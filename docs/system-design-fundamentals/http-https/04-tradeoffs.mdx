## The Core Tension

**HTTP's fundamental trade-off is connection overhead versus security and correctness.**

Every feature that makes HTTP reliable and secure — TLS encryption, keep-alive connections, status code semantics, header negotiation — adds latency, memory, or CPU cost. The simplest protocol (HTTP/1.0 over TCP, no keep-alive, no encryption) is the fastest per-request but the least capable. The richest protocol (HTTP/3 over QUIC with TLS 1.3, multiplexing, HPACK, 0-RTT) offers the best feature set but requires more complex infrastructure, larger connection state, and UDP-friendly network configuration.

There is no universal winner. The correct choice depends on your traffic pattern: an internal microservice API handling 1 million requests per second between two data centers has a different optimal configuration than a public-facing mobile API serving 10,000 requests per second over cellular networks.

To ground this: consider a single request. Under HTTP/1.1 with no keep-alive, one request costs one TCP handshake (1 RTT) and one TLS handshake (1-2 RTT). Total: 2-3 RTT before any application data flows. For a 50ms RTT, that is 100-150ms of overhead. Under HTTP/3 with 0-RTT session resumption, the same request costs 0 RTT of protocol overhead — the first packet carries application data. The trade-off: the HTTP/1.1 connection state is approximately 3KB of memory. The HTTP/3 connection state is approximately 15KB — 5x more memory for the connection state, but zero initial latency.

The tension is not resolvable with a single answer. It is a spectrum, and the right position on that spectrum depends on how many connections you need to maintain, what latency your users tolerate, and what resources (memory, CPU, network) you have available.

## Decision Matrix

| Decision | Option A | Option B | Choose A When | Choose B When |
|----------|----------|----------|---------------|---------------|
| HTTP vs HTTPS | Faster (no encryption overhead), simpler debugging, lower CPU usage on server | Protects data in transit, required for HTTP/2, trust signal to users, enables service worker APIs | Internal traffic within isolated VPC, no PII transferred, trusted network between services | Any cross-network traffic, user-facing APIs, regulatory requirements (GDPR, PCI, HIPAA, SOC2), mobile apps |
| HTTP/1.1 vs HTTP/2 | Universal compatibility, simple debugging with curl, lower CPU overhead per connection, mature infrastructure everywhere | Multiplexing reduces page load time by 40-60%, header compression saves 85% bandwidth on headers, server push (deprecated), single connection per origin | Low-latency networks, single-resource APIs, legacy proxy infrastructure that lacks HTTP/2 support | Multi-resource pages (40+ resources), high-latency networks, mobile apps, streaming responses, gRPC |
| TCP vs QUIC (HTTP/2 vs HTTP/3) | Mature infrastructure, deep tooling (tcpdump, Wireshark, ss), firewall-friendly (all networks allow TCP), kernel-level optimization (TCP offload) | 0-RTT reconnection, no head-of-line blocking, connection migration across network change, better lossy-network performance | Corporate networks that block UDP, internal DC-to-DC traffic over dedicated fiber, legacy clients (pre-2020) | Cellular/mobile users (20-40% packet loss typical), real-time services, global consumer audience, WiFi-to-cellular switching |
| Short connections vs keep-alive | Simple: close after each response, no state to manage on server, no idle connection memory cost | Reuses TCP+TLS handshake, saves 2-3 RTT per subsequent request, higher throughput for multiple requests | Very low request rates (under 1 req/minute per client), server memory constrained, clients that only make one request anyway | High request rates, high-latency networks, streaming responses, multiple related requests (a page and its sub-resources) |
| TLS termination at LB vs end-to-end | Lower backend CPU (offload encryption to LB), simpler certificate management at one point, LB can inspect content for WAF/routing | Full encryption to backend, no plaintext inside internal network, compliance-friendly, backend can authenticate client certs | Trusted internal network between LB and backend, performance-critical backends where CPU savings matter, compliance team accepts LB decryption boundary | Untrusted or shared network between LB and backend, strict compliance regimes (PCI DSS requires encryption to the application), zero-trust architecture |
| Stateless HTTP vs sessions/auth tokens | Simple: no server-side state, easy horizontal scaling, no session storage to fail, no session expiry management | User state persists across requests, enables shopping carts, login sessions, personalization, rate limiting per user | Each request is fully self-contained with all auth data in the request (JWT, signed headers), horizontal scaling is priority over personalization | User context spans multiple requests, legacy session-based frameworks (PHP, Rails before API mode), server-rendered applications |
| Content-Length vs chunked encoding | Simple: single header, client knows total size upfront, required for HTTP/1.1 persistent connections, enables download progress bars | Streaming: send data as it becomes available, no need to buffer entire response in memory, enables indefinite-length responses (server-sent events) | Response size known before first byte is sent, small payloads under 1MB, file downloads where progress indication matters | Dynamic or large responses where total size is unknown upfront, server-sent events, real-time streaming, generated content (reports, video transcoding) |
| Compression (gzip/brotli) vs raw | Saves 60-80% bandwidth on text responses (JSON, HTML, CSS, JS), faster page loads on constrained bandwidth | Saves CPU cycles (no compression or decompression), no latency for compression, works for already-compressed content | API responses with JSON bodies, HTML/CSS/JS responses, bandwidth-cost-sensitive deployments (CDN egress fees) | Binary content (images are already compressed), CPU-bound servers where every cycle matters, responses under 1KB where compression overhead may exceed savings |

## CAP Theorem Positioning

HTTP itself is not a distributed data system — it does not store state. But every HTTP-based service sits on top of distributed storage (databases, caches, queues). The CAP positioning of that storage system determines how HTTP responds under partition.

For a typical HTTP API backed by a CP database (PostgreSQL, MySQL with synchronous replication):

**CP (Consistency + Partition Tolerance)**: When a network partition occurs, the database favors consistency over availability. If the primary cannot confirm that a quorum of replicas has the latest write, it stops accepting writes. The HTTP API returns 503 Service Unavailable during the partition. Clients see errors but never see stale or conflicting data.

The impact on the HTTP API is that every mutating request becomes unreliable during a partition. Clients must handle the 503 by queuing the request locally or showing the user an error. The API should communicate the nature of the failure clearly in the response body: `{"error": "write_unavailable", "retry_after_seconds": 30}`. This gives the client actionable information — retry after 30 seconds, when the partition may have healed.

For an AP database (Cassandra, DynamoDB with eventual consistency):
**AP (Availability + Partition Tolerance)**: The database accepts writes from any reachable node during a partition. The API returns 200 OK for every request. But during the partition, different clients may see different data — Client A sees the order as confirmed, Client B sees it as pending. When the partition heals, the database reconciles the conflict using its configured strategy (last-write-wins, CRDT merge, or application-level conflict resolution).

From the HTTP layer, the API must communicate to clients that the write was accepted but consistency is eventual. The response might include a header: X-Consistency-Level: eventual. Clients that need strong consistency (read-your-writes) should include a header like X-Consistency: quorum, which the API gateway uses to route the read to the primary node instead of a replica.

## Failure Modes

### Failure Mode 1: TLS Certificate Expiry

**What happens**: The server's TLS certificate expires. Clients that verify certificates (all modern browsers, curl with --proxy-ca-cert, any API client using the system trust store) refuse to connect. The TLS handshake fails with a certificate error.

**Root cause**: The certificate's notAfter date has passed. Most TLS certificates are valid for 90 days (as of 2026, following Apple and Google's 90-day maximum validity policy). The short lifetime means renewal is needed every 3 months — a forgetful operations team without automation will eventually miss a renewal.

**Blast radius**: All HTTPS traffic to the affected domain stops. HTTP (port 80) traffic continues if the server still listens on port 80. Users see browser security warnings with red "Not Secure" indicators. API clients fail with SSL_ERROR_CERTIFICATE_EXPIRED or similar errors. Internal services that talk to this server also fail if they verify certificates (which they should).

**Detection**: Monitor certificate expiry dates on all load balancers, reverse proxies, and edge servers. Alert when a certificate has fewer than 14 days remaining. Use Certificate Transparency (CT) logs to catch domain-name squatting on soon-to-expire domains. Run a synthetic check from an external monitoring service every 5 minutes: `curl -vI https://example.com 2>&1 | grep "SSL certificate verify ok"`.

**Recovery**: Issue a new certificate. Deploy it to the load balancer or web server. Automated renewal (ACME/certbot) should handle this without human intervention in under 60 seconds. If you manage certificates manually, create a calendar reminder 30 days before every expiry. Practice the renewal procedure during a maintenance window.

### Failure Mode 2: Connection Pool Exhaustion

**What happens**: The database connection pool behind the HTTP API reaches its maximum size. New requests that need a database connection block waiting for one to become available. If the wait timeout expires, the request fails with a 503 or 500.

**Root cause**: A backend service slows down (slow query, lock contention, upstream API latency), causing each database transaction to take longer. The pool's connections are held open for longer. New requests queue up. Queue depth grows faster than connections become available. The pool resembles a traffic jam: once the queue reaches a critical density, it does not clear on its own even if the original slow query finishes.

**Blast radius**: All API endpoints that require database access fail. Any endpoint that does not touch the database continues working (health checks, static data, cached responses). If the pool exhaustion is caused by a specific slow query on one table, only the endpoint using that query fails initially — but as the queue backs up, the pool's shared nature means all endpoints eventually fail. The blast radius expands from one endpoint to the entire service.

**Detection**: Pool utilization metric (currently-active / max-active). Alert at 80% utilization. Monitor connection wait time — if it exceeds 100ms at p50, the pool is near capacity and action is needed before it exhausts.

**Recovery**: Kill the slow queries (PostgreSQL: pg_cancel_backend, MySQL: KILL QUERY). Increase pool size temporarily (double it as a stopgap). Add read replicas to offload read queries from the primary pool. Implement query timeout at the database driver level (e.g., Postgres statement_timeout = 30s, MySQL max_execution_time = 30000). After recovery, investigate the slow query's execution plan and add missing indexes.

### Failure Mode 3: Retry Storm (Thundering Herd)

**What happens**: A downstream service becomes slow. Its clients (upstream HTTP services) all have retry configured. They all wait for their first attempt to time out at approximately the same time, then all retry simultaneously. The downstream service, already struggling, receives N times the normal traffic.

**Root cause**: Clients use fixed retry intervals (retry after exactly 2 seconds) instead of jittered exponential backoff. When the downstream fails, N clients all retry at exactly the same instant. The amplification factor is clients × retries: 100 clients × 3 retries = 300 requests where only 100 were intended.

**Blast radius**: The downstream service fails completely under the amplified load. The retry storm cascades: upstream service A calls downstream B. B's 503 causes A's clients to retry, amplifying the load on A, which then retries B even more. This cascade can take down an entire data center in under 30 seconds.

**Detection**: Sudden spike in request rate to a specific endpoint combined with elevated error rates. Monitor client-side retry counts. A sustained retry rate above 5% of total requests is a warning sign.

**Recovery**: Implement jittered exponential backoff on all clients. Use circuit breakers: after a configurable error threshold (e.g., 5 consecutive failures), the circuit opens and all requests fail fast (no retries) for a cooldown period (30 seconds). This gives the downstream service time to recover without being overwhelmed by retries.

### Failure Mode 4: DNS Propagation Delay During Migration

**What happens**: You change the DNS A record for api.example.com from 203.0.113.10 to 203.0.113.20. Clients with cached DNS records continue sending requests to the old IP for the duration of the TTL. The old server is being decommissioned.

**Root cause**: DNS TTL was set too high (e.g., 86400 seconds = 24 hours). Changing the DNS record does not flush client-side caches — each client respects the TTL from the previous query.

**Blast radius**: A percentage of users proportional to DNS TTL multiplied by request rate. During a 24-hour TTL window, clients gradually migrate. If the old server is taken offline immediately, those clients experience connection failures. For a service with 10 million daily users, a 24-hour TTL means up to 10 million users could be affected.

**Detection**: Monitor request rates to old versus new IP addresses. A gradual decline on the old IP confirms the migration is in progress. If requests to the old IP fall off a cliff (drop 100% in 1 minute), the old server was likely taken offline before all DNS caches expired.

**Recovery**: Before migration, lower DNS TTL to 60 seconds at least 48 hours in advance (2x the old TTL to allow the old high-TTL records to expire everywhere). After migration, keep the old server online for 2x the new TTL window (120 seconds minimum, but keep it for 24 hours as a safety margin). Use a load balancer that can route traffic to both old and new infrastructure during the migration window. Use multiple A records with the same TTL rather than replacing a single record.

### Failure Mode 5: HTTP/2 Head-of-Line Blocking at TCP Level

**What happens**: HTTP/2 multiplexes multiple streams over a single TCP connection. A TCP packet loss affects all streams on that connection, even though only one stream's data was in the lost packet. All streams stall until TCP retransmits the lost segment.

**Root cause**: TCP guarantees in-order delivery. If packet 5 (carrying stream 3 data) is lost, packets 6-10 (carrying streams 1, 2, 5 data) cannot be delivered to the application until packet 5 is retransmitted. This is TCP-level head-of-line blocking, different from HTTP/1.1's application-level head-of-line blocking.

**Blast radius**: All streams sharing the connection with the lost packet — potentially 100+ concurrent requests — stall simultaneously. On a network with 2% packet loss, this adds 50-200ms of delay per loss event. For mobile users on cellular with 20% packet loss, HTTP/2 can be slower than HTTP/1.1 because HTTP/1.1's 6 independent connections limit the blast radius to 1/6 of requests.

**Detection**: Monitor TCP retransmission rate on the server side (netstat -s | grep retransmit). Compare per-request latency under HTTP/2 versus HTTP/1.1. If HTTP/2 is slower than HTTP/1.1 on lossy networks, TCP HoL blocking is the cause.

**Recovery**: Migrate to HTTP/3 over QUIC, which eliminates TCP-level HoL blocking. QUIC delivers packets from each stream independently — packet loss on stream 3 does not delay stream 5's data. As a partial mitigation while on HTTP/2, limit max_concurrent_streams to 32 or 64 to reduce the blast radius of a single packet loss.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: 200 OK for Everything

**The mistake**: The API always returns 200 OK, with the actual status encoded in the response body. For example: `{"error": "user_not_found", "code": 404}`.

**Why it was reasonable**: "Consistent parsing" — the client always checks for 200, then switches on the body's status field. This made error handling look like success handling. Some frameworks (especially early REST implementations) encouraged this pattern for simplicity.

**Production impact**: Monitoring systems (load balancers, API gateways, service meshes) log error rates based on HTTP status codes. A service returning 200 for every response shows 0% error rate. Alerts never fire. Caches cache error responses as if they were success. Retry logic never triggers. A catastrophic failure becomes invisible. At one company, this anti-pattern caused a 4-hour outage: the order service returned 200 with `{"error": "database_unavailable"}` for every request, and nobody noticed for 4 hours because the HTTP error rate was zero.

**Correct pattern**: Use HTTP status codes as specified in the RFC. 200 means success. 4xx means client error. 5xx means server error. The body provides details, but the status code is the primary signal that all infrastructure components (load balancers, monitoring, caches, API gateways) rely on.

### Anti-Pattern 2: Retry Without Backoff

**The mistake**: A client retries failed HTTP requests immediately, with no delay between retries.

**Why it was reasonable**: "The failure might be transient. Try again right away." Some developers assume that the fastest retry is the best retry.

**Production impact**: Retry at 1 million requests/second amplifies into 2 million requests/second instantly. The downstream service collapses. Other services sharing the same infrastructure also degrade. This is how a single slow database query becomes a company-wide outage. In 2017, this pattern caused a 6-hour outage at a major e-commerce company: a retry storm from 50,000 mobile apps took down the entire checkout pipeline on Black Friday.

**Correct pattern**: Exponential backoff with jitter: first retry after 100ms, second after 400ms, third after 1.6s, fourth after 6.4s, with +/- 50% random jitter. Maximum retries: 3-5. Circuit breaker: stop retrying after N consecutive failures and fail fast for a cooldown period.

### Anti-Pattern 3: Infinite Keep-Alive

**The mistake**: No keep-alive timeout configured. Connections stay open forever.

**Why it was reasonable**: "The client might send another request any time. Closing and reopening would be wasteful." Without a timeout, no one thinks about closing connections until the server runs out of memory.

**Production impact**: Each idle connection consumes 3-30 KB of server memory (depending on protocol version — HTTP/1.1 is cheapest, HTTP/3 is most expensive). With 1 million concurrent idle connections, that is 3-30 GB of RAM wasted. The server runs out of memory and restarts. All connections are lost. The cycle repeats. Engineers wonder why the server restarts every 4 hours and blame the OS.

**Correct pattern**: Set keep-alive timeout to 10-75 seconds. Set a maximum number of keep-alive requests per connection (e.g., 1000 in Nginx). The connection manager should aggressively reclaim idle connections when memory pressure is high.

### Anti-Pattern 4: Ignoring 429 (Too Many Requests)

**The mistake**: The client does not handle the 429 status code. It either crashes, retries immediately, or treats 429 as success.

**Why it was reasonable**: "Our API never returns 429. We did not think we needed to handle it." Many developers only handle the status codes they expect — 200, 404, 500.

**Production impact**: The client sends requests at full rate despite the server asking it to slow down. The server's rate limiter keeps returning 429, consuming CPU on both sides to generate and process the responses. Eventually, the server's rate limiter's own resources become exhausted, and the server falls over. During traffic spikes, this pattern turns a temporary overload into a complete outage.

**Correct pattern**: Every client must handle 429 by reading the Retry-After header (or RateLimit-Reset) and waiting the specified duration before the next request. Log 429 responses for visibility. Implement a local rate limiter that matches the server's limits so the client self-regulates before the server returns 429.

### Anti-Pattern 5: Self-Signed Certificate in Production

**The mistake**: Using a self-signed TLS certificate in production instead of a CA-issued certificate.

**Why it was reasonable**: "It encrypts the traffic. That is the important part. A CA certificate costs money or setup effort." In the early days of TLS, certificates cost $100-500/year, and automation was not widely available.

**Production impact**: Every client must explicitly disable certificate verification (--insecure flag in curl, SSLVerify=none in Java), or add the self-signed cert as a pinned exception. The verification bypass means the client is now vulnerable to MITM attacks — exactly what TLS is supposed to prevent. New developers or services connecting to this endpoint forget to bypass verification and spend hours debugging "SSL_ERROR_UNKNOWN_CA" errors. When auditors discover the self-signed certificate, they flag it as a security finding that must be remediated.

**Correct pattern**: Use Let's Encrypt (free, automated) for public-facing services. Use an internal CA (e.g., HashiCorp Vault PKI, cert-manager in Kubernetes) for internal services. Never bypass certificate verification in production code.

### Anti-Pattern 6: Tunneling All Traffic Through a Single Connection

**The mistake**: All client requests go through a single HTTP/1.1 keep-alive connection. Requests are serialized.

**Why it was reasonable**: "One connection is simpler. No connection pool to manage. Less code." This is common in prototypes and scripts that evolved into production services.

**Production impact**: A single slow request blocks all subsequent requests. If one API call takes 10 seconds (e.g., a report generation), all other API calls on that connection wait. A mobile app that makes 20 API calls on page load would take 20 sequential round trips = potentially 10+ seconds of wall-clock time. With a pool of 10 connections, that same work completes in 2 rounds.

**Correct pattern**: Use HTTP/2 for multiplexing over one connection. If HTTP/1.1 is required, use a connection pool with at least 10 connections per destination. Use multiple connections from different client instances.

## When NOT to Use This

HTTP is not always the right protocol. Over-engineering an HTTP solution when a simpler protocol suffices wastes infrastructure and complexity. Recognizing when HTTP is the wrong tool is a senior engineer's skill.

**Signs you are over-engineering HTTP**:

- **Real-time bidirectional communication**: You have polling endpoints that clients hit every 200ms. Replace with WebSocket (for arbitrary bidirectional messages) or Server-Sent Events (for unidirectional server-to-client streaming). This reduces server load by 99% (from 5 requests/second per client to 0 if using push) and improves latency from 200ms-polling to sub-10ms push.

- **Service-to-service IPC within the same process**: You run two modules that communicate over HTTP on localhost. Replace with direct function calls or shared memory. HTTP over localhost still serializes (JSON or protobuf to bytes), encrypts (TLS if HTTPS), decrypts, and parses — overhead that is wasted when both sides are in the same process and share memory space. A function call is nanoseconds; HTTP on localhost is microseconds — three orders of magnitude slower.

- **Streaming large binary data**: You use HTTP range requests to download large files. For downloads exceeding 100 MB, consider a dedicated binary protocol (BitTorrent for massive distribution, S3 Multipart Upload for uploads, rsync for sync over SSH) rather than HTTP GET with byte ranges. HTTP's per-request overhead adds up: 256 range requests on a 1 GB file at 200ms each = 51 seconds of HTTP overhead.

- **IoT sensor data at high frequency**: Sensors sending 1,000 readings per second per device. The HTTP request-response model adds protocol overhead of 400+ bytes per reading. At 1,000 readings/s, that is 400 KB/s of protocol overhead per device. Switch to MQTT (3-byte minimum header) or CoAP (UDP-based, designed for constrained devices). These protocols are designed for the IoT use case — low bandwidth, intermittent connectivity, constrained devices.

- **Internal microservices with strict SLOs under 5ms**: HTTP/1.1 or HTTP/2 adds 1-3ms of serialization/deserialization overhead per hop. For sub-5ms latency requirements, use gRPC (binary protocol, 0.2ms overhead) or raw TCP with a custom framing. Every millisecond counts when your SLO is 5ms — HTTP overhead consumes 20-60% of your budget.

**Scale thresholds where simpler alternatives work**:

| Current setup | At this scale | Simplify to | Why |
|---------------|---------------|-------------|-----|
| HTTPS for internal K8s services | Under 500 req/s per service | HTTP within same namespace | Network is isolated by Kubernetes network policies, no sensitive data crosses pod boundaries, TLS overhead (CPU + latency) is pure waste |
| Load balancer with TLS termination | Under 100 req/s per server | Single server, no LB | One server handles the load. LB adds cost ($20-100/month for cloud LB) and complexity (config management, health checks) |
| CDN for a B2B API | Under 10,000 req/s global | Single region with caching | B2B clients (other companies) are not geographically distributed like consumers. If your API serves 20 enterprise customers, they do not need 330 edge PoPs. CDN adds latency for non-cacheable API responses — every cache miss costs an extra TCP/TLS round trip to the edge |
| HTTP/3 on a data-center-to-data-center link | Dedicated fiber between DCs | HTTP/2 over TCP | Zero packet loss on dedicated fiber means QUIC's loss-recovery benefits do not apply. TCP is faster on zero-loss links because kernel TCP stacks are more optimized than userspace QUIC stacks. QUIC's encryption is mandatory (no unencrypted QUIC), which adds CPU overhead on high-throughput DC links |

**The golden rule**: Use the simplest protocol that meets your latency, security, and throughput requirements. Add complexity only when you have measured that the simpler option does not work — not because a more complex option exists. A common mistake is designing for 10 million users when you have 10,000. The simplest solution that works for 10K will usually work for 100K with minor tuning. You will know when you need HTTP/3 because your p99 latency on cellular will exceed your SLO and you will have measured that TCP packet loss is the cause.
