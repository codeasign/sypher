## The Problem Story

On a Tuesday morning at 09:47, a major airline's check-in system went offline globally. Passengers at 47 airports could not check in, print boarding passes, or tag luggage. The root cause was not a server crash, a database failure, or a cloud outage. A single HTTP response — sent from a load balancer to a mobile app — contained a status code the app had never seen before. The app interpreted the unfamiliar code as a success, displayed "Baggage tagged successfully" to the passenger, and did not forward the bag to the aircraft. By the time engineers traced the issue, 1,200 bags were in the wrong cities.

The gap was not a bug in the code. It was a gap in understanding how the protocol that carries every web request and response actually works. Every API call, every page load, every image fetch runs on HTTP or its encrypted sibling HTTPS. When you understand the protocol, you predict failure modes before they happen. When you treat it as a black box, a single unexpected status code grounds 1,200 bags.

Consider a food delivery app that loses orders at peak dinner hours. The mobile app sends a POST request to place an order, gets back a 200 OK, and shows "Order confirmed" to the customer. The restaurant never sees the order. The customer waits 90 minutes, then calls support. The support agent sees the order in the database. The restaurant manager's tablet shows nothing. Where did the order go? The answer lies in the difference between HTTP at the load balancer, HTTP between the app and the API gateway, and HTTP between the gateway and the internal service mesh — three distinct conversations over the same protocol, with different headers, timeouts, and retry policies.

A ride-sharing company experienced a similar outage during New Year's Eve. Surge pricing triggered a 429 (Too Many Requests) on fare estimates. The mobile SDK treated 429 as a transient error and retried immediately — 50,000 times per second. The retry storm collapsed the pricing service. The protocol did what it was told. The problem was nobody had modeled how HTTP flows across 15 microservices under 100x load.

## The Insight

HTTP (Hypertext Transfer Protocol) is a request-response protocol that defines how one computer asks another computer for data and how the second computer replies — using a uniform set of methods, status codes, and headers that any implementation must follow. HTTPS is HTTP wrapped in Transport Layer Security (TLS) encryption so that nobody between the two computers can read or modify the conversation.

A postal system runs on the same principles. You write a letter (the request) on a standardized envelope — destination address (URL), return address (origin IP), and handling instructions (headers). You drop it in a mailbox. The postal service routes it through sorting facilities (routers, load balancers, proxies) to the recipient. The recipient opens it, writes a reply, and drops it back. The postal service does not guarantee delivery — letters get lost, delayed, or damaged. If you need proof of delivery, you pay extra for registered mail (HTTPS with certificate verification). If you need a guaranteed answer, you use certified mail with return receipt (HTTP with idempotency keys and retry logic).

The critical insight: HTTP is stateless. Each letter is independent. The postal service does not remember what you mailed yesterday. If you need continuity — a conversation, not isolated letters — you and the recipient must invent a convention: "include your previous letter's tracking number in every new letter." That convention is what cookies, session tokens, and correlation IDs do.

## Core Vocabulary

**HTTP Method** — The verb that describes the action the requester wants to perform: GET (retrieve), POST (create), PUT (replace), PATCH (modify), DELETE (remove). Methods matter because they define idempotency: repeating a GET or PUT produces the same result; repeating a POST may create duplicates.

**Status Code** — A three-digit number in every response that tells the requester what happened. 1xx (informational), 2xx (success), 3xx (redirection), 4xx (client error), 5xx (server error). Status codes matter because they are the primary signal for error handling — a 503 tells the caller to back off; a 404 says the resource does not exist.

**Header** — Key-value metadata sent with every request and response: content type, caching policy, authentication tokens, compression preferences. Headers matter because they control behavior that the URL and body cannot express — like "do not cache this response" (Cache-Control: no-store) or "the body is compressed with gzip" (Content-Encoding: gzip).

**URL (Uniform Resource Locator)** — The address of the resource being requested: scheme, host, port, path, query parameters, fragment. URLs matter because they are the only way the client tells the server what it wants — a mistyped URL produces a 404 or, worse, hits the wrong environment.

**TLS Handshake** — The cryptographic negotiation that happens before any HTTPS request: the client and server agree on a cipher suite, exchange certificates, and derive session keys. The handshake matters because it adds 1-3 round trips of latency before the first byte of application data flows.

**Certificate** — A digitally signed document that binds a public key to a domain name, issued by a Certificate Authority (CA). Certificates matter because they prevent man-in-the-middle attacks — without one, an attacker on the same Wi-Fi network can intercept every request.

**Idempotency** — The property that repeating the same operation produces the same result as a single execution. HTTP PUT and DELETE are idempotent by design; POST is not. Idempotency matters for retry safety — if a PUT times out, the client can retry safely; if a POST times out, the client needs an idempotency key to prevent duplicate charges.

**Content-Type** — The header that tells the receiver how to interpret the body: application/json, text/html, multipart/form-data, application/octet-stream. Content-Type matters because the wrong value causes parsers to reject valid data or misinterpret binary payloads.

**Connection Keep-Alive** — A mechanism where the TCP connection stays open after a response so multiple requests reuse the same socket, avoiding the overhead of repeated TLS handshakes. Keep-alive matters because opening a new TLS connection costs 2-3 round trips of latency — reusing a connection saves 80-150ms per request.

**CORS (Cross-Origin Resource Sharing)** — A set of HTTP headers that tells the browser whether a web application running at one origin can access resources at another origin. CORS matters because browsers enforce the same-origin policy by default — missing CORS headers cause valid API calls to fail silently with opaque errors.

The stack operates as a set of nested envelopes. The application layer (HTTP) produces a message — method, URL, headers, body. If using HTTPS, the presentation layer wraps that message in encryption (TLS record). The transport layer (TCP) chunks the encrypted data into segments, assigns sequence numbers, and ensures reliable delivery. The internet layer (IP) puts each segment into a packet with source and destination addresses and routes it across the network. The receiving side reverses the process: IP delivers the packets, TCP reassembles them in order, TLS decrypts the records, and HTTP presents the request to the server application.

## How It Actually Works — Step by Step

### Step 1: URL Parsing and DNS Resolution

The client (browser or app) parses the URL into components: scheme (https), host (api.example.com), port (443, implied), path (/v2/users), and query parameters (?page=1). It checks its DNS cache for the host's IP address. On a cache miss, it sends a DNS query — typically to a recursive resolver, which traverses the DNS hierarchy: root nameserver, TLD nameserver (.com), authoritative nameserver (example.com). The authoritative server returns the IP address (e.g., 203.0.113.42).

### Step 2: TCP Three-Way Handshake

The client initiates a TCP connection to port 443 on the resolved IP address. The three-way handshake establishes a reliable byte stream:

1. Client sends SYN (synchronize) with a random sequence number (ISN-C).
2. Server responds with SYN-ACK (synchronize-acknowledge), its own ISN-S, and acknowledges ISN-C+1.
3. Client sends ACK acknowledging ISN-S+1.

After the handshake, both sides maintain a connection state: send buffer, receive buffer, sequence numbers, and window size for flow control. This state lives in kernel memory on both machines.

### Step 3: TLS Handshake

Since the URL uses HTTPS, the client and server negotiate encryption before any HTTP data flows. Modern TLS 1.3 reduces the handshake to one round trip (1-RTT) or zero round trips (0-RTT with session resumption):

1. Client sends ClientHello: supported TLS versions, cipher suites, and a random nonce.
2. Server responds with ServerHello: chosen TLS version and cipher suite, its certificate chain, and a digital signature proving it owns the private key corresponding to the certificate.
3. Client verifies the certificate chain against its trust store. It checks: each certificate is signed by the next CA in the chain, none have expired, none are revoked, and the server's certificate's Common Name or Subject Alternative Name matches the hostname in the URL.
4. Both sides derive the same session key using ephemeral Diffie-Hellman key exchange. The key never crosses the network — each side computes it independently from public parameters and their own private value.

### Step 4: HTTP Request Construction

The client constructs the HTTP request. A GET request has no body. A POST request includes a body with the Content-Type and Content-Length headers. The request is serialized into bytes using the ASCII encoding for the start line and headers, and the encoding specified by Content-Type for the body.

```
GET /v2/users?page=1 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Accept: application/json
Accept-Encoding: gzip
Cache-Control: no-cache
User-Agent: SypherApp/1.0
Connection: keep-alive
```

For HTTP/2 and HTTP/3, the request is compressed using HPACK (HTTP/2) or QPACK (HTTP/3) header compression — a Huffman-coded dictionary that reduces repetitive headers like Cookie and User-Agent by 85-90%.

### Step 5: Network Transit

The encrypted bytes travel through the network stack: the client's OS encapsulates them in TCP segments, then IP packets, then Ethernet frames (or Wi-Fi frames). Each frame passes through the client's default gateway (home router or office firewall), across the internet backbone through a series of routers (12-25 hops typically), through the server's firewall and load balancer, and finally to the server's network interface.

At each hop, the router looks at the destination IP, consults its forwarding table, and sends the packet to the next hop. Intermediate routers do not read the encrypted TLS payload — they only inspect the IP headers and (for TCP) the port number.

### Step 6: Load Balancer Termination

If the server uses a load balancer (software like Nginx or HAProxy, or cloud like AWS ALB), the load balancer terminates the TLS connection. It decrypts the request, examines the Host header to determine which backend pool should handle it, and either:

- Forwards the decrypted request to the backend over HTTP (TLS termination at LB)
- Re-encrypts and forwards over HTTPS (TLS pass-through or re-encryption)

The load balancer adds headers like X-Forwarded-For (the client's original IP), X-Forwarded-Proto (the original protocol), and X-Request-Id (a unique trace ID).

### Step 7: Server Processing

The backend server receives the HTTP request. The web server (e.g., Tomcat, uWSGI, Express.js) parses the request line, headers, and body into its internal data structures. It routes the request to the appropriate handler based on the URL path and HTTP method.

The handler executes application logic: authenticates the user from the Authorization header, queries a database, calls other services, constructs the response. During processing, the server may write to a database, call a cache, or forward requests to internal services — each of which is another HTTP conversation over the same protocol.

### Step 8: HTTP Response Construction

The server constructs the HTTP response:

```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 452
Cache-Control: public, max-age=300
ETag: "abc123"
X-Request-Id: req-7a81f2c4
Date: Tue, 01 Jul 2026 09:47:00 GMT

{"users":[{"id":1,"name":"Alice"},...]}
```

The response includes the status code, headers, and body. The server may compress the body with gzip or brotli depending on the client's Accept-Encoding header.

### Step 9: Response Transit and Decryption

The response travels back through the same network path. If TLS is used end-to-end, the load balancer encrypts the response before sending it to the client. The client's TLS layer decrypts each record as it arrives and passes the plaintext bytes to the HTTP parser.

### Step 10: Response Parsing and Handling

The client's HTTP library parses the response: validates the status code, parses headers into a key-value map, decompresses the body if needed, and deserializes the body according to Content-Type. The application then acts on the response — render the page, update the UI, trigger the next request.

If the status code indicates an error (4xx or 5xx), the application should handle it: retry for 5xx (server error), fail gracefully for 4xx (client error), follow the Location header for 3xx (redirect). Most production failures come from status codes that were not handled — 429 (rate limited) treated as success, 503 (unavailable) retried immediately without backoff.

## Variants and Flavors

### HTTP/1.1 — The Baseline

HTTP/1.1, standardized in 1997, is the version most developers first learn. It uses text-based request and response messages separated by CRLF line breaks. Each request-response pair consumes one connection for the duration of the exchange. Multiple requests on the same connection are serial (pipelining exists but is rarely used due to head-of-line blocking).

**Use when**: You need maximum compatibility. Every HTTP library, every server, every proxy speaks HTTP/1.1. It is the baseline that all other versions are measured against.

**Do not use when**: You have many small requests (an API client fetching 50 resources on page load) or high-latency networks (mobile). The head-of-line blocking adds cumulative latency: 50 requests at 100ms each = 5 seconds serial.

### HTTP/2 — Multiplexed and Binary

HTTP/2 (2015) replaces text framing with a binary protocol that multiplexes multiple streams over a single TCP connection. Streams are independent — a slow response on stream 1 does not block stream 2. Headers are compressed with HPACK, reducing overhead by 85-90%. The server can push resources the client has not yet requested (server push), though browser adoption has been limited.

**Do not use when**: Your infrastructure lacks HTTP/2 support (legacy load balancers, old proxy servers), or you need raw throughput for very large single transfers (HTTP/1.1 pipelining is simpler for streaming downloads).

### HTTP/3 — QUIC-Based

HTTP/3 (2022) replaces TCP with QUIC, a transport protocol built on UDP. QUIC integrates TLS 1.3 natively, eliminates head-of-line blocking at the transport layer, and provides 0-RTT connection establishment for returning clients. Each stream is truly independent — packet loss on stream 3 does not delay stream 5.

**Do not use when**: Your network infrastructure blocks UDP (some corporate firewalls), your users are on very old clients (pre-2020 operating systems), or you need the simplest possible debugging tooling (HTTP/3 traffic is harder to capture with Wireshark than HTTP/2 or HTTP/1.1).

### HTTP vs. HTTPS — The Encryption Dimension

This is not a version variant but a security variant. HTTP sends all data as plaintext. HTTPS encrypts everything except the destination IP address and the SNI (Server Name Indication) field. The difference matters at every level: HTTP on port 80 can be intercepted by anyone on the same network; HTTPS on port 443 requires a valid certificate chain and protects against passive eavesdropping, active tampering, and man-in-the-middle attacks.

For internal service-to-service communication within a VPC or Kubernetes cluster, HTTP is sometimes acceptable because the network itself is isolated. For anything crossing a network boundary — including between cloud regions, or between a service and a database — HTTPS should be the default.

### HTTP/1.0 vs HTTP/1.1 — The Connection Reuse Dimension

HTTP/1.0 (1996) opened a new TCP connection for every request-response pair. After each response, the server closed the connection. Loading a page with 10 images required 10 separate TCP connections, each with its own three-way handshake. On a 50ms RTT connection, that is 500ms of TCP handshake overhead alone before any data transfer.

HTTP/1.1 (1997) introduced persistent connections (Connection: keep-alive by default) and pipelining. A single connection can handle multiple request-response pairs sequentially. The TCP and TLS handshake cost is paid once; subsequent requests on the same connection have zero handshake overhead.

**Use when**: Users make multiple requests to the same server (nearly every web application). HTTP/1.1 is the minimum viable version for any production service today.

**Do not use when**: Clients make only one request and never return (IoT sensors that send telemetry once per session). In that case, HTTP/1.0 is simpler and the connection reuse feature provides no benefit.

### Connection: close vs keep-alive — The Per-Request Dimension

Every HTTP request includes a Connection header. The value determines whether the TCP connection persists after the response:

- **Connection: close**: The server closes the TCP connection after sending the response. The next request from the same client must open a new connection (new TCP handshake, new TLS handshake).
- **Connection: keep-alive**: The server keeps the connection open for a configurable timeout (default 75s in Nginx, 5s in AWS ALB). The next request from the same client reuses the existing connection.

The performance difference is dramatic. Without keep-alive, each request costs 2-3 RTT for TCP+TLS handshakes. With keep-alive, each subsequent request costs 0 RTT for handshakes. For an API client making 50 requests per session, keep-alive saves approximately 100-150 RTTs worth of latency.

**Use when**: Clients make multiple requests over a short time period (web browsers, mobile apps, API clients).

**Do not use when**: Clients make infrequent requests (once per minute or less). The server wastes memory holding idle connections for clients that may not return within the keep-alive timeout.

## Mental Model Check

**Question 1**: A mobile app sends a POST request to create a payment. The server processes the payment successfully but the TCP connection drops before the response arrives. The client retries. Is that a problem?

**Answer**: Yes, unless the server uses idempotency keys. The first request completed the payment. The retry (identical POST) creates a second payment. HTTP POST is not idempotent by default. The fix: the client includes an Idempotency-Key header with a unique value per request. The server checks if it has already processed that key — if yes, it returns the stored response instead of processing again. Stripe's API processes over $5 billion in payments per year using exactly this pattern.

**Question 2**: You open a website that loads 80 resources (HTML, CSS, JS, images, fonts). With HTTP/1.1 and 6 parallel connections, how many round trips does it take?

**Answer**: 80 resources divided by 6 connections = 14 batches (13 full + 1 partial). At 80ms RTT with 20ms server processing per request, total load time is 14 x (80 + 20) = 1,400ms. With HTTP/2 multiplexed over one connection, all 80 requests can be sent simultaneously: one RTT for all requests, plus server processing time for the slowest response. HTTP/2 load time is approximately 80 + 200 = 280ms (assuming some resources need backend computation). The improvement is 5x faster page load.

**Question 3**: Your API returns 200 OK with an empty body because of an internal error instead of returning 503. What happens?

**Answer**: The client interprets 200 OK as success and processes the empty body. If the client expects a JSON array, parsing [] succeeds silently. The upstream system thinks the request was successful. No alert fires — 200 OK responses are not monitored as errors. The bug is invisible until a downstream system discovers missing data. This is why returning accurate status codes matters: 200 OK means "I got your request, understood it, processed it, and here is the result." Returning 200 when processing failed is breaking the protocol contract.

**Question 4**: What happens if your server sends a Set-Cookie header in a response, but the client never received the response because the connection dropped?

**Answer**: The cookie is never stored. The client does not know about the cookie. The next request from the client does not include that cookie. This is a common source of "sometimes the user is logged in, sometimes not" bugs. The fix: the server should treat cookie setting as part of a transactional operation — either the full response (including the Set-Cookie) is delivered, or none of it is. Using idempotency keys combined with response caching on the server side ensures that retry requests receive the same cookies.

**Question 5**: Your API is under DDoS attack. The attackers send SYN packets to port 443 at 1 million per second but never complete the TCP handshake. What happens to legitimate users?

**Answer**: The server's SYN backlog fills up with half-open connections. Once the backlog (typically 1024 on Linux default) is full, the kernel drops new SYN packets — including legitimate ones. Legitimate users see connection timeouts. This is called a SYN flood. Mitigations: SYN cookies (the kernel responds to SYNs without allocating state until the ACK arrives), rate-limiting of SYNs per source IP, and DDoS scrubbing services like Cloudflare or AWS Shield that absorb the attack before it reaches your server.

**Question 6**: Your API gateway adds an X-Request-Id header to every request. Your downstream services propagate this header in their HTTP calls to each other. What problem does this solve?

**Answer**: This solves the problem of correlating a single user request across multiple service hops. When a user's request travels through API gateway → service A → service B → database, any single component failure produces logs with different connection IDs and process IDs. Without a shared trace ID, reconstructing the path of a single request from user to database requires manual correlation of timestamps and IP addresses. With X-Request-Id, you grep for the trace ID across all services and see every hop in order. This is the foundation of distributed tracing — every major tracing system (OpenTelemetry, Zipkin, Jaeger) uses an HTTP header (usually traceparent or x-request-id) to propagate trace context.

**Question 7**: You deploy a new version of your API that has a bug — it reads from the wrong database table, returning empty results. You realize the bug 10 seconds after deployment. What HTTP caching behavior could make this worse?

**Answer**: If the API responses include Cache-Control: public, max-age=3600, a CDN or proxy cache will serve the empty results to all users for up to 1 hour. Rolling back the deployment does not fix the problem because the cached empty responses persist. Even after the fix is deployed, users still see empty data until the cache TTL expires or the cache is manually purged. The lesson: when deploying a new version, either (a) set Cache-Control to no-cache during rollout, or (b) ensure the new version uses new URLs (e.g., /v3/ instead of /v2/) so old cached responses are not affected, or (c) purge the cache immediately after rollback.
