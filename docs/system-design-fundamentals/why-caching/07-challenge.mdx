These challenges test your understanding of caching concepts, your ability to design caching systems under realistic constraints, and your judgment about when caching helps and when it harms. Complete them in order — each tier builds on the previous.

## Tier 1 — Foundation

These challenges verify you understand the core concepts and vocabulary. Each should take 5-10 minutes.

### Challenge 1.1: Explain the Cache Miss Cost

A team is considering adding a cache to their web application. They expect a cache hit ratio of 90%. A database query costs 150 milliseconds on average. A cache lookup costs 2 milliseconds on average. They currently serve 100,000 requests per second directly from the database, which uses 15 database instances at 70% CPU.

Calculate the following:

1. What is the average response time after caching (combining hit and miss paths)?
2. How many database queries per second reach the database after caching?
3. How much database capacity can they reclaim (how many instances)?
4. What happens if the hit ratio drops to 50% due to a bad deployment?

**Model answer:**

1. Average response time = (hit ratio x cache latency) + (miss ratio x (cache latency + database latency)) = (0.90 x 2ms) + (0.10 x (2ms + 150ms)) = 1.8ms + 15.2ms = 17ms. The average response time drops from 150ms to 17ms — a 9x improvement.

2. Database queries per second after caching = 100,000 x 0.10 = 10,000 QPS. A 10x reduction from the original 100,000 QPS.

3. The original database handled 100,000 QPS across 15 instances at 70% CPU, or roughly 6,667 QPS per instance. After caching, 10,000 QPS requires 10,000 / 6,667 = 1.5 instances — but you need at least 2 for redundancy. This means reclaiming 13 of 15 database instances, saving approximately 85% of database costs.

4. At a 50% hit ratio, the average response time = (0.50 x 2ms) + (0.50 x 152ms) = 1ms + 76ms = 77ms. Still better than 150ms, but 50,000 QPS reach the database — 50% of the original load. The database would need 50,000 / 6,667 = 7.5 instances, still above the original 15 but well within what 15 instances can handle. The system survives the bad deployment, though with reduced performance margin.

### Challenge 1.2: Compare Cache Strategies

You are designing a caching system for a social media application's user profile service. Profiles are read 10,000 times per second and written 50 times per second. Profiles change infrequently (users update their bio, profile picture, or settings).

Compare the following three caching strategies for this workload:

1. Cache-aside with TTL of 5 minutes and explicit invalidation on write.
2. Write-through caching (every write goes through the cache to the database).
3. No caching at all.

For each strategy, describe the average read latency, write latency, database load, and staleness characteristics.

**Model answer:**

Strategy 1 — Cache-aside with TTL + invalidation: Read latency is 2ms (cache hit, 99.5% of reads) or 152ms (cache miss, 0.5% of reads). Average read latency is approximately 2.75ms. Write latency is 5ms (database write only) plus 1ms (cache invalidation command) = 6ms total. Database load is 50 writes/second + 50 reads/second (approximately 0.5% of 10,000 reads miss the cache) = 100 QPS, down from 10,050 QPS without caching. Staleness window is zero for explicitly invalidated entries — the invalidation happens synchronously after the write, so the next read always gets fresh data. Entries that expire naturally via TTL may be up to 5 minutes stale, but this only affects rarely accessed profiles.

Strategy 2 — Write-through: Read latency is 2ms (cache hit) for every read — no misses because the cache is always populated. Average read latency is 2ms. Write latency increases to 5ms (database write) + 2ms (cache write) = 7ms, assuming the cache and database writes are sequential. (Parallel writes reduce latency but add complexity.) Database load is 50 writes/second only — zero read queries reach the database. Staleness window is zero — every write immediately updates the cache.

Strategy 3 — No caching: Read latency is 150ms for every read. Write latency is 5ms. Database load is 10,050 QPS. Staleness is not applicable (no cache). This is the worst option for this workload — high latency, high database cost, no benefit.

For this specific workload (read-heavy, infrequent writes), cache-aside with explicit invalidation is the best choice. The writes are so infrequent that write-through's extra write latency is negligible, but the additional complexity of write-through is unnecessary when invalidation provides the same freshness guarantee at lower write cost.

### Challenge 1.3: Identify the Cache Problem

A team deploys a cache to reduce database load. After deployment, they observe:

- Cache hit ratio: 2%
- Average response time: 165ms (was 150ms before caching)
- Database CPU: 95% (was 90% before caching)
- Cache CPU: 5%

Identify the most likely root cause. List three specific things you would check. Propose a fix.

**Model answer:**

The cache is making the system worse. A 2% hit ratio means 98% of requests miss the cache, so the application is paying the cost of a cache lookup (2-5ms) on every request without benefiting. The database CPU actually increased because the application servers are processing more work (cache check + database query) per request.

Three things to check:

1. **Cache key format:** Are the cache keys deterministic, or do they include dynamic components like timestamps, random numbers, or user-agent strings? A common bug is a cache key like `feed:{user_id}:{timestamp}` which guarantees every request generates a unique key, producing a 0% hit ratio. Check the cache key generation code.

2. **Cache vs origin data format:** Is the cache serializing and deserializing data correctly? If deserialization fails silently, the application may treat every cache hit as a miss and re-query the database. Check the logs for serialization errors.

3. **Cache connectivity:** Is the cache actually reachable? If the application cannot connect to the cache, every lookup fails and falls through to the database. But the cache CPU is at 5% — it IS receiving requests. This suggests the cache is reachable but returning results that the application does not recognize as valid.

The most likely root cause (given the symptoms): the cache key includes a dynamic component. The fix is to remove the dynamic component and use a deterministic key.

## Tier 2 — Design

These challenges require designing a caching system for a specific problem. Each should take 20-30 minutes.

### Challenge 2.1: Design a Cache for a URL Shortener

A URL shortener service handles 100 million redirects per day and 10,000 new shortened URLs per day. Each redirect requires looking up the original URL from the short code. The lookup currently takes 10ms (database index scan). The team wants to reduce latency to under 2ms.

Design a caching solution. Specify cache key format, TTL, memory requirements, eviction policy, and how you handle a new URL being created for a short code that already has a cached entry.

**Requirements:**
- 100 million reads/day = approximately 1,157 reads/second average, 3,500 peak
- 10,000 writes/day = approximately 0.12 writes/second
- 100 million unique short codes in the database (total dataset)
- Each cache entry: short code (7 bytes) + original URL (200 bytes average) + metadata = approximately 250 bytes
- Target: under 2ms read latency

**Rubric:**

| Aspect | Poor | Good | Excellent |
|--------|------|------|-----------|
| Cache key | Not specified | `url:{short_code}` | `url:{short_code}` with version prefix for schema migration |
| TTL | Not specified | 24 hours (implicit) | 24 hours with explicit invalidation on write |
| Memory calculation | Not calculated | "250MB should be enough" | "100M entries x 250 bytes x 1.3 overhead = 32.5GB. With 20% active working set = 6.5GB. Add replica = 13GB." |
| Cache technology | Not specified | Redis | Redis with primary-replica for HA |
| Invalidation | Not addressed | TTL only | "On write, DELETE the cache entry. The next read fetches the fresh URL from database." |

**Model answer:**

Cache key: `short:{short_code}` (e.g., `short:abc123`). Deterministic, fixed length, easy to hash.

TTL: 24 hours. Shortened URLs, once created, almost never change. A 24-hour TTL provides near-100% hit ratio. Combined with explicit invalidation on write, the staleness window is zero for the extremely rare case of an overwritten URL.

Memory: The total dataset is 100 million entries at 250 bytes each = 25GB raw. With jemalloc overhead (30%), that is 32.5GB for the full dataset. But at 1,157 reads/second, only a fraction of entries are accessed within 24 hours — likely under 20% (approximately 20 million entries). The active working set is 20 million x 250 bytes x 1.3 = 6.5GB. Adding one replica for fault tolerance brings the total to 13GB. A Redis cluster with 2 shards at 8GB each provides headroom for growth.

Eviction policy: `allkeys-lru`. If the cache exceeds memory, evict the least recently used keys. Since most keys are never accessed again after the initial redirect, LRU naturally retains the popular entries.

Invalidation: When a new URL is created for an existing short code, the write path issues a `DEL short:abc123` command after the database write. This is almost never needed (the overwrite rate is under 0.01%), but implementing it is trivial and eliminates the only source of staleness.

Result: Read latency drops from 10ms to under 2ms for cache hits (99.9%+ of reads). Database load drops from 1,157 QPS to effectively 0 — only the negligible overwrite misses reach the database.

### Challenge 2.2: Cache for Search Autocomplete

Design a caching system for a search autocomplete feature. Users type a query, and the system shows the top 10 matching suggestions. The suggestions are pre-computed from search logs and refreshed every hour. The system handles 500,000 searches per day, and each keystroke in the search box triggers an autocomplete request (average 3 requests per search, so 1.5 million autocomplete requests per day).

**Requirements:**
- 1.5 million requests/day = 17 requests/second average, 100 peak
- 1 million unique prefix queries in the suggestion database
- Each entry: prefix (50 bytes) + top 10 suggestions (500 bytes JSON) = 550 bytes
- Pre-computed suggestions are refreshed every hour via batch job
- Target: under 50ms response time (user typing expects fast feedback)

**Rubric:**

| Aspect | Poor | Good | Excellent |
|--------|------|------|-----------|
| Cache type | Redis only | Redis + local cache | Multi-layer: local LRU cache + Redis cluster + CDN |
| Key design | `prefix:{query}` | `autocomplete:{query}` | `ac:v2:{query}` with versioning |
| TTL | Not specified | 1 hour (matches refresh cycle) | 55 minutes (slightly under refresh cycle to avoid serving stale data during batch window) |
| Memory | Not calculated | Calculated working set | Calculated + pre-warm from batch job + compression (50% savings) |
| Cold start | Not addressed | Accept degraded | Pre-warming from batch output |

**Model answer:**

Multi-layer caching: Suggestions are the same for all users, making them perfect for CDN caching at the edge. The CDN handles geographically distributed users with under 20ms latency. Behind the CDN, Redis acts as the origin cache. Behind Redis, a local in-process cache in each application server provides sub-millisecond lookups for the most popular prefixes.

Cache key: `ac:v2:{normalized_prefix}`. The `v2` prefix enables cache invalidation on schema change — when the suggestion format changes, change the version and all old entries are automatically ignored.

TTL: 55 minutes. The suggestions are refreshed every hour. Setting TTL to 55 minutes gives the batch job a 5-minute window to complete and populate the cache before the old entries expire.

Memory: 1 million prefixes at 550 bytes each = 550MB raw. With compression (the suggestions JSON is highly repetitive — "restaurants near me" and "restaurants in NY" share many suggestion tokens), LZ4 compression achieves approximately 50% reduction. The effective memory is 275MB. Adding 30% jemalloc overhead: 360MB. A single Redis instance with 1GB memory is more than sufficient. No clustering needed.

Pre-warming: The batch job that computes suggestions writes the output directly to Redis at the end of each run. This ensures the cache is warm before the old entries expire. There is no cold start — the cache population is part of the batch pipeline.

### Challenge 2.3: Cache for a Social Media Feed

Design a caching system for a social media feed similar to Facebook's News Feed or Instagram's Explore page. The feed shows ranked posts from followed users, with posts scored by recency, engagement (likes, comments, shares), and relationship strength.

**Requirements:**
- 500 million daily active users
- Each user's feed is personalized (different ranking per user)
- Feed must load in under 500ms p99
- Posts are scored and ranked by a machine learning model that runs every 5 minutes
- Each user follows an average of 400 accounts
- The top 100 posts are shown per page load

**Rubric:**

| Aspect | Poor | Good | Excellent |
|--------|------|------|-----------|
| Key insight | "Cache the feed" | "Cache pre-computed ranked feed" | "Two-level cache: pre-computed ranked feed + real-time insertion of new posts" |
| Write strategy | Invalidate entire feed on any new post | Invalidate only when follower follows a new account | Pre-compute ranked feed every 5 minutes, insert new posts from recently followed accounts in real-time using a merge-sort on read |
| Memory calculation | Not calculated | Calculated for 500M users | "5M active users x 100 posts x 200 bytes = 100GB. Working set of top 10% active users = 10GB with one replica = 20GB." |
| Cache key | Not specified | `feed:{user_id}` | `feed:v2:{user_id}:page:{page_num}` with versioning |
| Cold start | Not addressed | "Pre-warm from batch job" | "Stagger batch job completion across 30-minute window. Each user's feed is computed in the batch and written to cache immediately. The cache is always warm because the batch completes before the old entries expire." |

### Challenge 2.4: Cache for an E-Commerce Product Page

Design a caching system for an e-commerce product page. The page shows product details, price, inventory status, reviews, ratings, and related products. Each product page gets 10,000 views per day on average, with the top 100 products getting 100,000 views per day each. There are 10 million products in the catalog.

**Requirements:**
- Product details change hourly (price updates, inventory changes)
- Reviews change throughout the day
- Related products change daily
- Inventory status must be accurate within 30 seconds
- A stale price would cause customer complaints and lost revenue
- 100 million page views per day

**Rubric:**

| Aspect | Poor | Good | Excellent |
|--------|------|------|-----------|
| Data segmentation | One cache for everything | Separate caches per data type | CDN for product HTML, Redis for JSON API data, separate TTL per component |
| Inventory TTL | Same as product details | 30 seconds | 30 seconds with invalidation on inventory change events |
| Price freshness | TTL only | TTL + invalidation | "Price is cached with 5-minute TTL but invalidated immediately via a message queue when price changes in the database" |
| Partial invalidation | Flush all | Delete individual keys | "Use edge-side includes (ESI) to cache page components independently. Invalidate only the price component when price changes." |

**Model answer:**

This is a heterogeneous caching problem — different components of the page have different freshness requirements. A unified cache with a single TTL would either serve stale prices or unnecessarily re-fetch reviews every 30 seconds.

The solution is component-level caching with per-component TTL:

- **Product details (name, description, images):** 1-hour TTL. Changes hourly. Cache key: `pd:{product_id}`.
- **Price:** 5-minute TTL with immediate invalidation on price change. Cache key: `price:{product_id}`. The product management system publishes a `price.updated` event to a Kafka topic when prices change. A cache invalidation consumer listens and issues `DEL price:{product_id}` within 100ms.
- **Inventory status:** 30-second TTL. Cache key: `inv:{product_id}`. No invalidation — 30-second staleness is acceptable for "in stock" indicators and avoids the complexity of real-time inventory invalidation.
- **Reviews:** 15-minute TTL. Reviews change infrequently and staleness is invisible. Cache key: `rev:{product_id}:page:{page_num}`.
- **Related products:** 1-hour TTL. Computed nightly. Cache key: `related:{product_id}`.

Memory: 100 million page views/day. Top 100 products account for 10 million views. Working set: the top 10,000 products account for 80% of views. Total cache memory: approximately 15GB across all data types. A Redis cluster with 3 shards at 8GB each (24GB total) provides headroom.

## Tier 3 — Expert

These challenges require making judgment calls with incomplete information. There is no single correct answer — the evaluation is based on the quality of your reasoning. Each should take 30-45 minutes.

### Challenge 3.1: Cache or Not to Cache

Your team is building a real-time ride-sharing platform similar to Uber. The core data includes:

- Driver location updates (every 3 seconds per driver, 100,000 drivers active)
- Rider ride requests (submitted, matched, in-progress, completed)
- Pricing data (surge multipliers change every 2-5 minutes)
- Driver ratings (updated after each ride, read when matching)
- Rider payment methods (updated rarely, read on ride completion)

Propose a caching strategy for each data type. For each, state whether caching is appropriate, and if so, specify the cache type, TTL, and invalidation mechanism. If caching is inappropriate, explain why.

**Evaluation dimensions:**

- Does your analysis correctly identify which data types benefit from caching? (Driver location updates every 3 seconds have a near-0% hit ratio — caching them is pointless. Driver ratings change after each ride and are read during matching — caching with a short TTL makes sense.)
- Do you consider the consistency requirements? (Pricing must be accurate to prevent financial disputes — a stale surge multiplier overcharges or undercharges riders. This requires write-through or extremely short TTL with invalidation.)
- Do you account for the matching algorithm's data dependencies? (The matching algorithm reads driver locations and ratings for potentially hundreds of nearby drivers per request — caching aggregated per-driver data at the application layer reduces database load.)
- Do you propose appropriate technologies? (Driver location updates are a streaming problem, not a caching problem. Redis streams or Kafka are more appropriate. Rider ratings fit Redis sorted sets. Pricing fits cache-aside with short TTL.)

**Discussion:**

Driver location data is the key insight question. A naive designer proposes caching driver locations with a 3-second TTL. But driver locations change every 3 seconds, so the TTL expires at the same rate as the updates, producing a hit ratio near 0%. The cache check becomes pure overhead. Driver locations should be stored in a streaming data store (Redis Streams, Apache Kafka, or a dedicated geospatial database) optimized for fast writes and range queries — not a cache optimized for fast reads of static data.

Rider payment methods are the opposite: they change rarely (when a rider adds or removes a card), are read on every ride completion (millions per day), and must be accurate to avoid charging errors. Cache them with a 1-hour TTL and explicit invalidation on payment method changes. The invalidation event should block the ride completion until confirmed — stale payment info causes failed charges.

### Challenge 3.2: Multi-Region Cache for a Global Product Catalog

A global e-commerce company has data centers in US-East, EU-West, and AP-Southeast. Each region has its own application servers and a shared PostgreSQL database in US-East. The product catalog has 500 million products and is read-heavy (95% reads, 5% writes). Writes are primarily price and inventory updates from suppliers worldwide.

Product catalog reads must be fast in every region — under 50ms p99. Currently, AP-Southeast users experience 300ms latency because queries must cross the Pacific Ocean to the US-East database.

Design a multi-region caching solution. Address:

1. Cache topology (where does data live, how is it distributed?)
2. Write propagation (how does a price update in EU-West affect the cache in AP-Southeast?)
3. Consistency model (what staleness is acceptable, and how do you bound it?)
4. Failure scenarios (a region's cache cluster loses connectivity to the primary database)
5. Capacity planning (memory, bandwidth, node count for each region)

**Discussion:**

This is a global-scale caching problem. The strongest approach involves three layers:

**Layer 1 — CDN:** The product catalog is a natural fit for a CDN. Product pages are the same for all users (unlike personalized pages). A CDN at 50+ edge locations serves cached product HTML and API responses with under 20ms latency worldwide. The CDN origin is the regional cache.

**Layer 2 — Regional Redis cluster:** Each region has its own Redis cluster storing frequently accessed product data. The cluster caches product details, prices, inventory, and images. Read requests hit the regional cluster first, then the CDN (as origin fetch), then the database.

**Layer 3 — Global database:** The PostgreSQL primary in US-East is the source of truth. Read replicas in EU-West and AP-Southeast reduce cross-region query latency for cache misses.

Consistency model: Strong consistency for inventory (must not oversell), eventual consistency for product descriptions and images (minutes of staleness is acceptable). Price has bounded staleness: the cache TTL for prices is 2 minutes, and price updates are propagated via a global invalidation queue (Kafka MirrorMaker or a dedicated service).

Write propagation: A price update arrives at the EU-West application server. The server writes to the global database primary in US-East. The invalidation service in US-East picks up the change (via database trigger or change data capture) and publishes invalidations for the affected cache keys to a global queue. Regional invalidation consumers in EU-West, US-East, and AP-Southeast receive the event within 1-5 seconds (cross-region latency) and issue `DEL` commands to their respective Redis clusters. The maximum staleness window is 2 minutes (TTL) + 5 seconds (propagation lag) = 125 seconds. This is acceptable for prices, but not for inventory.

For inventory, the solution is different: the "add to cart" flow bypasses the cache entirely and reads inventory directly from the regional database read replica, which replicates from the US-East primary with under 100ms lag. This ensures no overselling.

Failure scenario: If AP-Southeast loses connectivity to the US-East database, the regional cache continues serving cached data. Writes are queued locally and replayed when connectivity is restored. The CDN continues serving cached product pages independently — it has its own TTL and origin fallback. The region is degraded (no price updates, no inventory writes) but remains available for reads.

### Challenge 3.3: Cache for a Real-Time Messaging System

Design a caching strategy for a real-time messaging platform similar to WhatsApp or Signal. The system handles:

- 1 billion messages sent per day
- 500 million active users
- Each message must be delivered with under 100ms end-to-end latency
- Messages are stored in the database for history but must be available instantly for the most recent conversations
- Users access their last 100 conversations on app open (the inbox)
- Each conversation loads the last 50 messages
- Media messages (images, videos) are stored separately in blob storage

Decide what to cache, what not to cache, and at which layer (client, edge, application, database). Address the consistency requirements: a user who sends a message must see it immediately. A user who receives a message must see it within 100ms. Account for the fact that messages, once sent, are immutable (they cannot be edited, only deleted).

**Discussion:**

**What to cache — the inbox (last 100 conversations per user):** This is the highest-value cache target. When a user opens the messaging app, the inbox must load instantly. The inbox is pre-computed and cached in Redis with key `inbox:{user_id}`. Each entry in the cached inbox is a lightweight conversation summary: conversation ID, last message preview, timestamp, unread count. The TTL is 5 minutes, but the inbox is immediately invalidated when a new message arrives for that user (the message event triggers a cache delete for the recipient's inbox). This ensures the sender-immediacy requirement: when a user sends a message, the recipient's inbox is invalidated and the next read fetches the updated state.

**What to cache — recent messages per conversation:** The last 50 messages of the most recent conversations are cached in Redis with key `msgs:{conversation_id}:before:{timestamp}`. Messages, once sent, are immutable — they never change. This means the cache never has a staleness problem for individual messages. The only invalidation needed is for new messages, which are appended to the cached message list.

**What NOT to cache — message content (media):** Images and videos are stored in blob storage (S3, GCS) and served directly to the client via signed URLs. The CDN caches media assets at the edge for fast delivery. The application caches only the metadata (URL, thumbnail, dimensions) and not the binary content.

**Client-side cache:** The mobile app maintains an in-memory cache of the last 20 conversations and the last 50 messages per conversation. When the app is backgrounded and foregrounded, it fetches a delta update — only conversations that have changed since the last check. This eliminates the need to fetch the entire inbox on every app open.

**Consistency strategy:** This system is a good fit for eventual consistency with read-your-writes for the sender. The sender's message must appear immediately in their own conversation view. This is achieved by optimistic local rendering: the app adds the sent message to the local conversation view immediately (before the server confirms receipt) and updates it when the server confirms. For the recipient, the 100-millisecond delivery requirement means the cache invalidation must be fast — within 50ms of the message being stored, the recipient's inbox cache and conversation message cache must be invalidated. This is achievable with an in-memory event bus within the same datacenter.

**What is NOT cached:** Message delivery status (read receipts, typing indicators) is not cached — it is ephemeral state that is pushed via WebSocket to active recipients and ignored after delivery. User authentication tokens are not cached in the shared Redis — they are validated against the authentication service on every connection. End-to-end encryption keys are not cached outside the client device.&#x60;
