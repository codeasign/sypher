Every engineering decision involves trade-offs. Caching is no exception. The core tension is simple to state but hard to navigate: caching trades memory for speed, and freshness for latency reduction. Every cache design decision sits somewhere on these two axes. Understanding where your system falls on each axis determines whether a caching strategy succeeds or introduces catastrophic failure modes.

## The Core Tension

Memory versus speed: a cache stores copies of data in fast memory to avoid fetching from slow storage. The more memory you allocate to the cache, the more data you can store, the higher the hit ratio, and the faster the average response time. But memory is finite and expensive. At some point, adding more cache memory yields diminishing returns — the long tail of infrequently accessed data does not benefit from caching.

Freshness versus latency: a cache returns data faster by returning data that may be stale. The longer you let data live in the cache (higher TTL), the higher the hit ratio and the faster the responses — but the greater the chance a user sees outdated information. A shorter TTL keeps data fresher but reduces the hit ratio, increasing load on the origin.

These two tensions interact. A cache with very low memory will have a low hit ratio because it evicts entries aggressively. A cache with very long TTL will have a high hit ratio but serve stale data. The optimal configuration sits at the intersection of acceptable freshness and acceptable memory cost.

## Decision Matrix

Every caching decision involves choosing between options that optimize for different outcomes. This matrix helps you navigate the most common binary choices.

| Decision | Option A | When to Choose | Option B | When to Choose |
|----------|----------|---------------|----------|---------------|
| Location | Local cache | Single process, microsecond latency needed, inconsistent data acceptable | Distributed cache | Multiple processes, shared data, consistency needed |
| Expiration | TTL-based | Data changes on a predictable schedule, staleness is acceptable | Explicit invalidation | Data changes unpredictably, staleness is not acceptable |
| Write strategy | Write-through | Writes are infrequent, cache must always be fresh | Write-back | Writes are frequent, write latency matters more than read freshness |
| Population | Lazy (on miss) | Simple implementation, data is requested by users anyway | Eager (pre-warm) | Cold start is unacceptable, data is known in advance |
| Scope | Client cache | Reduce bandwidth, offline support, public content | Server cache | Reduce origin load, secure data, controlled invalidation |
| Eviction | LRU | General purpose, recency predicts future access | LFU | Content delivery, frequency predicts future access |
| Storage | In-memory | Sub-millisecond latency needed, data fits in RAM | SSD-backed | Data exceeds RAM budget, slightly higher latency is acceptable |

No single combination is universally correct. The right choice depends on your access patterns, consistency requirements, and operational constraints.

During a network partition (when the cache cannot reach the database to verify freshness), an AP cache continues serving requests with whatever data it has — even if that data is stale. This is the correct behavior for a cache. Blocking during a partition would defeat the purpose of having a cache, which is to provide low-latency reads.

If your application requires strong consistency — every read must return the most recent write — you should not use a cache in the read path, or you must use a cache that supports leases or read-your-writes consistency with serialized access to the origin. These consistency mechanisms add latency and reduce throughput, but guarantee correctness.

Reddit's experience illustrates this trade-off. Reddit uses a cache for the front page. During a database outage, the cache continued serving a slightly stale version of the front page. Users saw posts that were 30 seconds old instead of a 500 error. The company considered this a win — stale beats broken for a content browsing experience.

## Failure Modes

A cache that is badly configured, poorly monitored, or mismatched to its workload can cause failures more severe than having no cache at all. These are the most common failure modes.

### Failure Mode 1: Cache Stampede

A cache stampede occurs when a popular cache key expires and a large number of concurrent requests arrive for that key before any single request can repopulate it. Every request sees a cache miss and queries the database simultaneously.

**Detection:** A sudden spike in database query latency and connection count, coinciding with a drop in cache hit ratio for one or more keys. The `X-Cache: MISS` ratio jumps from under 10% to 100% for affected keys.

**Recovery:** Rate-limit database queries per cache key. Only allow one request per key to query the database; other requests either wait or receive a stale-but-acceptable value. Most cache client libraries support "lock around the miss" or "probabilistic early expiration" patterns.

**Prevention:** Use probabilistic early recomputation (refresh the cache entry before it expires). Stagger TTLs for batch-loaded entries (add a random jitter of +-10% to each key's TTL so they do not expire simultaneously). Use a hot-key backup: replicate popular keys across multiple cache nodes so demand is distributed.

### Failure Mode 2: Thundering Herd

The thundering herd problem is a specific type of cache stampede that occurs during system startup or after a cache cluster restart. Every request hits a completely cold cache. The database receives the full read load.

**Detection:** Immediately after a deployment or cache restart, database CPU spikes to 100%, request latency spikes, and error rates increase.

**Recovery:** Pre-warm the cache by replaying recent traffic or running a data-loading script before directing user traffic to the system. Implement connection throttling on the database to prevent overload.

**Prevention:** Use a cache with persistence (Redis RDB/AOF) so the cache survives restarts. Deploy cache changes with a rolling update rather than a full cluster restart. Introduce new cache nodes gradually using consistent hashing.

A cold start happens when a cache has no data — after a fresh deployment, a cluster restart, or scaling up with new nodes. The hit ratio starts at 0% and climbs gradually as requests populate the cache. The database must handle 100% of read traffic during the warm-up period.

The danger is that the warm-up period coincides with peak traffic. Deploying a cache restart at 2 PM when traffic is highest is asking for a database overload and cascading failure.

**Detection:** Hit ratio graph shows a steady upward slope from near 0% over 10-30 minutes.

**Recovery:** No recovery — you must wait for the cache to warm up. The only mitigation is to ensure the database has enough headroom to absorb the full read load during the warm-up period.

**Prevention:** Pre-warm the cache. Before directing traffic to the new cache, run a script that requests the most popular keys (the top 1% of keys that account for 80% of requests). This warms the cache in minutes instead of tens of minutes. Redis supports key scanning and data migration tools for this purpose.

### Failure Mode 4: Cache Poisoning

Cache poisoning occurs when malicious or invalid data is written to the cache and served to subsequent users. This can happen through:

- An application bug that stores corrupted data in the cache.
- An attacker exploiting an unvalidated input that becomes a cache key or value.
- A deserialization vulnerability that executes code when the cached value is read.

**Detection:** Users report seeing incorrect data. Monitoring shows unusual cache values (size outliers, unexpected content types).

**Recovery:** Flush the affected cache keys. Fix the application bug that allowed the bad data to enter the cache. If the attack is in progress, consider temporarily disabling the cache for the affected endpoints.

**Prevention:** Validate all data before storing it in the cache. Never cache raw user input without sanitization. Use integrity checks (checksums or signatures) on cached values to detect corruption. Limit the maximum size of cache values to prevent memory exhaustion attacks.

### Failure Mode 5: Memory Exhaustion

When the cache's memory is exhausted and the eviction policy cannot keep up — because the write rate exceeds the eviction rate, or because keys have no TTL and the eviction policy is `noeviction` — the cache returns errors on write operations.

**Detection:** Cache SET operations return OOM errors. Hit ratio drops as eviction rates spike.

**Recovery:** Increase `maxmemory`. Reduce TTLs. Implement a more aggressive eviction policy. Consider vertical scaling (larger instance) or horizontal scaling (more shards).

**Prevention:** Set `maxmemory-policy` to an eviction policy other than `noeviction`. Monitor memory usage and set an alert at 80% of `maxmemory`. Use `maxmemory-samples` (Redis) to control the accuracy of the LRU approximation — higher samples mean better eviction decisions but higher CPU cost.

## Anti-Patterns

These patterns appear frequently in production systems and reliably cause problems.

### Anti-Pattern 1: Caching Everything

The instinct to cache everything "for performance" leads to wasted memory and complexity. Not everything benefits from caching. Small lookup tables that fit in the application's working set (country codes, status enums) are already fast enough without a cache. Data that changes on every request (real-time stock prices, game scores) has a 0% hit ratio and only adds overhead.

**The fix:** Cache only data that is expensive to compute and requested more than once per TTL window. Profile your application to identify the top 5 most expensive queries and cache those first.

### Anti-Pattern 2: Infinite TTL

Setting a very long TTL (days, weeks) or no TTL at all creates a cache that never refreshes. When the data changes, users see stale content indefinitely. The cache becomes a liability rather than an optimization.

**The fix:** Always set a TTL. Even for data that "never changes," set a TTL of at least 24 hours. Implement explicit invalidation for time-sensitive updates. If you must cache data without a TTL, implement a background refresh mechanism that periodically re-fetches from the origin.

### Anti-Pattern 3: Cache-Aside Without Fallback

If the cache read returns an error, the application must fall through to the database. Some implementations treat a cache error as a failure and return an error to the client. This is wrong — the cache is not the source of truth.

**The fix:** Always implement a fallback. If the cache is unreachable or returns an error, read from the database. Log the error and alert, but serve the response. This is the fail-open pattern described in the deep dive.

### Anti-Pattern 4: Oversized Values

Storing multi-megabyte values in a cache consumes memory disproportionately to the value's access frequency. A single 10MB value stored in a 1GB cache can evict hundreds of smaller, more frequently accessed entries.

**The fix:** Set a maximum value size. In Redis, configure `proto-max-bulk-len` to limit incoming value sizes. In the application, refuse to cache values above a threshold (typically 100KB-1MB). Store large blobs separately (in S3 or a blob store) and cache only the reference.

### Anti-Pattern 5: No Monitoring

A cache deployed without hit ratio, latency, and eviction monitoring is a blind spot. You cannot know whether the cache is helping, hurting, or silently failing. Many teams discover their cache has a 20% hit ratio only after a database outage forces them to investigate.

**The fix:** Instrument every cache operation. Log hit ratio, latency (p50/p99/p999), eviction rate, and error count. Set up dashboards and alerts. Redis provides the `INFO` command with most of these metrics; Memcached provides similar stats via the `stats` command.

### Anti-Pattern 6: Cache as a Database

Using cache as the primary data store for data that must survive crashes. A cache cluster restart loses all data. If the data is important, store it in a database and cache it for performance. The only exception is explicitly ephemeral data — session tokens, rate limit counters, real-time leaderboards — where data loss is acceptable.

**The fix:** For every piece of data in the cache, ask: "If this data disappeared right now, would the system still function?" If the answer is no, the data should not be solely in the cache. Add a database or persistent store as the source of truth.

### Anti-Pattern 7: Cache-aside Without Locking

Implementing cache-aside by checking the cache, falling through to the database on miss, and storing the result — without any protection against concurrent misses. Under high concurrency, N simultaneous requests for the same missing key all query the database. This is the thundering herd problem described earlier. It wastes database resources and may cause a stampede.

**The fix:** Implement lock-based deduplication around the miss path. Only one request queries the database; the others wait for the cache to be populated. Most cache client libraries provide this feature, or you can implement it with Redis `SETNX` (SET if Not Exists) as a distributed lock.

### Anti-Pattern 8: No Cache Key TTL

Storing data in the cache without a TTL. Entries accumulate indefinitely. Memory fills up. Eviction eventually removes entries — but eviction removes the least recently used entries, which may be valuable. Without TTL, the cache memory is consumed by data that may never be accessed again.

**The fix:** Always set a TTL, even if it is long (24 hours). The TTL acts as a safety valve that eventually removes unused entries. If you have data that truly has no staleness concern, set a TTL of 24 hours and implement a background refresh job that re-fetches the data before it expires.

### Cache Consistency Models

Different applications require different levels of cache consistency. Choosing the right model determines how you handle writes, invalidations, and stale data.

**Eventual consistency:** The cache may serve stale data for a bounded period after a write. All caches with TTL are eventually consistent. This is the simplest model and appropriate for most read-heavy workloads. Users see slightly outdated data for at most TTL seconds. The Netflix recommendation service uses eventual consistency — a 5-minute stale recommendation is indistinguishable from a fresh one.

**Read-your-writes consistency:** After a user performs a write, their subsequent reads must reflect that write. Other users may see stale data, but the writer always sees their own writes. This is important for user-facing features like profile updates or status changes. Implementation: after a write, either invalidate the cache for that specific user or use a write-through cache for that user's data.

**Monotonic read consistency:** Once a user has seen a particular value, they must never see an older value. This is violated when requests are routed to different cache nodes with different data freshness levels. Implementation: use sticky sessions (route the user to the same cache node for the duration of their session) or use a single shared cache with global ordering.

**Strong consistency:** Every read returns the most recent write. This is impossible to guarantee with an asynchronous cache invalidation model. Strong consistency requires either: (1) serializing all writes through the cache (write-through with a single serialization point), (2) using distributed locks to coordinate cache updates, or (3) using a database that provides strong consistency and bypassing the cache for critical reads. Strong consistency is expensive and rarely justified for caching — if you need strong consistency, you may not need a cache at all for that data.

The consistency model you choose affects the eviction policy, the invalidation strategy, and the operational complexity. Start with eventual consistency and add stronger guarantees only for specific data types that need them.

### Cost Analysis: Is Caching Worth It?

Caching is not free. The operational costs include infrastructure, engineering time, and ongoing maintenance.

The cost of a cache cluster:

| Cost Category | Monthly Estimate |
|--------------|-----------------|
| Compute (3-node Redis cluster, r6g.large) | $150-200 |
| Cross-AZ data transfer | $50-100 |
| Engineering time (build, integrate, maintain) | $5,000-15,000 |
| Monitoring and alerting | $20-50 |
| Backup storage (RDB snapshots) | $5-15 |
| **Total** | **$225-15,365** |

The engineering time dominates. This is not a reason to skip caching — it is a reason to be deliberate about where caching provides the most value for the engineering cost.

A simple cost-benefit framework:

1. Measure the current p95 read latency and database CPU utilization.
2. Estimate the latency improvement from caching (typically 10-50x for cached reads).
3. Estimate the database cost reduction (fewer read replicas, smaller instance types).
4. Calculate the net infrastructure savings: database savings minus cache infrastructure cost.
5. Compare against the engineering investment. If the net is positive over 12 months, caching is worth it.

For most applications, caching is worth it when: (a) the database is the bottleneck and scaling it up is expensive, (b) the read-to-write ratio exceeds 10:1, and (c) the working set fits in a reasonably sized cache cluster (under 100GB). For applications that do not meet these criteria, optimizing the database queries or application code is a better first step.

## When NOT to Use

Caching is not a universal performance solution. These scenarios indicate that caching may cause more harm than good.

**Real-time data:** Stock prices, sports scores, multiplayer game state, live chat. Data that changes at sub-second granularity cannot benefit from caching because the TTL would need to be under one second, providing negligible hit ratio.

**Small datasets that already fit in memory:** If your entire database is 1GB and the application server has 32GB of RAM, you do not need a separate cache. The database itself caches data in its buffer pool. A separate cache adds complexity without benefit.

**Write-heavy workloads:** If your system processes 90% writes and 10% reads, caching provides minimal benefit. Most writes to the cache are immediately invalidated by the next write. The overhead of maintaining the cache outweighs the read savings.

**Strong consistency requirements:** If your application requires linearizability — every read returns the most recent write — a cache adds uncertainty. You would need strict invalidation on every write, which turns every write into a synchronous cross-system operation, defeating the purpose of caching.

**Ephemeral or short-lived data:** Data that appears and disappears within seconds (real-time event streams, monitoring metrics) does not benefit from caching. By the time the cache is populated, the data is already irrelevant.

**DoorDash learned this lesson the hard way.** The company implemented caching for its delivery tracking system, but the tracking data changed every few seconds. The cache hit ratio was under 10% because entries expired before they could be read from cache. The cache added latency (cache check + miss + database query) without any benefit. DoorDash removed the cache and optimized the database queries instead, improving p99 latency by 40%.

### Caching and Data Freshness SLAs

Every caching design should start with a data freshness SLA — a clear statement of how stale data may be, measured in seconds or minutes. This SLA determines the TTL, the invalidation strategy, and the acceptable failure modes.

| Data Type | Typical Freshness SLA | TTL Strategy | Invalidation |
|-----------|----------------------|--------------|--------------|
| User session | Session lifetime (15-30 min) | TTL = session timeout | Expire on logout |
| User profile | 5 minutes | TTL = 300s | Delete on profile update |
| Product price | 1 minute | TTL = 60s | CDN purge + cache delete on price change |
| Inventory count | 30 seconds | TTL = 30s | No invalidation (accept staleness) |
| Real-time score | 0 seconds | No caching | N/A — cache not appropriate |
| Search index | 1 hour | TTL = 3600s | Reindex triggers cache flush |
| News article | 15 minutes | TTL = 900s | Purge on article update |
| Weather data | 30 minutes | TTL = 1800s | No invalidation (API refresh) |

The SLA drives the architecture. If your SLA is 5 minutes of staleness, a TTL of 300 seconds with cache-aside is sufficient. If your SLA is 100 milliseconds of staleness, you need write-through caching or no caching at all, and you need to consider whether the complexity is justified.

When you cannot meet the SLA with caching, you have three options: (1) Accept the SLA violation and reduce the TTL until the staleness window is acceptable. (2) Implement aggressive invalidation with sub-millisecond propagation. (3) Do not cache the data — optimize the database or API instead.
