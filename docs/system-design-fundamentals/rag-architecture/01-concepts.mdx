## The Problem Story

Imagine a medical chatbot deployed by a large health system. A patient asks: "What
is the recommended dosage of metformin for a 70 kg adult with stage 3 kidney
disease?" The chatbot, powered by a general-purpose LLM, responds confidently with
a standard dosage. The LLM has never seen this patient's chart. It has no access
to the hospital's formulary. It was trained on data that cuts off 18 months ago.
The dosage it recommends is wrong for a patient with impaired kidney function.

The patient follows the advice and suffers complications.

This is not a hypothetical. Every major LLM provider has documented cases of
confident hallucination in high-stakes domains. The root cause is structural: a
pre-trained model knows only what it saw during training. It cannot see your
documents, your database, or your real-time data. When asked about information
outside its training set, it does not say "I don't know" — it generates the most
plausible-sounding completion, which may be entirely wrong.

The problem is not that LLMs are unreliable. The problem is that we were asking
them to answer questions without giving them the source material.

## The Insight

Retrieval-Augmented Generation (RAG) solves this by adding a simple but powerful
step: before the LLM generates an answer, you retrieve relevant documents from
your own knowledge base and insert them into the prompt as context. The LLM then
generates its answer grounded in those documents.

Think of a library. A traditional LLM is like a librarian who has memorized every
book but has not read anything published after their training cutoff. A RAG system
is the same librarian, but now you hand them the specific bookshelf relevant to
the question before they answer. The librarian still provides the synthesis and
explanation, but the source material is verified and current.

The analogy works for a chef too. A chef who cooks from memory may serve a dish
with stale ingredients. A RAG chef checks the pantry first, retrieves the freshest
ingredients, and then cooks. The cooking skill (generation) is the same — the
quality of ingredients (retrieved context) is what changes.

## Core Vocabulary

**Retrieval** — The process of finding relevant documents from a knowledge base
given a query. This is the "R" in RAG. Implemented via vector similarity search,
keyword matching, or a hybrid of both.

**Generation** — The process of producing a natural language answer from the LLM
given the retrieved context and the original query. This is the "G" in RAG.

**Augmentation** — The step that combines the retrieved documents with the
original query into a structured prompt. This is the "A" in RAG. The augmentation
step defines how context is formatted, truncated, and prioritized before being
passed to the LLM.

**Chunking** — The strategy for splitting documents into smaller, retrievable
pieces. A document about a medical trial might be chunked by section
(abstract, methodology, results, conclusion). Chunk size and overlap
directly affect retrieval quality.

**Embedding** — A dense vector representation of text. A sentence like "What is
the dosage of metformin?" is converted into a list of hundreds of floating-point
numbers that capture its semantic meaning. Similar sentences produce similar
vectors.

**Vector Database** — A database optimized for storing and querying embedding
vectors. Examples include Pinecone, Weaviate, Qdrant, and Milvus. These databases
support approximate nearest neighbor (ANN) search to find the most similar vectors
to a query vector in milliseconds.

**Query Encoding** — The process of converting a user's question into an embedding
vector using the same embedding model used to index the documents. If the query
encoder and document encoder are not aligned, retrieval quality degrades.

**Document Indexing** — The pipeline that ingests raw documents, cleans them,
chunks them, generates embeddings, and stores the vectors with metadata in the
vector database. This runs offline, before any queries arrive.

**Relevance Scoring** — The mechanism for ranking retrieved documents by how
useful they are for answering the query. Common scores include cosine similarity,
dot product, or learned relevance models like Cohere's rerank.

**Context Window** — The maximum number of tokens the LLM can process in a single
request. Retrieved documents must fit within this window. Exceeding it requires
truncation, summarization, or sliding window strategies.

## How It Works Step by Step

### Step 1: Document Ingestion

Raw documents arrive from various sources — PDFs, websites, databases, internal
wikis. Each document is cleaned (HTML tags removed, formatting normalized) and
stored in a staging area.

Each chunk is passed through an embedding model (e.g., text-embedding-3-small,
BGE, E5) to produce a dense vector. The vector captures the semantic meaning of
the chunk. Similar chunks produce vectors that are close together in the embedding
space.

The query vector is sent to the vector database, which performs an ANN search to

## Variants and Flavors

### Naive RAG

The simplest variant. Documents are indexed once. Every query performs the same
retrieve-augment-generate cycle. No caching, no re-ranking, no query rewriting.

<AsciiDiagram
  id="rag-architecture/concepts-naive-rag"
  title="Naive RAG performs a single retrieve-augment-generate cycle"
  content={`
    ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
    │  Query   │────>│  Vector  │────>│  Augment │────>│  LLM     │
    │          │     │  Search  │     │  Context │     │  Generate│
    └──────────┘     └──────────┘     └──────────┘     └──────────┘
  `}
/>

**Characteristics:**
- Single-pass retrieval, no iteration
- Fixed chunk size, no dynamic adjustment
- No query decomposition for complex questions
- Low latency, but lower quality for ambiguous queries

### Advanced RAG (with Re-ranking)

Adds pre-retrieval and post-retrieval steps. Pre-retrieval includes query rewriting
and query expansion. Post-retrieval includes re-ranking and context filtering.

<AsciiDiagram
  id="rag-architecture/concepts-advanced-rag"
  title="Advanced RAG adds pre-retrieval and post-retrieval optimization"
  content={`
    ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
    │  Query   │────>│  Query   │────>│  Vector  │────>│  Re-rank │────>│  LLM     │
    │  Rewrite │     │  Expand  │     │  Search  │     │  Top-3   │     │  Generate│
    └──────────┘     └──────────┘     └──────────┘     └──────────┘     └──────────┘
  `}
/>

**Improvements:**
- Query rewriting handles ambiguous or poorly phrased questions
- Re-ranking catches ANN false positives
- Context filtering removes irrelevant chunks before generation
- 10-20% improvement in answer accuracy over naive RAG

### Modular RAG

Each component of the pipeline is a swappable module. Different retrieval
strategies, embedding models, and generation models can be composed differently
for different use cases.

## Mental Model Check

**Question 1:** What is the single most important difference between a standard
LLM and a RAG system?

**Answer:** A standard LLM generates answers from its training data only. A RAG
system retrieves relevant documents from an external knowledge base and inserts
them into the prompt, so the LLM generates answers grounded in those documents.
This means RAG can answer questions about information that was not in the training
data, and it can cite sources.

**Question 2:** Why does chunking strategy matter for retrieval quality?

**Answer:** Chunking determines the granularity of retrieval. Chunks that are too
large may contain irrelevant content that drowns out the relevant passage. Chunks
that are too small may lack sufficient context for the LLM to understand them.
The optimal chunk size depends on the document structure and the typical query
scope. Medical papers benefit from section-level chunks, while code documentation
benefits from function-level chunks.

**Question 3:** What happens if the query encoder and document encoder are
different models?

**Answer:** The query vector and document vectors will live in different embedding
spaces. Cosine similarity between them will be meaningless. The retriever will
return semantically irrelevant results. Always use the same embedding model for
both indexing and query encoding, or use a model specifically designed for
asymmetric retrieval (e.g., BGE for query-document pairs).

**Question 4:** Why is re-ranking sometimes necessary even with a good vector
database?

**Answer:** ANN search is an approximation. It may return false positives —
documents that are vector-close to the query but semantically irrelevant. A
re-ranker (typically a cross-encoder) evaluates each query-document pair more
thoroughly, catching these false positives. Re-ranking is expensive (O(n) per
query) but applied to a small set (top-20 or top-50), so the cost is manageable.

**Question 5:** What is the context window constraint in RAG, and how do you
handle it?

**Answer:** The LLM has a maximum number of tokens it can process. Retrieved
context must fit within this window, leaving room for the query and the system
prompt. If the retrieved documents exceed the window, you must truncate (keep
the highest-ranked chunks), summarize (use another LLM call to condense), or
use a sliding window approach (process chunks in batches and aggregate results).
