## Company Case Studies

### Notion AI: RAG for Workspace Q&A

Notion AI uses RAG to answer questions about a user's workspace documents, notes,
and databases. The system must handle a wide variety of document types (free text,
tables, kanban boards, code blocks) and respect per-user access controls.

**Architecture:**
Notion indexes each workspace's content into a per-tenant vector index. When a
user asks a question, the system retrieves relevant blocks from their workspace
and augments the prompt. The retrieval is scoped to documents the user has
permission to see.

**Key design decisions:**
- Per-tenant indexing rather than a shared index. This simplifies access control
  but increases storage costs (each tenant's documents are indexed separately).
- Block-level retrieval rather than page-level. A Notion page can contain dozens
  of blocks (paragraphs, headings, toggles), and retrieving at the block level
  provides more precise context.
- Caching is aggressive because workspace content changes slowly. A user's
  documents are re-indexed only when they edit them, not on a schedule.

**Challenges encountered:**
- Block-level indexing created millions of small vectors per tenant. The vector
  database had to handle high cardinality with many small indexes.
- Access control filtering at retrieval time required metadata tagging on every
  vector, increasing storage.
- Users expected real-time indexing within seconds of editing a page. Notion
  implemented an event-driven pipeline that triggers re-indexing on document
  change events.

### GitHub Copilot: Code Retrieval for Code Generation

GitHub Copilot uses a form of RAG to retrieve relevant code context from a user's
repository and generate code suggestions. The retrieval is implicit — Copilot
uses the current file and open tabs as context, along with semantically similar
code snippets from the repository.

**Architecture:**
Copilot Code Review and Copilot Chat use explicit retrieval. When a developer
asks a question about a codebase, Copilot retrieves relevant code files, function
definitions, and documentation, then generates an answer grounded in the retrieved
code.

**Key design decisions:**
- Code is chunked at the function or class level, not at the token level. This
  preserves syntactic structure and makes the code understandable to the LLM.
- The embedding model is fine-tuned on code — OpenAI's text-embedding-3 models
  are further trained on code repositories to understand code semantics.
- Retrieval is scoped to the user's repository by default, with optional
  cross-repository search for enterprise customers.

**Challenges encountered:**
- Code retrieval is fundamentally different from text retrieval. A function that
  is semantically similar may use different variable names or have a different
  structure. The embedding model must understand code semantics, not just text
  similarity.
- Large repositories (millions of lines) require efficient indexing. GitHub
  indexes only the default branch and uses a priority queue for the most
  frequently accessed files.
- Security is critical. Copilot must not retrieve code from repositories the user
  does not have access to, even if the code is semantically similar.

### Perplexity AI: Real-Time Web + RAG

Perplexity AI combines real-time web search with RAG. When a user asks a question,
Perplexity searches the web, retrieves the most relevant pages, and generates an
answer with citations to the source URLs.

**Architecture:**
Perplexity uses a multi-stage pipeline: web search (Bing API or self-hosted index)
to find relevant URLs, page fetching and parsing to extract content, chunking and
re-ranking to select the best passages, and LLM generation to produce the answer.

**Key design decisions:**
- Web search is the primary retrieval mechanism. Perplexity maintains its own
  web index for some queries and uses third-party search APIs for others.
- The re-ranking stage is critical because web search results can be noisy.
  Perplexity uses a cross-encoder model to score each passage against the query.
- Citations are surfaced inline with the answer, and each citation links to the
  source URL. This is a key differentiator from traditional search engines.

- Citation accuracy is a differentiator. Perplexity tracks which passage was used
  for each claim and surfaces the source URL directly.

### Glean: Enterprise Search with RAG

Glean is an enterprise search platform that uses RAG to answer questions across
a company's internal tools (Slack, Google Drive, Notion, Jira, Salesforce, and
more). The system must handle hundreds of connectors, fine-grained access control,
and cross-document reasoning.

**Architecture:**
Glean indexes content from every connected application into a unified search
index. Each document is enriched with metadata about its source, access control
list, and freshness. Queries search across all connected sources and return
unified answers.

**Key design decisions:**
- Connector-based architecture with standardized metadata. Each connector
  (Slack, Google Drive, etc.) normalizes content into a common format.
- Fine-grained access control at the document level. A user can only see results
  from documents they have permission to access in the source application.
- Cross-document reasoning. Glean can answer questions that require synthesizing
  information from a Slack message, a Google Doc, and a Jira ticket.

## Open Source Implementations

### LangChain

LangChain was the first widely adopted framework for building RAG pipelines. It
provides a modular abstraction layer over retrievers, embedding models, and LLMs.

**Strengths:**
- Extensive ecosystem of integrations (50+ vector DBs, 50+ LLMs, 30+ document
  loaders)
- Rich chain composition patterns (LCEL — LangChain Expression Language)
- Active community with thousands of examples

**Weaknesses:**
- Abstraction leaks are common. Debugging a failed pipeline often requires
  understanding the internals of multiple integrations.
- Performance overhead from the abstraction layer. Each chain step adds 1-5ms
  of orchestration overhead.
- Rapid API evolution. Code written for LangChain v0.1 often breaks on v0.2.

### LlamaIndex

LlamaIndex (formerly GPT Index) focuses specifically on data indexing and
retrieval for LLMs. It provides data structures specifically designed for RAG.

**Strengths:**
- Strong data ingestion capabilities. Handles PDFs, databases, APIs, and
  unstructured text with sophisticated parsing.
- Advanced indexing strategies (tree index, keyword table, vector store index,
  document summary index)
- Built-in evaluation tools for retrieval quality

**Weaknesses:**
- Steeper learning curve than LangChain for simple use cases
- Indexing abstractions can be confusing (when to use each index type is not
  always obvious)
- Smaller community than LangChain

### Haystack

Haystack is a production-oriented framework by deepset. It focuses on building
deployable, scalable RAG pipelines.

**Strengths:**
- Production-ready pipeline architecture with built-in caching and batching
- Strong document processing pipeline (clean, split, embed, store)
- Native integration with Hugging Face models and embedding pipelines

**Weaknesses:**
- Smaller integration ecosystem than LangChain
- Fewer tutorials and examples for complex scenarios
- REST API integration is less mature than the core pipeline

### Chroma

Chroma is an open-source, embeddable vector database designed specifically for
RAG. It is not a full pipeline framework but a storage and retrieval component.

**Strengths:**
- Simple API, easy to embed in existing Python applications
- In-memory and persistent modes for development and production
- Automatic embedding management (can use your own embedding function)

**Weaknesses:**
- Limited to vector storage and retrieval — does not provide pipeline
  orchestration, chunking, or LLM integration
- Not designed for massive scale (best for 1M-10M vectors)
- Fewer query-time features than Pinecone or Weaviate

### Open Source Comparison

| Feature | LangChain | LlamaIndex | Haystack | Chroma |
|---------|-----------|------------|----------|--------|

## Cloud Provider Implementations

### AWS Bedrock Knowledge Bases

AWS Bedrock Knowledge Bases provides a fully managed RAG service. You connect
your data sources (S3, SharePoint, Confluence), and Bedrock handles chunking,
embedding, indexing into a vector database (Aurora PostgreSQL or Pinecone), and
retrieval.

**Key features:**
- Managed chunking with configurable chunk size and overlap
- Automatic embedding using Amazon Titan or Bedrock-supported models
- Integration with Aurora PostgreSQL (pgvector) or Pinecone
- Guardrails for filtering harmful content

**Limitations:**
- Limited to AWS ecosystem. If your data is not in AWS, you need to move it.
- Less control over the retrieval pipeline than a custom implementation.
- Chunking strategies are limited compared to LlamaIndex or custom pipelines.

### GCP Vertex AI RAG

Vertex AI provides a RAG engine that integrates with Google's vector search
(Vertex AI Vector Search) and Gemini models.

**Key features:**
- ScaNN-based vector search (Google's proprietary ANN algorithm) that is among
  the fastest available
- Direct integration with Google Workspace (Docs, Drive, Gmail) for data sources
- Automatic embedding using the text-embedding-gecko model family

**Limitations:**
- Tied to Google Cloud ecosystem
- Limited customization of the retrieval pipeline
- Cost can be higher than self-managed alternatives for high-volume use cases

### Azure AI Search

Azure AI Search is a managed search service that has been extended to support
vector search and RAG. It integrates with Azure OpenAI Service.

**Key features:**
- Hybrid search (keyword + vector) with semantic ranking built in
- Integration with Azure OpenAI for embedding and generation
- Role-based access control integrated with Azure AD

**Limitations:**
- Tied to Azure ecosystem
- Vector search is a newer feature and may not have the same maturity as
  dedicated vector databases
- Indexing throughput is limited compared to Pinecone or Milvus

## Lessons Learned

### Lesson 1: Measure Retrieval Quality Before Anything Else

Every production RAG deployment reveals that retrieval quality is the bottleneck.
Teams spend months optimizing prompt engineering and generation parameters, only
to discover that 40% of retrievals return irrelevant documents. The quality of
the retrieved context determines the upper bound of the system's answer quality.
Measure hit rate, relevance score distribution, and answer acceptance rate from
day one.

### Lesson 2: Caching Is Not Optional

Without caching, a RAG system with 1000 QPS costs approximately $2000 per day
in LLM API calls alone. With a three-tier cache (query, embedding, LLM response),
the effective cost drops to $600-800 per day. Caching also reduces p99 latency
from 2 seconds to under 100ms on cache hits. Invest in caching infrastructure
early.

### Lesson 3: Metadata Filtering Prevents Many Failures

Most RAG failures come from retrieving irrelevant documents. A simple metadata
filter (by domain, source, date range, access level) eliminates 50-60% of
retrieval failures. Tag every chunk with metadata during indexing and pass
applicable filters with every query.

### Lesson 4: The Re-Ranker Is Worth the Latency

A cross-encoder re-ranker adds 100-300ms of latency but improves answer quality
by 15-25%. In production testing, users consistently prefer the slightly slower
response with a re-ranker over the faster response without one. The re-ranker
catches the false positives that ANN search inevitably produces.

### Lesson 5: Monitoring Must Be Full-Stack

You need observability into every stage of the pipeline to debug production
issues. The most common debugging scenario is "the answer was wrong" and the
question is "was it a retrieval failure, a context assembly failure, or a
generation failure?" Without full-stack tracing, this question is unanswerable.
Instrument every stage with latency, relevance score, and error rate metrics.

### Lesson 6: Embedding Model Choice Matters More Than You Think

Teams often default to the cheapest or most popular embedding model without
testing it against their specific documents. In one case study, a fintech company
using text-embedding-3-small for financial documents observed a 72% hit rate.
Switching to a finance-tuned embedding model (FinBERT embeddings) improved the
hit rate to 91% — a 19 percentage point improvement that completely changed the
user experience. The lesson is that generic embedding models are good for
prototyping, but domain-specific models or fine-tuned embeddings are worth the
investment for production.

### Lesson 7: Document Processing Is the Most Underestimated Component

The chunking step is often treated as a simple text split. In reality, document
processing (PDF parsing, HTML cleaning, image extraction, table parsing) is the
most labor-intensive part of building a RAG system. A study of 10 production RAG
deployments found that teams spent 40% of their development time on document
processing and chunking, 30% on retrieval tuning, 20% on prompt engineering, and
only 10% on LLM selection. Poor document processing corrupts the data at the
source, and no amount of downstream tuning can fix garbage input.

### Lesson 8: User Feedback Is Your Best Evaluation Signal

Offline evaluation metrics (hit rate, MRR, NDCG) are useful for development, but
they do not always correlate with user satisfaction. In one case, a team improved
their MRR by 15% but saw no change in user feedback scores. The reason was that
users cared about answer comprehensiveness, not just precision. The lesson is to
collect user feedback (thumbs up/down, follow-up questions, rephrased queries)
and use it as the primary evaluation signal. Offline metrics are diagnostic tools,
not targets.

## Emerging Trends

### GraphRAG

GraphRAG combines knowledge graphs with vector search. Instead of retrieving
only semantically similar chunks, GraphRAG also traverses relationships between
entities. For example, a query about "metformin" would retrieve documents about
metformin and also traverse the graph to find related documents about diabetes,
kidney disease, and drug interactions. Microsoft Research's GraphRAG paper
showed a 25% improvement in answer comprehensiveness for multi-part questions.

### Agentic RAG

In agentic RAG, the LLM is given a set of tools (vector search, web search,
code execution, database query) and decides which tools to use to answer the
query. The agent can perform multiple rounds of retrieval, synthesize information
across tools, and even ask clarifying questions. This is the direction Perplexity
AI and Google's Gemini are moving toward.

### Streaming RAG

Streaming RAG processes query results incrementally. As the first chunks are
retrieved, the LLM begins generating an answer. As more chunks arrive, the
answer is refined. This reduces perceived latency because the user sees the
first tokens of the answer within 200ms, even though the full retrieval pipeline
takes 500ms.
