## Tier 1: Beginner

### Challenge 1: Cookie Security Flags

You are reviewing a colleague's session implementation. The `Set-Cookie` header is:
```http
Set-Cookie: session_id=abc123; Path=/; Max-Age=3600
```

**Task:** Identify three security vulnerabilities in this cookie configuration and explain how to fix each one.

<details>
<summary>Solution</summary>

**Vulnerability 1: Missing HttpOnly flag**
- **Risk:** JavaScript can read `document.cookie` and steal the session ID via XSS.
- **Fix:** Add `HttpOnly` to prevent JavaScript access.

**Vulnerability 2: Missing Secure flag**
- **Risk:** The cookie is sent over HTTP connections. An attacker on the same network can intercept it via packet sniffing.
- **Fix:** Add `Secure` to ensure the cookie is only sent over HTTPS.

**Vulnerability 3: Missing SameSite attribute**
- **Risk:** The cookie is sent on cross-site requests, enabling CSRF attacks.
- **Fix:** Add `SameSite=Lax` (or `Strict` for higher security).

**Fixed header:**
```http
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=3600
```
</details>

### Challenge 2: Session ID Generation

You find this code in production:
```python
import random
import time

def generate_session_id(user_id):
    timestamp = int(time.time())
    random_part = random.randint(0, 1000000)
    return f"session_{user_id}_{timestamp}_{random_part}"
```

**Task:** Identify three problems with this session ID generation approach and provide a corrected implementation.

<details>
<summary>Solution</summary>

**Problem 1: Predictable.** The session ID encodes the user_id and timestamp. An attacker can guess valid session IDs by iterating user IDs and timestamps.

**Problem 2: Insufficient entropy.** `random.randint(0, 1000000)` provides only ~20 bits of entropy. With 1 million active sessions, an attacker has a high probability of guessing a valid session ID.

**Problem 3: Non-cryptographic random.** `random` uses a Mersenne Twister PRNG, which is predictable if the attacker can observe a sequence of outputs.

**Corrected implementation:**
```python
import secrets

def generate_session_id():
    """Generate a cryptographically secure, opaque session ID."""
    return secrets.token_hex(32)  # 256-bit random value
```
</details>

### Challenge 3: Session Expiry Logic

You are implementing session expiry for a user-facing web application. The product team wants users to stay logged in as long as they are active, but the security team requires that no session lives longer than 24 hours.

**Task:** Write the session validation logic that implements both requirements.

<details>
<summary>Solution</summary>

```python
from datetime import datetime, timedelta

def validate_session(session):
    """
    Validate a session with sliding + absolute expiry.
    
    - Sliding: session extends by 1 hour on each activity
    - Absolute: no session lives longer than 24 hours from creation
    """
    now = datetime.utcnow()
    
    # Check absolute expiry (24 hours from creation)
    if now > session["created_at"] + timedelta(hours=24):
        return False  # Session expired (absolute limit)
    
    # Check sliding expiry (1 hour from last activity)
    if now > session["last_access"] + timedelta(hours=1):
        return False  # Session expired (idle timeout)
    
    # Session is valid. Update last_access to extend sliding window.
    session["last_access"] = now
    return True
```

**Test cases:**
- Session created at T=0, accessed at T=1: valid (both checks pass)
- Session created at T=0, last accessed at T=23.5, accessed at T=23.6: valid (sliding OK, absolute OK)
- Session created at T=0, last accessed at T=1, accessed at T=25: invalid (absolute limit exceeded)
- Session created at T=0, last accessed at T=0.5, accessed at T=1.6: invalid (sliding timeout exceeded)
</details>

### Challenge 4: Distributed Session Store Race Condition

Your e-commerce platform uses Redis as a session store. The shopping cart is stored in the session. A user has two browser tabs open. In Tab A, they add item X. In Tab B, they add item Y. Both tabs send requests simultaneously.

The session store flow:
1. Tab A reads session (cart: [])
2. Tab B reads session (cart: [])
3. Tab A appends X, writes session (cart: [X])
4. Tab B appends Y, writes session (cart: [Y])
5. Result: item X is lost.

**Task:** Implement a solution that prevents this race condition using Redis transactions.

<details>
<summary>Solution</summary>

**Solution using Redis Lua script (atomic execution):**

```lua
-- add_to_cart.lua
-- Args: session_key, item_to_add
-- Returns: updated cart as JSON array

local session_data = redis.call('GET', KEYS[1])
local session = cjson.decode(session_data)

-- Add item to cart
table.insert(session.cart, ARGV[1])

-- Update session
redis.call('SET', KEYS[1], cjson.encode(session), 'EX', 3600)

return cjson.encode(session.cart)
```

```python
import redis

r = redis.Redis()
add_to_cart_script = r.register_script(open('add_to_cart.lua').read())

def add_item_to_cart(session_id, item):
    session_key = f"session:{session_id}"
    result = add_to_cart_script(keys=[session_key], args=[json.dumps(item)])
    return json.loads(result)
```

**Alternative: Redis WATCH/MULTI/EXEC (optimistic locking):**

```python
def add_item_to_cart_optimistic(session_id, item, max_retries=3):
    session_key = f"session:{session_id}"
    
    for attempt in range(max_retries):
        r.watch(session_key)
        session_data = r.get(session_key)
        session = json.loads(session_data)
        
        session["cart"].append(item)
        
        pipe = r.pipeline(transaction=True)
        pipe.multi()
        pipe.set(session_key, json.dumps(session), ex=3600)
        
        try:
            pipe.execute()
            return session["cart"]  # Success
        except redis.WatchError:
            if attempt == max_retries - 1:
                raise  # Give up after max retries
            continue  # Retry with fresh data
```

The Lua script approach is preferred because it is atomic (no race window) and requires only one round trip to Redis.
</details>

### Challenge 5: Session Architecture for a Microservices Platform

A platform has 12 microservices (auth, user, orders, payments, inventory, shipping, notifications, search, recommendations, reviews, analytics, admin). Each service needs to know the current user and their permissions.

**Task:** Design a session architecture that allows all 12 services to access session data without each service making its own Redis call.

<details>
<summary>Solution</summary>

**Approach 1: Session Token (JWT) — Recommended**

The auth service issues a signed JWT containing user identity and permissions. The JWT is passed via HTTP header from service to service.

```python
# Auth service issues JWT on login
token = jwt.encode({
    "user_id": 42,
    "role": "customer",
    "permissions": ["read:profile", "write:orders"],
    "session_id": "abc123",
    "exp": 1720003600
}, SECRET_KEY, algorithm="RS256")

# Each service validates the JWT (no Redis call needed)
def authenticate_request(request):
    token = request.headers.get("Authorization", "").replace("Bearer ", "")
    payload = jwt.decode(token, PUBLIC_KEY, algorithms=["RS256"])
    request.user = payload
    return payload
```

**Pros:** No inter-service dependency. No shared session store. Each service independently validates the token.
**Cons:** Cannot revoke sessions server-side. Token size grows with permissions.

**Approach 2: Session Context Header**

The API gateway validates the session and injects a signed session context header to downstream services.

```python
# API gateway validates session and creates context header
session = redis.get(f"session:{session_id}")
context = {
    "user_id": session["user_id"],
    "role": session["role"],
    "permissions": session["permissions"]
}
# Sign the context header to prevent tampering
context_token = jwt.encode(context, INTERNAL_SECRET, algorithm="HS256")
# Forward to downstream services
request.headers["X-Session-Context"] = context_token
```

**Pros:** Single Redis call per request (at the gateway). All downstream services get session data without extra calls.
**Cons:** Gateway becomes a bottleneck. Internal services must trust the gateway's signature.

**Approach 3: Sidecar Session Cache**

Each service runs a sidecar (local Redis or Memcached) that caches session data. The sidecar reads from the central session store on cache miss.

```python
def get_session(session_id):
    # Try local cache first
    data = local_cache.get(f"session:{session_id}")
    if data:
        return data
    
    # Miss: read from central Redis
    data = central_redis.get(f"session:{session_id}")
    if data:
        # Cache locally for 5 minutes
        local_cache.set(f"session:{session_id}", data, ex=300)
    return data
```

**Pros:** Most reads are served from local cache (~1 ms). Reduced load on central Redis.
**Cons:** Stale data possible (5-minute cache TTL). More infrastructure (sidecar per service).
</details>

### Challenge 6: Session Migration

You are migrating from a legacy PHP application using file-based sessions to a new Python application using Redis sessions. Both applications must be able to read and write sessions during the migration period (which will last 2 weeks).

**Task:** Design a session migration strategy that allows both applications to share session data during the transition.

<details>
<summary>Solution</summary>

**Strategy: Dual-Read, Single-Write**

1. **Phase 1: Deploy Redis.** Add Redis alongside the existing file-based session store.
2. **Phase 2: Dual-read.** The PHP application reads from Redis first, then falls back to file-based sessions. The Python application reads from Redis only.
3. **Phase 3: Single-write (PHP to Redis).** The PHP application writes new sessions to Redis. Existing file-based sessions are read from files but migrated to Redis on first access.
4. **Phase 4: Validation.** Monitor both stores for consistency. Verify that all active sessions are in Redis.
5. **Phase 5: Cutover.** Disable file-based session reads. All sessions live in Redis.
6. **Phase 6: Cleanup.** Remove file-based session storage code. Delete old session files.

```python
# PHP session handler (dual-read, single-write)
class MigrationSessionHandler implements SessionHandlerInterface {
    private $redis;
    private $fileHandler;
    
    public function read($session_id) {
        // Try Redis first
        $data = $this->redis->get("session:$session_id");
        if ($data !== false) {
            return $data;
        }
        
        // Fall back to file-based session
        $data = $this->fileHandler->read($session_id);
        if ($data !== false) {
            // Migrate to Redis on first read
            $this->redis->set("session:$session_id", $data, 3600);
        }
        return $data;
    }
    
    public function write($session_id, $data) {
        // Write to both stores during migration
        $this->redis->set("session:$session_id", $data, 3600);
        $this->fileHandler->write($session_id, $data);
    }
}
```

**Rollback plan:** If the migration causes issues, revert to file-based sessions. The file-based sessions were never deleted, so they are still available as the fallback.
</details>

## Tier 3: Advanced

### Challenge 7: Design a Session System for 10 Billion Requests Per Day

A global social media platform handles 10 billion requests per day (approximately 115,000 requests per second). Each request requires session validation. The platform has users in 200+ countries.

**Task:** Design a session system that handles this scale with p99 latency under 10 ms.

<details>
<summary>Solution</summary>

**This is not a Redis problem.** At 115,000 req/s, a Redis cluster would need 10-20 nodes and would be expensive. More importantly, cross-region latency makes a centralized session store infeasible.

**The Solution: Token-Based Sessions with Regional Rotation**

**1. JWT access tokens (15-minute TTL)**
- Signed with a regional private key
- Contain user_id, role, and a session version
- Validated with a cached public key (no Redis call)
- 15-minute TTL limits the damage from a stolen token

**2. Regional auth services**
- Each geographic region has its own auth service
- Tokens signed with the region's private key
- Public keys distributed to all regions via a config service

**3. Session version (for revocation)**
- Each user has a session version stored in a regional Redis cache
- On password change or force logout, the version is incremented
- The version check is a fast Redis read (1-2 ms) in the user's home region
- Version is cached for 5 minutes in the application server

**4. Home region routing**
- Each user is assigned a home region based on registration location
- Requests are routed to the user's home region for session validation
- Read-only operations use cached data

```
Architecture overview:

User in Japan ──> CDN ──> AP-East Gateway ──> Auth Service (AP-East)
                                                    │
                                                    ├─ Validate JWT (local, no call)
                                                    ├─ Check version cache (local Redis)
                                                    └─ Route to service (local)

User visiting Japan from Brazil ──> CDN ──> AP-East Gateway
                                           Authenticated via JWT
                                           Version check: cache hit (fast)
                                           Full version check: redirected to home region
```

**5. Capacity numbers**

| Component | Requirement | Capacity |
|---|---|---|
| JWT validation | 115,000/s | Unlimited (CPU-bound, ~10K/s per core) |
| Version check | 115,000/s | 12 Redis nodes x 10K/s = 120K/s |
| Token refresh | 115,000/900 = 128/s | Trivial (15-min TTL, 1/900th of requests) |
| Revocation | Rare | Redis set per user, deleted on TTL expiry |
</details>

### Challenge 8: Session Store DoS Attack

An attacker creates 10 million session entries in your Redis session store, exhausting memory and causing evictions of legitimate sessions.

**Task:** Design defenses against session store exhaustion attacks.

<details>
<summary>Solution</summary>

**Defense 1: Rate limit session creation**
```python
# Rate limit: max 10 session creations per IP per minute
def can_create_session(ip_address):
    key = f"rate_limit:session_create:{ip_address}"
    count = redis.incr(key)
    if count == 1:
        redis.expire(key, 60)  # 1 minute window
    return count <= 10
```

**Defense 2: Validate before creating**
```python
def create_session(request):
    ip = request.remote_addr
    
    # Check rate limit
    if not can_create_session(ip):
        raise RateLimitExceeded("Too many session creation attempts")
    
    # Check CAPTCHA for suspicious IPs
    if is_suspicious_ip(ip):
        if not verify_captcha(request):
            raise CaptchaRequired("Please complete the CAPTCHA")
    
    # Validate user agent
    if not is_valid_user_agent(request.user_agent):
        raise InvalidRequest("Invalid user agent")
    
    # Create session
    return do_create_session(request)
```

**Defense 3: Session creation requires proof of work**
For anonymous sessions (not authenticated), require a client-side proof of work:
```python
# Client sends: session_request + nonce + hash
# Server verifies: hash(session_request + nonce) starts with "0000"
def verify_proof_of_work(session_request, nonce, hash_value):
    expected = hashlib.sha256(f"{session_request}{nonce}".encode()).hexdigest()
    return hash_value == expected and hash_value.startswith("0000")
```

**Defense 4: Memory limits and eviction policies**
```redis
# Redis configuration
maxmemory 8gb
maxmemory-policy allkeys-lru  # Evict least recently used sessions
```

**Defense 5: Anomaly detection**
Monitor session creation rate and alert on spikes:
```python
# Monitor: session creation rate per second
session_create_rate = redis.get("stats:session_creates_per_second")
if session_create_rate > 1000:  # 10x normal
    alert("Possible session store DoS attack detected")
```

**Defense 6: Session creation costs**
For unauthenticated sessions, add a small cost (CAPTCHA, proof of work, or rate limit). Authenticated sessions are not rate-limited because the user already proved their identity.
</details>

### Challenge 9: Session Consistency Across Data Centers

Your platform has data centers in US-East, EU-West, and AP-East. A user in Tokyo logs in and is routed to AP-East. The session is created in AP-East Redis. Thirty seconds later, the user's request is routed to US-East (due to a load balancer issue). The US-East Redis does not have the session.

**Task:** Design a solution that handles this session consistency problem across regions.

<details>
<summary>Solution</summary>

**Approach 1: Stick to home region (recommended)**
Assign each user to a home region. Route all requests for that user to their home region.

```python
def get_home_region(user_id):
    """Assign user to a home region based on user_id hash."""
    region_hash = hash(user_id) % 100
    if region_hash < 40:
        return "us-east"
    elif region_hash < 70:
        return "eu-west"
    else:
        return "ap-east"
```

**Pros:** Simple. No cross-region session replication needed.
**Cons:** Users outside their home region experience higher latency. Load balancer must route by user_id.

**Approach 2: Session replication with local read**
Session writes go to the primary region. Session reads first check the local region, then fall back to the primary region.

```python
def get_session(session_id, user_region):
    # Try local Redis first
    local_data = local_redis.get(f"session:{session_id}")
    if local_data:
        return json.loads(local_data)
    
    # Fall back to home region Redis
    home_region = get_home_region_for_session(session_id)
    if home_region != user_region:
        data = remote_redis.get(f"session:{session_id}")
        if data:
            # Cache locally for 5 minutes
            local_redis.setex(f"session:{session_id}", 300, data)
            return json.loads(data)
    
    return None
```

**Pros:** Fast local reads for most requests. Graceful fallback for cross-region requests.
**Cons:** Stale data possible (5-minute local cache). Cross-region reads are slow (100-300 ms).

**Approach 3: DynamoDB Global Tables**
Use DynamoDB Global Tables for the session store. DynamoDB automatically replicates writes across regions with sub-second latency.

```python
import boto3

dynamodb = boto3.resource('dynamodb', region_name='ap-east-1')
table = dynamodb.Table('Sessions')

def get_session(session_id):
    # Read from local region (DynamoDB Global Tables
    # replicates writes from all regions)
    response = table.get_item(
        Key={'session_id': session_id},
        ConsistentRead=False  # Eventual consistency
    )
    return response.get('Item')
```

**Pros:** Fully managed. Automatic cross-region replication. No custom replication logic.
**Cons:** Higher latency than Redis (5-20 ms vs 1-2 ms). Eventual consistency (sub-second replication lag). Higher cost per operation.
</details>
