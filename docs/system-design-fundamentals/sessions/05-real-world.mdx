## How Companies Handle Sessions at Scale

Session management at internet scale requires solving problems that are invisible at small scale: session store throughput, multi-region latency, server failure handling, and security at scale. Each major platform has evolved a unique approach.

---

## Case Study 1: E-Commerce — Session Architecture at Scale

An e-commerce platform handling 10 million daily active users with 100,000 concurrent shoppers needs a session system that handles high throughput, survives server failures, and supports multi-region deployments.

### The Problem

- 100,000 concurrent authenticated sessions
- Shopping cart, browse history, and preferences in session
- 10,000 session reads per second during peak
- 1,000 session writes per second (cart updates)
- Multi-region (US, EU, AP)

### Key Design Decisions

**1. Minimal session data.** The session stores only user_id, role, and a cart_id reference. The actual cart contents live in a separate DynamoDB table. This keeps session data small (under 1 KB) and session operations fast.

```python
# Session data (in Redis, ~1 KB)
session = {
    "user_id": 42,
    "role": "customer",
    "cart_id": "cart_abc123",
    "session_start": 1720000000
}

# Cart data (in DynamoDB, potentially large)
cart = {
    "cart_id": "cart_abc123",
    "items": [
        {"sku": "SHIRT-BLUE-M", "qty": 2, "price": 2999},
        {"sku": "PANTS-BLK-32", "qty": 1, "price": 4999},
        {"sku": "SOCKS-WHT-O", "qty": 3, "price": 499}
    ],
    "total": 10995,
    "updated_at": 1720000050
}
```

**2. Redis Cluster for throughput.** Six master nodes each handle ~1,600 session operations per second. Each master has a replica for failover. The cluster automatically handles node failures.

**3. Multi-region active-passive.** US-East is the primary region. All session writes go to US-East Redis. EU-West and AP-East app servers read from US-East Redis (200-300 ms latency) but writes are handled by the origin region. A planned future upgrade would use DynamoDB Global Tables for lower cross-region latency.

**4. Graceful degradation.** If Redis is unavailable, the site operates in read-only anonymous mode. Authenticated users cannot add items to cart or check out, but the site remains accessible and browseable.

### How They Handle Peak Traffic

- **Black Friday**: 5x normal traffic. Redis Cluster auto-scales by adding more shards (nodes). Application servers auto-scale horizontally.
- **Session store saturation**: If Redis reaches 80% memory, the ops team increases TTL eviction frequency and adds more shards.
- **Server failure**: On app server failure, all sessions are intact because the session store is external. Users see no interruption.

## Case Study 2: Social Media Platform — Token-Based Sessions

A social media platform with 500 million monthly active users uses JWT-based sessions for their mobile API. The architecture is fundamentally different from the e-commerce approach.

### Why the Different Approach

- **Scale**: 500 million users means 50 million+ concurrent sessions. A Redis cluster of this size would be extremely expensive.
- **Read-heavy**: Most API requests are reads (feed, profiles, notifications). Session revocation is a minor concern because users rarely need to be force-logged-out.
- **Mobile-first**: The API is consumed by mobile apps, not browsers. Cookies are not the primary transport.
- **Multi-region by default**: Users are distributed globally. A centralized session store would add 100-500 ms latency to every request.

### The Architecture

```python
# JWT token structure
# Token is ~500 bytes, sent as Authorization: Bearer <token>
token_payload = {
    "sub": "user_42",           # Subject (user ID)
    "role": "user",              # Role
    "iat": 1720000000,           # Issued at
    "exp": 1720003600,           # Expires (1 hour)
    "scope": "read:feed write:post",  # Permissions
    "iss": "auth.service",       # Issuer
    "jti": "unique-token-id"     # JWT ID (for revocation if needed)
}
```

### Session Revocation Strategy

Token-based sessions cannot be revoked server-side, so the platform uses a hybrid approach:

1. **Short-lived access tokens (1 hour)**: The main API token. Not revocable. If compromised, the attacker has access for up to 1 hour.
2. **Long-lived refresh tokens (30 days)**: Stored in a database. Used to obtain new access tokens. Revocable: the server deletes the refresh token on logout or password change.
3. **Token blocklist**: A small Redis set of revoked JWT IDs (jti). Only populated when a user explicitly logs out or changes their password. The blocklist is checked on every API request.

```python
def validate_token(token):
    try:
        payload = jwt.decode(token, PUBLIC_KEY, algorithms=["RS256"])
    except jwt.InvalidTokenError:
        return None

    # Check if token is explicitly revoked
    if redis.sismember("revoked_tokens", payload["jti"]):
        return None  # Token has been revoked

    return payload

def revoke_user_sessions(user_id):
    """Revoke all sessions for a user (e.g., on password change)."""
    # Add all active JWT IDs to the blocklist
    active_jwts = redis.smembers(f"user_jwts:{user_id}")
    for jti in active_jwts:
        redis.sadd("revoked_tokens", jti)
    redis.delete(f"user_jwts:{user_id}")
```

### How They Handle Scale

- **No session store**: The JWT contains all session data. The server validates the signature using a public key. No database or Redis lookup for the common case.
- **Public key distribution**: The auth service signs tokens with a private key. All API servers cache the public key. Token validation is a single cryptographic operation.
- **Refresh tokens**: Stored in a sharded PostgreSQL database. Refresh token lookups are relatively rare (one per hour per user, compared to dozens of API calls per minute).
- **Token size**: 500 bytes per token. At 50 million active users, the total token storage is 25 GB — but this is distributed across client devices, not on the server.

## Case Study 3: Banking Application — Strict Security

A banking application takes the opposite approach: maximum security over convenience.

### Session Constraints

- 15-minute absolute session timeout (no sliding)
- Single session per user (login on a new device invalidates the old session)
- Session ID rotated on every request (not just on login)
- IP address and user-agent fingerprinting
- Session invalidated on any suspicious activity
- All session data encrypted at rest in the session store
- Session audit log: every session read and write is logged

### The Architecture

```python
def create_banking_session(user_id, request):
    old_session_id = validate_session(request)
    
    if old_session_id:
        rotate_session(old_session_id)
    
    session_id = secrets.token_hex(32)
    
    session_data = {
        "user_id": user_id,
        "created_at": datetime.utcnow(),
        "expires_at": datetime.utcnow() + timedelta(minutes=15),
        "ip_address": request.remote_addr,
        "user_agent": request.user_agent.string,
        "device_id": request.cookies.get("device_id"),
        "last_activity": datetime.utcnow()
    }
    
    # Encrypt session data before storing
    encrypted = encrypt_session(session_data)
    
    session_store.set(f"bank_session:{session_id}", encrypted, ex=900)  # 15 min
    
    # Audit log
    audit_log.info(f"Session created: user={user_id}, session={session_id[:8]}...")
    
    return session_id

def validate_banking_session(request):
    session_id = request.cookies.get("session_id")
    if not session_id:
        return None
    
    encrypted = session_store.get(f"bank_session:{session_id}")
    if not encrypted:
        return None
    
    session_data = decrypt_session(encrypted)
    
    # Check expiry
    if datetime.utcnow() > session_data["expires_at"]:
        session_store.delete(f"bank_session:{session_id}")
        audit_log.warning(f"Expired session used: {session_id[:8]}...")
        return None
    
    # Check IP address
    if request.remote_addr != session_data["ip_address"]:
        session_store.delete(f"bank_session:{session_id}")
        audit_log.warning(f"IP mismatch: session={session_id[:8]}...")
        return None
    
    # Check user-agent
    if request.user_agent.string != session_data["user_agent"]:
        session_store.delete(f"bank_session:{session_id}")
        audit_log.warning(f"UA mismatch: session={session_id[:8]}...")
        return None
    
    # Rotate session ID on every request
    new_session_id = secrets.token_hex(32)
    session_data["last_activity"] = datetime.utcnow()
    session_data["expires_at"] = datetime.utcnow() + timedelta(minutes=15)
    encrypted = encrypt_session(session_data)
    session_store.set(f"bank_session:{new_session_id}", encrypted, ex=900)
    session_store.delete(f"bank_session:{session_id}")
    
    return new_session_id, session_data
```

### The Tradeoffs

- **User experience**: Users are logged out after 15 minutes of inactivity. They must re-authenticate frequently. This is intentional.
- **Infrastructure cost**: Session rotation on every request means 2x session store operations (delete + create). Audit logging adds additional storage cost.
- **Security**: A stolen session ID is valid for only one request (because it is rotated). IP and user-agent checks prevent most session hijacking attempts.

## Case Study 4: Serverless Platform — DynamoDB Sessions

A serverless e-commerce platform built on AWS Lambda uses DynamoDB for session storage. The architecture is fundamentally different from the Redis-based approach.

### Why DynamoDB?

- **Serverless compatibility**: Lambda functions are ephemeral. An in-memory session store is not possible.
- **No infrastructure to manage**: DynamoDB is fully managed. No Redis cluster to maintain.
- **Pay-per-request**: Session storage cost scales with actual usage, not provisioned capacity.
- **Multi-region replication**: DynamoDB Global Tables replicate sessions across regions with sub-second latency.

### The Architecture

```python
import boto3
from boto3.dynamodb.conditions import Key
import time
import json

dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('Sessions')

def lambda_handler(event, context):
    session_id = get_session_id_from_cookie(event)
    
    if not session_id:
        # New session
        session_id = create_session()
        return respond_with_cookie(event, session_id)
    
    # Validate session
    session = get_session(session_id)
    if not session:
        # Session expired or invalid
        session_id = create_session()
        return respond_with_cookie(event, session_id)
    
    # Process request with session data
    return process_request(event, session)

def get_session(session_id):
    response = table.get_item(
        Key={'pk': f'SESSION#{session_id}'},
        ConsistentRead=False  # Eventual consistency for lower latency
    )
    item = response.get('Item')
    if not item:
        return None
    
    # Check TTL
    if int(time.time()) > item['ttl']:
        table.delete_item(Key={'pk': f'SESSION#{session_id}'})
        return None
    
    return json.loads(item['data'])

def create_session():
    session_id = secrets.token_hex(32)
    ttl = int(time.time()) + 86400  # 24 hours
    
    table.put_item(
        Item={
            'pk': f'SESSION#{session_id}',
            'data': json.dumps({
                'created_at': int(time.time()),
                'cart': []
            }),
            'ttl': ttl
        }
    )
    return session_id
```

### Key Design Decisions

- **TTL attribute**: DynamoDB automatically deletes expired items based on the TTL attribute. No manual cleanup needed.
- **Eventually consistent reads**: Session reads use eventual consistency for lower latency. This means a session write may not be immediately visible to subsequent reads, but the window is typically under 100 ms.
- **Single table design**: Sessions are stored in a single DynamoDB table with a `SESSION#` prefix. This keeps the data model simple and cost-effective.

## Case Study 5: Real-Time Collaboration — Conflict Resolution

A collaborative document editing platform (like Google Docs or Notion) uses sessions to track user presence and editing state. The session architecture must handle real-time updates and conflict resolution.

### The Challenge

- Hundreds of users editing the same document simultaneously
- Real-time presence (who is viewing which section)
- Session data changes frequently (cursor position, selected text)
- Low latency required (sub-100 ms for presence updates)

### The Approach

The platform uses a hybrid architecture:

1. **Session store (Redis)**: Stores user identity, document permissions, and session metadata. TTL of 24 hours.
2. **Presence channel (WebSocket)**: Real-time presence data is not stored in the session. It is published to a pub/sub channel and is ephemeral.
3. **Document state (CRDT)**: The actual document content is managed by a CRDT (Conflict-free Replicated Data Type), not by the session system.

```python
# Session data (Redis, TTL=24h)
session = {
    "user_id": 42,
    "name": "Alice",
    "avatar_url": "https://...",
    "permissions": ["read", "write", "comment"],
    "active_docs": ["doc_abc", "doc_def"]
}

# Presence data (Redis pub/sub, ephemeral)
# Published to "presence:doc_abc" channel
presence = {
    "user_id": 42,
    "doc_id": "doc_abc",
    "cursor_position": {"line": 120, "col": 45},
    "status": "editing",
    "last_seen": 1720000050
}
```

The key insight: session data is relatively static (user identity, permissions) and stored in Redis. Presence data is highly dynamic (changes every keystroke) and is not stored in the session at all. Mixing the two would overload the session store with unnecessary writes.

## Common Patterns Across Companies

| Pattern | Companies Using It | Why |
|---|---|---|
| Redis session store | Most e-commerce, SaaS, social platforms | Fast, mature, built-in TTL |
| JWT tokens | Mobile-first platforms, APIs | Stateless, no server-side store |
| DynamoDB sessions | Serverless platforms, AWS-native apps | Managed, pay-per-request, multi-region |
| Session rotation on login | All security-conscious apps | Prevents session fixation |
| Sliding + absolute expiry | Most consumer apps | Balances UX and security |
| Minimal session data | All production systems | Performance and cost |
| Separate cart/presence from session | E-commerce, collaboration tools | Session store should not be the primary data store |
| Graceful degradation on store failure | Large-scale platforms | Site availability over session state |

The common thread: session data should be minimal, the session store should be fast and reliable, and the system should degrade gracefully when the session store is unavailable. No single architecture works for every scale — the right choice depends on your specific requirements for latency, revocability, operational complexity, and cost.
