## The Problem: HTTP Is Stateless

HTTP was designed as a stateless protocol. Each request-response pair is independent. The server does not know that request A and request B came from the same client, or that request B is a follow-up to request A. This design was intentional — statelessness simplifies server design, improves scalability, and makes caching straightforward. But it creates a fundamental problem for any application that needs to remember something about a user across multiple requests.

Consider these scenarios:

**Scenario 1: The Shopping Cart.** A user browses an e-commerce site, adds a shirt to the cart, continues browsing, adds pants, then proceeds to checkout. Between each click, the server must remember what is in the cart. Without state, every page load would show an empty cart.

**Scenario 2: The Authenticated Dashboard.** A user logs in with their credentials. The server verifies the password and redirects to the dashboard. Every subsequent page load requires the server to know that this user has already authenticated. Without state, the user would need to re-enter their password on every page.

**Scenario 3: The Multi-Step Form.** A user fills out a loan application across five pages: personal info, employment, assets, references, and review. The data from step 1 must be available when the user reaches step 5. Without state, the application would lose all progress between steps.

**Scenario 4: The Rate-Limited API.** A developer calls an API endpoint 100 times in one minute. The API must track how many calls this developer has made within the current window. Without state, every call would appear to be the first call.

All four scenarios share the same root cause: **HTTP is stateless, and these applications need state**.

---

## Insight

A session is the mechanism that adds state on top of a stateless protocol. The core idea is simple: the server creates a temporary container for data that lives across multiple requests from the same client. Both the client and the server agree on a unique identifier — the session ID — that links each request to its session container.

Compare these communication patterns:

| Pattern | Example | How State Is Maintained |
|---|---|---|
| Email | Alice sends an email to Bob. Bob replies. Each email is independent. | State is embedded in the message chain (reply headers, thread IDs) |
| Phone call | Both parties talk in real-time, picking up the conversation where they left off. | State is maintained by both parties' memory during the call |
| Library hold | You request a book at the front desk. The librarian puts it aside with a slip bearing your name. | State is stored on a physical slip (the session) |
| Hotel room key | You check in, receive a key card, and the key identifies you as the occupant of room 214 for the duration of your stay. | State is stored in the hotel's system; the key is the token |
| Vending machine | You insert coins, make a selection. The machine tracks your credit until the transaction completes. | State is maintained by the machine's internal counters during the transaction |
| Barcode on luggage | You check a bag at the airport. Each bag gets a barcode that links it to your itinerary. | State is stored in the airline's system; the barcode is the identifier |

The session sits between the purely stateless (email) and the continuously connected (phone call). It provides a persistent context without requiring a persistent connection.

## Core Vocabulary

Master these terms before moving to the deep dive.

**Session** — A temporary, server-side container that holds state associated with a specific client across multiple requests. The session exists for a defined lifetime and is destroyed when the user logs out, the session expires, or the server reclaims the resources.

**Session ID** — A unique, opaque identifier that links a client to its session. Typically a cryptographically random string stored in a cookie or returned in the response body.

**Session Store** — The backend storage where session data lives. Options include in-memory maps, Redis, Memcached, relational databases, and distributed key-value stores.

**Cookie** — A small piece of data that the server sends to the client and the client sends back on subsequent requests. HTTP cookies are the most common transport for session IDs.

**Secure Cookie** — A cookie with the `Secure` flag, meaning it is only sent over HTTPS. Prevents session ID leakage over unencrypted connections.

**HttpOnly Cookie** — A cookie with the `HttpOnly` flag, meaning it is not accessible to JavaScript in the browser. Prevents session ID theft via XSS attacks.

**SameSite Cookie** — A cookie with the `SameSite` attribute (Strict, Lax, or None), which controls whether the cookie is sent with cross-site requests. Prevents CSRF attacks.

**Session Fixation** — An attack where an attacker forces a victim to use a known session ID, then hijacks the session after the victim authenticates.

**Session Hijacking** — An attack where an attacker steals a valid session ID and uses it to impersonate the victim.

**Session Expiry** — The time limit after which a session is automatically invalidated. Can be absolute (fixed TTL) or sliding (reset on each activity).

**Absolute Timeout** — A session lifetime measured from creation. The session expires at creation time + timeout regardless of activity.

**Sliding Expiration** — A session lifetime measured from the last activity. Each request resets the expiry clock. Also called rolling expiration.

**Session Replay** — An attack where a captured session ID is reused after the legitimate session has ended but before the server has invalidated the session.

**Session Affinity** — A load-balancing strategy that routes all requests from the same client to the same server. Also called sticky sessions. Avoided in modern architectures but still used in legacy systems.

**Distributed Session Store** — A shared, network-accessible storage backend that all application servers can read from and write to. Enables sessions to survive server failures and scale horizontally.

**Token-Based Session** — An alternative to server-side sessions where all session state is encoded into a token (like a JWT) and sent to the client. The server validates the token on each request rather than looking up server-side state.

**Session Rotation** — The practice of generating a new session ID after a significant state change (like login). Prevents session fixation attacks.

## Step by Step

### Step 1: Client sends a request

The client sends an HTTP request to the server. At this point, no session exists. The request is just another stateless HTTP message.

### Step 2: Server identifies the need for a session

The server determines that the request requires stateful handling. This could be a login request, a shopping cart add, or any operation that needs to remember something across requests.

### Step 3: Server generates a session ID

The server generates a cryptographically random session ID. The ID should be:
- **Unique**: No two sessions share the same ID. Collision probability should be negligible.
- **Unpredictable**: An attacker cannot guess a valid session ID.
- **Opaque**: The ID reveals nothing about the user or the session data.

Good session IDs are generated using a cryptographically secure random number generator (CSPRNG) and are typically 128 bits or longer.

```python
import secrets

def generate_session_id() -> str:
    """Generate a cryptographically secure session ID."""
    return secrets.token_hex(32)  # 256-bit random hex string
```

### Step 4: Server creates the session store entry

The server stores the session data in the session store, keyed by the session ID. The initial data includes the creation time, last access time, and any application-specific data.

### Step 5: Server sends the session ID to the client

The server sets a `Set-Cookie` header in the HTTP response, instructing the client to store the session ID. The cookie should have security flags set:

```http
Set-Cookie: session_id=abc123def456; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=3600
```

- `HttpOnly`: Prevents JavaScript access. Mitigates XSS-based session theft.
- `Secure`: Only sent over HTTPS. Prevents interception over plain HTTP.
- `SameSite=Lax`: Not sent on cross-site requests. Mitigates CSRF attacks.
- `Path=/`: Sent with all requests to the domain.
- `Max-Age=3600`: Cookie expires in one hour. The server should independently enforce session expiry.

### Step 6: Client stores the cookie

The browser stores the cookie and attaches it to every subsequent request to the same domain that matches the cookie's path and security attributes.

### Step 7: Client sends a subsequent request with the session ID

The client includes the cookie in the `Cookie` header:

```http
GET /dashboard HTTP/1.1
Host: example.com
Cookie: session_id=abc123def456
```

### Step 8: Server looks up the session

The server extracts the session ID from the cookie, validates it, and looks up the session data in the session store.

### Step 9: Server validates session expiry

The server checks whether the session has expired. Two common expiry strategies:

- **Absolute timeout**: The session expires at `created_at + TTL`, regardless of activity. Example: A banking app expires sessions after 15 minutes, even if the user is actively clicking.
- **Sliding expiration**: The session expires at `last_access + TTL`. Each request extends the session. Example: A CMS extends the session as long as the editor is actively working.

```python
def is_session_valid(session: dict, strategy: str = "sliding") -> bool:
    now = datetime.utcnow()
    if strategy == "absolute":
        return now < session["expires_at"]
    elif strategy == "sliding":
        # Reset expiry on each access
        session["last_access"] = now
        session["expires_at"] = now + timedelta(hours=1)
        return True
    return False
```

### Step 10: Server processes the request using session data

The server uses the session data to customize the response. For example, the user's ID from the session is used to fetch their personal data, their role determines what they can see, and their preferences control the page layout.

### Step 11: Session rotation on privilege escalation

When a user's privilege level changes — most commonly when they log in from anonymous to authenticated — the server should rotate the session ID. This prevents session fixation attacks where an attacker sets a known session ID before the user logs in.

```python
def rotate_session(old_session_id: str, session_data: dict) -> str:
    """Create a new session ID and delete the old one."""
    new_session_id = generate_session_id()
    session_store.set(new_session_id, session_data)
    session_store.delete(old_session_id)
    return new_session_id
```

### Step 12: Session termination

The session ends when one of these conditions is met:

1. **Explicit logout**: The user clicks "Logout." The server deletes the session and clears the client cookie.
2. **Expiry**: The session TTL elapses. The server rejects requests with expired session IDs.
3. **Idle timeout**: The sliding expiration interval elapses without activity. The session is reclaimed.
4. **Server-side revocation**: An administrator forces logout for all sessions (e.g., password reset, account suspension).
5. **Resource pressure**: The session store runs out of memory and evicts the least recently used sessions.

### Step 13: Session cleanup

Sessions that are not explicitly deleted continue to occupy storage. The server must implement a cleanup mechanism:

- **Lazy expiration**: Check expiry on each access. Expired sessions are not deleted but are treated as invalid. Background processes periodically purge expired entries.
- **TTL-based eviction**: Redis and Memcached support automatic key expiry. The server sets a TTL on each session entry, and the store handles deletion.
- **Periodic sweep**: A cron job scans the session store and deletes expired entries.

## Session Storage Options

| Store | Read Latency | Durability | Scalability | Complexity |
|---|---|---|---|---|
| In-memory (app server) | under 1 ms | Lost on restart | Single node | None |
| Memcached | under 1 ms | Lost on restart | Distributed | Low |
| Redis | under 1 ms | Configurable (RDB/AOF) | Distributed | Low |
| PostgreSQL | 1-5 ms | Durable | Read replicas | Medium |
| DynamoDB | 5-20 ms | Durable | Multi-region | High |

**In-memory** is the simplest but does not survive server restarts and cannot be shared across servers. Use for development, single-server applications, or non-critical session data.

**Redis** is the most common production choice. It provides sub-millisecond reads, built-in TTL, and cluster support. The session data is ephemeral by design, so Redis's persistence is a convenience for recovery, not a requirement.

**PostgreSQL** is used when session data must be durable or when the application already has a database and the operational overhead of adding Redis is not justified. Session reads are slower than Redis but still fast enough for most workloads.

**DynamoDB** is chosen for multi-region deployments where sessions must survive region-level failures. The higher latency is acceptable when the alternative is a complete loss of session state.

## Variants of Session Architecture

### 1. Server-Side Sessions (Cookie-Based)

The session ID is stored in a cookie. All session data lives on the server. This is the traditional approach and the one described in the steps above.

**Pros:** Full control over session data. Can store arbitrarily large amounts. Can revoke sessions server-side. No client-side tampering.
**Cons:** Requires a session store lookup on every request. Session store is a single point of failure. Adds latency to every request.

### 2. Token-Based Sessions (JWT)

All session state is encoded into a signed token and sent to the client. The server validates the token on each request but does not need to look up session state.

**Pros:** No server-side session store. Stateless servers. Scales horizontally trivially. Lower latency per request.
**Cons:** Tokens cannot be revoked server-side (unless a blocklist is maintained). Token size grows with session data. Can encode only small amounts of data (typically 1-4 KB in a cookie).

### 3. Hybrid Sessions

The server stores a minimal session identifier and a small amount of critical state in a token, while the remainder of the session data lives in a server-side store.

**Pros:** Reduces session store lookups for common cases. Maintains revocability for critical operations.
**Cons:** More complex. Two systems to manage.

## Mental Model

When reasoning about sessions in a system design interview, ask yourself these six questions:

**Question 1: How much state do I need to remember?**

If the state is small (user ID, role, preferences), a token-based approach works. If the state is large (shopping cart contents, multi-step form data, feature flags), you need a server-side session store.

**Question 2: Can I afford a session store lookup on every request?**

If your p99 latency target is under 10 ms and your session store is in the same region, a Redis lookup (1-2 ms) is acceptable. If your latency budget is under 2 ms or your servers are in multiple regions, consider token-based sessions.

**Question 3: Do I need to revoke sessions server-side?**

If you need to force logout of specific users (password reset, account suspension, stolen device), you need server-side sessions or a token blocklist. Token-based sessions without a blocklist cannot be revoked.

**Question 4: How many concurrent sessions per user?**

Single-session-per-user (one device at a time) is simpler to implement. Multi-session (same user on phone, laptop, and tablet simultaneously) requires tracking multiple session IDs per user and handling conflicts.

**Question 5: What happens when the session store goes down?**

If the session store is unavailable, all authenticated requests fail. Mitigations include local session caches (serve stale data), read replicas, and multi-region replication. Token-based sessions inherently avoid this problem.

**Question 6: How long should sessions live?**

Short sessions (15-30 minutes) are more secure but create a poor user experience. Long sessions (days or weeks) are convenient but increase the window of vulnerability if a session ID is stolen. Most applications use a sliding expiration with a maximum absolute timeout — for example, a session expires after 1 hour of inactivity but is forcibly terminated after 24 hours regardless of activity.

## Key Takeaways

- Sessions bridge the gap between stateless HTTP and stateful applications by maintaining shared state across requests.
- A session ID is a cryptographically random, opaque identifier that links a client to its server-side session data.
- Cookies are the most common transport for session IDs, with HttpOnly, Secure, and SameSite flags for security.
- Session stores range from in-memory maps to Redis, PostgreSQL, and DynamoDB, each with different latency, durability, and scalability characteristics.
- Absolute timeouts and sliding expiration are the two primary expiry strategies, each suited to different use cases.
- Session rotation on privilege escalation prevents session fixation attacks.
- Session cleanup must be implemented (lazy expiry, TTL-based eviction, or periodic sweep) to prevent storage leaks.
- Token-based sessions (JWT) eliminate the session store lookup but sacrifice server-side revocability.
- The choice between server-side and token-based sessions is a tradeoff between latency, revocability, and scalability.
- Distributed session stores enable horizontal scaling but add latency and complexity.
