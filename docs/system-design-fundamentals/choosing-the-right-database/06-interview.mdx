# Choosing the Right Database --?" Interview Playbook

Database selection questions are among the most common in system design interviews because they test your ability to evaluate trade-offs under constraints --?" a skill that maps directly to real engineering decisions. This page prepares you for every variant of the question.

### System Design Questions with Database Emphasis

Some interview questions are system design questions that surface database selection as the key decision. These questions test whether you can identify the database as the bottleneck and propose appropriate alternatives.

**Question: "Design a URL shortener like bit.ly."** The database decision: you need to store short code to long URL mappings at high throughput with low latency. The short code is the key. Reads dominate (99 percent of requests are redirects). The data set might be 1 billion records at 1 KB each (roughly 1 TB). 

The right answer points to: key-value store (DynamoDB, Cassandra) or a sharded relational database (PostgreSQL with Citus). The key requirement is fast point lookups by short code. Range queries are not needed. Strong consistency is optional for redirects (a user seeing "page not found" and refreshing to find the page works fine). Eventual consistency with read-your-writes guarantees is sufficient.

**Question: "Design an online multiplayer leaderboard."** The database decision: you need to maintain a global sorted ranking of millions of players by score. Players query their rank and the top 100 players. Score updates happen at high velocity (10,000 updates/second). The leaderboard must reflect near-real-time scores.

The right answer: Redis sorted sets (ZADD, ZRANK, ZREVRANGE). Redis handles the high-frequency score updates and the sorted set query in under 1 millisecond. A single Redis instance handles 100,000+ score updates/second. For scale beyond a single instance, use Redis Cluster with sharding by player ID (each shard maintains a local sorted set) and an aggregator that merges the top 100 from each shard.

**Question: "Design a content delivery network (CDN) like Cloudflare or Akamai."** The database decision: you need to store cached content close to users across 200+ edge locations, cache metadata (origin URL, cache key, TTL, invalidation rules), and access logs for analytics. The edge cache needs sub-millisecond lookups. The control plane needs strong consistency for cache invalidation.

The right answer: Use an in-memory key-value store at each edge location (Varnish, Redis, or a custom in-memory cache) for cached content. Use a globally replicated database (DynamoDB Global Tables or Cassandra multi-region) for cache metadata. Use a strongly consistent database (PostgreSQL or CockroachDB) for the control plane: account data, cache invalidation rules, and billing. The edge caches communicate with the control plane through a pub/sub system for real-time invalidation.

### Additional Phone Screen Questions

At the phone screen level, database questions are often simpler and focus on fundamental understanding:

"Explain the difference between SQL and NoSQL databases." The interviewer wants to hear you describe the relational model (tables, schemas, joins, ACID) versus non-relational models (documents, key-value, flexible schemas, BASE). Mention specific examples for each.

"What is an index and how does it work?" The interviewer wants a concise explanation of B-trees or hash indexes, the read-speed vs write-speed trade-off, and the concept of covering indexes. A concrete example (CREATE INDEX on a user's email column accelerates login queries) demonstrates practical understanding.

"Your database queries are getting slower as data grows. What do you do?" The interviewer wants you to diagnose the problem systematically: check the slow query log, check for missing indexes, check the buffer pool hit ratio, check for blocking queries, and check the hardware utilization. Mention EXPLAIN plans explicitly.

## How This Topic Appears

Database selection questions come in three categories: open-ended design, troubleshooting, and comparative evaluation.

### Open-Ended Design Questions

These questions present a system and ask you to choose the database. The interviewer watches how you gather requirements and weigh options.

1. "Design Twitter. What database would you use for tweets, users, and the timeline?"
2. "Design a ride-hailing system like Uber. How do you store driver locations and trip history?"
3. "Design an e-commerce platform. What database handles the product catalog versus the shopping cart versus orders?"
4. "Design a real-time chat system like WhatsApp. How do you store messages and ensure delivery across devices?"
5. "Design a video streaming service like Netflix. How do you store user progress, recommendations, and billing?"

### Troubleshooting Questions

These questions describe a failing system and ask you to diagnose and fix the database issue.

6. "Your team's application was fast at launch, but now queries have degraded 10x in latency. What do you check?"
7. "Your DynamoDB table that stores user sessions is throttling during peak hours. What went wrong?"
8. "Your read replicas are consistently 30 seconds behind the primary. How do you fix this?"

### Comparative Evaluation Questions

These questions ask you to compare specific databases for a given workload.

9. "Compare MongoDB and PostgreSQL for a content management system with blog posts, comments, and authors."
10. "Compare DynamoDB and Cassandra for an IoT telemetry pipeline ingesting 100,000 events per second."
11. "When would you choose CockroachDB over a sharded PostgreSQL cluster?"
12. "Your startup has 1,000 users and uses MongoDB. Should you migrate to Postgres before you raise Series A?"

### Follow-Up Questions and How to Handle Them

Interviewers rarely accept your first answer. They probe deeper to test your understanding. Expect these follow-ups:

**Follow-up: "Why not just use [alternative database]?"** Do not get defensive. Acknowledge the alternative's strengths, then explain why your primary choice is better for this specific workload. Example: "You could use DynamoDB here, and it would handle the write throughput well. I chose PostgreSQL because the query pattern includes joins between orders and line items. DynamoDB would require denormalizing that data and managing consistency in application code, which adds complexity that PostgreSQL handles natively."

**Follow-up: "What happens when your database gets 10x more traffic?"** Describe the scaling path. Do not just say "we add more nodes." Be specific: read replicas for read scaling, sharding for write scaling, cache for hot data, connection pooler for connection scaling. Describe the trigger for each intervention (e.g., "when the primary reaches 80 percent CPU, we add a read replica").

**Follow-up: "How do you handle a database migration without downtime?"** Describe the expand-migrate-contract pattern. Mention dual-writes, backfill, validation, and rollback. Acknowledge that zero-downtime migrations require careful planning and that some schema changes (e.g., adding a NOT NULL column to an existing table) require multiple phases.

**Follow-up: "What metrics do you monitor?"** List specific metrics with alert thresholds: query latency (p99 over 100 ms), connection count (over 80 percent of max), replication lag (over 10 seconds), disk usage (over 80 percent), cache hit ratio (below 90 percent). Mention that the slow query log is reviewed weekly.

**Follow-up: "What would make you change your database choice?"** Define specific measurable triggers. Not "if it gets slow" but "if the write throughput exceeds 5,000 writes/second, we would evaluate CockroachDB or sharded PostgreSQL." This shows you have thought about the evolution path.

## What the Interviewer Is Actually Testing

**Requirements gathering.** The single most important signal in a database selection interview. Candidates who immediately name a database without asking about the data model, query patterns, consistency needs, and scale fail this test even if they name the "correct" database. The interviewer wants to see that you know which questions to ask before you answer.

**Trade-off awareness.** There is no perfect database. The interviewer wants to hear you articulate what you gain and what you sacrifice with each choice. A candidate who says "use DynamoDB" without mentioning its limitations with joins, aggregations, or strongly consistent global reads signals shallow understanding.

**Depth of knowledge.** Vague answers like "PostgreSQL is reliable" do not differentiate candidates. Strong answers reference specific mechanisms: "PostgreSQL's B-tree indexes make range scans efficient for the timeline query. Its MVCC implementation means read queries never block writes, which is critical for a read-heavy social media workload."

**Operational realism.** Junior candidates propose architectures with six synchronized databases. Senior candidates acknowledge the operational cost of each component. The interviewer wants to see that you understand what it means to run a database in production --?" backups, replication lag, connection management, upgrade procedures, and incident response.

**Evolution thinking.** A database chosen for 1,000 users will differ from one chosen for 100 million users. The interviewer wants to see that you can describe how the database architecture evolves as the system grows, not just the end state.

## Level-by-Level Expectations

### L3/L4 (Early Career)

**Expectation:** You can describe the basic properties of relational databases and NoSQL databases. You know when to use each at a high level. You can list ACID and BASE properties.

**Sample answer to "Design Twitter's database":** "I would use Cassandra for tweets because Twitter has very high write throughput and tweets are keyed by tweet ID. I would use PostgreSQL for users and relationships because user data needs joins between followers and followees. For the timeline, I would precompute it and store it in Redis."

**What this shows:** You understand the basic split between read-heavy and write-heavy workloads. You have heard of the major database categories.

**Gap:** You did not ask about the specific query patterns (timeline construction), consistency requirements (freshness of follows), or scale (how many users, how many tweets per second). You also did not address the operational complexity of managing three databases.

### L5 (Senior)

**Expectation:** You systematically gather requirements before proposing a database. You articulate specific trade-offs between options and reference concrete implementation details. You consider operational implications and propose an evolutionary path.

**Sample answer to "Design Twitter's database":** "Before choosing a database, I need to understand the workload. Tweet writes are probably 5,000-10,000 per second at peak, and each tweet is roughly 1 KB with text, media links, and metadata. Tweet reads are mostly the home timeline, which needs to merge tweets from the last 500 follows ordered by time. User profiles are read-heavy with occasional updates. The follow graph is adjacency-list data.

"For tweets, I see two serious options. Cassandra's wide-column model stores tweets per user as a column family ordered by timestamp, which maps perfectly to the timeline query --?" fetch the last 200 columns for each followed user and merge in application code. But this requires a fan-out query on every timeline load. An alternative is PostgreSQL with partitioning on user_id and an index on created_at. PostgreSQL handles the range scan efficiently, it supports transactional integrity if we need it for tweet deletion or edit features, and it avoids introducing a new database system.

"For the follow graph, PostgreSQL handles this naturally with a follows table (follower_id, followee_id, created_at). This is a textbook relational join.

"The key decision is whether the write throughput for tweets exceeds what a single PostgreSQL primary can handle --?" roughly 5,000 writes per second. At Twitter's scale of millions of tweets per day, the answer is yes, so I would start with Cassandra for tweets and PostgreSQL for users and graphs. I would evaluate whether Cassandra's eventual consistency is acceptable for the timeline --?" users must see their own tweets immediately, so I would use read-your-writes consistency for the author and eventual consistency for other users.

"Operationally, I would use managed services where possible: Amazon Keyspaces for Cassandra and RDS for PostgreSQL. I would set up CDC from PostgreSQL to a secondary system for analytics."

**What this shows:** You gathered requirements before proposing. You compared concrete databases on specific mechanisms. You considered the operational burden. You acknowledged trade-offs. You proposed an evolutionary path.

### L6 (Staff)

**Expectation:** You present a framework for making the decision that generalizes to any database selection problem. You reference how real companies handle the same problem. You discuss failure modes specific to the chosen architecture. You consider cost, team size, and migration path.

**Sample additions to L5 answer:** "The framework I use for database selection has six dimensions: data model shape, query patterns, consistency requirements, scale trajectory, operational maturity of the team, and cost constraints. Let me walk through each for the Twitter workload...

"For the timeline fan-out, a common approach at scale is to precompute home timelines and store them in a distributed cache. Redis sorted sets with tweet IDs ordered by timestamp let you push new tweets to all followers' timelines in real-time. This moves the merge computation to write time, reducing read latency. The trade-off is write amplification: a celebrity with 10 million followers generates 10 million cache writes per tweet. Twitter solved this by only precomputing timelines for active users --?" the rest get the merge-at-read-time path.

"On operational failure modes: if I use Cassandra for tweets, the compaction storm failure mode is the biggest risk. At Twitter's tweet volume, SSTable compaction consumes significant I/O. I would monitor compaction throughput, schedule it during low-traffic periods, and pre-provision IOPs above peak write throughput.

"On cost: Cassandra's per-node cost with replication factor 3 means roughly $2,000 per node per month on AWS. Twitter-scale deployments need dozens of nodes. If the team is under 20 engineers, I would strongly consider whether managed alternatives like DynamoDB or Aurora can handle the workload with less operational overhead, even if the per-request cost is higher."

**What this shows:** Framework thinking. Reference to real implementations. Awareness of failure modes. Cost consciousness.

### L7/L8 (Principal/Director)

**Expectation:** You shape the organizational approach to database selection. You balance short-term pragmatism against long-term architecture. You articulate when NOT to use the framework and how to handle organizational resistance.

**Sample additions to L6 answer:** "The hardest part of database selection is not technical --?" it is organizational. The engineering team wants to use Cassandra because it is exciting. The VP wants to use a single Postgres instance because it is simpler. The database administrator knows MongoDB is the worst choice but does not say so in the meeting. As a principal engineer, my job is to create a decision process that surfaces the objective trade-offs and aligns the team on the decision criteria.

"I would run a database selection workshop. The outcome is not just a database name. It is a documented decision with: the workload assumptions, the alternatives considered, the trade-offs accepted, the migration plan, the rollback plan, and the criteria that would trigger a re-evaluation. This document survives engineering turnover and prevents the next team from re-litigating the decision.

"The other key decision at this level is when to accept a less-optimal database choice because the team cannot operate the optimal one. A Cassandra cluster that falls over from compaction storms is less reliable than a PostgreSQL instance that runs smoothly. The best database is the one your team can operate well."

### Common Mistake Patterns in Interview Answers

Learn from candidates who failed specific database questions:

**Mistake: Naming the wrong database family.** A candidate proposed Cassandra for an e-commerce system where every query required joins across orders, line items, and products. Cassandra is wide-column key-value store with no join support. The candidate should have recognized the relational query pattern and proposed PostgreSQL.

**Mistake: Ignoring the team context.** A candidate proposed a 20-node Cassandra cluster for a team of 3 engineers building an MVP. The proposal showed technical knowledge but zero operational awareness. The better answer: start with PostgreSQL or DynamoDB (managed), add complexity when the product proves it needs the scale.

**Mistake: No evolution plan.** A candidate proposed DynamoDB for a social media application and stopped there. When asked what happens at 10x scale, the candidate had no answer. The interviewer wanted to hear about DynamoDB's auto-scaling, DAX (DynamoDB Accelerator) for caching, Global Tables for multi-region, and the threshold where a relational sidecar might be needed for analytics queries.

**Mistake: Over-engineering.** A candidate proposed Redis + PostgreSQL + MongoDB + Elasticsearch + Kafka for a blogging platform with 100 users. The architecture looked impressive but was completely inappropriate for the scale. The candidate failed the "keep it simple" test that interviewers apply to senior roles.

**Mistake: Ignoring the data.** A candidate recommended MongoDB for a payment system without asking about consistency requirements. When questioned about ACID compliance, the candidate had to backtrack. The mistake was proposing a database before understanding the workload.

### Mock Interview Transcript --?" Strong Candidate

**Interviewer:** Design the database for a real-time collaborative document editor like Google Docs.

**Candidate:** Before I propose any database, I need to understand a few things. First, what are the read and write patterns? Document editing is write-heavy with frequent small updates (each keystroke is a write). Reads are less frequent --?" you load the document once and then receive updates via WebSocket. The key consistency requirement: a user should see their own edits immediately, and edits from collaborators should appear within a few hundred milliseconds. Conflicts must be resolved when two people edit the same paragraph simultaneously.

Given these requirements, I see three distinct data models within this system:

**Document storage:** The authoritative copy of each document. Structured metadata (title, owner, sharing settings) in one table. Document content stored as snapshots with a version number. This is a relational workload with moderate throughput (millions of documents, not billions). I would use PostgreSQL for this.

**Real-time editing state:** Current cursor positions, active collaborators, unsaved changes. This is ephemeral, high-frequency, and needs pub/sub for broadcasting changes to collaborators. I would use Redis for this --?" it handles the pub/sub pattern natively and can store ephemeral state with TTL.

**Operational transforms (OT) or conflict resolution:** Each edit operation (insert character at position N, delete characters M through N) needs to be stored and applied in order. This is an append-only sequence. I would use a time-series or log-structured approach here --?" either Kafka (if I need to replay the entire edit history) or a simple append-only table in PostgreSQL with an incrementing sequence number.

The database architecture would be:

1. PostgreSQL for authoritative document storage and metadata.
2. Redis for real-time collaboration state and pub/sub.
3. An append-only log (PostgreSQL table or Kafka topic) for edit operations, used for conflict resolution and history.

The key trade-off: I am using multiple databases for different parts of the workload. The operational complexity is justified because each database is optimized for its specific role. If the team is small (under 5 engineers), I would simplify by using PostgreSQL for everything except the real-time pub/sub (which needs Redis regardless), and add the append-only log as a PostgreSQL table.

**Interviewer:** How does PostgreSQL handle the edit log at Google Docs scale --?" billions of edits per day?

**Candidate:** At that scale, the append-only log table would be partitioned by document ID or time range. I would use PostgreSQL declarative partitioning on document_id (hash-based) combined with range partitioning on created_at. Old partitions (over 30 days) would be moved to a cheaper storage tier. The active partition set would be sized to fit in the buffer pool for fast append operations. If write throughput exceeds what a single PostgreSQL instance can handle (roughly 5,000 writes/second per partition), I would move the edit log to Kafka, which handles millions of writes per second natively. The migration trigger: when PostgreSQL's write latency for the edit log exceeds 10 milliseconds at p99.

## FAANG Framework for Answering

Use this six-step framework for any database selection question.

**Step 1: Clarify requirements (2 minutes).** Ask about each dimension before proposing anything:
- What is the data model? Structured, semi-structured, graph, blob?
- What are the read and write query patterns? Point lookups, range scans, joins, aggregations, full-text?
- What are the consistency requirements? Is eventual consistency acceptable? Are transactions needed?
- What is the scale? Data volume, read QPS, write QPS. Current and projected for 12 and 24 months.
- What is the team size and operational maturity? Can they operate a distributed database, or do they need managed services?
- What is the budget? Free tier, startup budget, enterprise budget?

**Step 2: Identify viable database families (1 minute).** Map requirements to database families:
- Structured data with joins and transactions: RDBMS (PostgreSQL, MySQL).
- Semi-structured documents with schema flexibility: Document store (MongoDB, Firestore).
- High-throughput key-value with no joins: Key-value store (DynamoDB, Redis).
- Graph-shaped data with traversals: Graph database (Neo4j, Neptune).
- Time-series data: Wide-column (Cassandra, Bigtable) or specialized (InfluxDB).
- Full-text search: Search engine (Elasticsearch, Solr).
- Need both ACID and horizontal scale: NewSQL (CockroachDB, Spanner).

**Step 3: Propose concrete databases with rationale (2 minutes).** Name specific products, not just families. State what you gain and what you sacrifice.

**Step 4: Describe the operational plan (1 minute).** How is this database deployed? Managed service or self-managed? What monitoring is needed? What does backup and recovery look like?

**Step 5: Identify failure modes and mitigations (1 minute).** What is the first thing that breaks? How do you detect it? How do you recover?

**Step 6: Outline the evolution path (1 minute).** How does this choice change at 10x scale? What triggers the migration to a different database?

## Phrase Bank

Use these phrases to sound like you have real production experience with database selection.

**On requirements gathering:**
- "Before I name a database, I need to understand the query patterns and consistency requirements."
- "Let me walk through the workload dimensions with you."
- "The most important constraint is usually the write throughput. Can we estimate that?"
- "Is strong consistency required here, or can reads see stale data for a few hundred milliseconds?"

**On trade-offs:**
- "PostgreSQL gives us joins and ACID, but we pay for it in write throughput. At this projected write volume, I think it still works for the first 18 months."
- "DynamoDB's operational simplicity is hard to beat, but the lack of joins means we need to denormalize data and manage consistency in application code."
- "Redis is incredibly fast for single-key lookups, but the data must fit in memory and range queries are essentially full-table scans."
- "CockroachDB provides the consistency model of a relational database with the horizontal scale of a NoSQL system. The cost is higher write latency due to the consensus protocol."

**On operational reality:**
- "The operational cost of running a Cassandra cluster --?" monitoring compaction, repairing nodes, managing gossip --?" is significant. I would only choose it if the throughput justifies it."
- "I always start with the simplest operational posture and add complexity only when measured bottlenecks prove it is necessary."
- "A managed database is almost always the right choice for teams under 10 engineers."

**On failure modes:**
- "The biggest risk with DynamoDB is a hot partition from a poorly chosen partition key. I would monitor partition utilization and alert if any partition exceeds 80 percent of its throughput."
- "Cassandra compaction storms can cause 10x latency spikes. I would set compaction throughput limits and monitor the pending compaction count."
- "Replication lag is the silent killer of read-replica architectures. I always set up replication lag alerts and design the application to tolerate stale reads."

**On evolution:**
- "I would start with a single PostgreSQL instance and add read replicas when the read-to-write ratio exceeds 10:1."
- "The threshold for migrating from a single-node database to a distributed one is roughly 5,000 writes per second."
- "I would choose a database that has a clear migration path to the next tier. PostgreSQL can scale up (bigger instance), scale out (read replicas), and scale distributed (Citus, CockroachDB)."
- "Start with the simplest architecture that works. Accept that you will migrate the database as you grow. Design the application to make that migration possible."

**On multi-tenancy and isolation:**
- "For multi-tenant systems, I would start with a shared database and Row-Level Security. Move large tenants to dedicated databases only when measured query performance degrades."
- "The noisy neighbor problem in a shared database is real. I would monitor query latency per tenant and alert if any tenant's queries take significantly longer than the median."

**On cost optimization:**
- "DynamoDB on-demand pricing is convenient but can be 5-10x more expensive than provisioned for predictable workloads. I would start with on-demand and switch to reserved capacity once the traffic pattern stabilizes."
- "The biggest database cost at scale is not compute or storage --?" it is the engineering team's time spent on operations. A more expensive managed database is often cheaper than a cheaper self-managed database when you factor in team cost."
- "Read replicas are cheap compared to the cost of re-architecting when you outgrow a single node. Add replicas early, even before the bottleneck materializes."

**On NoSQL skepticism:**
- "Document databases are a great fit when the data is naturally document-shaped and you do not need joins. If you find yourself implementing joins in application code, you should have used a relational database."
- "The development velocity advantage of schema-less databases is real for early-stage products. The cost is paid later when the schema becomes implicit in the application code and every developer must know the undocumented structure."
- "I have seen teams choose MongoDB because they were afraid of schema migrations. Six months later, they had a database with 10 different document shapes for the same entity type and no way to query across them."

**On selling the decision:**
- "The database is the hardest component to change. I want to invest 2 weeks upfront in a proof-of-concept rather than 6 months in a migration."
- "Let me document the trade-offs we are accepting with this choice so that when the bottlenecks appear, the team knows what to expect and when to revisit the decision."
- "The best database is the one your team can operate. I am not recommending Cassandra because I do not think we have the operational expertise to keep it healthy."

## Red Flags

**Red Flag 1: Naming a database without asking any requirements questions.** You look like you memorized answers instead of understanding trade-offs.

**Red Flag 2: Proposing a polyglot architecture for a startup with 1,000 users.** You look inexperienced --?" you do not know that operational complexity kills early-stage startups.

**Red Flag 3: Claiming one database is universally better than another.** "PostgreSQL is always better than MongoDB." This statement is false in both directions and signals a lack of nuance.

**Red Flag 4: Ignoring consistency requirements.** Assuming eventual consistency is fine for a payment system, or assuming strong consistency is needed for a social media like counter.

**Red Flag 5: Proposing DynamoDB for a workload that needs complex joins and aggregations.** DynamoDB is optimized for key-value access patterns. Proposing it for a reporting system signals you do not understand DynamoDB's data model.

**Red Flag 6: Ignoring the operational burden of your proposed architecture.** Proposing Cassandra (self-managed) for a two-person engineering team without acknowledging the operations cost.

**Red Flag 7: Saying "we can just add a cache layer" without specifying cache invalidation strategy.** Cache invalidation is one of the hardest problems in distributed systems. Hand-waving it as an afterthought is a red flag.

**Red Flag 8: Proposing a complete database migration without discussing the rollback plan.** Every production database migration goes wrong at some point. Not having a rollback plan is reckless.

**Red Flag 9: Assuming all data in a system can use the same database.** Complex systems have multiple data models and access patterns. The candidate who proposes one database for Twitter's tweets, users, timelines, and analytics has not thought deeply about the workload.

**Red Flag 10: Using buzzwords without substance.** Saying "we need eventual consistency and BASE properties" without explaining what that means for the application code.

**Red Flag 11: Proposing a single database for a workload that clearly has multiple data models.** If your system stores user profiles (document-shaped), financial transactions (relational), and analytics events (time-series), and you propose one database for all three, you have not analyzed the workload.

**Red Flag 12: Saying "we can just use an ORM to handle the database."** ORMs abstract away simple CRUD but leak badly for complex queries, performance tuning, and migrations. Proposing an ORM as a substitute for understanding the database is a sign of inexperience.

**Red Flag 13: Ignoring the network in a distributed database discussion.** Distributed databases are sensitive to network latency, bandwidth, and reliability. Proposing a multi-region active-active setup without discussing network latency and bandwidth costs shows a gap in operational awareness.

**Red Flag 14: Describing a database migration as "easy."** Real database migrations at scale involve schema mapping, data validation, downtime minimization, application code changes, performance regression testing, and rollback planning. Calling it easy signals inexperience.

**Red Flag 15: Proposing a cache for every database performance problem.** Caching is a powerful tool, but cache invalidation is hard, cache misses under load cause cascading failures, and caching write-heavy data is wasteful. Use caching for read-dominant, stale-tolerant workloads only.

**Red Flag 16: Not knowing the data size.** Proposing a specific database without asking about the expected data volume. A database that works well at 10 GB may fail at 10 TB, and vice versa.

## Practice Drills

**Drill 1 (5 minutes):** You are designing the database for a blogging platform like Medium. Authors write posts with text, images, and tags. Readers view posts and leave comments. The home page shows a personalized feed of recent posts from followed authors. What databases do you choose and why?

**Drill 2 (5 minutes):** Your IoT platform ingests sensor readings from 100,000 devices, each sending a reading every 30 seconds. Users query recent readings for a device, historical trends, and alerts when values exceed thresholds. What database do you choose?

**Drill 3 (10 minutes):** You inherit a system that uses MongoDB for everything --?" user profiles, orders, inventory, analytics, and full-text search. The system has 100,000 users and is growing at 10 percent per month. Queries are getting slow. Walk through your analysis and proposed changes.

**Drill 4 (10 minutes):** Your team of five engineers runs a SaaS application with 500 customers on a single PostgreSQL database. Your CTO says you should migrate to DynamoDB because "it scales infinitely." Walk through your response.

**Drill 5 (15 minutes):** Design the database architecture for a global video-conferencing platform like Zoom or Google Meet. Consider room metadata, participant presence, chat messages, recording storage, and billing data. Address multi-region latency, consistency across regions, and the difference between real-time data and business data.

**Drill 6 (15 minutes):** Your team is building a financial portfolio tracking application. Users connect their brokerage accounts, and the system imports transactions, calculates portfolio value, tracks gains and losses, and generates tax reports. What database do you use for each data type? What consistency guarantees do you need for portfolio calculations versus historical reporting? How do you handle the daily market close when thousands of users' data needs simultaneous recalculation?

**Drill 7 (15 minutes):** You are the database architect at a company that is migrating from a monolithic Rails application with PostgreSQL to a microservices architecture. Each microservice needs its own database. How do you decide which data stays in the shared PostgreSQL instance and which data moves to a new database per service? How do you handle data consistency across services during the migration? What about queries that currently join across what will become separate databases?
