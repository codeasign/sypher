# Choosing the Right Database --?" Challenges

Each challenge tests a specific skill from the database selection toolkit. Foundation challenges verify basic knowledge. Design challenges require you to apply the framework to realistic scenarios. Expert challenges push you to handle ambiguity, conflicting constraints, and operational reality.

## Tier 1: Foundation

These challenges take roughly 5 minutes each. They test whether you understand the basic properties of different database types and can map workloads to appropriate families.

### Challenge 1.1: Shopping Cart Database

You are building a shopping cart for an e-commerce website. The cart needs to:
- Store cart items per user (product ID, quantity, added timestamp).
- Support adding and removing items.
- Load the full cart on every page view.
- Maintain inventory counts accurately (no overselling).
- Support 10,000 users concurrently.

Which database type would you choose? What specific product? What trade-offs are you accepting?

**Model answer:** Use PostgreSQL. The shopping cart is structured data with clear relationships between cart items, products, and users. The inventory count needs ACID guarantees --?" you cannot oversell a product because two concurrent transactions see stale inventory. PostgreSQL's built-in row-level locking and serializable isolation handle this correctly.

If the scale grows to 100,000 concurrent users, you add read replicas and cache the cart content in Redis (keyed by session ID). The cache reduces database load for the most frequent operation --?" loading the cart on page view. For inventory, you still read from PostgreSQL to get the accurate count, possibly adding an inventory-specific caching layer with pessimistic locking for the checkout flow.

The trade-off: PostgreSQL's strict schema means changing the cart model requires a migration. If the product team frequently adds new cart features (gift wrapping, notes, split shipments), the migration overhead accumulates. You might address this by using PostgreSQL's JSONB column for flexible cart metadata while keeping the core relational structure in normal columns.

### Challenge 1.2: Session Store Selection

Your web application needs to store user sessions. Each session contains user ID, login timestamp, preferences, and an authentication token. Sessions are created on login, read on every page load, and invalidated on logout or expiry (24 hours). You expect 50 million sessions active at any time, with 500,000 reads per second and 10,000 writes per second.

Which database class fits best? List two specific products, one managed and one self-managed.

**Model answer:** Use a key-value store. Sessions are the textbook key-value workload: the lookup key is the session token (a high-cardinality string), the value is the session data, and the only query pattern is point read by key. No joins, no range scans, no aggregations.

The managed choice is DynamoDB with on-demand capacity. Create a table with session_token as the partition key, set TTL to 24 hours for automatic session expiry, and pay per request. DynamoDB handles the 500K reads/second without provisioning.

The self-managed choice is Redis Cluster with 16 nodes (32 GB each). Store sessions as Redis hashes keyed by session token. Enable eviction (allkeys-lru) as a safety net. Redis handles this workload on a cluster of modest size, and the operational cost is lower than DynamoDB at this read volume if you have the team to manage it.

The trade-off: DynamoDB costs more per request at this volume but requires zero operational effort. Redis is cheaper at scale but requires cluster management, failover configuration, and persistence tuning.

### Challenge 1.3: Index Strategy Design

You have a PostgreSQL table `orders` with columns: `order_id` (UUID PK), `user_id` (UUID, FK to users), `status` (text: pending, confirmed, shipped, delivered, cancelled), `total_amount` (decimal), `created_at` (timestamp), `shipped_at` (nullable timestamp). The table has 50 million rows and grows by 100,000 rows per day.

The most common queries are:
- Get all orders for a specific user ordered by created_at DESC (called 50 times per second).
- Get orders in a specific status created in the last 7 days for a reporting dashboard (called once per minute).
- Get a single order by order_id (called 200 times per second).
- Get the count of orders per status for today (called every 5 seconds for a real-time dashboard).

Design the index strategy. Which columns do you index? In what order? What are the trade-offs for write throughput?

**Model answer:**

Query 1 (orders by user, ordered by time): Create a composite index on `(user_id, created_at DESC)`. This is a covering index for the query --?" PostgreSQL can scan the index in order and return the rows without sorting. The index order matches the query's ORDER BY, so no additional sort step is needed.

Query 2 (status + date range): Create a composite index on `(status, created_at)` where status comes first to filter, then created_at to range-scan. However, this index is only useful for the reporting query, which runs once per minute. The write cost of maintaining this index (updates to created_at do not change, but status changes from pending to confirmed to shipped) means the index is updated 3 times per row over its lifetime. If status updates happen at high volume, this index creates meaningful write amplification.

Query 3 (order_id lookup): The primary key index on `order_id` handles this. UUID indexes are less efficient than sequential integer indexes because UUIDs are random and cause index page splits. Consider using a sequential ID or a UUID v7 (time-ordered) to reduce index fragmentation.

Query 4 (count by status for today): A materialized view refreshed every 5 seconds would be more efficient than scanning the table. Create a materialized view: `SELECT status, COUNT(*) FROM orders WHERE created_at >= now() - interval '1 day' GROUP BY status`. Refresh it concurrently every 5 seconds. The materialized view query scans a small portion of the table (one day's data) and returns instantly.

Trade-offs: The user_id + created_at index is essential --?" without it, Query 1 scans 50 million rows per execution. It adds roughly 20 percent to write I/O for INSERT operations. The status index is optional --?" if the reporting dashboard can tolerate 5-second delays, consider removing the index and using a materialized view instead. Every unnecessary index on this table adds roughly 100 MB of storage and 0.5 ms of write latency per insert.

### Challenge 1.4: CAP Classification

Classify each of the following databases as CP or AP during a network partition, and describe what happens to writes on the minority side of the partition: (a) PostgreSQL with streaming replication, (b) Cassandra, (c) MongoDB with write concern majority, (d) DynamoDB, (e) etcd.

**Model answer:**
- (a) PostgreSQL with streaming replication is CP if you configure synchronous replication and use the primary for all writes. During a partition, the primary continues to accept writes, but replicas cannot acknowledge the sync commit, so write latency increases (the primary waits for the replica's acknowledgment). If you are using asynchronous replication (the default), PostgreSQL is effectively CP for the primary side and the replica is a stale read target.
- (b) Cassandra is AP. Both sides of the partition accept writes. When the partition heals, last-writer-wins conflict resolution merges divergent data. Users on the minority side can continue writing.
- (c) MongoDB with write concern majority is CP. The minority-side replica set members stop accepting writes because the write concern cannot be satisfied (the majority is on the other side of the partition). Clients connected to the minority replica set see write errors.
- (d) DynamoDB is AP. Both sides of a partition continue to accept writes. Conflicts are resolved using last-writer-wins.
- (e) etcd is CP. etcd uses the Raft consensus protocol, which requires a majority of nodes to agree. During a partition, the minority side cannot form a quorum and stops accepting writes.

## Tier 2: Design

These challenges take roughly 15 minutes each. They test your ability to apply the database selection framework to a realistic scenario with competing constraints.

### Challenge 2.1: Real-Time Analytics Dashboard

A SaaS company needs a real-time analytics dashboard for their customers. The system ingests events (page views, clicks, form submissions) at 50,000 events per second. Each event is roughly 500 bytes. Customers query their data through a dashboard that shows counts, top pages, funnel conversions, and time-series charts. Queries must return in under 3 seconds for the current day's data. Historical data (older than 90 days) is queried less frequently and can take up to 30 seconds.

Design the database architecture. Consider the data ingestion path, the query path, and the data retention strategy.

**Model answer:**

This workload has two fundamentally different halves: the ingestion path (high-throughput writes, append-only) and the query path (ad-hoc aggregations, range scans on time). No single database is optimal for both.

**Ingestion path:** Use a stream processor (Apache Kafka) to buffer the events. Kafka handles 50,000 events/second on a 3-node cluster with appropriate partitioning. Each event is written to a topic partitioned by customer_id to enable parallel consumption.

**Real-time query path:** Use ClickHouse for the real-time dashboard queries. Stream the events from Kafka into ClickHouse using the Kafka table engine or a streaming connector like ClickHouse Kafka Connect. ClickHouse's columnar storage compresses events by 5-10x (events store well in columns --?" lots of repeated values for page paths, event types, user agents). The vectorized query engine computes counts and aggregations over 100 million events in under 1 second.

**Historical path:** After 90 days, move raw events from ClickHouse to cheaper object storage (S3, GCS) in columnar format (Parquet, ORC). ClickHouse can still query this data using its S3 table engine, but queries are slower (30 seconds vs 3 seconds). For very old data, only pre-computed daily aggregations are retained in ClickHouse as materialized views.

**Data retention strategy:** Raw events in ClickHouse for 90 days. Daily aggregations in ClickHouse for 2 years (stored as materialized views --?" 100x smaller than raw data). Raw events in S3 for 7 years for compliance, queryable on-demand.

**Key trade-offs in this design:**
- ClickHouse is excellent at aggregations but poor at point lookups and transactional workloads. You would not use it for the application's operational database.
- The architecture requires a streaming pipeline (Kafka), which adds operational complexity.
- Eventual consistency is inherent --?" the dashboard shows data within seconds of ingestion, not instantly.

### Challenge 2.2: Global Social Feed

Design the database for a social feed serving 100 million monthly active users. Each user creates 2 posts per day on average. Posts are 500 bytes (text + media URL). Each user follows 200 accounts on average. The home feed shows the 50 most recent posts from followed accounts. Feed load is the most frequent operation --?" users check it 10 times per day. Posts must appear in followers' feeds within 5 seconds of publishing.

Consider the fan-out problem: a celebrity with 10 million followers creates one post, and you need to deliver it to 10 million feeds.

**Model answer:**

The core challenge is the read-to-write ratio. Each write (post) must be visible to N followers. If N averages 200, each post generates 200 reads-worth of work. For celebrities with 10 million followers, that is 10 million reads-worth of work from one write. You must choose between fan-out on write (precompute feeds) and fan-out on read (compute feeds at query time).

**Fan-out on write (push model):** When a user creates a post, insert the post ID into the feed cache of every follower. Use Redis sorted sets keyed by user_id, with post_id as the member and created_at timestamp as the score. The home feed query is `ZREVRANGE user:{user_id}:feed 0 49` --?" a single Redis command returning the 50 most recent post IDs.

For celebrity users, the fan-out on write is impractical (10 million Redis writes per post). Use a hybrid approach: push to active followers (those who checked their feed in the last 7 days), and let inactive followers pull on next login.

**Storage for posts:** Use Cassandra or ScyllaDB for the post storage. Posts are naturally keyed by post_id and ordered by created_at. Cassandra's wide-column model stores posts per partition efficiently. The write throughput (100M users x 2 posts/day / 86,400 seconds = roughly 2,300 posts/second) is well within Cassandra's capabilities on a modest cluster.

**Storage for the follow graph:** Use PostgreSQL. The follow graph is a classic adjacency list: table follows (follower_id, followee_id, created_at) with composite indexes on both columns. This supports "who does user X follow" and "who follows user X" queries efficiently.

**Key trade-offs:** This design uses three databases (Cassandra for posts, Redis for feeds, PostgreSQL for follows), which adds operational complexity. The fan-out-on-write approach means feed delivery latency is near real-time but requires careful handling of celebrity accounts. The alternative (fan-out on read) would use fewer databases but increase feed load latency for users following many accounts.

### Challenge 2.3: Migration from DynamoDB to Relational

Your company's core product runs on DynamoDB. It powers a project management tool with users, projects, tasks, comments, and assignments. The DynamoDB schema uses a single-table design with sort keys for different entity types. As the product grew, the limitations have become painful:

- Cross-entity queries (show all tasks assigned to user X across all projects) require client-side joins and multiple round trips.
- Ad-hoc reporting requires exporting data to a separate analytics database.
- Schema changes require application-level migrations because DynamoDB has no schema enforcement.
- The team spends 30 percent of development time working around DynamoDB's access pattern limitations.

Design the migration to a relational database. Address the schema design, the migration strategy, and the rollback plan.

**Model answer:**

**Target schema:** Use PostgreSQL with the following table structure:
- `users` (id, name, email, created_at)
- `projects` (id, name, owner_id FK, created_at)
- `project_members` (project_id FK, user_id FK, role)
- `tasks` (id, project_id FK, title, description, status, assignee_id FK, created_at, updated_at)
- `comments` (id, task_id FK, author_id FK, body, created_at)
- `task_assignments` (task_id FK, user_id FK, assigned_at)

Add indexes on foreign keys, task status, and created_at for common query patterns.

**Migration strategy (expand-migrate-contract):**

Phase 1 --?" Expand (2 weeks): Set up PostgreSQL alongside DynamoDB. Modify the application to write new data to both databases. Existing data is still read from DynamoDB. This is the riskiest phase --?" a dual-write failure (one database accepts, the other rejects) causes divergence. Use a write-through pattern: write to PostgreSQL first, then to DynamoDB. If the DynamoDB write fails, log the error and reconcile later.

Phase 2 --?" Migrate (3 weeks): Backfill all existing data from DynamoDB to PostgreSQL. Run validation queries: for every entity in DynamoDB, verify it exists in PostgreSQL with the correct field values. Parallel-read a random 0.1 percent sample and compare results. Switch read traffic to PostgreSQL gradually --?" first 10 percent, then 50 percent, then 100 percent --?" while monitoring error rates and latency.

Phase 3 --?" Contract (1 week): Stop writing to DynamoDB. Remove the DynamoDB write code from the application. Keep DynamoDB data for 30 days as a rollback target. After 30 days with no incidents, drop the DynamoDB tables.

**Rollback plan:** If PostgreSQL latency exceeds DynamoDB latency by more than 20 percent during Phase 2, or if error rates exceed 0.1 percent, flip the read traffic back to DynamoDB. The write path remains dual so no data is lost during the rollback. The rollback is a configuration change, not a code deployment.

**Key risks and mitigations:**
- Data divergence during dual-writes: Implement a reconciliation job that runs daily during Phase 1 and flags discrepancies.
- Query performance regression: Profile the top 10 queries in staging with the PostgreSQL schema and compare latency to DynamoDB before cutting over production traffic.
- Application code complexity during dual-writes: Abstract the data access layer behind an interface so the dual-write logic is contained in one module.
- Team knowledge of PostgreSQL: Run a training session before the migration starts. The team needs to understand PostgreSQL-specific features (transactions, connection pooling, indexing) that differ from DynamoDB.

### Challenge 2.4: Multi-Tier Data Lifecycle

You are designing a video surveillance system. Cameras record 24/7 video streams at 1080p (roughly 10 Mbps per camera). You have 10,000 cameras. Video must be retained for 24 hours for immediate review, 30 days for incident investigation, and 1 year for compliance archiving.

Users query the system in three modes:
- **Live view:** View a camera's current feed with under 2-second delay.
- **Instant replay:** View the last 5 minutes from a specific camera with under 3-second seek time.
- **Investigation:** Search for motion events across multiple cameras in a specific time window, then view the corresponding video clips.

Design the database architecture for metadata (camera info, motion event timestamps, retention policies) and the storage architecture for the video data itself. What databases serve each layer?

**Model answer:**

This system has three distinct data types with very different access patterns and scalability requirements.

**Video data (blob storage):** The primary storage challenge is raw throughput. 10,000 cameras at 10 Mbps generates 100 Gbps of continuous write throughput. That is 4.5 PB of raw video per day. The only practical storage for this volume is object storage (S3, GCS, Azure Blob) with segmented video files.

Divide each camera's stream into segments of 1 minute (roughly 75 MB per segment). The camera writes each segment as a separate S3 object named by camera_id and timestamp: `camera/0001/2024-06-01/143000.mp4` for the segment starting at 14:30:00 UTC. This naming convention enables efficient listing for replay queries (list objects with prefix `camera/0001/2024-06-01/` to get all segments for a specific hour).

**Metadata (relational --?" PostgreSQL):** Camera configuration, user permissions, retention policies, and motion event metadata. This workload is low throughput (tens of operations per second) and benefits from PostgreSQL's relational model for joins between cameras, events, and users. PostgreSQL's declarative partitioning by camera ID group supports the range queries for investigation ("find all motion events on cameras 1000-2000 between 2pm and 3pm").

**Motion events (time-series --?" Cassandra or TimescaleDB):** Motion detection generates events: camera_id, timestamp, motion_duration, bounding_box coordinates. This is a time-series workload --?" append-only writes, range scans by time for investigation queries, and data retention with automatic expiration. Cassandra's wide-column model with camera_id as partition key and timestamp as clustering column handles millions of events per day with predictable query performance. TimescaleDB on PostgreSQL provides time-series-specific optimizations (hypertables, automatic partitioning, compression) if you prefer to keep everything in the PostgreSQL ecosystem.

**Hot video cache (Redis or in-memory):** The "instant replay" feature (last 5 minutes) is the most latency-sensitive query. Store the last 5 minutes of each camera's video as a rolling buffer in an in-memory cache. Each camera stores roughly 37.5 MB (5 minutes at 10 Mbps). For 10,000 cameras, that is 375 GB of cache --?" feasible with a Redis cluster of 16 nodes (32 GB each). Use Redis Streams for the video segment index and store segments in S3 with pre-signed URLs for direct access.

**Data lifecycle automation:**
- Hot tier (24 hours): In-memory cache for instant replay. S3 for all segments with STANDARD storage class.
- Warm tier (30 days): S3 STANDARD_IA (Infrequent Access) storage. Drop from cache. Retain motion event metadata in PostgreSQL/TimescaleDB.
- Cold tier (1 year): S3 GLACIER or DEEP_ARCHIVE for compliance. Only metadata retained in the database for search.

**Key decision points:**
- Why not a single database? The workload diversity --?" continuous blob writes, time-series event ingestion, structured metadata queries, and latency-sensitive replay --?" requires specialized systems. No single database handles all of them efficiently.
- Why S3 over HDFS or local storage? S3's virtually unlimited capacity and per-object retention policies match the video lifecycle perfectly. Local storage would require manual capacity management and data migration between tiers.
- Why PostgreSQL for metadata? The investigation queries need joins: find cameras in a specific location, then query events for those cameras in a time range, then retrieve the relevant video segments. PostgreSQL handles these joins naturally.

## Tier 3: Expert

These challenges take roughly 30 minutes each. They test your ability to handle ambiguous requirements, conflicting stakeholder priorities, and extreme scale.

### Challenge 3.1: Multi-Tenant SaaS with Workload Diversity

You are the database architect for a multi-tenant SaaS platform serving 10,000 businesses. Each business uses your platform for different purposes --?" some use it as a CRM, some as a project management tool, some as an inventory system. The data models vary significantly between tenants. Some tenants have 10 users and store 100 MB of data. Others have 10,000 users and store 100 GB.

The system must provide:
- Strong consistency within a tenant's data (tenant A's writes are immediately visible to tenant A's reads).
- Isolation between tenants (tenant A cannot read tenant B's data).
- 99.99 percent availability.
- The ability for tenants to customize their data schema (add custom fields, create custom entities) without database migrations.
- Reporting queries that join across a tenant's custom fields with standard fields.

Design the database architecture. Consider the multi-tenancy model (separate database per tenant vs. shared database with tenant isolation), the custom schema requirement, and the scale range from 10 MB tenants to 100 GB tenants.

**Discussion guide:**

This problem has no perfect solution. Every approach involves significant trade-offs. The discussion should cover at least these dimensions:

**Multi-tenancy model:** Separate database per tenant provides the strongest isolation (a noisy tenant cannot affect others) and allows per-tenant backup and restore. However, managing 10,000 databases is operationally expensive --?" connection pooling, schema migrations, monitoring all multiply by 10,000.

A shared database with row-level tenant isolation (every table has a tenant_id column, and Row-Level Security (RLS) in PostgreSQL enforces tenant isolation) is simpler operationally but risks the "noisy neighbor" problem --?" one tenant's query can consume resources and degrade performance for other tenants.

A hybrid approach: use a shared database for the 9,000 small tenants (under 1 GB each) and separate databases for the 1,000 large tenants. The operational complexity is 1,000 databases, not 10,000, and the small tenants benefit from shared resource efficiency.

**Custom schema:** The requirement for tenant-customizable schemas rules out a fully normalized relational model (you cannot add columns to every table for every tenant's custom field). The standard approach is the Entity-Attribute-Value (EAV) pattern, but EAV queries are painful for reporting.

A better approach: store custom data as JSONB columns in PostgreSQL. Standard fields stay in normal columns with indexes. Custom fields go into a JSONB column. PostgreSQL's JSONB supports indexing (GIN indexes on JSONB paths) and querying (jsonb_path_query, @> containment operator). For reporting, materialized views can extract common custom fields into normal columns daily.

**Scale range:** The 100 GB tenant needs a different database approach than the 10 MB tenant. For tenants under 10 GB, a shared PostgreSQL instance with RLS handles everything. For tenants over 10 GB, consider moving them to a dedicated instance where you can tune the configuration for their specific workload. This is known as "tenant placement" --?" a placement service decides which database infrastructure serves each tenant based on their size, growth rate, and activity level.

**Consistency and availability:** Strong consistency within a tenant is achievable with a single write master per tenant (or per shard of tenants). The 99.99 percent availability target suggests multi-AZ deployment with automatic failover. For the shared database, this means RDS Multi-AZ or a PostgreSQL replication setup. For dedicated instances, each tenant's database has its own failover configuration.

**Key tensions to explore in your response:**
- Custom schema versus query performance: JSONB is flexible but slower than normal columns for indexed queries.
- Tenant isolation versus operational simplicity: Separate databases are cleaner but harder to manage at 10,000 instances.
- Reporting on custom fields: Materialized views add latency (daily refresh) but enable efficient reporting queries.
- The right time to move a tenant from shared to dedicated.

### Challenge 3.2: You Are the Database Advisor

The CTO of a 50-person startup comes to you with this proposal: "We are building a next-generation social network for gamers. We have raised 20 million dollars. We are going to use Neo4j as our primary database because the social graph is the core of our product. We are also going to use Elasticsearch for search, Redis for session storage, and Apache Kafka for event streaming. We have 2 backend engineers."

Evaluate this proposal. What is right about it? What concerns do you have? What would you change?

**Discussion guide:**

This challenge tests your ability to evaluate a concrete proposal and give actionable advice, not just criticize. The CTO has thought about the problem --?" graph queries for a social network, search for discoverability, caching for performance, streaming for events. But the proposal has several problems that a more experienced engineer would catch.

**What is right:**
- Using a graph database for the social graph is technically appropriate. Friend-of-friend queries, recommendation traversals, and community detection are graph workloads. Neo4j handles these far more efficiently than a relational database with recursive CTEs.
- Elasticsearch for search is a mature choice.
- Redis for session storage is standard.
- Kafka for event streaming is appropriate for a real-time social platform.

**What is concerning:**
- Five database systems for a 50-person startup with 2 backend engineers is extreme. Each database requires monitoring, backup, upgrade procedures, and operational knowledge. The team will spend more time operating databases than building product features.
- Neo4j as the primary database is the highest-concern choice. Graph databases are excellent for traversal queries but poor at transactional workloads, bulk operations, and simple CRUD. User profiles (name, bio, avatar URL) are not graph data --?" they are structured documents. Using Neo4j for CRUD data adds unnecessary complexity.
- The proposal lacks a relational database entirely. User authentication, billing, content moderation, and reporting all need SQL with ACID guarantees. These workloads map poorly to graph databases.

**What I would change:**

1. Add PostgreSQL as the primary operational database. It handles user profiles, authentication, billing, content, and metadata. PostgreSQL can also handle social graph queries up to a moderate scale (recursive CTEs for friend-of-friend queries, indexed joins for follower relationships).

2. Use Neo4j only for graph-specific queries at scale. If PostgreSQL's graph queries become slow at 1 million users, add Neo4j as a read-replicated graph index. Stream data from PostgreSQL to Neo4j via CDC. Neo4j handles the recommendation engine and friend-of-friend traversals. PostgreSQL remains the source of truth.

3. Reduce the stack to PostgreSQL + Redis + Elasticsearch for the MVP. Kafka and Neo4j are added only when measured traffic proves PostgreSQL is the bottleneck. This reduces the operational burden from 5 systems to 3.

4. Hire at least one engineer with database operations experience before deploying to production. The CTO's plan assumes the 2 backend engineers can learn and operate 5 database systems. This is unrealistic for a 50-person startup.

**The broader lesson:** This challenge illustrates the most common database mistake at early-stage companies --?" over-engineering the database architecture based on future scale rather than current needs. The CTO is planning for the scale of a company that has not yet proven product-market fit. The right approach is to build with the simplest architecture that works now, design a clean migration path, and add complexity when the data proves it is necessary.

### Challenge 3.3: Migrating from Cloud to Self-Managed

Your company has outgrown DynamoDB for its core operational workload. The monthly DynamoDB bill is $80,000 for a single table with 10 TB of data, 50,000 reads per second, and 20,000 writes per second. The workload is relational in nature --?" orders, line items, shipments, returns --?" but was shoehorned into a single DynamoDB table using the single-table design pattern with composite sort keys for different entity types.

The CTO wants you to migrate to a self-managed Cassandra cluster running on EC2 to reduce the monthly cost to $15,000. Evaluate this proposal, design the migration, and identify the risks.

**Discussion guide:**

**Cost analysis:** The CTO's cost estimate of $15,000 per month for Cassandra is plausible for a 12-node cluster (6 TB per node, replication factor 3, using i3en.2xlarge instances at roughly $1,200/month each). However, the cost comparison is incomplete. DynamoDB's $80,000 includes zero operations cost --?" no database engineers, no backup management, no patching, no failover testing. Cassandra's $15,000 is just the infrastructure cost. Add one senior database engineer at $30,000/month (fully loaded), and the real cost is $45,000/month. Add the migration cost (6 months of engineer time), and the first year cost is $540,000 infrastructure plus $360,000 engineering time: $900,000 versus DynamoDB's $960,000. The savings are $60,000 in the first year --?" not the dramatic reduction the CTO expects.

**Data model mismatch:** The single-table DynamoDB design pattern uses composite sort keys to represent different entity types (e.g., sort key "ORDER#12345" or "LINEITEM#12345#001"). Cassandra's data model is fundamentally different --?" it uses partition keys and clustering columns. The migration requires redesigning the schema for Cassandra's access patterns. This is not a simple export-import; it is a re-architecture.

**Write throughput:** The workload is 20,000 writes per second. Cassandra handles this on a 12-node cluster with ease (roughly 1,700 writes per second per node, well within Cassandra's 10,000 writes per second per node capacity). Read throughput of 50,000 reads per second is also manageable with proper caching and bloom filter configuration.

**Operational maturity:** The company is migrating FROM a managed service TO a self-managed database. This is the opposite of the typical migration path. The team must learn: Cassandra's gossip protocol, node repair, compaction tuning, hinted handoff, read repair, consistency level configuration, and backup/restore procedures. If the team does not have this expertise, the migration increases operational risk significantly.

**Migration strategy:**
1. Set up the Cassandra cluster alongside the DynamoDB table. Dual-write all data to both systems. This phase requires careful error handling: if Cassandra write fails, the DynamoDB write must still succeed, and the Cassandra write must be retried asynchronously.
2. Backfill all existing data from DynamoDB to Cassandra. Use DynamoDB's Scan operation (paginated) and bulk-load into Cassandra. This takes days for 10 TB.
3. Validate data consistency between the two systems. Sample-based comparison with automatic flagging of discrepancies.
4. Migrate read traffic gradually: 10 percent to Cassandra, then 50 percent, then 100 percent, monitoring latency and error rates at each step.
5. Stop writing to DynamoDB after 2 weeks of stable operation. Delete the DynamoDB table after 30 days.

**Key risks:**
- Cassandra compaction storms during the backfill phase, causing latency spikes for the dual-write path.
- Data divergence during dual-writes if the reconciliation mechanism fails.
- Application code complexity from supporting two database backends simultaneously.
- Team knowledge gap: if the only Cassandra expert leaves, the operational burden falls on engineers who do not understand Cassandra's failure modes.
- Vendor lock-in reversal: the company is moving from a managed service that requires no operational expertise to a self-managed system that demands it. This is a strategic decision, not just a cost decision.

**Recommendation for the CTO:** Before committing to the migration, run a 2-week proof of concept with a smaller workload (1 TB, 5,000 writes/second). Measure the operational overhead. Document the procedures. Estimate the engineering time required to maintain the Cassandra cluster long-term. Present the true cost --?" infrastructure + engineering + training --?" rather than just the EC2 instance cost. If the true cost is within 20 percent of DynamoDB's cost, the migration may not be worth the operational risk.
