# Understanding Choosing the Right Database

## The Late-Night Pager Dilemma

It was 2:47 AM when Sarah's phone lit up with a PagerDuty alert. The startup's monolith database — a PostgreSQL instance they had used since day one — was hitting 99% CPU. Connection pools were exhausted. Queries that ran in 200 milliseconds six months ago were crawling past 30 seconds. The user base had grown from 500 beta testers to 85,000 paying customers, and the schema they had celebrated as "elegant" now looked like a Jackson Pollock painting of migrations, polymorphic associations, and JSONB columns used as dumping grounds.

Sarah was the fourth engineer to join the team, back when they had twelve people and a shared belief that "we'll figure out the database later." Later had arrived. The CTO, pacing behind her, kept asking the same question: "Should we switch to MongoDB? Or Cassandra? Or — I heard about this thing called CockroachDB?"

The team had fallen into a classic trap: treating database selection like a religious war instead of a systematic engineering decision.

This page gives you a mental model for making that decision systematically. You won't get a single answer. You'll get a framework for finding the right answer for your specific context.

## The Insight: Your Database Is a Toolbox, Not a Religion

Imagine walking into a woodworking shop. The craftsman has twenty different chisels, a dozen planes, seven saws, and three types of clamps. You ask: "Which tool should I use?" The craftsman looks at the piece of wood in your hand and says: "Depends on what you're making."

Databases are the same. There is no best database. There is only the best database for your particular workload, data shape, consistency needs, and operational constraints.

## Core Vocabulary: Decision Criteria You Need to Know

Before you can choose, you need vocabulary to describe your requirements. These are the dimensions along which databases differ.

### Data Model

This is the shape of your data as it naturally exists.

| Data Shape | Description | Natural Database |
|---|---|---|
| Tabular / Relational | Rows and columns with foreign keys | PostgreSQL, MySQL |
| Document / Hierarchical | Nested objects with varying fields | MongoDB, Couchbase |
| Key-Value | Opaque blobs accessed by unique key | Redis, DynamoDB |
| Wide Column | Sparse rows with dynamic columns | Cassandra, HBase |
| Graph | Nodes and edges with traversals | Neo4j, DGraph |
| Time-Series | Data points indexed by timestamp | InfluxDB, TimescaleDB |
| Vector | Embeddings with similarity search | Pinecone, Qdrant, pgvector |
| Search / Text | Full-text indexes and faceted search | Elasticsearch, Meilisearch |

### Consistency Model

What do you see when you read data after a write?

- **Strong Consistency** — Every read returns the most recent write. Higher latency, lower availability during partitions (CP in CAP theorem).
- **Eventual Consistency** — Reads may return stale data briefly. Higher availability and lower latency (AP in CAP theorem).
- **Read-Your-Writes** — A session sees its own writes. Others may see stale data for a window.
- **Causal Consistency** — Causally related writes are seen in order by all readers.

### Transaction Isolation

What anomalies are possible when transactions run concurrently?

| Level | Dirty Read | Non-Repeatable Read | Phantom Read | Write Skew |
|---|---|---|---|---|
| Read Uncommitted | Possible | Possible | Possible | Possible |
| Read Committed | Prevented | Possible | Possible | Possible |
| Repeatable Read | Prevented | Prevented | Possible | Possible |
| Snapshot Isolation | Prevented | Prevented | Possible | Possible |
| Serializable | Prevented | Prevented | Prevented | Prevented |

### Access Patterns

How does your application interact with the database?

- **Point Reads** — Single record by primary key. Most databases excel at this.
- **Range Scans** — Records where a key falls in a range. B-tree indexes support this; hash-based systems do not.
- **Full-Text Search** — Find documents matching keywords with relevance scoring. Specialized capability.
- **Aggregations** — Sums, averages, counts over groups. OLAP and columnar stores excel here.
- **Joins** — Combine data from multiple tables. Relational databases win; NoSQL requires application-level joins.
- **Graph Traversals** — Follow edges from a node. Exponentially faster in graph databases beyond two hops.
- **Time-Based Window Queries** — Fetch data within a time range. Time-series databases have specialized indexes.

### Operational Characteristics

What does it cost to run this database in production?

- **Throughput** — Writes and queries per second. Cassandra and DynamoDB handle hundreds of thousands. Single-node SQL databases cap in the low thousands.
- **Latency P99** — In-memory stores (Redis) achieve sub-millisecond P99. Disk-based stores range from 1-50ms.
- **Replication Model** — Single leader (simple consistency, limited write scaling), multi-leader (conflict resolution), leaderless (highest availability, complex consistency).
- **Sharding** — Does the database distribute data automatically (MongoDB, Cassandra) or require manual effort?
- **Backup and Restore** — Consistent backups without downtime? Restore time for multi-terabyte databases?
- **Migration Tooling** — Relational databases have mature tools (Alembic, Flyway). Document databases are schema-less but still need data migrations.

### Ecosystem and Team

The most overlooked dimension.

- **Driver Maturity** — Production-quality, idiomatic driver for your language?
- **Observability** — Prometheus exporters, Datadog integrations, built-in dashboards?
- **Hosted vs Self-Managed** — Does your team have the operational expertise to run this database?
- **Community Support** — Active community? Commercial backing? Consulting ecosystem?

## How It Works: A Step-by-Step Decision Framework

### Step 1: Characterize Your Data Shape

Data shape is the most fundamental constraint. Pick the wrong data model and everything else gets harder.

**Relational data** has fixed attributes per entity, well-defined relationships (one-to-many, many-to-many), and data integrity enforced through foreign keys. Most queries join across multiple entities. Example: an e-commerce order with a customer, shipping address, line items, and payments.

**Document data** contains nested, heterogeneous records with evolving schemas. Relationships are embedded rather than referenced. Example: a product catalog where electronics have wattage, clothing has size and fabric, and books have ISBN.

**Graph data** derives value from connections between entities. Queries traverse paths of variable length. Example: fraud detection finding whether two accounts are connected through a chain of transactions and devices.

**Time-series data** is append-heavy and rarely updated. Queries are time-range based. Old data is less valuable than recent data. Example: server metrics ingested at 10,000 data points per second.

```python
def categorize_data_shape(sample_data, relationships, access_patterns):
    has_varying_fields = any(
        len(r.keys()) != len(sample_data[0].keys()) for r in sample_data[1:]
    )
    needs_traversal = any("traverse" in p for p in access_patterns)
    is_append_only = all(p in ("insert", "range_scan") for p in access_patterns)

    if needs_traversal:
        return "graph"
    if is_append_only:
        return "time_series"
    if has_varying_fields:
        return "document"
    return "relational"
```

### Step 2: Determine Consistency and Transaction Requirements

Ask these questions:

1. **Can a user ever see stale data?** If never, you need strong consistency. Candidates narrow to relational databases and CP NoSQL systems (DynamoDB strongly consistent reads, MongoDB majority read concern).

2. **Can concurrent transactions interfere?** If multiple users read then write the same record, you need transaction isolation. Examples: booking the last seat on a flight, transferring money, decrementing inventory.

3. **Can you tolerate conflict?** In multi-leader systems, concurrent writes can happen. Resolution strategies: last-write-wins (simple, may lose data), CRDTs (automatic convergence, complex), or application-level resolution.

```python
from enum import Enum, auto

class ConsistencyLevel(Enum):
    STRONG = auto()
    READ_COMMITTED = auto()
    EVENTUAL = auto()
    CAUSAL = auto()

def needed_consistency(has_financial: bool, has_booking: bool,
                       has_sessions: bool, has_analytics: bool):
    if has_financial:
        return ConsistencyLevel.STRONG
    if has_booking:
        return ConsistencyLevel.READ_COMMITTED
    if has_sessions:
        return ConsistencyLevel.CAUSAL
    return ConsistencyLevel.EVENTUAL
```

### Step 3: Map Your Access Patterns

Your access patterns are your queries — both current and anticipated.

**Point reads** (get by ID) — Almost every database handles these well. Redis: under 1ms. PostgreSQL: 1-5ms. Cassandra: 5-20ms. If point reads dominate, an in-memory cache (Redis) on top of your durable store is common.

**Range scans** (records where timestamp > X) — Relational databases handle this with B-tree indexes. Key-value stores without range capability cannot do efficient range scans.

**Joins** (order + customer + line items in one query) — Relational databases have optimized join algorithms. Document databases discourage joins — you either embed or do multiple queries. Graph databases treat joins as traversals.

**Aggregations** (total revenue by month) — Relational GROUP BY works for moderate volumes. For large-scale aggregations, columnar databases (ClickHouse, Redshift) are needed. Document databases perform poorly on GROUP BY equivalents.

**Full-text search** (find products matching "wireless headphones") — PostgreSQL offers basic FTS (tsvector). For serious search, Elasticsearch or Meilisearch are purpose-built.

```python
def score_access_pattern_match(candidates, patterns):
    caps = {
        "postgresql":     {"point_read": 10, "range_scan": 9, "join": 10,
                           "aggregation": 8, "full_text": 7, "graph": 3},
        "mongodb":        {"point_read": 10, "range_scan": 8, "join": 3,
                           "aggregation": 6, "full_text": 7, "graph": 2},
        "cassandra":      {"point_read": 9,  "range_scan": 8, "join": 0,
                           "aggregation": 3, "full_text": 0, "graph": 0},
        "redis":          {"point_read": 10, "range_scan": 3, "join": 0,
                           "aggregation": 2, "full_text": 0, "graph": 0},
        "elasticsearch":  {"point_read": 8,  "range_scan": 7, "join": 0,
                           "aggregation": 9, "full_text": 10, "graph": 0},
        "neo4j":          {"point_read": 7,  "range_scan": 4, "join": 0,
                           "aggregation": 3, "full_text": 0, "graph": 10},
    }
    scores = {db: sum(caps.get(db, {}).get(p, 0) for p in patterns)
              for db in candidates}
    return dict(sorted(scores.items(), key=lambda x: -x[1]))
```

### Step 4: Estimate Scale Requirements

Scale is deceptive. Distinguish between:
- **What you need today** (the actual load)
- **What you need in 12 months** (reasonable projection)
- **What you dream about** (pitch-deck scale that may never arrive)

Choose for the 12-month projection. Optimizing for the pitch deck is premature optimization. Optimizing for today is short-sighted.

```python
from dataclasses import dataclass

@dataclass
class ScaleRequirements:
    writes_per_second: int
    reads_per_second: int
    total_data_gb: int
    geo_regions: int

def classify_scale_tier(req: ScaleRequirements) -> str:
    if req.geo_regions > 1:
        return "global"
    if (req.writes_per_second > 50000 or req.reads_per_second > 200000
        or req.total_data_gb > 10000):
        return "hyperscale"
    if (req.writes_per_second > 5000 or req.reads_per_second > 20000
        or req.total_data_gb > 500):
        return "medium"
    return "small"
```

| Tier | Candidate Databases |
|---|---|
| Small | PostgreSQL, MySQL, SQLite, MongoDB |
| Medium | PostgreSQL, MySQL, MongoDB, Redis + SQL |
| Hyperscale | Cassandra, DynamoDB, Spanner, CockroachDB, ScyllaDB, ClickHouse |
| Global | Spanner, CockroachDB, Cassandra, DynamoDB Global Tables |

### Step 5: Assess Operational Reality

A technically superior database your team cannot operate is worse than a mediocre database that runs smoothly.

1. **Who will be on call for this?** If the answer is "the same two people running PostgreSQL," that constrains you.
2. **What is the team's existing expertise?** A team that knows PostgreSQL deeply can handle most workloads well. The cost of learning a new database often outweighs the theoretical benefit.
3. **Is there a managed service?** For most teams, yes. Managed services (RDS, Aurora, MongoDB Atlas, Cassandra Astra) remove most operational burden.
4. **How critical is this data?** If losing one transaction is catastrophic, you need a mature database with point-in-time recovery and WAL archiving.
5. **What is your backup RTO/RPO?** Restoring within 5 minutes of failure in 30 minutes requires significant infrastructure.

### Step 6: Build a Shortlist and Prototype

You should now have 2-3 candidates. Validate with real data and real queries.

**Prototype:**
1. The three most complex queries — rewrite them in the candidate's query language.
2. The write path under load — can the database ingest data at 3x expected peak?
3. The worst-case consistency scenario — read data immediately after writing.
4. The disaster recovery scenario — delete the data directory and recover.

```python
import time, statistics

def benchmark_writes(db_client, num_writes=10000):
    latencies = []
    start = time.monotonic()
    for i in range(num_writes):
        t0 = time.monotonic()
        db_client.insert({"id": i, "value": f"record_{i}", "ts": time.time()})
        latencies.append(time.monotonic() - t0)
    elapsed = time.monotonic() - start
    return {
        "throughput_wps": num_writes / elapsed,
        "avg_latency_ms": statistics.mean(latencies) * 1000,
        "p99_latency_ms": statistics.quantiles(latencies, n=100)[-1] * 1000,
    }

def benchmark_reads(db_client, num_reads=10000):
    latencies = []
    for i in range(num_reads):
        t0 = time.monotonic()
        db_client.find_one({"id": i})
        latencies.append(time.monotonic() - t0)
    return {
        "avg_latency_ms": statistics.mean(latencies) * 1000,
        "p99_latency_ms": statistics.quantiles(latencies, n=100)[-1] * 1000,
    }
```

### Step 7: Account for Growth Vectors

Databases are not static choices. Consider:

- **Data volume growth** — If 10x growth is expected, does the database scale vertically or horizontally? Vertical is simpler but has hard limits.
- **Query complexity growth** — Startups begin with CRUD and add analytics, search, and reporting later. Does the database support these patterns?
- **Team growth** — Document databases with flexible schemas become maintenance burdens without strong conventions. Relational databases enforce structure through the schema.
- **Feature pivot risk** — Schema flexibility (NoSQL) helps with pivots. Schema enforcement (SQL) prevents data corruption.

### Step 8: Plan the Migration

The decision is never final. Plan for the possibility that you will migrate.

**When to decide:**
- Greenfield project: before writing data access code
- Before a major feature requiring new access patterns
- When operational pain exceeds migration pain
- After a growth milestone (10x re-evaluate)

**Migration phases:**
1. **Dual-write** — Write to both old and new databases. Compare results.
2. **Read from new** — Route reads to the new database. Keep writes going to both.
3. **Cutover** — Switch all traffic to the new database. Old one as fallback.
4. **Decommission** — After confirming stability, turn off the old database.

```python
class DualWriteManager:
    def __init__(self, old_db, new_db, log_func=print):
        self.old = old_db
        self.new = new_db
        self.log = log_func
        self.divergence_count = 0

    def write(self, collection, document):
        old_result = self.old.insert(collection, document)
        new_result = self.new.insert(collection, document)
        if old_result != new_result:
            self.divergence_count += 1
            self.log(f"DIVERGENCE in {collection}: old={old_result}, new={new_result}")
        return new_result

    def read(self, collection, query):
        result = self.new.find_one(collection, query)
        if result is None:
            result = self.old.find_one(collection, query)
        return result

    def health_check(self):
        return {
            "old_count": self.old.count_all(),
            "new_count": self.new.count_all(),
            "diff": self.old.count_all() - self.new.count_all(),
            "divergences": self.divergence_count,
        }
```

## Variants: Decision Frameworks for Different Contexts

### The Startup Variant: Speed First, Migrate Later

**Framework shortcut:**
1. Can I use PostgreSQL? If yes, use PostgreSQL. It handles 95% of startup workloads.
2. Do I need something else? Only if: your data is naturally document-shaped (MongoDB), you need sub-millisecond reads (Redis), or full-text search is a core feature (Elasticsearch).
3. Can I avoid choosing? Use a managed service (Supabase, Neon, Railway) that abstracts the database entirely.
4. When to rethink? When traffic exceeds 10,000 QPS, data exceeds 500 GB, or queries take longer than 100ms and cannot be optimized.

**Common pattern:** Postgres for primary data + Redis for caching + one specialized service (search, analytics) when needed. Add databases one at a time, only when the pain is real.

### The Enterprise Variant: Compliance and Governance

**Additional criteria:**
1. **Compliance certifications** — SOC 2, HIPAA, GDPR, PCI-DSS. Not all databases have the certifications you need.
2. **Data residency** — Can you control which geographic region stores data? Some databases (CockroachDB, Spanner) have configurable region constraints.
3. **Vendor lock-in risk** — Can you export data in a portable format? Is there an open-source equivalent?
4. **Audit logging** — Who accessed what data, when, and from where? Oracle and SQL Server have built-in audits. Open-source databases need add-ons.
5. **Support contract** — Can you call someone at 3 AM? Enterprise-grade SLAs?

**Common pattern:** Oracle or SQL Server for core financial systems, PostgreSQL for new services, specialized databases for specific workloads.

### The Data-Heavy Variant: Analytics and Big Data

**Key differences:**
1. **Columnar storage** — Analytical queries read many rows but few columns. Columnar storage reads only needed columns.
2. **MPP architecture** — Massively Parallel Processing (Redshift, Snowflake, ClickHouse) distributes query execution across many nodes. A query on 10 TB runs in seconds, not minutes.
3. **Separation of storage and compute** — Modern analytical databases (Snowflake, BigQuery) separate cheap object storage from elastic compute.
4. **Materialized views** — ClickHouse, PostgreSQL, and TimescaleDB continuous aggregates reduce query times from seconds to milliseconds.
5. **Data ingestion** — Analytical databases often need ETL pipelines (Airbyte, Fivetran, dbt) to load data from operational databases.

**Common pattern:** OLTP database (PostgreSQL) + CDC (Debezium, Kafka) + OLAP database (ClickHouse, Snowflake) + dbt for transformations. Or a unified database (SingleStore, YugabyteDB) for moderate workloads.

### The Polyglot Persistence Variant

Polyglot persistence uses multiple databases for different workloads. This is the most mature approach, and the most operationally complex.

**When to use:** Your system has fundamentally different workloads (data shape, access patterns, consistency). You have the operational capacity to run multiple databases. The value of optimal databases exceeds integration cost.

**You will eventually accept polyglot persistence.** No single database handles all use cases well. The question is when, not whether.

**Avoiding complexity:**
- Use PostgreSQL extensions (pgvector, PostGIS, TimescaleDB) before adding separate databases.
- Add auxiliary databases one at a time, when the pain is undeniable.
- Standardize on one managed provider rather than self-hosting each.
- Use an abstraction layer (database router, service mesh) to keep topology invisible to app code.

```python
class DatabaseRouter:
    def __init__(self):
        self.databases = {}

    def register(self, workload_type, db_client):
        self.databases[workload_type] = db_client

    def write_user(self, user_data):
        self.databases["primary"].insert("users", user_data)
        self.databases["cache"].set(
            f"session:{user_data['id']}", user_data, ttl=3600
        )
        self.databases["search"].index(
            index="users", id=user_data["id"], body=user_data
        )

    def search_users(self, query):
        return self.databases["search"].search(
            index="users",
            body={"query": {"match": {"name": query}}}
        )

    def get_user_profile(self, user_id):
        return (self.databases["cache"].get(f"session:{user_id}")
                or self.databases["primary"].find_one("users", {"id": user_id}))
```

## Mental Model Check

### The SQL vs NoSQL False Dichotomy

The most common mistake is treating "SQL" and "NoSQL" as meaningful categories. They are marketing terms.

**What people think they mean:**
- SQL = old, rigid, hard to scale
- NoSQL = new, flexible, scales infinitely

**What they actually mean:**
- SQL = relational model with declarative query language — strong foundation for data integrity
- NoSQL = variety of models (document, key-value, graph) — optimized for specific access patterns

**Reality:**
- PostgreSQL handles many NoSQL use cases (JSONB, full-text search, GIN indexes, partitioning, logical replication)
- Many "NoSQL" databases now support SQL-like query languages (CQL for Cassandra, N1QL for Couchbase)
- The choice should be about data shape and access patterns, not about SQL syntax

Think of it this way: SQL vs NoSQL is like asking "should I use a hammer or a power tool?" It doesn't make sense until you specify what you're building.

### The CAP Theorem Trap

CAP says you cannot have all three of Consistency, Availability, and Partition Tolerance. "Pick two" is misleading.

**Nuance:**
- CAP assumes a network partition is happening. During normal operation, you can have all three.
- The real trade-off is consistency vs latency. Strong consistency across geo-distributed nodes requires coordination, which adds latency.
- For most systems, network partitions are rare. The CAP "choice" matters most during these rare events.

### The "One Size Fits All" Myth

No single database handles all workloads well. Using a database for a workload it was not designed for causes:
- Poor performance (slow queries, high latency)
- Operational pain (scaling bottlenecks, complex workarounds)
- Data integrity issues (losing guarantees you assumed existed)
- Higher cost (over-provisioned infrastructure)

**Litmus test:** Can you state in one sentence what your database is optimized for, and does that match your primary workload? If no, you have the wrong database.

## Summary: The Database Decision Checklist

Before you make a database decision, run through this checklist:

1. **What is the natural shape of my data?** (relational / document / graph / time-series / key-value)
2. **What are my three most important queries?** (write them out)
3. **What consistency and isolation do I need?** (can you tolerate stale reads? concurrent write conflicts?)
4. **What is my 12-month scale projection?** (writes/second, reads/second, data volume)
5. **Who will operate this database?** (team size, existing expertise, willingness to learn)
6. **Is there a managed service?** (yes reduces operational burden significantly)
7. **What is my migration path?** (data export format, compatible alternatives)

**If the answer to any question is "I don't know," start with PostgreSQL. Database decisions made without understanding your workload are guesses, not engineering.**

## Exercises

### Exercise 1: Diagnose the Pain

A startup uses MongoDB for a social media platform (users, posts, likes, follows). As they grew:
- Feed queries (joining posts, likes, follows) became slow
- Follower counts are sometimes wrong
- Complex application code maintains data consistency

**Questions:**
1. What is the root cause of slow feed queries?
2. Why is the follower count inconsistent?
3. Would PostgreSQL be a better fit? Why?
4. If they stay on MongoDB, what patterns could mitigate these issues?

### Exercise 2: Choose the Database

For each scenario, recommend a database and explain why:

1. A real-time chat application with 10M DAU. Messages must be in order. Users access chat history. Three regions.

2. A financial reporting system processing 50M transactions/day. Reports run on daily/weekly/monthly aggregates. Accuracy is critical.

3. A recommendation engine finding "users who bought X also bought Y" in under 50ms over 10M products.

4. A server monitoring system ingesting 200,000 metrics/second. Queries are time-range based. Data older than 90 days is downsampled.

### Exercise 3: Migration Risk Assessment

Your team migrates from self-hosted MongoDB to Amazon Aurora (PostgreSQL). You keep MongoDB for the product catalog.

**Identify risks:**
1. What could go wrong during dual-write?
2. What if cutover fails and you need to roll back?
3. How do you handle nested MongoDB documents needing normalization?
4. What performance differences might surprise the application team?
5. How will you verify data integrity after migration?

## Key Takeaways

1. **There is no best database.** Only the best database for your specific context.
2. **Start with your data's natural shape.** Data model is the most fundamental constraint.
3. **Consistency is a spectrum, not a binary.** Know which operations need strong consistency.
4. **Scale predictions are usually wrong.** Optimize for 12 months, not 5 years.
5. **Operational reality matters more than technical superiority.** A database your team cannot operate will fail.
6. **Polyglot persistence is inevitable, but introduce it lazily.** Add databases one at a time.
7. **PostgreSQL is the default for a reason.** It handles 95% of workloads well.
8. **Plan for migration from day one.** Every database choice is temporary.
9. **The SQL vs NoSQL framing is a distraction.** Focus on data model and access patterns.
10. **Benchmark, don't guess.** Run realistic prototypes with your data before committing.
