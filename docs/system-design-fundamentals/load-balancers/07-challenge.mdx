## Tier 1 -- Foundation

Five minutes each. Answer in 2-3 sentences per question.

### Challenge F1: Round-Robin Trap

**Prompt**: You have 4 identical backend servers. In production, one server is 3x slower than the others due to a noisy neighbor (another VM on the same hypervisor consuming CPU). You use round-robin. Describe the exact user-facing impact.

**Evaluation criteria**: Did you identify that the slow server processes requests slowly while still receiving 25% of traffic? Did you mention that queues form on the slow server and response time increases for everyone? Did you suggest least-connections as the fix?

**Model answer**: Round-robin sends 25% of requests to the slow server. Those requests take 3x longer to process, so p99 latency rises by 75-100ms for the affected user group. Least-connections would detect the backlog on the slow server (its active connection count stays high because requests take longer) and route fewer requests to it. If the slow server reaches max connections, new requests automatically skip it.

### Challenge F2: Health Check Depth

**Prompt**: Your health check endpoint is `GET /health` which returns `{"status": "ok"}` immediately. Users report that every 5-10 minutes, they get 5-second timeouts. The health check never fails. What is the most likely cause and how do you fix it?

**Evaluation criteria**: Did you recognize that the health check is too shallow? Did you propose a health check that exercises dependencies? Did you mention passive health checks as a complementary mechanism?

**Model answer**: The health check only verifies the web server process is running, but the application has a dependency failure (database connection pool exhaustion, Redis latency spike, external API timeout). The fix is to make the health check query each dependency: `SELECT 1` to Postgres, `PING` to Redis, and check internal queue depth. Only return 200 if all dependencies are healthy. Additionally, enable passive health checking -- if the load balancer observes 5 consecutive 5xx responses from a backend, mark it down even if the active health check passes.

### Challenge F3: TLS Handshake Overload

**Prompt**: Your load balancer handles 100,000 new TLS connections per second on a c5.4xlarge (16 vCPU). CPU is at 98%. Backend servers are at 15% CPU. The response is a JSON payload under 1 KB. What is wrong and what three things can you do?

**Evaluation criteria**: Did you identify TLS termination as the bottleneck? Did you suggest at least two practical mitigations? Did you consider the latency impact of each option?

**Model answer**: TLS handshake is CPU-intensive (asymmetric crypto). One c5.4xlarge can do approximately 15,000-20,000 new TLS handshakes per second. At 100,000/second, it is 5x overloaded. Three fixes: (1) Reduce new connection rate by enabling HTTP/2 multiplexing and keep-alive -- clients reuse existing connections instead of opening new ones. (2) Offload TLS to a cloud LB (AWS NLB with TLS termination, or CloudFront) which handles TLS in dedicated hardware. (3) Add more LB instances scale horizontally to 5-6 instances.

## Tier 2 -- Design

Fifteen minutes each. Draw a diagram, list components, justify decisions.

### Challenge D1: E-Commerce Checkout

**Prompt**: Design a load balancing layer for an e-commerce checkout system. Requirements: 10,000 checkouts per second at peak. Each checkout involves 5 internal API calls (cart service, inventory, payment, shipping, notification). Checkout must complete within 2 seconds. Payments are idempotent after 30 seconds. Peak traffic is 3x average during Black Friday.

**Rubric**:

| Criterion | Pass (2 pts) | Good (4 pts) | Excellent (6 pts) |
|---|---|---|---|
| LB architecture | Single LB layer | Multi-tier with L4 + L7 | Multi-tier + circuit breakers + canary deployment |
| Algorithm choice | Round-robin | Least-connections with stickiness | Consistent hashing + weighted connections |
| Failure handling | Mentions health checks | Mentions circuit breakers and draining | Full drain + retry with idempotency + slow start |
| Scale math | Rough numbers | Correct capacity with headroom | Exact instance count + bandwidth + cost estimate |
| Black Friday | Just "auto-scale" | Pre-warming + overprovisioning plan | Predictive scaling + load shedding + degraded mode |

**Model answer outline**:

1. **Architecture**: Cloud L4 LB (AWS NLB) for global entry + internal L7 (Nginx x 4 c5.4xlarge) for path routing + per-service LBs for checkout microservices. L7 LB handles TLS termination, rate limiting (200 req/s per client IP), and circuit breaking (10 consecutive 5xx = 60 second cooldown).

2. **Algorithm**: Least-connections for the L7 LB because request processing time varies (5ms for cart lookup, 300ms for payment). Consistent hashing via session ID to ensure the same backend handles all 5 calls for one checkout (solves the "shopping cart not found" problem).

3. **Capacity**: 10,000 req/s checkout peak. Each request averages 50 KB total (body + response). Bandwidth: 4 Gbps. Nginx L7 tier: 4 instances handle ~160,000 rps with TLS, giving 4x headroom. During Black Friday (30,000 req/s), add 4 more instances pre-warmed.

4. **Failure handling**: Connection draining set to 5 seconds (checkout timeout 2s x 2 + 1s buffer). Circuit breaker trips per-backend-pool, not per-instance, to handle payment gateway degradation. Retry with 200ms backoff for idempotent steps (payment after 30s window).

5. **Black Friday**: Pre-warm 8 LB instances instead of 4. Enable load shedding: return 503 with Retry-After header when queue depth exceeds 1000 requests. Degraded mode: polling for shipping status disabled, notifications batched.

### Challenge D2: Multi-Region Database

**Prompt**: Design a load-balancing system for a globally distributed Postgres database with 1 primary (US-East) and 6 read replicas (2 US-East, 2 EU-West, 2 AP-Southeast). Application servers run in all three regions. Each region has 10,000 read qps and 1,000 write qps. Reads should use local replicas when possible. Writes go to the primary. Failover under 60 seconds.

**Rubric**:

| Criterion | Pass (2 pts) | Good (4 pts) | Excellent (6 pts) |
|---|---|---|---|
| Read routing | Single LB | Regional L4 LB per replica pool | Regional LB + latency-based routing + local fallback |
| Write routing | Direct to primary | Internal LB with health check | Multi-region LB with primary detection + failover |
| Failover | DNS failover | DNS failover + LB health checks | Anycast + automated primary promotion + DNS TTL management |
| Replication lag | Mentions async | Handles stale reads with session stickiness | Read-your-writes consistency with timestamp-based routing |

**Model answer outline**:

1. **Read routing**: Each region has an L4 HAProxy (port 5432, TCP mode, least-connections) fronting the 2 local replicas. Application connects to `replica-db.us-east.internal:5432`. The DNS name resolves to the local HAProxy VIP. If both local replicas are down, the HAProxy has a `default-server init-addr last` fallback to a replica in the next-closest region.

2. **Write routing**: All writes go to `primary-db.internal:5432`, an L4 LB with a single backend (the current primary). If the primary fails, the L4 target is updated to the new primary. Health check: custom PostgreSQL check (`pg_isready` + `SELECT pg_is_in_recovery() = false`).

3. **Failover**: If the primary fails, etcd-based failover tool promotes the least-lagged replica. The DNS record for `primary-db.internal` gets a new IP (TTL: 5 seconds). The L4 LB target group is updated via API. Total failover time: 10 seconds (detection) + 5 seconds (promotion + LB update) + 5 seconds (DNS propagation) = under 20 seconds.

4. **Read-your-writes**: Write operations return a timestamp (Postgres transaction ID or application-generated request ID). Read replicas track applied LSN (Log Sequence Number). The application sends the last-write timestamp as a query parameter (e.g., `?min_lsn=12345678`). The LB routes to a replica that has applied at least that LSN. If no local replica qualifies, route to the primary.

### Challenge D3: WebSocket Chat (10M Concurrent)

**Prompt**: Design load balancing for a real-time chat app with 10 million concurrent WebSocket connections. Each user maintains a WebSocket to receive messages. Messages must be delivered in under 100ms. Users stay connected for 8+ hours. Rolling restarts happen weekly. One region.

**Rubric**:

| Criterion | Pass (2 pts) | Good (4 pts) | Excellent (6 pts) |
|---|---|---|---|
| Connection stickiness | Mentions IP hash | Cookie-based stickiness | Cookie-based with fallback + Redis session lookup |
| Connection limit | Single number | Per-server max + connection draining | Per-server max + queue + timeout + graceful degradation |
| Rolling restart | Restart all at once | Draining one at a time | Slow-start after restart + canary group |
| Health check | TCP only | HTTP /ws health | WebSocket ping/pong + traffic-based passive check |

**Model answer outline**:

1. **Architecture**: Cloud L4 LB (AWS NLB) for DDoS + self-managed L7 (Nginx on c5n.4xlarge, 25 instances) for WebSocket termination + chat server pool (100 instances, 100,000 connections each).

2. **Stickiness**: Nginx sets `Set-Cookie: route=chat-server-42` on first WebSocket upgrade. Reconnecting clients send the cookie and route to the same chat server. If that server is down, Nginx routes to any available server (the client reconnects and the application rebuilds state from Redis).

3. **Capacity**: 10 million connections / 100,000 per server = 100 chat servers. Nginx instances: 10 million / 500,000 per instance = 20 instances, round up to 25 for headroom. Memory: 10 million x 8 KB per connection = 80 GB across Nginx fleet.

4. **Rolling restart**: Nginx: blue-green. Spin up 25 new instances, switch NLB target group, validate, drain old (60 second drain timeout). Chat servers: rolling restart with 5 servers at a time. 60-second drain, 60-second slow start (10% traffic, ramp over 60 seconds). Total rollout: 100 servers / 5 at a time x 2 minutes per batch = 40 minutes.

5. **Health check**: WebSocket ping/pong every 10 seconds. If ping not returned within 5 seconds, server marked unhealthy. Passive: if Nginx sees >1% of WebSocket connections from a server dropping with errors, eject it.

## Tier 3 -- Expert

Thirty minutes each. No single correct answer. Discussion required.

### Challenge E1: Global Social Feed

**Prompt**: Design a load balancing layer for a global social network feed system. Each user sees a feed with posts from people they follow. Feeds are cached. The cache hit ratio is 85%. Traffic is 80% read, 20% write. Peak: 1 million req/s globally. Writes must be visible to the writer within 1 second (read-your-writes consistency). Other users can see stale data for up to 5 minutes. Regions: US-East, EU-West, AP-Southeast.

**Key design tensions**:
- Caching works best when the same user always hits the same backend (local cache), but the user may travel or connect from different regions.
- Write path must go to a primary region, but reads prefer local replicas. How do you route a user's read-your-writes request without adding 180ms of cross-region latency?
- Auto-scaling adds backends which invalidate consistent hash ring assignments. How do you prevent cache misses during scaling events?

**Discussion points**:
- Do you use consistent hashing per user ID to maximize cache locality? What happens when a user travels from US to Japan? Their traffic shifts to a different region but their hash key still maps to a US backend.
- If you replicate cache entries across regions, does the cache invalidation traffic exceed the read traffic savings?
- How do you handle a regional cache warming event (new server, scaled-up region) without a thundering herd of cache misses hitting the database?
- Is a global load balancer a single point of failure? How does anycast BGP handle this?

**Model answer** (one possible approach):
Use user-ID-based consistent hashing within each region, so all requests for user 12345 go to the same backend in US-East. For cross-region travel, the DNS anycast routes to the nearest region. If user 12345 lands in Japan, the Japan region's load balancer sees an unknown user -- no local cache exists. It proxies the read to the user's home region (US-East) and caches the response locally. The user's subsequent requests hit the Japan cache. Write-your-reads: the write goes to US-East (primary region) and the response includes a timestamp. The Japan LB marks the local cache entry for this user as stale until the replication confirms the write has propagated. Write visibility: US-East users see it in under 100ms. EU-West users see it in under 1 second (80ms replication + LB routing). AP-Southeast users see it in under 2 seconds (180ms replication).

### Challenge E2: Incident Post-Mortem

**Prompt**: At 2:14 PM, your monitoring system shows a spike to p99 response time from 120ms to 8 seconds. 23% of requests return 503. After 12 minutes, the system recovers on its own. The load balancer logs show no anomalies except a 30% increase in traffic starting at 2:12 PM. Backend CPU never exceeded 40%. Database CPU never exceeded 25%. Reconstruct what happened. What monitoring gap allowed this to go undetected for 12 minutes? What changes do you make?

**Key investigation threads**:
- If backend CPU is only 40%, why were there 503 errors? The bottleneck is not CPU. Is it a connection pool? Thread pool? File descriptor limit?
- The load balancer shows 30% traffic increase. If a single API endpoint causes the spike, which one? What percentage of requests hit that endpoint?
- 23% error rate but only ONE backend pool? If the error is specific to one pool, the LB should have ejected it. Why did it not? What was the health check returning?
- "Recovered on its own" suggests the problem was self-limiting. Was it a degraded dependency that recovered (external API timeout)? A cache TTL expiration that resolved when the cache repopulated? A temporary resource exhaustion that freed up when requests backed off?

**Model answer** (one reconstruction):
The spike was caused by a Redis cluster node failing over. The application's Redis client had no circuit breaker: it waited 2 seconds for a response, then threw an exception. The health check endpoint (`GET /health`) returned 200 because it did not check Redis connectivity. The LB continued sending full traffic to all backends. Backends accumulated pending connections as each request waited for the Redis timeout. Backend thread pool exhaustion caused 503s. After 12 minutes, Redis failover completed, requests started succeeding, the backlog drained, and response times returned to normal.

**Fixes**:
1. Health check must verify Redis connectivity (`PING` with 500ms timeout).
2. Application Redis client needs a circuit breaker (3 timeouts in 10 seconds = trip for 30 seconds).
3. Load balancer should have a passive health check: if a backend produces 5% 503s in 60 seconds, mark it down.
4. Add Redis-connected health dashboard with per-endpoint latency breakdown.
5. Auto-diagnosis rule: "if p99 > 500ms and backend CPU under 50%, check external dependencies."
