**Tier 1 — Foundation**

**Challenge 1: Explain State Drift (5 minutes)**

A teammate says: "I noticed that the production database in the AWS console has backup retention set to 35 days, but our Terraform config specifies 7 days. Is this a problem?"

Write a clear explanation addressing:
- What state drift is
- Why it happened (someone changed backup retention through the console)
- The risk: next `terraform apply` will reset backup retention to 7 days
- The fix: either update the Terraform config to 35 days (if 35 is correct) or run apply to set it back to 7 days

**Model answer:**

"Yes, this is state drift. The actual infrastructure configuration has diverged from what Terraform manages. It probably happened because someone with console access changed the backup retention setting directly in the AWS console. Terraform is still tracking the backup retention as 7 days in its state file.

The risk is real: any time someone runs `terraform apply` on the production configuration, Terraform will see that the actual backup retention (35 days) does not match the desired state (7 days), and it will reset it back to 7 days. If the team intentionally wanted 35 days (e.g., for compliance reasons), they would lose that change.

The correct fix is to decide which value is correct. If 35 days is what we want, update the Terraform config to `backup_retention_period = 35` and run apply — no actual change will happen because the cloud already matches. If 7 days is what we want, run apply and it will reset to 7 days. Either way, the Terraform config and the actual infrastructure must be in agreement."

**Common mistake:** Saying drift is always bad. Drift itself is not bad — it signals a discrepancy. The question is whether the Terraform config or the manual change represents the correct desired state.

**Challenge 2: Explain Idempotency (5 minutes)**

Explain why idempotency is a fundamental property of Infrastructure as Code. Use a specific example of a resource creation to demonstrate.

**Model answer:**

"Idempotency means that running the same IaC configuration multiple times produces the same result, and running it when the infrastructure already matches does nothing.

Consider a security group defined in Terraform:

```hcl
resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Security group for web servers"

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

The first time you run `terraform apply`, Terraform calls the AWS API to create this security group. It receives a security group ID (sg-12345) and stores it in the state file.

The second time you run `terraform apply`, Terraform reads the state file, sees that `aws_security_group.web` already exists with ID sg-12345, compares its attributes (name, description, ingress rules) against what the config says, finds no difference, and reports "No changes. Infrastructure is up-to-date."

This is idempotency. It is what makes IaC safe to run repeatedly. It also means that if someone manually deletes the security group, the next `terraform apply` will recreate it automatically — because Terraform sees it in the state file but cannot find it in the cloud, it marks it as needing creation.

Without idempotency, a manual `terraform apply` run twice would create duplicate resources — two security groups with the same name (which would fail) or duplicate entries in a route table (which would succeed but cause routing ambiguity)."

**Common mistake:** Confusing idempotency with immutability. Idempotency means the result is the same regardless of how many times you run the operation. Immutability means you never modify existing resources — you replace them.

**Challenge 3: Choose the Right IaC Approach (5 minutes)**

You have these requirements:
1. A startup with 2 engineers building their first product on AWS
2. A financial services company with 50 engineers, 3 environments, and PCI compliance
3. A solo developer running a personal blog on a single VPS

For each scenario, recommend an IaC approach and justify your choice.

**Model answer:**

**Scenario 1 — Startup, 2 engineers, AWS:**
Use Terraform with S3 remote state. Simple structure: a single directory with `main.tf`, `variables.tf`, `outputs.tf`. One state file. CI/CD is optional at this stage — engineers can run from their laptops with state locking disabled (small team, low risk). The goal is to get the infrastructure defined in code without over-engineering the pipeline.

**Scenario 2 — Financial services, 50 engineers, PCI compliance:**
Use Terraform with a multi-account AWS organization. Dedicated management account for state. Separate state files per environment with DynamoDB locking. Policy as code with Sentinel for compliance rules. CI/CD pipeline with mandatory plan approval for production. Secrets management via Vault or AWS Secrets Manager. Immutable AMIs built by Packer. IaC modules published to an internal registry with semantic versioning. The compliance requirement drives the complexity — every change must be auditable, every resource must meet security standards.

**Scenario 3 — Solo developer, personal blog, single VPS:**
Do not use Terraform. A single VPS is overkill for IaC. Use a simple script (bash or Ansible playbook) to configure the server. Manual backups of the configuration files. If the server dies, create a new one, run the script, and restore the backup. The overhead of Terraform state management, CI/CD, and module structure is not justified for a single server.

**Tier 2 — Design**

**Challenge 4: Design IaC for a Data Pipeline (15 minutes)**

Design an IaC strategy for a data pipeline with these requirements:
- Ingest 10 TB of data per day from external partners via SFTP
- Process data using Spark on EMR (10 nodes, r5.2xlarge)
- Store processed data in S3 (500 TB total, lifecycle policy to Glacier after 90 days)
- Run once per day, 6-hour processing window
- 3 environments: dev, staging, production

**Evaluation rubric:**

| Criteria | Weak Answer | Strong Answer |
|----------|-------------|---------------|
| State separation | Single state file for everything | Separate state files per environment, and optionally per component (network, compute, storage) |
| EMR lifecycle | EMR cluster always running | EMR cluster created on-demand for each processing run, destroyed after completion (saves cost) |
| S3 configuration | Basic bucket creation | Bucket with versioning, encryption, lifecycle policy, bucket policy for access control |
| SFTP considerations | Not mentioned | SFTP server (AWS Transfer Family) with Elastic IP, IAM authentication, logging |
| Cost optimization | Not discussed | Spot instances for EMR, S3 lifecycle to Glacier, cluster auto-termination |
| Deployment process | "Run terraform apply" | CI/CD pipeline with plan review, environment promotion (dev -> staging -> prod), data validation checks |

**Model answer:**

"I would structure this as three Terraform workspaces or directories, one per environment, each with its own state file in S3 with locking.

The Terraform configuration would have these modules:

```hcl
module "network" {
  source   = "./modules/vpc"
  env      = var.environment
  vpc_cidr = var.vpc_cidr
}

module "sftp" {
  source        = "./modules/sftp"
  env           = var.environment
  vpc_id        = module.network.vpc_id
  # SFTP server with endpoint in private subnet
  # IAM users for each external partner
  # CloudWatch logging for audit
}

module "emr" {
  source        = "./modules/emr"
  env           = var.environment
  vpc_id        = module.network.vpc_id
  subnet_ids    = module.network.private_subnet_ids
  # Creates EMR cluster with auto-termination
  # Bootstrap actions to install Spark configs
  # Spot instances for task nodes (70% cost saving)
}

module "storage" {
  source         = "./modules/s3-data-lake"
  env            = var.environment
  # Bucket with SSE-S3 encryption
  # Lifecycle: standard (30d) -> infrequent (60d) -> Glacier (90d)
  # Versioning enabled for data recovery
}
```

For the EMR lifecycle, I would use a Lambda function triggered by CloudWatch Events that calls `terraform apply` with a variable `emr_action = "create"` at the scheduled start time, and `emr_action = "destroy"` after the processing window. This avoids paying for an idle cluster for 18 hours per day.

For SFTP, I would use AWS Transfer Family with an Elastic IP so external partners have a stable endpoint. IAM authentication maps each partner to a specific S3 prefix for data isolation. All transfers are logged to CloudWatch for compliance auditing.

The CI/CD pipeline would: (1) run `terraform plan` on every PR, (2) apply to dev on merge, (3) promote to staging after dev tests pass, (4) promote to production after manual approval. Data validation tests (row counts, schema checks) would run between each promotion to catch data issues early."

**Challenge 5: Refactor to IaC — Migration Strategy (15 minutes)**

A company has 200 EC2 instances, 20 RDS databases, 50 S3 buckets, and 30 Elastic Load Balancers — all created manually over 3 years. They want to move to Terraform. Design the migration strategy with zero downtime.

**Evaluation rubric:**

| Criteria | Weak Answer | Strong Answer |
|----------|-------------|---------------|
| Import approach | "Start fresh, recreate everything" | "Import existing resources into Terraform state, one service at a time" |
| Downtime consideration | Not mentioned | "Zero downtime: import first, then manage through Terraform" |
| Risk management | "Convert everything in one sprint" | "Phased migration with rollback plan per service" |
| State file strategy | "One giant state file" | "Separate state files per service or per component" |

**Model answer:**

The migration follows a "strangler fig" pattern — gradually wrapping existing resources in Terraform management without recreating them.

**Phase 1 — Discovery (1 week):**
Use AWS Config and Tag Editor to inventory all resources. Identify untagged resources. Assign ownership tags to every resource. Group resources by service and environment.

**Phase 2 — State file bootstrap (1 week per service):**
For each service, write the Terraform configuration that matches the existing infrastructure. Use `terraform import` to bring each existing resource into Terraform state. The key command:

```bash
terraform import aws_instance.web_server i-0a1b2c3d4e5f67890
```

After importing, run `terraform plan`. It should show "No changes." If it shows changes, adjust the configuration to match exactly. This step requires careful attention to detail — every tag, every security group rule, every RDS parameter must match.

**Phase 3 — Verification (2 weeks):**
The team operates normally, but all changes go through the new Terraform pipeline. For the first two weeks, any Terraform apply is reviewed by two engineers — one from the migration team and one from the service team. No production incidents should result from the migration.

**Phase 4 — Cleanup (1 week):**
After all services are migrated, remove console-level IAM permissions for resource creation. Enforce IaC-only policy. Run drift detection weekly.

**Key technical considerations:**
- S3 buckets must be imported carefully — the bucket already exists, Terraform must not recreate it
- RDS databases should be imported with `terraform import` pointing to the existing DB instance
- Auto-scaling groups require importing the ASG itself, then Terraform can manage the launch configuration
- IAM roles and policies can be imported but must not disrupt existing permissions during the transition

**Challenge 6: Multi-Region Disaster Recovery (15 minutes)**

Design an IaC system for a global SaaS application that must:
- Run in 3 AWS regions (us-east-1, eu-west-1, ap-southeast-1)
- Fail over between regions within 30 minutes
- Have independent IaC management per region so one region's failure does not block operations in other regions
- Share global resources (Route53 zone, CloudFront distribution)

**Evaluation rubric:**

| Criteria | Weak Answer | Strong Answer |
|----------|-------------|---------------|
| State per region | Single state file for all regions | Separate state file per region |
| Global vs regional | Not separated | Global state file for cross-region resources |
| Failover | Manual DNS change | Route53 health checks with failover routing |
| Data replication | Not addressed | Cross-region read replicas for RDS, S3 CRR for storage |

**Model answer:**

```hcl
# Directory structure
terraform/
├── global/                          # Resources shared across all regions
│   ├── main.tf                      # Route53 zone, CloudFront, IAM
│   └── state/                       # Single state file for global
├── us-east-1/
│   ├── main.tf                      # Regional resources
│   ├── variables.tf                 # region = us-east-1
│   └── state/                       # Separate state file
├── eu-west-1/
│   ├── main.tf
│   ├── variables.tf                 # region = eu-west-1
│   └── state/
└── ap-southeast-1/
    ├── main.tf
    ├── variables.tf                 # region = ap-southeast-1
    └── state/
```

Each region has its own Terraform state file stored in S3 with a region-prefixed key:
- `terraform-state/global/terraform.tfstate`
- `terraform-state/us-east-1/terraform.tfstate`
- `terraform-state/eu-west-1/terraform.tfstate`
- `terraform-state/ap-southeast-1/terraform.tfstate`

The global module creates the Route53 zone and CloudFront distribution. Regional modules create VPC, EKS cluster, RDS read replicas, and application infrastructure. Each regional module reads the Route53 zone ID from the global state using `terraform_remote_state`.

For failover, Route53 health checks monitor the ALB endpoint in each region. If us-east-1 fails, Route53 routes traffic to eu-west-1. The RDS read replica in eu-west-1 is promoted to primary. This failover is defined in Terraform:

```hcl
resource "aws_route53_health_check" "primary" {
  fqdn              = aws_lb.main.dns_name
  port              = 443
  type              = "HTTPS"
  resource_path     = "/health"
  failure_threshold = 3
  request_interval  = 10
}

resource "aws_route53_record" "app" {
  zone_id = data.terraform_remote_state.global.outputs.route53_zone_id
  name    = "app.example.com"
  type    = "A"
  set_identifier = var.region

  failover_routing_policy {
    type = var.region == "us-east-1" ? "PRIMARY" : "SECONDARY"
  }

  alias {
    name                   = aws_lb.main.dns_name
    zone_id                = aws_lb.main.zone_id
    evaluate_target_health = true
  }
}
```

The deployment pipeline deploys to each region independently. A failure in the ap-southeast-1 apply does not block us-east-1 deployments. Drift detection runs per region, so if someone manually changes resources in eu-west-1, it is caught independently of other regions.

**Tier 3 — Expert**

**Challenge 7: Platform Engineering — IaC at Scale (30 minutes)**

You are the infrastructure lead at a company that has grown from 20 to 200 engineers in 18 months. Each team manages their own infrastructure using Terraform, but there is no standardization. Some teams use one state file, others use workspaces. Some teams commit their state files to Git. Some teams have no CI/CD pipeline and run `terraform apply` from laptops.

Design a platform engineering approach that:
- Standardizes IaC practices across all teams without blocking velocity
- Enforces security and compliance rules automatically
- Provides self-service infrastructure for common patterns (web service, batch job, data pipeline)
- Works with both AWS and GCP (the company uses both)
- Includes a migration plan from the current chaos to the target state

**Evaluation discussion:**

A strong answer explores multiple dimensions beyond the technical Terraform setup:

**Organizational dimension:**
Create a platform team (3-4 engineers) that owns the IaC framework, not the infrastructure itself. The platform team builds and maintains modules. Application teams consume modules. The platform team is not a bottleneck because they provide a library, not a service. Each application team is responsible for their own infrastructure configuration, but they must use the platform's approved modules.

**Module registry:**
Publish all modules to an internal Terraform registry (Terraform Cloud, or a custom S3-based registry). Each module has semantic versioning. Module versions are pinned in application configurations. Breaking changes require a major version bump. Modules include compliance checks internally — an "S3 bucket" module always enables encryption, never allows public access, and always requires tags.

**Policy as code:**
Use Sentinel or OPA to enforce organization-wide policies regardless of which module version is used. Policies are enforced in the CI/CD pipeline, not in the modules:
- All resources must have `Environment`, `Team`, and `CostCenter` tags
- No security groups may allow inbound 0.0.0.0/0 on port 22
- All S3 buckets must have encryption and versioning enabled
- All RDS instances must have backup retention of at least 7 days
- Terraform version must be 1.7 or later

**Multi-cloud abstraction:**
Create a common interface module per resource type. For example, a "compute" module that abstracts EC2 (AWS) and GCE (GCP). The application team passes `provider = "aws"` or `provider = "gcp"`, and the module selects the appropriate resource definitions. This hides provider-specific complexity from application teams.

**Self-service portal:**
Build a simple web UI (or use Backstage) where teams can request infrastructure. The portal generates a pull request with the Terraform configuration pre-filled with the team's inputs. This reduces the barrier to entry for teams that are not comfortable writing Terraform directly.

**Migration plan:**
1. Month 1: Platform team builds the first 5 modules (VPC, compute, database, load balancer, DNS) and the CI/CD pipeline template. No teams are required to adopt yet.
2. Month 2: Pilot team (most cooperative, smallest footprint) migrates to the new platform. Iterate based on feedback.
3. Month 3-4: Two teams per sprint adopt the platform. Platform team provides pair-support during migration. Each migration is a "lift and shift" — import existing resources, do not recreate.
4. Month 5: Launch the self-service portal. New teams default to the platform. Legacy teams have a deadline (month 8) to migrate.
5. Month 6+: Platform team shifts to maintenance mode — fixing bugs, adding new modules on demand, updating policies as compliance requirements evolve.

**Anti-patterns that weak answers exhibit:**
- "The platform team manages all infrastructure." This creates a bottleneck and does not scale.
- "Force all teams to use one module for everything." Over-standardization kills innovation and frustrates teams with legitimate differences.
- "No migration plan — start fresh." This would require rebuilding everything, causing months of downtime.

**Challenge 8: Incident Response and IaC (30 minutes)**

A severe incident occurs: at 2:00 AM, the production database crashes due to a corrupted index. The on-call engineer determines that the fix requires:
1. Increasing the RDS instance class to handle the load during recovery
2. Disabling the read replica temporarily to speed up recovery
3. Running a manual SQL repair command

However, the company policy requires all infrastructure changes to go through Terraform. The CI/CD pipeline takes 15 minutes for a plan + apply cycle. The recovery is time-sensitive — every minute of downtime costs the company $10,000.

Design the incident response process that respects the IaC requirement while enabling fast recovery. The answer must address:
- What the engineer should do in the first 5 minutes
- How to reconcile the emergency changes with IaC after recovery
- How to prevent the same scenario from causing the same delay next time
- The trade-offs between speed and process enforcement

**Evaluation discussion:**

This challenge has no single correct answer. A strong response explores the tension between process and speed and proposes a pragmatic solution.

**Immediate response (first 5 minutes):**

The engineer follows the pre-defined break-glass procedure:
1. Announce the incident in the incident channel: "Following break-glass procedure for PROD-DB-01 failure"
2. Make the emergency changes through the AWS console:
   - Modify the RDS instance class (console or AWS CLI)
   - Disable the read replica
   - Run the SQL repair command
3. Record every action taken in the incident document
4. Fix the immediate issue

The break-glass procedure is a documented, pre-approved exception to the IaC-only policy. It exists specifically for scenarios like this. The company accepts that the 90-second console fix saves $10,000/minute of downtime, and the IaC policy is temporarily suspended.

**Post-incident reconciliation (within 24 hours):**

After the incident is resolved:
1. Run `terraform plan` to see the diff between the configured state and the actual state
2. Update the Terraform configuration to match the desired state:
   - If the larger instance class should be permanent: update the config with the new instance class
   - If the original instance class should be restored: run `terraform apply` to restore it
3. Re-enable the read replica (update config and apply)
4. Document the incident and the reconciliation in the IaC repository
5. Run a post-mortem within 48 hours

**Preventing recurrence:**

1. Implement a faster IaC pipeline for production emergencies. Options:
   - A separate, lightweight pipeline that skips non-essential checks in emergencies (test in dev first)
   - A "fast-path" approval process: pre-approved changes (instance class changes) can be applied faster
   - Terraform Cloud's run tasks with parallel approval

2. Build a runbook Terraform module:
```hcl
# An emergency runbook module
module "emergency_db_scale" {
  source          = "./modules/db-emergency-runbook"
  db_identifier   = var.db_identifier
  target_class    = "db.r6g.4xlarge"
  apply_immediate = true
}
```
This module is pre-approved in the CI/CD pipeline for emergency use. The engineer triggers it with a single command, and it runs through the normal pipeline but with a shorter review window.

3. Schedule regular game days where the team practices the break-glass procedure. The goal is that everyone knows the procedure by heart, and no one hesitates when an incident happens.

**Trade-offs:**

The fundamental trade-off in this scenario is consistency vs availability. The IaC-only policy prioritizes consistency — every change is tracked, reviewed, and repeatable. But during an incident, availability is more important. The break-glass procedure is the mechanism that temporarily switches the priority from consistency to availability, with the explicit understanding that consistency will be restored afterward.

A team that never breaks the IaC policy cannot respond to emergencies quickly. A team that always breaks the IaC policy has no IaC at all — the infrastructure will drift until it no longer reflects the configuration. The balance is found through a documented, tested, and practiced break-glass procedure.

**Anti-patterns that weak answers exhibit:**
- "The engineer should wait for the CI/CD pipeline." This costs $10,000 per minute in downtime. The company is losing money while waiting.
- "Use Terraform import after the fix." Import brings the resource under management but does not update the configuration file. The engineer still needs to update the .tf file.
- "Disable all IaC requirements during incidents." This creates lasting drift because the reconciliation step is forgotten. A weak plan has no procedure for post-incident reconciliation.
