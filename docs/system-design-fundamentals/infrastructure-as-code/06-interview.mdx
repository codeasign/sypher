**How This Topic Appears**

Infrastructure as Code questions appear across multiple categories in FAANG system design interviews. They are most common at the senior level (L5+) where the interviewer expects you to think operationally, not just architecturally.

**Direct questions:**

1. "Explain how Terraform works under the hood. Walk me through what happens when you run `terraform apply`."
2. "What is the difference between declarative and imperative infrastructure management? When would you use each?"
3. "How does Terraform handle state? What happens if the state file is lost or corrupted?"
4. "Explain the concept of Infrastructure as Code to a non-technical product manager."

**Design questions:**

5. "Design an IaC pipeline for a company with 50 microservices, 3 environments, and 20 engineers. How do you organize the code, manage state, and enforce security?"
6. "Design a multi-region infrastructure deployment system. The company has users in North America, Europe, and Asia-Pacific. Walk through the IaC approach."
7. "Your company is migrating from a monolith to microservices. Design the IaC strategy that enables teams to independently deploy their services."

**Debugging questions:**

8. "A developer runs `terraform apply` and it creates a resource that already exists. The provider returns an error and the apply fails. Walk me through debugging this."
9. "Two engineers ran `terraform apply` at the same time and the state file is now corrupt. The infrastructure is fine but nobody can make changes. How do you recover?"

**Trade-off questions:**

10. "Why would you choose Terraform over CloudFormation, or CloudFormation over Terraform?"
11. "Your team is debating between a monorepo with a single state file and multiple repos with per-service state files. What are the trade-offs?"
12. "Compare IaC with GitOps. When would you choose one over the other?"
13. "You need to choose between Terraform and Pulumi for a new startup. What factors influence your decision?"
14. "Your team wants to manage Kubernetes resources through Terraform's Kubernetes provider. What are the risks and alternatives?"

**What the Interviewer Is Actually Testing**

| Question Type | What They Are Testing | 3/4 Answer | 4/4 Answer |
|---------------|----------------------|------------|------------|
| Direct "Explain Terraform" | Depth of understanding — do you know the internals or just surface-level syntax | "Terraform compares config to state and makes API calls" | "Terraform builds a dependency graph, performs a three-way diff, executes in topological order, writes state incrementally, and handles partial failures" |
| "Design an IaC pipeline" | Operational maturity — have you actually run IaC in production | "Use modules, store state in S3, run in CI/CD" | "Separate state per environment, remote state with locking, CI/CD with plan approval gates, policy as code, drift detection, break-glass procedures" |
| Debugging: concurrent applies | Distributed systems thinking — can you reason about concurrent state mutations | "Use state locking" | "State locking via DynamoDB, but also need idempotent providers, retry logic, and a lock timeout to prevent deadlocks" |
| Trade-off: Terraform vs CloudFormation | Decision-making under constraints — can you make a non-dogmatic choice | "Terraform is multi-cloud, CloudFormation is AWS-only" | "Terraform for multi-cloud or when you need a consistent workflow across providers; CloudFormation when you are all-in on AWS and want native integration with StackSets, IAM, and CloudTrail" |
| Debugging: state file corruption | Incident response skills | "Restore from backup" | "Restore from S3 versioning, verify with refresh-only, check for data loss, implement prevention measures" |

**Level-by-Level Expectations**

**L3/L4 — Junior / New Grad**

**Minimum expected knowledge:**
- Know what Infrastructure as Code is and why it matters
- Understand the difference between declarative and imperative
- Can define: state file, idempotency, drift
- Can run `terraform init`, `terraform plan`, `terraform apply`

**Vocabulary they must know:**
declarative, imperative, state, plan, apply, idempotent, drift

**What a passing answer looks like:**

Interviewer: "What is Infrastructure as Code?"

Answer: "Infrastructure as Code is the practice of managing servers, databases, networks, and other infrastructure through configuration files stored in version control, rather than through manual processes like SSH or cloud consoles. The key idea is that you describe the desired state of your infrastructure in a file, and a tool like Terraform or CloudFormation makes the necessary API calls to reach that state. The benefits are repeatability — you can recreate the same infrastructure from the same config files — and auditability — every change is tracked in Git."

**L5 — Mid-level**

**Expected depth of understanding:**
- Can design a multi-environment IaC setup (dev, staging, prod)
- Understands state management: remote backends, locking, state file structure
- Knows about modules and how to structure them
- Can explain Terraform's resource graph and dependency resolution
- Understands the difference between Terraform provisioners and configuration management tools like Ansible

**Vocabulary they must know:**
DAG, resource graph, topological sort, remote state, state locking, module registry, provider plugin, lifecycle hooks, taint, refresh

**Example question + model answer:**

Interviewer: "How would you structure Terraform code for a company with 10 microservices and 3 environments?"

Answer: "I would use a monorepo with a modules directory and an environments directory. The modules directory contains reusable components — a VPC module, an ECS service module, an RDS module. Each module has well-defined inputs and outputs. The environments directory has a folder per environment — dev, staging, prod — each with its own root configuration that calls the modules with appropriate parameters. Each environment gets its own state file stored in S3 with DynamoDB locking. The CI/CD pipeline runs `terraform plan` on pull requests and `terraform apply` on merge to each environment's branch. For production, I would add a manual approval gate in the pipeline before apply. I would also set up drift detection — a scheduled job that runs `terraform plan` nightly and alerts if the actual infrastructure has diverged from the configuration."

**L6 — Senior**

**Expected to drive the conversation proactively:**
- Must raise failure modes before being asked
- Must quantify everything — plan duration, state file sizes, operational overhead
- Must discuss policy as code, compliance, and security implications
- Must address organizational concerns — team structure, ownership boundaries, training
- Must compare multiple approaches and justify the chosen one

**Vocabulary they must know:**
policy as code, sentinel, OPA, cross-state references, data sources, remote_state, workspace, provider aliases, assume_role, secure secret injection

**Example question + model answer:**

Interviewer: "Design an IaC system for a multi-account AWS organization with 10 accounts and 100 engineers."

Answer: "I would use a multi-account approach with a dedicated management account for Terraform state. Each account (dev, staging, prod, shared-services, security, etc.) would have its own Terraform backend configured in S3 with encryption and DynamoDB locking. The state bucket would have versioning enabled and cross-region replication for disaster recovery.

The repository structure would be a monorepo with three top-level directories: `modules/` for reusable components, `accounts/` for root configurations per account, and `global/` for organization-wide resources like IAM roles and Route53 zones that are not account-scoped.

The CI/CD pipeline would use AWS IAM roles with `sts:AssumeRole` — each pipeline step assumes a role specific to the target account. This means a developer's CI/CD configuration for the dev account cannot accidentally target production.

For policy enforcement, I would use Sentinel policies or OPA to enforce rules like: 'all S3 buckets must have encryption enabled,' 'all RDS instances must have backup retention of at least 7 days,' and 'no security groups may allow inbound traffic on port 22 from 0.0.0.0/0.'

For secrets management, I would integrate with AWS Secrets Manager and Parameter Store. Database passwords would be generated by Secrets Manager and rotated automatically. Terraform would reference the secret ARN, never the secret value itself. The state file would not contain plaintext secrets because the secret values are stored outside Terraform.

The operational considerations include: scheduled drift detection, automated state backups, a documented break-glass procedure for emergency changes, and a module versioning strategy using semantic versioning with automated changelogs."

**L7/L8 — Staff / Principal**

**Expected to discuss org-level impact:**
- Must connect IaC strategy to business outcomes: deployment velocity, incident frequency, compliance requirements
- Must address organizational change — how do you get 100 engineers to adopt IaC?
- Must discuss cost implications — IaC tooling, state storage, CI/CD compute, training
- Must address migration strategies — how to move from manual to IaC without downtime

**Vocabulary they must know:**
declarative infrastructure, immutability, GitOps, continuous reconciliation, policy guardrails, golden images, secure-by-default, compliance as code, progressive delivery

**Example question + model answer:**

Interviewer: "Our company has 200 engineers across 20 teams. Infrastructure is currently managed manually through the AWS console. We want to adopt IaC. How do you approach this transition?"

Answer: "This is as much an organizational challenge as a technical one. The technical solution is straightforward — Terraform modules, remote state, CI/CD pipeline. The real challenge is adoption and migration without breaking production.

Phase 1 — Foundation (weeks 1-4): Set up the management account, state bucket, CI/CD pipeline, and module registry. Create the core modules — VPC, EC2, RDS, EKS — with secure defaults. Everything is designed with compliance in mind: encryption enabled by default, no public access by default, mandatory tags including cost center and owner.

Phase 2 — Pilot team (weeks 5-8): Select one team with a relatively simple infrastructure footprint. Work with them to convert their existing infrastructure to IaC. This means importing existing resources into Terraform state. Use `terraform import` for each resource. During this phase, the IaC team provides white-glove support — the pilot team should not feel slower with IaC than without it. The pilot validates that the modules work, the pipeline works, and the developer experience is good.

Phase 3 — Documentation and training (weeks 9-12): Create runbooks, video tutorials, and example repositories. Each module has a README with examples. Create an IaC office hours — twice a week, engineers can ask questions. The training covers: Terraform basics, how the modules work, what to do in an emergency, and how to contribute new modules.

Phase 4 — Rollout (weeks 13-26): Each team gets a two-week migration slot. The IaC team pairs with the team to convert their infrastructure. Migration happens gradually — no big-bang conversions. Each service is converted independently, tested, and validated. The key metric is not 'percent converted' but 'zero incidents during conversion.'

Phase 5 — Governance (ongoing): Once all infrastructure is defined in code, implement policy as code. Use automated compliance checks in the CI/CD pipeline. Enforce tagging standards, security rules, and naming conventions. Publish a monthly compliance report to each team showing their adherence score. Teams that fail compliance checks cannot deploy to production.

The most important thing to get right is the break-glass procedure. Engineers need to know they can make emergency changes if production is down. The procedure is: (1) make the change in the console to fix the incident, (2) create a GitHub issue documenting what was changed, (3) within 24 hours, update the Terraform config and run apply to reconcile. Without this safety valve, engineers will resist IaC because they fear being unable to respond to emergencies."

**The FAANG Framework for Answering**

Use these six steps when answering any IaC-related system design question:

**Step 1: Clarify requirements (1 minute)**

Ask clarifying questions before designing:
- "How many environments do we need?"
- "Is this single-cloud or multi-cloud?"
- "How large is the team and what is their IaC experience level?"
- "Are there compliance requirements?"
- "What is our deployment frequency target?"
- "Do we need to manage existing infrastructure or start fresh?"

What not to ask: Asking "what IaC tool should I use?" too early. Decide the requirements first.

**Step 2: Estimate scale (1 minute)**

Calculate the rough numbers:
- Number of resources: 10 services x 20 resources each = 200 resources
- State file size: 200 resources x 5 KB = approximately 1 MB
- Plan duration: approximately 30 seconds for 200 resources with refresh
- Apply duration: varies by resource type, estimate 2-30 seconds per resource

**Step 3: High-level design (3 minutes)**

Draw the architecture on the whiteboard:
- Repository structure (monorepo vs multi-repo)
- State management (backend, encryption, locking)
- Account structure (single account vs multi-account)
- CI/CD pipeline (lint, validate, plan, approve, apply)

Start with a simple design and say "I will add complexity as needed."

**Step 4: Deep dive (5 minutes)**

Dive into the component the interviewer seems most interested in:
- If they push on state management: explain state structure, locking, backup, recovery
- If they push on security: explain IAM roles, assume_role, secret management, policy as code
- If they push on multi-region: explain state-per-region, provider aliases, data replication
- If they push on GitOps: explain reconciliation loop, drift detection, auto-remediation

**Step 5: Trade-offs (3 minutes)**

Volunteer trade-offs even if not asked:
- "Monorepo vs multi-repo: A monorepo simplifies cross-service refactoring but makes team ownership harder. Multi-repo gives team autonomy but requires cross-state references."
- "Single state file vs multiple: Single file simplifies dependency management but creates a single point of failure and slow apply times. Multiple files are faster but require careful dependency management."
- "Terraform vs Pulumi: Terraform has a larger ecosystem and more mature state management. Pulumi offers more flexibility through general-purpose languages."

**Step 6: Wrap up (1 minute)**

Summarize the key decisions and why you made them:
- "To summarize: single monorepo for consistency, separate state files per environment in encrypted S3 with DynamoDB locking, multi-account AWS organization for isolation, CI/CD pipeline with manual approval for production, and a break-glass procedure for emergencies. The primary trade-off is standardization overhead versus team autonomy."

**Phrase Bank**

**When explaining Terraform's execution model:**
- "Terraform builds a directed acyclic graph of resource dependencies, then traverses it in topological order to ensure each resource is created only after its dependencies are satisfied."

**When discussing state management:**
- "The state file is Terraform's only source of truth for what exists. If the state file is lost, Terraform cannot manage the infrastructure — you would have to import every resource manually."
- "I treat state file backups with the same rigor as database backups. S3 versioning, cross-region replication, and automated backup verification are non-negotiable."

**When discussing drift:**
- "Drift is inevitable in any production system. The question is not whether drift will happen, but how quickly you detect and remediate it."
- "A scheduled drift detection job that runs terraform plan nightly is the minimum viable monitoring for IaC."

**When discussing the declarative vs imperative trade-off:**
- "Declarative configuration tells the tool what you want; imperative tells it how to get there. Declarative is safer because the tool can decide the most efficient and safe way to reach the desired state."
- "I prefer declarative for cloud infrastructure because the state space is well-defined. For software configuration inside servers, imperative tools like Ansible are more practical."

**When discussing CI/CD for IaC:**
- "Every infrastructure change should go through the same review process as application code. Plan output should be visible in the pull request for review."
- "The manual approval gate between plan and apply is not bureaucracy. It is the last line of defense against mistakes that could cost the company millions."

**When discussing organization-level IaC:**
- "IaC adoption is a cultural change, not just a technical one. The team needs to trust that the IaC process will not slow them down in emergencies."
- "I always design a break-glass procedure alongside the IaC pipeline. Without it, the first production incident will destroy the IaC adoption effort."

**When discussing state file security:**
- "State files contain every attribute of every resource, including secrets. Encrypted state backends are not optional — they are table stakes for production use."

**When discussing migration to IaC:**
- "The hardest part of adopting IaC is not writing the configuration — it is importing the existing resources into the state file without causing downtime."

**When discussing module design:**
- "A good Terraform module is opinionated about secure defaults but flexible about inputs. It should make the right thing easy and the wrong thing hard."

**Red Flags**

**Red Flag 1: "I just run terraform apply and it works."** This signals that the candidate has only used Terraform in a tutorial or sandbox environment. Every production engineer has broken infrastructure with Terraform. Acknowledge the failure modes.

**Red Flag 2: "State files are not important, you can always rebuild."** This signals inexperience with production infrastructure. Rebuilding 500 resources from scratch is not practical, and some resources (like S3 bucket contents or RDS data) cannot be rebuilt from Terraform configuration alone.

**Red Flag 3: "I would never use CloudFormation, Terraform is always better."** Dogmatic tool choices are a red flag at all levels. The correct answer depends on constraints: team skills, cloud provider, existing tooling, compliance requirements.

**Red Flag 4: No mention of security.** If a candidate designs an IaC pipeline without discussing state file encryption, IAM permissions, secret management, or policy enforcement, they are not thinking about production security.

**Red Flag 5: "We stored state in Git."** This is a dangerous anti-pattern. State files contain sensitive data and are not designed for Git merging. Remote backends with encryption are the standard.

**Red Flag 6: "I use terraform apply -auto-approve in production."** Auto-approve bypasses the most important safety mechanism in the IaC workflow. Acceptable in CI/CD only after a manual approval step has already happened.

**Red Flag 7: "Provisioners are the best way to configure servers."** Provisioners are a last resort, not a primary strategy. The correct approach is to use configuration management tools or immutable images.

**Red Flag 8: "I have never had a Terraform apply fail."** Either the candidate is not working on real infrastructure, or they are not paying attention. Failure is a routine part of IaC operations. The correct answer describes how you handle failures, not that you avoid them.

**Practice Drills**

**Drill 1: 2-minute explain — "Explain Terraform state to a junior engineer"**

Set a timer for 2 minutes. Your explanation must cover:
- What the state file is (a JSON document mapping Terraform resources to real cloud resource IDs)
- Why it exists (Terraform needs to know what it already created so it does not duplicate resources)
- Where it is stored (local disk for development, remote backend for team use)
- What happens if it is lost (Terraform cannot manage existing resources)

**Drill 2: 5-minute design — "Design an IaC setup for a startup with 3 engineers"**

The startup has:
- One AWS account
- 2 environments (dev, prod)
- 4 microservices, each with a database
- All in us-east-1

Design the Terraform structure, state management, and deployment process.

**Drill 3: 10-minute deep dive — "Design IaC for a multi-region, multi-account healthcare platform"**

The platform must:
- Operate in 3 AWS regions (us-east-1, eu-west-1, ap-southeast-1)
- Have separate accounts for dev, staging, prod, and security
- Comply with HIPAA (all data encrypted, all access logged)
- Support 50 microservices owned by 10 teams
- Each team can deploy independently

Walk through: directory structure, state management, CI/CD pipeline, security controls, compliance enforcement, disaster recovery.

**Drill 4: 5-minute debugging — "State file is corrupt. Walk through recovery."**

Your CI/CD pipeline fails with: "Error loading state: unexpected end of JSON input"

Walk through:
1. How to confirm the state file is corrupt (check file size, try to parse it)
2. How to recover (restore from S3 versioning, restore from backup, or rebuild from cloud resources)
3. How to prevent recurrence (S3 versioning enabled, state file backups, monitoring state file size)

**Drill 5: 10-minute design — "Design a Terraform module for deploying a web application"**

The module must:
- Accept inputs: app name, environment, instance type, min/max instances, VPC ID
- Create: launch template, auto-scaling group, ALB target group, security group, CloudWatch alarms
- Output: ALB DNS name, auto-scaling group name, security group ID
- Include secure defaults: health check path, encryption at rest, no public IPs

Write the module interface (variables and outputs) and explain your design choices.

**Drill 6: 10-minute design — "Design an IaC migration strategy for a legacy monolith"**

The company has 200 servers managed manually, 50 databases in RDS, and 100 S3 buckets. All created over 5 years through the AWS console. Design a phased migration to Terraform with zero downtime.

**Key points that a strong answer covers:**
1. Phase 1 — Discovery and inventory using AWS Config, Resource Groups Tag Editor, and AWS CLI
2. Phase 2 — State bootstrap using `terraform import` for each resource
3. Phase 3 — Module creation for common patterns (web server, database, storage)
4. Phase 4 — Gradual cutover, one service at a time, with rollback plans
5. Phase 5 — IAM lock-down to enforce IaC-only going forward
6. A clear rollback procedure: if a migrated service has issues, the Terraform state for that service is removed and the service continues to be managed manually until the issue is resolved

**Model answer for Drill 5:**

```hcl
variable "app_name" {
  description = "Application name used for resource naming"
  type        = string
}

variable "environment" {
  description = "Environment name (dev, staging, prod)"
  type        = string
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.medium"
}

variable "min_size" {
  description = "Minimum number of instances"
  type        = number
  default     = 2
}

variable "max_size" {
  description = "Maximum number of instances"
  type        = number
  default     = 10
}

variable "vpc_id" {
  description = "VPC ID to deploy into"
  type        = string
}

output "alb_dns_name" {
  description = "DNS name of the ALB"
  value       = aws_lb.main.dns_name
}

output "asg_name" {
  description = "Name of the auto-scaling group"
  value       = aws_autoscaling_group.main.name
}

output "security_group_id" {
  description = "Security group ID for the web tier"
  value       = aws_security_group.web.id
}
```

Design choices: (1) Sensible defaults for instance type and min/max size so the module works out of the box. (2) Validation on the environment variable to catch typos early. (3) The VPC ID is required (no default) because the module should not create its own VPC — it should be injected by the parent configuration.

**Additional Scenario: Multi-Cloud IaC Strategy**

**Question:** "Design an IaC strategy for a company that uses both AWS and GCP, with 10 services in each cloud. The team wants a single tool and consistent workflow."

**Model answer:** "I would use Terraform with a provider-agnostic module structure. Each service gets a module that defines its infrastructure abstractly, with the cloud provider selection as a variable. The module uses Terraform's `count` and `for_each` with provider-specific resources behind conditional logic.

The directory structure would be:

```
terraform/
├── modules/
│   ├── compute/            # EC2 (AWS) or GCE (GCP)
│   ├── database/            # RDS (AWS) or Cloud SQL (GCP)
│   └── networking/          # VPC (AWS) or VPC (GCP)
├── environments/
│   ├── aws-us-east-1/
│   ├── aws-eu-west-1/
│   ├── gcp-us-central1/
│   └── gcp-europe-west1/
└── providers/
    ├── aws.tf
    └── gcp.tf
```

The key trade-off is abstraction vs. native features. Abstracting away provider differences means you lose access to provider-specific features (AWS Lambda vs GCP Cloud Functions have different APIs). I would accept this trade-off because the team values consistency over using every native feature.

The CI/CD pipeline runs the same Terraform code against both clouds. State files are stored in a shared S3-backed backend (AWS) with a DynamoDB lock table. GCP engineers access the state via `terraform_remote_state` data sources that read from the S3 bucket using AWS credentials stored as CI/CD secrets."

**Additional Practice Drill: 15-minute Multi-Account Migration**

**Scenario:** Your company has 200 AWS EC2 instances, 50 RDS databases, and 100 S3 buckets spread across 2 AWS accounts. All were created manually. You need to migrate everything to Terraform. Design the migration plan with zero downtime.

**Key points a strong answer covers:**
1. Discovery phase: Use AWS Config, Resource Groups, and Tag Editor to inventory all resources across both accounts
2. Import phase: Use `terraform import` for each resource, working one service at a time
3. State organization: Each account gets its own state file, with separate states for network, compute, and storage
4. Rollback plan: Each import is reversible — if the Terraform config does not match, the import can be removed and the resource continues to be managed manually
5. Validation: After each service is imported, run `terraform plan` and verify it shows "No changes"
6. Cutover: Once all resources are imported, enforce IaC-only policy through IAM — remove console write permissions for all users except the break-glass admins

**Common Mistake Candidates Make:** Trying to migrate everything in one weekend. The correct approach is a phased migration over weeks, with each phase validated before moving to the next.

**Additional Interview Phrases:**

**When discussing the organizational impact of IaC:**
- "IaC is a force multiplier for platform engineering teams. A well-designed module registry allows a team of 3 platform engineers to support 50 application teams."
- "The biggest cost of IaC is not the tooling — it is the organizational change management. Training, documentation, and migration support are 80% of the effort."

**When discussing failure recovery:**
- "In a disaster recovery scenario, the IaC configuration is your runbook. You can recreate the entire infrastructure from scratch in a new region by running `terraform apply` with a different backend configuration."
- "The recovery time objective (RTO) for infrastructure is directly proportional to how well your IaC is organized. If the config is a monolith, the RTO is hours. If it is modular, the RTO is minutes."

**When discussing multi-cloud:**
- "Multi-cloud IaC is always a trade-off between abstraction and leverage. The more you abstract away provider differences, the less you can use each provider's unique capabilities."
- "I recommend against multi-cloud for most companies. The operational complexity of maintaining expertise in two clouds is rarely justified. Choose one cloud and use IaC to manage it well."

**When discussing security:**
- "The state file is the crown jewel of your IaC setup. It contains every secret, every IP address, every configuration detail. It must be encrypted, access-controlled, and backed up with the same rigor as your production database."
- "Policy as Code is the security boundary that prevents 'I didn't know that was a security issue' mistakes. It catches the S3 bucket that should have been private but was accidentally configured as public."

**Sample Interview Questions with Model Answers**

**Question: "Walk me through a time when Terraform broke your infrastructure."**

**Model answer:** "I was working on a migration from an older Terraform setup to a modular one. We had a monolith state file managing 200 resources. I needed to split it into service-specific state files. I used `terraform state mv` to move resources between state files. The process went smoothly for the first 180 resources. Then I moved a security group that was referenced by an EC2 instance. The move succeeded, but the new state file did not have the correct dependency mapping. The next `terraform plan` showed that the EC2 instance needed to be recreated because the security group reference was lost. I caught this in the plan output before applying. The fix was to update the EC2 instance configuration to explicitly reference the security group by its ID in the new state file. The lesson: always verify the plan output after state moves, and never assume the move preserved all dependencies correctly."

**Question: "How would you convince a skeptical team to adopt IaC?"**

**Model answer:** "I would not try to convince them with arguments about best practices. I would show them a concrete example. Pick the most painful manual process the team deals with — something that takes hours, breaks frequently, and requires tribal knowledge. Write a Terraform configuration for it. Open a PR. Let them see the plan output. Let them see that the plan catches mistakes before they happen. Let them see that the apply takes 30 seconds instead of 2 hours.

The second step is to address their fear about losing control. Create a break-glass procedure. Show them that IaC does not prevent emergency changes — it just requires that the change is documented afterward. Once they see that they can still fix production issues quickly, the resistance drops.

The third step is to make them the heroes. Give them the credit when the IaC setup prevents an incident. Celebrate the first time someone catches a mistake in a plan review. The cultural shift from 'I fix things' to 'I prevent things' is the real transformation."

**Question: "Design a Terraform module for a multi-tier web application."**

**Model answer:** "The module would have three sub-modules: networking, compute, and database.

The networking sub-module creates a VPC with public and private subnets across two availability zones. It creates an internet gateway for the public subnets and NAT gateways for the private subnets. It outputs subnet IDs and security group IDs.

The compute sub-module takes subnet IDs and security group IDs as inputs. It creates an auto-scaling group with a launch template. The launch template specifies the AMI, instance type, and user data script. The auto-scaling group has health checks, min/max sizes, and scaling policies. It also creates an ALB in the public subnets and a target group for the auto-scaling group.

The database sub-module creates an RDS instance in the private subnets. It accepts the VPC ID and subnet IDs. It creates a security group that only allows traffic from the compute tier's security group. It enables Multi-AZ, automated backups, and encryption.

The module is called once for each environment with different parameters:

```hcl
module "production" {
  source          = "./modules/web-app"
  environment     = "prod"
  instance_type   = "m5.xlarge"
  min_instances   = 5
  max_instances   = 20
  db_instance_class = "db.r5.xlarge"
  db_storage_gb   = 500
  enable_multi_az = true
}
```"

**Question: "What happens when a Terraform provider has a bug?"**

**Model answer:** "Provider bugs are rare but they happen. When they do, the symptoms are usually:
- A resource is created but the provider reports an error, leaving the resource in a "tainted" state
- A resource is updated but the provider stores incorrect values in the state
- A resource fails to be destroyed because the provider's delete function has a bug

When this happens, the first step is to check if the bug is known. Search the provider's issue tracker on GitHub. If it is a known bug, there is usually a workaround documented in the issue.

If it is not a known bug, file a new issue with:
- The Terraform version and provider version
- The minimal configuration that reproduces the bug
- The full error output
- Whether `terraform plan` shows the expected changes

Meanwhile, work around the bug:
- For create bugs: use `terraform state rm` to remove the resource from state and recreate it manually
- For update bugs: use `terraform state replace-provider` to force a state refresh
- For destroy bugs: manually delete the resource through the cloud console, then use `terraform state rm` to remove it from state

In extreme cases, downgrading the provider to a previous version can bypass the bug until a fix is released."

**Drill 9: 10-minute debugging — "Terraform plan shows 200 resources to destroy"**

You run `terraform plan` in production and it shows 200 resources to destroy, 0 to create, 0 to change. The infrastructure is running and healthy. Walk through the debugging process.

**Key steps:**
1. Check if the state file is corrupt or empty — is it the right state file?
2. Check if the backend configuration changed — are you pointing to the right S3 bucket?
3. Check if someone ran `terraform state rm` on all resources
4. Check if the entire configuration file was replaced or emptied
5. Run `terraform plan -refresh-only` to sync state without changes
6. If state is empty, restore from S3 versioning or backup
7. If configuration is empty, restore the configuration from Git

**Mock Interview Walkthrough: Full 45-Minute Session**

Below is a simulated interview session for an L5 candidate answering an IaC design question. Use this to practice your own pacing and responses.

**Interviewer:** "Design an infrastructure-as-code system for a company with 100 microservices, 20 engineers, and 3 environments running on AWS."

**Candidate (0-2 min — clarify):** "Let me clarify a few things. First, are all 100 microservices currently running, or are we designing for future growth? Second, do you have compliance requirements? Third, what is the current state of infrastructure management — is it manual, partially automated, or greenfield?"

**Interviewer:** "Assume 50 services exist today, 50 planned. No specific compliance requirements but security is important. Currently everything is managed through the AWS console. You are starting from scratch with IaC."

**Candidate (2-5 min — estimate):** "For 100 microservices, I would estimate approximately 20-30 infrastructure resources per service — load balancer, target group, auto-scaling group, security groups, IAM role, ECR repository, CloudWatch alarms, and potentially a database or cache. So roughly 2,000-3,000 resources total. This means a single state file would be too large — approximately 10-15 MB and plan times would be 10-20 minutes. I need to split state by service or by team."

**Candidate (5-15 min — high-level design):** "Here is my proposed structure:

Directory structure: single repository with a terraform/ directory containing modules/ and environments/. Modules include vpc, ecs-service, rds-database, redis-cache, and iam-role. The environments directory has dev/, staging/, and prod/ folders, each with their own state configuration pointing to S3 with DynamoDB locking.

State organization: one state file per service per environment. So dev/service-a, dev/service-b, etc. This keeps each state file under 500 resources and plan times under 2 minutes.

CI/CD: GitHub Actions with two workflows. PR workflow runs terraform plan. Merge workflow runs terraform apply. Production has a manual approval gate."

**Candidate (15-30 min — deep dive):** "Let me dive deeper into the module design. The ecs-service module accepts the service name, environment, container image, CPU/memory, and scaling parameters. It creates an ECS task definition, service, target group, security group, CloudWatch log group, and alarms. The module enforces encryption, health checks, and logging by default. Teams can override scaling parameters but cannot disable security features.

For secrets management, I would use AWS Secrets Manager. Each service gets a secret for its database password, which is automatically rotated. Terraform references the secret ARN, never the secret value.

For the CI/CD pipeline, every PR triggers a plan. The plan output is posted as a PR comment. A GitHub action checks that no more than 10 resources are changed per PR — large changes must be broken into smaller PRs. For production, a designated approver reviews and approves the plan."

**Candidate (30-40 min — trade-offs):** "The key trade-offs I want to call out:

Monorepo vs multi-repo: I chose a monorepo because it makes cross-service refactoring easier and provides a single source of truth. The downside is that all teams share the same repository, which can cause merge conflicts. I mitigate this by enforcing per-team directory ownership.

Single state file vs per-service: Per-service state files increase operational complexity (more backends to manage) but provide faster deploys and better isolation. When service-a's apply fails, service-b's deployment is not affected.

Terraform vs alternatives: Terraform is the right choice here because of its large provider ecosystem and mature state management. Pulumi would work if the team prefers TypeScript, but Terraform has better community support and more documentation for AWS."

**Candidate (40-45 min — wrap up):** "To summarize: monorepo with modules, per-service state files in S3 with DynamoDB locking, CI/CD with plan review and manual production gate, secrets via AWS Secrets Manager, drift detection running nightly, and a break-glass procedure for emergencies. The system scales to 100 services, supports 20 engineers working simultaneously, and provides a complete audit trail for every infrastructure change."
