## Tier 1 — Foundation (5 minutes each)

### Challenge 1: Explain TCP Slow Start

**Prompt**: Open a terminal on your machine. Find a network interface (eth0, en0, wlan0, etc.) and run a traceroute or ping to google.com. Now, in your own words, explain how TCP slow start works and why the first few packets in a TCP connection use the smallest window size.

**Evaluation criteria**:
- Describes slow start correctly: initial cwnd = 10 segments (Linux default), doubles every RTT
- Explains the rationale: the sender does not know the available bandwidth until it probes
- Mentions ssthresh (slow start threshold): typically 64 KB, after which additive increase takes over
- Connects to real-world observation: the traceroute output shows increasing RTT as the cwnd opens

**Model answer**: TCP slow start begins with a congestion window of 10 segments (10 * 1460 = 14,600 bytes). The sender transmits 10 segments and waits for ACKs. For every ACK received, the window increases by 1 segment. This means the window doubles every RTT (exponential growth). The reason: the sender has no idea what the bottleneck bandwidth is. Starting small means a slow link does not get overwhelmed while the sender probes its capacity. Once the window reaches the ssthresh (slow start threshold), TCP transitions to congestion avoidance (additive increase at +1 segment per RTT). In the traceroute output, you may see the first hop return faster than later hops because the first 10 segments fill a small part of the buffer, while later segments fill more.

### Challenge 2: Identify the Protocol

**Prompt**: A service sends data over the network. The client sends a datagram, and 3ms later the server sends a response. The client sends another datagram one second later. There is no handshake at connection startup. If a datagram is lost, the application does not notice. Which transport protocol is this, and what are the implications?

**Evaluation criteria**:
- Correctly identifies UDP (connectionless, no handshake, no built-in loss detection)
- Explains that the application does not know if its request was received
- Describes at least one real-world scenario where this is correct behavior (DNS, monitoring, heartbeats)
- Describes at least one scenario where this is dangerous (financial transaction, state-changing operation)

**Model answer**: This is UDP. The absence of a handshake is the defining characteristic — UDP sends each datagram independently without establishing a connection first. The 3ms response time is the application round-trip time, not the protocol setup time. The implication is that the application is responsible for detecting and handling lost datagrams. This is correct for DNS (resolver just retries with a different server), monitoring (next metric push replaces the lost one), or heartbeats (if you miss one, wait for the next). It is dangerous for any operation that changes state — a payment authorization, a database write, or a user account update — because the application cannot distinguish between "the server did not receive my request" and "the server received it and the response was lost."

### Challenge 3: Trace the Packet

**Prompt**: A web browser sends an HTTP request to https://example.com. List every TCP header field that is set in the first data-carrying packet after the handshake. Explain what each field means and what value it likely holds.

**Evaluation criteria**:
- Lists at least 5 TCP header fields correctly (source port, dest port, sequence number, ACK number, flags, window size, checksum)
- Explains source port is ephemeral (random high port), destination port is 443
- Explains sequence number = ISN + 1 (client's ISN incremented after SYN)
- Explains ACK number = server ISN + 1 (acknowledging the server's SYN-ACK)
- Explains flags: ACK is set (this is not a SYN), PSH may be set (push data to application)

**Model answer**: The first data-carrying packet after the three-way handshake has these TCP header fields:
- Source port: ephemeral port (e.g., 54321), randomly chosen by the OS
- Destination port: 443 (HTTPS)
- Sequence number: client_ISN + 1 (e.g., 1001 if ISN was 1000; SYN consumed one sequence number)
- ACK number: server_ISN + 1 (e.g., 5001 if server ISN was 5000)
- Flags: ACK set (always set after SYN), PSH set (tell the receiver to push data to the application immediately, not buffer)
- Window size: typically 65535 or larger with window scaling (tells the server how much data the client can accept)
- Checksum: TCP checksum over the pseudo-header and payload
- Options: Maximum Segment Size (MSS = 1460), window scaling factor (e.g., 7 = multiply window by 128), timestamp (for RTT estimation and PAWS — protection against wrapped sequences)
- Data offset: typically 5 (20 bytes of TCP header, no options) or more if options present

## Tier 2 — Design (15 minutes each)

### Challenge 4: Design a Real-Time Notification System

**Prompt**: Design a transport protocol for a push notification system that delivers alerts to 1 million mobile devices. Notifications must arrive within 5 seconds of being published. The system tolerates 0.1% notification loss (99.9% reliability target). Each notification is 4 KB. The system publishes notifications in bursts of 50,000 at peak (breaking news events).

**Scale**: 1 million devices, 50,000 notifications per burst, 4 KB per notification = 200 MB per burst.

**Design considerations**:
- TCP would require 1 million connections — each consuming 128 KB of kernel buffer = 128 GB of RAM
- TCP handshake for each device would take 200ms+ per device for push
- UDP eliminates connection state but loses 0.1% of notifications

**Rubric**:
- Transport choice explained with constraint analysis (4/4)
- Connection management strategy described (3/4)
- Loss detection and retry mechanism specified (4/4)
- Capacity math present (3/4)
- Scale limits identified (2/4)

**Model answer**: Use UDP for the push delivery channel and TCP for registration and retry. The push server maintains no per-device state. Each notification carries a sequential notification ID and a message-level checksum in the application header. Devices detect gaps in notification IDs and request retransmission of missing IDs over a TCP connection to the retry endpoint. For the 0.1% lost notifications: 50,000 x 0.001 = 50 missed per burst. Each missed device opens a TCP connection to the retry endpoint, sends the last received ID, receives the missing notifications, and closes. This is 50 TCP connections total versus 1 million — a 20,000x reduction. The push server uses SO_REUSEPORT with 8 worker processes to handle 200 MB bursts at line rate. Capacity: 50,000 x 4 KB = 200 MB per NIC at 25 Gbps = 64ms per burst. The retry endpoint needs capacity for 50 connections per burst — trivial.

### Challenge 5: Design a Video Conferencing Transport

**Prompt**: Design the transport architecture for a video conferencing system that must operate with sub-200ms end-to-end latency. The system uses VP9 or AV1 codecs. Participants can be on WiFi (up to 5% packet loss), cellular (up to 10% loss), or wired Ethernet (under 0.1% loss). Handling 10 participants per call.

**Scale**: 10 participants per call. Video bitrate: 2 Mbps average. Audio bitrate: 64 Kbps. Total throughput per participant: approximately 2.1 Mbps.

**Design considerations**:
- TCP's retransmission at 5% loss causes 200ms+ stalls — breaks the sub-200ms latency
- UDP's lack of reliability at 10% loss produces visible artifacts
- Forward error correction adds 50% overhead but eliminates retransmission delay

**Rubric**:
- Transport choice with per-stream justification (4/4)
- FEC or error correction strategy described (3/4)
- Audio vs video vs control channel separation (3/4)
- Jitter buffer sizing with numbers (3/4)
- Bandwidth estimation technique specified (3/4)

**Model answer**: Use UDP for audio and video streams. Use TCP for control signaling (join, leave, mute, screen-share toggle) where reliability matters and latency tolerance is higher (500ms+). Audio and video each get their own UDP port pair (or same port with stream IDs in the application header). Forward Error Correction: every 4 video packets carry 1 redundant FEC packet (20% overhead). At 5% loss, FEC recovers all lost packets. At 10% loss, FEC recovers most but not all — remaining loss is concealed by the decoder. Audio uses higher redundancy (every 2 packets carry 1 FEC packet, 50% overhead) because audio loss is more perceptible. Jitter buffer: 80ms for video, 40ms for audio (audio needs lower delay but higher tolerance for loss concealment). Bandwidth estimation: use Google's GCC (Google Congestion Control) algorithm, which estimates bandwidth from packet arrival patterns and adjusts bitrate every 200ms. At 5% loss, the video encoder drops from 2 Mbps to 1.5 Mbps. At 10% loss, it drops to 1 Mbps. Audio stays at 64 Kbps. Total UDP packets per second: 50 video (4 KB frames, ~41 ms per frame), 50 audio (160 bytes per 20ms packet), 10 FEC = 110 packets/sec per participant.

### Challenge 6: Design a Cross-Region Database Replication Protocol

**Prompt**: Design a transport layer for synchronous cross-region database replication between US East (us-east-1) and EU West (eu-west-1). RTT is 80ms. Target: the write must be applied on both regions before the client gets confirmation. Writes are 100 bytes to 10 KB in size. Throughput target: 10,000 writes per second.

**Scale**: 80ms RTT, 10,000 writes/sec, each write confirmed by both regions before client response.

**Design considerations**:
- Synchronous replication adds at least 80ms to every write latency — cannot avoid this
- TCP's three-way handshake per write would add 200ms + slow start = unacceptable
- Connection pooling is mandatory
- A single TCP connection with pipelining maximizes throughput

**Rubric**:
- Connection management strategy with numbers (4/4)
- Pipelining or batching mechanism described (4/4)
- Failure detection and retry semantics (3/4)
- Capacity calculation for bandwidth-delay product (3/4)
- Timeout and consistency guarantee explanation (2/4)

**Model answer**: Use a persistent TCP connection (pooled) between the two regions. One connection, no handshake overhead after setup. Pipeline writes: the sender transmits write 1, write 2, write 3, ..., write N on the same TCP connection without waiting for individual ACKs. The receiver acknowledges each write as it persists to disk. The maximum number of in-flight writes equals bandwidth-delay product: 10 KB * 10,000 writes/sec = 100 MB/s throughput. At 80ms RTT, the in-flight data is 100 MB/s * 0.08s = 8 MB. At 10 KB per write, this is 800 concurrent in-flight writes. The TCP send buffer must be at least 8 MB per connection — set `tcp_wmem` max to 16 MB to be safe. Confirmation arrives in exactly 1 RTT (80ms) + processing time (5-10ms) = approximately 90ms p50. Failure: if the TCP connection drops mid-transfer, writes that were sent but not confirmed are retried on the new connection. Use idempotency keys on each write so the receiver deduplicates. Consistency: synchronous replication means the client sees latency equal to the write-through time. If the cross-region link becomes partitioned, the system must either refuse writes (CP) or accept writes in one region and replicate asynchronously (AP). Define this choice before implementation.

## Tier 3 — Expert (30 minutes each)

### Challenge 7: Design a Transport for the Interplanetary Internet

**Prompt**: You are designing the transport layer for a communication system between Earth and a Mars base. The one-way light time varies from 3 minutes to 22 minutes depending on orbital positions. Round-trip time varies from 6 to 44 minutes. The link has 1 Mbps bandwidth. Packet loss is approximately 1% due to cosmic radiation. Data is 1 MB images, 100 MB video clips, and real-time telemetry (100 bytes every 10 seconds). TCP is fundamentally broken at this RTT. Design a UDP-based transport that achieves 90%+ link utilization while tolerating the loss rate.

**Scale**: 1 Mbps link, 6-44 minute RTT, 1% packet loss, 1% cosmic ray bit-flip rate.

**Design considerations**:
- TCP's retransmission timeout (RTO) would be 6-44 minutes — a single lost packet costs 6+ minutes
- TCP's slow start would take hours to reach 1 Mbps at 44-minute RTT
- CUBIC and BBR both assume sub-second RTT feedback
- ACKs returned 44 minutes later are useless for window-based adaptation

**Model answer**: Use a rate-controlled datagram protocol with erasure coding.

File transfer: split each file into N blocks of 1400 bytes each. Compute K = N * 0.33 additional Reed-Solomon parity blocks. Continuously transmit all N+K blocks in a circular carousel. The receiver needs any N of N+K blocks to reconstruct the file. At 1% block loss and 33% parity overhead, the probability that fewer than N of N+K blocks arrive is under 1 in 10^100 (essentially zero). No ACKs. The transmitter sends at 900 Kbps (90% link utilization, 10% margin for fluctuation). The carousel cycle time for a 1 MB file at 900 Kbps is 1 MB / 112 KBps = approximately 9 seconds. The receiver collects all N+K blocks in approximately 3 cycles (27 seconds worst case), regardless of which specific blocks were lost each cycle.

Telemetry: telemetry (100 bytes) is sent as a single UDP datagram every 10 seconds with sequence numbers. No parity blocks needed — the next telemetry point (10 seconds later) supersedes a lost one.

This design achieves 90% link utilization, zero protocol overhead from ACKs (all bandwidth goes to payload), and tolerates up to 25% block loss. The downside: 27-second delivery delay for files — but on a 44-minute RTT link, that is acceptable.

### Challenge 8: Debug a Production Incident Involving Both TCP and UDP

**Prompt**: Your team runs a high-frequency trading system. Market data arrives over UDP (NASDAQ's OUCH protocol). Trade orders are sent over TCP to the exchange. At 9:30 AM market open, you see the following symptoms:

1. The trade execution rate drops from 500 orders per second to 10 orders per second for 3 minutes.
2. Market data packet loss spikes from 0.01% to 5%.
3. The TCP retransmission rate on the order connection jumps from 0.1% to 12%.
4. CPU usage on the trading server stays at 40%.
5. Memory usage stays flat.

The network team reports no link errors, no packet drops at the switch, and no congestion on the 10 Gbps link (utilization: 2 Gbps). Diagnose the problem. The cause is not obvious.

**Scale**: 10 Gbps link at 2 Gbps utilization. 500 orders/sec TCP. 5% packet loss on UDP. Everything correlates but root cause is not link-level.

**Design considerations**:
- Hint: the symptoms correlate (TCP retransmission spikes at the same time as UDP loss spikes)
- Hint: no link errors, no switch drops, and only 2 Gbps of 10 Gbps used
- Hint: CPU and memory are fine — the problem is not server overload
- Root cause: UDP receive queue overflow on the server causes the kernel to drop UDP packets silently, AND the same kernel CPU starvation causes delayed ACK processing for TCP, which triggers TCP retransmissions.

**Discussion points**:
- The market data feed at 500,000 packets per second (OUCH protocol) arrived in a burst at market open (the "opening print" burst — all symbols print their opening trade simultaneously)
- The server's UDP receive queue (default 208 KB = ~200 datagrams) overflowed but no error is visible in `netstat` because the application processes UDP in the same thread
- The kernel's softirq handler, overwhelmed by the UDP burst, delays processing TCP ACK arrivals
- The exchange's TCP stack sees delayed ACKs and assumes packet loss, triggering fast retransmit
- The trade execution rate drops because each order takes longer to confirm (due to TCP retransmission)

**Model answer**:
Root cause: UDP receive queue overflow on port 11000 (market data feed). At market open, all symbols send their opening prints simultaneously, creating a micro-burst of 50,000 packets in approximately 2 milliseconds. The UDP receive queue overflows. The kernel silently drops datagrams (UdpRcvbufErrors increments). The application reads what it can from the receive queue but processes fewer updates than expected. The application thread's processing of the UDP data (order book updates) takes 15 microseconds per update. With only 200 datagrams per batch (receive queue size), the application spends 3ms draining the queue. Meanwhile, the next 50,000 datagrams arrive — but the queue refills before the application returns to the event loop, causing persistent overflow.

Collateral effect: the kernel's network softirq handler spends 90% of its time processing UDP bursts. TCP ACK processing is starved. The exchange sees its ACKs for trade orders delayed by 100-500ms. The exchange's TCP stack has a retransmission timeout of 200ms. It retransmits orders that were already delivered. The server sees duplicate orders and must deduplicate. The duplicate processing further slows the application, worsening the TCP ACK delay. The system enters a self-reinforcing degradation cycle.

Detection: `netstat -s | grep UdpRcvbufErrors` shows 30,000 increments in 5 seconds. `cat /proc/net/softnet_stat` shows the third column (processed in softirq) exceeding the first column (received) by a factor of 3 — indicating the softirq backlog is falling behind.

Fix: Increase UDP receive buffer to 16 MB for the market data socket. Use `SO_REUSEPORT` to distribute the UDP feed across 4 worker threads (4 receive queues, each with 16 MB). Increase `net.core.netdev_max_backlog` from 1000 to 10,000 to absorb the initial burst. Apply `tcp_congestion_control=bbr` to make TCP less sensitive to ACK delay (BBR probes rate, not loss). Pre-allocate the order book data structures before market open to avoid JIT or GC pauses during the opening burst.

Total time to diagnose: 30 minutes. Time to implement fix: 2 hours. Time to verify: next market open.
