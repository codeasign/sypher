## Company Case Studies

### Netflix — Open Connect CDN

Netflix serves over 200 million subscribers streaming 140 million hours of content per day. The content delivery is an engineering challenge on an enormous scale.

**Scale**: 15+ terabits per second peak egress. 4000+ Open Connect appliances deployed at ISP edge locations.

**Problem**: Delivering streaming video at high bitrate (4K HDR at 25 Mbps) with zero rebuffering events over an unreliable internet. TCP's retransmission behavior causes playback stalls when packets are lost — the video player waits for the retransmitted segment before it can decode the next keyframe.

**Solution**: Netflix built their own TCP stack called "Guevara" (patches to Linux TCP). Key innovations:
- **TCP pacing**: Send packets at a smooth rate rather than in bursts (TCP's default behavior sends a full window at once, causing micro-bursts that overflow buffer).
- **FQ (Fair Queuing) CoDel within the ISP appliance**: The Open Connect appliance uses active queue management to keep latency low, preventing bufferbloat on the last-mile link.
- **Application-layer bitrate adaptation**: The client measures actual TCP throughput and requests the appropriate bitrate from the ABR (Adaptive Bitrate) ladder.

### Uber — Ringpop for Service Discovery

Uber's services needed fast detection of failed nodes in a cluster of thousands.

**Scale**: 2000+ microservices, 100,000+ container instances across multiple data centers.

**Problem**: Traditional service discovery using ZooKeeper or etcd (both TCP-based) could not converge fast enough when a node failed. A 60-second failure detection window caused cascading failures — clients sent requests to dead servers, got timeouts, and the timeout storms increased latency for surviving services.

**Solution**: Ringpop uses a SWIM-based (Scalable Weakly-consistent Infection-style protocol) gossip mechanism over UDP:
- Each member gossips its state to a random subset of the cluster every 200ms.
- A failure is detected when N consecutive ping-pongs (UDP echo) to a target are lost.
- Detection time: configurable, typically 5-10 seconds (versus 30-60 seconds for TCP-based health checks).
- UDP is chosen because gossip does not require reliability — if one gossip message is lost, the next one usually arrives. The cost of a false positive failure detection is a rebalancing event; the cost of slow detection is a cascading failure.

### Discord — Real-Time Voice with UDP

Discord serves 140 million monthly active users with real-time voice communication requiring under 200ms end-to-end latency.

**Scale**: 150 million active users. 4 million concurrent voice users at peak. 200 voice regions globally.

**Problem**: Voice packets are time-sensitive: a packet that arrives 300ms after it was spoken is useless. TCP's retransmission of lost packets would deliver the old voice data too late and cause the "robot voice" effect — the audio decoder receives packets out of order and produces garbled output.

**Solution**: Discord uses UDP for all voice traffic. The application layer implements:
- **Forward Error Correction (FEC)**: Each packet carries both the current audio frame and a redundant copy of the previous frame. If packet N is lost, packet N+1 fills the gap. Every other packet doubles as a repair.
- **Opus codec**: The codec has built-in packet loss concealment — it interpolates up to 20ms of lost audio from surrounding context.
- **Jitter buffer**: The client buffers 40-80ms of audio before playback, absorbing network jitter. If a packet arrives late (but within the buffer window), it plays normally.

### Cloudflare — HTTP/3 (QUIC) at Edge

Cloudflare handles 20% of the world's web traffic and was one of the earliest adopters of HTTP/3 (QUIC).

**Scale**: 20% of global web traffic. 28 million HTTP requests per second at peak. 330+ edge locations.

**Problem**: TCP+TLS connection setup requires 2-3 round trips (200-600ms on a typical mobile connection). For a mobile user switching between WiFi and cellular, TCP connections break (IP address changes), requiring a new handshake. This made the mobile web feel slow and unreliable.

**Solution**: Cloudflare deployed HTTP/3 (QUIC) on all 330+ edge locations:
- QUIC uses UDP and combines the TCP handshake with TLS 1.3 into a single round trip.
- Connection migration: when a mobile user switches from WiFi to cellular, the QUIC connection identifier (not IP address) identifies the session. The server continues the connection transparently.
- Zero RTT (0-RTT): repeat visitors send their HTTP request in the very first packet, eliminating the handshake entirely.

## Open Source Implementations

### Nginx — TCP and UDP Load Balancing

Nginx, serving 30%+ of the world's top websites, supports both TCP and UDP stream proxying.

**TCP proxy**: Nginx terminates TCP connections from clients and opens new TCP connections to upstream servers. It maintains a connection table mapping client tuples to upstream connections. Under the hood, it uses epoll (Linux) or kqueue (macOS) for event-driven connection handling.

**UDP proxy**: Nginx forwards UDP datagrams between clients and upstream servers. It maintains minimal state (source IP/port mapping to upstream) for 60 seconds by default. This is used for DNS, syslog, and QUIC traffic proxying.

```nginx
# TCP streaming (load balancing for Redis)
stream {
    upstream redis_backend {
        server 10.0.0.1:6379 max_fails=3 fail_timeout=30s;
        server 10.0.0.2:6379 max_fails=3 fail_timeout=30s;
    }
    server {
        listen 6379;
        proxy_pass redis_backend;
        proxy_connect_timeout 5s;
    }
}

# UDP streaming (DNS load balancing)
stream {
    upstream dns_backend {
        server 10.0.0.1:53;
        server 10.0.0.2:53;
    }
    server {
        listen 53 udp;
        proxy_pass dns_backend;
        proxy_timeout 2s;
    }
}
```

### HAProxy — Healthcare TCP

HAProxy is the standard for high-availability reverse proxying and uses TCP health checks to determine backend liveness. The `option tcp-check` directive performs a sequence of sends and expects on a TCP connection to verify not just that the port is open, but that the service is responding correctly.

```haproxy
backend mysql_replicas
    option tcp-check
    tcp-check connect
    tcp-check send PING\r\n
    tcp-check expect string +OK
    server db1 10.0.0.1:3306 check inter 1s fall 3 rise 2
```

HAProxy also supports UDP health checks via the `check` parameter on `udp` listeners, but UDP health checks can only verify port availability — they cannot distinguish between "the service is healthy" and "the service is accepting packets but returning garbage."

### Redis — Combined TCP and UDP

Redis uses TCP for all data commands (GET, SET, HGETALL, etc.) but supports optional UDP for `PUBLISH`/`SUBSCRIBE` notifications in Redis Sentinel (the high-availability module). The reasoning: losing one subscription notification to a sentinel is acceptable (the next heartbeat will catch it), but losing a SET command to the database is not.

### ZooKeeper and etcd — TCP Consensus

Both ZooKeeper and etcd run the Raft or Zab consensus protocols over TCP exclusively. A consensus round requires ordered, guaranteed delivery of proposal messages to all nodes. Losing a single proposal packet would break the consensus cluster's log consistency guarantees.

## Cloud Provider Implementations

### AWS

| Service | Transport | Data Plane | Why |
|---------|-----------|------------|-----|
| ALB (Application Load Balancer) | TCP | HTTP/2 multiplexed over TCP | Needs request/response ordering |
| NLB (Network Load Balancer) | TCP + UDP | Direct forwarding | TLS termination offloaded, preserves client IP |
| Route 53 (DNS) | UDP (primary), TCP (fallback) | UDP port 53 | Speed, 4K response via EDNS0 |
| ElastiCache (Memcached) | TCP + UDP | UDP gets, TCP sets | Same pattern as Memcached |

**Cost considerations**: AWS charges per load balancer capacity unit (LCU). A TCP LCU processes up to 800 new connections per second; a UDP LCU processes up to 400,000 packets per second. For high-traffic UDP workloads (DNS, gaming), NLB is dramatically cheaper per packet than ALB.

### GCP

GCP's Cloud Load Balancing uses Maglev (Google's custom load-balancing layer) with support for both TCP and UDP on the same forwarding rule. GCP's global load balancer terminates TCP near the user at the edge of Google's network, then forwards over Google's internal backbone.

**Key difference**: GCP supports QUIC (UDP-based HTTP/3) on all external load balancers. AWS does not offer ALB-level QUIC termination as of 2026 — you must terminate QUIC on your instances behind an NLB.

### Azure

Azure Load Balancer supports UDP directly. Azure Front Door (global HTTP load balancer) uses TCP+TLS termination at 160+ edge locations. Azure DNS supports UDP on port 53 with EDNS0.

## Lessons Learned

### 1. Know Your Loss Budget

Every production system should define its acceptable packet loss rate before choosing a transport:
- Financial transactions: loss budget = 0. Use TCP, always.
- Real-time voice: loss budget = 1-5%. Use UDP with FEC.
- Video streaming (on-demand): loss budget = 0. Use TCP with buffer.
- Video streaming (live): loss budget = 0.5%. Use UDP with retransmission of keyframes only.
- Gaming (state sync): loss budget = 10% (position updates). Use UDP with interpolation.

### 2. Always Layer Application Heartbeats on UDP

If you use UDP for any production service, implement a heartbeart mechanism. The kernel sends you no signal when a packet is lost. The application must detect gaps. Idiomatic pattern: every 5 seconds, the client sends a sequence-numbered echo request. If three consecutive responses are missing, declare the server dead.

### 3. Monitor Kernel Counters, Not Application Logs

By the time your application logs detect a transport issue, thousands of packets have already been affected. Monitor `TcpExtListenOverflows`, `UdpRcvbufErrors`, and `TcpExtLostRetransmit` at 1-second granularity through your monitoring system. These counters tick before the application sees any impact.

### 4. Respect Middlebox Surprises

Enterprise firewalls, load balancers, and cloud security groups apply different rules to TCP and UDP:
- Some enterprises block all UDP on port 443 (breaking QUIC and HTTP/3).
- Some cloud NAT gateways have different session timeouts for TCP (typically 350 seconds) and UDP (typically 30 seconds).
- Some load balancers cannot forward UDP traffic at all.
Always verify your transport choice is feasible through your entire network path, end to end.

### 5. Tune Before Rewriting

Before building a custom transport layer on top of UDP, tune TCP for your workload:
- Enable BBR congestion control: `sysctl -w net.ipv4.tcp_congestion_control=bbr`
- Enable `TCP_NODELAY` in your application code.
- Increase buffer sizes for high-bandwidth-delay-product links.
- Use connection pooling instead of one connection per request.
These changes typically deliver 90% of the benefit of a custom protocol at 1% of the implementation cost.
