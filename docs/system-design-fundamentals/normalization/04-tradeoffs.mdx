## Core Tension

Normalization reduces data duplication but increases query complexity. Every fact lives in one place, so updates are fast and consistent. But reading that fact requires navigating through joins, indexes, and foreign key relationships.

This is the fundamental trade-off: **you trade write efficiency and consistency for read overhead**.

A write to a denormalized table touches one row. A write to a normalized schema touches:
- 1 row in `customers`
- 1 row in `orders`
- 3 rows in `order_items`
- 3 FK checks across `products`, `orders`, and `customers`
- 5 index updates for FK columns

That is 4 row writes and 5 index updates versus 1 row write and 1 index update for denormalized. Write latency increases by 3x to 5x. But update consistency goes from "you might miss a row" to "guaranteed atomic."

## Decision Matrix

Seven key decision points determine where your schema lands on the normalization spectrum.

### Decision 1: 1NF vs Non-1NF

Choose 1NF unless the data is never queried by individual sub-values.

**1NF applies when**: You query individual values in the array. You need referential integrity. You update individual values.

**Skip 1NF when**: The column holds a monolithic value (JSON blob for a config object). You never filter on sub-values. The data is always read and written as a unit.

### Decision 2: 2NF vs 3NF

2NF removes partial dependencies. 3NF removes transitive dependencies. The difference matters when your entities have attributes that depend on other non-key attributes.

**2NF is sufficient when**: Entities have few attributes. No attribute depends on another non-key attribute. The schema is used for a single-purpose application.

**3NF is needed when**: Entities have complex relationships. Attributes like `department_building` depend on `department_id`, not `employee_id`. Multiple applications share the schema.

### Decision 3: BCNF vs 3NF

BCNF catches cases where 3NF misses redundancy due to overlapping candidate keys. In practice, 3NF is sufficient for 95 percent of production schemas.

**BCNF is needed when**: The table has two or more candidate keys that overlap. The overlap creates the potential for a non-candidate-key determinant. Example: `(student, subject, teacher)` where each teacher teaches one subject, but a subject may have multiple teachers.

**Stick with 3NF when**: The schema has one clear candidate key. Overlapping candidate keys do not exist. The team is not familiar with BCNF analysis.

### Decision 4: 3NF vs Denormalization

This is the most common trade-off decision in production systems.

**Stay normalized (3NF) when**: Write consistency is critical (financial transactions, inventory). Storage costs are a concern. The system has more writes than reads. Multiple applications or services share the database.

**Denormalize when**: Read performance is the bottleneck. The join overhead exceeds 40 percent of query time. The read-to-write ratio exceeds 50:1. The data is primarily read, rarely updated. You are pre-computing for an API endpoint.

### Decision 5: Strict FK vs Application-Level Integrity

**Strict FK constraints** guarantee referential integrity at the database level. The database prevents orphaned rows. Cascade operations handle related deletes. Cost: every INSERT and UPDATE checks the referenced table, requiring a lock on the referenced row.

**Application-level integrity** removes FK constraints and trusts the application to maintain relationships. Cost: orphaned rows can appear from bugs, race conditions, or failed deployments. Benefit: higher write throughput, no FK lock contention, easier sharding.

Use application-level integrity only when:
- The database must be sharded (FKs cannot span shards).
- Write throughput is the primary bottleneck.
- The application team can enforce consistency through application logic and reconciliation jobs.

### Decision 6: Cascade vs Restrict

**CASCADE**: Deleting a customer deletes all their orders and order items. Simple, automatic. Risk: accidental cascade storms that lock the database for minutes.

**RESTRICT**: Deleting a customer fails if they have orders. The application must explicitly handle the deletion logic. Safer but requires more code.

Use RESTRICT for production systems. Implement cascade logic in application code with explicit batching and progress tracking.

### Decision 7: Single-Column Identity vs Composite Natural Keys

Natural keys (ISBN for books, SSN for people) avoid extra join columns. Surrogate keys (auto-increment IDs) separate identity from business meaning.

| Aspect | Natural Key | Surrogate Key |
|---|---|---|
| Readability | Self-documenting | Opaque |
| Join performance | Wider keys, slower | Narrow integer, faster |
| Stability | Can change (e.g., ISBN reissue) | Never changes |
| Normalization | Same | Same |
| Migration difficulty | Very high if key changes | Trivial |

Use surrogate keys for all primary keys. Add unique constraints on natural keys for lookup queries.

## CAP Positioning

Normalized relational databases operate in the CP quadrant of the CAP theorem: Consistent and Partition-tolerant, but not Available during partitions.

When a network partition splits the database cluster, the system must choose between consistency and availability. A normalized RDBMS with foreign key constraints chooses consistency:

- The partition minority stops accepting writes. The database rejects any write that cannot be verified against all referenced tables.
- The partition majority continues operating. Foreign key checks succeed within the majority.
- After the partition heals, the minority re-joins and catches up from the write-ahead log.

The impact on normalized schemas: during a partition, writes that span multiple tables may fail even if the referenced rows exist. A write to `order_items` referencing `products` fails if the `products` table is on the minority side of the partition.

This is why normalized RDBMS are not ideal for multi-region active-active deployments. The consistency requirements force a single writer or a complex consensus protocol.

## Failure Modes

### Deadlocks from Multi-Table Transactions

Normalized schemas spread a single business operation across multiple tables. Placing an order: INSERT into `orders`, then INSERT into `order_items`, then UPDATE `inventory`. If two transactions access these tables in different orders, they deadlock.

**Example deadlock**:

```
Transaction A:                     Transaction B:
1. INSERT orders (id=100)         1. INSERT orders (id=200)
2. INSERT order_items             2. UPDATE inventory (product=42)
   (order_id=100, product=42)         
3. UPDATE inventory (product=42)  3. INSERT order_items
   ---- DEADLOCK DETECTED ----        (order_id=200, product=42)
```

Transaction A holds a lock on `orders` row 100 and needs the `inventory` lock. Transaction B holds the `inventory` lock and needs a lock on `order_items`. Neither can proceed.

**Prevention**: Access tables in the same order in all transactions. Always update `inventory` first, then insert `order_items`, then insert `orders`. This creates a consistent lock ordering.

### Cascade Storms

A single DELETE cascading to millions of rows:

```sql
-- This DELETE cascades to millions of rows
DELETE FROM customers WHERE id = 42;
-- Customer 42 has 50,000 orders
-- Each order has 3 order items = 150,000 rows
-- Plus addresses, payments, reviews, returns
-- Total: 200,000+ rows deleted
```

The cascade holds exclusive locks on every affected row. Other transactions cannot read or write these rows. If the delete runs for 30 seconds, all reads for those rows block for 30 seconds. Timeouts cascade to application servers. Users see 5xx errors.

**Prevention**: Batch deletes with explicit progress tracking:

```sql
-- Batch delete with sleep to reduce lock contention
DO $$
DECLARE
    batch_size INT := 1000;
    deleted INT;
BEGIN
    LOOP
        WITH batch AS (
            SELECT id FROM order_items
            WHERE order_id IN (
                SELECT id FROM orders WHERE customer_id = 42
            )
            LIMIT batch_size
            FOR UPDATE SKIP LOCKED
        )
        DELETE FROM order_items WHERE id IN (SELECT id FROM batch);
        GET DIAGNOSTICS deleted = ROW_COUNT;
        EXIT WHEN deleted = 0;
        COMMIT;
        PERFORM pg_sleep(0.1);  -- 100ms pause between batches
    END LOOP;
END $$;
```

### Slow Joins on Deep Normalized Hierarchies

A query joining 8 normalized tables for a single page:

```sql
SELECT * FROM customers c
JOIN orders o ON c.id = o.customer_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
JOIN product_categories pc ON p.category_id = pc.id
JOIN suppliers s ON p.supplier_id = s.id
JOIN addresses a ON o.shipping_address_id = a.id
JOIN shipments sh ON o.id = sh.order_id
WHERE c.id = 42;
```

This query joins 8 tables. The optimizer must choose among 8 factorial (40,320) join orders. A suboptimal choice leads to a 10-second query for a page that should load in 200 ms.

**Prevention**: Limit join depth to 4 tables in a single query. Use materialized views for deeper hierarchies. Denormalize the hot path.

### Lock Escalation in Normalized Schemas

When a transaction modifies many rows across multiple normalized tables, the database may escalate row-level locks to page-level or table-level locks. This blocks all access to affected tables.

Example: An ETL job updates 50,000 order records:

```sql
BEGIN;
UPDATE orders SET status = 'archived'
WHERE created_at < '2020-01-01';
-- 50,000 row locks acquired
-- Database may escalate to table-level lock
UPDATE order_items SET archived = true
WHERE order_id IN (
    SELECT id FROM orders WHERE created_at < '2020-01-01'
);
COMMIT;
```

During the lock escalation, no reads or writes to the `orders` or `order_items` tables are possible. For a busy OLTP system, this causes a minutes-long outage.

**Prevention**: Process deletes and updates in small batches (1,000 rows per batch). Use `pg_repack` or `pt-online-schema-change` for bulk operations. Schedule bulk jobs during maintenance windows.

### Index Bloat from FK Indexes

Every foreign key column needs an index. These indexes consume disk space and memory. For a normalized schema with 10 tables and 15 FK columns, that is 15 indexes — approximately 6 GB of index storage for a 48 GB database.

Index bloat occurs when indexes grow faster than the data they index. An FK index on `order_items.order_id` might be 50 percent larger than the data in the `order_items` table because of the overhead of storing and updating the index.

**Prevention**: Rebuild indexes periodically with `REINDEX` (PostgreSQL) or `OPTIMIZE TABLE` (MySQL). Monitor index-to-table size ratios using `pg_stat_user_indexes`.

### FK Constraint Failures During Replication Lag

When using read replicas, a common pattern is to write to the primary and immediately read from a replica. If the replica has not received the write yet, a FK lookup can fail:

```python
# Race condition with read replicas
def create_order(customer_id, product_ids):
    # Write to primary
    primary.execute("INSERT INTO orders (customer_id) VALUES (%s)", (customer_id,))
    order_id = primary.lastrowid

    # Read from replica for verification (lag: 50ms)
    # This might return 0 rows because the replica is behind
    result = replica.execute(
        "SELECT 1 FROM orders WHERE id = %s", (order_id,)
    )
    if not result:
        raise Exception("Order not found on replica")  # False positive
```

**Prevention**: Route read-after-write queries to the primary. Use session-level hints to force primary reads for the current transaction.

## Anti-Patterns

### Over-Normalization

Joining 15 tables to display a simple page. Each join adds 1-5 ms of latency. At 15 joins, the query takes 20-75 ms even with perfect indexes. Without indexes, the query takes seconds.

Symptoms: Every query joins more than 6 tables. The schema has tables with one or two columns. Developers complain about schema complexity. ORM queries generate SQL with 10+ JOIN clauses.

Fix: Denormalize the hot path. Create a summary table for the common query. Use a materialized view. Accept 2NF instead of insisting on BCNF.

### Under-Normalization

Storing JSON blobs in relational columns and then querying individual fields:

```sql
-- Anti-pattern: JSON blob in relational column
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_data JSONB,  -- Contains name, email, address, phone
    product_data JSONB,   -- Contains array of products with prices
    payment_data JSONB    -- Contains card type, last four, expiry
);

-- Querying inside JSON is slow and hard to index
SELECT * FROM orders
WHERE customer_data->>'email' = 'alice@example.com';
```

JSON is not a replacement for normalization. Use JSON for truly flexible schemas (user preferences, config settings). Use normalized columns for queryable data.

### One True Lookup Table

A single table that holds all reference data:

```sql
-- Anti-pattern: one true lookup table (OTLT)
CREATE TABLE lookup (
    id INT PRIMARY KEY,
    type VARCHAR(50),      -- 'category', 'status', 'role', 'country'
    code VARCHAR(50),
    value VARCHAR(200),
    description TEXT
);

-- Queries become confusing
SELECT l.value AS category_name
FROM lookup l
JOIN products p ON p.category_id = l.id
WHERE l.type = 'category';
```

**Problems**: Impossible to enforce referential integrity. One table has millions of rows across all types. Every query filters by type. Join performance degrades as the table grows. Foreign keys cannot reference specific subtypes.

**Fix**: Create separate tables: `product_categories`, `order_statuses`, `user_roles`, `countries`.

### Entity-Attribute-Value (EAV) Anti-Pattern

Storing entity attributes as rows instead of columns:

```sql
-- Anti-pattern: EAV
CREATE TABLE product_attributes (
    product_id INT,
    attribute_name VARCHAR(100),  -- 'color', 'weight', 'size', 'material'
    attribute_value VARCHAR(500),
    PRIMARY KEY (product_id, attribute_name)
);

-- Reconstructing a product requires 10+ joins or pivot queries
SELECT product_id,
    MAX(CASE WHEN attribute_name = 'color' THEN attribute_value END) AS color,
    MAX(CASE WHEN attribute_name = 'weight' THEN attribute_value END) AS weight
FROM product_attributes
GROUP BY product_id;
```

EAV appears normalized — each attribute is a separate row — but it destroys query performance. Use JSONB for variable attributes or create specific columns for common attributes and a JSONB column for rare ones.

### Multi-Column Join Anti-Pattern

Using multiple columns in a join when a single surrogate key would work:

```sql
-- Anti-pattern: joining on multiple natural key columns
SELECT *
FROM orders o
JOIN customers c
  ON o.customer_first_name = c.first_name
 AND o.customer_last_name = c.last_name
 AND o.customer_zip = c.zip_code;

-- Better: single surrogate key join
SELECT *
FROM orders o
JOIN customers c ON o.customer_id = c.id;
```

Multi-column joins are slower than single-column joins because the index must match all columns. The index on `(first_name, last_name, zip_code)` is larger and slower than an index on `id`. Use surrogate keys for all joins.

### Polymorphic Association Anti-Pattern

A foreign key column that references multiple tables:

```sql
-- Anti-pattern: polymorphic association
CREATE TABLE comments (
    id INT PRIMARY KEY,
    comment_text TEXT,
    source_type VARCHAR(20),  -- 'blog_post', 'product', 'video'
    source_id INT             -- FK to one of three tables
);

-- Cannot declare a proper foreign key
-- source_id could reference blog_posts.id, products.id, or videos.id
-- No referential integrity
```

The database cannot enforce the foreign key. An application bug can reference a non-existent source. Fix: create separate `blog_post_comments`, `product_comments`, and `video_comments` tables, or use a junction table.

### Premature Optimization Anti-Pattern

Denormalizing before measuring:

An engineering team hears that joins are slow. They denormalize the product name into the `order_items` table before the system has 1,000 orders. The denormalization adds update complexity (when a product name changes, all order_items must be updated) without any performance benefit — the join would have taken 2 ms.

Always measure before denormalizing. Run the normalized query with `EXPLAIN ANALYZE`. If the join time is under 5 ms, denormalization adds complexity without benefit.

## When NOT to Use

Normalization is not the right choice for every workload. Recognize these cases.

### Time-Series Data

Metrics, logs, and sensor readings are append-only. Each data point is independent. Updates are rare or nonexistent. Joins add no value.

**Store time-series data in**: InfluxDB, TimescaleDB, Prometheus, or a flat table with a timestamp index. Normalizing to 3NF adds write overhead without any read benefit.

### Event Logs

Application events, audit trails, and user activity streams are written once and read in bulk. Queries are range scans over time, not point lookups with joins.

**Store event logs in**: A wide table with a timestamp primary key. Consider columnar storage (Parquet, ORC) for analytics. Normalization adds complexity without benefit.

### High-Write-Volume Append-Only Workloads

A system processing 100,000 writes per second with zero updates or deletes. Normalization adds FK checks, index updates, and multi-table transactions. Each FK check is an extra index lookup. Each index update adds write amplification.

**Design for these workloads**: Use a log-structured merge-tree (LSM) database like Cassandra or ScyllaDB. Accept denormalized wide rows. If you must use an RDBMS, batch writes and defer constraint checking.

### Full-Text Search

Search engines need denormalized documents, not normalized tables. A blog post with author name, category, tags, and content is one document in Elasticsearch, not five tables in PostgreSQL.

**Store search data in**: Elasticsearch, Meilisearch, Algolia. Use the normalized RDBMS as the source of truth and index denormalized documents into the search engine.

### Caching Layers

A Redis cache holding session data, API responses, or computed aggregations. Caches are key-value stores. Normalization has no meaning in a cache.

**Store cache data in**: Redis, Memcached. Use the simplest key structure that serves the read pattern. Do not apply normalization principles to cache schemas.

### Machine Learning Feature Stores

ML feature stores store pre-computed features for model training and inference. Features are typically wide denormalized rows: one row per entity with all features as columns.

**Store feature data in**: Feast, Tecton, or a denormalized Redis/Aerospike cluster. Normalization adds join latency to model inference, which is unacceptable for real-time ML.

### Graph Data

Social graphs, recommendation networks, and knowledge graphs are naturally represented as nodes and edges. A normalized relational schema requires separate tables for nodes, edges of each type, and properties — then recursive CTEs to traverse the graph.

**Store graph data in**: Neo4j, Dgraph, or a dedicated graph database. Graph traversals that would require 10+ JOINs in a normalized relational schema execute in milliseconds in a graph database.

### Document Storage

Content management systems, product catalogs with variable attributes, and user profiles with sparse fields benefit from document-oriented storage. The schema is flexible — each document can have different fields.

**Store document data in**: MongoDB, Couchbase, DynamoDB. A normalized schema would require dozens of tables to capture all possible attribute combinations.

## The Decision Framework

When faced with a normalization decision, run through this checklist:

1. **Identify the workload**: Is this OLTP (many small transactions) or OLAP (large scans, aggregations)?
2. **Measure the write pattern**: How many writes per second? Are they INSERTs (new data) or UPDATEs (changing existing data)?
3. **Measure the read pattern**: How many tables per query? What is the acceptable latency?
4. **Assess consistency requirements**: Can the application tolerate stale data? Is there financial or legal liability?
5. **Evaluate team expertise**: Can the team maintain a complex normalized schema? Do they understand join optimization?
6. **Plan for scale**: Will the schema need to change as the data grows? How hard is it to add new tables?

The output is a decision: normalize to 3NF, stop at 2NF, or denormalize intentionally. Each choice comes with trade-offs that you should document and revisit as the system evolves.
