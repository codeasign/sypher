## The Problem Story

A startup called QuickCart launches an e-commerce platform. The founding engineer, racing to ship, creates one table:

```sql
CREATE TABLE orders (
    order_id INT,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100),
    customer_address TEXT,
    product_name VARCHAR(200),
    product_category VARCHAR(100),
    product_price DECIMAL(10,2),
    quantity INT,
    order_date DATE,
    shipping_address TEXT,
    payment_method VARCHAR(50),
    coupon_code VARCHAR(20)
);
```

QuickCart gets its first 10 customers. The table works. The team celebrates.

At 100 customers, problems surface. Customer Alice orders three times. Her name, email, and address are stored in three separate rows. When Alice moves, the support team must find and update every row containing her address. They miss one. Alice's third order ships to her old apartment.

At 1,000 customers, the table has 15,000 rows. The same product names appear hundreds of times. A product category change — "Electronics" renamed to "Gadgets" — requires updating 8,000 rows. The UPDATE runs for 47 seconds, locking the table. Orders fail during the window. The CEO gets angry calls.

At 10,000 customers, the anomalies compound. Inserting a new product requires creating a fake order because there is no separate products table. Deleting a canceled order also deletes the only record of a product's existence. The database is a lie.

This is the normalization problem: **data redundancy creates inconsistency, and inconsistency destroys trust.**

### Three Specific Anomaly Types

The QuickCart story demonstrates three classes of anomalies. Each one corresponds to a specific violation of normalization.

**Update anomaly**: One fact exists in multiple places. Updating it in one place but not others creates inconsistency. QuickCart's customer address change required finding every row — a task that is impossible to do correctly at scale. The database had no way to enforce that all copies of the address were updated together.

**Insert anomaly**: You cannot record a fact because another fact is missing. QuickCart could not add a new product to the catalog without creating a fake order. The schema forced a dependency between products and orders that does not exist in the real world.

**Delete anomaly**: Deleting one fact accidentally deletes another. Canceling QuickCart's last order for a product also deleted the product's name, category, and price — information that should have survived independently.

These three anomalies are the symptoms of a schema that has not been normalized. They are not bugs in the database software. They are structural problems in how the data is organized.

### The Domain Analogy: Filing Cabinets

Imagine an office with one filing cabinet. Every document — customer records, invoices, product catalogs, employee timesheets — goes into the same cabinet in any available folder. To find a customer's invoice, you search every folder. To update a customer's address, you check every folder. To add a new product, you must file it under an existing customer's folder because there is no product folder.

This is the unnormalized office. It works for 10 customers. It collapses at 100.

A normalized office has separate cabinets: one for customers, one for invoices, one for products, one for employees. Each cabinet has its own indexing system. Documents reference documents in other cabinets by an ID number — the invoice references the customer ID, but does not copy the customer's entire address. Updating the customer's address means opening one folder in one cabinet.

## The Insight

Normalization is the art of organizing data into separate tables so each fact lives in exactly one place.

Think of a library. Without normalization, every visitor who borrows a book copies the entire text into their personal notebook. When the author releases a corrected edition, each visitor must manually update their copy. Most never do. The library holds 47 different versions of the same book, and nobody knows which is current.

With normalization, the library stores one master copy of each book. Visitors borrow it, read it, return it. The catalog card references the single copy by its shelf location. When the author releases an update, the library updates one book. Every visitor who borrows it afterward gets the correct version.

Databases work the same way. A normalized schema stores each piece of information — customer, product, order — once. Everything else references it by a key. Updates are atomic. Consistency is the default.

## Core Vocabulary

Master these terms before proceeding to the deep dive.

**Normalization** — The process of structuring a relational database to reduce data redundancy and improve data integrity. Organized into normal forms, each with stricter rules.

**First Normal Form (1NF)** — Every column contains atomic values. No repeating groups or arrays. Each row has a unique identifier (primary key).

**Second Normal Form (2NF)** — Meets 1NF. Every non-key column is fully functionally dependent on the entire primary key. No partial dependencies.

**Third Normal Form (3NF)** — Meets 2NF. Every non-key column is non-transitively dependent on the primary key. No column depends on another non-key column.

**Boyce-Codd Normal Form (BCNF)** — A stricter version of 3NF. Every determinant in the table must be a candidate key. Handles edge cases that 3NF misses.

**Primary Key** — A column or set of columns that uniquely identifies each row in a table. Enforces entity integrity.

**Foreign Key** — A column that references the primary key of another table. Enforces referential integrity.

**Functional Dependency** — A relationship where knowing the value of one column (or set of columns) uniquely determines the value of another column. Written as X → Y (X determines Y).

**Candidate Key** — A minimal set of columns that can uniquely identify a row. One candidate key is chosen as the primary key; the rest are alternate keys.

**Composite Key** — A primary key consisting of two or more columns.

**Transitive Dependency** — When a non-key column determines another non-key column. X → Y and Y → Z means Z is transitively dependent on X.

**Partial Dependency** — When a non-key column depends on only part of a composite primary key, not the whole thing.

**Insert Anomaly** — Cannot insert a fact because another required fact is missing. For example, cannot add a new product without creating an order.

**Update Anomaly** — Updating a fact in one row leaves inconsistent copies in other rows. For example, updating a customer's address in only 3 of 5 order rows.

**Delete Anomaly** — Deleting a row also removes facts you wanted to keep. For example, deleting the last order for a product also deletes the product record.

**Atomicity (of columns)** — Each column holds a single value, not a list or set. "John, Jane" in a single column violates atomicity.

**Denormalization** — The intentional reintroduction of redundancy for performance reasons. A trade-off, not a mistake.

## Step by Step: Normalizing an E-Commerce Schema

Follow these steps to normalize a single example from a denormalized mess to 3NF.

### Step 1: Start with a Single Giant Table

```sql
CREATE TABLE sales (
    invoice_id INT,
    invoice_date DATE,
    customer_name VARCHAR(100),
    customer_phone VARCHAR(20),
    customer_email VARCHAR(100),
    customer_city VARCHAR(50),
    customer_state VARCHAR(50),
    product_name VARCHAR(200),
    product_category VARCHAR(100),
    unit_price DECIMAL(10,2),
    quantity INT,
    discount_pct DECIMAL(5,2),
    total_amount DECIMAL(10,2)
);
```

Problems: customer details repeat across every invoice. Product details repeat across every invoice. A customer moving to a new city requires updating every row.

### Step 2: Identify the Primary Key

The natural key is `(invoice_id, product_name)` — each invoice can list multiple products. This is a composite key.

### Step 3: Apply 1NF — Eliminate Repeating Groups

The table is already in 1NF because it has no arrays or repeating groups. Every column holds atomic values. If `product_name` held comma-separated values or if there were `product_1`, `product_2` columns, those would violate 1NF.

**Before**: Columns like `product_1, qty_1, product_2, qty_2` (repeating groups)
**After**: Each product appears in its own row. The `sales` table already does this.

### Step 4: Find Partial Dependencies

With composite key `(invoice_id, product_name)`, identify what depends on the full key versus what depends on only part of it.

- `total_amount` depends on `(invoice_id, product_name)` — the full key. You need both to compute it.
- `invoice_date` depends only on `invoice_id` — a partial dependency.
- `customer_name` depends only on `invoice_id` — a partial dependency.
- `customer_city` depends only on `invoice_id` — a partial dependency.
- `product_category` depends only on `product_name` — a partial dependency.
- `unit_price` depends only on `product_name` — a partial dependency.

### Step 5: Apply 2NF — Remove Partial Dependencies

Split the table into three tables to eliminate partial dependencies.

**Before**: One table with partial dependencies on both parts of the composite key.

**After**:

```sql
CREATE TABLE invoices (
    invoice_id INT PRIMARY KEY,
    invoice_date DATE,
    customer_name VARCHAR(100),
    customer_phone VARCHAR(20),
    customer_email VARCHAR(100),
    customer_city VARCHAR(50),
    customer_state VARCHAR(50)
);

CREATE TABLE products (
    product_name VARCHAR(200) PRIMARY KEY,
    product_category VARCHAR(100),
    unit_price DECIMAL(10,2)
);

CREATE TABLE invoice_items (
    invoice_id INT,
    product_name VARCHAR(200),
    quantity INT,
    discount_pct DECIMAL(5,2),
    total_amount DECIMAL(10,2),
    PRIMARY KEY (invoice_id, product_name),
    FOREIGN KEY (invoice_id) REFERENCES invoices(invoice_id),
    FOREIGN KEY (product_name) REFERENCES products(product_name)
);
```

### Step 6: Find Transitive Dependencies

In the `invoices` table, examine dependencies:

- `invoice_id → customer_name` (functional)
- `customer_name → customer_phone` (customer name determines phone)
- `customer_name → customer_email` (customer name determines email)
- `customer_name → customer_city` (customer name determines city)
- `customer_name → customer_state` (customer name determines state)

The phone, email, city, and state are transitively dependent on `invoice_id` through `customer_name`. If a customer changes their phone number, you must update every invoice row for that customer.

### Step 7: Apply 3NF — Remove Transitive Dependencies

Split the `invoices` table to eliminate transitive dependencies.

**Before**: One table mixing invoice facts with customer facts.

**After**:

```sql
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    customer_phone VARCHAR(20),
    customer_email VARCHAR(100),
    customer_city VARCHAR(50),
    customer_state VARCHAR(50)
);

CREATE TABLE invoices (
    invoice_id INT PRIMARY KEY,
    invoice_date DATE,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

CREATE TABLE products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(200),
    product_category VARCHAR(100),
    unit_price DECIMAL(10,2)
);

CREATE TABLE invoice_items (
    invoice_item_id INT PRIMARY KEY,
    invoice_id INT,
    product_id INT,
    quantity INT,
    discount_pct DECIMAL(5,2),
    total_amount DECIMAL(10,2),
    FOREIGN KEY (invoice_id) REFERENCES invoices(invoice_id),
    FOREIGN KEY (product_id) REFERENCES products(product_id)
);
```

### Step 8: Verify the Result

Check that the schema is now in 3NF:

- **1NF satisfied**: All columns are atomic. No repeating groups.
- **2NF satisfied**: The composite key problem is gone. Each table has a single-column primary key or a composite key where all non-key columns depend on the full key.
- **3NF satisfied**: No transitive dependencies. Customer facts live in the `customers` table. Product facts live in the `products` table. Invoice facts live in the `invoices` table.

### Step 9: Check Anomaly Elimination

**Insert anomaly**: Now you can add a new product without creating an invoice. You can register a customer before they place an order.

**Update anomaly**: Changing a customer's phone number requires updating exactly one row in the `customers` table. Every invoice for that customer automatically reflects the change through the foreign key reference.

**Delete anomaly**: Deleting an invoice no longer deletes the customer or product record. Customer and product data persists independently.

### Step 10: Measure the Storage Impact

The original single table with 10,000 invoices and 3 items per invoice (30,000 rows) would store redundant customer and product data in every row. Assuming 200 bytes of redundant customer data per row and 100 bytes of redundant product data per row:

- Original table: 30,000 rows x 600 bytes = 18 MB
- Normalized tables: 10,000 invoices x 100 bytes = 1 MB, plus 30,000 invoice items x 60 bytes = 1.8 MB, plus 3,000 customers x 300 bytes = 0.9 MB, plus 500 products x 150 bytes = 0.075 MB
- Total normalized: approximately 3.8 MB
- Storage savings: 14.2 MB (79 percent reduction)

The savings compound as the data grows. At 1 million invoices, the original table exceeds 1.8 GB while the normalized schema stays under 400 MB.

### Step 11: Identify Further Optimization Opportunities

The 3NF schema is correct, but it may not be optimal for all query patterns. Consider these additional analyses:

**Query pattern analysis**: If 80 percent of queries fetch a customer's recent orders with product names, the 3NF schema requires three joins. You might add a denormalized `product_name` column to `invoice_items` to eliminate the join with `products` for this hot path.

**Index analysis**: Every foreign key column needs an index. The `invoice_items` table has two foreign keys (`invoice_id`, `product_id`), and both should be indexed. Without these indexes, every join scans the full table.

**Materialized view analysis**: For a monthly revenue report that joins all four tables and aggregates by date and category, create a materialized view that pre-computes the aggregation. Refresh it nightly.

### Step 12: Document the Schema

A normalized schema with multiple tables requires documentation. Each table should have a clear description of its purpose, its relationships, and its key columns. For the e-commerce schema:

```
tables/
  customers        One row per customer. Source of truth for customer identity.
  invoices         One row per invoice. References customers.
  products         One row per product. References categories.
  invoice_items    One row per line item on an invoice. References invoices and products.

relationships/
  customers 1──N invoices       A customer can have many invoices.
  invoices  1──N invoice_items  An invoice can have many line items.
  products  1──N invoice_items  A product can appear on many line items.
```

This documentation helps new team members understand the schema without reading every column definition.

## Identifying Functional Dependencies in Practice

Functional dependencies are the foundation of normalization. Here is how to identify them in a real schema.

### Given Column Values

If column A's value uniquely determines column B's value, then A → B is a functional dependency. Test this with SQL:

```sql
-- Test if customer_email → customer_name is a functional dependency
SELECT customer_email, COUNT(DISTINCT customer_name) AS names
FROM orders
GROUP BY customer_email
HAVING COUNT(DISTINCT customer_name) > 1;
```

If this query returns any rows, the dependency is violated. Multiple names exist for the same email, meaning email does not determine name.

```sql
-- Test if customer_id → customer_email is a functional dependency
SELECT customer_id, COUNT(DISTINCT customer_email) AS emails
FROM customers
GROUP BY customer_id
HAVING COUNT(DISTINCT customer_email) > 1;
```

If this query returns zero rows (or the `customer_id` is the primary key), the dependency holds.

### From Business Rules

Functional dependencies often come from business rules, not data analysis:

- Each employee has exactly one department → `employee_id → department_id`
- Each department has exactly one manager → `department_id → manager_id`
- Each product has exactly one category → `product_id → category_id`
- Each invoice belongs to exactly one customer → `invoice_id → customer_id`
- A student can have many enrollments, but each enrollment is for one course → `(student_id, course_id) → grade`

Document these rules. They are the specification that determines the correct normal form.

### Implied Dependencies

Functional dependencies can chain. If `A → B` and `B → C`, then `A → C` is also a functional dependency (by transitivity). This is how you discover transitive dependencies:

- `employee_id → department_id` (given)
- `department_id → manager_email` (given)
- Therefore: `employee_id → manager_email` (implied transitive dependency)

The transitive dependency is redundant once you have the chain. Eliminating it is the goal of 3NF.

## Real-World Red Flags

These signs indicate that your schema may need normalization.

**Sign 1**: A table has columns named `field1`, `field2`, `field3`, or `extra_data`, `misc`, `notes`. These are often symptoms of an unnormalized schema where developers add columns for data that belongs in separate tables.

**Sign 2**: Column names contain numbers: `phone_1`, `phone_2`, `phone_3`. This is a repeating group in disguise. Create a separate `customer_phones` table instead.

**Sign 3**: A large percentage of rows in a table have NULL values for most columns. This indicates that different row subtypes are mixed together. Split into separate tables per subtype.

**Sign 4**: A single UPDATE statement touches more than 100 rows. This is likely an update anomaly caused by redundant data. Normalize the schema so that the UPDATE targets one row.

**Sign 5**: Application code loops over query results to transform data before using it. If the loop extracts related data from denormalized rows, the normalization is incomplete.

### Common Objections to Normalization and the Responses

**Objection 1**: "Joins are slow. I will avoid them by storing everything in one table."

**Response**: Joins are slow only when indexes are missing. With proper indexing on foreign key columns, a 3-table join takes under 10 ms for tables with millions of rows. The cost of one join is one index lookup — approximately 0.5 ms. Three joins add 1.5 ms to a query. The application will not notice.

**Objection 2**: "My data is JSON. Normalization does not apply."

**Response**: JSON data that is never queried by individual fields can stay as JSON. But if you query JSON fields in WHERE clauses, you are paying the query performance cost without getting the consistency benefit. Extract queryable fields into normalized columns and keep the JSON blob for the flexible parts.

**Objection 3**: "We use an ORM. It handles the joins automatically."

**Response**: ORMs generate SQL, but they do not choose join methods, create indexes, or optimize query plans. An ORM hides the complexity of a normalized schema but does not eliminate the performance considerations. Always profile the SQL that your ORM generates.

**Objection 4**: "Our data is read-only. Normalization does not matter."

**Response**: Read-only data does not need normalization for consistency — there are no updates to cause anomalies. However, normalized read-only data saves storage. If you have 100 GB of read-only data, normalization can reduce it to 30 GB, saving storage costs and improving scan performance.

## Variants

Normal forms build on each other. Each normal form eliminates a specific class of redundancy.

**First Normal Form (1NF)** requires that every column contains atomic values and every row has a unique identifier. No arrays, no lists, no repeating groups. Example violation: a `phone_numbers` column containing "555-0100, 555-0101, 555-0102". Fix: create a separate `customer_phones` table with one row per phone number.

**Second Normal Form (2NF)** requires 1NF plus the elimination of partial dependencies. A partial dependency occurs when a non-key column depends on only part of a composite primary key. Example violation: in a table with primary key `(student_id, course_id)`, the column `student_name` depends only on `student_id`. Fix: split into `students`, `courses`, and `enrollments` tables.

**Third Normal Form (3NF)** requires 2NF plus the elimination of transitive dependencies. A transitive dependency occurs when a non-key column depends on another non-key column. Example violation: `employee_id → department_id → department_building` means `department_building` is transitively dependent on `employee_id`. Fix: create a separate `departments` table.

**Boyce-Codd Normal Form (BCNF)** is a stricter version of 3NF. It requires that every determinant in the table is a candidate key. A determinant is any column or set of columns that functionally determines another column. BCNF catches cases where 3NF allows redundancy because of overlapping candidate keys. Example: a table of `(student, subject, teacher)` where each teacher teaches only one subject, but a subject may have multiple teachers. The determinant `teacher → subject` is not a candidate key, violating BCNF. Fix: split into `(student, teacher)` and `(teacher, subject)`.

**Fourth Normal Form (4NF)** requires that the table has no multi-valued dependencies. A multi-valued dependency occurs when one attribute determines a set of independent values. Example: a table of `(employee, skill, language)` where skills and languages are independent of each other. An employee with 3 skills and 2 languages requires 6 rows. Fix: split into `(employee, skill)` and `(employee, language)`.

**Fifth Normal Form (5NF)** requires that the table cannot be decomposed into smaller tables without losing information. Also called project-join normal form (PJ/NF). Very rarely encountered in practice — most schemas stop at 3NF or BCNF.

### Choosing the Right Normal Form

Most production databases target 3NF. Here is a practical guide:

| Workload Type | Target Normal Form | Reasoning |
|---|---|---|
| Financial transactions | 3NF or BCNF | Consistency is non-negotiable |
| E-commerce OLTP | 3NF | Balance of read/write performance |
| Content management | 2NF | Many reads, fewer writes, simpler schema |
| Analytics / reporting | 1NF or star schema | Denormalized fact tables for query speed |
| Mobile app local storage | 2NF | Limited storage, simpler queries |
| IoT sensor data | 1NF or no normalization | Append-only, no updates, no joins |

The decision is not about what is "correct" in an abstract sense. It is about what is appropriate for the workload.

### A Historical Note on Normal Forms

Normalization theory was developed by Edgar F. Codd in 1970 while he was working at IBM. Codd defined 1NF, 2NF, and 3NF in his original paper. Raymond F. Boyce and Codd developed BCNF in 1974. Ronald Fagin introduced 4NF in 1977 and 5NF in 1979.

The progression reflects a deepening understanding of redundancy. Each normal form catches a class of redundancy that the previous forms missed. In practice, most schemas do not reach 4NF or 5NF because the redundancy they eliminate is rare in real-world data.

## Mental Model Check

Test your understanding with these five questions.

**Question 1**: A table has columns `(order_id, product_id, product_name, quantity)`. The primary key is `(order_id, product_id)`. Is this table in 2NF?

**Answer**: No. `product_name` depends only on `product_id`, which is a partial dependency on the composite key. This violates 2NF. Split into `products(product_id, product_name)` and `order_items(order_id, product_id, quantity)`.

**Question 2**: A table has columns `(student_id, advisor_id, advisor_office)`. The primary key is `student_id`. Each student has one advisor. Each advisor has one office. Is this table in 3NF?

**Answer**: No. `advisor_id → advisor_office` is a transitive dependency. `student_id → advisor_id` and `advisor_id → advisor_office`, so `advisor_office` is transitively dependent on `student_id`. Split into `students(student_id, advisor_id)` and `advisors(advisor_id, advisor_office)`.

**Question 3**: You have a table of `(employee_id, department_id, department_name)`. The primary key is `employee_id`. Each department has one name. Is this in 3NF?

**Answer**: No. `department_id → department_name` is a transitive dependency. `employee_id → department_id` and `department_id → department_name`, so `department_name` is transitively dependent on `employee_id`. Create a separate `departments(department_id, department_name)` table.

**Question 4**: What is the difference between a partial dependency and a transitive dependency?

**Answer**: A partial dependency occurs when a non-key column depends on only part of a composite primary key. It requires a composite key to exist. A transitive dependency occurs when a non-key column depends on another non-key column. It does not require a composite key. Both violate 3NF, but they are eliminated at different stages: partial dependencies at 2NF, transitive dependencies at 3NF.

**Question 5**: A table has columns `(student_id, course_id, instructor_name, instructor_office, grade)`. The primary key is `(student_id, course_id)`. `instructor_name` depends on `course_id`. `instructor_office` depends on `instructor_name`. `grade` depends on the full primary key `(student_id, course_id)`. What normal form is this table in?

**Answer**: This table is in 1NF but not 2NF. The partial dependency `course_id → instructor_name` violates 2NF. Fixing that also eliminates the transitive dependency `course_id → instructor_name → instructor_office`. Fix: create `courses(course_id, instructor_name, instructor_office)`, `enrollments(student_id, course_id, grade)`, and `instructors(instructor_name, instructor_office)`.

**Question 6**: Your table has a composite primary key `(order_id, product_id)` and columns `order_date, customer_id, customer_name, product_name, quantity, discount`. Identify which columns violate 2NF and which violate 3NF.

**Answer**: 2NF violations: `order_date` and `customer_id` depend only on `order_id`. `product_name` depends only on `product_id`. 3NF violation: `customer_name` depends on `customer_id`, which is a transitive dependency from `order_id`. Eliminate 2NF first by splitting into `orders(order_id, order_date, customer_id)`, `products(product_id, product_name)`, and `order_items(order_id, product_id, quantity, discount)`. Then eliminate the 3NF violation by moving `customer_name` to a `customers` table.

**Question 7**: A table stores `(book_id, book_title, author_id, author_name, author_birth_year, publisher_id, publisher_name, publisher_city)`. The primary key is `book_id`. What normal form violations exist?

**Answer**: This table is in 2NF (no composite key, so no partial dependencies) but not in 3NF. Two transitive dependencies exist: `author_id → author_name` and `author_id → author_birth_year`, and `publisher_id → publisher_name` and `publisher_id → publisher_city`. The author and publisher facts depend on `author_id` and `publisher_id`, not on `book_id`. Fix: create `authors(author_id, author_name, author_birth_year)`, `publishers(publisher_id, publisher_name, publisher_city)`, and `books(book_id, book_title, author_id, publisher_id)`.

**Question 8**: You have a choice between a 3NF schema with 5 tables and a denormalized schema with 2 tables. The query pattern is 95 percent reads, 5 percent writes. Which do you choose?

**Answer**: The denormalized schema is the better choice for a read-heavy workload. The 95:5 read-to-write ratio means the update anomalies of denormalization affect only 5 percent of operations. The normalized schema would add join overhead to 95 percent of queries. However, consider the update pattern carefully: if the 5 percent of writes update data that is widely replicated across denormalized rows, the anomaly risk may be unacceptable. If the writes are to append-only data (new orders, new events), denormalization is safe.

## Why Normalization Matters for System Design

In a system design interview or a production system, normalization decisions ripple through every other design choice.

**Caching**: A normalized schema stores data in multiple tables. You can cache individual table results (a customer record, a product record) separately. A denormalized schema stores everything in one blob — you cache the entire blob or nothing. Normalization gives you finer-grained cache control.

**Replication**: A normalized schema with separate tables allows table-level replication. You can replicate the `products` table to all regions (it is small and changes rarely) while keeping the `orders` table in a single region (it is large and latency-sensitive). Denormalized schemas force you to replicate everything or nothing.

**Sharding**: A normalized schema allows per-table sharding strategies. The `customers` table can be sharded by `customer_id`. The `products` table may not need sharding at all — 50K products fit in a single node. Denormalized schemas force a single sharding strategy for all data.

**Migration risk**: A normalized schema can be migrated one table at a time. Add a column to `customers` without touching `orders`. A denormalized schema migration touches every row of a giant table, requiring more downtime and carrying more risk.

**Team ownership**: A normalized schema allows different teams to own different tables. The payments team owns the `payments` table. The catalog team owns the `products` table. Each team can evolve their schema independently, as long as the foreign key relationships are preserved.

These system design implications are why normalization is not just a database concept. It is an architecture decision.

Normalization is a mindset more than a checklist. The mindset asks: "Where does each fact live? Is it stored in exactly one place? If I update it, will all references reflect the change?"

The three normal forms are tools for answering these questions. 1NF eliminates repeating groups and ensures atomic values. 2NF eliminates partial dependencies — every non-key column must depend on the whole key. 3NF eliminates transitive dependencies — every non-key column must depend on nothing but the key.

When you hold this mindset, you design schemas that are consistent by construction. The database prevents anomalies. You do not need application code to maintain consistency — the schema itself enforces it.
