## Naive Implementation

The naive approach to database design is to skip normalization entirely. One table holds everything. This works for a prototype but fails at every scaling threshold.

**At 100 users**: The first data inconsistency appears. A customer's email is updated in one order row but not the other nine. Support spends 20 minutes tracking down which version is correct. The team decides to "be more careful" next time — a policy that never works.

**At 100K users**: The single table has 5 million rows. The `customer_name` column is 100 bytes, `customer_address` is 200 bytes, `product_name` is 100 bytes. Every row carries 400 bytes of redundant data. The table is 4 GB, but 3.2 GB of that is duplicated. A query to find all products purchased by a customer in the last month scans 5 million rows. Response time: 47 seconds. The page times out.

**At 10M users**: The table has 500 million rows. Disk space: 400 GB. An UPDATE to change a product category touches 20 million rows. The update takes 14 minutes and locks the entire table. All writes fail during the window. The site goes down for maintenance. The CTO declares a data architecture rewrite.

## Production-Grade Architecture

A production-grade normalized database architecture combines multiple patterns to deliver both consistency and performance.

### Core Schema: 3NF with Strategic Denormalization

```sql
-- Customers
CREATE TABLE customers (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name VARCHAR(200) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Products
CREATE TABLE products (
    id BIGSERIAL PRIMARY KEY,
    sku VARCHAR(50) UNIQUE NOT NULL,
    name VARCHAR(200) NOT NULL,
    category_id INT REFERENCES product_categories(id),
    base_price DECIMAL(10,2) NOT NULL,
    current_price DECIMAL(10,2) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_sku ON products(sku);

-- Orders
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,
    customer_id BIGINT NOT NULL REFERENCES customers(id),
    status VARCHAR(20) NOT NULL DEFAULT 'pending',
    shipping_address_id BIGINT REFERENCES addresses(id),
    billing_address_id BIGINT REFERENCES addresses(id),
    subtotal DECIMAL(12,2) NOT NULL,
    tax DECIMAL(12,2) NOT NULL,
    total DECIMAL(12,2) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_orders_customer ON orders(customer_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_created ON orders(created_at);

-- Order Items
CREATE TABLE order_items (
    id BIGSERIAL PRIMARY KEY,
    order_id BIGINT NOT NULL REFERENCES orders(id),
    product_id BIGINT NOT NULL REFERENCES products(id),
    quantity INT NOT NULL CHECK (quantity > 0),
    unit_price DECIMAL(10,2) NOT NULL,
    total_price DECIMAL(12,2) NOT NULL
);

CREATE INDEX idx_order_items_order ON order_items(order_id);
CREATE INDEX idx_order_items_product ON order_items(product_id);
```

### Indexing Strategy

Every foreign key column gets a B-tree index. Composite indexes cover the most common query patterns:

```sql
-- Composite indexes for common access patterns
-- Orders by customer, sorted by date (order history page)
CREATE INDEX idx_orders_customer_date
    ON orders(customer_id, created_at DESC);

-- Order items with product details (order detail page)
CREATE INDEX idx_order_items_order_product
    ON order_items(order_id, product_id)
    INCLUDE (quantity, unit_price, total_price);

-- Partial index for active orders only
CREATE INDEX idx_orders_active
    ON orders(customer_id, created_at DESC)
    WHERE status IN ('pending', 'processing', 'shipped');
```

### Materialized Views for Common Joins

Materialized views pre-compute the joins that are requested most frequently. They refresh on a schedule or on demand.

```sql
-- Daily sales summary: joins 4 tables
CREATE MATERIALIZED VIEW daily_sales AS
SELECT
    DATE(o.created_at) AS sale_date,
    p.category_id,
    COUNT(DISTINCT o.id) AS order_count,
    COUNT(DISTINCT o.customer_id) AS customer_count,
    SUM(oi.quantity) AS units_sold,
    SUM(oi.total_price) AS revenue
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE o.status = 'delivered'
GROUP BY DATE(o.created_at), p.category_id;

CREATE UNIQUE INDEX idx_daily_sales_date_cat
    ON daily_sales(sale_date, category_id);
```

Refresh strategy: refresh the materialized view during off-peak hours (every 6 hours), or use a logical replication slot to incrementally update it.

### Query Caching Layer

A Redis cache sits between the application and the database. The cache stores the results of frequent normalized queries.

```python
# Cache-aside pattern for normalized queries
import redis
import json

cache = redis.Redis(host='cache-cluster', port=6379, decode_responses=True)

def get_customer_order_history(customer_id, limit=10):
    cache_key = f"customer_orders:{customer_id}:{limit}"

    # Try cache first
    cached = cache.get(cache_key)
    if cached:
        return json.loads(cached)

    # Query normalized schema
    result = db.execute("""
        SELECT o.id, o.created_at, o.status, o.total,
               json_agg(json_build_object(
                   'product', p.name,
                   'quantity', oi.quantity,
                   'price', oi.unit_price
               )) AS items
        FROM orders o
        JOIN order_items oi ON o.id = oi.order_id
        JOIN products p ON oi.product_id = p.id
        WHERE o.customer_id = %s
        GROUP BY o.id
        ORDER BY o.created_at DESC
        LIMIT %s
    """, (customer_id, limit))

    # Cache for 5 minutes
    cache.setex(cache_key, 300, json.dumps(result))
    return result
```

### Connection Pooling

A connection pool (PgBouncer, HikariCP) manages database connections. Without pooling, each application instance opens connections to the normalized database. With 100 application instances, each maintaining 10 connections, the database handles 1,000 connections. Each connection consumes memory for query state, plan cache, and temporary buffers.

Connection pool configuration:

```yaml
# PgBouncer configuration
[databases]
primary = host=db-primary port=5432 dbname=production

[pgbouncer]
pool_mode = transaction
default_pool_size = 50
max_client_conn = 500
max_db_connections = 100
reserve_pool_size = 10
reserve_pool_timeout = 5.0
```

## Integration Patterns

### Normalized Schema + ORM

Object-Relational Mapping (ORM) tools like SQLAlchemy, Django ORM, and Hibernate map normalized relational schemas to application objects. The ORM generates SQL queries, manages connections, and handles result mapping.

```python
# SQLAlchemy model for normalized schema
from sqlalchemy import Column, Integer, String, ForeignKey, DateTime
from sqlalchemy.orm import declarative_base, relationship

Base = declarative_base()

class Customer(Base):
    __tablename__ = 'customers'
    id = Column(Integer, primary_key=True)
    email = Column(String, unique=True, nullable=False)
    name = Column(String, nullable=False)
    orders = relationship('Order', back_populates='customer')

class Order(Base):
    __tablename__ = 'orders'
    id = Column(Integer, primary_key=True)
    customer_id = Column(Integer, ForeignKey('customers.id'))
    status = Column(String, default='pending')
    total = Column(Integer)
    customer = relationship('Customer', back_populates='orders')
    items = relationship('OrderItem', back_populates='order')

class Product(Base):
    __tablename__ = 'products'
    id = Column(Integer, primary_key=True)
    name = Column(String, nullable=False)
    price = Column(Integer)

class OrderItem(Base):
    __tablename__ = 'order_items'
    id = Column(Integer, primary_key=True)
    order_id = Column(Integer, ForeignKey('orders.id'))
    product_id = Column(Integer, ForeignKey('products.id'))
    quantity = Column(Integer)
    unit_price = Column(Integer)
    order = relationship('Order', back_populates='items')
    product = relationship('Product')
```

ORM pitfalls with normalized schemas: the N+1 query problem. Accessing `order.items` for each order in a loop generates one query per order. Use eager loading (`joinedload`, `selectinload`) to generate JOINs instead.

### Normalized + Read Replicas

Read replicas offload join-heavy queries from the primary database. The primary handles writes (INSERT, UPDATE, DELETE) and enforces referential integrity. Read replicas handle SELECT queries, including JOINs across normalized tables.

```yaml
# Application database configuration
primary:
  host: db-primary.example.com
  port: 5432
  database: production
  pool_size: 20
  # Only primary handles writes

replicas:
  - host: db-replica-1.example.com
    port: 5432
    database: production
    pool_size: 50
    # Replicas handle reads
  - host: db-replica-2.example.com
    port: 5432
    database: production
    pool_size: 50
```

Replication lag is the critical metric. A write to the primary takes 10-50 ms to reach a replica in the same region and 100-500 ms to reach a cross-region replica. The application must tolerate reading slightly stale data from replicas, or route read-after-write requests to the primary.

### Normalized + Search Index (Elasticsearch)

Normalized relational databases are not designed for full-text search or complex filtering. Elasticsearch provides these capabilities by indexing denormalized documents.

```python
# Index denormalized documents from normalized schema
def index_order_for_search(order_id):
    order = db.execute("""
        SELECT o.id, o.created_at, o.status, o.total,
               c.name AS customer_name, c.email AS customer_email,
               json_agg(json_build_object(
                   'product_name', p.name,
                   'product_category', pc.name,
                   'quantity', oi.quantity,
                   'price', oi.unit_price
               )) AS items
        FROM orders o
        JOIN customers c ON o.customer_id = c.id
        JOIN order_items oi ON o.id = oi.order_id
        JOIN products p ON oi.product_id = p.id
        JOIN product_categories pc ON p.category_id = pc.id
        WHERE o.id = %s
        GROUP BY o.id, c.name, c.email
    """, (order_id,))

    es.index(index='orders', id=order_id, body={
        'id': order['id'],
        'created_at': order['created_at'],
        'status': order['status'],
        'total': order['total'],
        'customer_name': order['customer_name'],
        'customer_email': order['customer_email'],
        'items': order['items'],
        'suggest': {
            'input': [order['customer_name'], order['customer_email']]
        }
    })
```

The normalized schema remains the source of truth. Elasticsearch is a secondary index optimized for search, not for consistent reads.

### Normalized + Cache (Redis)

Redis caches the results of expensive normalized queries. Cache invalidation is the challenge: when data in one normalized table changes, all cached queries that reference that table must be invalidated.

```python
# Cache invalidation strategy
def invalidate_order_caches(customer_id, order_id):
    # Invalidate specific caches, not all caches
    pattern = f"customer_orders:{customer_id}:*"
    for key in cache.scan_iter(match=pattern):
        cache.delete(key)
    cache.delete(f"order_detail:{order_id}")
    cache.delete("recent_orders")
```

## Multi-Region

Normalization interacts with geo-distributed databases in challenging ways.

### Referential Integrity Across Regions

Foreign key constraints require that the referenced row exists in the same database. In a multi-region deployment, a customer in us-east-1 might have an order referencing a product stored in eu-west-1. Standard foreign key constraints cannot span database instances.

Solutions:

**Application-level integrity**: Remove foreign key constraints and enforce relationships in application code. Accept that referential integrity may be eventually consistent across regions.

**Global tables**: Use a database that supports global tables (AWS DynamoDB Global Tables, CockroachDB). These replicate data across regions and maintain consistency. However, cross-region foreign key checks add 100-300 ms of latency.

**Region-local references**: Partition data by region. A customer's orders and their referenced products must reside in the same region. This limits flexibility but maintains referential integrity.

### Conflict Resolution

When two regions independently insert rows with the same primary key, the conflict must be resolved. Strategies:

**Last-writer-wins (LWW)**: The most recent write survives. Acceptable for non-critical data like product descriptions. Dangerous for financial data.

**CRDT-based merging**: Use conflict-free replicated data types for specific columns. Counters, sets, and registers can be merged automatically.

**Manual reconciliation**: Flag conflicts for human review. Appropriate for high-value data where automatic resolution is too risky.

### Read Replica Placement

Place read replicas in each region where read traffic originates. The primary database in us-east-1 replicates to replicas in us-west-2, eu-west-1, and ap-southeast-1. Application instances in each region read from their local replica.

Replication lag across regions: 200-500 ms for us-east-1 to eu-west-1, 100-300 ms for us-east-1 to us-west-2. Queries that require fresh data route to the primary, accepting the cross-region latency penalty.

### Multi-Region Query Routing for Normalized Schemas

A multi-region application must route each query to the correct database instance based on consistency requirements:

```python
def get_order_details(order_id, user_region, require_fresh=False):
    if require_fresh:
        # Route to primary for read-after-write consistency
        db = get_primary_connection()
    else:
        # Route to local replica for lower latency
        db = get_replica_connection(user_region)

    # This normalized query joins 3 tables
    return db.execute("""
        SELECT o.id, o.order_date, o.total,
               c.name AS customer_name,
               json_agg(json_build_object(
                   'product', p.name,
                   'qty', oi.quantity,
                   'price', oi.unit_price
               )) AS items
        FROM orders o
        JOIN customers c ON o.customer_id = c.id
        JOIN order_items oi ON o.id = oi.order_id
        JOIN products p ON oi.product_id = p.id
        WHERE o.id = %s
        GROUP BY o.id, c.name
    """, (order_id,))
```

The routing decision is made per-query, not per-request. An order detail page may have 5 normalized queries: 3 can tolerate stale data (read from replica), 2 require fresh data (read from primary).

### Backup and Disaster Recovery for Normalized Schemas

A normalized schema's referential integrity depends on all tables being backed up consistently. A point-in-time backup must capture all related tables at the same transactionally consistent moment:

```bash
# PostgreSQL: transactional consistent backup of a normalized schema
pg_dump --transactional --format=custom \
  --table=customers --table=orders --table=order_items \
  --table=products --table=inventory \
  --file=/backups/normalized_schema.dump \
  production_db

# Verify referential integrity after restore
psql -d restored_db -c "
SELECT COUNT(*) AS orphaned_order_items
FROM order_items oi
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.id = oi.order_id
);"
```

A backup restored without transactional consistency can have orphaned foreign key references. Always use `--transactional` (PostgreSQL) or `--single-transaction` (MySQL) for normalized schema backups.

## Capacity Planning

Designing a normalized schema for a 10M-user SaaS requires concrete math.

### Table Size Estimates

Assume: 10M users, each user places 20 orders on average, each order has 3 items, there are 50K products.

| Table | Row Count | Row Size | Total Size | Index Size |
|---|---|---|---|---|
| customers | 10,000,000 | 300 B | 3 GB | 400 MB |
| addresses | 15,000,000 | 250 B | 3.75 GB | 500 MB |
| products | 50,000 | 200 B | 10 MB | 5 MB |
| orders | 200,000,000 | 150 B | 30 GB | 4 GB |
| order_items | 600,000,000 | 80 B | 48 GB | 6 GB |
| **Total normalized** | | | **84.8 GB** | **10.9 GB** |

Compare to a denormalized single table with 600M rows (one row per order item, with customer and product data duplicated):

| Table | Row Count | Row Size | Total Size | Index Size |
|---|---|---|---|---|
| sales (denormalized) | 600,000,000 | 500 B | 300 GB | 15 GB |

The normalized schema saves 215 GB of storage — a 72 percent reduction.

### Join Cost Analysis

For a query that fetches the last 10 orders for a customer with product details:

- Lookup customer by ID: index scan, under 1 ms.
- Lookup orders by customer_id: index scan on `idx_orders_customer_date`, returns 20 rows, under 1 ms.
- Join with order_items: index scan on `idx_order_items_order`, returns 60 rows, under 2 ms.
- Join with products: index scan on primary key, 60 lookups, under 3 ms.
- Total: under 7 ms with proper indexing.

Without proper indexing, the same query would scan the full orders table (200M rows) and use nested loop joins, resulting in response times of 30 seconds or more.

### Storage Overhead of Foreign Keys

Every foreign key column adds 8 bytes (BIGINT) plus index overhead. For `order_items`:

- `order_id` FK: 8 bytes + 16 bytes index entry = 24 bytes per row
- `product_id` FK: 8 bytes + 16 bytes index entry = 24 bytes per row
- Total FK overhead: 48 bytes per row

For 600M order items, FK overhead is 28.8 GB. This is a real cost. But compare to the alternative: storing the same data denormalized (500 bytes per row instead of 80 bytes) would cost 300 GB versus 48 GB. The FK overhead is 28.8 GB. The savings from eliminating duplication is 252 GB. FKs are a net win.

## Deployment

### Schema Migrations with Zero Downtime

Adding a normalized table or modifying a schema in production requires zero-downtime migrations. The standard approach uses the expand-migrate-contract pattern.

**Phase 1: Expand**

Add the new table and populate it. The application continues to use the old schema.

```sql
-- Add the new normalized table while old table still exists
CREATE TABLE customers_new (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name VARCHAR(200) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Backfill from the old orders table
INSERT INTO customers_new (email, name)
SELECT DISTINCT customer_email, customer_name
FROM orders_old
ON CONFLICT (email) DO NOTHING;
```

**Phase 2: Migrate**

Update the application to write to both the old and new tables. Read from the new table. Run a reconciliation job to verify consistency.

**Phase 3: Contract**

Remove the old columns and tables. This is the riskiest phase. Perform it after confirming zero data loss over at least one full business cycle.

```sql
-- Drop the old denormalized columns
ALTER TABLE orders_old DROP COLUMN customer_email;
ALTER TABLE orders_old DROP COLUMN customer_name;
-- Drop the old table if no longer needed
DROP TABLE IF EXISTS orders_old;
```

### Monitoring and Observability for Normalized Schemas

A normalized schema requires different monitoring than a denormalized one. The key metrics focus on join performance and referential integrity.

**Join performance monitoring**:

```sql
-- Identify queries with the most join overhead
SELECT
    queryid,
    query,
    calls,
    total_exec_time / calls AS avg_time_ms,
    shared_blks_hit,
    shared_blks_read,
    rows
FROM pg_stat_statements
WHERE query ~ 'JOIN'
ORDER BY total_exec_time DESC
LIMIT 10;
```

**Referential integrity violations**:

```sql
-- Check for orphaned rows across all normalized tables
SELECT 'order_items' AS table_name, COUNT(*) AS orphans
FROM order_items oi
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.id = oi.order_id)
UNION ALL
SELECT 'orders', COUNT(*)
FROM orders o
WHERE NOT EXISTS (SELECT 1 FROM customers c WHERE c.id = o.customer_id)
UNION ALL
SELECT 'order_items', COUNT(*)
FROM order_items oi
WHERE NOT EXISTS (SELECT 1 FROM products p WHERE p.id = oi.product_id);
```

**Cache hit ratio for join-heavy queries**:

```sql
-- Buffer cache hit ratio across all tables
SELECT
    relname,
    heap_blks_hit, heap_blks_read,
    CASE WHEN heap_blks_hit + heap_blks_read = 0 THEN 0
         ELSE ROUND(heap_blks_hit::numeric / (heap_blks_hit + heap_blks_read), 4)
    END AS hit_ratio
FROM pg_statio_user_tables
ORDER BY relname;
```

A hit ratio below 0.99 for a frequently joined table indicates that the working set does not fit in memory. Increase `shared_buffers` or add more RAM to the database server.

### Rollback Strategy for Normalization Migrations

Every schema migration must have a rollback plan. Normalization migrations are particularly sensitive because they involve data transformation:

```yaml
# Rollback: reverse the normalization in case of issues
  - changeSet:
      id: rollback-5-restore-denormalized-columns
      author: platform-team
      changes:
        - addColumn:
            tableName: orders
            columns:
              - column: { name: customer_email, type: VARCHAR(255) }
              - column: { name: customer_name, type: VARCHAR(200) }
        - sql: >
            UPDATE orders o SET
              customer_email = c.email,
              customer_name = c.name
            FROM customers c
            WHERE o.customer_id = c.id;
```

Test the rollback in a staging environment before running the forward migration. The rollback should complete in under 30 minutes for the full production dataset.

### Data Integrity Verification After Migration

After a normalization migration, verify that no data was lost:

```sql
-- Verify row counts match between old and new schemas
SELECT 'old_schema' AS source, COUNT(*) AS row_count
FROM orders_old

UNION ALL

-- New schema: reconstruct old rows by joining normalized tables
SELECT 'new_schema', COUNT(*)
FROM orders o
JOIN customers c ON o.customer_id = c.id;

-- Compare: should match exactly

-- Verify specific values
SELECT o.id, c.email AS old_email, c2.email AS new_email
FROM orders_old o
JOIN customers c ON o.customer_email = c.email
JOIN orders o2 ON o.id = o2.id
JOIN customers c2 ON o2.customer_id = c2.id
WHERE c.email <> c2.email;
-- Should return zero rows
```
