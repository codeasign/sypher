## Question Types

Write-back cache questions appear in system design interviews in several common forms. Each type tests a different aspect of your understanding.

### Type 1: Pure Design

**"Design a write-back cache for a system that handles 1 million writes per second."**

This is the most common type. The interviewer wants to see your understanding of the write-back pattern, trade-offs, and scaling properties. Start with the basic architecture, identify the scaling bottlenecks ("the single flusher thread cannot handle 1M writes/sec"), and iterate toward a distributed solution.

### Type 2: Problem-First

**"Our like counter system is falling over under load. Write-through is too slow. What do you do?"**

This tests your ability to diagnose the problem and recommend the right solution. The interviewer wants you to explain why write-through fails, propose write-back as the solution, and discuss the trade-offs. They may push back on data loss concerns.

### Type 3: Compare and Contrast

**"Compare write-back vs write-through vs write-around. When would you use each?"**

This tests your fundamental understanding. Build a comparison matrix (latency, throughput, data loss risk, consistency). Use real-world examples: write-back for like counters, write-through for payment transactions, write-around for content delivery.

### Type 4: Failure Analysis

**"What happens if the write-back cache crashes? How do you recover?"**

This tests your understanding of the durability trade-off. Explain the WAL, WAL replay, and data loss scenarios. Discuss async vs sync WAL. Mention the dead-letter queue for failed flushes.

### Type 5: Real-World Design

**"Design the like system for a social media app with 1 billion users."**

This combines write-back caching with distributed systems concepts. You need consistent hashing, sharding, replication, and fault tolerance. Use Facebook's McSqueal architecture as inspiration.

## What the Interviewer Evaluates

### L3 (Junior / New Grad)

**Expectations**: Understand the basic concept. Write-back is faster than write-through. Write-back means writes go to cache first, then DB later. Can draw a simple diagram.

**Good answer**: "Write-back cache stores writes in memory and writes them to the database asynchronously. This makes writes faster because the app doesn't wait for the database."

**Great answer**: Same, but identifies the durability trade-off: "The downside is that if the cache crashes before the DB write, data is lost."

### L4 (Mid-Level)

**Expectations**: Understand the mechanics. Can describe the dirty buffer, flusher, flush interval, batch writes. Knows about write coalescing.

**Good answer**: "The cache has a dirty buffer where writes are stored. A background thread periodically flushes dirty entries to the database. Multiple writes to the same key are coalesced into one database write."

**Great answer**: Explains the flush trigger (time vs threshold), batch sizing trade-offs, and the data loss window. Provides numbers: "With a 500ms flush interval and 100ms batch write time, the durability window is 600ms."

### L5 (Senior)

**Expectations**: Deep understanding of trade-offs. Can design for specific requirements. Knows failure modes and recovery strategies.

**Good answer**: Covers WAL, backpressure, dirty ratio monitoring. Discusses flush ordering and consistency.

**Great answer**: Provides concrete configuration for a specific scenario. "For a like counter, I would use async WAL with 500ms flush interval and threshold of 2,000 dirty entries. For user profile updates, I would use sync WAL with 100ms flush interval. The monitoring dashboard tracks dirty ratio, flush latency, and backlog size."

### L6 (Staff)

**Expectations**: Multi-region, active-active, conflict resolution. Capacity planning from first principles.

**Good answer**: Designs a distributed write-back cache with consistent hashing, per-node WAL, flush coordinator, and regional buffers.

**Great answer**: Works through the exact math. "10M DAU with 10 writes/user during a 2-hour peak gives 14K writes/sec average, 70K writes/sec peak. Each cache node handles 35K writes/sec. With 500ms flush interval, the dirty buffer needs 17.5 MB per node. WAL grows at 3.5 MB/s per node. We need 4 database shards running at under 20% write capacity."

### L7 (Principal / Distinguished)

**Expectations**: Design the system from the database's perspective. Discusses trade-offs in the storage engine itself.

**Good answer**: Draws parallels to InnoDB buffer pool, Cassandra commit log, RocksDB WAL. Designs a custom write-back engine optimized for the workload.

**Great answer**: Proposes a novel approach. "Instead of a generic write-back cache, I would use a log-structured merge tree with a bloom filter. Writes go to the L0 memtable, which is flushed to L1 as SSTables. Reads check the bloom filter before touching the database. This gives us write-back performance with better read efficiency."

## FAANG Interview Framework: RACE

For write-back cache questions, use the RACE framework: **R**equirements, **A**rchitecture, **C**omponents, **E**valuation.

### Requirements (2-3 minutes)

Always clarify what the system needs before designing.

- **Functional**: The system must accept writes and make them visible quickly. Writes must eventually reach the database.
- **Non-functional**: Target write latency (under 5ms), write throughput (your expected QPS), data loss tolerance (how much can you lose?), read latency (1ms for cache hits).

**Example dialogs**:

- "How much data loss is acceptable?" *("A few seconds is fine for non-critical data.")*
- "What is the write throughput requirement?" *("100,000 writes per second during peak.")*
- "Is read-your-write consistency required?" *("Yes. If I write and immediately read, I should see my write.")*

### Architecture (5-7 minutes)

Draw the high-level architecture. Start simple, then add complexity.

1. **Single node**: App, cache (dirty buffer), flusher thread, database.
2. **Distributed**: Consistent hashing, multiple cache nodes, per-node WAL.
3. **Production**: Flush coordinator, circuit breaker, retry queue, monitoring.

### Components (10-15 minutes)

Dive into each component. Be opinionated about design choices.

| Component | Key Design Points |
|-----------|------------------|
| Dirty buffer | Concurrent hash map. Shard for parallelism. Track dirty flag. |
| WAL | Append-only log. Async vs sync vs group commit. Checksums. |
| Flusher | Thread pool. Batch accumulator. Coalescing logic. |
| Retry queue | Exponential backoff. Max retries. Dead letter queue. |
| Circuit breaker | Error rate threshold. Half-open probes. |
| Backpressure | Progressive: warn, throttle, reject. |

Provide configuration numbers. "I would set the flush interval to 500ms, giving a durability window of approximately 600ms. The dirty buffer threshold would be 25% of cache capacity, equivalent to 500,000 entries for a 4 GB cache."

### Evaluation (3-5 minutes)

Summarize the system's properties and trade-offs.

| Requirement | How Write-Back Meets It |
|------------|------------------------|
| Low latency | Write ACK in under 2ms (local) or 5ms (distributed) |
| High throughput | 10x-100x over write-through |
| Data loss | Acceptable window configured via flush interval |
| Consistency | Eventual. Read-your-write from cache. |
| Availability | Available during DB partition (buffer absorbs writes) |

## Phrase Bank

Use these phrases to demonstrate expertise during the interview.

**"The dirty buffer acts as a shock absorber between write throughput and database capacity."**
Shows you understand the buffering function of write-back.

**"Write coalescing turns a firehose of writes into a trickle of database operations."**
Demonstrates your understanding of the efficiency gain.

**"The durability window is the price we pay for performance."**
Shows you understand the fundamental trade-off.

**"A WAL with group commit gives us near-write-through durability with write-back performance."**
Demonstrates knowledge of WAL configurations.

**"Backpressure is not a failure; it is a controlled degradation."**
Shows operational maturity.

**"The dirty ratio is the canary in the coal mine for write-back systems."**
Shows you know what to monitor.

**"Causal dependencies must be flushed together or in order. Priority queues break causal ordering."**
Demonstrates deep understanding of consistency.

**"At 70,000 writes per second with a 5:1 coalescing ratio, the database sees only 14,000 writes per second."**
Shows you do the math.

**"Circuit breaker prevents a database outage from becoming a total system outage."**
Shows knowledge of failure patterns.

**"I would use consistent hashing with 256 virtual nodes per physical node for even key distribution."**
Shows operational precision.

**"The LSM tree is a natural evolution of write-back for persistent storage."**
Shows broader systems knowledge.

**"Read-repair during flush is an optimization that provides eventual consistency without extra round trips."**
Shows database-specific knowledge.

**"I would separate the flush path into a control plane and data plane. The control plane manages scheduling; the data plane moves bytes."**
Shows software architecture sophistication.

**"For counters, write-back with delta accumulation is strictly better than absolute value writes."**
Shows domain-specific optimization knowledge.

**"The most common write-back failure is not crash data loss; it is silent flush failure where the cache thinks data was flushed but the database never received it."**
Shows experience with real production incidents.

## Red Flags (What to Avoid)

Avoid these mistakes during the interview.

**"Write-back never loses data."** False. Write-back always has a durability window. Acknowledging this up front shows honesty and depth.

**No WAL discussion.** If you design a write-back cache without mentioning a Write-Ahead Log, the interviewer will assume you have not thought about crash recovery.

**Ignoring dirty buffer memory.** If you allocate the dirty buffer as an unbounded data structure, the interviewer will ask about OOM. Have an answer for dirty buffer capacity.

**Single flusher thread at scale.** If you design a system for 100K writes/sec with a single flusher thread, the interviewer will point out the bottleneck.

**No backpressure mechanism.** If the dirty buffer fills up and your answer is "more memory," you have missed the point. Backpressure is essential.

**Assuming in-order flush.** If you use a priority queue without discussing causal dependencies, the interviewer may probe deeper.

**Forgetting read-your-write consistency.** Write-back provides this naturally (reads hit the cache), but if you do not mention it, the interviewer may wonder if you understand the read path.

**Ignoring monitoring.** A system without monitoring is a system that will fail silently. Mention dirty ratio, flush latency, and backlog monitoring.

## Practice Drills

### Drill 1: Whiteboard (15 minutes)

Draw the complete write-back architecture from memory. Include: app, cache, dirty buffer, WAL, flusher thread pool, batch accumulator, database, retry queue, circuit breaker. Label all connections. Identify the durability window. Add monitoring metrics. Time yourself.

### Drill 2: Explain to a Non-Technical Person (5 minutes)

Explain write-back caching to someone who knows databases but not caching. Use the restaurant waiter analogy. Explain the trade-off: write-back is faster but may lose a few seconds of data on a crash.

**Starter**: "Imagine a waiter who takes orders from five tables before going to the kitchen. That is write-back. A write-through waiter runs to the kitchen after every single table. The write-back waiter serves more tables per hour, but if the waiter drops the order pad..."

### Drill 3: Capacity Calculation (10 minutes)

Given: 10M DAU, 5 writes/user/day, 4-hour peak window, each write is 500 bytes, flush interval 1 second, coalescing ratio 3:1, batch size 500.

Calculate: writes/sec at peak, dirty buffer memory, database writes/sec, WAL growth per minute.

**Starter**: "50M writes/day / 14,400 seconds = 3,472 writes/sec average. With a 5:1 peak-to-average ratio: 17,361 writes/sec peak. Dirty buffer: 17,361 * 1s = 17,361 entries at 500 bytes = 8.7 MB. Coalesced: 5,787 DB writes/sec. Batch size 500: 12 batch writes/sec."

### Drill 4: Debug the System (10 minutes)

The interviewer describes a failing write-back system. You diagnose the cause.

**Scenario**: "Dirty ratio is 70%. Write latency has increased from 1ms to 50ms. The database CPU is at 30%."

**Diagnosis**: The database is not saturated, so the bottleneck is not there. The high dirty ratio with low DB load suggests the flusher is not keeping up. Possible causes: single flusher thread is blocked (deadlock, I/O wait), flusher is configured with too large a batch size, or the flusher thread has crashed. Recommendation: check flusher thread health, reduce batch size, add more flusher threads.

**Starter**: "Since the database is at 30% CPU, it is not the bottleneck. The problem is in the flusher. I would check..."

### Drill 5: Trade-Off Debate (5 minutes)

Argue for write-back. Then argue for write-through. Practice switching perspectives.

**For write-back**: "Our users need instant feedback. 50ms write latency is unacceptable. We accept the risk of losing a few seconds of data because we have monitoring and alerts. The database cost savings from coalescing are significant."

**For write-through**: "We cannot lose any data. Regulatory requirements mandate synchronous durability. We will invest in faster storage and larger connection pools to make write-through perform acceptably."

### Drill 6: Design with Constraints (15 minutes)

Design a write-back cache with an unusual constraint.

**Constraint A**: "The database is 100ms away (cross-region)." Solution: Increase batch size to 5,000 entries. Accept 100ms flush latency. Add WAL for crash safety. The inter-region latency is now amortized over 5,000 writes.

**Constraint B**: "Data loss of more than 1 second is unacceptable." Solution: Use sync WAL. Set flush interval to 500ms. Ensure the flusher can complete in under 500ms. Monitor flush latency with p99 alerting.

**Constraint C**: "The cache nodes have only 256 MB of memory." Solution: Aggressive backpressure at 20% dirty ratio. Small batch sizes (100). Frequent flushes (every 200ms). Priority-based eviction. Monitor memory usage closely.
