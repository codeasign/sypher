## How This Topic Appears

Model routing questions appear in system design interviews at every level. Interviewers use them to assess your understanding of LLM infrastructure, decision systems, cost optimization, and operational maturity. The topic sits at the intersection of distributed systems, machine learning operations, and practical engineering judgment.

Here are the common phrasings you will encounter:

**Direct Questions:**
1. "What is model routing and when would you use it?"
2. "Explain how you would build a model router for an AI application."
3. "What factors go into choosing which model to route a request to?"

**Design Questions:**
4. "Design a model routing system for a customer support chatbot that handles 1 million conversations per day."
5. "Design a cost-optimized routing pipeline for a platform that offers 20 different LLM models."
6. "How would you build a multi-region model routing system with failover?"

**Debugging Questions:**
7. "Your model router is misclassifying 15 percent of requests. How do you debug this?"
8. "The p99 latency of your routing system just spiked from 30ms to 500ms. What do you check first?"

**Trade-off Questions:**
9. "Compare rule-based routing versus ML-based routing. When would you choose each one?"
10. "Your team wants to route every request through a classifier before sending it to the model. What are the risks?"

## What Interviewer Is Testing

Each question type tests a different dimension of your understanding:

**Direct Questions** test fundamental knowledge. The interviewer wants to confirm you understand what model routing is, why it matters, and the core concepts. Strong answers name concrete decision criteria (cost, latency, capability, safety) and give real examples. Weak answers stay abstract and theoretical.

**Design Questions** test architectural thinking. The interviewer evaluates your ability to decompose a problem into components, make reasonable assumptions about scale, and design a system that works at production traffic levels. Strong answers include numbers (QPS estimates, latency budgets, cache hit ratios) and discuss failure modes. Weak answers describe a single approach without trade-off analysis.

**Debugging Questions** test operational maturity. The interviewer wants to see your systematic approach to diagnosing problems in a running system. Strong answers follow a logical narrowing process: check metrics first, then logs, then configuration, then code. Weak answers jump to conclusions or suggest restarting everything.

**Trade-off Questions** test engineering judgment. The interviewer evaluates whether you understand that every design decision has costs. Strong answers frame the trade-off explicitly ("This buys us X accuracy at the cost of Y latency") and give decision rules for when to choose each option. Weak answers treat one approach as universally better.

## Level-by-Level Expectations

### L3 / L4 (Early Career)

At this level, the interviewer expects you to understand the basic concept and implement a simple version.

**Sample Answer for "What is model routing?":**

"Model routing is a decision layer that examines each incoming request to an AI application and sends it to the most appropriate LLM based on factors like request complexity, cost constraints, latency requirements, and the capabilities of available models. For example, simple queries like 'What is the weather in Tokyo?' would route to a cheap, fast model like GPT-4o-mini, while complex tasks like 'Analyze this contract for legal risks' would route to a more capable model like GPT-4 or Claude Opus. The router typically uses a classifier -- either rule-based or ML-based -- to determine the best model for each request. The key benefits are cost savings (you are not paying for a large model to answer simple questions), latency improvement (fast models respond faster), and reliability (you can fall back to a different model if the primary one is unavailable)."

**What distinguishes a strong L3 answer:** The candidate names specific models, gives concrete examples, and mentions fallback behavior. They do not just define model routing -- they show they have thought about why it matters.

### L5 (Senior)

At this level, the interviewer expects you to design a production-grade routing system.

**Sample Answer for "Design a model routing system":**

"I would design a model routing system with four main components. First, a request parser that extracts the input text, metadata (user tier, request type), and constraints (max latency, max cost). Second, a classifier that determines the request category -- this could be a lightweight ML model, a prompt-based classifier using a cheap LLM, or a set of rules. Third, a routing table stored in a fast key-value store like Redis that maps categories to model assignments with fallback chains. Fourth, a model gateway that makes the actual API calls, handles retries, and collects latency and cost metrics.

For a customer support chatbot handling 1M conversations per day, I would estimate roughly 3,000 requests per second at peak. The classifier needs to run in under 20ms to stay within the latency budget. I would use a small embedding model for classification -- embed each request and match against predefined category centroids. The routing table would be cached in memory with a TTL of 60 seconds, refreshed from etcd. Each category would have a primary model and two fallbacks. Metrics would flow to a time-series database for monitoring routing accuracy, fallback rate, and per-model cost.

The main trade-off is classification accuracy versus latency. A prompt-based classifier gives higher accuracy but adds 200-500ms. An embedding classifier runs in 5-10ms but may misclassify edge cases. I would default to embedding-based routing and add a prompt-based recheck for ambiguous requests."

**What distinguishes a strong L5 answer:** The candidate provides specific numbers, discusses caching, mentions fallback chains, includes observability, and frames trade-offs explicitly.

### L6 (Staff)

At this level, the interviewer expects you to design for global scale, operational excellence, and evolving requirements.

**Sample Answer for "Design multi-region model routing":**

"I would design a globally distributed model routing system with regional routing service clusters, each backed by a local model registry replica. The global topology would use a leader-follower pattern for the routing table: one primary region (us-east-1) owns the authoritative routing configuration, which replicates asynchronously to secondary regions (eu-west-1, ap-southeast-1). Each regional router caches the routing table locally and serves requests independently -- no cross-region dependency for the hot path.

Region failover works at two levels. At the model level, if the primary model in a region is unavailable, the router falls back to the secondary model in the same region. At the region level, if the entire regional routing service degrades, a global load balancer routes traffic to the nearest healthy region. The routing table replication uses a version vector for conflict detection -- if a secondary region has stale data, it still routes correctly using the cached version while the sync catches up.

The hard part is cost-aware routing across regions. Model pricing varies by region (us-east-1 GPT-4 might cost 20 percent less than eu-west-1). I would add a cost layer in the routing service that adjusts model priority based on regional pricing. This creates a trade-off: routing to a cheaper region adds cross-region latency. The solution is to route latency-sensitive requests to the local region regardless of cost, and route batch or async requests to the cheapest region."

**What distinguishes a strong L6 answer:** The candidate discusses global topology, replication strategies, version vectors, cost-latency trade-offs across regions, and operational concerns like stale caches.

### L7 / L8 (Principal / Fellow)

At this level, the interviewer expects you to discuss the strategic dimensions: when model routing is the wrong solution, how the pattern evolves as the industry changes, and what meta-lessons generalize to other systems.

**Sample Answer for "The future of model routing":**

"Model routing as we practice it today is a transitional pattern. The current landscape has dozens of models with different capabilities, pricing, and latency profiles, so routing adds clear value. But as models converge in capability and as model providers adopt more granular pricing (per-token latency tiers, reserved capacity), the routing layer needs to become more dynamic and more integrated with the application logic.

The strategic question is whether routing should be a separate infrastructure layer or embedded in the application. My view is that it will move into the model gateway -- the same way load balancing is an infrastructure concern, not an application concern. Companies like Portkey and Helicone are already building this. The routing decisions will be driven by trained ML models that optimize for a composite objective function: minimize (cost) at penalty for (latency violations + accuracy failures + availability incidents). This is a constrained optimization problem, not a rules engine.

The anti-pattern I see most often is teams building elaborate routing systems before they have traffic data. You should start with a simple route-to-all approach, collect telemetry, analyze the actual cost and latency distribution, and only then build routing logic. Otherwise you are optimizing based on assumptions, not data."

**What distinguishes a strong L7 answer:** The candidate shows strategic perspective, identifies the pattern as transitional, frames routing as an optimization problem, and warns against premature optimization.

## FAANG Framework

Use this 6-step framework to answer any model routing design question:

### Step 1: Clarify Scope (2 minutes)

Ask clarifying questions before designing:
- "What is the scale? QPS, number of models, number of users?"
- "What are the constraints? Cost budget, latency budget, model availability?"
- "Is this for chat, batch processing, or real-time API?"
- "Do we control the models or call third-party APIs?"
- "What is the fallback behavior? Fail open or fail closed?"

### Step 2: Define Requirements (2 minutes)

State the requirements explicitly:

**Functional:**
- Route each request to the best model based on predefined criteria
- Support fallback if the primary model is unavailable
- Collect metrics on routing decisions and model performance
- Allow dynamic updates to routing policies without downtime

**Non-Functional:**
- Routing decision latency: under 20ms p50, under 100ms p99
- Routing accuracy: over 95 percent correct classification
- Availability: 99.99 percent (routing service must not be a single point of failure)
- Throughput: support 10K QPS per routing instance

### Step 3: High-Level Design (5 minutes)

Sketch the architecture:

```
Client Request
    │
    v
Load Balancer ──> Routing Service Cluster
    │                   │
    │                   ├── Classifier (embedding model)
    │                   ├── Routing Table (cached from registry)
    │                   └── Cost/Latency Predictor
    │                   │
    v                   v
Model Gateway ──> [Model A] [Model B] [Model C]
    │
    v
Response + Metrics ──> Monitoring
```

### Step 4: Deep Dive (10 minutes)

Go deep on 2-3 specific areas:

**Classification Strategy:**
- Rule-based: keyword matching, regex patterns. Fast (under 1ms) but brittle.
- Embedding-based: embed query, nearest centroid. Fast (5-10ms), good accuracy.
- Prompt-based: ask cheap LLM to classify. Slow (200-500ms), highest accuracy.
- Hybrid: embedding for fast path, prompt for ambiguous cases.

**Routing Table Design:**
- Key: category ID or embedding centroid hash
- Value: ordered list of `{model_id, weight, cost_cap, timeout_ms}`
- Storage: in-memory cache with 60-second refresh from etcd
- Versioning: monotonic version number for cache invalidation

**Fallback Chain:**
- Each route defines a primary, secondary, and tertiary model
- On failure (timeout, error, rate limit), try the next in chain
- Track fallback rate as an alerting metric
- Implement circuit breaker: if a model fails 5 times in 30 seconds, skip it

### Step 5: Edge Cases and Failure Modes (5 minutes)

Discuss what breaks:
- "What if the classifier is wrong?" -- Add a confidence threshold; route low-confidence requests to the most capable model.
- "What if all models fail?" -- Return a graceful error with a retry hint.
- "What if the routing table is stale?" -- Use TTL with background refresh; serve stale data under load.
- "What if cost spikes?" -- Enforce a per-request cost cap; alert on average cost increase over 10 percent.

### Step 6: Scale and Optimize (3 minutes)

Address production concerns:
- "How do we scale?" -- Stateless routing service, horizontally scalable. Add instances based on QPS. Each instance caches the routing table locally.
- "How do we monitor?" -- Track routing latency, classification distribution, fallback rate, per-model cost, average response quality score.
- "How do we deploy?" -- Canary deployments: route 1 percent of traffic to the new routing configuration, compare metrics, then roll out globally.

## Phrase Bank

Use these phrases to sound experienced and precise:

1. **"The routing decision must happen in the hot path, so the classifier needs sub-20ms latency."** -- Shows you understand the performance constraint.

2. **"I would use a fallback chain with circuit breakers, not just a primary-secondary pair."** -- Shows operational maturity beyond basic failover.

3. **"The cost-accuracy-latency triangle is the core trade-off in every routing decision."** -- Shows you frame problems in terms of trade-offs.

4. **"I would embed the requests and use nearest-centroid matching for classification."** -- Shows you know a specific, practical technique.

5. **"Route ambiguous requests to the most capable model by default."** -- Shows you have a default handling strategy for edge cases.

6. **"The routing table is a cache of the model registry, not the source of truth."** -- Shows you understand data flow and consistency.

7. **"I would use a hybrid classifier: fast embedding-based routing for the common case, with a prompt-based recheck for low-confidence classifications."** -- Shows you know how to balance speed and accuracy.

8. **"The fallback rate should be under 5 percent in steady state. If it exceeds that, something is wrong with the primary model selection."** -- Shows you have concrete operational thresholds.

9. **"I would separate routing policy from routing execution. Policy is configuration, execution is code."** -- Shows clean architectural thinking.

10. **"Cost-optimized routing is a constrained optimization problem: minimize cost subject to latency and accuracy constraints."** -- Shows analytical framing.

11. **"The model registry must be highly available because every routing decision depends on it."** -- Shows you think about dependencies.

12. **"I would add a routing audit log to trace why each request went to a particular model."** -- Shows you care about debuggability.

13. **"Cache hit ratio for routing decisions should be over 99 percent for repeated request types."** -- Shows you set performance targets.

14. **"Model routing is only as good as your observability. If you cannot measure routing accuracy, you cannot improve it."** -- Shows you connect monitoring to system quality.

15. **"I would version the routing configuration and support A/B testing of routing policies."** -- Shows you think about evolution and experimentation.

## Red Flags

Avoid these behaviors:

1. **Not mentioning cost.** Model routing is primarily a cost optimization pattern. If you only discuss latency or capability, you miss the main reason companies build it.

2. **Proposing a single, fixed routing strategy.** A good answer discusses multiple strategies and when to use each. A routing system with one strategy is brittle.

3. **Ignoring fallback behavior.** Every real routing system needs fallback. If you forget this, the interviewer knows you have not operated a routing system.

4. **Treating classification as perfect.** All classifiers make mistakes. A good answer discusses misclassification rates, confidence thresholds, and handling of ambiguous requests.

5. **No numbers.** Estimates without QPS, latency targets, cache sizes, or cost figures sound theoretical. Ground your design in concrete numbers.

6. **Ignoring the model registry.** The routing table must come from somewhere. If you do not mention how models are registered, discovered, and health-checked, you miss a critical component.

7. **Designing without observability.** If you do not mention metrics, logging, or tracing, the interviewer knows you have not operated a production system.

8. **Over-engineering the classifier.** Using a 70B parameter model to classify requests defeats the purpose of routing. The classifier must be lightweight.

9. **No mention of caching.** The routing decision is made for every request. Without caching, the classifier or lookup must run at full QPS, which is expensive.

10. **Failing to discuss deployment.** How do you update routing policies without downtime? How do you roll back a bad configuration? If you skip this, the design is incomplete.

## Common Question Deep Dives

### Deep Dive: "Design a model routing system"

This is the most common design question. Here is a complete walkthrough of a strong answer.

**Clarifying questions to ask:**
- "How many models will we support? 3, 10, or 50?"
- "What is the traffic volume? Millions of requests per day or thousands?"
- "Are these hosted models (OpenAI, Anthropic) or self-hosted?"
- "What are the latency requirements? Real-time chat or batch processing?"
- "Do we control the model selection or does the user specify a preference?"

**Requirements (state explicitly):**
Functional: Route each request to the best model, support fallback on failure, collect cost and latency metrics, allow dynamic policy updates.
Non-functional: Routing latency under 20ms p50, routing accuracy over 95 percent, availability 99.99 percent, support 10K QPS per instance.

**Architecture (draw at high level):**
1. Load balancer in front of stateless routing service instances
2. Each instance has: request parser, classifier (embedding model), routing table (in-memory cache from etcd), model gateway client
3. Model registry (etcd/Consul) stores the authoritative routing configuration
4. Metrics pipeline (Prometheus + Grafana) collects routing decisions and model performance
5. Fallback chain with circuit breakers per model

**Deep dive areas to choose from:**
- Classification strategy: embedding-based with hybrid fallback for low confidence
- Routing table schema: key = (category, complexity_tier), value = ordered model list
- Cache design: in-memory hash map with 60-second TTL, background refresh
- Fallback strategy: primary model -> same-tier model -> higher-tier model -> graceful error
- Cost optimization: cheapest model meeting quality floor, recomputed on pricing changes

**Edge cases to discuss:**
- Model unavailable mid-request: fallback chain activates, circuit breaker opens
- Classification low confidence: route to most capable model, log for tuning
- Flapping between two models: apply hysteresis (only switch if cost difference > 5 percent)
- Stale routing table: serve from cache, log staleness, probe registry in background

**Numbers to have ready:**
- 10M DAU -> approximately 1,000 requests/second average, approximately 3,000 peak
- Each routing instance handles 5K-10K QPS
- Embedding model adds 5-10ms latency, 50-100MB memory
- Cache with 60-second TTL and 80 percent hit ratio reduces classification load 5x
- Monthly cost savings: 40-60 percent for typical customer support traffic

### Deep Dive: "Debug a routing latency spike"

**Situation:** The routing system's p99 latency suddenly increased from 20ms to 200ms. All models are healthy. No deployment changes were made.

**Systematic debugging approach:**

Step 1 -- Check classification latency:
- Is the embedding model slow? (p99 embedding time went from 8ms to 42ms)
- Is the classification service under load? (QPS is normal)
- Has the embedding model been updated? (No recent changes)
- Check embedding model host: CPU is at 95 percent utilization, insufficient capacity.

Step 2 -- Check routing table lookup:
- Is the cache hit ratio normal? (Yes, 85 percent)
- Is the registry connection healthy? (Yes, sub-1ms latency)
- Are there any lock contention issues? (No)

Step 3 -- Check model gateway:
- Are any model API calls timing out? (No, all healthy)
- Is the connection pool exhausted? (No)

Root cause: The embedding model instance is CPU-bound because a co-located service is consuming resources. The fix is to isolate the embedding model on dedicated instances or scale horizontally.

**What the interviewer is looking for:** A systematic narrowing process, knowledge of which metrics to check and in what order, and the ability to identify the root cause from symptoms.

### Deep Dive: "Compare rule-based vs. ML-based routing"

**When the interviewer asks this, they want to see that you understand the trade-offs and can make a justified recommendation.**

**Rule-based advantages:**
- Deterministic: same request always produces the same routing decision
- Explainable: any routing decision can be traced to the rule that triggered it
- Fast: under 1ms per decision
- No training data required: rules are authored by domain experts
- Easy to debug: if a request is misrouted, you see which rule matched

**Rule-based disadvantages:**
- Brittle: synonyms and novel phrasings can miss rules
- Maintenance burden: every new request pattern needs a new rule
- Cannot handle nuanced categories: "I need help with my bill" could be billing or service cancellation
- Rules accumulate: 50+ rules become hard to audit for conflicts

**ML-based advantages:**
- Flexible: handles synonyms and novel phrasings naturally
- Lower maintenance: update centroids or retrain model instead of adding rules
- Better accuracy: captures semantic similarity that rules miss
- Scales across languages: multilingual embedding models work out of the box

**ML-based disadvantages:**
- Non-deterministic: embeddings can drift as the underlying model is updated
- Harder to debug: "why was this request classified as technical support?" is harder to answer
- Slower: adds 5-10ms for embedding generation
- Requires labeled data: categories need representative samples for centroids
- Model dependency: the routing system depends on the embedding model's availability

**Recommendation framework:**
- Choose rule-based when: categories are well-defined and non-overlapping, the team needs full explainability, latency is critical (under 5ms budget), or there is no labeled training data
- Choose ML-based when: categories overlap semantically, request volume is high (over 1K QPS), the system supports multiple languages, or request patterns change frequently
- Choose hybrid when: you need both speed and accuracy, have budget for development, or have high-confidence requirements (over 99 percent effective accuracy)

## Common Follow-Up Questions

Interviewers often ask follow-up questions to probe depth. Here are the most common ones and how to handle them:

**Q: "How do you handle a new model that is cheaper and better than all existing models?"**

A: "Register the new model in the registry, update the routing table to make it the primary for appropriate categories, and run a shadow evaluation for 24 hours to validate quality. Then gradually shift traffic: 10 percent on day 1, 25 percent on day 2, 50 percent on day 3, 100 percent on day 5. Monitor for quality regression at each step."

**Q: "What if a user explicitly requests a specific model?"**

A: "User-specified model selection should override the router. The router should only make decisions when the user does not specify a preference. This is the 'user override' pattern: the routing decision is a suggestion, not a mandate. If the user's specified model is unavailable, ask the user if they want to fall back to the router's recommendation."

**Q: "How do you measure routing quality?"**

A: "Use a sampling approach: for every 100th request, send the same request to all available models and compare the responses. Use automated evaluation metrics (Rouge-L, BERTScore) and periodic human evaluation. Track the percentage of requests where the router's selected model produced the best response. This is the 'routing accuracy' metric."

**Q: "How do you handle model versioning?"**

A: "Each model in the registry has a version field. When a new version is deployed, it is registered as a new model entry (e.g., 'gpt-4o-v2'). The routing table is updated to point to the new version. Old versions are deprecated and removed from the registry after a grace period. This allows A/B testing between versions and provides a rollback path."

## Behavioral Questions for Routing Roles

Some interviews combine system design with behavioral questions. Here are common behavioral questions that relate to model routing and how to answer them.

**Q: "Tell me about a time you had to make a trade-off between cost and quality."**

Frame your answer around a routing-like decision: "At my previous company, we used GPT-4 for all customer support queries. I proposed routing simple queries to GPT-4o-mini, which would save $20K/month. The VP of Product was concerned about quality degradation. I ran a 2-week pilot with 5 percent of traffic routed to the cheap model. We measured user satisfaction scores and found no statistically significant difference for simple queries. I presented the data to the VP, and we rolled out the routing system. It saved $18K/month in the first quarter."

**Q: "Describe a time when a system you built failed in production."**

This is a classic behavioral question. Frame it around a routing failure: "Our routing system had a bug where the classifier was not handling empty requests. A partner integration sent empty string requests, which caused the classifier to return a default category with high confidence. Those requests were routed to GPT-4o-mini, which produced nonsensical responses. We detected the issue through a spike in negative user feedback. The fix was adding input validation: reject empty requests before classification. We also added a 'minimum confidence' threshold: any classification below 0.5 confidence routes to the most capable model as a safety net."

**Q: "How do you stay current with the rapidly changing LLM landscape?"**

Discuss your process for monitoring new models, pricing changes, and industry trends: "I subscribe to model provider changelogs, follow the LLM infrastructure community, and maintain a spreadsheet of model pricing and capabilities. Every month, I review whether our routing policies are still optimal given the current model landscape. I also run a weekly script that benchmarks new models against our production traffic to evaluate whether they could improve our routing decisions."

## Negotiation and Scope Management

In senior-level interviews, the interviewer may test your ability to scope a routing system appropriately.

**When the interviewer says "We have unlimited budget," push back:** "Even with unlimited budget, routing every request to the most expensive model is not the right approach. The most capable model is not always the best -- it may be slower, have different safety characteristics, or be overkill for simple requests. The goal is to match the request to the most appropriate model, not the most expensive one."

**When the interviewer says "We need this in production next week," negotiate scope:** "I can build a simple rule-based router in a week that handles 80 percent of the traffic correctly. The remaining 20 percent will go to a default model, and we can add ML-based classification in the next sprint. This gives us cost savings immediately while we build the more accurate system."

**When the interviewer says "We have 50 models to support," validate the premise:** "Do all 50 models need to be in the routing table? I would start by identifying the 5-10 models that handle 90 percent of our traffic. The remaining models can be added as special cases or managed through a separate path. Trying to build a routing system that handles 50 models from day one will take too long and may not be necessary."

## Key Takeaways for Interview Day

Arrive at your interview with these key points ready to deploy:

1. **Model routing is a cost optimization pattern first.** Always lead with the cost motivation. Mention that routing can reduce LLM costs by 40-70 percent while maintaining quality.

2. **The three classification approaches are your anchor.** Rule-based (under 1ms, 70-85 percent), embedding-based (5-20ms, 85-95 percent), prompt-based (200-800ms, 92-98 percent). Use these numbers to frame every classification discussion.

3. **Always include fallback.** A router without fallback is not production-ready. Mention fallback chains, circuit breakers, and cascading failure prevention.

4. **Specific numbers build credibility.** Have QPS estimates, latency budgets, cost savings, and cache hit ratios ready. Even rough estimates show you have thought about scale.

5. **The routing table is a cache, not the source of truth.** This shows you understand data flow and consistency. The model registry is the source of truth.

6. **Focus on the fastest path to value.** In a design interview, you do not need to design the perfect system. Design a system that works, then discuss how to improve it. Start simple, then layer on optimization.

7. **Connect to business outcomes.** Routing is not just a technical pattern. It saves money, improves user experience, and enables teams to use multiple models without complexity. Connect your design decisions to these outcomes.

8. **Practice the trade-off language.** Use phrases like "the cost-accuracy-latency triangle," "routing accuracy versus routing latency," and "constrained optimization." These signal that you think in trade-offs, not absolutes.

9. **Your calmness under pressure matters.** When the interviewer challenges your design -- "That classification will be too slow" or "Your fallback chain will cause a cost spike" -- do not get defensive. Say "That is a valid concern. Let me explain why I made that choice, and then we can discuss alternatives." This shows you can handle technical disagreements professionally.

10. **End with operations.** After describing your design, always add: "And then I would set up monitoring for routing accuracy, fallback rate, and cost per request, with alerts on any of these metrics exceeding their thresholds." This closes the loop from design to operations.

11. **Prepare a story from your experience.** Even if you have not built a model routing system, you can frame related experience. Load balancers, API gateways, message routers, or any decision-layer system follows similar patterns. Connect your experience to routing concepts: "The load balancer I built used health checks and circuit breakers, which is exactly the same pattern I would use for model fallback." This shows you can transfer knowledge across domains.

12. **Bring your own framework.** Interviewers remember candidates who use a structured approach. Use the FAANG framework (clarify, define, design, deep dive, edge cases, optimize) from this playbook. When you say "Let me use a structured approach here," the interviewer pays attention.

13. **Review your answers aloud.** Before the interview, record yourself answering the practice drills. Listen for unclear explanations, filler words, and gaps in your reasoning. The best interviewers sound confident, not because they know everything, but because they communicate clearly about what they know and what they would need to figure out.

14. **Stay calm on the hard questions.** If the interviewer throws a curveball, take a breath and say "Let me think about that for a moment." Silence is acceptable. A thoughtful pause followed by a structured answer is better than a fast, unstructured answer. Interviewers value quality over speed.
