## Tier 1: Foundation

### Challenge 1: Explain Model Routing

**Prompt:** Explain the concept of model routing to a junior engineer who knows what LLMs are but has never designed an AI system. Include the core motivation, the key components, and at least one concrete example. Your explanation must be understandable without any diagrams.

**Evaluation Criteria:**
- Clearly defines model routing as a decision layer between client and models
- Names at least three decision factors (cost, latency, capability, safety)
- Gives a concrete example with specific model names
- Explains why routing matters for production systems
- Total time: under 3 minutes when spoken

**Model Answer:**
"Model routing is a decision layer that sits between your application and the LLM models you use. Instead of sending every request to a single model like GPT-4, the router examines each request and sends it to the best model based on what the request needs.

The core motivation is cost optimization. GPT-4 costs 30 times more than GPT-4o-mini. If you send all requests to GPT-4, you burn money on simple tasks like translation or keyword extraction. A router classifies each request: simple queries go to cheap models, complex reasoning goes to expensive models.

The key components are a classifier (which determines request type), a routing table (which maps request types to models), and a fallback chain (which handles model failures). For example, a customer support chatbot routes greetings to GPT-4o-mini, account questions to Claude Haiku, and legal analysis to GPT-4 or Claude Opus. The router saves cost without sacrificing quality for complex queries."

**Common Mistakes:**
- Only discussing one decision factor (usually cost)
- Not mentioning fallback behavior
- Using vague model descriptions instead of naming specific models
- Not connecting the explanation to a real use case

### Challenge 2: Classify Routing Strategies

**Prompt:** You are given a list of five routing strategies. For each one, classify it as rule-based, ML-based, or hybrid. Then rank them from fastest to slowest in terms of routing decision time, and from least to most accurate in classification.

**Strategies:**
1. "If the request contains 'code' or 'debug', route to Codex. If it contains 'translate', route to translation model."
2. "Embed each request into a 384-dimensional vector, then find the nearest centroid among 20 predefined category clusters."
3. "Send each request to a small LLM with the prompt 'Classify this request into one of these 10 categories.' Use the response to select the model."
4. "Use a keyword matcher that runs first. If the keyword confidence is below 0.7, fall through to an embedding-based classifier."
5. "Train a random forest classifier on a labeled dataset of 100K requests. Use the predicted category to select the model."

**Evaluation Criteria:**
- Correctly identifies each strategy type
- Ranking is logically consistent with known performance characteristics
- Understands the relationship between speed and accuracy

**Model Answer:**
- Strategy 1: Rule-based. Fastest routing (under 1ms). Least accurate (misses nuanced requests).
- Strategy 5: ML-based (random forest). Fast (1-5ms inference). Medium accuracy (depends on training data quality).
- Strategy 2: ML-based (embedding). Fast (5-10ms including embedding). Good accuracy (captures semantic similarity).
- Strategy 4: Hybrid. Medium speed (5-15ms total). High accuracy (catches both keyword and semantic patterns).
- Strategy 3: ML-based (prompt-based LLM). Slowest (200-500ms). Highest accuracy (LLM understands nuance).

**Common Mistakes:**
- Ranking embedding-based as slower than prompt-based
- Not identifying Strategy 4 as hybrid
- Assuming ML-based is always slower than rule-based
- Not accounting for embedding generation time

### Challenge 3: Identify Routing Needs

**Prompt:** You are consulting for three different startups. For each one, determine whether they need model routing and explain why or why not. If they need it, describe the simplest routing setup that would help.

**Startup A:** A writing assistant that uses GPT-4 exclusively. They process 500 requests per day. Their monthly API bill is $600. Users are satisfied with response quality but some mention it feels slow.

**Startup B:** An enterprise document analysis platform that uses 8 different models for different tasks: Claude Opus for legal analysis, GPT-4 for financial analysis, Mistral for summarization, and smaller models for extraction, classification, translation, proofreading, and formatting. They process 50K requests per day.

**Startup C:** A batch processing pipeline that runs 500K documents through a single fine-tuned model every night. The model costs $5 per 1K documents. The pipeline takes 8 hours to complete and has no real-time requirements.

**Evaluation Criteria:**
- Correctly identifies which startups need routing and which do not
- Justifies the recommendation with specific reasoning
- Suggests appropriate routing complexity for each case

**Model Answer:**
Startup A: Model routing would help marginally. At 500 requests per day and $600/month, they could save 40-60 percent by routing simple prompts to GPT-4o-mini. However, the absolute savings are only $240-360 per month, which may not justify building and maintaining a routing system. Recommendation: Use a simple keyword-based router that sends proofreading and translation requests to GPT-4o-mini. Do not build an ML classifier until scale increases.

Startup B: Model routing is essential. They already use 8 models but presumably route manually in the application code. A dedicated routing layer would let them centralize routing logic, add fallback chains, and track costs. The volume (50K/day) justifies the investment. Recommendation: Build an embedding-based router with a routing table in Redis. Each document type maps to a specific model with fallback. Add cost tracking per model.

Startup C: Model routing is not needed. A single model with no real-time requirements and batch processing does not benefit from routing. The cost is already low ($5/1K documents) and the 8-hour window is acceptable. Recommendation: Optimize the existing pipeline for throughput. Do not add routing complexity.

**Common Mistakes:**
- Recommending routing for Startup C without considering the batch/non-real-time nature
- Over-engineering the solution for Startup A's small scale
- Not recognizing that Startup B already has an implicit routing problem

## Tier 2: Design

### Challenge 4: Design Router for Customer Support Chatbot

**Prompt:** Design a model routing system for a customer support chatbot that handles 200K conversations per day. The chatbot supports 15 languages and needs to handle simple FAQs, account issues, billing questions, technical support, and escalation to human agents. The monthly budget for model API costs is $10,000.

**Evaluation Criteria:**
- Estimates QPS and traffic distribution across categories
- Specifies the classification approach with rationale
- Designs routing table schema with fallback chains
- Addresses multilingual requirements
- Stays within the $10K/month budget with a cost breakdown

**Model Answer:**
200K conversations per day at 8-hour peak = approximately 25K conversations per peak hour = approximately 7 conversations per second peak. Each conversation has 3-5 messages, so approximately 28 requests per second.

Traffic distribution estimate: FAQ (40 percent), account (20 percent), billing (15 percent), technical (15 percent), escalation (10 percent).

Classification approach: Embedding-based using a multilingual embedding model (e.g., multilingual-e5-small). Embed each message and match against 15 language-specific category centroids. Latency: 10ms. Accuracy: 92 percent for clear cases.

Routing table:
- FAQ, translations, greetings: GPT-4o-mini ($0.15/1M input tokens)
- Account, billing: Claude Haiku ($0.25/1M input tokens)
- Technical support: GPT-4o ($2.50/1M input tokens)
- Legal/compliance: Claude Opus ($15.00/1M input tokens)
- Escalation: Route to human agent (free)

Fallback chain: Primary -> GPT-4o-mini -> GPT-4o (for any category where primary fails)

Cost estimate: At approximately 500K tokens per day average, with distribution: 40 percent cheap + 35 percent medium + 15 percent expensive + 10 percent human = approximately $8,500/month. Under budget.

Multilingual: Use a single multilingual embedding model. Category centroids are computed per language cluster.

**Common Mistakes:**
- Not estimating traffic distribution
- Using prompt-based classification for a latency-sensitive chatbot
- Ignoring the multilingual requirement
- Not providing a cost estimate that fits the budget

### Challenge 5: Design Cost-Optimized Routing Pipeline

**Prompt:** Your team manages a content generation platform that produces 50K pieces of content per day. You have access to 12 models from 3 providers with different pricing, latency, and quality profiles. The requirement is to minimize cost while maintaining quality above a minimum threshold. Design the routing pipeline.

**Evaluation Criteria:**
- Defines a quality metric and measurement approach
- Designs a cost-optimization strategy with clear decision logic
- Includes a feedback loop for continuous improvement
- Handles model provider outages and pricing changes

**Model Answer:**
Quality metric: Use a composite score from automated evaluation (Rouge-L, BERTScore) plus human rating samples. Threshold: minimum composite score of 0.85.

Pipeline design:
1. Request type classifier: Embedding-based, 10 categories (blog post, product description, email, social media, etc.)
2. Quality predictor: For each category, maintain a performance matrix mapping each model to its average quality score, cost, and latency.
3. Optimization engine: Solve a knapsack-like optimization: select the cheapest model whose quality score exceeds 0.85 for that category.
4. Fallback: If the cheapest qualifying model fails, try the next cheapest, up to 3 attempts.

Feedback loop: For every 100th request, route to a random model (including more expensive ones) to measure whether cheaper models still meet the quality bar. Update the performance matrix daily.

Provider outage handling: Maintain a health-check endpoint for each provider. If a provider is down, exclude all their models from the optimization and reroute.

Pricing change handling: Ingest pricing updates from provider APIs daily. Recompute the optimization matrix on change.

**Common Mistakes:**
- Only minimizing cost without a quality threshold
- Not including a feedback loop for quality measurement
- Ignoring provider-level failures (only handling per-model failures)
- Assuming static model quality (quality degrades over time without monitoring)

### Challenge 6: Design Multi-Region Routing

**Prompt:** A global SaaS company needs model routing across 5 regions (us-east, us-west, eu-west, ap-southeast, ap-northeast). They use 6 models from 2 providers. Traffic is 10M requests per day, distributed across regions with latency sensitivity (users expect under 500ms total response time). Design the routing system.

**Evaluation Criteria:**
- Describes the global routing topology
- Explains routing table replication strategy
- Addresses regional model availability differences
- Includes latency-based routing with failover
- Models the math for regional capacity

**Model Answer:**
Global topology: Each region has a local routing service cluster. A global model registry in us-east serves as the source of truth. Configuration replicates asynchronously to regional registries using a change-data-capture pipeline.

Routing table replication: Each region's routing service caches the table locally with a 30-second TTL. The table version is a monotonic counter. Regional routers serve from cache; they never block on a cross-region read. If the cache is stale, the router still serves the last known configuration.

Regional model availability: Not all models are available in all regions. The routing table includes a region-specific availability mask. The router only considers models available in the user's region.

Latency-based routing: For each request, the router calculates:
- Expected local model latency (from latency histogram, p50)
- Expected cross-region latency (p50 network time to nearest region with the model)
If local model meets the latency budget, use local. Otherwise, route to the nearest region with the model available.

Failover: At the model level -- if primary model in region fails, try secondary in same region. At the region level -- if the entire regional routing service is degraded, DNS-based failover to a healthy region.

Capacity math: 10M requests/day = approximately 116 QPS average, approximately 350 QPS peak. With p99 latency target of 100ms for routing, each routing instance handles 500 QPS. Need 1-2 instances per region with 2x headroom. Total: 10-20 instances globally.

**Common Mistakes:**
- Designing a single-region routing system and calling it multi-region
- Not addressing regional model availability differences
- Ignoring cross-region latency in the routing decision
- Over-provisioning without capacity math

## Tier 3: Expert

### Challenge 7: Design Routing for a 50-Model Platform

**Prompt:** Your company maintains 50 fine-tuned models for different enterprise clients. Each model has different capabilities, pricing, latency, and availability guarantees. Clients send requests through a shared gateway. Design a routing system that ensures each client's requests go to the correct model, handles capacity constraints (each model instance can handle limited concurrent requests), and optimizes for cost.

**Evaluation Criteria:**
- Handles multi-tenant routing (client-to-model mapping)
- Addresses capacity management per model instance
- Includes request queuing with priority
- Designs a cost-allocation system
- Discusses isolation between client traffic

**Model Answer:**
Multi-tenant routing: Each client request includes a tenant ID. The routing table maps tenant ID to one or more allowed models. The router validates that the requested model is in the tenant's allowed list. This prevents cross-tenant routing errors.

Capacity management: Each model instance reports its current queue depth and estimated time-to-completion via a health endpoint. The routing service maintains a capacity matrix: `{model_id, instance_id, queue_depth, max_concurrency, avg_completion_time}`. When selecting an instance for a request, the router picks the instance with the shortest expected queue time. If all instances of a model are at capacity, the router queues the request or routes to a fallback model with client approval.

Request queuing with priority: Use a priority queue per model. Each tenant has a priority level (platinum, gold, silver, free). Higher priority requests jump the queue. The queue depth per tenant is capped to prevent any single tenant from starving others. Queue wait time is exposed as a metric.

Cost allocation: Track cost per tenant per model per request. Generate daily cost reports. Implement per-tenant cost caps: if a tenant exceeds their daily budget, their requests route to a cheaper fallback model for the remainder of the day.

Tenant isolation: Each model instance serves one tenant exclusively (no multi-tenancy on the GPU). The routing table ensures tenant A's requests never reach tenant B's model instance. Authentication and tenant validation happen before routing.

**Common Mistakes:**
- Ignoring tenant isolation (routing is not just about model selection)
- Not addressing capacity management (model instances have limits)
- Forgetting cost allocation and per-tenant budgets
- Designing a flat routing table without multi-tenant considerations

### Challenge 8: Dynamic Cost Optimization

**Prompt:** Design a model routing system with dynamic cost optimization. Model pricing varies by time of day (off-peak discounts), by provider (competing pricing), and by usage tier (volume discounts). Your system must continuously optimize routing decisions to minimize total cost while meeting latency and quality constraints. Real-time pricing changes must be incorporated without downtime.

**Evaluation Criteria:**
- Models the optimization as a constrained optimization problem
- Handles dynamic pricing changes
- Includes real-time pricing ingestion
- Discusses the trade-off between optimization frequency and stability
- Addresses cold-start problem (no historical data)

**Model Answer:**
Optimization formulation: Minimize `sum(cost(m, r))` for all requests r, subject to `latency(r) under L` and `quality(r) over Q`. This is a variant of the assignment problem. Since the full problem is NP-hard for 50 models and 10K requests/second, use a greedy approximation: for each request, select the cheapest model that meets the latency and quality constraints.

Dynamic pricing ingestion: Each provider exposes a pricing API. A pricing poller runs every 5 minutes, fetches current prices, and updates the pricing matrix. Price changes trigger a recomputation of the optimal routing policy. To avoid routing flapping (oscillation between models due to tiny price differences), apply a hysteresis threshold: only switch models if the cost difference exceeds 5 percent.

Real-time optimization loop:
1. Request arrives at router
2. Look up current pricing matrix (in-memory, updated every 5 minutes)
3. Filter models that meet latency budget for this request type
4. Filter models that meet quality threshold for this request type
5. Select cheapest remaining model
6. Check circuit breaker: if model has failed more than N times in window, skip it
7. Route request and record actual cost, latency, quality

Cold-start solution: Without historical data, route a representative sample of requests to each available model for the first hour. Collect latency, cost, and quality data. Seed the quality model with these samples. During the cold-start period, use a conservative default: route to the medium-cost, medium-capability model.

Stability optimization: Only adjust routing policies every 5 minutes (not per-request). Use a shadow evaluation: compute what the optimal routing would have been for the last 5-minute window and compare it to what was actually used. Log the gap as a metric. This prevents overfitting to transient pricing changes.

**Common Mistakes:**
- Solving the wrong optimization (minimizing cost without quality/latency constraints)
- Not addressing routing flapping due to pricing changes
- Ignoring the cold-start problem
- Optimizing per-request instead of per-window (creates thrashing)
- Not validating against a baseline (no evaluation of whether optimization actually helps)

### Tier 1 Solutions

**Challenge 1 Model Answer (Expanded):**

"Model routing is a decision layer that sits between your application and the AI models you call. Instead of sending every request to a single model, the router examines each request and selects the best model based on four factors: cost, latency, capability, and safety.

For example, a customer support chatbot that sends every query to GPT-4 might ask a simple question like 'What are your hours?' and pay $0.01 for a response that a $0.0003 model could answer perfectly. Over 1 million requests per day, that difference adds up to $10,000 per month in unnecessary cost.

The key components are: a classifier that determines request type and complexity, a routing table that maps request types to model assignments, and a fallback chain that handles model failures. When a request arrives, the router classifies it, looks up the best model, validates constraints, and sends the request to the selected model. If that model fails, the fallback chain tries the next model in line.

The goal is to send the right request to the right model -- not the most powerful model for every request, but the most appropriate model for each specific request."

## Tier 2 Challenge: Detailed Solutions

### Challenge 4: Detailed Solution

**Traffic estimation:** 200K conversations/day at 3 messages per conversation = 600K requests/day. Peak hour is 8x average = 75K requests/hour = 21 requests/second peak. With 2x headroom, design for 50 QPS.

**Classification approach:** Embedding-based with multilingual model. Use sentence-transformers with a multilingual model (distiluse-base-multilingual-cased-v2). Pre-compute centroids for each category in each language. Total: 15 languages * 5 categories = 75 centroids. Each centroid is a 512-dimensional vector. Total centroid storage: 75 * 512 * 4 bytes = 154KB.

**Routing table schema:**
```
Key: "{language}:{category}"
Value: {
  "primary": "model_name",
  "fallback": "model_name",
  "cost_cap": 0.001,
  "max_latency_ms": 2000,
  "confidence_threshold": 0.7
}
```

**Fallback chain:** For each category, define two fallback models. Example for FAQ: primary = GPT-4o-mini, fallback1 = Claude Haiku, fallback2 = GPT-4o. For escalation: primary = human agent (via ticket system), fallback = GPT-4o (with note that this is an AI-generated response).

**Budget breakdown:**
- 600K requests/day * 30 days = 18M requests/month
- 60 percent simple (10.8M): GPT-4o-mini at $0.0001/request = $1,080
- 25 percent moderate (4.5M): Claude Haiku at $0.0003/request = $1,350
- 15 percent complex (2.7M): GPT-4o at $0.002/request = $5,400
- Total: $7,830/month (under $10K budget)

### Challenge 5: Detailed Solution

**Quality metric design:** Use a composite score of three signals:
1. Automated: BERTScore (0-1, weight 0.4)
2. User feedback: thumbs up/down (0-1, weight 0.3)
3. Task completion: did the output meet the specified format? (0-1, weight 0.3)

Composite score threshold: 0.85 for production quality.

**Optimization algorithm:**
```python
def select_model(request, performance_matrix, cost_matrix):
    category = classify(request.text)
    eligible = []
    for model in performance_matrix[category]:
        if performance_matrix[category][model]["score"] >= 0.85:
            # passes quality floor
            cost = cost_matrix[model]["input"] * request.input_tokens
            cost += cost_matrix[model]["output"] * request.expected_output_tokens
            eligible.append((model, cost))
    if not eligible:
        # No model meets quality floor, select the best available
        return max(performance_matrix[category], 
                   key=lambda m: performance_matrix[category][m]["score"])
    return min(eligible, key=lambda x: x[1])[0]  # cheapest that qualifies
```

**Feedback loop:** Every 100th request, send to a random model (including those below the quality floor) to verify that the floor is still appropriate. If a model consistently scores above the floor, update the performance matrix. If a model drops below the floor, exclude it from the eligible list.

### Challenge 6: Detailed Solution

**Regional model availability matrix:**
| Model | us-east | us-west | eu-west | ap-southeast | ap-northeast |
|-------|---------|---------|---------|-------------|-------------|
| GPT-4o | yes | yes | yes | yes | yes |
| GPT-4o-mini | yes | yes | yes | yes | yes |
| Claude Sonnet | yes | yes | yes | no | no |
| Claude Haiku | yes | yes | yes | no | no |

**Latency-based routing logic:**
```python
def select_region(request, user_location):
    local_region = get_nearest_region(user_location)
    model = select_model(request)
    
    # Check if model is available in local region
    if model in regional_availability[local_region]:
        expected_latency = latency_estimator.predict(local_region, model)
        if expected_latency < request.max_latency_ms:
            return local_region, model
    
    # Find nearest region with the model that meets latency budget
    candidates = []
    for region in regions:
        if model in regional_availability[region]:
            network_latency = latency_estimator.network_latency(user_location, region)
            model_latency = latency_estimator.predict(region, model)
            total_latency = network_latency + model_latency
            if total_latency < request.max_latency_ms:
                candidates.append((region, total_latency))
    
    if candidates:
        return min(candidates, key=lambda x: x[1])  # fastest
    
    # No region meets latency budget, use best effort
    return local_region, select_fallback_model(request)
```

**Capacity math:** 10M requests/day = 115 QPS average, 350 QPS peak. Each router instance handles 5K QPS. Need 1 instance per region with 2x headroom for failover. Total: 2 instances * 5 regions = 10 instances. Each instance caches the routing table locally (500KB). Total memory: 10 * 500KB = 5MB for routing tables.
