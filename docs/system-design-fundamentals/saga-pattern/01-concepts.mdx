# Understanding the Saga Pattern

## The Problem Story

A customer books a complete vacation package through your travel booking platform. The request hits the API and the system needs to coordinate three separate services:

1. **Flight Service** — Reserve seat AA-1234, New York to London, July 15.
2. **Hotel Service** — Book the Standard King at The Kensington, July 15-20.
3. **Car Rental Service** — Reserve a midsize sedan at Heathrow, pickup July 15, drop-off July 20.

Each service runs its own database. There is no shared transaction coordinator, no global lock, and no two-phase commit protocol. The three databases are independent — managed by different teams, deployed on different clusters, possibly in different regions.

The booking request arrives. The Flight Service confirms the reservation. The Hotel Service confirms the booking. The Car Rental Service responds: "No midsize sedans available for those dates."

Now the customer has a confirmed flight and hotel but no car. The customer cannot cancel the flight without a penalty. The hotel cancellation window has already passed. The customer disputes the charge. The support team spends hours untangling the mess. The customer never books with this platform again.

This is the distributed transaction problem in its purest form. In a monolithic database with ACID transactions, a `ROLLBACK` would undo the flight and hotel reservations atomically. In a distributed system with independent services and databases, there is no `ROLLBACK` across service boundaries. The system needs a different approach.

**The core challenge**: how do you maintain data consistency across multiple services and databases when a single atomic transaction is not possible?

---

## The Insight

The Saga pattern solves this by breaking a long-lived distributed transaction into a sequence of smaller, local transactions — each one executed against a single service and its local database. Each local transaction is followed by a compensating transaction that undoes it if a later step fails.

Think of it like planning a road trip with multiple stops. You book a hotel in Boston, then a rental car, then a hotel in New York. If the New York hotel is unavailable, you do not roll back the entire trip — you cancel the Boston hotel and the rental car individually. Each cancellation is a compensating action for the original booking. The trip either has all reservations confirmed, or every reservation is cancelled and you are back to the starting point.

A saga guarantees one of two outcomes:

- **Forward path**: Every local transaction succeeds. The saga is complete.
- **Backward path**: One or more local transactions fail. Every successful transaction is undone by its compensating action. The system returns to its initial state.

This is eventual consistency with bounded recovery. The system is never in an inconsistent state for longer than the saga execution time.

## Core Vocabulary

| Term | Meaning |
|---|---|
| Saga | A sequence of local transactions where each has a compensating action for rollback |
| Local transaction | An ACID transaction executed against a single service's database |
| Compensating transaction | An operation that semantically undoes a previous local transaction (e.g., "Cancel Booking" for "Create Booking") |
| Choreography | Each saga participant decides what to do next by reacting to events from other participants — no central coordinator |
| Orchestration | A central coordinator tells each participant what to do and tracks progress — single point of decision |
| Saga log | A durable record of every step and its outcome, used for recovery after crashes |
| Compensation action | The code or API call that reverses the effect of a committed local transaction |
| Backward recovery | Undoing completed steps when a step fails — compensating in reverse order |
| Forward recovery | Retrying a failed step rather than undoing previous steps — useful when the failure is transient |
| Saga participant | A service that executes a local transaction and provides the compensating action |
| Saga coordinator | In orchestration, the component that manages the saga's execution and state |
| Idempotency key | A unique identifier for each saga step so that retries do not cause duplicate side effects |

### Step-by-Step Walkthrough — Happy Path

**Step 1: Coordinator sends "Reserve flight" to Flight Service.**
The Flight Service checks seat availability on AA-1234. The seat is available. It creates a reservation record in its own database and marks the seat as held. It returns success to the coordinator. The coordinator writes "step 1 = COMPLETED" to the saga log.

**Step 2: Coordinator sends "Book hotel" to Hotel Service.**
The Hotel Service checks room availability. The Standard King is available for July 15-20. It creates a booking record and returns success. The coordinator writes "step 2 = COMPLETED" to the saga log.

**Step 3: Coordinator sends "Reserve car" to Car Rental Service.**
The Car Rental Service checks fleet availability. A midsize sedan is available. It creates a reservation and returns success. The coordinator writes "step 3 = COMPLETED".

**Step 4: Coordinator sends "Charge payment" to Payment Service.**
The Payment Service authorizes the credit card for $2,450 and captures the funds. It creates a transaction record and returns success. The coordinator writes "step 4 = COMPLETED" and marks the entire saga as "COMPLETED".

The booking is confirmed. All four services have committed their local transactions. The system is consistent.

### Step-by-Step Walkthrough — Failure Path

**Step 1: Flight reservation succeeds.** Same as happy path. The saga log records step 1 as COMPLETED.

**Step 2: Hotel booking succeeds.** Same as happy path. The saga log records step 2 as COMPLETED.

**Step 3: Car rental fails.** The Car Rental Service checks availability and finds no midsize sedans. It returns a failure response to the coordinator. The coordinator writes "step 3 = FAILED" to the saga log.

**Step 4: Coordinator begins compensation.** The saga log now has two completed steps and one failure. The coordinator reads the log and determines it must compensate steps 2 and 1 in reverse order.

**Step 5: Compensate step 2 — Cancel hotel.** The coordinator sends a compensation request to the Hotel Service with the original booking ID. The Hotel Service cancels the booking, releases the room, and waives any cancellation fees. It returns success. The coordinator records "step 2 compensation = COMPLETED".

**Step 6: Compensate step 1 — Cancel flight.** The coordinator sends a compensation request to the Flight Service. The Flight Service cancels the reservation, releases seat AA-1234, and waives fees. It returns success. The coordinator records "step 1 compensation = COMPLETED".

**Step 7: Mark saga as COMPENSATED.** All compensations succeeded. The customer is never charged. The system is consistent.

### What If a Compensation Fails?

If the hotel cancellation fails — perhaps the Hotel Service is down — the coordinator has a problem. It cannot leave the flight booked with no compensation for the hotel. The coordinator retries the compensation. If the Hotel Service remains down after all retries are exhausted, the coordinator marks the saga as COMPENSATION_FAILED and alerts an operator for manual intervention.

## Variants: Choreography vs Orchestration

The saga pattern has two main implementation styles. Each style makes different trade-offs in coupling, complexity, and observability.

### Choreography Saga

In a choreographed saga, there is no central coordinator. Each service executes its local transaction and then emits an event. Other services listen for events and decide what to do next. The logic is distributed across all participants.

**How choreography works:**
1. The Flight Service books the seat and publishes a "FlightBooked" event.
2. The Hotel Service subscribes to "FlightBooked" events. When it receives one, it books the room and publishes a "HotelBooked" event.
3. The Car Rental Service subscribes to "HotelBooked" events. It reserves the car and publishes a "CarBooked" event.
4. The Payment Service subscribes to "CarBooked" events. It charges the customer.

**Compensation in choreography:**
If the Car Rental Service fails to reserve a car, it publishes a "CarBookingFailed" event. Both the Hotel Service and Flight Service subscribe to failure events and each independently cancels its own booking. Each service is responsible for its own compensation.

**Pros:** No single point of failure. No coordinator to deploy, maintain, or scale. Simple to understand for small numbers of participants.
**Cons:** Complex to reason about for more than 3-4 participants. No single place to view saga progress. Event contracts create coupling between services. Difficult to add new steps without understanding the full event chain.

### Orchestration Saga

In an orchestrated saga, a central Saga Coordinator manages the entire flow. The coordinator decides which service to call next, handles failures, and triggers compensation.

**How orchestration works:**
1. The Saga Coordinator receives the booking request.
2. The coordinator calls the Flight Service directly (HTTP or gRPC). It receives a success response and records step 1 as COMPLETED in the saga log.
3. The coordinator calls the Hotel Service. Success. Records step 2 as COMPLETED.
4. The coordinator calls the Car Rental Service. Failure. Records step 3 as FAILED.
5. The coordinator looks up the saga definition and determines it must compensate steps 2 and 1 in reverse order.
6. The coordinator calls the Hotel Service with a cancel request. Success. Records compensation as COMPLETED.
7. The coordinator calls the Flight Service with a cancel request. Success. Records compensation as COMPLETED.
8. The coordinator marks the saga as COMPENSATED.

**Pros:** Single place to manage all logic. Easy to add, remove, or reorder steps. Full visibility into saga state. Each service only needs to implement its own API — no event schema coordination required.
**Cons:** Single point of failure (the coordinator). The coordinator must be highly available and durable. Additional latency from all calls routing through the coordinator.

### Comparison

| Aspect | Choreography | Orchestration |
|---|---|---|
| Coordination model | Event-driven, decentralized | Command-driven, centralized |
| Coupling between services | Tight (event schemas) | Loose (service APIs) |
| Single point of failure | No | Yes (coordinator) |
| Observability | Distributed (many event logs) | Centralized (one saga log) |
| Complexity scales with | Linear per service | Linear per saga definition |
| Adding new participants | May require changing existing services | Update coordinator definition only |
| Debugging difficulty | High (trace across services) | Moderate (trace in coordinator) |
| Best for | Simple, linear workflows | Complex workflows with branching |

## Mental Model Check

Answer these five questions to verify your understanding. Each targets a common point of confusion about the saga pattern.

### Question 1

A developer says: "A saga is just a distributed ACID transaction with extra steps." Is this statement correct?

**Answer:** No. A saga provides atomicity across services (all-or-nothing through compensation), but it does not provide isolation. In an ACID transaction, other transactions see either the full result or nothing at all. In a saga, intermediate states are visible — after step 1 commits, other consumers can see the flight reservation even though the saga might later compensate. Sagas provide eventual consistency with bounded recovery, not the strict isolation of ACID.

### Question 2

A team implements a choreographed saga with four services. The Car Rental Service emits a "CarBooked" event. The Payment Service receives it and charges the customer. But another instance of the Car Rental Service, processing a different saga, also emits a "CarBooked" event for the same booking ID. The customer is charged twice. What went wrong?

**Answer:** Two issues: (1) The system lacks idempotency — the Payment Service should check whether it has already processed this booking ID before charging. (2) The Car Rental Service should deduplicate its event emission or use an idempotency key so downstream services can detect duplicates. In event-driven systems, at-least-once delivery means duplicates are expected, not exceptional.

### Question 3

A saga coordinator crashes after recording step 2 as COMPLETED but before calling step 3. The saga log is persisted to a database. On restart, what must the coordinator do?

**Answer:** The coordinator reads the saga log from the database and finds a saga in the ACTIVE state with step_1 = COMPLETED and step_2 = COMPLETED, but no entry for step_3. The coordinator must determine which step to execute next. Since step 3 was never called, the coordinator resumes from step 3. If there is ambiguity — the coordinator cannot tell whether step 3 was called but not logged — it must check with the Car Rental Service using the idempotency key. If the Car Rental Service already processed step 3, the coordinator skips it. If not, the coordinator calls step 3.

### Question 4

An engineer argues that sagas completely replace two-phase commit (2PC). Under what conditions is 2PC still the better choice?

**Answer:** 2PC is the better choice when: (1) The data resources support XA-compliant transactions (most relational databases do). (2) The transaction spans only a few participants in the same network with low latency. (3) Strict isolation is required — intermediate states must not be visible. (4) The transaction volume is low enough that the coordinator does not become a bottleneck. Sagas are better when participants are long-running, involve external APIs you cannot lock, or require high availability over strict atomicity.

### Question 5

A payment saga includes a "Reserve Funds" step followed by "Capture Funds." The reserve succeeds. The capture fails because the payment gateway rejects the request. What compensation is needed?

**Answer:** The "Reserve Funds" step must be released. If the payment gateway has an API for releasing a reservation (voiding a hold), the compensation calls that API. If the reservation expires automatically after a timeout, the compensation may simply wait for expiration — but this is risky because the funds remain unavailable to the customer during the holding period. The safer approach is an explicit release or void call with idempotency to handle retries.

### Question 6

A team designs a choreographed saga with four services. The hotel service subscribes to the "FlightBooked" event. The car rental service subscribes to the "HotelBooked" event. The hotel service crashes and misses several "FlightBooked" events while down. When it recovers, it does not retroactively process the missed events. What design change ensures the hotel service does not miss events during downtime?

**Answer:** In a choreographed saga, every service must handle the possibility of being down when events are published. Two solutions: (1) Use a message queue with persistent subscriptions — the queue retains messages for each subscriber until they are consumed. When the hotel service recovers, it reads all unconsumed messages from its queue. (2) Implement a reconciliation process that periodically checks for incomplete sagas and re-publishes events for any that are stuck. Most production event buses (Kafka, RabbitMQ, SQS) support persistent subscriptions out of the box and this is the recommended approach.

### Question 7

An orchestrated saga for a food delivery app has steps: accept order, notify restaurant, assign driver, dispatch food. The "notify restaurant" step sends a push notification to the restaurant's tablet. The tablet is offline. The notification fails. The saga coordinator does not retry because push notifications are fire-and-forget. The restaurant never receives the order. The driver arrives at the restaurant and finds no food prepared. What is the flaw in the saga design?

**Answer:** The "notify restaurant" step should not be a fire-and-forget push notification. It should be a reliable step that confirms the restaurant acknowledged the order. Acceptable designs: (1) The step sends the order to the restaurant's order management API and waits for an HTTP 200 response confirming receipt. (2) The step commits a database record that the restaurant's system polls. (3) The step uses a confirmed delivery mechanism (like a queue with acknowledgment) instead of a push notification. The key principle: every saga step must have a verifiable outcome. Fire-and-forget operations have no verifiable outcome and cannot be part of a saga.
