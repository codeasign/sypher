# Saga Pattern at Scale

## Naive Implementation: Choreography with Callbacks

The simplest saga implementation uses choreography with direct HTTP callbacks between services. Each service, after completing its step, calls the next service's API directly.

## Production Architecture: Orchestration with Saga Coordinator

A production saga system uses a dedicated saga coordinator deployed as a horizontally scalable service. The coordinator communicates with participants through a message queue for reliable delivery and uses a state store for durability.

### Command Queues

The coordinator does not call saga participants directly. Instead, it publishes commands to a message queue. Each participant subscribes to its command queue and processes messages as they arrive.

**Why use queues instead of direct calls:**
- **Decoupling**: The coordinator does not wait for a response. It publishes a command and moves on.
- **Buffering**: If a participant is slow or down, commands accumulate in the queue instead of blocking the coordinator.
- **Exactly-once delivery semantics**: Message queues provide at-least-once delivery. Combined with idempotency keys in each message, the system achieves effectively-once processing.
- **Backpressure**: If participants cannot keep up, the queue grows rather than the coordinator timing out. This provides natural backpressure.

### Event Queues

After processing a command, each participant publishes a result event (success or failure with payload) to an event queue. The coordinator subscribes to these events and processes them to advance the saga state machine.

**Event schema:**
```json
{
  "event_type": "saga.step.completed",
  "saga_id": "ord_789",
  "step_seq": 2,
  "service": "payment",
  "status": "COMPLETED",
  "payload": {
    "charge_id": "ch_abc123",
    "amount": 150.00,
    "currency": "USD"
  },
  "idempotency_key": "evt_ord_789_pay_2_001"
}
```

The coordinator processes events asynchronously. It reads the event, looks up the saga instance, and transitions the state. This means the coordinator is never blocked waiting for a participant — it processes events at its own pace.

### Dead-Letter Handling

When a saga cannot complete — either because a forward step keeps failing or a compensation keeps failing after exhausting all retries — the coordinator moves the saga instance to a dead-letter queue.

```python
def handle_saga_failure(saga_id, reason):
    # Move to dead letter queue
    dead_letter_queue.publish({
        "saga_id": saga_id,
        "reason": reason,
        "failed_step": saga.current_step,
        "attempts": saga.retry_count,
        "last_error": saga.last_error,
        "created_at": now()
    })

    # Alert the operations team
    alerting.send(
        channel="saga-failures",
        severity="CRITICAL",
        message=f"Saga {saga_id} failed after {saga.retry_count} retries: {reason}"
    )
```

Operators have tools to inspect dead-lettered sagas and take action:
- **Retry**: Manually trigger another compensation attempt.
- **Skip compensation**: Acknowledge the failure and mark the saga as requiring manual reconciliation.
- **Force complete**: Manually apply the compensating action and close the saga.

## Integration Patterns

### Integration with Message Queues

Message queues serve as the backbone for command and event communication in a production saga system.

```python
# Coordinator publishes a command to a queue
def execute_step(saga_id, step_definition):
    command = {
        "command_type": f"saga.{step_definition.service}.{step_definition.action}",
        "saga_id": saga_id,
        "step_seq": step_definition.seq,
        "payload": step_definition.payload,
        "idempotency_key": f"{saga_id}_{step_definition.service}_{step_definition.seq}"
    }

    queue.publish(
        exchange="saga.commands",
        routing_key=step_definition.service,
        message=command,
        # Message persists to disk — survives queue crash
        delivery_mode=2,
        # TTL: message expires after step timeout
        expiration=str(step_definition.timeout_ms)
    )
```

The participant's consumer is idempotent — it checks for duplicate messages using the idempotency key before processing.

```python
def handle_command(channel, method, properties, body):
    command = json.loads(body)

    # Idempotency check
    if is_already_processed(command["idempotency_key"]):
        channel.basic_ack(delivery_tag=method.delivery_tag)
        return

    try:
        # Execute the business logic
        result = execute_business_logic(command)

        # Store the result for idempotency
        mark_as_processed(command["idempotency_key"], result)

        # Publish success event
        event_queue.publish({
            "event_type": "saga.step.completed",
            "saga_id": command["saga_id"],
            "step_seq": command["step_seq"],
            "status": "COMPLETED",
            "payload": result
        })

        channel.basic_ack(delivery_tag=method.delivery_tag)

    except Exception as e:
        # Log failure and publish failure event
        event_queue.publish({
            "event_type": "saga.step.failed",
            "saga_id": command["saga_id"],
            "step_seq": command["step_seq"],
            "status": "FAILED",
            "error": str(e)
        })
        channel.basic_ack(delivery_tag=method.delivery_tag)
```

### Integration with Databases

The saga log database must support the coordinator's access patterns: point lookups by saga_id and range queries for sagas in certain states.

```sql
-- Query: Find all sagas that need recovery (stuck in ACTIVE or COMPENSATING)
SELECT saga_id, status, updated_at
FROM saga_instances
WHERE status IN ('ACTIVE', 'COMPENSATING')
  AND updated_at < NOW() - INTERVAL '5 minutes'
ORDER BY updated_at ASC
LIMIT 100;

-- Query: Find all COMPENSATION_FAILED sagas for operator review
SELECT saga_id, updated_at, payload
FROM saga_instances
WHERE status = 'COMPENSATION_FAILED'
ORDER BY updated_at DESC
LIMIT 50;
```

For high-throughput systems, the saga log should be sharded by saga_id to distribute write load.

### Idempotency Across Regions

Idempotency keys must be globally unique to prevent duplicate processing during region failover. Include a region prefix and a timestamp component.

```
Format: {region}_{saga_id}_{step_seq}_{attempt}
Example: use1_ord_789_pay_2_001
```

When a saga fails over to a standby region, the new coordinator reads the saga state from the replicated log and reissues any in-flight commands with the same idempotency keys. Participants in the original region see the idempotency key and return cached responses instead of executing duplicate work.

### Lease-Based Saga Ownership

Each saga instance is owned by exactly one coordinator at a time, enforced through a lease mechanism:

```sql
-- Acquire lease on a saga instance (optimistic lock)
UPDATE saga_instances
SET owner = 'crd_us_east_1_001',
    lease_expires_at = NOW() + INTERVAL '30 seconds'
WHERE saga_id = 'ord_789'
  AND (owner IS NULL OR lease_expires_at < NOW());
```

If the coordinator crashes, its leases expire after 30 seconds. A coordinator in another region picks up the orphaned saga and resumes it.

## Capacity Planning

### Request Throughput

Each saga step consumes coordinator resources:
- **CPU**: Serializing/deserializing commands and events, running state machine logic.
- **Memory**: Active saga instances in memory (state, payload, retry counters).
- **Database IOPS**: Each step requires 2-3 database writes (step start, step completion, saga status update).

**Formula for coordinator capacity:**
```
max_sagas_per_second = coordinator_instances * steps_per_saga * db_write_latency

Example:
  5 coordinator instances
  x 4 steps per saga (average)
  x 5ms per database write (p99)
  = 250 sagas/second
```

Scale by adding more coordinator instances behind the load balancer. Each instance is stateless — it reads and writes saga state to the shared saga log.

### Memory for Active Sagas

Each active saga instance in memory consumes approximately 1-5 KB for its state. If the average saga takes 3 seconds to complete and the system handles 250 sagas/second:

```
Active sagas in memory = 250 sagas/second x 3 seconds = 750 sagas
Memory required = 750 sagas x 3 KB = 2.25 MB

Plus overhead for retry queues, dead letter queues, and idle connections:
Total coordinator memory: ~64 MB per instance (negligible)
```

The coordinator is not memory-bound. The bottleneck is database write IOPS for the saga log.

### Startup Recovery

On startup, each coordinator instance runs a recovery scan:

```python
def recovery_scan():
    # Find all sagas that were ACTIVE or COMPENSATING but had no progress
    # for more than 5 minutes (stuck sagas from a crashed coordinator)
    stuck_sagas = db.query("""
        SELECT saga_id FROM saga_instances
        WHERE status IN ('ACTIVE', 'COMPENSATING')
          AND updated_at < NOW() - INTERVAL '5 minutes'
    """)

    for saga_row in stuck_sagas:
        saga = load_saga(saga_row.saga_id)
        if saga.current_step is None:
            # Saga was ACTIVE but no step was in progress — resume from step 1
            execute_step(saga, step_seq=1)
        elif saga.current_step.status == "IN_PROGRESS":
            # Step was in flight — check with participant or resume
            check_or_resume_step(saga, saga.current_step)
        elif saga.current_step.status == "FAILED":
            # Step failed — begin compensation
            begin_compensation(saga)
```

Each coordinator instance takes a lease on the sagas it recovers to avoid multiple instances recovering the same saga.

A production saga coordinator must expose metrics, health checks, and trace data so operators can detect and diagnose issues before they affect users.

### Health Check Endpoints

The coordinator exposes three health check endpoints for load balancers and orchestrators:

```
GET /health/liveness — Returns 200 if the process is alive. Used by Kubernetes liveness probe.
GET /health/readiness — Returns 200 if the coordinator can accept new sagas and connect to the saga log database. Used by Kubernetes readiness probe and load balancer.
GET /health/startup — Returns 200 if the coordinator completed its recovery scan. Used by Kubernetes startup probe.

GET /health/detailed — Returns JSON with internal status:
{
  "status": "healthy",
  "db_connected": true,
  "queue_connected": true,
  "sagas_active": 124,
  "sagas_pending": 5,
  "uptime_seconds": 345600
}
```

### Distributed Tracing

Include distributed trace headers in every command and event so operators can trace a saga's execution across service boundaries:

```python
def publish_command(saga_id, step_def, trace_context):
    headers = {
        "X-Saga-Id": saga_id,
        "X-Step-Seq": str(step_def.seq),
        "X-Trace-Id": trace_context.trace_id,
        "X-Span-Id": generate_span_id(),
        "X-Parent-Span-Id": trace_context.span_id,
    }
    queue.publish(
        exchange="saga.commands",
        routing_key=step_def.service,
        message=command,
        headers=headers
    )
```

Each participant propagates these headers into its own internal logging and tracing system. This allows operators to search for a saga_id and see every log line, database query, and API call that the saga produced across all services.

### Graceful Shutdown

When the saga coordinator receives a SIGTERM (Kubernetes pod termination, deployment rollout), it must drain in-flight sagas before shutting down:

```python
import signal
import time

class SagaCoordinatorServer:
    def __init__(self):
        self.shutdown_requested = False
        self.active_steps = set()
        signal.signal(signal.SIGTERM, self.handle_sigterm)

    def handle_sigterm(self, signum, frame):
        self.shutdown_requested = True
        self.log("SIGTERM received. Draining active steps...")
        # Stop accepting new sagas
        self.stop_accepting_new_sagas()
        # Wait for active steps to complete (max 30 seconds)
        deadline = time.time() + 30
        while self.active_steps and time.time() < deadline:
            time.sleep(0.1)
        self.log(f"Drain complete. {len(self.active_steps)} steps still in flight.")
        # Any in-flight steps are recovered by another coordinator on restart
```

This ensures that sagas in mid-execution are not abruptly terminated. The incomplete sagas are recovered by another coordinator instance during its startup recovery scan.
