# Transactions — Challenges

Each challenge maps to a specific skill: knowledge of isolation levels and anomalies, transaction design for realistic systems, and distributed transaction strategies. Foundation challenges take 5-10 minutes. Design challenges take 15-20 minutes. Expert challenges take 30-45 minutes.

---

## Tier 1: Foundation

These challenges test whether the concepts are solid.

### Challenge 1.1: Identify the Anomaly

A database is running at Read Committed isolation. The following sequence executes:

```
Time 1: Transaction A: SELECT balance FROM accounts WHERE id = 1;  → $500
Time 2: Transaction B: UPDATE accounts SET balance = 600 WHERE id = 1;
Time 3: Transaction B: COMMIT;
Time 4: Transaction A: SELECT balance FROM accounts WHERE id = 1;  → $600
Time 5: Transaction A: COMMIT;
```

A developer says: "Transaction A saw different values for the same query. This is a dirty read." Is the developer correct? If not, what anomaly occurred and why?

**Model answer:** The developer is not correct. A dirty read means reading a value written by an uncommitted transaction. Transaction B had already committed at time 4. This is a non-repeatable read — the same query returns different results because another transaction committed a change between the two reads.

Read Committed allows non-repeatable reads. To prevent this, the isolation level would need to be Repeatable Read or Serializable.

### Challenge 1.2: WAL Recovery Trace

A database crashes. The WAL contains the following records (in order):

```
[LSN 0/10, XID 101, BEGIN]
[LSN 0/20, XID 101, UPDATE accounts SET balance=4000 WHERE id='A']
[LSN 0/30, XID 101, UPDATE accounts SET balance=1200 WHERE id='B']
[LSN 0/40, XID 101, COMMIT]
[LSN 0/50, XID 102, BEGIN]
[LSN 0/60, XID 102, UPDATE accounts SET balance=3000 WHERE id='C']
[LSN 0/70, XID 103, BEGIN]
[LSN 0/80, XID 103, UPDATE inventory SET qty=50 WHERE id='widget']
```

The last checkpoint was at LSN 0/10. Describe what happens during REDO phase and UNDO phase. What is the final state of the database?

**Model answer:**

**REDO phase:** The database starts at LSN 0/10 (last checkpoint) and replays every record forward:

- Replay LSN 0/10: RECORD BEGIN for XID 101 (no data change).
- Replay LSN 0/20: Set Account A balance to 4000.
- Replay LSN 0/30: Set Account B balance to 1200.
- Replay LSN 0/40: RECORD COMMIT for XID 101 (no data change).
- Replay LSN 0/50: RECORD BEGIN for XID 102.
- Replay LSN 0/60: Set Account C balance to 3000.
- Replay LSN 0/70: RECORD BEGIN for XID 103.
- Replay LSN 0/80: Set inventory qty to 50 for widget.

After REDO, all changes are applied. The database is in the state at the time of crash.

**UNDO phase:** The database identifies active transactions at crash time:

- XID 102: Was ACTIVE (has BEGIN but no COMMIT or ABORT). Roll back its changes.
- XID 103: Was ACTIVE (has BEGIN but no COMMIT or ABORT). Roll back its changes.

Undo XID 102: Revert the UPDATE to Account C. balance is restored to its value before LSN 0/60. The database writes an ABORT record for XID 102.

Undo XID 103: Revert the UPDATE to inventory widget qty. The database writes an ABORT record for XID 103.

**Final state:** XID 101's work is committed and visible. XID 102 and XID 103's work is rolled back.

### Challenge 1.3: Choose the Right Isolation Level

Match each workload to the minimum isolation level that prevents its critical anomalies:

| Workload | Requirements | Isolation Level |
|---|---|---|
| Account balance transfer | Total funds must not change | |
| E-commerce inventory check | Must not oversell, acceptable to retry | |
| Blog post reader count | Approximate counts OK | |
| Analytics dashboard | Consistent snapshot for 30 minutes | |
| Seat reservation system | No double-booking allowed | |

**Model answer:**

| Workload | Isolation Level | Reasoning |
|---|---|---|
| Account balance transfer | Serializable | Phantom reads could cause incorrect totals across accounts |
| E-commerce inventory check | Repeatable Read | Lost updates prevented, phantoms unlikely for single-row inventory |
| Blog post reader count | Read Uncommitted | No critical anomaly possible for counters |
| Analytics dashboard | Repeatable Read | Consistent snapshot needed; phantom rows could affect aggregates |
| Seat reservation system | Serializable | Phantoms cause double-booking (row that didn't exist before appears) |

## Tier 2: Design

These challenges require designing the transaction logic for a specific system.

### Challenge 2.1: Bank Transfer Design

Design the transaction logic for a bank transfer system that handles:

- Money transfers between accounts
- Balance inquiries
- Monthly interest accrual
- Transfer limits ($10,000 per day per account)
- Fraud detection flagging

The system must be correct even under crash recovery. No money can be lost or created. Write the SQL and describe the internal state transitions.

**Requirements:**

- The transfer must be a single transaction with Serializable isolation.
- The balance check and the debit must happen atomically — no separate SELECT then UPDATE.
- The daily limit check must use the current day's total, not a stale value.
- Fraud detection runs asynchronously after the transaction commits.

**Model answer approach:**

```sql
BEGIN ISOLATION LEVEL SERIALIZABLE;
-- Atomic: check balance AND deduct in one statement
UPDATE accounts
SET balance = balance - 1000,
    daily_debit_total = daily_debit_total + 1000
WHERE id = 'A'
  AND balance >= 1000
  AND daily_debit_total + 1000 <= 10000;

-- Check that the debit actually affected a row
-- If not, rollback (insufficient funds or daily limit exceeded)

UPDATE accounts
SET balance = balance + 1000
WHERE id = 'B';

INSERT INTO transfer_log (from_id, to_id, amount, timestamp)
VALUES ('A', 'B', 1000, NOW());

COMMIT;
```

The key design choices:

1. **Atomic balance check**: The WHERE clause in the UPDATE prevents race conditions. No separate SELECT.
2. **Daily limit as same-row update**: The daily_debit_total column is on the same row as the balance, so both are atomically updated.
3. **Serializable isolation**: Prevents any concurrent transaction from seeing inconsistent state.
4. **Idempotency**: The transfer_log should have a unique constraint on (from_id, to_id, amount, date) to prevent duplicates if the client retries.
5. **Async fraud detection**: After COMMIT, publish an event with the transfer details. Fraud detection consumes this event and can roll back via a compensating transaction if needed.

### Challenge 2.2: Inventory Reservation for Flash Sale

Design the transaction system for a flash sale where 10,000 units of a product are available and 500,000 users try to buy simultaneously.

Constraints:
- Only one unit per user.
- No overselling (negative inventory).
- The inventory count must be accurate after every purchase.
- Users who acquire a unit have 15 minutes to complete payment or the unit is released.

**Model answer approach:**

The key insight is that contention on the inventory row is the bottleneck. A single row with 500,000 concurrent UPDATEs will serialize at the database level, achieving roughly 10,000 TPS — just enough for 10,000 units.

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
-- Reserved inventory table: reserve one unit for the user
INSERT INTO reservations (product_id, user_id, expires_at)
VALUES (42, 100, NOW() + interval '15 minutes');

-- Update available inventory atomically
UPDATE inventory
SET available = available - 1
WHERE product_id = 42
  AND available > 0;

-- If UPDATE affected 0 rows, rollback (sold out)
GET DIAGNOSTICS rows = ROW_COUNT;
IF rows = 0 THEN
    ROLLBACK;
    RETURN 'sold out';
END IF;

COMMIT;
```

Handling expiration:

- A background job runs every 60 seconds: DELETE expired reservations and increment available inventory.
- The background job runs in its own transaction to ensure the inventory increment is atomic with the reservation deletion.

Performance optimization: shard inventory by SKU. Since each SKU is independent, multiple 10,000-unit flash sales for different products do not contend on the same row.

### Challenge 2.3: Ride Payment with Multiple Services

A ride-hailing app processes payments after a ride completes. The payment involves:

1. Charge the rider's card (external payment gateway)
2. Credit the driver (internal platform credit)
3. Apply any promo code discounts
4. Send receipt to rider
5. Send earnings summary to driver

Design the transaction approach. Which steps must be atomic? Which can be eventually consistent? How do you handle failures (card declined, service timeout, crash during processing)?

**Model answer approach:**

This is a natural Saga. The core financial operations (1-3) should be as atomic as possible, but the external gateway (step 1) forces an eventual consistency approach.

```
Saga: process_ride_payment(ride_id)

Step 1 (within local transaction):
  - Create a "pending charge" record for the rider's card
  - Reserve the driver credit in a pending state
  - Deduct the promo code usage count
  - Commit local transaction

Step 2 (external call, not in transaction):
  - Call payment gateway to charge the rider
  - If success: proceed
  - If failure: execute compensating transaction
    - Refund the promo code usage
    - Cancel the pending driver credit
    - Mark charge as failed

Step 3 (within local transaction, after payment success):
  - Mark rider charge as completed
  - Mark driver credit as available
  - Send receipt (async, outside transaction)
  - Send earnings summary (async, outside transaction)
  - Commit local transaction
```

The compensating transaction for a failed charge: revert the promo code usage, the driver credit, and the pending charge record. This ensures the rider is not charged and the driver does not receive phantom earnings.

## Tier 3: Expert

These challenges require deep knowledge of distributed transaction patterns.

### Challenge 3.1: Global Transaction Strategy for Multi-Region E-Commerce

A rapidly growing e-commerce company expands from one region (us-east) to three regions (us-east, eu-west, ap-southeast). Each region has its own PostgreSQL database. The inventory is shared globally — a customer in any region can buy the last item.

Design the transaction strategy that satisfies:

- No overselling across regions.
- Sub-500 ms checkout latency for a typical purchase.
- Reads must always return the most recent committed data.
- The system must remain available (for reads) during a network partition.

**Model answer approach:**

**The core challenge:** Serializable transactions across 3 regions with 240 ms round-trip latency between the farthest regions. Two-phase commit would add 3 x RTT = 720 ms minimum, exceeding the 500 ms latency budget.

**Strategy: Geo-partitioned inventory with local checkout.**

```
Strategy: Location-Based Allocation

1. Partition inventory by region. Each region owns a fixed percentage of 
   total inventory (e.g., us-east 40%, eu-west 30%, ap-southeast 30%).
   
2. A customer checking out from eu-west uses inventory allocated to eu-west.
   Only a local transaction is needed. Latency: < 50 ms.

3. If eu-west runs out of its allocation, it can borrow from another 
   region. The borrow request uses a distributed transaction (2PC) 
   but only happens on the rare path (< 5% of checkouts).

4. Rebalance inventory allocations nightly (async batch process, not 
   in the critical path).

5. For read-your-writes: route the checkout GET request back to the 
   same region's primary. Replica lag is irrelevant because the 
   checkout write was local.
```

This approach accepts a trade-off: during a partition, a region can only sell from its own allocation. If it runs out, the customer sees "sold out" even though another region has stock. This is better than allowing overselling (which causes angry customers and chargebacks).

### Challenge 3.2: SAGA for Microservice Payment Flow

A fintech platform processes payments through a pipeline of microservices:

1. `FraudService` — Check for fraud (call external API)
2. `LedgerService` — Debit from sender account
3. `CurrencyService` — Convert currency if needed
4. `SettlementService` — Credit to receiver account (external bank API)
5. `NotificationService` — Send confirmation

Each service has its own database. No distributed transaction coordinator is available. Design a Saga that guarantees:

- All steps complete successfully, OR all steps are rolled back.
- If any step is retried, it is idempotent.
- The system can recover from crashes mid-Saga.
- The Saga can be monitored and audited.

**Model answer approach:**

```python
# Saga state machine stored in the saga orchestrator's database
SagaState:
  id: uuid
  status: PENDING | RUNNING | COMPLETED | FAILED
  current_step: int
  steps: [
    {name: "FraudService", action: call_fraud_api, 
     compensation: None},  # no-op, fraud check is idempotent
    {name: "LedgerService", action: debit_sender, 
     compensation: credit_sender},  # reverse the debit
    {name: "CurrencyService", action: convert_currency, 
     compensation: reverse_conversion},  # convert back
    {name: "SettlementService", action: credit_receiver, 
     compensation: debit_receiver},  # reverse the credit
    {name: "NotificationService", action: send_notification,
     compensation: None},  # no-op, notification is informational
  ]
```

**Execution protocol:**

1. The orchestrator creates a Saga record with status=PENDING in its own database.
2. For each step:
   a. Execute the step action (call the microservice).
   b. If success: update Saga state with the result, move to next step.
   c. If failure: enter compensation mode — execute compensation for this step, then for each completed step in reverse order.
3. After all steps succeed: update Saga state to COMPLETED.
4. After all compensations complete: update Saga state to FAILED.

**Idempotency:** Each step action accepts a saga_id and returns a result. The microservice checks if it has already processed this saga_id. If yes, return the previous result (idempotent replay). If no, execute the action and store the result keyed by saga_id.

```sql
-- Idempotent debit microservice
INSERT INTO ledger_entries (saga_id, from_account, amount, status)
VALUES ('saga-123', 'acct-456', 100.00, 'PENDING')
ON CONFLICT (saga_id) DO NOTHING;

-- If the INSERT did nothing, this saga_id was already processed
-- Return the previous result
```

**Crash recovery:** The orchestrator's database has the Saga state. After a crash, the orchestrator reads all Sagas with status=RUNNING and resumes from the last completed step. Since each step is idempotent, replaying steps that already completed is safe — they return the cached result.

**Monitoring:**

```sql
-- Find stuck Sagas
SELECT id, current_step, created_at
FROM saga_state
WHERE status = 'RUNNING'
  AND updated_at < NOW() - interval '5 minutes';
```

## Challenge Scoring Rubric

| Dimension | Foundation | Design | Expert |
|---|---|---|---|
| Isolation Level Knowledge | Identify anomalies correctly | Choose minimum level for workload | Explain SSI and conflict detection |
| Recovery Knowledge | Trace WAL recovery phases | Design recovery strategy | Analyze recovery time bottlenecks |
| Transaction Design | Write correct BEGIN/COMMIT | Design for crash and contention | Multi-region distribution |
| Failure Handling | Identify failure modes | Implement retry + compensation | Design for FLP impossibility |
| Trade-off Awareness | Recognize fsync cost | Choose sync/async commit | Accept weaker isolation for scale |
