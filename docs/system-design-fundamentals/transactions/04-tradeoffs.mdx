# Transactions — Trade-offs & Failure Modes

## Core Tension

The central trade-off in transaction design: stronger guarantees provide more safety but reduce throughput and increase latency. Every ACID property comes at a cost.

### The Cost of Each ACID Property

**Atomicity cost**: To support atomicity, the database must track every change made during a transaction and be able to undo it. This requires undo logs, which consume storage and write bandwidth. On rollback, the database spends CPU and I/O applying the undo records.

**Consistency cost**: Enforcing constraints (foreign keys, unique constraints, CHECK constraints) requires the database to validate every operation within a transaction. An UPDATE in the middle of a transaction that would violate a UNIQUE constraint causes the entire transaction to abort.

**Isolation cost**: Higher isolation levels require more locks or more overhead in snapshot management. Serializable isolation in PostgreSQL uses a serialization dependency graph and aborts transactions that would produce non-serializable behavior. This can cause many transaction aborts under contention.

**Durability cost**: The fsync system call on every COMMIT is the single largest performance cost. An fsync to a local NVMe SSD takes about 0.02 ms. An fsync to a network-attached volume (EBS gp3) can take 1-5 ms. An fsync to a synchronous replica across regions takes 50-200 ms.

Transactions sit squarely on the CP (Consistency + Partition Tolerance) side of the CAP theorem. Serializable transactions require coordination, and during a network partition, the system must choose between consistency and availability.

### CP vs AP Decision

For most single-datacenter deployments, CP is the right choice: the database provides transactions, and the application can always reach the database. The partition tolerance requirement only becomes relevant when the database spans multiple datacenters or regions.

**Choose CP (Serializable, 2PC) when**: The data represents money, inventory, or legal obligations. Losing consistency means losing money or violating regulatory requirements.

**Choose AP (Saga, BASE) when**: The data represents user sessions, activity logs, or cached data. Temporary inconsistency is acceptable. The system must remain available during a network partition.

## Failure Modes

### Long-Running Transactions Blocking VACUUM / Compaction

In PostgreSQL, VACUUM removes dead row versions. Long-running transactions prevent VACUUM from cleaning up rows that the transaction might still see. This leads to table bloat — the table grows even when data is deleted or updated.

```sql
-- Transaction A is running for 30 minutes
BEGIN;
-- does some work, then goes idle
-- VACUUM cannot remove any rows newer than this transaction's start time
```

**Impact**: If the table has high update volume, it can grow from 10 GB to 100 GB in hours. Queries scan more pages and slow down. The entire database suffers.

**Mitigation**: Set `idle_in_transaction_session_timeout = '30000'` (30 seconds). Monitor `pg_stat_activity` for transactions running longer than 5 seconds.

### Transaction ID Wraparound

PostgreSQL assigns a 32-bit XID to every transaction. After about 2 billion transactions, the XID counter wraps around, and old XIDs suddenly appear to be in the future.

PostgreSQL uses `pg_database.datfrozenxid` to track the oldest unfrozen XID. When `age(datfrozenxid)` approaches 2 billion, the database starts emitting warnings at transaction start. At 2 billion minus 1 million, the database enters single-user mode for emergency vacuuming.

```sql
-- Check XID age (should be well under 1 billion)
SELECT datname, age(datfrozenxid) FROM pg_database;

-- Check the current XID
SELECT txid_current();

-- How many transactions until wraparound?
SELECT 2147483647 - age(datfrozenxid) AS transactions_remaining
FROM pg_database WHERE datname = current_database();
```

**Mitigation**: Regular `VACUUM` (or `autovacuum`) freezes old XIDs. Set `autovacuum_freeze_max_age` to 200 million (default) — VACUUM runs automatically when any table's XID age exceeds this. Monitor XID age in alerts. Set `vacuum_freeze_table_age` to allow early table-level freezing.

### Serialization Failures Under Contention

With Serializable isolation, PostgreSQL detects conflicts that would produce non-serializable behavior and aborts one of the conflicting transactions. Under high contention (many transactions touching the same rows), this causes many aborts.

```sql
-- Transaction A                    -- Transaction B
BEGIN ISOLATION LEVEL SERIALIZABLE; BEGIN ISOLATION LEVEL SERIALIZABLE;
UPDATE accounts SET balance = ...   UPDATE accounts SET balance = ...
WHERE id = 'A';                     WHERE id = 'A';
COMMIT;                             COMMIT;
-- One of these aborts with:
-- ERROR: could not serialize access due to read/write dependencies
```

**Impact**: Under contention of 10+ concurrent transactions on the same row, 30-60 percent of transactions abort and must be retried. Throughput drops accordingly.

**Mitigation**: Use Read Committed or Repeatable Read if Serializable is not strictly required. Implement retry logic with exponential backoff. Reduce contention by sharding hot rows.

### Replication Lag and Read-Your-Writes

With asynchronous replication, a transaction commits on the primary but the change has not yet arrived at replicas. A subsequent read from a replica returns stale data.

```text
Time 0: Transaction commits (write "balance=4000" to primary)
Time 1: Read from replica (returns "balance=5000" — old data)
Time 2: WAL arrives at replica, change applied
Time 3: Read from replica (returns "balance=4000" — current)
```

**Impact**: The user sees their own actions inconsistently. "I just transferred $1000 but my balance still shows $5000." This damages trust.

**Mitigation**: Route read-your-writes queries to the primary. Use synchronous replication (but accept the throughput cost). Set `synchronous_commit = remote_apply` for strict consistency on replicas.

### Orphaned XA Transactions

With two-phase commit, a prepared transaction that the coordinator never completes is an orphan. It holds locks and WAL space indefinitely.

```sql
-- Find prepared transactions
SELECT gid, prepared, owner, database FROM pg_prepared_xacts;

-- Manually resolve an orphan
ROLLBACK PREPARED 'global-txn-001';
-- or: COMMIT PREPARED 'global-txn-001';
```

**Impact**: Orphaned prepared transactions block VACUUM and can cause XID wraparound if accumulated.

**Mitigation**: Monitor `pg_prepared_xacts`. Set a timeout for prepared transactions (PostgreSQL has no built-in timeout — build it into the application). Implement a heartbeat mechanism so the coordinator's failure is detected.

### Two-Phase Commit Coordinator Failure

If the 2PC coordinator crashes between PREPARE and COMMIT, participants hold locks on prepared data indefinitely. This is the blocking problem: the coordinator is a single point of failure.

**Impact**: All participants are blocked. No other transactions can modify the locked rows. System throughput drops to zero for those rows.

**Mitigation**: Run the coordinator as a highly available service (etcd, Zookeeper, or a replicated state machine). Implement a timeout and recovery protocol: participants can query each other to determine the outcome if the coordinator is down.

## Anti-Patterns

### Transactions Spanning User Interactions

The application begins a transaction, reads some data, presents a form to the user, and waits for the user to click Submit before committing. While the user is deciding, the transaction holds locks.

```python
# BAD: Transaction spans user interaction
def edit_user_page(user_id):
    db.execute("BEGIN")
    user = db.execute("SELECT * FROM users WHERE id = ? FOR UPDATE", (user_id,))
    render_form(user)  # User takes 30 seconds to edit the form
    # Transaction is still open, holding the lock!
    updated = read_form_input()
    db.execute("UPDATE users SET ... WHERE id = ?", (updated, user_id))
    db.execute("COMMIT")
```

This is the worst transaction pattern. The lock is held for 30+ seconds. Every other transaction that touches this user's row blocks. Connection pool exhausts. The entire application freezes.

**Fix**: Read the data without a write lock. Present the form. When the user submits, start a new transaction, read the current state, check for conflicts (optimistic locking), and write.

### Distributed Transactions for Everything

Centralized systems sometimes use distributed transactions for operations that span only a single database. This adds complexity and latency with no benefit.

```sql
-- BAD: Distributed transaction for a single-node operation
BEGIN;
PREPARE TRANSACTION 'txn-001';
-- Only one database involved
COMMIT PREPARED 'txn-001';
```

**Fix**: Use flat transactions for single-database operations. Use distributed transactions only when the operation genuinely spans multiple independent databases or systems.

### Running Batch Jobs Inside Transactions

A batch job processes 100,000 rows in a single transaction. The transaction holds locks for minutes. Other operations are blocked.

```python
# BAD: Batch job as one transaction
with db.transaction():
    for order in orders:
        db.execute("UPDATE orders SET status = 'processed' WHERE id = ?", (order.id,))
        # Locks held for all 100,000 updates
        # Other queries that touch orders are blocked for minutes
```

**Fix**: Process batches in smaller chunks, each with its own transaction. 1,000 rows per transaction. If the job crashes mid-way, only the most recent chunk is lost.

### Ignoring Row Lock Order

When multiple transactions update the same set of rows in different orders, deadlocks are guaranteed.

```python
# BAD: Inconsistent lock order
# Transaction A:
with db.transaction():
    db.execute("UPDATE accounts SET balance = balance - 50 WHERE id = 1")
    db.execute("UPDATE accounts SET balance = balance + 50 WHERE id = 2")

# Transaction B:
with db.transaction():
    db.execute("UPDATE accounts SET balance = balance - 50 WHERE id = 2")
    db.execute("UPDATE accounts SET balance = balance + 50 WHERE id = 1")
```

Transaction A locks row 1 then waits for row 2. Transaction B locks row 2 then waits for row 1. Deadlock.

**Fix**: Always acquire locks in a consistent order across all transactions.

```python
def transfer(from_id, to_id, amount):
    # Always lock the smaller ID first
    first, second = sorted([from_id, to_id])
    with db.transaction():
        db.execute("UPDATE accounts SET balance = balance - ? WHERE id = ?",
                   (amount if from_id == first else -amount, first))
        db.execute("UPDATE accounts SET balance = balance - ? WHERE id = ?",
                   (amount if from_id == second else -amount, second))
```

### Using Transactions for Read-Modify-Write Without Locks

A common anti-pattern is reading data, computing a new value in application code, and writing it back in a separate transaction. This breaks atomicity even within a single transaction if the isolation level is not high enough.

```python
# BAD: Read-modify-write without proper isolation
with db.transaction():
    balance = db.execute("SELECT balance FROM accounts WHERE id = 1").fetchone()[0]
    # Time passes - another transaction may have changed the balance
    new_balance = balance - 100
    db.execute("UPDATE accounts SET balance = ? WHERE id = 1", (new_balance,))
    # If another transaction also deducted between the SELECT and UPDATE,
    # this UPDATE overwrites their deduction. Lost update!
```

**Fix**: Use atomic UPDATE with WHERE clause, or use SELECT ... FOR UPDATE, or use optimistic locking with version numbers.

```sql
-- Atomic: combines read and write in one statement
UPDATE accounts SET balance = balance - 100 WHERE id = 1 AND balance >= 100;

-- Or: explicit locking
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- Now lock is held; read is safe
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
```

### Ignoring Serialization Failures

Applications that use Serializable isolation but do not implement retry logic will silently lose data.

```python
# BAD: No retry logic for serialization failures
from psycopg2 import errors

try:
    with db.transaction(isolation_level=ISOLATION_LEVEL_SERIALIZABLE):
        update_inventory(product_id, -1)
        create_order(user_id, product_id)
except errors.SerializationFailure:
    pass  # WRONG: The update was lost silently!
```

**Fix**: Always implement retry logic with exponential backoff:

```python
def execute_with_retry(fn, max_retries=3):
    for attempt in range(max_retries):
        try:
            return fn()
        except errors.SerializationFailure:
            if attempt == max_retries - 1:
                raise
            time.sleep(0.1 * (2 ** attempt))  # Exponential backoff
```

## When NOT to Use Transactions

Transactions are not free. In some scenarios, the overhead of transactions outweighs their benefit.

| Scenario | Why Transactions Are Wrong | Alternative |
|---|---|---|
| Read-only workloads | No writes to protect | No transaction needed (or READ ONLY) |
| Event sourcing | All operations are appends | No locks or rollbacks needed |
| Simple single-row CRUD | No multi-row atomicity needed | Individual statements suffice |
| High-throughput logging | fsync overhead limits throughput | Batch insert, async WAL |
| Time-series metrics | Transient data, loss-tolerant | No transaction needed |
| Cache population | Data will be overwritten anyway | No transaction needed |

For these workloads, skipping transactions or using `synchronous_commit = off` can boost throughput by 10-100x while accepting the risk of losing a small window of data.
