# Understanding Transactions

## The Problem Story

At 2:14 PM, a bank transfer executes. Account A has $5,000. Account B has $200. The command issues: move $1,000 from A to B.

The database performs two operations:

1. Debit A: `UPDATE accounts SET balance = balance - 1000 WHERE id = 'A';`
2. Credit B: `UPDATE accounts SET balance = balance + 1000 WHERE id = 'B';`

Operation 1 succeeds. The power supply in the server room trips. Operation 2 never runs.

Account A now has $4,000. Account B still has $200. One thousand dollars has vanished into the void. The customer calls support. The support agent checks the database. The database says A lost the money. Nobody knows where it went.

This is not a hypothetical. Data corruption from partial updates is one of the oldest problems in computing. Before transactions, databases had no mechanism to guarantee that a group of operations either all completed or all rolled back. Every application had to implement its own recovery logic — and most got it wrong.

The transaction primitive exists specifically to solve this problem: it wraps a group of operations into an atomic unit that either fully succeeds or fully fails, with no in-between state visible to the outside world.

---

### Why This Matters for Every Engineer

Transactions are not just a database feature. They are a fundamental correctness primitive that every system relies on. Whether the application uses PostgreSQL, MySQL, MongoDB, DynamoDB, or a custom in-memory store, the question is always the same: how does the system maintain correctness when multiple operations must happen together, when concurrent users touch the same data, and when hardware fails at the worst possible moment?

Understanding transactions means understanding:

1. **Correctness**: How the system guarantees that data is never corrupted by partial updates.
2. **Concurrency**: How the system handles multiple users reading and writing the same data simultaneously.
3. **Recovery**: How the system survives crashes without data loss.

Without transactions, every application must solve these three problems itself. Almost no application gets it right. Transactions provide a battle-tested primitive that handles all three.

## The Insight

A transaction is a group of database operations that must all succeed or all fail together — an atomic unit of work.

Think of it like sending a letter with tracking: either the entire process completes (letter picked up, sorted, transported, delivered, signed for) or the sender is refunded and it is as if the letter was never sent. There is no state where the sender lost the letter and the recipient never received it.

In database terms:

- **BEGIN** — Start the transaction. The database records that a transaction is beginning.
- **Operations** — Execute reads and writes against the database. These changes are not visible to other transactions yet.
- **COMMIT** — Make all changes permanent and visible to others.
- **ROLLBACK** — Discard all changes, restoring the database to its state before BEGIN.

The bank transfer becomes:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 1000 WHERE id = 'A';
UPDATE accounts SET balance = balance + 1000 WHERE id = 'B';
COMMIT;
```

If the server crashes between the two UPDATEs, the database's recovery process sees that the transaction never committed and rolls back the first UPDATE. The $1,000 never left Account A.

## Core Vocabulary

| Term | Meaning |
|---|---|
| Transaction | A group of database operations treated as a single atomic unit |
| Atomicity | All operations in a transaction complete, or none do |
| Consistency | A transaction brings the database from one valid state to another |
| Isolation | Concurrent transactions do not interfere with each other |
| Durability | Once committed, changes survive system failures |
| COMMIT | Make all changes in a transaction permanent |
| ROLLBACK | Undo all changes in a transaction |
| Savepoint | A marker within a transaction to roll back to a specific point |
| Write-Ahead Log (WAL) | Append-only log written before data files are modified |
| Dirty Read | Reading uncommitted data from another transaction |
| Non-Repeatable Read | Reading the same row twice and getting different values |
| Phantom Read | A query returns different sets of rows on successive executions |
| Serialization Anomaly | The result of concurrent transactions is inconsistent with any serial execution |

### The Four Key State Transitions

**BEGIN** — The database assigns the transaction a unique transaction ID (XID) and adds it to the set of running transactions. The transaction enters the ACTIVE state.

**ACTIVE** — SQL statements execute. The database acquires locks on rows or tables that the transaction reads or modifies. Write operations write records to the Write-Ahead Log (WAL) before modifying the actual data pages. Other transactions may or may not see these changes depending on the isolation level.

**COMMIT** — The database writes a COMMIT record to the WAL and fsyncs the WAL to disk. Once the fsync completes, the transaction is durable. The database releases all locks held by the transaction. The transaction's changes become visible to other transactions.

**ROLLBACK** — The database reads undo records from the WAL and reverses every change the transaction made. It releases all locks. From the perspective of other transactions, it is as if the transaction never existed.

## Step by Step: Bank Transfer Transaction

This sequence walks through every step of a bank transfer transaction, showing exactly what happens in the WAL, buffer pool, and lock manager at each stage.

```sql
-- Transaction begins
BEGIN;
-- Step 1: Read balance of Account A
SELECT balance FROM accounts WHERE id = 'A';
-- Step 2: Check if sufficient funds
-- (Application checks: balance >= 1000)
-- Step 3: Debit Account A
UPDATE accounts SET balance = balance - 1000 WHERE id = 'A';
-- Step 4: Credit Account B
UPDATE accounts SET balance = balance + 1000 WHERE id = 'B';
-- Step 5: Commit
COMMIT;
```

### Step-by-Step Internal Trace

**Step 0 — BEGIN:**
The database assigns XID = 723. It writes a BEGIN record to the WAL: `[XID 723, BEGIN, timestamp]`. The transaction enters the ACTIVE state. No locks are held yet.

**Step 1 — Read Balance A:**
The database acquires a shared lock on the row for Account A. It reads the account page from the buffer pool (or disk if not cached). The balance is $5,000. No WAL write (reads are not logged). The shared lock allows other transactions to read but not modify this row.

**Step 2 — Check Balance:**
The application layer checks that $5,000 >= $1,000. This happens outside the database. No database activity.

**Step 3 — Debit A:**
The database acquires an exclusive lock on Account A's row. It writes a WAL record: `[XID 723, UPDATE, table=accounts, row=A, old_balance=5000, new_balance=4000]`. It modifies the page in the buffer pool: Account A's balance changes from 5000 to 4000. The page is now dirty (modified but not yet written to disk).

**Step 4 — Credit B:**
The database acquires an exclusive lock on Account B's row. It writes a WAL record: `[XID 723, UPDATE, table=accounts, row=B, old_balance=200, new_balance=1200]`. It modifies the page in the buffer pool: Account B's balance changes from 200 to 1200. The page is now dirty.

**Step 5 — COMMIT:**
The database writes a COMMIT record to the WAL: `[XID 723, COMMIT, timestamp]`. It fsyncs the WAL to disk (ensuring all WAL records from this transaction are physically written). It releases all locks held by XID 723. The transaction becomes visible to others. The transaction is now durable.

**After COMMIT (asynchronous):**
Eventually, the database's background writer or checkpoint process flushes the dirty pages from the buffer pool to the actual data files on disk. If the server crashes before this flush, the WAL is replayed during recovery to reconstruct the changes.

## The Write-Ahead Log (WAL)

The WAL is the heart of transaction durability. Every modification to the database is first written to the WAL before it touches the data files. This is the fundamental rule: **WAL before data**.

The WAL enables a property called STEAL/NO-FORCE, which means:

- **STEAL**: The database can flush dirty pages from the buffer pool to disk before the transaction commits. If the transaction later rolls back, the WAL contains the undo information to revert those changes.
- **NO-FORCE**: The database does not need to flush dirty pages to disk at commit time. It only needs to flush the WAL. This means COMMIT is fast — it is bounded by WAL write speed, not by random page write speed.

### Read Uncommitted

Transactions can see uncommitted changes from other transactions. This is dangerous for most applications. Example: Transaction A debits $1,000 from Account A but has not committed. Transaction B reads Account A's balance and sees $4,000. Transaction A rolls back. Transaction B made a decision based on data that never existed.

Used in: Rarely. Some reporting systems where approximate values are acceptable.

### Read Committed (Default in PostgreSQL, SQL Server, Oracle)

Transactions see only committed data. Every query within a transaction sees a fresh snapshot of committed data. However, two successive reads of the same row might return different values if another transaction commits an update between the reads.

Example: Transaction A reads Account A balance: $5,000. Transaction B debits $1,000 and commits. Transaction A reads Account A balance again: $4,000. Different values for the same read — a non-repeatable read.

Used in: Most web applications. The default for good reason.

### Repeatable Read

Every query within a transaction sees the same snapshot of data taken at the first query. If Transaction A reads Account A's balance once, it will see the same value on every subsequent read, regardless of what other transactions commit.

However, Repeatable Read does not prevent phantom reads: if Transaction A queries "all accounts with balance > 3000" and gets 2 rows, then Transaction B inserts a new account with balance 4000 and commits, Transaction A's second query might return 3 rows.

PostgreSQL's Repeatable Read actually prevents phantoms (it uses snapshot isolation, which is stronger than the standard). In MySQL/InnoDB, Repeatable Read prevents phantoms for most queries.

Used in: Financial systems, reporting, any system where consistent reads matter.

### Serializable

Transactions execute as if they ran one after another — serially. The database prevents all anomalies by detecting conflicts and aborting one of the conflicting transactions.

Serializable is the safest level but has the highest overhead. Applications must be prepared to retry aborted transactions.

Used in: Systems where correctness is paramount: banking ledgers, inventory management, ticketing systems.

### Anomaly Walkthroughs

**Dirty Read Example:**

```
Transaction A: UPDATE accounts SET balance = 400 WHERE id = 1;
Transaction B: SELECT balance FROM accounts WHERE id = 1;
Transaction A: ROLLBACK;
```

With Read Uncommitted, Transaction B sees balance = 400, which never committed. With Read Committed or higher, Transaction B sees the original value.

**Lost Update Example:**

```
Transaction A: SELECT balance FROM accounts WHERE id = 1;  → $500
Transaction B: SELECT balance FROM accounts WHERE id = 1;  → $500
Transaction A: UPDATE accounts SET balance = 450 WHERE id = 1;
Transaction B: UPDATE accounts SET balance = 300 WHERE id = 1;
Transaction A: COMMIT;
Transaction B: COMMIT;
-- Final balance: $300. Transaction A's $50 deduction is lost.
```

This is prevented by Repeatable Read or Serializable isolation. With Repeatable Read, Transaction B's UPDATE would block until Transaction A commits, then recalculate.

**Write Skew Example:**

```
-- Hospital shift scheduling: at least one doctor must be on call
-- Two doctors try to take themselves off call simultaneously
Transaction A: SELECT COUNT(*) FROM on_call WHERE on_call = true;  → 2
Transaction B: SELECT COUNT(*) FROM on_call WHERE on_call = true;  → 2
Transaction A: UPDATE doctors SET on_call = false WHERE id = 1;  -- sees 2, OK
Transaction B: UPDATE doctors SET on_call = false WHERE id = 2;  -- sees 2, OK
Transaction A: COMMIT;
Transaction B: COMMIT;
-- Result: zero doctors on call. Constraint violated.
```

Write skew is prevented by Serializable isolation but allowed by Repeatable Read. Neither transaction wrote data the other read, so Repeatable Read detects no conflict. Serializable detects the dependency through predicate locks.

## Variants of Transactions

Not all transactions follow the simple BEGIN/COMMIT pattern. Different workloads need different transaction models.

### Flat Transactions

The standard model: BEGIN, operations, COMMIT or ROLLBACK. If any operation fails, the entire transaction rolls back.

```sql
BEGIN;
INSERT INTO orders (user_id, total) VALUES (42, 29.99);
INSERT INTO order_items (order_id, product_id, qty) VALUES (currval('orders_id_seq'), 101, 1);
COMMIT;
```

This is the most common form. Used for: bank transfers, order placement, any multi-step write.

### Nested Transactions (Savepoints)

A savepoint creates a marker within a transaction. The transaction can roll back to a savepoint without aborting the entire transaction.

```sql
BEGIN;
INSERT INTO orders ...;
SAVEPOINT order_created;
INSERT INTO order_items ...;
-- Something goes wrong with the items
ROLLBACK TO order_created;
-- The order still exists, items are rolled back
INSERT INTO order_items ...;  -- Try different items
COMMIT;
```

Nested transactions trade atomicity for granularity: parts of the transaction succeed independently.

### Distributed Transactions (Two-Phase Commit / XA)

A transaction that spans multiple databases. The two-phase commit protocol coordinates the commit across all participants.

```sql
-- Phase 1: Prepare
PREPARE TRANSACTION 'txn-001';
-- Database A is ready to commit
-- Database B is ready to commit

-- Phase 2: Commit
COMMIT PREPARED 'txn-001';
-- Both databases commit
```

Two-phase commit guarantees that either all databases commit or none do. The cost: a coordinator is required, and if the coordinator crashes, prepared transactions may block until it recovers.

### Compensating Transactions (Saga Pattern)

Instead of locking resources for the duration of a long-running operation, each step has a compensating action that undoes it. If step 3 fails, the system executes the compensations for steps 2 and 1.

```sql
-- Saga for booking a trip
Step 1: Book flight (compensation: cancel flight booking)
Step 2: Book hotel (compensation: cancel hotel booking) 
Step 3: Book car (compensation: cancel car booking)

-- If step 3 fails:
Execute compensation for step 2: cancel hotel
Execute compensation for step 1: cancel flight
```

The Saga pattern trades isolation for availability. Intermediate states are visible to other transactions, but the system guarantees eventual consistency. This is the dominant pattern in microservice architectures.

### Chain Transactions

Each transaction in a chain transfers its partial results to the next transaction without making the results visible to others until the final commit. This is a performance optimization for long sequences of operations where intermediate results should not be visible.

## Mental Model Check

These five questions will confirm whether the mental model is solid.

**Question 1:** A transaction writes $100 to Account A and $200 to Account B. The server crashes after writing to A but before the COMMIT record reaches the WAL. What happens to Account B's write?

**Answer:** Both writes are rolled back. The WAL contains no COMMIT record for this transaction, so during crash recovery, the database reads the undo information from the WAL and reverts Account A's write. Neither write is visible after recovery. The transaction might as well never have started.

**Question 2:** What is the difference between a dirty read and a non-repeatable read?

**Answer:** A dirty read means reading data written by a transaction that has not yet committed. The data might disappear if the transaction rolls back. A non-repeatable read means reading the same row twice and getting different values because another transaction committed an update between the two reads. Dirty reads involve uncommitted data; non-repeatable reads involve committed data that changed between reads.

**Question 3:** Why does COMMIT only need to flush the WAL, not the data pages?

**Answer:** Because of the WAL-before-data rule. The WAL contains enough information to reconstruct the changes during crash recovery. If the server crashes after the WAL is flushed but before data pages are written to disk, the database replays the WAL during recovery. Flushing just the WAL is much faster than writing random data pages because the WAL is a sequential append-only structure.

**Question 4:** When would a Saga be a better choice than a distributed transaction?

**Answer:** A Saga is better when the transaction spans microservices or when the individual steps take a long time (minutes to hours). Distributed transactions via 2PC hold locks for the entire duration, which blocks other transactions and reduces throughput. Sagas release resources after each step and use compensating actions to undo failures, at the cost of weaker isolation (intermediate states are visible).

**Question 5:** A reporting dashboard queries the total balance across all accounts. Why might Repeatable Read give different results than Serializable?

**Answer:** Repeatable Read prevents the reporting query from seeing rows that change during its execution, but it might allow phantom reads — rows inserted or deleted by other transactions that change the set of accounts being summed. Serializable prevents phantoms, guaranteeing that the set of accounts is fixed for the entire transaction. For an accurate total, Serializable (or a snapshot taken at a single point in time) is required.
