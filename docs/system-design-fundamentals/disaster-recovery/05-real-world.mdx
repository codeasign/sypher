# Disaster Recovery in Production

## Company Case Studies

### Netflix: Chaos Engineering at Scale

Netflix is the most famous example of disaster recovery done right. Their entire infrastructure philosophy is built around the assumption that everything fails — and that the system must survive without human intervention.

**The problem**: Netflix runs on AWS. Before 2011, they experienced several high-profile outages caused by unexpected failures. A single EBS volume failure or a single AWS availability zone issue could take down the entire streaming service for millions of users. They needed a system that could survive any single failure without degrading the user experience.

**The solution**: Netflix created the Simian Army — a suite of tools that intentionally inject failures into the production system. The most famous member is Chaos Monkey, which randomly terminates EC2 instances during business hours.

```python
# Simplified Chaos Monkey logic
def chaos_monkey(instance_list):
    target = random.choice(instance_list)
    if target.is_eligible_for_termination():
        target.terminate()
        log_event("chaos_monkey", f"Terminated {target.id}")
```

The logic is deliberately simple: terminate a random instance and see if the system survives. If it does, the redundancy is working. If it does not, engineering fixes the gap before a real failure finds it.

Netflix's approach has evolved to include:

- **Chaos Kong**: Simulates a full AWS region failure by routing all traffic away from a region. Netflix runs Chaos Kong in production every few months.
- **Chaos Gorilla**: Simulates an AWS availability zone failure.
- **Latency Monkey**: Introduces artificial latency between services to simulate network degradation.
- **Conformity Monkey**: Finds instances that do not conform to best practices (e.g., not in an auto-scaling group) and flags them.

**Results**: Netflix can survive a full AWS region failure without any user-facing impact. During a 2014 AWS region outage that took down many other services, Netflix remained fully operational because their traffic automatically shifted to the remaining regions. Their RTO for a single-region failure is effectively zero.

**Key lesson**: Treat infrastructure as "cattle, not pets." If you are afraid to terminate an instance, you have not built enough redundancy. Chaos engineering makes failure routine and survivable.

### Uber: The Multi-Region Migration

Uber's disaster recovery journey is a story of learning from catastrophic failure. In 2017, Uber experienced a major outage in their primary data center that took down their entire platform for several hours.

**The problem**: Uber's infrastructure was built on a single data center with a single MySQL database cluster. When the data center experienced a power issue, the entire platform went offline. Rides could not be dispatched, payments could not be processed, and drivers could not get trip requests.

**The solution**: Uber embarked on a multi-year project to rebuild their infrastructure for multi-region operation. The key architectural decisions were:

1. **Database sharding**: Uber's monolithic MySQL database was split into 4,000+ shards, each replicated across multiple regions. A shard failure only affects a small fraction of users.

2. **Ringpop**: Uber built a distributed application-layer routing protocol called Ringpop that routes requests to the correct shard regardless of region. Ringpop uses consistent hashing to minimize reshuffling when nodes are added or removed.

3. **Cross-region request routing**: If a user's data is in the us-east-1 shard but the user's request arrives at us-west-2, Ringpop forwards the request to the correct region. This allows any region to serve any request, even during a partial failure.

4. **Pilot light for non-critical services**: Low-priority services (analytics, reporting) run in pilot light mode in the secondary region. When a disaster occurs, these services are scaled up.

**Results**: Uber achieved an RTO of under 5 minutes for a single-region failure. The system can survive a full data center outage without losing trip data. The multi-region architecture also improved latency for users in different geographic regions.

**Key lesson**: Sharding was the critical enabler for Uber's DR strategy. Without sharding, the monolithic database was a single point of failure that could not be replicated across regions efficiently. Sharding also improved performance and scalability.

### Discord: The Database Failover That Almost Wasn't

Discord's disaster recovery story illustrates the importance of testing. In 2020, Discord experienced a catastrophic database failure that required a full failover to their secondary region.

**The problem**: Discord's primary database cluster in us-east-1 experienced a hardware failure that caused corruption in the metadata table. The database could not be recovered in place. The team had to fail over to the us-west-2 region.

**The failover**: Discord's DR plan called for promoting the read replica in us-west-2 to primary. The team had tested this procedure in staging. But the production failover revealed several issues:

- The read replica was 12 seconds behind the primary. The replication lag was higher than expected because of the write volume during the incident.
- The secondary region's cache was cold. The first 30 minutes of operation after failover were extremely slow as data was fetched from the database and cached.
- Some application instances had hardcoded database endpoints instead of using the service discovery system. They continued pointing to the downed primary and had to be manually restarted.

**The recovery**: The team successfully failed over in 18 minutes — within their 30-minute RTO target. They lost 12 seconds of data (the replication lag gap). The data was reconstructed from application logs within 24 hours.

**Post-mortem action items**:

1. Reduce DNS TTL from 300 seconds to 60 seconds
2. Eliminate all hardcoded database endpoints
3. Implement cache warm-up scripts that run automatically after failover
4. Increase replication monitoring to alert on lag exceeding 5 seconds
5. Test failover in production, not just staging

**Key lesson**: Staging tests are not enough. The production environment has different traffic patterns, different data volumes, and different configuration than staging. A production DR test, even if it is a read-only test, reveals issues that staging never will.

### Cloudflare: The Multi-Cloud Backup

Cloudflare operates a global network that routes traffic for 20% of the web. Their disaster recovery strategy involves a secondary infrastructure provider — a multi-cloud backup.

**The problem**: Cloudflare's primary infrastructure runs on their own bare-metal servers in data centers around the world. If a data center fails, traffic is rerouted to other data centers. But what if a software bug causes a global control plane failure? This happened in 2019 when a WAF rule caused a global outage.

**The solution**: Cloudflare maintains a secondary control plane in a separate cloud provider (AWS). The secondary control plane is a minimal version of the primary — it can serve DNS and basic routing but cannot provision new services. If the primary control plane fails, the secondary takes over critical functions.

```json
{
  "cloudflare_secondary_control_plane": {
    "provider": "AWS",
    "regions": ["us-east-1", "eu-west-1"],
    "capabilities": [
      "DNS resolution",
      "Basic routing",
      "Health check responses",
      "Rate limiting (static rules)"
    ],
    "limitations": [
      "No new service provisioning",
      "No configuration changes",
      "No analytics",
      "No dashboard"
    ],
    "failover_trigger": "Primary control plane unreachable for 60 seconds",
    "test_frequency": "Monthly"
  }
}
```

**Results**: Cloudflare's secondary control plane provides a safety net for control plane failures. They have never needed to use it for a real disaster, but the regular testing ensures it works.

**Key lesson**: Multi-cloud DR is expensive and complex. Most organizations should not use it. But for companies like Cloudflare that provide infrastructure for millions of other sites, the cost is justified. For most applications, multi-region DR within a single cloud provider is sufficient.

### GitLab: The Backup Restoration That Saved Everything

GitLab's disaster recovery story is a cautionary tale about backups. In 2017, a GitLab engineer accidentally deleted a production database. The team discovered that the backup process had been broken for six months — no usable backups existed.

**The problem**: During a routine database maintenance, an engineer ran a `DELETE` command on the wrong database. The production database for GitLab.com was deleted. The team discovered that:

- Daily backups had been running but the backup process was silently failing
- The backup files were empty
- There was no cross-region backup
- The database replication was in the same data center

**The recovery**: GitLab's team worked for 24 hours to recover the data. They were able to reconstruct the database from a combination of:

1. A 6-hour-old database snapshot that had been taken for a different purpose
2. Database replication logs from the production database
3. Application logs that contained write operations
4. User-submitted data that was cached in the application layer

The team recovered 99.5% of the data. The remaining 0.5% was reconstructed from user reports over the following weeks.

**Post-mortem action items**:

1. Backups must be validated — automated restore tests run daily
2. Backups must be copied to a different region
3. Write access to production databases must use role-based access with approval gates
4. All database modifications must go through a code review process
5. The `DELETE` command should require a confirmation flag

**Key lesson**: A backup that is not validated is not a backup. GitLab's backup process reported success but the output files were empty. After this incident, GitLab implemented automated backup validation that restores the backup to a test environment and verifies data integrity.

### Dropbox: The Migration to Structured DR

Dropbox provides another interesting case study. In 2013, Dropbox migrated from a single data center to a multi-region architecture. The migration was driven by a series of outages that affected their 500 million users.

**The problem**: Dropbox's infrastructure was built on a single data center in San Jose, California. When the data center experienced a power outage, all Dropbox users lost access to their files. The outage lasted 4 hours and affected millions of users.

**The solution**: Dropbox rebuilt their infrastructure to support multi-region operation. The key architectural change was moving from a monolithic MySQL database to a sharded architecture with multiple replicas per shard. Each shard was replicated to a secondary data center in a different geographic region.

Dropbox also implemented a feature called "remote wipe" protection: if a data center failure required failing over to a secondary region, the system could be configured to serve read-only traffic until the data was verified as consistent. This prevented data corruption from propagating to users.

**Key lesson**: The migration to structured DR took Dropbox two years. The technical work (sharding, replication, failover automation) was the easy part. The hard part was the cultural change: moving from a team that assumed the data center was always available to a team that designed for failure as a default.

### Slack: The Global Cache Architecture

Slack's DR strategy is built around a globally distributed cache layer. Slack serves 10 million daily active users, and their infrastructure must handle traffic spikes during business hours in each time zone.

**The problem**: Slack's database was a single PostgreSQL instance in a single data center. As the user base grew, the database became a bottleneck. A region failure would take the entire service offline.

**The solution**: Slack deployed a global cache layer using memcached, with caching servers in 12 data centers around the world. Each cache served reads from the local region. Writes were forwarded to the primary database, which was replicated to a secondary database in a different region.

The cache layer provided two benefits: it reduced read latency for users in different regions, and it provided a buffer during a region failure. If the primary database region failed, Slack could serve stale data from the cache while the database failover was in progress.

**Key lesson**: Slack's DR strategy was not designed as a separate DR system. It was a side effect of their performance optimization. The cache layer that improved read latency also provided the foundation for their DR strategy. This is a common pattern: the best DR systems are built into the architecture, not bolted on as an afterthought.

### Amazon: The DynamoDB Global Tables Story

Amazon built DynamoDB Global Tables to solve the DR problem for their own internal services. The architecture is instructive because it shows how to design a system where DR is a built-in feature, not an add-on.

**The problem**: Amazon's internal services needed to survive region failures. Each service was building its own DR solution — some using RDS with cross-region replication, some using application-level replication, some using custom database solutions. This was inefficient and fragile. Amazon needed a managed database service that handled DR automatically.

**The solution**: DynamoDB Global Tables provide multi-region, multi-writer database tables. A write to any region is automatically propagated to all other regions within seconds. There is no concept of primary and secondary — every region is equal. This eliminates the need for failover entirely.

The key technical innovation is the use of **last-writer-wins (LWW)** conflict resolution. If two writes to the same item occur in different regions at the same time, the write with the latest timestamp wins. This means the system cannot lose data but can lose individual writes that are overwritten by newer writes from another region.

```json
{
  "dynamodb_global_table": {
    "table_name": "Orders",
    "regions": ["us-east-1", "us-west-2", "eu-west-1"],
    "replication": "automatic, sub-second",
    "conflict_resolution": "last-writer-wins",
    "write_model": "multi-writer (no failover needed)",
    "rpo": "sub-second (conflicts possible)",
    "rto": 0
  }
}
```

**Key lesson**: DynamoDB Global Tables solve the DR problem by making it irrelevant. If every region is equal and can accept writes independently, there is nothing to fail over. The cost is the complexity of conflict resolution — last-writer-wins semantics may not be acceptable for all use cases. Financial transactions, for example, require different conflict resolution strategies.

### Atlassian: The Backup Validation Story

Atlassian (the company behind Jira and Confluence) learned the importance of backup validation the hard way. In 2022, a misconfiguration in their backup system caused all backups to silently fail for several weeks.

**The problem**: Atlassian's backup system was configured to take daily snapshots of their database and store them in a different region. The backups appeared to be running successfully — the backup logs showed "completed" for each day. But a storage configuration change had caused the backup files to be written to a location that was not monitored. The files existed but were empty.

**The discovery**: During a routine DR drill, the team attempted to restore from a backup and discovered the backup was unusable. Further investigation revealed that the last valid backup was from 3 weeks ago. The team had been operating under a false sense of security for 21 days.

**The fix**: Atlassian implemented a three-tier backup validation system:

1. **Tier 1 (automated, daily)**: A script restores the latest backup to a test environment and verifies data integrity by running checksums.
2. **Tier 2 (automated, weekly)**: A script runs a set of application-level queries against the restored backup and verifies the results match expected patterns.
3. **Tier 3 (manual, monthly)**: An engineer logs into the test environment and manually validates that the restored data is consistent and complete.

**Key lesson**: Backups are not a set-it-and-forget-it operation. They require continuous validation. An automated backup that is not automatically validated is no better than no backup at all.

## Open Source Implementations

### Velero (formerly Heptio Ark)

Velero is an open-source tool for backing up and restoring Kubernetes clusters and persistent volumes. It provides disaster recovery for containerized workloads.

```bash
# Install Velero with AWS S3 as backup storage
velero install \
    --provider aws \
    --plugins velero/velero-plugin-for-aws:v1.0.0 \
    --bucket velero-backups \
    --backup-location-config region=us-west-2 \
    --snapshot-location-config region=us-west-2

# Schedule daily backups
velero schedule create daily-backup \
    --schedule "0 1 * * *" \
    --ttl 720h \
    --include-namespaces production

# Restore from backup
velero restore create --from-backup daily-backup-20260702
```

Velero supports backup of Kubernetes resources (deployments, services, config maps) and persistent volumes (EBS snapshots, Azure disks, GCE persistent disks). It can restore to a different cluster or a different region.

### Patroni

Patroni is an open-source HA (High Availability) solution for PostgreSQL. It provides automated failover for PostgreSQL databases, which is the foundation of any database DR plan.

```yaml
# Patroni configuration for cross-region replication
scope: mydb
name: postgresql0
restapi:
  listen: 0.0.0.0:8008
  connect_address: 10.0.0.1:8008
etcd:
  host: etcd-cluster:2379
bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576
    postgresql:
      use_pg_rewind: true
      parameters:
        wal_level: logical
        hot_standby: "on"
        max_wal_senders: 10
        max_replication_slots: 10
        wal_keep_segments: 1000
```

Patroni uses a distributed consensus store (etcd or ZooKeeper) to coordinate leader election. When the primary fails, the replicas hold an election, and the replica with the most recent WAL position is promoted to primary.

## Cloud Provider Implementations

### AWS Disaster Recovery Services

AWS provides a comprehensive suite of DR services:

**AWS Backup**: Centralized backup service for AWS resources. Supports cross-region and cross-account backup copies. Automated backup policies with retention rules.

**AWS Elastic Disaster Recovery (DRS)**: Continuous replication of EC2 instances to a different region. Supports pilot light and warm standby configurations. Automated failover and failback testing.

**AWS Global Accelerator**: Routes traffic to the optimal endpoint based on health, latency, and geography. Provides instant failover by rerouting traffic to a different region when health checks fail.

**AWS Route53 Application Recovery Controller**: A managed service for multi-region application recovery. Provides readiness checks, routing controls, and failover automation.

## Lessons Learned

### Lesson 1: Test in Production

The single most important lesson from every major DR incident is that testing in staging is not enough. Staging environments have different traffic patterns, different data volumes, different configurations, and different timing. A DR test in staging proves only that the staging environment works — not that the production environment will survive.

Netflix tests in production by running Chaos Monkey during business hours. Discord learned the hard way that their staging DR test did not reveal the production issues. The companies that test their DR in production are the companies that survive real disasters.

### Lesson 2: Automate, But Keep a Kill Switch

Full automation is the goal, but every automation should have a manual override. The DR orchestrator should handle the common cases automatically but must have a "stop" button that halts all automated actions and escalates to a human.

The kill switch is especially important for the failover decision itself. A false positive failover (failing over when the primary is actually healthy) can be as damaging as a real outage. The cooldown timer and manual approval gate prevent automation from making things worse.

### Lesson 3: The DR Plan Is Code

A DR plan stored in a PDF on a shared drive is a plan that will fail. The DR plan should be version-controlled alongside the infrastructure code. Every change to the infrastructure should trigger a review and update of the DR plan. The execution of the plan should be automated — runbooks should be scripts, not checklists.

### Lesson 4: Measure Everything

You cannot improve what you do not measure. Every DR drill should produce measured RTO and RPO values. These values should be tracked over time to detect trends. If the measured RTO is increasing, the DR plan is decaying. If the measured RPO is stable, the replication is working.

Track these metrics:

- **Time to detect**: How long between the failure occurring and the system detecting it
- **Time to decide**: How long between detection and the failover decision
- **Time to execute**: How long between decision and traffic flowing to the secondary
- **Time to validate**: How long between traffic flowing and the system being declared healthy
- **Replication lag at failover**: The actual data loss at the moment of failover
- **Cache hit rate after failover**: How quickly the cache recovers from cold start

### Lesson 5: Plan for the Disaster After the Disaster

The primary disaster (the region failure) is bad enough. But the secondary effects — the cascading failures, the human errors, the configuration drift — can be worse. A DR plan that only plans for the primary disaster is incomplete.

Plan for these scenarios:

- The on-call engineer is on vacation and cannot be reached
- The secondary region has a different database schema than the primary
- The backup files are corrupt
- The DNS provider is also affected by the same disaster
- The failover orchestrator itself fails
- The primary region recovers and the system must fail back during peak traffic

The companies that survive disasters are not the ones with the best infrastructure. They are the ones that have planned for the worst-case scenario — and tested that plan.

### Lesson 6: DR Is a Product, Not a Project

The most successful DR programs treat disaster recovery as a product with its own roadmap, on-call rotation, and user feedback loop. The DR system is not built once and forgotten. It is continuously improved based on testing results, incident post-mortems, and changes to the primary system.

A product mindset for DR means:
- DR has a dedicated budget and engineering team
- DR has a roadmap with quarterly improvement goals
- DR team participates in the on-call rotation for the primary system
- DR incidents are analyzed with the same rigor as production incidents
- DR improvements are prioritized using the same framework as product features

This approach costs more upfront but pays for itself the first time a real disaster occurs. Companies that treat DR as a one-time project are the ones that discover their DR plan is broken during the actual disaster.

The companies that survive disasters are not the ones with the best infrastructure. They are the ones that have planned for the worst-case scenario — and tested that plan.
