# Disaster Recovery at Scale

## Naive Implementation: The Single-Server Approach

### Phase 1: 1,000 Users -- No DR at All

At 1,000 users, the system is a single server co-located in a data center or running on a single cloud instance. There is no disaster recovery plan. Backups are manual — the founder runs `pg_dump` once a week and stores the result on the same machine.

```yaml
# docker-compose.yml -- the "disaster recovery" setup
services:
  app:
    image: myapp:latest
    ports:
      - "80:80"
    depends_on:
      - db
  db:
    image: postgres:15
    volumes:
      - ./pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: "${DB_PASSWORD}"
```

If the server's disk fails, all data is lost. If the cloud provider's availability zone goes down, the site is offline until a new instance is provisioned. This is acceptable for a prototype or an internal tool — not for a business that needs uptime.

**Failure scenario**: The data center loses power. The server shuts down. The site is down for 4 hours until power is restored. The last backup was 6 days ago. Six days of user data is lost.

### Phase 2: 100,000 Users -- Backup to Object Store

At 100,000 users, the system has moved to a managed cloud service. The database runs on a managed database service with automated daily snapshots. Backups are stored in an object store in a different region.

```json
{
  "primary_setup": {
    "compute": "Auto-scaling group, 5-20 instances, us-east-1",
    "database": "RDS PostgreSQL, db.r6g.xlarge, us-east-1a",
    "backups": {
      "automated_snapshots": true,
      "retention_period_days": 35,
      "cross_region_copy": {
        "enabled": true,
        "target_region": "us-west-2",
        "copy_tags": true
      }
    },
    "storage": "S3 standard, cross-region replication enabled"
  }
}
```

This setup protects against instance failure (auto-scaling replaces the instance) and availability zone failure (RDS multi-AZ provides automatic failover to the standby in a different AZ within the same region). It does NOT protect against a full region failure.

**Failure scenario**: The entire us-east-1 region becomes unavailable. The team must provision a new database in us-west-2 from the latest snapshot. The snapshot is 4 hours old — 4 hours of data is lost. Provisioning and restoring takes 2 hours. Total downtime: 2 hours for restore plus 30 minutes for DNS changes. RTO is approximately 2.5 hours. RPO is 4 hours.

### Phase 3: 10 Million Users -- Multi-Region with Pilot Light

At 10 million users, downtime costs tens of thousands of dollars per minute. The system deploys to two regions with a pilot light setup.

```json
{
  "primary_region": {
    "name": "us-east-1",
    "compute": "Auto-scaling group, 50-200 instances, c6g.large",
    "database": "RDS Aurora PostgreSQL, db.r6g.2xlarge, Multi-AZ",
    "cache": "ElastiCache Redis Cluster, 10 nodes, r6g.large"
  },
  "secondary_region": {
    "name": "us-west-2",
    "compute": "Auto-scaling group, 2 min instances, scales to 200",
    "database": "RDS Aurora read replica, db.r6g.2xlarge",
    "cache": "ElastiCache Redis Cluster, 2 nodes (warm standby)"
  },
  "replication": {
    "database": "Aurora cross-region replication (async)",
    "storage": "S3 CRR (Cross-Region Replication)",
    "dns": "Route53 failover routing, TTL=60s"
  },
  "monitoring": {
    "health_checks": "Route53 health checks from 3 regions",
    "synthetic_transactions": "Every 5 minutes from external locations",
    "alerts": "PagerDuty with 24/7 on-call rotation"
  }
}
```

The secondary region runs a minimal footprint: 2 application instances and a database read replica. When a disaster is declared, the auto-scaling group is instructed to scale to full capacity, the database replica is promoted to primary, and DNS records are updated. The RTO is 15 minutes. The RPO is 5 seconds (asynchronous replication lag).

**Failure scenario**: A major AWS event in us-east-1. The failover orchestrator detects the outage within 60 seconds. It promotes the Aurora replica in us-west-2, scales the application fleet, and updates DNS. Within 12 minutes, traffic is flowing through us-west-2. Approximately 8 seconds of data is lost (replication lag at the moment of failover). Revenue impact: approximately $12,000 (at $1,000/minute downtime).

## Integration Patterns

Disaster recovery integrates with every other system in the stack. Here are four critical integration patterns.

### Pattern 1: Database Integration with Replication

The database is the heart of disaster recovery. Every major database system provides cross-region replication capabilities.

**PostgreSQL logical replication**: The primary database streams WAL changes to a logical replica in another region. The replica applies changes in near-real-time. On failover, the replica is promoted to primary.

```sql
-- Primary: Create publication
CREATE PUBLICATION dr_pub FOR ALL TABLES;

-- Replica: Create subscription
CREATE SUBSCRIPTION dr_sub
CONNECTION 'host=primary.example.com dbname=mydb'
PUBLICATION dr_pub;
```

**DynamoDB Global Tables**: AWS-managed multi-region replication with multi-writer support. Writes to any region are automatically replicated to all others. No promotion needed during failover — the application simply points to the local table.

```python
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('Orders')
table.put_item(Item=order)
# DynamoDB automatically replicates to all global table regions
```

**Aurora Global Database**: Amazon Aurora's cross-region feature provides fast replication (typically under 1 second latency in the same continent). Failover is a single API call.

```bash
aws rds failover-global-cluster \
    --global-cluster-identifier my-global-cluster \
    --target-db-cluster-identifier arn:aws:rds:us-west-2:...
```

### Pattern 2: Queue and Message Integration

Message queues must be configured to survive a region failure. If the primary queue becomes unavailable during a disaster, messages must not be lost.

**SQS with dead-letter queues and backup queues**:

```json
{
  "queue_config": {
    "primary_region": "us-east-1",
    "backup_region": "us-west-2",
    "queue_name": "order-processing",
    "redrive_policy": {
      "deadLetterTargetArn": "arn:aws:sqs:us-east-1:...:order-processing-dlq",
      "maxReceiveCount": 3
    }
  }
}
```

The application writes to both the primary and backup queue simultaneously (dual-write pattern). If the primary region fails, the application reads from the backup queue. Messages that cannot be delivered are moved to a dead-letter queue for manual processing.

### Pattern 3: Cache Integration

Caches present a unique challenge for disaster recovery because they are volatile by design. A failover leaves the secondary region's cache cold.

**Global Redis with replication groups**: Redis Enterprise and AWS ElastiCache for Redis support cross-region replication. A global datastore replicates data asynchronously, and read replicas in the secondary region serve traffic.

```json
{
  "global_cache": {
    "primary_cluster": "mycache-primary.us-east-1",
    "secondary_cluster": "mycache-secondary.us-west-2",
    "replication_mode": "async",
    "failover_behavior": "promote_read_replica",
    "write_concern": "majority"
  }
}
```

During normal operation, applications in both regions read from their local cache. Writes go to the primary cluster and are replicated asynchronously. On failover, the secondary cluster is promoted to primary. The cache is partially warm — data that was written recently is available immediately, but older entries that were evicted from the secondary before replication are cold.

### Pattern 4: CDN Integration

CDNs cache static assets at edge locations. A disaster recovery event requires the CDN to serve from a new origin.

```json
{
  "cdn_config": {
    "distribution_id": "E1ABCDEFGHIJKL",
    "origins": {
      "primary": "origin-primary.example.com",
      "fallback": "origin-secondary.example.com"
    },
    "behavior": {
      "origin_failover": true,
      "failover_codes": [500, 502, 503, 504]
    }
  }
}
```

CloudFront supports origin failover: if the primary origin returns an error, CloudFront automatically retries the request against the fallback origin. This provides CDN-level DR without changing DNS.

## Multi-Region and Global Scale

### Three-Region Strategy

For the highest availability, three regions provide better protection than two. With two regions, if the secondary region also experiences an issue (unlikely but possible), you have no fallback.

A three-region strategy typically uses:

- **Two active regions**: Traffic is split between Region A and Region B (60% / 40%). Both serve reads and writes.
- **One witness region**: Region C runs a minimal footprint and serves as a tiebreaker for consensus protocols. It also hosts the quorum for distributed locking.

### Global Traffic Routing

A global service routes traffic based on the user's geographic location. During normal operation, most users hit the nearest region. During a disaster, traffic is rerouted.

- **Latency-based routing**: DNS routes users to the region with the lowest latency from their location. A user in New York goes to us-east-1. A user in London goes to eu-west-1.
- **Geolocation routing**: Users are explicitly assigned to regions based on their country or continent.
- **Weighted routing**: Traffic is distributed between regions using weighted records. Adjust weights during failover.

## Capacity Planning

### Math for a Global E-Commerce Platform

Let us design the disaster recovery capacity for an e-commerce platform with the following characteristics:

- **Peak traffic**: 100,000 requests per second
- **Average request**: 200ms processing time, 50ms database time
- **Database writes**: 5,000 writes per second at peak
- **Database size**: 2 TB
- **Target RTO**: 5 minutes
- **Target RPO**: 0 (zero data loss for payments, 5 seconds for catalog)

**Step 1: Calculate compute capacity per region**

```
Single-instance throughput = 1000 ms / 200 ms = 5 req/sec per instance
Instances needed at peak = 100,000 / 5 = 20,000
Assume 70% utilization target = 20,000 / 0.7 ≈ 28,600 instances
```

This is too many instances for a single auto-scaling group. In practice, the application would run on larger instances. With c6g.4xlarge (16 vCPU), each instance handles about 80 req/sec.

```
Instances at peak: 100,000 / 80 = 1,250 instances
Secondary region capacity: 1,250 instances (auto-scaled on failover)
Pilot light footprint: 10 instances (minimum for warm standby)
```

**Step 2: Calculate database capacity**

```
Write throughput: 5,000 writes/sec
Aurora writes per second per node: ~10,000 (2TB instance, r6g.8xlarge)
Required write capacity: 5,000 / 10,000 = 0.5 nodes ≈ 1 writer node
Read traffic: 95,000 reads/sec
Aurora read replicas needed: 95,000 / 30,000 = 3.2 ≈ 4 read replicas
Secondary region: 1 read replica (promoted on failover)
```

**Step 3: Calculate network bandwidth for replication**

```
WAL generation rate at peak writes:
  Each write ≈ 2 KB of WAL data
  Total: 5,000 × 2 KB = 10 MB/sec
  Cross-region bandwidth needed: 10 MB/sec ≈ 80 Mbps
  Required: 100 Mbps link minimum (with headroom)
```

**Step 4: Cache capacity**

```
Working dataset for read-heavy endpoints: 500 GB
Redis cluster with replication factor 2: 1 TB total
Cross-region replication bandwidth for cache writes:
  10% write rate: 10,000 cache writes/sec
  Each cache write ≈ 1 KB: 10 MB/sec additional bandwidth
```

**Step 5: Total cost estimate**

```
Primary region (us-east-1, on-demand):
  Compute: 1,250 instances × $0.408/hr = $510/hr
  Database: 1 writer + 4 readers × $2.464/hr = $12.32/hr
  Cache: 10 r6g.xlarge nodes × $0.504/hr = $5.04/hr
  Storage: 2 TB × $0.115/GB-month = $230/month
  Data transfer: ~$500/month

Secondary region (us-west-2, pilot light):
  Compute: 10 instances × $0.408/hr = $4.08/hr
  Database: 1 reader × $2.464/hr = $2.46/hr
  Cache: 2 nodes × $0.504/hr = $1.01/hr
  Replication bandwidth: ~$100/month

Total normal operation: ~$530/hr (primary) + ~$8/hr (secondary)
DR overhead: ~1.5% of production cost

Total during failover (secondary at full scale):
  Secondary compute: 1,250 instances × $0.408/hr = $510/hr
  Total during failover: ~$530/hr (same as primary)
```

## Deployment and Operations

### Infrastructure as Code for DR

Disaster recovery infrastructure must be defined in code, deployed through CI/CD, and tested automatically. Terraform, Pulumi, or CloudFormation templates define the complete multi-region setup.

```hcl
# Primary region module
module "primary" {
  source = "./modules/region"
  region = "us-east-1"
  is_primary = true
  app_count = 100
  db_instance_class = "db.r6g.2xlarge"
}

# Secondary region module
module "secondary" {
  source = "./modules/region"
  region = "us-west-2"
  is_primary = false
  app_count = 2  # pilot light
  db_instance_class = "db.r6g.2xlarge"
  replication_source = module.primary.db_arn
}

# DR orchestrator
module "dr_orchestrator" {
  source = "./modules/orchestrator"
  primary_region = "us-east-1"
  secondary_region = "us-west-2"
  failover_rto_target = 300  # seconds
  health_check_endpoint = module.primary.app_health_url
  notification_channels = ["slack:#ops", "pagerduty:xxx"]
}
```

### CI/CD Pipeline for DR

Every deployment must be replicated to both regions. The pipeline applies the same changes to both regions and verifies the secondary has been updated.

The deployment process for a multi-region system follows a careful sequence:

1. Deploy to the secondary region first. This allows you to validate the deployment on live infrastructure without affecting users.
2. Run automated smoke tests against the secondary region. Verify that the new version passes health checks and synthetic transactions.
3. If the secondary region tests pass, deploy to the primary region. Use a rolling deployment to minimize impact.
4. After deployment, verify that the replication lag between regions is within acceptable limits. The cross-region replication should be streaming the new schema correctly.
5. Run the automated DR validation suite against the secondary region. This verifies that the failover path still works with the new deployment.

```yaml
deploy:
  stages:
    - build_and_test
    - deploy_primary:
        region: us-east-1
        verify: true
    - deploy_secondary:
        region: us-west-2
        wait_for_replication: true
    - dr_validation:
        run: verify_dr_health.sh
        assert:
          - "secondary_replication_lag < 30s"
          - "secondary_app_health == healthy"
          - "failover_dry_run == success"
```

### Runbook Automation

The DR runbook is not a PDF — it is executable code. Each step is a script or a Lambda function that runs automatically.

```python
# dr_runbook.py -- executed by the DR orchestrator on failover
def execute_runbook(event, context):
    phase = event["phase"]

    if phase == "detect":
        return detect_disaster(event)
    elif phase == "decide":
        return decide_failover(event)
    elif phase == "execute":
        return execute_failover(event)
    elif phase == "validate":
        return validate_failover(event)
    elif phase == "notify":
        return notify_team(event)

def execute_failover(event):
    results = {}
    # Step 1: Acquire failover lock
    results["lock"] = acquire_lock(event["new_primary_region"])
    if not results["lock"]:
        raise Exception("Could not acquire failover lock")

    # Step 2: Update DNS in parallel
    results["dns"] = update_dns_records(
        event["new_primary_region"],
        event["old_primary_region"],
    )

    # Step 3: Promote database
    results["db_promotion"] = promote_database(
        event["new_primary_region"],
        event["db_identifier"],
    )

    # Step 4: Reload application configuration
    results["config_reload"] = reload_app_config(
        event["new_primary_region"],
        event["new_db_endpoint"],
    )

    # Step 5: Warm cache
    results["cache_warmup"] = trigger_cache_warmup(
        event["new_primary_region"],
    )

    return results
```

### Game Day Drills

A game day is a scheduled, controlled disaster simulation. The goal is to validate the DR system under realistic conditions.

**Sample game day schedule**:

```markdown
## Game Day: Region Failure Simulation

### Scenario
us-east-1 experiences a complete failure. All compute, database, 
and cache instances become unreachable.

### Prerequisites
- [ ] Rollback plan documented
- [ ] Engineering team briefed (not including participants)
- [ ] Stakeholders notified (read-only observation)
- [ ] Monitored: RTO, RPO, error rates, p99 latency

### Timeline
00:00 - Simulator injects failure (blocks all traffic to us-east-1)
00:01 - Health checks begin failing
00:30 - Composite trigger evaluates, initiates failover
01:00 - DNS failover initiated
01:30 - Database promotion begins
02:15 - Config reload complete
03:00 - Traffic flowing to us-west-2
04:30 - Validation complete, failover declared successful
05:00 - Measure and record: RTO = 3min 17sec, RPO = 12sec

### Post-Game Review
- Did failover complete within RTO target (5 min)? YES (3:17)
- Was data loss within RPO target (30 sec)? YES (12 sec)
- Were any manual steps required? NO
- Were all monitoring alerts accurate? 1 false positive from probe 2
- Action items:
  1. Investigate probe 2 latency spike (false positive)
  2. Reduce DNS TTL from 60s to 30s (recover 30s from RTO)
  3. Add cache warm-up step to validation suite
  4. Schedule next game day in 30 days
```

## Testing the DR System

### Automated DR Validation

The DR system should be validated continuously, not just during scheduled drills. Automated validation runs every deployment and checks that the failover path is healthy.

```python
# Automated DR validation -- runs in CI/CD after every deployment
def validate_dr():
    checks = {}
    # Check that the secondary region is running the correct version
    checks["version_match"] = (
        get_app_version("us-east-1") == get_app_version("us-west-2")
    )

    # Check that replication lag is within limits
    lag = get_replication_lag("us-west-2")
    checks["replication_lag"] = lag < 30  # seconds

    # Check that health check endpoints are reachable
    checks["secondary_health"] = check_health("us-west-2")

    # Check that DNS failover records are configured correctly
    checks["dns_config"] = verify_dns_records()

    # Run a synthetic transaction against the secondary
    checks["synthetic_transaction"] = run_synthetic(
        "us-west-2", read_only=True
    )

    # Report
    failed = [k for k, v in checks.items() if not v]
    if failed:
        raise Exception(f"DR validation failed: {failed}")
    print("DR validation passed")
```

### Observability for DR Readiness

DR readiness should be visible on a dashboard. The dashboard answers: "If the primary region failed right now, would the failover work?"

Key metrics on the DR dashboard:

- **Replication lag (seconds)**: Current lag between primary and secondary. Alert if over 10 seconds.
- **Secondary health score**: Composite health score of the secondary region. Red if under 0.6.
- **Failover test results**: Pass/fail status of the last automated failover test.
- **Days since last game day**: If more than 90 days, the DR readiness is considered stale.
- **Configuration drift**: Number of configuration differences between primary and secondary.

## Cost Optimization for DR

DR infrastructure is expensive because it duplicates resources. Several strategies can reduce the cost:

**Reserved instances for DR capacity**: The secondary region's compute capacity can run on reserved instances at a 30-40% discount compared to on-demand. Since the secondary capacity is always running (for warm standby), the reservation is fully utilized.

**Spot instances for non-critical DR workloads**: For DR testing and validation environments, spot instances provide 60-90% discounts. Only the actual failover infrastructure needs on-demand or reserved instance pricing.

**Storage tiering for backups**: Use lifecycle policies to move backups through storage tiers. Recent backups (7 days) stay in hot storage for fast restore. Older backups (30+ days) move to cold storage at 1/10th the cost. Archives (90+ days) move to glacier storage at 1/100th the cost.

**Scheduled scaling for the secondary**: During normal operation, the secondary region only needs pilot light capacity. Use scheduled scaling to reduce the secondary footprint during low-traffic periods. Scale up during business hours when a failover would be most impactful.

**Regional traffic shifting**: A cost optimization specific to active-active setups is to shift more traffic to the cheaper region during normal operation. If one region has lower compute costs or reserved instances, route 60% of traffic to that region. During a disaster, the remaining region handles 100% of traffic. This reduces the average cost per request while maintaining the redundancy needed for DR.

## Security Considerations for DR

Disaster recovery introduces security considerations that differ from normal operation. The secondary region must have the same security posture as the primary region, even though it runs at reduced capacity.

**Encryption**: All cross-region replication traffic must be encrypted in transit. AWS cross-region replication uses TLS by default. Verify that the encryption in transit is enabled for all replication channels. The secondary region must also have encryption at rest enabled, using the same key management system or a cross-region key replica.

**Access control**: The secondary region should have the same IAM roles and policies as the primary region. If the primary region's IAM is not replicated to the secondary region, the failover will fail because the application cannot access the database. Use a centralized IAM provider or automate the replication of IAM policies.

**Secrets management**: Database passwords, API keys, and encryption keys must be available in the secondary region. Use a secrets manager that supports cross-region replication or a third-party vault that is accessible from all regions. During a failover, the application must be able to retrieve the same secrets from the secondary region.
