## Core Tension

Every distributed scheduler faces the same fundamental tension: **you cannot simultaneously minimize latency, maximize throughput, and guarantee exactly-once execution at scale**.

Pick any two. This is not a limitation of a particular implementation. It is a consequence of distributed systems theory.

- **Low latency + High throughput** = You accept duplicates (at-least-once). Workers claim tasks eagerly, leases are short, and the system optimizes for speed over strict deduplication.

- **Low latency + Exactly-once** = Throughput is limited by coordination overhead. Each task requires consensus or a distributed transaction, which adds latency and reduces parallelism.

- **High throughput + Exactly-once** = You accept higher latency. Tasks are batched, claims are coordinated, and the system trades responsiveness for reliability.

The right choice depends on your domain. A payment scheduler needs exactly-once and accepts higher latency. A notification scheduler optimizes for latency and can tolerate occasional duplicates.

### Concrete Examples

**Notification system (low latency + high throughput)**: You send 10 million push notifications per hour. A duplicate rate of 0.1 percent is invisible to users. The system uses 200 workers polling every 100ms, with 30-second leases. Each worker claims batches of 50 tasks. The schedule index runs on Redis for sub-millisecond poll latency. If Redis loses a pending task during failover, the notification is simply never sent -- acceptable for a promotional campaign. The system processes 50,000 tasks per second at p99 latency of 150ms.

**Payment processing system (low latency + exactly-once)**: You process 50,000 transactions per hour. A duplicate payment would trigger a financial reconciliation incident. The system uses 10 workers with 60-second leases and a consensus-based claim mechanism. Each worker claims one task at a time. The schedule index uses PostgreSQL with serializable isolation. The system processes 14 tasks per second at p99 latency of 400ms. Throughput is 3,500 times lower than the notification system for the same hardware.

**Data pipeline (high throughput + exactly-once)**: You process 5 million analytics events per hour. Idempotent writes to the data warehouse make duplicates harmless, but you still need each event to be processed at least once. The system batches 200 tasks per claim, with 5-minute leases. Workers poll every 10 seconds. The schedule index uses DynamoDB with conditional writes. The system processes 1,400 tasks per second at p99 latency of 3 seconds. The latency is higher, but the infrastructure cost per task is 10 times lower than the payment system.

### The Middle Ground

In practice, most systems do not sit at one vertex of the triangle. They find a practical compromise:

- **Bounded at-least-once**: The system guarantees at-least-once delivery but adds a lightweight deduplication layer. In-memory bloom filters track recently completed task IDs. Duplicates are detected within 30 seconds and dropped. This gives you 90 percent of the latency benefit of at-least-once with 90 percent of the reliability benefit of exactly-once.

- **Optimistic exactly-once**: The system assumes no contention and uses optimistic locking. Each task claim includes a version number. If two workers claim the same task, the second claim fails. The worker backs off and claims the next task. This works well when contention is below 1 percent of claims.

- **Quorum-based leases**: Instead of full consensus, the system uses a lightweight quorum. A majority of replicas must agree on lease ownership. This adds 5-10ms to the claim operation but avoids the 100ms+ overhead of Paxos or Raft for each task.

## Decision Matrix

The following matrix maps common scheduler requirements to the recommended design decisions. Each row addresses a specific requirement and the trade-offs it creates.

| Requirement | Recommendation | Trade-off |
|---|---|---|
| Sub-second latency (p99 less than 500ms) | Push-based notification + 100ms backup poll | Higher system complexity, message queue dependency |
| Exactly-once execution | Idempotent tasks + at-least-once delivery | Application-level deduplication overhead |
| 100,000+ tasks per second | 32+ shards, Redis sorted sets, batch claims | Higher operational complexity, potential data loss on Redis failover |
| Durability over latency | PostgreSQL or DynamoDB backing store | 10-50ms higher latency per operation |
| Multi-tenancy | Task-level isolation with tenant ID prefix | Cross-tenant fairness issues, noisy neighbor problem |
| Unlimited queue depth | Database-backed with paginated polling | Higher database cost, slower index scans |
| Cron-style fixed schedule | Use cron wrapper over the scheduler | Loss of dynamic scheduling flexibility |
| FIFO ordering per partition | Single-worker per partition or ordered message queue | Throughput limited to single worker per partition |
| Cost-optimized (low infra) | Database polling with fewer, larger workers | Higher latency, higher risk during worker failures |
| Global multi-region | Active-active with local queues + global replication | Eventual consistency, cross-region conflict resolution |
| Task versioning and schema evolution | Versioned task payloads + migration handler | Additional storage per version, migration logic |
| Batch processing of similar tasks | Coalescing scheduler: group by task type + delay window | Higher latency for individual tasks, added buffering complexity |
| Priority-based execution | Priority queue with weighted scheduling | Starvation of low-priority tasks if not weighted correctly |

### Decision Matrix Deep Dive

**Sub-second latency** requires push-based notification because polling alone cannot achieve sub-10ms latency at scale. A push notification from the schedule index to the worker cuts the poll cycle from 500ms to 2ms. The backup poll at 100ms catches missed push notifications. The trade-off is a persistent connection between the schedule index and each worker, which adds connection management overhead.

**Exactly-once execution** is best achieved through idempotent tasks with at-least-once delivery. The scheduler cannot guarantee exactly-once at the infrastructure level -- it can only guarantee at-least-once. The final deduplication must happen in the task handler. Each task carries an idempotency key. The downstream system checks this key before processing. If the key exists, the task is skipped. This pattern works for 99.99 percent of use cases.

**100,000+ tasks per second** requires sharding because a single Redis instance handles approximately 50,000 operations per second. With 32 shards, each shard handles 3,125 tasks per second. Batch claims of 50 tasks per poll reduce the claim operation rate by 50x. The risk is that a Redis failover during a rebalance window can lose up to 1 percent of pending tasks.

**Durability over latency** means choosing a database that writes to disk before acknowledging. PostgreSQL with synchronous replication adds 10-50ms to each write operation. DynamoDB with consistent reads adds 10-20ms. The benefit is that a power failure at the data center does not lose pending tasks.

**Priority-based execution** requires a priority queue. Each task has a priority level from 0 to 10. Higher priority tasks are scheduled before lower priority tasks. The schedule index maintains a separate sorted set per priority level. Workers poll higher priority levels first. The trade-off is that low-priority tasks may starve if high-priority tasks are continuously enqueued. Weighted scheduling mitigates this: each priority level gets a minimum share of worker capacity.

## CAP Theorem Analysis

The distributed scheduler is a distributed system, and CAP theorem applies. The scheduler is a CP system by nature: it cannot sacrifice partition tolerance because partitions happen. The trade-off is between Consistency and Availability.

### CP Configuration (Consistency over Availability)

The scheduler waits for a majority of replicas to acknowledge every enqueue, claim, and completion. If a partition separates the shard leader from its followers, the leader stops accepting new tasks until the partition heals.

**When to choose CP**: Payment processing, financial reconciliation, inventory management. Any domain where a missed or duplicated task has monetary or compliance consequences.

**CP scenario in detail**: A payment scheduler has 3 replicas per shard. The leader writes to a PostgreSQL table with `synchronous_commit = on`. Every enqueue waits for 2 of 3 replicas to acknowledge. If replica 2 is partitioned, the leader has only 2 of 3 replicas. Enqueues proceed because the leader still has a majority (itself + replica 1). If both replicas 1 and 2 are partitioned, the leader has only 1 of 3 replicas. Enqueues stop. The scheduler is unavailable for writes but guarantees that no task is lost or duplicated.

**CP cost to throughput**: Each synchronous write adds 10-50ms. With 3 replicas, the write amplification is 3 times. The effective throughput is 33 percent of the theoretical maximum. For 50,000 tasks per hour, you need 3 times the infrastructure of an AP configuration.

### AP Configuration (Availability over Consistency)

The scheduler accepts enqueues and claims on any replica. During a partition, replicas may diverge. The system relies on conflict resolution after the partition heals. Tasks may execute more than once.

**When to choose AP**: Notifications, analytics, cache warming, log processing. Any domain where occasional duplicates are acceptable and uptime is critical.

**AP scenario in detail**: A notification scheduler has 3 replicas per shard. Each replica accepts writes independently. During a partition, replica 1 accepts enqueue for task A, and replica 2 accepts enqueue for task A (duplicate). After the partition heals, the conflict resolution detects the duplicate. Both tasks execute. The downstream notification service receives two identical notifications. The notification service uses an idempotency key to drop the second notification. The user sees one notification.

**AP cost to consistency**: The system accepts that during a partition, the task count may be off by up to 5 percent. Tasks may execute 2-3 times. The benefit is that the scheduler remains available for writes during any partition scenario. The notification system meets its 99.99 percent uptime SLA.

### Practical CAP: The Graceful Degradation Pattern

Most production schedulers use a hybrid approach. They default to CP but degrade to AP when the partition affects only the state store, not the schedule index.

**Graceful degradation rules**:

1. The schedule index (pending tasks) always operates in CP mode. No exceptions. Losing a pending task is data loss.

2. The state store (completed tasks) operates in AP mode. During a partition, the state store accepts writes on any replica. After the partition heals, the system reconciles duplicates.

3. If the schedule index cannot achieve a majority, the scheduler enters read-only mode. Workers can still poll and execute tasks that are already claimed, but no new tasks are enqueued and no existing tasks are reclaimed.

4. After 30 seconds in read-only mode, the scheduler escalates to controlled shutdown. Workers finish their current tasks and stop accepting new ones. The operator is alerted.

### The Scheduler-Specific CAP Insight

The scheduler has an unusual CAP property: the state that matters most is the future state (pending tasks), not the past state (completed tasks). Losing a completed task record is acceptable (the task already executed). Losing a pending task is catastrophic (the task never executes). The schedule index (pending tasks) must be CP. The state store (completed tasks) can be AP.

## Failure Modes

### Failure Mode 1 -- Thundering Herd on Recovery

A worker pool of 200 machines polls every 500ms. All 200 workers crash simultaneously. After 30 seconds, all 200 workers restart simultaneously. Each worker immediately polls for due tasks. The schedule index receives 200 queries in 100ms. The database CPU spikes to 100 percent. Queries time out. Workers retry. The database stays saturated.

**Mitigation**: Implement jittered startup. Each worker delays its first poll by a random value between 0 and poll_interval. With 200 workers and 500ms poll interval, the startup delay spreads the initial queries over 500ms instead of 100ms. Additionally, use a circuit breaker that limits database connections per shard.

### Failure Mode 2 -- Lease Thundering Herd

A task that triggers a high-latency external API has a 30-second lease. After 30 seconds, the external API is slow but not down. The worker is still executing but has not completed. The lease expires. A second worker claims the task. Now two workers execute the same task. The downstream system sees a duplicate.

**Mitigation**: Set lease duration to p99 execution time multiplied by 3. Implement heartbeats for long-running tasks. Use a grace period: the reclaim scanner skips tasks whose lease expired less than grace_duration ago, giving the original worker time to complete and update the status.

### Failure Mode 3 -- Skewed Shard Distribution

Tasks are distributed across shards by hashing the task ID. If a single tenant enqueues 500,000 identical tasks (e.g., send promotional emails), the hash of their IDs may concentrate on one shard. That shard becomes overloaded while others sit idle.

**Mitigation**: Use two-level sharding: first by tenant ID, then by task ID within the tenant. Implement per-tenant rate limiting. Monitor shard imbalance and trigger rebalancing when one shard exceeds 20 percent more tasks than the median.

### Failure Mode 4 -- Zombie Worker

A worker is partitioned from the schedule index but not from its task execution target. The worker continues executing tasks because it cannot reach the index to check lease expiry. These zombie executions duplicate work that other workers are also performing.

**Mitigation**: Workers should periodically check their lease status by contacting the lease manager. If the lease manager is unreachable, the worker should pause execution. This is the distributed equivalent of a kill switch. Workers should also record execution results with an idempotency key that the downstream system uses for deduplication.

### Failure Mode 5 -- Clock Skew Cascade

Worker A has a clock 10 seconds behind NTP. It polls for tasks due at or before 14:00:00, but the actual time is 14:00:10. Worker A is 10 seconds late polling. Meanwhile, the reclaim scanner (on a machine with accurate time) sees Worker A's leases as expired and reclaims them. Tasks end up executing twice or thrice.

**Mitigation**: All time comparisons must use an authoritative clock source (the database or a consensus-based monotonic clock). Workers should compare lease expiry against the schedule index's clock, not their local clock. NTP should be configured with tight error bounds.

### Failure Mode 6 -- Database Connection Pool Exhaustion

A scheduler has 50 workers, each maintaining 10 persistent database connections. The database connection pool is configured for 200 connections. During a peak load event, the schedule index needs 150 connections for claim operations. The remaining 50 connections are shared among all workers for polling. Each poll takes 200ms. With 50 workers polling every 500ms, the queue of pending poll requests grows to 20 requests. Each poll waits 4 seconds for a connection. Tasks that are due are not discovered until the poll completes. The latency spikes from 500ms to 4.5 seconds.

**Mitigation**: Separate the poll connection pool from the claim connection pool. The poll pool uses 50 connections (one per worker). The claim pool uses 150 connections. This prevents claim operations from starving poll operations. Set a connection timeout on the poll pool: if a connection is not available within 100ms, the poll is skipped. The poll interval is short enough that the next poll catches the due tasks.

### Failure Mode 7 -- Rebalance Storm During Cluster Expansion

You add 10 new shards to a 20-shard cluster to handle increased load. The rebalance process moves 50 percent of the tasks from the existing 20 shards to the new 10 shards. Each move operation requires a read from the source shard, a write to the destination shard, and a delete from the source shard. With 500,000 tasks per shard, the rebalance generates 10 million move operations. The move operations compete with the poll and claim operations for database capacity. Poll latency increases from 500ms to 5 seconds. Tasks are delayed by minutes.

**Mitigation**: Use a gradual rebalance strategy. Move 10 percent of the tasks per hour instead of all at once. Throttle the move rate to 20 percent of the database capacity. Schedule the rebalance during low-traffic periods. Use a background rebalance worker that yields to poll and claim operations. If the poll latency exceeds 1 second, the rebalance pauses until latency drops below 500ms.

## Anti-Patterns

### Anti-Pattern 1 -- Using the Scheduler as a Workflow Engine

You build complex multi-step workflows with dependencies, conditional branches, and error handling directly in the scheduler task definitions. The scheduler becomes a workflow engine.

**Why it fails**: Schedulers are optimized for time-based execution, not state machines. Workflow logic embedded in task payloads is opaque, hard to debug, and impossible to monitor. Real workflow engines like Temporal or Step Functions provide explicit state management, compensation, and visibility.

**Use instead**: A dedicated workflow engine for orchestration. Use the scheduler for individual timed tasks within the workflow.

### Anti-Pattern 2 -- Polling Interval Set to Zero

You set the poll interval to 0 or 1 millisecond to minimize latency. Workers continuously hammer the schedule index with poll queries.

**Why it fails**: The database or Redis instance spends 90 percent of its CPU handling poll queries that return zero results (because no tasks are due). The system burns resources on polling, not execution. The latency gain is marginal because the claim operation and task execution dwarf the poll time.

**Use instead**: Set the poll interval to 10 times the p99 database query latency. Use push notifications for near-zero latency. A 500ms poll interval with push notification gives sub-10ms latency with 1 percent of the database load.

### Anti-Pattern 3 -- Infinite Retries

You set max_retries to a very high number (100+) to avoid any task being lost. Tasks that will always fail are retried indefinitely, consuming resources and blocking the queue.

**Why it fails**: A permanently failing task (e.g., invalid payload, deleted resource) is retried 100 times before being dead-lettered. Each retry consumes a lease, a poll cycle, and execution capacity. The dead-letter rate is zero, but the system is wasting 99 percent of its throughput on doomed tasks.

**Use instead**: Set max_retries to 3 for transient failures and 0 for permanent failures. Implement a dead-letter queue with manual inspection. The DLQ should alert operators who can fix the issue and re-enqueue the task.

### Anti-Pattern 4 -- Monolithic Task Payloads

You store the complete execution payload (including large binary data, full API responses, or serialized objects) in the schedule index.

**Why it fails**: The schedule index is optimized for time-ordered queries, not blob storage. Large payloads increase memory pressure, slow down index scans, and increase claim latency by 10 times or more. A 1MB payload per task means 1 GB per 1,000 tasks in the queue.

**Use instead**: Store large payloads in an object store (S3, GCS) and reference them by key in the scheduler task. The task payload should be a lightweight reference (typically less than 1KB).

### Anti-Pattern 5 -- Synchronous Task Execution in the Poll Loop

You execute the task within the poll function, blocking subsequent polls until the task completes.

```python
# ANTI-PATTERN: blocking execution in poll loop
def poll():
    tasks = get_due_tasks()
    for task in tasks:
        execute_sync(task)  # blocks here for seconds
    time.sleep(0.5)
```

**Why it fails**: A single slow task delays every subsequent task by seconds or minutes. The poll interval becomes meaningless. The system degrades from sub-second latency to minutes during any high-latency task.

**Use instead**: Submit tasks to an async execution pool. The poll function should spend minimal time per task, delegating execution to background threads or processes.

### Anti-Pattern 6 -- Ignoring Task Priority

You treat all tasks as equal. A batch of 10,000 low-priority analytics tasks fills the queue ahead of 50 high-priority payment tasks. The payment tasks wait behind the analytics tasks for 5 minutes.

**Why it fails**: Without priority, the scheduler processes tasks in FIFO order by due time. Low-priority batch tasks that are enqueued first block high-priority tasks that are enqueued later. The system meets the SLA for batch tasks but misses the SLA for priority tasks.

**Use instead**: Implement a priority queue with at least 3 levels: high, normal, low. Workers poll high-priority queues first. High-priority tasks get 50 percent of worker capacity. Normal and low priority tasks share the remaining 50 percent. Set a maximum wait time for each priority level. If a low-priority task has waited longer than the maximum, it is promoted to normal priority.

### Anti-Pattern 7 -- Global Locking

You use a distributed lock (e.g., Redis Redlock, ZooKeeper lock) to ensure that only one worker processes the schedule at a time. This eliminates contention but serializes the entire scheduler.

**Why it fails**: The global lock reduces the system to single-worker throughput. With 100 workers and a global lock, only 1 worker is active at any time. The effective throughput is 1 percent of the available capacity. The lock acquisition and release add 10-50ms per poll cycle, further reducing throughput.

**Use instead**: Use per-shard locks instead of a global lock. Each shard has its own lock, allowing up to N workers to operate concurrently (one per shard). If the scheduler has 32 shards, 32 workers can process tasks simultaneously. The lock granularity should match the shard granularity. Per-shard locks add 2ms per poll cycle instead of 50ms for a global lock.

## When NOT to Use a Distributed Scheduler

A distributed scheduler is a powerful tool, but it is not the right solution for every timing problem.

### You Do NOT Need It When

**You have fewer than 1,000 tasks per hour**: A simple cron job, a single-process scheduler with a database, or a lightweight library like APScheduler or node-schedule will suffice. The operational complexity of a distributed scheduler is not justified at this scale.

**Your tasks are all immediate (no delay)**: If every task executes within milliseconds of being enqueued, you do not need scheduling at all. Use a message queue directly. The scheduler adds latency with its time-indexed structure.

**Your delay requirements are coarse (minutes to hours)**: If the minimum delay is 5 minutes, a cron-based approach with minute-level granularity is simpler and more reliable. Use a message queue with a delay plugin (RabbitMQ delayed messages, SQS delay queues) instead of a full scheduler.

**You need a state machine with complex branching**: A distributed scheduler handles time-triggered execution, not conditional branching. Use a workflow engine (Temporal, Cadence, AWS Step Functions, Airflow) for stateful orchestration with decision points.

**Your tasks must run in strict FIFO order**: A distributed scheduler with multiple workers will reorder tasks based on execution time variance. For strict FIFO, use a single-consumer queue with ordered delivery (Kafka partition with one consumer, SQS FIFO queue).

**You need sub-millisecond latency**: The overhead of the schedule index lookup (1-5ms for Redis, 10-50ms for a database) makes sub-millisecond scheduling impossible. Use an in-memory timer wheel or a real-time event stream (Kafka with consumer groups) for sub-millisecond delivery.

**Your tasks are all idempotent and can tolerate loss**: If losing a task is acceptable and the task handler is idempotent, you do not need a scheduler at all. A simple message queue with at-least-once delivery provides better throughput and lower latency.

**You have a single machine**: A distributed scheduler on a single machine adds network overhead, serialization cost, and operational complexity with zero benefit. Use a single-process scheduler. The single-machine version of BullMQ or Celery gives you 90 percent of the functionality with 10 percent of the complexity.

### Signs You Have Outgrown Your Current Solution

You started with a cron job or a simple message queue. Here are the signs that you need to migrate to a distributed scheduler:

1. **Cron jobs are stacking**: You have 20+ cron jobs, and managing their schedules, dependencies, and failure handling is a full-time operational burden.

2. **Tasks are delayed without explanation**: A single slow cron job blocks the execution of other jobs. You cannot explain why a task that should run at 14:00 runs at 14:07.

3. **You manually retry failed tasks**: An operator spends 2 hours per day identifying and re-enqueuing failed tasks. The system has no automatic retry or dead-letter queue.

4. **You cannot trace task execution**: When a task fails, you cannot determine which worker executed it, how long it took, or what the error was. The only evidence is a log line on a machine that was recycled.

5. **Your queue is growing unbounded**: The number of pending tasks grows faster than the workers can process them. You add more workers, but the bottleneck is the queue itself, not the processing capacity.

6. **You need execution guarantees**: A task that fails during a deployment is silently lost. You need at-least-once delivery, retries, and a dead-letter queue for permanently failed tasks.

## Migrating Between Scheduler Types

As your system grows, you will outgrow simpler scheduling solutions. The migration path follows a predictable evolution.

### Evolution Path

**Phase 1 -- Cron**: You start with 5-10 cron jobs on a single server. Each cron job runs a script that processes a batch of work. The cron jobs have no retry logic, no monitoring, and no execution history. This works for 1,000 tasks per hour with minute-level granularity.

**Phase 2 -- Database Polling**: You add a polling loop that queries a database table for due tasks. The poll runs every 5 seconds. Tasks have a `scheduled_at` timestamp and a `status` column. This gives you sub-minute precision and basic task tracking. You can handle 10,000 tasks per hour.

**Phase 3 -- Redis-based Scheduler**: You adopt a Redis-backed scheduler like BullMQ or Sidekiq. The scheduler handles 100,000 tasks per hour with sub-second precision. Redis sorted sets provide efficient time-ordered queries. You get automatic retries, dead-letter queues, and basic monitoring. The schedule index is in memory, so latency is 1-5ms per operation.

**Phase 4 -- Full Distributed Scheduler**: You build or adopt a fully distributed scheduler with sharding, multi-region support, and exactly-once semantics. The scheduler handles 1 million tasks per hour with sub-second latency and self-healing capabilities. This is where the trade-offs and failure modes in this page become critical.

### Migration Strategies

**Dual-write strategy**: Run both the old and new scheduler in parallel. Every task is enqueued to both systems. The old system remains the primary. The new system runs in shadow mode: it processes tasks but does not complete them. After 2 weeks of validation, switch the primary to the new system. The dual-write period gives you time to detect data loss, latency issues, or correctness bugs in the new system.

**Drain-and-switch**: Stop enqueuing new tasks to the old system. Let the old system drain its remaining tasks. Once the old queue is empty, switch all traffic to the new system. This strategy works best when the old system can drain within a maintenance window. The drain time depends on the queue depth. A queue of 500,000 tasks with 50 workers takes 2-3 hours to drain.

**Gradual shard migration**: Move one shard at a time from the old system to the new system. Each shard handles a subset of tasks. As you validate each shard, you move the next one. This strategy minimizes risk because a failure affects only one shard. The migration takes longer but is safer. With 32 shards and a 1-day validation period per shard, the migration takes 32 days.

### Migration Checklist

| Step | Action | Validation |
|---|---|---|
| 1 | Deploy new scheduler in shadow mode | No data loss, no duplicate tasks |
| 2 | Compare task execution counts | Less than 0.1 percent discrepancy |
| 3 | Compare latency p99 | Within 10 percent of old system |
| 4 | Enable dual-write for 10 percent of tasks | No errors, no performance regression |
| 5 | Increase to 50 percent of tasks | Stable for 1 week |
| 6 | Switch 100 percent to new system | Monitor for 2 weeks |
| 7 | Drain and decommission old system | Zero tasks in old queue |

## Community Patterns and Ecosystem

The distributed scheduler ecosystem has converged on several architecture patterns. Understanding these patterns helps you choose the right foundation for your system.

### Redis-Based Schedulers

Redis-based schedulers (BullMQ, Sidekiq, RQ, Celery with Redis broker) are the most popular choice for high-throughput, low-latency workloads. Redis sorted sets provide O(log N) time-ordered queries. The schedule index is entirely in memory, giving 1-5ms poll latency.

**Best for**: Notification systems, real-time analytics, cache warming, and any workload where losing a few pending tasks during a failover is acceptable.

**Limitations**: Redis persistence is not as strong as a database. A Redis failover during a write can lose up to 1 second of data. The total memory limits the queue depth. A Redis instance with 16GB of RAM can hold approximately 16 million tasks with 1KB payloads.

### Database-Backed Schedulers

Database-backed schedulers (pg_timetable, DBMS_SCHEDULER, or custom-built schedulers on PostgreSQL, MySQL, DynamoDB) provide full durability and ACID guarantees. The schedule index is on disk, giving 10-50ms poll latency.

**Best for**: Payment processing, financial reconciliation, inventory management, and any workload where task loss is unacceptable.

**Limitations**: Poll latency is 10-50ms, which is 10-50 times higher than Redis. The database becomes the bottleneck under high throughput. Scaling beyond 100,000 tasks per second requires careful sharding and connection management.

### Cloud-Native Schedulers

Cloud-native schedulers (AWS SQS with Lambda triggers, Google Cloud Tasks, Azure Queue Storage) provide managed scheduling with pay-per-use pricing. The cloud provider handles scaling, durability, and failover.

**Best for**: Serverless workloads, startups, and teams that want to minimize operational overhead.

**Limitations**: Vendor lock-in is significant. Migrating from AWS SQS to Google Cloud Tasks requires a full rewrite of the scheduling layer. Cloud-native schedulers have less flexibility in retry policies, lease management, and custom scheduling logic.

## Cost-Benefit Summary

| Factor | Benefit of Distributed Scheduler | Cost |
|---|---|---|
| Scale | Handles 1 million tasks per second | 3 times infrastructure cost vs. single-node |
| Reliability | Survives machine, AZ, region failures | 2 times operational complexity |
| Latency | Sub-second task execution | 5 times database load vs. batch polling |
| Durability | Tasks survive process crashes | Write amplification from durability config |
| Flexibility | Dynamic scheduling, cancel, retry, DLQ | Learning curve for operators |
| Observability | Full task lifecycle visibility | Storage cost for task history |
| Multi-tenancy | Isolated task queues per tenant | Shard imbalance and noisy neighbor issues |
| Rebalancing | Adaptive capacity through rebalancing | Risk of rebalance storm during scaling events |
| Monitoring | Built-in metrics for queue depth, latency, throughput | Alert fatigue from false positives on transient delays |
| Extensibility | Plugin system for custom scheduling policies | API stability burden for long-term maintenance |
