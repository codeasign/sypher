## How It Appears in Interviews

The distributed scheduler pattern appears in system design interviews more often than you might expect. Interviewers rarely ask for a scheduler directly. Instead, the scheduler emerges as the solution to a timing problem within a larger design.

Here are 10 common contexts where the distributed scheduler pattern appears, along with what the interviewer is specifically looking for when you propose it.

**1. Design a payment system**: Payments have timeouts, retries, and settlement delays. The scheduler handles payment expiry (hold for 30 minutes, then release or capture), retry on failure, and recurring billing. The interviewer expects you to identify that payment timeouts have strict accuracy requirements -- a payment that clears a few seconds late could mean double-spending. They want to hear about idempotency keys and exactly-once semantics for capture requests. You should mention that the scheduler must distinguish between soft declines (retryable, for example, timeout) and hard declines (not retryable, for example, insufficient funds), and have separate handling paths for each. The interviewer also watches for your understanding of settlement delays: payments that are authorized but not captured within a window must be released, and the scheduler tracks that window precisely.

**2. Design a food delivery platform**: Orders have timed states (notify restaurant immediately, dispatch driver in 3 minutes, escalate if late by 15 minutes). The scheduler is the backbone of order lifecycle management. The interviewer watches for your awareness that state transitions must survive the complete failure of the order service -- if the scheduler restarts, it must pick up all pending transitions exactly where it left off. You should discuss the concept of "at-least-once state transitions" combined with idempotent state handlers: transitioning from ORDER_PLACED to DISPATCHING twice must produce the same result as transitioning once. The interviewer expects you to talk about ordering guarantees: the scheduler must not dispatch a driver before the restaurant has acknowledged the order, even if both timeouts fire close together.

**3. Design a notification system**: Scheduled notifications (reminders, digests, follow-ups) are the core feature. The scheduler must handle millions of scheduled notifications with different delivery times. The interviewer tests whether you understand that most scheduled notifications are flexible by a few seconds (reminder emails) but some require strict timing (one-time passwords). They want to see you separate the two categories with different scheduling tiers. You should propose two tiers: a low-latency tier using an in-memory scheduler for time-sensitive notifications and a high-throughput tier using a database-backed scheduler for batch notifications. The interviewer also probes how you handle notification deduplication: if a user marks a notification as read on one device, the scheduler must cancel the scheduled follow-up on all devices.

**4. Design a social media platform**: Scheduled posts, content expirations, periodic digest emails, and rate-limit cooldowns all require a scheduler. The interviewer probes the mixed workload challenge: scheduled posts require exact execution time, but digest emails have batch character. They expect you to propose different queues or tiers for each workload type. You should discuss how content expiration (for example, stories that disappear after 24 hours) differs from scheduled posts: expiration is a batch operation that affects thousands of posts at once, while scheduled posts are individual operations. The interviewer checks whether you design separate worker pools for batch and individual scheduling to avoid head-of-line blocking.

**5. Design a ride-sharing platform**: Ride timeouts (driver did not accept in 30 seconds), surge pricing recalculation every 5 minutes, and periodic driver location updates. The interviewer tests low-latency scheduling: ride timeouts need sub-second precision because the rider experience degrades rapidly. They want to see a push-based notification path for short-interval tasks and a poll-based path for periodic tasks. You should design a two-path architecture: a fast path using an in-memory timer wheel for 30-second ride timeouts and a slow path using a database-backed scheduler for 5-minute surge recalculations. The interviewer evaluates whether you understand the memory cost of the timer wheel: holding 100,000 concurrent ride timers in memory requires approximately 50 MB, which is acceptable for a single node.

**6. Design a distributed cron service**: The explicit ask: design a cron replacement that works across multiple machines, survives failures, and handles millions of jobs. The interviewer expects you to handle the cron expression parsing, daylight saving time transitions (a cron job scheduled at 2:30 AM may or may not run on the DST change day), and the leap year edge case. You should discuss how you handle the case where a cron job is scheduled for a time that does not exist (spring-forward DST transition) or occurs twice (fall-back DST transition). The standard resolution is to skip non-existent times and run twice for duplicated times, but the interviewer wants to hear that you have considered the problem explicitly.

**7. Design a workflow engine**: Workflow steps have timeouts, delays, and retry schedules. The scheduler component handles all time-based transitions. The interviewer checks whether you realize that timeouts must be cancellable -- a workflow step that completes before its timeout fires must not trigger both the completion handler and the timeout handler. You should explain that the timeout is stored in a cancellable timer data structure: when the step completes, a cancellation message is sent to the scheduler, which removes the timeout from the pending list before it fires. The interviewer also checks whether you handle the race condition between completion and timeout: both operations must be serialized through a per-workflow lock to prevent the timeout from firing moments before the cancellation is processed.

**8. Design a rate limiter**: Rate limit windows reset at specific times. A distributed scheduler manages the window transitions and token bucket refills. The interviewer tests whether you understand that the refill mechanism must be consistent across all rate limiter nodes to avoid a coordinated reset storm. You should propose a staggered refill strategy: each node refills its tokens at a slightly different time within the window, spreading the load on the central counter store. The interviewer also evaluates your understanding of the difference between fixed-window and sliding-window rate limiting, and how the scheduler handles window alignment differently for each.

**9. Design a data pipeline**: Batch processing at scheduled intervals, with dependencies between stages and time-based triggers for data freshness checks. The interviewer evaluates your understanding of DAG-based scheduling and the complexity of handling upstream failures: if stage A fails, all downstream stages in the same batch window must be skipped or retried. You should discuss how the scheduler tracks DAG dependencies: each stage has a list of upstream stages, and the stage runs only when all upstream stages have completed successfully. If a stage fails, downstream stages enter a BLOCKED state and are not scheduled until the upstream issue is resolved. The interviewer checks whether you have considered cascading retries: retrying a failed stage should not retry all downstream stages automatically.

**10. Design a leader election system**: Lease renewal is a scheduling problem. The leader must periodically renew its lease or step down. The interviewer checks whether you understand the relationship between lease duration, renewal interval, and network latency -- a lease too short causes false-positive leader changes; a lease too long increases downtime when the leader fails. You should explain the lease duration formula: lease_duration = 2 * (renewal_interval + max_network_latency). For a 5-second renewal interval and 100ms network latency, the lease duration should be at least 10.2 seconds. The interviewer also checks whether you use a monotonic clock for lease renewal to avoid time-adjustment issues from NTP corrections.

## What the Interviewer Tests

Interviewers have specific evaluation criteria when they hear you propose a distributed scheduler. They are testing these five dimensions. Each dimension includes what a good answer sounds like versus a bad one.

**Dimension 1 -- Correctness**: Can you identify the consistency requirements? Do you understand the difference between at-least-once, at-most-once, and exactly-once? Can you design for idempotent execution?

- **Good**: "This scheduler provides at-least-once execution. Every task carries an idempotency key. The downstream system deduplicates using that key. Together they achieve effectively-once semantics. For payment tasks, the idempotency key is the payment reference plus the attempt number."
- **Bad**: "The scheduler guarantees exactly-once execution because we check the database before running the task." This misses the reality that the check and the execution are not atomic across a crash.

**Dimension 2 -- Reliability**: What happens when a worker crashes? What happens when the schedule index fails? How do you prevent data loss? How do you handle network partitions?

- **Good**: "Workers acquire a lease before executing. The lease expires after 30 seconds. A reclaim scanner runs every 10 seconds and resets expired leases. If the schedule index fails, we fail over to a synchronous replica within 5 seconds. During failover, no new tasks are leased, but in-flight tasks continue on their leases."
- **Bad**: "If the worker crashes, another worker picks up the task." No mechanism described, no timeline, no lease concept.

**Dimension 3 -- Scalability**: How does the scheduler handle 10x the current load? Where are the bottlenecks? How do you shard the schedule index? How do workers scale?

- **Good**: "The schedule index is sharded by task ID hash into 16 shards. Each shard is a Postgres database with read replicas. When load increases, we double the shard count to 32 using consistent hashing. The re-sharding migrates tasks incrementally without downtime. Workers are stateless and auto-scale based on queue depth per shard."
- **Bad**: "We use a single database and add more workers." The database becomes a bottleneck and the interviewer will ask the follow-up question.

**Dimension 4 -- Latency**: What is the trade-off between poll interval and database load? How do you achieve sub-second latency? When is a push-based approach necessary?

- **Good**: "We use a two-tier approach. For latency-sensitive tasks under 10 seconds, we use a push-based notifier that scans the schedule index every 100 milliseconds and sends notifications to workers via an internal message queue. For latency-tolerant tasks above 10 seconds, workers poll every 5 seconds. This keeps the database read load under 10,000 queries per second at the push tier."
- **Bad**: "We poll the database every millisecond for maximum latency." The database will collapse under the query load.

**Dimension 5 -- Operations**: How do you monitor the scheduler? What alerts do you configure? How do you recover from a shard failure? What is the runbook for a stuck scheduler?

- **Good**: "The four critical metrics are queue depth per shard, drain rate, lease expiry rate, and the age of the oldest pending task. We alert if any shard queue depth exceeds 50,000, if drain rate drops below 1,000 per second for 30 seconds, or if the oldest pending task is over 5 minutes late. The runbook for a shard stall is: verify the shard database is healthy, check worker connectivity, and if needed, redistribute pending tasks to another shard within 60 seconds."
- **Bad**: "We monitor the dashboard." No specific metrics, no alert thresholds, no recovery procedure.

## Level-by-Level Expectations

### L3 / Entry-Level

**What you must demonstrate**: You understand the basic concept of a scheduler. You can describe a single-node scheduler with a database-backed task queue. You know the difference between cron and a programmatic scheduler.

**What to prepare**: Implement a simple polling loop. Explain the task state machine (PENDING, RUNNING, COMPLETED, FAILED). Describe what happens when a process crashes.

**Evaluation criteria**: The interviewer checks whether you can articulate the state machine without being prompted. They expect you to name the four states in order and describe the transition conditions. They will ask "What state is a task in when the worker crashes?" -- the correct answer is RUNNING, and the task stays RUNNING until a recovery process resets it. They also expect you to mention that FAILED tasks can be retried, while COMPLETED tasks are terminal unless the business logic requires re-execution.

**Sample question**: "Design a simple task scheduler that runs delayed jobs."

**Sample answer outline**: Use a relational database table with columns for id, trigger_at, status, and payload. A polling thread runs every second, selects tasks where trigger_at is in the past and status is PENDING, locks them with an UPDATE, executes the task logic, and marks them COMPLETED or FAILED. On crash, any task in RUNNING state is detected on restart and reset to PENDING after a timeout.

**What to avoid**: Proposing a distributed architecture. You are not expected to design a multi-shard system at this level.

**Common follow-up at this level**: "What is the maximum throughput of your design?" Your answer should estimate based on the database write capacity: a single Postgres instance can handle approximately 5,000 writes per second, so the maximum throughput is approximately 5,000 tasks per second.

### L4 / Mid-Level

**What you must demonstrate**: You can design a scheduler that runs across multiple machines. You understand the lease mechanism, worker pools, and the need for a durable backing store. You can identify the CAP trade-offs.

**What to prepare**: Describe the lease acquire-execute-release cycle. Compare database-backed vs. Redis-backed schedule indices. Explain sharding by task ID range. Discuss the thundering herd problem and its mitigations.

**Evaluation criteria**: The interviewer expects you to explicitly state that the schedule index is the source of truth and the workers are stateless. They check whether you mention the thundering herd problem without being prompted -- when multiple workers poll for tasks simultaneously, they create a database contention spike. The fix is staggered polling with jitter. They also check whether you understand that caching the schedule index introduces consistency problems: a cached view of pending tasks may be stale, causing the scheduler to miss tasks or double-execute them.

**Sample question**: "Design a distributed cron service that serves 10,000 customers, each with up to 100 scheduled jobs."

**Sample answer outline**: The schedule index stores 1M cron jobs in a Postgres database with read replicas. Workers poll in staggered intervals with 100ms jitter. Each worker acquires a 30-second lease per task. A lease scanner runs every 10 seconds to reclaim expired tasks. Jobs are sharded by customer ID to prevent any single tenant from overwhelming a shard. The cron expression parser runs on task creation to compute the next trigger time, avoiding runtime parsing overhead.

**What to avoid**: Ignoring the worker crash scenario. Not discussing the retry and dead-letter mechanisms.

**Common follow-up at this level**: "How do you prevent the same task from being executed by two workers simultaneously?" Your answer should explain that the lease acquire operation is an atomic compare-and-swap: the worker runs an UPDATE on the task row with a WHERE clause that checks the current status is PENDING. The database row lock ensures only one worker succeeds.

### L5 / Senior

**What you must demonstrate**: You can design a scheduler that handles 1M+ tasks per second with 99.99% reliability. You understand the operational complexity and can propose monitoring and alerting. You know the failure modes and how to design for them.

**What to prepare**: Multi-region active-active scheduling. Graceful degradation under load. Consistency models for the schedule index vs. state store. Priority-based scheduling and preemption. Cost analysis of different scheduling approaches.

**Evaluation criteria**: The interviewer expects you to derive the 99.99% availability requirement into concrete uptime numbers -- that is 52 minutes of downtime per year. They want to hear about multi-region deployment with active-active scheduling, cross-region replication lag, and the consistency impact during a regional failover. They expect a cost calculation: at 1M tasks per second, storing 30 days of task history costs approximately 2.6 TB of storage per day, and cold storage after 90 days reduces the hot storage cost by 70%. They also expect you to discuss graceful degradation: under load, the scheduler should prioritize high-priority tasks over low-priority tasks, and signal backpressure to upstream systems.

**Sample question**: "Design the scheduling backend for a global payment system that processes 1M transactions per hour with retries and timeouts."

**Sample answer outline**: Active-active deployment across three regions with synchronous replication for the schedule index within each region and asynchronous replication between regions. The lease duration is 3x the p99 payment processing time of 500ms, giving a 1.5-second lease. Workers poll a local shard of the schedule index. Each shard handles 20,833 tasks per minute. The retry policy uses exponential backoff: 1 hour, 6 hours, 24 hours, with a maximum of 3 retries. At-least-once delivery with idempotency keys at the payment gateway.

**What to avoid**: Proposing a single-database-backed scheduler. Not addressing the data retention problem. Ignoring the cost implications of your design.

**Common follow-up at this level**: "How do you handle cross-region failover for the schedule index?" Your answer should describe the replication lag between regions (typically 1 to 5 seconds for async replication) and the consistency check on failover: the new primary region must wait for all pending tasks from the old region to be accounted for before accepting new tasks.

### L6 / Staff

**What you must demonstrate**: You can design a scheduler that is the foundation of a platform used by multiple teams. You consider the organizational impact, the API design, and the service-level objectives. You can compare your design with existing solutions (Airflow, Temporal, AWS Scheduler) and explain why your design is better.

**What to prepare**: Multi-tenant isolation with noisy neighbor protection. API versioning and backward compatibility. SLO derivation from business requirements. Cost optimization at scale (50%+ reduction in infrastructure cost). Migration strategy from existing systems.

**Evaluation criteria**: The interviewer expects you to mention at least two existing solutions and give specific reasons why your design addresses their shortcomings. For example: "Temporal provides excellent workflow durability but has a higher per-task overhead of 200 microseconds. For internal teams running 10B events per day, that adds up to 23 days of CPU time per month just for scheduling overhead. Our design uses a lighter-weight task model with 20-microsecond overhead, at the cost of weaker workflow guarantees." They also look for evidence that you have considered the developer experience, such as providing SDKs in multiple languages and a web UI for debugging.

**Sample question**: "Design a scheduling platform to be used by 50 internal teams at a company processing 10B events per day."

**Sample answer outline**: Each team gets a dedicated shard with configurable capacity limits. The API supports versioning from day one: /v1/tasks, /v2/tasks, with a deprecation policy of 18 months. The SLO is 99.95% of tasks must start execution within 60 seconds of their scheduled time, derived from a business requirement that no team experiences more than 4 hours of delayed task starts per year. Cost optimization: use spot instances for workers with a 90% cost reduction compared to on-demand, with a fallback to on-demand when spot is unavailable. Migration strategy: provide a compatibility layer that accepts Airflow DAG definitions and translates them into the platform-native scheduling model.

**What to avoid**: Designing for a single use case. Ignoring the migration path from existing systems. Not considering the developer experience of the API.

**Common follow-up at this level**: "How do you isolate noisy tenants?" Your answer should describe per-tenant rate limiting at the API gateway, per-shard capacity quotas, and a scheduler-level fairness mechanism: if one tenant submits 10M tasks and another submits 10 tasks, the scheduler allocates a minimum share of worker capacity to the second tenant to ensure it is not starved.

### L7-L8 / Principal and Above

**What you must demonstrate**: You can design a system that sets the industry standard. You consider the long-term evolution of the architecture, including the impact of new hardware (persistent memory, disaggregated storage) and new software paradigms (serverless, edge computing). You can articulate the unsolved problems in distributed scheduling.

**What to prepare**: The future of scheduling: global consistency with low latency using new consensus protocols. The economics of scheduling: when does it make sense to build vs. buy. The human factors: how the scheduler design affects developer productivity and incident response time.

**Evaluation criteria**: The interviewer expects you to identify at least two unsolved problems in the space. Examples: "No existing distributed scheduler provides both global consistency and sub-100ms latency at planetary scale. The fundamental barrier is the speed of light -- synchronously replicating state across regions costs a minimum of 50-100ms of latency. We solve this by using a local-first architecture where each region is fully autonomous and reconciliation happens asynchronously." They also expect a build-versus-buy analysis: "For a company processing fewer than 100K tasks per day, buying AWS Scheduler or using Temporal Cloud is the correct choice. Our build investment is justified only above 10M tasks per day where the per-task cost savings of 0.0001 cents per execution saves 1M dollars per year."

**Sample question**: "Design the next-generation distributed scheduler for a cloud provider. It must support 10M customers with sub-100ms latency and 99.999% availability."

**Sample answer outline**: The design uses a two-layer architecture: a fast path for sub-second tasks and a durable path for everything else. The fast path uses in-memory replicated state machines with Raft consensus across three nodes per shard. The durable path writes tasks to a distributed log (Kafka-style) and consumes from it. Each customer gets a dedicated micro-shard. The scheduling algorithm uses a calendar queue data structure in memory for O(1) task insertion and O(1) task retrieval. The system is designed for serverless deployment -- workers spin up on demand per task type, with cold start mitigated by maintaining a warm pool of 1,000 pre-warmed workers.

**What to avoid**: A design that is incrementally better than existing solutions. At this level, the solution must be a step change in capability or cost.

**Common follow-up at this level**: "How does your design evolve over 5 years?" Your answer should describe the roadmap: year one, establish the core two-layer architecture with 100 shards; year two, add multi-region active-active with autonomous per-region schedulers; year three, introduce serverless worker execution with sub-second cold start; year four, add global task routing based on data locality; year five, add predictive scheduling using machine learning to pre-allocate workers based on historical load patterns.

## FAANG Framework

Use this 5-step framework when you encounter a scheduling problem in an interview. It covers the key dimensions that interviewers evaluate.

### Step 1: Requirements Clarification

Ask these questions before starting the design. Each question has a reason behind it:

- What is the expected throughput? (tasks per second) -- This determines whether a single-node scheduler is sufficient or you need sharding.
- What is the latency requirement? (p99, p999) -- This determines whether you need a push-based or poll-based architecture.
- What is the durability requirement? (acceptable data loss?) -- This determines the replication strategy and the consistency model.
- What is the execution semantics? (at-least-once, exactly-once, at-most-once) -- This determines whether you need idempotency keys and deduplication.
- Are tasks idempotent? (can the same task execute twice?) -- If not, you must design for at-most-once execution, which is substantially harder.
- What is the expected task duration? (short, long, mixed) -- Short tasks need fast lease cycles; long tasks need heartbeat mechanisms.
- How many tenants? (single tenant, multi-tenant) -- Multi-tenant needs isolation, rate limiting, and noisy neighbor protection.
- What is the deployment topology? (single region, multi-region) -- Multi-region adds replication lag, clock skew, and failover complexity.

**Sample dialogue**: "Let me confirm the requirements. You have 500,000 tasks per second with a p99 latency of 1 second. Tasks are idempotent and you need at-least-once delivery. Task duration varies from 100 milliseconds to 5 minutes. This is a single-tenant system deployed in two regions for disaster recovery. Is that correct?"

### Step 2: Core Components

Identify the essential components and explain how they interact:

- **Schedule Index**: Data structure for time-ordered tasks (sorted set, priority queue, database table). This is the heart of the system. The choice of data structure determines the scheduling throughput. A sorted set in Redis gives O(log N) insertion and O(1) range queries. A database table with an index on trigger_at gives O(log N) for both but adds durability.
- **Lease Manager**: Mechanism for task claiming and expiry. This is the fault-tolerance backbone. Without leases, a worker crash loses the task forever. The lease manager must handle clock skew, network delays, and the race condition between lease expiry and task completion.
- **Worker Pool**: Execution capacity. Workers pull tasks from the schedule index, execute them, and update the task state. Workers must be stateless -- they hold no authoritative state. The worker pool autoscales based on queue depth.
- **State Store**: Durable task metadata. This stores the current state of every task. It must support fast reads (workers check task status) and fast writes (workers update task state on completion).
- **API Gateway**: Enqueue, cancel, status operations. This is the entry point for external systems. It must validate input, authenticate requests, and route operations to the correct shard.

**Sample dialogue**: "The system has five core components. The schedule index is a Postgres table with a composite index on (trigger_at, status). Each shard owns one Postgres database. The lease manager is a background process that runs inside each shard coordinator. Workers are a Kubernetes deployment that autoscales based on the queue depth metric exposed by each shard. The state store is the same Postgres database as the schedule index -- they share the table. The API Gateway is a stateless HTTP service that hashes the task ID to determine the target shard."

**Interaction between components**: The API gateway receives a new task, hashes the task ID to find the correct shard, and writes the task to that shard schedule index. The lease manager in the shard coordinator periodically scans the schedule index for tasks whose trigger_at is in the past and status is PENDING. When it finds such tasks, it atomically updates the status to LEASED and assigns a worker ID. The worker polls the lease manager, receives the task, executes it, and updates the state store to COMPLETED or FAILED. The reclaim scanner continuously checks for expired leases and resets LEASED tasks back to PENDING.

### Step 3: Data Model

Define the task schema with an explanation for each field:

```
task {
  id: UUID,               -- unique identifier, used for shard routing and idempotency
  task_type: string,       -- determines the handler code path
  payload: bytes,          -- serialized task parameters (JSON, Protobuf, Avro)
  trigger_at: timestamp,   -- the earliest time this task can execute
  status: enum (PENDING, LEASED, RUNNING, COMPLETED, FAILED, DEAD_LETTER),
  worker_id: string,       -- which worker holds the lease (null if unclaimed)
  lease_until: timestamp,  -- when the lease expires (null if unclaimed)
  retry_count: int,        -- how many times this task has been retried
  max_retries: int,        -- maximum retries before moving to dead-letter
  next_retry_at: timestamp, -- scheduled time for the next retry
  created_at: timestamp,   -- when the task was created
  updated_at: timestamp    -- last modification time, used for optimistic locking
}
```

**Index strategy**: The primary index is a composite B-tree index on (status, trigger_at). This supports the primary query: tasks with status in (PENDING, FAILED) ordered by trigger_at ASC. A secondary index on (worker_id, status) supports queries for tasks currently assigned to a specific worker. A tertiary index on (id) supports point lookups for task status checks.

**Sample dialogue**: "The task schema is designed for two access patterns. The primary read is a range scan on (status, trigger_at): select tasks where status in (PENDING, FAILED) and trigger_at is before now, ordered by trigger_at. The primary write is an update by id with an optimistic lock on updated_at: update tasks set status = LEASED where id = X and updated_at = previous_value."

### Step 4: Consistency and Fault Tolerance

Address these scenarios with specific mechanisms:

- **Worker crash during execution**: The lease expires after 30 seconds. A reclaim scanner identifies expired leases and resets the task to PENDING. The retry count increments. If the task has exhausted retries, it moves to DEAD_LETTER.
- **Schedule index database failure**: The primary database fails over to a synchronous replica within 5 seconds. During failover, new tasks are queued in a buffer (Kafka topic) and replayed after the failover completes. In-flight workers complete their current tasks and report results after the failover.
- **Network partition between workers and index**: Workers continue executing tasks for the duration of their lease (30 seconds). If the lease expires without a refresh, the task is reclaimed. After the partition heals, workers must reconcile: execute status queries for the tasks they completed during the partition and update the state store accordingly.
- **Clock skew across machines**: All lease expiry comparisons use the schedule index clock, not the worker clock. Workers call a `now()` function provided by the schedule index. This eliminates clock skew as a source of correctness bugs.
- **Data corruption in the schedule index**: Each task row has a checksum field. A background integrity scanner reads all tasks, recomputes the checksum, and reports mismatches. Corrupted tasks are recreated from the write-ahead log.
- **Runaway task that never completes**: Each task has a maximum execution time. If the task exceeds this limit, the worker process is killed and the lease is reclaimed. A sentinel process monitors overall task duration and hard-kills workers that exceed the maximum.

**Sample dialogue**: "Fault tolerance is built on three layers. Layer one is the lease: if a worker dies, the lease expires and the task is reclaimed. Layer two is the retry policy with exponential backoff: transient failures are retried, and permanent failures are sent to a dead-letter queue. Layer three is the runbook automation: a health monitor detects shard failures, redistributes pending tasks, and initiates database failover when needed."

### Step 5: Scaling and Operations

Describe how the system handles growth and operational scenarios:

- **10x traffic increase**: Double the number of shards using consistent hashing. The re-sharding process runs in the background over 8 hours. Workers auto-scale based on queue depth. Read replicas per shard are added to handle the increased query volume.
- **Single shard failure**: The remaining shards rebalance by taking over the failed shard task ID range. The rebalancing is automated: a coordinator detects the failure, updates the shard mapping, and redistributes pending tasks within 60 seconds.
- **Worker fleet failure**: A large-scale worker failure (for example, an entire Kubernetes node pool goes down) triggers a rapid re-lease cycle. The reclaim scanner detects the expired leases and resets them. New workers spin up within 2 minutes and immediately start claiming tasks.
- **Data volume growth (task history)**: Completed tasks are archived to cold storage (S3, GCS) after 30 days. The archive is partitioned by month for efficient query. Dead-letter tasks are deleted after 90 days. The cold storage cost is approximately 0.1 cents per GB per month.
- **Monitoring and alerting**: The four critical metrics are queue depth per shard, drain rate in tasks per second, lease expiry rate, and the age of the oldest pending task. Alerts fire when any shard queue depth exceeds 50,000 or when the drain rate drops below 10% of the target for 30 seconds.
- **Backup and restore**: The schedule index is backed up every 6 hours. Point-in-time recovery is supported via write-ahead log replay. The recovery time objective is 15 minutes. The recovery point objective is 5 minutes (up to 5 minutes of task data may be lost in a catastrophic failure).

**Sample dialogue**: "For scaling, the key insight is that sharding the schedule index removes the bottleneck. Each shard handles 50,000 tasks per second. When throughput reaches 500,000 tasks per second, we add 10 shards for a total of 20. For operations, every shard exposes a health endpoint that reports queue depth, drain rate, and the timestamp of the last successful lease cycle. The operations dashboard shows these metrics for every shard in a single view."

## Phrase Bank

Use these phrases to articulate your design decisions clearly and confidently.

**Opening the problem**:
- "A distributed scheduler must separate the clock from the hands. The schedule index is the clock, and the workers are the hands."
- "The core tension is between latency, throughput, and exactly-once execution. You can pick any two."
- "The fundamental challenge is coordination: multiple workers must agree on which task each worker owns."
- "The scheduler is the heartbeat of the system. If it stalls, nothing else moves."
- "Every scheduling problem reduces to a time-ordered list and a polling loop. The complexity comes from making both distributed and reliable."

**Explaining the lease mechanism**:
- "The lease is a time-limited lock. The worker holds the lease while executing. If the lease expires, the task is available for reclaim."
- "Lease duration must be set to 3x the p99 execution time to prevent false-positive reclaims."
- "The reclaim scanner is the safety net. It detects expired leases and resets tasks to PENDING."
- "A heartbeat mechanism extends the lease for long-running tasks. Without it, the task is reclaimed before it completes."
- "The lease acquire operation must be atomic: compare-and-swap on the task row, or a LUA script in Redis."

**Addressing consistency**:
- "No distributed scheduler can guarantee exactly-once execution. The Two Generals Problem applies. We design for at-least-once and make tasks idempotent."
- "The schedule index must be CP. The state store can be AP. Losing a pending task is catastrophic. Losing a completed task record is acceptable."
- "We use an authoritative clock source for all time comparisons to avoid clock skew issues."
- "The atomic unit of work is a single task. We never batch-claim tasks because a partial batch failure creates an inconsistent state."

## Red Flags

Interviewers watch for these mistakes. Avoid them.

**Red Flag 1 -- Ignoring worker crashes**: You describe a design where tasks are claimed but never mention what happens if the claiming worker dies. The interviewer immediately knows you have not thought about fault tolerance.

**Fix**: Always mention the lease mechanism, lease expiry, and the reclaim scanner. Describe the timeline: worker crashes, lease expires in 30 seconds, reclaim scanner resets the task, another worker picks it up.

**Red Flag 2 -- Proposing exactly-once execution without idempotency**: You claim the scheduler guarantees exactly-once execution without explaining how. The interviewer knows that exactly-once is impossible in distributed systems without downstream support.

**Fix**: Say "We provide at-least-once execution from the scheduler side. The downstream system must implement idempotent handling using an idempotency key. Together, they achieve effectively-once semantics."

**Red Flag 3 -- Single point of failure in the schedule index**: You propose a single database as the schedule index without replication or sharding. The interviewer asks: "What happens when this database fails?"

**Fix**: Describe the database replication strategy (synchronous replication, Raft or Paxos consensus). Explain how the system fails over to a replica. If the throughput requires it, describe sharding.

**Red Flag 4 -- No retry policy**: You assume tasks always succeed on the first attempt. The interviewer knows that in production, 5% to 10% of tasks fail due to transient errors.

**Fix**: Describe the retry policy with exponential backoff and jitter. Explain the maximum retry count and the dead-letter queue. Discuss the trade-off between retry frequency and downstream system load.

**Red Flag 5 -- Poll interval set to zero**: You propose polling every millisecond to minimize latency, ignoring the database load this creates.

**Fix**: Explain the trade-off between poll interval and database load. Propose a reasonable poll interval (500ms to 1s) combined with push notifications for low-latency requirements.

**Red Flag 6 -- No data retention policy**: You store all completed tasks forever. The interviewer asks about the storage cost at 10,000 tasks per second.

**Fix**: Describe the data retention policy: archive completed tasks to cold storage after 30 days, delete dead-letter tasks after 90 days. Explain the storage cost calculation.

**Red Flag 7 -- No backpressure mechanism**: You accept tasks at any rate, ignoring the scheduler capacity limits. The interviewer asks what happens during a traffic spike.

**Fix**: Describe the backpressure mechanism: per-shard token buckets, HTTP 429 responses with Retry-After headers, message queue buffering, and admission control at the API gateway.

**Red Flag 8 -- Assuming monotonic clocks are reliable**: You rely on local clock time for lease expiration, ignoring clock skew between machines.

**Fix**: Use an authoritative clock source (database NOW() function, consensus-based monotonic clock). Workers should compare lease expiry against the schedule index clock, not their local clock.

**Red Flag 9 -- Designing without an SLA**: You present a design but never state the throughput, latency, or availability you are targeting. The interviewer cannot evaluate whether your design meets the requirements.

**Fix**: State numeric targets explicitly: "This design handles 500,000 tasks per second with p99 latency under 1 second and 99.99% availability." Then describe how each component supports those numbers.

**Red Flag 10 -- No monitoring or observability**: You design the system but never mention how you would detect failures, measure performance, or diagnose issues.

**Fix**: Describe the key metrics, the alerting thresholds, and the dashboard layout. Mention the smoke test task that runs every minute and triggers an alert if it does not complete on time.

## Common Follow-up Questions

Interviewers often ask these follow-up questions after you complete the initial design. Prepare your answers in advance.

**Q1: How do you handle a thundering herd when all workers poll the schedule index simultaneously?**

**A**: You introduce staggered polling with jitter. Each worker waits a random offset before polling. The offset is between 0 and the poll interval. For example, with a 1-second poll interval, each worker waits a random number of milliseconds between 0 and 1,000 before querying the database. This spreads the database queries evenly across the interval. The database query itself should use pagination with a LIMIT clause and fetch only a small batch of tasks (100 to 1,000) per poll cycle. At the database level, you can also use advisory locks to serialize concurrent lease acquisitions for the same task ID range.

**Q2: What happens when the schedule index database is overloaded?**

**A**: You have three options depending on the severity. Option one: increase the poll interval from 500ms to 2s to reduce query volume by 75%. Option two: add read replicas to handle the poll query load while keeping writes on the primary. Option three: if the overload is persistent, increase the shard count to distribute the load across more database instances. The decision is driven by the database CPU and connection count metrics. In extreme cases, you can activate admission control at the API gateway to limit the rate of new task submissions until the backlog clears.

**Q3: How do you test a distributed scheduler?**

**A**: You use three testing layers. Layer one is unit tests for the task state machine, lease logic, and retry policy. Layer two is integration tests with a real database, testing the polling loop, lease acquisition, and reclaim scanner. Layer three is chaos engineering: you kill workers mid-execution, partition workers from the database, and inject clock skew to verify the fault-tolerance mechanisms. A simulation framework generates random task schedules and verifies that every task completes or moves to the dead-letter queue within a bounded time. You also run load tests to verify the throughput and latency targets: ramp up from 1,000 to 500,000 tasks per second and measure p50 and p99 scheduling delay at each step.

**Q4: How do you handle tasks that are scheduled years in advance?**

**A**: You separate the index into two tiers: a fast tier for tasks due within the next 24 hours and a cold tier for tasks due after 24 hours. A promotion job runs every hour and moves tasks from the cold tier to the fast tier. The fast tier is stored in memory or a high-performance database. The cold tier is stored in a cheaper database or object storage. This keeps the hot working set small and the poll queries fast. The promotion job must be resilient to crashes: if the job fails, it resumes from the last checkpoint without duplicating tasks. The checkpoint stores the last promoted task ID and timestamp, and the promotion query uses those as the starting point for the next run.

**Q5: How do you ensure the scheduler does not become a single point of failure?**

**A**: Each shard of the schedule index is replicated with a primary-replica configuration using synchronous replication. If the primary fails, a replica is promoted within 5 seconds. The shard coordinator process runs as an active-passive pair: the active coordinator handles lease management, and the passive coordinator takes over if the active fails, detected by a lease on the coordinator role itself. The API gateway is stateless and runs behind a load balancer with health checks. The reclaim scanner runs on both coordinator instances but only the active coordinator performs reclaim operations -- the passive coordinator runs in dry-run mode and takes over when promoted.

**Q6: How do you measure and enforce the SLA?**

**A**: Every task records its scheduled time and its actual start time. The difference is the scheduling delay. You track the p50, p99, and p999 of scheduling delay across all tasks. The SLA (for example, 95% of tasks start within 60 seconds of the scheduled time) is enforced by alerting when the p95 delay exceeds 60 seconds over a 5-minute window. The SLA also accounts for the schedule index being unavailable: if the primary database fails, tasks that missed their scheduled time during the failover are counted against the SLA, and the failover time is reported in the monthly SLA report. You publish an SLA dashboard that shows the rolling 7-day and 30-day compliance against each SLO metric.

## Architecture Decision Record

When interviewers ask why you chose one approach over another, use these decision rationales.

**Decision: Database-backed schedule index vs. in-memory schedule index**

- Database-backed: Use when durability is critical. Tasks survive worker crashes, database restarts, and full system failures. Throughput is limited by database write capacity (approximately 5,000 to 10,000 writes per second per instance).
- In-memory (Redis): Use when latency is critical and some data loss is acceptable. Throughput is higher (100,000+ operations per second per node) but data is lost if the Redis node fails without persistence enabled.
- Hybrid: Use when both latency and durability matter. Use Redis for fast polling and persist tasks to Postgres for durability. This adds complexity because the two stores must stay consistent.

**Decision: Poll-based vs. push-based task distribution**

- Poll-based: Workers periodically query the schedule index for due tasks. Simpler to implement. Database load increases with poll frequency. Best for latency-tolerant tasks (more than 10 seconds).
- Push-based: The schedule index notifies workers when tasks are due (via message queue or HTTP callback). Lower latency. More complex to implement because notifications must be reliable. Best for latency-sensitive tasks (less than 1 second).
- Hybrid: Use push for latency-sensitive tasks and poll for latency-tolerant tasks. This is the most common production approach.

**Decision: Sharding by task ID vs. sharding by tenant**

- Task ID sharding: Uniform distribution across shards. No single shard is overloaded. But a single tenant tasks are spread across all shards, making per-tenant queries expensive.
- Tenant sharding: All tasks for one tenant are on the same shard. Per-tenant queries are efficient. But one large tenant can overload a single shard.
- Hybrid: Shard by tenant with per-tenant capacity limits and overflow shards for large tenants. More complex but provides both isolation and scalability.

## Full Worked Example

Walk through this complete example to see how the FAANG framework applies to a real interview problem.

**Problem**: Design the scheduling layer for a food delivery platform that operates in 50 cities across 5 countries. The platform processes 2M orders per day. Each order goes through 6 timed states: order placed (immediate), restaurant notified (immediate), expected prep time (configurable per restaurant, average 15 minutes), dispatch driver (must find a driver within 5 minutes of food being ready), delivery in progress (average 25 minutes), and order delivered (terminal). If any state timeout is exceeded, the system must escalate and compensate the customer.

**Step 1 -- Requirements Clarification**:
- Throughput: 2M orders per day equals approximately 23 orders per second average, with peaks at meal times of approximately 200 orders per second.
- Latency: Restaurant notification must happen within 5 seconds. Driver dispatch must complete within 30 seconds of food-ready signal or escalate.
- Tasks per order: Each order generates 4 to 6 timed tasks (restaurant notify timeout, food-ready dispatch deadline, delivery progress check, escalation). Total: approximately 100 tasks per second average, 1,000 tasks per second at peak.
- Task duration: Most tasks fire within 30 minutes of order creation. The longest task is delivery progress check at 60 minutes.
- Execution semantics: At-least-once. If a timeout fires twice, the state machine must handle the duplicate transition idempotently.
- Fault tolerance: Losing a timeout means a customer does not receive their food or does not receive a refund. Zero data loss is required.

**Step 2 -- Core Components**:
- Schedule index: Postgres sharded by city ID. Each city is a separate shard because there is no cross-city ordering coordination. The shard count is 50, one per city.
- Lease manager: Built into the shard coordinator. Lease duration is 3x the p99 task execution time of 200ms, so 600ms.
- Worker pool: A Kubernetes deployment with 10 worker pods per city. Workers autoscale to 50 pods during peak meal hours (11 AM to 1 PM and 6 PM to 9 PM local time).
- State store: The same Postgres instance as the schedule index. The order state machine and the task schedule coexist in the same database per city.
- API gateway: Routes order lifecycle events to the correct city shard based on the order's city ID.

**Step 3 -- Data Model**:
- The task table follows the standard schema with one addition: a `correlation_id` field links the task back to the order ID. This allows the escalation handler to look up all pending tasks for a single order and cancel them when the order completes.
- The trigger_at field is computed from the order creation time plus the restaurant's configured prep time. For restaurants with variable prep time, the trigger_at is updated dynamically when the restaurant reports a change.

**Step 4 -- Consistency and Fault Tolerance**:
- If a worker crashes mid-task, the 600ms lease expires and the task is reclaimed by another worker within 1 second. The task retry count increments, but this is safe because timeout tasks are idempotent.
- If the Postgres shard for a city fails, all orders in that city are affected. The failover to a read replica happens within 5 seconds. During failover, the API gateway holds new orders for that city in a buffer. After failover, the buffered orders are submitted.
- If a city has cascading failures (database down, workers unreachable), a global escalation triggers: all pending orders in that city are flagged for manual review by the regional operations team.

**Step 5 -- Scaling and Operations**:
- Each city shard handles 20 to 200 tasks per second. A single Postgres instance can handle this comfortably. As the platform expands to new cities, a new shard is added per city.
- The key metric is the "late task ratio": the percentage of tasks that miss their scheduled time by more than 30 seconds. The alert threshold is 0.5% over a 5-minute window.
- The escalation runbook: when a food-ready task fires but no driver has been dispatched, the escalation system issues a customer compensation credit and notifies the restaurant. The scheduler tracks the escalation state to ensure no duplicate compensation.

This worked example demonstrates how to take the abstract framework and apply it to a concrete business scenario with specific numbers and operational details.

## Storage Cost Estimation Reference

Interviewers frequently ask about storage costs. Memorize these reference numbers:

- One task row with metadata and payload: approximately 1 KB with metadata, 10 KB to 100 KB with payload.
- At 100,000 tasks per second: 100 MB per second in raw data, 8.6 TB per day, 260 TB per month.
- At 1,000,000 tasks per second: 1 GB per second in raw data, 86 TB per day, 2.6 PB per month.
- Postgres storage cost: approximately 0.10 to 0.25 dollars per GB per month for managed cloud databases.
- S3 cold storage cost: approximately 0.01 dollars per GB per month.
- Savings from archiving: moving completed tasks to cold storage reduces hot storage costs by 90%.
- Index overhead: B-tree indexes add 20% to 50% additional storage on top of the raw data.

You can estimate storage cost on the fly: throughput in tasks per second times 1 KB gives you the write rate. Multiply by 86,400 seconds per day and then by the number of days you retain data in hot storage. Multiply by the storage cost per GB.

For example: 500,000 tasks per second, 30-day retention, 1 KB per row. Total: 500,000 times 86,400 times 30 equals 1.3 trillion rows. At 1 KB each, that is 1.3 PB. At 0.15 dollars per GB per month, hot storage costs 195,000 dollars per month. Archiving to S3 after 30 days at 0.01 dollars per GB reduces the hot storage to the most recent 30 days and the cold storage to the archive, saving approximately 175,000 dollars per month.

**Interviewer**: "How do you monitor this system?"

## Edge Cases to Discuss

Interviewers love edge cases. Prepare these to demonstrate thoroughness.

**Edge case: Task scheduled for February 29 on a non-leap year**: The task creation API must validate the trigger_at date. If a task is created with trigger_at of February 29, 2025 (non-leap year), the API should reject it. For recurring tasks, the scheduler should compute the next valid date: a yearly task scheduled for February 29 runs on February 28 in non-leap years or on March 1, depending on the business requirement.

**Edge case: Task trigger_at is in the past at creation time**: The API should accept the task and execute it as soon as possible. The scheduler should not reject tasks with past trigger_at because this can happen due to network delays. The task should be available for immediate pickup by the next poll cycle.

**Edge case: Two tasks scheduled for the exact same millisecond**: The scheduler must define a deterministic ordering. Use the task ID as a tiebreaker. The task with the lower UUID (lexicographic order) executes first. This prevents non-deterministic behavior in dependent task chains.

**Edge case: Worker completes a task but the database write fails**: The worker retries the status update. If the update fails after 3 retries, the worker writes the completion event to a local log and terminates. A recovery process reads these local logs and reconciles the task status. The task may appear PENDING in the schedule index during the reconciliation window, leading to a duplicate execution -- which is safe because tasks are idempotent.

**Edge case: The schedule index database runs out of disk space**: The monitoring alert fires when disk usage exceeds 80%. If the disk fills up completely, the database rejects writes but still accepts reads. Workers cannot update task status, so all tasks remain LEASED indefinitely. The recovery procedure: delete archived tasks, extend the disk, and restart the reclaim scanner to reset all LEASED tasks that were orphaned during the outage.

**Edge case: A network partition isolates a subset of workers from the schedule index but not from downstream systems**: Workers continue executing tasks from their local cache and calling downstream APIs. When the partition heals, the workers must reconcile: they query the schedule index for all tasks they completed during the partition and verify that the task status is COMPLETED. If any task is still PENDING (because the lease expired and the task was reclaimed), the worker must re-execute it or report the discrepancy for manual review.
