## How to Use These Challenges

Each challenge has a time limit, evaluation criteria, and a model answer. Attempt the challenge within the time limit before reading the model answer. The challenges progress from foundation (5 minutes) through design (15 minutes) to expert (30 minutes).

Passing all challenges at a tier means you have mastered that level. Move to the next tier only after you consistently pass.

---

## Tier 1: Foundation

### Challenge 1.1: Explain the Bulkhead Pattern (5 minutes)

**Prompt.** Write a 250-word explanation of the bulkhead pattern. Include the definition, the analogy, and the two main implementation variants. Use no jargon beyond the terms defined in the pattern. Your audience is a junior engineer who has heard the term but does not understand it.

**Time limit.** 5 minutes.

**Evaluation criteria.** The explanation defines the bulkhead pattern correctly. It uses the ship compartment analogy. It mentions thread pool isolation and semaphore isolation as the two main variants. It explains that the pattern prevents cascading failures by isolating resources.

**Common mistakes.** Confusing bulkheads with circuit breakers. Not mentioning the ship compartment analogy. Using implementation-specific jargon.

**Model answer.** The bulkhead pattern is a resource isolation strategy. It prevents a single slow or failing service from exhausting all system resources. The name comes from shipbuilding: ships have watertight compartments called bulkheads. If one compartment is breached, the water stays in that compartment. The ship stays afloat. In software, the pattern works the same way. Instead of a single shared thread pool for all downstream services, you create separate pools. Each pool is assigned to a specific service. If that service slows down, it only fills its own pool. The other pools remain available. There are two main implementations. Thread pool isolation creates a separate thread pool for each service. This provides strong isolation but uses more memory. Semaphore isolation uses a lightweight counter instead of a thread pool. This is more efficient but does not provide timeout isolation. The choice between them depends on the service's latency profile.

### Challenge 1.2: Identify the Isolation Requirement (5 minutes)

**Prompt.** For each system, decide whether bulkhead isolation is needed and which variant (thread pool or semaphore) is appropriate. Justify each decision in one sentence.

Systems: a) A notification service that sends emails with 50ms average latency, b) A payment gateway with 500ms p99 latency and occasional 5-second timeouts, c) A cache read service with 2ms average latency, d) A video transcoding service with 30-second average processing time, e) A user authentication service with 20ms average latency and 99.99% uptime.

**Time limit.** 5 minutes.

**Evaluation criteria.** Each choice is correct and justified. Notification service: semaphore (fast, consistent latency). Payment gateway: thread pool (slow, variable latency). Cache read: semaphore (very fast, consistent). Video transcoding: thread pool (very slow, long-running). Authentication: semaphore (fast, reliable).

**Common mistakes.** Choosing thread pool for all services without considering the overhead. Choosing semaphore for the payment gateway, which needs timeout isolation. Not providing justification.

**Model answer.** Notification service: semaphore isolation, because 50ms latency is fast and consistent. Payment gateway: thread pool isolation, because 500ms p99 with occasional 5-second timeouts requires timeout isolation. Cache read: semaphore isolation, because 2ms latency is extremely fast and the thread switching overhead of a thread pool would be significant. Video transcoding: thread pool isolation, because 30-second processing time means threads are held for extended periods and must not block the calling thread. Authentication: semaphore isolation, because 20ms latency with 99.99% uptime means the risk of resource exhaustion is minimal.

### Challenge 1.3: Pool Sizing Math (5 minutes)

**Prompt.** A downstream service handles 200 requests per second with an average latency of 250 milliseconds. Calculate the following:

a) The minimum number of threads needed for the pool.
b) The recommended thread pool size with a safety margin of 2.0.
c) The queue capacity for a maximum acceptable wait time of 3 seconds.
d) The memory overhead at 256 KB per thread stack.

**Time limit.** 5 minutes.

**Evaluation criteria.** Correct application of Little's Law. Correct safety margin calculation. Correct queue capacity calculation. Correct memory calculation.

**Model answer.** a) Minimum threads = 200 * 0.25 = 50 threads. b) Recommended pool size = 50 * 2.0 = 100 threads. c) Queue capacity = 3 * 200 = 600 slots. d) Memory overhead = 100 * 256 KB = 25,600 KB = 25 MB. The total memory overhead for the thread pool is 25 MB. The queue memory overhead is 600 * 80 bytes = 48 KB, which is negligible.

**Common mistakes.** Forgetting the safety margin. Using the wrong units (milliseconds vs seconds). Confusing queue capacity with pool size. Not calculating the memory overhead.

## Tier 2: Design

### Challenge 2.1: Multi-Service Bulkhead Design (15 minutes)

**Prompt.** You are designing a checkout service for an e-commerce platform. The service handles 500 transactions per second. It calls five downstream services:

- Inventory service: 200 req/s, 100ms average latency, 2-second p99.
- Pricing service: 500 req/s, 50ms average latency, 200ms p99.
- Payment gateway: 500 req/s, 500ms average latency, 5-second p99.
- Shipping service: 100 req/s, 200ms average latency, 1-second p99.
- Notification service: 500 req/s, 30ms average latency, 100ms p99.

Design the bulkhead architecture. Specify the pool size, queue capacity, isolation variant, and fallback strategy for each service. Calculate the total thread count and memory overhead.

**Time limit.** 15 minutes.

**Evaluation criteria.** Correct pool sizing for each service. Appropriate isolation variant selection. Reasonable fallback strategies. Correct total thread count and memory overhead calculation. Justification for each decision.

**Model answer.** Inventory service: thread pool isolation. Pool size = 200 * 0.1 * 2.0 = 40 threads. Queue = 2 * 200 = 400 slots. Fallback: return "in stock" (degraded). Pricing service: semaphore isolation. Permits = 500 * 0.05 * 2.0 = 50. No queue. Fallback: return cached price. Payment gateway: thread pool isolation. Pool size = 500 * 0.5 * 2.0 = 500 threads. Queue = 3 * 500 = 1,500 slots. Fallback: return "payment pending" and retry asynchronously. Shipping service: thread pool isolation. Pool size = 100 * 0.2 * 2.0 = 40 threads. Queue = 2 * 100 = 200 slots. Fallback: return standard shipping estimate. Notification service: semaphore isolation. Permits = 500 * 0.03 * 2.0 = 30. No queue. Fallback: discard notification (non-critical).

Total threads: 40 (inventory) + 500 (payment) + 40 (shipping) = 580 threads. Thread pool memory overhead: 580 * 256 KB = 145 MB. Semaphore memory overhead: negligible. Total memory: approximately 150 MB. This is acceptable for a server with 4 GB of RAM.

**Common mistakes.** Using thread pool for all services, which would add 150 MB of overhead for pricing and notification services that are fast and reliable. Not providing fallback strategies. Sizing pools without considering the safety margin.

### Challenge 2.2: Multi-Tenant SaaS Bulkhead (15 minutes)

**Prompt.** You are designing a multi-tenant SaaS platform. The platform has three tiers: Free (10,000 customers), Standard (1,000 customers), and Premium (100 customers). Each tier has different throughput and latency requirements.

- Free tier: 10 req/s per customer, 100ms average latency, 500ms p99.
- Standard tier: 100 req/s per customer, 50ms average latency, 200ms p99.
- Premium tier: 500 req/s per customer, 20ms average latency, 100ms p99.

Design a bulkhead architecture that prevents one tenant from affecting other tenants. Specify the pool sizes, isolation strategy, and fallback for each tier. Include the total resource usage.

**Time limit.** 15 minutes.

**Evaluation criteria.** Correct identification of per-tenant vs per-tier isolation. Appropriate pool sizing for each tier. Consideration of the noisy neighbor problem. Total resource usage calculation.

**Model answer.** I will use per-tier thread pools with per-tenant semaphore sub-partitions. Each tier has a dedicated thread pool. Within each pool, tenants are limited by semaphores.

Free tier: 10,000 customers * 10 req/s = 100,000 req/s total. Pool size = 100,000 * 0.1 * 2.0 = 20,000 threads. This is too many. I will use 100 instances, each handling 1,000 req/s. Per-instance pool size = 1,000 * 0.1 * 2.0 = 200 threads. Per-tenant semaphore: 10 * 0.1 * 2.0 = 2 permits. This means one free tenant can use at most 2 concurrent calls per instance.

Standard tier: 1,000 customers * 100 req/s = 100,000 req/s total. 100 instances, each handling 1,000 req/s. Per-instance pool size = 1,000 * 0.05 * 2.0 = 100 threads. Per-tenant semaphore: 100 * 0.05 * 2.0 = 10 permits.

Premium tier: 100 customers * 500 req/s = 50,000 req/s total. 50 instances, each handling 1,000 req/s. Per-instance pool size = 1,000 * 0.02 * 2.0 = 40 threads. Per-tenant semaphore: 500 * 0.02 * 2.0 = 20 permits.

Total threads per instance: 200 + 100 + 40 = 340 threads. Memory per instance: 340 * 256 KB = 85 MB. Total across 100 instances: 8.5 GB.

Fallback strategy: Free tier returns a degraded response (cached data). Standard tier returns a cached response with a 1-minute TTL. Premium tier is never rejected; if the pool is full, the request is queued with a 10-second timeout.

**Common mistakes.** Using a single shared pool for all tiers, which allows a noisy free tenant to exhaust Standard or Premium resources. Not considering the instance count. Forgetting the per-tenant semaphore within the per-tier pool.

### Challenge 2.3: Bulkhead with Adaptive Concurrency (15 minutes)

**Prompt.** You are designing a system that calls a third-party API. The API's latency varies unpredictably: normally 50ms, but can spike to 3 seconds during traffic surges. The API has a rate limit of 1,000 requests per second. You need to maximize throughput while protecting the API from overload and protecting your system from slow responses.

Design a bulkhead architecture with adaptive concurrency limiting. Explain how the adaptive algorithm works, how it detects the need for change, and how it adjusts the concurrency limit. Include the initial configuration, the adaptation logic, and the safety mechanisms.

**Time limit.** 15 minutes.

**Evaluation criteria.** Correct design of adaptive concurrency limiting. Reasonable initial configuration. Clear adaptation logic. Safety mechanisms to prevent oscillations. Integration with circuit breaker.

**Model answer.** The architecture uses a thread pool bulkhead with dynamic pool sizing. The initial configuration: pool size = 50 threads (based on 1,000 req/s * 0.05s). Queue capacity = 100 slots. The pool size is adjusted dynamically based on observed latency.

The adaptation logic uses a variant of the CoDel algorithm. Every 100 milliseconds, the system measures the average queueing delay. If the queueing delay exceeds 5 milliseconds, the pool size is reduced by 20%. If the queueing delay is below 2 milliseconds for 10 consecutive measurements, the pool size is increased by 1.

The pool size is bounded between 10 and 200 threads. The lower bound prevents the pool from becoming too small to handle any traffic. The upper bound prevents the pool from consuming too many resources.

The adaptation logic is smoothed to prevent oscillations. Each adjustment is limited to 20% of the current pool size. Adjustments are made at most once per second. The pool size is persisted and restored on application restart.

A circuit breaker is paired with the bulkhead. If the API failure rate exceeds 50% in a 30-second window, the circuit breaker opens. The fallback handler returns a cached response. The circuit breaker probes the API every 5 seconds. When the probe succeeds, the circuit breaker closes and the bulkhead resumes normal operation.

The monitoring tracks the adaptive pool size, queueing delay, and adaptation events. Alerts fire when the pool size changes by more than 50% in 5 minutes (indicating instability) or when the queueing delay exceeds 10ms (indicating persistent overload).

**Common mistakes.** Setting the pool size as a fixed value without adaptation. Not bounding the adaptive range. Not smoothing the adaptation to prevent oscillations. Forgetting the circuit breaker integration.

## Tier 3: Expert

### Challenge 3.1: Global E-Commerce Platform Bulkhead (30 minutes)

**Prompt.** You are designing the bulkhead architecture for a global e-commerce platform. The platform operates in 10 AWS regions. Each region has 20 microservices. The services communicate within the region and across regions. The platform handles 1,000,000 transactions per second globally.

The following services are the most latency-sensitive:

- Product search: 200,000 req/s per region, 20ms latency, 100ms p99.
- Checkout: 50,000 req/s per region, 500ms latency, 3-second p99.
- Payment: 50,000 req/s per region, 1-second latency, 10-second p99.
- Inventory: 200,000 req/s per region, 50ms latency, 500ms p99.
- Recommendations: 100,000 req/s per region, 100ms latency, 1-second p99.

Design the complete bulkhead architecture. Address per-service, per-region, and cross-region isolation. Calculate the total resource usage. Provide the configuration for each service, including pool size, isolation variant, queue capacity, fallback strategy, and circuit breaker integration. Describe the monitoring and alerting strategy.

**Time limit.** 30 minutes.

**Evaluation criteria.** Complete architecture covering all isolation levels. Correct pool sizing for each service. Appropriate cross-region bulkhead design. Total resource usage calculation. Monitoring and alerting strategy. Justification for all decisions.

**Model answer.** The architecture has three layers of bulkheads: per-service thread pools within each region, per-region resource limits, and cross-region connection pools.

**Layer 1: Per-Service Bulkheads (per region, per 100-instance cluster)**

Product search: semaphore isolation. Permits = 2,000 * 0.02 * 2.0 = 80 per instance. Fallback: return cached search results, 5-minute TTL.

Checkout: thread pool isolation. Pool = 500 * 0.5 * 2.0 = 500 threads per instance. Queue = 5 * 500 = 2,500 slots. Fallback: return "checkout pending, we will notify you."

Payment: thread pool isolation. Pool = 500 * 1.0 * 2.0 = 1,000 threads per instance. Queue = 10 * 500 = 5,000 slots. Fallback: return "payment pending" and enqueue for async processing.

Inventory: thread pool isolation. Pool = 2,000 * 0.05 * 2.0 = 200 threads per instance. Queue = 2 * 2,000 = 4,000 slots. Fallback: return "in stock" and verify at checkout.

Recommendations: semaphore isolation. Permits = 1,000 * 0.1 * 2.0 = 200 per instance. Fallback: return empty recommendations list.

**Layer 2: Per-Region Limits**

Each region has a maximum of 10,000 threads across all services. This prevents one region from consuming all threads in the global pool. The limit is enforced by a global semaphore. Each thread acquired from a per-service pool must also acquire a region permit.

**Layer 3: Cross-Region Bulkheads**

Cross-region calls use separate connection pools. Each cross-region destination gets a pool of 100 connections. The pool is sized for the 60-100ms cross-region latency. Fallback: return cached data from the local region. Cross-region calls have a 2-second timeout.

**Total resource usage per region**

Per-instance: 500 + 1,000 + 200 = 1,700 threads (checkout, payment, inventory). 100 instances: 170,000 threads. Memory: 170,000 * 256 KB = 41.6 GB. This is distributed across 100 instances, so 416 MB per instance. Acceptable for 8 GB instances.

**Monitoring and alerting**

Per-service metrics: active count, queue depth, rejection count, p50/p99 latency. Per-region metrics: total thread usage, region pool utilization. Cross-region metrics: connection pool usage, cross-region latency.

Alerts: rejection rate > 0 (critical), queue depth > 50% capacity (warning), region thread usage > 80% of limit (critical), cross-region latency > 2 seconds (warning).

**Common mistakes.** Not addressing cross-region isolation. Sizing pools without considering the instance count. Not providing region-level limits. Forgetting to include fallback strategies for cross-region failures.

### Challenge 3.2: Designing a Bulkhead Configuration System (30 minutes)

**Prompt.** You are designing a centralized bulkhead configuration system for a company with 500 microservices. Each service has 5 to 20 downstream dependencies. The current configuration is manual: each team sets their own pool sizes, queue capacities, and fallback strategies. This has led to inconsistent configurations, frequent incidents, and high operational overhead.

Design a system that automatically recommends and enforces bulkhead configurations. The system should:

- Collect metrics from all services.
- Analyze the metrics to detect misconfiguration.
- Recommend optimal pool sizes based on observed throughput and latency.
- Enforce minimum and maximum limits.
- Detect anomalies and alert the operations team.

Provide the architecture, the algorithm for configuration recommendation, the enforcement mechanism, and the anomaly detection logic.

**Time limit.** 30 minutes.

**Evaluation criteria.** Complete architecture covering all components. Clear algorithm for configuration recommendation. Practical enforcement mechanism. Robust anomaly detection. Justification for all decisions.

**Model answer.** The system has four components: a metrics collector, a configuration analyzer, a configuration store, and an enforcement agent.

**Metrics collector.** A sidecar agent runs alongside each service. The agent collects per-bulkhead metrics every 10 seconds: active count, queue depth, rejection count, p50 latency, p99 latency, throughput. The metrics are sent to a central time-series database (Prometheus).

**Configuration analyzer.** A batch job runs every 5 minutes. The job reads the last 30 minutes of metrics for each bulkhead partition. For each partition, the job calculates:

- Recommended pool size = p99 throughput * p99 latency * 1.5 safety margin.
- Recommended queue capacity = max(100, p99 throughput * 5 seconds).
- Current pool utilization = p99 active count / current pool size.

If the current pool utilization exceeds 80% or the rejection rate exceeds 0.1%, the partition is flagged as "under-provisioned." If the current pool utilization is below 20% and the rejection rate is zero for 24 hours, the partition is flagged as "over-provisioned."

**Configuration store.** The recommended configurations are stored in a key-value store (etcd). Each partition has a key: `bulkheads/{service}/{dependency}/config`. The value is the recommended pool size, queue capacity, and isolation variant.

**Enforcement agent.** The sidecar agent watches the configuration store for changes. When a new configuration is published, the agent updates the local bulkhead configuration. The update is applied without restarting the service. The agent logs the configuration change and sends an event to the operations team.

**Anomaly detection.** The analyzer detects three types of anomalies:

- Sudden increase in rejection rate: the pool size is too small for the current traffic. The analyzer automatically increases the pool size by 50% and alerts the operations team.
- Sudden increase in latency: the downstream service is degraded. The analyzer recommends reducing the pool size by 50% to prevent thread exhaustion. The operations team must approve the reduction.
- Configuration drift: a team manually changed the configuration outside the system. The analyzer detects the drift and reverts the configuration to the recommended value.

**Safety mechanisms.** All automatic changes are bounded: pool sizes cannot exceed 200% of the initial configuration or fall below 50%. All automatic changes are logged and can be rolled back. The operations team receives a summary of all changes every 15 minutes.

**Common mistakes.** Not including the enforcement agent. Not bounding automatic changes. Not detecting configuration drift. Not providing a rollback mechanism. Not considering the operations team's workflow.
