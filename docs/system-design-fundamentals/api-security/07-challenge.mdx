### Exercise 1.1: Explain Defense in Depth

You are designing the security posture for a new microservices platform that handles sensitive user data. Explain the concept of defense in depth and list at least five distinct layers of defense you would implement, describing what each layer protects against.

**Model Answer:**

Defense in depth is a security strategy that layers independent defensive mechanisms so that if one layer is breached, the next layer continues to provide protection. No single security control is perfect — defense in depth acknowledges this and builds redundancy into the security architecture.

```
Five layers of defense for a microservices platform:

Layer 1: Network perimeter (WAF + DDoS protection)
  Protects against: volumetric DDoS attacks, SQL injection,
  XSS, protocol attacks at the edge
  Implementation: CloudFront + AWS WAF with rate-based rules,
  managed rule sets (OWASP top 10), geo-blocking

Layer 2: Authentication and authorization
  Protects against: unauthorized access, credential theft,
  privilege escalation
  Implementation: OAuth 2.0 + OIDC, short-lived JWTs,
  API keys for services, mutual TLS for service-to-service,
  RBAC with least-privilege policies

Layer 3: Transport security
  Protects against: eavesdropping, man-in-the-middle,
  replay attacks
  Implementation: TLS 1.3 for all endpoints, HSTS headers,
  certificate pinning for critical services, perfect forward
  secrecy cipher suites

Layer 4: Input validation and sanitization
  Protects against: injection attacks (SQL, NoSQL, command),
  IDOR, buffer overflows
  Implementation: Allow-list validation, parameterized queries,
  rate limiting, request size limits, content-type enforcement

Layer 5: Data protection at rest
  Protects against: data breaches from storage compromise,
  insider threats, physical theft
  Implementation: AES-256 encryption at rest, envelope encryption
  with KMS, field-level encryption for PII, access auditing,
  automatic key rotation

Each layer assumes the layer below it has been compromised.
A WAF bypass should not help an attacker if authentication
is working correctly. An authentication bypass should not
help if input validation catches the payload. This redundancy
is the core of defense in depth.
```

### Exercise 1.2: Explain CORS

A frontend application running at `https://app.example.com` makes API calls to `https://api.example.com`. The developer reports that the browser is blocking requests with a CORS error. Explain what CORS is, why it exists, how the browser enforces it, and what configuration is needed on the API server to resolve this specific scenario.

**Model Answer:**

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that controls which origins are allowed to access resources on a different origin. It exists because the Same-Origin Policy (SOP) blocks cross-origin requests by default to prevent malicious sites from reading sensitive data from other sites.

```
How CORS works:

1. The browser enforces CORS, NOT the server.
   An attacker can bypass CORS by using curl or Postman —
   CORS only protects browser-based clients.

2. For simple requests (GET, HEAD, POST with certain content types):
   Browser adds the Origin header: Origin: https://app.example.com
   Server responds with: Access-Control-Allow-Origin: https://app.example.com
   Browser checks the response header — if it doesn't match, the
   response is blocked in the browser (the server still processes it).

3. For preflight requests (PUT, DELETE, custom headers, non-standard types):
   Browser sends an OPTIONS request first:
     OPTIONS /resource HTTP/1.1
     Origin: https://app.example.com
     Access-Control-Request-Method: DELETE
     Access-Control-Request-Headers: X-Custom-Header

   Server responds:
     Access-Control-Allow-Origin: https://app.example.com
     Access-Control-Allow-Methods: GET, POST, PUT, DELETE
     Access-Control-Allow-Headers: X-Custom-Header
     Access-Control-Max-Age: 86400

   Only after receiving this response does the browser send the
   actual DELETE request.

Configuration for https://api.example.com to accept requests
from https://app.example.com:

  Access-Control-Allow-Origin: https://app.example.com

Do NOT use a wildcard (*) in production — it allows any site to
read the response. This is acceptable only for public APIs.

Additional headers to consider:
  Access-Control-Allow-Credentials: true
    (required if using cookies for auth, and then the origin cannot be *)
  Access-Control-Expose-Headers: X-Request-Id, X-RateLimit-Remaining
    (exposes custom response headers to the browser)

Common CORS mistakes:
  1. Using * with Allow-Credentials (browser rejects this)
  2. Forgetting to handle OPTIONS preflight at the application
     level (it should be handled before auth middleware)
  3. Reflecting the Origin header without validation
     (opens the door to origin spoofing)
```

### Exercise 1.3: Explain CSRF

Your web application processes financial transactions. A user reports that money was sent from their account without their knowledge. Investigators discover the user visited a malicious site while logged into your application. Explain what CSRF is, how this attack works, and design a complete CSRF defense strategy for your application.

**Model Answer:**

CSRF (Cross-Site Request Forgery) is an attack where a malicious site tricks a user's browser into making an unintended request to a trusted application where the user is authenticated. The attack exploits the fact that the browser automatically includes cookies with requests to the target origin.

```
How the CSRF attack works:

1. User logs into bank.com — the browser stores the session cookie.
2. User visits attacker.com (in another tab) without logging out.
3. attacker.com contains:
     <form action="https://bank.com/transfer" method="POST">
       <input name="amount" value="10000" />
       <input name="to" value="attacker" />
     </form>
     <script>document.forms[0].submit();</script>
4. The browser sends the POST request to bank.com INCLUDING the
   session cookie automatically — the server sees a legitimate
   authenticated session and processes the transfer.

Why this works:
  - The Same-Origin Policy does NOT prevent sending requests —
    it prevents reading responses. CSRF is a write-only attack.
  - Cookies are sent automatically by the browser regardless of
    which site originated the request.

CSRF defense strategy (implement all three layers):

Layer 1: CSRF tokens (primary defense)
  - Server generates a cryptographically random token bound to
    the user's session
  - Token is embedded in every form or sent as a custom header:
      <input type="hidden" name="_csrf" value="abc123..." />
    or:
      X-CSRF-Token: abc123...
  - Server validates the token on every state-changing request
  - Attacker cannot read the token from the malicious site
    (SOP prevents cross-origin reads)

Layer 2: SameSite cookie attribute
  - Set the session cookie with SameSite=Strict or SameSite=Lax:
      Set-Cookie: session=xxx; SameSite=Strict; Secure; HttpOnly
  - SameSite=Strict: cookie is never sent for cross-origin requests
  - SameSite=Lax: cookie is sent for top-level GET navigations
    (safe for links) but not for POST forms from other sites

Layer 3: Custom request headers
  - Require a non-standard header for all state-changing requests:
      X-Requested-By: XMLHttpRequest
  - Cross-origin requests cannot set custom headers without CORS
    preflight, and the browser will not send cookies with preflight
    unless the server explicitly allows it

Additional measures:
  - Double-submit cookie pattern: send the CSRF token both as a
    cookie and as a request header — server validates they match
    (useful for stateless APIs without server-side sessions)
  - Re-authentication for sensitive actions (transfers, password
    changes): require the user to enter their password again
  - Origin header validation: verify the Origin or Referer header
    matches your application's allowed origins
```

### Exercise 2.1: API Security for a Fintech

Design the API security architecture for a fintech platform that allows users to transfer money, view transaction history, and manage linked bank accounts. The API is consumed by a web application and a mobile app. Compliance requirements include SOC 2 and GDPR.

**Your task:** Define the complete API security architecture including authentication, authorization, transport security, rate limiting, audit logging, and data protection. Include specific technologies and configuration decisions.

**Model Answer:**

```
Architecture overview:

                    ┌─────────────┐
                    │  WAF + CDN  │
                    └──────┬──────┘
                           │ TLS 1.3
                    ┌──────▼──────┐
                    │  API Gateway│
                    │  (Kong)     │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼────┐ ┌────▼────┐
        │ Auth      │ │Transfer│ │History  │
        │ Service   │ │Service │ │Service  │
        └─────┬─────┘ └───┬────┘ └────┬────┘
              │            │            │
              └────────────┼────────────┘
                           │
                    ┌──────▼──────┐
                    │  Database   │
                    │  (encrypted)│
                    └─────────────┘

Authentication:
  - OAuth 2.0 + OIDC for user authentication
  - PKCE flow for both web and mobile (no client secret exposed)
  - Access tokens: short-lived JWTs (15 minutes)
  - Refresh tokens: opaque, 7 days, stored securely (HttpOnly cookie
    for web, device keychain for mobile)
  - Biometric authentication for high-value transfers (>$1,000)

Authorization:
  - RBAC with scopes: transfer:read, transfer:write, account:read,
    account:write, profile:read, profile:write
  - Transaction-level authorization: each transfer is checked against
    daily limits ($10,000 default), velocity checks (max 5 transfers/hour),
    and fraud scoring
  - API Gateway enforces scope-based access before requests reach
    backend services

Transport security:
  - TLS 1.3 minimum, HSTS with max-age=31536000, preload
  - Certificate pinning in mobile apps
  - Mutual TLS for service-to-service communication within the VPC
  - API Gateway terminates TLS and re-encrypts for backend services

Rate limiting:
  - Per-user: 100 requests/minute for standard endpoints
  - Per-IP: 1,000 requests/minute (against DDoS)
  - Per-endpoint: 10 transfers/minute per user (reduces fraud blast radius)
  - Rate limit headers: X-RateLimit-Limit, X-RateLimit-Remaining,
    X-RateLimit-Reset (helps clients self-regulate)

Audit logging:
  - Every state-changing request is logged with:
    user_id, timestamp, action, resource, IP, user-agent,
    request_id (correlation ID across services), geo-location
  - Logs are append-only, written to immutable storage (AWS CloudTrail
    + S3 with object lock)
  - Retention: 1 year hot storage, 7 years cold storage (GDPR requires
    documented retention policy)

Data protection:
  - Encryption at rest: AES-256, envelope encryption with AWS KMS
  - Field-level encryption for PII (email, phone, SSN)
  - Tokenization for payment card data (replace PAN with tokens)
  - Database is encrypted at the storage layer + application layer
    (defense in depth for data)

Mobile-specific:
  - Certificate pinning prevents MITM even with compromised CAs
  - Device attestation (Android SafetyNet / iOS DeviceCheck)
  - Biometric gate for sensitive operations
  - App transport security (ATS) enforces HTTPS in iOS
```

### Exercise 2.2: API Security for a Public API

Design the security architecture for a public REST API that allows third-party developers to query weather data, access historical climate statistics, and subscribe to weather alert webhooks. The API offers a free tier and a paid tier.

**Your task:** Define authentication, authorization, rate limiting, abuse prevention, and webhook security for a developer-facing public API. Address how you handle API key rotation, usage tracking, and the differences between the free and paid tiers.

**Model Answer:**

```
Authentication model:

Two authentication methods:
  Method 1: API Key (for server-to-server, simple use cases)
    - Generated per developer account via developer portal
    - Sent as header: X-API-Key: sk_live_abc123...
    - Key format: prefix (sk_live_/sk_test_) + random 32 bytes
      (base64 encoded, roughly 48 characters)
    - Stored as SHA-256 hash — the plaintext key is shown once
      at creation and never stored
    - Rate limited per key, revoked from developer portal

  Method 2: OAuth 2.0 (for acting on behalf of end users)
    - Used when a third-party app needs to access user-specific data
      (e.g., saved locations, alert preferences)
    - Client Credentials grant for machine-to-machine
    - Authorization Code + PKCE when user delegation is needed

Authorization:
  - Free tier: read-only access, 1,000 requests/day, 1 API key
  - Paid tier: read + webhook write, 100,000 requests/day, 5 API keys
  - Enterprise tier: full access, 10M requests/day, unlimited keys,
    custom rate limits, SLA-backed uptime
  - Scopes: weather:read, climate:read, webhook:write
  - API Gateway validates tier and scope before proxying

Rate limiting:
  Per-key sliding window counters in Redis:
    Free: 1 req/s sustained, burst to 10
    Paid: 10 req/s sustained, burst to 100
    Enterprise: 100 req/s sustained, burst to 500

  Response headers:
    X-RateLimit-Limit: 1000
    X-RateLimit-Remaining: 842
    X-RateLimit-Reset: 1625097600
    Retry-After: 45

  On rate limit exceeded:
    429 Too Many Requests with Retry-After header
    JSON body: {"error":"rate_limit_exceeded","retry_after_seconds":45}

Abuse prevention:
  - Anomaly detection: monitor for sudden spikes from a single key
    (indicates leaked key or compromised account)
  - Honeytokens: API keys that look valid but are flagged on use
    (identifies unauthorized data scraping)
  - Usage quotas that escalate: not just hard limits, but soft warnings
    at 80% and 95% of quota
  - Bot detection: rate patterns, User-Agent analysis, IP reputation

API key rotation:
  - Keys expire after 90 days (configurable for enterprise)
  - Grace period: old key continues to work for 48 hours after rotation
    (allows developers to migrate without downtime)
  - Rotation notification: email 14 days, 7 days, and 1 day before expiry
  - Developer portal supports generating a new key without revoking the
    old one during the grace period
  - Key hashes in database include a version field for parallel validation

Usage tracking and billing:
  - Each request is logged with: developer_id, key_id, endpoint,
    response_time, status_code, request_size, timestamp
  - Logs are streamed to the billing system in near-real-time (5-minute
    batches via Kinesis)
  - Daily usage summaries emailed to developers
  - Overages: paid tier supports metered billing above quota
    (automatic or with cap — configurable by developer)

### Exercise 2.3: Design Webhook Security

You are building a webhook delivery system that sends real-time event notifications to hundreds of third-party services. Events include order confirmations, payment status changes, and fraud alerts. The webhooks contain sensitive customer data.

**Your task:** Design the complete webhook security architecture. Address payload signing, delivery guarantees, retry logic, replay attacks, IP allowlisting versus signature verification, secret rotation, and dead-letter handling.

**Model Answer:**

```
Webhook delivery architecture:

  Event Source ──> Event Bus ──> Webhook Dispatcher ──> Recipient
                    (Kafka)      (Dedicated service)      (HTTP)

Payload signing (MANDATORY):
  Every webhook payload is signed with an HMAC-SHA256 signature
  derived from a per-recipient shared secret:

    signature = HMAC-SHA256(secret, timestamp + "." + body)

  Headers sent with each webhook:
    X-Webhook-Id: wh_evt_abc123           (unique delivery attempt ID)
    X-Webhook-Timestamp: 1625097600        (Unix timestamp, prevents replay)
    X-Webhook-Signature: v1=abc123def456   (comma-separated for key rotation:
                                            v1=old,v2=new)

  Recipient verification (pseudo-code):
    expected = HMAC-SHA256(secret, timestamp + "." + body)
    if constant_time_compare(signature, expected):
        process_webhook(body)
    else:
        reject_with_401("invalid signature")

  Why HMAC and not JWT:
    - HMAC is simpler, faster, and sufficient for webhook signing
    - JWTs are useful when the payload contains claims about the
      sender (iss, aud, exp), but for webhooks the recipient only
      needs to verify the payload hasn't been tampered with
    - HMAC avoids the complexity of JWT libraries and key management

Replay attack prevention:
  The timestamp in the signature prevents replay:
    1. Recipient checks that timestamp is within 5 minutes of current time
    2. Recipient checks X-Webhook-Id against a deduplication cache (expire
       after 1 hour — after that, replays are harmless because the
       timestamp check catches them)
    3. If X-Webhook-Id is already in the cache: return 200 OK without
       processing (idempotent consumption)

Delivery guarantees and retry:
  At-least-once delivery with idempotent consumption:
    - Attempt 1: immediate delivery
    - Retry 1: 60 seconds (transient failure recovery)
    - Retry 2: 5 minutes (network issue recovery)
    - Retry 3: 30 minutes (service restart)
    - Retry 4: 2 hours (infrastructure incident)
    - Retry 5: 6 hours (worst-case delay)
    - After 6 failed attempts: move to dead-letter queue (DLQ)

  Each retry uses exponential backoff with jitter:
    delay = min(base * 2^attempt + random(0, 1000ms), max_delay)

  Recipient should return:
    2xx: success, stop retrying
    4xx: client error, stop retrying (recipient is misconfigured)
    429: rate limited, retry with Retry-After header respect
    5xx: server error, retry

IP allowlisting vs signature verification:
  Signature verification is strictly superior to IP allowlisting:
    - IPs can be spoofed (especially with VPC peering, NAT changes)
    - Managing IP ranges is operationally expensive (cloud providers
      publish changing IP ranges in JSON documents)
    - IP allowlisting does not prevent replay attacks
    - IP allowlisting alone provides NO cryptographic proof of
      payload integrity

  Recommendation: do NOT use IP allowlisting. Use signature
  verification exclusively. If the recipient insists on IP
  allowlisting as a SECONDARY measure, publish a fixed set of
  egress IPs and rotate them quarterly.

IP allowlisting as a secondary measure (if required):
  Publish egress IPs in a well-known location:
    https://api.example.com/.well-known/webhook-ips.json
  Limit to 4-8 IPs, rotate quarterly with 2-week overlap

Secret rotation:
  Each recipient has a primary and secondary secret:
    - Primary: used for signing new webhooks
    - Secondary: kept active for signature verification during rotation
  Rotation process:
    1. Generate new secret, set as secondary
    2. Sign webhooks with primary, include both in X-Webhook-Signature:
       X-Webhook-Signature: v1=old_sig,v2=new_sig
    3. Recipient verifies with either secret
    4. After 48 hours, promote secondary to primary
    5. Generate a new secondary for the next rotation
  Rotation frequency: every 90 days, or immediately on security incident

Dead-letter queue:
  Webhooks that fail all 6 retry attempts go to the DLQ:
    - DLQ stores: webhook payload, recipient URL, failure reasons,
      timestamps of all attempts, X-Webhook-Id
    - Alerting: DLQ with > 10 messages triggers PagerDuty
    - Manual intervention: support team contacts the recipient
    - Automatic suspension: if a recipient has > 50 DLQ messages in
      24 hours, automatically suspend webhook deliveries and notify
      the recipient via email and dashboard notification
    - DLQ replay: support team can replay DLQ messages after the
      recipient confirms the issue is resolved
```

### Exercise 3.1: API Security for a PCI-Compliant Payment System

Design the API security architecture for a payment processing system that handles credit card transactions and must comply with PCI DSS (Payment Card Industry Data Security Standard). The system processes 10,000 transactions per minute across web, mobile, and POS (point-of-sale) channels.

**Your task:** Define the complete security architecture covering PCI DSS requirements 3 (protect stored cardholder data), 4 (encrypt transmission), 6 (develop and maintain secure systems), 7 (restrict access by need-to-know), 10 (track and monitor access), and 11 (test security). Address how you handle card data at rest, in transit, and in use. Include your approach to tokenization, key management, and quarterly ASV scanning.

**Model Answer:**

```
PCI DSS scope reduction strategy:

THE CORE PRINCIPLE: Minimize the PCI DSS compliance scope.
Every system that stores, processes, or transmits cardholder
data (CHD) is IN SCOPE. Our goal is to isolate CHD to the
smallest possible footprint.

Approach: Tokenization with a third-party token vault.

Architecture:

  Client App (web/mobile/POS)
       │
       │ Card info goes DIRECTLY to token provider
       │ (Stripe/Braintree/Adyen — PCI DSS Level 1)
       ▼
  Token Provider (Stripe)
       │ Returns a payment token: tok_abc123
       │ Token is NOT a PAN — it has no mathematical
       │ relationship to the real card number
       ▼
  Our API (receives token, never sees PAN)
       │
       │ Stores token in database (OUT OF SCOPE for PCI)
       │ Sends token + amount to processor
       ▼
  Payment Processor
       │ Resolves token to PAN internally (IN their scope)
       │ Processes the transaction
       ▼
  Card Network / Bank

Result: Our systems NEVER touch the PAN. PCI scope is limited
to the physical and network controls around the system, not
the application itself.

If self-hosting tokenization is unavoidable:

Requirement 3: Protect stored cardholder data
  - Never store the full PAN. If storage is required:
    - Tokenize to a random token (16-19 chars, Luhn-check-passing
      for format preservation)
    - Truncate displayed PAN to first-6-last-4 (BIN + last 4 digits)
    - Encrypt the token-to-PAN mapping table with AES-256-GCM
    - Encryption keys stored in a Hardware Security Module (HSM)
      or KMS with automatic rotation every 90 days
    - Key hierarchy:
      Master Key (HSM, rotated yearly) → Key Encryption Key
      (KMS, rotated quarterly) → Data Encryption Key (per volume,
      rotated monthly) → per-field IVs
    - Never write the PAN to logs, error messages, or monitoring
      dashboards

Requirement 4: Encrypt transmission
  - TLS 1.2 minimum, TLS 1.3 preferred for all external connections
  - Mutual TLS between internal services handling tokenized data
  - Perfect Forward Secrecy (ECDHE) cipher suites only
  - Strong ciphers: TLS_AES_256_GCM_SHA384 or
    TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
  - Disable TLS 1.0, 1.1, and all CBC-mode ciphers
  - HSTS with max-age=31536000 and includeSubDomains
  - Certificate pinning in mobile apps for the payment endpoint

Requirement 6: Develop and maintain secure systems
  - All code changes go through security review
  - SAST (Static Analysis) on every commit: CodeQL, Semgrep
  - DAST (Dynamic Analysis) weekly: OWASP ZAP in CI/CD pipeline
  - Dependency scanning: Dependabot for known vulnerabilities
  - SCA (Software Composition Analysis): Snyk for open-source
    dependency vulnerabilities
  - Secure coding standards: OWASP ASVS Level 3 (highest)
  - Separate development, test, and production environments
    with no data sharing between environments

Requirement 7: Need-to-know access
  - Role-Based Access Control with PCI-specific roles:
    - PCI_ADMIN: manages encryption keys (2-person control required)
    - PCI_AUDITOR: read-only access to audit logs
    - PCI_OPERATOR: monitor system health, no data access
    - PCI_DEVELOPER: access to code but NOT to production data
  - All database access requires approved change tickets
  - Just-in-time (JIT) access: no standing admin permissions,
    approval workflow for temporary elevated access

Requirement 10: Track and monitor access
  - Immutable audit logs of all CHD access:
    user_id, timestamp, action, resource, IP, query
  - Logs written to append-only storage (S3 Object Lock in
    compliance mode) with 7-year retention (PCI minimum)
  - Log monitoring: SIEM (Splunk/ELK) with 10+ alerts:
    - Multiple failed login attempts (10+ in 5 minutes)
    - Access to CHD by non-PCI_ADMIN role
    - API calls from unexpected IP ranges
    - Database queries returning > 100 CHD records (data scraping)
    - Changes to firewall rules or access control lists
  - All clocks synchronized to atomic time (NTP with multiple
    upstream sources)

Requirement 11: Test security
  - Quarterly ASV (Approved Scanning Vendor) scans of all external
    IPs and domains — automated via Nessus or Qualys
  - Penetration testing: annual external + internal, with quarterly
    re-test after any significant infrastructure change
  - Intrusion detection: IDS/IPS on all network segments containing
    CHD, with signature updates within 24 hours of vendor release
  - File integrity monitoring: detect unauthorized changes to
    critical system files (Tripwire / OSCAP)
  - Segmentation testing: quarterly verification that CHD systems
    are isolated from non-CHD systems (network scans to prove
    no cross-VPC traffic routes are open)

Key management specifics:
  Key hierarchy:
    Root of Trust: HSM (CloudHSM / Luna) — never leaves the HSM
      └─ Master Key (MK): stored in HSM, rotated yearly
          └─ Key Encryption Key (KEK): stored in KMS, rotated quarterly
              └─ Data Encryption Key (DEK): stored with data,

    - MK creation requires 2 PCI_ADMINs to be present (physical or
      cryptographically verified remote ceremony)
    - MK backup: split into 5 shards with Shamir's Secret Sharing,
      3 required to reconstruct
    - MK destruction: cryptographically erase using the HSM's
      destroy function, confirmed by two independent admins

  1. Identify all external IPs/DNS in the cardholder data environment
  2. Run Qualys/Nessus PCI scan against each IP
  3. Remediate any findings classified as High or Critical:
     - Critical: must be fixed within 72 hours (potential data
       compromise window)
     - High: must be fixed before the next scan (within 90 days)
  4. Submit passing scan reports to acquirer
  5. Maintain evidence of passing scans for 3 years

  Common ASV findings to preemptively address:
    - TLS 1.0/1.1 enabled (disable them)
    - Weak Diffie-Hellman parameters (use 2048-bit minimum, or ECDHE)
    - Self-signed certificates accessible externally (use trusted CA)
    - Open/forwarded ports not required for business (default-deny)
    - Outdated OpenSSL or Nginx versions (keep patched)

### Exercise 3.2: API Security for a GraphQL API

You are designing a GraphQL API for a multi-tenant SaaS platform. The schema exposes user profiles, billing data, analytics, and admin functions. The API is consumed by web and mobile applications, and by third-party integrations via API keys.

**Your task:** Design the complete security architecture for this GraphQL API. Address authentication, authorization at the field and type level, rate limiting specific to GraphQL (query complexity analysis, depth limiting, batching), input validation, and protection against GraphQL-specific attacks (introspection abuse, data scraping via nested queries, denial of service via expensive queries).

**Model Answer:**

```
Authentication strategies:

Three authentication methods depending on the consumer:
  1. Session cookie (web app): HttpOnly, Secure, SameSite=Strict
  2. Bearer token (mobile app): JWT with PKCE OAuth 2.0 flow
  3. API Key (third-party): static key with scopes, for server-to-server

All methods resolve to the same internal structure:
  ```json
  { "user_id": "...", "tenant_id": "...", "roles": ["admin", "viewer"], "scopes": ["read", "write"] }
  ```

Authorization at field and type level:

We implement a resolver-level authorization middleware that checks
the authenticated user's tenant, role, and scope against the
resource being accessed.

```
Type-level authorization example:
```graphql
  type Query {
```
    # Public — available to all authenticated users
    me: User

    # Tenant-scoped — user can only query their own tenant's data
```graphql
    users(tenantId: ID!): [User!]!   @auth(require: "tenant_access")
```

    # Admin-only — requires admin role
```graphql
    billingReports(month: String!): [BillingReport!]!
      @auth(require: "admin")
```

    # System-only — restricted to internal API keys
```graphql
    systemHealth: HealthStatus    @auth(require: "internal")
  }
```

Field-level authorization example:
```graphql
  type User {
    id: ID!
    name: String!                 @auth(require: "self_or_admin")
    email: String!                @auth(require: "self_or_admin")
    billingHistory: [Invoice!]    @auth(require: "admin_or_billing")
    ssn: String                   @auth(require: "compliance_officer")
  }
```

Implementation pattern (pseudo-middleware):
  Each resolver group wraps the base resolver with authorization:
    1. Extract tenant_id from the authenticated context
    2. Check if the requested tenant_id matches the user's tenant
    3. Check if the user's role allows the operation
    4. If field-level: check field permissions before resolving
    5. Return null or throw AuthorizationError if unauthorized

  Critical: Never return a different error message for "field
  exists but unauthorized" vs "field does not exist" — this
  leaks schema information to unauthorized users.

GraphQL-specific rate limiting:

REST APIs rate limit by endpoint. GraphQL has a single endpoint
so we need QUERY COMPLEXITY ANALYSIS.

Query complexity scoring:
  Each field has a cost (default: 1, arrays: count * cost):
  ```graphql
  type Query {
    user(id: ID!): User!         cost: 1
    users: [User!]!              cost: 10 (expensive list query)
    searchUsers(query: String!): [User!]!  cost: 20 + 2 * limit
    analyticsReport(type: String!): JSON!  cost: 50 (compute-heavy)
  }
  ```

  Query complexity = sum of resolved field costs:
  ```graphql
  query {
    me { name email }            complexity: 3 (1 for me + 1 + 1)
    users { id name }            complexity: 30 (10 for users + 10*1 + 10*1)
    Total: 33
  }
  ```

  Limits:
    Free tier: max complexity 100 per query, 1000 per minute
    Paid tier: max complexity 500 per query, 5000 per minute
    Enterprise: custom

Query depth limiting:
  Prevents deeply nested queries that create exponential joins:
  ```graphql
  query {
    user(id: "1") {                     depth 1
      friends {                          depth 2
        friends {                        depth 3
          friends {                      depth 4 — too deep
            friends {                    depth 5
              name
            }
          }
        }
      }
    }
  }
  ```

  Limit: max depth of 5 for standard users, 7 for internal API keys.
  A banned query above the limit returns:
  ```json
  { "errors": [{ "message": "Query depth exceeds maximum of 5" }] }
  ```

Batch request limiting:
  Prevent GraphQL batching attacks where the client sends many
  queries in a single request:
  ```graphql
  query {
    u1: user(id: "1") { ... }
    u2: user(id: "2") { ... }
    ...
    u100: user(id: "100") { ... }
  }
  ```

  Limit: max 10 top-level query fields per request.
  For N+1 bulk queries: provide a dedicated batch endpoint instead
  ```graphql
  batchUsers(ids: ["1","2",...,"100"]): [User!]!
  ```

GraphQL-specific attack mitigations:

1. Introspection abuse:
   Attack: attacker queries __schema to dump the entire schema
     (all types, fields, arguments, and their descriptions)
   Mitigation: disable introspection in production
     For development/staging: requires admin API key
     For production: completely disabled, or whitelist internal IPs
     Partial alternative: expose a curated schema document instead
       (documentation, not machine-readable introspection)

2. Data scraping via nested queries:
   Attack: query nested list fields to download all data:
  ```graphql
  query {
    allUsers {           # returns every user
      orders {           # all orders for each user
        lineItems {      # all line items per order
          product {
            supplier { name }
          }
        }
      }
    }
  }
  ```
  Mitigations:
     - All list fields require pagination (first, after, limit)
       enforce: max 100 items per page
     - Query complexity cost on list fields is amplified:
       orders cost: 20 * count
     - Rate limit by total items returned per minute, not just
       queries per minute

3. GraphQL DoS via expensive queries:
   Attack: send a query with many aliases to exhaust server:
  ```graphql
  query {
    a1: user(id:"1") { ... }
    a2: user(id:"2") { ... }
    ...
    a500: user(id:"500") { ... }
  }
  ```
  Mitigations:
     - Max aliases: 50 per request
     - Query complexity limit catches this (complexity = sum of all
       aliased fields)
     - Request timeout: 10 seconds max query execution time
     - Response size limit: 1MB per response (truncate or reject)

4. SQL injection via GraphQL arguments:
   Attack: attacker sends id: "1; DROP TABLE users"
   Mitigation:
     - Parameterized queries/ORMs for database access (never
       string-concatenate GraphQL arguments into SQL)
     - Input validation with allow-lists for enum values
     - GraphQL's type system handles JSON-level validation, but
       application-level validation is still needed for strings
```

Performance considerations for authorization:

Authorization decisions must not become a bottleneck:
  1. Cache authorization policies in Redis (tenant_id → policy document)
     TTL: 5 minutes, invalidate on policy change
     Reduces policy evaluation from 50ms DB query to 1ms cache lookup

  2. Preload roles and permissions at authentication time:
     The auth token (JWT) includes the user's role and explicit
     permission list. Resolvers check permissions from the token
     without making additional DB calls.

  3. Batch authorization checks:
     When resolving a list of 100 users, don't check authorization
     individually. Batch the check: "is this user authorized for
     all 100 users?" — one check instead of 100.

  4. Bypass authorization for trusted internal callers:
     Service-to-service requests with mutual TLS can skip field-level
     authorization (they already passed stronger network-level auth).
     But still enforce type-level authorization.

Schema design for security:

  Separate public and internal schemas:
    Public schema: accessible by all authenticated users
    Internal schema: accessible by service accounts only
    Admin schema: accessible by admin role only

    Avoid conditionally including fields in the same schema —
    attackers can detect hidden fields through error message
    differences and timing attacks.

  Use dedicated mutation types for sensitive operations:
  ```graphql
  type Mutation {
    updateProfile(input: UpdateProfileInput!): User!
    changePassword(input: ChangePasswordInput!): Boolean!
    # Separate mutation for sensitive operation
    initiatePayment(input: PaymentInput!): PaymentResult!
      @auth(require: "payment_scope")
  }
  ```

    Why separate mutations instead of a single updateUser with an
    optional password field: it creates a clear audit trail for
    sensitive operations, simplifies authorization rules, and
    prevents accidental password changes through a general-purpose
    updateUser mutation.
