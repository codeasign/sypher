## The $2 Million Endpoint

You launch a fintech platform. Users trade stocks, check balances, and manage portfolios through a sleek mobile app. The engineering team works around the clock to ship features. Security takes a back seat — after all, the API is behind authentication, right?

Wrong.

A developer exposes an internal health-check endpoint at `/api/v2/_internal/debug/positions`. No authentication. No rate limiting. No logging. It was meant for internal debugging only. The developer forgot to remove it before the production deploy.

A malicious actor discovers the endpoint through automated scanning. They write a script that calls the endpoint once per second for each of 10,000 simulated user IDs. The endpoint returns real portfolio data: stock holdings, cash balances, margin limits, open orders. The actor builds a complete picture of your users' financial positions.

Then they execute a coordinated attack. They place massive sell orders on thinly traded stocks right before market close, knowing exactly which users hold those positions. The panic selling triggers a cascade. Your users lose $2 million in a single afternoon.

The class-action lawsuit names your company. The regulatory fine follows. The CTO resigns. The postmortem headline reads: **"An unprotected endpoint cost us everything."**

This is not a hypothetical. Similar breaches have happened at real fintech companies, crypto exchanges, and healthcare platforms. In every case, the root cause was the same: the organization treated API security as an afterthought rather than a first-class architectural concern.

You are here to make sure that never happens to a system you design.

## What Is API Security

API security is the practice of protecting APIs from unauthorized access, data exfiltration, injection attacks, and denial of service by enforcing authentication, authorization, input validation, rate limiting, encryption, and monitoring at every layer of the request lifecycle.

Think of API security as the security system for a 50-story office building.

The **lobby guard** checks every person who walks through the front door — is this person allowed to be here at all? That is authentication. The guard asks for ID, checks the ID against a database, and decides whether to let the person past the lobby. The guard does not care where the person is going; the guard only cares that the person is who they claim to be.

Once inside, the person approaches the **elevator bank**. The elevator requires a keycard to activate the buttons for specific floors. Even with a valid ID from the lobby, the person cannot reach floor 40 unless their keycard permits it. That is authorization. Authentication says "you are who you say you are." Authorization says "you may go where your role permits."

On each floor, a **department receptionist** checks that the person has an appointment with someone on that floor. This is input validation — verifying that each request is reasonable and expected before acting on it. If someone walks in claiming to see an employee who does not work there, the receptionist turns them away.

Throughout the building, **CCTV cameras** monitor activity, log every door swipe, and flag unusual behavior. A person who visits 30 different floors in one hour triggers an alert. That is monitoring and anomaly detection — the security team investigates before the person reaches something sensitive.

Your API needs the same layered defense. The lobby guard, the elevator keycard system, the floor receptionists, and the CCTV cameras each handle a distinct security concern. If one layer fails, the others still protect the building. That is defense in depth.

## Core Vocabulary

**API Gateway**
A dedicated server that sits between clients and backend services, acting as the single enforcement point for all security policies. The gateway validates every request before it reaches any service. It runs authentication checks, enforces rate limits, filters malicious payloads, and logs all traffic in one centralized location. In security architecture, the gateway is your first line of defense — the lobby guard.

**Rate Limiting**
A mechanism that restricts how many requests a client can make within a specific time window. Rate limiting prevents brute-force attacks, credential stuffing, and resource exhaustion. If a single IP address sends 10,000 login requests in one minute, rate limiting blocks all but the first 100. Implementations range from simple fixed-window counters to sophisticated token-bucket and sliding-window algorithms backed by Redis.

**Input Validation**
The process of verifying that every piece of data in a request matches expected formats, types, lengths, and ranges before processing it. Input validation prevents injection attacks, malformed data exploits, and buffer overflow attempts. A well-designed API rejects invalid input at the gateway with a clear error message, never forwarding garbage to backend services. Validate everything. Trust nothing.

**SQL Injection**
An attack where the attacker inserts malicious SQL statements into an API input field — typically a form input, query parameter, or header — tricking the database into executing unintended commands. A classic example: a login endpoint that constructs SQL as `SELECT * FROM users WHERE username = '` + userInput + `'` allows the attacker to enter `' OR '1'='1` and bypass authentication entirely. Parameterized queries and prepared statements are the only reliable defenses. Never concatenate user input into SQL strings.

**XSS — Cross-Site Scripting**
An attack where the attacker injects malicious JavaScript into web pages served by your API. When other users view the page, the injected script executes in their browser, stealing cookies, session tokens, or personal data. A comments API endpoint that returns user-submitted HTML without sanitization is a common vector. Output encoding, Content Security Policy headers, and context-aware escaping are your primary defenses. For APIs that serve data to web clients, XSS is a top threat.

**CSRF — Cross-Site Request Forgery**
An attack that tricks an authenticated user into performing an action on your API without their knowledge or consent. The attacker crafts a malicious page that, when the user visits it, triggers a request to your API using the user's existing session cookies. If your API relies on cookies for authentication and does not verify the request's origin, the forged request is processed as legitimate. Anti-CSRF tokens, SameSite cookies, and origin header validation prevent this attack.

**CORS — Cross-Origin Resource Sharing**
A browser security mechanism that controls which origins (domains) are allowed to make requests to your API. When a web app hosted on `evil.com` tries to fetch data from your API at `api.yourplatform.com`, the browser checks the API's CORS headers before allowing the request. Misconfigured CORS — specifically `Access-Control-Allow-Origin: *` combined with `Access-Control-Allow-Credentials: true` — opens your API to cross-origin data theft. Configure CORS as narrowly as possible. Allow only the specific origins your application needs.

**AuthN — Authentication**
The process of verifying that a client is who they claim to be. Authentication answers the question "Who are you?" Common mechanisms include API keys (simple, low-security), JWT tokens (stateless, widely adopted), OAuth2 (delegated authorization with token exchange), and mTLS (mutual TLS using client certificates). Authentication is always the first security check — you cannot authorize an unauthenticated request.

**AuthZ — Authorization**
The process of determining what an authenticated client is permitted to do. Authorization answers "What are you allowed to access?" Role-based access control (RBAC) assigns permissions to roles and roles to users. Attribute-based access control (ABAC) evaluates policies against user attributes, resource attributes, and environment conditions. A request that passes authentication but fails authorization returns a 403 Forbidden.

**API Key**
A simple token — typically a long, randomly generated string — that identifies the client making the request. API keys are passed in headers, query parameters, or request bodies. They are easy to implement but provide weak security because they never expire by default and are often hard-coded in source code. API keys are suitable for identifying projects or applications, but not for authenticating individual users. Treat an API key like a shared office password — useful for basic gates, not vault doors.

**HMAC — Hash-Based Message Authentication Code**
A cryptographic mechanism that uses a shared secret key to sign each request. The client computes a hash over the request content using the secret, includes the hash in a header, and the server recomputes and verifies the hash. HMAC ensures request integrity — if the request body is modified in transit, the hash does not match and the server rejects the request. AWS uses HMAC signing for all API requests. HMAC is more secure than API keys because the secret is never transmitted over the wire — only the hash is sent.

**mTLS — Mutual TLS**
A transport-layer security protocol where both the client and the server present certificates to each other during the TLS handshake. In standard TLS, only the server proves its identity. In mTLS, the client also proves its identity by presenting a certificate signed by a trusted certificate authority. mTLS is the standard for service-to-service communication in zero-trust architectures. It provides encryption (TLS), server authentication, and client authentication in a single protocol handshake.

**WAF — Web Application Firewall**
A security appliance — hardware, software, or cloud-based — that filters, monitors, and blocks HTTP traffic to and from a web application or API. A WAF inspects each request against a set of rules that detect common attack patterns: SQL injection signatures, XSS payloads, path traversal attempts, and known malicious IP addresses. AWS WAF, Cloudflare WAF, and ModSecurity are common implementations. The WAF acts as your outermost security layer, catching known threats before they reach your API gateway.

**DDoS — Distributed Denial of Service**
An attack that overwhelms your API with traffic from multiple compromised systems, making it unavailable to legitimate users. DDoS attacks range from volumetric floods (saturating network bandwidth) to application-layer attacks (sending slow, seemingly legitimate requests that exhaust server resources). Defenses include traffic scrubbing at the network edge, rate limiting per IP, connection limiting, and cloud-based DDoS protection services like AWS Shield or Cloudflare. DDoS is not a vulnerability in your API code — it is an availability attack on your infrastructure.

Each layer filters out a different class of threats. The network edge stops volumetric attacks and known exploit patterns. The API gateway enforces transport security and request quotas. The authentication layer verifies identity and permissions. The application layer validates inputs and enforces business rules. The data layer uses secure queries and access controls to protect the raw data.

No single layer is sufficient. A WAF catches known SQL injection patterns but does not validate that the request is properly authenticated. Authentication confirms identity but does not check whether the request body contains malicious content. Defense in depth means each layer operates independently, and a failure in one layer does not compromise the entire system.

### Step 1: DNS and CDN Edge — DDoS Scrubbing

The request arrives at your DNS resolver, which points to a CDN or DDoS protection service like Cloudflare, AWS Shield, or Akamai. Before the request reaches your infrastructure, the edge service analyzes traffic patterns. Traffic from known malicious IPs is dropped. Traffic that matches volumetric attack signatures (too many requests from too many IPs in too short a time) is scrubbed. Only traffic that passes this initial triage reaches your origin servers.

This layer does not care what the request contains — it only cares about the traffic pattern and the source. A single legitimate request passes through immediately. Ten thousand requests from a botnet are dropped before your servers feel the load.

### Step 2: Web Application Firewall — Pattern Matching

If the request survives the DDoS scrubber, it hits your WAF. The WAF inspects the request body, headers, query parameters, and path against a rule set. The rule set includes signatures for SQL injection (`' OR '1'='1`, `UNION SELECT`, `DROP TABLE`), XSS payloads (`<script>`, `onerror=`, `javascript:`), path traversal (`../../../etc/passwd`), and known CVEs.

A request containing `' OR '1'='1` in a query parameter is blocked with a 403 Forbidden response. The WAF never forwards known attack patterns to your API. This protects against unsophisticated attackers and automated scanning tools. The WAF operates on pattern matching — it cannot catch novel attacks or context-dependent exploits, but it eliminates the noise of automated attacks.

### Step 3: TLS Termination — Transport Security

The request arrives at your API gateway or load balancer over HTTPS. The gateway terminates the TLS connection by decrypting the request. This step ensures that the request was encrypted in transit from the client to your infrastructure and that the client verified your server's certificate.

For mTLS connections, this is also where the gateway verifies the client's certificate against your certificate authority. If the client does not present a valid certificate, the TLS handshake fails and the request never reaches any application code.

After decryption, the gateway may re-encrypt the request with a different certificate for internal forwarding, or forward it over plain HTTP within the trusted network. The choice depends on your internal security requirements.

### Step 4: Rate Limiting — Quota Enforcement

The decrypted request enters the rate limiter. The gateway extracts the client's identifier from the request — typically the IP address, an API key, or a JWT claim. It checks the identifier against a rate limit counter stored in Redis or a similar fast data store.

If the client has exceeded their quota (for example, 1,000 requests per minute), the gateway returns a 429 Too Many Requests response with a Retry-After header indicating when the client can retry. If the client is within their quota, the counter is incremented and the request proceeds.

Rate limiting runs before authentication because it needs minimal processing per request. Rejecting a rate-limited request at this point costs microseconds. Rejecting it after authentication and authorization would cost milliseconds and consume authentication service resources. Rate limiting first is an efficiency optimization and a security best practice.

### Step 5: Authentication — Identity Verification

The request reaches the authentication layer. The gateway or authentication service extracts credentials from the request. The mechanism depends on your API's authentication scheme.

For JWT-based authentication, the gateway validates the token signature using the public key, checks the token expiration (`exp` claim), verifies the issuer (`iss` claim), and confirms the token is not in a revocation list. If any check fails, the gateway returns a 401 Unauthorized response.

For OAuth2, the gateway may exchange an authorization code for an access token by calling the authorization server. For API keys, the gateway looks up the key in a database or cache and verifies it has not been revoked.

Authentication failures must be logged with the IP address, timestamp, and failure reason. A high rate of authentication failures from a single IP signals a brute-force or credential-stuffing attack.

### Step 6: Authorization — Permission Check

Authentication confirmed who you are. Authorization checks where you are allowed to go. The gateway or an authorization service evaluates the authenticated client's permissions against the requested resource and action.

In role-based access control (RBAC), the gateway looks up the client's role (admin, editor, viewer) and checks whether that role permits the requested operation. An editor role may allow reading and writing posts but not deleting users. The gateway returns 403 Forbidden if the role lacks permission.

In attribute-based access control (ABAC), the gateway evaluates policies against multiple attributes: the user's department, the resource's classification level, the time of day, and the client's geographic location. ABAC is more flexible than RBAC but requires a policy engine and adds latency.

Authorization is context-dependent. The same user may have permission to read their own profile but not another user's profile. The authorization layer must evaluate resource ownership, not just role membership.

### Step 7: Input Validation — Payload Sanitization

The authenticated and authorized request now reaches the application layer, where input validation runs. Every field in the request body, every query parameter, and every header is checked against a schema or set of rules.

A POST endpoint that expects a JSON body like `{"email": "user@example.com", "age": 30}` is validated against a schema: `email` must be a valid email format, `age` must be an integer between 0 and 150, no extra fields are allowed. If the body contains `{"email": "user@example.com", "age": "thirty", "role": "admin"}`, validation rejects it — `age` is not an integer and `role` is an unexpected field.

Input validation also applies to file uploads: check file type by inspecting magic bytes (not just the file extension), enforce maximum file size, scan files for malware. Reject requests that fail validation with a 422 Unprocessable Entity response and a clear error message describing which field failed and why.

### Step 8: Business Logic Enforcement

After input validation passes, the request enters your business logic layer. This is where security meets application-specific rules. Business logic enforcement includes:

- **Idempotency checking**: If the request includes an idempotency key, the service looks up whether this key was already processed. Duplicate requests with the same key are rejected, preventing double charges in payment APIs.
- **State validation**: The service checks that the requested state transition is valid. You cannot cancel an order that was already shipped. You cannot withdraw more money than your account balance allows.
- **Quota enforcement**: Per-user or per-tenant limits on resource creation. A free-tier user may create only 10 projects. Creating project number 11 is rejected.
- **Rate-of-change limits**: A trading API may allow only one order modification per second per user, even if the general rate limit allows 100 requests per second.

Business logic enforcement is the hardest layer to implement correctly because it requires deep domain knowledge. Automation tools cannot guess your business rules. You must design and test them yourself.

### Step 9: Secure Query Execution

The request reaches the data layer. Every query against a database must use parameterized queries or prepared statements. Never concatenate user input into SQL strings, even if the input was validated in step 7.

Safe: `SELECT * FROM users WHERE email = ?` with the email passed as a parameter.
Unsafe: `"SELECT * FROM users WHERE email = '" + email + "'"`

Parameterized queries separate the SQL structure from the data. The database treats the parameters as values, never as executable code. An attacker who manages to bypass input validation and send a malicious payload in the email field still cannot inject SQL because the parameterized query prevents the payload from being interpreted as SQL syntax.

The same principle applies to NoSQL databases, LDAP queries, and command execution. If your code constructs a query string by concatenating user input, you have a vulnerability.

### Step 10: Audit Logging

Every request that reaches step 10 is logged with sufficient detail for forensic analysis. The audit log captures:

- Request timestamp, source IP, and client identifier
- Request method, path, and query parameters
- Authentication result (which user, which token)
- Authorization result (which permissions were checked)
- Action performed and resources affected
- Response status code and processing time
- Any security-relevant events (rate limit near threshold, suspicious headers)

Audit logs must be immutable — append-only storage that cannot be modified by an attacker who gains access to the system. Write logs to a separate logging service or a dedicated log database with restricted access. Encrypt logs at rest. Retain logs for the period required by your compliance obligations (typically 1-7 years depending on industry).

Audit logs are not just for post-incident forensics. Monitor logs in real time for patterns that indicate an active attack: a sudden spike in 403 responses, requests to endpoints that should not exist, or a single user accessing resources they have never accessed before.

## Variants of API Security

Different API protocols expose different attack surfaces and require different security configurations. The three most common API protocols in distributed systems — REST, GraphQL, and gRPC — each have unique security considerations.

### REST API Security

REST APIs are the most widely deployed API type. REST over HTTP uses standard verbs (GET, POST, PUT, DELETE, PATCH) and URL path hierarchies. The security model for REST is well understood and has mature tooling.

Key security concerns for REST APIs:

- **Token-based authentication**: JWTs or OAuth2 bearer tokens in the Authorization header. Each request is stateless — the token carries all the information needed to authenticate. No server-side session storage needed.
- **CORS configuration**: REST APIs are frequently consumed by web browsers, making CORS a critical security control. Restrict origins to specific domains. Never use `Access-Control-Allow-Origin: *` with credentials.
- **HTTP method enforcement**: Each endpoint should accept only the HTTP methods that are valid for its purpose. A read-only endpoint that accepts DELETE is a vulnerability waiting to happen.
- **Path traversal protection**: REST URL paths like `/api/users/../../etc/passwd` must be normalized and rejected. The WAF and input validation layer should strip path traversal sequences.
- **Verbose error handling**: REST APIs often return detailed error messages in the response body. An endpoint that returns `User with email 'attacker@test.com' not found` leaks information about which email addresses are registered. Return generic error messages in production.

### GraphQL Security

GraphQL introduces a fundamentally different attack surface compared to REST. Instead of fixed endpoints, GraphQL exposes a single endpoint (typically `/graphql`) where clients specify exactly which data they need in a query. This flexibility creates unique security challenges.

Key security concerns for GraphQL:

- **Query depth limiting**: A GraphQL client can request deeply nested relations: `user -> posts -> comments -> author -> posts -> comments -> ...`. An attacker can craft a deeply nested query that overwhelms the server, even if each individual field is cheap. Set a maximum query depth (typically 5-7 levels for most APIs).
- **Query complexity analysis**: Not all fields are equal. A field that triggers a database join should have higher cost than a field that reads from cache. Implement per-field cost scoring and reject queries whose total cost exceeds a threshold.
- **Batched query attacks**: GraphQL allows multiple queries in a single request via aliases or batching. An attacker can send `query { a: user(id:1), b: user(id:2), c: user(id:3) ... z: user(id:26) }` — 26 queries in one request. Limit the number of queries per request.
- **Introspection disabling**: GraphQL introspection exposes your entire schema — every type, field, argument, and relationship. Disable introspection in production, or restrict it to authenticated admin users.
- **Authorization at the field level**: A REST endpoint that returns a user profile sends a fixed set of fields. GraphQL lets the client request specific fields. If `email` and `ssn` are fields on the User type, a viewer role should not be able to request `ssn` even if they can request `name` and `email`. Implement field-level authorization.

### gRPC Security

gRPC is a high-performance RPC framework that uses Protocol Buffers for serialization and HTTP/2 for transport. gRPC security differs from REST and GraphQL because gRPC is typically used for internal service-to-service communication, not public-facing APIs.

Key security concerns for gRPC:

- **mTLS as the default**: gRPC is designed with mTLS as the primary authentication mechanism. Every gRPC connection uses TLS. With mTLS, each service presents a certificate, and both sides verify each other's identity. This is the standard for zero-trust service meshes (Istio, Linkerd).
- **No browser exposure**: gRPC is not typically consumed by browsers (though gRPC-Web exists). This eliminates CSRF, CORS, and XSS as attack vectors for native gRPC. The threat model shifts from browser-based attacks to service impersonation, man-in-the-middle, and data interception.
- **Stream-based attacks**: gRPC supports bidirectional streaming. An attacker could open a streaming RPC and never send data, keeping the connection alive indefinitely. Set connection timeouts, stream idle timeouts, and maximum message size limits.
- **Proto-based validation**: Protocol Buffers generate strongly typed code. gRPC automatically validates message fields against the proto schema at deserialization time. This provides built-in input validation — malformed messages fail to deserialize before reaching your handler code. Fields that are unknown in the schema are rejected or silently dropped.
- **HTTP/2 multiplexing risks**: gRPC uses HTTP/2 multiplexing — multiple streams over a single TCP connection. A single compromised service could exhaust the HTTP/2 stream ID space, starving other services sharing the same connection. Configure per-stream flow control and connection limits.

### Choosing the Right Security Model

Your choice of API protocol determines which security controls you need and which threats you face:

| Aspect | REST | GraphQL | gRPC |
|---|---|---|---|
| Primary client | Browsers, mobile apps | Web SPAs, mobile apps | Internal services |
| Authentication | JWT, OAuth2, API keys | JWT, OAuth2 | mTLS (primary) |
| CORS required | Yes | Yes | No (internal) |
| CSRF risk | Yes (cookie auth) | Yes (cookie auth) | No |
| Rate limiting | Per-endpoint | Per-query complexity | Per-stream |
| Input validation | Ad-hoc or schema | Schema (GraphQL types) | Proto schema (built-in) |
| Attack surface | Wide (many endpoints) | Narrow (one endpoint) | Narrow (RPC methods) |
| Hardest challenge | Path + method control | Depth + complexity limiting | mTLS certificate management |

In a typical distributed system, you will use all three. REST for public-facing APIs consumed by browsers and mobile apps. GraphQL for complex data-fetching requirements where clients need flexible queries. gRPC for internal service-to-service communication where performance and strong identity matter.

Each protocol requires a different security configuration. The same gateway and WAF protect all three, but the authentication layer, rate limiting logic, and input validation rules differ per protocol.

## Mental Model Check

Test your understanding with these five questions. Think through each one before reading the answer. The goal is to internalize the layered security model, not to memorize facts.

### Question 1

You deploy a REST API that accepts POST requests with JSON bodies. A security scan reveals that your API is vulnerable to SQL injection through the `email` field in the signup endpoint. Which three security layers could have caught this, and in what order would they catch it?

**Answer:** Three layers could catch SQL injection. First, the WAF should catch common SQL injection patterns in the request body — strings containing `' OR '1'='1`, `UNION SELECT`, or `DROP TABLE` trigger a WAF rule. Second, input validation should reject the payload because the `email` field does not match the expected email format — the injected value is not a valid email. Third, the database layer should use parameterized queries, which render the injection harmless even if it passes the WAF and input validation. If any one of these three layers is missing or misconfigured, the injection succeeds. All three must be present for defense in depth.

### Question 2

Your team is designing a public GraphQL API for a social media platform. A user can query `user(id)` to get profile information. What prevents a malicious user from querying `comments -> author -> posts -> comments -> author -> posts` infinitely and overwhelming the server?

**Answer:** Three controls prevent this. Query depth limiting rejects queries that exceed a configurable nesting depth — typically 5-7 levels. Query complexity scoring assigns costs to each field and rejects queries whose total cost exceeds a threshold (a deeply nested query accumulates high cost). Query timeout terminates any query that runs longer than a configurable duration (typically 10-30 seconds for GraphQL). At least the first two controls should be enforced at the GraphQL gateway before the query reaches your resolver logic. The timeout is a safety net at the execution layer.

### Question 3

You are using mTLS for internal service-to-service communication. A developer asks: "Why not just use API keys? They are simpler." The developer's service communicates with three other services. What is the security disadvantage of API keys compared to mTLS in this scenario?

**Answer:** API keys have four disadvantages compared to mTLS for service-to-service communication. First, API keys are transmitted with each request — if an attacker intercepts the request (through a compromised service or network tap), they capture the key and can impersonate the service. In mTLS, the certificate never leaves the service; only a signed proof of possession is transmitted. Second, API keys have no built-in expiration mechanism — a key exposed in a log file is valid forever. mTLS certificates have a configurable validity period (typically 24 hours to 30 days) and must be rotated. Third, API keys provide no encryption — you still need TLS for transport encryption. mTLS provides authentication and encryption in one protocol. Fourth, mTLS offers strong identity binding through certificate SANs (Subject Alternative Names) that specify which service the certificate belongs to. A compromised key grants access to everything the service has access to. A compromised certificate is valid only for the service named in the certificate and expires automatically.

### Question 4

Your API gateway enforces rate limiting at 1,000 requests per minute per user. A user submits 2,000 requests in one minute from the same IP address, and 1,500 are rejected with 429 Too Many Requests. The user complains that legitimate requests were rejected. What is the problem with your rate limiting configuration?

**Answer:** The rate limiter is working correctly — it rejected the excess requests and protected the backend. The issue is that rate limiting operates on a per-IP basis, not a per-user basis. If the user shares an IP address with other users (common in corporate networks, NAT gateways, or mobile networks with carrier-grade NAT), all users behind that IP share the same rate limit. A single abusive user from the same IP can exhaust the limit for all legitimate users. The fix is to rate limit by user identity (JWT claim, API key, or session token) rather than by IP address, or to use a dual rate limit — per-user for authentication-based limits and per-IP as a safety net for unauthenticated requests. For authenticated requests, the rate limiter should use the user ID from the JWT token, not the source IP.

### Question 5

Your API accepts JSON payloads with an `age` field. A user sends `{"age": -5}`. The input validation layer accepts this value because it passes type checking (age is an integer) and range checking (age is between -100 and 200 — your validation allows negative numbers for some reason). The service stores -5 in the database. Later, a reporting system crashes when it tries to compute average age and divides by a negative number. Which security layer should have caught this, and what is the root cause?

**Answer:** The input validation layer should have caught this, but the root cause is that the validation rules were too permissive. Input validation for `age` should reject negative values — an age of -5 is never valid in any domain context. The validation rule allowed a range of -100 to 200, which was likely copied from a "signed integer range" template without domain-specific constraints. The fix is to define validation rules based on domain semantics, not data types. The `age` field should validate for a non-negative integer between 0 and 150 (or whatever the maximum supported age is for your application). Beyond input validation, this is also a business logic failure — the service should have checked that the value makes sense in context before storing it. The lesson: validation rules must encode domain constraints, not just type constraints. A valid integer is not the same as a valid age.
