## Naive Architecture: Single Reverse Proxy

The simplest API gateway architecture is a single reverse proxy instance sitting between clients and a single backend service. This is the starting point for most projects — before the microservices explosion, before the need for rate limiting, before the traffic reaches thousands of requests per second.

A single NGINX or HAProxy instance listens on port 443, terminates TLS, and proxies requests to a backend server. The configuration is trivial: define a location block, point it at the backend address, and you are done.

```nginx
server {
    listen 443 ssl;
    server_name api.example.com;

    location /api/ {
        proxy_pass http://backend:8080/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

This works for a single service with modest traffic. The problems start when you add a second service. The location block pattern repeats, but now you need separate backends for /users, /orders, and /products. Authentication must be duplicated in each service. Rate limiting is nonexistent. Monitoring is per-service with no central view.

The naive architecture hits its limit at roughly 500 requests per second and 3-5 services. Beyond that, you need a purpose-built gateway.

## Production Architecture: Gateway Cluster

The production architecture replaces the single instance with a cluster of gateway nodes behind a load balancer. Each node is identical and stateless on the request path. Configuration is shared via a central database or management server.

### Components

**Global Load Balancer:** Routes traffic to the nearest gateway cluster. Uses DNS-based routing or Anycast. Detects cluster health and stops routing to dead clusters.

**Gateway Cluster:** A group of identical gateway instances (3-10 nodes per region). Each instance runs the same configuration. New instances can be added and removed without downtime.

**Configuration Store:** A database (PostgreSQL for Kong, etcd for Envoy) that holds the shared configuration. Gateway instances poll for changes or receive push updates.

**Redis Cluster:** Stores rate limiter state. The Lua scripts ensure atomic token bucket operations. Redis Sentinel or Redis Cluster provides high availability.

**Service Registry:** A registry (Consul, Eureka, Kubernetes DNS) that maps service names to instance IPs. The gateway queries the registry during the service discovery step.

**Monitoring Stack:** Prometheus collects metrics from each gateway instance. Grafana dashboards show request rates, error rates, and latencies. Alerts fire on anomaly detection.

### How the Cluster Handles a Request

1. The global load balancer receives the request and picks a healthy gateway node (us-east-1b).
2. The gateway node parses the request and runs the middleware chain.
3. For rate limiting, the gateway calls Redis with a Lua script: "read tokens, check, decrement."
4. For authentication, the gateway validates the JWT token using its local JWKS cache (fetched from the config store or an OIDC provider).
5. For routing, the gateway walks its local routing trie (loaded from the config store at startup and updated on config changes).
6. For service discovery, the gateway looks up the target service in Consul or Kubernetes DNS.
7. The gateway calls the backend service, collects the response, transforms it, and sends it to the client.
8. The gateway emits metrics (latency, status count, request count) to Prometheus.

### Cluster Sizing

A three-node cluster in one region handles approximately 30,000 requests per second (10,000 per node) with room for one node failure. For 100,000 requests per second, you need ten nodes across two availability zones.

The sizing formula:
```
nodes_needed = ceil(peak_rps / (node_capacity × (1 - buffer)))
```
Where `buffer` is the spare capacity for failover (typically 30-40 percent).

```
Example:
  Peak RPS = 50,000
  Node capacity = 10,000 RPS
  Buffer = 40%

  nodes_needed = ceil(50000 / (10000 × 0.6))
               = ceil(50000 / 6000)
               = ceil(8.33)
               = 9 nodes
```

Nine nodes across three availability zones give you 90,000 RPS of theoretical capacity with 50,000 RPS of peak load. If one zone goes down (3 nodes lost), the remaining 6 nodes handle 60,000 RPS — enough to absorb the peak.

## Integration Architecture

The gateway does not operate in isolation. It integrates with the broader infrastructure: databases, caches, message queues, external APIs, and identity providers.

### Database Integration

The gateway itself does not directly access business databases. However, it integrates with the configuration database (PostgreSQL, etcd) and authentication databases for token validation and API key management.

Configuration updates flow as follows:
1. An admin updates a route in the management UI
2. The management API writes the new route to PostgreSQL
3. The gateway polls PostgreSQL every 30 seconds for config changes (or receives a webhook notification)
4. The gateway rebuilds its in-memory routing trie atomically
5. The new route is live without any downtime

### Cache Integration

The gateway can cache responses from backend services. A cache layer (Redis or Memcached) sits between the gateway and the backend. The gateway checks the cache before calling the backend.

Cache invalidation is the hard part. The gateway supports:
- TTL-based invalidation: cache entries expire after a set time (60 seconds for user profiles, 300 seconds for product catalog)
- Cache tags: the gateway tags cached responses with entity IDs. When an entity is updated, the backend sends a cache invalidation request to the gateway (or directly to Redis)
- Write-through: the gateway updates the cache when it receives a write request from the client

### Message Queue Integration

For high-latency or async operations, the gateway can publish requests to a message queue (Kafka, RabbitMQ, SQS) instead of calling a backend directly. The gateway returns a 202 Accepted with a tracking ID. The client polls a status endpoint for the result.

This pattern is useful for:
- File processing jobs (upload an image, get a thumbnail later)
- Bulk data exports (request a report, download it when ready)
- Webhook delivery (incoming webhooks are queued, processed asynchronously)

### External API Integration

When the gateway needs to call external third-party APIs (payment gateways, shipping providers, auth providers), it adds special handling:

- Retry with exponential backoff for transient failures
- Circuit breaking if the external API is degraded
- Response caching for idempotent external calls
- Credential rotation for API keys stored in the gateway's config

## Multi-Region Architecture

A global application needs gateways in multiple geographic regions. Each region has its own gateway cluster, its own Redis for rate limiting, and its own service instances.

### Active-Active

Both regions handle traffic simultaneously. Clients are routed to the nearest region by DNS-based global load balancing (Route53 latency routing, Cloudflare geo-routing).

In active-active mode, each region is self-sufficient. Rate limiter state is per-region. A client that switches regions (because the nearest region became unavailable) gets a fresh rate limit counter in the new region. This is acceptable — the 5-10 minute window of slightly higher traffic in the surviving region is preferable to denying all traffic.

Configuration is replicated across regions. The config store in each region is a read replica of a central configuration database. Configuration changes are written to the primary database and propagated to all region replicas.

### Active-Passive

One region handles all traffic. The second region runs hot standby — gateway instances are running and ready but receive no traffic. If the primary region fails, DNS is updated to route traffic to the standby region.

Active-passive is simpler (no cross-region state to manage) but wastes 50 percent of capacity. It is suitable for applications where latency is not critical and failover time under 5 minutes is acceptable.

### Cross-Region Rate Limiting

For strict global rate limits (a client can make at most 1,000 requests per hour across all regions), you need a cross-region rate limiter. The token bucket state lives in a globally replicated Redis cluster (Redis Enterprise with CRDTs or a custom solution).

Global rate limiting adds 20-50 ms of cross-region latency to each request. Most applications use per-region rate limiting and tolerate the small overage when clients switch regions.

## Capacity Planning

Capacity planning for an API gateway answers four questions:

1. How many requests per second at peak?
2. What is the average request size?
3. How many concurrent TLS connections?
4. What is the acceptable latency overhead?

### Request Rate Sizing

Gateway nodes scale linearly. If one node handles 10,000 RPS, two nodes handle 20,000 RPS, assuming the load balancer distributes evenly.

The peak-to-average ratio matters. If your average is 10,000 RPS but your peak is 50,000 RPS (a 5:1 ratio common in e-commerce during flash sales), you provision for the peak, not the average.

```
Tier         Daily Avg RPS    Peak RPS    Nodes (10K/node)
──────────────────────────────────────────────────────────
Small        1,000             5,000       1 node + 1 spare
Medium       10,000            50,000      5 nodes + 2 spare
Large        50,000            250,000     25 nodes + 10 spare
Enterprise   200,000           1,000,000   100 nodes + 40 spare
```

### Memory Sizing

Gateway memory scales with concurrent connections, not request rate. If each request takes 50 ms to process and you receive 10,000 RPS, you have 500 concurrent requests in flight at any moment.

```
RPS            Latency    Concurrency    Memory (50 KB each)
─────────────────────────────────────────────────────────────
10,000         50 ms      500            25 MB
50,000         50 ms      2,500          125 MB
50,000         200 ms     10,000         500 MB
200,000        200 ms     40,000         2 GB
```

Memory is rarely the bottleneck. CPU is the bottleneck for TLS termination and request parsing.

### Network Sizing

The gateway's network interface must handle both ingress and egress traffic. With request aggregation, egress traffic can be larger than ingress traffic (the gateway fetches data from multiple services and returns a combined response).

```
Request size:    2 KB (typical API payload)
Response size:   10 KB (aggregated from 5 services)
Ingress:         10,000 RPS × 2 KB   = 20 MB/s (160 Mbps)
Egress:          10,000 RPS × 10 KB  = 100 MB/s (800 Mbps)
Total:           120 MB/s (960 Mbps)  →  need 1 Gbps NIC
```

For 100,000 RPS, you need 10 Gbps NICs or multiple bonded 1 Gbps NICs.

## Deployment Strategies

### Rolling Deployment

New gateway instances are deployed one at a time. The load balancer removes each old instance from rotation, a new instance takes its place, and traffic is directed to it. Zero downtime.

This works well for stateless gateway clusters. Configuration changes are independent of the deployment — the new instances load the same configuration from the shared config store.

### Blue-Green Deployment

Two identical gateway clusters run simultaneously: blue (current) and green (new). The load balancer points all traffic to blue. When green is fully deployed and tested, the load balancer switches to green. If the switch causes issues, you switch back to blue.

Blue-green requires double the gateway capacity during deployment. This is expensive but provides instant rollback.

### Canary Deployment

A new gateway version is deployed to a small subset of nodes (one out of ten). A small percentage of traffic (5-10 percent) is routed to the canary nodes. The team monitors error rates and latencies. If the canary looks healthy for 15-30 minutes, the new version is rolled out to all nodes.

## Observability

A gateway is a critical piece of infrastructure. When it breaks, everything breaks. Observability is not optional.

### Metrics (Prometheus)

Essential gateway metrics:
- `gateway_requests_total{status, method, route}` — request count by outcome
- `gateway_request_duration_ms{route}` — latency histogram
- `gateway_upstream_duration_ms{service}` — backend call latency
- `gateway_connections_active` — concurrent connection count
- `gateway_rate_limit_exceeded_total{client}` — rate limit hits
- `gateway_circuit_breaker_open{service}` — circuit breaker state

### Logging

Structured JSON logs capture every request for audit and debugging:
```json
{
  "request_id": "abc-123",
  "method": "GET",
  "path": "/v2/users/123",
  "status": 200,
  "latency_ms": 45,
  "upstream": "user-service:8080",
  "client_ip": "203.0.113.5",
  "user_id": "user-456",
  "route_id": "get-user"
}
```

### Tracing

Distributed tracing (OpenTelemetry, Jaeger) traces a request from client through gateway to backend services. The gateway injects trace context into each forwarded request. This lets you see the full picture when debugging latency issues.

A trace shows:
- Time spent in the gateway middleware chain
- Time spent waiting for each backend call
- Time spent in the backend service itself
- Any retries or circuit breaker events

## Auto-Scaling the Gateway Cluster

The gateway cluster must handle traffic spikes without manual intervention. Auto-scaling is the mechanism that adds or removes gateway nodes based on demand.

### Metrics for Scaling

The most reliable scaling metric is CPU utilization. Gateway nodes are CPU-bound (TLS termination, request parsing, plugin execution). When CPU crosses 70 percent, add a node. When CPU drops below 30 percent, remove a node.

Secondary metrics provide earlier warning:
- Request rate per node (trigger at 8,000 RPS per node, scale at 10,000 capacity)
- Connection count per node (trigger at 5,000 concurrent connections)
- Latency p99 (trigger at 10 ms, scale before it hits 20 ms)

### Scaling Policy

```yaml
auto_scaling:
  min_nodes: 3              # per availability zone
  max_nodes: 20             # per availability zone
  scale_up:
    metric: cpu_utilization
    threshold: 70%
    cooldown: 60 seconds    # wait before next scale-up
    increment: 2            # add 2 nodes at a time
  scale_down:
    metric: cpu_utilization
    threshold: 30%
    cooldown: 300 seconds   # 5 minutes before scale-down
    decrement: 1            # remove 1 node at a time
```

The asymmetric cooldown is intentional. Scale up quickly to handle load spikes. Scale down slowly to avoid oscillation.

### Scaling During a Traffic Spike

1. Normal state: 6 nodes, 40 percent CPU, 4,000 RPS each
2. Traffic spike begins: 10,000 RPS arrives. CPU climbs to 75 percent
3. Scale-up triggers: 2 nodes added. New nodes register with the load balancer
4. Traffic redistributes: 8 nodes, 50 percent CPU, 5,000 RPS each
5. Spike continues: CPU climbs again. Another scale-up adds 2 more nodes
6. Steady state: 10 nodes, 55 percent CPU, 6,000 RPS each
7. Spike subsides: CPU drops to 35 percent. Scale-down timer starts
8. After 5 minutes: 1 node removed. CPU rises to 40 percent. Stable

## Disaster Recovery

A gateway cluster can fail in ways that affect entire regions. Disaster recovery planning ensures the system survives catastrophic failures.

### Regional Failure

If an entire AWS region (us-east-1) becomes unavailable, the global load balancer must route traffic to surviving regions. The DNS-based failover detects the failure through health checks.

**Failover timeline:**
- t=0: us-east-1 gateway cluster becomes unreachable
- t=10s: Global load balancer health checks fail (3 consecutive failures)
- t=30s: DNS TTL expires. Clients in us-east-1 re-resolve the gateway hostname
- t=40s: DNS returns eu-west-1 gateway IPs. Clients begin connecting to Europe
- t=60s: All traffic rerouted to eu-west-1 and ap-southeast-1
- t=120s: Surviving clusters auto-scale to handle the additional load (50 percent increase)

**During the failover window:**
- 30 seconds of partial availability (some clients with cached DNS still try us-east-1)
- Surviving regions experience higher latency for former us-east-1 customers (100-200 ms additional due to geographic distance)
- Rate limits are per-region, so us-east-1 clients get fresh counters in the new region

### Configuration Store Failure

If the PostgreSQL configuration database fails, the gateway continues operating with its last known good configuration. No new routes can be added, but existing routes continue working.

The database recovery plan:
1. Promote the read replica to primary (5 minutes)
2. Update the gateway's database connection string in the deployment config
3. Gateway instances reconnect to the new primary within 30 seconds
4. Full configuration management restored

During the recovery window, the ops team cannot add or modify routes. The system is frozen in its current state but continues serving traffic.

### Redis Cluster Failure

If the Redis cluster for rate limiting fails, the gateway degrades gracefully:
1. Each gateway instance detects the Redis connection failure
2. Gateway switches to local in-memory rate limiting
3. The local rate limiter uses a best-effort token bucket with no coordination between instances
4. A warning alert fires: "Rate limiter degraded — Redis unavailable"
5. When Redis recovers, the local rate limiter reconnects and syncs

During the degradation, rate limiting accuracy drops from 99 percent to approximately 80 percent (some clients may exceed limits by hitting multiple gateway instances). This is acceptable for short outages (under 5 minutes).

## Cost Optimization

Gateway costs can be significant at scale. Understanding the cost drivers helps you optimize.

### Self-Hosted vs Managed

| Cost Factor | Self-Hosted (Kong/Envoy) | Managed (AWS API Gateway) |
|---|---|---|
| Compute (100K RPS) | 10 nodes × $200/month = $2,000 | $3.50/million requests = $907,200/month |
| Redis cluster | 3 nodes × $50/month = $150 | Included |
| Database | 1 node × $50/month = $50 | Included |
| Operations | 0.5 FTE × $10,000/month = $5,000 | $0 |
| Total | $7,200/month | $907,200/month |

At 100,000 requests per second (259 billion requests per month), self-hosted gateways are 100x cheaper than managed gateways. The trade-off is operational overhead.

### Cache Optimization

Response caching reduces the number of requests that reach backend services. A 50 percent cache hit rate halves the load on backend services.

Cache tuning:
- Short TTL (10-60 seconds) for user-specific data (profile, recommendations)
- Medium TTL (5-15 minutes) for catalog data (products, pricing)
- Long TTL (1-24 hours) for static data (countries, currencies, configuration)

Each percentage point of cache hit rate saves 1 cent per million requests at the gateway level. For a system doing 10 billion requests per month, improving cache hit rate from 40 percent to 60 percent saves $200,000 per month.

### Data Transfer Costs

The gateway multiplies data transfer costs because it receives data from clients and sends data to backend services, then receives the response and sends it back to the client.

```text
Request path data transfer:
  Client → Gateway: 1 KB (request)
  Gateway → Backend: 1 KB (forwarded request)
  Backend → Gateway: 10 KB (response)
  Gateway → Client: 10 KB (transformed response)
  Total: 22 KB per request (includes internal transfer)

At 100,000 RPS:
  22 KB × 100,000 × 86,400 seconds = 190 TB per day
  At $0.09/GB (AWS data transfer): $17,100 per day
```

Optimization: deploy the gateway and backend services in the same availability zone to reduce data transfer costs. Internal traffic between AZs is charged. Traffic within an AZ is free.

## Security Architecture

The gateway is the most security-critical component in your infrastructure. It is the first point of contact for every external request.

### TLS Configuration

Minimum TLS 1.2, prefer TLS 1.3. Disable weak cipher suites. Use HSTS headers to force HTTPS. The gateway should score an A+ on SSL Labs.

```yaml
tls:
  min_version: "1.2"
  prefer_server_ciphers: true
  ciphers:
    - "TLS_AES_128_GCM_SHA256"      # TLS 1.3
    - "TLS_AES_256_GCM_SHA384"       # TLS 1.3
    - "ECDHE-RSA-AES128-GCM-SHA256"  # TLS 1.2
  hsts:
    max_age: 31536000
    include_subdomains: true
```

### Authentication and Authorization

The gateway supports multiple authentication methods:
- JWT bearer tokens (for mobile and SPA clients)
- API keys in headers (for third-party API consumers)
- OAuth2 / OIDC flows (for web applications)
- Mutual TLS (for service-to-service calls)
- Basic auth (legacy, discouraged)

Authorization is enforced after authentication. The gateway checks the client's claims against the resource's required permissions. For fine-grained access control, the gateway delegates to a policy engine like Open Policy Agent (OPA).

### DDoS Protection

The gateway is the first line of defense against DDoS attacks. Capabilities include:
- Rate limiting per client IP, API key, or token
- Connection limiting (max concurrent connections per client)
- Request size limiting (reject requests with bodies larger than 1 MB)
- Slow loris protection (timeout for slow headers/body reads)
- IP allowlisting for internal endpoints

### Request Validation

The gateway validates incoming requests before forwarding:
- Header validation (Content-Type must match Accept header)
- Body size limits (reject oversized payloads)
- Schema validation (JSON Schema for request bodies)
- SQL injection pattern detection (basic, at the edge)
- Path traversal prevention (strip ".." from paths)

## Summary

Production API gateway architecture replaces a single reverse proxy with a clustered design: multiple stateless gateway nodes behind a global load balancer, backed by a shared configuration store, a Redis cluster for rate limiting, and a service registry for backend discovery.

Integration extends the gateway to caches, message queues, and external APIs. Multi-region deployment uses active-active or active-passive patterns with regional configuration replicas. Capacity planning must account for request rate, memory for concurrent connections, and network bandwidth for aggregated responses.

Observability through metrics, logs, and traces is mandatory — the gateway is a critical path component that touches every request. Security configuration at the gateway level protects the entire backend infrastructure from common attack vectors.
