## Foundation Challenges

### Challenge 1: Rate Limiter Design

A startup has 10,000 API clients. Each client can make 100 requests per second. The gateway runs on a single node. Design a rate limiter that enforces these limits.

**Requirements:**
- Each client gets its own 100 request/second limit
- The system must handle bursts of up to 50 requests in a single second
- Memory usage must stay under 50 MB
- Operations must complete in under 1 millisecond

**Model Answer:**

Use a token bucket algorithm with in-memory storage. Each client has a bucket with capacity 100 and refill rate 100 tokens/second. The bucket is a hash map keyed by client ID.

```
Data structure:
  buckets: {
    "client-1": { tokens: 45, last_refill: 1719876543 },
    "client-2": { tokens: 100, last_refill: 1719876543 },
    ...
  }

Algorithm (per request):
  1. Look up client's bucket in hash map
  2. Calculate elapsed = now - last_refill
  3. tokens = min(100, tokens + elapsed * 100)
  4. If tokens < 1 → reject request (429)
  5. tokens -= 1
  6. Update last_refill = now
  7. Allow request
```

Memory calculation: 10,000 clients × 2 fields (8 bytes each) + hash map overhead ≈ 10,000 × 64 bytes = 640 KB. Well under the 50 MB limit.

Time complexity: O(1) hash map lookup plus arithmetic. Under 1 microsecond.

The burst requirement is handled by the bucket capacity. A client that has not sent requests for 1 second accumulates 100 tokens. They can send 100 requests in the first second, then 1 request per second after that.

### Challenge 2: Route Conflict Resolution

A gateway has two route definitions that conflict:

```
Route A: GET /users/:id      → user-service:getUser
Route B: GET /users/search   → user-service:searchUsers
```

When a client sends `GET /users/search`, which route matches? What data structure prevents this ambiguity?

**Model Answer:**

The trie data structure resolves this. The trie walker always prefers exact matches over parameterized matches. The route tree looks like:

```
/users
  ├── "search" (exact) → Route B
  └── ":id" (parameterized) → Route A
```

When the path `/users/search` is walked, the first segment `/users` matches. The second segment `search` is compared against the children. The exact match `search` takes priority over the parameterized match `:id`. Route B is selected.

The trie enforces the rule: exact matches beat parameterized matches. This is how all production gateways resolve path conflicts. The same rule applies to wildcard routes — they have the lowest priority.

If the administrator defines a route that truly conflicts (same path, same method, different upstreams), the gateway should reject the configuration at validation time. The Admin API should detect the conflict and return an error.

### Challenge 3: Gateway Health Check

Design a health check endpoint for a gateway cluster. The health check must verify that the gateway is fully operational, not just that the process is running.

**Model Answer:**

The health check endpoint (`/health`) returns a 200 OK with a JSON body containing detailed status. The check validates five subsystems:

```json
{
  "status": "healthy",
  "version": "3.2.1",
  "config_version": 1042,
  "checks": {
    "database": { "status": "ok", "latency_ms": 2 },
    "redis": { "status": "ok", "latency_ms": 1 },
    "service_registry": { "status": "ok", "services": 47 },
    "tls_certificate": {
      "status": "ok",
      "expires_in_days": 45
    },
    "connection_pool": {
      "status": "ok",
      "active_connections": 234,
      "idle_connections": 1200,
      "max_connections": 5000
    }
  },
  "uptime_seconds": 864000
}
```

The health check fails (503) if any critical subsystem is degraded:
- Database unreachable (gateway cannot load new configuration)
- Redis unreachable (rate limiting degraded)
- TLS certificate expiring in under 7 days
- Connection pool exhaustion (over 90 percent utilization)

The load balancer uses this health check to remove unhealthy gateway instances from rotation. The health check endpoint is served on a separate port (8080) from the main API traffic (443) so health check requests do not compete with client traffic.

### Challenge 4: Connection Pool Tuning

A gateway handles 50,000 requests per second across 10 backend services. Each service has 10 instances. The average response time is 100 ms. Design the connection pool configuration for the gateway.

**Requirements:**
- No single service's slow response should exhaust the gateway's connections
- The gateway should reuse connections aggressively
- Idle connections should be closed after 30 seconds
- Connections should be re-established every 5 minutes to avoid stale connections

**Model Answer:**

**Connection pool configuration per service:**
```yaml
connection_pool:
  max_connections: 200          # per service
  max_idle_connections: 50      # per service
  idle_timeout: 30s             # close idle connections after 30 seconds
  connection_ttl: 300s          # re-establish every 5 minutes
  max_requests_per_connection: 1000  # close after 1000 requests
```

**Rationale for each parameter:**

`max_connections: 200` — At 50,000 RPS across 10 services, each service handles 5,000 RPS. With 100 ms average response time, each service needs 500 concurrent connections (5,000 RPS × 0.1 seconds). With 10 instances per service, each instance needs 50 connections. Setting max_connections to 200 per service allows 4x headroom for spikes.

`max_idle_connections: 50` — Pre-warm 50 connections per service so they are ready for the next request. This eliminates the TCP handshake latency for the first 50 concurrent requests.

`connection_ttl: 300s` — TCP connections that stay open for hours may encounter middleboxes that drop them silently. Re-establishing connections every 5 minutes prevents silent connection drops.

`max_requests_per_connection: 1000` — HTTP/2 connections multiplex requests. After 1,000 requests, the connection is closed and a new one is opened. This prevents one slow request from holding up subsequent requests on the same connection.

**Bulkheading:**
Each service gets its own pool. Service A's pool of 200 connections is separate from Service B's pool. If Service A becomes slow and its pool fills up, Service B's pool is unaffected. This is the key isolation mechanism.

**Total connection count:**
10 services × 200 max connections = 2,000 connections total. Each connection consumes approximately 5 KB. Total memory for connections: 2,000 × 5 KB = 10 MB. Well within the gateway's memory budget.

### Challenge 5: Gateway Authentication Design

Design an authentication layer for an API gateway that supports three authentication methods: JWT tokens, API keys, and mutual TLS. The gateway must verify authentication in under 3 milliseconds per request.

**Requirements:**
- Support JWT (JSON Web Tokens) with RS256 signature verification
- Support API key validation against a database
- Support mTLS with client certificate validation
- Authentication must complete in under 3 ms at p99
- The gateway handles 20,000 requests per second

**Model Answer:**

**Architecture:**
Use a tiered authentication approach. The first tier is the fastest check. Each subsequent tier is slower but handles more complex cases.

**Tier 1 — mTLS (fastest, under 0.5 ms):**
The TLS handshake validates the client certificate. The gateway extracts the client identity from the certificate's Common Name (CN) or Subject Alternative Name (SAN). No additional validation needed. This is the fastest authentication method because it happens during the TLS handshake, before the request is even parsed.

**Tier 2 — API Key (1-2 ms):**
The gateway reads the API key from the X-API-Key header. It looks up the key in a local Redis cache (hash table, O(1)). If found, the client is authenticated. If not found, the gateway queries the authentication database (PostgreSQL) and caches the result in Redis for 5 minutes.

**Tier 3 — JWT (2-3 ms):**
The gateway reads the JWT from the Authorization header. It decodes the JWT header to find the key ID (kid). It looks up the corresponding public key from a local JWKS cache (downloaded from the OIDC provider at startup and refreshed every hour). It verifies the JWT signature using the cached public key. If the signature is valid and the token is not expired, the client is authenticated.

**Caching strategy:**
- JWKS keys: cached in memory, refreshed every 60 minutes
- API key lookups: cached in Redis, TTL of 5 minutes
- Revoked token list: cached in Redis, TTL of 1 minute
- First-time API key validation: hits PostgreSQL, result cached for 5 minutes

**Performance budget:**
```
Authentication method          p50        p95        p99
─────────────────────────────────────────────────────────
mTLS (certificate CN)          0.2 ms     0.4 ms     0.5 ms
API key (Redis cache hit)      0.8 ms     1.2 ms     1.5 ms
API key (cache miss, DB)       5.0 ms     8.0 ms    12.0 ms
JWT (local JWKS cache)         1.5 ms     2.0 ms     2.5 ms
```

The 3 ms target is met for the common cases (cache hits). Cache misses are rare (under 1 percent of requests) and the higher latency is acceptable for the occasional miss.

## Design Challenges

### Challenge 6: Multi-Region Gateway for a Global E-Commerce Platform

Design a gateway architecture for a global e-commerce platform serving 50 million monthly active users across North America, Europe, and Asia-Pacific.

**Requirements:**
- 200,000 requests per second at peak (Black Friday)
- P99 latency under 100 ms for the complete request (client to gateway to backend to client)
- 99.99 percent availability
- Users in one region should not see degraded performance when another region has a failure
- Rate limiting is global (10,000 requests per hour per user, regardless of region)

**Model Answer:**

**Architecture:**

Deploy active-active gateway clusters in three regions: us-east-1, eu-west-1, and ap-southeast-1. Each cluster has 20 gateway nodes behind a regional load balancer. A global DNS load balancer (Route53 latency routing) directs each user to the nearest region.

**Per-region components:**
- 20 Envoy gateway nodes (auto-scaled to 40 during peak)
- Redis cluster for local rate limiting (fast path, 1 ms latency)
- PostgreSQL read replica for configuration
- Full microservice deployment

**Cross-region components:**
- Central configuration primary database in us-east-1 (async replication to eu-west-1 and ap-southeast-1 read replicas)
- Cross-region Redis replication for global rate limiting (Redis Enterprise CRDTs, eventually consistent)
- Global load balancer with health-based failover

**Rate limiting strategy:**
- Primary: local rate limit check per region (Redis, 1 ms)
- Secondary: cross-region Redis sync every 5 seconds
- The effective limit is 10,000 requests/hour per user with a small overage window (5-10 percent) when a user switches regions

**Capacity planning:**
- Peak: 200,000 RPS across 3 regions = 67,000 RPS per region
- Each Envoy node handles 10,000 RPS at p99 under 5 ms
- Nodes per region: ceil(67,000 / 10,000) = 7 nodes, plus 40 percent buffer = 10 nodes, plus one-AZ failure = 20 nodes

**Regional failure handling:**
If us-east-1 fails completely:
1. Global DNS updates within 30 seconds (Route53 health check)
2. us-east-1 traffic is rerouted to eu-west-1 and ap-southeast-1
3. Surviving regions handle 100,000 RPS each (50 percent above normal)
4. Auto-scaling adds 10 nodes per surviving region within 2 minutes
5. During the 2-minute window, the surviving gateways perform load shedding (5 percent of traffic gets 503 with Retry-After of 1 second)

### Challenge 7: Migrating from Monolith to Microservices with a Gateway

A company has a monolithic API serving 10 million daily active users. The monolith is deployed on 20 EC2 instances behind an ELB. The engineering team wants to extract five microservices (auth, users, products, orders, notifications) over 6 months without breaking existing clients.

Design the migration strategy using an API gateway as the enabler.

**Model Answer:**

**Phase 1 — Gateway in Front of Monolith (Week 1):**
Deploy an API gateway (Kong) in front of the ELB. All traffic routes through the gateway to the monolith. No changes to clients. The gateway adds request logging and rate limiting immediately.

Configuration:
```yaml
services:
  - name: monolith
    url: http://internal-elb:8080
routes:
  - paths: ["/"]
    service: monolith
```

**Phase 2 — Extract Auth Service (Week 3):**
Extract the authentication logic from the monolith into a new auth-service. Add a route in the gateway:

```yaml
routes:
  - paths: ["/v2/auth/login", "/v2/auth/register"]
    service: auth-service
  - paths: ["/v1/auth/*"]
    service: monolith
```

New clients use /v2/auth endpoints. Old clients continue using /v1/auth. The monolith still handles auth for old clients.

**Phase 3 — Extract User Service (Week 6):**
Extract user management. The gateway now has three routes:

```yaml
routes:
  - paths: ["/v2/auth/*"]
    service: auth-service
  - paths: ["/v2/users/*"]
    service: user-service
  - paths: ["/v1/*"]
    service: monolith
```

**Phase 4 — Repeat for Remaining Services (Weeks 8-20):**
Extract products, orders, and notifications one at a time. Each extraction follows the same pattern: new v2 endpoint on the new service, old v1 endpoint on the monolith, clients migrate at their own pace.

**Phase 5 — Decommission Monolith (Week 24):**
Once all traffic is on v2 endpoints and no clients use v1, the monolith is shut down. The gateway now routes all traffic to the five microservices.

**Key considerations:**
- Data migration: Shared data between monolith and new services requires a shared database during migration (dual-writes with a sync mechanism, or a shared database schema)
- Client migration: Provide a migration window of 4 weeks per service. Monitor old endpoint traffic. When it drops below 1 percent, contact the remaining clients and migrate them manually
- Rollback: If a new service has issues, the gateway can route its traffic back to the monolith by changing one route definition

### Challenge 8: Designing a Gateway for a Serverless Architecture

Design an API gateway for a serverless architecture using AWS Lambda. The gateway must handle 100,000 requests per second, support WebSocket connections, and integrate with Lambda for request processing.

**Model Answer:**

Use AWS API Gateway (HTTP API type) as the frontend, with Lambda function integrations for each endpoint.

**Architecture:**

```
Clients → AWS API Gateway (HTTP API) → Lambda Functions → DynamoDB / S3 / SQS
```

**Route configuration:**

```
POST /v2/users          → create-user-lambda (Lambda)
GET  /v2/users/{id}     → get-user-lambda      (Lambda)
POST /v2/orders         → create-order-lambda   (Lambda + SQS for async processing)
WS   /v2/notifications  → connection-lambda     (WebSocket)
```

**Key design decisions:**

**API Gateway type:** HTTP API over REST API. HTTP API costs $1.00 per million requests vs $3.50 for REST API. The cost savings at 100,000 RPS (259 billion requests per month) is $648,000 per month. HTTP API latency is 30-50 ms vs 50-100 ms for REST API.

**Lambda concurrency:** At 100,000 RPS with 200 ms average Lambda execution time, concurrent Lambda invocations = 100,000 × 0.2 = 20,000. Lambda has a default concurrency limit of 1,000 per region. Request a concurrency limit increase to 25,000 from AWS support.

**Caching:** Enable API Gateway caching for read-heavy endpoints. Use a 5-minute TTL for user profiles and product catalog. Cache hit rate targeted at 60 percent, reducing Lambda invocations to 40,000 RPS.

**WebSocket handling:** The WebSocket API maintains persistent connections. The connection lambda handles connect/disconnect events. Messages are routed through the gateway to the appropriate Lambda function. Use DynamoDB to store connection IDs for broadcasting.

**Rate limiting:** Use API Gateway's per-route usage plans. Set a limit of 10,000 requests per second per API key for premium customers, 1,000 for standard customers.

**Cold start mitigation:** Use Lambda Provisioned Concurrency for the most latency-sensitive endpoints (login, checkout). Set Provisioned Concurrency to 5,000 for critical functions. This adds $0.000004 per second per function but eliminates cold starts.

**Monitoring:** CloudWatch metrics for API Gateway (4XX count, 5XX count, latency, integration latency). Lambda metrics (invocations, duration, throttles, concurrent executions). CloudWatch dashboards for real-time monitoring.

## Expert Challenges

### Challenge 9: Distributed Rate Limiter with Global Consistency at 1 Million RPS

Design a distributed rate limiter for a global API gateway handling 1,000,000 requests per second. The rate limit must be globally consistent: a client's limit of 1,000 requests per minute must be enforced across all gateway nodes in all regions. The acceptable error margin is under 0.1 percent.

**Model Answer:**

This is a hard problem. Standard Redis-based rate limiting with eventual consistency produces 5-10 percent error margins at this scale. Achieving under 0.1 percent error requires a fundamentally different approach.

**Architecture: CRDT-based rate limiting with local counters and periodic reconciliation.**

**Design:**

Each client's rate limit is tracked independently in each region. The limit is split proportionally: for a limit of 1,000 requests per minute across 3 regions, each region gets 333 requests per minute. Within each region, the limit is enforced by the regional Redis cluster using token buckets.

When a client's traffic shifts between regions (for example, user travels from US to Europe), the regional limits are rebalanced. The rebalancing happens every 60 seconds based on observed traffic patterns.

**Rebalancing algorithm:**

```
1. Every 60 seconds, each region reports its current usage:
   us-east-1: 420 requests this minute (out of 333)
   eu-west-1: 120 requests this minute (out of 333)
   ap-southeast-1: 60 requests this minute (out of 333)

2. Central coordinator calculates new allocations:
   Total observed: 600 requests
   Expected: 1,000 requests/minute
   Remaining capacity: 400 requests

3. Allocate remaining capacity proportionally:
   us-east-1: 60% of traffic → 240 of remaining 400
   eu-west-1: 30% of traffic → 120 of remaining 400
   ap-southeast-1: 10% of traffic → 40 of remaining 400

4. New allocations for next minute:
   us-east-1: 333 + 240 = 573
   eu-west-1: 333 + 120 = 453
   ap-southeast-1: 333 + 40 = 373
```

**Error analysis:**
- During the 60-second rebalancing window, a region may exceed its allocation by up to 50 percent
- At 1,000 requests/minute, the worst-case overage is 500 requests over one minute, or 0.83 requests/second
- Over 1,000,000 RPS, this represents 0.000083 percent overage — well under the 0.1 percent target

**Data structures:**
- Local Redis: token bucket per client per region (standard Lua script)
- Central coordinator: CRDT counter per client per region (replicated across three coordinator nodes for fault tolerance)
- Communication: gRPC streams between regional coordinators and the central coordinator

**Performance:**
- Local rate limit check: 1 ms (Redis, local region)
- Rebalancing: 100 ms every 60 seconds (negligible)
- Total overhead per request: 1 ms (local check only, no cross-region calls on the hot path)

### Challenge 10: Gateway for a Real-Time Trading Platform

Design an API gateway for a high-frequency trading platform. The requirements are extreme: sub-100-microsecond p99 latency, zero tolerance for gateway-induced failures, and 500,000 messages per second.

**Model Answer:**

This design abandons HTTP entirely. HTTP's protocol overhead, header parsing, and connection management add too much latency for HFT.

**Protocol:**
Use a custom UDP-based protocol with a fixed-length binary header. No TLS (trading networks are private and physically secured). No connection management (UDP is connectionless). The gateway processes messages without allocating memory on the hot path.

**Architecture:**

```
Trading Clients → Custom UDP Gateway → Matching Engine → Market Data Feed
```

The gateway is a single-threaded C++ application running on a dedicated bare-metal server with a 10 Gbps NIC. The thread uses a busy-spin event loop (no kernel context switches). Each CPU core is isolated (isolcpus kernel parameter) with no other processes competing for CPU time.

**Message processing pipeline (all in the same thread, no locks):**

```
1. Receive UDP datagram from NIC (zero-copy via DPDK)
2. Parse fixed-length binary header (12 bytes, no allocation)
3. Authenticate client ID (precomputed hash table, O(1))
4. Rate limit check (preallocated counter per client, O(1))
5. Route to matching engine (shared memory ring buffer, O(1) enqueue)
6. Emit acknowledgement (preallocated response buffer, write to NIC)
```

**Total pipeline time: under 100 microseconds.**

**Key design decisions:**

**No TLS:** TLS handshake adds 30-150 ms. TLS per-message encryption adds 5-50 microseconds. The trading platform uses a private network with physical security. No TLS needed.

**No HTTP:** HTTP header parsing adds 1-5 microseconds. HTTP body parsing adds more. A custom binary protocol with fixed-length fields adds under 0.1 microseconds.

**No kernel networking:** Standard Linux networking (TCP/IP stack) adds 10-50 microseconds per packet. DPDK (Data Plane Development Kit) bypasses the kernel entirely, reducing latency to under 1 microsecond per packet.

**Single-threaded:** Multi-threading adds synchronization overhead (locks, atomics). A single thread on a dedicated core avoids all synchronization. The thread uses busy-spin (polling) instead of interrupts to avoid context switch latency.

**No heap allocation:** Memory is preallocated at startup. Message buffers, client tables, and rate limiter counters are statically allocated. No malloc/free on the hot path. No garbage collection.

**No logging on the hot path:** Logging is a blocking I/O operation. All logging happens asynchronously. The gateway writes to a ring buffer. A separate thread reads from the ring buffer and writes to disk.

**Redundancy:**
Primary and backup gateway instances run in parallel on separate hardware. The backup receives the same UDP datagrams (via multicast) and maintains the same state. If the primary fails (detected by a hardware watchdog timer), the backup takes over with zero state loss. The failover is transparent to clients.

**Capacity:**
500,000 messages per second, each 100 bytes. Total throughput: 50 MB/s, well within a single 10 Gbps NIC (1.25 GB/s). A single core at 3 GHz processes each message in 6,000 CPU cycles — easily within the 100-microsecond budget.

### Challenge 11: Gateway Configuration for a Zero-Trust Architecture

Design an API gateway configuration for a zero-trust security architecture. Every request must be authenticated, authorized, and encrypted regardless of its origin (internal or external).

**Requirements:**
- No request is trusted based on network location
- Every request must carry a valid JWT, mTLS certificate, or both
- Authorization is evaluated per-request against a policy engine
- All traffic between gateway and backend services is encrypted with mTLS
- The gateway must not decrypt and re-encrypt payloads unnecessarily

**Model Answer:**

**Architecture:**
The gateway operates in a zero-trust model where every request, regardless of source IP, must pass authentication and authorization checks.

**Ingress authentication:**
- External clients: JWT in the Authorization header, validated against the gateway's JWKS cache
- Internal clients: mTLS certificates issued by the internal certificate authority, validated during the TLS handshake
- Service-to-service: mTLS with SPIFFE-compliant identities (spiffe://example.com/ns/default/sa/payment-service)

**Egress authentication:**
The gateway forwards requests to backend services using mTLS. Each backend service has its own certificate. The gateway verifies the backend's certificate before forwarding the request.

**Per-request authorization:**
After authentication, the gateway evaluates authorization using Open Policy Agent (OPA). The OPA policy checks:
- The client's role (from JWT claims or certificate attributes)
- The requested resource (HTTP method + path)
- The resource's required permissions (from the route configuration)
- Any contextual attributes (time of day, client IP, request size)

Example OPA policy (Rego language):
```rego
default allow = false

allow {
  input.method = "GET"
  input.path = ["users", user_id]
  input.user_id = user_id  # users can access their own profile
}

allow {
  input.method = "GET"
  input.path = ["users", user_id]
  input.role = "admin"  # admins can access any profile
}
```

**mTLS without payload decryption:**
For sensitive data, the gateway can forward the encrypted payload without decrypting it. The client encrypts the payload with the backend service's public key. The gateway validates the envelope (authentication, integrity) without decrypting the payload. The backend service decrypts the payload after receiving it from the gateway.

**Configuration:**
```yaml
zero_trust:
  ingress:
    - protocol: mTLS
      ca_cert: /etc/certs/internal-ca.pem
      require_client_cert: true
    - protocol: TLS + JWT
      require_jwt: true
      jwks_url: https://auth.example.com/.well-known/jwks.json
  egress:
    protocol: mTLS
    verify_backend_cert: true
    sni: "*.svc.cluster.local"
  authorization:
    engine: opa
    policy_url: http://opa:8181/v1/data/gateway/authz
    cache_ttl: 60s
```

## Summary

The five foundation challenges covered rate limiter design with token buckets, route conflict resolution with tries, gateway health checks, connection pool tuning, and authentication layer design. The three design challenges covered multi-region e-commerce gateways, monolith-to-microservices migration with the strangler fig pattern, and serverless gateway architectures. The three expert challenges pushed the boundaries with CRDT-based global rate limiting at 1,000,000 RPS, a sub-100-microsecond UDP gateway for high-frequency trading, and a zero-trust gateway configuration.
