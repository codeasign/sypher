## Netflix: Zuul — The Gateway That Handles 10 Million Requests Per Second

Netflix built one of the earliest and most influential API gateways. In 2012, they extracted Zuul from their edge service layer and open-sourced it. Today, the Netflix API gateway handles over 10 million requests per second at peak, serving 200+ million subscribers across every device type.

### The Problem

Netflix's backend is a sprawling system of 1,000+ microservices. Each service handles a specific domain: recommendations, search, playback, billing, profiles, ratings, content metadata, subtitles, and personalization. Clients — web browsers, mobile apps, smart TVs, gaming consoles, and streaming sticks — all need access to these services.

Before Zuul, each client team built its own server-side integration layer. Each team reimplemented authentication, routing, error handling, and logging. The duplication was unsustainable. A new device type required months of backend work.

### The Architecture

Zuul is a JVM-based gateway built on Netflix's own asynchronous framework. It uses a filter chain architecture where each filter is a Groovy script that runs at a specific point in the request lifecycle.

**Filter types:**
- **Pre-filters:** Authentication, rate limiting, request routing, header injection
- **Route filters:** Forward the request to the backend service (built on Netflix's Hystrix circuit breaker and Ribbon load balancer)
- **Post-filters:** Response transformation, compression, header manipulation
- **Error filters:** Handle failures and return fallback responses

Each filter is a Groovy script that can be changed at runtime without restarting the gateway. This is the key innovation — Zuul can change its behavior in milliseconds by deploying a new filter script.

```
Zuul filter example (simplified):
  class RateLimiter extends ZuulFilter {
    String filterType() { return "pre" }
    int filterOrder() { return 1 }

    boolean shouldFilter() { return true }

    Object run() {
      String clientId = RequestContext.currentContext.getRequest()
        .getHeader("X-Netflix-Client-Id")
      if (rateLimitExceeded(clientId)) {
        throw new ZuulException(429, "Rate limit exceeded")
      }
      return null  // continue the filter chain
    }
  }
```

### Key Lessons from Netflix

**Filters are the right abstraction.** Zuul's filter architecture lets teams deploy routing logic independently of the gateway binary. A rate limiting change does not require a gateway restart. A new authentication scheme can be tested on 1 percent of traffic by deploying a canary filter.

**Asynchronous matters.** Zuul 2 is fully asynchronous (Netty-based). Synchronous gateways waste threads waiting for I/O. Netflix's traffic volume requires thousands of concurrent connections per node. Asynchronous I/O lets a single thread handle hundreds of connections, reducing the node count by 50 percent compared to the synchronous Zuul 1.

**Monitoring is the foundation.** Netflix's gateway emits detailed metrics for every filter, every backend call, and every failure. They knew about problems before their customers did. Their Chaos Monkey regularly tested gateway behavior under failure conditions.

**The gateway must be fast.** Netflix measured the gateway's latency contribution in microseconds, not milliseconds. A 10-millisecond gateway overhead multiplied by 10 million requests per second means 100,000 seconds of additional latency per second — or 100,000 CPU-seconds wasted on overhead. They optimized every microsecond.

## Amazon: AWS API Gateway — Managed Gateway at Cloud Scale

AWS API Gateway is a fully managed service. You do not provision servers, configure clusters, or manage Redis instances. You define your API in a YAML or JSON specification (OpenAPI / Swagger), deploy it, and AWS handles the rest.

### The Architecture

Behind the scenes, AWS API Gateway runs on a fleet of proxy servers distributed across AWS regions. Each proxy server handles TLS termination, route matching, and request forwarding. The control plane manages API definitions, deployments, and monitoring.

**API Types:**
- **HTTP API:** Low-cost, low-latency proxy for HTTP backends (ALB, Lambda, HTTP endpoints). 30-50 ms added latency at p50.
- **REST API:** Feature-rich API with request validation, API keys, usage plans, caching, and SDK generation. 50-100 ms added latency at p50.
- **WebSocket API:** Bidirectional communication for real-time applications. Persistent connections with pub/sub messaging.

**Key Features:**
- **Request validation:** Validates request bodies against a JSON Schema. Returns 400 for malformed requests before they reach your backend.
- **Usage plans:** Define rate limits and quota limits per API key. Useful for monetizing APIs or tiered access.
- **Caching:** Optional cache layer (managed Redis) that caches responses per endpoint. Configurable TTL from 0 to 3600 seconds.
- **Canary deployments:** Route a percentage of traffic to a different deployment stage. Validate changes before full rollout.
- **WAF integration:** AWS WAF attaches to the API Gateway to filter common web exploits (SQL injection, XSS).

### Pricing Model

```
API Type     Per million requests   Cache (1.6 GB/hour)
──────────────────────────────────────────────────────
HTTP API     $1.00                  Not available
REST API     $3.50                  $0.068/hour
WebSocket    $1.00 (connect) +      Not available
             $0.25 (messages)
```

At 10 million requests per month, the difference between HTTP API ($10) and REST API ($35) is significant. For high-volume workloads, HTTP APIs are 70 percent cheaper.

### Key Lessons from AWS

**Managed services shift operational burden.** AWS API Gateway eliminates the need to maintain gateway clusters, Redis instances, and configuration databases. The trade-off is less control — you cannot add custom plugins or inspect the underlying infrastructure.

**Latency is the cost of management.** AWS API Gateway adds 30-100 ms of latency compared to 1-6 ms for a self-hosted Kong or Envoy. For latency-sensitive applications, a self-hosted gateway may be better.

**The OpenAPI specification becomes the source of truth.** AWS API Gateway imports OpenAPI specs directly. The spec defines the routes, request/response models, authentication, and integration targets. The spec is the API contract between frontend and backend teams.

## Kong: The Open-Source Gateway Powering Enterprise APIs

Kong is the most popular open-source API gateway. It is built on OpenResty (NGINX + LuaJIT) and uses a plugin architecture that supports Lua, Go, and JavaScript plugins.

### The Architecture

Kong has two components: the gateway nodes (data plane) and a management layer (control plane).

**Kong Gateway (Data Plane):** NGINX workers that process requests through a plugin chain. Each worker is an NGINX worker process with a LuaJIT VM that executes plugins.

**Kong Manager / Admin API (Control Plane):** The RESTful Admin API manages services, routes, plugins, and upstreams. Changes are written to the database and propagated to all gateway nodes.

**Database:** PostgreSQL for production. Cassandra for high-availability multi-datacenter deployments.

**Plugin Hub:** 200+ pre-built plugins for authentication, security, traffic control, serverless, analytics, and transformations.

### Enterprise Use Cases

Kong is used by 10,000+ organizations including Samsung, Siemens, Cisco, and Yahoo Japan.

**Samsung:** Kong manages APIs for Samsung SmartThings, handling authentication and routing for 50 million connected devices. Each device sends telemetry through Kong, which authenticates the device, validates the payload, and routes it to the appropriate smart home service.

**Yahoo Japan:** Kong powers the API platform for Yahoo Japan's e-commerce, news, and auction services. They run 100+ Kong nodes across multiple regions, handling 30,000 requests per second.

## Envoy: The High-Performance Service Proxy

Envoy was built by Lyft and donated to CNCF. It is a high-performance L7 proxy designed for service mesh architectures. While not strictly an API gateway, Envoy is increasingly used as the data plane for gateway solutions.

### Performance

Envoy is written in C++ and uses an event-driven architecture with a thread-per-core model. Each worker thread runs an event loop that handles connections independently with no shared state. This eliminates lock contention.

```
Thread-per-core model:
  Core 0: Event loop → handles 10,000 connections
  Core 1: Event loop → handles 10,000 connections
  Core 2: Event loop → handles 10,000 connections
  Core 3: Event loop → handles 10,000 connections

  Result: 40,000 concurrent connections, 50,000 RPS
  P99 latency added: under 3 ms
```

### xDS Protocol

Envoy introduced the xDS (Discovery Service) protocols that revolutionized how gateways are configured. Instead of polling a database, Envoy receives configuration updates over gRPC streams from a management server.

**xDS types:**
- **LDS (Listener Discovery):** Which ports and protocols to listen on
- **RDS (Route Discovery):** Routing rules (path → cluster)
- **CDS (Cluster Discovery):** Backend service definitions
- **EDS (Endpoint Discovery):** Individual backend instances
- **SDS (Secret Discovery):** TLS certificates

The management server pushes configuration changes to all Envoy instances simultaneously. Configuration convergence happens in seconds instead of minutes.

### Envoy as API Gateway

While Envoy is typically a sidecar proxy in service mesh, it also works as a standalone gateway. Projects like Contour (Kubernetes ingress), Ambassador (API gateway), and Gloo (API gateway) are built on top of Envoy.

Envoy's gateway features include:
- Full L7 routing (path, header, method, query parameter matching)
- Rate limiting (via the global rate limit filter, backed by Redis)
- Circuit breaking (per-cluster configurable thresholds)
- Retry policies (configurable per route)
- TLS termination with SNI-based certificate selection
- Access logging (structured, configurable format)
- Distributed tracing (OpenTelemetry, Zipkin, Jaeger)
- WebSocket and HTTP/2 support
- gRPC passthrough

## Traefik: Cloud-Native Gateway with Automatic Service Discovery

Traefik was built for the container era. It integrates natively with container orchestrators (Kubernetes, Docker Swarm, Nomad) and automatically discovers new services without manual configuration.

### Automatic Discovery

When a new service is deployed in Kubernetes, Traefik automatically detects it via the Kubernetes API. The service's Ingress or CRD definitions tell Traefik how to route traffic to it. No Admin API calls, no database writes, no configuration reloads.

```
Kubernetes Ingress → Traefik discovers automatically:
  apiVersion: networking.k8s.io/v1
  kind: Ingress
  metadata:
    name: user-service-ingress
  spec:
    rules:
      - host: api.example.com
        http:
          paths:
            - path: /users
              pathType: Prefix
              backend:
                service:
                  name: user-service
                  port: 8080
  → Traefik creates route: api.example.com/users → user-service:8080
  → No manual gateway configuration needed
```

### Key Features

- **Automatic HTTPS:** Traefik integrates with Let's Encrypt and automatically provisions TLS certificates for every domain it discovers
- **Middleware chain:** Similar to Kong's plugins but configured as Kubernetes CRDs or Docker labels
- **Dashboard:** Built-in web UI showing routes, services, and health status
- **Multiple providers:** Kubernetes, Docker, Consul, etcd, Redis, ZooKeeper, file-based

## Lessons Learned from Production Deployments

### Lesson 1: Start Simple, Evolve Slowly

Every production gateway deployment the team has studied started with a simple reverse proxy (NGINX or HAProxy). They added features one at a time as the need arose: first authentication, then rate limiting, then aggregation, then circuit breaking.

Teams that started with a full-featured gateway from day one spent months configuring features they did not yet need. The complexity overhead delayed their initial launch.

### Lesson 2: Measure the Gateway's Latency Budget

Before adding a new feature to the gateway, measure its latency cost. A new authentication plugin that adds 10 ms may be fine at 1,000 RPS but costs 10,000 ms of wall-clock time per second at 1,000,000 RPS.

Netflix's rule: every gateway filter must justify its latency cost. If a filter adds more than 1 ms on average, it must be optional (applied only to the routes that need it).

### Lesson 3: The Gateway Is the First Thing to Fail Under Load

In every major outage the team studied, the gateway was the first component to fail. Not the database, not the backend services — the gateway. It is the frontline component that absorbs the initial shock of every traffic spike.

The mitigation is not "make the gateway stronger" but "make the gateway fail gracefully." Load shedding, circuit breaking, and bulkheading prevent the gateway from taking down the rest of the system when it fails.

### Lesson 4: Separate Configuration from Code

Gateway configuration changes more frequently than gateway code. Routes are added when a new service is deployed. Rate limits change when traffic patterns shift. Auth schemes change when security requirements evolve.

Treat configuration as a separate artifact with its own CI/CD pipeline. Validate config changes in a staging environment before applying them to production. Every config change should be auditable and revertible.

### Lesson 5: Know When to Replace the Gateway

A gateway that was right for your system two years ago may be wrong today. Teams outgrow gateways in two directions:

**Outgrowing the simple gateway:** Your NGINX configuration has grown to 2,000 lines with custom Lua scripts for authentication and rate limiting. You need plugin support, an Admin API, and better observability. Replace it with Kong, Envoy, or a managed service.

**Outgrowing the complex gateway:** Your team spends more time managing the gateway than building features. You have 50 plugins enabled and the gateway's latency overhead doubled. Simplify: move to a managed gateway (AWS API Gateway) or split into per-service BFFs.

## GCP Apigee: Enterprise API Management

Google Cloud's Apigee is an API management platform that includes an API gateway. It targets enterprise organizations that need API analytics, developer portals, monetization, and security policies.

### Key Features

**API Analytics:** Apigee provides built-in analytics dashboards showing API usage, error rates, latency distributions, and developer adoption. The analytics layer processes billions of API calls and surfaces trends without custom instrumentation.

**Developer Portal:** Apigee includes a portal where third-party developers can discover APIs, read documentation, register for API keys, and test endpoints. The portal reduces the support burden for API providers.

**Security Policies:** Apigee offers 50+ pre-built policies including:
- OAuth v2.0, SAML, JWT, and LDAP authentication
- JSON and XML threat protection
- API key validation and rotation
- Traffic rate limiting with spike arrest

**API Monetization:** Enterprises can charge for API access with usage-based, package-based, or rate-card pricing models. Apigee tracks usage per developer and generates invoices.

### Performance Profile

Apigee adds 15-50 ms of latency per request. It is not intended for sub-10-millisecond use cases. The latency comes from the policy evaluation engine, analytics pipeline, and multiple network hops through Google's infrastructure.

### When to Choose Apigee

Apigee is the right choice when you need an API management platform, not just an API gateway. If your organization exposes APIs to external developers, needs a developer portal, requires API analytics, and has budget for a premium product, Apigee fits. If you need a low-latency gateway for internal microservices, Apigee is overkill.

## Azure API Management: Microsoft's Managed Gateway

Azure API Management (APIM) is Microsoft's managed API gateway. It offers Consumption, Developer, Basic, Standard, and Premium tiers.

### Tier Comparison

| Tier | Cost/month | Capacity | SLA | Features |
|---|---|---|---|---|
| Consumption | Pay-per-call | Auto-scale | 99.9% | Serverless, no reserved capacity |
| Developer | $50 | Low | None | Development only |
| Basic | $150 | 200 calls/sec | 99.9% | Single region |
| Standard | $650 | 1,000 calls/sec | 99.95% | Multi-region read |
| Premium | $2,500+ | 5,000+ calls/sec | 99.95% | Multi-region write, VNet injection |

### Unique Features

**Policy snippets:** APIM policies are XML-based snippets that can be reused across APIs. Microsoft provides a gallery of pre-built policies for common scenarios (rate limiting, IP filtering, CORS, JWT validation).

**Backend circuit breaker:** APIM supports a circuit breaker pattern where requests to unhealthy backends are failed fast. The circuit breaker is configurable per backend with custom failure thresholds and cooldown periods.

**Azure AD integration:** APIM natively integrates with Azure Active Directory for authentication and authorization. No custom JWT validation code is needed for Azure AD-backed APIs.

## Lessons from Migration Stories

### Migration Story 1: From NGINX to Kong at a FinTech Startup

A FinTech startup grew from 5 to 50 microservices in 18 months. Their NGINX configuration grew from 50 lines to 3,000 lines. Custom Lua scripts handled JWT validation and rate limiting. Deploying a new route required a full NGINX reload, causing 500ms of downtime per deployment.

**Migration:**
1. Deployed Kong alongside NGINX. Both gateways received traffic (50 percent each).
2. Migrated routes one at a time from NGINX to Kong.
3. Kong's Admin API allowed route changes without reloads.
4. After 2 weeks, NGINX received zero traffic. NGINX was decommissioned.

**Result:** Route deployment time went from 10 minutes (edit config, test, reload, verify) to 30 seconds (API call, auto-propagated). Gateway downtime during deployments dropped to zero.

### Migration Story 2: From Self-Hosted to Managed at a SaaS Company

A SaaS company ran a self-hosted Envoy gateway for 3 years. The operational burden grew: 2 DevOps engineers spent 50 percent of their time managing gateway clusters, Redis instances, and configuration databases. The company decided to migrate to AWS API Gateway.

**Challenges:**
- AWS API Gateway adds 30-100ms latency, compared to 3ms for Envoy
- Some features (custom plugins, advanced circuit breaking) were not available in the managed service
- The company's traffic pattern (steady-state with predictable peaks) did not justify the custom gateway's flexibility

**Result:** The migration saved $120,000 per year in DevOps costs. Latency increased by 40ms at p50, but the application's SLA was 200ms, so the increase was acceptable. Two DevOps engineers were redeployed to product work.

### Migration Story 3: From One Gateway to BFFs at an E-Commerce Company

An e-commerce company had one Kong gateway serving mobile, web, and third-party APIs. The gateway's response payload grew to accommodate all client needs. Mobile clients received large responses with fields they did not need. Web clients needed additional fields that the gateway had to fetch from three extra services.

**Migration:**
1. Created three BFF gateways: mobile-bff, web-bff, and partner-bff.
2. Each BFF had its own Kong instance with route-specific plugins.
3. The mobile BFF aggregated data aggressively (5 services per endpoint).
4. The web BFF served rich responses with minimal aggregation.
5. The partner BFF enforced strict rate limits and API key authentication.

**Result:** Mobile response size dropped by 60 percent. Mobile page load time dropped by 40 percent. Web BFF added new features without affecting mobile. Each team owned their BFF independently.

## Summary

Four production gateway deployments — Netflix Zuul, AWS API Gateway, Kong, and Envoy — demonstrate the range of gateway architectures available. Netflix built a custom filter-based gateway handling 10 million RPS. AWS offers a fully managed gateway with 30-100 ms latency overhead. Kong provides a plugin-rich open-source gateway with a control plane/data plane split. Envoy offers a high-performance proxy with dynamic xDS configuration.

Five lessons emerge from production experience: start simple and evolve, measure latency budget, expect the gateway to fail first under load, separate configuration from code, and know when to replace the gateway. The choice between open-source and managed, between plugin-rich and high-performance, depends on your team's operational capacity and latency requirements.
