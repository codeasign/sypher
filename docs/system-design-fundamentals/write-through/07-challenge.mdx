## Tier 1: Fundamentals

These challenges test your understanding of basic write-through concepts and their application.

### Challenge 1: Write-Through for a User Profile Service

You are designing a user profile service that needs write-through caching. Users update their profiles, and any subsequent read must return the updated data immediately.

**Your task**: Design the write path. Describe the sequence of operations from the moment the user submits a profile update to the point where the next read returns fresh data. Include the cache and database interactions.

**Key considerations**: What happens if the database is temporarily slow? What if the cache runs out of memory? How do you handle two simultaneous updates to the same profile?

**Model answer**:

The write path follows this sequence:

1. The application receives the profile update request (for example, update display name).
2. The application writes the new profile data to the cache using the user ID as the key.
3. The cache updates its internal storage and writes the data to its write-ahead log for durability.
4. The cache initiates a database write: `UPDATE users SET display_name = ? WHERE id = ?`.
5. The database executes the write, commits to its transaction log, and returns success.
6. The cache confirms the write is complete and returns success to the application.
7. The application sends a success response to the user.
8. When the user's next request reads the profile, the application hits the cache and retrieves the updated data directly — no database round-trip needed.

**Handling a slow database**: Set a timeout on the database write (for example, 2 seconds). If the timeout expires, roll back the cache update, return an error to the application, and enqueue the write for retry with exponential backoff.

**Handling cache memory pressure**: Configure the cache with a maximum memory limit and an eviction policy (LRU by default). When the cache is full, evicted entries must still be in the database — the next read will miss the cache and fetch from the database.

**Handling concurrent updates**: Use optimistic locking with version numbers. Each profile has a version field. Before writing, the application reads the current version. The cache checks the version during the write — if the version has changed, the write is rejected with a conflict error, and the application must retry with the latest version.

### Challenge 2: Read-Your-Writes Guarantee

A ride-sharing application uses write-through caching for trip state. A driver accepts a trip, which updates the trip status to "accepted." Immediately after, the rider checks the trip status.

**Your task**: Explain how write-through caching guarantees that the rider sees the "accepted" status immediately, even if the database write takes 50ms.

**Model answer**:

Write-through guarantees read-your-writes through the order of operations:

1. The driver app writes `trip_id: 123, status: "accepted"` to the cache.
2. The cache immediately updates its in-memory state with the new status.
3. The cache initiates the database write (this takes 50ms).
4. During those 50ms, the cache holds the updated status in memory.
5. The rider app reads `trip_id: 123` — the cache returns `status: "accepted"` immediately (microseconds).
6. The rider sees the accepted status before the database write has even completed.
7. After 50ms, the database writes complete. The cache already has the data.

The key insight is that the cache is updated BEFORE the database write completes. The cache serves the fresh data instantly, while the database write is still in flight. The write-through acknowledgment waits for the database, but the cache state is already updated for reads.

This means cache reads always return the latest written data, regardless of database latency. The write penalty affects write acknowledgment latency, not read freshness.

### Challenge 3: Write-Through vs. Write-Back Decision

Your social media platform has a "like" feature. Each post has a like count. You need to decide between write-through and write-back caching for this feature.

**Your task**: Which caching strategy would you choose? Justify your answer with specific reasoning about consistency requirements, write volume, and failure tolerance.

**Model answer**:

Write-back is the better choice for a "like" counter. Here is the reasoning:

**Consistency requirements**: Like counts do not require strong consistency. If a user likes a post and sees the count update 2 seconds later, that is acceptable. Eventual consistency is sufficient. Unlike financial transactions or inventory counts, slightly stale like counts are not a problem.

**Write volume**: Like buttons generate enormous write volume. A popular post can receive thousands of likes per second. Forcing every like through a database write in a write-through pattern would overwhelm the database. Write-back allows the cache to accumulate like events and flush them to the database in batches.

**Failure tolerance**: If a like is lost due to a cache crash before the batch flush, the consequence is minimal — a single like on a post with thousands of likes is not noticed. The cost of data loss is low.

**Write-back implementation**: The cache increments the like count in memory immediately (1-2ms). The user sees the updated count instantly. The cache periodically flushes aggregated counts to the database (every 10 seconds or every 100 updates). If the cache crashes between flushes, a small number of likes may be lost, but this is acceptable.

If you chose write-through, you would guarantee that every like is immediately durable. However, the database would need to handle 10,000+ writes per second during peak traffic, the user would experience 50ms+ latency on every like, and the infrastructure cost would be significantly higher.

## Tier 2: Applied Design

These challenges require you to design write-through systems for real-world constraints.

### Challenge 4: Distributed Write-Through with Sharding

You are designing a write-through cache for a multi-tenant SaaS platform. Each tenant has their own data, and tenants are isolated from each other. The system must handle 1,000 tenants, each with 10,000 operations per second.

**Your task**: Design a distributed write-through architecture using consistent hashing for cache sharding. Explain how writes are routed, how the write-through mechanism works across shards, and how you handle the failure of a cache node.

**Model answer**:

**Architecture**:

- The cache cluster consists of 20 Redis nodes, each with 32GB of memory.
- Consistent hashing distributes keys across nodes: `hash(tenant_id + ":" + key) % 20`.
- Each cache node maintains its own write-through adapter to the database.
- The database is sharded by tenant ID across 10 PostgreSQL instances.

**Write flow**:

1. The application tenant ID and key, computes the hash, and sends the write to the appropriate cache node.
2. The cache node updates its local state and writes the WAL entry.
3. The cache node opens a connection to the correct database shard (determined by tenant ID).
4. The database write executes and commits.
5. The cache node acknowledges the write to the application.
6. All subsequent reads for this key route to the same cache node and return fresh data.

**Handling a cache node failure**:

1. The load balancer detects the node is down and removes it from the ring.
2. Consistent hashing remaps the node's key range to the next node in the ring.
3. The replacement node has no cached data — initial reads from this node will be cache misses.
4. For write-through to work correctly on the replacement node:
   - Writes go to the replacement node and through to the database.
   - The cache miss on reads triggers a database fetch and cache population.
5. The data is eventually consistent after one read-through per key.
6. To accelerate recovery, the failed node's WAL can be replayed if the node restarts, or a background warmup process can pre-populate critical keys from the database.

### Challenge 5: Atomic Write-Through Across Cache and Database

Your payment system requires atomic writes — either both the cache and database are updated, or neither is. A partial update would cause financial inconsistency.

**Your task**: Design an atomic write-through mechanism. How do you ensure that a cache update is always paired with a successful database update, and that a failure in either causes a complete rollback?

**Model answer**:

True atomicity across two independent systems (cache and database) requires a coordination mechanism. Here are three approaches:

**Approach 1: Two-Phase Commit (2PC)**

1. **Phase 1 (Prepare)**: The cache writes the data to its local WAL (marked PREPARED). The database writes the data and marks the transaction as PREPARED. Both return "ready."
2. **Phase 2 (Commit)**: The coordinator sends COMMIT to both. The cache marks the WAL entry as COMMITTED. The database commits the transaction.
3. **Failure handling**: If either system fails in Phase 1, both roll back. If the coordinator crashes after Phase 1, a recovery process checks both systems and completes on restart.

**Approach 2: Write-Ahead Log with Replay**

1. Write the operation to a durable WAL (shared log) before touching either cache or database.
2. Update the cache.
3. Update the database.
4. Mark the WAL entry as COMPLETED.
5. On any failure, replay the WAL: check if the cache has the data, if the database has the data, and complete any missing write.
6. If the WAL entry cannot be completed, mark it for manual review.

**Approach 3: Outbox Pattern**

1. Write the data to the cache.
2. Write the operation to a database outbox table (in the same database transaction).
3. A background process reads the outbox table and ensures the cache write is complete.
4. If the cache write fails, the outbox entry remains and is retried.
5. The transaction is atomic because the outbox write is part of the same database transaction.

**Recommendation**: For most payment systems, the outbox pattern is preferred because it avoids the complexity of 2PC while providing strong guarantees. The cache is treated as a secondary store that is kept in sync by the outbox processor.

### Challenge 6: Write-Through with Read Replicas

Your database has a primary node for writes and multiple read replicas. You want to use write-through caching but replicas may be slightly behind the primary.

**Your task**: Design the write-through cache to work correctly with read replicas. How do you ensure that reads return fresh data when the read replicas may not yet have the latest write?

**Model answer**:

The write-through cache already solves this problem for cache reads. The cache is updated during the write, so any read that hits the cache gets fresh data regardless of replica lag.

The challenge is with cache misses. When the cache does not have a key, it reads from a replica. If the replica is behind, the read returns stale data, which is then cached as stale.

**Solution: Read-from-primary on cache miss**:

Configure the cache to read from the database primary on cache miss, not from replicas. This ensures that stale data never enters the cache.

**Alternative: Version-checked reads**:

1. The primary database assigns a version number (or timestamp) to each write.
2. This version is stored in the cache with the data.
3. On cache miss, the cache reads from a replica and also checks the primary for the latest version.
4. If the replica version matches the primary version, the data is fresh and is cached.
5. If the replica version is behind, the cache reads from the primary, gets the fresh data, and caches it.
6. This approach is more complex but reduces load on the primary.

**Alternative: Replica lag monitoring**:

1. Monitor replica lag continuously (track seconds_behind_master or equivalent).
2. Route cache-miss reads only to replicas with lag under a threshold (for example, under 100ms).
3. If no replica is within the threshold, read from the primary.
4. This is a simpler heuristic but does not guarantee correctness during lag spikes.

## Tier 3: Advanced Systems

These challenges require deep system design skills and are appropriate for senior and staff-level interviews.

### Challenge 7: Global Write-Through with Strong Consistency

Your company is expanding globally. You need a write-through cache that spans three regions (US, EU, Asia) and provides strong consistency. A user in any region must read their own writes immediately.

**Your task**: Design a globally distributed write-through cache with strong consistency guarantees. Discuss the trade-offs between consistency, latency, and availability.

**Model answer**:

**Architecture: Active-Passive with Global Write Coordinator**

- One region (US) is the leader for writes. All writes route to the leader region.
- The leader region has a write-through cache and a primary database.
- Other regions (EU, Asia) serve reads from their local caches.
- Cross-region replication replicates the database from the leader to follower regions.
- Follower caches are invalidated by a global invalidation bus.

**Write flow**:

1. A user in EU updates their profile. The request routes to US (the write leader).
2. The US write-through cache updates and writes through to the US primary database.
3. The write completes and the user gets acknowledgment (150-200ms latency from EU to US).
4. The US database replicates to EU and Asia databases asynchronously.
5. The global invalidation bus sends a message to EU and Asia caches.
6. The EU and Asia caches invalidate the old entry (but the new entry is not yet in their cache).
7. If the user reads from EU immediately, the cache miss triggers a read from the EU database replica. If the replica has not yet received the replication, the read is stale.

**The read-your-writes problem**: If the user writes from the EU and then reads from the EU, the cache miss reads the EU replica which may be stale. To guarantee read-your-writes, the system must either:

- Route reads to the leader region for a configurable window after a write (for example, 5 minutes).
- Require cache-miss reads to check the leader's version before returning data.
- Maintain a session-affinity table that tracks which region processed the user's last write and routes subsequent reads there.

**Trade-off analysis**:

- **Consistency**: Strong within the leader region. Eventual for follower regions after replication (unless read-routing is used).
- **Latency**: Writes from non-leader regions pay 150-200ms extra for cross-region round-trip. Reads are local (1-5ms).
- **Availability**: If the leader region goes down, the system must fail over to another region. During failover, writes are paused. Read-your-writes consistency may break during the transition.

**Alternative: Active-Active with CRDTs**

For lower write latency, use an active-active architecture with conflict-free replicated data types (CRDTs):

- Each region accepts writes locally (no cross-region routing).
- Each region has its own write-through cache and database.
- Regions replicate writes to each other asynchronously using CRDT merge logic.
- CRDTs ensure that conflicting writes (same key written in two regions) are merged deterministically.

**Limitations**: CRDTs work well for counters, sets, and maps but not for arbitrary data structures or operations (for example, "decrement inventory by 1" is non-commutative across regions).

### Challenge 8: Write-Through with a Database That Cannot Keep Up

Your database can handle 1,000 writes per second, but your write-through cache needs to handle 10,000 writes per second during peak traffic. The database cannot be upgraded or replaced.

**Your task**: Design a write-through system that respects the database's capacity limit while still providing write-through guarantees. Do not use write-back.

**Model answer**:

**Approach: Write-Through with Admission Control and Batching**

**Component 1: Admission Controller**

- A rate limiter sits between the cache and the database.
- It tracks the write rate at 1-second granularity.
- When the rate approaches 900 writes/sec (90% capacity), it signals the cache to slow down.
- When the rate hits 1,000 writes/sec, it rejects excess writes and returns a backpressure signal.

**Component 2: Write Buffer with Priority**

- The cache maintains a priority write buffer with three tiers:
  - **Tier 1 (high priority)**: Writes for active sessions, recent user interactions. Flushed to DB immediately.
  - **Tier 2 (medium priority)**: Background updates, non-critical state changes. Flushed when DB capacity is available.
  - **Tier 3 (low priority)**: Analytics, counters, logging. Flushed when DB is idle (for example, overnight).
- Tier 2 and Tier 3 writes can hold in the buffer for up to 5 minutes before being flushed.

**Component 3: Batch Writer**

- Instead of one write per database statement, the batch writer groups writes by target table.
- It accumulates writes and sends them as batch INSERT/UPDATE statements (100 writes per batch).
- A batch of 100 writes takes roughly the same time as a single write (the DB overhead is in the connection and transaction, not the row count).
- This effectively multiplies the DB's write capacity by the batch size.

**Result**:

- The DB processes 1,000 batches per second, each containing ~50 writes on average.
- Effective write throughput: 50,000 writes per second.
- High-priority writes (Tier 1, ~20% of traffic) are flushed every 100ms.
- Medium-priority writes (Tier 2, ~50%) are flushed every 500ms.
- Low-priority writes (Tier 3, ~30%) are flushed every 5 seconds.

**Consistency guarantee**: Tier 1 writes maintain the write-through contract (acknowledged only after DB commit). Tier 2 and 3 writes weaken the contract — the cache is updated immediately but the DB write is deferred. This is a conscious trade-off: you weaken the durability guarantee for non-critical data to maintain the system's overall throughput.
