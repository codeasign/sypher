## The Problem Story

Imagine you run an e-commerce platform. A customer updates their shipping address in the profile settings. The application dutifully writes the new address to the database. But the cache layer still holds the old address from the previous read. When the customer loads the order confirmation page, the system reads from cache — and shows the old address. The package ships to the wrong location.

This is the **stale data problem**. After every cache miss, the cache is populated with whatever was in the database at that moment. But subsequent writes go directly to the database, leaving the cache untouched. The cache and database diverge. The window of inconsistency can last anywhere from a few seconds (if a TTL expires) to several minutes (if the cache entry has a long TTL or no expiration).

The product team receives complaints: profiles not updating, order details showing wrong information, inventory counts that never reflect recent purchases. Engineers scramble to invalidate cache entries manually. Some teams resort to setting absurdly short TTLs — under 5 seconds — which defeats the purpose of caching altogether. Others build elaborate invalidation schemes that are fragile and hard to maintain.

The root cause is clear: the system writes to one store (the database) while the cache holds a snapshot from an earlier read. The two stores diverge the moment a write happens.

## The Insight

Write-through cache solves this by making the cache an active participant in every write operation. Instead of writing only to the database, the application writes to the cache first. The cache then writes to the database. Only when both writes succeed does the application receive confirmation.

**Definition**: Write-through cache is a caching strategy where every write operation updates both the cache and the backing database within the same operation, ensuring the cache always contains fresh data.

**Analogy**: Think of maintaining a personal journal and a public blog. If you first write an entry in your journal, then later manually copy it to the blog, there is a period where the blog is out of date. If someone reads the blog during that gap, they see old content. Write-through is like writing the entry simultaneously in both places — the journal entry and the blog post are created together. Anyone reading either source sees identical, up-to-date information.

The consequence is powerful: **read-your-writes consistency**. When you write data and immediately read it back, you always see the latest version. There is no staleness window. Every subsequent read from cache hits the freshly written data. The cache never serves stale data because it was never allowed to fall behind.

The trade-off is equally clear: every write now pays a latency penalty for updating two systems instead of one. This is called the **write penalty** — the extra time spent writing through to the database.

## Core Vocabulary

**Write-through** — A caching strategy where writes update both cache and database in a single synchronous operation. You always read fresh data but pay extra latency on every write.

**Write penalty** — The additional latency incurred because each write must complete against both the cache and the backing database before being acknowledged. This is the fundamental cost of write-through caching.

**Synchronous write** — A write operation that waits for confirmation from the target system before proceeding. In write-through, the cache writes to the database synchronously — the cache does not respond to the application until the database confirms.

**Cache coherency** — The property that all reads of a given key return the most recently written value. Write-through guarantees strong cache coherency because the cache is always updated before any read can see stale data.

**Write amplification** — The phenomenon where a single logical write generates multiple physical writes across the system. In write-through, one application write produces one cache write and one database write. Under failures or retries, amplification can increase.

**Atomic write** — A write operation that either completes fully across both cache and database or has no effect. True atomicity across two independent systems is difficult — write-through typically uses transactional patterns to approximate it.

**Dual-write** — The pattern of writing to two different data stores as part of a single operation. Write-through is a specific form of dual-write where the cache is the first target and the database is the second.

**Write-back** — An alternative caching strategy where writes go only to the cache and are asynchronously flushed to the database later. Provides lower write latency but risks data loss if the cache fails before the flush completes.

**Write-around** — An alternative where writes bypass the cache entirely and go directly to the database. The cache is only populated on subsequent reads. Avoids cache pollution but means reads after a write will miss the cache.

**Cache consistency** — A measure of how closely the data in the cache matches the data in the backing store. Write-through provides strong consistency — the cache always matches the database for written keys.

**Read-your-writes consistency** — A consistency guarantee that once a write completes, any subsequent read by the same client sees that write. Write-through provides this guarantee naturally.

**Staleness window** — The time during which the cache holds data that differs from the database. In write-through, the staleness window is effectively zero for successfully written keys.

**Write buffer** — A temporary queue within the cache layer that holds pending writes before they are sent to the database. Helps batch operations and manage backpressure.

**Commit** — The point at which a write is considered durable. In write-through, the commit happens when both the cache and the database have confirmed the write.

## How It Actually Works

Write-through caching follows a precise sequence of steps for every write operation. Understanding this sequence reveals both the guarantees and the costs.

### Step Descriptions

**Step 1 — Issue Write**: The application sends a write request to the cache layer. This could be a key-value set, a document update, or a SQL write depending on the cache technology. The application expects a synchronous response — it blocks until it hears back.

**Step 2 — Receive Write**: The cache accepts the write request on its network listener. The request enters the cache's processing pipeline, where it is parsed and validated.

**Step 3 — Validate and Serialize**: The cache checks the write for correctness — field types, size limits, and any schema constraints. The data is then serialized into the internal format used by both the cache storage engine and the database adapter. This is also where encoding transformations happen (JSON to internal bytes, for example).

**Step 4 — Update Internal State**: The cache updates its local data structures. This might mean inserting into a hash table, updating a B-tree, or appending to a sorted set. At this point, the data exists in the cache's memory but has not yet been persisted to the database.

**Step 5 — Initiate Database Write**: The cache opens a database connection from its connection pool and sends the write command. This is the moment the write penalty begins — the cache waits on the database.

**Step 6 — Commit to Disk**: The database receives the write and commits it to durable storage. This typically involves writing to a write-ahead log (WAL) and updating the database's internal data structures.

**Step 7 — Write-Ahead Log**: The database records the write in its transaction log before acknowledging. This ensures durability — even if the database crashes immediately after acknowledgment, the write can be recovered from the log.

**Step 8 — Database Acknowledge**: The database sends a success response back to the cache, confirming the write is durable.

**Step 9 — Confirm Write Success**: The cache marks the write as complete. If the write updated the cache's internal state speculatively in Step 4, this is now confirmed. If Step 8 fails, the cache must roll back its internal state.

**Step 10 — App Acknowledged**: The cache responds to the application with a success confirmation. The application can now proceed, knowing the data is in both the cache and the database.

**Step 11 — Next Read**: When the application reads the same key, it hits the cache directly. No database round-trip is needed.

**Step 12 — Return Fresh Data**: The cache returns the data that was just written. No staleness — the read-your-writes guarantee is satisfied.

## Variants

Write-through caching is not a single pattern. Different environments and requirements produce several distinct variants.

### Write-Through with Synchronous Database

This is the classic write-through pattern described above. The cache waits for the database to confirm before responding to the application.

Instead of writing each item to the database individually, the cache accumulates multiple writes and flushes them in a batch. Individual writes return quickly, but the batch commit introduces a slight delay in durability.

## Mental Model Check

Test your understanding of write-through cache with these questions.

**Q1: What happens to the cache if the database write fails in a write-through setup?**

The cache must roll back its internal state to before the write was applied. The data that was speculatively stored must be removed or reverted. The application receives an error response. The cache may log the failure for later analysis or retry, but it must not serve the uncommitted data.

**Q2: Why does write-through guarantee read-your-writes consistency?**

Because the cache is updated before the application receives acknowledgment. When the application reads the same key immediately after the write, the read hits the cache — which now holds the freshly written data. There is no window where the cache could return the old value because the old value was replaced before the write was acknowledged.

**Q3: What is the primary performance cost of write-through caching?**

The write penalty. Each write operation must wait for both the cache update and the database commit before returning. This adds the database write latency (typically 5-50ms) on top of the cache write latency (1-5ms). For write-heavy workloads, this can significantly reduce throughput.

**Q4: How does write-through differ from write-back in terms of data durability?**

Write-through ensures data is durable in the database before acknowledging the write. If the cache crashes after acknowledging, the data is safe in the database. Write-back acknowledges after the cache write and flushes to the database asynchronously — a cache crash before the flush loses data.

**Q5: Can write-through be used with a distributed cache cluster?**

Yes. Distributed write-through adds complexity: the write must be coordinated across cache nodes, and all nodes must agree on the state before writing to the database. This typically requires a consensus protocol or a designated write coordinator to avoid race conditions and conflicts.
