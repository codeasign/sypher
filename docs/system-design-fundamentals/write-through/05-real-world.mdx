## Company Case Studies

Real-world systems use write-through caching in specific ways that reveal how the pattern works under production constraints.

### Netflix — Write-Through to Cassandra

Netflix uses write-through caching extensively with Apache Cassandra as the backing database. The EvCache (EVCache) layer, built on memcached, operates as a write-through cache in several critical paths.

**Why write-through**: Netflix needs read-your-writes consistency for user playback state. When a user pauses a video and resumes on another device, the resume position must be accurate. Write-through ensures the cache is fresh, so reads from any device return the correct position.

**Scale**: EVCache clusters handle millions of writes per second across Netflix's infrastructure. Each EVCache node is a memcached instance with a write-through adapter that sends writes to Cassandra using a configurable consistency level (typically QUORUM).

**Failure handling**: If Cassandra is unavailable, EVCache can fall back to a local write buffer that replays writes when Cassandra recovers. This is a relaxation of pure write-through — the cache accepts the write locally (acknowledging quickly) and defers the database write.

Uber uses write-through caching for trip data and driver state. The architecture uses Redis as the write-through cache with a PostgreSQL or Cassandra backend.

**Why write-through**: Trip state must be consistent. If a driver accepts a trip, both the driver and the rider must see the accepted state. Any inconsistency leads to confusion — drivers arriving at wrong locations, riders waiting for no-show drivers.

**Scale**: Uber handles millions of trips daily across hundreds of cities. The write-through cache absorbs read traffic that would otherwise overwhelm the database.

**Coordination**: Uber uses Ringpop (a consistent hashing library) to route writes to the correct Redis node. This ensures that all writes for a given trip are handled by the same cache node, avoiding cross-node coordination for single-key operations.

### Shopify — Write-Through for Product and Inventory

Shopify uses write-through caching for merchant product data and inventory counts. The architecture uses a distributed cache layer that writes through to MySQL.

**Why write-through**: Inventory consistency is critical. When a customer places an order, inventory must immediately reflect the change. A cache that serves stale inventory data could cause overselling — accepting orders for products that are actually out of stock.

**Scale**: Shopify serves hundreds of thousands of merchants. Each merchant has its own product catalog and inventory. The write-through cache ensures that each merchant sees their own writes immediately, even before the database fully replicates.

**Implementation**: Shopify uses a per-merchant cache shard. Writes go to the cache shard and the primary database. The cache shard is always fresh because it sits in the write path.

### Stripe — Write-Through for Balance and Transaction Data

Stripe uses write-through caching for account balances and transaction state in the payment processing pipeline.

**Why write-through**: Financial data demands strong consistency. A cached account balance must reflect the actual balance in the database. Serving a stale balance could lead to incorrect authorization decisions, double-spending, or accounting errors.

**Implementation**: Stripe's write-through cache sits between the API layer and the database. Every balance update writes through to the database. Read requests hit the cache, which guarantees fresh data. The cache also stores computed values (available balance, pending balance) that are derived from the raw transaction log.

## Open Source Implementations

### Redis with AOF Persistence

Redis with Append-Only File (AOF) persistence is a form of write-through caching. Every write command is appended to the AOF file on disk before being applied to the in-memory data store.

- **Write-through behavior**: The `appendfsync always` mode calls `fsync()` after every write, ensuring the write is on disk before Redis acknowledges. This is a single-node write-through: Redis is both cache and durable store.
- **Latency**: `appendfsync always` reduces Redis throughput by up to 50% compared to `appendfsync everysec`.
- **Use case**: Systems that need the speed of Redis with the durability of a database, and can tolerate the write throughput reduction.

### Apache Cassandra

Cassandra is a distributed database that internally uses a write-through pattern:

1. Writes are first recorded in the commit log (a write-ahead log on disk).
2. Then written to the memtable (an in-memory write-back cache).
3. The memtable periodically flushes to SSTables on disk.
4. Reads check the memtable first (which always has fresh data), then bloom filters, then SSTables.

The commit log acts as the \"durable store\" equivalent, and the memtable as the \"cache.\" This ensures reads always see the latest writes (read-your-writes consistency within a Cassandra node).

### Hazelcast

Hazelcast is an in-memory data grid that supports write-through persistence to external databases. Its `MapStore` interface provides a write-through implementation:

- `store(key, value)` — called after the cache is updated, writes to the external database.
- `storeAll(map)` — batch variant for bulk writes.
- `delete(key)` — called after the cache entry is removed, deletes from the database.

Hazelcast's write-through can be configured with:

- **Write coalescing**: Multiple writes to the same key are coalesced into a single database write.
- **Write delay**: A configurable delay before the database write is initiated, allowing batching.
- **Retry configuration**: Number of retries and backoff strategy for failed database writes.

### Apache Ignite

Apache Ignite is an in-memory computing platform with native write-through support. Its `CacheStore` interface allows custom write-through and write-behind (write-back) implementations.

Ignite distinguishes itself by supporting:

- **Transactional write-through**: Writes to cache and database are part of the same distributed transaction.
- **Cross-partition consistency**: Ignite ensures that writes to the same key are serialized across the cluster.
- **Write-behind option**: Per-cache configuration to switch between write-through and write-back.

### Couchbase

Couchbase is a NoSQL database that uses a write-through cache at its core. The in-memory cache layer is always updated on writes before the data is persisted to disk:

- **Ejection and write-through**: If the cache is full, the least recently used items are ejected. Ejected items are still in the database. When accessed again, they are read from disk back into the cache.
- **Write path**: Set key → update cache → write to disk → acknowledge.

## Cloud Provider Implementations

### AWS ElastiCache with Write-Through

AWS ElastiCache (Redis/Memcached) does not natively support write-through. You implement it in the application layer. The standard pattern is:

1. Write to ElastiCache first.
2. Write to the database (RDS, DynamoDB, Aurora).
3. Verify the database write succeeded.
4. If the database write fails, remove the key from ElastiCache and return an error.

**Best practice**: Use DynamoDB with DAX (DynamoDB Accelerator). DAX is a write-through cache for DynamoDB — writes to DAX propagate to DynamoDB synchronously. Reads from DAX return fresh data automatically. No application-level coordination needed.

### GCP Memorystore

Google Cloud Memorystore (Redis) follows the same pattern as ElastiCache — the application manages the write-through logic. For stronger consistency guarantees, combine Memorystore with Cloud Spanner or Cloud SQL using the application-level write-through pattern.

### Azure Redis Cache

Azure Redis Cache provides a `RedisEnterprise` tier that supports active geo-replication. For write-through:

1. Write to Azure Redis.
2. Write to Azure SQL Database or Cosmos DB.
3. Handle failures with Azure's retry and circuit breaker patterns.

## Lessons Learned

### Lesson 1: Consistency Comes at a Cost

Every company that uses write-through caching acknowledges the write penalty. The trade-off is explicit: you pay more for writes to guarantee fast, consistent reads. Always measure your read-to-write ratio before adopting write-through. If reads are fewer than 10x writes, consider an alternative strategy.

### Lesson 2: Always Have a Fallback

Write-through couples the cache to the database. When the database is unavailable, the cache cannot serve writes. Always have a fallback mode:

- **Fail-open**: Queue writes locally and replay when the database recovers.
- **Fail-close**: Reject writes and return a clear error to the user.
- **Degrade**: Switch to write-around (bypass cache) and accept the read latency cost.

### Lesson 3: Monitor Write Latency Relentlessly

Write latency is the single most important metric for a write-through cache. A 20ms increase in write latency doubles the time the cache holds database connections, halving effective throughput. Set aggressive alerts on p50, p99, and p999 write latency.

### Lesson 4: Test Database Failures Exhaustively

Most teams test the happy path — writes succeed, caches update, reads return fresh data. The failures are where write-through systems break. Run chaos experiments:

- Kill the database mid-write.
- Introduce network latency between cache and database.
- Fill the write buffer.
- Exhaust the connection pool.
- Test your circuit breaker.

### Lesson 5: Plan for Burst Writes

Black Friday, product launches, flash sales — write traffic can spike 10-100x in seconds. Write-through systems must handle these bursts without failing. Strategies include:

- **Over-provision the connection pool** by 3-5x for expected burst levels.
- **Queuing at the application layer** with backpressure signals.
- **Rate limiting** on the write path to protect the database.
- **Auto-scaling** the cache cluster and database connections.
