## The Core Tension

Write-through caching embodies a fundamental trade-off in distributed systems: consistency versus latency.

**The guarantee**: Write-through ensures that the cache always contains fresh data. Every read returns the most recently written value. There is no staleness window, no TTL-based guessing, no cache invalidation logic to debug.

**The cost**: Every write must complete against both the cache and the database before the application can continue. This doubles the write latency compared to write-back caching (which only waits for the cache) and adds a database round-trip compared to write-around (which bypasses the cache entirely).

This tension is irreducible. You cannot have both instant writes and guaranteed cache freshness. Every design decision in a write-through system is a negotiation between these two forces.

## Decision Matrix

Choosing between write-through, write-back, and write-around depends on your workload characteristics and consistency requirements.

| Dimension | Write-Through | Write-Back | Write-Around |
|---|---|---|---|
| Write latency | High (+DB round-trip) | Low (cache only) | None (bypasses cache) |
| Read latency (hot data) | Very low (always fresh) | Very low (cache hit) | Low to high (cache miss after write) |
| Read-your-writes | Guaranteed | Not guaranteed | Not guaranteed |
| Data durability | High (DB confirms) | Low (cache flush risk) | High (DB confirms) |
| Cache utilization | High (every write populates) | High (every write populates) | Low (only reads populate) |
| Write amplification | 2x (cache + DB) | 1x (deferred DB flush) | 0x on cache (DB only) |
| Consistency model | Strong | Eventual | DB-consistent, cache stale |
| Failure recovery | Complex (rollback or WAL) | Risk of data loss | Simple (cache miss falls back) |
| Best for | Read-heavy, consistency-critical | Write-heavy, some data loss OK | Write-heavy, cache is optional |

### Synchronous vs. Asynchronous Write-Through

| Variant | Commit | Latency | Durability | Use Case |
|---|---|---|---|---|
| Write-through, sync DB | Cache + DB both confirm | Highest | Highest | Financial transactions |
| Write-through, async DB | Cache confirms, DB queued | Medium | Medium (risk on cache crash) | User profiles, session data |
| Write-through, batch | Cache confirms, batched to DB | Medium-low | Medium (batch loss risk) | Analytics, logging |
| Write-through, quorum | Cache + N of M DB nodes confirm | High | Very high | Distributed databases |

### Single vs. Batch Commit

| Aspect | Single Commit | Batch Commit |
|---|---|---|
| Latency per write | Full DB latency | Cache latency only |
| Throughput | Limited by DB connections | Higher (batches reduce overhead) |
| Durability gap | None | Writes in batch buffer at risk |
| Complexity | Low | Moderate (batch sizing, flush timing) |
| Best for | Small-scale, strict consistency | High-throughput, some delay OK |

## Failure Modes

Understanding how write-through fails is essential for building resilient systems.

### Failure Mode 1: Database Failure (Write Amplification)

When the database experiences an outage, the write-through cache cannot complete writes. If the database is slow (not down), writes queue up in the cache's write buffer. As the buffer fills, the cache applies backpressure to applications. If the database is down, every write fails immediately.

### Failure Mode 2: Write Buffer Overflow

Under write-heavy burst traffic, the write buffer fills faster than the database can drain it. The buffer hits its configured limit and must decide: block incoming writes (increasing application latency) or drop writes (losing data).

**Signs**: Write latency starts at 5ms (normal), grows to 50ms (buffer filling), then spikes to 500ms+ (buffer full, blocking). Error rate climbs as timeout thresholds are exceeded.

**Mitigation**: Configure buffer size based on worst-case burst modeling. Use backpressure signals to slow down write sources before the buffer fills. Consider a secondary flush pathway during extreme load.

### Failure Mode 3: Deadlock

Write-through caches that use locking to prevent concurrent writes to the same key can deadlock if not carefully implemented.

**Scenario**: Application A holds a lock on key X and needs to write to key Y. Application B holds a lock on key Y and needs to write to key X. Both wait forever.

### Failure Mode 4: Transaction Timeout

A single slow database write can hold up the entire write pipeline. If the transaction manager has a timeout of 10 seconds and the database occasionally takes 15 seconds on a write, that write blocks a connection and a transaction slot for the full timeout duration.

**Impact**: Connection pool depletes. Other writes queued. Latency spikes. If the slow write is the tip of an iceberg (the database is generally slow), cascading failures follow.

**Mitigation**: Use aggressive timeouts (2-3 seconds) and retry logic for slow writes. Separate timeouts for different operation types (simple writes vs. batch writes). Monitor slow query logs in the database.

### Failure Mode 5: Cascading Failure

A cascade in a write-through system starts with a small failure that amplifies through the system's dependencies.

**Chain**: Minor DB slowdown → more in-flight writes → connection pool exhaustion → retry queue fills → write buffer fills → application threads block → health checks fail → load balancer drains instances → remaining instances overloaded → more failures.

## Anti-Patterns

These anti-patterns appear repeatedly in production write-through systems.

### Anti-Pattern 1: Unlimited Write Buffer

**What**: The write buffer has no size limit. Under write bursts, it grows unbounded, consuming all available memory.

**Why it fails**: The cache process runs out of memory and is killed by the OOM killer. All buffered writes are lost. The cache restarts cold, and every subsequent read hits the database, causing a read storm.

**Fix**: Set a hard limit on the write buffer. When the limit is reached, apply backpressure (slow down clients) rather than accepting more data.

### Anti-Pattern 2: No Retry Limit

**What**: Failed database writes are retried indefinitely with no maximum.

**Why it fails**: A single unprocessable write (for example, a schema violation) is retried forever, consuming connections, CPU, and logging resources. Other writes that could succeed are starved.

**Fix**: Set max retries (3-5). Move persistently failing writes to a dead-letter queue for manual inspection.

### Anti-Pattern 3: Synchronous Writes to Slow Database

**What**: The write-through cache waits for a database that averages 100ms+ per write.

**Why it fails**: Write latency is dominated by the database. Throughput is limited by database connection count. The cache provides no speed benefit for writes, and reads only benefit if the cache hit rate is high.

**Fix**: Optimize the database write path first. Consider batching writes (multiple writes per DB call). If the database cannot be optimized, consider write-back or a different caching strategy.

### Anti-Pattern 4: Ignoring Write Failures

**What**: The cache writes through to the database, but if the database write fails, the cache returns success anyway.

**Why it fails**: The cache has the new value, but the database does not. If the cache crashes, the write is lost. If a different cache node serves the key, it gets the old value from the database. Consistency is broken silently.

**Fix**: Always verify the database write before acknowledging the client. If the DB write fails, return an error and roll back the cache state.

### Anti-Pattern 5: No Circuit Breaker

**What**: When the database is slow or down, the cache continues trying to write through, compounding the failure.

**Why it fails**: Every write attempt to a failing database consumes resources and time. The cache fills with pending writes. Applications time out. The failure cascades.

**Fix**: Implement a circuit breaker that detects repeated database failures and switches to a fallback mode (reject writes, switch to write-around, or queue writes) until the database recovers.

## When NOT to Use Write-Through

Write-through caching is not a universal solution. These conditions suggest a different approach.

**Write-heavy workloads**: If your system performs more writes than reads, the write penalty dominates. Every write pays extra latency, and the read benefit is minimal. Write-back or write-around are better suited.

**High throughput requirements**: Write-through is inherently throughput-limited by the database. If you need hundreds of thousands of writes per second, the database will be the bottleneck. Consider write-back with batch flushing or an in-memory database.

**Eventual consistency is acceptable**: Many applications do not need read-your-writes consistency. Social media feeds, analytics dashboards, recommendation engines — these can tolerate seconds or minutes of staleness. For these workloads, write-through adds unnecessary cost.

**Database cannot handle the write load**: If your database is already near its write capacity limit, adding write-through caching makes the problem worse. Optimize the database first, or use a caching strategy that reduces database write load (write-back, for example).

**Geographically distributed writes**: In multi-region active-active setups, write-through becomes impractical due to cross-region latency. Use conflict-free replicated data types (CRDTs) or a global write coordinator with eventual consistency instead.
