# NoSQL Overview — Under the Hood

## Internal Architecture

A NoSQL database is not a monolith. It is a stack of layers, each solving a specific problem: wire protocol, request routing, consistency coordination, data distribution, storage engine, and failure detection. Understanding each layer is the difference between operating a database and being operated by it.

## Data Structures and Algorithms

Every NoSQL database is built on a small set of fundamental data structures. If you understand these five, you understand the core of almost every system.

### LSM-Tree (Log-Structured Merge Tree)

The LSM-tree is the single most important data structure in NoSQL. It is the engine behind Cassandra, HBase, LevelDB, RocksDB, ScyllaDB, Bigtable, and DynamoDB's storage layer.

An LSM-tree inverts the traditional B-tree approach. Instead of modifying pages in place (which requires random I/O), it batches writes in memory (a memtable, typically implemented as a balanced tree or skip list), then flushes them as immutable sorted files (SSTables) to disk. Reads must merge data from the memtable and all relevant SSTables.

**Write cost**: O(log N) amortized for the memtable insert, plus O(1) sequential I/O for the flush. Compare to a B-tree's O(log N) random I/O per write, which is 10-100x slower on disk.

**Read cost**: O(log N) for the memtable, plus O(K * log N) for K SSTables. This is the LSM-tree's weakness: reads can be slow if compaction falls behind and too many SSTables accumulate.

### Bloom Filter: The Negative-Space Accelerator

A bloom filter is a probabilistic data structure that answers "have I seen this key before?" It can say "definitely not" (100% accurate) or "maybe yes" (some false positive rate). It cannot produce false negatives.

Internally, a bloom filter is a bit array of size M with K independent hash functions. To insert a key, you hash it K times and set the corresponding K bits to 1. To check membership, you hash it K times and check all K bits — if any is 0, the key is definitely absent; if all are 1, the key is probably present.

**False positive rate**: approximately (1 - e^(-KN/M))^K. With M=10 bits per key and K=7 hash functions, the false positive rate is under 1%.

### Consistent Hashing: Adding Nodes Without Chaos

Consistent hashing solves the rehashing problem. In a traditional hash table, adding or removing a server requires remapping (N-1)/N of all keys. In a system with 100 servers, adding one server moves 99% of the data. Consistent hashing reduces this to K/N where K is the number of virtual nodes — typically under 10%.

The idea: both keys and servers are hashed onto the same circular space (a ring from 0 to 2^32 - 1). Each server is assigned one or more points on the ring (virtual nodes, or vnodes). A key is assigned to the nearest server clockwise from its hash position. When a server joins or leaves, only the keys in its immediate ring neighborhood need to move.

### Version Vectors and Vector Clocks

In a distributed system with no leader, two nodes can independently accept writes for the same key. Version vectors (also called vector clocks) track causality so the system can detect and resolve conflicts.

A version vector is a map of `{node_id: counter}` attached to each value. When a node writes, it increments its own counter. When a node receives a value from another node, it merges the vectors. Two values are in conflict if neither vector dominates the other (i.e., each has a counter the other does not).

Dynamo-style databases return all conflicting values to the application during a read. The application (or a configured policy like "last-write-wins" based on timestamp) resolves the conflict and writes the resolved value back — this is called **read-repair**.

### Gossip Protocol: Rumor Propagation at Scale

The gossip protocol lets every node discover the state of every other node without a central registry. Each second, each node picks 1-3 random peers and exchanges its view of the cluster — which nodes are up, which are suspected down, what token ranges they own, and what their load is.

Information propagates exponentially: after log(N) rounds, every node in an N-node cluster has seen a given piece of information. If a node goes down, its neighbors start gossiping about it within seconds, and soon the entire cluster marks it as unavailable. The phi-accrual failure detector adds sophistication: instead of a binary "up/down", it computes a continuous suspicion level based on heartbeat timing history. A node that occasionally misses a heartbeat is not immediately declared dead — useful in environments with network jitter.

### Write Lifecycle: The Full Path with Consistency Edge Cases

A write follows a similar path but with different coordination: the coordinator writes locally, sends to replicas, and waits for acknowledgment. The key complexity is handling failures.

**Scenario: One replica is down (hinted handoff).**

Coordinator (Node C) sends write to Node D (acknowledges) and Node E (no response). After a timeout, Node C stores the write locally with a hint: "deliver to Node E." When Node E comes back online, Node C (or whichever node holds the hint) forwards the accumulated writes. The racioale: the system stays available for writes even with partial failures.

**Scenario: Network partition (two replicas cannot see each other).**

If the cluster splits into two groups that cannot communicate, each group's coordinator continues accepting writes to the same key. Both groups produce versions of the data that are now divergent. When the partition heals, a read may return both versions. This is where **last-write-wins** (LWW) based on a wall-clock timestamp can lose data (due to clock skew), or **vector clocks** can detect and expose the conflict for application-level resolution.

## Consistency and State Handling

### Crashes and Recovery

When a node crashes and restarts, it replays the commit log to reconstruct any writes that were acknowledged but not yet flushed to SSTables. This ensures durability: a write acknowledged to the client is never lost, even if the node crashes immediately after.

The recovery sequence:
1. Node starts, reads its commit log from disk.
2. All entries in the commit log are replayed into the memtable.
3. The memtable is flushed to SSTables.
4. The commit log is truncated (cleared) since all data is now in SSTables.
5. The node announces its availability via gossip.
6. Other nodes notice it is back and begin sending hinted handoffs that accumulated during the outage.

### Concurrent Writes and Last-Write-Wins

When two clients write to the same key at nearly the same time, the system needs a tiebreaker. Most NoSQL databases use **last-write-wins** (LWW) with a timestamp. The write with the latest timestamp wins. The problem: timestamps come from client machines with potentially unsynchronized clocks. If Client A's clock is 5 seconds ahead of Client B's clock, Client A's writes will always win, even if Client B's write happened later in real time.

Better systems use hybrid logical clocks (HLC) that combine wall-clock time with a logical counter, guaranteeing that timestamps respect causality even when clocks drift. Cassandra's implementation uses `TimeUUID` (type 1 UUID) which encodes both timestamp and node ID, guaranteeing unique ordering.

### Stale Reads

Stale reads occur when a read is served by a replica that has not yet received the latest write. This is inherent in eventually-consistent systems. The window of inconsistency depends on:
- **Replica lag**: how fast writes propagate (typically under 100ms in the same datacenter, can be seconds across regions).
- **Consistency level**: CL=ONE can read stale data; CL=QUORUM reduces the probability; CL=ALL eliminates it (but sacrifices availability).
- **Read-repair**: triggered on read if digests differ, which eventually fixes stale replicas.

## Performance Characteristics

### Latency by Operation

Performance varies dramatically by data model, consistency level, and whether data fits in memory.

| Operation | Memtable hit | SSTable hit (few) | SSTable hit (many) | Quorum (same DC) | Quorum (cross-DC) |
|-----------|-------------|-------------------|-------------------|------------------|-------------------|
| Get (key) | 0.5-1 ms | 2-5 ms | 10-50 ms | 5-20 ms | 50-200 ms |
| Put (key) | 0.5-2 ms | 0.5-2 ms | 0.5-2 ms | 5-20 ms | 50-200 ms |
| Range scan | 1-5 ms | 5-20 ms | 20-200 ms | 10-50 ms | 100-500 ms |
| Batch write | 2-10 ms | 2-10 ms | 2-10 ms | 10-50 ms | 100-500 ms |

Key insight: writes are consistently fast because they only touch the commit log and memtable. Reads degrade as SSTables accumulate. This is why compaction scheduling is critical — letting compaction fall behind can turn a 2ms read into a 50ms read.

### P50 / P99 / P999 Latency

In a well-tuned 6-node Cassandra cluster with replication factor 3, serving a workload of 50% reads / 50% writes at 10,000 ops/sec:

| Metric | Read (CL=ONE) | Read (CL=QUORUM) | Write (CL=ONE) | Write (CL=QUORUM) |
|--------|--------------|------------------|---------------|-------------------|
| P50 | 1 ms | 3 ms | 0.8 ms | 2 ms |
| P99 | 8 ms | 25 ms | 5 ms | 15 ms |
| P999 | 50 ms | 150 ms | 20 ms | 80 ms |

The jump from P99 to P999 is where compaction pauses, garbage collection (for JVM-based databases like Cassandra), and network jitter live. If P999 is more than 5x P99, look for compaction backlogs or GC tuning issues.

### Throughput Scaling

NoSQL databases scale throughput nearly linearly with nodes for write-heavy workloads — this is their primary advantage over SQL databases.

| Nodes | Writes/sec (Cassandra, RF=3) | Reads/sec (CL=ONE) | Reads/sec (CL=QUORUM) |
|-------|------------------------------|--------------------|----------------------|
| 3 | 30,000 | 60,000 | 40,000 |
| 6 | 65,000 | 130,000 | 85,000 |
| 12 | 140,000 | 280,000 | 180,000 |
| 24 | 300,000 | 600,000 | 380,000 |

Scaling is not perfectly linear due to gossip overhead, cross-node coordination for QUORUM operations, and compaction I/O sharing disk bandwidth. ScyllaDB (written in C++) achieves closer-to-linear scaling by avoiding JVM garbage collection pauses.

### Memory Sizing

A common error is underestimating memory requirements. The rule of thumb: your working set (frequently accessed keys) should fit in memory for consistent sub-millisecond reads. If your working set is 500 GB and each node has 32 GB of RAM, you need at least 16 nodes (plus replication overhead).

Memory consumers per node:
- **Memtable**: 64 MB to 1 GB (configurable). Larger memtables batch more writes before flushing, reducing compaction frequency but requiring more RAM.
- **Row cache**: 0 to N GB. Caching frequently read rows can dramatically improve read performance for hot keys. Set to 10-25% of the working set size.
- **Bloom filters**: approximately 1-2 GB per TB of data. Each SSTable has its own bloom filter loaded into memory.
- **Key cache / partition index**: approximately 1-2% of data size. Maps keys to their on-disk locations.
- **Compaction overhead**: temporary memory for compaction operations, typically 1-2 GB.
- **OS page cache**: the OS caches recently read SSTable blocks — 20-30% of the data size on a well-tuned system.

Total: for 1 TB of data per node, expect 30-60 GB of RAM just for the database overhead, before you count the working set.

## Configuration and Tuning

### The Critical Knobs

**Memtable size** (`memtable_heap_space_in_mb`): Larger memtables reduce flush frequency and compaction pressure but increase recovery time on crash and memory pressure. Default 25% of heap (Cassandra) or 64 MB per shard (ScyllaDB).

**Commit log sync** (`commitlog_sync`): `periodic` (sync every N ms, default 10,000) gives better write performance but risks losing up to 10 seconds of data on crash. `batch` (sync after every write) is slower but never loses acknowledged writes.

**Compaction strategy**:
- **SizeTieredCompactionStrategy (STCS)**: Default. Triggers when N similarly-sized SSTables exist. Good for write-heavy workloads with infrequent deletes.
- **LeveledCompactionStrategy (LCS)**: Maintains a fixed number of levels (L0 is small, L1 is 10x larger, etc.). Optimizes read performance at the cost of more write I/O. Good for read-heavy workloads.
- **TimeWindowCompactionStrategy (TWCS)**: Designed for time-series data. Compacts within time windows. Excellent for TTL-expiring data like metrics and logs.

**Read repair chance** (`read_repair_chance`): Probability that a read triggers a repair check on all replicas even if digests match. Default 0.1 (10%). Higher values increase consistency at the cost of read latency.

**Concurrent readers/writers** (`concurrent_reads`, `concurrent_writes`): Controls the number of parallel I/O operations. Default 32 for reads, 32 for writes. Tune up for SSDs (64-128), down for HDDs (8-16).

### Aggressive Tuning for Write-Heavy Workloads

If you are ingesting IoT sensor data at 500,000 writes per second:
- Set `commitlog_sync` to `periodic` with a 5-second sync interval — losing 5 seconds of sensor data is acceptable for the gain in throughput.
- Use `TimeWindowCompactionStrategy` with 1-hour windows — compaction only runs on one hour's worth of data at a time, never the full dataset.
- Disable row cache entirely — each sensor reading is written once and read once (or never), so caching provides no benefit.
- Increase `memtable_heap_space` to 25% of available RAM — more write buffering means fewer flushes means less compaction I/O contention.
- Set `concurrent_writes` to 128 to keep all CPU cores busy.
- Disable read-repair (`read_repair_chance: 0.0`) — if a sensor reading is inconsistent, the next write will overwrite it anyway.

### Aggressive Tuning for Read-Heavy Workloads

If you are serving a product catalog at 200,000 reads per second with 10,000 writes per second:
- Use `LeveledCompactionStrategy` — reads only need to check one SSTable per level.
- Increase row cache to 25% of RAM — product catalog data is read repeatedly and rarely changes.
- Enable caching of partition summaries — trade RAM for faster partition location.
- Set `concurrent_reads` to 128 to handle many simultaneous lookups.
- Consider adding a Redis caching layer in front for the hottest 5% of products.
- Tune bloom filter `false_positive_probability` to 0.01 (1%) — lower values use more memory but skip more SSTable lookups.

### Monitoring: What to Watch

The five metrics that tell you if your NoSQL database is healthy:

1. **Compaction backlog**: the total number of pending compactions. A growing backlog means writes are arriving faster than compaction can merge them. Read latency will increase.
2. **P99 read latency by table**: not just overall P99. One hot table with a bad data model can drag down the entire cluster.
3. **Hinted handoff count**: a spike means nodes are going down and coming back. Sustained high handoff means a node is flapping (repeatedly crashing and restarting).
4. **Live disk usage trend**: if disk usage grows faster than the TTL expiration rate, your TTLs are too long, your compaction strategy is wrong, or your data model creates too much tombstone overhead.
5. **Garbage collection pause duration** (JVM databases): if GC pauses exceed 1 second, Cassandra's P999 latency will spike. Tune heap size or switch to a non-GC database (ScyllaDB, Redis).

The difference between a database that works and a database that works well is almost always in the compaction and memory configuration. Start with the defaults, watch the five metrics, and tune one knob at a time. Measure before and after each change — most "tuning" is actually just fixing problems that the defaults create at your specific scale.
