# NoSQL Overview — Challenges

## Tier 1: Foundation

These challenges take approximately 5 minutes each. They test your ability to make basic technology choices and reason about fundamental NoSQL concepts.

### Challenge 1: Event Tracking Database

You are building an analytics service that tracks page views for 500 websites. Each website sends an event every time a visitor loads a page. Each event includes: website_id, visitor_id, page_url, timestamp, user_agent, referrer, and custom properties (a JSON object with variable fields). You ingest 50,000 events per second. Queries are: "how many page views did website X have yesterday?" and "show me the last 100 visitors to page Y."

Choose a NoSQL database and design the data model. Explain your reasoning.

**Model Answer**:

Use Cassandra. The workload is write-heavy (50,000 writes/sec) with simple read queries (count by date, last 100 visitors per page). Cassandra's write throughput scales linearly, and range scans by timestamp are efficient.

Data model:

```
CREATE TABLE page_views_by_site (
    website_id text,
    date text,               // "2024-01-15"
    hour int,                // 0-23 (for rolling windows)
    visitor_id text,
    page_url text,
    timestamp timeuuid,
    user_agent text,
    referrer text,
    properties map<text, text>,
    PRIMARY KEY ((website_id, date, hour), timestamp)
) WITH default_time_to_live = 7776000;  // 90 days
```

The partition key `(website_id, date, hour)` keeps each partition to approximately 50,000 events * 1 KB = 50 MB — well within Cassandra's recommended limit of 100 MB per partition. The `timestamp` clustering column provides sorted order for the "last 100 visitors" query. TTL automatically expires data after 90 days, avoiding manual cleanup.

For the "count by date" query, you would either: (a) query all 24 hourly partitions for a site-date and count rows client-side (acceptable for under 1 million events), or (b) use a separate counter table `CREATE TABLE counters (website_id text, date text, count counter, PRIMARY KEY (website_id, date))` and increment the counter on each event.

### Challenge 2: Schema Evolution Design

Your product catalog started with 10 fields per product: name, description, price, category, image_url, sku, brand, weight, dimensions, color. After 6 months, your team needs to add: material (for clothing), wattage (for appliances), edition (for books), and age_range (for toys). Some products will have these new fields, others will not. You cannot afford any downtime during the schema change.

How do you handle this in: (a) a document store like MongoDB, and (b) a wide-column store like Cassandra?

**Model Answer**:

**For MongoDB**: No schema change required. Each new product document includes the fields relevant to that product. Existing documents simply lack the new fields. Your application code checks for field existence: `const material = product.material || "unknown"`. Add indexes on the new fields if they become query filters: `db.products.createIndex({ material: 1 })` — this is an online operation in MongoDB 4.2+ (background index build). Migration of existing products is optional: you can add the new fields to existing products via a batched background migration that reads 1,000 documents at a time, adds the new field (with a null or default value), and writes them back.

**For Cassandra**: Add the new columns with `ALTER TABLE products ADD material text;` — this is an online, metadata-only operation in Cassandra. The new columns default to null for existing rows. Your application code handles null: `String material = row.getString("material") != null ? row.getString("material") : "unknown"`. For the wide-column model, each product is a single row with dynamic columns. Adding columns does not require rewriting data.

Both approaches give zero downtime. MongoDB requires more care with index creation on large collections; Cassandra's ALTER TABLE is instantaneous but adds metadata overhead for each column definition (keep the total columns under 200-300 for performance).

### Challenge 3: Cache Performance Projection

You have a DynamoDB table serving a product catalog. Each read costs 1 RCU (read capacity unit, roughly 4 KB). The catalog has 1 million products. Read traffic is 10,000 requests per second. You add a Redis cache in front and achieve a 90% cache hit rate.

Calculate the cost savings in RCUs and estimate the Redis cluster size needed.

**Model Answer**:

**Baseline (no cache)**: 10,000 reads/sec * 1 RCU = 10,000 RCUs. At DynamoDB pricing of approximately $0.00013 per RCU-hour (on-demand): 10,000 * $0.00013 * 730 hours/month = approximately $949/month.

**With Redis cache (90% hit rate)**: 1,000 reads/sec hit DynamoDB (10% of 10,000). DynamoDB cost: 1,000 * $0.00013 * 730 = approximately $95/month.

**Redis cost**: You need to cache the most frequently accessed 10% of products (100,000 products). Each product record is approximately 2 KB (assuming name, price, image URL, description). Cache size: 100,000 * 2 KB = 200 MB. A single Redis instance with 1 GB memory (cache.r6g.large at approximately $0.12/hour = $86/month) is sufficient. Total cost: $95 (DynamoDB) + $86 (Redis) = $181/month — a 5x reduction from $949/month.

**Additional benefit**: Read latency drops from 5-15ms (DynamoDB eventual consistent read) to under 0.5ms (Redis in-memory read). P99 latency drops from 30ms to 2ms.

## Tier 2: Design

These challenges take approximately 15 minutes each. They require designing a complete data model and architecture.

### Challenge 4: Real-Time Leaderboard for Mobile Game

Your mobile game has 5 million daily active users. Each user plays an average of 8 games per day. After each game, the user's score is submitted. You need:

- A global top-100 leaderboard, updated in real-time (under 1 second delay).
- Per-user rank display: "You are rank #42,298 of 5,000,000."
- Historical leaderboard snapshots (daily top-100 for the past 30 days).
- Friend leaderboard: rank among a user's friends.

Design the complete data model and architecture. The read-to-write ratio is approximately 20:1 (users check the leaderboard far more often than they submit scores).

**Model Answer**:

**Three-tier architecture**:

**Tier 1: Redis Sorted Sets (real-time leaderboard)**.

```
GLOBAL_LEADERBOARD (sorted set) → score is the user's best score
  ZADD GLOBAL_LEADERBOARD user_id score
  ZREVRANGE GLOBAL_LEADERBOARD 0 99 WITHSCORES  → top 100
  ZREVRANK GLOBAL_LEADERBOARD user_id  → user's rank
```

Redis Sorted Sets are O(log N) for ZADD and ZREVRANK, and O(log N + M) for ZREVRANGE. With 5 million users, each operation completes in under 1ms.

Alternative design: Use a sorted set per game type if scores are per-game. This challenge assumes a single score type (e.g., best score across all games).

**Tier 2: DynamoDB (durable leaderboard storage and history)**.

```
leaderboard_current:
  PK: "GLOBAL"  SK: top_100#<rank>
  → user_id, score, last_updated

leaderboard_history:
  PK: "SNAPSHOT#2024-01-15"  SK: top_100#<rank>
  → user_id, score
  TTL: 60 days

user_scores:
  PK: "USER#" + user_id
  → best_score, total_games, last_game_timestamp
```

Every 5 minutes, a background worker reads the top 100 from Redis and writes a snapshot to `leaderboard_current` in DynamoDB (overwriting previous data). Every day at midnight, the worker saves a historical snapshot to `leaderboard_history`.

**Tier 3: Friend leaderboard**:

Store friend lists in Redis Sets per user:
```
FRIENDS:user_id → set of friend user_ids
```

For the friend leaderboard query:
1. `SMEMBERS FRIENDS:user_id` gets all friend IDs.
2. `ZREVRANK GLOBAL_LEADERBOARD friend_id1 friend_id2 ...` gets ranks for all friends.
3. Sort and return top 10.

This is O(N) where N is the friend count. For users with under 500 friends (typical), this completes in under 10ms.

**Write path**: After each game, the client submits the score to an API endpoint. The endpoint writes to Redis Sorted Set (ZADD) and asynchronously writes to DynamoDB `user_scores` via an SQS queue. The write to Redis is synchronous (the client needs the updated rank); the DynamoDB write is asynchronous.

**Why not use DynamoDB alone?** DynamoDB does not support server-side sorted set operations. Computing "rank among 5 million users" would require scanning the entire table — infeasible at this scale. Redis Sorted Sets provide this natively.

### Challenge 5: IoT Sensor Data Pipeline

You are building a platform for 100,000 industrial sensors. Each sensor reports temperature, pressure, humidity, and vibration every 5 seconds. Total: 20,000 writes per second. Each reading is approximately 200 bytes. Retention requirement: 90 days for raw data, 2 years for hourly aggregates.

Queries:
- "Show all readings for sensor X in the last hour" (dashboard, real-time).
- "Show hourly average temperature for sensor X for the past week" (analytics).
- "Alert if sensor X's reading exceeds threshold Y" (real-time, under 100ms).

Design the pipeline from ingestion through storage.

**Model Answer**:

**Ingestion layer**:

Use Kafka as the ingestion buffer. 20,000 writes/sec at 200 bytes each = 4 MB/sec. A 3-node Kafka cluster with 6 partitions handles this trivially. Kafka provides durability and replay capability — if storage goes down, data accumulates in Kafka until recovery.

**Storage layer**:

Use Cassandra with TimeWindowCompactionStrategy (TWCS) for raw data:

```
CREATE TABLE raw_readings (
    sensor_id text,
    hour_bucket int,          // epoch hour: 2024-01-15T10:00:00Z → 1705305600
    reading_time timestamp,
    temperature double,
    pressure double,
    humidity double,
    vibration double,
    PRIMARY KEY ((sensor_id, hour_bucket), reading_time)
) WITH default_time_to_live = 7776000  // 90 days
  AND compaction = { 'class': 'TimeWindowCompactionStrategy' };
```

The partition key `(sensor_id, hour_bucket)` keeps one partition to approximately 720 readings (5-second interval * 60 minutes * 60 seconds / 5 = 720). At 200 bytes each, that is 144 KB per partition — well within limits.

TWCS is ideal here because data is written in time order and never updated. Each hour bucket is compacted independently. No cross-hour compaction overhead.

**Analytics (hourly aggregates)**:

A separate Cassandra table stores pre-computed hourly aggregates:

```
CREATE TABLE hourly_aggregates (
    sensor_id text,
    date text,                 // "2024-01-15"
    hour int,                  // 0-23
    avg_temp double,
    min_temp double,
    max_temp double,
    avg_pressure double,
    reading_count int,
    PRIMARY KEY ((sensor_id, date), hour)
) WITH default_time_to_live = 63072000;  // 2 years
```

A streaming processor (Kafka Streams or Spark Structured Streaming) reads raw data from Kafka, aggregates by sensor-hour, and writes to the aggregates table. It also triggers alerts when thresholds are exceeded.

**Alerting**:

Alerts are evaluated in the streaming processor, not in the database. The streaming processor maintains a sliding window of the last 60 seconds of data per sensor (in memory, using Kafka Streams state store). When a value exceeds a threshold, the processor writes to a Redis-backed alert channel that the monitoring dashboard reads. This gives under 50ms alert latency.

**Read path for dashboard**:

The dashboard queries `raw_readings` for the latest hour's data from the current sensor. Because the partition `(sensor_id, hour_bucket)` matches the current hour, Cassandra reads from a single partition containing at most 720 rows — complete in under 5ms.

**Why not InfluxDB or TimescaleDB?** Those are purpose-built time-series databases and would also work. This solution uses Cassandra because you already operate it for other workloads (reducing operational complexity). If this were a greenfield project with no existing database, InfluxDB would be simpler.

### Challenge 6: Multi-Tenant SaaS Data Isolation

You are designing a SaaS platform where 500 tenants share a single Cassandra cluster. Each tenant has between 1,000 and 10 million user records. Tenants must not be able to see each other's data. Write throughput varies: small tenants write 10 records/sec, large tenants write 10,000 records/sec.

Design the data model and operational strategy to handle uneven tenant sizes without hot spots.

**Model Answer**:

**Data model**: Each table includes `tenant_id` as the partition key prefix.

```
CREATE TABLE tenant_users (
    tenant_id text,
    user_id text,
    email text,
    profile_data text,
    created_at timestamp,
    PRIMARY KEY ((tenant_id, user_id))
);
```

**Hot tenant problem**: A tenant with 10 million users generates 10,000 writes/sec. If all those writes go to the same `tenant_id` partition, that partition becomes a hot spot. Solution: add a shard key within the tenant.

Revised model for large tenants:

```
CREATE TABLE tenant_users (
    tenant_id text,
    shard text,             // for large tenants: "large_tenant:0" to "large_tenant:99"
    user_id text,
    email text,
    profile_data text,
    created_at timestamp,
    PRIMARY KEY ((tenant_id, shard), user_id)
);
```

For small tenants: `shard = "0"` (single partition). For large tenants: `shard = tenant_id + ":" + random(0-99)`, distributing writes across 100 partitions.

**Detection**: Monitor partition size and request rate per partition. When a tenant's partition crosses 100 MB or 1,000 writes/sec, automatically flag it for sharding. Apply the sharding configuration in a metadata table:

```
CREATE TABLE tenant_config (
    tenant_id text PRIMARY KEY,
    tier text,            // "small", "medium", "large"
    shard_count int,      // 1, 10, 100
    created_at timestamp,
    updated_at timestamp
);
```

The application reads `tenant_config` on startup and caches it (with 5-minute refresh) to determine the shard key scheme for each tenant.

**Operational strategy**: Small tenants are free or low-tier (single partition, up to 10,000 users). Medium tenants get 10 shards (up to 1 million users). Large/enterprise tenants get 100 shards (up to 100 million users). Tenant tier is part of the pricing model, creating a natural incentive to stay within shard limits.

**Data isolation**: The `tenant_id` partition prefix ensures that no query can accidentally read another tenant's data without specifying the tenant_id. The application authenticates the tenant on every request and injects the tenant_id into every query. Row-level security is enforced at the application layer, not the database layer.

### Challenge 7: Conflict-Free Replicated Data Types for Multi-Region Shopping Cart

Your e-commerce platform serves 20 million users across 5 regions. You are migrating from a single-region MySQL database to a multi-region DynamoDB setup with active-active replication so that users always write to their nearest region.

The problem: if a user adds items to their cart in the US, then flies to Europe and continues shopping, two regions may have conflicting versions of the same cart. Last-write-wins (LWW) loses items — if the US cart has items A and B, and the EU cart has items A and C, LWW based on timestamp would keep whichever cart was last written, silently dropping either B or C.

Design a conflict resolution strategy that never loses items. Describe how you would implement it using DynamoDB constructs.

**Model Answer**:

**Approach: Use DynamoDB as a CRDT (Conflict-Free Replicated Data Type) store.**

Each cart item is stored as a separate DynamoDB item (not as a list within a single item). The cart is a DynamoDB collection: `PK = "cart:" + user_id`, `SK = "item:" + item_id`.

When a user adds an item to their cart, the application performs a DynamoDB `PutItem` with `ConditionExpression: "attribute_not_exists(item_id)"`. If the item already exists (added in another region), the condition fails and the operation is a no-op — the item is already in the cart.

When a user removes an item, the application performs a DynamoDB `DeleteItem`. A tombstone (a deleted-item marker) is written to a separate table: `PK = "cart_tombstone:" + user_id`, `SK = "removed:" + item_id`, with TTL of 7 days (long enough for cross-region replication to complete).

When reading the cart, the application:
1. Queries all items: `PK = "cart:" + user_id`.
2. Queries all tombstones: `PK = "cart_tombstone:" + user_id`.
3. Filters out any items that appear in the tombstone list.

**Conflict scenario**: User has item A in cart (added in US). User flies to EU and removes item A. EU region writes a tombstone. The tombstone replicates to all regions within seconds. Result: item A is not in the cart anywhere. No conflict.

**Edge case**: User adds item B in EU at the same time as removing item B in US. Both operations succeed in their respective regions (add uses `attribute_not_exists`, remove is unconditional). The tombstone for item B exists. The item B also exists. The read reconciliation filters out item B because the tombstone exists. The add is effectively undone by the remove — this is correct behavior (the last intended action wins for the item's lifecycle).

**Item quantity updates**: For items where quantity matters (e.g., 3x T-shirts), use DynamoDB `UpdateItem` with `ADD` (atomic increment). If US adds 2 and EU adds 1 concurrently, both operations apply, and the final quantity is 3. This is a commutative operation (order does not matter), making it naturally conflict-free.

**Advantage over LWW**: LWW on the entire cart loses items. This approach treats each cart item independently, using idempotent operations that are safe under concurrent updates from multiple regions. No item is silently lost.

**Trade-off**: The cart is no longer a single DynamoDB item, so you cannot use DynamoDB transactions across the entire cart. If you need atomic "add 3 items together," you would need a multi-item transaction or handle the "partial cart" state in the UI.

### Challenge 8: Global-Scale Multi-Model Database

Your company has acquired a startup that uses MongoDB and another that uses Cassandra. The CTO wants to consolidate both workloads into a single database platform to reduce operational costs. The workloads are:

- **Workload A (MongoDB)**: Product catalog, 10 million products, document model with 50+ field types, ad-hoc queries (search by name, category, price range, attributes), read-heavy (20,000 reads/sec), write-light (500 writes/sec). Requires secondary indexes on 10 fields.
- **Workload B (Cassandra)**: Event tracking, 5 billion events/month, write-heavy (100,000 writes/sec at peak), simple queries (events by user_id in time range), TTL-based data expiration after 90 days, multi-region (3 regions).

The CTO wants to know: is a single NoSQL database capable of handling both workloads? If so, which one? If not, what is the right strategy?

**Model Answer**:

**Short answer**: No single NoSQL database handles both workloads optimally. The two workloads are at opposite ends of the NoSQL spectrum — MongoDB excels at document queries and schema flexibility, Cassandra excels at write throughput and linear scalability. Forcing both into one database would compromise one or both workloads severely.

**Detailed analysis**:

**Option A: Use Cassandra for both**. Workload A (product catalog) would suffer: Cassandra does not support ad-hoc secondary index queries efficiently. Querying "all products in category X with price under $50 and color = red" requires either (a) a materialized view for every query pattern you need to support, or (b) application-side filtering. Secondary indexes in Cassandra are only efficient for low-cardinality columns on small datasets — the product catalog is neither. This would degrade Workload A's read performance by 10-50x and require extensive application-level query logic.

**Option B: Use MongoDB for both**. Workload B (event tracking) would suffer: MongoDB's write throughput is limited compared to Cassandra. At 100,000 writes/sec, MongoDB requires significant sharding (12-24 shards) and the shard balancer would constantly move data between shards. MongoDB's multi-region replication (via replica sets) is master-slave, not active-active — writes go to the primary in one region, causing 100-300ms write latency for remote users. Cassandra handles this natively with active-active multi-region.

**Option C: Separate clusters with divergent configurations on the same engine**. Even if you use the same database product (e.g., both on MongoDB or both on Cassandra), the configuration requirements are opposite: Workload A needs large nodes with lots of RAM (for indexes and working set) and SSDs. Workload B needs many smaller nodes optimized for write throughput with horizontal scalability. Running these on the same cluster would force suboptimal configuration for both.

**Recommended strategy: Two clusters, same orchestration layer**.

Keep both MongoDB and Cassandra running, but standardize the operational layer: use the same deployment tooling (Terraform, Kubernetes operators), monitoring stack (Prometheus + Grafana), backup strategy (consistent snapshots to S3), and CI/CD pipeline for schema changes. This gives you 80% of the operational savings (common runbooks, shared alerting, single team) without compromising either workload's performance.

**Alternative**: If the CTO's primary motivation is cost, evaluate whether Workload A could be migrated to PostgreSQL with the JSONB data type. PostgreSQL with JSONB supports JSON document storage, GIN indexes on JSON fields, and full SQL query capability. This handles Workload A's ad-hoc queries well and adds ACID transactions. The PostgreSQL team uses the same operational tooling as the Cassandra team. Total cost: one Cassandra cluster (Workload B) + one PostgreSQL cluster (Workload A) = two databases to manage, but each is the right tool for its job.

**The key lesson**: Consolidating databases for operational simplicity is valuable, but forcing workloads into the wrong data model creates more operational cost than it saves — in degraded performance, complexity of workarounds, and engineer frustration. The right consolidation target is operational tooling, not the database engine itself.
