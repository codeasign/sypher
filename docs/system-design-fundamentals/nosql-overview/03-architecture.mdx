# NoSQL Overview at Scale

## The Naive Implementation

When you first start building with a NoSQL database, the architecture is simple: one application, one database cluster, one datacenter. You deploy a few nodes, configure replication, and move on. This works at 1,000 users. At 100,000 users, the cracks appear. At 10 million users, the naive architecture becomes your biggest liability.

### What Fails at Each Tier

**At 1,000 users (everything works)**: Your 3-node Cassandra cluster serves all reads in under 5ms. Write latency is under 2ms. You have not thought about compaction, bloom filters, or consistency levels. Life is good.

**At 100,000 users (first cracks)**: Your 6-node cluster now handles 5,000 ops/sec. P99 read latency climbs to 50ms. A few bad queries — `SELECT * FROM events WHERE user_id > 0 ALLOW FILTERING` — cause nodes to time out under the load. Your monitoring dashboard shows compaction falling behind during peak hours. You add 3 more nodes and the problem subsides, but you realize: the architecture is not wrong, the data model and query patterns need discipline.

**At 1,000,000 users (system under stress)**: Your 12-node cluster handles 50,000 ops/sec with replication factor 3. P99 read latency hits 200ms during peak. Disk utilization is at 70% and growing. The weekend batch job that rebuilds materialized views now takes 6 hours and causes read timeouts across the cluster. You add a Redis caching layer for the top 1% of keys. You split your workload into two Cassandra clusters: one for real-time data (high write throughput, low retention) and one for historical analytics (lower throughput, longer retention).

**At 10,000,000 users (naive architecture fails)**: Your 30-node cluster handles 500,000 ops/sec, but you have hit hard limits. Cross-region users experience 200-400ms latency. A regional network issue takes down the entire database. Rolling upgrades require 8+ hours of careful orchestration. Compaction storms during peak hours cause P999 latency to spike to 5 seconds. You realize you need multi-region deployment, read replicas, application-level caching, and a completely different approach to data modeling — the naive architecture cannot scale here.

### Key Architectural Decisions in Production

**Read replicas**: A separate pool of nodes that receive asynchronous replication from the primary cluster. You route read-only queries (reports, dashboards, batch jobs) to replicas so they do not degrade write performance. Replicas can be configured with weaker consistency (CL=ONE) since they serve non-critical queries.

**Caching layer (Redis or Memcached)**: Sits in front of the NoSQL database and caches hot keys (the 5% of keys that receive 95% of reads). A cache hit returns in under 1ms. A cache miss falls through to the NoSQL database and populates the cache. This reduces read load on the database by 60-90% for read-heavy workloads.

**Workload isolation**: Real-time customer-facing queries (user profiles, session data) are served by the primary cluster. Analytics and batch queries (reports, data exports) are served by a separate analytics cluster. Mixing these workloads causes the "noisy neighbor" problem where a heavy analytics query starves real-time queries of I/O.

**Connection pooling and circuit breakers**: Application servers maintain persistent connection pools to the database. Circuit breakers (Hystrix, resilience4j) detect when the database is slow or failing and stop sending requests before the application itself runs out of threads. A tripped circuit breaker returns a fast failure (or cached fallback) instead of hanging for the database timeout (typically 10-30 seconds).

### Pattern 4: Hybrid SQL + NoSQL (Polyglot Persistence)

You use both SQL and NoSQL in the same application, each for what it does best. SQL for accounting, reporting, and strongly-consistent transactions. NoSQL for high-throughput, schema-flexible, latency-sensitive workloads.

A typical split: PostgreSQL for user accounts and payments (where ACID matters). DynamoDB for session data, activity feeds, and leaderboards (where low latency and high throughput matter). The two databases are synchronized via application-level dual-writes or change data capture (CDC).

## Multi-Region and Global Scale

### Replication Topologies

**Active-Passive**: One region handles all writes. Data replicates asynchronously to one or more passive regions that serve only reads. Failover requires DNS propagation and promoting a passive region to active. Write latency from non-primary regions is high.

**Active-Active (multi-master)**: Multiple regions accept writes independently. Data is replicated between regions asynchronously. This gives the best write latency (each client writes to the nearest region) but requires handling conflict resolution — different regions may accept conflicting writes to the same key.

### Consistency Tradeoffs Across Regions

When you replicate across continents, the speed of light imposes a hard floor on synchronous replication. Round-trip time between New York and London is roughly 80ms. Between New York and Sydney, about 200ms. If you use synchronous replication across regions, every write must wait for the slowest region. This is typically unacceptable for user-facing workloads.

The practical approach: use **synchronous replication within a region** (for strong consistency within the region) and **asynchronous replication between regions** (for availability and low latency globally). Accept that cross-region reads may be stale by seconds or, during network issues, minutes.

### Failover Strategy

When a region goes down:
1. **Health checks** on the load balancer detect that the region's database is unreachable (3 consecutive failed health checks, ~15 seconds).
2. **DNS failover** (Route53 or equivalent) updates the DNS record to point to the healthy region.
3. **Application servers** in the affected region are scaled down or terminated.
4. **Traffic** from the affected region is redirected to the nearest healthy region.
5. **The application** should be designed to tolerate the increased latency of being served from a distant region.
6. When the failed region recovers, data replication catches up (using hinted handoffs and batch replay).
7. Traffic is gradually shifted back to the recovered region using weighted DNS routing (10%-25%-50%-100% over 30 minutes).

## Capacity Planning

### QPS Calculation

You have a social app with 10 million monthly active users (MAU). Each user performs 50 read operations and 10 write operations per day.

- Daily reads: 10M users * 50 reads = 500M reads/day
- Peak reads per second: (500M / 86,400 seconds) * 2 (peak-to-average ratio of 2x) = approximately 11,600 reads/sec
- Daily writes: 10M * 10 writes = 100M writes/day
- Peak writes per second: (100M / 86,400) * 2 = approximately 2,300 writes/sec
- Total peak operations: approximately 14,000 ops/sec

A well-configured 6-node Cassandra cluster with RF=3 handles this comfortably with headroom.

### Storage Sizing

- Average write size: 1 KB (JSON document)
- Replication factor: 3
- Storage per day: 100M writes * 1 KB * 3 (RF) = 300 GB/day
- Storage per year: 300 GB * 365 = 109 TB/year
- With 30-day data retention (expired via TTL): 300 GB * 30 = 9 TB after steady state
- With 30% overhead for compaction and temp files: 9 TB * 1.3 = 11.7 TB total cluster storage
- Per node (6 nodes, no dedicated replication): approximately 2 TB per node

### Bandwidth Sizing

- Write bandwidth: 2,300 writes/sec * 1 KB = 2.3 MB/sec inbound
- Read bandwidth: 11,600 reads/sec * 1 KB = 11.6 MB/sec outbound
- Cross-node replication bandwidth (RF=3): 2.3 MB/sec * 2 (replica writes) = 4.6 MB/sec
- Cross-region replication bandwidth: 2.3 MB/sec (write volume to remote region)
- Total cluster bandwidth: approximately 18 MB/sec — manageable on 1 Gbps network links (theoretical max 125 MB/sec).

### Compute Sizing

- Each node should have 8-16 vCPUs
- Memory: enough for working set plus overhead (see deep-dive memory section)
- Disk: SSDs only — HDDs cannot sustain the compaction I/O. Provision 3x the expected data volume per node to account for compaction space amplification.
- Network: at least 10 Gbps for clusters handling 50,000+ ops/sec

## Deployment and Operations

### Zero-Downtime Rolling Upgrade

Upgrading a NoSQL database version without downtime:

1. **Plan**: Read the release notes for breaking changes, incompatible SSTable formats, or protocol changes.
2. **Seed nodes first**: Upgrade one node in each datacenter (seed nodes act as gossip contact points).
3. **Roll in waves**: Upgrade 25% of nodes at a time. After each wave, verify cluster health: gossip status, compaction throughput, P99 latency.
4. **Use `nodetool drain` (Cassandra)**: Before stopping a node, drain it to flush memtables, complete in-flight compactions, and gracefully hand off ownership.
5. **Upgrade SSTables**: After all nodes are upgraded, run `nodetool upgradesstables` to rewrite SSTables in the new format. This can take hours on large datasets. Schedule during maintenance windows.
6. **Monitor**: Watch for increased compaction pressure during SSTable upgrades. Disk space can temporarily double as old and new format SSTables coexist.

### Monitoring and Alerting

Essential monitoring setup for a NoSQL cluster:

- **Latency by percentile and consistency level**: alert when P99 exceeds 2x baseline for 5 minutes.
- **Compaction backlog**: alert when pending compactions exceed 100.
- **Disk usage**: alert at 70% (plan capacity), warn at 85% (urgent), critical at 92% (read-only mode may activate).
- **Node up/down status**: alert if any node is down for more than 2 minutes.
- **Hinted handoff count**: alert if hints exceed 10,000 per node, indicating a flapping node.
- **Garbage collection (JVM databases)**: alert if GC pause exceeds 1 second or GC time exceeds 10% of runtime.
- **Replication lag (cross-region)**: alert if lag exceeds 30 seconds.

### Debugging Common Issues

**High CPU, low throughput**: Likely compaction storm. Check `nodetool compactionstats`. If pending compactions are growing, reduce write load, add nodes, or switch to a more write-friendly compaction strategy (STCS instead of LCS).

**High latency for reads, low CPU**: Likely bloom filter misses or too many SSTables. Check `nodetool cfstats` — look at "SSTables per read" count. If it is above 5, compaction is not keeping up.

**Disk filling faster than expected**: Check for tombstone accumulation. Deletes in NoSQL databases create tombstones (delete markers) that occupy space until compaction. A table with frequent deletes can accumulate huge tombstone overhead. Verify TTLs are set correctly and compaction is running.

**Uneven load across nodes**: Token ring imbalance. Some nodes own more data or serve more requests. Use virtual nodes (vnodes) with 256 tokens per node for better distribution. If already using vnodes, check if new nodes were added without properly rebalancing.

### Schema and Data Modeling Migrations

Unlike SQL's `ALTER TABLE`, NoSQL schema changes are additive. You add a new field to your documents: old documents lack the field, new documents have it. Your application code handles both cases:

```python
# Reading a user profile that may or may not have "preferences"
profile = db.get("user:42")
prefs = profile.get("preferences", {})  # default to empty dict
theme = prefs.get("theme", "light")  # default to light theme
```

For backwards-incompatible changes (e.g., changing a field name from "lastName" to "family_name"), you run a background migration that reads old records, transforms them, and writes the new format. The application handles both formats during the migration window (typically 24-72 hours). After the migration completes, you add a data validation check to ensure no old-format records remain.

This approach — schema evolution via application code, not database commands — is the core reason NoSQL databases can support zero-downtime schema changes at any scale. It puts the responsibility on you, but it also gives you complete control over the migration speed and validation.
