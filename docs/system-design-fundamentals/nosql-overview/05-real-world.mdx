# NoSQL Overview in Production

## Company Case Studies

### Amazon: The Origin of DynamoDB

In 2004, Amazon's shopping cart service ran on a commercial relational database with a single master. The holiday shopping season pushed the master past its limit. A master failure during peak traffic meant the entire shopping cart service was down — and every page on Amazon that showed the cart count was broken. This was Amazon's "Google File System moment": the problem was not their code, it was the database architecture.

Amazon's engineers published the Dynamo paper in 2007, describing a key-value store designed for "always-on" availability. The design principles: always writable (even during failures), eventually consistent, optimized for high availability over strong consistency, and designed to run on commodity hardware. Every node in Dynamo was equal — no master, no single point of failure.

Key design decisions:
- **Consistent hashing** with virtual nodes to distribute load evenly across hundreds of servers.
- **Object versioning** with vector clocks to detect and resolve conflicts.
- **Hinted handoff** to survive temporary node failures without losing writes.
- **Gossip-based membership** so no centralized registry was needed.
- **Sloppy quorum** — during failures, writes go to the first N healthy nodes, not the first N in the ring. This sacrifices some consistency guarantees but keeps the system available.

DynamoDB, the managed service launched in 2012, is the public version of this design. It adds SSD-backed storage, predictable pricing per read/write unit, auto-scaling, global tables, and strong consistency as an option. At AWS re:Invent 2023, DynamoDB processed over 100 trillion requests per day across all customers. The largest single DynamoDB table holds over 50 TB and handles over 10 million requests per second.

### Google: Bigtable and the Birth of Wide-Column Storage

Google faced a different problem in the early 2000s. Crawling and indexing the web — billions of pages with trillions of links — required a storage system that could handle petabytes of data across thousands of servers. Existing databases could not scale to this size. Google's answer was Bigtable, described in the 2006 paper "Bigtable: A Distributed Storage System for Structured Data."

Bigtable is a wide-column store that runs on top of GFS (Google File System) and uses the Chubby lock service for coordination. Data is stored in tablets — contiguous ranges of rows — that are distributed across tablet servers. Each tablet is an immutable SSTable on GFS, with a memtable in memory for recent writes.

Key innovations:
- **SSTables**: immutable, sorted, on-disk data files that are never modified after writing. Compaction merges SSTables to reclaim space and improve read performance.
- **Bloom filters**: per-SSTable probabilistic filters that prevent unnecessary disk reads for non-existent keys.
- **Tablets**: data is split into 200 MB tablets. When a tablet grows past a threshold, it splits into two. Tablets are load-balanced across hundreds of tablet servers.
- **Compression**: Bigtable compresses individual columns using column-specific compression schemes, achieving 10:1 compression ratios for web crawl data.

Bigtable's legacy is enormous: it directly inspired HBase (the open-source Bigtable clone, now Apache HBase), Cassandra (which combines Bigtable's data model with Dynamo's distribution), and Google Cloud Bigtable (the managed service). Bigtable-based workloads at Google include Google Search (web index), Google Maps (tile and location data), Google Analytics (event tracking), and YouTube (video metadata and recommendations).

### Netflix: Cassandra as the Backbone

Netflix runs one of the largest Cassandra deployments in the world. In 2024, Netflix operates over 2,500 Cassandra nodes across multiple AWS regions, storing over 20 PB of data. Every piece of data that drives the Netflix experience — viewing history, recommendations, user profiles, streaming session metadata, A/B test assignments, and content metadata — lives in Cassandra.

Key lessons from Netflix's Cassandra journey:
1. **Separate user-facing and analytics clusters.** Mixing workloads causes the "noisy neighbor" problem — a heavy analytics query starves real-time queries of CPU and I/O.
2. **Restrict the Thrift API.** Netflix saw security incidents from raw Thrift access. They now route all database access through a RESTful API layer with authentication and rate limiting.
3. **Extensive monitoring.** Netflix monitors each table's P99 latency, compaction status, and disk usage. Alerts trigger when P99 exceeds a table-specific threshold, not a global one.
4. **Dark launches.** Before moving production traffic to a new cluster or configuration, Netflix replays recorded production traffic against the new setup to validate performance.
5. **Cassandra-sidecar.** Netflix runs Cassandra as a sidecar process on each EC2 application instance. Applications read cassandra data from their local sidecar, which reduces network hops and latency.

### Uber: From MySQL to Schemaless to DocStore

Uber's data platform evolution is a cautionary tale about migrating from SQL to NoSQL. By 2014, Uber's MySQL database was struggling with schema changes — every time a new city launched with different data requirements, a migration was needed. The engineering team built "Schemaless," a MySQL-based document store that stored JSON blobs with indexed fields, effectively turning MySQL into a NoSQL-like system.

In 2016, Uber migrated from Schemaless to its own document store built on Cassandra, called DocStore. DocStore stores JSON documents in Cassandra tables, with customizable secondary indexes. The schema is defined per "entity type" and consists of a list of indexed fields (called "indices") that are automatically maintained as documents are written.

Why Cassandra instead of MongoDB: Uber needed the multi-datacenter replication (Cassandra's native cross-DC replication) and the operational predictability of a system they could run themselves at global scale. MongoDB at the time did not have mature multi-datacenter support.

The lesson: building your own database layer on top of another database (Schemaless on MySQL, DocStore on Cassandra) is sometimes the right call when your requirements are specific enough. But it is expensive — Uber's database infrastructure team grew from 5 to over 50 engineers during the transition.

Discord stores every message ever sent — over a trillion messages — in Cassandra. Each channel is a partition, and messages within a channel are stored sorted by timestamp. The write path is simple: when you send a message, it is appended to the channel's partition as a new row.

The read path is also simple: when you open a channel, Cassandra does a range scan for the most recent 50 messages within that partition. This range scan is extremely fast because all messages for one channel are stored contiguously on disk, sorted by timestamp.

Discord's scaling challenge: some channels (like the official Fortnite channel) have hundreds of thousands of members sending thousands of messages per second. The partition for this channel becomes a hot spot. Discord mitigates this with a two-tier approach: frequently-accessed channel data is cached in Redis; Cassandra serves as the durable, canonical store.

**Apache Cassandra**: The most widely deployed open-source NoSQL database for write-heavy workloads. Combines Dynamo's distribution model with Bigtable's data model. Written in Java. Used at Netflix, Apple, Spotify, Instagram. Key strengths: multi-datacenter replication, linear write scalability, tunable consistency. Key weakness: read performance degrades under compaction pressure.

**ScyllaDB**: A Cassandra-compatible rewrite in C++. Replaces the JVM with Seastar's shared-nothing architecture (one thread per core). Claims 10x the throughput of Cassandra on the same hardware. Used at Discord (message store), Expedia (booking data), and IBM (cloud infrastructure). No garbage collection pauses — a major advantage for P999 latency.

**MongoDB**: The dominant document store, used at Coinbase, Uber (early days), Meta (some internal tools), and GitLab. MongoDB 5.0+ added native time-series collections, multi-document ACID transactions, and queryable encryption. Key weakness: the global write lock (removed in MongoDB 4.0 with document-level locking) was a historic bottleneck.

**Redis**: The standard for in-memory key-value storage. Used at Twitter (cache layer), GitHub (sidekiq queue), Pinterest (feed cache), and Stack Overflow (session store). Redis 7.0+ adds Redis Search, JSON, TimeSeries, and Probabilistic data structures as modules. Not a primary database — Redis trades durability for speed.

**Apache HBase**: The open-source Bigtable clone running on HDFS. Used at Meta (messages), Apple (Siri data), and Salesforce (marketing analytics). Where Cassandra is "write anywhere," HBase has a single HMaster that coordinates region assignment — a potential single point of failure.

## Lessons Learned

### Pattern 1: Start with Document Stores, Migrate to Wide-Column at Scale

MongoDB is the most common "first NoSQL database" for startups because it feels familiar (JSON, queries that resemble SQL-ish syntax). It handles schema flexibility well and scales to terabytes. But when you cross the petabyte threshold or need 100,000+ writes per second, the operational complexity of MongoDB (shard balancing, secondary index overhead, election storms during replica set failures) often drives teams to Cassandra or ScyllaDB.

The migration path: start with MongoDB for velocity. When MongoDB's operational complexity becomes the bottleneck, move the highest-volume data stores to Cassandra. Keep MongoDB for the data that benefits from its query flexibility (complex ad-hoc queries, aggregations, geospatial).

### Pattern 2: Never Build Your Own Query Engine on Top of NoSQL

Teams that try to build a SQL-like query engine on top of Cassandra or DynamoDB invariably fail. The impedance mismatch is too large: Cassandra does not support joins, subqueries, or arbitrary aggregations. Attempts to add these at the application layer either lead to O(N) queries or require maintaining materialized views that essentially replicate the data into another system.

Instead: use a purpose-built tool for each query type. Use Cassandra for high-throughput primary-key lookups. Use Elasticsearch for full-text search. Use Spark or Presto for analytical queries over Cassandra data. Accept that NoSQL means asking the database to do less — and doing the rest in dedicated systems.

### Pattern 3: Plan for Compaction from Day One

Compaction is the single most neglected operational concern in NoSQL. Teams plan for read throughput, write throughput, storage, and memory. They do not plan for compaction overhead — and compaction consumes 20-40% of disk I/O on a write-heavy cluster.

The rule: budget 30% of your disk I/O capacity for compaction overhead. If your application needs 1,000 IOPS for writes, ensure the disk can deliver 1,400 IOPS (the extra 400 is for compaction). Monitor compaction backlog as a first-class alert — it is often the leading indicator of an impending performance crisis.

### Pattern 4: Cache Relentlessly — But Invalidate Thoughtfully

A cache reduces read load on the database by 70-95% for most workloads. The trap is cache invalidation. The three strategies, from simplest to most complex:

**TTL-based (simplest)**: Set a TTL and accept stale data. Works for non-critical data where seconds of staleness are acceptable (leaderboards, trending topics, suggestions).

**Write-through (medium complexity)**: On every write to the database, also update the cache. Guarantees cache freshness for the data you write. Does not handle data modified by other paths (batch jobs, database migrations, CDC streams).

**Event-based (most complex)**: Publish invalidation events when data changes. Separate services subscribe and invalidate cache entries. Handles all write paths. Requires an event infrastructure (SNS+SQS, Kafka, Redis Pub/Sub).

### Pattern 5: Have a Fallback — Every System Goes Down

At some point, your NoSQL database will be unavailable or too slow to serve traffic. The best teams plan for this by implementing degradable experiences:

- **Leaderboard**: if Cassandra is slow, show cached scores from 5 minutes ago with a "refreshing" spinner.
- **User profile**: if DynamoDB is unavailable, serve a cached version from Redis with a "data may be stale" banner.
- **Product catalog**: if MongoDB is down, serve from a static JSON file deployed with the application.

These fallbacks turn a "site is down" incident into a "temporarily degraded" incident. Users forgive degraded. They do not forgive unavailable.
