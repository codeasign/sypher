## Tier 1: URL Shortener Revisited

**Problem:** A startup asks you to design a URL shortening service for internal use. Their marketing team sends 200 tracked links per day through SMS campaigns. Each link needs click tracking (who clicked, when, what device). The system must be operational in one week.

**Your task:** Run through the six-step process. Write down your requirements, estimates, architecture, deep dive, trade-offs, and wrap before reading the model answer below.

### Model Answer — Tier 1

**Requirements clarification:**
- 200 URLs created per day. Not 100 million. Two hundred.
- Reads: each link is clicked 50-500 times. At 200 links and 500 clicks max, that is 100,000 clicks per day.
- No custom aliases needed (internal team uses generated keys).
- Click tracking: timestamp, device type, approximate location (from IP), referrer.
- One-week deadline means no infrastructure setup time. Use existing tools.
- No high availability requirement — if the system is down for 10 minutes, the marketing team waits.

**Capacity estimation:**
- Writes: 200 URLs/day / 86,400 = 0.002 writes/sec. Effectively zero.
- Reads: 100,000 clicks/day / 86,400 = 1.2 reads/sec.
- Storage: 200 URLs x 500 bytes each = 100 KB new data per day. One million URLs over 5 years = 500 MB. Fits on a phone.
- Click storage: 100,000 clicks x 200 bytes each = 20 MB per day. 36.5 GB over 5 years. Still fits on a single machine.

**High-level design:**

This system fits on one server. The architecture is a single web application with SQLite as the database. No load balancer. No cache. No queue. No CDN.

```
Single server: nginx → Flask/Django app → SQLite database
```

**API design:**
```
POST /shorten
  Request:  { "url": "https://example.com/campaign-1", "campaign": "spring-sale" }
  Response: { "short_url": "https://short.ly/a1b2c3" }

GET /{short_key}
  Response: HTTP 302 redirect
  Side effect: INSERT INTO clicks (short_key, clicked_at, device, location)

GET /{short_key}/stats
  Response: { "clicks": 342, "top_devices": [...], "top_locations": [...] }
```

**Deep dive:**

Database schema:

```sql
CREATE TABLE urls (
    short_key    VARCHAR(10) PRIMARY KEY,
    original_url TEXT NOT NULL,
    campaign     VARCHAR(100),
    created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE clicks (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    short_key    VARCHAR(10) NOT NULL,
    clicked_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    device       VARCHAR(50),
    location     VARCHAR(100),
    referrer     TEXT,
    FOREIGN KEY (short_key) REFERENCES urls(short_key)
);

CREATE INDEX idx_clicks_key ON clicks (short_key);
```

Key generation: use a random 8-character alphanumeric string. At 200 URLs per day, collisions are essentially impossible (62^8 = 218 trillion keys).

**Trade-offs:**
- SQLite was chosen over PostgreSQL. SQLite requires zero setup, runs in-process, handles 100,000 reads/day with no problem, and the entire database fits in memory. PostgreSQL would require a server process, connection management, and configuration — wasted effort for this scale.
- No caching needed. The database query takes under 1ms for a 500 MB database. A cache layer would add complexity without benefit.
- No queue needed for click events. 1.2 writes per second is trivially handled by SQLite directly. Kafka would be absurd.

**Wrap:** Single-server Flask application with SQLite, deployed in 2 days. Handles 200 URLs/day with click tracking. Cost: zero dollars in new infrastructure (runs on existing internal server). If traffic grows 100x (20,000 URLs/day), migrate SQLite to PostgreSQL on the same server — takes one afternoon.

## Tier 2: Distributed Key-Value Store

**Problem:** Design a distributed key-value store similar to Amazon DynamoDB. The system must store 10 billion key-value pairs, handle 1 million reads per second and 500,000 writes per second, and provide eventual consistency with tunable consistency levels. Keys are strings up to 256 bytes. Values are binary blobs up to 1 MB.

### Model Answer — Tier 2

**Requirements clarification:**
- 10 billion key-value pairs, average value size 10 KB (mix of small and large values).
- 1M reads/sec, 500K writes/sec peak.
- Eventual consistency with tunable consistency per operation (read-your-writes available, strong consistency available at higher latency).
- Keys distributed uniformly (no hot keys to worry about for partitioning).
- Data must survive individual node failures (no single point of failure).
- Multi-region replication is not required.
- Operations: get(key), put(key, value), delete(key).

**Capacity estimation:**
- Storage: 10B x (256 bytes key + 10 KB average value + 100 bytes metadata) = 10B x 10.35 KB = 103.5 TB.
- With 3x replication: 310 TB.
- With 20% overhead for indexing and compaction: 372 TB.
- Node count: 48 nodes at 8 TB each.
- Network: 1M reads/sec x 10 KB = 10 GB/s read bandwidth. 500K writes/sec x 10 KB = 5 GB/s write bandwidth.
- A 10 Gbps network provides 1.25 GB/s per link. Need multiple links per node or 40 Gbps NICs.

**High-level design:**

Use consistent hashing to partition keys across nodes. Each node is a peer — no master, no coordinator. Nodes use gossip protocol for membership. Data is replicated to N nodes (N=3 by default).

```
Key → consistent hash → node responsible for key range
Each key stored on 3 nodes (preference list)
Read: contact all 3 nodes, return latest version by vector clock
Write: contact all 3 nodes, wait for R responses (tunable)
```

**API design:**
```
get(key) → { value, version, metadata }
put(key, value, consistency_level=QUORUM) → { success, version }
delete(key) → { success }
```

Consistency levels: ONE (any 1 node responds), QUORUM (N/2 + 1 nodes), ALL (all N nodes).

**Deep dive — Partitioning and Replication:**

**Trade-offs:**
- Consistent hashing was chosen over range partitioning (like Bigtable). Consistent hashing minimizes data movement when nodes join or leave. With range partitioning, adding a node requires splitting a range and moving half the data — expensive at 103 TB.
- Vector clocks were chosen over last-writer-wins (LWW). LWW loses data when concurrent writes happen — the last timestamp wins, even if it is an older write that arrived late. Vector clocks track causality and allow the application to resolve conflicts. Trade-off: vector clocks grow with the number of nodes that modified a key. Practical limit: 10-20 nodes per key.
- Gossip protocol was chosen over a centralized membership service (like ZooKeeper). Gossip is fully distributed — no single point of failure, no coordinator bottleneck. Trade-off: membership changes take 30-60 seconds to propagate. For 48 nodes, this is acceptable.
- 3x replication was chosen over 2x or 5x. 2x is insufficient (loss of one node loses data). 5x provides better read throughput but 67% more storage cost. For 103 TB, 3x (310 TB) is the sweet spot.

## Tier 3: Global Real-Time Analytics Platform

**Problem:** Design a real-time analytics platform that tracks user events (page views, clicks, purchases) and provides sub-second query results over rolling time windows. Scale: 100 million events per second. Query latency under 200ms. Dashboards must update in real time. Events have 50 fields (user_id, event_type, timestamp, page, referrer, device, location, etc.). 100,000 unique dashboards run simultaneously.

### Model Answer — Tier 3

**Requirements clarification:**
- 100 million events per second. At 500 bytes per event: 50 GB per second. 4.3 PB per day.
- Sub-second query latency: under 200ms for 99th percentile.
- Real-time dashboards: data visible within 5 seconds of the event occurring.
- 100,000 unique dashboards, each with its own query filter and aggregation.
- Retention: raw data for 7 days, aggregated data for 1 year.
- Queries are time-range aggregations: count of events by type per minute, average purchase value per hour, top 10 pages by visits per day.

**Capacity estimation:**
- Ingestion: 100M events/sec x 500 bytes = 50 GB/sec. Need 40 servers at 1.25 GB/s each just to ingest.
- Storage (raw, 7 days): 50 GB/sec x 86,400 sec/day x 7 days = 30.2 PB.
- Storage (aggregated, 1 year): depends on granularity. If pre-aggregated to 1-minute buckets with 5 dimensions, storage drops to under 100 TB.
- Query QPS: 100,000 dashboards refreshing every 30 seconds = 3,300 queries per second. Each query scans pre-aggregated data — under 100ms per query.
- Lambda architecture: streaming layer (Apache Kafka + Flink) for real-time data + batch layer (Spark) for correction.

**High-level design:**

Lambda architecture with Kafka for ingestion, Flink for stream processing, Druid for real-time OLAP, and Presto for batch queries.

**Deep dive — Flink streaming aggregation:**

Flink consumes from 256 Kafka partitions with 128 Flink task slots. Each task slot runs a windowed aggregation: 1-minute tumbling windows with 5 dimensions (event_type, page, country, device, user_segment).

```java
// Flink streaming job — pseudo-code
DataStream<Event> events = env.addSource(kafkaConsumer);

events
    .keyBy(event -> event.eventType + "|" + event.page)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new CountAndSumAggregator())
    .addSink(druidSink);  // Write to Druid for real-time queries

// The batch layer (Spark) recomputes the same aggregation
// from raw Parquet files to correct any speed-layer errors
```

**Trade-offs:**
- Lambda architecture was chosen over Kappa architecture (stream-only). Kappa is simpler (one code path) but requires reprocessing the entire stream for any aggregation error. With 100M events/sec, reprocessing 7 days of data (60 PB) takes hours. Lambda allows fast corrections through the batch layer.
- Druid was chosen over Elasticsearch for real-time OLAP. Druid is purpose-built for time-series aggregations with sub-second queries on pre-aggregated data. Elasticsearch excels at log search and text queries — a different use case. Druid also handles 100,000 concurrent dashboards more efficiently through its segment-based architecture.
- Kafka with 256 partitions was chosen over a smaller partition count. Each partition is consumed by one Flink task slot. 256 partitions gives 256-way parallelism for ingestion and aggregation. Fewer partitions would limit throughput. More partitions would increase Kafka's metadata overhead.
- Pre-aggregation was chosen over raw-event querying. At 100M events/sec, querying raw data in sub-second time is impossible — even a 1-minute scan covers 6 billion events. Pre-aggregation to 1-minute buckets reduces the scanned data by a factor of 10,000.

## Bonus Challenge — Design a Parking Garage System

This is a common real-world system design problem that tests your ability to apply the process to a physical system with hardware constraints. Work through it on your own before reading the outline below.

**Prompt:** Design a parking garage system for a 5,000-space parking garage near a major airport. The system must track available spaces, accept reservations, handle license plate recognition at entry and exit, process payments, and provide real-time availability on a website. Peak traffic is 500 cars per hour entering and 500 cars per hour exiting during flight departure/arrival windows.

**Process outline (your task: expand each step into a complete design):**

1. **Clarify:** What sensors detect occupancy? How do reservations work (prepaid, time-slot, or first-come)? What payment methods? Do we need monthly passes? What is the latency for gate opening (must be under 2 seconds — cars cannot wait)?

2. **Estimate:** 500 cars/hour peak = 8.3 cars/minute peak. Each car stays 3-7 days (airport parking). 5,000 spaces at 80% occupancy = 4,000 occupied. Reservation rate: 2,000 advance reservations per day. Storage: 100 bytes per event x 10 million events per year = 1 GB. This system fits on a Raspberry Pi with PostgreSQL.

3. **Design:** Three subsystems: entry gate (license plate camera + ticket dispenser + barrier), exit gate (license plate camera + payment terminal + barrier), reservation system (web app + database). Architecture: local server at the garage (for reliability — internet goes down) with cloud sync for the reservation website.

4. **Deep dive:** License plate recognition (OCR accuracy at 95% — need fallback for unreadable plates: manually enter plate at kiosk). Payment processing (credit card at exit kiosk, pre-pay online with QR code). Occupancy tracking (magnetic loop sensors in each space or camera-based counting at entry/exit). Reservation expiration (hold a space for 30 minutes past reservation time, then release).

5. **Trade-offs:** Local server vs cloud (local is more reliable when internet is down, cloud is easier to maintain — decision: local with cloud sync because garage gates must work 100% of the time). Magnetic sensors vs camera counting (sensors are more accurate but more expensive per space — decision: camera counting at entry/exit is cheaper and sufficient). QR code vs RFID for reservation validation (QR code works with any smartphone, RFID is faster but requires hardware distribution — decision: QR code for convenience).

6. **Wrap:** System handles 5,000 spaces with 500 cars/hour peak. Local server runs PostgreSQL and gate control software. Cloud sync pushes availability data to the website every 30 seconds. License plate recognition at entry and exit. QR code for reservation validation. Total infrastructure cost: approximately $50,000 for hardware (cameras, barriers, kiosks, server) plus $200/month for cloud services.

## Additional Practice Problems

Use the six-step process to design these systems. Each one emphasizes a different aspect of the process.

**Problem 1: Distributed Rate Limiter**
Design a rate limiter that enforces "100 requests per minute per user" across a cluster of 50 application servers. The rate limiter must be accurate within 5% of the configured limit and must not become a bottleneck.

Process emphasis: The deep dive focuses on the rate-limiting algorithm (token bucket vs sliding window vs leaky bucket) and the coordination mechanism (centralized Redis vs distributed counters with gossip). The estimation phase calculates the number of Redis operations per request and the memory required for the counter data.

**Problem 2: Global File Storage and Sharing Service**
Design a Dropbox-like service with 1 billion users. Each user stores 2 GB of files. The service supports sharing, version history (30 days), and cross-device sync.

Process emphasis: The estimation phase calculates massive storage requirements (2 exabytes total, 50 TB of new data per day). The high-level design focuses on the sync protocol (delta sync vs full file sync) and the storage architecture (blob storage for file content, metadata database for file index). The deep dive covers conflict resolution (last-writer-wins vs version branching).

**Problem 3: Real-Time Leaderboard**
Design a real-time leaderboard for a mobile game with 10 million daily active players. The leaderboard updates within 1 second of a score being submitted. Players can see their rank, the top 100 players, and the rank of their friends.

Process emphasis: The estimation phase calculates 1 million score updates per minute (16,667 writes/sec) and 10 million leaderboard views per day (115 reads/sec). The deep dive focuses on the sorted set data structure (Redis sorted sets with O(log N) insertion and O(1) rank retrieval) and the partitioning strategy (shard by score range or use a single sorted set with a cluster).

**Problem 4: URL Shortener — Revisited**
Design a URL shortener, but this time with a twist: the system must handle 10 billion URLs per day (not 100 million). This is the scale of a global link management platform like bit.ly at its peak.

Process emphasis: The estimation phase now shows 115,740 writes per second — well beyond a single PostgreSQL instance. The architecture must change. The deep dive must address multi-tier sharding (shard by key hash, then by date), the key generation service must handle 115K keys per second without becoming a bottleneck, and the analytics pipeline must handle 1 trillion click events per day.

**Problem 5: Uber-Like Ride Hailing System**
Design a ride-hailing system that matches 10 million riders with 1 million drivers per day across a metropolitan area.

Process emphasis: The estimation phase is complex — you must calculate the number of active drivers per hour, the ride request rate (max 5,000 requests per second at peak), and the geolocation update rate (each driver sends location every 3 seconds). The deep dive focuses on the geospatial index (quad-tree or S2 geometry for finding nearby drivers) and the dispatch algorithm (nearest driver vs highest-rated driver vs minimum wait time). The trade-off analysis compares geospatial databases (Redis with GeoSet, PostgreSQL with PostGIS, Elasticsearch with geo queries).

You have now completed three full design exercises spanning three orders of magnitude of scale, plus a bonus physical system and five additional practice problem outlines. The same six-step process — clarify, estimate, design, deep dive, trade-offs, wrap — produced radically different architectures for each problem. A URL shortener at 200 URLs per day uses SQLite on one server. A URL shortener at 100 million URLs per day uses PostgreSQL with replicas, Redis cache, and RabbitMQ. A URL shortener at 10 billion URLs per day uses Cassandra with multi-tier sharding and a distributed key generation service. Same process, different inputs, different outputs. The process does not give you the answer. It gives you the structure to find the answer yourself. Apply it to every system design problem you encounter, and you will never start a design by guessing.
