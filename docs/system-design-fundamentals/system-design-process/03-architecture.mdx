The system design process produces different architectures depending on how rigorously you apply it. A team that skips the process draws a different system than a team that follows it. A team designing for one data center draws a different system than a team designing for five continents. This section shows the architecture that emerges at each level of process maturity.

## Naive Architecture — No Process

Without the process, engineers draw what they know. A junior engineer who learned about microservices yesterday designs microservices for everything. A senior engineer who spent five years at a Kafka shop designs Kafka for everything. The resulting architecture reflects the designer's toolbox, not the problem's requirements.

The naive architecture for a URL shortener looks like this: a load balancer, three microservices (create, redirect, analytics), a Redis cache, a PostgreSQL database, and a Kafka pipeline. The engineer drew this because it is the standard FAANG architecture they memorized. They did not ask how many URLs per day. They did not calculate storage. They did not check whether the write volume justifies Kafka.

**Problems with the naive approach:**

- Three microservices are unnecessary for 12,000 QPS. A monolith handles this comfortably. Each microservice adds deployment overhead, latency between services, and debugging complexity. The team chose microservices because "that is what Netflix does," not because the URL shortener needs independent scaling of the create and redirect paths.
- A single PostgreSQL instance with no read replica creates a single point of failure. An index scan on the urls table at 12,000 QPS pushes a single node to 70% CPU. When the checkpoint fires every 5 minutes, CPU spikes to 100% and queries queue up. One server, one failure mode.
- Redis caching every URL consumes 42 TB of memory for the full dataset. The hot set (top 1% of URLs) fits in 5 GB. The naive designer threw money at the cache instead of analyzing access patterns.
- Kafka is overkill for 11,600 click events per second at 190 bytes each. RabbitMQ handles this volume with lower latency and simpler operations. Kafka's replay capability adds no value for click events that are consumed once and written to an analytics database.

## Production Architecture — Process Applied

Now apply the six-step process. Requirements clarified: 100 million URLs per day, 1 billion redirects per day, p99 under 50ms, 99.99% availability. Capacity estimated: 23 GB of new storage per day, 126 TB over 5 years with replication, 12,000 QPS peak. These numbers drive every architectural decision.

**What the process changed:**

- **Monolith over microservices.** The capacity estimate shows 12,000 QPS. A single monolith on 6 servers handles this easily. Microservices add latency, complexity, and operational cost without benefit. The process forced the team to ask: "Do we need independent scaling of the create and redirect paths?" The answer was no — both paths grow at the same rate.
- **PostgreSQL primary + 2 replicas.** The 99.99% availability requirement means a single database is unacceptable. Two read replicas handle read traffic and provide failover targets. The process forced the question: "What happens when the database goes down?"
- **Redis caches only hot URLs.** The process included analyzing the access pattern. 10% of URLs get 90% of traffic. Caching all 42 TB of URLs would require 126 TB of Redis memory — prohibitively expensive. Caching the hot set (top 5 million URLs, ~5 GB) achieves a 99% cache hit rate. The process forced the question: "Which URLs are actually read frequently?"
- **RabbitMQ instead of Kafka.** The click event stream is 11,600 events per second at 190 bytes each. RabbitMQ handles this with one-third the operational complexity of Kafka. The replay capability and 1M+ messages/second throughput of Kafka add no value here. The process forced the question: "What throughput and features does the queue actually need?"
- **ClickHouse for analytics storage.** Columnar storage compresses click event data by 10x compared to row-oriented storage. The 190 GB of daily click data compresses to 19 GB in ClickHouse. The process forced the question: "What is the query pattern on analytics data?" The answer (aggregations over time ranges) points directly to a columnar database.

## Integration Architecture — Fitting into an Existing System

The process changes when you are integrating into an existing system rather than building greenfield. You have constraints that the greenfield process does not account for: existing database schemas, existing API contracts, existing team ownership boundaries, and existing infrastructure.

**Process modifications for integration:**

- **Requirements phase** must include existing system constraints. "What databases are already in use? What message queue infrastructure exists? What is the team's deployment cadence? What monitoring infrastructure must the new system integrate with?"
- **Capacity estimation** must account for shared infrastructure. The new system may share a database cluster, a Kafka cluster, or a load balancer with existing services. Capacity is not just the new system's traffic — it is the new system's traffic added to the existing baseline.
- **High-level design** must show integration points explicitly. Existing services are "brownfield" — you cannot change their interfaces. The new system must adapt to their API contracts, not the other way around.
- **Deep dive** must include migration strategy. "How do we move data from the old system to the new system without downtime? How do we handle the transition period where both systems are live?"

## Multi-Region Architecture — Global Scale

The process produces a different architecture when the requirements include global scale. A single-region design and a multi-region design share the same six steps, but the answers at each step differ.

**How the process changes for multi-region:**

- **Requirements phase** must answer: "Which regions initially? Which regions in 2 years? Do we need active-active (writes in any region) or active-passive (writes in one region, reads in others)? What is the cross-region latency budget? What regulatory constraints exist (GDPR for EU data, China firewall for Asia)?"
- **Capacity estimation** must account for inter-region bandwidth. Replicating data between US East and EU West costs money and adds latency. The replication traffic may exceed the user-facing traffic.
- **High-level design** must include global load balancing (DNS-based), per-region deployments, and cross-region data replication.
- **Deep dive** must include conflict resolution for concurrent writes. If user A in New York and user B in London both create the same custom alias within 10 milliseconds of each other, which one wins?

## Deployment Architecture — Iterating Through the Process

The process is not a one-time activity. It is a cycle you repeat as the system evolves. The initial design targets year 1 requirements. Year 2 requires a new pass through the process with updated numbers. Year 3 requires another pass. Each iteration changes the architecture.

**Iteration 0 (Year 0, MVP):** 10,000 URLs per day. Single server. SQLite. No cache. No queue. No analytics. Deployed in 2 weeks. The process gave the answer: "minimal viable architecture."

**Iteration 1 (Year 1, Launch):** 1 million URLs per day. PostgreSQL. Redis cache. RabbitMQ queue. Simple analytics. Three application servers. Deployed in 2 months. The process gave: "a properly sized architecture for launch."

**Iteration 2 (Year 2, Growth):** 100 million URLs per day. PostgreSQL primary + 2 replicas. Redis cluster with layered TTLs. RabbitMQ with batch consumers. ClickHouse for analytics. Six application servers. The process gave: "horizontal scaling for 100x growth."

**Iteration 3 (Year 3, Global):** 500 million URLs per day. Multi-region active-passive. Global DNS routing. Per-region database clusters with cross-region replication. Per-region cache clusters. Regional queue infrastructure. The process gave: "global architecture for international expansion."

## Case Study Walkthrough — Process Applied to a Video Platform

To solidify how the process drives architecture, here is a complete walkthrough of a video platform design. Each phase of the process produces a specific architectural output.

### Phase 1: Requirements -> Architecture Impact

Requirements: 500 million DAU, 500 hours of video uploaded per minute, p99 first-frame latency under 200ms, supports 720p/1080p/4K, global delivery.

Architecture impact of each requirement:
- 500M DAU means you need a CDN. Self-hosting video delivery for 500M users requires 30,000+ servers and 250 Tbps of bandwidth. A CDN handles this for a fraction of the operational cost.
- 500 hours uploaded per minute means you need an async transcoding pipeline. A user cannot wait for 4K transcoding to finish before seeing their video. The upload must return immediately, with the video becoming available progressively as transcoding completes.
- 200ms p99 first-frame latency means the CDN must be configured for video streaming (not just HTTP caching). You need segments (HLS/DASH), adaptive bitrate, and edge-optimized delivery.
- Global delivery means you need multiple CDN origins (one per continent) or a single origin with a global CDN provider that has points of presence in every region.

### Phase 2: Estimation -> Architecture Impact

Estimates:
- Storage: 500 hours/min x 500 MB/hour average bitrate = 4.2 TB of new video per minute. 6 PB per day. 2.2 exabytes per year.
- Bandwidth: 500M DAU x 30 minutes of viewing per day x 5 Mbps = 75,000 Tbps total. With CDN offloading 95%: 3,750 Tbps from origin.
- Transcoding compute: 500 hours/min of raw footage requires approximately 15,000 transcoding hours per minute (each video is transcoded to 3 resolutions x 2 codecs = 6 variants). At 3x real-time transcoding speed on a GPU instance: 5,000 GPU-minutes per minute of wall clock time. You need 5,000 concurrent GPU instances.

Architecture impact:
- 2.2 exabytes per year means you cannot store processed video on spinning disk. You need a tiered storage strategy: hot (CDN cache, hours), warm (SSD-based origin storage, weeks), cold (tape or long-term S3 archival, months).
- 5,000 concurrent GPU instances means you cannot use a single queue with a single worker pool. You need a partitioned job queue (one partition per codec/resolution combination) with auto-scaling groups per partition.

### Phase 3: High-Level Design -> Architecture

The high-level architecture for the video platform:

```
User uploads video → Load balancer → Upload service → Raw storage (S3)
                                                  → Transcoding job queue
                                                  → Transcoding workers (5,000 GPUs)
                                                  → Processed storage (S3 + CDN)
```

```
User watches video → CDN edge → If miss: CDN origin server → Processed storage (S3)
```

The architecture separates the upload path (write-heavy, async) from the viewing path (read-heavy, latency-sensitive). The upload path handles 4.2 TB of new data per minute. The viewing path handles 75,000 Tbps of outbound traffic.

### Phase 4: Deep Dive -> Architecture Decisions

Deep dive into the transcoding pipeline reveals that the job queue must be partitioned by resolution. 4K transcoding takes 10x longer than 720p transcoding. If all resolutions share one queue, 4K jobs starve the workers and 720p jobs wait behind them.

Architecture decision: separate queues per resolution with dedicated auto-scaling groups.

```
Queue: "transcode-720p" → Workers: 1,000 GPU instances (fast, cheap instances)
Queue: "transcode-1080p" → Workers: 1,500 GPU instances (mid-range instances)
Queue: "transcode-4k" → Workers: 2,500 GPU instances (high-end instances)
```

This ensures that a spike in 4K uploads does not delay 720p availability. 720p video is available within 30 seconds of upload. 4K may take 5 minutes.

### Phase 5: Trade-Offs -> Architecture Decisions

Trade-off: HLS vs DASH for streaming format.

HLS (Apple HTTP Live Streaming): longer segments (6 seconds), wider device support (iOS native), simpler to implement. DASH (Dynamic Adaptive Streaming over HTTP): shorter segments (2-4 seconds), better adaptive bitrate switching, more codec flexibility.

Decision: support both (HLS for iOS/Safari, DASH for Chrome/Android). Two packaging pipelines = more compute but better user experience.

Trade-off: MP4 vs fragmented MP4 for storage format.

MP4: single file, simple, but requires the entire file to be downloaded before playback starts. Fragmented MP4: broken into segments, supports byte-range requests (skip to middle of video without downloading the beginning), and is the native format for HLS/DASH.

Decision: fragmented MP4. The 10% storage overhead (fragmented headers are duplicated per segment) is worth the streaming flexibility.

### Phase 6: Wrap -> Architecture Roadmap

Current architecture: CDN + S3 + GPU transcoding pipeline + HLS/DASH packaging.

Year 2 roadmap: Add user-generated content moderation pipeline (AI-based content scanning between raw storage and CDN). Implement predictive pre-fetching to edge CDN nodes based on trending content.

Year 3 roadmap: Migrate from CDN to a peer-to-peer delivery network for popular content (reduces CDN costs by 30%). Implement regional storage for GDPR compliance (EU user data stays in EU).

## Architecture Anti-Patterns — When the Process Produces Bad Architecture

Even when you follow the process correctly, you can produce bad architecture. Here are three anti-patterns that the process does not automatically prevent.

### Anti-Pattern 1: The Hedgehog — All Eggs in One Technology

The process told you to use PostgreSQL for the database. You used PostgreSQL for the primary database. You used PostgreSQL for the analytics database (with a different schema). You used PostgreSQL for the session store. You used PostgreSQL for the job queue (with `SELECT ... FOR UPDATE` polling). You have one technology solving every problem — but PostgreSQL is not optimized for session storage (Redis is better), job queues (Kafka/RabbitMQ are better), or analytics (ClickHouse is better).

The hedgehog anti-pattern happens when the process does not ask the question: "What is the best tool for this specific problem?" It only asks: "What database should we use?" — and you pick one answer for everything.

### Anti-Pattern 2: The Goldilocks — Three Options, One Always Wins

Every trade-off analysis in your design has three options, and the middle option always wins. PostgreSQL vs Cassandra vs SQLite — you choose PostgreSQL. Kafka vs RabbitMQ vs Redis streams — you choose Kafka. S3 vs HDFS vs local disk — you choose S3. The goldilocks anti-pattern means you are not really analyzing trade-offs — you are defaulting to the familiar choice and naming two straw-man alternatives.

The fix: for each trade-off, commit to a specific reason why the extreme options fail. "SQLite fails because the write volume exceeds its single-writer limitation." "Cassandra is overkill because we do not need 1M writes/sec or multi-region replication." A genuine trade-off analysis eliminates alternatives with specific, falsifiable reasons.

### Anti-Pattern 3: The Waterfall — No Iteration Within the Process

You move through the six steps linearly and never go back. You estimated 100 million URLs per day. In the deep dive, you realize the key generation service cannot produce unique keys at 1,200 per second using the approach you designed. You should go back to the estimation phase and recalculate key generation throughput. You should go back to the high-level design and add a key generation service as a separate component. But you do not — you are committed to the linear process.

The waterfall anti-pattern treats the process as a one-way street. In reality, the process is a cycle. Each deep dive may reveal that the high-level design needs adjustment. Each trade-off analysis may reveal that the capacity estimate was wrong. The fix: explicitly build a feedback loop into your process. After each step, ask: "Does this change anything I decided in a previous step?"

## The Process as a Lens for Architecture Reviews

Architecture reviews are where the process meets reality. A team presents their existing architecture, and you evaluate it using the same six steps.

**Reviewing an architecture with the process:**

1. **Check the requirements** — are they documented? Do the non-functional requirements have specific numbers (under 200ms p99 latency, 99.99% availability) or vague goals ("fast," "reliable")? Vague requirements produce vague architectures.

2. **Check the capacity estimation** — are the numbers on the diagram (10M DAU, 1,200 QPS, 8.4 TB/year)? Or is every component sized by guesswork? If the database is "sized for growth" without specifying the growth number, the estimation step was skipped.

3. **Check the high-level design** — does every component have a clear responsibility? Can you trace a request from entry to storage and back? Are there components whose purpose is unclear? Unclear components are symptoms of unclear requirements.

4. **Check the deep dive** — are the schemas written down? Is the cache strategy defined (what keys, what TTL, what eviction policy)? Is the partitioning strategy explicit? If the deep dive is missing for critical components (database, cache), the design is not ready for implementation.

5. **Check the trade-offs** — are there documented alternatives? Did the team consciously reject other approaches? A design with no documented trade-offs is a design where decisions were made by default, not by analysis.

6. **Check the wrap** — is there a migration plan? A rollback plan? A monitoring plan? A design without a rollout plan is a design that will break in production.

## Example Architecture Review Walkthrough

Consider reviewing a team's design for a notification service. The team proposes: 3 microservices (email, SMS, push notifications), Kafka for event ingestion, PostgreSQL for user preferences, Redis for rate limiting, and a React admin dashboard.

**Review using the process:**

Requirements check: "What is the scale?" Team: "1 million notifications per day." Requirement target: 99.99% delivery within 5 seconds. Gap: the team did not document the latency requirement. Without it, they cannot verify whether Kafka adds unnecessary latency.

Estimation check: "1 million per day = 12 per second. Notifications are 1 KB each = 12 KB/s. This fits on one server with nginx. Do you need 3 microservices or can this be a monolith?" The team's architecture assumes massive scale that does not exist.

High-level check: "The React admin dashboard reads from PostgreSQL directly. At 12 QPS, this is fine. But what about the Kafka pipeline — why do you need Kafka for 12 events per second?" The team chose Kafka because they read that "Kafka is for notifications." They do not need Kafka.

Deep dive check: "The PostgreSQL schema has a notifications table with 15 columns. The team is storing the full notification payload plus delivery status plus retry count. This is fine for 12 writes per second. But the schema is missing an index on (user_id, created_at). Without it, the 'fetch my last 50 notifications' query does a sequential scan."

Trade-off check: "The team chose PostgreSQL over MySQL. Reason: 'PostgreSQL is better.' This is not a trade-off analysis. At 12 writes per second, both databases work equally well. The real trade-off is: single server (simpler, no failover) vs replicated setup (more complex, HA). The team chose replication but cannot articulate why."

Wrap check: "No migration plan is documented. The team plans to 'turn it on and see what happens.' This is not acceptable for a notification service that must have 99.99% delivery." The review identifies the missing rollout plan as a blocker.

**Review outcome:** The architecture is over-engineered (Kafka, 3 microservices) for the actual scale (12 events/sec). The team should simplify to a monolith with PostgreSQL and skip Kafka. The deep dive reveals a missing index. The wrap is missing entirely. The review redirects the team to a simpler, correct architecture.

## Architecture as Code — Diagrams as Living Documentation

The process output is usually a static diagram or document. In modern engineering teams, the architecture should be "as code" — version-controlled, reviewable, and automatically generated from configuration.

Options for architecture-as-code:

- **PlantUML:** Text-based UML diagram generator. Write `@startuml` blocks in your design doc. The diagram is generated from text, so it can be reviewed in a pull request. PlantUML supports component diagrams, sequence diagrams, and deployment diagrams.
- **Mermaid:** JavaScript-based diagram generator. Native support in GitHub Markdown. Write ````mermaid` blocks in your README or design doc. The diagram renders inline in the GitHub UI.
- **Structurizr:** DSL for C4 model diagrams. Define your system, containers, components, and relationships in a structured DSL. Generate multiple diagram views from the same model. The C4 model (Context, Container, Component, Code) maps directly to the system design process: Context = requirements, Container = high-level design, Component = deep dive.
- **Diagrams as Python code:** The `diagrams` Python library lets you define infrastructure as Python code. `from diagrams import Diagram, Cluster; from diagrams.aws.compute import EC2; with Diagram("Web Service"): EC2("web")`. The output is a PNG that updates automatically when the code changes.

Architecture-as-code solves the biggest problem with design documents: they go stale. A handwritten diagram in a wiki page is out of date the moment the system changes. An architecture-as-code diagram is regenerated from the source of truth and stays accurate.
