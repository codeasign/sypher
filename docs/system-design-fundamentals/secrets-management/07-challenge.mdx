## Tier 1 -- Foundation

Five minutes each. Answer in 2-3 sentences per question.

### Challenge F1: Secret Rotation Strategy

**Prompt**: Your team stores a database password as a string literal in a Kubernetes Secret. The password has been unchanged for 18 months. An auditor flags this as a security risk. You need to rotate the password without downtime. The application connects to the database using a connection pool of 20 connections. The database has no connection draining mechanism. Describe the rotation steps and the risk of not rotating.

**Evaluation criteria**: Did you identify that the application holds stale connections that will break after rotation? Did you propose a zero-downtime strategy (dual-credential window or connection refresh)? Did you explain why the 18-month gap is a real threat (credential leakage window, compliance violation)?

**Model answer**: The key risk of not rotating for 18 months is that any leaked credential remains valid indefinitely -- a compromised CI/CD token, a developer's laptop with the secret in shell history, or an insider exfiltration event all grant access forever. Compliance frameworks (SOC 2, PCI-DSS, ISO 27001) all require periodic rotation specifically to limit this exposure window. Zero-downtime rotation requires a dual-credential window: first, update the database to accept both the old and new password simultaneously. Most databases support this -- PostgreSQL accepts `ALTER USER ... PASSWORD` while keeping the old password in a grace period via `PASSWORD_OLD`. MySQL supports `ALTER USER ... IDENTIFIED BY` with `RETAIN CURRENT PASSWORD`. Next, deploy the application with the new password -- this can be done by updating the Kubernetes Secret and letting the pod restart. Since both passwords work, old connections survive until the pods restart and establish new connections using the new password. Once all pods have recycled and the old connections are replaced, revoke the old password. The common mistake is updating the Kubernetes Secret but not restarting pods -- the application process continues using the stale in-memory credential from the connection pool until the next error-driven reconnect, which causes a brief outage wave. Another common mistake is rotating the password without a dual-credential window: every pod restarts simultaneously, they all try to connect with the new password, and any timing issue causes a cascading failure.

### Challenge F2: Dynamic Secrets Explained

**Prompt**: Your DevOps lead says "just use static secrets -- they're simpler and we rotate them every 30 days." Explain what dynamic secrets are, why they are more secure than short-lived static secrets, and describe the runtime cost of dynamic secrets that the DevOps lead may be concerned about.

**Evaluation criteria**: Did you clearly distinguish dynamic secrets (generated on-demand, valid for one session/task, automatically revoked) from static secrets (long-lived, manually rotated)? Did you explain the security advantage (reduced blast radius, no credential sharing, no rotation overhead)? Did you honestly address the cost (latency of secret generation, load on the secrets backend, need for application code changes)?

**Model answer**: Dynamic secrets are credentials that a secrets engine (like HashiCorp Vault) generates on-demand for a specific workload, with a configurable TTL (typically 1-60 minutes). Each application instance gets a unique credential, valid only for its lifetime. The credential is automatically revoked when the TTL expires or when the lease is explicitly revoked. Compare to static secrets: with a 30-day rotation, if a developer commits a static secret to GitHub, it is exposed for up to 30 days before rotation removes its validity. With dynamic secrets, the exposed credential expires in minutes -- the blast radius shrinks from 30 days to 30 minutes. The security advantage is profound across three dimensions: (1) no credential sharing -- each instance has its own identity, so a breach of one instance does not expose other instances; (2) automatic revocation -- the secret disappears when the TTL expires, eliminating orphaned credentials from decommissioned instances; (3) no rotation process -- rotation is continuous and automatic, removing the operational burden of scheduling, testing, and verifying manual rotations. The runtime cost the DevOps lead worries about is real and should not be dismissed: each secret generation requires a round-trip to Vault (5-10ms latency for the API call plus TLS handshake overhead), Vault must track every issued secret in its storage backend (storage and memory overhead), the database must handle frequent credential creation and revocation (connection overhead), and the application must handle re-authentication when secrets expire (reconnection logic, error handling). For high-throughput services (10,000+ requests per second), this overhead can be significant. Mitigation: use a caching layer that renews secrets before they expire (Vault Agent sidecar or a local secret cache that renews at 60-80% of the TTL). This reduces the Vault call rate from per-request to per-instance-per-TTL.

### Challenge F3: Lease Management

**Prompt**: Your Kubernetes cluster uses Vault for dynamic database credentials. Each pod requests a new credential on startup. After a rolling update of 50 pods, you notice that Vault's storage backend has grown by 2 GB and the number of leases in the database engine has increased by 500, even though only 50 pods are running. Explain what leases are, why the lease count is 10x the pod count, and how to fix the storage leak.

**Evaluation criteria**: Did you explain that a lease is Vault's tracking record for every issued secret, which includes metadata (creation time, TTL, renewals, revocation status)? Did you identify that pods are not revoking their leases on shutdown (which is why the count is 10x -- each previous deployment left orphaned leases)? Did you propose a solution (lease revocation on pod shutdown, shorter TTLs, or Vault's lease cleanup mechanism)?

**Model answer**: A lease in Vault is a metadata record that tracks every credential the system has issued. Each lease contains the credential's creation time, TTL, renewal count, revocation status, and the identity of the requesting entity. Each lease consumes storage in Vault's backend (typically 1-5 KB per lease for the metadata plus indexing overhead). The lease count is 10x the pod count because each rolling deployment creates new pods while old pods are terminated without revoking their leases. Over 10 deployments, 500 orphaned leases accumulate in the storage backend. This is a classic "credential leak" pattern that has no immediate security impact (the credentials expire when their TTL elapses) but causes a slow-burn storage exhaustion problem.

The fix is threefold:

(1) **preStop lifecycle hook**: Register a `preStop` hook in the Kubernetes pod spec that calls Vault's `sys/leases/revoke` API before the pod terminates. This ensures the lease is cleaned up immediately when the pod shuts down gracefully. The hook should have a reasonable timeout (e.g., 30 seconds) and should block the pod's termination until revocation completes or the timeout fires.

```yaml
lifecycle:
  preStop:
    exec:
      command:
        - vault
        - lease
        - revoke
        - "$(cat /vault/lease-id)"
  terminationGracePeriodSeconds: 45
```

(2) **TTL enforcement**: Set a reasonable TTL (e.g., 4 hours) and a max TTL (e.g., 24 hours) on the database role in Vault. Even if the `preStop` hook fails or is not invoked (e.g., `kubectl delete pod --force`, node failure, OOM kill), the lease expires automatically. Vault's background garbage collection process reclaims the storage for expired leases. The TTL should be long enough that pods do not need to renew credentials during normal operation but short enough that leaked credentials expire quickly.

(3) **Grace period configuration**: Configure Vault's lease expiry grace period (`lease_expiry_grace_period = 30m` in Vault's server configuration). This controls how frequently Vault scans for and purges expired leases. A shorter grace period means faster storage reclamation but more CPU overhead from the scan. For a large cluster (500+ leases), set this to 30 minutes to balance storage efficiency with CPU cost.

Without these measures, the storage backend grows unbounded, increasing Vault's read and write latency as the index size grows and eventually exhausting disk space. A production alert should fire when the lease count exceeds 2x the expected maximum or when storage usage exceeds 80% of the allocated backend capacity. Additionally, a periodic audit script should verify lease counts are within expected bounds and flag any unexpected accumulation.

To understand why leases accumulate even with short TTLs, consider the scenario where a pod crashes (OOM, segmentation fault, or liveness probe failure). The Kubernetes API server immediately terminates the pod without running the `preStop` hook. The lease remains active until its TTL expires. In a cluster with frequent crash-loops, this can generate hundreds of orphaned leases per hour. A Vault role with `max_ttl = 4h` ensures these orphaned leases expire within 4 hours. Without a `max_ttl` cap, pods configured with long TTLs (e.g., 24 hours) would leak credentials that remain valid for an entire day. Always set both `default_ttl` and `max_ttl` on every Vault role, and ensure the `max_ttl` is no longer than necessary for your workload.

### Challenge D1: Microservices Secrets Management

**Prompt**: Design a secrets management system for a microservices application with 40 services across 5 Kubernetes clusters (production, staging, dev, eu-prod, us-prod). Requirements: (1) each service must have a unique database credential per environment, (2) credentials must rotate automatically every 24 hours, (3) a compromised service should not be able to access another service's secrets, (4) secret access must be auditable, (5) zero-downtime deployments.

**Rubric**:

| Criterion | Pass (2 pts) | Good (4 pts) | Excellent (6 pts) |
|---|---|---|---|
| Authentication | Static token per cluster | Kubernetes auth method with service account | Kubernetes auth + workload identity federation + SPIFFE |
| Authorization | Global policy | Per-service policy with path prefix | Namespace-scoped policies with tenant isolation + ACL audit |
| Secret delivery | Sidecar injection at startup | Sidecar with ongoing renewal | Vault Agent sidecar with template rendering + auto-renewal |
| Rotation | Manual rotation script | Scheduled rotation via cron | Dynamic secrets with TTL + automatic re-issuance before expiry |
| Disaster recovery | Single Vault instance | Vault HA with Raft backend | Active-active Vault clusters with cross-region replication |

**Model answer outline**:

1. **Architecture**: Deploy a dedicated Vault cluster (3 nodes, Raft integrated storage) in each Kubernetes cluster for write locality and reduced latency. Use Vault's Kubernetes auth method in every cluster: each service account maps to a Vault policy with minimum necessary permissions. The production Vault cluster (us-prod) acts as the primary for secrets data. The other clusters (eu-prod, staging, dev) replicate from the primary using Vault's Performance Replication for read availability while maintaining a single source of truth for writes. This design ensures that the production cluster remains fully operational even if the primary cluster experiences a regional outage.

2. **Authentication**: Each pod runs with a dedicated Kubernetes service account (e.g., `service-account: payment-service-prod` in the `prod` namespace). Vault's Kubernetes auth method works by verifying the pod's JWT token against the Kubernetes TokenReview API. The pod authenticates as `vault-role: payment-service-prod`. No static API tokens or shared secrets ever need to be distributed to pods. The auth method also supports bound service account names -- only pods running under the exact `payment-service-prod` service account in the `prod` namespace can authenticate as this role. This eliminates a class of attacks where a compromised pod could impersonate another service.

3. **Authorization**: Vault policies are scoped to the most granular paths possible. Each service gets a policy that grants `read` and `list` capabilities only on its own paths:

   ```hcl
   path "database/creds/payment-service-prod" {
     capabilities = ["read"]
   }
   path "transit/encrypt/payment-service-prod" {
     capabilities = ["create", "update"]
   }
   path "transit/decrypt/payment-service-prod" {
     capabilities = ["create", "update"]
   }
   path "kv/data/shared/certificates" {
     capabilities = ["read", "list"]
   }
   ```

   Deny-all is the default stance. A compromised `payment-service` pod cannot read `inventory-service` secrets, cannot write to `kv/data/shared`, and cannot generate credentials for any database other than its own. ACL audit reports are generated weekly to detect and flag any policy that has drifted toward granting broader permissions than intended.

4. **Secret delivery**: Use the Vault Agent sidecar injection pattern (enabled via the Vault mutating webhook). On pod creation, the webhook injects a Vault Agent container alongside the application container. The Agent authenticates, retrieves the secret, and writes it to a shared `emptyDir` volume as a plain text file or as a template-rendered configuration file. The application container reads from this file. The Agent renews the lease before TTL expiry (at 50% TTL by default) and rewrites the file on credential change. The application's file watcher (or `inotify` listener) detects the file change and triggers a connection pool refresh without restarting the pod. This achieves zero-downtime credential rotation entirely through configuration -- no application code changes required.

5. **Rotation**: Use dynamic secrets via Vault's database engine for all database credentials. Configure each database role with `default_ttl = 24h` and `max_ttl = 48h`. Vault Agent renews the lease at 12 hours (50% of TTL). If a pod restarts, it receives a fresh credential -- there is no need to track which credential was previously assigned. No manual rotation scripts, no cron jobs, no ceremony. For services that cannot use dynamic secrets (e.g., legacy applications with hardcoded connection strings), use Vault's KV v2 engine with a rotation schedule that pushes new secrets and triggers a rolling deployment.

6. **Audit**: Enable Vault's file audit device and stream logs to Splunk (or ELK) via Fluentd. Every secret read logs: service name, Kubernetes namespace, Vault role, timestamp, secret path, operation type, and client IP. A log analysis dashboard alerts on anomalous access patterns -- for example, a frontend service (which should never need database credentials) reading `database/creds/payment-service-prod` triggers a critical-severity alert. Audit logs are retained for 1 year for compliance and 30 days in hot storage for operational debugging.

7. **Disaster recovery**: Performance Replication replicates secret data from the primary Vault cluster (us-prod) to the secondary clusters (eu-prod, staging, dev). If the primary cluster is lost (catastrophic failure of the us-prod Kubernetes cluster), promote the eu-prod secondary to primary. Applications reconnect using the same Vault address (DNS CNAME failover). The RPO (Recovery Point Objective) is under 5 minutes -- any secrets created or updated in the 5 minutes before failure may be lost. The RTO (Recovery Time Objective) is under 2 minutes -- the time to update DNS and promote the secondary. Test this failover process quarterly in a game-day exercise.

### Challenge D2: Database Secrets Rotation

**Prompt**: Design a system to rotate the root and application credentials for a PostgreSQL database serving a payment processing application. The database is self-managed on EC2. Requirements: (1) zero downtime during rotation, (2) rotation must complete within 10 minutes, (3) no application code changes, (4) if rotation fails, the system must roll back to the working credential, (5) all rotation events must be logged for compliance.

**Rubric**:

| Criterion | Pass (2 pts) | Good (4 pts) | Excellent (6 pts) |
|---|---|---|---|
| Credential model | Single password | Dual-credential window | Dual-credential window + connection poisoning prevention |
| Rotation trigger | Cron job | Vault scheduled rotation | Vault rotation with pre-rotation validation + post-rotation verification |
| Failure handling | Log error and stop | Automatic rollback to old credential | Rollback + alert + health check before retry |
| App impact | Brief restart required | Connection pool drain with no restart | File-watch triggered reload + gradual connection migration |
| Audit | App logs | Vault audit log + DB audit log | Structured audit trail + compliance report + alert on failure |

**Model answer outline**:

1. **Dual-credential window**: PostgreSQL supports a dual-password model through the `password` and `password_old` parameters (available since PostgreSQL 16) or by creating two roles with identical privileges and rotating which role the application uses. The recommended rotation process using `password_old`:
   - **Phase 1 (t=0)**: Set a new password for the application role while keeping the old password valid.
     ```sql
     ALTER ROLE app_user PASSWORD 'new-password-generated-by-vault';
     ALTER ROLE app_user PASSWORD_OLD 'old-password';
     ```
     Both passwords now work for authentication. Existing connections using the old password continue to function. New connections can use either password.
   - **Phase 2 (t=0 to t=5min)**: Deploy the new password to all application instances. Since both passwords work, this deployment is non-disruptive. Old connections survive until the pod or process restarts naturally. The deployment can be rolling without any special coordination.
   - **Phase 3 (t=5min)**: After confirming that all application instances have received the new password, remove the old password.
     ```sql
     ALTER ROLE app_user PASSWORD_OLD NULL;
     ```
     Any connection still using the old password will fail authentication on its next reconnect attempt. If your monitoring shows a spike in authentication errors during this phase, it indicates that some instances did not receive the new password -- abort and investigate.

2. **Connection poisoning prevention**: The most dangerous failure mode in this process is "connection poisoning" -- a scenario where a connection pool silently caches a failing credential. For example, if the new password contains a typo or encoding issue, the pool might mark the new credential as "unreachable" and continue using the old credential. When the old password is removed in Phase 3, all connections fail simultaneously. Mitigation: the rotation script sends a test authentication with the new password at every phase boundary using a separate disposable connection:

   ```python
   def test_credential(db_host, db_port, db_name, username, password):
       try:
           conn = psycopg2.connect(
               host=db_host, port=db_port, dbname=db_name,
               user=username, password=password, connect_timeout=5
           )
           conn.close()
           return True
       except Exception as e:
           logger.error(f"Credential test failed: {e}")
           return False
   ```

   If the test fails at any phase, the rotation is immediately aborted, an alert fires to the SRE team, and the old password remains valid. The compliance report logs: `"Rotation attempt: FAILED at phase 2 -- new password rejected by database. Rollback initiated."`

3. **Root credential rotation**: The PostgreSQL superuser (`postgres`) password is stored in Vault and rotated every 30 days. The rotation process is simpler because the root credential is only used by Vault's database engine and by DBAs during emergency maintenance -- it is never read by application code. Vault connects to PostgreSQL using the current root password, runs `ALTER USER postgres PASSWORD 'new-password'`, stores the new password in its encrypted storage, and verifies authentication with the new password. If verification fails, Vault reverts to the stored previous password automatically. A DBA is alerted if two consecutive root rotation attempts fail. Root credentials must never appear in application configuration, CI/CD variables, or documentation.

4. **Application credential rotation (no code change)**: Applications read database credentials from a file that is mounted into the container by the Vault Agent sidecar. The sidecar watches the secret path in Vault and rewrites the file when the credential changes. The application's database driver detects the file change -- drivers that support `pgpass` file reloading (like `pgx` with `PgxPoolConfig`) handle this natively. For drivers without file-watch support, a lightweight wrapper process sends `SIGHUP` to the application process to trigger a configuration reload. This approach requires zero application code changes -- only deployment configuration changes to add the Vault Agent sidecar and mount the shared volume. The connection pool itself handles the transition: old connections are retired naturally, and new connections use the updated file content.

5. **Rollback procedure**: If Phase 3 (old password removal) fails or if monitoring detects authentication errors after rotation, the rollback procedure is:
   ```sql
   -- Restore old password as primary
   ALTER ROLE app_user PASSWORD 'old-password';
   -- Clear the password_old that was set
   ALTER ROLE app_user PASSWORD_OLD NULL;
   ```
   Then redeploy the application configuration with the old password. The compliance report must capture: the rotation attempt ID, the failure reason (e.g., "authentication error rate exceeded threshold of 0.1%"), the rollback decision timestamp, and the operator who approved the rollback (if manual). Automated rollback should complete within 30 seconds to minimize the impact window.

6. **Audit trail and compliance**: Every rotation step is logged to a structured audit record:

   ```json
   {
     "rotation_id": "rot-20260701-abc123",
     "target": "postgresql://payment-db.internal:5432/payments",
     "role": "app_user",
     "phase": 1,
     "action": "SET_NEW_PASSWORD",
     "timestamp": "2026-07-01T03:00:00Z",
     "operator": "vault-system",
     "result": "SUCCESS",
     "duration_ms": 234
   }
   ```

   Logs are sent to a SIEM (Splunk or ELK) via Vault's audit log socket device and retained for 7 years to meet PCI-DSS retention requirements. A monthly compliance report is auto-generated, summarizing: total rotations, failed rotations (with failure reasons), rollbacks, average rotation time, and any manual interventions. The report is distributed to the security team and the external auditor.

### Challenge D3: CI/CD Secrets Pipeline

**Prompt**: Design a system for managing secrets in a CI/CD pipeline. The pipeline builds and deploys 20 microservices using GitHub Actions. Requirements: (1) secrets must never appear in build logs, (2) each service's secrets must be scoped to its deployment, (3) a developer should not be able to read production secrets through a pull request, (4) the pipeline must inject secrets at deploy time, not build time, (5) secrets must be rotated quarterly with zero pipeline downtime.

**Rubric**:

| Criterion | Pass (2 pts) | Good (4 pts) | Excellent (6 pts) |
|---|---|---|---|
| Secret storage | GitHub Actions secrets | Vault with GitHub Actions OIDC | Vault with OIDC + dynamic secrets + per-runner lease |
| Secret injection | Environment variable at build time | Environment variable at deploy time | Vault Agent sidecar in deployment pod + OIDC token exchange |
| Leak prevention | Log scrubbing | Log scrubbing + secret detection in PR | Secret scanning + PR comment auto-block + CI gate on credential exposure |
| Rotation | Manual update in GitHub UI | Vault rotation + GitHub sync | Vault rotation + dynamic secrets + zero-downtime deploy with new creds |
| Audit | GitHub audit log | GitHub audit log + Vault audit log | Centralized audit trail + deployment attestation + compliance report |

**Model answer outline**:

1. **Architecture**: GitHub Actions workflows authenticate to Vault using OpenID Connect (OIDC). When a workflow job starts, GitHub issues a short-lived OIDC token (a JWT signed by GitHub's OIDC provider at `token.actions.githubusercontent.com`). This JWT contains claims identifying the repository, environment, branch, and workflow name. Vault validates the JWT's signature against GitHub's JWKS endpoint, verifies the claims against a configured bound claim set, and issues a Vault token with narrowly scoped policies. The Vault token has a TTL of 10 minutes (the expected duration of a single deploy job) and is automatically revoked when the job completes. No long-lived GitHub Actions secrets, no static API tokens, no shared passwords. The entire authentication chain is cryptographically verified and ephemeral.

   The GitHub Actions workflow configuration for OIDC authentication:

   ```yaml
   jobs:
     deploy:
       runs-on: ubuntu-latest
       permissions:
         id-token: write   # Required for OIDC token generation
         contents: read
       steps:
         - name: Authenticate to Vault
           uses: hashicorp/vault-action@v3
           with:
             url: https://vault.internal.example.com
             method: jwt
             role: ci-deployer
             jwt_path: ${{ env.ACTIONS_ID_TOKEN_REQUEST_TOKEN }}
             secrets: |
               deploy/payment-service/config | DEPLOY_CONFIG ;
               deploy/payment-service/version | DEPLOY_VERSION
         - name: Deploy to Kubernetes
           run: |
             echo "Deploying version $DEPLOY_VERSION..."
             helm upgrade payment-service ./charts/payment-service \
               --set image.tag=$DEPLOY_VERSION
   ```

   The `permissions.id-token: write` block is critical -- without it, GitHub does not issue the OIDC token, and the job cannot authenticate to Vault. The `vault-action` step exchanges the OIDC token for a Vault token and reads only the secrets needed for deployment (the deploy config and version, not the actual database credentials).

2. **Vault OIDC configuration**: The Vault role for CI/CD access is configured with strict binding:

   ```hcl
   path "deploy/*" {
     capabilities = ["create", "read", "update"]
   }
   ```

   The bound claims ensure that only the specific repository and environment can authenticate:

   ```hcl
   bound_claims = {
     "repository": "myorg/myapp",
     "environment": "production",
     "ref": "refs/heads/main"
   }
   ```

   A pull request from a fork -- even from the same repository -- would have a different `ref` claim and would be denied. The `environment` claim check enforces that only deployments to production (triggered by a GitHub Deployment event) can access production secrets. Dev or staging environments get their own Vault roles with less restrictive policies.

3. **Secret injection at deploy time (not build time)**: The pipeline is strictly divided into build and deploy phases. The build phase compiles code and produces a container image -- no secrets are needed, and none are available. The image is pushed to a container registry. The deploy phase (triggered by a GitHub Deployment event, executed by ArgoCD or a Helm operator in the target cluster) injects a Vault Agent sidecar into the newly created Kubernetes deployment pod. The sidecar authenticates to Vault using the pod's Kubernetes service account (Kubernetes auth method, not OIDC) and retrieves the secrets. The application never receives secrets through environment variables from the CI pipeline -- only from the sidecar's shared volume at runtime. This is the critical security property: no secret ever passes through GitHub's logging infrastructure, the CI runner's memory, or the build artifact.

4. **Scoped access**: Each service's Vault policy is defined as Infrastructure as Code in a `policies/` directory checked into the infrastructure repository. The CI pipeline's own Vault role is scoped exclusively to `deploy/*` paths -- the pipeline can trigger a deployment, but it cannot read the actual secrets. The deployment process (ArgoCD or Helm) creates pods with Vault Agent sidecars that authenticate using their own Kubernetes service accounts, which have policies scoped to that specific service's secrets:

   ```hcl
   # Vault policy for payment-service
   path "database/creds/payment-service" {
     capabilities = ["read"]
   }
   path "kv/data/payment-service/*" {
     capabilities = ["read", "list"]
   }
   ```

   This two-layer scoping means the CI pipeline cannot exfiltrate secrets even if compromised -- it can only trigger a deploy, and the deploy process can only read secrets for the specific service being deployed.

5. **Leak prevention**: Four layers of defense prevent secret leakage:
   - **Pre-commit**: A pre-commit hook (using `trufflehog` or `git-secrets`) scans every commit for potential secrets. If matched, the commit is blocked.
   - **PR scan**: A GitHub Actions workflow triggered on PR creation runs `trufflehog` against the diff. If a secret pattern is detected, the workflow fails, and a bot comments on the PR: "Potential secret detected in `src/config.py` at line 42. Please revoke this credential and use Vault instead."
   - **Build log scrubber**: All CI output is routed through a log scrubber (custom Fluentd filter or GitHub's built-in secret scanning API) that detects secret patterns (AWS keys, JWT tokens, etc.) in real time. If detected, the pipeline is immediately failed, and the log line is redacted.
   - **Post-deployment scan**: A periodic scan checks public and private GitHub repositories for leaked credentials that match known Vault secret paths. If a match is found, the secret is automatically rotated and the security team is alerted.

6. **Rotation**: Quarterly rotation is fully automated. For services using dynamic secrets (database credentials, service mesh mTLS), rotation is continuous and invisible -- no quarterly operation is needed. For services using static secrets in Vault's KV v2 engine, a scheduled job (Vault cron or a Kubernetes CronJob) runs every quarter:
   - Generate a new secret value using `vault write -f kv/data/payment-service/api-key`
   - The KV v2 engine automatically tracks the new version and keeps the old version
   - Trigger a rolling Kubernetes deployment of all pods that use this secret
   - Vault Agent sidecars detect the new secret version and update the file on disk
   - The deployment uses the dual-credential pattern (or blue-green deploy) to ensure zero downtime
   - A verification job confirms all pods are using the new secret version
   - Old KV versions are pruned after 30 days (to allow rollback if needed)

7. **Audit trail**: Every Vault access from CI/CD is logged with structured metadata: the GitHub Actions run ID, workflow name, repository, branch, the requesting service, the secret path, the operation type, and the timestamp. A monthly compliance report is auto-generated by querying Vault's audit log and GitHub's audit log API:

   ```sql
   -- Pseudocode for the compliance query
   SELECT run_id, workflow, secret_path, operation, timestamp
   FROM vault_audit_log
   WHERE auth_method = 'oidc'
     AND timestamp > date_trunc('month', now()) - interval '1 month'
     AND operation = 'read'
     AND secret_path NOT LIKE 'deploy/%'
   ORDER BY timestamp
   ```

   This query catches any CI run that read a secret outside the expected `deploy/` prefix. If any such read is found, it is flagged for investigation. The compliance report certifies: (a) no secrets were read at build time, (b) all secrets are rotated on schedule, (c) no developer has accessed production secrets through CI.

### Challenge E1: Multi-Cloud Secrets Management

**Prompt**: Design a secrets management system for a multi-cloud deployment spanning AWS, GCP, and on-premises bare metal. The deployment has 200 services across 10 environments. Requirements: (1) a single control plane manages all secrets, (2) services in each cloud can authenticate without hardcoded cloud credentials, (3) secrets are replicated across clouds but must be revocable globally in under 30 seconds, (4) each cloud has its own encryption key for secrets at rest, (5) the system must survive a full cloud provider outage.

**Key design tensions**:

- A single Vault cluster is a single point of failure, but a Vault cluster per cloud introduces fragmentation. How do you balance consistency (single source of truth) with availability (survive a cloud outage)?
- Cross-cloud replication latency can be 50-200ms. If a secret is revoked in AWS, how quickly does GCP learn about the revocation? What happens if GCP's replication link is down? Does the revocation wait for the link, or does GCP eventually learn about it through a separate channel?
- Each cloud has its own KMS for encryption at rest. Should Vault use cloud KMS for auto-unseal, or should it use a separate HSM that spans clouds? Cloud KMS is simpler to operate by leveraging each cloud's IAM and audit trails, but it adds a dependency on the cloud provider's availability during the unseal process. A standalone HSM (like Thales Luna) is portable across clouds but requires specialized operational knowledge and dedicated network connectivity.
- On-premises bare metal cannot use cloud KMS. How do you manage the same encryption key hierarchy across clouds and on-prem? If you use a single HSM accessible from all locations, the HSM becomes a single point of failure and a network latency bottleneck. If you use separate HSMs per location, you need a cross-location key synchronization mechanism.
- The "single control plane" requirement conflicts with "survive a cloud provider outage." If the control plane lives in AWS and AWS goes down, secrets cannot be managed. You could designate a secondary control plane in GCP, but then you have a split-brain risk if both think they are the primary.

**Discussion points**:

- Can you use a global Vault cluster with Performance Replication (active-active, multi-region) where each cloud region runs a Vault cluster that replicates data from a primary? How does this handle a primary cloud outage? Performance Replication is read-only on secondaries -- if the primary goes down, secondaries continue serving reads but cannot create new secrets or rotate credentials. Is this acceptable for your services? For many applications, read-only mode is tolerable for minutes to hours, but a payment processing system that needs new database credentials every 24 hours would fail after the TTL of existing credentials expires.
- Is a federation model better -- each cloud has its own Vault cluster, and a global policy engine (e.g., Crossplane, a custom controller, or a centralized Terraform pipeline) pushes secrets to each cluster as needed? This avoids the single point of failure entirely -- each cloud can operate independently. The cost is secret drift: a secret updated in AWS may take minutes or hours to propagate to GCP and on-prem. For short-lived dynamic secrets, this drift is unacceptable because a credential valid in AWS would be rejected by GCP until the sync completes.
- How do you handle the on-premises environment that cannot use cloud KMS for auto-unseal? Three approaches: (1) a hardware security module (HSM) deployed on-prem that is accessible from both cloud and on-prem through a VPN or Direct Connect link -- this works but the HSM becomes a network-accessible key server with all the security implications; (2) Shamir's secret sharing approach where the unseal key is split into shares distributed across clouds and on-prem operators -- for example, 5 shares with a threshold of 3, where AWS holds one share, GCP holds one, on-prem holds one, and two operators hold the remaining shares. This eliminates the HSM dependency but requires manual intervention for unseal (which may be unacceptable for automated scaling); (3) the cloud clusters use their native KMS, and the on-prem cluster uses a local HSM, with each cluster managing its own unseal independently. This is the most practical approach and works with Vault Performance Replication since the replication stream is encrypted independently of the cluster's unseal key.
- Revocation: if you use short-lived dynamic secrets (TTL 5 minutes), revocation is automatically handled by TTL expiry. The "revoke in under 30 seconds" requirement is met implicitly because the maximum exposure of a compromised dynamic secret is 5 minutes. However, some secrets cannot be dynamic -- API keys for third-party services, TLS certificates, SSH keys. For these, a webhook-based revocation approach is recommended: each cloud's Vault cluster subscribes to a revocation topic on a message bus that spans all clouds (AWS SQS + GCP Pub/Sub + on-prem RabbitMQ, bridged by a federation layer). When a secret is revoked, the primary publishes a revocation event to all three message buses. Each spoke cluster applies the revocation locally. If a spoke's message bus is unreachable, the revocation is queued and applied when connectivity is restored.

**Model answer** (one possible approach):

Use a hub-and-spoke Vault architecture with three Vault Enterprise clusters: one primary in AWS (hub), and two secondaries in GCP and on-prem (spokes). The primary cluster is the source of truth for all secret data. Performance Replication (Vault Enterprise feature) replicates secrets from the primary to the secondaries with near-real-time latency -- typically sub-second under normal conditions, up to 200ms during peak load. Each cluster has its own independent encryption key for at-rest encryption, managed by its cloud-native KMS or a local HSM:

- **AWS cluster**: uses AWS KMS (`aws/auto-unseal`) for auto-unseal. The KMS key is a Customer-Managed Key (CMK) in the AWS account, with automatic key rotation enabled. IAM policies control which services and users can access the KMS key.
- **GCP cluster**: uses GCP Cloud KMS for auto-unseal. The key is a Cloud HSM key in the GCP project (backed by a hardware security module within GCP's infrastructure). This provides FIPS 140-2 Level 3 compliance for the GCP region.
- **On-prem cluster**: uses a dedicated hardware security module (HSM) for auto-unseal (e.g., Thales Luna Network HSM or YubiHSM 2). The HSM is deployed in a redundant pair within the on-prem data center (active/passive). The on-prem team manages the HSM through its own key ceremony process.

For **authentication**, each cloud uses its native identity system:

- **AWS**: Vault's AWS auth method. EC2 instances present their instance identity document (signed by the AWS API) as proof of identity. Vault validates the document's signature against AWS's public certificate. Optionally, the auth can be bound to a specific AWS account ID, VPC ID, or IAM role. This means that only EC2 instances in the correct account and VPC can authenticate, and only those running under a specific IAM role.
- **GCP**: Vault's GCP auth method. GCE instances authenticate using their instance identity token (a JWT signed by Google's metadata server). Vault validates the JWT signature against Google's JWKS endpoint and verifies the instance's project, zone, and service account. No hardcoded credentials, no service account key files.
- **On-prem**: Vault's TLS certificate auth method combined with an internal CA (e.g., HashiCorp Vault PKI or a smallstep CA). Each bare-metal server receives a client certificate at provisioning time. The certificate is signed by the internal CA and is bound to the server's hostname and IP address. Vault validates the TLS connection against the CA certificate.

For **global revocation**: The primary cluster is the single revocation point. When a secret is revoked, the primary updates the replication stream. The secondaries receive the revocation within the replication latency (typically under 1 second). For secrets that must be revoked in under 30 seconds even if the replication link is down -- for example, a compromised API key for a payment gateway -- a fallback mechanism is deployed: each spoke cluster runs a local webhook that subscribes to a revocation topic on a message bus native to that cloud (AWS SQS for AWS, GCP Pub/Sub for GCP, RabbitMQ for on-prem). The primary publishes revocation events to all three message buses in parallel with the replication update. If the replication link is down, the webhook processes the revocation event directly from the message bus. The 30-second target accounts for: primary detects compromise (0-5s), publishes to message bus (1-2s), message propagates across cloud boundaries via a federation bridge (5-15s), spoke webhook processes revocation (1-2s), spoke Vault updates local leases (1-2s). Total: 8-26 seconds.

For **surviving a cloud outage**: If the primary AWS cluster goes down (catastrophic failure of the AWS region), the GCP and on-prem clusters continue serving existing secrets -- they are read replicas with full cached data. Existing application workloads continue to authenticate and read secrets without disruption. However, new secrets cannot be created, existing secrets cannot be updated, and secret rotations cannot be initiated until the primary is restored. This is an acceptable trade-off for most workloads: the system operates in read-only mode during a primary outage, and existing credentials continue to function until their TTL expires. For services that require continuous secret creation (e.g., auto-scaling groups that create new instances every few minutes), the auto-scaler must be configured with a startup grace period that accounts for this read-only window -- or the instances must cache credentials locally. To improve availability, a secondary cluster (GCP) can be promoted to primary if the outage exceeds 30 minutes. The promotion process requires operator approval (to avoid split-brain scenarios) and involves: verifying that the AWS cluster is truly unrecoverable, updating DNS to point the global Vault address to the GCP cluster, and reconfiguring the replication topology. The promoted GCP cluster becomes the new primary, and the on-prem cluster replicates from it. A full recovery postmortem is conducted after the promotion.

### Challenge E2: Regulated Fintech with Audit Requirements

**Prompt**: Design a secrets management system for a fintech startup processing payments in 15 countries. The company must comply with PCI-DSS, SOC 2 Type II, and GDPR. Requirements: (1) all secrets must be encrypted at rest and in transit, (2) every secret access must be logged and retained for 7 years, (3) secrets must be rotated every 90 days minimum (PCI-DSS requirement 8.2.4), (4) production secrets must be accessible only through a break-glass mechanism with dual approval, (5) the system must support instant credential revocation in case of a security incident, (6) encryption keys must be stored in a hardware security module (HSM) with FIPS 140-2 Level 3 certification.

**Key design tensions**:

- PCI-DSS requirement 3.5.2 states that no one person should have access to all parts of any cryptographic key. How do you implement dual control for the HSM root key? Smart cards with split-knowledge (shards distributed to two key custodians) is the standard approach, but what happens when both custodians are on vacation or unreachable during a critical incident? A third custodian introduces a new trust risk. A time-locked recovery mechanism (e.g., after 72 hours, the system can be unsealed by a third-party escrow) is a possible solution but adds complexity.
- GDPR's right to erasure (Article 17, "right to be forgotten") directly conflicts with PCI-DSS requirement 10.7, which mandates 7-year retention of audit logs. If a customer requests deletion of their personal data, you cannot delete the audit log (PCI-DSS), but you cannot continue storing the customer's personal data (GDPR). The resolution is to remove the link between the customer's identity and the log data without deleting the log itself -- often done by replacing PII with a one-way salted hash and then discarding the salt.
- SOC 2 CC6.1 and CC6.3 require periodic review of logical access permissions. With 200 services, each having 3-5 secrets, and Vault policies granting path-based access, you might have 2000+ policy grants. How do you audit all of them quarterly and detect excessive permissions? Manual review is infeasible at this scale. Automated policy analysis (e.g., running `vault policy list` and comparing the union of all grants against a declared "least privilege" baseline) is necessary. A policy that grants access to more paths than the service's declared baseline should be flagged and auto-remediated.
- Break-glass access: the entire point of a secrets management system is that no human can read production secrets in the normal course of operations. But during a critical incident (database corrupted, application crashing, need to manually connect to a production database), authorized engineers need temporary access. How do you provide this access without permanently lowering security? A physical safe with a sealed envelope containing a one-time recovery code is the most common approach, but it requires two people to be physically present in the data center. For a company operating in 15 countries, this may be impractical during off-hours. A digital equivalent (multi-party computation with SMS-based second factor) may be more practical but introduces additional attack surface.

**Discussion points**:

- Should you use Vault Enterprise with HSM integration (PKCS#11) or a managed cloud HSM like AWS CloudHSM? Cloud HSM is simpler to operate because the cloud provider manages the hardware, firmware updates, and availability. However, it creates a dependency on the cloud provider's availability and pricing. Vault Enterprise with a standalone HSM is more portable across clouds and on-prem but requires specialized operational knowledge for HSM key ceremonies, firmware upgrades, and disaster recovery.
- For 7-year audit retention, Vault's audit log can be streamed to an immutable object store. AWS S3 with Object Lock in COMPLIANCE mode ensures that even the root AWS user cannot delete or overwrite log objects before the retention period expires. Azure Blob Storage with immutability policies provides equivalent functionality. The log format should be JSON Lines (one JSON object per line) for easy processing by SIEM tools. The total storage cost for 7 years of Vault audit logs at 200 services with moderate secret access is approximately 10-50 GB per year.
- For SOC 2 access review automation, use a policy-as-code approach: declare each service's required access in a YAML file, generate Vault policies from this file, and run a CI/CD gate that prevents policy deployment if the generated policy deviates from the declared baseline. A quarterly cron job compares the deployed policies against the baseline and generates a report of any discrepancies.
- Break-glass design: a four-eyes principle with multi-party computation. Each authorized break-glass responder has a hardware security key (YubiKey or smart card) enrolled with the system. To initiate break-glass access, two responders must authenticate using their keys and PINs. The system performs multi-party computation to decrypt a one-time recovery code that grants a highly-scoped, time-limited, audited Vault token. The token is restricted to specific secret paths (e.g., only production database credentials, not TLS private keys or encryption keys) and has a TTL of at most 1 hour with a maximum of 5 operations. Every operation performed with the token is logged and triggers an alert to the CISO and the security team.

**Model answer** (one possible approach):

1. **Architecture**: Vault Enterprise with HSM integration (Thales Luna Network HSM, FIPS 140-2 Level 3 certified). The HSM generates and stores the master encryption key within its tamper-proof hardware -- the key never leaves the HSM boundary in plaintext. Vault uses the HSM for `auto-unseal` and all encryption and decryption operations via the PKCS#11 interface. The Vault cluster is deployed in an active-active configuration across two AWS regions (us-east-1 and eu-west-1) for disaster recovery and geographic redundancy. Performance Replication ensures secret data is available in both regions with sub-second replication latency. Each region has its own HSM (or an HSM partition on a clustered HSM pair) so that a regional outage does not affect the other region's ability to operate.

2. **Encryption key hierarchy**: A four-layer hierarchy ensures that a compromise at any layer does not expose the layers above:
   - **Layer 1**: HSM root key (never leaves the HSM, generated on the HSM using its internal entropy source).
   - **Layer 2**: Vault master key (wrapped/unwrapped by the HSM root key. The master key is generated by Vault during initialization and stored encrypted. It is only unwrapped inside the HSM for brief operations).
   - **Layer 3**: Vault keyring (a set of encryption keys managed by Vault's `sys/key` endpoints. Each key in the keyring is encrypted by the master key. The keyring supports key rotation -- a new key is added, and data is re-encrypted lazily).
   - **Layer 4**: Individual secrets (each secret is encrypted by the current keyring key using AES-256-GCM. The encrypted ciphertext is stored in Vault's storage backend).

3. **Dual control for key management**: The HSM requires two key custodians to authorize any key ceremony operation (initialization, key rotation, disaster recovery). Each custodian has a smart card (PIV credential) with a unique PIN. During a key ceremony:
   - Both custodians present their smart cards to the HSM management workstation.
   - Each enters their PIN independently.
   - The HSM requires both smart cards to be present for the operation to proceed.
   - A video recording of the key ceremony is retained for audit and compliance purposes.
   - A third emergency custodian is designated (CISO or VP of Engineering) whose smart card is stored in a separate sealed envelope in a geographically different safe. This custodian is only used if both primary custodians are unavailable, and their use requires a documented exception approved by the Board.

4. **Audit logging for 7-year retention**: Vault's audit log is streamed to two destinations simultaneously:
   - **Hot destination**: Splunk (or ELK) for real-time alerting and operational queries. Retained for 90 days.
   - **Cold destination**: AWS S3 bucket with Object Lock in COMPLIANCE mode. Retention period: 7 years. The S3 bucket is in a separate AWS account (log archive account) with strict IAM policies that prevent any user or service from modifying or deleting objects. Logs are written in JSON Lines format, batched every 5 minutes, and compressed with gzip. Approximate storage: 50 GB per year.

   The audit log schema captures every secret access:

   ```json
   {
     "timestamp": "2026-07-01T12:34:56.789Z",
     "source": "vault-cluster-us-east-1",
     "operation": "read",
     "path": "database/creds/payment-service-prod",
     "client_ip": "10.0.1.42",
     "identity": {
       "type": "kubernetes",
       "service_account": "payment-service",
       "namespace": "prod",
       "cluster": "eks-us-east-1"
     },
     "response_code": 200,
     "lease_id": "s.abc123def456",
     "user_pseudonym": "sha256$salt_value$hash_output"
   }
   ```

5. **GDPR compliance via pseudonymization**: Audit logs never contain raw personally identifiable information. The user identifier (employee email, developer ID, or service account name) is replaced with a salted SHA-256 hash. The salt is generated per-user and stored in a separate AWS KMS key (not in the HSM) with a 24-hour cache. When a customer or employee requests data erasure (GDPR Article 17):
   - Delete the salt from KMS. The pseudonym becomes irreversible -- the hash cannot be reversed to recover the original identifier without the salt.
   - The audit log remains intact and readable for PCI-DSS compliance.
   - The data subject is notified that their data has been anonymized in the audit trail.
   - A compliance report certifies that erasure was performed without affecting the immutable audit log.

6. **Rotation compliance (PCI-DSS 8.2.4)**: Dynamic secrets for databases have a TTL of 24 hours and a max TTL of 48 hours -- this far exceeds the 90-day requirement. For static secrets (API keys, TLS certificates), Vault's rotation schedule forces rotation every 90 days via a `period` setting:

   ```hcl
   # Vault KV secret with enforced rotation
   path "kv/data/payment-service/api-key" {
     capabilities = ["read", "list"]
     rotation_period = "2160h"  # 90 days
   }
   ```

   A compliance dashboard (built with Grafana) shows the last rotation date for every secret in the system. Secrets that are overdue for rotation by more than 7 days trigger a P1 alert to the security team. A weekly compliance report is emailed to the CISO listing: all secrets, their last rotation dates, and any overdue items. The external auditor receives a quarterly summary.

7. **Break-glass access procedure**: The break-glass procedure is documented, tested quarterly in a tabletop exercise, and reviewed annually by the external auditor:
   - **Step 1**: An engineer identifies an incident requiring production secret access (e.g., database corruption requiring direct SQL intervention).
   - **Step 2**: The engineer files an incident report and requests break-glass access. Two authorized responders (on-call from a pre-approved list) are paged.
   - **Step 3**: Both responders authenticate to the break-glass portal (a hardened web application) using their YubiKey hardware security keys and individual PINs.
   - **Step 4**: The portal performs multi-party computation to decrypt a recovery code from the HSM. The recovery code is valid only for 1 hour and only for the specific secret paths requested (e.g., `database/creds/payment-service-prod`).
   - **Step 5**: The portal generates a time-limited Vault token with those constraints and presents it to the requesting engineer.
   - **Step 6**: Every operation performed with the break-glass token is logged. A real-time alert is sent to the CISO and the security team. The alert includes: who requested access, who approved, which secrets were accessed, and the timestamp.

8. **Instant revocation for security incidents**: The incident response team maintains a runbook for credential revocation:
   - **Tier 1 incident** (suspected credential exposure, confirmed within 1 hour): Revoke all leases for the compromised path via Vault API: `vault lease revoke -prefix database/creds/payment-service-prod`. The revocation is propagated to both Vault clusters via Performance Replication within 5 seconds. For database credentials, Vault's database engine additionally executes a `REVOKE ALL ON payments FROM current_user;` statement at the database level to immediately invalidate the credential server-side.
   - **Tier 2 incident** (confirmed credential exposure, immediate threat): Revoke all leases on the entire Vault cluster: `vault lease revoke -prefix database/creds/`. This invalidates ALL database credentials for ALL services. This is a last-resort action that causes a full application restart. It is only used when the scope of the compromise cannot be determined.
   - **Post-revocation**: A compliance dashboard confirms that no active leases remain for the revoked credential. A report is generated documenting: the incident ID, the revoked path, the number of leases revoked, the time to full revocation, and any application impact observed.

For additional depth on the fintech system, consider how the system handles the HSM key ceremony and break-glass access in practice. The HSM generates the root key using a FIPS 140-2 Level 3 certified random number generator. The root key is split into 5 shards using Shamir's Secret Sharing algorithm with a threshold of 3. Three shards are distributed to designated key custodians on smart cards, one shard is stored in a sealed envelope in the corporate safe, and one shard is stored in a separate sealed envelope in a geographically diverse location (a bank safe deposit box). During a key ceremony, the custodians present their smart cards, and the HSM reconstructs the root key in memory for the duration of the operation. The root key is never stored in software or in a file on disk. The HSM itself is tamper-resistant: if an attacker attempts to physically open the HSM, the tamper circuits detect the intrusion and zeroize the key material within milliseconds.

The break-glass system is tested quarterly in a full walkthrough exercise. The exercise simulates a scenario where the Vault cluster is unsealed but a critical production secret (e.g., the payment processor API key) is unavailable due to a configuration error. The on-call engineer initiates the break-glass procedure, two responders authenticate, and the recovery code is used to generate a temporary token. The entire process is timed and documented. The average time from incident detection to credential access is under 5 minutes. The exercise report is reviewed by the security team, and any delays or procedural issues are addressed in the next iteration. The external auditor reviews the break-glass test results annually as part of the SOC 2 Type II audit.

The GDPR erasure process is also tested quarterly. A test subject is created in the system, the pseudonymization salt is generated, and the audit log records operations for this test subject. The erasure request is submitted, the salt is deleted, and the system verifies that the audit log records are no longer linkable to the test subject's identity. The test confirms that: (a) the audit log is still intact and readable, (b) the pseudonymized identifier cannot be reversed to the original identity, (c) the erasure is complete within 24 hours of the request (GDPR requirement). The test results are documented in the compliance report.
