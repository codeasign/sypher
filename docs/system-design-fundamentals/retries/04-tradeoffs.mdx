## The Core Tension

The core tension in retry system design is between reliability and stability. More retries improve the chance of eventual success. More retries also increase the risk of cascading failure, load amplification, and resource exhaustion.

Every retry configuration is a bet on the failure characteristics of the downstream system. If failures are short and sparse, aggressive retries with short delays are optimal. If failures are long and correlated, conservative retries with long delays are necessary. The wrong bet makes the system worse.

This tension manifests in every design decision: the number of retries, the delay between them, the budget for rate limiting, the interaction with circuit breakers, and the timeout for the overall operation.

## The CAP Perspective on Retries

Retries interact with the CAP theorem in a subtle but important way. During a network partition, retries amplify the problem. The client retries the request, which requires network communication. If the network is partitioned, the retry will also fail. The retry adds load without improving the outcome.

### Consistency vs. Availability in Retries

In a CP system (ZooKeeper, etcd), retries are dangerous. The system may sacrifice availability during a partition. Retries against a minority node will always fail because the node has stopped accepting requests. The retry system must detect this and stop retrying immediately.

In an AP system (Cassandra, DynamoDB), retries are more useful. The system remains available during a partition. Retries against a different replica may succeed. The retry system can route retries to replicas that are reachable.

### Partition Tolerance and Retries

Retries are not partition-tolerant. They assume that the network will eventually recover. If the partition is permanent, retries are wasted. The retry system must have a mechanism to detect partitions and suppress retries after a threshold.

The retry timeout is the partition tolerance mechanism. If the total retry time exceeds the timeout, the operation fails. The timeout must be shorter than the expected partition duration. If partitions are expected to last minutes, the retry timeout should be seconds.

## Failure Modes

### Failure Mode 1: Retry Storm

The retry storm is the most dangerous failure mode. A small failure triggers a large number of retries, which amplifies the load, which causes more failures, which triggers more retries. The system enters a death spiral.

The fix for retry storms is layered: jitter prevents retry waves, exponential backoff gives the server time to recover, retry budgets limit the total retry rate, and circuit breakers stop retries when the server is unhealthy.

### Failure Mode 2: Retry Cascade Across Services

A retry cascade occurs when multiple services retry each other. Service A calls Service B, which calls Service C. A failure in Service C causes B to retry, which causes A to retry B, which causes B to retry C again. The retries amplify each other.

The amplification is multiplicative. If each service has 3 retries, a single failure in Service C generates 3 retries from B to C, and 3 retries from A to B (each of which results in 3 retries from B to C). The total requests to C from a single original request are: 1 (original) + 3 (B retries C) + 3 (A retries B, each B retries C) = 1 + 3 + 9 = 13 requests.

The fix for retry cascades is to use different retry configurations at each layer. The innermost service (C) has the fewest retries. The outermost service (A) has the most retries. Circuit breakers at each layer prevent the cascade from propagating.

### Failure Mode 3: Budget Starvation

A retry budget that is too small can starve legitimate retries. If the budget is 5 tokens with a refill rate of 1 token per second, and 10 requests fail simultaneously, only 5 retries fire. The other 5 requests fail without retry.

Budget starvation is a safety mechanism by design, but it can cause user-visible failures. The budget must be sized to handle the expected burst of retries. The expected burst is: maxConcurrentClients * failureRate * maxRetries.

### Failure Mode 4: Timeout Stacking

Timeout stacking occurs when the transport timeout and the retry backoff are not coordinated. The transport timeout is set to 500ms. The retry backoff is 100ms, 200ms, 400ms. The total retry time is 700ms, which exceeds the transport timeout. The third retry is canceled by the transport before the server responds.

The fix is to coordinate the transport timeout with the retry configuration. The transport timeout should be at least: initialDelay * (multiplier^maxRetries - 1) / (multiplier - 1) + serverResponseTime. Alternatively, use a per-attempt timeout and an overall deadline.

### Failure Mode 5: Idempotency Key Collision

An idempotency key collision occurs when two different requests use the same idempotency key. The server sees the second request as a duplicate and returns the cached result of the first request. The second request is silently ignored.

The fix is to generate idempotency keys that are unique across all requests. A UUID is sufficient. The idempotency key should include a prefix that identifies the client and the operation type, making collisions astronomically unlikely.

## Anti-Patterns

### Anti-Pattern 1: Infinite Retry

Infinite retry is the most common anti-pattern. The developer sets maxRetries to a very large number or omits the max retries check entirely. The retry loop runs forever. The server is hammered continuously. The client never returns a failure to the caller.

The fix is to always set a finite maxRetries. The maximum recommended value is 5. Beyond 5 retries, the total wait time exceeds 30 seconds, and the amplification factor exceeds 2x for most failure rates.

### Anti-Pattern 2: Retrying Non-Idempotent Operations

Retrying a non-idempotent operation without idempotency keys causes duplicate data. A charge endpoint that is retried twice charges the customer twice. An insert endpoint that is retried twice creates duplicate rows.

The fix is to either make the operation idempotent (by adding idempotency keys) or not retry it. If the operation cannot be made idempotent, the client must accept the failure and surface it to the user.

### Anti-Pattern 3: No Jitter with Many Clients

A retry system with no jitter and many clients creates a thundering herd. All clients retry at exactly the same time. The server sees a coordinated wave of retries. The wave amplifies the load.

The fix is to always add jitter when the number of concurrent clients exceeds 5. Full jitter is the safest default. Equal jitter is appropriate when you need a minimum delay between retries.

### Anti-Pattern 4: Retrying Without Circuit Breaker

A retry system without a circuit breaker does not know when to stop retrying. The server is unhealthy. The circuit breaker is open. The retry system continues to retry, wasting resources and delaying the server's recovery.

The fix is to integrate the retry system with a circuit breaker. The retry interceptor checks the circuit breaker before each attempt. If the circuit is open, the retry is skipped.

### Anti-Pattern 5: Same Retry Configuration for All Operations

A single retry configuration applied to all operations ignores the different failure characteristics of each operation. A read operation that is fast and cheap should have fewer retries than a write operation that is expensive and idempotent.

The fix is to use per-operation retry configuration. The retry interceptor should accept a configuration map that maps operation names to retry policies. The default policy is conservative. Each operation can override the defaults.

## When NOT to Retry

There are specific scenarios where retries are harmful and should not be used.

### Real-Time Systems

In real-time systems with strict latency requirements, retries are typically not acceptable. An audio streaming service that loses a packet should not retry — the retry would arrive too late to be useful. The system should accept the loss and continue.

### User-Facing Operations with Strict SLAs

A user-facing API with a 200ms SLA should not retry 3 times with 100ms backoff. The retries alone would consume 700ms, exceeding the SLA by 3.5x. The system should fail fast and return the error to the user.

### Disaster Scenarios

During a disaster (region outage, data center fire), retries are wasted. The downstream system is not coming back within the retry window. The retry system should detect the catastrophic failure and fail fast, allowing the user to route traffic to a different region.

### Operations That Are Not Safe to Repeat

Any operation that produces side effects and is not idempotent should not be retried. Sending an email, triggering a push notification, or writing to a log are examples. Retrying these operations produces duplicates.

The optimal zone depends on the server's recovery time distribution. If the server recovers in 500ms 90% of the time, the retry configuration should have a total delay of at least 500ms before the last retry. If the total delay is 300ms, 90% of retries miss the recovery window. If the total delay is 2000ms, 90% of retries wait too long.

The optimal configuration is derived from the server's recovery time histogram. Measure the p50, p95, and p99 recovery times. Set the total retry delay to match the p95 recovery time. This catches 95% of recoverable failures while keeping the latency within acceptable bounds.
