# Understanding Object Storage

## The Story

Picture this: you are the first engineer at a fast-growing photo-sharing startup. Your users upload 50,000 pictures a day. You started storing them on the application server's local disk, organized in nested folders by user ID and date. It worked for the first thousand users. But now, with 200,000 users and 3 million photos, your single server is running out of disk space. Worse, a disk failure last week wiped out 12,000 vacation photos — and your users are furious.

You migrate to a network-attached storage (NAS) array. Now you have more space, but the hierarchical file system means listing all photos for a user takes 800 milliseconds because the OS has to traverse thousands of directory entries. Renaming a million files? That is a 45-minute operation. And when your second application server needs to read the same files, you discover that file locking in a POSIX filesystem is not designed for concurrent access from dozens of machines. Your team is burning weekend after weekend fighting storage bottlenecks instead of building features.

The breaking point arrives when your CTO asks you to store 4K video files that are 2 GB each. The file system's 4 GB per-file limit on your NAS is fine today, but next year's 8K footage will be 8 GB per clip. You need a storage system that handles trillions of objects, survives disk failures without losing data, works across hundreds of servers simultaneously, and has no practical file size limit. You need object storage.

## The Insight

Think of object storage as a giant automated warehouse, not a library. A library (file system) organizes books on shelves by category, then author, then title — a strict hierarchy. To find a book, you navigate the shelves. A warehouse (object storage) assigns every item a unique barcode, tosses it onto any shelf that has space, and records the barcode-to-location mapping in a computer. When you need an item, you give the computer the barcode, and a robot fetches it from wherever it is sitting. The warehouse does not care about the item's shape, size, or contents — it only cares about the barcode. This is exactly how object storage works: every file (object) gets a unique identifier (key), stored in a flat namespace (bucket), and the system figures out where to physically place it and how to retrieve it.

## Core Vocabulary

**Object** — The fundamental unit of storage. An object contains your data (the file contents), metadata (key-value pairs describing the data), and a globally unique identifier. Unlike a file in a traditional filesystem, an object is immutable — you cannot edit part of it; you must upload a new version.

**Bucket** — A flat container that holds objects. Think of a bucket as the top-level namespace. Buckets have their own access policies and configuration (region, versioning, encryption). You can have up to 100 buckets per account by default, and each bucket can hold an unlimited number of objects.

**Key** — The unique identifier for an object within a bucket. The combination of bucket name plus key forms the object's complete address. Despite looking like a file path (e.g., `photos/2024/vacation/beach.jpg`), this is just a string — there is no real directory hierarchy underneath.

**Flat Namespace** — The defining structural property of object storage. Unlike a filesystem tree where directories nest inside directories, object storage has only two levels: bucket and key. The "path" in a key is cosmetic; the system does not maintain directory metadata, rename parent directories, or enforce path constraints.

**PUT** — The HTTP method used to upload an object. A successful PUT returns an HTTP 200 response with the object's ETag (a hash of the content). You can also use multipart uploads for objects over 100 MB, splitting the file into parts and uploading them in parallel.

**GET** — The HTTP method used to retrieve an object. A successful GET returns the object body with headers containing metadata, content type, and content length. Range GETs let you download specific byte ranges — essential for resuming interrupted downloads.

**DELETE** — The HTTP method used to remove an object. Object deletion is immediate and permanent (unless versioning or soft-delete is configured). There is no "trash" by default.

**Durability** — The probability that an object will not be lost over time. AWS S3 standard storage offers 99.999999999% (eleven 9s) durability per object. This means if you store 10 billion objects, statistically you expect to lose one object every 10,000 years. Durability is achieved through automatic replication across multiple devices and facilities.

**Availability** — The probability that you can successfully read or write an object when requested. S3 standard offers 99.99% availability (about 53 minutes of downtime per year). Availability depends on the service being reachable and the data being accessible.

**Replication** — The mechanism that copies objects across multiple physical locations. Synchronous replication waits for all copies to be written before confirming success. Asynchronous replication (cross-region replication) copies data to another region after the primary write completes. Replication is how object storage achieves its durability.

**Multi-Part Upload** — A feature for uploading large objects (recommended over 100 MB). You split the file into parts (up to 10,000), upload each independently and in parallel, then send a complete request to assemble them. If a part fails, you re-upload only that part instead of restarting the entire upload. This dramatically improves throughput for large objects.

**ETag** — An HTTP response header containing a hash of the object content (typically MD5). The ETag changes when the object content changes, making it useful for cache validation and change detection. For multi-part uploads, the ETag is a hash of the concatenated part hashes.

**Pre-signed URL** — A temporary, permission-granting URL that lets someone else perform a specific operation (GET, PUT, DELETE) on an object without having AWS credentials. You generate the URL by signing it with your credentials and setting an expiration time. Pre-signed URLs are widely used for direct browser uploads and time-limited downloads.

**Versioning** — A feature that keeps multiple versions of the same object. When you overwrite an object with versioning enabled, the old version is preserved rather than replaced. This protects against accidental deletion and overwrites. You can list, retrieve, and restore previous versions individually.

For objects over a threshold (typically 5-10 MB), the system splits the incoming data stream into fixed-size chunks (typically 8-64 MB each). Each chunk is assigned a unique content hash (SHA-256). This chunking is transparent to you — you still see one object, but internally the system works with many pieces.

### Step 4: Metadata Registration

Before any data is written to disk, the system registers the object in the metadata store. This creates a record mapping the object key (`my-home-videos/wedding-highlights.mp4`) to the list of chunk hashes and their intended locations. The metadata includes the object's size, content type, creation timestamp, and user-defined tags.

### Step 5: Parallel Data Write to Storage Nodes

The system sends each chunk to multiple storage nodes in parallel. Modern object storage systems like S3 use a technique called "shard over enough independent devices": each chunk is written to N different disks (typically 3 for standard storage, 2 for reduced redundancy). The storage nodes are organized into placement groups, and the system ensures no two replicas of the same chunk are on the same physical host or rack.

### Step 6: Quorum Confirmation

The write operation does not complete until a configurable number of storage nodes acknowledge success. For a replication factor of 3, the system might require 2 out of 3 acknowledgments before considering the write successful (write quorum = 2). This balances durability against latency — waiting for all 3 would be slower but more durable.

### Step 7: Background Integrity Verification

After the write is confirmed, background processes verify the data's integrity. They compute checksums on stored data and compare them to the expected values. If a mismatch is found, the system automatically fetches a good replica and repairs the bad one. This is how object storage achieves eleven 9s durability — continuous self-healing.

### Step 8: Object Retrieval

When you request the object, the system looks up the key in the metadata store, retrieves the list of chunk hashes and their locations, fetches the chunks in parallel from the storage nodes, assembles them in order, and streams the reconstructed object to you. If a storage node is unavailable, the system fetches a different replica transparently.

## Variants and Flavors

### Variant 1: Standard Object Storage (S3 Standard)

This is the default variant. Data is replicated synchronously across three Availability Zones within a region. Designed for frequently accessed data. Offers eleven 9s durability and 99.99% availability.

### Variant 2: Infrequent Access (S3 Standard-IA)

Same durability and replication as standard, but with lower storage cost and higher retrieval fees. Designed for data accessed less than once per quarter. The system stores the data identically but uses cheaper storage media (slower HDDs instead of SSDs) and charges a retrieval fee per GB.

### Variant 3: One-Zone IA (S3 One Zone-IA)

Data is replicated within a single Availability Zone, not across three. This reduces cost by roughly 20% but sacrifices durability if the zone is destroyed. Appropriate for secondary backups or easily reproducible data. The diagram is identical to standard but with only one AZ and a note that durability drops to 99.99%.

## Mental Model Check

**Q1: What is the fundamental difference between a filesystem and an object store?**
A: A filesystem organizes data in a hierarchy of nested directories and files, with each file occupying a specific path. Operations like "rename this directory" require updating every child path. An object store has a flat namespace with only two levels: bucket and key. The key is just a string — the system does not interpret slashes as directory separators. This flatness enables horizontal scaling to trillions of objects.

**Q2: Why is object storage more durable than a RAID array on a single server?**
A: RAID protects against disk failure within one server, but not against server failure, power loss, network outage, or site destruction. Object storage replicates each object across multiple independent servers, racks, and often multiple physical facilities (Availability Zones). Background scrubbers continuously verify data integrity. If one copy is corrupted, the system repairs it from a good copy. If a facility is destroyed, copies in other facilities survive.

**Q3: You have a 10 GB database backup file. Should you use PUT or Multi-Part Upload?**
A: Use Multi-Part Upload. A single PUT requires the entire file to be uploaded in one stream. If the connection drops after 9.5 GB, you must restart from scratch. Multi-Part Upload splits the file into parts (recommend 100-200 MB each, so 10 GB = 50-100 parts). Each part can be uploaded independently and retried individually. If one part fails, you only re-upload that 200 MB, not the entire 10 GB.

**Q4: What happens when you "delete" an object with versioning enabled?**
A: With versioning disabled, the object is permanently deleted immediately and irreversibly. With versioning enabled, the system adds a delete marker (a special version with a null ID and a deletion timestamp) on top of the existing versions. The object still exists — it is simply hidden from list operations. You can restore the object by removing the delete marker. Previous versions remain accessible if you request them by version ID.

**Q5: How does object storage handle concurrent writes to the same key?**
A: The last write wins by default. Object storage does not support file locking or transactions. If two clients PUT to the same key simultaneously, the system processes both writes through the metadata store's consistency mechanism. The second write to complete overwrites the metadata pointer. The first writer's data chunks remain on storage nodes but are now unreferenced and will be garbage collected. This is acceptable for many use cases (logs, media, backups) but not for databases that need row-level concurrency control.
