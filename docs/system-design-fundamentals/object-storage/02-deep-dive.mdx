# Object Storage — Under the Hood

## Internal Architecture

The metadata plane and the data plane operate independently. The metadata plane handles object keys, permissions, and location mappings. The data plane stores and retrieves raw bytes. This separation is what allows each plane to scale independently — you can add storage nodes without touching the metadata servers.

### Component Responsibilities

**Request Router** — The load balancer and protocol adapter. It terminates HTTPS connections, validates AWS Signature V4 authentication, enforces rate limits (default 3,500 PUT/copy/post/delete requests per second per prefix), and forwards valid requests to the metadata server leader.

**Metadata Server (Leader)** — The single authority for all metadata operations in a partition. It serializes writes to the metadata store, assigns chunk locations, and handles bucket configuration changes. It uses a consensus protocol (Paxos or Raft) to elect a leader and maintain a write-ahead log (WAL) for crash recovery.

**Metadata Store (Database)** — A distributed key-value database that maps object keys to their metadata. This database must handle millions of reads and writes per second. Amazon S3 uses a custom database called ShardStore (based on Dynamo principles) internally. The metadata store does not store the object data — only the references (chunk hashes and locations).

**Location Manager** — An in-memory service that maintains the current mapping of chunk hashes to physical disk locations. It is updated whenever chunks are written, moved, or repaired. Because this data changes frequently and must be accessed at very low latency, it lives in memory backed by a persistent log.

**Placement Engine** — The algorithm that decides which disks receive which chunk replicas. It must satisfy constraints: no two replicas on the same device, no two in the same failure domain (rack/zone), balanced disk utilization across the cluster, and minimized cross-rack traffic.

**Scrubber Daemon** — A background process that reads each chunk on every disk, recomputes its checksum, and compares it to the stored checksum. It runs continuously with a low priority to avoid impacting production traffic. A typical scrubber verifies about 10,000 chunks per second per storage node.

**GC Daemon** — The garbage collector that deletes unreferenced chunks. When you DELETE an object, the metadata entry is removed, but the chunks remain on disk. The GC daemon scans for chunks not referenced by any metadata entry and securely erases them after a retention period.

**Repair Daemon** — Monitors the cluster for under-replicated chunks. When a disk fails, the repair daemon identifies all chunks that lost a replica and schedules rebuild operations to restore the replication factor. It prioritizes chunks with the fewest remaining replicas.

## Data Structures and Algorithms

### Consistent Hashing Ring

Object storage uses consistent hashing to distribute objects across storage nodes without requiring a full rebalance when nodes are added or removed. Each node is assigned multiple positions (virtual nodes) on a hash ring. The object key is hashed and the closest clockwise virtual node determines placement.

### Merkle Tree for Integrity Verification

Each storage node organizes chunk checksums into a Merkle tree. The root hash represents the checksum of all chunks on that node. When two nodes claim to have the same chunk, comparing Merkle roots quickly identifies whether they agree. If they disagree, the tree structure allows binary-search descent to find the exact divergent chunk.

### Bloom Filters for Object Existence Checks

Before querying the metadata database, the system checks a Bloom filter to determine whether an object key might exist. A Bloom filter is a space-efficient probabilistic data structure that can say "definitely not present" or "possibly present." This avoids unnecessary database lookups for non-existent keys (common in LIST operations with prefixes that do not match).

### Write-Ahead Log (WAL)

Every mutation to the metadata store is first appended to a sequential log on durable storage before the metadata record is updated. This ensures crash recovery: if the metadata server crashes mid-write, on restart it replays the WAL to reconstruct any committed operations that had not yet been reflected in the metadata store.

## Consistency and State Handling

### Strong Read-After-Write Consistency

Modern object storage (S3 since December 2020) provides strong read-after-write consistency. Once you receive a 200 OK for a PUT, any subsequent GET always returns the latest data. This is achieved by routing all requests for a given key through a single metadata server that serializes operations using the WAL.

### Concurrent Write Handling

When two clients write to the same key simultaneously, the system serializes the writes through the metadata leader. The last WAL entry to be committed wins. The losing write's data chunks become orphaned and are cleaned up by the GC daemon. There is no locking or atomic compare-and-swap for object writes (though S3 does offer conditional writes via If-Match / If-None-Match headers).

### Stale Read Prevention

To prevent stale reads, the request router checks the metadata server's current term (epoch) before serving reads. If a metadata server has been partitioned from the leader and missed updates, it cannot serve read requests until it catches up. This is enforced by the consensus protocol — a server with a stale term is rejected by clients.

## Performance Characteristics

### Latency by Object Size

| Object Size | p50 Latency | p99 Latency | p999 Latency |
|-------------|-------------|-------------|--------------|
| 1 KB        | 5 ms        | 20 ms       | 80 ms        |
| 100 KB      | 10 ms       | 35 ms       | 120 ms       |
| 1 MB        | 25 ms       | 80 ms       | 250 ms       |
| 10 MB       | 80 ms       | 250 ms      | 800 ms       |
| 100 MB      | 400 ms      | 1,200 ms    | 3,500 ms     |
| 1 GB        | 2,500 ms    | 7,000 ms    | 18,000 ms    |

These numbers assume a single PUT or GET with standard network bandwidth (1 Gbps). Multi-Part Upload can improve large-object latency by parallelizing the transfer.

### Throughput Limits

A single prefix (path prefix within a bucket) supports at least 3,500 PUT/COPY/POST/DELETE or 5,500 GET/HEAD requests per second. To achieve higher throughput, you distribute objects across multiple prefixes. The system achieves this by partitioning the metadata store by key hash prefix — each partition handles its own request rate.

### Memory Footprint

- **Metadata server**: ~2 GB per million objects for in-memory location cache
- **Storage node**: ~500 MB for Merkle tree root nodes (regardless of data volume)
- **Bloom filter**: ~12.5 MB per 10 million keys (at 1% false positive rate)
- **WAL buffer**: ~64 MB per partition

## Configuration and Tuning

### Key Knobs

**Replication Factor** — Controls how many copies of each chunk are maintained. Default: 3. Minimum: 1 (single-copy, no durability guarantee). Maximum: 10+ (in some systems). Higher factor increases durability at the cost of storage cost per GB.

**Write Quorum** — Number of replicas that must acknowledge a write before it is confirmed. Default: W=2 (for RF=3). Setting W=3 provides stronger durability but adds latency proportional to the slowest replica.

**Read Quorum** — Number of replicas contacted during a read to verify the latest version. Default: R=1 (read from fastest replica). Setting R=3 provides read-your-writes consistency for multi-datacenter configurations at the cost of latency.

**Chunk Size** — The target size for data chunks. Default: 64 MB. Smaller chunks (8-16 MB) reduce latency for partial reads and improve parallelism for large objects. Larger chunks (128-256 MB) reduce metadata overhead for archive workloads.

**Scrub Interval** — How often the scrubber verifies each chunk. Default: every 30 days. More aggressive: every 7 days. The scrubber runs at low I/O priority, so aggressive intervals do not significantly impact production traffic.

### Aggressive Settings for High Performance

| Setting | Default | Aggressive | Impact |
|---------|---------|------------|--------|
| Chunk size | 64 MB | 8 MB | 8x more metadata entries, faster partial reads |
| Max concurrent part uploads | 4 | 25 | Higher throughput for large objects, more connections |
| Metadata cache TTL | 300s | 0s | Always consistent, higher metadata load |
| Storage node connections | 100 | 500 | Higher throughput, more memory for sockets |
| Read-ahead | 1 chunk | 4 chunks | Faster sequential reads, wasted I/O for random reads |

### Monitoring Key Indicators

| Metric | Warning | Critical | Meaning |
|--------|---------|----------|---------|
| Write latency p99 | above 200 ms | above 500 ms | Metadata contention or slow disks |
| Read latency p99 | above 100 ms | above 300 ms | Network congestion or overloaded nodes |
| Replication lag | above 5 seconds | above 30 seconds | Follower falling behind, risk of data loss |
| Under-replicated chunks | above 0.1% | above 1% | Disk or node failures need investigation |
| Scrubber errors | above 0.001% | above 0.01% | Silent data corruption or hardware issues |
| GC backlog | above 10 TB | above 50 TB | Cleanup not keeping up with deletes |
