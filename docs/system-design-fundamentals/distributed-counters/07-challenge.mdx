## Tier 1 — Foundation (5 minutes each)

### Problem 1: Redis Shard Sizing

You need a distributed counter that handles 2 million increments per second at peak. Each Redis shard can sustain 50,000 INCR operations per second. Your load balancer distributes keys using consistent hashing with 256 virtual nodes per physical shard. You want 40% headroom for traffic spikes.

**Calculate the minimum number of physical Redis shards required. Show your work.**

<details>
<summary>Model Answer</summary>

Target throughput: 2,000,000 increments/sec.

Per-shard capacity: 50,000 increments/sec.

Raw shard count without headroom: 2,000,000 / 50,000 = 40 shards.

With 40% headroom: target utilization = 60% of capacity.

Effective per-shard capacity for planning: 50,000 × 0.6 = 30,000 increments/sec.

Required shards with headroom: 2,000,000 / 30,000 = 66.7, round up to 72 shards.

**Answer: 72 physical shards.**

**Common mistake**: Forgetting to account for headroom, or applying headroom to the total instead of per-shard capacity. Some candidates calculate 40 shards and stop, missing the spike tolerance.

**Evaluation**: Correct calculation with headroom = 3 points. Stated shard count without calculation = 1 point. No headroom = 1 point deduction.
</details>

### Problem 2: Staleness Window Analysis

Your distributed counter system has the following parameters:
- Read cache TTL: 200ms
- Async replication lag: p50 = 2ms, p99 = 40ms, p999 = 200ms
- Gossip merge interval: 100ms
- Gossip convergence rounds: 5 rounds

**Calculate the worst-case staleness window for a read that misses the cache. Then identify the single component you would optimize to reduce the worst-case staleness by 50%.**

<details>
<summary>Model Answer</summary>

Worst-case staleness for a cache miss read:
- The read itself: 0ms (it is a cache miss, so it goes directly to the aggregator)
- The aggregator reads from one shard replica: 200ms replication lag (p999)
- The gossip merge is not on the read path — it affects the shard's state but the read gets whatever the shard's current value is

Wait — the staleness depends on what we measure. Two scenarios:

**Scenario A: Read-after-write staleness.** A client writes an increment, then immediately reads the counter. The write was just applied to the primary. The read goes:
- Cache miss (the cache has not been updated since the write)
- Aggregator fans out to all shards
- One shard's replica responds with a value that does not include the latest write (replication lag)
- Worst case: read returns value that is 200ms stale

**Scenario B: Cross-region staleness.** An increment is applied in US-East-1. A read in EU-West-1 must wait for cross-region replication + gossip merge:
- Cross-region replication lag: 1,000ms (typical estimate for Kafka MirrorMaker)
- Gossip merge interval: 100ms × 5 rounds = 500ms
- Read cache TTL: 200ms
- Total worst-case: 200ms + 1,000ms + 500ms = 1,700ms

To reduce the worst-case staleness by 50%, optimize the component with the highest contribution. In scenario A: reduce async replication lag by using synchronous replication (adds 5ms to write latency but drops p999 replication lag from 200ms to 1ms). In scenario B: reduce the read cache TTL from 200ms to 50ms and the gossip merge interval from 100ms to 20ms — the wildcard is cross-region replication, which requires a faster link or active geo-replication.

**Evaluation**: Correctly identifies the largest contributor = 2 points. Proposes a specific optimization with trade-off = 2 points. Forgets to mention trade-off of the optimization = 1 point deduction.
</details>

### Problem 3: Idempotency Key Design

You are implementing idempotency keys for a distributed counter. Each client generates a unique idempotency key per request and sends it in the HTTP header `Idempotency-Key: <uuid>`. The counter shard stores seen keys in a Redis Set with a TTL.

**Draw the flow diagram and describe: what happens when a key expires before the retry arrives? What happens when two different requests (different idempotency keys) for the same counter arrive at the same millisecond? Is there any risk of double-counting?**

<details>
<summary>Model Answer</summary>

**Key expiry before retry**: The idempotency key has a TTL of 300 seconds. If the client's retry arrives 310 seconds after the original request, the key has expired. The shard does not recognize the key as a duplicate and applies the increment a second time. The counter overcounts.

**Mitigation**: Set the idempotency key TTL to 5x the maximum expected retry window. If the client timeout is 5 seconds and it retries up to 3 times with exponential backoff (5s, 10s, 20s max), the retry window is 35 seconds. Set TTL to 175 seconds (5 × 35).

**Two different requests, same millisecond**: They have different idempotency keys. Both are new. Both pass the idempotency check. Both increments are applied. The counter increments by 2. This is correct behavior — these are two distinct requests.

**Risk of double-counting**: Yes, if the shard crashes between receiving the INCR and recording the idempotency key in the Redis Set. The client times out (no response), retries with the same idempotency key. The new primary has no record of the key. It applies the increment again. The counter overcounts.

**Mitigation**: Use a write-ahead log (Redis AOF) that records both the INCR and the idempotency key insertion atomically. Or use a database transaction that commits the increment and the idempotency key together.

**Evaluation**: Identifies the key-expiry scenario = 1 point. Identifies the crash-before-record scenario = 2 points. Proposes a concrete mitigation = 2 points.
</details>

## Tier 2 — Design (15 minutes each)

### Problem 4: Social Media Like Counter

Design a like counter for a social media platform with 100 million daily active users. Peak writes occur during a major event (like the Super Bowl) when 500,000 likes per second arrive. The like counter must display on the post within 500 milliseconds of the user tapping the button. The platform needs to handle 10,000 distinct posts receiving likes simultaneously.

**Provide:**
- Shard count and placement
- Read and write path descriptions
- Throughput and latency calculations
- Hot key strategy for viral posts
- Idempotency approach
- Cost estimate for the counter infrastructure

<details>
<summary>Model Answer</summary>

**Throughput**: 500,000 writes/sec peak. Each write = 1 INCR. Redis per-shard capacity: 50,000 ops/sec. Target utilization: 60%. Effective per shard: 30,000 ops/sec. Required: 500,000 / 30,000 = 17 shards. Use 20 shards for headroom.

**Shard placement**: 20 Redis instances, each cache.r6g.xlarge (8 GB, 2 vCPU, $240/month). 2 replicas per shard in different AZs (total 60 instances). Consistent hashing with 256 virtual nodes per physical shard for even distribution.

**Write path**: Client → HTTP POST with idempotency key → API Gateway (rate limit 100 req/s per client) → Consistent Hash Router → shard Redis INCR → shard idempotency Set (TTL 300s) → AOF (fsync 1s) → async replica → HTTP 200 response. Latency: 3-15ms.

**Read path**: Client → GET post → CDN edge (TTL 30s) → cache miss → web app → Read Aggregator → fan-out to 20 shards → sum → response + cache update (TTL 200ms). Latency (cache hit): 1-2ms. Latency (cache miss): 10-30ms.

**Hot key strategy**: Pre-shard each post counter into 64 sub-keys. For post `abc123`, create keys `like:abc123:0` through `like:abc123:63`. Increment reads a random sub-key or hashes the user_id to a sub-key. Read sums all 64 sub-keys. A viral post's 500K likes spread across 64 keys = 7,800 reads/sec per key — well within Redis capacity.

**Idempotency**: Client generates `<user_id>:<post_id>:<timestamp_ms>` as the idempotency key. Stored in Redis Set on the same shard as the counter. TTL: 300 seconds.

**Cost estimate**: 20 primary × $240 + 40 replicas × $240 = $14,400/month. Add aggregator (4 t3.medium instances at $60/month): $240. Add API Gateway: approximately $500/month (first 300M API calls free, then $3.50/M). Data transfer: approximately $2,000/month. **Total: $17,140/month.**

**Common mistakes**: Not pre-sharding hot keys; forgetting idempotency keys; using more write capacity than needed (over-provisioning shards to 100+ without justification).
</details>

### Problem 5: Rate Limiting Counter

Design a distributed rate limiter that enforces a limit of 100 requests per minute per API key across 100 application servers. The rate limiter must detect violations within 1 second of the 100th request arriving. Each application server processes 10,000 requests per second.

**Provide:**
- Counter data structure and merge semantics
- Write path (each request increments the counter)
- Read path (checking whether the limit is exceeded)
- How you handle the 1-second detection SLA
- What happens during a network partition

<details>
<summary>Model Answer</summary>

**Counter structure**: A sliding window counter using a sorted set in Redis. Each request is stored as a member with score = timestamp. Count = number of members with score > (now - 60 seconds). This avoids the clock synchronization problems of simple counters.

Alternative: A fixed window counter per (api_key, minute_bucket). Key = `ratelimit:api_key_123:2026-07-01T14:05`. INCR the key. If value exceeds 100, reject. Reset every minute. Simpler but allows bursts at window boundaries (101 requests in the last second of the window + 99 in the first second of the next window = 200 in 2 seconds).

**Choice**: Fixed window with minute bucket. The boundary burst is acceptable for rate limiting (the spec says "100 requests per minute," not "100 requests in any sliding 60-second window").

**Write path**: Each application server sends `INCR ratelimit:api_key:minute_bucket` to the Redis cluster. If the response is under 100, the request passes. If 100 or above, reject with 429.

**Read path**: The INCR response _is_ the read — we get the new value atomically. No separate read needed. This is the key optimization for rate limiting: write and read are a single operation.

**1-second SLA**: INCR on a sharded Redis cluster completes in under 1ms (same-AZ). Even with 100 app servers × 10,000 req/sec = 1,000,000 writes/sec aggregate, 20 Redis shards handling 50,000 ops/sec each keep per-request latency under 5ms. The 1-second SLA is easily met.

**Partition behavior**: If the Redis shard is unreachable, the application server cannot check the rate limit. Two approaches: (1) Fail open — allow the request (risk: attacker can exceed the limit during the partition). (2) Fail closed — reject the request (risk: legitimate users are blocked). For rate limiting, fail open is standard — the cost of a false positive (blocking a user) is higher than the cost of a false negative (allowing extra requests during a brief partition).

**Evaluation**: Chooses appropriate counter structure = 2 points. Identifies INCR as read+write combined = 2 points. Discusses partition behavior = 1 point.
</details>

### Problem 6: Real-Time Leaderboard Counter

Design a real-time leaderboard for a gaming platform with 1 million concurrent players. Each player's score changes multiple times per second during gameplay. The leaderboard must show the top 100 players updated within 2 seconds of a score change.

**Provide:**
- Counter data structure
- How you maintain the top-100 sorted list
- Write throughput calculations
- How you handle a player whose score increases by 100 points in a single 100ms burst (10 score updates)

<details>
<summary>Model Answer</summary>

**Counter structure**: Redis Sorted Set. Key = `leaderboard:game_id`. Member = `player_id`. Score = `current_score`. Each score increment is `ZINCRBY leaderboard:game_id <delta> <player_id>`. Redis Sorted Sets use a skip list internally, giving O(log N) for both ZINCRBY and ZRANGE.

**Top-100 reads**: `ZREVRANGE leaderboard:game_id 0 99 WITHSCORES`. This is O(log N + K) where K = 100. For 1 million players, each read is approximately 20 operations on the skip list — completes in under 1ms.

**Write throughput**: Each score update is one `ZINCRBY`. With 1 million concurrent players generating 3 updates/sec each, total writes = 3,000,000/sec. Redis single instance handles 50,000 ops/sec. Need 60 shards. But a sorted set cannot be sharded — the top-100 query needs to see all players. Solutions:

1. **Single leaderboard per game**: One Redis instance handles all 3M writes/sec. At 50K ops/sec, one instance handles approximately 50K writes/sec — 60x too slow. Solution: batch score updates locally on game servers and flush every 200ms. With 100 game servers, each flushes 30,000 updates (500 updates × 100ms window / 200ms flush... recalculate: 3M updates/sec × 200ms = 600,000 updates per batch). Each batch has 600,000 / 100 game servers = 6,000 updates per server. Pipeline these as ZINCRBY commands. Redis processes 6,000 commands in a pipeline at approximately 100,000 ops/sec on a single connection = 60ms. Total: 200ms batch + 60ms Redis = 260ms. Within the 2-second SLA.

2. **Hierarchical leaderboard**: Shard players by score range or region, maintain per-shard top-100 lists, and merge periodically. More complex but scales to higher throughput.

**Burst handling**: A burst of 10 score updates in 100ms for the same player. With local batching, the game server accumulates all 10 updates in the buffer and flushes a single ZINCRBY with the sum (100 points) instead of 10 individual commands. This is both correct (the order of updates within a 200ms window does not matter for the total score) and efficient (1 Redis command instead of 10).

**Evaluation**: Identifies sorted set as the data structure = 1 point. Recognizes the single-shard bottleneck = 2 points. Proposes batching or hierarchical solution = 2 points.
</details>

## Tier 3 — Expert (30 minutes each)

### Problem 7: Global Counter with Regional Independence

Design a global view counter for a video platform that must handle 10 million views per minute across 5 AWS regions (US-East, US-West, EU-West, AP-Southeast, AP-Northeast). The counter must provide a globally consistent total within 5 seconds of any individual view. Each region must continue operating independently if inter-region connectivity is lost for up to 30 minutes.

**Provide:**
- Counter data structure with merge semantics
- Region topology and cross-region replication architecture
- Read path for both regional and global totals
- Write path and latency per region
- Recovery after a 30-minute partition
- Reconciliation with the authoritative event log

**Evaluation dimensions:**
- **Correctness**: Does the merge function preserve all increments across all regions during a partition? (30%)
- **Latency**: Is the write path sub-5ms within a region, and does the global read converge within 5 seconds? (25%)
- **Survivability**: Do all 5 regions continue operating during a 30-minute partition? (25%)
- **Operational details**: Is there a monitoring and reconciliation plan? (20%)

**Anti-patterns to avoid:**
- Using wall-clock LWW (clock skew between regions drops increments)
- Synchronous cross-region writes (adds 200-500ms to every write)
- Single-region aggregation for global reads (single point of failure)
- Missing idempotency (double-counting from retries across regions)
- Ignoring tombstone GC in PN-Counter design

<details>
<summary>Discussion Points</summary>

**Counter structure**: G-Counter (grow-only counter) with one vector element per region. Total = sum of each region's contribution. G-Counter is sufficient for view counting (no decrements). Merge = element-wise max. During a partition, each region increments its own element. When the partition heals, merge via element-wise max recovers all increments from both sides.

5 vector elements:
```
Region US-EAST-1:  [45,000,000, 0, 0, 0, 0]
Region EU-WEST-1:  [0, 0, 32,000,000, 0, 0]
...merged...
All regions: [45,000,000, 12,000,000, 32,000,000, 28,000,000, 15,000,000]
Total = 132,000,000
```

**Region topology**: Each region runs its own 64-shard Redis cluster for the regional counter. Cross-region replication: each region's counter shards asynchronously replicate their G-Counter vector to a regional aggregator, which publishes the vector to a cross-region Kafka topic. Other regions consume this topic and merge the incoming vector into their local state.

**Write path**: Client in US-East-1 → API Gateway → consistent hash → shard INCR (0.08ms). The increment updates the local G-Counter element for US-East-1. Google Cloud Spanner is NOT needed — no cross-region coordination on writes.

**Regional read path**: Read aggregator fans out to 64 local shards → sum → cache (TTL 200ms). Returns regional total within 10ms.

**Global read path**: Read aggregator reads all 5 regional vectors from the local cache (each region's latest published vector is cached with 1-second TTL) → merge via element-wise max → sum → cache (TTL 1s). Returns global total within 50ms.

**30-minute partition recovery**: Each region's G-Counter element continues incrementing independently. After 30 minutes, US-East-1 has processed 15 million views. EU-West-1 has processed 10 million views. Both elements in each region's vector are preserved. When connectivity restores, each region's gossip exchange merges the vectors. The global total after merge = sum of both elements = 25 million additional views. No loss.

**Reconciliation with event log**: Each view event is also written to S3 (in the region's own bucket) with a timestamp. A Spark job runs daily, replays all 5 regions' event logs (sorted by timestamp), and computes the authoritative total. If the distributed counter is off by more than 0.1%, the counter is corrected and an incident is filed.
</details>

### Problem 8: Warehouse Inventory Counter

Design a distributed counter for warehouse inventory at a large e-commerce company. The system must track available units for 10 million SKU across 500 warehouse pick stations. Each pick station decrements inventory when an item is picked, and the receiving dock increments inventory when new stock arrives. The system must prevent overselling (inventory dropping below zero) while maintaining sub-10ms decrement latency. Pick stations operate on unreliable local networks that experience 3-second partitions several times per hour.

**Provide:**
- Counter data structure with merge semantics
- How you handle decrement-under-zero prevention (the critical requirement)
- The behavior during a 3-second partition at a pick station
- Reconciliation when the partition heals
- How you handle the receiving dock (which increments inventory and must eventually see the current level)

**Evaluation dimensions:**
- **Safety**: Does the system prevent inventory from going below zero? (40%)
- **Liveness**: Does the system continue operating during a partition? (20%)
- **Throughput**: Can 500 pick stations each do 5 picks per second (2,500 decrements/sec total)? (20%)
- **Operational simplicity**: Is the system understandable and debuggable? (20%)

**Anti-patterns to avoid:**
- Using PN-Counter without tombstones (net negative not detectable)
- Relying on wall-clock timestamps for ordering
- Pessimistic locking across all 500 stations (creates a centralized bottleneck)
- Letting the counter go negative then correcting later (violates the "no overselling" requirement)

<details>
<summary>Discussion Points</summary>

This is the hardest distributed counter problem because the core requirement — prevent overselling — conflicts with the distributed nature of the system. A distributed counter is eventually consistent by default, which means it CAN oversell because two stations may each think inventory is 1 and each decrement.

**Solution**: A hybrid approach with a centralized inventory tracker and distributed reservation counters.

**Centralized inventory tracker**: A single PostgreSQL instance (or Spanner for global) stores the authoritative inventory count per SKU. This is the system of record for "how many units exist." Writes use `UPDATE inventory SET qty = qty - :picked WHERE sku = :sku AND qty >= :picked RETURNING qty`. This is an atomic compare-and-swap — if the transaction sees qty < picked, the UPDATE returns 0 rows and the decrement is rejected.

**Distributed reservation counter**: Each pick station maintains a local counter of "reserved but not yet picked" units. When a station receives a pick request, it first reserves inventory against the centralized tracker (the atomic UPDATE above). If the reservation succeeds, the station increments its local counter. The picker then fulfills the pick, and the station decrements the local counter.

**Behavior during partition**: The centralized tracker is unreachable during the 3-second partition. New pick requests cannot reserve inventory. The pick station continues processing picks that were already reserved (its local counter tracks pending picks). After 3 seconds, connectivity restores and the station can reserve again.

**Behavior at the receiving dock**: The dock increments the centralized tracker: `UPDATE inventory SET qty = qty + :received WHERE sku = :sku`. This is a simple increment with no conflict. The new inventory becomes available for reservations immediately.

**Why not a distributed counter**: The requirement to never oversell means you need a single source of truth for the absolute quantity. A distributed counter cannot provide this — it can only converge to the correct value after the fact. The centralized tracker with atomic compare-and-swap is the safe choice. The distributed counter is used only for per-station tracking of in-flight picks.

**Evaluation**: Recognizes that a distributed counter alone cannot prevent overselling = major insight. Proposes the hybrid architecture with centralized atomic decrement + local reservation counters = full marks.
</details>
