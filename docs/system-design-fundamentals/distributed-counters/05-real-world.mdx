## Company Case Studies

Four production systems show how distributed counters handle extreme scale.

### Netflix: The Viewing Activity Counter

Netflix tracks over 1 billion viewing hours per week across 260 million subscribers. Each viewing session generates multiple counter increments: play start, resume, pause, stop, and chapter transitions. The counter system must handle 500,000+ increments per second at peak.

**Architecture**: Netflix uses a sharded counter built on Apache Cassandra. Each counter row has a partition key (content_id + date) and multiple counter columns (play_start, play_end, chapter_view). Cassandra's distributed counter implementation uses a PN-Counter internally — each coordinator node stores the counter delta locally and reconciles during read repair.

**Key fact from engineering blogs**: Netflix's Cassandra counters use a two-phase increment internally. The coordinator writes the delta to a local commit log, then asynchronously propagates it to all replicas. If the coordinator crashes before propagation, the increment is lost. Netflix mitigates this by running 3 replicas per counter and accepting that the final count may undercount by up to 0.1%.

**Trade-off**: Cassandra counters sacrifice strong consistency for write throughput. A read must contact all replicas and sum their values to get the latest count. Netflix optimizes by running a periodic batch job that computes authoritative totals from the event log (S3 + Spark) and backfills the counters.

### Discord: The Message Reaction Counter

Discord processes 4 billion server messages daily. Each message can have multiple emoji reactions, each reaction increments a counter. With hundreds of thousands of concurrent users reacting to popular messages, the counter system must be fast and reliable.

**Architecture**: Discord initially used vanilla Redis `INCR` counters but ran into hot key issues on large community servers where a single announcement message could receive 50,000+ reactions in minutes. They moved to a custom sharded counter: each emoji reaction is stored in a separate Redis key, and the total reaction count for a message is the sum of all emoji-specific keys.

**Key fact from engineering blogs**: Discord's key insight was to pre-shard the hot key at the application layer. Instead of one counter per message, they create one counter per emoji per message. A message with 10 emojis has 10 counter keys, each on potentially different Redis shards. The write load for a single message is distributed across up to 10 shards.

**Trade-off**: Read complexity increases — reading the total requires summing 10 keys instead of 1. But reads are cached (1-second TTL), and the write throughput on hot messages scales linearly with the number of emojis.

### Uber: The Trip Counter

Uber tracks trips, earnings, surge multipliers, and driver incentives across 70+ countries. Each event (trip start, trip end, payment) increments multiple counters for different aggregations (per driver, per city, per hour, per day).

**Architecture**: Uber uses a two-layer counter with a local buffer on each application server and a global Redis-based aggregator. Each application server maintains a per-second buffer of counter increments. Every 200ms (or every 500 increments, whichever comes first), the server flushes the buffer to a central Redis cluster via multi-key `MSET` or `INCRBY` in a pipeline.

**Key fact from engineering blogs**: Uber's counter system processes 15 million increments per second at peak. The local buffer approach reduces Redis write load by a factor of 200: each application server sends one batch of 500 increments instead of 500 individual INCR commands. Network bandwidth for counter traffic dropped by 85% after implementing batching.

**Trade-off**: Data loss on application server crash. Each server's unflushed buffer (up to 500 increments, approximately 200ms of work) is lost when the server dies. Uber accepted this trade-off because trip data is also persisted in the trip database — the counter is a real-time estimate, not the system of record.

### Cloudflare: The Rate Limit Counter

Cloudflare's rate limiting service tracks the number of requests from each IP address across 330 data centers worldwide, enforcing rate limits at the edge with sub-50ms detection.

**Architecture**: Cloudflare uses a CRDT-based distributed counter. Each edge data center maintains a local G-Counter for each (IP, time_bucket) pair. Counters are merged across data centers using a gossip protocol every 10 seconds. When a client exceeds the rate limit threshold, the edge returns HTTP 429.

**Key fact from engineering blogs**: Cloudflare's rate limiting counters have a unique requirement: they MUST be eventually consistent because a globally consistent counter would require cross-datacenter coordination on every request, adding 200-500ms latency. Cloudflare accepts that a determined attacker may briefly exceed the rate limit due to counter staleness. The system compensates by over-provisioning the limit by 5% to absorb the staleness-induced overshoot.

**Trade-off**: Over-provisioning means legitimate users may be rate-limited slightly earlier than the nominal threshold (at 105 requests instead of 100). Cloudflare determined that 5% false positives is acceptable for rate limiting, where the cost of a missed attack (server overload) far exceeds the cost of a slightly over-zealous limit.

## Open Source Implementations

Three open-source systems implement distributed counters with different trade-offs.

### Redis (with Redis Cluster)

**How it works**: Redis `INCR` and `INCRBY` commands provide atomic single-node counters. Redis Cluster adds automatic sharding across nodes with consistent hashing. Replication is asynchronous.

**Strengths**: Sub-millisecond latency; atomic INCR on the hot path; cluster mode scales to 1,000+ nodes; AOF persistence provides crash recovery within the fsync window.

**Weaknesses**: Single-threaded per shard (hot key bottleneck); no native CRDT support (split-brain requires manual recovery); memory-bound (all counters in RAM).

**Best for**: High-throughput counters where the cost of RAM is acceptable and hot keys are rare.

### Cassandra

**How it works**: Cassandra's `COUNTER` column uses a PN-CRDT internally. Each coordinator writes to a local commit log and asynchronously propagates to replicas. Read queries contact all replicas and merge their values.

**Strengths**: Native CRDT counter with automatic convergence; tunable consistency (ONE, QUORUM, ALL for both reads and writes); survives node failures without data loss (within the async replication window); no single point of contention per counter.

**Weaknesses**: Read latency is higher (must contact all replicas); counter reads with QUORUM can timeout if a replica is slow; decommissioning a node requires migrating counter state carefully; write throughput per counter is lower than a single Redis INCR.

**Best for**: Multi-datacenter deployments where counters must survive node and rack failures.

### Riak (with Riak DT)

**How it works**: Riak's data types module implements PN-Counters as part of its CRDT library. Each value is stored with its full vector clock. Reads trigger read repair: if a read encounters a replica with a stale vector clock, it automatically repairs the replica.

**Strengths**: Automatic read repair; configurable replication factor N, read quorum R, write quorum W; counters survive partitions with automatic merge.

**Weaknesses**: Riak's community edition is no longer actively maintained; vector clock metadata grows large for counters with many concurrent writers; read-heavy workloads require careful tuning.

**Best for**: Systems that need CRDT counters with automatic read repair and are already running Riak.

## Cloud Provider Implementations

Managed cloud services that implement distributed counter primitives.

### AWS DynamoDB

**Feature**: DynamoDB `UpdateItem` with `ADD` operation on a Number attribute provides atomic increment. The increment is applied server-side by the DynamoDB service.

**Cost/ops trade-off**: Each increment consumes 1 write capacity unit (WCU) per 1 KB of item size. A counter item is typically under 200 bytes, so 1 increment = 1 WCU. At $1.25 per million WCUs, 100 million increments per day costs $125. On-demand mode costs $1.4286 per million writes. **Total: $142.86/day for 100M increments in on-demand mode.**

**Strengths**: Managed — no server to operate; auto-scaling; DAX (DynamoDB Accelerator) for sub-millisecond reads; Global Tables for multi-region replication.

**Weaknesses**: Write throughput per partition is limited to 1,000 WCUs (1,000 increments per second per partition key) unless you use DynamoDB adaptive capacity. Hot keys can still throttle. Cost at scale is higher than self-managed Redis.

### Google Cloud Firestore (Datastore Mode)

**Feature**: Firestore `sum` aggregate query provides a server-side counter. Use `set` with `merge: true` to update only the counter field.

**Strengths**: Managed; strong consistency (reads always see the latest write within a regional instance); integrates with Cloud Functions for event-driven processing.

**Weaknesses**: Write throughput is limited to 1,000 operations per second per document. Exceeds this, and writes are rejected. Firestore is not designed for high-frequency counter increments.

### Azure Redis Cache

**Feature**: Azure Redis Cache provides managed Redis with `INCR` semantics identical to open-source Redis.

**Cost/ops trade-off**: P1 tier (6 GB, 1,000 connections) at $420/month handles approximately 50,000 ops/sec. For 100,000 ops/sec, you need two instances at $840/month. Adding clustering (C1 tier) for automatic sharding: approximately $1,500/month for 4 shards.

**Strengths**: Fully compatible with open-source Redis commands; geo-replication across regions; active geo-replication with conflict resolution (using last-writer-wins — accepts clock skew risk).

## Lessons Learned

Five key lessons from production distributed counter implementations.

**Lesson 1: Accept loss on the write path, reconcile on the batch path.** Every eventually-consistent distributed counter loses some increments. Netflix loses 0.1%; Uber loses 0.05%. The production pattern is not to eliminate loss entirely — that requires consensus on every write, which caps throughput. Instead, build a separate batch reconciliation pipeline that computes authoritative totals from the event log and corrects the counter.

**Lesson 2: The hot key problem is the ceiling on throughput.** Even with 1,000 shards, a single viral post or a single popular emoji can overwhelm one shard. Pre-sharding at the application layer (Discord's approach: one counter per emoji per message) is the only reliable mitigation. Auto-scaling shards does not help — the hot key is a single point of serial execution.

**Lesson 3: Idempotency keys are not optional.** Every production incident where counters overcount traced back to missing or expired idempotency keys. Budget the idempotency TTL generously (at least 5x the maximum expected retry window). Monitor idempotency hit rates — a sudden increase means clients are experiencing timeouts.

**Lesson 4: Always have a second source of truth.** The distributed counter is a real-time estimate. The authoritative value comes from replaying the event log offline. Cloudflare's rate limiting counters are the exception (no second source of truth needed for an ephemeral rate limit). For like counters, impression counters, trip counters — the event log is the truth, and the distributed counter is a cached projection.

**Lesson 5: Monitor staleness, not just throughput.** The most common failure mode is not "counter is down" but "counter is wrong." Monitor p99 staleness (not just p50), and alert when staleness exceeds twice the target SLA. Discord discovered that their p50 staleness was 5ms but p999 was 8 seconds during a hot key event — a 1,600x difference that went unnoticed because they were only monitoring p50.
