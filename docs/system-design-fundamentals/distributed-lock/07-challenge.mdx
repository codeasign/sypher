## Tier 1: Foundation

These challenges verify that you understand the core concepts of distributed locks: the differences between implementations, the safety and liveness guarantees each provides, and how fencing tokens prevent the delayed-writer problem.

### Challenge 1.1: Choose the Right Lock Implementation

You are consulting for three different startups. Each has a distinct set of requirements for a distributed lock. Your job is to recommend the correct lock implementation -- Redis SETNX, ZooKeeper ephemeral sequential znodes, etcd leases, or database SELECT FOR UPDATE -- and justify your choice with specific properties of the implementation.

**Startup A -- Payment Processing Pipeline**: Processes 500,000 transactions per hour. A duplicate lock grant would cause a double-charge to the customer. The lock service runs on a Kubernetes cluster with 12 worker nodes. The maximum acceptable lock acquisition latency is 20 milliseconds at P99. Lock hold time is 3 to 50 milliseconds per transaction. The team has three infrastructure engineers who manage 15 other services.

**Startup B -- Analytics ETL Job Scheduler**: Runs 200 batch jobs per hour across 50 worker nodes. If two workers execute the same job, the result is duplicate data in the analytics database, which is acceptable because downstream queries use DISTINCT. Lock acquisition latency up to 200 milliseconds is acceptable. The lock hold time is 30 seconds to 5 minutes. The team has one infrastructure engineer shared across five teams.

**Startup C -- Configuration Management Server**: Distributes configuration updates to 200 application servers. Only one administrator should modify a configuration at a time. The lock is held for 10 to 300 seconds. A lock failure causes a temporary inconsistency that is resolved at the next sync cycle (every 60 seconds). The lock service must survive the failure of any two nodes. The team has two infrastructure engineers.

**Model Answer**:

**Startup A should use etcd leases with fencing tokens.** The zero-tolerance for duplicate lock grants requires a consensus-based lock service. Redis SETNX provides only probabilistic safety -- a master failure during key replication can silently grant the lock to two processes. ZooKeeper and etcd both provide linearizable consistency through consensus, but etcd is easier to operate on Kubernetes (etcd-operator) and the team's Kubernetes expertise reduces operational burden. The 20ms P99 latency requirement is achievable -- etcd writes commit in 3 to 8 milliseconds in the same region. The team of three engineers is sufficient to manage a 3-node or 5-node etcd cluster.

**Startup B should use Redis SETNX.** Duplicate job execution is acceptable because the analytics queries use DISTINCT. Redis provides the simplest implementation with the lowest operational overhead. The 200ms latency requirement is easily met -- Redis operations complete in under 1 millisecond. The 30-second to 5-minute hold time is well within Redis TTL capabilities. A single Redis instance with a replica provides sufficient availability. The single infrastructure engineer can manage Redis alongside their other responsibilities.

**Startup C should use ZooKeeper ephemeral sequential znodes.** Configuration management benefits from ZooKeeper's watch-based notification -- waiters are notified immediately when the lock is released, which aligns well with the 10 to 300 second hold times where polling would waste resources. The requirement to survive two node failures means a 5-node ZooKeeper ensemble where any 3 nodes form a majority. ZooKeeper's session-based ephemeral znodes automatically release the lock if the administrator's connection drops, which is appropriate for the interactive configuration use case. The two engineers can manage a ZooKeeper ensemble as part of their infrastructure.

**What to watch for**: You must connect specific implementation properties to the requirements. Recommending ZooKeeper for the payment pipeline because it is "more reliable" is not sufficient -- you must explain that etcd's Raft consensus and Kubernetes-native operation make it a better fit for that specific context. Recommending Redis for the payment system without acknowledging the duplicate grant risk is a red flag. You must show that you understand the trade-offs, not just list features.

### Challenge 1.2: Calculate Lock Contention and Throughput

A payment processing system uses a single distributed lock to protect a shared account balance table. The system has the following characteristics:

- 40 worker processes compete for the lock
- Each worker holds the lock for 80 milliseconds to read, update, and verify a balance
- Lock acquisition latency: 2 milliseconds
- Lock release latency: 1 millisecond
- The system operates 24 hours per day
- Peak load is 3x the average load and lasts for 4 hours each day
- Each successful lock acquisition allows one transaction to be processed
- Average revenue per transaction: $0.35

**Questions**:

A. What is the maximum throughput of the system in transactions per second? What is the maximum throughput in transactions per hour?

B. How many transactions can the system process in a 24-hour period, accounting for the 4-hour peak at 3x average load?

C. If you add 40 more workers (80 total), what is the new throughput? What happens to the lock contention rate?

D. The payment processing system must handle 150 transactions per second during peak. Currently it uses a single lock. What change do you recommend? Your solution must not introduce additional infrastructure cost beyond what already exists.

**Model Answer**:

A. Each transaction cycle takes 80ms (work) + 2ms (acquire) + 1ms (release) = 83ms. With a single lock, workers serialize. Maximum throughput = 1000ms / 83ms = 12.05 transactions per second. Per hour: 12.05 * 3600 = 43,373 transactions per hour.

B. Let average throughput be T transactions per second. Peak at 3x average means 3T transactions per second. There are 4 peak hours and 20 off-peak hours per day. Total daily transactions = (20 hours * T * 3600) + (4 hours * 3T * 3600) = 72,000T + 43,200T = 115,200T. The system can handle 12.05 TPS, so daily throughput = 115,200 * 12.05 = 1,388,160 transactions. Daily revenue at $0.35 per transaction = 1,388,160 * $0.35 = $485,856.

C. Adding 40 more workers does not increase throughput. The bottleneck is the single lock, not the number of workers. With 40 workers, all 40 are idle while one holds the lock. With 80 workers, all 80 are idle while one holds the lock. Throughput remains 12.05 TPS. The lock contention rate increases -- more workers are waiting for the same lock, which increases the number of failed acquisition attempts per second. The contention rate is the number of workers that cannot acquire the lock in a given second. With 40 workers at 83ms cycles, approximately 39 workers fail to acquire the lock on each cycle, resulting in approximately 39 / 83ms = 470 failed attempts per second. With 80 workers, this doubles to approximately 940 failed attempts per second.

D. At 150 TPS during peak, the single lock provides only 12.05 TPS. You need a 12.5x improvement. The recommendation is to switch from coarse-grained locking (single lock for the entire balance table) to fine-grained locking (one lock per account ID). With 10,000 active accounts and the same lock service, contention per account is negligible. Each worker locks only the specific account it is processing. The lock operations remain identical -- SETNX on Redis or SELECT FOR UPDATE on the database -- but the key or row includes the account identifier instead of a global lock name. This increases throughput from 12.05 TPS to the point where the bottleneck becomes the database write throughput, not the lock. If the database can handle 1,000 writes per second and each transaction touches 3 account locks sequentially, the system can process approximately 330 TPS, which exceeds the 150 TPS peak requirement.

**Evaluation rubric**:
- Correct throughput calculation: 3 points
- Correct daily throughput with peak accounting: 3 points
- Correct explanation of why more workers do not increase throughput: 3 points
- Correct contention rate calculation: 3 points
- Practical fine-grained lock recommendation without new infrastructure: 4 points
- Acknowledgment that lock granularity is the lever, not worker count: 4 points
- Total: 20 points. Pass: 14 points. Pass with distinction: 18 points.

### Challenge 1.3: Tracing the Fencing Token Guarantee

You are debugging a production incident in a distributed payment system. The system uses Redis SETNX locks with a 5-second TTL. Workers process payment batches, each batch containing 500 to 5,000 individual payments. A worker must hold the lock for the entire batch processing time, which averages 2 seconds.

The incident report states that two workers processed the same batch ID #48192. The first worker, W1, acquired the lock at timestamps 14:30:00.000 and began processing. The second worker, W2, also processed batch #48192 and wrote 427 payments that were already written by W1.

The relevant timestamps from the logs are:

| Event | Timestamp |
|---|---|
| W1 acquires lock | 14:30:00.000 |
| W2 acquires lock | 14:30:04.800 |
| W1 finishes processing | 14:30:05.200 |
| W2 finishes processing | 14:30:07.100 |

W1's Java garbage collection log shows a 3.5-second STW (stop-the-world) GC pause starting at 14:30:00.500 and ending at 14:30:04.000.

**Questions**:

A. Explain the exact sequence of events that caused both workers to process the same batch. Identify the root cause.

B. The lock TTL is 5 seconds. Did the lock expire before W2 acquired it? Show the math.

C. Did W1 know it had lost the lock when it resumed after the GC pause? Why or why not?

D. Design a fencing token system that would have prevented this incident. Specify the token source, how W1 would include the token in its writes, and how the database would reject W2's writes if W2 had a stale token.

E. If W1's writes are rejected due to a stale fencing token, what happens to the 427 payments that W2 already wrote? Who is responsible for cleanup?

**Model Answer**:

A. W1 acquired the lock at 14:30:00.000. At 14:30:00.500, a 3.5-second GC pause started. W1 was paused until 14:30:04.000. During this pause, W1 could not renew the lock or write to the database. The lock TTL of 5 seconds means expiry at 14:30:05.000. But how did W2 acquire at 14:30:04.800 if lock expiry is at 14:30:05.000?

The root cause is likely that an automated lock reaper process (a common anti-pattern in Redis lock deployments) cleaned up the lock key after detecting no recent activity from W1 during the GC pause. The lock reaper, configured with a 3-second stale threshold, detected that the lease was not renewed and forcefully deleted the lock key at approximately 14:30:03.700. W2 then acquired the lock at 14:30:04.800 because the key no longer existed. W1 resumed at 14:30:04.000 and continued processing, unaware that the lock had been removed.

B. The lock TTL of 5 seconds means the lock key expires at 14:30:05.000. However, the lock reaper deleted the key earlier at approximately 14:30:03.700. The lock was not released by TTL expiry but by an external process. The mathematical TTL was not the issue -- the lock reaper was.

C. W1 did not know it had lost the lock. After the GC pause ended at 14:30:04.000, W1 resumed execution as if nothing happened. The distributed lock library used a simple SETNX acquire that returned a boolean indicating success, but it did not check the lock status after a GC pause. W1's code path assumed that once the lock was acquired, it was held until explicitly released. This is the fundamental flaw in lock implementations without fencing tokens.

D. The fencing token system requires three components:

1. **Token source**: A monotonically increasing counter stored in the lock service (or a separate atomic counter in Redis). Each lock grant atomically increments the counter and returns the new value as the fencing token. When W1 acquires the lock, it receives token 1001. When W2 acquires the lock after the reaper removes W1's key, W2 receives token 1002.

2. **Token verification in writes**: Every write to the database includes the fencing token. The database table has a `last_token` column that stores the highest token that has written to this batch.

3. **Token enforcement**: The database checks the token before each write. If the write's token is higher than the stored `last_token`, the write proceeds and updates `last_token`. If the write's token is lower or equal to `last_token`, the write is rejected.

With fencing tokens, W1's writes (token 1001) that arrived after the GC pause would be rejected because W2 already wrote with token 1002. The database would return an error for the first write W1 attempted after resuming.

E. The 427 payments that W2 already wrote are valid because they were written with token 1002, which is the highest token seen. These payments are not duplicate -- they are the correct payments for batch #48192. The 0 payments that W1 wrote before the GC pause may also be in the database. If W1's pre-GC writes completed before the GC pause, those are also valid. If W1 only processed writes after the GC pause, none of its writes are in the database.

The responsible cleanup path: W1's writes are rejected by the fencing token check, so no cleanup is needed for W1. W2's writes are correct. The system should log the incident (W1 attempted stale writes that were rejected) for monitoring purposes. The batch #48192 is fully processed by W2.

**What to watch for**: You must trace the exact timeline and identify that the lock reaper (or equivalent) caused the issue, not the TTL itself. The GC pause prevented renewal, the reaper removed the lock, and W1 had no mechanism to detect this. Fencing tokens are the only complete solution -- simply increasing the TTL would not help because the reaper uses an independent timeout.

## Tier 2: Design

These challenges require you to design distributed lock solutions for realistic systems with specific constraints. Each challenge includes an evaluation rubric and a detailed model answer walkthrough.

### Challenge 2.1: Design Locks for a Multi-Region Inventory System

A global e-commerce company operates in four regions: US-East, US-West, EU-West, and AP-Southeast. Each region has 20 to 50 application servers that handle inventory updates for 2 million SKUs. The inventory database is a sharded PostgreSQL cluster with 64 shards distributed across regions.

When a customer places an order, the system must decrement the inventory count for that SKU. Two concurrent orders for the same SKU must not decrement below zero. The system processes 8,000 orders per second at peak.

**Constraints**:
- Inventory updates must be consistent across all regions -- overselling is not permitted
- P99 latency for inventory decrement must be under 50 milliseconds
- Each region must be able to operate independently for up to 30 minutes during a network partition: customers in that region should still be able to place orders and the inventory should not be oversold locally
- After the partition heals, inventory counts must converge to the correct global value
- 30 second write outage is acceptable during reconciliation
- Operational budget: two database administrators and one SRE

**Questions**:

A. Propose a locking strategy that satisfies all constraints. Be specific about the lock service, lock granularity, and lock topology.

B. Explain how your design handles a 15-minute network partition between US-East and AP-Southeast. Walk through the exact sequence of lock operations during and after the partition.

C. What is the maximum number of concurrent locks your system must support at peak? Show the calculation.

D. Estimate the lock acquisition latency for a same-region operation and a cross-region operation.

**Evaluation rubric**:
- Lock service selection justified against requirements: 4 points
- Lock granularity (per-SKU vs per-category vs global) justified: 4 points
- Topology described with specific node counts per region: 4 points
- Partition behavior explained step by step: 4 points
- Concurrent lock calculation correct: 2 points
- Latency estimates plausible with source explanation: 2 points
- Total: 20 points. Pass: 14 points. Pass with distinction: 18 points.

**Model Answer**:

A. **Lock service**: Use a 5-node etcd cluster with nodes distributed as US-East (3 nodes), EU-West (1 node), AP-Southeast (1 node). The US-East region hosts the leader because it has the most nodes and the lowest latency to the primary inventory shard. All lock writes require Raft consensus, which requires a majority of 3 out of 5 nodes. The majority can always be achieved in US-East (3 nodes) even if both other regions are partitioned.

**Lock granularity**: Per-SKU locking. Each SKU has a dedicated lock key in etcd. With 2 million SKUs and 8,000 orders per second, per-SKU locking provides maximum concurrency because orders are distributed across SKUs.

**Lock topology**: The etcd cluster spans three regions but the majority lives in US-East. Lock acquisition always goes through the etcd leader. If the leader is in US-East, all lock requests go to US-East, regardless of which region the worker is in.

B. **Partition scenario -- US-East isolated from AP-Southeast for 15 minutes**:

Minute 0: Network partition occurs. US-East nodes 1, 2, 3 can talk to each other but cannot reach AP-Southeast nodes 4 and 5.

Minute 0 to 15 (during partition):
- US-East: Nodes 1, 2, 3 form a majority (3 of 5). Workers in US-East and EU-West continue processing orders normally.
- AP-Southeast: Nodes 4 and 5 cannot reach the majority. They cannot participate in lock writes. AP-Southeast switches to local mode using a regional Redis cluster as a fallback lock service. Orders placed during the partition decrement a local counter.

Minute 15: Partition heals. AP-Southeast nodes 4 and 5 rejoin the etcd cluster. Reconciliation starts:
1. AP-Southeast's local Redis inventory counters are compared with etcd's global counters.
2. For each SKU where AP-Southeast sold units during the partition, the global inventory is decremented.
3. If global inventory is insufficient, excess AP-Southeast orders are flagged for compensation.

C. Maximum concurrent locks: 8,000 orders per second. Each order touches approximately 1.2 SKUs on average. That is 9,600 SKU operations per second. Each operation holds the lock for approximately 15 milliseconds. Maximum concurrent locks at any instant = 9,600 * 0.015 = 144 concurrent locks. Accounting for hot SKUs, approximately 1,150 concurrent locks. A 5-node etcd cluster handles this comfortably.

D. **Same-region latency**: Worker in US-East, etcd leader in US-East. Network round trip: 0.5 to 1 ms. Raft consensus: 3 to 5 ms. Total P99: 4 to 8 ms.

**Cross-region latency**: Worker in AP-Southeast, etcd leader in US-East. Network round trip across the Pacific: 120 to 160 ms. Raft consensus: 3 to 5 ms. Total P99: 125 to 170 ms.

The cross-region latency is a problem. Mitigation: workers in AP-Southeast batch their lock operations. Instead of acquiring one lock per order, they aggregate multiple orders for the same SKU into a single lock acquisition and batch update. For single-item orders on cold SKUs, the 150ms latency may exceed the 50ms P99 constraint. These orders are queued locally and processed asynchronously.

**What to watch for**: You must address the partition scenario concretely. Saying "etcd handles partitions" is not enough -- you must explain what happens to workers in the partitioned region. The cross-region latency mitigation (batching) shows that you recognize the latency problem and have a practical solution.

### Challenge 2.2: Design a Fair Lock for Ticket Allocation

A ticket marketplace sells 50,000 tickets for a major concert. The on-sale event generates 200,000 concurrent users attempting to buy tickets. The system must be fair: tickets should be allocated in the order users joined the virtual waiting room.

**Constraints**:
- FIFO ordering: the first user to join the waiting room gets the first ticket
- Each user buys 1 to 4 tickets but only one user can process at a time for a specific section
- The lock must support per-section locking (Floor, Lower Bowl, Upper Bowl, Balcony), each with 5,000 to 20,000 tickets
- No user should wait more than 5 minutes for a ticket allocation decision
- The system must handle the case where a user abandons their session while holding the section lock
- Lock acquisition must use Redis as the backend (the company already runs a 6-node Redis cluster)
- The implementation must prevent starvation

**Questions**:

A. Design the fair lock data structure in Redis. Specify the exact key names, data types, and values. Explain how the FIFO queue is implemented.

B. Walk through the lock acquisition flow for a user who is second in the queue. Show the exact Redis commands executed and the expected responses.

C. How does your design prevent starvation?

D. Propose a mechanism to detect and prevent multi-tab queue jumping without introducing a new infrastructure dependency.

**Evaluation rubric**:
- Redis data structure designed correctly for FIFO ordering: 4 points
- Lock acquisition flow explained with exact Redis commands: 4 points
- Starvation prevention explained with specific mechanism: 4 points
- Multi-tab detection without new infrastructure: 4 points
- Edge cases handled (session abandon, crash during queue): 4 points
- Total: 20 points. Pass: 14 points. Pass with distinction: 18 points.

**Model Answer**:

A. **Redis data structures**:

Each ticket section has three keys:

1. `ticket:section:{section_id}:queue` -- Redis List. A FIFO queue of user IDs waiting for this section. Users are added with `RPUSH`. The first user in the queue is at index 0.

2. `ticket:section:{section_id}:holder` -- Redis String. The user ID of the current lock holder for this section. Set with `SET NX` and a TTL of 30 seconds.

3. `ticket:section:{section_id}:next_allocation_time` -- Redis String (integer). A Unix timestamp in milliseconds representing when this section lock was acquired.

Additionally, a global counter tracks user position: `ticket:waiting_room:position_counter` -- Redis String (integer). Atomically incremented with `INCR` each time a user joins the waiting room.

B. **Acquisition flow for user U2 who is second in the queue**:

Step 1: U2 joins the section queue.
```
RPUSH ticket:section:floor:queue U2
```
Response: (integer) 2

Step 2: U2's client polls every 500ms to check if U2 is at the front.
```
LINDEX ticket:section:floor:queue 0
```
Response: "U1" (U1 is first)

Step 3: When U1 finishes, U2's poll returns:
```
LINDEX ticket:section:floor:queue 0
```
Response: "U2"

Step 4: U2 attempts to acquire the section lock.
```
SET ticket:section:floor:holder U2 NX PX 30000
```
Response: "OK"

Step 5: U2 removes themselves from the queue.
```
LPOP ticket:section:floor:queue
```
Response: "U2"

Step 6: U2 performs the purchase while holding the lock. Periodic renewal:
```
PEXPIRE ticket:section:floor:holder 30000
```
Response: (integer) 1

Step 7: U2 releases the lock after purchase.
```
DEL ticket:section:floor:holder
```
Response: (integer) 1

C. **Starvation prevention**: Starvation is structurally impossible in a pure FIFO system because every user moves toward the front as previous users complete. Two edge cases exist:

**Edge case 1: A user holds the lock indefinitely.** The 30-second TTL prevents this. After 30 seconds, the lock key expires and the next user in the queue can acquire it. The expired user's entry in the queue is cleaned up lazily.

**Edge case 2: A user with a slow connection holds up the queue.** Each user holds the lock for at most 30 seconds, so at most 10 users can be in the queue before the 5-minute limit is reached. If the queue exceeds 10 users, the system dynamically reduces the lock TTL to 10 seconds for that section.

D. **Multi-tab detection**: Use browser fingerprinting via session cookie combined with IP address and user-agent string. When a user joins the waiting room, create a hash: `SHA256(session_cookie + ip_address + user_agent)`. Store this hash in Redis with a 30-minute TTL at key `ticket:waiting_room:user:{hash}:position`. If a second join request arrives with the same hash within 30 minutes, check whether the existing position has already been served. If it is still in the queue or is the current holder, reject the new request.

This works without external dependencies because it uses Redis, which is already part of the infrastructure.

### Challenge 2.3: Design a Lock for Cache Rebuild Coordination

A social media platform serves 50 million daily active users. The feed service runs on 200 containers and uses a Redis cache for user feeds. When a user posts new content, the cache for all their followers must be invalidated. The cache rebuild pattern uses optimistic locking with a version counter.

Currently, when a user posts content, all 200 containers may simultaneously detect the stale cache and attempt to rebuild it from the database. This thundering herd causes 200 identical database queries per post, which scales linearly with the number of followers (average 300 followers per user, so 60,000 queries per post).

**Constraints**:
- Cache rebuild must happen within 500 milliseconds of a post
- The database can handle 5,000 queries per second -- exceeding this causes query queueing and timeouts
- The platform has 2,000 posts per second at peak
- Each follower cache rebuild requires 3 database queries
- Lock acquisition overhead must not exceed 5 milliseconds per attempt
- The solution must use existing infrastructure (Redis and PostgreSQL)

**Questions**:

A. Design a lock-based cache rebuild coordinator. Specify the lock key structure, TTL, and exact pseudocode.

B. Calculate the database query load under your design. How does this compare to the 5,000 QPS database limit?

C. What happens when a cache rebuild takes longer than the lock TTL?

D. The system uses Redis SETNX with a single-master deployment. The master fails and a replica is promoted with a 200ms write backlog. What happens?

**Evaluation rubric**:
- Lock key structure and TTL justified: 4 points
- Pseudocode demonstrates correct rebuild pattern: 4 points
- Database QPS calculation matches the design and respects the limit: 4 points
- TTL extension or early expiry handling explained: 4 points
- Redis failover analysis correct with specific behavior: 4 points
- Total: 20 points. Pass: 14 points. Pass with distinction: 18 points.

**Model Answer**:

A. **Lock key structure**: `cache:rebuild:follower:{user_id}:{follower_id}`. Each follower's cache rebuild is a separate lock.

**TTL**: 2 seconds. The rebuild requires 3 database queries plus cache write, which completes in under 200 milliseconds. The 2-second TTL provides a 10x safety margin.

**Pseudocode**:

```
function rebuild_follower_cache(user_id, follower_id):
    lock_key = "cache:rebuild:follower:" + user_id + ":" + follower_id
    owner_id = container_id + "-" + thread_id + "-" + uuid()

    # Step 1: Check cache (optimistic fast path)
    cache_version = REDIS.GET("cache:follower:" + follower_id + ":version")
    if cache_version != NULL and cache_version >= expected_version:
        return CACHE_VALID

    # Step 2: Acquire lock
    lock_acquired = REDIS.SET(lock_key, owner_id, NX=true, PX=2000)
    if not lock_acquired:
        return LOCK_HELD_BY_ANOTHER

    try:
        # Step 3: Double-check cache
        cache_version = REDIS.GET("cache:follower:" + follower_id + ":version")
        if cache_version != NULL and cache_version >= expected_version:
            return CACHE_ALREADY_REBUILT

        # Step 4: Rebuild cache from database
        user_profile = DB.QUERY("SELECT * FROM profiles WHERE user_id = ?", user_id)
        recent_posts = DB.QUERY("SELECT * FROM posts WHERE user_id = ? ORDER BY created_at DESC LIMIT 20", user_id)
        engagement = DB.QUERY("SELECT engagement_score FROM user_metrics WHERE user_id = ?", user_id)

        # Step 5: Write to cache
        cache_data = format_feed(user_profile, recent_posts, engagement)
        REDIS.SET("cache:follower:" + follower_id, cache_data)
        REDIS.SET("cache:follower:" + follower_id + ":version", expected_version)
        return CACHE_REBUILT_SUCCESSFULLY

    finally:
        # Step 6: Release lock (only if we still own it)
        lua_script = """
        if redis.call("GET", KEYS[1]) == ARGV[1] then
            return redis.call("DEL", KEYS[1])
        end
        return 0
        """
        REDIS.EVAL(lua_script, 1, lock_key, owner_id)
```

B. **Database query load calculation**: 2,000 posts per second at peak, average 300 followers per post, total potential rebuild operations per second = 600,000. Under the lock-based approach: only one container rebuilds per follower per post. But database QPS = 600,000 unique rebuilds * 3 queries = 1,800,000 QPS. This still exceeds the 5,000 QPS limit by 360x!

The correct design uses batching. Instead of rebuilding one follower at a time, execute a single batch query for all 300 followers' profiles (reducing 300 queries to 1 query), a single batch query for recent posts (1 query instead of 300), and compute engagement metrics in memory. Total database QPS becomes 2,000 posts * 3 batch queries = 6,000 QPS. With write-behind caching that batches 50 posts' worth of rebuilds, this is reduced to 4,000 QPS -- within the limit.

The key insight: the lock prevents duplicate work, but batching is what actually reduces database QPS.

C. **Cache rebuild exceeding lock TTL**: The double-check pattern (Step 3) handles this. After acquiring the lock, the code checks whether the cache was already rebuilt. If the version is already at or above the expected version, the rebuild is skipped. However, if the TTL expires and another container rebuilds the cache first, the original container may overwrite with stale data.

Use a global version counter. Before writing to the cache, check that the version matches the current global version. If another container already incremented the version, discard the result.

D. **Redis master failure with 200ms backlog**: The SETNX command was sent to the master but not yet replicated. The replica is promoted without the lock key. Container C1 believes it holds the lock. Container C2 attempts to acquire -- the key does not exist on the new master, so C2's SETNX succeeds. Both containers believe they hold the lock.

Impact: two containers may rebuild the same follower's cache concurrently. Since the rebuild is idempotent (same database queries produce the same result), the worst case is duplicate work rather than data corruption. The version check on the cache ensures both containers write the same data. During the 200ms failover window, the database impact is approximately 800 extra queries -- acceptable.

**What to watch for**: You must calculate the database QPS and recognize that simple per-follower locking does not solve the thundering herd -- the database queries themselves are the bottleneck. Batching is the primary mechanism for reducing QPS.

## Tier 3: Expert

These challenges have no single correct answer. There are multiple valid approaches, each with different trade-offs. You must argue for your chosen approach, acknowledge the alternatives, and explain why your approach is superior for the given context.

### Challenge 3.1: Build a Global Lock Service or Use Existing Infrastructure?

You are the principal engineer at a mid-stage fintech company that processes $50 million in daily transaction volume across 15 microservices running on 400 containers. The company currently uses Redis SETNX distributed locks for all coordination needs. Over the past six months, there have been three production incidents where duplicate lock grants caused double-processing of financial transactions, resulting in $80,000 in total losses.

Your VP of Engineering asks you to evaluate two approaches:

**Approach A -- Build a Lock Service on Raft Consensus**: Deploy a dedicated 5-node etcd or Raft-based lock service. Build a client library providing fencing tokens, automatic lease renewal, and monitoring. Estimated effort: 3 months for two engineers. Estimated infrastructure cost: $3,000 per month.

**Approach B -- Fix the Redis Lock Implementation**: Keep Redis but fix the three root causes: (1) add fencing tokens using a Redis atomic counter, (2) add a server-side lock reaper, and (3) implement auto-extending lock pattern. Estimated effort: 6 weeks for one engineer. Estimated infrastructure cost: $1,000 per month.

**Questions**:

A. Argue for Approach A. What makes the fintech context specifically demand a consensus-based lock service?

B. Argue for Approach B. Why is the Redis approach sufficient despite the past incidents?

C. Under what conditions would you switch from Approach B to Approach A? Define a specific threshold.

D. Propose a hybrid approach that combines elements of both.

**Model Answer**:

A. **Approach A -- Build argument**: The fintech context demands Approach A for three reasons. First, $80,000 in six months from lock-related incidents is material. At $160,000 annualized, the $36,000 annual infrastructure cost plus $120,000 in engineering time totals $156,000 in year one. The current Redis approach costs $160,000/year in incidents plus $12,000/year in Redis infrastructure plus approximately $40,000/year in engineering time investigating incidents. Total Redis cost: $212,000/year. Approach A breaks even in year two.

Second, the regulatory trajectory matters. Fintech companies processing $50 million daily are approaching the scale where auditors will ask about coordination mechanisms. A Raft-based lock service with fencing tokens is an auditable, verifiable system.

Third, the incident cost grows with transaction volume. At $50 million daily, the loss rate is 0.009 percent of volume. If volume doubles to $100 million, the loss doubles to $160,000 per six months. Approach A's infrastructure cost is fixed.

B. **Approach B -- Fix argument**: The $80,000 in losses came from three specific root causes, each fixable with targeted engineering. The fencing token fix alone prevents all three incidents. The estimated 6 weeks of engineering time at $40,000 plus $6,000 annual Redis cost totals $46,000 in year one -- $110,000 cheaper than Approach A.

The residual risk: if the Redis master fails during a lock write, the lock may not be replicated, and the new master may grant the same lock to another process. However, the fencing token prevents the stale writer from committing data to the database. The worst case is duplicate work rather than data corruption.

C. **Switch threshold**: When annualized lock-related incident cost exceeds $200,000, or when a regulatory audit requires a consensus-based coordination mechanism, adopt Approach A.

D. **Hybrid approach**: Use both. Transactions that move money between accounts (payment processing, settlement) use the etcd lock service with fencing tokens. Non-critical transactions (balance inquiries, transaction history) use the fixed Redis lock. The routing decision is made at the service level. This limits the etcd footprint to 3 critical operations. Engineering effort: 4 months with 1.5 engineers, costing $96,000.

**Trade-offs discussion**: The build argument prioritizes correctness and future-proofing. The fix argument prioritizes engineering efficiency. The hybrid approach acknowledges that not all locks are equal. The key insight: the cost of a lock failure depends on what the lock protects. Allocate the strongest lock service to the operations with the highest failure cost.

### Challenge 3.2: Should You Use Distributed Locks at All?

A ride-hailing company is designing a driver assignment system. When a rider requests a ride, the system must assign the nearest available driver. Two concurrent requests must not assign the same driver. The system operates on 50 containers across 3 AWS availability zones.

The engineering team is debating whether to use a distributed lock for driver assignment:

**Camp 1 -- Lock-based**: Use a per-driver Redis lock. When a ride request arrives, acquire the driver's lock, check availability, assign the ride, update the driver status, release the lock.

**Camp 2 -- Lock-free**: Use a database transaction: `UPDATE drivers SET status = 'assigned', current_trip_id = $1 WHERE status = 'available' AND driver_id = $2`. If the UPDATE updates 0 rows, the driver is already assigned -- pick the next available driver.

**Questions**:

A. Argue for Camp 1. What does the distributed lock provide that the database transaction does not?

B. Argue for Camp 2. Why is the distributed lock an unnecessary abstraction here?

C. The company plans to expand to 200 cities in 5 years, each with 500 to 5,000 drivers. Does your answer change with scale?

D. Propose a third approach that neither camp has considered. Explain the trade-offs.

**Model Answer**:

A. **Camp 1 -- Lock-based argument**: The distributed lock provides two things the database transaction does not. First, the lock provides a contention queue. When multiple requests contend for the same driver, the lock queues the requests. The first request acquires the lock. The second fails and moves to the next driver. The lock gives the system visibility into contention -- it can measure how many requests contended for each driver, track lock wait times, and adjust the driver pool.

Second, the lock provides cross-resource coordination. A ride request may need to lock multiple resources: the driver (for assignment), the rider's payment method (for hold), and the surge pricing zone (for fare calculation). A distributed lock can coordinate all three.

The lock-based approach outperforms under high contention. If a driver is in high demand, 50 containers may simultaneously attempt to assign that driver. Redis serializes this at sub-millisecond latency, faster than PostgreSQL row-level locking at 1-3 milliseconds.

B. **Camp 2 -- Lock-free argument**: The SQL UPDATE statement is atomic and provides exactly the same guarantee as the distributed lock. Two concurrent updates for the same driver cannot both succeed because the first update changes `status` from `available` to `assigned`, making the WHERE clause false for the second. The second update returns 0 rows affected, and the system moves to the next driver.

The cost of the additional complexity in Camp 1 includes: deploying and managing Redis for the lock service, writing and testing the lock client library, handling Redis failover, monitoring lock metrics, and debugging lock incidents. This is approximately 0.5 FTE per year in operational overhead.

For the ride-hailing use case, the lock-free approach is clearly superior because the assignment is a single operation, the database is the source of truth for driver status, the latency of a single UPDATE (under 2ms) is well within acceptable range, and the operational simplicity means fewer failure modes.

C. **Scaling analysis**: The lock-based approach becomes superior when the system needs to coordinate across multiple databases or data stores. When the company expands to 200 cities, driver assignment may involve city-specific databases, separate pricing services, and a global dispatch queue. A distributed lock at the application layer coordinates the operation across all of them.

The threshold: when the driver assignment operation needs to atomically coordinate across 3 or more distinct data stores or services, the distributed lock becomes the simpler approach.

D. **Third approach -- Queue-based**: Push the ride request onto a per-driver queue (Redis list or Kafka partition). A driver-side worker consumes from its own queue. Each ride request is added to the queue of the nearest available driver. The driver worker dequeues requests one at a time.

The queue-based approach eliminates lock contention entirely because each driver has their own queue. If 50 containers all try to assign the same driver, they all push to the same queue. The driver's worker processes them sequentially. No lock contention, no transaction conflicts, no serialization overhead.

**Trade-offs summary**:

| Approach | Latency | Complexity | Failure Mode | Scale Sweet Spot |
|---|---|---|---|---|
| Lock-free (DB txn) | 1-3ms | Low | DB connection loss releases row locks silently | Single data store, under 100 cities |
| Lock-based (Redis) | 0.5-2ms | Medium | Redis failover may grant duplicate locks | Cross-store coordination, high contention |
| Queue-based (per-driver) | 1-5ms | High | Driver queue blocks if worker crashes | Very high contention, large scale |

The recommendation depends on the company's current scale. At current scale (single database, 50 containers), the lock-free approach is the right choice. Plan to migrate to a queue-based approach when coordination spans multiple data stores or when per-city driver contention exceeds 500 requests per second per driver.

**What to watch for**: You must engage with both camps seriously. The queue-based approach is a strong third option that neither camp mentioned -- proposing it shows breadth of thinking. The scaling analysis must be specific rather than vague.

**Tier 1 Foundation**: Complete all three. Write out your answers fully and check them against the model answers. If your reasoning deviates from the model answer, identify whether your alternative is also valid or whether you missed a key concept.

**Tier 3 Expert**: Complete at least one of the two. There is no single correct answer. Your goal is to produce a well-reasoned argument that acknowledges the alternatives, identifies the trade-offs, and makes a specific recommendation tied to the context.
