## Company Case Studies

### Case Study 1: Stripe's Lock Service for Payment Processing

Stripe uses a distributed lock service as part of its payment processing infrastructure. The lock service coordinates access to financial resources such as merchant accounts, balance ledgers, and payment method tokens. Every payment authorization, capture, and refund requires a lock on the affected merchant account.

**Architecture**: Stripe's lock service is built on top of FoundationDB, a distributed database with strict serializable transactions. The lock records are stored in a FoundationDB table. Lock acquisition is a FoundationDB transaction that reads the current lock state and writes a new lock record atomically. The transaction is retried with exponential backoff if a conflict is detected.

**Key design decisions**:

- **Strict serializable isolation**: FoundationDB provides the strongest consistency model available in a distributed database. Each lock acquisition is a serializable transaction. This eliminates the clock drift problem entirely because the lock state is determined by the transaction order, not by wall clock time.
- **No TTL for financial locks**: Stripe does not use TTL-based lock expiry for financial locks. Instead, locks are explicitly released by the holder. If a holder crashes without releasing the lock, an operator must manually release it. This is acceptable because financial operations have a low crash rate and the manual release provides an extra safety check.
- **Audit trail**: Every lock operation is logged to an immutable audit trail. The audit trail is used for compliance and postmortem analysis. The audit log includes the fencing token, the resource identifier, the owner identifier, and the timestamp of every operation.

**Production scale**: Stripe's lock service handles approximately 50,000 lock acquisitions per second during peak hours. The P99 latency is 15 milliseconds for lock acquisition and 8 milliseconds for lock release. The service has been running in production since 2017.

**Lessons learned**:

- **FoundationDB transactions are expensive**: Each lock acquisition requires a FoundationDB transaction with strict serializable isolation. The transaction overhead adds 5 to 15 milliseconds of latency. Stripe optimized by batching multiple lock operations into a single transaction where possible, but this added complexity to the lock release logic.
- **Operator training is essential**: Manual lock release requires trained operators who understand the lock system. Stripe runs quarterly drills where operators practice identifying and releasing stuck locks. The drill includes a simulated failure scenario where a lock holder crashes mid-operation.
- **Integration with incident management**: When a lock is stuck for more than 5 minutes, an incident is automatically created in PagerDuty. The incident includes the resource identifier, the owner identifier, and the lock duration. The on-call engineer evaluates the lock and decides whether to release it manually.

### Case Study 2: Google Chubby -- The Original Production Lock Service

Google's Chubby lock service has been running in production since 2005. It is the foundation of Google's distributed systems infrastructure. Chubby provides a coarse-grained lock service with a simple file-system-like API.

**Architecture**: Chubby runs on a 5-node cluster using the Paxos consensus protocol. One node is the master. The master handles all read and write requests. Read requests are served from the master's in-memory cache. Write requests go through Paxos.

**Key design decisions**:

- **Coarse-grained locks**: Chubby is designed for locks that are held for minutes to hours. Fine-grained locks with high churn would overwhelm the Paxos write path.
- **File-system API**: Chubby exposes locks as files in a namespace. Clients open a file to acquire the lock. The lock is released when the file handle is closed.
- **Lease-based**: The master has a lease from the cluster. Clients have leases from the master. Lease durations are configurable, typically 10 to 30 seconds.
- **Event notifications**: Clients register for events (file modified, child added, lock acquired). Events are delivered via a callback mechanism.

**Production scale**: Chubby serves approximately 100,000 clients across Google's production infrastructure. The service handles millions of lock acquisitions per day with a P99 latency under 10 milliseconds for read operations and under 50 milliseconds for write operations.

**Lessons learned**:

- **Fencing tokens are essential**: Chubby assigns a monotonically increasing epoch number to each lock acquisition. The epoch number is included in all communications with the protected resource. Google's experience showed that without fencing tokens, delayed writers caused data corruption in approximately 1 in 100,000 lock acquisitions.
- **Session management is critical**: Chubby's session timeout mechanism was refined over years of production experience. The initial timeout of 10 seconds was too short for Java applications with GC pauses. The timeout was increased to 30 seconds with a configurable timeout multiplier.
- **Cache coherency is hard**: Chubby's read cache became a source of stale reads during master failover. The team implemented an invalidation protocol that flushed the cache on master change.

### Case Study 2: Uber's Ringpop -- Consistent Hashing with Locks

Uber built a distributed lock service called Ringpop that combines consistent hashing with lock management. Ringpop is used for sharded job scheduling and resource allocation across Uber's microservice fleet.

**Architecture**: Ringpop uses a consistent hashing ring to partition lock responsibility across nodes. Each node is responsible for a subset of resources. Lock operations are forwarded to the responsible node. The node uses a local in-memory lock table with Raft-based replication for fault tolerance.

**Key design decisions**:

- **Sharded lock management**: Each node manages locks for its assigned resources. The consistent hashing ring distributes the load evenly across nodes.
- **Local lock tables**: Lock state is stored in an in-memory hash map on the responsible node. The in-memory approach provides sub-millisecond lock acquisition latency.
- **Raft replication**: Lock state is replicated to a Raft group for fault tolerance. If the responsible node fails, the ring is rebalanced and lock state is recovered from the Raft log.

**Production scale**: Ringpop handles over 50,000 lock acquisitions per second across Uber's fleet. The P99 lock acquisition latency is under 5 milliseconds. The service has been running in production since 2016.

**Lessons learned**:

- **Shard rebalancing is expensive**: When nodes are added or removed from the ring, locks must be migrated to new nodes. The migration process caused temporary unavailability for some resources. Uber implemented a gradual rebalancing mechanism that migrates locks over a 5-minute window.
- **Head-of-line blocking**: A slow lock acquisition on a node blocked subsequent acquisitions for different resources on the same node. Uber implemented per-resource queues to prevent head-of-line blocking.
- **Clock drift detection**: Ringpop's TTL-based expiry was affected by clock drift. Uber implemented a clock drift detection mechanism that compares node clocks and adjusts TTLs accordingly.

### Case Study 3: Netflix's Cassandra-Based Lock Service

Netflix uses Apache Cassandra as a distributed lock service for resource management in its content delivery infrastructure. The lock service coordinates access to encoding resources, CDN bandwidth, and storage allocations.

**Architecture**: The lock service uses Cassandra's lightweight transactions (LWT) with the `IF NOT EXISTS` clause for atomic lock acquisition. Lock records are stored in a Cassandra table with a TTL column. The LWTs provide linearizable consistency within a single partition.

**Key design decisions**:

- **Leverage existing infrastructure**: Netflix already runs Cassandra for its primary data store. Using Cassandra for locks eliminated the need to operate a separate lock service.
- **TTL-based cleanup**: Cassandra's built-in TTL mechanism handles lock expiry. If the holder does not update the TTL, Cassandra automatically deletes the lock record.
- **Partitioned lock namespace**: Locks are partitioned by resource type. Each partition corresponds to a Cassandra partition key. The LWT provides linearizable consistency within the partition.

**Production scale**: The Cassandra-based lock service handles approximately 10,000 lock acquisitions per second during peak hours. The P99 latency is 15 to 40 milliseconds, depending on the Cassandra cluster load.

**Lessons learned**:

- **LWT overhead is significant**: Cassandra LWTs are 4 to 5 times slower than regular writes. The LWT requires a consensus round (Paxos) within the partition. For high-throughput scenarios, the LWT overhead became a bottleneck.
- **Partition hot spots**: Some resource types had much higher lock rates than others. The hot partitions caused uneven load on the Cassandra cluster. Netflix implemented a custom hash-based sharding within each partition to distribute the load.
- **TTL versus lease confusion**: The Cassandra TTL mechanism does not provide lease semantics. If the holder crashes, the lock is released after the TTL, but there is no mechanism to detect that the holder is still alive. Netflix implemented a client-side heartbeat that updates the TTL periodically.

### Case Study 4: Cloudflare's Distributed Lock Service for Durable Execution

Cloudflare built a distributed lock service as part of its Durable Objects platform. The lock service coordinates access to stateful objects across Cloudflare's global network of 330 data centers.

**Architecture**: Each Durable Object has a single storage shard. The shard is replicated across three data centers using a custom consensus protocol. The lock is part of the storage shard -- the shard leader holds the lock for the object. All write requests must go through the shard leader.

**Key design decisions**:

- **Lock integrated with storage**: The lock is not a separate service. It is an integral part of the storage shard. This eliminates the network round trip between the lock service and the storage system.
- **Global distribution**: Objects can be created in any data center. The storage shard is replicated across three geographically distributed data centers for fault tolerance.
- **Tunable consistency**: Cloudflare allows developers to choose between strong consistency (lock required) and eventual consistency (no lock) on a per-operation basis.

**Production scale**: The lock service handles over 1 million lock acquisitions per second across Cloudflare's global network. The P99 latency varies by geographic region, ranging from 10 milliseconds within the same data center to 200 milliseconds across continents.

**Lessons learned**:

- **Cross-datacenter latency is the bottleneck**: The consensus protocol across geographically distributed data centers adds 50 to 200 milliseconds of latency to each write. Cloudflare optimized the protocol to batch operations and use speculative execution.
- **Hot shards are inevitable**: Some objects (top customer accounts, popular API endpoints) generate much more lock traffic than others. Cloudflare implemented automatic shard splitting that divides hot shards into smaller shards.
- **Idempotency is critical**: Cloudflare's experience confirmed that no lock service provides perfect safety. All operations are designed to be idempotent, so that duplicate execution is harmless.

### Case Study 5: Stripe's Idempotency Keys -- Locks Without Locks

Stripe famously solves a class of distributed locking problems without explicit locks. When a client sends a payment request, it includes an idempotency key. Stripe's API checks if a request with that key has already been processed. If so, it returns the previous result. If not, it processes the request and stores the result keyed by the idempotency key.

**Design insight**: Idempotency keys eliminate the need for distributed locks in many API scenarios. The client is responsible for generating a unique key for each operation. The server is responsible for deduplicating requests with the same key. The result is exactly-once semantics without the complexity of distributed locking.

**Lessons learned**:

- **Idempotency is cheaper than locking**: An idempotency check is a single database read with a conditional insert. A distributed lock requires multiple network round trips, lease renewal, and fencing token validation.
- **Idempotency keys handle crashes**: If the server crashes after processing the request but before returning the response, the client retries with the same idempotency key. The server returns the cached result. No lock needed.
- **Lifetime matters**: Idempotency keys must be retained for a sufficient period to cover retry windows. Stripe retains them for 24 hours. After that, the same key can be reused for a different operation.

## Open Source Implementations

### Apache ZooKeeper

ZooKeeper is the most widely used open-source lock service. It provides a simple recipe for distributed locks using ephemeral sequential znodes. The recipe is documented in the ZooKeeper documentation and implemented in most ZooKeeper client libraries.

ZooKeeper's lock recipe is used by Apache Kafka for controller election, by Apache HBase for region server assignment, and by Apache Solr for leader election.

**Strengths**: Strong consistency via Zab protocol, session-based lock management, watch-based notification.

**Weaknesses**: Lower throughput than Redis, requires minimum 3 nodes, lock recipe is not built-in but implemented in client code.

### etcd

etcd provides a built-in concurrency package with distributed lock primitives. The `concurrency` package in etcd's Go client library provides a `Mutex` type that implements the lock algorithm using leases and keys.

etcd's lock is used by Kubernetes for controller leader election, by HashiCorp Consul for service coordination, and by many other cloud-native projects.

**Strengths**: Built-in lock primitive in Go client, Raft consensus, lease-based management, watch API.

**Weaknesses**: Go-specific concurrency package, other languages require custom implementation.

### Redis Redlock

Redis Redlock is an algorithm for distributed locking using multiple independent Redis nodes. The algorithm acquires the lock from a majority of N Redis nodes (typically 5). The Redlock algorithm was proposed by Redis creator Salvatore Sanfilippo.

Redlock is controversial in the distributed systems community. Martin Kleppmann published a critique arguing that Redlock does not provide the safety guarantees it claims because it depends on synchronized clocks. The debate highlights the fundamental challenge of distributed locking: without consensus, perfect safety is impossible.

### HashiCorp Consul

Consul provides distributed locking through its session mechanism. A session is created with a TTL. The session is associated with a key-value pair. The client holding the session holds the lock. Consul's lock is used for service mesh configuration and leader election.

**Strengths**: Integrated with service discovery, simple API, session-based.

**Weaknesses**: Lower throughput than dedicated lock services, session management overhead.

### Apache Curator

Curator is a Java library that simplifies ZooKeeper client development. It provides a higher-level API for distributed locks, leader election, and barriers. Curator's `InterProcessMutex` implements reentrant distributed locks. `InterProcessReadWriteLock` implements read-write distributed locks.

**Strengths**: Production-proven abstractions, handles connection loss and session expiry, implements the ZooKeeper lock recipe correctly.

**Weaknesses**: Java-specific, requires ZooKeeper infrastructure.

## Cloud Provider Managed Services

### Amazon ElastiCache for Redis

AWS provides managed Redis clusters that can be used for distributed locking. ElastiCache supports Redis SETNX with TTL for simple locks. For stronger guarantees, ElastiCache offers Redis Cluster mode with sharding and replication.

**Pricing**: Starting at $0.017 per hour for a cache.t2.micro instance. A production cluster with replication costs $50 to $200 per month.

**Limitations**: ElastiCache does not support Redlock natively. You must implement the Redlock algorithm in your application code. The managed service provides monitoring and automatic failover.

### Azure Cache for Redis

Azure provides managed Redis with similar capabilities. Azure Cache for Redis supports Redis SETNX with TTL. The Premium tier offers data persistence and clustering.

**Pricing**: Starting at $0.022 per hour for a Basic instance. A Premium cluster costs $100 to $500 per month depending on size.

**Limitations**: Same as ElastiCache -- no native Redlock support. Azure's managed service provides automatic failover with up to 10 seconds of downtime during failover events.

### Google Cloud Memorystore for Redis

Google Cloud provides managed Redis with support for SETNX and TTL-based locks. Memorystore offers standard and Premium tiers with high availability.

**Pricing**: Starting at $0.020 per hour for a Basic instance. A Premium instance with replication costs $60 to $250 per month.

**Limitations**: Memorystore's failover behavior is similar to other managed Redis services. Lock state may be lost during failover.

### Amazon MemoryDB for Redis

MemoryDB is a Redis-compatible, durable, in-memory database service. MemoryDB uses a multi-AZ transaction log to provide durable Redis storage. For distributed locks, MemoryDB provides stronger consistency than ElastiCache because writes are replicated to the transaction log before being acknowledged.

**Pricing**: Starting at $0.076 per hour for a db.r6g.large instance. A production deployment costs $150 to $600 per month.

**Advantages**: Multi-AZ durability reduces the risk of lock loss during failover. Write acknowledgement requires replication to the transaction log, providing stronger consistency than standard Redis replication.

### Amazon DynamoDB Lock Client

The DynamoDB Lock Client is an open-source Java library that uses DynamoDB as a distributed lock service. The library uses DynamoDB's conditional update API for atomic lock acquisition. Lock records include a lease duration, owner identifier, and fencing token.

**Pricing**: Pay per DynamoDB read and write capacity units. A lock service handling 10,000 acquisitions per second costs approximately $50 to $200 per month in DynamoDB capacity.

**Advantages**: Fully managed service with no servers to operate. Multi-AZ replication provides high availability. Conditional updates provide atomic check-and-set semantics.

**Limitations**: DynamoDB's write throughput is limited by the provisioned capacity. A sudden spike in lock acquisitions can be throttled. The fencing token mechanism requires application-level implementation.

## Lessons Learned

### Lesson 1: Always Use Fencing Tokens

Every production incident involving distributed locks traces back to a delayed writer. The lock holder pauses, the lock expires, a new holder acquires the lock, and the original holder resumes writing with stale authority. Fencing tokens eliminate this class of failure.

The fence token must be checked on the resource side, not the client side. If the client checks the fencing token, a GC pause between the check and the write allows stale data through. The resource must reject any write with a token lower than the last-seen token.

### Lesson 2: Plan for Lock Service Failure

The lock service will eventually fail. A network partition will separate clients from the lock service. A software bug will cause the lock service to crash. A configuration error will misconfigure the lock service.

The application must handle lock service failure gracefully. Options include:

- **Fail open**: Continue operations without the lock, accepting the risk of data corruption.
- **Fail closed**: Stop all operations until the lock service is restored, accepting the risk of unavailability.
- **Degraded mode**: Use a local lock approximation or best-effort mutual exclusion until the lock service is restored.

The choice depends on the cost of data corruption versus the cost of unavailability.

### Lesson 3: Monitor Everything

Undetected lock failures cause silent data corruption. Every lock operation must be monitored. The critical metrics are acquisition latency, contention rate, lease expiry rate, and wait queue depth.

Alert thresholds must be set based on baseline measurements. A 10 percent increase in acquisition latency may indicate a problem. A sudden spike in lease expirations may indicate a network partition or lock service crash.

### Lesson 4: Test Under Realistic Conditions

Testing a distributed lock under ideal conditions with no network delay, no clock drift, and no process pauses does not validate the lock's safety properties. The lock must be tested under fault injection.

Fault injection scenarios should include network partitions, process crashes, simulated GC pauses, clock skew, and resource exhaustion. The test must verify that no two processes ever hold the lock simultaneously, even under these fault conditions.

### Lesson 5: The Simpler the Better

The most reliable distributed lock implementations are the simplest. Chubby uses a 5-node Paxos cluster with a file-system API. The implementation is straightforward and well-understood. The team resisted the temptation to add features.

Complex lock algorithms such as Redlock and custom consensus protocols introduce failure modes that are hard to analyze. Stick with well-understood protocols (Raft, Paxos, Zab) and simple APIs (acquire, release, renew).

### Lesson 6: Idempotency Is Your Safety Net

No lock service provides perfect safety. Even with fencing tokens, there is always a residual risk. The ultimate safety net is idempotency -- designing operations so that executing them multiple times produces the same result as executing them once.

Idempotency is not always possible. Debit operations, append operations, and resource allocation are inherently non-idempotent. But for operations that can be made idempotent, idempotency eliminates the cost of lock failures.

## Production Runbook Example

The following runbook outlines the steps for diagnosing and resolving a distributed lock incident in production.

### Incident: High Lock Acquisition Latency

**Symptoms**: P99 lock acquisition latency exceeds 100 ms for more than 5 minutes.

**Step 1**: Check the lock service cluster health.
```
etcdctl endpoint status --cluster
```
Verify that all nodes are reachable and the leader is healthy.

**Step 2**: Check the lock service resource usage.
- CPU utilization: Should be below 80 percent.
- Memory utilization: Should be below 80 percent.
- Disk I/O: Should be below 70 percent of capacity.
- Network bandwidth: Should be below 50 percent of capacity.

**Step 3**: Check the lock registry size.
- A large registry (more than 100,000 active locks) may indicate leaked locks.
- Check for holders that have crashed without releasing their locks.
- The lease manager should eventually clean up expired locks.

**Step 4**: Check the client-side logs.
- Are clients experiencing connection errors to the lock service?
- Are clients retrying connection establishment?
- Are clients reporting high GC pause durations?

**Step 5**: Check for network issues.
- Is there packet loss between clients and the lock service?
- Is there asymmetric network latency?
- Are any network load balancers or proxies misconfigured?

**Step 6**: If all else fails, escalate to the infrastructure team.
- The issue may be at the network, DNS, or cloud provider level.
- Prepare a timeline of events and relevant metrics for the escalation.

### Incident: Duplicate Lock Grants

**Symptoms**: Data corruption detected. Investigation reveals two processes both believed they held the same lock simultaneously.

**Step 1**: Immediately stop all lock acquisition and release operations.
- The affected system should enter a degraded mode where no critical operations are performed.

**Step 2**: Identify the root cause.
- Was there a network partition?
- Was there a lock service failover?
- Was there a clock drift event?
- Was there a GC pause on the lock holder?

**Step 3**: Assess the data corruption.
- Identify the affected resources.
- Determine the extent of the damage.
- Initiate data recovery procedures.

**Step 4**: Implement a fix.
- If the root cause was a network partition, consider implementing a partition detection mechanism.
- If the root cause was a GC pause, consider increasing the TTL or implementing fencing tokens.
- If the root cause was clock drift, consider switching to a consensus-based lock service.

**Step 5**: Verify the fix.
- Run the lock safety test to verify that no two processes can hold the lock simultaneously under the same fault conditions.
- Run the fault injection test to verify the fix under realistic conditions.

**Step 6**: Document the incident.
- Create a postmortem with the root cause, impact, and fix.
- Share the findings with the team.
- Update the runbook with the new knowledge.

## Cloud-Native Patterns

### Kubernetes Leader Election

Kubernetes uses distributed locks for leader election in its controller manager, scheduler, and other control plane components. The lock is implemented as a `Lease` object in the Kubernetes API server. The component that holds the lease is the leader.

```yaml
apiVersion: coordination.k8s.io/v1
kind: Lease
metadata:
  name: kube-controller-manager
  namespace: kube-system
spec:
  holderIdentity: controller-manager-pod-1
  leaseDurationSeconds: 15
  renewTime: "2026-07-03T12:00:05Z"
  acquireTime: "2026-07-03T11:59:50Z"
  leaseTransitions: 3
```

Kubernetes's approach is notable for its simplicity: the lease is just another API object stored in etcd, with no separate lock service. The controller manager's leader election code is approximately 200 lines of Go.

### Service Mesh Coordination

Istio uses distributed locks for configuration coordination across its control plane. The Istio pilot agents use etcd-backed locks to ensure that configuration updates are applied in a consistent order. The locks prevent race conditions where two agents simultaneously update the same Envoy proxy configuration.

### Database Migration Coordination

Distributed locks are commonly used to coordinate database schema migrations across multiple application instances. Only one instance should run the migration at a time. The lock key is the database migration version. This pattern is used by Flyway, Liquibase, and Alembic.

```python
def run_migration(migration_version):
    lock_key = f"db_migration:{migration_version}"
    lock = lock_service.acquire(lock_key, ttl=600)
    if not lock:
        logger.info("Migration %s running on another instance", migration_version)
        return
    try:
        apply_migration(migration_version)
        record_migration(migration_version)
    finally:
        lock_service.release(lock)
```

### Cache Warming Coordination

In a distributed cache, multiple replicas can simultaneously detect a cache miss and attempt to rebuild the cache entry. A distributed lock ensures that only one replica rebuilds the cache. The others wait for the result or serve stale data.

This is one of the most common use cases for distributed locks in production. The lock TTL is typically short, 10 to 30 seconds, because the cache rebuild operation is fast. The cost of a duplicate cache rebuild is low -- wasted CPU cycles but no data corruption -- so a simple Redis SETNX lock is adequate.
