## Core Tension

The fundamental tension in distributed lock design is the trade-off between safety and liveness. Safety means never granting the lock to two processes simultaneously. Liveness means always granting the lock to a requesting process when no one holds it.

Safety and liveness are at odds because of the unreliable nature of distributed systems. A process that holds a lock may be alive but unreachable. The lock service must decide: do I wait for the process to become reachable again (safety), or do I grant the lock to another process (liveness)?

Choosing safety means the system can become unavailable. If the lock holder is unreachable and the lock service never releases the lock, all other processes stall. The system is safe but not live.

Choosing liveness means the system can violate mutual exclusion. If the lock service releases the lock prematurely, two processes may hold it simultaneously. The system is live but not safe.

This tension is not resolvable in theory. Every distributed lock implementation makes a choice between safety and liveness, and each choice has consequences. The choice manifests in the TTL value, the lease renewal mechanism, and the behavior during network partitions.

### Decision Framework

Use the following decision framework to select the appropriate lock implementation:

1. **If you can tolerate rare lock violations**: Choose Redis SETNX. It is the fastest and simplest option. The risk of duplicate lock grants is approximately 1 in 1,000 acquisitions under normal operation, increasing to 1 in 100 during failover events.

2. **If you need strong consistency but already use Redis**: Choose Redlock. Redlock acquires the lock from a majority of 5 independent Redis nodes. It reduces the risk of duplicate grants but does not eliminate it. The Redlock paper has been criticized for its dependence on synchronized clocks.

3. **If you need strong consistency and can accept higher latency**: Choose ZooKeeper or etcd. Both provide linearizable consistency through consensus protocols. The latency penalty of 5 to 15 milliseconds is acceptable for most coordination workloads.

4. **If you already have a relational database and the lock rate is low**: Choose database locks. The acquisition latency of 10 to 100 milliseconds is acceptable for operations that run once per minute or less.

5. **If you want a fully managed service with zero operational overhead**: Choose DynamoDB Lock Client. The managed service handles failover, replication, and scaling automatically. The trade-off is vendor lock-in and higher per-operation cost.

### Behavior During Partition

| Lock Service | Behavior During Partition | Safety Impact |
|---|---|---|
| Redis (single master) | Master in majority partition continues serving. Locks in minority partition are unavailable. | If master fails and replica promotes, locks may be lost. Duplicate grants possible. |
| Redlock | Majority of 5 nodes can serve locks. Minority partition cannot acquire locks. | If clock drift causes time to move backward, duplicate grants possible. |
| ZooKeeper | Writes blocked if quorum lost. Reads return stale data if not using sync reads. | Linearizable reads require quorum. Stale reads can cause incorrect lock state observation. |
| etcd | Writes blocked if quorum lost. Linearizable reads also require quorum. | Serialized access to lock registry is maintained. No duplicate grants. |
| Database (single) | Locks unavailable if database unreachable. | No duplicate grants because the database is the single source of truth. |

## Failure Modes

A distributed lock system can fail in many ways. Understanding each failure mode is essential for building a reliable system.

### Failure Mode 1: Delayed Writer

A process acquires a lock, starts processing, and then pauses for a long time due to a garbage collection pause, a page fault, or a scheduling delay. The lock TTL expires. Another process acquires the lock. The original process resumes and writes to the protected resource, believing it still holds the lock.

**Impact**: Two processes write to the same resource concurrently. Data corruption is guaranteed if the writes are not idempotent.

**Probability**: Low for short GC pauses (under 1 second). High for long GC pauses (over 10 seconds), which can occur in Java applications with large heaps.

**Mitigation**: Fencing tokens on the protected resource. The resource rejects writes with stale tokens. The lock holder must check the token before each write operation.

### Failure Mode 2: Session Expiry Race

A ZooKeeper client holds a lock through an ephemeral znode. The client's ZooKeeper session times out due to a network hiccup. The ephemeral znode is deleted. A new client acquires the lock. Meanwhile, the original client's network recovers, but the client does not detect the session expiry until it tries to use the session.

**Impact**: The original client continues processing, believing it holds the lock. The new client also processes. Both write to the protected resource.

**Probability**: High. ZooKeeper session timeouts are typically 10 to 20 seconds. Any network disruption longer than the timeout triggers this race.

**Mitigation**: Use a session watch to detect session expiry. The client must abort the critical section when the session expires. Fencing tokens provide a second line of defense.

### Failure Mode 3: Clock Drift

A TTL-based lock service uses the server's clock to determine when a lock expires. If the server's clock drifts forward, locks expire early. If the clock drifts backward, locks expire late.

**Impact**: Clock drift forward causes premature lock expiry. The lock is released while the holder is still processing. Another process acquires the lock. Two processes hold the lock simultaneously. Clock drift backward causes delayed lock expiry. The lock is held longer than intended, causing unnecessary blocking.

**Probability**: Medium. NTP keeps clocks within 1 to 10 milliseconds of the reference time under normal operation. Clock drift becomes significant when NTP is misconfigured or when the server lacks a hardware clock.

**Mitigation**: Use a consensus-based lock service such as ZooKeeper or etcd that does not rely on clock time for lease expiry. Use session-based expiry instead of TTL-based expiry.

### Failure Mode 4: Split-Brain Lock Service

A lock service cluster experiences a network partition. The cluster splits into two groups, each believing it is the majority. Both groups accept lock acquisition requests. Two processes acquire the same lock from different groups.

**Impact**: Complete safety violation. Two processes hold the same lock simultaneously. The protected resource receives conflicting writes.

**Probability**: Very low for consensus-based systems. Raft and Zab are designed to prevent split-brain. The probability is non-zero for marginally configured clusters with an even number of nodes.

**Mitigation**: Use an odd number of nodes (3, 5, 7). Configure the cluster with strict quorum requirements. Monitor the cluster for split-brain conditions.

### Failure Mode 5: Resource Exhaustion

The lock service runs out of memory, disk space, or file descriptors. The lock service cannot process new lock acquisition requests. Existing locks may be expired or released unexpectedly.

**Impact**: The lock service becomes unavailable. Existing lock holders may lose their locks if the lease manager crashes. New lock acquisitions fail.

**Probability**: Medium. Memory exhaustion is the most common cause. etcd's default database size limit of 2 GB can be reached within days for high-throughput systems.

**Mitigation**: Set resource limits on the lock service process. Configure automatic compaction for etcd. Monitor resource usage and alert before exhaustion. Implement graceful degradation in the client library.

### Failure Mode 6: Client-Side Renewal Failure

The client fails to renew the lease before the TTL expires. The renewal may fail due to a network issue, a client-side timeout, or a client crash.

**Impact**: The lock is released while the client is still processing. Another process acquires the lock. Two processes operate concurrently.

**Probability**: High. Network issues are common in distributed systems. The client must handle renewal failures gracefully.

**Mitigation**: The client should retry renewal with exponential backoff. The renewal interval should be one-third of the TTL, providing three chances to renew before expiry. The client should detect renewal failure and abort the critical section.

### Failure Mode 7: Lock Release by Wrong Owner

A bug in the client code or a malicious actor causes a lock release request that specifies a different owner than the current holder. The lock service incorrectly processes the release, removing the lock record.

**Impact**: The lock is released while the real holder is still processing. Another process acquires the lock. The real holder writes to the protected resource without the lock.

**Probability**: Low. The Lua script or compare-and-delete operation prevents this if implemented correctly.

**Mitigation**: Always verify the owner in the release operation. Use a Lua script that compares the owner before deleting. Use a unique owner identifier that cannot be guessed.

## Anti-Patterns

These are common design mistakes that engineers make when implementing distributed locks.

### Anti-Pattern 1: Using a Lock as a Mutex

A distributed lock is not a mutex. A mutex protects shared memory within a single process. A distributed lock protects shared resources across a network. The latency difference is three orders of magnitude: a mutex acquisition takes nanoseconds, while a distributed lock acquisition takes milliseconds.

Using a distributed lock where a mutex would suffice adds unnecessary latency and complexity. If the resources being protected are in the same process memory, use a mutex.

### Anti-Pattern 2: Infinite TTL

Setting the TTL to a very large value or disabling it entirely to prevent the lock from expiring while the holder is still processing. This creates a permanent blockage if the holder crashes. The system becomes unavailable until the lock is manually released.

The correct approach is to set the TTL to the maximum expected critical section duration plus a safety margin, and use lease renewal for long-running operations.

### Anti-Pattern 3: Mixing Critical and Non-Critical Locks

Using the same lock service for both critical locks such as financial transactions and non-critical locks such as cache rebuilds. A failure in the non-critical lock's operation can bring down the entire lock service, affecting critical operations.

The correct approach is to use separate lock service instances for critical and non-critical locks. If this is not possible, use different resource namespaces with different TTL and priority settings.

### Anti-Pattern 4: Ignoring Fencing Tokens

Using a distributed lock without fencing tokens on the protected resource. The lock provides probabilistic safety at best. Any GC pause, network delay, or clock drift can cause two processes to hold the lock simultaneously.

The correct approach is to always implement fencing token checks on the protected resource. The fencing token is the only mechanism that guarantees safety in the presence of delays.

### Anti-Pattern 5: Locking Too Broadly

Locking a large resource such as an entire database table when a fine-grained lock such as a single row would suffice. Broad locking reduces concurrency and increases contention. The system becomes serialized unnecessarily.

The correct approach is to start with fine-grained locks and escalate only when contention or overhead becomes a problem. Measure the contention rate before deciding to broaden the lock granularity.

### Anti-Pattern 6: Ignoring Clock Drift

Using a TTL-based lock service without considering clock drift. The TTL is measured by the lock service's clock, but the lock holder's clock may be different. If the clocks drift, the TTL behavior becomes unpredictable.

The correct approach is to use a consensus-based lock service where lease expiry is based on session state rather than absolute time. If you must use a TTL-based service, configure NTP carefully and monitor clock drift.

### Anti-Pattern 7: Single-Node Lock Service

Running the lock service on a single node. The node is a single point of failure. If the node goes down, the entire lock system is unavailable. No distributed system can make progress without a lock service.

The correct approach is to run the lock service on a cluster of at least 3 nodes. The cluster tolerates 1 node failure. For production systems, use 5 nodes to tolerate 2 node failures.

### Anti-Pattern 8: No Monitoring

Deploying a distributed lock system without monitoring. The lock acquisition latency, contention rate, and lease expiry rate are unknown. Problems are discovered only when they cause production incidents.

The correct approach is to expose metrics for all lock operations and configure alerts for abnormal conditions. The monitoring system should track acquisition latency, contention rate, lease expiry rate, and wait queue depth.

### Anti-Pattern 9: Holding a Lock Across an Unreliable Network Call

Holding a lock while making a network call to an external service that can hang or fail. If the call hangs, the lock blocks all other processes until the TTL expires. If the call fails after the TTL expires, the lock is released while the operation is still in progress.

The correct approach is to hold the lock only for local operations. If you must make a network call, use a short TTL with automatic retry, and implement idempotency on the external service side.

### Anti-Pattern 10: Polling Without Backoff

A process that fails to acquire a lock immediately retries in a tight loop. Each retry sends a request to the lock service, which must check the lock registry, find the lock is still held, and return a failure response. The tight loop generates unnecessary load.

The correct approach is to use exponential backoff with jitter. Start with a 100-millisecond delay. Double the delay on each retry up to a maximum of 5 seconds. Add random jitter of plus or minus 25 percent.

## When NOT to Use

Distributed locks are a powerful tool, but they are not the right solution for every problem. Here are the scenarios where you should not use a distributed lock.

### You Do Not Need Mutual Exclusion

If the operation is idempotent, you do not need a lock. An idempotent operation produces the same result regardless of how many times it is executed. Processing the same batch of payments twice is safe if the payment gateway deduplicates. Processing the same file twice is safe if the output is a set of unique records.

Idempotency is often cheaper and simpler than distributed locking. Before implementing a lock, ask whether the operation can be made idempotent.

### Optimistic Locking Is Sufficient

If the contention rate is low, below 5 percent, optimistic locking is more efficient than pessimistic locking. The process reads the current state, performs the operation locally, and attempts to commit with a version check. If the version changed, the operation fails and retries.

Optimistic locking avoids the overhead of lock acquisition, lease renewal, and lock release. It is ideal for scenarios where conflicts are rare.

### The Resource Is Immutable

If the resource is immutable -- write once, read many -- you do not need a lock. The first write sets the value. Subsequent writes are rejected because the resource already exists. The resource itself enforces the mutual exclusion.

This pattern is used in event sourcing, where each event is appended to an immutable log. The append operation is atomic, and the log maintains the order of events.

### The Operation Is Atomic

If the operation can be made atomic at the storage layer, you do not need a lock. Atomic operations include Redis `INCR`, database `UPDATE ... WHERE version = X`, and conditional writes to object storage.

Atomic operations are simpler and faster than distributed locks. They are the preferred approach for single-resource operations.

### You Need High Throughput

If the throughput requirement exceeds 10,000 operations per second for a single resource, a distributed lock is the wrong tool. The lock serializes access to the resource, limiting throughput to the inverse of the acquisition latency.

For high-throughput scenarios, consider partitioning the resource so that each partition has its own lock, or use an optimistic approach that avoids locking entirely.

### The System Is Eventually Consistent

If the system is designed for eventual consistency, adding a distributed lock introduces a strongly consistent component that conflicts with the system's design goals. The lock becomes a bottleneck and a single point of failure.

Eventual consistency systems should use application-level conflict resolution, such as last-writer-wins with conflict-free replicated data types (CRDTs), rather than distributed locks.

### You Have a Single Server

If the system runs on a single server, use a local mutex. A distributed lock adds unnecessary latency and complexity. The network round trip to the lock service is wasted when the lock could be acquired in nanoseconds from shared memory.

## Summary of Trade-offs

The following table summarizes the key trade-offs you face when designing a distributed lock system.

| Trade-off | Choose Left | Choose Right |
|---|---|---|
| Safety vs. Liveness | ZooKeeper / etcd (CP) | Redis (AP) |
| Latency vs. Consistency | Redis (1 ms) | ZooKeeper / etcd (5-15 ms) |
| Throughput vs. Safety | Redis (50,000 ops/s) | ZooKeeper / etcd (10,000 ops/s) |
| Complexity vs. Simplicity | Database (simple, no new infra) | Multi-node consensus (complex, reliable) |
| Cost vs. Reliability | Single-node (cheap, fragile) | 5-node cluster (expensive, resilient) |
| Availability vs. Partition Tolerance | Redis (always available) | ZooKeeper (blocks during partition) |
| Self-managed vs. Managed | etcd cluster (full control) | DynamoDB Lock Client (zero ops) |

The key insight is that there is no single best distributed lock implementation. The right choice depends on your specific requirements for safety, liveness, latency, throughput, and operational complexity. Measure your workload, understand your failure modes, and choose accordingly. When in doubt, start with the simplest implementation that meets your requirements and evolve as you learn more about your system's behavior under production load.
