## Tier 1 — Foundation (5 minutes each)

### Challenge 1.1: Cache Hit Ratio Analysis

You manage a CDN for a news website. The cache hit ratio is 62 percent. The engineering team wants to increase it to 90 percent. The origin is in us-east-1. Most readers are in North America and Europe.

**Explain:**
- Three possible root causes of the low hit ratio
- What data you would collect to identify which root cause is responsible
- The change you would make for each root cause

**Evaluation criteria:**
- Identifies that cache keys may include unique query parameters (session IDs, timestamps, tracking params)
- Identifies that TTLs may be too short (5-minute TTLs on news articles that do not change for hours)
- Third cause could be: content is personalized per user (Auth headers in cache key), unversioned URLs (new articles have same URL pattern as old), or mixed cacheable/non-cacheable endpoints

**Model answer:**

The three most likely root causes are:

1. **Query parameter pollution:** If the news site includes tracking parameters (`?utm_source=twitter`, `?ref=recommended`) in URLs, each variant creates a unique cache entry. Fix: configure the CDN to strip tracking parameters from the cache key.

2. **TTL too short:** If the origin sets `Cache-Control: max-age=60`, the CDN revalidates every 60 seconds. For news articles, a 1-hour TTL is safe because articles do not change after publication (except for corrections, which are rare). Fix: set `Cache-Control: max-age=3600` on article pages.

3. **Cache key includes cookies or auth headers:** If the CDN includes `Cookie` or `Authorization` headers in the cache key, every logged-in user creates unique entries. Fix: bifurcate the cache — authenticated users bypass cache entirely (low volume), anonymous users share a single cache key.

**Data to collect:** Enable CDN logs with cache status per request. Group requests by URL pattern and check which patterns have the highest miss rate. Check for query parameter variability. Check origin response headers for Cache-Control values.

### Challenge 1.2: CDN vs No CDN Decision

Your startup has 5,000 daily active users, all in the United States. The app serves 500KB of static assets and 50KB of dynamic JSON per page load. Origin is in us-east-1 with 99.5 percent uptime. Average page load time is 500ms.

**Explain:**
- Would you add a CDN? Why or why not?
- What would be the expected improvement in page load time?
- What is the cost trade-off?

**Evaluation criteria:**
- Recognizes that 5,000 DAU all in one country does not justify a CDN
- Identifies that 500ms load time is acceptable (but improvable)
- Considers cost of CDN ($20-200/month) vs benefit (latency improvement from 500ms to 200-300ms)
- Mentions alternative: Nginx reverse proxy cache in the same data center

**Model answer:**

At 5,000 daily active users all in the US, a CDN is not the right investment. The latency benefit would be modest — users are 30-80ms from us-east-1, so a CDN would improve load time from 500ms to approximately 350-400ms. The improvement is primarily in static asset delivery (images, CSS, JS), which total 500KB. Dynamic JSON (50KB) must still fetch from origin.

A better investment: add an Nginx reverse proxy cache in the same data center as the origin. This caches static assets with 1-year TTLs and costs nothing extra beyond the existing server. This would improve load time to approximately 400ms. If latency is still a concern, the bottleneck is likely the dynamic API, not static assets — optimize that first (response compression, database query tuning).

Cache deeper infrastructure and optimize the dynamic path before adding bandwidth costs with a CDN.

### Challenge 1.3: TTL Selection

You have three content types on your platform. Assign appropriate TTLs to each:

| Content | Update Frequency | User Impact of Staleness |
|---------|-----------------|--------------------------|
| Product catalog JSON | Every 6 hours | Low (prices slightly outdated) |
| User avatar images | Never (immutable) | None (old versions acceptable) |
| Live sports scores | Every 5 seconds | High (must be real-time) |

**Explain:** Your TTL choices for each content type, including the cache-control headers.

**Evaluation criteria:**
- Product catalog: TTL of 1-6 hours with stale-while-revalidate of 1 hour
- User avatars: TTL of 1 year (versioned), or max-age 31536000
- Live sports: No CDN caching (max-age=0, s-maxage=0, must-revalidate) — or use SSE/WebSocket directly

**Model answer:**

| Content | TTL | Cache-Control Header | Rationale |
|---------|-----|---------------------|-----------|
| Product catalog | 4 hours | `Cache-Control: public, max-age=14400, stale-while-revalidate=3600` | Updates every 6 hours, so 4-hour cache ensures freshness while absorbing origin delays |
| User avatars | 365 days | `Cache-Control: public, immutable, max-age=31536000` | Immutable content. `immutable` directive tells browser never to revalidate. Version URLs for mutable avatars. |
| Live scores | 0 seconds | `Cache-Control: no-cache, no-store, must-revalidate` | Real-time data cannot be cached. Use Server-Sent Events or WebSocket to push updates directly. No CDN benefit. |

## Tier 2 — Design (15 minutes each)

### Challenge 2.1: Design a CDN for Video Streaming

**Problem:** A video streaming platform has 10 million monthly active users. Average video bitrate is 5 Mbps. Videos are 10 minutes long average. Users expect under 2 seconds of startup delay. Users are distributed: 40 percent North America, 30 percent Europe, 20 percent Asia, 10 percent rest of world. Budget: $500,000/month for content delivery.

**Design:**
- CDN architecture (POP locations, tiers, storage)
- Pre-positioning strategy (which content is pre-cached? how?)
- Caching strategy for adaptive bitrate streaming (HLS/DASH)
- Cost and capacity math

**Rubric:**

| Criterion | Poor | Good | Excellent |
|-----------|------|------|-----------|
| POP placement | "All major continents" | Lists specific regions with traffic rationale | Calculates number of POPs from bandwidth math |
| Pre-positioning | "Cache everything" | Positions top 20% of content (Zipf distribution) | Pre-positions based on regional popularity (content A may be popular in US but not in Asia) |
| ABR caching | "Cache all bitrates" | Caches segments independently, optimized for first few seconds | Caches first 10 seconds of every video at all bitrates (startup), rest on demand |
| Cost | No budget consideration | Stays within budget with margin | Optimizes per-region bandwidth costs (cheaper transit in Europe than Asia) |

**Model answer:**

**POP placement:** I need POPs within 500km of users to achieve under 20ms latency for the manifest file and first video segment. Minimum POPs: 8 in North America (NYC, IAD, ORD, DFW, LAX, SEA, ATL, MIA), 5 in Europe (LON, FRA, PAR, AMS, MIL), 4 in Asia (TYO, SIN, HKG, BOM), 1 in Oceania (SYD), 1 in South America (GRU). Total: 19 POPs.

**Storage:** Each POP stores the most popular content. With Zipf distribution, the top 5 percent of videos serve 80 percent of requests. For 10M users: if the catalog is 100,000 videos, store the top 5,000 (at 3 bitrates each = 15,000 segments per video segment = 5 bitrates × 10 min × 60s/10s segments = 300 segments per video; 5,000 × 300 = 1.5M segments; at 10MB per segment average = 15TB per POP). Each POP needs 15TB of NVMe storage.

**Pre-positioning:** Use a popularity model. Track per-region watch data. Pre-position the top 5 percent of content for each region during off-peak hours (3-6 AM local time). First-request latency for less popular content: accept 1-2 second startup (fetch from regional cache or origin).

**Startup optimization:** Pre-position the first 10 seconds of every video at all bitrates (this creates 3-second cache window at 5 Mbps). This ensures instant startup. The remaining content is cached on demand after the first user request (subsequent users get full cache).

**Cost math:**
- Bandwidth: 10M users × 1 hour/week average = 10M hours/week × 5 Mbps = 50,000 Gbps-week = approximately 200,000 TB/month
- CDN pricing: $0.02/GB average (negotiated) = $4,000,000/month
- With 80% cache hit ratio (pre-positioning): 200,000 × 0.2 = 40,000 TB from origin → $800,000/month. But pre-positioned traffic is intra-POP and costs nothing.
- With pre-positioning effectively eliminating transit for the top 80%: 200,000 × 0.2 = 40,000 TB from origin at $0.02 = $800,000. Plus pre-positioning storage costs: 19 POPs × 15TB × $0.10/GB/month = $28,500. Total: ~$830,000/month. Over budget.

**Optimization:** Negotiate CDN pricing to $0.015/GB, reducing to $600,000/month. Accept 70% cache hit ratio (less aggressive pre-positioning). Final: $420,000/month. Within budget.

### Challenge 2.2: Design a Cache Invalidation Strategy

**Problem:** You operate a CDN for a hotel booking platform. Prices change every 15-30 minutes. Availability changes in real time (last room booked). SEO-optimized HTML pages (cached for performance) must reflect current prices and availability. The platform has 50 million monthly users across 200 countries.

**Design:**
- Invalidation strategy that balances price freshness with performance
- How to handle the "last room" scenario (decisions based on stale availability cause overbooking)
- How to handle global price changes (a hotel chain changes prices for 10,000 hotels at once)
- Monitoring and alerting for invalidation failures

**Rubric:**

| Criterion | Poor | Good | Excellent |
|-----------|------|------|-----------|
| Price freshness | "Short TTLs (30s)" | Tiered TTLs with stale-while-revalidate | Hybrid: CDN for HTML shell, AJAX for dynamic pricing (ESI at edge) |
| Overbooking prevention | "Jus don't cache availability" | Short TTL (5s) on availability data + conditional requests | Edge Side Includes (ESI): cache HTML page at edge, fetch availability fragment per-request (not cached) |
| Bulk price change | "Purge each URL" | Use a purge queue with prioritization with a CDN API tag-based purge | Tag-based purge: set Cache-Tag header on each object, purge by tag (one API call for 10,000 hotels) |
| Monitoring | "Check logs" | Real-time invalidated-content-served metric with alert at 0.1% | Synthetic user monitoring: bot that checks prices on cached pages vs origin every 60s, alert on mismatch |

**Model answer:**

The key insight is that hotel prices and availability have different staleness tolerances. Price can be 15 minutes stale. Availability must be real-time.

**Architecture:** Use Edge Side Includes (ESI). The CDN caches the entire HTML page for 1 hour. Within the page, two fragments are injected by the edge:
- `<!--# include virtual="/fragment/price?hotel=123" -->` — cached for 15 minutes
- `<!--# include virtual="/fragment/availability?hotel=123" -->` — never cached (bypass to origin)

This gives:
- HTML page loads in 15ms (fully cached)
- Price loads within 15ms to 150ms (cached at edge 85% of time)
- Availability loads in 100-300ms (direct to regional API)

**Bulk price changes:** Use CDN tag-based purging. Each hotel's data gets a `Cache-Tag: hotel_123` response header. When a chain updates prices for 10,000 hotels, the platform sends one API call:

```
POST /purge/tag
Host: cdn.example.com
Content-Type: application/json

{"tags": ["chain_marriott"]}
```

This invalidates all objects tagged with `chain_marriott` in under 10 seconds globally.

**Last-room scenario:** The availability fragment bypasses CDN cache entirely. Every page load fetches real-time availability from the booking engine. This adds 100-200ms to page load (acceptable because it is a single fragment, not the full page) but guarantees zero overbooking.

**Monitoring:** Deploy a synthetic monitoring bot that runs every 60 seconds. It requests the cached HTML page, extracts the displayed price, then requests the origin directly. If prices differ by more than 10 percent, trigger an alert. If over 1 percent of synthetic checks show availability discrepancies, page the on-call engineer.

### Challenge 2.3: Multi-CDN Strategy

**Problem:** Your company uses a single CDN provider. The CDN has two outages per year lasting 30-60 minutes. Each outage costs $2 million in lost revenue. Executive leadership asks you to design a multi-CDN strategy.

**Design:**
- How many CDN providers and which ones
- How traffic is split between them
- How failover works (manual vs automatic, detection, time to failover)
- Configuration synchronization across providers
- Cost impact

**Rubric:**

| Criterion | Poor | Good | Excellent |
|-----------|------|------|-----------|
| Provider selection | "Two random CDNs" | Geographic selection: CDN A for Americas, CDN B for Europe/Asia | Diverse: picks providers with different underlying infrastructure (Cloudflare + Fastly, not Cloudflare + Akamai which both use Level 3 transit) |
| Traffic splitting | "50/50 always" | DNS-based weighted: primary 70%, secondary 30% (live traffic tests) | Per-region traffic steering with real-time performance data |
| Failover | "Manual, flip DNS" | Auto-failover: health check every 10s, shift on 3 consecutive failures | Graceful degradation: shift 20% of traffic to healthiest provider every 15s, not 100% instantly |
| Config sync | "Same config on both" | Automated pipeline: deploy to both via Terraform/API | Canonical config in YAML pushed to both CDN APIs + validation suite for equivalence |
| Cost | "Double the CDN cost" | Negotiate lower rates with each (20% discount for committing 30% traffic) | Active-active: use both providers simultaneously, negotiate multi-year commits for 40% off list price |

**Model answer:**

**Provider selection:** I choose CloudFront (AWS) as primary and Fastly as secondary. Reason: they use different transit providers and different cache software. CloudFront is built on custom software; Fastly uses Varnish-based infrastructure. A routing problem affecting CloudFront (for example, BGP issue with AWS) will not affect Fastly.

**Traffic split:** Active-active, not active-passive. 70 percent to CloudFront, 30 percent to Fastly. Rationale: the secondary provider gets real production traffic, so its configuration and performance are continuously validated. If CloudFront fails, Fastly already handles 30 percent of traffic and scales to 100 percent seamlessly.

**Failover mechanism:** DNS-based with health checks. A monitoring service (independent, not on either CDN) checks the health of both CDNs from 10 global locations every 10 seconds. If 3 consecutive checks from 3+ locations detect errors (HTTP 5xx, latency > 1s, or connection failure), the monitoring service updates Route53 weight to shift 100 percent of traffic to the healthy CDN. Time to failover: 60-90 seconds (DNS TTL + health check windows). No manual step needed.

**Configuration sync:** Store CDN configuration in Terraform. Two modules — `cdn_cloudfront` and `cdn_fastly` — that produce equivalent configurations from the same input variables. After each configuration change, run a validation suite that: (a) deploys to staging on both CDNs, (b) sends test requests to both, (c) compares the response bodies, headers, and status codes. Only deploy to production if validation passes.

**Cost impact:** Two CDN providers cost more than one, but the discount negotiation offsets it. Single CDN at 100 percent traffic: $500K/month (list price $0.085/GB). Two CDNs: CloudFront at 70 percent ($350K/month, negotiate 20% discount = $280K/month) and Fastly at 30 percent ($150K/month, negotiate 20% discount = $120K/month). Total: $400K/month. The savings from discount ($100K/month) partially offset the infrastructure cost of running two CDNs. Annual cost: $4.8M (compared to $6M for single CDN at list price). Plus, avoided outage cost: $2M × 2 outages = $4M/year saved. Net benefit: $5.2M/year.

## Tier 3 — Expert (30 minutes each)

### Challenge 3.1: Design a CDN for a Real-Time Multiplayer Game

**Problem:** A multiplayer battle royale game has 50 million monthly active players. Each game session has 100 players. The game produces 30 updates per second per player (position, state, actions). Game sessions last 20 minutes on average. Players are global. The game must maintain under 50ms tick-to-render latency. You cannot cache game state because it is constantly changing.

**Design a CDN architecture for this workload that:**
- Reduces latency for game state updates without caching
- Handles 50 million concurrent players
- Provides DDoS protection
- Stays within $2 million/month content delivery budget

**Challenge notes:**

This is an intentionally ambiguous problem. Game networking is fundamentally different from web CDN workloads. A good answer recognizes that standard CDN caching does not help here. The solution involves:

- **Edge proximity:** Deploying relay servers close to players (not caching, but proxying). The "CDN" in this context is a network of game relay servers that forward UDP packets between players and the game server. Each relay adds under 2ms of processing time but saves 100ms of cross-ocean round trips.
- **Regional game servers:** Instead of one global game server, deploy regional matchmaking and game servers in each region. Players within a region play together on a regional server. Cross-region play uses relayed connections through the edge.
- **UDP acceleration:** Standard CDNs optimize TCP. Game traffic is UDP. The solution uses a custom UDP acceleration protocol (KCP, Noise, or QUIC-based) over the CDN's network.
- **DDoS mitigation:** The CDN's network capacity absorbs volumetric DDoS (100 Gbps+). Rate limiting at the edge drops non-game UDP traffic.

**Model answer:**

A standard HTTP CDN is irrelevant for real-time game state — you cannot cache frames. The correct architecture is:

**Regional game server deployment:** Deploy game server clusters in 8 AWS regions: us-east-1, us-west-2, eu-west-1, eu-central-1, ap-northeast-1, ap-southeast-1, sa-east-1, ap-south-1. Players are matched within their region. Cross-region play uses a relay between the two relevant regions.

**Edge relay network:** Deploy lightweight UDP relay servers in 50+ edge locations (same locations as CDN POPs). Each relay simply forwards encrypted UDP packets between the player and the regional game server. The relay uses a custom protocol that:
- Batches small packets (under 100 bytes) into larger UDP datagrams (up to 1400 bytes) for efficient transit
- Applies forward error correction (FEC) to recover from 5 percent packet loss without retransmission
- Measures RTT every 100ms and reports to the matchmaking service

**Latency math:**
- Player in Berlin: 5ms to Frankfurt edge relay, relay to Frankfurt game server: 2ms (same city). Total: 7ms. Well under 50ms.
- Player in Berlin matched with player in Sydney (cross-region): Berlin to Sydney = 160ms (fiber). Then relay relays one-way. Total: 167ms. Over 50ms threshold. Solution: do not enable cross-region matchmaking. Players in different regions cannot play together. (This is the trade-off most games accept.)

**DDoS protection:** The edge relays are the first line of defense. They validate that every UDP packet has a valid session token (issued by the matchmaking service). Packets without valid tokens are dropped at the relay with under 1 microsecond of CPU. Invalid traffic never reaches the game servers. The relay network absorbs up to 1 Tbps of attack traffic across 50 POPs.

**Cost:**
- Game server compute: 8 regions × 200 servers (c4.xlarge equivalent) × $0.10/hour × 730 hours = $116,800/month
- Edge relays: 50 locations × 10 servers × $0.05/hour × 730 hours = $18,250/month
- Bandwidth (game traffic): 50M players × 20 min/day × 30 updates/s × 100 bytes × 30 days = 540 PB/month. At $0.01/GB negotiated rate = $5.4M/month. Over budget.

**Budget optimization:** Reduce bandwidth by sending only delta updates (changed state, not full state). Average delta = 20 bytes (not 100). Bandwidth drops to 108 PB/month = $1.08M/month. Within budget.

### Challenge 3.2: Design a CDN for an AI Inference Cache

**Problem:** An AI image generation service (like Midjourney) generates images on demand. Each image takes 5-10 seconds of GPU compute to generate. Generated images are often requested multiple times (same prompt, same parameters). The service sees 100 million API calls per day. Images average 2MB each (512×512 PNG). Users are global.

**Design a caching layer that:**
- Caches generated images so repeated identical prompts return instantly
- Handles the cache key problem: "two cats playing chess" and "a cat and a dog playing chess" are different prompts and produce different images, even if the text is similar
- Manages the huge storage requirement (100M requests/day × even 10% repeat rate = 10M new images/day = 20TB/day)
- Handles the cost of GPU compute (cache misses are expensive, $0.01 per image at GPU time)

**Challenge notes:**

This problem tests: cache key design for non-URL content, storage tiering for massive write throughput, and cost optimization (GPU compute is the bottleneck, not bandwidth).

**Model answer:**

**Cache key:** The cache key is the SHA-256 hash of the full prompt + model version + seed + parameters (guidance scale, steps, size). Two prompts that differ by even one character produce different cache keys. This is by design — AI generates different images for "cat" vs "cat " (trailing space). The cache stores the key and the image.

**Cache tiers:**

| Tier | Storage | Access Pattern | Write Pattern | Cost |
|------|---------|---------------|---------------|------|
| L1: RAM (Hot) | 1TB per POP | Most popular 50K images | Read-heavy (100:1) | $0.30/GB/month |
| L2: NVMe (Warm) | 20TB per POP | Next 500K images | Write-heavy at peak | $0.10/GB/month |
| L3: S3 (Cold) | Unlimited | Rare prompts, long tail | Write-once, read rarely | $0.023/GB/month |

**Storage math:** 10M unique images/day × 2MB = 20TB/day. After 30 days: 600TB. This cannot fit in expensive cache tiers. Strategy:
- L1/L2 keep only images generated in the last 7 days (140TB per POP → requires 7 POPs with 20TB each)
- L3 (S3) holds everything forever (prompt reuse can happen months apart)
- When a cache miss occurs at L1/L2, check L3 before generating

**GPU cost optimization:**
- A cache hit saves $0.01 (GPU compute) + $0.0001 (storage retrieval)
- A cache miss costs $0.01 (GPU) + $0.0001 (storage write to L3)
- Target cache hit ratio: 70 percent
- Daily savings: 100M requests × 70% hit ratio × $0.01 = $700,000/day
- A 1 percent improvement in hit ratio saves $10,000/day

**Cache key extension (prompt normalization):** The main challenge is that "two cats" and "2 cats" are semantically identical but hash differently. Implement prompt normalization before hashing:
- Strip punctuation
- Normalize numbers (2 → two, 5 → five)
- Lowercase
- Strip extra whitespace
- Sort parameters (seed first, then steps, then guidance, etc.)

This increases hit ratio by an estimated 15 percent, saving $150,000/day in GPU costs.

**Eviction policy:** LFU (Least Frequently Used) with decay. An image that was popular last week but unused today decays in priority. LFU with logarithmic aging ensures popular images stay in cache while the long tail moves to S3 eventually.
