## Internal Architecture

A production CDN edge server is not a simple reverse proxy. It is a layered pipeline of specialized modules. Understanding each layer is essential because tuning the wrong layer causes cascading failures that are difficult to diagnose.

Each layer in this pipeline is independently configurable and independently monitored. A bottleneck at any layer degrades overall performance. In practice, layer 4 (cache lookup) and layer 5 (origin fetcher) are where most tuning effort concentrates because they determine hit ratio and origin load.

## Data Structures and Algorithms

### Cache Key Hash Table (Primary Index)

**Structure:** Open-addressed hash table with 32-byte keys (SHA-256 of the normalized URL) and 8-byte value pointers.

**Why this structure:** Cache lookup must complete in under 1 millisecond to keep p50 latency below 10ms. A hash table with O(1) average lookup satisfies this requirement. Open addressing (rather than chaining) avoids pointer chasing and cache misses at the CPU level — the hash table itself fits in L3 cache for tables under 4GB.

**Time complexity:** O(1) average, O(n) worst case (resize). Real-world load factor stays under 0.7 to keep collision rate low.

**Space complexity:** 40 bytes per entry (32 key + 8 pointer), plus the variable-length object storage. A 100 million entry table consumes approximately 4GB for the index alone.

### LRU Eviction List (Cache Replacement)

**Structure:** Doubly-linked list with a hash table for O(1) find-and-move operations. The most recently accessed entry sits at the head; the least recently accessed sits at the tail.

**Why this structure:** CDN cache slots are finite (typically 80 percent of available RAM/SSD is allocated to cache storage). When a new entry must be inserted and the cache is full, something must be evicted. LRU assumes that content accessed recently will be accessed again soon — which is true for content that follows a Zipf-like access distribution where 80 percent of requests hit 20 percent of content.

**Time complexity:** O(1) for lookup, insert, delete, and promote-to-head. The hash table provides the pointer; the list provides the ordering.

**Space complexity:** 24 bytes per entry (two pointers + key) for the list node, plus the hash table entry (40 bytes). Total overhead per cached object: approximately 64 bytes.

### Adaptive TTL Timer Wheel

**Structure:** Hierarchical timing wheel with 4 levels (milliseconds, seconds, minutes, hours) using a circular buffer of linked lists.

**Why this structure:** Each cached object has its own TTL. A naive approach checks every object's TTL on every access, which wastes CPU. A timer wheel groups objects by their expiry time and processes only expired buckets when time advances. The hierarchical design supports TTLs from 1 millisecond to 30 days without allocating 2.6 billion slots.

**Time complexity:** O(1) insert, O(1) tick, O(n) per tick for expired entries (n = objects in the expired bucket, ideally few).

**Space complexity:** 1024 slots per level × 4 levels = 4096 head pointers, plus the entries themselves.

### Bloom Filter for Hot Object Detection

**Structure:** Bit array of size m with k hash functions. Insertion sets k bits. Lookup checks all k bits — if any is 0, the object is definitely not in the hot set.

**Why this structure:** Not all cached objects are equal. The top 1 percent of objects serve 90 percent of traffic. Detecting hot objects allows the CDN to prioritize them in RAM (fast tier) versus SSD (slow tier). A bloom filter provides a fast, memory-efficient membership test with tunable false-positive rate.

**Time complexity:** O(k) per operation (k hash functions, typically 7-10).

**Space complexity:** For 10 million objects at 1 percent false positive rate: approximately 12MB (1.2 bytes per element). An equivalent hash set would require 400MB.

### Consistent Hash Ring for Cache Sharding

**Structure:** A ring of 2^64 positions. Each cache node maps to multiple virtual positions on the ring. Each cache key maps to exactly one position. The owner of a key is the nearest node clockwise from the key's position.

**Why this structure:** When the cache cluster adds or removes nodes, a naive modulo-N hash would remap every key (catastrophic miss storm). Consistent hashing remaps only K/N keys, where K is total keys and N is number of nodes.

**Time complexity:** O(log V) per lookup with binary search on virtual node positions, where V is the number of virtual nodes (typically 100-200 per physical node).

**Space complexity:** O(V) for the sorted position list (negligible).

### Range Bloom Filter for Range Request Handling

**Structure:** A set of bloom filters, one per 1MB chunk of a large file, plus a prefix bloom filter that summarizes the full set.

**Why this structure:** Video downloads use HTTP Range requests (partial content). Each byte range translates to one or more chunks. The CDN must know which chunks are cached to respond with the correct byte range or fetch missing chunks from origin. Scanning all chunks on every request is too slow.

**Time complexity:** O(chunks_in_range) per request. Typical range covers 2-5 chunks.

**Space complexity:** 1.2 bytes per chunk at 1 percent FPR. A 10GB video file divided into 1MB chunks uses 12KB of bloom filter data.

## Configuration and Tuning

### Cache-Control Header Directives

| Directive | Purpose | Safe Default | When to Change | What Breaks |
|-----------|---------|-------------|----------------|-------------|
| `max-age=N` | How long (seconds) the CDN caches without revalidation | 3600 for HTML, 31536000 for static assets | Shorter for dynamic content, longer for evergreen assets | Too long = stale content served; too short = origin overload |
| `s-maxage=N` | Overrides max-age specifically for shared caches (CDNs) | Same as max-age | When you want CDN to cache longer than browser cache | Browser vs CDN TTL mismatch causes confusion |
| `public` | Allows any cache to store the response | Set for all cacheable content | Remove for authenticated/user-specific responses | Exposing private data if set incorrectly |
| `private` | Prevents CDN from caching (browser only) | Set for user-specific content | Remove for public content | Zero CDN benefit for that endpoint |
| `no-cache` | Must revalidate with origin before every serve | Omit for cacheable content | Set for content that changes every request | Kills cache hit ratio (forces every request to origin) |
| `no-store` | Never store any copy (CDN or browser) | Set for sensitive data (PII, tokens) | Remove for public content | Zero caching possible |
| `stale-while-revalidate=N` | Serve stale during background revalidation for N seconds | 86400 (1 day) for static; 0 for dynamic | Higher for origin reliability | Serves slightly stale content during revalidation window |
| `stale-if-error=N` | Serve stale content if origin returns error for N seconds | 86400 for non-critical content | 0 for transactional endpoints | Masks real origin errors |

### CDN Platform Configuration

| Setting | What It Controls | Safe Default | When to Change | What Breaks |
|---------|-----------------|-------------|----------------|-------------|
| Cache key template | Which URL parts form the cache key | Host + path + extension | Strip query params for static assets | Too narrow = low hit ratio; too broad = cache collisions |
| TTL override | Force TTL on objects that lack Cache-Control headers | 3600s | When origin cannot set headers | Overriding a long-lived TTL with a short one reduces performance |
| Query parameter filtering | Which query params are included in cache key | Strip all except known params | API responses using query-based differentiation | Cache serves wrong content variant |
| Origin connect timeout | How long to wait for origin TCP connection | 5s | Increase for slow origin under load | Slow failure detection on dead origin |
| Origin read timeout | How long to wait for origin response body | 30s | Increase for large file origins | Premature 504 errors on slow responses |
| Max object size | Largest single object the CDN will cache | 1GB | Increase for video content | Cache fills with huge objects, evicting many smaller ones |
| Compression level | Brotli/gzip compression ratio | Auto (brotli 4) | Lower for CPU-bound edge servers | Too high = increased CPU at edge reducing throughput |
| Rate limiting | Max requests per IP per second | 1000 req/s/IP | Lower for API protection | Blocking legitimate users if set too low |
| WAF sensitivity | How aggressively to block suspicious requests | Medium | Lower for traffic-heavy applications | Blocking legitimate traffic (false positives) |
| Origin shield level | Number and distribution of shield layers | Regional shield only | Add global shield for flash crowd protection | More layers = higher stale content propagation delay |

When tuning a CDN, change one parameter at a time and monitor the cache hit ratio, origin load, and p99 latency. A change that improves hit ratio by 5 percent but doubles origin timeout errors is a net negative. The most common production tuning sequence is: normalize cache keys first (removing session IDs and timestamps), then extend TTLs, then add shield layers, then tune compression.
