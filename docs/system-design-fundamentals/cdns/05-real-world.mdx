## Company Case Studies

### Netflix — Open Connect CDN

**Scale:** 260 million subscribers, 15+ exabytes of monthly content delivery, 15 percent of global internet traffic at peak hours.

**Problem:** Netflix started as a DVD-by-mail company. When it migrated to streaming in 2007, it used third-party CDNs (Akamai, Limelight). By 2010, Netflix accounted for 30 percent of North American internet traffic during peak evening hours. The cost of third-party CDN bandwidth was unsustainable — estimated at $100+ million annually. Additionally, third-party CDNs could not guarantee quality of experience for 4K streaming, which requires 25 Mbps sustained throughput.

**Solution:** Netflix built Open Connect, its own CDN. Instead of caching video files in standard edge POPs, Netflix deployed dedicated streaming appliances directly inside ISP data centers. These appliances pre-load popular content during off-peak hours and serve it to subscribers over the ISP's internal network — the traffic never leaves the ISP's infrastructure.

**Result:** Netflix saves an estimated $1 billion per year in bandwidth costs. Average video start time dropped from 5 seconds to under 2 seconds. Open Connect handles 95 percent of Netflix's global traffic. The remaining 5 percent (less popular content) is served from AWS.

**Key lesson:** The most efficient CDN is the one that does not cross any network boundaries. By putting content inside the ISP, Netflix eliminated transit costs entirely.

### Uber — Edge Caching for Real-Time Rides

**Scale:** 130 million monthly active users, 25 million trips per day across 70 countries.

**Problem:** Uber's ride availability data is highly dynamic — driver locations change every few seconds. A standard CDN with TTL-based caching would serve stale GPS coordinates, making the app useless. But every ride request going back to a central origin added 200-800ms of global latency.

**Solution:** Uber uses CloudFront (AWS CDN) with Lambda@Edge for ride-hail caching. Instead of caching ride availability data, Uber caches the static parts of each page (maps tiles, UI framework, CSS) at the edge with 24-hour TTLs. The dynamic ride data is fetched directly from Uber's regional API gateways, which are deployed in 6 AWS regions worldwide. Users connect to the nearest regional gateway via Route53 Latency-Based Routing.

### Cloudflare — Reverse Proxy as Global CDN

**Scale:** 30 million+ websites, 20 percent of all web traffic, 330 data centers worldwide.

**Problem:** Cloudflare itself IS a CDN, but it had to solve a unique challenge: how to serve 30 million websites from shared infrastructure without cache key collisions. Two different customers could both have the URL `https://example.com/style.css` — but they reference different content.

**Solution:** Cloudflare's cache key includes a `zone_id` that disambiguates between customers. Every customer's content is isolated, even if URLs collide. Cloudflare also pioneered the use of Argo Smart Routing, which dynamically finds the fastest path between Cloudflare edge servers and the origin.

**Key lesson:** Multi-tenant CDNs require cache key isolation at the tenant level. Two customers with identical URLs must not see each other's content.

### Dropbox — LAN Sync as Local CDN

**Scale:** 700 million users, 2 billion files synced daily, 500PB of storage.

**Problem:** When multiple Dropbox users in the same office all sync the same large file, each user independently downloads it from Dropbox's servers. The download is repeated N times over the office's internet connection, consuming bandwidth and slowing every user.

**Solution:** Dropbox implemented LAN Sync. When a user downloads a file, Dropbox advertises the file's checksum on the local network. Other Dropbox clients detect the advertisement and download the file from the local peer instead of from Dropbox's servers. This turns every Dropbox client into a miniature CDN edge node.

## Open Source Implementations

### Nginx (Reverse Proxy + Cache)

Nginx can function as a simple CDN edge server. It is the most widely deployed open-source CDN building block.

```nginx
# Production Nginx CDN edge configuration
http {
    # Cache configuration
    proxy_cache_path /data/nginx/cache levels=1:2 
                     keys_zone=cdn:500m max_size=500g
                     inactive=7d use_temp_path=off;
    
    # Cache lock for stampede prevention
    proxy_cache_lock on;
    proxy_cache_lock_timeout 5s;
    
    # Stale serving
    proxy_cache_use_stale error timeout updating http_500 http_502;
    
    server {
        listen 80;
        location / {
            proxy_cache cdn;
            proxy_cache_key "$scheme$host$uri$is_args$args";
            proxy_cache_valid 200 302 60m;
            proxy_cache_valid 404 1m;
            proxy_cache_bypass $http_x_purge;
            proxy_pass http://origin_upstream;
        }
    }
}
```

**Strengths:** Extremely stable, well-documented, handles 10,000+ concurrent connections per worker process.
**Weaknesses:** Single-node only (no built-in clustering), no multi-tenant isolation, basic cache invalidation.
**Best for:** Simple reverse proxy CDN in front of a single origin service.

### Varnish Cache (High-Performance HTTP Cache)

Varnish is purpose-built for HTTP caching. It sits in front of your application server and caches responses in virtual memory.

```vcl
# Varnish VCL configuration (simplified)
vcl 4.1;

backend origin {
    .host = "origin.example.com";
    .port = "80";
}

sub vcl_recv {
    # Normalize cache key: strip utm_* parameters
    if (req.url ~ "(\?|&)(utm_)") {
        set req.url = regsuball(req.url, "&(utm_[^&]*)", "");
        set req.url = regsuball(req.url, "\?(utm_[^&]*)&?", "?");
        set req.url = regsuball(req.url, "\?$", "");
    }
    
    # Bypass cache for authenticated endpoints
    if (req.url ~ "^/api/user/") {
        return (pass);
    }
}

sub vcl_backend_response {
    # Override TTL from origin
    if (beresp.ttl under 3600s) {
        set beresp.ttl = 3600s;
    }
    
    # Stampede protection
    set beresp.ttl = beresp.ttl + std.random(0, 600s);
}

sub vcl_hit {
    # Stale-while-revalidate
    if (obj.ttl >= 0s && obj.ttl under 10s) {
        return (deliver);  # Serve stale, trigger async revalidate
    }
}
```

**Strengths:** Sub-millisecond cache lookup (in-memory hash table), flexible VCL domain-specific language for cache logic, real-time purge via PURGE request.
**Weaknesses:** Single-threaded event loop (CPU-bound), no built-in clustering, requires tuning for high concurrency.
**Best for:** High-performance HTTP cache with complex caching logic.

### Apache Traffic Server (ATS — Full CDN Software)

Apache Traffic Server is the only complete open-source CDN server. It is used as the core of many commercial CDNs (Yahoo, Comcast, Akamai historically).

**Strengths:** Multi-threaded (scales across CPU cores), native HTTP/2 and HTTP/3 support, plugin system for custom logic, built-in DNS for anycast, native clustering and cache peering.
**Weaknesses:** Complex configuration (300+ settings), steep learning curve, smaller community than Nginx.
**Best for:** Building a custom CDN where you need full control.

### Redis (Distributed Cache Layer)

Redis is not a CDN server, but many CDNs use it as a shared cache coordination layer.

```python
# Redis-based request coalescing for CDN stampede prevention
import redis
import requests

r = redis.Redis(host='redis-cluster', port=6379)

def serve_or_fetch(url, cache_key, ttl=3600):
    # 1. Try local cache first
    local_entry = local_storage.get(cache_key)
    if local_entry:
        return local_entry
    
    # 2. Redis SETNX for distributed lock
    lock_acquired = r.setnx(f"lock:{cache_key}", "1")
    if lock_acquired:
        r.expire(f"lock:{cache_key}", 5)  # Lock expires in 5s
        # This server fetches from origin
        response = requests.get(url)
        local_storage.set(cache_key, response.content, ttl)
        r.delete(f"lock:{cache_key}")  # Release lock
        return response.content
    else:
        # Another server is fetching. Wait, then retry.
        time.sleep(0.05)
        return serve_or_fetch(url, cache_key, ttl)
```

**Best for:** Coordinating cache operations across multiple edge servers (stampede prevention, cache peering).

## Lessons Learned

### Lesson 1: Cache Key Design Is the Most Important Optimization

Every case study reinforces the same lesson: get the cache key right, and everything else follows. Netflix uses content hash + bitrate + language as the cache key. Uber uses URL path + file hash for static assets only. Cloudflare uses zone_id + URL for multi-tenant isolation.

**Pattern:** A cache key must uniquely identify content, not requests. Strip every request-specific parameter (session IDs, timestamps, tracking parameters) from the key. Add only the parameters that change the response content.

### Lesson 2: Flash Crowd Protection Must Be Automatic, Not Manual

Every company that operates at scale has experienced a thundering herd. The successful ones automated protection. Netflix's Open Connect pre-positions content. Cloudflare's cache lock prevents simultaneous origin fetches. Dropbox's LAN Sync reduces external dependency.

**Pattern:** Implement request coalescing at the cache layer. When 100 requests arrive for the same missing key, only 1 should go to origin. The other 99 should wait for the result.

### Lesson 3: Partition Tolerance Is Not Optional at Global Scale

Network partitions are not rare anomalies — they happen weekly across 300+ POPs. The correct response is to serve stale content (AP) rather than returning errors (CP). Every CDN makes this choice.

**Pattern:** Set `stale-while-revalidate` to at least 86400s (1 day) for static content. For dynamic content, accept that brief downtime during partitions is better than serving incorrect data.

### Lesson 4: Cost Per Gigabyte Determines Your Architecture

Bandwidth is the dominant cost at scale. Netflix's entire Open Connect architecture was driven by bandwidth economics. Dropbox's LAN Sync was driven by the same math. When bandwidth costs exceed compute costs, the architecture shifts to minimize data transfer.

**Pattern:** Optimize for cache hit ratio above all other metrics. A 1 percent improvement in hit ratio saves 1 percent of total bandwidth, which at Netflix's scale is $10 million+ per year.

### Lesson 5: Multi-CDN Is the Endgame for Large Deployments

Companies with over 10 million DAU typically use 2+ CDNs simultaneously. Reasons: price competition between providers, geographic coverage gaps (no single CDN covers every region equally), and failover resilience.

**Pattern:** Use DNS-based load balancing to split traffic between CDNs. Assign 70 percent to the primary CDN and 30 percent to the secondary. Monitor error rates per CDN and shift traffic if one degrades. Netflix uses a custom multi-CDN routing layer that accounts for real-time performance.
