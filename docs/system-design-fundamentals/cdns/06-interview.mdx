## How This Topic Appears

CDN questions appear in two forms: as the primary topic ("design a CDN") or as a sub-topic within a larger system design ("design Netflix", "design YouTube", "design a global e-commerce site"). The sub-topic form is more common at FAANG interviews.

### Primary Topic Questions

1. "Design a CDN system." (Google, Amazon — L5+)
2. "How would you build a global content delivery network from scratch?" (Meta, L6+)
3. "Design a system for serving static assets to 100 million users worldwide." (Microsoft, L5+)
4. "Architect a CDN that handles 10 Tbps of video traffic." (Netflix, L6+)

### Sub-Topic Questions

5. "Your video streaming service is slow in Asia. Walk me through how you would fix it." (Any company, any level)
6. "How does a CDN improve page load time? Walk through the steps." (Phone screen, L3/L4)
7. "Our cache hit ratio dropped from 95% to 40%. How do you debug it?" (On-site, L4/L5)
8. "Should we use a CDN for our real-time chat application?" (System design probing, L5)
9. "Compare edge caching and origin caching strategies for our API." (L5, design review)
10. "How do you handle a flash crowd / viral event on a cached system?" (L5+)

## What the Interviewer Is Actually Testing

### For "Design a CDN" (primary)

| What They Test | How They Probe | Red Flag |
|---------------|----------------|----------|
| Do you understand DNS and anycast? | "Why doesn't traffic go to the wrong POP?" | Guesses at DNS behavior |
| Can you reason about cache hierarchies? | "Should we have 1 or 3 layers of cache?" | Picks a number without reasoning |
| Do you know the CAP implications? | "What happens during a network partition?" | "It just works" |
| Can you plan capacity? | "How many servers do we need for 100M DAU?" | Afraid to do math |
| Do you understand cache invalidation? | "How do you push new content?" | "We purge everything" |
| Can you handle failure scenarios? | "What if the origin goes down?" | "It never goes down" |

### For "The app is slow in region X" (sub-topic)

| What They Test | How They Probe |
|---------------|----------------|
| Can you trace latency to root cause? | "Is it the CDN, the origin, or the last mile?" |
| Do you know CDN configuration? | "What headers would you check?" |
| Can you prioritize fixes? | "Should we add a POP there or optimize the origin?" |
| Do you understand the cost-performance trade-off? | "A new POP costs $50K/month. Is it worth it?" |

## Level-by-Level Expectations

### L3 / L4 (Entry / Mid-level)

**Expectation:** Explain what a CDN does at a high level. Identify when a CDN helps. Set up a basic CDN configuration.

**Sample question:** "How does a CDN improve website performance?"

**Sample answer:** "A CDN improves performance by caching content on edge servers located closer to users. When a user requests a file, the CDN routes them to the nearest edge server. If the file is cached there (a cache hit), it returns in under 50ms instead of the 200-600ms it would take to reach the origin server. The key metric is the cache hit ratio, which should be above 85 percent for static assets. The CDN also reduces origin server load because most requests never reach it — only cache misses do, which is typically 5-15 percent of total traffic."

**What the interviewer wants:** You know the core concept, the latency numbers, and the cache hit ratio metric. You can explain it to a non-technical stakeholder.

### L5 (Senior)

**Expectation:** Design a CDN from scratch. Compare providers. Diagnose cache-related issues from symptoms. Handle failure scenarios.

**Sample question:** "Design a CDN for a global social media platform with 500 million monthly active users."

**Sample answer structure:**

1. **Requirements:** 500M MAU, 20 images/user/day, 500KB average image size. 50 percent of users outside US. Read-heavy (99% reads). Target p50 latency under 20ms globally.

2. **Architecture:** Three-tier cache: edge POPs in 20 cities (anycast DNS), regional caches per continent, origin shield in us-east-1 and eu-west-1 (active-active). Each edge POP has 8 servers with 4 NVMe drives each. Cache key: host + path + content hash (strip all query parameters except format).

3. **Cache strategy:** Static assets (images, CSS, JS) get 1-year TTLs with versioned URLs. API responses get 60-second TTLs with stale-while-revalidate. User content gets 24-hour TTLs with cache lock for stampede prevention.

4. **Scale math:** 500M users × 50 requests/day = 25B daily requests. Peak RPS = 2.5M (10x average). With 95% cache hit ratio, origin sees 125K RPS peak. Need 20 edge servers per POP (2.5M ÷ 20 POPs = 125K RPS per POP; each server handles 10K RPS → 12.5 servers → 20 for headroom).

5. **Failure handling:** Partition → serve stale. Origin down → serve stale for up to 7 days. Cache stampede → request coalescing at every layer. BGP hijack → RPKI + multi-CDN backup.

**What the interviewer wants:** You can handle ambiguity, do the math, justify architectural choices, and handle failures.

### L6 (Staff)

**Expectation:** Design a multi-tenant CDN. Optimize for cost at scale. Handle catastrophes. Design the control plane. Know open source CDN internals.

**Sample question:** "Design Cloudflare's CDN infrastructure."

**Sample answer additions beyond L5:**

- Multi-tenant isolation via zone-scoped cache keys and separate rate limit budgets per customer.
- Control plane architecture: etcd-backed config store, gRPC-based config distribution to POPs, eventual consistency (POPs can operate offline for 24 hours with local config cache).
- Cost optimization: negotiate transit costs per POP (some regions are 10x more expensive than others), use peering at IXPs to reduce transit, deploy servers based on traffic density not geographic coverage.
- Edge compute platform: Wasm-based sandbox for customer code execution at edge, with per-request CPU limits (50ms budget), memory limits (128MB), and network access control.
- Multi-CDN failover: DNS-based with health-checked traffic steering. Automatically shift 20% of traffic to backup CDN if primary error rate exceeds 1% for 60 seconds.

**What the interviewer wants:** Deep architectural thinking, multi-tenancy awareness, cost optimization, and control plane knowledge.

### L7 / L8 (Principal / Distinguished)

**Expectation:** Design a new CDN protocol or algorithm. Predict industry trends. Set technical strategy for the next 3-5 years.

**Sample question:** "How would you design a CDN protocol from scratch, independent of HTTP?"

**Sample answer themes:** Mosh (mobile-optimized protocol with unreliable transport), QUIC-based multicast for live streaming, erasure coding for cache redundancy (reduce replicas from 3 to 1.5x), information-centric networking (ICN) where cache keys are content hashes not URLs, peer-to-peer CDN extensions for last-mile delivery.

**What the interviewer wants:** Research-level thinking, ability to challenge HTTP assumptions, understanding of transport layer optimizations, vision for future internet architecture.

## The FAANG Framework for Answering

### 6-Step Framework for CDN System Design Questions

**Step 1 — Scope and Requirements (2 minutes)**

State what you are and are not building. Clarify scale.

"I will design a CDN for serving static assets (images, CSS, JS) to a global audience of 500 million monthly active users. I am not building a video streaming CDN, though I will note where the designs diverge. My goals are: p50 latency under 20ms, 99.9 percent availability, and 95 percent cache hit ratio for static assets."

**Step 2 — Traffic Math and Capacity (3 minutes)**

Calculate requests per second, bandwidth, storage, and server count.

"500M MAU, 10 requests/user/day average, 50 percent cacheable. That is 2.5B daily requests, 57,870 RPS average, 578,700 RPS peak. With 500KB average response size, peak bandwidth is 231 Gbps. I need 20 POPs with 12 edge servers each..."

**Step 3 — High-Level Architecture (3 minutes)**

Draw the three layers: DNS, edge, origin. Explain the data flow.

"The system has three layers: DNS with anycast routing directs users to the nearest POP. Each POP has a cluster of edge servers with local SSD cache. Behind the edges is a regional cache layer per continent. Behind that is an origin shield that deduplicates requests to the origin cluster. The control plane manages config distribution, purge propagation, and monitoring."

**Step 4 — Deep Dive on One Component (5 minutes)**

Pick the most interesting component (cache key design, stampede protection, invalidation strategy). Explain with specifics.

"I want to focus on cache key design because it determines the entire hit ratio. The cache key is `zone_id + host + path + content_type`. I strip all query parameters except `?format=`. This prevents the billion unique cache keys that would come from ignoring session IDs in URLs. For API responses, I add the `Accept-Language` header to the key so different language versions are cached separately."

**Step 5 — Failure Modes and Mitigations (3 minutes)**

Cover at least three failure scenarios with specific mitigation.

"Three failure modes: first, a cache stampede when viral content causes simultaneous misses at every POP — mitigated by request coalescing and an origin shield. Second, a BGP hijack that routes traffic to an attacker — mitigated by RPKI and multi-CDN failover. Third, stale configuration pushes that accidentally set cache TTL to zero on all assets — mitigated by canary deployments on the config rollout (1% of POPs, then 10%, then 100%)."

**Step 6 — Trade-offs and Alternatives (2 minutes)**

Summarize what you sacrificed and what you would do differently at different scales.

"Key trade-offs: I optimized for read latency over write consistency (AP design). Cached content may be up to 60 seconds stale. At smaller scale (under 10M DAU), a single-region CDN with CloudFront would be more cost-effective. At larger scale (over 1B DAU), I would add a peer-to-peer delivery layer and custom edge compute for personalization. I chose SSD over RAM for cache storage because at 500TB of total content, SSD provides the best cost-performance ratio."

## Phrase Bank

Use these phrases in your answers. Each includes the context where it fits naturally.

**Phrase 1 (when scoping):** "I will narrow the scope to static asset delivery. Dynamic content and video streaming have fundamentally different caching requirements, and I want to focus on getting the static path right."

**Phrase 2 (when discussing DNS):** "Anycast DNS allows a single IP address to route users to the nearest POP via BGP. The beauty of anycast is that failover is automatic — if a POP goes down, BGP withdraws the route and traffic shifts to the next-closest POP within 60 seconds."

**Phrase 3 (when discussing cache keys):** "The golden rule of cache keys is: include only what differentiates the content, exclude everything else. Session IDs, tracking parameters, and timestamps must never be cache key components."

**Phrase 4 (when discussing TTLs):** "I use a graduated TTL strategy: short TTLs at the edge (1 hour), longer at the regional cache (4 hours), and longest at the shield (24 hours). This way, a single object expiry does not cascade through all layers simultaneously."

**Phrase 5 (when discussing stampede prevention):** "Request coalescing is mandatory at every cache layer. Whether it is Nginx's proxy_cache_lock or Varnish's hit-for-pass, only one request per unique key goes upstream. The rest queue and receive the cached result."

**Phrase 6 (when discussing stale serving):** "Cache invalidation is hard. I prefer versioned URLs over purge operations because versioning eliminates the need for global cache invalidation — old cache entries expire naturally via TTL."

**Phrase 7 (when discussing partitions):** "CDNs are AP systems in CAP terms. During a network partition, every edge POP continues serving cached content independently. Different POPs may return different versions of the same URL, but the system stays available."

**Phrase 8 (when discussing origin protection):** "The origin shield is the most important layer for protecting your origin at scale. Without it, a cache miss for a popular asset at 50 POPs results in 50 concurrent requests to the origin. With it, the shield deduplicates to 1 request."

**Phrase 9 (when discussing monitoring):** "I monitor three primary metrics: cache hit ratio (target above 85%), p99 edge latency (target under 50ms), and origin load percentage (target under 60%). A drop in hit ratio is the earliest indicator of a CDN problem."

**Phrase 10 (when discussing multi-CDN):** "At scale, multi-CDN provides price competition, geographic diversity, and failover resilience. No single CDN covers every region equally. I would use DNS-based traffic steering with real-time health checks to distribute load across providers."

**Phrase 11 (when discussing cost):** "Bandwidth is the dominant cost. A 1 percent improvement in cache hit ratio at 100 Gbps throughput saves approximately $7,000 per month at standard CDN pricing. Every optimization should be evaluated by its impact on bandwidth costs."

**Phrase 12 (when discussing edge compute):** "Edge compute shifts the trade-off: instead of caching raw content, you cache rendered fragments. A personalized homepage would be assembled at the edge from cached components (header, footer, recommendations) rather than fetched entire from the origin."

**Phrase 13 (when discussing write-through vs write-behind):** "For content updates, I use a write-through pattern: the origin accepts the write, updates its database, then invalidates the CDN cache for that URL. If purge propagation takes more than 5 seconds, I version the URL instead."

**Phrase 14 (when discussing security):** "Cache poisoning is prevented by validating origin response headers before caching. I verify the Content-Type matches expectations and the response does not contain security headers that contradict the cache policy (for example, Set-Cookie on a publicly cached response)."

**Phrase 15 (when discussing configuration rollouts):** "Config changes follow a canary deployment: push to 1 percent of POPs, monitor for 10 minutes, then 10 percent, then 100 percent. A single bad config (for example, disabling cache for all routes) would be detected before it affects all users."

## Red Flags

### Red Flag 1: "The CDN handles everything automatically"

**Why it is a red flag:** It suggests you have not operated a CDN in production. CDNs require careful configuration for cache key normalization, TTL settings, and origin protection. They do not "just work."

**Better answer:** "A CDN provides the right abstractions, but each deployment requires tuning. The most critical configuration is the cache key template. The default typically includes all query parameters, which destroys cache hit ratio for sites with tracking parameters. I always override this."

### Red Flag 2: "I would purge the entire cache on every deploy"

**Why it is a red flag:** It shows you have not thought about cache stampedes or the cost of repopulating caches globally. A full cache flush causes minutes of elevated latency and origin load.

**Better answer:** "I use versioned URLs so that old cache entries expire naturally. Purge operations are expensive and slow. If I must purge, I purge only the specific changed paths, not wildcards or the entire distribution."

### Red Flag 3: "We set TTL to maximum so everything is cached forever"

**Why it is a red flag:** Infinite TTLs prevent cache invalidation. If content must change (security patch, legal takedown), you cannot force users to see the new version.

**Better answer:** "I set TTLs based on content lifecycle. Versioned static assets get 1-year TTLs. HTML pages get 1-hour TTLs. API responses get 60-second TTLs. Each TTL choice balances freshness against performance."

### Red Flag 4: "The CDN will protect us from DDoS automatically"

**Why it is a red flag:** CDNs help with volumetric DDoS but require specific rate limiting and WAF rules. Without them, a cache-busting DDoS (unique URLs in every request) bypasses the cache and hits the origin.

**Better answer:** "A CDN helps with DDoS but is not a silver bullet. I configure rate limiting at the edge (per-IP limits), set a maximum origin request rate, and enable WAF rules for known attack patterns. Cache-busting DDoS is specifically mitigated by setting `proxy_cache_bypass` only for authenticated users."

### Red Flag 5: "Latency will be under 10ms for all users"

**Why it is a red flag:** Light in fiber travels at 200,000 km/s. A user in Australia connecting to a US-based POP will always have at least 80ms of round-trip latency due to physics. Promising sub-10ms globally is impossible without POPs in every country.

**Better answer:** "Latency is bounded by the speed of light and the distance to the nearest POP. With POPs on every inhabited continent, I can achieve under 20ms for users in major metro areas. Users in regions without a POP (for example, parts of Africa) will see 100-200ms latency. The solution is to add POPs where traffic density justifies the cost."

### Red Flag 6: "I would not cache dynamic content"

**Why it is a red flag:** Many types of dynamic content can and should be cached with short TTLs (10-60 seconds). Blanket "no caching" for dynamic content wastes significant performance.

**Better answer:** "I categorize dynamic content by staleness tolerance. Product catalog data (TTL 60s), news headlines (TTL 30s), and weather forecasts (TTL 300s) are all cacheable with short TTLs. Only user-specific data (cart contents, session tokens, personalized recommendations) bypass the cache entirely."

### Red Flag 7: "We do not need an origin shield"

**Why it is a red flag:** It indicates you have not experienced a flash crowd. Without a shield, 50 POPs simultaneously missing for the same new content will send 50 concurrent requests to the origin.

**Better answer:** "The origin shield is non-negotiable for any deployment with more than 5 POPs. It provides request coalescing at the origin boundary, converting N concurrent misses into 1 origin request. The shield also isolates the origin from edge POP failures: if the shield is up, the origin is protected even if edges are misbehaving."

### Red Flag 8: "My design has only edge servers and an origin, no intermediate layers"

**Why it is a red flag:** A flat two-layer architecture (edge → origin) does not scale. Every cache miss goes directly to origin. There is no deduplication, no isolation, and no graduated TTL strategy.

**Better answer:** "I use a three-tier cache: edge POPs (first layer, small capacity, low latency), regional caches (second layer, larger capacity, moderate latency), and origin shield (third layer, largest capacity, closest to origin). Each layer has a longer TTL than the layer above, so a miss at the edge likely hits at the regional layer."

## Practice Drills

### Drill 1: Phone Screen Warm-up (2 minutes)

**Prompt:** "Explain how a CDN works to a non-technical product manager in 2 minutes. Do not use any jargon."

**Self-evaluation:** Record yourself. Did you use an analogy? Did you hit the key points (distance, caching, faster load times)? Did you avoid jargon? Aim for: clear, simple, compelling.

### Drill 2: Cache Hit Ratio Debug (5 minutes)

**Prompt:** "Your e-commerce site normally has a 92 percent cache hit ratio. Today it dropped to 45 percent. Walk through your debugging approach."

**Checklist:** Did you check for a recent deploy (changed cache keys)? Check for new query parameters? Check origin response headers (missing Cache-Control)? Check for a purge that went too broad? Check CDN config changes? Measure the hit ratio per URL pattern (all URLs or just one pattern?).

### Drill 3: Architecture Sketch (10 minutes)

**Prompt:** "Design a CDN for serving high-resolution satellite imagery to a global mapping application. 2 billion tiles in the catalog, 100 million daily active users, tile size averages 256KB."

**Checklist:** Calculate total storage (2B × 256KB = 512TB). Calculate RPS (100M users × 50 tiles/day = 5B daily, 57,870 RPS average, 578,700 RPS peak). Plan cache tiers: hot tiles in RAM (most-requested 1% = 5TB), warm tiles on SSD (next 9% = 46TB), cold on HDD or pre-fetched. Use consistent hashing for tiles to maximize cache reuse (Z-order curve for spatial locality). Use range requests for progressive loading (256KB tiles split into 16KB chunks for quick initial render).

### Drill 4: Failure Scenario (5 minutes)

**Prompt:** "The origin server is down. Your CDN has 50 POPs worldwide. What happens? What do you do?"

**Checklist:** Cache hits continue serving (all POPs have cached content). Cache misses return 502 or stale content. Purge commands fail (cannot reach POPs). Recovery brings origin up, then POPs revalidate stale content via conditional GETs. Measure: how much traffic is cached (ratio-dependent), how long until origin recovers (assume 15 minutes for hot spare), what percentage of users are affected (the miss percentage).

### Drill 5: Trade-off Deep Dive (10 minutes)

**Prompt:** "Should you use versioned URLs or purge-based invalidation for your CDN? Compare both approaches."

**Checklist:** Versioned URLs: no purge needed, old entries expire via TTL, cache always warm, requires build pipeline changes. Purge: immediate removal of old content, no URL changes needed, but purge propagation takes 1-30 seconds, costs money (CloudFront charges per path), and can fail silently. Recommendation: versioned URLs for build-time assets, purge for emergency content takedowns.
