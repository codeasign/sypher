## Tier 1 — Foundation

### Challenge 1: Explain the Pipeline

**Prompt:** Your colleague is a backend engineer who hasn't worked with ML. Explain, in plain terms, how a semantic search system works end-to-end. Include the embedding model, the index, and the query flow.

**Time limit:** 5 minutes.

**Evaluation criteria:** Your answer must cover:
1. What the embedding model does (converts text to vectors)
2. Why vectors are useful (they encode meaning as position in space)
3. How the index works (organizes vectors for fast search)
4. How the query flows (embed query → search index → return results)
5. What cosine similarity measures (angle between vectors = semantic distance)
6. The key insight: query and documents must use the SAME embedding model

**Common mistakes:**
- Skipping the normalization step or not knowing it matters
- Using different models for indexing and querying
- Not explaining why ANN is needed (the brute-force approach doesn't scale)
- Confusing the vector store with the document store

**Model answer:**

"Every semantic search system has three stages: indexing, querying, and retrieval.

**Indexing:** We take each document and pass it through an embedding model — a neural network that converts text into a list of numbers called a vector. For example, 'how to reset my password' becomes a vector with 1,536 numbers. These numbers are coordinates in a 'meaning space' — documents about similar things get similar coordinates. We store these vectors in a special data structure called an approximate nearest neighbor index, which is optimized to quickly find vectors that are 'close' in this space.

**Querying:** When a user types a search, we pass their query through the exact same embedding model to get a query vector. We then ask the index: 'which vectors are closest to this one?' using cosine similarity, which measures the angle between vectors. The index returns the top K results (usually 10 or 100).

**Retrieval:** We take the IDs of those closest vectors and look up the original documents in a separate database. The results are returned ranked by similarity score.

The critical constraint is that the same embedding model must be used for both indexing and querying, because different models produce vectors in different coordinate spaces — you can't compare them, just like you can't compare GPS coordinates with a different planet's coordinate system."

### Challenge 2: Compare Cosine Similarity and Euclidean Distance

**Prompt:** Your team is debating whether to use cosine similarity or Euclidean distance for comparing embeddings. Explain the difference, when each is appropriate, and why cosine similarity is the standard choice for text embeddings.

**Time limit:** 5 minutes.

**Evaluation criteria:**
1. Must explain the formula difference (angle vs. straight-line distance)
2. Must explain why cosine similarity is scale-invariant (length doesn't matter)
3. Must give a concrete example where the two metrics produce different rankings
4. Must mention L2-normalization and cosine = dot product for normalized vectors

**Common mistakes:**
- Saying cosine similarity is always better (it's not — for some embedding types, Euclidean distance works fine)
- Not understanding the math (dot product vs. cosine formula)
- Forgetting to mention normalization

**Model answer:**

"Cosine similarity measures the angle between two vectors, ignoring their length. Euclidean distance measures the straight-line distance between their endpoints.

For text embeddings, cosine similarity is preferred because document length varies. A paragraph about 'machine learning' and a sentence about 'machine learning' should have high similarity — they're about the same topic. But the paragraph's vector is longer (larger values in each dimension) because it has more tokens. Euclidean distance would see them as far apart because the endpoints are far apart. Cosine similarity sees they point in the same direction and scores them as close.

Concrete example: Vector A = [2, 1], Vector B = [4, 2] (B is 2x A).

Cosine similarity: A and B point in the same direction (angle = 0 degrees), similarity = 1.0.

Euclidean distance: sqrt((4-2)^2 + (2-1)^2) = sqrt(5) = 2.24 — not zero at all!

In practice, we L2-normalize both vectors to unit length before indexing. With normalized vectors, cosine similarity and dot product are equivalent: cosine_sim(A, B) = A · B = sum(A[i] * B[i]). This is computationally cheaper because we can use a single dot product instead of computing angles."

### Challenge 3: Choose a Dimensionality

**Prompt:** An embedding model offers two output sizes: 256 dimensions and 1536 dimensions. Your dataset has 10 million documents. Each query must return in under 100ms. Which do you choose and why?

**Time limit:** 5 minutes.

**Evaluation criteria:**
1. Must discuss the trade-off (accuracy vs. storage vs. speed)
2. Must do the math on storage requirements
3. Must consider the impact on index size and query latency
4. Must make a recommendation with reasoning

**Common mistakes:**
- Always choosing higher dimensions (cost matters!)
- Not considering the recall difference between 256-d and 1536-d
- Ignoring memory constraints

**Model answer:**

"I'd start with the 1536-dimensional vectors but apply quantization.

Here's the math: At 10M documents, 1536-d float32 vectors require 10M x 1536 x 4 bytes = 61.4 GB of raw storage. With HNSW graph overhead, that's roughly 180 GB total. For 256-d vectors: 10M x 256 x 4 = 10.2 GB, roughly 30 GB with HNSW.

The storage difference is real — 180 GB vs 30 GB. But the recall difference is also significant. Going from 1536 to 256 dimensions typically drops recall by 5-15% on semantic similarity tasks.

My recommendation: Use 1536-d vectors with Product Quantization (PQ) compression. PQ compresses each vector to 1 byte per dimension (from 4 bytes), reducing the raw vector storage from 61 GB to 15 GB. HNSW overhead stays similar, but the total is around 75 GB. This fits on a single large-memory node. The recall loss from PQ is typically 1-3%, far less than the 5-15% loss from reducing dimensions.

If I were cost-constrained and couldn't afford a large-memory node, I'd use 256-d vectors as a fallback. But the accuracy improvement from 1536-d is usually worth the additional infrastructure cost."

## Tier 2 — Design

### Challenge 4: Design a Product Search System

**Prompt:** You're building a semantic product search for an e-commerce platform with 2 million products. Users search using natural language ("red running shoes for women"). Key requirements:
- p99 latency under 300ms
- Products updated hourly
- Must support category filtering ("only shoes")
- Team has no GPU infrastructure
- Monthly budget under $10,000

**Time limit:** 15 minutes.

**Scale estimates:**
- 2 million products
- Assume each product description is ~200 words = ~250 tokens
- Peak traffic: 500 QPS
- Storage: 2M x 1536 x 4 bytes (float32) = 12.3 GB raw vectors. ~36 GB with HNSW overhead.

**Key decisions:**
1. **Embedding model:** Use OpenAI text-embedding-3-small via API. No GPU needed, and at $0.13/1M tokens, embedding 2M products costs about $65 one-time, plus ongoing for new products. Monthly embedding cost at 500 QPS x 250 tokens x $0.13/1M = ~$42/month.

2. **Index:** Use Pinecone or Chroma as the vector database. For 2M vectors, a Pinecone pod index handles it easily. Cost: ~$300/month for the starter tier. Alternatively, use FAISS on a single EC2 instance (c6i.4xlarge, ~$400/month).

3. **Hybrid search:** Combine embedding search with category filtering. Store the category as metadata. Use a filtered ANN search to restrict results to the requested category.

4. **Caching:** Use Redis with 5-minute TTL for query embeddings. Expected hit rate: 30%.

5. **Re-ranking:** Skip the cross-encoder re-ranker to stay under budget. The embedding model alone should provide 80-85% recall.

6. **Data freshness:** Use a daily batch pipeline to re-embed updated products. New products appear in search within 1 hour.

**Evaluation rubric:**

| Criterion | Strong Answer | Weak Answer |
|-----------|--------------|-------------|
| Budget awareness | Picks API over self-hosted GPU | Proposes GPU cluster ($3K+/mo) |
| Latency awareness | Accounts for API call latency in p99 | Ignores network latency |
| Filter support | Uses filtered ANN or post-filter mitigation | Ignores filtering entirely |
| Scale constraints | Picks appropriate index for 2M docs | Proposes brute-force search |
| Freshness | Batch update with acceptable delay | Ignores update strategy |

**Model answer:**

"Given the constraints — no GPU, under $10K/month, 2M products, under 300ms — here's my design:

**Embedding:** Use OpenAI's text-embedding-3-small via API. At 500 QPS and ~250 tokens per query, that's about $42/month in API costs. The one-time embedding cost for 2M products is about $65. Well within budget.

**Index:** Use Pinecone's pod index with the s1 pod type. For 2M vectors at 1536 dimensions, Pinecone handles this easily with automatic sharding and replication. Cost is about $350/month. If we have DevOps capacity, we could run FAISS on a c6i.4xlarge EC2 instance for $400/month, but Pinecone reduces operational overhead.

**Caching:** Deploy a Redis instance ($50/month on ElastiCache) to cache query vectors. At 500 QPS with 30% cache hit rate, we reduce embedding API calls to 350 QPS.

**Filtering:** Store product category as metadata. Pinecone supports metadata filtering during ANN search — this applies the filter before searching, not after. It's much more efficient than post-filtering.

**Freshness:** Run a daily batch job that queries for products updated in the last 24 hours, re-embeds them, and upserts them to Pinecone. For hourly updates, run the job every hour with a smaller batch.

**Total cost:** ~$450/month. p50 latency: ~80ms (50ms API + 20ms index search + 10ms retrieval). p99 latency: ~250ms. Under all constraints."

### Challenge 5: Design a Duplicate Content Detector

**Prompt:** Your content platform hosts 50 million articles. You need to detect near-duplicate articles as they're published — before they appear on the site. Near-duplicates are articles that are 80%+ identical but not exact copies (e.g., rewritten introductions, different formatting). Detection must complete within 5 seconds of submission.

**Time limit:** 15 minutes.

**Key decisions:**
1. **Granularity:** Chunk each article into sentence-level or paragraph-level embeddings. A near-duplicate will share most of its chunks with the original.

2. **Index strategy:** Use a two-tier approach. Tier 1: article-level embedding for coarse filtering. Tier 2: chunk-level embeddings for precise matching.

3. **Similarity threshold:** If the article-level cosine similarity exceeds 0.85, compare at the chunk level. If 60%+ of chunks match with over 0.9 similarity, flag as near-duplicate.

4. **Index size:** 50M articles x 10 chunks per article = 500M vectors. This requires a sharded index with at least 10 shards.

5. **Streaming ingestion:** New articles are embedded and searched within the same request. The index must support real-time insert + search.

**Evaluation rubric:**

| Criterion | Strong Answer | Weak Answer |
|-----------|--------------|-------------|
| Granularity | Proposes chunking strategy | Embeds whole articles only |
| Threshold tuning | Discusses precision-recall trade-off | Uses arbitrary threshold |
| Scale | Plans for 500M+ vectors | Underestimates by 10x |
| Real-time | Accommodates 5-second window | Proposes batch processing |

**Model answer:**

"**Architecture:** Two-tier embedding comparison.

Tier 1 uses an article-level embedding (the entire article as one vector). Tier 2 uses paragraph-level embeddings (each paragraph chunk separately).

**Flow on submission:**
1. The new article is chunked into paragraphs (average 10 per article).
2. Each chunk is embedded using text-embedding-3-small.
3. The article-level embedding is also computed (mean of all chunk embeddings).
4. Search the article-level index for articles with cosine similarity above 0.85.
5. If no candidates found: article is unique. Insert chunk embeddings into the index.
6. If candidates found: for each candidate, compute the fraction of chunks that match (cosine similarity > 0.9). If 60%+ match, flag as near-duplicate.

**Scale planning:**
- 50M articles x 10 chunks = 500M vectors at 1536 dimensions
- Raw storage: 500M x 6 KB = 3 TB. With HNSW at 2x overhead: 9 TB.
- Shard across 10 nodes, each handling 50M vectors (~300 GB RAM per node with PQ compression).
- Use FAISS with IVF_PQ for memory efficiency.

**5-second constraint:**
- Embedding: ~100ms per article (10 chunks batched)
- Tier 1 search: ~10ms
- Tier 2 chunk comparison: ~200ms for worst case (100 candidates x 10 chunks each)
- Total: under 500ms. Well within the 5-second window.

**Precision-recall tuning:** The tier 1 threshold of 0.85 catches most duplicates but may miss some aggressively rewritten ones. I'd set this threshold lower (like 0.75) and use the tier 2 chunk comparison as the decision maker. The chunk-level 60% match threshold is tunable based on false-positive rates in production."

### Challenge 6: Design a Real-Time Recommendation System

**Prompt:** A music streaming service with 10 million songs and 50 million active users wants to add a "Discover Weekly"-style personalized recommendation feature. Users should get 30 new song recommendations every Monday. The system must generate these recommendations within 2 hours during off-peak (Sunday night). The team has a budget for GPU infrastructure.

**Time limit:** 15 minutes.

**Key decisions:**
1. **Audio embeddings:** Extract audio features (tempo, key, energy, danceability) and combine with text embeddings from song metadata.
2. **User embeddings:** Build a user profile vector by averaging the embeddings of songs the user has listened to, weighted by listen count and recency.
3. **Two-stage retrieval:** First, find the closest songs to the user's profile vector (candidate generation). Second, diversify candidates across genres and moods to avoid recommending 30 similar songs.
4. **Scale:** 50M users x 30 recommendations = 1.5B similarity searches per week. Must batch efficiently.

**Evaluation rubric:**

| Criterion | Strong Answer | Weak Answer |
|-----------|--------------|-------------|
| Personalization | Uses user behavior vectors | Only uses content similarity |
| Diversity | Includes diversity logic | Recommends 30 identical songs |
| Scale | Batches efficiently | Does real-time search |
| Infrastructure | Uses GPU for batch inference | Ignores hardware needs |

**Model answer:**

"**Embedding strategy:** Two embedding spaces.

Song embeddings combine: (1) audio features from the track analysis (tempo, key, energy, etc.), (2) text embeddings of the title, artist, and genre metadata. These are fused into a single 1536-dimensional vector.

User embeddings are computed weekly: average the embeddings of the last 200 songs the user listened to, with exponential time-weighting (songs from this week count 2x more than songs from last week).

**Batch generation (Sunday night, off-peak):**
1. Load all 50M user profile vectors into a GPU cluster (10 A10G instances).
2. For each user vector, search the song embedding index using FAISS GPU (brute force is feasible at 10M songs on GPU, ~5ms per query).
3. Return top 200 candidates per user.
4. Apply diversity re-ranking: for each user, select 30 songs that maximize (similarity to user profile + diversity from already-selected songs).
5. Write the 30 recommendations to a key-value store (DynamoDB) keyed by user ID.

**Compute estimate:**
- 50M users x 5ms per FAISS search = 250,000 seconds total
- Distributed across 10 GPUs: 25,000 seconds = ~7 hours
- Need more GPUs or a faster index to hit the 2-hour window
- With HNSW index on GPU: ~1ms per search -> 50,000 seconds / 10 GPUs = 1.4 hours. Within window.

**Diversity algorithm:** Maximal Marginal Relevance (MMR). Select the first song that best matches the user profile. For each subsequent song, pick one that maximizes: (similarity to profile) - lambda * max(similarity to already-selected songs). Lambda=0.5 balances relevance and diversity. This prevents the top 30 from all being the same genre."

## Tier 3 — Expert

### Challenge 7: Cross-Lingual Search at Global Scale

**Prompt:** A global e-commerce platform operates in 45 countries and supports 30 languages. They have 500 million products. Users search in their local language, but the product catalog is primarily in English (70%), Chinese (15%), and Spanish (10%), with the remaining 5% in 27 other languages. The system currently uses keyword search per language, which misses cross-lingual matches.

Design a cross-lingual semantic search system that:
- Lets a user search in any supported language and find relevant products from any language's catalog
- Returns results within 300ms p99
- Handles 10,000 peak QPS
- Supports continuous ingestion of 1 million new products daily
- Must work for languages with very different scripts (Arabic, Cyrillic, CJK, Latin)

**Time limit:** 30 minutes.

**Evaluation dimensions a strong answer explores:**

**Embedding model selection:** The critical choice is using a multilingual embedding model like multilingual-E5-large, LaBSE, or text-embedding-3-small (which supports all 30 languages). The model must map all languages to a shared vector space where similar meanings converge regardless of language. A dedicated multilingual model outperforms per-language models because it learns cross-lingual alignment during training.

**Query normalization:** Different languages have different tokenization properties. CJK languages have no spaces between words. Arabic is right-to-left with contextual letter forms. The embedding model must handle all of these consistently. The normalization step must account for the model's specific tokenizer behavior per language.

**Language-specific chunking:** Product descriptions in English average 150 tokens, but German descriptions are typically longer (compound words), and CJK descriptions are shorter. The chunking strategy should use token count (not word count) and adjust max chunk size per language based on the model's tokenizer statistics.

**Cross-lingual evaluation:** Standard recall metrics don't capture cross-lingual quality. You need a separate evaluation set with queries in language A and relevant products described in language B. The MTEB benchmark includes cross-lingual retrieval tasks — use those as a starting point, then build a domain-specific evaluation set.

**Index strategy for 500M products:** Use a multi-tier approach. Tier 1: product-level embedding for broad matching. Tier 2: product description chunk-level for precise matching. The index must support 500M+ vectors. Use FAISS with IVF_PQ on a 32-node cluster.

**Anti-patterns weak answers exhibit:**
- Proposing separate indexes per language (defeats the purpose of cross-lingual search!)
- Not mentioning multilingual evaluation (a tell that you haven't done this before)
- Ignoring the tokenization differences between languages (CJK, Arabic, etc.)
- Proposing machine translation instead of embeddings (translation adds latency and cost)
- Suggesting a single embedding model without verifying it supports all 30 languages

**Discussion of multiple valid approaches:**

**Approach A — Multilingual embedding model (recommended):** Use a single model trained on parallel data across all 30 languages. Simplest architecture, single vector space. Works well for major languages but may have uneven quality for low-resource languages.

**Approach B — Translate-all:** Translate all product descriptions to a single language (English), then use a monolingual embedding model. Pro: Works with any embedding model; high quality for English. Con: Expensive (translating 500M products), adds latency, loses nuance, and some languages don't translate well.

**Approach C — Hybrid:** Use multilingual embeddings for search, but add a translation fallback for low-confidence queries. A query in Swahili gets embedded directly. If the top results have similarity scores below 0.6, translate the query to English and re-search. Adds 2-3 seconds for the fallback path but catches cases where the multilingual model is weak.

The best approach for this scale is Approach A with Approach C as a fallback for low-resource languages. The trade-off is that A is simpler and faster for 95% of queries, while C catches the long tail where the multilingual model underperforms.

### Challenge 8: Embedding System with Privacy Constraints

**Prompt:** A healthcare company wants to build an embedding-based search over patient medical records. There are 100 million patient records, each containing clinical notes, diagnoses, and treatment plans. The system must support:
- Semantic search across medical concepts (e.g., "patients with hypertension who might need beta-blockers")
- Strict HIPAA compliance: embeddings must not contain personally identifiable information (PII)
- The embedding model cannot be hosted on external APIs (data cannot leave the network)
- Queries must complete under 1 second
- Regulatory requirement: any data used for model training must be auditable

**Time limit:** 30 minutes.

**Evaluation dimensions a strong answer explores:**

**PII removal before embedding:** Embeddings trained on PII-laden text will encode that information into the vector space. The solution is not to sanitize the vectors afterward — it's to sanitize the input text before embedding. Strip all PII (names, dates, SSNs, addresses, phone numbers, medical record numbers) using a combination of regex patterns and a fine-tuned NER model. Replace with synthetic placeholders ("Patient X", "Date Y").

**Self-hosted models:** Since external APIs are forbidden, use a self-hosted open-source model like BioBERT or PubMedBERT, which are pre-trained on medical literature. Fine-tune on the PII-sanitized clinical notes to adapt the vocabulary to clinical language. Deploy on internal GPU infrastructure.

**Auditability:** Every embedding operation must be logged: which text was embedded (PII-sanitized version), which model version was used, when it was run, and by which system component. The training data pipeline must record which records were used, their source, and the exact preprocessing steps applied.

**Differential privacy for training:** If fine-tuning on patient data, use differentially private training (DP-SGD) to guarantee that the model doesn't memorize individual records. This adds a privacy budget tracking system — each fine-tuning run consumes some of the privacy budget.

**Access control in search:** Embeddings are only useful if you can search them, but healthcare data requires strict access control. The solution is to associate each embedding with a document-level access control list (ACL). The search system filters results based on the searcher's permissions before returning them. This is implemented as metadata filtering in the vector index.

**The fundamental tension:** Search quality requires rich embeddings, but privacy requires minimizing the information in each vector. This is the core tension of privacy-preserving embeddings. One approach is to use dimensionality reduction (from 1536 to 256 dimensions) to reduce the encoding capacity of each vector, making it harder to extract PII. The trade-off is reduced recall.

**Anti-patterns weak answers exhibit:**
- Proposing an external embedding API (HIPAA violation)
- Not addressing PII in embeddings (regulatory violation)
- Saying "we'll just encrypt the vectors" (encryption protects data at rest, but the vector space still encodes PII)
- Ignoring access control (HIPAA requires least-privilege access)
- Assuming the standard embedding model works for medical text (medical vocabulary is highly specialized)

**Discussion of multiple valid approaches:**

The strongest approach for healthcare is: (1) Build a PII-sanitization pipeline using a fine-tuned clinical NER model, (2) Fine-tune a self-hosted biomedical embedding model on the sanitized data with DP-SGD, (3) Index the sanitized embeddings with document-level ACLs, (4) Log everything for audit, (5) Use dimensionality reduction to limit encoding capacity.

This gives the best recall while satisfying regulatory requirements. The cost is significant engineering effort for the PII pipeline and the self-hosted model infrastructure.

## Tier 4 — System Design Whiteboard

### Challenge 9: Design a Real-Time Embedding Pipeline for Social Media

**Prompt:** A social media platform with 500 million monthly users wants to use embeddings to power:
- Real-time content moderation (detect toxic content within 100ms of posting)
- Personalized content recommendations
- Search across posts in 50+ languages

Requirements: 10,000 posts/second peak, must detect and filter toxic content before it reaches other users, must handle multilingual content with code-switching.

**Time limit:** 45 minutes.

**Scale estimates:**

- 500 million monthly active users
- 10,000 posts per second peak = 864 million posts per day
- Average post length: 150 characters = ~40 tokens
- Embedding dimension: 1536 (text-embedding-3-small)
- Raw vector storage per day: 864M x 1536 x 4 bytes = 5.3 TB/day
- Moderation latency budget: 100ms total (embedding + search + classification)
- 50+ languages, with code-switching in at least 20% of posts

**Architecture diagram description:**

```
                         ┌─────────────────┐
                         │   User Posts    │
                         │   (10K/sec)     │
                         └────────┬────────┘
                                  │
                                  ▼
                         ┌─────────────────┐
                         │  Content Router │
                         │  (Kafka/GCP)    │
                         └────────┬────────┘
                                  │
                    ┌─────────────┼─────────────┐
                    ▼             ▼             ▼
           ┌──────────────┐ ┌──────────┐ ┌──────────┐
           │Embedding     │ │Embedding │ │Embedding │
           │Worker Pool 1 │ │Worker 2 │ │Worker N  │
           │(Moderation)  │ │(Search)  │ │(Recs)    │
           └──────┬───────┘ └──────────┘ └──────────┘
                  │
                  ▼
          ┌──────────────┐
          │Toxic Content  │
          │Classifier     │
          │(NN + Rule)    │
          └──┬───────┬────┘
             │       │
    ┌────────┘       └────────┐
    ▼                         ▼
┌──────────┐           ┌──────────┐
│Flagged   │           │Clean Post│
│Queue     │           │Index     │
│(Review)  │           │(ANN)     │
└──────────┘           └──────────┘
```

The architecture uses a streaming ingestion pipeline with Kafka as the message bus. Posts are fanned out to three parallel embedding worker pools: one for moderation (must complete before the post is visible), one for search indexing, and one for recommendation candidate generation. The moderation pipeline runs a lightweight neural classifier on the embedding to detect toxicity, with a rules-based override layer for edge cases. Clean posts are indexed into the ANN search index. Flagged posts go to a human review queue.

**Key decisions:**

1. **Embedding model:** Use a multilingual embedding model (text-embedding-3-small or multilingual-E5-large) that supports all 50+ languages natively. This is critical because code-switching (mixing languages in a single post) requires a model that understands multiple languages simultaneously. A monolingual model would fail on mixed-language posts.

2. **Streaming vs. batch:** The moderation pipeline must be streaming (real-time) because toxic content must be flagged before it reaches other users. The search and recommendation pipelines can use micro-batching (1-second windows) to reduce compute cost.

3. **Moderation classifier:** Use a two-stage approach. Stage 1: a lightweight logistic regression classifier on the embedding vector (sub-millisecond inference). Stage 2: a small BERT-based classifier for borderline cases (10-20ms). This keeps p99 under 100ms while maintaining high accuracy.

4. **Index freshness:** The search index must support real-time insertion. Use a two-tier index: a small "hot" index for the last 24 hours of posts (in-memory, real-time insert) and a large "cold" index for historical posts (daily rebuild, batch). Hot index: 864M posts/day x 40% kept after moderation = ~345M posts/day. This is too large for a single-node hot index. Use a 10-node cluster where each node handles a hash-based shard of posts.

5. **Code-switching strategy:** Posts with code-switching are detected via language identification at the word level. The embedding model handles mixed-language input, but the moderation classifier may need to be specifically trained on code-switching examples, which are common in social media.

6. **Caching:** Embedding results are cached by content hash. A post that is identical to a previously embedded post (e.g., a viral meme being reposted) skips embedding. Expected cache hit rate: 15-20%.

**Evaluation rubric:**

| Criterion | Strong Answer | Weak Answer |
|-----------|--------------|-------------|
| Real-time constraint | Clearly separates streaming (moderation) from batch (search/recs) | Treats all pipelines as real-time |
| Code-switching | Addresses multilingual model requirement | Ignores language mixing |
| Scale estimates | Includes storage, throughput, and node count | Omits numeric estimates |
| Moderation latency | Designs a sub-100ms path | Proposes multi-second NLP pipeline |
| Index freshness | Uses hot/cold index strategy | Ignores real-time insertion challenge |

**Model answer:**

"The system has three distinct pipelines sharing the same embedding infrastructure.

**Moderation pipeline (real-time, sub-100ms):**
1. Post arrives via Kafka. Language detection runs on the first 50 characters to identify the language(s) present.
2. The post is sent to the embedding worker pool. Each worker runs text-embedding-3-small on GPU (NVIDIA T4, which handles ~500 requests/second per GPU). With 20 GPU workers, we handle 10,000 posts/second.
3. The embedding is passed to a lightweight logistic regression classifier with 256 hidden units. This classifier is trained on a labeled dataset of toxic/non-toxic posts across all 50+ languages. Inference takes 0.5ms.
4. If the classifier score is above 0.9 (high confidence toxic), the post is immediately blocked and sent to a human review queue.
5. If the score is between 0.5 and 0.9, a second-stage BERT classifier runs for 15ms to make a final determination.
6. If the score is below 0.5 (clean), the embedding is forwarded to the search index and the post is published.

**Search pipeline (near-real-time, 1-second batches):**
1. Clean posts are batched every second and inserted into the hot index (10-node FAISS cluster with HNSW).
2. The hot index stays in memory. Every 24 hours, the hot index is merged into the cold index (S3-backed, IVF_PQ compressed).
3. Search queries are embedded with the same model and searched across both hot and cold indexes, with results merged and ranked by timestamp.

**Recommendation pipeline (batch, hourly):**
1. User interest vectors are computed from recent browsing history (last 7 days of impressions).
2. New posts are scored against each user's interest vector using approximate nearest neighbor search.
3. A diversity filter ensures recommendations aren't all from the same topic or language.

**Language handling for code-switching:** The multilingual embedding model handles mixed-language input natively. However, the toxicity classifier is trained on synthetic code-switching data (randomly swapping words between language pairs in the training set). This ensures the model doesn't just learn that 'switching to language X' is a signal of toxicity."

### Challenge 10: Design a Multi-Modal Embedding System for E-Commerce

**Prompt:** An e-commerce company wants to build a unified embedding system that combines:
- Product images (visual similarity for "find similar looks")
- Product descriptions (text similarity for semantic search)
- User purchase history (behavioral similarity for recommendations)

Requirements: 100 million products, 1 billion user interactions, cross-modal search (search by image, find by text).

**Time limit:** 45 minutes.

**Scale estimates:**

- 100 million products
- 1 billion user interactions (purchases, views, clicks, saves)
- Each product has 3-8 images (average 5), 200-word description, 10 metadata fields
- Image embedding dimension: 2048 (ResNet-50 or CLIP ViT)
- Text embedding dimension: 1536 (text-embedding-3-small)
- Combined embedding dimension: 1536 (after fusion projection)
- Raw vector storage: 100M x 1536 x 4 bytes = 614 GB (uncompressed)
- Cross-modal queries: 2,000 QPS peak
- User interaction data grows at 50M interactions/day

**Architecture:**

The architecture has three separate embedding pathways that converge into a fusion layer. Images are embedded with a CLIP ViT model (vision transformer), text with a standard text embedding model, and metadata (category, price range, brand) with a small neural network. The fusion layer uses cross-attention to produce a unified 1536-dimensional embedding that captures all modalities.

**Embedding fusion strategy:**

1. **Late fusion (recommended):** Embed each modality separately, then combine via a learned projection layer. This lets each modality use its optimal embedding model and dimensionality. The fusion layer is a 2-layer transformer with cross-attention between modalities. Output: a single 1536-dimensional vector.

2. **Early fusion:** Concatenate raw pixel data and text tokens into a single input and train a multi-modal model from scratch. Pro: potentially richer interactions between modalities. Con: requires training a massive model from scratch, expensive to update.

3. **Hybrid (ranking-stage fusion):** Use late fusion for candidate generation (fast, scalable). Use a cross-attention re-ranker on the top 100 candidates (more accurate, but too slow for all candidates). This is the most practical approach for production.

**Index strategy:**

1. **Single unified index:** Store all 100M products as 1536-dimensional vectors in a single FAISS index with HNSW. Query by embedding the user's input (image or text) through the same fusion pipeline and searching the unified index. Pros: simple, one index to maintain. Cons: the unified embedding may not be optimal for any single modality's search task.

2. **Per-modality indexes (alternative):** Maintain separate indexes for image embeddings, text embeddings, and metadata embeddings. On query, search all three and merge results. Pros: each index is optimized for its modality. Cons: three indexes to maintain, harder to rank across modalities.

3. **Recommended approach:** Use a single unified index for candidate generation (top 200), then apply a modality-specific re-ranker on the candidates. The unified index captures cross-modal relationships (e.g., "a dress that looks like this image AND has the word 'floral' in the description"). The re-ranker fine-tunes the ranking for the specific query type (image search vs. text search).

**Trade-offs:**

| Approach | Cross-modal quality | Latency | Storage | Maintenance |
|----------|-------------------|---------|---------|-------------|
| Single unified index | High | Fast (1 index) | 614 GB | Low |
| Separate indexes per modality | Medium (harder to fuse) | Moderate (3 searches) | 1.2 TB | High |
| Unified + re-ranker | Highest | Moderate (2 phases) | 620 GB | Medium |

| Trade-offs | Discusses accuracy vs. latency trade-offs | Only presents one option |

**Model answer:**

"The system uses a late-fusion architecture with a cross-attention re-ranker for the best balance of accuracy and latency.

**Ingestion pipeline:**
1. When a product is created or updated, the event triggers the embedding pipeline.
2. Images are embedded using CLIP ViT-L/14 (768-dimensional output). Up to 5 images per product are embedded, and the mean of all image embeddings is taken as the product's visual representation.
3. The product description is embedded using text-embedding-3-small (1536-dimensional). The title, description, and bullet points are concatenated (up to 512 tokens) before embedding.
4. Metadata (category, brand, price tier, color, material) is encoded as a 256-dimensional embedding through a small feed-forward network.
5. All three embeddings are projected into a shared 512-dimensional space through learned linear projections. They are combined using cross-attention: each modality attends to the other two, producing a context-aware representation.
6. The output is a 512-dimensional unified embedding (compress from 1536 to 512 to reduce storage, with minimal accuracy loss).

**Query pipeline:**
1. For text queries: the text is embedded using text-embedding-3-small, then projected through the text projection layer. The resulting vector is used to search the unified index.
2. For image queries: the image is embedded using CLIP ViT-L/14, then projected through the image projection layer. The resulting vector searches the same unified index.
3. For hybrid queries (user uploads an image AND types 'red dress'): both modalities are embedded, projected, and fused through the cross-attention layer. The fused vector searches the unified index.
4. Top 200 candidates are retrieved from the FAISS HNSW index (~10ms).
5. A lightweight cross-modal re-ranker scores the top 200 candidates using a more expensive cross-attention between the query and each candidate. This adds ~50ms but improves recall by 8-12%.
6. Final results are returned in under 200ms p99.

**User interaction integration:**
1. User interactions (purchases, views, clicks) are stored daily.
2. Each user's interaction history is encoded as a weighted average of the unified embeddings of products they've interacted with, weighted by interaction type (purchase=3x, click=1x, view=0.5x).
3. The user embedding is used for personalized search: each query vector is blended with the user embedding (70% query, 30% user profile) before searching. This personalizes results without changing the index.

**Scalability:**
- 100M products x 512 dimensions x 4 bytes = 205 GB raw vectors. With HNSW overhead (~2x), ~410 GB total. Fits on a single large-memory node (e.g., AWS x1e.32xlarge with 4 TB RAM).
- Use 2 nodes for redundancy and read replication. Each node handles 2,000 QPS easily.
- Daily ingestion of new products: ~10,000 products/day. Batch embedding nightly, index rebuild is incremental with HNSW dynamic insertion."

## Tier 5 — Debugging Scenarios

### Challenge 11: The Case of the Silent Recall Drop

**Scenario:** Your team deployed an embedding model upgrade last week. Nothing crashed. No latency spikes. But the product team just reported that user click-through rate on search results dropped 15%. Walk through the debugging process.

**Time limit:** 30 minutes.

**Prompt:** Determine the root cause of the CTR drop after the embedding model upgrade.

**Step-by-step debugging checklist:**

1. **Confirm the metric change.** Is the 15% drop real? Check if CTR was computed the same way before and after the deployment. Sometimes a tracking code change coincides with the model deployment and skews the numbers. Verify against at least two independent metrics (e.g., search-to-purchase conversion rate, session duration after search).

2. **Isolate the change.** The model upgrade changed two things: the embedding vectors themselves AND the scoring function (if the new model has a different similarity distribution). A/B test the old model vs. new model on a 5% traffic split. If the old model shows higher CTR, the embedding model is the root cause. If both show the same CTR, look elsewhere.

3. **Check the similarity distribution.** The new model may produce vectors with different properties. Compute the distribution of cosine similarity scores between queries and their top-10 results. If the new model produces narrower score ranges (all results between 0.7 and 0.8 instead of 0.5 to 0.95), the ranking is less discriminative. Users see less relevant results because the model can't distinguish good matches from mediocre ones.

4. **Inspect the nearest neighbors qualitatively.** Take 100 sample queries. For each query, compare the top-5 results from the old model vs. the new model. Are the new results topically relevant? Are they less specific? Are they returning different categories? This qualitative inspection often reveals the problem immediately.

5. **Check for calibration drift.** The old model may have been trained on in-domain data, while the new model is a general-purpose model. The new model might be better at semantic understanding but worse at capturing the specific signals that drive CTR (e.g., popularity, recency, price competitiveness). CTR is not just about relevance — it's about relevance + desirability.

6. **Test with known golden queries.** You should have a set of 100-500 "golden queries" with known ideal results, maintained by your search quality team. Run these through both models and compute recall@10. If the new model has lower recall, the embedding model is the problem. If recall is the same or better, the problem is elsewhere.

7. **Check the re-ranking layer.** If the system uses a cross-encoder re-ranker after the embedding retrieval, the new embedding model may produce candidates that the re-ranker wasn't trained on. The re-ranker might be scoring the new model's candidates differently (and incorrectly). Re-train the re-ranker on the new model's output distribution.

8. **Monitor the log data.** Are users clicking on the first result less often? Are they scrolling further down the results page? Are they reformulating queries more frequently? Each of these signals tells a different story about what went wrong.

**Possible root causes:**

1. **Embedding model mismatch (most likely).** The new model produces vectors in a different embedding space. Even if both models are semantically strong, the distance relationships between points change. The ANN index was built with the new model's vectors, but the query embeddings are also from the new model — so the search should be internally consistent. But the RELEVANCE of the results changed. The new model may be worse at the specific domain of your product catalog.

2. **Score distribution shift.** The new model's cosine similarity scores are compressed into a narrower range. If the system uses a fixed threshold for filtering results (e.g., "show only results with similarity > 0.7"), the new model may be filtering out results that are actually relevant. Fix: recalibrate thresholds.

3. **Re-ranker staleness.** If a cross-encoder re-ranker was trained on the old model's candidate distribution, it may assign poor scores to the new model's candidates. The re-ranker needs to be re-trained on the new model's output.

4. **Popularity bias change.** The old model may have implicitly learned to boost popular items (through training data that over-represents popular items). The new model, being a general-purpose model, doesn't have this bias. Popular items rank lower, and users click less on less-popular items. Fix: add a popularity boost to the scoring function.

5. **Data leakage or drift.** The new model's training data may have been different from the deployment data. If the model was trained on a different distribution (e.g., general web text vs. e-commerce product data), it will produce different embeddings for the same text.

**How to test each one:**

| Root cause | Test | Expected signal |
|------------|------|-----------------|
| Embedding mismatch | A/B test old vs. new model | Old model has higher CTR |
| Score distribution shift | Plot similarity score histograms for both models | New model has narrower distribution |
| Re-ranker staleness | Re-rank old model's candidates with new model's re-ranker | CTR drops for old model candidates too |
| Popularity bias change | Check correlation between result position and popularity score | New model has lower correlation |
| Data drift | Compute embedding drift between training and production data | High drift score |

**The most likely cause:** Score distribution shift combined with a threshold-based filter. The new model is a general-purpose embedding model that produces vectors with lower variance in similarity scores. The system's filtering threshold was tuned for the old model's score distribution. The new model's top-10 results all cluster between 0.68 and 0.72 similarity, while the old model's ranged from 0.55 to 0.95. The threshold of 0.7 now filters out the 8th-10th results that were previously included. Users see fewer results per query, and the results they see are less differentiated. This leads to lower engagement.

**The fix:**

1. Recalibrate the similarity threshold for the new model. Compute the optimal threshold by running a grid search over thresholds (0.5 to 0.9 in 0.05 increments) on a held-out validation set, measuring recall@10 for each threshold.
2. If the new model's scores are inherently less discriminative, apply a temperature scaling factor to the similarity scores: `score = exp(cosine_sim / temperature)`. A temperature of 0.5 (or similar) stretches the distribution back to the old model's range.
3. Re-train the re-ranker (if one exists) on the new model's candidate distribution.
4. Add a popularity boost component to the ranking: `final_score = 0.7 * similarity + 0.3 * normalized_popularity`. This compensates if the new model lost the implicit popularity bias of the old model.
5. Run a shadow deployment for 48 hours: serve the new model's results but log what the old model would have returned. Compare the two result sets offline. This validates the fix before full rollout.

### Challenge 12: The Cache Stampede

**Scenario:** Your embedding system serves 5,000 QPS during peak hours. The Redis cache has a 60-second TTL. At 2:30 PM, the top 10 queries all have their cache entries expire simultaneously. The embedding service (running on 2 GPU nodes) is hit with 5,000 embedding requests at once.

**Time limit:** 30 minutes.

**Prompt:** Describe what happens and how to prevent it.

**The failure cascade:**

1. **Trigger:** At 2:30 PM, the top 10 queries (accounting for 30% of total traffic, or 1,500 QPS each) have their cached embeddings expire simultaneously. This is a coincidence of TTL timing — all 10 queries were cached at roughly the same time 60 seconds ago.

2. **Cache miss storm:** The Redis cache has no entries for these 10 queries. All 5,000 requests arriving per second need fresh embeddings. The cache miss rate spikes from 5% to 95%.

3. **GPU overload:** The embedding service has 2 GPU nodes, each handling ~50 requests/second (batch size 32, ~6ms per batch). Total capacity: 100 requests/second. But 5,000 requests/second are now arriving. The GPU queue grows by 4,900 requests per second.

4. **Queue explosion:** After 10 seconds, the queue has 49,000 pending requests. Each request now waits 490 seconds before being processed. Response time goes from 50ms to 490 seconds.

5. **Connection pool exhaustion:** The application servers have finite connection pools to the embedding service. With 5,000 requests/second piling up, all connections are consumed. New requests cannot even establish a connection.

6. **Backpressure collapse:** The application servers' thread pools fill up waiting for embedding responses. No threads are available to handle any other work. The entire search service becomes unresponsive.

7. **Downstream cascading:** The search service stops responding. The API gateway times out. Users see error pages or blank search results. The monitoring system triggers alerts as error rates spike.

8. **Recovery:** After 60 seconds, the first batch of completed embeddings returns. But the requests that were waiting have already timed out (typical timeout: 30 seconds). The system needs several minutes to drain the queue. During this time, the cache is repopulated, and the system gradually returns to normal.

**The symptoms:**

- p99 latency spikes from 100ms to 30+ seconds
- Error rate jumps from 0.1% to 40% (timeouts)
- GPU utilization spikes to 100% but throughput doesn't increase (all capacity is consumed by queued requests)
- Redis CPU usage drops (fewer cache hits, fewer lookups)
- Downstream services show increased error rates on search endpoints
- Support tickets spike as users report "search is broken"

**Three prevention strategies:**

**Strategy 1: Staggered TTL (Recommended)**

Instead of a fixed 60-second TTL, add a random jitter to each cache entry's TTL. For each entry, set the TTL to `60 + random(-10, +10)` seconds. This ensures that entries expire at different times. Even if all entries were written simultaneously, they now expire over a 20-second window.

Trade-offs:
- Pro: Simple to implement (one line of code change)
- Pro: No additional infrastructure
- Con: Does not prevent cascading if the database is restarted (all entries have the same age)
- Con: If the burst is large enough, a 20-second window may not be enough to spread the load

**Strategy 2: Cache-Aside with Mutex Locking**

When a cache miss occurs, the first request to miss acquires a distributed lock (via Redis SETNX) and is responsible for computing the embedding. All other requests for the same query wait briefly (up to 50ms) for the lock holder to populate the cache. If the lock holder completes within 50ms, the waiting requests are served from cache. If the lock holder fails or times out, a backup request computes the embedding.

```
function getEmbedding(query):
    cached = redis.get(query)
    if cached is not None:
        return cached

    lock = redis.setnx(query + ":lock", "1", ex=5)
    if lock:
        try:
            embedding = embeddingService.compute(query)
            redis.setex(query, 60, embedding)
            return embedding
        finally:
            redis.delete(query + ":lock")
    else:
        // Wait for the lock holder to populate the cache
        sleep(10ms)
        for i in range(5):
            cached = redis.get(query)
            if cached is not None:
                return cached
            sleep(10ms)
        // Timeout: compute the embedding ourselves
        return embeddingService.compute(query)
```

Trade-offs:
- Pro: Prevents the stampede entirely — only one request per query hits the embedding service
- Pro: Can be layered on top of any TTL strategy
- Con: Adds 10-50ms latency for the waiting requests (in the happy path, this is 0-10ms)
- Con: Lock contention on Redis (minor, SETNX is fast)
- Con: More complex code, potential for deadlock if the lock is not released

**Strategy 3: Proactive Cache Refresh (Background Refresh)**

A background worker runs every 30 seconds and refreshes cache entries before they expire. The TTL is set to 120 seconds (longer than usual). The worker iterates over the most popular queries (maintained in a separate data structure) and re-embeds them proactively. When a cache entry is within 30 seconds of expiring, it's already been refreshed by the worker.

```
// Background worker, runs every 30 seconds
function refreshCache():
    popularQueries = topKQueries.get(100)
    for query in popularQueries:
        ttl = redis.ttl(query)
        if ttl < 45: // refresh when less than 45 seconds remain
            embedding = embeddingService.compute(query)
            redis.setex(query, 120, embedding)
```

Trade-offs:
- Pro: Cache entries never expire unexpectedly — they are always refreshed before they expire
- Pro: Embedding service load is spread evenly across the refresh interval
- Pro: No impact on user-facing latency
- Con: Requires a background worker and a "top queries" data structure
- Con: May waste compute if queries are refreshed but never served (if traffic patterns change)
- Con: More infrastructure to maintain

**Recommended approach:**

Combine Strategy 1 and Strategy 2:
1. Use staggered TTLs (60 + random(-10, +10) seconds) as the baseline. This handles the common case of coincidental expiration.
2. Add cache-aside with mutex locking as a safety net for the rare case of mass expiration (e.g., after a cache flush or restart).
3. If the system has a known set of top queries (e.g., the top 1,000 queries account for 80% of traffic), add Strategy 3's background refresh for those specific queries. This is the belt-and-suspenders approach.

**Implementation notes:**
- The mutex lock must have a timeout (5 seconds) to prevent deadlock if the lock holder crashes.
- The background worker should use a separate, lower-priority queue to avoid competing with user-facing requests.
- Monitor the cache hit rate and the mutex contention rate. If the mutex is acquired more than 1% of the time, the TTL strategy needs adjustment.
- Test the system under failure conditions: simulate a cache flush during peak traffic and verify that the system degrades gracefully instead of collapsing.
