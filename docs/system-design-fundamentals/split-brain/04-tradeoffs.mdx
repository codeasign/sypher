## Core Tension

The central trade-off in split brain prevention is between availability during network partitions and data consistency after the partition heals. Every system must choose a point on this spectrum.

### Quorum Size Trade-off

Larger quorums provide more safety but reduce availability because more nodes must be online to make decisions.

```
Quorum size controls two competing properties:

Safety increases: More nodes must agree, harder to corrupt.
N=3, quorum=2: 2 nodes must agree (67%)
N=5, quorum=3: 3 nodes must agree (60%)
N=7, quorum=4: 4 nodes must agree (57%)

Availability decreases: More nodes must be online.
N=3 with 99% uptime per node:
  P(all 3 up) = 0.99^3 = 97.0%
  P(quorum lost) = P(2 down) + P(3 down)
  = 0.0285 + 0.000001 = 2.85%

N=5 with 99% uptime per node:
  P(all 5 up) = 0.99^5 = 95.1%
  P(quorum lost) = P(3 down) + P(4 down) + P(5 down)
  = 0.098 + 0.0048 + 0.0000001 = 10.3%
```

A 5-node cluster has 10.3% chance of quorum loss per event with 99% per-node uptime. This is 3.6 times worse than a 3-node cluster for the same per-node reliability. The trade-off is fault tolerance versus availability probability.

## Failure Modes

### Failure Mode 1: Undetected Split Brain

The cluster fails to detect that a split brain has occurred. Both sides continue to operate as the authoritative cluster. This is the most dangerous failure mode because it produces silent data corruption.

```
Causes:
- Heartbeat timeout too long (partition detected late)
- Asymmetric partition (one-way communication)
- False positive in partition detection (wrong node fenced)
- Fencing mechanism failure (power-off command not received)

Detection difficulty: HIGH
- Both sides report healthy cluster status
- Monitoring shows all nodes operational
- Data corruption is discovered only when conflicts surface
- Typical discovery: user reports, data integrity checks

Remediation: Automated conflict detection with checksums
             Periodic data consistency audits
             Application-level idempotency keys
```

### Failure Mode 2: Fencing That Kills the Wrong Node

The cluster's fencing mechanism targets a healthy node instead of the partitioned node. This causes unnecessary downtime and potential data loss.

### Failure Mode 3: Rejoin Data Loss

When a partitioned node rejoins the cluster and discards its writes, data can be permanently lost. This is expected and documented behavior, but applications often rely on acknowledged writes that were later discarded.

```
Scenario:
1. Client sends write to Node A (leader, token 1)
2. Node A acknowledges write (reply: "committed")
3. Partition separates Node A from cluster
4. Nodes B and C elect new leader (token 2)
5. Partition heals, Node A rejoins
6. Node A's token-1 writes are discarded
7. Client's write — already acknowledged — is gone

Impact: Application received "200 OK" for a write
        that no longer exists. If the application has
        no retry logic, this is permanent data loss.

Mitigation:
- Idempotency keys on all writes
- Write confirmation from quorum (not just leader)
- Application-level write verification after acknowledgement
- Commit log on client side for replay
```

### Failure Mode 4: Cascading Quorum Loss

A single node failure triggers a cascade that causes additional nodes to fail, eventually causing quorum loss across the entire cluster.

```
Cascade sequence:
1. Node A fails (hardware crash)
2. Quorum drops from 3 to 2 (in a 3-node cluster)
3. Node B and C continue, but with no redundancy
4. Network spike causes Node C's heartbeat to Node B to timeout
5. Node B stops writes (quorum lost: 1/3)
6. Node C also stops writes (quorum lost: 1/3)
7. Entire cluster is unavailable

Root cause: No redundancy after first failure.
            Tight heartbeat timeout in degraded mode.
            No retry mechanism for heartbeat.

Prevention:
- Increase heartbeat timeout in degraded mode
- Use 5-node clusters for production
- Implement exponential backoff for heartbeat failures
```
### Failure Mode 5: False Positive Partition Detection

The system detects a partition that does not exist. A slow node is mistaken for a dead node. This triggers an unnecessary failover and fencing, causing disruption to a healthy cluster.

```
Causes:
- Garbage collection pause on a node (200ms becomes 5s)
- CPU throttling due to noisy neighbor
- Network congestion causing packet loss
- NTP time jump on a monitoring node

Detection: Hard to distinguish from real partition
           because the symptoms are identical

Impact:
- Unnecessary leader election
- Unnecessary fencing (potential downtime)
- Possible data loss from stale leader writes
- Reduced cluster capacity during failover

Mitigation:
- Use suspicion-based failure detection (Phi Accrual)
- Implement slow-node detection separately from dead-node detection
- Require multiple consecutive failures before actioning
- Verify from multiple monitoring nodes before fencing
```

### Failure Mode 6: Configuration Drift

Cluster nodes have different configurations for heartbeat timeouts, quorum sizes, or fencing methods. This causes inconsistent behavior during partitions.

```
Example:
- Node A:    heartbeat_timeout = 3s, quorum = majority
- Node B:    heartbeat_timeout = 5s, quorum = majority
- Node C:    heartbeat_timeout = 3s, quorum = 2 (hardcoded)

During partition:
- Node A detects partition at 3s, triggers election
- Node B still sees A as alive (5s timeout)
- Node B refuses to participate in election
- Node C participates (3s), but quorum requirement is 2
- Only Nodes A and C agree → "quorum" = 2 → proceed
- But Node B disagrees → actual split brain

Prevention:
- Configuration as code, reviewed and deployed together
- Configuration checksum verification at cluster formation
- Reject nodes with mismatched configuration
- Centralized configuration management (etcd, ZooKeeper)
```

## Anti-Patterns

### Anti-Pattern 1: Two-Node Cluster Without Witness

A 2-node cluster has no way to break a tie. Both nodes will claim leadership during a partition. This is always unsafe.

```
Reality: 2-node clusters are deployed frequently for cost.
         Operators justify it as "good enough" for staging.
         Production incidents prove otherwise.

Mandatory: Either add a witness node or accept downtime.
           A shared S3 bucket as witness is better than nothing.
```

### Anti-Pattern 2: Tuning Timeouts to "Eliminate" Partitions

Setting heartbeat timeouts to 60 seconds or more to "avoid" false partitions. This does not eliminate partitions — it merely delays detection, increasing the window for data divergence.

```
Bad configuration:
  heartbeat_timeout: 60s
  lease_duration: 120s
  election_timeout: 90s

Result: A partition takes 60 seconds to detect.
        During that time, both sides accept writes.
        The divergence window is 60+ seconds.
        More data corrupted when partition is detected.

Better: Accept that partitions happen.
        Optimize for fast detection and safe recovery.
        heartbeat_timeout: 3-5s
```

### Anti-Pattern 3: Ignoring Asymmetric Partitions

Many partitions are asymmetric: Node A can reach Node B, but Node B cannot reach Node A. The system must handle this correctly.

```
Asymmetric partition handling:
- Node A receives writes from clients
- Node A cannot replicate to Node B
- Node A deduces: B is dead, continue as sole leader
- Node B receives heartbeats from A, deduces: A is alive, B is follower
- Node B's data is stale but B does not know this
- Clients read from B get stale data

Safe handling:
- Node A must NOT continue if it cannot replicate to quorum
- Even if A can receive client requests, lack of replication
  means A is on the wrong side of a quorum loss
```

### Anti-Pattern 4: Race Conditions in Election Plus Fencing

Starting fencing and election simultaneously. The fencing might target the newly elected leader instead of the old leader.

```
Race:
1. Partition occurs
2. Node B triggers election
3. Node B sends "fence Node A" command
4. Node A goes offline
5. Node B is elected leader
6. Unbeknownst to B, Node A was already reconnected and had quorum
7. Node A was the correct leader. B fenced the wrong node.

Prevention: Complete election before fencing.
            Verify fencing target is actually the stale leader.
            Implement fencing token validation.
```

### Anti-Pattern 5: Manual Override Without Safety Checks

An operator manually intervenes in a split brain scenario without understanding the fencing state.

```
Scenario:
- Split brain detected
- Operator's instinct: "restart both nodes"
- Both nodes restart simultaneously
- Neither can prove leadership
- Cluster deadlock
- Recovery requires external intervention

Better: Follow the runbook. One node at a time.
        Verify fencing state before restarting.
        Never restart both sides simultaneously.
```

## When Quorum Is Unnecessary

Quorum-based split brain prevention is not required in all scenarios. Some systems can operate safely without it.

**Read-only systems:** If the cluster serves only read traffic and data is loaded from an immutable source, split brain is not a concern. Each node can serve reads independently. Examples: CDN edge caches, read replicas for reporting.

**Stateless services:** If no data is stored locally, there is nothing to diverge. A partitioned node simply fails to serve requests. Examples: API gateways, load balancers, authentication services.

**Immutable data stores:** If data is write-once and never modified, a split cannot cause conflicts. Each node has the same data. Examples: version control repositories (read-only), archival storage, blockchain nodes.

**CRDT-based systems:** Conflict-free Replicated Data Types are designed to handle concurrent updates without conflict. They can safely accept writes on both sides of a partition. Examples: collaborative editors, distributed counters, shopping carts.

## Decision Framework

A structured decision framework for choosing a split brain prevention strategy based on system requirements.

### Step 1: Classify Data Criticality

```
Data Category         Consistency Need    Split Brain Tolerance
────────────────      ────────────────    ─────────────────────
Financial ledgers     Strict             Zero tolerance
User authentication   Strict             Zero tolerance
Inventory counts      Moderate           Low tolerance
User profiles         Moderate           Low tolerance
Analytics events      Low                High tolerance
Session data          Low                High tolerance
Logs                  None               No concern
```

### Step 2: Classify Partition Probability

```
Deployment Type       Partition Risk     Typical Prevention
────────────────      ───────────────    ──────────────────
Single datacenter     Low                Quorum + STONITH
Two datacenters       Moderate           Quorum + Witness
Three datacenters     Moderate           Cross-region quorum
Multi-region          High               Hierarchical quorum
Multi-cloud           Very high          AP + CRDT
Edge                  Very high          AP + LWW + manual
```

### Step 3: Select Strategy

```
Strict consistency + low partition risk:
  Strategy: CP with quorum + STONITH
  Cluster: 3 nodes in 3 AZs
  RTO: ~10 seconds, RPO: zero

Strict consistency + moderate partition risk:
  Strategy: CP with quorum + fencing tokens
  Cluster: 5 nodes across 2 datacenters
  RTO: ~5 seconds, RPO: zero

Moderate consistency + high partition risk:
  Strategy: AP with CRDT-based merge
  Cluster: 3 nodes per region
  RTO: instantaneous, RPO: eventual consistency

Low consistency + very high partition risk:
  Strategy: AP with last-writer-wins
  Cluster: peer-to-peer (Cassandra-style)
  RTO: instantaneous, RPO: possible data loss
```

## How to Present Trade-offs in an Interview

When discussing split brain trade-offs in system design interviews, structure your answer around three axes: consistency, availability, and operational complexity.

```
Framework:
1. State the requirement: "My system needs X uptime and
   Y consistency. Split brain is a risk because Z."

2. Propose the primary strategy: "I recommend quorum-based
   prevention with N nodes because it provides the best
   trade-off between safety and cost for this use case."

3. Address the trade-off explicitly: "The cost of this
   choice is that the cluster will lose quorum if M nodes
   fail simultaneously, causing M minutes of downtime.
   The alternative of using M+1 nodes would cost more
   but provide higher availability."

4. Layer additional protection: "I would also add fencing
   tokens as a safety net, even though they add complexity
   to the rejoin process."

5. Cover recovery: "In the event of split brain despite
   prevention, we have an automated recovery procedure
   that fences the stale leader, reconciles data using
   last-token-wins strategy, and resynchronizes from the
   authoritative cluster."

Keep the answer concrete. Use numbers. Avoid hedge words
like "probably" or "might". Commit to specific cluster sizes,
timeouts, and strategies.
```
