## Naive Architecture — Active-Passive Without Fencing

The simplest approach to high availability is an active-passive pair: one primary node handles all requests while a standby node waits to take over. This is the most common pattern in early-stage deployments.

  id="split-brain/architecture-naive"

  caption="A naive active-passive pair has no protection against split brain if both nodes assume leadership"
 content={`
                          ┌──────────────┐
                          │  Load        │
                          │  Balancer    │
                          └──────┬───────┘
                                 │
                 ┌───────────────┼───────────────┐
                 │               │               │
                 ▼               │               ▼
        ┌────────────────┐       │      ┌────────────────┐
        │  Primary       │<──────┼──────│  Standby       │
        │  Node A        │       │      │  Node B        │
        │  Accepts writes│       │      │  Rejects writes│
        │  x = 5         │       │      │  x = 5         │
        └───────┬────────┘       │      └───────┬────────┘
                │                │              │
                └────────────────┼──────────────┘
                                 │
                         Replication Channel
                         (Synchronous or Async)

   Failure scenario:
   Network partition → Both believe other dead
   Standby B promotes self → Two primaries → Split brain
   No fencing mechanism exists to stop A from continuing
`} />

Without fencing, the active-passive pair is vulnerable. When a partition occurs, the standby promotes itself to primary. The original primary continues serving. Both accept writes. The system has no mechanism to prevent this.

Recovery requires manual intervention. An operator must determine which node had the most recent writes, force that node to be primary, and discard writes from the other node. During this process, the database is unavailable.

## Production Architecture — Active-Passive with STONITH

The first step toward safety is adding STONITH fencing. When the standby promotes itself, it first fences the original primary by cutting its power.

<AsciiDiagram
  id="split-brain/architecture-stonith"
  title="Active-Passive with STONITH Fencing"
  alt="Two-node active-passive cluster with STONITH fencing via IPMI, showing the standby cutting power to the primary before promoting itself"
  caption="STONITH fencing prevents split brain by physically powering off the old primary before promoting the standby"
 content={`
                          ┌──────────────┐
                          │  Load        │
                          │  Balancer    │
                          └──────┬───────┘
                                 │
                 ┌───────────────┼───────────────┐
                 │               │               │
                 ▼               │               ▼
        ┌────────────────┐       │      ┌────────────────┐
        │  Primary       │       │      │  Standby       │
        │  Node A        │       │      │  Node B        │
        │  Active        │       │      │  Monitoring    │
        └───────┬────────┘       │      └───────┬────────┘
                │                │              │
                │      Heartbeat Channel        │
                │◄─────────────────────────────►│
                │                               │
                │    Fencing Channel (IPMI)      │
                │◄══════════════════════════════►│
                │                               │
                ▼                               ▼
        ┌──────────────────────────────────────────┐
        │  Management Network                      │
        │  (Out-of-band, independent of data path) │
        │                                          │
        │  Node B detects heartbeat loss →         │
        │  Node B sends IPMI power-off to Node A → │
        │  Node B verifies Node A is offline →     │
        │  Node B promotes self to primary         │
        └──────────────────────────────────────────┘
`} />

The management network is a separate physical network from the data network. This ensures that the fencing command can reach Node A even when the data network is partitioned. Without a separate management network, a network partition can block both data and fencing, making STONITH impossible.

Cost: two servers plus dedicated IPMI management network. The primary limitation is that the cluster can only tolerate one node failure — if the primary fails and the standby cannot fence it, both must stop.

## Multi-Node Architecture with Quorum

For production deployments requiring higher availability, a multi-node cluster with quorum-based voting is the standard approach.

<AsciiDiagram
  id="split-brain/architecture-quorum"
  title="Multi-Node Cluster with Quorum Voting"
  alt="Five-node database cluster with quorum-based leader election, showing the leader, three followers, and the quorum calculation flow"
  caption="A 5-node cluster uses quorum voting to ensure only one leader exists at any time"
 content={`
                          ┌──────────────┐
                          │  Client      │
                          │  Applications│
                          └──────┬───────┘
                                 │
                                 ▼
               ┌ ─ ─ ─ ─ ─ ─ ─ ┼ ─ ─ ─ ─ ─ ─ ─ ─ ┐
                 Quorum-based  │   Leader Election
               │              │                     │
                 ┌────────────┴────────────┐
               │ │   Node A (Leader)       │       │
                 │   Term: 5               │
               │ │   Data: x=10, y=20     │       │
                 └────────┬────────────────┘
               │           │                          │
                          │ Replication
               │ ┌────────┼────────┬────────┐       │
                 │        │        │        │
               │ ▼        ▼        ▼        ▼      │
           ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
           │Node B│ │Node C│ │Node D│ │Node E│
           │Term:5│ │Term:5│ │Term:5│ │Term:5│
           └──────┘ └──────┘ └──────┘ └──────┘
   Quorum size: 3 (floor(5/2)+1)
   Write acknowledgement: Node A + any 2 followers = 3 total
   Failure tolerance: 2 nodes can fail simultaneously
   During partition:
   - Side with 3+ nodes → maintains quorum, continues
   - Side with 2 nodes  → loses quorum, stops writes
   - No fencing needed — quorum ensures safety
`} />

The quorum approach has a critical advantage over STONITH: it does not require a separate management network. Safety is enforced through counting votes rather than through physical isolation. However, the cluster becomes unavailable when quorum is lost. A 5-node cluster that loses 3 nodes stops accepting writes entirely.

### Quorum and Witness Nodes

Even-node clusters require witness nodes to achieve a majority. A 2-node cluster with a witness becomes a 3-vote system requiring 2 votes for decisions.

```
Data Nodes: 2
Witness:    1
Total Votes: 3
Quorum:     2

Failure tolerance: 1 data node or the witness
But not: both data nodes (lose all data)
         or 1 data node + witness (lose quorum)
```

## Multi-Region Architecture

Geo-distributed deployments introduce additional complexity. Network latency between regions is higher than within a region, and regional partitions are more likely.

   │  └──┘  └──┘      │      │  └──┘  └──┘      │     │  └──┘          │

   │  Zone: primary    │      │  Priority: 2     │     │  Priority: 3   │
   └────────┬──────────┘      └────────┬─────────┘     └────────┬───────┘
            │                          │                       │
            │       Cross-Region       │                       │
            │       Replication        │                       │
            │◄────────────────────────►│◄──────────────────────►│
            │                          │                       │
            ◄──────────────────────────────────────────────────►│
   Quorum Configuration:
   ─────────────────────
   Total nodes: 5
   Quorum: 3
   Regional preference:
     us-east-1: 2 votes
     us-west-2: 2 votes
     eu-west-1: 1 vote
   Quorum regions:
     us-east-1 alone:    2/5 → no quorum
     us-east-1 + eu:     3/5 → quorum (leader stays in us-east-1)
     us-west-2 + eu:     3/5 → quorum (leader moves to us-west-2)
     us-east-1 + west-2: 4/5 → quorum (leader in us-east-1)
`} />

Cross-region quorum introduces latency for writes. Each write must be acknowledged by nodes in remote regions. Typical cross-region latency is 60-200 milliseconds for us-east to us-west, and 150-350 milliseconds for us-east to eu-west. This adds 60-350 milliseconds to write latency.

### Regional Partition Handling

When a full region is isolated, the remaining regions must decide whether they have quorum.

```
Scenario: us-east-1 partitioned away
- Remaining: us-west-2 (2 votes) + eu-west-1 (1 vote) = 3 votes
- Quorum required: 3 votes total / 2 = 2 votes
- 3 >= 2 → quorum maintained
- us-west-2 becomes new leader region
- eu-west-1 participants in replication
- us-east-1 detects quorum loss, stops writes

Scenario: us-west-2 and eu-west-1 both partitioned away
- Remaining: us-east-1 (2 votes)
- Quorum required: 3
- 2 < 3 → quorum lost
- us-east-1 stops writes
- Cluster is unavailable
```

## Integration with Databases

### PostgreSQL with Patroni

Patroni manages PostgreSQL high availability using a Distributed Configuration Store (DCS) like etcd or ZooKeeper to provide quorum-based leader election.

```
Patroni Architecture:
- PostgreSQL: data storage
- Patroni: cluster manager (runs on each node)
- etcd/ZooKeeper: DCS for leader election and fencing tokens
- HAProxy or similar: connection routing

Split brain prevention:
- Leader holds a lease in etcd with fencing token
- Replicas monitor leader via etcd and PostgreSQL streaming replication
- On leader failure, replicas attempt to acquire lock in etcd
- First replica to acquire lock becomes new leader
- Old leader's connection is rejected by etcd (stale token)
- Patroni uses the "no overdue leader" safety check
```

### MongoDB Replica Sets

MongoDB uses a Raft-like consensus algorithm for replica set elections. A replica set has a primary and multiple secondaries.

```
MongoDB Protection:
- Majority quorum required for primary election
- A secondary with higher priority can trigger election
- In a 3-node set, 2 nodes needed for quorum
- "rollback" phase recovers writes from former primary

Concern: MongoDB's automatic failover can trigger split brain
if network partitions are asymmetric and election timeouts
are not configured correctly. Setting electionTimeoutMillis
to at least 10 seconds is recommended for production.
```

### Cassandra

Cassandra uses a peer-to-peer architecture with no single leader. Each node accepts writes for the keys it owns. This eliminates leader-based split brain but introduces a different problem: conflicting writes from different nodes.

```
Cassandra Protection:
- No leader election → no split brain from dual leadership
- Hinted Handoff: stores writes for unavailable nodes
- Read Repair: fixes inconsistencies on read
- Last-writer-wins per column using timestamps
- Gossip protocol detects node failures

Split brain-like scenario:
- Two nodes in different datacenters accept writes for same key
- Both writes have timestamps
- Last timestamp wins when nodes sync
- Problem: clock skew can cause wrong version to win
- Solution: use Cassandra's monotonic clock improvements
```

## Capacity Planning

### Node Count Selection

```
Cluster Size   Writes/sec     Latency p99     Cost
  3            10,000         5 ms            Medium
  5            30,000         8 ms            High
  7            50,000         12 ms           Very High
  9            70,000         15 ms           Extreme

Recommendations:
- Development/staging:   3 nodes
- Production (standard): 5 nodes
- Production (mission-critical): 7 nodes across 3 AZs
- Global (multi-region): 9 nodes across 3 regions
```

### Scaling Up

When adding nodes to an existing cluster, consider the quorum impact during the transition.

```
Current: 3 nodes, quorum = 2
Adding: 2 new nodes

Transition states:
1. Add Node D as learner (no vote) → quorum still 2
2. Add Node E as learner (no vote) → quorum still 2
3. Promote D and E to voters (now 5 nodes) → quorum becomes 3
4. Critical: during step 3, a partition could leave 2 old + 2 new
   This would be 4 votes, quorum=3, still safe

Never remove a node without first confirming quorum is maintained.
Never reduce quorum below N/2+1 of the full cluster size.
```

## Operations

### Monitoring for Split Brain

Key metrics to monitor for early detection of split brain conditions.

```
Metric                    Warning        Critical       Action
────────                  ───────        ────────       ──────
Quorum loss risk          N-2 nodes      N-1 nodes      Add nodes
Heartbeat success rate    <95%           <85%           Check network
Election count/hour       >2             >5             Investigate
Lease renewal failures    3 in 5 min     5 in 5 min     Check DCS
Fencing failures          1 in 1 hour    Any            Fix IPMI
Rejoin duration           >30s           >5min          Check sync
Token gaps found          Any            >2 tokens      Emergency
```

### Split Brain Recovery Runbook

```
1. DETECT
   - Alert: split brain detected or suspected
   - Check cluster logs for dual leadership messages
   - Verify via independent monitoring system

2. ASSESS
   - Identify which side has the authoritative data
   - Compare fencing tokens or term numbers
   - Determine which writes would be lost on each side

3. FENCE
   - If STONITH available: power off the non-authoritative side
   - If STONITH unavailable: disable switch ports
   - If neither works: shut down the application on that side

4. RECOVER DATA
   - Export any unique writes from the non-authoritative side
   - Apply application-level conflict resolution
   - Initiate full data sync from authoritative to non-authoritative

5. REJOIN
   - Bring non-authoritative nodes back online
   - Verify full synchronization
   - Return cluster to normal operations

6. POST-MORTEM
   - Root cause analysis
   - Implement network redundancy
   - Review fencing configuration
   - Update monitoring thresholds
```

### Testing Split Brain Resilience

```
Test Type                Frequency       Method
────────                 ────────        ──────
Network partition        Monthly         Chaos Mesh, Gremlin
Node isolation           Monthly         Block network interface
Power failure            Quarterly       Physical power cut
STONITH failure          Quarterly       Simulate IPMI failure
Clock drift              Quarterly       NTP manipulation
Recovery drill           Quarterly       Full runbook execution
```

Each test should verify that the cluster detects the condition, applies fencing correctly, and recovers without data loss. Document the expected behavior before each test and compare with the actual outcome.

## Kubernetes Architecture for Split Brain Awareness

Kubernetes adds another layer of complexity. Pods, nodes, and clusters each have their own partition scenarios.

### StatefulSet with 3 Replicas

```
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: db-cluster
spec:
  replicas: 3
  podManagementPolicy: OrderedReady
  updateStrategy:
    type: RollingUpdate
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringScheduling:
            - labelSelector:
                matchExpressions:
                  - key: app
                    operator: In
                    values:
                      - db-cluster
              topologyKey: topology.kubernetes.io/zone
      containers:
        - name: database
          image: database:3.5
          livenessProbe:
            exec:
              command: ["cluster-check", "--quorum"]
            initialDelaySeconds: 30
            periodSeconds: 10
          readinessProbe:
            exec:
              command: ["cluster-check", "--leader"]
            initialDelaySeconds: 10
            periodSeconds: 5
```

The pod anti-affinity ensures each replica runs in a different availability zone. The liveness probe checks quorum health — a pod that loses quorum is restarted. The readiness probe checks leader status — a pod that is not the leader is removed from service.

### Kubernetes Network Policies

Network policies prevent cross-namespace access during partitions, acting as a fencing mechanism.

```
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-fencing
spec:
  podSelector:
    matchLabels:
      app: db-cluster
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: db-cluster
      ports:
        - port: 5432
        - port: 8080
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: db-cluster
      ports:
        - port: 5432
        - port: 8080
```

When a node is fenced, the Kubernetes control plane can update the network policy to block traffic to and from the fenced pod. This provides network-level fencing without IPMI access.

### Operator Pattern

Kubernetes operators can manage the full split brain lifecycle by watching cluster state and taking action.

```
Operator responsibilities:
- Watch cluster membership changes (via StatefulSet status)
- Detect leader status (via application-level health checks)
- Trigger fencing (via pod deletion, network policy update, or node cordoning)
- Manage recovery (via init containers that sync state before starting)
- Generate fencing tokens (via CRD based on generation counter)

Kubernetes-native fencing:
1. Operator detects stale leader via application health check
2. Operator deletes the stale leader pod
3. StatefulSet controller replaces the pod (new PVC if needed)
4. New pod starts with empty data directory
5. Init container syncs from current leader
6. Application container starts as replica
```

### DNS-Based Routing

Split brain awareness requires routing clients to the correct leader. DNS-based discovery must account for partition state.

```
Standard approach:
  db-primary.example.com → DNS A record → Leader IP
  db-replica.example.com → DNS A record → Replica IPs

Split brain aware:
  Each node runs a health check that reports:
    - Is this node the leader? (boolean)
    - Does the cluster have quorum? (boolean)
    - What is the current fencing token? (integer)
    - Can this node reach a majority? (boolean)

  Health check endpoint: GET /cluster/status
  Response: { leader: true, quorum: true, token: 7, healthy: true }

  Load balancer uses health check to route traffic:
    - Only route to nodes that report leader=true AND quorum=true
    - If no node reports both, reject all traffic (quorum lost)
    - If multiple nodes report leader=true, this is split brain
      - Load balancer blocks all traffic
      - Alert triggers immediately
```

## DNS and Service Discovery Integration

Service discovery systems like Consul and etcd provide distributed key-value stores that can be used for split brain monitoring.

```
Consul health checks:
  check {
    id: "db-leader"
    name: "Database Leader Status"
    notes: "Returns true if this node is the cluster leader"
    http: "http://localhost:8080/cluster/status"
    interval: "5s"
    timeout: "2s"
    deregister_critical_service_after: "30s"
  }

  When a node reports it is no longer leader or loses quorum:
    1. Consul marks the service instance as unhealthy
    2. Traffic is routed away from the unhealthy instance
    3. Service mesh (Envoy, Linkerd) removes the instance from
       its connection pool
    4. If all instances are unhealthy, the entire service is
       flagged as degraded
```

## Integration with Message Queues

Message queues introduce additional split brain considerations because they store unprocessed messages that can be lost or duplicated.

### Kafka Partition Reassignment

Kafka's controller can become split if the ZooKeeper ensemble that manages cluster metadata splits. Each side might assign different partition leaders.

```
Kafka Controller Split:
- ZooKeeper ensemble has 3 nodes
- Partition isolates controller from 2 ZooKeeper nodes
- Controller cannot write to ZooKeeper (no quorum)
- Remaining ZooKeeper nodes elect new controller
- Two controllers exist — split brain

Impact:
- Both controllers attempt partition leadership assignments
- Some partitions have multiple leaders
- Producers write to both leaders
- Consumers read from one leader, missing writes from the other
- Data is duplicated or lost

Mitigation:
- Use 5-node ZooKeeper ensemble (survives 2 failures)
- Set unclean.leader.election.enable=false
- Configure min.insync.replicas=2 for critical topics
- Use Kafka's built-in fencing via epoch numbers
```

### RabbitMQ Queue Mirroring

RabbitMQ's queue mirroring across nodes requires careful split brain handling.

```
RabbitMQ quorum queues:
- Use Raft consensus for queue state
- Require majority of nodes to acknowledge messages
- Survive N/2 node failures
- Automatic leader election when current leader fails
- Fencing via Raft term numbers

Policy recommendation:
  rabbitmqctl set_policy ha-quorum \
    "^critical\." \
    '{"queue-mode":"quorum","quorum-initial-group-size":3}' \
    --apply-to queues

  This ensures all queues matching "critical." use quorum
  replication with 3 nodes minimum.
```
