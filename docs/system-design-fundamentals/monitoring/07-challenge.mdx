## Tier 1: Foundation

### Challenge 1: Agent Buffer Configuration

Your application runs on 100 instances, each emitting 2,000 metrics per second. The agent uses a ring buffer with 10,000 event capacity and a flush interval of 15 seconds. The monitoring aggregator occasionally experiences 30-second outages due to deployments.

**Question**: Will the agent lose data during a 30-second aggregator outage? Calculate the buffer capacity in seconds and explain what happens when the buffer fills.

**Model Answer**: The agent emits 2,000 metrics per second, and the buffer holds 10,000 events. The buffer provides 10,000 / 2,000 = 5 seconds of capacity before it fills. The flush interval is 15 seconds, meaning the agent sends data every 15 seconds but the buffer can only hold 5 seconds of data. During a 30-second aggregator outage, the agent attempts to flush at t=15s and t=30s. Both attempts fail. The buffer fills at t=5s (5 seconds into the outage). For the remaining 25 seconds, all new metrics overwrite the oldest unread data in the ring buffer.

The data loss window is 25 seconds out of every 30-second outage. The fix is to increase the buffer capacity to at least 60,000 events (30 seconds x 2,000 metrics/second) or reduce the flush interval to 5 seconds so data is sent more frequently, reducing the accumulation rate.

### Challenge 2: Cardinality Debugging

Your team deploys a new version of the recommendation service. Within 10 minutes, the time-series database starts rejecting writes, and all dashboards for all services go blank. The TSDB team reports: "too many series" errors.

**Question**: What is the most likely cause, and how do you find and fix it? What metrics should you check to confirm your hypothesis?

**Model Answer**: The most likely cause is a cardinality explosion from a new metric dimension introduced in the deployment. The recommendation service added a label that creates a unique value per user or per request -- such as `user_id`, `session_id`, or `request_id` -- turning a metric with 100 series into a metric with 1 million series.

To confirm, check the series count metric for the TSDB. Most monitoring systems expose a metric like `tsdb_series_count` or `prometheus_tsdb_head_series_count`. If the count jumped from 10,000 to 1,010,000 at the time of deployment, the hypothesis is confirmed.

To find the offending metric, query the TSDB for the metrics with the highest series count. In Prometheus, use `count by (__name__) ({__name__=~".+"})`. The metric with 1 million series will be immediately visible.

The fix is to roll back the deployment immediately, then remove the high-cardinality label from the instrumentation code. The label should be moved from a metric dimension to a log field or trace attribute, where high cardinality is expected and handled efficiently.

### Challenge 3: Alert Fatigue Analysis

Your team has 300 alerting rules configured across 20 services. The on-call engineer receives an average of 40 alerts per night. Post-incident analysis shows that 95 percent of alerts are false positives. The team has started ignoring alerts, and a real incident went undetected for 2 hours.

**Question**: Design a systematic approach to reduce alert fatigue by 90 percent within 30 days. What specific steps do you take, and how do you measure success?

**Model Answer**: The approach has four phases:

Phase 1 (Day 1): Audit all 300 alert rules. For each rule, answer: "When was the last time this alert fired during a real incident?" If the answer is "never" or "more than 30 days ago," the alert is removed. This typically removes 50-60 percent of rules.

Phase 2 (Day 7): For the remaining rules, add a `for` duration. Start with 5 minutes for latency-based alerts and 1 minute for error-rate-based alerts. The `for` duration prevents alerts from firing on transient spikes, which are the most common source of false positives.

Phase 3 (Day 14): Replace static thresholds with dynamic thresholds. Use the previous 7 days of data as a baseline and alert on deviations from the baseline rather than absolute values. For example, instead of "CPU > 80 percent," use "CPU > 2 standard deviations above the 7-day baseline."

Phase 4 (Day 30): Measure the result. The number of alerts per night should drop from 40 to 4. The false positive rate should drop from 95 percent to 20 percent. The remaining false positives are investigated and addressed individually.

Success metrics: alerts per night, false positive rate, mean time to detect real incidents, and on-call satisfaction survey scores.

## Tier 2: Design

### Challenge 1: Design a Metrics Pipeline for a Fintech Platform

**Design a real-time metrics pipeline that handles 500,000 data points per second with 99.999 percent durability. The pipeline must support sub-second query latency for the last 5 minutes of data and 30-day retention with 1-minute resolution.**

**Model Answer**: The pipeline uses four layers:

Layer 1 -- Agent: Every instance runs a Go-based agent with a 500,000-event ring buffer. The agent uses protobuf serialization with gzip compression, sending batches of 10,000 events every 10 seconds. The agent retries failed sends with exponential backoff (1s, 2s, 4s, 8s, 16s) before dropping data. The buffer provides 1,000 seconds of capacity at the expected rate, ensuring durability through prolonged aggregator outages.

Layer 2 -- Aggregator: A cluster of 20 aggregator instances behind a load balancer, each handling 25,000 data points per second. Each aggregator validates schema, enforces cardinality limits (100,000 series per metric), and routes data to one of 16 TSDB partitions based on hash of the metric name. If an aggregator fails, the load balancer redistributes traffic to the remaining instances.

Layer 3 -- Storage: 16 TSDB partitions, each with 3 replicas (2 within-region, 1 cross-region). Each partition uses a WAL for durability, a 256 MB memtable for in-memory buffering, and LSM-tree segments on disk. The hot tier (5 minutes) is entirely in memory, served by the memtable. The warm tier (5 minutes to 7 days) is on SSD. The cold tier (7 to 30 days) is downsampled to 1-minute resolution and stored on HDD.

Layer 4 -- Query: 4 query frontends that fan out to all 16 partitions. Queries over the last 5 minutes are served from the in-memory memtable with sub-millisecond latency. Queries over longer ranges read from segment files and are cached in a Redis cluster with a 60-second TTL.

Durability guarantee: Data is written to the WAL and acknowledged to the agent before the write is considered successful. The WAL is replicated to 3 nodes before acknowledgment. In the worst case (all 3 replicas fail simultaneously), data is lost only for the last 1 ms of writes (the WAL fsync interval).

### Challenge 2: Design a Multi-Tenant Monitoring System

**Design a multi-tenant monitoring system where 500 teams each run 10 services. Each team must see only their own metrics, and the system must enforce per-team resource quotas. The total scale is 100,000 services and 10 million data points per second.**

**Model Answer**: The system uses a tenant-isolated architecture with a shared control plane.

Data isolation: Each tenant's data is routed to a dedicated TSDB instance or a dedicated partition within a shared TSDB cluster. For small tenants (under 10,000 data points per second), use shared partitions with tenant-ID labels. For large tenants (over 10,000 data points per second), use dedicated TSDB instances. This hybrid approach balances isolation granularity with operational efficiency.

Quota enforcement: The aggregator enforces per-tenant quotas at three levels. First, a data point rate limit (per-tenant token bucket). Second, a time series limit (100,000 series per tenant). Third, a storage quota (1 TB per tenant). When a tenant exceeds a quota, the aggregator either rejects the data with a 429 response or drops the oldest data, depending on the tenant's plan.

Authentication: Each tenant has an API key that is included in every agent request. The aggregator validates the API key against a tenant database and assigns the request to the tenant's routing and quota configuration.

Global dashboard: A shared Grafana instance uses per-tenant data sources. Each Grafana user is scoped to their tenant's data sources through Grafana's organization feature. Cross-tenant dashboards are available only to the platform team.

Operational isolation: A single tenant's data storm cannot affect other tenants. If one tenant's cardinality explodes, the aggregator's per-tenant cardinality limit rejects the excessive data. If one tenant's TSDB instance fails, only that tenant's dashboards are affected. The platform team monitors per-tenant pipeline health and receives alerts when any tenant's error rate exceeds 1 percent.

### Challenge 3: Design a Global SLO Tracking System

**Design a system that tracks SLO compliance for 200 services across 3 regions with 99.99 percent accuracy. The SLO window is 30 days, and the system must process 1 billion requests per day.**

**Model Answer**: The system uses a two-phase architecture: counting and reporting.

Phase 1 -- Counting: Each request generates an SLO event that records whether the request was successful (within latency budget and no errors) or failed. SLO events are emitted as structured logs from the application, not as metrics, because SLO events are high cardinality (one per request). The log stream is processed by a stream processor (Flink or Kafka Streams) that maintains a rolling 30-day count of good events and total events per service, per region.

The stream processor maintains a sliding window with 1-minute granularity. For each service-region pair, it stores the count of good events and total events for each minute. The 30-day window is exactly 43,200 minutes. The stream processor uses RocksDB-backed state for durability and checkpointing for crash recovery.

Phase 2 -- Reporting: A reporting service queries the stream processor at 1-minute intervals to compute the current SLO compliance percentage: `good_events / total_events * 100`. The result is stored as a metric in the monitoring TSDB for dashboarding and alerting.

Accuracy: The counting uses exactly-once semantics from the stream processor. The Flink/Kafka Streams checkpointing mechanism ensures that each SLO event is counted exactly once, even through process crashes and restarts. The 99.99 percent accuracy target is achievable because the counting is deterministic -- no sampling, no approximation, no aggregation loss.

Alerting: Multi-burn-rate alerts fire when the error budget is consumed at a rate that would exhaust the budget within 1 hour (critical), 6 hours (warning), or 3 days (info). Each alert includes the current burn rate, the remaining error budget, and the projected budget exhaustion time.

## Tier 3: Expert

### Challenge 1: Design a Monitoring System for a Self-Healing Infrastructure

**Your infrastructure runs 50,000 containers across 2,000 nodes in a Kubernetes cluster. The cluster auto-scales, auto-heals, and self-upgrades. Containers are ephemeral -- they are created and destroyed every few minutes. The monitoring system must handle this churn without generating false alerts for every container termination.**

**Discussion**: This challenge tests your understanding of monitoring in dynamic environments where the concept of a "node" or "instance" is ephemeral.

The key insight is that the monitoring system must track the lifetime of each container and correlate metrics across container generations. A container that is terminated as part of a normal scaling event should not generate an alert. A container that crashes unexpectedly should generate an alert, but only if the crash is not part of a rolling update.

Approach: Use Kubernetes service discovery to track container lifecycles. Each container is assigned a unique ID at creation. The monitoring agent is a sidecar container that shares the lifecycle with the application container. When the application container is terminated, the monitoring sidecar emits a final metrics batch and then terminates.

The `up` metric is not useful for individual containers because they are expected to be terminated frequently. Instead, track the `up` metric at the Kubernetes deployment level. If the deployment has 10 replicas and only 8 are running, fire an alert. If the deployment has 10 replicas and 10 are running, but 5 were replaced in the last minute, do not fire an alert -- that is a normal rolling update.

For alerting, use the `for` duration parameter aggressively. A container crash that is immediately replaced by a new container should not generate an alert. Only alert if the replacement does not happen within 60 seconds.

The discussion should also cover how to handle metrics from terminated containers. Old container metrics should be garbage collected from the TSDB within 15 minutes of container termination. The TSDB's retention policy should use a TTL based on the container's last reported timestamp, not the current time.

### Challenge 2: Cost-Optimize a Monitoring System at Petabyte Scale

**Your monitoring system stores 2 PB of data and costs $500,000 per month in storage and compute. The CFO asks you to reduce costs by 50 percent without reducing the number of metrics collected or the alerting coverage. The system tracks 50,000 services and 10,000 team members use the dashboards daily.**

**Discussion**: This challenge tests your ability to optimize a large-scale system within business constraints.

The constraint "no reduction in metrics collected or alerting coverage" eliminates the obvious solution of sampling less. The optimization must come from storage efficiency, query efficiency, and tiered storage.

Approach 1 -- Compression: The TSDB's default compression achieves 3:1. Switching to a custom compression algorithm optimized for time-series data can achieve 10:1 for most metric types. Uber's M3 achieves 1.5 bytes per data point using XOR compression. At 2 PB, 10:1 compression reduces storage to 200 TB, cutting the storage cost by 80 percent.

Approach 2 -- Tiered Storage: Move data older than 7 days to object storage (S3, GCS) instead of keeping it on SSD. Object storage is 10x cheaper than provisioned SSD. At 2 PB with 80 percent of data older than 7 days, moving 1.6 PB to object storage at $0.023/GB/month saves $36,800 per month compared to SSD at $0.08/GB/month.

Approach 3 -- Query Optimization: The biggest compute cost comes from expensive queries that scan millions of time series. Implement query cost tracking: each query is assigned a "cost score" based on the number of series scanned and the time range. Expensive queries are throttled or cached. Dashboard queries use pre-computed recording rules instead of running raw aggregations on every load. This reduces compute cost by 40 percent.

Approach 4 -- Data Lifecycle Management: Not all metrics need 30-day retention at full resolution. Apply retention policies by metric category. Error rate metrics: 90 days at full resolution. Request latency metrics: 30 days at full resolution, 1 year at 1-minute downsampling. Infrastructure metrics (CPU, memory): 14 days at full resolution, 90 days at 5-minute downsampling. This reduces storage by 60 percent before any other optimization.

Combined impact: 80 percent storage reduction from compression, 50 percent compute reduction from query optimization, and 60 percent storage reduction from tiered retention. The total cost reduction is approximately 60 percent, exceeding the 50 percent target. The remaining $200,000 per month is the baseline cost of operating a petabyte-scale monitoring system, and further reductions would require reducing the number of metrics collected or the retention period.

## Additional Practice Challenges

### Challenge 4: Monitoring Pipeline Reliability

Your monitoring pipeline has been experiencing intermittent data loss. The agents report that they are sending data successfully, but the TSDB shows gaps in the data. The gaps occur randomly across different services, times, and nodes. Diagnose the possible causes and propose a systematic approach to finding the root cause.

**Discussion**: This challenge tests your ability to debug a distributed monitoring pipeline. The fact that agents report success but data is missing means the failure is in the transport or storage layer.

Possible causes in order of likelihood:

1. Network connectivity issues between agents and aggregators. The agent might successfully send data to a load balancer that then fails to reach the aggregator. The agent receives a TCP ACK from the load balancer but the data never reaches the aggregator.

2. Aggregator rate limiting. The aggregator might be rate-limiting data from certain agents without returning an error code. Some rate limiters silently drop data when the rate limit is exceeded, especially if token bucket configurations are misaligned.

3. TSDB write rejection. The TSDB might reject data due to out-of-order timestamps, schema violations, or cardinality limits. The aggregator might log these rejections but not report them to the agent.

4. Aggregator routing misconfiguration. The aggregator might route data to the wrong partition or drop data that does not match any routing rule.

5. Clock skew between agents and aggregators. If the agent's clock is more than 5 minutes off from the aggregator's clock, the TSDB rejects the data as "too old" or "too far in the future."

Systematic approach: add a unique sequence number to each batch of data from each agent. The TSDB records the highest sequence number seen per agent. The monitoring team compares the expected sequence number (sent by the agent) with the actual sequence number (stored in the TSDB). Any gap indicates data loss. Correlation with aggregator logs, TSDB logs, and network metrics identifies the root cause.

### Challenge 5: Anomaly Detection Design

Design an anomaly detection system that identifies unusual patterns in request latency without manual threshold configuration. The system must handle daily traffic patterns (higher during business hours, lower at night), weekly patterns (weekdays vs weekends), and seasonal patterns (holiday traffic spikes).

**Discussion**: This challenge tests your understanding of statistical methods for monitoring and the tradeoffs between different approaches.

Approach 1 -- Seasonal Decomposition: Use a time-series decomposition algorithm (STL or Prophet) to split the latency metric into three components: trend, seasonality, and residual. The residual represents the deviation from expected behavior. If the residual exceeds 3 standard deviations from its historical mean, fire an anomaly alert.

```python
# Pseudocode for seasonal decomposition
from statsmodels.tsa.seasonal import STL

def detect_anomaly(latency_series, timestamp):
    # Decompose into trend + seasonal + residual
    stl = STL(latency_series, period=1440)  # 24 hours at 1-minute resolution
    result = stl.fit()
    residual = result.resid

    # Anomaly if residual exceeds 3 sigma
    threshold = 3 * residual.std()
    current_residual = residual.iloc[-1]
    return abs(current_residual) > threshold
```

Approach 2 -- Moving Average with Dynamic Threshold: Use a moving average of the last 24 hours of data as the baseline. Set the threshold as the moving average plus 3 times the moving standard deviation. This approach adapts to changing traffic patterns automatically.

```python
# Pseudocode for dynamic threshold
def dynamic_threshold(values, window_size=1440):
    rolling_mean = values.rolling(window=window_size).mean()
    rolling_std = values.rolling(window=window_size).std()
    upper_bound = rolling_mean + 3 * rolling_std
    lower_bound = rolling_mean - 3 * rolling_std
    return upper_bound, lower_bound
```

Approach 3 -- Machine Learning: Train a model (Isolation Forest or Autoencoder) on historical latency data. The model learns the normal pattern of latency and flags deviations. This approach handles complex patterns that simple statistical methods miss, but requires training data, model retraining, and monitoring of the model itself.

Tradeoffs: Seasonal decomposition is simple and interpretable but requires manual configuration of the period. Moving averages adapt automatically but can be slow to react to gradual changes. ML models are powerful but complex to maintain. For most production systems, seasonal decomposition with a 24-hour period catches 90 percent of anomalies with minimal configuration.

### Challenge 6: Monitoring Pipeline Migration

You are migrating from a legacy monitoring system (Nagios + Graphite) to a modern system (Prometheus + Grafana + Loki). The legacy system monitors 500 services across 2,000 nodes. The migration must be completed within 3 months with zero gaps in monitoring coverage. Design the migration strategy.

**Discussion**: This challenge tests your ability to plan and execute a complex infrastructure migration without downtime.

Phase 1 (Weeks 1-2): Assessment and Planning
- Inventory all existing Nagios checks and Graphite metrics
- Categorize each check as "critical" (pages on-call), "important" (dashboard only), or "nice to have" (no action on failure)
- Map each check to a Prometheus equivalent
- Design the target architecture: 2 Prometheus servers, Grafana, Alertmanager, Loki for log storage

Phase 2 (Weeks 3-4): Infrastructure Setup
- Deploy Prometheus, Grafana, Alertmanager, and Loki on new infrastructure
- Set up the monitoring pipeline for the monitoring system itself
- Create the top-level overview dashboard in Grafana
- Configure alert routing to the existing PagerDuty integration

Phase 3 (Weeks 5-8): Service Migration
- Migrate services in batches of 50 per week
- For each service: deploy Prometheus node exporter, configure Prometheus scrape targets, create Grafana dashboard, configure alert rules, validate that data matches the legacy system
- Run both systems in parallel during migration
- Validate that every critical check in Nagios has an equivalent Prometheus alert

Phase 4 (Weeks 9-10): Cutover
- Switch alert routing from Nagios to Alertmanager
- Announce the cutover to all service teams
- Provide a 2-week window for teams to report issues
- Keep Nagios running but disabled for alerting

Phase 5 (Weeks 11-12): Decommissioning
- Remove Nagios and Graphite from production
- Archive Graphite data for historical reference
- Document the new monitoring system architecture and runbooks
- Train the on-call team on the new system

Key risk: data gaps during migration. Mitigation: run both systems in parallel during Phase 3. If the new system has issues, the old system is still running. The cutover in Phase 4 is a soft cutover -- if any team reports issues, revert to the old system for that service and investigate.

### Challenge 7: Monitoring for a Multi-Tenant SaaS Platform

Design a monitoring system for a SaaS platform that hosts 1,000 tenants. Each tenant runs their own instance of the application on shared infrastructure. The monitoring system must provide per-tenant dashboards, per-tenant alerting, and per-tenant SLO tracking. The system must also detect when one tenant's behavior affects other tenants (the noisy neighbor problem).

**Discussion**: This challenge tests your ability to design monitoring for a multi-tenant environment with isolation requirements.

The key architectural decision is whether to use one monitoring pipeline per tenant or a shared pipeline with tenant labels.

For 1,000 tenants, a shared pipeline with tenant labels is more cost-effective. Each metric and log event includes a `tenant_id` label. The TSDB stores all tenants' data in the same partition, and queries filter by `tenant_id`. The challenge is that a single tenant's high cardinality can affect all tenants.

Cardinality isolation: enforce a per-tenant cardinality limit of 10,000 series. If a tenant exceeds this limit, the aggregator rejects the data for that tenant only. Other tenants' data continues to flow normally.

Query isolation: run a separate query frontend for each tenant's dashboards. Each frontend is configured to add a `tenant_id` filter to every query, preventing tenants from accessing each other's data.

Noisy neighbor detection: compute per-tenant resource usage (CPU, memory, disk IO, network) and compare it to the tenant's allocation. If a tenant exceeds 80 percent of their allocation for more than 5 minutes, generate a warning. If a tenant exceeds 100 percent of their allocation, generate a critical alert and throttle the tenant's traffic.

Per-tenant SLO tracking: each tenant has a custom SLO based on their contract. The SLO tracking system computes per-tenant SLI values from the shared metrics and alerts when any tenant's SLO is at risk. The SLO compliance report is available as a per-tenant dashboard.

### Challenge 8: Log Management at Scale

Design a log management system that ingests 100 TB of log data per day from 10,000 services. The system must support real-time log search with sub-second latency for the last 24 hours of data and historical search with sub-10-second latency for data up to 30 days old.

**Discussion**: This challenge tests your understanding of log management architecture and the tradeoffs between indexing speed, query performance, and storage cost.

The architecture uses a three-tier storage approach with a streaming ingest pipeline.

Ingest pipeline: each service sends structured JSON logs to a local agent. The agent batches logs and sends them to a Kafka cluster with 100 partitions. Logs are partitioned by service name to ensure that all logs for a single service are in the same partition, enabling efficient queries.

Tier 1 (Hot storage, 24 hours): Logs are stored in a distributed in-memory store (Elasticsearch or ClickHouse). The hot store indexes all log fields for sub-second search. The index is built in memory during ingest and flushed to disk every 10 minutes. The hot store runs on 20 nodes with 128 GB RAM each, providing 2.5 TB of in-memory storage.

Tier 2 (Warm storage, 7 days): Logs are stored on SSD-backed Elasticsearch nodes. The warm store indexes only the most important fields: timestamp, service name, severity level, and trace ID. Other fields are stored but not indexed, reducing index size by 60 percent. The warm store runs on 50 nodes with 4 TB SSD each.

Tier 3 (Cold storage, 30 days): Logs are compressed with gzip (5:1 compression ratio) and stored in object storage (S3 or GCS). The cold store has no index. Queries over cold data scan the compressed files sequentially, using the file's time range to skip irrelevant files. A query over cold data takes 5-10 seconds depending on the time range.

Search routing: a query router receives all search requests. If the search time range is within the last 24 hours, the router queries the hot store. If the range is within 7 days, the router queries the warm store or a combination of hot and warm. If the range is beyond 7 days, the router queries the cold store. The router merges results from all stores and returns the combined result.

Cost breakdown: the three-tier approach costs approximately 40 percent of a single-tier approach that stores all 30 days on SSD with full indexing. The hot store is the most expensive per GB but the smallest. The cold store is the cheapest per GB but the largest. The tradeoff is that cold queries are 10-100x slower than hot queries, which is acceptable for historical debugging but not for real-time alerting.

## Appendix: Key Formulas

Use these formulas during your interview to calculate monitoring system parameters without hesitation.

### Data Points Per Second

```text
DPS = I x M / Si

Where:
  I = Number of instances
  M = Metrics per instance
  Si = Scrape interval in seconds

Example: 2,000 instances x 100 metrics / 15s = 13,333 DPS
```

### Agent Buffer Capacity in Seconds

```text
BufferSeconds = C / DPS

Where:
  C = Buffer capacity in events
  DPS = Data points per second from this agent

Example: 50,000 events / 2,000 DPS = 25 seconds of buffer
```

### Storage Per Day

```text
StoragePerDay = DPS x BytesPerPoint x 86,400

Where:
  DPS = Data points per second
  BytesPerPoint = Average bytes per data point (typically 16 for protobuf)

Example: 13,333 x 16 x 86,400 = 18.4 GB/day for metrics
  Logs: 835,000 log lines/s x 512 bytes x 86,400 = 36.9 TB/day
  With 5:1 compression: 7.4 TB/day
```

### Total Storage with Retention

```text
TotalStorage = StoragePerDay x RetentionDays x ReplicationFactor

Example (metrics only):
  18.4 GB/day x 30 days x 2 replicas = 1.1 TB
```

### Number of Aggregator Instances

```text
Aggregators = TotalDPS / CapacityPerInstance

Where:
  CapacityPerInstance = Typically 5,000 DPS per aggregator

Example: 13,333 DPS / 5,000 = 3 aggregators (round up to 4 for HA)
```

### Number of TSDB Partitions

```text
Partitions = TotalDPS / ThroughputPerPartition

Where:
  ThroughputPerPartition = Typically 2,000 DPS per partition

Example: 13,333 DPS / 2,000 = 7 partitions (round up to 8 for balanced)
```

### Error Budget

```text
ErrorBudget = (1 - SLO / 100) x TotalRequests x Window

Example:
  SLO = 99.9 percent
  Requests per month = 10 million x 30 = 300 million
  Error budget = 0.001 x 300 million = 300,000 failed requests allowed
  Error budget in time = 43.2 minutes of allowed violation
```

### Burn Rate

```text
BurnRate = CurrentErrorRate / AllowedErrorRate

Example:
  SLO = 99.9 percent, allowed error rate = 0.1 percent
  Current error rate = 1 percent
  Burn rate = 1 / 0.1 = 10x

  Time to exhaust budget = 30 days / 10 = 3 days
```

### Sampling Cost Savings

```text
CostSavings = 1 - SampleRate

Example:
  Full collection: $10,000/month for log storage
  10 percent sampling: $1,000/month for log storage
  Savings: 90 percent

  But recall: probability of catching a specific event = SampleRate
  So 10 percent sampling = 10 percent chance of catching a specific error
```
