## Internal Architecture

A production recommendation engine is not a single model. It is a pipeline of several models and services, each with a well-defined responsibility. Understanding the internal architecture is essential for debugging, optimizing, and scaling the system.

### Component Breakdown

The pipeline consists of 6 major components, ordered by their position in the request flow:

1. **Feature Store** -- serves real-time and batch-computed features for both candidate generation and ranking. It must respond in under 5 milliseconds with 99.99% availability.

2. **Retrieval Service** -- runs multiple candidate generation strategies in parallel and merges results. It must reduce the catalog from millions to 500 items in under 50 milliseconds.

3. **Ranking Service** -- scores each candidate using a deep neural network with feature engineering. It must score 500 items in under 100 milliseconds, including feature assembly time.

4. **Post-Processor** -- applies filtering, diversity reranking, and business rules. It must complete all post-processing in under 20 milliseconds.

5. **Logging Layer** -- records every recommendation impression and user interaction for training data. This is asynchronous and adds zero latency to the request path.

6. **Training Pipeline** -- offline job that reads historical data, trains models, and publishes artifacts. It runs every 6 hours and must complete within a 6-hour window.

### Component Deep Dive: The Feature Store

The feature store is the most critical component for latency. Every other component depends on it for data. If the feature store is slow, the entire pipeline is slow.

The feature store has three layers:

**Layer 1: Hot Cache (Redis).** Stores the most frequently accessed features: user embeddings, item embeddings, user profile summaries, and trending lists. Data is stored in Redis hashes with TTLs ranging from 5 minutes (trending) to 6 hours (embeddings). The Redis cluster has 8 shards, each running on an r6g.2xlarge instance with 64 GB of RAM. Total memory footprint: 66 GB.

**Layer 2: Warm Store (DynamoDB).** Stores persistent feature data: full user profiles, item catalog metadata, and historical feature snapshots. DynamoDB is configured with auto-scaling: 5,000 read capacity units (RCU) baseline, scaling to 20,000 RCU under peak load. The table uses a composite key: partition key = `feature_type`, sort key = `entity_id`.

**Layer 3: Cold Store (S3).** Stores historical feature data for training. Features are stored as Parquet files partitioned by date. The training pipeline reads 30 days of feature data from S3.

### Component Deep Dive: The Training Pipeline

The training pipeline runs every 6 hours and produces new model artifacts. It has four stages:

**Stage 1: Data Ingestion.** Read 30 days of impression and interaction logs from S3. The logs are stored as compressed JSON, approximately 500 GB per day. The Spark job reads the last 30 days, which is 15 TB of data. This takes approximately 30 minutes on a 100-node Spark cluster.

**Stage 2: Feature Generation.** Join impressions with user profiles and item catalog. Generate labels: did the user click or watch the item? (positive = 1) or not? (negative = 0). Apply negative sampling: for each positive interaction, sample 5 negative items that the user was shown but did not interact with. This corrects for the implicit feedback bias (users only interact with items they like, so all observed interactions are positive -- you need negatives to train a binary classifier). Output: 200 GB of TFRecord files.

**Stage 3: Model Training.** Train a two-tower DNN model on a GPU cluster with 8 NVIDIA A100 GPUs. The model trains for 5 epochs with early stopping if validation loss does not improve for 3 consecutive epochs. Training takes approximately 2 hours. The output includes: user tower weights, item tower weights, and the ranking head weights.

**Stage 4: Artifact Publishing.** Compute embeddings for all 50 million users and 10 million items using the trained model. Publish user embeddings to Redis (25 GB), item embeddings to FAISS index (5 GB), and the ranking model to TensorFlow Serving model store. Update the config service with the new model version number. This process takes approximately 30 minutes.

The total pipeline time is approximately 3.5 hours, leaving 2.5 hours of buffer within the 6-hour window.

## Data Structures

Three data structures are fundamental to building a recommendation engine at scale. Each solves a specific problem and has specific performance characteristics.

### Data Structure 1: User-Item Interaction Matrix

The user-item interaction matrix is the foundation of collaborative filtering. It is a sparse matrix where rows represent users, columns represent items, and cells represent interactions (ratings, clicks, purchases, watch events).

In a platform with 100 million users and 10 million items, the matrix has 10^15 possible cells. In practice, fewer than 0.01% of these cells are filled -- most users have interacted with fewer than 100 items. This sparsity is the fundamental challenge of collaborative filtering.

The matrix is typically stored as a sparse tensor in formats like COO (coordinate list) or CSR (compressed sparse row). For production systems, the matrix is partitioned across multiple machines in a Spark cluster.

**Storage format comparison:**
- COO format: stores (row, col, value) tuples. Simple but memory-intensive. For 100M interactions: 100M x (4 + 4 + 4) bytes = 1.2 GB
- CSR format: stores compressed row pointers, column indices, and values. More compact for row-major access. For 100M interactions: approximately 800 MB
- Block format: stores blocks of the matrix. Good for GPU processing. For 100M interactions: varies by block size

The interaction matrix is stored on disk as Parquet files partitioned by date. The training pipeline reads the matrix for the last 30 days, which is approximately 30 billion interactions.

**Sparsity analysis:**
- Total possible interactions: 100M users x 10M items = 10^15
- Actual interactions: 100M users x 100 avg interactions = 10^10
- Sparsity: 10^10 / 10^15 = 0.001% (99.999% empty)
- This is why collaborative filtering is hard: the signal is extremely sparse

### Data Structure 2: Embedding Vectors

Embeddings are dense vector representations of users and items in a shared latent space. A typical embedding has 64 to 256 dimensions. Each dimension captures a latent feature -- for movies, these might correspond to "action vs. drama," "modern vs. period," "serious vs. comedic," "blockbuster vs. indie."

The dot product of a user embedding and an item embedding gives a relevance score. If the embeddings are well-trained, the dot product correlates with the user's preference for that item.

**How embeddings are computed:**
1. The two-tower model takes user features (interaction history, demographics, context) and passes them through the user tower -- a feedforward neural network with 3 layers of 256 units each. The output of the last layer is the user embedding.

2. The same model takes item features (metadata, content features, popularity) and passes them through the item tower -- a separate feedforward network with the same architecture. The output of the last layer is the item embedding.

3. The model is trained to maximize the dot product between user and item embeddings for positive interactions and minimize it for negative interactions. The loss function is binary cross-entropy with negative sampling.

**Storage and serving:**
- User embeddings: 50 million x 128 dimensions x 4 bytes (float32) = 25 GB. Stored in Redis as a hash per user. Key: `user:{user_id}:embedding`
- Item embeddings: 10 million x 128 dimensions x 4 bytes = 5 GB. Stored in both Redis (for feature assembly) and FAISS (for ANN search)
- FAISS index: adds approximately 3 GB of index structure on top of the raw vectors. Total FAISS memory: 8 GB

**Update frequency:**
- Embeddings are recomputed every 6 hours by the training pipeline
- Online learning can update embeddings incrementally between training runs using streaming gradient updates
- User embeddings are more volatile than item embeddings (users change, items do not)

### Data Structure 3: Item Similarity Matrix (Precomputed)

The item similarity matrix stores pairwise similarities between items. For item-based collaborative filtering, you compute the similarity between every pair of items based on co-occurrence patterns. Two items are similar if users who interact with one tend to also interact with the other.

The similarity metric is typically cosine similarity or Pearson correlation:

```
sim(i, j) = |users_i ∩ users_j| / sqrt(|users_i| * |users_j|)
```

For a catalog of 10 million items, computing all-pairs similarity is O(n^2) -- 10^14 pairs, which is computationally infeasible. Instead, the system computes similarities only for items that have co-occurred in at least N user sessions (typically N = 5). This reduces the number of pairs to a manageable 10^9 to 10^10.

The resulting matrix is still sparse. Each item has, on average, 50 to 200 similar items. The matrix is stored as an adjacency list in a key-value store like Redis, keyed by item ID, with values being sorted lists of (similar_item_id, similarity_score) pairs.

**Storage calculation:**
- 10 million items x 100 neighbors x 25 bytes (item_id + score + overhead) = 25 GB
- This fits in a single Redis node (r6g.2xlarge with 64 GB RAM)
- For redundancy, use a Redis cluster with 2 shards, each holding 5 million items

**Update frequency:**
- The item similarity matrix is recomputed every 6 hours
- Unlike embeddings, the similarity matrix does not need online updates because item relationships are stable
- A new item gets its initial similarities from content-based features, then the similarity matrix is updated at the next batch computation

### Data Structure 4: FAISS Vector Index for ANN Search

The FAISS index is the workhorse of the retrieval stage. It enables approximate nearest neighbor search on 10 million item embeddings in under 15 milliseconds.

**Index type: IVF (Inverted File Index) with Flat encoding.**
- IVF partitions the embedding space into 4,096 clusters (centroids) using k-means clustering
- During indexing, each item is assigned to its nearest centroid
- During search, only the 32 nearest centroids are probed, reducing the search space from 10 million to approximately 80,000 items
- The Flat encoding means the full vectors are stored for exact distance computation within the probed clusters

**Index parameters:**
- `nlist` (number of centroids): 4,096
- `nprobe` (number of centroids to probe): 32
- `metric`: inner product (dot product)
- Training: 2 hours on a single GPU (A100)

**Performance characteristics:**
- Recall@100: 97% (compared to exact search)
- Latency: 8-15ms per query
- Memory: 5 GB (vectors) + 3 GB (index) = 8 GB
- Throughput: 5,000 QPS per GPU

**Index building:**
1. Read all 10 million item embeddings from the training pipeline output
2. Train the IVF centroids using k-means on a random sample of 1 million embeddings
3. Assign each item embedding to its nearest centroid
4. Build the inverted index mapping each centroid to its assigned items
5. Save the index to disk and load into memory for serving

### Data Structure 5: User Profile Cache

The user profile cache stores a compact representation of each user's state for fast retrieval during the request path.

**Cache contents:**
- User ID, account tier, language, region
- Interaction history summary: last 50 items interacted with, with timestamps and interaction types
- Negative preferences: set of item IDs the user has explicitly disliked
- Session state: current session duration, items seen in this session, search queries

**Cache strategy:**
- Cache-aside pattern: read from cache, write-through to DynamoDB
- TTL: 5 minutes for profile data, 1 hour for embeddings
- Eviction: LRU when memory is full
- Size: 50M users x 2 KB = 100 GB (too large for single Redis node -- sharded across 8 nodes)

## Request Lifecycle: Complete 15-Step Walkthrough

A single recommendation request goes through 15 distinct steps. Each step contributes to the overall latency and accuracy of the response. Here is the complete walkthrough.

### Step 1: User Identification (under 2ms)

The API gateway receives an HTTP request with a session cookie or API token. The gateway validates the token by checking the signature, expiry, and user ID. It extracts the user ID and forwards the request to the recommendation service via gRPC.

The gRPC request includes: user ID, session ID, device type, client version, request timestamp, and a flag indicating whether this is a fresh session or a continuation.

### Step 2: Profile Lookup (under 5ms)

The recommendation service looks up the user profile from the feature store. The profile contains:
- User embedding (128-dim float vector)
- User attributes (age, region, language, subscription tier, account age)
- Recent interaction history (last 50 items interacted with, with timestamps, interaction types, and interaction durations)
- Watch history summary (movies watched, series partially watched, items started but abandoned)
- Explicit preferences (thumbs up, thumbs down, "not interested" flags)

The profile is cached in Redis with a 5-minute TTL. Cache hit rate should be above 95%. If the profile is not in cache, the service reads it from DynamoDB, which adds 20-50ms but happens for fewer than 5% of requests.

### Step 3: Context Assembly (under 5ms)

The system combines the user profile with the current request context to build a complete context vector. This includes:

**Session context:** how many items the user has seen in this session (affects exploration rate -- more items seen means more exploration needed), search queries entered in this session (indicates current intent), items added to watchlist or cart, time elapsed since session start, number of sessions in the last 7 days (indicates churn risk)

**Temporal context:** hour of day (morning commute, lunch break, evening relaxation, late night), day of week (weekday vs. weekend), season (holiday season, summer, back-to-school), special events (sports final, awards season, holiday shopping)

**Device context:** screen size (affects number of items to show), network bandwidth (affects thumbnail quality and video preview autoplay), device type (mobile, TV, desktop, tablet), operating system

**Social context:** whether the user is browsing alone or with others (family profile, shared account), whether there are shared watchlists, whether the user is in a group viewing session

### Step 4: User Embedding Retrieval (under 2ms)

The system retrieves the user embedding from the embedding store. The embedding is a 128-dimensional float vector that represents the user's latent preferences. It is retrieved from Redis using the key `user:{user_id}:embedding`.

For a new user with no embedding, the system uses a default embedding based on the user's region, language, and device type. This default embedding is computed by averaging the embeddings of all users in the same region and language group.

### Step 5: Candidate Generation -- ANN Retrieval (15-25ms)

The system takes the user embedding and queries the FAISS IVF index for the top 200 nearest neighbors. The FAISS index is loaded into memory on a GPU instance. The search probes the 32 nearest centroids and returns the top 200 results.

The FAISS query returns a list of (item_id, distance) pairs. The distance is the inner product between the user embedding and the item embedding. Higher inner product means higher relevance.

### Step 6: Candidate Generation -- Item-to-Item Retrieval (3-5ms)

The system takes the last 3 items the user interacted with (from the profile cache) and queries the item similarity matrix for each. Each query returns 100 similar items. The system merges these into a set of up to 300 items.

The items are fetched from Redis using the key `item_sim:{item_id}`. The Redis response includes the top 100 similar items with their similarity scores. Items that the user has already interacted with are filtered out at this stage.

### Step 7: Candidate Generation -- Complementary Retrievers (2-5ms)

In parallel with steps 5 and 6, the system runs 3 additional retrievers:

**Trending retriever:** retrieves the top 20 trending items in the user's region. Trending is computed as a weighted combination of interaction velocity (interactions per hour) and total interaction count. The trending list is cached in Redis and updated every 5 minutes.

**Fresh content retriever:** retrieves the 20 newest items added to the catalog in the user's preferred genres. Fresh items are boosted by a factor of 1.5x for their first 7 days. The list is retrieved from DynamoDB, keyed by genre and date.

**Editorial picks retriever:** retrieves 10 items from human-curated lists. The lists are personalized to the user's language and region. The picks are stored in a configuration service (etcd) and loaded into Redis for fast access.

### Step 8: Candidate Merging and Deduplication (under 3ms)

The system merges all candidates from the 5 retrievers into a single set. Duplicates are removed (an item that appears in both ANN results and trending results is counted once). Items the user has already interacted with (watched, purchased, disliked) are filtered out using the negative preferences set from the user profile.

The merging logic uses a hash set of item IDs. Each item is added to the set with its source retriever tag (for logging and analysis). After merging, the system checks the pool size.

If the pool is below 300 items (which can happen for niche users with very specific preferences), the system runs a fallback retriever that returns the top 200 most popular items globally. This ensures the ranking stage always has enough candidates.

The resulting pool contains 400-600 unique items.

### Step 9: Feature Assembly for Ranking (under 10ms)

For each candidate item, the system assembles a feature vector for the ranking model. The features are assembled from three sources:

**User features (128 floats):** user embedding, user engagement score (0-1), average rating given, genre preference distribution, time-of-day preference vector, device preference vector, session depth, days since last visit

**Item features (128 floats):** item embedding, genre vector, popularity score, freshness score, average rating, completion rate, language, content rating (PG, PG-13, R), runtime, release year

**Cross features (94 floats):** user-genre affinity (dot product of user genre preference with item genre vector), user-director affinity, user-actor affinity, user-language affinity, average rating user gives to items in this genre, time since last user interaction with this genre, time since last user interaction with this director, whether the user has watched other items in this series, whether the user has watched other items by this director

**Context features (20 floats):** hour-of-day embedding (8-dim), day-of-week embedding (7-dim), device-type embedding (4-dim), session depth (1-dim)

Total feature vector: 370 dimensions. For 500 candidates, this is 500 x 370 = 185,000 floats = 740 KB.

### Step 10: DNN Scoring -- GPU Inference (15-30ms)

The ranking model is a 3-layer feedforward neural network:
- Input: 370 features
- Layer 1: 256 units, ReLU, batch normalization, dropout 0.2
- Layer 2: 128 units, ReLU, batch normalization, dropout 0.2
- Layer 3: 64 units, ReLU, batch normalization
- Output: 1 unit, sigmoid activation

The model is served using TensorFlow Serving on NVIDIA A100 GPU instances. The batch of 500 candidates is packed into a single inference call. With GPU acceleration, the entire batch is scored in 15-30 milliseconds.

The model's output is a relevance score between 0 and 1 for each candidate. The score represents the predicted probability that the user will positively interact with the item (click, watch, purchase).

### Step 11: Business Logic and Filtering (under 5ms)

The ranked list passes through a business logic layer with deterministic rules:

**Filtering rules:**
- Remove items the user has explicitly disliked (thumbs down, "not interested")
- Remove items the user has already watched or purchased
- Remove items not available in the user's region (licensing restrictions)
- Remove items not appropriate for the user's age/parental control settings
- Remove items that are out of stock or no longer available
- Remove items the user has already seen in the current session (to avoid repetition)

**Boosting rules:**
- Freshness boost: items published within the last 7 days get a 1.5x multiplicative boost
- High-quality boost: items with average rating above 4.5 get a 1.2x boost
- Subscription boost: items exclusive to the user's subscription tier get a 1.1x boost

**Sponsored insertion:**
- 2 sponsored items are inserted at positions 2 and 5
- Sponsored items must have a relevance score in the top 30% of all candidates
- Sponsored items are clearly labeled as "Sponsored" in the response

### Step 12: Diversity Reranking (MMR) (under 10ms)

The diversity reranking step applies Maximum Marginal Relevance (MMR) to ensure the final list is diverse. The algorithm:

```
selected = []
remaining = all candidates sorted by score descending

for i in range(50):
    best_score = -inf
    best_item = None
    for item in remaining:
        mmr_score = lambda * relevance(item)
                   - (1 - lambda) * max_{s in selected} sim(item, s)
        if mmr_score > best_score:
            best_score = mmr_score
            best_item = item
    selected.append(best_item)
    remaining.remove(best_item)
```

Parameters:
- lambda = 0.7 (70% relevance weight, 30% diversity weight)
- Sliding window size = 5 (diversity is only enforced within the last 5 items)
- Item similarity is computed using the item embedding dot product

The sliding window variant improves efficiency: instead of computing similarity against all selected items, only the last 5 are considered. This reduces the complexity from O(n^2) to O(n * k) where k is the window size.

### Step 13: Exploration Injection (under 2ms)

The system injects exploration items into the final list. For each batch of 10 items, the last position is reserved for an exploration item. Exploration items are randomly sampled from a "cold start pool" -- items that have low interaction counts and need exposure.

The exploration rate is controlled by the user's engagement level:
- New users (under 10 interactions): 20% exploration to quickly learn preferences
- Active users (10-500 interactions): 10% exploration to discover new interests
- Power users (500+ interactions): 5% exploration (system already knows preferences well)

### Step 14: Logging (asynchronous, 0ms on request path)

Every recommendation impression is logged to a Kafka topic called `rec_impressions`. The log entry includes:

- User ID, session ID, request timestamp, model version
- Full list of recommended item IDs with their positions, scores, and source retriever tags
- Which items were explored vs. exploited
- A/B test assignment (control or treatment variant)
- User features and context used for the request
- Latency breakdown for each stage

The logging is asynchronous: the response is sent to the user before the Kafka message is produced. This ensures zero latency impact from logging.

### Step 15: Response Delivery and Client Rendering (under 5ms)

The final list of 50 items is serialized as JSON and returned to the client. The response includes:
- Item IDs, titles, thumbnail URLs, and brief descriptions
- Recommendation reason for each item ("Because you watched X", "Trending now", "New release", "Because you like Y genre")
- Tracking pixel URLs for impression logging (client fires an HTTP request to log the impression)
- Experiment assignment metadata

The client renders the recommendations in the appropriate UI components. The response is cached on the client side for 5 minutes to reduce server load during rapid scrolling.

## Consistency and State Management

Recommendation systems have different consistency requirements than traditional transactional systems. Here are four key scenarios that affect consistency.

### Scenario 1: Eventual Consistency for Interaction Data

When a user watches a movie on Netflix, that interaction goes through a multi-step pipeline before it affects recommendations:

1. **Immediate (0-5 seconds):** The interaction is recorded in the user's session state. When they refresh the page, the movie is removed from their recommendations via a rule-based filter that checks a Redis set of recently watched items.

2. **Short-term (5-60 seconds):** The interaction is written to a Kafka topic and consumed by the real-time feature store. Related item co-occurrence counts are updated. The user's session embedding is updated via online learning.

3. **Medium-term (minutes to hours):** The interaction is consumed by the streaming feature computation pipeline. User features (engagement counts, genre distribution) are recomputed. The changes propagate to the feature store.

4. **Long-term (hours to days):** The training pipeline reads the interaction data and retrains the model. New embeddings and factor tables are published. All users see the updated recommendations.

This means a user who watches a movie does not see it reflected in their model-based recommendations for hours. The system compensates with rule-based filters that immediately remove watched items from the candidate set.

### Scenario 2: Read-Your-Writes for User Preferences

When a user explicitly dislikes an item, the system must respect this immediately. Unlike implicit signals, explicit negative feedback is treated as a strong signal.

The implementation uses a "negative preferences" set stored in Redis:
- Key: `user:{user_id}:disliked`
- Value: Set of item IDs
- Every recommendation request reads this set and filters out disliked items

This guarantees read-your-writes consistency for negative feedback. The user dislikes an item, refreshes the page, and that item is gone.

### Scenario 3: Stale Embeddings During Training Window

Between model training runs, embeddings become increasingly stale. User preferences drift, new items are added, and old items decline in popularity. The system must handle this gracefully.

Production systems mitigate staleness with:
- **Online learning:** Incrementally update embeddings based on real-time events using a streaming optimizer like Adam on GPU. This bridges the gap between batch training runs.
- **Feature freshness indicators:** Each embedding has a timestamp. If the embedding is older than 48 hours, the system increases the exploration rate for that user to compensate for potentially stale preferences.
- **Shadow evaluation:** The new model runs in shadow mode for 24 hours before deployment, comparing predictions against the live model. If the shadow model's predictions diverge significantly from the live model, the deployment is paused.

### Scenario 4: A/B Experiment Consistency

When A/B tests are running, the system must consistently assign the same user to the same experiment variant. This is critical for statistical validity.

The implementation uses consistent hashing:
- User ID is hashed with the experiment ID using a deterministic hash function (SHA-256)
- The hash is taken modulo the number of variants
- The hash is stable across sessions and devices
- If a user is in a treatment group, they always see the new model

### Throughput and Scaling Considerations

A single GPU instance (NVIDIA A100) can handle approximately 500 requests per second for the DNN scoring stage. For a system serving 10,000 requests per second, you need 20 GPU instances.

The retrieval stage is CPU-bound and can handle more than 2,000 QPS per instance. You need 5 retrieval instances.

Caching the top-100 recommendations for the most active 20% of users (power users) reduces the load on the pipeline by 40%. This cache is stored in Redis with a 1-hour TTL.

### Feature Configuration

The ranking model uses features from three categories. Each feature has a type, transformation, and source.

| Feature | Type | Transformation | Source | Dimension |
|---------|------|---------------|--------|-----------|
| user_embedding | float | raw | Redis | 128 |
| item_embedding | float | raw | Redis | 128 |
| genre_affinity | float | dot product | computed | 1 |
| director_affinity | float | dot product | computed | 1 |
| actor_affinity | float | dot product | computed | 1 |
| hour_of_day | int | 8-dim embedding | computed | 8 |
| day_of_week | int | 7-dim embedding | computed | 7 |
| device_type | int | 4-dim embedding | computed | 4 |
| session_depth | int | log normalization | computed | 1 |
| days_since_last_visit | int | log normalization | computed | 1 |
| engagement_score | float | z-score | batch | 1 |
| genre_distribution | float[20] | softmax | batch | 20 |
| item_popularity | float | min-max | batch | 1 |
| item_freshness | float | time decay | batch | 1 |
| item_avg_rating | float | raw | batch | 1 |
| completion_rate | float | raw | batch | 1 |
| user_language_pref | int | one-hot (50) | batch | 50 |
| user_region | int | one-hot (20) | batch | 20 |
