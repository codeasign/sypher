## Naive Implementation

Before you build a production-grade system, you need to understand why the naive approach fails. A startup typically starts with a single monolithic recommendation service. Here is what that looks like and where it breaks.

### The Monolith

A naive recommendation engine runs as a single service. It loads all user data, item data, and model weights into memory. For each request, it computes similarity scores between the target user and every item in the catalog.

```python
def recommend(user_id, num_results=10):
    user = load_user_profile(user_id)
    all_items = load_all_items_from_db()  # 10 million items
    scores = []
    for item in all_items:
        score = cosine_similarity(user.embedding, item.embedding)
        scores.append((item.id, score))
    scores.sort(key=lambda x: x[1], reverse=True)
    return scores[:num_results]
```

This approach works for a proof-of-concept with 1,000 users and 10,000 items. It catastrophically fails at scale.

### Failure Points

**Failure 1: Linear Scan of All Items.** Computing cosine similarity for 10 million items requires 10 million dot products. At 100 nanoseconds per dot product, that is 1 second per request. With 1,000 concurrent requests, the service is overwhelmed.

**Failure 2: Monolithic State.** Loading all data into a single process limits memory to what one machine can hold. A single machine can hold at most 500 GB of RAM. Your user embeddings (50M x 128 x 4 = 25 GB) plus item embeddings (10M x 128 x 4 = 5 GB) plus the item catalog (10M items x 1 KB = 10 GB) totals 40 GB. This fits in one machine, but adding any more data or models breaks it.

**Failure 3: No Caching.** Every request recomputes the same scores. Popular items are rescored for every user. A trending item is scored 50 million times per day, with the same result each time.

**Failure 4: No Separation of Concerns.** The same model handles retrieval, scoring, and filtering. You cannot optimize each stage independently. You cannot update the retrieval model without affecting the ranking model.

**Failure 5: No A/B Testing.** Every user sees exactly the same algorithm. There is no way to test improvements without a full redeploy. This makes it impossible to validate model improvements statistically.

**Failure 6: No Fault Tolerance.** If the single server crashes, the entire recommendation service goes down for all users. The mean time to recover (MTTR) is the time to restart the process and reload 70 GB of data into memory, which takes 10-15 minutes.

**Failure 7: No Incremental Updates.** To update the model, you must redeploy the entire service. There is no way to hot-swap a model version. This means every model update carries significant risk -- if the new model has a bug, all recommendations are broken until the rollback completes.

**Failure 8: No Monitoring or Observability.** The monolith has no visibility into its internal operations. You cannot see how long each stage takes. You cannot measure cache hit rates or model latency. Debugging a performance regression requires sifting through application logs.

## Production-Grade Architecture

A production-grade recommendation engine uses a microservices architecture with clear separation of concerns, horizontal scaling, caching, and fault tolerance.

### Service Decomposition

The monolithic service is split into 5 independent services:

1. **Retrieval Service** -- handles candidate generation. Stateless. Scaled horizontally based on QPS.
2. **Ranking Service** -- handles DNN scoring. GPU-backed. Scaled based on throughput.
3. **Post-Processing Service** -- handles filtering, diversity, and business rules. Stateless.
4. **Feature Service** -- serves real-time and batch features. Reads from Redis and DynamoDB.
5. **Training Service** -- handles model training. Runs on Spark and GPU clusters. Not in the request path.

Each service communicates via gRPC for low-latency, high-throughput internal calls. The API Gateway handles external HTTP requests and routes them to the internal services.

### Data Layer Decomposition

The data layer is also decomposed:

1. **Redis** -- caching layer for user profiles, embeddings, and temporary data. 8 shards, each running on an r6g.2xlarge instance.
2. **FAISS Cluster** -- vector index for ANN search. 3 shards, each holding a third of the item catalog.
3. **DynamoDB** -- persistent storage for user profiles, item metadata, and feature data. Auto-scaling based on read/write capacity.
4. **S3** -- data lake for raw events and training data. Lifecycle policies move data to S3 Glacier after 90 days.
5. **Kafka** -- event streaming for real-time interactions. 12 partitions, replicated across 3 availability zones.

| Service | Instances | Instance Type | QPS per Instance | Total Capacity | Scaling Trigger |
|---------|-----------|---------------|-----------------|----------------|-----------------|
| Retrieval | 10 | c6i.4xlarge (16 vCPU) | 2,000 | 20,000 | CPU > 70% |
| Ranking | 20 | g4dn.xlarge (GPU) | 500 | 10,000 | GPU utilization > 80% |
| Post-Processing | 5 | c6i.2xlarge (8 vCPU) | 3,000 | 15,000 | CPU > 70% |
| Feature | 8 | c6i.4xlarge (16 vCPU) | 4,000 | 32,000 | CPU > 70% |
| Redis | 8 | r6g.2xlarge (64 GB) | 20,000 | 160,000 | Memory > 80% |
| FAISS | 3 | r6g.4xlarge (128 GB) | 5,000 | 15,000 | QPS > 10,000 |
| Kafka | 12 | m6i.large (8 GB) | 10,000 | 120,000 | Disk > 80% |

The system is designed to handle 10,000 QPS with headroom for 2x traffic spikes. Auto-scaling policies add instances within 2 minutes of detecting elevated CPU or GPU utilization.

### Service Communication Protocol

All inter-service communication uses gRPC with Protocol Buffers for serialization. gRPC was chosen over REST for three reasons:

**Performance:** gRPC uses HTTP/2 multiplexing, which allows multiple requests to share a single TCP connection. For a system handling 10,000 QPS across 5 services, this reduces connection overhead by 80% compared to HTTP/1.1. Protobuf serialization is 3-5x faster than JSON and produces messages that are 30-50% smaller.

**Streaming:** gRPC supports bidirectional streaming, which is essential for fetching features from the feature store. The ranking service can open a streaming connection and send feature requests for each candidate batch without the overhead of establishing a new connection per request.

**Strong typing:** The Protobuf schema defines the exact structure of every request and response. This prevents the "silent field missing" bugs that are common with JSON APIs.

### API Design

The recommendation service exposes a single gRPC endpoint:

```protobuf
service RecommendationService {
  rpc GetRecommendations(RecommendationRequest) returns (RecommendationResponse);
}

message RecommendationRequest {
  string user_id = 1;
  string session_id = 2;
  string device_type = 3;
  string region = 4;
  int32 num_results = 5;  // default 50
  repeated string exclude_item_ids = 6;  // items to exclude
  Context context = 7;
}

message Context {
  int32 hour_of_day = 1;
  int32 day_of_week = 2;
  int32 session_depth = 3;  // items seen in this session
  string search_query = 4;  // if user is searching
}

message RecommendationResponse {
  repeated RecommendationItem items = 1;
  string request_id = 2;
  int64 processing_time_ms = 3;
  string model_version = 4;
}

message RecommendationItem {
  string item_id = 1;
  string title = 2;
  string thumbnail_url = 3;
  double score = 4;
  string reason = 5;  // "Because you watched X"
  string source = 6;  // "ann", "item_sim", "trending", "editorial"
}
```

### Request Routing and Load Balancing

The API Gateway uses consistent hashing to route requests. For a given user ID, all requests during a session are routed to the same set of service instances. This improves cache locality -- if the same retrieval instance processes all requests for a user, its Redis cache for that user's profile and embeddings stays warm.

The consistent hashing ring has 512 virtual nodes. When a service instance is added or removed, only 1/512th of user requests are rerouted. This minimizes cache disruption during scaling events and rolling deployments.

A recommendation engine does not exist in isolation. It must integrate with the broader platform. Here are 4 common integration patterns.

### Pattern 1: Embedded Widget (Netflix Homepage)

The recommendation engine provides an API that the frontend calls to populate widgets on the homepage. Each widget corresponds to a recommendation row: "Trending Now," "Because You Watched X," "Top Picks for You."

The API returns a structured response with widget groups:

```json
{
  "widgets": [
    {
      "id": "trending",
      "title": "Trending Now",
      "items": [{"id": "tt123", "title": "...", "score": 0.95}]
    },
    {
      "id": "because-you-watched",
      "title": "Because You Watched Stranger Things",
      "source_item": "tt_st",
      "items": [{"id": "tt456", "title": "...", "score": 0.89}]
    }
  ]
}
```

The frontend renders each widget independently. If one widget fails, the page still loads with the remaining widgets. Each widget has a fallback -- if the personalized recommendation fails, show the global trending widget.

### Pattern 2: Server-Side Fetch (Amazon PDP)

On an Amazon product detail page (PDP), the recommendation widget "Customers who bought this also bought" is rendered server-side. The PDP service makes a gRPC call to the recommendation service with the current product ID. The recommendation service returns a list of related product IDs.

This integration is synchronous. The PDP page cannot render without the recommendations. The p99 latency target is 50 milliseconds. If the recommendation service takes longer than 50ms, the PDP falls back to a static list of best-selling items.

### Pattern 3: Personalized Search (YouTube)

YouTube combines search and recommendations. When a user searches for "cooking tutorials," the search results are reranked based on the user's recommendation profile. A user who watches high-production Gordon Ramsay videos sees different cooking results than a user who watches low-budget home cooking channels.

The integration works as follows:
1. Search service retrieves 1,000 results based on text relevance
2. Search service sends the results to the recommendation service with the user context
3. Recommendation service scores each result using the ranking model
4. The final rank is a weighted blend of text relevance (60%) and personalized score (40%)

### Pattern 4: Real-Time Session Personalization (TikTok)

TikTok's recommendation engine integrates at the session level. As a user scrolls through the feed, every swipe (like or skip) is sent to the recommendation engine in real time. The engine updates the user's session state and adjusts the remaining items in the current batch.

This pattern requires sub-100 millisecond feedback loops. The architecture uses:
- A WebSocket connection between the client and a session service
- The session service maintains an in-memory state for active sessions (TTL: 30 minutes)
- Each like or skip updates the user's temporary embedding in the session state
- The next batch of items is retrieved using the updated embedding

### Pattern 5: Notification Recommendations (Push)

The recommendation engine generates recommendations for push notifications. This is a batch pattern -- the system computes the best item to recommend to each user and sends it as a push notification.

The key difference from inline recommendations is that push notifications compete for user attention outside the app. The recommendation must be highly compelling to justify interrupting the user. The system uses a higher relevance threshold (score must be in the top 1% of all recommendations) and includes the recommendation reason in the notification body ("A new movie from Christopher Nolan is now available").

The push notification pipeline runs every 6 hours and selects the single best item for each user. Notifications are throttled to a maximum of 2 per week per user to prevent notification fatigue.

### Pattern 6: Email Campaign Recommendations

The recommendation engine powers personalized email campaigns. The system generates a batch of 5 recommended items for each user and inserts them into the weekly marketing email.

This pattern has no real-time requirements -- the recommendations are precomputed 24 hours before the email is sent. The system uses the same ranking model but applies different post-processing rules: items that require immediate action (limited-time offers, expiring content) are boosted, and items with high relevance but low recency are demoted.

The email campaign system reads the precomputed recommendations from S3, renders the email templates, and sends via the email delivery service. Each email includes tracking pixels to measure click-through rates on recommended items.

## Multi-Region Architecture

A global recommendation engine must serve users across multiple geographic regions with low latency. The architecture must handle data sovereignty, regional catalog differences, and disaster recovery.

### Regional Deployment

Each region (US-East, EU-West, AP-Southeast) has its own deployment of the recommendation stack:
- **Retrieval, Ranking, Post-Processing** -- full stack deployed in each region
- **Redis** -- regional cache, populated from the local training pipeline
- **FAISS** -- regional index, replicated from the central training pipeline
- **DynamoDB** -- regional copy for user profiles and item catalog (cross-region replication enabled)
- **Kafka** -- regional event streams that replicate to the central data lake

### Central Training

Training data from all regions is consolidated in a central S3 data lake in US-East. The training pipeline runs in US-East and produces a single global model. The model artifacts (embeddings, factor tables, ranking model) are replicated to all regions using S3 cross-region replication.

This means the same model runs everywhere, but the results differ because users, interactions, and regional item catalogs differ.

### Data Sovereignty

For regions with data sovereignty requirements (EU under GDPR), user interaction events are stored in regional S3 buckets and are not replicated to US-East. The training pipeline ingests only metadata about the interactions (aggregated counts), not the raw interaction data. The model is trained in each region independently.

### Disaster Recovery

The multi-region deployment supports two disaster recovery scenarios:

**Scenario 1: Service Degradation (single service fails in one region).** Each service runs behind a load balancer with health checks. Health checks are performed every 5 seconds. If a service instance fails 3 consecutive health checks, it is removed from the load balancer. Traffic is redirected to healthy instances. The auto-scaling group replaces the failed instance within 2 minutes.

**Scenario 2: Complete Region Failure (entire region goes down).** An AWS Route 53 health check monitors the primary region's API Gateway endpoint. If the endpoint is unreachable for 60 seconds, DNS-based failover routes traffic to the nearest healthy region. The failover region loads the replicated model artifacts from S3 and initializes the FAISS index within 5 minutes. User profiles are available via DynamoDB global tables (cross-region replication with eventual consistency under 1 second).

During failover, the user experience degrades in three ways: (1) session state is lost -- the user's current session context is not available in the failover region, so recommendations revert to the user's long-term profile, (2) latency increases by the round-trip time between the user's original region and the failover region (US-East to EU-West adds 80-100ms), and (3) exploration may increase because the failover region's embeddings are up to 6 hours stale.

The failover target is recovery time objective (RTO) of 5 minutes and recovery point objective (RPO) of 1 hour (acceptable data loss window for interaction events that have not been replicated).

### Regional Model Differences

While the same base model runs in all regions, each region applies regional adjustments:

- **US-East:** Standard model with no modifications. Full 10 million item catalog.
- **EU-West:** Standard model plus GDPR compliance filters. Item catalog reduced to 8 million items (some content rights do not extend to EU). User embeddings computed from aggregated data only.
- **AP-Southeast:** Standard model with language-specific item catalog. Additional boost for locally produced content. Item catalog of 5 million items relevant to the APAC market.

These regional differences are handled at the post-processing layer, not at the model layer. The same ranking model scores all candidates, but regional filters remove items that are not available or relevant in the user's region. This approach keeps the model simple and the training pipeline unified.

## Capacity Planning

Building a recommendation engine requires careful capacity planning. Here is how to estimate your needs.

### Request Volume Estimation

Start with your daily active users (DAU) and average requests per user per session:

| Metric | Small App | Medium App | Large App | Netflix Scale |
|--------|-----------|------------|-----------|---------------|
| DAU | 100K | 10M | 100M | 200M |
| Sessions per day per user | 1 | 2 | 3 | 2 |
| Requests per session | 5 | 10 | 20 | 15 |
| Total daily requests | 500K | 200M | 6B | 6B |
| Peak QPS (10x hourly avg) | 58 | 23,000 | 694,000 | 694,000 |

For a medium app (10M DAU), you need to handle 23,000 peak QPS. Your system must be designed for this peak, not the average.

### Compute Estimation

The compute requirements depend on the model architecture. For a two-tower model with 128-dim embeddings:

| Component | Computation per Request | Total per Second (23K QPS) | Required Instances |
|-----------|------------------------|---------------------------|-------------------|

| Feature Assembly | 5ms | 115 seconds | 8 (c6i.4xlarge) |
| Post-Processing | 5ms | 115 seconds | 5 (c6i.2xlarge) |

### Network Bandwidth Estimation

The inter-service bandwidth of 2.3 GB/s requires all services to be deployed in the same AWS availability zone with 10 Gbps networking enabled. Cross-zone traffic would add latency and incur data transfer costs.

## Load Shedding and Graceful Degradation

No system operates perfectly at all times. A recommendation engine must degrade gracefully under extreme load or partial failure.

### Traffic Spikes

Under a 10x traffic spike (230,000 QPS), the system cannot handle the full pipeline for every request. The system uses three levels of degradation:

**Level 1: Cache Hit Preferred (under 3x spike).** The system prioritizes users whose recommendations are in the batch precompute cache. For cached users (top 20% most active), the response comes from Redis in under 5ms, bypassing the full pipeline. For uncached users, the full pipeline runs with a reduced candidate pool (250 items instead of 500) to reduce ranking load.

**Level 2: Ranking Bypass (under 5x spike).** The ranking service is bypassed entirely. Candidates are scored using the dot product of the user embedding and item embedding (cheap, approximate scoring) instead of the expensive DNN. The scoring time drops from 30ms to 2ms per request. The ranking quality drops by approximately 15%, but the system stays operational.

**Level 3: Retrieval Only (over 5x spike).** The entire ranking and post-processing stages are bypassed. The system returns candidates from the retrieval stage directly, sorted by the ANN distance score. No diversity reranking, no business rules, no sponsored items. The response is returned in under 30ms. Users see less relevant and less diverse recommendations, but they see something.

### Partial Service Failures

When a specific service fails, the system degrades the affected component:

**Redis cache failure:** If the Redis cluster is degraded, the system reads all data from DynamoDB. Latency increases by 50-100ms. Cache miss rate goes to 100%.

**FAISS index failure:** If FAISS is unavailable, the system falls back to exact nearest neighbor search using a precomputed item similarity matrix. This is 50x slower but works for catalogs under 500,000 items. For larger catalogs, the system falls back to trending and popularity-based retrieval only.

**GPU ranking failure:** If all GPU instances are unavailable, the system runs the ranking model on CPU. It is 5-10x slower, so throughput drops to 1,000-2,000 QPS. The system reduces the candidate pool to 200 items and activates the ranking bypass level for excess traffic.

**Kafka failure:** If Kafka is unavailable, the event logging layer buffers events in local files (up to 1 GB per instance). When Kafka recovers, the buffered events are replayed. This prevents data loss for up to 10 minutes of Kafka downtime.

## Data Lifecycle Management

A recommendation engine generates massive amounts of data. Managing the data lifecycle is essential for cost control and regulatory compliance.

### Event Data Retention

| Data Type | Hot Storage | Warm Storage | Cold Storage | Deletion |
|-----------|-------------|--------------|--------------|----------|
| Raw interaction events | Kafka (7 days) | S3 Standard (30 days) | S3 Glacier (6 months) | After 6 months |
| Impression logs | Kafka (7 days) | S3 Standard (30 days) | S3 Glacier (1 year) | After 1 year |
| Training data | S3 Standard (30 days) | S3 Standard (90 days) | S3 Glacier (2 years) | After 2 years |
| Model artifacts | S3 Standard (current) | S3 Standard (last 10) | S3 Glacier (last 100) | Keep forever |
| User profiles | DynamoDB (active) | DynamoDB (TTL 1 year) | S3 Glacier (5 years) | After 5 years inactive |
| Embeddings | Redis (current) | S3 Standard (last 30) | None | Keep last 30 versions |

### Data Compression Ratios

| Data Type | Raw Size | Compressed Size (Snappy) | Compression Ratio |
|-----------|----------|--------------------------|-------------------|
| Raw JSON events | 500 GB/day | 150 GB/day | 3.3:1 |
| Parquet events | 200 GB/day | 80 GB/day | 2.5:1 |
| TFRecord training data | 100 GB/day | 40 GB/day | 2.5:1 |

Switching from JSON to Parquet for the training data reduces S3 storage costs by 60% and speeds up Spark reads by 40% (Parquet is columnar and supports predicate pushdown).

### Data Archival Strategy

Data moves through a lifecycle defined by S3 lifecycle policies:

1. **Day 0-30:** Data in S3 Standard. Available for training pipeline reads. Queryable via Athena and Spark.
2. **Day 31-90:** Data transitions to S3 Infrequent Access. 40% cost savings. Still queryable but with higher latency.
3. **Day 91-365:** Data transitions to S3 Glacier Instant Retrieval. 70% cost savings. Retrieval takes milliseconds but costs more per request.
4. **Day 366+:** Data transitions to S3 Deep Archive. 95% cost savings. Retrieval takes 12 hours.

## Operations

Running a recommendation engine in production requires robust operational practices.

### Monitoring

Each service exposes metrics for:
- **Latency:** p50, p95, p99, p999 for each stage
- **Throughput:** QPS per stage, per model version
- **Error rate:** 4xx and 5xx responses
- **Cache hit rate:** per cache type (profile, embedding, trending)
- **GPU utilization:** for ranking service
- **Model staleness:** hours since last model update

### Alerts

| Alert | Threshold | Action |
|-------|-----------|--------|
| p99 latency > 200ms | 5 minutes | Page on-call engineer |
| Error rate > 1% | 1 minute | Auto-scale, page on-call |
| Cache hit rate < 90% | 10 minutes | Investigate, potential cache poisoning |
| Training pipeline failure | 1 hour | Page ML engineer |
| Model staleness > 12 hours | 30 minutes | Investigate training pipeline |
| FAISS rebuild failure | 1 hour | Page platform engineer |

### Deployment

The recommendation system supports rolling deployments with zero downtime:

1. **Model update:** New model artifacts are published to S3. Each service instance picks up the new model on the next refresh cycle (every 5 minutes). No restart needed.
2. **Service update:** New service binaries are deployed via blue-green deployment. A full fleet of new instances is created, tested with synthetic traffic, then traffic is switched over.
3. **Configuration update:** Configuration changes (A/B experiments, business rules) are pushed via the config service (etcd). Changes take effect within 30 seconds.
