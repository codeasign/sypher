# PACELC at Scale

From a single-region eventual consistency system to a global multi-region PACELC-aware topology. Each architectural stage reveals new trade-offs.

---

### What Breaks at Scale

**Problem 1: Replication lag grows with write volume.** At 10K writes/second, replication lag is typically 50ms. At 100K writes/second, lag grows to 500ms or more. At 500K writes/second, the async replication pipeline saturates and lag can reach seconds to minutes. Reads increasingly return stale data.

**Problem 2: Single region is a single point of failure.** If us-east-1 goes down, the entire system is unavailable. Data may be permanently lost if the region does not recover.

**Problem 3: Hot partitions.** With PA/EL and CL=ONE, one replica can absorb 70% of reads if the load balancer distributes unevenly. The hot replica's latency degrades while other replicas sit idle.

### Consistency Level by Operation Pattern

| Operation | Read CL | Write CL | Rationale |
|---|---|---|---|
| Product browse | LOCAL_ONE | LOCAL_ONE | Speed over freshness for catalog |
| Add to cart | LOCAL_ONE | LOCAL_QUORUM | Cart count must be accurate |
| Checkout | LOCAL_QUORUM | EACH_QUORUM | Financial transaction, must be consistent globally |
| User profile read | LOCAL_ONE | LOCAL_ONE | Stale profile is acceptable |
| Password change | LOCAL_QUORUM | LOCAL_QUORUM | Auth data must be strongly consistent |
| Inventory check | LOCAL_QUORUM | LOCAL_QUORUM | Avoid overselling |

### PACELC Decision Engine Per Request

The decision engine evaluates three inputs:
1. **Partition status** — is a partition active? (from the partition detector)
2. **Consistency requirement** — what does the application request?
3. **Latency budget** — what is the maximum acceptable response time?

For a checkout operation during a partition: the engine sees partition=True, requirement=strong, and rejects the request rather than risking inconsistency. For a product browse during a partition: the engine sees partition=True, requirement=weak, and serves from the local replica.

### Database Path

The primary data store. For DynamoDB, the path applies PA/EL by default with optional ConsistentRead parameter for PC/EC. For Cassandra, the consistency level is set per query.

### Cache Path

Caches are inherently PA/EL — they store a copy of data that may be stale. The invalidation strategy determines staleness bounds. Write-through caches reduce staleness but add write latency. Write-behind caches have higher throughput but longer inconsistency windows.

### Message Queue Path

Queues decouple producers from consumers. Producers write to the queue with at-least-once delivery semantics. This is PA/EL for the producer — the write is accepted quickly and replicated asynchronously. Consumers must handle duplicate delivery (idempotent processing).

### External API Path

Third-party APIs (payment gateways, identity providers) are outside your PACELC control. Treat them as PC/EC: assume strong consistency on their side, implement idempotency on yours.

### Active-Active (PA/EL Between Regions)

Both regions accept writes. Data is replicated asynchronously between them. Each region uses PA/EL internally with LOCAL_QUORUM for writes and LOCAL_ONE for reads. Cross-region reads may see stale data up to the replication lag (typically 100ms to 2 seconds depending on distance).

Use when: The application can tolerate temporary cross-region inconsistency. User session data, product catalogs, content feeds.

### Active-Passive (PC/EC Between Regions)

One region is active. The passive region accepts reads but not writes. All writes go through the active region and are replicated synchronously to the passive region (or asynchronously, but reads from passive are guaranteed to see all data from active within bounded staleness).

Use when: Financial data, inventory systems, any data requiring global strong consistency.

### Per-Region Trade-off

The PACELC trade-off is made twice: once within a region (how many local replicas must agree) and once between regions (how many regions must agree before responding). A system with RF=3 per region and 3 regions has 9 total replicas. Requiring all 9 to agree makes the system PC/EC but creates extreme write latency (200ms+). Requiring 1 of 9 to agree makes it PA/EL but creates high inconsistency.

## Capacity Planning Math

### PA/EL Capacity

Each node in a PA/EL system handles approximately 10K to 50K operations per second for mixed read/write workloads. To handle 1M writes/second with RF=3:

- Total write amplification: 3x (each write goes to 3 replicas)
- Effective throughput needed: 3M ops/s across the cluster
- Nodes needed: 3M / 30K ops/node = 100 nodes
- At 16 cores and 64GB RAM per node: 100 nodes

### PC/EC Capacity

Each node in a PC/EC system handles approximately 1K to 10K operations per second due to coordination overhead. To handle 100K writes/second with RF=3:

- Each write requires prepare + commit: 2x message amplification
- Quorum requires 2 of 3 nodes: 2/3 fraction
- Total amplification: 3 * 2 * (2/3) = 4x
- Effective throughput needed: 400K ops/s
- Nodes needed: 400K / 5K ops/node = 80 nodes
- At 32 cores and 128GB RAM per node: 80 nodes

### Cross-Region Bandwidth

For PA/EL async replication: bandwidth = write_throughput * average_record_size * replicas_per_region.

For 1M writes/s at 1KB per record, 3 regions: 1M * 1KB * 2 (replicas in other regions) = 2GB/s per region pair. This requires dedicated cross-region links.

For PC/EC sync replication: bandwidth = write_throughput * (2 round trips) * average_record_size * replicas_per_region. Same 1M writes pattern: 2GB/s * 2 = 4GB/s per region pair.

## Deployment and Operations

### Deployment Topology

- Each region: 3 to 5 availability zones.
- Each AZ: 8 to 32 nodes per data partition.
- Replication factor: 3 per region.
- Consistency levels: LOCAL_ONE for default, LOCAL_QUORUM for critical.

### Monitoring

Key PACELC metrics to alert on:

| Metric | Warning | Critical | Action |
|---|---|---|---|
| Replication lag | > 1 second | > 10 seconds | Check network, reduce write throughput |
| Read repair count | > 100/min | > 1000/min | Nodes may be unhealthy, check anti-entropy |
| Hinted handoff depth | > 10K | > 100K | Node may be down, check and repair |
| Quorum failures | > 0.1% | > 1% | Nodes down or partition in progress |
| p99 write latency (PA/EL) | > 50ms | > 200ms | Check coordinator node load |
| p99 write latency (PC/EC) | > 200ms | > 500ms | Check lock contention or TrueTime errors |

### Operational Runbooks

**Partition detected (PA/EL):** Continue serving. Monitor conflict resolution. No immediate action needed. Verify partition heals within expected window.

**Partition detected (PC/EC):** Alert immediately. Minority-side clients are seeing errors. Force failover if the partition does not heal within 30 seconds.

**Node failure (PA/EL):** No impact. New writes go to other replicas. Hinted handoff accumulates. Repair or replace the node within 3 hours to avoid hint expiry.

**Node failure (PC/EC):** Check quorum. If majority remains, continue. If quorum lost, stop writes. Replace failed node urgently.
