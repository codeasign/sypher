# PACELC — Challenges

Three tiers of exercises to verify your understanding of the PACELC theorem. Each tier builds on the previous one. Complete a tier before moving to the next.

---

## Tier 1: Foundation (3 Questions)

### Question 1: PACELC Classification

Classify each scenario into its PACELC quadrant (PA/EL, PC/EC, PA/EC, or PC/EL):

1. A Cassandra cluster with RF=3, read consistency LOCAL_ONE, write consistency LOCAL_ONE.
2. A Cassandra cluster with RF=3, read consistency QUORUM, write consistency QUORUM.
3. A DynamoDB table with default settings.
4. A DynamoDB GetItem with ConsistentRead=True.
5. A Spanner read-only transaction with exact-staleness read at timestamp 10 seconds ago.
6. A Zookeeper ensemble for leader election during a network partition.

### Answer

1. **PA/EL.** Read and write use ONE replica. Fast response. Stale reads possible. During a partition, both sides accept requests.
2. **PC/EC.** Quorum provides strong consistency. During a partition, the minority side loses quorum and rejects requests.
3. **PA/EL.** DynamoDB default writes to one replica and propagates async. Reads return from the nearest replica.
4. **PA/EC.** The write path is still PA/EL (async to replicas), but this specific read waits for all replicas to respond with the latest version. During partition, this read may fail on the minority side.
5. **PC/EL.** The base system is PC/EC (Spanner), but this specific read avoids strong consistency by accepting exact-staleness. The read returns immediately from any replica that has the data at that timestamp. During a partition, this read would still work on the minority side.
6. **PC/EC.** Zookeeper requires majority for all operations. During a partition, the minority side becomes unavailable.

### Question 2: Regime Identification

Describe what happens in each regime for a PA/EL system handling user profile data:

- **Else regime (normal operation):**
- **Partition regime (network split):**
- **Partition healing:**
- **Single node failure:**

### Answer

**Else regime:** A read for a user profile returns data from the nearest replica in 1-3ms. The profile may be up to 200ms stale if the user just updated their name. A write to the profile writes to one replica, responds in 2-5ms, and propagates asynchronously.

**Partition regime (3-node cluster split 2-1):** The majority side (2 nodes) continues accepting reads and writes normally. The minority side (1 node) also accepts reads and writes. If a user updates their profile on the majority side and another update happens on the minority side, the two values will diverge. Both users see their update immediately. Neither sees the other's change.

**Partition healing:** The minority node reconnects. The system detects the version conflict for the user profile key. LWW resolution compares timestamps: the later write wins. The earlier write is discarded. The node receives the winning value and the cluster converges.

**Single node failure:** New writes for keys whose primary replica is the failed node go to other replicas via hinted handoff. Reads for those keys are served by the remaining healthy replicas. When the node recovers, hinted handoff forwards the buffered writes. Read repair fixes any remaining inconsistent keys.

### Question 3: Latency vs Consistency

A system has 5 replicas. It switches from read consistency ONE (PA/EL) to read consistency QUORUM (PC/EC). The write consistency remains QUORUM.

- What happens to p50 read latency?
- What happens to p99 read latency?
- What happens to write latency?
- What happens to stale read probability?

### Answer

- p50 read latency increases from 1-3ms to 8-15ms. The coordinator must now wait for 3 of 5 replicas to respond instead of 1.

- p99 read latency increases from 20-50ms to 100-300ms. The slowest replica in the quorum dictates the tail latency. With ONE, the coordinator only waits for the fastest replica.

- Write latency is unchanged because write consistency is already QUORUM. The write path is the same.

- Stale read probability drops from approximately 5% (depending on replication lag) to effectively 0%. With W=3 and R=3, the read quorum overlaps with the write quorum by at least 1 node, guaranteeing the read sees the latest write.

## Tier 2: Design (3 Questions)

### Question 4: Multi-Tier PACELC

Design a ride-sharing platform (like Uber or Lyft) with these subsystems: rider app, driver app, trip matching, pricing, payment, and ride history.

For each subsystem, specify:
- The PACELC quadrant
- The consistency level(s) used
- What happens during a partition
- What happens during normal operation with high latency

### Answer

**Rider app (location and map data):**
- Quadrant: PA/EL
- Consistency: CL=ONE reads and writes
- During partition: Rider sees their own location on their side of the split. Driver locations shown to riders may be stale or missing on the other side.
- High latency normal operation: Location updates may be queued and delivered in batches. Riders see slightly delayed driver positions.

**Driver app (location updates):**
- Quadrant: PA/EL
- Consistency: CL=ONE writes (high-frequency location pings)
- During partition: Driver's location updates are accepted on their local region partition. Cross-region riders may see stale location.
- High latency normal operation: GPS pings arrive late but are still accepted. Driver availability state is eventually consistent.

**Trip matching:**
- Quadrant: PC/EC for the matching decision
- Consistency: Read and write both CL=QUORUM for the match record
- During partition: Only the majority side can create new trips. Riders on the minority side cannot be matched. Drivers on the minority side are shown as unavailable.
- High latency normal operation: Matching takes longer (waiting for quorum) but is correct. No double-matching risk.

**Pricing (surge pricing):**
- Quadrant: PA/EL for displayed price estimates, PC/EC for the final charged price
- Consistency: Estimates use CL=ONE, charge uses CL=QUORUM
- During partition: Surge estimates may be different on each side. The final charge is computed from the partition side that handles the trip.
- High latency normal operation: Price estimates respond quickly. The final charge may take longer to compute.

**Payment:**
- Quadrant: PC/EC
- Consistency: All operations use CL=QUORUM or CL=ALL. Idempotency keys for all payment calls.
- During partition: Payments on the minority side fail. Riders cannot complete trips on that side. Payments on the majority side proceed normally.
- High latency normal operation: Payment processing takes 50-200ms but guarantees correctness.

**Ride history:**
- Quadrant: PA/EL
- Consistency: CL=ONE reads, CL=LOCAL_QUORUM writes
- During partition: Recent trip history may show different data on each side. Historical data (older than 1 hour) is fully replicated and consistent.
- High latency normal operation: History reads are fast. New trip records are written with local quorum for durability.

### Question 5: Partition Tolerance at Extreme Scale

Design a social media platform's notification system. The system delivers 500M notifications per day across 5 regions. A user expects to see their notification count update within 5 seconds of the event. During a partition between us-east-1 and eu-west-1, what does each region show? How do you reconcile the data when the partition heals?

### Answer

**Architecture:** Cassandra or DynamoDB with PA/EL across all regions. Each region stores notifications for users in that region. Notifications for a user are always written to the user's home region first, then replicated asynchronously to other regions.

**During partition between us-east-1 and eu-west-1:**
- A user in us-east-1 with home region us-east-1: No impact. All notifications are written and read locally. The user sees updates in real time.
- A user in eu-west-1 with home region us-east-1: The notification write happens in us-east-1. Replication to eu-west-1 is blocked by the partition. The user in eu-west-1 sees stale notification counts (last replicated before the partition). New notifications are queued in us-east-1. When reading from eu-west-1, the user sees data that is minutes or hours old.
- A user in eu-west-1 with home region eu-west-1: No impact. Notifications written and read locally within eu-west-1.

**Partition healing reconciliation:**
1. The replication channel reopens. Queued notifications from us-east-1 flow to eu-west-1.
2. Each notification has a unique ID and timestamp. No conflict resolution is needed — notifications are append-only, each with a unique key.
3. The notification count is recomputed from the notification list. Since notifications are additive, the count converges to the correct total.
4. Users who were on the "wrong" side of the partition see their notification count update within 30 seconds of the partition healing.

**Monitoring:** Replication lag between regions is the key metric. Alert when lag exceeds 30 seconds. During partition, the alert fires continuously and is expected — suppress it. When the partition heals, monitor for the lag to drop below 5 seconds.

### Question 6: Hybrid PACELC for Financial Data

A fintech app processes stock trades. It has two data stores: one for market data (stock prices, charts) and one for order data (buys, sells, account balances). The market data store receives 50K updates/second. The order data store receives 500 writes/second.

Design the PACELC strategy for each store. Justify your choices with concrete numbers.

### Answer

**Market data store:**
- Quadrant: PA/EL with RF=3, CL=ONE for reads and writes.
- Rationale: Stock prices change constantly. A stale price by 200ms is acceptable. Users expect the chart to update smoothly, not to block waiting for the latest tick. The system must handle 50K writes/second, and strong consistency would reduce throughput by 10x.
- During partition: Each side shows the prices from its own data feed. Prices may diverge by a few cents. When the partition heals, LWW reconciliation picks the later price. The divergence is temporary and self-correcting.
- Performance: p50 write 2ms, p50 read 1ms, 50K writes/node.

**Order data store:**
- Quadrant: PC/EC with RF=5, W=3, R=3.
- Rationale: Account balances and order execution must be strongly consistent. A double-execution due to stale reads could cause financial loss. The write volume is low (500 writes/second), so the coordination overhead is acceptable.
- During partition: Only the majority side (3+ of 5 nodes) can execute orders. The minority side rejects order attempts and returns an error. This is acceptable — an order rejected during a network issue is better than an order executed twice.
- Performance: p50 write 15ms, p50 read 10ms, 5K writes/node maximum. With 500 writes/second across 5 nodes, the system operates at 2% of capacity — well within limits.

**Cross-system interaction:** When an order is placed, the system reads market data (PA/EL — fast, possibly stale) and writes the order (PC/EC — strong, correct). The trade-off is that the order executes against a price that may be 200ms old. For a brokerage, this is acceptable: the fill price is confirmed by the exchange, and the system records the actual fill, not the displayed price.

## Tier 3: Expert (2 Questions)

### Question 7: Designing a New PACELC Variant

A team proposes a new system with the following behavior:
- During normal operation: writes complete in 1ms, but the system guarantees that any read within 500ms of a write will see that write.
- During a partition: both sides accept reads, but writes are only accepted if more than 50% of nodes are reachable.

What PACELC quadrant does this system occupy? Is it achievable? What mechanisms would be needed?

### Answer

**PACELC classification:** PA with a session-like consistency guarantee in the Else regime, and CP in the Partition regime. The Else regime is not pure EL (eventual low latency) and not pure EC (strong consistency). It is EL with a bounded staleness guarantee.

**Is it achievable?** Partially. The normal-operation guarantee (read-your-writes within 500ms) is achievable with session consistency: the coordinator maintains a mapping of which keys each client has written and routes reads for those keys to replicas that have acknowledged the latest write.

The 500ms bound requires careful engineering:
- All replicas must acknowledge the write within 500ms (easily achievable within a data center).
- The coordinator must track write completion per key-client pair (memory bounded by active clients).
- Reads must go to replicas known to be up-to-date (may increase latency to 3-5ms instead of 1ms).

The partition regime behavior (writes require majority) is CP-like. This is achievable with a quorum-based commit protocol. The challenge is the transition: the system must detect that it has lost majority and switch from fast writes to quorum writes. This adds latency during the detection window.

**Limitations:**
- The 500ms guarantee only applies to read-your-writes, not to cross-client consistency. Client A writing and client B reading within 500ms may still see stale data.
- The partition detection delay (3-15 seconds) means the system may accept non-majority writes during a partial partition before declaring majority loss. This creates a window of potential divergence.

**Mechanisms needed:**
1. Client-coordinator session table with last-written timestamp per key.
2. Read routing logic that checks session table before choosing a replica.
3. Partition detector with consensus-based majority determination.
4. Write path selector: switch between one-replica write (fast) and quorum write (safe).

### Question 8: PACELC in a Multi-AZ, Multi-Region Deployment

Design a global PACELC strategy for a cloud-native database that must serve 1M writes/second with the following constraints:
- 3 regions (us-east-1, eu-west-1, ap-southeast-1)
- 3 availability zones per region
- Replication factor 3 per region
- Cross-region replication: async or sync configurable per keyspace
- p99 write latency must stay under 50ms for 95% of keyspaces
- One keyspace (financial transactions) requires cross-region strong consistency

### Full Design

**Within-region architecture:** RF=3 across 3 AZs in each region. Default consistency: LOCAL_ONE for PA/EL, with optional LOCAL_QUORUM.

**Cross-region topology:** Asynchronous replication for 95% of keyspaces. Synchronous replication (using a consensus protocol across regions) for the financial keyspace.

**Per-keyspace PACELC configuration:**

| Keyspace | Intra-Region | Cross-Region | PACELC |
|---|---|---|---|
| product_catalog | LOCAL_ONE | Async | PA/EL |
| user_sessions | LOCAL_ONE | Async | PA/EL |
| activity_feed | LOCAL_ONE | Async | PA/EL |
| order_history | LOCAL_QUORUM | Async | PA/EC (intra-region) |
| financial_txns | LOCAL_QUORUM | Sync (Paxos) | PC/EC (global) |

**Latency analysis:**
- PA/EL keyspace: write completes in 2-5ms within region, async cross-region adds 100-500ms to full propagation. User always sees fast local writes.
- PC/EC keyspace (financial): write requires LOCAL_QUORUM (2 of 3 nodes) plus sync Paxos across 3 regions (2 of 3 region leaders). Expected latency: 50-150ms per write. This exceeds the 50ms p99 constraint for this specific keyspace, so a different approach is needed.

**Optimization for financial keyspace:** Use a two-phase approach:
1. Write to local region with LOCAL_QUORUM (12ms).
2. Replicate asynchronously to other regions (100ms background).
3. Before confirming to the client that the transaction is visible to other regions, wait for async acknowledgment from at least one other region (60ms total).

This gives a p99 of 60-70ms for the financial keyspace — slightly above the general 50ms budget but acceptable for financial data. The alternative (fully synchronous Paxos across all regions) would push p99 to 200-500ms, which would break SLAs for the overall system.

**Failure modes addressed:**
1. **Region failure:** PA/EL keyspaces switch to the nearest healthy region. Data staleness up to async replication lag. Financial keyspace: use an active-passive model where a second region takes over as write leader. RPO (recovery point objective) is near-zero due to the async-acknowledgment design.
2. **AZ failure:** LOCAL_QUORUM within the surviving AZs. If only 2 of 3 AZs survive, LOCAL_QUORUM still works (need 2 out of 2 or 2 out of 3). If only 1 AZ survives, writes are rejected for the financial keyspace (quorum loss).
3. **Global partition:** If the US and Europe are partitioned, PA/EL keyspaces continue in both regions with eventual reconciliation. The financial keyspace leader election detects the partition and promotes one region leader. Writes go to the promoted leader. The partitioned region reads from its local copy (possibly stale) until the partition heals.
