# Vertical Scaling — Trade-offs and Failure Modes

## The Core Tension

Vertical scaling trades operational simplicity and strong consistency for a hard physical ceiling and a super-linear cost curve that eventually makes continued growth uneconomical.

## Decision Matrix

Every vertical scaling decision involves choosing between scaling up (bigger machine) and scaling out (more machines). The right choice depends on workload, cost constraints, and growth timeline.

## CAP Theorem Positioning

A vertically scaled single-node system is a CA system in CAP theorem terms — it provides Consistency and Availability because there is no network partition within a single machine. There is no P (Partition Tolerance) trade-off because there is no distribution.

## Failure Modes

Vertical scaling avoids distributed systems failure modes (network partitions, split-brain, clock skew) but introduces machine-level failure modes that are different in character.

### Failure 1: The OOM Killer Strikes

**What happens**: The Linux Out-Of-Memory (OOM) killer terminates a process when system memory is exhausted. On a database server, the OOM killer often targets the process using the most memory — which is the database itself.

**Why it happens**: A background job, a new application deployment with a memory leak, or an unexpected query that builds an enormous hash table consumes memory beyond the allocated budget. The kernel's OOM killer selects the process with the highest `oom_score` — usually the database because it has the largest memory footprint.

**Blast radius**: Total database outage. All connections drop. In-flight writes are lost (not yet in WAL). Recovery: WAL replay on restart, taking 5-60 minutes depending on data size.

### Failure 2: NUMA Cross-Socket Saturation

**What happens**: On a multi-socket server, threads pinned to one socket access memory allocated on another socket. Cross-socket memory access is 50-100% slower than local access. When cross-socket traffic exceeds 30% of total memory access, throughput drops significantly.

**Why it happens**: Default NUMA balancing in Linux automatically migrates memory pages between sockets. This is designed to balance memory usage but adds migration overhead. Without explicit NUMA pinning, the database process may have threads running on socket 0 accessing memory on socket 1.

**Blast radius**: 20-40% throughput degradation. No outage, but performance is well below what the hardware should deliver.

**Detection**: Run `numastat -p postgres`. If `other_node` exceeds 20% of total memory access, NUMA tuning is needed.

**Recovery**: Restart the process with `numactl --cpunodebind=0 --membind=0` or configure `numa_memory = off` in PostgreSQL.

### Failure 3: EBS Bandwidth Throttling

**What happens**: EBS volumes have burst credits for IOPS. When credits are exhausted, IOPS throttle to the baseline rate. For gp3 volumes, the baseline is 3,000 IOPS regardless of volume size. A database that exceeds this for sustained periods will see I/O latency spike.

**Why it happens**: Instance types have EBS bandwidth limits. A `r7g.4xlarge` can drive up to 20 Gbps to EBS. The gp3 volume may support only 1,000 MB/s throughput. If the database writes WAL at 500 MB/s (checkpoint burst), the volume saturates and I/O queue depth increases.

**Blast radius**: Write latency increases from 2ms to 50ms+. Read latency also increases because writes consume I/O capacity. Application timeouts if queries exceed statement_timeout during this period.

**Detection**: CloudWatch EBS metrics: `VolumeQueueLength` over 100, `VolumeWriteBytes` at volume maximum.

**Recovery**: Increase provisioned IOPS on the volume (io2 with higher IOPS), or spread WAL across multiple volumes, or use instance-store NVMe for temporary WAL storage.

### Failure 4: Connection Pool Starvation

**What happens**: The application connection pool is too small for the number of concurrent requests. All connections are in use, and new requests queue. If queuing exceeds the pool timeout, requests fail.

**Why it happens**: After a vertical CPU scaling, the application can handle more concurrent requests — but the connection pool was not increased correspondingly. With 32 cores and 16 gunicorn workers, each worker may need multiple database connections for parallel requests, exhausting the pool.

**Blast radius**: Partial outage — percentage of requests fail proportionally to how many exceed the pool capacity. Cascade: failed requests are retried by clients, creating a retry storm that further consumes pool connections.

```
Connection pool math:
  App workers: 16 (gunicorn)
  Connections per worker (peak): 3 (parallel queries)
  Pool size: 20
  
  At peak: 16 * 3 = 48 connections needed
  Available: 20
  Queued requests: 28
  Pool timeout: 5 seconds
  
  After 5 seconds: 28 requests fail with "pool timeout"
  Clients retry: 28 more requests queued
  Pool stays saturated: 20 active, rest failing
```

### Failure 5: Storage Volume Pinning

**What happens**: When upgrading instance types, the root EBS volume may be unpinned from the old instance type's optimized path. IOPS and throughput drop because the new instance type uses a different EBS optimization configuration.

**Why it happens**: Instance types have different EBS-optimized bandwidth. A `c5.xlarge` supports up to 2.5 Gbps EBS bandwidth. A `c5.4xlarge` supports up to 4.75 Gbps. If an EBS volume was attached to the `c5.xlarge` and then the instance is stopped and started as `c5.4xlarge`, the volume inherits the new instance's EBS configuration — but if the volume was attached during the stop, it may not be recognized as EBS-optimized by the new instance until explicitly re-attached.

**Blast radius**: I/O throughput drops to non-optimized levels (approximately 50% of expected). Database performance degrades until the volume is detached and re-attached.

**Recovery**: Detach and re-attach the EBS volume after the instance type change, or use the AWS Console's "Modify Instance" action which handles EBS optimization correctly.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Scaling CPU for an I/O-Bound Workload

Adding CPU cores when the bottleneck is disk I/O. The new cores sit idle because threads are stuck waiting on I/O. Cost increases with zero throughput gain.

- **Why it happens**: The monitoring dashboard shows high CPU, but the CPU is in `iowait` state, not user/system time. `vmstat` shows `wa` (wait) at 30%+. 
- **What to do instead**: Check `iostat -x` before upgrading CPU. If `await` is above 10ms or disk utilization is above 80%, upgrade storage, not CPU.

### Anti-Pattern 2: Vertical Scaling Without Cooldown Monitoring

Upgrading an instance and immediately declaring victory without monitoring for 72 hours. The new instance may have a different performance profile, and the system may have shifted to a new bottleneck.

- **Why it happens**: The symptom (e.g., slow queries) disappears after the upgrade. The team assumes the problem is solved. Three weeks later, a different symptom appears (e.g., connection timeouts) and it takes days to diagnose.
- **What to do instead**: After every vertical scaling event, run a 72-hour monitoring watchlist. Check CPU, memory, I/O, network, and connection metrics daily. Document the new baseline.

### Anti-Pattern 3: Ignoring NUMA Topology

Deploying a database on a 64+ core instance without NUMA configuration. The database suffers 25-40% throughput loss due to cross-socket memory access penalties.

- **Why it happens**: The team is familiar with smaller instances (up to 16 cores) that are single-socket. They deploy to a larger instance without understanding the multi-socket NUMA topology.
- **What to do instead**: Run `lscpu` to check NUMA node count before deploying. If more than one NUMA node exists, pin the database process to a single node or configure NUMA-aware memory allocation.

### Anti-Pattern 4: Over-Provisioning on Premises

Buying a physical server with 4x the current capacity to avoid future scaling costs. The server sits at 15% utilization for 2 years while the hardware depreciates.

- **Why it happens**: Capital expenditure (CapEx) mindset — buying hardware upfront is cheaper than monthly cloud costs over 3 years. But the utilization curve means you pay for capacity you do not use.
- **What to do instead**: Use cloud instances with 6-12 month reservations. Scale up on demand rather than purchasing for 3 years of growth. The cloud premium is worth it when utilization is variable.

### Anti-Pattern 5: License Cost Trap

Running a commercial database (Oracle, SQL Server Enterprise) on a large instance without accounting for per-core licensing. The license cost for 64 cores at $47,500/core totals $3,040,000 — dwarfing the instance cost.

- **Why it happens**: The engineering team focuses on cloud instance costs and neglects the licensing cost model. Procurement approves the instance upgrade without checking the license agreement.
- **What to do instead**: Compare total cost of ownership (instance + license) before scaling up. For commercial databases, consider migrating to PostgreSQL before hitting the per-core cost wall.

### Anti-Pattern 6: Cache All the Things on One Node

Adding more and more cache (Redis, Memcached, in-process caches) to the same single machine instead of addressing the underlying data architecture. The machine becomes a caching appliance with a side of application logic.

- **Why it happens**: Adding cache is easy — install Redis, point the app at it, see latency drop. Doing it repeatedly avoids the hard work of sharding or moving to a distributed database.
- **What to do instead**: Design a cache strategy: what is cached, what TTL, what eviction policy, and when is cache not enough. Set a hard limit on cache memory per instance (e.g., Redis uses at most 40% of instance RAM). When that limit is hit, it is time to architect for horizontal scaling.

Vertical scaling is the wrong choice in these situations:

**When growth exceeds the largest instance within 12 months.** If projections show the working set will exceed the largest available instance in 12 months, start designing for horizontal scaling now. The migration from a single node to a distributed system takes 6-18 months for a complex application. Waiting until the ceiling is hit creates a crisis.

**When the workload has unpredictable spikes of 10x+.** A single machine cannot handle a 10x traffic spike that lasts 30 minutes. Auto-scaling horizontally costs less during quiet periods and handles spikes via additional instances.

**When zero-downtime upgrades are mandatory.** Every vertical scaling instance upgrade requires a stop-start or blue-green switch. For four-nines (99.99%) uptime targets, this window is too large. Horizontal deployments allow rolling upgrades without downtime.

**When the data is already partitioned by nature.** If the application naturally operates on tenant-based or region-based data, horizontal scaling maps well to the domain. A multi-tenant SaaS application is better served by per-tenant database instances than a single massive machine.

**When the team lacks capacity to manage a single critical machine.** A vertically scaled database is a single point of failure. If the team does not have the operational maturity to manage backups, replication, failover, and capacity planning for this single machine, the risk of catastrophic data loss is high. Horizontal systems distribute this risk across smaller machines, each with less blast radius.
