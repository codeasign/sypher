# Vertical Scaling at Scale

Vertical scaling is not a single architectural choice — it is a series of architectural patterns that evolve as the system outgrows each machine size. The naive implementation works for 1,000 users. The production-grade architecture handles 100,000 users. The multi-region variant pushes toward 1,000,000 users on a single-node stack. Each stage has specific patterns for storage, caching, deployment, and operations.

## Naive Implementation

The simplest vertical scaling architecture: everything on one machine. This is the starting point for most systems and is correct for small scale.

### What Breaks at Each Scale

**At 1,000 users (breakpoint: database and app compete for resources)**
Both the application and database run on the same machine. When the app spawns workers to handle requests, each worker uses memory and CPU that the database needs. A single slow query from a new feature can cause the app to time out because PostgreSQL consumed all available RAM for sorting. The database's `shared_buffers` and the application's memory footprint fight for the same pool.

**At 100,000 users (breakpoint: single-machine limits)**
The application and database now run on separate machines (the first vertical scaling upgrade). The database server is a large instance — 64 GB RAM, 16 vCPU. But the 64 GB RAM is only 60% of the working set. Queries that used to hit the buffer cache now hit disk. The cache hit ratio drops from 98% to 65%. Query latency jumps from 5ms to 200ms. The application server has spare CPU, but the database is the bottleneck. You upgrade to 128 GB RAM. Cache hit ratio returns to 98%. Problem solved — for another 12 months.

**At 10,000,000 users (breakpoint: no larger instance exists)**
The database is on a `r7g.16xlarge` (512 GB RAM, 64 vCPU) with `io2` Block Express volumes (256,000 IOPS). Cost: $4,000/month for the instance, $3,000/month for the storage. The working set is 480 GB — 94% of available RAM. Any growth pushes it over the edge. There is no larger memory-optimized instance in the current generation. You must either migrate to the `u-*` high-memory family (up to 24 TB RAM, but at $50,000+/month) or design a horizontal scaling architecture.

## Integration Patterns

Vertical scaling succeeds or fails based on how the application integrates with external dependencies. Four critical integration patterns determine whether the system degrades gracefully or collapses under load.

### Pattern 2: Cache Integration

A single-node Redis or Memcached instance must be configured for vertical scaling. The key settings are memory limit, eviction policy, and persistence strategy.

```
Redis on r7g.4xlarge (128 GB RAM, allocation for Redis = 100 GB):
  maxmemory: 100gb
  maxmemory-policy: allkeys-lru
  maxmemory-samples: 10

  When memory fills: LRU eviction removes least-used keys
  Target: eviction rate under 100 keys/second
  At eviction > 500 keys/sec: too little RAM — vertical scaling needed

Cache miss penalty:
  Cache hit:   ~1ms (Redis read, local network)
  Cache miss:  ~10ms (database query)
  At 90% hit rate: average latency = 0.9*1ms + 0.1*10ms = 1.9ms
  At 70% hit rate: average latency = 0.7*1ms + 0.3*10ms = 3.7ms
  At 50% hit rate: average latency = 0.5*1ms + 0.5*10ms = 5.5ms

  When hit rate drops below 80%, memory is the bottleneck.
  Solution: vertical scaling of Redis instance or adding a Redis cluster.
```

### Pattern 3: Message Queue Integration

A single-node message queue (RabbitMQ, Redis Streams, or Kafka on one broker) is simple but has clear limits.

```
Single-node queue limits:
  RabbitMQ:    ~50,000 messages/sec (single queue, persistent)
  Redis       ~100,000 messages/sec (Streams, non-persistent)
  Streams:
  Kafka:      ~200,000 messages/sec (single broker, 3 partitions)

When these limits are hit on a single node:
  - Add more partitions (Kafka) — still on one server, but more parallelism
  - Add more consumer threads — uses more CPU
  - Eventually: split queue across separate nodes (horizontal)

Vertical scaling bottleneck for queues:
  - Network throughput: 10 Gbps ENA max per instance
  - Disk I/O: 16,000 IOPS gp3 max per volume
  - Memory: queue backlog size limited by RAM
```

## Capacity Planning

Capacity planning for vertical scaling answers one question: how long until the current instance is insufficient?

### Capacity Math Example

```
System: Real-time analytics dashboard
Current state:
  Instance: r7g.4xlarge (16 vCPU, 128 GB RAM)
  Working set: 80 GB (62% of RAM)
  Cache hit ratio: 97%
  Query throughput: 20,000 QPS
  CPU at peak: 60%
  Storage: 2 TB NVMe (55% used)
  
Growth projections:
  Data growth: 15 GB/month (from new data ingestion)
  Query growth: 8% month-over-month (from new users)

Calculations:

1. Memory saturation:
   Current RAM: 128 GB
   Working set: 80 GB
   Growth: 15 GB/month
   Time until working set = 80% of RAM (102 GB):
     (102 - 80) / 15 = 1.47 months until cache pressure
   Time until working set = 95% of RAM (121 GB):
     (121 - 80) / 15 = 2.73 months until critical
  
   Next instance: r7g.8xlarge (256 GB RAM, $1,612/month)
   Time until saturation on new instance:
     (204 GB = 80%) in (204 - 80) / 15 = 8.3 months
     (243 GB = 95%) in (243 - 80) / 15 = 10.9 months

2. CPU saturation:
   Current: 20,000 QPS at 60% CPU
   Max at 80% (scale-out threshold): 26,667 QPS
   Max at 100%: 33,333 QPS
   Growth: 8% monthly
   Time to 80%: 20,000 * 1.08^t = 26,667 => t = 3.8 months
   Time to 100%: 20,000 * 1.08^t = 33,333 => t = 6.5 months
  
   Next instance: c7g.8xlarge (32 vCPU, ~40,000 QPS at 60%)
   Time to 80%: 40,000 * 1.08^t = 53,333 => t = 4.0 months

3. Storage saturation:
   Current: 1.1 TB used of 2 TB (55%)
   Growth: 15 GB/month
   Time to 80% (1.6 TB): (1600 - 1100) / 15 = 33.3 months
   Time to 95% (1.9 TB): (1900 - 1100) / 15 = 53.3 months
   (Storage is not the immediate bottleneck)

4. Upgrade decision:
   Memory runs out in 2.7 months
   CPU runs out in 6.5 months
   Storage runs out in 33 months
   
   Recommendation: Upgrade to r7g.8xlarge in 2 months (memory)
   That provides 8 more months of memory headroom.
   CPU will need re-evaluation at that point.

5. Cost analysis:
   Current: r7g.4xlarge at $589/month
   Proposed: r7g.8xlarge at $1,177/month
   Cost increase: $588/month ($7,056/year)
   
   Alternative: Add Redis cache (r7g.xlarge at $147/month) 
   to reduce working set pressure by caching query results.
   Combined cost: $589 + $147 = $736/month
   Extends memory timeline from 2.7 to ~8 months at lower cost.
   
   Recommendation: Add Redis cache first, delay instance upgrade.
```

## Deployment and Operations

Vertical scaling requires specific operational patterns to avoid downtime and data loss.

### Zero-Downtime Instance Upgrade

Upgrading from one instance type to a larger one traditionally requires downtime. Here is the blue-green approach:

```
Zero-downtime database upgrade (PostgreSQL):

  Phase 1: Provision new instance
    1. Launch new instance (e.g., r7g.8xlarge) in same subnet
    2. Configure PostgreSQL from template
    3. Configure WAL streaming from old instance
    4. Start PostgreSQL in recovery mode (replica)
  
  Phase 2: Sync data
    1. Wait for replication lag to reach 0
    2. Validate data: row count checksum on largest tables
    3. Run ANALYZE on new instance (warm caches)
  
  Phase 3: Switch traffic
    1. Pause application (connection drain — 30s window)
    2. Promote new instance to primary (SELECT pg_promote())
    3. Update DNS/connection string to new instance IP
    4. Resume application traffic
    5. Verify: query latency, cache hit ratio, connections
  
  Phase 4: Decommission old instance
    1. Keep old instance running for 48 hours (rollback window)
    2. Snapshot old data volume (final backup)
    3. Terminate old instance

  Total downtime: 30-60 seconds (connection drain window)
```

### Monitoring

Key metrics for a vertically scaled node:

```
Critical alerts (P0 — page on-call):

  1. memory.used_ratio > 90% for 5 minutes
     - Working set exceeds RAM, cache misses increasing
     - Action: Scale up or add cache layer
  
  2. cpu.utilization > 90% for 5 minutes
     - CPU saturated, request queue growing
     - Action: Scale up or move CPU-heavy work to async jobs
  
  3. disk.io_util > 80% for 10 minutes
     - EBS IOPS limit being approached
     - Action: Increase provisioned IOPS or move to NVMe
  
  4. postgresql.cache_hit_ratio < 90% for 15 minutes
     - Buffer cache inadequate for working set
     - Action: Increase shared_buffers or add RAM

Warning alerts (P1 — Slack notification):

  1. memory.used_ratio > 75% for 30 minutes
  2. cpu.utilization > 75% for 15 minutes
  3. disk.used_ratio > 80%
  4. postgresql.connection_usage > 80%
  5. redis.eviction_rate > 100/sec
  6. query.p99_latency > 500ms for 5 minutes
```

### Debugging

When a vertically scaled system misbehaves after an upgrade, trace through these steps:

```
Post-upgrade debugging checklist:

  1. Is the bottleneck actually the resource you scaled?
     - Check top/vmstat: CPU idle? iowait?
     - Check free -h: memory used? swap activity?
     - Check iostat -x: await high? queue depth?
  
  2. Did application configuration update correctly?
     - Check shared_buffers: cat /proc/sys/kernel/shmmax
     - Check connection pool size
     - Check max_connections
  
  3. Is NUMA topology causing cross-socket penalties?
     - Run: numastat -p postgres
     - If "other_node" > 20% of total, NUMA pin the process
  
  4. Is the kernel I/O scheduler appropriate?
     - Check: cat /sys/block/nvme0n1/queue/scheduler
     - Should be "none" for NVMe, "none" or "mq-deadline" for EBS
  
  5. Did the instance retain previous EBS volume attributes?
     - Check: aws ec2 describe-volumes | grep Iops
     - If IOPS is lower than provisioned, EBS may be throttling
  
  6. Are application-level caches correctly sized?
     - Redis maxmemory set correctly?
     - Application in-memory cache limits?
```

### Upgrade Strategy Summary

```
Upgrade type              Method                    Downtime      Frequency
─────────────────────────────────────────────────────────────────────────
CPU/RAM                   Blue-green instance swap  30-60 sec     Every 6-18 months
Disk (increase IOPS)      Modify volume (online)    0 sec         As needed
Disk (increase size)      Modify volume + grow2fs   0 sec         As needed
Database tuning           Reload config (pg_reload) 0 sec         After upgrades
OS kernel update          Reboot                    30-90 sec     Quarterly
PostgreSQL major version  pg_upgrade + blue-green   30-60 sec     Annually
Redis upgrade             Replica promotion         5 sec         As needed
```

The architecture patterns described here — naive through multi-region — show that vertical scaling is never a one-time decision. Each stage of growth requires a specific architectural pattern, from co-located everything to separated components with cgroup isolation to multi-region active-passive deployments.
