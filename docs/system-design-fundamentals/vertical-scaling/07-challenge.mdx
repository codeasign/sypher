# Vertical Scaling — Challenges

These challenges test your understanding of vertical scaling through hands-on exercises. For each challenge, sketch your architecture on paper or a whiteboard before writing any code. No AsciiDiagram is provided — you will draw your own as part of the solution.

## Tier 1 — Foundation (5 minutes each)

### Challenge 1.1: Identify the Bottleneck

A web application serves 5,000 requests per second from a single `c5.4xlarge` instance (16 vCPU, 32 GB RAM). At peak load:

- CPU: 75% utilization (all cores)
- Memory: 28 GB of 32 GB used
- Disk I/O: 2,000 IOPS of 3,000 IOPS baseline (gp3)
- Network: 2 Gbps of 10 Gbps available
- Query P99 latency: 850ms
- Connection pool: 50 connections, all active

**Task**: Identify the primary bottleneck. Which resource is saturated and which dependency is failing? Describe your diagnostic process step by step. What single change would produce the largest improvement?

**Evaluation criteria**:
- Correct resource identified as the bottleneck
- Correct diagnostic command or metric named
- Proposed change directly addresses the bottleneck
- Fails if: suggests adding a resource that is not the bottleneck (e.g., "add more RAM" when CPU is the issue)

**Common mistakes**:
- Saying "add more CPU" without checking if the CPU is spending time in iowait
- Blaming "the database" without isolating whether it is CPU, memory, I/O, or connection pool at the database level
- Proposing horizontal scaling when vertical scaling is the simpler first step

**Model answer**: "The connection pool is fully saturated at 50 connections. All threads are active. New requests queue. The average query takes 850ms at P99, so each connection processes approximately 1.2 requests per second. 50 connections x 1.2 RPS = 60 concurrent requests served, but 5,000 RPS incoming means 4,940 requests are waiting for a connection. The pool timeout would cause most of those to fail. The fix: increase the connection pool to at least 200 connections (if database max_connections allows) and verify that the database has sufficient resources to handle the increased connection count."

### Challenge 1.2: Working Set Calculation

A PostgreSQL database has the following characteristics:
- Total data: 500 GB
- Row access frequency (from `pg_stat_user_tables`): 80% of queries hit 60 GB of frequently accessed data, 15% hit another 120 GB, 5% hit the remaining 320 GB
- Current RAM: 64 GB
- Cache hit ratio: 78%

**Task**: Calculate the size of the working set. How much RAM would you need to achieve a 99% cache hit ratio? What is the cost difference between keeping the current instance and upgrading to one that fits the working set?

**Evaluation criteria**:
- Correct identification of the working set (frequently accessed data that should fit in RAM)
- Correct RAM calculation for 99% hit ratio
- Cost calculation with specific instance types and prices
- Fails if: quotes total data size as the working set

**Common mistakes**:
- Assuming working set equals total data size (500 GB). The working set is the set of data accessed frequently enough that caching it in RAM materially improves performance.
- Not accounting for PostgreSQL overhead (shared_buffers at 25% of RAM, connections, OS cache)

**Model answer**: "The working set is the 60 GB that receives 80% of accesses, plus a portion of the 120 GB that receives 15% of accesses. For 99% cache hit ratio, you need the 60 GB frequently accessed data plus approximately 90 GB of the secondary data (enough to bring the miss rate from 20% to 1%), totaling approximately 150 GB of working set. With PostgreSQL overhead (25% of RAM for shared_buffers, plus OS cache), you need approximately 200 GB of RAM. Recommended: r7g.4xlarge (128 GB) for approximately 95% hit ratio, or r7g.8xlarge (256 GB) for 99%. Cost: $589/month vs $1,177/month."

### Challenge 1.3: The 80/20 Rule

A system serves 10,000 requests per second. Database queries account for 80% of response time. Of those queries:
- 20% of the queries (hot queries) take 2ms each when cached
- 80% of the queries (cold queries) take 50ms each (always hit the database)

**Task**: Calculate the current weighted average query latency. If you add a cache that covers the 20% hot queries and reduces their latency to 0.5ms (cache hit), what is the new weighted average? What percentage of total response time is now database vs cache? How much would you need to reduce cold query latency to halve the overall response time?

**Evaluation criteria**:
- Correct weighted average calculation
- Correct before-and-after comparison
- Correct percentage allocation
- Fails if: arithmetic errors in weighted average

**Model answer**: "Current weighted latency: (0.20 x 2ms) + (0.80 x 50ms) = 0.4 + 40 = 40.4ms average. With cache: (0.20 x 0.5ms) + (0.80 x 50ms) = 0.1 + 40 = 40.1ms. The cache only reduced the overall latency by 0.3ms because the cold queries dominate. Database is now 40ms out of 40.1ms = 99.75% of response time. To halve overall latency (20.2ms), you need to reduce cold queries from 50ms to (20.2 - 0.1) / 0.80 = 25.1ms — a 50% reduction in cold query latency. This illustrates: optimizing the fast path does little if the slow path dominates."

## Tier 2 — Design (15 minutes each)

### Challenge 2.1: Scale an Analytics Database

A real-time analytics dashboard currently handles:
- 100 concurrent dashboards, each polling every 30 seconds
- Each dashboard runs 5 aggregate queries (SELECT COUNT, SUM, AVG with GROUP BY time windows)
- Average query time: 800ms
- Data volume: 1 TB, growing at 20 GB/month
- Current instance: r7g.2xlarge (8 vCPU, 64 GB RAM, 5,000 IOPS gp3)
- Working set: 50 GB (frequently accessed last 90 days of data)

**Task**: Design a vertical scaling plan for the next 24 months. Specify instance upgrades at each phase, expected performance improvement, and cost. After the 24-month period, determine whether the system can continue vertical scaling or must migrate to horizontal.

**Scale estimates**:
- Dashboard count grows at 10% per quarter
- Data grows at 20 GB/month
- Query complexity increases by 5% per year (more aggregate functions per dashboard)

**Diagram**: Draw the architecture showing the instance sizes at each phase, the cache hit ratio trajectory, and the point at which a horizontal migration becomes necessary.

**Rubric**:
- Phase 1 upgrade (months 0-8) matches the correct bottleneck
- Phase 2 upgrade (months 8-16) shows awareness of the new bottleneck
- Phase 3 (months 16-24) identifies the ceiling
- Cost analysis includes instance cost and storage cost separately
- Fails if: proposes a single upgrade for the full 24 months

### Challenge 2.2: Cache Layer Integration

A web application with 200,000 daily active users runs on a single server:
- Application: Python (Gunicorn, 8 workers)
- Database: PostgreSQL on the same `c5.4xlarge` (16 vCPU, 32 GB)
- Current P99 latency: 1.2 seconds
- Database cache hit ratio: 72%
- Read-to-write ratio: 20:1
- Average query time (cache hit): 5ms
- Average query time (cache miss): 45ms

**Task**: Design a caching strategy that addresses the cache hit ratio problem. Should you add Redis on the same server, add Redis on a separate server, or vertically scale the database instance? Include:
- Cache key design
- Memory sizing for Redis
- Eviction policy
- Cache invalidation strategy
- Expected improvement in P99 latency
- Cost comparison with vertical scaling the database

**Diagram**: Draw the architecture before and after adding the cache. Show the request flow: user -> app -> cache -> database.

### Challenge 2.3: Blue-Green Database Upgrade

A production PostgreSQL database runs on `r5.4xlarge` (16 vCPU, 128 GB RAM) with 4 TB of data. It must be upgraded to `r5.8xlarge` (32 vCPU, 256 GB RAM) due to memory pressure. The system has a 99.9% uptime SLA (maximum 43 minutes of downtime per month).

**Task**: Design a blue-green deployment plan for the upgrade. Specify:
- Provisioning steps for the new instance
- Data synchronization method and expected sync time
- Replication lag monitoring strategy
- Cutover steps and expected downtime window
- Rollback plan if the cutover fails
- Post-upgrade validation checklist

**Scale estimates**:
- WAL generation rate: 50 MB/second
- Network bandwidth between old and new instance: 10 Gbps
- Total data to sync: 4 TB (initial) + WAL during setup

**Diagram**: Draw the timeline showing old instance, new instance, replication stream, cutover point, and rollback path.

## Tier 3 — Expert (30 minutes each)

### Challenge 3.1: Migrate from Vertical to Horizontal

A SaaS application serves 500 enterprise customers. Each customer has their own data in a shared PostgreSQL database on a single `r7g.16xlarge` (64 vCPU, 512 GB RAM). The largest 20 customers generate 60% of the write traffic. Data is 8 TB total and growing at 25% per year. Write throughput is approaching the single-node ceiling (approximately 50,000 TPS for PostgreSQL with group commit).

**Task**: Design a 12-month migration from the single-node architecture to a horizontally scaled architecture. The new architecture must maintain ACID guarantees per customer and provide a migration path with zero data loss.

**Evaluation dimensions**:
- Migration phase design (is it in-order, with rollback at each phase?)
- Customer isolation strategy (per-customer databases or sharded across many?)
- Dual-write strategy during transition (how to handle writes during migration?)
- Data consistency verification (how to prove migrated data is correct?)
- Rollback plan (what if the horizontal architecture does not perform as expected?)
- Query routing (how does the application know which node to query?)
- Operational runbook (what does the on-call rotation look like for 50 database nodes vs 1?)
- Cost comparison (does horizontal reduce or increase the total bill?)

**Anti-patterns to avoid**:
- Migrating all customers at once (should be gradual, per-customer cutover)
- Assuming the same query patterns work on sharded data (joins across shards become expensive)
- Neglecting to benchmark the new architecture before migration starts
- Assuming the team can manage 50 PostgreSQL instances with the same playbook as 1 instance

**Discussion**: This challenge mirrors the actual migration decisions made by Atlassian (Jira), Discord (messages), and other companies that outgrew single-node architectures. Reflect on why each company chose the path they did — and what you would do differently.

### Challenge 3.2: Extreme Vertical Scaling — Design a 24 TB Server

Your company needs to process 5 PB of reference data for machine learning training pipelines. The data is read-only (ingested once, queried millions of times). Random reads must complete in under 10ms P99. The budget for the database server is $250,000/year.

**Task**: Design a server configuration that handles this workload. Specify:
- Instance type (cloud or dedicated hardware)
- CPU, RAM, and storage specifications
- Database software choice and configuration
- Storage layout (which data on which volume type)
- Benchmark results you would expect (IOPS, throughput, latency)
- Power and cooling requirements (if on-premises)
- Backup strategy for 5 PB on a single machine
- Disaster recovery approach (what happens when this machine fails?)

**Evaluation dimensions**:
- Realistic hardware choices (can this actually be built within budget?)
- NUMA awareness (how to avoid cross-socket penalties with 5 PB of data)
- Storage bottleneck analysis (5 PB on one machine means the storage layer must handle massive throughput)
- Database software suitability (is PostgreSQL appropriate for 5 PB of reference data, or should you use ClickHouse, SingleStore, or a custom solution?)
- Backup window calculation (can you back up 5 PB within a reasonable window?)
- Failure scenario analysis (what happens to the ML pipeline if this machine is down for a week?)

**Anti-patterns to avoid**:
- Proposing a single-server solution for a workload that clearly exceeds the largest available instance (5 PB on one machine is not feasible — the instance you want does not exist)
- Underestimating the backup and recovery time for 5 PB (at 10 Gbps, restoring 5 PB takes 46 days)
- Ignoring the blast radius of a single machine failure (what trains the model while the "database server" is down?)

**Discussion**: This challenge is designed to reveal the hard limits of vertical scaling. If your analysis concludes that this workload cannot be served by a single machine of any practical size, that is the correct answer. Articulate why the limit exists and what alternative architecture (distributed file system, object storage with caching, sharded database) would work instead.

These three tiers of challenges progress from diagnosing a clear bottleneck through designing a caching strategy to planning a complex migration and confronting the hard limits of vertical scaling. Completing all three tiers with the specified time limits demonstrates the depth of understanding expected at the senior and staff engineering levels.
