## Naive Architecture: Single Event Store

The simplest event sourcing architecture runs everything on a single server: the event store, one or more projections, and a read model database. This is a valid starting point for applications with under 1,000 events per second and a single deployment region.

### Single-Node Implementation

In a single-node setup, you can use a regular relational database table as the event store:

```sql
CREATE TABLE events (
    global_position BIGSERIAL PRIMARY KEY,
    stream_id       VARCHAR(255) NOT NULL,
    stream_version  INT NOT NULL,
    event_type      VARCHAR(255) NOT NULL,
    data            JSONB NOT NULL,
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    correlation_id  VARCHAR(255),
    causation_id    VARCHAR(255),

    UNIQUE(stream_id, stream_version)
);

CREATE INDEX idx_events_stream ON events(stream_id, stream_version);
CREATE INDEX idx_events_global ON events(global_position);
CREATE INDEX idx_events_type ON events(event_type);
CREATE INDEX idx_events_correlation ON events(correlation_id);
```

This approach works for small to medium systems. PostgreSQL can handle 5,000 to 10,000 writes per second on modest hardware. With proper indexing, read operations on the event store are fast.

The limitations become apparent at higher scale:

- The single PostgreSQL instance becomes a bottleneck for writes.
- The read model shares the same database, leading to resource contention.
- Projections run in-process, competing for CPU and memory with the application.
- A database failure takes down both the event store and the read model.
- There is no isolation between the write path (event store) and the read path (projections/queries).

## Production Architecture: Distributed Event Store with CQRS

A production architecture separates the write path from the read path using CQRS (Command Query Responsibility Segregation). Event store nodes are dedicated to writes, and read model nodes are dedicated to queries. Projections run as independent services.

### CQRS Separation

CQRS separates the command model (writes) from the query model (reads). In event sourcing, this separation is natural:

- **Command side.** The event store and aggregates handle all writes. The command API accepts commands, validates them through aggregates, and stores events. The command side never returns data to the client (beyond a confirmation and version number).

- **Query side.** Read models handle all queries. The query API accepts read requests and returns data from the read model. The query side never modifies data.

```python
# Command API (write path)
@app.post("/accounts/{id}/withdraw")
def withdraw(id: str, request: WithdrawRequest):
    # All operations go through the aggregate
    account = load_aggregate(id)
    events = account.process_withdrawal(request.amount)
    results = event_store.append(
        stream_id=f"acct:{id}",
        expected_version=account.version,
        events=events
    )
    return {"status": "ok", "version": results[-1].version}


# Query API (read path)
@app.get("/accounts/{id}")
def get_account(id: str):
    # Read directly from the projection, never from the event store
    row = read_model.fetchone(
        "SELECT * FROM account_summary WHERE account_id = ?", (id,)
    )
    return dict(row)
```

The advantages of this separation:

- **Independent scaling.** The write path scales based on command volume; the read path scales based on query volume. A read-heavy application can run 100 read replicas and 3 write replicas.
- **Independent failure modes.** If the read model database crashes, writes continue to flow to the event store. When the read model recovers, it rebuilds from the event store. If the event store is temporarily unavailable, reads continue from the read model cache.
- **Technology diversity.** The command side uses an event-optimized store (EventStoreDB, Kafka). Each read model can use the best technology for its purpose: PostgreSQL for structured queries, Elasticsearch for full-text search, Redis for caching, a data warehouse for analytics.

### Stream Processing

In a distributed architecture, events are published to a stream processing platform (typically Apache Kafka, Pulsar, or Kinesis). The stream processor guarantees at-least-once delivery and allows multiple projection services to consume the same events independently.

## Integration Patterns

### Relational Database for Projections

The most common read model is a relational database with tables optimized for specific queries. Each table is maintained by one or more projections.

```sql
-- Balance summary table (kept up-to-date by balance projection)
CREATE TABLE account_summary (
    account_id      VARCHAR(255) PRIMARY KEY,
    owner_name      VARCHAR(255) NOT NULL,
    balance         DECIMAL(15,2) NOT NULL DEFAULT 0,
    currency        VARCHAR(3) NOT NULL DEFAULT 'USD',
    status          VARCHAR(20) NOT NULL DEFAULT 'active',
    last_activity   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    version         BIGINT NOT NULL DEFAULT 0
);

-- Daily transaction count (maintained by analytics projection)
CREATE TABLE daily_transaction_counts (
    account_id      VARCHAR(255) NOT NULL,
    date            DATE NOT NULL,
    deposit_count   INT NOT NULL DEFAULT 0,
    withdrawal_count INT NOT NULL DEFAULT 0,
    total_deposits  DECIMAL(15,2) NOT NULL DEFAULT 0,
    total_withdrawals DECIMAL(15,2) NOT NULL DEFAULT 0,
    PRIMARY KEY (account_id, date)
);

-- Active accounts list (maintained by account-status projection,
-- filters out closed accounts)
CREATE VIEW active_accounts AS
SELECT account_id, owner_name, balance
FROM account_summary
WHERE status = 'active';
```

### Cache for Read Models

For high-throughput reads (over 10,000 queries per second), add a cache layer in front of the relational read model. The cache stores the most frequently accessed read models.

```python
class CachedReadModel:
    def __init__(self, db, cache):
        self.db = db
        self.cache = cache
        self.cache_ttl = 60  # seconds

    def get_account(self, account_id: str) -> dict:
        # Try cache first
        cached = self.cache.get(f"account:{account_id}")
        if cached:
            return json.loads(cached)

        # Fall back to database
        row = self.db.fetchone(
            "SELECT * FROM account_summary WHERE account_id = ?",
            (account_id,)
        )
        if row is None:
            return None

        result = dict(row)
        # Populate cache
        self.cache.setex(
            f"account:{account_id}",
            self.cache_ttl,
            json.dumps(result)
        )
        return result

    def invalidate(self, account_id: str):
        # Called by projection when new events arrive
        self.cache.delete(f"account:{account_id}")
```

### Event Store as Message Queue

The event store can serve double duty as a message queue for internal service communication. When events are stored, they are published to subscribers. This eliminates the need for a separate message broker in smaller deployments.

```python
class EventBus:
    """In-process event bus for single-node deployments."""

    def __init__(self):
        self.subscribers: dict[str, list[Callable]] = {}

    def subscribe(self, event_type: str, handler: Callable):
        if event_type not in self.subscribers:
            self.subscribers[event_type] = []
        self.subscribers[event_type].append(handler)

    def publish(self, event: EventRecord):
        handlers = self.subscribers.get(event.event_type, [])
        for handler in handlers:
            handler(event)

# Usage
event_bus = EventBus()
event_bus.subscribe("FundsDebited", balance_projection.handle_debit)
event_bus.subscribe("FundsCredited", balance_projection.handle_credit)
event_bus.subscribe("AccountOpened", account_summary_projection.handle_open)
```

For larger deployments, replace the in-process bus with a Kafka topic or RabbitMQ exchange.

## Multi-Region Architecture

When your application spans multiple geographic regions, event replication becomes critical. You need to ensure that events from region A are eventually available in region B, and that global aggregates (entities accessed from multiple regions) maintain consistency.

### Replication Strategy

The recommended replication strategy for multi-region event sourcing:

1. **Writes go to the local region.** A client in Europe writes to the EU event store. The write latency is low (under 10 ms) because the request does not cross continents.

2. **Events replicate asynchronously.** Events from the EU event store are replicated to the US event store (and vice versa) using a log-based replication mechanism. The replication lag is typically 1-5 seconds between regions.

3. **Each region has independent projections.** Projections in each region process events as they arrive, building region-specific read models. Event ordering is preserved within each stream.

4. **Global aggregates use a single writer.** If an aggregate is global (accessed from multiple regions), designate a single region as the writer for that aggregate. All commands for that aggregate must be routed to the writer region. This avoids distributed consistency problems.

```yaml
multi_region:
  # Replication configuration
  replication:
    # Source and target for event log replication
    regions:
      - name: "us-east"
        event_store_url: "esdb://useast-cluster:2113"
        is_writer_for: ["acct:*", "order:*"]  # All accounts and orders

      - name: "eu-west"
        event_store_url: "esdb://euwest-cluster:2113"
        is_writer_for: ["eu-account:*"]  # Only EU-based accounts

    # Replication channel
    channel:
      type: "kafka"  # or "log-shipping", "grpc-stream"
      topic: "global-event-replication"
      # Each region produces to and consumes from this topic

    # Replication lag alert
    max_replication_lag_seconds: 10
```

### Global Aggregates

Global aggregates — entities that can be accessed from any region — require special handling. The key insight is that you can designate a single region as the authoritative writer for each aggregate.

```python
def route_command(aggregate_id: str, command: dict) -> str:
    """Route a command to the correct region based on aggregate ID."""

    # Simple routing: hash the aggregate ID to a region
    region_count = len(REGIONS)
    region_index = hash(aggregate_id) % region_count
    region = REGIONS[region_index]

    return region
```

This approach works well when the routing is stable (an aggregate always routes to the same region) and when the number of regions is small (2-3). For systems with many regions, use a more sophisticated routing strategy such as consistent hashing.

### Conflict Resolution

In multi-region setups, conflicts can arise if two regions modify the same aggregate concurrently. Event sourcing's optimistic concurrency control handles this naturally for the writer region. But what about read replicas?

The rule is simple: **the writer region always wins.** When events replicate from region A (writer) to region B (reader), the replicated events have higher authority than any local modifications in region B. If region B's event store contains events for an aggregate it should not have written to, those events are marked as divergent and the aggregate is rebuilt from the authoritative copy.

In practice, this situation should not arise if you correctly route commands to the designated writer region. The routing table must be enforced at the API gateway level.

## Capacity Planning

### Write Capacity

The write capacity of an event store depends on the number of partitions and the hardware of each partition.

| Component | EventStoreDB | Kafka | PostgreSQL |
|-----------|-------------|-------|------------|
| Max writes/sec (single node) | 15,000 | 50,000 | 5,000 |
| Max writes/sec (3-node cluster) | 45,000 | 150,000 | 15,000 |
| Storage per day (1 KB events, 10,000/s) | 864 GB | 864 GB | 864 GB |
| Recovery time (1 TB, single node) | 30 min | 20 min | 60 min |

### Read Capacity

Read capacity is determined by your read model infrastructure, not the event store. Each read model is independently scalable.

| Read Model | Queries/sec (single node) | Latency p99 | Best For |
|-----------|--------------------------|-------------|----------|
| PostgreSQL (indexed) | 10,000 | 5 ms | Structured queries |
| Redis (cached) | 100,000 | 1 ms | Key-value lookups |
| Elasticsearch | 5,000 | 20 ms | Full-text search |
| Read replica PostgreSQL | 10,000 | 10 ms | Reporting |

### Projection Build Time

The time to rebuild a projection from scratch depends on the total number of events and the processing throughput of the projection.

```python
def estimate_rebuild_time(
    total_events: int,
    events_per_second: int,
    batch_size: int
) -> tuple[float, str]:
    """Estimate projection rebuild time."""

    seconds = total_events / events_per_second
    batches = total_events / batch_size
    network_overhead = batches * 0.01  # 10ms per batch

    total_seconds = seconds + network_overhead

    if total_seconds < 60:
        return total_seconds, f"{total_seconds:.0f} seconds"
    elif total_seconds < 3600:
        return total_seconds, f"{total_seconds / 60:.1f} minutes"
    else:
        return total_seconds, f"{total_seconds / 3600:.1f} hours"
```

| Total Events | Events/sec (single projection) | Rebuild Time |
|-------------|-------------------------------|-------------|
| 1,000,000 | 5,000 | 200 seconds |
| 10,000,000 | 5,000 | 33 minutes |
| 100,000,000 | 5,000 | 5.5 hours |
| 1,000,000,000 | 50,000 (parallel) | 5.5 hours |

For very large event stores (over 100 million events), parallelize the rebuild by partitioning events by stream or time range.

```python
def rebuild_projection_parallel(
    projection: Projection,
    worker_count: int = 4
):
    """Rebuild a projection in parallel across multiple workers."""

    # Each worker processes a range of global positions
    total_events = event_store.get_global_position()
    chunk_size = total_events // worker_count

    def worker(chunk_start: int, chunk_end: int):
        for event in event_store.read_range(chunk_start, chunk_end):
            projection.handle(event)

    with ThreadPoolExecutor(max_workers=worker_count) as executor:
        futures = []
        for i in range(worker_count):
            start = i * chunk_size
            end = (i + 1) * chunk_size if i < worker_count - 1 else total_events
            futures.append(executor.submit(worker, start, end))

        for future in futures:
            future.result()
```

## Migration Between Architectures

Moving from a single-node architecture to a distributed architecture requires careful planning. You cannot simply replace the event store — you must migrate events and rebuild projections.

### Migration Path: Single Node to CQRS

1. **Add a stream processing layer.** Introduce Kafka (or a similar platform) as a buffer between the event store and the projections. The event store writes events to Kafka; projections consume from Kafka. This decouples the event store from the projections.

2. **Extract projections to separate services.** Move each projection from in-process code to a standalone service. Each service has its own database for the read model. This allows independent scaling.

3. **Scale the event store.** Replace the single-node PostgreSQL event store with a clustered EventStoreDB or a Kafka-based event store. Use a one-time migration script to copy events from the old store to the new store.

4. **Verify correctness.** Rebuild all projections from scratch using the new event store. Compare the read models with the old read models. Any discrepancy indicates a bug in the migration.

### Migration Path: Single Region to Multi-Region

1. **Add a second region in read-only mode.** Deploy a replica event store in a second region. All reads from the second region go to the replica. Writes still go to the primary region.

2. **Implement event replication.** Set up asynchronous event replication from the primary region to the replica. Initially, use a simple log-based replication (stream the event log over gRPC or Kafka).

3. **Verify read model consistency.** Build projections in the second region using replicated events. Verify that the read models in both regions converge to the same state.

4. **Enable writes in the second region.** For regional aggregates (entities specific to the second region), redirect write traffic to the local event store. Route global aggregate writes to the primary region.

5. **Repeat for additional regions.** Each region follows the same pattern: deploy in read-only mode, establish replication, verify consistency, enable writes for regional aggregates.

## Summary

Scaling event sourcing from a single node to a distributed, multi-region architecture requires separating the write path from the read path, introducing stream processing for event distribution, and carefully planning capacity.

| Architecture | Event Volume | Regions | Complexity |
|-------------|-------------|---------|-----------|
| Single node | Under 1,000/s | One | Low |
| CQRS + stream | 1,000 to 50,000/s | One | Medium |
| Distributed + multi-region | Over 50,000/s | Multiple | High |

Choose the architecture that matches your current scale and growth trajectory. Start simple — a single node with PostgreSQL as the event store — and add complexity as your event volume and query patterns demand it. The event store is the foundation: as long as your events are safely stored, you can always rebuild projections and expand into more sophisticated architectures.
