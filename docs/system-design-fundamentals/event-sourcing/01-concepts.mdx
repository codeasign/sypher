## The Problem: Lost History

Imagine you run a payment processing platform. A customer calls to dispute a charge: they claim they never authorized a $450 withdrawal from their account. You open your database and see:

| Field | Value |
|-------|-------|
| account_id | 12345 |
| balance | $850 |
| updated_at | 2026-06-28 14:32:01 |

The balance is $850. That number is the sum of every deposit and withdrawal that ever happened. But you cannot answer the simplest question: *How did the balance become $850?* You have no record of the individual transactions. The current balance is a single number that buries all context.

This is the fundamental weakness of the CRUD (Create-Read-Update-Delete) approach to state management. Every time a record changes, the old state is overwritten. The history is destroyed. When a bug, a dispute, or an audit request arrives, you have nothing to inspect.

**A real example:** In 2018, a major cryptocurrency exchange suffered a $40 million loss from a bug in their withdrawal processing logic. Because they stored only current balances — not the sequence of events that produced them — it took engineers weeks to reconstruct what had happened. The replay of events had to be approximated from server logs that were never designed for that purpose. With event sourcing, the answer would have been a single query: *replay all events for this account and find the first one that produced an incorrect balance.*

The deeper problem is that CRUD treats state as a snapshot — a single point of truth that can be overwritten. In any system where correctness matters (finance, inventory, compliance, audit), overwriting truth is destructive. You need a system that treats every change as permanent and inspectable.

## The Insight: Events as Truth

Event sourcing flips the model. Instead of storing the current state, you store every state-changing event. The current state becomes a derived value — you compute it by replaying the events from the beginning.

Think of a bank statement. Your bank does not just tell you your current balance and erase the past. Every month, you receive a statement listing every deposit, every withdrawal, every fee. The statement is the source of truth. Your current balance is just the result of adding up all those entries.

An event-sourced system works the same way:

- **Events are facts.** Once recorded, they are never modified or deleted.
- **State is derived.** The current state is the result of replaying all events in order.
- **Time travel is free.** You can replay events up to any point in time to see what the state looked like then.
- **Audit is built-in.** Every event carries a timestamp, a type, and the data that changed. There is no separate audit log to maintain.

## Core Vocabulary

Event sourcing uses a specific vocabulary. Each term describes a distinct responsibility in the system.

**Event.** A record of something that happened in the past. Events are facts — they are immutable and named in the past tense: `OrderPlaced`, `PaymentReceived`, `InventoryReserved`. Each event carries the data relevant to that occurrence. An event is not a command (a request to do something) and not a state (a snapshot of the system). It is a fact that something happened.

**Event Store.** A database designed specifically for storing events. Unlike a traditional database, an event store only supports append operations — you can add new events but you cannot modify or delete existing ones. The event store is the system of record. Popular event stores include EventStoreDB, Apache Kafka (used as an event store), and purpose-built solutions on top of PostgreSQL or MySQL.

**Aggregate.** A cluster of related data treated as a single unit for consistency. In domain-driven design terms, an aggregate is a consistency boundary. For example, a `BankAccount` aggregate contains the account balance, owner information, and transaction history. All operations on the account must go through the aggregate, which validates the operation and emits events. The aggregate enforces invariants — for example, that a withdrawal cannot exceed the current balance.

**Snapshot.** A point-in-time record of an aggregate's state, stored alongside the event stream. Snapshots speed up replay: instead of replaying 1,000,000 events to get the current state, you load the most recent snapshot and replay only the events after it. Snapshots are purely a performance optimization — they do not affect correctness because you can always replay from the beginning.

**Replay.** The process of reading events in order and applying them to rebuild state. Replay is the fundamental operation of event sourcing. You can replay all events (full rebuild), events for a specific aggregate (targeted rebuild), or events up to a specific timestamp (time travel). Replay must be deterministic: given the same events in the same order, you must always get the same resulting state.

**Projection.** A function that reads events from the event store and produces a read-optimized view. Projections transform the event stream into a form that serves specific queries efficiently. For example, a projection might build a table of all active customer accounts and their balances, updated as new events arrive. Projections are also called "automata" or "processors" in different event sourcing frameworks.

**Read Model.** The output of a projection — a data structure optimized for queries rather than writes. Read models can be stored in a relational database, a cache, a search index, or any other query-optimized store. The read model is eventually consistent with the event store: there may be a small delay between an event being stored and the read model reflecting it.

**Event Stream.** The ordered sequence of events for a single aggregate or entity. Each stream has a unique identifier (the aggregate ID) and events are appended with monotonically increasing sequence numbers. Streams are the unit of consistency: events within a stream are totally ordered; events across different streams have no implicit ordering.

**Event Version.** A number attached to each event that identifies which version of the aggregate produced it. Event versions enable optimistic concurrency control: when two commands try to modify the same aggregate simultaneously, the version check ensures only one succeeds. The version also helps during replay — you can verify that no events were missed or duplicated.

**Tombstone Event.** An event that marks the deletion of an aggregate. Even when an entity is "deleted" from the system, event sourcing does not physically remove the event stream. Instead, you append a tombstone event (such as `AccountClosed` or `UserDeleted`) and then filter it out in projections. This preserves the full history while logically removing the entity from active queries.

**Upcasting.** The process of transforming old-format events to new-format events during replay. As your system evolves, event schemas change. A field might be renamed, a new field added, or an event split into multiple events. Upcasting transforms historical events on-the-fly during replay so that the projection code always sees the current schema. This is analogous to database migrations, but applied to individual events rather than table schemas.

**Event Migration.** A batch process that rewrites historical events to a new schema in the event store. Unlike upcasting (which transforms at read time), migration transforms at write time — it reads old events, converts them, and writes the new versions. Migration is typically done to improve read performance or to simplify upcasting logic. It is a delicate operation because you must ensure no events are lost and the resulting state is identical.

### Step 3: Validation

The aggregate checks that the command can be executed without violating invariants.

```python
class BankAccount:
    def __init__(self):
        self.balance = 0
        self.version = 0
        self.is_closed = False
        self.pending_holds = 0

    def apply_transfer_funds(self, command: TransferFunds):
        if self.is_closed:
            raise ValueError("Account is closed")
        available = self.balance - self.pending_holds
        if command.amount > available:
            raise ValueError(
                f"Insufficient funds: {available} available, "
                f"{command.amount} requested"
            )
        # Validation passed — emit events
        return [
            FundsDebited(
                amount=command.amount,
                destination=command.to_account,
                remaining_balance=self.balance - command.amount
            ),
            FundsCredited(
                amount=command.amount,
                source=command.from_account,
                new_balance=0  # Account B balance unknown here
            )
        ]
```

### Step 7: Projection Updates Read Model

The `BalanceProjection` updates the read model table that stores current balances.

```sql
-- Read model table: account_balances
-- Before the transfer:
-- | account_id | balance | last_updated           |
-- | A          | 700.00  | 2026-07-01 15:30:00    |
-- | B          | 300.00  | 2026-07-01 16:45:00    |

-- BalanceProjection processes FundsDebited(A, $200):
UPDATE account_balances
SET balance = balance - 200,
    last_updated = '2026-07-02T10:00:00Z'
WHERE account_id = 'A';
-- Result: A balance = $500

-- BalanceProjection processes FundsCredited(B, $200):
UPDATE account_balances
SET balance = balance + 200,
    last_updated = '2026-07-02T10:00:00Z'
WHERE account_id = 'B';
-- Result: B balance = $500

-- Final read model:
-- | account_id | balance | last_updated           |
-- | A          | 500.00  | 2026-07-02 10:00:00    |
-- | B          | 500.00  | 2026-07-02 10:00:00    |
```

### Step 8: Query Returns Current State

A client queries the read model for the current balance. The read model returns the balance immediately with no event replay needed.

```python
@app.get("/accounts/{account_id}/balance")
def get_balance(account_id: str):
    # Read from the projection (not the event store)
    row = db.execute(
        "SELECT balance FROM account_balances WHERE account_id = ?",
        (account_id,)
    ).fetchone()
    return {"account_id": account_id, "balance": row.balance}
```

### Step 9: Audit — Trace Every Event

When the customer disputes the $200 transfer, you query the event store directly to see every event for account A.

```python
@app.get("/accounts/{account_id}/events")
def get_events(account_id: str):
    events = event_store.read_stream(f"acct:{account_id}")
    return [
        {
            "seq": e.sequence_number,
            "type": e.event_type,
            "data": e.data,
            "timestamp": e.timestamp
        }
        for e in events
    ]
```

This returns the complete history of every change to the account, in order, with no gaps.

### Step 10: Time Travel — Replay to Any Point

If a bug was introduced on June 30th, you can replay all events up to June 29th to see what the state looked like before the bug.

```python
def get_state_at(account_id: str, at_timestamp: datetime):
    events = event_store.read_stream(f"acct:{account_id}")
    account = BankAccount()
    for event in events:
        if event.timestamp > at_timestamp:
            break
        account.apply(event)
    return account
```

This gives you the exact state of the account at any moment in history. No restore from backup. No approximation. Exact deterministic replay from the event record.

## Variants of Event Sourcing

Event sourcing is not a single approach. There are several variants, each with different trade-offs.

### Pure Event Sourcing

In pure event sourcing, the event store is the only source of truth. There is no current-state cache, no snapshot store, no read model that is treated as authoritative. Every time you need the state of an aggregate, you replay all events from the beginning.

This is the simplest variant to reason about because there is no secondary state that could become inconsistent. However, it has poor performance for aggregates with long event streams — replaying 1,000,000 events to answer a single query is slow.

**Use when:** The number of events per aggregate is small (under 1,000) and correctness must be absolute.

### Event Sourcing with Snapshots

This variant adds periodic snapshots of aggregate state. During replay, you load the most recent snapshot and then replay only the events after it. This dramatically reduces replay time for aggregates with long event streams.

The snapshot is stored alongside the event stream. It is never treated as authoritative — if a snapshot becomes corrupted or outdated, you ignore it and replay from an earlier snapshot or from the beginning.

**Use when:** Aggregates can accumulate tens of thousands of events and query latency matters.

```python
class SnapshotRepository:
    def load_aggregate(self, aggregate_id: str) -> BankAccount:
        snapshot = self.snapshot_store.get(aggregate_id)
        if snapshot:
            account = snapshot.state  # Deserialized aggregate state
            # Replay only events after the snapshot
            events = self.event_store.read_stream(
                aggregate_id,
                from_version=snapshot.version + 1
            )
        else:
            account = BankAccount()
            events = self.event_store.read_stream(aggregate_id)

        for event in events:
            account.apply(event)
        return account

    def save_snapshot(self, aggregate: BankAccount):
        # Save snapshot every 100 events
        if aggregate.version % 100 == 0:
            self.snapshot_store.put(
                aggregate.id,
                Snapshot(
                    version=aggregate.version,
                    state=aggregate.serialize()
                )
            )
```

### Hybrid: Event Sourcing with Current-State Cache

This variant adds a current-state cache (typically in a relational database or Redis) that serves as the read-optimized view. All writes go through the event store, but reads come from the cache. This is the most common production variant.

The cache is built and maintained by projections. When events are committed, projections update the cache. If the cache is ever lost (crash, corruption), it can be rebuilt entirely from the event store.

**Use when:** You need low-latency reads (under 10 ms) and high read throughput (tens of thousands of queries per second).

## Mental Model Check

Answer these questions to verify you understand event sourcing.

**Question 1:** Your event-sourced system has stored 500,000 events for a single aggregate. A projection is rebuilt from scratch. How many events must be replayed?

<details>
<summary>Answer</summary>

All 500,000 events must be replayed. However, if you use snapshots, you would load the most recent snapshot (say, at version 493,700) and replay only the 6,300 events after it. If you do not use snapshots and need to improve rebuild time, you would add a snapshot strategy.
</details>

**Question 2:** A bug in version 2.5 of your software caused incorrect `Withdrawal` events to be stored. You fixed the bug in version 2.6. What do you do about the incorrect events already in the event store?

<details>
<summary>Answer</summary>

You never modify or delete events in the event store. Instead, you have two options:
1. **Compensation event:** Append a `WithdrawalCorrected` event that reverses the incorrect withdrawal.
2. **Upcast + rebuild:** Write an upcaster that transforms the incorrect events during replay, then rebuild all projections. If the events are fundamentally wrong, compensation events are usually the cleaner approach because they leave the audit trail intact.
</details>

**Question 3:** Your read model database goes down and loses all data. How do you recover?

<details>
<summary>Answer</summary>

You rebuild the read model from the event store. Reset all projections to the beginning of the event stream and replay every event. This is a standard operational procedure for event-sourced systems. The read model is always disposable — the event store is the source of truth. During the rebuild, you might serve degraded responses or redirect traffic to a secondary read model.
</details>

**Question 4:** Two customers each send a `Withdraw($100)` command for the same account at exactly the same time. How does event sourcing prevent a race condition that could allow the account to go negative?

<details>
<summary>Answer</summary>

Event sourcing uses optimistic concurrency control. Each event has a version number. When storing events, the event store checks that the expected version matches the current stream version. Only the first command to append succeeds. The second command receives a concurrency error and must retry — it will reload the aggregate (seeing the first withdrawal), re-validate (the balance is now lower), and decide whether the second withdrawal is still valid. This is identical to optimistic locking in traditional databases.
</details>

**Question 5:** You need to add a new query: "Show me all accounts that have not had any activity in the last 90 days." How do you implement this without replaying every event?

<details>
<summary>Answer</summary>

You create a new projection that subscribes to all account events and maintains a read model of `last_activity_date` per account. Initially, you backfill it by replaying all historical events. After that, the projection updates incrementally as new events arrive. The query then reads from this projection — a simple lookup rather than a full stream scan.
</details>

## Summary

Event sourcing stores every state change as an immutable event in an append-only log. The current state is derived by replaying events. This provides built-in audit, time travel, and full history reconstruction that CRUD systems cannot match.

Key takeaways:

- Events are facts — they are stored, never modified, and never deleted.
- State is derived — you compute it by replaying events in order.
- The event store is the source of truth; everything else is a derived view.
- Snapshots optimize replay performance without sacrificing correctness.
- Projections build read-optimized views from the event stream.
- Event sourcing pairs naturally with CQRS (Command Query Responsibility Segregation), which is covered in the next topic.

The cost is operational complexity: you need an event store, a projection infrastructure, and schema migration tooling. For systems where correctness, auditability, and historical analysis matter, this cost is justified.
