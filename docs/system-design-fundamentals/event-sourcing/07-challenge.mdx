## Tier 1: Foundational

### Challenge 1: Event Store Schema

Design the database schema for an event store that will handle events from an e-commerce system. The store must support:

- Storing events of different types with different data payloads.
- Reading all events for a specific order in order.
- Reading all events of a specific type (for example, all `OrderShipped` events).
- Optimistic concurrency control (expected version check).
- Global ordering of all events.

**What to do:** Write the SQL schema (or the data model for a NoSQL store), explaining your design decisions.

<details>
<summary>Model Answer</summary>

```sql
CREATE TABLE events (
    -- Global ordering
    global_position BIGSERIAL PRIMARY KEY,

    -- Stream identification
    stream_type     VARCHAR(50) NOT NULL,  -- "order", "payment", etc.
    stream_id       VARCHAR(255) NOT NULL,  -- Aggregate ID
    stream_version  INT NOT NULL,           -- Version within stream

    -- Event metadata
    event_type      VARCHAR(255) NOT NULL,  -- "OrderPlaced", "OrderShipped"
    event_version   INT NOT NULL DEFAULT 1, -- Schema version of this event

    -- Payload
    data            JSONB NOT NULL,         -- Event-specific data
    metadata        JSONB NOT NULL DEFAULT '{}', -- correlation_id, causation_id

    -- Timing
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- Uniqueness constraint for optimistic concurrency
    UNIQUE(stream_type, stream_id, stream_version)
);

-- Indexes
CREATE INDEX idx_events_stream
    ON events(stream_type, stream_id, stream_version);

CREATE INDEX idx_events_type
    ON events(event_type, global_position);

CREATE INDEX idx_events_created
    ON events(created_at);
```

**Design decisions:**

- `stream_type + stream_id` identifies the aggregate stream. Separating type from ID allows efficient filtering by aggregate type.
- `stream_version` enables optimistic concurrency: when appending, check that the current max version for the stream matches the expected version.
- `global_position` is a BIGSERIAL, giving a monotonically increasing global order. This is used by projections to track their position.
- `data` is JSONB, allowing flexible event schemas without schema changes. The `event_version` field tracks schema version for upcasting.
- The unique constraint on `(stream_type, stream_id, stream_version)` prevents duplicate appends and enables version checking.
</details>

### Challenge 2: Transfer Aggregate

Implement the `process_transfer` method on a `BankAccount` aggregate. The method takes a `TransferFunds` command and returns a list of events. It must enforce:

- The source account must have sufficient funds.
- The transfer amount must be positive.
- The source account must not be frozen or closed.
- The destination account must exist.
- The method must not modify the aggregate state directly — it only validates and emits events.

**What to do:** Write the Python code for the `process_transfer` method.

<details>
<summary>Model Answer</summary>

```python
from dataclasses import dataclass, field
from enum import Enum
from typing import Optional


class AccountStatus(Enum):
    ACTIVE = "active"
    FROZEN = "frozen"
    CLOSED = "closed"


@dataclass
class TransferFunds:
    from_account: str
    to_account: str
    amount: float
    correlation_id: str


@dataclass
class FundsDebited:
    account: str
    amount: float
    destination: str
    remaining_balance: float
    correlation_id: str


@dataclass
class FundsCredited:
    account: str
    amount: float
    source: str
    correlation_id: str


@dataclass
class TransferFailed:
    account: str
    reason: str
    amount: float
    correlation_id: str


class BankAccount:
    def __init__(self):
        self.account_id: str = ""
        self.balance: float = 0.0
        self.status: AccountStatus = AccountStatus.ACTIVE
        self.version: int = 0

    def process_transfer(self, command: TransferFunds) -> list:
        """
        Validate the transfer command and emit events.
        Does not modify aggregate state — only emits events.
        """
        events = []

        # Validate account status
        if self.status == AccountStatus.CLOSED:
            events.append(TransferFailed(
                account=self.account_id,
                reason="Account is closed",
                amount=command.amount,
                correlation_id=command.correlation_id
            ))
            return events

        if self.status == AccountStatus.FROZEN:
            events.append(TransferFailed(
                account=self.account_id,
                reason="Account is frozen",
                amount=command.amount,
                correlation_id=command.correlation_id
            ))
            return events

        # Validate amount
        if command.amount <= 0:
            events.append(TransferFailed(
                account=self.account_id,
                reason="Transfer amount must be positive",
                amount=command.amount,
                correlation_id=command.correlation_id
            ))
            return events

        # Validate sufficient funds
        if command.amount > self.balance:
            events.append(TransferFailed(
                account=self.account_id,
                reason=(
                    f"Insufficient funds: "
                    f"{self.balance} available, {command.amount} requested"
                ),
                amount=command.amount,
                correlation_id=command.correlation_id
            ))
            return events

        # All validations passed — emit success events
        events.append(FundsDebited(
            account=self.account_id,
            amount=command.amount,
            destination=command.to_account,
            remaining_balance=self.balance - command.amount,
            correlation_id=command.correlation_id
        ))

        events.append(FundsCredited(
            account=command.to_account,
            amount=command.amount,
            source=self.account_id,
            correlation_id=command.correlation_id
        ))

        return events
```

**Key design points:**

- The method returns events; it does not apply them. Applying events is the responsibility of the `apply()` method used during replay.
- Validation failures produce `TransferFailed` events rather than throwing exceptions. This preserves the audit trail — you can see that a transfer was attempted and rejected.
- The source account emits `FundsDebited` and `FundsCredited` includes the destination. The destination account processes the `FundsCredited` event independently.
</details>

### Challenge 3: Snapshot Strategy

You have an inventory management system with 100,000 SKUs. Each SKU aggregate receives approximately 50 events per day (stock additions, sales, adjustments, returns). The aggregates are loaded frequently (every time a customer views a product page or adds to cart). After 1 year of operation, each SKU has approximately 18,250 events.

Without snapshots, loading an SKU aggregate requires replaying 18,250 events, taking approximately 900 ms. The target load time is under 20 ms.

**What to do:** Design a snapshot strategy. Calculate the optimal snapshot interval and estimate the storage overhead.

<details>
<summary>Model Answer</summary>

**Step 1: Determine events per load.** Each SKU processes 50 events/day. After 1 year: 50 x 365 = 18,250 events per SKU. Target replay time is 20 ms. Measured time per event is 900 ms / 18,250 = 0.049 ms per event.

**Step 2: Target events per load.** 20 ms / 0.049 ms per event = 408 events. Round to 400.

**Step 3: Snapshot interval.** With an interval of 400 events, the maximum events to replay after a snapshot is 399. This achieves the 20 ms target.

**Step 4: Storage overhead.** Each snapshot stores the complete aggregate state (approximately 200 bytes for an SKU). Snapshots are taken every 400 events. Each SKU has 18,250 events / 400 = 45.6 snapshots per year. 100,000 SKUs x 46 snapshots x 200 bytes = 920 MB per year for snapshots. This is negligible compared to the event storage (100,000 x 18,250 x 200 bytes = 365 GB per year for events).

**Step 5: Implementation.**

```python
SNAPSHOT_INTERVAL = 400  # Take snapshot every 400 events

def load_sku(sku_id: str) -> SkuAggregate:
    snapshot = snapshot_store.get(sku_id)
    if snapshot:
        sku = SkuAggregate.deserialize(snapshot.state)
        from_version = snapshot.sku_version + 1
    else:
        sku = SkuAggregate(sku_id)
        from_version = 0

    events = event_store.read_stream(
        f"sku:{sku_id}",
        from_version=from_version
    )
    for event in events:
        sku.apply(event)

    return sku

def save_snapshot(sku: SkuAggregate):
    if sku.version % SNAPSHOT_INTERVAL == 0:
        snapshot_store.put(
            sku.sku_id,
            Snapshot(
                sku_version=sku.version,
                state=sku.serialize()
            )
        )
```

**Result:** Load time drops from 900 ms to under 20 ms. Snapshot storage adds less than 1 GB per year.
</details>

### Challenge 4: Order Lifecycle with Saga

An order in your e-commerce system requires a saga (a long-running process across multiple aggregates). When an order is placed:

1. Reserve inventory (Inventory aggregate).
2. Process payment (Payment aggregate).
3. Ship order (Shipping aggregate).
4. Update order status (Order aggregate).

If any step fails, compensating events must undo the previous steps.

**What to do:** Design the events, saga coordinator, and compensation logic. Write the saga coordinator code.

<details>
<summary>Model Answer</summary>

**Events for each step:**

```python
# Step 1: Reserve inventory
@dataclass
class InventoryReserved:
    order_id: str
    sku: str
    quantity: int

@dataclass
class InventoryReservationFailed:
    order_id: str
    sku: str
    quantity: int
    reason: str

# Compensation for step 1
@dataclass
class InventoryReservationReleased:
    order_id: str
    sku: str
    quantity: int

# Step 2: Process payment
@dataclass
class PaymentProcessed:
    order_id: str
    amount: float
    transaction_id: str

@dataclass
class PaymentFailed:
    order_id: str
    amount: float
    reason: str

# Compensation for step 2
@dataclass
class PaymentRefunded:
    order_id: str
    transaction_id: str
    amount: float

# Step 3: Ship order
@dataclass
class OrderShipped:
    order_id: str
    tracking_number: str
```

**Saga coordinator:**

```python
class OrderSagaCoordinator:
    """
    Coordinates the order placement saga across multiple aggregates.
    If any step fails, it issues compensating events.
    """

    def __init__(self, event_store, event_bus):
        self.event_store = event_store
        self.event_bus = event_bus

    def handle(self, command: PlaceOrder):
        """Start the order saga."""
        saga_id = f"saga:{command.order_id}"
        saga_state = SagaState(order_id=command.order_id)

        # Step 1: Reserve inventory
        self.event_bus.publish(ReserveInventory(
            order_id=command.order_id,
            items=command.items
        ))

        # Wait for response (via event subscription)
        # The saga continues in on_event()

    def on_event(self, event):
        """Handle events from saga steps."""
        saga_id = f"saga:{event.order_id}"
        saga_state = self._load_saga(saga_id)

        if isinstance(event, InventoryReserved):
            saga_state.step = "reserved"
            # Step 2: Process payment
            self.event_bus.publish(ProcessPayment(
                order_id=event.order_id,
                amount=saga_state.total_amount
            ))

        elif isinstance(event, InventoryReservationFailed):
            # Saga fails here — no compensation needed
            saga_state.status = "failed"
            self.event_bus.publish(FailOrder(
                order_id=event.order_id,
                reason=event.reason
            ))

        elif isinstance(event, PaymentProcessed):
            saga_state.step = "paid"
            saga_state.transaction_id = event.transaction_id
            # Step 3: Ship order
            self.event_bus.publish(ShipOrder(
                order_id=event.order_id,
                shipping_address=saga_state.shipping_address
            ))

        elif isinstance(event, PaymentFailed):
            # Compensate: release inventory
            saga_state.status = "failed"
            saga_state.failure_reason = event.reason
            self.event_bus.publish(ReleaseInventory(
                order_id=event.order_id,
                items=saga_state.items
            ))
            self.event_bus.publish(FailOrder(
                order_id=event.order_id,
                reason=event.reason
            ))

        elif isinstance(event, OrderShipped):
            saga_state.step = "shipped"
            saga_state.tracking_number = event.tracking_number
            saga_state.status = "completed"

        elif isinstance(event, InventoryReservationReleased):
            # Compensation complete
            saga_state.compensation_complete = True

        self._save_saga(saga_id, saga_state)
```

**Key points:**

- The saga is itself event-sourced. Its state is stored as events in the event store.
- Each step emits events on success and failure. Failure events trigger compensation.
- Compensation events are also stored as immutable facts, preserving the full saga trail.
- The saga coordinator is idempotent: if it receives the same event twice, it does not re-execute the step.
</details>

### Challenge 5: Multi-Region Event Replication

Your event-sourced system operates in three regions (US, EU, APAC). Each region has its own event store. Some aggregates are regional (only used in one region). Others are global (accessed from multiple regions).

**What to do:** Design the replication strategy. Describe how events flow between regions. Address conflict resolution.

<details>
<summary>Model Answer</summary>

**Replication strategy:**

1. **Regional aggregates** (for example, "eu-customer:123"). Writes happen in the local region only. Events are replicated to all other regions asynchronously for read model building. No conflicts because only one region writes.

2. **Global aggregates** (for example, "product:SKU-001"). A single region is designated as the writer for each global aggregate. Use consistent hashing to assign aggregates to regions. All commands for that aggregate must be routed to the writer region. The writer region replicates events to other regions.

3. **Replication flow:**
   - Each region has a Kafka topic for outgoing event replication.
   - A replication service consumes the local event store's global log and produces to the local outgoing topic.
   - Cross-region replication (Kafka MirrorMaker or custom consumer) copies events from region A's outgoing topic to region B's incoming topic.
   - Each region's event store subscribes to its incoming topic and appends replicated events.

4. **Conflict resolution:** There are no conflicts because writes for a given aggregate happen in only one region. If a stale write somehow arrives (for example, from a misconfigured client), the event store checks the stream version and rejects the write if it does not match.

5. **Replication lag:** 1-5 seconds typically. The read model in each region lags by this amount for events from other regions. Local events are visible immediately.
</details>

### Challenge 6: Production Incident — Lost Events

Your production monitoring shows that the BalanceProjection has processed 500,000 events, but the event store's global position is at 1,000,000. The projection is 500,000 events behind and the gap is growing at 100 events per second. The projection handler is not reporting errors. What do you do?

**What to do:** Write a step-by-step incident response plan.

<details>
<summary>Model Answer</summary>

**Step 1: Determine the root cause (5 minutes).**
- Check if the projection is crashing silently. Examine the projection service logs for error messages, OOM kill, or crash loops.
- Check if the event store is reachable from the projection service. A network issue could cause the projection to skip events.
- Check the projection's last processed position vs the event store's latest position. Are there gaps in the projection's processed positions?

**Step 2: Stop the bleed (5 minutes).**
- If the projection is crashing, deploy a fix or roll back to the last known good version.
- If the event store is unreachable, fix the network issue or restart the connectivity.
- If the projection is simply too slow, increase the number of projection instances.

**Step 3: Catch up (10 minutes).**
- If the projection is healthy but behind, let it catch up naturally. Monitor the lag to confirm it is decreasing.
- If the lag is growing because the projection cannot process events fast enough, scale projection instances horizontally.
- If the projection has missed events, rebuild it from scratch.

**Step 4: Investigate the 500,000 event gap (ongoing).**
- Check if the projection skipped events (for example, because of a `continue` on error).
- Check if the event store published events that the projection never received.
- Check if a deployment caused the projection to reset its position.

**Step 5: Restore correctness (after investigation).**
- If some events were missed and cannot be reprocessed, rebuild the projection from the event store.
- If events were lost from the event store, restore from backup.

**Preventive measures:**

- Add a dead-letter queue for events that the projection fails to process.
- Add a health check that alerts if projection lag exceeds 30 seconds.
- Implement a position checkpoint after every batch, so the projection can resume from the last checkpoint on restart.
</details>

### Challenge 7: Schema Migration at Scale

Your event store has 2 billion events across 5 years of operation. You need to rename the `OrderPlaced` event type to `OrderSubmitted` and add a `discount_code` field. The old event type has the `items` field as a JSON array; the new format should have `items` as a structured field with product ID and quantity.

You cannot afford downtime. The migration must complete within 24 hours.

**What to do:** Design the zero-downtime migration plan.

<details>
<summary>Model Answer</summary>

**Phase 1: Add new event type (Day 1)**

1. Create the new `OrderSubmitted` event type with the updated schema.
2. Write an upcaster that can transform old `OrderPlaced` events to `OrderSubmitted` during replay.
3. Deploy new projection code that handles both `OrderPlaced` and `OrderSubmitted` events. Old projections that only handle `OrderPlaced` continue to work because the upcaster runs before the projection sees the event.
4. Update the aggregate to emit `OrderSubmitted` instead of `OrderPlaced` for new commands.
5. No downtime so far — both event types coexist.

**Phase 2: Verify correctness (Day 1-2)**

1. Run a shadow rebuild of all projections using the new code. Compare the shadow read model with the live read model.
2. Verify that the upcaster correctly transforms old events. Sampling: pick 10,000 old events, manually verify the transformation.
3. Verify that new events with the `OrderSubmitted` type are correctly handled.

**Phase 3: Migrate historical events (optional — Day 2-3)**

1. If you want to rewrite old events to the new format, run a background migration that reads old `OrderPlaced` events, transforms them, and writes new `OrderSubmitted` events. Write the old events' global position in a migration tracking table so the process is idempotent.
2. This step is optional. If your upcaster is efficient, you can keep old events in their original format and transform on-the-fly forever.

**Phase 4: Phase out old event type (after all projections updated)**

1. Once all projections have been updated to handle only `OrderSubmitted` (or to use the upcaster transparently), you can stop writing the old `OrderPlaced` events.
2. Remove the old event type from the aggregate's emit method.
3. Optionally, delete old events if storage is a concern and you have verified the migration.

**Verification check:**

```python
def verify_migration():
    """Verify that replaying old + new events produces the same state."""

    # Take a sample of 100 orders from the last 5 years
    sample_orders = select_orders_spanning_all_event_types()

    for order_id in sample_orders:

            f"Mismatch for order {order_id}: "
```
