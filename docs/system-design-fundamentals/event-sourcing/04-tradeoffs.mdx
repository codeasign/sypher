## The Core Tension

Event sourcing provides auditability, time travel, and full history reconstruction. These benefits come at a cost: operational complexity, storage growth, and a fundamentally different programming model.

The core tension is between **auditability** (preserving every change) and **complexity** (managing an event store, projections, snapshots, and schema evolution). For every project, you must evaluate whether the benefits justify the costs.

### When the Tension Bites

Consider a team managing a financial trading platform. They use event sourcing for their order book — every trade, every quote, every cancellation is stored as an event. The audit trail is invaluable for regulatory compliance. But the engineering team spends 30% of their time on event-sourcing infrastructure: managing schema migrations, rebuilding projections after deployment failures, monitoring projection lag, and training new team members on the event-sourcing model.

A different team runs a blog commenting system. They use CRUD because the comments have no audit requirement, no complex queries, and no regulatory oversight. Storing every edit as an event would add complexity without benefit.

The decision depends on context, not dogma.

## Decision Matrix

Use this matrix to evaluate whether event sourcing is the right choice for your component.

| Criterion | Event Sourcing | CRUD | Hybrid |
|-----------|---------------|------|--------|
| **Audit requirement** | Must preserve all history | No audit needed | Selective audit (audit log + current state) |
| **Query complexity** | Multiple read models optimized for different queries | Single query interface | Few read models, simple projections |
| **Write throughput** | 5,000-50,000 events/s (event store dependent) | 10,000-100,000 updates/s | Varies (split write path) |
| **Read latency** | 1-10 ms via read model | 1-5 ms direct read | 2-15 ms (via cache + read model) |
| **Schema evolution** | Upcasting, migration tools required | ALTER TABLE, migrations | Migrations for current state, upcasts for events |
| **Storage cost** | High (all events retained) | Low (current state only) | Medium (events + current state cache) |
| **Consistency model** | Eventual (read model may lag) | Strong (single DB) | Tunable (per read model) |
| **Development speed** | Slower initial development | Faster initial development | Moderate |
| **Team expertise** | Requires event sourcing knowledge | Standard SQL/ORM knowledge | Mix of both |
| **Operational tooling** | Event store, snapshots, projections | Standard database | Standard DB + event store |

### Decision Flow

```python
def recommend_approach(
    needs_full_audit: bool,
    needs_time_travel: bool,
    query_complexity: int,  # 1-10
    team_es_experience: int,  # 1-10
    event_volume_per_day: int
) -> str:
    """Recommend event sourcing vs CRUD vs hybrid."""

    es_score = 0

    # Audit and time travel are the strongest signals
    if needs_full_audit:
        es_score += 40
    if needs_time_travel:
        es_score += 20

    # High query complexity favors event sourcing (multiple read models)
    es_score += min(query_complexity * 3, 20)

    # Low team experience is a risk factor
    if team_es_experience < 3:
        es_score -= 15
    elif team_es_experience < 6:
        es_score -= 5

    # Very high event volume favors CRUD (storage cost concern)
    if event_volume_per_day > 100_000_000:
        es_score -= 10
    elif event_volume_per_day > 10_000_000:
        es_score -= 5

    if es_score >= 40:
        return "event_sourcing"
    elif es_score >= 25:
        return "hybrid"
    else:
        return "crud"
```

## CAP Theorem and Event Sourcing

Event sourcing systems are typically AP (Availability and Partition tolerance) systems under the CAP theorem. They prioritize availability and eventual consistency over strong consistency.

- **Availability.** The event store and read models can serve reads and writes even if other components are down. A projection service failure does not prevent event storage.
- **Partition tolerance.** The system continues operating if network partitions occur between the event store and read model services.
- **Eventual consistency.** Read models may lag behind the event store by a small window (typically under 1 second in the same region).

If your application requires strong consistency — reading your own writes, for example — you have two options:

1. **Read from the event store directly** for critical operations that must see the latest state. This sacrifices read performance for consistency.
2. **Use synchronous projections** that block the write response until the read model is updated. This sacrifices write latency for consistency.

```python
# Option 1: Read from event store for strong consistency
@app.post("/withdraw")
def withdraw(request: WithdrawRequest):
    account = load_aggregate(request.account_id)
    events = account.process_withdrawal(request.amount)
    event_store.append(account.stream_id, account.version, events)

    # Return the new balance computed from the aggregate,
    # not from the (possibly stale) read model
    return {"new_balance": account.balance}


# Option 2: Synchronous projection (blocking write)
@app.post("/withdraw-sync")
def withdraw_sync(request: WithdrawRequest):
    account = load_aggregate(request.account_id)
    events = account.process_withdrawal(request.amount)
    event_store.append(account.stream_id, account.version, events)

    # Block until the balance projection has processed this event
    wait_for_projection("balance", account.id, account.version, timeout=5.0)

    # Now the read model is consistent
    return read_model.get_balance(request.account_id)
```

## Five Failure Modes

### Failure Mode 1: Event Schema Evolution Failure

**What happens:** You rename a field in your event schema. Old events still have the old field name. Your projection code, written for the new schema, crashes when it encounters an old event during replay.

**Consequences:** Projection rebuilds fail. New deployments cannot build read models from scratch. Time travel to dates before the schema change is broken.

**Root cause:** No upcasting mechanism. The projection code assumes all events use the current schema.

**Solution:** Implement upcasters that transform old events to the current schema during replay.

```python
# Upcaster registry
class Upcaster:
    @abstractmethod
    def can_upcast(self, event_type: str, event_version: int) -> bool: ...

    @abstractmethod
    def upcast(self, event: EventRecord) -> list[EventRecord]: ...


# Example: v1 Withdrawal event had "dest" field, v2 uses "destination"
class WithdrawalUpcasterV1ToV2(Upcaster):
    def can_upcast(self, event_type: str, event_version: int) -> bool:
        return event_type == "WithdrawalExecuted" and event_version == 1

    def upcast(self, event: EventRecord) -> list[EventRecord]:
        data = json.loads(event.data)
        data["destination"] = data.pop("dest")
        data["_schema_version"] = 2
        return [EventRecord(
            event_type=event.event_type,
            data=json.dumps(data),
            metadata={**event.metadata, "upcasted": True}
        )]
```

**Prevention:** Always include a schema version number in your events. Never deploy projection code that cannot handle at least one schema version back. Run integration tests that replay old events through new upcasters.

### Failure Mode 2: Projection Lag

**What happens:** A read model falls behind the event store. A user queries the read model and sees stale data — an order they just placed does not appear, or a withdrawal they made still shows in the balance.

**Consequences:** User confusion, support tickets, and — for financial applications — regulatory risk if stale data is shown to customers.

**Root cause:** Projection service cannot keep up with the event volume. Common causes: slow database queries, insufficient projection instances, or a single-threaded projection bottleneck.

**Solution:**

1. **Monitor projection lag** as a critical metric (the difference between the latest global event position and the last event processed by the projection).
2. **Scale projection instances** horizontally (each instance handles a partition of events).
3. **Optimize slow projections** — batch database writes, use prepared statements, add indexes.

```python
# Projection lag monitoring
class ProjectionMonitor:
    def __init__(self, projection_name: str):
        self.projection_name = projection_name
        self.last_processed_position = 0

    def record_progress(self, global_position: int):
        self.last_processed_position = global_position

    def get_lag(self) -> int:
        latest = event_store.get_global_position()
        return latest - self.last_processed_position

    def lag_in_seconds(self) -> float:
        lag = self.get_lag()
        # Estimate: measure events per second over last minute
        eps = self.get_events_per_second()
        return lag / eps if eps > 0 else float('inf')
```

**Prevention:** Set a maximum acceptable lag (for example, under 5 seconds). Alert when lag exceeds this threshold. In high-volume systems, build projections that can process events in batches of 100-1,000 at a time.

### Failure Mode 3: Event Store Corruption

**What happens:** A hardware failure, filesystem corruption, or a software bug damages the append-only log. Events are lost, reordered, or the index becomes inconsistent.

**Consequences:** Data loss. If no backup or replica exists, the system's entire history is compromised.

**Root cause:** Single node deployment without replication. No backup strategy. Hard drive failure or software bug.

**Solution:**

1. **Run a cluster.** EventStoreDB and Kafka provide clustered configurations where events are replicated across 3 or more nodes. A single node failure does not cause data loss.
2. **Take regular backups.** Back up the event store at periodic intervals, including the index and log files.
3. **Test recovery.** Periodically practice restoring from backup in a staging environment.

**Prevention:** Always run event store in a clustered configuration for production. The cost of three nodes is justified by the durability guarantee. Never rely on a single node for the event store.

### Failure Mode 4: Snapshot Inconsistency

**What happens:** A snapshot is saved but the event stream is in a different state. A command loads the snapshot, skips replaying events, and operates on incorrect state.

**Consequences:** Data corruption. The aggregate makes decisions based on wrong state.

**Root cause:** Race condition in snapshot saving: the snapshot is saved while events are still being written, or the snapshot state does not match the snapshot version.

**Solution:**

1. **Always replay events after a snapshot.** Never trust the snapshot alone. Load the snapshot, then replay all events with version greater than the snapshot version.
2. **Validate snapshots periodically.** Run a consistency check that compares the aggregate state from replay vs the aggregate state from the snapshot.

```python
def verify_snapshot(aggregate_id: str):
    """Verify that a snapshot is consistent with event replay."""

    # Replay from scratch (no snapshot)
    account_from_scratch = BankAccount()
    all_events = event_store.read_stream(aggregate_id)
    for event in all_events:
        account_from_scratch.apply(event)

    # Replay from snapshot
    snapshot = snapshot_store.load(aggregate_id)
    if snapshot:
        account_from_snapshot = BankAccount.deserialize(snapshot.state)
        remaining_events = event_store.read_stream(
            aggregate_id,
            from_version=snapshot.version + 1
        )
        for event in remaining_events:
            account_from_snapshot.apply(event)

        # Compare
        if account_from_scratch.version != account_from_snapshot.version:
            raise SnapshotInconsistency(
                f"Version mismatch: {account_from_scratch.version}"
                f" vs {account_from_snapshot.version}"
            )
        if account_from_scratch.balance != account_from_snapshot.balance:
            raise SnapshotInconsistency(
                f"Balance mismatch: {account_from_scratch.balance}"
                f" vs {account_from_snapshot.balance}"
            )
```

**Prevention:** The snapshot should be saved in the same transaction as the events, or saved after events are confirmed committed. Use a two-phase approach: save snapshot marker, save snapshot data, then mark as complete.

### Failure Mode 5: Event Flooding

**What happens:** A bug in a client application or a misconfigured automation script sends thousands of commands per second to the same aggregate. The event store accepts them all, creating a flood of events.

**Consequences:** The event store grows rapidly. Projections fall behind due to the spike. Read models become stale. Storage costs increase.

**Root cause:** No rate limiting at the command level. No circuit breaker for runaway clients.

**Solution:**

1. **Rate limit command processing** per aggregate. Allow a maximum number of commands per second per aggregate.
2. **Throttle event storage.** If the event store's write throughput exceeds a threshold, reject new commands with a 429 (Too Many Requests) response.
3. **Detect and blacklist abusive clients.** Monitor command frequency per client ID and temporarily block clients that exceed limits.

```python
class RateLimiter:
    def __init__(self, max_commands_per_second: int):
        self.max_per_second = max_commands_per_second
        self.tokens = max_commands_per_second
        self.last_refill = time.monotonic()

    def check_command(self, aggregate_id: str) -> bool:
        now = time.monotonic()
        elapsed = now - self.last_refill
        self.tokens = min(
            self.max_per_second,
            self.tokens + elapsed * self.max_per_second
        )
        self.last_refill = now

        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False
```

**Prevention:** Always implement rate limiting at the API gateway level. Monitor event volume per aggregate and alert on anomalies. Consider circuit breakers that stop processing commands for an aggregate if the event rate exceeds a threshold.

## Five Anti-Patterns

### Anti-Pattern 1: Storing Events in a Regular Database Table

**Description:** You use a standard SQL table for events but allow UPDATE and DELETE operations. Someone inevitably modifies or deletes an event because "it was a mistake" or "the data was wrong."

**Why it fails:** The immutability guarantee is broken. You cannot trust the event stream. Replay produces different results depending on who modified which events. The audit trail is compromised.

**Fix:** Use an event store that enforces append-only semantics at the storage level. If you must use a standard database, restrict access to the events table through a stored procedure or API that only allows INSERT operations. Grant DELETE and UPDATE privileges to no one, including yourself.

### Anti-Pattern 2: No Schema Versioning

**Description:** Your events have no version field. When you add a field to an event, you simply start writing the new format. Old events still have the old format, and your projection code has no way to distinguish them.

**Why it fails:** When you deploy new projection code, it crashes on old events because they lack the new field. You cannot rebuild projections from scratch. Time travel to dates before the change produces errors.

**Fix:** Include a schema version in every event:

```json
{
  "event_type": "OrderPlaced",
  "schema_version": 3,
  "data": {
    "order_id": "ORD-123",
    "items": [{"sku": "ABC", "qty": 2}]
  }
}
```

Write upcasters that transform version-1 events to version-3 before passing them to projection code. Test upcasters with historical events.

### Anti-Pattern 3: Rebuilding Projections from Production

**Description:** When a projection needs to be rebuilt (due to a code change or data inconsistency), you run the rebuild directly against the production event store and production read model database.

**Why it fails:** A bug in the new projection code can corrupt the production read model. If the rebuild is slow, it degrades production read model performance. If you need to roll back, you have no clean way to restore the old read model.

**Fix:** Always use the shadow rebuild pattern. Build the new projection in a shadow table, verify its correctness against the current projection, and then atomically swap. Run the rebuild against a read replica of the event store, not the primary.

### Anti-Pattern 4: Mutable Events

**Description:** You store events as mutable records. When you discover an error in an event, you fix it in place — updating the event data rather than appending a correction.

**Why it fails:** This destroys the trust in the event store. If events can be changed after the fact, you cannot rely on replay producing correct results. Audit is meaningless because the record can be altered.

**Fix:** Never modify an event. To correct an error, append a new event:

- **Error event:** Append `WithdrawalAmountCorrected {original_event_id: 123, old_amount: 450, new_amount: 400}`.
- **Compensation event:** Append `WithdrawalReversed {original_event_id: 123, reason: "incorrect_amount"}` followed by the correct `WithdrawalExecuted` event.
- **Settlement event:** Append `BalanceSettled {correct_balance: 500}` to snap the state back to correct.

The event store is append-only. Corrections are events, not edits.

### Anti-Pattern 5: Event per Field Change

**Description:** For every single field change in your domain model, you create a separate event type: `NameChanged`, `EmailChanged`, `AddressLine1Changed`, `AddressCityChanged`, `AddressZipChanged`.

**Why it fails:** The event stream becomes incredibly fine-grained. To rebuild a `Customer` aggregate, you must replay 30 events for a simple address update. Projections become complex because they must handle many event types. Querying the event store for debugging requires inspecting dozens of tiny events to understand what happened.

**Fix:** Group related field changes into domain-meaningful events:

```python
# Instead of 6 events for an address update:
# ❌ AddressLine1Changed, AddressLine2Changed,
#    AddressCityChanged, AddressStateChanged,
#    AddressZipChanged, AddressCountryChanged

# Use 1 event:
# ✅ CustomerMoved {
#       customer_id: "CUST-123",
#       old_address: {...},
#       new_address: {...}
#   }
```

Events should represent meaningful domain state changes, not mechanical field updates. A good event type is one that makes sense to a business stakeholder: "Customer moved" is meaningful; "Address line 2 changed" is not.

## When NOT to Use Event Sourcing

### 1. Simple CRUD Applications

If your application is a simple record-keeping system with no audit requirements, no complex queries, and no need for time travel, CRUD is the better choice. Blog comments, todo lists, simple content management — these do not benefit from event sourcing enough to justify the complexity.

### 2. High-Volume IoT Telemetry

If you receive 1,000,000 sensor readings per minute, storing each reading as an individual event in an event store is wasteful. The storage cost is high, and replaying 1,000,000 events to determine the average temperature provides no advantage over a time-series database.

**Better approach:** Use a time-series database (InfluxDB, TimescaleDB) for the raw data. Use event sourcing only for the metadata and configuration changes (device registered, calibration updated, sensor decommissioned).

### 3. Systems with PII in Events

If your events contain personally identifiable information and you must comply with right-to-erasure (GDPR Article 17), event sourcing creates a tension: events are immutable, but PII must be deletable.

**Better approach:** Store only non-PII identifiers in events. Store PII in a separate encrypted store that can be deleted on request. Or use a data model where PII is stored as current state (deletable) and events reference only opaque IDs.

```python
# ❌ PII in events (immutable — cannot be deleted)
event = CustomerRegistered(
    customer_id="CUST-123",
    name="Alice Smith",  # PII — cannot delete from event store
    email="alice@example.com"  # PII — cannot delete from event store
)

# ✅ PII stored separately (deletable)
event = CustomerRegistered(
    customer_id="CUST-123",
    personal_data_ref="pii://alice-smith-uuid"  # Reference, not data
)
# PII stored in encrypted KV store — deletable when requested
pii_store.set("pii://alice-smith-uuid", {
    "name": "Alice Smith",
    "email": "alice@example.com"
})
```

### 4. Systems Where Team Has No Event Sourcing Experience

If your team is not familiar with event sourcing, the learning curve will slow development significantly. The concepts (immutable events, snapshots, projections, upcasting) are not intuitive to developers accustomed to CRUD.

**Better approach:** Start with CRUD for most functionality. Introduce event sourcing for a single bounded context where the benefits are clearest (for example, the order history for a financial application). Let the team learn event sourcing in a controlled scope before expanding.

### 5. Real-Time Systems with Strict Consistency

If your application requires under 5 ms response times with strong consistency guarantees, event sourcing's eventual consistency model may not meet your requirements. The read model lag, even if measured in milliseconds, violates the consistency contract.

**Better approach:** Use a traditional database with synchronous replication. Or use event sourcing for the write side only and implement read-own-writes by reading from the aggregate in memory for the current request.

## Summary

Event sourcing is a powerful pattern, but it is not universally applicable. The auditability and time travel capabilities come with operational costs that must be justified by the domain requirements.

## Cost Analysis Framework

Before adopting event sourcing for a new project, estimate the incremental cost compared to CRUD.

### Development Cost

Event sourcing requires additional development effort in these areas:

- **Aggregate design.** Defining the aggregate boundaries, event types, and projection logic. Expect 2-3x the design time compared to CRUD for the first aggregate.
- **Event store setup.** Deploying and configuring the event store (EventStoreDB, Kafka, or a relational database used as an event store). This is comparable to setting up any database.
- **Projection development.** Building the projections that maintain read models. This is new work that does not exist in CRUD.
- **Tooling.** Event viewer, projection rebuild CLI, consistency checker. These are required but not needed in CRUD.
- **Testing.** Testing event schema evolution (upcasting) requires creating and replaying historical events.
- **Team training.** Team members must learn event sourcing concepts. Budget 2-4 weeks of learning time per developer.

### Operational Cost

- **Infrastructure.** Event store cluster (3 nodes minimum for production), snapshot store, projection services. This typically doubles the infrastructure cost compared to a simple CRUD database.
- **Monitoring.** Projection lag, event store health, snapshot consistency — additional metrics to track.
- **On-call.** Event store failures are harder to diagnose than CRUD database failures. The on-call team needs event sourcing knowledge.
- **Backup/restore.** Event store backups are larger (all events vs current state) and restore takes longer (must replay events to rebuild state).

### ROI Calculation

```python
def estimate_roi(
    audit_requests_per_month: int,
    avg_time_per_audit_hours: float,
    developer_day_rate: float,
    dev_cost_multiplier: float,  # 2-3x for ES vs CRUD
    operational_lift: float      # 1.5-2x for ES vs CRUD
) -> dict:
    """Estimate ROI of event sourcing vs CRUD over 3 years."""

    # CRUD baseline
    crud_dev_cost = 100 * developer_day_rate  # 100 developer days
    crud_ops_cost = 12 * 3 * 1000  # $1,000/month for 3 years
    crud_audit_cost = audit_requests_per_month * 12 * 3 * \
                      avg_time_per_audit_hours * developer_day_rate / 8

    # Event sourcing
    es_dev_cost = crud_dev_cost * dev_cost_multiplier
    es_ops_cost = crud_ops_cost * operational_lift
    es_audit_cost = audit_requests_per_month * 12 * 3 * \
                    0.25 * developer_day_rate / 8  # 15 min per audit

    crud_total = crud_dev_cost + crud_ops_cost + crud_audit_cost
    es_total = es_dev_cost + es_ops_cost + es_audit_cost

    return {
        "crud_3yr_total": crud_total,
        "es_3yr_total": es_total,
        "difference": es_total - crud_total,
        "break_even_audits_per_month": (
            # How many audits/month before ES becomes cheaper
            (es_dev_cost + es_ops_cost - crud_dev_cost - crud_ops_cost) /
            ((crud_audit_cost / (audit_requests_per_month or 1)) -
             (es_audit_cost / (audit_requests_per_month or 1)))
        )
    }


# Example: 50 audit requests/month, 4 hours each for CRUD
result = estimate_roi(
    audit_requests_per_month=50,
    avg_time_per_audit_hours=4,
    developer_day_rate=800,
    dev_cost_multiplier=2.5,
    operational_lift=1.5
)
# Result: Event sourcing breaks even at 8 audit requests/month.
# At 50/month, event sourcing saves $400,000+ over 3 years.
```

The break-even point is surprisingly low. For a system that handles more than 5-10 audit requests per month, event sourcing can be cheaper than CRUD within 2-3 years — the savings from faster audit resolution offset the higher development and operational costs.

## Summary

| Decision | When | Example |
|----------|------|---------|
| Use event sourcing | Finance, compliance, complex domains with history requirements | Banking, accounting, order management |
| Use hybrid | Partial audit needs, moderate complexity | E-commerce with order history |
| Use CRUD | Simple data, no audit, low complexity | CMS, blog, todo app |

The most expensive mistake is adopting event sourcing for every component in your system. Be selective: use it where the benefits clearly outweigh the costs, use CRUD everywhere else.
