## How the Topic Appears

Dashboard design questions appear in system design interviews across all levels. The interviewer is testing your ability to design a data-intensive system that handles high-throughput ingestion, low-latency queries, and a clear mental model of what to visualize. Here are the most common phrasings:

1. "Design a real-time monitoring dashboard for a distributed system."
2. "Design a metrics and monitoring system like Grafana or Datadog."
3. "Design a dashboard that shows the health of 500 microservices."
4. "How would you build a time series database for a dashboard platform?"
5. "Design a system that can display 10,000 metrics in real time."
6. "Design an observability platform that shows metrics, logs, and traces."
7. "How would you build a real-time analytics dashboard for a video streaming service?"
8. "Design a dashboard that can show 30 days of metric data in under 3 seconds."
9. "Design an executive dashboard that shows business and operational metrics."
10. "How would you build a dashboard platform that supports multi-tenant isolation?"

## What the Interviewer Tests

Every dashboard design question tests four specific skills. The interviewer is not looking for a perfect architecture — they are looking for evidence that you understand these four dimensions:

**Data ingestion and pipeline design.** Can you design a system that collects metrics from thousands of services without losing data or overwhelming the services? This tests your knowledge of pull vs push models, scrape intervals, buffering, and backpressure. Strong candidates discuss Kafka or similar message queues as ingestion buffers.

**Storage and query optimization.** Can you design a storage system that handles billions of data points per day and still returns queries in milliseconds? This tests your knowledge of time series storage formats (Gorilla compression), index structures (inverted index), and query acceleration (caching, rollups, materialized views).

**Scalability and fault tolerance.** Can you scale the system from 10 services to 10,000 services? This tests your understanding of horizontal scaling, sharding, replication, and multi-region deployment. Strong candidates discuss the trade-offs between different sharding strategies (hash-based vs range-based) and consistency models.

**Dashboard design principles.** Can you decide what to show on a dashboard and how to lay it out? This tests your understanding of the human side of dashboard design: top-down hierarchy, signal-to-noise ratio, color schemes, and drill-down paths. Interviewers care about this because a technically perfect dashboard platform is useless if the dashboards themselves are unreadable.

## Level-by-Level Expectations

### L3 / Entry Level (0-2 years experience)

At this level, the interviewer expects you to identify the basic components of a dashboard system. You should be able to draw a diagram with services, a metrics database, and a dashboard frontend. You should mention caching and basic query optimization.

**What you must cover:** Single-server architecture, pull-based metric collection (Prometheus-style), basic dashboard layout (top-down health indicators). Identify that the time series database is the central component.

**What gets you down-leveled:** Proposing a relational database as the primary metric store without discussing time series-specific optimizations. Ignoring the difference between metric storage and log storage. Suggesting that all metrics should be stored at full resolution forever.

**Sample answer structure:**
1. Services expose a `/metrics` endpoint
2. A metrics agent scrapes each service every 15 seconds
3. Metrics are stored in a time series database
4. The dashboard API queries the TSDB and returns results
5. The web frontend renders charts using a charting library
6. The dashboard refreshes every 30 seconds
7. Caching reduces load on the TSDB

### L4 / Mid Level (2-5 years experience)

At this level, the interviewer expects you to identify scaling bottlenecks and propose solutions. You should discuss the trade-off between query latency and storage cost. You should mention rollups, downsampling, and query caching.

**What you must cover:** Separation of ingestion and query paths, rollup tables for long-range queries, Redis or Memcached for query caching, horizontal scaling for the ingestion tier, basic HA with active-passive replication.

**What gets you down-leveled:** Proposing a solution that cannot handle more than 10 million data points per day. Not discussing the cost of high-cardinality labels. Ignoring the dashboard refresh cycle and its impact on query load.

**Sample answer structure:**
1. Design the ingestion pipeline with a message queue for buffering
2. Discuss the time series data format and Gorilla compression
3. Propose rollup tables at 1-minute, 1-hour, and 1-day resolution
4. Add a Redis cache layer that caches query results for 30 seconds
5. Discuss sharding the TSDB by metric name hash
6. Implement active-passive replication for the TSDB
7. Discuss dashboard layout principles: 3-5 panels on top row, drill-down

### L5 / Senior Level (5-8 years experience)

At this level, the interviewer expects you to design a production-grade system that handles high throughput and multi-tenancy. You should discuss consistency models, cardinality management, and detailed capacity planning.

**What you must cover:** Multi-tenant isolation (data separation, rate limiting, resource quotas), cardinality limits and label validation, detailed capacity planning with numbers (samples per second, storage per day, query QPS), multi-region deployment with federation, eventual consistency and its implications for dashboards.

**What gets you down-leveled:** Proposing a system that cannot handle 500,000+ active time series. Not discussing the operational cost of the system (storage, compute, bandwidth). Ignoring the multi-tenant use case (different teams, different dashboards).

**Sample answer structure:**
1. Design a multi-tenant ingestion pipeline with tenant-level rate limiting
2. Implement label validation and cardinality enforcement at ingestion time
3. Design the TSDB with consistent hash-based sharding (16-64 shards)
4. Implement a multi-tier cache: browser cache, CDN cache, Redis cache, TSDB cache
5. Design rollup policies with retention tiers: 7d raw, 30d 1m, 13mo 1h, 7yr 1d
6. Discuss multi-region: each region has its own TSDB, global view aggregates rollups
7. Present capacity numbers: 260K samples/sec, 22.5B samples/day, 416GB total storage
8. Discuss dashboard-as-code and CI/CD for dashboard definitions

### L6 / Staff Level (8-12 years experience)

At this level, the interviewer expects you to discuss the system architecture in the context of organizational needs. You should address cost optimization, reliability guarantees, and the trade-offs between different open-source and commercial solutions.

**What you must cover:** Cost analysis of different storage options (SSD vs object store vs in-memory), RPO and RTO targets for the dashboard system, comparison of Prometheus vs Thanos vs VictoriaMetrics vs Datadog, organizational alignment (who owns the dashboard platform, how teams create and maintain dashboards), SLO-based dashboard design.

**What gets you down-leveled:** Proposing expensive solutions without cost analysis. Ignoring the human and organizational aspects of dashboard maintenance. Not discussing failure modes and mitigation strategies.

**Sample answer structure:**
1. Define SLOs for the dashboard platform: 99.9% availability, p99 query latency under 1 second
2. Compare open-source options and make a build-vs-buy recommendation
3. Design a cost-optimized storage tier: local SSD for hot data, S3 for warm, Glacier for cold
4. Discuss organizational patterns: platform team owns infrastructure, service teams own dashboard content
5. Propose dashboard lifecycle management: auto-expire unused dashboards, alert on stale metrics
6. Design the incident response workflow: dashboard → runbook → traces → logs
7. Discuss monitoring the monitor: how to know when the dashboard platform itself is degraded

### L7 / Principal Level (12+ years experience)

At this level, the interviewer expects you to shape the conversation based on the company's context. You should adapt your design to the specific industry, scale, and constraints. You should discuss how the dashboard system evolves over 3-5 years as the company grows.

**What you must cover:** Multi-year evolution strategy, industry-specific patterns (fintech needs different dashboards than social media), cost optimization at scale (reducing metric volume by 50% without losing signal), influencing upstream (convincing service teams to standardize metrics), trade-offs between standardization and flexibility.

**What gets you down-leveled:** Proposing a one-size-fits-all architecture. Not discussing the organizational change management required to adopt a new dashboard platform. Ignoring the business impact of dashboard reliability.

**Sample answer structure:**
1. Start with the business context: what decisions will this dashboard influence?
2. Define the evolution: phase 1 (Prometheus + Grafana for 100 engineers), phase 2 (custom ingestion pipeline at 500 engineers), phase 3 (multi-region federated system at 2,000 engineers)
3. Discuss the metric standardization effort: define common metric names, labels, and units across the entire organization
4. Design the organizational interface: dashboard API with ACLs, dashboard review process, quarterly dashboard audit
5. Discuss cost optimization: eliminate unused metrics, reduce scrape frequency for non-critical services, implement adaptive sampling
6. Address the 5-year horizon: what happens when the company grows 10x in services and 100x in data volume

## FAANG Framework

Use the FAANG framework to structure your dashboard system design answer. Each letter represents a phase of the answer, and the interviewer expects you to spend roughly the indicated proportion of time on each phase.

### F — Frame the Problem (2 minutes)

Start by clarifying the scope and requirements. Do not start drawing boxes until you have confirmed what the interviewer wants you to build.

"Let me start by clarifying the scope. Are we designing a dashboard platform for internal engineering use, or a customer-facing analytics dashboard? Are we building the time series database from scratch, or are we integrating with existing open-source tools like Prometheus? For this discussion, let me assume we are building an internal operational dashboard platform for 500 microservices, ingesting about 500,000 metrics, with a team of 10 on-call engineers who need real-time visibility."

Confirm these requirements with the interviewer before proceeding:
- Number of services and hosts
- Metrics per service and per host
- Dashboard viewers and concurrent query load
- Retention requirements (raw and rollup)
- Availability and latency SLOs
- Multi-tenancy requirements

### A — Architecture Overview (5 minutes)

Draw the high-level architecture: ingestion pipeline, time series database, query API, dashboard frontend. Label each component and the data flow between them. Do not dive into details yet.

"Here is the high-level architecture. On the left, the ingestion pipeline collects metrics from services. In the middle, the time series database stores and indexes the data. On the right, the query API and dashboard frontend serve the data to viewers. Let me walk through each component..."

Draw:
- Services with `/metrics` endpoints
- Scrape agents or push gateways
- Message queue (Kafka) for buffering
- Time series database cluster (sharded and replicated)
- Query API with cache layer
- Dashboard frontend served from CDN
- User viewing the dashboard

### A — Analyze Deep Dive (15 minutes)

Pick the two or three most important components and dive deep. The interviewer will guide you to the areas they want to explore. Common deep-dive areas include:

**Storage deep dive:** "Let me dive into the time series storage format. Each data point is stored as a compressed chunk using Gorilla compression. The delta-of-delta encoding for timestamps and XOR for values gives us about 1.37 bytes per sample. The inverted index maps label values to bitsets of series IDs, enabling sub-millisecond label matching. Let me walk through what happens when a query comes in..."

**Ingestion deep dive:** "Let me look at how metrics flow from a service to the database. The service exposes a `/metrics` endpoint that returns metrics in Prometheus exposition format. A scraper polls this endpoint every 15 seconds, parses the text, and sends the samples to a Kafka topic. The ingestion consumer reads from Kafka, validates labels, checks cardinality, applies tenant rate limits, and writes to the TSDB..."

**Query deep dive:** "Let me trace a dashboard query from the browser to the database and back. The browser requests the dashboard definition, then fires parallel query requests for each panel. The query API checks the Redis cache first. On a cache miss, it sends the query to the TSDB, which scans the inverted index, reads compressed chunks, decompresses, aggregates, and returns the result..."

### N — Navigate Constraints (5 minutes)

Discuss the trade-offs and constraints in your design. Acknowledge weaknesses and propose mitigations.

"The main constraint in this design is the trade-off between data resolution and storage cost. Storing 10-second resolution data for 30 days would require about 1.78 TB for 500K series. My mitigation is to store raw data for only 7 days and use 1-minute rollups for 30 days and 1-hour rollups for 13 months. This reduces storage by about 5x while keeping 90% of query patterns fast.

"Another constraint is query latency on high-cardinality queries. If someone queries across 5,000 time series for 30 days, the query could take 18 seconds. The mitigation is to set a query timeout of 30 seconds and display a progress indicator to the user. We can also detect these expensive queries and suggest they use rollups or shorter time ranges."

### G — Gather Requirements and Recap (3 minutes)

Summarize your design and confirm it meets the requirements. This shows you can close a loop and communicate clearly.

"To recap, this dashboard platform ingests 260K samples per second from 500 services, stores them in a sharded time series database with compression and rollups, and serves dashboard queries through a cached query API. The system handles 400+ TSDB queries per second at peak, with p99 query latency under 500ms for the last 6 hours. Raw data is stored for 7 days, rollups for 13 months. The total storage is about 416 GB. The system is deployed across three regions with independent regional TSDBs and a global aggregation layer for cross-region dashboards. Does this design meet the requirements we discussed?"

## Quantitative Reasoning for Interviews

Dashboard design interviews often require quantitative reasoning. You must be able to estimate storage costs, query throughput, and infrastructure requirements on the fly.

### Storage Estimation

Interviewers expect you to compute storage requirements based on the number of metrics, scrape interval, and retention policy.

**Example question:** "We have 10,000 hosts, each exporting 200 metrics. Scrape interval is 15 seconds. We need 7 days of raw data and 13 months of 1-hour rollups. How much storage do we need?"

**Answer:** "Let me walk through the calculation. Each data point is approximately 1.37 bytes after Gorilla compression. 10,000 hosts times 200 metrics gives us 2 million time series. At a 15-second scrape interval, each series produces 5,760 data points per day. Total raw data per day: 2 million times 5,760 times 1.37 bytes equals 15.8 GB per day. For 7 days, that is 111 GB of raw storage."

"For the 1-hour rollups: the rollup reduces 360 data points (6 per minute times 60 minutes) to 1 data point. So the rollup data per series per day is 24 data points. 2 million series times 24 times 0.17 bytes (rollup compression is better) equals 8.2 MB per day. For 395 days, that is 3.2 GB."

"Total storage: 111 GB raw plus 3.2 GB rollups, plus approximately 200 MB for the inverted index. Approximately 115 GB total."

### Query Throughput Estimation

**Example question:** "We have 500 engineers viewing dashboards. Each dashboard has 12 panels and refreshes every 30 seconds. Cache hit rate is 70%. How many queries per second does the TSDB need to handle?"

**Answer:** "Let me calculate. 500 engineers times 12 panels per dashboard divided by 30 seconds equals 200 queries per second to the dashboard API. With a 70% cache hit rate, 30% of those queries reach the TSDB. That is 60 queries per second to the TSDB. Each query scans approximately 5 time series, so the TSDB handles 60 queries per second, with each query resolving through the inverted index and reading approximately 180 chunks per series for a 6-hour time range."

### Capacity Estimation

**Example question:** "How many ingestion nodes do we need for 5 million samples per second?"

**Answer:** "A standard ingestion node handles approximately 100,000 samples per second. For 5 million samples per second, we need 50 ingestion nodes. However, we should add 20% overhead for headroom and redundancy, bringing us to 60 nodes. We would use a consistent hash ring to distribute the scrape targets across the nodes, so adding or removing nodes requires minimal reconfiguration."

## Phrase Bank

Use these phrases to communicate your design decisions with precision and confidence.

### Data Ingestion

1. "I will use a pull-based model where a centralized scraper polls each service's metrics endpoint at a configurable interval, because pull models are simpler to debug — if a metric is missing, you know the scraper could not reach the target."
2. "The scrape interval is set to 15 seconds by default. This gives us a good balance between data freshness and storage cost. Critical services can be configured for 5-second scraping."
3. "A message queue between the scrapers and the TSDB acts as a shock absorber. If the TSDB is slow (compaction, GC pause), the queue buffers incoming data instead of dropping it."
4. "Label validation at ingestion time prevents cardinality explosions. I will reject labels that match patterns like session IDs or user IDs, and I will set a hard limit of 10,000 unique label values per metric."

### Storage and Query

5. "Gorilla compression reduces each data point from 16 bytes to about 1.37 bytes. This is critical for keeping storage costs manageable at our scale."
6. "The inverted index maps label values to bitsets of series IDs. Intersecting two bitsets is a single bitwise AND operation that takes under 100 CPU cycles."
7. "I will implement a two-tier caching strategy: a Redis cache in the query API layer (30-second TTL) and a browser cache in the dashboard frontend (matching the refresh interval)."
8. "Rollup tables reduce query latency for long time ranges by pre-computing aggregations. A query over 30 days of raw data scans 260K points per series. The same query against 1-hour rollups scans 720 points per series — a 360x reduction."

### Scalability and Reliability

9. "Consistent hashing on the metric name distributes time series evenly across shards. When we add a new shard, only 1/N of the data needs to be redistributed."
10. "Each region operates its own dashboard stack independently. This ensures regional availability during a network partition — engineers in EU can still see their dashboards even if the US region is unreachable."
11. "The query API is stateless, so it scales horizontally behind a load balancer. Adding more query nodes instantly increases query capacity."
12. "I will protect the TSDB with query timeouts (30 seconds), query concurrency limits (50 per node), and query complexity budgets (reject queries that scan more than 10 million data points)."

### Dashboard Design

13. "The dashboard follows a top-down hierarchy: the first row shows system-wide health indicators, the second row shows per-service breakdowns, and the third row provides drill-down details."
14. "I will use a consistent color scheme across all dashboards: green for healthy, yellow for warning, red for critical. The same color for the same metric type across all panels."
15. "Every panel must include a previous-period comparison. A single time series tells you the current value. A comparison tells you whether that value is normal."
16. "The dashboard is code. Dashboard definitions live in Git alongside service code, go through code review, and have a designated team owner."
17. "Annotations are the most powerful debugging tool on a dashboard. Every deployment, configuration change, and feature flag toggle should be automatically overlaid on every chart."

## Red Flags

Awareness of these red flags helps you avoid common mistakes that can lead to a negative interview outcome.

1. **Not clarifying scope.** Starting to design without confirming the number of services, metrics, viewers, or retention requirements. This signals that you jump to solutions without understanding the problem.

2. **Using a relational database for time series data.** Proposing PostgreSQL or MySQL as the primary metric store without mentioning time series optimizations. This signals lack of domain knowledge. If you mention PostgreSQL, pair it with an explanation of partitioning, BRIN indexes, and why it is still suboptimal compared to a TSDB.

3. **Ignoring compression.** Not discussing how data is stored efficiently. A dashboard system at scale generates terabytes of data per day. Without compression, the storage cost alone makes the system infeasible.

4. **No discussion of rollups.** Proposing that all data is stored at full resolution forever. This signals that you have not thought about storage costs or query performance for long time ranges.

5. **Ignoring the dashboard refresh cycle.** Treating each dashboard load as a one-time query rather than a recurring pattern. The refresh cycle multiplies the query load by 2x, 10x, or 100x depending on the number of concurrent viewers.

6. **Proposing a monolith.** A single server that does ingestion, storage, query, and presentation. This works at small scale but fails completely at medium or large scale. The interviewer is testing whether you can separate concerns.

7. **No caching strategy.** Proposing that every dashboard query hits the TSDB directly. This signals that you have not thought about the most basic performance optimization for read-heavy systems.

8. **Ignoring multi-tenancy.** Not discussing how different teams, services, or customers are isolated from each other. In any real deployment, teams need to see their own metrics and not be affected by other teams' queries.

9. **No failure mode discussion.** Proposing an architecture without discussing what happens when components fail. Every distributed system fails. The interviewer wants to hear your mitigation strategies.

10. **Not discussing the human side.** Focusing only on the technical architecture and ignoring how teams create, maintain, and use dashboards. A technically perfect dashboard platform fails if no one maintains the dashboards.

## Negotiating the Interview

In a system design interview, you are not just answering questions. You are negotiating the scope and direction of the conversation. A strong candidate guides the interviewer through their logic and redirects when the interviewer goes too deep into a tangential topic.

### When the Interviewer Interrupts

Interviewers often interrupt to test your composure and your ability to handle feedback. When you are interrupted, pause, acknowledge the interruption, and incorporate the feedback.

**Interviewer:** "I think you are spending too much time on the ingestion pipeline. Can we focus on the storage layer?"

**Candidate:** "You are right, I have been spending too long on ingestion. Let me summarize the ingestion design in one sentence: pull-based scraping with a Kafka buffer to decouple ingestion from storage. Now let me dive into the storage layer, which I believe is the most critical component for this design."

This response shows that you can take feedback, summarize concisely, and move in the direction the interviewer wants.

### When the Interviewer Challenges Your Design

Interviewers challenge your design to test whether you understand the trade-offs and can defend your decisions. Do not get defensive. Acknowledge the challenge and explain your reasoning.

**Interviewer:** "Why are you using pull-based scraping? Push-based is faster and simpler."

**Candidate:** "That is a fair point. Push-based scraping is simpler to implement because the service just sends metrics without needing a scrape endpoint. However, I chose pull-based for two reasons. First, discovery: in a pull model, the scrape agent discovers targets through service discovery, so new services are automatically scraped without configuration. Second, debuggability: if a metric is missing, I know the scrape agent could not reach the target, which is easier to diagnose than a push failure where the service thinks it sent data but the receiver dropped it. Could you share more about your context? For short-lived tasks like Lambda functions, I would recommend push-based, but for the long-lived services in this design, pull-based is more reliable."

This response shows that you understand the trade-off, can articulate your reasoning, and can adapt to the interviewer's context.

### When You Do Not Know the Answer

It is acceptable to say "I don't know" if you genuinely do not know something. But follow it up with your reasoning about what you would do to find the answer.

**Interviewer:** "How does Gorilla compression work at the bit level?"

**Candidate:** "I don't know the exact bit-level encoding, but I know the high-level approach: it uses delta-of-delta encoding for timestamps and XOR encoding for values, achieving approximately 1.37 bytes per sample. For a system design discussion, I would not need to implement Gorilla from scratch — I would use an existing TSDB like Prometheus or VictoriaMetrics that already implements it. In practice, the compression ratio is what matters for storage cost, and Gorilla gives us a 10x improvement over uncompressed storage."

This response shows that you are honest about your limitations but can still provide useful context from your knowledge.

Set a timer for 5 minutes. On a whiteboard or paper, draw the complete data flow for a dashboard query: from the browser loading the page to the final chart rendering. Include every intermediate step: CDN, load balancer, query API, Redis cache, TSDB inverted index, chunk read, decompression, aggregation, JSON serialization, and chart rendering. Do this until you can draw the entire flow from memory without pausing.

### Drill 2: Capacity Planning Calculation (10 minutes)

Calculate the infrastructure requirements for these scenarios. Do not look up answers — estimate based on the numbers in this chapter.

- Scenario A: 1 million DAU, 200 services, 300 hosts, 15-second scrape interval, 7-day raw retention, 13-month rollup. Calculate samples per second, storage per day, total storage, and query QPS.
- Scenario B: 50 million DAU, 5,000 services, 10,000 hosts, 5-second scrape interval for critical services (20%), 15-second for rest. Same retention. Calculate total storage and cluster size.
- Scenario C: Customer-facing analytics dashboard, 10,000 customers each tracking 100 metrics, 1-second resolution, 90-day retention. Calculate the ingestion rate and storage cost.

Find a partner or use a recording device. Give yourself 45 minutes to design a dashboard system for a fictional company. Use the FAANG framework. Record your answer and review it for the following:

- Did you discuss trade-offs explicitly?
- Did you cover ingestion, storage, query, and presentation?

- Did you discuss failure modes and mitigations?

### Drill 4: Anti-Pattern Identification (10 minutes)

Review a real dashboard (from your workplace or a public example). Identify as many anti-patterns as you can:

- Is there a clear top-down hierarchy?

Apply the fixes from the Trade-offs chapter and redesign the dashboard layout.

### Drill 5: Phrase Bank Recitation (5 minutes)

Recite the 17 phrase bank entries from this chapter from memory. Then, for each phrase, explain in one sentence why that specific phrasing is effective in an interview setting. This builds recall speed so you can use these phrases naturally during an interview.

## Common Mistakes Candidates Make in Dashboard Interviews

### Mistake 1: Designing a Dashboard for Yourself, Not the User

Candidates often design dashboards that show the metrics they personally find interesting, not the metrics that the on-call engineer needs. A dashboard that shows every metric the system emits is not a dashboard; it is a data dump.

**What to do instead:** Ask the interviewer: "Who is the primary user of this dashboard? Are they an on-call engineer looking for problems, a manager looking for trends, or a developer debugging a specific issue?" Design the dashboard for that user's specific needs.

### Mistake 2: Ignoring the Cost of Storage

Candidates propose storing all metrics at full resolution forever. This is economically infeasible at scale. A system with 500K series storing at 10-second resolution generates 1.78 TB of data per day. After 30 days, that is 53 TB. After a year, that is 650 TB.

**What to do instead:** Discuss compression (Gorilla reduces to 1.37 bytes per point), rollups (1-minute, 1-hour, 1-day), and tiered storage (SSD for hot, S3 for cold). Show that you understand the cost implications of your design decisions.

### Mistake 3: No Mention of Caching

Candidates design the query path without any caching layer. Every dashboard query hits the TSDB directly. This works for a single user, but fails when 100 engineers are viewing dashboards simultaneously.

**What to do instead:** Always mention caching. The cache layer is the most important performance optimization for a dashboard system. Discuss cache TTL, cache invalidation, and the difference between browser cache, CDN cache, API cache, and TSDB cache.

### Mistake 4: Overlooking the Scrape Interval

Candidates design a dashboard that refreshes every 5 seconds but has a scrape interval of 60 seconds. The dashboard is refreshing 12x faster than the data is collected, wasting resources without providing any benefit.

**What to do instead:** Match the dashboard refresh interval to the scrape interval. A 15-second scrape interval supports a 15-30 second dashboard refresh. A 60-second scrape interval supports a 60-second refresh. Never refresh faster than the scrape interval.

### Mistake 5: No Discussion of Series Churn

Candidates design a time series database without considering series churn. They assume the number of time series is stable. In reality, metrics with high-cardinality labels (user IDs, session IDs, container IDs) can create millions of new series per day, causing the TSDB to run out of memory.

**What to do instead:** Discuss label validation, cardinality limits, and the difference between "active series" (currently emitting data) and "total series" (all series ever seen). Explain how you would handle a cardinality explosion: reject high-cardinality labels, route them to a separate TSDB, or use a TSDB that handles high cardinality well (VictoriaMetrics, ClickHouse).

## Interview War Stories

### Story 1: The Dashboard That Crashed the TSDB

A candidate was asked to design a dashboard for 500 microservices. Their design included a single panel showing "all metrics for all services on one chart." The interviewer asked: "What happens when this panel loads?" The candidate did not realize that a chart with 500 overlapping lines is unreadable and that the query would scan every time series in the database, causing a query timeout and degrading performance for all other viewers. The candidate was rejected for not understanding the impact of high-cardinality queries.

**Lesson:** Always consider the cost of a dashboard query. Every panel should be justified by answering: "What actionable information does this panel provide?" If the answer is "it shows everything," the panel is poorly designed.

### Story 2: The 30-Second Refresh

A candidate designed a dashboard that refreshed every 3 seconds. The interviewer asked: "Why 3 seconds?" The candidate said: "To show real-time data." The interviewer asked: "What is your scrape interval?" The candidate said: "60 seconds." The interviewer pointed out that the dashboard was refreshing 20 times more often than the data was being collected, wasting database and network resources without providing any benefit.

**Lesson:** The dashboard refresh interval should match the scrape interval. Refreshing faster than the scrape interval shows the same data repeatedly. Refreshing slower than the scrape interval means the dashboard is always behind.

### Story 3: The Ten Thousand Panel Dashboard

A candidate was asked to design a dashboard for a company with 2,000 microservices. Their design included a single dashboard with 2,000 panels — one panel per service. The interviewer asked: "How long would this dashboard take to load?" The candidate estimated 30-40 seconds. The interviewer asked: "Is that acceptable?" The candidate realized that a dashboard that takes 40 seconds to load is never loaded, and all 2,000 panels are wasted.

**Lesson:** A dashboard with more than 15 panels is a catalog, not a dashboard. Use drill-down navigation to organize hundreds of services into a hierarchy: overview (3-5 panels), service group (10-15 panels), individual service (10-15 panels).

- [ ] Did you discuss compression (Gorilla or similar)?
- [ ] Did you mention the inverted index and bitset operations?
- [ ] Did you include a caching strategy with specific TTLs?
- [ ] Did you discuss rollups at multiple resolutions?
- [ ] Did you draw a clear architecture diagram?

## System Design Problem Breakdown

When the interviewer asks a dashboard design question, break it down into these sub-problems and address each one. This structure shows that you can decompose a complex problem into manageable pieces.

### Sub-problem 1: Metric Ingestion (15% of answer)

"How do metrics get from the service to the database?" Discuss the pull vs push model, scrape intervals, target discovery, and the ingestion pipeline (service → scrape agent → buffer → TSDB). Mention service discovery (Kubernetes, Consul, file-based) and how you handle transient targets.

### Sub-problem 2: Data Storage (25% of answer)

"How are metrics stored and indexed?" Discuss the time series data model, inverted index for label matching, Gorilla compression, chunk-based storage, WAL for durability, and MemTable for buffering. Mention storage tiers (SSD for hot, HDD for warm, S3 for cold) and retention policies.

### Sub-problem 3: Query Execution (20% of answer)

"How does a dashboard query go from the browser to the database and back?" Discuss the query API, caching layer, inverted index lookup, chunk selection and decompression, aggregation computation, and result serialization. Mention query timeout, concurrency limits, and query optimization techniques (predicate pushdown, vector matching, recording rules).

### Sub-problem 4: Dashboard Rendering (15% of answer)

"How does the dashboard render charts in the browser?" Discuss the SPA architecture, dashboard definition JSON, parallel query execution for multiple panels, chart rendering (Canvas vs SVG), refresh cycle, and drill-down navigation. Mention the 12-panel limit and the F-pattern layout.

### Sub-problem 5: Scaling and Reliability (15% of answer)

"How does the system handle growth and failures?" Discuss horizontal scaling for each tier, sharding for the TSDB, multi-region deployment, disaster recovery, and monitoring the dashboard platform itself. Mention specific failure modes: TSDB crash, scrape agent failure, network partition, and query overload.

### Sub-problem 6: Multi-Tenancy and Governance (10% of answer)

"How does the system support multiple teams?" Discuss namespace-based isolation, RBAC, rate limiting per team, dashboard ownership, lifecycle management, and audit logging. Mention the distinction between platform-level multi-tenancy (sharing the TSDB) and dashboard-level multi-tenancy (sharing the dashboard tool).

## Common Follow-Up Challenges

Interviewers often present follow-up challenges to test your ability to adapt. Here are common challenges and how to handle them.

**Challenge: "Our dashboard is showing incorrect data. How do you debug it?"** Walk through the data pipeline step by step. Check the scrape: is the service's `/metrics` endpoint returning the correct value? Check the ingestion: is the scrape agent storing the data correctly? Check the storage: is the TSDB returning the correct value for the exact timestamp? Check the query: is the dashboard query using the correct aggregation and time range? Start at the source (the service) and work forward through the pipeline.

**Challenge: "Our dashboard is too slow. What do you optimize first?"** Start with the most impactful optimization: caching. Add a Redis cache with a 30-second TTL. If that is not enough, optimize the queries: use recording rules to pre-compute expensive aggregations. If the dashboard is still slow, optimize the storage: add rollups for long time ranges. If the dashboard is still slow, optimize the hardware: add more query nodes or upgrade to faster SSDs.

**Challenge: "Our team has 500 dashboards and no one maintains them. What do you do?"** Implement dashboard lifecycle management. First, audit all dashboards: check the view count, the last modification date, and the metric freshness. Archive dashboards that have not been viewed in 90 days. Flag dashboards with broken queries. Assign each dashboard to a team owner. Require an annual review for each dashboard. Automate the process: a weekly cron job sends a report of dashboard health to each team.

**Challenge: "Our dashboard is showing a spike, but we cannot find the cause. What is missing?"** The dashboard is missing annotations. Deployments, configuration changes, and feature flag toggles should be automatically overlaid on every chart. If the spike coincides with a deployment annotation, the cause is the deployment. If the spike does not coincide with any annotation, the cause is likely a third-party dependency or a network issue. Add third-party dependency metrics and network metrics to the dashboard.

## Post-Interview Checklist

**How does your dashboard system relate to your alerting system?** The answer should discuss the feedback loop: alerts fire based on dashboard metrics, the dashboard shows alert annotations, and the dashboard is the first tool the on-call engineer opens when an alert fires. The dashboard and alerting systems share the same TSDB but serve different purposes: dashboards for situational awareness, alerts for notification.

**How do you integrate logs and traces into your dashboards?** The answer should discuss linking from dashboard panels to log queries and trace views. When an engineer sees a latency spike on the dashboard, they click the spike and see the relevant log lines and trace spans. The integration is implemented as URL parameters: `/logs?service=auth-api&time=14:32-14:35&level=error` opens the log viewer filtered to the error spike.

**How would you design a dashboard for non-technical stakeholders?** The answer should discuss simplifying metrics (use business terms like "revenue" instead of "payment_success_rate"), using single stat panels with large fonts, providing period-over-period comparisons, and avoiding any technical jargon. The executive dashboard should answer: "Are we doing better or worse than last month?"

**How do you handle dashboard localization?** For organizations with global teams, dashboards must support multiple languages. The answer should discuss externalizing dashboard labels and descriptions into translation files, and detecting the viewer's locale from their browser or user profile. The metric names themselves stay in English, but the dashboard labels, legends, and descriptions can be translated.

## Negotiation Tactics for System Design Interviews

System design interviews are, in part, a negotiation. You are negotiating the scope of the problem, the constraints you will work under, and the level of detail you will go into. Effective negotiators get better outcomes.

**Tactic 1: Set the scope early.** "Let me confirm the scope. We are designing a dashboard platform for 500 microservices with 100 on-call engineers. The system needs to handle 15,000 samples per second at ingestion and 40 queries per second at peak. Is that the right scope, or should I adjust it?" This tactic ensures you are solving the problem the interviewer wants solved, not the problem you assumed.

**Tactic 2: Offer choices.** "I see two approaches here: a pull-based ingestion model using Prometheus, which is simpler to debug, or a push-based model using StatsD, which handles short-lived tasks better. Which approach would you like me to explore?" This tactic shows that you understand trade-offs and gives the interviewer a sense of control over the direction.

**Tactic 3: Use the "yes, and" approach.** When the interviewer proposes a constraint or a requirement, do not push back. Accept it and build on it. "Yes, we need to support 10-year retention, and that means we need a cost-effective long-term storage solution. Let me discuss how we can use S3 Glacier with 1-day rollups to keep storage costs under $100 per year."

**Tactic 4: Admit what you do not know, then pivot.** "I do not know the exact throughput of a single Prometheus server, but I know it is approximately 500,000 active series. For our design with 2.5 million series, we would need 5 Prometheus servers, which aligns with our sharding strategy." This tactic shows honesty and the ability to reason from first principles.

## Rapid-Fire Questions

These are short-answer questions that test your fundamental knowledge. Practice answering each one in 30 seconds or less.

**Q: What is a time series?** A sequence of data points indexed by timestamps, representing a metric over time.

**Q: What is Gorilla compression?** A compression algorithm that uses delta-of-delta for timestamps and XOR for values, achieving 1.37 bytes per sample.

**Q: What is an inverted index?** A data structure that maps label values to bitsets of series IDs, enabling sub-millisecond label matching.

**Q: What is the four golden signals?** Latency, traffic, errors, and saturation — the four metrics every service dashboard should show.

**Q: What is the USE method?** Utilization, Saturation, and Errors — a framework for infrastructure dashboard design.

**Q: What is the RED method?** Rate, Errors, and Duration — a framework for service dashboard design.

**Q: What is a scrape interval?** The frequency at which a metrics agent polls a service for data, typically 15-60 seconds.

**Q: What is a recording rule?** A pre-computed expression that runs periodically and stores its result as a new time series.

**Q: What is cardinality?** The number of unique label value combinations for a metric, which determines the number of time series.

**Q: What is a rollup?** An aggregated version of metric data at lower resolution, used for long-term storage and fast queries.

## Advanced Design Considerations

### Handling Bursty Metrics

Some metrics are bursty: they remain at zero for long periods and then spike suddenly. Examples include error rates (zero until something fails), queue depths (zero until a backlog forms), and deployment metrics (zero between deployments).

**The challenge:** A dashboard that auto-scales the y-axis makes bursty metrics invisible until they explode. The chart shows a flat line at zero for hours, then a spike that shoots off the chart. The auto-scaling y-axis adjusts to the spike, but by the time the engineer sees it, the spike may have already passed.

**The solution:** Use a logarithmic y-axis for bursty metrics. A log scale shows both the zero baseline and the spike in the same chart. The spike is visible as a sudden rise from the baseline, and the engineer can see both the pre-spike and post-spike values. Alternatively, use a heatmap with a color scale that highlights spikes.

### Handling Sparse Data

Some metrics are sparse: they are collected infrequently or only when specific events occur. Examples include batch job metrics (collected once per day), cleanup task metrics (collected once per hour), and cron job metrics (collected on a fixed schedule).

**The challenge:** A line chart with gaps shows broken lines that are hard to interpret. The engineer cannot tell whether the gap means "no data" (the metric was not collected) or "zero data" (the metric was collected and the value is zero).

**The solution:** Use a step chart (where the value stays constant until it changes) instead of a line chart. A step chart shows the last known value as a horizontal line until the next data point arrives. The engineer can see both when the data was collected (the step point) and the value at that time (the horizontal line).

### Handling Missing Data

Missing data is a fact of life in dashboard systems. A scrape fails, a service is down, a network partition splits the cluster. The dashboard must handle missing data gracefully.

**The "no data" state:** When a metric has no data for the selected time range, the dashboard panel should show a clear "No data" message with the metric name and the time range. It should not show a blank panel or a chart with a single data point at zero.

**The "partial data" state:** When some data points are missing within the time range, the line chart should show a gap (a break in the line) at the missing point. It should not interpolate across the gap or show a zero value for the missing point. The gap tells the engineer that the data source was unavailable for that period.

**The "stale data" state:** When the most recent data point is older than the expected scrape interval (e.g., the last scrape was 5 minutes ago and the scrape interval is 15 seconds), the dashboard should show the stale data in a different color (gray or dimmed) and display a warning: "Data may be stale. Last update: 5 minutes ago."

## Mock Interview Script

Here is a complete mock interview transcript between a candidate (C) and an interviewer (I). Study the candidate's responses for effective framing, structure, and vocabulary.

**I:** "Design a monitoring dashboard system for a company with 500 microservices."

**C:** "Before I start designing, let me clarify the scope. How many engineers will view these dashboards? What is the data retention requirement? And are we building the monitoring infrastructure from scratch or integrating with existing tools?"

**I:** "About 100 on-call engineers. 30-day retention for most metrics, 7 days for high-resolution data. You can assume we have Prometheus available but we can extend it."

**C:** "Good. Let me define the requirements: 500 services, each exposing approximately 50 metrics, scraped every 15 seconds. That gives us about 500 x 50 = 25,000 active time series and 25,000 / 15 = 1,667 samples per second from scraping. Adding infrastructure metrics (CPU, memory, disk) from 2,000 hosts gives us about 2,000 x 100 = 200,000 infrastructure time series. Total: approximately 225,000 active time series and 15,000 samples per second. For 100 concurrent viewers each loading dashboards with 12 panels that refresh every 30 seconds, we need to handle about 100 x 12 / 30 = 40 queries per second. Does this sound right?"

**I:** "Yes, those numbers are reasonable."

**C:** "Here is my high-level architecture. On the left, the ingestion tier. Each service exposes a /metrics endpoint. Prometheus servers scrape these endpoints in a pull model. I will run 4 Prometheus servers, each responsible for 125 services and 500 hosts. Data from Prometheus is sent to a Thanos sidecar, which uploads compressed blocks to S3 for long-term storage.

"In the middle, the query tier. A Thanos query frontend accepts PromQL queries and fans them out to the individual Prometheus servers and the S3-backed store. This gives us a unified query interface. I will add a Redis cache with a 30-second TTL in front of the query frontend to absorb repeated queries from multiple viewers.

"On the right, the presentation tier. Grafana serves dashboards to the engineers. Dashboard definitions are stored in Git and deployed through a CI/CD pipeline. Each team owns their dashboards, and the platform team reviews them for consistency every quarter.

"The key design decisions are using Thanos for long-term storage (unlimited retention at S3 cost), Redis for query caching (60-80% cache hit rate), and Git-based dashboard management (preventing dashboard drift). The system handles 15,000 samples per second at ingestion and 40 queries per second at peak, with p99 query latency under 500ms for the last 6 hours and under 2 seconds for 30-day queries."

**I:** "How do you handle a situation where one team's expensive query slows down everyone else's dashboards?"

**C:** "I would implement three protections. First, query timeout: any query taking longer than 30 seconds is cancelled. Second, per-tenant rate limiting: each team has a quota of 50 queries per second. Teams that exceed the quota receive 429 responses. Third, query classification: I would mark dashboard queries as 'priority' and ad-hoc queries as 'best-effort.' The TSDB prioritizes priority queries when it has resource contention. Additionally, I would detect expensive queries (queries that scan more than 10 million data points) and cache their results for 5 minutes instead of 30 seconds, so the expensive computation is shared across all viewers."

## Advanced Interview Topics

### Handling High Cardinality at Scale

High cardinality is the most common failure mode for TSDBs. Interviewers frequently ask: "What happens when a label like `user_id` or `session_id` is added to a metric?"

The correct answer covers three dimensions. First, detection: the ingestion pipeline validates labels against an allow list and rejects labels that match high-cardinality patterns (anything ending in `_id`, `_uuid`, `_hash`). Second, mitigation: if high-cardinality labels are necessary (e.g., per-customer metrics for a SaaS platform), route them to a separate TSDB that is optimized for high cardinality (like ClickHouse or a dedicated VictoriaMetrics cluster). Third, recovery: when cardinality exceeds the TSDB's capacity, the TSDB restarts with a clear index. The high-cardinality data in the WAL is replayed but may be truncated if it exceeds the cardinality limit.

### Multi-Tenant Isolation Strategies

For a dashboard platform used by 50 teams, each team must be isolated from the others. Interviewers ask: "How do you ensure that one team's heavy usage does not affect another team's dashboards?"

The answer involves three layers of isolation. At the metric level, each team's metrics are prefixed with the team namespace: `team-payments.request_latency`. At the storage level, metrics are sharded by namespace, so each team's data lives on dedicated shards. At the query level, each team has a dedicated query endpoint with its own rate limit and resource quota. The query endpoint routes to the team's shards. If one team's queries overload their shards, only that team's dashboards are affected.

### Dashboard as Code

Many companies now manage dashboards as code: dashboard definitions are stored in Git, reviewed through pull requests, and deployed through a CI/CD pipeline. Interviewers ask: "How would you implement dashboards as code?"

The answer covers the dashboard definition format (JSON or YAML), the CI/CD pipeline (lint, validate, deploy, smoke test), and the versioning strategy (semantic versioning for dashboards, with backward compatibility for linked dashboards). The dashboard-as-code approach ensures that every dashboard change is reviewed, tested, and documented. It also enables automated dashboard generation from service catalogs: when a new service is deployed, a CI/CD job generates a baseline dashboard from a template and opens a pull request for the service team.
