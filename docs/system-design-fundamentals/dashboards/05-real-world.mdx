## Company Case Studies

### Netflix: The Atlas Monitoring System

Netflix built its own monitoring system called Atlas because no off-the-shelf solution could handle the scale of Netflix's streaming infrastructure. Atlas ingests over 1.5 billion metrics per day from thousands of microservices running across multiple AWS regions.

The key innovation in Atlas is the **dimensional time series model**. Each metric carries an arbitrary number of tags (labels), and the query language lets you slice and aggregate across any combination of tags. This means a Netflix engineer can write a single query that computes the average streaming bitrate for all users on the "Stranger Things" title, grouped by device type and region.

**Dashboard design philosophy at Netflix:**

- **Every service owner builds their own dashboards.** The platform team provides the building blocks (Atlas, Grafana); service teams own their dashboard content. This distributes the maintenance burden and ensures the people who understand the service design its dashboards.

- **Dashboards are code.** Dashboard definitions are stored in Git alongside service code. A dashboard change goes through the same code review process as a service change. This prevents dashboard drift — every dashboard has a known owner and a review history.

- **Top-down drill structure.** Netflix's dashboards follow a strict hierarchy: executive overview (6 panels) → service overview (12 panels) → service detail (20 panels) → host detail (30 panels). An engineer going from 30,000 feet to a specific host follows a well-defined drill-down path with exactly 3 clicks.

### Uber: Real-Time and Historical Dashboards

Uber operates two distinct dashboard systems because no single system can serve both real-time operational monitoring and historical trend analysis at Uber's scale.

**M3 (real-time metrics platform)** — M3 is Uber's open-source time series database built on top of M3DB. It ingests over 10 billion data points per day from Uber's 4,000+ microservices. M3 powers Uber's real-time dashboards with sub-second query latency. Data flows from services through a Kafka ingestion pipeline into M3DB, which stores data in memory and on SSDs with 3x replication.

**Hive (historical analytics)** — For long-term trend analysis, Uber's dashboards query Hive, the data warehouse built on top of Hadoop and Presto. Historical dashboards show year-over-year comparisons, seasonal patterns, and capacity planning projections. Data is loaded into Hive as nightly ETL jobs from the M3 cold storage.

**Dashboard design philosophy at Uber:**

- **Two tiers of dashboards.** Real-time dashboards refresh every 15 seconds and show the last 6 hours. Historical dashboards refresh daily and show the last 2 years. Engineers use different dashboards for different tasks: the real-time dashboard for incident response, the historical dashboard for capacity planning.

- **Service-level objectives (SLOs) baked into every dashboard.** Every Uber dashboard has a row of SLO attainment panels at the top. These panels show whether each critical metric is meeting its target (green), approaching its limit (yellow), or breaching (red). This forces every team to define what "healthy" means.

- **Dashboard as incident commander tool.** During incidents, Uber's incident commander has a dedicated dashboard that shows cross-service health. This dashboard has a fixed, unchanging layout that every incident commander knows by heart. No customization is allowed — consistency trumps flexibility in incident response.

### Discord: From Prometheus to a Custom Pipeline

Discord started with standard Prometheus and Grafana for their monitoring stack. As they grew to over 150 million monthly active users and 500+ microservices, they hit three specific limits: cardinality explosion from user-specific metrics, query latency on large time ranges, and storage costs for high-resolution data.

**The cardinality problem.** Discord's per-guild metrics (metrics tagged with guild/server IDs) created hundreds of thousands of unique time series. The Prometheus TSDB's inverted index, held in memory, ballooned to 80GB. GC pauses from index management caused query timeouts. Discord's solution was to move high-cardinality metrics to a separate system (ClickHouse) that handles high-cardinality queries efficiently.

**Dashboard design philosophy at Discord:**

- **Minimal viable dashboard.** Discord's on-call dashboards show exactly 7 panels: overall error rate, p99 latency, request rate, active users, database connections, cache hit rate, and queue depth. Nothing else. The team found that every additional panel beyond 7 increased the average time to identify an incident.

- **Incident dashboards are distinct from monitoring dashboards.** When an incident is declared, Discord's incident command automatically activates a separate set of incident dashboards that show high-cardinality, high-resolution data. These dashboards would be too expensive to run continuously, but they are invaluable during an incident.

- **Correlated annotations.** Discord's dashboards automatically overlay deployment marks, configuration changes, and feature flag toggles on every chart. When latency spikes at 14:32, the annotations show a deployment at 14:30. The correlation is automatic and covers 90% of root cause explanations.

### Cloudflare: Real-Time Analytics Dashboard

Cloudflare serves over 25 million internet properties and handles tens of millions of requests per second. Their analytics dashboard shows each customer their traffic patterns, security events, and performance metrics in real time, with a maximum of 10 seconds of latency.

**The scale challenge.** Cloudflare's dashboard system ingests over 100 million data points per second from every server in their global network. Each data point carries 15-20 labels (customer ID, data center, protocol, status code, cache hit/miss, attack type). The total time series cardinality exceeds 5 billion.

**Dashboard design philosophy at Cloudflare:**

- **Pre-aggregation at the edge.** Cloudflare's servers aggregate metrics into 1-minute buckets locally before sending them to the central dashboard system. This reduces the data volume by a factor of 60 (from 1-second resolution to 1-minute resolution) before the data ever leaves the server. The edge aggregation is lossy — it only keeps min, max, sum, and count — but it is sufficient for every dashboard query.

- **No live queries on raw data.** The dashboard never queries raw time series data. Every dashboard query reads from pre-computed rollup tables that are generated by a continuous streaming aggregation pipeline. The rollup tables are partitioned by customer ID and time, so a query for one customer's last 24 hours reads exactly one partition.

- **Demo-based dashboard design.** Cloudflare's dashboard team sits with customers while they use the dashboard. They watch where customers click, what they search for, and which panels they ignore. This feedback loop drives dashboard design: any panel that customers don't look at is removed or demoted. The most-used panels are made larger and moved higher.

## Open Source Implementations

### Prometheus + Grafana

The most popular open source dashboard stack. Prometheus handles metric collection and storage. Grafana handles dashboard creation and visualization. The stack is simple to deploy, well-documented, and sufficient for teams with under 1,000 hosts.

```
Pros: Simple setup, active community, extensive integrations,
  mature alerting (Alertmanager), hundreds of dashboard templates
Cons: Single-node Prometheus is not highly available, limited
  long-term storage (local disk), query latency degrades with
  high cardinality (>500K series)

Best for: Teams with 50-500 hosts, standard web application monitoring
```

### Thanos

Thanos extends Prometheus with global query views, unlimited retention, and high availability. It wraps Prometheus instances and adds a sidecar that uploads data to object storage (S3, GCS). A query frontend aggregates results from multiple Prometheus instances and the object store.

```
Pros: Unlimited retention (object storage), global query view
  across multiple Prometheus instances, HA without replica
  duplication, downsampling for fast long-range queries
Cons: More complex than plain Prometheus, higher infrastructure
  cost (object storage), requires Kubernetes for easy deployment

Best for: Teams with 500+ hosts, multi-cluster Kubernetes, 
  long-term retention requirements
```

### VictoriaMetrics

VictoriaMetrics is a purpose-built time series database that is compatible with the Prometheus query language but offers better performance and lower resource usage. It handles 10x more active time series per node than Prometheus with the same hardware.

```
Pros: 10x higher cardinality per node than Prometheus, lower
  memory usage, faster queries on large time ranges, built-in
  downsampling, single binary deployment
Cons: Less mature ecosystem than Prometheus, fewer integrations,
  smaller community

Best for: Teams that need high cardinality support (>1M series)
  or want better performance on existing hardware
```

## Cloud Provider Implementations

### AWS CloudWatch

CloudWatch is AWS's managed monitoring service. It ingests metrics from AWS services (EC2, RDS, Lambda, ELB) and custom application metrics. CloudWatch dashboards support up to 500 widgets per dashboard with automatic scaling.

```
Pros: Zero setup for AWS services, no infrastructure to manage,
  integrated with AWS alarms and SNS, metric math for on-the-fly
  transformations
Cons: Limited custom metric resolution (1-minute minimum, 1-second
  for detailed monitoring at extra cost), query latency varies,
  dashboard rendering can be slow with many widgets
```

### Google Cloud Monitoring (formerly Stackdriver)

Google Cloud Monitoring provides dashboards integrated with GCP services and supports custom metrics via the Monitoring API. It offers uptime checks, alerting policies, and a dashboard API for programmatic dashboard management.

```
Pros: Native integration with GCP services, uptime checking,
  dashboard-as-code via API, good query performance on large
  data sets (backed by Bigtable)
Cons: Limited open-source compatibility, vendor lock-in,
  higher cost for high-resolution custom metrics
```

### Azure Monitor / Grafana on Azure

Azure Monitor collects metrics from Azure resources and sends them to Azure Managed Grafana for dashboard rendering. The managed Grafana service handles scaling, patching, and availability.

```
Pros: Managed Grafana (no ops), Azure-native integration,
  supports data sources beyond Azure (Prometheus, InfluxDB)
Cons: Limited customization compared to self-hosted Grafana,
  slower feature releases than open-source Grafana
```

## Lessons Learned

### Lesson 1: Every dashboard needs an owner

A dashboard without an owner is a dead dashboard. Metrics change, services are renamed, labels are modified — and the orphaned dashboard never gets updated. It slowly fills with broken queries and "No data" panels. Assign every dashboard a team owner in the dashboard metadata, and require the owner to review the dashboard quarterly.

### Lesson 2: The best dashboard is the one that does not exist

If a metric does not trigger any action — no alert, no decision, no investigation — then it does not belong on a dashboard. Every dashboard panel should answer a specific question, and that question should be documented in the dashboard description. If the panel answers "I don't know, it just seemed interesting," remove it.

### Lesson 3: Ratios are the enemy of clarity

A dashboard that shows "error rate" as a percentage hides the absolute magnitude. A 5% error rate looks bad when you have 1,000 requests per minute (50 errors). The same 5% looks acceptable when you have 1,000,000 requests per minute (50,000 errors). Show both the rate and the absolute count. Better yet, show the actual count and let the viewer judge the severity.

### Lesson 4: Dashboards for humans, not for machines

Automated systems (alerting, auto-scaling, incident response) should query the TSDB directly, not scrape dashboard panels. Dashboards are optimized for human visual pattern recognition. Automated systems need precise machine-readable thresholds and structured data. Do not make your alerting system parse a chart image to decide whether to page an engineer.

### Lesson 5: Always include the previous period comparison

A single time series tells you the current value. A time series with a comparison to the previous week tells you whether the current value is normal. A 20% error rate is alarming unless you see that it was 25% last Tuesday at the same time — which means the system is actually improving. Every dashboard panel should overlay the previous period as a dimmed or dashed series.

### Stripe: Dashboard-Driven Incident Response

Stripe's incident response process is built around dashboards. Every incident starts with a dashboard that shows the blast radius: which services are affected, how many users are impacted, and whether the incident is growing or shrinking.

**The incident dashboard.** When an incident is declared, a dedicated incident dashboard is automatically created. This dashboard shows the affected service's metrics at 15-second resolution (instead of the standard 60-second resolution), with annotations for every action taken during the incident (who was paged, when they joined the bridge, what changes were made). The incident dashboard persists after the incident is resolved and becomes the postmortem artifact.

**The error budget dashboard.** Every Stripe team has a dashboard that shows their SLO attainment and error budget. The dashboard is divided into three sections: the error budget remaining (as a percentage and a time estimate), the burn rate (how fast the budget is being consumed), and the top contributors to error budget consumption (which requests, which endpoints, which error codes). The error budget dashboard is reviewed in every weekly team meeting.

**The API consumer dashboard.** Stripe's external API consumers get their own dashboards that show their API usage, error rates, and latency. These dashboards are used by Stripe's developer relations team to identify customers who are experiencing degraded performance before the customers report it. The dashboards are embedded in Stripe's customer portal and are refreshed every 60 seconds.

### Etsy: Dashboards for Continuous Deployment

Etsy deploys code 50+ times per day. Their dashboard system is designed to catch deployment-induced regressions within 60 seconds of a deploy.

**The deploy dashboard.** Every Etsy deployment shows a dashboard that compares the canary (new version) to the baseline (old version) across 20 metrics: error rate, latency, CPU, memory, request rate, database queries per request, cache hit rate, and garbage collection pauses. The dashboard uses a difference chart: a horizontal line at zero means no change, bars above zero mean the canary is worse, bars below zero mean the canary is better. If any metric shows a statistically significant degradation, the deployment is automatically rolled back.

**The experiment dashboard.** Etsy runs A/B experiments on almost every feature. Each experiment has a dedicated dashboard that shows the experiment metrics split by variant. The dashboard uses Bayesian statistics to compute the probability that each variant is better than the control. The dashboard updates every 5 minutes as new data arrives.

### Lessons from the Case Studies

**Lesson 6: Separate dashboards for different audiences.** The operational dashboard that the on-call engineer uses is different from the business dashboard that the VP of Engineering uses. They show different metrics at different resolutions with different refresh rates. Trying to build one dashboard that serves both audiences results in a dashboard that serves neither.

**Lesson 7: Automate dashboard creation for incidents.** When an incident is declared, the team should not be building dashboards. The dashboard platform should automatically create an incident dashboard that shows the relevant metrics at high resolution. This requires pre-configuring incident dashboard templates for each service.

**Lesson 8: Dashboards are the source of truth for deployments.** Every deployment should have a dashboard that shows the health of the new version compared to the old version. This dashboard is the primary tool for deciding whether to proceed with the deployment or roll back. If the dashboard is not ready, the deployment should not proceed.

**Lesson 9: Measure dashboard effectiveness.** Track how long it takes on-call engineers to identify the root cause of an incident using the dashboard. If the average time increases, the dashboard needs to be simplified. If the average time decreases, the dashboard is working. Use this metric to drive dashboard improvements.

## Dashboards as a Cultural Artifact

At the most mature organizations, dashboards are not just monitoring tools. They are cultural artifacts that encode the team's understanding of their system. The dashboard layout reflects what the team believes is important. The metrics on the dashboard reflect what the team believes is healthy. The annotations on the dashboard reflect the team's understanding of how their system behaves.

When a new engineer joins the team, the first thing they should do is study the dashboard. The dashboard tells them: these are the metrics we care about, these are the thresholds that define healthy, and these are the patterns that indicate problems. The dashboard is the team's shared mental model of the system, rendered as a visual layout.

## Industry Benchmarks

Understanding how other organizations approach dashboards helps you calibrate your own designs. The following benchmarks are based on surveys and case studies from large-scale monitoring deployments.

### Case Study: Shopify's Dashboard at Scale

Shopify processes over 1 million requests per second during Black Friday. Their dashboard system must handle this load without breaking, and the dashboards must be useful for the operations team that manages the platform.

**The cardinality challenge.** During Black Friday, Shopify's metrics system sees a massive spike in cardinality. Each new merchant session creates new time series with merchant-specific labels. The series count grows from 10 million to 50 million during the peak. Shopify's TSDB (Thanos) must handle this 5x cardinality increase without crashing.

**The solution: pre-aggregation at the edge.** Shopify's edge servers aggregate metrics before sending them to the central TSDB. Instead of sending one metric per merchant, they aggregate by merchant tier (small, medium, large, enterprise) and send only the aggregated metrics. This reduces the cardinality from 50 million to 5,000, while still providing useful data for the dashboard.

**The dashboard layout.** Shopify's primary dashboard shows the top row as: requests per second, error rate, average response time, and active merchants. The second row shows the top 5 merchant tiers by error rate. The third row shows the infrastructure health (CPU, memory, disk) for the 200 most critical servers. The dashboard is the same for every Black Friday and is exercised in load tests throughout the year.

### The Cost of Bad Dashboards

Bad dashboards have a real, measurable cost to organizations. Understanding this cost helps justify investment in dashboard platform improvements.

**Cost 1: Mean time to resolution (MTTR) impact.** A study of incident response at large tech companies found that engineers spend an average of 15 minutes per incident navigating between monitoring tools and searching for the right dashboard. With 500 incidents per year and an average engineer cost of $150/hour, the annual cost of bad dashboard navigation is: 500 incidents times 15 minutes times $150/hour equals $18,750. For a team of 20 on-call engineers, the cost is $375,000 per year.

**Cost 2: Alert fatigue from unactionable dashboards.** When dashboards do not provide clear signals, teams add more alerts to compensate. Each additional alert increases noise. With an average of $50,000 per year in alert fatigue costs (burnout, missed incidents, high turnover), bad dashboards contribute to a measurable human cost.

**Cost 3: Delayed incident detection.** A bad dashboard that takes 30 seconds to load instead of 2 seconds delays incident detection by 28 seconds. In a high-revenue system processing $10,000 per minute, a 28-second delay in detecting a revenue-impacting incident costs $4,667 per incident. With 20 such incidents per year, the cost is $93,340.

**Cost 4: Lost engineer productivity.** Engineers who build and maintain dashboards spend time that could be spent on product development. A team of 50 engineers each spending 1 hour per week on dashboard maintenance costs: 50 hours times 52 weeks times $150/hour equals $390,000 per year.

**Total estimated cost of bad dashboards for a 50-engineer organization: $850,000-$1,500,000 per year.** A well-designed dashboard platform that reduces MTTR, eliminates alert fatigue, and automates dashboard maintenance can save the organization this amount.

### Team Topologies for Dashboard Platforms

How you organize the team that builds and maintains the dashboard platform determines its success as much as the technical architecture. Three common team topologies exist, each with different trade-offs.

**Centralized platform team.** A dedicated team of 3-5 engineers builds and maintains the dashboard platform. Service teams create and manage their own dashboards using the platform. The platform team handles infrastructure (TSDB scaling, query optimization, upgrades) and provides templates and training. This topology works well for organizations with 100-500 services.

**Decentralized (each team owns their monitoring).** Each service team runs their own Prometheus/Grafana instance. There is no central dashboard platform. This topology works for small organizations (under 50 services) but breaks at scale because there is no cross-service visibility, no consistent dashboard layout, and no shared infrastructure.

**Federated platform team.** A central platform team owns the shared infrastructure (TSDB cluster, Grafana instance, service discovery) and provides self-service tools. Service teams create dashboards through the self-service portal. The platform team focuses on reliability and scalability, while service teams focus on dashboard content. This topology works well for organizations with 500+ services.

**Key success factors for the platform team:**
- The team must have a clear charter: "We provide the platform, you provide the dashboards"
- Self-service is critical: service teams should not need to file a ticket to create a dashboard
- The platform team must measure and communicate their impact: dashboard availability, query latency, number of dashboards created, active dashboard viewers

### Migrating from One Dashboard System to Another

Many organizations eventually outgrow their initial dashboard system and need to migrate. The migration process is delicate because it involves moving the team's monitoring infrastructure without breaking their operational visibility.

**Phase 1: Side-by-side (2-4 weeks).** The new dashboard system runs alongside the old one. Both systems ingest the same metrics. The team creates copies of critical dashboards in the new system. No one stops using the old system yet. This phase validates that the new system produces correct data and has acceptable query latency.

**Phase 2: Gradual adoption (4-8 weeks).** The team starts using the new dashboards for daily operations. The old dashboards remain available but are no longer the primary view. Each team migrates their dashboards at their own pace. The platform team provides migration scripts that convert old dashboard definitions to the new format.

**Phase 3: Old system read-only (2-4 weeks).** The old system stops accepting new data but remains available for historical queries. The team can still access old dashboards for postmortems and trend analysis. This phase lasts until the retention period of the old system's data expires.

**Phase 4: Decommission (1 week).** The old system is shut down. All historical data that is still needed has been migrated to the new system's long-term storage. The DNS records for the old system are removed.

**Key lessons from migration stories:**
- Do not try to migrate all dashboards at once: pick 5 critical dashboards and migrate them first
- Automate the migration: write a converter script that handles 80% of dashboards automatically
- Keep the old system running for at least 30 days after the migration starts: you will discover edge cases that the new system does not handle
- Train the team on the new system before the migration: schedule 1-hour workshops for each team

### Dashboards in Regulated Industries

Regulated industries (finance, healthcare, government) have specific requirements that affect dashboard design. These requirements add complexity but are critical for compliance.

**Audit trails.** Every dashboard view is logged with the user identity, timestamp, and the exact dashboard state at the time of viewing. The audit trail is stored in an immutable, append-only database that is never pruned. Compliance teams can reconstruct exactly what any engineer saw at any point in time.

**Data retention.** Metrics must be retained for specific periods defined by regulation. Payment transaction metrics: 7 years. Healthcare patient data metrics: 6 years. Financial market data metrics: 5 years. The dashboard platform must support variable retention policies per metric namespace, with no metric deleted before its retention period expires.

**Immutable dashboards.** In some regulated environments, dashboards used for compliance reporting cannot be modified after they are created. The dashboard platform supports a "freeze" feature that makes a dashboard read-only with a cryptographic hash of its definition. Any attempt to modify the frozen dashboard is blocked and logged.

**Segregation of duties.** The engineer who creates a dashboard cannot be the same engineer who approves it for production use. The dashboard platform enforces a two-person rule: dashboard changes go through a review and approval workflow, and the reviewer must be different from the author.

### Open Source vs Commercial Decision Framework

| Factor | Open Source (Prometheus + Grafana) | Commercial (Datadog, Grafana Cloud) |
|--------|-----------------------------------|-------------------------------------|
| Upfront cost | Free (infrastructure cost only) | Per-host or per-series pricing |
| Operational overhead | High (you manage upgrades, patches, scaling) | Low (vendor manages the platform) |
| Customization | Full access to source code | Limited to API surface |
| Support | Community forums, GitHub issues | Vendor SLA, dedicated support |
| Feature velocity | Slower (depends on community contributions) | Faster (vendor releases quarterly) |
| Data residency | You control where data is stored | Data is stored in vendor's regions |
| Integration depth | Deep (you can write custom exporters) | Good (vendor SDK, but limited) |

**When to choose open source:** You have a dedicated SRE team, you need full control over data residency, you have specific compliance requirements, or you want to avoid vendor lock-in.

**When to choose commercial:** You have a small team with limited ops bandwidth, you need fast time-to-value, or you want to avoid managing your own infrastructure.

## Building a Dashboard Culture

The most successful dashboard implementations share a common trait: they have a dashboard culture. The organization values dashboards, uses them daily, and continuously improves them.

**Characteristics of a dashboard culture:**
- Every team starts their daily standup by looking at their dashboard
- On-call engineers are trained on dashboard reading as part of onboarding
- Dashboard changes are reviewed in the same way as code changes
- Dashboards are included in incident postmortems as evidence
- The dashboard platform team has a budget for improvements

**How to build a dashboard culture:**
1. Start with a small number of high-quality dashboards. A team that has 5 useful dashboards will use them. A team that has 50 mediocre dashboards will ignore them.
2. Make dashboards the default view. When an engineer opens a monitoring tool, the first thing they see should be the dashboard, not a blank page or a search bar.
3. Celebrate good dashboards. Recognize teams that create useful dashboards in team meetings. Share dashboard screenshots in the team chat.
4. Invest in dashboard training. Every new engineer should spend 1 hour learning how to read and create dashboards. The training should cover the four golden signals, the F-pattern layout, and the dashboard-as-code workflow.

## Dashboard Metrics That Matter

To measure whether your dashboard platform is effective, track these metrics:

**Dashboard adoption rate:** What percentage of services have a dashboard? A dashboard platform with 80%+ adoption is healthy. Below 50% indicates that teams do not find the platform useful.

**Dashboard freshness:** What percentage of dashboards have been updated in the last 90 days? A dashboard that has not been updated in 90 days is likely stale. Below 50% freshness indicates that the platform needs lifecycle management.

**Dashboard load time:** How long does the average dashboard take to load? A dashboard that takes longer than 5 seconds to load is not used. The target is p99 load time under 3 seconds.

**Dashboard-to-incident ratio:** How many dashboards are viewed during an incident? A high ratio (10+ dashboards per incident) indicates that the dashboards are not providing enough information. The target is 2-3 dashboards per incident: the overview dashboard, the service dashboard, and the debugging dashboard.

**Dashboard query efficiency:** How many TSDB queries per dashboard view? A high ratio (50+ queries per view) indicates that the dashboard is making too many queries. The target is 12-15 queries per view (one per panel).
