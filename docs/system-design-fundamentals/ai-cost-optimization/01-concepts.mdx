## The Problem Story

You just launched an AI-powered writing assistant. Within a week, you have 1,000 users. Within a month, 25,000. By day 60, you hit 100,000 active users. Growth is everything you dreamed of. Investors are calling. Product-market fit is real.

Then the AWS bill arrives.

$340,000 for LLM API calls alone. Not including hosting. Not including the vector database. Not including the engineering team. Just tokens -- input tokens and output tokens flowing through OpenAI's API.

You check your bank account. You raised a $3 million seed round. At this burn rate, you have 8 months of runway. If growth continues, you have 4 months. Your CTO runs the numbers again. And again. The math does not change: at 200,000 users, the bill would be $680,000. At 500,000 users, $1.7 million. You are growing fast and losing money on every user.

The writing assistant is excellent. Users love it. Your retention numbers are industry-leading. But the unit economics are inverted. Every $10 monthly subscription costs you $18 in API calls. You are paying users to use your product.

This is the cold math of AI cost optimization. You built a great product. The market wants it. But the underlying infrastructure costs can destroy your business if you do not manage them aggressively. The question is not whether you can build the AI capability. It is whether you can deliver it at a cost the market will bear.

## The Insight

AI cost optimization is the practice of systematically reducing the cost of LLM inference without sacrificing quality. It is not about cutting corners. It is about spending money where it matters and saving where it does not.

Think of a manufacturing factory. You have different machines for different jobs. A precision CNC machine can make any part, but it costs $200 per hour to run. A basic stamping press costs $20 per hour and can make simple parts just as well. If you use the CNC machine for every part, you burn cash. You need a system that routes each job to the right machine, buys materials in bulk for volume discounts, and designs parts to minimize waste.

LLM inference works the same way. You have expensive models (GPT-4, Claude 3) that can handle any task. You have cheap models (GPT-3.5, Llama 3 8B) that handle simple tasks well. You have caching, batching, and compression techniques that reduce waste. The art of cost optimization is knowing when to use each tool.

The core principle is: match the cost of computation to the value of the response. A customer support query about your return policy does not need a GPT-4 reasoning chain. It needs a fast, cheap answer from a small, fine-tuned model. A complex legal document review? That might justify the expensive model. The key is having the infrastructure to make this distinction automatically, at scale, with sub-50ms overhead.

## Core Vocabulary

**Cost per token** -- The fundamental unit of LLM pricing. Expressed as price per 1,000 or per 1,000,000 tokens. Typical costs: $0.15 to $30 per million input tokens, $0.60 to $60 per million output tokens depending on model tier.

**Input tokens** -- The tokens in your prompt, context, and instructions. You pay for these on every request. Long context windows (4,000 to 128,000 tokens) make input tokens the dominant cost driver in many applications.

**Output tokens** -- The tokens the model generates. Typically 2x to 4x more expensive than input tokens because generation requires sequential compute with no batching efficiency gains.

**Model tier** -- The category of model based on capability and cost. Tiers range from small open-weight models (free to self-host, $0.50-$2 per hour GPU cost) through mid-tier APIs (GPT-3.5, Claude Haiku: $1-$3/M tokens) to frontier models (GPT-4, Claude Opus: $30-$60/M tokens).

**Prompt compression** -- The technique of reducing prompt size by removing irrelevant context, truncating verbose instructions, or using learned compression. Can reduce token usage by 40-60% without affecting quality on most tasks.

**Context window** -- The maximum number of tokens a model can process in a single request. Larger context windows (128K tokens for some models) enable richer prompts but dramatically increase cost per request.

**Batching** -- Grouping multiple independent requests into a single API call. Many providers offer lower per-token pricing for batch processing (typically 50% discount). Batch processing is suitable for async workloads where throughput matters more than latency.

**Caching** -- Storing previous LLM responses and reusing them for identical or similar requests. Exact-match caching is simple and effective. Semantic caching (using embeddings to find similar queries) extends the hit rate to near-duplicate requests. Both reduce API calls to zero for cached requests.

**Rate limiting** -- Controlling the number of API requests per time window. Rate limits prevent cost spikes from runaway retry loops, abusive users, or misconfigured clients. They are a first line of defense against unexpected cost overruns.

**Cost allocation** -- The practice of attributing AI infrastructure costs to specific teams, features, or users. Enables chargeback models, budget accountability, and identification of cost optimization opportunities at a granular level.

**Chargeback** -- A financial model where each product team is billed for its actual AI infrastructure usage. Creates accountability and incentivizes cost-conscious behavior. Teams that would waste expensive model calls on trivial tasks suddenly optimize when they see the cost on their budget.

**TCO (Total Cost of Ownership)** -- The complete cost of an AI system including API fees, infrastructure, engineering time, maintenance, and opportunity cost. A cheaper API model may have higher TCO if it requires more prompt engineering and monitoring. A self-hosted model has lower per-token costs but higher infrastructure TCO.

**Token budget** -- A predefined limit on the number of tokens a feature, team, or user can consume in a given period. Enables cost predictability and prevents budget overruns.

**Cost-aware routing** -- Dynamically selecting which model to use for each request based on the cost-quality tradeoff. A classifier determines the complexity of the request and routes accordingly. Savings of 40-70% are typical.

## How It Actually Works -- Step by Step

### Step 1: Request Classification

Every incoming request is classified by complexity. A lightweight classifier -- typically a small ML model (DistilBERT) or a set of heuristics -- assigns a complexity score. Simple requests include: factual queries, template-based generation, straightforward summarization. Complex requests include: multi-step reasoning, code generation, creative writing, ambiguous queries with no clear answer.

The classifier is trained on a labeled dataset of past queries with their correct model tiers, and it is retrained monthly as query patterns evolve.

### Step 2: Model Router Decision

Based on the classification, the model router selects a target model. The router maintains a routing table with model tiers, pricing, latency guarantees, and current load. It applies two additional constraints: user tier (premium users get higher-quality models) and budget status (if the daily budget is nearly exhausted, even complex queries may be downgraded).

### Step 3: Prompt Optimization

Before sending the prompt to the model, the prompt optimizer reduces its size. Techniques include:

- **Context truncation**: Remove irrelevant retrieved documents. If the RAG system returned 10 documents but only 3 are relevant, drop the rest.
- **Instruction compression**: Replace verbose instructions with concise versions. "Please carefully analyze the following text and provide a detailed summary covering the main points" becomes "Summarize the text."
- **Semantic compression**: Use a small model to rewrite the prompt in fewer tokens while preserving meaning.
- **Dynamic context window**: Limit the context to the minimum needed for the task. If the task only needs the last 3 messages of conversation history, do not send all 50.

Prompt compression can reduce token count by 40-60% on most applications. The latency added by compression is 5-20ms, dwarfed by the latency of the LLM call itself (500ms to 5 seconds).

### Step 4: Cache Lookup

The request (now compressed) goes through the cache layer. Three cache levels are checked in order:

1. **Exact-match cache** (Redis): Keys are the full prompt string. TTL varies by use case: 1 hour for customer support, 24 hours for documentation lookups.
2. **Semantic cache** (vector database): The query is embedded and compared against cached query embeddings. If cosine similarity exceeds a threshold (typically 0.92), the cached response is returned.
3. **Prefix cache**: For streaming applications where prompts share a prefix (system prompt + recent context), the prefix computation can be reused across requests.

Cache hit rates vary by application: 20-40% for open-ended chat, 40-60% for customer support, 60-80% for template-driven generation.

### Step 5: Batch Decision

If the request is not urgent -- batch tolerance is configurable per feature -- the batch scheduler queues it. The scheduler waits for either a batch size threshold (8 requests) or a max wait time (5 seconds), whichever comes first. Batch processing typically receives a 50% discount from providers.

### Step 6: Inference and Cost Tracking

The request is sent to the selected model provider. The inference engine wraps the API call with cost tracking instrumentation. Each call emits a structured log containing: provider, model, input tokens, output tokens, latency, cost calculated from provider pricing, request ID, user ID, feature ID, and team ID.

This cost tracking data flows to a time-series database (Prometheus or InfluxDB) for real-time dashboards and to a data warehouse (Snowflake or BigQuery) for historical analysis.

### Step 7: Budget Monitoring

Every minute, the budget monitor checks current cost against budget thresholds:

- Below 50% of budget: normal operation
- 50-80% of budget: send summary notification to team
- 80-90% of budget: degrade non-critical requests to cheaper models
- 90-95% of budget: warn all requests to budget-conscious routing
- 95-100% of budget: throttle non-premium requests, alert on-call engineer
- Over 100% of budget: block all non-critical requests, page engineering leadership

### Step 8: Feedback Loop

The cost optimization system continuously learns. Usage patterns change. New models are released. Provider pricing changes. The feedback loop adjusts routing rules, cache TTLs, compression thresholds, and budget allocations automatically or through human review. Weekly reports show cost trends by feature, team, and model tier.

## Variants and Flavors

### Model-Level Optimization (Right-Sizing)

Route each request to the cheapest model that can handle it correctly. This is the highest-impact optimization for most applications. Implementation requires a classifier, a routing table, and a quality monitoring system.

Typical model tiers:

| Tier | Model | Cost per M output tokens | Use Case |
|---|---|---|---|
| Free | Self-hosted Llama 3 8B | $0.50/hour GPU | Simple classification, extraction |
| Budget | GPT-3.5 / Claude Haiku | $2 | FAQs, summarization, templates |
| Standard | GPT-4o mini / Claude Sonnet | $8 | Customer support, content generation |
| Premium | GPT-4 / Claude Opus | $60 | Complex reasoning, code, legal |

### Prompt-Level Optimization (Compression and Caching)

Reduce the number of tokens sent with each request and avoid redundant computation. Prompt compression reduces per-request tokens by 40-60%. Caching eliminates entire requests. These two optimizations compound: compressed prompts take less cache space and increase cache hit rates.

### Infrastructure-Level Optimization (Batching and Quantization)

Optimize the execution environment. Batch processing gets 50% provider discounts. Model quantization (FP16 to INT8) reduces GPU memory and compute requirements by 50% with minimal quality loss. Self-hosting with quantized models on dedicated GPUs can reduce per-token costs by 10x compared to API models.

### Business-Level Optimization (Tiered Access)

Let your business model drive cost optimization. Free tier users get cheap models with caching. Premium users get the best models with no caching restrictions. Enterprise users get self-hosted dedicated infrastructure. This aligns cost structure with revenue and ensures that the users who generate the most revenue get the best experience.

## Mental Model Check

**Q1: A customer support query about refund timing and a complex code review request arrive simultaneously. Which should use the expensive model, and why?**

A1: The code review should use the expensive model. Customer support queries have known answers that can be cached or handled by a smaller fine-tuned model. Code review requires understanding context, identifying subtle bugs, and generating accurate suggestions -- tasks where a large model significantly outperforms small models.

**Q2: Your caching system has a 35% hit rate. A colleague suggests increasing cache TTL from 1 hour to 24 hours to improve hit rate. What is your response?**

A2: Increasing TTL improves hit rate but introduces stale-data risk. If accuracy requirements allow 24-hour-old responses, do it. If the domain changes frequently (stock prices, news, availability), shorter TTLs are safer. A better approach is tiered TTLs: short TTL for time-sensitive content, long TTL for evergreen content (policies, documentation).

**Q3: Why would you ever use a more expensive model if a cheaper one exists?**

A3: Quality. Cheaper models make more errors on complex tasks. A 5% error rate on code generation could introduce bugs that cost more to fix than the savings from using a cheaper model. The expensive model is an insurance policy against quality degradation. The goal is to use the cheapest model that meets the quality bar for each specific task.

**Q4: A single request costs $0.05 to process. Describe three distinct ways to reduce this cost.**

A4: (1) Cache the response if a similar query was answered before -- reduces cost to $0.00 or the cache lookup cost. (2) Compress the prompt from 2,000 tokens to 800 tokens -- reduces cost by 60%. (3) If the task is simple, route to a cheaper model that costs $0.01 per request instead of $0.05.

**Q5: Your budget monitor shows that Team A has consumed 95% of their monthly AI budget in the first 10 days. What do you do?**

A5: First, alert Team A automatically with their cost breakdown. Second, check if their usage pattern has changed or if there is a bug (retry loop, misconfigured client). Third, degrade their requests to cheaper models for the remainder of the month. Fourth, schedule a review with Team A to understand their needs and adjust their budget for the next month if necessary. Fifth, add proactive alerts to catch this earlier (at 50% and 80% of budget) for future months.

## Deep Dive: The Economics of Token Pricing

Understanding why LLM providers charge what they do helps you make better optimization decisions. Token pricing is not arbitrary. It reflects the underlying cost of computation.

### Input Token Pricing

Input tokens are processed in parallel. When you send a prompt, the model computes attention over all input tokens simultaneously. This is compute-bound: the cost is proportional to the number of input tokens times the model size. Providers charge $2-$30 per million input tokens depending on the model.

The parallelism means that input processing is efficient. A single GPU can process thousands of input tokens in a single forward pass. This is why input tokens are cheaper than output tokens.

### Output Token Pricing

Output tokens are generated sequentially. The model produces one token at a time, using the previous token as input. This is memory-bound: the bottleneck is reading the KV cache and model weights. A single GPU generates 20-100 tokens per second for a large model.

The sequential nature means that output processing is inefficient. The GPU is idle while waiting for the next token to be generated. This is why output tokens are 2-4x more expensive than input tokens.

### The Cost of the KV Cache

The KV cache grows with sequence length. For a 4,000-token sequence with a 70B parameter model, the KV cache is approximately 4 GB per request. For a batch of 8 requests, the KV cache is 32 GB -- more than most GPUs have. This memory pressure limits batch sizes and drives up cost.

### Why Prompt Compression Saves Money

Prompt compression reduces input tokens, which are cheap. But it also reduces the KV cache size, which enables larger batches. A compressed prompt from 4,000 to 1,600 tokens reduces KV cache memory by 60%. This allows the inference system to batch more requests, improving throughput and reducing per-request cost.

The savings from compression are not just the direct token savings. The indirect savings from improved batch efficiency are often larger.

## Real-World Numbers: Cost Breakdown by Application

The cost structure of an AI application depends heavily on the use case. Here are three common patterns:

### Pattern 1: Customer Support Chatbot

In this pattern, users ask short questions and get short answers. The input is typically 500-1,000 tokens (including conversation history). The output is 100-200 tokens. The ratio of input to output is 5:1 to 10:1.

Cost breakdown using GPT-4:
- Input cost per request: 1,000 tokens x $30 / 1,000,000 = $0.03
- Output cost per request: 150 tokens x $60 / 1,000,000 = $0.009
- Total: $0.039 per request
- At 1 million requests per month: $39,000

Optimization opportunity: Caching is highly effective because customer support queries are repetitive. A 50% cache hit rate reduces cost to $19,500. Model routing to GPT-3.5 for simple queries reduces cost further to $5,000-$10,000.

### Pattern 2: RAG Document Analysis

In this pattern, users ask questions about a document corpus. The input is large (2,000-8,000 tokens including retrieved documents). The output is moderate (200-500 tokens). The input cost dominates.

Cost breakdown using GPT-4:
- Input cost per request: 5,000 tokens x $30 / 1,000,000 = $0.15
- Output cost per request: 300 tokens x $60 / 1,000,000 = $0.018
- Total: $0.168 per request
- At 500,000 requests per month: $84,000

Optimization opportunity: Prompt compression is the highest-impact optimization. Reducing input from 5,000 to 2,000 tokens saves 60% on input costs. Improved retrieval quality (fewer irrelevant documents) compounds the savings.

### Pattern 3: Code Generation Assistant

In this pattern, users provide code context and ask for completions. The input is moderate (1,000-2,000 tokens). The output is variable (50-500 tokens). The output cost can be significant because code generation produces long responses.

Cost breakdown using GPT-4:
- Input cost per request: 1,500 tokens x $30 / 1,000,000 = $0.045
- Output cost per request: 300 tokens x $60 / 1,000,000 = $0.018
- Total: $0.063 per request
- At 2 million requests per month: $126,000

Optimization opportunity: Batching is highly effective because code completions are short and independent. A smaller, fine-tuned model for common patterns (boilerplate, simple functions) can handle 70% of requests at 10% of the cost.

## Building a Cost-Aware Culture

Technical optimization is only half the battle. The other half is building a culture where every engineer thinks about cost.

### Cost Visibility

Make cost data visible to everyone. Every engineer should be able to answer: "How much does my feature cost per request?" Publish a cost dashboard that shows cost per feature, per team, and per model. Send weekly cost summaries to each team.

### Cost Accountability

Assign cost ownership to each feature team. The team that builds the feature is responsible for its cost. This creates natural incentives for optimization. When a team sees their feature cost increasing, they investigate and optimize.

### Cost Gates in CI/CD

Add cost estimation to the CI/CD pipeline. When a developer submits a change that introduces a new AI feature, the pipeline estimates the cost impact and requires approval if the cost exceeds a threshold. This catches expensive features before they reach production.

### Cost Reviews

Schedule quarterly cost reviews with each team. Review cost trends, optimization opportunities, and budget allocations. Celebrate teams that reduced costs. Investigate teams with increasing costs.

## The Unit Economics of AI Features

Understanding the unit economics of your AI features is essential for pricing and budgeting. The cost per request determines whether your product is profitable.

### Calculating Unit Cost

The unit cost of an AI feature has three components:

1. **Inference cost**: The direct cost of the LLM API call. This is the dominant cost for most applications.
2. **Infrastructure cost**: The cost of running the application, database, cache, and monitoring systems. Allocate this across features based on usage.
3. **Engineering cost**: The cost of building and maintaining the feature. Amortize this over the expected lifetime of the feature.

### Example: Writing Assistant

A writing assistant feature generates 200-word responses. Each request uses 1,000 input tokens and 250 output tokens.

- Inference cost: (1,000 x $30 + 250 x $60) / 1,000,000 = $0.045 per request
- Infrastructure cost (allocated): $0.005 per request
- Amortized engineering cost: $100,000 development / 10 million lifetime requests = $0.01 per request
- Total unit cost: $0.06 per request

If the product charges $10/month and users make 200 requests per month, the cost per user is $12. The product loses $2 per user. Cost optimization must reduce the unit cost by at least 17% to break even.

### Target Cost per Request

For different pricing models, the target cost per request varies:

| Pricing Model | Monthly Revenue | Target Cost per Request (at 70% margin) |
|---|---|---|
| $10/month, 100 requests | $10 | $0.07 |
| $10/month, 500 requests | $10 | $0.014 |
| $50/month, 500 requests | $50 | $0.07 |
| $50/month, 2,000 requests | $50 | $0.0175 |

The target cost per request determines which optimization strategies are viable. For a $0.07 target, GPT-4 with caching may work. For a $0.014 target, you need model routing and compression. For a $0.005 target, you need self-hosting.

## The Mental Model in Practice

Here is a step-by-step approach to applying the cost optimization mental model to any new AI feature:

### Step 1: Estimate Baseline Cost

Calculate the cost of the feature using the default model (typically the best model available). Use the formula: cost per request = (input tokens x input price + output tokens x output price) / 1,000,000. Multiply by expected daily volume.

### Step 2: Identify Optimization Opportunities

Analyze the request characteristics:
- Are requests repetitive? Yes -> caching is viable.
- Are requests mixed complexity? Yes -> model routing is viable.
- Are requests long-context? Yes -> prompt compression is viable.
- Are requests non-urgent? Yes -> batching is viable.

### Step 3: Apply Optimization in Order

Apply optimizations in order of ROI and risk:
1. Exact-match caching (always, no risk)
2. Rate limiting (always, no risk)
3. Model routing (highest ROI, medium risk)
4. Prompt compression (medium ROI, low risk)
5. Semantic caching (medium ROI, low risk)
6. Batching (medium ROI, low risk, requires latency tolerance)
7. Self-hosting (highest ROI, highest risk, last resort)

### Step 4: Measure and Iterate

After implementing each optimization, measure the actual cost savings. Compare to the estimated savings. Adjust parameters (cache TTL, routing thresholds, compression ratios) based on real data. Add monitoring to detect regression.

### Step 5: Set Budget and Alerts

Set a budget for the feature at 80% of the estimated cost. Configure alerts at 50%, 75%, 90%, and 100% of budget. Assign cost ownership to the feature team. Include cost in the feature's operational review.
