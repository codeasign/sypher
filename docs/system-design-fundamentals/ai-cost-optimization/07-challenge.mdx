## How to Use This Section

Each challenge includes a scenario, requirements, constraints, and evaluation criteria. You should aim to complete Foundation challenges within 10-15 minutes, Design challenges within 30-45 minutes, and Expert challenges within 60-90 minutes. After attempting each challenge, review the evaluation criteria to identify areas for improvement. Model answers are provided at the end so you can compare your approach.

## Foundation Tier

### Challenge F1: Cost Baseline Calculation

**Scenario:** Your AI application processes 200,000 requests per day. Each request uses 1,500 input tokens and 300 output tokens. You use GPT-4 at $30 per million input tokens and $60 per million output tokens. You also pay $0.02 per request for a classification model that determines request complexity.

**Tasks:**
1. Calculate the daily cost of GPT-4 inference.
2. Calculate the daily cost of the classification model.
3. Calculate the total monthly cost (30 days).
4. Determine the cost per request.
5. Identify the percentage of total cost that comes from input tokens, output tokens, and classification.

**Evaluation criteria:**
- All calculations must be correct and shown step by step.
- Cost per request must account for all components.
- Percentage breakdown must sum to 100%.
- You should identify that input tokens are the dominant cost driver.

**Model answer:**
- GPT-4 input cost per request: (1,500 x $30) / 1,000,000 = $0.045
- GPT-4 output cost per request: (300 x $60) / 1,000,000 = $0.018
- Classification cost per request: $0.02
- Total cost per request: $0.045 + $0.018 + $0.02 = $0.083
- Daily cost: 200,000 x $0.083 = $16,600
- Monthly cost: $16,600 x 30 = $498,000
- Breakdown: input tokens 54.2%, output tokens 21.7%, classification 24.1%

### Challenge F2: Cache Hit Rate Analysis

**Scenario:** You are implementing a semantic cache. You estimate that cache infrastructure costs $3,000 per month. The cache has a 40% hit rate. Without caching, your daily API cost is $12,000. The cache adds 100ms latency on cache hits and 200ms latency on cache misses (for embedding computation).

**Tasks:**
1. Calculate the monthly savings from caching.
2. Calculate the net monthly benefit (savings minus cache infrastructure cost).
3. Determine the break-even hit rate where caching becomes worthwhile.
4. Estimate the p95 latency change when the cache is introduced.

**Evaluation criteria:**
- Net benefit calculation must subtract infrastructure costs.
- Break-even analysis must be mathematically correct.
- Latency impact must consider both hit and miss scenarios.
- You should discuss whether the latency tradeoff is acceptable.

**Model answer:**
- Monthly API cost without cache: $12,000 x 30 = $360,000
- Monthly savings: $360,000 x 0.40 = $144,000
- Net monthly benefit: $144,000 - $3,000 = $141,000
- Break-even hit rate: $3,000 / $360,000 = 0.83% hit rate
- With 40% hit rate and 100ms hit latency / 200ms miss latency, p95 depends on the original p95. If original p95 was 1,500ms, the cache adds 100ms on 40% (1,600ms effective) and 200ms on 60% (1,700ms effective). Weighted average: 1,660ms.

### Challenge F3: Model Selection Decision

**Scenario:** You need to choose between two deployment strategies for serving 1 million requests per day. Each request averages 800 input tokens and 150 output tokens.

Option A: Use GPT-4 exclusively. $30/M input tokens, $60/M output tokens. No infrastructure cost.
Option B: Fine-tune and self-host Llama 3 70B. Training cost: $50,000 (one-time). Hosting cost: $5 per hour for GPU instances. Model runs on 4 x A100 GPUs, each at $2 per hour.

**Tasks:**
1. Calculate the monthly cost of Option A (30 days).
2. Calculate the monthly cost of Option B, amortizing training over 12 months.
3. Determine the break-even point in months where Option B becomes cheaper.
4. List three non-financial factors that might influence the decision.

**Evaluation criteria:**
- Amortization must be correctly calculated.
- Break-even analysis must account for both fixed and variable costs.
- Non-financial factors must be substantive.

**Model answer:**
- Option A monthly: 1,000,000 x ((800 x $30 + 150 x $60) / 1,000,000) x 30 = $990,000
- Option B monthly hosting: 4 x $2 x 24 x 30 = $5,760. Monthly ops: $5,760 + (100% overhead for engineering): total $11,520. Amortized training: $50,000 / 12 = $4,167. Total: $15,687.
- Break-even: $50,000 + $15,687 x M = $990,000 x M -> $50,000 = $974,313 x M -> M = 0.05 months (about 1.5 days)
- Non-financial factors: quality differences (Llama 3 vs GPT-4), maintenance burden of self-hosting, ability to customize the fine-tuned model, vendor lock-in risk

## Design Tier

### Challenge D1: Multi-Tier Cost Optimization System

**Scenario:** You work at a company building an AI-powered code review assistant. The system handles 50,000 code reviews per day. Each review processes 10 files on average. Each file is 200 lines of code (approximately 1,000 tokens when formatted as a prompt). The output is a review comment averaging 100 tokens per file.

The system currently uses GPT-4 for all reviews. Monthly API cost is $675,000. The CEO wants to reduce this by 60% within 3 months without significantly degrading review quality.

**Requirements:**
- Reduce costs by 60% from baseline.
- Response time for each review must stay under 30 seconds.
- Review quality must not decrease for complex code changes.
- The solution must be implementable within 3 months.

**Design tasks:**
1. Design the full cost optimization architecture. Include: model routing, caching, prompt compression, and any other techniques you consider.
2. Estimate the cost savings from each component and show they total at least 60%.
3. Describe the model routing strategy: what classifier do you use, how do you determine complexity, what are the fallback behaviors?
4. Design the caching strategy: what cache layers, what eviction policy, estimated hit rates, invalidation strategy.
5. Describe the monitoring and feedback system that ensures quality does not degrade.
6. Identify the biggest risk to your plan and how you mitigate it.

**Evaluation criteria:**
- Total savings calculation must sum component savings to at least 60%.
- Each savings estimate must be justified with reasoning.
- The routing classifier must be practical (not requiring another LLM just to classify).
- The caching strategy must include hit rate estimates and invalidation logic.
- Quality monitoring must include concrete metrics and thresholds.
- Risk mitigation must be specific, not generic.

### Challenge D2: Cost Allocation and Chargeback System

**Scenario:** Your organization has 12 product teams sharing a centralized AI infrastructure. The total monthly AI cost is $2.4 million. Teams are complaining that costs are allocated unfairly. Some teams use very large models for small tasks because "it is free to them." Other teams are being charged for shared infrastructure they do not use.

**Requirements:**
- Design a cost allocation system that attributes costs to each team accurately.
- Each team must have a monthly budget with enforcement.
- Teams should be able to see their real-time cost dashboard.
- The system must handle shared costs (base models, cache infrastructure, rate limit overhead).
- Budget enforcement must not cause production outages.

**Design tasks:**
1. Design the data model for cost attribution. What dimensions do you track?
2. Design the cost allocation algorithm. How do you handle shared costs?
3. Design the budget enforcement mechanism. What happens when a team exceeds budget?
4. Design the real-time cost dashboard schema and API.
5. Describe how you handle edge cases: concurrent requests at budget boundaries, retroactive price changes from providers, team restructures.

**Evaluation criteria:**
- The cost allocation model must handle both direct and shared costs.
- Budget enforcement must include soft limits (alerting) and hard limits (throttling) with grace periods.
- The dashboard must update within 1 minute of an API call completing.
- Edge case handling must be specific and practical.

### Challenge D3: Real-Time Cost Anomaly Detection at Scale

**Scenario:** Your AI platform processes 50,000 requests per minute across 200 different model endpoints. You need to detect cost anomalies in real time. A recent incident where a retry loop burned $80,000 in 15 minutes should never happen again.

**Requirements:**
- Detect cost anomalies within 1 minute of their start.
- False positive rate under 5%.
- The system must handle seasonal patterns (higher usage during business hours).
- Alerts must include root cause diagnosis information.
- The system must scale to 2x current volume.

**Design tasks:**
1. Design the anomaly detection algorithm. Justify your choice of statistical method.
2. Design the data pipeline for real-time cost metrics. Include ingestion, aggregation, and storage.
3. Design the alerting system: thresholds, escalation policies, notification channels.
4. Describe how you handle false positives and tune the system over time.
5. Design a runbook template for cost anomaly response.

**Evaluation criteria:**
- The anomaly detection algorithm must handle seasonality.
- The data pipeline must support sub-minute latency.
- Alerting must include automatic root cause analysis.
- False positive handling must include a feedback loop.
- The runbook template must be actionable, not theoretical.

## Expert Tier

### Challenge E1: Global Multi-Provider Cost Optimization

**Scenario:** You are building a global AI platform serving 100 million requests per day across North America, Europe, and Asia-Pacific. You have access to six model providers: OpenAI, Anthropic, Google (GCP Vertex AI), Amazon (AWS Bedrock), Microsoft (Azure OpenAI), and a self-hosted cluster of open-weight models running on your own hardware in three regions.

Each provider has different pricing in each region, different available models, different latency profiles, and different rate limits. Data transfer costs between regions add $0.02 per GB. Provider pricing changes quarterly. New models are released monthly.

**Requirements:**
- Minimize total cost across all providers and regions.
- p95 latency must stay under 3 seconds in all regions.
- No single provider should handle more than 40% of traffic (to avoid vendor lock-in).
- The system must automatically adapt to pricing changes without manual intervention.
- Cost per request must decrease by 10% quarter over quarter.

**Design tasks:**
1. Design the global routing architecture. How does a request in Tokyo get routed to the optimal provider?
2. Design the pricing data model that supports automatic updates and multi-region comparisons.
3. Design the dynamic load balancing algorithm that considers price, latency, and lock-in constraints.
4. Design the fallback strategy when a provider in a region becomes unavailable or expensive.
5. Describe how you measure and enforce the 10% quarterly cost reduction.
6. Identify the top three risks to your architecture and how you mitigate them.

**Evaluation criteria:**
- The routing architecture must minimize cross-region data transfer.
- The pricing data model must support versioned price updates.
- The load balancing algorithm must satisfy all constraints simultaneously.
- The quarterly cost reduction must be tracked with a clear mechanism.
- Risk mitigations must be specific to global multi-provider systems.

### Challenge E2: Cost-Driven Model Architecture Evolution

**Scenario:** Your company has an AI product that has grown from 10,000 to 10 million daily active users over 18 months. The AI cost has grown from $50,000 to $5 million per month. The current architecture uses a single large model (GPT-4 class) for all features. The product has 15 distinct features, each with different quality requirements and willingness to pay.

The company has raised Series C and needs to demonstrate a path to gross margin targets. The board has mandated that AI infrastructure costs must not exceed 20% of revenue. Current revenue is $15 million per month with 40% gross margin before AI costs.

**Requirements:**
- Design a model architecture evolution plan that achieves the gross margin target within 12 months.
- The plan must include intermediate milestones with measurable cost reductions.
- You may introduce new models, fine-tune existing models, or build custom small models.
- The plan must not reduce quality for premium features.
- The plan must account for engineering effort and opportunity cost.

**Design tasks:**
1. Analyze the current cost structure and identify the biggest opportunities.
2. Design a tiered model architecture for the 15 features. Map each feature to a model tier.
3. Propose a timeline for the evolution with milestones at months 1, 3, 6, and 12.
4. Calculate the projected cost at each milestone and show the path to the target.
5. Describe the quality validation process for each model change.
6. Identify what could cause the plan to fail and what contingencies you have.

**Evaluation criteria:**
- The cost analysis must identify specific features and their current costs.
- The model tiering must be justified by feature quality requirements.
- The timeline must be realistic about engineering capacity.
- Cost projections must be detailed and show month-over-month improvement.
- The quality validation process must include automated and manual evaluation.
- Failure analysis must identify specific, plausible failure modes.

## Reflection Questions

After completing each challenge, ask yourself:
- Did I consider the organizational and process implications, or only the technical ones?
- Did I quantify my savings estimates, or did I rely on vague percentages?
- Did I identify tradeoffs for each optimization I proposed?
- Did I consider what happens when the optimization fails?
- Did I account for the cost of the optimization infrastructure itself?
- Would my solution still work if request volume doubled?
- Would my solution still work if model prices changed by 50%?
- Did I consider the user experience impact of my cost optimizations?

## Quick Practice Scenarios

These mini-scenarios are designed for 5-minute practice sessions. Read the scenario, think through your answer, then check the key considerations.

### Scenario 1: The Viral App

Your AI app goes viral. User growth jumps from 10,000 to 100,000 DAU in a week. Your AI costs jumped from $15,000 to $150,000 per month. What is the first thing you do?

**Key considerations:** Do not panic-optimize. First, check if the growth is sustainable. If yes, negotiate a volume discount with your provider immediately (minimum 15-30% savings for 10x volume). Then implement caching (quickest ROI). Do not start a 3-month self-hosting project during a growth crisis.

### Scenario 2: The Quality Complaints

After implementing model routing, your support team reports a 20% increase in quality complaints. Users say the AI is "getting dumber." Your cost savings are 55%. What do you do?

**Key considerations:** Check the classifier accuracy first. Are complex queries being misrouted to cheap models? If yes, increase the confidence threshold. Implement a re-routing fallback for low-confidence cheap model responses. Consider whether some features should be excluded from routing entirely. Measure the cost of the quality complaints (churn, support time) against the savings.

### Scenario 3: The Budget Standoff

Your CFO wants to cut the AI budget by 30% because the company is not profitable. Your engineering team says they need the current budget to maintain quality. How do you resolve this?

**Key considerations:** A 30% cut is achievable with existing optimization techniques. Propose a plan: implement model routing (saves 40%) and prompt compression (saves 20%) with a quality monitoring guardrail. The net result: 30% savings with minimal quality impact for most users. Agree to a 2-week trial with the cheaper models and meet again to review quality metrics.

### Scenario 4: The New Model Release

A new model is released by a competitor. It costs 50% less than your current model and claims equivalent quality. What is your process for evaluating it?

**Key considerations:** Do not switch immediately. Run a blind A/B test: route 5% of traffic to the new model and compare quality metrics (automated evaluation, user feedback, task completion rates). Test on all feature types, not just the most common ones. Check for edge cases: long context performance, language support, formatting consistency. Only switch after 2 weeks of positive results.

### Scenario 5: The Multi-Cloud Lock-In

Your company has been using one cloud provider for all AI inference. The provider just announced a 25% price increase. You are locked into a 12-month contract. What do you do?

**Key considerations:** First, check if the contract has a price protection clause. Many enterprise contracts freeze prices for the term. If not, prepare for switching at contract end: start implementing a provider-agnostic abstraction layer (LiteLLM or custom), evaluate alternatives, run benchmark tests. Increase optimization efforts to offset the price increase in the short term.

## Scoring Rubric

Use this rubric to self-assess your answers to the challenges:

| Dimension | 1 (Needs Improvement) | 2 (Adequate) | 3 (Strong) | 4 (Outstanding) |
|---|---|---|---|---|
| Cost calculation | No numbers, vague estimates | Rough estimates, some errors | Correct calculations shown | Precise numbers, multiple scenarios considered |
| Tradeoff identification | Proposes optimizations without discussing downsides | Mentions 1-2 tradeoffs | Discusses tradeoffs for each optimization | Quantifies tradeoffs with dollar amounts and probabilities |
| System thinking | Single-component solution | 2-3 components, loosely connected | Layered architecture with clear data flow | Full system with feedback loops, monitoring, and iteration |
| Risk awareness | No risks mentioned | Mentions 1 generic risk | Identifies specific risks per optimization | Quantifies risk impact and proposes mitigations |
| Practical realism | Idealistic, ignores engineering cost | Somewhat practical | Realistic about engineering effort and timeline | Grounded in real-world constraints and tradeoffs
