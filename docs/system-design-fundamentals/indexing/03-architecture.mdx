## Naive Implementation: No Indexes

The simplest database deployment has zero indexes beyond the primary key. Every query performs a sequential scan. At small scale this is acceptable. At production scale it is catastrophic.

The progression of failure is predictable:
- At 100 rows, every query returns in under 2ms. No one notices a problem.
- At 10,000 rows, queries that filter on non-key columns take 40-80ms. Still acceptable for simple applications.
- At 1 million rows, the same queries take 2-8 seconds. The application feels sluggish. Users refresh the page, compounding the load.
- At 10 million rows, queries take 15-30 seconds. Connection pools fill up with waiting queries. New connections are rejected. The application is effectively down.
- At 50 million rows, queries take over a minute. The application is down. The database CPU is pinned at 100 percent from sequential scanning. The operations team is paged at 3 AM.

Every stage in this progression is avoidable with properly designed indexes. Yet this pattern repeats across thousands of production databases because indexes are added reactively, not proactively.

## Production-Grade Architecture

A production indexing strategy must balance read performance against write overhead, storage cost, and maintenance complexity. The following architecture patterns represent proven approaches at scale across companies like Uber, Netflix, and Cloudflare.

### Clustered + Secondary Index Strategy

A clustered index determines the physical order of table data. InnoDB always stores the table as a clustered index on the primary key. PostgreSQL uses a heap-based approach where the clustered index is a separate structure that reorders heap pages.

**Recommended pattern**: Choose a clustered index that matches the most common range-scan query pattern. If the application frequently queries by creation date range, cluster on `created_at`. If it queries by user ID ascending, cluster on `user_id`.

**Why clustered indexes matter for range queries**: A range query on a non-clustered index reads rows in index order, but each row fetch from the heap is a random I/O. With a clustered index, adjacent rows are on the same page, so a range scan reads pages sequentially. The performance difference is dramatic: a range scan returning 10,000 rows from a clustered index requires approximately 80 page reads (at 125 rows per page), while the same scan through a non-clustered index requires up to 10,000 random page reads.

**Key consideration for InnoDB users**: Secondary indexes in InnoDB store the primary key value as the row pointer, not a physical address. This means a lookup via a secondary index requires two B+Tree traversals: the secondary index (find the PK value), then the clustered index (find the row). The PK reference never changes even if the row moves due to page splits, which simplifies concurrency but doubles the tree-walk cost.

### Covering Indexes for Hot Queries

A covering index includes every column referenced by a specific query. The query can be answered entirely from the index pages without touching the heap table.

```sql
-- Instead of a general index (requires heap fetch for each column):
CREATE INDEX idx_orders_customer ON orders (customer_id);

-- Use a covering index for the hot query:
CREATE INDEX idx_orders_customer_covering
ON orders (customer_id) INCLUDE (order_date, status, total);

-- Query that is now covered:
SELECT customer_id, order_date, status, total
FROM orders
WHERE customer_id = 4237;
-- No heap access needed — all columns are in the index
```

The `INCLUDE` clause (PostgreSQL 11+, MySQL 8.0.13+, SQL Server 2012+) adds non-key columns to the leaf pages without including them in the B+Tree sort order. This means:
- The WHERE clause still benefits from the B+Tree's efficient search on the key columns.
- The INCLUDEd columns are stored only in the leaf pages, not in internal pages — they do not affect the fanout or tree depth.
- Index-only scans use the INCLUDEd columns without needing heap access.

**Trade-off**: Covering indexes increase index size proportionally to the included columns. Each included column adds its full width to every index entry. For a 100-million-row table, including a 16-byte UUID column adds 1.6GB to the index. The trade-off must be evaluated: the storage cost versus the I/O savings of eliminating heap fetches.

### Partial Indexes for Filtered Queries

A partial index covers only rows that satisfy a WHERE condition. The index is smaller, faster to scan, and cheaper to maintain because it excludes rows that will never match the query.

```sql
-- Index only active orders for a dashboard query
CREATE INDEX idx_orders_active
ON orders (created_at DESC)
WHERE status = 'active';

-- This index is tiny compared to a full index because
-- only a fraction of orders are "active" at any given time.
```

**Real-world impact**: A SaaS company with 500 million orders found that 95 percent of orders had `status = 'completed'` and were never queried via the dashboard. The dashboard-only query filtered `WHERE status = 'active'`. A partial index on the 25 million active orders was 20x smaller than a full index on all 500 million rows. The partial index also reduced write overhead because completed orders did not need index updates.

Partial indexes are especially valuable for:
- Soft-delete patterns (`WHERE deleted_at IS NULL`)
- Status-based filtering (`WHERE status = 'pending'`)
- Multi-tenant systems with a tenant filter (`WHERE tenant_id = 42`)
- Time-window queries (`WHERE created_at > NOW() - INTERVAL '30 days'`)

### Expression Indexes

Expression indexes store the result of a function or expression rather than the raw column value. They enable efficient indexing on transformed data.

```sql
-- Cannot use a normal index for this pattern:
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';

-- Create an expression index:
CREATE INDEX idx_users_lower_email ON users (LOWER(email));

-- Now LOWER(email) lookups use the index directly.
```

Expression indexes in PostgreSQL are maintained automatically — any INSERT or UPDATE that changes `email` also updates the index entry for `LOWER(email)`. The expression is evaluated once at index update time, not at query time. Expression indexes also work for date truncation, JSONB field extraction, and any immutable function.

### Maintenance Windows and Index Rebuilds

Indexes require periodic maintenance to prevent bloat and fragmentation:

| Operation | Blocks Writes? | Duration | Frequency |
|---|---|---|---|
| `REINDEX INDEX` (PostgreSQL) | Yes (ACCESS EXCLUSIVE) | Index rebuild time | Weekly/monthly |
| `REINDEX INDEX CONCURRENTLY` | No | ~2x standard rebuild | Weekly/monthly |
| `ALTER INDEX REBUILD ONLINE` (SQL Server) | No, with versioning overhead | Rebuild time | Weekly/monthly |
| `OPTIMIZE TABLE` (MySQL) | Yes (table copy) | Table copy time | Weekly |
| `VACUUM` (PostgreSQL) | No (clears dead tuples, not full rebuild) | Varies with bloat | Continuous (autovacuum) |

The scheduling strategy depends on write volume: a table with 1 million writes per day may need daily index maintenance, while a read-only table needs none.

## Integration Patterns: Indexing Across Databases

### PostgreSQL

- Default index type: B+Tree (referred to as "B-tree" in the documentation).
- Additional types: Hash, GiST, GIN, SP-GiST, BRIN, RUM (extension).
- Unique feature: `INCLUDE` columns for covering indexes since version 11.
- Concurrent builds: `CREATE INDEX CONCURRENTLY` for zero-downtime creation.
- Bloat management: VACUUM controls index bloat from dead tuples.
- Partitioning: Local indexes on each partition. Global indexes are not supported in native partitioning.
- Key monitoring view: `pg_stat_user_indexes` shows scan counts, tuples fetched, and page reads per index.

### MySQL (InnoDB)

- Clustered index on the primary key. If no explicit PK is defined, InnoDB creates a hidden 6-byte row ID.
- All secondary indexes reference the primary key (not a physical row ID), making the PK a critical design choice.
- `ALTER TABLE ... DROP INDEX, ADD INDEX` recreates the table (expensive for large tables).
- InnoDB buffer pool uses adaptive hash indexing for frequent lookups — a self-tuning mechanism.
- Since MySQL 8.0: invisible indexes (can test removing an index without actually dropping it), descending indexes, index skip scans.
- `MRR` (Multi-Range Read) optimization batches secondary index lookups into ordered primary key lookups, reducing random I/O.

### MongoDB

- Default index on `_id` (auto-generated ObjectId).
- Types: single-field, compound, multikey (array fields), text, geospatial, hashed, and wildcard indexes.
- Compound indexes support prefix matching (leftmost prefix rule, similar to relational databases).
- Covered queries: all returned fields are in the index — no document fetch required.
- Background index builds (version 3.2+) do not block writes on the database.
- Index intersection: MongoDB can use multiple indexes per query (since version 2.6), combining results in memory.

### Elasticsearch

- Inverted index on every field by default (dynamic mapping).
- Each field is indexed according to its data type (keyword, text, date, geo_point).
- Lucene segment-level indexes merge in the background; each segment is a self-contained inverted index.
- Elasticsearch indexes add to segment size and refresh time — not free.
- Index templates control mapping and settings per index pattern.
- The `refresh_interval` trades search freshness for indexing throughput: default 1 second, but can be set to 30 seconds for bulk loads.

### Capacity Planning for Index Storage

The storage requirement for an index depends on row count, key width, pointer width, fill factor, and overhead from page headers and internal node pages.

**Formula**:
```
leaf_pages = ceil(rows * (key_size + pointer_size + per_entry_overhead)
                  / (page_size - page_header) * fill_factor / 100)
internal_pages = leaf_pages / fanout + leaf_pages / fanout^2 + ... + 1
total_size = (leaf_pages + internal_pages) * page_size
```

**Example calculation for 100 million rows, 8-byte BIGINT key**:

| Parameter | Value | Notes |
|---|---|---|
| Rows | 100,000,000 | Fixed |
| Key size | 8 bytes | BIGINT |
| Pointer size | 6 bytes | PostgreSQL TID (4 bytes page + 2 bytes offset) |
| Per-entry overhead | 4 bytes | ItemId pointer in page header |
| Effective entry size | 18 bytes | Total per index entry |
| Page size | 8,192 bytes | Default PostgreSQL |
| Page header | 120 bytes | Fixed overhead |
| Available per page | 8,072 bytes | After header |
| Fill factor | 90 percent | Default |
| Usable per page | ~7,265 bytes | After fill factor |
| Entries per leaf page | ~400 | 7,265 / 18 |
| Leaf pages needed | 250,000 | 100M / 400 |
| Leaf space | ~2.0 GB | 250,000 * 8KB |
| Internal pages | ~500 | 250,000 / 500 fanout |
| Root pages | 1 | Single root |
| Total index size | ~2.0 GB | Approximately |

**Scaling estimates for different key sizes**:
- 100 million rows, 8-byte key: ~2GB per index.
- 100 million rows, composite (8B + 8B = 16B key): ~4GB per index.
- 100 million rows, varchar(100) email (30B average): ~5GB per index.
- 1 billion rows, 8-byte key: ~20GB per index.
- Same table with 5 indexes: ~100GB additional storage.

## Multi-Region Indexing

### Indexes on Read Replicas

In a multi-region deployment with read replicas, indexes on replicas work the same as on the primary — they are read-only copies. The primary concern is replication lag:

- An index on a replica is as current as the replica's data. If the replica is 2 seconds behind, the index reflects data from 2 seconds ago.
- A query that uses a replicated index to find rows inserted on the primary but not yet replicated will miss those rows.
- Read-your-writes consistency requires routing the read to the primary or waiting for replication confirmation (synchronous replication).

### Secondary Index Challenges in Distributed Databases

Distributed databases face a fundamental problem with secondary indexes: the indexed value may exist on a different node than the row it references. This is called the "distributed secondary index problem."

**Solution approaches**:

1. **Local indexes** (Cassandra, DynamoDB): Each node indexes only its own data. Queries that filter on non-partition-key columns must scatter requests to all nodes (fan-out query). Efficient only when the query includes the partition key.

2. **Global secondary indexes** (DynamoDB GSI, MongoDB, CockroachDB): A separate index table distributed across all nodes. Each index entry includes the shard key, allowing point queries to route directly to the correct node. Write amplification: every write to the primary table also updates the GSI, which may involve a distributed transaction.

3. **Materialized views with logical replication** (PostgreSQL): Maintain a separate table with a different primary key, kept in sync via triggers or logical replication. Offers flexibility but adds complexity.

4. **Interleaved indexes** (Cloud Spanner): Child table (index) rows are stored physically interleaved with parent table rows. Co-locates related data for fast local reads at the cost of reduced write throughput on the parent.

## Deployment

### Online Index Creation

The standard `CREATE INDEX` blocks writes on the table. For production deployments, use concurrent creation:

```sql
-- PostgreSQL: concurrent creation
CREATE INDEX CONCURRENTLY idx_orders_customer_id
ON orders (customer_id);
```

**Phases of CONCURRENTLY**:
1. Build the index structure via a first table scan (ACCESS SHARE lock — compatible with other operations).
2. Wait for all in-flight transactions to complete.
3. Second table scan to catch rows modified during Phase 1.
4. Brief ACCESS EXCLUSIVE lock to validate and make the index visible.

**Risk**: If the concurrent build fails (deadlock, unique violation), it leaves an invalid index that consumes space but is not used. The invalid index must be dropped and rebuilt.

### Monitoring Unused and Duplicate Indexes

Indexes that are never used waste write throughput and storage. Every production system should monitor for dead indexes.

**PostgreSQL monitoring query**:
```sql
SELECT
  schemaname,
  tablename,
  indexname,
  idx_scan AS index_scans,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  pg_relation_size(indexrelid) AS index_size_bytes
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;
```

**Duplicate index detection**:
```sql
-- Detects indexes with identical leading columns
SELECT
  a.indrelid::regclass AS table_name,
  a.indexrelid::regclass AS index_a,
  b.indexrelid::regclass AS index_b
FROM pg_index a
JOIN pg_index b
  ON a.indrelid = b.indrelid
  AND a.indexrelid < b.indexrelid
  AND a.indkey = b.indkey;
```

**Operational cadence**: Run these queries weekly. Drop any index with zero scans over a 7-day period. Investigate indexes with fewer than 10 scans — they may still be useful if the queries are rare but critical (e.g., monthly reporting).

## Schema Migration Strategies for Index Changes

Managing index changes as part of database schema migrations requires careful planning to avoid downtime.

**Adding a new index**:
- Development: `CREATE INDEX` (no CONCURRENTLY needed, zero data).
- Staging: `CREATE INDEX` on production-scale data to benchmark build time.
- Production: `CREATE INDEX CONCURRENTLY` to avoid write blocking.
- Rollback: `DROP INDEX CONCURRENTLY` if performance regression detected.

**Dropping an index**:
- Verify the index is unused: check `pg_stat_user_indexes.idx_scan` for at least 7 days.
- Drop with `DROP INDEX CONCURRENTLY` (PostgreSQL 9.6+).
- Monitor query performance for 24 hours after the drop.
- Have the DDL ready to recreate if the index was actually needed.

**Modifying an index** (changing columns, adding INCLUDE columns):
- Cannot modify an index in place in most databases.
- Pattern: create new index CONCURRENTLY, drop old index CONCURRENTLY.
- During the transition window, both indexes exist — insert performance degrades temporarily (extra index maintenance).

**Real-world migration example**:
```
Monday:  Profile queries, discover index idx_users_email not used by any query
Tuesday: Verify — check pg_stat_user_indexes for 7 days of zero scans
Wednesday: DROP INDEX CONCURRENTLY idx_users_email;
Thursday:  Monitor query performance — confirm no regression
Friday:    Storage savings: 2GB reclaimed
```

## Performance Benchmarking

When deploying a new index, benchmark the before-and-after performance:

```sql
-- Before index:
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE customer_id = 42;

-- After CREATE INDEX CONCURRENTLY:
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE customer_id = 42;
```

**What to look for in the output**:
- Plan changed from `Seq Scan` to `Index Scan` — expected and desired.
- `Buffers: shared hit=` shows pages read from cache vs `read=` from disk.
- Actual time drops from seconds to milliseconds — confirmation the index works.
- Planning time may increase slightly (more indexes to evaluate) — acceptable.

**When the index is NOT used by the optimizer**:
- The `WHERE` clause might not be selective enough.
- The optimizer's cost model might be wrong: check `random_page_cost` and `effective_cache_size`.
- Statistics might be stale: run `ANALYZE` and retry.
- The index might not match the query: check column order and expression.

## Bridge to Trade-offs

Production indexing architecture makes concrete the patterns of clustered, covering, partial, and expression indexes. However, each pattern carries trade-offs: covering indexes bloat storage, partial indexes miss unexpected queries, concurrent builds cost CPU and I/O, and global secondary indexes in distributed databases add write amplification. The trade-offs section explores these tensions systematically, mapping decision points for B-Tree versus Hash, clustered versus non-clustered, and covering versus minimal index design.
