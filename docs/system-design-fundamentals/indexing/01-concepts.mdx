## The Problem Story

A customer support database holds 50 million customer records. An agent searches for a customer by email address. The database reads every single row — all 50 million — comparing each one against the search email. Forty-five seconds pass before the result appears. The customer on the phone has been waiting over a minute. The agent cannot help other callers during this wait. The CEO demands answers.

This is a full table scan. Without an index, the database has no shortcut. It visits every page, every row, every time.

The fix is an index on the email column. With it, the same query returns in under 3 milliseconds. The agent handles the call, resolves the issue, and moves to the next customer. The CEO never learns about the fix because the problem disappears.

**The broader problem**: This scenario plays out thousands of times daily across production databases. A missing index on a foreign key column causes cascading sequential scans in JOIN queries. An index on a timestamp column that was dropped during a migration is never recreated. A composite index with columns in the wrong order sits unused while the optimizer chooses a full table scan. Each case costs seconds per query, minutes per incident, and hours of engineering time to diagnose. Indexing is not academic — it is the difference between a database that works and one that fails under load.

## The Insight

An index is a separate data structure that maps column values to their physical locations in a table. Instead of scanning every row, the database consults the index, finds the location of the matching rows, and fetches only those pages.

**Analogy: a book index.** An 800-page textbook on machine learning contains information about decision trees, gradient boosting, and neural networks. Without an index at the back, finding "gradient boosting" means flipping through every page. With an index, you look up "gradient boosting" in 2 pages, see page references 342 and 478, and flip directly there. You read 3 pages instead of 800.

The same principle applies to databases. An index on `email` maps `alice@example.com` to the specific data page and row offset where that record lives. The database reads one index page (instead of millions of table pages) and jumps directly to the data.

**A deeper analogy: the card catalog in a library.** A library of 100,000 books does not require you to walk every aisle reading every spine to find a specific title. The card catalog (the index) is organized alphabetically by title, author, and subject. You find the card, note the call number (Dewey decimal), and walk directly to the exact shelf. The card catalog is a separate set of drawers from the bookshelves — just as an index is a separate database object from the table. And when the library adds a new book, the librarian must write a new card for the catalog (index maintenance). The book goes on the shelf; the card goes in the drawer. Both must stay in sync.

**The database equivalent**: The table is the bookshelf. The index is the card catalog. The row pointer (TID or primary key) is the call number. The B+Tree is the organized drawer system. Index maintenance is the librarian writing new cards. A missing index is a library with no catalog — every search is a full walk of every aisle.

## Core Vocabulary

Master these 22 terms before moving to the deep dive.

**Index** — A data structure that maps column values to row locations, enabling fast lookup without scanning the full table. Indexes are stored as separate database objects with their own storage allocation and statistics.

**Primary Index** — Built automatically on the primary key. In most databases, the table data is stored in the order of the primary key. InnoDB's clustered index is the primary index — the table is the index. PostgreSQL uses a heap-based approach where the primary key creates a unique B+Tree index, and the table data remains in heap order.

**Secondary Index** — Any index that is not the primary index. Stores the indexed column values plus a pointer to the corresponding row in the primary index or heap table. A table can have many secondary indexes.

**B-Tree** — Balanced tree data structure where every leaf node is at the same depth. Each node contains keys and pointers to child nodes. The balance property guarantees that the tree depth grows logarithmically with the number of entries. Used by nearly all relational databases for general-purpose indexing.

**B+Tree** — Variant of B-Tree where only leaf nodes store actual data pointers. Internal nodes store only keys for routing. Leaf nodes are linked together via sibling pointers for efficient range scans. The standard index structure in PostgreSQL, MySQL (InnoDB), and SQLite.

**Hash Index** — Uses a hash function to map keys to bucket locations. Only supports equality lookups (no range queries, no ORDER BY support). Historically limited in PostgreSQL with no WAL logging, but hash indexes were fully crash-safe as of PostgreSQL 10.

**Composite Index** — An index on multiple columns. Column order is critical. An index on `(last_name, first_name)` helps queries filtering by `last_name` or both columns, but not queries filtering by `first_name` alone. This is the leftmost prefix rule.

**Clustered Index** — Determines the physical order of data in a table. A table can have only one clustered index. InnoDB stores table rows in the primary key B+Tree leaf pages. Benefits range scans on the clustered key because adjacent rows are stored on the same or adjacent pages.

**Non-Clustered Index** — A separate structure from the data rows. Contains indexed columns and a pointer (row ID or primary key value) to the actual data. A table can have many non-clustered indexes. Each non-clustered index lookup in InnoDB requires two B+Tree traversals: the secondary index, then the clustered index.

**Full Table Scan** — Reading every row in a table sequentially. The database reads all data pages from disk. Inevitable when no suitable index exists for the query's WHERE clause. Also called a sequential scan.

**Selectivity** — The fraction of rows matched by a query predicate. High selectivity (few rows match, such as 0.01 percent of rows) favors index usage. Low selectivity (many rows match, such as 50 percent of rows) may bypass the index in favor of a scan because the overhead of random reads to the heap exceeds the cost of a sequential scan.

**Cardinality** — The number of distinct values in a column. High cardinality (email, primary key) makes good index candidates. Low cardinality (boolean, gender) rarely benefits from B+Tree indexing alone because the index returns too many rows per lookup.

**Covering Index** — An index that contains all columns needed by a query. The query can be answered entirely from the index pages without touching the table at all. Also called an index-only scan. In PostgreSQL, the `INCLUDE` clause adds non-key columns to the leaf pages without affecting the sort order.

**Index Scan** — The database reads index pages sequentially, often returning many rows. Used for low-selectivity queries or when the index matches the ORDER BY clause. Each matching row may require a separate table lookup. Also called an index range scan.

**Index Seek** — The database navigates the B+Tree from root to a specific leaf page, finding rows with pointer precision. Used for high-selectivity equality lookups. Much faster than an index scan because only a few pages are read.

**Fill Factor** — A percentage that controls how full each B+Tree page is on creation. A fill factor of 70 means each page is filled to 70 percent capacity, leaving 30 percent room for future insertions without page splits. Lower fill factors reduce split frequency at the cost of more index pages.

**Page Split** — When a B+Tree page reaches capacity and a new entry must be inserted, the page splits into two. This is expensive: it requires locking the page, allocating new space, redistributing entries, and potentially cascading the split upward through parent pages. Page splits are a primary source of write latency spikes.

**Index Fragmentation** — Over time, page splits and deletions cause index pages to become partially empty or logically out of order on disk. Fragmentation wastes space and slows scans because the physical order diverges from the logical order. Rebuilding the index consolidates pages and restores physical order.

**TID (Tuple ID)** — PostgreSQL's row pointer format: a 6-byte value encoding the page number (4 bytes) and the offset within the page (2 bytes). B+Tree leaf entries in PostgreSQL store `(key, TID)` pairs. The TID is a physical address — if the row moves to a different page (e.g., via VACUUM FULL), the TID changes.

**Heap** — In PostgreSQL, the main data storage structure where table rows are stored in no particular order. Indexes reference rows via TIDs pointing into the heap. This is different from InnoDB's clustered index approach where the table data is the primary key B+Tree's leaf pages.

**Autovacuum** — PostgreSQL's background process that removes dead row versions and reclaims storage. Autovacuum also updates index statistics. If autovacuum falls behind, index bloat accumulates and query performance degrades.

**BRIN (Block Range INdex)** — A specialized index type for large tables where data is naturally ordered (e.g., time-series data). BRIN indexes store the minimum and maximum value for each block range rather than indexing each row individually. They are tiny (millions of times smaller than a B+Tree) and cheap to maintain, at the cost of less precise lookups.

## Step by Step: What Happens With and Without an Index

### Without an Index (Full Table Scan)

The database executes a sequential scan when no index is available for the query's WHERE clause. The following sequence shows exactly what happens:

1. **Parser**: The SQL text is tokenized into keywords, identifiers, and operators. The parser validates the grammar against the SQL standard and produces a parse tree.

2. **Binder**: The parse tree is bound against the system catalog. The binder resolves `customers` to the table's object ID and `email` to column number 4. It checks that the user has SELECT permission on the table and that `email` is a text-comparable column.

3. **Optimizer**: The optimizer examines available indexes — there are none on `email`. It estimates the cost of a full table scan using the formula: `relpages * seq_page_cost + reltuples * cpu_tuple_cost`. For 50 million rows across 400,000 pages, the cost is approximately 400,000 sequential page reads at 1.0 cost unit each, plus 50 million row comparisons at 0.01 cost units each. Total cost: approximately 900,000 units.

4. **Executor start**: The executor opens the heap relation. The buffer manager is asked to pin the first page. If the page is not in the shared buffer pool, a disk read is scheduled.

5. **Page-by-page read**: The first page (8KB) is read from disk. It contains approximately 125 rows. The executor scans every row on the page, evaluating `row.email = 'alice@example.com'` for each one.

6. **Row comparison**: Each row comparison is a string equality check. The database must compare the full email string byte-by-byte. No match on page 0. The executor moves to page 1.

7. **Buffer pool interaction**: Each page may or may not be in the buffer pool. On a cold cache (database restart), the first query experiences a 100 percent cache miss rate — every page is a disk read. On a warm cache, the pages may be cached from previous queries, but a full table scan of 50 million rows will evict other useful pages from the buffer pool.

8. **Scan continuation**: The executor reads page after page. The matching row appears somewhere around page 350,000. The executor finds it, but must keep scanning the remaining 50,000 pages because there may be more matches (unless the column has a UNIQUE constraint, in which case the database can stop at the first match).

9. **CPU cost**: Beyond the I/O cost, the CPU cost of 50 million string comparisons is significant. Each comparison involves checking the length prefix, then comparing bytes. At 50 million comparisons, this adds hundreds of milliseconds of CPU time.

10. **Result**: The query returns the row after 45 seconds. The database has read 400,000 pages from disk (approximately 3.2GB of data) and performed 50 million row comparisons. The buffer pool is now polluted with pages that are unlikely to be needed again.

### With an Index

1. **Parser and binder**: Same steps as above. The parse tree and bound query are identical.

2. **Optimizer**: The optimizer now sees an index on `email`. It estimates the cost: `index_pages * random_page_cost per level`. A B+Tree index for 50 million rows is roughly 3 levels deep with a fanout of 500. Cost: 3 random page reads at 4.0 cost units each, plus 1 heap page read at 1.0 cost unit. Total cost: approximately 13 cost units — versus 900,000 for the sequential scan.

3. **Plan selection**: The optimizer selects the index plan. The cheapest path is clearly the index. The executor receives the plan and prepares to call the index access method.

4. **Buffer pool lookup**: The index access method starts at the root B+Tree page. The root page is likely already in the buffer pool from a previous query — it is the most frequently accessed page in the index.

5. **Root page traversal**: The root page contains separator values: `["d", "h", "l", "p", "t"]`. The database performs a binary search on the root page to find the correct pointer. The search key "a" is less than "d", so it follows the leftmost pointer.

6. **Internal page traversal**: The internal page at level 2 is read. It contains approximately 400 separators: `["ab", "ac", "al", "an", "ar", "ba", "be", ...]`. Binary search finds that "alice" falls between "al" and "an". The database follows the third pointer.

7. **Leaf page read**: The leaf page is read. It contains approximately 200 key-pointer pairs in sorted order. The database performs a binary search on the leaf page and finds the entry for `"alice@example.com"` with TID `(4237, 12)` — page 4237, offset 12.

8. **Heap page fetch**: The executor reads heap page 4237. The buffer manager checks the buffer pool — if the page is cached, the read is a memory copy. If not, it issues a random disk read. The page is read and the row at offset 12 is extracted.

9. **Row projection**: The executor projects the requested columns from the row. The full row is returned to the client.

10. **Total accounting**: 4 page reads in total (3 B+Tree pages + 1 heap page). Total I/O: 32KB (versus 3.2GB for the full scan). Total time: under 10 milliseconds. The buffer pool is minimally impacted — only 4 pages are added, and the root page was already cached.

## Index Variants

### B+Tree Index

The dominant index type in every major relational database. Data is stored in a balanced tree where:
- Every path from root to leaf has the same length (balance property).
- Internal nodes contain routing keys and child pointers — no data.
- Leaf nodes contain key-pointer pairs and are linked left-to-right via sibling pointers.
- Range queries scan the leaf chain forward without traversing the tree again.
- The fanout (keys per internal node) is typically 200-500, keeping the tree shallow.

Used for: equality lookups, range queries, sorting, prefix matching, ORDER BY optimization.

### Hash Index

Maps keys to bucket locations using a hash function. A perfect hash structure offers O(1) lookup time, but only for equality predicates. Range queries are impossible because hash ordering does not preserve key ordering. PostgreSQL's hash index implementation was overhauled in version 10 to be crash-safe and WAL-logged. Before that, hash indexes were dangerous in production.

Used for: exact match lookups on columns with many distinct values, where range queries are never needed.

### GiST (Generalized Search Tree)

A balanced tree structure that supports custom data types and search predicates. GiST is not a single algorithm but a framework: you define the union, consistency, and penalty functions for your data type, and the GiST framework builds the tree. This makes GiST a foundation for many specialized index types.

Used for: full-text search (tsvector with GiST), geospatial queries (PostGIS geometry), ranges (int4range, daterange), network address lookups (inet, cidr), trigram similarity matching (pg_trgm), UUID order-preserving indexes, nearest-neighbor searches for (lat, lng) coordinates, and other distance-ranked searches.

**GiST vs GIN for full-text**: GiST is faster to build and update (better for write-heavy workloads). GIN is faster to search (better for read-heavy workloads). GIN produces smaller indexes. The choice depends on the write-to-read ratio.

### GIN (Generalized Inverted Index)

Stores a mapping from each distinct key to a list of locations (posting list) where that key appears. GIN is optimized for composite types — columns that contain multiple values. The key insight is that each index entry (e.g., a word in a document) maps to many rows, so the index is "inverted" compared to the normal row-to-column mapping.

Performance characteristics: GIN builds are 2-3x slower than GiST for the same data, but searches are 3-5x faster. GIN indexes are also smaller because the posting lists compress well. The `gin_pending_list_limit` parameter controls how many buffered inserts accumulate before being merged into the main index tree — tuning this parameter is critical for write-heavy workloads.

Used for: array columns (`@>` operator), JSONB query acceleration (`?`, `@>`, `?|` operators), inverted indexes for full-text search (tsvector), composite type search.

### Spatial (R-Tree) Index

Based on the R-Tree data structure, spatial indexes organize geometric data by bounding boxes. Each node in the tree has a bounding box that encloses all shapes in its subtree. Queries traverse the tree by checking whether the query region overlaps with each node's bounding box — if not, the entire subtree is pruned.

Used for: geographic information systems (PostGIS), location-based services (find nearby restaurants), spatial JOIN operations (find all parcels within a flood zone).

### Bitmap Index

Stores a bitmap (array of bits) for each distinct value in a column. Each bit indicates whether a row contains that value. The bitmap for each value is compressed using run-length encoding. Bitmap indexes excel on columns with low cardinality and are common in data warehousing.

**Bitwise operations**: The real power of bitmap indexes comes from combining multiple conditions. `WHERE region = 'EMEA' AND product_line = 'Cloud'` performs a bitwise AND of the two bitmaps. With 10 million rows and 10 conditions, the entire query can be answered with 10 bitwise AND operations — no row comparisons needed.

Used for: data warehouse star schemas, low-cardinality columns in analytical queries, complex Boolean combinations across multiple dimension tables.

## Mental Model Questions

1. **Question**: A query searches for `WHERE status = 'active'` on a table of 10 million rows. The `status` column has 2 distinct values. Should an index on `status` help?

   **Answer**: No — this is low selectivity with 5 million rows per value. The database would need to read half the table pages even with the index. A bitmap index might help in a data warehouse where the status column is combined with other filters via bitwise AND. In OLTP, a B+Tree index on this column would likely be ignored by the optimizer in favor of a full scan. The optimizer knows that reading 5 million rows through the index requires 5 million random heap page reads (each potentially 4x cost), which is more expensive than a sequential scan of all pages.

2. **Question**: A composite index exists on `(city, last_name)`. Does it help a query filtering by `last_name` alone?

   **Answer**: Possibly not — the index is ordered by `city` first. Without a `city` filter, the index cannot efficiently narrow the search. The database cannot skip the leading column. Some databases support index skip scans (PostgreSQL 15+, MySQL 8.0.13+) which can partially use the index by scanning each distinct city prefix, but this is less efficient than having the correct column order. The general rule: the leading column of a composite index must match the query's WHERE clause for the index to be used efficiently.

3. **Question**: Why do databases use B+Trees instead of binary search trees for indexes?

   **Answer**: B+Trees have a high fanout (hundreds of keys per node), keeping the tree depth shallow at 3-4 levels even for billions of rows. Binary search trees have a branching factor of 2, requiring 30+ levels for 1 billion rows. Each level in a BST is a separate disk page read. Thirty disk reads per lookup is unacceptable. B+Trees also pack keys densely into disk pages (8KB each), minimizing disk I/O — each node read retrieves hundreds of keys in one operation. The sequential leaf chain of B+Trees also enables efficient range scans, which BSTs cannot do without parent pointers.

4. **Question**: A covering index contains all columns referenced by a SELECT query. How does this change execution?

   **Answer**: The database can answer the query entirely from the index pages without reading the heap (table) pages. This is called an index-only scan. It saves the random I/O of fetching each matching row's data page. For a query returning 10,000 rows that are scattered across the table, a covering index eliminates 10,000 random heap page reads. In PostgreSQL, this requires the visibility map to confirm that the index entries are all visible — otherwise, a heap check is still needed for MVCC visibility.

5. **Question**: An INSERT statement on a table with 8 indexes takes 8 times longer than the same insert on a table with no indexes. Is this expected?

   **Answer**: Roughly yes — each index must be updated on every row insert. The B+Tree for each index requires locating the correct leaf page and inserting the new key-pointer pair, possibly triggering page splits. The overhead scales with both the number and size of indexes. However, the relationship is not perfectly linear because: (a) some indexes may share the same leaf page for adjacent keys, (b) the B+Tree traversal cost is shared across the root and internal pages (which are cached), and (c) the actual insertion cost depends on the key distribution and whether the insert causes a page split. This is the fundamental read-write tradeoff of indexing.

6. **Question**: A table has 100 million rows. A query searches by `id` (primary key) and returns in 2ms. The same query searches by `email` (non-unique, not indexed) and takes 30 seconds. Why the difference?

   **Answer**: The primary key `id` has a clustered index (InnoDB) or a unique B+Tree index (PostgreSQL). The database traverses the B+Tree directly to the row. The `email` column has no index — every query performs a full table scan of 100 million rows. The optimizer has no choice because no index exists for the `email` predicate. Creating an index on `email` would bring the second query to sub-millisecond speed.

7. **Question**: What is the difference between an index scan and an index seek in database execution plans?

   **Answer**: An index seek navigates the B+Tree from root to a specific leaf page, finding rows with pointer precision. It is used for high-selectivity equality lookups and typically reads only a few pages. An index scan reads index pages sequentially, often from the first leaf page to the last. It is used when the query needs many rows (low selectivity) or when the index matches the ORDER BY clause. Index scans are more expensive than seeks but still cheaper than full table scans if the index is smaller than the heap.

## Additional Practice Problems

### Problem 1: Choosing Between Single-Column Indexes

A table `orders` has columns `customer_id` (BIGINT, 10 million distinct values), `status` (VARCHAR, 5 distinct values), and `created_at` (TIMESTAMPTZ). The query is:

```sql
SELECT * FROM orders
WHERE customer_id = ? AND status = 'shipped'
ORDER BY created_at DESC;
```

**Question**: Which is better: two single-column indexes on `customer_id` and `status`, or one composite index on `(customer_id, status, created_at DESC)`? Explain why.

**Answer**: The composite index is superior. Two single-column indexes would require the database to use index intersection (bitmap AND) for the WHERE clause, then sort the results in memory. The composite index on `(customer_id, status, created_at DESC)` supports all three query components: the equality filter on `customer_id`, the equality filter on `status`, and the ORDER BY `created_at DESC`. Since the leading column `customer_id` has high selectivity (10 million distinct values), the index narrows the search to a few hundred rows immediately.

### Problem 2: Index Maintenance Cost

A table receives 1,000 INSERTs per second. It has 5 indexes. Each index update takes 0.3ms. How much CPU time per second is spent on index maintenance?

**Answer**: 5 indexes * 1,000 inserts/second * 0.0003 seconds = 1.5 seconds of CPU time per second. This means 150 percent of a single CPU core is consumed by index maintenance. With 16 CPU cores, this represents approximately 9 percent of total CPU capacity — acceptable. With 4 CPU cores, this represents 37 percent — expensive. Adding a 6th index would increase it to 1.8 seconds per second (45 percent of 4 cores).

### Problem 3: Range Scan Performance

A clustered index on `created_at` stores 100 million rows across 800,000 pages. A range query requests 30 days of data, approximately 3 million rows. With 125 rows per page, the range covers 24,000 pages. A sequential scan of these pages at 100 microseconds each (SSD) takes 2.4 seconds. A non-clustered index that requires random page fetches for 3 million rows at 100 microseconds each would take 300 seconds. The clustered index provides a 125x speedup for this range scan.
