## How These Challenges Work

Each tier builds on the previous one. Tier 1 establishes basic index design skills. Tier 2 applies those skills to realistic system design scenarios. Tier 3 extends to distributed and multi-region architectures.

Attempt each challenge without looking at the hints first. After completing each tier, verify your design against the evaluation criteria provided at the bottom.

---

## Tier 1 — Foundation

### Challenge 1: Simple Query Indexing

A database table stores user accounts:

```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL,
    username VARCHAR(100) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    full_name VARCHAR(200),
    avatar_url TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_login TIMESTAMPTZ,
    is_active BOOLEAN DEFAULT true
);
```

The application executes the following queries:
1. `SELECT * FROM users WHERE email = ?` — Login flow, 10,000 queries per second.
2. `SELECT id, username, avatar_url FROM users WHERE id = ?` — Profile page, 20,000 queries per second.
3. `SELECT id, email, full_name FROM users WHERE LOWER(email) LIKE ?` — Admin search, 10 queries per second.
4. `SELECT id, username, created_at FROM users WHERE is_active = true ORDER BY created_at DESC LIMIT 20` — Dashboard, 100 queries per second.

**Task**: Design the minimum set of indexes that cover all queries efficiently. For each index, specify:
- The index type (unique, non-unique, partial, expression).
- The columns and their order.
- Whether it should be a covering index (with INCLUDE columns).
- The rationale for the design.

**Hint**: Query 1 requires a unique constraint. Query 4 has 2 distinct values for `is_active` — consider whether an index helps here at all.

**Evaluation criteria**:
- Queries 1 and 2 should be covered by indexes that enable sub-millisecond lookups.
- Query 3 should be optimized for the admin use case without compromising login performance.
- Query 4 should have a clear trade-off discussion — an index on `is_active` is wasteful given low cardinality.
- Total indexes should be 4 or fewer. At least 2 should be covering indexes.

### Challenge 2: B+Tree Page Simulation

A B+Tree index has the following parameters:
- Page size: 8KB (8,192 bytes)
- Internal node key size: 8 bytes per key
- Internal node pointer size: 8 bytes per pointer
- Leaf node key size: 8 bytes
- Leaf node pointer size: 6 bytes (TID)
- Page header: 120 bytes
- Fill factor: 90 percent

**Task A**: Calculate the maximum number of leaf entries per page. Show each step of the calculation.

**Task B**: How many leaf pages are needed to store 100 million rows?

**Task C**: How many internal pages are needed if the fanout is 500?

**Task D**: What is the total index size in GB?

**Task E**: If the fill factor is reduced to 70 percent, how does the index size change? What is the trade-off in terms of page split frequency?

**Hint for Task E**: Lower fill factor means more pages (larger index) but each page has more headroom for future inserts, reducing the probability that an insert triggers a page split.

**Answer check**:
- Leaf entries per page: approximately 400 (403 calculated precisely)
- Leaf pages for 100M rows: 248,139 pages (approximately 2.0GB)
- Internal pages (fanout 454): 547 + 2 + 1 = 550 pages (approximately 4MB)
- Total size: approximately 2.0GB
- With fill factor 70: index grows to approximately 2.6GB, but page splits decrease by approximately 65 percent

### Challenge 3: Diagnose a Slow Query

A team reports that the following query suddenly became slow:

```sql
SELECT o.id, o.total, o.status, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'pending'
  AND o.created_at > '2025-01-01'
ORDER BY o.created_at DESC
LIMIT 100;
```

The table has:
- `orders`: 200 million rows. Indexes: `PRIMARY KEY (id)`, `INDEX (customer_id)`, `INDEX (status)`.
- `customers`: 50 million rows. Indexes: `PRIMARY KEY (id)`.
- The query previously returned in under 200ms. It now takes 8 seconds.

**Task**: List at least 5 possible causes for the regression, in order of likelihood. For each cause, describe how to verify it and what the fix would be.

**Hint**: Consider the optimizer's decision process. The query involves a JOIN and two WHERE conditions — the optimizer must choose between multiple index strategies.

**Possible causes to consider**:
1. Stale statistics leading to a bad query plan.
2. Index bloat from accumulated dead entries.
3. A missing index on the foreign key `orders.customer_id`.
4. A change in data distribution (many more pending orders now).
5. A PostgreSQL version upgrade that changed the optimizer's cost parameters.
6. A change in server configuration (random_page_cost, effective_cache_size).

**Evaluation criteria**:
- At least 3 of the 5 causes should match an actual regression scenario.
- The verification method should be specific (e.g., "Run EXPLAIN ANALYZE and compare the plan with a known-good baseline").
- The fix should be actionable (e.g., "Update statistics with ANALYZE", "Reindex with CONCURRENTLY", "Add a composite index on (status, created_at)").

## Tier 2 — Design

### Challenge 4: Social Media Feed Index Strategy

Design the indexing strategy for a social media platform's feed system.

**Schema**:
```sql
CREATE TABLE posts (
    id BIGSERIAL PRIMARY KEY,
    author_id BIGINT NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    likes_count INT DEFAULT 0,
    visibility VARCHAR(20) DEFAULT 'public'
);

CREATE TABLE follows (
    follower_id BIGINT NOT NULL,
    followee_id BIGINT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (follower_id, followee_id)
);
```

**Query patterns**:
1. **Timeline**: `SELECT id, content, author_id, created_at FROM posts WHERE author_id IN (?) ORDER BY created_at DESC LIMIT 50` — 100,000 QPS.
2. **User profile**: `SELECT id, content, created_at, likes_count FROM posts WHERE author_id = ? ORDER BY created_at DESC LIMIT 20` — 50,000 QPS.
3. **Trending**: `SELECT id, author_id, content, likes_count FROM posts WHERE visibility = 'public' AND created_at > NOW() - INTERVAL '24 hours' ORDER BY likes_count DESC LIMIT 100` — 1,000 QPS.
4. **Follow** (write): `INSERT INTO follows (follower_id, followee_id) VALUES (?, ?)` — 10,000 writes per second.
5. **New post** (write): `INSERT INTO posts (author_id, content, visibility) VALUES (?, ?, ?)` — 5,000 writes per second.

**Constraints**:
- Read latency: under 50ms p99.
- Write throughput must sustain peak with fewer than 5 percent degradation from index maintenance.
- Total index storage must not exceed 50 percent of table storage.

**Task**: Design the complete index strategy. Consider whether the timeline IN-list query requires a fan-out approach instead of a simple index solution. Address the trending query — how do you index `likes_count` efficiently for a time-windowed query?

### Challenge 5: E-Commerce Product Search

Design the indexing strategy for an e-commerce product catalog with faceted search.

**Schema**:
```sql
CREATE TABLE products (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(500) NOT NULL,
    description TEXT,
    category_id INT NOT NULL,
    brand VARCHAR(200),
    price DECIMAL(10,2) NOT NULL,
    rating DECIMAL(2,1) DEFAULT 0.0,
    stock_quantity INT DEFAULT 0,
    is_active BOOLEAN DEFAULT true,
    attributes JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Query patterns**:
1. **Search by keyword**: Full-text search on `name` and `description` — 20,000 QPS.
2. **Category browsing**: `SELECT * FROM products WHERE category_id = ? ORDER BY rating DESC, price ASC LIMIT 50` — 10,000 QPS.
3. **Faceted filter**: `SELECT * FROM products WHERE category_id = ? AND attributes @> '{"color": "red"}' AND price BETWEEN 10 AND 100 ORDER BY rating DESC` — 5,000 QPS.
4. **Stock check**: `SELECT id, stock_quantity FROM products WHERE id = ?` — 50,000 QPS.

**Constraints**: 100 million products across 5,000 categories. JSONB attributes contain 5-20 key-value pairs per product. 1,000 product updates per second.

**Task**: Design the index strategy. Consider PostgreSQL GIN for JSONB, GIN for full-text search, partial indexes for active products, and composite indexes for category browsing. Evaluate whether PostgreSQL full-text search handles 20,000 QPS or if Elasticsearch is needed.

### Challenge 6: Real-Time Analytics Dashboard

A real-time analytics system ingests 500,000 events per second and supports dashboard queries.

**Schema**:
```sql
CREATE TABLE events (
    id BIGSERIAL,
    tenant_id INT NOT NULL,
    event_type VARCHAR(50) NOT NULL,
    source_ip INET,
    duration_ms INT,
    status_code INT,
    occurred_at TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (occurred_at);
```

**Query patterns**:
1. **Time-series count**: `SELECT occurred_at::date, count(*) FROM events WHERE tenant_id = ? AND occurred_at BETWEEN ? AND ? GROUP BY day` — 1,000 QPS, sub-second required.
2. **Top sources**: `SELECT source_ip, count(*) FROM events WHERE tenant_id = ? AND occurred_at > NOW() - INTERVAL '1 hour' GROUP BY source_ip ORDER BY hits DESC LIMIT 10` — 500 QPS.
3. **Error rate**: `SELECT status_code, count(*) FROM events WHERE tenant_id = ? AND occurred_at > NOW() - INTERVAL '1 day' GROUP BY status_code` — 200 QPS.

**Constraints**: 500,000 writes per second. Partitioned by day (365 partitions). Partitions older than 90 days are detached. Dashboard queries need sub-second response.

**Task**: Consider BRIN indexes for time-series data, partial indexes on event_type, and materialized views for pre-aggregation. The 500,000 writes per second severely limit index maintenance budget.

## Tier 3 — Expert

### Challenge 7: Distributed Indexing for Global Multi-Region Database

A global social platform stores user messages across 6 regions. Each region has its own database cluster. The system must support local reads, global search, and cross-region writes.

**Schema per region**:
```sql
CREATE TABLE messages (
    message_id UUID PRIMARY KEY,
    sender_id BIGINT NOT NULL,
    recipient_id BIGINT NOT NULL,
    content TEXT,
    sent_at TIMESTAMPTZ NOT NULL,
    home_region VARCHAR(20) NOT NULL
);
```

**Query patterns**:
1. `SELECT * FROM messages WHERE sender_id = ? AND recipient_id = ? ORDER BY sent_at DESC LIMIT 50` — 20,000 QPS per region.
2. `SELECT * FROM messages WHERE recipient_id = ? ORDER BY sent_at DESC LIMIT 50` — 50,000 QPS per region.
3. `SELECT * FROM messages WHERE content ILIKE ? AND sender_id = ?` — 100 QPS global.
4. Cross-region writes: 5,000 per second. A message written in EU-West for a US-East user must be searchable within 5 seconds.

**Constraints**: 1 billion messages per region. Local reads under 50ms. Global search under 2 seconds.

**Task**: Design the indexing and data distribution strategy. Consider two-tier indexing: local B+Tree indexes per region and a separate global search system for cross-region queries.

### Challenge 8: Mixed Read-Write Conflict Resolution

A ride-sharing pricing system updates prices based on real-time demand. The core table has one row per city:

```sql
CREATE TABLE ride_pricing (
    city_id INT NOT NULL PRIMARY KEY,
    surge_multiplier DECIMAL(3,2) NOT NULL DEFAULT 1.00,
    base_fare DECIMAL(8,2) NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Query patterns**:
1. Read pricing: `SELECT * FROM ride_pricing WHERE city_id = ?` — 500,000 QPS.
2. Update pricing: `UPDATE ride_pricing SET surge_multiplier = ?, updated_at = NOW() WHERE city_id = ?` — 10,000 updates per second.

**Conflict**: Same row read 50 times per write. Every update changes `updated_at`, preventing HOT updates and causing index churn.

**Task**: Design a solution for sub-10ms reads while sustaining 10,000 updates per second on the same row. Consider splitting hot columns, using a cache layer, or redesigning the schema.

## Final Evaluation

| Criterion | Tier 1 (L4) | Tier 2 (L5) | Tier 3 (L6+) |
|---|---|---|---|
| Index type selection | B+Tree vs Hash | GIN, GiST, BRIN, Partial | Distributed GSI, CDC-based |
| Storage estimation | Single-index calculation | Multi-index with overhead | Cross-region cost |
| Query coverage | All queries have index plan | P0 queries have covering indexes | Complex IN-list and JOIN patterns |
| Write overhead | Mentioned qualitatively | Quantified as ms per write | Modeled as throughput impact |
| Operational awareness | CONCURRENTLY | Bloat monitoring, periodic reindex | CDC propagation, consistency |
| Trade-off depth | Read vs write | Index type trade-offs | Distributed vs local indexing |
