## Tier 1: Foundation

These challenges test your understanding of the core data structures and algorithms that power every search system. You should be able to complete each one with pen and paper or a simple script.

### Challenge 1: Build the Inverted Index

You are building the search index for a small document set. Below are four documents. Your job is to manually construct the inverted index with term frequencies and posting lists.

**Documents:**

```
Doc 1: "The quick brown fox jumps over the lazy dog"
Doc 2: "A quick brown dog chased the fox"
Doc 3: "The lazy dog sleeps under the brown table"
Doc 4: "Quick foxes jump over fences"
```

**Tasks:**

1. Tokenize all four documents (lowercase, split on whitespace, remove punctuation)
2. Remove stop words (consider these stop words: "the", "a", "over", "under")
3. Apply stemming (quick, quicker, quickest all stem to "quick"; fox, foxes stem to "fox"; jump, jumps stem to "jump"; sleeps, sleep stem to "sleep")
4. Build the inverted index: for each unique term, list the document IDs and term frequencies as `docID:tf` pairs
5. For the query "quick brown fox," list which documents match and compute the intersection order starting from the rarest term

**Full Model Answer:**

**Step 1: Tokenize and lowercase.**

| Document | Tokens |
|----------|--------|
| Doc 1 | the, quick, brown, fox, jumps, over, the, lazy, dog |
| Doc 2 | a, quick, brown, dog, chased, the, fox |
| Doc 3 | the, lazy, dog, sleeps, under, the, brown, table |
| Doc 4 | quick, foxes, jump, over, fences |

**Step 2: Remove stop words (the, a, over, under).**

| Document | Tokens |
|----------|--------|
| Doc 1 | quick, brown, fox, jumps, lazy, dog |
| Doc 2 | quick, brown, dog, chased, fox |
| Doc 3 | lazy, dog, sleeps, brown, table |
| Doc 4 | quick, foxes, jump, fences |

**Step 3: Apply stemming.**

| Document | Tokens |
|----------|--------|
| Doc 1 | quick(1), brown(1), fox(1), jump(1), lazy(1), dog(1) |
| Doc 2 | quick(1), brown(1), dog(1), chase(1), fox(1) |
| Doc 3 | lazy(1), dog(1), sleep(1), brown(1), table(1) |
| Doc 4 | quick(1), fox(1), jump(1), fence(1) |

Note: "chased" stems to "chase" and "fences" stems to "fence." Since "jumps" becomes "jump" (already in Doc 1), Doc 1 now has "jump" instead of "jumps." Doc 4 already has "jump" from "jumps." This is consistent.

**Step 4: Inverted index with term frequencies.**

| Term | Posting List (docID:tf) | Document Frequency |
|------|------------------------|-------------------|
| quick | 1:1, 2:1, 4:1 | 3 |
| brown | 1:1, 2:1, 3:1 | 3 |
| dog | 1:1, 2:1, 3:1 | 3 |
| fox | 1:1, 2:1, 4:1 | 3 |
| jump | 1:1, 4:1 | 2 |
| lazy | 1:1, 3:1 | 2 |
| chase | 2:1 | 1 |
| sleep | 3:1 | 1 |
| table | 3:1 | 1 |
| fence | 4:1 | 1 |

**Step 5: Query "quick brown fox" intersection.**

Terms ordered by document frequency (rarest first): all three terms have document frequency 3, so tiebreak alphabetically: brown(3), fox(3), quick(3).

Intersection of posting lists:
- brown: [1, 2, 3]
- fox: [1, 2, 4]
- quick: [1, 2, 4]

Intersect brown and fox: [1, 2]
Intersect [1, 2] with quick: [1, 2]

Result: Documents 1 and 2 match all three terms.

**Common mistakes:**

- Forgetting to remove stop words before building the index. Stop words like "the" appear in almost every document, making them useless for retrieval.
- Inconsistent stemming: stemming "jumps" to "jump" in Doc 1 but not in Doc 4. The stemmer must apply the same transformation everywhere.
- Counting term frequency wrong: "the" appears twice in Doc 1 (positions 2 and 7), but it is a stop word so it is removed entirely.
- Not sorting posting lists. Posting lists must be sorted ascending for the intersection algorithm to work. If you listed Doc 1 and Doc 4 as [4, 1] for "fox," the intersection would fail.

### Challenge 2: Compute BM25 Scores

You have a small corpus of five documents. A user searches for "solar energy." Compute the BM25 score for each document.

**Corpus:**

```
Doc 1: "Solar panels convert sunlight into energy for homes"
Doc 2: "Renewable energy sources include solar and wind"
Doc 3: "Energy storage is critical for solar power adoption"
Doc 4: "Solar energy reduces carbon emissions significantly"
Doc 5: "Wind energy is cheaper than solar in some regions"
```

**Constants:**

- k1 (saturation parameter) = 1.2
- b (length normalization) = 0.75
- Average document length (avgdl) = 8 words (all documents are 8 words)
- Total number of documents (N) = 5
- IDF formula: `IDF(t) = ln(1 + (N - df(t) + 0.5) / (df(t) + 0.5))`
- BM25 formula: `BM25(doc, Q) = sum_over_terms IDF(t) * (tf(t,d) * (k1 + 1)) / (tf(t,d) + k1 * (1 - b + b * |d| / avgdl))`

**Tasks:**

1. For each term in the query ("solar", "energy"), compute its document frequency (df) across the corpus
2. Compute IDF for each term using the formula above
3. For each document, compute the BM25 score for the query
4. Rank the documents by BM25 score from highest to lowest
5. Which document has the highest score and why?

**Full Model Answer:**

**Step 1: Document frequency.**

Both "solar" and "energy" appear in all five documents. Check each document:

- Doc 1: "Solar panels convert sunlight into energy for homes" -- solar(1), energy(1)
- Doc 2: "Renewable energy sources include solar and wind" -- energy(1), solar(1)
- Doc 3: "Energy storage is critical for solar power adoption" -- energy(1), solar(1)
- Doc 4: "Solar energy reduces carbon emissions significantly" -- solar(1), energy(1)
- Doc 5: "Wind energy is cheaper than solar in some regions" -- energy(1), solar(1)

df(solar) = 5, df(energy) = 5.

**Step 2: Compute IDF.**

```
IDF(solar) = ln(1 + (5 - 5 + 0.5) / (5 + 0.5))
           = ln(1 + 0.5 / 5.5)
           = ln(1 + 0.0909)
           = ln(1.0909)
           = 0.0870

IDF(energy) = same calculation = 0.0870
```

Both terms have low IDF because they appear in every document. The IDF is low because neither term is discriminative in this corpus. In a real corpus with 1 million documents, "solar" might appear in 10,000 documents and "energy" in 100,000 documents, giving them different IDF values.

**Step 3: Compute BM25 for each document.**

All documents have 8 words (|d| = 8). Average document length is 8. So `|d| / avgdl = 1.0`, and the length normalization factor `(1 - b + b * |d| / avgdl) = (1 - 0.75 + 0.75 * 1.0) = 1.0`. Since all documents have the same length, length normalization has no effect in this example.

Each document has tf(solar) = 1 and tf(energy) = 1.

```
BM25 term contribution = IDF * (tf * (k1 + 1)) / (tf + k1 * 1.0)
                       = 0.0870 * (1 * 2.2) / (1 + 1.2)
                       = 0.0870 * 2.2 / 2.2
                       = 0.0870

Total BM25 = 0.0870 (solar) + 0.0870 (energy) = 0.1740
```

Every document scores 0.1740. All documents are tied.

**Step 4: Ranked results.**

1. Doc 1: 0.1740 (tie)
2. Doc 2: 0.1740 (tie)
3. Doc 3: 0.1740 (tie)
4. Doc 4: 0.1740 (tie)
5. Doc 5: 0.1740 (tie)

**Step 5: Analysis.**

All documents score identically because both query terms appear exactly once in every document and every document has the same length. This highlights a limitation of BM25: when terms are not discriminative (appear in all documents equally), BM25 cannot distinguish between documents. In a production system, the ranking would need additional signals to break ties, such as:

- **Field boost**: a document with "solar" in the title gets a higher score than one with "solar" only in the body
- **Proximity**: documents where "solar" and "energy" appear close together score higher
- **Popularity boost**: documents with more views or clicks get a boost
- **Recency boost**: newer documents rank higher

To demonstrate a more interesting BM25 calculation, recompute with a modified corpus where term frequencies differ. Suppose Doc 4 had "Solar solar energy reduces carbon emissions" (tf(solar) = 2):

```
BM25(solar) for Doc 4 = 0.0870 * (2 * 2.2) / (2 + 1.2)
                       = 0.0870 * 4.4 / 3.2
                       = 0.0870 * 1.375
                       = 0.1196

BM25(energy) for Doc 4 = 0.0870 * (1 * 2.2) / (1 + 1.2)
                       = 0.0870 * 2.2 / 2.2
                       = 0.0870

Total = 0.1196 + 0.0870 = 0.2066
```

Doc 4 now scores higher because the term "solar" appears twice, but the BM25 saturation means doubling the frequency does not double the score. The second occurrence adds only 37 percent more score than the first (0.1196 vs 0.0870). This is the saturation effect: the first occurrence of a term contributes the most, and subsequent occurrences contribute diminishing returns.

### Challenge 3: Simulate Posting List Intersection

You have three posting lists from an inverted index. Write a step-by-step trace of the intersection algorithm using skip pointers.

**Posting lists:**

```
Posting List A (term "quantum"): [2, 5, 8, 12, 17, 21, 30, 35, 42, 50, 55, 61, 68, 73, 80, 88]
Posting List B (term "computing"): [3, 8, 14, 21, 27, 35, 41, 50, 57, 64, 71, 78, 85, 92]
Posting List C (term "algorithm"): [1, 8, 15, 22, 29, 35, 42, 50]
```

**Skip pointer configuration:**

Each list has skip pointers that point to the entry 4 positions ahead. For example:
- Posting List A at index 0 skips to index 4 (value 17)
- Posting List A at index 4 skips to index 8 (value 42)
- And so on.

**Tasks:**

1. Intersect A and B first, then intersect the result with C
2. For each step, show the pointer positions before and after the advance
3. Record how many comparisons you would save compared to a naive linear scan without skip pointers
4. Show the final intersection result

**Full Model Answer:**

**Phase 1: Intersect A and B.**

Initialize pointers: pA = 0 (value 2), pB = 0 (value 3).

|------|----|-------|----|-------|--------|----------------------|

| 2 | 1 | 5 | 0 | 3 | Check skip: pA skip from 1 goes to 5 (value 30). 30 > 3. Advance pA to 2. | 2 |
| 3 | 2 | 8 | 0 | 3 | A[2]=8 > B[0]=3, advance pB | 3 |
| 4 | 2 | 8 | 1 | 8 | Match! Record 8. Advance both. | 4 |

| 10 | 6 | 30 | 5 | 35 | A[6]=30 < B[5]=35, advance pA | 11 |
| 11 | 7 | 35 | 5 | 35 | Match! Record 35. Advance both. | 12 |
| 12 | 8 | 42 | 6 | 41 | A[8]=42 > B[6]=41, advance pB | 13 |
| 13 | 8 | 42 | 7 | 50 | A[8]=42 < B[7]=50, advance pA | 14 |

| 15 | 10 | 55 | 8 | 57 | A[10]=55 < B[8]=57, advance pA | 16 |

A intersect B = [8, 21, 35, 50]

**Phase 2: Intersect [8, 21, 35, 50] with C = [1, 8, 15, 22, 29, 35, 42, 50].**

Initialize: pAB = 0 (value 8), pC = 0 (value 1).

| Step | pAB | Val | pC | C[pC] | Action | Comparisons |
|------|-----|-----|----|-------|--------|-------------|
| 1 | 0 | 8 | 0 | 1 | AB[0]=8 > C[0]=1, advance pC | 1 |
| 2 | 0 | 8 | 1 | 8 | Match! Record 8. Advance both. | 2 |
| 3 | 1 | 21 | 2 | 15 | AB[1]=21 > C[2]=15, advance pC | 3 |
| 4 | 1 | 21 | 3 | 22 | AB[1]=21 < C[3]=22, advance pAB | 4 |
| 5 | 2 | 35 | 3 | 22 | AB[2]=35 > C[3]=22, advance pC | 5 |
| 6 | 2 | 35 | 4 | 29 | AB[2]=35 > C[4]=29, advance pC | 6 |
| 7 | 2 | 35 | 5 | 35 | Match! Record 35. Advance both. | 7 |
| 8 | 3 | 50 | 6 | 42 | AB[3]=50 > C[6]=42, advance pC | 8 |
| 9 | 3 | 50 | 7 | 50 | Match! Record 50. End. | 9 |

Final intersection: [8, 35, 50]

**Skip pointer savings analysis:**

Without skip pointers, Phase 1 would require comparing every element in A (16) with elements in B (14) in the worst case, or approximately 26 comparisons at best. With skip pointers, you used 26 comparisons -- in this case, the skip pointers did not save comparisons because the lists are relatively small and the skip granularity (4) is coarse for lists of this size.

In a real search system with posting lists of 1 million entries and skip pointers every 128 entries, the savings are dramatic. For two lists of 1 million entries each, a naive linear scan requires up to 2 million comparisons. With skip pointers, the algorithm skips over blocks of up to 128 entries at a time, reducing the number of comparisons to approximately 1,500 to 3,000 -- a 500x to 1,000x reduction.

**Common mistakes:**

- Not advancing both pointers when a match is found. If you advance only one pointer, you will match the same element again on the next iteration.
- Starting the intersection with the largest list instead of the smallest. The result size is bounded by the smallest list, so starting with the largest list wastes comparisons.
- Forgetting that skip pointers must be checked before linear advance. The algorithm checks the skip target and only falls back to linear advance when the skip overshoots.

### Challenge 4: Enterprise Document Search

**Scenario:**

Your company has 100 million internal documents (Word files, PDFs, emails, wikis, code comments) totaling 5 TB. Employees need to search this corpus from a web interface. The requirements are:

- 2,000 queries per second during business hours
- P99 latency under 500 milliseconds
- Support phrase queries (exact match of a quoted phrase)
- Support field-scoped search (search only the title, only the author, or only the body)
- Support date range filtering
- Index freshness within 5 minutes of document creation
- Multi-tenant: 10 departments, each should only see their own documents

Design the search system. Cover indexing pipeline, query flow, data model, distributed architecture, and caching strategy.

**Evaluation Rubric:**

| Criterion | Exceptional (5 pts) | Proficient (3 pts) | Developing (1 pt) |
|-----------|-------------------|-------------------|-------------------|
| Indexing pipeline | Full pipeline with CDC, document parsing, text extraction, analysis chain for multiple file formats | Basic pipeline with batch indexing, single format support | No pipeline described or pipeline has critical gaps |
| Data model | Complete field mapping with analyzers per field type, doc_values, norms configuration | Basic mapping with field types but no analyzer tuning | Missing field types or inappropriate types |
| Distributed architecture | Detailed shard count, node roles, replication factor with rationale | Generic distributed design without specific numbers | Single node or vague distributed description |
| Query flow | Step-by-step flow with optimization details (query planning, filter pushdown, segment pruning) | High-level flow without optimization details | Missing key stages |
| Multi-tenant isolation | Tenant filter injection, dedicated indices or routing, resource quotas | Tenant-aware but no resource isolation | Tenant ID as a regular filter field only |
| Caching strategy | Multi-level cache with hit rate estimates and eviction policies | Single-level cache described | No cache strategy |

**Full Model Answer:**

**Indexing Pipeline:**

Documents originate from multiple sources: SharePoint (Word, PDF), Exchange (email), Confluence (wiki), and GitHub (code comments). A CDC pipeline captures changes from each source.

1. A document is created or modified in any source system
2. A change data capture connector (Debezium for databases, webhooks for SaaS sources) captures the event
3. The event is published to a Kafka topic: `document-changes`
4. The document ingestion service consumes the event, downloads the document binary, and extracts text using Apache Tika (supports Word, PDF, HTML, email, and code formats)
5. The extracted text is analyzed: the standard tokenizer for body text, the English analyzer with stemming for English documents, a keyword analyzer for author and department fields
6. The analyzed document is indexed into Elasticsearch with a bulk index request (batch size: 500 documents or 5 MB, whichever is smaller)
7. The document is searchable within 5 minutes (triggered by a scheduled refresh or immediate flush)

**Data Model:**

```json
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "english",
        "fields": {
          "keyword": { "type": "keyword" }
        },
        "norms": true
      },
      "body": {
        "type": "text",
        "analyzer": "english",
        "norms": false
      },
      "author": {
        "type": "keyword",
        "doc_values": true
      },
      "department": {
        "type": "keyword",
        "doc_values": true
      },
      "created_at": {
        "type": "date",
        "format": "yyyy-MM-dd HH:mm:ss"
      },
      "source_type": {
        "type": "keyword"
      },
      "file_size": {
        "type": "long",
        "doc_values": true
      },
      "permissions": {
        "type": "keyword"
      }
    }
  },
  "settings": {
    "number_of_shards": 20,
    "number_of_replicas": 1,
    "refresh_interval": "60s"
  }
}
```

**Design rationale for key decisions:**

- `body.norms: false` saves memory because body length normalization is not useful for this use case
- `title` has both `text` and `keyword` sub-fields so you can search in title text and sort by title alphabetically
- `department` and `author` are `keyword` for exact matching and aggregation
- `permissions` as `keyword` enables document-level security filtering
- Refresh interval of 60 seconds instead of the default 1 second: the requirement is 5-minute freshness, so a 60-second refresh provides a healthy safety margin while reducing segment count by 60x compared to the default. Fewer segments means faster queries.

**Distributed Architecture:**

- 100 million documents at 5 TB raw data
- Inverted index overhead: 2.5x (12.5 TB)
- With 1 replica: 25 TB total
- 20 shards across 5 data nodes: 4 shards per node. Each node stores 4 primary shards and 4 replica shards
- Each data node: 64 GB RAM, 16 CPU cores, 2 TB SSD storage
- 2 coordinator nodes: 32 GB RAM, 8 CPU cores (stateless, only route queries and merge results)
- 3 master nodes: 16 GB RAM, 4 CPU cores (cluster management only)

**Query Flow:**

1. Employee types a query in the search UI: `"quarterly report" department:finance after:2025-01-01`
2. Query reaches the coordinator node via the load balancer
3. Coordinator parses the query: phrase query for "quarterly report," filter on department=finance, date range filter on created_at
4. Query rewriting: no spell correction needed (quoted phrase means exact match)
5. Coordinator checks the query cache. If the same query was executed in the last 60 seconds, return cached results
6. Coordinator fans out the query to all 20 shards (primary or replica, whichever is available and least loaded)
7. Each shard executes the query against its local segment
8. Segment pruning: segments whose date range is entirely before 2025-01-01 are skipped
9. Term lookup for "quarterly" and "report" in the inverted index. Since it is a phrase query, positions are checked
10. Result set: documents containing both "quarterly" and "report" as adjacent terms
11. Filter application: bitmap AND for department=finance AND date >= 2025-01-01
12. BM25 scoring with field boost: title matches get 3x boost over body matches
13. Each shard returns its top 20 results to the coordinator
14. Coordinator merges results from all shards into a globally sorted top 20
15. Coordinator fetches document metadata (title, author, date, snippet) from stored fields
16. Response is returned to the UI

Estimated latency per stage:
- Query parsing and planning: 2 ms
- Fan-out to 20 shards: 1 ms (network)
- Shard-level execution (segment pruning + lookup + filter + scoring): 40 ms
- Result merging: 3 ms
- Response building: 4 ms
- Total: 50 ms (well under the 500 ms requirement)

**Multi-Tenant Isolation:**

Each department's documents are tagged with a `department` field. The search service intercepts every query and injects a filter: `department:<user_department>`. This filter is added to the query tree before execution, not applied as a post-filter. Tenant filter injection happens at the API gateway level, so users cannot bypass it even with direct API access.

For resource isolation, departments with high query volume (Engineering, Sales) get dedicated indices on dedicated shards. Smaller departments share shards with dedicated routing. Resource quotas limit indexing rates and query concurrency per department.

**Caching Strategy:**

Three-level cache:

1. **Application-level cache (Redis)**: caches complete query results for frequent queries. TTL of 60 seconds. Expected hit rate: 25 percent. Covers queries like "HR policies" that many users search for daily.

2. **Node-level query cache**: Elasticsearch's built-in query cache caches the results of recent queries at the node level. TTL of 60 seconds. Expected hit rate: 20 percent.

3. **Filter cache**: caches the bitmaps for individual filter values. The filter for "department:finance" is computed once and reused across all queries from the finance department. This cache is never evicted for popular departments. Expected hit rate: 90 percent.

Combined cache effect: 40 percent of queries are served partly or entirely from cache. The average query latency drops from 50 ms to 15 ms for cached queries.

### Challenge 5: E-Commerce Search with Autocomplete

**Scenario:**

You are building search for an e-commerce platform with 10 million products. The search powers the main shopping experience. Requirements:

- 5,000 queries per second during peak (Black Friday)
- P99 latency under 150 milliseconds
- Autocomplete suggestions after 2 characters typed (first 2 characters return broad category suggestions, 3+ characters return product suggestions)
- Faceted navigation: brand, category, price range, rating, size, color
- Personalized ranking based on user purchase history
- Real-time inventory-aware results (out-of-stock items demoted)
- Support for typo tolerance: "nike" should match "Nike," "nkie" should suggest "Nike"

Design the system with special focus on the autocomplete pipeline and the personalized ranking layer.

**Evaluation Rubric:**

| Criterion | Exceptional (5 pts) | Proficient (3 pts) | Developing (1 pt) |
|-----------|-------------------|-------------------|-------------------|
| Autocomplete | Prefix trie with edit distance, tiered completion (categories first, then products), debounced queries, traffic estimates | Basic completion suggestions without tiers | No autocomplete design or single-strategy approach |
| Typo tolerance | Levenshtein automaton with configurable edit distance per string length, prefix-based correction | Simple fuzzy match | No typo tolerance |
| Personalized ranking | Multi-signal ranking with user history, purchase data, browsing, and real-time signals | Simple personalization (recent purchases only) | No personalization |
| Inventory awareness | Real-time inventory feed, two-phase ranking (score then demote) | Batch inventory sync with stale data | No inventory integration |
| Faceted navigation | Precomputed facet counts with sampling for large result sets | On-the-fly facet computation | Facets as post-filter only |
| Scale and performance | Specific numbers for index size, shard count, node count, cache sizes | Vague scaling estimates | No scaling discussion |

**Full Model Answer:**

**Autocomplete Pipeline:**

The autocomplete system has two tiers.

**Tier 1: Category Completion (2 characters typed).**

When the user types 2 characters (e.g., "ni"), show broad category suggestions:
- "Nike" (brand)
- "Nike Running Shoes" (category+brand)
- "Nintendo" (brand)

These suggestions come from a small, frequently accessed prefix trie stored in Redis. The trie is built from the top 10,000 search queries from the last 30 days, augmented with category and brand names. The trie nodes store the prefix string, expected completion count (how many products match this prefix), and top 5 completions precomputed.

The category completion trie is 2 MB in memory. It serves 10,000 requests per second with sub-millisecond latency.

**Tier 2: Product Completion (3+ characters typed).**

When the user types 3+ characters, show actual product suggestions. This tier uses a dedicated completion index in Elasticsearch with the `completion` suggester. The completion index is updated in near real-time (refresh interval: 5 seconds for completions, which can be slightly stale compared to the main index).

Top 5 product suggestions are returned within 5 to 15 milliseconds.

**Debouncing and traffic estimation:**

- Autocomplete fires on every keystroke, debounced at 150 milliseconds
- Average query length is 7 characters, so each full search produces approximately 5 autocomplete queries (characters 2 through 7, minus debounce skips)
- 5,000 full searches per second produce 25,000 autocomplete queries per second
- Each autocomplete query returns within 15 milliseconds
- CDN caching: top 1,000 autocomplete queries are cached at the edge for 30 seconds. Cache hit rate: 60 percent.
- Cached autocomplete queries are served from the CDN in under 5 milliseconds.

**Typo Tolerance:**

The system uses a Levenshtein automaton with the following edit distance rules:

- 1-2 character queries: no typo tolerance (too short, too many false positives)
- 3-4 character queries: maximum 1 edit (insertion, deletion, substitution)
- 5+ character queries: maximum 2 edits

The typo tolerance is implemented as a prefix trie augmented with Levenshtein automata. When the user types "nkie":

1. The system computes that "nkie" has edit distance 1 from "nike" (one substitution: k before i should be reversed)
2. The top suggestion becomes "Nike" with corrected spelling
3. The search results are for "Nike" but the UI shows "Showing results for _Nike_. Search instead for _nkie_?"

The typo tolerance trie is built from the product catalog and stored in memory (approximate size: 500 MB for 10 million products). Updates are applied daily during low-traffic hours.

**Personalized Ranking:**

Personalized ranking uses a two-phase approach.

**Phase 1: Candidate generation (all users, 50 ms budget).**

All matching products are scored using BM25 with basic boosts (recency, popularity). The top 500 candidates are passed to Phase 2.

**Phase 2: Personalized reranking (per user, 30 ms budget).**

Each user has a personalization profile stored in Redis. The profile includes last 50 purchased products (product IDs and categories), last 200 viewed products, last 50 searches, top 3 most-purchased brands and categories, and price range preferences (computed from purchase history).

The personalization score is computed as a weighted combination:

```
personalization_score = 0.3 * brandAffinity_brand
                      + 0.25 * categoryAffinity_category
                      + 0.15 * priceRangeFit_price
                      + 0.1 * recencyBoost
                      + 0.1 * popularityBoost
                      + 0.1 * randomExploration
```

The final score is a weighted blend of BM25 and personalization:

```
final_score = 0.6 * bm25_score + 0.4 * personalization_score
```

The blend ratio is different for new users (more BM25, less personalization since there is no history) versus returning users (more personalization).

**Inventory Awareness:**

Inventory data flows from the warehouse management system to the search index through a CDC pipeline with 10-second latency. Each product has an `in_stock` boolean field and a `stock_count` integer field.

Out-of-stock products are not removed from the index (they should still be findable by direct link or back-in-stock notifications). Instead, in-stock products receive a 2x ranking boost.

```
if (product.in_stock) {
    final_score *= 2.0
} else {
    final_score *= 0.5
}
```

During Black Friday, when stock changes rapidly, the CDC replication lag is monitored. If the lag exceeds 30 seconds, the search switches to a degraded mode: all products are treated as in-stock for ranking purposes, and the actual availability is checked at checkout time.

**Distributed Architecture for Scale:**

- 10 million products, average 2 KB per document (including descriptions, specs, images URLs) = 20 GB raw data
- With inverted index, stored fields, doc values: approximately 60 GB total
- With 1 replica: 120 GB
- 6 data nodes, each with 64 GB RAM and 8 CPU cores
- 12 shards (2 per node), each approximately 5 GB primary
- Each shard serves 416 queries per second (5,000 QPS / 12 shards)
- Each shard can handle 500 to 1,000 QPS comfortably on modern hardware

**Faceted Navigation:**

Facet counts are computed on the candidate set after the query phase but before the personalized reranking phase. This ensures facet counts are accurate for the current query.

For queries with more than 10,000 matching products, facet computation uses sampling: a random 10 percent of the candidate set is sampled, and facet counts are extrapolated. The error margin is under 5 percent for facets with at least 1,000 results, and the latency savings are 90 percent (from 100 ms to 10 ms).

High-cardinality facets (like product size: 50+ unique values) are limited to the top 10 values by count. The remaining values are grouped into "Other."

### Challenge 6: Real-Time Social Media Search

**Scenario:**

You are building the search feature for a social media platform with 500 million monthly active users. The platform's content includes posts, comments, images, videos, and user profiles. Requirements:

- 50,000 queries per second
- P99 latency under 200 milliseconds
- Content must be searchable within 2 seconds of posting
- Search must support text, mentions (@username), hashtags (#topic), and media type filters
- Trending topics must be identifiable from search volume
- Content moderation: flagged or removed content must disappear from search within 5 seconds
- Multi-language support: English, Spanish, Arabic, Japanese, and Hindi

Focus on the real-time indexing challenges, the trending topics pipeline, and the content moderation integration.

**Evaluation Rubric:**

| Criterion | Exceptional (5 pts) | Proficient (3 pts) | Developing (1 pt) |
|-----------|-------------------|-------------------|-------------------|
| Real-time indexing | Near-real-time indexing with refresh interval under 1 second, in-memory buffer, concurrent segments | Batch indexing with 10+ second delay | No mention of indexing latency |
| Trending topics | Two-phase detection: real-time counting in Redis + historical aggregation in analytics cluster, decay function | Simple count-based trending | No trending detection |
| Content moderation | Immediate deletion pipeline: reverse index lookup, segment deletion markers, write-ahead log for consistency | Deletion on next refresh only | No content moderation integration |
| Multi-language | Language-specific analyzers per field, language detection, CJK tokenizer, Arabic stemmer, Hindi normalizer | Single analyzer for all languages | No language support |
| Write throughput | 100,000 writes/second handling: bulk indexing, segment merging strategy, write queue with backpressure | Single writes acknowledged individually | No write throughput consideration |
| Query types | Type-specific indexing for text, @mentions, #hashtags, media filters | Generic text search for all types | Supports text search only |

**Full Model Answer:**

**Real-Time Indexing Pipeline:**

The platform generates 100,000 posts per second during peak hours. Each post is 500 bytes to 5 KB (text + metadata). The indexing pipeline must keep up with this write volume while maintaining sub-2-second searchability.

**In-memory buffer with near-real-time refresh:**

Each data node maintains an in-memory indexing buffer of 256 MB. When a post is created:

1. The post is written to the primary shard's in-memory buffer
2. The write is acknowledged to the client immediately (within 2 milliseconds)
3. The buffer is refreshed to a searchable segment every 500 milliseconds
4. When the buffer reaches 256 MB (approximately 50,000 posts), it is forced to flush

Since each node handles approximately 20,000 writes per second with a 256 MB buffer, the buffer fills in approximately 3 seconds. The forced flush at 256 MB creates segments of approximately 50,000 documents each.

Segment merging runs continuously in the background. The tiered merge policy combines segments of similar sizes. Merges run on dedicated merge threads (2 per node) to avoid blocking indexing or querying threads.

**Sharding and routing:**

- 500 million users, average 200 posts per user per month = 100 billion posts per month
- Average post metadata size: 200 bytes
- 100 billion posts * 200 bytes = 20 TB per month of indexed data
- With 30-day hot index and 6-month warm index: 120 TB hot + 600 TB warm
- 200 shards for the hot index (30 days of data), each 30 GB
- 20 data nodes (with 64 GB RAM), 10 shards per node
- Replication factor: 2 (tolerates 1 node failure per shard group)

**Writing 100,000 posts per second:**

Each write is a single document index request. 100,000 writes per second spread across 200 shards means 500 writes per second per shard. Each shard can handle 2,000 to 5,000 writes per second, so the write capacity is sufficient.

The bottleneck is the cluster coordinator, which must route 100,000 requests per second. Two strategies:

1. **Bulk indexing**: the API gateway batches writes into groups of 1,000 and sends bulk index requests. This reduces the coordinator load from 100,000 requests per second to 100 bulk requests per second.

2. **Distributed coordinator**: 5 coordinator nodes behind a load balancer, each handling 20,000 writes per second.

**Query Types: Specialized Indexing.**

The search system needs to handle different query types with appropriate indexing strategies.

- **Text search**: standard inverted index with language-specific analyzers. The English analyzer for English posts, the Arabic analyzer for Arabic posts, the Japanese (kuromoji) analyzer for Japanese posts, and the Hindi analyzer for Hindi posts.

- **@mention search**: mentions like @username are indexed as a separate `keyword` field. The field is indexed with a `lowercase` normalizer so @User123 and @user123 both match. The inverted index for mentions is kept separate from the text index because mention queries are exact match, not fuzzy.

- **#hashtag search**: hashtags like #breakingnews are indexed as a `keyword` field with lowercase normalization. Hashtags also feed into the trending detection pipeline. A separate completion suggester on hashtags enables typeahead as the user types "#bre..."

- **Media type filter**: `media_type` is a `keyword` field with values: text, image, video, link. Filter queries use roaring bitmaps.

**Multi-Language Support:**

The index uses a multi-field mapping for the post body. A language detection step runs before indexing. The post body is passed through a language detection model (Compact Language Detector v3 or a lightweight ML model). The detected language is stored in the `detected_language` field. During query time, the search runs against all language-specific fields, but language-appropriate fields get a 2x boost. A Japanese query gets results from all fields, but results from the Japanese field are boosted.

This approach handles multilingual content well: an Arabic speaker can search in Arabic and find Arabic results ranked highest, but English results about the same topic still appear.

**Trending Topics Pipeline:**

Trending topics are identified through a two-phase pipeline.

**Phase 1: Real-time counting (sub-second latency).**

Every time a post is indexed, its hashtags are extracted and sent to a Redis cluster. Redis maintains a sorted set for each rolling time window:

```
ZINCRBY trending:1min "#breakingnews" 1
ZINCRBY trending:5min "#breakingnews" 1
ZINCRBY trending:1hour "#breakingnews" 1
```

The 1-minute window captures sudden spikes. The 5-minute window smooths out noise. The 1-hour window identifies sustained trends.

**Phase 2: Historical aggregation (1-minute latency).**

Every 60 seconds, the top 100 hashtags from each time window are read from Redis and written to a separate trending analytics index in Elasticsearch. This index stores historical trending data for the last 30 days. The data is used for "rising" trends (hashtags that are trending faster than they were an hour ago), trend decay rates (how long a trend stays in the top 10), and trend prediction (forecasting which hashtags will trend next).

**Scoring function for trending:**

```
trend_score = current_count / baseline_count * decay_factor
decay_factor = exp(-time_since_first_appearance / half_life)
```

The decay factor prevents the same hashtag from trending indefinitely. A hashtag that appeared 2 hours ago with a 30-minute half-life gets a decay factor of `exp(-120 / 30) = exp(-4) = 0.018`. Even if the current count is high, the trend score is low because the hashtag has been around too long. This makes room for new, fresh trends.

**Content Moderation Integration:**

When a post is flagged and removed by the moderation team, it must disappear from search within 5 seconds. The deletion pipeline:

1. Moderator action triggers a deletion event in the moderation service
2. The event is published to a Kafka topic: `content-removals`
3. The search deletion consumer reads the event and gets the post ID
4. The deletion consumer issues a delete-by-query request to Elasticsearch
5. Elasticsearch marks the document as deleted (logical deletion) and refreshes the affected segment
6. The deletion is acknowledged within 3 seconds

For bulk removals (entire user account deleted), the system uses a delete-by-query with a filter on the author ID. Deleted documents are marked as deleted in the segment but not physically removed until the segment is merged. During this time, the deleted marker is checked during query execution. The performance impact of deleted documents is minimal (Lucene handles them efficiently, and the index maintains a "live docs" bitset).

For the most urgent removal cases (illegal content, policy violations), the system performs an immediate segment refresh on the affected shards. This ensures the deletion is visible within 1 second at the cost of creating additional small segments. The small segments are merged in the background within 60 seconds.

**Estimated resource requirements:**

- 20 data nodes (64 GB RAM, 16 CPU cores, 2 TB SSD each)
- 5 coordinator nodes (32 GB RAM, 8 CPU cores)
- 3 master nodes (16 GB RAM, 4 CPU cores)
- 3 Redis nodes (64 GB RAM) for trending and caching
- 5 Kafka nodes for the event streaming pipeline

Total cost estimate: approximately 40 to 60 thousand dollars per month in cloud infrastructure (AWS or GCP).

### Challenge 7: Build vs. Buy for a Search Platform

**Scenario:**

You are the infrastructure lead at a mid-stage startup with 200 employees and 5 million users. The company runs a content platform where users create and share documents (similar to Notion or Google Docs). Currently, search is implemented using PostgreSQL full-text search (tsvector), but as the document count has grown to 50 million, search quality and latency are degrading.

Your CEO has given you a budget of 500,000 dollars for the year to fix search. You have a team of 5 backend engineers who are currently maintaining the PostgreSQL search and building features. You must decide between three options:

**Option A: Elasticsearch Cloud.** 200,000 dollars per year for a managed Elasticsearch cluster. Setup time: 2 weeks. Ongoing maintenance: minimal (the cloud provider handles operations). Configuration and tuning will require 1 engineer 50 percent time.

**Option B: Custom Search Engine in Rust.** Build a search engine from scratch. Engineering effort: 4 engineers for 6 months (approximately 400,000 dollars in salary cost). Ongoing maintenance: 1 engineer full-time. After 6 months, you have a custom search engine optimized for your specific workload.

**Option C: Algolia.** 300,000 dollars per year for Algolia's search-as-a-service. Setup time: 1 week. Ongoing maintenance: minimal. Configuration and tuning requires 1 engineer 25 percent time.

Present your recommendation with specific trade-off analysis across these dimensions:

1. Cost (year 1 and year 3)
2. Time to market
3. Control and customization
4. Scalability headroom
5. Vendor lock-in risk
6. Team capability and focus

Then, for your chosen option, design the migration strategy from PostgreSQL full-text search to the new system with zero downtime.

**Trade-offs Discussion:**

There is no universal correct answer. The best choice depends on the company's stage, growth rate, and strategic priorities. Here is the analysis for each option.

**Option A: Elasticsearch Cloud -- The Balanced Choice.**

Cost:
- Year 1: 200,000 dollars (Elastic Cloud) + 100,000 dollars (0.5 engineer) = 300,000 dollars
- Year 3: 200,000 dollars + 100,000 dollars = 300,000 dollars per year (assuming no significant growth in cluster size)
- Total over 3 years: 900,000 dollars

Time to market: 2 weeks for initial setup. 4 weeks for migration. Total: 6 weeks.

Control and customization: Medium. Elasticsearch is highly configurable but within the limits of its architecture. Custom scoring plugins require Java development. The managed service limits access to some low-level settings.

Scalability headroom: Elasticsearch scales to hundreds of nodes. For 50 million documents (approximately 200 GB), a 6-node cluster handles 5,000 QPS comfortably. Headroom for 5x growth without architecture changes.

Vendor lock-in risk: Medium. Elasticsearch is open source. If the managed service becomes too expensive, you can self-host. However, migration to a different search engine (like Solr) would require significant effort because the query DSL and configuration are different.

Team capability: Your team is unlikely to have Elasticsearch expertise, but the learning curve is manageable. Elasticsearch has extensive documentation and community support. One engineer can become productive in 2 weeks.

**Option B: Custom Search Engine in Rust -- The Strategic Choice.**

Cost:
- Year 1: 400,000 dollars (4 engineers x 6 months) + 200,000 dollars (1 engineer full-time for remaining 6 months) = 600,000 dollars -- exceeds the 500,000 dollar budget
- Year 3: 200,000 dollars (1 engineer full-time) + 100,000 dollars (infrastructure) = 300,000 dollars per year
- Total over 3 years: 1,200,000 dollars (significantly more than other options)

Time to market: 6 months for the initial version. Migration: 2 months. Total: 8 months. Your users suffer with poor search quality for almost a year.

Control and customization: Maximum. You can implement any ranking algorithm, any query parsing, any indexing strategy. If your platform has unique requirements (like real-time collaborative document search with version history), a custom engine can handle them better than any off-the-shelf solution.

Scalability headroom: You control the architecture. You can design for your specific workload. But you must build sharding, replication, and fault tolerance yourself -- features that Elasticsearch has spent 15 years perfecting.

Vendor lock-in risk: Zero. You own the code. But you are now locked into your own code. If the engineer who built the search engine leaves, knowledge is lost.

Team capability: Building a search engine requires deep expertise in information retrieval, data structures (inverted index, BKD trees, roaring bitmaps), distributed systems, and concurrent programming. Your 5 backend engineers are currently doing feature development. Asking them to build a search engine means zero feature development for 6 months. The risk of building something with fundamental design flaws is high.

**Option C: Algolia -- The Fast Choice.**

Cost:
- Year 1: 300,000 dollars (Algolia) + 50,000 dollars (0.25 engineer) = 350,000 dollars
- Year 3: At 5 million users growing 50 percent per year, user count reaches 11 million by year 2 and 17 million by year 3. Algolia pricing is based on search operations and index size. Estimate: 500,000 dollars per year by year 3.
- Total over 3 years: 1,250,000 dollars (most expensive over time)

Time to market: 1 week setup, 3 weeks migration. Total: 4 weeks.

Control and customization: Low. Algolia provides a fixed set of features. You cannot customize the scoring algorithm, the indexing pipeline, or the query parser. If your platform needs a feature Algolia does not support (like phrase queries with slop), you are stuck.

Scalability headroom: Algolia handles the scaling for you. You do not need to provision nodes or tune shards. But the cost scales linearly with usage.

Vendor lock-in risk: High. Algolia uses a proprietary API and data format. Migrating away from Algolia requires a complete reindex and new query logic in your application code. The custom C++ engine is opaque -- you cannot debug slow queries or index issues yourself.

Team capability: No specialized search knowledge required. The API is simple and well-documented. Your team stays focused on feature development.

**Recommendation:**

For a mid-stage startup with 5 million users and a 500,000 dollar budget, **Option A (Elasticsearch Cloud) is the best choice.** Here is the rationale:

1. Year 1 budget fits: 300,000 dollars leaves 200,000 dollars for other initiatives.
2. Time to market is 6 weeks: your users get improved search in less than 2 months, not 8 months.
3. Your team stays focused: one engineer learns Elasticsearch part-time. The other 4 engineers continue building features.
4. Scalability headroom: Elasticsearch handles 5x growth without re-architecture.
5. Exit strategy: if Elastic Cloud becomes too expensive, you can self-host the open-source version. If you outgrow Elasticsearch entirely (unlikely at this scale), the migration path to Solr is well-documented.
6. Community and hiring: Elasticsearch expertise is common. If the search engineer leaves, you can hire a replacement.

**When Option B (Custom Engine) would be better:**

- Your platform has extreme search requirements that Elasticsearch cannot meet (for example, searching within real-time collaborative documents with per-character indexing)
- Your company has deep information retrieval expertise in-house
- Search is a core competitive differentiator for your product
- You have a budget of 2 million dollars+ and can afford the development time

**When Option C (Algolia) would be better:**

- Your team has zero bandwidth for infrastructure work
- Your search requirements are simple (filter + text search, no complex ranking)
- You expect moderate growth and can absorb the increasing cost
- Time to market is the highest priority

**Migration Strategy (for chosen Option A):**

The migration from PostgreSQL full-text search to Elasticsearch must happen with zero downtime. Here is the migration plan.

**Phase 1: Backfill (Week 1-2).**

1. Set up the Elasticsearch cluster on Elastic Cloud (6 data nodes, 9 shards, 1 replica)
2. Define the index mapping based on the document schema in PostgreSQL
3. Write a backfill script that reads all 50 million documents from PostgreSQL in batches of 10,000 and indexes them into Elasticsearch
4. The backfill runs at 5,000 documents per second and completes in approximately 3 hours
5. During backfill, PostgreSQL continues to serve all search queries

**Phase 2: Dual-Write (Week 3-4).**

1. Deploy a feature flag: `use_elasticsearch`. The flag is initially off for all traffic
2. Modify the document create/update/delete logic to write to both PostgreSQL and Elasticsearch
3. The primary write path remains PostgreSQL. Elasticsearch writes are fire-and-forget (log errors but do not block the primary write)
4. Run a reconciliation job every hour that compares document counts and flags discrepancies
5. Monitor Elasticsearch query latency and error rates under load testing (simulate 5,000 QPS)

**Phase 3: Shadow Reads (Week 5).**

1. For 10 percent of search traffic (random sampling), send the query to both PostgreSQL and Elasticsearch, but only show PostgreSQL results to the user
2. Compare result quality: for each shadow query, measure response latency (both systems), result count and precision at top 10, and whether the user clicked a result and how far down in the result set
3. Fix any issues discovered during shadow testing (mapping errors, scoring discrepancies, missing results)

**Phase 4: Cutover (Week 6).**

1. Set the `use_elasticsearch` flag to on for 1 percent of traffic
2. Monitor: query latency, error rate, user engagement metrics (click-through rate, bounce rate, search abandonment)
3. Gradually increase to 5 percent, then 25 percent, then 50 percent, then 100 percent over 3 days
4. Each increase happens only after confirming metrics are stable for 4 hours
5. Once 100 percent of traffic is on Elasticsearch, keep dual-writes to PostgreSQL for 2 weeks as a rollback option

**Rollback Plan:**

If Elasticsearch has issues during the cutover:

1. Set the `use_elasticsearch` flag to off for all traffic (instant rollback to PostgreSQL)
2. All documents written during the cutover period are still in PostgreSQL (dual-write ensures this)
3. Investigate the issue while users continue to use PostgreSQL search
4. Fix the issue and restart the cutover from Phase 2 (dual-write is already in place)

**Post-Migration (Week 7+):**

- Remove the dual-write to PostgreSQL (stop writing documents to PostgreSQL search, keep documents in PostgreSQL as the source of truth)
- Decommission the PostgreSQL full-text index (reclaim storage)
- Enable advanced Elasticsearch features: autocomplete, faceted navigation, personalized ranking (features that were not available in PostgreSQL search)
- Set up monitoring dashboards for search performance and relevance metrics

### Challenge 8: Multi-Region Search with Compliance Constraints

**Scenario:**

Your company operates a global e-commerce platform with customers in the United States, European Union, and China. You have 500 million products and 100,000 queries per second globally. You currently run a single Elasticsearch cluster in US-East (Virginia), and you are experiencing the following problems:

1. **Latency**: users in China experience 800 millisecond average search latency due to trans-Pacific round trips
2. **Compliance**: the EU requires that European user data stays within European borders (GDPR). China requires that the search service for Chinese users operates within China (data sovereignty laws)
3. **Consistency**: product inventory changes rapidly (prices change hourly, stock levels change every few seconds). A user who adds an item to their cart should see accurate stock levels when they search again. But cross-region replication causes 10 to 30 second delays
4. **Cost**: the single cluster in Virginia costs 1.2 million dollars per year. Adding two more regions would triple the infrastructure cost

Design a multi-region search architecture that addresses all four problems. Discuss the trade-offs between consistency, latency, compliance, and cost. Provide specific estimates for the additional cost of the multi-region solution.

**Trade-offs Discussion:**

This challenge requires balancing four competing constraints: latency (users need fast results), compliance (data must stay in-region), consistency (inventory must be accurate), and cost (budget is finite). There is no perfect solution. Every decision creates a trade-off.

**Proposed Architecture: Three-Region Active-Active with Local Differences.**

Region Setup:

| Region | Location | Purpose | Compliance |
|--------|----------|---------|------------|
| US-East | Virginia | US users, rest-of-world fallback | US data residency |
| EU-West | Frankfurt | EU users, EU product data | GDPR compliance (data stays in EU) |
| AP-Southeast | Singapore | Asia-Pacific users | No data sovereignty issues |
| China | Beijing | Chinese users only (operated through local partner) | China data sovereignty |

**Data Partitioning Strategy:**

Not all product data needs to be in every region. The catalog of 500 million products is partitioned:

- **Global catalog** (400 million products): available in all regions. These are products from international brands that ship worldwide. Stored in every region's index.
- **Regional catalog** (100 million products): available only in specific regions. These are products from local sellers, regional brands, and items that cannot ship internationally. Stored only in the relevant region's index.

When a user searches, the query automatically searches both the global catalog and their region's local catalog. Results from the local catalog receive a 1.5x relevance boost.

**Data Flow for Each Region:**

**Indexing path:**
1. Sellers update product information through the API
2. The API routes the request to the nearest region (based on the seller's location)
3. Each region has its own Kafka cluster for change data capture
4. The indexing service in each region indexes data into the local Elasticsearch cluster
5. Cross-region replication sends global catalog changes to other regions:
   - US-East to EU-West: 80 millisecond replication latency (fiber path)
   - EU-West to AP-Southeast: 150 millisecond replication latency
   - AP-Southeast to China: 200 millisecond replication latency (through VPN/proxy)

**Compliance enforcement:**
- EU sellers' data is tagged with `data_origin: eu`. This tag prevents the data from being indexed in non-EU regions. The CDC pipeline in EU-West filters out non-EU data from replication
- Chinese user data (search queries, click history, purchase data) never leaves China. The China cluster operates independently
- US-East stores data tagged as `data_origin: us` and `data_origin: international`. It also stores aggregated (non-personal) product data for global catalog products

**Consistency vs. Latency Trade-off:**

The critical consistency challenge is inventory accuracy. A search query should reflect recent inventory changes.

**Approach: Local-first inventory with best-effort global sync.**

Each region maintains its own inventory index. Inventory changes are:

1. Applied immediately to the local region's index (sub-second freshness)
2. Replicated asynchronously to other regions (10 to 30 second delay)

When a user searches in EU-West, they see inventory that is:
- Accurate for EU-West warehouses (sub-second freshness)
- Stale for US-East warehouses (up to 30 seconds stale)

If a user in EU-West searches for a product that is only available in a US-East warehouse, the search shows the last known inventory from US-East (up to 30 seconds stale). If the user tries to purchase the item, the order system checks real-time availability before confirming.

**Trade-off accepted:** users see slightly stale inventory for cross-region products. This is acceptable because:
1. Only 5 percent of searches involve cross-region products (most users search for locally available products)
2. The order system performs the final inventory check, preventing overselling
3. Users prefer fast search with slightly stale data over slow search with accurate data

**Latency Analysis:**

| User Location | Before (single US-East cluster) | After (multi-region) | Improvement |
|--------------|-------------------------------|---------------------|-------------|
| US-East | 50 ms | 50 ms | 0 percent |
| EU-West | 120 ms | 60 ms | 50 percent |
| AP-Southeast | 250 ms | 80 ms | 68 percent |
| China | 800 ms | 150 ms (in-country) | 81 percent |

The multi-region architecture reduces global average latency from 150 ms to 60 ms. P99 latency drops from 500 ms to 180 ms.

**Cost Analysis:**

| Component | Single Region | Multi-Region (4 regions) |
|-----------|--------------|-------------------------|
| Elasticsearch cluster | 600,000 dollars (US-East) | 1,800,000 dollars (450,000 dollars x 4) |
| Coordinator nodes | 150,000 dollars | 450,000 dollars (150,000 dollars x 3) |
| Cross-region data transfer | 0 dollars | 400,000 dollars |
| Compliance auditing | 50,000 dollars | 200,000 dollars |
| Operational overhead | 200,000 dollars (2 SREs) | 600,000 dollars (1 SRE per region + 1 global SRE) |
| China infrastructure | 0 dollars | 400,000 dollars (through local partner at premium pricing) |
| **Total** | **1,000,000 dollars** | **3,850,000 dollars** |

The multi-region solution costs 3.85x the single-region solution. The increase is driven by:

1. **Infrastructure duplication**: you need full cluster capacity in each region. You cannot share infrastructure across regions due to latency and compliance requirements
2. **China premium**: operating infrastructure in China requires a local partner (Alibaba Cloud, Tencent Cloud) which costs 30 to 50 percent more than US/EU providers
3. **Cross-region data transfer**: network egress costs are significant at 100,000 QPS with 50 KB response sizes
4. **Operational complexity**: each region needs local SRE coverage for incident response

**Cost Optimization Strategies:**

1. **Right-size each region based on traffic**: US-East handles 40 percent of traffic (40,000 QPS), EU-West handles 30 percent (30,000 QPS), AP-Southeast handles 20 percent (20,000 QPS), China handles 10 percent (10,000 QPS). Scale clusters proportionally. US-East needs 12 data nodes, EU-West needs 9, AP-Southeast needs 6, China needs 3.

2. **Use different storage tiers**: the global catalog in US-East uses SSD storage for low latency. The global catalog copies in other regions can use lower-cost storage (like Elasticsearch's cold tier) because most queries in those regions are for local products.

3. **Optimize replication**: only replicate global catalog product metadata (200 bytes per product), not user data, search logs, or analytics data. This reduces cross-region data transfer by 70 percent.

4. **Cache aggressively at the edge**: use a CDN (CloudFront, Cloudflare, Akamai) to cache search results for anonymous users (not logged in). Anonymous traffic accounts for 40 percent of all search queries. Edge caching eliminates backend search calls for these queries. Expected edge cache hit rate: 70 percent. This reduces backend QPS from 100,000 to 58,000.

**Trade-off Summary:**

| Priority | Decision | Sacrifice |
|----------|----------|-----------|
| Compliance | Data stays in-region, China isolated | Cannot serve Chinese users from other regions |
| Latency | Local clusters for each region | 3.85x infrastructure cost |
| Consistency | Local-first inventory, async replication | Cross-region inventory is 10 to 30 seconds stale |
| Cost | Proportionally scaled clusters, edge caching | No redundancy in China (single cluster, no cross-region failover for China) |

**China Special Considerations:**

Operating in China requires additional considerations beyond the standard multi-region architecture:

- **Local partner**: Elasticsearch is not available as a managed service in China. You must use a local cloud provider (Alibaba Cloud Elasticsearch or Tencent Cloud ES) or self-host
- **Censorship compliance**: certain search queries must be blocked or filtered according to Chinese regulations. The China cluster must integrate with local content moderation APIs
- **Isolation**: the China cluster stores only Chinese user data and global catalog data. No EU user data, no US user data. The China cluster is not part of the cross-region replication topology for user data
- **Disaster recovery**: due to cost constraints, the China region has a single cluster with no cross-region failover. If the China cluster fails, Chinese users are served from AP-Southeast with degraded latency (400 ms). This is a deliberate trade-off: cost savings now versus potential revenue loss during a prolonged outage

This design is not perfect. It accepts higher cost for better latency and compliance. It accepts stale inventory for cross-region products. It accepts limited disaster recovery for the China region. Every decision is a deliberate trade-off based on business priorities.
