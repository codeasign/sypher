## The Core Tension

Every search system faces a fundamental tension between three competing goals:

1. **Precision**: returning the most relevant results
2. **Latency**: returning results quickly
3. **Freshness**: reflecting the latest data

Improving any one of these degrades at least one of the others.

Improving precision requires more complex scoring, additional index lookups, and reranking. Each of these adds latency. Improving latency means using simpler scoring, fewer index lookups, and aggressive caching. Each of these reduces precision. Improving freshness means indexing documents faster, which creates more segments and increases query latency.

This is not a problem you solve once. It is a trade-off you manage continuously based on your workload and business requirements.

## Decision Matrix

The following matrix helps you choose the right search architecture based on your constraints.

| Decision | Option A | Option B | When to Choose A | When to Choose B |
|----------|----------|----------|-----------------|-----------------|
| Indexing strategy | Real-time (refresh < 1s) | Batch (refresh > 30s) | User-facing search, social feeds | Analytics, nightly jobs |
| Shard count | Few shards (1-5 per node) | Many shards (10+ per node) | Small clusters, heavy indexing | Large clusters, high QPS |
| Replication | 0 replicas | 2+ replicas | Dev/test, cost-sensitive | Production, high availability |
| Scoring | BM25 only | Learned ranking (LTR) | Simple use cases, low latency | High-relevance requirements |
| Cache strategy | Aggressive (TTL 60s) | Minimal (TTL 1s) | Stable content, high QPS | Rapidly changing content |
| Index storage | SSD only | SSD + object storage | Hot data, low latency | Cold data, cost savings |
| Query timeout | Short (5s) | Long (60s) | User-facing search | Background reporting |
| Result window | Shallow (100) | Deep (10,000) | Web search | Data export, compliance |
| Filter strategy | Pre-filter (filter then score) | Post-filter (score then filter) | Selective filters | Expensive scoring |
| Similarity model | TF-IDF | BM25 | Legacy systems, simple needs | Modern search, default |

### Decision 1: Pre-filter vs Post-filter

Pre-filtering applies filters before scoring. The filter reduces the candidate set, so fewer documents are scored. This is faster but can miss relevant documents if the filter is too restrictive.

Post-filtering scores all matching documents, then applies the filter. This returns the most relevant results within the filter criteria but scores 10x more documents.

**Rule of thumb**: use pre-filter when filters are selective (filter out more than 90 percent of documents). Use post-filter when filters are non-selective or when scoring accuracy is critical.

### Decision 2: Exact Count vs Approximate Count

When a user searches, the UI shows "12,847 results." Computing the exact count requires scanning all matching documents. For large result sets (over 10,000), exact counts add 50 to 200 milliseconds of latency.

Approximate counts use cardinality estimation (HyperLogLog or sampling). They return within 1 millisecond with 1-2 percent error.

**Rule of thumb**: use exact counts for result sets under 10,000. Use approximate counts for larger result sets. Most users never verify the exact count.

## CAP Theorem Implications

The CAP theorem states that a distributed system can provide at most two of: consistency, availability, and partition tolerance. Search systems make specific choices within this framework.

### Consistency: Eventual

Search systems are eventually consistent. A document that was just indexed may not appear in a query that is executing concurrently. The system does not provide linearizable reads.

This is acceptable because search is inherently a best-effort operation. Users do not expect search results to be perfectly consistent with the underlying data store. They expect fast, relevant results.

### Availability: High

Search systems prioritize availability. If a shard is unavailable, the query proceeds with the remaining shards. The results are partial but the system stays up.

Elasticsearch's default behavior on shard failure: return results from available shards and include a `_shards` section in the response indicating how many shards succeeded and failed. The client can decide whether to use the partial results.

### Partition Tolerance: Required

Search clusters span multiple nodes and often multiple data centers. Network partitions are inevitable. The system must tolerate them.

During a partition, the cluster enters a quorum-based decision:

- The majority side (nodes that can communicate) continues to serve queries
- The minority side stops serving queries to prevent split-brain
- When the partition heals, the minority side rejoins and reconciles state

### What CAP Means for Search Design

The PACELC extension adds nuance: even when the network is healthy (E, Else), there is a trade-off between latency (L) and consistency (C). Search systems choose low latency over strict consistency.

When you index a document, the system does not wait for all replicas to acknowledge. It writes to the primary shard and the specified number of replicas (default: 1). The write is acknowledged when the primary and one replica confirm. Other replicas receive the update asynchronously.

This means a read-your-write scenario can fail: you index a document, then search for it, and it does not appear because the read hit a replica that has not yet received the update. Applications that require read-your-write consistency must use a separate mechanism (like refreshing the index before the search or routing the read to the primary shard).

## Failure Modes

Search systems fail in specific ways. Understanding these failure modes helps you design resilient systems.

### Failure Mode 1: Thundering Herd

A thundering herd occurs when many users issue the same query simultaneously. This can happen during a popular event, a news breaking, or when a cache expires.

**Symptoms**: sudden spike in query latency, increased CPU usage, and cache miss rates.

**Mitigation**:
- Rate limiting on the coordinator nodes
- Stale-while-revalidate caching: serve stale cached results while refreshing the cache
- Query deduplication: coordinator merges concurrent identical queries into one shard request
- Circuit breaker for expensive queries

### Failure Mode 2: Expensive Query Exhaustion

A single expensive query can consume all available resources. Common expensive queries include:

- Wildcard queries on high-cardinality fields (`*:something`)
- Regular expression queries on unindexed fields
- Queries with many `should` clauses (each adds an index lookup)
- Deep pagination (page 1000 with 10 results per page)
- Aggregations on high-cardinality fields (50,000 unique values)

**Mitigation**:
- Query complexity limits: reject queries that exceed configuration limits
- Timeout per query: hard limit of 30 seconds
- Resource isolation: dedicated nodes for heavy analytics queries
- Query profiling: log and analyze expensive queries for optimization

### Failure Mode 3: Segment Explosion

Under high indexing load, the number of index segments grows rapidly. Each segment adds overhead to query execution. The merge process cannot keep up with the segment creation rate.

**Symptoms**: increasing query latency, high I/O wait, merge queue backlog.

**Mitigation**:
- Reduce the refresh interval (from 1s to 30s) during high indexing periods
- Increase the merge thread pool
- Use a larger indexing buffer to create larger segments
- Throttle the indexing rate
- Use separate clusters for indexing and querying

### Failure Mode 4: Memory Pressure

The Java heap is shared between indexing buffers, caches, and query execution. Under high load, the heap fills up, triggering garbage collection pauses. Long GC pauses cause the node to disconnect from the cluster.

**Symptoms**: node timeouts, increased query latency, circuit breaker trips.

**Mitigation**:
- Keep JVM heap at or below 32 GB (to use compressed OOPs)
- Monitor heap usage and set circuit breaker thresholds
- Reduce cache sizes if memory pressure is persistent
- Add nodes to distribute the memory load
- Use separate JVM instances for indexing and querying

### Failure Mode 5: Cluster Split-Brain

A network partition splits the cluster into two groups. Both groups elect a master. The two masters make conflicting decisions about shard allocation and cluster state.

**Symptoms**: duplicate shards, inconsistent cluster state, index corruption.

**Mitigation**:
- Use an odd number of master-eligible nodes (3 or 5)
- Set `discovery.zen.minimum_master_nodes` to N/2 + 1
- Use a quorum-based decision for master election
- Monitor cluster health for split-brain indicators (duplicate node IDs, conflicting routing tables)

### Failure Mode 6: Reindexing Backlog

When the mapping or analysis chain changes, existing documents must be reindexed. Reindexing 500 GB of data takes hours. During reindexing, the old index is still serving queries, but the new index is being built.

**Symptoms**: increased CPU usage, slower query response times on the old index (due to resource contention), slow reindexing progress.

**Mitigation**:
- Use index aliases to swap indexes atomically
- Reindex in the background with throttled rate
- Use a read replica for serving queries during reindexing
- Add temporary nodes for the reindexing workload

## Anti-Patterns

These anti-patterns appear regularly in search system designs. Recognizing them prevents costly mistakes.

### Anti-Pattern 1: Using a Database as a Search Engine

Replacing a search engine with a relational database is tempting for teams that want to reduce infrastructure complexity. The database has indexes, so it should be able to search, right?

The problem is that database full-text indexes lack the features that make search useful: relevance scoring, fuzzy matching, synonym expansion, faceted aggregation, and phrase queries. A database query returns all matches with equal weight, sorted by an arbitrary field. Users expect the most relevant results first.

The database also cannot handle the query volume. A search engine handles 5,000 QPS on a single node. A database falls over at 500 QPS with full-text queries.

### Anti-Pattern 2: Single Giant Index

Putting all data in one index with no partitioning. This index grows to hundreds of gigabytes or terabytes. Query latency degrades as the index grows. Reindexing takes days. Node recovery from failure takes hours.

**Solution**: partition by time (daily or weekly indices) or by tenant. Use index aliases to query across partitions transparently.

### Anti-Pattern 3: Over-Aggregation

Adding 50 aggregations to every query. Each aggregation scans the candidate set and computes field statistics. Fifty aggregations on a result set of 100,000 documents take 500 milliseconds or more.

**Solution**: limit aggregations to 10 per query. Use sampling for large result sets. Cache aggregation results for popular queries.

### Anti-Pattern 4: Ignoring the Query Cache

Every query hits the index, even when the same query runs 100 times per second. The query cache is turned off or configured too small.

**Solution**: enable the query cache with sufficient memory. A 20 percent cache hit rate saves 20 percent of query CPU. For dashboards and auto-suggest, the cache hit rate exceeds 80 percent.

### Anti-Pattern 5: Premature Distribution

Running a 3-node cluster for 100,000 documents. The overhead of distributed coordination (network round-trips, result merging, cluster state management) exceeds the benefits of distribution.

**Solution**: start with a single node. Scale to multiple nodes only when the index exceeds 100 GB or the query load exceeds 1,000 QPS.

### Anti-Pattern 6: Custom Relevancy without Testing

Tuning search relevance by gut feeling. Adding boosts for recency, popularity, and brand without measuring the impact. The search results get worse but no one notices because there are no precision metrics.

**Solution**: set up a relevance evaluation framework. Use a test set of 100 to 500 queries with known relevant documents. Track precision metrics after every relevancy change. A/B test in production.

### Anti-Pattern 7: Deep Paging

Allowing users to request page 10,000. Each deep page requires the system to score and sort all matching documents, then discard everything before the requested offset. For a result set of 1 million documents, page 10,000 (results 100,000 to 100,010) requires scoring all 1 million documents.

**Solution**: use search_after cursors for deep paging. Limit the result window to 10,000. For use cases requiring full result access (like data export), use a dedicated scan-and-scroll API that does not require scoring all results.

## When NOT to Use a Search System

A dedicated search system is not the right solution for every problem.

### Use a Database Instead When

- **You have fewer than 10,000 documents**. A database with a full-text index handles this easily.
- **You need ACID transactions**. Search systems do not provide transactional guarantees across documents.
- **Your queries are simple lookups**. If you only query by ID or exact field match, a database is faster and simpler.
- **Your data changes extremely fast**. If documents change 100 times per second and you need immediately consistent reads, a database is better suited.

### Use a Cache Instead When

- **You only need exact key lookups**. A cache (Redis, Memcached) provides faster lookups than a search engine for single-key access.
- **Your data fits in memory**. If you can store all data in a single machine's memory, a cache provides sub-millisecond lookups.
- **You do not need relevance ranking**. If any matching result is equally good, a cache with sorted sets is simpler.

### Use Stream Processing Instead When

- **You need real-time alerts on data patterns**. A stream processor (Kafka Streams, Flink) detects patterns in data streams and triggers alerts. A search engine is not designed for this.
- **You need complex event processing**. Sequence detection, temporal windows, and pattern matching are stream processing domains.

### Use Graph Database Instead When

- **Your queries focus on relationships between entities**. Friend-of-friend queries, recommendation paths, and influence networks are better served by a graph database like Neo4j or Amazon Neptune.
- **You need multi-hop traversals**. A search engine cannot efficiently answer "find all products that users who bought this product also bought."

## Summary

The search system pattern involves continuous trade-offs between precision, latency, and freshness. The decision matrix provides a framework for choosing the right options based on your workload.

The CAP theorem applies to distributed search: search systems choose availability and partition tolerance over strict consistency. This means eventual consistency is a feature, not a bug.

Six failure modes threaten production search systems: thundering herds, expensive query exhaustion, segment explosion, memory pressure, cluster split-brain, and reindexing backlog. Each has specific mitigations.

Seven anti-patterns waste resources and degrade user experience: using a database as a search engine, single giant index, over-aggregation, ignoring the query cache, premature distribution, custom relevancy without testing, and deep paging.

## Additional Decision Dimensions

### Real-Time vs Batch Indexing

The indexing strategy determines how quickly new documents appear in search results.

**Real-time indexing** (refresh interval under 1 second):
- Documents are searchable within 1 second of ingestion
- Creates many small segments, increasing query latency
- Higher CPU and I/O overhead for segment merging
- Best for: user-generated content, social feeds, chat applications

**Batch indexing** (refresh interval 30 seconds to 1 hour):
- Documents are searchable after the next refresh
- Creates fewer, larger segments, reducing query latency
- Lower overhead for segment merging
- Best for: nightly imports, analytics, reference data

**Hybrid approach**: Use a small in-memory buffer for real-time writes and a large batch indexer for bulk imports. The bulk indexer writes directly to a new segment without going through the in-memory buffer.

### Exact Match vs Fuzzy Match

**Exact match** requires the query term to match the indexed term exactly:
- Fast: no additional computation
- Precise: no false positives
- Brittle: misses relevant results due to spelling variations or stemming

**Fuzzy match** allows the query term to match similar indexed terms:
- Slower: requires computing edit distance for candidate terms
- More recall: catches spelling errors and variations
- Less precise: may return irrelevant results

The fuzzy match algorithm computes the Levenshtein distance between the query term and indexed terms. The maximum edit distance is typically 1 (for short terms) or 2 (for longer terms). Fuzzy matching adds 2 to 10 milliseconds to query time.

### Term Frequency Saturation

The BM25 parameter `k1` controls how quickly term frequency saturates:

- `k1 = 0`: no term frequency effect (binary scoring). A term either appears or not. This is equivalent to TF-IDF with binary TF.
- `k1 = 1.2` (default): moderate saturation. The first few occurrences matter, but the 10th occurrence adds little value.
- `k1 = 2.0`: slower saturation. Higher term frequency continues to significantly increase the score.

A high `k1` value is appropriate for long documents (where term frequency is a strong relevance signal). A low `k1` value is appropriate for short documents (like product titles).

### Length Normalization

The BM25 parameter `b` controls how much document length affects scoring:

- `b = 0`: no length normalization. A 100-page document and a 1-paragraph document are scored equally.
- `b = 0.75` (default): strong length normalization. Longer documents are penalized.
- `b = 1.0`: full length normalization. Score is proportional to the density of matching terms.

A high `b` value is appropriate for web search (where short documents are often more relevant). A low `b` value is appropriate for academic search (where longer documents contain more information).

## Failure Mode Deep Dives

### Failure Mode 7: Cache Poisoning

A user submits a query with unique parameters that bypasses the cache. If this query is expensive, it consumes resources without contributing to cache efficiency.

**Scenario**: A search-as-a-service platform serves 10,000 customers. Each customer has a unique tenant ID that is injected into every query. The query cache is keyed by the full query including tenant ID, so queries from different tenants never share cache entries.

**Impact**: The cache hit rate is 2 percent instead of the expected 30 percent. Query latency is 3x higher than expected.

**Mitigation**:
- Cache at the shard level instead of the query level. Shard-level caching is more granular and benefits from shared terms across queries.
- Use a separate cache for common sub-expressions (like filter bitmaps) that are shared across tenants.
- Implement a popularity-based cache admission policy: only cache queries that appear more than N times per minute.

### Failure Mode 8: Indexing Backpressure Failure

When the indexing rate exceeds the cluster's capacity, the indexing buffer fills up. The system applies backpressure by rejecting new indexing requests.

**Scenario**: A black Friday flash sale triggers 100,000 product updates per second. The cluster can process 50,000 updates per second. The indexing buffer fills in 10 seconds. The system starts rejecting indexing requests.

**Impact**: Products are not updated in search. Some products show incorrect prices or stock levels. Users see stale data.

**Mitigation**:
- Implement a write queue (Kafka) between the application and the search cluster. The queue absorbs spikes and provides backpressure.
- Monitor the indexing buffer fill rate and alert when it exceeds 80 percent.
- Shard the index by date or tenant to distribute the indexing load.
- Use a separate cluster for indexing and replicate to the query cluster.

### Failure Mode 9: Hot Shard

Uneven data distribution causes one shard to be significantly larger than others. This shard receives a disproportionate share of queries.

**Scenario**: An e-commerce platform routes products by category to shards. The "Electronics" category has 10 million products, while other categories have 1 million each. The Electronics shard is 10x larger than the others and receives 40 percent of all queries.

**Impact**: The hot shard is the bottleneck for all queries that include the Electronics category. Query latency for Electronics searches is 5x higher than for other categories.

**Mitigation**:
- Use a hash-based routing strategy that distributes documents evenly across shards.
- Custom routing keys should be designed to produce even distribution.
- If uneven distribution is unavoidable, allocate more shards to the hot category.
- Monitor shard size distribution and alert when the standard deviation exceeds 20 percent.

The infrastructure cost of a search system depends on the cluster size, storage type, and replication factor.

### Build vs Buy Decision

The build-vs-buy decision depends on the complexity of the search requirements and the team's expertise.

**Build (Elasticsearch/Solr on self-managed infrastructure)**:
- Lower infrastructure cost (pay for raw compute and storage)
- Higher operational cost (need search engineers)
- Full control over configuration and tuning
- Best for: organizations with search expertise and specific requirements

**Buy (Algolia, Meilisearch Cloud, Elastic Cloud)**:
- Higher infrastructure cost (pay for service markup)
- Lower operational cost (no in-house management)
- Limited customization
- Best for: teams without search expertise, standard search use cases

## Decision Trees

### Search Engine Selection Decision Tree

Follow this decision tree when choosing a search engine for a new project:

```
Is the data size under 10 GB and QPS under 100?
  YES -> Does the team have search expertise?
    YES -> Elasticsearch (single node, minimal config)
    NO  -> Meilisearch or Typesense (zero config, built-in typo tolerance)
  NO -> Is the data size under 500 GB and QPS under 5,000?
    YES -> Is multi-tenancy required?
      YES -> Elasticsearch with tenant routing
      NO  -> Elasticsearch or OpenSearch (3-5 node cluster)
    NO -> Is vector search required?
      YES -> Elasticsearch with dense_vector + HNSW
      NO  -> Elasticsearch or OpenSearch (10-20 node cluster)
```

### Relevancy Strategy Decision Tree

```
Is the search for a user-facing product?
  YES -> Does the product have explicit user preferences (ratings, purchase history)?
    YES -> Implement personalized ranking with boost signals
    NO  -> Start with BM25 and monitor user feedback
  NO -> Is the search for log analytics?
    YES -> Use standard BM25 (exact match is more important than relevance)
    NO -> Is the search for code or technical content?
      YES -> Use code-specific analyzer, no stemming, exact match preferred
      NO -> Use BM25 with field-level boosting
```

## Common Trade-Off Mistakes

### Mistake 1: Optimizing for the Wrong Metric

Teams optimize for query latency when precision is the real problem. Users do not notice 50ms vs 100ms latency, but they notice when the second result is irrelevant.

**Fix**: Measure and track precision metrics (NDCG, P@10) alongside latency. If precision is below 90 percent, invest in relevancy before performance.

### Mistake 2: Over-Caching

Teams cache everything with long TTLs, then wonder why search results are stale. Products go from out-of-stock to in-stock, but the cache still shows them as out-of-stock for 5 minutes.

**Fix**: Use short TTLs for dynamic data (inventory, price) and long TTLs for static data (product description, brand name). Invalidate cache entries when the underlying data changes.

### Mistake 3: Ignoring the Query Distribution

Teams optimize for the average query when the P99 query is 10x more expensive. A single expensive query (wildcard, regex, high-cardinality aggregation) can consume 50 percent of the cluster's resources.

**Fix**: Profile all queries. Identify the top 10 most expensive queries. Optimize or restrict them. Set per-query resource limits.

### Mistake 4: Replica Blindness

Teams count replica shards as capacity without considering the network overhead. Adding a replica doubles the query fan-out traffic. If the network is the bottleneck, more replicas make the problem worse.

**Fix**: When adding replicas, monitor network utilization. If network utilization exceeds 70 percent, add nodes before adding replicas.

A search system is not the right tool when your data fits in a database or cache, when you need ACID transactions, or when your query patterns are simple lookups. Choose the right tool for the job.
