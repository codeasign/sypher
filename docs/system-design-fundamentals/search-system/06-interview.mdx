## How It Appears

The Search System Pattern appears in system design interviews in these forms.

**Form 1: "Design a search engine."** The most common formulation. The interviewer asks you to design a search engine for a specific domain: web search, e-commerce, document search, or code search. The domain determines the specific requirements.

**Form 2: "Design the search feature for X."** The search feature is part of a larger system. Examples: "Design Airbnb search," "Design YouTube search," "Design Slack search." The search component must integrate with the broader system architecture.

**Form 3: "Design an autocomplete system."** This is a specialized variant of the search pattern. It focuses on prefix matching, trie data structures, and top-K completion suggestions.

**Form 4: "Design a typeahead search."** Similar to autocomplete but returns full results (not just suggestions) as the user types. This tests your understanding of instant search and query performance optimization.

**Form 5: "Design a full-text search for a document store."** A focused variant where the system indexes documents (PDFs, Word files, emails) and supports full-text queries. Tests your understanding of text analysis and inverted index internals.

**Form 6: "Design a product search for an e-commerce platform."** Combines text search with structured filters, faceted navigation, and personalized ranking. Tests your understanding of multi-criteria ranking and filter integration.

**Form 7: "Design a search for a social media platform."** Adds real-time indexing, recency ranking, and content moderation filters. Tests your understanding of high-throughput indexing and time-aware ranking.

**Form 8: "Design a geolocation-based search."** The query involves a location parameter. "Find all coffee shops within 2 miles of my current location." Tests your understanding of geohashing and spatial indexing.

**Form 9: "Design a job search platform."** A multi-faceted search combining text search (job title, skills), structured filters (salary range, location, seniority level), and personalized ranking (based on applicant profile). Tests your ability to integrate multiple search dimensions.

**Form 10: "Design a hotel search system."** Combines text search (destination), date range filtering (check-in, check-out), geospatial constraints (map viewport), and availability lookup. Tests your understanding of intersection between search and reservation systems.

## What the Interviewer Tests

The interviewer evaluates four dimensions when you design a search system.

### Depth of Understanding

Can you explain how the inverted index works? Do you know what a posting list is and how terms are intersected? Can you describe the difference between TF-IDF and BM25?

Interviewers probe depth by asking follow-up questions: "How would you handle phrase queries?" "What happens when you index a document with 100,000 words?" "How does the system decide which shard to search?"

Demonstrate depth by using precise terminology: inverted index, posting list, segment, term dictionary, BM25 scoring, filter bitmap, aggregation, and relevancy tuning.

### Awareness of Trade-offs

Can you identify the key trade-offs in the design? The interviewer wants to see that you know search is about compromises, not perfect solutions.

Key trade-offs to discuss unprompted:

- Precision versus latency: more complex scoring returns better results but takes longer
- Index freshness versus query performance: frequent refreshes create more segments and slow queries
- Memory versus disk: in-memory indexes are faster but more expensive
- Replication factor versus write throughput: more replicas improve query throughput but slow writes
- Exact counts versus approximate counts: exact counts are accurate but slow for large result sets

### Scalability Intuition

Can you estimate the resources needed to serve a given workload? The interviewer wants to see that you can translate query volume and data size into infrastructure requirements.

Be ready to estimate:

- How many nodes are needed for 10,000 QPS with 500 GB of data
- How much memory is required for the inverted index
- What is the bottleneck at each scale (CPU, memory, I/O, network)
- When to shard and how to distribute shards

Provide concrete numbers. "For 10,000 QPS with 500 GB of data, I estimate 12 to 16 nodes with 64 GB RAM and 8 CPU cores each. The bottleneck is likely CPU because scoring is the most expensive operation."

### System Design Process

Can you follow a structured design process? The interviewer evaluates how you approach the problem: gathering requirements, defining scope, estimating scale, designing components, discussing trade-offs, and identifying failure modes.

A structured process for search system design:

1. Clarify requirements (domain, data volume, query patterns, latency targets)
2. Estimate scale (data size, QPS, storage requirements)
3. Define the data model (documents, fields, types)
4. Design the indexing pipeline (document ingestion, analysis, index building)
5. Design the query flow (query parsing, index lookup, filtering, scoring, ranking)
6. Design the distributed architecture (sharding, replication, node roles)
7. Discuss trade-offs (precision vs latency, freshness vs performance)
8. Address failure modes (thundering herd, memory pressure, split-brain)

## Level-by-Level Expectations

Different experience levels have different expectations for search system design.

### L3 / Entry Level (0-2 years)

**Expectation**: understand the basic components of a search system.

You should be able to:

- Describe the inverted index and how it enables fast term lookup
- Explain the basic query flow: query parsing, index lookup, result return
- Identify the difference between a relational database and a search engine
- Use basic terminology correctly: document, term, score, match, filter

**Common pitfalls**: proposing a database as a search engine, not understanding the need for relevance scoring, ignoring the indexing pipeline.

**Interview tip**: focus on one component and explain it thoroughly. If asked about scaling, acknowledge that you are not sure and ask for guidance. "I know that production systems use sharding, but I would need to learn more about the specific implementation."

### L4 / Mid Level (2-5 years)

**Expectation**: design a search system for a single use case with moderate scale.

You should be able to:

- Design the inverted index with posting lists and term compression
- Explain BM25 scoring and compare it with TF-IDF
- Design a filter and facet system using bitmaps
- Discuss sharding and replication strategies
- Estimate resource requirements for 1,000 to 5,000 QPS

**Common pitfalls**: proposing a single-node solution for a workload that requires distribution, not accounting for index merging overhead, ignoring cache strategies.

**Interview tip**: demonstrate trade-off awareness explicitly. "I would use this approach because it optimizes for write throughput, and here is the trade-off it makes on query latency."

### L5 / Senior Level (5-8 years)

**Expectation**: design a production-grade search system with multiple features and high scale.

You should be able to:

- Design the full search pipeline: ingestion, analysis, indexing, querying, ranking, aggregation
- Compare Elasticsearch, Solr, and custom engines with detailed technical reasoning
- Design query rewriting pipelines (spell correction, synonym expansion, query relaxation)
- Design personalized ranking with multiple boost signals
- Discuss operational concerns: monitoring, rolling upgrades, backup and restore
- Handle failure modes: thundering herd, expensive queries, segment explosion
- Estimate resource requirements for 5,000 to 50,000 QPS

**Common pitfalls**: oversimplifying the ranking system, not addressing multi-tenancy, ignoring operational concerns.

**Interview tip**: discuss real-world experience. "In a previous project, we faced a segment explosion issue during Black Friday traffic. We mitigated it by reducing the refresh interval and throttling the indexing rate."

### L6 / Staff Level (8-12 years)

**Expectation**: design a search system that spans multiple teams and serves as a platform.

You should be able to:

- Design a multi-tenant search platform with tenant isolation, rate limiting, and billing
- Design a multi-region active-active architecture with cross-region replication
- Discuss the CAP theorem implications for search design
- Design a relevance framework for A/B testing and continuous improvement
- Address organizational concerns: team structure, migration strategy, operational runbooks

**Common pitfalls**: ignoring the organizational impact of the design, not addressing migration from legacy systems, underestimating operational complexity.

**Interview tip**: discuss platform-level concerns. "The design must support 50 teams independently managing their search configurations. Each team needs their own index templates, analyzers, and ranking profiles."

### L7 / Principal Level (12+ years)

**Expectation**: design a search system that shapes the company's technical strategy.

You should be able to:

- Define the long-term roadmap for search infrastructure
- Make build-vs-buy decisions with cost-benefit analysis
- Design for 10x growth without redesign
- Influence industry standards and open-source contributions
- Mentor multiple teams on search best practices

**Common pitfalls**: proposing a solution that is too complex for the company's maturity level, not considering the total cost of ownership.

**Interview tip**: discuss strategic decisions. "I would recommend adopting Elasticsearch for the first 18 months and investing in a custom search engine only after validating the search use cases and team capability."

### L8 / Distinguished Level (15+ years)

**Expectation**: define the search vision for the entire industry or a significant domain.

You should be able to:

- Design search systems that process billions of documents and millions of QPS
- Optimize at the hardware level (SIMD instructions for scoring, NUMA-aware memory allocation)
- Define the research agenda for the search team (learned ranking, neural search, hybrid retrieval)
- Evaluate and acquire search technology companies

**Common pitfalls**: proposing bleeding-edge technology that is not proven at scale.

**Interview tip**: focus on business impact. "The search system generates 30 percent of the company's revenue. A 1 percent improvement in search relevance translates to $50 million in additional annual revenue."

## FAANG Framework for Search Design

Use this structured framework to answer any search system design question.

### Step 1: Scope the Problem (3 minutes)

Ask clarifying questions to define the scope:

- What type of content is being searched? (web pages, products, documents, code, videos)
- How many documents? (thousands, millions, billions)
- How many queries per second? (hundreds, thousands, tens of thousands)
- What are the latency requirements? (100ms, 500ms, 2 seconds)
- What are the freshness requirements? (real-time, near real-time, batch)
- What query features are needed? (full-text, filters, faceting, geo, ranking)
- How is the system used? (user-facing, internal, API)

### Step 2: Estimate Scale (3 minutes)

Provide concrete estimates for:

- Total document count and data size
- Query volume per second (peak and average)
- Storage requirements (raw data, index, metadata, replication factor)
- Network bandwidth for query fan-out
- Number of nodes required

### Step 3: Design the Data Model (3 minutes)

Define the document schema:

- Which fields are searchable (text, keyword)
- Which fields require filtering (numeric, date, geo)
- Which fields need aggregation (keyword terms for faceting)
- Which fields are retrieved but not searched

Show the mapping for the search index:

```json
{
  "product": {
    "properties": {
      "title": {"type": "text", "analyzer": "english"},
      "description": {"type": "text", "analyzer": "english"},
      "brand": {"type": "keyword"},
      "category": {"type": "keyword"},
      "price": {"type": "float"},
      "rating": {"type": "float"},
      "created_at": {"type": "date"},
      "in_stock": {"type": "boolean"}
    }
  }
}
```

### Step 4: Design the Indexing Pipeline (5 minutes)

Describe how documents are indexed:

- Document ingestion: CDC from database, API, or batch load
- Text analysis: tokenizer, character filters, token filters, stemmer
- Index building: inverted index with posting lists, positions, and term frequencies
- Segment management: in-memory buffer, flush to disk, background merging

### Step 5: Design the Query Flow (5 minutes)

Describe what happens when a query arrives:

1. Query parsing and validation
2. Query rewriting: spell correction, synonym expansion
3. Query planning: term ordering, filter pushdown
4. Index segment selection: prune irrelevant segments
5. Term lookup and posting list intersection
6. Filter application using bitmaps
7. BM25 scoring with boost signals
8. Aggregation computation for faceting
9. Result merging across shards
10. Response building with metadata

### Step 6: Design the Distributed Architecture (5 minutes)

Describe how the system scales:

- Sharding: number of shards, routing strategy, shard sizing
- Replication: replication factor, query routing to replicas
- Node roles: coordinator nodes, data nodes, master nodes
- Query distribution: fan-out to all shards, merge results
- Fault tolerance: node failure, shard recovery, rebalancing

### Step 7: Discuss Trade-offs (5 minutes)

Identify and discuss the key trade-offs in your design:

- Precision vs latency: where your system operates on the curve
- Freshness vs performance: refresh interval choice and rationale
- Memory vs disk: what is cached and what is on disk
- Exact vs approximate counts: when to use each
- Pre-filter vs post-filter: which filter strategy and why

### Step 8: Identify Failure Modes (3 minutes)

Discuss how your system handles failures:

- Thundering herd: cache strategies and rate limiting
- Expensive queries: timeout and circuit breaker
- Segment explosion: merge policy and indexing throttling
- Memory pressure: circuit breaker and heap management
- Split-brain: quorum configuration

## Phrase Bank

Use these phrases to demonstrate expertise during the interview.

**On inverted index**. "The inverted index is built during indexing, not during query time. Each term maps to a sorted list of document IDs, which enables fast intersection during query execution."

**On posting lists**. "Posting lists are stored with skip pointers that let the intersection algorithm skip over blocks of documents that cannot match. This reduces intersection cost from O(N) to approximately O(sqrt(N))."

**On BM25**. "BM25 improves over TF-IDF by adding term frequency saturation. A term appearing 10 times is not 10x more relevant than appearing once. BM25 also normalizes for document length to prevent long documents from having an unfair advantage."

**On segmentation**. "The index is divided into immutable segments. New documents are written to a small in-memory segment that is searchable immediately. When the memory buffer fills, the segment is flushed to disk. Background threads merge small segments into larger ones."

**On query rewriting**. "Raw queries are rarely optimal. The rewriting pipeline applies spell correction, synonym expansion, and query relaxation. Each plugin in the pipeline can modify the query tree to improve recall or precision."

**On filter bitmaps**. "Filters are precomputed as roaring bitmaps. The AND or OR of two filters is a bitwise operation that completes in microseconds. This is much faster than scanning documents and checking filter conditions one by one."

**On scoring performance**. "Scaling is the most expensive operation in search. Each document in the candidate set is scored using BM25, which involves term frequency, inverse document frequency, and length normalization. For 10,000 candidate documents, this takes 5 to 15 milliseconds."

**On caching strategy**. "We cache at multiple levels. The filter cache stores precomputed bitmaps for each filter value. The query cache stores complete query results for frequent queries. The node-level request cache stores shard-level results."

**On sharding**. "Sharding distributes the index across multiple nodes. Each shard is a self-contained Lucene index. The routing function determines which shard stores each document. Query requires fan-out to all shards and result merging."

**On replication**. "Replicas serve two purposes: fault tolerance and query throughput. Each replica is a full copy of a shard. With R replicas, the cluster tolerates R-1 node failures and serves R times the query throughput."

**On relevancy tuning**. "Relevance is a weighted combination of signals. The base signal is BM25. Additional signals include recency boost, popularity boost, personalization score, and business rules. Weights are tuned through A/B testing."

**On aggregations**. "Aggregations compute facet counts on the candidate set. For the query 'headphones,' the system counts results per brand and per price range. Aggregations are computed on the same data that produced the search results, so facet counts are consistent."

**On distributed queries**. "A distributed query fans out to all relevant shards in parallel. Each shard returns its top N results. The coordinator merges the results into a globally sorted list. The total query time is the maximum of the individual shard times, not the sum."

**On index freshness**. "Refresh interval controls how quickly new documents appear in search results. A 1-second refresh means documents are searchable within 1 second of indexing. Shorter refresh intervals create more segments, which increases query latency."

**On multi-tenant isolation**. "Each tenant's data is stored in a separate index or on dedicated shards. The search-as-a-service proxy injects tenant filters, enforces rate limits, and tracks billing. Resource isolation prevents one tenant from affecting another's performance."

## Red Flags

Avoid these mistakes during the interview.

**Red flag 1: Proposing a database as a search engine**. Saying "we can use Postgres with LIKE queries." This shows you do not understand the fundamental difference between a database and a search engine. A database cannot do relevance scoring, faceted aggregation, or fuzzy matching at scale.

**Red flag 2: Ignoring the indexing pipeline**. Only discussing the query side and never explaining how documents get into the index. A search system needs a complete indexing pipeline with text analysis, segment management, and merge strategy.

**Red flag 3: No concrete numbers**. Proposing a design without estimating scale. "We need 10 nodes" is weak. "For 500 GB of data with 8,000 QPS, I estimate 12 data nodes with 64 GB RAM and 16 CPU cores each" shows depth.

**Red flag 4: Single-node design for large scale**. Proposing a single-node architecture for a workload that clearly needs distribution. "One server is enough" is wrong for any system with more than 100 GB of data or more than 500 QPS.

**Red flag 5: No discussion of trade-offs**. Presenting a design as perfect with no compromises. Every search system makes trade-offs. Not discussing them shows lack of experience.

**Red flag 6: Over-engineering the solution**. Proposing 50 nodes for 10,000 documents. An interview is not the time to demonstrate that you know every Elasticsearch feature. Keep the design proportional to the requirements.

**Red flag 7: Ignoring failure modes**. Designing for the happy path only. "What happens when a node fails?" is a standard follow-up. Not having an answer shows lack of operational awareness.

**Red flag 8: Confusing search with analytics**. Proposing the same system for search queries and analytical queries. Search queries are for finding specific documents. Analytical queries are for computing statistics. They have different optimization goals.

**Red flag 9: Forgetting about ranking**. Focusing only on finding matching documents and ignoring how they are ordered. A search system that returns unranked results is not useful. Discuss BM25, boost signals, and personalization.

**Red flag 10: No mention of caching**. Caching is essential for search performance at scale. Not discussing the query cache, filter cache, and node-level request cache suggests inexperience with production search systems.

## Practice Drills

Use these drills to prepare for search system design interviews.

### Drill 1: Five-Minute Diagram

Set a timer for 5 minutes. Draw the architecture of a search system on a whiteboard or paper. Include:

- Client, coordinator nodes, data nodes, master nodes
- Indexing pipeline: documents, analyzer, inverted index, segments
- Query flow: query parser, index lookup, filter, scorer, result builder
- Storage: inverted index on disk, cache in memory, document store

Repeat until you can draw the complete diagram from memory in under 5 minutes.

### Drill 2: Capacity Estimation

Practice estimating capacity for different scenarios:

- Scenario A: 5 million documents, 200 bytes per document, 2,000 QPS, 200ms latency target
- Scenario B: 500 million documents, 2 KB per document, 10,000 QPS, 100ms latency target
- Scenario C: 10 billion documents, 5 KB per document, 50,000 QPS, 50ms latency target

For each scenario, calculate: storage per shard, number of shards, number of nodes, memory per node, and network bandwidth.

### Drill 3: Query Flow Walkthrough

Verbally walk through the complete query flow for a specific query. Do it out loud.

Example: "The user types 'wireless headphones under $100' into an e-commerce search box. Walk through every step from query intake to result display."

Include: tokenization, normalization, query rewriting, inverted index lookup, posting list intersection, filter application for price range, BM25 scoring, recency boost, aggregation for facet counts, pagination, and response building.

### Drill 4: Trade-off Analysis

Pick a design decision and argue both sides:

- Should the index be in memory or on disk?
- Should filters be applied before or after scoring?
- Should the system use exact counts or approximations?
- Should the refresh interval be 1 second or 30 seconds?
- Should the system use BM25 or a machine-learned ranking model?

For each decision, state the criteria that would lead you to choose each option.

### Drill 5: Failure Mode Walkthrough

Pick a failure mode and describe your response:

- "A thundering herd of 10,000 identical queries hits your search cluster. What happens? How does your system respond?"
- "An indexing surge creates 10,000 small segments. Query latency doubles. What do you do?"
- "The query cache is empty after a node restart. Queries that were 5ms are now 200ms. How do you handle this?"

Describe the symptom, root cause, immediate mitigation, and long-term fix.

### Drill 6: Mock Interview

Find a partner to role-play a search system design interview. The interviewer picks a scenario (e-commerce search, document search, code search) and the interviewee designs the system using the FAANG framework. Timebox each step:

- Scope: 3 minutes
- Estimate: 3 minutes
- Data model: 3 minutes
- Indexing: 5 minutes
- Query flow: 5 minutes
- Distributed architecture: 5 minutes
- Trade-offs: 5 minutes
- Failure modes: 3 minutes

Total: 32 minutes (a typical system design interview time)

### Drill 7: Reverse Engineer

Pick a search system you use daily (Google, Amazon, GitHub, Slack, YouTube). Reverse engineer its search system based on observable behavior:

- How long does a search take? (latency)
- What happens when you type? (autocomplete, instant search)
- What filters are available? (faceting)
- How are results ordered? (ranking signals)
- What happens when you misspell a word? (spell correction)
- What happens when no results match? (query relaxation)

Write a one-page analysis of the system's architecture.

### Drill 8: Open Source Deep Dive

Pick one open-source search engine (Elasticsearch, Solr, Meilisearch, Typesense) and read its documentation for one specific component:

- How does Elasticsearch implement the inverted index?
- How does Meilisearch implement typo tolerance?
- How does Typesense implement vector search?

Write a summary of the implementation approach. Understanding how production systems implement the pattern builds credibility in interviews.

## Additional Interview Scenarios

### Scenario: Design Search for a Photo Sharing Platform

The platform has 1 billion photos. Each photo has tags, captions, location, and EXIF data. Users search by tag, caption text, location, camera model, or date range.

Key considerations:
- Tag search must be exact (no stemming, no fuzzy matching)
- Location search uses geohash indexing
- Date range search uses a BKD tree
- Caption search uses full-text with stemming
- Rank by recency and popularity (number of likes, shares)

### Scenario: Design Search for a Human Resources Platform

The platform has 10 million resumes. Recruiters search for candidates by skills, experience, education, location, and salary expectations.

Key considerations:
- Resume parsing: extract structured fields from unstructured text (skills, education, job titles)
- Skill synonym expansion: "Java" expands to include "Java 8," "Java 11," "Java 17," "Java EE"
- Experience range filter: "5+ years of Python" requires date-based filtering on job history
- Privacy: candidates must opt in to appear in search results
- Recency: resumes updated in the last 30 days are boosted

### Scenario: Design Search for a Legal Document Database

The platform has 50 million legal documents (court cases, statutes, regulations, contracts). Lawyers search by citation, party name, date, judge, and legal concepts.

Key considerations:
- Citation search: "Brown v. Board of Education" must match the exact citation format
- Boolean search: lawyers use AND, OR, NOT operators extensively
- Proximity search: "expert" within 5 words of "testimony" is a common query pattern
- Date range: cases from 1980 to 1990
- Faceted navigation: by court, by judge, by outcome, by year

## Common Follow-Up Questions

Interviewers often ask these follow-up questions after the initial design. Prepare answers for each.

**Q: How would you handle the case where the index is too large to fit in memory?**

A: The index does not need to fit entirely in memory. The term dictionary should fit in memory (FST is compact). Posting lists are cached by the OS page cache for frequently accessed terms. Cold posting lists are read from disk. The key is to ensure the working set fits in memory: the terms and documents that are most frequently queried.

**Q: How would you implement spell correction?**

A: Spell correction uses a two-level approach. First, compute the edit distance between the query term and indexed terms. Second, use a Levenshtein automaton that traverses the term dictionary FST and finds all terms within the edit distance threshold. The candidate corrections are scored by their document frequency (more common terms are preferred) and the edit distance (smaller distance is preferred).

**Q: How do you handle synonyms?**

A: Synonyms are handled at index time or query time. Index-time synonym expansion: the synonym "laptop" is expanded to `["laptop", "notebook", "ultrabook"]` during indexing, so all three terms are in the inverted index. Query-time synonym expansion: the query term "laptop" is expanded to `("laptop" OR "notebook" OR "ultrabook")` during query processing. Index-time is faster for queries but requires reindexing when synonyms change. Query-time is more flexible but slower.

**Q: How does search handle stop words?**

A: Modern search systems do not remove stop words; instead, they use a common terms query. The common terms query treats frequent terms (like "the," "a," "for") differently from rare terms. The rare terms are used for the initial search, and the common terms are used for scoring. This avoids the problem of removing stop words and losing phrase matches while still maintaining relevance.

**Q: How do you test search relevance?**

A: Search relevance is tested using a judgment list: a set of 100 to 500 queries with manually labeled relevant documents. The system runs each query and computes NDCG@10, P@10, and MAP. A regression test suite runs these metrics after every relevancy change. A/B testing in production compares the new ranking against the old ranking using user engagement metrics (click-through rate, time to first click, session duration).

## Final Checklist

Before your system design interview, confirm you can do each of the following:

**Core Concepts (must have)**:
- Explain the inverted index data structure
- Describe how posting lists are intersected
- Explain BM25 scoring and how it differs from TF-IDF
- Describe the indexing pipeline from document to searchable segment
- Explain the role of tokenization, normalization, and stemming

**Architecture (must have)**:
- Draw the architecture of a distributed search cluster
- Explain sharding strategy (hash routing, number of shards, sizing)
- Describe query flow from client to result
- Explain the CAP theorem implications for search
- Design a CDC pipeline from database to search index

**Advanced (good to have)**:
- Explain filter bitmaps and how they work
- Describe the difference between pre-filter and post-filter
- Explain query rewriting (spell correction, synonym expansion, relaxation)
- Design a multi-region search architecture
- Discuss personalized ranking with multiple boost signals

**Operational (good to have)**:
- Explain segment merging and its impact on performance
- Describe monitoring and alerting for search clusters
- Design a zero-downtime reindexing strategy
