## Company Case Studies

Real-world search systems reveal the pattern in action. Each company optimized for different constraints and made different trade-offs.

### Case Study 1: Netflix -- Search for 200 Million Members

Netflix's search system helps 200 million members find content from a catalog of 15,000 titles. The challenge is not document count but query complexity and personalization.

**The Problem**: Members search with vague queries ("funny action movies," "something like Stranger Things"). The search system must understand intent, not just match keywords. Results must be personalized to each member's viewing history and preferences.

**Architecture**:

Netflix uses Elasticsearch for the search index, but the search pipeline is heavily customized:

1. **Query understanding layer**: a machine learning model predicts the member's intent. The query "funny action movies" is classified as genre=Action, tone=Comedy, type=Movie.
2. **Personalized boosting**: the results are boosted by the member's viewing history, ratings, and watchlist. A member who watches Marvel movies sees superhero content ranked higher.
3. **Multi-objective ranking**: the ranking layer balances multiple objectives: relevance (does this match the query?), engagement (will they watch it?), and business goals (is this content promoted?).
4. **Real-time signals**: the system incorporates real-time signals like trending content, new releases, and what friends are watching.

**Key trade-off**: Netflix optimizes for member satisfaction over pure search precision. A query that returns 100 percent relevant results but causes the member to spend 5 minutes deciding is worse than one that returns 80 percent relevant results but leads to immediate playback.

**Scale**: 500 search queries per second during peak. P99 latency of 80 milliseconds per query. Index size of 50 GB (title metadata, not the video content).

### Case Study 2: GitHub -- Code Search for 100 Million Repositories

GitHub's code search indexes 100 million repositories and 2 billion files. The challenge is indexing code (not natural language) and handling the unique query patterns of developers.

**The Problem**: Code search is fundamentally different from text search. Developers search for symbols, function definitions, imports, and regex patterns. A search for `ThreadPoolExecutor` should find class declarations, usages, and imports. A regex search for `def test_.*` must match Python function definitions.

**Architecture**:

GitHub rebuilt its code search system in 2023 using a custom Rust-based search engine:

1. **Indexing at repository level**: each repository is indexed independently. The index stores symbol definitions, function signatures, and code structure (not just text tokens).
2. **Language-aware parsing**: the indexer uses language-specific parsers (tree-sitter) to extract symbols, function names, and references. A query for `class Foo` in Java matches only Java class declarations, not the word "class" in a comment.
3. **Regex support**: code search supports regular expressions. The index uses trigram indexing for fast regex filtering: the regex is decomposed into required trigrams, which are looked up in the inverted index.
4. **Ranking by relevance signals**: code search ranks results by file popularity (stars), repository relevance, symbol usage frequency, and code quality signals.

**Key trade-off**: GitHub prioritizes indexing throughput over real-time freshness. A new commit is searchable within 5 minutes. This is acceptable because code search is an exploratory tool, not a real-time monitor.

**Lessons learned**: GitHub's previous code search (based on Elasticsearch) could not handle the scale. The custom Rust engine provides 10x better performance and supports regex search, which Elasticsearch could not do efficiently.

### Case Study 3: Airbnb -- Search for 7 Million Listings

Airbnb's search system matches guests with 7 million listings across 220 countries. Search is the core user experience: every booking starts with a search query.

**The Problem**: Airbnb search combines text search (location, property type) with structured filters (price, bedrooms, amenities) and geospatial constraints (within this map viewport). Ranking must balance relevance, availability, pricing, and host quality.

**Architecture**:

Airbnb uses Elasticsearch with significant customization:

1. **Geo-spatial indexing**: every listing has exact coordinates. The search must return listings within the user's map viewport. Elasticsearch's geo_shape and geo_distance queries handle this.
2. **Availability filtering**: listing availability is stored as a bitmask of dates. The search query filters by the user's desired check-in and check-out dates. This is the most expensive filter because availability changes constantly.
3. **Smart ranking**: the ranking model is a gradient-boosted decision tree trained on booking data. Features include listing quality score, host response rate, price competitiveness, and personalized signals.
4. **Price-aware ranking**: the system optimizes for booking conversion, not just relevance. A slightly less relevant listing at the right price point converts better than a perfect match that is too expensive.

**Key trade-off**: availability filtering is a hard constraint (you cannot book a listing that is not available), but keeping the availability index in sync with the booking system is challenging. Airbnb uses a change data capture pipeline from the booking database to the search index with a 5-second delay.

**Scale**: 1,500 search queries per second during peak. P99 latency of 150 milliseconds. Index size of 200 GB.

### Case Study 4: Algolia -- Search-as-a-Service for 10,000 Customers

Algolia provides search infrastructure to 10,000+ customers, including Stripe, Shopify, and Medium. Each customer expects dedicated performance and isolation.

**The Problem**: Multi-tenant search is challenging. One customer's heavy indexing load should not affect another customer's query latency. Each customer has different data models, ranking requirements, and query patterns.

**Architecture**:

Algolia built a custom search engine from scratch in C++:

1. **Per-customer indices**: each customer's data is stored in separate indices on dedicated shards. This provides strong isolation.
2. **In-memory index**: the entire index is kept in memory (RAM). Algolia does not use disk-based storage for the hot path. This enables sub-10 millisecond query times.
3. **Typo tolerance**: Algolia is famous for its tolerance algorithm. It automatically handles typos and misspellings without requiring an explicit spell correction step. The algorithm uses a precomputed prefix trie with Levenshtein automata.
4. **Instant search**: results are returned as the user types (after each keystroke). This requires sub-50 millisecond latency for every intermediate query.

**Key trade-off**: in-memory indexing is fast but expensive. Algolia charges a premium for this speed. Customers pay for the memory footprint of their index. A 10 GB index costs approximately $500 per month.

**Scale**: 50,000 queries per second across all customers. Median latency of 7 milliseconds. P99 latency of 25 milliseconds.

## Open Source Solutions

Several open-source search engines implement the Search System Pattern. Each has different strengths.

### Elasticsearch

Elasticsearch is the most popular open-source search engine. It is built on Apache Lucene and provides a distributed, RESTful search and analytics engine.

**Strengths**: mature ecosystem, extensive documentation, rich query DSL, built-in aggregation framework, strong Kibana integration for log analytics, and large community.

**Weaknesses**: Java memory management can be tricky, cluster management is complex at scale, indexing performance degrades under heavy load without careful tuning, and the scoring model is difficult to customize without a plugin.

**Best for**: general-purpose search, log analytics, application monitoring, and e-commerce search.

### Apache Solr

Solr is the older sibling of Elasticsearch. Both are built on Lucene, but Solr was the original search server.

**Strengths**: mature and stable, excellent faceting performance, strong security features (role-based access control out of the box), and advanced query capabilities (spatial, graph, SQL-like queries).

**Weaknesses**: smaller community than Elasticsearch, less modern tooling, and the configuration format (XML-based) is less developer-friendly.

**Best for**: enterprise search, government systems, and applications that need strong security and precise control.

### Meilisearch

Meilisearch is a modern, Rust-based search engine designed for developer experience.

**Strengths**: instant search (typo-tolerant by default), simple API (single endpoint for search and indexing), minimal configuration (almost zero-config setup), and fast indexing.

**Weaknesses**: limited aggregation capabilities, smaller community, fewer language analyzers, and less mature distributed mode.

**Best for**: small to medium e-commerce sites, documentation search, and applications that prioritize simplicity over advanced features.

### Typesense

Typesense is another modern search engine, similar to Meilisearch in philosophy.

**Strengths**: typed schema with automatic type inference, built-in vector search, extremely fast (median latency under 10 milliseconds), and simple API.

**Weaknesses**: newer and less proven at scale, smaller ecosystem, and fewer third-party integrations.

**Best for**: search-as-a-service alternatives, applications that need vector search without a separate vector database, and low-latency requirements.

## Cloud Providers

All major cloud providers offer managed search services.

### Amazon OpenSearch Service

Amazon's managed Elasticsearch and OpenSearch service. It handles cluster provisioning, software upgrades, and automated backups.

**Strengths**: tight integration with AWS ecosystem (S3 snapshots, CloudWatch monitoring, IAM authentication), ultra-warm storage tier for cost-effective cold data, and cross-cluster search.

**Pricing**: approximately $0.10 to $0.50 per GB per month, depending on instance type and storage tier.

### Azure Cognitive Search

Microsoft's managed search service with built-in AI enrichment capabilities.

**Strengths**: AI enrichment pipeline (OCR, entity recognition, key phrase extraction from Azure Cognitive Services), semantic search (Microsoft's deep learning reranker), and integration with Azure data sources (Cosmos DB, Blob Storage, SQL Server).

**Pricing**: approximately $0.05 to $0.15 per GB per month, plus query costs.

### Google Cloud Search for Work

Google's enterprise search solution for searching across G Suite data (Gmail, Drive, Calendar).

**Strengths**: seamless integration with Google Workspace, AI-powered ranking, and enterprise-grade security and compliance.

**Limitations**: only works with Google Workspace data. Not available as a general-purpose search engine.

## Lessons Learned

Production search systems reveal patterns that apply across all implementations.

### Lesson 1: Measure Relevance, Not Just Performance

Many teams optimize search latency and throughput but never measure relevance. A search that returns results in 10 milliseconds is useless if the results are irrelevant.

Set up a relevance evaluation framework from day one. Create a test set of 50 to 200 queries with known relevant documents. Track NDCG@10 and P@10 after every change. Run A/B tests in production to validate that relevance improvements actually improve user engagement.

Netflix found that a 5 percent improvement in NDCG translated to a 2 percent increase in member engagement. Without measurement, they would never have known.

### Lesson 2: Cache Aggressively

Production search traffic is highly skewed. The top 1 percent of queries account for 30 to 50 percent of all query volume. Caching these queries eliminates the latency and CPU cost for a large fraction of traffic.

Algolia caches query results at multiple levels: application-level cache (Redis), CDN cache (Fastly), and node-level query cache. The combined cache hit rate exceeds 80 percent for most customers.

### Lesson 3: Indexing and Querying Are Different Workloads

Many teams deploy a single cluster for both indexing and querying. This is the default in Elasticsearch and Solr. But the two workloads have different resource requirements.

Indexing is I/O-intensive (writing segments). Querying is CPU-intensive (scoring and aggregating). When both workloads share the same nodes, they compete for resources. An indexing burst can cause query latency to spike.

Separate indexing and querying workloads onto different nodes. During peak indexing periods (like Black Friday for e-commerce), dedicate more nodes to indexing. During normal operations, more nodes serve queries.

### Lesson 4: Schema Design Matters More Than Tuning

A well-designed schema with appropriate field types, analyzers, and mappings outperforms a poorly designed schema with extensive tuning. Getting the schema right at the start saves weeks of optimization work.

Key schema design decisions:

- Choose the right field type: `text` for full-text search, `keyword` for exact match and aggregations
- Define analyzers for each language
- Use `norms.disabled: true` for fields that do not need length normalization (saves memory)
- Use `doc_values: false` for fields that do not need sorting or aggregations (saves disk space)
- Set `index: false` for fields you only need to retrieve, not search on

### Lesson 5: Plan for Failure

Search clusters fail. Nodes crash. Network partitions happen. Indexes get corrupted. Plan for these failures before they happen.

- Run with at least 2 replicas for production data
- Test failover scenarios regularly
- Take automated snapshots every 6 hours
- Document the recovery procedure
- Practice disaster recovery drills quarterly

Airbnb runs a "chaos monkey" for their search cluster: random nodes are periodically taken offline to verify that the system continues to function correctly.

### Lesson 6: The User Interface Shapes the Search System

The search UI determines what the search system must support. A search box with autocomplete requires different indexing than a search page with faceted navigation.

- **Autocomplete**: requires prefix indexing and fast lookups (sub-10 milliseconds). The index must support prefix queries and completion suggestions.
- **Faceted navigation**: requires aggregation capabilities. The system must compute facet counts for every query.
- **Instant search**: requires sub-50 millisecond end-to-end query time. Results must be returned after each keystroke.
- **Voice search**: requires natural language understanding and speech-to-text processing before the search query.

Design the search system and the UI together. The UI capabilities determine the indexing requirements, and the indexing capabilities determine what the UI can offer.

## Additional Case Studies

### Case Study 5: Spotify -- Podcast and Music Search

Spotify indexes 100 million tracks and 5 million podcast episodes. The search challenge is distinguishing between different types of content and handling artist names, song titles, and user-generated playlists.

**Architecture**: Spotify built a custom search stack using Google Cloud Platform services. The system uses a multi-phase ranking approach:

1. **Candidate generation**: The query is matched against multiple indices (tracks, artists, albums, playlists, podcasts, episodes). Each index returns its top 200 candidates.
2. **Cross-type ranking**: A machine learning model scores candidates across all types. The model predicts the likelihood that the user will click or play the result.
3. **Contextual boosting**: Results are boosted by user context (time of day, device type, listening history). Morning searches favor podcasts. Evening searches favor party playlists.
4. **Diversity enforcement**: The system enforces result diversity. No more than 3 results from the same artist can appear in the top 10.

**Scale**: 1,200 QPS during peak. P99 latency of 120 milliseconds. Search returns mixed results (tracks, artists, playlists, podcasts) ranked by predicted user engagement.

### Case Study 6: eBay -- Product Search for 1.5 Billion Listings

eBay operates one of the largest e-commerce search systems, indexing 1.5 billion active listings. The challenge is handling the diversity of listings (from individual sellers to large retailers) and detecting listing quality.

**Architecture**: eBay uses a custom search engine built on top of Apache Lucene, with significant modifications for e-commerce:

1. **Aspect-based search**: eBay extracted structured attributes (aspects) from listing titles and descriptions. For "iPhone 13 Pro Max 256GB Blue," the system extracts brand=iPhone, model=13 Pro Max, storage=256GB, color=Blue. Users can filter on any aspect.
2. **Listing quality score**: Each listing has a quality score based on seller reputation, return rate, shipping speed, and listing completeness. High-quality listings are boosted by 1.5x.
3. **Duplicate detection**: The same product listed by multiple sellers is detected and grouped. The search shows the best listing (by quality score) with links to alternatives.
4. **Best match ranking**: eBay's ranking algorithm considers 50+ signals including recency, price competitiveness, seller performance, listing completeness, and buyer preferences.

**Key trade-off**: eBay prioritizes listing quality over pure relevance. A lower-relevance listing from a trusted seller ranks above a high-relevance listing from an unknown seller. This increases buyer trust and reduces return rates.

### Case Study 7: Slack -- Channel and Message Search

Slack indexes messages, files, and channels across 100,000+ workspaces. The search must find relevant content within the user's workspace, respecting channel membership and permissions.

**Architecture**: Slack migrated from an in-house search system to Elasticsearch in 2018. The key challenges were:

1. **Workspace isolation**: each workspace's data must be fully isolated. A user from workspace A must never see results from workspace B.
2. **Permission-aware search**: within a workspace, a user can only search channels they belong to. The search system must inject the user's channel membership as a filter.
3. **Real-time indexing**: messages become searchable within 2 seconds of being sent. The indexing pipeline processes 10,000 messages per second.
4. **Search snippets**: search results show a snippet of the matching message with the query terms highlighted. The snippet is extracted from the message text using term position information.

**Scale**: 5,000 QPS across all workspaces. P99 latency of 200 milliseconds. Index size of several terabytes.

## Evolution of Search Systems

### Phase 1: Database Search

Most search systems start with a database. The team uses `LIKE` or `ILIKE` queries for text matching. This works for the first 10,000 documents and 10 QPS.

**Limitations hit at**: 50,000 documents or 50 QPS. Users complain about slow search and irrelevant results.

### Phase 2: Dedicated Search Engine

The team adopts Elasticsearch, Solr, or a managed service. The inverted index provides 100x faster queries. Relevance scoring and faceted navigation are added.

**Limitations hit at**: 100 GB of data or 1,000 QPS. The team hits performance bottlenecks and needs to optimize.

### Phase 3: Distributed Search

The search engine is distributed across multiple nodes. Sharding, replication, and dedicated node roles are configured. The system handles 10 TB of data and 10,000 QPS.

**Limitations hit at**: 100 TB of data or 50,000 QPS. The team faces diminishing returns from adding more nodes.

### Phase 4: Specialized Search

The team customizes the search engine for the specific domain. Custom analyzers, ranking scripts, and query rewriting pipelines are built. The search system is deeply integrated with the product domain.

**Examples**: Netflix's ML-based query understanding, GitHub's language-aware code parsing, Airbnb's availability-aware ranking.

### Phase 5: AI-Powered Search

The system incorporates vector search, neural reranking, and machine-learned ranking. Semantic understanding augments keyword matching. Search results are personalized using deep learning models.

**Current state**: Most large companies operate at Phase 4, with some at Phase 5. Newer systems can start at Phase 4 or 5 using managed services that include vector search and AI capabilities.

## Cost Optimization in Production

### Storage Optimization

Storage costs dominate the infrastructure budget for search systems. Optimizations include:

- **Cold storage tier**: move older or less-frequently queried indices to slower storage. Elasticsearch's cold tier uses spinning disks instead of SSDs, reducing storage cost by 70 percent.
- **Snapshot and delete**: take a snapshot of indices older than 30 days and delete from the cluster. Snapshots are restored on demand if needed.
- **Rollover indices**: create new indices when the current index reaches 50 GB. Smaller indices are easier to manage and merge. This prevents a single index from growing too large.
- **Field dropping**: remove fields that are no longer needed. Each field adds overhead to stored fields, doc values, and potentially the inverted index.

### Compute Optimization

Compute costs (CPU and memory) are the second-largest expense:

- **Right-sizing instances**: use instances with the optimal CPU-to-memory ratio for the workload. Search is typically CPU-bound, so CPU-optimized instances provide better value.
- **Spot instances**: use spot instances for non-critical nodes (cold data nodes, batch reindexing workers). Spot instances cost 60 to 90 percent less than on-demand.
- **Autoscaling**: add data nodes during peak hours and remove them during off-peak hours. This works well for clusters with consistent daily traffic patterns.
- **Graviton/Arm instances**: AWS Graviton instances provide 20 percent better price-performance for Elasticsearch compared to x86 instances.

### License Optimization

Open-source search engines avoid licensing costs but require operational expertise. Managed services include the license cost in the service fee.

- **Elasticsearch**: free (Apache 2.0 license). Elastic provides paid features (security, alerting, machine learning) under the Elastic License.
- **OpenSearch**: fully open source (Apache 2.0). Features are free. Forked from Elasticsearch and maintained by AWS and community contributors.
- **Algolia**: proprietary. Pricing is based on the number of search operations and index size.
- **Meilisearch Cloud**: proprietary. Pricing is based on the number of search operations.

## Migration Patterns

### Migrating from Database Search to Elasticsearch

The most common migration path is moving from PostgreSQL or MySQL full-text search to Elasticsearch.

**Migration strategy**:

1. **Shadow mode**: Index all documents to both the database and Elasticsearch. Continue serving queries from the database. Validate that Elasticsearch returns the same (or better) results.
2. **Dual-read mode**: Serve queries from Elasticsearch but also query the database in the background. Compare results and verify correctness. Log any discrepancies for analysis.
3. **Full cutover**: Route all read traffic to Elasticsearch. Keep the database as the source of truth for writes. The CDC pipeline keeps Elasticsearch in sync.
4. **Optimize**: Tune Elasticsearch for the workload. Adjust analyzers, add boost signals, and configure caching.

The full migration takes 4 to 12 weeks depending on the complexity of the search requirements and the size of the data.

### Migrating from Elasticsearch to a Custom Engine

Some companies outgrow Elasticsearch and build custom search engines. This migration is rare but necessary at extreme scale (100+ TB, 50,000+ QPS).

**Migration strategy**:

1. **Define the interface**: Create an abstraction layer that both Elasticsearch and the custom engine implement. The interface includes indexing, querying, and aggregation APIs.
2. **Build in parallel**: Run both systems in production. Route a percentage of traffic to the custom engine.
3. **Compare results**: Validate that the custom engine matches or exceeds Elasticsearch's precision metrics.
4. **Gradual cutover**: Increase traffic to the custom engine by 10 percent per week. Monitor performance and correctness at each step.
5. **Decommission**: Once traffic is fully on the custom engine, decommission the Elasticsearch cluster.

The migration takes 6 to 18 months for companies like GitHub, Netflix, and eBay.

## Future Trends

### Vector Search and Semantic Understanding

The most significant trend in search is the adoption of vector search. Instead of relying on exact keyword matches, vector search uses embeddings to find documents based on semantic similarity. A query about "cheap flights to Paris" finds documents about "budget airfare to France" even though they share no common keywords.

Production systems increasingly use hybrid search: combining BM25 keyword search with vector semantic search. The BM25 component handles exact matches and rare terms. The vector component handles synonyms, paraphrases, and conceptual relationships. The fusion algorithm (typically Reciprocal Rank Fusion) combines both result sets.

### Generative AI Integration

Large language models are transforming the search experience. Instead of returning a list of links, search systems now generate direct answers, summaries, and comparisons. The search system retrieves relevant documents, and a language model synthesizes the information into a coherent response.

This pattern (Retrieval-Augmented Generation, or RAG) requires the search system to support high recall (finding all relevant documents, not just the top 10) and to provide context windows large enough for the language model.

### Real-Time Personalization

Search systems are moving from batch-computed personalization to real-time personalization. User actions (clicks, purchases, dwell time) are streamed to the search system and incorporated into the ranking within seconds.

This requires a streaming architecture: user events are processed by a stream processor (Kafka Streams, Flink), which updates user profiles in real time. The search query includes the current profile as a ranking signal.

### Observability-Driven Optimization

Search systems are becoming more observable. Detailed telemetry on every query stage, every indexing operation, and every cache interaction is collected and analyzed. Teams use this data to identify bottlenecks, optimize resource allocation, and predict capacity needs.

Machine learning models analyze search quality metrics and automatically tune ranking parameters. If a new product category launches and users are not finding it, the system detects the drop in click-through rate and adjusts the ranking boost for that category.

The future of search is a continuous optimization loop: measure user behavior, detect quality issues, adjust ranking parameters, and validate the improvement. The search system becomes a self-optimizing platform that adapts to changing user needs and business conditions.

## Summary

Real-world search systems demonstrate the pattern in diverse contexts. Netflix optimizes for personalized relevance at 500 QPS using ML-based query understanding. GitHub rebuilt its search from scratch in Rust for code-specific indexing across 2 billion files. Airbnb combines geo-spatial search with availability filtering at 1,500 QPS. Algolia provides sub-10 millisecond search as a service using in-memory indexing. Each system made different trade-offs based on its domain requirements.

Open-source options range from the full-featured Elasticsearch and Solr ecosystems to the developer-friendly Meilisearch and Typesense. Cloud providers offer managed services with different integration points and pricing models. The build-vs-buy decision depends on your team's expertise and the complexity of your search requirements.

The key lessons from production are consistent across all case studies: measure relevance not just performance, cache aggressively, separate indexing from querying workloads, design the schema carefully before writing code, plan for failure with replicas and snapshots, and let the user interface shape the search system design. Apply these lessons and you can build a search system that scales from thousands to billions of documents while maintaining sub-second query times and consistently high relevance for your users.

No single architecture fits every use case. The right approach depends on your data size, query volume, latency requirements, and team expertise. Start simple, measure everything, and evolve your system as your understanding deepens and your scale grows.

The search systems profiled here demonstrate that there is no substitute for real-world validation. Each team built, measured, iterated, and improved over years. You can apply the same approach to your search system, regardless of its current scale. Whether you are running search for a startup with 10,000 documents or a global platform with billions, the same fundamental patterns apply across all scales of operation.

Study these case studies, extract the patterns relevant to your domain, and apply them to your own search system designs. The pattern is proven; your implementation is what makes it unique.
