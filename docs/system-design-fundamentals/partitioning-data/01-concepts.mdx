## The Problem Story

A single PostgreSQL table stores every ride taken by 50 million users over five years. The table has grown to 5.2 TB. Backups run nightly using pg_dump and take 8 hours and 23 minutes — they routinely exceed the maintenance window and spill into peak traffic hours.

A simple query — "find all rides for user 48192 in March" — scans the entire 5 TB table because the database has no way to skip the 99.99 percent of data that belongs to other users. The query takes 47 seconds. The application times out after 5 seconds.

An index on user_id reduces the scan to 240 GB of relevant pages. The query still takes 4.2 seconds because the index itself is 180 GB and spans millions of entries across the entire user base.

The worst incident comes on a Tuesday at 2:14 PM. A DBA runs `ALTER TABLE rides ADD COLUMN surge_multiplier DECIMAL(3,2)` to support a new pricing feature. PostgreSQL takes an ACCESS EXCLUSIVE lock on the table. Every read and write to the rides table blocks. For 6 minutes and 38 seconds, the entire rides service is down. The on-call engineer pages the entire database team. The VP of Engineering wants a postmortem by Friday.

The root cause is not a bad query, a missing index, or insufficient hardware. The root cause is that all 5.2 TB of data lives in one monolithic table. The database cannot parallelize maintenance, cannot isolate hot partitions, and cannot serve queries from subsets of data without scanning the whole.

This is the moment a team realizes they need **partitioning**.

The pattern repeats across every scaling story. No database survives a 5 TB table with mixed workloads forever. The question is not whether to partition, but when and how.

### Why Backups Take So Long on a Monolithic Table

The 8-hour backup time is not just an inconvenience — it is a structural risk. PostgreSQL's `pg_dump` runs a single-threaded read of the entire table. On a 5.2 TB table with 10 Gbps network, the theoretical minimum is 4,160 seconds (69 minutes) assuming perfect throughput and zero CPU overhead. In practice, the backup saturates the single-threaded read path, contends with production queries for buffer pool space, and causes page evictions that slow down every query running concurrently. Production queries that normally take 5ms spike to 500ms during the backup window. The engineering team learns to schedule deployments around the backup window, losing 8 hours of productive deploy time every night.

### The `ALTER TABLE` Catastrophe

The `ALTER TABLE` operation that locked the rides table for 6 minutes and 38 seconds is the starkest illustration of the monolithic table problem. PostgreSQL's `ADD COLUMN` for a non-nullable column with a default value rewrites every row in the table. On a 5.2 TB table, this requires reading 5.2 TB and writing 5.2 TB plus the new column — roughly 11 TB of I/O. During the rewrite, PostgreSQL holds an `ACCESS EXCLUSIVE` lock that blocks `SELECT`, `INSERT`, `UPDATE`, and `DELETE`. Every ride request during that window returns a database connection error. The rides team estimates 12,000 lost rides during the lock. Each lost ride is a customer who will use a competitor's app next time.

This is why the team needs partitioning: not just for query performance, but for operational survival.

## The Insight

Partitioning splits a large dataset into smaller, independent pieces that can be stored, queried, and managed separately while still appearing as a single logical table to the application. Each piece is called a **partition**. The system chooses which partition to access based on a **partition key** — a column or set of columns that determines where each row belongs.

Imagine a giant warehouse (the table) with 5 million boxes scattered across the floor. Finding any single box requires walking the entire warehouse. Now divide the warehouse into 100 numbered aisles. Label each box with an aisle number based on its contents. When someone asks for a box, the system checks the aisle number and walks only that aisle. The warehouse is the same size, but each search now touches 1 percent of the space.

The same principle applies to databases. A 5 TB table partitioned into 50 partitions of 100 GB each allows a query with a partition key to scan only 100 GB instead of 5 TB. Maintenance operations like `VACUUM`, `REINDEX`, and `ALTER TABLE` lock only one partition at a time instead of the entire dataset. Backups can run per-partition in parallel, shrinking the window from hours to minutes.

## Core Vocabulary

Master these 18 terms before moving to the deep dive.

**Partition** — A logical or physical subdivision of a dataset. Each partition holds a subset of rows or columns and can be stored on separate storage volumes or database nodes.

**Horizontal Partitioning (Sharding)** — Splitting a table by rows. Each partition contains the same columns but different rows. The most common form of partitioning in distributed systems.

**Vertical Partitioning** — Splitting a table by columns. Frequently accessed columns go into one partition; large, rarely accessed columns (blobs, text fields) go into another.

**Range Partitioning** — Dividing data by continuous value ranges. Example: orders partitioned by `order_date` into monthly ranges: January, February, March, and so on.

**List Partitioning** — Dividing data by discrete value categories. Example: customers partitioned by `region` into values: North America, Europe, Asia Pacific, and so on.

**Hash Partitioning** — Dividing data by applying a hash function to the partition key. Example: `HASH(user_id) % N` distributes rows uniformly across N partitions regardless of key distribution.

**Composite Partitioning** — Combining two or more strategies. Example: range partition by month, then hash partition within each month by user_id.

**Partition Pruning** — The query optimizer's ability to skip irrelevant partitions during query execution. A query with a `WHERE user_id = 48192` clause can read only the partition containing that user_id, ignoring all others.

**Partition Key** — The column or expression used to determine which partition a row belongs to. The single most important design decision in any partitioned system.

**Shard** — An independent database instance that holds a subset of a partitioned dataset. Each shard runs on its own server or cluster and handles its own queries, transactions, and storage.

**Shard Key** — The partition key in a sharded database system. Selecting a shard key with high cardinality and uniform access frequency is critical to avoiding hot spots.

**Cross-Shard Query** — A query whose predicate does not include the shard key. The query must be sent to all shards, each executes a partial result, and a coordinator merges the results. Cross-shard queries are expensive and should be minimized.

**Rebalancing** — The process of moving data between partitions when the distribution becomes uneven. Rebalancing can be triggered by adding or removing nodes, data skew, or changes in access patterns.

**Hot Spot** — A single partition that receives disproportionate traffic because its partition key value is heavily accessed. A celebrity user on a social network whose data all lands on one shard is a classic hot spot.

**Consistent Hashing** — A hashing technique that minimizes data movement when the number of partitions changes. Each node maps to multiple points on a hash ring. When a node joins or leaves, only the nearest neighbor transfers data.

**Virtual Nodes** — Multiple hash ring positions assigned to each physical node. Virtual nodes improve load distribution and speed up rebalancing by spreading each node's data across many small segments.

**Scatter-Gather** — A query execution pattern where a coordinator sends the same query to all partitions, each partition computes a partial result, and the coordinator merges results. Required when the query cannot be pruned to a single partition.

**Partition Independence** — The property that operations on one partition do not affect other partitions. Backups, indexing, schema changes, and maintenance can run on one partition while others remain available.

**Repartitioning (Resharding)** — The process of changing the number or boundaries of partitions. Repartitioning is triggered by data growth, uneven distribution, or changing access patterns. It is one of the hardest operations in distributed systems — it requires moving large amounts of data without downtime.

**Partition Router** — The component that maps a partition key value to a specific partition or shard. The router can be a separate proxy service (Vitess VTGate, ProxySQL), a sidecar process running alongside the application, or embedded logic within the application itself.

**Fan-Out** — The act of sending a query or write to multiple partitions simultaneously. In write fan-out, a single logical write (for example, a new post by a celebrity) is written to many followers' partitions. In read fan-out (scatter-gather), a query is sent to all partitions and results are merged.

**Directory-Based Routing** — A routing strategy where a centralized metadata service (ZooKeeper, etcd, Cassandra) stores the authoritative mapping from partition ranges to physical nodes. All routers watch the directory for changes. Changes to the partition topology are published to the directory and propagate to all routers within seconds.

PostgreSQL demonstrates partition pruning through its `EXPLAIN` output. A query against a 50-partition table shows:

```
EXPLAIN SELECT * FROM rides WHERE user_id = 42;
                                   QUERY PLAN
─────────────────────────────────────────────────────────────
 Seq Scan on rides_p3  (cost=0.00..8942.18 rows=412 width=128)
   Filter: (user_id = 42)
 Planning time: 0.45 ms
 Execution time: 12.34 ms
(3 rows)
```

The key detail: `Seq Scan on rides_p3` confirms that only partition 3 was scanned. If the partition key is missing from the `WHERE` clause, the same query scans all 50 partitions:

```
EXPLAIN SELECT * FROM rides WHERE date = '2024-03-15';
                                   QUERY PLAN
─────────────────────────────────────────────────────────────
 Append
   -> Seq Scan on rides_p0  (cost=...)
   -> Seq Scan on rides_p1  (cost=...)
   -> ... (50 partition scans)
 Planning time: 2.31 ms
 Execution time: 47300.00 ms
(52 rows)
```

The `Append` node confirms the planner is scanning every partition. The execution time is 47.3 seconds — 3,800 times slower than the pruned query. When you see an `Append` node scanning more partitions than expected, the partition key is not being used in the query predicate.

## Step by Step: The Partitioning Lifecycle

Building a partitioned system involves fifteen sequential decisions and actions.

**Step 1: Identify the natural partition key.** Look at the access pattern for the dataset. Find the column that appears in 80 percent or more of `WHERE` clauses. For a rides table, that is `user_id`. For an orders table, that is `customer_id` or `order_date`. The ideal partition key has high cardinality (many distinct values) and uniform access frequency (no single value dominates traffic).

**Step 2: Choose the partitioning strategy.** Map the access pattern to a strategy. If queries filter by date ranges, choose range partitioning. If queries filter by geographic region, choose list partitioning. If queries filter by an opaque ID with no natural ordering, choose hash partitioning or consistent hashing.

**Step 3: Estimate the number of partitions.** Calculate total data size and divide by the target partition size. A common target is 50 GB to 200 GB per partition — small enough for fast backups and maintenance, large enough to avoid an unwieldy number of partitions. For 5 TB with 100 GB partitions: 50 partitions.

**Step 4: Define partition boundaries.** For range partitioning, define explicit boundaries: `PARTITION BY RANGE (order_date) (PARTITION p2024_q1 VALUES LESS THAN ('2024-04-01'), ...)`. For hash partitioning, define the modulus: `PARTITION BY HASH (user_id) PARTITIONS 64`. For list partitioning, enumerate values: `PARTITION BY LIST (region) (PARTITION p_na VALUES IN ('US','CA','MX'), ...)`.

**Step 5: Create the partitioned table.** Execute the `CREATE TABLE` statement with the `PARTITION BY` clause. For each partition defined in step 4, create the partition table or shard. Verify that the query planner recognizes the partitions.

**Step 6: Migrate existing data.** Load data into the new partitions using `INSERT ... SELECT` or an `EXPORT/IMPORT` pipeline. For a live migration, use a dual-write strategy: write new data to both the old table and the new partitions, then backfill historical data in batches. Monitor replication lag and consistency.

**Step 7: Route queries to the correct partition.** Update the application layer or proxy to include the partition key in every query. Use a routing layer — database proxy (ProxySQL, pgBouncer), sidecar, or application-level routing — that maps partition keys to connection strings.

**Step 8: Verify partition pruning.** Run `EXPLAIN` on queries with the partition key in the `WHERE` clause. Confirm that the query plan shows a single partition scan instead of a full table scan. A query that should touch one partition but scans all of them is a performance bug.

**Step 9: Monitor partition health.** Track partition size, query latency per partition, and write throughput per partition. Set alerts for partition imbalance: any partition exceeding 150 percent of the average size or any partition with write throughput that is 2x the average.

**Step 10: Plan for growth and rebalancing.** Design the rebalancing strategy before it is needed. For range partitioning, add new range partitions ahead of time — create next month's partition before the current month fills. For hash partitioning, plan for re-sharding: double the number of partitions and move half the data from each old partition to a new one. For consistent hashing, virtual nodes make rebalancing incremental.

**Step 11: Build the query routing layer.** Deploy a partition-aware proxy or router that intercepts all database queries. The router extracts the partition key from each query, looks up the target partition in its routing table, and forwards the query. Stateless proxies (VTGate, ProxySQL, HAProxy with Lua scripting) allow horizontal scaling of the routing layer independently of the database shards.

**Step 12: Implement cross-shard query handling.** For the minority of queries that cannot include the partition key, implement a scatter-gather mechanism. The coordinator sends the query to all relevant partitions in parallel, collects partial results, and merges them. Set a timeout on scatter-gather queries to prevent a single slow shard from blocking the entire system. Consider routing scatter-gather queries to dedicated read replicas instead of the primary shards to avoid impacting OLTP traffic.

**Step 13: Set up per-partition monitoring and alerting.** Partition health is not the same as cluster health. A cluster might show average latency of 10ms, but one partition could be at 500ms while the rest are at 2ms. Track per-partition metrics: query latency (p50, p95, p99), disk usage (absolute and as percentage of capacity), write throughput (rows per second), replication lag (seconds behind primary for each replica), cache hit ratio (buffer pool hit rate), and active connections. Set alerts for any partition exceeding 150 percent of the cluster average on any metric. Partition-level alerting catches hot spots before users do.

**Step 14: Test failover and disaster recovery per partition.** Each partition should be independently recoverable. Design the backup strategy to capture per-partition snapshots that can be restored independently. Test the scenario where a single partition's primary fails and a replica must be promoted — confirm that the routing table updates correctly and that queries for that partition's key range route to the new primary within the expected recovery time objective (RTO). Run a disaster recovery drill at least quarterly: select one partition at random, kill its primary, and measure the time to full recovery.

**Step 15: Document partition topology and runbook.** Maintain an up-to-date document that records: the partition key and strategy, the number of partitions and their physical locations, the routing table schema, the rebalancing procedure with rollback steps, the recovery procedure for each failure scenario, and contact information for the on-call team for each shard's infrastructure. The runbook should be tested by someone who did not write it. Partitioning complexity means tribal knowledge is dangerous. Every engineer on the team should be able to explain the partition topology and execute the rebalancing procedure from the runbook alone.

## Partitioning Variants in Detail

Five major partitioning strategies exist, each with distinct tradeoffs. The right choice depends on data shape, access patterns, and query requirements.

### Horizontal Partitioning (Sharding)

Horizontal partitioning divides a table by rows. Each partition is a separate database instance with the same schema, storing a subset of rows. This is the foundation of virtually every distributed database system.

**How it works:** A routing layer maps each partition key value to a specific shard. The application or proxy includes the partition key in every query. The routing layer forwards the query to the correct shard.

**When to use:** Very large tables (over 1 TB), high write throughput (over 10,000 writes per second), workloads where most queries filter by a single key.

**Strengths:** Scales write throughput linearly with shard count. Each shard is an independent database — no shared contention. Backups and maintenance run per-shard. Failure isolation: one shard failing does not bring down the whole system.

**Weaknesses:** Cross-shard queries are expensive. Schema changes require rolling deploys across all shards. Adding or removing shards requires data migration (rebalancing). Transactions spanning multiple shards are complex and slow.

### Vertical Partitioning

Vertical partitioning splits a table by columns. Frequently accessed columns go into a "hot" partition. Large, rarely accessed columns (BLOBs, audit trails, raw JSON) go into a "cold" partition. The partitions are joined on the primary key when needed.

**How it works:** Create multiple tables with the same primary key. The hot table contains columns accessed in 95 percent of queries. The cold table contains large columns accessed in 5 percent of queries. The application joins on primary key when the cold columns are needed, or uses a view that hides the join.

**When to use:** Tables with wide rows where most queries access only a subset of columns. Common for entities with profile data plus large media attachments or raw text fields.

**Strengths:** Dramatically reduces the working set for hot queries. Indexes on the hot table are smaller and faster. The cold table can use cheaper, slower storage.

**Weaknesses:** Requires application-level joins or lazy loading. Schema changes must be coordinated across tables. Does not help with write throughput scaling — all writes still go to the same database node.

### Range Partitioning

Range partitioning divides data by continuous value ranges. Each partition holds rows whose partition key falls within a specified interval. The most common range is time-based: daily, monthly, or yearly partitions.

**How it works:** Define explicit range boundaries in the `CREATE TABLE` statement. The database checks the partition key value against each range and routes the row to the matching partition. New partitions are added as data grows beyond existing ranges.

**When to use:** Time-series data (logs, metrics, events), historical data where recent data is accessed more frequently, data that is naturally ordered by a monotonically increasing key.

**Strengths:** Natural alignment with time-based retention policies — drop old partitions instead of running `DELETE`. Excellent for time-range queries. Partition pruning is straightforward and predictable. New partitions can be created in advance.

**Weaknesses:** Can create hot spots on the most recent partition. If all writes go to "today's" partition, that single partition bears all write load. Requires careful monitoring and pre-creation of future partitions.

### List Partitioning

List partitioning divides data by discrete category values. Each partition holds rows whose partition key matches one of a specified set of values.

**How it works:** Define each partition with an `IN` clause specifying which values belong to it. The database checks the partition key against each partition's list and routes the row to the matching partition.

**When to use:** Data naturally divided by geographic region, business unit, tenant ID in a multi-tenant system, or status codes.

**Strengths:** Intuitive mapping — a partition per region or tenant. Partition pruning works well for queries filtering by the list column. Each partition can be stored in the appropriate geographic region for compliance.

**Weaknesses:** Cardinality limits — the number of distinct values should be reasonably small. Uneven distribution if some values dominate (one region has 90 percent of customers). Adding new values requires adding or expanding partitions.

### Hash Partitioning

Hash partitioning applies a hash function to the partition key and uses the hash value to determine the partition. This distributes rows uniformly across partitions regardless of the key's natural distribution.

**How it works:** The database computes `HASH(partition_key) % N` where N is the number of partitions. The result maps to a partition. The same key always maps to the same partition.

**When to use:** Opaque keys (UUIDs, generated IDs) with no natural ordering. Workloads requiring uniform data distribution. Cases where no column is a natural range or list partition key.

**Strengths:** Excellent load distribution — hash functions produce uniform output across the range. No hot spots from skewed key distributions. Simple to implement and reason about.

**Weaknesses:** Poor for range queries — neighboring keys are scattered across different partitions. Changing the number of partitions (N) requires rehashing all data. Cannot take advantage of time-based retention.

### Composite Partitioning

Composite partitioning combines two strategies. The first level uses one strategy (often range), and within each first-level partition, the second level uses another strategy (often hash or list).

**How it works:** Define the first-level partitioning scheme as usual. Within each first-level partition, define a second-level scheme. Oracle and PostgreSQL support sub-partitioning natively. In systems without native support, composite partitioning is implemented at the application layer.

**When to use:** Very large datasets where a single strategy is insufficient. Time-series data with geographic or tenant subdivisions. Systems that need both time-based retention and uniform load distribution.

**Strengths:** Combines the benefits of both strategies — time-based range for retention, hash for uniform distribution. Allows fine-grained partition pruning on both dimensions.

**Weaknesses:** Increased complexity in partition management. More partitions to monitor and maintain. Not all databases support composite partitioning natively.

## Mental Model Check

Five questions to test your understanding before moving to the Deep Dive.

**Question 1:** A social media platform stores posts in a table with columns `post_id`, `user_id`, `content`, `created_at`, and `likes_count`. The platform has 500 million posts and receives 10 million writes per day. Most queries are "show all posts for user X" ordered by `created_at`. What partition key and strategy would you choose?

<details>
<summary>Answer</summary>

Partition by `user_id` using hash partitioning with 128 shards. User-level queries always include `user_id`, so every query prunes to a single shard. Hash partitioning distributes users uniformly even if some users post more than others. Within each shard, a composite index on `(user_id, created_at)` provides efficient ordering. The 128 shard target gives each shard roughly 12.5 million posts and 78,000 writes per day — well within a single MySQL instance's capacity.

</details>

**Question 2:** A logging system ingests 50 TB of application logs per day. Logs are queried by time range (past hour, past day, past week) and occasionally by `trace_id` for debugging. Old logs are deleted after 90 days. What partitioning strategy fits?

<details>
<summary>Answer</summary>

Range partition by `timestamp` with daily partitions. Each daily partition holds roughly 50 TB. Logs older than 90 days are dropped by dropping the corresponding partition — a metadata operation that takes milliseconds instead of a `DELETE` that scans terabytes. Time-range queries prune to exactly the requested days. For trace_id lookups, a secondary index on `trace_id` is added per partition, but those queries must scatter-gather across all daily partitions in the time range.

</details>

**Question 3:** An e-commerce catalog has 200 million products. The products table has 80 columns including a `description` field averaging 2 KB and a `specifications` JSON field averaging 10 KB. Most queries return only `product_id`, `name`, `price`, and `stock_count`. What two partitioning strategies apply here?

<details>
<summary>Answer</summary>

Two strategies: vertical partitioning and horizontal sharding. First, vertically split the table into a hot schema (`product_id`, `name`, `price`, `stock_count`, `category_id`) and a cold schema (`product_id`, `description`, `specifications`, `manufacturer_info`). The hot table rows are under 100 bytes each — 20 GB total for 200 million products, easily cached. The cold table stores the large fields and is only queried on product detail pages. Second, horizontally shard both tables by `product_id` using hash partitioning to distribute write load and keep each shard manageable.

</details>

**Question 4:** What happens when a query does not include the partition key? Using the rides table example, what happens with `SELECT COUNT(*) FROM rides WHERE fare > 100`?

<details>
<summary>Answer</summary>

The query cannot be pruned to a single partition because the predicate `fare > 100` does not reference the partition key `user_id`. The database must execute a scatter-gather query: it sends `SELECT COUNT(*) FROM rides WHERE fare > 100` to every partition, each partition computes a partial count, and the coordinator sums the partial counts. If there are 50 partitions, the query runs on all 50 in parallel, but the total resource consumption is 50 times higher than a partition-pruned query. Scatter-gather queries are acceptable for analytical or batch workloads but should be avoided in real-time request paths.

</details>

**Question 5:** Your hash-partitioned database uses 16 shards. Traffic has grown and you need 32 shards. Describe the rebalancing process and its challenges.

<details>
<summary>Answer</summary>

Rebalancing from 16 to 32 shards requires moving approximately half the data. With `hash(key) % 16`, each key's partition depends on the modulus. Changing to `hash(key) % 32` changes the mapping for roughly half the keys. The process: (1) create 16 new empty shards (17-32), (2) for each old shard, compute which keys now map to new shards using the 32 modulus, (3) stream those keys from old shards to new shards, (4) verify consistency by comparing checksums, (5) switch the routing configuration atomically, (6) retire old shards after confirming all reads return correct data. The challenges are data movement bandwidth (moving 2.5 TB across the network takes time), consistency during migration (writes to in-flight keys must be dual-written or buffered), and query performance degradation during migration (network and CPU on shards are consumed by data transfer). Consistent hashing with virtual nodes mitigates these problems by limiting data movement to the nearest neighbors on the hash ring.

</details>

**Question 6:** Your company's database administrator proposes partitioning the `users` table by `country` column. The table has 200 million rows. The top 3 countries (US, India, Brazil) account for 65 percent of all users. The remaining 195 countries account for 35 percent. What problems will this partitioning strategy cause, and what alternative would you recommend?

<details>
<summary>Answer</summary>

Partitioning by `country` with list partitioning creates three massive partitions (US, India, Brazil) and 195 small partitions. The three large partitions are each 20-30 million rows — too large for effective partition pruning. The US partition becomes a hot spot: queries for US users dominate traffic, but the US partition scans 20 million rows. Meanwhile, 95 percent of the partitions (195 small countries) are underutilized.

The deeper problem: country is a low-cardinality partition key. Only 200 distinct values for 200 million rows means each partition holds an average of 1 million rows, but the distribution is highly skewed by population. A partition key should have high cardinality (many distinct values) and uniform distribution across values.

**Recommendation:** Use `user_id` as the partition key with hash partitioning across 256 shards. This distributes users uniformly regardless of country. If country-based queries are needed (for example, regulatory reporting), add a secondary index on `country` and accept the scatter-gather cost for those queries. Alternatively, use a composite key: partition by `country` (list) at the top level, then by `user_id` (hash) within each country. This keeps country-level data together for regulatory queries while distributing each country's users evenly across shards.

</details>

**Question 7:** A database administrator runs `EXPLAIN` on a query that should prune to one partition. The output shows `Append -> Seq Scan on all 32 partitions`. What is the most likely cause, and how do you fix it?

<details>
<summary>Answer</summary>

The most likely cause: the query predicate does not match the partition key exactly. For PostgreSQL declarative partitioning, partition pruning requires the `WHERE` clause to contain an expression that directly references the partition key column. Common mismatches:

1. **Wrapped in a function.** `WHERE DATE(created_at) = '2024-03-15'` does not prune, but `WHERE created_at >= '2024-03-15' AND created_at < '2024-03-16'` does. The planner cannot prove that `DATE(created_at)` maps to a single partition without evaluating the function for every row.

2. **Wrong data type.** `WHERE user_id = '42'` (string) when the partition key is `user_id BIGINT`. The planner may not recognize the equivalence. Fix: use the correct type: `WHERE user_id = 42`.

3. **Implicit type conversion.** `WHERE order_date = '2024-03-15'` when the partition key is `order_date TIMESTAMP`. PostgreSQL implicitly converts the string to TIMESTAMP, but the planner may or may not use the partition key for pruning depending on the execution plan. Fix: `WHERE order_date >= '2024-03-15 00:00:00' AND order_date < '2024-03-16 00:00:00'`.

4. **Partition key is part of a composite index but not the leading column.** PostgreSQL's partition pruning uses the partition key directly, not through indexes. The index structure is irrelevant to pruning.

**Fix:** First, check the `WHERE` clause for any function wrapping or type mismatch. Second, run `EXPLAIN (VERBOSE)` to see the exact filter condition applied. Third, ensure the partition key column is referenced directly without transformations.

</details>

**Question 8:** A team chooses list partitioning by `region` with values like `'NA'`, `'EU'`, `'APAC'`, `'LATAM'`. The application grows to serve 500 million users. The `'NA'` partition now holds 250 million users (50 percent) while `'LATAM'` holds 30 million (6 percent). How does this skew affect query performance, and what can the team do?

<details>
<summary>Answer</summary>

The NA partition is 8 times larger than the LATAM partition. Queries for NA users scan 8 times more data. The NA partition becomes a hot spot for both reads and writes. Maintenance operations on the NA partition (backup, indexing, `VACUUM`) take longer and block more data. The overall system is bottlenecked by the NA partition.

**Solutions (ordered by practicality):**

1. **Sub-partition within NA.** Partition NA by a second dimension, for example, `sub_region` (US, Canada, Mexico) or by `state`. This reduces the NA partition size by a factor of 3-10 depending on the sub-dimension.

2. **Composite partitioning.** Use hash partitioning by `user_id` as the primary dimension, with a secondary list by `region` for compliance queries. This distributes NA users across all shards, eliminating the hot spot.

3. **Accept and monitor.** If the NA partition is still within hardware limits (for example, 250M users on a single PostgreSQL instance with 64 GB RAM), the team may choose to accept the uneven distribution. This is a valid pragmatic choice for small-to-medium datasets.

4. **Replicate NA to multiple shards.** Use application-level logic to split the NA region across multiple physical shards while maintaining the logical partition structure. This is complex and error-prone.

</details>

## Common Misconceptions About Partitioning

Eight statements that sound plausible but are wrong, with explanations of the correct understanding.

**Misconception 1: "Partitioning is the same as indexing."** A B-Tree index on `user_id` allows the database to find rows for user 42 without scanning the entire table. Partitioning stores rows for user 42 in a separate physical location. The index is a data structure that points to row locations. The partition is a physical storage container. Both improve query performance, but they work through different mechanisms. Indexing is always the first optimization to try; partitioning is for when the table is too large for indexing alone.

**Misconception 2: "More partitions always mean better performance."** Each partition adds overhead: file descriptors, memory for metadata, checkpoint processes, and background workers. The optimal partition count is the smallest number that keeps each partition within a manageable size (50-200 GB). Adding more partitions beyond this point increases coordination overhead without improving query performance.

**Misconception 3: "Partitioning solves all query performance problems."** Partitioning only improves queries that filter by the partition key. Queries without the partition key become slower because they must scan all partitions. Partitioning is a tradeoff: it makes partition-key queries fast at the cost of making non-partition-key queries slower.

**Misconception 4: "Hash partitioning guarantees uniform load."** Hash partitioning distributes rows uniformly by count, but traffic is not determined by row count alone. A partition with 1 million rows that are all accessed 100 times per second has more load than a partition with 10 million rows that are accessed once per day. Access frequency skew is separate from data size skew.

**Misconception 5: "You can change the partition key later."** Changing the partition key after the table is populated requires moving every row to a new partition based on the new key. This is an online data migration that touches 100 percent of the data. It is possible but expensive and risky. Choose the partition key carefully the first time.

**Misconception 6: "Partitioning is only for large databases."** Partitioning is beneficial for tables over 100 GB on transactional systems and over 1 TB on analytical systems. Below these thresholds, the overhead of managing partitions outweighs the query performance benefits. Start with a single database and proper indexing.

**Misconception 7: "All databases support partitioning the same way."** PostgreSQL, MySQL, Oracle, and SQL Server each have different partition syntax, different limitations on partition count, and different partition pruning capabilities. PostgreSQL supports up to 5,000 partitions but partition pruning degrades beyond 500. MySQL supports up to 8,192 partitions with hash and range only. Check the documentation for your specific database before designing the partition scheme.

**Misconception 8: "Distributed systems always need partitioning."** Some distributed databases handle partitioning transparently. Google Spanner, CockroachDB, and Azure Cosmos DB automatically split and distribute data across nodes without requiring the application to specify a partition key. However, even these systems benefit from the application choosing a good partition key — the key determines which data is co-located for efficient transactions and queries.

## Practical Examples: Creating Partitioned Tables

### PostgreSQL Declarative Partitioning (Range)

PostgreSQL 12+ supports declarative partitioning. The following example creates a rides table with monthly range partitions:

```sql
-- Create the partitioned parent table
CREATE TABLE rides (
    ride_id    BIGSERIAL,
    user_id    BIGINT NOT NULL,
    fare       DECIMAL(10,2),
    distance   DECIMAL(8,2),
    pickup_ts  TIMESTAMP NOT NULL,
    dropoff_ts TIMESTAMP,
    status     VARCHAR(20)
) PARTITION BY RANGE (pickup_ts);

-- Create monthly partitions
CREATE TABLE rides_2024_q1 PARTITION OF rides
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');

CREATE TABLE rides_2024_q2 PARTITION OF rides
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');

CREATE TABLE rides_2024_q3 PARTITION OF rides
    FOR VALUES FROM ('2024-07-01') TO ('2024-10-01');

CREATE TABLE rides_2024_q4 PARTITION OF rides
    FOR VALUES FROM ('2024-10-01') TO ('2025-01-01');

-- Add indexes on each partition (or use a template)
CREATE INDEX idx_rides_2024_q1_user_id ON rides_2024_q1 (user_id);
CREATE INDEX idx_rides_2024_q2_user_id ON rides_2024_q2 (user_id);
CREATE INDEX idx_rides_2024_q3_user_id ON rides_2024_q3 (user_id);
CREATE INDEX idx_rides_2024_q4_user_id ON rides_2024_q4 (user_id);
```

Queries with a `WHERE pickup_ts` filter automatically prune to the correct partition:

```sql
-- This query scans only rides_2024_q2
EXPLAIN SELECT * FROM rides
WHERE pickup_ts >= '2024-05-01' AND pickup_ts < '2024-06-01'
AND user_id = 42;

-- Output shows: Seq Scan on rides_2024_q2
-- No scan of the other three partitions
```

### MySQL Hash Partitioning

MySQL supports hash partitioning for distributing rows across a fixed number of partitions:

```sql
CREATE TABLE rides (
    ride_id    BIGINT NOT NULL,
    user_id    BIGINT NOT NULL,
    fare       DECIMAL(10,2),
    pickup_ts  TIMESTAMP,
    PRIMARY KEY (ride_id, user_id)
) PARTITION BY HASH (user_id) PARTITIONS 64;
```

MySQL also supports key partitioning (using MySQL's internal hash function):

```sql
CREATE TABLE rides (
    ride_id    BIGINT NOT NULL,
    user_id    BIGINT NOT NULL,
    fare       DECIMAL(10,2),
    pickup_ts  TIMESTAMP,
    PRIMARY KEY (ride_id, user_id)
) PARTITION BY KEY (user_id) PARTITIONS 64;
```

The difference: `PARTITION BY HASH` uses a simple modulo of the integer value, which can produce poor distribution for non-uniform keys. `PARTITION BY KEY` uses MySQL's internal hash function (CRC32-based), which provides better distribution. For production systems, prefer `PARTITION BY KEY` over `PARTITION BY HASH` when the partition key is an integer.

### Vitess Sharding Configuration

Vitess configures sharding at the keyspace level, not at the table level. The VSchema defines how tables map to shards:

```json
{
  "sharded": true,
  "vindexes": {
    "hash": {
      "type": "hash"
    }
  },
  "tables": {
    "rides": {
      "column_vindexes": [
        {
          "column": "user_id",
          "name": "hash"
        }
      ]
    }
  }
}
```

This configuration tells Vitess to shard the `rides` table by `user_id` using Vitess's built-in hash vindex. Queries on `rides` that include `WHERE user_id = ?` are routed to exactly one shard. Queries without `user_id` in the WHERE clause are broadcast to all shards.
