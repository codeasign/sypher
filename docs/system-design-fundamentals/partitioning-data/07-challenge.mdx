## How to Approach These Challenges

Each challenge presents a concrete scenario. Solve it by identifying the partition key, choosing a strategy, estimating capacity, planning for skew, and explaining the tradeoffs. The three tiers progress from basic conceptual understanding to complex multi-region design.

All answers are expandable. Attempt each challenge before revealing the solution.

---

## Tier 1: Foundation

These challenges test your grasp of basic partitioning concepts. Each should take 3-5 minutes.

### Challenge 1.1: Choose the Right Partition Key

A video streaming platform stores a `views` table with columns: `view_id` (auto-increment PK), `video_id`, `user_id`, `timestamp`, `duration_watched_seconds`, `device_type`, `country`, `isp`. The table receives 10 million new rows per day. The most common queries are:

- "Show total views for video X in the last 24 hours" (70 percent of queries)
- "Show a user's watch history" (20 percent of queries)
- "Aggregate views by country for a monthly report" (10 percent of queries)

**Question:** Choose a partition key and strategy. Justify why the other candidates are worse.

<details>
<summary>Solution</summary>

**Partition key:** `video_id` with hash partitioning.

**Why video_id?** 70 percent of queries filter by video_id. The partition key should match the dominant access pattern. Hash partitioning distributes videos uniformly — each shard holds roughly the same number of videos and receives roughly the same query load.

**Why not user_id?** Only 20 percent of queries filter by user_id. Partitioning by user_id would mean 80 percent of queries become scatter-gather — the opposite of what partitioning should achieve.

**Why not timestamp?** Only the "last 24 hours" filter is time-constrained. But queries for different videos at the same time would hit the same partition, creating a temporal hot spot. All queries for the most popular video of the day would also be time-partitioned, creating a double hot spot.

**Why not composite (video_id + timestamp)?** Composite partitioning by video_id (hash) then timestamp (range) would provide both video-level pruning and time-range pruning. This is a valid alternative if the storage system supports it. However, the added complexity is not justified when 70 percent of queries already prune to a single shard with video_id alone.

**Capacity:** 10M rows/day x 100 bytes = 1 GB/day. For 90-day retention: 90 GB. With hash partitioning across 32 shards: approximately 3 GB per shard. Comfortably small.

</details>

### Challenge 1.2: Explain Rebalancing Without Downtime

An e-commerce platform uses hash partitioning with 8 shards, each running on a separate server. The system has grown to 50 TB. The team needs to add 4 more servers (going from 8 to 12 shards) to handle increased holiday traffic.

**Question:** Walk through the rebalancing process step by step. Assume the system cannot afford more than 5 seconds of downtime.

<details>
<summary>Solution</summary>

**Step 1: Pre-migration preparation.** Set up 4 new servers with the database software and schema. Configure them as empty shards in the routing layer. Ensure network connectivity between all servers.

**Step 2: Enable dual-writes.** Update the application to write each row to both the old shard (computed via `hash(key) % 8`) and the new shard (computed via `hash(key) % 12`). Initially, only the old shard is used for reads. This ensures that during the migration, all new data exists in both locations.

**Step 3: Backfill historical data.** For each of the 8 old shards, compute which rows must move to which new shards based on the new formula. Stream data from old shards to new shards in parallel batches. Rate-limit to 30 percent of network bandwidth to avoid saturation.

**Step 4: Verify consistency.** After backfill completes, run consistency checks: compare row counts and checksums on a 5 percent sample of migrated rows. If mismatches are found, identify the root cause and re-run the backfill for affected ranges.

**Step 5: Switch reads (read-only mode on old shards).** Update the routing table atomically so that reads go to the new shards. The routing layer (etcd or ZooKeeper) pushes the updated mapping to all proxy nodes. Old shards remain available for read verification.

**Step 6: Switch writes.** Remove the old shard from the write path. All writes now go only to the new 12 shards. Keep the old dual-write code active but configure it to no-op (or confirm it applies zero changes).

**Step 7: Monitor.** Observe query latency and error rates for 24 hours. Verify that the new shards are handling traffic correctly.

**Step 8: Cleanup.** Decommission the old shard servers. Remove the dual-write code from the application.

**Downtime analysis:** Steps 1-4 require zero downtime. Step 5 involves an atomic routing table update that takes under 1 second. Step 6 is also sub-second. Total downtime: under 2 seconds for the routing update, well within the 5-second budget.

</details>

### Challenge 1.3: Identify Hot Spots

A music streaming service partitions play history by `artist_id` using hash partitioning across 32 shards. The platform has 10 million artists. The top 100 artists account for 60 percent of all plays.

**Question:** Identify at least two hot spot problems in this design. Propose a specific fix for each.

<details>
<summary>Solution</summary>

**Hot spot 1: Artist-level data skew.** The top 100 artists (0.001 percent of artists) generate 60 percent of plays. With hash partitioning by artist_id, the shards containing these artists receive 60 percent of write traffic. Even though hash distribution is uniform by count, the access pattern is not. The 2-3 shards that contain these top artists are overloaded.

**Fix 1: Sub-shard hot artists.** For each top-100 artist, create 256 sub-keys: `artist_id_0` through `artist_id_255`. Hash each sub-key independently across the 32 shards. The application fans out writes and reads to all 256 sub-keys and merges. This spreads the 60 percent traffic across all 32 shards.

**Hot spot 2: Read amplification for artist pages.** When a user visits a popular artist's page, the system loads the artist's latest play data. If 1 million users visit the same artist's page simultaneously, all 1 million queries hit the same shard.

**Fix 2: Cache layer.** Place a Redis cache in front of each shard. Cache the artist's aggregated play data for 60 seconds. A million users querying the same artist in one minute result in only one database query per minute — the rest are cache hits. For the hottest artists, increase TTL to 300 seconds and proactively refresh when new plays are recorded.

**Alternative fix: Change partition key.** Instead of artist_id, partition by user_id. Each user's play history lives on one shard. The artist-specific queries become scatter-gather, but user-level queries (the most common interaction: "show my listening history") hit one shard. Artist-level analytics can be served from a separate aggregate cache.

</details>

### Challenge 2.1: Social Media Platform Feed

A social media platform has 300 million monthly active users. Each user posts 1.5 times per day on average. Each post has: `post_id`, `author_id`, `content` (text, 500 bytes average), `media_urls` (up to 10 URLs), `created_at`, `likes_count`, `comments_count`. The feed query is: "Show the 50 most recent posts from people I follow, ordered by created_at DESC." Approximately 85 percent of read queries are feed queries. The remaining 15 percent are profile queries ("show all posts by author X") and search queries.

**Question:** Design a partitioning strategy. Describe the partition key, the routing architecture, and how the feed generation system works.

<details>
<summary>Solution</summary>

**Partition key:** `author_id` with hash partitioning across 1,024 logical shards.

**Rationale:** The feed query requires posts from followed authors. The system uses fan-out-on-write: when author X posts, the post is inserted into the feed tables of all followers. Each follower's feed is stored on the shard determined by the follower's user_id. The post itself (the canonical copy) lives on the shard determined by the author's user_id.

**Table design:**
- `posts` table: partitioned by `author_id` hash. This is the authoritative copy of each post. Used for the author's profile page.
- `feed` table: partitioned by `follower_id` hash. Each row is (follower_id, author_id, post_id, created_at). This is the materialized feed. Queries filter by follower_id.

**Feed generation:**
1. Author X creates a post. It is written to the `posts` table on shard `hash(X) % 1024`.
2. A background worker reads X's follower list (stored in a separate followers table, partitioned by the followed user_id).
3. For each follower, the worker writes a row to the `feed` table on the follower's shard. This is parallelized: one worker thread per destination shard.
4. When a user views their feed, the query reads from the `feed` table on their shard: `SELECT * FROM feed WHERE follower_id = ? ORDER BY created_at DESC LIMIT 50`. This hits exactly one shard.

**Handling celebrities:** A user with 50 million followers generates 50 million feed writes per post. This is too much for a single background worker. Mitigation: use a queue per celebrity. Write the post to the queue. Fan-out workers read from the queue in batches, writing to follower shards at a rate of 10,000 writes per second per worker. Scale workers horizontally.

**Capacity:** 300M users x 1.5 posts/day = 450M posts/day. Each post in the `feed` table is approximately 50 bytes. 450M x 50 bytes = 22.5 GB/day. Over 30 days: 675 GB. Across 1,024 shards: approximately 660 MB per shard. Tiny. The `posts` table is larger: 450M posts/day x 600 bytes = 270 GB/day. Over 90 days: 24 TB. Across 1,024 shards: 24 GB per shard. Manageable.

</details>

### Challenge 2.2: E-Commerce Catalog

An e-commerce platform has 500 million products across 50 categories. Each product has: `product_id`, `category_id`, `name`, `description` (average 2 KB), `specifications` (JSON, average 5 KB), `price`, `stock_count`, `seller_id`, `created_at`, and `updated_at`. Common queries:

- "Show products by category with pagination" (60 percent of queries)
- "Search products by keyword" (25 percent)
- "Show product detail by ID" (10 percent)
- "Show all products by seller" (5 percent)

**Question:** Design a partitioning strategy. Consider whether vertical partitioning is applicable.

<details>
<summary>Solution</summary>

**Strategy: Vertical partitioning first, then horizontal.**

**Vertical split.** Product data has wide rows. Most queries return only name, price, and stock. Split into hot and cold tables:

- `product_summary` (hot): product_id (PK), name, price, stock_count, category_id, created_at. Row size: approximately 120 bytes. Total: 500M x 120 bytes = 60 GB.
- `product_detail` (cold): product_id (PK), description, specifications, seller_id, updated_at. Row size: approximately 7 KB. Total: 500M x 7 KB = 3.5 TB.

**Horizontal partition for product_summary.** Partition by `category_id` using list partitioning. This matches the dominant query pattern: 60 percent of queries filter by category. Each of the 50 categories becomes a partition. Partition sizes vary: electronics may have 50M products while others have 2M. This is acceptable because the hot table is only 60 GB — even the largest category partition fits in memory.

**Horizontal partition for product_detail.** Partition by `product_id` using hash partitioning across 32 shards. The cold table is 3.5 TB. Queries by product_id (10 percent of queries) hit exactly one shard.

**Search queries (25 percent):** These cannot be answered by the partitioned tables alone. Use a separate Elasticsearch index for full-text search. Elasticsearch indexes the product_summary and product_detail fields. Search queries hit Elasticsearch, return product_ids, and then fetch the full product detail from the appropriate shard.

**Seller queries (5 percent):** These are scatter-gather across the product_detail hash partitions. Acceptable at 5 percent of traffic because the query volume is low. If seller queries grow, add a secondary index on seller_id with a partitioned lookup table.

</details>

### Challenge 2.3: Time-Series Metrics System

A DevOps monitoring platform ingests 5 million metrics per second from 100,000 servers. Each metric is: `metric_name`, `server_id`, `timestamp`, `value` (float). Queries are: "Show metric X for server Y over the last hour" (80 percent), and "Show average of metric X across all servers" (20 percent).

**Question:** Design a partitioning strategy. Consider data retention (metrics older than 30 days are archived).

<details>
<summary>Solution</summary>

**Partition key:** Composite: range by `timestamp` (daily), then hash by `metric_name + server_id`.

**Why range first?** Time-range queries dominate (80 percent). Daily range partitioning ensures the "last hour" query reads only today's partition. Data retention is trivial: drop partitions older than 30 days.

**Why hash within each day?** Within a daily partition, hash by `(metric_name, server_id)` to distribute writes across all storage nodes. This prevents a hot server's metrics from concentrating on one shard.

**Storage engine:** Use a columnar time-series database (ClickHouse, TimescaleDB) that supports automatic partition creation and partition-based retention.

**Capacity:** 5M metrics/sec x 100 bytes = 500 MB/sec = 43 TB/day. With daily partitioning, each partition is 43 TB. This is too large for a single partition — use hourly sub-partitions: each hourly partition is approximately 1.8 TB. With retention of 30 days = 720 hourly partitions.

**Sharding across nodes:** Distribute hourly partitions across 48 storage nodes using consistent hashing. Each node handles 15 partitions at any time. Write throughput: 5M/sec / 48 = 104K writes/sec per node. Modern SSDs handle this.

**Scatter-gather for cross-server queries:** The 20 percent of queries that aggregate across all servers require scanning all hourly partitions in the time range. ClickHouse's columnar format and vectorized execution make this feasible: a 1-hour scan on 48 nodes in parallel takes approximately 2-5 seconds.

</details>

### Challenge 2.4: Notification System with Partitioning

A push notification system sends 1 billion notifications per day across mobile and web platforms. Each notification has: `notification_id`, `user_id`, `title`, `body`, `channel` (email, push, SMS, in-app), `status` (pending, sent, delivered, failed, opened), `created_at`, `sent_at`, and `opened_at`. The system must support queries like:

- "Show all notifications for user X in the last 30 days" (50 percent of queries)
- "Show aggregate notification stats: sent vs delivered vs opened by channel" (30 percent)
- "Show undelivered notifications for batch retry" (15 percent)
- "Delete all notifications older than 90 days" (5 percent)

**Question:** Design a partitioning strategy. Consider that user_id appears in 50 percent of queries, and the retention requirement is 90 days.

<details>
<summary>Solution</summary>

**Partition key:** Composite — range by `created_at` (monthly), then hash by `user_id` within each month.

**Why composite?** The retention requirement (90 days) maps naturally to monthly range partitions. Dropping a month-old partition deletes 30 days of data in a metadata operation — no expensive DELETE. For user-level queries (50 percent), the query can be further pruned by month: "show notifications for user X in the last 30 days" scans only the current month's partition and then prunes to the user's hash shard within that month.

**Table structure:**
- Each monthly partition is a separate table: `notifications_2024_01`, `notifications_2024_02`, and so on.
- Within each month, hash by `user_id` across 64 sub-partitions. Each sub-partition is a MySQL shard on a separate server.
- Retention: at the start of each month, create the new month's partition. After 90 days, drop the oldest partition: `DROP TABLE notifications_2024_01`.

**Aggregate queries (30 percent):** These queries need notification stats across all users. They are scatter-gather across all shards within the current month's partition. This is acceptable because aggregate queries are typically run by administrators or dashboards — not by individual users. The dashboard queries run every 5 minutes, so a 10-second scatter-gather is acceptable.

**Batch retry queries (15 percent):** These queries find undelivered notifications. They are filtered by `status = 'pending'` across all partitions. This requires scatter-gather across all monthly partitions and all shards within each partition. Mitigation: maintain a separate `pending_notifications` table that stores only notifications with status = 'pending'. This table is much smaller and can be queried efficiently. Partition the pending table by `created_at` (range, daily) to support time-based retry ordering.

**Capacity:** 1 billion notifications/day x 250 bytes = 250 GB/day. Monthly partition = 7.5 TB. With 64 shards, each shard holds 117 GB per month — within the 200 GB target. Retention of 3 months = 3 active partitions at any time = 21 TB total active data.

</details>

### Challenge 3.1: Global Multi-Region Data Partitioning

A global social platform operates in 50 countries including the European Union (GDPR), Brazil (LGPD), Japan (APPI), and California (CCPA). The platform has 800 million users. Each user has a home region determined by their account registration. Data types include: profile, posts, messages, friend connections, and activity logs.

**Requirements:**
- User data must not leave the user's home region without explicit consent (GDPR Article 44).
- Cross-region features: a user in the US must be able to view a public post from a user in Japan.
- Availability: 99.99 percent per region.
- Latency: feed load under 1 second regardless of user location.

**Question:** Design a multi-region partitioned architecture. Explain data placement, cross-region data flow, replication strategy, and how you handle compliance.

<details>
<summary>Solution</summary>

**Core principle: Region-local primary, global secondary cache.**

**Data placement:**
- Each user's home region is determined at account creation. All personally identifiable information (profile, private messages, friend connections) is stored only in the user's home region cluster.
- Public posts are stored in the user's home region with async replication to a global read-only cache.
- Activity logs are stored in the home region and never replicated.

**Partitioning within each region:** Hash by user_id across 256 shards per region. Each region has its own independent cluster with its own routing table and consistent hash ring. No cross-region shard coordination is needed.

**Cross-region data flow:**
1. User A (US) creates a public post. It is written to the US region cluster.
2. A change data capture (CDC) pipeline (Debezium) captures the post from the US cluster's WAL.
3. The post is published to a cross-region Kafka topic.
4. Each region subscribes to the Kafka topic and writes the post to its local read-only cache.
5. User B (Japan) views their feed. The feed query hits the Japan region cluster, which includes locally cached posts from global users.
6. For private data (messages), User B cannot access User A's messages unless User A is in the Japan region or has explicitly consented to cross-region data transfer.

**Compliance handling:**
- GDPR EU user data never leaves EU data centers. The EU region cluster is physically located in Frankfurt, Ireland, or Stockholm. No EU user data is replicated outside the EU.
- Cross-region data transfer for non-EU public posts cached in the EU region: apply pseudonymization. The post content is stored without the author's IP address and with only a pseudonymized user ID.
- Data deletion: when an EU user deletes their account, the CDC pipeline propagates the deletion to all regions within 24 hours (GDPR Article 17).

### Challenge 3.2: Real-Time Analytics Pipeline Partitioning

A real-time analytics platform ingests 20 million events per second from web and mobile applications. Each event has: `event_id` (UUID), `app_id`, `user_id`, `event_type` (page_view, click, purchase, login, error), `properties` (JSON, 500 bytes average), `timestamp`, `ip_address`, `user_agent`, and `country`. The platform must support:

- Real-time dashboards showing event counts by event_type for the last hour (refreshed every 10 seconds).
- Ad-hoc queries filtering by event_type, country, or user_id over any time range (up to 90 days).
- Exact count distinct user_id per event_type per day for billing.

**Question:** Design a partitioning strategy using a columnar database (ClickHouse). Consider the tension between real-time ingestion (20M events/sec, 10-second refresh) and ad-hoc queries (90-day range, arbitrary filters).

<details>
<summary>Solution</summary>

**Partition key:** Range by `toYYYYMMDD(timestamp)` (daily), with ORDER BY `(event_type, country, timestamp)` as the sorting key.

**Why daily range partitioning:** Data retention is 90 days. Dropping a daily partition deletes one day of data in a metadata operation. New data is written to the current day's partition. The real-time dashboard queries the current day's partition only — the data it needs fits in one partition.

**ORDER BY (event_type, country, timestamp):** ClickHouse uses the ORDER BY to build a sparse index (granularity index). Queries filtering by event_type skip directly to the relevant rows within each partition. Queries filtering by event_type + country skip even more precisely. This enables both the real-time dashboard (filter by event_type for the current day) and ad-hoc queries (filter by event_type, country over 90 days).

**Ingestion strategy:** Use ClickHouse's MergeTree engine with a 10-second flush interval. Events arrive continuously. ClickHouse buffers them in memory and flushes to disk every 10 seconds as a new part. The part is merged into the existing partition asynchronously. For 20M events/second, each 10-second flush writes 200 million rows as a new part — this is approximately 100 GB per flush. ClickHouse's LSM-tree merge process consolidates parts in the background.

**Real-time dashboard:** `SELECT count(), event_type FROM events WHERE timestamp > now() - INTERVAL 1 HOUR GROUP BY event_type`. This query prunes to the current day's partition (one partition), then scans the sparse index for the timestamp range. Total data scanned: approximately 240 GB (one day of data at 20M events/sec). ClickHouse's vectorized execution returns results in 200-500ms.

**Ad-hoc 90-day query:** `SELECT count(DISTINCT user_id) FROM events WHERE event_type = 'purchase' AND country = 'US' AND timestamp >= '2024-01-01'`. This query prunes to 90 daily partitions (one per day). Within each partition, the ORDER BY index skips directly to the rows matching `event_type = purchase, country = US`. Total data scanned: approximately 1-2 percent of 90 days = 200-400 GB. Returned in 2-10 seconds.

**Count distinct challenge:** ClickHouse's `uniqExact` function provides exact count distinct by hashing all user_id values in the query range. For 90-day queries over purchase events in the US, this may involve 50 million distinct user_ids. ClickHouse's HyperLogLog approximation (`uniq` or `uniqHLL12`) reduces memory usage at the cost of 1-2 percent error. For billing accuracy, use `uniqExact` with increased memory budget.

**Capacity:** 20M events/sec x 500 bytes = 10 GB/sec = 864 TB/day. Compression (ClickHouse LZ4) reduces this by a factor of 5-8x for event data: approximately 100-170 TB/day compressed. With 90-day retention: 9-15 PB compressed. Partition across 48 ClickHouse nodes using distributed table with consistent hashing by `event_id`. Each node handles 2-5 TB of compressed data per day — feasible with 10 TB NVMe SSDs per node.

</details>

### Challenge 3.3: Financial Transactions with Compliance

A global payment processing platform handles 2 billion transactions per month across 30 countries. Each transaction includes: `transaction_id`, `sender_account_id`, `receiver_account_id`, `amount`, `currency`, `timestamp`, `status`, `region`. Regulatory requirements:

1. Transactions must be auditable: any transaction must be retrievable within 5 seconds by transaction_id.
2. Anti-money laundering (AML) checks require aggregating all transactions for a given account across all regions.
3. Balance consistency: account balances must be strongly consistent within a region.
4. GDPR: EU transaction data must be deletable within 30 days of account closure request.

**Question:** Design a three-level partitioning scheme (region, account, time) that satisfies all requirements. Explain the tradeoffs.

<details>
<summary>Solution</summary>

**Partitioning scheme: Three-level composite.**

**Level 1: Range by month.** Transactions are partitioned into monthly buckets. This enables GDPR deletion by dropping an entire month's partition for the affected user (if the account is closed and all transactions are older than the retention period). Monthly partitions also enable efficient time-range queries for AML investigations.

**Level 2: Hash by region_code within each month.** Within each monthly partition, transactions are distributed across regional shards. Each region's transactions stay within that region for GDPR compliance. Cross-region balance checks require fan-out across regions.

**Level 3 (within region): Hash by account_id.** Within each region-month shard, transactions are sub-partitioned by `account_id`. This enables fast retrieval of all transactions for a specific account within a region-month. The 5-second audit SLA is met by querying the account's shard.

**Balance consistency:**
- Account balances are stored in a separate `balances` table, partitioned by `account_id` hash.
- Balances are strongly consistent within a region using synchronous replication (Raft-based consensus).
- Cross-region balances are not kept consistent in real-time. A reconciliation job runs daily.
- For AML checks that need global totals: the system queries all regional balance partitions and sums. This is a scatter-gather with 30 participants — acceptable for a background check that runs every few minutes.

**GDPR compliance:**
- When a user requests deletion, the system identifies all monthly partitions containing their data.
- For the current month (where data cannot be cleanly dropped without affecting other users), individual rows are anonymized: personal identifiers are overwritten with NULLs.
- For completed months, if the user is the only account in a partition (rare), the entire partition can be dropped. Otherwise, individual rows are tombstoned.
- The partition-by-month design limits the GDPR deletion scope to at most 13 partitions (13 months of data per regulatory requirement).

**Tradeoffs:**
- Three-level partitioning creates a complex routing table. The routing layer must match on month, then region, then account_id.
- Cross-region AML queries are scatter-gather across all regions. This is acceptable because AML checks are batch jobs, not real-time queries.
- Monthly partitions must be created ahead of time. The system pre-creates 3 months of empty partitions to avoid a hot spot at month boundaries.
- Account-level hash within a region-month partition means that all transactions for a single account within a month are co-located. This is efficient for audit queries but creates potential hot spots for high-volume accounts.

</details>

## Summary

These challenges cover the spectrum of partitioning design from fundamental key selection to multi-region, compliance-aware architectures. The key across all of them: the partition key must match the access pattern, and the scheme must account for skew, rebalancing, and operational constraints from day one.
