# Graph Databases in Production

## Company Case Studies

### LinkedIn — The Social Graph That Started It All

LinkedIn pioneered graph databases in production at massive scale. Its "People You May Know" feature recommends connections by analyzing the second-degree network: friends of friends who share companies, schools, or skills. The system processes 1 billion+ nodes and 50 billion+ edges.

LinkedIn originally built its own graph engine called **Voldemort** (now deprecated), then moved to a custom graph database called **LIquid** built on top of their internal key-value store. Key lessons: they precompute recommendations offline in batch jobs rather than traversing the graph in real time, and they use a scoring model that weights different edge types differently.

**Scale**: 1 billion+ members, 50 billion+ edges. Recommendation latency: under 50ms with precomputation. Write throughput: 10,000+ edge creations per second during peak hours.

### Facebook — The Social Graph as Infrastructure

Facebook runs the largest social graph in existence: 3 billion monthly active users connected by hundreds of billions of edges. Their graph infrastructure is called **TAO** (The Associations and Objects), a distributed graph store that serves over 2 billion reads per second.

Facebook's TAO is not a traditional graph database. It is a graph-optimized cache layer over MySQL. The graph itself is stored in MySQL shards (each user's edges stored in a row). TAO caches adjacency lists and serves them at 2 billion queries per second. The key insight: you do not need a traversal engine if your queries are "get all edges of type X from node Y" — you just need fast adjacency list caching.

### Netflix — Content Graph for Recommendations

Netflix uses a graph database for its content recommendation engine. The graph connects: users to their viewing history, content to genres/actors/directors, and content to other content viewed by similar users. Every recommendation is a traversal through this graph.

**Scale**: 260 million subscribers, 20,000+ titles, billions of viewing events. Recommendation latency target: under 100ms for the initial query, with additional context loaded asynchronously.

Key lesson: Netflix uses a hybrid approach. The graph database identifies candidate recommendations (traversal), but the ranking and personalization are done by a machine learning model that scores candidates. The graph database does not need to be the final decision maker — it is a candidate generator.

### Amazon Neptune as a Managed Service

Amazon Neptune is a fully managed graph database supporting both property graph (Gremlin, openCypher) and RDF (SPARQL) models. It powers fraud detection, social features, and knowledge graphs across AWS customers.

**Scale**: Up to 15 read replicas, up to 64 TB of storage, automatic replication across 3 Availability Zones. Neptune uses a shared storage volume, so writes are visible to all replicas with read-after-write consistency.

## Open Source Implementations

**Neo4j Community Edition** — The most popular open-source graph database. Single instance, no clustering. Supports Cypher, ACID transactions, and the property graph model. Used by small to mid-size deployments. The Enterprise edition adds clustering, security, and performance monitoring at significant cost.

**JanusGraph** — Open-source, distributed graph database backed by Cassandra, HBase, or Bigtable. Supports Gremlin query language. Scales horizontally but sacrifices traversal speed — index-free adjacency is lost when graph data is distributed across multiple storage nodes.

**ArangoDB** — Multi-model (document, graph, key-value) in a single engine. Good for applications that need both document storage and graph traversals. Uses its own query language (AQL). Simpler operations than managing separate databases.

**Dgraph** — Open-source, distributed graph database with a focus on high availability and horizontal scaling. Uses GraphQL as its query language. Written in Go. Supports sharding, replication, and ACID transactions across shards.

## Lessons Learned

### Lesson 1: Precompute What You Can

The fastest traversal is the one you do not run. Precompute popular traversals (friends of friends, content recommendations, fraud patterns) in batch jobs and cache the results. LinkedIn precomputes "People You May Know" nightly. Facebook caches adjacency lists in TAO. Netflix generates candidate sets every 6 hours. Real-time traversals should be the exception, not the rule.

### Lesson 2: Model for Your Query Patterns

Design your graph schema around the queries you will run, not the entities in your domain. If you always query "find friends of friends who like the same restaurants," add direct edges between users and restaurants (a "would_like" edge) rather than traversing through friends and reviews at query time. This is denormalization, and it is acceptable in graph databases.

### Lesson 3: Plan for Supernodes

Every graph database deployment eventually encounters a supernode problem. Identify potential supernodes at design time (celebrities, popular products, central services) and plan your modeling strategy: partition them, precompute their subgraphs, or set traversal thresholds.

### Lesson 4: Monitor Traversal Depth

Your monitoring system must alert on average traversal depth. If average depth increases over time, your query patterns are getting more expensive. A 10% increase in average depth means a 33% increase in edge traversal count (assuming uniform branching). Investigate before it becomes a performance incident.

### Lesson 5: Graph Databases Are Not Primary Databases

The most successful graph database deployments use the graph as part of a heterogeneous architecture: PostgreSQL or DynamoDB for primary storage, graph database for connected queries, Elasticsearch for search, Redis for caching. Do not try to fit everything into one graph database. Each tool has strengths — use the right one for each job.
