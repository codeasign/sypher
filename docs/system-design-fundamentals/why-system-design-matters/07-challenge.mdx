These exercises test your ability to apply design thinking — evaluating tradeoffs, anticipating failure modes, and calibrating design investment to context. Each tier increases in difficulty. Complete all tiers to build a well-rounded understanding of why system design matters.

## Tier 1: Foundation (Knowledge)

### Challenge 1 — Vocabulary Application

A junior engineer on your team proposes using a single table with no indexes for a new feature that stores user activity logs. The feature is expected to generate approximately 50,000 rows per day. The engineer argues that indexes slow down writes and the table will only be queried "occasionally."

Using at least five terms from the core vocabulary, write a response that explains why this approach is problematic and what design thinking would recommend.

**Time limit:** 5 minutes.

**Common mistake:** Focusing only on query performance without discussing the broader system impact — blast radius, operational overhead, and TCO.

**Model answer:**

"The concern about write performance is valid but incomplete. The key terms to consider are:

**Technical debt** — the index-free approach creates debt today that accumulates interest every time the table is queried. A table with 50,000 inserts per day accumulates 1.5 million rows per month. After six months, a table scan on 9 million rows to find a specific user's activity will take seconds, not milliseconds.

**Total cost of ownership** of an unindexed table includes not just write performance but query performance, debugging time when users complain about slow pages, and the engineering time required to add indexes later — which requires locking the table or using a migration tool.

**Scaling cost curve** — writes scale linearly with users, but query time scales linearly with table size when no index exists. The cost curve is steeper than it should be.

**Blast radius** of a missing index is large — a single slow query can consume database CPU and exhaust connection pools, affecting all other features that share the database.

**Operational overhead** of diagnosing these issues in production is far higher than the upfront cost of adding two B-tree indexes: one on user_id and one on created_at.

The design-thinking approach is to add the indexes upfront (approximately thirty minutes of work) and accept the marginal write overhead, which is less than five percent for the typical workload described."

### Challenge 2 — Design Philosophy Identification

You are reviewing three architecture proposals for a social media feed feature:

**Proposal A:** Use a monolithic Rails application with a single Postgres database. Generate the feed by querying all recent posts from followed users and sorting by timestamp at read time. Deploy to a single server. Add a caching layer if performance becomes a problem.

**Proposal B:** Design the system on paper for six months. Evaluate five different database technologies, three caching strategies, two message queue systems, and four deployment topologies. Produce a one-hundred-page architecture document. Begin implementation only after full stakeholder approval.

**Proposal C:** Build a feed service with event-driven architecture from day one. Use Kafka for post ingestion, Cassandra for the post store, Redis for the timeline cache. Deploy on Kubernetes with service mesh. Implement multi-region failover.

For each proposal, identify which design philosophy it represents (design-first, evolve-as-you-go, BDUF), describe the tradeoffs each makes, and recommend which proposal is most appropriate for a two-person startup with fewer than one thousand users.

**Time limit:** 5 minutes.

**Common mistake:** Dismissing Proposal A as "not scalable" without recognizing that it is the most appropriate for the given context — a two-person startup with under 1,000 users.

**Model answer:**

**Proposal A** represents **evolve-as-you-go**. It builds the simplest thing that works and adds complexity only when a concrete bottleneck appears. The tradeoff is that some complexity — like pre-computing timelines — becomes more expensive to add later. For a two-person startup with fewer than one thousand users, this is the correct approach. The team should ship first, measure actual usage patterns, and optimize when they observe real bottlenecks.

**Proposal B** represents **big design up front**. It attempts to solve all problems before writing code. The tradeoff is that the team will spend six months designing for a scale that may never materialize, and the requirements will likely change before implementation begins. For a startup, this approach is fatal — it burns through runway without shipping value.

**Proposal C** represents **design-first** taken to an extreme that borders on **premature optimization**. The architecture is correct for a system serving tens of millions of users, but the operational complexity of Kafka, Cassandra, Kubernetes, and multi-region failover will overwhelm a two-person team.

**Recommendation:** Proposal A with a clear migration path. Start with the monolith and single database. Document the assumptions (under 100,000 users, mostly reads, bounded post volume). Set trigger points for adding complexity: when read latency exceeds 500ms at P99, add Redis caching. When database CPU exceeds seventy percent sustained, add read replicas. When deployment coordination becomes the bottleneck, split services.

### Challenge 3 — Cost Calculation

A startup has a single Postgres instance serving both reads and writes. Current metrics: 5,000 queries per second (4,000 reads, 1,000 writes), average query time 20ms, database CPU at sixty percent, database cost 500 dollars per month.

The engineering team is preparing for a launch that they expect to increase traffic by a factor of ten — to 50,000 queries per second. The current database will not handle this load. Two solutions are proposed:

**Solution A:** Upgrade the database instance to a machine with ten times the CPU and memory. Cost: 4,500 dollars per month.

**Solution B:** Add a read replica cluster (three replicas), a caching layer (Redis cluster), and keep the original database instance for writes. Cost: 2,200 dollars per month.

Calculate the total cost of ownership for each solution over two years. Include engineering time: Solution A requires zero architectural changes. Solution B requires approximately four engineering-weeks of setup time at 8,000 dollars per week. Which solution has lower TCO over two years? Which solution is better designed? Why might the answers be different?

**Time limit:** 10 minutes.

**Common mistake:** Comparing only the infrastructure cost without including engineering setup time, or concluding that the cheaper option is automatically the better-designed one.

**Model answer:**

**Cost calculation over 24 months:**

Solution A — Instance upgrade:
- Infrastructure: 4,500 dollars/month × 24 months = 108,000 dollars
- Engineering: 0 dollars
- Total: 108,000 dollars

Solution B — Replicas + cache:
- Infrastructure: 2,200 dollars/month × 24 months = 52,800 dollars
- Engineering: 4 weeks × 8,000 dollars/week = 32,000 dollars
- Total: 84,800 dollars

Solution B has a lower two-year TCO by 23,200 dollars. The break-even point is approximately fifteen months — after that, Solution B becomes cheaper.

**Design quality:** Solution B is better designed because it separates read and write paths, allows independent scaling, and provides redundancy. If a read replica fails, traffic falls back to the cache and primary. If the primary fails in Solution A, all traffic is down.

**Why the answers might differ:** If the startup's runway is under twelve months, Solution A's lower upfront engineering cost preserves cash for product development. The TCO calculation favors B, but the cash-flow calculation favors A. This is a real tradeoff — the better-designed system costs less in the long run but more in the short run. The correct choice depends on the startup's financial position.

## Tier 2: Application (Design)

### Challenge 4 — Architectural Decision Record

An existing system uses a single Postgres database for all data. The team is adding a new feature that requires storing and querying time-series metrics — millions of data points per hour. The current Postgres database is already at seventy percent CPU utilization.

Write an ADR that evaluates three options for storing the time-series data:
1. Keep it in Postgres with a separate schema
2. Add a dedicated Postgres instance with TimescaleDB
3. Add a purpose-built time-series database (InfluxDB or similar)

Include the decision, the rationale, the tradeoffs considered, and the expected consequences.

**Time limit:** 15 minutes.

**Evaluation rubric:** A strong answer includes concrete numbers (data volume, cost), compares at least three options with explicit pros and cons, states a clear decision with rationale, and acknowledges the conditions under which the decision should be revisited.

**Model answer:**

**Title:** ADR-001: Time-series data storage for metrics pipeline

**Status:** Proposed

**Context:** The system collects two million metric data points per hour (approximately 550 per second, 48 million per day). Each data point is a tuple of (timestamp, metric_name, value, tags). Queries are predominantly time-range scans: "all values of metric X between time A and time B, aggregated to one-minute buckets."

**Options considered:**

Option 1 — Postgres same schema: Add a hypertable-style schema in the existing database. The existing database is at seventy percent CPU. Adding time-series inserts would push it to over ninety percent, causing contention with transactional workloads. The query pattern — range scans on timestamp — would consume buffer cache that the transactional workload needs. Estimated monthly cost increase: 1,200 dollars for a larger instance.

Option 2 — Dedicated Postgres instance with TimescaleDB: TimescaleDB is an extension of Postgres that automatically partitions data into time-based chunks. It compresses data older than a configurable threshold and provides optimized time-bucket aggregation functions. A separate instance isolates the time-series workload from the transactional workload. The engineering team already knows Postgres — no new operational knowledge required. Estimated monthly cost: 400 dollars for the separate instance. Engineering investment: one week for migration.

Option 3 — Purpose-built time-series database (InfluxDB): InfluxDB is optimized for time-series — high write throughput, automatic downsampling, built-in retention policies. It uses a different query language (Flux/InfluxQL) and a different operational model. The engineering team has no InfluxDB experience. Estimated monthly cost: 800 dollars for a managed InfluxDB instance. Engineering investment: three weeks for learning curve, data migration, and query rewrite.

**Decision:** Option 2 — Dedicated Postgres instance with TimescaleDB.

**Rationale:** The team already runs Postgres, eliminating the learning curve. TimescaleDB's chunk-based partitioning provides automatic time-based data management without custom sharding logic. The dedicated instance isolates the time-series workload, preventing CPU and I/O contention with the transactional database. Compression will reduce storage costs for data older than thirty days by approximately ninety percent.

**Tradeoffs:** InfluxDB would provide faster point-in-time queries at extreme scale (over one million points per second). If the metric volume grows beyond ten million points per hour, Option 2 may require re-evaluation. For the current volume and foreseeable growth over the next two years, TimescaleDB is sufficient.

**Consequences:** Positive — the transactional database is protected from overload; the team uses familiar tooling; the time-series data has its own scaling path. Negative — the team must manage one more database instance; the TimescaleDB extension adds minor operational complexity for upgrades and backups.

### Challenge 5 — Failure Mode Retrospective

A company launches a new e-commerce checkout service. The service is designed as a microservice that calls three external services — payment gateway, inventory service, and shipping calculator. The architect designed all calls as synchronous HTTP requests with five-second timeouts.

On launch day, the payment gateway suffers a transient slowdown — response times increase from 200ms to 4,000ms. Within sixty seconds, the checkout service's thread pool is exhausted. Within two minutes, the API gateway marks the checkout service as unhealthy and returns 503 errors to all users. The entire site cannot process orders for forty-five minutes.

Identify five design failures that contributed to this incident. For each failure, explain which design principle was violated and what the correct approach would have been.

**Time limit:** 15 minutes.

**Common mistake:** Listing generic failures without connecting each one to a specific design principle.

**Model answer:**

**Failure 1 — No timeouts tailored per dependency.** The architect used a uniform five-second timeout for all three external services. The inventory and shipping services respond in under 100ms. The payment gateway averages 200ms but can spike to 4,000ms. A per-service timeout should reflect each service's known latency distribution — 500ms for inventory and shipping, 3,000ms for the payment gateway. **Violated principle:** Explicit assumptions about dependency behavior.

**Failure 2 — No circuit breaker.** When the payment gateway became slow, every checkout request still attempted to call it. A circuit breaker pattern would detect that errors or timeouts exceed a threshold — for example, over fifty percent of calls fail within a thirty-second window — and open the circuit, immediately returning a fallback response without making the network call. **Violated principle:** Graceful degradation and failure containment.

**Failure 3 — No request prioritization or shedding.** When the thread pool began to exhaust, the service should have prioritized critical requests and shed lower-priority traffic. A production-grade approach uses thread pool isolation — one pool for payment flows, one pool for read-only flows — so that a payment slowdown does not block users from browsing. **Violated principle:** Bulkheading and blast radius limitation.

**Failure 4 — No graceful degradation.** When the payment gateway is down, the system should degrade gracefully rather than crash entirely. Users attempting checkout should see a message like "Payment processing is temporarily unavailable. Your cart has been saved." instead of HTTP 503. **Violated principle:** User-visible degradation over total failure.

**Failure 5 — No dependency health monitoring.** The team did not monitor the latency and error rate of each external dependency in real time. If they had, they would have detected the payment gateway slowdown within seconds and could have either switched to a backup gateway, throttled checkout traffic, or paged the payments team — all before the thread pool exhausted. **Violated principle:** Observability as a first-class architectural concern.

### Challenge 6 — Design Rigor Calibration

You are the technical lead for a team of six engineers building three systems simultaneously:

**System A:** An internal tool for the customer support team of fifteen people. It tracks support ticket status, resolution times, and customer satisfaction scores. Data volume: 500 tickets per day. Criticality: low — if the system is down, support agents use spreadsheets.

**System B:** A customer-facing API that processes credit card payments for a subscription service. Throughput: 10,000 payments per day, growing at five percent per month. Criticality: high — a thirty-minute outage costs approximately 50,000 dollars in failed transactions.

**System C:** A data pipeline that aggregates clickstream data from the company's web application. Volume: 100 million events per day. The pipeline feeds a machine learning model that powers the product recommendation engine. Criticality: medium — stale recommendations degrade conversion by an estimated two percent but do not cause a complete outage.

For each system, specify:
1. How much design rigor should be applied (low, medium, high)
2. What specific design activities are appropriate
3. What the deployment architecture should look like at launch
4. What operational tooling is needed

**Time limit:** 15 minutes.

**Evaluation rubric:** The answer must justify each calibration with concrete reasoning tied to the system's characteristics — user count, data volume, cost of failure, growth rate.

**Model answer:**

**System A — Internal support tool:**
- Design rigor: Low
- Activities: No ADRs needed. Define the database schema in thirty minutes. Use the team's existing web framework. Do not add caching, replication, or queueing.
- Deployment: Single server or shared Kubernetes namespace with a straightforward web app. One database instance. No load balancer needed for fifteen users.
- Operational tooling: Basic logging. A health check endpoint. Deploy via the team's existing CI/CD pipeline. No dedicated on-call — the team fixes it during business hours.
- Justification: The user count is bounded at fifteen. The cost of downtime is near zero. Any design time invested beyond the minimum is wasted.

**System B — Payment processing API:**
- Design rigor: High
- Activities: ADR for data model, consistency model, and retry strategy. Design review with at least one senior engineer who has built payment systems. Explicit consistency and durability guarantees documented. Idempotency key design. Fallback payment gateway integration.
- Deployment: Multi-AZ deployment with a load balancer and at least two application instances. Dedicated database with automated backups and point-in-time recovery. Read replicas for reporting queries. Dedicated Redis for idempotency key tracking.
- Operational tooling: Structured logging with correlation IDs. Metrics dashboard for latency, error rate, and throughput at P50, P95, P99. Alerts with page-worthy thresholds — any error rate above 0.1 percent, any P99 latency above 500ms. Canary deployment with automated rollback. Scheduled backup verification.
- Justification: The cost of failure is 50,000 dollars per thirty minutes. The growth rate of five percent per month means the system doubles in under fourteen months. Design investment that prevents failure pays for itself quickly.

**System C — Clickstream data pipeline:**
- Design rigor: Medium
- Activities: Evaluate two or three data pipeline frameworks. Design the schema for the raw events and the aggregated output. Define retention policies. Document the expected data volumes and the scaling plan.
- Deployment: Message queue for event ingestion. One or two worker instances for aggregation. A database or data lake for storage. This can start simple as long as the ingestion queue provides durability and replay capability.
- Operational tooling: Queue depth monitoring. Worker health checks. Basic latency tracking for end-to-end event processing. Alerts for queue growth beyond expected bounds (indicating a consumer failure).
- Justification: The system is important but not critical — stale recommendations reduce conversion by two percent but do not cause downtime. The data volume is large (100 million events per day) and demands some design attention, but the cost of a brief outage is modest. Medium rigor balances reliability with development velocity.

## Tier 3: Expert (Synthesis)

### Challenge 7 — System Evolution Strategy

A ride-sharing company has grown from a single-city operation to a fifty-city operation over three years. Their architecture evolved organically:

- Year 1: Monolith with Postgres. Everything in one codebase. Works for one city.
- Year 2: Added a separate driver-tracking service and a ride-matching service. Shared Postgres database between all services — they read and write each other's tables.
- Year 3: Added pricing optimization, surge detection, and ETA prediction as new services. The shared database now has over two hundred tables. No service owns its data — any service can read or write any table. Deployments are coordinated across all services because changing a schema affects every consumer.

Current problems: deploying a new version of any service requires all services to be deployed simultaneously because schema changes in one service's tables break other services. A bug in the pricing service can corrupt ride data because it writes directly to the ride table. The engineering team has grown from five to forty people, but velocity has decreased — they ship fewer features per sprint than they did with five people.

Design a three-phase migration strategy that moves this system from shared-database architecture to bounded-context architecture with per-service data ownership. Each phase should take three to six months and deliver measurable improvement.

**Time limit:** 30 minutes.

**What a strong answer covers:**
- Phase 1 identifies service boundaries, maps dependencies, creates read-only views
- Phase 2 uses the strangler fig pattern to wrap write paths behind APIs
- Phase 3 splits the shared database into per-context databases
- Each phase has a measurable improvement milestone
- The migration is incremental, not big-bang

**Anti-patterns (what weak answers do):**
- Proposing a big-bang rewrite ("move everything to new databases over a weekend")
- Skipping the dependency mapping phase and guessing boundaries
- No mention of how to handle the transition period when both old and new paths must work simultaneously

**Model answer:**

**Phase 1 — Boundary identification and read isolation (months 1-4):**

Goal: Identify service boundaries and stop untracked cross-service reads.

Approach:
1. Map every table to the services that read and write it. Use query logging, code analysis, and team interviews. Produce a dependency matrix showing which service touches which table.
2. Group tables into bounded contexts. Ride data (rides, fares, routes, driver_assignments) belongs to a Rides context. Pricing data (surge_multipliers, pricing_rules, event_logs) belongs to a Pricing context. Driver data (driver_profiles, vehicle_info, driver_availability) belongs to a Driver context.
3. For each bounded context, create a read-only view (a Postgres view or a materialized view) that exposes only the data that other contexts legitimately need. Redirect cross-service reads through these views.
4. Add a proxy layer that logs any direct table reads that bypass the views. Flag these as violations for the next phase.

Measurable improvement: Cross-service read tracking enables the team to quantify dependencies. The proxy logs reveal which teams are reading which tables — information that was previously invisible.

**Phase 2 — Write isolation with the strangler fig pattern (months 4-10):**

Goal: Give each service ownership of its writes. External services can no longer write to tables owned by another service.

Approach:
1. For each bounded context, wrap the write path behind a service API. Instead of the pricing service writing to the ride table directly, it calls the Ride Service's API endpoint: `POST /rides/{id}/pricing-data`.
2. Implement this as a strangler fig — introduce the API endpoint alongside the direct database access. Migrate consumers one at a time. During migration, both paths are active and produce the same write.
3. Use a validation layer that compares the output of the old path (direct write) with the new path (API call) and alerts on discrepancies.
4. Once all consumers of a table have migrated to the API, remove direct write access. Add a database trigger or a proxy that rejects direct writes and logs the violating service.

Measurable improvement: The blast radius shrinks. A bug in the pricing service can no longer corrupt ride data because the pricing service can only write pricing-related fields through the ride API, which validates the input.

**Phase 3 — Schema splitting and independent deployment (months 10-16):**

Goal: Split the shared database into per-context databases so each service owns its schema.

Approach:
1. Create a new database for each bounded context. Migrate tables to the new database using logical replication. The shared database still exists as the source of truth during migration.
2. The service that owns the context writes to the new database. Other services read from the old shared database through views (from Phase 1) or through the API (from Phase 2).
3. Once all consumers have migrated, drop the replicated tables from the shared database.
4. Each context's database is now deployed and scaled independently. Schema changes in one context do not affect others.

Measurable improvement: Deployment coordination drops from requiring all services to deploy simultaneously to zero coordination. Each team deploys independently.

**Key principle throughout:** Never perform a big-bang migration. Every phase works on the running system, every step is reversible, and every milestone delivers measurable reduction in coupling.

### Challenge 8 — Greenfield Design with Design Thinking

A company is building a greenfield platform for remote healthcare: video consultations, prescription management, medical record storage, appointment scheduling, insurance claims processing, and regulatory compliance (HIPAA, GDPR).

You are the principal architect. The company has twelve engineers, plans to launch in five cities within six months, and expects to expand to fifty cities within two years.

Design the system architecture with explicit justifications for each design choice. Address:
1. How you decompose the system into components
2. How you choose the data stores for each component
3. How you handle regulatory compliance (data residency, audit logging, access control)
4. How you plan for the growth from twelve engineers to forty engineers
5. How you handle the tension between the six-month launch deadline and the need for robust design

**Time limit:** 30 minutes.

**What a strong answer explores:**
- Decomposition by domain (bounded contexts), not by technology
- A modular monolith for the first six months, with extraction-ready boundaries
- Data store choices driven by access patterns, not familiarity
- Compliance built into the architecture from day one (data tagging, row-level security, append-only audit logs)
- A team growth plan that aligns with architectural extraction phases
- Explicit identification of which design investments are made upfront and which are deferred

**Anti-patterns (what weak answers look like):**
- Proposing microservices from day one for a team of twelve
- Forgetting compliance until "phase 2"
- No mention of how to handle the six-month deadline constraint
- Data store choice based on what is trendy rather than what fits the access pattern

**Model answer:**

**Architecture decomposition:**

The system decomposes into six bounded contexts based on domain analysis, not technical convenience:

| Context | Responsibility | Initial team | Grows to |
|---------|--------------|-------------|---------|
| Patient | Registration, authentication, profile, preferences | 2 engineers | 6 |
| Consultation | Video call orchestration, chat, session management | 3 engineers | 8 |
| Medical Records | Clinical data storage, structured data, FHIR API | 3 engineers | 10 |
| Pharmacy | Prescriptions, pharmacy integration, medication history | 2 engineers | 6 |
| Billing | Insurance claims, payment processing, invoicing | 1 engineer | 6 |
| Compliance | Audit logging, access control, data residency, reporting | 1 engineer | 4 |

**Initial deployment — modular monolith (first six months):**

All six contexts deploy as a single service in the first six months. This is not a traditional monolith — each context is a well-defined module with its own data access layer, its own tests, and a documented API boundary. The modules are deployed together for operational simplicity, but they are structured so that any module can be extracted into a standalone service later.

This decision explicitly trades deploy-time coupling for launch speed. The team of twelve engineers cannot operate six independently-deployed microservices — the operational overhead of CI/CD pipelines, service discovery, and cross-service debugging would consume the slack they need to build features. The modular monolith preserves the option to split later without paying the overhead today.

**Data store selection:**

- **Relational database (Postgres)** for patient profiles, appointments, prescriptions, and billing. These require ACID transactions, strong consistency, and complex relational queries. Postgres provides row-level security for compliance.
- **Document store (Postgres with JSONB + HNSW indexes)** for structured medical records. Medical records vary by specialty. JSONB provides schema flexibility while staying in the relational ecosystem.
- **Object storage (S3 or equivalent)** for video recordings, diagnostic images, and document scans. These are immutable blobs accessed by patient ID.
- **Search index (Elasticsearch or OpenSearch)** for full-text search across medical records, medications, and patient notes.

**Compliance implementation:**
- **Data residency:** Each patient's data is tagged with a region attribute. Write operations route to the database cluster in the patient's home region.
- **Audit logging:** Every data access — read or write — is logged to an append-only audit store. The audit record includes who accessed what data, when, from which IP, and what operation they performed.
- **Access control:** Role-based access control (RBAC) at the API gateway level plus row-level security at the database level. A doctor can see only their own patients' records.

**Engineering team growth plan:**

Phase 1 (0-6 months, 12 engineers): Monolith. Two squads share the codebase. Squad A owns Patient and Consultation, Squad B owns Medical Records and Pharmacy.

Phase 2 (6-18 months, 15-20 engineers): Begin extraction. When a context's module becomes a bottleneck — measured by deploy conflicts or team coordination overhead — extract it into a standalone service. The documented API boundary makes extraction a structured operation.

Phase 3 (18+ months, 20-40 engineers): Fully extracted services. Each context is a standalone service with its own deployment pipeline, database, and on-call rotation.

**Tension resolution — launch speed versus design robustness:**

The specific design investments made upfront:
1. Well-defined module boundaries with documented interfaces (cost: approximately one week per context)
2. Row-level security from day one (cost: approximately three days of database configuration)
3. Append-only audit logging (cost: approximately two days of pipeline setup)
4. Structured logging with correlation IDs (cost: approximately one day of middleware configuration)

These investments add approximately three weeks to the initial build timeline. In exchange, the system is audit-ready at launch, can be decomposed without a rewrite, and will not require a multi-million-dollar migration like PayPal's.

The design choices not made upfront — independent deployment pipelines, per-service databases, distributed tracing, orchestration platform — can be deferred until the team size and system scale justify their operational overhead. This is the calibrated design thinking that the entire topic advocates: invest where the cost of being wrong is high; defer where the cost of changing later is low.

The ability to reason at all three tiers — foundation knowledge, applied design, and expert synthesis — is the mark of an engineer who truly understands why system design matters.
