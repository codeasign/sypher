## How This Topic Appears

Distributed ID questions test your ability to design systems that are globally unique, scalable, and available. The topic appears across multiple question categories:

**Direct (you are told the topic):**
1. "Design a distributed unique ID generator for a system handling 10,000 IDs per second."
2. "How would you generate unique IDs across 100 database shards?"
3. "Compare and contrast Snowflake IDs, UUID v4, and database sequences."
4. "What bit layout would you choose for a 64-bit distributed ID?"

**Design (the topic is a sub-problem):**
5. "Design a URL shortener like TinyURL. How do you generate unique short codes?"
6. "Design a chat system like WhatsApp. How do you generate message IDs that order correctly in real-time?"
7. "Design a distributed event logging system. Each event needs a unique, sortable ID."
8. "Design a sharded database for a social media platform. How do you assign primary keys?"

**Debugging:**
9. "Your database has duplicate primary key errors. The IDs are generated by a distributed ID service. Walk through the debugging process."
10. "Events in your system are appearing out of order even though they are sorted by ID. What went wrong?"

**Trade-off:**
11. "Should we use UUID v4 or Snowflake IDs for our new microservice?"
12. "Our Snowflake generator has started blocking for 1-5ms on every generate() call. Diagnose and fix."
13. "Should we embed the shard ID in our primary key, or use a separate lookup table?"

## What the Interviewer Is Actually Testing

When interviewers ask about distributed IDs, they are evaluating:

**System design fundamentals:** Can you decompose a seemingly simple problem (generate a unique number) into components, constraints, and trade-offs? Distributed ID questions are a microcosm of systems design — they test bit-level optimization, distributed coordination, and operational monitoring in a single problem.

**Scalability intuition:** Can you estimate throughput requirements and identify bottlenecks? The interviewer wants to hear you calculate "4,096 IDs per millisecond per worker" and reason about whether that is enough.

**Failure awareness:** Do you know what breaks and why? Candidates who only describe the happy path (unique IDs, fast generation) miss the clock skew handling, partition behavior, and overflow scenarios that separate production-ready designs from toy implementations.

**Database internals knowledge:** Understanding B-tree index behavior, clustered vs. non-clustered indexes, and write amplification from random inserts distinguishes strong candidates.

**Practical engineering judgment:** Can you evaluate whether the system actually needs distributed IDs, or would a simpler solution (auto-increment, UUID v4) suffice? The best candidates match the solution to the scale, not the other way around.

## Level-by-Level Expectations

### L3 / L4 (Early Career)

**What they must know:**
- The basic problem: centralized ID generation does not scale
- UUID v4 exists and generates unique IDs
- Database auto-increment works for single-node systems
- 32-bit integer IDs overflow (2.1 billion limit)

**Good answer:** "I would use UUID v4 because it guarantees uniqueness without coordination. It is 128 bits and not sortable, but for our scale of 1 million records, that is fine."

**Bad answer:** "I would just use an auto-increment column."

### L5 (Senior Engineer)

**What they must know:**
- Snowflake algorithm structure and bit layout
- UUID v4 vs. Snowflake trade-offs (storage, ordering, index performance)
- Worker ID assignment and coordination
- Clock skew handling
- B-tree index fragmentation from random UUIDs

**Good answer:** "I would use a Snowflake-style 64-bit ID with a 41-bit timestamp, 10-bit worker ID, and 12-bit sequence. This gives us time-sortable IDs that fit in a BIGINT column, with zero coordination per ID generation. Worker IDs would be assigned via ZooKeeper, with a 60-second TTL lease. The system would handle 4 million IDs per second across 1,024 workers."

**Bad answer:** "I would generate a random 64-bit number and hope there are no collisions."

### L6 (Staff Engineer)

**What they must know:**
- Everything at L5 level, plus:
- Bit budget optimization for specific workload (different worker/sequence ratios)
- Multi-region deployment with regional prefixes
- Hybrid Logical Clocks as an alternative to physical clocks
- Pre-allocation and buffer tuning for high-throughput paths
- Monitoring and alerting for clock skew, sequence overflow, generation latency
- Capacity planning: number of workers needed per region at peak throughput

**Good answer:** "At our scale of 500,000 IDs per second across 3 regions, I would allocate 2 bits for the region identifier, 7 bits for worker ID (128 workers per region), and 14 bits for sequence (16,384 IDs per ms per worker). The extra sequence bits handle traffic spikes without spin-wait. I would embed the generation library as a sidecar in each service rather than use a remote ID service, keeping p99 latency under 0.2ms."

**Bad answer:** "This is a solved problem. Just copy Twitter's Snowflake code exactly."

### L7+ (Principal / Distinguished Engineer)

**What they must know:**
- Everything at L6 level, plus:
- Trade-offs of building vs. buying (Spanner TrueTime, Cosmos DB)
- Long-term migration strategies when ID format needs to change
- Organizational impact: which team owns the ID service, how it is versioned
- Business-level justification for ID choices (e.g., competitive blindness from sequential IDs, audit trail requirements)
- Cross-organizational standardization of ID format across dozens of services

**Good answer:** "I would standardize on a single ID format across all 200 services. The coordination service would be owned by the infrastructure platform team, offering a client library that handles worker registration, clock monitoring, and ID generation. Each service would embed the library — the cost of the extra 5MB per container is negligible compared to the operational savings of having zero ID collisions across the fleet for 5+ years."

## The FAANG Framework for Answering

Use this 6-step framework when you get a distributed ID design question:

**Step 1: Scope and requirements (2 minutes).** Ask clarifying questions:
- "What is the expected peak throughput (IDs per second)?"
- "What are the storage constraints? Are we using MySQL, Cassandra, S3?"
- "Do we need time-sortable IDs (e.g., for timeline display)?"
- "Is there any business requirement about sequential, gap-free IDs?"
- "How many regions do we deploy in?"

**Step 2: Constraint identification (1 minute).** Identify what forces choice:
- If under 10K IDs/sec and no ordering need: UUID v4 is fine
- If over 1M IDs/sec: must use Snowflake or ULID
- If multi-region: need region prefixes in the ID

**Step 3: Core design (3 minutes).** Present the chosen scheme:
- "I will use a 64-bit Snowflake-style ID with this bit layout..."
- "Worker IDs are assigned via a 3-node etcd cluster..."
- "Each worker generates IDs locally ensuring zero cross-node communication..."
- "The full system supports N million IDs per second..."

**Step 4: Deep dive — failure modes (2 minutes).** Address the hard parts before being asked:
- "Clock skew is handled by blocking generation until the clock catches up..."
- "Sequence overflow resolves within 1ms via spin-wait..."
- "Network partitions do not affect existing workers..."
- "Worker ID collisions are prevented by coordination service TTL leases..."

**Step 5: Scaling and optimization (2 minutes).** Discuss how it handles growth:
- "At 3x current traffic, we add more workers and increase sequence bits..."
- "For global deployment, we add a 2-bit region prefix..."
- "For latency-critical paths, we pre-allocate IDs into a local buffer..."

**Step 6: Alternatives (1 minute).** Show you considered other options:
- "The alternative is UUID v7, which trades 64-bit storage for simpler implementation..."
- "Another option is a database sequence with step increments, which is simpler but limits throughput..."

## Phrase Bank

Use these phrases in interviews to demonstrate depth:

1. **"The bit budget is a zero-sum game."** — When explaining that adding more sequence bits requires reducing timestamp or worker bits.

2. **"The sign bit is fixed at 0 for signed integer compatibility."** — Shows attention to the practical detail that Snowflake IDs must be positive integers for most languages' database drivers.

3. **"The sequence overflows when all 4,096 values per millisecond are exhausted, causing a spin-wait."** — Demonstrates understanding of the overflow mechanism rather than treating it as magic.

4. **"Clock skew is the most dangerous failure mode in timestamp-based ID schemes."** — Signals awareness of the fragility of system clocks in distributed systems.

5. **"Two workers on different sides of a network partition both continue generating IDs independently."** — Shows understanding of CP positioning and graceful degradation.

6. **"UUID v4 causes B-tree page splits on every insert, increasing write amplification by 10x at scale."** — Demonstrates database internals knowledge beyond the ID itself.

7. **"The 41-bit timestamp gives 69 years from the custom epoch, which is sufficient for any system built today."** — Shows you have done the math on the bit range.

8. **"Pre-allocation moves the generation cost off the critical path."** — Signals optimization thinking.

9. **"Hybrid Logical Clocks would eliminate the clock skew problem at the cost of adding a 64-bit clock field to every ID."** — Shows knowledge of the academic alternative.

10. **"The coordination service for worker ID assignment is the system's only single point of failure."** — Identifies the real operational risk.

11. **"At 8 bytes per ID, 1 billion IDs consume 8GB of storage for the ID column alone, plus index overhead."** — Shows quantitative thinking.

12. **"Emb edding shard IDs in the primary key eliminates one network round-trip per query."** — Connects ID format to query performance.

13. **"Sequence persistence is not needed because the timestamp component advances monotonically."** — Explains why crash recovery is safe without disk writes.

14. **"The cross-worker monotonicity guarantee is best-effort, not absolute."** — Sets realistic expectations about ordering across nodes.

15. **"Dual-write migration lets us change the ID format without a flag day."** — Shows operational maturity in format upgrades.

## Red Flags

1. **"UUID is unique because it uses random numbers."** — UUID does not guarantee uniqueness; it guarantees a very low probability of collision. The distinction matters for systems where collision is catastrophic.

2. **"Snowflake never blocks."** — Snowflake blocks on clock skew and sequence overflow. Claiming it never blocks shows you have not thought about edge cases.

3. **"We only need one worker because it generates 4 million IDs per second."** — Single-worker designs still have a single point of failure. Interviewers test redundancy reasoning.

4. **"I would use Snowflake for everything."** — Not every system needs distributed IDs. Using Snowflake for a blog with 100 posts/day is over-engineering.

5. **"The timestamp makes IDs globally ordered."** — See "Lessons Learned" about cross-worker ordering. Timestamps only guarantee monotonicity within a single worker.

6. **"We should make our IDs sequential for user-friendliness."** — Sequential IDs leak business metrics, create security concerns, and create contention. Explain why opaque IDs are better.

7. **"I would just use the database auto-increment."** — Works at small scale but fails at moderate scale. If the interviewer specified "distributed," they are testing beyond single-node knowledge.

8. **"Random 64-bit IDs have a negligible collision probability."** — The birthday paradox means you have a 50% collision probability after generating only 2^32 (4 billion) random 64-bit IDs. That is not negligible at scale.

## Practice Drills

**Drill 1 (5 minutes):** You are designing IDs for a URL shortener that handles 1,000 new URLs per second. The short codes should be 7-8 characters. Design the ID scheme. (Focus on: encoding 64-bit IDs to base62 for compact URLs, Snowflake timestamp for time-ordering, collision avoidance.)

**Drill 2 (5 minutes):** Your team uses UUID v4 for all database primary keys. Write queries against a 500GB table are slowing down over time. Diagnose and propose a fix. (Focus on: B-tree page splits, fragmented indexes, migration to UUID v7 or Snowflake.)

**Drill 3 (10 minutes):** Your Snowflake ID generator's workers use clock_gettime() on Linux. You notice p99 generation latency increased from 0.1ms to 5ms after a recent deployment. What could have changed? (Focus on: container CPU throttling, noisy-neighbor CPUs, kernel clock source switching from TSC to HPET.)

**Drill 4 (15 minutes):** Your system uses Snowflake IDs across 3 regions. You discover that IDs from region A are sometimes numerically larger than IDs from region B, even though the region A events happened after region B events. Is this a bug? (Focus on: cross-region clock skew is normal, ID timestamp does not guarantee real-time ordering across physically separate clocks, Hybrid Logical Clocks as a fix.)

**Drill 5 (15 minutes):** Design a distributed ID system for a payment processing platform. The platform handles 5,000 transactions per second, needs audit-compliant IDs (no gaps, sequential), and operates across 2 data centers with active-active replication. (Focus on: The gapless requirement conflicts with distributed generation — you must decide which requirement to compromise. Explore single-master failover for gapless sequences, or use Snowflake with a gap-acceptance audit trail.)
