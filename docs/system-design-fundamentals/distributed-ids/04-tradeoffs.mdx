## The Core Tension

Distributed ID systems trade perfect ordering and zero-collision guarantees for the ability to generate IDs without cross-node coordination — the less you coordinate, the more you risk collisions or ordering violations, and the more you coordinate, the slower and less available your ID generation becomes.

## Decision Matrix

| Decision Point | Option A | Option B | Choose A When | Choose B When |
|---|---|---|---|---|
| ID width | 64-bit integer | 128-bit (UUID) | Database space is constrained, IDs must fit in BIGINT, or you need B-tree efficiency | No storage constraints, you want zero-coordination guarantees, or IDs are generated client-side |
| Timestamp source | System clock | Hybrid logical clock (HLC) | NTP is reliable, clock skew is under 10ms most of the time | NTP is unreliable, containers frequently pause, or clock monotonicity is non-negotiable |
| Worker ID assignment | Static config file | Dynamic (ZooKeeper/etcd) | Under 10 workers, deployments are infrequent, manual error is acceptable | Over 10 workers, auto-scaling is used, or zero-touch deployments are required |
| Sequence persistence | Memory only | Redis/disk-backed | Crash recovery is acceptable (see next section) | Zero ID reuse after crash required, or audit compliance demands it |
| Encoding | Raw integer | Base62/Base64 string | IDs stay in internal systems and databases | IDs appear in URLs, user-facing pages, or need to be compact |
| Collision strategy | Prevention (math guarantee) | Detection + retry | Throughput and latency requirements fit within bit-budget guarantees | Unbounded scale, relaxed ordering requirements, or you accept probabilistic uniqueness |
| Generation location | Local library (embedded) | Remote service (RPC) | Single language monolith, or latency is critical (under 0.5ms) | Polyglot environment, or you want to centralize coordination logic |

## CAP Theorem Positioning

Distributed ID systems are **CP** (Consistency and Partition Tolerance) systems under the CAP theorem. They sacrifice availability during clock skew events to maintain consistency (no duplicate IDs). However, the availability loss is bounded (milliseconds) and local (only the affected worker).

**Partition behavior:** During a network partition, existing workers on each side continue generating IDs with no impact. New workers cannot register (coordination service is unreachable), but running workers are unaffected. This is the ideal partition behavior: the system degrades gracefully without violating consistency.

**Availability impact:** The availability sacrifice is limited to:
- Clock skew events (typically under 2 seconds per event, events per month on healthy NTP)
- Sequence overflow (under 1ms per overflow event)
- Coordination service unavailability for new workers only

## Failure Modes

### Failure 1: Worker ID Collision (Duplicate IDs)

**What:** Two workers are assigned the same worker ID, causing duplicate IDs.
**Why:** Coordination service bug, stale lease not expired, manual configuration error.
**Blast radius:** All IDs generated by both workers during the overlap period will collide when used as primary keys.

### Failure 2: Large Clock Jump

**What:** The system clock on a worker jumps forward by several seconds or minutes.
**Why:** NTP correction after the clock drifted, VM migration with unsynchronized host clock, or manual clock adjustment.
**Blast radius:** All IDs generated on this worker during the clock jump window appear to come from the future. Other workers generating IDs during the gap produce IDs with timestamps that are numerically smaller, breaking time-ordering.

**Detection:** The generator's clock skew detector catches backward jumps immediately. Forward jumps are harder to detect because timestamps naturally increase — there is no "too far forward" check in standard Snowflake implementations.

**Recovery:** Monitor the difference between the ID's embedded timestamp and the actual wall clock across all workers. If a worker's IDs consistently embed timestamps ahead of the current time, that worker's clock is fast. Manually reset the clock via NTP and wait for IDs to normalize.

### Failure 3: Sequence Overflow Storm

**What:** Multiple workers simultaneously hit sequence overflow (more than 4096 IDs in a millisecond).
**Why:** Unexpected traffic spike (e.g., flash crowd, DDoS, Super Bowl ad).
**Blast radius:** The affected workers spin-wait, which consumes 100% CPU on one core. Application threads blocked on `generate()` back up, causing cascading timeouts in dependent services.

**Detection:** CPU usage on ID generation workers spikes from under 5% to 100%. The application's `generate()` call latency increases from under 0.1ms to 1ms (the spin-wait duration).

**Recovery:** No manual action needed — the spin-wait resolves within 1ms. For repeated events, add more workers or increase sequence bit width. An emergency circuit breaker can switch to a backup ID scheme (e.g., fall back to random UUIDs) during sustained overflow.

### Failure 4: Coordination Service Outage

**What:** The ZooKeeper/etcd cluster loses quorum or becomes unavailable.
**Why:** Network partition, majority of nodes fail, or configuration error.
**Blast radius:** Existing workers continue generating IDs indefinitely (they only contact the coordinator at startup). New worker deployments fail to start. Auto-scaling events fail.

**Detection:** Deployment pipeline fails with "cannot register worker." Monitoring shows 0 healthy ID service instances in the auto-scaling group.

**Recovery:** Restore the coordination service cluster. Existing IDs remain valid. New workers register once the coordinator is healthy. No data loss — only a deployment freeze.

### Failure 5: Long-Term Monotonicity Break

**What:** IDs generated at time T+1 are numerically smaller than IDs generated at time T.
**Why:** This is an inherent property of cross-worker generation. Worker A at time T generates ID(T, 1, 0) = 1000. Worker B at time T+5 generates ID(T+5, 2, 0) = 5000. But Worker C (with a faster clock) at time T+2 generates ID(T+10, 3, 0) = 8000 — an ID from the future that beats Worker B's later ID.

**Detection:** Rarely caught proactively. Systems that depend on strict ID ordering discover this through bug reports (e.g., pagination skips an item, or a timeline shows items out of order).

**Recovery:** Accept that cross-worker monotonicity is a best-effort property. For strict ordering, use a single worker or use Hybrid Logical Clocks (HLCs) that synchronize clock state.

## Anti-Patterns Hall of Shame

**Anti-pattern 1: Using UUID v4 for clustered primary keys.** In MySQL InnoDB, the clustered index is physically ordered by primary key. UUID v4 inserts cause page splits on every single insert, increasing write amplification by 10-100x and causing index fragmentation that persists until the table is rebuilt. Real impact: a system with 500GB of data required 5TB of index space due to UUID v4 fragmentation.

**Anti-pattern 2: Using database sequences across shards.** Database sequences do not span shards. With 10 shards each using auto-increment, you get 10 records with ID 42. Merging query results across shards requires composite keys or application-level deduplication.

**Anti-pattern 3: Relying on NTP for cross-worker ordering.** NTP synchronization is typically within 1-100ms of true time, not 0ms. Two workers at NTP sync offsets of +5ms and -5ms have a 10ms window of unordered IDs. If strict ordering matters, use a centralized sequencer, not distributed timestamps.

**Anti-pattern 4: Putting the ID generator behind a load balancer.** Load balancers distribute requests across workers. Two consecutive requests from the same client may hit different workers, returning IDs that are not monotonic from the client's perspective. If monotonic IDs per client session matter, pin clients to a specific worker.

**Anti-pattern 5: Encoding business logic in IDs.** Embedding tenant IDs, shard numbers, or timestamps that reveal business information in IDs. IDs should be opaque to consumers. Exposing the shard number in the ID makes re-sharding a public API break. Exposing timestamps reveals the rate of new record creation (a security concern for competitive intelligence).

**Anti-pattern 6: Sequential IDs in public URLs.** If your API returns `/order/1001` and `/order/1002`, anyone can infer your order volume. Competitive analysis tools scrape sequential IDs to estimate growth rates. Use opaque IDs (e.g., base62-encoded Snowflake IDs) in public URLs.

**Anti-pattern 7: Assuming IDs are secret.** IDs are identifiers, not secrets. A UUID v4 has 122 bits of randomness, which provides some unpredictability, but Snowflake IDs encode predictable timestamps and sequential counters. Never use IDs as access control tokens, session identifiers, or CSRF tokens.

**Anti-pattern 8: Storing IDs as strings when they fit in integers.** A 64-bit Snowflake ID stored as a BIGINT in SQL or int64 in Go/Python uses 8 bytes. The same ID stored as its decimal string "1912345678901234567" uses 19 bytes of text plus variable-length overhead. String indexes are larger and slower than integer indexes.

## When NOT to Use This

Distributed ID generation (Snowflake/ULID style) is not the right choice in these situations:

- **You need global, strict monotonic ordering across all nodes.** Timestamp-based schemes can only guarantee ordering within a single worker, not across workers. Use a centralized sequencer (e.g., Google's TrueTime API, or a single-master database sequence).

- **Your system is a single-node application that will never scale horizontally.** A database auto-increment column is simpler, faster, and requires zero infrastructure. Distributed ID generation adds complexity with no benefit.

- **You are on resource-constrained hardware (IoT, mobile).** The clock read, CAS instructions, and spin-wait loops add overhead that matters on a microcontroller. Use pre-assigned ID blocks or random UUIDs.

- **Your compliance requirements demand sequential, gap-free IDs.** Regulated industries (invoicing, tax IDs) often require gapless sequential numbering. Distributed IDs produce gaps (from overflow, crashed workers, pre-allocated but unused IDs). Use a single-threaded centralized counter instead.

- **You need cryptographic unpredictability.** Snowflake IDs are predictable: given one ID, an attacker can estimate the next one within a small range. For opaque tokens that must be unguessable, use secure random IDs (v4 UUID or 128+ bits of CSPRNG output).
