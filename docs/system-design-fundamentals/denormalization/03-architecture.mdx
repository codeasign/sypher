## Naive Implementation

The simplest denormalization approach — a single table with all fields — is tempting but dangerous. Here is what happens when you try it.

```sql
-- Naive denormalization: one table to rule them all
CREATE TABLE everything (
  order_id BIGINT PRIMARY KEY,
  customer_id INT,
  customer_name VARCHAR(200),
  customer_email VARCHAR(200),
  customer_address_street VARCHAR(200),
  customer_address_city VARCHAR(100),
  customer_address_zip VARCHAR(20),
  order_date TIMESTAMP,
  order_total DECIMAL(10,2),
  product_id INT,
  product_name VARCHAR(200),
  product_category VARCHAR(100),
  product_sku VARCHAR(50),
  quantity INT,
  unit_price DECIMAL(10,2),
  line_total DECIMAL(10,2),
  payment_method VARCHAR(50),
  payment_status VARCHAR(20),
  shipping_method VARCHAR(50),
  shipping_status VARCHAR(20),
  -- 30 more columns...
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

### What Goes Wrong

**Storage bloat**: The table is 4x larger than the normalized alternative. With 100 million orders and 3 items per order, the table has 300 million rows. Each row is 2 KB. Total: 600 GB. The normalized version: 150 GB.

**Write amplification**: Every INSERT writes 2 KB. Every UPDATE of a customer name must scan and update all rows for that customer. If a customer with 50 orders changes their name, that is 50 row updates in one transaction.

**Update anomalies**: A customer updates their email address. You run:

```sql
UPDATE everything SET customer_email = 'new@email.com'
WHERE customer_id = 1234;
```

This updates 50 rows. One row was written by a concurrent process during the UPDATE — its NEW value was correct but the UPDATE already passed over it. That row now has the old email. You have a corrupted read.

Without an exclusive lock on all matching rows (which blocks concurrent writes), you cannot guarantee consistency during the UPDATE window.

**Schema rigidity**: Adding a column to this table requires a full-table ALTER TABLE, which can lock the table for hours on 600 GB.

```sql
ALTER TABLE everything ADD COLUMN loyalty_tier VARCHAR(20);
-- Locks table for 15-30 minutes on 600 GB
```

During the ALTER, no reads or writes can proceed. The application is down.

**Scalability ceiling**: Indexing this monstrosity is impractical. A B-tree index on 300 million rows of 2 KB each needs gigabytes. A full table scan becomes the fastest option for many queries.

**Conclusion**: The naive approach works for small datasets (under 100,000 rows) but collapses under production loads. A production-grade architecture separates the write path from the read path and uses a pipeline to keep denormalized data consistent.

### Layer 1: OLTP Write Path

The application writes to a normalized database. This is the system of record. Every insert, update, and delete goes through standard transactional processing.

**Key properties**:
- ACID transactions guarantee consistency
- Normalized schema optimizes write throughput
- Indexes serve point lookups, not full-table scans
- Data is immediately consistent after each transaction

**Example**: A customer places an order. The application inserts one row into `orders`, three rows into `order_items`, and updates the `inventory` count in `products`. All in one transaction. The normalized schema makes this fast and safe.

### Layer 2: Change Data Capture

CDC captures every change made to the OLTP database and streams it to a durable message queue.

**Implementation options**:
- **Debezium** (open source): Connects to PostgreSQL, MySQL, MongoDB, SQL Server, Oracle. Reads the WAL or binlog. Emits change events to Kafka.
- **AWS Database Migration Service (DMS)**: Supports continuous replication from Aurora, RDS, and on-prem databases.
- **Confluent Kafka Connect**: Managed Debezium connector with schema registry.

**Event format** (Debezium JSON):

```json
{
  "schema": { ... },
  "payload": {
    "op": "c",
    "before": null,
    "after": {
      "order_id": 123456,
      "customer_id": 7890,
      "order_date": "2025-11-15T14:30:00Z",
      "total": 149.99
    },
    "source": {
      "table": "orders",
      "ts_ms": 1731683400000
    }
  }
}
```

**Durability guarantee**: Kafka persists events with configurable replication factor (3 is standard). Even if the ETL pipeline is down, events are available for replay.

### Layer 3: Transformation

Two competing approaches:

**Streaming (Flink, Spark Streaming, Kafka Streams)**: Processes each event as it arrives. Denormalization happens in near-real-time.

```java
// Conceptual Flink job: denormalize order + customer
DataStream<OrderEvent> orders = env.addSource(kafkaSource("db.orders"));
DataStream<CustomerEvent> customers = env.addSource(kafkaSource("db.customers"));

orders
  .keyBy(order -> order.customerId)
  .connect(customers.keyBy(customer -> customer.customerId))
  .flatMap(new JoinFunction() {
    public void flatMap(OrderEvent order, CustomerEvent customer, Collector out) {
      out.collect(new DenormalizedOrder(
        order.orderId,
        customer.name,
        customer.email,
        order.orderDate,
        order.total
      ));
    }
  })
  .addSink(jdbcSink("order_reports"));
```

**Batch (dbt, Airflow, Spark SQL)**: Runs on a schedule. Processes all changes since the last run in one batch.

```sql
-- dbt model: order_reports.sql
SELECT
  o.order_id,
  c.name AS customer_name,
  c.email AS customer_email,
  o.order_date,
  o.total
FROM {{ ref('orders') }} o
LEFT JOIN {{ ref('customers') }} c ON o.customer_id = c.customer_id
{% if is_incremental() %}
WHERE o.order_date > (SELECT MAX(order_date) FROM {{ this }})
{% endif %}
```

**Which one to choose**:

| Factor | Streaming | Batch |
|--------|-----------|-------|
| Data freshness | Seconds | Minutes to hours |
| Infrastructure cost | Higher (always-on compute) | Lower (scheduled compute) |
| Operational complexity | Higher (state management, backpressure) | Lower (run on schedule, retry on failure) |
| Replay capability | Requires state rebuild | Natural with backfill |
| Change handling | Real-time DELETEs and UPDATEs | Scheduled refresh |

### Layer 4: Data Warehouse Storage

The warehouse stores denormalized data optimized for read workloads.

**Schema design**:
- **Fact tables**: Partitioned by date, sorted by most frequent filter column, columnar compression
- **Dimension tables**: Denormalized (flat), SCD Type 2 for slowly changing attributes
- **Aggregate tables**: Precomputed summaries at daily, weekly, monthly granularity

**Warehouse-specific optimizations**:

```sql
-- Snowflake: automatic clustering
CREATE TABLE fact_sales CLUSTER BY (date_sk, product_sk) AS
SELECT ...;

-- Redshift: distribution and sort keys
CREATE TABLE fact_sales (
  date_sk INT,
  product_sk INT,
  quantity INT,
  total DECIMAL(10,2)
)
DISTKEY (product_sk)
SORTKEY (date_sk, product_sk);

-- BigQuery: partitioning and clustering
CREATE TABLE fact_sales
PARTITION BY DATE(date_sk)
CLUSTER BY product_sk
AS SELECT ...;
```

### Layer 5: Cached Access

The warehouse is fast, but a cache in front of it is faster for the hottest queries.

**Redis cache for top-N queries**:
```python
def get_daily_summary(date):
    cache_key = f"daily_summary:{date}"
    result = redis.get(cache_key)
    if result:
        return json.loads(result)

    result = warehouse.execute("""
        SELECT category, SUM(total) AS revenue
        FROM fact_sales
        WHERE date_sk = %s
        GROUP BY category
    """, [date])

    redis.setex(cache_key, 300, json.dumps(result))  # 5 minute TTL
    return result
```

**Elasticsearch for search and exploration**:
Denormalized data is indexed into Elasticsearch for free-text search and complex filtering.

### Reconciliation Layer

A periodic job that validates the denormalized data against the source of truth.

```python
def reconcile():
    issues = []

    # Check row counts match
    source_count = db.query("SELECT COUNT(*) FROM orders")
    denorm_count = warehouse.query("SELECT COUNT(*) FROM fact_sales")
    if source_count != denorm_count:
        issues.append(f"Row count mismatch: {source_count} vs {denorm_count}")

    # Check aggregate consistency
    source_agg = db.query("""
        SELECT DATE_TRUNC('day', order_date) AS day, SUM(total)
        FROM orders GROUP BY 1 ORDER BY 1
    """)
    denorm_agg = warehouse.query("""
        SELECT day, SUM(total)
        FROM fact_sales GROUP BY day ORDER BY day
    """)
    for s, d in zip(source_agg, denorm_agg):
        if abs(s.total - d.total) > 0.01:
            issues.append(f"Drift on {s.day}: {s.total} vs {d.total}")

    if issues:
        alert_pagerduty(issues)
        trigger_incremental_refresh()
```

**Run frequency**: Every 6 hours for enterprise systems, hourly for critical systems.

**Example**: An e-commerce system:
- Write model: `orders` table (normalized) accepts commands like `PlaceOrder`
- Event: `OrderPlaced` emitted to Kafka
- Projection handler: listens for `OrderPlaced`, builds denormalized `order_details` document
- Read model: MongoDB collection with all order fields in one document

Querying is a single document lookup — no JOINs, no aggregation. The complexity is moved to the projection handler.

### Denormalized Reporting Database

Run a separate database instance dedicated to reporting queries, populated via ETL from the operational database.

```yaml
# docker-compose for sidecar reporting DB
version: '3'
services:
  app-db:
    image: postgres:16
    environment:
      POSTGRES_DB: app
    volumes:
      - app-data:/var/lib/postgresql/data

  reporting-db:
    image: postgres:16
    environment:
      POSTGRES_DB: reports
    volumes:
      - reporting-data:/var/lib/postgresql/data

  etl:
    image: python:3.12
    command: python /etl/run.py
    volumes:
      - ./etl:/etl
    depends_on: [app-db, reporting-db]
```

**Benefits**:
- Reporting queries do not compete with application queries for database resources
- The reporting DB can have different indexes, compression, and configuration
- The reporting DB can be scaled independently (more replicas for more dashboards)

**Cost**: Duplicate storage, ETL maintenance, data freshness lag.

### Denormalized Elasticsearch Index

Elasticsearch is inherently denormalized — each document contains all fields needed for search and filtering.

```python
# Index a denormalized order document into Elasticsearch
def index_order(order_id):
    # Fetch from normalized source
    order = db.query("""
        SELECT o.*, c.name, c.email, p.name AS product_name
        FROM orders o
        JOIN customers c ON o.customer_id = c.customer_id
        JOIN order_items oi ON o.order_id = oi.order_id
        JOIN products p ON oi.product_id = p.product_id
        WHERE o.order_id = %s
    """, [order_id])

    # Denormalize into one document
    doc = {
        "order_id": order.order_id,
        "customer_name": order.name,
        "customer_email": order.email,
        "product_names": [item.product_name for item in order.items],
        "total": order.total,
        "order_date": order.order_date.isoformat()
    }

    es.index(index="orders", id=order_id, body=doc)
```

**Elasticsearch as a read model**: The index is rebuilt from the source on every change. The search query never JOINs — it just searches the denormalized document.

### Denormalized Redis Cache

Redis caches denormalized responses, avoiding database lookups entirely.

```python
def get_order_summary(order_id):
    # Try cache first
    summary = redis.get(f"order:{order_id}:summary")
    if summary:
        return json.loads(summary)

    # Cache miss: build denormalized result from normalized source
    rows = db.query("""
        SELECT o.total, o.order_date, c.name,
               p.name AS product, oi.quantity
        FROM orders o
        JOIN customers c ON o.customer_id = c.customer_id
        JOIN order_items oi ON o.order_id = oi.order_id
        JOIN products p ON oi.product_id = p.product_id
        WHERE o.order_id = %s
    """, [order_id])

    summary = {
        "order_date": rows[0].order_date,
        "total": rows[0].total,
        "customer": rows[0].name,
        "items": [{"product": r.product, "qty": r.quantity} for r in rows]
    }

    redis.setex(f"order:{order_id}:summary", 3600, json.dumps(summary))
    return summary
```

**Cache invalidation**: When the source data changes, invalidate the cache entry and rebuild on next read. Use Redis pub/sub or a separate invalidation stream.

### Replication Strategy

**Cross-region warehouse replication**:
- Snowflake: Database replication across regions (near-real-time, automatic)
- Redshift: Cross-region snapshot copy (hourly, manual scheduling needed)
- BigQuery: Cross-region dataset copy (periodic via scheduled queries)

**Replication latency**:
- Same-continent: 1-5 seconds
- Cross-continent: 5-30 seconds
- Intercontinental: 10-60 seconds

### Consistency Challenges

A user in Europe places an order. The data flows: us-east-1 OLTP → ETL → us-east-1 warehouse → replication → eu-west-1 replica. Total propagation time: 10-30 seconds.

If the user immediately opens the dashboard, their order will not appear until the replication completes. This is eventual consistency across regions.

**Mitigations**:
- **Local writes**: For interactive applications, write to the nearest region and replicate asynchronously. Prefer CRDT or last-writer-wins conflict resolution.
- **Session stickiness**: Route the user to the same region for both read and write for the duration of their session, so they see their own writes immediately.
- **Read-after-write consistency**: The write response includes a timestamp or sequence number. Reads block until the replica has applied at least that sequence number.

### Capacity Planning

Plan storage and compute for a multi-region denormalized system.

**Storage**:
```
Source data: 500 GB (normalized)
Fact table:  800 GB (denormalized, pre-joined)
Dimensions:   50 GB
Aggregates:  200 GB
Indexes:     150 GB
Total (one region): 1.2 TB

Per additional region: 1.2 TB (full copy)
5 regions total: 6 TB
```

**Query capacity**:
A single warehouse instance handles 50 concurrent queries at sub-second latency. Each region needs its own compute capacity. Budget for peak concurrency:

```
Region      Peak concurrency    Warehouse compute    Cache size
us-east-1   200 queries/s       8XL (128 vCPU)      50 GB
eu-west-1   150 queries/s       4XL (64 vCPU)       30 GB
ap-southeast-1 100 queries/s    2XL (32 vCPU)       20 GB
sa-east-1   50 queries/s        XL (16 vCPU)        10 GB
```

**Network egress costs**:
Replicating 800 GB of denormalized data across 4 additional regions: 3.2 TB of cross-region data transfer. At typical cloud pricing ($0.02-$0.12/GB), this costs $64-$384 per full refresh.

## Deployment

### Blue-Green Deployment for Schema Changes

Schema changes on denormalized tables (adding a new column, changing a dimension attribute) require careful rollouts.

**Blue-green deployment process**:

1. **Prepare green warehouse**: Create a parallel set of denormalized tables with the new schema in the warehouse.

```sql
-- Blue (current): order_reports has 12 columns
-- Green (new): order_reports_v2 has 13 columns (added loyalty_tier)

CREATE TABLE order_reports_v2 AS
SELECT
  o.*,
  c.loyalty_tier  -- new column
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;
```

2. **Dual-write**: Run the ETL pipeline to both tables for a transition period.

```python
# Dual-write during transition
def etl_job():
    rows = extract_from_source()
    load_to_warehouse("order_reports", rows)    # Blue
    load_to_warehouse("order_reports_v2", rows)  # Green
```

3. **Validate**: Run reconciliation jobs against both tables. Verify query latency, correctness, and storage cost.

```sql
SELECT COUNT(*) FROM order_reports;
SELECT COUNT(*) FROM order_reports_v2;
-- Should be identical
```

4. **Switch queries**: Update the dashboard queries to use `order_reports_v2` instead of `order_reports`.

```sql
-- Old dashboard query
SELECT * FROM order_reports WHERE order_date >= '2025-01-01';

-- New dashboard query
SELECT * FROM order_reports_v2 WHERE order_date >= '2025-01-01';
```

5. **Drop blue**: After the validation period (typically 1-7 days), drop the old table and rename the new one.

```sql
DROP TABLE order_reports;
ALTER TABLE order_reports_v2 RENAME TO order_reports;
```

### Backfill Scripts for New Denormalized Columns

When adding a new column to an existing denormalized table, the backfill script populates historical rows.

```python
def backfill_loyalty_tier():
    batch_size = 10000
    last_id = 0

    while True:
        rows = warehouse.query(f"""
            SELECT r.order_id, c.loyalty_tier
            FROM order_reports r
            JOIN customers c ON r.customer_id = c.customer_id
            WHERE r.loyalty_tier IS NULL
              AND r.order_id > {last_id}
            ORDER BY r.order_id
            LIMIT {batch_size}
        """)

        if not rows:
            break

        for row in rows:
            warehouse.execute("""
                UPDATE order_reports
                SET loyalty_tier = %s
                WHERE order_id = %s
            """, [row.loyalty_tier, row.order_id])

        last_id = rows[-1].order_id
        print(f"Backfilled up to order_id {last_id}")
```

**Backfill strategies**:
- **Full backfill**: Rewrite the entire table. Accurate but takes hours.
- **Chunked backfill**: Process in batches of 10,000-100,000 rows as shown above. Incremental and can run alongside normal operations.
- **Lazy backfill**: Only populate the new column when the row is next read or refreshed. Uses a null sentinel value.
