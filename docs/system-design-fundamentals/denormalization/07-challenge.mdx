# Denormalization Challenges

⏱️ 45 minutes · Difficulty: Advanced

## What You Will Learn

By completing these challenges you will be able to:

- Convert a normalized schema to a denormalized schema and justify each decision
- Identify update anomalies in a denormalized schema and design a reconciliation strategy
- Design a denormalized reporting schema from business requirements
- Build a CQRS read model for a real-world application
- Design a hybrid normalized+denormalized architecture for a fintech system

## How to Approach These Challenges

Each challenge has a scenario, a set of requirements, and deliverables. Work through them in order — Tier 1 builds foundation, Tier 2 applies to design, Tier 3 combines everything.

Write your answers as SQL schema definitions, architecture diagrams, and trade-off analysis. Use the concepts from the previous pages to justify your decisions.

## Tier 1 — Foundation

### Challenge 1: Normalized to Denormalized Conversion

**Scenario**: You are given a normalized e-commerce schema. Convert it to a denormalized schema optimized for a dashboard that shows monthly sales by product category and customer region.

**Normalized schema**:

```sql
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(200),
    email VARCHAR(200),
    signup_date DATE,
    region_id INT REFERENCES regions(region_id)
);

CREATE TABLE regions (
    region_id INT PRIMARY KEY,
    region_name VARCHAR(100),
    country VARCHAR(100)
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id),
    order_date TIMESTAMP,
    total DECIMAL(10,2)
);

CREATE TABLE order_items (
    order_item_id INT PRIMARY KEY,
    order_id INT REFERENCES orders(order_id),
    product_id INT REFERENCES products(product_id),
    quantity INT,
    unit_price DECIMAL(10,2)
);

CREATE TABLE products (
    product_id INT PRIMARY KEY,
    name VARCHAR(200),
    category VARCHAR(100),
    sku VARCHAR(50)
);
```

**Requirements**:
- The dashboard runs one query: monthly revenue and order count by product category and customer region.
- The dashboard must return results in under 1 second.
- The data is updated nightly — no real-time requirement.
- The dataset has 50 million orders with 2.5 line items each.

**Deliverables**:
1. Design a denormalized star schema (fact table + dimension tables).
2. Write the CREATE TABLE statements for the star schema.
3. Write the dashboard query against your star schema.
4. Estimate the storage difference between the normalized and denormalized schemas.
5. Explain the refresh strategy and its trade-offs.

### Challenge 3: Update Anomaly Identification

**Scenario**: The denormalized table below contains customer and order data together. Identify all update anomalies that can occur.

**Dashboard queries**:

**Requirements**:

### Challenge 5: CQRS Read Model Design

**Scenario**: A social media platform needs to show a user's feed. The normalized source has:

```sql
CREATE TABLE users (
    user_id INT PRIMARY KEY,
    username VARCHAR(50),
    display_name VARCHAR(200),
    avatar_url VARCHAR(500),
    bio TEXT
);

CREATE TABLE posts (
    post_id INT PRIMARY KEY,
    author_id INT REFERENCES users(user_id),
    content TEXT,
    created_at TIMESTAMP
);

CREATE TABLE likes (
    like_id INT PRIMARY KEY,
    post_id INT REFERENCES posts(post_id),
    user_id INT REFERENCES users(user_id),
    created_at TIMESTAMP
);

CREATE TABLE comments (
    comment_id INT PRIMARY KEY,
    post_id INT REFERENCES posts(post_id),
    author_id INT REFERENCES users(user_id),
    content TEXT,
    created_at TIMESTAMP
);

CREATE TABLE follows (
    follow_id INT PRIMARY KEY,
    follower_id INT REFERENCES users(user_id),
    followee_id INT REFERENCES users(user_id),
    created_at TIMESTAMP
);
```

**The feed query** (current, too slow):
```sql
SELECT
    p.post_id,
    p.content,
    p.created_at,
    u.display_name AS author_name,
    u.avatar_url AS author_avatar,
    COUNT(DISTINCT l.user_id) AS like_count,
    COUNT(DISTINCT c.comment_id) AS comment_count,
    (SELECT COUNT(*) FROM likes WHERE post_id = p.post_id AND user_id = ?) AS user_liked
FROM posts p
JOIN users u ON p.author_id = u.user_id
LEFT JOIN likes l ON p.post_id = l.post_id
LEFT JOIN comments c ON p.post_id = c.post_id
WHERE p.author_id IN (
    SELECT followee_id FROM follows WHERE follower_id = ?
)
GROUP BY p.post_id, p.content, p.created_at, u.display_name, u.avatar_url
ORDER BY p.created_at DESC
LIMIT 50;
```

**Requirements**:
1. Design a denormalized read model (a single table or document) that serves the feed query without JOINs.
2. Explain how the read model is populated when a new post is created.
3. Explain how the read model is updated when a user likes a post.
4. Explain how the read model handles a user changing their display name or avatar.
5. Estimate the storage cost of the denormalized read model for 10 million users, each following 100 accounts, with 500 million posts.

### Challenge 6: Real-Time Dashboard Data Model

**Scenario**: A ride-sharing company needs a real-time dashboard showing:

- Current ride requests per city (last 5 minutes, refreshed every 10 seconds)
- Average driver wait time by neighborhood (last 15 minutes)
- Surge multiplier by zone (updated in real-time when supply/demand ratio changes)
- Completed rides in the last hour, aggregated by vehicle type

**Data sources**:
- Kafka stream: `ride_requests` (rider_id, lat, lon, timestamp, city, zone)
- Kafka stream: `driver_locations` (driver_id, lat, lon, timestamp, city, zone, status, vehicle_type)
- Kafka stream: `ride_completions` (ride_id, rider_id, driver_id, start_time, end_time, fare, city, zone)
- Lookup table: `city_zones` (city, zone, population, area_km2)

**Requirements**:
1. Design a denormalized data model optimized for the dashboard queries.
2. The dashboard refreshes every 10 seconds — each query must complete in under 500ms.
3. Data volume: 10,000 ride requests per minute, 8,000 active drivers, 5,000 ride completions per minute.
4. Design the ingestion pipeline that denormalizes the streaming data.
5. Explain how you handle the staleness budget for the dashboard.

## Tier 3 — Expert

### Challenge 7: Hybrid Normalized + Denormalized Architecture for Fintech

**Scenario**: A fintech company processes payment transactions. The system must satisfy two conflicting requirements:

**Requirement A — Transaction processing (OLTP)**:
- Every transaction must be recorded with strict consistency.
- No stale data allowed. Balances must be accurate to the penny.
- The system processes 1,000 transactions per second during peak hours.
- Schema must prevent double-spending and account overdrafts.

**Requirement B — Reporting and analytics (OLAP)**:
- Daily reports on transaction volume, value, and merchant performance.
- Monthly compliance reports showing all transactions with customer and merchant details.
- Real-time fraud detection dashboard that queries the last 5 minutes of transactions.
- Analysts need to run ad-hoc queries without impacting transaction processing.

**Current normalized schema**:

```sql
CREATE TABLE accounts (
    account_id INT PRIMARY KEY,
    customer_id INT,
    account_type VARCHAR(20),
    balance DECIMAL(15,2),
    created_at TIMESTAMP,
    status VARCHAR(20)
);

CREATE TABLE transactions (
    transaction_id INT PRIMARY KEY,
    from_account_id INT REFERENCES accounts(account_id),
    to_account_id INT REFERENCES accounts(account_id),
    amount DECIMAL(15,2),
    currency VARCHAR(3),
    transaction_type VARCHAR(30),
    status VARCHAR(20),
    created_at TIMESTAMP,
    gateway_response TEXT
);

CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(200),
    email VARCHAR(200),
    kyc_status VARCHAR(20),
    risk_score DECIMAL(3,2),
    created_at TIMESTAMP
);

CREATE TABLE merchants (
    merchant_id INT PRIMARY KEY,
    business_name VARCHAR(200),
    category VARCHAR(100),
    country VARCHAR(100),
    fee_percentage DECIMAL(4,2),
    status VARCHAR(20)
);
```

**Requirements**:
1. Design a hybrid architecture that keeps the normalized write path for transaction processing and adds a denormalized read path for reporting.
2. The denormalized reporting system must serve daily reports within 30 seconds and the fraud detection dashboard within 2 seconds.
3. The reporting system must not affect transaction processing latency, even during ETL operations.
4. Design the reconciliation strategy that ensures the reporting data matches the transaction data.
5. Explain how you handle schema evolution in the reporting system when a new transaction type is added.
6. Estimate the resource requirements (storage, compute, network) for the reporting system, assuming 30 million transactions per day, 10 million customers, and 500,000 merchants.

**Deliverables**:
1. Architecture diagram showing the write path and read path, including all components (databases, queues, ETL, caches).
2. Denormalized schema for the reporting system (fact and dimension tables).
3. ETL pipeline design (batch or streaming, refresh frequency, logic for each table).
4. Reconciliation query design (SQL that validates the reporting data against the source).
5. Schema evolution plan (how to add a new transaction type without downtime).
6. Monitoring and alerting strategy for data freshness and consistency.

## Submission Guidelines

For each challenge, provide:

1. **SQL schema definitions** — complete CREATE TABLE statements with appropriate data types, keys, and constraints
2. **Architecture description** — 3-5 paragraphs explaining your design decisions
3. **Trade-off analysis** — what you sacrificed and why
4. **Quantitative estimates** — storage, query latency, refresh frequency, staleness window

**Evaluation criteria**:
- **Correctness**: Does the design satisfy the requirements?
- **Depth**: Are the trade-offs explicitly identified and justified?
- **Practicality**: Would this design work in production with real data volumes?
- **Completeness**: Are all edge cases addressed (schema changes, failures, drift)?

## Answer Key

Review your answers against these key points after completing the challenges.

**Challenge 1 key points**:
- Star schema: fact_order_items (date_key, product_key, customer_key, region_key, quantity, unit_price, total) + dim_date, dim_product, dim_customer, dim_region
- Dim_product includes category as a flat column. Dim_region includes region_name and country.
- Dashboard query: SELECT dp.category, dr.region_name, SUM(foi.total) FROM fact_order_items foi JOIN dim_product dp ON foi.product_key = dp.product_key JOIN dim_region dr ON foi.region_key = dr.region_key JOIN dim_date dd ON foi.date_key = dd.date_key WHERE dd.month = ? GROUP BY dp.category, dr.region_name
- Storage: normalized ~200 GB, star schema ~280 GB, compressed ~90 GB
- Refresh: nightly full rebuild with incremental option for same-day data

**Challenge 2 key points**:
- Problems: 80 columns including TEXT fields (product_description, review_body), no partitioning, no column selection, update anomalies on customer fields, write amplification, schema migration cost, query performance for wide rows
- Better approach: star schema with fact table (10-15 columns) and dimension tables (10-20 columns each). Keep product_description and review_body in source tables.
- Storage savings: 80 columns x 400 bytes = 32 KB per row → 15 columns x 200 bytes = 3 KB per row. 10x reduction.

**Challenge 3 key points**:
- Anomalies: customer name change (update 3 rows), customer tier change, email change, address change
- Reconciliation: SELECT customer_id, COUNT(DISTINCT customer_name) FROM customer_orders GROUP BY customer_id HAVING COUNT(DISTINCT customer_name) > 1
- Fix: normalize customer data into a separate table, keep order data with customer_id foreign key

**Challenge 4 key points**:
- Fact tables: fact_readmissions, fact_procedures, fact_medication_adherence, fact_lab_results
- Dimensions: dim_patient (SCD Type 2 for diagnosis changes), dim_physician, dim_date, dim_department
- Partition by month, sort by diagnosis_id, physician_id

**Challenge 5 key points**:
- Denormalized feed table: feed_entries (post_id, author_id, author_name, author_avatar, content, created_at, like_count, comment_count, user_liked_bitmap)
- Populate on post creation: query the author's display_name and avatar_url, insert into feed_entries
- Handle name changes: update all feed_entries for that author (batch job, or accept staleness)
- Storage: 500M posts x 500 bytes = 250 GB, compressed ~50 GB

**Challenge 6 key points**:
- In-memory dashboard: use Redis or Pinot for sub-second queries
- Denormalize at ingestion time: JOIN ride_requests with city_zones lookup in the stream processor
- Precompute aggregates: maintain running 5-minute and 15-minute windows in the dashboard cache
- Staleness: accept 10 seconds of staleness, matching the refresh interval

**Challenge 7 key points**:
- Write path: normalized PostgreSQL for transaction processing (ACID, strict consistency)
- Read path: CDC to Kafka → ETL (dbt/Flink) → Columnar warehouse (Snowflake/Redshift)
- Separate databases: the warehouse query load does not affect the OLTP database
- Reconciliation: hourly batch job comparing transaction counts and totals between source and warehouse
- Schema evolution: add new columns to warehouse with NULL default, backfill in batches
- Storage: 30M transactions/day x 500 bytes = 15 GB/day fresh data, 5.5 TB/year compressed ~1.8 TB
