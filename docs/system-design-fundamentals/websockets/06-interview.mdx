## How This Topic Appears

WebSocket questions appear primarily in system design interviews and debugging/operational scenarios. While a WebSocket-specific round is rare, WebSocket infrastructure frequently shows up as a sub-component of larger designs: chat, live collaboration, real-time gaming, or streaming analytics.

### System Design Questions (WebSocket as Core Component)

1. "Design WhatsApp or Messenger. How would you handle real-time message delivery?"
2. "Design a real-time collaborative document editor like Google Docs."
3. "Design a live voting system for a TV show with millions of concurrent viewers."
4. "Design a real-time location tracking system for 10 million delivery drivers."
5. "Design a live sports score broadcast system that pushes updates to 100 million fans."

### System Design Questions (WebSocket as Sub-Component)

6. "Design Twitter. How would you implement the real-time timeline?"
7. "Design a stock trading platform. How would you stream price updates?"
8. "Design a multiplayer game server infrastructure."
9. "Design a notification system that delivers push, email, SMS, and in-app."

### Debugging and Architecture Questions

10. "Your WebSocket connections drop after exactly 60 seconds. What do you check?"
11. "Your WebSocket server runs out of memory after 50,000 connections. Why?"
12. "How would you deploy a new version of your WebSocket server without dropping connections?"
13. "Your users report that messages arrive out of order after reconnecting. What went wrong?"
14. "How would you migrate from HTTP polling to WebSocket without downtime?"

| Symptom | Hidden Test | Red Flag Answer | Green Flag Answer |
|---------|-------------|-----------------|-------------------|

## Level-by-Level Expectations

### L3/L4 (Entry-level / Mid-level)

**Baseline requirement:** Explain how WebSocket works at a high level. Recite the handshake. Describe the difference from HTTP polling.

**Sample answer for "Describe WebSocket handshake":** "The client sends an HTTP GET request with an Upgrade: websocket header and a Sec-WebSocket-Key. The server responds with HTTP 101 Switching Protocols and a Sec-WebSocket-Accept header, which is a SHA-1 hash of the key concatenated with a magic GUID. After the 101 response, the connection switches from HTTP protocol to WebSocket framing — both sides can send data at any time with 2-14 bytes of overhead per message."

**Sample answer for "WebSocket vs HTTP polling":** "HTTP polling sends requests at fixed intervals, with full HTTP headers (500-800 bytes) on every request, regardless of whether data changed. WebSocket establishes a persistent connection with a one-time handshake overhead of about 1 KB, then each message costs only 2-14 bytes of framing overhead. WebSocket latency equals the network round trip, while polling latency equals the poll interval plus one round trip."

### L5 (Senior)

**Baseline requirement:** Design a WebSocket system for 1 million concurrent users. Identify bottlenecks before they happen. Handle failures gracefully.

**Sample answer for "Design WebSocket for 1M concurrent connections":** "The bottleneck is not memory — 1M connections at 20 KB each is 20 GB, which fits on two modern servers. The bottleneck is connection management and broadcast. I would use a pool of WebSocket servers behind a TCP load balancer with sticky sessions (NLB with proxy protocol). Each server handles 50,000-100,000 connections. Cross-server communication goes through Redis Pub/Sub or Kafka. For broadcast, I would batch writes and yield between batches to avoid blocking the event loop. Heartbeats at 30 seconds with a 10-second timeout detect dead connections. Reconnection uses exponential backoff with jitter: 1s, 2s, 4s, up to 30s maximum. For zero-downtime deploys, I would send a 1012 close code signaling service restart, then clients reconnect to the new servers."

**Sample answer for "What happens when a broadcast blocks for 2 seconds":** "At 100,000 connections, a synchronous broadcast of one message takes approximately 500ms to 2 seconds depending on message size and network. During this time, no heartbeats are processed, leading to false timeouts and cascading disconnections. The fix is to batch the broadcast: iterate 1,000 connections, call writev() for the batch, yield to the event loop, then continue. This limits any single broadcast's blocking time to under 20ms. For very large fan-outs (1M connections), distribute the broadcast across servers — each server only broadcasts to its own connections."

### L6 (Staff)

**Baseline requirement:** Design for multi-region, 10 million+ concurrent users. Handle split-brain, rebalancing, and cross-region consistency. Challenge the requirements — know when not to use WebSocket.

**Sample answer for "Design real-time chat for 100 million users":** "A single server pool cannot handle 100 million connections. I would use a multi-region architecture with regional WebSocket pools. Each region handles 10-20 million connections. Cross-region message delivery uses a global Kafka cluster with geo-replication. Messages from User A in US-East to User B in EU-West take the path: Server A writes to local Kafka, Kafka MirrorMaker replicates to EU-West, EU-West server delivers to User B. Target latency: same-region under 50ms, cross-region under 500ms. The key trade-off is consistency versus latency: for cross-region messages, we accept eventual consistency within 2 seconds. For same-region messages within the same server, we guarantee strong ordering. I would also challenge whether all 100 million users need persistent WebSocket connections simultaneously. For less active users, we could downgrade to Server-Sent Events or HTTP/2 push after a period of inactivity, freeing WebSocket slots for active users."

**Sample answer for "How would you handle a thundering herd after a regional failover":** "When US-East fails and 20 million clients reconnect to EU-West simultaneously, the connection rate jumps from under 10,000/second to over 200,000/second. Three mitigations: first, DNS routing with health checks gradually shifts traffic over 60 seconds rather than all at once. Second, the load balancer rate-limits new connections to 50,000/second, and rejected clients back off with jitter. Third, we pre-provision EU-West with 20 percent extra capacity to absorb the initial spike. The load balancer's max_connection_rate setting is the primary control — it prevents the servers from being overwhelmed by TLS handshakes, which are CPU-intensive."

### L7/L8 (Principal / Distinguished)

**Baseline requirement:** Design a new protocol or infrastructure component. Analyze the architectural landscape and recommend build vs. buy. Influence the industry.

**Sample answer for "Should we build a custom WebSocket gateway or use a managed service":** "The build-vs-buy decision for WebSocket infrastructure depends on three factors: scale, differentiation, and operational capacity. Under 100,000 concurrent connections and with a team of under 10 engineers, buy: AWS API Gateway or Azure SignalR. Between 100K and 1M, consider a hybrid: use managed load balancers (ALB/NLB) but run your own WebSocket servers on Kubernetes for cost efficiency — AWS API Gateway costs approximately 0.80 USD per 1 million connections per month, so at 1M connections that is 19,200 USD per year just for connection charges before message costs, while self-hosted on EC2 costs roughly 6,000 USD per year. Above 1M connections, build a custom gateway if WebSocket is core to your product and you have a team to maintain it (Discord, Netflix both built custom). For most companies, the operational cost of running a custom WebSocket gateway (on-call, debugging, feature development) outweighs the infrastructure cost savings."

## The FAANG Framework for Answering

Use this 6-step framework for any WebSocket system design question:

### Step 1: Clarify Requirements (2 minutes)

Establish the scope before designing.

- "How many concurrent users?"
- "What is the message delivery latency requirement?"
- "What is the average message size? Text or binary?"
- "Do we need message persistence?"
- "What are the consistency requirements — is eventual consistency acceptable?"
- "What is the read-to-write ratio?"
- "Are there any platform constraints (browser support, mobile battery concerns)?"

### Step 2: Define the Communication Model (1 minute)

State your protocol choice and justify it.

- "I will use WebSocket with WSS for the real-time channel. The bidirectional push requirement rules out Server-Sent Events. The sub-100ms latency rules out HTTP polling. I will use JSON text frames for simplicity initially, with a migration path to Protobuf binary frames if throughput passes 10,000 messages/second."

### Step 3: Design the Connection Lifecycle (3 minutes)

Walk through connection establishment, data flow, and disconnection.

- "The connection lifecycle has five phases: handshake, active messaging, heartbeat, reconnect, and close. During handshake, the load balancer establishes a sticky session so subsequent messages reach the same server. During active messaging, the server reads frames through a state machine parser, processes extensions (deflate), and dispatches to the application handler. Heartbeats run at 30-second intervals. On unexpected disconnect, clients implement exponential backoff reconnect. On graceful shutdown, the server sends 1012 and drains connections."

### Step 4: Design the Scaling Model (3 minutes)

Show how the system handles load.

- "I will use N servers behind an NLB with proxy protocol for sticky sessions. Each server handles up to 100,000 connections. Cross-server communication goes through Redis Pub/Sub for ephemeral messages and Kafka for durable messages. Broadcasting to all connected clients fans out via Redis Pub/Sub to all servers, and each server delivers to its local connections in batches of 1,000 with event loop yields. I will add servers when average connection count per server exceeds 80,000."

### Step 5: Address Failure Modes (3 minutes)

Show you understand what breaks and how to fix it.

- "Five failure modes I must handle: server crash, network partition, slow consumer, broadcast amplification, and thundering herd. For server crash, clients reconnect with exponential backoff and recover state from Redis session store. For network partition, application-layer heartbeats detect half-open connections within 40 seconds. For slow consumers, per-connection write buffer caps at 2 MB, and connections exceeding the cap are closed. For broadcast amplification, batching with yields limits event loop blocking to under 20ms. For thundering herd, load balancer connection rate limiting spreads reconnection load over 30-60 seconds."

### Step 6: Discuss Trade-offs (2 minutes)

Show you evaluated alternatives and made deliberate choices.

- "The key trade-off is using WebSocket versus Server-Sent Events. SSE would be simpler — no subprotocol negotiation, automatic reconnection via EventSource API, HTTP/2 multiplexing. But SSE is unidirectional (server-to-client only) and limited to 6 connections per browser in HTTP/1.1. Since our application requires client-to-server messages for typing indicators and reactions, WebSocket is the correct choice despite the added complexity of managing persistent connections."

## Phrase Bank

**For explaining WebSocket basics:**
- "WebSocket upgrades an HTTP connection to a persistent full-duplex channel using the HTTP Upgrade mechanism. This means it works through standard HTTP infrastructure — port 80/443, firewalls, proxies — unlike raw TCP sockets."

**For comparing to polling:**
- "HTTP polling wastes resources proportional to the poll frequency multiplied by the number of clients. With 100,000 clients polling every 5 seconds, that is 20,000 requests per second even when no data has changed. WebSocket delivers zero-overhead idle — the connection stays open but no data is exchanged until an event occurs."

**For discussing sticky sessions:**
- "Sticky sessions couple a client to a specific server, which complicates rolling deployments and auto-scaling. I accept this coupling because the alternative — maintaining a global connection table that every server can read — adds latency and complexity that negates the benefits of statelessness. The compromise is to use a 30-second drain window during deployments."

**For WebSocket vs SSE decision:**
- "If the data flow is unidirectional — server pushes updates to the client — I would choose SSE over WebSocket because SSE runs over HTTP/2, has built-in reconnection in the EventSource API, and is simpler to implement on both client and server. SSE fails at bidirectional communication: I cannot send data from client to server without a separate HTTP request."

**For scaling discussion:**
- "The limiting factor for WebSocket servers is not bytes per second but the number of concurrent open file descriptors and the memory per connection. Each connection costs 10-20 KB of kernel buffer space plus application-level state. I budget 64 KB per connection for safe scaling calculations."

**For backpressure explanation:**
- "Backpressure in WebSocket means: when a consumer cannot keep up with the producer, the send buffer fills, the kernel's TCP buffer fills, and eventually the producer blocks on write. I handle backpressure by capping per-connection write buffers at 2 MB and disconnecting clients that exceed the cap with a 1009 close code."

**For reconnection strategy:**
- "Exponential backoff without jitter creates synchronized reconnection waves. Jitter spreads the wave into a trickle. I use `base * 2^attempt + random(0, 1000)` capped at 30 seconds maximum."

**For message ordering after reconnect:**
- "TCP guarantees ordering within a connection. A reconnect creates a new connection. To restore ordering across reconnects, I assign monotonically increasing sequence numbers to messages. The client reports its last received sequence on reconnect, and the server replays from that offset."

**For broadcast architecture:**
- "I use a two-phase broadcast. Phase 1: the publishing server writes the message to a durable log (Kafka) for exactly-once delivery. Phase 2: the server publishes the message ID to a fast fan-out channel (Redis Pub/Sub) for low-latency delivery. Servers that missed the Redis notification can catch up from Kafka on the next heartbeat."

**For cross-region delivery:**
- "Same-region messages take the fast path: WebSocket server to local Redis to local clients. Cross-region messages take the slow path: local Kafka, geo-replication to remote Kafka, remote Redis, remote clients. I design the system so that most messages are same-region by routing users to their closest data center."

## Red Flags

**Red Flag 1: "WebSocket guarantees message ordering."**

WebSocket inherits TCP ordering within a single connection. But across reconnections, or across different connections in a multi-server system, ordering is not guaranteed. If you need global ordering, sequence numbers at the application layer are required.

**Red Flag 2: "I will use a single server with async I/O."**

This works up to approximately 100,000 connections. Above that, you need horizontal scaling, sticky sessions, and a pub/sub layer. An interviewer asking about scale expects to hear about cross-server communication, not single-server optimization.

**Red Flag 3: "Ping/Pong is optional."**

Without application-layer heartbeats, half-open connections persist for minutes (TCP keepalive default is 2 hours). Any production WebSocket system must have heartbeats. Omitting them signals inexperience with production networking.

**Red Flag 4: "I will broadcast by iterating all connections and calling send()."**

At scale, this blocks the event loop. The interviewer expects you to mention batching, writev(), concurrency limits, and backpressure. Simply describing the naive for-loop pattern indicates you have not operated WebSocket at scale.

**Red Flag 5: "I will store WebSocket session state only in memory."**

This fails immediately on server restart, rolling deployment, or adding a second server. State must be externalized (Redis, DynamoDB, etc.) with a TTL-based cleanup strategy.

**Red Flag 6: "WebSocket is always better than HTTP polling."**

WebSocket is the right tool for real-time bidirectional communication. For infrequent updates (over 10-second intervals), HTTP polling is simpler and cheaper. Claiming WebSocket is universally superior signals that you do not evaluate trade-offs.

**Red Flag 7: "I will combine all client-server communication into one WebSocket connection."**

Mixing real-time events, request-response API calls, and file uploads over a single WebSocket channel creates coupling and debugging difficulty. Each communication pattern deserves its own channel. Using WebSocket for everything is an anti-pattern.

**Red Flag 8: "The load balancer handles all the complexity."**

Load balancers do not solve cross-server communication, state persistence, reconnection ordering, or broadcast fan-out. They solve one problem: distributing connections. Claiming the load balancer handles everything tells the interviewer you have not built a distributed WebSocket system.

## Practice Drills

### Drill 1: The One-Minute Whiteboard (2 minutes)

**Prompt:** "Explain WebSocket to a non-technical product manager who wants to understand why our chat app needs WebSocket instead of HTTP requests."

**Time:** 2 minutes.

**Check your answer against:** Did you use an analogy (telephone vs postal service)? Did you explain the cost difference (no repeated envelopes)? Did you mention the trade-off (persistent connection costs more server resources)?

### Drill 2: Architecture Sketch (5 minutes)

**Prompt:** "Sketch the architecture for a real-time notification system that pushes updates to 10 million mobile users. WebSocket is the delivery channel."

**Time:** 5 minutes.

**Must include:** Client connection flow, load balancer and sticky sessions, server pool with connection count per server, Redis/Kafka for cross-server messaging, state persistence, reconnection strategy, health monitoring.

### Drill 3: Debug the Failure (5 minutes)

**Prompt:** "Our WebSocket server handles 30,000 connections for 6 hours, then starts throwing 'Out of memory' errors. Connections are C-style WebSocket, no external state store. Diagnose the problem."

**Time:** 5 minutes.

**Possible answers:** (a) Connection leak — close handlers not firing for half-open connections. (b) Write buffer growth from slow consumers. (c) Fragment reassembly buffers holding incomplete large messages indefinitely. (d) Timer wheel memory leak — timers not being freed.

### Drill 4: Trade-Off Analysis (10 minutes)

**Prompt:** "Compare WebSocket and Server-Sent Events for a live sports score application. The app must push score changes to 500,000 concurrent viewers. Viewers do not send data back. What do you recommend and why?"

**Time:** 10 minutes.

**Rubric:** SSE is simpler for unidirectional data. HTTP/2 multiplexing solves the 6-connection limit. EventSource API handles reconnection automatically. SSE does not work with HTTP/1.1 proxies cleanly — some proxy between browser and server may buffer SSE responses. WebSocket is more battle-tested at extreme scale. Final recommendation: SSE for simplicity at 500K; if scale exceeds 10M viewers, consider WebSocket for finer control.

### Drill 5: Full Design (15 minutes)

**Prompt:** "Design a real-time collaborative whiteboard application that supports 100,000 concurrent users across 5,000 active whiteboards."

**Time:** 15 minutes.

**Must cover:** WebSocket connection lifecycle, multi-server architecture, broadcast to specific whiteboard participants (not all users), cursor position sharing (high-frequency, loss-tolerant), shape insertion (low-frequency, must-not-lose), undo/redo state per whiteboard, reconnection with state reconciliation.
