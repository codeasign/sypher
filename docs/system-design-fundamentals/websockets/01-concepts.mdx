## The Problem Story

You built a real-time dashboard showing stock prices. Every five seconds, your frontend sends an HTTP request to the server asking for the latest price. It works fine for ten users. Your server handles 200 requests per second without breaking a sweat.

Then you launch the feature to your full user base. Ten thousand traders open the dashboard simultaneously. Now your server is handling 60,000 HTTP requests per minute — most of them returning the same data, because prices haven't even changed between requests. Response times climb from 12 milliseconds to over 3 seconds. Mobile users drain their batteries because each request wakes the radio, sends headers, parses responses, and repeats every five seconds regardless of whether anything changed. Your infrastructure costs triple, and users still see stale data for up to five seconds.

You try reducing the poll interval to every 30 seconds. Now users complain the data is too stale. You try HTTP/2 server push, but the browser connection limits still bind you. You try long-polling — the server holds the request open until new data arrives. This cuts bandwidth, but now your server runs out of connection slots because every client holds a TCP connection open indefinitely, and reverse proxies time out idle connections after 60 seconds.

The core problem is architectural: HTTP was designed for a request-response pattern where the client initiates every interaction. Your application needs the server to push data the moment it changes. These are fundamentally mismatched communication models, and no amount of polling or connection reuse closes the gap.

## The Insight

A WebSocket is a persistent, bidirectional communication channel between a client and a server over a single TCP connection. Once established, either side can send data at any time without the overhead of HTTP request/response framing.

Think of it like a telephone call instead of a postal service. With HTTP (the postal service), you mail a letter (request), wait for a reply letter (response), and every letter requires a new envelope with addresses, stamps, and routing instructions. If the post office is slow, you mail the same letter again (retry). With a WebSocket (the telephone call), you dial once, the connection stays open, and both people can speak whenever they want. You hear the other person instantly because the line is always live. There is no envelope, no stamp, no repeated address lookup. But like a phone call, maintaining the line costs resources: a dedicated circuit stays open whether anyone is speaking or not.

## Core Vocabulary

**WebSocket** — A full-duplex communication protocol operating over a single TCP connection, standardized as RFC 6455. It upgrades from HTTP via a handshake and then switches to its own framing protocol. This matters because it eliminates the overhead of HTTP headers on every message, reducing per-message overhead from hundreds of bytes to as few as 2 bytes.

**Handshake** — The initial HTTP Upgrade request and 101 Switching Protocols response that transitions a connection from HTTP to WebSocket protocol. The client sends an Upgrade: websocket header with a Sec-WebSocket-Key. The server responds with Sec-WebSocket-Accept containing a hash of that key. This matters because it lets WebSocket connections traverse standard HTTP infrastructure (port 80/443, proxies, firewalls) without special configuration.

**Frame** — The basic unit of WebSocket communication. A frame contains an opcode (text, binary, close, ping, pong), a payload length, and the payload data. Frames can be fragmented across multiple frames. This matters because the framing protocol is lightweight — 2 to 14 bytes of overhead per message versus HTTP's 500-800 byte header overhead.

**Message** — A complete sequence of one or more frames, reassembled by the recipient. A message can be a single unfragmented frame or multiple frames chained together. This matters because application code works with messages, not frames — the protocol handles fragmentation transparently.

**Opcode** — A 4-bit field in each frame identifying the frame type: 0x1 (text), 0x2 (binary), 0x8 (close), 0x9 (ping), 0xA (pong). This matters because it lets the protocol distinguish data from control signals without application-level parsing.

**Ping/Pong** — Heartbeat frames used for connection health checking. Either side can send a Ping frame; the recipient must respond with a Pong frame containing the same payload. This matters because it enables detection of dead connections within seconds rather than waiting for TCP timeouts that can take minutes.

**Extension** — Protocol extensions negotiated during the handshake, such as permessage-deflate for compression. The client lists supported extensions in the handshake headers. This matters because extensions can dramatically reduce bandwidth — deflate compression reduces typical JSON payloads by 60-80 percent.

**Subprotocol** — An application-level protocol negotiated during the handshake, such as wamp, mqtt, or so Stomp. The client sends a Sec-WebSocket-Protocol header with supported subprotocols; the server picks one. This matters because subprotocols define message formats and routing rules above the raw WebSocket layer.

**Close Frame** — A control frame (opcode 0x8) that initiates connection termination. It includes an optional status code and reason string. Status code 1000 means normal closure; 1001 means endpoint going away; 1002 means protocol error. This matters because clean shutdown allows both sides to release resources and prevents half-open connections.

**Masking** — A security mechanism where client-to-server frames are XOR-masked with a 32-bit key. Server-to-client frames are never masked. This matters because masking prevents cache poisoning attacks where an attacker tricks a browser into sending data that an intermediary HTTP cache interprets as a new HTTP response.

## How It Actually Works — Step by Step

### Step 1: TCP Connection Establishment

Before any WebSocket handshake can occur, a TCP connection must exist. The client performs a three-way handshake with the server: SYN, SYN-ACK, ACK. This establishes a reliable, ordered byte stream between the two endpoints.

### Step 2: HTTP Upgrade Handshake

Once TCP is open, the client sends an HTTP Upgrade request. This is a standard HTTP GET request with special headers that ask the server to switch protocols.

```
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Sec-WebSocket-Extensions: permessage-deflate
Sec-WebSocket-Protocol: mqtt, wamp
Origin: https://example.com
```

The server validates the request and responds with status 101:

```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
Sec-WebSocket-Extensions: permessage-deflate
Sec-WebSocket-Protocol: mqtt
```

### Step 3: Protocol Switch

After the 101 response, the TCP connection shifts from HTTP to WebSocket framing. The server computes Sec-WebSocket-Accept by concatenating the client's Sec-WebSocket-Key with the magic GUID `258EAFA5-E914-47DA-95CA-C5AB0DC85B11`, then taking the SHA-1 hash and Base64 encoding it. The client verifies this value to confirm the server actually understands WebSocket.

### Step 5: Message Fragmentation

Messages larger than a single frame can be fragmented. The first frame has FIN=0, and the last frame has FIN=1. All fragments in a sequence share the same opcode type. This allows streaming large messages without buffering the entire payload.

### Step 6: Control Frames Interleave

Ping, Pong, and Close frames can appear between data frame fragments. The protocol processes control frames immediately, even in the middle of a fragmented message. This is how the protocol maintains responsiveness — a Ping doesn't wait for the current large data message to finish before getting a Pong.

### Step 7: Heartbeat and Keepalive

Either side sends Ping frames on a timer, typically every 30 to 60 seconds. The other side must respond with a Pong within a timeout window (commonly 10 seconds). If the Pong does not arrive, the connection is considered dead. This catches half-open connections where TCP still reports the connection as established but the remote side has crashed or the network path is broken.

### Step 8: Close Handshake

When either side wants to close, it sends a Close frame (opcode 0x8) with a status code. The other side responds with its own Close frame. After sending the response, each side waits for TCP FIN to fully tear down. Neither side should send data frames after sending or receiving a Close frame.

### Step 9: Extensions — permessage-deflate

The permessage-deflate extension compresses each message's payload using DEFLATE compression. The server and client negotiate compression parameters during the handshake. Compression reduces JSON payloads by 60 to 80 percent but adds CPU cost for compression and decompression. For small messages under 256 bytes, compression may actually increase size due to overhead.

### Step 10: Subprotocols — Application-Level Framing

Subprotocols define message formats above the WebSocket layer. For example, MQTT over WebSocket uses WebSocket as the transport and MQTT's publish-subscribe protocol for message routing. WAMP (Web Application Messaging Protocol) provides RPC and PubSub patterns. The subprotocol is negotiated once during the handshake, and all subsequent messages follow that protocol's rules.

**When to use:** You control both client and server. Messages are small (under 10 KB). You want maximum throughput with minimum overhead.

**When not to use:** You need message routing, pub/sub patterns, or interoperability with non-WebSocket clients. Raw WebSocket gives you none of these — you build everything yourself.

### Variant 2: WebSocket with permessage-deflate Extension

Standard WebSocket with message-level compression negotiated during the handshake.

**When to use:** Messages are large (over 1 KB). Bandwidth is expensive. You are serving mobile users on metered connections. Your payloads contain repetitive JSON keys.

**When not to use:** Messages are tiny (under 256 bytes). You are CPU-bound and cannot spare cycles for compression. Real-time audio/video where latency is critical.

## Mental Model Check

**Question 1:** Your WebSocket connection drops after exactly 120 seconds of inactivity. What is the most likely cause?

The server or an intermediate proxy has a 120-second idle timeout for TCP connections. WebSocket Ping/Pong frames must be sent before this timeout fires. Set up a heartbeat at 30 to 45 seconds to stay below the timeout threshold.

**Question 2:** You are sending 1 million small messages per second through a single WebSocket connection. The receiver cannot keep up and the send buffer grows. What happens?

The TCP send buffer fills up (typically 64 KB to 256 KB depending on OS settings). When the buffer is full, the send() call blocks (for blocking sockets) or returns EAGAIN/EWOULDBLOCK (for non-blocking sockets). Backpressure propagates: the sender must slow down. If the sender ignores backpressure, memory usage grows unbounded until the process is killed by OOM.

**Question 3:** You need to support 100,000 concurrent WebSocket connections on a single server. Is this feasible?

Yes, with proper tuning. Each WebSocket connection consumes approximately 10-20 KB of memory for TCP buffer and WebSocket state. For 100,000 connections, this is 1-2 GB of RAM. The OS must support enough file descriptors (ulimit -n), and the event loop must use non-blocking I/O (epoll on Linux, IOCP on Windows). A single modern server can handle 500,000 to 1 million connections with 8-16 GB RAM dedicated to connection state.

**Question 4:** Your WebSocket server sends large binary messages (50 MB each). Clients report intermittent disconnections. What is happening?

Many load balancers, proxies, and browsers have per-message size limits. Nginx defaults to 1 MB proxy_buffer_size. AWS ELB limits WebSocket messages to 1 MB. Browsers limit messages to 16 MB for Blob and up to 256 MB for ArrayBuffer. Messages exceeding these limits are dropped, causing protocol violations and disconnection. Solution: chunk large payloads at the application level.

**Question 5:** Your WebSocket server crashes. What happens to the client connections?

The OS closes each TCP connection with a FIN packet as the process exits. Clients receive a Close frame (status 1006 — abnormal closure). Clients should implement reconnection logic with exponential backoff: immediately retry, then wait 1 second, 2 seconds, 4 seconds, up to a maximum of 30 seconds. Add jitter to prevent thundering herd on server restart.
