## Company Case Studies

### Weights and Biases (WandB)

WandB is the dominant platform for ML experiment tracking. Their AI observability product, Weave, extends experiment tracking to LLM applications. The core insight: treat every LLM call as an experiment and log everything.

**How they do it.** Weave wraps any LLM call with a decorator. The decorator captures the input, output, model metadata, and latency. It also captures the call stack so you can see which function called the LLM, what parameters were passed, and how the output was used downstream.

```python
import weave

@weave.op()
def generate_response(user_message: str, context: list[str]) -> str:
    prompt = build_prompt(user_message, context)
    response = openai.chat.completions.create(
        model="gpt-4o",
        messages=prompt
    )
    return response.choices[0].message.content
```

**Key metrics tracked.** WandB logs 47 metrics per LLM call, including response time, token count, cost, model version, temperature, top_p, frequency penalty, presence penalty, and the full response content.

**Scale.** WandB processes over 1 billion LLM calls per month across their customer base. Their largest customer generates 50 million traces per day.

**Lesson learned.** The decorator pattern is simple and non-invasive. Teams can instrument their entire codebase by adding a single line to each function. The challenge is managing the sheer volume of data: WandB uses adaptive sampling that reduces the rate during peak hours and increases it during off-peak hours.

### LangSmith

LangSmith is the observability platform from LangChain, the most popular LLM framework. It provides end-to-end tracing for LangChain applications, from the initial user input through chains, agents, retrievers, and LLM calls.

**How they do it.** LangSmith hooks into LangChain's callback system. When a LangChain application runs, every step (chain start, LLM call, tool execution, chain end) generates a callback event. LangSmith captures these events and reconstructs the full execution trace.

```python
from langsmith import Client
from langchain.callbacks import LangSmithCallbackHandler

client = Client()
handler = LangSmithCallbackHandler(
    project_name="my-ai-agent",
    client=client
)

# The handler automatically captures all LangChain steps
response = chain.invoke(
    {"input": "What is the return policy?"},
    config={"callbacks": [handler]}
)
```

**Key features.** LangSmith provides a playground for comparing model outputs side-by-side. You can run the same prompt against GPT-4o, Claude 3.5, and Gemini 1.5 simultaneously, view the results, add human feedback, and export the comparison to a dataset. This A/B testing capability is built directly into the observability platform.

**Scale.** LangSmith handles 100 million traces per day across their customer base. The platform uses a combination of head-based sampling (for normal traffic) and tail-based sampling (for error traces) to manage storage costs.

**Lesson learned.** Deep integration with the framework (LangChain) provides rich traces automatically. The trade-off is vendor lock-in: teams that use LangSmith are incentivized to use LangChain, even when a simpler framework would suffice.

### Arize AI

Arize AI focuses on ML observability with a strong emphasis on drift detection and model performance monitoring. Their platform monitors both traditional ML models and LLMs.

**How they do it.** Arize uses a concept called "model performance monitoring." They ingest inference data (model inputs, outputs, predictions) and compare them against baselines. The comparison detects drift in input distributions, output distributions, and prediction accuracy.

```python
import arize

# Log a single LLM inference
arize.log_inference(
    model_id="gpt-4o-production",
    prediction_id="pred_12345",
    features={"prompt": "What is the return policy?", "temperature": 0.3},
    prediction="Our return policy allows returns within 30 days...",
    actual="Our return policy allows returns within 45 days...",  # ground truth
    tags={"environment": "production", "user_segment": "premium"}
)
```

**Key features.** Arize specializes in embedding drift detection. They embed the prompt text and the response text, then visualize the embeddings in a 2D space. When the embeddings drift away from the baseline cluster, it signals a change in the model's behavior. This technique caught a production issue where a prompt template change accidentally truncated the system message, causing the model to shift from formal to informal language.

**Scale.** Arize handles 5 billion inferences per month across their customer base. Their embedding drift detection runs on 1 percent of all inferences, sampled randomly.

**Lesson learned.** Embedding-based drift detection is a powerful technique for catching subtle behavioral changes. It catches issues that metric-based monitoring misses. The challenge is that generating embeddings for every inference is expensive (approximately 0.10 USD per 1,000 embeddings).

### Helicone

Helicone is a lightweight, open-source observability platform focused on cost and latency tracking. It is designed to be a drop-in replacement for the OpenAI API endpoint.

**How they do it.** Helicone acts as a proxy between your application and the LLM API. You configure your application to point to Helicone's endpoint instead of the OpenAI API directly. Helicone forwards the request to OpenAI, captures the response, logs all metrics, and returns the response to your application. No code changes are needed.

```python
# Instead of:
openai.api_base = "https://api.openai.com/v1"

# Use:
openai.api_base = "https://oai.hconeai.com/v1"
```

**Key features.** Helicone provides real-time cost dashboards, latency alerts, and a simple API for querying historical usage. The proxy approach captures all data without any code changes. Helicone processes 100 million requests per month, with an average added latency of 15 milliseconds.

**Scale.** Helicone's proxy handles 300 requests per second during peak hours. The proxy is deployed on AWS Lambda with auto-scaling from 10 to 1,000 concurrent executions.

**Lesson learned.** The proxy approach is the simplest integration path: zero code changes, zero instrumentation. The trade-off is limited visibility: the proxy sees the API request and response but not the application context (retrieved documents, guardrail results, user identity). For basic cost and latency tracking, the proxy is sufficient. For full observability, you need instrumentation.

### MLflow

MLflow is the open-source ML lifecycle platform from Databricks. Its AI observability features are built on top of its experiment tracking and model registry.

**How they do it.** MLflow uses the OpenTelemetry standard for tracing. They provide auto-instrumentation packages for popular LLM SDKs (OpenAI, Anthropic, LangChain). The traces are stored in MLflow's tracking server, which can be self-hosted or managed by Databricks.

```python
import mlflow

mlflow.openai.autolog()

# All OpenAI calls are now logged to MLflow
response = openai.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Summarize this document."}]
)

# The trace is stored in MLflow's tracking server
# Access it via: mlflow.search_traces(experiment_ids=["1"])
```

**Key features.** MLflow integrates observability with the model registry. When you log a trace, you can associate it with a specific model version. This enables you to trace production issues back to the model version that caused them. If quality drops after deploying model version 3, you can look at all traces for version 3 and compare them with version 2's traces.

**Scale.** MLflow is self-hosted. Databricks customers run MLflow on their own infrastructure, handling up to 10 million traces per day on a single tracking server.

**Lesson learned.** Integration with the model registry is underrated. The ability to correlate traces with model versions makes debugging significantly faster. When a quality issue is detected, the first question is "Which model version changed?" and MLflow answers it immediately.

### OpenTelemetry for LLMs

OpenTelemetry is the industry standard for observability instrumentation. The LLM-specific extensions are relatively new but rapidly maturing. The OpenTelemetry semantic conventions for LLM spans define standardized attribute names:

```yaml
# OpenTelemetry LLM semantic conventions (proposed)
attributes:
  llm.request.model: "gpt-4o"
  llm.request.temperature: 0.3
  llm.request.max_tokens: 1024
  llm.response.completions: 1
  llm.usage.prompt_tokens: 1847
  llm.usage.completion_tokens: 342
  llm.usage.total_tokens: 2189
  llm.response.finish_reason: "stop"
  llm.cost: 0.0023
```

The OpenTelemetry ecosystem includes collectors, processors, and exporters that can be composed into custom pipelines. The key advantage is vendor neutrality: you can switch between storage backends (Jaeger, Tempo, ClickHouse) without changing instrumentation.

### LangFuse

LangFuse is an open-source observability platform specifically designed for LLM applications. It provides:

- **Trace visualization:** A tree view showing the full execution flow of an LLM request.
- **Prompt management:** Version-controlled prompt templates with deployment tracking.
- **Evaluation:** Built-in evaluators for factuality, relevance, and toxicity.
- **Feedback collection:** Simple API for collecting user feedback (thumbs up/down).

LangFuse is self-hosted and uses PostgreSQL as its primary store. For high-scale deployments, it supports ClickHouse as a trace store. The platform handles 10 million traces per day on a single ClickHouse node.

### Phoenix (Arize Open Source)

Phoenix is Arize's open-source observability platform. It provides:

- **LLM trace visualization:** A web UI for exploring traces and spans.
- **Embedding visualization:** 2D and 3D visualizations of embeddings for drift detection.
- **Evaluation:** Built-in evaluators using LLM-as-judge.
- **Notebook integration:** Python API for exploring traces in Jupyter notebooks.

Phoenix is designed for development and staging environments. It runs in a single Python process and stores data in memory. For production use, Arize recommends the managed platform.

### Choosing between Open Source and Managed

| Factor | Open Source | Managed |
|--------|-------------|---------|
| Cost | Free (infrastructure cost only) | 500 to 5,000 USD/month |
| Setup time | 1 day to 2 weeks | 1 hour |
| Maintenance | Your team maintains it | Provider handles it |
| Customization | Full control | Limited to provider's features |
| Storage | You manage scaling | Provider handles scaling |
| Compliance | Full control over data location | Depends on provider |
| Reliability | Depends on your infrastructure | Provider SLA (99.9% typically) |

Most teams start with open source for development and staging, then use a managed service for production. The cost difference is justified by the reduced maintenance burden for production deployments.

## Cloud Provider Offerings

### AWS SageMaker

SageMaker provides Model Monitor, which detects drift in deployed models. For LLMs, SageMaker captures:

- **Input drift:** Distribution changes in the prompt text (using embeddings).
- **Output drift:** Distribution changes in the response text.
- **Model quality:** Comparison against a baseline dataset.

SageMaker Model Monitor integrates with CloudWatch for metrics and alerts, and with S3 for storing inference data. The pricing is based on the number of inferences monitored: 0.10 USD per 1,000 inferences for real-time monitoring.

**Limitations.** SageMaker's LLM observability is less mature than dedicated platforms. It focuses on drift detection rather than full trace analysis. You cannot view individual LLM call traces or drill into specific spans.

### GCP Vertex AI Experiments

Vertex AI Experiments provides experiment tracking and model evaluation. For LLMs, it captures:

- **Prompt and response:** The full text of each LLM call.
- **Model parameters:** Temperature, top_p, max_output_tokens.
- **Evaluation metrics:** Automatic evaluation using predefined metrics (BLEU, ROUGE, METEOR).
- **Comparison:** Side-by-side comparison of model outputs.

Vertex AI Experiments is integrated with the Vertex AI platform. If you are using Vertex AI for model deployment, the observability features are available with zero configuration.

**Limitations.** Vertex AI Experiments is designed for evaluation, not production monitoring. It does not provide real-time dashboards, alerting, or trace visualization. It is best used during model development and evaluation.

### Azure ML

Azure ML provides model monitoring with drift detection and data quality monitoring. For LLMs, it captures:

- **Prediction drift:** Changes in the distribution of model predictions.
- **Data drift:** Changes in the distribution of input data.
- **Model performance:** Accuracy, precision, recall against a baseline.

Azure ML Monitor integrates with Azure Monitor for alerts and Application Insights for tracing. The platform supports both batch and real-time monitoring.

**Limitations.** Azure ML's LLM observability is the least mature of the three cloud providers. It lacks LLM-specific features like token tracking, cost tracking, and prompt/response logging. Use it for infrastructure-level monitoring and supplement with a dedicated LLM observability platform.

### Lesson 1: Start with Trace ID Propagation

The most important infrastructure decision is propagating trace IDs across all services. Ensure that every service, every API call, every background job includes the trace ID as a header or context variable. Without this, you cannot stitch together a complete trace.

```python
# In every service
from opentelemetry import propagate

# Extract trace ID from incoming request
ctx = propagate.extract(request.headers)
# Use ctx in all downstream calls
```

### Lesson 2: Sampling is a Feature, Not a Bug

Teams often resist sampling because they want to see every request. In practice, sampling is liberating. It reduces costs by 90 percent while still capturing the most important events. The key is intelligent sampling: always keep errors, keep high-latency traces, and keep traces with unusual attributes.

### Lesson 3: Instrument Your Own Code, Not Just the LLM

Many teams instrument only the LLM call itself. They miss the context: the retrieval step, the guardrail check, the post-processing logic. A complete trace includes all steps from user input to final response. Without this, you cannot determine whether the issue was in the LLM, the retrieval, or the post-processing.

### Lesson 4: Feedback Loops Are More Valuable Than Dashboards

Dashboards show you what happened. Feedback loops (user feedback, automated evaluations, CI/CD regression tests) tell you whether the result was good or bad. Invest in building feedback loops early. A dashboard with no feedback is a data graveyard. A feedback loop with no dashboard is still useful.

### Lesson 5: Plan for Cost From Day One

LLM observability costs scale with LLM usage. A system that costs 10 USD per month at 10,000 requests per day costs 500 USD per month at 10 million requests per day. Plan your sampling strategy, retention policy, and storage tiering from the start. Retrofitting cost controls is harder than building them in.
