## Foundation Tier

### Challenge 1: Instrument a Simple LLM Call

**Task.** Given the following Python function that calls an LLM, add observability instrumentation that captures: model name, input tokens, output tokens, latency, and cost. Use the OpenTelemetry span API.

```python
import openai
import time

def get_response(user_message: str) -> str:
    start = time.time()
    response = openai.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": user_message}],
        max_tokens=500
    )
    duration_ms = (time.time() - start) * 1000
    return response.choices[0].message.content
```

**Evaluation criteria.**
- The span is created with the correct parent context.
- Span attributes include model name, input tokens, output tokens, latency, and cost.
- The span status is set to OK or ERROR based on the API response.
- The span is exported even if the API call raises an exception.
- Cost is calculated from token counts using the correct pricing.

**Model answer.**

```python
from opentelemetry import trace
from opentelemetry.trace import Status, StatusCode

tracer = trace.get_tracer("ai-agent", "1.0.0")

# GPT-4o pricing: $2.50 per 1M input tokens, $10.00 per 1M output tokens
INPUT_TOKEN_PRICE = 2.50 / 1_000_000
OUTPUT_TOKEN_PRICE = 10.00 / 1_000_000

def get_response(user_message: str) -> str:
    with tracer.start_as_current_span("llm.call") as span:
        span.set_attribute("llm.model", "gpt-4o")
        span.set_attribute("llm.max_tokens", 500)
        span.set_attribute("llm.temperature", 0.0)  # default

        start = time.monotonic()
        try:
            response = openai.chat.completions.create(
                model="gpt-4o",
                messages=[{"role": "user", "content": user_message}],
                max_tokens=500
            )
            duration_ms = (time.monotonic() - start) * 1000

            input_tokens = response.usage.prompt_tokens
            output_tokens = response.usage.completion_tokens
            cost = (input_tokens * INPUT_TOKEN_PRICE +
                    output_tokens * OUTPUT_TOKEN_PRICE)

            span.set_attribute("llm.input_tokens", input_tokens)
            span.set_attribute("llm.output_tokens", output_tokens)
            span.set_attribute("llm.total_tokens",
                               input_tokens + output_tokens)
            span.set_attribute("llm.latency_ms", duration_ms)
            span.set_attribute("llm.cost_usd", round(cost, 6))

            span.set_status(Status(StatusCode.OK))
            return response.choices[0].message.content

        except Exception as e:
            span.set_status(Status(StatusCode.ERROR, str(e)))
            span.record_exception(e)
            raise
```

### Challenge 2: Design a Sampling Strategy

**Task.** Your application handles 50 million requests per day. The observability pipeline costs 15,000 USD per month. Your VP of Engineering wants to reduce costs to 5,000 USD per month while maintaining the ability to detect production issues. Design a sampling strategy that achieves this.

**Constraints.** The team needs to detect: latency regressions (affecting more than 1 percent of traffic), quality regressions (affecting more than 0.5 percent of traffic), and error spikes (any error rate above 0.1 percent). The pipeline must provide actionable data within 10 minutes of an issue occurring.

**Evaluation criteria.**
- The strategy addresses all three detection requirements.
- The strategy provides specific sampling rates and justification.
- The strategy includes a plan for handling changes over time.
- The cost reduction is achievable with measurable assumptions.

**Model answer.**

The strategy uses three-tier sampling:

**Tier 1: Error traces (100 percent sampling).** Every trace with an error status is always kept. This costs approximately 500 USD per month and guarantees that all error incidents are captured.

**Tier 2: High-latency traces (100 percent sampling).** Every trace with latency above the P95 threshold (2 seconds) is always kept. This costs approximately 1,500 USD per month.

**Tier 3: Normal traces (tail-based importance sampling at 5 percent).** Remaining traces are sampled at 5 percent using importance weighting. Traces with unusual attributes (high token count, unusual model, first-time user) are sampled at 20 percent. Routine traces are sampled at 3 percent. This costs approximately 2,500 USD per month.

Total: 500 + 1,500 + 2,500 = 4,500 USD per month. This is below the 5,000 USD target.

For detection: latency regressions are detected via Tier 2 (100 percent of high-latency traces provide sufficient data). Quality regressions are detected via Tier 3 (5 percent of normal traces with 2 million samples per day provides statistically significant quality data). Error spikes are detected via Tier 1 (100 percent of error traces).

The strategy adapts over time: if the error rate increases, Tier 1 costs increase proportionally. If the team needs more data for quality analysis, Tier 3 sampling can be increased to 10 percent at an additional 2,500 USD per month.

### Challenge 3: Explain the CAP Trade-Off

**Task.** An observability platform stores trace data in a distributed database. The database is partitioned across 3 nodes. During a network partition, one node becomes unreachable. The database must choose between consistency and availability. Explain the trade-off and recommend a strategy for the observability use case.

**Evaluation criteria.**
- Clear explanation of the CAP theorem.
- Specific recommendation for the observability use case.
- Justification based on observability requirements.

**Model answer.**

The CAP theorem states that a distributed system cannot simultaneously provide Consistency, Availability, and Partition Tolerance. During a network partition, the system must choose between consistency (returning an error rather than stale data) and availability (returning potentially stale data rather than an error).

For an AI observability system, the recommendation is to prioritize availability over consistency. There are two reasons:

First, observability data is append-only. It is never updated. This means that the consistency risk is limited to reading data that has not yet been replicated — not reading stale data that has been overwritten. The consequence of eventual consistency is a slightly stale dashboard, not an incorrect one.

Second, observability systems are used for investigation and alerting. A dashboard that is 5 seconds behind is acceptable. A dashboard that returns an error is not. During an incident, the team needs the dashboard to work, even if it shows slightly delayed data.

The recommended strategy: configure the database with `quorum=one` for reads and `quorum=two` for writes. This means writes require confirmation from 2 out of 3 nodes (durability), but reads can be served from any single node (availability). During a partition, the system serves reads from the available nodes and buffers writes for the unreachable node.

## Design Tier

### Challenge 4: Design Observability for a Multi-Agent System

**Task.** Design an observability platform for a multi-agent AI system. The system has 3 agents: a coordinator agent, a research agent, and a writing agent. Each agent communicates with the others via function calls. The coordinator receives a user request, asks the research agent to gather information, asks the writing agent to compose a response, and returns the final answer.

**Requirements.**
- Trace a single user request through all 3 agents.
- Show the communication between agents as spans.
- Capture agent-specific metrics (research time, writing time, coordination time).
- Detect when an agent is stuck or looping.
- Support debugging by replaying the full trace.

**Evaluation criteria.**
- The trace structure captures the full agent interaction flow.
- Agent-specific metrics are correctly attributed.
- Loop detection is implemented.
- The design handles agent failures gracefully.

**Model answer.**

The trace structure uses a root span for the user request. Each agent call creates a child span. Agent-to-agent communication creates a nested span structure:

```
user.request (root)
  ├── coordinator.plan (50ms)
  ├── research.agent.call
  │   ├── research.query (2.1s)
  │   └── research.summarize (800ms)
  ├── writing.agent.call
  │   ├── writing.draft (1.4s)
  │   └── writing.review (300ms)
  └── coordinator.merge (100ms)
```

Each agent exports its own spans with the same trace ID. The collector stitches the spans together using the trace ID and parent span ID.

For loop detection: each agent span includes a `loop_counter` attribute. If the loop counter exceeds 5 for a single trace, the span is marked with a "loop detected" warning. The alerting system fires when loop detection occurs in more than 0.1 percent of traces.

For debugging: the full trace is stored with all span attributes, including the agent's internal state at each step. The trace explorer allows replaying the trace step-by-step, showing the state before and after each agent call.

For agent failures: if an agent does not respond within 30 seconds, the root span is marked as "partial" and the missing spans are listed. The dashboard shows the partial trace with a warning banner.

### Challenge 5: Design a Privacy-Preserving Observability System

**Task.** Design an observability system for a healthcare AI assistant. The system must comply with HIPAA. You cannot store any personal health information (PHI) in the observability platform. At the same time, the ML team needs to monitor quality and detect regressions.

**Requirements.**
- No PHI stored in any observability store.
- The ML team can detect quality regressions.
- The ML team can detect data drift.
- Incident investigation can identify the root cause without exposing PHI.
- Audit trail: the system must log which PHI was redacted and why.

**Evaluation criteria.**
- PII/PHI redaction is implemented at the earliest possible stage.
- The design provides useful data for ML teams without exposing PHI.
- Incident investigation is possible without PHI access.
- The audit trail is comprehensive.

**Model answer.**

The design uses three layers of protection:

**Layer 1: On-device redaction.** Before any span is emitted from the application, the instrumentation agent scans all text attributes for PHI patterns. The scanner detects: names (NER model), addresses (pattern matching), phone numbers (regex), email addresses (regex), medical record numbers (pattern matching), social security numbers (regex), and insurance IDs (pattern matching). Each detected PHI token is replaced with a placeholder: `[PHI_REDACTED:email]` or `[PHI_REDACTED:name]`. The original PHI is never stored or transmitted.

**Layer 2: Metadata-only observability.** The primary observability pipeline stores only metadata: latency, token count, cost, quality scores, and drift metrics. No text content is stored. The quality scores are computed from the redacted text, so the ML team can see whether quality is dropping without knowing the content of the conversations.

**Layer 3: Secure investigation vault.** When a quality regression is detected, the affected trace IDs are stored in a secure investigation vault. Only authorized personnel with HIPAA training can access the vault. The vault uses a just-in-time access model: access is granted for 24 hours with a specific purpose (incident investigation, compliance audit). All access is logged.

For the audit trail: every redaction event is logged to a separate audit log. The log includes: timestamp, span ID, attribute name, redaction type (email, phone, SSN), and the redaction reason ("HIPAA compliance"). The audit log is append-only with Write-Once-Read-Many (WORM) storage.

For drift detection: the system generates embeddings from the redacted text and compares them against a baseline. The embeddings are generated from the redacted version, so they capture changes in structure and topic without exposing PHI. If drift is detected, the ML team sees a notification: "Drift detected in user messages. Embedding centroid shifted by 0.03. Investigate via secure vault."

### Challenge 6: Design an Evaluation Pipeline

**Task.** Design a pipeline that evaluates every LLM response for quality, safety, and factual accuracy. The pipeline must process 10 million traces per day with an end-to-end latency of under 5 minutes.

**Requirements.**
- Each trace is evaluated by 10 evaluators.
- Evaluators include: factuality, relevance, toxicity, PII, sentiment, tone, format compliance, instruction adherence, safety, and bias.
- Evaluators run asynchronously to avoid delaying the application response.
- Evaluation results are available within 5 minutes of the trace being ingested.
- The evaluation pipeline must be fault-tolerant: if one evaluator fails, the evaluation continues.

**Evaluation criteria.**
- The pipeline architecture supports asynchronous evaluation.
- The pipeline handles evaluator failures gracefully.
- The pipeline meets the 5-minute latency requirement.
- The pipeline processes 10 million traces per day.

**Model answer.**

The pipeline uses a Kafka-based event-driven architecture:

1. When a trace is ingested, the collector publishes an event to Kafka topic `trace-evaluations`.
2. The evaluation pipeline is a Flink job with 10 parallel operators, one per evaluator.
3. Each operator reads from the same Kafka topic but processes independently.
4. As each evaluator finishes, it publishes the result to Kafka topic `evaluation-results`.
5. A merge operator collects all 10 results for each trace and publishes the complete evaluation to Kafka topic `trace-with-evaluations`.

For throughput: 10 million traces per day = 116 traces per second. Each evaluator takes 50 ms on average. A single evaluator instance processes 20 traces per second. With 10 evaluators, the pipeline needs approximately 6 instances per evaluator (116 / 20 = 5.8) to handle the load. Total: 60 instances.

For the 5-minute latency requirement: Kafka ingestion takes 100ms. Flink processing takes 50ms per evaluator, and evaluators run in parallel. The merge step takes 10ms. Total: 100ms + 50ms + 10ms = 160ms. The remaining 4 minutes and 50 seconds is budget for retries, backpressure, and queueing.

For fault tolerance: if a factuality evaluator fails, the Flink operator restarts and re-processes the last 1,000 events from the Kafka offset. The other evaluators continue unaffected. The merge operator waits for all 10 evaluator results. If one evaluator is down for more than 5 minutes, the merge operator marks the evaluation as "partial" and includes the available results.

## Expert Tier

### Challenge 7: Design a Global Observability Mesh

**Task.** Design an observability system for a global AI platform with 5 regions (US-East, US-West, EU-West, EU-Central, APAC) and 50 million requests per day. The system must provide a single pane of glass for global dashboards, while allowing regional teams to investigate their own traces.

**Requirements.**
- Global dashboards show aggregated metrics across all regions.
- Regional teams can investigate individual traces in their region.
- Cross-region traces are stitched together.
- A single region's observability failure does not affect other regions.
- The system must comply with EU data residency requirements (EU data stays in EU).

**Evaluation criteria.**
- The architecture isolates regions for failure.
- The architecture supports data residency requirements.
- Cross-region trace stitching is implemented correctly.
- Global dashboards have acceptable latency (under 10 seconds).

**Model answer.**

Each region runs an independent observability stack: OpenTelemetry collectors, Kafka, ClickHouse, and Grafana. Applications within a region send spans to the regional collector. The collector adds a `region` attribute to each span.

For global dashboards: each regional ClickHouse cluster exports aggregated metrics (not individual traces) to a global metrics store every 30 seconds. The export uses a push model: ClickHouse runs a scheduled query that computes aggregations for the last 30 seconds and sends the results to the global Prometheus/Mimir cluster. The global Grafana dashboard queries the global metrics store. Latency: under 30 seconds from application to global dashboard.

For cross-region trace stitching: when a trace spans multiple regions, the first region creates a "pending stitch" marker in the trace store. The marker includes the trace ID and the region. When the second region's trace is ingested, the collector checks for pending stitches. If found, the second region sends a copy of the trace to the first region's ClickHouse cluster. The first region's query engine merges the spans. The stitch process takes 50 to 200 milliseconds.

For EU data residency: EU regions do not export any data (including aggregated metrics) to non-EU clusters. The EU region has its own global dashboard that is hosted in EU. The worldwide global dashboard shows EU metrics as a single aggregate value, not broken down by EU region.

For failure isolation: if US-West fails, the other 4 regions continue normally. The global dashboard shows US-West data as "no data" for the last 30 seconds. After 30 seconds of no data, the global dashboard removes US-West from the aggregation and shows a warning banner. The US-West team investigates using their regional dashboard.

### Challenge 8: Design a Predictive Observability System

**Task.** Design an observability system that predicts quality regressions before they happen. The system must use historical observability data to train a model that predicts future quality drops.

**Requirements.**
- The system predicts quality drops 1 hour in advance.
- Leading indicators are identified from historical data.
- The prediction model is retrained daily.
- Predictions are surfaced as alerts in the dashboard.
- The system has a precision of at least 80 percent (no more than 20 percent false positives).

**Evaluation criteria.**
- The design identifies plausible leading indicators.
- The prediction model is appropriate for the task.
- The retraining pipeline is practical.
- The precision requirement is addressed.

**Model answer.**

The system uses a gradient-boosted decision tree (XGBoost) trained on historical observability data. The input features are derived from the last 60 minutes of observability data:

**Feature engineering:**
- Embedding drift: cosine distance between the current 5-minute embedding centroid and the baseline centroid (rolling 7-day window).
- Latency variance: standard deviation of latency over the last 15 minutes, normalized by the baseline.
- Token count shift: change in average token count per request compared to the same time yesterday.
- Error rate trend: slope of error rate over the last 30 minutes.
- Guardrail score variance: variance of the factuality score over the last 15 minutes.
- Model distribution: percentage of traffic going to each model version.
- Time of day: hour and day of week as categorical features.
- Prompt template version: categorical feature for the active prompt template.

**Training pipeline:**
- Label: quality drop (1 if the average quality score drops by more than 5 percent in the next hour, 0 otherwise).
- Training data: 90 days of historical observability data.
- Feature computation: sliding window of 60 minutes with 5-minute advance.
- Model: XGBoost with 500 trees, max depth 6, learning rate 0.1.
- Evaluation: 5-fold cross-validation with precision target of 80 percent.

**Feature importance analysis:**

The most important features from historical data are:
1. Embedding drift (importance: 0.31). When the embedding centroid shifts by more than 0.05, quality drops 1 hour later in 73 percent of cases.
2. Guardrail score variance (importance: 0.24). When factuality score variance exceeds 0.1, a quality drop occurs in 68 percent of cases.
3. Latency variance (importance: 0.18). When latency variance doubles, quality drops in 62 percent of cases.
4. Token count shift (importance: 0.15). When average token count changes by more than 20 percent, quality drops in 55 percent of cases.
5. Prompt template version (importance: 0.12). A new prompt template version is the strongest categorical predictor.

**Alerting integration:**
- When the model predicts a quality drop with probability above 0.7, an alert is sent to the AI team's Slack channel.
- The alert includes: predicted probability, top 3 contributing features, and recommended actions ("Check the latest prompt template deployment. Embedding drift is the primary contributor.")
- Precision is monitored weekly. If precision drops below 80 percent, the model is retrained with additional features or different hyperparameters.
- False positives are logged and analyzed. Common false positive patterns are used to create negative examples for the next training cycle.

The system is deployed as a Flink job that runs every 5 minutes. It reads the last 60 minutes of observability data from ClickHouse, computes features, and runs the XGBoost model. The prediction is written to a Kafka topic that feeds into the alerting system.
