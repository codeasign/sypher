## Data Structures and Algorithms

System design thinking is built on a foundation of data structures. The questions "how fast?", "how many?", and "what if it fails?" are answered by understanding the behavior of the underlying data structures at scale.

### Hash Tables for Load Distribution

Hash tables are the single most important data structure in system design. Every consistent hashing scheme, every partition strategy, and every cache shard is a hash table with a specific collision resolution strategy.

**What it is:** A hash table maps keys to values using a hash function. The hash function computes an index from the key, and the value is stored at that index. Collisions (two keys hashing to the same index) are resolved with chaining or open addressing.

**Why it matters for system design:** Hash tables determine how requests are distributed across servers. If your hash function distributes keys unevenly, some servers handle 10x the traffic of others. The hash function choice directly determines load balance quality.

**Complexity:** O(1) average case for get and put. O(n) worst case if all keys collide. In a distributed hash table (consistent hashing), each node handles O(k/n) keys where k is total keys and n is total nodes, assuming a uniform hash function.

**Real impact:** Facebook's memcached deployment uses consistent hashing to distribute cache keys across hundreds of servers. A poor hash function would cause hot spots where one server handles 30% of all requests while others handle 2%. They use a modified hash function that assigns each key to multiple servers for redundancy.

### Trees for Routing and Indexing

Trees maintain ordered data for range queries, prefix matching, and nearest-neighbor searches.

**B-Trees** are the default data structure for relational database indexes. A B-tree of order m maintains sorted data and allows search, sequential access, insertions, and deletions in O(log n) time. PostgreSQL indexes are B-trees. MySQL InnoDB uses B+ trees (a variant where only leaf nodes hold data pointers). A B-tree with a height of 3-4 can index billions of rows because each node stores hundreds of keys.

**Trie (prefix tree)** data structures power URL routers, autocomplete systems, and IP routing tables. A trie stores strings by sharing common prefixes. Finding all URLs with prefix "/api/v2/users/" is O(k) where k is the prefix length, regardless of how many URLs are stored.

**Consistent hashing (virtual ring)** is technically not a tree but uses a ring data structure organized as a binary search tree for O(log n) node lookup. When a request arrives for key K, the system finds the next node on the ring by binary search on the sorted node positions.

**Why it matters for system design:** When you design a database index strategy, you are designing a tree. When you design an API router, you are designing a trie. When you design a distributed cache, you are designing a consistent hash ring. Understanding the underlying data structure tells you the complexity, the failure modes, and the scaling limits.

### Queues for Async Processing

Queues are the backbone of asynchronous processing in distributed systems. A queue decouples the producer (who creates work) from the consumer (who processes work).

**What it is:** A FIFO (first-in, first-out) data structure, or a priority queue that orders by priority, or a topic-based queue that routes messages by topic.

**Why it matters for system design:** Without queues, every request must be processed synchronously. The system's throughput is limited by the slowest component. With queues, the web server accepts the request, enqueues it, and returns immediately. Background workers process the queue at their own pace. The web server never blocks on slow operations.

**Complexity:** Enqueue: O(1) (amortized). Dequeue: O(1). For priority queues: O(log n) for both operations.

**The critical insight about queues:** Queues absorb traffic spikes. If traffic jumps from 1,000 RPS to 10,000 RPS for 30 seconds, the web servers handle it (they just enqueue faster), but the database sees the same 1,000 RPS because workers dequeue at a controlled rate. The queue grows by 9,000 messages per second during the spike and drains afterward. The system survives the spike without crashing.

## The Request / Write / Read Lifecycle

Understanding every microsecond of a request lifecycle is what separates production system design from textbook system design. This section traces a single request through every layer.

### Step 1: DNS Resolution

The user types `short.ly/abc123` into their browser. The browser checks its local DNS cache. If not found, it queries the recursive DNS resolver, which queries the root nameserver, then the TLD nameserver, then the authoritative nameserver for short.ly. The entire DNS chain takes 20-120ms depending on cache state. The result is an IP address for the load balancer.

### Step 2: TCP Connection

The browser opens a TCP connection to the load balancer IP on port 443 (HTTPS). A three-way handshake occurs: SYN, SYN-ACK, ACK. This round trip takes 1-5ms within the same region, 50-200ms across continents. To avoid this overhead on every request, HTTP keep-alive reuses existing connections for multiple requests.

### Step 3: TLS Handshake

After TCP, the TLS handshake negotiates encryption parameters. The load balancer presents its certificate. The browser verifies it against a trusted CA. They exchange keys. This adds 2-3 round trips: 10-50ms on a fast connection, 100-300ms on a slow one. Production systems terminate TLS at the load balancer, not the application server, because TLS termination is CPU-intensive.

### Step 4: Load Balancer Routing

The load balancer receives the HTTPS request: `GET /abc123`. It terminates TLS, decrypts the request, and selects an application server. Selection algorithms:

**Round-robin:** Distributes evenly but ignores server load. Simple, predictable.
**Least connections:** Sends to the server with the fewest active connections. Better for variable-length requests.
**Least response time:** Sends to the server with the lowest average response time. Best for latency-sensitive apps.

The load balancer adds the `X-Forwarded-For` header with the original client IP and forwards the request to the chosen server over HTTP (already decrypted).

### Step 5: Application Server — Parse and Validate

The application server receives the request on a worker thread from its connection pool. The worker parses the HTTP method, path, headers, and body. It validates the path: does `/abc123` match the route pattern `GET /{short_key}`? Valid. It checks authentication (none needed for public redirects). It checks rate limiting (max 100 requests per second per IP). All validation passes.

### Step 6: Cache Lookup

The application constructs a cache key: `url:abc123`. It queries the Redis cluster. The query goes to the Redis node responsible for this key (determined by consistent hashing). Redis is in-memory, so the lookup takes 0.5-2ms. Two outcomes:

**Cache hit:** Redis returns the original URL. The application returns an HTTP 302 redirect immediately. Total time: under 10ms.

**Cache miss:** Redis returns nil. The application proceeds to the database.

### Step 7: Database Query

The application opens a database connection (from a pool, so no TCP handshake overhead — connection reuse). It sends:

```sql
SELECT original_url FROM urls WHERE short_key = 'abc123';
```

PostgreSQL checks its shared buffer cache. If the page containing this row is in memory, the read completes in 1-3ms. If the page was evicted, PostgreSQL reads from disk: 8-15ms for a random disk read on SSD. The B-tree index on `short_key` (primary key) is always in memory for a frequently accessed table — the index is small compared to the data.

### Step 8: Response and Redirect

The application receives the URL. It constructs an HTTP 302 response with the `Location` header set to the original URL. The response travels back through the load balancer, over the internet, to the user's browser. The browser receives the 302, reads the `Location` header, and initiates a new request to the target URL.

Total request lifecycle:
- Cache hit: 3-10ms (server processing only, plus network)
- Cache miss: 15-60ms (server processing only)

### Step 9: Asynchronous Side Effects

The redirect also triggers asynchronous operations that do not block the response:

1. A click event is emitted to Kafka: `{short_key: "abc123", timestamp: "...", referrer: "...", ip: "..."}`.
2. The analytics consumer picks up the event and increments the click counter for this URL.
3. If this is the first redirect for this URL, a background job marks the URL as "activated" — a signal for the billing system.

### Step 10: Complete Write Path (Creating a URL)

When a user creates a short URL, the write path differs significantly:

1. User sends `POST /shorten` with `{"url": "https://..."}`.
2. Application checks the local key cache (pre-allocated range of IDs).
3. Application converts numeric ID to base62 short key.
4. Application opens a database transaction.
5. Application executes `INSERT INTO urls (short_key, original_url, ...) VALUES (...)`.
6. Application commits the transaction.
7. Application writes to cache: `SET url:abc123 "https://..." TTL 3600`.
8. Application emits a `url.created` event to Kafka.
9. Application returns `201 Created` with the short URL.

The write path must complete in under 50ms. The critical resource is the database write throughput. At 1,200 writes per second, a single PostgreSQL primary can handle it with proper configuration. Beyond 5,000 writes per second, you need sharding or a distributed database.

## Consistency and State

Distributed systems face fundamental challenges around state and consistency. This section examines the specific scenarios that make system design hard.

### Node Crashes Mid-Operation

A node processing a write request crashes after writing to the database but before responding to the user. The user's client times out and retries. The retry arrives at a different node. That node checks the database, finds the data already inserted, and returns success. This is an **at-least-once** delivery semantic. The write succeeded, the user got a confirmation, but the system was temporarily inconsistent (between crash and retry).

The database handles this with **idempotency keys**. Each write request carries a unique idempotency key. If the server receives a request with a key it has already processed, it returns the previous result without re-executing the write.

### Network Partitions

A network partition occurs when two parts of a distributed system cannot communicate with each other. This is not a failure of either side — the network itself dropped packets, or a switch failed, or a firewall rule changed.

During a network partition, the system must choose: serve stale data or return an error. This is the CAP theorem in action. A partition-tolerant system must choose between availability (respond with possibly stale data) and consistency (refuse to respond until the partition heals).

Most production systems choose availability for read-heavy workloads. A user seeing slightly stale data (an unread notification that has been read) is better than a user seeing an error page.

### Concurrent Writes

Two users update the same resource simultaneously. Without coordination, the last write wins and the first write is lost. This is a **write-write conflict**.

Solutions:
- **Pessimistic locking:** Lock the row before writing. Other writers wait. Guarantees consistency. Reduces throughput.
- **Optimistic concurrency:** Read a version number, write only if the version has not changed. If another writer changed the version, retry. High throughput under low contention. High retry rate under high contention.
- **Last-write-wins:** Accept both writes, keep the one with the later timestamp. Simple, fast, loses data. Acceptable for non-critical data like "last viewed timestamp."

### Stale Reads (Read-Write Conflict)

A writer commits a change. A reader reads from a replica that has not yet received the change. The reader sees stale data.

This is **replica lag**. In a normal system, replica lag is under 50ms. Under heavy write load, it can grow to seconds or minutes. Stale reads during replica lag are expected.

Solutions:
- **Read-after-write consistency:** After writing, read from the primary for the next N seconds. Use a cookie or a database marker to track "I just wrote, send me to primary."
- **Quorum reads:** Read from multiple replicas and take the latest version. Works with versioned data (like DynamoDB).
- **Synchronous replication:** Wait for the replica to acknowledge the write before responding to the user. Increases write latency by 1-5ms. Guarantees no stale reads.

### Clock Skew and Logical Clocks

Distributed systems cannot rely on physical clocks. System clocks drift by up to 1-5 milliseconds per second on standard hardware. Running NTP (Network Time Protocol) synchronization every 60 seconds reduces drift to under 1ms, but clock resynchronization can cause time to jump forward or backward by tens of milliseconds.

**Why this matters:** If you use timestamps to determine the order of events, clock skew causes incorrect ordering. Event A on Server 1 might have timestamp 10:00:00.001, and Event B on Server 2 might have timestamp 10:00:00.000 — even though Event B actually happened after Event A. Without correction, Event A appears to be the latest event.

**Solutions:**
- **Lamport clocks:** Each server maintains a monotonically increasing counter. Every event includes the counter. When servers communicate, they exchange counters and the higher counter wins. Guarantees: if Event A happens before Event B, then A's counter is less than B's counter. Does NOT guarantee: that all servers agree on the total order of all events.
- **Vector clocks:** Each server maintains a vector of counters, one per server. Events record the full vector. Two events with equal vectors are concurrent (neither happened before the other). Concurrent events require conflict resolution. Amazon Dynamo uses vector clocks for last-write-wins conflict resolution.

**Production reality:** Most production systems accept that physical clock skew causes minor ordering errors. Log entries may be out of order by a few milliseconds. This is acceptable for analytics and logging. For systems that require strict ordering (distributed databases, consensus systems), use logical clocks or a centralized timestamp service (like Google's TrueTime, which provides a bounded clock uncertainty interval).

### Split-Brain Scenarios

**Split-brain** occurs when a distributed system has two or more nodes that each believe they are the leader. This happens during a network partition when a secondary node cannot reach the primary and promotes itself, but the original primary is still running.

**Consequences:** Both nodes accept writes. When the partition heals, the writes from both nodes must be reconciled. If reconciliation is not possible (a user updated their password on both nodes to different values), data is lost or inconsistent.

**Solutions:**
- **Quorum-based leader election:** A leader needs a majority (N/2 + 1) of votes to be elected. During a partition, only one side can have a majority. Example: with 3 nodes, the leader needs 2 votes. During a partition (2 nodes on one side, 1 on the other), only the side with 2 nodes can elect a leader.
- **Lease-based leadership:** The leader holds a lease with a time limit (e.g., 30 seconds). It must renew the lease before it expires. If the leader cannot renew (network partition), the lease expires, and another node can take over. This guarantees a bounded period of unavailability (30 seconds) and prevents split-brain.
- **Fencing tokens:** Each leader election generates a monotonically increasing token. Every write includes the current token. If a former leader (with an older token) tries to write, the database rejects it. This prevents the "zombie leader" problem.

| Branch mispredict | 3 nanoseconds | CPU pipeline flush |
| L2 cache reference | 7 nanoseconds | CPU cache |
| Mutex lock/unlock | 25 nanoseconds | Contended: 10-100 microseconds |
| Main memory reference | 100 nanoseconds | RAM access |
| Compress 1 KB with Snappy | 3 microseconds | Compression algorithm |
| Send 2 KB over 1 Gbps network | 20 microseconds | Within same data center |
| Read 1 MB sequentially from memory | 30 microseconds | Sequential access |
| Round trip within same data center | 500 microseconds | Two-tier app + DB |
| Disk seek | 2-10 milliseconds | SSD random read |
| Read 1 MB sequentially from SSD | 500 microseconds | Sequential SSD read |
| Read 1 MB sequentially from disk | 15 milliseconds | Spinning disk |
| Packet CA to Netherlands and back | 150 milliseconds | Transatlantic round trip |

**The key insight:** Reading 1 MB from SSD (500 microseconds) is faster than reading 1 MB from spinning disk (15 milliseconds) by a factor of 30. A cache hit in Redis (under 2ms) is 10x faster than a database read from SSD (2-10ms). Every layer of the system has a cost, and you pay the most for disk I/O and cross-region network calls.

### Latency Distribution: p50, p95, p99, p999

Average latency is meaningless for system design. A system with 100ms average latency might have 10ms p50 and 5,000ms p999 — meaning half of users get fast responses, but the slowest 0.1% wait five seconds.

**Percentile meanings:**
- p50 (median): 50% of requests are faster than this. Not useful for understanding tail behavior.
- p95: 95% of requests are faster than this. The "good enough" measurement for most systems.
- p99: 99% of requests are faster than this. The standard SLA target for production systems.
- p999: 99.9% of requests are faster than this. Critical for real-time systems.

**Example latency distribution for a well-tuned system at 10,000 RPS:**

| Percentile | Latency | User Experience |
|---|---|---|
| p50 | 8ms | Feels instant |
| p90 | 15ms | Feels instant |
| p95 | 25ms | Feels instant |
| p99 | 80ms | Noticeable delay |
| p999 | 350ms | Annoying |
| p9999 | 1,200ms | Likely timeout or error |

The jump from p99 (80ms) to p999 (350ms) reveals that 0.1% of requests hit a slow path — typically a cache miss combined with a database read from disk.

### The Relationship Between Latency and Throughput

Latency and throughput are not independent. As throughput approaches the system's capacity, latency increases non-linearly. This is modeled by **Little's Law**: L = lambda x W, where L is the number of requests in the system, lambda is the arrival rate (throughput), and W is the average time per request (latency).

**Practical implication:** At 50% capacity, latency is close to the no-load latency. At 80% capacity, latency is approximately 2x no-load. At 95% capacity, latency is 5-10x no-load. At 99% capacity, latency approaches infinity (the system collapses).

This is why production systems are never run at full capacity. The target utilization is:
- CPU: target 60-70% headroom for traffic spikes
- Memory: target 70-80% headroom for GC or allocation bursts
- Database connections: target 60% of max_connections
- Network bandwidth: target 50% of link capacity

**Why 50% for network:** Network links have non-linear latency curves. At 50% utilization, latency is near zero. At 80%, latency starts climbing. At 90%, latency doubles. At 95%, latency spikes by 10x. This is caused by queuing in network buffers.

### Backpressure and Load Shedding

When a system is overloaded, the worst thing you can do is accept all requests and let them queue up. Queued requests consume memory, time out, and waste resources that could be used to serve requests that might complete.

**Backpressure** is a mechanism where a component that is overloaded signals its upstream caller to slow down. In a gRPC call chain, the downstream service returns a `RESOURCE_EXHAUSTED` status code. The upstream service stops sending requests and retries with backoff.

**Load shedding** is the practice of dropping requests when the system is overloaded. The dropped request returns a 503 Service Unavailable immediately, rather than queuing and timing out later. The client receives immediate feedback and can retry with backoff.

**Implementation:**
```python
class LoadShedder:
    def __init__(self, max_queue_depth=1000, max_latency_ms=500):
        self.queue_depth = 0
        self.p50_latency = 0
        self.max_queue_depth = max_queue_depth
        self.max_latency_ms = max_latency_ms

    def should_accept_request(self):
        # Shed load if queue is too deep
        if self.queue_depth > self.max_queue_depth:
            return False
        # Shed load if latency exceeds threshold
        if self.p50_latency > self.max_latency_ms:
            return False
        return True
```

Throughput is limited by the slowest component in the critical path. Here are realistic throughput limits:

| Component | Max Throughput | Limiting Factor |
|---|---|---|
| Single web server (Node.js) | 10,000-30,000 RPS | CPU, available connections |
| Single web server (Go) | 50,000-100,000 RPS | Network bandwidth |
| Single PostgreSQL instance | 5,000-10,000 QPS | CPU, disk I/O |
| Single Redis instance | 100,000-200,000 QPS | Network, single-threaded CPU |

## Configuration and Tuning

A system design is theoretical until you set actual configuration values. The configuration knobs you turn determine whether the system survives production traffic or collapses under the first real load test.

### Operating System Tuning

Before any application-level tuning, the operating system must be configured for high-throughput network workloads.

**TCP settings on the application server:**
```
# /etc/sysctl.conf — TCP tuning for high-throughput servers

# Increase the range of ephemeral ports (default: 32768-60999)
net.ipv4.ip_local_port_range = 1024 65535

# Enable TCP fast open (reduces latency by 1 RTT on repeat connections)
net.ipv4.tcp_fastopen = 3

# Increase the maximum number of open files
fs.file-max = 1000000

# Enable TCP keepalive with shorter intervals
net.ipv4.tcp_keepalive_time = 120
net.ipv4.tcp_keepalive_intvl = 30
net.ipv4.tcp_keepalive_probes = 3

# Enable connection reuse in TIME_WAIT state
net.ipv4.tcp_tw_reuse = 1
```

Without these settings, a server handling 10,000 RPS exhausts its ephemeral port range in under 6 seconds. New connections are rejected. The system appears to be down. The root cause is not the application — it is the OS refusing to allocate ports.

### Garbage Collection Tuning (Java)

Java services require explicit GC tuning. The default GC is optimized for throughput on batch jobs, not for low latency on interactive services.

**For latency-sensitive services:**
```
-XX:+UseG1GC
-XX:MaxGCPauseMillis=50
-XX:G1HeapRegionSize=16m
-XX:InitiatingHeapOccupancyPercent=45
-Xms8g -Xmx8g
```

G1GC with a 50ms pause target keeps GC pauses under 50ms 99% of the time. A default GC without tuning can cause 500ms+ pauses that spike p99 latency.

**For throughput-oriented services (workers, batch processors):**
```
-XX:+UseParallelGC
-XX:ParallelGCThreads=4
-Xms16g -Xmx16g
```

Parallel GC maximizes throughput at the cost of longer individual pauses. This is fine for batch workers that process messages from a queue — the worker is not serving interactive requests.

### Database Configuration (PostgreSQL)

PostgreSQL default configuration is tuned for a laptop, not a production server. These are the most impactful settings to change:

```ini
# postgresql.conf — Production tuning for 32GB RAM, 8-core server

# Memory
shared_buffers = 8GB           # 25% of total RAM
effective_cache_size = 24GB    # 75% of total RAM
work_mem = 64MB                # Per-operation sort memory
maintenance_work_mem = 2GB     # For VACUUM, CREATE INDEX

# WAL (Write-Ahead Log)
wal_buffers = 64MB
wal_level = replica
max_wal_size = 4GB
min_wal_size = 1GB

# Connections
max_connections = 200
superuser_reserved_connections = 10

# Query tuning
random_page_cost = 1.1         # For SSD (default is 4.0 for spinning disk)
effective_io_concurrency = 200 # For SSD
```

**The single most impactful change:** Set `random_page_cost` to 1.1 for SSD storage. The default of 4.0 tells PostgreSQL that random I/O is expensive, making it prefer sequential scans over index scans. On SSD, random reads are almost as fast as sequential reads. With default settings, PostgreSQL often chooses the wrong query plan.

### Thread Pool Size

The thread pool determines how many requests can be processed concurrently. If the pool is too small, requests queue up and latency spikes. If the pool is too large, context switching overhead increases latency and memory usage grows.

**The formula:** The optimal thread pool size depends on whether the workload is CPU-bound or I/O-bound.

- CPU-bound (compress image, render template): threads = number of CPU cores + 1
- I/O-bound (database query, HTTP call, file read): threads = cores × (1 + wait_time / service_time)

For an I/O-bound web server where each request waits 50ms for the database and spends 5ms computing: threads = 8 × (1 + 50/5) = 8 × 11 = 88 threads.

**Safe default:** 50 threads per CPU core for I/O-bound workloads. 2 threads per CPU core for CPU-bound workloads.

### Connection Pool Size

The database connection pool prevents the application from overwhelming the database with connections. Each connection consumes memory (approximately 5-10 MB in PostgreSQL).

**The formula:** Connection pool size = (number_of_instances × connections_per_instance) should not exceed database max_connections.

For 20 application servers with 50 connections each: 20 × 50 = 1,000 connections. A PostgreSQL database with max_connections = 500 would be overwhelmed. Solution: use a connection pooler like PgBouncer between the application and database. PgBouncer handles 1,000 application connections with 50 actual database connections.

**Safe default:** 10-30 connections per application instance. Never let total connections exceed 80% of the database's max_connections.

### Batch Size

When writing or reading many records, batch operations dramatically improve throughput. Each batch operation incurs the same network and transaction overhead as a single operation, but processes N records.

**Database batch INSERT:**
- Single-row INSERT: 1,000 inserts = 1,000 round trips × 2ms = 2,000ms
- Batch INSERT (100 rows): 1,000 rows / 100 per batch = 10 round trips × 2ms = 20ms
- Improvement: 100x

**Safe default:** 100-1,000 rows per batch. Beyond 1,000 rows, diminishing returns and increased memory pressure.

### Timeouts

Every network call must have a timeout. Without timeouts, a slow or dead component causes cascading failures as threads pile up waiting for responses that never arrive.

**Timeout strategy:**
- Database query timeout: 5 seconds (recoverable), 30 seconds (hard limit)
- External HTTP call timeout: 2 seconds (connect), 10 seconds (read)
- Cache query timeout: 50ms (Redis is fast; if it is slower, treat as miss)
- Request timeout (total): 30 seconds per HTTP request

**Circuit breaker pattern:** If a downstream service fails N times consecutively (e.g., 5), stop sending requests to it for a recovery period (e.g., 30 seconds). This prevents cascading failures.

### Retry Policy

Not all failures should be retried. Client errors (4xx) should not be retried. Server errors (5xx) and network timeouts should be retried with backoff.

**Exponential backoff:**
- First retry: wait 100ms
- Second retry: wait 200ms
- Third retry: wait 400ms
- Fourth retry: wait 800ms
- Max retries: 3-5

**Jitter:** Add random variation to the wait time (e.g., ±25%) to prevent the thundering herd problem — all retrying clients hitting the recovering server simultaneously.
