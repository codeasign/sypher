## The Problem Story

A photo-sharing app launches on Product Hunt. The founder spent six months perfecting filters, building a beautiful feed, and tuning the color palette. On launch day, 5,000 people sign up in the first hour. Everything works. By hour three, 50,000 users are uploading photos, scrolling feeds, and leaving comments. The app slows to a crawl. By hour four, the server crashes with a connection pool exhaustion error. Users see a white screen. The founder restarts the server. It stays alive for eleven minutes before crashing again. The Product Hunt thread fills with "site is down" comments. The moment passes. The app never recovers its momentum.

An e-commerce company spends twelve months building a monolithic checkout system. Black Friday arrives. The marketing team drives 200,000 concurrent shoppers to the site. The checkout service tries to process every request in a single database transaction. The database hits 100% CPU at 8,000 transactions per second. Orders start timing out. The cart service, which shares the same database pool, cannot read or write. The inventory service, on the same application server, blocks waiting for database connections. Twenty minutes into the biggest shopping day of the year, the entire site is down. Revenue loss: 1.2 million dollars per hour.

A ride-sharing company designs its dispatch algorithm to assign the nearest driver to every rider. In a small city with 500 drivers, this works perfectly. The company expands to a megacity with 50,000 drivers and 200,000 active riders. The dispatch server now evaluates every driver within a 5-kilometer radius for every ride request. At 500 requests per second, the server performs 25 million distance calculations per second. The CPU melts. Riders wait fifteen minutes for a match. Drivers see empty screens. The algorithm was correct. The system design was wrong.

These stories share a single root cause: the teams built features without designing the system to handle load. They focused on functionality when they should have focused on architecture. They wrote code when they should have designed systems. The difference between an app that works for ten users and an app that works for ten million users is not better code. It is system design.

## The Insight

System design is the practice of defining the architecture, components, modules, interfaces, and data flow of a software system to satisfy specified requirements. It is the blueprint that determines how a system behaves under load, how it recovers from failure, how it scales from hundreds to millions of users, and how it evolves over years of development.

Think of system design as city planning. A single developer writing a CRUD app is like a carpenter building a single chair. You need skill, you need tools, and you need to know how wood behaves. System design is the urban planner who decides where the roads go, how wide the water pipes are, where the power substations sit, how many lanes the highway needs, and what happens when a sewer line breaks during rush hour. The carpenter builds one chair at a time. The urban planner ensures that ten million people can live, work, and move through the city without everything collapsing. The carpenter's work is visible immediately. The urban planner's work is invisible until the city floods, or the power goes out, or traffic gridlocks the downtown core on a Tuesday afternoon.

A system architect makes the same kind of invisible decisions: which database to use, how many servers to run, how to route traffic, where to cache data, what to do when a server fails, how to deploy new code without downtime, and how to recover from a data center outage. These decisions look invisible when they work. When they fail, the entire application stops working and you lose users, revenue, and trust.

## Core Vocabulary

**System design:** The process of defining the architecture, components, modules, interfaces, and data flow of a system to meet functional and non-functional requirements. Every system has a design. Some designs are intentional. Some designs emerge by accident. Intentional design wins.

**Architecture:** The highest-level structure of a system. Architecture is the set of decisions that are expensive to change later. Choosing a database is architecture. Naming a variable is not.

**Component:** A discrete, deployable unit of a system that provides a specific function. A web server is a component. A database is a component. A message queue is a component. A caching layer is a component. Each component does one thing and communicates with others through well-defined interfaces.

**Scalability:** The ability of a system to handle increased load by adding resources. Horizontal scalability means adding more machines. Vertical scalability means making each machine more powerful. The internet runs on horizontal scalability. The first question you ask about any system: does it scale horizontally?

**Availability:** The percentage of time a system is operational and accessible. Measured in nines: 99% (three days and 15 hours of downtime per year), 99.9% (8.7 hours), 99.99% (52 minutes), 99.999% (5.25 minutes). Production systems target 99.9% or higher. Critical financial and medical systems target 99.999%.

**Latency:** The time between sending a request and receiving a response. Measured in milliseconds. A database query takes 1-10ms on the same data center. A round trip to the user's browser takes 50-500ms. Every millisecond of latency above 100ms reduces user engagement measurably. Amazon calculated that 100ms of extra latency cost them 1% in revenue.

**Throughput:** The number of requests a system can process per unit of time. Measured in requests per second (RPS), queries per second (QPS), data per second (MB/s or GB/s). Throughput and latency are related: as throughput approaches capacity, latency spikes non-linearly.

**Trade-off:** Every engineering decision involves sacrificing one desirable property to gain another. Speed versus consistency. Availability versus cost. Simplicity versus flexibility. System design is the art of making these trade-offs deliberately rather than accidentally.

**Bottleneck:** The component that limits the throughput of the entire system. Finding the bottleneck is the most valuable debugging skill in system design. Fix the bottleneck, throughput improves. Fix anything else, nothing happens.

**Redundancy:** Running multiple copies of a critical component so that if one fails, another takes over. Redundant servers, redundant databases, redundant network paths, redundant data centers. Redundancy is the price of availability.

**Single point of failure:** A component whose failure causes the entire system to fail. A single database server. A single load balancer. A single network switch. A single power supply. A single engineer who knows how the deployment works. Eliminate single points of failure before you go to production.

**Load balancing:** Distributing incoming requests across multiple servers to prevent any single server from becoming a bottleneck. Load balancers also perform health checks, terminate TLS connections, and provide a fixed endpoint that hides the pool of servers behind it.

### Terminology in Practice

These terms are not academic. They appear in every design discussion, every architecture review, and every incident post-mortem. Here is how they show up in real conversations:

"When the cache evicts a hot key, 10,000 requests simultaneously miss and hit the database. That is a **thundering herd problem** caused by a single point of overload in an otherwise redundant system. The fix is a mutex lock on the cache key — only the first request talks to the database, the rest wait for the cache to populate."

"The **availability** target is 99.99%, which means we can tolerate 52 minutes of downtime per year. To hit that, we need **redundancy** at every layer: two load balancers in active-passive, three application servers behind an auto-scaling group, a primary database with a synchronous standby, and Redis with a replica cluster. Every **single point of failure** must be identified and eliminated."

"The **latency** target is p99 under 200ms. At 10,000 requests per second, p99 of 200ms means 100 requests per second take longer than 200ms. If we identify the **bottleneck** — a missing database index causing a sequential scan — and fix it, p99 drops to 30ms. The **throughput** increases because each request holds the connection for less time, freeing capacity for more concurrent requests."

"Every decision in this architecture is a **trade-off**. Using caching reduces read latency by 10x but introduces stale data. Adding a message queue decouples services but adds operational complexity. Choosing a NoSQL database gives us horizontal write scalability at the cost of transaction support. There is no right answer — there is only the answer that matches the **requirements**."

## How It Actually Works — Step by Step

Designing a system means making decisions in a specific order. Each decision constrains the next one. Here is the step-by-step process applied to designing a URL shortening service like bit.ly or TinyURL.

### Step 1: Gather Requirements

You start by asking questions. How many URLs per day? 100 million. How many reads per day? 1 billion. How long should URLs persist? Forever. Should we track clicks? Yes, with referrer data and timestamp. Should we support custom aliases? Yes, for premium users.

These numbers drive every subsequent decision. 100 million writes per day is approximately 1,200 writes per second. 1 billion reads per day is approximately 11,600 reads per second. Your system needs to handle roughly 12,000 RPS at peak with a read-to-write ratio of 10:1.

### Step 2: Estimate Storage

Each short URL record contains: a 6-character key (6 bytes), the original URL (average 200 bytes), a creation timestamp (8 bytes), a user ID for tracking (8 bytes), and a click counter (8 bytes). That is approximately 230 bytes per record. For 100 million new URLs per day, you need 23 GB of new storage per day. After one year, you need 8.4 TB of storage. This tells you that you cannot store everything on a single machine. You need a distributed database from the start.

### Step 3: Design the Data Model

The data model determines the API, the cache strategy, and the query patterns. For a URL shortener, you need two main tables: one for URL mappings and one for click events. URLs are read-heavy and need fast lookup by short key. Click events are write-heavy and never updated, only inserted and aggregated.

```
-- URL mappings table
CREATE TABLE urls (
    short_key    VARCHAR(6) PRIMARY KEY,
    original_url TEXT NOT NULL,
    created_at   TIMESTAMP NOT NULL DEFAULT NOW(),
    user_id      BIGINT,
    expires_at   TIMESTAMP
);

-- Index for quick lookup by short key is automatic (PK)
-- Hot data lives in cache, cold data lives in database

-- Click events table (append-only, time-series pattern)
CREATE TABLE clicks (
    id           BIGSERIAL,
    short_key    VARCHAR(6) NOT NULL,
    clicked_at   TIMESTAMP NOT NULL DEFAULT NOW(),
    referrer     TEXT,
    user_agent   TEXT,
    ip_address   INET,
    country_code CHAR(2)
) PARTITION BY RANGE (clicked_at);
```

Notice the partitioning on `clicks`. Without partitioning, a table with 500 billion rows becomes unqueryable. With monthly partitions, each partition holds roughly 3 billion rows, which is manageable.

### Step 4: Design the API

The API defines how components communicate. For the URL shortener, you need two primary endpoints and one internal streaming pipeline.

```
POST /shorten
  Request:  { "url": "https://example.com/very/long/path", "custom_alias": "myalias" }
  Response: { "short_url": "https://short.ly/myalias", "key": "myalias" }

GET /{short_key}
  Response: HTTP 302 redirect to original URL
  Side effect: emit click event to analytics pipeline

Internal: Click analytics stream
  Producer: redirect server → Kafka topic "clicks"
  Consumer: analytics worker → batch insert every 5 seconds
```

### Step 5: Choose the Storage Technology

Given the 10:1 read-to-write ratio, the storage system must handle fast point lookups by primary key. You have three options:

**PostgreSQL:** Mature, transactional, supports replication, handles 10,000+ QPS with proper indexing and connection pooling. Choose this if you need joins, transactions, or complex queries.

**Cassandra:** Distributed, write-optimized, handles millions of writes per second, no single point of failure. Choose this if you plan to exceed 100,000 writes per second or need multi-region writes.

**Redis:** In-memory key-value store, sub-millisecond latency, limited by RAM. Choose this as a cache layer, not primary storage for a URL shortener (URLs must persist).

For the URL shortener, the right choice is PostgreSQL for primary storage with Redis as a read cache. The 10:1 read ratio means caching eliminates 90% of database reads.

### Step 7: Generate Unique Keys

The short key must be globally unique and unpredictable. Sequential keys (1, 2, 3) can be scraped. UUIDs are 36 characters, too long for a short URL. The standard approach is base62 encoding of a unique numeric ID.

Base62 uses A-Z, a-z, and 0-9 (62 characters). With 6 characters, you have 62^6 = approximately 56 billion possible keys. With one year of data at 100M URLs per day, you use about 36.5 billion keys, so 6 characters gives you roughly 1.5 years of headroom.

The key generation must be distributed to avoid a single point of failure. One approach is a dedicated key-generation service that pre-allocates ranges:

```
KeyGen Service allocates range [500000, 599999] to Server A
Server A generates keys locally: 500000 → base62 → "2Rq3a"
Server A can generate 100,000 keys without talking to the KeyGen Service
KeyGen Service allocates [600000, 699999] to Server B
```

This eliminates the centralized ID generator as a bottleneck.

### Step 8: Choose the Communication Protocol

The communication protocol between the application server and other components determines latency, complexity, and debuggability. For the URL shortener, you have three protocol choices for the API:

**REST over HTTP:** The standard choice. Every resource has a URL. Every operation uses an HTTP verb (GET, POST, PUT, DELETE). Request and response bodies are JSON. Simple, cacheable, debuggable with curl. Every programming language has an HTTP client. This is the right default.

**gRPC:** Protocol Buffers over HTTP/2. Faster serialization (binary instead of JSON). Built-in streaming. Strongly typed contracts defined in .proto files. Better for internal service-to-service communication where you control both sides. Harder to debug (binary protocol). No browser support without a proxy.

**GraphQL:** The client specifies exactly which fields it needs. No over-fetching or under-fetching. Good for mobile apps with limited bandwidth. Complex server-side execution. Harder to cache. Not a good fit for a URL shortener that has exactly two operations (shorten a URL, redirect by key).

For the URL shortener, the right choice is REST over HTTP. The API surface is small (two endpoints). The clients are web browsers and mobile apps. The performance difference between REST and gRPC at 12,000 RPS is negligible compared to the debugging advantage of plain HTTP.

### Step 9: Handle the Write Path

When a user creates a short URL, the write path must be fast and durable. The user does not need to wait for cache population or analytics setup. The write path looks like this:

1. User sends POST /shorten with the original URL.
2. Application requests a key from the local key-allocator (no network call).
3. Application inserts into PostgreSQL: `INSERT INTO urls ...`
4. Application inserts into Redis: `SET url:abc123 original_url TTL 3600`
5. Application emits a message to Kafka topic "new-urls" for downstream processing.
6. Application returns the short URL to the user.

The entire write path should complete in under 50ms. The Kafka message is asynchronous and does not block the response.

### Step 9: Handle the Read Path at Scale

The read path must handle 11,600 reads per second with 99th percentile latency under 50ms. At this volume, every millisecond matters.

### Step 10: Handle Analytics

Click tracking adds write volume equal to the read volume (11,600 writes per second). You cannot insert into a database at this rate without overwhelming it. The solution is batching.

```
Click events go to Kafka → partitioned by short_key (16 partitions)
Analytics worker pool (4 workers per partition):
  - Buffer events in memory until buffer reaches 1,000 items
  - OR buffer time reaches 5 seconds
  - THEN batch INSERT into PostgreSQL
  - Batch size: 1,000 rows per INSERT
  - Writes per second to DB: 11,600 / 1,000 × (1/5) = 2.3 writes per second
```

The batch INSERT turns 11,600 individual writes into 2.3 batch writes per second. This is the difference between a database that works and a database that burns.

## Variants and Flavors

System design is not one-size-fits-all. The architecture you choose depends on your team size, traffic volume, latency requirements, and organizational constraints.

### Monolithic Architecture

A monolithic system runs all code in a single process. The web server, the business logic, the database access layer, and the background job processing all run inside the same application.

**Use monolithic when:** Your team has fewer than 10 engineers. Your traffic fits on one server (under 10,000 RPS). You are building an MVP or prototype. You need to ship fast and refactor later.

**Do NOT use monolithic when:** Your team has multiple autonomous teams. You need to scale different parts of the system independently. You require zero-downtime deployments for different components on different schedules.

A microservices system decomposes functionality into independently deployable services. Each service owns its data, exposes its API, and scales independently.

**Use microservices when:** Your team has multiple autonomous teams (5+ teams of 4-8 engineers). Different parts of the system have different scaling requirements. You need independent deployment cycles. Your system has grown to 50,000+ lines of code and a monolith has become unmanageable.

**Do NOT use microservices when:** Your team has fewer than 20 engineers. Your system has low traffic or simple functionality. You cannot invest in infrastructure for observability, deployment automation, and service discovery. Microservices without infrastructure are a distributed monolith with extra latency.

### Serverless Architecture

Serverless systems run code as stateless functions triggered by events. You do not manage servers. You do not manage containers. You write functions and configure triggers.

**Use serverless when:** Traffic is highly variable with long idle periods. You want to minimize operational overhead. Your workload is event-driven and short-lived (under 15 minutes per execution). You can tolerate cold start latency.

**Do NOT use serverless when:** You have sustained high traffic (cost becomes higher than provisioned servers). Your workload requires low-latency responses (under 10ms). You need long-running processes. Your application has high statefulness or requires local disk. You need predictable performance under load.

### Event-Driven Architecture

Event-driven systems communicate through events rather than direct API calls. When something happens (a user signs up, an order is placed, a payment is processed), an event is emitted. Other services react to events they subscribe to.

**Use event-driven when:** You need loose coupling between services. Multiple services need to react to the same event. You want to add new consumers without modifying existing producers. The system needs to replay past events (event sourcing pattern).

**Do NOT use event-driven when:** You need synchronous request-response between services. You need strong consistency guarantees across services. The debugging overhead of following event flows through multiple services is too high for your team size.

## Mental Model Check

Test your understanding of these core concepts before moving on.

**Question 1:** A team deploys a web application on a single server with a single database. The application works perfectly for 100 concurrent users at 200ms average latency. At 500 concurrent users, latency jumps to 2,000ms. What is the most likely bottleneck?

**Question 1:** A team deploys a web application on a single server with a single database. The application works perfectly for 100 concurrent users at 200ms average latency. At 500 concurrent users, latency jumps to 2,000ms. What is the most likely bottleneck?

The bottleneck is likely the single application server (CPU or connection limit) or the single database (connection pool exhaustion or query queue). With one server, there is no redundancy, no load distribution, and no way to scale beyond the machine's limits.

**Question 2:** An e-commerce site targets 99.99% availability. How many minutes of downtime does this allow per year, and what components must be redundant?

99.99% availability allows approximately 52 minutes of downtime per year. To achieve this, every component in the critical path must be redundant: multiple application servers, a load balancer with failover, a primary and standby database, and redundant network paths.

**Question 3:** A database serves 100 queries per second with an average latency of 5ms. A cache layer is added. The cache hit rate is 90%. Cache lookups take 1ms. Database queries that reach the database still take 5ms. What is the new average latency?

90% of requests hit cache at 1ms. 10% miss cache and go to the database at 5ms cache + 5ms database = 10ms. Average latency = 0.9 * 1ms + 0.1 * 10ms = 0.9 + 1.0 = 1.9ms. The cache cut latency by more than half.

**Question 4:** A company chooses microservices for a 5-person team building a prototype. The prototype takes 8 months to ship instead of 3. What went wrong?

The team chose a complex architecture before validating the product. Microservices impose overhead in service discovery, inter-service communication, deployment pipelines, observability, and debugging. A 5-person team should start with a monolith, validate the product, then decompose into services when the system grows beyond a single deployable unit.

**Question 5:** A read-heavy social media feed returns data in 500ms. The team adds Redis caching and the latency drops to 50ms. What trade-off did they make?

They traded cache coherence for speed. Cached data can be stale. If a user posts new content, the cache may serve the old feed for the cache TTL duration. They introduced a cache invalidation problem. The trade-off is acceptable for feeds where eventual consistency within seconds is acceptable. It would not be acceptable for a banking application where stale balance data could cause overdrafts.
