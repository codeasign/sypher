## How This Topic Appears

Notification system questions appear in FAANG interviews in four forms. Here are the most common phrasings:

### Direct Questions
- "How does a push notification system work?"
- "Explain the architecture of a notification system."
- "What happens when you send a push notification?"

### Design Questions
- "Design a notification system for a social media platform."
- "Design a system that sends 10 million notifications per day."
- "Design a notification service that supports push, email, and SMS."
- "Design a notification delivery system with at-least-once delivery guarantees."

### Debugging Questions
- "Our notification system is showing high latency. Walk me through debugging."
- "Users are receiving duplicate notifications. How do you investigate?"
- "The notification system is losing messages. What do you check?"

### Trade-off Questions
- "Why would you choose Kafka over RabbitMQ for a notification system?"
- "How would you handle rate limiting for push notifications?"
- "What are the trade-offs between batching and immediate delivery?"

## What the Interviewer Is Actually Testing

### For Direct Questions
**What they are testing:** Do you understand the fundamentals? Can you explain the system clearly? The interviewer wants to see that you know the basic components: event ingestion, queue, router, delivery workers, and feedback loop. A 3/4 answer lists these components. A 4/4 answer explains why each component exists and what breaks without it.

### For Design Questions
**What they are testing:** Can you design a distributed system under real constraints? The interviewer evaluates your ability to make trade-offs, account for failure modes, and design for scale. The signals they look for: you ask clarifying questions before designing, you address failure scenarios proactively, you provide concrete numbers for your design decisions, and you identify the bottlenecks in your own design.

### For Debugging Questions
**What they are testing:** Can you debug a distributed system systematically? The interviewer wants to see a structured approach: start with the user-facing symptom, narrow down to the layer, check the metrics, identify the root cause, and propose a fix. A weak answer jumps to conclusions. A strong answer follows a methodical process.

### For Trade-off Questions
**What they are testing:** Can you think in trade-offs? The interviewer wants to see that you understand the costs of your choices. The signal is: you mention both the benefits and the drawbacks of your choice, and you explain the conditions under which the other choice would be better.

## Level-by-Level Expectations

### L3/L4 — Junior / New Grad

**Minimum expected knowledge:**
- Know what a push notification is
- Know that push notifications go through APNs (Apple) or FCM (Google)
- Can explain the basic flow: event → queue → worker → provider → device
- Understands that notifications can fail and need retry

**Vocabulary they must know:** APNs, FCM, push token, notification payload, delivery receipt.

**What a passing answer looks like:**

Interviewer: "How does a push notification work?"

Candidate: "A push notification starts when the app server sends a request to the push notification provider — APNs for iOS or FCM for Android. The request includes the device's push token and the notification payload. The provider delivers the notification to the device. The device shows the notification to the user. If the token is invalid, the provider returns an error, and the server should remove the token."

**What distinguishes a 3/4 from a 4/4:** The 4/4 candidate mentions that the server should use a queue to decouple the notification request from the provider call, and that the server should handle the different error responses from the provider (invalid token, rate limit, server error).

### L5 — Mid-level

**Expected depth of understanding:**
- Can design a notification system that handles 1 million notifications per day
- Understands multi-channel delivery (push, email, SMS)
- Can explain the trade-offs between at-least-once and exactly-once delivery
- Knows about rate limiting, retry, and dead-letter queues
- Can discuss preference storage and user opt-out

**Example question + model answer:**

Interviewer: "Design a notification system that sends push, email, and SMS notifications."

Candidate: "I will start by clarifying the scale. How many notifications per day? What are the delivery SLAs? How many channels do we need to support? Let me assume 1 million notifications per day with a 99.9 percent delivery rate within 5 minutes.

The system has 5 layers. First, the ingestion layer accepts notification events through an API. The event includes the recipient, type, and payload. The ingestion layer validates the event and writes it to a Kafka topic.

Second, the processing layer enriches the event with user preferences, device tokens, and templates. This layer also checks for duplicates using a Bloom filter in Redis.

Third, the routing layer determines which channels to use. It checks the user's preferences — if the user has push enabled, send push. If push is not available, fall back to email.

Fourth, the delivery layer has separate workers for each channel. Push workers maintain HTTP/2 connections to APNs and FCM. Email workers connect to SendGrid or SES. SMS workers connect to Twilio.

Fifth, the feedback layer processes delivery receipts. Success receipts are logged. Failure receipts trigger retry. Invalid tokens are removed from the registry.

The key trade-off is between delivery speed and cost. For critical notifications, send all channels immediately. For non-critical notifications, batch by channel and time window. The rate limiter uses a token bucket per user and per provider to prevent abuse."

### L6 — Senior

**Expected depth of understanding:**
- Proactively discusses failure modes before the interviewer asks
- Quantifies everything — latency, throughput, storage, cost
- Discusses operational complexity: monitoring, deployment, upgrades
- Considers multi-region architecture and failover
- Addresses the business impact of design decisions

**Example question + model answer:**

Interviewer: "Design a notification system for a global social media platform with 100 million users."

Candidate: "Let me start by estimating the scale. With 100 million daily active users, each generating 50 notification events per day, we need to handle 5 billion events per day. That is roughly 58,000 events per second. At peak, 4x that, so 230,000 events per second. Each event is roughly 1 KB after enrichment, so 230 MB/s ingress.

The system must be multi-region because a user in Tokyo should not wait for a notification from a US-East data center. I will design for active-active in 3 regions: US-East, EU-West, and AP-Southeast. Each region has its own Kafka cluster. User preferences are replicated across regions using DynamoDB Global Tables. Device tokens are regional — the user registers with the nearest region.

For durability, I will use Kafka with replication factor 3 and min.insync.replicas=2. The system can tolerate one broker failure per partition without data loss.

For delivery, I will use a channel arbiter that considers user preferences, notification type, and delivery history. The arbiter assigns a priority to each notification. P0 (security alerts) are sent immediately through all channels. P1 (direct messages) are sent within 5 seconds. P2 (likes and follows) are batched within 30 seconds. P3 (marketing) are batched within 5 minutes.

The biggest operational risk is provider outages. I will implement circuit breakers for each provider. The circuit breaker opens when the error rate exceeds 5 percent over 30 seconds. When the circuit breaker is open, notifications are queued in the outbound topic. The circuit breaker transitions to half-open after 60 seconds and sends a probe request. If the probe succeeds, normal delivery resumes.

For cost management, SMS is the most expensive channel at $0.0075 per message. I will use SMS only for P0 notifications and only for users who have opted in. Email costs $0.0003 per message, so I will batch emails to reduce the count. Push notifications are nearly free, so I will use push as the default channel.

The monitoring system tracks 20 metrics: events per second, delivery rate per channel, latency per stage, consumer lag, circuit breaker state, and cost per channel. Alerts fire when the delivery rate drops below 99 percent, consumer lag exceeds 10,000, or any circuit breaker opens."

### L7/L8 — Staff / Principal

**Expected depth of understanding:**
- Discusses org-level impact — how does this design affect the engineering organization
- Connects technical decisions to business outcomes
- Addresses operational complexity: who owns the platform, how to onboard new teams
- Discusses cost optimization at the organizational level
- Considers the 5-year evolution of the system

**Example question + model answer:**

Interviewer: "Design a notification platform that serves multiple product teams across the company."

Candidate: "The key insight is that this is not just a technical system — it is an organizational platform. The design must account for how different product teams will use it, how we handle conflicting priorities, and how we evolve the platform without breaking existing integrations.

The platform provides a single API that all product teams use. The API accepts a notification event with a type, recipient, and payload. The platform handles channel selection, delivery, and feedback. The API is versioned (v1, v2) with a deprecation policy of 12 months.

Each product team gets a namespace for their notification types. The namespace configures the allowed channels, priority levels, and rate limits. Team A might have a rate limit of 10,000 events per second, while Team B has 1,000 events per second. The namespace configuration is self-service — teams can change their settings through a web UI or API.

The platform team provides service level objectives (SLOs) for each notification tier. Tier 1 (critical) notifications have a 99.99 percent delivery rate within 10 seconds. Tier 2 (normal) have 99.9 percent within 1 minute. Tier 3 (batch) have 99 percent within 1 hour. The platform team is on-call for Tier 1 SLO breaches. Product teams are alerted for Tier 2 and Tier 3 breaches.

The biggest organizational challenge is preventing abuse. A product team might send too many notifications, causing users to disable all notifications. The platform enforces a company-wide notification budget: each user receives at most 100 notifications per day across all product teams. If a team exceeds their share of the budget, their notifications are throttled.

The platform's cost is charged back to the product teams. Each team sees their monthly notification bill: $X for push, $Y for email, $Z for SMS. This creates a feedback loop — teams optimize their notification usage when they see the cost. The chargeback model reduced overall notification volume by 30 percent at companies that implemented it.

For the 5-year evolution, the platform should support new channels as they emerge (smart watches, smart home devices, chatbots). The provider adapter pattern allows adding new channels without changing the core pipeline. The template engine supports new message formats (rich media, interactive buttons, carousel messages). The platform's architecture should be extensible enough to support these without a rewrite."

## The FAANG Framework for Answering

### Step 1: Clarify Requirements

**What to ask:**
- How many notifications per day? (scale)
- What channels are required? (push, email, SMS, in-app)
- What is the delivery SLA? (latency, delivery rate)
- Are notifications persistent? (must survive restarts)
- Is multi-region required? (global user base)
- What is the budget? (cost constraints)

**What NOT to ask:**
- "What programming language should I use?" (irrelevant at the architecture level)
- "What database should I use?" (decide this during design, not requirements)
- "Can you tell me the exact requirements?" (you should define reasonable assumptions)

### Step 2: Estimate Scale

**Back-of-envelope calculations for notification systems:**
- DAU to events: 1 DAU = 50 notification events per day (average)
- Events per second: DAU × 50 / 86400 × peak factor (4x)
- Storage per day: events × 1.5 KB per event
- Kafka storage: 7 days × daily storage
- Notification history storage: 90 days × daily storage × 500 bytes per record
- Throughput per channel: events × channel percentage (push 60%, email 20%, SMS 5%, others 15%)

### Step 3: High-Level Design

**What to draw first:**
1. Event sources (left side) producing events
2. Ingestion layer (API gateway)
3. Dispatch queue (Kafka)
4. Router layer (consumer, dedup, preference, template)
5. Delivery layer (push, email, SMS workers)
6. Feedback layer (receipts, analytics)
7. User preferences database (queried by router)
8. Device token registry (queried by router)

### Step 4: Deep Dive

**What to expand on:**
- Dedup: Bloom filter with controlled false positive rate
- Rate limiting: token bucket per user, per provider, per channel
- Retry: exponential backoff with jitter, dead-letter queue
- Multi-region: active-active with regional Kafka clusters
- Fallback: push → email → SMS chain

### Step 5: Trade-offs

**What to volunteer:**
- "The trade-off between batching and latency is..."
- "I chose Kafka over SQS because..."
- "The Bloom filter trades memory for a small false positive rate..."
- "Multi-region adds complexity but provides lower latency for global users..."

### Step 6: Wrap Up

**How to close confidently:**
- "This design handles 10 million users. To handle 100 million, I would..."
- "The most important monitoring metric is the delivery success rate."
- "If I had more time, I would explore..."
- "I believe this design meets the requirements because..."

## Phrase Bank

### Framing the Trade-off
- "The trade-off here is between delivery speed and delivery cost. At this scale, I would optimize for cost by batching non-critical notifications."
- "One thing interviewers often want to see is whether you account for provider rate limits. I will address this by implementing a token bucket per provider connection."
- "The failure mode I would watch for here is the circuit breaker opening on APNs. When it opens, notifications are queued rather than dropped."

### Demonstrating Depth
- "The Bloom filter for dedup uses 10 bits per element with 7 hash functions. This gives a 1 percent false positive rate. For critical notifications, I add an exact-match check in Redis."
- "The retry policy uses exponential backoff with jitter. The first retry is at 10 seconds, the second at 60 seconds, and the third at 5 minutes. After 3 retries, the notification moves to the dead-letter queue."
- "The channel arbiter is a rule engine. Each rule has a condition (notification type, user preference, delivery history) and an action (send, defer, fallback, skip)."

### Handling Follow-Up Questions
- "If the user asks about cost, I would explain that SMS costs $0.0075 per message, so I limit SMS to critical notifications only."
- "If the user asks about monitoring, I would mention that we track consumer lag, delivery rate, and circuit breaker state as the top 3 metrics."
- "If the user asks about scale, I would describe the multi-region architecture and how it handles 10x traffic spikes."

### Closing Strong
- "The key insight is that notification delivery is a cross-cutting concern. A dedicated platform provides reliability, consistency, and cost optimization that ad-hoc solutions cannot match."
- "The most important lesson from production experience is that provider-specific error handling is mandatory. Generic retry logic causes more problems than it solves."

## Red Flags

### Red Flag 1: No Mention of Dedup
**What the candidate says:** "The notification is sent to the provider and delivered to the device."
**Why it signals a gap:** The candidate does not account for retries, duplicates, or race conditions. A real notification system must handle duplicate detection.

### Red Flag 2: No Queue
**What the candidate says:** "The app server calls the notification API directly."
**Why it signals a gap:** The candidate does not understand the need for decoupling. Direct calls couple the application to the notification provider, causing cascading failures.

### Red Flag 3: No Rate Limiting
**What the candidate says:** "The system sends all notifications as fast as possible."
**Why it signals a gap:** The candidate does not account for provider rate limits or user notification fatigue. The system will be throttled by providers and users will disable notifications.

### Red Flag 4: No Error Handling
**What the candidate says:** "The notification is sent successfully."
**Why it signals a gap:** The candidate assumes the provider always succeeds. Real providers fail. The system must handle failures, retries, and fallbacks.

### Red Flag 5: Single Queue for All Channels
**What the candidate says:** "All notifications go through one queue."
**Why it signals a gap:** The candidate does not understand channel-specific requirements. A single queue couples push, email, and SMS delivery, causing cross-channel interference.

### Red Flag 6: No Preference Store
**What the candidate says:** "The system sends notifications to all users."
**Why it signals a gap:** The candidate does not account for user opt-out, quiet hours, or channel preferences. Sending unwanted notifications leads to user churn.

### Red Flag 7: No Feedback Loop
**What the candidate says:** "The notification is sent. The system is done."
**Why it signals a gap:** The candidate does not check delivery receipts. Invalid tokens are never cleaned up. Delivery failures are never retried. The system has no way to improve.

### Red Flag 8: Ignoring Cost
**What the candidate says:** "Send all notifications through all channels."
**Why it signals a gap:** The candidate does not consider the cost of SMS or email. At scale, notification costs can reach millions of dollars per month. Cost optimization is a fundamental requirement.

## Practice Drills

### Drill 1: The 2-Minute Explain
**Prompt:** "Explain how a push notification system works as if I have never heard of it."

**What to cover:**
1. The event triggers the notification (one sentence)
2. The event is queued for processing (one sentence)
3. The system checks user preferences (one sentence)
4. The system sends to the provider (APNs/FCM) (one sentence)
5. The provider delivers to the device (one sentence)
6. The system handles the response (one sentence)

### Drill 2: The 5-Minute Design
**Prompt:** "You are building a notification system for a food delivery app. The app needs to send order status updates (confirmed, preparing, out for delivery, delivered) to customers via push and SMS. Design the notification architecture."

**What to cover:**
1. Scale estimate: 1 million orders per day × 4 status updates = 4 million notifications per day
2. Channel selection: push for active app users, SMS for critical updates
3. Queue: Kafka topic for order notification events
4. Router: preference check, dedup, template rendering
5. Delivery: push workers for FCM/APNs, SMS workers for Twilio
6. Feedback: delivery receipts, token cleanup

### Drill 3: The 10-Minute Deep Dive
**Prompt:** "Design a global notification platform for a social media company with 200 million users. The platform must support push, email, SMS, and in-app notifications. The platform must handle 10x traffic spikes during major events. The budget is $500,000 per month for notification infrastructure."

**What to cover:**
1. Requirements clarification (scale, channels, SLA, regions)
2. Scale estimation (200M users × 50 events/day = 10B events/day, 115k events/s, 460k peak)
3. System design (5 layers: ingress, dispatch, router, delivery, feedback)
4. Multi-region architecture (US, EU, AP regions with active-active Kafka)
5. Cost optimization (SMS only for critical, batch email, push default)
6. Traffic spike handling (over-provision ingress, 2-hour buffer in queue, auto-scaling workers)
7. Monitoring and alerts (delivery rate, consumer lag, circuit breaker, cost per channel)
8. Failure scenarios (provider outage, rate limit, token explosion)
