## Tier 1 — Foundation

### Challenge 1: Explain the Notification Pipeline

**Prompt:** You are interviewing a junior engineer who just joined your team. Explain the notification system pipeline in 5 minutes. Include the purpose of each layer, the data flow between layers, and what breaks if a layer is missing.

**Time limit:** 5 minutes

**Evaluation criteria:**
- Names all 5 layers: ingestion, dispatch, routing, delivery, feedback
- Explains the purpose of each layer in one sentence
- Describes the data flow: event → validate → enrich → queue → route → deliver → acknowledge
- Identifies what breaks without each layer
- Provides a concrete example (a like notification flowing through the pipeline)

**Common mistakes:**
- Forgetting the feedback layer (most common omission)
- Describing the layers as isolated components without showing the data flow
- Not explaining why the queue is necessary (decoupling, buffering)
- Confusing the dispatch queue with the channel queues

**Model answer:**

"The notification pipeline has 5 layers. First, the ingestion layer accepts events from application services. It validates the event format and enriches it with user context. Without this layer, malformed events would crash downstream services.

Second, the dispatch queue stores the events durably. I use Kafka for this. The queue decouples event production from delivery. Without it, a spike in event production would overwhelm the delivery system, and a provider outage would cause data loss.

Third, the routing layer processes events from the queue. It checks for duplicates, renders templates, and selects delivery channels. Without this layer, the system would not respect user preferences, and users would receive unwanted notifications.

Fourth, the delivery layer sends notifications through provider-specific workers. Each channel has its own queue and worker pool. Without this layer, the system would couple all channels together, and a failure in one channel would block others.

Fifth, the feedback layer processes delivery receipts. It updates notification status, removes invalid tokens, and triggers retries. Without this layer, the system would be blind to delivery failures and would continue sending to expired tokens."

### Challenge 2: Deduplication Strategy

**Prompt:** A user is receiving duplicate notifications. They liked a post, then unliked it, then liked it again. They received 3 separate notifications instead of 1. Explain how you would prevent this.

**Time limit:** 5 minutes

**Evaluation criteria:**
- Identifies the need for deduplication
- Explains the dedup key: `recipient_id + notification_type + resource_id`
- Describes the time window: dedup within 30 seconds to 5 minutes
- Mentions the Bloom filter approach with controlled false positive rate
- Explains the exact-match fallback for critical notifications
- Addresses the race condition: two concurrent requests passing the dedup check

**Common mistakes:**
- Suggesting dedup only at the provider level (providers do not dedup)
- Not considering the time window (permanent dedup would prevent legitimate re-notifications)
- Not addressing the concurrent request race condition

**Model answer:**

"Duplicate notifications arise from the same event being processed twice. This can happen when a user likes, unlikes, and likes again — the system should treat this as one notification, not three.

I implement dedup at the router layer. The dedup key is a composite of recipient ID, notification type, and resource ID. For the like example, the key is `like:user_789:photo_456`. The time window is 30 seconds — if the same key appears within 30 seconds, the notification is dropped.

I use a Bloom filter in Redis for the dedup store. The Bloom filter has a 0.1 percent false positive rate with 7 hash functions. The memory usage is 10 bits per element. For a system processing 1 million notifications per day, the Bloom filter uses 1.2 MB.

For critical notifications (password resets, payment receipts), I add an exact-match check in Redis. The exact-match store is a Redis set with a 60-second TTL. This eliminates the false positive risk for notifications where a duplicate would cause a serious problem.

The race condition of two concurrent requests passing the dedup check is handled by a distributed lock in Redis. The lock key is `lock:like:user_789:photo_456` with a 5-second TTL. The first request acquires the lock and sends the notification. The second request fails to acquire the lock and skips."

### Challenge 3: Rate Limiting Design

**Prompt:** Your notification system is being rate-limited by APNs. APNs allows 600 requests per second per connection. Your system is sending 1,000 requests per second. Design the rate limiting solution.

**Time limit:** 5 minutes

**Evaluation criteria:**
- Identifies the need for per-provider rate limiting
- Explains the token bucket algorithm
- Shows how to scale: multiple connections with each handling 600 req/s
- Discusses what happens when the rate limit is exceeded (queue, backoff, alert)
- Mentions per-user rate limits in addition to per-provider limits

**Common mistakes:**
- Only implementing per-user rate limiting (ignoring provider limits)
- Suggesting synchronous throttling (blocking the worker)
- Not accounting for the provider's rate limit window

**Model answer:**

"APNs allows 600 requests per second per HTTP/2 connection. I need to stay within this limit while handling 1,000 requests per second. The solution is two-fold: use multiple connections and implement a token bucket rate limiter.

I open 2 APNs connections. Each connection handles 600 requests per second, giving a total capacity of 1,200 requests per second. This provides headroom for bursts. The connection pool is managed by the push worker service.

Each connection has a token bucket rate limiter. The bucket has a capacity of 600 tokens and refills at 600 tokens per second. When a notification is ready to send, the worker acquires a token from the bucket for the appropriate connection. If no token is available (all connections are at capacity), the notification is re-queued with a 100ms delay.

I also implement per-user rate limiting. Each user can receive a maximum of 5 push notifications per minute. This is enforced by a separate token bucket keyed by `user_id:push`. The per-user limit is checked before the per-connection limit. If the user has exceeded their limit, the notification is deferred, not dropped.

When the rate limit is exceeded, the system logs the event and increments a `rate_limit.hit` metric. The circuit breaker is configured to open if the rate limit error rate exceeds 5 percent. The operations team is alerted if the rate limit hit rate exceeds 10 percent of total notifications."

## Tier 2 — Design

### Challenge 1: E-Commerce Notification System

**Prompt:** Design a notification system for an e-commerce platform that supports 10 million monthly active users. The platform needs to send the following notifications:
- Order confirmation (immediate, high priority)
- Shipping update (immediate, medium priority)
- Abandoned cart reminder (1 hour delay, low priority)
- Weekly promotional email (scheduled, batch)

The system must support push, email, and SMS channels. The email channel must have a 99.9 percent delivery rate. The SMS channel is only for order confirmations and shipping updates.

**Time limit:** 15 minutes

**Scale estimates:**
- 10 million MAU = roughly 3 million DAU
- 1 order per user per month = 10 million orders per month = 333,333 orders per day
- 4 notifications per order = 1.3 million notifications per day
- At peak, 4x average = 60 notifications per second
- Email: 70 percent of notifications = 900,000 per day
- Push: 25 percent = 330,000 per day
- SMS: 5 percent = 66,000 per day

**Key decisions:**
- Use Kafka as the dispatch queue (3 partitions, RF=3)
- Use separate topics for each priority level (immediate, delayed, batch)
- Order confirmations and shipping updates go to the immediate topic
- Abandoned cart reminders go to the delayed topic (scheduled for 1 hour later)
- Weekly promotional emails go to the batch topic (processed by a cron job)

**What a strong answer covers:**
- The preference store: users can opt out of promotional emails but keep order notifications
- The SMS fallback: if push delivery fails for an order confirmation, fall back to SMS
- The batching strategy: promotional emails are batched by recipient with a 5-minute window
- The monitoring: track delivery rate per notification type, alert if order confirmation delivery drops below 99.9 percent

**What a weak answer misses:**
- User preference check (sending promotional emails to opted-out users)
- SMS cost management (SMS costs $0.0075 per message — 66,000 SMS/day = $495/day)
- The abandoned cart delay mechanism (how to schedule a notification for 1 hour later)

**Model answer:**

"Let me start with the scale. 10 million MAU translates to roughly 3 million DAU. Each user makes 1 order per month, so 333,333 orders per day. Each order generates 4 notifications, so 1.3 million notifications per day. At peak, roughly 60 notifications per second. This is a modest scale.

The system has 5 layers. For the ingestion layer, a simple REST API endpoint receives notification events. The event schema includes the type, recipient, priority, and channel preferences.

The dispatch queue is Kafka with 3 partitions and replication factor 3. I use 3 Kafka topics: `immediate` for order confirmations and shipping updates, `delayed` for abandoned cart reminders, and `batch` for promotional emails.

The router layer checks user preferences. Order confirmations are always sent. Promotional emails are only sent if the user has opted in. The router also checks quiet hours — if it is 11 PM in the user's timezone, non-critical notifications are deferred.

For channel selection, order confirmations are sent via push and SMS. Shipping updates are sent via push and email. Abandoned cart reminders are sent via email. Promotional emails are sent via email only.

The delivery layer has separate workers for push, email, and SMS. The push workers use FCM (since it handles both Android and iOS through a single API). The email workers use SendGrid. The SMS workers use Twilio, but only for order confirmations and shipping updates.

The feedback layer processes delivery receipts. For order confirmations, if the push delivery fails, the system falls back to SMS. If SMS also fails, the system sends an email. The notification is not considered delivered until at least one channel succeeds.

For cost management, SMS is limited to 66,000 messages per day at $0.0075 each = $495 per day. To reduce this, I would explore using push as the primary channel and SMS only as a fallback for failed push deliveries."

### Challenge 2: Multi-Region Notification Hub

**Prompt:** Design a notification system for a global messaging app with 50 million daily active users. Users are spread across North America (40 percent), Europe (30 percent), Asia (25 percent), and other regions (5 percent). The app sends notifications for new messages, calls, and friend requests. The system must have under 500ms notification delivery latency for real-time messages.

**Time limit:** 15 minutes

**Scale estimates:**
- 50 million DAU
- 100 messages per user per day = 5 billion messages per day
- 1 notification per message = 5 billion notifications per day
- Peak: 4x average = 230,000 notifications per second
- 4 regions: US (40%), EU (30%), AP (25%), others (5%)

**Key decisions:**
- 3 active regions: US-East, EU-West, AP-Southeast
- Each region has its own Kafka cluster
- User preferences are replicated using DynamoDB Global Tables
- Device tokens are stored per-region (users register with the nearest region)
- DNS-based routing (Route53 latency records) directs users to the nearest region

**What a strong answer covers:**
- The 500ms latency requirement means the router and delivery must be in the same region as the user
- Cross-region notification delivery is not acceptable for real-time messages
- The dedup Bloom filter is not replicated across regions (accepting cross-region duplicates)
- Failover: if a region fails, traffic is rerouted to the next-nearest region
- The Kafka clusters are regional — no cross-region Kafka replication

**What a weak answer misses:**
- The latency budget (500ms end-to-end means each layer has a tight budget)
- The data replication complexity (user preferences must be replicated, but device tokens are regional)
- The failover capacity planning (each region must be over-provisioned by 50 percent to handle failover)

**Model answer:**

"50 million DAU generating 5 billion notifications per day is a significant scale. At peak, 230,000 notifications per second. The 500ms latency requirement means the system must process and deliver notifications within the same region.

I deploy in 3 active regions: US-East, Europe (EU-West), and Asia (AP-Southeast). Each region has its own Kafka cluster, router service, and delivery workers. The ingress gateway in each region accepts events from local application services.

User preferences are stored in DynamoDB Global Tables, which provides active-active replication across regions with sub-second latency. When a user updates their preferences in the US, the change is propagated to EU and AP within 1 second.

Device tokens are stored per-region. When a user's device starts, it connects to the nearest region via DNS latency-based routing. The device registers its push token with the local token registry. Notifications for the user are sent from the region where the event was generated.

The 500ms latency budget is allocated as follows: event ingestion (50ms), enrichment (100ms), Kafka write (20ms), queue read (20ms), routing (50ms), delivery to provider (200ms), provider to device (60ms). The total is 500ms.

For failover, each region is over-provisioned by 50 percent. If US-East fails, Route53 health checks detect the failure after 3 consecutive failures (30 seconds). Traffic is rerouted to EU-West and AP-Southeast. The surviving regions each handle 50 percent of the US traffic. The consumer lag may increase to 10 seconds during the failover, but no notifications are lost.

The dedup Bloom filter is not replicated across regions. This means a user who triggers events in both the US and EU within 30 seconds might receive duplicate notifications. This is an acceptable trade-off — the scenario is rare, and the impact is minor."

### Challenge 3: Notification Budget and Cost Optimization

**Prompt:** Your company's notification costs have tripled in the last quarter. The monthly bill is $1.5 million. The CTO asks you to reduce notification costs by 40 percent while maintaining a 99 percent delivery rate for critical notifications. Design the cost optimization strategy.

**Time limit:** 15 minutes

**Cost breakdown (assumptions):**
- SMS: $0.0075 per message, 50 million messages per month = $375,000
- Email: $0.0003 per message, 300 million emails per month = $90,000
- Push: $0.00001 per message, 1 billion pushes per month = $10,000
- Infrastructure: $1,025,000 (Kafka, Redis, compute, network)

**Key decisions:**
- Reduce SMS usage by 50 percent (switch to push as primary, SMS as fallback only)
- Implement email batching (reduce email count by 40 percent)
- Eliminate push notifications for inactive users (users who have not opened the app in 30 days)
- Implement a notification budget per user (max 50 notifications per day across all channels)

**What a strong answer covers:**
- The cost breakdown by channel (SMS is 25 percent of cost but only 3 percent of volume)
- The user-level notification budget (reduces total volume without affecting engagement)
- The inactive user optimization (sending to inactive users is wasted spend)
- The batching strategy (reduces email volume without reducing delivery)
- The monitoring and analytics (track cost per user, cost per notification type)

**What a weak answer misses:**
- The impact on user experience (aggressive cost cutting reduces engagement)
- The critical notification exception (SMS must still be used for critical notifications)
- The infrastructure cost (the largest cost component needs independent optimization)

**Model answer:**

"The $1.5 million monthly bill has 3 components: SMS ($375k), email ($90k), push ($10k), and infrastructure ($1,025k). The infrastructure cost is the largest, but the per-message costs are where we can make the biggest impact.

Step 1: Reduce SMS usage by 50 percent. SMS is currently the default channel for order confirmations. I will change the default to push, with SMS as a fallback only when push delivery fails. This saves $187,500 per month. The impact on delivery is minimal because push delivery succeeds 95 percent of the time for active users.

Step 2: Implement email batching. Currently, each notification triggers a separate email. I will batch emails by recipient with a 5-minute window. This reduces the email count by 40 percent, saving $36,000 per month. The batching window is configurable — users can choose between immediate delivery (no batching) and batched delivery (every 5 minutes).

Step 3: Eliminate push notifications for inactive users. Users who have not opened the app in 30 days have a push notification open rate under 1 percent. Sending them push notifications is wasted. I will suppress push notifications for inactive users and switch them to email. This reduces push volume by 20 percent, saving $2,000 per month.

Step 4: Implement a per-user notification budget. Each user can receive a maximum of 50 notifications per day across all channels. This is enforced by the preference checker. The budget is configurable per notification type — critical notifications are exempt from the budget. This reduces total notification volume by 15 percent, saving $150,000 per month.

Step 5: Optimize infrastructure costs. The Kafka cluster is over-provisioned. I will right-size the cluster from 10 brokers to 6 brokers, saving $400,000 per month. The Redis cluster can be reduced from 6 nodes to 4 nodes, saving $100,000 per month.

Total savings: $187,500 + $36,000 + $2,000 + $150,000 + $500,000 = $875,500 per month (58 percent reduction). The per-user notification budget is the most impactful optimization — it reduces costs while improving user experience by reducing notification fatigue."

## Tier 3 — Expert

### Challenge 1: Real-Time Notification Delivery at Planetary Scale

**Prompt:** You are the principal engineer at a global messaging platform with 1 billion users. The platform needs a notification system that delivers messages to users within 100ms end-to-end, supports 1000+ languages, and operates in 200+ countries. The system must handle 10x traffic spikes during major events (New Year's Eve, World Cup finals). The system must comply with data residency regulations in the EU (GDPR), China (data localization), and India (data localization). The team has 50 engineers across 4 timezones.

Design the notification system architecture. Discuss the technical challenges, regulatory constraints, and organizational approach.

**Time limit:** 30 minutes

**Dimensions a strong answer explores:**

**Technical:**
- Edge-based notification delivery (servers in 100+ locations, not just 3 regions)
- Protocol choice: WebSocket for real-time delivery, HTTP/2 for push notifications
- The "fan-out at the edge" pattern: broadcast the notification to all edge nodes, let the edge node closest to the user deliver it
- The gossip protocol for notification state synchronization across edge nodes
- The latency budget: 100ms end-to-end means 10ms for processing, 90ms for network transit

**Regulatory:**
- GDPR: user data must stay in the EU. EU users' notification preferences and device tokens cannot leave the EU
- China: data must be stored on servers in China. The notification system must have a separate deployment in China with a Chinese cloud provider (Alibaba Cloud, Tencent Cloud)
- India: data must be stored on servers in India. The notification system must have a separate deployment in India
- The solution: region-specific deployments with no cross-region data movement for user data. Aggregated analytics can cross regions (no PII)

**Organizational:**
- Team structure: 3 sub-teams (core platform, regulatory compliance, developer experience)
- The core platform team owns the ingestion, routing, and delivery components
- The regulatory compliance team owns the region-specific deployments and data handling
- The developer experience team owns the API, SDK, documentation, and onboarding
- The platform must support self-service onboarding for new product teams

**Anti-patterns a weak answer shows:**
- Suggesting a single global deployment (violates data residency regulations)
- Ignoring the 100ms latency requirement (the system must be designed for this from the start)
- Not addressing the 10x traffic spike (the system will fail on New Year's Eve)
- Proposing a centralized architecture (latency from a central location to remote users is too high)

**Discussion of multiple valid approaches:**

**Approach A: Edge-based delivery (Cloudflare-style)**
- Deploy notification servers in 100+ edge locations
- Each edge node has a local notification queue
- Notifications are broadcast to all edge nodes via a publish-subscribe mesh
- The edge node closest to the user delivers the notification
- Pros: lowest latency (15ms), best compliance (data stays in the region)
- Cons: complex operational model, 100+ deployments to manage, higher infrastructure cost

**Approach B: Regional hubs (AWS-style)**
- Deploy in 10 regional hubs (US-East, US-West, EU-West, EU-Central, AP-South, AP-Southeast, AP-Northeast, South America, Middle East, Africa)
- Each hub serves a group of nearby countries
- Cross-region replication for user preferences only
- Pros: simpler operations (10 deployments vs 100+), moderate latency (50ms)
- Cons: higher latency for edge countries (a user in Argentina connecting to US-East), partial compliance (some data must still leave the country)

**Approach C: Hybrid (edge + regional)**
- Use regional hubs for the core pipeline (ingestion, routing, delivery workers)
- Use edge nodes for the last-mile delivery (push to device, WebSocket connection)
- The edge node receives the notification from the regional hub and delivers it to the device
- Pros: best of both worlds, moderate operational complexity
- Cons: requires coordination between edge and regional layers

### Challenge 2: The Notification Slippery Slope

**Prompt:** Your company's user engagement team wants to increase the number of notifications sent to users. They believe that more notifications will increase user engagement and revenue. The data shows that users who receive 50 notifications per day are 2x more likely to make a purchase than users who receive 5 notifications per day.

The product team wants to increase the default notification frequency from 5 per day to 50 per day. The engineering team is concerned about the infrastructure cost and user experience.

Design the system and process that allows the company to increase notification frequency while maintaining user trust, controlling costs, and providing an escape hatch for users who feel overwhelmed.

**Time limit:** 30 minutes

**Dimensions a strong answer explores:**

**Technical:**
- The notification budget system: each user has a configurable daily budget. The budget is enforced by the router layer
- The budget allocation: different notification types (transactional, promotional, social) have different budget pools
- The budget rollover: unused budget from one day can be rolled over to the next day (up to a maximum)
- The budget escalation: users can escalate their budget, but the escalation must be confirmed by the user every 30 days
- The cost tracking: each notification type has a cost, and the budget system tracks the cost per user

**User Experience:**
- The notification preference center: users can see their notification budget, usage, and delivery history
- The notification digest: users can choose to receive a daily or weekly digest instead of real-time notifications
- The notification pause: users can pause all non-critical notifications for 1 hour, 1 day, or 1 week
- The notification feedback: users can rate notifications as "useful" or "not useful." The system learns from this feedback
- The notification fatigue detection: if a user has not opened any notifications in 7 days, the system automatically reduces their budget

**Business:**
- The A/B testing framework: roll out the increased frequency to 10 percent of users, measure engagement and churn
- The guardrail metrics: churn rate, notification opt-out rate, and user satisfaction score
- The cost-benefit analysis: the increased notification volume costs $X per month, and the expected revenue increase is $Y per month. The feature only launches if Y > 1.5X
- The sunset clause: if the guardrail metrics degrade by more than 5 percent in 30 days, the feature is rolled back

**Anti-patterns a weak answer shows:**
- Blindly increasing notification frequency (the data shows correlation, not causation)
- Not providing a user escape hatch (users have no way to reduce their notification load)
- Ignoring the cost increase (50x more notifications = 50x higher infrastructure cost)
- Not measuring the impact on user satisfaction (engagement metrics alone are misleading)

**Model answer:**

"The data shows a correlation between notification frequency and purchase likelihood, but correlation is not causation. Users who are highly engaged with the platform are more likely to both receive notifications and make purchases. The question is whether increasing notifications for less engaged users will increase their engagement or drive them away.

My approach is a controlled experiment with guardrails. The system implements a per-user notification budget. The current budget is 5 notifications per day for promotional notifications. I will increase the budget to 50 per day for a test group of 10 percent of users.

The experiment has 3 guardrail metrics: (1) churn rate — users who leave the platform within 30 days, (2) notification opt-out rate — users who disable notifications entirely, and (3) user satisfaction score — measured by a weekly survey. If any guardrail metric degrades by more than 5 percent, the experiment is paused.

The system provides a notification preference center where users can see their notification budget and usage. Users can set their own budget limit — the default is 50, but they can lower it to 5. The preference center also shows which notification types are consuming the most budget.

The cost increase is significant. At 50 notifications per user per day, the total volume is 50x the current volume. The infrastructure cost scales linearly with volume. The incremental cost is $500,000 per month. The expected revenue increase must exceed $750,000 per month (1.5x the cost) to justify the change.

The notification fatigue detection system monitors user engagement. If a user has not opened any notifications in 7 days, the system automatically reduces their budget to 5 notifications per day. If the user opens notifications again, the budget is gradually increased over 30 days.

The sunset clause: if the experiment does not show a positive return on investment within 60 days, the feature is rolled back. The rollback is automatic — the system reverts the budget to 5 notifications per day for all users.
