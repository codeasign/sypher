### What Breaks at Each Scale

| User Count | Symptoms | Root Cause |
|-----------|----------|------------|
| 1K | Nothing | Single server handles 50 QPS easily |
| 10K | Occasional slow queries at peak (30 reads per second)| Query cache thrashing, working set exceeds RAM |
| 100K | Connection errors, p99 latency over 500ms | Connection pool full, disk I/O saturation |
| 1M | Partial outages during traffic spikes, backup takes 4 hours | Single primary write bottleneck, table scans |
| 10M | Reads timeout, replication lag of minutes, OOM crashes | Cannot scale writes vertically any further |

The naive architecture fails because it provides no separation of concerns: reads compete with writes for the same connections, the same cache, and the same disk I/O. A single slow query (full table scan on a 50GB table) evicts the entire hot working set from the buffer pool, causing every subsequent query to hit disk.

### Component Responsibilities

**Primary (SQL)**: Handles all writes and strongly-consistent reads. Single writer prevents write conflicts. WAL ships to replicas asynchronously.

**Read Replicas**: Serve read-only queries. Three replicas provide redundancy — if one fails, the load balancer directs traffic to the other two. Replicas also serve as failover targets: if the primary crashes, promote a replica to primary.

**Connection Pooler (PgBouncer / ProxySQL)**: Front-end connection management. Without a pooler, each app connection creates a backend process on the database server, consuming memory. PgBouncer multiplexes thousands of app connections over a small pool of database connections.

**NoSQL (Cassandra)**: Handles write-heavy workloads that do not need transactions. User sessions, event logs, and time-series data live here. Each node accepts writes, so there is no write bottleneck.

**Redis Cache**: Caches hot data — product listings, user profiles, rate limit counters. TTL-based eviction ensures stale data is removed automatically. Redis sits in front of both SQL and NoSQL, intercepting the hottest queries.

**Monitoring Stack**: Tracks query latency, connection counts, replication lag, disk usage, and error rates. Alerts on p99 latency exceeding 100ms or replication lag exceeding 30 seconds.

## Integration Patterns

### Pattern 1: SQL Primary + Read Replicas

```python
# Application-level read/write split
def get_order(order_id):
    # Writes go to primary
    db.execute("UPDATE orders SET status = 'viewed' WHERE id = %s",
               (order_id,), using='primary')
    
    # Reads go to replica (accepts 1s stale data)
    result = db.execute("SELECT * FROM orders WHERE id = %s",
                        (order_id,), using='replica')
    return result
```

**When to use**: Applications with a clear read/write ratio (80% reads, 20% writes) and tolerance for slightly stale reads on replicas.

**Drawbacks**: Replication lag can cause read-your-write inconsistency — you update an order, then read the old version from a replica. Fix with session stickiness (route the same user to the same replica) or read-after-write consistency (read from primary for N seconds after a write).

### Pattern 3: Write-Around Cache with Redis

Hot data is cached in Redis. Writes go to the primary database and invalidate the cache. Reads check Redis first, then fall through to the primary if missing.

```python
def get_product(product_id):
    # Try cache first
    cached = redis.get(f"product:{product_id}")
    if cached:
        return json.loads(cached)
    
    # Cache miss — fetch from primary, populate cache
    product = db.execute(
        "SELECT * FROM products WHERE id = %s", (product_id,)
    )
    redis.setex(f"product:{product_id}", 300, json.dumps(product))
    return product
```

**Cache invalidation strategies:**
- **TTL-based**: Set expiration (300 seconds). Simple but can serve stale data.
- **Write-through**: Update cache on every write. Consistent but higher write latency.
- **Write-behind**: Update database, send async invalidation event. Lower latency, temporary inconsistency.
- **Cache-aside (above)**: Cache populated on read miss. Application invalidates on writes.

### Pattern 4: Transactional Outbox

When you need atomic updates across SQL and NoSQL (write to PostgreSQL AND publish to Kafka), use the outbox pattern:

```sql
-- In the same SQL transaction
BEGIN;
  INSERT INTO outbox (aggregate_id, event_type, payload)
  VALUES ('order-42', 'order.created', '{"id": 42, "total": 2900}');
  
  UPDATE orders SET status = 'created' WHERE id = 42;
COMMIT;

-- Background worker reads outbox and publishes to Kafka / NoSQL
-- Kafka consumer updates Cassandra read model
```

The outbox table ensures at-least-once delivery: the background worker polls the table, publishes events, and deletes processed rows. If the worker crashes, it picks up where it left off on restart.

## Multi-Region and Global Scale

For applications with users across continents, a single-region database adds 100-300ms of latency to every query from distant regions. Multi-region deployment is required.

### DynamoDB Global Tables

DynamoDB Global Tables provide active-active replication with last-writer-wins conflict resolution. Replication happens within seconds, typically under 1 second across US regions, 1-2 seconds across continents.

**Practical considerations:**
- Strongly consistent reads are only available within a single region. Cross-region reads are eventually consistent.
- Each write incurs replication cost: 2x write capacity units for global tables.
- Conflict resolution is automatic but destructive — the last write always wins, even if it is logically incorrect.
- Application-level conflict resolution requires conditional writes and a custom reconciliation layer.

## Capacity Planning

### Estimating Required QPS

For a social application with 10 million daily active users:

1. **Daily peak QPS**: Assume 10% of DAU active at peak hour. Each user makes 50 requests.
   - Peak hour requests: 10M * 10% * 50 = 50M requests/hour
   - Peak QPS: 50M / 3600 = ~14,000 QPS
   - Safety factor: 2x headroom = 28,000 QPS target

2. **Read/write split**: 90% reads, 10% writes.
   - Read QPS: 25,200 QPS
   - Write QPS: 2,800 QPS

3. **Database sizing**: Each read hits 1 row (2KB), each write writes 1 row.
   - Read throughput: 25,200 * 2KB = 50 MB/s (negligible — network handles it)
   - Write throughput: 2,800 * 2KB = 5.6 MB/s (fine for a single node with SSD)

4. **Storage for 3 years**:
   - Each user generates 100 writes/day (posts, likes, comments, session updates)
   - Daily write volume: 10M * 100 * 2KB = 2 TB/day
   - 3 years: 2 TB * 365 * 3 = 2,190 TB = ~2.2 PB
   - With 3x replication: 6.6 PB

5. **Nodes required** (Cassandra):
   - Storage per node: 4TB SSD
   - Effective capacity per node (with replication factor 3): 4TB / 3 = 1.33 TB
   - Raw storage needed: 2.2 PB * 3 = 6.6 PB
   - Nodes needed: 6,600 TB / 4 TB = 1,650 nodes

This is why Netflix runs over 75,000 Cassandra nodes — the storage requirements at scale are enormous.

### Compute Sizing

For a PostgreSQL primary handling 2,800 writes/sec:

| Resource | Calculation | Requirement |
|----------|------------|-------------|
| CPU cores | Each connection uses ~1% CPU. 2,800 writes/sec with 50ms per write needs 140 concurrent connections. Add 2x overhead for query parsing and planning. | 16-32 cores |
| RAM | Buffer pool should hold the active working set. For 10M users with 100KB hot data each: 10M * 100KB = 1TB. But 1TB RAM costs $10K+/month. Realistically: 64GB buffer pool + 32GB OS cache. | 96GB-128GB |
| Disk IOPS | Each write needs WAL fsync (one sequential write), data page write (one random write), index updates (2-3 random writes). 2,800 writes/sec * 4 IOS = 11,200 IOPS. Modern SSDs handle 50K-100K IOPS. | 8K-12K IOPS |
| Network | 2,800 writes/sec * 2KB = 5.6 MB/s. Also read traffic to replicas (WAL shipping). Replication adds 5-20 MB/s per replica. | 1 Gbps |

## Deployment and Operations

### Zero-Downtime Schema Migrations

Schema changes in SQL databases block writes (or reads, depending on the database and migration tool).

**Safe migration strategy for PostgreSQL:**

```sql
-- Step 1: Add column as nullable (no lock)
ALTER TABLE orders ADD COLUMN driver_id BIGINT;

-- Step 2: Backfill in batches (background job, 1000 rows at a time)
UPDATE orders SET driver_id = 0 WHERE driver_id IS NULL LIMIT 1000;

-- Step 3: Add NOT NULL constraint (requires exclusive lock but only for a moment)
ALTER TABLE orders ALTER COLUMN driver_id SET NOT NULL;

-- Step 4: Add index concurrently (non-blocking)
CREATE INDEX CONCURRENTLY idx_orders_driver_id ON orders(driver_id);
```

**NoSQL migration strategy (MongoDB):**

MongoDB's schema-on-read means you can add fields without a migration. Instead, handle both document shapes in application code:

```python
def get_driver_id(order):
    # Handle both old (no driver_id) and new documents
    if 'driver_id' in order:
        return order['driver_id']
    # Old document — return default
    return None
```

Then backfill in the background:

```python
# Backfill script (runs asynchronously, no downtime)
for order in db.orders.find({'driver_id': {'$exists': False}}):
    db.orders.update_one(
        {'_id': order['_id']},
        {'$set': {'driver_id': None}}
    )
```

```sql
-- Current replication lag

-- Cache hit ratio

FROM pg_stat_activity

GROUP BY 1, 2;
```

### Backup and Disaster Recovery

**Recovery Point Objective (RPO)** — how much data you can lose. **Recovery Time Objective (RTO)** — how long to restore.

| Tier | RPO | RTO | Method | Cost |
|------|-----|-----|--------|------|
| Bronze | 1 day | 4 hours | Daily pg_dump to S3 | Low |
| Silver | 1 hour | 30 minutes | WAL archiving to S3 + daily base backup | Medium |
| Gold | 5 minutes | 5 minutes | Streaming replication to standby in different AZ + WAL archiving | High |
| Platinum | 1 second | 1 minute | Synchronous replication across regions + automated failover | Very high |

**Cross-region DR strategy:**
1. Take daily base backups to S3 in the primary region.
2. Copy backups to a secondary region using S3 cross-region replication.
3. Stream WAL to the secondary region continuously (WAL files are small, typically 16MB each).
4. On primary region failure, restore the latest backup in the secondary region and replay WAL to the last known good state.
5. Update DNS CNAME to point at the new primary.

### Data Archival Strategy

As data grows, you need to archive old data to reduce the active dataset size. A common approach is to use a partitioned table with automated archival to S3.

```sql
-- PostgreSQL: partition by month
CREATE TABLE orders (
    id BIGSERIAL,
    created_at TIMESTAMP NOT NULL,
    data JSONB
) PARTITION BY RANGE (created_at);

CREATE TABLE orders_2026_01 PARTITION OF orders
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
CREATE TABLE orders_2026_02 PARTITION OF orders
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

-- Detach and archive partitions older than 6 months
ALTER TABLE orders DETACH PARTITION orders_2026_01;
-- Then dump to S3 and drop the orphaned table
```

For NoSQL databases like Cassandra, TTL-based expiration is built in. Set a `default_time_to_live` on the table:

```sql
CREATE TABLE events (
    event_id UUID,
    created_at TIMESTAMP,
    data TEXT,
    PRIMARY KEY (event_id)
) WITH default_time_to_live = 86400;  -- 24 hours
```

Cassandra automatically removes expired data during compaction. No separate archival process needed.
