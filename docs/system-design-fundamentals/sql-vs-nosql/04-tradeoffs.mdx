# SQL vs NoSQL — Trade-offs and Failure Modes

## Core Tension

The fundamental trade-off between SQL and NoSQL is **consistency versus availability under partition**.

Every distributed database faces a simple reality: the network can fail at any moment. When a partition severs communication between two halves of a cluster, you must choose: stop accepting writes (maintain consistency, sacrifice availability) or accept writes on both sides (maintain availability, risk inconsistency).

SQL databases choose consistency. NoSQL databases choose availability.

Everything else — schema, queries, transactions, joins — follows from this one choice. SQL enforces schema because consistency requires it: inconsistent data shapes break joins and aggregations. NoSQL accepts flexible schema because availability requires it: you cannot reject a write just because an optional field is missing.

## Decision Matrix

| Decision Point | Choose SQL | Choose NoSQL |
|---------------|-----------|-------------|
| Data relationships | Data has many joins and foreign key relationships (orders, customers, payments) | Data is self-contained (documents, key-value pairs, time-series points) |
| Query complexity | Queries are ad-hoc, analytical, or involve aggregations across many records | Queries are simple key lookups or single-collection filters |
| Consistency requirements | You cannot tolerate stale reads (banking, booking, inventory) | You can tolerate seconds of inconsistency (sessions, logs, profiles) |
| Write throughput | 5K-20K writes per second on a single node | 100K+ writes per second, horizontally scalable |
| Schema stability | Schema changes rarely (once per sprint) | Schema changes constantly (A/B tests, feature flags) |
| Data volume | Under 1TB per node (vertical scale works) | 100TB+ (must shard across many nodes) |
| Transaction scope | Two-phase commit or ACID across multiple records | Single-document atomicity is sufficient |
| Team expertise | Team knows SQL well | Team works with JSON objects natively |

### How to Make the Call

When evaluating a new feature or service, ask three questions:

1. **Does this data need to be consistent within milliseconds of a write?** If yes, SQL.
2. **Does this data have relationships I need to query across?** If yes, SQL.
3. **Will this data need to scale beyond a single server's write capacity?** If yes, NoSQL.

If the answer to all three is "yes," you need a distributed SQL database (Spanner, CockroachDB, YugabyteDB) — which gives you SQL semantics with NoSQL-like horizontal scaling but at higher latency and cost.

## CAP Theorem Positioning

The CAP theorem states a distributed database can guarantee at most two of:
- **C**onsistency: all nodes see the same data at the same time
- **A**vailability: every request gets a response (success or failure)  
- **P**artition Tolerance: the system continues operating despite network partitions

In practice, partitions are inevitable (network cables fail, switches drop packets, routers reconfigure), so you choose between CP and AP.

### PACELC Extension

The PACELC theorem extends CAP: if a partition occurs (P), trade off A and C. Otherwise (Else, E), trade off Latency (L) and Consistency (C).

- **Cassandra**: PA/EL — prefers availability during partitions, prefers low latency over consistency normally (tunable).
- **PostgreSQL**: PC/EC — prefers consistency during partitions, prefers consistency over low latency normally.
- **DynamoDB**: PA/EL — same as Cassandra, with tunable consistency on a per-query basis.
- **Spanner**: PC/EC — prefers consistency in all cases, using TrueTime to provide external consistency at the cost of some latency.

## Failure Modes

### Failure 1: Replication Lag and Stale Reads

A read replica falls behind the primary due to a slow query, a compaction burst, or network congestion. Reads return stale data.

**Symptoms**: Users report "I just updated my profile but it still shows the old name." Customer support tickets increase. Revenue is affected if stale data causes incorrect pricing or inventory.

**Root cause**: A read replica is processing a long-running query (maybe a nightly report) that consumes CPU and I/O, starving the WAL apply process.

**Solution**: Set `max_standby_streaming_delay` in PostgreSQL to a safe value (30 seconds). If lag exceeds this, the replica cancels conflicting queries. Add a monitoring alert on replication lag. Route post-write reads to the primary for 5 seconds using session stickiness.

### Failure 2: Compaction Storm (Cassandra)

All nodes in a Cassandra cluster trigger compaction simultaneously after a bulk data load, consuming all available disk IOPS. Write latency spikes from 2ms to 200ms.

### Failure 3: Hot Partition

A single partition key in a NoSQL database receives significantly more traffic than others. In Cassandra, a single node handles all writes and reads for that partition, creating a bottleneck. In DynamoDB, the hot partition throttles requests for the entire table.

**Scenario**: A social app stores notifications keyed by `user_id`. A celebrity user (1M followers) generates 10,000 notifications per second. All traffic hits one partition.

**Root cause**: Poor partition key design. The celebrity's notifications all map to one node.

**Solution**: Design partition keys to distribute load evenly:
- Use a composite key: `(user_id, notification_type)` or `(user_id, date_bucket)`.
- In DynamoDB, use an adaptive capacity auto-scaling policy or add a random suffix to the partition key.
- In Cassandra, choose a partition key that has high cardinality (unique values).

### Failure 4: Deadlock (SQL)

Two transactions hold locks on resources the other needs. Both wait forever (or until a timeout).

```sql
-- Transaction A: locks orders, then users
BEGIN;
UPDATE orders SET status = 'shipped' WHERE id = 42;
-- At this moment, Transaction B runs and locks users
UPDATE users SET last_order_id = 42 WHERE id = 99;
-- Transaction A now waits for users lock — DEADLOCK

-- Transaction B: locks users, then orders
BEGIN;
UPDATE users SET last_order_id = 42 WHERE id = 99;
UPDATE orders SET status = 'shipped' WHERE id = 42;
```

**Detection**: PostgreSQL detects deadlocks within 1 second (deadlock_timeout) and kills one transaction, returning a "deadlock detected" error.

**Prevention**: Always acquire locks in the same order across all transactions. If Transaction A always locks orders before users, deadlocks cannot happen.

### Failure 5: Unbounded Query (SQL)

A user or internal tool runs a query that scans a multi-million-row table without effective filters. The query consumes all available I/O, evicts the working set from the buffer pool, and causes cascading latency increases for all other queries.

```sql
-- The full table scan that kills the database
SELECT * FROM orders ORDER BY created_at DESC;
```

**Solution**:
- Set `statement_timeout` in PostgreSQL (30 seconds default, 5 seconds for web queries).
- Use `pg_stat_activity` to identify long-running queries and kill them.
- Implement query analysis in your connection pooler (PgBouncer or RDS Proxy can cancel queries that exceed time limits).
- For analytics queries, route them to a read replica or a dedicated analytics database (Redshift, ClickHouse).

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: Everything in MongoDB

You store financial transactions, user accounts, inventory counts, and relationships in MongoDB because "it is faster to develop." You find that:
- You cannot run a transaction across two collections (MongoDB transactions exist but perform poorly and have limited isolation).
- You re-implement joins in application code — and get them wrong (N+1 queries, inconsistent data).
- You have no foreign key constraints, so a user_id in the orders collection can point to a deleted user.
- Your aggregation pipeline (`$lookup`, `$group`) is slow because MongoDB's optimizer is not designed for analytical queries.

**Fix**: Use PostgreSQL for transactional data. Use MongoDB for documents that fit the document model (catalogs, content, profiles).

### Anti-Pattern 2: PostgreSQL for Time-Series at 100K Writes/sec

PostgreSQL can handle 5K-15K writes per second on a single node. At 100K writes per second:
- WAL write bottleneck: a single WAL writer cannot keep up.
- MVCC bloat: every update creates a new row version. `VACUUM` cannot keep up with dead tuples.
- Index bloat: B-tree indexes grow faster than they can be maintained.

**Fix**: Use TimescaleDB (PostgreSQL extension for time-series), Cassandra, or InfluxDB.

### Anti-Pattern 3: SELECT * on 50 Million Rows

You write `SELECT * FROM users` in a migration script to process all users. The database spends 5 minutes assembling a 10GB result set. The connection times out. You run it again. And again.

**Fix**: Use server-side cursors (`DECLARE cursor CURSOR FOR`) or batch processing with `LIMIT`/`OFFSET`:

```sql
-- Process 1,000 rows at a time
SELECT * FROM users WHERE id > :last_id ORDER BY id LIMIT 1000;
```

### Anti-Pattern 4: MongoDB without Indexes

You create a MongoDB collection for order data. You query by `order_id`, `customer_id`, and `status`, but you never create indexes. Every query does a full collection scan on a 100GB collection. p99 latency is 3 seconds.

**Fix**: Analyze your query patterns, then create indexes:

```javascript
// Create indexes for the three query patterns
db.orders.createIndex({ order_id: 1 });    // exact match
db.orders.createIndex({ customer_id: 1 }); // exact match
db.orders.createIndex({ status: 1, created_at: -1 }); // filter + sort
```

### Anti-Pattern 5: Multi-Leader Replication without Conflict Resolution

You set up multi-leader replication in MySQL (two primaries, bidirectional replication) to handle writes in two data centers. You do not implement conflict resolution. Two users update the same row simultaneously in different data centers — the row diverges, and replication breaks.

**Fix**: Use single-primary replication with read replicas in secondary regions. If you need multi-primary, use a database designed for it (Cassandra, DynamoDB) or implement application-level conflict resolution with CRDTs or version vectors.

## When NOT to Use This

### Do not use SQL when:

- You need **unlimited horizontal write scalability** and are willing to sacrifice joins and transactions. Cassandra is better.
- You need **sub-millisecond reads** on a dataset larger than RAM and the reads are key-value only. Use Redis with disk persistence or use Aerospike.
- Your data is **large blobs** (images, video). Object storage (S3) with a metadata database is better than storing blobs in either SQL or NoSQL.
- You need **full-text search** with faceting, scoring, and fuzzy matching. Elasticsearch is better.

### Do not use NoSQL when:

- You need **ACID transactions** across multiple records. Banking, accounting, and inventory systems need SQL.
- You run **ad-hoc analytical queries** with GROUP BY, window functions, and complex filters. SQL databases have query planners optimized for this.
- Your data has **complex relationships** that you need to traverse (friends-of-friends, org charts). Graph databases (Neo4j) or SQL with recursive CTEs are better.
- You have a **small dataset** (under 10GB) and a small team. A single PostgreSQL instance with no sharding solves the problem with much less operational complexity.

### The Polyglot Persistence Sweet Spot

Most production systems use multiple databases:

- **PostgreSQL**: Orders, accounts, financial data (needs ACID, normalizable).
- **Redis**: Sessions, rate limits, cache, leaderboards (needs speed, TTL).
- **Cassandra**: Event logs, time-series, notifications (needs write throughput).
- **Elasticsearch**: Search, analytics, logs (needs full-text search).
- **S3**: User uploads, backups, static assets (needs cheap blob storage).

Each database does what it is good at. The trade-off is operational complexity — you need expertise in each one, and you need infrastructure (Kubernetes, Terraform) to manage them.
