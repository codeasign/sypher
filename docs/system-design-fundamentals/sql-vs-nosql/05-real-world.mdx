# SQL vs NoSQL in Production

## Company Case Studies

### Netflix — Cassandra as the Backbone

Netflix migrated from Oracle to Cassandra in 2011 when their Oracle RAC cluster could not handle the write throughput required for their growing streaming business. Each user interaction — play, pause, search, scroll — generates write events. At 200 million subscribers, that is billions of writes per day.

**Netflix lessons learned:**
- **Embrace eventual consistency for high-scale writes**. User history events do not need to be immediately consistent; they need to be absorbed without dropping data.
- **Cassandra operations at scale require dedicated SRE teams**. Netflix has a team of 20+ engineers managing their Cassandra infrastructure. They built Priam (automated backup/restore), Astyanax (client library), and Sidecar (monitoring agent).
- **Compaction is the biggest operational challenge**. Netflix uses size-tiered compaction with carefully tuned thresholds to prevent compaction storms.
- **Do not use Cassandra for data that needs joins or transactions**. Netflix keeps financial data in MySQL (now migrated to RDS) and uses Cassandra for everything else.

### Uber — Polyglot Persistence at Global Scale

Uber's architecture evolved through three phases as they grew from a single-city dispatch service to a global logistics platform.

**Phase 1 (2010-2014): PostgreSQL monolith.**
Everything in a single PostgreSQL database — trips, drivers, riders, payments. As they expanded to 100+ cities, the single database could not keep up. Writes to the trips table during surge pricing caused lock contention that crashed the system.

**Phase 2 (2014-2018): Decompose into microservices with per-service databases.**
- **Trip service**: Schemaless (Uber's custom MySQL sharding layer) for trip data — needs high write throughput, simple queries by trip UUID.
- **Payment service**: PostgreSQL for transactions — needs ACID, complex financial operations.
- **Rider/driver profiles**: Cassandra for profile data — each profile is self-contained, high read volume, horizontal scale.
- **Geospatial index**: Redis with sorted sets for real-time driver location — sub-millisecond radius queries.

**Phase 3 (2018-present): DocumentDB and distributed SQL.**
Uber moved most of their Schemaless workloads to Amazon DocumentDB (MongoDB-compatible). They use Apache Hudi (Hadoop) for analytics and are experimenting with CockroachDB for distributed SQL.

**Uber's key lesson**: Let each service pick the right database. A trip record is different from a payment transaction. Forcing one database to handle both creates operational pain that exceeds the cost of running two databases.

**Discord's key lessons:**
- **MongoDB sharding fails for uneven workloads**. Discord's shard key was server_id, but popular servers had 100x more messages. MongoDB's shard balancer could not redistribute fast enough.
- **Cassandra's partition key design matters**. Discord uses `channel_id` as the partition key and `message_id` as the clustering key. Messages for a channel are sorted by ID, and reads paginate through them efficiently.
- **Compaction is expensive**. Discord runs compaction only during low-traffic hours. They use leveled compaction for message tables to reduce read amplification.
- **Monitor tombstone ratios**. Discord alerts when tombstones exceed 5% of total data — too many tombstones indicate an anti-pattern (frequent deletes, which Cassandra handles poorly).

### Amazon — DynamoDB for Shopping Cart

Amazon built DynamoDB originally for their shopping cart service. The cart is a simple data structure: a list of items keyed by user ID. Write throughput spikes during holiday sales (100x normal). Read throughput is consistently high (every page load checks the cart).

**Why DynamoDB for the cart:**
- **Simple key-value access**: `get(cart/user_id)` and `put(cart/user_id, items)`. No joins, no transactions needed at the database level.
- **Unlimited scalability**: DynamoDB automatically partitions data across storage nodes. During Black Friday, writes scale from 50K to 5M per second without manual intervention.
- **SSD-backed storage**: Consistent single-digit millisecond latency at any scale.
- **Multi-region replication**: Users in Europe see the same cart as users in the US, within seconds.
- **Strong consistency option**: For checkout, DynamoDB provides strongly consistent reads (at 2x the read capacity cost).

**Amazon's key lesson**: If your access pattern is simple (key-value), use a key-value store. DynamoDB is the shopping cart's natural home. Attempting to store carts in a SQL database would require constant shard management and add unnecessary query features.

## Open Source Implementations

### PostgreSQL Extensions for NoSQL Workloads

PostgreSQL has evolved to handle many NoSQL use cases through extensions:

- **PostgreSQL JSONB**: Store JSON documents in a column with indexing via GIN indexes. Query with `->` and `->>` operators. This gives you NoSQL-like document storage WITH joins and transactions.
- **Citus (now pg_citus)**: Horizontally scale PostgreSQL across multiple nodes. Distribute tables, run parallel queries, and get SQL semantics at Cassandra-like scale.
- **TimescaleDB**: PostgreSQL extension for time-series data. Automatic partitioning (hypertables), continuous aggregates, and columnar compression. Handles 1M+ writes per second on a single node.
- **PgVector**: Vector similarity search for AI applications. Builds on PostgreSQL's indexing to support nearest-neighbor queries.

```sql
-- JSONB example: document store inside PostgreSQL
CREATE TABLE products (
    id BIGSERIAL PRIMARY KEY,
    data JSONB NOT NULL
);
CREATE INDEX idx_products_data ON products USING GIN (data);
-- Query like a document store
SELECT * FROM products WHERE data @> '{"category": "electronics"}';
-- Join with relational tables
SELECT p.data->>'name', o.order_date
FROM products p
JOIN orders o ON o.product_id = p.id;
```

### RocksDB — The Embedded LSM Engine

RocksDB is Facebook's LSM-tree storage engine, used as the foundation for many NoSQL databases:
- **Cassandra**: RocksDB is the default storage engine (since Cassandra 4.0), replacing the original SSTable implementation.
- **TiKV**: Distributed key-value store, the storage layer for TiDB (distributed SQL).
- **Apache Flink**: Uses RocksDB as state backend for streaming applications.

## Cloud Provider Implementations

### AWS

- **Aurora**: MySQL/PostgreSQL-compatible, 5x faster than standard MySQL, auto-scaling storage up to 128TB, 6-way replication across 3 AZs. Serverless option for variable workloads.
- **DynamoDB**: Fully managed key-value and document database, single-digit millisecond latency at any scale, auto-scaling, multi-region via Global Tables, DAX (in-memory cache) for microsecond reads.
- **ElastiCache**: Managed Redis and Memcached. Cluster mode for Redis allows sharding across up to 500 nodes. Supports multi-AZ with automatic failover.
- **Keyspaces**: Managed Cassandra-compatible database. Serverless, auto-scaling, wire-protocol compatible with Cassandra drivers.
- **DocumentDB**: MongoDB-compatible document database. Supports MongoDB 5.0+ wire protocol.

### GCP

- **Cloud Spanner**: Globally distributed, strongly consistent, relational database. Google's crown jewel — used internally for AdWords, Gmail, YouTube. External consistency via TrueTime (GPS + atomic clocks). 99.999% availability SLA.
- **Firestore**: Serverless NoSQL document database. Real-time synchronization via web sockets. Strong consistency by default. Integrated with Firebase for mobile apps.
- **Cloud Bigtable**: Wide-column database (HBase-compatible). Designed for large analytical workloads. 100K+ writes per second per node.
- **Memorystore**: Managed Redis and Memcached.

### Azure

- **Cosmos DB**: Globally distributed, multi-model NoSQL. Supports Cassandra, MongoDB, Gremlin, Table, and SQL APIs. Turnkey multi-region writes. Five consistency levels: strong, bounded staleness, session, consistent prefix, eventual.
- **Azure SQL**: Managed SQL Server. Hyperscale tier auto-scales storage to 100TB. Serverless compute for intermittent workloads.
- **Azure Cache for Redis**: Managed Redis with clustering, persistence, and active geo-replication.

## Lessons Learned

### Pattern 1: Start with SQL, Add NoSQL When Needed

Every team that starts with a NoSQL-only architecture eventually adds SQL. Every team that starts with SQL-only eventually considers NoSQL. The common pattern is:

1. Start with PostgreSQL (single instance).
2. Add Redis (cache) when read latency matters.
3. Add read replicas (SQL) when read volume exceeds primary capacity.
4. Add a document store (MongoDB) when you need flexible schema for a new service.
5. Add Cassandra when you need write throughput above 100K/sec.
6. Add a search engine (Elasticsearch) when you need full-text search.

### Pattern 2: Own Your Consistency Boundaries

Draw a box around data that must be strongly consistent. Everything inside the box uses SQL or a transactional NoSQL system. Everything outside uses eventually consistent NoSQL.

```
            ┌─────────────────────────────────────┐
            │  STRONG CONSISTENCY BOUNDARY         │
            │  PostgreSQL                          │
            │  - Orders                            │
            │  - Payments                          │
            │  - Inventory                         │
            └──────────────┬──────────────────────┘
                           │ (eventual sync via CDC)
                           ▼
            ┌─────────────────────────────────────┐
            │  EVENTUAL CONSISTENCY ZONE           │
            │  Cassandra                           │
            │  - User sessions                     │
            │  - Event logs                        │
            │  - Notifications                     │
            │  Elasticsearch                       │
            │  - Search indexes                    │
            │  - Analytics aggregations            │
            └─────────────────────────────────────┘
```

### Pattern 3: Plan for Database Migration From Day One

No company picks the right database on the first try. Discord migrated from MongoDB to Cassandra. Uber migrated from PostgreSQL to multiple databases. Instagram migrated from PostgreSQL to Cassandra and back to PostgreSQL. Expect to change your database at least once.

**Design for migration:**
- Put a repository/DAO layer between your application and the database. Never write raw queries in business logic.
- Use database-agnostic serialization (Protobuf, Avro) for stored data.
- Keep a migration harness: dual-write to old and new databases, compare results, switch over.

### Pattern 4: Measure Before Optimizing

Teams often migrate databases for the wrong reasons. Before choosing a new database, measure:
- **Current p50, p99, p999 latency** for each query pattern. If p99 is under 100ms, PostgreSQL is fine.
- **Current read/write ratio and throughput**. If writes are under 10K/sec, SQL handles it.
- **Cache hit ratio**. A 95% cache hit ratio means the working set fits in memory. Adding indexes or RAM may solve the problem without a migration.
- **Query patterns**. Are slow queries caused by missing indexes or full table scans? Fix those first.

### Pattern 5: Accept Operational Complexity

Running two databases costs more than running one, but the right two databases cost less than fighting the wrong one. Discord's operational cost of running Cassandra (dedicated SRE, compaction monitoring, repair scheduling) is lower than the cost of MongoDB outages and data corruption.

The same applies to cloud-managed databases. DynamoDB costs more per GB than RDS (PostgreSQL), but the auto-scaling and zero-maintenance sharding reduce operational overhead. Compute the total cost of ownership — infrastructure cost plus engineering time plus downtime cost — before deciding.
