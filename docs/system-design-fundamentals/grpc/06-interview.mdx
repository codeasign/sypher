# gRPC — Interview Playbook

## How This Topic Appears

### System Design Questions

These questions explicitly call for discussing communication protocols. The interviewer expects you to evaluate gRPC, REST, GraphQL, and websockets and justify your choice.

1. "Design a real-time ride-sharing dispatch system" — evaluates streaming awareness and gRPC vs WebSocket trade-off.
2. "Design a YouTube-like video platform" — gRPC surfaces in the upload pipeline, metadata service, and recommendation backend.
3. "Design a multiplayer real-time game server" — bidirectional streaming knowledge is critical.
4. "Design a metrics monitoring system (Prometheus-like)" — server streaming for query results, protobuf for efficient data transfer.
5. "Design a global chat application like WhatsApp" — gRPC for backend services, persistent streaming for message delivery.

### Design-Deep-Dive Questions

These questions focus on gRPC-specific internals:

6. "How does gRPC handle serialization differently from REST/JSON, and what are the performance implications?"
7. "Explain how HTTP/2 multiplexing works in gRPC. What problem does it solve?"
8. "What happens when a gRPC server crashes mid-stream? How does the client detect and recover?"
9. "Compare gRPC load balancing strategies: client-side vs proxy-side. When would you use each?"
10. "How do you evolve a protobuf schema without breaking existing clients?"

## Level-by-Level Expectations

### L3/L4 (Entry-Level / Mid-Level)

**Expected knowledge**:
- gRPC is a high-performance RPC framework using protobuf and HTTP/2.
- Protobuf is smaller and faster than JSON.
- gRPC supports unary, server streaming, client streaming, and bidirectional streaming.
- gRPC uses a single persistent connection (HTTP/2).

**Sample answer to "Why would you use gRPC over REST?"** :

"A service making 40,000 downstream calls per second would waste significant CPU on JSON serialization and TCP connection setup with REST. gRPC uses protobuf binary serialization, which is roughly 10x faster than JSON and produces messages 3-10x smaller. It also uses HTTP/2 multiplexing, so all 40,000 calls share a single TCP connection instead of opening 40,000 separate connections. For latency-sensitive internal services, gRPC typically reduces p99 latency by 50-80% compared to REST/JSON."

**What they check**: Can you name the key benefits with concrete numbers? Do you know the four RPC types?

### L5 (Senior Engineer)

**Expected knowledge** — everything L3/L4 knows plus:
- Protobuf wire format: varint encoding, field tags, wire types, the 5-byte gRPC frame header.
- HTTP/2 frame structure: DATA, HEADERS, SETTINGS, GOAWAY, RST_STREAM.
- HPACK compression and its impact on per-request overhead.
- gRPC load balancing strategies: proxy-based (Envoy) vs client-side (round_robin, pick_first).
- Deadline propagation and context cancellation semantics.
- Schema evolution: backward compatibility rules, field tag reuse dangers.

**Sample answer to "How does gRPC handle request cancellation?"** :

"Every gRPC call carries a context with an optional deadline. When the deadline expires or the caller cancels, the gRPC library sends an HTTP/2 RST_STREAM frame to the server. The server-side library checks the context between protobuf deserialization steps and in blocking I/O calls. If the context is cancelled, it returns CANCELLED status immediately, freeing the server thread. On the client side, the in-flight response is discarded. The critical point: deadline propagation across service boundaries. If Service A calls B with a 5-second deadline, and B calls C without propagating that context, C might hang for 2 hours (TCP timeout) while B's thread is consumed. Always propagate the context: in Go, pass `ctx` through; in Python, check `context.timeout()` and pass it downstream."

**What they check**: Do you know the mechanics (RST_STREAM), not just the concept? Can you describe the cascade failure and how to prevent it?

### L6 (Staff Engineer)

**Expected knowledge** — everything L5 knows plus:
- gRPC max message size and flow control tuning for specific workloads.
- gRPC + Envoy health checking protocol integration.
- Multi-region gRPC routing: anycast, DNS-based regional affinity, cross-region failover.
- gRPC observability: interceptor-based metrics, distributed tracing propagation.
- Performance profiling: where gRPC overhead actually lives (serialization, not network).
- Retry budget management: preventing retry storms during cascading failures.

**Sample answer to "Design a multi-region gRPC service for a global chat application"** :

"I would deploy gRPC servers in 5-7 regions, each fronted by an Envoy proxy. Clients connect to the nearest region via DNS-based routing (latency-based routing with Route53 or Google Cloud DNS). Each Envoy runs gRPC health checks against its backends. The gRPC client uses the `round_robin` load balancing policy with subchannel sharing — one channel per region.

For write operations (send message), the gRPC server in the nearest region forwards the write to the primary region via a gRPC unary call with a 2-second deadline. For read operations (get messages), the server reads from the local replica.

The critical failure mode is region isolation. If the primary region becomes unreachable, writes must either fail fast (CP approach) or buffer locally and replay (AP approach). I would use CP for messages (consistency matters) and AP for read-only user presence. gRPC deadlines handle the fail-fast part — 2-second deadline, retry 2 times with 500ms backoff. If all fail, return UNAVAILABLE status with a retry hint.

For streaming (presence, typing indicators), each client maintains a bidirectional gRPC stream to its regional gateway. The stream is pinned by consistent hashing of user ID to a specific gateway instance. If the connection drops, the client reconnects with the last seen sequence number, and the server replays missed events from that point. gRPC flow control windows are set to 1 MB per stream to handle bursty typing indicator traffic without excessive ACKs."

**What they check**: Can you design for real scale with specific, justified decisions? Do you anticipate failure modes? Can you articulate trade-offs (CP vs AP for different features)?

### L7/L8 (Principal / Distinguished Engineer)

**Expected knowledge** — everything L6 knows plus:
- Can design or evaluate alternative RPC protocols (Cap'n Proto, FlatBuffers, Thrift) vs gRPC and justify the choice.
- Understands the impact of gRPC's choice on organizational structure: proto monorepo vs multi-repo, cross-team dependency management.
- Can design a company-wide gRPC migration strategy: phased approach, interoperability, fallback mechanisms.
- Can articulate when NOT to use gRPC at the organizational level, not just the service level.
- Understands the cost of gRPC adoption: training, tooling, debugging infrastructure, proto governance.

**Sample answer to "Your company has 300 microservices on REST/JSON. Should you migrate to gRPC?"** :

"The migration cost is approximately 6-9 months of engineering time across all teams. I would not approve an org-wide migration unless: (1) serialization CPU exceeds 10% fleet-wide, (2) average inter-service latency exceeds 50ms, or (3) we are adding streaming requirements that REST cannot efficiently handle.

If these conditions are not met, I would recommend a targeted migration: adopt gRPC for the new event-ingestion pipeline and the real-time notification service (both need streaming). Use gRPC-Gateway to expose REST endpoints for these services so existing consumers are unaffected. Over 12 months, measure CPU and latency improvements from these services. If the data supports it, expand to high-traffic services (above 5,000 RPS). Leave low-traffic services on REST — the migration cost is not justified.

The organizational cost is significant. Proto file governance requires a new team process: field tag allocation, breaking change review, cross-team dependency tracking. I would create a 'proto guild' — one representative per team — to review schema changes weekly. Without this, field tag conflicts and silent data corruption will occur within 6 months."

**What they check**: Can you think at the organizational level? Do you understand that technology choices have team and process implications? Can you design a measured, risk-aware adoption strategy?

## The FAANG Framework for Answering

### 6-Step Framework for gRPC Questions

**Step 1 — State the requirement**: "You as the interviewer are asking me to choose a communication protocol for a service that handles X RPS with Y latency requirements and Z client types. Let me clarify the constraints first."

**Step 2 — List the options**: "The main options are gRPC, REST/JSON, GraphQL, and WebSocket. Let me evaluate each against the requirements."

**Step 3 — Compare with concrete numbers**: "gRPC serialization is approximately 10x faster than JSON. HTTP/2 multiplexing means one connection handles thousands of concurrent calls. For services with over 1,000 RPS internal traffic, gRPC typically reduces CPU by 5-10 percentage points and latency by 50-80%."

**Step 4 — Apply to the specific use case**: "For the real-time dispatch system, we need driver GPS streaming upstream and match offers downstream. This maps directly to gRPC bidirectional streaming. REST cannot efficiently handle the upstream streaming requirement — we would need WebSocket on top of REST, which adds complexity."

**Step 5 — Address trade-offs and failure modes**: "gRPC adds operational complexity: protobuf schema management, HTTP/2 flow control tuning, and streaming error handling. If the streaming consumer is slow, the producer can exhaust memory through flow control buffer buildup. We would need server-side flow control with backpressure signaling."

**Step 6 — Conclude with a clear recommendation**: "I recommend gRPC for the dispatch backend with Envoy proxy for load balancing and mTLS. Use gRPC-Gateway for the driver app's REST endpoints if the app cannot use gRPC-Web directly. The expected outcome: 60-80% reduction in latency compared to the current REST-based system."

## Red Flags

1. **"gRPC is always faster than REST"** — This is false for small payloads on fast networks. The serialization overhead difference between protobuf (2 microsec) and JSON (25 microsec) for a 1 KB message matters only at high throughput. For a service handling 50 RPS with 1 KB payloads, the difference is negligible. You must qualify performance claims with scale context.

2. **"We use gRPC everywhere including browser clients"** — gRPC-Web exists but has limitations: no bidirectional streaming, no trailer support, experimental mobile support. A production system needs either a REST gateway or gRPC-Web with Envoy. Claiming native gRPC works in browsers shows you have not deployed gRPC in production.

3. **"Deadlines are optional, we just set them on critical paths"** — Deadlines are never optional. Every gRPC call must have a deadline. The cost of a missed deadline is a thread/goroutine blocked for minutes while the server recovers from a transient issue.

4. **"Protobuf field tags can be reused if the field is deleted"** — This causes silent data corruption. Old clients still running old code will deserialize the new field as the old field type. Use `reserved` to prevent reuse.

5. **"We use bidirectional streaming for all real-time features"** — Bidirectional streaming is significantly more complex than unary or server streaming. It should be a measured decision, not a default. Many real-time features (notification feeds, event streams) are better served with server streaming.

6. **"gRPC automatically handles retries, so we do not need idempotency"** — gRPC retry is transparent; the client retries the entire RPC. If the server received the first request, processed it, and crashed before responding, the retry will process it again — creating a duplicate. Idempotency keys are still required for mutations.

7. **"We do not need load balancing because gRPC uses a single connection"** — A single gRPC channel can connect to only one endpoint. To distribute traffic across multiple server instances, you need either client-side load balancing (DNS resolver with multiple A records) or a proxy (Envoy, NGINX) in front of your server pool.

8. **"Protobuf is self-documenting, so we do not need API documentation"** — Protobuf defines the wire format, not the API contract semantics (what values are valid, what error states exist, what side effects occur). API documentation is still required.

## Practice Drills

### Drill 1: The Elevator Pitch (2 minutes)

Explain to a product manager why the team should migrate the user service from REST to gRPC. Do not use jargon — explain in terms of user-facing latency and infrastructure cost. Say it aloud in under 2 minutes.

**Check**: Did you mention concrete numbers? Did you avoid protobuf internals? Did you explain the benefit in terms the PM cares about (faster feature delivery, lower cost, better reliability)?

### Drill 2: Protocol Deep-Dive (5 minutes)

Explain protobuf encoding to a fellow engineer who knows JSON but has never used protobuf. Cover: field tags, varint encoding, wire types, default value omission, schema evolution rules.

**Check**: Can you explain varint encoding with an example (300 → `0xAC 0x02`)? Can you explain why unknown fields in JSON are harmless but unknown field tags in protobuf must be handled with `reserved`?

### Drill 3: System Design — Notification Feed (5 minutes)

Design a real-time notification feed for 10 million users. Justify your protocol choice (gRPC, REST, WebSocket, or combination).

**Check**: Did you consider gRPC server streaming (client subscribes, server pushes notifications)? Did you address the streaming reconnect problem (last-seen ID for replay)? Did you discuss browser clients (gRPC-Web or REST gateway)?

### Drill 4: Whiteboard — gRPC Request Lifecycle (10 minutes)

Draw the full lifecycle of a gRPC unary call on a whiteboard, showing the bytes on the wire at each stage: protobuf → 5-byte gRPC frame → 9-byte HTTP/2 frame → TCP segment.

**Check**: Can you draw the byte layout? Do you know the gRPC frame header fields (compression flag, message length)? Do you know the HTTP/2 frame header fields (length, type, flags, stream ID)?

### Drill 5: Failure Post-Mortem (10 minutes)

Your team's gRPC service had an outage: p99 latency went from 50ms to 5 seconds for 15 minutes. All metrics showed healthy CPU, memory, and database. The service was not at capacity. Describe your debugging process and what you might find.

**Expected answer structure**: (1) Check gRPC client-side metrics — is latency high on the client or the server? (2) Check for slow downstream gRPC calls — one service in the call chain might be slow. (3) Check gRPC deadline propagation — did a downstream service lose its deadline? (4) Check for thread pool exhaustion from slow downstream calls. (5) Check for HTTP/2 flow control — if a streaming response is large, flow control window updates add latency. (6) Check for a recent deployment that changed a protobuf message size (larger messages = more serialization time).
