# gRPC in Production

## Company Case Studies

### Netflix: gRPC for Internal Service Mesh

**Scale**: 200+ million subscribers, 1,000+ microservices, billions of RPCs per day across 7 regions.

**Problem**: Netflix's internal service mesh (part of their Titus container platform) was using REST/JSON over HTTP/1.1. The serialization overhead consumed 15% of total CPU fleet-wide. Connection management -- opening new TCP connections for each request -- added 20-30ms of latency per call. With deeply nested service chains (a single user request could fan out to 40+ microservices), this overhead compounded to over 1 second of pure transport latency before any business logic ran.

**Solution**: Netflix migrated their internal service mesh to gRPC with Envoy sidecar proxies. Each service pod runs an Envoy proxy that terminates gRPC and handles load balancing, retries, and circuit breaking. They adopted client-side load balancing with the gRPC `round_robin` policy, removing the need for a centralized load balancer.

**Result**:
- Fleet-wide CPU for serialization dropped from 15% to 2%.
- Median inter-service latency dropped from 35ms to 8ms (77% reduction).
- Connection count dropped from 10,000+ per host to under 20 (one per destination service).
- gRPC deadlines and propagation gave them fine-grained control over cascading failure.

### Uber: gRPC for Real-Time Ride Matching

**Scale**: 25 million trips per day, 450 cities, sub-second ride matching requirement.

**Problem**: Uber's dispatch system needed to stream real-time driver location updates (GPS pings every 4 seconds from 3+ million drivers) to the matching engine and then push match decisions back to drivers. The previous system used long-polling HTTP which caused: (1) 40% bandwidth waste on empty polls (no new data), (2) 15-second average latency between a trip request and driver notification, and (3) massive connection churn as drivers polled and reconnected.

**Solution**: Uber replaced their dispatch and driver-tracking pipeline with gRPC bidirectional streaming. Each driver app opens a single gRPC bidirectional stream: the driver sends GPS coordinates upstream; the server sends match offers, routing updates, and trip events downstream -- all on the same stream. They built a custom gRPC load balancer (Ringpop) that uses consistent hashing to pin a driver to a specific backend, ensuring stream locality.

**Result**:
- Driver notification latency dropped from 15 seconds to under 800ms (p95).
- Bandwidth per driver dropped by 60% (no more empty polls).
- Connection count dropped from 4 connections per driver (polling) to 1 bidirectional stream.
- Server CPU per driver dropped by 45% (no HTTP connection overhead).

### Discord: gRPC for Voice and Chat Infrastructure

**Scale**: 150 million monthly active users, 19 million servers, 1 billion+ gRPC calls per day.

**Problem**: Discord's core chat service (the "Session" service) needed to route messages and state updates between users in real time. Their legacy REST-based approach required clients to poll for updates. For a server with 1,000 members at 1 poll per second, that is 1,000 HTTP requests per second -- each with TLS handshake, headers, and JSON parsing overhead.

**Solution**: Discord moved their Session service to gRPC bidirectional streaming. Each client maintains a persistent gRPC stream to the nearest regional gateway. The stream carries: presence updates (typing indicators, online status), message acknowledgements, and channel state changes. They run gRPC on top of their own custom transport layer (called "Orca") that adds application-level flow control above HTTP/2.

**Result**:
- Messages per second per gateway node: 250,000 (up from 30,000 with REST).
- Gateway CPU usage per connection dropped 70%.
- P99 message delivery latency: 15ms (down from 120ms).
- They were able to reduce their gateway fleet from 180 nodes to 45 for the same traffic.

### Cloudflare: gRPC for Edge Network Services

**Scale**: 28 million internet properties, 275+ edge locations, 45 Tbps network capacity.

**Problem**: Cloudflare's edge services (WAF rules, cache purging, DNS updates) communicate between edge nodes and the central control plane. The control plane pushes configuration updates (firewall rules, load balancing configs) to 275+ edge locations. The original system used HTTP/1.1 polling. Each edge location polled every 30 seconds, generating 550 requests per minute to the control plane -- mostly empty responses (no config change). The aggregated polling traffic consumed 30% of control plane capacity.

**Solution**: Cloudflare deployed gRPC server streaming for config pushes. Each edge node opens a persistent gRPC streaming connection to the control plane. When a config changes, the control plane pushes the update to all connected edge nodes simultaneously. The edge node never polls.

**Result**:
- Config propagation latency dropped from 30 seconds (poll interval) to under 500ms (push).
- Control plane load dropped by 85% (no polling traffic).
- Total bandwidth for config sync dropped by 90% (protobuf binary vs JSON, and no empty poll responses).
- Connection count: 275 (one per edge location) instead of 550 requests per minute.

### Dropbox: gRPC for Storage Backend Communication

**Scale**: 700 million users, 500+ PB of stored data, 100,000+ storage nodes.

**Problem**: Dropbox's "Magic Pocket" storage system (their custom-built block store) needed fast communication between metadata services and storage nodes. The metadata service determines which storage node holds a given block; the client then must read from that node. These lookups and transfers happen millions of times per second. Using REST, each block transfer required HTTP connection setup, sending the block ID in JSON, and receiving the block data -- typically 200ms overhead for a 4 KB block that takes 1ms to transfer.

**Solution**: Dropbox uses gRPC for all Magic Pocket internal RPCs. Block requests use unary RPC (request block ID, receive block bytes). Metadata updates use server streaming (subscribe to changes). Block replication between storage nodes uses bidirectional streaming (send blocks to replicate, receive acknowledgements).

**Result**:
- Internal RPC latency dropped 70% (from 200ms to 60ms per block lookup + transfer).
- Storage node CPU for protocol handling dropped from 12% to 3%.
- The protobuf contract allowed automatic code generation in Go (metadata services) and C++ (storage nodes), eliminating hand-rolled serialization.

## Open Source Implementations

### Envoy Proxy

Envoy is the most popular gRPC-capable proxy. It supports:

- **gRPC pass-through**: Terminates HTTP/2 and forwards gRPC frames transparently.
- **gRPC-Web bridge**: Translates gRPC-Web requests to native gRPC (and back).
- **gRPC health checking**: Implements the standard `grpc.health.v1.Health/Check` protocol.
- **gRPC statistics**: Exports per-method metrics (requests, errors, latency) to Prometheus.
- **gRPC access logging**: Logs method name, status code, duration, and metadata.

Configuration example:

```yaml
# Envoy gRPC proxy configuration (partial)
static_resources:
  listeners:
  - name: grpc_listener
  address:
  socket_address: { address: 0.0.0.0, port_value: 443 }
  filter_chains:
  - filters:
  - name: envoy.filters.network.http_connection_manager
  typed_config:
  "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
  codec_type: HTTP2
  stat_prefix: grpc
  http_filters:
  - name: envoy.filters.http.router
  route_config:
  virtual_hosts:
  - name: grpc
  domains: ["*"]
  routes:
  - match: { prefix: "/UserService/" }
  route: { cluster: user_service_grpc }
  clusters:
  - name: user_service_grpc
```

### etcd (gRPC-native)

etcd uses gRPC as its native communication protocol. All client-server interactions -- key-value operations, watch streams, lease management, cluster membership -- are defined in protobuf and served over gRPC.

Key gRPC features etcd uses:
- **Unary RPC**: `Put`, `Get`, `Delete`, `Txn` (transaction).
- **Server streaming**: `Watch` -- client sends a watch request, server streams all changes.
- **Bidirectional streaming**: `LeaseKeepAlive` -- client periodically sends keep-alives; server confirms.
- **gRPC Gateway**: etcd also serves REST/JSON endpoints via gRPC-Gateway, so CLI tools (like `etcdctl`) can use either.

etcd's gRPC choice influenced the Kubernetes ecosystem -- kube-apiserver communicates with etcd exclusively through gRPC.

### CockroachDB

CockroachDB uses gRPC for all inter-node communication: Raft consensus, SQL execution, distributed transactions, and cluster replication. Each CockroachDB node runs a gRPC server that speaks a custom protobuf-defined protocol (the "CockroachDB wire protocol").

Why gRPC for a database: (1) The distributed SQL engine sends millions of small messages per second between nodes; binary protobuf serialization cuts CPU overhead. (2) Streaming is essential for SQL result sets -- a `SELECT` that returns 1 million rows uses server streaming to avoid buffering all results in memory. (3) gRPC deadlines align with SQL query timeouts.

## Cloud Provider Implementations

### AWS

- **No native gRPC service**: AWS does not offer a managed gRPC service. You run gRPC on EC2, ECS, or EKS.
- **AWS App Mesh**: Supports gRPC for service mesh traffic. Configure gRPC listeners and routes through App Mesh's Envoy integration.
- **AWS Cloud Map**: Can serve as a gRPC-compatible service registry (DNS-based).
- **API Gateway**: Supports gRPC proxying (REST/HTTP APIs only, not WebSocket). HTTP/2 must be enabled on the API Gateway.

### GCP (Google Cloud)

- **Google Cloud Run**: Fully supports gRPC -- deploy a gRPC server as a container, get auto-scaling, HTTPS, and regional load balancing. gRPC health checks are built in.
- **Google Kubernetes Engine**: Native gRPC support with GKE Ingress (HTTP/2 passthrough) and Anthos Service Mesh (Envoy-based).
- **Traffic Director**: Google's managed gRPC-aware load balancer. Understands gRPC methods and can route, split traffic, and enforce policies per-method.
- **gRPC-Web via Cloud Endpoints**: Managed gRPC-Web proxy for browser clients.

### Azure

- **Azure Container Apps**: Supports gRPC with HTTP/2 ingress. Configure `ingress.exposedPort` for gRPC traffic.
- **Azure Kubernetes Service**: gRPC support through NGINX Ingress (with HTTP/2 annotation) or Application Gateway.
- **No native gRPC load balancer**: Azure Load Balancer (Layer 4) works for TCP, but L7 gRPC-aware routing requires NGINX/Envoy.

## Lessons Learned

### Pattern 1: Always Use mTLS for gRPC in Production

Every case study uses mutual TLS for production gRPC. gRPC's performance advantage does not excuse insecure communication. mTLS ensures that both client and server authenticate each other, preventing man-in-the-middle and unauthorized service access. Use a service mesh (Envoy with SPIRE or cert-manager) to automate certificate rotation.

### Pattern 2: gRPC Enables, It Does Not Replace, Your Service Mesh

gRPC and service mesh (Envoy, Linkerd) are complementary, not competing. gRPC handles the application-level call semantics (serialization, framing, streaming). The service mesh handles cross-cutting concerns (load balancing, retries, circuit breaking, observability, security). Netflix, Uber, and Discord all use gRPC with Envoy, not gRPC standalone.

### Pattern 3: Start with Unary, Add Streaming Only When Measured

Every company that adopted gRPC started with unary RPCs replacing REST endpoints. They added streaming only when they had metrics showing that polling or pagination was the bottleneck. Discord added bidirectional streaming for presence; Uber added it for GPS tracking; Netflix's core migration was entirely unary. Do not pre-optimize with streaming.

### Pattern 4: Schema Evolution Discipline Is Non-Negotiable

Organizations with the smoothest gRPC migrations enforced strict protobuf schema review processes: (1) never reuse field tags, (2) always use `reserved` for deleted fields, (3) add new fields in the middle, not at the end (protobuf encoding is field-tag-order-independent), (4) use `google.protobuf.Timestamp` and `google.type.Money` for semantically typed fields instead of raw int64. Violations caused silent data corruption that took weeks to detect.

### Pattern 5: gRPC Does Not Solve Browser and Mobile

Even at massive scale, every company runs a REST gateway or gRPC-Web proxy for browser and mobile clients. gRPC-Web exists but has limitations: no bidirectional streaming, no trailer-based status, and mobile support is still experimental. Dropbox uses REST for mobile clients and gRPC only for server-to-server. Netflix uses REST for their TV apps and gRPC internally. Plan for two API surfaces: internal (gRPC) and external (REST/gRPC-Web).
