# gRPC — Challenges

## Tier 1 — Foundation

### Challenge 1: Protobuf Wire Format (5 minutes)

A protobuf message contains a single field: `int32 age = 1;` with value `150`.

**Question**: What bytes appear on the wire for this field? Show your work.

**Evaluation criteria**: Correct application of varint encoding. Correct field tag + wire type composition.

**Model answer**:

Field tag 1, wire type 0 (varint) → `(1 <under 3) | 0 = 0x08`.

Value 150 encoded as varint:
- Binary: `10010110` (128 + 16 + 4 + 2 = 150? No — let me redo: 128 + 16 + 4 + 2 = 150. Correct, 8 bits.)
- Wait, 150 in binary is `10010110` — that is 8 bits, so it needs multiple varint bytes.

150 decimal = binary `10010110`:
- Split into 7-bit groups: `0000001` `0010110`
- Reverse groups (little-endian): `0010110` `0000001`
- Set MSB=1 on all but last: `10010110` `00000001`
- Hex: `0x96` `0x01`

Wire bytes: `0x08 0x96 0x01` (3 bytes total: tag + 2-byte varint).

### Challenge 2: Protocol Choice (5 minutes)

Your team is building an internal user-facing dashboard that loads 500 rows of data from a backend service. The dashboard reloads every 10 seconds. The backend handles 200 RPS from 50 dashboard instances. The data rows are approximately 100 bytes each.

**Question**: Should you use gRPC or REST/JSON for the dashboard-to-backend communication? Justify with concrete numbers.

**Evaluation criteria**: Correct analysis of scale. Recognition that gRPC overhead savings are negligible at this volume. Consideration of debugging and development speed.

**Model answer**:

Use REST/JSON. At 50 instances × 1 request every 10 seconds = 5 RPS, with 200 total RPS backend load. The payload is 500 × 100 bytes = 50 KB. Using REST/JSON, this payload becomes approximately 75 KB. Using gRPC/protobuf, approximately 52 KB. The bandwidth savings is 23 KB per request — negligible at 5 RPS (115 KB/second saved — nothing).

Development cost: the dashboard team would need to set up protobuf tooling, code generation, and a gRPC client library. The debugging overhead (cannot curl the endpoint to inspect responses) adds friction. At this scale, gRPC's performance advantage is not realized. REST/JSON is the correct choice.

If the dashboard were scaled to 10,000 instances (1,000 RPS) and payloads were 10 MB each, then gRPC's serialization speed and wire efficiency would matter. But at 50 instances, they do not.

### Challenge 3: Field Tag Management (5 minutes)

Your proto file has this message:

```protobuf
message User {
  string id = 1;
  string name = 2;
  string email = 3;
  bool is_active = 4;
  string phone = 5;
}
```

You need to remove `is_active` and add `email_verified` (bool) and `phone_verified` (bool). What is the correct way to modify this proto file?

**Evaluation criteria**: Correct use of `reserved` keyword. Correct field tag assignment (new fields get new tags, never reuse old ones).

**Model answer**:

```protobuf
message User {
  reserved 4;              // was is_active, removed in v2.1
  reserved "is_active";    // prevent field name reuse

  string id = 1;
  string name = 2;
  string email = 3;
  string phone = 5;
  bool email_verified = 6;  // new field — new tag
  bool phone_verified = 7;  // new field — new tag
}
```

The `reserved` keyword prevents field tag 4 and field name `is_active` from ever being reused. New fields get tags 6 and 7 — the next available tags after the existing ones. Tag reuse would cause silent data corruption if old clients still reference tag 4 as `is_active`.

## Tier 2 — Design

### Challenge 4: gRPC for a Stock Market Data Feed (15 minutes)

**Design a gRPC-powered stock market data feed that delivers real-time prices to 100,000 trading terminals.** Each terminal subscribes to a subset of 10,000 stocks. Prices update up to 1,000 times per second per stock. Terminals must receive updates within 10ms. Each price update is approximately 50 bytes.

**Rubric**:

| Criterion | Max | Pass |
|-----------|-----|------|
| Correct gRPC pattern (server streaming) | 2 | 1 |
| Bandwidth calculation | 2 | 1 |
| Connection management (channels, load balancing) | 2 | 1 |
| Flow control and backpressure | 2 | 1 |
| Failure handling (reconnect, replay) | 2 | 1 |
| **Total** | **10** | **5** |

**Model answer**:

**Pattern**: Server streaming RPC. Each terminal sends a `SubscribeRequest` (list of stock symbols) and receives a stream of `PriceUpdate` messages. This is the natural fit — one request, infinite stream of responses.

**Bandwidth**: 100,000 terminals × 10 stocks each × 1,000 updates/second/stock = 1 billion updates/second. At 50 bytes each (after protobuf), that is 50 GB/s of data. This is not feasible on a single connection or even a single region.

**Solution**: fan-out via regional gateways. Each terminal connects to a gRPC gateway in its nearest region. The gateway maintains a single server-streaming connection to a central price distributor that sends all 10,000 stocks. The gateway filters and forwards only the stocks each terminal subscribed to.

**Flow control**: Each terminal gateway sets a per-stream flow control window of 10 MB. If a terminal is slow (network congestion, CPU spike), the flow control window fills, and the gateway stops sending updates to that stream. The gateway buffers up to 1 second of updates per terminal (approximately 500 KB). If the terminal remains slow for more than 1 second, the gateway drops the connection — the terminal must reconnect with the last update timestamp.

**Reconnect**: Each PriceUpdate includes a sequence number. On reconnect, the terminal sends the last received sequence number, and the server replays missed updates from its in-memory circular buffer (last 2 seconds, approximately 100 MB per stock).

**Capacity planning**: Each gateway instance handles 5,000 terminals. A gateway processes 5,000 × 10 stocks × 1,000 updates = 50 million updates/second. At 50 bytes each and 2 microseconds per protobuf deserialization/serialization, each gateway instance needs approximately 8 CPU cores for protobuf alone. With 50 gateway instances, total fleet: 50 × 8 cores = 400 cores. Add 2x for headroom: 800 cores across 100 instances.

**Load balancing**: DNS-based routing to nearest region. Each terminal is pinned to one gateway via consistent hashing on terminal ID. The gRPC channel uses `pick_first` policy (connect to first resolved address). If the gateway fails, DNS is updated, and the terminal reconnects to the next gateway.

### Challenge 5: gRPC Schema Evolution Strategy (15 minutes)

You have 50 microservices sharing a common `User` protobuf message defined in a shared proto repository. The message currently has 12 fields (tags 1-12). You need to add a new `preferences` field (a map of string to string). Six of the 50 services need the new field; the other 44 do not.

**Design a rollout strategy** that allows adding this field without breaking any service, and handles the 6-month transition period where services are running different versions of the proto file.

**Rubric**:

| Criterion | Max | Pass |
|-----------|-----|------|
| Field tag allocation strategy | 2 | 1 |
| Backward compatibility (old deserialize new) | 2 | 1 |
| Forward compatibility (new deserialize old) | 2 | 1 |
| Rollout order and verification | 2 | 1 |
| Deprecation plan for old field tags | 2 | 1 |
| **Total** | **10** | **5** |

**Model answer**:

**Step 1 — Field tag allocation**: Add the new field with tag 13. Tags 1-12 are in use. Tags 13-15 encode in 2 bytes (still efficient). Tag 13 is safe.

```protobuf
message User {
  // existing fields tags 1-12 unchanged
  map<string, string> preferences = 13;  // new in v2.4
}
```

**Step 2 — Rollout order**: (1) Update the shared proto repository. All teams pull the new proto but do not deploy yet. (2) Deploy the 44 services that do NOT use the new field first. They serialize/deserialize messages with tag 13 as unknown bytes (protobuf parsers preserve unknown fields). (3) Deploy the 6 services that use the new field. They read/write tag 13 normally. (4) Verify: services that write tag 13, services that do not read it, but when they forward the message, they include the unknown bytes (protobuf's default behavior). The field passes through intact.

**Step 3 — Forward compatibility**: Services using the new proto must be able to receive messages from old services that omit tag 13. This works automatically — missing fields get the default value (empty map). The consuming service checks `if user.preferences != nil` before accessing.

**Step 4 — Verification**: Deploy a canary of the new proto to one service in each tier (producer, consumer, forwarder). Run integration tests that verify: (a) new proto can deserialize old proto's messages, (b) old proto can deserialize new proto's messages (unknown fields preserved), (c) forwarded messages preserve the new field even through services that do not know about it.

**Step 5 — No deprecation needed**: No old fields are being removed, so no `reserved` is required.

**Key risk**: if any service does `filter_unknown_fields = true` or equivalent, it will strip the new field during forwarding. Verify this configuration is off in all services before starting the rollout.

### Challenge 6: gRPC Service Mesh Design (15 minutes)

**Design a gRPC service mesh for a company with 200 microservices across 3 data centers.** Each service communicates with 10-30 downstream services. Traffic patterns are: 80% unary RPCs (request-response), 15% server streaming (event subscriptions), 5% bidirectional streaming (real-time collaboration).

**Rubric**:

| Criterion | Max | Pass |
|-----------|-----|------|
| Service discovery mechanism | 2 | 1 |
| Load balancing strategy | 2 | 1 |
| Security (auth, encryption) | 2 | 1 |
| Observability (metrics, tracing) | 2 | 1 |
| Failure handling (timeouts, retries, circuit breaking) | 2 | 1 |
| **Total** | **10** | **5** |

**Model answer**:

**Service discovery**: Use Kubernetes DNS with headless services for each microservice. The gRPC `dns` resolver discovers all healthy pods via A records. For cross-data-center discovery, use a Consul service mesh with WAN federation.

**Load balancing**: Deploy Envoy proxy as a sidecar container in each pod. Envoy handles: (a) L7 gRPC routing (by `:path`), (b) round-robin load balancing across upstream pods, (c) circuit breaking (max connections: 1,000, max pending requests: 1,000, max requests per connection: 0/unlimited), (d) retry budget (max 3 retries with exponential backoff, limited to 0.1% of total requests to prevent retry storms).

**Security**: mTLS between all Envoy sidecars. SPIFFE-based identity using SPIRE for certificate distribution and rotation. No application code handles TLS — Envoy terminates mTLS and forwards plain gRPC to the local container over localhost.

**Observability**: Each Envoy exports gRPC per-method metrics to Prometheus. Distributed tracing via OpenTelemetry with Envoy's Zipkin-compatible tracer. Trace context propagates through gRPC metadata. Each service exports: request rate, error rate (by gRPC status code), latency histograms (p50/p90/p99/p999). Centralized alerting: p99 > 500ms for any method triggers P2 ticket; error rate > 5% for any method triggers P1.

**Failure handling**: (a) Deadlines cascade through propagated context — each service sets a deadline equal to remaining parent deadline minus 100ms buffer. (b) Retries are limited per service with a retry budget: each service can spend 0.1% of its total request budget on retries. Once exhausted, retries stop to prevent cascade. (c) Circuit breaking detects when downstream error rate exceeds 50% over 30-second window. Open circuit for 60 seconds (half-open after 30). (d) Health checking: each Envoy probes its local backend's gRPC health endpoint every 5 seconds. Unhealthy backends are removed from the load balancer within 10 seconds.

## Tier 3 — Expert

### Challenge 7: gRPC at Hyperscale (30 minutes)

**Design a gRPC-based communication layer for a global financial exchange** that processes 500,000 orders per second. Each order event (submitted, filled, canceled, rejected) must be broadcast to all subscribers (brokerage terminals, regulatory systems, internal risk engines) within 1ms of the event timestamp.

Constraints:
- 5,000 subscriber endpoints across 200 brokerage firms, each in one of 10 global financial hubs.
- Each subscriber subscribes to a subset of trading symbols (total 50,000 symbols).
- Events per symbol: up to 100 per second, average 10 per second.
- Data loss tolerance: zero. An event not delivered is a regulatory violation.
- gRPC is required — the exchange has standardized on it.
- You may introduce additional infrastructure (message buses, caches, gateways).

**Model answer structure**: This is an open-ended expert challenge with no single correct answer. A strong response addresses these dimensions:

**1. Impossible in pure gRPC**: Point-to-point gRPC streaming from the exchange to each subscriber cannot meet the latency or reliability requirements. 5,000 subscribers × 10 events/second/symbol × 50,000 symbols = 2.5 billion event deliveries per second. Pure unicast gRPC would require 5,000 server-streaming connections per symbol aggregator — the per-connection overhead alone exceeds the 1ms window.

**2. Required architecture**: A multi-layer fan-out with message bus (Kafka/Pulsar) and local aggregators:
- Layer 1: Order matching engine writes events to a partitioned Kafka topic (keyed by symbol). Each partition has one producer and N consumers.
- Layer 2: Regional gRPC aggregators subscribe to Kafka partitions via gRPC server streaming. Each aggregator filters events by its region's subscribed symbols.
- Layer 3: Broker-side gRPC gateways (one per brokerage, running in the same data center) receive filtered streams from the regional aggregator. The gateway fans out to individual subscriber terminals within the brokerage.
- Layer 4: Subscriber terminals connect to their brokerage's gateway via gRPC server streaming, receiving only their subscribed symbols.

**3. Latency breakdown**: Kafka write: 0.5ms p99. Kafka consume: 0.3ms p99. Regional aggregator filter + forward: 0.1ms p99. Broker gateway fan-out: 0.1ms p99. Total: 1.0ms p99 — barely within requirement. Any variance requires QoS queuing and CPU isolation.

**4. Zero data loss**: gRPC streaming with at-least-once semantics. Each event carries a global sequence number. On reconnect, the subscriber sends the last received sequence number. The broker gateway replays missed events from an in-memory buffer (10 seconds, approximately 500 MB for peak activity). Kafka's log compaction provides the ultimate recovery source.

**5. Anti-patterns to avoid**: (a) Direct gRPC from exchange to each subscriber — N^2 connection scaling is infeasible. (b) Bidirectional streaming — subscribers only receive, never send events. Server streaming is correct. (c) Same gRPC connection for multiple symbols — each symbol stream should be isolated so a slow subscriber for symbol A does not block symbol B's delivery. Use separate gRPC streams per symbol group.

### Challenge 8: gRPC Anti-Patterns Debrief (30 minutes)

Your company has been using gRPC for 2 years across 150 microservices. The CTO is concerned about rising operational costs and asks you to evaluate whether gRPC was the right choice. Present your analysis as a decision document.

**Model answer structure**:

**Evidence to gather**:
1. CPU utilization breakdown: how much of fleet CPU is protobuf serialization vs business logic vs garbage collection vs kernel? Compare with pre-gRPC REST baseline if available.
2. Latency: p50, p99, p999 for inter-service calls. Compare with SLA targets. Is gRPC meeting or exceeding them?
3. Debugging time: hours spent on gRPC-specific issues per quarter (proto schema conflicts, streaming bugs, HTTP/2 flow control, dead connection detection). Compare with hours spent on REST issues pre-migration.
4. Developer productivity: time to define and publish a new API under gRPC (proto definition, code generation, deployment) vs REST (OpenAPI spec, documentation, client generation).
5. Outage count: incidents caused by or related to gRPC (schema incompatibility, connection pool exhaustion, deadline propagation failures).

**Recommendation framework**:
- If serialization CPU dropped by 5+ percentage points AND latency improved by 50%+ AND developer complaints are contained to 1-2 teams: gRPC was the right choice. Optimize the pain points.
- If serialization CPU is unchanged (small payloads, low throughput) AND debugging time increased by 2x: gRPC was a net negative for this organization. Plan a phased migration back to REST for low-traffic services.
- If gRPC meets performance goals but developer productivity dropped sharply: keep gRPC for high-traffic services (top 20 by RPS), move low-traffic services back to REST. Hybrid is operationally acceptable.

**Key anti-patterns to flag**:
1. Teams that used bidirectional streaming for simple request-response patterns — high complexity, low benefit.
2. Teams that never set deadlines — caused 5+ incidents from thread pool exhaustion.
3. Teams that reused field tags — caused 2 silent data corruption incidents.
4. Teams that shared one giant proto file — caused daily merge conflicts and cross-team dependency logjams.
5. Teams that used gRPC for browser-facing API endpoints — required extra gRPC-Web proxy infrastructure with no measurable benefit.

**Final verdict**: gRPC is a tool, not a strategy. Services with over 5,000 RPS or streaming requirements benefit significantly. Services under 200 RPS with simple request-response patterns see little benefit and incur unnecessary operational cost. The optimal path: segment services by communication pattern and apply gRPC only where its strengths align with the workload.
