# Replication — Trade-offs and Failure Modes

No replication strategy is universally correct. Every decision you make trades one property for another — consistency for availability, latency for durability, simplicity for throughput. This page maps those trade-offs so you know exactly what you are giving up with each choice.

## The Core Tension

**You cannot have low-latency writes, strong consistency, and high availability across a network partition — you must choose which one to sacrifice.**

This is the fundamental tension in replication. It is a direct consequence of the CAP theorem, but it shows up in concrete operational decisions: should I wait for the replica to confirm the write? Should I read from the replica even if it might be stale? Should I keep accepting writes when half my nodes are unreachable?

## Decision Matrix

| Decision Point | Option A | Option B | Choose A When | Choose B When |
|----------------|----------|----------|---------------|---------------|
| Replication mode | Synchronous | Asynchronous | Zero data loss is non-negotiable (financial transactions, auth credentials) | Write latency must be under 5 ms and eventual consistency is acceptable (feeds, analytics, logs) |
| Read routing | Always read from primary | Read from replicas | Users must see their writes instantly (read-your-writes consistency) | Read throughput exceeds what one primary can handle, and stale data is tolerable |
| Failover strategy | Automated (Patroni, Orchestrator) | Manual (on-call engineer promotes replica) | Blast radius of downtime is measured in revenue lost per second (e-commerce, payments) | You have tightly controlled maintenance windows and the cost of auto-failover bugs exceeds downtime cost |
| Replica count | Minimum (2-3 total) | Maximum (10+) | Each replica is synchronous and adds latency; each replica costs money | Read traffic is massive and unpredictable; you need headroom for traffic spikes |
| Conflict resolution (multi-leader) | Last-writer-wins | CRDTs or application-level merge | Simplicity matters more than correctness; timestamp skew is bounded tightly | Data semantics are rich (counters, sets, text) and losing a write has business impact |
| Cross-region Replication | Single primary with global replicas | Multi-leader per region | Most users are in one region; cross-region writes are rare | Users are globally distributed; local write latency is a product differentiator |
| Replica hardware | Same spec as primary | Lower spec than primary | Replicas serve production read traffic at full query complexity | Replicas serve cached or filtered queries; cost savings matter |
| WAL retention | Minimum (replica lags tracked) | Maximum (keep WAL for 7 days) | Replicas are reliable and have low latency | Replicas go offline frequently; you need point-in-time recovery |

**CP (Consistency + Partition Tolerance):** Single-leader with synchronous replication. If a replica cannot acknowledge a write, the primary stops accepting writes. The system chooses consistency over availability during a partition.

**AP (Availability + Partition Tolerance):** Multi-leader or leaderless replication. Each node continues accepting writes independently during a partition. The system chooses availability over consistency. Conflicts are resolved after the partition heals.

**Tunable:** Leaderless systems like Cassandra let you set `W` (write consistency) and `R` (read consistency) per query. Write with `W=ALL` for CP behavior. Write with `W=ONE` for AP behavior. This flexibility lets you choose per-operation.

### Partition Behavior

When a network partition splits a 3-node cluster into (2 nodes, 1 node):

- **Single-leader sync:** The side without the leader (1 node) cannot write. The side with the leader (2 nodes) continues to write. The isolated node's data goes stale. When the partition heals, the isolated node catches up from the WAL.

- **Multi-leader:** Both sides continue writing. When the partition heals, the system detects conflicts (version vectors diverge). Resolution may lose data depending on the strategy.

- **Leaderless with quorum W=2, R=2:** The side with 2 nodes can write and read. The side with 1 node cannot achieve write quorum and stops accepting writes. Reads from the single node return stale data only.

## Failure Modes

### Failure 1: Replication Lag Cascade

**What:** A replica falls behind due to a CPU-intensive query. The lag grows from 1 second to 60 seconds. The WAL sender on the primary keeps up, but the replica's applier cannot replay quickly enough.

**Why:** A long-running `REPORT` query on the replica blocks WAL replay. In PostgreSQL, `max_standby_streaming_delay` allows the query to complete, but the WAL applier queues up behind it. In MySQL, the single-threaded SQL thread falls behind.

**Blast radius:** All reads from that replica return data that is 60 seconds stale. If the primary fails, up to 60 seconds of writes may be unreplicated and lost.

**Detection:** Alert when `pg_stat_replication.replay_lag` exceeds 30 seconds.

**Recovery:** Cancel the blocking query on the replica. The WAL applier catches up. Add a dedicated reporting replica that is isolated from production read traffic.

### Failure 2: Split-Brain After Network Recovery

**What:** A network partition causes two regions to each promote their own primary. The partition heals, and both primaries accept writes simultaneously.

**Why:** The fencing mechanism (STONITH — Shoot The Other Node In The Head) failed. The old primary was never killed, so it continued accepting writes after the new primary was promoted.

**Blast radius:** Total data corruption. Two divergent data sets that cannot be automatically merged.

**Detection:** Replication conflict errors in logs. Data integrity checks fail. Application users see inconsistent data.

**Recovery:** Manual reconciliation. Identify the last common LSN. Compare row versions. Apply changes from the loser's WAL that do not conflict with the winner's data. This takes hours to days.

### Failure 3: WAL Disk Full

**What:** A replica disconnects for 8 hours. The primary keeps all WAL segments for that replica's slot. The WAL disk fills up. The primary crashes.

**Why:** The replication slot prevents WAL cleanup. Even if the replica is gone, the slot says "I need WAL from LSN X onward." The primary cannot delete those segments.

**Blast radius:** Complete write outage. The application cannot write any data. Users see errors on all write operations.

**Detection:** Disk usage alert on primary. Replication slot `restart_lsn` is far behind current LSN.

**Recovery:** Drop the replication slot for the disconnected replica. Delete old WAL segments. Restart the primary. Reconfigure the replica to rejoin from scratch.

### Failure 4: Stale Primary After Failover

**What:** The primary crashes. A replica is promoted. The old primary comes back online, still believing it is the primary. It accepts writes that conflict with the new primary.

**Why:** The old primary was not properly fenced. It either did not detect the failover or the fencing script failed.

**Blast radius:** Same as split-brain — divergent data.

**Recovery:** The new primary's WAL position is ahead of the old primary's. The old primary must be destroyed and rebuilt as a replica. Never rejoin a node that was a primary during a failover without a full rebuild.

### Failure 5: Cascading Failover

**What:** The primary fails. Replica A is promoted. Replica A fails 30 seconds later due to a latent bug (same database version, same schema, same data corruption). Replica B is promoted. Replica B also fails.

**Why:** All replicas ran the same data migration that caused the original primary failure. The fix must be applied before any replica can serve as primary.

**Blast radius:** Complete cluster failure. All nodes down. Full recovery from backup required.

**Detection:** All nodes are down. This is a P0 incident with CEO visibility.

**Recovery:** Restore from the most recent backup. Apply hotfix (roll back migration). Promote the restored node. This takes 30-60 minutes.

## Anti-Patterns Hall of Shame

**Anti-pattern 1: Read-your-writes with async replicas only.** The application writes to the primary, then immediately reads from a random replica. If the replica is behind, the user sees data loss. *Fix:* Read from the primary for N seconds after writing, or include the LSN in the write response and verify replicas are caught up.

**Anti-pattern 2: Synchronous replication across continents.** Every write to US-East waits for a confirmation from a replica in Sydney. Write latency goes from 1 ms to 300 ms. The application's throughput drops by 90%. *Fix:* Use async cross-region replication with a sync replica in the same region.

**Anti-pattern 3: Infinite replica scaling.** Adding more replicas seems like free read throughput. Each replica consumes WAL bandwidth on the primary. At 50 replicas, the primary's network is saturated with WAL traffic and cannot serve writes. *Fix:* Use a replica-of-a-replica cascade topology, or switch to a read cache layer for massive read scaling.

**Anti-pattern 4: Ignoring replication lag in application logic.** The database dashboard shows 0 seconds of lag, so the application assumes it can always read from replicas. At 3 AM during a backup run, lag spikes to 15 seconds. Users who update their profile see the old profile for 15 seconds. *Fix:* Build the application to tolerate stale reads gracefully. Show a "recently updated" indicator. Verify data after reads when consistency matters.

**Anti-pattern 5: No fencing in automated failover.** The failover script promotes a replica but does not kill the old primary. The old primary remains on the network and continues accepting writes. *Fix:* Use a STONITH mechanism. In cloud environments, this means shutting down the old primary's VM. In Kubernetes, deleting the pod or scaling the statefulset.

**Anti-pattern 6: Using the same replica for sync replication and heavy reporting.** The sync replica handles production write acknowledgments AND runs the heavy quarterly report. When the report runs, WAL replay blocks. The primary stops acknowledging writes. *Fix:* Dedicate separate replicas: one for sync replication (light queries only), one or more for async read traffic, and a separate reporting replica.

## When NOT to Use This

Replication adds complexity, cost, and operational risk. Do not use it when:

- **Your data fits on one server and downtime is acceptable.** A single PostgreSQL instance on a big VM serves 100,000 users with 10 ms queries. Backups restore in 15 minutes. Replication adds 2x cost and failover complexity for a problem you do not have.

- **Your write throughput exceeds what one primary can handle.** Replication does not increase write capacity — it distributes reads. If you need 100,000 writes/second, you need sharding, not replication.

- **Your application is an embedded database.** SQLite running on a mobile device does not need replication. Sync to the cloud when network is available.

- **Strong consistency is critical and latency must be low.** Synchronous replication adds at least one network round-trip. If you need both strong consistency and single-digit millisecond writes, use a single-node database with a filesystem-level snapshot for backups.

- **Your team cannot operate a distributed system.** Replication failures (split-brain, lag, slot management, WAL disk full) require experienced DBAs or SREs to diagnose and fix. A single-node database with a cron-based pg_dump is safer for a small team.
