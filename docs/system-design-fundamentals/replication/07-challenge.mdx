# Replication — Challenges

These challenges test your ability to apply replication concepts to real-world scenarios. Each tier increases in difficulty and time commitment. Draw your architecture diagrams on paper or a whiteboard, write out the trade-offs, and then compare against the model answers.

---

## Tier 1 — Foundation (5 minutes each)

These challenges verify that you understand the core mechanisms of replication. Complete all three before moving to Tier 2.

### Challenge 1.1: Read-Your-Writes Consistency

The scenario: A social media application has one primary and two async replicas. When a user posts a comment, the application writes to the primary and then immediately reads from a round-robin load balancer in front of the replicas to display the comment. Users report that their own comments sometimes disappear after posting, only to reappear on the next refresh.

**Your task:** Explain exactly what is happening. Draw the request flow that causes the bug. Propose three different fixes, each with its own trade-off.

*Evaluation criteria:* Do you identify replication lag as the root cause? Do your fixes include (a) session stickiness, (b) primary reads after write, and (c) LSN-based routing? Do you discuss the trade-offs of each approach?

*Common mistakes:* Blaming the application code without understanding replication. Proposing synchronous replication as the only fix. Not explaining why the read replica is behind.

*Model answer key:* Replication lag between the primary and the replica causes the stale read. Fix 1: session stickiness — route all reads from the same session to the same replica, but if that replica is the lagging one, the problem persists. Fix 2: primary reads for N seconds after a write — guarantees consistency but increases primary load. Fix 3: LSN-based routing — the write returns the LSN, and the read layer checks each replica's current LSN before routing; this is the most technically correct but requires application changes.

### Challenge 1.2: WAL Disk Full

The scenario: A production PostgreSQL primary has been running with two replicas for six months. Suddenly, write operations start failing with "No space left on device." The disk monitoring shows the WAL directory is using 95% of a 200 GB volume. The data directory is only 40 GB.

**Your task:** Diagnose the cause. List the immediate steps to restore service. Describe the permanent fix to prevent recurrence.

*Evaluation criteria:* Do you identify a disconnected replication slot as the likely cause? Do you mention checking `pg_replication_slots`? Do you propose dropping the stale slot as an immediate fix? Do you recommend monitoring and alerting on WAL disk usage for the permanent fix?

*Common mistakes:* Trying to manually delete WAL files without checking replication slots. Restarting the primary without clearing the slot. Not understanding that the slot prevents WAL cleanup.

*Model answer key:* A replica disconnected and its replication slot was not removed. The primary kept all WAL segments from LSN of disconnect to present (6 months of data, 160 GB). Immediate fix: identify the stale slot and drop it, then the WAL cleanup runs and frees space. Permanent fix: set `wal_keep_size` to a reasonable limit, set disk usage alerts at 60% and 80%, and implement a script that drops slots for replicas that have been disconnected for more than 24 hours.

### Challenge 1.3: Failover Latency Budget

The scenario: An e-commerce platform requires write availability at 99.99% uptime. A single node failure must cause no more than 30 seconds of write downtime. The database is PostgreSQL running on AWS RDS with Multi-AZ failover.

**Your task:** Walk through every step that happens during a failover and estimate the time for each step. Does the default RDS Multi-AZ failover meet the 30-second budget? If not, what changes would you make?

*Evaluation criteria:* Do you break down the failover into realistic steps (detection, DNS update, TCP connection draining, application retry)? Do you provide time estimates for each? Do you identify the DNS propagation delay as the likely bottleneck? Do you propose solutions like connection poolers with hot standby detection?

*Common mistakes:* Saying "failover takes 30 seconds total" without breaking it down. Ignoring application reconnection time. Assuming DNS updates are instant.

*Model answer key:* Detection (2-5 seconds via health check), DNS record update (30-60 seconds for TTL-based DNS, or instant if using Route 53 health checks), connection pool reconfiguration (1-2 seconds), application connection retries (1-5 seconds depending on backoff strategy). Total with standard DNS: 34-72 seconds, potentially exceeding the budget. Solution: use a connection pooler (PgBouncer) that detects failover and reconnects immediately, bypassing DNS. Or use RDS Proxy, which has sub-second failover detection.

## Tier 2 — Design (15 minutes each)

These challenges require you to design a replication architecture from scratch. Draw the full system diagram and write out your reasoning.

### Challenge 2.1: Global Document Storage Service

The scenario: Design a replication architecture for a global document storage service (similar to Dropbox or Google Drive). The service must support:

- 500 million users worldwide
- Files up to 5 TB each
- Users access files from any device (phone, laptop, desktop)
- Files sync across devices within 5 seconds (p99)
- Conflicts must be detectable and resolvable

**Scale estimates:**

- 500 million users, 100 files per user average = 50 billion files
- Average file size: 2 MB
- Total storage: 100 PB
- 5% of files modified daily = 2.5 billion file modifications

**Your task:** Design the replication strategy for:
1. File metadata (filename, size, permissions, version)
2. File content blocks
3. The sync protocol between devices

For each, state your replication mode (single-leader, multi-leader, leaderless), conflict resolution strategy, and the trade-off you are making.

*Rubric (out of 10):*

| Criteria | 0-2 | 3-5 | 6-8 | 9-10 |
|----------|-----|-----|-----|------|
| Metadata replication | No strategy described | Single-leader basic design | Multi-leader with version vectors | Multi-leader with CRDTs and logical clocks |
| Content replication | No strategy described | Single-region blob store | Multi-region blob store with erasure coding | Content-addressable store with geo-replication |
| Conflict resolution | Not mentioned | Last-writer-wins | Version vectors with application merge | CRDTs with semantic merge and user-facing conflict UI |
| Sync protocol | Not described | Poll-based sync | WebSocket push + diff sync | OT/CRDT-based real-time sync with offline queue |
| Trade-off analysis | None | Mentions one trade-off | Discusses 3+ trade-offs explicitly | Compares 3+ strategies with quantitative reasoning |

*Model answer key:* Metadata uses multi-leader replication with CRDTs (each device is a leader for its own changes). File content uses content-addressable storage (CAS) with geo-replicated blob stores. Conflict resolution: identical files (same hash) are deduplicated; divergent versions create conflicted copies. Sync protocol: WebSocket for real-time notifications, incremental diff for small changes, chunk-level sync for large files. Trade-off: consistency model is "per-file eventual with cross-file causal ordering" — within a file, order is preserved; across files, concurrent edits to different files may conflict but are handled by CRDT merge.

### Challenge 2.2: Real-Time Leaderboard

The scenario: A gaming platform needs a real-time leaderboard that updates scores as players complete matches.

**Requirements:**

- 10 million daily active players
- 50,000 score updates per second at peak
- Leaderboard must be strongly consistent (players must see their correct rank)
- p99 read latency under 50 ms
- p99 write latency under 10 ms

**Your task:** Design the replication architecture for the leaderboard database. Consider that this is a write-heavy workload with very different characteristics from the typical read-heavy pattern.

*Rubric (out of 10):*

| Criteria | 0-2 | 3-5 | 6-8 | 9-10 |
|----------|-----|-----|-----|------|
| Write throughput | Not addressed | Suggests single leader | Single leader with batched writes | Multi-leader or partition-per-leader |
| Read consistency | Not addressed | Suggests all reads from replicas | Reads from primary for consistency | Consistent caching layer with invalidation |
| Architecture | No diagram | Single node | Primary + replicas | Partitioned + replicated + cached |
| Peak handling | Not addressed | "Add more replicas" | Write buffer + batch commit | Sharded counters + in-memory DB + periodic flush |

*Model answer key:* Use Redis with CRDT-based replication (active-active geo-distribution) for the real-time scores. Shard by score range or player ID. Use a write-behind cache pattern: players write to Redis, Redis appends to a WAL-like log, and a background process periodically persists to the durable database. Each shard has its own leader to distribute write load. Leaderboard reads hit the Redis replicas (sub-millisecond latency). Strong consistency is achieved by routing a player's score read to the same shard and leader that handles their writes.

### Challenge 2.3: Multi-Tenant SaaS Database

The scenario: A B2B SaaS platform hosts 5,000 tenants on a shared PostgreSQL cluster. Each tenant has their own schema (schema-per-tenant model). The platform needs to offer a replication SLA to enterprise customers: RPO less than 1 second, RTO less than 30 seconds.

**Your task:** Design a replication strategy that meets the SLA for enterprise tenants while minimizing cost for the other 4,800 tenants. Consider how replication interacts with the multi-tenant architecture.

*Model answer key:* Use a two-tier approach. Standard tenants share a primary with async replicas (RPO: minutes, RTO: 5 minutes). Enterprise tenants get dedicated database instances with synchronous replication (RPO: zero, RTO: 30 seconds). The sync replicas for enterprise tenants are in different availability zones. Failover uses Patroni with etcd quorum. The key trade-off: cost — each enterprise tenant requires at least 2x the database cost (primary + sync replica), so only 200 of the 5,000 tenants opt in.

## Tier 3 — Expert (30 minutes each)

These challenges require deep architectural thinking across multiple system dimensions.

### Challenge 3.1: Offline-First Collaboration Platform

The scenario: Design a replication system for a project management tool (Trello/Asana/Notion analog) where users work offline for extended periods (flights, conferences, field work) and sync when they reconnect.

**Key requirements:**

- Users can create, edit, and delete cards, lists, and boards offline
- Multiple users may edit the same board simultaneously, some online and some offline
- When offline users reconnect, their changes must sync with no data loss
- Users must be able to see what changed while they were offline
- The system must handle 100 million users, 10 billion boards

**Evaluation dimensions:**

- **Conflict resolution model:** How do you handle concurrent edits to the same card by two offline users? By one offline and one online user? When both users reconnect?
- **Replication topology:** Is this multi-leader, single-leader, or something else? What is the "leader" when a device is offline?
- **Version vector design:** What metadata do you track per object? How do you detect conflicts?
- **Merge semantics:** How do you merge list reordering, card moves between lists, and text edits? Are there any operations that cannot be merged automatically?
- **Sync protocol:** What does the client upload when reconnecting? How do you handle large boards (1000+ cards)?
- **Operational concerns:** How do you prevent the server from being overwhelmed when a plane lands and 200 users reconnect simultaneously?

*Anti-patterns to avoid:*
- Using last-writer-wins for all operations (loses data)
- Requiring online connectivity for basic operations (defeats offline-first)
- Single-leader with the server as the only authority (devices must be leaders too)
- Ignoring causality: if a user renames a list and then adds a card to it, the operations are causally related and must be applied in order

*Discussion points:*
- CRDTs versus OT (Operational Transformation): CRDTs are preferred for offline-first collaboration because they do not require a central server for ordering. Google Docs uses OT (central server), while Trello/Asana use a form of CRDT or MVCC.
- The "reconnection storm" problem: exponential backoff with jitter for client sync. Batch the sync into pages of 100 changes. Prioritize metadata over content.
- Semantic versus structural merge: reordering cards in a list is a structural change that CRDTs handle well. Editing card descriptions is a semantic change that may require three-way merge.

### Challenge 3.2: Financial Ledger with Audit Requirements

The scenario: A fintech startup needs a double-entry accounting system that processes 10 million transactions per day. Regulatory requirements demand:

- Every transaction is auditable: you must prove the order of all transactions
- Two-phase commit across accounts is required for transfers
- The system must survive the failure of an entire AWS Availability Zone
- RPO: zero (no transactions can be lost)
- RTO: under 60 seconds
- Write throughput: 500 transactions per second at peak

**Your task:** Design the replication strategy. This is the hardest scenario because financial systems require strong consistency properties that conflict with most replication models.

*Evaluation dimensions:*

- **Transaction ordering guarantee:** How do you ensure all replicas agree on the order of transactions? This is the core challenge. Deterministic database ordering (Calvin, Spanner TrueTime) versus consensus-based ordering (Raft, Paxos).
- **Replication and two-phase commit interaction:** If a transfer debits account A and credits account B across different shards or replicas, how does replication interact with the two-phase commit? Hint: the transaction log must capture both the prepare and commit phases.
- **Disaster recovery:** Two AWS AZs are not enough for an RPO of zero across a full AZ failure — you need a third AZ or a second region. How do you handle the cross-region replication latency?
- **Audit trail:** The replication log itself is the audit trail. How do you guarantee that the log is immutable and cannot be modified retroactively?

*Anti-patterns to avoid:*
- Async replication (violates RPO=zero)
- Multi-leader (conflicts cannot exist in a financial ledger)
- Last-writer-wins (loses the "losing" transaction)
- Single-file WAL with manual rotation (too easy to tamper with)

*Model approach:* Use a single-leader architecture with synchronous replication to at least 2 replicas across 3 AZs. The replication log is written to append-only storage (S3-compatible object store with object lock enabled for immutability). Transaction ordering is deterministic — the primary assigns monotonically increasing sequence numbers before committing. The two-phase commit coordinator is co-located with the primary to minimize latency. Cross-region: asynchronous replication to a second region for read-only disaster recovery; the second region can be promoted only if the first region's data is recovered from the immutable audit log.

This challenge is realistic at the highest engineering levels. Companies like Stripe, Square, and Plaid operate systems with these exact constraints, and their architects make millions of dollars in total compensation because solving this problem correctly is hard.
