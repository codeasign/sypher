# Replication in Production

The theory of replication is well-understood. The practice is where reality diverges from the textbook. This page examines how major engineering organizations use replication in production, the incidents that shaped their architectures, and the open-source and cloud tools that implement replication at scale.

## Company Case Studies

### Netflix: Replicating Cassandra Across 8 Regions

Netflix runs Cassandra as its operational database, storing 50 TB of data across 8 AWS regions with 100+ nodes per region. Their replication strategy is leaderless with tunable consistency.

**The architecture:** Each region runs its own Cassandra ring with a replication factor of 3. Writes use `W=2` (wait for 2 nodes to confirm) and reads use `R=2`, giving consistency 75% of the time. Cross-region replication uses Cassandra's built-in gossip-based replication: each region asynchronously replicates to the other 7 regions.

**The incident that changed everything:** In 2011, a major AWS RDS outage took down Netflix's single-region, single-primary architecture. Netflix was down for hours. This incident drove the migration to Cassandra with multi-region replication. The key lesson was that a single-region primary is a single point of failure, regardless of how many replicas you have inside that region.

**Current state:** Netflix runs Chaos Monkey to randomly kill Cassandra nodes in production. The system survives node failures transparently because every piece of data lives on 3 nodes in every region. Read repair and hinted handoff handle inconsistencies.

**Public reference:** Netflix Tech Blog, "Cassandra at Netflix: Past, Present, and Future" (2018, 2021 presentations at Cassandra Summit).

### Uber: From MySQL to Schemaless and Back

Uber's replication story is a journey from single-primary PostgreSQL to a custom MySQL-based sharded architecture called Schemaless, and eventually to a tiered storage system with DocStore.

**The architecture:** Uber shards its database by city. Each shard is a MySQL primary with 2-3 replicas. Write traffic goes to the shard's primary. Read traffic is distributed across the replicas. Cross-shard queries are handled by the application layer using scatter-gather.

**The replication challenge:** At Uber's scale (15 million trips per day at peak), maintaining replication consistency across shards is the primary challenge. A driver status update and a trip creation must be atomic within a shard but can tolerate eventual consistency across shards. Uber uses asynchronous replication within each shard and synchronous writes only for critical state transitions (trip start, trip end, payment).

**The incident:** In 2014, a replication lag spike during a major promotion caused surge pricing to use stale data for 20 minutes. Drivers were shown incorrect demand maps. Uber's fix was to add a "primary read" option for latency-sensitive queries — any query that affects pricing reads from the primary for that shard.

**Public reference:** Uber Engineering Blog, "Building Schemaless: Uber's Scalable Datastore" (2016), "Uber's Journey to DocStore" (2020).

### Discord: PostgreSQL Replication for Real-Time Messaging

Discord runs one of the largest PostgreSQL deployments in the world: 50+ database clusters, 200+ PostgreSQL instances, serving 140 million daily active users.

**The architecture:** Each Discord guild (server) is assigned to a database shard. Each shard is a PostgreSQL primary with 2 replicas. Discord uses Cascading Replication — replicas that replicate from other replicas rather than the primary — to reduce WAL bandwidth on the primary.

**The replication challenge:** Discord's real-time nature means users expect sub-second propagation of messages across devices. Replication lag of greater than 2 seconds causes users to see messages on one device but not another. Discord monitors lag per-replica with a custom Prometheus exporter and will not serve reads from a replica with lag above 2 seconds.

**The incident:** In 2019, a cascade topology bug caused replicas to fall behind during a traffic spike from a gaming event. WAL replication stalled because the L1 replicas could not keep up with applying writes AND forwarding WAL to L2 replicas. Discord's fix: added more L1 replicas and increased their memory allocation for the WAL receiver buffer.

**Public reference:** Discord Engineering Blog, "How Discord Stores Billions of Messages" (2017), "Scaling Discord" (2021 presentation at PGConf).

### Cloudflare: Replicating Edge Cache State with Durable Objects

Cloudflare's Durable Objects provide strongly consistent storage at the edge — replication across the Cloudflare network of 330+ data centers.

**The architecture:** Each Durable Object is a single JavaScript VM that holds state in memory and persists to a storage engine. Read replicas are not used. Instead, Cloudflare uses a single-leader approach per object, with the leader in one data center and the storage replicated to 3 data centers for durability.

**The replication challenge:** Replicating across 330+ data centers introduces latency that varies from 1 ms (same PoP) to 300 ms (worldwide). Cloudflare cannot use traditional synchronous replication — the latency would be devastating. Instead, each Durable Object's write is acknowledged when 2 of 3 storage copies confirm (quorum W=2), and the object leader handles all reads for strong consistency.

**Public reference:** Cloudflare Blog, "Durable Objects: A New Approach to Edge State" (2020), "Replication at the Edge" (2022 presentation at Cloudflare Meetup).

### Dropbox: Multi-Leader with Custom Conflict Resolution

Dropbox uses a multi-leader replication model where each user's desktop client acts as a leader, and the server mediates conflicts.

**The architecture:** Changes can originate from any device (phone, laptop, desktop, web). The server maintains the authoritative state but allows multiple concurrent writers. When two devices modify the same file simultaneously, Dropbox stores both versions and lets the user resolve the conflict (the "conflicted copy" pattern that creates `filename (Conflicted Copy).ext`).

**The replication challenge:** Dropbox's replication model is unique because the replicas are user devices, not servers. Devices go offline for days or weeks. The server must reconcile divergent state when devices come back online. Dropbox uses a version-vector-based conflict detection system and content-addressable storage (the Magic Pocket system) to avoid storing duplicate data when files are renamed.

**Public reference:** Dropbox Engineering Blog, "Magic Pocket: Infrastructure Behind Dropbox" (2016), "How Dropbox Scales Up" (2018).

## Open Source Implementations

### PostgreSQL Streaming Replication

PostgreSQL's built-in streaming replication is the most widely deployed open-source replication system. It uses a single-leader, WAL-based architecture.

**Strengths:** Zero configuration for basic setup (wal_level = replica, a few lines in recovery.conf). Synchronous and asynchronous modes. Cascading replication. Hot standby (replicas serve read queries).

**Weaknesses:** No built-in automatic failover (requires Patroni or repmgr). Single-threaded WAL apply until PostgreSQL 14 (parallel WAL apply is still limited). No multi-master support (BDR is a separate extension).

### MySQL Group Replication

MySQL's Group Replication provides multi-leader replication with built-in conflict detection, using a Paxos-based group communication system.

**Strengths:** Automatic failover. Multi-leader with conflict detection. Built into MySQL 8.0+. No external coordinator needed.

**Weaknesses:** Performance degrades under write contention (conflict detection adds overhead). Group size limited to 9 nodes. Network partition can cause the entire group to stop (if quorum is lost).

### Cassandra

Apache Cassandra implements leaderless replication with a Dynamo-style architecture. Each node accepts reads and writes. Data is partitioned across nodes using consistent hashing, and each partition is replicated to N nodes (replication factor).

**Strengths:** Linear scalability. No single point of failure. Tunable consistency (per-query W and R settings). Multi-datacenter support built in. Hinted handoff for temporarily unavailable nodes.

**Weaknesses:** Eventual consistency by default. No SQL joins, foreign keys, or transactions. Higher operational complexity. Read repair and anti-entropy operations consume significant CPU.

**Operational trade-offs:**

- **AWS RDS:** Automated backups, automated failover (35-60 seconds), but limited control over replication settings. Cannot set `synchronous_standby_names` directly.
- **GCP Cloud SQL:** Built-in failover (under 60 seconds), automated storage scaling, but cross-region replicas are read-only and cannot be promoted without manual intervention.
- **Azure SQL:** Built-in HA with read-scale, zone-redundant option, but the Business Critical tier is expensive and replica count is fixed.

## Lessons Learned

1. **Failover is the most dangerous operation you will perform.** Every major outage in replication systems (GitHub's 2018 MySQL failover that took 43 seconds but felt like an eternity, Slack's 2020 PostgreSQL failover that caused 30 minutes of degraded performance) occurred during or immediately after a failover. Test failover in staging every week, not once a year.

2. **Replication lag is never zero under load.** Claiming "our replicas are always caught up" is false for any system under sustained write load. Measure your lag in production. Build your application to tolerate at least 10 seconds of lag. If you cannot, accept the latency cost of synchronous replication.

3. **Monitor the replica apply rate, not just the lag.** Lag tells you how far behind a replica is, but apply rate tells you whether the replica is keeping up with the write rate. If the primary writes at 10,000 entries/second and the replica applies at 9,500 entries/second, lag will grow indefinitely — just slowly.

4. **Cascading replication saves primary bandwidth but adds latency risk.** Each cascade level adds 1-5 ms of additional lag. More than 2 levels of cascade are risky. Discord's incident shows that cascade nodes can become bottlenecks.

5. **You need a plan for WAL growth that does not involve the on-call engineer at 3 AM.** Set WAL retention limits. Alert on WAL disk usage at 60% and again at 80%. Have an automated script that drops the oldest replication slots if disk crosses 90%.
