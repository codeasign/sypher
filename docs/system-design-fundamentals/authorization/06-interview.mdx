## Authorization in System Design Interviews

Authorization questions test whether you can design access control systems that are secure, scalable, maintainable, and aligned with business requirements. Unlike authentication questions (who the user is), authorization questions drill into *what the user is allowed to do* — and how you enforce those rules without leaking data, slowing down requests, or creating a maintenance nightmare.

### Why Authorization Questions Are Hard

Three factors make authorization questions difficult:

**Factor 1: There is no single right answer.** Role-based access control (RBAC) works for 80% of systems but breaks down under fine-grained conditions. Attribute-based access control (ABAC) is more expressive but harder to manage. Relationship-based access control (ReBAC) is emerging for social and multi-tenant systems. The interviewer wants to see you pick the right model for the requirements.

**Factor 2: Authorization is a cross-cutting concern.** Every API endpoint, every data query, every background job must enforce authorization. Many candidates design an authorization system that covers the happy path but forgets about batch operations, streaming, asynchronous processing, and admin interfaces.

**Factor 3: Security requirements are non-negotiable.** A flaw in authorization is a security vulnerability, not a performance bug. The interviewer will probe for failure modes: Can one user see another user's data? Can a lower-privilege user escalate? Can a deleted user still access resources?

## Question Types and Phrasings

Authorization questions fall into four categories. Here are 10 phrasings you should be ready for.

### Direct Questions

**Q1: "What is the difference between RBAC and ABAC, and when would you use each?"**

This is a foundational question. The interviewer wants to confirm you understand the core models before diving into a design.

**Q2: "How would you implement authorization for a multi-tenant SaaS application?"**

Multi-tenant authorization is the most common real-world scenario. Data isolation between tenants is the primary concern.

**Q3: "What is the principle of least privilege, and how do you enforce it at scale?"**

Tests your understanding of security fundamentals and their practical implementation.

### Design Questions

**Q4: "Design an authorization system for a document collaboration platform like Google Docs."**

The challenge is granular permissions: read, comment, suggest, edit, share, own — each with different scoping rules. Documents can be nested in folders, shared with individuals, groups, or the entire organization.

**Q5: "Design an access control system for a healthcare platform that must comply with HIPAA."**

Adds regulatory requirements. Certain roles (doctor, nurse, admin, billing) have different access to patient records. Audit logging is mandatory. Break-glass access for emergencies must be supported.

**Q6: "Design a permission system for a payment processing platform with 10,000 merchants."**

Each merchant has their own staff with different roles. Some permissions are global (view reports across all merchants — platform admin), some are merchant-scoped (view transactions for one merchant). Role hierarchy matters.

### Debugging Questions

**Q7: "A user with the 'admin' role cannot access the admin panel. The database shows the role assignment is correct. Walk me through your debugging process."**

Tests your understanding of how authorization is enforced at every layer: middleware, route handlers, database queries, and the frontend.

**Q8: "After a deployment, users report that they can see other users' data. The deployment only changed the API response format. What went wrong?"**

Tests whether you understand how authorization can be silently broken by unrelated changes — for example, a query that no longer filters by tenant ID after a refactor.

**Q9: "Our performance team reports that authorization checks now take 200ms per request, up from 5ms. What changed?"**

Tests your understanding of authorization performance: caching, policy evaluation, database queries, and the overhead of remote calls to an authorization service.

### Trade-off Questions

**Q10: "Should authorization be a centralized service or embedded in each service?"**

The classic distributed systems trade-off: centralized enforcement (consistent, single source of truth, but a potential bottleneck and SPOF) vs. embedded enforcement (fast, no network call, but inconsistent policies across services).

## What the Interviewer Tests

### Security Mindset

The interviewer is evaluating whether you think like a security engineer. This means you do not just design for the happy path — you actively consider attack vectors, privilege escalation, and data leakage.

**Signal**: You mention "default deny" unprompted. You design for the case where a user is not assigned any role.
**Signal**: You discuss audit logging and how to detect unauthorized access attempts.
**Signal**: You consider the case where a service account is compromised and how to limit blast radius.

### Model Selection

Authorization is not a one-size-fits-all problem. The interviewer wants to see that you can match the access control model to the requirements.

**Signal**: You explain why RBAC is sufficient for a simple system and why it would break for a complex one.
**Signal**: You identify when ABAC is needed (fine-grained, attribute-based rules) and the complexity cost it brings.
**Signal**: You discuss hybrid approaches — RBAC for coarse-grained access, ABAC for fine-grained rules within a role.

### Performance Awareness

Authorization must be fast. A check that adds 200ms to every request is unacceptable. The interviewer wants to see that you understand the performance implications of your design.

**Signal**: You discuss caching authorization decisions, not just raw policies.
**Signal**: You mention the overhead of policy evaluation and how to optimize it (pre-computed decision trees, indexed lookups).
**Signal**: You consider the latency of remote authorization calls vs. local enforcement.

### Operational Maturity

Authorization systems must be maintainable. Policies change, new roles are added, and the system must support auditing and debugging.

**Signal**: You discuss how to add a new role without redeploying every service.
**Signal**: You mention how to debug "why was this request denied" with clear error messages.
**Signal**: You discuss how to migrate from one authorization model to another without downtime.

### Scope and Granularity

The hardest part of authorization is getting the scope right. Users have roles, but roles apply within a scope (organization, project, document). The interviewer tests whether you understand scoping.

**Signal**: You distinguish between global roles (admin of the entire platform) and scoped roles (admin of a specific project).
**Signal**: You discuss resource-level permissions (can edit this document) vs. collection-level permissions (can view all documents in this folder).
**Signal**: You handle inheritance — a user with edit access to a folder should have edit access to documents inside it.

## Level Expectations

### L3 (Entry-Level / New Grad)

**Expectation**: Understands the difference between authentication and authorization. Can define RBAC. Knows why you cannot just check user identity directly.

**Good answer**: "Authorization determines what a user can do after they are authenticated. The most common approach is RBAC, where users are assigned roles, and roles have permissions. When a user makes a request, the system checks if their role has the required permission."

**Great answer**: "Authorization is enforcement of access control rules. I would use RBAC because it maps to how organizations naturally structure permissions. Users are assigned roles, roles have permissions, and permissions map to specific actions. The key insight is that roles are a layer of indirection — you do not assign permissions to individual users because that does not scale. When a user's role changes, all their permissions change automatically."

### L4 (Mid-Level)

**Expectation**: Can design a complete authorization system for a single-service application. Understands middleware-based enforcement, role hierarchy, and basic scope management.

**Good answer**: "I would use middleware to enforce authorization. The middleware extracts the user and their roles from the session, checks the required permission for the endpoint, and denies the request if the user lacks the permission. Roles have a hierarchy — admin inherits editor permissions, editor inherits viewer permissions. The permission check is fast because it is a simple lookup in memory."

**Great answer**: "I would implement authorization in three layers. First, route-level middleware validates that the user has the broad category of access (authenticated, admin, public). Second, the endpoint handler checks fine-grained permissions using a policy engine. Third, the data access layer enforces row-level security — even if an endpoint accidentally returns all rows, the database layer filters by tenant ID and user scope. The role hierarchy is defined as a directed acyclic graph: admin subsumes editor, editor subsumes viewer. Permission checks traverse the graph upward. For performance, I cache the user's effective permissions after login and invalidate the cache when the role assignment changes. The cache is local to the application instance with a 5-minute TTL, and a message queue broadcast triggers early invalidation for critical role changes."

### L5 (Senior)

**Expectation**: Can design a distributed authorization system across multiple services. Understands centralized vs. decentralized enforcement trade-offs. Can design for policy-as-code and auditability.

**Good answer**: "I would use a centralized authorization service. Each service calls the authorization service with the user, resource, and action. The authorization service evaluates the policy and returns allow or deny. This ensures consistent enforcement across all services. The authorization service caches policies in memory and evaluates them in under 5ms. For resilience, each service caches the last-known authorization decisions locally with a short TTL."

**Great answer**: "I would use a hybrid approach. A centralized policy service stores and manages all authorization policies. However, enforcement is distributed — each service has a local policy engine that evaluates decisions without a network call. The policies are versioned and distributed via a sidecar proxy or a local policy file that is updated asynchronously. This gives us the consistency of central policy management with the performance of local enforcement.

The authorization service exposes a gRPC API for policy CRUD and a webhook for policy change notifications. When a policy changes, the service broadcasts an event to a message queue. Each service's sidecar subscribes to the queue and updates the local policy cache. The policy format is Rego (from Open Policy Agent), which is declarative, testable, and supports both RBAC and ABAC patterns.

For auditability, every authorization decision is logged to a centralized audit store. The log includes the user, resource, action, decision, policy version, and a unique request ID. This satisfies compliance requirements and enables debugging. The audit store is append-only with immutable logs — even administrators cannot delete audit entries."

### L6 (Staff / Principal)

**Expectation**: Designs authorization systems at the organizational level. Considers regulatory compliance, multi-region data governance, and the operational cost of policy management. Defines the authorization strategy for the entire engineering organization.

**Good answer**: "I would establish an authorization framework that every service must follow. The framework defines standard patterns: all services must use a centralized policy registry, all authorization decisions must be logged, and all services must support a minimum set of roles (admin, writer, reader). Services can extend this framework with service-specific roles and permissions. The framework is enforced through design reviews and automated linting."

**Great answer**: "I would implement a policy-as-code platform. Authorization policies are written in a declarative language (Rego or Cedar), stored in a version-controlled repository, and deployed through a CI/CD pipeline. Every policy change goes through code review and automated testing. The test suite includes unit tests for individual policies, integration tests for end-to-end scenarios, and property-based tests that verify invariants (no user can ever escalate their own permissions, no cross-tenant data access is possible).

The platform supports multiple authorization models. Services choose the model that fits their requirements: RBAC for coarse-grained, ABAC for fine-grained attribute-based, and ReBAC for relationship-based access (used in social features and document sharing). The platform provides a unified API so the choice of model is transparent to the calling service.

For compliance, the platform generates reports automatically. A SOC 2 auditor can request "list all users who accessed patient records in the last 30 days" and the platform produces the report in minutes. Break-glass access (emergency override of normal permissions) is supported with mandatory justification, time-limited access, and post-hoc review by a security team.

The operational cost of the platform is tracked through metrics: policy evaluation latency (p99 under 5ms), cache hit ratio (target 99.5%), and false positive rate (policies that deny legitimate requests, tracked through a feedback loop)."

### L7 / L8 (Distinguished / Fellow / VP)

**Expectation**: Sets the authorization strategy for the entire organization. Defines the security posture, regulatory compliance approach, and engineering culture around access control. Influences industry standards.

**Good answer**: "Authorization is a first-class concern in our software development lifecycle. Every design document includes a section on access control. Every security review evaluates the authorization model. Every incident response includes a review of whether authorization was bypassed. The authorization strategy is driven by data classification — the more sensitive the data, the stricter the authorization model."

**Great answer**: "I would establish a zero-trust authorization architecture. No user, service, or device is implicitly trusted. Every request is authenticated, authorized, and logged. Authorization decisions are based on multiple signals: the user's identity, their role, the resource being accessed, the context of the request (time of day, location, device), and the risk score of the action.

The architecture is built on three pillars:

**Pillar 1: Policy as code.** All authorization policies are version-controlled, tested, and deployed through CI/CD. The policy repository is the single source of truth. Every policy has an owner and a review date. Stale policies are automatically flagged.

**Pillar 2: Continuous verification.** Authorization is not a one-time check at login. Every API request is re-authorized. Session tokens have short lifetimes (15 minutes for humans, 5 minutes for services). Privileged actions require step-up authentication.

**Pillar 3: Audit and response.** Every authorization decision is logged. The audit log is immutable and retains for 7 years to meet regulatory requirements. Automated alerts fire on suspicious patterns: a user accessing resources they have never accessed before, a user denied access multiple times in a short window, or a role assignment that grants excessive permissions.

The organizational impact is significant. Authorization is no longer an afterthought — it is a core architectural concern. The platform team invests in tooling that makes it easy for product teams to implement correct authorization. The security team focuses on threat modeling and policy review rather than manually auditing every endpoint. The result is a system that is secure by default, auditable by design, and maintainable at scale."

## FAANG Interview Framework for Authorization

Use this six-step framework when answering authorization design questions. The framework ensures you cover all aspects of the system without missing critical details.

### Step 1: Requirements Gathering

Start by understanding the authorization requirements before designing the system.

**Ask the interviewer:**
- What types of resources need protection? (documents, API endpoints, database rows, infrastructure)
- Who are the users? (individuals, organizations, service accounts, external partners)
- What are the access patterns? (read, write, delete, share, administer)
- Are there regulatory requirements? (HIPAA, SOC 2, GDPR, PCI-DSS)
- What is the scale? (number of users, resources, permission checks per second)
- Is there an existing authorization model to migrate from?

**Document:**
- Resource hierarchy: organization > project > folder > document
- User roles: admin, editor, viewer, custom
- Permission types: create, read, update, delete, share, manage
- Scoping requirements: roles are scoped to a resource level

### Step 2: Choose the Authorization Model

Select the appropriate access control model based on the requirements.

**RBAC** — Use when:
- Permissions are coarse-grained (read, write, admin)
- Role hierarchy is natural (admin > editor > viewer)
- Number of roles is small and stable (under 50)
- Example: internal admin panel, simple CMS

**ABAC** — Use when:
- Permissions depend on user attributes, resource attributes, or environment attributes
- Rules are complex (allow read if user.department = resource.department AND user.clearance >= resource.classification)
- Policy changes are frequent and need to be made without code changes
- Example: healthcare system, financial trading platform

**ReBAC** — Use when:
- Permissions depend on relationships between users and resources (owner, collaborator, shared-with)
- Resources are organized in a hierarchy with inheritance
- Examples: document sharing, social media, cloud storage

**Hybrid** — Use when:
- Coarse-grained roles are needed for broad access categories
- Fine-grained rules are needed within a role scope
- This is the most common real-world approach

### Step 3: Design the Data Model

Design the data structures that represent users, roles, permissions, and resources.

**Core tables:**
```
users: id, name, email, organization_id
roles: id, name, description, is_system_role
permissions: id, action, resource_type, scope
role_permissions: role_id, permission_id
user_roles: user_id, role_id, scope_type, scope_id
resource_hierarchy: resource_id, parent_id, resource_type
```

**Policy evaluation:**
```
function check_permission(user, action, resource):
    roles = get_user_roles_in_scope(user, resource.scope)
    for role in roles:
        permissions = get_role_permissions(role)
        if (action, resource.type) in permissions:
            return ALLOW
    return DENY
```

**Key design decisions:**
- Role assignments are scoped (user has admin role on organization X, viewer role on organization Y)
- Permission inheritance traverses the resource hierarchy (edit on folder implies edit on documents inside)
- Deny rules override allow rules (explicit deny for specific resources)
- Default deny is the baseline — no explicit permission means deny

### Step 4: Design the Enforcement Architecture

Design how authorization is enforced at every layer of the system.

**API Gateway / Middleware Layer:**
- Validates authentication token
- Extracts user identity and claims
- Passes user context to downstream services

**Service Layer:**
- Calls authorization service or local policy engine
- Passes user, action, resource, and context
- Receives allow/deny decision
- Denies request with 403 Forbidden if denied

**Database Layer:**
- Row-level security (RLS) filters query results by tenant ID
- Parameterized queries prevent SQL injection that could bypass authorization
- Views or materialized views pre-filter data by user scope

**Data Access Layer:**
- Repository pattern enforces authorization in the data access method
- Every query includes a tenant ID filter
- Batch operations check authorization for each resource in the batch

### Step 5: Plan for Performance and Scale

Authorization must be fast. Design for low latency and high throughput.

**Caching strategy:**
- Cache user permissions at login with a session-level cache
- Cache role definitions in application memory (reload on change)
- Cache policy evaluation results for frequently accessed resources
- Use write-through cache for role assignments to maintain consistency

**Latency budget:**
- Local policy evaluation: under 1ms
- Remote authorization service call: under 5ms (including network)
- Database-level RLS: under 1ms (indexed tenant ID filter)
- Total authorization overhead: under 10ms per request

**Scaling considerations:**
- Authorization service is stateless and horizontally scalable
- Policy cache is distributed via Redis or Memcached
- Cache invalidation uses pub/sub for near-real-time updates
- Read replicas serve policy queries for non-critical authorization checks

### Step 6: Plan for Operational Excellence

Design for maintainability, auditability, and failure modes.

**Audit logging:**
- Log every authorization decision: user, action, resource, decision, timestamp
- Log policy changes: who changed what and when
- Log role assignments: who granted which role to whom
- Store logs in immutable, append-only storage

**Debugging:**
- Provide a "why was this denied" endpoint for debugging
- Include the policy evaluation trace in the error response (for admin users only)
- Support dry-run mode where requests are evaluated but not enforced

**Failure modes:**
- If the authorization service is unreachable, fall back to cached policies (degraded mode)
- If the cache is empty, deny by default (fail closed, not fail open)
- If the policy cannot be parsed, deny the request and alert the operations team
- Regularly test authorization failure scenarios in non-production environments

**Testing:**
- Unit tests for individual policies
- Integration tests for end-to-end authorization flows
- Regression tests for known security issues
- Penetration testing for authorization bypass attempts
- Compliance audits for regulatory requirements

## Red Flags — 8+ Signs of a Weak Answer

### Red Flag 1: Confusing Authentication with Authorization

**What it looks like**: "The user logs in with their credentials, and then we check the session to see who they are."

**Why it matters**: This describes authentication, not authorization. Authentication answers "who are you?" Authorization answers "what are you allowed to do?" The interviewer needs to see that you understand the distinction.

**Fix**: "After authentication, the system checks authorization — what resources is this user allowed to access? Authentication verifies identity, authorization verifies permissions."

### Red Flag 2: Permissions Assigned Directly to Users

**What it looks like**: "Each user has a list of permissions stored in the user record."

**Why it matters**: This does not scale. With 1,000 users and 10 permissions each, changing a permission type requires updating 1,000 records. Role-based systems are the standard for a reason.

**Fix**: "Users are assigned to roles. Roles have permissions. If the permission for 'edit document' changes, I update one role definition, not every user's record."

### Red Flag 3: No Scope Management

**What it looks like**: "A user with the admin role can do anything."

**Why it matters**: In multi-tenant systems, an admin of one organization should not be an admin of another organization. Scope is essential.

**Fix**: "Roles are scoped. A user has the admin role within the scope of organization X. The same user may have the viewer role within the scope of organization Y. Every permission check includes the scope."

### Red Flag 4: No Mention of Default Deny

**What it looks like**: "The user has permissions a, b, and c. If the action is one of those, they are allowed."

**Why it matters**: What happens if the user has no permissions at all? What happens if the action is not in the list? Without an explicit default deny, the system might accidentally allow access.

**Fix**: "The default is deny. If no policy explicitly allows the action, the request is denied. This is the fundamental security posture."

### Red Flag 5: No Performance Considerations

**What it looks like**: "The authorization service checks each permission by querying the database."

**Why it matters**: A database query per authorization check does not scale to thousands of requests per second. Caching, in-memory evaluation, and pre-computed policies are necessary.

**Fix**: "Permissions are cached in memory. The policy engine evaluates decisions without a database query for the common case. Cache invalidation is handled through an event-driven mechanism."

### Red Flag 6: No Audit Trail

**What it looks like**: "If there is a problem, we can check the logs."

**Why it matters**: Authorization decisions must be auditable for security and compliance. Vague logging is not sufficient. The audit trail must be complete, immutable, and queryable.

**Fix**: "Every authorization decision is logged with the user, resource, action, decision, and a unique request ID. The audit log is immutable and stored in a separate system from the main database."

### Red Flag 7: Fail Open Mentality

**What it looks like**: "If the authorization service is down, we let the request through to avoid downtime."

**Why it matters**: This is a security vulnerability. An attacker could disable the authorization service and gain unrestricted access.

**Fix**: "The system fails closed. If the authorization service cannot be reached, the request is denied. We cache policies locally to handle temporary outages without blocking all requests."

### Red Flag 8: No Migration Strategy

**What it looks like**: "We rewrite the authorization system from scratch and deploy it."

**Why it matters**: Authorization systems cannot be replaced in a single deployment. The risk of a security gap during migration is too high. A phased migration with dual evaluation is needed.

**Fix**: "I would migrate in phases. Phase 1: run the new authorization system in parallel with the old one, comparing results but not enforcing. Phase 2: enforce the new system but log every decision that would have been different under the old system. Phase 3: retire the old system after validating the new system in production."

### Red Flag 9: No Discussion of Hierarchy and Inheritance

**What it looks like**: "Permissions are checked for each resource individually."

**Why it matters**: In real systems, resources are organized in hierarchies (organization > team > project > document). Granting access to a parent should grant access to children. Without inheritance, the system is unmanageable.

**Fix**: "Permissions inherit through the resource hierarchy. If a user has read access to a folder, they can read all documents in that folder. Inheritance is transitive — access to an organization grants access to all teams, projects, and documents within it."

### Red Flag 10: No Consideration of Service Accounts

**What it looks like**: "Authorization is only about human users."

**Why it matters**: Real systems have service accounts, automated jobs, CI/CD pipelines, and third-party integrations. Each needs authorization with least privilege.

**Fix**: "Authorization applies to all principals, not just human users. Service accounts have their own roles and permissions. Machine-to-machine communication uses API keys or OAuth client credentials, each with a defined scope of access."

## Practice Drills — 5 Prompts

### Drill 1: Model Selection

For each scenario, determine the appropriate authorization model (RBAC, ABAC, ReBAC, or hybrid) and justify your choice.

**Scenario A**: A company's internal expense reporting system. Employees can submit expenses. Managers can approve expenses for their direct reports. Finance team can view all expenses. Admins can configure the system.

**Scenario B**: A cloud file storage platform. Users can create folders and files. Users can share files with individuals or groups. Sharing can be view-only, comment-only, or edit. Files inherit permissions from their parent folder.

**Scenario C**: A hospital patient record system. Doctors can view and update records for their assigned patients. Nurses can view records and add observations. Billing staff can view billing-related fields only. Emergency override allows any doctor to access any patient record with mandatory audit logging.

**Scenario D**: A multi-tenant e-commerce platform. Each tenant (merchant) has their own staff with roles. Platform admins can view any tenant's data for support. Merchants can customize roles for their staff.

**Scenario E**: A financial trading system. Traders can view market data and execute trades within their risk limits. Risk limits depend on trader's experience level, account balance, and the type of instrument. Compliance officers can view all trades but cannot trade. Automated trading bots have their own authorization with strict rate limits.

### Drill 2: Design a Permission Check Endpoint

Design the API endpoint for checking whether a user has permission to perform an action on a resource. Consider the following requirements:

- The endpoint must return allow or deny in under 10ms
- The endpoint must support dry-run mode (evaluate without enforcing)
- The endpoint must include the policy trace (which policy matched, which policy denied)
- The endpoint must be idempotent

**Define:**
- Request and response format
- Caching strategy
- Database queries
- Error handling
- Rate limiting

### Drill 3: Debug the Authorization Bug

A production system has the following behavior:

- A user with the "editor" role on a document can edit the document
- A user with the "viewer" role on a document can only view the document
- A user with the "admin" role on the organization can access any document
- Bug: A user with the "viewer" role on document A can also edit document A
- The role assignment in the database is correct (viewer role, not editor)

**Debugging approach:**
1. Check the policy evaluation trace for the affected user
2. Check if the user has any other role assignments (maybe through a group membership?)
3. Check if the document has inheritance that gives the user edit access through a parent resource
4. Check if the policy has a bug (e.g., a condition that always evaluates to true)
5. Check if the cache is returning stale data
6. Check if the database-level RLS is misconfigured

**Trace the exact cause using the data model:**
- What tables do you query?
- What indexes do you check?
- What caching layers do you inspect?

### Drill 4: Performance Optimization

Your authorization system is processing 10,000 requests per second. The current latency is 50ms per authorization check. The target is under 5ms.

**Identify the bottlenecks:**
- For each potential bottleneck, describe how you would measure it
- Propose specific optimizations with expected latency improvements

**Current architecture:**
- Each request calls a centralized authorization service via HTTP
- The authorization service queries a PostgreSQL database for the user's roles
- The authorization service queries the database for the resource's permissions
- The authorization service evaluates the policy in Python
- The response is returned to the calling service

**Optimization ideas to consider:**
- Caching user roles in Redis (reduce database queries)
- Caching policy evaluation results (reduce computation)
- Moving from HTTP to gRPC (reduce serialization overhead)
- Pre-computing effective permissions on role assignment (reduce evaluation cost)
- Using a local policy engine (eliminate network call)
- Adding a CDN for public resource permissions (edge caching)

### Drill 5: Migration Strategy

You are migrating from a legacy authorization system to a new system. The legacy system stores permissions directly on user records. The new system uses RBAC with a centralized policy service.

**Migration requirements:**
- Zero downtime
- Zero security gaps during migration
- All existing permissions must be preserved
- The migration must complete within 3 months
- 500 services must be migrated

**Design the migration plan:**

Phase 1 — Audit:
- Extract all existing permissions from the legacy system
- Map each permission to a role in the new system
- Identify any permissions that do not fit the RBAC model
- Identify any stale permissions (users who no longer exist, resources that no longer exist)

Phase 2 — Parallel Run:
- Deploy the new authorization system in parallel
- Route all requests through both systems
- Compare results — log any discrepancy
- Fix discrepancies by adjusting policies or role assignments
- Do not enforce the new system yet

Phase 3 — Shadow Enforcement:
- Switch to the new system for enforcement
- Log decisions from the legacy system for comparison
- Monitor for authorization errors (false positives and false negatives)
- Roll back to the legacy system if error rate exceeds threshold

Phase 4 — Decommission:
- Verify that the legacy system has no discrepancies for 30 days
- Archive the legacy audit logs
- Remove the legacy authorization code
- Update documentation and runbooks

## Full Interview Walkthrough

Here is a complete mock interview showing how to apply the framework.

### Question: "Design an authorization system for a document collaboration platform like Google Docs."

**Think about:**
- Documents can be created, viewed, commented on, suggested, edited, and shared
- Documents are organized in folders
- Users can share documents with individuals, groups, or the entire organization
- Some users are organization admins who can manage all documents
- Audit logging is required for compliance
- The platform has 10 million users and 100 million documents

**Sample Answer (Intermediate Level):**

"I would start with the requirements. The key resources are documents and folders. The key actions are create, read, comment, suggest, edit, share, delete, and administer. The actors are individual users, groups, and the organization itself.

For the authorization model, I would use a hybrid approach. RBAC handles the broad roles: admin, editor, commenter, viewer. ABAC handles fine-grained rules like 'can edit only if the document is in draft status' or 'can share only if the user is the document owner.'

The data model has a users table, a groups table, a documents table, and a permissions table. The permissions table links a principal (user or group) to a resource (document or folder) with a specific role. For example: user_id=123, resource_id=doc456, role=editor.

Permission inheritance works through the folder hierarchy. If a user has editor access to a folder, they inherit editor access to all documents in that folder. The inheritance is evaluated at query time by traversing the resource hierarchy.

Permission checking works like this: when a user tries to edit a document, the system checks if the user has a direct permission on the document for the edit action. If not, it checks the parent folder. If not, it checks the grandparent folder, up to the organization level. If no permission is found, the request is denied.

For performance, I cache the user's permissions for each resource they access. The cache is keyed by user_id + resource_id and has a 5-minute TTL. When a permission is changed, the cache is invalidated for the affected user and resource.

For audit logging, every authorization decision is logged to an append-only audit store. The log includes the user, document, action, decision, and timestamp."

**Sample Answer (Senior Level):**

"Let me approach this systematically using the six-step framework.

**Step 1: Requirements**

The platform has a resource hierarchy: organization > team > folder > document. The key actions are create, read, comment, suggest, edit, share, delete, and administer. The principals are users, groups, and the organization itself. The platform needs to support 10 million users and 100 million documents with a p99 authorization latency under 10ms.

**Step 2: Authorization Model**

I would use a hybrid model. The core is relationship-based access control (ReBAC), which naturally models the sharing patterns of a document platform. In ReBAC, permissions are defined by relationships between users and resources: user A is an editor of document B, user C is a viewer of folder D. The relationship graph is traversed at evaluation time.

On top of ReBAC, I add RBAC for organization-level roles. Organization admins have a special relationship that grants them access to all documents in the organization.

**Step 3: Data Model**

The core table is the relationship table:

```
relationships:
  principal_type: enum(user, group, organization)
  principal_id: uuid
  resource_type: enum(document, folder, team, organization)
  resource_id: uuid
  relation: enum(owner, editor, commenter, viewer, admin)
  created_at: timestamp
  expires_at: timestamp (optional, for time-limited access)
```

The relationship table is indexed on (principal_id, resource_id) for direct permission checks, and on (resource_id, relation) for finding all users with a specific relation to a resource.

The resource hierarchy table tracks parent-child relationships:

```
resource_hierarchy:
  resource_type: enum(document, folder, team, organization)
  resource_id: uuid
  parent_resource_type: enum(folder, team, organization)
  parent_resource_id: uuid
  level: integer (0 for root, incremented for each level down)
```

**Step 4: Enforcement Architecture**

Authorization is enforced at three layers:

1. **API Gateway**: Validates the authentication token and extracts the user's identity. Passes the user context to downstream services via HTTP headers.

2. **Service Layer**: Each service calls a local policy engine that evaluates permissions against the cached relationship graph. The policy engine is a sidecar process that loads the relationship cache from a local Redis instance.

3. **Database Layer**: Row-level security filters query results by the user's organization ID. This is a defense-in-depth measure — even if the service layer is bypassed, the database prevents cross-organization data access.

**Step 5: Performance**

The relationship cache is the key to performance. Relationships are cached in Redis with the following structure:

- Key: `rels:{user_id}:{resource_type}:{resource_id}`
- Value: list of relations (owner, editor, viewer)
- TTL: 5 minutes, refreshed on access

For inheritance, the cache also stores the resource hierarchy path. When checking permission for a document, the system checks the document's relationships, then the folder's relationships, then the team's relationships, up to the organization level. Each level is a single Redis lookup.

The average permission check requires 2 Redis lookups (document + folder). The p99 is 4 lookups (document + folder + team + organization). At 1ms per Redis lookup, the p99 authorization latency is under 5ms.

**Step 6: Operational Excellence**

Audit logging is implemented as a separate service that receives a stream of authorization decisions from a message queue. The audit service writes to an append-only log stored in S3 with Glacier archival for older entries.

For debugging, the system supports a `X-Auth-Debug` header. When an admin user includes this header, the response includes the authorization trace: which relationships were checked, which policies matched, and which decision was made.

For failure modes, the system fails closed. If the local policy engine cannot reach Redis, it uses a last-known-good cache that is persisted to disk. If the disk cache is also unavailable, the request is denied with a 503 error.

The system is tested through regular chaos experiments. Every week, a random service's authorization cache is cleared to test the cache rebuild path. Every month, the authorization service is taken offline to verify that the fail-closed mechanism works correctly."

## Common Follow-Up Questions

These questions may come after your initial answer. Have answers prepared.

### "How do you handle permission inheritance in a deeply nested hierarchy?"

Key points: Use a materialized path or nested set model. The resource hierarchy is stored as a tree. At evaluation time, traverse from the resource up to the root, checking for permissions at each level. Cache the path for frequently accessed resources. For very deep hierarchies (10+ levels), consider pre-computing the effective permissions on write rather than evaluating on read.

### "What if two users edit the same document simultaneously and they have different permission levels?"

Key points: Permissions are evaluated at the time of the request, not at the time the document is loaded. If a user's permission is revoked while they are editing, the next save operation should be rejected. This is enforced by checking authorization on every write operation, not just on the initial load. The user should see a message that their access has been revoked and their changes cannot be saved.

### "How do you handle group membership changes?"

Key points: When a user is added to or removed from a group, all permissions that were granted through the group must be updated. This is handled by the relationship cache: the cache key includes the principal type (user or group). When a group membership changes, the cache for that group is invalidated, and all users who belong to the group will have their effective permissions recalculated on the next request. For large groups (10,000+ members), use a batched cache invalidation approach to avoid a thundering herd.

### "How do you audit a specific user's access across all resources?"

Key points: This is a common compliance requirement. The audit log must be queryable by user ID. The relationship table is indexed by principal_id, so you can list all relationships for a user. For historical access, the audit log provides a complete record of every authorization decision. The audit log is stored in a columnar format (Parquet) for efficient querying. For compliance reports, pre-compute daily snapshots of all user-role assignments.

### "How do you implement time-limited access?"

Key points: Add an `expires_at` field to the relationship table. The policy engine checks `expires_at IS NULL OR expires_at > NOW()` as part of the evaluation. For time-limited access, the relationship is created with an expiration timestamp. The cache includes the expiration in the cache key, so the evaluation is correct even if the cache is stale. A background job cleans up expired relationships and invalidates the cache.

### "What if a user is both a direct editor and a group viewer of the same document?"

Key points: The most permissive role wins by default, but this depends on the requirements. For a document platform, having more access through a direct relationship is expected. If the requirement is deny-overrides, then an explicit deny on any relationship chain overrides any allow. The key is to define the policy evaluation rules clearly: allow-overrides (most permissive wins) or deny-overrides (most restrictive wins). Most systems use allow-overrides with explicit deny support.

## What Interviewers Really Want

At the end of any authorization-focused question, the interviewer is asking themselves:

1. **Does this candidate understand that authorization is a security concern, not just a feature?** You should talk about default deny, fail-closed behavior, and audit logging unprompted.

2. **Can this candidate match the authorization model to the requirements?** You should not propose RBAC for every problem. You should evaluate the requirements and choose the appropriate model, explaining why.

3. **Does this candidate think about real-world operational concerns?** You should discuss caching, performance, migration, and debugging. The interviewer wants to know you have operated an authorization system, not just read about one.

4. **Can this candidate design for scale and security simultaneously?** Authorization is a cross-cutting concern that touches every request. Your design must be both fast and secure. The interviewer wants to see you balance these competing concerns.

5. **Does this candidate understand the failure modes?** You should discuss what happens when the authorization service is down, when the cache is stale, and when a policy is misconfigured. The interviewer wants to see that you have thought about what breaks.

If you hit all five signals, you will ace the authorization question at any company.
