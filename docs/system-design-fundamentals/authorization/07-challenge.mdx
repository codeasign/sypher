## Tier 1: Foundation

These challenges test your grasp of the basic concepts and terminology. You should be able to answer each in 3-5 minutes.

### Challenge 1.1: RBAC versus ABAC

Your team is building a document management platform for a mid-sized company with 200 employees. The platform has five departments: Engineering, Sales, HR, Finance, and Legal. The current authorization system uses simple boolean flags (canRead, canWrite, canDelete) assigned per user per document, and it has become unmanageable.

You are asked to evaluate Role-Based Access Control (RBAC) and Attribute-Based Access Control (ABAC) for this system.

**Question:** Explain the key differences between RBAC and ABAC. For each model, describe how you would model the following scenario: "Only HR managers can view salary documents in the HR folder." Then recommend which model fits this company and state your reasoning.

<details>
<summary>Model Answer</summary>

**RBAC (Role-Based Access Control):**

RBAC grants permissions based on a user's role within the organization. Permissions are pre-assigned to roles, and users are assigned to roles.

Key characteristics:
- Roles are static and predefined (Employee, Manager, HR Manager, Admin).
- Permissions are coarse-grained (read document, write document, delete document).
- Adding a new user means assigning them to a role -- all permissions follow automatically.
- Auditing is straightforward: you inspect the role-permission matrix.

For the scenario "Only HR managers can view salary documents in the HR folder," you would create a special role:

```
Role: HR_Manager
Permissions:
  - read (folder: /hr/salary) = true
  - write (folder: /hr/salary) = true

Role: HR_Employee
Permissions:
  - read (folder: /hr/salary) = false
  - read (folder: /hr/general) = true
```

This works, but it requires creating granular roles for every combination of role and resource boundary. If Finance later wants a similar rule, you need a new role: Finance_Manager with salary document permissions.

**ABAC (Attribute-Based Access Control):**

ABAC evaluates access based on attributes of the user, resource, action, and environment. A policy engine evaluates boolean expressions at request time.

Key characteristics:
- Policies are dynamic expressions, not static role-permission maps.
- Attributes can be anything: department, seniority, document classification, time of day, location.
- Fine-grained control without creating new roles.
- Policy evaluation happens at runtime: every request evaluates potentially dozens of attribute conditions.

For the same scenario, you write a single policy rule:

```
PERMIT access IF:
  user.department == "HR"
  AND user.level >= "manager"
  AND resource.type == "salary_document"
  AND resource.folder == "/hr/salary"
  AND action == "read"
```

This one rule covers HR managers. If Finance later wants a similar rule, you add one condition:

```
PERMIT access IF:
  (user.department == "HR" OR user.department == "Finance")
  AND user.level >= "manager"
  AND resource.type == "salary_document"
  AND resource.folder == "/hr/salary" OR resource.folder == "/finance/salary"
  AND action == "read"
```

**Recommendation:** Start with RBAC. For a 200-person, 5-department company, RBAC is simpler, faster to implement, and easier to audit. Define roles for each department (HR_Employee, HR_Manager, Engineering_Employee, Engineering_Manager, and so on) and assign document-level permissions per role. Move to ABAC only if you hit one of these triggers:
- You need cross-cutting policies that span departments (for example, "any manager can view any document in their department").
- You need context-aware rules (for example, "only allow access during business hours from the office network").
- The number of roles exceeds 50 and the role-permission matrix becomes hard to manage.

RBAC is the 80-percent solution. ABAC handles the remaining 20 percent of edge cases at the cost of higher system complexity and a policy evaluation engine.

</details>

### Challenge 1.2: Policy Evaluation Walkthrough

A cloud storage platform uses a policy-based authorization engine inspired by AWS IAM. Each request triggers the evaluation of multiple policies. The system has three types of policies:

- Identity-based policies: attached to the user or their group.
- Resource-based policies: attached to the resource being accessed.
- Deny override: any explicit deny statement overrides any allow.

Consider this concrete scenario:

```
User: Alice
Groups: Engineering, Project-X
Resource: S3 bucket named "proj-x-data"
Action: s3:GetObject

Identity policies applicable to Alice:

Policy A (attached to Engineering group):
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::proj-x-data/*"
}

Policy B (attached to Alice directly):
{
  "Effect": "Deny",
  "Action": "s3:*",
  "Resource": "arn:aws:s3:::proj-x-data/secret-config.json"
}

Resource policy on the bucket:

Policy C (resource-based):
{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::proj-x-data/*",
  "Condition": {"IpAddress": {"aws:SourceIp": "10.0.0.0/8"}}
}
```

Alice sends a request from IP address 10.0.1.50 to read the object `proj-x-data/reports/2025-q1.json`.

**Question:** Walk through the policy evaluation step by step. Determine the final decision (allow or deny) and explain which policies matched, which did not, and why.

<details>
<summary>Model Answer</summary>

**Step 1: Gather applicable policies.**

Collect all identity-based policies that apply to Alice:
- Policy A: from the Engineering group. Applies to Alice because she is a member of Engineering.
- Policy B: attached directly to Alice. Applies to Alice.

Collect all resource-based policies that apply to the target resource:
- Policy C: attached to the bucket `proj-x-data`. Applies because the resource is in this bucket.

**Step 2: Determine the decision for each policy.**

Policy A evaluation:
- Effect: Allow
- Action: `s3:GetObject` --- Alice is requesting `s3:GetObject`. Match.
- Resource: `arn:aws:s3:::proj-x-data/*` --- the requested object is `proj-x-data/reports/2025-q1.json`, which matches `proj-x-data/*`. Match.
- Result: **Allow.**

Policy B evaluation:
- Effect: Deny
- Action: `s3:*` --- Alice is requesting `s3:GetObject`, which matches `s3:*`. Match.
- Resource: `arn:aws:s3:::proj-x-data/secret-config.json` --- Alice is requesting `proj-x-data/reports/2025-q1.json`, which does NOT match the specific path `secret-config.json`. No match.
- Result: **Not applicable** (the resource does not match).

Policy C evaluation:
- Effect: Allow
- Action: `s3:GetObject` --- Match.
- Resource: `arn:aws:s3:::proj-x-data/*` --- Match.
- Condition: `{"IpAddress": {"aws:SourceIp": "10.0.0.0/8"}}` --- Alice's IP is 10.0.1.50, which falls within the CIDR range 10.0.0.0/8. Match.
- Result: **Allow.**

**Step 3: Combine decisions with deny override.**

Applicable decisions:
- Policy A: Allow
- Policy B: Not applicable (skipped)
- Policy C: Allow

Rules for combining:
1. If any applicable policy results in an explicit Deny, the final decision is Deny (deny override).
2. If at least one applicable policy results in Allow and no policy results in Deny, the final decision is Allow.
3. If no policy results in Allow or Deny (implicit deny), the final decision is Deny.

In this case, there is no explicit Deny (Policy B did not match because the resource did not match). Policy A and Policy C both result in Allow.

**Final decision: Allow.** Alice can read `proj-x-data/reports/2025-q1.json` from IP 10.0.1.50.

**Key learning point:** The deny in Policy B is narrowly scoped to a single file (`secret-config.json`). If Alice had requested `secret-config.json`, Policy B would have applied and the result would have been Deny, overriding both Allow decisions from Policies A and C.

**Evaluation order:**

```
Request: Alice, s3:GetObject, proj-x-data/reports/2025-q1.json, IP=10.0.1.50

Identity policies (Engineering group):
  Policy A → Allow ✓ (effect=Allow, action=match, resource=match)

Identity policies (direct):
  Policy B → N/A ✗ (resource does not match)

Resource policies (bucket):
  Policy C → Allow ✓ (effect=Allow, action=match, resource=match, condition=match)

Combine:
  Deny encountered? No → Allow
  Allow encountered? Yes → Allow

Result: ALLOW
```

</details>

### Challenge 1.3: The Least Privilege Violation

You join a company as a security engineer. During your first week, you audit the production authorization system and discover the following configuration:

```
Database: PostgreSQL with 200 tables

Service account: "api-server-prod"
Assigned role: "db_admin" (superuser)
  - Can create/drop databases
  - Can read, write, truncate, and drop any table
  - Can modify system settings
  - Can create users and grant privileges

Number of microservices: 12
Number of service accounts: 12 (one per microservice)
Number of service accounts with "db_admin" role: 12
```

The explanation you receive: "We gave everyone db_admin because developers kept hitting permission errors, and it was faster than debugging each one."

**Question:** Explain the principle of least privilege. Identify at least four specific violations in this setup. Then describe a concrete remediation plan, including the role design and migration approach, that does not require rewriting any application code.

<details>
<summary>Model Answer</summary>

**The principle of least privilege:**

Every user, process, or system component should have the minimum set of permissions required to perform its function and nothing more. The principle applies at four levels:
- What a user can do (actions).
- What resources a user can access (scope).
- When a user can access resources (time-bound).
- How a user can access resources (channel or method).

**Violations in this setup:**

1. **Superuser access for every service.** Twelve microservices, each with a vastly different function (API gateway, background workers, reporting, analytics, and so on), all share the same superuser role. A reporting service that only needs read access to 3 tables can drop entire databases.

2. **No read-write separation.** A service that reads cached session data should never have write access to the user accounts table. Giving every service full DDL and DML privileges means any compromised service can destroy the entire database.

3. **Unlimited DDL privileges.** Most services should never create, alter, or drop tables. Granting CREATE, ALTER, TRUNCATE, and DROP to every service means a bug in any service could result in data loss. Dropping a table should require a separate deployment with a privileged migration account.

4. **No resource scoping.** Each microservice should have access only to the tables it needs. For example:
   - User service: `users`, `sessions`, `profiles`
   - Orders service: `orders`, `order_items`, `payments`
   - Inventory service: `products`, `inventory`
   - Reporting service: read-only on `orders`, `products` (no write)

5. **Granting privileges to other users.** The `db_admin` role can create new users and grant them superuser privileges. This means any compromised service account can create a backdoor account, bypassing all security controls.

**Remediation plan:**

**Phase 1 -- Audit and inventory (Week 1):**

Map each microservice to the tables it touches. Use query logging to discover actual access patterns. Your goal is to build a permission matrix:

| Service | Tables needed | Access type | Frequency |
|---|---|---|---|
| user-svc | users, sessions, profiles | R/W | High |
| order-svc | orders, order_items, payments | R/W | High |
| inventory-svc | products, inventory | R/W | Medium |
| report-svc | orders, products, payments | Read-only | Low |
| migration-svc | all tables | DDL | Rare |

**Phase 2 -- Role creation (Week 2):**

Create database roles per service using the narrowest permissions possible:

```sql
-- Example for the reporting service
CREATE ROLE reporting_role;
GRANT SELECT ON orders, order_items, products TO reporting_role;

-- Example for the inventory service
CREATE ROLE inventory_role;
GRANT SELECT, INSERT, UPDATE ON products, inventory TO inventory_role;
GRANT SELECT ON orders TO inventory_role;  -- cross-table read only

-- Example for the user service
CREATE ROLE user_service_role;
GRANT SELECT, INSERT, UPDATE ON users, sessions, profiles TO user_service_role;
GRANT SELECT ON orders TO user_service_role;  -- read-only for order history
```

Create a separate migration role for DDL operations that is only usable during deployments:

```sql
CREATE ROLE migration_role WITH CREATEDB CREATEROLE;
-- This role is used only by CI/CD pipeline, never by running services
```

**Phase 3 -- Migration (Weeks 3-4):**

For each service, in order of lowest risk first:

1. Create the new service-specific role in the database.
2. Assign the new role to the service account.
3. Remove the `db_admin` role assignment.
4. Run the service's integration tests.
5. Monitor for 24 hours for permission errors.
6. Move to the next service.

For reporting and read-only services, you can migrate multiple at once. For write-heavy services like order-svc, migrate during low-traffic hours.

**Phase 4 -- Enforcement (Week 5):**

- Set `REVOKE ALL ON SCHEMA public FROM PUBLIC` to ensure no default grants exist.
- Use PostgreSQL row-level security for multi-tenant data scoping.
- Implement automated permission auditing: a weekly script that checks that no service account has superuser privileges.
- Add a CI check that fails if a migration creates a user with superuser privileges without an explicit security review approval.

**Result:** Instead of 12 superuser accounts, you have 12 narrowly scoped service roles plus one tightly controlled migration role. A compromise of any single service is contained to that service's data domain.

</details>

### Challenge 2.1: Multi-Tenant SaaS Authorization

You are designing the authorization system for a SaaS project management platform. The platform has the following characteristics:

- 10,000 tenants (companies), each with 5-500 users.
- Each tenant has projects, tasks, and team members.
- Roles per tenant: Owner, Admin, Member, Viewer.
- Some tenants want custom roles (for example, "Project Manager" who can manage tasks but cannot invite new members).
- Billing is per-seat, and only the Owner can change the billing plan.

**Question:** Design the authorization data model and policy evaluation flow. Explain how you handle tenant isolation, default roles, and custom roles. Address both the database schema and the runtime evaluation flow.

<details>
<summary>Model Answer</summary>

**Data model:**

Use a hybrid RBAC model with tenant-scoped roles and optional custom roles.

**Core tables:**

```
tenants
  id: UUID (PK)
  name: string
  plan: string (free, pro, enterprise)

users
  id: UUID (PK)
  email: string

tenant_memberships
  user_id: UUID (FK -> users)
  tenant_id: UUID (FK -> tenants)
  role_id: UUID (FK -> roles)
  PRIMARY KEY (user_id, tenant_id)

roles
  id: UUID (PK)
  tenant_id: UUID (FK -> tenants, nullable for system defaults)
  name: string (Owner, Admin, Member, Viewer)
  is_system_role: boolean

role_permissions
  role_id: UUID (FK -> roles)
  resource_type: string (project, task, team, billing, settings)
  action: string (create, read, update, delete, invite, manage_billing)
  PRIMARY KEY (role_id, resource_type, action)

projects
  id: UUID (PK)
  tenant_id: UUID (FK -> tenants)
  name: string

project_members
  user_id: UUID
  project_id: UUID
  role_id: UUID (FK -> roles, optional for per-project override)
```

**Default roles:**

| Role | Permissions |
|---|---|
| Owner | All actions on all resource types, including manage_billing and invite |
| Admin | All actions except manage_billing |
| Member | Create, read, update on projects, tasks; no delete, no invite, no billing |
| Viewer | Read on projects, tasks; no writes of any kind |

System roles are stored with `tenant_id = NULL` and `is_system_role = true`. When a tenant is created, the system copies the four default roles into the tenant's scope (`tenant_id = tenant.id`) so the tenant can modify them without affecting other tenants.

**Custom roles:**

When a tenant creates a custom role (for example, "Project Manager"), the system:

1. Creates a new row in `roles` with `tenant_id = tenant.id`, `is_system_role = false`.
2. Inserts the selected permissions into `role_permissions`.
3. The owner assigns this role to any member via `tenant_memberships.role_id`.

Custom roles are scoped to a single tenant. No other tenant sees or can use them.

**Runtime policy evaluation flow:**

```
Request: User Alice wants to DELETE task #42 in tenant "Acme Corp"

1. Authentication layer identifies Alice (user_id resolved).

2. Tenant resolution: determine which tenant context. This comes from
   the request subdomain (acme.app.com) or a header (X-Tenant-Id).

3. Fetch Alice's role in this tenant:
   SELECT role_id FROM tenant_memberships
   WHERE user_id = 'alice' AND tenant_id = 'acme-corp'

   Result: role_id = "admin_role" (Admin role)

4. Fetch permissions for the role on resource type "task" action "delete":
   SELECT action FROM role_permissions
   WHERE role_id = 'admin_role'
   AND resource_type = 'task'
   AND action = 'delete'

   Result: "delete" found in the list.

5. Decision: Allow.

6. (Optional) Project-level override check. If Alice has a project-level
   role on project #42, that role's permissions override the tenant-level
   role for resources within that project.
```

**Caching strategy:**

Cache the role-to-permissions mapping aggressively. This rarely changes (only when an admin edits roles). Use a per-tenant cache with a 5-minute TTL:

```
Cache key: "authz:tenant:{tenant_id}:role:{role_id}:permissions"
Cache value: ["create", "read", "update", "delete"] (list of allowed actions per resource)
```

Invalidate the cache for a tenant whenever an admin modifies roles or assigns a user to a role.

**Tenant isolation enforcement:**

1. All queries include `WHERE tenant_id = :tenant_id`. The ORM layer or a database RLS policy enforces this automatically.
2. The application layer passes the tenant context from the authentication token to every data access call.
3. Cross-tenant access is prevented at the query level, not just the UI level. If a user tampered with an API request to change the tenant ID, the query still returns no rows for the wrong tenant.

**Scaling considerations:**

- 10,000 tenants x 500 users = 5 million membership rows. This is trivially handled by a single PostgreSQL instance.
- The role-permissions cache for all tenants fits in a few hundred MB of Redis.
- Policy evaluation is O(1): one cache lookup per resource-action pair.
- For enterprise tenants with 500+ users and complex custom roles, the per-tenant cache hit rate stays above 99 percent because role definitions change infrequently.

</details>

### Challenge 2.2: Fine-Grained Document Sharing

You are building a document sharing application similar to Google Docs or Notion. The system must support:

- Documents organized in a folder hierarchy (folders can contain folders and documents).
- Permissions can be set at the folder level or the document level.
- Inheritance: documents inherit permissions from their parent folder unless overridden.
- Sharing modes: private, shared with specific users, shared with a link, public.
- Users can have different roles on a document: Viewer, Commenter, Editor, Owner.

**Question:** Design the authorization model for this system. Address the permission inheritance algorithm, conflict resolution when a user has multiple permission paths (direct, group, folder inheritance), and the permission check query for a user viewing a document.

<details>
<summary>Model Answer</summary>

**Data model:**

```
documents
  id: UUID (PK)
  name: string
  parent_id: UUID (FK -> folders, nullable for root documents)
  owner_id: UUID (FK -> users)
  sharing_mode: enum (private, shared, link, public)
  link_role: enum (viewer, commenter, editor) -- only when sharing_mode = link

folders
  id: UUID (PK)
  name: string
  parent_id: UUID (FK -> folders, nullable for root folders)
  owner_id: UUID (FK -> users)

document_permissions
  document_id: UUID (FK -> documents)
  grantee_type: enum (user, group, domain)
  grantee_id: UUID (user_id, group_id, or domain pattern)
  role: enum (viewer, commenter, editor, owner)
  PRIMARY KEY (document_id, grantee_type, grantee_id)

folder_permissions
  folder_id: UUID (FK -> folders)
  grantee_type: enum (user, group, domain)
  grantee_id: UUID
  role: enum (viewer, commenter, editor, owner)
  PRIMARY KEY (folder_id, grantee_type, grantee_id)

group_members
  group_id: UUID
  user_id: UUID
  PRIMARY KEY (group_id, user_id)
```

**Permission inheritance algorithm:**

When checking a user's permission on a document, follow this resolution order:

1. Check for an explicit permission on the document itself (`document_permissions`). If found, that is the effective role.

2. If no explicit permission, walk up the folder hierarchy from the document's parent folder to the root folder, collecting all folder permissions that include the user.

3. If multiple ancestor folders grant different roles, apply the most permissive role. The hierarchy is: viewer < commenter < editor < owner.

4. If no folder permission is found, check the document's `sharing_mode`:
   - `private`: no access.
   - `shared`: no access via link (user must be explicitly granted).
   - `link`: use `link_role` as the effective role.
   - `public`: viewer role.

5. If the user is the `owner_id` of the document, they always have Owner-level access regardless of other settings. This is a hard override.

**Conflict resolution -- multiple permission paths:**

A user may have access through multiple paths: direct permission, group membership, folder inheritance, and so on. The resolution rule is:

**Take the highest role across all applicable paths, then check deny (if applicable).**

```
Example:
- Document D is in Folder F.
- Alice has Editor role on Folder F (inherited by D).
- Alice has Viewer role directly on Document D.
- Alice is a member of Group G, which has Commenter role on Document D.

Resolution:
- Direct permission on D: Viewer (lowest)
- Group permission on D: Commenter (middle)
- Folder inheritance: Editor (highest)

Final: Editor (the highest of all applicable permissions).
```

Why take the highest? In document sharing, users expect the most permissive access they have from any source. If Alice was invited as a Viewer but someone gave her group Editor access, she should be able to edit. Reducing a user's access should be done by explicitly changing or removing the highest-level permission, not by relying on the lowest.

**Permission check query:**

To check whether user Alice can edit document D, you need a single efficient query that considers all paths:

```sql
WITH RECURSIVE folder_ancestors AS (
    -- Base case: start from the document's parent folder
    SELECT f.id, f.parent_id, fp.role, 0 as depth
    FROM folders f
    LEFT JOIN folder_permissions fp
        ON fp.folder_id = f.id
        AND (
            fp.grantee_type = 'user' AND fp.grantee_id = 'alice-uuid'
            OR fp.grantee_type = 'group' AND fp.grantee_id IN (
                SELECT group_id FROM group_members WHERE user_id = 'alice-uuid'
            )
        )
    WHERE f.id = (SELECT parent_id FROM documents WHERE id = 'doc-d-uuid')

    UNION ALL

    -- Recursive: walk up to parent folders
    SELECT f.id, f.parent_id, fp.role, fa.depth + 1
    FROM folders f
    INNER JOIN folder_ancestors fa ON f.id = fa.parent_id
    LEFT JOIN folder_permissions fp
        ON fp.folder_id = f.id
        AND (
            fp.grantee_type = 'user' AND fp.grantee_id = 'alice-uuid'
            OR fp.grantee_type = 'group' AND fp.grantee_id IN (
                SELECT group_id FROM group_members WHERE user_id = 'alice-uuid'
            )
        )
)
, effective_permissions AS (
    -- Direct permission on document
    SELECT role, 0 as priority FROM document_permissions
    WHERE document_id = 'doc-d-uuid'
        AND (
            grantee_type = 'user' AND grantee_id = 'alice-uuid'
            OR grantee_type = 'group' AND grantee_id IN (
                SELECT group_id FROM group_members WHERE user_id = 'alice-uuid'
            )
        )

    UNION ALL

    -- Inherited from folder hierarchy
    SELECT role, depth + 1 as priority FROM folder_ancestors
    WHERE role IS NOT NULL
)
-- Take the highest role across all paths
SELECT 'editor' as effective_role
FROM effective_permissions
WHERE role = 'editor' OR role = 'owner'
LIMIT 1;
```

In practice, you would cache the effective permission for each (user, document) pair with a TTL of 60 seconds. When any permission changes on the document or any ancestor folder, you invalidate the cache for all users who have access to that document.

**Edge cases:**

- **Move operation:** When a document is moved from Folder A to Folder B, the effective permission changes. The system should invalidate all cached permissions for the document after the move and optionally notify affected users.

- **Circular folder references:** The `parent_id` constraint should prevent cycles. Enforce this with a check constraint or application logic that verifies the new parent is not a descendant of the folder being moved.

- **Orphaned permissions:** When a user is removed from the system, cascade delete all their permission rows. Use `ON DELETE CASCADE` foreign keys or a cleanup job.

- **Link sharing override:** If a document is shared with a link (viewer role) but a specific user has been granted editor access directly, the user sees editor-level access. Only users without explicit access see the link-level role.

</details>

### Challenge 2.3: Enterprise Role Hierarchy

A large enterprise with 50,000 employees is adopting a new internal platform for knowledge management, project collaboration, and resource planning. The enterprise has the following organizational structure:

- 8 divisions (Engineering, Sales, Marketing, HR, Finance, Legal, Operations, Executive).
- Each division has 3-10 departments.
- Each department has 2-5 teams.
- Roles exist at three levels: division-level, department-level, and team-level.
- The hierarchy is: Executive > Division Head > Department Head > Team Lead > Team Member.

**Question:** Design a role hierarchy system that supports this structure. Address how role inheritance works across levels, how to handle a user who holds multiple roles in different parts of the organization, and how permission escalation (a user temporarily needs a higher role) should be handled. Provide the database schema and the algorithm for resolving a user's effective permissions.

<details>
<summary>Model Answer</summary>

**Role hierarchy model:**

Define roles as a directed acyclic graph (DAG) where higher roles inherit permissions from lower roles plus additional permissions.

```
Role hierarchy:
  Executive (level 5): all permissions in the enterprise.
    |
  Division Head (level 4): all permissions within one division.
    |
  Department Head (level 3): all permissions within one department.
    |
  Team Lead (level 2): all permissions within one team, plus manage team members.
    |
  Team Member (level 1): basic permissions within one team.
```

Inheritance is upward: a Division Head implicitly has all permissions of a Department Head, Team Lead, and Team Member within their division.

**Database schema:**

```sql
-- Organizational structure
divisions
  id: UUID (PK)
  name: string (Engineering, Sales, ...)

departments
  id: UUID (PK)
  division_id: UUID (FK -> divisions)
  name: string

teams
  id: UUID (PK)
  department_id: UUID (FK -> departments)
  name: string

-- Role assignments (one row per user per scope)
role_assignments
  id: UUID (PK)
  user_id: UUID (FK -> users)
  role: enum (team_member, team_lead, department_head, division_head, executive)
  scope_type: enum (team, department, division, enterprise)
  scope_id: UUID (team_id, department_id, division_id, or NULL for enterprise)
  valid_from: timestamp
  valid_until: timestamp (NULL for permanent)
  granted_by: UUID (FK -> users, for audit)

-- Role definitions and their permission sets
roles
  name: string (PK)
  level: integer (1-5)
  description: string

role_permissions
  role_name: string (FK -> roles)
  resource_type: string (knowledge_base, project, resource_plan, people, billing)
  action: string (create, read, update, delete, approve, manage_roles)
```

**Role inheritance algorithm:**

To resolve a user's effective permissions for a given resource:

```
function getEffectivePermissions(userId, resourceContext):
    // resourceContext = { division_id, department_id, team_id }

    assignments = fetchRoleAssignments(userId)

    applicableAssignments = []

    for each assignment in assignments:
        if assignment.scope_type == 'enterprise':
            // Executive level: applies everywhere
            applicableAssignments.append(assignment)

        elif assignment.scope_type == 'division':
            // Applies if resource is in this division
            if resourceContext.division_id == assignment.scope_id:
                applicableAssignments.append(assignment)

        elif assignment.scope_type == 'department':
            // Applies if resource is in this department
            if resourceContext.department_id == assignment.scope_id:
                applicableAssignments.append(assignment)

        elif assignment.scope_type == 'team':
            // Applies if resource is in this team
            if resourceContext.team_id == assignment.scope_id:
                applicableAssignments.append(assignment)

    // Sort by role level (highest first)
    sort(applicableAssignments by role.level descending)

    // Take the highest role applicable to this context
    effectiveRole = applicableAssignments[0].role

    // Fetch permissions for this role
    return fetchRolePermissions(effectiveRole)
```

**User with multiple roles:**

Consider Alice, who holds these roles:

- Division Head of Engineering (scope: Engineering division, level 4).
- Team Member of the Data Science team (scope: Data Science team under Engineering, level 1).

When Alice accesses a resource in the Data Science team, the algorithm finds two applicable assignments:

1. Division Head of Engineering (level 4) --- applies because Data Science is part of Engineering.
2. Team Member of Data Science (level 1) --- applies directly.

The algorithm takes the highest level (Division Head, level 4). Alice can manage the Data Science team as a Division Head, not as a Team Member.

When Alice accesses a resource in the Sales division, the algorithm finds no applicable assignments (her Division Head scope is Engineering, and her Team Member scope is Data Science under Engineering). Result: no access.

**Permission escalation:**

Temporary escalation uses a time-bound role assignment with `valid_from` and `valid_until`:

```sql
INSERT INTO role_assignments
    (user_id, role, scope_type, scope_id, valid_from, valid_until, granted_by)
VALUES
    ('alice-uuid', 'department_head', 'department', 'eng-platform-dept',
     '2026-07-02 09:00:00', '2026-07-09 18:00:00',
     'bob-uuid');
```

Alice holds Department Head permissions in the Engineering Platform department for one week. The system enforces `valid_until` at the query level:

```sql
SELECT * FROM role_assignments
WHERE user_id = 'alice-uuid'
  AND (valid_until IS NULL OR valid_until > NOW())
  AND (valid_from IS NULL OR valid_from <= NOW());
```

Escalation requires approval from a role at least two levels above the target role. For example, to grant a Department Head escalation, the approver must be a Division Head or Executive. An audit log records every escalation grant, including the reason, approver, and exact time window.

**Audit trail:**

Every permission check is logged:

```
audit_log
  id: UUID
  timestamp: timestamp
  user_id: UUID
  action: string
  resource_id: UUID
  decision: allow / deny
  reason: string (which role and scope produced the decision)
  escalation_ref: UUID (nullable, links to the escalation grant if applicable)
```

This audit trail satisfies compliance requirements (SOX, SOC 2) and supports post-incident reviews.

**Caching:**

Cache the effective role for each (user, resource_context) pair. Invalidate when:

- A role assignment is created, modified, or deleted for that user.
- An organizational unit (team, department, division) is restructured.
- A time-bound escalation expires (use Redis TTL equal to the remaining validity).

For 50,000 employees and an average of 3 role assignments per user, the cache stores approximately 150,000 key-value pairs, easily fitting in a single Redis instance with replication.

### Challenge 3.1: Healthcare Authorization with HIPAA Compliance

Design an authorization system for a national healthcare records platform. The system connects hospitals, clinics, laboratories, pharmacies, and insurance providers. Each patient's medical records must be accessible only to authorized healthcare providers directly involved in that patient's care.

**Requirements:**

- 5,000 healthcare organizations, 200 million patients, 2 million healthcare providers.
- Compliance with HIPAA (US) including the Minimum Necessary standard: a provider should access only the minimum data needed for their role.
- Break-glass access: in emergencies, a provider can temporarily access a patient's records even without pre-existing authorization. This must be logged and audited.
- Patient consent management: patients can restrict who sees their data (for example, "hide mental health records from my primary care provider").
- Fine-grained access at the record type level: demographics, lab results, medications, diagnoses, mental health notes, surgery history, insurance claims.
- Delegation: a provider can delegate access to a covering provider during absences.
- Access must be revocable within 1 minute of a patient revoking consent.

**Question:** Design the authorization model, the policy engine architecture, and the data model. Address break-glass access, consent management, and patient-level data segmentation.

<details>
<summary>Model Answer</summary>

**Authorization model: ABAC with hierarchical resource types.**

Use Attribute-Based Access Control as the foundation because healthcare authorization depends on context: who the provider is, their relationship to the patient, the type of data, the purpose of access, the time of day, and the patient's consent preferences. RBAC alone is insufficient because the same role (for example, a nurse) may have different access levels for different patients based on consent and care relationships.

**Core attributes:**

```
User attributes:
  user.id, user.role (doctor, nurse, lab_tech, pharmacist, biller, admin)
  user.organization_id, user.department, user.specialty
  user.license_status (active, suspended, revoked)

Resource attributes:
  resource.type (demographics, lab_results, medications, diagnoses,
                 mental_health_notes, surgery_history, insurance)
  resource.patient_id, resource.organization_id
  resource.sensitivity (normal, sensitive, restricted)
  resource.date_created

Environment attributes:
  env.time_of_day, env.location (on_premise, remote)
  env.access_purpose (treatment, payment, operations, research, emergency)

Relationship attributes (computed at query time):
  relationship.type (primary_care, specialist, lab_requester,
                     emergency_responder, delegated_cover)
  relationship.active (boolean, based on care team membership)
```

**Policy engine architecture:**

```
Provider (web/mobile)
    |
API Gateway
    |
    v
Authorization Service (stateless, horizontally scalable)
    |
    +-- Policy Evaluation Engine (OPA / Cedar / custom)
    |       |
    |       +-- Loads policies from Policy Store
    |       |
    |       +-- Fetches attributes from:
    |           - Identity Provider (user attrs)
    |           - Patient Relationship DB (relationship attrs)
    |           - Consent Service (patient's consent preferences)
    |           - Resource Registry (resource attrs)
    |
    +-- Decision Cache (Redis, 30-second TTL for non-emergency)
    |
    +-- Audit Logger (async, writes to immutable log)
```

**Policy examples:**

```
Policy: Primary care physician can read medications for their patients

PERMIT access IF:
  user.role IN ("doctor", "nurse", "pharmacist")
  AND relationship.type == "primary_care"
  AND relationship.active == true
  AND resource.type IN ("medications", "lab_results", "diagnoses")
  AND resource.sensitivity != "restricted"
  AND env.access_purpose IN ("treatment", "payment")


Policy: Minimum necessary for billing staff

PERMIT access IF:
  user.role == "biller"
  AND relationship.type IN ("billing_contact", "insurance_admin")
  AND resource.type IN ("demographics", "insurance")
  AND env.access_purpose == "payment"
  -- Explicitly deny clinical data to billing

DENY access IF:
  user.role == "biller"
  AND resource.type IN ("mental_health_notes", "surgery_history",
                        "lab_results", "medications", "diagnoses")
```

**Patient consent management:**

Each patient has a consent record that modifies how policies apply:

```json
{
  "patient_id": "patient-123",
  "consent_restrictions": [
    {
      "restricted_to_role": ["mental_health_professional"],
      "resource_types": ["mental_health_notes"],
      "description": "Only mental health professionals can view mental health notes"
    },
    {
      "blocked_providers": ["provider-456"],
      "resource_types": ["*"],
      "reason": "Patient filed complaint against Dr. Smith"
    },
    {
      "allow_research_use": false,
      "allow_marketing_use": false
    }
  ],
  "updated_at": "2026-06-28T14:30:00Z",
  "version": 12
}
```

The policy engine checks consent as an additional filter after the base policy evaluation:

```
1. Base policy: does the provider have a care relationship that
   permits accessing this resource type?
   - If NO: Deny.
   - If YES: proceed to consent check.

2. Consent check: does the patient's consent record allow this
   specific provider to access this resource type?
   - If the provider is in the blocked list: Deny.
   - If the resource type is restricted to a role the provider
     does not hold: Deny.
   - Otherwise: Allow.
```

**Break-glass access:**

In emergencies (code blue, trauma bay, unexpected adverse event), any licensed provider can access any patient's records for treatment purposes.

```
BREAK_GLASS access IF:
  env.access_purpose == "emergency"
  AND user.license_status == "active"
  AND user.role IN ("doctor", "nurse", "paramedic")
  AND user.is_physically_present_at_facility(patient_location)
```

Break-glass access triggers immediate actions:

1. **Log the access** to a tamper-proof audit trail:
   ```json
   {
     "event_type": "BREAK_GLASS_ACCESS",
     "provider_id": "provider-789",
     "patient_id": "patient-123",
     "resource_type": "all",
     "timestamp": "2026-07-02T03:15:00Z",
     "reason": "Code blue in ER bay 4, patient unconscious, no ID band",
     "location": "ER-bay-4",
     "expires_at": "2026-07-02T03:45:00Z"
   }
   ```

2. **Notify the patient's designated primary care provider** within 1 minute via SMS and email.

3. **Set a 30-minute expiry** on the break-glass session. After 30 minutes, the provider must re-authenticate the emergency or transition to a standard care relationship.

4. **Flag for compliance review.** A compliance officer must review all break-glass events within 24 hours. If the provider is found to have accessed records without a legitimate emergency, disciplinary action follows.

**Revocation within 1 minute:**

The 1-minute revocation requirement is achieved through a combination of caching strategy and active invalidation:

```
Revocation flow:

1. Patient revokes consent via the patient portal.
2. Consent Service updates the consent record in the database.
3. Consent Service publishes a revocation event to a message queue:
   topic: "consent.revoked"
   payload: { patient_id, provider_id, resource_types, timestamp }

4. The Authorization Service consumes the event and:
   a. Invalidates all cached policy decisions for this
      (patient, provider) pair.
   b. Publishes a session invalidation event to the API Gateway.

5. The API Gateway terminates any active sessions for this
   provider on this patient's data (WebSocket disconnect,
   session token revocation).

6. All running operations (report generation, data export)
   are terminated with a "consent revoked" error.
```

**Caching strategy for consent:**

| Cache | TTL | Behavior |
|---|---|---|
| Policy decisions | 30 seconds | Fast path for repeated same-context checks |
| Consent records | 10 seconds | Frequent consent changes must propagate quickly |
| Provider-relationship | 5 minutes | Changes less frequently (care team updates) |
| Break-glass sessions | 30 minutes | Aligned with emergency access expiry |

The 10-second TTL on consent records means a consent revocation is reflected within 10 seconds on the fast path and within 1 second on the slow path (cache miss triggers a fresh database read). This satisfies the 1-minute requirement with a comfortable margin.

**Data model:**

```sql
-- Care relationships (the core authorization link)
care_relationships (
    id UUID PRIMARY KEY,
    patient_id UUID NOT NULL,
    provider_id UUID NOT NULL,
    organization_id UUID NOT NULL,
    relationship_type VARCHAR(50) NOT NULL,
    -- primary_care, specialist, lab_requester, delegated_cover
    role VARCHAR(50) NOT NULL,
    -- doctor, nurse, lab_tech, pharmacist
    active BOOLEAN DEFAULT true,
    valid_from TIMESTAMP NOT NULL,
    valid_until TIMESTAMP,
    delegation_ref UUID, -- points to the original provider if delegated
    created_at TIMESTAMP DEFAULT NOW()
);

-- Patient consent restrictions
consent_restrictions (
    id UUID PRIMARY KEY,
    patient_id UUID NOT NULL,
    restriction_type VARCHAR(50),
    -- blocked_provider, role_restriction, resource_restriction
    restricted_role VARCHAR(50),
    blocked_provider_id UUID,
    resource_type VARCHAR(50),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Break-glass access log (immutable, append-only)
break_glass_log (
    id UUID PRIMARY KEY,
    provider_id UUID NOT NULL,
    patient_id UUID NOT NULL,
    emergency_location VARCHAR(200),
    reason TEXT NOT NULL,
    started_at TIMESTAMP NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    reviewed_by UUID, -- compliance officer
    reviewed_at TIMESTAMP,
    review_status VARCHAR(20) -- pending, approved, flagged
);

-- Full audit trail
access_audit_log (
    id UUID PRIMARY KEY,
    provider_id UUID NOT NULL,
    patient_id UUID NOT NULL,
    resource_type VARCHAR(50),
    action VARCHAR(20), -- read, write, share, export
    decision VARCHAR(10), -- allow, deny, break_glass
    consent_version INTEGER, -- version of consent at decision time
    purpose VARCHAR(50), -- treatment, payment, operations, emergency
    ip_address INET,
    session_id UUID,
    timestamp TIMESTAMP DEFAULT NOW()
);
```

### Challenge 3.2: Global Authorization with Offline Policy Evaluation

Design a global authorization system for a multinational content platform with the following characteristics:

- 500 million users across 30 countries.
- Users access the platform from web, mobile apps, and IoT devices.
- Network connectivity is unreliable in some regions (offline periods of 2-12 hours).
- Content access decisions must work offline: the device must be able to determine whether a user can access a piece of content without contacting a server.
- Policies change in real time (content creators update sharing settings, administrators revoke access, subscription plans change).
- Policy updates must propagate to all devices within 5 minutes under normal conditions and within 30 minutes under degraded conditions.
- The system must support 100,000 policy evaluation requests per second globally.
- Compliance with GDPR (right to erasure must be enforced within 48 hours).

**Question:** Design the offline-first authorization system. Address the policy distribution mechanism, the offline evaluation engine on the device, conflict resolution when the device comes back online, and the tradeoffs between policy freshness and offline availability.

<details>
<summary>Model Answer</summary>

**Architecture overview:**

The system uses a signed policy bundle approach inspired by the OPA (Open Policy Agent) bundle protocol. Policy bundles are pre-compiled, signed, and distributed to devices. The device evaluates policies locally using a lightweight embedded policy engine. When the device reconnects, it syncs its local decisions with the server for auditing and receives updated policy bundles.

```
                            Policy Authoring
                                   |
                                   v
                            Policy Compiler
                            (Rego / Cedar / custom DSL)
                                   |
                                   v
                        Policy Signing Service
                        (signs with HSM-backed key)
                                   |
                                   v
                           CDN + Policy Store
                     (distributes signed bundles)
                          /         |         \
                         v          v          v
                    Device A    Device B    Device C
                 (offline Eva) (offline Eva) (offline Eva)
```

**Policy bundle format:**

A policy bundle is a signed archive containing compiled policies, metadata, and a manifest.

```json
{
  "bundle_id": "bundle-v384",
  "created_at": "2026-07-02T06:00:00Z",
  "expires_at": "2026-07-02T06:05:00Z",
  "signature": {
    "algorithm": "Ed25519",
    "key_id": "policy-signer-v2",
    "value": "base64-encoded-signature"
  },
  "policies": [
    {
      "id": "content_access_v3",
      "compiled": "base64-encoded-compiled-policy",
      "version": 3
    },
    {
      "id": "subscription_check_v2",
      "compiled": "base64-encoded-compiled-policy",
      "version": 2
    }
  ],
  "data": {
    "content_owner_map": { ... },
    "blocked_users": ["user-123", "user-456"],
    "country_restrictions": {
      "US": ["content-aaa", "content-bbb"],
      "DE": ["content-ccc"]
    },
    "subscription_plans": {
      "free": { "max_bitrate": "720p", "ad_supported": true },
      "premium": { "max_bitrate": "4k", "ad_supported": false }
    }
  },
  "manifest": {
    "total_size_bytes": 245760,
    "policy_count": 5,
    "data_entry_count": 15000,
    "compression": "gzip"
  }
}
```

**Offline evaluation engine on the device:**

The device embeds a Wasm-based policy evaluator. Compiled policies run in a sandboxed WebAssembly runtime with a fixed memory budget (16 MB).

```
Evaluation flow:

1. User taps "Play content X"
2. App sends evaluation request to local policy engine:
   {
     "input": {
       "user": { "id": "user-456", "plan": "premium", "country": "DE" },
       "resource": { "id": "content-X", "type": "video", "owner": "creator-789" },
       "context": { "timestamp": 1720274400, "offline": true }
     }
   }
3. Policy engine loads the compiled content_access_v3 policy from the bundle.
4. Engine evaluates the policy against the input using the bundled data
   (country_restrictions, blocked_users, subscription_plans).
5. Engine returns decision: Allow or Deny with a reason code.
6. App caches the decision locally with a time-to-live (TTL).

Decision cache (local SQLite):
  decision_cache:
    user_id TEXT
    resource_id TEXT
    decision TEXT (allow, deny)
    reason TEXT
    policy_version INTEGER
    evaluated_at TEXT (ISO 8601)
    expires_at TEXT (ISO 8601)
    PRIMARY KEY (user_id, resource_id)
```

**Policy distribution mechanism:**

Devices poll for new bundles on a configurable schedule:

| Connectivity state | Poll interval | Behavior |
|---|---|---|
| Online (WiFi) | Every 1 minute | Fetch bundle, apply immediately |
| Online (cellular) | Every 5 minutes | Fetch bundle, defer large data updates |
| Online (roaming) | Every 15 minutes | Metadata-only check, skip large bundles |
| Offline | N/A | Use current bundle until reconnection |

Push-based updates supplement polling:
- Long-lived WebSocket connection from the device to the policy distribution service.
- When a critical policy changes (user blocked, content ownership change), the server sends a `policy.update_available` notification.
- The device fetches the new bundle on next poll, or immediately if the notification flags it as urgent.

**Delta updates for large bundles:**

Full policy bundles for 500 million users would be enormous. Instead, use delta (diff) bundles:

```
Full bundle (baseline, ~250 KB):
  Contains all policies and data for the user's region.

Delta bundle (incremental):
  Contains only the changes since the last baseline.
  Format: list of add/update/delete operations on the data.
  Typical size: 1-10 KB.

Policy compiler outputs both a full bundle and a delta bundle
at each compilation cycle. The device applies deltas to its
local bundle to produce the new state.
```

**Conflict resolution on reconnection:**

When a device comes back online after an offline period, it must reconcile its local decision cache with the server.

```
Reconciliation flow:

1. Device sends its decision cache summary to the server:
   {
     "device_id": "device-789",
     "decisions": [
       {"user_id": "user-456", "resource_id": "content-X",
        "decision": "allow", "policy_version": 384, "evaluated_at": "..."},
       {"user_id": "user-456", "resource_id": "content-Y",
        "decision": "allow", "policy_version": 384, "evaluated_at": "..."}
     ],
     "local_bundle_version": 384
   }

2. Server checks each decision against the current policies:
   - If the user was blocked after the device went offline,
     the server marks the decision as "stale" and flags it for
     invalidation.
   - If the content was taken down, same treatment.
   - If no relevant policy change occurred, the decision is confirmed.

3. Server responds with invalidation instructions:
   {
     "invalidations": [
       {"user_id": "user-456", "resource_id": "content-Y",
        "reason": "user_blocked"}
     ],
     "new_bundle_version": 387,
     "bundle_url": "https://policies.cdn.example.com/bundle-v387.signed"
   }

4. Device invalidates the stale decisions and fetches the new bundle.
5. Device logs all offline decisions to the server for audit.
```

**GDPR right-to-erasure enforcement:**

When a user requests deletion under GDPR:

1. The user's data is deleted from the primary database.
2. The policy compiler generates a delta bundle that adds the user's ID to the `blocked_users` list or removes their content.
3. The bundle is signed and published to the CDN.
4. The push notification service sends a `policy.update_urgent` notification to all devices that have cached decisions for this user.
5. Devices receiving the urgent notification immediately invalidate the user's cached decisions and re-evaluate with the new bundle.
6. The 48-hour GDPR window is met because policy bundles are published within 5 minutes under normal conditions and within 30 minutes under degraded conditions.

**Performance and scale:**

| Metric | Target | Mechanism |
|---|---|---|
| Offline evaluations/second per device | Unlimited (local) | Wasm evaluator, ~2 microseconds per evaluation |
| Server-side evaluations/second | 100,000 | Horizontally scaled stateless evaluators |
| Bundle generation latency | Less than 30 seconds | Incremental compilation, 8-core workers |
| Bundle size (full) | Less than 500 KB | Delta compression, region-scoped data |
| Bundle signing latency | Less than 100 ms | Ed25519 with HSM |
| Offline decision cache size | Less than 50 MB per device | LRU eviction, TTL-based expiry |

**Tradeoffs:**

| Tradeoff | Choice | Rationale |
|---|---|---|
| Freshness vs. availability | Availability wins | Stale decisions are better than no decisions. Users can access content they should not be able to for up to 5 minutes, but they are never locked out of content due to network issues. |
| Bundle size vs. granularity | Region-scoped bundles | A global bundle would be gigabytes. Scoping by region reduces each device's bundle to under 500 KB. The tradeoff is that a roaming user may need to download multiple bundles. |
| Local evaluation vs. server audit | Both | Local evaluation provides offline availability. Server-side audit provides compliance (GDPR, SOC 2). Every offline decision is logged and reconciled on reconnection. |
| Push vs. poll | Hybrid | Push provides sub-minute updates for critical changes. Polling provides reliable delivery even if push notifications fail. The 5-minute maximum staleness is bounded by the polling interval. |
| Policy complexity vs. evaluation speed | Compile to Wasm | Wasm evaluation is deterministic, sandboxed, and fast. Complex policies (ABAC with 50 attributes) evaluate in under 10 microseconds on modern mobile CPUs. |

**Security considerations:**

- **Bundle signing:** Every bundle is signed with an Ed25519 key stored in an HSM. The device rejects unsigned or tampered bundles.
- **Replay attacks:** Bundles include an `expires_at` field. Devices reject bundles older than 60 minutes.
- **Denial of service:** The Wasm evaluator enforces a 5-microsecond evaluation budget per request. Policies that exceed the budget are terminated and return a deny decision.
- **Side-channel resistance:** Policy evaluation is constant-time for the same input size. The evaluation result does not leak information about which policy rule was matched.

</details>
