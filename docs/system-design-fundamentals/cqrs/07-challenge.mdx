## Tier 1: Foundation Challenges

These challenges test your understanding of CQRS basics. Complete them before moving to Tier 2.

### Challenge 1: Identify the Command vs. Query

For each operation below, determine whether it is a command, a query, or both. If both, explain why it violates CQS and how you would split it.

| Operation | Context |
|---|---|
| `getUserById(id)` | Returns user profile |
| `createOrder(items, customer)` | Creates an order and returns the order ID |
| `updateInventory(sku, quantity)` | Updates stock count, returns new count |
| `searchProducts(query, filters)` | Searches catalog, returns matching products |
| `processPayment(amount, method)` | Processes payment, returns transaction ID |

**Answer.** `getUserById` is a pure query. `searchProducts` is a pure query. `updateInventory` is a command — it should not return data. Return success or failure instead. `createOrder` is a command disguised as a query — returning the order ID is a side effect. Have the command side emit an event with the order ID, and the client polls the read side to get it. `processPayment` is a command — the transaction ID should be obtained through a separate query.

### Challenge 2: Draw a CQRS Architecture

Draw a CQRS architecture for a library management system. The system tracks books, members, and loans. Members can search the catalog, borrow books, return books, and view their loan history.

**Checklist.**

- Two sides: command and query
- Command bus routes: `BorrowBook`, `ReturnBook`, `RegisterMember`
- Query bus routes: `SearchCatalog`, `GetMemberLoans`, `GetBookDetails`
- Write database: normalized tables for books, members, loans
- Read database: denormalized views for catalog search (book + availability), member loans (book details + due dates)
- Sync mechanism: event bus with events like `BookBorrowed`, `BookReturned`, `MemberRegistered`

### Challenge 3: Identify the Anti-Pattern

A team implements CQRS with a shared database. The command handler inserts into the `orders` table. The query handler reads from the same `orders` table. The read model is identical to the write model. The command handler also updates a Redis cache after each write, and the query handler checks Redis first.

Is this CQRS? Why or why not? What would you change?

**Answer.** This is not CQRS. The read model and write model use the same schema and the same database. The Redis cache is just a cache — it does not make this CQRS. True CQRS requires the read model to be structurally different from the write model and optimized for queries. The Redis cache does not count because the read model shape is identical to the write model shape. To make this CQRS, the team should create a denormalized read model — for example, a `order_summary` table that pre-joins order headers, line items, and payment information.

## Tier 2: Applied Challenges

These challenges require you to design CQRS solutions for realistic scenarios.

### Challenge 4: Design a Notification System

Design a CQRS-based notification system. Users can have multiple notification channels (email, SMS, push). Notifications are sent asynchronously. Users should be able to view their notification history, mark notifications as read, and configure which channels they use.

**Requirements.**

- 100,000 users
- Each user receives 50 notifications per day on average
- Notifications must be delivered within 5 minutes
- Notification history must be queryable for up to 90 days
- Users can filter notification history by type, date range, and read status

**Design considerations.**

- Write side: `SendNotification` command creates a notification record. A separate delivery worker processes pending notifications through the appropriate channel.
- Write model: `notifications` table with fields for user ID, channel, type, content, delivery status, timestamps. `user_channels` table for channel preferences.
- Read side: `notification_history` materialized view with denormalized user name, channel name, type label, and delivery status. Indexed by user ID and created_at for fast filtering.
- Sync: When a notification is created or its delivery status changes, emit an event. The read model updater processes the event and updates the history view.
- Query side: `GetNotificationHistory` supports pagination, filtering by type/date/status, and ordering by date descending.

### Challenge 5: Migrate with Zero Downtime

Your team maintains a monolith with a single PostgreSQL database supporting an e-commerce platform. You want to migrate to CQRS without downtime. Design the migration plan.

**Phase 1: Dual writes.** Add a read model table to the existing database. Modify the monolith to write to both the existing tables and the new read model table in the same transaction. Run both paths in parallel. Verify correctness.

**Phase 2: Separate read database.** Create a read database with the same read model schema. Set up PostgreSQL logical replication from the monolith database to the read database. Point read queries to the read database for a subset of traffic. Verify correctness.

**Phase 3: Extract command service.** For one bounded context (orders), extract the write path into a separate command service. The command service writes to a new command database. It also writes to an outbox table for event publication. A separate process publishes events to a Kafka topic. The read model updater subscribes to the topic and updates the read database.

**Phase 4: Complete migration.** Repeat Phase 3 for each bounded context (inventory, payments, shipping). Phase out the monolith. The system is now fully CQRS.

### Challenge 6: Debug a Production Incident

Your team receives an alert: the "order summary" dashboard shows incorrect data for 15% of orders. The dashboard queries the read database. The write database has correct data.

**Investigation steps.**

1. Compare a sample of incorrect records in the read database against the write database. Identify the specific fields that are wrong.
2. Check the event bus for failed or unprocessed events related to those orders.
3. Check the read model updater logs for exceptions related to those order IDs.
4. Check for recent schema changes to either the event schema or the read model schema.

**Common causes and fixes.**

| Symptom | Likely Cause | Fix |
|---|---|---|
| Specific field null on read | Optional field not populated by updater | Add field mapping in projection handler. Rebuild affected read models. |
| Order count off by 2% | Double-counted events (duplicate delivery) | Make projections idempotent. Use upsert instead of insert. |
| Data is 6 hours old | Subscriber crashed, events accumulated | Restart subscriber. Process backlog. |
| Some orders missing entirely | Outbox failure — event never published | Fix outbox publisher. Rebuild read model for missing orders from write database. |
| Field has wrong value | Race condition in projection — two events updated same row out of order | Add per-row versioning. Use conditional update (only apply if event version > current row version). |

## Tier 3: Design Challenges

These challenges require full system design with written or diagrammed solutions.

### Challenge 7: Design a Multi-Tenant CQRS System

Design a CQRS system for a SaaS analytics platform. The platform serves 500 tenants. Each tenant has up to 10,000 users. Tenants ingest events from their applications and view analytics dashboards.

**Write volume.** Each tenant sends 1,000 events per second during peak hours. Total: 500,000 events per second.

**Read volume.** Each tenant has 50 dashboard viewers who refresh every 30 seconds. Each dashboard runs 10 queries. Total: 8,333 dashboard queries per second.

**Constraints.** Tenant isolation. Up to 5 minutes of sync staleness is acceptable. 99.9% uptime SLA.

**Key design decisions.**

- **Tenant isolation model.** Shared event bus with tenant-partitioned topics. Separate read databases per tenant (or per tenant group for small tenants). Write database shared with tenant ID as partition key.
- **Event ingestion.** Kafka with one partition per tenant for ordered delivery per tenant. Event schema includes tenant ID as the partition key.
- **Read model.** Columnar database (ClickHouse) optimized for time-series analytics. Pre-computed aggregates per tenant. Dashboards query materialized views, not raw events.
- **Sync.** Real-time streaming from Kafka to ClickHouse using Kafka Connect. Windows of sub-5 second latency.
- **Staleness management.** Dashboard shows "last updated N seconds ago" indicator. Users are aware the data is near-real-time.

### Challenge 8: Design an Audit System

Design a system that records every state change in a financial application: account creation, deposit, withdrawal, transfer, interest accrual, fee assessment. The system must support auditing, dispute resolution, and regulatory reporting.

**Requirements.**

- Immutable record of every state change
- Ability to reconstruct account balance at any point in time
- Regulatory reports (monthly statements, annual summaries)
- Dispute resolution (show the sequence of events for a specific transfer)

**This is a natural fit for CQRS with event sourcing.**

- **Event store.** Append-only log of every financial event. Each event has a unique ID, aggregate ID (account ID), version, timestamp, event type, and payload.
- **Command side.** `Deposit`, `Withdraw`, `Transfer`, `AssessFee` handlers. Each validates the command against current aggregate state before accepting.
- **Read side.** `account_balance` projection for current balance. `transaction_history` projection for per-account transaction lists. `monthly_statement` batch projection for regulatory reports.
- **Audit.** The event store is the audit log. Every state change is recorded permanently. Auditors can replay events for any account to verify correctness.
- **Disputes.** To investigate a disputed transfer, load the aggregate's event stream for the relevant time period. Replay events to reconstruct the exact state at the time of the transfer.

### Challenge 9: Optimization Challenge

Your CQRS system handles 100,000 commands and 2 million queries per day. The read database is PostgreSQL. Query latency is 200ms on average. The read model is a normalized schema identical to the write model. Propose optimizations.

**Issues.**

- Read model duplicates write model — no benefit from CQRS
- Normalized schema requires joins on every query
- No caching layer

**Optimizations.**

1. **Denormalize the read model.** Create flat tables that match the queries. For example, if queries frequently join `orders` with `customers` and `line_items`, create an `order_summary` table with all the needed fields.
2. **Add a caching layer.** Use Redis between the query service and the read database. Cache popular queries with a TTL appropriate to your staleness tolerance.
3. **Use covering indexes.** Create indexes that include all columns a query returns. This allows index-only scans, eliminating table lookups.
4. **Consider a different read database.** If the queries are document-oriented, migrate the read model to MongoDB. If the queries are aggregation-heavy, use Elasticsearch or ClickHouse.
5. **Add query result pagination.** If queries return hundreds of rows, enforce pagination limits to reduce query size.

## How to Use These Challenges

Work through the challenges in order. Tier 1 confirms you understand the basics. Tier 2 tests your ability to apply CQRS to realistic scenarios. Tier 3 pushes you to design complete systems.

For each challenge, write out your solution or draw the architecture. Compare your answer against the model answer. If your approach differs, evaluate the tradeoffs — there is rarely one correct answer.

For interview preparation, focus on Tier 1 and Tier 2. Tier 3 challenges are more appropriate for system design study groups and deep architectural exploration.
