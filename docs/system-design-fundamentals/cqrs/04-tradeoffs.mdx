## The Core Tension

CQRS introduces a fundamental tradeoff: optimized reads and writes versus synchronization complexity.

A single-model system gives you immediate consistency, a single source of truth, and simple infrastructure. Every query sees the latest write. Every write is visible to the next query. There is no sync gap, no stale data, no drift.

CQRS gives you independently optimized read and write stacks. The read side can use a completely different data model, storage engine, and indexing strategy than the write side. This decoupling enables performance that a single-model system cannot match.

The cost is that you now have two data stores that must be kept in sync. The synchronization mechanism is a new point of failure. Eventual consistency means stale reads are possible. Schema changes must be coordinated across two models.

This tension is inescapable. Every CQRS decision is a negotiation between independence and coherence.

## Decision Matrix

Use this matrix to evaluate whether CQRS is the right choice for your situation.

| Factor | Favors CQRS | Favors Single Model |
|---|---|---|
| Read-to-write ratio | Heavily read-biased (100:1 or higher) | Balanced or write-biased |
| Read/write data shape | Significantly different | Nearly identical |
| Query complexity | Complex joins, aggregations, projections | Simple lookups |
| Write complexity | Complex business rules, multi-step workflows | Simple CRUD |
| Consistency requirements | Eventual consistency acceptable | Strong consistency required |
| Team size | Multiple teams (command vs. query ownership) | Small team |
| Scale requirement | Need independent read/write scaling | Moderate throughput |
| Schema evolution | Read and write schema change independently | Schema changes are coupled |

### Interpreting the Matrix

If seven or more factors favor CQRS, the pattern will likely provide significant value.

If four to six factors favor CQRS, the pattern may help but consider starting with a simpler variant like single-database CQRS.

If three or fewer factors favor CQRS, do not use CQRS. The complexity will outweigh the benefits.

## CQRS and the CAP Theorem

The CAP theorem states that a distributed data store can only provide two of three guarantees: Consistency, Availability, and Partition tolerance. CQRS systems, by introducing separate databases connected by a synchronization mechanism, are inherently CAP-relevant.

The write side of CQRS prioritizes consistency and partition tolerance. The write database enforces ACID transactions and rejects writes that violate constraints, even if the read side is unreachable.

The read side prioritizes availability and partition tolerance. If the read database cannot sync with the write side, it continues serving the data it has. If the sync mechanism fails, reads return stale data rather than errors.

The overall system is AP — it remains available during partitions but sacrifices cross-side consistency. This is a deliberate architectural choice.

## Five Failure Modes

### Failure Mode 1: Stale Data on Reads

The most common CQRS failure. A client writes data, then immediately reads and sees the old state. This happens because the sync has not yet propagated the change to the read database.

**Impact.** User confusion, potential for incorrect decisions based on stale data, support tickets.

**Causes.** Sync lag exceeds the time between write and read. The client polls the read database before the event is processed. The sync pipeline has a backlog.

**Detection.** Monitor sync lag as a percentile distribution. Track the p99 sync lag. Alert when the sync lag exceeds your acceptable threshold.

**Mitigation.** 
- Return the write acknowledgment with a version token. The client includes the version in the query. If the read database has not caught up, the query is routed to the write database.
- For critical post-write reads, bypass the read database and query the write database directly.
- Reduce sync latency by optimizing the event pipeline (faster processing, smaller batches, more consumer instances).

### Failure Mode 2: Sync Pipeline Failure

The synchronization mechanism stops working. Events are produced on the write side but not consumed on the read side. The read database becomes increasingly stale.

**Impact.** The read model becomes increasingly inaccurate. Eventually the divergence becomes visible to users. Reports show wrong numbers. Dashboard data is incomplete.

**Causes.** Subscriber crash, network partition between event bus and subscriber, event bus cluster failure, schema mismatch (subscriber cannot deserialize a new event version), resource exhaustion (subscriber runs out of memory).

**Detection.** Alert on event bus backlog (unconsumed partition lag for Kafka, queue depth for RabbitMQ). Alert on subscriber health check failures. Monitor the last sync timestamp for each read model.

**Recovery.** Restart the subscriber. If the issue is a poison event, skip or dead-letter it. If the backlog is large, scale up consumer instances to process faster. For severe divergence, rebuild the read model from scratch.

### Failure Mode 3: Schema Divergence

The write model schema and the read model schema evolve independently. Over time, the gap between what the write side produces and what the read side expects widens.

**Impact.** Sync failures. Events that cannot be processed. Incomplete reads. Deployments that break synchronization.

**Causes.** A developer adds a field to the write model without updating the read model. A write model column is renamed without updating the read model projection. The read model adds a new field that the write model does not populate.

**Mitigation.** 
- Treat read models as consumers of events with a contract. Changes to the event schema require coordinated updates to all consumers.
- Use schema registry (Avro, Protobuf, JSON Schema) to enforce compatibility between event producers and consumers.
- Run integration tests that verify read model processing against the latest event schemas.
- Make all schema changes backward-compatible. Add fields as optional. Never remove fields from events.

### Failure Mode 4: Write Model Corruption

The write database becomes corrupted or inconsistent. Because the read model is derived from the write model, the corruption propagates to the read side during sync.

**Impact.** Incorrect data served on both read and write sides. Business logic errors. Cascading failures.

**Causes.** Software bug in a command handler that violates an invariant. Concurrent write conflict that was not properly handled. Database-level corruption (hardware failure, disk errors). Manual data fix applied directly to the write database without triggering the corresponding event.

**Recovery.** 
- Identify and fix the root cause in the command handler.
- Repair the write database using database restore or point-in-time recovery.
- Rebuild all read models from the corrected write database.
- In event-sourced CQRS, you can replay events from before the corruption occurred.

### Failure Mode 5: Read Model Inconsistency

The read model contains data that is internally inconsistent — for example, an order that shows as "shipped" in one field but "pending" in another field. This happens when two events update the read model in an unexpected order.

**Impact.** Users see contradictory information. Automated decisions based on read model data are unreliable.

**Causes.** Out-of-order event processing. Two subscribers updating the same read model concurrently with conflicting values. Partial event batch failure where some events in a batch succeed and others fail.

**Mitigation.** 
- Process events in order per aggregate key.
- Use idempotent upsert operations for read model updates.
- Use transactional writes per read model update.
- Consider per-row versioning on the read model to detect stale updates.

## Five Anti-Patterns

### Anti-Pattern 1: CQRS without Need

A team adopts CQRS because it sounds sophisticated, not because their problem requires it. They build two databases, an event bus, and sync infrastructure for a simple CRUD application with 500 users.

**Why it fails.** The operational overhead of managing two databases and an event bus far exceeds any benefit. The team spends more time managing sync failures than building features. The application would have performed perfectly with a single database.

**The fix.** Start with a single database. Extract a separate read model only when you can measure that the single database is causing problems.

### Anti-Pattern 2: Same Schema for Both Sides

The write model and read model use identical schemas. The system has CQRS in name only — there is no difference between what is stored and what is queried.

**Why it fails.** CQRS adds complexity without benefit. If the read model is a 1:1 copy of the write model, you gain nothing from the separation. You still pay the sync cost.

**The fix.** Design the read model to match specific query patterns. The read model should be denormalized, pre-joined, and projected to serve queries efficiently. If the schemas are identical, you do not need CQRS.

### Anti-Pattern 3: Sync in the Transaction

The command handler updates the write database and the read database in the same distributed transaction. This defeats the purpose of separation.

**Why it fails.** If the read database is unavailable, the write fails. If the read database is slow, the write is slow. The synchronous coupling means the write side's availability and latency are tied to the read side's availability and latency.

**The fix.** Use asynchronous synchronization. The command handler writes to the write database and emits an event. A separate process handles the read model update. The write does not wait for the read update to complete.

### Anti-Pattern 4: Ignoring Eventual Consistency

The development team treats the system as if it were strongly consistent. They write automated tests that assume immediate consistency between write and read. They build client code that reads immediately after writing without handling staleness.

**Why it fails.** Tests are flaky — they pass or fail depending on sync timing. Clients show stale data to users. The team burns time debugging phantom issues that are actually consistency windows.

**The fix.** Accept that the read side is eventually consistent. Design for it. Write tests that expect eventual consistency. Build client code that handles stale reads gracefully. Make the staleness window explicit in documentation and API contracts.

### Anti-Pattern 5: Over-Normalized Read Models

The read model preserves the same normalized structure as the write model. Queries still need to join across multiple tables to build a response.

**Why it fails.** The query side does not deliver the performance improvement that CQRS should provide. Queries are still slow because they join across the same normalized relationships.

**The fix.** Denormalize aggressively in the read model. Store data in the shape that queries need. If every query returns `{order_id, customer_name, item_count, total}`, store those five fields in a single read model row. Do not normalize on the read side.

## When Not to Use CQRS

CQRS is not the default choice. These situations strongly suggest you should not use it.

**Your application is CRUD.** If every operation is create, read, update, delete on the same entity shape, CQRS adds nothing. Standard REST or GraphQL with a single database is the right answer.

**You need strong consistency on every read.** If your application cannot tolerate any stale data, the eventual consistency of CQRS is a dealbreaker. Applications like banking ledgers, real-time inventory, and collaborative editing require strong consistency.

**Your team is small.** A team of three developers should not maintain two databases, an event bus, and sync infrastructure for a CQRS system. The overhead reduces velocity.

**You have no measurable read performance problem.** Do not optimize in advance. If your queries are fast enough with a single database, keep the single database. CQRS is a solution to a measured problem, not a theoretical improvement.

**Your read and write shapes are identical.** If the data you write looks exactly like the data you read, there is no optimization to gain from separation. You pay the complexity cost for zero benefit.
