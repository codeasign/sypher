## Internal Architecture

A CQRS system consists of three major subsystems that work together: the command stack, the query stack, and the synchronization mechanism that connects them. Each stack has its own runtime, its own data structures, and its own optimization goals.

### Command Stack

The command stack is the write path. It receives commands, validates them against business rules, applies state changes to the write database, and emits events. The command stack prioritizes data integrity over query performance.

Every command follows this internal path:

Each layer has a specific responsibility.

**Deserialization.** The incoming request payload is parsed into a strongly-typed command object. Schema validation catches malformed payloads before they reach business logic. A `PlaceOrder` command expects `customerId` (UUID), `items` (array of `{sku, quantity, unitPrice}`), `shippingAddress` (object), and `paymentMethod` (enum).

**Validation.** Structural validation ensures required fields exist and values are within acceptable ranges. Quantity must be positive. SKU must match a known pattern. Shipping address must include at least city and country. Validation errors return immediately without touching the business layer.

**Authorization.** The command is checked against access control rules. Can this customer place an order? Is this user allowed to cancel orders in this state? Authorization failures are logged for audit purposes.

**Command Bus Middleware.** The command bus runs a middleware pipeline. Typical middleware includes: logging (command type, timestamp, payload hash), idempotency check (reject duplicate command IDs), retry logic (transient failures trigger retry with exponential backoff), metrics (counter increments, duration recording). Middleware runs in order and can short-circuit the pipeline.

**Command Handler.** The handler contains the domain logic. It loads the current aggregate state from the write database, applies the command's intent against the current state, checks business invariants, computes the new state, and writes the changes. The handler is where business rules execute.

**Write Database Transaction.** All state changes happen inside a database transaction. The handler reads the current aggregate state, verifies the command is valid against that state, applies changes, and commits. The transaction ensures atomicity — either all changes succeed or none do.

**Event Publisher.** After a successful commit, the handler emits one or more domain events. These events describe what happened in the past tense: `OrderPlaced`, `InventoryReserved`, `PaymentProcessed`. Events are published to an event bus or message queue for consumption by the sync mechanism.

### Query Stack

The query stack is the read path. It receives queries, retrieves data from the read database, and returns results. The query stack prioritizes speed over absolute freshness.

The query stack layers serve different purposes from their command stack counterparts.

**Deserialization.** Query parameters are parsed. Unlike commands, queries carry no intent to change state — only filtering, sorting, and pagination criteria.

**Query Bus.** The query bus middleware handles authentication, rate limiting, and logging. It does not run idempotency checks — queries are safe to repeat. It may enforce read-time limits to prevent runaway queries.

**Cache Check.** The query bus checks a cache layer before invoking the handler. If a cached result exists and the cache TTL has not expired, the cached result is returned immediately, bypassing the handler and database entirely.

**Query Handler.** The handler receives the query parameters and builds a database query. It applies filters, sorts, paginates, and projects. Query handlers are stateless — they do not maintain any context between invocations.

**Read Database Access.** The read database executes the query. Because the read model is denormalized, most queries are simple SELECT statements with WHERE clauses and LIMIT/OFFSET. Joins are rare — the denormalized schema pre-joins related data.

**Result Projection.** The raw database result is projected into the response shape. Field names may be renamed, nested structures assembled, and null values handled. The projection ensures the API contract is stable even if the read schema evolves.

Three synchronization strategies are common.

**Synchronous (inline).** The command handler updates the read database in the same operation as the write. This eliminates eventual consistency but couples the two databases. If the read database is unavailable, the write fails. This approach is not recommended for production CQRS.

**Event-driven (async).** The command handler emits an event after committing the write. A separate subscriber process receives the event and updates the read model. If the read database is down, the write still succeeds — the event is retried later. This is the standard approach.

**Change Data Capture (CDC).** The synchronization attaches to the write database's transaction log. As changes are committed, CDC picks them up and streams them to the read side. This approach does not require the command side to explicitly publish events.

Each stage has characteristic failure modes.

**Stage 3 — Validation fails.** The command is rejected immediately. No state change occurs. The client receives a validation error response. This is fast and cheap.

**Stage 7 — Business rule violation.** The handler detects a violation (insufficient inventory, credit limit exceeded). The transaction is aborted. No state change occurs. The client receives a business error response.

**Stage 9 — Transaction conflict.** Two commands try to modify the same aggregate concurrently. Optimistic locking detects the version mismatch. One command succeeds, the other retries or fails.

**Stage 10 — Event publish fails.** The transaction committed but the event bus is unreachable. The system must handle this with an outbox pattern — writing the event to a database table within the same transaction, then a background process publishes from the outbox.

**Stage 12 — Subscriber fails to process.** The read model updater throws an exception. The event should be retried and eventually sent to a dead letter queue for manual inspection.

**Stage 14 — Read database write fails.** The read database is unreachable or the write violates a constraint. The event is retried. Meanwhile, the read database is stale for that aggregate.

## Consistency Model

CQRS systems fundamentally operate under eventual consistency. Understanding exactly what guarantees exist and what can go wrong is critical.

### Eventual Consistency Window

Between stage 10 (event emitted) and stage 14 (read model updated), the read database is stale. This window is typically measured in milliseconds but can stretch to seconds or minutes under load or during infrastructure failures.

### Stale Read Handling

Three strategies manage stale reads.

**Strategy 1: Accept staleness.** The application acknowledges that reads may be slightly stale. This works for dashboards, reporting, catalog browsing, and any scenario where read freshness is not critical.

**Strategy 2: Read from write database for critical queries.** Certain queries bypass the read database and hit the write database directly. This guarantees freshness but defeats the separation benefit. Use this only for queries that absolutely require the latest state, such as "show me my order confirmation immediately after placing it."

**Strategy 3: Version-tagged reads.** The read model includes a version number or timestamp. The client can request "give me data fresher than version X." If the read database cannot satisfy this, the query router falls back to the write database.

## Performance Characteristics

### Read Throughput

The read side of a CQRS system can dramatically outperform a single-model system for query-heavy workloads. Because the read model is denormalized, a single index scan replaces what would be a multi-table join. Because the read database is separate, it can use different storage engines and indexing strategies.

Read-intensive workloads benefit from columnar storage, covering indexes, and in-memory caching. Some CQRS implementations use a dedicated document database (MongoDB, Couchbase) for the read side and a relational database (PostgreSQL, MySQL) for the write side.

### Write Throughput

Write throughput is also improved because the write database is not competing with heavy read queries. There are no long-running SELECT statements holding locks or consuming I/O.

The write database can be tuned for write performance: smaller pages, write-ahead log optimizations, reduced indexing. Some writes may be asynchronous — the command handler acknowledges receipt and processes the command in the background.

## Configuration Surface

A production CQRS system requires configuration in several areas.

**Command Bus Configuration.** Thread pool size for handler execution, retry policy (max retries, backoff multiplier, max backoff), timeout per command, middleware pipeline order.

**Event Bus Configuration.** Transport type (in-memory, RabbitMQ, Kafka, SQS), topic/queue names per event type, consumer group configuration, delivery guarantees (at-least-once, exactly-once), dead letter queue settings.

**Read Model Configuration.** Database connection pool size, cache TTL per view, refresh strategy (per-event vs. scheduled batch), staleness thresholds.

**Synchronization Configuration.** Batch size for bulk processing, poll interval for CDC, retry policy for failed projections, alert thresholds for sync lag.
