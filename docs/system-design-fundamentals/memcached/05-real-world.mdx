## Company Case Studies

### Facebook / Meta — The Largest Memcached Deployment

Facebook runs the largest Memcached deployment in the world, with thousands of servers serving billions of requests per second. Their architecture is a case study in pushing Memcached to its absolute limits — and extending it where the vanilla version falls short.

**Scale.** In 2020, Facebook reported over 9,000 Memcached servers across multiple data centers, serving over 60 billion operations per day. Each server handles 100,000-300,000 QPS.

**Key challenges Facebook solved:**
1. **Connection scaling.** Each application server connects to thousands of Memcached nodes. Facebook built mcrouter, a proxy that multiplexes connections and reduces the per-node connection count from thousands to tens.
2. **Replication without complexity.** Vanilla Memcached has no replication. Facebook added replication through mcrouter: writes go to a primary and a replica, reads fan out across both.
3. **Warmup.** When a new Memcached node joins, it starts cold. Facebook's mcrouter supports cold-start warmup: misses on the new node trigger reads from the old node, transparently populating the cache.
4. **Slab rebalancing.** Facebook added automatic slab rebalancing to their Memcached fork. The server monitors eviction rates per slab class and moves pages from underutilized classes to overutilized ones.
5. **Multi-cluster.** Facebook's data centers have multiple Memcached clusters for different use cases (user data, graph data, page content), each tuned independently.

### Wikipedia — Page Cache at Scale

Wikipedia serves billions of page views per month with a fraction of the infrastructure of commercial social media platforms. Memcached is central to their caching strategy.

**Architecture.** Wikipedia uses Varnish as an edge cache (caching entire rendered HTTP responses) and Memcached as a backend cache for rendered page fragments and parsed templates.

**What Wikipedia learned.** Page caching with Memcached dramatically reduces database load for read-heavy workloads. Their cache hit ratio for rendered page fragments exceeds 90%, meaning the database handles only a fraction of the requests it would handle without caching.

**Key insight.** Wikipedia does not cache entire pages in Memcached — they cache parsed page fragments and templates. When a page is edited, only the relevant fragments invalidate, not the entire page. This fine-grained caching is more efficient than whole-page caching for a dynamic content site.

### Twitter — Timeline Cache (Originally)

Twitter's early architecture used Memcached to cache user timelines. When a user viewed their timeline, Twitter would fetch the latest tweets from Memcached. On a miss, it would fetch from the database and populate the cache.

Twitter eventually moved to Redis for timeline caching because they needed richer data structures (sorted sets for timeline ordering) and persistence for critical data. Their transition illustrates the Memcached-to-Redis migration path as caching requirements grow beyond key-value semantics.

### Reddit — Memcached Cache

Reddit uses Memcached for caching subreddit listings, post metadata, and user sessions. Their deployment is smaller than Facebook's but still serves billions of requests per day across multiple data centers.

**Reddit's architecture highlights:**
- Memcached for frequently accessed subreddit listings and comments.
- PostgreSQL as the primary database, with Memcached reducing read load by 80%+.
- Local application cache as L1, Memcached as L2.
- LRU eviction handles memory pressure without application intervention.

## Open Source Ecosystem

### Memcached (Original)

The canonical implementation, written in C, maintained by the Memcached community. Current stable version: 1.6.x.

```bash
# Build from source
git clone https://github.com/memcached/memcached.git
cd memcached
autoreconf -fvi
./configure --prefix=/usr/local/memcached
make && make install
```

### Twemproxy (nutcracker) — Twitter

A proxy that provides consistent hashing, connection pooling, and failover for Memcached (and Redis). Lightweight, single-threaded, very stable.

```bash
# Install
git clone https://github.com/twitter/twemproxy.git
cd twemproxy
autoreconf -fvi
./configure && make
```

**Strengths.** Extremely stable, low overhead (single-threaded Rust-like C code), simple YAML configuration, proven at Twitter and other large deployments.

**Limitations.** Single-threaded (limited by one core), no replication support, no warmup or advanced routing.

### mcrouter — Facebook

A feature-rich proxy that provides replication, sharding, warmup, shadowing, and rich routing rules. More complex than Twemproxy but far more capable.

```bash
# Install via GitHub releases
wget https://github.com/facebook/mcrouter/releases/download/v0.41.0/mcrouter-ubuntu-xenial.deb
sudo dpkg -i mcrouter-ubuntu-xenial.deb
```

**Strengths.** Replication, warmup, shadow testing, rich routing rules (regex-based, prefix-based), active development by Facebook.

**Limitations.** More complex configuration, higher resource usage than Twemproxy, C++ codebase (more dependencies to build).

### libmemcached — C/C++ Client Library

A C library providing a Memcached client API. Used as the underlying implementation for many higher-level language bindings.

### Spymemcached — Java Client Library

The standard Java Memcached client. Features consistent hashing, connection pooling, binary protocol support, and CAS.

### memcached — PHP Extension

The standard PHP Memcached client, used by WordPress, Drupal, and many PHP applications.

### pymemcache — Python Client Library

The standard Python Memcached client. Simple API, connection pooling, consistent hashing via `HashClient`.

```python
from pymemcache.client.base import Client
from pymemcache.client.hash import HashClient

# Single node
client = Client(('localhost', 11211))

# Cluster with consistent hashing
cluster = HashClient([
    ('mc1', 11211),
    ('mc2', 11211),
    ('mc3', 11211),
])
```

## Cloud Provider Support

### AWS ElastiCache for Memcached

AWS offers Memcached as a managed service through ElastiCache. This is the most popular managed Memcached option.

**Features:**
- Auto-discovery (clients discover nodes via DNS).
- Automatic failover (new node replaces failed one — cache is cold).
- CloudWatch monitoring for standard metrics.
- Multi-AZ support (nodes in different availability zones).
- Automated patching and maintenance.

```bash
# Connect to ElastiCache cluster via auto-discovery endpoint
# Python automatically discovers all nodes
from pymemcache.client.hash import HashClient
client = HashClient('my-cluster.xxxxxx.cfg.use1.cache.amazonaws.com:11211')
```

**Limitations.** No Auto Scaling (you must manually add/remove nodes), no replication (each node is independent), warmup time after node replacement is identical to any Memcached deployment.

### GCP Memorystore

GCP's Memorystore for Memcached provides a managed Memcached service. Note that GCP primarily supports Redis through Memorystore; Memcached support is available but less featured.

**Features:**
- Managed instance groups with auto-healing.
- Cloud Monitoring integration.
- VPC-native access (no public IPs).

**Limitations.** Fewer instance sizes available compared to Redis, no cross-region capability, GCP recommends Redis for most caching use cases.

### Azure Cache for Redis

Azure does not offer a managed Memcached service. Their managed caching service is Azure Cache for Redis, which supports the Redis API (not the Memcached protocol). For Memcached on Azure, you run it yourself on VMs or use a third-party offering.

## Lessons Learned

### 1. Client-Side Consistent Hashing Is Critical

The most common production failure with Memcached is not understanding that distribution is entirely the client's responsibility. Without consistent hashing, adding or removing a single server invalidates nearly every cached key. The thundering herd that follows can take down your database.

**Rule.** Always use a client library with consistent hashing enabled by default. Verify this before writing a single line of caching code.

### 2. Monitor Eviction Rate as Your Primary Health Metric

Cache hit ratio matters, but eviction rate is the actionable signal. A non-zero eviction rate means your working set exceeds your cache memory. The evicted items had to be re-fetched from the database, adding load that you thought the cache was eliminating.

**Rule.** Alert on `evictions > 0` for more than 5 minutes. When evictions climb, increase `-m` or add nodes. Do not ignore evictions because "the cache should be full" — a full cache is fine; a thrashing cache (evicting and re-caching the same items repeatedly) is a performance disaster.

### 3. Use CAS for Concurrent Updates

If multiple application servers or threads update the same cache key, use CAS to prevent lost updates. Avoiding CAS because it "complicates the code" leads to subtle, hard-to-debug data corruption.

```python
# Pattern for CAS updates
def update_with_cas(key, update_fn, max_retries=3):
    """Read, apply update function, CAS-write. Retry on conflict."""
    for attempt in range(max_retries):
        result = client.gets(key)
        if result is None:
            # Key does not exist — try to create it
            value = update_fn(None)
            if client.add(key, value):
                return value
            continue  # Another client created it — retry

        current_value, cas_id = result
        new_value = update_fn(current_value)

        if client.cas(key, new_value, cas_id):
            return new_value

    raise Exception("CAS update failed after retries")
```

### 4. Prepare Helper Scripts for Cache Warmup

After a node restart, the cache is empty and hit ratio drops to zero. Your database must handle the full load. Speed up recovery with warmup scripts:

```python
# Warmup script: re-populate the N most critical keys
def warmup_cache(client, db, key_list, ttl=3600):
    """Pre-warm cache with the most-accessed keys."""
    for key, query_fn in key_list:
        existing = client.get(key)
        if existing is None:
            data = query_fn(db)
            client.set(key, data, expire=ttl)
            print(f"Warmed: {key}")
        else:
            print(f"Skipped: {key} (already in cache)")
```

Maintain a ranked list of your top-1000 cache keys (by access frequency) and run this script after any restart.

### 5. Plan for Node Failures — They Will Happen

Memcached nodes fail. Not frequently, but when they do, the impact is immediate — all data on that node is lost. Your architecture must assume node failures are routine.

| Failure Scenario | Impact | Mitigation |
|---|---|---|
| Single node crash | Keys on that node → cold (misses) | Cache-aside handles it; DB must survive load spike |
| Multiple node crash | Larger share of cache → cold | Include DB capacity headroom for cache warmup |
| Network partition | Clients cannot reach some nodes | Client library removes unreachable nodes from ring |
| Slow node (high latency) | Requests to that node time out | Client library marks node dead after timeout threshold |
| Full cluster restart | All cache data lost | Staggered restarts; warmup scripts; DB capacity planning |

**Rule.** Every Memcached-dependent system must be able to run without the cache for 30 minutes. If it cannot, you are using Memcached as a database — which is a design error.
