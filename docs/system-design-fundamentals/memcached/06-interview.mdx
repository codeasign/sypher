## Question Types

Memcached appears in system design interviews in three ways:

**1. Direct questions.** "Design a distributed cache." "Compare Memcached and Redis." "How would you build a caching layer for a social network?" These test your understanding of caching fundamentals and Memcached's specific design.

**2. Embedded in larger designs.** "Design Twitter." "Design YouTube." "Design a URL shortener." Caching is a key component of every large-scale system, and Memcached (or its concepts) is the most common reference point.

**3. Deep-dive follow-ups.** "How does Memcached handle memory allocation?" "What happens when a Memcached node fails?" "How would you scale Memcached to 10,000 QPS?" These test your depth of knowledge on a specific topic.

## Level Expectations

### L3 / Entry-Level (0-2 years)

**What you should know:**
- What Memcached is (distributed in-memory cache, key-value store).
- Basic cache-aside pattern (check cache, miss → query DB, populate cache).
- Difference between cache hit and cache miss.
- Simple TTL/expiry concept.

**What you might miss:**
- Consistent hashing (you should know it exists but may not implement it).
- Slab allocator (external fragmentation is a common gap).
- CAS for concurrency.

**Interview expectation.** You can describe the basic architecture and the cache-aside pattern. You should be able to sketch a simple system with one Memcached node.

### L4 / Mid-Level (2-5 years)

**What you should know:**
- Consistent hashing (how it works, why it matters).
- Memcached vs Redis comparison (at least 5 dimensions).
- Cache-aside, write-through, write-around patterns.
- Cache invalidation strategies.
- Hot key and hot partition problems.
- Slab allocator fundamentals.

**What you might miss:**
- Slab imbalance across classes.
- Connection exhaustion and pooling.
- Network bandwidth as a bottleneck.
- CAS for concurrent writes.

**Interview expectation.** You can design a multi-node Memcached cluster, explain key trade-offs, and handle a scaling discussion.

### L5 / Senior (5-8 years)

**What you should know:**
- Everything in L4, plus:
- Internal architecture: libevent event loop, worker threads, hash table, LRU lists.
- Slab allocator internals: chunk size growth, internal fragmentation, slab imbalance.
- CAS details: refcount, race conditions, use-after-free prevention.
- Mcrouter and Twemproxy architecture.
- Capacity planning: working set estimation, QPS calculation, network bandwidth.
- Multi-region deployment patterns.
- Failure modes: hot keys, slab imbalance, OOM, network bottlenecks, connection exhaustion.

**What you might miss:**
- Facebook's specific extensions (slab rebalancing, warmup, replication via mcrouter).
- Detailed performance numbers (p50/p99 latency, QPS per node).
- Binary protocol advantages over ASCII.

**Interview expectation.** You can design a production-grade Memcached deployment from scratch, including capacity planning, failure handling, and operational monitoring. You can explain the internal architecture and identify subtle failure modes.

### L6 / Staff (8-12 years)

**What you should know:**
- Everything in L5, plus:
- Memcached source code architecture (key files, data structures, threading model).
- Performance optimization: NUMA awareness, hugepages, kernel bypass (DPDK), XDP.
- Facebook's published papers on Memcached at scale.
- Distributed caching theory: consistent hashing variants (ketama, jump hash, rendezvous hashing), cache coherence protocols, thundering herd prevention.
- Alternative designs: when to build a custom caching layer vs. using Memcached.
- Trade-offs between Memcached, Redis, CDN caching, application-level caching, and database-level caching.

**What you might miss:**
- Specific kernel tuning for Memcached (net.core.somaxconn, tcp_tw_reuse, tcp_defer_accept).
- Latest Memcached 1.6 features (extstore, proxy layer, TLS).
- Comparative analysis of proxy performance (Twemproxy vs mcrouter vs Envoy).

**Interview expectation.** You can debate the architectural trade-offs of Memcached vs. alternatives at a deep level, design custom caching solutions, and optimize for extreme performance.

### L7 / Principal (12+ years)

**What you should know:**
- Everything in L6, plus:
- You have opinions on when Memcached should not be used and can articulate specific alternative architectures.
- You can design a caching system from scratch that is better than Memcached for a specific use case.
- You understand the evolution of caching: from local caches to distributed caches to CDN-based caching to database-native caching.
- You can evaluate the cost-performance trade-offs of caching decisions at the organizational level.
- You have experience with post-mortems of caching-related outages.

**Interview expectation.** You lead the architecture discussion, not just answer questions. You identify the subtle problems, propose mitigations, and make trade-off decisions that balance performance, cost, and operational complexity.

## FAANG Interview Framework

Use this framework when answering any Memcached-related system design question.

### Step 1: Clarify Requirements

Start by bounding the problem:

- **Scale.** How many users? How many requests per second? What is the cache hit ratio target?
- **Data.** What is the key size? Value size? Total working set? TTL requirements?
- **Consistency.** Is eventual consistency acceptable? Do you need CAS? Need read-your-writes?
- **Availability.** Can the cache go down? What happens during a cache failure?
- **Budget.** Cost constraints? Team size? Operational maturity?

### Step 2: Define the API

```
GET(key) → value or nil
SET(key, value, ttl) → OK
DELETE(key) → OK
```

That is the entire Memcached API. If you need more, you are building on top of it.

### Step 3: Design the Data Model

```
Key: string (max 250 bytes)
Value: blob (max 1 MB)
TTL: integer (seconds, 0 = no expiry)
```

### Step 4: Architecture Diagram

Sketch:
- Application servers
- Client library with consistent hashing
- Cluster of Memcached nodes
- Database as source of truth

## Phrase Bank

Build your interview vocabulary with these phrases and concepts:

**Architecture.** "Memcached uses a slab allocator to prevent memory fragmentation. Memory is divided into pages, each page is divided into fixed-size chunks per slab class, and items are stored in the smallest chunk that fits them."

**Consistent hashing.** "The client library uses consistent hashing to distribute keys across nodes. When a node is added or removed, only K/N keys move instead of nearly all keys, which prevents thundering herds on the database."

**Cache-aside.** "The application checks the cache first. On a miss, it falls back to the database and populates the cache. This is the simplest and most resilient caching pattern because the cache can fail without taking down the application."

**Eviction.** "Memcached uses LRU eviction per slab class. When a slab class runs out of free chunks, it evicts the least recently used item in that class. This is not global LRU — one class can be thrashing while another has free space."

**Hot keys.** "A hot key is a single key that receives a disproportionate share of requests. The node hosting that key becomes overloaded while other nodes sit idle. Mitigations include local L1 caching, key replication, and mcrouter fan-out."

**Performance.** "A single Memcached node can handle 100,000-500,000 QPS for small objects. The main bottleneck is the global lock on the hash table and LRU lists, which limits scaling beyond 8 worker threads."

**Failover.** "Memcached has no built-in replication. When a node fails, the client library detects the failure, removes the node from the consistent hash ring, and routes requests to remaining nodes. The cache is cold for the lost keys — the cache-aside pattern handles repopulation."

**Slab imbalance.** "If one slab class fills up while another has free space, the full class evicts items even though the server has available memory. Vanilla Memcached does not rebalance — you need Facebook's fork or mcrouter for automatic rebalancing."

**Comparison.** "Memcached is simpler, faster per-operation, and uses less memory overhead than Redis. Redis provides persistence, data structures, replication, and clustering. Choose Memcached when you need maximum speed and minimal overhead for key-value caching. Choose Redis when you need data structures, persistence, or replication."

**Capacity planning.** "For a system with 10 million DAU, estimate the working set by multiplying active items by size per item, add 12% for slab overhead, double for headroom, and divide by per-node memory to get the node count."

## Red Flags

Interviewers watch for these mistakes. Avoid them:

**1. "Memcached stores data persistently."**
No. Memcached has no persistence. Node crash = total data loss. This is a fundamental misunderstanding.

**2. "Memcached is a database."**
Memcached is a cache, not a database. It offers no durability, no query language, no transactions, no replication. Using it as a database is a design error.

**3. "I would use modulo hashing for distribution."**
When you add a node, nearly all keys move. This causes a massive thundering herd. Always use consistent hashing.

**4. "Cache hit ratio is 95% — that is fine with evictions."**
Evictions mean the cache is too small. Even with 95% hit ratio, the 5% of requests that miss are paying the full database cost. If evictions are high, the miss rate is actually higher than 5% for the most popular items (the ones being evicted and re-fetched).

**5. "Memcached handles replication natively."**
Vanilla Memcached has no replication. Facebook's mcrouter adds it, but it is not part of the standard open-source version.

**6. "I would put all cache keys in one slab class."**
Different key sizes need different slab classes. Ignoring slab class behavior leads to slab imbalance and wasted memory.

**7. "The cache can be down; the application handles it."**
This is true for cache-aside, but only if the database has capacity to handle the full load. A cache failure can trigger a cascading database failure if the database is not provisioned for the load spike.

**8. "I would use the default settings."**
The default `-m` is 64 MB. Running a production cache with 64 MB means you are not caching anything useful. Every production deployment must size the cache for the working set.

### Drill 2: Design a Cache for Twitter Timeline

Given: 100 million DAU, each user views their timeline 10 times per day. Each timeline has 200 tweets (280 bytes each). Cache TTL: 60 seconds. Estimate memory, QPS, node count.

**Answer.** 
- 100M users × 10 views = 1B timeline views per day = ~11,574 QPS average.
- Each timeline ≈ 200 × 280 bytes = 56 KB + overhead ≈ 60 KB.
- Working set: 100M active users × 60 KB = 6 TB. This is too large for Memcached.
- Better approach: cache only the top 1% of users (1M users = 60 GB, which fits in 2-3 nodes).
- For the other 99%: generate timelines on demand from database, cache individual tweets not full timelines.

### Drill 4: Compare Memcached and Redis Verbally

Practice a 60-second comparison covering:
- Data structures (key-value vs. rich types)
- Persistence (none vs. RDB/AOF)
- Memory efficiency (better in Memcached)
- Multi-core (4-8 threads vs. single-threaded)
- Replication (none vs. built-in)
- Clustering (client-driven vs. built-in)
- Use cases (pure caching vs. caching + data structures + pub/sub + streams)

### Drill 5: Hot Key Mitigation Design

Design a system to handle a hot key receiving 500,000 QPS (single Memcached node capacity is 200,000 QPS).

**Answer.**
1. Add L1 local cache (in application memory, 5-second TTL). This reduces the Memcached load from 500K to ~100K QPS (since the same clients see the same data within 5 seconds).
2. If the hot key is known (e.g., a celebrity profile), replicate it across 3 nodes. The client randomly selects one replica.
3. Use mcrouter with replication rules for the hot key prefix.
4. If the hot key is dynamic (viral content), use a machine learning model to detect hot keys and replicate them automatically.
