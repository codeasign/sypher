## The Problem Story

Before Memcached existed, every major website faced the same fundamental problem: the database was the bottleneck, and caching solutions were custom-built per team, poorly engineered, and nearly impossible to share between projects.

In 2003, LiveJournal was one of the fastest-growing social platforms on the internet. It had millions of users, each generating page views that required database queries for friends lists, blog entries, comments, and user profiles. The engineering team, led by Brad Fitzpatrick, watched their MySQL database buckle under the load. Every page request meant multiple SQL queries. As user counts climbed, the database became a furnace — more disks, more replication lag, more connections, more pain.

The standard approach at the time was to buy bigger database hardware or add read replicas. But this was expensive, fragile, and scaled in big expensive jumps rather than small cheap ones. Fitzpatrick recognized that the vast majority of database reads were for the same popular data — user profiles, recent blog posts, friend lists — repeated millions of times per day.

He needed something radically simple. Not a database. Not a storage system. A pure, dumb, in-memory cache that tossed data away when memory ran out and never pretended to be durable. The design goals were brutal: sub-millisecond latency, minimal memory overhead, distributed by design (not as an afterthought), and so simple that you could trust it under fire.

The result was Memcached. It changed how the internet handled scale.

## The Insight

Memcached is a distributed memory caching system that stores key-value data in RAM for extremely fast retrieval, offloading database reads and reducing latency for applications at scale.

**The coat-check analogy.** Imagine a massive theater with 50,000 seats. Every time someone needs their coat, they must go back to their seat, find the coat, and return. That is your database — you can find anything, but it takes time and effort. Now imagine a coat check room at the entrance. You hand your coat to an attendant who tags it and hangs it on a rack. When you return, you give them the ticket, and they grab it in seconds. You don't search the entire building. You don't wait in line at every seat row. The attendant is Memcached.

In computing terms, your application server queries Memcached by key. If the data is there (a cache hit), Memcached returns it in under a millisecond. If it is not there (a cache miss), the application falls back to the database, retrieves the data, and writes it into Memcached for next time. This simple pattern — the cache-aside pattern — is the foundation of virtually every high-traffic system on the web.

## Core Vocabulary

**Memcached** — A distributed, open-source, in-memory key-value store designed for caching. It is not a database — it offers no persistence, no transactions, and no querying. This lack of features is its superpower: it is tiny, fast, and trivially simple to operate.

**Slab allocator** — Memcached's memory management system. Instead of using malloc/free for every item (which causes fragmentation), it pre-divides allocated memory into slabs, and each slab is further divided into chunks of a specific size. This prevents memory fragmentation and keeps performance predictable.

**Slab class** — A group of slabs that all have the same chunk size. Memcached creates multiple slab classes with different chunk sizes (typically growing by a factor of 1.25x). Items are assigned to the slab class whose chunk size is the smallest that fits the item. For example, a 200-byte item goes into the 256-byte slab class, not the 512-byte class.

**Chunk** — The individual storage unit within a slab. Each chunk can hold exactly one cached item (key + value + metadata). Chunks are fixed-size within a slab class. An item smaller than its chunk wastes the remaining space (internal fragmentation). An item larger than its chunk cannot be stored in that slab class.

**LRU (Least Recently Used) eviction** — The eviction policy Memcached uses per slab class. When a slab class runs out of free chunks, Memcached evicts the least recently used item in that slab class to make room for a new one. Crucially, LRU is per slab class, not global — one slab class can be thrashing while another has free space.

**Item** — A single key-value entry stored in Memcached. Each item has a key (up to 250 bytes), a value (up to 1 MB), flags (arbitrary 32-bit integer set by the client), an expiration time, a CAS (Compare-And-Swap) identifier, and access time metadata.

**Key-value store** — The simplest possible data model. A key is a string (max 250 bytes). A value is an opaque blob of bytes (max 1 MB). Memcached does not know or care about the structure of the value — it is a black box. Your application serializes data (JSON, pickle, msgpack) when writing and deserializes when reading.

**CAS (Compare-And-Swap)** — A concurrency control mechanism. Each item has a 64-bit CAS identifier that changes every time the item is modified. When you read an item, you get its CAS value. When you write it back, you include the CAS value. Memcached rejects the write if the CAS value has changed since your read, preventing lost updates in concurrent scenarios. Think of it as an optimistic lock.

**ASCII protocol** — The original, human-readable text protocol for communicating with Memcached. Commands like `get mykey` and `set mykey 0 3600 5` are sent as plain text over TCP. Simple to debug with telnet. Higher overhead than binary due to parsing.

**Binary protocol** — A more efficient, machine-readable protocol added in 2010. Uses fixed-length headers and avoids string parsing on the server. Enables features like quiet commands (getq) that suppress responses for cache misses, reducing bandwidth in multi-get operations.

**Consistent hashing** — A distributed hashing scheme used by Memcached client libraries to determine which server in a cluster stores a given key. Unlike simple modulo hashing (key.hash() % N), consistent hashing minimizes key redistribution when servers are added or removed. Only K/N keys move instead of nearly all keys, where K is total keys and N is total servers.

**Memcached client library** — The software running inside your application that speaks the Memcached protocol, implements consistent hashing, manages connection pools, and handles server failures. Examples: `pymemcache` (Python), `spymemcached` (Java), `memcached` (PHP), `node-memcached` (Node.js). The client handles distribution — Memcached servers themselves know nothing about each other.

**GET / SET / DELETE** — The three core operations. `GET key` retrieves a value. `SET key flags exptime bytes\r\ndata` stores a value. `DELETE key` removes a value. That is the entire command surface. Memcached does not have increment-by-key as a native atomic operation in the ASCII protocol (the incr/decr commands exist but are rarely the path to high performance).

**Expiry time** — A TTL (Time-To-Live) set on each item when stored. Can be a relative number of seconds (up to 30 days) or a Unix timestamp for absolute expiration. When expired, the item is not immediately deleted — it is invalidated on the next `GET` and lazily evicted. You can also set expiry to 0, meaning the item never expires (evicted only by LRU pressure).

**Max item size (1 MB)** — A hard limit on the value size. The default maximum chunk size in Memcached is 1 MB. Items larger than 1 MB are rejected. You can increase this with the `-I` flag, but doing so changes the slab class sizing and wastes memory. If you need to cache larger objects, split them into chunks and reassemble on the client.

The architecture is remarkably simple. Application servers do not talk to Memcached servers directly in most cases — they talk through a client library that handles distribution. The client library uses consistent hashing to decide which Memcached node stores each key. The Memcached nodes themselves are completely unaware of each other. There is no cluster membership protocol, no replication, no gossip. Each node is a standalone process that stores whatever keys the client sends it.

This simplicity is intentional. By pushing distribution logic to the client, the server stays small, fast, and reliable. The hard problems (partitioning, failover, topology changes) are handled in the client layer, where they can be updated without restarting cache servers.

### Step 1: Client Initialization

Your application creates a Memcached client instance with a list of server addresses. The client builds a consistent hash ring from these addresses, assigning each server multiple virtual nodes on the ring for balanced distribution.

```python
from pymemcache.client.hash import HashClient

client = HashClient([
    ('mc-node-1', 11211),
    ('mc-node-2', 11211),
    ('mc-node-3', 11211),
    ('mc-node-4', 11211),
    ('mc-node-5', 11211),
], use_pooling=True)
```

### Step 3: Getting a Value (GET Command)

When you call `client.get('user:1234')`:

1. Client hashes key `user:1234` using consistent hashing, determines which server owns it.
2. Client sends `get user:1234\r\n` over TCP to the selected server.
3. Server receives data in libevent event loop, dispatches to worker thread.
4. Worker thread parses the command, computes hash, looks up in hash table (O(1) average).
5. If found: checks expiry — if expired, treats as miss and lazily evicts.
6. If valid: touches the item (marks as recently used in the LRU list). This is an O(1) pointer relocation operation.
7. Server sends back `VALUE user:1234 0 45\r\n{json blob}\r\nEND\r\n`.

The slab allocator prevents memory fragmentation, which was a major problem in earlier caching systems. ### ASCII vs Binary Protocol

Memcached supports two wire protocols. Understanding the trade-off helps you choose the right client library.

| Feature | ASCII Protocol | Binary Protocol |
|---|---|---|
| Human-readable | Yes — telnet-debuggable | No — hex dumps |
| Parsing overhead | String parsing on server | Fixed-length headers |
| Quiet commands | No | Yes (getq — no response on miss) |
| Bandwidth (multi-get) | Higher (verbose keys in response) | Lower (compact headers) |
| CAS support | Yes (gets/cas commands) | Yes (native field) |
| Authentication | None | Optional SASL |
| Client complexity | Minimal | Moderate |

The binary protocol reduces per-request overhead by roughly 25-30% for small objects and is the standard choice for high-throughput production deployments. The ASCII protocol remains popular for debugging and simple scripting (curl, telnet, netcat).

## Mental Model Check

**1. What happens when a Memcached node crashes?**
Answer: All data on that node is lost. This is expected behavior — Memcached is not persistent. The client library detects the failure (connection refused or timeout), marks the node as down, and stops routing keys to it. Application requests that would have hit that node experience cache misses, fall through to the database, and populate other nodes as they warm up. This is the cache-aside pattern doing its job.

**2. Why does Memcached use a slab allocator instead of malloc?**
Answer: Malloc causes memory fragmentation over time. When you allocate and free objects of different sizes, the heap becomes a patchwork of small gaps — none large enough to satisfy a new allocation. The slab allocator pre-divides memory into fixed-size chunks per slab class, so every allocation is exactly the same size within a class. This eliminates fragmentation at the cost of some internal waste (unused space within a chunk).

**3. Can two Memcached nodes share the same key?**
Answer: No — and yes. Memcached nodes are unaware of each other, so they have no mechanism to prevent duplicate keys. However, the consistent hashing algorithm in the client library ensures that every client maps a given key to the same node (assuming the cluster topology is the same). If clients have different server lists, the same key could land on different nodes. This is a configuration error — all clients must use the same server list.

**4. What is the difference between expiration and eviction?**
Answer: Expiration is set by the application via the TTL (exptime) parameter on SET. It is a time-based deletion — after 3600 seconds, the item is considered expired and will not be returned. Eviction is the LRU mechanism that kicks in when memory is full. Memcached evicts the oldest items (by access time) in each slab class to make room for new writes. An expired item can still occupy memory until it is accessed (lazy eviction) or the slab class evicts it.

**5. Why does the FAQ say "Memcached is not a database"?**
Answer: Because calling it a database creates dangerous expectations. Data in Memcached has no durability guarantee — a crash wipes everything. Memcached has no query language, no indexes, no transactions, no rollback, no replication (in the vanilla version), and no backup. It is a cache. Storing the only copy of important data in Memcached is a design error. Use it to speed up access to data that lives permanently in a database or other durable store.
