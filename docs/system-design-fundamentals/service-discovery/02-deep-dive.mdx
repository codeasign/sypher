## Internal Architecture

A production-grade service registry is itself a distributed system. It must be highly available, consistent (to a defined degree), and performant under load. The internal architecture breaks down into four major subsystems.

### Storage Layer

The storage layer holds the mapping from service names to instance lists. In systems like Consul and etcd, this is a replicated key-value store. ZooKeeper uses a hierarchical namespace (ZNode tree). The storage layer must support:

- **Write operations**: Register, deregister, heartbeat (lease renewal)
- **Read operations**: Query by service name, query by instance ID, watch for changes
- **Replication**: Synchronize state across cluster nodes

### Health Checker

The health checker subsystem is responsible for verifying that registered instances are alive and functional. It operates in two modes:

- **Passive**: Receives heartbeats from instances and updates lease timestamps
- **Active**: Periodically probes instances by making HTTP, TCP, or gRPC connections

### API Layer

The API layer exposes the registry's functionality to clients over the network. Typical operations:

- `PUT /v1/agent/service/register` — Register a service instance
- `PUT /v1/agent/service/deregister/{id}` — Deregister an instance
- `GET /v1/health/service/{name}` — Query healthy instances
- `PUT /v1/agent/health/service/checks/{id}` — Send a heartbeat

### Replication / Consensus Layer

This layer ensures all nodes in the registry cluster agree on state. Most registries use the Raft consensus protocol for strong consistency (etcd, Consul) or a gossip protocol for eventual consistency (Eureka).

## Data Structures and Algorithms

### The Core Data Structure: Hash Map of Service Name to Instance List

At its heart, a service registry is a distributed hash map. The key is the service name (a string like `video-transcoder`). The value is a list of instance descriptors, each containing IP address, port, metadata, and lease expiration timestamp.

```
Registry State (in-memory):
{
  "api-gateway": [
    { id: "gw-1", address: "10.0.1.10", port: 8080, lease_expires: 1719876543, healthy: true },
    { id: "gw-2", address: "10.0.1.11", port: 8080, lease_expires: 1719876543, healthy: true }
  ],
  "video-transcoder": [
    { id: "tr-3", address: "10.0.3.45", port: 9090, lease_expires: 1719876540, healthy: true },
    { id: "tr-7", address: "10.0.3.89", port: 9090, lease_expires: 1719876537, healthy: true }
  ],
  "user-service": [
    { id: "us-1", address: "10.0.2.20", port: 8080, lease_expires: 1719876545, healthy: true }
  ]
}
```

### Consistent Hashing for Load Distribution

When a consumer receives a list of instances, it must select one. Consistent hashing ensures that a given request key (for example, user ID or session ID) is always routed to the same instance, as long as the instance set has not changed. This is critical for services that maintain local caches or in-memory state.

```
Consumer selects an instance:
  hash_value = hash("user-12345")
  instance_list = sorted(all_instances)
  selected = instance_list[hash_value % len(instance_list)]
```

When instances are added or removed, consistent hashing minimizes the number of requests that are rerouted to different instances. Only the requests closest to the changed instance on the hash ring are affected.

### Gossip Protocol for Heartbeat Propagation

In systems like Consul and Eureka, the gossip protocol distributes health information across the cluster without requiring all nodes to synchronize with a central leader for every heartbeat. Each node periodically exchanges state with a small set of random peers. Within seconds, all nodes converge on the same view of which instances are healthy.

```
Gossip Cycle (every 1 second):
  Node A selects 3 random peers from its membership list
  Node A sends its current state summary:
    {
      known_instances: 142,
      suspect_failures: ["tr-7"],
      recently_joined: ["gw-3"]
    }
  Peer Node B responds with its state summary
  Both nodes merge any new information
```

### Watch-Based Notification

Many registries support watch operations where consumers subscribe to changes for a specific service. Instead of polling the registry every few seconds, the consumer receives a push notification when the instance list changes. This is implemented using long-polling (HTTP) or persistent streams (gRPC).

```
Watch Mechanism:
  1. Consumer calls Watch("video-transcoder")
  2. Registry holds the connection open (up to 60 seconds)
  3. When an instance is added/removed for "video-transcoder":
     a. Registry sends the updated instance list to the consumer
     b. Consumer updates its local cache
     c. Consumer re-establishes the watch
  4. If no changes occur within 60 seconds:
     a. Registry sends a keepalive message
     b. Consumer continues waiting
```

### Detailed Step Descriptions

**Step 1 — Consumer Connection**: The consumer opens a TLS connection to the registry API endpoint. The connection is authenticated using a token or mutual TLS.

**Step 2 — Authentication and Authorization**: The API layer validates the consumer's credentials and checks whether the consumer is authorized to discover the requested service.

**Step 3 — Cache Lookup**: The API layer checks its in-memory cache for a recent response. Caching avoids hitting the storage layer on every query.

**Step 4 — Cache Miss**: If the cache entry is empty or expired, the API layer proceeds to the storage layer.

**Step 5 — Storage Read**: The API layer reads the service entry from the storage layer. On a follower node, this is a local read. On the leader, it reads from the leader's authoritative state.

**Step 6 — Consistency Check**: For strong consistency reads, the leader verifies that its state is up to date with the latest committed Raft log entry. For eventually consistent reads, the follower returns whatever it has.

**Step 7 — Health Filtering**: The returned list is filtered to include only instances whose lease has not expired and whose health check status is passing.

**Step 8 — Result Serialization**: The filtered list is serialized into JSON or Protocol Buffers for the response.

**Step 9 — Cache Population**: The result is stored in the API layer's cache with a short TTL (for example, 5 seconds). Subsequent queries for the same service can be served from cache.

**Step 10 — Response Delivery**: The response is sent back to the consumer, typically with a cache-control header indicating how long the consumer may cache the result locally.

## Consistency and State

### Crash Scenarios

When a registry node crashes, the remaining nodes must detect the failure and continue operating. In a Raft-based cluster, the leader sends periodic heartbeats to followers. If a follower does not hear from the leader within the election timeout (typically 150-300 milliseconds), it starts a new election.

**Leader crash**: Followers detect the absence of heartbeats and hold an election. One follower becomes the new leader. During the election (typically 100-500 milliseconds), write operations are unavailable. Reads may continue on followers.

**Follower crash**: The leader detects the missing follower heartbeats and removes it from the cluster. Writes and reads continue as long as a majority of nodes remain.

### Partition Scenarios

A network partition splits the cluster into two groups. If the partition separates the leader from a majority of followers, the leader steps down. The majority side elects a new leader and continues operating. The minority side stops accepting writes.

**What happens to services on the minority side?** Services registered with the minority side cannot complete their registrations or heartbeats because the registry cannot achieve consensus. The services must retry and eventually connect to the majority side when the partition heals.

### Simultaneous Registrations

Two instances of the same service may try to register at the exact same time. The registry must handle this race condition. With Raft consensus, all write operations go through the leader, which serializes them deterministically. The result is that both registrations succeed, and both instances appear in the instance list.

**Conflicts**: If two instances claim the same instance ID, the last write wins. Each registration request includes a timestamp or sequence number, and the registry keeps the most recent one.

### Stale Reads

When a consumer reads from a follower that has not yet received the latest updates, it receives stale data. This stale read window depends on replication lag.

**Stale read window = max(network latency, replication delay)**

For Raft-based systems, replication latency is typically 1-10 milliseconds. For gossip-based systems, convergence time is typically 1-5 seconds.

## Performance Characteristics

### Latency

Service discovery operations must be fast because they sit on the critical path of every inter-service call.

| Operation | Target Latency | Acceptable | Degraded |
|---|---|---|---|
| Cache hit (local query) | under 1 ms | under 5 ms | over 5ms |
| Register instance | under 10 ms | under 50 ms | over 50ms |
| Query (follower read) | under 5 ms | under 20 ms | over 20ms |
| Query (leader read) | under 10 ms | under 30 ms | over 30ms |
| Heartbeat renew | under 10 ms | under 50 ms | over 50ms |
| Watch notification | under 50 ms | under 200 ms | over 200ms |

### Throughput

A single Consul or etcd node can handle approximately 5,000 to 10,000 write operations per second and 50,000 to 100,000 read operations per second. Clusters scale horizontally for reads.

### Memory

Each stored instance descriptor consumes roughly 200-500 bytes in memory. For a system with 10,000 service instances, the in-memory state is approximately 2-5 MB per node. The watch state (tracking which consumers are watching which services) can consume more memory than the service data itself. For 10,000 consumers each watching 5 services, expect 50-100 MB of watch state per node.

## Configuration and Tuning

### TTL and Heartbeat Intervals

The TTL value controls how quickly the registry detects a failed instance. Shorter TTL means faster detection but more network traffic. The general guideline is:

```
TTL = heartbeat_interval × 2
```

For most production systems, a heartbeat interval of 10-15 seconds with a TTL of 25-30 seconds works well. This gives the instance two missed heartbeats before eviction.

| Scenario | Heartbeat Interval | TTL | Detection Time |
|---|---|---|---|
| Fast failover (critical) | 5s | 12s | 12-17s |
| Default (balanced) | 15s | 35s | 35-50s |
| Low overhead (stable) | 30s | 70s | 70-100s |

### Replication Factor

The replication factor determines how many nodes store each piece of data. In a three-node cluster, a replication factor of 3 (default for etcd and Consul) means every node stores all data. This provides maximum durability but limits the total capacity to the smallest node.

### Health Check Configuration

Active health checks should be configured with careful intervals and timeouts:

- **Check interval**: How often the registry probes the instance. For critical services, 10 seconds. For less critical services, 30-60 seconds.
- **Check timeout**: How long the registry waits for a response. Typically 2-5 seconds.
- **Deregister critical service after**: If health checks fail continuously for this duration, deregister the instance automatically. Typically 1-3 minutes.

### Watch Configuration

Consumers that use watch-based discovery should set a reasonable watch timeout:

- **Watch timeout**: 60 seconds (standard for long-polling)
- **Consumer-side cache TTL**: 5-15 seconds (fallback if watch disconnects)
- **Consumer-side retry interval**: 1-2 seconds (how fast to re-establish a broken watch)

## Advanced Algorithms

### Bloom Filters for Quick Existential Checks

When a consumer asks "does service X exist?" the registry can use a Bloom filter to answer quickly without querying the storage layer. The Bloom filter is a probabilistic data structure that can answer "definitely not" or "probably yes" about set membership.

```
Bloom Filter Parameters:
  - n = 10,000 expected service names
  - p = 0.001 false positive probability
  - k = 10 hash functions
  - m = ~144 KB filter size

Operation:
  - Insert "video-transcoder" → hash 10 times, set 10 bits
  - Query "video-transcoder" → hash 10 times, check 10 bits
    All 10 bits set → "probably exists" (0.1% chance wrong)
    Any bit not set → "definitely does not exist"
```

The Bloom filter eliminates storage lookups for nonexistent services. If a consumer queries for a service that was misspelled as `video-transcoderr` (two r's), the Bloom filter quickly determines the service does not exist, and the registry returns an empty result without touching the storage layer.

### Merkle Trees for Anti-Entropy

When two registry nodes need to verify they have the same state, exchanging the full state is expensive. A Merkle tree allows nodes to compare state efficiently.

```
Merkle Tree Construction:
  Root hash = hash(hash(hash(instance1) + hash(instance2)) + ...)

  Node A's root: 0x7A3F...B2
  Node B's root: 0x7A3F...B2
  → Both nodes have identical state (no sync needed)

  Node A's root: 0x7A3F...B2
  Node B's root: 0x9C1D...E8
  → Nodes differ. Walk the tree to find the specific subtree
    that differs. Only exchange the differing subtree.
```

This reduces the data exchanged during anti-entropy from O(N) (full state) to O(log N) in the common case where nodes are in sync.

### Raft Log Compaction

The Raft log grows with every write operation. Without compaction, the log would grow unboundedly. The registry periodically snapshots its state and truncates the log up to the snapshot point.

```
Raft Log Compaction Cycle:
  Time 1: Log has 100,000 entries since last snapshot
  Time 2: Registry creates a snapshot of current state
          (stops accepting writes briefly)
  Time 3: Log is truncated to entries after the snapshot
  Time 4: New nodes start from the snapshot, not from
          the beginning of the log

  Snapshot size: 2-5 MB for 10,000 instances
  Log size before compaction: 50-100 MB
  Log size after compaction: 1-2 MB
```

## Security Considerations

### Registry Authentication

The registry must authenticate every registration and query request. Unauthenticated access allows an attacker to register fake services, redirect traffic to malicious endpoints, or deregister legitimate services.

```
Authentication methods:
  - Mutual TLS (mTLS): Both client and server present certificates
  - API tokens: Bearer tokens in HTTP headers
  - JWT: Signed JSON Web Tokens with claims

Recommended: mTLS for service-to-registry communication.
Tokens and JWTs are vulnerable to token theft.
```

### Access Control

Not every service should be able to discover every other service. A fine-grained access control system limits which consumers can discover which providers.

```
ACL Policy Example (Consul format):
  service "payment-gateway" {
    policy = "deny"
    # Only the billing service can discover payment-gateway
  }
  service "billing" {
    policy = "write"
    # Billing can register itself
  }
```

### Rate Limiting

The registry must protect itself from abusive clients. A misconfigured or malicious client that sends 100,000 queries per second can overwhelm the registry.

```
Rate limiting strategies:
  - Per-client token bucket: 1,000 queries/second per client
  - Global rate limit: 50,000 queries/second total
  - Connection-level limit: 100 concurrent connections per client
  - Backpressure: Return HTTP 429 Too Many Requests with
    a Retry-After header
```

## Operational Scenarios

### Scenario 1: Rolling Deployment with Zero Downtime

During a rolling deployment, old instances are replaced with new ones. The registry must handle the transition without dropping traffic.

```
1. New instance boots, registers with registry
2. Registry health check passes (new instance is ready)
3. New instance appears in query results
4. Old instance is sent a SIGTERM signal
5. Old instance deregisters from registry
6. Registry removes old instance from query results
7. Consumers detect the change and route only to new instances
8. Gap: 1-2 seconds between old instance deregistration and
   new instance being fully ready
Midigation: Keep old instance in the pool for 5 seconds
after the new instance is confirmed healthy.
```

### Scenario 2: Network Partition Between Consumer and Registry

A consumer loses connectivity to the registry but can still reach service instances. The consumer must continue operating without registry updates.

```
Consumer behavior during partition:
  Second 0-15: Consumer uses cached instance list (TTR = 15s)
  Second 15-30: Cache expires. Consumer retries registry.
                Retry fails. Consumer uses last known list.
  Second 30-60: Consumer marks registry as "unavailable"
                Uses last known list with TTL extended to 300s
  Second 60+: Consumer periodically retries registry
              (every 30s with exponential backoff)
              If a connection to a cached instance fails,
              remove it from the local list
              If all instances are exhausted, return error
```

### Scenario 3: Registry Capacity Planning for a Deployment

You need to deploy 500 new instances of a service in 5 minutes. Each instance must register before accepting traffic.

```
Registration burst capacity:
  500 instances / 300 seconds = 1.67 registrations/second
  (This is well within the capacity of a 3-node Consul cluster)

  But: Each registration triggers a watch notification to
  200 consumers. That is 500 × 200 = 100,000 notification
  messages.

  To handle this:
  1. Batch notifications: Collect all changes within a
     100ms window and send one combined notification
  2. Rate-limit notification delivery to 10,000 per second
  3. Consumers should process notifications asynchronously
```

## Cross-Referencing with Other Patterns

Service discovery does not exist in isolation. It interacts with several other distributed systems patterns.

**Load balancing**: Discovery provides the list of healthy instances. Load balancing selects one instance from that list. The two are tightly coupled. Most discovery client libraries include built-in load balancing.

**Circuit breakers**: When a consumer discovers a service instance and the connection fails, the circuit breaker prevents repeated retries to the same instance. The consumer removes the instance from its local cache and tries another instance.

**Health checks**: Discovery depends on health checks to know which instances are healthy. The health check system is the source of truth for instance liveness. If health checks are wrong, discovery is wrong.

**Configuration management**: Service registries often double as configuration stores (Consul KV store, etcd). The same infrastructure that stores service locations also stores configuration values. This is convenient but creates coupling between the discovery system and the configuration system.
