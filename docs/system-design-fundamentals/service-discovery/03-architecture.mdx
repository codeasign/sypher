## Naive Implementation

Every team starts somewhere. The most naive approach to service discovery is hardcoded IP addresses in configuration files.

```java
// Naive вҖ” hardcoded IPs in application config
public class ServiceConfig {
    public static final String USER_SERVICE_URL = "http://10.0.2.20:8080";
    public static final String TRANSCODER_URL = "http://10.0.3.45:9090";
    public static final String BILLING_URL = "http://10.0.4.10:8080";
}
```

This approach works when you have three services running on three servers. It breaks catastrophically at scale for several reasons.

### What Breaks at Scale

**Configuration drift**: When you have 50 microservices, each deployed across 10 instances, you have 500 places where IP addresses are hardcoded. A single IP change requires updating all 500 references. In practice, some get missed, causing mysterious failures.

**Scaling events are impossible**: When auto-scaling adds 5 new instances of the transcoder service, no configuration file knows about them. The new instances sit idle while existing instances are overloaded.

**Rolling deployments cause downtime**: When you deploy a new version of the user service, old instances are replaced with new ones on different IPs. Every consumer must be restarted with updated configuration. During the deployment window, many consumers point at dead instances.

**Manual toil does not scale**: A team of 5 engineers can manage 20 hardcoded IPs through spreadsheet coordination. A team of 50 engineers managing 500 IPs will have constant outages from stale configurations.

### Improvement: Configuration File with Service Names

A slight improvement is to use a configuration file that maps service names to hostnames, and rely on DNS round-robin:

```yaml
services:
  user-service: "http://user-service.internal.example.com"
  transcoder: "http://transcoder.internal.example.com"
  billing: "http://billing.internal.example.com"
```

This is better but still fragile. DNS caching introduces 30-300 second delays before changes propagate. DNS has no built-in health checking. A dead instance continues to receive traffic until its DNS TTL expires and a new DNS lookup returns only healthy instances.

### etcd-Based Architecture

etcd is a strongly consistent, distributed key-value store commonly used for service discovery in Kubernetes. The architecture stores service data under a well-defined key prefix:

```
/registry/services/specs/<service-name>/<instance-id>
/registry/services/status/<service-name>/<instance-id>
```

The `specs` path contains the registration metadata. The `status` path contains health and lease information. Kubernetes uses this pattern with its own API server layer on top of etcd.

### Consul-Based Architecture

Consul is designed specifically for service discovery. It provides a higher-level API than etcd, with built-in health checking, DNS interface, and a web UI. Consul uses the gossip protocol for membership management and Raft for data consistency.

Key Consul features:
- **Agent-based**: Each node runs a Consul agent that handles registration, health checks, and forwarding to the server cluster.
- **DNS interface**: Services are accessible via `<service>.service.consul` DNS names.
- **KV store**: General-purpose key-value store for configuration, health check definitions, and service metadata.

### ZooKeeper-Based Architecture

Apache ZooKeeper predates both etcd and Consul. It uses a hierarchical namespace (ZNode tree) and the Zab consensus protocol. ZooKeeper is still used in legacy systems and in Apache Kafka for cluster coordination.

ZooKeeper's ephemeral ZNodes are a natural fit for service discovery. When a service instance registers, it creates an ephemeral ZNode under a service path. When the instance disconnects or crashes, the ZNode is automatically removed.

```
/transcoder/instances/instance-001  (ephemeral, data: 10.0.3.45:9090)
/transcoder/instances/instance-002  (ephemeral, data: 10.0.3.89:9090)
```

## Integration Patterns

### Integration with Load Balancers

Load balancers need to know which instances are healthy to route traffic. In a server-side discovery pattern, the load balancer queries the registry.

The load balancer polls the registry every 10-30 seconds and updates its backend pool. Alternatively, the load balancer can use a watch to receive push notifications when instances change. HAProxy, NGINX, and Envoy all support this pattern.

### Integration with Kubernetes

Kubernetes has its own integrated service discovery model based on three key concepts: Services, Endpoints, and kube-proxy.

**Services** define a logical set of Pods and a policy for accessing them. Each Service has a stable virtual IP and DNS name.

**Endpoints** are the actual IP-port pairs of the Pods matching the Service's selector. The Endpoints controller watches Pod changes and updates the Endpoints object.

**kube-proxy** runs on every node and watches Services and Endpoints via the Kubernetes API server. It programs iptables or IPVS rules to route traffic to healthy Pods.

```
User request в”Җв”Җв–ә Service VIP в”Җв”Җв–ә kube-proxy в”Җв”Җв–ә Pod IP:port
                (10.96.0.1)     (iptables      (10.244.1.5:8080)
                                 DNAT rules)
```

Kubernetes also supports **headless Services** (no cluster IP) where DNS returns the Pod IPs directly. This is used by stateful applications like databases.

### Integration with Service Mesh

A service mesh like Istio or Linkerd adds a sidecar proxy (Envoy or linkerd-proxy) to every Pod. The sidecar intercepts all traffic and consults the mesh control plane for routing and discovery decisions.

```
Istio Architecture:
  Control Plane (istiod):
    - Pilot: service discovery and traffic management
    - Citadel: certificate issuance for mTLS
    - Galley: configuration validation

  Data Plane:
    - Envoy sidecar per Pod
    - Discovers services via xDS APIs from Pilot
    - Routes traffic based on VirtualService rules
    - Handles retries, timeouts, circuit breaking
```

In a service mesh, the application code is completely unaware of service discovery. The sidecar handles everything transparently.

### Integration with DNS

DNS-based integration is the simplest and most universal. Consul provides a DNS interface where services are queried by name:

```bash
# Query Consul DNS for transcoder service instances
dig transcoder.service.consul

; <<>> DiG 9.16 <<>> transcoder.service.consul
;; ANSWER SECTION:
transcoder.service.consul. 0 IN A 10.0.3.45
transcoder.service.consul. 0 IN A 10.0.3.89

# With SRV records for port information
dig _transcoder._tcp.service.consul SRV

;; ANSWER SECTION:
_transcoder._tcp.service.consul. 0 IN SRV 1 1 9090 transcoder-1.node.dc1.consul.
_transcoder._tcp.service.consul. 0 IN SRV 1 1 9090 transcoder-2.node.dc1.consul.
```

The DNS TTL is set to 0 to prevent caching. Each DNS query returns the current healthy instances.

### Federation

Most registries support federation, where clusters in different regions share service information. Strategies include:

**Gossip federation**: Consul servers in different regions participate in a WAN gossip pool. They exchange summaries of service health and availability. Updates propagate in 5-30 seconds depending on network conditions.

**Hierarchical federation**: A top-level registry aggregates information from regional registries. Consumers query the top-level registry for cross-region discovery.

**Read-through federation**: The local registry, on a miss for a service name, queries the remote registry and caches the result locally.

### Zone-Aware Routing

Zone-aware routing ensures that traffic stays within the same availability zone when possible. This reduces latency and cross-AZ data transfer costs (which can be significant at scale).

```
Consumer request:
  1. Get all healthy instances for "transcoder"
  2. Filter to instances in same AZ (us-east-1a)
  3. If same-AZ instances exist, select one via round-robin
  4. If no same-AZ instances, fall back to same-region
  5. If no same-region instances, fall back to cross-region
```

## Capacity Planning for 10M DAU

Let's design the service discovery infrastructure for a system serving 10 million daily active users. The system has 200 microservices, each running an average of 10 instances.

### Traffic Estimates

```
Service instances: 200 services Г— 10 instances = 2,000 total instances
Registration rate per second: 2,000 / 86,400 = 0.02/s (evenly distributed)
Peak registration rate (deployments): 500 instances/s during rolling deploy
Heartbeat rate: 2,000 instances Г— 1/15s = 133 heartbeats/second
Query rate (per instance): 50 requests/s (typical for an API service)
Total query rate: 2,000 instances Г— 50 queries = 100,000 queries/second
Watch subscriptions: 2,000 instances Г— 5 watched services = 10,000 watches
```

### Infrastructure Requirements

```
Registry cluster: 5 nodes (tolerates 2 failures)
Each node: 4 CPU, 16 GB RAM, 100 GB SSD
Network: 1 Gbps internal
Total registry traffic: ~100 MB/s (queries + heartbeats + replication)
```

### Bottlenecks

- **Write throughput during deployments**: 500 registrations per second during rolling deployments. Each registration requires Raft consensus (2 round trips). The leader node handles 500 writes/s Г— 3 internal messages = 1,500 messages/s.
- **Watch fan-out**: A single service change triggers notifications to all watchers. For the most popular service (watched by 500 consumers), a deployment updating 10 instances generates 500 Г— 10 = 5,000 notification messages.
- **Heartbeat bursts**: When the network recovers from a partition, all instances reconnect and send heartbeats simultaneously. The heartbeats rate may spike to 10,000 per second for 1-2 seconds.

## Deployment and Operations

### Deployment Topology

Deploy the registry as a dedicated cluster, not as part of application instances. Mixing registry and application workloads on the same machines causes resource contention and reduced reliability.

```
Recommended topology:
  - 3-5 dedicated nodes for the registry cluster
  - Each node in a different availability zone
  - Low-latency network between nodes (same region)
  - SSD storage for write-ahead log and state
  - Separate network interface for cluster replication traffic
```

### Monitoring and Alerting

Monitor the following metrics to maintain a healthy registry:

- **Leader stability**: If the leader changes more than once per hour, investigate network issues or resource pressure.
- **Registration lag**: Time between a service sending a registration and the registration being visible to all consumers. Should be under 1 second for Raft-based systems.
- **Stale watch count**: Number of watchers that have not received an update within the expected interval.
- **Heartbeat success rate**: Percentage of heartbeats that arrive within the TTL window. Should be above 99.9%.
- **Eviction rate**: Number of instances evicted per minute. A spike indicates a network partition or widespread instance failures.

### Backup and Disaster Recovery

- **Snapshot the registry state** every 6 hours to persistent storage.
- **Maintain a hot standby cluster** in a different AWS region for critical systems.
- **Test failover procedures** monthly: kill the leader, verify a new leader is elected within 1 second, and verify all watches reconnect.
- **Document manual fallback**: If the entire registry is down, document how to manually configure routes at the load balancer level as a last resort.

### Runbook for Common Incidents

**Incident: Registry leader is flapping (changing repeatedly)**

```
Symptoms: Leader changes more than 5 times in 10 minutes.
          Registration failures increase.
          Heartbeat timeouts increase.

Diagnosis:
  1. Check network latency between registry nodes
     ping node-b, ping node-c
  2. Check disk I/O on each node
     iostat -x 1
  3. Check CPU and memory on each node
     top, free -m

  Root cause usually: High disk latency (>100ms) causing
  Raft heartbeats to be delayed. Or: Network latency >100ms
  between nodes in different AZs.

Resolution:
  1. If disk latency: Move to faster SSD or reduce log size
     via compaction
  2. If network latency: Verify all nodes are in same region.
     If spread across AZs, increase Raft election timeout
     to 500ms
  3. If resource pressure: Add more nodes to the cluster
```

**Incident: Heartbeat success rate drops below 90%**

```
Symptoms: Many instances are being evicted.
          Consumers see shrinking instance pools.
          Error rates increase.

Diagnosis:
  1. Check if the registry is overloaded
     consul info (check "last_contact" and "commit_time")
  2. Check if there is a network issue between instances
     and the registry
  3. Check if the registry is dropping heartbeats due to
     rate limiting

Resolution:
  1. If registry overloaded: Scale up registry nodes (more CPU)
  2. If network issue: Check firewall rules, DNS resolution
     for registry nodes
  3. If rate limiting: Increase rate limit or add more nodes
  4. As emergency: Increase heartbeat interval to 30s and
     TTL to 90s to reduce load on the registry
```

**Incident: All consumers report stale data**

```
Symptoms: New instances are not discoverable.
          Dead instances are still in query results.
          Deployments show 0% traffic to new instances.

Diagnosis:
  1. Check if watch notifications are being delivered
  2. Check consumer-side cache TTL settings
  3. Check if the registry is processing writes but not
     propagating to followers

Resolution:
  1. Force a cache refresh on all consumers
     (restart or send admin signal)
  2. Verify Raft replication is working
     consul operator raft list-peers
  3. If replication is broken: Restart the follower that
     is not receiving updates
```

## Tiered Discovery Architecture

For very large systems, a single discovery mechanism is insufficient. A tiered approach uses different mechanisms at different layers.

### Layer 0: Direct Connections (Performance Critical)

For the most performance-critical paths (database connections, in-memory cache), use direct connections with static VIPs. These paths are not allowed to change dynamically.

```
Example: Web server to Redis cache
  - Static VIP: 10.0.0.100:6379
  - VIP is moved by the operations team when cache cluster changes
  - No discovery overhead вҖ” every microsecond counts
```

### Layer 1: Client-Side Discovery (Latency Sensitive)

For internal services that need sub-5ms discovery latency, use client-side discovery with a local registry cache. The service instances embed a discovery client library.

```
Example: API gateway to user service
  - Client library caches instance list for 5 seconds
  - Registry lookup on cache miss: 2-5ms
  - Direct connection to selected instance: 0.5ms
  - Total discovery overhead: 2.5-5.5ms
```

### Layer 2: Server-Side Discovery (Standard Traffic)

For most inter-service communication, use server-side discovery through a load balancer. The load balancer handles discovery and routing.

```
Example: Recommendation service to content service
  - Consumer calls load balancer: localhost:8080
  - Load balancer discovers healthy instances: 1-2ms
  - Total overhead: 1-2ms (no DNS lookup, no client library)
```

### Layer 3: DNS-Based Discovery (Infrequent Connections)

For services that are called infrequently (batch jobs, admin tools, reporting), use DNS-based discovery. The 30-60 second propagation delay is acceptable.

```
Example: Reporting tool to analytics service
  - DNS lookup: 10-20ms (cached by OS resolver)
  - Connection to resolved IP: direct
  - If the analytics service changes, the reporting tool
    discovers the change within 60 seconds
```

When this job is submitted, Nomad automatically registers 5 instances of `api-gateway` with Consul, sets up health checks, and deregisters them when the job is stopped.

## Security Architecture for Service Discovery

### Zero-Trust Discovery

A zero-trust security model assumes no network is safe. Every request between services must be authenticated and encrypted. Service discovery plays a key role in zero-trust because it controls which services can find each other.

```
Zero-trust discovery principles:
  1. Every registration must be authenticated (mTLS or token)
  2. Every query must be authorized (ACLs determine who
     can discover which services)
  3. Every connection between services must use mTLS
     (certificates are distributed by the registry's CA)
  4. The registry itself must be hardened against attack
     (API rate limiting, intrusion detection, audit logging)
```

### Certificate Management for mTLS

When services use mTLS, each service instance needs a certificate. The registry often doubles as a certificate authority (Consul Connect, Istio's Citadel).

```
Certificate lifecycle:
  1. Service instance starts
  2. Instance calls registry's CA endpoint with its
     registration token
  3. CA issues a short-lived certificate (24-48 hour TTL)
  4. Instance uses this certificate for all service-to-service
     communication
  5. Instance renews the certificate before expiry
     (typically every 12 hours)
  6. If the instance crashes and restarts, it gets a new
     certificate during re-registration

### Operational Readiness Checklist

Before deploying a service discovery system to production, verify each item on this checklist:

- [ ] Registry cluster has at least 3 nodes across 3 availability zones
- [ ] Heartbeat interval and TTL are configured per service criticality
- [ ] Consumer-side caching is configured with a fallback strategy
- [ ] Watch notifications are enabled and working
- [ ] ACLs are configured to restrict service discovery access
- [ ] mTLS is enabled for registry-to-service communication
- [ ] Monitoring alerts are set up for leader changes (>1/hour)
- [ ] Backup snapshots are scheduled (every 6 hours)
- [ ] Disaster recovery plan is documented and tested
- [ ] Rate limiting is configured on the registry API
- [ ] Circuit breakers are configured on the consumer side
- [ ] Cross-region discovery is tested (if applicable)
```
