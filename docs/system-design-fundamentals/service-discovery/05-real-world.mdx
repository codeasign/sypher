## Netflix — Eureka

Netflix was one of the first companies to publicly discuss their service discovery architecture at scale. In 2012, Netflix ran over 500 microservices on AWS, each with hundreds of instances. They needed a discovery system that could handle constant instance churn from auto-scaling, deployments, and AWS instance failures.

### The Architecture

Netflix built **Eureka**, an eventually-consistent (AP) service registry designed for the cloud. Eureka was designed with the assumption that failures are the norm, not the exception.

### Key Design Decisions

**AP over CP**: Netflix chose eventual consistency over strong consistency. Eureka servers do not require a quorum. If one server fails, the remaining servers continue operating. Clients can detect the failed server and fall back to cached data.

**Self-registration**: Every service instance registers itself using the Eureka client library. There is no external orchestrator or sidecar. This makes the architecture simple but requires every service to embed the client library.

**Region and zone awareness**: Each Eureka server knows its AWS region and availability zone. Clients prefer instances in the same zone, then the same region, and only fall back to other regions if no local instances are available.

**Heartbeat-based health**: Instances send heartbeats every 30 seconds. Eureka does not perform active health checks. If a heartbeat is missed for 3 intervals (90 seconds), the instance is evicted.

**Client-side caching**: The Eureka client caches the instance list locally. If the Eureka server is unreachable, the client uses its last known cache for up to 30 minutes. This provides resilience against Eureka server outages.

**No DNS dependency**: Eureka does not use DNS for service resolution. All discovery goes through the REST API. This avoids DNS caching issues.

### What Netflix Learned

- The 90-second eviction window is too slow for fast-failing services. They later added a faster eviction mechanism for critical services.
- Client-side caching for 30 minutes means that even if Eureka is down for half an hour, services continue to work.
- The peer-to-peer replication model works but requires careful capacity planning. Each Eureka server sends full replication to all peers, and the replication traffic grows linearly with the number of servers.

## Uber — Ringpop

Uber faced a different challenge. Their monolithic architecture was transitioning to microservices, but their services ran across thousands of hosts managed by a custom scheduler. They needed a discovery system that could handle frequent host reassignments without a centralized registry.

### The Architecture

Uber built **Ringpop**, a library that implements a distributed hash ring with consistent hashing and gossip-based membership. Ringpop is not a traditional service registry — it is a peer-to-peer discovery and routing layer.

Key characteristics:
- **No central registry**: Every node runs the Ringpop library and participates in a gossip protocol to learn about other nodes.
- **Consistent hashing**: Requests are routed to nodes based on the hash of a key (for example, user ID or trip ID). This provides affinity without requiring a dedicated registry lookup.
- **Failure detection**: Ringpop uses SWIM (Scalable Weakly-consistent Infection-style Membership protocol) for failure detection. Nodes suspect failures based on gossip, and if a majority confirms, the node is marked as dead.

### What Uber Learned

- Gossip-based discovery converges slowly (5-15 seconds for a membership change to propagate to all nodes). This is acceptable for Uber's use case because instances are long-lived.
- The decentralized approach eliminated the single point of failure of a centralized registry. No Eureka cluster to maintain.
- Consistent hashing meant that when instances were added or removed, only a fraction of requests were rerouted. This was critical for Uber's stateful routing scenarios.

## Airbnb — SmartStack

Airbnb's SmartStack is a server-side discovery system that combines Nerve (a local health checker and registrar) with Synapse (a local HAProxy configuration generator).

### The Architecture

Every host in Airbnb's infrastructure runs both Nerve and Synapse:

- **Nerve** runs alongside each service instance and performs health checks. When an instance starts, Nerve registers it with the service registry (ZooKeeper). When an instance fails its health check, Nerve deregisters it.
- **Synapse** watches the registry for changes to services it cares about. When instance lists change, Synapse regenerates the local HAProxy configuration with the new backend pool and reloads HAProxy.

The result is that every host has a local HAProxy that knows about all healthy instances of all services. Applications on that host connect to `localhost:SERVICE_PORT` and are transparently routed to a healthy instance.




### What Airbnb Learned

- Running a local HAProxy on every host uses extra CPU and memory (about 50-100 MB per host). At thousands of hosts, this adds up to significant infrastructure cost.
- The ZooKeeper cluster was a single point of failure. When ZooKeeper had issues, all service discovery stopped working. Airbnb later migrated to Consul.
- The SmartStack model works well for services that use HTTP. For non-HTTP protocols (gRPC, Thrift), they needed additional configuration.

## Kubernetes — etcd-based Discovery

Kubernetes has the most widely deployed service discovery system in the world. Every Kubernetes cluster uses etcd as the backing store for all cluster state, including service discovery.

### How Kubernetes Service Discovery Works

Kubernetes provides two modes of service discovery: environment variables and DNS.

**Environment variable mode**: When a Pod is created, Kubernetes injects environment variables for each Service in the namespace. For example, a Service named `redis-master` creates `REDIS_MASTER_SERVICE_HOST` and `REDIS_MASTER_SERVICE_PORT` environment variables.

**DNS mode**: Kubernetes runs a DNS server (CoreDNS) as part of the cluster. Services are accessible via names like `my-svc.my-namespace.svc.cluster.local`. CoreDNS watches Services and Endpoints and returns A records matching healthy Pod IPs.

```
DNS resolution in Kubernetes:
  my-svc → A records matching healthy Pod IPs
  
  For a Service with 3 Pods:
    dig my-svc.default.svc.cluster.local
    ;; ANSWER:
    my-svc.default.svc.cluster.local. 30 IN A 10.244.1.5
    my-svc.default.svc.cluster.local. 30 IN A 10.244.2.8
    my-svc.default.svc.cluster.local. 30 IN A 10.244.3.2
```

### Key Design Decisions

- **etcd as source of truth**: All service and endpoint data is stored in etcd. etcd uses Raft for strong consistency.
- **Watch-based updates**: The Kubernetes API server and CoreDNS use watches on etcd to receive real-time updates when Services or Endpoints change.
- **No client-side discovery**: Kubernetes uses a server-side model where kube-proxy (on every node) programs iptables rules to route traffic. The application does not need to do discovery.
- **Headless services**: For stateful applications, headless Services return Pod IPs directly, allowing the application to do its own load balancing.

## Open Source Implementations

### Consul

Consul by HashiCorp is the most fully-featured open source service discovery tool. It provides:

- **Service discovery** via DNS and HTTP API
- **Health checking** with configurable checks (HTTP, TCP, gRPC, shell scripts)
- **KV store** for configuration data
- **Multi-datacenter** support via WAN gossip
- **Service mesh** with Consul Connect (mTLS, L7 traffic management)
- **ACL-based security** for controlling access to registry operations

Consul uses a hybrid model: Raft for data consistency, gossip for membership management. This gives strong consistency for service registration data with fast failure detection through gossip.

### etcd

etcd by Red Hat / CNCF is a distributed key-value store that powers Kubernetes. It provides:

- **Strong consistency** via Raft consensus
- **Watch support** for real-time change notifications
- **Lease mechanism** for TTL-based key expiration (used for heartbeats)
- **Simple API** over gRPC (range, put, delete, txn, watch)
- **High write throughput** (10,000+ writes/second)

etcd is not a service discovery system per se — it is a building block. You build discovery on top of etcd by storing service information under a structured key scheme and using watches to detect changes.

### Apache ZooKeeper

ZooKeeper is the oldest of the three. It provides:

- **Hierarchical namespace** (ZNodes similar to a filesystem)
- **Ephemeral ZNodes** that disappear when the creating client disconnects
- **Sequential ZNodes** for ordering operations
- **Watches** for change notifications
- **Zab consensus protocol** (similar to Raft)

ZooKeeper is battle-tested at massive scale (used by Apache Kafka, Solr, and many other distributed systems). However, it has more complex operational characteristics than etcd or Consul.

## Cloud-Managed Services

### AWS Cloud Map

AWS Cloud Map is a fully managed service registry. It allows you to register application resources (EC2 instances, Lambda functions, ECS tasks) with custom attributes. Cloud Map integrates with Route 53 to provide DNS-based discovery (A records, SRV records) and with CloudWatch for health checking.

Key features:
- **Namespace-based**: Resources are organized into namespaces representing environments or applications.
- **API-based discovery**: Query via `DiscoverInstances` API call.
- **DNS-based discovery**: Instances are automatically added to Route 53 DNS records.
- **Health integration**: Instances are marked unhealthy when their CloudWatch health check fails.

### GCP Service Directory

Google Cloud Service Directory is a managed registry that supports both DNS and API-based discovery. It integrates with Google Cloud's existing networking and load balancing services.

Key features:
- **Multiregional**: Supports services that span multiple regions.
- **gRPC and HTTP APIs**: Both protocols supported for service registration and lookup.
- **IAM integration**: Access control through Google Cloud IAM.
- **Cloud DNS integration**: Automatically creates DNS records for registered services.
