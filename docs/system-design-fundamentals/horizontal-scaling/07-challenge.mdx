# Horizontal Scaling — Challenges

## Instructions

Solve each exercise within the stated time limit. Draw your architecture diagram on paper or a whiteboard. Describe your reasoning aloud as you work. After solving, compare with the model answer.

**General rubric:**
- Does your solution include concrete numbers (QPS, server count, shard count, latency)?
- Does it identify the bottleneck at each stage?
- Does it discuss configuration parameters (pool sizes, TTLs, thresholds)?
- Does it include failure modes and mitigations?

## Tier 1 — Foundation (5 minutes each)

### Problem 1: Connection Pool Sizing

Your application has 50 stateless app servers behind an ALB. Each server processes 300 RPS. Each request makes 2 database queries. The database is PostgreSQL with `max_connections = 400`. Each connection uses 2 MB of RAM.

**Calculate the connection demand vs supply. What pool size should each server use? How much RAM does the database spend on connections?**

<details>
<summary>Model Answer</summary>

**Demand calculation:**
Total RPS: 50 servers * 300 RPS = 15,000 RPS
Database queries per second: 15,000 * 2 = 30,000 QPS
Average query time: assume 10ms (well-indexed, in cache)
Concurrent connections needed: 30,000 * 0.010 = 300 concurrent connections

**Supply calculation:**
Database max_connections = 400
Reserve 50 connections for maintenance, admin console, monitoring
Available for app: 350 connections
Pool per server: 350 / 50 = 7 connections per server

**RAM calculation:**
350 connections * 2 MB = 700 MB RAM for connection overhead

**Pool configuration per server:**
```python
pool_size = 7          # 7 * 50 = 350 total
max_overflow = 2       # Burst capacity, total 100 extra
pool_timeout = 5       # Fast fail — don't queue for 30 seconds
max_lifetime = 1800    # Recycle every 30 minutes to avoid firewalls
```
</details>

### Problem 2: Cache Sizing and Hit Rate

You have a 200 GB database. The working set is 60 GB of frequently accessed data. You configure a Redis cluster with 4 nodes, each with 16 GB of RAM. Redis overhead is approximately 25% (replication buffers, command buffers, connection overhead).

**Can this cache hold the working set? What happens when the working set grows to 80 GB? What is the eviction policy, and how does it affect hit rate?**

<details>
<summary>Model Answer</summary>

**Usable cache capacity:**
Per node: 16 GB * 0.75 = 12 GB usable
Total: 4 nodes * 12 GB = 48 GB usable
Working set: 60 GB
Cache can hold 80% of the working set (48/60)

**Result:** 20% of working set data is always evicted. Cache hit rate will be 80% at best. Frequently accessed data is evicted and re-queried from the database, adding 10-50ms per miss.

**When working set grows to 80 GB:**
Usable cache: 48 GB
Working set: 80 GB
Hit rate drops to 60% (48/80). Database load increases by 2x.

**Solution:** Add 2 more nodes (6 * 12 = 72 GB usable), or upgrade to 32 GB nodes (4 * 24 = 96 GB usable).

**Eviction policy:** LRU (Least Recently Used). With the allkeys-lru policy, Redis evicts the least recently accessed key when memory is full. This is the right choice for most applications — the most popular data stays cached.
</details>

### Problem 3: Auto-Scaling Thresholds

Your auto-scaling group runs 10 instances during normal traffic. Traffic spikes to 3x normal at lunchtime. Instances take 6 minutes to initialize and register with the load balancer. Your current rule: scale out when CPU exceeds 70% for 3 minutes, scale in when CPU drops below 30% for 10 minutes.

**A lunch rush traffic spike goes from 1,000 RPS to 3,000 RPS in 30 seconds. Existing instances are at 80% CPU at 1,000 RPS. Walk through the timeline. How many users see errors before new instances come online?**

<details>
<summary>Model Answer</summary>

**Timeline:**
t=0s: Traffic starts rising from 1,000 RPS
t=30s: Traffic hits 3,000 RPS. CPU on 10 instances = 240% of normal load. Instances cannot handle 3x traffic — they saturate at 100% CPU, handling about 1,250 RPS.
t=30s to 3min: 1,750 RPS is dropped (HTTP 503 or connection timeout). Total dropped requests: 1,750 * 150s = 262,500 requests.
t=3min: Auto-scaling alarm fires (CPU above 70% for 3 minutes). Launch 4 instances (typical scale-out increment).
t=6min: Instances boot, register with ALB, pass health checks (total 9 minutes from spike start).
t=9min: 14 instances online. If each handles 125 RPS at 100% CPU: 14 * 125 = 1,750 RPS. Still cannot handle 3,000 RPS.
t=12min: Second scale-out triggers. 4 more instances launch.
t=18min: 18 instances handle 2,250 RPS. Still short.
t=24min: 22 instances handle 2,750 RPS. Close to target.

**Result:** Approximately 263,000 requests are dropped in the first 3 minutes. After that, partial capacity causes degraded performance for 20+ minutes.

**Fix:** Predictive auto-scaling (provision to 3x capacity before lunch), or scheduled scaling (set desired capacity to 30 at 11:30 AM, reduce at 2:00 PM).
</details>

## Tier 2 — Design (15 minutes each)

### Problem 4: Design a Horizontally Scaled URL Shortener

**Requirements:**
- 100 million URLs created per month
- 500 million redirections per day
- P99 latency under 20ms for redirection
- URLs live forever (no TTL-based expiry)
- Must handle traffic spikes (viral link posted to social media)

**Scale estimates:**
URLs created per second: 100M / (30 * 86,400) = 39 writes/second
Redirections per second: 500M / 86,400 = 5,787 reads/second
Spike capacity: 10x normal = 57,870 reads/second
Total storage after 5 years: 100M * 12 * 5 = 6 billion URLs

**What to design:**
1. The application tier (stateless, behind load balancer)
2. The cache tier (Redis — what size, how many nodes)
3. The database tier (sharded or single, shard key, number of shards)
4. How reads and writes route through the system
5. How you handle the viral-link traffic spike

<details>
<summary>Model Answer</summary>

**Architecture:**
- Stateless API servers behind ALB, auto-scaled (min: 5, max: 50). Each handles 2,000 RPS.
- Redis cache cluster: Read-heavy workload. 90% of reads should hit cache. Cache stores short_code to long_url mapping with no TTL (URLs are permanent). Use LRU eviction to drop least-popular URLs.
- Database: sharded MySQL or PostgreSQL by short_code hash. 8 shards with 2 replicas each. The shard key is the first byte of the base62-decoded short code.

**Cache sizing:**
Total URLs after 5 years: 6 billion
Each cache entry: ~200 bytes (key + value + overhead)
Total cache data: 6B * 200 bytes = 1.2 TB
Working set (top 10% most-accessed URLs): 120 GB

Cache configuration: Redis cluster, 12 nodes at 16 GB each (144 GB total, about 108 GB usable). Enough for the working set. LRU eviction handles the rest.

**Viral link handling:**
When a URL goes viral, its cache entry is hot — it is always in cache because LRU promotes it. The entire traffic spike is absorbed by the cache. The cache key is a single Redis GET operation (under 0.5ms). The database sees zero load for that URL because it is cache-resident.
</details>

### Problem 5: Design Auto-Scaling for a Flash Sale Platform

**Requirements:**
- E-commerce platform expecting 1 million concurrent users during a flash sale
- Normal traffic: 10,000 concurrent users
- Flash sale starts at exactly 20:00 UTC
- Products go on sale for 2 hours
- Every user refreshes at 19:59, causing a 100x traffic spike in seconds

**What to design:**
1. How many servers during normal vs flash sale traffic
2. Auto-scaling strategy — reactive, scheduled, or predictive
3. How to handle the cold-start problem (instances starting too late)
4. Cost optimization — what runs during normal vs flash sale

<details>
<summary>Model Answer</summary>

**Server count:**
Normal traffic: 10,000 concurrent / 500 per instance = 20 instances
Flash sale: 1,000,000 / 500 = 2,000 instances needed

**Strategy: Scheduled scaling, NOT reactive.**
Reactive auto-scaling will fail — by the time CPU crosses a threshold and instances start, the sale is 10 minutes in and users have already left.

**Scheduled scaling plan:**
- 19:00: Scale from 20 to 500 instances (pre-warm 1 hour before)
- 19:30: Scale to 1,000 instances
- 19:45: Scale to 1,500 instances
- 19:55: Scale to 2,000 instances
- 22:00: Scale down aggressively (sale ends, admin closes remaining orders)
- 22:30: Scale to 20 instances (normal operations)

**Cold start mitigation:**
- Pre-pull container images on all worker nodes
- Use warm pools: keep 50 instances at 0% CPU (they exist but handle no traffic) for instant scale-up
- Database connection pools: scale database connection limits BEFORE the event (double `max_connections` temporarily)

**Cost optimization:**
- Use spot instances for 70% of the flash sale capacity (savings: 60-70%)
- On-demand instances for the baseline that must survive spot termination
- Reserve instances for the pre-warmed capacity

**Prayer rule of auto-scaling:**
Do not let reactive auto-scaling handle known traffic events. Schedule for what you know, react only to what you cannot predict.
</details>

### Problem 6: Design a Horizontally Scaled Real-Time Chat System

**Requirements:**
- 1 billion messages per day
- 500 million active users
- Messages must be delivered in under 100ms P99
- History must be available for 30 days
- Users are grouped into channels of 2 to 10,000 members

**What to design:**
1. How messages flow from sender to all recipients
2. How channels with 10,000 members handle fan-out
3. How to store 1 billion messages per day (30 TB/month)
4. How to horizontally scale WebSocket connections

<details>
<summary>Model Answer</summary>

**Architecture:**
- WebSocket servers behind NLB (NLB can handle millions of concurrent connections, ALB would cost more for WebSocket idle connections). Each WebSocket server handles 100,000 concurrent connections.
- 500M users / 100K per server = 5,000 WebSocket servers needed.
- Message broker: Kafka. Each channel is a Kafka topic partition. A message to a channel is written to that partition. All members of the channel consume from that partition.

**Fan-out optimization for large channels:**
- Channels under 1,000 members: direct fan-out — each member gets the message individually
- Channels over 1,000 members: Kafka-based fan-out — write once to the channel partition, each WebSocket server's consumer group consumes it and broadcasts to its connected members on that channel

**Storage:**
1 billion messages * 1 KB average = 1 TB/day
30-day retention: 30 TB total
Database: Cassandra (natively horizontally scalable, no single point of failure). 30 nodes with 1 TB each, replication factor 3.
Time-based partition key: (channel_id, message_timestamp_hour) for efficient range queries.

**Horizontal WebSocket scaling:**
- WebSocket servers are stateful (they hold open connections). Use consistent hashing to route a user to the same server for the duration of their session.
- Use a Redis-based session registry: when a user connects, register (user_id -> server_id). When a message needs to be pushed, look up which server holds that user's connection and forward the message.
</details>

## Tier 3 — Expert (30 minutes each)

### Problem 7: Design a Horizontally Scaled Global Payment System

**Requirements:**
- 100 million payment transactions per day across 50 countries
- ACID compliance required for each transaction
- P99 latency under 100ms for the payment processing API
- 99.999% uptime (5 minutes of downtime per year)
- Regulatory requirements: data must stay within the user's country (GDPR, Indian data localization, Chinese data laws)
- The system must survive the loss of an entire AWS region

**Key constraints:**
ACID compliance means you cannot use eventually consistent databases. Data localization means you cannot use a single global database. Zero-downtime means you cannot have a single point of failure.

**Evaluation dimensions:**
- How do you achieve ACID transactions across shards that are in different countries?
- How do you handle cross-border payments (payer in EU, merchant in US)?
- How do you test for regional failure? What is your failover drill schedule?
- How do you reconcile transactions that were in-flight during a regional failure?
- What does the read-after-write path look like for a user checking their transaction history?

**Anti-patterns to avoid:**
- A single global database violates data localization laws
- Active-active writes without conflict resolution leads to double-charges
- Synchronous cross-region commits would add 200ms+ latency to every cross-border payment
- Storing all data in a single region and serving globally violates GDPR explicitly

<details>
<summary>Discussion Points</summary>

This is an open-ended expert-level problem. A strong solution would address:

1. **Per-country cell architecture**: Each country is an independent cell with its own database, cache, and compute. Data never leaves the country.

2. **Two-phase commit within a cell**: Each cell runs its own database (PostgreSQL with synchronous replication across 3 AZs within the same region). ACID transactions work within a cell.

3. **Cross-border payments via idempotent queues**: A payment from EU to US is broken into two local operations. The EU cell records a debit to the EU account and enqueues a "credit US account" message. The US cell consumes the message and credits the US account. Each message is idempotent (same message processed twice produces the same result).

4. **Regional failover**: Each cell has a warm standby in another region within the same geopolitical boundary. US data has a standby in us-west-2 if primary is us-east-1. Failover is tested monthly and automated (Route53 health checks + RDS cross-region read replica promotion).

5. **Reconciliation**: A batch process runs every 15 minutes comparing the total debits and credits across all cells. Discrepancies trigger alerts and manual investigation.

6. **Read-after-write consistency**: The user's write goes to their local cell. Their next read routes to the same cell. The read is strongly consistent within that cell.
</details>

### Problem 8: Design a Horizontally Scaled Video Upload and Processing Pipeline

**Requirements:**
- 500,000 video uploads per day (average 100 MB each = 50 TB/day)
- Each upload must be transcoded to 3 resolutions (480p, 720p, 1080p)
- Transcoding must complete within 5 minutes of upload
- Users expect to see their video in their library within 30 seconds of upload
- The system must handle 10x spikes during major events (Super Bowl halftime, awards shows)
- Videos are stored for 5 years
- Upload bandwidth peaks at 5 Gbps during normal operation, 50 Gbps during spikes

**What to design:**
1. How uploads are received and acknowledged quickly
2. How the processing pipeline scales (transcoding is CPU/GPU intensive)
3. How storage scales (50 TB/day * 365 * 5 = 91 PB over 5 years)
4. How the spike-handling strategy works

<details>
<summary>Discussion Points</summary>

1. **Upload path**: Users upload directly to S3 via pre-signed URLs. The API server validates the upload token and generates a presigned URL in 2ms, then returns immediately. No file data passes through the application server. This makes the upload path horizontally scalable with zero state.

2. **Queue for transcoding**: S3 put-event triggers an SQS message. A pool of transcoding workers (auto-scaled based on queue depth) consumes messages from SQS. Each worker is a GPU instance (G4dn or similar) that transcodes one video at a time. Transcoding time per video per resolution: approximately 30 seconds on GPU. Three resolutions: 90 seconds total. 500,000 videos * 90 seconds = 45 million compute-seconds per day = 520 GPU-instance-days. At 24 hours per day: 500,000 * 90 / 86,400 = 520 concurrent GPU instances needed.

3. **Storage**: S3 with lifecycle policies. Uploads stored in S3 Standard (fast access during first week). After 30 days, move to S3 Infrequent Access. After 1 year, move to S3 Glacier. After 3 years, move to S3 Glacier Deep Archive. This reduces storage cost by approximately 80% compared to keeping everything in Standard.

4. **Spike handling**: The queue absorbs the spike naturally. During the Super Bowl, uploads may spike to 5 million in 10 minutes. The queue grows to several million messages. Workers auto-scale to 10x their normal count. The video processing extends from 5 minutes to 30+ minutes during the spike, but nothing fails. Users see a "processing" status immediately because the upload itself (S3 presigned URL) is instant.
</details>

## Reflection

After completing these challenges, review your solutions against the rubric:

1. Did you calculate concrete numbers, or rely on hand-waving?
2. Did you identify the bottleneck at each scaling threshold?
3. Did you discuss failure modes and mitigations?
4. Did your architecture have any single points of failure?
5. Could your design survive a 10x traffic spike without manual intervention?

The gap between what you designed and what would actually work in production is where the most valuable learning happens. Repeat the exercises with different constraints (half the budget, 10x the traffic, stricter latency requirements) to deepen your understanding.
