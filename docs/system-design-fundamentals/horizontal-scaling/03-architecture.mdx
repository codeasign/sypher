# Horizontal Scaling at Scale

## Naive Implementation: The Simplest Version

A single application server behind a basic load balancer. This is the starting point for most systems.

### What Breaks at 1,000 Users

Nothing. This setup handles 1,000 concurrent users with P95 latency under 100ms. The database handles 500 QPS comfortably. The app server runs at 20% CPU. The load balancer's health check pings every 5 seconds and the single server always responds.

### What Breaks at 100,000 Users

The app server runs at 95% CPU. The database handles 5,000 QPS and hits 80% CPU. Response times degrade to 2 seconds. The single app server is a single point of failure — if it crashes, the entire system goes down.

**Fix**: Add more app servers behind the load balancer. The app must be stateless for this to work.

### What Breaks at 10,000,000 Users

Everything. The single database cannot handle 50,000 QPS. The load balancer itself becomes a bottleneck (single ALB handles up to 25,000 connections per second — beyond that, you need multiple LBs). The app servers need to be in multiple regions. The naive implementation was never designed for this scale.

### Component Breakdown

**1. Global DNS (Route53 / Cloud DNS)**: Latency-based routing directs users to the nearest region. Health checks stop routing to a region if its load balancers are unhealthy. TTL is set to 60 seconds to allow fast failover without excessive DNS queries.

**2. Dual Load Balancers (NLB + ALB)**: Network Load Balancer handles TCP termination at the edge (millions of packets per second). Application Load Balancer handles HTTP routing, path-based routing, and TLS termination. The NLB shields the ALB from DDoS attacks and provides static IP addresses for DNS.

**3. App Auto-Scaling Group**: 20-100 instances per region, auto-scaled based on CPU, memory, and request queue depth. Instances are stateless — session data lives in Redis. Each instance registers with the ALB target group on startup and de-registers on shutdown (connection draining handles in-flight requests).

**4. Redis Cluster**: 6 nodes per region, each with a replica. Data is sharded across 16,384 hash slots. Redis Sentinel handles automatic failover. The cluster provides session storage, cache, and rate-limit counters.

**5. Database Shard Ring**: 4 shards per region, each with 2 replicas. Shards are distributed by consistent hashing of user ID. Read replicas handle 90% of query traffic. Cross-region replication runs asynchronously — writes replicate to other regions with 50-500ms latency.

**6. Kafka Multi-Region**: 24 partitions, replication factor 3. MirrorMaker 2 syncs topics across regions. Producers write to the local region's broker. Consumers in each region read from their local broker. This provides exactly-once semantics for order processing and event sourcing.

## Integration Patterns

### Pattern 1: RDBMS with Read Replicas

The most common horizontal scaling pattern for relational databases. Writes go to the primary; reads distribute across replicas.

/>

**Key consideration**: Replication lag. The analytics replica may lag by 200ms or more under heavy load. If your application reads its own writes on the analytics replica, users will see stale data. Route user-facing reads to the primary or a low-lag replica.

**Key consideration**: Cache invalidation. When a product price changes, the cache must be invalidated or updated. Write-through cache updates the cache on every write (adds 1ms to write latency). Write-behind cache batches updates (higher throughput, risk of stale data). TTL-based expiration is simplest but can serve stale data for up to the TTL duration.

## Multi-Region and Global Scale

When a single region cannot serve all users with acceptable latency, the system must span multiple geographic regions.

  alt="Multi-region global topology with 4 regions: us-east, eu-west, ap-south, and sa-east, with us-east as primary write region, eu-west and ap-south as active read regions with async replication, and sa-east as read-only with local cache"
  caption="Multi-region horizontal scaling: active-active regions for reads, single write region to avoid conflicts, async replication for data propagation"
/>

**Key design decisions:**

1. **Active-Passive vs Active-Active**: Active-passive has one region accepting writes, others serving reads. Active-active has multiple regions accepting writes. Active-active requires conflict resolution — last-write-wins (LWW) is simplest but loses data; CRDTs preserve data but are complex.

2. **Cross-region replication latency**: us-east to eu-west is roughly 80-100ms. us-east to ap-southeast is 150-200ms. Write latency for the user is the round-trip to the nearest write region. If writes go to us-east regardless of user location, a user in Tokyo sees 150ms write latency.

3. **Failover strategy**: If the primary write region fails, traffic must reroute to another region. This requires DNS failover (TTL 60s), database failover (promote replica to primary), and cache warm-up (regional cache starts cold). Total failover time: 5-15 minutes.

## Capacity Planning

Capacity planning for horizontal scaling means answering: how many nodes of what size in how many regions?

### Realistic Example

**Requirements:**
- 50 million daily active users
- 10 requests per user per day (5 reads, 5 writes)
- 50:1 read-to-write ratio
- Peak traffic is 3x average (lunch rush)
- Target P99 latency: under 200ms

**Step 1: Calculate peak throughput**

```
Daily requests: 50M users * 10 requests = 500M requests/day
Average RPS: 500M / 86,400 seconds = 5,787 RPS
Peak RPS: 5,787 * 3 = 17,361 RPS
Reads: 17,361 * 50/51 = 17,020 RPS
Writes: 17,361 * 1/51 = 341 RPS
```

**Step 2: Size the application tier**

```
Each app server handles 500 RPS (typical for Python async framework)
Servers needed: 17,361 / 500 = 35 servers
Add 30% headroom for failover: 35 * 1.3 = 46 servers
Add 30% for regional distribution: 46 * 1.3 = 60 servers
```

**Step 3: Size the database tier**

```
Each database shard handles 2,000 read QPS + 200 write QPS
Read shards needed: 17,020 / 2,000 = 9 shards
Write shards needed: 341 / 200 = 2 shards
Total shards: 9 (reads dominate, combine read/write per shard)
Add 2 replicas per shard: 9 * 3 = 27 database nodes per region
```

**Step 4: Size the cache tier**

```
Working set: 50% of data accessed frequently
Total data: 500 GB
Working set: 250 GB
Redis node memory: 32 GB (with overhead, ~24 GB usable)
Cache nodes: 250 / 24 = 11 nodes
Add replica: 11 * 2 = 22 nodes per region
```

**Step 5: Cost estimate**

```
App tier: 60 servers * $50/month = $3,000/month
Cache tier: 22 nodes * $200/month = $4,400/month
Database tier: 27 nodes * $500/month = $13,500/month
Load balancers: 2 * $25/month = $50/month
Kafka cluster: 12 nodes * $300/month = $3,600/month
Network transfer: 50 TB * $0.09/GB = $4,500/month

Per region total: ~$29,050/month
Three regions: ~$87,150/month
```

## Deployment and Operations

### Zero-Downtime Deploy

A horizontally scaled system enables zero-downtime deployments through rolling updates.

```
1. Deploy to 1 node (10% of traffic)
   - Monitor error rate, latency, CPU for 5 minutes
   - If metrics degrade: rollback, investigate
   - If metrics stable: proceed

2. Deploy to 25% of nodes
   - Monitor for 3 minutes
   - Check for gradual issues (memory leaks, slow connection leaks)

3. Deploy to 50% of nodes
   - Monitor for 3 minutes

4. Deploy to 100% of nodes
   - Monitor for 10 minutes
```

### Monitoring

Every horizontal scaling layer needs specific metrics:

```
Load Balancer:
  - Active connections per target
  - Request count per second
  - 5xx error rate
  - Target health status

Application Server:
  - CPU, memory, network I/O
  - Thread pool utilization (active, queued, rejected)
  - Connection pool utilization (active, idle, waiting)
  - Circuit breaker state per dependency
  - P50/P95/P99 latency

Database:
  - Connections per shard
  - Replication lag per replica
  - Query latency per shard
  - Cache hit ratio per index

Cache:
  - Hit rate per node
  - Memory usage per node
  - Eviction rate
  - Command latency
```

### Debugging Common Issues

| Symptom | Likely Cause | Check |
|---------|-------------|-------|
| P99 latency spikes after deploy | Connection pool exhaustion | Active connections vs pool size |
| One node shows 100% CPU, others 20% | Uneven load distribution | LB scheduling algorithm, sticky sessions |
| Cache hit rate drops from 90% to 40% | Cache key format changed, or TTL too short | Cache keys in new deploy, TTL config |
| Errors on every Nth request | Connection leak in pool | Max connections, active connections over time |
| Auto-scaling adds instances but throughput stays flat | Database bottleneck | DB CPU, query latency, connection count |
| New instances fail health checks for 5 minutes | Slow startup / cache warm-up | Startup time, dependency initialization order |

This architecture forms the foundation for evaluating trade-offs — the topic of the next section.
