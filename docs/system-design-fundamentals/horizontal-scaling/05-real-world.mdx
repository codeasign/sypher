# Horizontal Scaling in Production

## Company Case Studies

### Netflix: Stateless Microservices at 260 Million Subscribers

Netflix operates one of the largest horizontally scaled systems in the world, serving 260 million subscribers across 190+ countries. Every day, Netflix's content delivery infrastructure streams petabytes of video, and its control plane handles billions of API requests.

**The challenge**: Netflix's monolith (built in 2008) could not scale further. A single deploy took weeks. A single bug could take down the entire streaming service. The team needed to split the monolith into hundreds of independently scalable services.

**The solution**: Netflix migrated to a fully stateless microservice architecture on AWS. Each microservice is horizontally scaled behind a load balancer. The key architectural components are:

- **Eureka** (service discovery): Every service registers itself on startup and deregisters on shutdown. Services discover each other via DNS-like lookups. Eureka is itself horizontally scaled across multiple availability zones.
- **Zuul** (API gateway): Routes requests to the appropriate backend service. Handles authentication, rate limiting, and request routing at the edge. Zuul runs in an auto-scaling group behind an ELB.
- **Hystrix** (circuit breaker): Every service-to-service call goes through Hystrix. Circuit breakers, thread pool isolation, and fallbacks prevent cascading failures. Hystrix has three states (CLOSED, OPEN, HALF-OPEN) and tracks failure metrics in a rolling 10-second window.
- **Scryer** (predictive auto-scaling): A machine learning model that forecasts traffic 1-3 hours ahead based on historical patterns, content releases, and day-of-week. Scryer provisions instances before demand arrives, eliminating the 8-10 minute cold-start lag.

### Uber: Sharded Database at 93 Million Monthly Active Users

Uber's core trip database handles 93 million monthly active users and 25 million trips per day. Every trip generates writes to multiple tables — the trip request, driver assignment, GPS tracking points, payment records, and rider feedback.

**The challenge**: Uber's original monolithic PostgreSQL database (a single instance) could not keep up with growth. By 2015, the database was writing 10 million GPS points per minute. The primary was at 100% CPU during peak hours. Schema migrations took hours and required downtime.

**The solution**: Uber built a custom sharding layer called **Schemaless** (later deprecated in favor of DocStore) that split data across MySQL shards. The key innovations were:

- **Cell-based architecture**: Each "cell" is a self-contained set of shards serving a geographic region. Cells are completely independent — a failure in one cell does not affect others. New cells are added as cities are launched.
- **Dynamic shard splitting**: When a shard exceeds capacity, it is automatically split into two shards. The split is transparent to application code. Data is migrated in the background using a custom replication tool.
- **Ringpop** (consistent hashing): Uber's Ringpop library implements consistent hashing for service-to-service communication. It uses a SWIM gossip protocol for membership detection and failure discovery. When a node fails, Ringpop redistributes its keys to remaining nodes in O(1/N) time.

### Discord: From Monolith to Horizontally Scaled Microservices

Discord serves 150 million monthly active users and handles 25 million messages per day across millions of concurrent voice and text channels.

**The challenge**: Discord's original architecture was a single Python monolith with a MongoDB database and a single Redis instance. By 2017, the system was hitting hard limits: MongoDB could not handle 1.2 million concurrent connections, and the single Redis instance ran out of memory for session storage.

**The solution**: Discord horizontally scaled each layer:

- **Redis sharding**: Session data was sharded across 20 Redis nodes using consistent hashing. Each node holds 1/20th of all sessions. When a node fails, only 5% of sessions are lost (not 100%).
- **Database migration to Cassandra**: Discord moved real-time data (messages, voice state) to Cassandra, which is natively horizontally scalable. Cassandra distributes data across nodes using consistent hashing with tunable consistency levels. Discord runs 200+ Cassandra nodes across multiple data centers.
- **Kafka for event streaming**: All important events — messages sent, users joined, permissions changed — flow through Kafka. This decouples the write path from downstream consumers. Kafka uses 12 partitions per topic, allowing up to 12 concurrent consumers.

### Cloudflare: Global Load Balancing at Edge Scale

Cloudflare operates a global network of 330+ data centers serving 25+ million internet properties. Their horizontally scaled edge network is one of the largest in the world.

**The challenge**: Every DNS query and HTTP request must be routed to a healthy edge data center within milliseconds. The routing system must handle 35 million HTTP requests per second and survive the failure of entire data centers.

**The solution**: Anycast routing. Cloudflare announces the same IP addresses from all 330+ data centers. BGP routing automatically directs traffic to the nearest healthy data center. When a data center fails, BGP withdraws the route and traffic shifts to the next closest data center within 30-60 seconds. No load balancer, no DNS TTL — the internet's routing protocol handles the failover automatically.

## Open Source Implementations

### Cassandra

Apache Cassandra is natively horizontally scalable. It distributes data across nodes using consistent hashing on a ring. Each node owns a range of hash tokens. Data is replicated to N nodes (replication factor, typically 3). Reads can be served by any replica.

**Trade-offs:**
- Writes are always available (AP system): any node accepts writes at any time, even during network partitions
- Read consistency is tunable: ONE (read from one replica, may be stale), QUORUM (read from majority), ALL (read from all replicas)
- Operations-heavy: repair, compaction, and node replacement require experienced operators

### CockroachDB

CockroachDB is a horizontally scalable SQL database that provides ACID transactions across nodes. It uses the Raft consensus algorithm for replication and a distributed key-value store underneath the SQL layer.

**Trade-offs:**
- Strong consistency at the cost of write latency (Raft requires majority acknowledgment)
- SQL compatibility means existing applications can be migrated with minimal changes
- Higher latency than Cassandra for writes (serialized through Raft)

### Kubernetes

Kubernetes is the de facto standard for deploying horizontally scaled applications. Its core abstraction — the ReplicaSet — ensures N copies of a pod are running. The Horizontal Pod Autoscaler (HPA) adjusts replica counts based on CPU, memory, or custom metrics.

**Trade-offs:**
- Pod startup time: container image pull and initialization takes 10-60 seconds (not suitable for sub-minute traffic spikes)
- Cluster autoscaling (adding nodes to the cluster) takes 2-5 minutes
- StatefulSets for stateful applications require careful design (PersistentVolumeClaims are not auto-migrated)

## Lessons Learned

1. **Stateless is the foundation of horizontal scaling.** Every company that successfully scaled horizontally — Netflix, Uber, Discord — invested heavily in making services stateless. Session data goes to Redis. File uploads go to S3. Local disk is empty. If your service writes to local disk, it cannot scale horizontally without complex shared storage.

2. **Horizontal scaling exposes every hidden serial dependency.** A database query that took 2ms under light load takes 200ms under high load because of queueing. A synchronous API call that chained 3 services added negligible latency at 100 RPS. At 10,000 RPS, it adds seconds. Measure and remove serial dependencies before they become bottlenecks at scale.

3. **Circuit breakers are not optional.** Every team at Netflix, Uber, and Discord has a story of a cascading failure that was stopped by a circuit breaker. The one team that did not implement circuit breakers caused a production outage. The cost of implementing circuit breakers (a few hours per service) is dwarfed by the cost of a single P0 outage.

4. **Auto-scaling is a lagging indicator.** By the time CPU crosses a threshold, traffic has already been high for minutes. Predictive and scheduled scaling are essential for traffic patterns with known peaks. Even Amazon's own retail site uses scheduled scaling for Prime Day, not reactive auto-scaling.

5. **Cross-region writes are the hardest problem in distributed systems.** Conflict resolution, replication lag monitoring, and failover testing are not solved problems at any company. The simplest approach — active-passive with a single write region — is the most reliable for 95% of use cases.
