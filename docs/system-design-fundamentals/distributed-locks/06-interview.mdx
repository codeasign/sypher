## Level-by-Level Expectations

### Entry Level (0-2 years)

At this level, the focus is on understanding the basic concept and being able to implement a simple lock using existing infrastructure.

**Must know**: What a distributed lock is and why it is needed. The basic SETNX pattern in Redis. The difference between a distributed lock and a local mutex. The concept of a TTL and why it matters.

**Can mention**: The job scheduler problem (two servers running the same job). The bathroom key analogy. Redis SETNX with TTL.

**Should NOT claim**: Deep expertise in consensus protocols or fencing token internals. It is acceptable to say "I have not worked with ZooKeeper locks in production."

### Mid Level (2-5 years)

At this level, the focus is on implementation details and trade-offs.

**Must know**: How to implement a distributed lock with Redis (SETNX + Lua script for release). How to choose TTL values. The risk of asynchronous replication. The difference between Redis-based and ZooKeeper-based locks.

**Can mention**: The Redlock algorithm and its controversy. ZooKeeper ephemeral sequential znodes. Lease-based locks. Retry with backoff and jitter.

**Should understand**: Clock drift and its effect on locks. The fencing token concept. How to implement safe lock release with owner checking. The thundering herd problem and wait queues.

### Senior Level (5-8 years)

At this level, the focus is on designing lock systems and handling edge cases.

**Must know**: How to design a production lock service with consensus (Raft or Zab). Fencing tokens in detail. How to handle network partitions. The safety versus liveness trade-off in depth.

**Can mention**: Google Chubby architecture. ZooKeeper lock recipe internals. etcd concurrency package. Redlock critique by Kleppmann. Multi-region lock strategies.

**Should understand**: How Raft log replication affects lock latency. How to implement a fencing token generator. How to handle clock drift without relying on NTP. The CAP theorem implications for locks.

### Staff and above (8+ years)

At this level, the focus is on system architecture, cross-team coordination, and advanced failure modes.

**Must know**: How to design a lock service used across multiple teams and services. Cost-benefit analysis of different lock implementations. Capacity planning for lock clusters. When NOT to use distributed locks.

**Can mention**: Geo-distributed consensus (Geo-Raft). CRDT-based conflict resolution as an alternative to locks. Formal verification of lock algorithms. Industry case studies and their outcomes.

**Should understand**: The mathematical foundations of consensus (Paxos, Raft). The impossibility of perfect clock synchronization. The FLP result and its implications for liveness. How to write a design document for a lock service.

## Phrase Bank

**For explaining the problem**: "In a distributed system, multiple processes may attempt to modify the same resource concurrently, and without coordination this can lead to data corruption. A distributed lock provides a single source of truth for who has the right to access the resource at any moment."

**For defining a lock**: "A distributed lock is a mutual exclusion mechanism that works across network boundaries. It guarantees that at most one process can hold the lock at any time, preventing concurrent access to the protected resource."

**For explaining TTL**: "The TTL serves as a safety valve. If the lock holder crashes without releasing the lock, the TTL ensures the lock is eventually released so other processes can make progress. Without a TTL, a crash would permanently block access to the resource."

**For explaining fencing tokens**: "A fencing token is a monotonically increasing number that the lock service assigns with each lock grant. The protected resource checks the token and rejects any operation with a stale token. This protects against the delayed-writer problem where a process continues writing after its lock has expired."

**For explaining the Redis versus ZooKeeper trade-off**: "Redis provides speed and simplicity but sacrifices consistency guarantees. ZooKeeper provides strong consistency through the Zab protocol but has higher latency and operational complexity. The choice depends on whether a rare lock duplication is acceptable or catastrophic."

**For explaining Redlock**: "Redlock uses N independent Redis nodes and requires a majority to acquire the lock. It improves safety over single-master Redis but does not provide fencing tokens, which limits its safety guarantees under certain failure scenarios like extended GC pauses."

**For explaining clock drift**: "Clock drift between servers is a real problem for TTL-based locks. If the lock service clock runs fast, the TTL expires early. If it runs slow, the TTL expires late. Consensus-based systems like ZooKeeper avoid this by using session timeouts instead of absolute timestamps."

**For explaining lease renewal**: "Lease renewal is the heartbeat mechanism that keeps the lock alive. The holder periodically sends renewal requests to the lock service. If the renewals stop, the lease eventually expires and the lock is released. The renewal interval is typically one-third of the TTL."

**For explaining lock contention**: "High lock contention means many processes are waiting for the same lock. This reduces throughput because processes are blocked instead of doing useful work. Solutions include using finer-grained locks, reducing the critical section duration, and using wait queues instead of polling."

**For explaining when NOT to use locks**: "Distributed locks are expensive. Each lock acquisition requires a network round trip and potentially a consensus round trip. Before reaching for a lock, consider whether the operation is idempotent, whether optimistic concurrency control works, or whether the operation can tolerate eventual consistency."

**For explaining the safety-liveness trade-off**: "There is a fundamental tension between safety and liveness. Stronger safety guarantees require more coordination, which makes the system less available. Weaker safety can lead to data corruption but keeps the system running under failure conditions. The right balance depends on the cost of each failure mode."

**For explaining deadlock prevention**: "The most effective deadlock prevention strategy is lock ordering. All processes acquire locks in the same global order. If every process acquires lock A before lock B, the circular wait condition for deadlock cannot occur."

**For explaining multi-region locking**: "Cross-region locking is dominated by the speed of light. A lock acquisition from Singapore to us-east-1 takes at least 150ms in network latency. For this reason, multi-region lock architectures typically use a single leader region or partition locks by resource to the nearest region."

**For explaining the thundering herd problem**: "When a lock is released, all waiting processes may attempt to acquire it simultaneously. This creates a spike in load on the lock service and most attempts fail. Jitter in retry intervals spreads the attempts over time, and wait queues eliminate the problem entirely."

## Red Flags

**Red Flag 1**: The candidate suggests using a distributed lock without considering alternatives. Lock-free approaches, optimistic concurrency, idempotent operations, and CRDTs should all be considered before reaching for a distributed lock.

**Red Flag 2**: The candidate does not mention TTL or expiry. A lock without a TTL can never be released if the holder crashes. This indicates a lack of understanding of distributed system failure modes.

**Red Flag 3**: The candidate treats all lock implementations as equivalent. Redis locks and ZooKeeper locks have fundamentally different safety guarantees. Treating them as interchangeable indicates shallow understanding.

**Red Flag 4**: The candidate ignores clock drift. Assuming that clocks are perfectly synchronized is a common mistake. The candidate should at least mention NTP and the possibility of clock skew.

**Red Flag 5**: The candidate does not know about fencing tokens. For senior roles, this is a critical gap. Fencing tokens are the key mechanism that makes distributed locks safe.

**Red Flag 6**: The candidate suggests infinite lock hold times. Holding a lock for hours or days defeats the purpose of a distributed lock. If an operation takes hours, it should be broken into smaller steps.

**Red Flag 7**: The candidate ignores network partitions. A distributed system can always be partitioned. The candidate should discuss what happens when the lock service and the lock holder are on different sides of a partition.

**Red Flag 8**: The candidate claims distributed locks are perfectly safe. No distributed lock system provides perfect safety. Every system has failure modes. The candidate should acknowledge the limitations.

## Practice Drills

### Drill 1: Design a Distributed Lock for a Job Scheduler

A company has 20 worker servers that process jobs from a shared queue. Each job must be processed exactly once. Design a distributed lock mechanism to ensure no job is processed twice.

**Approach**:
1. Use a lock per job with the job ID as the lock key
2. Redis SETNX with 60-second TTL (jobs complete in under 10 seconds)
3. Lua script for safe release (check owner before deleting)
4. Worker generates a unique owner ID on startup
5. If lock acquisition fails, worker polls the queue for a different job
6. Monitor lock hold duration — alert if any lock is held for more than 50 seconds

**The twist**: What if a job takes longer than 60 seconds? Use a shorter polling interval for job status checking. If the worker that acquired the lock is making progress on the job, extend the TTL. If the worker crashes, the lock expires and another worker picks up the job.

### Drill 2: Fix the Redlock Problem

A team is using Redlock with 5 Redis nodes for a payment processing system. They are experiencing rare duplicate payments. Explain why and redesign the system.

**Approach**:
1. The root cause is the lack of fencing tokens — a delayed worker can continue writing after the lock expires
2. Add a monotonic fencing token: each lock grant includes an increasing number
3. The payment database stores the last processed token for each payment batch
4. Before processing, the worker checks that its token is higher than the stored token
5. After processing, the worker updates the stored token

**The twist**: The team cannot modify the payment database to store fencing tokens. In this case, use a stricter lock service (etcd or ZooKeeper) with shorter TTLs (1 second instead of 30 seconds). This reduces the window of unsafety but does not eliminate it. Document the residual risk and accept it, or redesign the payment pipeline to be idempotent.

### Drill 3: Design a Multi-Region Lock Service

A company operates in three regions (us-east-1, eu-west-1, ap-southeast-1). They need a distributed lock that works across all regions with maximum availability.

**Approach**:
1. Use etcd with a 5-node cluster spanning all three regions: 2 nodes in us-east-1, 2 in eu-west-1, 1 in ap-southeast-1
2. Resource affinity: resources are hashed to their nearest region. Lock acquisition for a resource goes to its home region first. If the home region is unavailable, fall back to a secondary region.
3. Global fallback: if a resource must be locked across regions, use a global etcd cluster with a single leader region. Accept 150ms+ acquisition latency.
4. Compromise: For non-critical resources, use local Redis locks with best-effort coordination. Accept occasional duplicate operations.

**The twist**: The cost of a single lock failure is $1 million. Use ZooKeeper with 7 nodes across regions. Accept the latency. The cost of the lock service ($10,000/month) is negligible compared to the cost of failure. Add a formal verification step to validate the lock implementation.

### Drill 4: Migrate from Redis to ZooKeeper

A team uses Redis SETNX for critical locks. They have experienced two lock duplication incidents in six months. Design the migration to ZooKeeper.

**Approach**:
1. Phase 1: Deploy a 3-node ZooKeeper cluster alongside the existing Redis setup
2. Phase 2: Modify the application to try ZooKeeper first, fall back to Redis if ZooKeeper is unavailable
3. Phase 3: Monitor dual-path operation for 30 days. Compare lock failure rates between ZooKeeper and Redis.
4. Phase 4: Remove the Redis fallback. Require ZooKeeper for lock operations.
5. Rollback plan: If ZooKeeper has issues, switch back to Redis using a feature flag.

**The twist**: The ZooKeeper cluster is slower than Redis, and the application's throughput drops by 30%. Increase the lock granularity (fewer, larger locks) to reduce the number of acquisitions per second. Or combine ZooKeeper locks with Redis caching: check the Redis cache first (millisecond), only contact ZooKeeper if the resource is likely free (10-20ms).

## Common Interview Questions

**Q**: Design a distributed lock service for a system that processes 10,000 transactions per second.

**A**: Start by understanding the transaction nature — are these financial transactions that require strict safety, or non-critical operations? For 10,000 ops/sec with strict safety, use ZooKeeper with locking sharded across multiple lock paths to distribute load. For non-critical ops, use Redis with multiple shards. Include fencing tokens either way. Plan for 3-5 node clusters, monitor contention ratios, and design client-side retry with backoff.

**Q**: How would you handle a situation where a database row needs to be locked across 50 application servers?

**A**: Use a distributed lock with the row ID as the lock key. The lock service (Redis or ZooKeeper) manages the lock state. The application acquires the lock before reading or writing the row. The database uses optimistic concurrency control as a second line of defense. Row version numbers ensure that even if the lock fails, the database detects concurrent modifications.

**Q**: What happens if the lock service becomes unavailable?

**A**: All operations that require locks must either fail immediately, queue the work locally, or proceed without a lock with a known risk of data corruption. The safest approach is to fail closed — stop processing operations that require locks. Some systems use a circuit breaker: if the lock service is unavailable for more than N seconds, stop accepting new work to prevent a backlog.

**Q**: How do you test a distributed lock implementation?

**A**: Three layers of testing. Unit tests: verify lock acquisition, release, TTL expiry, and owner checking logic. Integration tests: run multiple processes that contend for locks and verify that no two processes hold the lock simultaneously. Chaos tests: inject process crashes, network partitions, and clock skew while running the lock service. Measure the rate of safety violations (if any) under each failure scenario.

## Additional Interview Questions

### Q5: Design a Lock for a Distributed Task Queue

A distributed task queue has 1,000 workers consuming tasks from 10,000 queues. Each queue processes tasks sequentially. Design a locking mechanism that ensures each queue has at most one active worker.

**Approach**: Use a lock per queue with the queue ID as the lock key. Redis with SETNX and a 60-second TTL is sufficient because task processing is non-critical (a duplicate task processing is handled by downstream idempotency). Each worker acquires the lock for its assigned queue before processing a task. If lock acquisition fails (another worker is processing the queue), the worker moves to a different queue.

**Edge cases to discuss**:
- What happens if a worker crashes mid-task? The TTL ensures the lock is released and another worker picks up the queue.
- What happens if the task takes longer than the TTL? The worker extends the TTL in a background thread. The extension uses a Lua script that checks ownership.
- What happens if the lock service goes down? Workers continue processing tasks from queues they already hold locks for. New queues are not processed until the lock service recovers.

### Q6: How Does a Distributed Lock Handle Network Partitions?

A network partition splits your cluster into two groups. Group A holds the lock on resource X. Group B cannot reach the lock service. What happens?

**Approach**: The lock service uses leases with TTL. When the lock holder (in Group A) cannot renew its lease due to the partition, the lease expires and the lock is released. Group B, which cannot contact the lock service, cannot acquire the lock either — it has no way to communicate with the lock service. Neither group holds the lock. The resource becomes unavailable until the partition heals.

**This is the correct behavior**: A partition should cause the lock to be unavailable rather than allowing two groups to hold the same lock. The safety property (no two holders) is preserved at the cost of liveness (no one holds the lock).

**More nuanced situation**: If the lock service is on Group B's side of the partition, and Group A holds the lock but cannot reach the lock service to renew the lease, the lease expires. If Group B has connectivity to the lock service but not to Group A, a Group B process can acquire the lock after the lease expires. This is safe as long as Group A detects the lease expiry and stops operating (fencing tokens or synchronous lock loss detection).

### Q7: Compare Lock-Based and Transaction-Based Coordination

A system needs to coordinate access to a shared database record. Compare using a distributed lock versus using a database transaction with optimistic concurrency control.

**Trade-off analysis**:

| Aspect | Distributed Lock | Database Transaction |
|--------|-----------------|---------------------|
| Latency | 2-20ms per lock operation | 1-5ms per transaction |
| Throughput | Limited by lock service | Limited by database |
| Safety | Depends on fencing tokens | ACID guarantees |
| Complexity | Separate infrastructure | Built into database |
| Scalability | Limited by lock granularity | Limited by database sharding |
| Failure handling | TTL-based recovery | Automatic rollback |

**Decision criteria**:
- Use distributed locks when the coordination spans multiple databases or services.
- Use database transactions when all coordination happens within a single database.
- Use distributed locks with database-level optimistic concurrency as a second line of defense for critical operations.

### Q8: Design a Lock That Supports Read and Write Modes

Several processes read a configuration file. One process writes to it. Multiple readers can read concurrently. Writers must have exclusive access. Design a distributed read-write lock.

**Approach**: Use a lock service that supports read-write lock semantics. In ZooKeeper, this is the Shared/Exclusive lock recipe. In Redis, implement two keys: a write lock key and a read count key.

```
Read lock acquisition:
- Acquire read lock: increment read count
- If write lock is held by another process, wait
- If write lock is not held, proceed with read access

Write lock acquisition:
- Set exclusive write lock key
- Wait until read count reaches zero
- Proceed with write access

Release read lock: decrement read count
Release write lock: delete exclusive write lock key
```

**Considerations**:
- Read-write locks are more complex than exclusive locks. The lock service must handle the state of multiple readers atomically.
- Starvation is possible: if readers continuously acquire the read lock, writers may never get the write lock. Solution: writers have priority once they are queued.
- The read count must be tracked accurately. Redis Lua scripts or ZooKeeper transactions ensure the read count is updated atomically with the read lock state.

### Q9: How Do You Handle Lock Prioritization?

A system has high-priority and low-priority operations. Both compete for the same lock. High-priority operations should never wait behind low-priority operations.

**Approach**: Priority-based lock with preemption. Each lock request includes a priority level. The lock service maintains a queue of waiting processes ordered by priority. When the lock is released, the highest-priority waiting process acquires it.

```python
def acquire_lock_with_priority(lock_name, owner, priority):
    # Add to priority queue in lock service
    lock_service.add_to_wait_queue(lock_name, owner, priority)
    
    # Check if we are the highest priority waiter
    while True:
        current_holder = lock_service.get_holder(lock_name)
        highest_waiter = lock_service.get_highest_priority_waiter(lock_name)
        
        if highest_waiter == owner and current_holder is None:
            # We are the highest priority and the lock is free
            lock_service.set_holder(lock_name, owner)
            return True
        
        if priority > current_holder.priority:
            # Preempt the current holder
            lock_service.preempt(lock_name, current_holder.owner)
            # Wait for current holder to release
            time.sleep(preemption_timeout)
            continue
        
        # Wait for the lock
        time.sleep(backoff_interval)
```

**Trade-offs**: Priority locks can cause starvation for low-priority operations. A common solution is priority aging — the priority of a waiting process increases over time until it reaches the maximum priority and must be served.

### Q10: Design a Lock for Idempotency Keys

An API receives requests with idempotency keys. If a duplicate request arrives, the system must detect it and return the previous response. Design a lock system that prevents two requests with the same idempotency key from being processed concurrently.

**Approach**: The idempotency key itself serves as the lock key. When a request arrives with idempotency key K, the system acquires a lock on K before processing. If the lock acquisition succeeds, this is the first request for K. The system processes the request and stores the response. If lock acquisition fails, another request for K is in progress. The system waits for the lock to be released (response is now stored) and returns the stored response.

```python
def handle_request_with_idempotency_key(request, idempotency_key):
    # Attempt to acquire lock on the idempotency key
    owner = acquire_lock(f"idempotency:{idempotency_key}", ttl=5000)
    
    if owner:
        # We are the first to process this key
        try:
            # Check if response already exists (edge case)
            cached = get_cached_response(idempotency_key)
            if cached:
                return cached
            
            # Process the request
            response = process_request(request)
            
            # Cache the response
            cache_response(idempotency_key, response)
            return response
        finally:
            release_lock(f"idempotency:{idempotency_key}", owner)
    else:
        # Another request is in progress
        # Wait for response to become available
        response = wait_for_response(idempotency_key, timeout=5000)
        if response:
            return response
        else:
            # Timeout — the other request may have failed
            return handle_retry(request, idempotency_key)
```

**Edge cases**: What if the first request acquires the lock but crashes before storing the response? The lock expires (TTL), the second request acquires the lock, processes the request, and stores the response. The TTL must be long enough to cover the entire request processing time but short enough to allow recovery from crashes.

### Q11: Design a Distributed Semaphore

A system has 10 database connection slots. Each operation needs one slot. Design a distributed semaphore that ensures no more than 10 operations use the database concurrently.

**Approach**: A distributed semaphore is a generalization of a distributed lock. Instead of one holder, the semaphore allows N holders.

Redis implementation: Use a Redis counter. Each acquisition decrements the counter. Each release increments it. If the counter is zero, acquisition fails.

```python
class DistributedSemaphore:
    def __init__(self, redis_client, name, max_count):
        self.redis = redis_client
        self.name = name
        self.max_count = max_count
        self.semaphore_key = f"semaphore:{name}"
        self.owner_key = f"semaphore:{name}:owners"
        
    def acquire(self, owner, timeout_ms=5000):
        deadline = time.monotonic() + timeout_ms / 1000
        
        while time.monotonic() < deadline:
            # Use Redis Lua script for atomic check-and-decrement
            result = self.redis.eval("""
                local count = redis.call("GET", KEYS[1])
                if not count then
                    count = ARGV[1]  -- max count
                end
                if tonumber(count) > 0 then
                    redis.call("DECR", KEYS[1])
                    redis.call("SADD", KEYS[2], ARGV[2])
                    return 1
                end
                return 0
            """, 2, self.semaphore_key, self.owner_key, self.max_count, owner)
            
            if result:
                return True
            
            time.sleep(0.05)  # 50ms backoff
        
        return False
    
    def release(self, owner):
        self.redis.eval("""
            redis.call("INCR", KEYS[1])
            redis.call("SREM", KEYS[2], ARGV[1])
        """, 2, self.semaphore_key, self.owner_key, owner)
```

**Considerations**: The semaphore must handle the case where a holder crashes without releasing. A TTL per acquisition prevents this. However, the TTL also causes premature release under normal operation if the operation runs long. The solution is to set the TTL generously (e.g., 60 seconds for a 10-second operation) and rely on lease renewal for longer operations.

### Q12: How Do You Implement a Distributed Barrier?

Multiple processes need to reach a synchronization point before any of them proceeds. Design a distributed barrier.

**Approach**: A distributed barrier tracks how many processes have arrived at the barrier. When all expected processes have arrived, the barrier is lifted and all waiting processes proceed.

ZooKeeper implementation: Each process creates an ephemeral znode under a barrier path. A watcher on the barrier path counts the children. When the count reaches the expected number, the barrier node is deleted and all watchers are notified.

```
Barrier path: /barrier/job_batch_17
Expected count: 5

Process 1: creates /barrier/job_batch_17/p1 (ephemeral)
Process 2: creates /barrier/job_batch_17/p2 (ephemeral)
Process 3: creates /barrier/job_batch_17/p3 (ephemeral)
Process 4: creates /barrier/job_batch_17/p4 (ephemeral)
Process 5: creates /barrier/job_batch_17/p5 (ephemeral)

When count reaches 5: barrier deleted, all processes proceed
```

**Failure handling**: If a process crashes before reaching the barrier, its ephemeral znode is deleted. The barrier count never reaches the expected number. A timeout mechanism detects this and either allows the remaining processes to proceed or aborts the barrier, depending on the application requirements.

### Q13: Design a Global Lock for Multi-Region Deployment

A service runs in three AWS regions (us-east-1, eu-west-1, ap-southeast-1). Some operations require global coordination. Design a lock that works across all regions with maximum availability.

**Approach**: Use a 7-node etcd cluster distributed across three regions. The leader region handles all lock writes. This approach accepts 150-300ms latency for cross-region lock acquisition.

Architecture decisions:
- 3 nodes in us-east-1 (leader region, lowest latency to most regions)
- 2 nodes in eu-west-1
- 2 nodes in ap-southeast-1
- All lock writes go through the leader region (Raft consensus)
- Read-only replicas or local caching for lock status queries

**Failover strategy**: If the leader region becomes unavailable, a new leader is elected from the remaining nodes within 15-30 seconds. During the failover, lock operations are unavailable. Applications must handle this window gracefully — retry lock acquisition with backoff, queue operations locally.

**Better approach**: Avoid global locks entirely. Partition resources by region. Each resource is assigned to a home region. Lock operations go to the home region. Cross-region operations (rare) use a two-phase commit pattern with per-region locks.

### Q14: How Would You Debug a Lock That Is Never Released?

A production incident shows a lock that has been held for 6 hours. The lock holder process appears healthy. Diagnose the problem.

**Diagnostic steps**:
1. Check the lock TTL. If the TTL is infinite (no expiry), this is the root cause. The lock must always have a TTL.
2. Check the lock holder's code path. Is there a code path where the lock is acquired but not released (missing try/finally)?
3. Check for nested lock acquisition. If the process acquires lock A, then tries to acquire lock B (which is held by another process), and the code path does not handle this, the process is deadlocked holding lock A.
4. Check the lease renewal thread. If the process has a renewal thread, verify that it is still running. A silent crash of the renewal thread would cause the process to continue operating without the lock.
5. Check network connectivity between the lock holder and the lock service. A one-way network issue could prevent the holder from releasing the lock.
6. Check for lock service bugs. Some lock implementations have bugs where a lock is never released from certain code paths.

**Recovery steps**:
1. Force-release the lock from the lock service (administrative override).
2. Verify that the lock holder stops operating (it should detect the lock loss).
3. If the lock holder does not stop, terminate it forcefully.
4. Investigate the root cause in post-mortem.

## More Phrase Bank Entries

**For explaining lock granularity**: "Lock granularity determines the scope of exclusion. A coarse lock protects a large resource but creates contention. A fine lock allows more concurrency but increases overhead and the risk of deadlock. The right granularity balances these trade-offs."

**For explaining why locks fail**: "Distributed locks fail in two ways: safety failures where two processes hold the same lock, and liveness failures where no process can acquire the lock. Consensus-based systems prevent safety failures but can have liveness failures during leader elections."

**For explaining the cost of locks**: "Every lock operation has a measurable cost. A Redis lock costs 1-3 milliseconds and a few bytes of memory. A ZooKeeper lock costs 5-15 milliseconds and a consensus round trip. At 10,000 locks per second, the lock service consumes significant infrastructure resources."

**For explaining lock vs lease**: "A lock grants exclusive access until explicitly released. A lease grants exclusive access for a fixed duration. In distributed systems, all locks are effectively leases because the TTL provides the crash recovery mechanism. The practical difference is whether the holder renews the lease or provides its own health signal."

**For explaining the split-brain problem**: "Split-brain occurs when a network partition creates two groups that each believe they are the leader. Distributed locks prevent split-brain by ensuring at most one process holds the lock. However, if the lock itself is not partition-tolerant, the lock system can create the split-brain it was supposed to prevent."

**For explaining lock ordering**: "Lock ordering is a deadlock prevention technique where all processes acquire locks in a fixed global order. If every process acquires lock A before lock B, a circular wait condition cannot occur. This is enforced by the code convention, not by the lock service."

**For explaining starvation**: "Lock starvation occurs when a process never acquires the lock because other processes always acquire it first. This is common in polling-based locks where a lucky process always wins the race. Wait queues eliminate starvation by serving processes in order."

**For explaining why consensus matters**: "Consensus algorithms like Raft and Zab ensure all nodes in a lock service agree on the lock state. Without consensus, a network partition can cause two nodes to independently believe they granted the lock. Consensus makes this impossible — at least one node must be involved in both decisions and would detect the conflict."

**For explaining the ZooKeeper sequential znode trick**: "The sequential znode is a brilliant mechanism. The sequence number is generated by ZooKeeper in a monotonically increasing order. When multiple processes create sequential znodes, the order of creation determines who gets the lock. The process with the lowest sequence number holds the lock."

**For explaining why fencing tokens need external storage**: "A fencing token is only useful if the protected resource checks it. The lock service generates the token, but the storage layer must validate the token on every write. If the storage layer does not check tokens, the fencing token provides no protection."

**For explaining lock sharding**: "Lock sharding splits the lock namespace across multiple lock service instances. Each instance handles a subset of lock keys. This increases throughput because each instance handles fewer operations. Sharding requires a consistent hash function to map lock keys to instances."

**For explaining the need for chaos testing**: "Distributed lock systems fail in ways that unit tests cannot reproduce. Network partitions, clock skew, and process crashes interact in unexpected ways. Chaos testing systematically introduces these failures and measures the lock service behavior. It is the only way to verify safety properties under realistic failure conditions."

**For explaining lock observability**: "A lock system without observability is dangerous. Every lock acquisition, release, timeout, and failure must be logged with timestamps and owner identifiers. A centralized dashboard showing lock contention, hold duration, and lease renewal success rate is essential for operating lock systems in production."

**For explaining when to use Redlock**: "Use Redlock when the cost of a rare lock failure is low, the throughput requirement is high, and the infrastructure already includes Redis. Redlock is not suitable for financial systems, resource allocation, or any system where duplicate operations have significant business impact."

**For explaining the difference between coordination and synchronization**: "Distributed locks provide synchronization — mutual exclusion across processes. Distributed coordination is broader, including leader election, configuration management, and service discovery. Locks are one tool in the coordination toolbox, not the only tool."

### Question: Design a Lock for a Multi-Tenant SaaS Platform

## Estimation Questions

### Estimation 1: Redis Lock Throughput

**Question**: How many lock operations per second can a single Redis instance handle?

**Approach**:
- Redis is single-threaded for command execution. A single instance can process approximately 100,000 simple commands per second.
- Each lock operation involves two Redis commands (SETNX for acquire + DEL or Lua for release). Each command is approximately 50 bytes.
- Redis benchmark results: SET commands at 100,000 ops/sec on modern hardware.
- Lock operations: acquire (SETNX) + release (DEL) = 2 commands per lock cycle.
- Maximum throughput: 50,000 lock cycles per second for a single Redis instance.
- With pipelining, this can reach 80,000-90,000 lock cycles per second.

**Adjustments for contention**:
- Under high contention, most SETNX calls fail (lock is held). Redis still processes these commands, so throughput in terms of commands handled is the same. However, effective lock grants are much lower.
- At 90% contention, only 10% of acquire attempts succeed. Effective throughput is 5,000 successful acquisitions per second.

**Key insight**: Redis does well at high command throughput but poorly at high lock acquisition throughput under contention. For contended locks, the effective throughput is much lower than the raw command throughput.

### Estimation 2: ZooKeeper Lock Latency

**Question**: What is the expected latency for a ZooKeeper lock acquisition with a 3-node ensemble in the same data center?

**Approach**:
- ZooKeeper uses the Zab protocol. A write request goes to the leader, which proposes to followers, and commits when a majority acknowledges.
- Network round trip within a data center: approximately 0.5ms between any two nodes.
- Leader to follower proposal + acknowledgment: 1 round trip = 1ms.
- Client to leader + leader to client: 1 round trip = 0.5ms.
- Total ZooKeeper write latency: approximately 1.5-3ms for the znode creation.
- Additional overhead: serialization, deserialization, queueing, disk sync.
- Real-world P50 latency: 5-10ms for ZooKeeper lock acquisition.
- P99 latency: 20-50ms under moderate load.

**Comparison with etcd**:
- etcd uses Raft, which has similar latency characteristics to Zab.
- etcd P50: 3-8ms for lock acquisition.
- etcd typically has lower latency because it uses Go's goroutine model and has less serialization overhead.

**Key insight**: The consensus round trip dominates lock latency. Within a data center, this is 2-5ms. Cross-data center, this is 50-200ms depending on distance.

### Estimation 3: Cost of Lock Operations at Scale

**Question**: What is the cost of operating a distributed lock service for a system processing 10,000 locks per second?

**Approach**:
- Infrastructure cost: 3-node Redis cluster (r6g.large) at approximately $0.20/hour.
- Monthly infrastructure cost: 3 nodes x 730 hours x $0.20 = $438/month.
- At 10,000 locks/sec, monthly operations: 10,000 x 86,400 x 30 = 25.9 billion operations.
- Cost per operation: $438 / 25.9 billion = $0.0000000169 per operation.
- This is essentially zero per operation.

**Hidden costs**:
- Engineering time to implement and maintain the lock system: $50,000-200,000/year.
- Opportunity cost of lock contention: each millisecond a thread waits costs compute time.
- At 10,000 locks/sec with 5ms average contention wait per lock: 50 seconds of CPU time per second (50 threads blocked).
- Need 50 extra vCPUs to handle the blocked threads: additional $500/month.

**Key insight**: The infrastructure cost of locks is negligible. The hidden costs — engineering time, contention overhead, and failure recovery — dominate the total cost of ownership.

## Debugging Scenarios

### Scenario 1: Intermittent Lock Acquisition Failure

**Symptom**: Lock acquisition fails intermittently for 1-2 seconds, then recovers. The pattern repeats every 30 minutes. The lock service is a 3-node Redis cluster with Sentinel.

**Diagnosis steps**:
1. Check Redis Sentinel logs for the failure period. Sentinel performs health checks every 10 seconds. If a node is slow to respond, Sentinel may mark it as down, triggering a failover.
2. Check the Redis slow log for the failure period. A slow query (e.g., a large Lua script or a blocking operation) could delay lock operations.
3. Check the lock client logs for the exact error message. A connection timeout suggests the client could not reach Redis. An execution error suggests the Redis command failed.
4. Check the network monitoring for the failure period. Packet loss or latency spikes between the client and Redis would cause intermittent failures.

**Likely root cause**: Redis has a background save (RDB or AOF rewrite) configured every 30 minutes. During the save, Redis forks and the fork process can cause a latency spike. The fork takes time proportional to the memory size. During this time, Redis does not respond to commands quickly, causing timeouts.

**Fix**: Disable automatic saves or schedule them during low-traffic periods. Use a larger instance with more memory bandwidth to reduce fork time. Consider using Redis with disabled persistence and rely on replication for durability.

### Scenario 2: Lock Hold Duration Spikes

**Symptom**: Monitoring shows that lock hold duration has increased from 50ms (P50) to 2 seconds (P50) over the past week. The throughput has dropped by 40%.

**Diagnosis steps**:
1. Check if the critical section code has changed. A recent deployment may have introduced a slow operation within the critical section.
2. Check if the protected resource is slower. If the lock protects a database write, check the database performance for the same period.
3. Check if contention has increased. If more processes are contending for the same lock, each process spends more time waiting for the lock, increasing the effective hold duration.
4. Check if the lock TTL has changed. An increased TTL causes processes to wait longer for lock expiry.
5. Check for a slow background operation. If a maintenance task (backup, compaction, garbage collection) runs on the lock holder, it can slow down the critical section.

**Likely root cause**: A new service was deployed that also uses the same lock. The number of contenders increased from 5 to 50. Each process waits longer for its turn. The critical section time is the same, but the queue wait time increased.

**Fix**: Shard the lock namespace. Use more granular locks so each lock has fewer contenders. Alternatively, reduce the critical section time by batching operations or using a different access pattern.

### Scenario 3: Lock Duplication Under Load

**Symptom**: Under peak load (3x normal), the system experiences rare cases where two processes hold the same lock simultaneously. Lock service is Redis with master-replica replication.

**Diagnosis steps**:
1. Reproduce the failure under controlled load. Use a test environment that mirrors production.
2. Check the Redis replication lag during peak load. If the replica lag exceeds the lock TTL, a failover can cause the new master to lack the lock key.
3. Check the Redis failover configuration. Does Sentinel require the replica to be up to date before promoting it to master?
4. Check the lock acquisition code. Is the SETNX with PX flags sent atomically? A non-atomic check-then-set could allow two processes to acquire the same lock.

**Likely root cause**: Under peak load, Redis replication lag reaches 1-2 seconds. The lock TTL is 5 seconds. A Redis master failure during peak load causes Sentinel to promote a replica that does not have the lock key. The new master has no lock key, so a new process can acquire the lock while the original holder still operates.

**Fix**: Increase the Redis min-slaves-max-lag configuration to prevent promotion of lagging replicas. Alternatively, use Redis with WAIT command to ensure the write is replicated before returning success. For critical locks, migrate to ZooKeeper or etcd.

## Code Review Scenarios

### Code Review 1: Redis Lock Acquisition Bug

```python
# Review this code for bugs
def acquire_lock_redis(redis_client, lock_name, ttl_ms):
    # Check if lock exists
    exists = redis_client.exists(lock_name)
    if exists:
        return None
    
    # Generate owner ID
    owner_id = str(uuid.uuid4())
    
    # Set the lock
    redis_client.set(lock_name, owner_id, px=ttl_ms)
    return owner_id
```

**Bugs to identify**:
1. **Race condition**: The `exists` check and `set` are separate commands. Between the check and the set, another process can create the lock. Two processes can both pass the `exists` check and both create the lock. Fix: use `SET lock_name owner_id NX PX ttl_ms` as a single atomic command.
2. **Missing NX flag**: The `set` command does not use the `NX` flag. Even if the check passes, the set will overwrite an existing lock without the NX flag. Fix: add `nx=True` to the set call.
3. **Owner ID generation after check**: The owner ID is generated after the exists check. If two processes pass the check simultaneously, they both generate different owner IDs and both create the lock. This is not the primary bug (the race condition is) but it is a code organization issue.

**Fixed version**:
```python
def acquire_lock_redis(redis_client, lock_name, ttl_ms):
    owner_id = str(uuid.uuid4())
    result = redis_client.set(lock_name, owner_id, nx=True, px=ttl_ms)
    if result:
        return owner_id
    return None
```

### Code Review 2: Lock Release Bug

```python
# Review this code for bugs
def release_lock(redis_client, lock_name, owner_id):
    # Get the current lock owner
    current_owner = redis_client.get(lock_name)
    
    # Only release if we still own the lock
    if current_owner == owner_id:
        redis_client.delete(lock_name)
        return True
    return False
```

**Bugs to identify**:
1. **Race condition**: The `get` and `delete` are separate commands. Between the get and the delete, the lock could expire and another process could acquire it. Then our delete would remove another process's lock. Fix: use a Lua script to atomically check the owner and delete.
2. **Missing encoding handling**: The `get` returns bytes in Python Redis client. The comparison `current_owner == owner_id` may fail if one is bytes and the other is string. Fix: decode the bytes or use byte comparison.
3. **No error handling**: If the Redis connection fails between get and delete, the lock is not released. Fix: add timeout and retry logic.

**Fixed version**:
```python
def release_lock(redis_client, lock_name, owner_id):
    lua_script = """
    if redis.call("GET", KEYS[1]) == ARGV[1] then
        return redis.call("DEL", KEYS[1])
    else
        return 0
    end
    """
    result = redis_client.eval(lua_script, 1, lock_name, owner_id)
    return result == 1
```


**Bugs to identify**:
1. **No owner check**: The renewal uses `EXPIRE` without checking if the lock is still owned by this process. If the lock was released and re-acquired by another process, this renewal extends the new owner's TTL. Fix: use a Lua script that checks the owner before expiring.

3. **No health check**: There is no way to verify that the renewal thread is running. If the thread crashes, the lock expires silently. Fix: add a heartbeat counter that external monitoring can check.
4. **Race condition on stop**: The `running` flag is not volatile/synchronized. Another thread may call `stop` but the renewal thread may not see the change. Fix: use a threading.Event instead of a boolean.

## System Design Interview Scripts

### Script 1: Opening the Distributed Lock Interview

The following script demonstrates how to open the distributed lock portion of a system design interview.

**Interviewer**: "Design a distributed lock service for a job scheduler that runs on 100 servers."

**Candidate opening response**:
"Before I design the lock service, I want to understand the requirements. First, what kind of jobs are being scheduled? Are these idempotent jobs where a duplicate run is safe, or are they operations where duplicates cause data corruption? Second, what is the expected frequency of lock operations — how many jobs per second? Third, what is the maximum acceptable latency for lock acquisition?"

**Continued discussion**:
"If the jobs are idempotent and throughput is under 1,000 per second, Redis with SETNX and a TTL is the simplest solution. Each job ID serves as the lock key. The worker acquires the lock before processing the job and releases it after completion. The TTL protects against worker crashes."

"If the jobs are non-idempotent and require strict exactly-once processing, ZooKeeper or etcd with fencing tokens is necessary. The lock acquisition provides a monotonically increasing token that the job processor must present when writing results."

"If the throughput exceeds 5,000 per second, I would shard the lock namespace. For Redis, shard by job ID hash across multiple Redis instances. For ZooKeeper, multiple lock paths under a shared namespace."

### Script 2: Handling the Unexpected Question

**Interviewer**: "What happens if the network is partitioned while a process holds your lock?"

**Candidate response**:
"This is the most important failure mode for distributed locks. Let me reason through it carefully."

"The lock holder is on side A of the partition. The lock service is on side B. The lock holder cannot reach the lock service to renew its lease. The lease expires after the TTL. The lock is released on the lock service side."

"Now a process on side B can acquire the lock. This process starts working on the protected resource, which may also be on side B."

"The original lock holder on side A is still operating. It believes it holds the lock because it received a successful lock grant before the partition. It may continue writing to the resource."

"The critical question is: can the original holder on side A write to the resource? If the resource is on side B (behind the partition), it cannot — the partition prevents it. So for resources collocated with the lock service, this scenario is safe."

"If the resource is accessible from both sides of the partition, we need fencing tokens. The lock service generates a monotonically increasing token with each lock grant. The resource checks the token on every write. When side B acquires the lock, it gets a higher token. Side A's writes have a lower token and are rejected."

### Script 3: Deep Dive into Fencing Tokens

**Interviewer**: "How would you implement fencing tokens for a lock service that uses Redis?"

**Candidate response**:
"Fencing tokens require a monotonically increasing counter. Redis does not natively provide this with strong consistency. I would implement it in one of three ways."

"Option 1: Use a dedicated Redis key as a counter. Each lock grant uses INCR to atomically increment the counter. The resulting value is the fencing token. This works as long as the Redis instance does not fail. If the Redis instance fails and recovers from a snapshot that has an older counter value, the token sequence would reset. This is a safety risk."

"Option 2: Use Redis with AOF persistence and fsync=always. This ensures the counter is durable on every write. If the Redis instance restarts, it replays the AOF log and the counter value is restored. This provides better safety than Option 1 but has a performance cost."

"Option 3: Use an external token generator. A separate service generates monotonically increasing tokens using a file-backed counter or a consensus-based approach. The lock service acquires a token from the external generator for each lock grant. This decouples token generation from lock management and provides the strongest guarantees."

"For most applications, Option 2 is the right balance. The performance cost of AOF fsync=always is approximately 20-30%, which is acceptable for most lock workloads. For truly critical systems, Option 3 with a consensus-based token generator is the correct choice."

### Script 4: When the Candidate Needs Guidance

**Interviewer** (after a candidate proposes a flawed lock design): "How would your design handle a process crash?"

**Candidate**: "The process releases the lock when it recovers from the crash."

**Interviewer guidance**: "This is an important consideration. If a process crashes, it cannot execute any code during the crash. The lock release must happen without the process's cooperation. What mechanism would provide this?"

**Expected revised response**: "You are pointing out that the lock must be released automatically when the process crashes. I need a TTL or lease mechanism. When the process acquires the lock, it also sets a TTL. If the process crashes, the TTL expires and the lock is released. If the process is healthy, it periodically extends the TTL. This way, lock release does not require the holder to be alive."

**Discussion**: The candidate should also mention that the TTL must be set atomically with the lock acquisition (using the same command) to avoid the race condition where the process crashes between acquiring the lock and setting the TTL.
