## Tier 1: Foundational

### Challenge 1.1: Redis SETNX Lock Implementation

Implement a basic distributed lock using Redis SETNX and a Lua script for safe release. The lock must support a configurable TTL and owner-specific release.

**Requirements**:
- Implement lock acquisition using SET with NX and PX flags
- Implement safe lock release using a Lua script that checks the owner
- Handle the case where the lock is already held by another process
- The owner identifier must be unique per process

**Model answer**: The implementation uses `SET resource owner NX PX ttl` for acquisition. The `NX` flag ensures the key is created only if it does not exist. The `PX` flag sets an expiry in milliseconds. For release, a Lua script atomically checks that the current value matches the owner before deleting. This prevents a process from releasing another process's lock. The owner identifier combines hostname, process ID, and a UUID to ensure uniqueness across all servers.

**What to watch for**: The candidate must use the Lua script for release — checking the owner and deleting in separate commands creates a race condition. The TTL must be set at acquisition time, not separately. The owner identifier must be unique across all processes in the system.

### Challenge 1.2: ZooKeeper Lock Recipe

Implement a distributed lock using ZooKeeper's ephemeral sequential znode pattern. The lock must handle session expiry and process crashes.

**Requirements**:
- Create an ephemeral sequential znode under a lock path
- Determine lock ownership by checking the lowest sequence number
- Watch the preceding znode for deletion notifications
- Handle session expiry correctly (the lock is automatically released)

**Model answer**: The ZooKeeper lock recipe creates an ephemeral sequential znode at `/locks/resource/lock-`. The process gets all children of the lock path and checks if its znode has the lowest sequence number. If yes, it holds the lock. If no, it sets a watch on the preceding znode (sequence number one lower) and waits. When the watch fires (the preceding znode was deleted), the process re-lists children and checks again. Session expiry automatically deletes the ephemeral znode.

**What to watch for**: The candidate must handle the watch correctly — watching the preceding znode, not the lock path root. Watching the root creates a thundering herd on every release. The candidate must also handle the race condition where the preceding znode is deleted between listing children and setting the watch.

### Challenge 1.3: Lock Contention Analysis

Given 50 worker processes and a single lock protecting a resource, calculate the throughput of the system under the following conditions:

- Each worker requires the lock for 100ms to process one item
- Lock acquisition latency: 2ms
- Lock release latency: 1ms

**Model answer**: In the best case (no contention), each worker acquires the lock, processes for 100ms, and releases. The total time per worker per item is 103ms. With 50 workers and a single lock, the workers must serialize. Each worker completes one item every 103ms, but only one worker can process at a time. The throughput is 1 item per 103ms, or approximately 9.7 items per second. The 50 workers share this throughput — each worker processes approximately 0.19 items per second.

**What to watch for**: The candidate must recognize that a single lock serializes access regardless of the number of workers. Adding more workers does not increase throughput for a single lock — it increases contention, which decreases throughput due to lock acquisition overhead.

## Tier 2: Intermediate

### Challenge 2.1: Lease Renewal Implementation

Extend the basic Redis lock to support lease renewal. The lock should automatically extend its TTL while the holder is actively working. Implement a background thread for lease renewal.

**Requirements**:
- After acquiring the lock, start a background thread that renews the lease every TTL divided by 3 milliseconds
- If a renewal fails, retry 3 times with 100ms backoff
- If all retries fail, invoke a callback to notify the holder that the lock is lost
- The holder must cease all operations on the protected resource after lock loss notification

**Model answer**: The lease renewal thread runs every TTL divided by 3 milliseconds. Each renewal executes a Lua script: `if redis.call("GET", KEYS[1]) == ARGV[1] then return redis.call("PEXPIRE", KEYS[1], ARGV[2]) else return 0 end`. If the script returns 0 (owner changed), the thread stops and notifies the holder. The notification sets a flag that the holder checks before each write operation. The renewal interval must include network latency — if the TTL is 30 seconds and renewal takes 5ms, the effective renewal budget is 29,995ms, which is ample.

**What to watch for**: The candidate must use a Lua script for renewal (not separate GET and PEXPIRE calls). The renewal must check the owner before extending. The lock loss notification must be synchronous — the holder must stop operating before another renewal attempt. The thread must be a daemon thread that stops when the holder releases the lock.

### Challenge 2.2: Fencing Token Implementation

Implement a fencing token generator for a lock service. The generator must produce monotonically increasing tokens that survive leader failover.

**Requirements**:
- Each lock grant receives a unique, monotonically increasing integer
- The token sequence must persist across leader failovers
- The token generator must handle concurrent lock grants
- The token should be a 64-bit integer

**Model answer**: Store the current token value in the consensus log (Raft or Zab). Each lock grant appends an increment operation to the log. The token value is the new current value after the increment. For example, if the log contains entries for token increments 1040, 1041, 1042, the next grant gets token 1043. The Raft log ensures that even after leader failover, the new leader replays all committed log entries and continues from the last value.

**What to watch for**: The candidate must ensure the token is part of the consensus log, not stored in memory only. An in-memory counter resets on leader failover. Using the wall clock as a token source is not safe (clock drift can cause non-monotonic values). Using a UUID is not safe (no ordering guarantee).

Design a system that detects deadlocks in a distributed lock environment. The system must identify circular wait conditions.

**Requirements**:
- Build a wait-for graph from lock requests
- Detect cycles in the graph
- Resolve deadlocks by aborting one of the involved processes
- Minimize the cost of the aborted transaction

**Model answer**: The deadlock detector maintains a wait-for graph. Each lock request that cannot be granted (because the lock is held by another process) adds an edge from the requesting process to the holding process. A cycle detection algorithm (DFS-based) runs periodically on the graph. When a cycle is detected, the detector selects a victim process to abort. The victim selection minimizes the cost — typically the process with the fewest locks or the shortest elapsed execution time. The aborted process releases all its locks, breaking the cycle.

**What to watch for**: The candidate must distinguish between distributed deadlock detection (wait-for graph) and distributed deadlock prevention (lock ordering). The graph must account for multiple lock resources, not just a single lock. The victim selection must consider transaction cost, not just lock count.

## Tier 3: Advanced

### Challenge 3.1: Design a Lock Service for a Global Payment System

Design a distributed lock service for a global payment system that processes $1 billion in transactions per day. The system operates in 5 regions across 3 continents. A lock failure can cause duplicate payments, which cost $50 per incident in gateway fees plus reconciliation overhead.

**Requirements**:
- Zero tolerance for lock duplication (safety is absolute)
- Maximum availability — the lock service must survive a full region failure
- Maximum lock acquisition latency of 200ms (P99)
- Support 5,000 concurrent locks
- Each lock is held for 50-500ms

**Model answer**:

Use a 7-node etcd cluster with a per-region topology. Place 3 nodes in us-east-1 (primary), 2 in eu-west-1, and 2 in ap-southeast-1. The primary region handles all lock writes. Read-only replicas in other regions serve lock status queries but all acquisition goes through the primary.

Fencing tokens are mandatory. Each lock grant includes a monotonically increasing token. The payment database stores the last-seen token per payment batch and rejects operations with stale tokens.

TTL is set to 2 seconds — 4x the expected maximum operation time of 500ms. Lease renewal every 500ms.

For disaster recovery, the lock service is backed by an asynchronous replication stream to a backup cluster. If the primary region fails entirely, operations shift to eu-west-1 with a 60-second cutover, accepting the risk of lost lock state during the 60-second window.

**What to watch for**: The candidate must prioritize safety over liveness for a payment system. The TTL must be generous enough to prevent premature expiry but short enough to ensure quick recovery after crashes. Fencing tokens are non-negotiable. The cross-region topology must balance availability (multi-region) with latency (single-leader).

### Challenge 3.2: Design a Lock-Free Alternative

For a specific use case, design a lock-free coordination mechanism that replaces the distributed lock. The use case: 100 workers processing items from a shared work queue. Each item must be processed at least once. No item should be processed twice.

**Requirements**:
- No distributed lock service
- The workers use a shared database for state
- At-least-once processing guaranteed
- Best-effort exactly-once processing

**Model answer**: Replace the distributed lock with a database transaction that atomically marks the job as claimed. Each worker begins a transaction, reads a job with status `pending`, updates the status to `claimed` with the worker's identifier and a claim timestamp, and commits. If the update affects zero rows (another worker claimed it first), the transaction aborts and the worker retries with a different job.

For exactly-once processing, each worker writes the result with the job ID as the primary key. A duplicate write (same job ID from a different worker) fails because the primary key already exists. This provides at-least-once processing by design and exactly-once processing through idempotent writes.

**What to watch for**: The candidate must show that a distributed lock is not always the answer. The database transaction provides the same safety as a distributed lock without the additional infrastructure. The candidate must also handle the edge case where a worker claims a job but crashes before writing the result — a separate cleanup process must detect and re-queue stale claimed jobs.

### Challenge 3.3: Predict and Prevent a Lock Failure at 10x Scale

A lock service that handles 1,000 locks per second needs to scale to 10,000 locks per second. Predict the failure modes and design the scaled system.

**Requirements**:
- The current system uses a single 3-node etcd cluster
- All locks use a 30-second TTL
- Average hold time: 5 seconds
- Peak load is 3x average

**Model answer**:

**Predicted failures at 10x scale**:
1. The etcd cluster reaches its write throughput limit (5,000 ops/sec max). At 10,000 ops/sec with a 3x peak (30,000 ops/sec), the cluster falls behind. Queueing delays cause lock acquisition timeouts.
2. The lease manager at 10,000 concurrent leases with 100ms ticks processes 100,000 lease checks per second. CPU becomes the bottleneck.
3. Network saturation — each lock operation is a write to etcd (requires consensus). 10,000 ops/sec means 30,000 network messages per second (leader to 2 followers).

**Design for 10x scale**:
1. Shard the lock namespace across 5 etcd clusters. Resource keys hashed to one of 5 clusters. Each cluster handles 2,000 ops/sec (within its 5,000 ops/sec limit).
2. Reduce the TTL to 10 seconds for operations that complete in under 1 second. Shorter TTLs reduce the lease manager load because expired locks are cleaned up faster.
3. Increase the lease manager tick interval to 200ms and process expirations in batches.
4. Add a Redis caching layer for read-only lock checks. Workers check a local cache first (the lock is probably held if it was held 100ms ago). Only contact etcd for acquisition and release.

**What to watch for**: The candidate must consider the scaling limits of the consensus protocol, not just the application. Sharding is the correct solution for consensus-bound systems. Caching helps for read-heavy workloads but does not help for write-heavy workloads (all lock acquisitions are writes in consensus systems).

### Challenge 3.4: Redlock Critique and Defense

A team is using Redlock with 5 Redis nodes for a job scheduling system. The system occasionally processes the same job twice. Analyze the root cause and propose a fix.

**Requirements**:
- Understand the Redlock algorithm in detail
- Identify the specific failure mode causing duplicate processing
- Propose a fix that does not require replacing Redis

**Model answer**: The root cause is the lack of fencing tokens in Redlock. When a worker experiences a GC pause or network delay, its lock TTL expires and another worker acquires the lock. The original worker resumes and continues processing, duplicating work.

The fix: Add fencing tokens to the Redlock implementation. Maintain an atomic counter in one of the Redis nodes (or a separate Redis instance). Each lock grant atomically increments the counter and includes the resulting token. The job queue stores the token in the job metadata. Before processing a job, the worker checks if its token is the highest seen for that job. If not, it aborts.

This fix does not replace Redis but adds a monotonic counter for token generation. The counter must survive failures — use Redis RDB or AOF persistence with fsync to ensure the counter value is not lost on restart.

**Model answer**: The automatic TTL extension uses a background thread that runs on a configurable interval. Each extension executes a Lua script that atomically checks ownership and extends the TTL. The extension interval is typically one-third of the TTL to ensure there is time for retries if an extension fails. The thread maintains a retry counter — after three consecutive failures, the thread invokes a callback that signals the holder to stop operating. The thread must be a daemon thread that terminates when the holder releases the lock.

```python
class AutoExtendingLock:
    def __init__(self, redis_client, lock_name, owner_id, ttl_ms=30000):
        self.redis = redis_client
        self.lock_name = lock_name
        self.owner_id = owner_id
        self.ttl_ms = ttl_ms
        self.renewal_interval = ttl_ms / 3
        self._renewal_thread = None
        self._stop_event = threading.Event()
        self._lock_lost = False
        
    def acquire(self):
        result = self.redis.set(self.lock_name, self.owner_id, 
                                nx=True, px=self.ttl_ms)
        if result:
            self._start_renewal()
            return True
        return False
        
    def release(self):
        self._stop_renewal()
        lua = """
        if redis.call("GET", KEYS[1]) == ARGV[1] then
            return redis.call("DEL", KEYS[1])
        end
        return 0
        """
        self.redis.eval(lua, 1, self.lock_name, self.owner_id)
        
    def _start_renewal(self):
        self._stop_event.clear()
        self._lock_lost = False
        self._renewal_thread = threading.Thread(
            target=self._renew_loop, daemon=True)
        self._renewal_thread.start()
        
    def _renew_loop(self):
        retries = 0
        lua = """
        if redis.call("GET", KEYS[1]) == ARGV[1] then
            return redis.call("PEXPIRE", KEYS[1], ARGV[2])
        end
        return 0
        """
        while not self._stop_event.wait(self.renewal_interval / 1000):
            try:
                result = self.redis.eval(
                    lua, 1, self.lock_name, self.owner_id, self.ttl_ms)
                if result == 1:
                    retries = 0
                else:
                    retries += 1
            except Exception:
                retries += 1
                
            if retries >= 3:
                self._lock_lost = True
                break
```

### Challenge 1.5: Handle Lock Release Failure

Extend the lock implementation to gracefully handle failures during the lock release process. The lock release may fail because of network issues, lock service unavailability, or race conditions with lock expiry.

**Requirements**:
- Implement a robust release mechanism with retry logic
- Handle the case where the lock expired between the check and release
- Handle the case where the lock was acquired by another process between the check and release
- Log appropriate diagnostics for each failure mode
- Implement a force-release mechanism for administrative use

**Model answer**: Lock release failures must be handled carefully. A failed release does not automatically mean the lock is still held — the lock may have expired. The release logic must distinguish between these cases:

1. **Lock owned and released successfully**: The Lua script returns 1. Success.
2. **Lock owned but Redis connection failed during release**: The holder retries with exponential backoff. If the retry succeeds, the release is confirmed.
3. **Lock expired naturally**: The Lua script reads the key as nil. The holder does not need to retry — the lock is already released.
4. **Lock acquired by another process**: The Lua script reads the key with a different owner value. The holder does nothing — the lock belongs to another process now.

```python
def robust_release(redis_client, lock_name, owner_id, max_retries=3):
    lua_script = """
    local value = redis.call("GET", KEYS[1])
    if not value then
        return 2     -- Lock already expired
    end
    if value == ARGV[1] then
        redis.call("DEL", KEYS[1])
        return 1     -- Successfully released
    else
        return 0     -- Lock owned by another process
    end
    """
    
    for attempt in range(max_retries):
        try:
            result = redis_client.eval(lua_script, 1, lock_name, owner_id)
            if result == 1:
                log.info("Lock released successfully", lock=lock_name)
                return True
            elif result == 2:
                log.warning("Lock already expired", lock=lock_name)
                return True  # Lock is released from our perspective
            else:
                log.warning("Lock owned by another process", 
                           lock=lock_name, expected_owner=owner_id)
                return False
        except redis.ConnectionError as e:
            log.error("Redis connection failed during release", 
                     lock=lock_name, attempt=attempt, error=str(e))
            time.sleep(0.1 * (2 ** attempt))  # Exponential backoff
    
    log.critical("Failed to release lock after all retries", 
                lock=lock_name)
    return False
```

For administrative force-release, an authenticated user with elevated permissions can execute a direct DELETE on the lock key. This is used only when the lock holder is known to be dead (process crashed, node failed) and the TTL has not yet expired.

**What to watch for**: The candidate must not attempt to release a lock that belongs to another process. The candidate must distinguish between lock expiry (no key) and lock ownership change (key with different owner). The candidate should log the release outcome for debugging and monitoring.

**Model answer**: Use Redis for lock management. Each row maps to a Redis lock key with the format `dblock:{table_name}:{row_id}`. The lock is acquired using SETNX with a 5-second TTL. The TTL is generous because row operations should complete within 500ms. A renewal thread extends the lock if the operation takes longer than expected.

For operations that involve multiple rows, acquire all locks in a consistent order (by row ID, ascending). This prevents circular wait conditions and eliminates the possibility of deadlock. If a lock cannot be acquired within a timeout, release all previously acquired locks and retry.

**What to watch for**: The candidate must address the deadlock prevention strategy (lock ordering). The candidate must handle the edge case where a row lock is acquired but the row does not exist in the database. ### Challenge 2.5: Implement Fair Lock (FIFO Ordering) with Redis

Implement a distributed lock that guarantees FIFO ordering — processes acquire the lock in the order they requested it. The lock must prevent starvation, where a process never acquires the lock because other processes always win the race.

**Requirements**:
- Processes acquire the lock in FIFO order (first-come, first-served)
- No polling — processes should not spin-wait for the lock
- Handle process crashes: if a waiting process crashes, the next process in line should acquire the lock
- The implementation must use Redis as the backend
- Support a configurable timeout: a process gives up waiting after N seconds

**Model answer**: A fair distributed lock uses a combination of a Redis list and a lock key. When a process wants the lock, it pushes its owner identifier onto a Redis list (the waiting queue). It then checks if it is at the front of the list and the lock is free. If so, it pops from the list and acquires the lock. If not, it blocks with a timeout, waiting for a notification that the lock is available.

```python
class FairRedisLock:
    def __init__(self, redis_client, lock_name, owner_id, timeout_ms=10000):
        self.redis = redis_client
        self.lock_name = lock_name
        self.owner_id = owner_id
        self.timeout_ms = timeout_ms
        self.queue_key = f"fairlock:{lock_name}:queue"
        self.lock_key = f"fairlock:{lock_name}:holder"
        
    def acquire(self):
        deadline = time.monotonic() + (self.timeout_ms / 1000)
        
        # Add ourselves to the waiting queue
        queue_id = self.redis.rpush(self.queue_key, self.owner_id)
        
        try:
            while time.monotonic() < deadline:
                # Check if we are at the front of the queue
                front = self.redis.lindex(self.queue_key, 0)
                if front != self.owner_id:
                    # Someone else is ahead of us — wait
                    time.sleep(0.05)
                    continue
                
                # We are at the front. Try to acquire the lock.
                acquired = self.redis.set(
                    self.lock_key, self.owner_id, 
                    nx=True, px=self.timeout_ms
                )
                if acquired:
                    # Remove ourselves from the queue
                    self.redis.lpop(self.queue_key)
                    return True
                
                # Lock is held by someone else — wait and retry
                time.sleep(0.05)
            
            # Timeout expired
            return False
            
        finally:
            # If acquisition failed, remove ourselves from the queue
            # Use a Lua script for atomic removal
            lua = """
            local queue = KEYS[1]
            local value = ARGV[1]
            local list = redis.call("LRANGE", queue, 0, -1)
            for i, v in ipairs(list) do
                if v == value then
                    redis.call("LSET", queue, i-1, "__REMOVED__")
                    redis.call("LREM", queue, 0, "__REMOVED__")
                    break
                end
            end
            return 1
            """
            self.redis.eval(lua, 1, self.queue_key, self.owner_id)
```

**What to watch for**: The candidate must handle the crash of a waiting process. If a process adds itself to the queue but crashes before acquiring the lock, the queue would be blocked permanently. The solution is to use a TTL for queue entries or a cleanup mechanism that periodically removes stale entries from the queue. The candidate must also handle the race condition where a process is at the front of the queue but the lock is not yet released — the process must wait and retry, not give up immediately.

**Model answer**: Use Raft consensus with a 5-node cluster. Raft requires a majority of nodes (3 out of 5) to commit a log entry. With 2 node failures, 3 nodes remain, which is a majority. Lock operations continue normally.

The key design decisions are:

1. **Leader election**: If the leader is one of the failed nodes, the remaining 3 nodes hold a new election. The election succeeds because 3 nodes are available (same as a majority). The election completes within the leader election timeout (150-300ms).

2. **Log replication**: The new leader replicates lock operations to the 2 remaining followers. The 3-node majority commits each operation. When the failed nodes rejoin, they catch up on the log entries they missed.

3. **Read-only mode**: The remaining nodes can serve read-only lock state queries immediately. Write operations (lock acquisition, release) require the full consensus path.

4. **Recovery**: When a failed node rejoins, it enters a catch-up phase. It downloads all log entries it missed from the leader. During catch-up, it does not participate in voting or log commitment. Once caught up, it resumes normal operation.

**What to watch for**: The candidate must explain the majority concept clearly. With 5 nodes, any 3 nodes form a majority. With 2 failures, 3 remain — operations continue. With 3 failures (or a network partition that splits 2-3), the side with 2 nodes cannot form a majority and must reject lock operations. This is correct behavior — safety is preserved at the cost of availability.

**Model answer**: The priority-based lock maintains three pieces of state per lock: the current holder, the holder's priority, and the preemption state. When a high-priority request arrives, it checks the current holder's priority. If the requester's priority is higher, it marks the lock for preemption.

The preempted holder receives a notification through a subscription channel. The holder has a grace period to complete and release the lock. If the holder does not release within the grace period, the lock service force-releases the lock and grants it to the high-priority requester.

```python
class PriorityLock:
    def __init__(self, redis_client, lock_name):
        self.redis = redis_client
        self.lock_name = lock_name
        self.holder_key = f"priorlock:{lock_name}:holder"
        self.priority_key = f"priorlock:{lock_name}:priority"
        self.grace_key = f"priorlock:{lock_name}:grace_deadline"
        self.channel = f"priorlock:{lock_name}:notify"
        
    def acquire(self, owner_id, priority, ttl_ms=30000):
        # Lua script for priority-based acquisition
        lua = """
        local holder = redis.call("GET", KEYS[1])
        
        if not holder then
            -- Lock is free, acquire it
            redis.call("SET", KEYS[1], ARGV[1])
            redis.call("SET", KEYS[2], ARGV[2])
            redis.call("PEXPIRE", KEYS[1], ARGV[3])
            redis.call("PEXPIRE", KEYS[2], ARGV[3])
            return "ACQUIRED"
        end
        
        local current_priority = tonumber(redis.call("GET", KEYS[2]))
        local request_priority = tonumber(ARGV[2])
        
        if request_priority > current_priority then
            -- High priority requester — set preemption
            redis.call("SET", KEYS[3], ARGV[4])  -- grace deadline
            redis.call("PUBLISH", KEYS[4], ARGV[1])
            return "PREEMPTING"
        end
        
        return "DENIED"
        """
        
        grace_deadline = int(time.time() * 1000) + 3000  # 3s grace period
        result = self.redis.eval(
            lua, 4,
            self.holder_key, self.priority_key,
            self.grace_key, self.channel,
            owner_id, priority, ttl_ms, grace_deadline
        )
        
        if result == "PREEMPTING":
            # Wait for the lock to be released
            return self._wait_for_preemption(owner_id, priority, grace_deadline)
            
        return result == "ACQUIRED"
        
    def _wait_for_preemption(self, owner_id, priority, deadline_ms):
        deadline = time.monotonic() + (deadline_ms / 1000)
        pubsub = self.redis.pubsub()
        pubsub.subscribe(self.channel)
        
        while time.monotonic() < deadline:
            # Check if lock is now available
            try:
                result = self.redis.set(
                    self.holder_key, owner_id,
                    nx=True, px=30000
                )
                if result:
                    self.redis.set(self.priority_key, priority)
                    return True
            except:
                pass
            time.sleep(0.05)
        
        return False  # Grace period expired without acquiring
```

**What to watch for**: The candidate must address priority inversion. Priority inversion occurs when a high-priority process waits for a low-priority process because a medium-priority process is running. The solution is priority inheritance: when a low-priority holder is preempted, its priority is temporarily raised to the requester's priority, preventing medium-priority processes from preempting the low-priority holder during the grace period.

The candidate must also discuss the notification mechanism. Redis pub/sub is suitable for this use case. The preempted holder subscribes to a notification channel. When preemption is triggered, the lock service publishes a message to the channel. The holder receives the message and starts its graceful shutdown. If the holder does not respond within the grace period, the lock is force-released.
