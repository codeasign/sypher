# Understanding Leader Election

Build the mental model that explains how distributed systems select a single coordinator to serialize writes and maintain consistency.

---

## The Problem Story

A distributed database runs on 5 nodes across 3 availability zones. Each node accepts writes from application clients. At 2:14 PM, two customers add items to the same shopping cart simultaneously. One write lands on node 2 in us-east-1a. The other lands on node 4 in us-east-1b. Both nodes accept the write. Both return "success" to the client.

Now the cart contains duplicate items. The inventory count is wrong. The total price is double what it should be.

The engineering team traces the issue. The root cause is clear: every node believed it could accept and process writes independently. There was no single authority to order the operations. Without a coordinator, conflicting writes passed each other like ships in the night.

The team tried a simple fix: node 1 would be the designated writer. They hardcoded its IP address into every application config. This worked for three days. Then node 1 lost power during a maintenance window. Writes stopped entirely. The configs had no fallback. The application was down for 37 minutes while someone manually updated every config file to point at node 2.

This is the problem leader election solves.

A deeper look reveals the broader pattern. A distributed system with N nodes can suffer from conflicting state mutations when no coordination mechanism exists. The challenge is not specific to databases. Any shared state — a distributed lock, a configuration key, a task queue — faces the same problem. Without a single authority, the system cannot guarantee a total order of operations. The result is inconsistency, data corruption, and hard-to-debug application bugs.

The team learned this lesson the hard way. They had assumed the database layer would handle ordering. Instead, they discovered that distributed databases without leader election or consensus are eventually consistent at best. The inventory overcount was not a bug in the application code. It was a systems design failure: no leader, no ordering, no consistency guarantee.

---

## The Insight

Leader election is the process by which nodes in a distributed system select one member to act as the sole coordinator for write operations and state changes. The elected leader serializes all mutations, ensuring that two conflicting writes never commit independently.

The sports referee analogy clarifies the concept. A basketball game has 10 players on the court, but only one referee whose decisions are final. Players can disagree with a call, but they follow it. If the referee collapses, the game stops. After a timeout, a new referee is chosen. The game resumes. Every player accepts the new referee's authority because the mechanism for selection is agreed upon in advance.

In distributed systems, the leader is the referee. Followers accept the leader's decision about the order of writes. If the leader fails, a new one is elected using a predefined protocol. The cluster continues operating.

The core insight is that leader election trades the complexity of conflict resolution for the simplicity of a single decision-maker. The system accepts a brief period of unavailability during elections in exchange for a clear, consistent ordering of operations at all other times.

This trade-off is fundamental to distributed systems theory. The CAP theorem states that a distributed system cannot simultaneously provide Consistency, Availability, and Partition tolerance. Leader election leans into consistency: it sacrifices a small window of availability (during the election) to maintain strong consistency at all other times. This is why most strongly consistent databases use leader election as part of their consensus protocol.

## Core Vocabulary

**Leader:** The single node that coordinates write operations and manages state changes. All other nodes accept the leader's ordering of events.

**Follower:** A non-leader node that accepts writes forwarded from the leader and replicates data. Followers respond to heartbeat messages and can participate in leader elections.

**Candidate:** A follower that has detected the absence of a leader and transitions to candidate state, requesting votes from other nodes to become the new leader.

**Term / Epoch:** A monotonically increasing number that identifies a specific leadership period. Each successful election starts a new term. Terms prevent a node from one term from issuing stale commands in a later term.

**Election timeout:** A randomized timer each follower maintains. When the timer expires without hearing from the leader, the follower assumes the leader is dead and starts an election. Typical values range from 150 ms to 300 ms for Raft-based systems.

**Heartbeat:** A periodic message the leader sends to followers to announce its continued authority. Heartbeats carry no log entries in steady state and consume roughly 100 bytes per node per interval.

**Bully algorithm:** A leader election algorithm where the node with the highest ID wins. If a node detects leader failure, it broadcasts its candidacy to all higher-ID nodes. If no higher-ID node responds, it becomes the leader.

**Ring algorithm:** A leader election algorithm for ring topologies where nodes pass an election token around the ring. The node with the highest ID circulating in the token becomes the coordinator.

**Quorum:** The minimum number of nodes that must agree on a leader for the election to be valid. Typically a simple majority: floor(N/2) + 1 for a cluster of N nodes.

**Leader lease:** A time-bound grant of leadership authority. The leader holds the lease for a fixed duration and must renew it before expiry. If the lease expires, followers assume the leader has failed and begin a new election.

**Split vote:** A scenario where two candidates each receive votes from a subset of nodes but neither achieves a quorum. The election stalls until the timeout triggers a new round.

**Pre-vote:** An optimization where a candidate checks with other nodes before starting a real election. If the candidate is behind on log entries, other nodes reject the pre-vote, sparing the cluster from a disruptive election cycle.

**Quorum size:** For a cluster of N nodes, quorum is N/2 + 1. A 5-node cluster needs 3 votes to elect a leader. A 7-node cluster needs 4 votes. This ensures that no two overlapping majorities can elect different leaders simultaneously.

**AppendEntries:** The RPC that Raft leaders send to followers. When the entries list is empty, this serves as a heartbeat. When entries are present, it carries log entries to replicate.

**RequestVote:** The RPC that a candidate sends to other nodes during an election. It carries the candidate's term, last log index, and last log term, enabling the log completeness check.

### Step-by-Step Election

A complete leader election proceeds through these steps:

**Step 1: All nodes start as Followers.** Every node begins in the follower state. This is the default role, ensuring that no node assumes leadership without a proper election. Each follower has a randomized election timeout between 150 ms and 300 ms. Followers expect periodic heartbeat messages from an active leader.

**Step 2: The leader fails.** Node 4, the current leader, loses network connectivity to the cluster. No heartbeats reach the followers. The cause could be a hardware failure, a network partition, a kernel panic, or a process crash. From the followers' perspective, the leader has disappeared.

**Step 3: Follower timeouts expire.** Follower 1 has an election timeout of 210 ms. After 210 ms without a heartbeat, it transitions to candidate state. Follower 2 times out at 180 ms. Follower 3 at 250 ms. The first follower to time out acts first.

**Step 4: Candidate requests votes.** Follower 2 (timeout 180 ms, the shortest) becomes Candidate. It increments the current term from 3 to 4. It sends RequestVote messages to every other node in the cluster. Each RequestVote includes the candidate's term, candidate ID, last log index, and last log term.

**Step 5: Nodes vote.** Each node receives the RequestVote from Candidate 2. Nodes check:
- Is this the highest term they have seen?
- Is the Candidate's log at least as up-to-date as their own?
- Have they already voted in this term?

If all conditions are satisfied, the node grants its vote.

**Step 6: Majority check.** The Candidate counts the votes. In a 5-node cluster, the quorum is 3 votes (majority of 5). The Candidate must receive at least 3 votes including its own.

**Step 7: Candidate becomes Leader.** With 3 votes secured, Candidate 2 transitions to Leader. It immediately begins sending heartbeat messages to all nodes at every heartbeat interval (typically 50 ms).

**Step 8: Leader starts accepting writes.** The new leader begins receiving client write requests, appending them to its log, and replicating them to followers.

**Step 9: Other Candidates revert to Follower.** If Follower 1 or 3 also started an election, they receive the new leader's heartbeat and see term 4. They revert to follower state.

**Step 10: Steady state resumes.** The cluster operates normally with Node 2 as the leader. Heartbeats flow every 50 ms. Election timers reset with each heartbeat.

During steady state, every write goes through a strict pipeline:
1. Client sends write to the leader.
2. Leader appends the write to its log (persisted to disk).
3. Leader sends AppendEntries to all followers.
4. Followers append the entry to their logs and acknowledge.
5. When a majority of followers acknowledge, the leader commits the entry.
6. Leader applies the entry to its state machine and responds to the client.
7. Followers apply the entry to their state machines asynchronously.

This pipeline ensures that even if the leader crashes immediately after responding to the client, the write is replicated on a majority of nodes and will survive.

### The Randomized Timeout Mechanism

The election timeout randomization is critical. Without it, every follower would time out simultaneously, all become candidates, and all send RequestVote simultaneously. The result is a split vote where no candidate reaches a majority, forcing the entire cluster to wait for another timeout round.

The probability of a split vote depends on the randomization spread and the number of nodes. With a spread of 150 ms (e.g., 150 to 300 ms), the probability that exactly one node times out first in the critical window is approximately 95% for a 5-node cluster. This drops to about 80% for a 9-node cluster, which is why larger clusters sometimes use wider randomization spreads.

With randomized timeouts between 150 and 300 ms, the probability of simultaneous timeouts drops dramatically. The first node to time out becomes a candidate and starts requesting votes before other nodes have left the follower state.

### Leader Lease Mechanics

Once elected, the leader holds a time-limited lease of authority. The lease duration is typically 500 ms to 1000 ms. During the lease:
- Followers accept the leader's authority without question.
- The leader must renew the lease before it expires by receiving heartbeat acknowledgments from a majority.

If the leader cannot renew the lease (due to network partition or overload), it voluntarily steps down. This prevents split-brain scenarios where the old leader continues accepting writes after a new leader has been elected.

The lease also enables read-after-write consistency for followers. A follower can safely serve a read from its local state if the leader's lease has not expired, because no other node can be the leader during the lease period.

## Variants

### Bully Algorithm

The Bully algorithm selects the node with the highest ID as the leader. When a node detects the leader has failed, it sends an ELECTION message to all nodes with higher IDs. If none respond, it becomes the leader. If a higher-ID node responds, that node takes over the election process.

The algorithm is called "Bully" because the highest-ID node bullies lower-ID nodes into submission. This makes the election deterministic: in a stable cluster, the highest-ID node is always the leader. This predictability simplifies debugging and operations, but it means the leader is not necessarily the most capable node. A high-ID node with slow disk I/O could be the leader while a low-ID node with fast hardware stands by.

### Ring Algorithm

The Ring algorithm organizes nodes in a logical ring topology. When a node detects leader failure, it starts an election by sending an ELECTION message containing its node ID to its successor. Each node adds its own ID and forwards the message. When the message returns to the originator, the node with the highest ID is elected.

The ring algorithm has the advantage of O(N) message complexity — exactly one message per node per election. However, it requires maintaining a consistent ring membership view across all nodes. If the ring topology changes (nodes join or leave), the election may fail and must be retried.

### Raft-Style Election

Raft uses randomized election timeouts, request-vote RPCs, and term-based leadership to guarantee safety. Each term has at most one leader. A leader is elected only if it has the most up-to-date log entries, ensuring no committed entries are overwritten.

Raft's key innovation over earlier algorithms is the log completeness check. When a node votes for a candidate, it verifies that the candidate's log is at least as up-to-date as its own. This means a slow node with a stale log cannot win an election, even if its timeout fires first. The elected leader always has all committed log entries, making log recovery straightforward.

### ZooKeeper-Based Election

ZooKeeper implements leader election using ephemeral sequential znodes. Each node creates an ephemeral sequential znode under an election path. The node with the smallest sequence number becomes the leader. Other nodes watch the preceding znode and take over if it disappears.

This approach leverages ZooKeeper's linearizable write guarantees. The sequential znode creation ensures a total order. The watch mechanism enables immediate notification of leadership changes without polling. The ephemeral nature of the znodes ensures that if a node crashes, its znode is automatically deleted and a new leader is elected.

## Mental Model Check

**Question 1:** A 3-node cluster runs a Raft-based database. The leader crashes. How long does the cluster remain unavailable?

**Answer:** Between 150 ms and 300 ms (one election timeout). The two remaining followers have randomized timeouts. The first to time out becomes candidate, requests votes, and with 2 out of 3 votes (majority) becomes the new leader. Total downtime is roughly one timeout period plus network round-trip time.

**Question 2:** What happens if two followers have election timeouts of 150 ms and 151 ms?

**Answer:** If the leader crashes, Follower A (150 ms) times out first by 1 ms. Follower A becomes candidate, sends RequestVote to Follower B. Follower B checks: it has not timed out yet (still at 149 ms), and it has not voted this term. Follower B grants its vote. Follower A has 2 votes (self + B) out of 3. Election succeeds within a single round-trip.

**Question 3:** In the Bully algorithm, can a lower-ID node ever become leader?

**Answer:** No. The Bully algorithm guarantees the highest-ID node is always elected. Higher-ID nodes always respond to ELECTION messages and take over. If a higher-ID node is down, the next-highest-ID node becomes leader. This ensures stability but means leadership is deterministic based on node IDs.

**Question 4:** A ZooKeeper ensemble has 5 servers. Server A (node-0000000001) is the leader. Server A loses its ZooKeeper session due to a network partition. What happens?

**Answer:** The ephemeral znode /election/node-0000000001 is deleted automatically by ZooKeeper when Server A's session expires. Server B, which watches node-0000000001, receives a notification. Server B checks the /election path and finds node-0000000002 (its own znode) is now the smallest. Server B becomes the leader. Server A, after reconnecting, creates a new ephemeral sequential znode (node-0000000005) and becomes a follower.

**Question 5:** How does Raft prevent a node with stale logs from becoming leader?

**Answer:** Raft's RequestVote RPC includes the candidate's last log term and index. Receiving nodes compare: if the voter's last log entry has a higher term, or the same term but a higher index, the voter rejects the request. A candidate must have at least as complete a log as a majority of nodes to be elected, ensuring the leader has all committed entries.

**Question 6:** A 5-node Raft cluster has 3 nodes in us-east-1a and 2 nodes in us-east-1b. The us-east-1a availability zone fails entirely. What happens?

**Answer:** The 2 nodes in us-east-1b cannot form a majority (need 3 votes for a 5-node cluster). No election succeeds. The cluster becomes unavailable for writes. When us-east-1a recovers, the 3 nodes there can elect a leader with 2 out of 3 votes (majority of the 3 available nodes? No — majority of the cluster is always N/2 + 1 = 3). The 3 surviving nodes elect a leader. This is why cross-AZ deployment should distribute nodes evenly, not concentrate them.

**Question 7:** You hardcode the leader's IP address in a configuration file. The leader crashes. After restarting, the leader's IP changes because it uses DHCP. What happens?

**Answer:** The application cannot find the leader. Writes fail. An operator must update the configuration with the new IP. This is the manual failover anti-pattern. In a production system, a consensus service (etcd, ZooKeeper) should track the leader's address, and applications should query the service rather than using hardcoded IPs.
