# Leader Election in Production

Examine how ZooKeeper, etcd, MongoDB, and Redis implement leader election in production systems. Extract lessons from real-world outages and design decisions.

## ZooKeeper — Ephemeral Znodes for Kafka Leadership

Apache ZooKeeper is the coordination service used by Apache Kafka, Apache Hadoop, and many other distributed systems. ZooKeeper provides a strongly consistent hierarchical namespace with ephemeral sequential znodes that are the building blocks of leader election.

### Real-World Lesson: ZooKeeper Session Timeout Trade-off

Kafka's ZooKeeper session timeout is typically set to 6-18 seconds. This means a failed controller takes 6-18 seconds to be detected and replaced. The trade-off is deliberate: shorter timeouts (3 seconds) would detect failures faster but risk false positives during GC pauses. Longer timeouts (30 seconds) would increase stability but extend unavailability. Kafka chose 18 seconds as the default, favoring stability over recovery speed.

### ZooKeeper's Zab Protocol

ZooKeeper uses the Zab (ZooKeeper Atomic Broadcast) protocol for its own leader election. Zab is similar to Raft but predates it. Key differences:

- Zab uses epochs (equivalent to Raft's terms).
- Zab elects a leader using a Fast Leader Election (FLE) algorithm that converges in O(log N) rounds.
- Zab guarantees that messages are delivered in order (total order broadcast).
- Zab's leader election is separate from the broadcast phase, unlike Raft's integrated approach.

### etcd Performance Tuning

For production etcd deployments, key tuning parameters include:
- **Disk I/O:** etcd is sensitive to disk write latency because every Raft entry and vote is fsynced to disk. Use SSDs with low p99 latency (under 10ms). Avoid network-attached storage with variable latency.
- **Network latency:** Ensure inter-node latency is under 10ms for stable heartbeats. For cross-datacenter deployments, increase heartbeat-interval to 200-500ms.
- **Snapshot frequency:** A large Raft log slows down new nodes catching up. Set the snapshot interval to keep the log under 100,000 entries.

### Cluster of Size 3 vs 5

Kubernetes production deployments typically use 3 or 5 etcd nodes:

- **3 nodes:** Tolerates 1 failure. Quorum = 2. This is the minimum for production.
- **5 nodes:** Tolerates 2 failures. Quorum = 3. Recommended for production clusters with over 100 worker nodes.

A 3-node etcd cluster with a leader in us-east-1a and followers in us-east-1b and us-east-1c provides AZ-level fault tolerance. If us-east-1a fails, the other two nodes (both in different AZs) maintain quorum.

### MongoDB Election Performance

MongoDB's default election timeout is 10 seconds — significantly higher than Raft's 150-300ms. This is a deliberate design choice:

- **MongoDB is often deployed over WAN links** where latency and packet loss are higher.
- **MongoDB uses a heartbeat interval of 2 seconds** (compared to Raft's 50ms).
- **The high timeout favors stability over speed.** A 10-second failover is acceptable for many workloads.

Faster election timeouts can be configured by setting `settings.electionTimeoutMillis` to a lower value, but this increases the risk of spurious elections.

### MongoDB Arbiter Nodes

MongoDB supports arbiter nodes that participate in elections but do not replicate data. An arbiter has a single vote and no data storage. In a 2-node replica set, adding an arbiter creates a 3-vote system where 2 votes are needed for majority. This allows a 2-node + arbiter deployment to survive a single node failure. The arbiter is lightweight (runs on a small VM) and requires no data storage.

### Redis Sentinel Down-After-Milliseconds

Redis Sentinel's down-after-milliseconds (default: 30000ms) is the heartbeat timeout. This 30-second default is extremely conservative compared to Raft's 150-300ms. The justification:

- Redis Sentinel is often deployed with cross-datacenter replication, where network latency can be highly variable.
- Redis Sentinels are lightweight processes that may share hosts with other applications.
- Redis failover is not guaranteed to be split-brain safe. The 30-second timeout reduces the probability of false positives.

## Open Source Implementations

### ZooKeeper (Apache)

- **Algorithm:** Zab (ZooKeeper Atomic Broadcast), based on Paxos
- **Election details:** Fast Leader Election (FLE) protocol
- **Session timeout:** 1-30 seconds (configurable)
- **Use cases:** Kafka, HBase, Solr, distributed coordination

### etcd (CNCF / CoreOS)

- **Algorithm:** Raft consensus
- **Election details:** Randomized timeout, term-based, log completeness check
- **Heartbeat interval:** 100ms default
- **Election timeout:** 1000ms default
- **Use cases:** Kubernetes, service discovery, distributed config

### Consul (HashiCorp)

- **Algorithm:** Raft consensus
- **Election details:** Similar to etcd, with serf (gossip) for membership
- **Heartbeat interval:** 200ms default
- **Election timeout:** 500-1500ms default
- **Use cases:** Service discovery, health checking, key-value store

## Cloud Offerings

### AWS ElastiCache for Redis (Cluster Mode)

- **Algorithm:** Redis Sentinel (managed by AWS)
- **Election details:** Automatic failover within 30 seconds of primary failure
- **Multi-AZ:** Required for automatic failover
- **Failover behavior:** DNS record updated to point to new primary

### Azure Cosmos DB

- **Algorithm:** Proprietary multi-master with leader election per partition
- **Election details:** Logical timestamp-based election with conflict resolution
- **Failover:** Automatic within seconds
- **Consistency levels:** Configurable (strong, bounded staleness, session, eventual)

### Google Cloud Spanner

- **Algorithm:** Paxos-based across paxos groups
- **Election details:** TrueTime-based leader election with GPS clocks
- **Failover:** Designed for zero-downtime maintenance
- **Global distribution:** Supports configurable replication across continents

## Lessons from Production Outages

### Outage Analysis: etcd Leader Flapping (Kubernetes, 2021)

**Symptom:** kube-apiserver reported "etcd leader changes" at 50+ per minute. Pods failed to schedule. API latency exceeded 30 seconds.

**Root cause:** A misconfigured disk IOPS throttle caused etcd disk writes to stall for 200-500ms. Since etcd uses `fsync` for every write, the leader's AppendEntries responses were delayed, triggering followers' election timers. Each election produced a new leader, but the new leader also hit the same disk throttle. The cycle repeated.

**Impact:** 4 minutes of degraded control plane. No data loss.

**Resolution:** Increased the disk IOPS limit from 100 to 1000. After the fix, leader changes dropped to fewer than 1 per month.

### Outage Analysis: ZooKeeper Session Expiration (Kafka, 2020)

**Symptom:** Kafka brokers were repeatedly re-electing the controller. Topic metadata became inconsistent. Produce requests returned LEADER_NOT_AVAILABLE errors.

**Root cause:** A network switch failure caused intermittent packet loss between Kafka brokers and the ZooKeeper ensemble. The session timeout (18 seconds) was shorter than the time to repair the switch (45 seconds). Brokers lost their ZooKeeper sessions, triggering controller re-elections on reconnect.

**Impact:** 25 minutes of partial unavailability. 3 major incidents over 2 days.

**Resolution:** Increased the ZooKeeper session timeout to 30 seconds for the affected cluster. The network switch was replaced during a maintenance window.

### Lesson Summary

1. **Disk latency matters more than CPU.** Leader election requires persisting votes to disk. Slow disk I/O can trigger flapping even on a fast CPU.
2. **Test under load.** Heartbeat and election behavior changes dramatically under production load.
3. **Monitor leadership metrics.** Track `leader_changes_total` and `election_duration_seconds` as cardinal health metrics.
4. **Set the timeout to 10x heartbeat.** This provides a safety margin against transient delays without significantly increasing failover time.
5. **Deploy in 3 AZs.** A 5-node cluster across 3 availability zones can survive one full zone failure while maintaining quorum.
6. **Use dedicated instances.** Shared instances with CPU steal time or I/O throttling cause flapping. Dedicated instances eliminate noisy neighbor problems.
7. **Test disk throttling.** Cloud disks with burst IOPS (like AWS gp2/gp3) can throttle after sustained writes, causing sudden latency spikes during leader election. Provision adequate baseline IOPS.
