# Leader Election at Scale

Evolve leader election from a naive single-leader setup to a multi-region, multi-cluster production architecture. Integrate with databases, caches, and message queues.

---

The simplest leader election architecture uses a single leader with hardcoded failover. This works for small deployments but breaks under real-world conditions.

### Component Responsibilities

Each layer in the production architecture has a specific responsibility:

- **Traffic Layer:** Distributes incoming requests using anycast VIP or DNS-based routing. Does not need to know which node is the leader initially.
- **Proxy Layer:** Performs health checks against all nodes. Discovers the current leader from the consensus service. Routes write requests to the leader. Routes read requests to any node (for read-after-write consistency, reads should also go to the leader).
- **Consensus Service:** Provides the coordination primitive for leader election. etcd or ZooKeeper offers linearizable writes and watch notifications.
- **Application Nodes:** Run the leader election protocol (Raft, Bully, or ZooKeeper-based). The leader handles write serialization.
- **Monitoring Layer:** Tracks leader changes, election durations, and term changes. Alerts on anomalies.

## Multi-Region Leader Election

In multi-region deployments, leader election must account for geographic latency and regional isolation. The standard approach is to have a separate leader within each region with regional autonomy.

The challenge with multi-region leader election is that writes must be ordered globally, but network latency between regions (50-300ms) is comparable to or larger than the election timeout. A single global leader would be too slow for most applications. The solution is to accept per-region leaders with asynchronous cross-region replication.

### Cross-Region Consistency Trade-off

With regional leaders, consistency guarantees are limited:

- **Strong consistency within a region:** Reads and writes within the same region go to the same leader. Read-after-write is guaranteed.
- **Eventual consistency across regions:** A write in us-east may take 50-300ms to propagate to eu-west. Reads in eu-west may see stale data until propagation completes.
- **Global consistency break:** If a user writes in us-east and reads in eu-west 10ms later, they may not see their own write. Applications must tolerate this or route reads to the write region.

## Capacity Planning

### Cluster Size

| Nodes | Election Quorum | Fault Tolerance | Downtime After Failure |
|---|---|---|---|
| 1 | 1 | 0 nodes | Immediate |
| 3 | 2 | 1 node | 150-300ms |
| 5 | 3 | 2 nodes | 150-300ms |
| 7 | 4 | 3 nodes | 150-300ms |
| 9 | 5 | 4 nodes | 150-300ms |

- **3 nodes** is the minimum for fault tolerance. Losing 1 node still allows quorum.
- **5 nodes** is the sweet spot for most deployments. Withstands 2 simultaneous failures.
- **7+ nodes** adds overhead without significant benefit unless cross-datacenter deployment.

### Network Requirements

- **Latency between nodes:** Less than 10ms round-trip for heartbeats. Higher latency increases effective election time.
- **Bandwidth:** Minimal — 100 bytes per heartbeat per node per heartbeat interval. A 5-node cluster at 50ms interval uses 8 KB/sec.
- **Jitter tolerance:** Heartbeat intervals should be at least 3x the maximum observed jitter to prevent spurious elections.

### Node Placement Strategy

The distribution of nodes across availability zones directly affects fault tolerance:

| Nodes per AZ | 3-Node Cluster | 5-Node Cluster | 7-Node Cluster |
|---|---|---|---|
| 3 AZs (recommended) | 1-1-1 | 2-2-1 | 3-2-2 |
| 2 AZs | 2-1 | 3-2 | 4-3 |
| 1 AZ | 3 | 5 | 7 |

**Recommendation:** Deploy with 3 availability zones. For a 5-node cluster, the 2-2-1 distribution means that losing any single AZ leaves 3 or 2+1=3 nodes, which still forms a majority. Losing any two AZs leaves only 1 node, which cannot form a majority.

### Storage Considerations

Leader election protocols require persistent storage for the votedFor field and the Raft log. Storage latency directly impacts election performance:

- **SSD (NVMe):** 0.1-0.5ms fsync latency. Ideal for Raft clusters.
- **SSD (SATA):** 0.5-2ms fsync latency. Acceptable for most workloads.
- **HDD:** 5-15ms fsync latency. Not recommended for leader election clusters. The high latency increases the risk of split votes and flapping.
- **Network storage (EBS, SAN):** 1-50ms fsync latency. Use with caution. Provision sufficient IOPS to avoid throttling.

### Memory Sizing

For etcd and similar Raft-based systems, memory sizing rules:

- **Working set:** The total size of all keys should fit in memory for optimal read performance.
- **Raft log:** The unapplied portion of the Raft log is in memory. Compact aggressively to keep this under 100,000 entries.
- **Snapshot:** Memory-mapped snapshot files. Each snapshot is a point-in-time copy of the entire data store.
- **Recommendation:** For production etcd clusters with 10,000 keys, allocate 2-4 GB of RAM per node.

## Disaster Recovery

### Backup and Restore

Leader election clusters need careful backup strategies:

- **Snapshot backup:** Take periodic snapshots of the Raft log and state machine. The snapshot file contains all committed state at a point in time.
- **Snapshot frequency:** Every 100,000 log entries or every hour, whichever comes first.
- **Restore process:** Restore a snapshot on a new node. The node starts with the snapshot's state and catches up on any missing log entries from the leader.
- **Important:** Do not restore a snapshot on a running cluster. The snapshot may contain state that conflicts with the current leader's log.

### Node Replacement

When replacing a failed node in a leader election cluster:

1. Provision a new node with the same configuration (storage, memory, network).
2. Start the new node without joining it to the cluster.
3. Add the new node to the cluster's membership list.
4. The new node joins as a follower and synchronizes its log from the leader.
5. Verify the new node is fully caught up before removing the failed node.

### Cluster Rebuild

In extreme cases (all nodes lost), a full cluster rebuild is necessary:

1. Restore the most recent snapshot on a single node.
2. Start the node. It becomes the leader (single-node cluster).
3. Add the remaining nodes as followers.
4. Verify the cluster is healthy before routing production traffic.

## Operations

### Rolling Restarts

When upgrading nodes in a leader-election cluster, perform a rolling restart:

1. Identify the current leader.
2. Restart follower nodes one at a time. Wait for each to sync fully before proceeding.
3. Restart the leader last. The leader's departure triggers an election (150-300ms downtime).
4. Verify the new leader is operational and heartbeats are flowing.

### Leader Transfer

For planned maintenance, gracefully transfer leadership:

1. Issue a `step-down` command to the current leader.
2. The leader sends a `timeout-now` message to a chosen follower, triggering an immediate election.
3. The chosen follower wins (it has the most up-to-date log) and becomes leader.
4. Total downtime: 1-10ms (election triggered immediately, no timeout wait).

### Monitoring

Track these metrics for leader election health:

- **leader_changes_total:** Spikes indicate instability or flapping.
- **leader_election_duration_seconds:** Values above 1 second suggest network issues.
- **term_changes_total:** More than 10 per minute indicates a problem.
- **split_votes_total:** Any split vote indicates poor timeout randomization or network partition.
- **leader_uptime_seconds:** Frequent resets suggest flapping.
- **fsync_duration_seconds:** High values indicate disk I/O issues that can cause flapping.
- **proposal_failed_total:** Failed proposals indicate the leader lost quorum.
- **client_request_latency:** Elevated latency during elections signals write unavailability.

Set up alerting rules:

- Alert if leader_changes exceeds 5 per minute over a 5-minute window.
- Alert if election_duration exceeds 1 second.
- Alert if no leader is elected for more than 2 seconds.
- Alert if fsync p99 exceeds 100ms for more than 5 minutes.

### Automation Playbooks

For operational efficiency, implement these automation playbooks:

**Playbook 1: Leader Step-Down**
```
Trigger: Planned maintenance on leader node.
Actions:
1. Ansible/SSH to leader → `etcdctl transfer-leadership`
2. Wait for new leader to be elected (verify via health endpoint).
3. Confirm the old leader is now a follower.
4. Proceed with maintenance on the old leader.
```

**Playbook 2: Dead Node Replacement**
```
Trigger: Node heartbeat timeout > 30 seconds.
Actions:
1. Confirm the node is unreachable (network check).
2. Provision a replacement node with identical configuration.
3. Add the replacement to the cluster (etcdctl member add).
4. Monitor the replacement's log sync progress.
5. Remove the dead node (etcdctl member remove).
```

**Playbook 3: Cluster Health Check**
```
Trigger: Routine (every 5 minutes).
Actions:
1. Verify all nodes are reachable.
2. Verify exactly one leader exists.
3. Verify the leader's term is not changing rapidly.
4. Verify disk latency is under 10ms p99.
5. Verify clock skew between nodes is under 100ms.
```
