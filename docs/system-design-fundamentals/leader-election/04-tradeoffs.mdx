# Leader Election — Trade-offs and Failure Modes

Analyze the core tension between fast failure detection and election stability. Understand when leader election is the right tool and when it is not.

---

## The Core Tension

The fundamental trade-off in leader election is between **speed of failure detection** and **stability of the elected leader**.

This trade-off is captured by the relationship between the heartbeat interval (H), the election timeout (T), and the network jitter (J). For stable operation, T must be greater than H + J. If network jitter is 50ms and the heartbeat interval is 50ms, the minimum safe timeout is 100ms. In practice, T is set to 3x-10x H to provide a safety margin.

The cost of a false positive (spurious election) is significant: during the election window (150-300ms), the cluster cannot accept writes. Additionally, the log synchronization process after the election consumes bandwidth and CPU. A single spurious election is cheap, but frequent spurious elections can reduce throughput by 50% or more.

## Decision Matrix

| Requirement | Recommended Algorithm | Rationale |
|---|---|---|
| Strong consistency, log replication | Raft | Term-based, log completeness guarantee |
| High availability, low latency | ZooKeeper ephemeral znodes | Watch-based, immediate failover |
| Simple implementation, small cluster | Bully algorithm | O(N) election, easy to understand |
| Ring topology, peer-to-peer | Ring algorithm | Token-based, O(N) messages |
| Multi-region, high latency | Pre-vote Raft | Pre-vote prevents disruption |
| Immutable infrastructure | etcd-based election | Auto-joins, self-managing |
| Java ecosystem, existing ZooKeeper | ZooKeeper leader latch | Industry standard, well-tested |
| Go ecosystem, Kubernetes native | etcd + Raft | Native Kubernetes integration |

## Failure Modes

### Failure Mode 1: Leader Flapping

Leader flapping occurs when a leader repeatedly loses and regains its position, causing frequent elections and log synchronizations.

The root cause is typically transient delays that exceed the heartbeat interval but resolve quickly. Common triggers include:

- JVM garbage collection pauses of 100-500ms.
- CPU steal time in virtualized environments (noisy neighbor on the hypervisor).
- Network micro-bursts causing 1-5% packet loss.
- Disk I/O stalls during backup windows or compaction.
- Memory pressure causing swapping or OOM killer activity.

### Failure Mode 2: Split Vote Loop

When randomized timeouts are insufficiently randomized, multiple nodes consistently time out simultaneously, causing repeated split votes.

**Root cause:** Identical or near-identical timeout values across nodes. If all 5 nodes have a timeout of exactly 200ms, they all fire simultaneously.

**Impact:** 5-10x increase in election time. Each split vote adds 150-300ms of delay. Three consecutive split votes can cause 1 second of downtime.

**Mitigation:** Use a randomization spread of at least 150ms. Ensure the random seed is different for each node (use node ID as part of the seed).

### Failure Mode 3: Quorum Loss

When too many nodes fail simultaneously, no quorum exists and the cluster becomes unavailable.

**Threshold:** For a 5-node cluster, losing 3 nodes means no quorum. The remaining 2 nodes cannot elect a leader.

**Impact:** Complete write unavailability. Reads may still be served from the remaining nodes, but they are stale.

**Mitigation:** Deploy at least 5 nodes in production. Use a tiebreaker or witness node to increase the effective quorum size.

### Failure Mode 4: Slow Leader

A leader that is still running but too slow to process requests can degrade performance without triggering an election.

**Root cause:** Resource exhaustion (CPU, memory, disk I/O) on the leader node. The leader sends heartbeats on time but processes requests slowly.

**Impact:** Write latency increases by 5-10x. Clients time out. The system is degraded but not down.

**Mitigation:** Implement lease-based quality of service checks. If the leader's write latency exceeds a threshold, trigger a voluntary step-down.

### Failure Mode 6: Zeno's Paradox of Elections

In edge cases, a node can be repeatedly elected but immediately fail before sending a heartbeat. This creates an infinite loop of elections, each consuming resources and extending unavailability.

**Root cause:** A node crashes immediately after winning an election but before sending the first heartbeat. Followers detect the absence of heartbeats and re-elect the same node (because its timeout fires first again).

**Impact:** Complete cluster unavailability during the loop. Resources are wasted on repeated elections.

**Mitigation:** Implement a backoff mechanism. A node that has been recently elected should wait for a full heartbeat interval before accepting the leader role. If it crashes before completing the interval, it must wait for a longer timeout before starting another election.

### Failure Mode 7: Clock Skew

Leader election protocols assume bounded clock skew between nodes. If clocks drift significantly, the timeout mechanism breaks.

**Root cause:** Clock skew exceeding 50% of the election timeout. A node with a slow clock may never time out. A node with a fast clock may time out prematurely.

**Impact:** The fast-clock node constantly starts elections. The slow-clock node is perpetually the leader (its lease never expires from its perspective).

**Mitigation:** Use NTP with tight synchronization (sub-millisecond precision). Monitor clock skew and alert if it exceeds 100ms. Increase the election timeout to accommodate known clock skew in the deployment.

### Failure Mode 5: Network Asymmetry

Network partitions that are not symmetric can cause scenarios where a node can send messages but not receive them, or vice versa.

**Impact:** A leader may believe it is still leader (it can send heartbeats) but followers cannot receive them. Followers elect a new leader, creating a temporary split-brain until the old leader discovers the higher term.

**Mitigation:** Use bidirectional network checks. A leader should confirm that at least one heartbeat was acknowledged before extending its lease.

## Anti-Patterns

### Anti-Pattern 1: Hardcoded Leader

Hardcoding a node's IP address as the leader in configuration files or application code. This eliminates the failover capability entirely.

**Why it fails:** The leader is a single point of failure. Manual failover takes minutes to hours. The hardcoded IP becomes out of date during maintenance.

### Anti-Pattern 2: Small Election Timeout

Setting election timeouts too low (under 100ms) to achieve fast failover.

**Why it fails:** Normal network jitter and GC pauses trigger spurious elections. The cluster spends more time electing leaders than serving requests. Throughput drops by 50-80%.

### Anti-Pattern 3: Ignoring Pre-Vote

Disabling the pre-vote extension to reduce complexity.

**Why it fails:** A partitioned node that reconnects can force a new election even though its log is stale. The disruption is unnecessary and wastes cluster resources. Pre-vote eliminates this.

### Anti-Pattern 4: Single Region Deployment

Deploying all nodes in a single availability zone.

**Why it fails:** A single AZ failure takes down the entire cluster. If the AZ loses power, there is no way to elect a leader. Deploy across 3 AZs for 5-node clusters.

### Anti-Pattern 5: Unequal Voting Power

Assigning different vote weights to different nodes (e.g., a primary node gets 2 votes).

**Why it fails:** This violates the majority principle. The weighted node becomes a single point of failure. If it goes down, no quorum is possible even if other nodes are healthy.

### Anti-Pattern 6: Manual Leadership Transfers

Requiring an operator to manually trigger leader transfers for planned maintenance.

**Why it fails:** Manual steps are error-prone. The operator may forget. The system should support graceful leader transfer via API.

### Anti-Pattern 7: Over-Scaling the Cluster

Deploying 25 nodes for a leader-election cluster when 5 would suffice.

**Why it fails:** More nodes means more heartbeats, more potential split votes, and more log replication traffic. The election timeout must be widened for larger clusters to avoid split votes, which increases failover time. For most workloads, 5 nodes provides enough fault tolerance (2 simultaneous failures) without the overhead of a larger cluster.

### Anti-Pattern 8: Mixing Leader Election with Application Logic

Embedding leader election logic directly into the application code instead of using a dedicated consensus service.

**Why it fails:** The election logic must be tested, audited, and verified for correctness. A bug in the application's election logic can bring down the entire system. Using etcd or ZooKeeper as the coordination layer separates concerns and leverages battle-tested code.

## When NOT to Use Leader Election

Leader election is not always the right tool. Consider alternatives in these scenarios:

| Scenario | Alternative | Rationale |
|---|---|---|
| Two-node cluster | Paxos or multi-leader replication | Two nodes cannot form a majority (2/2 = 100% required, no fault tolerance) |
| Read-only workload | Anycast or DNS round-robin | No writes to coordinate, no leader needed |
| Eventually consistent data | Gossip protocol (Cassandra, DynamoDB) | No single coordinator, conflict resolution is deferred |
| High-write, low-latency requirement | CRDTs (Conflict-free Replicated Data Types) | No coordination needed, each node writes independently |
| Single-node deployment | Direct local access | No election protocol needed for one node |
| Strictly ordered global transactions | Global consensus (Spanner, TrueTime) | Leader election alone is insufficient for global ordering |
| Edge devices with intermittent connectivity | Local-first sync (offline-first databases) | Devices disconnected for hours cannot participate in elections |
| Append-only logs with immutable entries | Kafka-style partitioned logs | Each partition has its own leader, but no global leader needed |
