# Leader Election — Interview Playbook

Master leader election system design questions with structured frameworks, phrase banks, and practice drills.

---

## Level-by-Level Expectations

### Entry Level (0-2 years)

**What you must know:**
- Explain the purpose of leader election in a distributed system.
- Describe the difference between a leader and a follower.
- Draw the basic state machine: Follower → Candidate → Leader.
- Explain what a heartbeat is and why it matters.
- Understand that a leader election happens when the current leader fails.
- Describe the basic flow: leader crashes, timeout expires, candidate emerges, votes collected, new leader starts.

**Sample question:** "A 3-node database cluster needs leader election. Walk me through what happens when the leader crashes."

**Sample answer structure:**
1. Start: "The cluster has 3 nodes. Node 1 is the leader. Node 2 and Node 3 are followers."
2. Detection: "When Node 1 crashes, Node 2 and Node 3 stop receiving heartbeats. Each has a randomized election timeout."
3. Election: "Node 2's timeout fires first at 180ms. It increments the term, becomes a candidate, and sends RequestVote to Node 3."
4. Vote: "Node 3 checks its log, sees Node 2's log is up-to-date, and grants its vote."
5. Result: "Node 2 has 2 votes (self + Node 3) out of 3 — a majority. Node 2 becomes the new leader."
6. Recovery: "Node 2 immediately sends heartbeats to Node 3. Node 1, when it recovers, receives a heartbeat with a higher term and steps down to follower."

### Mid Level (2-5 years)

**What you must know:**
- Compare Bully algorithm, Ring algorithm, and Raft-style election.
- Explain randomized timeouts and why they are necessary.
- Describe the split vote scenario and how it resolves.
- Understand quorum and why majority is N/2 + 1.
- Identify trade-offs between fast failover and stability.
- Explain the leader lease concept and its role in preventing split-brain.
- Compare at-least-once vs exactly-once semantics for writes during elections.

**Sample question:** "Design a leader election mechanism for a 5-node cache cluster. How do you handle network partitions?"

**Sample answer structure:**
1. Choose Raft: "I will use Raft-style leader election with randomized timeouts of 150-300ms and heartbeats every 50ms."
2. Quorum: "A majority of 3 out of 5 nodes is needed to elect a leader. This ensures at most one leader per term."
3. Partition handling: "If the cluster splits into 3 nodes and 2 nodes, only the 3-node partition can elect a leader. The 2-node partition cannot reach quorum and rejects writes."
4. Recovery: "When the partition heals, the 2 nodes receive heartbeats from the 3-node leader, see a higher term, and step down to follower."
5. Lease: "I will use a leader lease of 500ms. The leader must receive acknowledgments from a majority within the lease period, or it steps down."
6. Safety: "The log completeness check ensures the elected leader has all committed entries."

### Senior Level (5-8 years)

**What you must know:**
- Implement or describe Raft's RequestVote safety check (log completeness).
- Design multi-region leader election with separate regional leaders.
- Analyze failure modes: flapping, asymmetric partitions, slow leader.
- Discuss pre-vote optimization and its benefits.
- Compare ZooKeeper ephemeral znode approach vs Raft.
- Design for cloud deployments with auto-scaling groups.
- Calculate the probability of split votes given cluster size and timeout spread.
- Discuss the trade-offs between synchronous and asynchronous replication during elections.

**Sample question:** "Design a globally distributed configuration service that uses leader election. How does the system handle a network partition between us-east and eu-west?"

**Sample answer structure:**
1. Regional leaders: "Each region has its own leader elected using Raft within the region. Global state is replicated asynchronously."
2. Cross-region partition: "If us-east and eu-west are partitioned, each region continues with its own leader. Writes are accepted in both regions."
3. Conflict resolution: "When the partition heals, the global configuration service reconciles the two versions using last-write-wins or a version vector."
4. Consistency guarantee: "Strong consistency within each region. Eventual consistency across regions. Applications must be designed for this."
5. Global observer: "A observer node in a third region monitors regional leaders. If a region is unreachable for 30 seconds, the observer triggers a failover."

### Staff / Principal Level (8+ years)

**What you must know:**
- Design a custom consensus protocol for a specific workload.
- Analyze and fix production incidents involving leader election.
- Compare ZooKeeper Zab, Raft, and Paxos election mechanisms.
- Design conflict-free leader election for edge computing (hundreds of nodes).
- Optimize election latency for sub-millisecond failover using hardware clocks.
- Evaluate when NOT to use leader election and use CRDTs instead.
- Design a leader election system that tolerates byzantine faults.
- Write a formal proof of safety for a novel election algorithm.

**Sample question:** "Our 5-node Raft cluster experiences leader flapping every 4 hours during the daily backup window. Diagnose and fix the issue."

**Sample answer structure:**
1. Diagnosis: "The backup window suggests a cron job or scheduled task. The backup process likely causes disk I/O spikes on the backup target node."
2. Root cause: "If the leader is the backup target, fsync latency increases during the backup. Followers detect the leader's heartbeat delay and trigger elections. Each new leader also hits the same I/O spike."
3. Check: "Verify disk p99 latency during backup window. Check if leader changes correlate with backup node identity."
4. Fix: "Exclude the backup target node from being a leader by setting its priority to 0. Or schedule the backup during off-peak hours. Or increase the election timeout to 1000ms during the backup window."
5. Prevention: "Add a metric to track leader_changes_by_node. Use a dashboard to correlate leader changes with system events."

## Phrase Bank

Use these phrases to articulate leader election concepts clearly and confidently.

### Starting the Explanation

- "Leader election solves the problem of coordination in a distributed system by selecting a single node to serialize all write operations."
- "The core insight is that a single decision-maker is simpler than distributed conflict resolution, as long as the system can recover quickly when the decision-maker fails."
- "The protocol guarantees that at most one leader exists at any given time, preventing conflicting writes."

### Describing the State Machine

- "Every node begins as a follower. Followers expect periodic heartbeats from the leader."
- "When a follower's election timeout fires without receiving a heartbeat, it transitions to candidate state."
- "The candidate increments the current term and requests votes from all other nodes."
- "If a candidate receives votes from a majority of the cluster, it becomes the leader for that term."
- "If a candidate receives a heartbeat from another node with a higher term, it steps down to follower."

### Explaining Safety

- "Raft guarantees log completeness: a candidate must have all committed log entries to be elected."
- "The RequestVote RPC includes the candidate's last log term and index, which receivers compare against their own logs."
- "Each node votes at most once per term. The votedFor field is persisted to stable storage."
- "A leader confirms it still holds the lease by verifying a quorum of heartbeat responses."

### Discussing Trade-offs

- "Faster election timeouts reduce failover time but increase the risk of spurious elections from transient network delays."
- "The sweet spot for most deployments is 150-300ms for Raft, which balances detection speed with stability."
- "Pre-vote prevents a partitioned node from forcing a disruptive election when it reconnects."
- "A larger cluster tolerates more failures but increases election latency due to more RPCs and higher split-vote probability."

### Handling Failure Scenarios

- "In a split vote, no candidate reaches a majority. The election stalls until the next timeout round."
- "Leader flapping occurs when a node repeatedly loses and regains leadership, often due to GC pauses or disk I/O spikes."
- "An asymmetric network partition can create a split-brain where the old leader continues accepting writes while a new leader is elected."
- "To mitigate split-brain, the leader must verify quorum for each write operation, not just heartbeats."

### Connecting to the Big Picture

- "Leader election is one component of a consensus protocol. It handles the problem of selecting a coordinator, but the full protocol must also handle log replication and safety."
- "The CAP theorem explains why leader election is needed: during a partition, the system must choose between consistency and availability. Leader election leans toward consistency by sacrificing a brief window of availability."

## Red Flags

Watch for these mistakes in your own answers and in candidate responses.

1. **Ignoring the log completeness check.** Claiming that any candidate can win an election is wrong. Raft requires the candidate's log to be at least as up-to-date as a majority of nodes.

2. **Suggesting manual failover.** In a distributed system interview, manual failover (changing config files by hand) signals inexperience. Always describe automated election.

3. **Assuming all nodes vote identically.** Each node evaluates the candidate's log independently. A node with a more up-to-date log rejects a candidate with a stale log.

4. **Not mentioning timeouts.** If you describe leader election without explaining election timeouts, you miss the core mechanism that triggers the process.

5. **Saying leader election handles all failures.** Leader election detects a failed leader and selects a new one. It does not magically restore consistency, recover lost writes, or handle byzantine faults.

6. **Proposing a 2-node cluster.** Two nodes cannot form a majority (2/2 = 100%). A single failure takes out quorum. A 2-node cluster with leader election offers zero fault tolerance.

7. **Ignoring the split vote scenario.** Every leader election algorithm must handle the case where multiple candidates split the vote. Describing how split votes resolve is essential.

8. **Equating leader election with consensus.** Leader election is a component of consensus, not the whole thing. Consensus requires safe log replication (Raft AppendEntries, Paxos accept phase) in addition to leader election.

9. **Not discussing persistence.** The votedFor field must be persisted to disk before granting a vote. If a node crashes and restarts, it must remember its vote to prevent voting for two candidates in the same term.

10. **Assuming monotonic clocks.** Leader election relies on timeouts, but node clocks may drift. Using randomized timeouts mitigates this, but a candidate with a clock that runs slow may be consistently outcompeted by faster-clock nodes.

## Practice Drills

### Drill 1: Whiteboard a 3-Node Election

You have 5 minutes. On a whiteboard, draw a 3-node cluster running Raft. The current leader is Node 1. It crashes. Walk through:
1. What timeout value triggers the election?
2. Which node becomes candidate?
3. How many votes are needed?
4. What happens if Node 2 and Node 3 time out simultaneously?
5. What prevents Node 3 (which has a stale log) from winning?

**Answer key:**
1. The first follower to reach its randomized timeout, between 150ms and 300ms.
2. Whichever follower times out first.
3. 2 votes out of 3 (majority of 3 = 2).
4. Both become candidates and send RequestVote. Each receives one vote (self). Neither reaches 2. Split vote. Both wait for a second timeout round. The one with the shorter timeout wins the next round.
5. The log completeness check: Node 3's lastLogTerm or lastLogIndex is lower than the receiving node's. The voter rejects the RequestVote.

### Drill 2: Design for Multi-Region

Design a leader election system for a global application with regions in us-east, eu-west, and ap-southeast. Each region has 3 nodes. Writes must be strongly consistent within a region but can be eventually consistent across regions.

Consider:
- How does each region elect its own leader?
- What happens when cross-region latency spikes to 300ms?
- How does a global configuration change propagate?

**Answer key:**
- Each region runs Raft independently with 3 nodes. Each region elects its own leader.
- Regional leaders are unaffected by cross-region latency spikes because they do not depend on cross-region heartbeats.
- Global configuration changes are written to a designated primary region. The primary region replicats the change asynchronously to other regions. Regional leaders apply the change when they receive it.

### Drill 3: Diagnose Flapping

A 5-node etcd cluster shows leader changes every 30-60 seconds. The cluster is deployed on m5.large instances in us-east-1. The etcd data directory is on EBS gp2 volumes.

Diagnose the root cause. Propose at least 3 potential causes and their fixes.

**Answer key:**
- **Cause 1:** EBS gp2 volume throttling after sustained writes. The volume's burst credits are exhausted, causing disk I/O to throttle to baseline IOPS (for gp2, baseline = volume size in GB × 3). Fix: Increase volume size or switch to gp3 with provisioned IOPS.
- **Cause 2:** CPU steal time from a noisy neighbor. The m5.large instance shares a hypervisor with another instance that is consuming CPU. Fix: Use dedicated instances or a larger instance size.
- **Cause 3:** Network jitter from the hypervisor. Fix: Increase the election timeout from 1000ms to 2000ms. This is a quick mitigation while investigating the root cause.

### Drill 4: Compare Algorithms

Create a comparison table for Bully, Ring, and Raft leader election:

| Aspect | Bully | Ring | Raft |
|---|---|---|---|
| Messages | O(N^2) worst case | O(N) | O(N) |
| Leader selection | Highest ID | Highest ID in token | First to timeout + log check |
| Split vote handling | Not applicable (deterministic) | Not applicable (deterministic) | Randomized timeout + retry |
| Log safety | None | None | Log completeness guarantee |
| Network topology | Any | Logical ring | Any |
| Complexity | Low | Low | Medium |

### Drill 5: Design a Health-Based Leader

Design a leader election algorithm where the leader is selected based on health metrics (CPU, memory, disk latency, queue depth) rather than ID or timeout.

Constraints:
- Nodes should elect the most capable node as leader.
- If the leader becomes unhealthy, it should step down voluntarily.
- The cluster must survive the simultaneous failure of 2 out of 5 nodes.

**Answer key:**
- Each node periodically broadcasts its health score (0.0 to 1.0) to all other nodes.
- The node with the highest health score becomes the leader.
- If the leader's health score drops below 0.5, it sends a step-down message and triggers a new election.
- If a node does not receive a health broadcast from the leader within 2 broadcast intervals, it assumes the leader is dead and starts a new election.
- The health score is computed as a weighted average: 0.4 × (1 - CPU utilization) + 0.3 × (1 - memory pressure) + 0.2 × (1 - disk latency percentile) + 0.1 × (1 - queue depth ratio).

## Common System Design Questions

### Design a Distributed Key-Value Store

**Leader election component:** Use Raft with 5 nodes. Leader handles all writes. Followers serve read-after-write queries with leader-forward for strong consistency.

**Key decisions:**
- Cluster size: 5 nodes for fault tolerance (tolerates 2 failures).
- Election timeout: 200ms base, 150ms randomization spread.
- Heartbeat interval: 50ms.
- Leader lease: 500ms with quorum-based renewal.
- Write path: Client → Leader → Log append → Replicate to followers → Commit → Apply → Respond.
- Read path: Client → Leader (or follower with safety check) → Local read → Respond.

### Design a Distributed Task Scheduler

**Leader election component:** The leader schedules tasks and distributes them to worker nodes. Use ZooKeeper ephemeral znodes for leader election. Workers create ephemeral znodes and the scheduler detects their availability through watches.

**Key decisions:**
- Coordination: ZooKeeper with 3-node ensemble.
- Election: Ephemeral sequential znodes under /scheduler/election.
- Worker discovery: Workers create /workers/worker-N (ephemeral).
- Leader responsibilities: Task queue management, worker assignment, health monitoring.
- Fallback: If the leader fails, the node with the next-smallest znode takes over within one session timeout.

### Design a Global Configuration Service

**Leader election component:** etcd cluster with Raft leader election. Each region has a local etcd cluster for regional configuration. A global etcd cluster acts as the source of truth for cross-region configuration.

**Key decisions:**
- Regional clusters: 3-node etcd per region, 100ms heartbeat, 1000ms election timeout.
- Global cluster: 5-node etcd across 3 regions, 500ms heartbeat, 5000ms election timeout.
- Propagation: Global cluster pushes configuration changes to regional clusters via watch notifications.
- Consistency: Strong consistency within each region (etcd guarantees this). Eventual consistency across regions.
- Failure handling: If a regional cluster loses its leader, it elects a new one locally. The global cluster is unaffected.

### Design an Auto-Scaling Group Coordinator

**Leader election component:** The leader decides when to scale up or down based on aggregated metrics. Other nodes act as hot standbys. Use a leader lease with 5-second duration to prevent flapping during rapid metric changes.

**Key decisions:**
- Election: Raft-lite with 3 nodes. No log replication needed (the leader's decision is ephemeral).
- Lease: 5-second lease renewal. The leader must send a heartbeat every 2 seconds.
- Metric aggregation: The leader collects metrics from all nodes every 10 seconds.
- Scaling decision: If average CPU > 70% for 3 consecutive intervals, scale up by 2 instances.
- Crash recovery: If the leader crashes, a new leader is elected within 5 seconds. The new leader uses the last known metrics to make a decision.
