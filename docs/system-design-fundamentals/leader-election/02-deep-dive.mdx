# Leader Election — Under the Hood

Trace the internal machinery of leader election algorithms: state transitions, message formats, correctness proofs, and failure recovery.

---

## Internal Architecture

Every leader election algorithm implements a state machine with three roles. The internal state of each node tracks the current term, the voted-for candidate, and the election timeout.

The persistent state is the key to Raft's safety guarantees. The `currentTerm` and `votedFor` fields are written to stable storage (disk) before any node transitions to candidate state or grants a vote. This ensures that after a crash and restart, a node remembers which term it was in and whom it voted for. Without this persistence, a restarted node could vote for two different candidates in the same term, violating the safety guarantee.

## Algorithms

### Bully Algorithm — Detailed Message Flow

The Bully algorithm operates on the principle that the node with the highest ID should be the coordinator. The message flow involves four message types: ELECTION, OK, COORDINATOR, and WORK.

The Bully algorithm has O(N^2) message complexity in the worst case. When a node with the lowest ID initiates an election, it sends ELECTION messages to all N-1 other nodes. Each of those nodes, if still alive, responds with OK and initiates its own election to higher-ID nodes. In a cluster of 100 nodes, Node 1 would send 99 messages, Node 2 would send 98, and so on, totaling roughly N(N-1)/2 messages.

Despite this high message count, the Bully algorithm is simple to implement and understand. For small clusters (3-5 nodes), the message overhead is negligible.

#### Complexity and Correctness Analysis

**Message complexity:** O(N^2) in the worst case, O(N) in the best case. The worst case occurs when the lowest-ID node initiates the election. The best case occurs when the second-highest-ID node initiates, requiring only one round.

**Time complexity:** O(N) rounds. Each escalation to a higher-ID node adds one round. With N nodes, at most N rounds are needed.

**Correctness guarantee:** The Bully algorithm guarantees that the highest-ID node in the cluster is always elected. If the highest-ID node is alive, no other node can become leader. If the highest-ID node is dead, the second-highest-ID node becomes leader. This monotonic property guarantees convergence.

**Liveness concern:** If a node with a high ID repeatedly crashes and restarts, it can trigger back-to-back elections. A node that crashes after sending OK but before completing the election stalls the process until its timeout fires.

**Safety guarantee (termination):** The algorithm terminates because each election round reduces the set of candidate nodes. The highest-ID node in the reachable set will eventually declare itself coordinator because no higher-ID node exists to challenge it.

#### Raft Election Rules

A Raft leader election follows these strict rules:

1. **Term monotonicity:** Each term has at most one leader. A candidate increments the current term before starting an election.
2. **Majority vote:** A candidate must receive votes from a majority of the cluster to become leader.
3. **Log completeness:** A candidate's log must be at least as up-to-date as a majority of nodes to be elected, ensuring the leader has all committed log entries.
4. **One vote per term per node:** Each node can vote for at most one candidate per term. The votedFor field is persisted to stable storage.
5. **Heartbeat suppression:** Followers reset their election timer each time they receive a valid AppendEntries RPC from the leader, preventing unnecessary elections.

#### Raft Safety Proof

Raft guarantees that a leader contains all committed log entries. The proof relies on the interaction between the Log Completeness Property and the Election Restriction:

- **Log Completeness Property:** If a log entry is committed in a given term, that entry is present in the logs of a majority of servers.
- **Election Restriction:** A candidate must have all committed entries to be elected. Since committed entries are on a majority of servers, and the candidate must contact a majority of servers to receive votes, at least one of those servers has the committed entry. The server with the committed entry only votes for the candidate if the candidate's log is at least as up-to-date, which means the candidate must have the committed entry.
- **By induction:** When the new leader appends entries, all previous committed entries are already in its log. Therefore no committed entry is ever overwritten.

The formal proof is structured as follows. Let C be the set of servers that committed entry E in term T. C is a majority of the cluster. When the next leader L is elected in term U (U > T), L must receive votes from a majority of servers D. Since C and D are both majorities, they must overlap: there exists at least one server S that is in both C and D. Server S has entry E in its log (because S was in C). Server S only voted for L if L's log was at least as up-to-date as S's log. Therefore L has entry E. By induction, all committed entries survive across all future terms.

## Consistency and State

### Split Vote Scenario

A split vote occurs when multiple candidates each request votes simultaneously and neither achieves a majority. This is most likely when election timeouts are poorly randomized or when nodes restart simultaneously.

The split vote probability can be calculated. For N nodes with uniform random timeouts in [Tmin, Tmax], the probability that the first timeout is unique is approximately (N * (1/N)) = 1/e for large N, or roughly 37%. This means that 37% of elections start with a simultaneous timeout. However, the split vote does not always result in a failed election — one candidate may still receive its RequestVote and responses faster than the other. The actual split vote failure rate depends on network latency and processing speed.

### Network Partition — Split-Brain Scenario

In a network partition, the cluster splits into two groups that cannot communicate with each other. Each group may elect its own leader. This creates a split-brain condition where two leaders exist simultaneously.

Raft handles this through the majority requirement. Only the partition containing a majority of nodes (3 out of 5, for example) can elect a leader. The minority partition (2 out of 5) cannot reach quorum and cannot elect a leader. This prevents true split-brain in Raft.

### Old Leader Reconnects

When a partitioned old leader reconnects to the cluster, it must recognize the new leader and step down. This is enforced by term numbers and heartbeat messages.

The old leader may still believe it is the leader if it was in a partition with a majority. However, the old leader's term is now stale. The new leader will have a higher term. When the old leader receives a heartbeat from the new leader with a higher term, it steps down.

If the old leader was in the minority partition, it could not have committed any writes. Its uncommitted writes are discarded. Any clients that received acknowledgments from the old leader during the partition will see their writes rolled back. This is why clients must be designed to handle write failures and retries.

### Leader Crash During Term

If the leader crashes mid-term before replicating its last entry to a majority of nodes, the next leader will not have that entry. The entry is lost, which is correct behavior — only committed entries (replicated to a majority) survive.

This means clients that receive an acknowledgment from the leader for an uncommitted write may see that write disappear after a leader crash. Strongly consistent systems require the leader to wait for the write to be committed to a majority before acknowledging, but this adds one round trip of latency. Some systems offer a trade-off: fast acknowledgment with the risk of lost writes, or safe acknowledgment with higher latency.

## Performance

Leader election performance is measured in three dimensions: election latency, convergence under contention, and steady-state overhead.

### Election Latency

In a typical Raft cluster with 5 nodes and election timeouts of 150-300ms, election completes in 50-200ms from the timeout expiry. The breakdown is approximately:

- **Timeout wait:** 150-300ms (randomized, average 225ms)
- **RPC round-trip:** 1-10ms per message exchange
- **Log equality check:** 0.01-0.1ms (in-memory comparison)
- **Persist vote:** 0.1-1ms (fsync to disk)
- **Total from leader crash:** 200-500ms

### Convergence Under Contention

When multiple nodes become candidates simultaneously, split votes delay convergence. The expected delay for each split vote is one timeout round (150-300ms). The probability of K consecutive split votes decreases exponentially with the quality of timeout randomization.

For a well-randomized system (spread of 150ms or more), the probability of a single split vote is approximately 5-10%. The probability of two consecutive split votes is 0.25-1%. In practice, most elections complete in one round, and the expected downtime is approximately 250ms.

### Heartbeat Overhead

Heartbeat messages consume minimal bandwidth. Each heartbeat is an AppendEntries RPC with no entries, approximately 100 bytes per message. For a 5-node cluster with 50ms heartbeat intervals:

- **Messages per second:** 5 leaders × 4 followers × 20 heartbeats/sec = 80 messages/sec
- **Bandwidth:** 80 messages × 100 bytes = 8 KB/sec
- **Annual bandwidth:** ~250 GB (trivial for any modern network)

## Configuration Parameters

| Parameter | Typical Range | Recommended | Effect |
|---|---|---|---|
| Election timeout | 150-300ms | 200ms | Lower = faster failover, higher = more stability |
| Heartbeat interval | 10-100ms | 50ms | Lower = faster discovery, higher = less overhead |
| Leader lease | 300-1000ms | 500ms | Lower = faster handoff, higher = more tolerance for latency |
| Pre-vote enabled | true/false | true | Prevents unnecessary elections from stale candidates |
| Election timeout randomization | 100-500ms range | 150ms spread | Wider = fewer split votes, too wide = slow elections |

### Election Timeout Configuration Guidelines

- **Set the base timeout at least 3x the heartbeat interval** to prevent spurious elections from delayed heartbeats.
- **Use a randomization spread of at least 150ms** to ensure a single node typically times out first.
- **For high-latency networks** (e.g., cross-region), increase the timeout to 500-1000ms to accommodate packet loss and retransmission.
- **For mission-critical systems** where downtime is expensive, use 100ms timeout with fast failure detection.

### Pre-Vote Configuration

Pre-vote is an extension to the standard Raft protocol that prevents a partitioned node from disrupting the cluster when it reconnects. When pre-vote is enabled:

1. A candidate first sends a PreVote request with the next term (not incrementing the actual term).
2. Other nodes respond based on whether they would vote for this candidate.
3. Only if a majority of pre-votes is received does the candidate increment the term and send real RequestVote RPCs.

This prevents a node that was partitioned (and thus has a stale log) from forcing a term increment and election when it rejoins, which would disrupt the existing leader unnecessarily.
