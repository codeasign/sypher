# OIDC — Challenges

Work through these challenges tier by tier. Each challenge gives you a time limit, evaluation criteria, common mistakes, and a model answer. Compare your own reasoning to the model answer, then assess where you need more practice.

---

## Tier 1 — Foundation

Three challenges that test your grasp of core OIDC concepts. You should be able to articulate each one clearly in under 10 minutes.

### Challenge 1: ID Token vs Access Token

**Time limit:** 10 minutes

Explain the difference between the ID Token and the Access Token in OIDC. Cover what each contains, who the intended audience is, who should be able to read each token, and why confusing them creates security vulnerabilities.

**Evaluation criteria:**
- You correctly identify the ID Token as an authentication token (JWT consumed by the client) and the Access Token as an authorization credential (consumed by the resource server)
- You state that the ID Token contains identity claims (sub, iss, aud, exp, nonce, name, email) and the client must validate its signature, issuer, audience, and expiration
- You state that the Access Token conveys authorization grants and the client must treat it as opaque — it must not inspect its payload
- You explain that an ID Token misused for API access will fail audience validation, and an Access Token misused for UI identity display may contain no identity claims

**Common mistakes:**
- Claiming the Access Token is always a JWT (it can be opaque, especially in production IdPs)
- Saying the ID Token authenticates the user to the resource server (it authenticates to the client only)
- Assuming the Access Token audience is the client (it is the resource server)
- Overlooking the `at_hash` claim that binds the ID Token to the Access Token

**Model answer:**

The ID Token is a JSON Web Token issued by the OpenID Provider and consumed by the client (relying party). It proves that the end-user has been authenticated and carries identity claims: `sub` (subject identifier), `iss` (issuer), `aud` (audience — the client_id), `exp` (expiration), `iat` (issued at), and optionally `nonce`, `auth_time`, `acr`, `at_hash`. The client MUST validate the signature (using the OP's JWKS), verify `iss` matches the expected issuer, verify `aud` contains its own client_id, and check `exp` has not passed. The ID Token is a **security token** — it tells the client who the user is.

The Access Token is a credential that the client presents to the resource server (the API) to gain access to protected resources. The client should treat the Access Token as opaque — it must not inspect its payload. The resource server validates the Access Token independently, either by calling the OP's introspection endpoint or by validating its own JWT structure if it shares a signing key with the OP. The Access Token is an **authorization credential** — it tells the resource server what the client may do on behalf of the user.

The critical distinction is audience. The ID Token is for the client. The Access Token is for the API. If a resource server accepts an ID Token as authorization, an attacker who steals a client-side ID Token can impersonate the user on every API call. If a client parses the Access Token payload for UI display, and the IdP switches to opaque tokens, the client breaks silently.

### Challenge 2: OIDC Discovery

**Time limit:** 10 minutes

Explain OIDC Discovery. What is the `.well-known/openid-configuration` endpoint? What does it return? How does a client use it? Why is it essential for dynamic client registration and multi-tenant deployments?

**Evaluation criteria:**
- You state the standard URL format: `https://{provider}/.well-known/openid-configuration`
- You list at least five key fields: `issuer`, `authorization_endpoint`, `token_endpoint`, `userinfo_endpoint`, `jwks_uri`, `id_token_signing_alg_values_supported`, `response_types_supported`, `scopes_supported`, `claims_supported`
- You explain how the client uses the discovery document to dynamically configure itself without hardcoded endpoints
- You note that the `issuer` field must be verified against the expected issuer to prevent issuer confusion attacks
- You explain that the `jwks_uri` tells the client where to fetch public keys for ID Token signature verification

**Common mistakes:**
- Forgetting that the discovery document must be fetched over HTTPS and the client must validate the TLS certificate
- Assuming the discovery document is static — it can change (rotated keys, new endpoint URLs), so clients should re-fetch periodically
- Not verifying the `issuer` claim in the discovery document matches the expected issuer
- Confusing OIDC Discovery with WebFinger (which discovers the IdP from an email address)

**Model answer:**

OIDC Discovery is defined in OpenID Connect Discovery 1.0. The OpenID Provider publishes a JSON document at a well-known URL: `https://op.example.com/.well-known/openid-configuration`. A client, knowing only the issuer URL, fetches this document to learn all the provider's endpoints, supported features, and cryptographic parameters.

A typical response includes:

```json
{
  "issuer": "https://op.example.com",
  "authorization_endpoint": "https://op.example.com/authorize",
  "token_endpoint": "https://op.example.com/token",
  "userinfo_endpoint": "https://op.example.com/userinfo",
  "jwks_uri": "https://op.example.com/jwks",
  "scopes_supported": ["openid", "profile", "email"],
  "response_types_supported": ["code", "code id_token"],
  "id_token_signing_alg_values_supported": ["RS256", "ES256"],
  "subject_types_supported": ["public", "pairwise"],
  "claims_supported": ["sub", "iss", "name", "email", "picture"]
}
```

The client uses this document to dynamically configure its OIDC integration. Instead of hardcoding endpoint URLs, the client fetches the discovery document at startup and uses the declared endpoints. This is essential in multi-tenant deployments: each tenant may have a different issuer URL, and the client discovers the correct endpoints per tenant. It also enables key rotation — the provider updates the `jwks_uri` contents, the client re-fetches when signature validation fails, and the rotation completes without downtime.

The client must validate that the `issuer` in the response matches the expected issuer value. If an attacker redirects the client to a malicious discovery document, a mismatched issuer alerts the client to the attack.

### Challenge 3: The Nonce

**Time limit:** 10 minutes

Explain the nonce parameter in OIDC. What problem does it solve? How is it generated, included in the authentication request, and validated in the ID Token? When is it optional versus required?

**Evaluation criteria:**
- You identify the nonce as a defense against ID Token replay attacks
- You explain that the nonce is a cryptographically random value generated per authentication request, per session
- You describe the full flow: client generates nonce, includes it in the authorize request, the OP includes it in the ID Token, and the client validates the match
- You state the nonce is required for implicit flow and optional but strongly recommended for authorization code flow
- You explain that an attacker cannot replay a captured ID Token because the nonce will not match the current session

**Common mistakes:**
- Confusing nonce with the CSRF `state` parameter (state protects the authorization code exchange; nonce binds the ID Token to the request)
- Saying the nonce is stored in a cookie or local storage without specifying the client-side session store
- Assuming the nonce makes the ID Token one-time-use (the ID Token still has an expiration; the nonce prevents cross-session replay, not within-session replay)
- Forgetting that the nonce must be checked against a single-use store, not just verified for presence

**Model answer:**

The nonce is a single-use, cryptographically random string generated by the client and bound to the ID Token. It solves ID Token replay: an attacker who intercepts an ID Token from one session cannot replay it to a different client session because the nonce will not match.

The flow:

1. The client generates a nonce: `nonce = crypto.randomUUID()` or 16+ bytes from a CSPRNG, base64url-encoded.
2. The client stores the nonce in session-scoped memory (server-side session or `sessionStorage` in a browser) associated with this specific authentication request.
3. The client includes the nonce as a query parameter: `GET /authorize?response_type=code&client_id=...&nonce=abc123`.
4. The OP includes the nonce in the ID Token: `{ "iss": "...", "sub": "...", "nonce": "abc123", ... }`.
5. The client receives the ID Token, validates the signature, issuer, and audience, then extracts the nonce claim.
6. The client looks up the stored nonce for this session and verifies it matches exactly. It marks the nonce as consumed (single-use enforcement).
7. If the nonce does not match or is missing, the client rejects the authentication response.

The nonce is **required** for the implicit flow (`response_type=id_token`) because the ID Token is returned directly in the front channel without an authorization code exchange that provides a separate binding. It is **optional but strongly recommended** for the authorization code flow, where the code-to-token exchange provides an additional layer of binding. Even so, using a nonce in the authorization code flow provides defense-in-depth against token substitution attacks.

## Tier 2 — Design

Three challenges that require you to design a complete OIDC integration for a specific client type. Allow 15 minutes each.

### Challenge 4: OIDC for a Web Application

**Time limit:** 15 minutes

Design an OIDC integration for a traditional server-side web application (Django, Rails, Spring Boot). The app needs user authentication through an external OpenID Provider. Users must sign in, access protected pages, and sign out. Sessions must survive server restarts. The app has 50,000 daily active users.

**Scale estimates:**
- 50,000 DAU, peak at 5,000 concurrent sign-ins per hour
- Each sign-in involves 3 network round trips (authorize, token, userinfo)
- Session store needs 50 MB for active sessions (assuming 1 KB per session)

**Key decisions:**
- Which OIDC flow to use
- Where ID Token validation occurs
- Whether to call the UserInfo endpoint
- Session management strategy
- Logout mechanism

**Model answer:**

**Flow choice:** Authorization Code Flow with `response_type=code` and the `openid` scope. This is the recommended flow for server-side web apps. The server has a secure backchannel where it can store the `client_secret` and exchange the authorization code for tokens. The `state` parameter prevents CSRF on the code exchange.

**ID Token validation:** The server validates the ID Token immediately after receiving it from the token endpoint. Steps:
1. Verify the JWT signature using the OP's public keys fetched from `jwks_uri` (with caching, re-fetching on unknown `kid`).
2. Verify `iss` matches the expected issuer.
3. Verify `aud` contains the client's `client_id`.
4. Verify `exp` has not passed.
5. Verify `nonce` if included in the original request.
6. Verify `azp` (authorized party) if `aud` has multiple values.

**UserInfo endpoint:** Call the UserInfo endpoint with the Access Token in the `Authorization: Bearer` header to fetch fresh user claims. This is optional if the ID Token already contains all needed claims, but calling UserInfo is the standard OIDC way to obtain current profile data. The Access Token must have the `openid profile email` scopes.

**Session management:** After successful authentication, the server creates its own session (signed cookie with a server-side session store in Redis). The session cookie is `httpOnly`, `Secure`, `SameSite=Lax`. The session data survives restarts because Redis is external.

### Challenge 5: OIDC for a Mobile App

**Time limit:** 15 minutes

Design an OIDC integration for a native mobile application (iOS/Android). The app needs user authentication through an external OpenID Provider. Users should authenticate once and remain signed in across app restarts. The app handles sensitive financial data and must use biometric unlock. The app has 1 million installs with 200,000 monthly active users.

**Scale estimates:**
- 200,000 MAU, 20,000 DAU
- Peak sign-in rate: 100 requests per second during morning commute
- Token refresh: each user refreshes on average 2 times per day
- Secure storage: 2 KB per user (refresh token + metadata)

**Key decisions:**
- Why PKCE is mandatory for mobile
- How the app opens the browser for authentication (ASWebAuthenticationSession / Chrome Custom Tabs)
- Where tokens are stored and how biometric unlock works
- Refresh token rotation policy
- Handling app kill and restore

**Model answer:**

**Flow choice:** Authorization Code Flow with PKCE. PKCE is mandatory for mobile because the app cannot securely store a `client_secret`. The app generates a `code_verifier` (high-entropy random string, 43-128 characters) and derives a `code_challenge` = `base64url(sha256(code_verifier))`. The `code_challenge` is sent in the authorize request, and the raw `code_verifier` is sent in the token request. The OP verifies the challenge matches, proving the same app that started the flow is completing it.

**System browser, not WebView:** The app uses `ASWebAuthenticationSession` (iOS) or Chrome Custom Tabs (Android). This runs in the system browser, not an embedded WebView. An embedded WebView can be manipulated by the app, and the user cannot verify the URL. The system browser shares cookies with the browser the user trusts, enabling seamless SSO.

**Token storage:** Store tokens in the OS keychain:
- iOS Keychain with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` and `kSecAccessControlBiometryCurrentSet`
- Android EncryptedSharedPreferences backed by Android KeyStore with `setUserAuthenticationRequired(true)`

This means the refresh token is encrypted at rest and requires biometric unlock before the app can read it.

**Biometric unlock flow:**
1. After initial sign-in, store the refresh token with biometric protection in the keychain.
2. On app foreground, check if the Access Token is expired. If yes, trigger biometric prompt.
3. Biometric success decrypts the refresh token.
4. Perform a refresh token grant to obtain a new Access Token.
5. After an idle timeout, re-lock the keychain item.

**Refresh token rotation:** The OP issues a new refresh token with each refresh and invalidates the old one. If a stolen refresh token is used after the legitimate client has rotated, the OP detects the fraud and revokes all tokens for that session. This is token rotation with reuse detection from RFC 6819.

**App kill and restore:** The refresh token survives app kill because it is in the OS keychain. On app relaunch, the app checks for an existing refresh token, prompts biometric unlock, and requests new tokens.

### Challenge 6: OIDC for a Single-Page Application

**Time limit:** 15 minutes

Design an OIDC integration for a browser-based SPA (React, Vue, Angular). The SPA calls a backend API with user context. The SPA must handle silent token refresh and logout across multiple browser tabs. The application has no server-side backend except the API layer.

**Scale estimates:**
- 100,000 DAU, peak 10,000 concurrent authenticated sessions
- Average session length: 30 minutes
- Refresh token usage: each session refreshes every 5 minutes (Access Token lifetime)
- Token storage: ephemeral only

**Key decisions:**
- Why the implicit flow is deprecated and you must use authorization code flow with PKCE
- Where to store tokens (in-memory vs storage)
- How to handle refresh tokens in a browser (with appropriate security restrictions)
- Cross-tab logout synchronization
- Handling the redirect back to the SPA after authentication

**Model answer:**

**Flow choice:** Authorization Code Flow with PKCE. The implicit flow (`response_type=id_token token`) is deprecated by OAuth 2.0 Security BCP (RFC 9700) because the Access Token is exposed in the browser URL fragment, which appears in browser history, referer headers, and can leak through extensions. With PKCE and authorization code flow, the code is single-use, short-lived, and exchanged for tokens on the OP's token endpoint, keeping tokens out of the URL.

The SPA does not have a `client_secret` because it cannot keep one secret. The PKCE `code_verifier` serves as proof of possession for the code exchange.

**Token storage:** Store tokens in JavaScript memory (a closure or module variable), not in `localStorage` or `sessionStorage`. Neither storage API is accessible from JavaScript, which means any XSS vulnerability leaks tokens stored there. In-memory storage means tokens are lost on full page reload, but this is an acceptable trade-off: on reload, the SPA redirects to the OP for a new silent authentication, which succeeds without user interaction if the OP session is still active.

**Refresh tokens in SPAs:** The refresh token, if used, is stored in an httpOnly, SameSite=Strict, Secure cookie set by a lightweight backend-for-frontend (BFF) proxy. The SPA never reads the refresh token from JavaScript. The BFF performs the token refresh on behalf of the SPA and sets a new httpOnly cookie with the fresh Access Token. Alternatively, the OP issues a very short-lived Access Token (5 minutes) and the SPA re-authenticates silently via `prompt=none` in the authorization request.

**Cross-tab logout:** Use the `storage` event for cross-tab synchronization:
- Tab A performs logout, clears in-memory tokens, sets a logout flag in `sessionStorage`.
- Tab B hears the `sessionStorage` change via the `storage` event, clears its own tokens, and redirects to the login page.
- Alternatively, use the OP's session management iframe with `postMessage` polling to detect when the OP session ends.

**Redirect handling:** After the OP redirects back to the SPA with the authorization code, the SPA extracts the code from the URL query string and immediately exchanges it for tokens via `fetch` to the OP's token endpoint. After the exchange, the SPA uses `window.history.replaceState` to remove the code from the URL bar without triggering a page reload.

## Tier 3 — Expert

Two challenges that simulate full system design interviews. Allow 30 minutes each.

### Challenge 7: OIDC Provider for 100 Million Users

**Time limit:** 30 minutes

Design an OpenID Connect provider that serves 100 million users across 10,000 tenant organizations. The provider must support dynamic client registration, OIDC Discovery per tenant, key rotation without downtime, session management, and consent handling. The system must be globally available with P99 latency under 200 ms for the authorize and token endpoints.

**Dimensions to explore:**
- Multi-tenant architecture: shared vs siloed infrastructure per tenant
- Issuer per tenant vs shared issuer with tenant routing
- Key management: how signing keys are generated, stored, rotated, and distributed
- JWKS endpoint design: caching, region-awareness
- Session store: scale, sharding, consistency model
- Consent management: per-client vs per-tenant consent
- Availability and durability of the token issuance path

**Anti-patterns to avoid:**
- Single shared signing key for all tenants (a breach exposes all tenants)
- Synchronous key generation at startup (slows boot and can fail)
- Storing full session state in a single-region database
- Allowing unlimited concurrent sessions per user without configuration
- Leaking tenant identity through timing side-channels in the authentication endpoint

**Model answer — multiple valid approaches:**

There are two valid architectural approaches for the tenant model.

**Approach A: Shared issuer, tenant routed by client_id.**

Each client registration includes the tenant. The provider routes by extracting a tenant prefix from the `client_id`. The discovery document is at a single `https://oidc.example.com/.well-known/openid-configuration`. The `issuer` is always the same. Tenant-specific configuration lives in a tenant config table keyed by `client_id`.

Pros: Simpler DNS, single TLS certificate, simpler caching. Cons: Leaked `client_id` reveals tenant; cannot customize discovery document per tenant; all tenants share the same issuer, so pairwise subject identifiers must be tenant-aware.

**Approach B: Per-tenant subdomain and issuer URL.**

Discovery is at `https://{tenant}.oidc.example.com/.well-known/openid-configuration`. The `issuer` includes the tenant subdomain. The provider uses a wildcard TLS certificate. Tenant-specific JWKS endpoints at `https://{tenant}.oidc.example.com/jwks` with per-tenant signing keys.

Pros: Strong tenant isolation; pairwise subject identifiers are trivially scoped; each tenant can enable or disable features independently; compliance boundaries per tenant. Cons: More complex DNS and TLS management; caching layer must be tenant-aware.

**Key management:**

Each tenant has its own signing key pair, rotated every 90 days with a 30-day overlap. The old key remains in the JWKS for verification of tokens issued under the old key, but new tokens are signed with the new key.

```
Key lifecycle:
  1. Generate: HSM or cloud KMS (AWS KMS, Azure Key Vault)
     - Key type: RSA 2048-bit (RS256) or EC P-256 (ES256)
     - Stored in HSM, never in plaintext on disk
  2. Publish: key_id (kid) added to tenant JWKS
     - JWKS cached at CDN edge (60-second TTL)
     - Clients discover new key when they see an unknown kid
  3. Active: 90-day signing window
     - All new ID Tokens signed with the active key
     - Previous key remains in JWKS for validation
  4. Retire: 30 days after rotation
     - Removed from JWKS
     - Tokens signed with retired key rejected
```

**Session management:**

Use a distributed session store on a high-performance KV store. Each session record:

```json
{
  "session_id": "sess_abc123",
  "user_id": "user_456",
  "tenant_id": "tenant_789",
  "client_id": "client_app01",
  "auth_time": 1740000000,
  "expires_at": 1740086400,
  "scopes": ["openid", "profile"],
  "consent_granted": true
}
```

Store sharded by `user_id % N` across 64 shards. Each shard is a Redis cluster with read replicas in each region. Write quorum to the primary in the home region; read from local replica with eventual consistency.

**Consent handling:**

Three consent modes:

| Mode | Behavior | Use Case |
|------|----------|----------|
| Automatic | No consent screen, trust established by tenant admin | Enterprise SSO, internal apps |
| Per-client | Consent screen shown once per client, then remembered | Third-party app integrations |
| Per-request | Consent screen shown on every authorization | High-security, financial data |

Consent records are stored per user-client pair, keyed by `(tenant_id, user_id, client_id)`, with a timestamp and scope snapshot. When requested scopes change, the consent is invalidated and re-prompted.

**Global availability:**

Each region has a full replica of the token-signing infrastructure. The authorize endpoint hits the user's home region (routed by geo-DNS). The token endpoint hits any region. If the authorization code was issued by another region, the code is stored in a cross-region replicated store (DynamoDB Global Tables, Cassandra with multi-region replication). Set the code lifetime to 60 seconds; typical replication lag is under 1 second within the same continent.

### Challenge 8: OIDC Federation Across Multiple IdPs

**Time limit:** 30 minutes

Design a system that allows users to authenticate through any of multiple external OpenID Providers (Google, Microsoft, GitHub, a corporate AD FS) and access a single application. The application must present a unified user identity regardless of which IdP the user chose. Users who have accounts on multiple IdPs must be recognized as the same person. The system must handle account linking and claim mapping. Conflicts between claims from different IdPs must be resolved.

**Dimensions to explore:**
- Broker or proxy pattern: the application delegates to a central identity broker
- Account linking: deterministic (email-based) vs probabilistic (behavioral signals)
- Claim mapping: how to normalize different IdPs' claim sets into a canonical schema
- Conflict resolution: what happens when two IdPs give different values for the same claim
- Session management across IdP sessions: simultaneous vs single-session
- Handling IdP outages

**Anti-patterns to avoid:**
- Requiring the user to manually link accounts without clear guidance
- Storing the IdP's internal `sub` as the primary user identifier (breaks if the IdP changes issuer)
- Assuming all IdPs support the same scopes and claims
- Silent linking by email without user consent (spoofable, opens account takeover vectors)
- Keeping the user's session alive against a failed IdP without re-authentication

**Model answer — discussion of multiple valid approaches:**

**Broker architecture:**

Place a central identity broker between the application and all external IdPs. The broker speaks OIDC to each IdP (as a client) and speaks OIDC to the application (as a provider). The application only knows about the broker's discovery endpoint and issuer.

```
User ---> Application ---> Broker ---> Google OIDC
                                |
                                +---> Microsoft OIDC
                                |
                                +---> GitHub OIDC
                                |
                                +---> Corporate AD FS
```

The broker maintains a database that maps external IdP identities to a canonical user:

```json
{
  "canonical_user_id": "cu_abc123",
  "identities": [
    { "provider": "google", "sub": "goog_12345", "email": "alice@gmail.com" },
    { "provider": "microsoft", "sub": "ms_67890", "email": "alice@contoso.com" }
  ],
  "canonical_claims": {
    "name": "Alice Johnson",
    "email": "alice@contoso.com",
    "picture": "https://google.com/picture/alice.jpg"
  }
}
```

**Account linking approaches:**

**Deterministic (email-based):** If the user authenticates with Google (email: `alice@gmail.com`) and later with Microsoft (email: `alice@contoso.com`), these are different emails and the broker treats them as separate accounts. If the user authenticates with Google (email: `alice@work.com`) and then with corporate AD FS (email: `alice@work.com`), the broker can ask: "We found an existing account with this email. Do you want to link them?" The user confirms by re-authenticating with both IdPs in a single session.

**User-initiated linking:** The user signs in with IdP A, goes to account settings, clicks "Link another account," authenticates with IdP B, and the broker records the link. This is the most reliable approach.

**Probabilistic linking (advanced):** For large-scale systems, the broker may compute similarity scores based on email, name, profile picture, and IP address. Linking is proposed but not automatic. False positives are user-resolvable from account settings.

**Claim mapping:**

Each IdP returns a different claim structure. The broker normalizes to a canonical schema:

| Canonical Claim | Google | Microsoft | GitHub |
|-----------------|--------|-----------|--------|
| sub | sub | sub | id (integer) |
| name | name | displayName | login |
| email | email | email | email (may be null) |
| picture | picture | (not standard) | avatar_url |
| locale | locale | preferredLanguage | (not standard) |

**Conflict resolution strategy:**

When the same canonical claim has different values from different IdPs:

| Strategy | Description | When to use |
|----------|-------------|-------------|
| Last-write-wins | The most recently authenticated IdP's value is canonical | General purpose, simplest |
| Priority-sourced | IdPs have a configured priority; highest priority wins | Enterprise with authoritative HR IdP |
| User-selected | The user chooses which value to keep at linking time | User-facing applications |
| Union | Collect values from all IdPs; expose lowest-common-denominator | Read-only profile aggregation |

A practical approach is priority-sourced with user override: the broker marks one IdP as authoritative for specific claims (e.g., corporate AD FS is authoritative for `name` and `email`), and other IdPs' values are secondary. The user can override any claim from their profile settings.

**IdP outage handling:**

The broker monitors each IdP's health via health-check endpoints and error rates. If Google is down:
1. New sign-ins with Google are blocked with a clear error message.
2. Existing users who previously authenticated through Google and have linked another IdP can still sign in using the other IdP.
3. Active sessions authenticated through Google remain valid until the broker's session cookie expires (typically 1 hour). The broker does not attempt to refresh tokens against a failed IdP.
4. If the user only has one linked IdP and that IdP is down, there is no workaround. The broker communicates the service status and estimated resolution time.

**Security considerations:**
- The broker must validate each IdP's ID Token independently, using each IdP's JWKS URI
- The broker's ID Token to the application includes the canonical `sub` and a `provider` claim indicating which IdP was used for this session
- Account linking must require proof of control over both accounts (re-authentication of both at linking time)
- The broker must support the `max_age` parameter to force re-authentication when linking accounts
