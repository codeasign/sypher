# OIDC — Under the Hood

OpenID Connect is an identity layer on top of OAuth 2.0. While OAuth 2.0 authorizes access to resources, OIDC authenticates users and provides a verifiable identity token. This deep dive walks through every component, validation step, and performance consideration you need to design production-grade authentication systems.

---

## 1. The Big Picture

When you integrate OIDC, your application (the Relying Party, or RP) delegates authentication to an OpenID Provider (OP). The OP returns an ID Token — a signed JSON Web Token (JWT) that proves the user is who they claim to be.

## 2. The ID Token

The ID Token is the core of OIDC. It is a JWT — three base64url-encoded segments separated by dots: header, payload, and signature.

### 2.2 Required Claims

Every ID Token must contain these claims:

| Claim | Purpose | Your Validation |
|---|---|---|
| `iss` | Issuer identifier — the OP's URL | Must match your known issuer |
| `sub` | Subject — unique user identifier at the OP | Store as your user's primary key |
| `aud` | Audience — the client ID this token is for | Must include your client ID |
| `exp` | Expiration time (Unix timestamp) | Must be in the future |
| `iat` | Issued at time (Unix timestamp) | Used for clock skew window |

### 2.4 The `sub` Claim — Your User Identity Anchor

The `sub` claim is the most important value you will extract from the ID Token. It is the OP's unique identifier for the user — stable, never reassigned, and case-sensitive. You should:

- Store `sub` as your primary user identifier in the local database.
- Never rely on `email` or `name` as the user key (email addresses change).
- Use a `sub` of at least 255 characters in your database schema (spec does not guarantee a max length).

## 3. UserInfo Endpoint

The ID Token gives you a compact, signed assertion about the user. For additional claims (profile, address, phone numbers), you call the UserInfo endpoint. This is a standard OAuth 2.0 protected resource that returns a JSON body.

### 3.1 When to Use UserInfo vs ID Token Claims

| Situation | Use |
|---|---|
| You need essential identity: `sub`, `iss`, `auth_time` | ID Token |
| You need a guaranteed fresh profile on every request | UserInfo endpoint |
| You need email, email_verified status | ID Token (usually included) |
| You need address, phone, or custom profile attributes | UserInfo endpoint |
| You are offline or the access token expired | ID Token (already cached) |

### 3.2 UserInfo Response Structure

```json
{
  "sub": "user|12345",
  "name": "Jane Doe",
  "given_name": "Jane",
  "family_name": "Doe",
  "middle_name": "Elizabeth",
  "nickname": "janie",
  "preferred_username": "jane.doe",
  "profile": "https://example.com/jane",
  "picture": "https://example.com/jane.jpg",
  "website": "https://jane.dev",
  "email": "jane@example.com",
  "email_verified": true,
  "gender": "female",
  "birthdate": "1990-01-15",
  "zoneinfo": "America/New_York",
  "locale": "en-US",
  "phone_number": "+1-555-123-4567",
  "phone_number_verified": false,
  "address": {
    "street_address": "123 Main St",
    "locality": "Anytown",
    "region": "NY",
    "postal_code": "12345",
    "country": "US"
  },
  "updated_at": 1767037056
}
```

The UserInfo endpoint is an OAuth 2.0 protected resource. You call it with the Access Token (not the ID Token). The response is not individually signed — the channel (HTTPS) provides integrity.

### 3.4 UserInfo Error Handling

The UserInfo endpoint may return errors. Your app must handle each gracefully.

| HTTP Status | Meaning | Your Action |
|---|---|---|
| `200 OK` | Success | Parse and merge claims |
| `401 Unauthorized` | Access Token expired or invalid | Re-authenticate the user |
| `403 Forbidden` | Insufficient scopes | Request additional scopes on next auth |
| `429 Too Many Requests` | Rate limited | Retry with exponential backoff |
| `5xx Server Error` | Provider issue | Retry with backoff, fall back to ID Token claims |

Never crash or expose a 500 error to the user because a UserInfo call failed. Fall back gracefully: the ID Token already contains the core identity claims (`sub`, `iss`, `auth_time`). You can serve the application with those claims and queue a background retry for the profile data.

### 3.5 UserInfo Caching Strategy

Cache UserInfo responses by `sub` with a TTL that matches your application's staleness tolerance:

```
Session-first cache strategy:

  Login:
    1. Extract sub, email, name from ID Token (signed, authoritative)
    2. Call UserInfo for additional claims
    3. Store full profile in session with timestamp TTL

  Profile display (user views their profile page):
    1. Read from session cache (instant)
    2. If cache is older than profile_ttl, fetch in background

  Administrative requests (check admin status):
    1. Call UserInfo synchronously with fresh Access Token
    2. Do not rely on cached admin status for authz decisions
```

## 4. The Discovery Document

Every OIDC-compliant provider publishes a Discovery Document at a well-known URL. This is your entry point for all configuration.

### 4.1 Discovery URL

```
https://accounts.example.com/.well-known/openid-configuration
```

### 4.2 Discovery Document Structure

```json
{
  "issuer": "https://accounts.example.com",
  "authorization_endpoint": "https://accounts.example.com/oauth/authorize",
  "token_endpoint": "https://accounts.example.com/oauth/token",
  "userinfo_endpoint": "https://accounts.example.com/oauth/userinfo",
  "jwks_uri": "https://accounts.example.com/oauth/jwks",
  "registration_endpoint": "https://accounts.example.com/oauth/register",
  "scopes_supported": ["openid", "profile", "email", "address", "phone"],
  "response_types_supported": ["code", "id_token", "token id_token"],
  "response_modes_supported": ["query", "fragment", "form_post"],
  "grant_types_supported": [
    "authorization_code",
    "implicit",
    "refresh_token",
    "client_credentials"
  ],
  "acr_values_supported": [
    "urn:mace:incommon:iap:silver",
    "urn:mace:incommon:iap:bronze"
  ],
  "subject_types_supported": ["public", "pairwise"],
  "id_token_signing_alg_values_supported": ["RS256", "ES256"],
  "token_endpoint_auth_methods_supported": [
    "client_secret_basic",
    "client_secret_post",
    "private_key_jwt"
  ],
  "claims_supported": [
    "sub", "iss", "auth_time", "name",
    "email", "email_verified", "picture"
  ],
  "request_parameter_supported": true,
  "request_uri_parameter_supported": true,
  "require_request_uri_registration": false
}
```

Always fetch the Discovery Document dynamically in production. Hardcoding endpoints is a common source of integration failures when providers update their infrastructure.

### 4.1 Dynamic Provider Discovery

When your application supports multiple providers (social login, enterprise SSO), you have two approaches:

```
Multi-Provider Discovery Strategies

Strategy A: Pre-configured with validation
  1. Store issuer URLs in your app config (not the full endpoint list)
  2. Fetch .well-known/openid-configuration from each issuer
  3. Validate that the returned issuer matches the URL you requested
  4. Cache per-provider

Strategy B: User-provided issuer
  1. User enters their provider's issuer URL
  2. Your app fetches the Discovery Document
  3. Validate the issuer claim matches
  4. Present supported features to the user
  (Used by self-serve identity platforms)
```

### 4.2 Scopes and Claims Mapping

The `scope` parameter determines which claims the OP returns. The `openid` scope is required — without it, you get an OAuth 2.0 token, not an OIDC ID Token.

```json
Scope            Claims Returned
─────            ───────────────
openid           sub (always), iss, auth_time
openid profile   name, given_name, family_name, middle_name,
                 nickname, preferred_username, profile, picture,
                 website, gender, birthdate, zoneinfo, locale,
                 updated_at
openid email     email, email_verified
openid address   address (JSON object with street, locality, region,
                 postal_code, country)
openid phone     phone_number, phone_number_verified

Combined:        "openid profile email"
Returns:         sub + all profile claims + all email claims
```

### 4.3 Request Object vs Request URI

OIDC supports two ways to send the authorization request:

**Request Object (by value):**
- The entire request is a signed JWT passed as the `request` parameter.
- The OP verifies the JWT signature using a pre-registered public key.
- Use this when you need to ensure the request was not tampered with.

**Request URI (by reference):**
- You POST the request JWT to the OP and get back a `request_uri`.
- Pass the URI as the `request_uri` parameter in the auth request.
- The OP fetches the request object from the URI.
- Use this for very large requests that exceed URL length limits.

```python
# Request object JWT payload example
{
  "iss": "your-client-id",
  "aud": "https://accounts.example.com",
  "response_type": "code",
  "client_id": "your-client-id",
  "redirect_uri": "https://yourapp.com/callback",
  "scope": "openid profile email",
  "state": "xyz789",
  "nonce": "abc123",
  "max_age": 3600,
  "claims": {
    "id_token": {
      "acr": {"essential": true, "values": ["2", "3"]}
    }
  }
}
```

### Step-by-Step Breakdown

**Step 1: User Initiation** — The user clicks your "Sign In" button. Your app determines which OP to use and begins the flow.

**Step 2: Discovery Fetch** — Your app fetches `/.well-known/openid-configuration` to get all endpoint URLs and supported features. Cache this document for 24 hours and re-fetch on 401 or signature validation failures.

**Step 3: JWKS Fetch** — Your app fetches the provider's public keys from the `jwks_uri`. These keys verify the ID Token signature. Cache aggressively — rotate keys are rare but must be handled gracefully.

**Step 4: Auth Request Construction** — Your app builds the authorization URL with these parameters:
- `client_id`: your registered client identifier
- `redirect_uri`: where the OP sends the user after authentication
- `response_type`: `code` for the Authorization Code flow
- `scope`: must include `openid` (plus `profile`, `email`, etc.)
- `state`: a cryptographically random string bound to the user's session (CSRF protection)
- `nonce`: a cryptographically random string bound to this specific authentication request (replay protection)

**Step 5: Browser Redirect** — Your app sends a 302 redirect to the OP's authorization endpoint. The user's browser follows it.

**Step 6: User Authentication** — The OP authenticates the user (password, SSO, biometrics, etc.) and presents a consent screen if required.

**Step 7: Authorization Code Delivered** — The OP redirects the browser back to your `redirect_uri` with a `?code=...&state=...` query string. Your app MUST validate that the `state` matches the one from Step 4.

**Step 8: Token Exchange** — Your app sends a POST request to the OP's token endpoint with:
- `grant_type=authorization_code`
- `code`: the code from Step 7
- `redirect_uri`: must match exactly
- `client_id` and `client_secret` (or other auth method)

The OP returns an ID Token and an Access Token.

**Step 9: ID Token Validation** — This is critical. See Section 2.3 for the complete checklist.

**Step 10: UserInfo Fetch** — Your app calls the UserInfo endpoint with the Access Token to get additional claims.

**Step 11: Session Creation** — Your app creates a local session (signed cookie, server-side session store, etc.) and maps it to the `sub` claim.

**Step 12: Application Loaded** — Your app returns the authenticated application to the user.

### 5.1 Token Types in the Response

The token endpoint returns three possible token types. Know the difference.

```
Token Type        Purpose                        Lifetime
──────────        ───────                        ────────
ID Token          Prove identity (who you are)   5-60 minutes
                  Never send to other services
                  Validate locally

Access Token      Authorize API calls            15-60 minutes
                  (what you can do)              Depends on provider
                  Sent to UserInfo and your
                  own backend APIs

Refresh Token     Get new tokens without          Days to months
                  re-authenticating the user      Configurable
                  Stored securely, never sent
                  to the browser
```

### 5.2 PKCE Extension

The Authorization Code flow with PKCE (Proof Key for Code Exchange) is mandatory for public clients (mobile apps, SPAs). PKCE prevents authorization code interception attacks.

```
PKCE Flow

1. Client generates:
     code_verifier = crypto.randomBytes(32).toString('base64url')
     code_challenge = base64url(sha256(code_verifier))

2. Auth request includes:
     code_challenge=<challenge>
     code_challenge_method=S256

3. Token exchange includes:
     code_verifier=<original_verifier>

4. OP verifies:
     sha256(code_verifier) === code_challenge
```

### 5.3 Session State Management

After successful authentication, your app must manage its own session. OIDC does not define session semantics — that is your responsibility.

```
Session Architecture Options

Option A: Signed Cookie (stateless)
  - Store user claims encrypted in a JSON Web Token
  - Set as an HttpOnly, Secure, SameSite=Lax cookie
  - Validate on every request (no server-side store)
  - Trade-off: revocation is hard (no central kill switch)

Option B: Server-side Session Store (stateful)
  - Store a session ID in the cookie
  - Keep user claims in Redis / Memcached / DB
  - Look up on every request
  - Trade-off: more infrastructure, but instant revocation
  - Use for high-security applications

Option C: Hybrid
  - Store core claims (sub, email) in a signed cookie
  - Store extended profile in server-side cache
  - Fast path for most requests, fallback to cache when needed
  - Best balance of performance and control
```

## 6. Consistency Considerations

### 6.1 Clock Skew

ID Token `exp` and `iat` timestamps come from the OP's clock. Your server's clock may differ. OIDC implementations should accept a clock skew of up to 60 seconds (the standard recommendation). Do not accept more than 120 seconds — excessive skew windows enable token replay.

```
Valid if: token.iat - clock_skew <= now <= token.exp + clock_skew
```

### 6.2 Key Rotation

Providers rotate their signing keys periodically and in emergency scenarios (key compromise). Your system must handle this gracefully.

### 6.3 Nonce Replay Prevention

The `nonce` parameter prevents replay attacks. Here is how it works:

1. **Generation**: Your app generates a cryptographically random string (minimum 16 bytes, preferably 32 bytes from `crypto.randomUUID()`).
2. **Binding**: The nonce is included in the authorization request and stored in the user's session.
3. **Assertion**: The OP includes the same nonce in the ID Token.
4. **Verification**: Your app compares the nonce in the ID Token to the stored value.
5. **One-time Use**: The nonce is consumed and cannot be reused for another authentication.

Never reuse nonces. Never use predictable values (timestamps, sequential integers).

## 7. Performance Considerations

### 7.1 ID Token Verification Latency

ID Token verification involves fetching and caching JWKS, then performing a cryptographic signature verification. This is fast but not free.

| Operation | Typical Latency | Notes |
|---|---|---|
| JWKS fetch (first request) | 50-200 ms | TLS handshake + HTTP GET |
| JWKS fetch (cached) | 0 ms | In-memory cache hit |
| RSA-256 signature verify | 1-3 ms | Using native crypto bindings |
| EC-256 signature verify | 0.5-1.5 ms | Faster than RSA |
| Full token validation suite | 3-10 ms | All checks including cache |

**Performance Rules for Production:**
- Cache JWKS aggressively (TTL of 1 hour minimum, re-fetch on validation failure).
- Use EC (ES256, ES384) keys over RSA when the provider supports them — faster verification and smaller tokens.
- Validate the JWT structure and `exp` before doing signature verification (fail fast on expired tokens).
- Use a crypto library with native bindings (not pure JavaScript).

### 7.2 UserInfo Endpoint Latency

Each UserInfo call adds a network round trip. Mitigate this with:

```
Strategy: Selective Fetching

Auth request scopes: openid  (ID Token only, no UserInfo call)
                     openid profile email (UserInfo called once on login)

Do NOT call UserInfo on every page load. Cache the profile in your
session store after the initial fetch and refresh periodically.
```

### 7.3 Token Refresh Performance

When the Access Token expires, the Refresh Token flow avoids re-authenticating the user. Understanding its performance profile helps you size your infrastructure.

```

                     │  User authenticated via OIDC      │
```

**Refresh Token Rotation**: Always accept a new Refresh Token and discard the old one when the OP provides one. This limits the damage window if a Refresh Token is compromised.

### 7.4 Pre-fetching Strategy

Optimize the first authentication for your users by warming caches before the auth flow begins.

```python
# Pre-fetch strategy pseudocode
def pre_fetch_provider_config(provider_issuer):
    # Warm the Discovery Document
    discovery = fetch_discovery_document(provider_issuer)
    # Warm the JWKS keys
    jwks = fetch_jwks(discovery["jwks_uri"])

    return {
        "authorization_endpoint": discovery["authorization_endpoint"],
        "token_endpoint": discovery["token_endpoint"],
        "jwks": jwks
    }
```

Run this pre-fetch in the background when your application starts or when a user lands on the login page.

## 8. Configuration Options

### 8.1 Claim Types

The OP can return claims in three places. You configure which claim types you want through scopes.

| Claim Type | Where Delivered | Access Scope |
|---|---|---|
| `normal` | UserInfo endpoint response | Depends on scope requested |
| `aggregated` | UserInfo endpoint as a JWT | Requires special scope |
| `distributed` | UserInfo endpoint as a URL to another endpoint | Requires special scope |

In practice, most implementations use `normal` claims only. Aggregated and distributed claims are for complex identity federation scenarios.

### 8.2 ID Token Lifetime Configuration

The `exp` claim in the ID Token is controlled by the OP, but you can request a specific `max_age` in the authorization request.

```
max_age Behavior

  max_age=0         Force re-authentication (user must actively
                    authenticate, not just have an active session)

  max_age=3600      User must have authenticated within the last
                    hour. If not, re-authentication is required.

  max_age (omitted) Use the OP's default session lifetime.
```

The ID Token's `auth_time` claim tells you when the user last authenticated. Compare it to your `max_age` requirement:

```python
# Server-side max_age enforcement
def validate_auth_time(id_token, max_age_seconds):
    if "auth_time" not in id_token:
        return False  # Can't enforce without auth_time
    elapsed = current_timestamp() - id_token["auth_time"]
    if elapsed > max_age_seconds:
        return False  # Re-authentication required
    return True
```

### 8.3 Subject Identifier Types

OIDC supports two subject types that control how the `sub` value is constructed:

**Public (`sub` = `user|12345`)**
- Same `sub` value across all clients within the same OP.
- Enables cross-application user correlation (the OP can see that user 12345 uses both App A and App B).
- Simpler to implement but weaker privacy.

**Pairwise (`sub` = `abc123def456`)**
- A different `sub` value for every client-issuer pair.
- The OP computes `sub = hash(client_id + user_id + sector_identifier)`.
- Prevents correlation across clients.
- Required for privacy-sensitive applications.

```python
# Pseudocode for pairwise sub calculation
def pairwise_sub(client_id, user_id, sector_identifier):
    raw = f"{client_id}:{user_id}:{sector_identifier}"
    return base64url(sha256(raw.encode()))
```

### 8.4 Prompt Types

The `prompt` parameter controls how the OP interacts with the user.

| Prompt Value | Behavior | Use Case |
|---|---|---|
| `none` | No UI — fail if the user is not already authenticated | Silent checks, iframe-based session checks |
| `login` | Force re-authentication even if the user has an active session | High-security flows |
| `consent` | Prompt for consent even if already granted | Sensitive data access |
| `select_account` | Show the account chooser | Multi-account providers |

Combine prompts: `prompt=login%20consent` forces both re-authentication and consent.

### 8.5 ACR Values

The `acr_values` parameter specifies the Authentication Context Class Reference — the level of assurance for the authentication event.

```
Common ACR values:

acr_value                           Meaning
──────────                          ───────
0                                   Default / lowest
1                                   Standard password authentication
2                                   Multi-factor authentication
3                                   MFA + hardware token / biometric
urn:mace:incommon:iap:bronze       InCommon Bronze
urn:mace:incommon:iap:silver       InCommon Silver

Example request:
  acr_values=urn:mace:incommon:iap:silver

Your app should pass its minimum required ACR and validate
that the returned acr claim in the ID Token meets or exceeds it.
```

### 9.2 Logout Considerations

| Concern | Your Implementation |
|---|---|
| `id_token_hint` | Include the ID Token so the OP knows which session to terminate |
| `post_logout_redirect_uri` | Register this URI with the OP. Validate it matches on the OP side |
| State parameter | Include `state` in the logout request and verify on return |
| Single Logout (SLO) | OIDC does not define SLO across multiple apps. Use back-channel logout or session management |
| Local vs Remote | Always clear your local session FIRST, then redirect to the OP |
| Partial logout | Watch for the case where the OP session is cleared but the user's session at other RPs remains active |

```python
# RP-Initiated Logout URL construction
def build_logout_url(provider_config, id_token, post_logout_uri, state):
    params = {
        "id_token_hint": id_token,
        "post_logout_redirect_uri": post_logout_uri,
        "state": state
    }
    encoded = urlencode(params)
    return f"{provider_config['end_session_endpoint']}?{encoded}"
```

Not all OPs advertise an `end_session_endpoint` in their Discovery Document. Check the Discovery Document at runtime — if the endpoint is missing, only your local session can be terminated.
