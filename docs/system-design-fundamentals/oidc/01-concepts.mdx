## The Problem Story

A team of five engineers builds a consumer analytics dashboard. They spend four months on the product: real-time event ingestion, a clean React frontend, a data pipeline that processes 50,000 events per second. They use OAuth2 for API access because every tutorial says "OAuth2 is the industry standard for authorization." They deploy to production. Within two hours, the integration team from their first enterprise customer calls. The customer wants to use their own Okta instance for authentication — their security policy requires it. The team says: "No problem, we support OAuth2. Just send us an access token." The customer sends their access token. The team decodes it. It is an opaque string — no user name, no email, no profile picture, no role information. The team writes custom code to call Okta's `/userinfo` endpoint. Three weeks later, the second customer arrives. They use Azure AD. The team integrates with Azure AD's `/me` endpoint. The endpoint returns data in a completely different format. The field called `displayName` in Okta is called `name` in Azure AD. The field called `mail` in Okta is called `email` in Azure AD. There is no `picture` field. There is no `role` field. Every customer requires a custom integration. Every integration takes two to three weeks. The team has ten engineers. Six of them are now full-time identity integration specialists. The product has not shipped a new feature in eight weeks.

A startup building a mobile health app tells a similar story. They support "Sign in with Google" and "Sign in with Apple." They store the user's email from Google in one database column. They store the user's email from Apple in a different column — because Apple's privacy relay generates a unique email per app. When the same user signs in with Google on their laptop and Apple on their phone, the app creates two separate user records. The user's workout history is split across two invisible silos. The team writes a background job that tries to merge users by matching hashed email variants. The job accidentally merges 3,000 accounts that belong to different people. The team reverts the merge. The workout history remains split. The user reports a bug: "I lost six months of data." The team has no way to fix it because there is no stable, cross-provider user identifier. The team never shipped identity linking because OAuth2 does not define one.

A third team builds a machine learning platform. They authenticate users through OAuth2. An engineer notices that the access token they received from one identity provider can be decoded as a JWT — it contains the user's name, email, and organization. The engineer starts using that token as proof of identity. The access token expires in one hour. The user is logged out every hour with no warning. The team implements token refresh. The refresh token also happens to be a JWT — an entirely different format with entirely different claims. The team writes parsing logic for two different token formats. The access token format changes when the identity provider upgrades their system. The team's parsing logic breaks. Every user in the organization is locked out for 90 minutes on a Monday morning. The team never knew that access tokens have no standard structure. The team never knew that OAuth2 defines no standard way to learn who the user is — only what the client is allowed to do.

OAuth2 is the industry standard for delegated authorization. It lets a user grant an application access to their resources without sharing their password. But OAuth2 was never designed to tell you who the user is. An access token says "this client can read this resource." An access token does not say "this token belongs to Alice from the engineering team, and she has verified her email and completed multi-factor authentication." Every OAuth2 deployment that needs user identity builds a custom layer on top. Every custom layer is slightly wrong. Every custom layer creates a new integration burden. Every integration burden slows the team down until identity work consumes their entire roadmap.

## The Insight

**OIDC (OpenID Connect)** is an identity layer built on top of OAuth2 that provides a standard way to verify the identity of an end user and obtain their basic profile information.

Think of OAuth2 as a passport that proves you have permission to enter a building. You show the passport at the door. The guard checks the visa page. The guard lets you in. But the passport itself says nothing about who you are — it only proves that you are authorized to be in that building. Now think of OIDC as that same passport plus a government-issued identity card stapled to the inside cover. The identity card has your photo, your full name, your date of birth, your national ID number, and the issuing authority's seal. When you show this combined document, the guard knows not only that you may enter the building but also exactly who you are. The passport is OAuth2 — the authorization layer. The identity card is OIDC — the authentication layer. The staple that holds them together is the ID Token, a cryptographically signed JSON Web Token that the identity provider issues alongside the OAuth2 access token. The identity provider is the government agency that issued both documents. The relying party — your application — is the guard who inspects both before granting access.

OAuth2 asks "what is this client allowed to do?" OIDC asks "who is this user, and can they prove it?" The two questions are different. The two layers are complementary. You can use OAuth2 without OIDC — but your application will never have a standardized way to learn the user's identity. You can use OIDC only with OAuth2 — because OIDC depends on the OAuth2 framework for token issuance, scopes, and consent. The two protocols together form the foundation of modern identity on the web.

OpenID Connect was published in 2014. It is the third major version of the OpenID protocol. The first version launched in 2005. The second version, OpenID 2.0, launched in 2007 but was complex and vulnerable to security issues. The third version — OpenID Connect — reuses the OAuth2 framework to keep the protocol simple, secure, and interoperable. It is the protocol behind Google Sign-In, Microsoft Entra ID, Apple Sign-In, Amazon Cognito, Auth0, Okta, Keycloak, and every major identity platform in production today.

## Core Vocabulary

**OIDC (OpenID Connect)** — An identity protocol built on top of OAuth2 that enables clients to verify the identity of an end user based on the authentication performed by an authorization server, as well as to obtain basic profile information about the end user in an interoperable, REST-like manner. OIDC matters because it standardizes user identity across every provider — a client that speaks OIDC can authenticate users from any OIDC-compliant identity provider without custom integration code.

**ID Token** — A JWT (JSON Web Token) issued by the OpenID Provider that contains claims about the authentication event and the end user. The ID Token is the core innovation of OIDC — it is the cryptographically signed proof that the user authenticated successfully and that you know who they are. The ID Token contains at minimum the issuer (`iss`), the subject (`sub`), the audience (`aud`), the issued-at time (`iat`), and the expiration time (`exp`). The ID Token is signed using JSON Web Signature (JWS). It can optionally be encrypted using JSON Web Encryption (JWE). The client MUST validate the ID Token signature, expiration, issuer, and audience before trusting its contents.

**UserInfo Endpoint** — An OAuth2-protected resource that returns claims about the authenticated end user, accessed using the access token obtained during the authentication flow. The UserInfo endpoint matters because it provides a standardized way to fetch additional identity data beyond what fits in the ID Token — profile data like address, phone number, and custom attributes. The response from the UserInfo endpoint MUST be a JSON object containing claims identical to or a superset of the claims in the ID Token. The endpoint is typically located at a well-known path relative to the issuer URL, such as `https://provider.example.com/userinfo`.

**Claims** — Individual pieces of information asserted about an entity, typically the end user. A claim is a key-value pair in JSON format. Standard claims defined in the OIDC specification include `sub` (subject identifier), `name`, `given_name`, `family_name`, `preferred_username`, `email`, `email_verified`, `phone_number`, `picture`, `profile`, `address`, and `updated_at`. Claims matter because they define the contract between the identity provider and the relying party — both parties agree on what data is available and what it means, eliminating the custom-parsing problem that plagues raw OAuth2 integrations.

**Issuer** — The party that issues the ID Token, identified by the `iss` claim in the ID Token. The issuer value is a case-sensitive URL using the HTTPS scheme that uniquely identifies the OpenID Provider. The issuer claim matters because it prevents token substitution attacks — if an attacker obtains a valid ID Token from a different provider, the client can detect that the issuer does not match the expected provider and reject the token.

**Subject** — A locally unique and never reassigned identifier for the end user, conveyed as the `sub` claim in the ID Token. The subject is a case-sensitive string. The subject matters because it is the stable identifier you use to link the authenticated user to your internal user record. The subject MUST NOT exceed 255 ASCII characters. The subject MUST be unique within the issuer's namespace. The combination of `iss` and `sub` is globally unique across all OIDC providers — no two providers should produce the same `iss` + `sub` pair for different users.

**Audience** — The intended recipient of the ID Token, identified by the `aud` claim. The audience is a case-sensitive string or array of strings. The audience matters because it prevents a token intended for one client from being used by a different client. The client MUST verify that its own `client_id` appears in the `aud` claim. If the audience check fails, the client MUST reject the token regardless of whether the signature is valid.

**Authentication Request** — The initial request sent by the Relying Party to the OpenID Provider's authorization endpoint to initiate the OIDC flow. The request includes the required parameters `scope` (MUST include `openid`), `response_type`, `client_id`, and `redirect_uri`, along with optional parameters `state`, `nonce`, `display`, `prompt`, `max_age`, `ui_locales`, `id_token_hint`, `login_hint`, and `acr_values`. The authentication request matters because it is the single entry point for every OIDC interaction — everything starts with this request.

**Discovery URL** — A well-known endpoint at `https://{provider}/.well-known/openid-configuration` that returns a JSON document describing the OpenID Provider's configuration, including its authorization endpoint, token endpoint, userinfo endpoint, JWKS URI, supported scopes, response types, response modes, grant types, encryption algorithms, and signing algorithms. The discovery URL matters because it enables fully dynamic client configuration — a client can learn everything about a provider by fetching a single JSON document, enabling automatic integration without hardcoded endpoints.

**JWKS (JSON Web Key Set)** — A JSON document that contains the public keys used by the OpenID Provider to sign ID Tokens, available at the provider's JWKS URI. The JWKS matters because it enables the client to verify the ID Token's signature using the provider's public key. The client fetches the JWKS, selects the key matching the `kid` (key ID) header in the ID Token, and cryptographically verifies the token signature. This is the mechanism that prevents token forgery — without the provider's private key, no attacker can produce a valid signature.

**RP (Relying Party)** — The application or service that relies on the OpenID Provider to authenticate end users. The RP is the OAuth2 client that initiates the authentication request, validates the ID Token, and uses the identity information to establish a session or grant access. Every application you build that uses "Sign in with Google" is a Relying Party. The RP MUST register with the OP to obtain a `client_id` and a `client_secret` (for confidential clients). The RP MUST handle redirects, validate tokens, and manage sessions.

**OP (OpenID Provider)** — The OAuth2 authorization server that can authenticate end users and issue ID Tokens and access tokens to Relying Parties. The OP is the identity system — Google, Microsoft, Okta, Auth0, Keycloak. The OP performs user authentication, obtains consent, issues tokens, and exposes the UserInfo endpoint. The OP publishes its configuration at the discovery URL. The OP maintains the key pairs used to sign ID Tokens. The OP is the trusted third party in the OIDC trust model — both the RP and the user trust the OP to correctly verify identity.

**nonce** — A random, unique string value sent from the RP to the OP in the authentication request and included in the ID Token to associate a client session with an ID Token and to mitigate replay attacks. The nonce matters because it prevents an attacker from replaying an old ID Token from a different session. The RP generates a fresh nonce for every authentication request, stores it in the session, and validates that the nonce in the ID Token matches the stored value. The nonce value MUST be unpredictable — typically a cryptographically random string of at least 128 bits.

### Terminology in Practice

These terms appear in every OIDC design conversation. Here is how they show up in real discussions:

"The RP sends an authentication request to the OP with `scope=openid profile email` and `response_type=code`. The OP authenticates the user, obtains consent, and redirects back with an authorization code. The RP exchanges the code for an ID Token and an access token at the token endpoint. The RP validates the ID Token signature against the OP's JWKS, checks the issuer, verifies the audience matches its client_id, confirms the nonce matches the session, and extracts the subject to look up the local user. If the user does not exist locally, the RP creates a new user record using the claims from the ID Token and optionally fetches additional claims from the UserInfo endpoint."

"The discovery URL is the single entry point for OIDC integration. The RP fetches `https://accounts.google.com/.well-known/openid-configuration` and learns the authorization endpoint, token endpoint, userinfo endpoint, JWKS URI, and supported scopes. If Google changes any endpoint URL in the future, the RP picks up the change automatically on the next fetch. No configuration updates needed. No deployment required."

"The nonce prevents replay attacks. Every authentication request includes a fresh nonce. The OP includes that exact nonce in the ID Token. The RP stores the nonce in the session before redirecting and validates it after the redirect completes. If an attacker intercepts the ID Token and tries to replay it in a different browser session, the nonce check fails — the stored nonce does not match the token's nonce — and the RP rejects the token."

## The Big Picture

OIDC is not a replacement for OAuth2. OIDC is an identity layer that extends OAuth2 with user authentication. Every OIDC flow is an OAuth2 flow with additional parameters and a specific token type — the ID Token. The core idea is simple: the OAuth2 authorization server becomes an OpenID Provider by adding the `openid` scope to the authentication request. When the scope includes `openid`, the token response includes an ID Token in addition to (or in place of) the access token.

### The ID Token Structure

The ID Token is a JSON Web Token (JWT) with a specific set of registered claims defined by the OIDC specification. Every ID Token MUST contain the following claims:

**`iss` (Issuer)** — The URL of the OpenID Provider that issued the token. The RP must verify that the issuer matches the expected provider. Example: `https://accounts.google.com`.

**`sub` (Subject)** — A locally unique identifier for the end user, never reassigned. This is the primary key for user identity. Example: `110169484474386276334`.

**`aud` (Audience)** — The client ID that this token is intended for. The RP must verify that its own client ID is in the audience. Example: `your-client-id.apps.googleusercontent.com`.

**`exp` (Expiration Time)** — The Unix timestamp after which the token is no longer valid. The RP must verify that the token has not expired. Example: `1719876543`.

**`iat` (Issued At)** — The Unix timestamp at which the token was issued. Example: `1719876243`.

The ID Token MAY also contain these common optional claims:

**`auth_time`** — The time when the end user last authenticated. Useful for policies that require recent authentication.

**`nonce`** — The string value sent in the authentication request, included to associate the token with a specific session and prevent replay attacks.

**`acr` (Authentication Context Class Reference)** — A string indicating the authentication method level. Example: `"urn:mace:incommon:iap:silver"` for a higher-assurance authentication.

**`amr` (Authentication Methods References)** — An array of strings indicating the methods used for authentication. Example: `["pwd", "otp"]` for password plus one-time passcode.

**`azp` (Authorized Party)** — The client ID for which the token was issued, used when the token has multiple audiences.

Beyond these OIDC-defined claims, the ID Token can include any standard profile claims from the `profile`, `email`, `address`, and `phone` scopes — provided the RP requested those scopes and the user consented.

### Step 1: User Initiates Sign-In

You are building a web application. A user visits your app and clicks the "Sign In" button. Your application — the Relying Party — has no idea who this user is. The user is anonymous. The first thing your application must do is determine which identity provider the user wants to use. In a multi-tenant SaaS application, you might show a list of providers: "Sign in with Google," "Sign in with Microsoft," "Sign in with Okta." In a single-tenant enterprise application, the provider is configured and the sign-in button is labeled simply "Sign In."

### Step 2: Prepare the Authentication Request

Your application prepares an OIDC Authentication Request. This is a set of parameters that defines exactly what you are asking for. The required parameters are:

**`scope`** — MUST include `openid` to request OIDC authentication. Additional scopes like `profile`, `email`, `address`, and `phone` request access to specific claim sets.

**`response_type`** — For the Authorization Code flow, this is `code`. The oAuth2 authorization server will return an authorization code, not the token directly.

**`client_id`** — The identifier you received when you registered your application with the OP. Think of this as your application's identity badge.

**`redirect_uri`** — The URL where the OP will send the user after authentication. This URL must be registered with the OP in advance. If the redirect URI does not match a registered value, the OP rejects the request. This prevents open redirector attacks.

**`state`** — A random, unguessable string that your application generates and stores in the user's session. The OP returns this value unchanged in the redirect back to your application. The `state` parameter prevents CSRF attacks on the redirect flow — if an attacker intercepts the redirect and sends a forged response, the state will not match and your application rejects it.

**`nonce`** — A random, unguessable string that your application generates fresh for every authentication request. The OP includes this value in the ID Token. Your application verifies that the nonce in the ID Token matches the nonce you sent. The `nonce` parameter prevents replay attacks — even if an attacker intercepts the ID Token, they cannot reuse it in a different session because the nonce will not match.

Your application generates both the `state` and `nonce` as cryptographically random values, typically 16 to 32 bytes encoded as hex or base64. You store both values in the user's session (server-side or in a secure, HTTP-only cookie).

### Step 3: Redirect to the OpenID Provider

Your application constructs the authorization URL and redirects the user's browser to the OP. The URL looks like this:

```
https://provider.example.com/authorize?
  scope=openid%20profile%20email&
  response_type=code&
  client_id=your_client_id&
  redirect_uri=https%3A%2F%2Fyour-app.com%2Fcallback&
  state=xyz789abc123&
  nonce=def456ghi789
```

The user's browser navigates to this URL. The user is now at the OP's authentication page. The user sees the OP's login form, not your application's login form. This is key — the OP handles authentication directly. Your application never sees the user's password. Your application never handles credential storage.

### Step 4: User Authenticates at the OP

The user enters their credentials at the OP. Depending on the OP's policy, the user may also complete multi-factor authentication — a one-time code from an authenticator app, a push notification to their phone, a hardware security key tap, or a biometric scan. The OP enforces whatever authentication policy the organization has configured.

During this step, the OP also displays a consent screen. The consent screen shows the user what information your application is requesting: "YourApp wants to access your profile information including your name, email address, and profile picture." The user must consent. The consent screen prevents applications from silently requesting user data without the user's knowledge.

### Step 5: OP Redirects Back with Authorization Code

After successful authentication and consent, the OP redirects the user's browser back to your application's `redirect_uri`. The redirect URL includes two query parameters:

**`code`** — The authorization code, a short-lived string that your application will exchange for tokens. The authorization code is typically valid for one to ten minutes and can be used only once. If an attacker intercepts the authorization code, they have a very narrow window to use it — and they cannot use it more than once.

**`state`** — The same `state` value you sent in the authentication request. Your application must verify that this value matches the one you stored in the user's session.

The redirect URL looks like this:

```
https://your-app.com/callback?code=AUTH_CODE_123&state=xyz789abc123
```

### Step 6: Verify State

Your application receives the redirect. The first thing you do — before touching the authorization code — is verify that the `state` parameter matches the value you stored in the user's session. If the state does not match, you reject the entire response. This check prevents CSRF attacks where an attacker tricks a user into completing a flow they did not initiate.

After state verification, you delete the stored state value — each value is single-use.

### Step 7: Exchange the Authorization Code for Tokens

Your application makes a server-to-server HTTP POST request to the OP's token endpoint. This is a back-channel request — it happens from your server to the OP's server, not through the user's browser. The request includes:

- `grant_type=authorization_code` — tells the OP you are exchanging an authorization code
- `code=AUTH_CODE_123` — the authorization code from the redirect
- `redirect_uri=https://your-app.com/callback` — must match the redirect URI used in the initial request
- `client_id=your_client_id` — your application's identifier
- `client_secret=your_client_secret` — your application's secret (for confidential clients only; public clients like SPAs use PKCE instead)

The token endpoint returns a JSON response containing three tokens:

```json
{
  "id_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6Im...",
  "access_token": "ya29.a0AfH6SMBI...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "1//0eABCDEFGHIJK..."
}
```

### Step 8: Validate the ID Token

Your application now has the ID Token. Before trusting it, you must validate it thoroughly. This is the most critical step — if you skip validation, you are vulnerable to token forgery attacks.

Your validation checklist includes:

1. **Signature verification** — Fetch the OP's JWKS from the discovery URL. Find the key that matches the `kid` in the ID Token header. Use that key to verify the JWT signature. If the signature is invalid, reject the token.

2. **Issuer verification** — Confirm that the `iss` claim matches the expected OP URL. For Google, this is `https://accounts.google.com`. If the issuer is unexpected, reject the token.

3. **Audience verification** — Confirm that your `client_id` appears in the `aud` claim. If someone else's client ID is the audience, the token was intended for a different application. Reject it.

4. **Expiration verification** — Confirm that the current time is before the `exp` claim. Allow a small clock skew (typically 60 seconds). If the token has expired, reject it.

5. **Nonce verification** — Confirm that the `nonce` claim in the ID Token matches the nonce you stored in the user's session. Delete the stored nonce after verification — each nonce is single-use.

6. **Additional checks** — If your application requires MFA, verify that the `amr` claim includes a multi-factor authentication method. If your application requires a specific authentication level, verify the `acr` claim.

### Step 9: Extract User Identity

After validation, you extract the user's identity from the ID Token's claims. The `sub` claim is your stable user identifier. Use `iss` + `sub` as the globally unique key to look up or create a local user record:

```python
user = User.find_or_create_by(
    provider="https://provider.example.com",
    subject=id_token["sub"]
)
user.update(
    name=id_token.get("name"),
    email=id_token.get("email"),
    picture=id_token.get("picture")
)
```

If you need additional claims that are not in the ID Token — for example, the user's full address or phone number — you fetch them from the UserInfo endpoint using the access token.

### Step 10: Establish Session

Your application creates a local session for the authenticated user. This session is managed entirely by your application — you can use cookies, server-side sessions, or local JWTs. The session is the link between the OIDC authentication event and your application's ongoing interaction with the user. The user is now logged in. Your application knows exactly who they are. Every subsequent request goes through your application's session check.

When the ID Token expires — typically after one hour — your application can use the refresh token to obtain a new ID Token and access token without requiring the user to re-authenticate. The refresh token is long-lived. The refresh token can be revoked by the OP at any time, which effectively logs the user out.

## Flow Variants

OIDC supports three flow variants that correspond to OAuth2 grant types. Each flow offers different tradeoffs between security, complexity, and supported client types.

### Authorization Code Flow (Recommended)

This is the flow described in detail above. The RP receives an authorization code through the front channel (browser redirect) and exchanges it for tokens through the back channel (server-to-server HTTP request). The authorization code is short-lived and single-use. The token exchange happens server-side, so the ID Token never reaches the user's browser.

**When to use it:** Web applications with a backend server, mobile applications, and any confidential client. This is the recommended flow for almost every application.

**Security properties:** The ID Token and access token never pass through the user's browser. The authorization code is useless without the client secret (or PKCE verifier). The refresh token can be stored securely on the server.

**PKCE (Proof Key for Code Exchange):** For public clients — single-page applications and mobile apps that cannot securely store a client secret — OIDC mandates the use of PKCE. The RP generates a cryptographic challenge and sends the challenge in the authentication request. The RP sends the verifier when exchanging the authorization code. The OP verifies that the verifier matches the challenge. PKCE prevents authorization code interception attacks — even if an attacker intercepts the authorization code, they cannot exchange it without the verifier.

### Implicit Flow (Deprecated)

The Implicit flow returns the ID Token directly from the authorization endpoint as a fragment in the redirect URL — no authorization code exchange, no back-channel request. The ID Token arrives in the user's browser as a URL fragment: `https://your-app.com/callback#id_token=eyJ...&access_token=ya29...&state=xyz...`.

**Why it exists:** The Implicit flow was designed for browser-only applications that have no backend server to perform the code exchange. The idea was to eliminate the server-side step and deliver the token directly.

**Why it is deprecated:** The Implicit flow has significant security weaknesses. The ID Token is exposed in the browser URL, visible to browser history, referrer headers, and any JavaScript on the page. The token cannot be securely stored — it lives in the browser's address bar and browser memory. There is no way to rotate or revoke a token delivered via the Implicit flow. The OAuth2 Security Best Current Practice document (BCP 195) explicitly recommends against using the Implicit flow.

**What to use instead:** The Authorization Code flow with PKCE. PKCE was specifically designed to make the Authorization Code flow safe for public clients — single-page applications and mobile apps — without exposing tokens in the URL.

### Hybrid Flow

The Hybrid flow combines elements of the Authorization Code flow and the Implicit flow. In a Hybrid flow, the OP returns both an authorization code and a token directly from the authorization endpoint. The response type parameter indicates which tokens are returned immediately: `response_type=code id_token`, `response_type=code token`, or `response_type=code id_token token`.

**When to use it:** Hybrid flow is useful when your application needs an ID Token immediately — to show the user's name on the loading screen before the token exchange completes — while still using the authorization code for secure token exchange. This is an advanced pattern used in specific high-performance scenarios.

**How it works:** The authorization endpoint returns both an authorization code (via the query string) and one or more tokens (via the URL fragment). The RP uses the ID Token from the fragment for immediate user identification and then exchanges the authorization code for the full set of tokens via the back channel.

**Security tradeoff:** The ID Token is exposed in the URL fragment during the Hybrid flow, inheriting the exposure risk of the Implicit flow. The access token and refresh token are only obtained through the back-channel code exchange, so they are not exposed. The Hybrid flow is a compromise — you accept some token exposure for the benefit of immediate user identification.

## Mental Model Check

Work through these five questions to verify that you understand OIDC at the depth required for system design interviews.

### Question 1

Your team is building a SaaS platform for enterprise customers. Each customer wants to use their own identity provider — Okta, Azure AD, or OneLogin. Your CEO asks you to estimate the integration effort. She expects "each provider will take about three weeks." How do you respond?

**Answer:** You respond that each provider will take one or two days of configuration work, not three weeks. OIDC standardizes the authentication flow across all compliant providers. Your application fetches each provider's discovery URL at `/.well-known/openid-configuration`, learns the endpoints and supported features dynamically, and uses the standard Authorization Code flow with PKCE. The integration code is the same for every provider — only the `client_id`, `client_secret`, and `issuer` value change. The reason teams historically spent weeks per provider is that they built custom OAuth2 integrations without OIDC — every provider returned user data in a different format, required different parameters, and had different token semantics. OIDC eliminates this variation. If a provider claims OIDC support, your application works with that provider on day one.

### Question 2

Your application receives an ID Token with `iss: "https://evil-provider.com"`, `aud: "your-client-id"`, and a valid signature from the evil provider's JWKS. Is this token valid?

**Answer:** No, this token is invalid. Your application must verify that the `iss` claim matches the expected OpenID Provider. If you configured your application to use Google as the identity provider, the only acceptable issuer is `https://accounts.google.com` (or `https://accounts.google.com`). An ID Token from any other issuer must be rejected regardless of whether the signature is valid. The signature proves only that the token was issued by the provider named in the `iss` claim — it does not prove that the issuer is the provider you trust. An attacker could spin up their own OIDC-compliant provider, register your client ID, issue tokens for any user they choose, and those tokens would have valid signatures from the attacker's provider. The issuer check is the gate that stops this attack.

### Question 3

Your application uses OIDC for authentication. A user signs in, receives an ID Token, and establishes a local session. One hour later, the user's ID Token expires. The user is still actively using your application. What happens?

**Answer:** The user continues using your application without interruption. The ID Token is an authentication artifact used during the initial sign-in to verify the user's identity. It is not used for ongoing session management. Your application establishes its own session — a cookie, a server-side session, or a local JWT — after the OIDC authentication completes. The session is independent of the ID Token. The ID Token expires after one hour, but your session can last for hours or days. If your application needs to verify the user's identity again — for example, to perform a sensitive operation like changing a password — you use the refresh token to obtain a new ID Token from the OP. The refresh token is long-lived and does not require the user to re-authenticate. If the refresh token is revoked or expires, the user must sign in again.

### Question 4

A single-page application uses OIDC with the Authorization Code flow and PKCE. The attacker intercepts the authorization code during the redirect. Can the attacker exchange the code for tokens?

**Answer:** No, the attacker cannot exchange the authorization code for tokens because the attacker does not have the PKCE code verifier. When the SPA initiated the authentication request, it generated a random code verifier, computed the SHA-256 hash as the code challenge, and sent the challenge to the OP. The code verifier was stored in the SPA's memory. When the SPA exchanges the authorization code, it must send the original code verifier. The attacker has the authorization code but does not have the code verifier — the verifier was never transmitted to the OP or included in any URL. The OP verifies that the code verifier matches the code challenge before issuing tokens. Without the correct verifier, the code exchange fails. This is the precise attack that PKCE was designed to prevent, and it is why the Authorization Code flow with PKCE is secure for public clients.

### Question 5

A financial application requires that users authenticate using multi-factor authentication (MFA). The application uses OIDC with an enterprise identity provider. How does the application enforce the MFA requirement?

**Answer:** The application uses the `acr` (Authentication Context Class Reference) claim and `amr` (Authentication Methods References) claim in the ID Token to enforce the MFA requirement. The application configures the authentication request to include the `acr_values` parameter specifying the required authentication context — for example, `acr_values=urn:mace:incommon:iap:silver` for an MFA-required level. The OP authenticates the user according to the requested `acr_values`. If the user does not have an MFA method configured or does not complete MFA, the OP does not issue the ID Token — the authentication fails. When the ID Token is returned, it includes the `acr` claim indicating the achieved authentication level and the `amr` claim listing the methods used (for example, `["pwd", "otp"]` for password plus one-time passcode). The application verifies that the `acr` value meets the required level and that the `amr` list includes at least two distinct authentication methods. If the authentication does not meet the MFA requirement, the application rejects the ID Token and does not establish a session. This enforcement happens at the application level, independent of the OP's configuration — even if the OP changes its MFA policy, your application's verification code remains the same.
