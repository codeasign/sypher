# ACID vs BASE at Scale

The previous page examined the internal machinery of each system. This page
connects them into real production architectures — the patterns and pitfalls
of combining ACID and BASE systems in a single application.

---

## Naive Implementation

Every team starts with the simplest possible architecture: one database, one
schema, one consistency model. This works until it does not.

### Phase 1: 100 Users — One ACID Database

An early-stage startup puts everything in PostgreSQL: users, orders, inventory,
sessions, analytics, and logs. The schema has 40 tables with foreign keys and
check constraints. Everything is ACID. It works perfectly. Writes complete in
2-5ms. Reads complete in under 1ms. The CTO sleeps well.

### Phase 2: 10,000 Users — Write Throughput Bottleneck

The database handles 2,000 writes per second during peak hours. The WAL fsync
is the bottleneck. The team moves sessions to Redis (BASE — eventual
consistency is fine for sessions). This frees up capacity. Writes drop to 1,500
per second on PostgreSQL, which is sustainable.

But inventory updates now contend with order writes. Transactions are aborted
and retried. The team notices that `SERIALIZABLE` isolation causes 5% of
transactions to fail due to serialization conflicts. They switch to `REPEATABLE
READ`. The conflict rate drops to 0.2%.

### Phase 3: 1,000,000 Users — The ACID Wall

At a million daily active users, the single PostgreSQL primary cannot handle the
write volume. The team scales vertically: 64 cores, 256 GB RAM, NVMe SSDs. This
gets them to 10,000 writes per second. It costs $4,000 per month for the
database instance.

But the application needs 20,000 writes per second during flash sales. The
single primary physically cannot fsync the WAL fast enough. The team is at the
**ACID wall**: the write throughput of a single node is the hard limit.

### What Happens Next

The team has three options:

1. **Shard PostgreSQL:** Split data across multiple PostgreSQL instances using
   application-level sharding. This works but loses cross-shard ACID guarantees.
   A transaction that spans shards is not atomic.

2. **Switch to a distributed ACID database:** Use CockroachDB or YugabyteDB.
   These provide ACID guarantees across nodes, but with higher latency and lower
   throughput than a single-node PostgreSQL.

3. **Adopt polyglot persistence:** Use ACID for what must be ACID (orders,
   payments, inventory) and BASE for what can be BASE (catalog, sessions,
   analytics, logs). This is the most common path at scale.

### Data Domain Mapping

| Domain | Storage | Model | Rationale |
|---|---|---|---|
| Payments, Ledger | PostgreSQL | ACID | Every cent must be accounted for. Atomicity and durability are non-negotiable. |
| Orders (write) | PostgreSQL | ACID | An order must be written exactly once. No duplicate or partial orders. |
| Product Catalog | Cassandra | BASE | Catalog data is read-heavy. Stale reads (seconds) are acceptable. Write throughput matters for flash sales. |
| User Sessions | Redis | BASE | Session data is ephemeral. Losing a session forces a login — annoying but not catastrophic. |
| Search Index | Elasticsearch | BASE | Index updates lag behind the source of truth by seconds to minutes. This is expected. |
| Analytics Events | S3 / Kafka | BASE | Append-only log. There is no conflict to resolve. Durability comes from S3 replication. |
| Inventory Counts | PostgreSQL | ACID | Double-selling an item is a direct revenue loss. Must use strong consistency. |

## Integration Patterns

Mixing ACID and BASE systems creates a new challenge: how do you keep data
consistent across systems with different consistency models?

### CQRS — Command Query Responsibility Segregation

CQRS separates the write model (commands) from the read model (queries). The
write model uses ACID to guarantee that commands are processed correctly. The
read model uses BASE for fast, scalable reads. The read model is updated
asynchronously from the write model through events.

**Example:** An e-commerce order service:

1. Order service writes the order to PostgreSQL (ACID). This is the command
   side — it must succeed or fail atomically.
2. The order write generates an event: `OrderPlaced { orderId, items, total }`.
3. Kafka delivers this event to the catalog service and the search service.
4. The catalog service updates an inventory dashboard (Cassandra, BASE).
5. The search service indexes the order for customer search (Elasticsearch,
   BASE).

The read side is eventually consistent with the write side. If the read side is
behind by 200 milliseconds, the customer might see an old order status
momentarily. The system acknowledges this and provides a refresh button.

### Event Sourcing

Event sourcing stores every state change as an immutable, append-only event.
The current state is derived by replaying all events. The event store is
BASE-friendly (append-only, no locks, no updates). The derived state can be
materialized into any read model.

**Example:** A banking application:

1. Every deposit, withdrawal, and transfer is an event: `Deposited(accountId,
   amount, timestamp)`.
2. Events are appended to the event store (Kafka or a dedicated event store
   like EventStoreDB).
3. The current balance is computed by summing all events for an account.
4. If the bank needs a new view (daily summary), it replays events through a
   new projector. No migration needed.

Event sourcing avoids the ACID locking problem because writes are append-only.
But the balance computation must be correct — this is typically done in a
strongly consistent projection.

### SAGA Pattern

A SAGA is a sequence of local transactions where each step publishes an event
that triggers the next step. If a step fails, the SAGA executes compensating
transactions to undo the previous steps. This pattern allows ACID-like
guarantees across multiple systems, each of which may use a different
consistency model.

**Example:** A ride-sharing ride:

1. **Payment service (ACID):** Charge the rider's card. If this succeeds, emit
   `PaymentCharged`.
2. **Dispatch service (BASE):** Assign a driver. If no driver is available,
   emit `DispatchFailed`. The compensating action is `PaymentRefunded`.
3. **Notification service (BASE):** Send push notification to rider and driver.
   If this fails, the ride still proceeds — notifications are best-effort.

The SAGA is an orchestration pattern. A coordinator (orchestrator) tracks state
and issues compensating actions on failure. The SAGA does not provide global
ACID isolation — a rider could see `PaymentCharged` before `DispatchFailed` is
detected. This is acceptable because the compensating transaction refunds the
payment.

### Change Data Capture (CDC)

CDC reads the WAL of an ACID database and streams changes to other systems.
This bridges the ACID-to-BASE gap: the source system keeps ACID guarantees,
and downstream systems consume changes with BASE semantics.

**Tools:** Debezium (Kafka Connect), AWS DMS, Striim.

**Example:** PostgreSQL → Debezium → Kafka → Cassandra

1. A write lands in PostgreSQL. The WAL record is captured by Debezium.
2. Debezium emits an event to Kafka: `{"op": "c", "after": {"id": 42,
   "status": "shipped"}}`.
3. A Kafka consumer writes the update to Cassandra for the customer-facing
   order tracker.
4. If the Kafka consumer is down for 30 seconds, Cassandra lags behind
   PostgreSQL by 30 seconds. This is acceptable for the order tracker.

## Multi-Region Deployment

When your application serves users globally, you must replicate data across data
centers. ACID and BASE take fundamentally different approaches.

### ACID Multi-Region — Synchronous Replication

An ACID database that spans regions must use synchronous replication. The write
is not acknowledged until it is fsynced on both the primary and at least one
replica in a different region.

**Cost:** Cross-region latency. A round trip between US-East (Virginia) and
EU-West (Ireland) is approximately 80 milliseconds. Every write incurs this
latency. Write throughput drops to approximately 10-20 writes per second for
a synchronous cross-region ACID setup.

**Worse:** If the replica in the other region is unavailable (network partition,
region failure), the primary cannot acknowledge writes. The system is down
until the replica comes back or an operator promotes another replica.

**Used by:** CockroachDB, Google Spanner (with TrueTime), YugabyteDB.

### BASE Multi-Region — Asynchronous Replication

A BASE database replicates across regions asynchronously. Writes are
acknowledged in the local region and propagated to other regions in the
background.

**Cost:** No cross-region latency on the write path. Each region serves writes
independently at its local throughput. The cost is possible data loss: if a
region fails before its writes are propagated, those writes are lost.

**Used by:** Cassandra (multiple data centers), DynamoDB Global Tables, Riak.

### Hybrid Approach — ACID per Region, BASE Between Regions

Many production systems use ACID databases within a region and BASE replication
between regions. Each region has its own PostgreSQL primary. Cross-region
replication uses async streaming (not synchronous). If a region fails, the
application fails over to another region — losing the last few seconds of
writes.

This is the approach used by GitHub, Stripe, and many financial systems. It
accepts that cross-region consistency is eventual, even within an ACID system.

## Capacity Planning

Estimating the cost of ACID vs BASE at scale requires understanding the
underlying physics of each system.

### ACID Cost Model

```
Total write cost = N_writes x (fsync_time + replication_overhead)

fsync_time = 1-3ms per fsync (SSD)
replication_overhead = 0-80ms for synchronous replica

Example: 100M writes/day on ACID PostgreSQL
  Writes per second (peak): ~10,000
  Fsync time per write: 2ms
  Replication factor: 1 primary + 1 sync replica (80ms cross-region)
  Total write latency: 82ms
  Max throughput: 1,000,000ms / 82ms = ~12,000 writes/sec
  Instance cost: 64 vCPU, 256 GB RAM ~ $3,500/month
  Storage: 1TB NVMe SSD ~ $500/month
  Total: ~$4,000/month
```

### BASE Cost Model

```
Total write cost = N_writes / fanout x quorum_cost

fanout = replication factor (typically 3)
quorum_cost = max(coordinator_tx, replica_tx) for quorum (2 of 3)

Example: 100M writes/day on Cassandra
  Writes per second (peak): ~10,000
  Replication factor: 3
  Consistency level: QUORUM (2 of 3)
  Coordinator time: 0.3ms
  Replica time: 0.3ms
  Per-write latency: 0.3ms (no fsync on write path)
  Max throughput per node: ~50,000 writes/sec
  Nodes needed: 6 (to handle 10K/sec with RF=3)
  Instance cost (6 x 16 vCPU, 64 GB): ~$3,000/month
  Storage (6 x 500GB SSD): ~$600/month
  Total: ~$3,600/month
```

### Cost Comparison for 100M Writes/Day

| Factor | ACID (PostgreSQL) | BASE (Cassandra) |
|---|---|---|
| Write latency | 2-5ms (single node) | 0.3-1ms |
| Max throughput | ~12K writes/sec | ~300K writes/sec (6 nodes) |
| Infrastructure cost | ~$4,000/month | ~$3,600/month |
| Consistency | Immediate | Configurable |
| Cross-region latency | 80-160ms | 1-5ms |
| Risk | Writes rejected on overload | Stale reads, conflict windows |

The BASE system provides more throughput for roughly the same cost. But the
comparison does not capture the engineering cost: building for eventual
consistency is harder than building for strong consistency.

## Deployment and Monitoring

### ACID Deployment — Primary-Replica Failover

A production ACID deployment uses a primary (handles writes) and one or more
replicas (handle reads). Failover requires promoting a replica to primary.
Modern tools (Patroni, pg_auto_failover, RDS Multi-AZ) automate this.

**Monitoring:**

- Transaction commit latency (p50, p95, p99). Rising latency indicates lock
  contention or I/O saturation.
- Deadlock rate. More than zero deadlocks per minute indicates a design
  problem.
- Replication lag. In PostgreSQL, this is the difference between WAL position
  on primary and replica. More than 30 seconds of lag indicates slow replicas
  or network issues.
- Connection pool utilization. Running out of connections causes write
  rejection.

### BASE Deployment — Ring Topology

A BASE deployment distributes nodes across availability zones. The ring
topology has no single point of failure. Nodes join and leave without manual
intervention.

**Monitoring:**

- Hinted handoff queue depth. A growing queue indicates nodes are
  persistently down.
- Read repair counts. A spike indicates widespread inconsistency.
- Gossip convergence time. Nodes should converge on cluster state within 3-5
  gossip rounds (3-5 seconds).
- Consistency violations. Track the number of stale reads returned to
  clients. This is the measure of how well eventual consistency is working.
- Merkle tree comparison frequency. Should run on every gossip round.
  Discrepancies should trigger immediate repair.

## Sharding Strategies for ACID at Scale

When a single ACID instance is no longer sufficient, sharding is the primary
strategy for horizontal scaling. Each shard is an independent ACID database
that holds a subset of the data. The application routes each query to the
correct shard.

### Key-Based Sharding

Each row is assigned to a shard based on a hash of the primary key:
`shard = hash(key) % N`. This distributes data evenly as long as the hash
function is uniform. Consistent hashing reduces data movement when adding or
removing shards.

**Example:** An e-commerce platform shards orders by `customer_id`. All orders
for customer 12345 go to shard 7. A lookup: `shard = murmur3(customer_id) %
32`. To find a customer's orders, the application queries shard 7 only.

**ACID within a shard:** Each shard provides full ACID guarantees for data on
that shard. Transactions that span shards are not atomic — the application
must handle this via the SAGA pattern or two-phase commit.

**Cross-shard transactions:** Avoid them where possible. If an order write
spans two shards (customer data on shard 7, payment data on shard 3), the
application cannot atomically write to both. Solution: co-locate related data
on the same shard. In the e-commerce example, store order and payment data in
the same shard as the customer data.

### Range-Based Sharding

Rows are assigned to shards by key range. Shard 1 holds keys A-D, shard 2
holds keys E-H, and so on. This makes range scans efficient but risks hotspot
shards (shard 4 handles 40% of traffic because it contains the most active
keys).

**Example:** Discord shards message storage by server creation date. Newer
servers go to newer shards. Active servers are spread across shards.

**Rebalancing challenge:** When a shard grows too large (hotspot), its range
must be split. This requires moving data from one shard to another while the
system is live. Tools like Vitess (MySQL sharding) automate this.

### Directory-Based Sharding

A lookup service maps each key to a shard. This allows arbitrary routing rules
and easy rebalancing (update the directory entry). The directory itself must
be highly available and strongly consistent — it is typically backed by
ZooKeeper or etcd.

**Example:** A SaaS platform assigns each tenant to a shard. When a tenant
grows, its data is moved to a larger shard by updating the directory entry.
No data is redistributed — only the tenant's data moves.

**Trade-off:** The directory lookup adds one network hop per query. Latency
increases by 1-3ms. The directory is also a single point of failure if not
properly replicated.

### Vitess — Sharded MySQL in Production

Vitess is the most mature open-source ACID sharding solution. It was built by
YouTube to scale MySQL and is now a CNCF project.

**How Vitess works:**

1. Vitess presents a single MySQL-compatible interface to the application. The
   application sends queries to Vitess as if it were a single MySQL instance.
2. Vitess parses the query, determines which shard owns the data, and routes
   the query to the correct MySQL backend.
3. For cross-shard queries (for example, "find all orders for this customer"
   when customer data spans shards), Vitess fans out the query to all shards
   and aggregates results.
4. Vitess manages schema changes across all shards. An `ALTER TABLE` is
   applied to each shard sequentially.

**Limitations:** Vitess supports ACID within a shard but does not support
cross-shard ACID transactions. A two-phase commit across shards is
architecturally possible but rarely used in practice.

## Tiered Storage in Production

A production system rarely stores all data on the same storage tier. Combining
ACID and BASE with a tiered storage strategy reduces cost while maintaining
performance.

### Hot / Warm / Cold Architecture

| Tier | Storage | Consistency | Cost/GB/Month | Use Case |
|---|---|---|---|---|
| Hot | In-memory (Redis) | BASE event. | $15-30 | Session cache, real-time counters |
| Warm | SSD (PostgreSQL, Cassandra) | ACID/BASE | $0.10-0.30 | Active orders, current catalog |
| Cold | HDD / S3 / GCS | BASE (S3 read-after-write) | $0.01-0.02 | Historical orders, logs, backups |

### Migration Between Tiers

Data moves from hot to warm to cold based on access patterns. The migration is
asynchronous and uses CDC:

1. **Hot to warm:** When a session expires (30 minutes idle), Redis evicts the
   data. If the session is needed later, it is reconstructed from the warm
   store (PostgreSQL or Cassandra).

2. **Warm to cold:** Orders older than 90 days are moved to S3. An archival
   job reads from PostgreSQL, writes to Parquet files in S3, and deletes from
   the active table. The application has a "search orders" feature that first
   checks PostgreSQL, then falls back to S3.

3. **Cold to warm:** If a customer views their 2-year-old order, the
   application reads from S3, caches the result in Redis (hot), and serves the
   response. Subsequent views hit the hot cache.

### Cost Impact

For an e-commerce platform with 500 million orders:

- All orders in PostgreSQL (warm): 500M rows x 2KB = 1TB. Warm SSD cost:
  ~$300/month.
- Orders older than 90 days moved to S3: ~80% of data = 0.8TB in S3 Standard.
  Cost: ~$20/month.
- Active orders in PostgreSQL (warm): 100M rows = 0.2TB. Cost: ~$60/month.

Savings: ~$220/month for this single table, with no performance impact because
the active dataset is smaller.

## Observability — Monitoring ACID/BASE Health

A polyglot persistence architecture is harder to monitor than a single-database
architecture. Each system has its own metrics and failure modes.

### Unified Dashboard Metrics

| Metric | ACID (PostgreSQL) | BASE (Cassandra) | Threshold |
|---|---|---|---|
| Write latency (p99) | `pg_stat_activity` query time | `nodetool cfstats` write latency | ACID: under 10ms, BASE: under 5ms |
| Read latency (p99) | `pg_stat_activity` query time | `nodetool cfstats` read latency | ACID: under 5ms, BASE: under 20ms |
| Error rate | Connection errors, deadlocks | UnavailableException, timeout | less than 0.1% of operations |
| Throughput | Writes/sec | Writes/sec | Monitor trend, not absolute |
| Replication lag | `pg_stat_replication` | `nodetool netstats` | ACID: under 1s, BASE: under 5s |
| Storage utilization | Table size growth rate | SSTable count per node | Grow less than 10% per week |

### Alerting Rules

1. **Read repair rate > 1% in Cassandra:** Indicates replicas are diverging.
   Investigate network issues or a recently down node.

2. **PostgreSQL deadlock rate > 0 per hour:** Indicates lock ordering problem.
   Transactions must acquire locks in the same order.

3. **CDC lag > 30 seconds:** Kafka consumer is falling behind. Scale consumer
   group or investigate topic partition imbalance.

4. **Dual-write divergence > 0:** If you are dual-writing to ACID and BASE,
   compare the values and alert on any divergence.

5. **Hinted handoff queue growing:** A node is persistently down. Investigate
   before the hint expiration window (default 3 hours) expires.

## Disaster Recovery — ACID vs BASE

### ACID Disaster Recovery

ACID databases typically use streaming replication. The replica stays in sync
with the primary within milliseconds. If the primary fails, the replica is
promoted automatically (tools like Patroni or RDS Multi-AZ handle this).

**Recovery point objective (RPO):** For synchronous replication, zero data
loss. For asynchronous replication, the RPO is the replication lag (typically
under 1 second).

**Recovery time objective (RTO):** 30-120 seconds for automated failover.
Manual failover can take 5-30 minutes depending on operator response time.

**Pitfall:** If the primary fails and the replica has lag, some committed
transactions are lost. Promising zero data loss with asynchronous replication
is a configuration error.

### BASE Disaster Recovery

BASE databases are inherently multi-node. A single node failure does not cause
downtime — the other nodes continue serving requests. Recovery is
node-by-node: replace the failed node, let it stream data from peers.

**RPO:** Configurable. With write consistency ALL, zero data loss but reduced
availability. With write consistency ONE, the RPO is several seconds of data
(the window between the write acknowledgment and the last successful gossip
round).

**RTO:** A single node failure is invisible (no RTO impact). A full data center
failure requires DNS or routing changes, which take 1-5 minutes for most cloud
configurations.

### Multi-Region DR Comparison

| Factor | ACID (PostgreSQL) | BASE (Cassandra) |
|---|---|---|
| Replication | Synchronous (sync replica in other region) | Asynchronous (eventual convergence) |
| Cross-region RPO | Zero (sync) or ~1s (async) | Configurable, typically seconds |
| Cross-region failover | Manual or automated promotion | Traffic routing change |
| Consistency during failover | All writes blocked during promotion | No writes blocked, possible stale reads |
| Cost | High (dedicated replica in each region) | Moderate (each node serves requests) |
