Your production system was serving 8,000 requests per second. At 3:14 AM, the p99 latency climbed from 120ms to 12 seconds. Users started seeing 503 errors. The on-call engineer SSHed into a box, ran `tail -f /var/log/app.log`, and found a single line: `ERROR - database connection pool exhausted`. That one log line told them exactly where to look: the database connection pool. Without it, the engineer would have spent 30 minutes guessing — is it the load balancer? The application? The CDN? A memory leak? The network? Logging turned a needle-in-a-haystack debugging session into a 90-second fix.

Every distributed system generates an avalanche of events: requests arriving, database queries completing, errors firing, caches being hit or missed, authentication succeeding or failing, background jobs starting and finishing, network connections opening and closing, configuration changes taking effect, memory usage crossing thresholds, and thousands of other signals. Logging is the discipline of recording those events in a structured, searchable, and durable way so that you can reconstruct what happened at any point in time, determine why something went wrong, and understand how the system is behaving under load.

Without logging, you are operating in the dark. You cannot debug failures, you cannot audit access, you cannot measure latency breakdowns, you cannot detect anomalies, you cannot trace a single request across twenty microservices, and you cannot answer the most basic question in operations: what just happened?

This section builds the mental model for logging from first principles. You will learn what a log is, how structured logs differ from plain text, what log levels mean and when to use each one, how log aggregation works across a distributed system, and why every engineer must think about logging before writing a single line of application code.

## The Insight

A log entry is an immutable, timestamped record of an event that occurred in a system. The key insight is that logs are both the most granular source of observability data and the most voluminous. Every request, every database query, every cache lookup, every authentication attempt, every configuration change, every error, every scheduled job — each of these is a log event. A modestly-sized service handling 10,000 requests per second generates over 860 million log entries per day.

Structured logging transforms raw text into queryable data. Instead of writing `"User 12345 logged in at 2026-07-02 14:30:00"`, a structured logger writes:

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00Z", "service": "auth-service", "event": "user_login", "user_id": 12345, "ip": "192.168.1.100", "duration_ms": 45}
```

This structure is machine-readable. You can filter by `level`, aggregate by `event`, correlate by `user_id`, calculate average `duration_ms` over time, and build dashboards without regex. Structured logs are the foundation of every modern observability platform.

Three properties define a good logging system:

1.  **Completeness**: every significant event is recorded. The system does not silently drop events.

2.  **Searchability**: you can find relevant logs across millions of entries in seconds. This requires indexing on key fields.

3.  **Durability**: once written, logs survive process crashes, node failures, and disk corruption. They are the system's persistent memory.

### Why These Three Properties Matter

Completeness matters because the log event that is missing is often the most important one. When a request fails, the absence of a log where one should exist is itself a signal. If the shipper drops events under load, you lose the evidence of what caused the load in the first place.

Searchability matters because the value of logs is proportional to how quickly you can find relevant entries. A log system that stores 10 PB of data but takes 10 minutes to return a query result is useless during an incident. Engineers need sub-second search to correlate events in real time.

Durability matters because logs are the system's memory. When a node crashes, the logs from before the crash are the only record of what led to the failure. If those logs are lost, the crash is effectively unrecoverable from a debugging perspective.

### The Three Pillars of Observability

Logging is one of three pillars of observability, alongside metrics and traces. Each pillar serves a different purpose:

- **Metrics**: numerical measurements aggregated over time. Answer "what is happening?" Error rate, latency, throughput.
- **Logs**: discrete event records. Answer "what exactly happened?" Each log entry is a specific event with a timestamp and context.
- **Traces**: end-to-end request paths across services. Answer "where did the request go?" Trace spans connect logs from different services into a single story.

## Core Vocabulary

These terms form the shared language of logging. Master them before moving on.

**Log event**: a single record representing something that happened in the system. Contains a timestamp, severity level, message, and structured metadata. Example: `{"level": "error", "timestamp": "...", "message": "connection refused", "service": "api-gateway", "remote_addr": "10.0.0.5"}`.

**Log level**: the severity classification of a log event. Standard levels from most to least severe: FATAL, ERROR, WARN, INFO, DEBUG, TRACE. In production, only FATAL through INFO are typically enabled; DEBUG and TRACE are activated during troubleshooting.

**Structured logging**: writing log events in a structured format (JSON, protobuf, or Avro) rather than free-form text. Enables programmatic querying, filtering, and aggregation without parsing.

**Log aggregation**: the process of collecting logs from multiple servers, services, or containers into a centralized store. Without aggregation, you must SSH into individual machines and grep log files — impossible at scale.

**Log shipping**: the mechanism that transports logs from the application process to the aggregation system. Common shippers include Filebeat, Fluentd, Logstash, Vector, and the AWS CloudWatch Agent.

**Log rotation**: the practice of closing the current log file and opening a new one based on size or time, then eventually deleting or archiving old files. Prevents a single log file from consuming all available disk space.

**Index**: in a log search system like Elasticsearch, an index organizes log events by time and field schema. Queries are run against indices to return matching events in milliseconds.

**Log retention**: the policy that determines how long logs are kept in fast storage (hot tier) versus slow storage (warm/cold tier) versus deleted. Typical hot retention is 7-30 days; cold retention may extend to years for compliance.

**Correlation ID**: a unique identifier attached to every log event generated during a single request or transaction. Enables tracing the request's path across multiple services, processes, and threads. Also called trace ID or request ID.

**Log shipper agent**: a lightweight daemon running on each node that tails log files or listens on a socket, applies basic transformations (filtering, parsing, tagging), and forwards events to the aggregation pipeline. Designed to be resilient: if the aggregation endpoint is unreachable, it buffers logs on disk and replays them later.

**Syslog**: a legacy standard (RFC 5424) for log message transport. Defines facility codes (kernel, mail, daemon, auth, etc.) and severity levels. Still widely used by network devices and Linux system services, but being replaced by structured JSON pipelines in modern applications.

**Log obfuscation**: the process of redacting sensitive information (passwords, credit card numbers, PII, access tokens) from log events before they are stored. Critical for compliance with GDPR, HIPAA, PCI-DSS, and SOC 2.

**Log sampling**: the practice of recording only a fraction of log events to reduce volume and cost. Common strategies include head-based sampling (capture every Nth request) and tail-based sampling (capture all events for a subset of requests selected after observing their outcome).

**Log format**: the schema or structure of a log event. Common formats include JSON (most common for modern systems), Logfmt (key=value pairs, popular in Go ecosystems), Apache Common Log Format (for web servers), and Syslog (RFC 3164 and RFC 5424).

**Structured metadata**: additional key-value pairs attached to a log event beyond the standard fields. Examples include environment, deployment version, data center, pod name, and user agent. Metadata enables richer filtering and correlation.

**Log buffer**: a temporary storage area in memory or on disk where log events are held before being written to the final destination. Buffers improve throughput by batching writes but risk data loss if the process crashes before flushing.

**Hot/warm/cold tier**: a storage tiering strategy for logs. Hot tier (SSD) stores recent logs for fast queries. Warm tier (HDD) stores slightly older logs for slower queries. Cold tier (S3/Glacier) stores archived logs for compliance.

**Log line noise**: the ratio of irrelevant log events to actionable ones. High noise means valuable signals are buried under DEBUG output, health check logs, or repetitive INFO messages. Reducing noise is a key operational goal.

**Multiline log**: a log event that spans multiple lines, such as a Java stack trace or a Python traceback. The log shipper must detect and combine these into a single event to preserve the full context.

**Log buffer**: a temporary storage area where log events are held before being written to the final destination. Buffers improve throughput by batching writes but risk data loss if the process crashes before flushing. The trade-off between buffer size and data loss risk is a fundamental configuration decision.

**Structured metadata**: additional key-value pairs attached to a log event beyond the standard fields. Examples include environment (production, staging), deployment version (v2.3.1), data center (us-east-1a), pod name, and user agent. Metadata enables richer filtering and correlation. Without structured metadata, all logs look the same regardless of where they were generated.

**Log routing**: the process of directing log events to different destinations based on their content. For example, ERROR logs might go to a high-priority index with 7-day retention on SSD, while INFO logs go to a standard index with 30-day retention on HDD. Routing is typically configured in the aggregator layer.

**Log enrichment**: the process of adding context to log events during collection. Examples include adding the Kubernetes pod name and namespace, the data center region, the environment name, and geo-IP data from the client IP address. Enrichment makes logs more useful without changing the application code.

**Log schema**: the definition of which fields a log event can contain and what types those fields have. A good schema is a contract between the application team and the operations team. It defines what fields are searchable, what fields are required, and what fields are deprecated.

**Log lifecycle**: the stages a log event passes through from creation to deletion. The lifecycle includes: creation (application writes the event), collection (shipper reads it), transport (shipper sends it over the network), ingestion (aggregator receives it), indexing (indexer makes it searchable), storage (indexer writes it to disk), query (user searches for it), and deletion (retention policy removes it).

**Bulkhead pattern**: a resilience pattern where the logging system is isolated from the application so that a logging failure does not affect the application. Implementing the bulkhead pattern means using asynchronous loggers, separate threads for log flushing, and circuit breakers for log shipping.

**Log shipping reliability**: the guarantee that the log shipper will deliver each event at least once. Achieving at-least-once delivery requires: (1) writing events to a local file before acknowledging them, (2) tracking the last acknowledged position in the file, and (3) replaying from the last acknowledged position on restart. The cost is potential duplication of events during recovery.

The diagram above shows the end-to-end logging pipeline. Each stage has a specific responsibility:

- **Application process**: generates structured log events as the code executes. Each event is written to stdout (in containerized environments) or to a log file on disk.
- **Log shipper**: runs alongside each application instance, reads log output, optionally parses and enriches it, and forwards it to the aggregation layer. Handles network failures by buffering on disk.
- **Aggregator**: receives logs from many shippers, applies transformations (field extraction, type coercion, enrichment with geo-IP or environment metadata), and publishes to the indexing system.
- **Indexer**: builds inverted indices on key fields (timestamp, level, service, trace ID) so that queries return in milliseconds even across billions of events.
- **Storage**: hot tier (SSD-backed) holds recent logs for fast interactive queries. Cold tier (S3/GCS) stores archived logs at lower cost for compliance and historical analysis.
- **Query interface**: provides the user-facing search and visualization layer. Engineers issue queries like `level:error AND service:auth-service` and get results in seconds.

### Why This Pipeline Exists

The pipeline exists because the application cannot be responsible for its own log storage. If every application process wrote directly to a shared database, the database would be overwhelmed by write throughput, and the application's performance would degrade due to network I/O and lock contention. The pipeline decouples log generation from log storage, allowing each layer to scale independently.

The shipper handles the first-mile problem: getting log data off the application node. The aggregator handles the transformation problem: normalizing data from different sources into a common schema. The indexer handles the search problem: making billions of events queryable in milliseconds. The storage tier handles the durability and cost problem: keeping data safe while managing storage costs.

### Step 2: Application service logs the request

The API gateway or edge service creates a correlation ID and logs the incoming request with all metadata.

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00.123Z", "service": "api-gateway", "trace_id": "abc-123-def", "event": "request_start", "method": "POST", "path": "/api/orders", "client_ip": "203.0.113.5", "user_agent": "Mozilla/5.0"}
```

### Step 3: Service authenticates the user

The auth middleware extracts the JWT, validates the signature, checks expiration, and logs the result.

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00.145Z", "service": "api-gateway", "trace_id": "abc-123-def", "event": "auth_check", "user_id": "user_789", "auth_method": "jwt", "duration_ms": 22, "result": "success"}
```

### Step 4: Service calls the database

The orders service queries PostgreSQL to check inventory. The query and its latency are logged.

```json
{"level": "debug", "timestamp": "2026-07-02T14:30:00.180Z", "service": "orders-service", "trace_id": "abc-123-def", "event": "db_query", "query": "SELECT stock FROM inventory WHERE product_id = ?", "duration_ms": 15, "rows_returned": 1}
```

### Step 5: Service calls a downstream API

The payment service is invoked over HTTP. The request and response are logged, including any retries.

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00.210Z", "service": "orders-service", "trace_id": "abc-123-def", "event": "http_call", "method": "POST", "url": "http://payment-service/charge", "duration_ms": 87, "status_code": 200, "retry_attempt": 0}
```

### Step 6: Service writes to a message queue

The order confirmation is published to a Kafka topic. The producer logs the partition and offset.

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00.300Z", "service": "orders-service", "trace_id": "abc-123-def", "event": "kafka_publish", "topic": "order-confirmed", "partition": 3, "offset": 1402, "duration_ms": 5}
```

### Step 7: Background consumer processes the event

A separate consumer service picks up the message from Kafka and logs the processing.

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00.350Z", "service": "notification-service", "trace_id": "abc-123-def", "event": "process_order_confirmed", "order_id": "ORD-98765", "duration_ms": 120}
```

### Step 8: Notification is sent

The notification service sends an email and logs the result.

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00.480Z", "service": "notification-service", "trace_id": "abc-123-def", "event": "email_sent", "recipient": "user@example.com", "template": "order_confirmation", "provider": "sendgrid", "duration_ms": 130}
```

### Step 9: Response is returned

The API gateway logs the final response with total duration and status code.

```json
{"level": "info", "timestamp": "2026-07-02T14:30:00.500Z", "service": "api-gateway", "trace_id": "abc-123-def", "event": "request_end", "status_code": 201, "total_duration_ms": 377, "bytes_sent": 1204}
```

### Step 10: An error occurs in a parallel path

A separate request hits a database timeout. The error is logged with full context.

```json
{"level": "error", "timestamp": "2026-07-02T14:30:00.520Z", "service": "inventory-service", "trace_id": "ghi-456-jkl", "event": "db_timeout", "query": "UPDATE stock SET quantity = quantity - 1 WHERE product_id = ?", "duration_ms": 30000, "error": "connection timeout", "pool_state": {"active": 24, "idle": 0, "pending": 8, "max": 30}}
```

The correlation ID `abc-123-def` allows an operator to reconstruct the entire 377ms journey across five services. Without it, each log line is an orphan — you see an email was sent but have no idea what order triggered it.

### Step 11: Alert fires based on error rate

A monitoring system counts `level:error` events per minute. When the rate exceeds a threshold (50 errors per minute), an alert fires. The operator opens the logging dashboard, searches for `level:error AND service:inventory-service`, and identifies the database connection pool exhaustion.

### Step 12: Root cause analysis with the correlation ID

Using the correlation ID `ghi-456-jkl` from the failed request, the operator traces back through the inventory service's logs and finds that the connection pool had 30 maximum connections, all 30 were active, and 8 queries were pending. The database server was overloaded because a batch job was running simultaneously. The fix: add a read replica for the batch job, or throttle the batch job during peak hours.

### Why Each Step Matters

Every one of the 12 log events above serves a specific purpose. The `request_start` event establishes that the request arrived. The `auth_check` event confirms the user was authenticated and records which user. The `db_query` events show what data was read and written at each step. The `http_call` events show downstream service dependencies. The `kafka_publish` event shows async communication. The `request_end` event provides the total duration. The `db_timeout` event captures the error with full context about the pool state.

Without any one of these events, the operator would have a gap in their understanding. For example, if the `kafka_publish` event were missing, the operator would not know that the order confirmation was sent to the queue, and they would assume the notification service was never triggered.

## Log Levels in Detail

Choosing the right log level for each event is a skill that develops with experience. Here is a practical guide for each level.

### FATAL

The system is unusable. The application cannot continue running. A FATAL log is typically followed by a process exit. Examples: configuration file is missing on startup, a required database migration cannot be applied, the JVM runs out of native memory.

Only log at FATAL when the application is about to crash. Operators should treat FATAL as a pageable event. No more than 1 FATAL log per 100,000 requests.

### ERROR

Something went wrong, but the application can continue serving other requests. The error needs investigation. Examples: a database query fails, an external API returns a 500, a message cannot be deserialized, a cache write fails.

ERROR logs should always include enough context to diagnose the issue: the operation that failed, the error message, the relevant IDs, and the state of the system at the time of the failure. A good ERROR log is self-contained — an operator should not need to read the source code to understand what failed.

### WARN

Something unexpected happened, but the system handled it gracefully. The event may indicate a future problem. Examples: a retry succeeded on the second attempt, a deprecated API endpoint was called, a configuration value is close to its limit, a cache miss is more frequent than expected.

WARN logs are often the first sign of a developing problem. If you see the same WARN log repeatedly, it is time to investigate. Teams should review WARN logs weekly as part of operational hygiene.

### INFO

Normal, expected behavior that is useful for understanding system operation. Examples: a request starts and ends, a user logs in, a payment is processed, a background job runs, a configuration change is applied.

INFO logs are the backbone of operational visibility. They should tell the story of what the system is doing. Each INFO log should be a complete sentence: "request_start", "auth_check", "email_sent". The combined INFO logs across all services should allow an operator to reconstruct the flow of any request.

### DEBUG

Detailed information that helps a developer diagnose a specific code path. Examples: the value of a variable, the result of a conditional check, the iteration count of a loop, the intermediate state of a computation.

DEBUG logs are for developers, not operators. They are typically disabled in production. When enabled, they generate 10-100x more volume than INFO logs. Use DEBUG for events that are only useful when you are reading the source code.

### TRACE

The most granular level. Every function entry and exit, every line of a loop, every cache lookup. TRACE logs are used during development and never enabled in production. They are typically written to the console during development and discarded.

### Variant 2: Structured JSON logging (modern, containerized)

Modern microservices emit structured JSON logs to stdout. In Kubernetes, container stdout is automatically captured by the container runtime and forwarded by a daemon set running on each node. This is the most common pattern for cloud-native applications.

Benefits:
- Fields are automatically parsed by the log shipper — no regex required.
- Indexing is done on field names, not on full-text search.
- Tools like jq can process logs directly from the command line.
- Schema evolution is straightforward: add a new field to the JSON without breaking existing parsers.

```json
{"level":"info","service":"user-service","event":"user_created","user_id":12345,"duration_ms":12}
```

### Variant 3: Binary/protobuf logging (high throughput)

At very high throughput (over 500,000 events per second per node), JSON serialization becomes a bottleneck due to string allocation and garbage collection. Systems like Apache Kafka, ScyllaDB, and custom game servers use binary log formats (protobuf, FlatBuffers, or custom wire formats) to reduce CPU and memory overhead.

The trade-off is that logs are not human-readable on disk — you need a decoder tool to inspect them. This is acceptable when the log pipeline is the primary consumer and human debugging is done through the query interface, not by tailing files.

### Variant 4: Logfmt logging (Go ecosystem)

Logfmt is a key=value log format popularized by the Go ecosystem. It is more readable than JSON on the command line while still being machine-parseable. Example:

```text
level=info ts=2026-07-02T14:30:00Z service=user-service event=user_created user_id=12345 duration_ms=12
```

Logfmt strikes a balance between plaintext and JSON: it is more structured than plaintext and more readable than JSON. However, it requires more careful parsing for values with spaces or special characters, and it does not support nested structures natively.

## Mental Model Check

Test your understanding with these five questions.

**Question 1**: You have a service that writes plaintext logs like `"ERROR: could not reach database"`. A colleague wants to switch to structured JSON. What are the top three benefits?

**Answer**: (1) Queryability — you can filter by `level:error` without grepping. (2) Field extraction — values like database hostname and error code are separate fields, not embedded in a string. (3) Schema evolution — adding a new field does not break existing parsers; the field is simply absent in older entries.

**Question 2**: What is the difference between a log shipper and a log aggregator?

**Answer**: A log shipper is a lightweight agent running on each node that collects logs and forwards them. It handles backpressure, buffering, and basic filtering. A log aggregator is a central service that receives logs from many shippers, applies transformations (parsing, enrichment, routing), and publishes to the storage/indexing layer. The shipper is distributed; the aggregator is a central pipeline.

**Question 3**: Your team has 50 microservices, each writing 10,000 log events per second. Your Elasticsearch cluster has 10 nodes. What is the approximate ingestion rate, and what factors limit it?

**Answer**: 50 services x 10,000 events/s = 500,000 events/s total. At 500 bytes per JSON event, that is 250 MB/s of raw data. Elasticsearch ingestion is limited by (a) indexing throughput per node — roughly 10,000-20,000 events/s per node depending on the number of fields and mappings, (b) the number of shards and their distribution across nodes, (c) the refresh interval — increasing from 1s to 30s improves bulk throughput by 3-5x, and (d) disk I/O on the hot tier — SSD-backed nodes are essential.

**Question 4**: You find that log events are being silently dropped during peak traffic. What are three possible causes?

**Answer**: (1) The log shipper's buffer is full and its disk queue is full — events are discarded. (2) The aggregator cannot keep up and applies backpressure, causing shippers to timeout and drop batches. (3) The application's logging library uses an asynchronous, non-blocking appender with a fixed-capacity queue — when the queue is full, events are discarded silently.

**Question 5**: When should you log at DEBUG level versus INFO level?

**Answer**: INFO is for events that an operator needs to understand normal system behavior: request starts and ends, database queries, external API calls, authentication decisions, configuration changes. DEBUG is for events that help a developer diagnose a specific code path: variable values, loop iterations, conditional branches, intermediate computation results. In production, DEBUG is typically disabled because the volume is 10-100x higher than INFO and the signal-to-noise ratio is low for operations. You enable DEBUG when troubleshooting a specific issue, then disable it again.

## Common Logging Patterns

Beyond the basics, several patterns recur across logging systems.

### Pattern 1: Request-scoped logging

Every log event within a request scope carries the same correlation ID. This is the most important pattern in distributed logging. Without it, you cannot connect events across services.

```python
import structlog

logger = structlog.get_logger()
logger.bind(trace_id=request.headers.get("X-Trace-ID"))
logger.info("request_start", path="/api/orders")
```

### Pattern 2: Contextual enrichment

Add context to your log events as they flow through the system. A database query log should include the query, the duration, and the result count. An HTTP call log should include the URL, method, status code, and duration.

```python
logger.info("db_query", query="SELECT * FROM users", duration_ms=15, rows=42)
```

### Pattern 3: Error aggregation

When an error occurs, log the error type, the error message, the stack trace, and the state of the system at the time of the error. Do not log the error at multiple levels (e.g., both WARN and ERROR for the same event).

```python
try:
    process_payment(order_id, amount)
except PaymentError as e:
    logger.error("payment_failed", order_id=order_id, amount=amount, error=str(e))
    # Do NOT also log at WARN level here
```

### Pattern 4: Health check suppression

Health check endpoints generate log noise. Suppress them at the shipper level or filter them in the application.

```python
if request.path != "/healthz":
    logger.info("request_start", path=request.path)
```

### Pattern 5: Startup and shutdown logging

Log when the application starts, when it connects to dependencies, and when it shuts down. These events are critical for understanding deployment issues.

```python
logger.info("app_starting", version="2.3.1", config_path="/etc/app/config.yaml")
logger.info("db_connected", host="db-primary.example.com", pool_size=30)
logger.info("app_shutting_down", reason="SIGTERM received")
```

## Distributed Logging Challenges

Logging in a distributed system introduces challenges that do not exist in a single-process application.

### Challenge: Clock Synchronization

In a distributed system, each node has its own clock. If the clocks are not synchronized, log events from different nodes appear in the wrong order. A database query log may appear 2 seconds before the request that triggered it, making debugging confusing and timeline reconstruction impossible.

The solution is NTP (Network Time Protocol) running on every node with frequent synchronization. For most systems, NTP with 60-second sync intervals provides millisecond accuracy. For systems that need sub-millisecond accuracy, PTP (Precision Time Protocol) with hardware timestamping is required.

### Challenge: Volume and Cost

Each microservice generates its own log stream. With 50 microservices handling 10,000 requests per second each, the total log volume is 500,000 events per second. At 500 bytes per event, that is 250 MB/s of ingestion. Storage cost alone is $50,000-$100,000 per month for a self-hosted solution.

The solution is sampling: log only a fraction of high-volume events. The key is to sample intelligently — capture all events for failed requests while sampling 1:100 for successful requests.

### Challenge: Correlation Across Services

Without a shared identifier, log events from different services are orphans. You cannot connect a payment failure in the payment service to the request that triggered it in the API gateway.

The solution is the correlation ID: a unique identifier generated at the edge of the system and propagated through every service via HTTP headers. Every log event includes the correlation ID, enabling operators to search for all events related to a single request.

### Challenge: Structured Schema Evolution

As the system evolves, the log schema changes. New fields are added, old fields are deprecated, and field types change. If the indexer does not handle schema changes gracefully, indexing breaks.

The solution is to use a flexible schema at the indexer (Elasticsearch dynamic mapping) and enforce schema constraints at the application level. New fields are automatically added to the index. Deprecated fields are gradually removed from queries but remain in the index until the retention period expires.

### Challenge: Multi-Tenancy

In a shared logging system, each team or service should only see their own logs. Access control is required to prevent one team from accidentally deleting another team's data.

The solution is to index logs from different teams in separate indices with separate access control policies. In Elasticsearch, this is achieved with index-level security and role-based access control (RBAC).

## How Logging Fits Into Incident Response

Logging is not just about recording events — it is a tool for incident response. Here is how logging supports each phase of an incident.

### Detection

Alerts fire based on log patterns: "5 or more ERROR events from auth-service in 1 minute." The alerting system queries the logging system in real-time and sends a notification to the on-call engineer.

### Triage

The on-call engineer opens the logging dashboard and searches for the pattern that triggered the alert. They look at the ERROR events in the affected service and read the error messages. They can often determine the root cause from the log message alone: "database connection pool exhausted" is a specific signal that points to a database capacity issue.

### Diagnosis

The engineer searches for related log events using the correlation ID. They trace the failing request across all services, looking at timestamps and durations. They identify which service is the bottleneck and what error occurred at each hop.

### Resolution

The engineer applies a fix (restart the database, scale the pool, roll back a deployment). They search the logs to confirm that the error rate is dropping and the system is recovering.

### Post-Mortem

The engineering team reviews the logs from the incident period. They identify the sequence of events that led to the failure and determine what monitoring gaps exist. They add new log events and alerts to prevent the same incident from recurring.
