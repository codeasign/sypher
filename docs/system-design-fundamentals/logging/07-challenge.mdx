## Tier 1: Foundation

These challenges test your understanding of logging fundamentals. Each should take 10-15 minutes.

### Challenge 1: Log Level Audit

You inherit a codebase where every log statement uses `logger.info()`. The application runs at 200 requests per second, and each request generates 15 INFO-level log events. The operations team is complaining that the logs are too noisy and they cannot find real errors. The application also has a 500ms p99 latency requirement.

**Task**: Design a log level strategy that reduces noise without losing critical information. Specify which events should be at which level and how to configure the change.

**Model Answer**: Start by categorizing the 15 log events per request:

- INFO (keep): request start, auth decision, database query (with duration), external API call, response sent. These are business-relevant events that operators need to understand normal system behavior.
- DEBUG (move): loop iteration count, variable values, intermediate computation results, cache hit/miss, function entry/exit. These are only useful during development debugging.

Implementation steps:
1. Change the 5 business-relevant events to INFO and the remaining 10 to DEBUG.
2. Set the production log level to `INFO`.
3. Verify that the application still logs 5 events per request (1000 events/s instead of 3000 events/s).
4. Enable DEBUG level per-service via an environment variable for debugging sessions.

Impact: log volume drops by 67 percent, signal-to-noise ratio improves, and the 500ms latency SLA is unaffected because the logging library only evaluates the log message string if the level is enabled.

### Challenge 2: Log Rotation Setup

Your application runs on a single server with a 100 GB disk. The application writes logs to `/var/log/app/app.log`. Currently, there is no log rotation. The application generates 10 GB of logs per day. You need to ensure the disk never fills up and that logs are available for at least 7 days.

**Task**: Design the log rotation strategy. Specify the rotation frequency, file size limits, retention, and compression settings.

**Model Answer**: Configure log rotation with these parameters:

- Max file size: 500 MB. At 10 GB/day, this creates 20 files per day.
- Max backup files: 140. At 20 files per day, this gives 7 days of retention.
- Compression: enabled. Compressing old files reduces storage by 5-10x.
- Naming: `app.log.YYYY-MM-DD.N` (dated, numbered).
- Post-rotation action: signal the application to reopen the log handle.

```python
import logging.handlers

handler = logging.handlers.RotatingFileHandler(
    "/var/log/app/app.log",
    maxBytes=500 * 1024 * 1024,  # 500 MB
    backupCount=140
)
handler.setFormatter(logging.Formatter(
    "%(asctime)s %(levelname)s %(message)s"
))
```

Storage calculation:
- 140 files x 500 MB = 70 GB max for uncompressed logs
- With compression: approximately 10-15 GB for 7 days of compressed logs
- Headroom: 85 GB free on a 100 GB disk

The disk usage is 15 percent with compression, leaving 85 percent for the application and system. This is safe.

### Challenge 3: Correlation ID Propagation

You have 5 microservices (api-gateway, orders, payments, inventory, notifications). When a request fails in the payments service, you cannot find the corresponding log events in the api-gateway because there is no shared identifier.

**Task**: Design a correlation ID propagation scheme. Specify where the ID is generated, how it is passed between services, and how it appears in log events.

**Model Answer**: The correlation ID is generated at the entry point (api-gateway) and propagated through HTTP headers:

1. **Generation**: the api-gateway generates a UUID v4 at the start of each request. If the incoming request already has an `X-Correlation-ID` header (from the client or a load balancer), it is used instead.

2. **Propagation**: each service passes the correlation ID to downstream services via the `X-Correlation-ID` HTTP header. The OpenTelemetry library handles this automatically if configured.

3. **Logging**: every log event includes the correlation ID as a structured field:

```json
{"level": "info", "service": "payments", "trace_id": "abc-123-def", "event": "charge_attempt", "amount": 99.95}
```

4. **Middleware**: a middleware function in each service extracts the correlation ID from the incoming request header and makes it available to the logging context:

```python
import uuid
from flask import g, request

@app.before_request
def ensure_correlation_id():
    g.trace_id = request.headers.get(
        "X-Correlation-ID",
        str(uuid.uuid4())
    )

@app.after_request
def add_correlation_header(response):
    response.headers["X-Correlation-ID"] = g.trace_id
    return response
```

5. **Querying**: to debug a failed request, search for `trace_id:abc-123-def` in the logging system. All 5 services' log events appear in chronological order.

These challenges require you to design a logging system for a specific scenario. Each should take 20-30 minutes.

### Challenge 4: Logging for a Growing Startup

A startup has 10 microservices running on 20 Kubernetes pods. They currently use `kubectl logs` to debug issues. They are growing and expect to have 100 microservices on 500 pods within 6 months. Their monthly budget for observability is $5,000.

**Task**: Design a logging system that supports their current needs and scales to their 6-month target. Include component choices, estimated costs, and a migration plan.

**Model Answer**: The current state (kubectl logs) is fine for 10 services but will break at 100 services. The 6-month target requires a centralized system.

**Phase 1 (current, 0-2 months)**: deploy a lightweight centralized logging stack:
- Log shipper: Filebeat (DaemonSet, 0.1 CPU per node)
- Buffering: none needed at this volume
- Storage/index: Grafana Loki single binary (no Elasticsearch overhead)
- Query: Grafana (already used for metrics)
- Estimated cost: $200/month (2 CPU cores, 8 GB RAM, 100 GB SSD)

**Phase 2 (growth, 2-6 months)**: migrate to a more scalable stack:
- Log shipper: Filebeat or Vector (scale with pods)
- Buffering: Kafka (3 brokers, small instances)
- Storage/index: Elasticsearch (3 data nodes) or migrate to a managed service
- Query: Grafana or Kibana
- Estimated cost: $1,000/month (self-hosted) or $3,000/month (managed Elastic Cloud)

**Phase 3 (6-month target)**: evaluate managed services:
- Grafana Cloud Loki: $0.50/GB ingested, approximately 100 GB/day = $1,500/month
- Datadog: $1.50/GB, approximately 100 GB/day = $4,500/month (over budget)
- Self-hosted ELK: $2,000/month (infrastructure) + $1,000/month (ops time)

The recommendation: start with Grafana Cloud Loki. It is within budget, requires minimal operational overhead, and scales to 500 GB/day without architectural changes.

### Challenge 5: Real-Time Log-Based Alerting

You need to build a system that fires an alert when any of these patterns appear in logs:
- 5 or more 500 errors from the same service in 1 minute
- Any log line containing "database connection pool exhausted"
- P99 latency exceeding 2 seconds for any service

**Task**: Design the alerting pipeline. Specify the architecture, the query language, and the alert routing.

**Model Answer**: The architecture uses a stream processing engine (Flink or Kafka Streams) to evaluate alert rules in real-time:

1. **Real-time rules** (pattern matching): "database connection pool exhausted" is matched by a simple filter on the log message. A Kafka Streams application reads from the log topic, filters for the pattern, and publishes to an alert topic.

2. **Windowed aggregation rules** (rate-based): "5 or more 500 errors in 1 minute" uses a tumbling window of 60 seconds. The stream processor counts errors per service within each window and publishes an alert when the count exceeds 5.

3. **Metric-based rules** (p99 latency): P99 latency is a metric, not a log. The alerting system queries the metrics store (Prometheus) for the p99 latency metric. If it exceeds 2 seconds, fire an alert.

```python
# Pseudo-code for stream processing alert rules
from kafka import KafkaConsumer, KafkaProducer
from collections import defaultdict
import time

consumer = KafkaConsumer("application-logs")
producer = KafkaProducer()
error_counts = defaultdict(int)  # service -> count
window_start = time.time()

for event in consumer:
    log = parse_event(event)
    
    # Rule 1: Pattern match (instant)
    if "database connection pool exhausted" in log.message:
        producer.send("alerts", {"type": "critical", "message": log.message})
    
    # Rule 2: Rate-based (windowed)
    if log.level == "error" and log.status_code >= 500:
        error_counts[log.service] += 1
    
    # Reset window every 60 seconds
    if time.time() - window_start >= 60:
        for service, count in error_counts.items():
            if count >= 5:
                producer.send("alerts", {
                    "type": "high_error_rate",
                    "service": service,
                    "count": count,
                    "window": "60s"
                })
        error_counts.clear()
        window_start = time.time()
```

Alert routing: critical alerts (pattern match) go to PagerDuty for immediate response. Warning alerts (rate-based) go to Slack for the on-call team. Info alerts (latency warnings) go to a dashboard.

### Challenge 6: Multi-Region Centralized Logging

A company operates in 3 regions (US-East, EU-West, AP-Southeast). Each region has 200 microservices generating 500 GB of logs per day. The company needs to search logs across all regions with sub-5-second query latency for the last 7 days. The budget is $200,000 per month.

**Task**: Design the multi-region logging architecture. Specify the component placement, data flow, query strategy, and cost breakdown.

**Model Answer**: Each region runs an independent logging stack. A global query layer fans out queries to all regions.

**Per-region stack**:
- Ingestion: Vector (DaemonSet on each node)
- Buffering: Kafka (3 brokers, 12 partitions, 48-hour retention)
- Indexing: Elasticsearch (6 hot nodes, 3 warm nodes per region)
- Storage: hot tier = 30-day SSD, warm tier = 90-day HDD
- Cold storage: S3 per region, 1-year retention

**Global query layer**:
- A stateless API service that accepts queries and fans them out to each region's Elasticsearch cluster
- Results are merged, deduplicated, and sorted by timestamp
- Query timeout: 5 seconds
- Cache: Redis for frequent queries (same query within 5 minutes)

**Cost breakdown (per region)**:
- Elasticsearch hot nodes: 6 x $500 = $3,000/month
- Elasticsearch warm nodes: 3 x $200 = $600/month
- Kafka brokers: 3 x $300 = $900/month
- S3 cold storage: 15 TB x $0.023/GB = $345/month
- Network transfer: moderate (cross-region queries only for specific searches)
- Per-region total: approximately $5,000/month

**Total cost**: 3 regions x $5,000 + $2,000 (global query layer) = $17,000/month. This is well within the $200,000 budget.

**Query flow**:
1. User submits a query with a time range and filter.
2. The global query layer sends the query to all 3 regions in parallel.
3. Each region's Elasticsearch cluster executes the query against its local data.
4. Results are returned to the global layer, merged, and sorted.
5. The user sees a unified result set.

For queries covering data older than 7 days, the global layer queries the S3 cold storage via Athena or Presto, with higher latency (10-30 seconds).

### Challenge 7: Zero-Trust Logging

You are designing a logging system for a financial platform that processes $10 billion in daily transactions. The system must satisfy SOC 2 Type II, PCI-DSS, and GDPR requirements. Key constraints:

- Audit logs must be immutable: no one, including the database administrator, can modify or delete them.
- Audit logs must be retained for 7 years.
- Sensitive data (credit card numbers, PII) must never appear in logs.
- The system must detect and alert on tampering within 5 minutes.
- Query performance for audit logs: any query across 7 years of data must return within 1 minute.

**Discussion Points**:

**Immutable storage**: write audit logs to a write-once-read-many (WORM) storage system. S3 Object Lock in compliance mode prevents any object from being deleted or overwritten for the retention period. The lock is applied at the bucket level with a 7-year retention. Even the root AWS account cannot delete locked objects.

**Hash chain**: each log entry contains the SHA-256 hash of the previous entry. This creates a chain where modifying any entry breaks the chain. Periodically (every hour), the latest hash is published to a public blockchain or a separate secure storage for verification.

```python
import hashlib

class AuditLogger:
    def __init__(self, initial_hash=None):
        self.prev_hash = initial_hash or "0" * 64
    
    def log(self, event):
        entry = {
            "timestamp": datetime.utcnow().isoformat(),
            "event": event,
            "prev_hash": self.prev_hash,
        }
        entry_hash = hashlib.sha256(
            json.dumps(entry, sort_keys=True).encode()
        ).hexdigest()
        entry["hash"] = entry_hash
        self.prev_hash = entry_hash
        
        # Write to WORM storage
        s3.put_object(
            Bucket="audit-logs",
            Key=f"logs/{datetime.utcnow():%Y/%m/%d/%H/%M}_{uuid.uuid4()}",
            Body=json.dumps(entry),
            ObjectLockMode="COMPLIANCE",
            ObjectLockRetainUntilDate=datetime(2033, 7, 2),
        )
        return entry_hash
```

**Sensitive data redaction**: apply redaction at the application layer before the log entry is serialized. Use a whitelist approach: only fields explicitly marked as safe are logged. A separate redaction service scans the log stream for any missed sensitive data and quarantines violating entries.

**Query performance**: store audit logs in Parquet format partitioned by year/month/day. Use Presto/Trino for SQL queries. Presto can scan 10 TB of Parquet data in under 30 seconds on a 10-node cluster. For common queries (specific user, specific date range), maintain a materialized view in Elasticsearch separate from the immutable storage.

**Tamper detection**: a background process runs every 5 minutes, reads the hash chain from the beginning, and verifies that every entry's hash matches the recomputed value. If a mismatch is found, an alert fires immediately.

### Challenge 8: Logging as a Product

Your company is building a "Logging as a Service" platform for external customers. Each customer sends 10,000 to 100 million log events per day. You need to support multi-tenancy with strict isolation, per-customer retention policies, per-customer query performance SLAs, and per-customer pricing (by volume, retention, and query count).

**Discussion Points**:

**Multi-tenancy strategy**: use a separate Elasticsearch index per customer. Each index has its own mapping, retention policy, and replica count. This provides strong isolation: one customer's query cannot affect another customer's data. The downside is the overhead of managing thousands of indices — Elasticsearch has a practical limit of approximately 1,000 indices per cluster.

**For more than 1,000 customers**: use a routing layer that maps customer IDs to specific Elasticsearch clusters. Each cluster holds 500-1000 customers. A gateway service inspects the API key, determines the customer's cluster, and routes the query.

**Pricing model**: charge by:
- Volume ingested: $0.50 per GB ingested, with tiered discounts (10 TB+ at $0.30/GB)
- Retention: $0.10 per GB per month for hot storage, $0.01 per GB per month for cold storage
- Query count: 10,000 free queries per month, then $0.001 per additional query
- Sampling: offer a lower-cost plan that samples 1:100 events

**Performance isolation**: use Elasticsearch's `index.routing.allocation` to pin high-paying customers to dedicated nodes. Use `thread_pool.write.queue_size` and `thread_pool.search.queue_size` to prevent noisy neighbors on shared nodes.

**Operational complexity**: the platform team must manage:
- Cluster health monitoring (disk usage, CPU, memory, query latency)
- Index lifecycle management (per-customer retention policies)
- Capacity planning (adding nodes as customer volume grows)
- Disaster recovery (cross-region replication for critical customers)
- Security (API key management, encryption at rest and in transit)

This is the business model of Datadog, Splunk, Grafana Cloud, and every other logging SaaS provider. The engineering challenge is not just the technology — it is the operational automation at scale.

### Challenge 9: Logging Migration Strategy

You are joining a company that has been using a homegrown logging system for 5 years. The system works but is struggling to scale: it uses MySQL for storage, has no structured logging, and has no correlation IDs. The team wants to migrate to a modern stack without losing access to historical logs.

**Tasks**:
1. Design the migration strategy (phases, timeline, risk mitigation).
2. Specify how to handle historical data.
3. Design the dual-write phase where both systems run simultaneously.
4. Specify the cutover criteria.

**Discussion Points**:

**Phase 1: Instrumentation (2 weeks)**: add structured JSON logging to all services. Keep the existing plaintext logging running in parallel. Add correlation IDs to every request. This phase does not change the storage backend — it only changes the log format.

**Phase 2: Parallel pipeline (4 weeks)**: deploy the new stack (Filebeat, Kafka, Elasticsearch) alongside the existing MySQL-based system. Route all new logs to both systems. The existing system continues to receive logs. The new system is validated for correctness and performance.

**Phase 3: Validation (2 weeks)**: compare the two systems. Verify that every log event in the old system appears in the new system. Verify that search queries return the same results. Tune the new system's performance.

**Phase 4: Cutover (1 week)**: switch the primary query interface to the new system. Keep the old system running in read-only mode. Monitor for issues. The cutover is complete when the team has used the new system for 1 week with no issues.

**Phase 5: Historical migration (4 weeks)**: export historical logs from MySQL, convert to structured JSON, and index into the new system. This is done in parallel with normal operations. The historical data is not available through the new system until the migration is complete, but the old system continues to serve historical queries.

**Historical data handling**: do not migrate all historical data immediately. Only migrate the last 90 days of data. Older data is archived to S3 and made available through a separate query interface (Presto/Trino). This avoids a 5-year data dump that would overwhelm the new system.

**Risk mitigation**: keep the old system running for 30 days after cutover. If the new system has issues, fall back to the old system. The dual-write phase ensures that no data is lost during the migration.

### Challenge 10: Logging for a Real-Time Analytics Platform

You are building a real-time analytics platform that processes 1 million events per second. The platform needs to provide sub-second query results for the last hour of data. The logging system for this platform must capture every step of the processing pipeline: ingestion, transformation, aggregation, and output.

**Tasks**:
1. Design the logging system for the analytics platform.
2. Specify how to achieve sub-second query latency for the last hour.
3. Design the sampling strategy for the logging system itself.
4. Specify how to minimize the impact of logging on the analytics pipeline's throughput.

**Discussion Points**:

**Challenge**: the analytics pipeline processes 1 million events per second. If the logging system logs every event at every pipeline stage, the logging system would need to handle 4 million events per second (ingestion, transformation, aggregation, output). This is more than the analytics pipeline itself.

**Solution**: the logging system for the analytics pipeline uses a different approach than standard application logging:

1. **Pipeline-stage logging**: only log at pipeline stage boundaries. Do not log every event — log every 1,000th event. This captures the pipeline's behavior without overwhelming the logging system.

2. **Sliding window storage**: for the last hour of data, store logs in Redis with a 60-minute TTL. Redis provides sub-millisecond read latency. Queries for the last hour hit Redis directly, bypassing Elasticsearch.

3. **Aggregated logging**: instead of logging individual events, log aggregated metrics: events processed per second, latency p50/p99/p999, error rate, throughput per stage. These metrics are stored in a time-series database (Prometheus) and queried for real-time dashboards.

4. **Error sampling**: for individual events, only log errors. Successful events are not logged at all. This reduces the logging volume by 99.9 percent (assuming a 0.1 percent error rate).

5. **Separate debug pipeline**: when debugging a specific issue, enable a "verbose mode" that logs every event for a specific pipeline stage. The verbose logs are stored in a separate, short-lived index with 24-hour retention.

**Sub-second query latency**:
- Hot data (last 1 hour): Redis, sub-millisecond
- Warm data (last 24 hours): Elasticsearch with 1-second refresh interval, 10-50ms
- Cold data (older than 24 hours): Elasticsearch with 30-second refresh interval, 100-500ms

**Performance impact**: the logging system must not add more than 1 percent overhead to the analytics pipeline. To achieve this:
- Use async loggers with ring buffers (sub-microsecond per event).
- Limit the log volume to 1,000 events per second per pipeline stage.
- Use a separate CPU core for the log shipper (not shared with the analytics pipeline).
- If the logging system is saturated, drop events before throttling the analytics pipeline.

### Challenge 11: Logging Cost Optimization

Your company's logging bill has grown to $200,000 per month. The CFO has asked you to reduce it by 50 percent without significantly impacting the engineering team's ability to debug production issues.

**Tasks**:
1. Analyze the current logging cost structure.
2. Design a cost reduction plan that achieves the 50 percent target.
3. Specify the impact of each reduction on debugging capability.
4. Design a monitoring system to track costs and prevent future growth.

**Discussion Points**:

**Current cost structure**:
- Elastic Cloud managed service: $150,000/month (100 TB/day ingested, 30-day hot retention, 2x replication)
- S3 cold storage: $30,000/month (3 PB stored)
- Network transfer: $20,000/month

**Cost reduction plan**:

1. **Reduce hot retention from 30 to 7 days (-$50,000/month)**: impact: queries for data older than 7 days use warm tier (HDD) with 1-5 second latency instead of sub-second. Acceptable for most debugging scenarios.

2. **Implement head-based sampling at 1:10 for INFO logs (-$60,000/month)**: impact: 10 percent of INFO-level events are stored. DEBUG and ERROR events are not sampled. 90 percent of debugging scenarios are unaffected because engineers primarily search for ERROR and trace_id.

3. **Remove replicas for non-critical indices (-$20,000/month)**: impact: if a node fails, data on that node is lost. Only apply to low-priority services (development, staging, internal tools). Production services keep 1 replica.

4. **Reduce cold storage retention from 365 to 90 days (-$15,000/month)**: impact: data older than 90 days is permanently deleted. Compliance requirements may prevent this.

5. **Switch from Elastic Cloud to self-hosted Elasticsearch (-$30,000/month)**: impact: requires 1-2 FTE for operations. The $30,000/month savings is offset by the engineering cost.

**Total savings**: $175,000/month (87 percent reduction). The target is 50 percent, so the team can implement a subset of these changes.

**Monitoring system**:
- Track per-service log volume (GB/day) and compare to budgets.
- Alert when a service's log volume exceeds its budget by 20 percent.
- Track the cost per GB and alert when it increases.
- Track the sampling rate and alert if it drops below the configured threshold.
- Generate a monthly cost report for each service and share it with the engineering team.

**Prevention**: implement a log budget system. Each service has a monthly log budget (e.g., 1 TB/month). When the budget is exceeded, the shipper drops events from that service. The budget is configurable per service and reviewed quarterly.
