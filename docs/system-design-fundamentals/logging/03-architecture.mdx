A single-node logging system works fine for a prototype or a small team. But as your organization grows from one service to fifty, from one developer to a hundred, and from 10,000 log events per day to 10 million per minute, the architecture must evolve. This section walks through the progression from a naive implementation to a production-grade, multi-region logging system that handles 10 million events per second with 99.99 percent availability and sub-second query latency.

## Naive Implementation

The simplest possible logging system: every application writes to its own log file on the local disk, and developers SSH into individual machines to inspect logs.

### At 1,000 Users

Your application runs on a single server. Logs are written to `/var/log/app.log`. When you need to debug, you run `tail -f /var/log/app.log` or `grep ERROR /var/log/app.log`. This works perfectly for a single instance.

Problems that appear at this scale:
- No log rotation: the log file grows until it fills the disk.
- No structured format: logs are plain text, hard to grep reliably.
- No correlation ID: you cannot link a log event to a specific request.
- Single point of failure: if the server crashes, the log file may be lost.

### At 100,000 Users

Your application now runs on 10 servers behind a load balancer. Each server has its own log file. To find an error, you must SSH into each server or use a tool like `pdsh` to run grep across all machines in parallel.

Problems that appear at this scale:
- No centralized search: you must search each server individually.
- No correlation across servers: a single request may hit multiple servers, but you cannot trace it.
- Disk space management: log files on each server consume disk space independently.
- No access control: every developer needs SSH access to all servers.
- No alerting: you cannot set up automated alerting on error rates.

### At 10 Million Users

Your application runs on 200 servers across 3 microservices. The naive approach has completely broken down. You cannot SSH into 200 servers. You cannot grep 200 log files. You need a centralized logging system.

### Ingestion Layer

Each node runs a log shipper (Filebeat, Vector, or Fluentd) as a DaemonSet or systemd service. The shipper tails log files, parses JSON, enriches with metadata (hostname, Kubernetes labels, environment), and publishes to the buffering layer.

The shipper must handle:
- **Backpressure**: if the buffering layer is slow, the shipper queues events on disk. When the disk queue is full, it drops events (oldest first).
- **Restart recovery**: the shipper tracks its position in each log file using a registry file. On restart, it resumes from the last committed position.
- **Multi-line events**: stack traces span multiple lines. The shipper must detect and combine them into a single event.
- **Filtering**: noisy events (health checks, heartbeat messages) are dropped at the shipper to reduce volume.

### Buffering Layer

Kafka is the standard buffering layer for production logging. It provides:
- **Durability**: events are replicated across 3 brokers. A single broker failure does not lose data.
- **Ordering**: events within a partition are ordered. Logs from the same service and same shard key (e.g., trace ID) go to the same partition, preserving chronological order.
- **Replay**: consumers can rewind and reprocess events from any point in time. This is invaluable for debugging ingestion pipeline issues.
- **Backpressure handling**: Kafka can absorb traffic spikes that exceed the indexing layer's capacity. Events accumulate in Kafka until the indexer catches up.

```yaml
# Kafka topic configuration for logs
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaTopic
metadata:
  name: application-logs
spec:
  partitions: 24
  replicas: 3
  config:
    retention.ms: 172800000  # 48 hours
    cleanup.policy: delete
    compression.type: gzip
```

### Transformation Layer

Logstash or Vector processes events from Kafka. Each transformation pipeline stage is a filter:

1. **Parse**: extract fields from JSON, grok patterns for legacy plaintext logs, split multi-line stack traces.
2. **Filter**: drop events that match noise patterns (e.g., health check endpoints, debug logs in production).
3. **Enrich**: add geo-IP data from source IP addresses, translate user IDs to usernames, insert environment metadata.
4. **Route**: send events to different Elasticsearch indices based on log level or service name. Errors go to a high-priority index with more replicas; debug logs go to a low-priority index with fewer replicas.
5. **Obfuscate**: redact sensitive fields (credit card numbers, passwords, access tokens) using regex or field path patterns.

### Indexing Layer

Elasticsearch runs as a cluster of nodes. Each node is assigned one or more roles:

- **Data node**: stores indexed data and handles search and aggregation queries. Hot data nodes use SSDs; warm data nodes use HDDs.
- **Master node**: manages cluster state — which nodes are part of the cluster, which shards are allocated where. Three master nodes for production (quorum-based election).
- **Coordinating node**: accepts client requests, routes them to the appropriate data nodes, and merges results. Dedicated coordinating nodes prevent query workload from interfering with indexing.

### Storage Tiering

Index lifecycle management (ILM) moves data through tiers:

| Tier | Storage | Nodes | Replicas | Retention | Cost per GB/month |
|------|---------|-------|----------|-----------|-------------------|
| Hot | SSD (NVMe) | 10 | 1 | 7-30 days | $0.17 |
| Warm | HDD (SATA) | 6 | 1 | 30-90 days | $0.05 |
| Cold | S3/GCS | 0 | 0 | 90 days - 7 years | $0.01 |
| Frozen | S3 Glacier | 0 | 0 | 7+ years | $0.001 |

## Integration Patterns

Logging does not exist in isolation. It integrates with every other system in the observability stack.

### Pattern 1: Logs + Metrics

Metrics provide the "what" (error rate is 5 percent), logs provide the "why" (error is `connection_refused` to database host `db-01`). The integration point is the error count metric: when the metric crosses a threshold, an operator clicks through to the correlating logs.

```python
from prometheus_client import Counter
error_counter = Counter("app_errors_total", "Total errors", ["service", "error_type"])

def handle_request():
    try:
        process_request()
    except ConnectionError as e:
        error_counter.labels(service="orders", error_type="connection").inc()
        logger.error("connection_error", error=str(e), host=e.host)
```

### Pattern 2: Logs + Traces

Distributed tracing provides the request-level view; logs provide the detailed event stream within each span. The correlation ID shared between logs and traces is the key integration point.

### Pattern 3: Logs + Alerting

Log-based alerting fires when a log pattern matches within a time window. This is distinct from metric-based alerting: it can detect patterns that are too rare or complex to express as metrics.

```yaml
# Prometheus Alertmanager rule for log-based alerting
groups:
  - name: logging
    rules:
      - alert: HighErrorRate
        expr: rate(log_events_total{level="error"}[5m]) > 50
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Error rate above 50 per minute for {{ $labels.service }}"
```

### Pattern 4: Logs + Audit

Compliance requirements (SOC 2, HIPAA, PCI-DSS, GDPR) mandate that certain events are logged, tamper-proof, and retained for a specific period. Audit logs are written to a separate, immutable index with append-only access control.

```json
{"level": "audit", "timestamp": "2026-07-02T14:30:00Z", "event": "user_permission_change", "actor": "admin@example.com", "target": "user_12345", "action": "grant_role", "role": "billing_admin", "source_ip": "10.0.0.1"}
```

Audit logs are typically:
- Written to a separate Kafka topic with infinite retention.
- Indexed in a separate Elasticsearch cluster with strict access control.
- Signed with a hash chain to detect tampering (each log entry contains the hash of the previous entry).
- Retained for 3-7 years depending on regulatory requirements.

### Pattern 5: Logs + Debugging

During incident response, operators need to enable detailed logging without restarting services. This is achieved through dynamic log level configuration:

```python
import os

def get_log_level():
    feature_flag = os.environ.get("DEBUG_SERVICE", "")
    if feature_flag == "orders-service":
        return "DEBUG"
    return "INFO"
```

### Pattern 6: Logs + Cost Management

Each log event costs money to store, index, and query. Teams need visibility into their log spending. The integration between logging and cost management uses per-service budgets:

```python
# Per-service log budget tracking
log_budgets = {
    "orders-service": {"gb_per_day": 10, "current_gb": 0},
    "payment-service": {"gb_per_day": 5, "current_gb": 0},
    "notification-service": {"gb_per_day": 2, "current_gb": 0},
}

def can_log(service_name, event_size_bytes):
    budget = log_budgets.get(service_name)
    if not budget:
        return True  # No budget = no limit
    gb_today = budget["current_gb"]
    gb_limit = budget["gb_per_day"]
    if gb_today >= gb_limit:
        logger.warn("log_budget_exceeded", service=service_name)
        return False
    budget["current_gb"] += event_size_bytes / (1024**3)
    return True
```

## Multi-Region Architecture

Cross-region query flow:
1. User submits a query with a time range (e.g., last 24 hours).
2. The global query layer sends the query to each region's Elasticsearch cluster.
3. Each cluster returns results from its local hot tier (last 24 hours is in hot tier).
4. The global layer merges, deduplicates, and sorts results by timestamp.
5. Results are returned to the user.

For queries covering older data, each region's cold tier (S3) is queried. S3 first-byte latency is 5-50ms, so cold queries are slower but still acceptable for historical analysis.

### Cross-Region Data Flow

Data does not flow between regions for normal operation. Each region's logs stay in that region. This is important for:
- **Latency**: the application writes logs to the local shipper with sub-millisecond latency. Cross-region writes would add 50-200ms.
- **Cost**: cross-region data transfer is expensive. Keeping logs local avoids egress charges.
- **Compliance**: some regulations require data to stay within geographic boundaries.

### Failover Strategy

If a region's Elasticsearch cluster fails, the global query layer marks that region as unavailable. Queries return results from the remaining regions. The failed region's logs are still in Kafka and can be replayed when the cluster recovers.

If a region's Kafka cluster fails, the shippers switch to a backup Kafka cluster in another region. The shipper's disk queue provides 1-5 GB of buffer, which gives the operations team 10-30 minutes to restore the primary cluster.

## Capacity Planning

You need to plan capacity for a system handling 10 million daily active users (DAU) with 50 microservices. Here is the math.

### Assumptions

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| DAU | 10 million | Given |
| Requests per DAU per day | 100 | Typical for a web application |
| Log events per request | 10 | Request start, auth, DB queries, API calls, response end |
| Average event size (JSON) | 500 bytes | After gzip compression in transit |
| Retention (hot) | 30 days | Industry standard |
| Retention (warm) | 90 days | Industry standard |
| Retention (cold) | 1 year | Compliance requirement |
| Replication factor | 2 | One primary + one replica |

### Volume Calculations

**Events per day**:
```
10 million DAU × 100 requests/day × 10 events/request = 10 billion events/day
```

**Events per second (peak)**:
```
Peak factor = 3x average (daily traffic pattern)
Average events/second = 10B / 86400 = 115,740 events/s
Peak events/second = 115,740 × 3 = 347,220 events/s
```

**Storage per day (raw)**:
```
10 billion × 500 bytes = 5 TB/day raw
5 TB × 2 (replicas) = 10 TB/day stored
```

**Storage per month (hot tier)**:
```
10 TB × 30 days = 300 TB
```

**Storage per year (cold tier)**:
```
5 TB × 365 days = 1.8 PB (raw, no replicas in cold)
```

### Elasticsearch Cluster Sizing

Each data node can handle approximately 20,000 events/second for indexing and hold up to 4 TB of data (assuming 10:1 compression ratio, SSD-backed).

**Indexing nodes needed (hot tier)**:
```
347,220 events/s / 20,000 events/s per node = 18 nodes
Rounded up with 20 percent headroom: 22 nodes
```

**Storage nodes needed (hot tier)**:
```
300 TB / (4 TB/node × 0.7 usage limit) = 108 nodes
```

The storage requirement dominates. You need 108 nodes for the hot tier. This is why most organizations use managed Elasticsearch (Elastic Cloud, AWS OpenSearch) or a log-as-a-service provider (Datadog, Splunk, Grafana Cloud) — managing 100+ Elasticsearch nodes is a full-time job for a team.

### Cost Estimate

| Component | Quantity | Unit Cost | Monthly Cost |
|-----------|----------|-----------|--------------|
| Hot tier nodes (SSD) | 108 | $500/node | $54,000 |
| Warm tier nodes (HDD) | 20 | $200/node | $4,000 |
| Kafka brokers | 6 | $500/broker | $3,000 |
| Log shippers | 50 | $0 (existing nodes) | $0 |
| Network transfer | 5 TB/day | $0.02/GB egress | $3,000 |
| S3 cold storage | 1.8 PB | $0.023/GB/month | $42,000 |
| **Total** | | | **$106,000/month** |

This is why managed log services (Datadog at approximately $1.50/GB ingested) often cost more than self-hosting but include the operational expertise. At 5 TB/day, Datadog would cost approximately $225,000/month — roughly double the self-hosted cost.

### Scaling Down: Smaller Systems

Not every system needs 108 Elasticsearch nodes. Here is how the math scales at different sizes:

| System Size | DAU | Events/sec | Hot Storage | ES Nodes | Monthly Cost |
|-------------|-----|------------|-------------|----------|-------------|
| Small startup | 10K | 350 | 30 GB | 1 | $500 |
| Growth stage | 100K | 3,500 | 300 GB | 3 | $3,000 |
| Mid-size | 1M | 35,000 | 3 TB | 10 | $15,000 |
| Large | 10M | 350,000 | 300 TB | 108 | $106,000 |
| Enterprise | 100M | 3.5M | 3 PB | 500+ | $500,000+ |

## Deployment and Operations

      - name: varlog

          path: /var/log

### Operational Runbooks

**Runbook 1: Indexing lag is growing**

1. Check the Kafka consumer lag for the logstash consumer group: `kafka-consumer-groups --bootstrap-server kafka:9092 --group logstash --describe`.
2. If lag is growing, the indexer is the bottleneck. Check Elasticsearch CPU and disk I/O.
3. Increase the number of Logstash workers: `pipeline.workers: 8`.
4. If the Elasticsearch cluster is saturated, temporarily reduce the replica count to 0 to free indexing capacity.
5. Once lag is cleared, restore replicas.

**Runbook 2: Logs are missing for a specific service**

1. Check if the service is running and producing stdout: `kubectl logs $POD --tail=50`.
2. Check if the log shipper is running on the node: `kubectl get pods -n logging -o wide | grep $NODE`.
3. Check the shipper's registry file: `cat /var/lib/filebeat/registry/filebeat/log.json`.
4. Check if the shipper is filtering out the logs: examine the shipper's configuration for `drop_event` or `drop_fields` processors.
5. Check if the Kafka topic is receiving events: `kafka-console-consumer --topic application-logs --from-beginning --max-messages 10`.

**Runbook 3: Disk space is low on hot tier nodes**

1. Check ILM policy: `GET _ilm/policy`. The hot phase rollover should trigger when the index reaches 50 GB.
2. If rollover is not triggering, force a rollover: `POST _rollover/logs-000001`.
3. Check if warm tier migration is working: `GET _ilm/explain/logs-000001`.
4. If the warm tier is full, add more warm nodes or reduce retention.
5. As a last resort, delete old indices: `DELETE logs-000001`.

**Runbook 4: Elasticsearch cluster is red**

When the cluster status is red, at least one primary shard is unassigned. This means data is unavailable.

1. Check unassigned shards: `GET _cat/shards?v=true&h=index,shard,prirep,state,node&s=state`.
2. Check the reason for unassignment: `GET _cluster/allocation/explain`.
3. If a node is down, restart it or replace it.
4. If the node cannot be recovered, reallocate the shards: `POST _cluster/reroute?retry_failed`.
5. If the shard data is lost, reindex from Kafka or accept the data loss.

### Monitoring the Logging System

The logging system itself must be monitored. Key metrics:

| Metric | Warning | Critical | Action |

### Disaster Recovery

The logging system's disaster recovery plan must address three scenarios:

**Scenario 1: Single node failure**
- Elasticsearch: the replica shard on another node takes over. No data loss.
- Kafka: the broker's partitions are reassigned to remaining brokers. No data loss.
- Recovery: replace the failed node. Data rebalances automatically.

**Scenario 2: Entire cluster failure (region)**
- Elasticsearch: data in hot and warm tiers is lost if replicas are in the same region.
- Kafka: data in Kafka is lost if replication is within the same region.
- Recovery: restore from S3 cold storage. Replay Kafka from the last backup.
- RTO: 4-8 hours. RPO: up to 24 hours (last S3 snapshot).

**Scenario 3: Catastrophic failure (multi-region)**
- All regions are simultaneously unavailable.
- Recovery: fail operations to a new region. Restore all data from S3 cold storage.
- RTO: 24-48 hours. RPO: up to 24 hours (last S3 snapshot).

### Migrating Between Tiers

Moving from one deployment model to another is disruptive. Plan the migration carefully:

1. **Phase 1: Dual write**. Run both the old and new logging systems simultaneously. Route a percentage of traffic to the new system while keeping the old system running.
2. **Phase 2: Parallel operation**. Run both systems at full capacity. Verify that the new system produces correct results.
3. **Phase 3: Cutover**. Switch all traffic to the new system. Keep the old system running for 30 days as a fallback.
4. **Phase 4: Decommission**. After 30 days with no issues, shut down the old system.

### Cost Optimization Strategies

Beyond the basic tiered storage, several strategies reduce logging costs:

**Remove replicas for non-critical indices**: application INFO logs can use 0 replicas. Audit logs should use 3 replicas.

**Aggressive sampling**: sample 1:1000 for successful requests. Store 100 percent of failed requests.

**Reduce field indexing**: only index the 5-10 fields that are used in queries. Store the remaining fields without indexing.

**Shorter retention in hot tier**: keep only 7 days in hot SSD storage, then move to warm HDD storage. The warm tier costs 70 percent less.

**Compression at the shipper**: compress logs before sending them to Kafka. zstd compression achieves 6-12x reduction.

**Remove noisy services**: some services generate more logs than all other services combined. Investigate and fix the root cause.

**Use log budgets per service**: each service has a daily log budget. When the budget is exceeded, the shipper drops events from that service.

### Security and Access Control

The logging system contains sensitive data. Access must be controlled:

**Network security**: the logging pipeline runs on a private network. The shipper connects to Kafka over TLS. The indexer connects to Elasticsearch over TLS. The query interface is behind a VPN or SSO.

**Authentication**: the shipper authenticates to Kafka using SASL/SCRAM. The indexer authenticates to Elasticsearch using API keys. The query interface uses OAuth2 with SSO.

**Authorization**: Elasticsearch RBAC controls which users can access which indices. Developers can search logs from their own services. Only the operations team can search all logs. Only the security team can delete logs.

**Audit**: all access to the logging system is logged. Every query, every export, every configuration change is recorded in an audit log that is stored in a separate, immutable index.

### Elasticsearch Index Design

The index design determines query performance and storage efficiency. For logging, use time-based indices with a daily or hourly rollover.

**Index naming convention**: `logs-{service}-{YYYY.MM.DD}`. Example: `logs-orders-2026.07.02`. This naming convention makes it easy to manage retention: delete indices older than 30 days.

**Shard count**: 1 shard per 50 GB of data per day. A service generating 10 GB/day needs 1 shard. A service generating 500 GB/day needs 10 shards. Too many shards wastes memory and slows cluster management. Too few shards limits parallelism and slows queries.

**Replica count**: 1 for hot tier, 0 for warm tier. Replicas provide failover but double the storage cost. For logging, the risk of losing a few hours of data is acceptable for most applications.

**Mapping design**: define explicit mappings for the 10 most commonly queried fields. Use dynamic mapping for the rest. This balances query performance against schema flexibility.
