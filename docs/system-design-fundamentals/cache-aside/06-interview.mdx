## How This Topic Appears

### Category 1: System Design Questions

These questions ask you to design a system, and caching is a key component.

1. "Design Twitter's home timeline. How do you handle 500 million users loading their feeds?"
2. "Design a URL shortening service like TinyURL. How do you handle 10,000 reads per second?"
3. "Design an e-commerce product page. A celebrity mentions a product and 1 million users load the page simultaneously."
4. "Design a real-time leaderboard for a gaming platform with 10 million active players."
5. "Design a social media feed aggregator that pulls from Twitter, Instagram, and LinkedIn."

**What they are testing:** Can you identify read-heavy components, apply caching, consider cache stampede, and reason about staleness trade-offs.

### Category 2: Deep Dive Questions

These questions focus specifically on caching architecture.

6. "Explain cache-aside vs read-through vs write-through. When would you choose each?"
7. "Walk me through what happens when a cache key expires and 1,000 requests arrive at the same time."
8. "How would you design a cache invalidation strategy for a multi-service architecture?"
9. "What happens to your cache when you deploy a new version of your application?"
10. "How do you calculate the memory requirements for a cache serving 50,000 requests per second?"

**What they are testing:** Depth of understanding of cache mechanics, failure modes, and capacity planning.

## Level-by-Level Expectations

### L3 / Junior

**Expectation:** Understand the basic cache-aside pattern. Can explain the difference between a cache hit and a cache miss. Can write pseudocode for cache-aside logic.

**Sample answer for "Explain cache-aside":**
"Cache-aside is a caching pattern where the application checks the cache first. If the data is in the cache (a hit), it returns the cached data. If the data is not in the cache (a miss), it queries the database, stores the result in the cache, and returns it. This improves read performance because the cache is faster than the database."

**Sample answer for "What happens when the cache is full?":**
"The cache evicts old entries based on its eviction policy -- typically LRU, which removes the least recently used entries first. This makes room for new entries without the application crashing."

**Missing the mark:**
- Cannot explain TTL or why it matters
- Does not mention cache stampede as a failure mode
- Thinks the cache is a database replacement

### L4 / Mid-Level

**Expectation:** Understand cache-aside trade-offs, can design a simple caching layer, knows when to use cache-aside vs other patterns. Can identify cache stampede and propose mutex-based mitigation.

**Sample answer for "Design a cache layer for a product catalog":**
"I would use cache-aside with Redis. Each product has a unique cache key like `product:{id}`. Reads check Redis first. On miss, the app queries the database and writes the result to Redis with a 10-minute TTL. On product update, the app deletes the cache key so the next read fetches fresh data. I would set the eviction policy to allkeys-lru and size the Redis cluster based on the number of products -- approximately 10,000 products at 10 KB each = 100 MB, so a single Redis instance with 1 GB is sufficient. I would add a mutex lock with `SET NX` for hot products to prevent cache stampede."

**Missing the mark:**
- Only knows one cache pattern
- Does not consider memory sizing
- Does not handle cache failure (no fallback)
- Does not consider hot keys or stampede

### L5 / Senior

**Expectation:** Deep knowledge of caching internals. Can design for scale (multi-node, replication). Understands serialization overhead, connection pooling, and circuit breakers. Can analyze and fix production caching problems.

**Sample answer for "Design Twitter's home timeline":**
"Twitter's timeline is a fan-out system. When a user tweets, the tweet ID is pre-pushed into the Redis timeline caches of all followers. This is eager population at write time. When a user reads their timeline, the app fetches the list of tweet IDs from Redis and hydrates the tweet content (separate cache-aside per tweet). For users with more than 10,000 followers, Twitter uses a hybrid approach -- followers beyond the first 10,000 pull the tweet on read (lazy fan-out). The Redis cluster is sharded by user ID with consistent hashing. For a celebrity with 50 million followers, the fan-out is too expensive, so the celebrity's tweets are handled via fan-out-on-read: when a follower loads their timeline, the app checks if the celebrity tweeted recently and includes those tweets. Cache memory is sized based on active users: 300 million daily active users, each with approximately 200 tweet IDs in their timeline cache, at 8 bytes per ID = 480 GB of Redis across the cluster."

**Missing the mark:**
- Designs a naive single-instance Redis cache for 500 million users
- Does not consider the difference between active and inactive users
- Does not handle the celebrity follower problem
- Does not consider Redis memory costs

### L6 / Principal

**Expectation:** Can design multi-region cache-aside with geo-distribution. Understands the trade-offs between consistency and performance across regions. Can design for zero-downtime cache cluster upgrades. Thinks about operational costs, capacity planning, and organizational impact.

**Sample answer for "How would you design a global caching strategy for a social network?":**
"I would use a two-tier caching strategy. Each region has a local Redis cluster for cache-aside, with local database read replicas as the miss fallback. This keeps 95% of reads within the region. For cross-region consistency, I would use a publish-subscribe invalidation bus (Kafka): when data is updated in one region, an invalidation event is published to all other regions. The invalidation is async, so cross-region read-after-write consistency is not guaranteed -- this is acceptable for a social network where eventual consistency is the norm. Cache key TTLs vary by data type: session data (15 minutes), user profiles (5 minutes), feed content (1 minute). The cluster is sized at 3x headroom above baseline to handle traffic spikes and node failures. Deployments use blue-green with cache pre-warming to avoid cold starts."

**Missing the mark:**
- Assumes cross-region strong consistency is achievable without significant cost
- Does not consider the invalidation bus as a single point of failure
- Does not optimize for regional traffic patterns

### L7/L8 / Distinguished / Fellow

**Expectation:** Can design caching infrastructure used across the entire organization. Thinks about caching abstractions, developer APIs, and platform tooling. Influences the caching culture and best practices across engineering teams.

**Sample answer for "Design a caching platform for the entire company":**
"I would build a caching-as-a-service platform. Each team registers their cache namespace and configures TTL, eviction policy, and consistency requirements via an API or configuration file. The platform manages the Redis clusters, handles capacity planning, provides a unified client library with built-in circuit breakers, retry logic, and metrics export. The client library supports multiple cache patterns (cache-aside, read-through, write-through) and automatically selects the optimal serialization format based on value size. The platform exposes a dashboard showing hit ratio, latency, memory usage, and cost per team. It automatically scales clusters based on memory utilization and traffic patterns. It provides a warmup API that teams call during deployment to pre-load their hot keys. The platform handles cross-region replication for teams that need it. It integrates with the organization's incident management system to alert when cache hit ratios drop below thresholds."

**Missing the mark:**
- Designs a single Redis cluster for the whole company (no isolation)
- Does not consider multi-tenancy and noise isolation
- Does not provide developer APIs or abstractions

## FAANG Framework for Cache-Aside Questions

Use this 6-step framework when a system design question involves caching.

### Step 1: Identify the Read-Read-Read Components

Before talking about caching, identify the components that are read-heavy. The cache is only useful for data that is read more than once.

```
Traffic breakdown:
  - 90% reads, 10% writes -> caching is essential
  - 50% reads, 50% writes -> caching is optional
  - 20% reads, 80% writes -> caching is detrimental
```

### Step 2: State the Assumption

Explicitly state what you are optimizing for:

"I am assuming 90% of requests are reads, 10% are writes. The read-to-write ratio is 9:1, which makes cache-aside a good fit. I am optimizing for low read latency (under 5 ms p99) and high availability. Eventual consistency within 1 minute is acceptable."

### Step 3: Choose the Pattern

State which caching pattern you are using and why.

"I will use cache-aside with Redis. Cache-aside gives the application explicit control over the cache lifecycle, which is appropriate for this workload because it has a clear read path and a clear write path. Redis is chosen over Memcached because it supports data structures (lists for timeline entries) and has built-in replication for high availability."

### Step 4: Design the Key Structure

Show the cache key naming convention.

"Each entity gets its own cache key:
- User profile: `user:{user_id}:profile` (TTL: 5 minutes)
- User timeline: `user:{user_id}:timeline` (TTL: 7 days)
- Tweet content: `tweet:{tweet_id}` (TTL: 24 hours)

The key includes the entity type and the identifier. The TTL is chosen based on how frequently the data changes and how stale it can be."

### Step 5: Address Failure Modes

Proactively discuss cache stampede (thundering herd), cold start, stale data, and cache node failure.

"Three failure modes I need to address:
1. **Cache stampede:** For hot keys, I will use Redis `SET NX` to create a distributed mutex. Only the lock holder queries the database. Other requests wait briefly and retry.
2. **Cold start:** After deployment, I will run a warmup script that loads the top 10,000 keys into the cache before routing traffic.
3. **Cache node failure:** I will run Redis with replication (one primary, one replica per shard) and multi-AZ deployment. The circuit breaker in the client library will fall back to the database if a node becomes unavailable."

### Step 6: Calculate and Optimize

Show rough capacity planning math.

"With 10 million DAU, each user reads 100 timeline items per day. That is 1 billion reads per day = 11,574 reads per second peak. With 90% hit ratio, the cache handles 10,416 requests per second. Each entry is 8 bytes (tweet ID) + 25 bytes overhead = 33 bytes. For 200 entries per user * 10 million users, that is 66 GB of data. But only 20% of users are active on any given day, so approximately 13 GB of active data. I will provision 3 Redis nodes with 8 GB each = 24 GB total. This gives 1.8x headroom for growth and replication overhead."

## Phrase Bank

Use these phrases in your interview answers to demonstrate fluency.

### Opening the Topic
- "Before I talk about caching, let me identify which components are read-heavy."
- "I will use cache-aside for this component because reads significantly outnumber writes."
- "The key assumption I am making is that eventual consistency within N seconds is acceptable."
- "Let me start by identifying the read-to-write ratio, which determines whether caching adds value."

### Explaining the Pattern
- "Cache-aside means the application explicitly manages the cache lifecycle -- it checks the cache first, queries the database on miss, and populates the cache."
- "The cache sits alongside the database, not between the application and the database."
- "The miss penalty is the database query plus the cache population cost. We want to minimize the miss rate to minimize the miss penalty."
- "Each cache miss pays the full cost of the database query plus the overhead of serialization and network transfer to the cache."

### Addressing Consistency
- "Cache-aside provides eventual consistency bounded by the TTL."
- "For strong consistency, I would use write-through caching or explicit invalidation on writes."
- "The staleness window is at most the TTL duration, but in practice it is shorter because most data changes infrequently."
- "The consistency-performance trade-off is the fundamental question the interviewer wants you to address."

### Handling Failure
- "The cache is an optimization, not a source of truth. The system must function correctly without it."
- "I will implement graceful degradation: if the cache is unreachable, fall back to the database."
- "For cache stampede, I will use a distributed mutex so only one request queries the database when a hot key expires."
- "I will add random jitter to TTLs to prevent batches of keys from expiring simultaneously."
- "I will implement a circuit breaker so that repeated cache failures do not cascade into database failures."

### Talking About Operations
- "I will monitor hit ratio, eviction rate, p50/p99 latency, and memory usage."
- "I will set up alerts: hit ratio below 80% for 5 minutes triggers a PagerDuty notification."
- "Capacity planning: memory = (unique keys in TTL window) * (average entry size + overhead)."
- "I will create a dashboard showing per-node memory, hit ratio trend, and top hot keys."

### Responding to Follow-Up Questions

Interviewers often follow up with specific scenarios to probe your depth. Here is how to handle the most common ones.

**Follow-up: "What if the database write succeeds but the cache delete fails?"**

"The cache delete failure means the stale cache entry remains. The next read returns stale data. The TTL bounds the staleness to at most the configured duration. This is acceptable for an eventually consistent system. To reduce the window, I would retry the cache delete with exponential backoff. If the delete still fails after 3 retries, I log the error and let TTL handle it. In systems where this is unacceptable, I would use a two-phase approach: mark the cache entry as 'pending refresh' and serve stale data only if the refresh flag is not set."

**Follow-up: "How do you handle cache serialization for complex nested objects?"**

"For simple objects (under 10 KB), JSON is the default. It is debuggable, widely supported, and fast enough for most workloads. For complex nested objects or high throughput, I switch to Protocol Buffers. The schema is defined in a .proto file, and the serialization is type-safe and compact. For objects that include binary data (images, files), I store the binary data in blob storage (S3) and cache only the metadata and URL. Serialization format should be a configuration parameter, not hardcoded, so it can be changed without code changes."

**Follow-up: "What happens when you deploy and the cache is full of old-format data?"**

"This is the version skew problem. When you deploy a new application version that changes the serialization format (e.g., adds a new field to a Protocol Buffer), the cache may contain entries in the old format. The solution is to handle both formats during deserialization: try the new format first, and if it fails, fall back to the old format. Alternatively, use a schema version field in the cache key: `user:42:v2:profile`. Old entries with `v1` are automatically missed when the application requests `v2`. The old entries expire naturally via TTL, and the cache gradually fills with v2 entries."

**Follow-up: "How do you test cache-aside behavior in unit tests?"**

"I mock the cache client. The unit test verifies: (1) on cache hit, the database is not called; (2) on cache miss, the database is called and the result is stored in the cache; (3) on cache error, the database fallback is triggered. The mock should be configurable to simulate hits, misses, and errors. Integration tests use a real Redis instance (ephemeral, spun up per test suite) and verify that cache-aside integration works end-to-end."

## Red Flags

### Red Flag 1: Not Mentioning Cache Stampede

If you design a cache system and do not mention what happens when a popular key expires, the interviewer assumes you have not operated a cache in production.

**What to say instead:** "When a hot key expires, I need to handle the thundering herd. I would use a distributed mutex with Redis `SET NX` to ensure only one request queries the database. Other requests wait briefly and retry."

### Red Flag 2: Treating Cache as a Database

If you design around the assumption that the cache always has the data you need, the interviewer knows you misunderstand caching.

**What to say instead:** "The cache is best-effort. Every cache access has a fallback to the database. The cache can be empty, partitioned, or evicting entries at any time."

### Red Flag 3: Not Discussing Memory Sizing

If you propose caching without estimating memory requirements, the interviewer questions your practical experience.

**What to say instead:** "Let me calculate the memory required. Each entry is 5 KB. We have 1 million unique keys accessed within the TTL window. That is 5 GB of data, plus 15% overhead for Redis data structures, total approximately 5.75 GB. I will provision 8 GB per node with 3 nodes for headroom."

### Red Flag 4: No TTL

If you design a cache without TTLs, the interviewer knows you have not thought about stale data.

**What to say instead:** "Every cache entry has a TTL. The TTL depends on the data type: user profiles get 5 minutes, blog posts get 1 hour, configuration gets 2 minutes. The TTL bounds the staleness window."

### Red Flag 5: Assuming Infinite Cache

If you design a cache that holds all data forever, the interviewer questions your understanding of memory constraints.

**What to say instead:** "Cache memory is finite. When memory fills up, the eviction policy (LRU) removes the least recently used entries. This means cold keys are evicted first, and the hit ratio degrades gracefully."

### Red Flag 6: No Serialization Discussion

If you talk about caching without mentioning serialization, you miss a key performance consideration.

**What to say instead:** "Serialization format matters for performance. I will start with JSON for debuggability, then switch to Protocol Buffers if serialization overhead becomes a bottleneck at scale."

### Red Flag 7: Over-Engineering

If you design a multi-region, replicated, sharded cache cluster for 10,000 users, the interviewer thinks you cannot match complexity to scale.

**What to say instead:** "For 10,000 users, a single Redis instance with replication is sufficient. The cache memory requirement is approximately 500 MB. I will scale to multiple nodes when traffic grows beyond 50,000 requests per second."

### Red Flag 8: No Monitoring Plan

If you design the cache but do not mention how you will know it is working, the interviewers see a gap in operational maturity.

**What to say instead:** "I will instrument the cache client to export hit ratio, miss latency, eviction rate, and memory usage. Alerts will fire when hit ratio drops below 80% or miss latency exceeds 200 ms p99."

## Practice Drills

### Drill 1: Design Cache-Aside for a Social Media Post (5 minutes)

Design a cache-aside system for a social media platform's post viewing. Each post has text, images, comments, and like counts. Posts can go viral and receive millions of views.

**Focus areas:** Cache key structure, TTL values, hot key handling (viral posts), cache stampede prevention.

### Drill 2: Plan the Migration from In-Process to Distributed Cache (5 minutes)

You have an application using an in-process Python dict as a cache-aside store. It runs on a single server. The company is scaling to 10 servers. Plan the migration to a distributed cache.

**Focus areas:** Why the in-process cache breaks with multiple servers, choosing Redis, connection pooling, client configuration, deployment strategy.

### Drill 3: Debug a Production Cache Issue (5 minutes)

The cache hit ratio dropped from 92% to 40% over 10 minutes. The application and cache code have not been deployed. Database CPU is at 80%. What could be wrong? How do you diagnose and fix it?

**Focus areas:** TTL-related mass expiry (hash-tag problem), cache node failure, network partition, hot key explosion, deployment of a data change.

### Drill 4: Calculate Memory Requirements (3 minutes)

You are designing a cache-aside system for a product catalog with 500,000 products. Each product is 15 KB of JSON when serialized. The TTL is 1 hour. At peak, 10,000 products per second are accessed. How much Redis memory do you need?

**Focus areas:** Active keys math, overhead calculation, headroom sizing.

### Drill 5: Explain Cache-Aside vs Read-Through (2 minutes)

Explain the difference between cache-aside and read-through to a non-technical product manager. Why would you choose one over the other?

**Focus areas:** Clear communication, eliminating jargon, focusing on trade-offs that matter to the business (development cost vs maintenance cost).

### Drill 6: Design Invalidation Strategy for a Comment System (5 minutes)

Design a cache-aside invalidation strategy for a blog comment system. Users can post comments, edit comments, and delete comments. Comments are displayed on blog post pages. The system serves 10 million readers per day.

**Focus areas:** When to invalidate (on create, update, delete), granularity (per-post vs per-comment), TTL trade-offs, handling the read-after-write consistency problem.

### Drill 7: Handle a Cache Node Failure in a 10-Node Cluster (3 minutes)

You have a 10-node Redis cluster serving cache-aside for a read-heavy API. One node fails completely. Walk through the impact, the immediate response, and the recovery process.

**Focus areas:** Impact on hit ratio (drops from 95% to approximately 85.5%), impact on database (10x more queries for the affected keys), consistent hashing redistribution, auto-failover, recovery steps.

### Drill 8: Optimize a Slow Cache-Aside System (5 minutes)

Your cache-aside system has a 95% hit ratio but the p99 latency for cache hits is 50ms. The cache and application are in the same data center. What is wrong and how do you fix it?

**Focus areas:** Network latency vs serialization vs connection pooling. Check: (1) Is the connection pool too small? (2) Is the serialization format inefficient? (3) Are there large values causing network saturation? (4) Is the Redis instance overloaded (slow commands, high CPU)?

I proposed adding a cache-aside layer with Redis. **What the interviewer is looking for:** Did you identify the bottleneck? Did you choose the right solution? Can you articulate the impact in business terms?

**Cache-aside angle:** "I deployed a new version of the product service that changed the cache serialization format from JSON to Protocol Buffers. I forgot to handle the old format during the transition. For 10 minutes, deserialization errors caused cache misses for every request, and the database was overwhelmed by the traffic spike. The fix was to add a try-catch that falls back to JSON deserialization for old-format entries. **What the interviewer is looking for:** Honesty about the mistake, understanding of the root cause, the fix, and the systemic improvement.

### Question: "Tell me about a time you had to make a trade-off between consistency and performance."

**Cache-aside angle:** "Our product team wanted real-time inventory counts displayed on the website. The database updates were accurate but slow (200ms per read). I proposed cache-aside with a 30-second TTL. The product team was concerned about showing stale inventory. I ran an experiment: 99.97% of inventory counts did not change within a 30-second window. The business accepted the 0.03% staleness risk for a 20x improvement in page load time. We added a cache invalidation on purchase to reduce the staleness window for high-turnover items."

**What the interviewer is looking for:** Can you quantify the trade-off? Can you negotiate with stakeholders? Do you have data to support your decision?

## Sample Answers for Common Questions

### Sample Answer 1: "Design a Cache System for Twitter Timeline"

**Opening:** "I would use cache-aside with Redis for Twitter's home timeline. The core optimization is a fan-out pattern where tweet IDs are eagerly pushed into followers' timeline caches when the tweet is created."

**Key design decisions:**
- Separate the timeline index (list of tweet IDs) from tweet content (individual items)
- Timeline index stored as a Redis list with a 7-day TTL
- Tweet content stored with cache-aside per tweet ID with a 24-hour TTL
- For celebrities with over 10,000 followers, switch to fan-out-on-read to avoid massive write amplification

**Failure handling:**
- Cache stampede: Use Redis SET NX for the tweet content population. Only the first miss populates the cache.
- Cold timeline: If a user's timeline is not in cache, rebuild it from the database with a background job.
- Node failure: Redis replication with automatic failover. If a shard goes down, its keys are reassigned.

**Capacity math:** "For 500 million users, only about 20% are daily active. Each active user has approximately 200 tweet IDs in their timeline cache. That is 100 million timelines * 200 IDs * 17 bytes per ID = 340 GB of timeline data plus 30% overhead = approximately 450 GB. I would use 12 Redis shards with 48 GB each, replicated across 2 availability zones. Total: 24 nodes * 48 GB = 1.15 TB provisioned."

### Sample Answer 2: "How Do You Prevent Cache Stampede?"

**Opening:** "Cache stampede happens when a popular key expires and multiple concurrent requests all miss the cache simultaneously. The mitigation is a distributed mutex that serializes cache population."

**Implementation:**
```
lock_key = "lock:" + cache_key
lock_acquired = redis.set(lock_key, "1", nx=True, ex=5)
if lock_acquired:
    # I am the designated populater
    data = database.query(...)
    redis.set(cache_key, serialize(data), ex=300)
    redis.delete(lock_key)
    return data
else:
    # Someone else is loading the data, wait briefly
    import time
    time.sleep(0.01)  # 10ms
    # Retry the cache read (the other request may have populated it)
    cached = redis.get(cache_key)
    if cached is not None:
        return deserialize(cached)
    # If still not cached, retry the mutex or fall back to DB
    # (with a cap on retries to avoid infinite loops)
```

**Trade-offs:**
- Adds approximately 1 millisecond of latency for the lock check
- If the lock holder crashes, the lock expires after 5 seconds (TTL)
- For extremely hot keys, a dedicated background refresher is better than on-demand population

### Sample Answer 3: "How Do You Calculate Cache Memory Requirements?"

**Opening:** "Cache memory depends on three factors: the number of unique keys accessed within one TTL window, the average entry size, and the overhead per entry."

**Formula:**
```
Unique keys per TTL window = read_rate * TTL_duration
Memory = unique_keys * (value_size + overhead)

Example:
  read_rate = 50,000 requests/second
  TTL = 300 seconds
  value_size = 5 KB
  overhead = 72 bytes (Redis dict entry + LRU + SDS)

  unique_keys = 50,000 * 300 = 15,000,000 keys
  memory = 15,000,000 * (5120 + 72) = 15,000,000 * 5192 bytes
         = 77.9 GB

  Add 30% headroom for peak loads = 101 GB
  Provision: 6 shards * 20 GB = 120 GB
```

**Important caveats:**
- Not all keys are accessed within the TTL window. Use the effective unique key count, not the theoretical maximum.
- Hot keys may have longer TTLs, increasing their residency time.
- Compression reduces the value size but adds CPU overhead.

### Sample Answer 4: "Design a Cache-Aside System for a Social Media Feed"

**Opening:** "I would use cache-aside with Redis for the feed. The feed is a list of post IDs from followed users. Each post ID is hydrated with post content from a separate cache."

**Key design decisions:**
- Feed index: Redis list of post IDs per user, TTL of 7 days
- Post content: cache-aside per post ID, TTL of 24 hours
- Feed population: On post creation, push post ID to followers' feed lists (fan-out on write)
- Feed hydration: On feed read, fetch post IDs from the feed list, then batch GET post content from the post cache
- For users with over 10,000 followers, use lazy fan-out (pull on read)

**Cache stampede prevention:**
- Batch hydrate: When loading a feed page (20 posts), batch GET all 20 in a single pipeline
- If any post is a miss, populate the cache for that post only
- Use a distributed mutex per post ID for the cache population

**Capacity math:**
```
10 million DAU, each reads feed 5 times per day
50 million feed reads/day = 578 reads/sec peak
Each feed: 20 posts, each post: 5 KB
Cache hits: 578 * 20 * 0.95 = 10,982 reads/sec
Cache misses: 578 * 20 * 0.05 = 578 reads/sec
Memory: 10 million active users * 200 post IDs * 17 bytes = 34 GB
Post content: 1 million recent posts * 5 KB = 5 GB
Total: 39 GB + 30% overhead = 51 GB
Provision: 6 shards * 12 GB = 72 GB
```

## How to Structure Your Answer in Real Time

When the interviewer asks a cache-aside question, use this 30-second mental preparation before speaking.

**Seconds 0-5:** Pause. Breathe. Identify the read-to-write ratio from the problem statement. If the problem does not state it, ask: "What is the read-to-write ratio for this system?"

**Seconds 5-15:** Choose your high-level approach. "I will use cache-aside with Redis because reads dominate the workload and eventual consistency is acceptable."

**Seconds 15-30:** Plan your structure. Use the FAANG framework: (1) identify what is read-heavy, (2) state your consistency assumption, (3) choose the pattern, (4) design keys, (5) address failures, (6) calculate capacity.

Then speak. Start with: "Let me walk through my approach. First, I will identify which components are read-heavy..."

### Common Interview Mistakes to Avoid

Beyond the red flags listed above, here are specific mistakes that candidates make during cache-aside interview questions.

### Mistake 1: Forgetting to Mention the Write Path

Many candidates describe the cache-aside read path in detail but forget to describe what happens on writes. The interviewer is looking for whether you understand both halves of the pattern.

**Instead of:** "On a cache miss, we query the database and populate the cache."
**Say:** "On a cache miss, we query the database and populate the cache. On a write, we update the database and either delete the cache key or update it directly, depending on the consistency requirements."

### Mistake 2: Describing the Cache as a Single Node

When asked to design a system for millions of users, describing a single Redis instance signals inexperience with scale.

**Instead of:** "I will use Redis for the cache."
**Say:** "I will use a Redis cluster with consistent hashing to distribute keys across nodes. For 50,000 requests per second, I estimate 3-4 shards with replication."

### Mistake 3: Not Mentioning Serialization

Cache-aside involves serializing data to store in the cache and deserializing when reading. Ignoring this suggests you have not implemented cache-aside in practice.

**Instead of:** "The application stores the data in the cache."
**Say:** "The application serializes the database result to JSON (or Protocol Buffers) and stores the serialized bytes in the cache. On read, the bytes are deserialized back into an application object."

### Mistake 4: Over-Optimizing Too Early

Designing a multi-region, sharded, replicated cache cluster for 10,000 users makes you look inexperienced with matching complexity to scale.

**Instead of:** Immediately jumping to a 12-node Redis cluster.
**Say:** "For 10,000 users, a single Redis instance with a read replica is sufficient. I would scale to a cluster as traffic grows past 50,000 requests per second."

### Mistake 5: Avoiding the Consistency Question

When the interviewer asks about stale data, do not deflect. Address the trade-off directly.

**Instead of:** "The cache is always up to date because we invalidate on writes."
**Say:** "Cache-aside provides eventual consistency. The staleness window is bounded by the TTL and the invalidation strategy. With explicit invalidation on writes, the window is typically under 1 second. Without invalidation, it is the TTL duration. This trade-off is acceptable for read-heavy workloads."

### Mistake 6: Not Quantifying Anything

Interviewers expect numbers. Cache sizes, latency targets, throughput numbers, TTL durations. Avoid vague statements.

**Instead of:** "The cache will be pretty fast."
**Say:** "Cache hits will take approximately 1-3 milliseconds. Cache misses will take 20-50 milliseconds including the database query. With a 90% hit ratio, the average latency is around 4 milliseconds."
