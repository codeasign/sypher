**How cache-aside works for Twitter timelines:**

- **Cache population:** When a tweet is created, the fan-out service pushes the tweet ID into the timeline cache (Redis list) of all followers. This is cache population done eagerly at write time for the followers -- a twist on classic cache-aside.

- **Cache read:** When a user opens their timeline, the application checks the Redis cache first. If the timeline exists (list with recent tweet IDs), it returns the cached list. If the cache is empty (cold start, or user hasn't been active in a while), the application falls back to a database fan-out query.

- **TTL:** Timeline entries have a 7-day TTL. After 7 days of inactivity, the timeline is evicted from the cache. This handles the pareto principle: 80% of reads come from 20% of users who check Twitter daily.

- **Hit ratio:** 95%+ for active users. Cold users' timelines are rebuilt on demand from the database.

**Key lesson:** Cache-aside with pre-warming (fan-out on write) gives the best of both worlds: the timeline is pre-populated when the tweet is created, so the read is always a hit for users who are actively checking, but the TTL bounds memory usage for inactive users.

**How cache-aside works for GitHub repositories:**

- **Cache population:** On cache miss, the application queries MySQL, serializes the repository metadata, and stores it in Redis with a TTL of 5 minutes.

- **Cache invalidation:** GitHub uses explicit invalidation. When a repository is updated (push, star, fork), the application immediately deletes the relevant cache keys. The next read triggers a miss and fetches fresh data.

- **Cache granularity:** Each repository has multiple cache keys: `repo:owner/name:meta` for basic info, `repo:owner/name:stargazers` for count, `repo:owner/name:readme` for the README. This allows granular invalidation -- a star only invalidates the stargazers key, not the README.

- **Hit ratio:** Very high. Popular repositories receive thousands of views per update.

**Key lesson:** Granular cache keys reduce invalidation blast radius. A star on a popular repository only invalidates the stargazer count, not the entire repository cache.

- **Invalidation:** When a new answer is posted or an edit is made, the application purges both the CDN cache (via API) and the Redis cache (explicit delete).

**Key lesson:** Multi-tier caching with different TTLs and invalidation rules. The CDN handles anonymous traffic (the majority), while Redis handles authenticated users and dynamic data.

**How cache-aside works for Airbnb search results:**

- **Cache key granularity:** The cache key incorporates city, dates, guest count, and a hash of all filters. This creates a large number of cache keys but ensures that search results are specific to each request.

- **TTL:** 60 seconds. Short enough that recently booked listings are reflected in search results within a minute.

- **Cache population:** On the first search for a given combination of parameters, the application queries Elasticsearch, serializes the top 200 results, and stores them in Redis with a 60-second TTL.

- **Cache invalidation:** Natural TTL expiry is the primary invalidation mechanism. When a listing is booked, the search results for the affected city/dates become stale within 60 seconds.

**Key lesson:** Cache keys should be granular enough to capture the uniqueness of the request but coarse enough to produce cache hits. Airbnb's keys include all search parameters, but they accept that many unique searches cause misses.

## Open Source Caching Systems

### Redis

Redis is the most popular cache-aside backend. It is an in-memory data structure store that supports strings, hashes, lists, sets, sorted sets, and more.

**Why Redis for cache-aside:**
- Sub-millisecond latency (in-memory, event-loop architecture)
- TTL support with automatic key expiration
- LRU, LFU, and TTL-based eviction policies
- Lua scripting for atomic operations (mutex, locking)
- Cluster mode for horizontal scaling
- Persistence (RDB snapshots, AOF logs) for cache recovery

**Pitfalls:**
- Single-threaded event loop: slow commands (KEYS, SMEMBERS on large sets) block all other operations
- Memory: no automatic compression, all data in RAM
- Replication lag: async replication can lose recent writes

### Memcached

Memcached is a simpler, faster alternative to Redis for cache-aside. It is a pure key-value store with no data structures.

**Why Memcached for cache-aside:**
- Simpler protocol means lower overhead per operation
- Multi-threaded architecture: better CPU utilization on multi-core machines
- Automatic slab allocation: memory-efficient for uniform-sized values
- Lower latency for simple get/set operations

**Pitfalls:**
- No built-in clustering (client-side consistent hashing required)
- No persistence (all data lost on restart)
- No data structures beyond key-value (no lists, sets, sorted sets)
- No replication (no failover)

### Caffeine (Java)

Caffeine is a high-performance Java in-process caching library, often used as a first-level cache in front of a distributed cache.

**Why Caffeine for cache-aside:**
- Sub-microsecond read latency (in-process, no network call)
- Window TinyLFU eviction policy -- adaptive, high hit ratio
- Automatic refresh (asynchronously reloads expiring keys)
- Size-based and time-based eviction

**Pitfalls:**
- In-process cache: not shared across instances, must be paired with a distributed cache
- Memory competition with application heap

### Django Cache Framework

Django provides built-in cache-aside support through its cache framework, supporting Redis, Memcached, and local memory backends.

```
from django.core.cache import cache

def get_profile(user_id):
    key = f"profile:{user_id}"
    result = cache.get(key)
    if result is None:
        result = Profile.objects.get(id=user_id)
        cache.set(key, result, timeout=300)  # 5-min TTL
    return result
```

### Rails Low-Level Cache

Ruby on Rails provides `Rails.cache` for cache-aside operations, supporting Redis and Memcached backends.

```ruby
def get_profile(user_id)
  Rails.cache.fetch("profile:#{user_id}", expires_in: 5.minutes) do
    Profile.find(user_id)
  end
end
```

## Lessons Learned

### Lesson 1: Key Naming Conventions Matter

Use a consistent key naming convention that supports pattern matching (where supported) and debugging:

```
{namespace}:{type}:{identifier}:{field}

Examples:
  user:42:profile      -> user profile data
  user:42:settings     -> user preferences
  product:99:price     -> product price (frequently invalidated)
  product:99:details   -> product details (rarely invalidated)
  session:abc123:data  -> session data (short TTL)
  search:paris:2024-06  -> search result (short TTL)
```

**Why it matters:** Namespacing allows batch invalidation. If a user deletes their account, you can iterate over `user:42:*` keys and delete them. Consistent naming also helps with debugging and monitoring.

### Lesson 2: Serialization Choice Is a Performance Decision

JSON is the default serialization format for most applications. It is human-readable, widely supported, and easy to debug. But at scale, JSON's performance characteristics matter:

```
1 MB of JSON:
  Serialize:   3-5 microseconds per KB = 3-5 milliseconds
  Deserialize: 2-4 microseconds per KB = 2-4 milliseconds
  Size:        ~1x (baseline, can be 2-3x with nested structures)

1 MB of Protocol Buffers:
  Serialize:   1-2 microseconds per KB = 1-2 milliseconds
  Deserialize: 0.5-1 microseconds per KB = 0.5-1 milliseconds
  Size:        0.3-0.5x of JSON (compressed binary format)
```

**Guideline:** Start with JSON for developer productivity. Profile the serialization cost. If it exceeds 5% of total request time, switch to Protocol Buffers or MessagePack.

### Lesson 3: Connection Pooling Prevents Resource Exhaustion

Every cache operation opens a TCP connection to the cache server. Without a connection pool, the application opens and closes TCP connections for every request:

```
Without pooling:
  Request 1: TCP connect (3-way handshake, ~1-3ms)
             GET key
             TCP close (4-way teardown, ~0.5ms)
  Request 2: TCP connect again
             GET key
             TCP close again
  Overhead per request: ~2-4ms just for connection management

With pooling:
  Request 1: Borrow pool[0] (already connected)
             GET key
             Return pool[0]
  Request 2: Borrow pool[0] (reused)
             GET key
             Return pool[0]
  Overhead per request: ~0ms for connection management
```

**Guideline:** Use connection pooling with a pool size of 10-20 per cache node. Monitor the pool utilization metric. If it reaches 80%+, increase the pool size.

### Lesson 5: Graceful Degradation Is Essential

The cache is a performance optimization, not a correctness guarantee. The system must function without it.

```
def get_with_cache(key, fetch_func, ttl=300):
    try:
        result = cache.get(key)
        if result is not None:
            return deserialize(result)
    except CacheConnectionError:
        # Cache unavailable: skip to database
        pass
    except CacheTimeoutError:
        # Cache slow: skip to database, don't wait for retry
        pass

    try:
        result = fetch_func()
        try:
            serialized = serialize(result)
            cache.set(key, serialized, ex=ttl)
        except (CacheConnectionError, CacheTimeoutError):
            # Cache population failed: log and continue
            logger.warning(f"Failed to set cache key {key}")
        return result
    except DatabaseError:
        logger.error(f"Database query failed for {key}")
        raise
```

**Guideline:** The cache path should be wrapped in try/except, with a database fallback. The cache population should also be wrapped separately so a cache SET failure does not fail the request.

### Lesson 6: Monitoring Cache Health in Production

Without monitoring, you are flying blind. The minimum set of metrics every cache-aside system should expose:

**Application-level metrics (emitted by the cache client):**
- `cache.operations.total` -- count of get/set/delete operations
- `cache.hits` and `cache.misses` -- partition by key pattern for granular insight
- `cache.latency.p50` and `cache.latency.p99` -- measure the time for each cache operation
- `cache.errors` -- connection failures, timeouts, serialization errors
- `cache.connection_pool.active` and `cache.connection_pool.idle` -- pool utilization

**Cache server metrics (emitted by Redis / Memcached):**
- `redis.memory.used_memory` and `redis.memory.maxmemory` -- memory utilization
- `redis.stats.evicted_keys` -- number of evictions per second
- `redis.stats.expired_keys` -- number of TTL expirations per second
- `redis.stats.instantaneous_ops_per_sec` -- throughput
- `redis.stats.hit_ratio` -- calculated from keyspace_hits / (keyspace_hits + keyspace_misses)

**Alert thresholds for production:**
```
CRITICAL:
  - cache.hit_ratio < 60% for 5 minutes  (cache not effective)
  - redis.memory.used_memory > 90% of maxmemory (imminent OOM)
  - cache.errors > 5% of total operations (cache connection issues)

WARNING:
  - cache.hit_ratio < 80% for 15 minutes (degraded performance)
  - cache.latency.p99 > 10ms for 5 minutes (cache slow, check network)
  - redis.stats.evicted_keys > 100/sec (insufficient memory)
  - cache.connection_pool.active > 80% of pool size (need more connections)
```

**Dashboards:** Create a dedicated cache monitoring dashboard that shows:
- Hit ratio over time (last 1 hour, last 24 hours)
- Latency heatmap (p50, p95, p99) per cache node
- Memory usage per node with maxmemory line
- Top 10 cache keys by access frequency (identify hot keys)
- Top 10 cache keys by miss count (identify cache-busters)

## Summary

Production cache-aside use cases share common patterns: granular cache keys, explicit invalidation on writes, short TTLs for dynamic data, and graceful degradation when the cache is unavailable. The choice of caching backend (Redis, Memcached, in-process) depends on latency requirements, data complexity, and operational maturity.
