## Naive Implementation

Every metrics system starts simple. A single server, a single database, a single dashboard. This works until it does not. Understanding where the naive implementation breaks is the key to designing a system that scales.

### The Single-Server Setup

The simplest possible metrics architecture: one Prometheus server running on a VM, scraping a handful of application instances, storing data locally, and serving a Grafana dashboard.

This setup handles:
- 10 to 100 application instances
- 10,000 to 50,000 active time series
- 15-second scrape intervals
- 30 days of data retention

The server runs on a single machine with 8 GB of RAM, 4 CPU cores, and a 500 GB SSD. The total cost is roughly $100 per month on a cloud provider.

### Breaks at 1,000 Users

At 1,000 users, the application scales to 10 microservices, each running 3 replicas. That is 30 application instances, each emitting 500 metrics with 5 labels each. The scrape targets are 30, and the active time series count is 30 multiplied by 500 multiplied by 5 multiplied by 3 equals 22,500.

The single server still handles this, but scrape latency starts to climb. Each scrape takes 50 milliseconds instead of 5. The server is spending 30 targets multiplied by 50ms equals 1.5 seconds per scrape cycle. With a 15-second scrape interval, the server is idle for 13.5 seconds. It works, but the headroom is shrinking.

### Breaks at 100,000 Users

At 100,000 users, the application is running 50 microservices with 10 replicas each. That is 500 application instances. Each instance emits 1,000 metrics with 10 labels. The active time series count is 500 multiplied by 1,000 multiplied by 10 equals 5 million.

The single server now faces:
- **Scrape duration**: 500 targets at 100ms each equals 50 seconds per scrape cycle. The scrape interval is 15 seconds. The server is permanently behind — it cannot finish one scrape cycle before the next one starts.
- **Memory pressure**: 5 million active series at 2 KB each for the index equals 10 GB of RAM. The server has 8 GB. The operating system starts swapping.
- **Disk I/O**: 5 million series at 15-second intervals produces 5,000,000 divided by 15 equals 333,333 samples per second. The WAL cannot keep up; writes queue and eventually time out.
- **Query latency**: a simple query over 1 hour touches 5 million series. The index lookup takes 5 seconds, and chunk decompression takes another 10 seconds. The Grafana dashboard times out at 30 seconds.

The system is completely broken. The team cannot see dashboards, alerts fire randomly (or not at all), and the monitoring system needs its own incident response.

The deeper problem is that the monitoring system is now a single point of failure. When the metrics server is down, the team has no visibility into production. They cannot see CPU usage, error rates, or latency. They cannot receive alerts. They are operating completely blind. The tools that were supposed to help them detect and respond to incidents are themselves an incident.

This is a classic second-order effect: a system that fails in a way that makes it harder to fix itself. The metrics server is down, so you cannot see that the metrics server is down. You have to SSH into the metrics server box and run commands manually to diagnose why it crashed. The crash log might tell you it ran out of memory, but you cannot see the memory trend because the metrics server was the thing that tracked memory trends.

The solution is to break the single point of failure through horizontal sharding, hierarchical aggregation, and independent regional stacks. Each component must be independently scalable and independently replaceable. The system must be designed so that the failure of any single component does not cause a complete loss of visibility.

### Breaks at 10,000,000 Users

At 10 million users, the naive approach is not just broken — it is dangerous. The metrics server is OOM-killed every 30 minutes. The on-call engineer silences alerts because they are all false positives caused by the broken monitoring system. The team has no visibility into production. When a real incident happens, they discover it through user complaints on social media.

The problem is not just the scale of the data. It is the coupling between the data collection and the data storage. A single-server system couples collection, storage, and query into a single process. If the process fails, all three functions are lost. A production-grade systems decouples these functions into separate, independently scalable components.

The lesson from every large-scale metrics deployment is the same: the metrics system must be designed as a distributed system from the start, with the same attention to fault tolerance, scalability, and operability as the application it monitors.

## Production-Grade Architecture

A production metrics system at scale uses horizontal sharding, hierarchical aggregation, and tiered storage. Each component is independently scalable.

### Horizontal Sharding (Hash Rings)

Metrics are distributed across multiple servers using consistent hashing. Each server is responsible for a subset of the time series. The hash function maps (metric_name, label_set) to a server ID.

```
hash("http_requests_total{method=GET, status=200}") % N = server 3
hash("http_requests_total{method=POST, status=200}") % N = server 7
```

Consistent hashing minimizes reshuffling when servers are added or removed. When a server joins or leaves, only 1/N of the hash space is reassigned. With 10 servers, adding one reassigns 10 percent of data.

### Hierarchical Aggregation

Collection is split into two tiers: leaf servers and aggregation servers.

**Leaf servers** scrape application instances directly. Each leaf handles 50 to 200 targets. Leaf servers are stateless for scraping: if a leaf fails, another leaf can take over its targets.

**Aggregation servers** pull data from leaf servers and compute global aggregates. They run recording rules, handle downsampling, and serve queries.

The hierarchy reduces the number of connections each server must manage. A leaf talks to 200 targets and 1 aggregation server. An aggregation server talks to 10 leaf servers. Without hierarchy, a single server would need to talk to 2,000 targets directly.

### Tiered Storage

Hot data (last 7 days) lives on SSDs for fast queries. Warm data (7 to 90 days) lives on HDDs or cheaper SSDs. Cold data (90+ days) lives in object storage (S3, GCS).

The query engine routes queries to the appropriate tier automatically. A query over the last hour reads only from the hot tier. A query over the last 6 months reads from all three tiers.

### Query Routing

A query frontend receives all API requests and routes them to the appropriate shards. The frontend:
1. Parses the query to identify which time series are needed.
2. Determines which shards hold those series.
3. Fans out the query to those shards in parallel.
4. Merges the results and returns them.

The query frontend is the most latency-sensitive component. A slow query frontend makes the entire dashboard feel slow.

## Integration Patterns

A metrics system does not exist in isolation. It integrates with databases, caches, message queues, CDNs, and CI/CD pipelines. Each integration has a specific pattern.

### Pattern 1: Database Integration

Databases expose metrics through their own /metrics endpoints or through plugins. The metrics system scrapes these endpoints to collect:

- Connection pool utilization (gauge)
- Query latency (histogram)
- Active transactions (gauge)
- Replication lag (gauge)
- Disk usage (gauge)
- Cache hit ratio (rate)

For PostgreSQL, the pg_stat_statements extension exposes per-query latency. For MySQL, the performance_schema provides similar data. For Redis, the INFO command output is parsed into metrics. For Kafka, the JMX exporter exposes broker and consumer metrics.

**Critical metric**: database connection pool utilization. When this hits 100 percent, new connections queue up and application requests time out. Alert at 80 percent to give you time to scale.

### Pattern 2: Cache Integration

Cache metrics tell you whether your caching is effective. The most important cache metric is hit ratio.

```
cache_hit_ratio = rate(cache_hits_total[5m]) / (rate(cache_hits_total[5m]) + rate(cache_misses_total[5m]))
```

A hit ratio above 90 percent is good. Below 80 percent, the cache is not helping. Below 50 percent, the cache is wasting resources and should be bypassed.

Other cache metrics: eviction rate (high evictions mean the cache is too small), memory usage, and connection count.

**Critical metric**: eviction rate. A sudden spike in evictions means the working set has grown beyond the cache size. This is a leading indicator of performance degradation — it happens before latency increases.

### Pattern 3: Message Queue Integration

Message queue metrics measure the health of async processing pipelines.

- **Queue depth**: number of messages waiting to be processed. A growing queue depth means consumers are falling behind.
- **Consume rate**: messages consumed per second. Should match the produce rate in steady state.
- **Lag**: difference between the latest produced offset and the latest consumed offset. For Kafka, this is the primary health metric.
- **Dead letter queue count**: messages that failed processing and cannot be retried. Should be zero.

**Critical metric**: consumer lag. A growing lag means the system is accumulating a backlog. If the backlog exceeds the retention period, messages are permanently lost.

### Pattern 4: CDN Integration

CDNs provide metrics through their APIs. Pull these metrics into your system for a unified view.

- **Edge request rate**: requests served by the CDN, not your origin.
- **Cache hit ratio at edge**: requests served from CDN cache without hitting origin.
- **Origin offload**: percentage of requests that the CDN handles without contacting your origin servers.
- **Error rates by edge location**: geographic distribution of errors.
- **Bandwidth**: bytes served by the CDN.

**Critical metric**: origin offload. If this drops, your origin servers are taking more traffic, which increases latency and cost. Investigate cache invalidation patterns or changes in traffic distribution.

### Pattern 5: CI/CD Pipeline Integration

CI/CD pipelines emit metrics for deployment frequency, change failure rate, and deployment duration.

- **Deployment frequency**: how often you deploy to production.
- **Deployment duration**: time from commit to production.
- **Change failure rate**: percentage of deployments that cause incidents.
- **Mean time to recovery**: time to restore service after a failure.

These metrics are the DORA metrics, used to measure DevOps maturity. They are tracked as counters or histograms.

## Multi-Region Architecture

A global service needs metrics from every region. The naive approach (scrape all regions from one central server) fails because of network latency, bandwidth costs, and cross-region failure domains.

### Regional Aggregation

Each region runs its own metrics stack: leaf servers, aggregation servers, and storage. Regional stacks are independent. A failure in one region does not affect metrics collection in another region.

The regional stack is sized to handle the peak traffic of that region. A region with 2 million users needs more capacity than a region with 500,000 users. Each region's capacity is planned independently based on the traffic patterns of that region.

The regional stack includes:
- **Leaf scrapers**: one per availability zone, scraping all targets in that zone.
- **Aggregation servers**: merging data from all leaf scrapers in the region.
- **Regional storage**: keeping 30 days of raw data on SSD.
- **Regional query endpoint**: serving dashboards and alerts for that region.
- **Regional alerting**: evaluating alerting rules against regional data.

The regional stack also includes regional meta-metrics: metrics about the health of the metrics system itself. The meta-metrics are scraped by a separate, independent collector that is outside the regional stack. This provides an independent health check: if the regional stack is down, the application still has a way to report that the metrics system is down.

```
Region US-East:  [Leaf] → [Agg] → [Store] → [Dashboard]
Region EU-West:  [Leaf] → [Agg] → [Store] → [Dashboard]
Region AP-South: [Leaf] → [Agg] → [Store] → [Dashboard]
```

### Global View

A global aggregation layer reads from each regional store and computes global metrics. The global layer is read-only: it does not participate in collection. It runs on a separate set of servers that are less resource-intensive because they handle only queries, not ingestion.

Global dashboards query the global layer. Regional dashboards query the regional layer. Drill-down goes from global to regional to individual service.

The global layer has its own storage for global metrics: aggregated data that spans all regions. These global aggregates include:
- **Total request rate**: sum of request rates across all regions.
- **Global error rate**: total errors divided by total requests across all regions.
- **Global p99 latency**: the latency value that covers 99 percent of all requests worldwide.
- **Regional comparison**: a breakdown of the same metrics by region for quick comparison.

The global layer does not need raw data. It operates on downsampled data (1-minute or 5-minute resolution) because the purpose of global dashboards is trend visibility, not per-request debugging. The raw data stays in the regional stores where it is available for deep dives.

The global query endpoint is a Thanos querier or equivalent that fans out queries to all regional stores. The fan-out adds network latency equal to the slowest region. If the EU-West store is 100ms away (one-way) and the AP-South store is 200ms away, the global query takes at least 200ms plus query execution time. Most organizations accept this latency for global dashboards. If cross-region queries need to be faster, replicate regional aggregates to a central store that is closer to the query frontend.

### Cross-Region Query

For queries that span regions, the query frontend fans out to all regional stores and merges the results. The fan-out adds latency equal to the slowest region. To mitigate this, regional stores return results within a configurable deadline (typically 2 seconds). If a region misses the deadline, the query returns partial results.

## Capacity Planning

Capacity planning for a metrics system is a math problem. Given your user count, service count, and metric cardinality, you can compute the required resources.

### Assumptions for 10 Million DAU

- 100 microservices
- 10 replicas per service (1,000 total instances)
- 500 metrics per instance
- 10 labels per metric (average)
- 15-second scrape interval
- 30-day raw retention
- 12-month downsampled retention

### Step 1: Active Time Series

Active time series = instances × metrics per instance × labels per metric.

```
1,000 instances × 500 metrics × 10 labels = 5,000,000 active series
```

### Step 2: Ingestion Rate

Ingestion rate = active series / scrape interval.

```
5,000,000 / 15 seconds = 333,333 samples per second
```

### Step 3: Storage per Day

Daily storage = samples per second × seconds per day × bytes per sample.

```
333,333 samples/sec × 86,400 seconds/day × 2 bytes/sample = 57.6 GB per day
```

### Step 4: Storage for Retention

Raw (30 days): 57.6 GB × 30 = 1.7 TB
Downsampled (5-min, 6 months): 57.6 GB × 0.03 (compression ratio) × 180 = 311 GB
Downsampled (1-hour, 12 months): 57.6 GB × 0.0025 × 365 = 52.5 GB

Total: 1.7 TB + 311 GB + 52.5 GB = 2.1 TB

### Step 5: Memory

Memory = index memory + memtable memory + query cache.

```
Index: 5 million series × 256 bytes = 1.28 GB
Memtable: 5 million series × 128 bytes = 640 MB
Query cache: 2 GB
Total: 4 GB per server
```

### Step 6: Server Count

With 10 shards, each server handles:
- 500,000 active series
- 33,333 samples per second
- 200 GB of storage
- 4 GB of memory

Each server needs: 8 CPU cores, 16 GB RAM, 500 GB SSD. The cluster needs 10 servers.

### Step 7: Cost Estimate

```
10 servers × $500/month (cloud) = $5,000/month
Object storage (cold tier): $50/month
Total: $5,050/month
```

This is the infrastructure cost for monitoring a 10 million DAU system. It is roughly 2 to 5 percent of the total infrastructure cost, which is the industry standard for observability.

## Deployment and Operations

### Deployment Strategies

**Kubernetes Operator**: Use the Prometheus Operator for Kubernetes deployments. The Operator manages the metrics stack as custom resources. It handles service discovery, configuration, and scaling automatically.

**Nomad / Consul**: For HashiCorp-based deployments, use Consul for service discovery and Nomad for scheduling. The metrics server discovers targets by querying the Consul catalog.

**Manual**: For small deployments, a single binary with a configuration file is sufficient. The configuration file lists scrape targets, recording rules, and alerting rules.

### Monitoring the Monitors

The metrics system must monitor itself. These meta-metrics detect when the monitoring infrastructure is failing:

```
prometheus_target_scrapes_exceeded_total
prometheus_tsdb_compactions_failed_total
prometheus_tsdb_out_of_order_ingestion_total
prometheus_rule_evaluation_failures_total
prometheus_alertmanager_alerts_invalid_total
```

If these meta-metrics show problems, the monitoring system is broken and needs immediate attention. The meta-metrics should be scraped by a separate, independent monitoring system for redundancy.

### Upgrades

Upgrade the metrics cluster one shard at a time. Each shard is drained of traffic, upgraded, tested, and rejoined. The drain period is the time it takes for the query frontend to detect the shard is gone and stop routing queries to it.

Drain before upgrade:
1. Mark the shard as leaving in the hash ring.
2. Wait for the query frontend to update its routing table (30 seconds).
3. Stop the metrics server process.
4. Upgrade the binary.
5. Start the process. The WAL is replayed on startup.
6. Mark the shard as active in the hash ring.
7. Verify the shard is serving queries correctly.

### Testing and Validation

A metrics system that produces incorrect data is worse than no metrics at all. Testing and validation are essential parts of the deployment process.

**Unit testing instrumentation**: Each metric emitted by a service should have a unit test that verifies:
1. The metric is emitted with the correct name and label values.
2. The metric value is computed correctly.
3. The metric is incremented or recorded on the correct code path.
4. The metric has the correct type (counter, gauge, histogram).

Unit tests for metrics are often overlooked because metrics are considered side effects rather than functional behavior. But incorrect metrics lead to incorrect dashboards, incorrect alerts, and incorrect operational decisions.

**Integration testing the pipeline**: Before deploying a new metric or changing the scraping configuration, run an integration test that:
1. Deploys a test instance of the service.
2. Configures the scraper to scrape the test instance.
3. Verifies that the scraper successfully collects the metrics.
4. Verifies that the metrics appear in the database.
5. Verifies that the dashboard shows the expected values.
6. Verifies that the alerting rule evaluates correctly.
7. Tears down the test instance and verifies that the metrics are cleaned up.

**Regression testing**: When upgrading the metrics system (new version of Prometheus, new version of Thanos), run a regression test that compares query results before and after the upgrade. The regression test should cover all critical dashboards and alerts. Any difference in query results should be investigated before the upgrade is deployed.

**Load testing**: Before scaling the metrics system to handle a new peak traffic level, run a load test that simulates the expected increase. The load test should verify that: scrape duration stays under the scrape interval, query latency stays under 2 seconds for the last hour, and the system does not run out of memory or disk.

### Backup and Disaster Recovery

The WAL provides crash recovery, but it does not protect against disk failure. For disaster recovery:

- **Daily snapshots**: take a snapshot of the data directory and store it in object storage.
- **Cross-region replication**: stream metrics to a secondary region for disaster recovery.
- **Backup retention**: keep 7 daily snapshots, 4 weekly snapshots, and 12 monthly snapshots.

Test restore procedures quarterly. A backup that has never been restored is a backup that does not work. The restore test should simulate a full disaster: wipe the primary metrics cluster, restore from the most recent snapshot, and verify that all dashboards and alerts work correctly. The restore time target is 4 hours: the team should be able to restore from backup and have the metrics system operational within 4 hours of the disaster declaration.

### Monitoring the Monitors, Revisited

The meta-metrics that monitor the metrics system must be collected by a separate infrastructure. The meta-metrics collector should be simpler, more reliable, and independently deployed. A common approach is a small script that runs on a separate VM, emits meta-metrics to a lightweight file-based database, and serves a minimal dashboard.

The meta-metrics dashboard should have exactly one panel: is the metrics system healthy? The panel shows a green checkmark if all meta-metrics are within normal ranges, or a red X with the failing component name. This dashboard is the first thing the on-call engineer checks. If it is green, the monitoring system is trustworthy and any alerts are real.

### Kubernetes Integration

For Kubernetes deployments, the metrics system follows a specific architecture that leverages Kubernetes-native service discovery.

**Service discovery**: The Prometheus Operator uses Custom Resource Definitions (CRDs) to define scrape targets. ServiceMonitor and PodMonitor CRDs describe which services and pods to scrape and what labels to attach. The Operator watches for changes to these CRDs and updates the Prometheus configuration automatically.

**Node-level collection**: The kubelet on each node exposes a /metrics endpoint for node-level metrics (CPU, memory, disk, network). The Prometheus server scrapes this endpoint for every node. The cadvisor metrics (container-level CPU and memory) are also exposed through the kubelet.

**Cluster-level metrics**: The Kubernetes API server, controller manager, scheduler, and etcd all expose metrics endpoints. These are scraped by a dedicated Prometheus instance or by the cluster-wide Prometheus.

**Pod-level instrumentation**: Each pod exposes a /metrics endpoint on a configurable port. The ServiceMonitor CRD tells Prometheus which pods to scrape, on which port, and with which labels. The labels are automatically enriched with Kubernetes metadata: namespace, pod name, node name, and deployment name.

**Kubernetes-specific labels**: The following labels are automatically attached to every metric scraped from a pod: namespace, pod, service, deployment, node, and container. These labels are essential for debugging but increase cardinality. A deployment with 100 pods produces 100 time series for each metric, one per pod. The cardinality impact is proportional to the number of pods.

**Auto-scaling**: The Prometheus Operator can auto-scale the Prometheus StatefulSet based on the number of scrape targets. When new services are deployed, the Operator adds them to the scrape configuration and scales the Prometheus cluster accordingly.

### Security Considerations

A metrics system exposes sensitive information about your infrastructure. Attackers can use metric data to find vulnerable services, identify traffic patterns, and plan attacks.

**Network segmentation**: The metrics system should run on a separate network segment from the application traffic. Scrape requests should only be allowed from the metrics collector, not from the public internet. Use network policies in Kubernetes or security groups in cloud providers to restrict access.

**Authentication**: The /metrics endpoint should require authentication for write access. Prometheus supports basic authentication, bearer tokens, and mTLS. The authentication method should be consistent across all services.

**Encryption**: Metric data in transit should be encrypted using TLS. Metrics data at rest should be encrypted using the storage layer's encryption mechanism. This is especially important for metrics that contain sensitive information (e.g., error rates that reveal business logic).

**Rate limiting**: The collection layer should rate-limit incoming scrape requests. A misconfigured scraper that scrapes too frequently can overload the target. The rate limit should be per-scraper, not global.

**Data retention for compliance**: Some industries require metric data to be retained for specific periods. Healthcare (HIPAA) requires 6 years. Financial services (PCI DSS, SOX) require 1 to 7 years. The metrics system must support configurable retention policies that meet these requirements. The compliance data should be stored in a separate, immutable storage tier that cannot be modified or deleted.

### Capacity Planning as a Continuous Process

Capacity planning for the metrics system is not a one-time exercise. It should be reviewed monthly as the organization grows. The key leading indicator is the active time series count. If it is growing faster than expected, investigate which services are adding metrics. Common causes: new services being added without proper instrumentation limits, or existing services adding high-cardinality labels.

Set up a trend alert on active series count. If the 7-day growth rate exceeds 10 percent, review the metric additions with the engineering teams. A cardinality review should be part of every quarterly planning cycle.

### The Observability Maturity Model

The metrics system should evolve as the organization's observability maturity grows. The maturity model has five stages:

**Stage 1: Reactive**: The team has basic CPU and memory metrics. They detect incidents through user complaints. The metrics system is a single Prometheus server that was set up by one engineer and has no documentation.

**Stage 2: Standardized**: Every service emits the four golden signals. Dashboards exist for each service. Alerts are configured with appropriate thresholds. The metrics system is documented and maintained by a dedicated team.

**Stage 3: Correlated**: Metrics are linked across services. A global dashboard shows the health of the entire system. Cross-service alerts detect cascading failures. The metrics system is integrated with the incident response system.

**Stage 4: Automated**: Capacity planning is automated based on metric trends. The metrics system auto-scales based on ingestion rate. Alert thresholds are automatically tuned based on observed behavior. The team spends more time on improvements than on firefighting.

**Stage 5: Predictive**: The metrics system uses machine learning to detect anomalies before they become incidents. The system predicts capacity needs weeks in advance. Metrics are used to drive business decisions, not just operational decisions.

Most organizations should aim for Stage 3: correlated visibility. Stage 4 and 5 require significant investment in automation and machine learning infrastructure.

### Metrics as an Internal Product

The metrics infrastructure team should treat the metrics system as an internal product. The users are the engineers who rely on the system for their daily work. The product should have:

**SLAs**: The metrics system should have an SLA for query latency, data freshness, and uptime. A typical SLA: p99 query latency under 2 seconds for the last 7 days, data freshness under 30 seconds, and uptime of 99.9 percent.

**Roadmap**: The metrics team should maintain a public roadmap that shows planned improvements, new features, and deprecations. The roadmap should be informed by user feedback and usage data.

**User feedback**: The metrics team should collect feedback from users through surveys, user interviews, and a feedback channel. Common feedback themes: dashboard load time, alert accuracy, and onboarding ease.

**Self-service**: Users should be able to add their service to the metrics system without filing a ticket. A self-service portal lets them register their service, select a dashboard template, and configure alerting rules. The onboarding process should take under 30 minutes.

### Service Level Objectives (SLO) Integration

The metrics infrastructure is the foundation for SLO monitoring. SLOs are defined in terms of metrics: the ratio of good events to total events over a rolling window.

**SLO metric definition**: An SLO is defined as a PromQL expression. For example, the SLO for HTTP request availability is: `sum(rate(http_requests_total{status!~"5.."}[30d])) / sum(rate(http_requests_total[30d]))`. This computes the ratio of non-5xx requests to total requests over a 30-day rolling window.

**Burn rate alerting**: Instead of alerting when the SLO is breached, alert when the SLO is being consumed faster than expected. A burn rate alert fires when the error rate over a short window (1 hour) is above a threshold that would consume the entire error budget within a target time (7 days).

The burn rate formula: if the SLO target is 99.9 percent, the allowed error rate is 0.1 percent. If the current error rate is 1 percent (10x the allowed rate), the error budget is being consumed at 10x the expected rate. The alert fires because the SLO will be breached within 7 days / 10 = 0.7 days if the error rate continues.

**SLO dashboard**: The SLO dashboard shows:
- Current SLO value (as a percentage) over the rolling window.
- Remaining error budget (as a percentage and absolute count).
- Burn rate (how fast the error budget is being consumed).
- Remaining time before SLO breach at current burn rate.

**Multi-window approach**: Use multiple windows for burn rate detection. A short window (1 hour) catches fast burn rates. A longer window (6 hours) catches slow burn rates. If any window shows a concerning burn rate, the alert fires.

### Incident Response Integration

The metrics system should integrate with the incident response system. When an alert fires, it should automatically create an incident ticket, notify the on-call engineer, and include a link to the relevant dashboard.

**Auto-created incident**: The alert notification includes: incident title, severity, affected service, metric query, current value, threshold, and a link to the relevant dashboard panel. The on-call engineer opens the dashboard from the notification and sees the problem immediately.

**Dashboard annotations**: When an incident is created (manually or automatically), the dashboard should show an annotation at the incident time. This helps the on-call engineer see what happened before, during, and after the incident. Annotations are automatically added to dashboard panels when the incident response system creates an incident.

**Post-incident review**: After the incident is resolved, the review process uses metric data to analyze the timeline. The dashboard shows the relevant metric panels with the incident annotations. The team traces the metric changes from the start of the incident through the recovery. The post-incident review identifies whether the metrics system provided the right signals at the right time.
