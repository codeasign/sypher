## The Core Tension

The central trade-off in event-driven architecture is **decoupling vs complexity**.

Decoupling gives you resilience, scalability, and extensibility. Each service operates independently. A failure in one service does not cascade to others. You scale each service independently. You add new consumers without touching producers.

But decoupling comes at a cost: you trade the simplicity of synchronous calls for the complexity of asynchronous event flows. A single request in a monolith becomes a chain of events across multiple services. Debugging requires distributed tracing. Testing requires orchestrating multiple services. Error handling requires retry policies, dead letter queues, and idempotency.

## Decision Matrix

| Factor | Event-Driven | Request-Driven | Hybrid |
|---|---|---|---|
| Latency requirement | Under 100ms end-to-end | Under 10ms per call | Mix: sync for reads, async for writes |
| Consistency requirement | Eventually consistent | Strongly consistent | Strong for critical paths, eventual for others |
| Number of consumers | Many (2-20+) | Few (1-5) | Varies by endpoint |
| Failure tolerance | High (isolated failures) | Low (cascading failures) | Moderate |
| Development speed | Slow initially, fast later | Fast initially, slow later | Moderate |
| Debugging difficulty | High (distributed tracing needed) | Low (single request trace) | Medium |
| Testing complexity | High (multiple services) | Low (single service) | Medium |
| Operational cost | High (broker + infrastructure) | Low (HTTP endpoints) | Medium |
| Schema evolution | Strict (registry enforced) | Loose (versioned endpoints) | Mixed |

## CAP Theorem in Event-Driven Systems

CAP theorem states that distributed systems can have at most two of Consistency, Availability, and Partition Tolerance. Event-driven systems face this trade-off directly.

In event-driven systems:

- **Partition tolerance is non-negotiable**: The broker must operate during network partitions. You cannot sacrifice P.
- **Availability is the primary goal**: Event-driven systems prioritize availability over consistency. If the broker loses contact with a consumer, it buffers events and delivers them when the consumer reconnects. The broker does not block the producer.
- **Consistency is eventual**: Because events are processed asynchronously, the system is not strongly consistent at any point in time. It converges over time.

This makes event-driven systems **AP systems** — they prioritize Availability and Partition Tolerance over Consistency. If you need strong consistency, consider request-driven communication or a database with distributed transactions.

### Consistency Levels

| Level | Guarantee | Event-Driven Feasibility |
|---|---|---|
| Strong | All reads return the latest write | Not achievable in standard EDA |
| Sequential | Operations from each producer appear in order | Achievable with partition keys |
| Causal | Related events appear in order | Achievable with correlation IDs |
| Eventual | All events converge to the same state | Default for EDA |

## Failure Modes

### Failure Mode 1: Event Loss

Events can be lost at multiple points in the pipeline:

- **Producer crash before publish**: The producer creates an event but crashes before sending it to the broker.
- **Broker crash before replication**: The broker acknowledges the event but crashes before replicating to followers.
- **Consumer crash before processing**: The consumer receives the event but crashes before processing.
- **Network partition**: The producer cannot reach the broker. Events are queued locally and may overflow.

**Mitigations**:
- Use producer acknowledgements (acks=all in Kafka)
- Replicate events across multiple broker nodes
- Use at-least-once delivery semantics
- Monitor consumer lag and set alerts
- Implement a local event buffer on the producer side

### Failure Mode 2: Ordering Violations

Event ordering is guaranteed only within a partition. Events for the same entity must be assigned to the same partition. If two events for the same order go to different partitions, they may be processed out of order.

Example: `PaymentCaptured` (partition 1) is processed before `PaymentAuthorized` (partition 2). The billing service sees a capture without an authorization and rejects it.

**Mitigations**:
- Use consistent partition keys (always use order_id for order events)
- Accept that cross-partition ordering is not guaranteed
- Design events to be commutative where possible
- Use sequence numbers and reject out-of-order events

### Failure Mode 3: Schema Mismatch

A producer upgrades its schema and starts publishing events with new fields. A consumer that has not been updated receives an event with fields it does not understand. If the consumer was not designed for forward compatibility, it crashes.

**Mitigations**:
- Enforce schema evolution rules (backward and forward compatible)
- Use a schema registry to validate every event
- Maintain consumer compatibility testing in CI/CD
- Version your schemas and consumers
- Migrate consumers before producers when breaking changes are required

### Failure Mode 4: Cascade Failure

An event-driven system should isolate failures, but in practice, failures can cascade through shared infrastructure:

1. Consumer A starts processing slowly (database contention).
2. Consumer A falls behind. Broker storage grows as unconsumed events accumulate.
3. Broker hits storage limits and starts deleting old events.
4. Consumer B loses events it has not yet processed because they were deleted.
5. Consumer B cannot reconcile its state and starts failing.

**Mitigations**:
- Use separate topics for different event types with separate retention policies
- Set consumer lag alerts and auto-scale consumers
- Use tiered storage to offload old events to cheaper storage
- Implement per-consumer quotas on broker storage

### Failure Mode 5: Debugging Difficulty

When an event flow breaks, the failure might be in the producer, the broker, any of the consumers, or the network between them. Each service has its own logs. Correlating them requires tracing infrastructure.

A team spent 3 days investigating why orders were not being shipped. The producer was fine. The broker was fine. The inventory consumer was fine. The billing consumer was fine. The shipping consumer was receiving events but processing them with outdated customer addresses. The root cause: a schema migration in the order service three weeks earlier had changed the customer address field name, but the shipping consumer was still looking for the old field name. The event carried valid data, but the consumer ignored it.

**Mitigations**:
- Mandatory distributed tracing (OpenTelemetry)
- Structured logging with correlation IDs in every log line
- Dashboard showing event flow with per-stage latency and error rates
- Dead letter queue with detailed failure metadata
- Regular event replay testing to verify the full pipeline

## Anti-Patterns

### Anti-Pattern 1: Over-Eventizing

Every minor state change becomes an event. A user types a character in a form field -> `CharacterTyped` event. The form field loses focus -> `FieldBlurred` event. The user scrolls -> `ScrollEvent`. Events flood the broker. Most consumers ignore them. The broker fills up with noise.

**Fix**: Only emit events for meaningful business occurrences. A character typed is not a business event. An order placed is.

### Anti-Pattern 2: God Events

A single event type carries data for every possible consumer. The `OrderEvent` includes fields for inventory (stock levels), billing (payment details), shipping (address, weight, dimensions), notifications (email templates), and analytics (user agent, session ID). Every consumer receives the same massive event payload, even though each consumer only needs a subset.

**Fix**: Split god events into focused event types. `OrderPlaced` for order data. `PaymentProcessed` for payment data. `ShipmentCreated` for shipping data. Each consumer subscribes only to the events it needs.

### Anti-Pattern 3: No Schema

Events are published as raw JSON with no schema validation. Producers change field names, types, and structure without coordination. Consumers crash when they receive unexpected data. The system degrades silently — no consumer is sure it is receiving valid data.

**Fix**: Use a schema registry from day one. Validate every event against its schema. Enforce compatibility rules on schema evolution.

### Anti-Pattern 4: No Idempotency

The team assumes at-most-once delivery. When the broker delivers the same event twice (which it will due to retries and consumer restarts), the consumer processes it twice. Inventory is double-decremented. Customers are double-charged. Duplicate notifications are sent.

**Fix**: Design every event handler to be idempotent. Use deduplication keys (event_id), upsert operations, or conditional writes. Test idempotency by replaying events.

### Anti-Pattern 5: Synchronous Event Handling

The consumer makes synchronous API calls within the event handler, blocking the event processing thread. When the downstream API is slow, the consumer processing rate drops, lag accumulates, and the broker storage fills up.

```typescript
// Anti-pattern: synchronous call in handler
async function handleOrderPlaced(event: Event): Promise<void> {
  const customer = await customerApi.getCustomer(event.payload.customerId); // blocks!
  const emailSent = await emailService.sendOrderConfirmation(customer.email); // blocks!
  // Or worse:
  await eventBus.publish("notification", notificationEvent); // blocking publish
}
```

**Fix**: Event handlers should be fast. Offload synchronous calls to a separate thread pool or use asynchronous processing. The handler should do local work (validate, transform, store) and emit a new event if external work is needed.

## When NOT to Use Event-Driven Architecture

Event-driven architecture is powerful, but it is not the right answer for every problem. Recognize when it adds complexity without benefit.

### Avoid EDA for CRUD APIs

If your service receives a request, performs a database operation, and returns a response, a synchronous REST or gRPC API is the right choice. Adding an event bus between the request and response adds latency, complexity, and operational overhead with no benefit.

### Avoid EDA for Strong Consistency Requirements

If your system requires strong consistency — every read returns the latest write — event-driven architecture cannot provide it. The asynchronous nature of events means there is always a window of inconsistency. Use a database with distributed transactions or a synchronous API.

### Avoid EDA for Low-Latency Requirements Under 10ms

Event brokers add latency. Kafka adds 2-5ms of latency per hop. Multiple hops in an event chain compound this. If your end-to-end latency requirement is under 10ms, synchronous calls are more appropriate.

### Avoid EDA for Simple Workflows with 2-3 Services

If your system has two services that communicate, and the communication pattern is simple (Service A calls Service B), an event bus adds unnecessary complexity. Use direct HTTP calls or a simple message queue.

### Avoid EDA When You Cannot Operate the Infrastructure

Event brokers require specialized operational knowledge. Kafka requires understanding of replication, partitioning, consumer groups, offsets, and rebalancing. A misconfigured Kafka cluster can lose data, corrupt events, or become unavailable. If your team does not have the operational capacity, use a managed service (Confluent Cloud, AWS MSK, Redpanda) or skip EDA altogether.
