## The Problem That Created Kafka

In 2010, LinkedIn faced a data infrastructure crisis. The engineering team had built dozens of services — profile servers, connection recommenders, analytics pipelines, search indexes, newsfeed generators — and each service talked to every other service through direct point-to-point integrations.

A single team that wanted data from four sources had to negotiate four separate APIs, four authentication mechanisms, four data formats, and four delivery guarantees. When a new consumer joined, every producer had to add another integration. When a producer changed its schema, every consumer had to update. The system resembled a tangled web of custom pipes, each built by a different engineer under different constraints.

The numbers tell the story. LinkedIn's infrastructure processed over 100 billion events per day by 2010. The application servers generated terabytes of log data each day. The search index had to be rebuilt every few hours from hundreds of source systems. The team estimated that 30% of engineering time went toward building and maintaining data pipelines — not toward the product features that users actually saw.

Jay Kreps, Neha Narkhede, and Jun Rao set out to build something different. They designed a system with a single insight: instead of having every service negotiate its own data transfer, all data should flow through a centralized, append-only log. Any producer writes to this log. Any consumer reads from this log. The log decouples producers from consumers in both time and space.

The result was Kafka, open-sourced in early 2011. By 2024, Kafka processes over 100 trillion messages per day across installations at LinkedIn, Uber, Netflix, Spotify, Airbnb, and tens of thousands of other companies.

## The Insight: A Distributed Commit Log

A commit log is one of the oldest data structures in computing. It is an append-only sequence of records. Every new record goes at the end. Records are never modified in place. The log records what happened, in order, as it happened.

Kafka's core insight was to take this simple idea — the commit log — and distribute it across a cluster of machines while preserving its essential properties: ordering within a partition, durability through replication, and the ability to replay history from any point.

Think of Kafka as a classified ads newspaper. Producers place ads (messages) in specific sections (topics). Subscribers (consumers) browse the sections that interest them. The newspaper never takes ads down (retention period). If you missed last week's ads, you can go back and read them. If you want to read ads as they arrive, you wait for the next edition. The newspaper has multiple copies stored in different libraries (replication), so losing one library does not destroy the archive.

This decoupling is powerful. A producer does not know or care which consumers read its messages. A consumer does not know or care which producers wrote the messages it reads. The log sits between them, absorbing bursts, tolerating failures, and preserving history.

## Core Vocabulary

These terms form the foundation of every Kafka conversation. You will encounter them in documentation, architecture reviews, and interview questions.

### Topic

A topic is a named category or feed that stores records. Topics are the primary abstraction through which data is organized. If you are building an e-commerce platform, you might have topics named `order-placed`, `payment-processed`, `inventory-updated`, and `shipping-label-generated`. Topics are multi-subscriber — any number of consumers can read from the same topic independently.

### Partition

A topic is divided into partitions. A partition is an ordered, immutable sequence of records. Each record within a partition gets a sequential ID called an offset. Partitions are the unit of parallelism in Kafka. A topic with 12 partitions can be consumed by up to 12 consumers in a consumer group running in parallel.

The ordering guarantee is partition-scoped. Records within partition 0 of topic `orders` are strictly ordered. Records across partitions have no global order. This is a critical design choice that enables horizontal scaling.

### Producer

A producer is a client that publishes records to a Kafka topic. Producers choose which partition to write to. They can distribute records round-robin, hash on a key (so all records with the same key go to the same partition), or use a custom partitioner. Producers can request acknowledgments for writes, trading throughput for durability.

### Consumer

A consumer is a client that reads records from one or more partitions. Consumers poll Kafka for new records. Each consumer maintains its position in each partition using an offset. When a consumer starts, it can choose to read from the earliest offset, the latest offset, or a specific offset.

### Consumer Group

A consumer group is a set of consumers that coordinate to consume a topic. The partitions of the topic are divided among the consumers in the group. If you have 8 partitions and 4 consumers in a group, each consumer handles 2 partitions. If you add an 8th consumer, each handles 1 partition. If you add a 9th consumer, it stays idle — there is no work for it.

Consumer groups enable horizontal consumption. They also provide fault tolerance: if a consumer in a group fails, the partitions it was handling are reassigned to the remaining members.

### Broker

A broker is a single Kafka server. Brokers form a cluster. Each broker stores some partitions and handles produce and consume requests for those partitions. A typical production cluster has 6 to 100+ brokers. Brokers are stateless from the client's perspective — clients connect to any broker and the broker routes the request to the appropriate partition leader.

### Offset

An offset is a sequential integer that identifies each record within a partition. Offsets are monotonic but not necessarily contiguous — if a producer writes and the broker crashes before acknowledging, the client retries and the record may be written at a higher offset while the gap remains. Consumers track their position by storing the last processed offset, which enables exactly-once semantics when combined with idempotent producers.

### Log

The log is the physical storage structure on disk. Each partition maps to one or more log segments. Records are appended to the active segment. When the segment reaches a size or time threshold, the broker creates a new segment. Older segments are eligible for compaction or deletion based on the retention policy. The log is the foundation of Kafka's durability guarantee.

### Replication

Replication is the mechanism that prevents data loss when a broker fails. Each partition has a configurable replication factor, typically 3. One replica is the leader; the rest are followers. The leader handles all produce and consume requests. Followers replicate the leader's data asynchronously. If the leader fails, a follower is elected as the new leader.

### Leader and Follower

The leader is the replica that serves all client requests for a partition. Followers keep in sync by fetching records from the leader. A partition has exactly one leader at any time. Leadership is distributed across brokers — broker 1 might be the leader for partition 0 of topic A and a follower for partition 0 of topic B.

### In-Sync Replica (ISR)

An ISR is a replica that is fully caught up with the leader. If a follower falls behind by more than `replica.lag.time.max.ms` (default 30 seconds), the leader removes it from the ISR set. When the producer uses `acks=all`, the leader waits for acknowledgment from all ISRs before responding to the producer. ISR management is the core of Kafka's consistency model.

### ZooKeeper and KRaft

ZooKeeper was the original metadata store for Kafka. It tracked broker membership, topic configurations, and partition leadership. ZooKeeper introduced operational complexity — it required its own cluster with its own tuning, monitoring, and failure modes.

KRaft (Kafka Raft) is the modern replacement. Introduced in Kafka 2.8 as experimental and production-ready in Kafka 3.5+, KRaft removes the ZooKeeper dependency by implementing the Raft consensus protocol directly in Kafka. KRaft controllers manage metadata and coordinate partition leadership elections. KRaft simplifies operations — one system instead of two.

### Retention

Retention is the policy that determines how long Kafka keeps records. Two policies exist: time-based (`retention.ms`, default 7 days) and size-based (`retention.bytes`). Records that exceed the retention threshold are eligible for deletion. Kafka does not delete individual records — it deletes entire log segments.

### Compaction

Compaction is an alternative to deletion-based retention. Instead of deleting old records, compaction retains the most recent record for each key. This is useful for topics that store state — such as the latest known address for each customer or the current inventory level for each SKU. Kafka logs are append-only, so compaction is the mechanism that prevents the log from growing unbounded while preserving the latest value for each key.

### Schema Registry

The Schema Registry is a separate service that stores and validates schemas for Kafka messages. It enforces schema evolution rules — for example, adding a field with a default value is allowed, but removing a field is not. Producers register schemas when they write; consumers fetch schemas when they read. The Schema Registry uses Apache Avro, JSON Schema, or Protobuf.

### Record Batch

Kafka does not send individual records over the network. It groups records into batches. A batch is a collection of records destined for the same partition. Batching is essential for performance — sending 100 records in one batch requires one network round trip instead of 100. The producer accumulates records into batches based on `batch.size` and `linger.ms`. The broker stores and serves batches as atomic units.

### Watermark

The watermark is the offset up to which all records have been replicated to all in-sync replicas. Consumers with `isolation.level=read_committed` see only records up to the watermark. Records beyond the watermark are not yet committed and may be lost if the leader fails. The watermark advances as the leader receives acknowledgments from ISR followers.

### Client ID

Each producer and consumer instance is identified by a `client.id`. This identifier appears in broker logs, metrics, and quota tracking. Standardizing client ID naming conventions is essential for operational visibility. A common convention is `{service}-{environment}-{instance}` — for example, `order-processor-prod-01`.

### Group Coordinator

Each consumer group is managed by a group coordinator, which is one of the brokers in the cluster. The coordinator handles consumer group membership, partition assignment, and offset management. The coordinator is elected based on the consumer group's name (hash to a broker). The coordinator is the single point of authority for the group's state.

### Consumer Offset Topic

Kafka stores consumer offsets in an internal topic called `__consumer_offsets`. This topic has a configurable number of partitions (default 50). Each consumer group maps to one partition of this topic. The offset topic is compacted — Kafka retains only the latest offset for each consumer group and partition. This prevents the offset topic from growing unbounded while ensuring that consumers can always find their last committed offset.

### Quota

Kafka supports quota-based throttling for producers and consumers. Two types of quotas exist: `producer_byte_rate` (bytes per second for produce requests) and `consumer_byte_rate` (bytes per second for fetch requests). Quotas prevent a single misbehaving client from degrading the cluster. When a client exceeds its quota, the broker delays the response, causing the client to back off. Quotas are configurable per client ID or per user.

### Step 1: Topic Creation

An operator creates a topic called `orders` with 6 partitions and a replication factor of 3. The control plane records this metadata in ZooKeeper or KRaft. The metadata broker assigns partition leaders. For a 3-broker cluster, partition 0 might have leadership on broker 1 with followers on brokers 2 and 3.

### Step 2: Producer Connection

The producer initializes by connecting to any broker and requesting metadata about the `orders` topic. The broker responds with the partition layout and which broker is the leader for each partition. The producer caches this metadata and refreshes it periodically.

### Step 3: Partition Assignment

The producer receives a message: `{"orderId": "ORD-12345", "userId": 42, "amount": 2999, "timestamp": "2026-07-02T10:30:00Z"}`. The producer applies its partitioner. If using key-based partitioning with `userId` as the key, Kafka hashes the key to determine the partition. `hash(42) % 6 = partition 3`. The producer now knows to send this record to the leader of partition 3, which is broker 2.

### Step 4: Producer Sends

The producer sends the record to broker 2 with configuration: `acks=all`, `compression=snappy`, and a 30-second timeout. The producer batches this record with other records bound for partition 3 to improve throughput.

### Step 5: Broker Append

Broker 2 (leader for partition 3) receives the produce request. It validates the record. It appends the record to the current log segment for partition 3. The append is sequential I/O — the record goes to the end of the file. Sequential writes on modern hardware operate at 200-500 MB/s per disk, compared to 1-10 MB/s for random I/O.

### Step 6: Replication to Followers

The leader (broker 2) sends the record to the followers on brokers 1 and 3. Followers acknowledge once they have appended the record to their local logs. When all in-sync replicas have acknowledged, the leader considers the write committed. With `acks=all` and `min.insync.replicas=2`, the leader waits for at least one follower acknowledgment before responding to the producer.

### Step 7: Producer Acknowledgment

Broker 2 sends an acknowledgment back to the producer. The acknowledgment includes the partition and offset where the record was written. The producer uses this offset for ordering and deduplication.

### Step 8: Consumer Poll

On the consumer side, a consumer in the `order-processor` group polls for new records. It sends a fetch request to broker 2 (leader of partition 3), specifying the partition and its current offset. The broker sends back any records written since the consumer's last poll.

### Step 9: Consumer Processes

The consumer processes the records. It might write to a database, call an API, update a cache, or trigger a downstream workflow. The processing logic is entirely the consumer's responsibility — Kafka does not validate or enforce processing semantics.

### Step 10: Offset Commit

After processing, the consumer commits its offset. It sends the new offset (the last processed offset plus one) to the internal `__consumer_offsets` topic. This allows the consumer to resume from the correct position if it restarts or if a rebalance moves partitions to a different consumer.

If the consumer crashes between processing and committing, it will reprocess some records when it restarts — this is the at-least-once delivery semantic. To achieve exactly-once semantics, consumers must use Kafka's transactional API or implement idempotent processing.

### Step 11: Consumer Rebalance (Failure Handling)

If consumer 3 in the group crashes, the remaining consumers detect the failure after `session.timeout.ms` passes without a heartbeat. The group coordinator triggers a rebalance. It removes the failed consumer from the group and reassigns its partitions (partition 3 and 4 in our example) to the remaining consumers. Consumer 1 now handles partitions 0, 1, and 3. Consumer 2 handles partitions 2, 4, and 5.

During the rebalance, all consumers pause processing. Once the rebalance completes, consumers resume from the last committed offset for each partition.

### Step 12: Consumer Group Scaling

The operations team adds 3 more consumers to the `order-processor` group, bringing the total to 6 consumers for 6 partitions. Each consumer now handles exactly one partition. Throughput increases because the processing load for each partition is isolated — a slow consumer for partition 0 does not affect partition 1.

The group coordinator detects the new consumers and triggers a rebalance. The cooperative rebalancing protocol moves partitions one at a time, minimizing the disruption. Each consumer drops one partition, which is picked up by the new consumer.

## Variants and Alternatives

Kafka's success has spawned several variants and managed services. Understanding the landscape helps you choose the right tool for your context.

### Apache Kafka (Open Source)

The original. Maintained by the Apache Software Foundation. Free, self-hosted. Requires operational expertise for ZooKeeper or KRaft management. Community-driven with releases every few months. Suitable for organizations that have the engineering capacity to operate infrastructure.

### Confluent Kafka

Founded by the original creators of Kafka. Confluent builds on Apache Kafka with proprietary extensions: the Schema Registry (open source core, with enterprise features), ksqlDB, Kafka Connect connectors, role-based access control, and Multi-Region Clusters. Confluent Platform is self-hosted; Confluent Cloud is a fully managed service.

Key differentiators: Cluster Linking for multi-region replication without MirrorMaker, Kora Engine for cloud-native broker architecture, and Partition Reassignment for automated cluster balancing.

### Redpanda

A Kafka-compatible event streaming platform written in C++. Redpanda replaces ZooKeeper with the Raft consensus protocol, eliminates the JVM dependency (no garbage collection pauses), and uses a thread-per-core architecture. It achieves 10x lower latency than Apache Kafka for some workloads while maintaining API compatibility.

Redpanda does not support all Kafka features — Kafka Streams, Connect, and the full admin API are not fully replicated. It excels in latency-sensitive environments where the Kafka API is used primarily for produce and consume.

### Amazon MSK (Managed Streaming for Kafka)

AWS's managed Kafka service. MSK provisions the Kafka cluster, handles broker replacement, and integrates with AWS IAM for authentication. It supports Apache Kafka's full API, so existing clients work without modification. MSK Serverless is a recent addition that abstracts cluster management entirely.

MSK is not a fork — it is vanilla Apache Kafka with AWS management tooling. You pay for broker instances and storage. MSK Serverless charges per partition-hour and data written.

### Confluent Cloud

Confluent's fully managed Kafka service running on AWS, Azure, or GCP. Provides Schema Registry, ksqlDB, and Connect as managed services. Offers multi-region clusters with synchronous replication. Charges based on throughput rather than cluster size.

### Apache Pulsar

A separate technology often compared to Kafka. Pulsar uses a two-layer architecture — a serving layer for handling requests and a bookkeeping layer (Apache BookKeeper) for storage. This separation allows Pulsar to support more partitions and provide stronger durability guarantees at the cost of higher latency. Pulsar supports native geo-replication and multi-tenancy.

### Google Pub/Sub

Google's fully managed messaging service. It provides Kafka-like semantics with automatic scaling and no cluster management. Pub/Sub uses a pull or push delivery model. It supports exactly-once delivery and global message ordering when using ordering keys. Pub/Sub does not expose offsets or consumer groups in the same way Kafka does.

### Kafka vs. Traditional Message Queues

Understanding the difference between Kafka and traditional message queues (RabbitMQ, ActiveMQ, SQS) clarifies when to use each.

| Feature | Kafka | Traditional Message Queue |
|---------|-------|--------------------------|
| **Data model** | Append-only log, partitioned | Queue, typically single-consumer per message |
| **Delivery semantics** | At-least-once by default, configurable to exactly-once | At-most-once, at-least-once, or exactly-once |
| **Message retention** | Configurable (time/size), messages persist after consumption | Message deleted after acknowledgment |
| **Consumption model** | Publish-subscribe (multiple consumer groups) | Queue (one consumer per message) or pub-sub |
| **Ordering** | Per-partition ordered | Best-effort or single-queue ordered |
| **Throughput** | Millions of messages per second (partitioned) | Thousands to hundreds of thousands per second |
| **Message routing** | Topic-based (partition key determines destination) | Exchange-based (routing keys, bindings, headers) |
| **Message state** | Broker tracks offset per consumer group | Broker tracks per-message acknowledgment |
| **Replay** | Any consumer can replay from any offset | Not supported (message deleted after ack) |
| **Latency** | 2-15 ms typical | Sub-millisecond to 5 ms typical |
| **Operational complexity** | High (multi-broker, ZooKeeper/KRaft) | Low to moderate |

**When to choose a message queue over Kafka:**
- Each message has independent processing that must be individually acknowledged.
- You need complex routing (topic exchanges, header matching, delayed queues).
- Sub-millisecond latency is required.
- Operational simplicity matters more than throughput.
- The workload is under 10,000 messages per day.

**### Kafka's Relationship with Stream Processing

Kafka's log-based architecture is the foundation of stream processing. Stream processing frameworks — Kafka Streams, Apache Flink, Apache Spark Streaming — all consume from Kafka topics, process records, and produce results back to Kafka topics. Kafka provides the durable, replayable, ordered event stream that stream processing requires.

The relationship between Kafka and stream processing can be summarized as:

- Kafka provides the storage layer for event streams.
- Stream processing provides the computation layer on top of those streams.
- Kafka's offsets enable exactly-once processing guarantees when combined with transactional APIs.
- Kafka's partitions enable parallel stream processing — each partition can be processed independently by a separate task.

This separation of storage and computation is a key architectural pattern. The event stream is durable and immutable in Kafka. The processing logic is decoupled and can be changed, replayed, or deployed independently of the storage.

### Kafka and Change Data Capture

Change Data Capture (CDC) is one of Kafka's most powerful use cases. CDC connectors like Debezium read the database transaction log and produce change events to Kafka topics. Each database row change becomes a Kafka record.

CDC with Kafka provides:

- **Real-time replication**: Changes appear in Kafka within milliseconds of the database commit.
- **Decoupled consumers**: Multiple downstream systems (search index, cache, data warehouse, analytics) consume from Kafka independently.
- **Replay capability**: If a downstream system needs to rebuild its state, it can replay the change events from the beginning of the Kafka topic.
- **Schema evolution**: The Schema Registry tracks each version of the database schema, enabling consumers to handle schema changes gracefully.

The combination of Kafka and CDC has largely replaced batch ETL pipelines in modern data architectures. Instead of running hourly or daily batch jobs to move data between systems, changes flow continuously through Kafka.

When to choose Kafka over a message queue:**
- You need to replay historical data.
- Multiple consumer groups need independent access to the same data stream.
- Throughput exceeds 100,000 messages per second.
- You need long-term retention of event history.
- The workload is streaming (continuous data flow) rather than queuing (discrete jobs).

## Mental Model Check

These seven questions test your understanding of Kafka's core concepts. Try to answer each one before reading the explanation.

### Question 1

You have a topic with 4 partitions and a consumer group with 6 consumers. How many partitions will the 6th consumer handle?

**Answer:** Zero. A consumer group can have at most as many active consumers as there are partitions in the subscribed topics. Extra consumers remain idle until another consumer in the group fails. This is by design — partitions are the unit of parallelism, and each partition is consumed by exactly one consumer within a group.

### Question 2

What happens to records in a partition when a consumer has already read them?

**Answer:** Nothing. Records remain in the log until the retention policy deletes their log segment. Multiple consumer groups can read the same records independently because each group maintains its own offset. Consumer group A reading offset 1000 does not affect consumer group B reading offset 500. This is the publish-subscribe model — each subscriber maintains independent state.

### Question 3

Your producer uses `acks=all` and the topic has `replication.factor=3` with `min.insync.replicas=2`. One broker goes offline. The remaining two brokers include the leader and one follower. The follower is in the ISR. Can the producer continue writing?

**Answer:** Yes. With two brokers still running and one follower in the ISR, the condition `min.insync.replicas=2` is satisfied (leader + follower = 2). The leader waits for acknowledgment from the single ISR follower before responding. When the failed broker returns, it resumes following the leader and rejoins the ISR once it catches up.

### Question 4

You change the partitioning key from `userId` to `orderId`. What impact does this have on existing records?

**Answer:** None for existing records — they remain in their original partitions. The change affects only new records. If a consumer assumes that all records for a given `userId` land in the same partition, this assumption breaks for records written after the change. Old records for user 42 are in partition 3; new records for user 42 might land in partition 1. This is why the partitioning strategy must be stable for the lifetime of a use case.

### Question 5

A consumer reads a message, processes it, and crashes before committing the offset. What happens when it restarts?

**Answer:** The consumer starts from the last committed offset, which is the offset before the crashed message. The consumer processes that message again. This is at-least-once delivery. To avoid duplicate processing, the consumer must make its processing idempotent — performing the same operation twice produces the same result as performing it once. Alternatively, the consumer can use Kafka's transactional API for exactly-once semantics, but this adds latency and complexity.

### Question 6

You produce 10,000 records with keys to a topic with 6 partitions. The keys are distributed across 1,000 unique values. How many partitions receive records, assuming the default murmur2 partitioner?

**Answer:** Likely all 6, but not evenly. The murmur2 hash distributes keys pseudorandomly across partitions. With 1,000 keys and 6 partitions, the expected number of keys per partition is 1,000 / 6 = 167. In practice, the distribution is close to uniform but not exact. Some partitions receive 150 keys, others 185. If the volume per key varies (a small number of keys generate most of the traffic), the skew can be severe. This is why key-based partitioning can create hot partitions.

### Question 7

You set `auto.offset.reset=earliest` on a consumer group that has never consumed before (no committed offset). The topic has 50 million records. What happens when the consumer starts?

**Answer:** The consumer reads from offset 0 and starts processing all 50 million records. If the consumer's processing speed is 1,000 records per second, it takes 50,000 seconds (approximately 14 hours) to catch up to the latest record. During this time, the consumer is processing old data while new data continues to arrive. New records are appended to the log beyond offset 50 million. When the consumer eventually catches up, it starts processing near-real-time records normally.

### Question 8

Two consumer groups read from the same topic. Group A is lagging by 100,000 records. Group B is up to date. Does Group B's consumption affect Group A's ability to catch up?

**Answer:** No. Each consumer group maintains its own offset. Group A can read from its lagging position while Group B reads new records. The groups do not interact through offsets. The only shared resource is broker I/O — if both groups read heavily, they compete for disk and network bandwidth. But the offsets themselves are independent.

## Kafka in Microservices Architecture

Kafka is not just a messaging system — it is an architectural pattern for microservices communication.

### Event-Driven Communication

In a traditional request-driven microservice architecture, Service A calls Service B directly via HTTP or gRPC. This creates tight coupling: Service A must know Service B's endpoint, must handle Service B's latency, and must deal with Service B's failures.

In an event-driven architecture with Kafka, Service A produces an event to a Kafka topic. Any service that cares about that event consumes it independently. Service A does not know which services consume its events. Service B, Service C, and Service D can each consume the event without Service A knowing about them.

```
Request-driven:    A ──HTTP──▶ B
Event-driven:      A ──Kafka──▶ B
                               ├──▶ C
                               └──▶ D
```

### Event Sourcing with Kafka

Event sourcing is the practice of storing the full sequence of events that led to the current state, rather than storing the state itself. Kafka's append-only log is a natural fit for event sourcing.

In an event-sourced system:
- Every state change is an event appended to a Kafka topic.
- The current state is rebuilt by replaying the events.
- Events are never deleted (compaction can remove old events for the same key).
- The event log is the source of truth. The database is a derived view.

### CQRS with Kafka

Command Query Responsibility Segregation (CQRS) separates read operations from write operations. Kafka supports CQRS by acting as the write-side event store. Read-side views are maintained by consuming from Kafka topics and building specialized projections.

For example, an e-commerce system might use:
- A `commands` topic for write requests (place order, cancel order).
- An `events` topic for state changes (order-placed, order-shipped, order-cancelled).
- Read-side consumers that build order-search indexes, customer-history views, and inventory snapshots from the events topic.

### Kafka and Observability

Kafka plays a central role in observability infrastructure. Three use cases dominate:

**Log aggregation:** Services produce structured log events to Kafka topics. A log consumer (Filebeat, Fluentd, Logstash) reads from Kafka and writes to Elasticsearch or a cloud log service. Kafka provides the buffer that absorbs log bursts from thousands of services.

**Metrics pipeline:** Time-series metrics (CPU usage, request latency, error rates) are collected from services and produced to Kafka. A stream processor aggregates metrics into time windows and writes to a time-series database (Prometheus, Thanos, VictoriaMetrics).

**Distributed tracing:** Trace spans from services are produced to Kafka topics. A trace consumer reads spans, correlates them by trace ID, and writes the complete traces to a trace storage backend (Jaeger, Tempo). Kafka ensures that trace data is not lost even during high-traffic periods.

The common pattern across all three use cases is the same: services produce observability data to Kafka asynchronously, decoupling data collection from data processing. The observability pipeline can scale independently of the services producing the data.

## Summary

Kafka solves the problem of data integration at scale by providing a distributed commit log. Topics organize data into categories. Partitions enable parallelism. Replication provides durability. Consumer groups enable scalable consumption. The log is the central abstraction — an immutable, ordered, replayable sequence of records.

**Key takeaways:**

- Kafka is a distributed commit log, not a traditional message queue. Records are persisted and can be replayed.
- Topics are split into partitions for parallelism. Ordering is guaranteed within a partition, not across partitions.
- Replication provides fault tolerance. The ISR mechanism handles consistency.
- Consumer groups enable multiple independent consumers to read the same data stream.
- Kafka's performance derives from sequential I/O, zero-copy data transfer, and OS page cache management.
- Producers and consumers are decoupled in time and space. The log sits between them, absorbing bursts and preserving history.
- Kafka is a CP system by default, favoring consistency over availability during failures.
- The Schema Registry, Kafka Connect, and Kafka Streams form the extended Kafka ecosystem for integration and stream processing.
