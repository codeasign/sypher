## Tier 1: Configuration and Operations

### Challenge 1: The Slow Consumer

**Problem:**

You manage a Kafka cluster that processes order events for an e-commerce platform. The `orders` topic has 12 partitions with a replication factor of 3 on a 6-broker cluster. The produce rate is 20,000 records per second at an average size of 1 KB. A consumer group called `order-processor` with 4 consumers handles the events.

Over the past week, `order-processor` consumer lag has grown from 1,000 records to 50,000 records during peak hours (2 PM to 4 PM). The service team reports processing time of 200 ms per record, which includes a database write and a cache update.

**Constraints:**
- The database cannot handle more than 5,000 writes per second.
- The cache update is fast (less than 1 ms).
- The consumer group cannot use more than 8 consumers due to connection pool limits.

**Questions:**

1. What is the root cause of the growing consumer lag?
2. What changes would you make to eliminate the lag?
3. What monitoring would you put in place to prevent recurrence?

<details>
<summary>**Model Answer**</summary>

**Root cause:**

The `order-processor` consumer group has 4 consumers and the topic has 12 partitions. Each consumer handles 3 partitions. At 20,000 records per second total, each consumer receives 5,000 records per second (20,000 / 4). At 200 ms per record, a single consumer processes at 5 records per second. This means each consumer ingests 5,000 records per second but processes only 5 records per second. The lag grows by 4,995 records per second per consumer.

**Three compounding factors:**
- Processing time per record (200 ms) is far too high for the ingestion rate.
- The database limit of 5,000 writes per second is not the bottleneck вҖ” the consumers are not even reaching that limit.
- At the current throughput, one consumer processes 5 records per second x 3600 seconds = 18,000 records per hour. During peak 2 hours, 40,000 records arrive per consumer (20,000 / 4 x 2), but only 180 records are processed per consumer (5 x 3600 x 2 / 1,000).

Wait вҖ” the math: 20,000 records per second across 4 consumers = 5,000 per consumer per second. At 200 ms per record, each consumer processes 5 per second. So each consumer falls behind by 4,995 records per second. Over 2 hours, that is 4,995 x 7200 = 35,964,000 records of lag per consumer. But the problem states lag grew to 50,000 records total. This suggests the produce rate during peak is not 20,000 continuous вҖ” it is bursty, with 20,000/sec peaks and off-peak periods where consumers catch up partially.

**Changes:**

1. Batch the database writes. Instead of writing one record at a time (200 ms per write), batch 100 records into a single batch write. Database vendors typically support batch inserts. Measure the batch write latency, but it will be closer to 300 ms for 100 records than 20,000 ms. This changes processing time from 200 ms per record to (300 ms + 1 ms cache update) per 100 records = 3.01 ms per record.

2. Increase consumer count. With 12 partitions, the maximum effective consumers is 12. Add 8 more consumers (total 12). This distributes the 20,000 records/s across 12 consumers at 1,667 records/s per consumer. With batched writes at 3 ms per record equivalent, each consumer processes 333 records per second, well above the 1,667 ingestion rate. Wait вҖ” at 3 ms per record batch (100 records), that is 33,333 records per second per consumer. This is far above the 1,667 needed. The database limit of 5,000 writes/s is now the bottleneck. With 12 consumers each writing 5 batches of 100 records per second, that is 12 x 5 = 60 writes per second to the database, well within the 5,000 limit.

3. If the database remains a bottleneck, introduce a write buffer. Accumulate records in memory (or a Redis queue) and flush to the database every second.

**Monitoring:**
- Alert on `records-lag-max` for the `order-processor` group at thresholds of 1,000 (warning) and 10,000 (critical).
- Monitor `records-lag` per partition to detect hot partitions.
- Track processing time per record (consumer-side metric).
- Track database write throughput and latency.
</details>

### Challenge 2: The Expanding Disk

**Problem:**

You receive a disk alert: broker 3 is at 82% disk capacity. The cluster has 6 brokers, each with 4 TB NVMe drives. The `events` topic uses 80% of the total storage and has a 7-day retention policy.

**Investigation findings:**

- The `events` topic has 120 partitions and RF=3.
- Storage distribution: broker 3 has 3.2 TB used. Other brokers average 1.8 TB.
- The produce rate to `events` has doubled over the last month from 100 MB/s to 200 MB/s.
- The `events` topic is keyed by `user_id`.

**Questions:**

1. Why is broker 3 disk usage significantly higher than other brokers?
2. What is the projected time to disk full?
3. What actions would you take to resolve the imbalance and prevent recurrence?

<details>
<summary>**Model Answer**</summary>

**Cause of imbalance:**

Broker 3 likely hosts more partition leaders or partition replicas than other brokers. With 120 partitions and RF=3, there are 360 replica assignments across 6 brokers вҖ” 60 per broker on average. Broker 3 has more than 60.

The key-based partitioning by `user_id` may create hot partitions вҖ” certain user ranges produce significantly more data. If the hot partitions land on broker 3, its disk fills faster. This is partition skew combined with leadership imbalance.

**Projection:**
- Broker 3: 3.2 TB used out of 4 TB. Remaining: 0.8 TB.
- Growth rate: 200 MB/s produce rate with RF=3 across 6 brokers. If broker 3 hosts 60 replicas (average), its storage grows at 200 MB/s x 3 (RF) / 6 (brokers) x (broker 3 replica count / average replica count). If broker 3 has 80 replicas instead of 60, its growth rate is 200 x 3 / 6 x 80/60 = 133 MB/s.
- Time to full: 800 GB / 133 MB/s = 6,015 seconds = 1.7 hours.

This is an emergency. Action must be taken within the hour.

**Immediate actions:**

1. Free space by reducing retention for the `events` topic: `kafka-configs.sh --alter --entity-type topics --entity-name events --add-config retention.ms=259200000` (3 days instead of 7). This immediately marks segments older than 3 days for deletion. Expect to recover 50-60% of space.

2. Trigger log cleaner to run immediately: increase `log.cleaner.backoff.ms` to accelerate segment deletion.

3. After freeing space, investigate the partition distribution. Run `kafka-topics.sh --describe --topic events` to identify which partitions are on broker 3.

**Structural fix:**

1. Generate a partition reassignment plan to move partitions off broker 3, rebalancing replica count across all 6 brokers.

2. If the hot partitions are concentrated on certain brokers because of key-based partitioning, consider adding partitions to the `events` topic (if the partition count cannot be increased without downtime, create a new topic with more partitions and migrate producers).

3. Enable rack awareness if not already active (assuming 3 racks, 2 brokers per rack).

4. Set per-broker disk usage monitoring with alerts at 70% warning, 85% critical.

5. Add a capacity plan: with the produce rate growing at 2x per month, add 2 brokers per month for the next 3 months, or reduce retention further.
</details>

### Challenge 3: The Failing Rebalance

**Problem:**

Your team deploys a new version of a consumer application. After the deployment, the consumer group enters a continuous rebalance loop. The consumer group has 8 consumers processing a topic with 16 partitions. During the deployment, all 8 instances are restarted within 2 minutes.

**Error logs show:**
```
[Consumer client] Member consumer-1-1234 sending LeaveGroup request
[Consumer client] Attempt to heartbeat failed since group is rebalancing
[Consumer client] Revoking previously assigned partitions
[Consumer client] Rebalance failed due to Member consumer-2-5678 was fenced by the group coordinator
```

**Configuration:**
- `session.timeout.ms`: 10,000
- `heartbeat.interval.ms`: 3,000
- `max.poll.interval.ms`: 30,000
- `max.poll.records`: 500
- Average processing time per record: 50 ms

**Questions:**

1. What is causing the continuous rebalance?
2. What configuration changes would prevent this?
3. How would you change the deployment process to avoid the rebalance storm?

<details>
<summary>**Model Answer**</summary>

**Root cause:**

The deployment restarts all 8 consumers within 2 minutes. Each consumer leaving triggers a rebalance. With 8 consumers leaving in quick succession, the group coordinator initiates 8 separate rebalances. During each rebalance, no consumer can process records. The 30-second `max.poll.interval.ms` means any consumer that takes longer than 30 seconds to process during startup (for example, warming caches, connecting to databases) is considered failed, triggering another rebalance.

The average processing time of 50 ms per record with `max.poll.records=500` means a single poll batch takes 25 seconds to process. This is dangerously close to the 30-second `max.poll.interval.ms`. If garbage collection or network latency adds 5 seconds, the consumer exceeds the limit.

**Configuration changes:**

1. Increase `session.timeout.ms` to 45,000 (the Kafka default). This prevents false-positive failures during GC pauses and network jitter.

2. Increase `max.poll.interval.ms` to 60,000. This provides a 2x safety margin above the 25-second processing time per batch.

3. Set `heartbeat.interval.ms` to 15,000 (1/3 of session.timeout.ms). This matches the recommended ratio.

4. Reduce `max.poll.records` to 200. This lowers processing time per batch to 10 seconds, providing a 6x safety margin.

5. Enable the cooperative rebalancing protocol: set `partition.assignment.strategy` to `org.apache.kafka.clients.consumer.CooperativeStickyAssignor`. Cooperative rebalancing does not stop all consumers вҖ” only the affected partitions are reassigned, and remaining consumers continue processing.

**Deployment process changes:**

1. Use a rolling restart with a 5-minute delay between each consumer restart. This gives the group coordinator time to complete each rebalance before the next restart.

2. Implement a graceful shutdown hook that sends a controlled LeaveGroup request and commits offsets before shutting down.

3. Use static group membership: set `group.instance.id` on each consumer. With static membership, the group coordinator holds the consumer's position for `session.timeout.ms` after the consumer disconnects. If the consumer reconnects within that window, no rebalance occurs.

4. Add a startup grace period: increase `max.poll.interval.ms` temporarily during consumer startup to accommodate cache warming and database connection setup.
</details>

### Challenge 4: Multi-Tenant Event Platform

**Problem:**

Design a Kafka-based event platform that serves 50 internal teams. Each team produces 5-50 MB/s of event data. The platform must provide:

- Tenant isolation: one team's high traffic should not affect other teams.
- Per-team quotas: each team has a maximum produce rate.
- Per-team retention: each team can configure retention from 1 to 30 days.
- Shared infrastructure: the platform should use fewer than 20 brokers.

**Key constraints:**
- Maximum throughput: 50 teams x 50 MB/s = 2,500 MB/s total produce.
- Maximum retention: 30 days at peak throughput.
- Cluster must survive 2 broker failures without data loss.

**Questions:**

1. What is the required storage capacity?
2. How do you implement tenant isolation?
3. What quota enforcement mechanism do you use?
4. How many partitions per topic? How many topics?

<details>
<summary>**Model Answer**</summary>

**Storage capacity:**

Each team produces up to 50 MB/s. With RF=3 and snappy compression (2:1 ratio), the data per team per day is 50 MB/s x 86,400 x 3 / 2 = 6.48 TB per team per day. With 50 teams, that is 324 TB per day. At 30-day retention, that is 9,720 TB total storage.

This does not fit on 20 brokers with standard hardware. The solution is tiered storage: keep 3-7 days on broker-attached NVMe and move older data to object storage (S3, GCS).

- Hot tier (NVMe): 3 days = 324 x 3 = 972 TB. With 20 brokers, 48.6 TB per broker. Achievable with 6 x 8 TB NVMe drives per broker.
- Cold tier (S3): 27 days = 324 x 27 = 8,748 TB. S3 standard storage at $0.023/GB/month = $201,000 per month for all 50 teams.

**Tenant isolation:**

1. **Dedicated topics per team**: Each team has its own topics. No topic sharing between teams. This provides natural isolation вҖ” Kafka's per-topic configuration allows separate retention, partitioning, and quotas.

2. **Rack-aware clustering**: Assign each team's topic partitions across specific brokers. Use rack IDs to ensure team A's partitions are not on the same brokers as team B's partitions.

3. **Broker-level isolation**: For the top 5 high-traffic teams, dedicate specific brokers to their partitions. Use partition reassignment to pin partitions.

**Quota enforcement:**

Use Kafka's built-in quota mechanism:

```bash
# Set produce quota for team-a to 50 MB/s
kafka-configs.sh --bootstrap-server localhost:9092 \
  --alter --add-config 'producer_byte_rate=51200' \
  --entity-type users --entity-name team-a
```

Kafka's quota enforcement throttles producers that exceed their byte rate. Throttled producers receive throttling responses with a backoff time. Client libraries automatically respect the backoff.

Set quotas per client-id prefix (each team uses a standardized client-id prefix).

**Partition sizing:**

Assume each team has 3 topics (raw-events, processed-events, errors) with varying throughput. The total partition count across 50 teams:

- Per team: 3 topics x 24 partitions (8 MB/s per partition for 50 MB/s peak) = 72 partitions.
- Total: 50 x 72 = 3,600 partitions.
- Per broker: 3,600 / 20 = 180 partitions per broker. This is well within the recommended maximum of 4,000 per broker.

Each team configures retention per topic via topic-level configuration. The platform enforces a maximum of 30 days by overriding `log.retention.ms` at the broker level for the topic.
</details>

### Challenge 5: Change Data Capture Pipeline

**Problem:**

Design a change data capture (CDC) pipeline that streams database changes from PostgreSQL to downstream consumers. The source database handles 50,000 transactions per second. Changes must be delivered to consumers within 5 seconds of commit.

**Consumers include:**
- A search index (Elasticsearch) that needs full record state.
- A cache (Redis) that needs key changes only.
- A data warehouse (Snowflake) that needs all changes with 1-hour delay.

**Questions:**

1. What tool captures changes from PostgreSQL?
2. What Kafka topology serves all three consumers?
3. How do you handle schema changes?
4. What happens when the source database schema evolves?

<details>
<summary>**Model Answer**</summary>

**CDC tool:**

Debezium is the standard CDC connector for PostgreSQL. It runs as a Kafka Connect source connector. Debezium captures changes from PostgreSQL's write-ahead log (WAL) using logical replication slots. It produces one Kafka record per database row change.

Debezium provides:
- At-least-once delivery: each change is captured even if the connector restarts.
- Schema change events: DDL changes are captured separately.
- Snapshot: Initial load of existing data before streaming.

**Kafka topology:**

Use three topics:

1. `db-changes-full`: The raw CDC events from Debezium. Contains the full record state after each change (using PostgreSQL's REPLICA IDENTITY FULL or the default primary-key-based before/after images).

2. `db-changes-keys`: A processed topic with only the primary key and the changed column values. A lightweight Kafka Streams job reads from `db-changes-full`, and projects to key-only records.

3. `db-changes-hourly`: A compacted topic with the latest state per key, updated every hour. A Kafka Streams job with a 1-hour window reads from `db-changes-full`, aggregates the latest state per key, and writes to this compacted topic.

**Consumer-specific topology:**

- Elasticsearch consumer reads from `db-changes-full` directly. Uses a sink connector that indexes each record. Indexes within seconds of the change.

- Redis consumer reads from `db-changes-keys`. Uses a sink connector that sets the key in Redis. Minimal data transfer since only keys and changed columns are sent.

- Snowflake consumer reads from `db-changes-hourly`. A Kafka Connect S3 sink connector writes hourly batches to Parquet files in S3. Snowpipe or an external stage loads these into Snowflake on a scheduled basis.

**Schema changes:**

1. Debezium publishes schema change events to a separate `dbz-schema-changes` topic when DDL is executed on PostgreSQL.

2. The Schema Registry validates the new schema against the old schema. All consumers must accept the new schema within the compatibility constraints (backward compatible by default).

3. When a backward-incompatible schema change is required (for example, dropping a column), the team must:
   - Create a new topic version (for example, `db-changes-full-v2`).
   - Migrate consumers to the new topic.
   - Stop the old topic after migration.

**Failure handling:**

- If the source database WAL grows faster than Debezium can read (50,000 tps is high), increase the Debezium poll interval and fetch size. Monitor the WAL lag.
- If Debezium fails, it resumes from the last committed offset. PostgreSQL retains the WAL until the logical replication slot is consumed. Set `slot.max_lsn_lag` to prevent unbounded WAL growth.
- If the target is unavailable, Debezium's producer retries with backoff. The database WAL continues to grow. Monitor and alert on replication slot lag.
</details>

## Tier 3: Advanced Design Challenges

These challenges are for staff-plus level preparation. They require novel thinking and deep distributed systems knowledge.

### Challenge 6: Ordering Across Partitions

**Problem:**

You need to process bank account transactions with strict ordering per account. Users have millions of accounts. Transactions arrive at 500,000 per second. Each transaction for the same account must be processed in order.

Kafka provides ordering within a partition. If you partition by `account_id`, all transactions for the same account go to the same partition. But with millions of accounts, a single partition can be a hot spot. The top 1% of accounts generate 50% of the traffic вҖ” a single account might generate 50,000 transactions per second.

**Questions:**

1. How do you handle a hot partition caused by a single high-traffic account?
2. Can you maintain per-account ordering while using multiple partitions?
3. What trade-off do you make?

<details>
<summary>**Model Answer**</summary>

**The core tension:**

A single partition can handle approximately 10-50 MB/s вҖ” roughly 5,000-25,000 transactions per second at 2 KB each. The top account at 50,000 transactions per second exceeds this capacity. But you cannot split this account across multiple partitions without losing ordering.

**Solution 1: Shard within the partition (tiered partitioning)**

Create a secondary partition key within the account: `account_id` + `sub_shard` = `transaction_id % N`. Route transactions within the same account to a sub-shard based on the transaction ID modulo the number of sub-shards. Each sub-shard maps to a Kafka partition.

The trade-off: transactions for the same account with different transaction IDs are in different partitions and are not ordered relative to each other. But if the application only requires ordering for transactions that affect the same balance field вҖ” and most transactions are independent (debits and credits to different sub-ledgers) вҖ” then this is acceptable.

**Solution 2: Buffer and reorder with a stream processor**

Route all transactions for hot accounts to a single partition using an external queue (Redis Streams or a dedicated Kafka partition). A stream processor reads from this partition, buffers transactions per account, and applies them in order.

The trade-off: this adds latency (buffer delay). The buffer drains at a rate that depends on the downstream processing capacity. If the buffer fills faster than it drains, it grows unbounded.

**Solution 3: Accept eventual ordering with application-level sequencing**

Route transactions by `account_id` to one of N partitions using a two-level key: `hash(account_id) % N`. Each transaction carries a sequence number from the source system. Consumers read from all N partitions and sort by sequence number before processing.

The trade-off: consumers must maintain a buffer of up to N windows and sort before processing. This adds complexity to the consumer. The buffer must hold enough records to bridge the maximum timing skew across partitions (typically 100-500 ms in Kafka).

**Recommended approach for bank accounts:**

Use Solution 3 with the following design:

- Partition by `account_id` using a hash with 256 partitions.
- Each transaction includes a strictly incrementing sequence number from the source.
- The consumer reads from all partitions and maintains a per-account buffer. It processes records in sequence-number order.
- The buffer is bounded вҖ” it processes records up to the point where all partitions have delivered at least through sequence number `N`. This guarantees ordering through sequence `N`.

This design sacrifices simplicity for throughput. It is the pattern used by real-world payment processing systems that need both per-account ordering and high throughput.
