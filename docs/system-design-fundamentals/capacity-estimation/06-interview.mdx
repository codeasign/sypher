# Capacity Estimation in Interviews

## Why Interviewers Test Capacity Estimation

System design interviews at FAANG and similar companies always include capacity estimation. Interviewers are not looking for a precisely correct number. They are looking for evidence that:

- You know which numbers matter (DAU, QPS, storage, bandwidth)
- You can derive one number from another without guessing
- You handle orders of magnitude correctly (K, M, B, T)
- You sanity-check your results (does 1 billion QPS make sense?)
- You articulate assumptions clearly before calculating

A candidate who says "let me assume 100 million DAU, each user does 20 requests per day, that gives us roughly 23,000 QPS" demonstrates structured thinking. A candidate who says "we need 50 servers because that sounds right" does not.

## The Signals Interviewers Read

| Signal | Strong Candidate | Weak Candidate |
|--------|-----------------|----------------|
| Starting assumptions | States DAU, actions/user, peak multiplier | Launches into calculation without framework |
| Unit handling | Converts between GB/TB correctly, uses powers of two | Mixes up MB and GB by 1,000x |
| Order of magnitude | Knows 10M DAU -> roughly 2,000 QPS | Thinks 10M DAU = 10M QPS |
| Peak awareness | Calculates both average and peak | Only uses average |
| Sanity checks | "23,000 QPS seems reasonable for 10M DAU" | Accepts 1 billion QPS without blinking |
| Headroom | Adds 30% to 50% buffer | Provisions exactly to the number |
| Adaptability | Adjusts when interviewer changes a number | Recalculates from scratch or gets stuck |

## Level-Specific Expectations

### Entry-Level (L3/E3)
- Basic multiplication and division
- Knows what DAU and QPS mean
- Can estimate servers from QPS with guidance
- Gets units right (MB vs. GB)

**Bar:** Given DAU and actions per user, calculate QPS and server count within 2x of the interviewer's expectation.

### Mid-Level (L4/E4)
- Independent estimation with clear assumptions
- Handles read/write ratio and storage
- Accounts for peak-to-average ratio
- Adds headroom automatically
- Sanity-checks own numbers

**Bar:** Estimate a complete system (compute, storage, bandwidth) from scratch with no assistance.

### Senior (L5/E5)
- Drives the entire estimation conversation
- Handles tradeoffs (more cache vs. more database)
- Considers consistency impact on capacity
- Discusses cost implications
- Adapts estimates as requirements change

**Bar:** Produces credible estimates for any unfamiliar system and explains the reasoning behind each assumption.

### Staff+ (L6/E6)
- Estimation covers business metrics, not just technical
- Discusses growth planning, regional distribution
- Recommends architectural changes based on estimates
- Challenges interviewer's assumptions productively

**Bar:** Capacity estimate is a small part of a complete system design that includes cost, growth, and operational considerations.

## The Five-Minute Interview Framework

Use this framework every time. Interviewers recognize and reward structured approaches.

```
Minute 1: Clarify and scope
  "Let me confirm: 100 million DAU, text-only messaging, no media?
  What's the peak-to-average ratio we should plan for? 5x? 10x?"

Minute 2: Traffic estimation
  "100M DAU, 20 messages/day/user = 2B messages/day.
  95% reads, 5% writes.
  2B / 86,400 = 23,148 avg QPS.
  At 5x peak = 115,740 peak QPS.
  Reads: 110K, Writes: 5.8K."

Minute 3: Compute estimation
  "Each server handles 750 QPS dynamic.
  115,740 / 750 = 154 servers.
  + 30% headroom = 200 servers."

Minute 4: Storage estimation
  "Each message is 1 KB. 100M users store 10K messages each.
  100M x 10K x 1 KB = 1 PB.
  3x replication = 3 PB + 30% indexes = 4 PB.
  Daily growth: 5.8K writes x 86,400 x 1 KB = 500 GB/day."

Minute 5: Bandwidth and summary
  "Outbound reads: 110K peak QPS x 5 KB response = 4.4 Gbps.
  Inbound writes: 5.8K peak QPS x 1 KB = 46 Mbps.
  Summary: 200 servers, 4 PB storage, 4.4 Gbps bandwidth."
```

## Fifteen Essential Phrases

Use these phrases during interview estimation:

**Setting up:**
1. "Let me confirm my understanding: we are designing for (X) DAU with (Y) actions per user."
2. "I will assume a 90:10 read-to-write ratio unless you tell me otherwise."
3. "For peak traffic, I will use 5x the average. Does that seem reasonable?"

**Calculating:**
4. "Moving to rough mental math: 1 billion divided by 100,000 seconds gives us 10,000 QPS."
5. "That gives us (X) QPS, which translates to roughly (Y) servers at (Z) QPS each."
6. "For storage, each user stores (X) of data, so (N) users times (X) is (total)."
7. "With 3x replication and 30% overhead, that becomes (total)."

**Checking:**
8. "Let me sanity-check that number. (X) QPS for (Y) DAU seems consistent with a system like Twitter."
9. "Do these numbers align with your expectations so far?"
10. "That is roughly (X) servers. For context, a medium-sized social app runs on 100-500 servers."

**Pivoting:**
11. "If that assumption changes, the estimate shifts significantly. Could we revisit the (X) assumption?"
12. "For a different approach, if we introduce caching, our database QPS drops to (X)."
13. "If growth is 50% per year, we need to plan for (X) instead of (Y)."

**Concluding:**
14. "To summarize, I estimate (X) servers, (Y) storage, and (Z) bandwidth for this system."
15. "The biggest sensitivity is the peak multiplier. If peak is 10x instead of 5x, servers double to (X)."

## Eight Red Flags

These behaviors hurt your estimation signal:

**Red Flag 1: No assumptions stated.**
You say "the QPS is 50,000" without explaining how you got there. The interviewer cannot evaluate your reasoning.

**Red Flag 2: Magically precise numbers.**
You say "we need 47.382 servers." Round to the nearest whole server or bracket (e.g., "50-70 servers").

**Red Flag 3: Off by more than 10x.**
You estimate 10 million QPS for an app with 1 million DAU. That is 10 requests per user per second — a botnet, not a social app.

**Red Flag 4: Ignoring peak traffic.**
You design for average QPS only. The interviewer will ask about peak, and you will have to redo your work.

**Red Flag 5: Wrong units.**
You say "1000 MB" when you mean 1 GB. Or you confuse bits and bytes (8 Gbps vs. 1 GB/s). This is a basic competence signal.

**Red Flag 6: No headroom.**
You provision exactly 50 servers for a system handling 50 servers' worth of traffic. One server failure causes a cascade. Add 30% minimum.

**Red Flag 7: Refusing to estimate.**
You say "it depends" or "we would need benchmarks." The interview is testing your ability to make reasonable assumptions under uncertainty.

**Red Flag 8: Contradicting your own numbers.**
You estimate 100 servers, then later say 500. Or you estimate 1 GB/day storage and 10 PB total — inconsistent by a factor of 10,000.

## Five Practice Drills

### Drill 1: The Daily User QPS Conversion (2 minutes)

Memorize this: for every 1 million DAU doing 10 actions per day:
- 10M / 86,400 = 116 QPS (roughly 100 QPS)
- With 5x peak: 500 QPS
- Servers at 750 QPS each: 1 server

Practice: "10M DAU, 50 actions/day = ? QPS"
Answer: 10M x 50 / 86,400 = 5,787 QPS

### Drill 2: Storage Size Estimation (2 minutes)

Memorize these reference sizes:
- Text message: 1 KB
- Photo (compressed): 1 MB
- Video per minute: 500 MB (HD)
- User profile with metadata: 100 KB
- 1 hour of logs per server: 1 GB

Practice: "100M users, 200 photos each = ? storage"
Answer: 100M x 200 x 1 MB = 20,000 TB = 20 PB

### Drill 3: The Peak Multiplier Exercise (3 minutes)

For each system type, estimate a reasonable peak multiplier:
- Social media feed app: 3x to 5x
- E-commerce site: 10x to 50x
- Real-time chat: 2x to 3x
- Video streaming: 2x to 5x
- Stock trading: 100x during market open

Practice: "E-commerce site with 50M DAU during Black Friday. Normal peak: 5x. Black Friday: ?"
Answer: 50x to 100x

### Drill 4: End-to-End Estimation (10 minutes)

Pick a system and run the full framework:

```
System: URL shortener (like bit.ly)
DAU: 10 million
Actions: 0.1 shortens/day, 1 redirect/day
Message size: 100 bytes per entry, 5 KB per redirect page

1. Traffic: 10M x 1.1 actions / 86,400 = 127 QPS, 635 peak (5x)
2. Compute: 635 / 750 = 1 server, + headroom = 2 servers
3. Storage: 10M x 10 links = 100M entries x 100 bytes = 10 GB
4. Bandwidth: 635 peak QPS x 5 KB = 3 MB/s = 24 Mbps
5. Summary: 2 servers, 10 GB storage, 24 Mbps
```

### Drill 5: Sanity Check Calibration

Learn to spot wrong numbers immediately:

| Number | Verdict | Why |
|--------|---------|-----|
| 100M DAU generating 100M QPS | Wrong | That is 1 QPS per user 24/7 — a single user cannot click once per second all day |
| 1PB database on a single server | Wrong | A single server with 1PB would cost $50K+ and have terrible latency |
| 10 servers for YouTube-scale video | Wrong | YouTube needs 10,000+ servers minimum |
| 100% cache hit ratio | Wrong | No cache has 100% hit ratio — cold starts, new content |
| Zero bandwidth cost for internal services | Wrong | Cross-AZ traffic costs money in every cloud |

## The Estimation Conversation Flow

A strong interview answer is a conversation, not a monologue. Here is how a good candidate paces the interaction:

```
Interviewer: "Design a URL shortener. How many servers do you need?"

Weak candidate response:
  "We need 10 servers. Because a URL shortener is simple."

Strong candidate response:
  "Let me start by clarifying the scope. How many DAU are we designing for?
  And what is the expected scale — thousands, millions, or billions of URLs?"

Interviewer: "Assume 100 million DAU, and 10 million new URLs per month."

Strong candidate:
  "Great. Let me make a few assumptions:
   1. 100M DAU with 40% daily active rate = 40M DAU.
   2. Each user creates 0.01 short URLs per day (1 per 100 days) and views 0.1.
   3. Peak-to-average ratio of 5x for reads, 2x for writes.

   Let me calculate reads first.
   40M DAU x 0.1 views/day = 4M views/day.
   4M / 86,400 = 46 QPS average.
   At 5x peak: 230 QPS peak.

   Writes:
   10M new URLs / 30 days = 333K/day.
   333K / 86,400 = 4 QPS average.
   At 2x peak: 8 QPS peak.

   Total peak QPS: 238.
   Each server handles 750 dynamic QPS: so 1 server is enough.

   With 30% headroom and 1 failover: 2 to 3 servers total.

   Does that align with what you were thinking?"
```

The candidate stated assumptions, showed math, rounded reasonably, added headroom, and checked with the interviewer. This is the pattern to practice.

## Handling Curveballs

Interviewers often change parameters mid-discussion. Practice these adjustments.

**Curveball 1: "Double the user count."**

```
40M DAU → 80M DAU
Everything linear: QPS doubles from 238 to 476
Servers: 476 / 750 = 1 → still 1 server with 50% headroom
Storage: doubles. URL entries: 98M → 196M. Storage: 29 GB → 58 GB.
No infrastructure change needed — still fits on one server.
```

**Curveball 2: "Now each URL has analytics — 10 tracking events per view."**

```
Each view generates 10 additional write events.
Views: 46 QPS avg, 230 peak.
Analytics writes: 230 x 10 = 2,300 peak write QPS.
Total QPS: 230 + 2,300 = 2,530 peak.
Servers: 2,530 / 750 = 4 servers + 30% = 6 servers.
Analytics storage: 2,300 writes x 500 bytes each = 1.15 MB/s = 99 GB/day.
Storage now grows at 3 TB/month for analytics alone.
Big change — need a time-series database and possibly data retention policy.
```

**Curveball 3: "Users upload profile photos now."**

```
Each user uploads 1 photo = 500 KB.
100M users: 100M x 500 KB = 50 TB storage.
New daily uploads: 1% of users per day = 1M photos/day x 500 KB = 500 GB/day.
With 3x replication: 1.5 TB/day from photos.
Bandwidth: 1M x 500 KB per day = 500 GB/day = 46 Mbps extra upload bandwidth.
Now storage dominates — need object store, not just a database.
```

**How to handle curveballs:**

1. Acknowledge the change: "Interesting, that changes the storage profile significantly."
2. Identify which dimensions are affected: "This primarily affects storage and write QPS."
3. Recalculate only the affected numbers: "Let me recalculate storage first, then servers."
4. State the new result clearly: "With photos, storage jumps from 29 GB to 50 TB, and we need an object store."

## Question-Specific Estimation Strategies

Each system design question has specific estimation shortcuts.

### Design Twitter

**Key numbers to estimate:**
- Tweets per day: DAU x tweets per user per day (500M x 1 = 500M tweets/day)
- Timeline reads: DAU x timeline refreshes per day (500M x 30 = 15B reads/day)
- Fan-out writes: tweets/day x average followers (500M x 200 = 100B fan-out writes)

**Shortcut:** Twitter is a fan-out-heavy system. The write amplification (1 tweet = 200 fan-out writes) makes writes dominate even though the read-to-write ratio appears high.

```
Timeline reads: 15B / 86,400 = 173,611 QPS average, 520,833 peak (3x)
Tweet writes: 500M / 86,400 = 5,787 QPS
Fan-out writes: 100B / 86,400 = 1,157,407 fan-out writes/second

Writes dominate when fan-out is considered.
This is why Twitter uses a fan-on-read strategy for celebrities
and fan-out-on-write for regular users.
```

### Design YouTube

**Key numbers to estimate:**
- Hours watched per day: DAU x hours per user (2B x 1 = 2B hours/day)
- Concurrent streams: total hours / 24 hours (2B / 24 = 83M concurrent)
- Uploads per day: 500 hours/min x 60 x 24 = 720,000 hours/day
- Storage per day: 720,000 hours x 3 GB/hour (compressed) = 2.16 PB/day

**Shortcut:** YouTube has three independent scaling challenges: storage (uploads grow continuously), bandwidth (streaming dominates), and compute (transcoding). Estimate each separately.

### Design WhatsApp

**Key numbers to estimate:**
- Messages per day: DAU x messages/user (2B x 50 = 100B messages/day)
- Peak messages per second: 100B / 86,400 x 5x = 5.8M peak writes/second
- Storage per day: 100B x 1 KB = 100 TB/day
- Connection count: DAU x 1.5x timezone multiplier (100M concurrent connections)

**Shortcut:** WhatsApp is a connection-count problem, not a QPS problem. 100 million persistent WebSocket connections require careful connection management, not high QPS on any single server.

## Common Interview Questions Involving Capacity

**Design Twitter:**
Estimate tweets per day, timeline reads, storage for tweets and media, cache size for hot tweets.

**Design YouTube:**
Estimate video uploads per day, storage per minute of video, CDN bandwidth, transcoding compute.

**Design WhatsApp:**
Estimate messages per day, read/write ratio, storage for message history, server count for WebSocket connections.

**Design Uber:**
Estimate ride requests per day, real-time location update frequency, database read/write for ride matching.

**Design Instagram:**
Estimate photo uploads per day, feed generation QPS, storage for images, CDN bandwidth.

For each question, use the five-minute framework. Clarify scope, estimate traffic, compute servers, calculate storage, determine bandwidth. End with a summary and the single biggest sensitivity in your estimate.

## Common Mistakes by Experience Level

### Entry-Level Mistakes

**Mistake: Not knowing the latency numbers.**
You quote 10 million QPS for a single database server. The interviewer knows this is impossible. A single database handles 1,000 to 10,000 QPS, not 10 million.

**Fix:** Memorize the latency reference table. A server handles 500 to 1,000 dynamic QPS. A database handles 1,000 to 10,000 read QPS. A cache handles 100,000+ QPS.

**Mistake: Mixing up MB and GB by 1,000x.**
You say "each user stores 5 MB of data" but then say "100 million users need 500 MB total." That is 5 MB x 100M = 500,000,000 MB = 500 TB, not 500 MB.

**Fix:** Write the units explicitly in every calculation. "5 MB x 100M = 500 million MB = 500,000 GB = 500 TB."

**Mistake: Using the wrong divisor for QPS.**
You divide by 3,600 (seconds per hour) instead of 86,400 (seconds per day). You are off by a factor of 24.

**Fix:** Use 100,000 seconds for rough mental math. 1 billion / 100,000 = 10,000 QPS. Adjust to 86,400 if you need precision.

### Mid-Level Mistakes

**Mistake: Only estimating servers.**
You calculate 50 servers and stop. The interviewer asks about storage, bandwidth, and cache. You have to work through three more calculations under time pressure.

**Fix:** Always cover all five dimensions in order: traffic, compute, storage, bandwidth, cache. Do them in under 5 minutes.

**Mistake: Ignoring replication.**
You say "100 TB of data" and stop. The interviewer asks about replication. You forgot: 3x replication means 300 TB raw storage.

**Fix:** Always add "with 3x replication, that becomes X." State your replication assumption upfront.

**Mistake: Not adjusting for read vs. write.**
You estimate total QPS and divide by server capacity. But reads and writes have different resource costs. A write with index updates is 10x more expensive than a read.

**Fix:** Estimate reads and writes separately. Apply different per-server capacities for each.

### Senior-Level Mistakes

**Mistake: Underestimating operational overhead.**
You calculate 100 servers for the workload. But you forgot that each server needs 20% CPU for monitoring agents, log shipping, health checks, and OS processes. Effective capacity is 80% of raw capacity.

**Fix:** Apply a 20% to 30% overhead factor to all resource estimates. "At 80% usable capacity, 100 servers = 80 servers of effective compute."

**Mistake: Assuming linear scaling.**
You estimate that doubling traffic means doubling servers. But database connection pools, cache hit ratios, and locking behavior do not scale linearly. At 2x traffic, you may need 3x servers for the database tier.

**Fix:** Mention non-linearity: "Server count scales roughly linearly for stateless tiers. Database and cache tiers may need more due to connection overhead and cache miss amplification."

**Mistake: Forgetting about network topology.**
You estimate bandwidth as total QPS x response size. But cross-AZ traffic costs money and adds latency. If servers are in 3 availability zones, your internal bandwidth estimate needs to account for replication and request routing.

**Fix:** Add 10% to 20% for internal traffic (replication, health checks, intra-service calls).

## Mock Interview Transcript

Below is a full 10-minute interview transcript for the question "Design Twitter" with capacity estimation focus.

```
Interviewer: "Design Twitter. Start with capacity estimation for the tweet pipeline."

Candidate: "Let me clarify the scope. Should I assume current Twitter scale of roughly
500 million DAU, or a smaller scale?"

Interviewer: "Assume 500 million DAU."

Candidate: "Great. Let me estimate traffic first.

For a typical user, they read tweets constantly — say 50 timeline refreshes per day.
They write tweets less often — roughly 1 tweet per user per day on average.

Daily actions:
  Reads: 500M DAU x 50 refreshes = 25B timeline reads/day
  Writes: 500M DAU x 1 tweet = 500M tweets/day

QPS:
  Avg read QPS: 25B / 86,400 = 289,351
  Peak read QPS (3x): 868,054
  Avg write QPS: 500M / 86,400 = 5,787
  Peak write QPS (3x): 17,361

Now, fan-out. Each tweet goes to the followers' timelines.
Average follower count: 200.
Total fan-out writes: 500M x 200 = 100B fan-out writes/day
Fan-out QPS: 100B / 86,400 = 1,157,407 avg, 3.5M peak

So the system does roughly 3.5M peak fan-out writes + 868K reads ≈ 4.4M peak operations.

For compute: each server handles roughly 750 dynamic QPS.
4.4M / 750 = 5,867 servers.
Warm standby + headroom: 7,500 servers.

For storage:
Each tweet: 280 chars text + metadata = ~500 bytes.
But most tweets include media: photos (1 MB) or video (5 MB).
Assume 50% of tweets have media:
  250M text tweets/day x 500 bytes = 125 GB
  250M media tweets/day x 2 MB avg = 500 TB
  Total: ~500 TB/day

With 3x replication: 1.5 PB/day.
Yearly storage (365 days): 547.5 PB replicated.

For bandwidth:
Peak reads: 868K x 50 KB (timeline response) = 43.4 GB/s = 347 Gbps
Peak writes: 17,361 x 500 bytes = 8.7 MB/s — negligible
Peak fan-out: internal, not bandwidth-constrained

Summary:
  7,500 servers (multi-region)
  500 TB/day new storage
  350 Gbps peak outbound bandwidth
  Biggest constraint: fan-out write amplification at 200x

One optimization: celebrities with millions of followers should use
fan-on-read instead of fan-out-on-write to avoid massive fan-out costs."

Interviewer: "Good. That covers the key estimation points."
```

This transcript shows the ideal flow: state assumptions clearly, calculate sequentially, identify the dominant constraint, and suggest an optimization based on the estimation results.
