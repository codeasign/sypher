# Capacity Estimation in Architecture

## The Naive Approach

A team that skips capacity estimation typically guesses server counts. The conversation sounds like this:

"We need a photo-sharing service. Probably 10 million users. Let's spin up 10 servers. That should be enough."

This guess is wrong in one of two ways. If traffic is lower than expected, you waste money on idle servers. If traffic is higher, the system collapses. Both outcomes are expensive.

```
Naive guess: 10 servers → 10 x $200/month = $2,000/month

What the system actually needs:
  10 million users
  x 4 sessions/day
  x 30 actions/session
  = 1.2B actions/day
  = 13,889 QPS average
  = 55,556 peak QPS (4x)

  At 750 QPS per server:
  55,556 / 750 = 74 servers

  + 30% headroom = 96 servers

Error: the guess was off by 9.6x
```

## Production Approach

Production capacity planning is an ongoing process, not a one-time calculation. It combines traffic models, monitoring data, and auto-scaling.

### Traffic Modeling

Production teams build traffic models from three sources:

**Analytics data:** Real user behavior from production. You know DAU, session length, actions per session, and peak times because you measure them.

**Growth projections:** Product team forecasts for user growth, feature launches, and marketing campaigns. A new video feature might increase storage per user by 10x.

**Seasonal patterns:** Known traffic cycles. E-commerce teams model Black Friday, New Year promotions, and summer sales. Social media teams model election cycles and major sporting events.

```
January baseline: 10 million DAU, 12,000 avg QPS
March product launch: estimated 15 million DAU, 18,000 avg QPS
July summer season: 12 million DAU, 14,000 avg QPS
November holiday: 20 million DAU, 40,000 avg QPS

Plan capacity for November: 40,000 avg QPS → 160,000 peak QPS
```

### Monitoring and Feedback Loops

Production capacity systems use three signals to adjust resources:

**CPU utilization:** If servers run above 70% sustained, add more. If below 30%, scale down.

**Request latency:** P99 latency rising is a leading indicator of capacity shortage. If P99 goes from 50 ms to 200 ms, you are approaching saturation.

**Queue depth:** If request queues grow faster than they drain, capacity is insufficient. Queue depth is the earliest warning signal.

```
Monitoring Dashboard Values:
  Current CPU: 65% (across 120 servers)
  P99 latency: 85 ms (threshold: 200 ms)
  Queue depth: 1,200 (threshold: 5,000)
  Error rate: 0.02% (threshold: 1%)

Action: no scaling needed. Current capacity adequate.
Alert when any threshold within 70% of limit.
```

### Auto-Scaling Configuration

Auto-scaling turns capacity estimates into dynamic resource management. You configure minimum, maximum, and scaling policies.

```
Production auto-scaling policy for API tier:

  Minimum servers: 50 (base capacity for baseline traffic)
  Maximum servers: 500 (budget or infrastructure limit)
  
  Scale-up policy:
    CPU > 65% for 3 consecutive minutes
    Add 20% of current server count
    Cooldown: 3 minutes between scale-ups
  
  Scale-down policy:
    CPU under 30% for 10 consecutive minutes
    Remove 10% of current server count
    Cooldown: 10 minutes between scale-downs

Estimated monthly cost at 50 servers idle: 50 x $200 = $10,000
Estimated monthly cost at peak 300 servers: 300 x $200 = $60,000
(with 60% utilization over month, average 120 servers): $24,000
```

Auto-scaling is not a substitute for capacity planning. If your base estimate is 50 servers but traffic is 300 servers worth, auto-scaling just adds the correct amount — it does not reduce the total cost.

## Integration with Monitoring Systems

Capacity estimation connects to monitoring systems through thresholds and dashboards.

**Alert thresholds** are derived from your estimates:
- Server count drop below minimum: alert
- CPU above 70% sustained: alert
- P99 latency above 200 ms: alert
- Error rate above 1%: alert

**Dashboard panels** show capacity vs. utilization:
- Current server count vs. estimated needed
- Current QPS vs. estimated peak
- Current storage vs. projected capacity date
- Current bandwidth vs. available bandwidth

**Capacity forecasting** tools (e.g., AWS Compute Optimizer, GCP Recommender) use past 14 days of metrics to predict future needs. They recommend right-sizing instances and reserved capacity.

## Integration with Cost Management

Every capacity decision has a cost impact. Teams that integrate capacity estimation with cost management avoid budget surprises.

```
Monthly cost breakdown for a 40M DAU social app:

Compute (650 servers x $200):              $130,000
Database (7 nodes x $1,500):               $10,500
Cache (2 TB Redis x $5/GB):                $10,000
Storage (30 PB hot + 120 PB cold):         $25,000
CDN bandwidth (50 TB egress x $0.08/GB):   $4,000
Network (cross-AZ, inter-region):          $8,000
Monitoring and logging:                    $3,000
Load balancers and DNS:                    $2,000
Total monthly:                             $192,500
Per-user monthly cost:                     $0.005
```

Armed with this estimate, the team knows whether the unit economics work. If each user generates $0.10 in monthly revenue, a $0.005 infrastructure cost is 5% — healthy. If revenue is $0.01 per user, infrastructure at 50% of revenue is unsustainable.

## Multi-Region Estimation

Global systems need capacity in each region. You estimate per-region capacity based on user distribution.

```
User distribution:
  North America: 40% = 16 million DAU
  Europe: 30% = 12 million DAU
  Asia-Pacific: 25% = 10 million DAU
  Rest of World: 5% = 2 million DAU

Per-region server count (North America example):
  16 million DAU
  x 4 sessions/day
  x 20 actions/session
  = 1.28B actions/day
  = 14,815 avg QPS
  = 59,260 peak QPS
  = 79 servers (at 750 QPS each)
  + 30% headroom = 103 servers
```

## Worked Example: 100M DAU Social Media App

This is a full capacity estimation for a hypothetical social media platform with 100 million daily active users.

### Traffic Estimate

```
100 million DAU
x 5 sessions per day
x 25 actions per session (scrolling, liking, posting, messaging)
= 12.5 billion actions per day

12.5B / 86,400 = 144,675 avg QPS

Read/write ratio: 95% reads, 5% writes
Read QPS: 137,441
Write QPS: 7,234

Peak multiplier: 3x
Peak read QPS: 412,323
Peak write QPS: 21,702
Peak total QPS: 434,025
```

### Server Estimate

```
Per-server capacity:
  Dynamic API requests: 750 QPS per server
  Concurrent connections: 5,000 per server

Server count from QPS:
  434,025 / 750 = 579 servers

Concurrent users:
  100M DAU, 20 min average session, 16 waking hours
  20 min = 1,200 seconds
  1,200 / 57,600 = 2.08% active at once
  100M x 2.08% = 2.08M concurrent users
  At 5,000 connections/server: 416 servers (QPS constraint dominates)

Total: 579 servers + 30% headroom = 753 servers
```

### Storage Estimate

```
Per user:
  Profile data: 10 KB
  Photos (200 photos x 1 MB each): 200 MB
  Messages (10K messages x 1 KB): 10 MB
  Feed cache data: 5 MB
  Total: 215 MB per user

Total storage:
  100M x 215 MB = 21.5 PB

3x replication: 64.5 PB
Index and overhead (30%): 84 PB

Daily writes generate:
  7,234 write QPS x 86,400 = 625M writes/day
  Average write size: 3 KB
  625M x 3 KB = 1.8 TB/day
  x 365 = 657 TB/year (replicated: 2 PB/year)

Photo uploads:
  Assume 0.5% of actions generate a photo
  625M x 0.5% = 3.1M photos/day
  x 1 MB = 3.1 TB/day
  x 365 = 1.1 PB/year
  x 3 replication = 3.3 PB/year

Storage growth: ~5.3 PB/year (replicated)
```

### Bandwidth Estimate

```
Peak read bandwidth:
  412,323 read QPS
  x 30 KB average response
  = 12.4 GB/s = 99 Gbps

Peak write bandwidth:
  21,702 write QPS
  x 3 KB average request
  = 65 MB/s = 0.5 Gbps

Media upload bandwidth:
  Assume 0.5% of peak QPS are uploads
  434,025 x 0.5% = 2,170 uploads/sec
  x 1 MB each = 2.17 GB/s = 17.4 Gbps

Total outbound bandwidth: 99 Gbps
Total inbound bandwidth: 17.9 Gbps
```

### Cost Estimate

```
Compute: 753 servers x $250/month = $188,250
Database: 30 nodes x $2,000/month = $60,000
Cache: 10 TB Redis x $5/GB = $51,200
Storage (hot): 30 PB x $50/TB/month = $1,500,000
Storage (cold): 60 PB x $10/TB/month = $600,000
CDN bandwidth: 99 Gbps sustained = ~30 PB/month transfer
  30 PB x $0.08/GB = $2,400,000
Network internal: $30,000
Load balancers, DNS, misc: $15,000

Total monthly: ~$4.84 million
Per-user monthly cost: ~$0.048 (5 cents per user)
```

## CPU vs. Memory vs. IO Bound Systems

Different workloads hit different resource limits first. Your capacity estimate must identify which resource is the bottleneck.

### CPU-Bound Systems

The bottleneck is compute capacity. Example: video transcoding, real-time ML inference, cryptographic operations.

```
Transcoding one minute of video: 120 seconds of CPU time per resolution
For 100 minutes/hour upload with 3 resolutions:
  100 x 3 x 120 = 36,000 CPU-seconds per hour
  = 36,000 / 3,600 = 10 CPU cores continuously busy

If each server has 16 cores:
  10 / 16 = 1 server for processing
But with peak (4x upload during evening): 4 servers needed

Memory per server: 2 GB per transcode pipeline
  4 concurrent pipelines per server = 8 GB RAM
CPU is the constraint, not memory. Add servers until CPU < 70%.
```

### Memory-Bound Systems

The bottleneck is RAM capacity. Example: in-memory caches, large graph databases, real-time analytics.

```
System: Real-time leaderboard for 10M concurrent gamers
Data: 10M user profiles x 1 KB each = 10 GB
Plus sorted sets for scores: 10M entries x 100 bytes = 1 GB
Plus session state: 10M x 512 bytes = 5 GB
Plus working buffers: 4 GB

Total RAM needed: 20 GB RAM for one Redis instance
With replication (2 replicas): 40 GB
With sharding across 8 nodes: 5 GB per node

Memory determines server count. Each server needs at least
32 GB to handle a shard + OS overhead + peak spikes.
CPU utilization may be under 10% — but you add servers
anyway because you need the RAM.
```

### IO-Bound Systems

The bottleneck is disk throughput. Example: write-heavy logging, large batch processing, data warehousing.

```
System: Event logging pipeline ingesting 1 million events/second
Each event: 1 KB
Total write throughput: 1 GB/s

Single SSD: 500 MB/s sequential write
Drives needed for throughput: 1 GB/s / 500 MB/s = 2 drives

But consider IOPS for random reads (queries on logs):
  10% of events get queried within 24 hours
  100K query QPS x 1 IO = 100K IOPS

Single SSD: 10,000 random IOPS
Drives needed for IOPS: 100K / 10K = 10 drives

IOPS constraint dominates. You need 10 drives even though
throughput only requires 2. This is an IO-bound system.
```

### Multi-Bound Detection

Real systems often hit multiple limits simultaneously. A web server might be CPU-bound during API processing and memory-bound during WebSocket handling.

## Deployment — Sizing Staging and Testing Environments

Staging environments rarely need production-scale capacity, but they must be representative enough to catch capacity-related bugs.

### Guidelines for Staging Sizing

| Environment | Size Relative to Production | Purpose |
|------------|---------------------------|---------|
| Dev | 1% to 5% | Feature development |
| Staging | 10% to 25% | Integration testing, load testing |
| Pre-prod | 50% to 100% | Final validation before deploy |
| Canary | 5% to 10% | Live traffic testing |

### Staging Capacity Configuration

```
Production: 500 servers, 20 TB database, 2 TB cache
Staging: 50 servers (10%), 2 TB database (10%), 200 GB cache (10%)

Staging still needs realistic traffic for load testing:
  Use recorded production traffic at 10% scale
  Or synthetic load generators targeting 50% of staging capacity
  Monitor for performance regressions before deploying

Key metric: staging must reproduce production bottlenecks at scale.
If a bottleneck only appears at 400+ servers, staging needs enough
servers to trigger it — otherwise the bug hits production first.
```

### Synthetic Load Generation

You validate capacity estimates by generating synthetic traffic at your estimated peak load. Tools like Locust, k6, or Gatling simulate user behavior at scale.

```
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '5m', target: 10000 },   // ramp to 10K QPS
    { duration: '10m', target: 50000 },  // ramp to 50K QPS
    { duration: '10m', target: 100000 }, // ramp to 100K QPS
    { duration: '5m', target: 100000 },  // hold at 100K QPS
    { duration: '5m', target: 0 },       // ramp down
  ],
  thresholds: {
    http_req_duration: ['p(99)under 200'],    // 99% under 200ms
    http_req_failed: ['rateunder 0.01'],      // under 1% errors
  },
};

export default function () {
  const res = http.get('https://api.example.com/feed');
  check(res, { 'status was 200': (r) => r.status === 200 });
  sleep(0.1); // simulate think time
}
```

Run this load test against staging. If P99 latency stays under 200 ms at 100,000 QPS, your capacity estimates are validated. If latency spikes at 60,000 QPS, your bottleneck is lower than expected — revise estimates.

## Data-Driven Capacity Planning Process

Production systems evolve their capacity planning through maturity levels.

### Level 1: Reactive (Startup)

No capacity planning. When the system goes down, you add more servers.

```
Traffic grows → server runs out of connections → outage → add 1 server → repeat
Cost: unpredictable. Outages every 2 to 4 weeks.
Utilization: wild swings from 10% to 120%.
```

### Level 2: Proactive (Growth Stage)

Basic capacity monitoring with alert thresholds. You add servers before they fail.

```
CPU at 70% → alert → add 2 servers within 24 hours
Weekly capacity review: check trends, add servers
Cost: moderate. Fewer outages, but some over-provisioning.
Utilization: steady at 40% to 60%.
```

### Level 3: Predictive (Scale Stage)

Traffic models predict growth weeks and months ahead. Auto-scaling handles daily fluctuations.

```
Traffic model predicts 30% growth next month → order servers 2 weeks ahead
Daily: auto-scaling handles +/- 20% fluctuations
Weekly: review model accuracy, adjust parameters
Monthly: full capacity review with updated traffic data
Cost: optimized. Over-provisioning under 20%.
Utilization: steady at 60% to 75%.
```

### Level 4: Autonomous (Hyper-Scale)

Machine learning models predict capacity needs from leading indicators (signups, feature adoption, marketing campaigns). Capacity adjusts automatically.

```
ML model predicts traffic 30 days ahead with 95% accuracy
Capacity changes are automated — servers provisioned and de-provisioned without human intervention
Cost: near-optimal. Over-provisioning under 10%.
Utilization: steady at 70% to 80%.
```

### Cloud-Specific Capacity Considerations

Each cloud provider has unique constraints that affect capacity planning.

**AWS-specific:**
- Instance types matter for your estimate. A c5.large (2 CPU, 4 GB) handles far less than a c5.24xlarge (96 CPU, 192 GB). Your server count estimate only makes sense with a specific instance type in mind.
- Reserved instances (1-year or 3-year commitment) save 30% to 60% compared to on-demand. Your capacity estimate determines how many RIs to buy.
- Spot instances can reduce compute cost by 60% to 90% but can be terminated with 2 minutes notice. Use them for batch processing, not for real-time serving.

**GCP-specific:**
- Preemptible VMs (equivalent to spot) have a 24-hour maximum lifetime. Batch jobs must be checkpointable.
- Committed use discounts require a 1-year or 3-year commitment per region. If your growth estimate is wrong, you overpay.
- GKE cluster auto-scaling adds nodes in 2 to 3 minutes — faster than AWS ASG in some configurations.

**Azure-specific:**
- Azure Reservations are region-specific. Moving to a new region means forfeiting the discount.
- Azure Spot VMs have no maximum lifetime — better for long-running batch jobs.
- Azure VM series (e.g., Ev5, Dv5) have different CPU-to-memory ratios. Match your workload profile to the right series.

**Multi-cloud capacity note:**
Running across multiple clouds means provisioning 2x the capacity (peak for each cloud + headroom). This increases cost by 30% to 50% compared to a single-cloud deployment. The tradeoff is vendor lock-in reduction and better geographic coverage.
