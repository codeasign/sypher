## The Estimation Lifecycle — 10 Steps

Each step builds on the previous one. You start with user numbers and end with a server count and monthly bill.

### Step 1: Estimate DAU

DAU is your starting point. Get this right or the rest is meaningless.

```
Addressable market: 500 million smartphone users in Region A
Expected adoption: 20% in year 2
Expected DAU rate: 40% of registered users

500 million x 20% = 100 million registered
100 million x 40% = 40 million DAU
```

Use 40 million DAU as your baseline.

### Step 2: Estimate Sessions Per User Per Day

How many times does a user open the app? A social media app gets 3 to 8 sessions per day. A utility app gets 1 to 2 sessions per day.

```
40 million DAU
x 4 sessions per user per day
= 160 million sessions per day
```

### Step 3: Estimate Actions Per Session

Each session generates multiple actions. Scroll through news feed: 15 posts viewed. Send messages: 2 messages. Like posts: 3 likes. Create a post: 0.2 posts (1 post every 5 sessions).

```
160 million sessions
x 20 actions per session (feed view, like, message, post)
= 3.2 billion actions per day
```

### Step 4: Calculate Average QPS

Divide total daily actions by seconds in a day. Use 100,000 seconds for mental math (close to 86,400).

```
3.2 billion actions / 86,400 seconds
= 37,037 average QPS

Rough mental math: 3.2B / 100K = 32,000 QPS
```

Call it 37,000 QPS average.

### Step 5: Determine Read-to-Write Ratio

Characterize the workload. A social media feed is roughly 90% reads, 10% writes. A logging system might be 10% reads, 90% writes.

```
37,000 average QPS
90% reads = 33,300 read QPS
10% writes = 3,700 write QPS
```

Reads and writes have different resource costs. A read typically consumes less CPU and disk I/O than a write (which updates indexes, triggers replicas).

### Step 6: Apply Peak Multiplier

Average QPS is misleading. You must design for peak traffic. For social media, peak is typically 3x to 5x average.

```
37,000 average QPS
x 4 peak multiplier
= 148,000 peak QPS
```

### Step 7: Estimate Concurrent Users

Concurrent users drive connection pool sizing and memory for connection overhead.

Assume average session lasts 15 minutes. Users are active throughout the day. 16 waking hours = 57,600 seconds.

```
Each user spends 15 minutes = 900 seconds active per day
900 / 57,600 = 1.56% of users active at any moment (on average)

40 million DAU x 1.56% = 624,000 concurrent users at average

At peak, concurrent users are roughly 4x higher:
624,000 x 4 = 2.5 million concurrent users at peak
```

Each concurrent connection consumes roughly 10 KB to 100 KB of memory (TCP buffer + application state). At 50 KB per connection:

```
2.5 million concurrent users
x 50 KB per connection
= 125 GB of connection memory across all servers
```

### Step 8: Calculate Server Count

Servers handle both CPU work and connection overhead. A single server handles roughly 5,000 concurrent connections and 500 to 1,000 QPS for dynamic requests.

```
From QPS:
148,000 peak QPS / 750 QPS per server = 197 servers

From concurrent connections:
2.5 million concurrent / 5,000 connections per server = 500 servers

The connection constraint dominates.
Use 500 servers as the baseline count.

Add 30% headroom:
500 x 1.3 = 650 servers
```

### Step 9: Calculate Storage

Storage estimation has three layers: data per user, data generated per day, and long-term retention.

**User data (persistent):**
```
40 million users
x 500 MB per user (profile, photos, messages)
= 20,000 TB = 20 PB

3x replication for durability
= 60 PB raw storage

2x for indexes and overhead
= 120 PB total
```

**Daily data generation:**
```
3.7 million writes per day (from 37,000 write QPS)
x 2 KB per write average (message + metadata)
= 7.4 GB per day

x 365 days retention
= 2.7 TB per year

Replicated: 8.1 TB per year
```

**Media storage:**
```
Assume 1% of actions generate media (photos, videos)
37,000 QPS x 1% = 370 media uploads per second
370 x 86,400 = 32 million media items per day

32 million x 5 MB average (compressed photo)
= 160 TB per day

x 365 days = 58.4 PB per year (unreplicated)
x 3 replication = 175 PB per year
```

Clearly, media storage dominates. You need tiered storage: hot storage for recent 30 days (160 TB x 30 = 4.8 PB), cold storage for everything older.

### Step 10: Calculate Bandwidth

Bandwidth depends on response sizes and QPS.

```
Read requests: 33,300 avg read QPS
x 50 KB average response (feed data, profile)
= 1.67 GB/s = 13.3 Gbps

Write requests: 3,700 avg write QPS
x 2 KB average request size
= 7.4 MB/s = 59 Mbps

Media upload: 370 uploads/s
x 5 MB average
= 1.85 GB/s = 14.8 Gbps

Total inbound: 14.8 Gbps
Total outbound: 13.3 Gbps
```

At peak (4x): 53 Gbps outbound, 59 Gbps inbound. You need multiple 40 Gbps or 100 Gbps network links.

## How Consistency Affects Capacity

Consistency guarantees directly multiply your infrastructure costs. Stronger consistency means more data copies, more round trips, and higher latency.

**Eventual consistency:**
- Single write to primary
- Async replication to replicas
- Read from any replica (stale data possible)
- Cost: 1x write, 1x read

**Strong consistency (quorum-based):**
- Write to N/2 + 1 nodes
- Read from N/2 + 1 nodes
- Cost: 2x write (with 3 replicas), 2x read

**Synchronous replication:**
- Write to primary, wait for all N replicas to confirm
- Cost: Nx write (3x for 3 replicas), 1x read

**Distributed consensus (Paxos/Raft):**
- Log replication + state machine apply
- Cost: roughly 3x to 5x write amplification

**Practical impact:**

Your write QPS of 3,700 with quorum consistency requires:
- 3,700 writes to coordinator
- Plus 3,700 x 2 replica writes = 7,400 internal writes
- Total database load: 11,100 writes per second (internal)

If your database handles 5,000 writes per second per node, you need:
- 11,100 / 5,000 = 3 database nodes just for writes
- Plus read replicas for 33,300 read QPS at 10,000 reads/node = 4 read replicas
- Total: 7 database nodes

With eventual consistency, same workload:
- 3,700 writes to primary
- Async replication: negligible load on primary
- 1 write node sufficient
- 4 read replicas (or even 3 with caching)
- Total: 5 database nodes

Consistency costs 2 extra database nodes for this example.

## Configuration — Safety Margins, Headroom, and Growth

### Safety Margins

Never provision to the exact number. Systems have unpredictable behavior at scale.

| Dimension | Recommended Headroom |
|-----------|---------------------|
| CPU utilization | Max 70% sustained |
| Memory utilization | Max 75% sustained |
| Disk utilization | Max 60% sustained |
| Network utilization | Max 50% sustained |
| Connection pool | Max 70% of max connections |

### Growth Allowance

Build for growth, not just current load.

```
Quarter 1: 100 servers
Quarter 2: 130 servers (30% growth)
Quarter 3: 169 servers
Quarter 4: 220 servers

If you provision 100 servers in Q1, you will need 220 by Q4.
Provision 220 from day one if lead times are long.
Provision 130 + auto-scaling if the cloud allows rapid provisioning.
```

### Tiered Storage Strategy

Not all data needs the same performance tier. Design storage tiers to match access patterns.

| Tier | Access Pattern | Cost | Example |
|------|---------------|------|---------|
| L1: Cache (Redis) | Millisecond access, seconds TTL | $50/GB/month | Hot feed data |
| L2: SSD Database | Sub-10ms access, hours to days | $5/GB/month | Active user data |
| L3: HDD Database | 10-50ms access, weeks | $0.5/GB/month | Older content |
| L4: Object Store | Seconds access, years | $0.02/GB/month | Archived content |

**Example cost comparison for 100 TB of user data:**

```
All on SSD: 100 TB x $5/GB/month = $512,000/month
Tiered (10% SSD, 30% HDD, 60% object store):
  10 TB x $5 = $51,200
  30 TB x $0.5 = $15,360
  60 TB x $0.02 = $1,200
  Total = $67,760/month

Savings: $444,240/month (87% reduction)
```

## Advanced Estimation: Handling Burst Traffic

Some workloads have extreme burst characteristics that standard peak multiples do not capture. You need specialized approaches for these.

### Flash Sale Burst

A flash sale (e.g., 1,000 items at 90% off) generates traffic spikes that dwarf normal patterns.

```
Normal traffic: 10,000 QPS
Flash sale start: 500,000 QPS in 5 seconds (50x spike)
Duration: 2 minutes of intense traffic, then back to normal

This is not a 3x or 5x peak — it is 50x.
Standard headroom of 30% would fail instantly.
```

**Strategy:**

```
Pre-warm servers: 10 minutes before sale, scale to 10x normal
Queue-based admission: requests enter a queue, processed at max system rate
Throttling: reject requests beyond a threshold with a "try again" response
Read-only mode: disable writes that are not related to the sale
Post-sale cooldown: scale back down within 5 minutes
```

### Viral Content Burst

A post goes viral and traffic spikes 20x within 30 minutes.

```
Normal: 50,000 QPS
Viral peak: 1,000,000 QPS (20x)
Growth rate: doubles every 5 minutes

Auto-scaling cannot keep up if it takes 10 minutes to provision a server.
By the time the first server is ready, traffic has quadrupled.
```

**Strategy:**

```
Over-provision baseline: run at 40% utilization normally to absorb 2.5x bursts
Reserved capacity pool: 20% extra servers in warm standby
Instant cache expansion: increase cache TTL, reduce origin load
Static content serving: serve cached versions of viral content from CDN
Feature degradation: disable comments, recommendations during peak
```

## Database Capacity Estimation in Depth

Databases require their own estimation model separate from application servers.

### Connection Pool Sizing

Each database connection consumes memory and a file descriptor. A PostgreSQL connection uses roughly 5 MB to 10 MB of memory just sitting idle.

```
2.5 million concurrent users
x 10% hitting the database at once (due to caching)
= 250,000 database connections needed

Each connection: 10 MB
250,000 x 10 MB = 2.5 TB of RAM just for connections

You cannot run 250,000 connections on one database.
You need connection pooling (PgBouncer, ProxySQL):
  Pool reduces active connections from 250K to 5K
  5,000 x 10 MB = 50 GB — manageable on one database node

PGBouncer servers needed:
  250,000 / 10,000 connections per pooler = 25 pooler servers
```

### IOPS Estimation

Database performance is often IOPS-bound, not CPU-bound.

```
Disk type: SSD with 10,000 random IOPS per volume
Each database read: 1 IO for small row, 3-5 IOs for larger row
Each database write: 3 IOs (log write, data page, index page)

Read QPS to DB: 33,300 (after cache)
  At 2 IOPS per read: 66,600 IOPS needed

Write QPS to DB: 3,700
  At 3 IOPS per write: 11,100 IOPS needed

Total IOPS: 77,700
SSD volumes at 10,000 IOPS each: 8 volumes
Or use a single high-IOPS SSD (e.g., AWS io2 at 256K IOPS)
```

## Cache Sizing Methodology

Cache sizing requires understanding the working set — the subset of data accessed frequently.

### Working Set Estimation

```
Total data: 100 TB (all user data)
Working set: 20% of data accessed 80% of the time (Pareto principle)
Hot data: 5% of data accessed 50% of the time

To cache the hot data: 5% x 100 TB = 5 TB cache
To cache the working set: 20% x 100 TB = 20 TB cache

Cache TTL determines how much of the working set fits:
  5 TB cache at 50% hit ratio (too small — 50% of requests hit DB)
  10 TB cache at 80% hit ratio
  20 TB cache at 95% hit ratio

Cache cost vs. DB savings:
  20 TB cache x $5/GB/month = $102,400/month cache cost
  Without cache: 148,000 peak QPS to DB = ~30 DB nodes = $60,000
  With cache (95%): 7,400 peak QPS to DB = ~3 DB nodes = $6,000
  Savings: $54,000/month in DB cost, minus $102,400 cache = net -$48,400

  In this case, the cache is more expensive than the DB it replaces.
  Use a smaller cache: 5 TB at 50% hit ratio
  Cache cost: $25,600/month
  DB load: 74,000 QPS (50% of peak) = ~15 DB nodes = $30,000
  Without cache: 30 DB nodes = $60,000
  Savings: $30,000 - $25,600 = $4,400/month net positive
```

### Cache Hierarchy

Multi-tier caching reduces cost for the same hit ratio.

```
L1 cache (in-memory, application-local):
  Size: 1 GB per server
  Hit ratio: 30%
  Cost: $0 (included in server RAM)
  Latency: under 1 ms

L2 cache (shared Redis/Memcached):
  Size: 5 TB cluster
  Hit ratio: 60% on remaining 70% = 42% additional
  Cumulative hit ratio: 72%
  Cost: $25,600/month

L3 cache (CDN for static/media):
  Size: unlimited (CDN edge)
  Hit ratio: 90% on remaining 28% = 25% additional
  Cumulative hit ratio: 97%
  Cost: CDN bandwidth included in egress pricing

Total cache cost: $25,600/month
Total effective hit ratio: 97%
DB load reduction: from 148,000 QPS to 4,440 QPS (97% reduction)
```
