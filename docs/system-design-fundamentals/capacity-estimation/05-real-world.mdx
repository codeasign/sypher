# Capacity Estimation in the Real World

## Facebook — News Feed Capacity Modeling

Facebook's News Feed is one of the most heavily optimized systems in the world. Every time you open the app, an algorithm ranks thousands of potential posts and selects roughly 15 for your feed. The capacity math behind this is staggering.

**The scale in 2024:**
- 2+ billion DAU
- Each user sees roughly 15 posts per session
- Users open the app 8 times per day
- Each feed request evaluates roughly 3,000 candidate posts
- Feed requests per day: 2 billion x 8 = 16 billion

**The estimation problem:**

Facebook does not just serve 16 billion requests. Each request triggers ranking computation — a composite of neural network inference, feature extraction, and candidate retrieval. A single feed request might require 500 to 2,000 model evaluations.

```
16 billion requests
x 1,000 model evaluations per request
= 16 trillion model evaluations per day

If one GPU handles 100,000 evaluations per second:
16T / (100,000 x 86,400) = 1,852 GPU-seconds per day
At peak (3x): 5,556 GPU-seconds per second
GPU fleet: roughly 5,000 to 10,000 GPUs just for feed ranking
```

**What Facebook actually does:**

Facebook does not run this full computation for every request. They pre-compute feed candidates using a tiered system:

1. **Pre-compute layer:** For high-affinity connections, pre-generate candidate sets offline. This handles 80% of feed requests at a fraction of the cost.
2. **Light ranking:** For mid-affinity, use a smaller model with 50 features instead of 5,000.
3. **Full ranking:** Only for the top candidates that survive pruning.

The lesson: capacity estimation exposes which operations are too expensive. The numbers motivate architectural changes — not just bigger servers.

## Twitter — Estimating for Super Bowl Traffic

Twitter has a well-documented pattern of traffic surges during live events. The Super Bowl is the most extreme: tweets per minute spike from 6,000 average to over 400,000 during key moments.

**The estimation problem:**

Twitter must provision capacity for events that last 30 minutes, happen a few times a year, and generate 70x normal traffic. Provisioning for Super Bowl peak all year would waste 95% of capacity.

```
Normal traffic: 6,000 TPM (tweets per minute)
Super Bowl peak: 400,000+ TPM (record: 2013 Super Bowl)

Normal write QPS: 6,000 / 60 = 100 QPS
Super Bowl write QPS: 400,000 / 60 = 6,667 QPS

Read traffic (timeline reads):
Normal: 10,000 QPS
Super Bowl: 700,000 QPS

Server count for normal: 50 write + 200 read = 250 servers
Server count for Super Bowl: 350 write + 3,500 read = 3,850 servers
```

**How Twitter solved it:**

Twitter uses a combination of three strategies:

1. **Pre-provisioned capacity reserved for events.** AWS and Google Cloud have reserved instances that are cheaper than on-demand but available on short notice. Twitter reserves this capacity in advance, paying a retainer rather than full price.

2. **Feature degradation.** During peak events, non-critical features turn off. Favorite counts update asynchronously. Search indexing pauses. Timeline refresh rate decreases from 15 seconds to 60 seconds. This effectively doubles server capacity without adding servers.

3. **Capacity testing with historical replay.** Twitter replays Super Bowl traffic patterns against their staging environment to find bottlenecks before game day. They discovered that their database write path saturated at 5,000 TPM on the first test, forcing a rewrite of the fan-out service.

**The lesson:** For event-driven capacity, do not provision for year-round peak. Provision for baseline plus critical event support. Use feature degradation as a safety valve.

## Netflix — Chaos Plus Capacity

Netflix is famous for Chaos Engineering — the practice of intentionally breaking systems in production to find weaknesses. But chaos is only one side of the coin. Netflix also invests heavily in capacity estimation.

**The scale:**
- 260 million+ subscribers
- 2 billion+ streaming hours per month
- Peak streaming: Friday and Saturday evenings, 8 PM to 11 PM
- Peak-to-average ratio: roughly 4x

**Capacity through chaos:**

Netflix uses Chaos Monkey to randomly terminate production instances. This forces the system to handle capacity loss gracefully. But the system must start with enough capacity to absorb the loss.

```
If you have 100 servers and Chaos Monkey kills 5:
95 remaining must handle 100% of traffic.
So each server must run at 95% utilization at most.
→ Run at 75% to have headroom for monkey casualties.

If Netflix runs at 75% average utilization:
Storage for 3 replicas of all content
CDN edge nodes at 60% capacity for streaming surge
Transcoding pipeline at 50% for batch processing backlog
```

**Content delivery estimation:**

Netflix pre-positions content on CDN edge nodes during off-peak hours. The capacity estimation determines how much content can be moved in the available window.

```
Each edge node: 100 TB storage
Content library: 20,000 titles
Average title: 5 GB (4K stream)
Total library: 100,000 GB = 100 TB

One edge node holds the entire library.
But during peak, each node serves 10,000 concurrent streams.
10,000 x 15 Mbps (4K) = 150 Gbps outbound
Edge node network: 100 Gbps → bottleneck.

Netflix limits concurrent streams per edge node or upgrades to 400 Gbps.
Both decisions come from capacity math.
```

**The lesson:** Capacity estimation and chaos engineering work together. Chaos finds weaknesses at current capacity. Estimation determines what capacity to inject.

## Google — Borg Resource Estimation

Google's Borg system (predecessor to Kubernetes) manages millions of containers across Google's global data centers. Every job submitted to Borg includes a resource estimate — CPU, memory, disk, and network. These estimates determine scheduling, bin packing, and overall data center utilization.

**The problem:**

Google uses roughly 60% to 70% of available data center capacity on average. The remaining 30% to 40% is headroom for traffic spikes, machine failures, and maintenance. Every percentage point improvement in utilization saves millions of dollars.

**Borg's estimation approach:**

1. **Users specify resource requests** in their job configuration (e.g., "4 CPU cores, 16 GB RAM, 100 GB disk"). These are upper bounds — the job gets killed if it exceeds them.

2. **Borg tracks actual usage** versus requested usage. Most jobs use far less CPU than requested. A batch data pipeline might request 8 cores but average 1.2 cores.

3. **Capacity planning tools** analyze historical usage patterns to recommend right-sized requests. If a job consistently uses 1.2 cores but requests 8, Borg suggests reducing the request to 2 cores.

4. **Overcommitment** — Borg schedules more jobs than the hardware can theoretically handle, betting that actual usage will not spike on all jobs simultaneously. If too many jobs spike at once, Borg preempts low-priority jobs.

**The math:**

```
Data center capacity: 100,000 CPU cores
Sum of all job requests: 150,000 CPU cores (1.5x overcommit)
Sum of actual average usage: 50,000 CPU cores
Actual peak usage: 80,000 CPU cores
Headroom: 20,000 CPU cores

Without overcommit: 50,000 cores used → 50% utilization, but no risk of overload
With overcommit (1.5x): 75,000 cores effectively used → 75% utilization, some preemption risk
Savings: 25,000 cores x power/cooling/hardware = millions per year
```

## Amazon — AWS Capacity Planning

AWS has a unique capacity planning challenge: they must provision data centers before they know which customers will use them. A new AWS region costs $1 billion+ and takes 2 to 3 years to build.

**The estimation problem:**

AWS must predict compute, storage, and network demand 3 years into the future, at a granularity of individual instance types, across 30+ regions, without knowing what products customers will launch.

**How AWS approaches this:**

1. **Traffic modeling from existing regions.** New regions follow adoption curves calibrated from existing regions. If Singapore grew 40% in year 1, Mumbai will likely follow a similar pattern.

2. **Customer pipeline analysis.** Enterprise customers share migration plans with AWS sales teams. Aggregating these pipelines gives a demand signal 6 to 12 months ahead.

3. **Buffer zones.** AWS builds data center shells (empty buildings with power and cooling) that can be equipped with servers in 8 to 12 weeks. This gives them time to react to unexpected demand while the shell is already built.

4. **Instance family mixing.** Instead of predicting exactly how many m5.xlarge instances will be needed, AWS provisions a pool of generic compute capacity that can be configured into any instance type.

**AWS capacity cost:**

```
A single data center (25 MW): $150M to $300M build cost
Equipment: $200M to $400M
Total: $350M to $700M per facility

Annual power cost: 25 MW x 24 x 365 x $0.05/kWh = $11M
Cooling: roughly equal to compute power → $11M
Total annual operating: $22M

At a 5-year amortization:
($500M / 5) + $22M = $122M per year total cost of ownership
$122M / (25,000 servers x $200/month) = breakeven at ~20% utilization
```

AWS cannot run at 20% utilization. They aim for 60% to 70%, making their data centers highly profitable once demand materializes.

## Slack — Estimating for Enterprise Chat

Slack serves enterprise teams with a fundamentally different capacity profile than consumer messaging apps.

**The difference:**
- Enterprise customers expect 99.99% uptime (52 minutes downtime per year max)
- Messages must be indexed and searchable instantly
- File attachments (documents, PDFs, spreadsheets) dominate storage
- Each message renders in multiple channels with potential for thousands of members

**Slack's capacity challenge:**

```
10 million DAU (enterprise)
Each user sends 200 messages per day (2x consumer — heavier collaboration)
= 2 billion messages/day

But enterprise messages are longer and include rich embeds:
Average message size: 5 KB (vs. 1 KB for consumer chat)
Storage per day: 2B x 5 KB = 10 TB/day

File storage:
  10% of messages include a file attachment
  Average file: 1 MB (documents, screenshots, PDFs)
  200M files/day x 1 MB = 200 TB/day

Indexing:
  Every message must be full-text searchable within seconds
  Index size: roughly 50% of raw text = 5 TB text + 3.5 TB index/day
  This requires significant write amplification — 3x to 5x for the search index

Search QPS:
  20% of users do 5 searches/day = 10M searches/day
  = 116 search QPS average, 580 peak
  Each search scans the index: 10 ms to 100 ms depending on query complexity
  Search infrastructure needed: roughly 20 to 50 nodes (Elasticsearch cluster)
```

**The lesson:** Enterprise capacity profiles differ from consumer profiles even at similar DAU. Files are larger, messages are more frequent, search is required, and uptime SLAs are stricter. Estimate based on your specific use case, not generic benchmarks.

## Uber — Estimating for Real-Time Matching

Uber's capacity estimation is driven by real-time supply-demand matching, which has a unique constraint: location updates.

**The scale:**

```
50 million DAU (riders)
5 million drivers (active)
15 million trips per day

Per trip:
  1 ride request (rider-side)
  1 ride acceptance (driver-side)
  20 location pings during the ride (2 pings/minute for 10 min avg trip)
  1 payment transaction
  1 rating

= 24 operations per trip
= 360 million trip-related operations per day
= 4,167 QPS average, 12,500 QPS peak (3x)
```

**The real challenge: driver location updates.**

Uber's biggest capacity challenge is not ride-matching — it is location updates.

```
5 million active drivers
Location ping every 4 seconds
= 1.25 million location updates per second
= 1,250,000 writes per second

Each location ping:
  Driver ID: 8 bytes
  Latitude: 8 bytes
  Longitude: 8 bytes
  Timestamp: 8 bytes
  Speed, heading, accuracy: 12 bytes
  Total: 44 bytes

Write throughput: 1.25M x 44 bytes = 55 MB/s = 440 Mbps

This is 300x the QPS of the trip-related operations.
Location updates dominate the write path.
```

**How Uber handles this:**

1. **Geospatial index sharding** — location data is sharded by geographic region (hexagonal grid cells), not by driver ID. Each cell is small enough that a single server can handle all drivers in that area.

2. **Temporal decay** — if a driver stops sending updates, their location is marked stale after 15 seconds and removed after 60 seconds. This keeps the active set at 5 million, not growing unboundedly.

3. **Read replicas for geo-queries** — rider location lookups (find nearby drivers) are served from read replicas updated asynchronously. The rider does not need the driver's exact location — within 100 meters is good enough.

```
Driver location writes:   1,250,000 QPS → 500 DB nodes (Cassandra, 2,500 writes/node)
Geo-query reads:          500,000 QPS → 100 read replicas (5,000 reads/node)
Ride matching compute:    1,000 QPS → 50 servers (heavier computation)
Total infrastructure:     roughly 650 nodes + geo-index layer
```

## Stripe — Estimating for Financial Infrastructure

Stripe processes hundreds of billions of dollars in payments annually. Financial capacity estimation has unique constraints: every operation must be durable, auditable, and consistent.

**The constraints:**

```
Transaction volume: 1 billion+ API requests/day
= 11,574 QPS average, 35,000 QPS peak

But each payment involves:
  1 API request → 1 authorization → 1 capture → 1 settlement → 1 reconciliation
  Plus retries: 5% of transactions have 1 to 3 retries
  Plus webhooks: each event generates 3 to 5 webhook deliveries

Amplification: 1 user action = 5 to 10 internal operations
Effective system QPS: 35,000 x 7 = 245,000 internal QPS
```

**Storage requirements:**

```
Each transaction stores:
  Payment details: 2 KB
  Log entries (10 per transaction): 10 KB
  Audit trail: 5 KB
  Dispute evidence if applicable: 100 KB (rare — 0.1% of transactions)
  Metadata and webhook logs: 3 KB
  Total per transaction: roughly 20 KB average

Daily storage: 1B x 20 KB = 20 TB/day
7-year retention (regulatory): 20 TB x 365 x 7 = 51.1 PB

Replication (3x synchronous — financial, cannot lose data):
  153.3 PB of raw database storage

Write amplification from indexes:
  Each transaction is indexed by: ID, customer, merchant, date, amount, status, card fingerprint
  7 indexes x 20 KB per transaction = 140 KB index write per transaction
  1B transactions x 140 KB = 140 TB/day index writes

Total write I/O: 20 TB (data) + 140 TB (indexes) = 160 TB/day write amplification
This is 8x amplification — the database must handle 8x the raw transaction volume in disk I/O.
```

**The lesson:** Regulated industries multiply capacity needs. Data retention laws (7 years), audit requirements, and synchronous replication (can't use async for payments) drive 10x to 50x more storage and write capacity than an unregulated system with the same transaction volume.

## Five Lessons from Real-World Systems

**Lesson 1: Estimates drive architecture, not just cost.**

Facebook did not build bigger GPU clusters. They changed their ranking architecture because the estimate showed the full-compute approach was infeasible. Capacity estimation should influence design decisions, not just provisioning numbers.

**Lesson 2: Peak is the enemy. Design for peaks.**

Twitter provisions for normal traffic plus event spikes using reserved + elastic + degradation. Netflix provisions for Friday evening 8 PM. AWS provisions for Black Friday. Every real system is shaped by its peak, not its average.

**Lesson 3: Build feedback loops.**

Google Borg measures actual usage and feeds it back into capacity recommendations. Facebook A/B tests new ranking models on 1% of traffic before rolling to 100%. Without measurement, your estimates remain guesses. Close the loop: estimate, measure, adjust.

**Lesson 4: Over-provision in some dimensions, optimize in others.**

Netflix over-provisions storage (every edge node has the full library) but optimizes network (limits concurrent streams per node). Twitter over-provisions for events but optimizes for normal days. Amazon over-provisions power and cooling (shells are cheap) but optimizes server placement. Know which dimensions matter to your system and allocate accordingly.

**Lesson 5: Unit economics matter at scale.**

Amazon's per-server economics: $122M/year total cost for 25,000 servers = $4,880 per server per year. If a server generates $10,000 in revenue, the margin is 51%. If it generates $3,000, the margin is negative. Every capacity estimation at scale leads back to a simple question: does the infrastructure cost less than the revenue it enables?

## Applying the Lessons to Your Systems

Take these five lessons and apply them to a system you build or use:

**Step 1: Identify your dominant constraint.**
Is your system compute-bound (transcoding), storage-bound (photo uploads), bandwidth-bound (video streaming), or IO-bound (logging)? The real-world examples show that each company's biggest challenge was different.

**Step 2: Find your event peaks.**
If you run a B2B SaaS product, your peak might be the first Monday of the month (reports). If you run e-commerce, your peak is Black Friday. If you run social media, your peak is evening hours and major events. Measure your actual peak, do not guess.

**Step 3: Build a feedback loop.**
Start with an estimate, deploy with monitoring, compare actual vs. estimated usage, and adjust your next estimate. After 3 to 4 cycles, your estimates should be within 20% of reality.

**Step 4: Know your per-unit cost.**
Calculate cost per DAU, cost per request, and cost per GB stored. Track these over time. If cost per DAU rises, your efficiency is declining. If it drops, your optimizations are working.

**Step 5: Plan for the worst day, budget for the average.**
Provision capacity for peak traffic (with auto-scaling). Budget based on average traffic (because peak is only a few hours per day). The difference between peak capacity and average cost is the headroom you pay for reliability.
