## Case Studies

### Google Chubby (Paxos)

Google Chubby was the first widely-known production consensus system. Deployed in 2005, Chubby provides a distributed lock service and a small-file storage system used by Google's distributed systems infrastructure. Chubby uses the Paxos consensus algorithm internally to replicate its state across five replicas.

Chubby's key design decision: a lock service with a filesystem-like interface. Applications acquire locks to perform operations, and the lock data is stored in a replicated, Paxos-backed filesystem. Chubby handles 30,000 to 50,000 operations per second on typical hardware with a 5-node cluster. It uses a lease-based approach for caching: clients cache file contents and metadata, and Chubby invalidates the cache when the data changes.

Google's lessons from Chubby shaped the industry. They found that Paxos, while correct, was difficult to implement and debug. Chubby's developers spent significant effort on testing and formal verification. The complexity of Paxos motivated the development of Raft, which prioritizes understandability. Chubby itself has been largely replaced by other systems within Google, but its design influenced ZooKeeper, etcd, and Consul.

### ZooKeeper (Zab)

Apache ZooKeeper is an open-source coordination service used by Kafka, HBase, Hadoop, Solr, and thousands of other systems. ZooKeeper implements the Zab (ZooKeeper Atomic Broadcast) protocol, which is similar to Paxos but tailored for ZooKeeper's workload: a hierarchical namespace (znodes) with watches for change notifications.

ZooKeeper's design emphasizes high read throughput. Reads are served from any node (including followers), while writes go through the leader. This design serves read-heavy coordination workloads well. A 5-node ZooKeeper ensemble on modern hardware processes 50,000 to 80,000 read operations per second and 10,000 to 20,000 write operations per second.

Key lessons from ZooKeeper production use: session timeouts must be configured carefully -- short timeouts detect failures quickly but cause frequent reconnections during GC pauses or network jitter. ZooKeeper's watches (one-shot triggers) require careful programming to avoid missed events. The community has learned that ZooKeeper performs best with dedicated hardware and careful JVM tuning.

### etcd (Raft)

etcd is an open-source distributed key-value store that uses the Raft consensus algorithm. It is the most widely deployed Raft-based system, serving as the backing store for Kubernetes, the most popular container orchestration platform.

etcd stores Kubernetes cluster state: all pod definitions, service definitions, secrets, configmaps, and custom resource definitions. A Kubernetes cluster with 1,000 nodes typically uses a 3-node or 5-node etcd cluster. etcd serves 100 to 10,000 operations per second depending on cluster size and hardware.

Key lessons from etcd production use: etcd's performance is highly sensitive to disk latency. An etcd write must fsync to disk before the leader responds. Slow disks (spinning disks, overloaded shared storage) cause latency spikes and cluster instability. etcd operators monitor the backend commit duration metric (99th percentile < 10ms is healthy). Multi-region etcd deployments require careful network planning, as cross-region latency directly adds to write latency.

### Consul (Raft)

HashiCorp Consul uses Raft for service discovery and configuration management. Consul provides service registration, health checking, key-value storage, and multi-datacenter federation.

Consul's key insight: use Raft for the local data center and a gossip-based protocol for cross-data-center federation. Each data center runs its own Raft cluster (3 or 5 nodes). Consul's gossip protocol (based on Serf) handles failure detection and membership in each data center, while the Raft cluster provides strongly consistent storage within the data center.

### Open-Source Implementations

**etcd** -- Raft-based key-value store. Written in Go. Used by Kubernetes, OpenStack, and many other projects. Provides a simple key-value API with watches and leases. Production deployment: 3 to 7 nodes, SSDs, dedicated hosts.

**ZooKeeper** -- Zab-based coordination service. Written in Java. Used by Kafka, HBase, Solr, and Hadoop. Provides a hierarchical namespace with watches. Production deployment: 3 to 7 nodes, 8-16GB heap, SSD storage.

**Consul** -- Raft-based service discovery and configuration. Written in Go. Provides service mesh, health checks, and KV store. Built-in multi-datacenter support via gossip protocol.

**TiKV** -- Raft-based distributed key-value store. Written in Rust. Used as the storage layer for TiDB, a distributed SQL database. TiKV uses Raft for consensus and Multi-Raft for scaling across many regions.

**CockroachDB** -- Raft-based distributed SQL database. Uses Raft for range replication. Each range of data is replicated using Raft with configurable replication factors (typically 3 or 5). CockroachDB's design demonstrates that consensus can be composed into a scalable system by using many small Raft groups (one per data range) rather than one large group per node.

### Cloud Managed Services

**AWS MemoryDB for Redis** -- A Redis-compatible, durable, in-memory database that uses a Multi-Paxos consensus protocol for replication and durability. Each write is replicated to a quorum of nodes and durably stored in a transaction log. MemoryDB provides Redis semantics with the durability of a consensus-based system.

**Google Cloud Spanner** -- A globally distributed, strongly consistent relational database. Spanner uses the Paxos consensus algorithm for each tablet (a partition of data). Spanner's TrueTime API (GPS and atomic clocks) enables external consistency (linearizability) across global deployments. Each Spanner write is replicated to 3 to 5 Paxos replicas.

**Amazon DynamoDB** -- Uses a quorum-based replication system (N=3, W=2, R=2) that provides configurable consistency. DynamoDB's internal replication is based on the Dynamo paper's quorum system, not a full consensus protocol. It uses hinted handoff and version vectors for conflict resolution.

### Lessons Learned

**Lesson 1: Consensus is slow.** Every consensus write requires an fsync on at least a majority of nodes. On typical SSDs, fsync takes 1-2ms. The network RTT adds another 1-10ms in the same region. Total write latency is 3-15ms. Applications expecting sub-millisecond writes should use eventually consistent systems.

**Lesson 2: Consensus does not scale to hundreds of nodes.** A single consensus group with 50 nodes would require 26 nodes to acknowledge each write, and the leader would need to maintain connections to 49 followers. The throughput would be severely limited. Real deployments use 3-7 nodes per consensus group.

**Lesson 3: Multi-Raft composes consensus for scale.** Systems like CockroachDB and TiKV use many small Raft groups rather than one large group. Each data range (64MB in CockroachDB) is a separate Raft group. The data is partitioned across ranges, and each range has its own leader and followers. This design provides linear scalability while maintaining consensus guarantees per range.

**Lesson 4: Monitor the right things.** The most important consensus metrics are: leader changes per hour, commit latency (p99), pending proposals, log compaction lag, and follower commit index lag. Alert on any sustained increase. A leader change that takes more than 10 seconds indicates a problem.

**Lesson 5: Test failure scenarios.** Consensus algorithms are complex, and subtle bugs can remain dormant for years. Test regular failure scenarios: kill the leader, kill a follower, partition the network, saturate the disk, and add a slow node. Use Jepsen-style testing to verify that the algorithm behaves correctly under failures.

**Lesson 6: Plan for reconfiguration.** Adding or removing nodes from a consensus cluster is a risky operation. A misconfigured reconfiguration can cause quorum loss. Always verify the cluster health before and after reconfiguration. Prefer automated cluster management (etcd operator, ZooKeeper dynamic reconfiguration) over manual steps.

## Production Incident Case Studies

### Incident 1: etcd Quorum Loss in a Kubernetes Cluster

A large Kubernetes cluster with a 3-node etcd cluster experiences two simultaneous node failures. One node fails due to a kernel panic. A second node fails due to disk corruption during a routine compaction. The third node is alive but cannot form quorum alone (needs 2 of 3). The entire Kubernetes control plane becomes unavailable. Pods continue running on worker nodes but no new pods can be scheduled, no configuration changes can be made, and no deployments can be rolled out.

Resolution: the corrupted node is restored from a recent backup. The remaining two nodes form quorum. The cluster recovers within 12 minutes. The lesson: a 3-node cluster provides minimal fault tolerance. For production-critical systems, a 5-node cluster would tolerate the two simultaneous failures.

### Incident 2: ZooKeeper Session Expiry Cascade

A ZooKeeper ensemble experiences a brief network jitter (2 seconds of packet loss). During the jitter, 20% of the ZooKeeper clients lose their sessions. The session expiry triggers watches on all ephemeral znodes. The watches trigger callbacks in all connected applications. The callbacks generate a burst of writes to ZooKeeper, overwhelming the leader. The leader becomes slow, causing more clients to time out. The cascade continues for 90 seconds before stabilizing.

Resolution: the session timeout was configured at 4 seconds, which was too short. Increasing the session timeout to 10 seconds made the system resilient to brief network jitter. The lesson: session timeouts must account for worst-case network latency plus GC pauses. A common recommendation is 10x the expected network RTT, with a minimum of 5 seconds.

### Incident 3: Raft Replay After Network Recovery

A Consul cluster with 5 nodes experiences a network partition that separates 2 nodes from the other 3 for 5 minutes. The majority side (3 nodes) continues operating normally. When the partition heals, the 2 isolated nodes have missed 300,000 log entries. The leader begins replicating the missing entries. The replication consumes 100% of the leader's CPU and network bandwidth for 8 seconds. During this time, new client requests time out.

Resolution: configure the cluster with request rate limiting to protect the leader during catch-up. The lesson: log catch-up after a long partition can temporarily degrade performance. Rate limiting prevents complete unavailability during catch-up.

## Vendor Comparison

### When to Use Each Implementation

**etcd**: Best for Kubernetes and cloud-native environments. Written in Go, easy to deploy. Provides a simple key-value API. Consumes minimal resources. The default choice for new projects that need a consensus-based coordination service.

**ZooKeeper**: Best for systems that need hierarchical data (znodes) and watches. Used by Kafka, HBase, and Solr. Mature and battle-tested. Requires Java runtime. Higher resource consumption than etcd. The right choice when the ecosystem already includes ZooKeeper (Kafka, Hadoop).

**Consul**: Best for service discovery and mesh networking. Built-in health checking, DNS interface, and multi-datacenter support. More features than etcd or ZooKeeper but also more complexity. The right choice when service discovery is the primary use case.

## Real-World Failure Numbers

Production consensus systems accumulate failure data over time. The following numbers are based on reported incidents from major cloud providers and open-source projects.

**etcd**: The median time between leader changes in a healthy 3-node cluster is 30+ days. The 99th percentile leader change duration (time from last heartbeat to new leader serving requests) is 400ms. Most leader changes are caused by kernel soft lockups (40%), disk latency spikes (30%), network jitter (20%), and other causes (10%).

**ZooKeeper**: A 5-node ZooKeeper ensemble serving 1,000 clients experiences approximately 2 leader elections per year in a well-maintained environment. Each leader election takes 200-800ms. The most common cause of ZooKeeper failures is session expiration cascades triggered by GC pauses.

**Consul**: A 5-node Consul cluster serving a 2,000-node service mesh experiences approximately 1 leader change per month during normal operations (planned maintenance). Unplanned leader changes occur once every 3-4 months. Consul's gossip protocol for health checking adds resilience: even if the Raft leader is briefly unavailable, the health check data propagates via gossip, maintaining the service mesh.

## Evolution of Consensus Systems

Classical consensus (Paxos, Raft, Zab) assumes crash failures: nodes either work correctly or stop working. Modern distributed systems face a broader range of failure modes, including Byzantine faults (arbitrary behavior), performance faults (slow nodes), and configuration faults (misconfigured nodes).

Newer consensus variants address these challenges:
- **EPaxos (Egalitarian Paxos)**: Eliminates the leader bottleneck by allowing any node to propose values. Provides lower latency in geographically distributed deployments.
- **Flexible Paxos**: Relaxes the quorum intersection requirement to allow faster decisions in specific configurations.
- **WPaxos (Wide-area Paxos)**: Optimizes Paxos for wide-area deployments by using multiple leaders for different key ranges.
- **Byzantine Fault Tolerance (BFT)**: Consensus that tolerates arbitrary (Byzantine) failures. Used in blockchain systems. Requires 3f+1 nodes to tolerate f Byzantine faults.
- **Proof-of-Stake**: Consensus for blockchain systems that uses stake-based voting rather than leader-based coordination. Sacrifices strong consistency for decentralization.
