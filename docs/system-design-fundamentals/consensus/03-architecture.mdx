## Naive Architecture

A naive consensus deployment places all nodes in a single data center on the same network segment. The leader handles all writes. Reads go to any node. If the leader fails, a new leader is elected from the remaining nodes. This design works for small systems but fails under realistic production conditions.

## Production Architecture

A production consensus deployment spans three data centers. Each data center contains one or two consensus nodes. The cluster size is odd to prevent split-brain: 3 nodes (one per data center) or 5 nodes (two in the primary region, one in each of two secondary regions, or similar distribution).

The leader resides in one data center (typically the primary region). All other nodes are followers. Write latency includes the cross-region RTT to the farthest follower that the leader needs for quorum. With a 3-node cluster across US East, US West, and EU West, write latency is approximately 80ms (the time to reach EU West and back). Reads can be served locally by the nearest follower if eventual consistency is acceptable, or routed to the leader for strong consistency.

### Integration with Other Systems

Consensus clusters are typically embedded in or paired with other infrastructure components.

**Integration with Databases**

A consensus cluster can be the backend for a distributed database like etcd (used by Kubernetes), ZooKeeper (used by Kafka, HBase), or Consul (used for service discovery). The database runs the consensus algorithm internally. Application databases use the consensus cluster for leader election, configuration storage, and distributed locks.

**Integration with Caches**

A consensus cluster can manage cache topology and invalidation. Redis Sentinel uses a gossip-based consensus (not Raft/Paxos but a custom protocol) to monitor Redis instances and perform failover. Redis Cluster uses its own gossip protocol for slot assignment. For applications that use Redis as a cache, the consensus cluster stores the mapping of which cache node holds which shard.

**Integration with Queues**

Apache Kafka uses ZooKeeper (a consensus service based on Zab) for controller election, cluster membership, and topic configuration. ZooKeeper stores the Kafka broker metadata: which broker is the controller, which partitions are assigned to which brokers, and the ISR (in-sync replica) set for each partition. Kafka's availability depends on ZooKeeper's consensus guarantees.

**Integration with External APIs**

External API gateways may use a consensus cluster to store rate limit configurations, API keys, and routing rules. The consensus cluster provides a strongly consistent source of truth that all API gateway instances read from. When a new API key is added, the write goes to the consensus cluster, and all gateways see the update within milliseconds.

### Capacity Planning

**Write Capacity**

Each consensus write requires a majority of nodes to persist the entry. The throughput is bounded by the leader's network bandwidth and CPU. For a 3-node cluster on 1 Gbps links with 1 KB entries:
- Maximum throughput: 10,000 to 30,000 writes per second
- Bottleneck: leader's outbound bandwidth (3 nodes x 1 KB x 30,000 writes/sec = 90 MB/s, ~720 Mbps)

**Storage Estimation**

Each consensus node stores the full replicated log. For etcd with 1 KB entries, 1 million entries consume 1 GB of disk. With 10,000 writes per second, the log grows at 10 MB per second, 36 GB per hour. Production deployments enable automatic compaction (compression + deletion of old entries) to bound the log size. A typical retention period is 1 to 7 days of entries, plus a snapshot of the current state.

**Network Considerations**

Cross-region consensus clusters require careful network planning. The write latency is dominated by the slowest link in the quorum. For a 5-node cluster with 2 nodes in US East, 2 in US West, and 1 in EU West, a quorum requires 3 of 5 nodes. The leader in US East can get acknowledgments from the other US East node and one US West node, avoiding the 80ms EU West hop. This reduces write latency from ~160ms to ~60ms. The leader selection algorithm should prefer a node in the region with the most nodes.

## Operations

### Monitoring

Key metrics for consensus cluster health:

- **Leader changes per minute**: A healthy cluster has 0 leader changes per minute. Any leader change indicates a failure or network issue.
- **Term changes per minute**: Similar to leader changes; rapid term changes indicate instability.
- **AppendEntries RTT**: The time for a leader to receive a majority of acknowledgments. Should be stable and close to the average network RTT.
- **Pending proposals**: The number of proposals waiting for acknowledgment. A growing backlog indicates a slow follower or network problem.
- **Log size**: Should be bounded by compaction. Unbounded growth indicates compaction failure.
- **Commit index lag**: The difference between the leader's commit index and each follower's last applied index. Large lag indicates a slow follower.

### Backup

Consensus clusters store their state in durable storage (disk). Full backups should be taken regularly. An etcdctl snapshot command captures the full database state. For ZooKeeper, a zkSnapShot captures the tree. Backups should be stored off-cluster and tested regularly by restoring to a separate cluster.

### Failure Scenarios

**Leader failure**: If the leader fails, the cluster elects a new leader within 150 to 300 milliseconds. Clients may experience a write failure or timeout during the election. The new leader picks up from the last committed entry in its log.

**Follower failure**: A follower failure does not affect writes as long as quorum is maintained. The leader continues to operate with the remaining followers. When the failed follower recovers, it catches up by receiving all missed log entries from the leader.

**Network partition**: If a partition splits the cluster, the side with a majority continues to operate. The minority side cannot elect a leader or commit writes. When the partition heals, the minority side catches up.

**Disk full**: If any node's disk is full, it cannot persist new log entries. The node becomes unavailable. In Raft, the leader detects the failure through failed AppendEntries and decreases the commit progress for that node. The cluster continues as long as a majority of nodes have available disk.

**Clock skew**: Consensus algorithms that use clock time only for election timeouts (Raft) are tolerant of moderate clock skew. Algorithms that use clock timestamps for ordering (Paxos with wall-clock proposal numbers) are vulnerable to clock skew. All production deployments should run NTP with proper monitoring.

### Manual Intervention

Operations teams should be prepared for these manual procedures:
- **Manual leader transfer**: Move leadership to a different node for maintenance.
- **Member addition**: Add a new node to the cluster. This is a multi-step process that must be done safely to maintain quorum.
- **Member removal**: Remove a faulty node. The cluster reconfigures to the new membership.
- **Cluster reconfiguration**: Change cluster size (e.g., from 3 to 5 nodes). Must be done one node at a time to maintain quorum.
- **Disaster recovery**: Restore the cluster from backup if all nodes fail.

## Detailed Deployment Topologies

### Single-Region, 3-Node Cluster

The simplest production topology. All three nodes in the same data center, connected by a low-latency network. The leader is automatically elected. Writes require 2 of 3 acknowledgments. The cluster tolerates 1 node failure.

This topology is appropriate for development environments, small production deployments, and systems that do not need multi-region failover. The write latency is typically 1-5ms p99.

### Single-Region, 5-Node Cluster

Five nodes in the same data center. The leader requires 3 of 5 acknowledgments. The cluster tolerates 2 node failures. This is the standard production topology for critical systems.

The 5-node cluster provides redundancy for upgrades and maintenance. An operator can take one node offline for maintenance, another node can fail, and the cluster continues operating. This is the minimum configuration for rolling upgrades.

### Multi-Region, 3-Node Cluster

One node in each of three data centers. The leader is typically in the data center with the lowest latency to the other two. Writes require 2 of 3 acknowledgments. The cluster tolerates 1 data center failure.

Write latency: approximately 2x the cross-region RTT to the farthest node in the quorum. If the leader is in US East, the quorum can be the leader (US East) plus one follower (US West, 40ms). Write latency is ~40ms. If the US East node fails, the leader election takes place between US West and EU. The new leader in US West needs one more acknowledgment. The farthest node (EU) is 80ms away. Write latency becomes ~80ms.

### Multi-Region, 5-Node Cluster

Two nodes in the primary data center, one node in each of two secondary data centers, and one node in a third data center or co-located with the primary. The leader is in the primary data center. Quorum requires 3 of 5 acknowledgments. The leader can get quorum from the second primary node and one secondary node, avoiding the distant third data center.

This topology provides the best balance of fault tolerance and performance. The primary data center can operate independently if both secondary data centers become unreachable (the primary's 2 nodes plus the leader form a quorum of 3). Write latency is bounded by the closest secondary data center.

## Observability Deep Dive

### Telemetry to Collect

Every consensus node should expose these metrics for monitoring and alerting.

**Leadership metrics**: current leader (which node), term number, leader changes per hour, time since last leader change. A stable system has zero leader changes per hour. A single leader change is acceptable. More than 5 leader changes per hour indicates instability.

**Commit metrics**: commit index, last applied index, time to commit a proposal (p50, p99, p999), number of pending proposals. The commit time should be stable. A sudden increase in commit time indicates a slow follower or network issue.

**Replication metrics**: per-follower next index, match index, AppendEntries RTT, number of failed AppendEntries. A follower that consistently fails AppendEntries may be experiencing hardware issues.

**Storage metrics**: log size on disk, snapshot size, compaction progress, disk write latency. The log size should be bounded by the compaction policy. Unbounded log growth indicates a compaction failure.

### Health Check Patterns

A liveness check confirms that the node is running. A readiness check confirms that the node can serve requests. A consensus node is ready only if it has caught up with the leader's log.

Pattern 1: check that the node's commit index is within a threshold of the leader's commit index (e.g., within 100 entries). This ensures that reads from the node return reasonably fresh data.

Pattern 2: check that the node is the leader (for leader-only read paths). If the node is not the leader, the health check fails, and the load balancer removes the node from rotation.

Pattern 3: check that the node's disk is not full and that the fsync latency is within limits. This catches storage issues before they affect the cluster.

### Capacity Sizing Examples

**Example 1: Kubernetes control plane (etcd)**. A 300-node Kubernetes cluster requires approximately 100 writes per second and 500 reads per second. A 3-node etcd cluster with 4 GB RAM and 64 GB SSD is sufficient. The etcd database size is typically under 10 GB for a 300-node cluster.

**Example 2: Large Kafka deployment (ZooKeeper)**. A 100-broker Kafka cluster with 10,000 topic partitions requires approximately 200 writes per second and 1,000 reads per second from ZooKeeper. A 5-node ZooKeeper ensemble with 16 GB RAM and 128 GB SSD is appropriate. The ZooKeeper database size is typically under 5 GB.

**Example 3: Global service discovery (Consul)**. A 2,000-node service mesh with 10,000 registered services requires approximately 500 writes per second and 5,000 reads per second from Consul. A 5-node Consul cluster with 8 GB RAM and 64 GB SSD is appropriate. Consul scales reads horizontally by serving from any node.

## Security Considerations

**TLS encryption**: All consensus communication between nodes should be encrypted with TLS. Without encryption, an attacker on the network can read, modify, or inject consensus messages. This can compromise the integrity of the consensus decisions.

**Authentication**: Nodes should authenticate each other before accepting consensus messages. This prevents a rogue node from joining the cluster and participating in decisions. Most consensus implementations support mutual TLS (mTLS) for node-to-node authentication.

**Role-based access control**: Client access to the consensus cluster should be restricted. Only authorized clients should be able to write to the cluster. Read access may be granted to a broader set of clients. etcd and Consul both support role-based access control (RBAC).

**Audit logging**: All consensus state changes should be logged for audit purposes. The audit log records who changed what and when. This is important for compliance and for debugging configuration changes.
