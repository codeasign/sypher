## Company Case Studies

### Netflix: The Alerting Culture

Netflix operates one of the most sophisticated alerting systems in the world. With hundreds of microservices, thousands of instances, and millions of streaming customers, alerting at Netflix is a first-class engineering discipline.

**The Philosophy:** Netflix alerting philosophy is based on the concept of "alerting on customer impact." If a metric does not directly correlate to a customer experience degradation, it should not be paged. This means Netflix ignores most infrastructure metrics — CPU, memory, disk — and focuses on service-level metrics.

**The Implementation:** Netflix uses a combination of Atlas (their in-house monitoring system) and a custom alerting engine called Kabana. Kabana evaluates alert rules against Atlas metrics and routes alerts through a multi-tier notification system.

**Key Practice — The Alert Review:** Every month, Netflix runs an alert review. Each alert rule is evaluated for its signal-to-noise ratio. Alerts that fired more than 10 times in the month without a corresponding incident are candidates for deletion. Alerts that never fired are candidates for deletion. Alerts that fired and were actionable are kept. This quarterly pruning keeps the alert count manageable.

**Key Practice — The Chaos Engineering Connection:** Netflix uses Chaos Monkey to simulate failures. When a failure is simulated, the alerting system is expected to fire. If it does not fire, the missing alert is a bug. Chaos experiments validate the alerting system continuously. Netflix runs chaos experiments in production during business hours. If the alerting system does not catch the simulated failure, a bug is filed against the alerting team.

### Uber: Alerting at Global Scale

Uber runs a real-time platform across 70+ countries. Their alerting system must handle diverse traffic patterns, multiple time zones, and dozens of regional deployments.

**The Challenge:** Diurnal traffic patterns create a problem for static thresholds. A 100 QPS (queries per second) increase during peak hours in New York is normal. The same increase at 3 AM in Tokyo is a crisis. Static thresholds cannot distinguish between these scenarios.

**The Solution:** Uber built a dynamic thresholding system called M3 (Metrics, Monitoring, and More). M3 uses machine learning to model each metric expected behavior at every hour of the day, day of the week, and day of the year. The alert threshold is computed as the expected value plus a margin based on historical variance.

**Key Practice — Regional Alerting:** Uber runs alerting stacks in each region. A regional alerting engine evaluates rules against local metrics. If the region loses connectivity to the central monitoring system, regional alerting continues to work. The central system aggregates alerts from all regions for global visibility.

**Key Practice — Incident Triage Automation:** When an alert fires, Uber system automatically runs a set of diagnostic checks. The checks include: checking recent deployments, querying the service dependency graph, and analyzing log patterns. The results are attached to the alert before the responder is paged. This saves the responder 10-15 minutes of manual investigation. The diagnostic checks are implemented as serverless functions that run in parallel.

### Discord: Alerting for Real-Time Communication

Discord serves 150 million monthly active users with real-time voice and text communication. Every millisecond of latency matters. Every outage means millions of users cannot communicate.

**The Challenge:** Discord alerting system must differentiate between a regional internet outage (not Discord fault), a cloud provider issue (Discord responsibility to mitigate), and a Discord-specific bug (Discord responsibility to fix). Each scenario requires a different response.

**The Solution:** Discord uses a layered alerting approach. The first layer detects symptoms at the user-facing level — message send failures, voice connection drops, gateway latency. The second layer detects causes at the infrastructure level — database query latency, cache hit rates, queue depths. The third layer correlates alerts across layers to identify the root cause.

**Key Practice — Alert Triage Runbooks:** Every Discord alert has a detailed runbook. The runbook includes: the exact dashboard to check, the logs to search, the common root causes with their frequency (top 5 causes account for 80 percent of incidents), and the exact fix for each cause. New on-call engineers can resolve 80 percent of incidents by following the runbook without deeper investigation.

**Key Practice — Postmortem-Driven Alert Creation:** Every incident postmortem at Discord asks: "What alert should have caught this earlier?" If the answer is "none," a new alert rule is created. If the answer is "the existing alert fired but nobody responded," the alert severity, channel, or escalation is adjusted. This ensures the alerting system improves with every incident.

### Cloudflare: Alerting at Internet Scale

Cloudflare operates a global network that handles 20 percent of the world internet traffic. Their alerting system must detect problems across 300+ cities, 50+ TBit/s of traffic, and millions of customer origins.

**The Challenge:** At Cloudflare scale, false positives are catastrophic. A single false-positive alert can trigger a global incident response, wasting hours of engineering time across multiple teams. The cost of a false positive is orders of magnitude higher than the cost of a delayed alert.

**The Solution:** Cloudflare uses a multi-stage alert confirmation pipeline. Before an alert is sent to a human, it must be confirmed by multiple independent sources. A latency spike must be confirmed by: the edge node local metrics, the global traffic dashboard, and the customer upstream probe. Only when all three sources agree does the alert fire.

**Key Practice — Traffic-Aware Alerting:** Cloudflare alerting system knows that global traffic patterns are not uniform. A spike in traffic from Asia during the Asian business hours is normal. The same spike from Europe at 3 AM is suspicious. The alerting system uses traffic pattern models per region per time of day.

**Key Practice — Autotuning Thresholds:** Cloudflare alerting system automatically adjusts thresholds based on the previous 7 days of data. If a threshold has not been breached in 7 days, the system lowers it by 5 percent. If the threshold has been breached more than 3 times in 7 days, the system raises it by 5 percent. This continuous tuning keeps the alert count stable even as traffic patterns change.

### PagerDuty: Building the On-Call Platform

PagerDuty, the company that powers incident response for thousands of organizations, uses its own platform internally. Their architecture provides a reference for how on-call scheduling and escalation should work.

**The Architecture:** PagerDuty internal alerting system receives alerts from multiple monitoring tools, normalizes them into a standard event format, and routes them through the PagerDuty platform. The platform handles on-call scheduling, escalation policies, and notification delivery.

**Key Practices:**
- Escalation policies are versioned and tested. Changes are deployed to a staging environment first.
- On-call schedules are automated. Engineers do not manually update who is on-call. The schedule is generated from the team rotation configuration.
- Postmortems are mandatory for any incident that resulted in a page. The postmortem template includes a section on alerting system performance.

## Open Source Implementations

### Prometheus Alertmanager

The most widely used open-source alerting system. Alertmanager receives alerts from Prometheus servers, handles deduplication and grouping, and routes notifications through the configured receivers. It supports grouping, inhibition, silencing, and routing based on labels.

**Strengths:** Simple to configure, integrates natively with Prometheus, supports multiple notification channels, mature and well-documented. Community support is excellent. The configuration is YAML-based and version-controlled.

**Weaknesses:** No built-in dashboard, limited deduplication options, no escalation policies, no incident management integration. Alertmanager is designed to be a component of a larger system, not a standalone solution. It requires Prometheus for metric collection and evaluation.

### Grafana Alerting

Grafana built-in alerting system supports multiple data sources (Prometheus, Graphite, Loki, InfluxDB, etc.). It provides a unified alerting interface across all your monitoring tools. Alert rules are configured through the Grafana UI.

**Strengths:** Unified alerting across multiple data sources, easy to set up, good UI for rule management, supports alert grouping and silences. The Grafana alerting system is increasingly popular as organizations standardize on Grafana for observability.

**Weaknesses:** Can be slow with many rules, limited to Grafana evaluation engine, no built-in escalation. Grafana alerting is part of the Grafana ecosystem and requires Grafana to be running.

### Alerta

An open-source alert aggregator that receives alerts from multiple sources and presents them in a unified dashboard. Alerta is designed for environments where multiple monitoring tools produce alerts that need to be correlated.

**Strengths:** Excellent for multi-tool environments, good deduplication, useful API for automation. Alerta can receive alerts from Prometheus, Nagios, Datadog, and any system that can send webhooks.

**Weaknesses:** Requires additional infrastructure, not a standalone alerting engine, limited notification options. Alerta is a correlation and visualization layer, not an evaluation engine.

## Cloud Provider Implementations

### AWS CloudWatch Alarms

AWS CloudWatch Alarms evaluate CloudWatch metrics and trigger actions on SNS topics. Alarms support static and anomaly detection thresholds. The evaluation interval is 1 minute for detailed monitoring metrics.

**Strengths:** Deep integration with AWS services, no infrastructure to manage, simple to set up. CloudWatch Alarms work out of the box with EC2, RDS, Lambda, ELB, and most AWS services.

**Weaknesses:** Limited to CloudWatch metrics, expensive at scale, no built-in escalation, noisy by default. CloudWatch Alarms are suitable for small to medium AWS deployments but become expensive and hard to manage at scale.

### Google Cloud Monitoring (formerly Stackdriver)

Google Cloud Monitoring provides alerting policies that evaluate metrics from GCP services and external systems. Policies support conditions, notifications, and documentation.

**Strengths:** Good integration with GCP, supports log-based metrics, includes SLO burn rate alerting, comprehensive documentation. GCP Monitoring has the most mature SLO-based alerting of the three major cloud providers.

**Weaknesses:** Limited to GCP ecosystem, can be complex to configure for advanced use cases, no built-in deduplication. The pricing model is based on the number of metric bytes ingested, which can be unpredictable.

### Azure Monitor Alerts

Azure Monitor Alerts evaluate metrics and log queries from Azure resources. Alerts support action groups that define notification channels and escalation.

**Strengths:** Deep Azure integration, good for log-based alerts, supports dynamic thresholds, integrated with Azure DevOps. Azure Monitor Alerts work well for organizations that are fully invested in the Azure ecosystem.

**Weaknesses:** Limited to Azure, can be slow to evaluate complex log queries, pricing can be unpredictable. The query-based alerts can take several minutes to evaluate, which is too slow for critical alerting.

## Lessons Learned

**Lesson 1: Start with the runbook, not the alert.** Before you create an alert rule, write the runbook. If you cannot write a runbook for the alert, you do not understand the failure mode well enough to alert on it. The runbook is the test of whether the alert is actionable. A good runbook takes 30 minutes to write. A bad alert wastes hours of responder time.

**Lesson 2: Measure your signal-to-noise ratio.** Track the number of alerts that fire per day, per week, per month. Track the number of alerts that result in an incident. If the ratio of alerts to incidents is under 10 percent, you have too much noise. Reduce thresholds, increase durations, or delete rules. A healthy alerting system has a 20-30 percent alert-to-incident ratio.

**Lesson 3: Prune aggressively.** An alert rule that has not fired in 90 days is a candidate for deletion. An alert rule that has fired but never resulted in an action is a candidate for deletion. An alert rule that has a runbook that is never updated is a candidate for deletion. Alert rules are liabilities, not assets. Every rule requires maintenance, review, and tuning.

**Lesson 4: Validate with chaos.** Regular chaos engineering exercises validate that your alerting system works. When you simulate a failure, the alerting system must detect it. If it does not, you have a gap. Chaos exercises should be routine, not special. Run them at least once per quarter.

**Lesson 5: Invest in the responder experience.** The responder first interaction with an alert should be fast and productive. The alert should include: the metric value, the threshold, the duration, the runbook link, the dashboard link, the recent changes, and the recent deployments. Every second the responder spends context-switching is a second they are not fixing the problem.

**Lesson 6: Automate the common responses.** If 80 percent of incidents have the same root cause, automate the response. A runbook that says "restart the service" should be automated, not documented. The alerting system should be able to trigger a remediation workflow automatically. Manual intervention should be reserved for the 20 percent of incidents that are novel.

**Lesson 7: Test alerts in staging before production.** An untested alert rule is a liability. Create a staging environment that mirrors production traffic patterns. Deploy new alert rules to staging first. Let them run for 48 hours. Review the alerts they produce. Only promote to production after the staging review is complete.

**Lesson 8: Build an alert review into the incident postmortem.** Every postmortem should ask: "Did the alerting system perform as expected?" If the answer is no, create a follow-up task to fix the alert. This closes the loop and ensures the alerting system improves with every incident. Over six months, alert quality measurably improves as each postmortem reveals and fixes one alert gap.

**Lesson 9: Measure the mean time to acknowledge and mean time to resolve per alert rule.** These metrics tell you whether an alert is useful. An alert that consistently has a high mean time to acknowledge means responders are ignoring it. An alert that consistently has a high mean time to resolve means the runbook is missing critical steps or the problem is poorly understood. Track these metrics in a dashboard.

**Lesson 10: Do not let perfect be the enemy of good.** A alerting system that catches 80 percent of incidents with a 20 percent false-positive rate is better than no alerting system at all. Start with simple static thresholds, add dynamic thresholds as you collect data, and evolve to SLO-based alerting as your organization matures. The perfect alerting system does not exist. The good alerting system gets better every quarter.

### Stripe: Incident Response at Scale

Stripe processes billions of dollars in payments annually. Their incident response system is one of the most mature in the industry.

**The Philosophy:** Stripe treats every incident as a learning opportunity. The postmortem is not a blame exercise — it is a systems analysis. The goal is to understand why the defenses failed and what can be improved.

**The Implementation:** Stripe uses a custom incident response platform that integrates with their monitoring system, deployment pipeline, and ticketing system. When an alert fires, the platform automatically creates an incident, assigns a severity, and pages the appropriate team.

**Key Practice — The Incident Commander Model:** Stripe uses the incident commander model for incident response. The incident commander is responsible for coordinating the response, not for fixing the problem. The commander delegates investigation tasks to subject matter experts. This separation of concerns ensures that the response is coordinated even when the problem is complex.

**Key Practice — The 5 Whys:** Every postmortem at Stripe uses the 5 Whys technique. The team asks "why" five times to trace the root cause from the symptom to the fundamental issue. The 5 Whys often reveal systemic issues — insufficient testing, inadequate monitoring, poor documentation — that go beyond the immediate technical failure.

### Additional Open Source Tools

**Cortex:** An open-source alerting system that provides a unified interface for alert management. Cortex integrates with Prometheus, Grafana, and other monitoring tools. It provides a horizontal scaling layer for Prometheus, allowing the alerting system to scale beyond a single Prometheus server.

**Thanos:** An open-source system that provides a global query view and unlimited retention for Prometheus metrics. Thanos does not evaluate alert rules itself, but it provides the metrics data that the alerting engine queries. Thanos is used by organizations that need to query metrics across multiple Prometheus servers.

**KubeMon:** An open-source monitoring system for Kubernetes that provides built-in alerting rules for common Kubernetes failure modes. KubeMon is pre-configured with alerts for pod health, node health, cluster capacity, and resource exhaustion. It is a good starting point for teams that are new to Kubernetes monitoring.

### Emerging Trends: AI-Driven Alerting

**Anomaly detection as a service:** Cloud providers and third-party vendors are offering anomaly detection as a managed service. The service ingests your metrics, trains a model on your historical data, and alerts when the metrics deviate from the expected pattern. The service handles the model training, tuning, and deployment. This reduces the operational burden of running a dynamic thresholding system.

**Alert correlation with large language models:** LLMs can analyze alert text and correlate alerts based on semantic similarity. An alert about "high latency in payment service" and an alert about "payment service timeout" are semantically similar even if they have different labels and metrics. The LLM-based correlator groups these alerts together, reducing noise.

**Automated runbook generation:** LLMs can generate runbook drafts based on the alert configuration and the metric history. The generated runbook includes: the most common causes of the alert, the investigation steps, and the remediation steps. The runbook is reviewed by a human before it is published.

### Additional Cloud Provider Details

**AWS CloudWatch Composite Alarms:** CloudWatch supports composite alarms that combine multiple conditions. A composite alarm can fire when the error rate exceeds 5 percent AND the request count exceeds 1,000 per minute. Composite alarms reduce false positives by requiring multiple conditions to be met. The conditions are evaluated as individual alarms, and the composite alarm fires based on the state of the individual alarms.

**GCP Alerting Conditions:** GCP alerting policies support multiple conditions per policy. A policy can have up to 6 conditions, each with a different threshold and duration. The conditions are combined with AND or OR logic. This allows complex alerting logic without creating multiple policies.

**Azure Smart Detection:** Azure Monitor includes a smart detection feature that automatically detects anomalies in your application metrics. Smart detection uses machine learning to identify patterns that indicate a problem. It does not require manual threshold configuration. Smart detection is available for Azure Application Insights and Azure Monitor metrics.

### The Future of Alerting

**Alerting as a platform:** Organizations are moving from ad-hoc alerting to alerting platforms that provide a unified interface for all monitoring tools. The platform handles rule management, evaluation, deduplication, and notification routing. The monitoring tools are plug-ins that provide data to the platform.

**Observability-driven alerting:** The next generation of alerting systems will be driven by observability data, not just metrics. The system will correlate metrics, logs, and traces to identify the root cause of an alert. The alert will include the full observability context: the metric that triggered the alert, the log entries that preceded the alert, and the trace of the affected requests.

**Self-healing alerting:** The alerting system will not just detect problems — it will fix them. The system will have a library of automated remediation actions. When an alert fires, the system will execute the appropriate remediation action. If the remediation succeeds, the alert is resolved without human intervention. If the remediation fails, the alert is escalated to a human.

### Incident Response Best Practices from the Industry

**The 15-Minute Triage Clock:** Many organizations follow the 15-minute triage clock. From the moment an alert fires, the responder has 15 minutes to determine the severity, the impact, and the appropriate response. If the triage is not complete in 15 minutes, the incident is automatically escalated to the next level. The 15-minute clock forces the responder to make a quick assessment rather than getting lost in investigation.

**The War Room Protocol:** For critical incidents, organizations activate a war room. The war room is a dedicated communication channel (Slack, Zoom, or physical room) where the incident response team coordinates. The war room has a structured protocol: the incident commander runs the room, the communications lead handles external updates, the technical lead drives the investigation. The alerting system automatically creates the war room when a critical alert fires.

**The Postmortem Template:** A standardized postmortem template ensures consistency across incidents. The template includes: incident summary, timeline, root cause, impact assessment, action items, and follow-up. The postmortem is written within 48 hours of the incident resolution. The postmortem is reviewed by the team and the action items are tracked in the project management system.

### The Alerting Playbook

A playbook is a collection of runbooks organized by scenario. Unlike a runbook, which covers a specific alert, a playbook covers a scenario that might involve multiple alerts. The playbook is the operational guide for the on-call engineer.

**Playbook structure:** Each playbook has: the scenario description, the expected alerts, the investigation steps, the remediation steps, and the escalation criteria. The playbook is stored in a version-controlled repository and reviewed quarterly.

**Playbook examples:**
- Playbook for "Database slow query scenario": alerts from database proxy, expected alerts from dependent services, investigation steps (check slow query log, check connection pool, check index usage), remediation steps (kill the slow query, add an index, increase pool size), escalation criteria (if the query kills the database, escalate to the DBA team).
- Playbook for "Deployment failure scenario": alerts from the deployment pipeline, expected alerts from the service, investigation steps (check the deployment logs, check the service health, check the deployment diff), remediation steps (roll back the deployment, fix the deployment configuration, re-deploy), escalation criteria (if the rollback fails, escalate to the platform team).

### Quantitative Alert Performance Metrics

Track these metrics to measure the alerting system performance over time. Each metric is tracked per alert rule and aggregated across the system.

**Alert firing rate:** The number of times each alert fires per day. A high firing rate indicates a noisy alert. A low firing rate indicates a well-tuned alert or an alert that is never triggered.

**Alert-to-incident ratio:** The number of alerts that result in a documented incident divided by the total number of alerts. A ratio above 20 percent is healthy. A ratio below 10 percent indicates noise.

**Time to acknowledge (TTA):** The time from alert firing to the first acknowledgment. A low TTA (under 5 minutes) indicates that the alert is reaching the right person quickly. A high TTA (over 30 minutes) indicates that the alert is not reaching the right person or that the responder is ignoring it.

**Time to resolve (TTR):** The time from alert firing to the resolution. A low TTR (under 30 minutes) indicates that the alert is actionable and the runbook is effective. A high TTR (over 4 hours) indicates that the alert is complex or the runbook is missing critical steps.

**False positive rate:** The percentage of alert firings that are false positives. A false positive rate under 10 percent is healthy. A rate over 50 percent requires immediate intervention.

**Missed detection rate:** The number of incidents that were not detected by any alert divided by the total number of incidents. A missed detection rate under 5 percent is healthy. A rate over 20 percent indicates a significant gap in the alerting coverage.

### Building an Alerting Culture

An alerting culture is the organizational mindset that treats alerting as a first-class engineering discipline. Building this culture requires:

**Leadership commitment:** Engineering leadership must commit to the alerting culture. The commitment includes: allocating time for alert maintenance, supporting the on-call rotation, and treating alert fatigue as a critical issue. Without leadership commitment, the alerting system will degrade over time.

**Training and onboarding:** Every new engineer should be trained on the alerting system. The training covers: how to respond to alerts, how to write runbooks, how to create and tune alert rules, and how to participate in the on-call rotation. The training is part of the onboarding process.

**Recognition and rewards:** The on-call role should be recognized and rewarded. Engineers who respond to critical incidents, improve the alerting system, or write excellent runbooks should be recognized. The recognition can be formal (bonus, promotion) or informal (shout-out in the team meeting).

**Continuous improvement:** The alerting system is never finished. It must be continuously improved. The improvement cycle is: create an alert, observe it for 30 days, tune it based on the observation, and review it quarterly. The cycle is automated where possible.

**Postmortem-driven improvement:** Every postmortem should produce at least one alerting improvement. The improvement might be: a new alert rule, a threshold adjustment, a runbook update, or a notification channel change. The improvement is tracked in the project management system and followed up in the next postmortem.

### The Incident Response Drill

Regular incident response drills keep the team sharp. The drill simulates a real incident without affecting production systems.

**Drill structure:** The drill starts with a simulated alert. The on-call engineer responds as if the alert were real. The engineer follows the runbook, investigates the simulated problem, and declares the incident resolved. The drill is timed. The team reviews the drill performance and identifies areas for improvement.

**Drill frequency:** The team runs a drill once per month. The drill scenarios are rotated. Each drill covers a different failure mode. The drill scenarios are designed by the team lead or the incident commander.

**Drill scenarios:**
- Scenario 1: A database service is down. The team must detect the outage, investigate the root cause, and restore the service.
- Scenario 2: A deployment causes a performance regression. The team must detect the regression, roll back the deployment, and fix the configuration.
- Scenario 3: A security breach is detected. The team must contain the breach, investigate the impact, and restore the service.
- Scenario 4: A third-party API is unavailable. The team must detect the dependency failure, mitigate the impact, and communicate with the third-party provider.

### The Alerting System Maturity Assessment

Use this assessment to evaluate your alerting system maturity. Score each area from 1 (immature) to 5 (mature).

**Alert creation (score 1-5):** 1 = alerts are created ad-hoc, no templates, no review. 3 = alerts are created from templates, reviewed by a second person. 5 = alerts are created from templates, reviewed, tested in staging, and deployed through CI/CD.

**Alert tuning (score 1-5):** 1 = thresholds are set once and never changed. 3 = thresholds are reviewed quarterly. 5 = thresholds are tuned automatically based on historical data.

**Runbook quality (score 1-5):** 1 = no runbooks exist. 3 = runbooks exist for critical alerts. 5 = runbooks exist for all alerts, are reviewed quarterly, and include automated remediation steps.

**On-call process (score 1-5):** 1 = no on-call rotation, the same person is always on-call. 3 = on-call rotation with 4 engineers, weekly rotation. 5 = on-call rotation with 8 engineers, weekly rotation, automated scheduling, and escalation.

**Postmortem quality (score 1-5):** 1 = no postmortems. 3 = postmortems are written for critical incidents. 5 = postmortems are written for all incidents, reviewed by the team, and action items are tracked.

**Total score (5-25):** 5-10 = immature system, needs significant improvement. 11-15 = developing system, making progress. 16-20 = mature system, well-maintained. 21-25 = world-class system, industry-leading.

### Common Pitfalls in Real-World Alerting

**Pitfall 1: Alerting on every deployment.** Every deployment generates alerts. The deployment causes a brief spike in latency, a brief increase in error rate, and a brief decrease in throughput. The team learns to ignore these alerts. The fix: silence alerts during the deployment window. Only alert if the deployment causes a sustained degradation.

**Pitfall 2: Alerting on every dependency.** The service depends on 10 other services. Each dependency has its own alert. When one dependency fails, 10 alerts fire. The team learns to ignore these alerts. The fix: use hierarchical alerting. Alert on the dependency failure, not on the dependent service degradation.

**Pitfall 3: Alerting on every metric.** The service exposes 100 metrics. Each metric has an alert. The team cannot keep up with the alert volume. The fix: only alert on the 4 golden signals. Add custom alerts only for service-specific failure modes.

**Pitfall 4: Alerting on every error.** Every error is treated as an alert. The team is paged for every 500 error, every timeout, every exception. The fix: aggregate errors. Alert on the error rate, not on individual errors. Use the duration window to confirm the error rate is sustained.

**Pitfall 5: Alerting on every threshold breach.** Every threshold breach is treated as an alert. The threshold is set too low, so the alert fires constantly. The fix: set the threshold based on historical data. Use the p99 value of the metric during normal operation. The threshold should be high enough that the alert only fires during abnormal conditions.

### The On-Call Experience Survey

Regularly survey the on-call engineers to measure the alerting system effectiveness. The survey asks:

1. How many alerts did you receive during your on-call shift? (0-10, 11-20, 21-50, 50+)
2. How many alerts were actionable? (All, Most, Some, Few, None)
3. How many alerts had a useful runbook? (All, Most, Some, Few, None)
4. How long did it take to investigate the average alert? (Under 5 minutes, 5-15 minutes, 15-30 minutes, 30+ minutes)
5. How many alerts were false positives? (0 percent, 1-25 percent, 26-50 percent, 51-75 percent, 76-100 percent)
6. How many alerts did you ignore? (0, 1-3, 4-10, 10+)
7. How many times were you paged at night? (0, 1-3, 4-10, 10+)
8. How would you rate the overall on-call experience? (Excellent, Good, Fair, Poor, Terrible)

The survey results are used to identify the most problematic alerts. The alerts with the highest false-positive rate and the lowest runbook quality are the first candidates for deletion or tuning.
