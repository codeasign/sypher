## Tier 1: Foundation

### Challenge 1: The Threshold Tuning Problem

Your team has a critical alert on the payment service p99 latency. The threshold is 500ms with a 5-minute duration. Over the past 30 days, the alert fired 12 times. Each time, the on-call engineer investigated and found that the latency spike was caused by a scheduled database backup that runs every 4 hours and takes approximately 3 minutes.

**Your task:** Propose three changes to reduce false positives without increasing the risk of missing a real latency incident.

**Model answer:**

Change 1 — Silence the alert during the backup window. The backup runs at a known schedule: every 4 hours starting at midnight, lasting approximately 3 minutes. Add a silence rule that suppresses the payment latency alert from 0-5 minutes past each backup start time. This eliminates the 12 false positives entirely without changing the threshold or duration.

Change 2 — If silencing is not possible because the backup timing varies, increase the duration from 5 minutes to 6 minutes. The backup lasts 3 minutes, so a 5-minute duration is too short to wait it out. A 6-minute duration ensures the backup completes before the alert fires. The trade-off is that a real latency incident would take 1 minute longer to detect.

Change 3 — Add a second condition to the alert rule. Instead of alerting on p99 latency alone, require both high latency AND a non-zero error rate. During the backup, latency is high but errors are zero. A real incident would typically have both high latency and errors. This improves specificity without sacrificing sensitivity for real incidents.

### Challenge 2: The Alert Storm Scenario

Your database cluster experiences a 5-second network partition. During those 5 seconds, 500 connections timeout. After the partition ends, all connections succeed. However, the 500 timeouts trigger the following alerts:

- 15 high-latency alerts from the database proxy (one per endpoint)
- 30 error-rate alerts from application services (one per service)
- 5 alerts from dependent services (queue, cache, search)
- 50 separate Slack messages to the #incidents channel

**Your task:** Design a deduplication and grouping strategy that would have collapsed these 50 alerts into a single incident.

**Model answer:**

The strategy has three components:

Component 1 — Time-window deduplication. Alerts that fire within the same 60-second window are candidates for grouping. In this scenario, all 50 alerts fire within seconds of each other, so they are all in the same window.

Component 2 — Root-cause correlation. The alert manager maintains a dependency graph. When the database proxy fires an alert, the alert manager checks which services depend on the database. Those service alerts are automatically grouped under the database alert as related alerts. The grouping creates a hierarchy: the database proxy alert is the parent, the application service alerts are children, and the dependent service alerts are grandchildren.

Component 3 — Smart notification. Instead of sending 50 notifications, the system sends one notification: "Database partition detected — 50 related alerts grouped into this incident." The notification includes a link to the incident page where the responder can see all related alerts. The responder acknowledges one incident, not 50. When the database alert resolves, all dependent alerts are automatically resolved.

### Challenge 3: The Missing Alert

Your team discovers that a critical service has been silently failing for 4 hours. The service error rate increased from 0.1 percent to 15 percent at 2 PM, and a user finally reported the issue at 6 PM. You check the alerting system and find that no alert fired.

**Your task:** Identify three possible reasons the alert did not fire and propose a fix for each.

**Model answer:**

Reason 1 — The alert rule was never created. The service was deployed without any alert rules. This is common when a new service is launched in a hurry. Fix: Add a deployment checklist that requires alert rules to be configured before a service is allowed to serve production traffic. The deployment pipeline should reject a deployment that has no alert rules.

Reason 2 — The alert rule existed but was silenced. Someone silenced the alert for maintenance and forgot to unsilence. Fix: Add a maximum silence duration. Auto-unsilence any alert rule after 24 hours of being silenced. Require a ticket number for every silence. The ticket triggers a reminder to unsilence.

Reason 3 — The threshold was set too high. The alert rule exists and was not silenced, but the threshold was set at 20 percent error rate. The error rate was 15 percent, so the alert never fired. Fix: Use SLO-based burn rate alerting instead of static thresholds. Set the burn rate alert to fire if the error budget burns faster than 2 percent per hour. This catches degradations long before they reach the static threshold.

### Challenge 4: The Flapping Alert Fix

Your team has a warning alert on the search service p50 latency. The threshold is 200ms with a 2-minute duration. The alert fires and resolves approximately every 5 minutes. The latency oscillates between 180ms and 220ms, crossing the threshold with each oscillation. The on-call engineer has started ignoring the alert.

**Your task:** Design a fix using hysteresis and explain the mathematics of the dead zone.

**Model answer:**

The problem is that the threshold is too close to the metric normal operating range. The latency oscillates naturally around the 200ms threshold, causing the alert to flap.

The fix is to introduce hysteresis. Instead of a single threshold, use two thresholds:
- Firing threshold: 200ms (alert fires when exceeded)
- Resolution threshold: 150ms (alert resolves when drops below)

The dead zone between 150ms and 200ms prevents flapping. When the latency exceeds 200ms, the alert fires. When the latency drops below 200ms but stays above 150ms, the alert continues firing. The alert only resolves when the latency drops below 150ms, which is clearly below the normal fluctuation range.

The dead zone width should be at least 2 standard deviations of the metric normal variation. If the latency standard deviation is 20ms, the dead zone should be 40ms. The resolution threshold is firing threshold minus 2 sigma.

## Tier 2: Design

### Challenge 5: Design a Multi-Tenant Alerting System

Your company provides a monitoring platform as a SaaS product. Each customer has their own services, metrics, and alert rules. Customers should only see their own alerts. One customer alert storm must not affect another customer alert delivery.

**Your task:** Design a multi-tenant alerting architecture that isolates tenants while sharing infrastructure.

**Model answer:**

The architecture uses three isolation strategies:

Tenant-level sharding — Each tenant is assigned to a specific evaluator shard. All rules for tenant A go to shard 1, all rules for tenant B go to shard 2. This prevents a noisy tenant from consuming CPU or memory on another tenant shard. The assignment is managed by a tenancy router that reads the tenant ID from the rule metadata.

Resource quotas — Each tenant has a maximum number of alert rules, a maximum evaluation frequency, and a maximum notification rate. If a tenant exceeds their quota, new rule creation is blocked and existing rules are de-prioritized. Quotas prevent a single tenant from overwhelming the system.

Tenant-level rate limiting in the notification router. If tenant A triggers 1,000 alerts in 5 minutes, the notification router queues tenant A notifications and delivers them at a controlled rate. Tenant B notifications are not affected. The rate limit is configurable per tenant.

Database-level isolation uses separate database schemas per tenant. Each tenant rule configuration and alert state are stored in separate tables. This simplifies data management and prevents cross-tenant data leaks.

### Challenge 6: Design a SLO-Based Alerting System

Your company wants to move from static threshold alerting to SLO-based burn rate alerting. Each service has an SLO (for example, 99.9 percent uptime, 200ms p99 latency). You need to design an alerting system that alerts when the error budget is burning too fast.

**Your task:** Design the burn rate alerting algorithm and the notification strategy.

**Model answer:**

The burn rate algorithm compares the actual error rate against the SLO budget.

The SLO for a service is 99.9 percent availability over a 30-day window. This means the error budget is 0.1 percent of total requests — 43 minutes of downtime per 30 days.

The burn rate is the ratio of the actual error rate to the maximum allowed error rate. If the actual error rate is 1 percent and the SLO allows 0.1 percent, the burn rate is 10. This means the error budget is being consumed 10 times faster than expected.

The alerting strategy uses two burn rate windows:

A fast burn rate window (1 hour). If the error rate exceeds the SLO budget consumption rate by 10x or more over the last hour, a critical alert fires. This catches severe outages quickly.

A slow burn rate window (6 hours). If the error rate exceeds the SLO budget consumption rate by 2x or more over the last 6 hours, a warning alert fires. This catches gradual degradations that would not trigger the fast burn rate alert.

When a burn rate alert fires, the notification includes: the current burn rate, the remaining error budget in hours, the expected time until the error budget is exhausted at the current burn rate, and a link to the SLO dashboard.

Escalation is based on the remaining error budget. If the error budget is 50 percent consumed, a warning goes to the team Slack. If 80 percent consumed, a critical alert pages the on-call. If 100 percent consumed, the incident is escalated to the engineering manager.

### Challenge 7: Design an On-Call Escalation System

Your alerting system needs to route alerts to the right person based on time of day, day of week, and skill level. You need to support multiple teams, each with their own on-call schedule.

**Your task:** Design the on-call scheduling and escalation system.

**Model answer:**

The system has four components:

Schedule management — Each team defines a schedule: "primary on-call is person A from 9 AM to 5 PM weekdays, person B from 5 PM to 9 AM weekdays, person C on weekends." Schedules are stored in a calendar database. Schedules can be overridden for vacations, holidays, or special events. Overrides are managed through the calendar API.

Contact information — Each engineer provides their contact methods: PagerDuty push, SMS, and phone call. Each contact method has a priority. The system tries PagerDuty push first, then SMS, then phone call. If all contact methods fail for the primary, escalation triggers.

Escalation chain — Each team defines a chain: primary on-call to secondary on-call to team lead to engineering manager. Each level has an acknowledgement timeout: 5 minutes for primary, 10 minutes for secondary, 15 minutes for team lead, 30 minutes for engineering manager. The timer starts when the alert fires. If the primary does not acknowledge within 5 minutes, the alert escalates to the secondary and sends a new notification.

Global fallback — If the full escalation chain is exhausted, the alert is routed to a global incident response team. The global team handles critical alerts that no team has acknowledged. The global team is the last resort before a customer-impacting outage goes unaddressed.

## Tier 3: Expert

### Challenge 8: Design Alert Correlation for a Service Mesh

Your company runs 500 microservices in a service mesh (Istio). Each service communicates with an average of 5 other services. When a downstream service fails, it causes cascading failures that trigger alerts in upstream services. You need to distinguish between root-cause alerts and symptom alerts.

**Your task:** Design an alert correlation system that identifies the root cause of cascading failures in a service mesh.

**Discussion:**

This is one of the hardest problems in alerting at scale. The key insight is that in a service mesh, the direction of failure propagation is opposite to the direction of the request flow. When service A calls service B and B is slow, A becomes slow too. The alert in service A is a symptom. The alert in service B is the root cause.

The correlation system must understand the service dependency graph. The graph is dynamic — dependencies change with each deployment, each configuration change, each feature flag toggle. The system must query the service mesh control plane to get the current dependency graph.

The algorithm works in three phases:

Phase 1 — Detection. When alerts fire, the system records the alert time, the service, and the metric. Alerts are grouped into windows. A window is 5 minutes. All alerts in the same window are candidates for correlation.

Phase 2 — Walk the dependency graph. For each alert in the window, the system walks the dependency graph downstream. If service A has alerts and service B (downstream) has alerts, service B is a candidate for root cause. If service B also depends on service C and C has alerts, C is a stronger candidate. The system produces a ranked list of candidate root causes based on the depth in the dependency graph.

Phase 3 — Validation. The system validates the candidate root causes by checking whether the downstream service failure mode is consistent with the upstream service alerts. If service B has a database connection pool exhaustion and service A has timeout errors, the correlation is consistent. If service B has high CPU and service A has timeout errors, the correlation is less clear — high CPU should cause latency, not necessarily timeouts.

The notification for a correlated incident shows one root-cause alert and N symptom alerts. The responder is told: "Service B has a database connection pool exhaustion. This is affecting 15 upstream services. The upstream service alerts are symptoms of this root cause."

The challenge is that correlation takes time. You cannot wait for all alerts in the window before notifying. The system should notify immediately with the first alert and update the notification as more alerts arrive and the correlation analysis completes. The notification is a living document that evolves as the system learns more.

### Challenge 9: Design Predictive Alerting

Your system currently alerts when problems have already happened. Your company wants you to design a predictive alerting system that alerts when problems are about to happen — before they impact users.

**Your task:** Design a system that predicts impending failures and alerts proactively.

**Discussion:**

Predictive alerting is the frontier of the field. The fundamental approach is to detect early indicators — micro-patterns that precede a failure by minutes or hours.

**Approach 1 — Trend-based prediction.** Some metrics have leading indicators. Memory usage trends upward before an OutOfMemoryError. Request latency trends upward before a connection pool exhaustion. The predictive system tracks the rate of change of each metric. If the rate of change has been positive for N consecutive evaluation cycles, the system predicts a threshold breach within T minutes. Example: if memory usage has increased by 5 percent per hour for 4 consecutive hours, and the threshold is 90 percent, the system predicts a breach in 2 hours.

**Approach 2 — Correlation-based prediction.** Certain event patterns precede failures. A deployment combined with an increase in garbage collection pauses precedes an OutOfMemoryError. An increase in log ERROR messages from service A combined with a decrease in cache hit rate for service B precedes a cascading failure. The predictive system learns these correlations from historical incident data.

**Approach 3 — ML-based anomaly detection.** Train a model on historical metrics to predict the expected value of each metric at each point in time. When the actual value deviates from the predicted value, the system evaluates whether the deviation is likely to lead to a failure. The model is trained on labeled data — metric patterns that preceded incidents. The challenge is that training data is sparse (incidents are rare) and the failure modes change as the system evolves.

**Practical considerations:**
- Predictive alerts should have a lower severity than reactive alerts. A warning that a problem might happen in 2 hours is different from a critical alert that a problem is happening now.
- Predictive alerts must include the confidence level. "Memory is trending upward and is predicted to breach 90 percent in approximately 2 hours (90 percent confidence)." This helps the responder prioritize.
- Predictive alerts should suggest preventive actions. "Consider increasing memory allocation or investigating the memory leak in service B." The runbook should include both reactive and preventive steps.
- Start with the simplest approach (trend-based prediction) and add complexity only when the simple approach misses too many incidents. Trend-based prediction catches 60-70 percent of impending failures with minimal engineering investment.
- Validate the predictive model against historical incidents. If the model would have predicted 3 out of the last 5 incidents, it is useful. If it would have predicted none, or if it produces too many false positives, go back to trend-based prediction.

### Challenge 10: Design an Alerting System for a Multi-Cloud Deployment

Your company runs services across AWS, GCP, and Azure. Each cloud provider has its own monitoring and alerting tools. You need a unified alerting system that works across all three clouds.

**Your task:** Design a multi-cloud alerting architecture that handles provider-specific metrics, cross-provider dependencies, and provider outages.

**Discussion:**

Multi-cloud alerting adds complexity because each provider has different metrics, different APIs, and different availability characteristics. The solution is a provider-agnostic abstraction layer.

**Abstraction layer:** Define a common metric format that all providers can map to. The format includes: metric name, value, timestamp, labels, and source provider. Each provider has a connector that translates provider-specific metrics into the common format. The connector runs as a sidecar or agent on the provider infrastructure.

**Cross-provider dependencies:** Services running on AWS might depend on services running on GCP. The alerting system must understand these cross-provider dependencies. The dependency graph includes nodes from all three providers. The correlation algorithm walks the dependency graph across provider boundaries.

**Provider outage handling:** If a provider experiences an outage, the alerting system must detect it and adjust. The alerting system should not fire alerts for services that are down because of the provider outage. The solution is to detect provider-level anomalies and suppress alerts for dependent services.

**Provider failover:** If the alerting system is running on a provider that goes down, the system must fail over to another provider. The alerting system is deployed in a multi-region, multi-provider configuration. Each instance of the alerting system monitors the other instances. If one instance is unreachable, the other instances take over its workload.

### Challenge 11: Design Alerting for a Real-Time Streaming Platform

Your company operates a real-time video streaming platform with 100 million monthly active users. The platform processes 10 TB of streaming data per second. Latency is measured in milliseconds. Any degradation in the streaming pipeline causes immediate user impact.

**Your task:** Design an alerting system that can detect streaming pipeline degradation within 5 seconds.

**Discussion:**

Real-time streaming alerting requires a fundamentally different architecture. The evaluation interval is 5 seconds, not 5 minutes. The data volume is 10 TB per second, not 10 GB per hour.

**Streaming evaluation engine:** The alerting engine is embedded in the stream processing pipeline. Instead of polling a metrics database, the engine evaluates rules inline as each data point passes through the pipeline. The evaluation is a side-effect of the stream processing — it does not add additional latency.

**Sliding window evaluation:** The evaluation uses a sliding window of 5 seconds. The window is maintained in memory by the stream processor. Each window tracks the count, sum, min, max, and histogram for each metric. The alerting rule is evaluated against the window contents.

**Hierarchical evaluation:** The evaluation is hierarchical. The first level evaluates individual streams. The second level evaluates aggregate streams (all streams from a region, all streams from a content type). The third level evaluates global aggregates. An alert at a higher level suppresses alerts at lower levels.

**Notification latency:** The notification latency must be under 5 seconds. The notification router is embedded in the stream processing pipeline. The route is pre-computed — the alert rule specifies the notification channel at configuration time, so the router does not need to look up the channel at runtime.

### Challenge 12: Design an Alert Audit and Compliance System

Your company is subject to regulatory compliance requirements (SOC 2, HIPAA, PCI-DSS). The compliance requirements specify that certain alerts must be created, must have specific thresholds, and must be reviewed quarterly.

**Your task:** Design an alert audit system that validates compliance requirements and reports on alert configuration health.

**Discussion:**

Compliance alerting is about proving that the alerting system exists and works, not about the alerts themselves. The audit system must provide evidence that the required alerts are configured and have been tested.

**Compliance rule templates:** Each compliance requirement maps to a set of alert rule templates. The template specifies the metric, the threshold, the duration, and the notification channel. The audit system checks that the required alerts exist and match the template.

**Alert configuration validation:** The audit system validates every alert rule against the compliance requirements. The validation checks: the rule exists, the threshold is within the required range, the duration is set, the notification channel is configured, the runbook exists, and the rule has been reviewed in the last 90 days.

**Alert testing evidence:** The audit system records the results of periodic alert testing. The testing process simulates the alert condition and verifies that the alert fires and the notification is delivered. The test results are stored as evidence for the compliance audit.

**Alert review tracking:** The audit system tracks the review status of every alert rule. A review is required every 90 days. The review includes: the alert performance (how many times it fired, how many resulted in incidents), the runbook accuracy, and the threshold appropriateness. The review results are stored for the compliance audit.

### Challenge 13: Design Alerting for a Serverless Architecture

Your company is migrating to a serverless architecture using AWS Lambda. Each function is ephemeral — it runs for milliseconds to seconds and then shuts down. Traditional metrics collection (CPU, memory, disk) does not work because the functions do not have persistent monitoring agents.

**Your task:** Design an alerting system for a serverless architecture with 500 Lambda functions, each invoked 10,000 times per minute.

**Discussion:**

Serverless alerting requires a fundamentally different approach. You cannot collect metrics from the function host because the host does not exist when the function is not running. Instead, you must collect metrics from the function output and the platform logs.

**Approach 1 — Cold start tracking.** Lambda cold starts occur when a new instance is created. Cold starts add 500-2000ms to the function execution time. A sudden increase in cold start rate indicates a scaling event or a platform issue. Track the cold start rate and alert if it exceeds 5 percent of total invocations.

**Approach 2 — Error rate tracking.** Monitor the function error rate from CloudWatch Logs. Alert if the error rate exceeds 1 percent over a 5-minute window. The error rate is the most reliable signal for serverless alerting because it directly measures function health.

**Approach 3 — Duration tracking.** Track the p50, p95, and p99 function execution duration. Alert if the p95 duration exceeds the function timeout by more than 50 percent. A sudden increase in duration indicates a performance regression or a downstream dependency issue.

**Approach 4 — Throttle tracking.** Lambda throttles functions when concurrency limits are reached. Track the throttle count and alert if it exceeds 0.1 percent of total invocations. Throttling directly impacts user experience.

**Approach 5 — Cost tracking.** In serverless, cost is a proxy for health. A sudden increase in function invocations or duration directly increases cost. Track the cost per function per hour. Alert if the cost exceeds the baseline by 2x.

### Challenge 16: Design an Alerting System for a Video Streaming Platform

Your company operates a video streaming platform with 200 million monthly active users. The platform delivers 10 PB of video content per day. The user experience is measured in milliseconds of buffering, seconds of startup time, and percentage of successful playback.

**Your task:** Design an alerting system that detects video streaming quality degradation within 10 seconds and isolates the root cause.

**Discussion:**

Video streaming alerting focuses on the user experience, not the infrastructure. The key metrics are: video startup time, buffering rate, playback failure rate, and video quality (resolution).

**User experience metrics:** The alerting system tracks the user experience metrics from the client-side. The metrics are collected by the video player and sent to the monitoring system. The metrics include: the time to first frame, the number of buffering events, the duration of buffering events, the playback resolution, and the playback errors.

**CDN health metrics:** The alerting system tracks the CDN health metrics. The metrics include: the edge node latency, the cache hit rate, the origin server load, and the throughput per edge node. The CDN metrics are collected from the CDN provider API.

**Content delivery metrics:** The alerting system tracks the content delivery metrics. The metrics include: the encoding latency, the packaging latency, the encryption latency, and the delivery latency. The content delivery metrics are collected from the content processing pipeline.

**Correlation analysis:** The alerting system correlates the user experience metrics with the CDN and content delivery metrics. If the user experience degrades in a specific region, the system checks the CDN health in that region. If the CDN is healthy, the system checks the content delivery pipeline. The correlation analysis narrows the root cause to a specific component.

**The alerting thresholds:** The thresholds are based on the user experience. The video startup time must be under 2 seconds. The buffering rate must be under 1 percent of playback time. The playback failure rate must be under 0.1 percent. The video quality must be at least 720p for 90 percent of users. The thresholds are reviewed monthly and adjusted based on the user feedback.

**The notification strategy:** The alerts are routed to the appropriate team. The video quality team handles the user experience alerts. The CDN team handles the CDN health alerts. The content delivery team handles the content delivery alerts. The escalation is to the platform engineering team if the alert is not resolved within 15 minutes.

IoT alerting has unique challenges. The devices are remote, the connectivity is unreliable, and the data volume is high. The alerting system must distinguish between device-level issues, connectivity issues, and platform-level issues.

**Platform health metrics:** The alerting system tracks the platform health metrics. The metrics include: the message throughput, the processing latency, the error rate, the queue depth, and the resource utilization. The platform health metrics are collected from the platform infrastructure.

**Connectivity loss detection:** The alerting system detects device connectivity loss. If a device does not send a telemetry message for 10 minutes, the system sends an alert. The alert is escalated if the device is critical (medical device, industrial sensor). The escalation is to the field service team.

**Data quality loss detection:** The alerting system detects data quality issues. If the message rate drops by 50 percent, the system sends an alert. If the data completeness drops below 90 percent, the system sends an alert. The alert is escalated to the data engineering team.

**The notification strategy:** The alerts are routed to the appropriate team. The device team handles the device health alerts. The data team handles the data quality alerts. The platform team handles the platform health alerts. The escalation is to the IoT engineering team if the alert is not resolved within 30 minutes.

### Challenge 14: Design an Alerting System for a Financial Trading Platform

Your company operates a financial trading platform that processes 1 million transactions per second. The platform must be available 24/7 with 99.9999 percent uptime. Any alerting delay can result in financial losses of $1 million per minute.

**Your task:** Design an alerting system that detects trading platform degradation within 1 second and notifies the on-call team within 5 seconds.

**Discussion:**

Financial trading alerting requires fundamentally different design decisions. The evaluation interval is 1 second. The notification latency is 5 seconds. False positives are expensive but false negatives are catastrophic.

**Hardware-level alerting:** The first layer of alerting runs on the trading hardware itself. The FPGA or ASIC that processes trades runs a hardware-level alerting module. The module monitors: the transaction latency, the throughput, the error rate, and the power consumption. The module triggers an alert if any metric exceeds the threshold. The alert is sent to the monitoring system through a dedicated network path.

**Software-level alerting:** The trading application exposes a set of metrics: the order book depth, the spread, the match rate, the rejection rate, and the latency. The alerting engine evaluates these metrics every 1 second. The evaluation engine runs on the same hardware as the trading application to minimize latency.

**Redundant alerting paths:** The alerting system has two independent paths. The primary path is the software-level alerting engine. The secondary path is the hardware-level monitoring module. If the primary path fails, the secondary path takes over. The failover is automatic and takes under 100ms.

**The on-call response:** The on-call team must respond within 30 seconds. The team is on-site (not remote) and is always within 100 meters of the trading hardware. The team has a dedicated escalation path that bypasses the standard notification system. The escalation is direct to the team lead, the engineering manager, and the executive team.

**The cost of false positives:** A false positive in a trading system costs $1 million per minute in lost trading opportunity. The alerting system must have a false-positive rate under 0.01 percent. The solution is to require confirmation from multiple independent sources. An alert about transaction latency must be confirmed by the hardware module, the software module, and a third-party monitoring service before it is considered a real alert.

### Challenge 15: Design an Alerting System for a Healthcare Platform

Your company operates a healthcare platform that monitors 10,000 ICU patients in real-time. The platform must detect patient deterioration within 30 seconds and alert the medical staff. The platform is subject to HIPAA compliance requirements.

**Your task:** Design an alerting system that meets the medical requirements and the compliance requirements.

**Discussion:**

Healthcare alerting has unique requirements. The alerts are about patient health, not system health. The alerting system must be certified by regulatory bodies (FDA, HIPAA). The alerting system must be available 24/7 with 99.999 percent uptime.

**Patient monitoring:** The platform collects vital signs from each patient: heart rate, blood pressure, oxygen saturation, temperature, and respiratory rate. The monitoring system evaluates the vital signs against clinical thresholds. The thresholds are defined by the medical team and are specific to each patient and each condition.

**Clinical alerting rules:** Each patient has a set of clinical alerting rules. The rules are defined by the attending physician. The rules include: vital sign thresholds (heart rate above 120 beats per minute), trend detection (blood pressure dropping by 20 percent in 30 minutes), and combination detection (high heart rate AND low blood pressure). The rules are updated as the patient condition changes.

**Medical staff notification:** The alert is sent to the medical staff through a dedicated notification system. The notification goes to the attending physician, the nurse, and the critical care team. The notification includes the patient information, the vital signs, the trend data, and the recommended intervention.

**Compliance requirements:** The alerting system must comply with HIPAA requirements. The compliance requirements include: audit logging of all alert activity, encryption of all patient data, access control for medical staff, and data retention for 7 years. The alerting system is certified by the FDA as a medical device.

**The alerting redundancy:** The healthcare alerting system has triple redundancy. The primary system runs on the hospital network. The secondary system runs on a cloud provider. The tertiary system runs on a separate cloud provider. The three systems are synchronized. If the primary system fails, the secondary system takes over within 1 second. If the secondary system fails, the tertiary system takes over within 1 second.

### Challenge 16: Design an Alerting System for a CDN (Content Delivery Network)

Your company operates a CDN with 500 edge nodes distributed across 100 cities worldwide. The CDN delivers 50 TB of content per second. The CDN must be available 24/7 with 99.999 percent uptime.

**Your task:** Design an alerting system that detects edge node failures, cache issues, and origin server issues.

**Discussion:**

CDN alerting focuses on the content delivery performance. The key metrics are: edge node health, cache hit rate, origin server load, and throughput per edge node.

**Edge node health metrics:** The alerting system tracks the edge node health metrics.
