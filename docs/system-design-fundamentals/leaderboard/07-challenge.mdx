### Challenge 1.1: The Redis Sorted Set Calculator

You are designing a leaderboard for a mobile game. The game has 1 million active players in the daily window. Each player ID is 16 bytes. You are using a Redis sorted set to store the leaderboard.

**Question:** Calculate the approximate Redis memory required for this leaderboard. Then calculate the memory required for 10 million players. Explain the assumptions behind your calculation.

<details>
<summary>Model Answer</summary>

Each entry in a Redis sorted set consumes memory for:
- The skip list node: approximately 32 bytes (forward pointers at multiple levels, typically up to 32 levels for large sets)
- The member (player ID): 16 bytes
- The score (double): 8 bytes
- The hash map entry for O(1) member lookup: approximately 8 bytes (pointer plus hash)
- Total per entry: approximately 64 bytes

For 1 million players:
- 1,000,000 x 64 bytes = 64 MB

For 10 million players:
- 10,000,000 x 64 bytes = 640 MB

Additional overhead:
- Redis key overhead: negligible (approximately 100 bytes per key)
- Redis internal fragmentation: approximately 10-20 percent overhead depending on the allocator
- Total with fragmentation for 10 million: approximately 700-770 MB

The calculation shows that a single Redis instance with 1 GB of memory can comfortably handle 10 million players. This is why Redis sorted sets are the default choice for leaderboard systems at this scale. A `cache.r6g.xlarge` instance with 32 GB of memory can handle over 100 million players in a single sorted set.

</details>

### Challenge 1.2: Cache Invalidation Strategy

Your leaderboard system uses a Redis sorted set with a cache for the top-100 leaderboard. The cache is invalidated on every score write. The system handles 50,000 score writes per second. Only 2 percent of those scores are high enough to enter the top 100.

**Question:** Calculate the current cache hit rate. Then propose a change that improves the cache hit rate to over 90 percent. Show the math.

<details>
<summary>Model Answer</summary>

**Current state:**
- Cache invalidated on every write: 50,000 invalidations per second
- Cache TTL: 30 seconds
- Cache rebuild time: 5ms
- With 50,000 invalidations per second, the average time between invalidations is 0.02ms

The cache is invalidated approximately every 0.02ms on average. Since the cache TTL is 30 seconds but the cache is invalidated long before the TTL expires, the cache is effectively always being rebuilt. During each rebuild window of 5ms, the cache is empty and all reads hit the sorted set.

**Hit rate calculation:**
- The cache is valid for approximately 5ms out of every 30 seconds (the rebuild time is 5ms, then it is immediately invalidated again)
- Wait -- this is not right. Let me recalculate.

Actually, the cache is invalidated 50,000 times per second. Each invalidation triggers a cache rebuild that takes 5ms to complete (reading from the sorted set and writing to the cache). So the cache is empty for 50,000 x 5ms = 250,000ms per second, or 250 seconds out of every 1 second. This is impossible -- the cache rebuilds are queuing up.

The correct calculation: The system can handle at most 1,000 cache rebuilds per second (each rebuild takes 1ms of processing time, assuming the sorted set read is fast). At 50,000 invalidations per second, the invalidations queue up. The cache is never fully populated because it is invalidated faster than it can be rebuilt.

**Cache hit rate:** Approximately 0 percent. Every read misses because the cache is always in the process of being rebuilt.

**Proposed change: Conditional cache invalidation.**

Only invalidate the cache when the new score exceeds the current top-100 minimum score. Since only 2 percent of scores qualify:

- Invalidations per second: 50,000 x 0.02 = 1,000
- Cache rebuilds per second: 1,000
- Each rebuild takes 1ms
- Cache is empty for 1,000 x 1ms = 1 second out of every 30 seconds
- Cache hit rate: (30 - 1) / 30 = 96.7 percent

The cache hit rate improves from approximately 0 percent to 96.7 percent. The 1,000 invalidations per second that do occur are necessary because the top 100 actually changed in those cases.

</details>

### Challenge 1.3: Friends Leaderboard Design

You are adding a friends leaderboard feature to a social gaming platform. Each player has an average of 50 friends. The platform has 10 million active players. The existing global leaderboard uses a Redis sorted set. You cannot change the global leaderboard architecture.

**Question:** Design the data structure and query strategy for the friends leaderboard. How do you compute a player's rank among their friends? What is the expected latency per query? Estimate the additional Redis memory required.

<details>
<summary>Model Answer</summary>

The friends leaderboard shows the rank of a player among their friends, not among all players. This is a fundamentally different problem from the global leaderboard.

**Design:**

1. Store a set of friends for each player in Redis using a Redis Set (not a sorted set):
   `SADD friends:{player_id} friend_1 friend_2 friend_3 ...`

2. When a player requests their friends leaderboard:
   - Fetch the friend list: `SMEMBERS friends:{player_id}` -- O(50), negligible time
   - Query each friend's score from the existing global sorted set using pipelining:
     ```
     pipe = redis.pipeline()
     for friend in friends:
         pipe.zscore(global_key, friend)
     scores = pipe.execute()  # All 50 queries in one round trip
     ```
   - If the player's own score is needed, include their player_id in the friend list
   - Sort friends by score descending locally in application memory:
     ```
     friend_scores = sorted(zip(friends, scores), key=lambda x: -x[1])
     rank = next(i for i, (f, s) in enumerate(friend_scores) if f == player_id) + 1
     ```

3. Cache the result: `friends_lb:{player_id}` with a TTL of 30 seconds. Invalidate the cache when the player's score changes.

**Expected latency:**
- SMEMBERS call: 1ms
- 50 pipelined ZSCORE calls: 2ms (one round trip)
- Local sort of 50 items: 0.01ms
- Total: approximately 3ms uncached, under 1ms cached

**Additional memory for friend sets:**
- 10 million players x 50 friends x 16 bytes per friend ID = 8 GB
- Plus Redis set overhead (hashtable): approximately 20-30 percent
- Total: approximately 10-11 GB

This is significant but manageable with a Redis cluster. The friend relationships can also be stored in a dedicated Redis instance separate from the leaderboard sorted set to avoid interference.

</details>

### Challenge 1.4: Weekly Leaderboard with Time-Based Decay

Player scores in a weekly contest decay by 1 percent every hour after submission. A player submits a score of 10,000 at hour 0 and their effective score drops to approximately 7,860 after 24 hours. After 48 hours, it reaches approximately 6,180. The weekly contest has 1 million participants. Players check the leaderboard frequently during the first 24 hours, then activity drops off significantly.

**Question:** Design a Redis-based leaderboard that implements hourly score decay. How do you apply decay to 1 million scores without overwhelming Redis? What is the cost in operations per day? How do you handle decreasing player activity over the week?

<details>
<summary>Model Answer</summary>

The naive approach applies decay to all 1 million scores every hour: 24 million ZADD operations per day. This wastes CPU on players who never check the leaderboard during the later days of the week. A better approach uses lazy decay combined with selective recomputation.

**Design:**

1. **Encode the submission timestamp in the sorted set member.** Store the member as `{player_id}:{timestamp}`. The sorted set score is the raw score at submission time. This embeds the information needed to compute the effective score without additional lookups.

2. **Lazy decay on individual rank queries.** When a player requests their rank, fetch their raw score and compute the effective score using the embedded timestamp:
   ```
   raw_score, timestamp = parse(member_string)
   hours_since = (current_time - timestamp) / 3600
   effective_score = raw_score * pow(0.99, hours_since)
   ```
   This costs 1 ZSCORE operation per query with zero decay overhead for non-top-1000 players.

3. **Periodic deep decay for the top 1000.** Every 5 minutes, fetch the top 1000 entries, recompute their effective scores, and update the sorted set. Players above rank 1000 are the only ones who see visible rank changes from decay, so they are the only ones who need decayed scores in the sorted set.

4. **Active player tracking.** Maintain a Redis set called `active_players`. Only apply decay to these players during the periodic deep decay pass. Players who have not logged in for 24 hours are removed from the active set and decayed lazily on their next login.

**Cost analysis per day:**
- Top-1000 deep decay: 288 runs per day x 2,000 Redis calls (ZRANGE plus ZADD per entry) = 576,000 operations
- Active player decay (peak 50,000 active): 288 runs x 250 ZADD per run = 72,000 operations
- Individual rank queries: approximately 500,000 ZSCORE operations
- Total: approximately 1.15 million Redis operations per day

This is approximately 5 percent of the cost of naive hourly decay (24 million ZADD operations).

**Memory:**
- Sorted set with 1 million entries: approximately 64 MB
- Active player set: approximately 4 MB
- Total: under 70 MB for the complete leaderboard

The key insight is that 99 percent of players do not need their scores decayed in the sorted set. Decay is applied lazily on read for most players and eagerly only for the top 1000.

</details>

### Challenge 1.5: Team Leaderboard Aggregation

Your game has teams of up to 10 players. The team leaderboard shows the sum of each team's top 5 players' scores. Each player can be on exactly one team at a time. You have 2 million teams and 10 million players. A player's score changes approximately 5 times per day on average.

**Question:** Design the data model and computation strategy for the team leaderboard. How do you update the team score when a player's individual score changes? How do you handle a player leaving one team and joining another? What is the cost in Redis operations per score update?

<details>
<summary>Model Answer</summary>

The team leaderboard requires maintaining team aggregates that update atomically when individual player scores change. The challenge is propagating individual score changes to the correct team aggregate without expensive full recomputation.

**Data model:**

1. **Player-to-team mapping.** Store the team membership in a Redis string:
   ```
   SET player:team:{player_id} {team_id}
   ```

2. **Team member scores.** Store each team's individual player scores in a per-team sorted set:
   ```
   ZADD team:scores:{team_id} {player_score} {player_id}
   ```
   This sorted set holds up to 10 entries per team.

3. **Team aggregate score.** Store the team's total score (sum of top 5) in the team leaderboard:
   ```
   ZADD leaderboard:teams {team_score} {team_id}
   ```

**Score update flow (5 Redis operations per update):**

When `player_id` on `team_id` achieves `new_score`:

1. Look up the player's team: `GET player:team:{player_id}` -- 1 operation
2. Update the team member scores: `ZADD team:scores:{team_id} {new_score} {player_id}` -- 1 operation
3. Fetch the top 5 scores: `ZREVRANGE team:scores:{team_id} 0 4 WITHSCORES` -- 1 operation
4. Sum values -- client-side, 0 Redis operations
5. Update the team leaderboard: `ZADD leaderboard:teams {new_team_score} {team_id}` -- 1 operation

**Atomic update with Lua scripting:**

Combine steps 2-5 into a single atomic Lua script call:

```lua
local player_id = KEYS[1]
local team_id = KEYS[2]
local new_score = tonumber(ARGV[1])
local team_scores_key = "team:scores:" .. team_id

redis.call("ZADD", team_scores_key, new_score, player_id)

local top5 = redis.call("ZREVRANGE", team_scores_key, 0, 4, "WITHSCORES")
local sum = 0
for i = 2, #top5, 2 do
    sum = sum + tonumber(top5[i])
end

redis.call("ZADD", "leaderboard:teams", sum, team_id)
return sum
```

This reduces the 5 operations to 1 Lua script call.

**Handling team transfers:**

When a player leaves `team_a` to join `team_b`:

1. Remove the player from `team_a`: `ZREM team:scores:{team_a} {player_id}`
2. Recompute `team_a` top-5 sum: `ZREVRANGE team:scores:{team_a} 0 4 WITHSCORES`
3. Add the player to `team_b`: `ZADD team:scores:{team_b} {new_score} {player_id}`
4. Recompute `team_b` top-5 sum
5. Update both teams in the leaderboard sorted set
6. Update player-to-team mapping: `SET player:team:{player_id} {team_b}`

This is O(1) for add and remove and O(5) for computing the top-5 sum.

**Memory estimate:**
- 2 million team sorted sets with up to 10 entries each: approximately 2 million x 10 x 64 bytes = 1.28 GB
- Team leaderboard sorted set with 2 million entries: approximately 128 MB
- Player-to-team mapping: 10 million x 20 bytes = 200 MB
- Total: approximately 1.6 GB

A single `cache.r6g.xlarge` Redis instance (32 GB) can handle this comfortably.

</details>

### Challenge 2.2: The Hot Key Problem

A popular streamer is playing your game. Their score is updated 100 times per second because their fans are sending in-game boosts. The streamer's entry in the sorted set is a hot key -- every write to the sorted set acquires a lock on the skip list node, causing all other writes to queue up.

**Question:** Design a solution to the hot key problem. How do you handle a single player being updated 100x more frequently than the average player? Provide at least two alternative solutions and explain when each is appropriate.

<details>
<summary>Rubric</summary>

Score your answer against these criteria. Aim for 10 out of 12 or higher.

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) | Not addressed (0 pts) |
|---|---|---|---|---|
| Problem diagnosis | Explains why hot keys are problematic: skip list global lock causes all writes to queue up | Mentions "hot key is slow" without explaining the mechanism | Says "hot key is bad" | No diagnosis |
| Solution 1: Debouncing | Complete design with local counter, flush threshold (time + count), atomic update | Mentions debouncing without threshold design | Mentions "batch the updates" vaguely | Not addressed |
| Solution 2: Separated sorted set | Design for top-player set with merge logic; explains when to use this vs. debouncing | Mentions separate set without merge details | Mentions "use a different set" | Not addressed |
| Solution 3 (bonus) | Client-side batching or adaptive rate limiting with clear tradeoff analysis | Mentions one alternative but no tradeoff analysis | Mentions a third option vaguely | Not addressed |

**Model answer key points:**

**Solution 1: Server-side debouncing (best for most cases)**
- Maintain a local counter per hot player: `debounce:{player_id}:{window}`
- Accumulate score deltas: `HINCRBYFLOAT debounce key delta 1`
- Flush to sorted set when counter >= 1000 OR 5 seconds have elapsed
- Reduces 100 writes/sec to 1 write per 5 seconds (500x reduction)
- Tradeoff: up to 5 seconds of staleness for the hot player's rank

**Solution 2: Separate sorted set for top players (best for competitive tournaments)**
- Maintain two sorted sets: `leaderboard:regular` and `leaderboard:champions`
- Hot players with frequent updates go into the champions set
- The champions set is smaller (only top 1% of players) so lock contention is lower
- Global leaderboard merges both sets at read time
- Tradeoff: read complexity increases (merge two sets instead of one)

**Solution 3: Client-side batching (best when you control the client)**
- The game client batches score boosts and sends them in bulk every 5 seconds
- Server receives one update instead of 500
- Tradeoff: requires client update and cannot be enforced for third-party clients

</details>

### Challenge 2.4: Streak-Based Scoring with 50 Million Users

Your fitness app awards bonus points for consecutive days of activity. A player earns 10 base points per day, plus a streak multiplier: day 1 is worth 10 points, day 2 is worth 20 points, day 3 is worth 30 points, day N is worth N x 10 points. The leaderboard shows total accumulated score.

**Constraints:**
- 50 million users
- 20 percent of users active in the same 30-minute window each day
- Each user has at most one streak change per day
- The leaderboard must show both global rank and friends rank

<details>
<summary>Rubric</summary>

Score your answer against these criteria. Aim for 11 out of 12 or higher.

| Criteria | Excellent (3 pts) | Good (2 pts) | Fair (1 pt) | Not addressed (0 pts) |
|---|---|---|---|---|
| Streak computation | Daily batch process computes streaks for all active users; handles timezone offsets and missed days; idempotent so reruns are safe | Batch process described but no idempotency or timezone handling | Mentions streaks without computation strategy | No streak computation |
| Peak write throughput | Calculates throughput (10 million users spread across 30 minutes = 5,555 writes per second); explains how pipelining handles this | Mentions peak throughput without exact calculation | Says "it might be slow at peak" | No throughput analysis |
| Leaderboard update strategy | Batched updates: accumulates daily score changes in a hash, applies in one pipeline batch | Updates leaderboard per user without batching | Mentions updating scores without mechanism | No leaderboard update design |
| Friends leaderboard integration | Uses the friends set from Challenge 1.3 combined with the streak-based sorted set; explains caching with 5-minute TTL | Mentions friends integration without caching strategy | Says "friends can see streaks" vaguely | No integration |

**Model answer key points:**

**Streak computation flow:**

Run a daily batch process that iterates over all 50 million users in batches of 10,000. For each user, query their last activity date from a Redis hash (`user:streak:{user_id}`). Compare it to today's date:

```
function compute_streak(user_id):
    streak_data = redis.hgetall("user:streak:{user_id}")
    last_active = streak_data.get("last_active")
    if last_active is None:
        streak = 1
    elif date(last_active) == today - 1:
        streak = int(streak_data.get("count", 0)) + 1
    elif date(last_active) == today:
        return  # Already processed today, skip
    else:
        streak = 1  # Streak broken

    daily_score = streak * 10
    total_score = int(streak_data.get("total", 0)) + daily_score
    redis.hset("user:streak:{user_id}", "count", streak)
    redis.hset("user:streak:{user_id}", "total", total_score)
    return (user_id, total_score)
```

Process all 50 million users in approximately 5 minutes using parallel batch workers. Each batch of 10,000 takes 1-2 seconds.

**Peak write throughput:**

10 million users spread across 30 minutes:
- 10,000,000 users / 1,800 seconds = 5,555 writes per second peak
- Redis can handle this on a single instance (approximately 100,000 operations per second)
- Use pipelining to batch all updates:
  ```
  pipe = redis.pipeline()
  for user_id, total in updates:
      pipe.zadd("leaderboard:global", total, user_id)
  pipe.execute()
  ```
- The pipeline executes in approximately 30 seconds, so the leaderboard is at most 30 seconds stale during peak

**Leaderboard read path:**
- Global rank query: `ZRANK leaderboard:global {user_id}` -- O(log N), approximately 0.1ms
- Top-100 query: `ZREVRANGE leaderboard:global 0 99 WITHSCORES` -- O(log N + 100), approximately 0.2ms
- Friends rank: combine the friends set from Challenge 1.3 with the streak-based sorted set; cache each player's friends leaderboard with a 5-minute TTL

**Storage:**
- Streak counters (50 million): 50 million x 8 bytes = 400 MB in a Redis hash
- Last activity dates (50 million): 50 million x 8 bytes = 400 MB in a Redis hash
- Global leaderboard sorted set (50 million): 50 million x 64 bytes = 3.2 GB
- Total: approximately 4 GB

</details>

### Challenge 3.1: Billion-Player Leaderboard

You are designing a leaderboard for a global platform with 1 billion registered users and 200 million daily active users. The leaderboard must support daily, weekly, and monthly views. The infrastructure budget is $500,000 per month.

**Constraints:**
- 1 billion registered users, 200 million daily active users
- Daily, weekly, and monthly leaderboard views
- Most players do not need exact ranks -- only the top 1 percent need exact rank precision
- Budget is $500,000 per month for the complete leaderboard system
- The leaderboard must be responsive within 100ms for any request

<details>
<summary>Model Answer</summary>

At 200 million daily active users, a single Redis sorted set with 200 million entries would consume approximately 12.8 GB of memory. The write throughput of 200 million score submissions per day works out to approximately 2,300 writes per second on average, with peaks of 20,000 writes per second during high-traffic hours. A single Redis instance can handle this throughput, but the cost can be optimized by using appropriate instance sizes and sharding.

**Architecture:**

1. **Sharded Redis sorted sets.** Shard the player base into 32 shards. Each shard handles approximately 6.25 million entries. Each shard runs on a `cache.r6g.xlarge` instance (32 GB memory, approximately $0.50/hour). Total Redis cost: 32 x $500 = $16,000 per month. This is within the budget.

2. **Approximate rank for 99 percent of players.** Use a score bucket array with 100 buckets covering the full score range. The bucket array provides O(1) rank estimation with a bounded error. Only the top 1 percent of players (2 million players) are looked up in the exact sorted set for their exact rank.

3. **Conditional caching with larger top-N.** Cache the top-1000 leaderboard (not just top-100) because at this scale, the top-1000 changes less frequently than the top-100. The larger cache reduces the frequency of cache rebuilds. The top-1000 minimum score is maintained as a separate Redis key.

4. **Database persistence with Aurora.** Use Amazon Aurora PostgreSQL for durable storage and historical analytics. Aurora provides 5x throughput of standard PostgreSQL at a similar cost. The score history table stores all submitted scores with player_id, score, window_type, and timestamp.

5. **Processing tier with Kafka.** Apache Kafka with 32 partitions (one per shard) handles score ingestion. The auto-scaling consumer group processes scores and updates the appropriate shard. Each partition maps to one Redis shard.

6. **CDN caching for the top-1000.** Cache the formatted top-1000 leaderboard at the CDN edge with a 30-second TTL. The CDN serves cache hits in under 5ms from the nearest edge location. Only the origin server handles cache misses.

**Expected latency:**
- Approximate rank query (99 percent of queries): 0.1ms (bucket array lookup)
- Exact rank query (top 1 percent of players): 2-5ms (sharded sorted set query plus merge)
- Top-100 query: 2-5ms (CDN cache hit) or 10-20ms (cache miss, read from all 32 shards)
- Global rank query: 10-50ms (read from all 32 shards, merge, compute rank)

**Cost breakdown:**
- 32 Redis shards at $500/month each: $16,000
- Processing tier (Kafka + consumers): $5,000
- Database (Aurora PostgreSQL): $4,000
- Cache tier (Redis cache nodes): $3,000
- CDN: $2,000
- Other (monitoring, networking, load balancers): $3,000
- Total: approximately $33,000 per month

The key tradeoff is accepting approximate ranks for 99 percent of players. This is acceptable because:
- Players in the middle of the leaderboard do not need exact ranks
- The difference between rank 50,000,000 and rank 50,000,005 is invisible to the player
- Only the top players (top 1 percent) and players near the cutoff for the next tier need exact ranks
- The bucket array provides rank estimates within plus or minus 5 positions, which is sufficient for 99 percent of use cases

</details>

### Challenge 3.2: Real-Time Stock Market Leaderboard

You are designing a system that ranks stocks by their performance (percentage change) in real time during market hours. The system tracks 50,000 stocks. This use case is fundamentally different from a gaming leaderboard in two ways: the number of entities is small (50,000 stocks versus millions of players), but the update frequency is extremely high (up to 50 million updates per second versus 1-10 updates per second per player). At 50 million updates per second, a single Redis sorted set cannot handle the throughput because Redis processes approximately 100,000 operations per second on one CPU core.

#### Why Redis Is the Wrong Tool Here

Redis sorted sets are designed for leaderboards with many members and infrequent updates per member. Each ZADD operation is O(log N) where N is the set size. With 50,000 stocks, log2(50,000) is approximately 16 comparisons per insert. Redis can perform approximately 100,000 of these operations per second on a single core. At 50 million updates per second, you would need 500 Redis instances running in parallel. This is impractical for a single leaderboard view.

The deeper problem is that Redis is single-threaded for data operations. You cannot parallelize ZADD calls targeting a single sorted set. The O(log N) cost per update is not the primary bottleneck -- the single-threaded execution model is.

#### Data Structure Selection

The correct approach uses an in-memory data structure in a single application process running on a multi-core machine:

**Recommended structure: ConcurrentSkipListMap (Java) or std::map (C++) with read-write locks.**

| Data Structure | Insert | Delete | Min/Max | Memory Overhead | Concurrent Safe |
|---|---|---|---|---|---|
| ConcurrentSkipListMap | O(log N) | O(log N) | O(1) peek, O(log N) remove | High (multi-level pointers) | Yes (lock-free) |
| std::map (red-black tree) | O(log N) | O(log N) | O(log N) | Medium (2-3 pointers per node) | No (needs external locking) |
| std::multiset | O(log N) | O(log N) | O(log N) | Medium | No |
| Binary heap | O(log N) | O(log N) | O(1) | Low (array-based) | No |
| B-tree (in-memory) | O(log B N) | O(log B N) | O(log B N) | Medium | With careful locking |

The ConcurrentSkipListMap is the best choice because it provides lock-free concurrent reads, O(log N) cost for all operations, and O(1) access to the first and last entries (top losers and top gainers).

**Dual data structure pattern:**

Maintain two structures in the same process:

1. A `ConcurrentHashMap<String, Double>` mapping stock symbol to current performance (percentage change). This provides O(1) lookup for individual stock queries from the WebSocket clients.

2. A `ConcurrentNavigableMap<Double, Set<String>>` mapping performance values to sets of stock symbols at that performance. This provides O(log N) access to the top gainers and losers and handles ties correctly (multiple stocks at the same performance percentage).

#### Batch Updates and Partitioning

Accumulate price changes in lock-free queues for 100ms and apply them in a single batch. Partition the stock universe across CPU cores for parallel processing:

```python
class StockMarketLeaderboard:
    def __init__(self, num_partitions=8):
        self.stocks = {}
        self.sorted_performances = SortedDict()
        self.partitions = [LockFreeQueue() for _ in range(num_partitions)]
        self.running = True
        self.worker_threads = []

    def start(self):
        for i in range(num_partitions):
            t = Thread(target=self._process_partition, args=(i,))
            self.worker_threads.append(t)
            t.start()

    def _process_partition(self, partition_id):
        while self.running:
            batch = self.partitions[partition_id].drain()
            if batch:
                self._apply_batch(batch)

    def _apply_batch(self, updates):
        for symbol, performance in updates:
            old_perf = self.stocks.get(symbol)
            self.stocks[symbol] = performance
            if old_perf is not None:
                self.sorted_performances[old_perf].discard(symbol)
                if not self.sorted_performances[old_perf]:
                    del self.sorted_performances[old_perf]
            self.sorted_performances.setdefault(performance, set()).add(symbol)

    def get_top_gainers(self, n=10):
        result = []
        for perf, symbols in reversed(self.sorted_performances.items()):
            for symbol in symbols:
                result.append((symbol, perf))
                if len(result) >= n:
                    return result
        return result

    def get_top_losers(self, n=10):
        result = []
        for perf, symbols in self.sorted_performances.items():
            for symbol in symbols:
                result.append((symbol, perf))
                if len(result) >= n:
                    return result
        return result
```

#### WebSocket Push Architecture

Every 100ms, the top 10 gainers and top 10 losers are published to a message bus. WebSocket servers across regions push the update to connected clients:

- Payload size: approximately 400 bytes (20 symbols at 10 bytes each plus performance values)
- Message bus: Kafka with 2 partitions (one for gainers, one for losers)
- WebSocket fan-out: use Redis Pub/Sub or a dedicated fan-out service to reach all WebSocket server instances
- Client-side update rate: approximately 10 messages per second (100ms interval)
- Bandwidth per client: approximately 4 KB per second (downstream)

#### Monitoring and Observability

For a production stock market leaderboard, monitor these metrics:

1. **Update latency.** Track the time from price arrival to leaderboard update. Alert if p99 exceeds 150ms during market hours. Use a histogram metric with 10ms buckets.

2. **Batch processing time.** Each 100ms batch should complete in under 25ms. If batches start taking 50ms or more, the system is falling behind and needs more partitions or faster hardware.

3. **Queue depth.** Monitor the lock-free queue depths. If any queue grows beyond 1 million entries, the system cannot keep up with the data feed. This indicates the need for more partitions or faster processing.

4. **Memory usage.** Track heap usage over time. For 50,000 stocks, set a 5 GB heap limit. A growing heap may indicate a memory leak in the sorted structure (entries not being cleaned up for de-listed stocks).

5. **Stale stock detection.** If a stock has not been updated for 5 seconds during market hours, it may indicate a data feed failure. Log this condition and display a warning on the operations dashboard.

#### Edge Cases

**Market open surge.** When the market opens, all 50,000 stocks receive their first price update of the day within 1-2 seconds. This creates a burst of 50,000 writes in a single batch cycle. Pre-warm the data structure by seeding all stocks with the previous day's closing price before the market opens. This distributes the initial writes across the warm-up period (approximately 5 minutes) instead of concentrating them at the open.

**Trading halt.** When a stock is halted, its price stops updating. The leaderboard should freeze the last known performance value for halted stocks. Display a "H" marker next to halted stocks in the top-10 display. If a halt lasts more than 1 hour, remove the stock from the sorted structure to keep the leaderboard responsive.

**Stock split.** A stock split changes the price but not the percentage change calculation. The performance formula uses price-adjusted data from the market data feed. No special handling is needed in the leaderboard logic as long as the feed provides adjusted prices. Validate this assumption during integration testing.

**After-hours trading.** Low trading volume during after-hours sessions means far fewer updates (1-10 per second per stock). The architecture handles this trivially. Consider increasing the push interval to 1 second during after-hours to reduce WebSocket traffic and server load.

#### Scaling to 500,000 Stocks

To scale from 50,000 to 500,000 stocks:

1. **Increase partition count.** Use 32 partitions instead of 8. Each partition handles approximately 15,625 stocks. This keeps the per-partition update rate at approximately 150,000 updates per second (equivalent to the 50,000-stock case with 8 partitions).

2. **Increase heap memory.** 500,000 stocks at approximately 2 KB per stock (symbol string, performance value, sorted structure entries) equals approximately 1 GB of heap. The total heap requirement is approximately 4 GB including overhead, which still fits on a single large instance.

3. **Shard the sorted structure.** Use consistent hashing to assign stocks to separate sorted structures. The top-10 gainers query reads the top-10 from each shard (approximately 10 x 32 = 320 entries) and merges them in application memory. This adds 1-2ms to the query time but keeps write throughput manageable.

4. **Horizontal scaling by sector.** If a single machine cannot handle the load, split the stock universe across multiple machines by sector (technology, healthcare, finance). Each sector machine maintains its own leaderboard. A coordinator machine merges the top-10 from each sector for the overall top-10 gainers and losers.

#### Performance Characteristics

- Batch update (partition-level, 8 partitions on a 32-core machine): approximately 3ms per partition
- Top-10 gainers query: O(1) via `lastEntry()` on the sorted structure (constant time, essentially instant)
- WebSocket push interval: 100ms
- End-to-end latency from price change to client update: approximately 103ms (3ms batch processing plus 100ms push interval)

#### Deployment

- Primary and warm standby instances for fault tolerance
- The primary handles all price updates and WebSocket publishing
- The standby replicates state from the primary every 60 seconds via a serialized snapshot
- On failure: the standby takes over in under 1 second
- During failover: clients receive the last published top-10 (100ms stale) until the standby publishes its first update
- Run the primary and standby in different availability zones to survive an AZ failure

This design shows how the leaderboard pattern must be adapted when the workload has a high write-to-read ratio. The core insight is that the data structure must match the workload characteristics: many writes per entity requires an in-memory concurrent data structure, not a centralized Redis sorted set.
