## Naive Implementation

The naive leaderboard implementation uses a single relational database table with an index on the score column. This works for small scale but breaks under load.

### The Single-Table Approach

```sql
CREATE TABLE leaderboard (
    player_id VARCHAR(64) PRIMARY KEY,
    score INTEGER NOT NULL DEFAULT 0,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_leaderboard_score_desc ON leaderboard (score DESC);

-- Get top 100
SELECT player_id, score FROM leaderboard ORDER BY score DESC LIMIT 100;

-- Get player rank
SELECT COUNT(*) + 1 AS rank FROM leaderboard
WHERE score > (SELECT score FROM leaderboard WHERE player_id = 'player_42');

-- Update score
INSERT INTO leaderboard (player_id, score, updated_at)
VALUES ('player_42', 15000, NOW())
ON CONFLICT (player_id)
DO UPDATE SET
    score = GREATEST(leaderboard.score, EXCLUDED.score),
    updated_at = NOW();
```

This approach fails at approximately 100,000 active players. The rank query for a player in the bottom half takes over 1 second. The database cannot handle more than 5,000 concurrent score submissions per second. The connection pool of 500 is exhausted during peak hours, causing connection timeouts.

### What Breaks First

The rank query is the first thing to break. The `COUNT(*) WHERE score > X` query requires a full index scan. For a player with a score at the 50th percentile, this means scanning 25 million index entries. Even with a fast SSD, this takes hundreds of milliseconds.

The second thing to break is the write throughput. Every score submission requires a write to the `leaderboard` table. At 5,000 writes per second, the database write-ahead log (WAL) becomes a bottleneck. Checkpointing falls behind, and the database spends more time flushing WAL segments than processing queries.

The third thing to break is the connection pool. PostgreSQL has a practical limit of about 500 simultaneous connections. At 10,000 concurrent players submitting scores and querying the leaderboard, the connection pool is exhausted, and new connections are queued or rejected.

## Production-Grade Architecture

A production-grade leaderboard architecture uses three tiers: Redis for the live sorted set, a cache for the top-N leaderboard, and a database for durable storage. The system is horizontally scalable at every tier.

### Tier Specifications

**Ingestion Tier:**
- Autoscaling group of game servers behind a load balancer
- Maximum 1 score submission per second per player
- Ingestion limit: 50,000 requests per second

**Message Queue:**
- Apache Kafka with 12 partitions for parallel processing
- Retains scores for 7 days for replay capability
- Delivery guarantee: at-least-once

**Processing Tier:**
- Consumer group with up to 12 workers (one per Kafka partition)
- Each worker maintains a connection to Redis primary
- Maximum throughput: 50,000 score updates per second

**Redis Tier:**
- Primary node: `cache.r6g.xlarge` (32 GB memory)
- Two read replicas in different availability zones
- Replication lag: typically under 5ms
- Persistence: AOF with fsync every second

**Cache Tier:**
- Redis cluster with 3 nodes for high availability
- TTL for cached top-N: 30 seconds
- Cache hit rate target: above 95 percent

**Storage Tier:**
- PostgreSQL RDS with Multi-AZ deployment
- Automated backups with 35-day retention
- Read replica for analytics queries

**Serving Tier:**
- Autoscaling API nodes (target CPU: 60 percent)
- Response time SLA: p99 under 50ms
- Global edge caching via CDN for static leaderboard pages

## Integration Patterns

### Pattern 1: Real-Time Score Update with WebSocket Push

When a player's score changes, the system pushes the update to connected clients via WebSockets. This provides a real-time experience without polling.

```python
# WebSocket push on score update
async def handle_score_update(player_id: str, window_key: str, old_score: int, new_score: int):
    # Compute the rank change
    redis_key = f"leaderboard:daily:{window_key}"
    new_rank = await redis.zrevrank(redis_key, player_id)

    # Publish to Redis Pub/Sub for WebSocket distribution
    update_message = json.dumps({
        "type": "score_update",
        "player_id": player_id,
        "old_score": old_score,
        "new_score": new_score,
        "new_rank": new_rank + 1,
        "timestamp": time.time()
    })
    await redis.publish("leaderboard:updates", update_message)
```

### Pattern 2: Scheduled Snapshot for Analytics

A scheduled job captures a snapshot of the leaderboard at regular intervals for analytics and historical comparison.

```python
# Scheduled leaderboard snapshot (every 5 minutes)
async def capture_leaderboard_snapshot(window_type: str = "daily"):
    window_key = get_current_window_key(window_type)
    redis_key = f"leaderboard:{window_type}:{window_key}"

    # Get all entries (or top 10,000 for large leaderboards)
    entries = await redis.zrevrange(redis_key, 0, 9999, withscores=True)

    # Store snapshot in database
    snapshot_id = str(uuid4())
    await db.execute(
        "INSERT INTO leaderboard_snapshots (id, window_type, window_key, captured_at) "
        "VALUES ($1, $2, $3, NOW())",
        snapshot_id, window_type, window_key
    )

    # Bulk insert entries
    async with db.transaction():
        for rank, (player_id, score) in enumerate(entries, 1):
            await db.execute(
                "INSERT INTO leaderboard_snapshot_entries "
                "(snapshot_id, rank, player_id, score) "
                "VALUES ($1, $2, $3, $4)",
                snapshot_id, rank, player_id.decode(), int(score)
            )

    return snapshot_id
```

### Pattern 3: Event-Driven Score Processing with Kafka

Using Apache Kafka as the message queue enables reliable, scalable score processing with exactly-once semantics.

```python
# Kafka producer for score events
from aiokafka import AIOKafkaProducer

producer = AIOKafkaProducer(
    bootstrap_servers=['kafka-1:9092', 'kafka-2:9092', 'kafka-3:9092'],
    value_serializer=lambda v: json.dumps(v).encode(),
    acks='all',  # Wait for all in-sync replicas to acknowledge
    retries=3,
)

async def publish_score_event(player_id: str, score: int, metadata: dict):
    event = {
        "event_type": "score_submitted",
        "player_id": player_id,
        "score": score,
        "metadata": metadata,
        "timestamp": time.time(),
        "event_id": str(uuid4()),
    }
    # Partition by player_id for ordering guarantees
    key = player_id.encode()
    await producer.send("scores", key=key, value=event)
```

### Pattern 4: Multi-Game Leaderboard with Tenant Isolation

A platform hosting multiple games needs tenant isolation. Each game gets its own Redis database or key prefix to prevent interference.

```python
# Multi-game leaderboard with key prefix isolation
class MultiGameLeaderboard:
    def __init__(self, redis_client, game_id: str):
        self.redis = redis_client
        self.prefix = f"game:{game_id}"

    async def update_score(self, player_id: str, score: int, window_type: str):
        key = f"{self.prefix}:leaderboard:{window_type}:{get_current_window_key(window_type)}"
        await self.redis.zadd(key, {player_id: score})

    async def get_top_n(self, window_type: str, n: int = 100):
        key = f"{self.prefix}:leaderboard:{window_type}:{get_current_window_key(window_type)}"
        return await self.redis.zrevrange(key, 0, n - 1, withscores=True)

    async def get_player_rank(self, player_id: str, window_type: str):
        key = f"{self.prefix}:leaderboard:{window_type}:{get_current_window_key(window_type)}"
        return await self.redis.zrevrank(key, player_id)
```

### Pattern 5: Batch Score Reconciliation with Periodic Full Sync

Mobile games with intermittent connectivity need to reconcile scores when players come back online. Score events are cached on the device, submitted as a batch, and validated server-side for tampering and duplicates.

```python
# Offline score cache on device (client-side pseudo-code)
class OfflineScoreCache:
    def __init__(self):
        self.pending_scores: List[dict] = []
        self.max_batch_size = 50

    def record_score(self, score: int, game_state_hash: str):
        self.pending_scores.append({
            "score": score,
            "game_state_hash": game_state_hash,
            "device_timestamp": time.time(),
            "session_id": self.current_session_id,
        })

    async def sync_batch(self, api_client):
        response = await api_client.post(
            "/v1/leaderboard/sync",
            json={
                "player_id": self.player_id,
                "scores": self.pending_scores[-self.max_batch_size:],
                "device_id": self.device_id,
            }
        )
        if response.status == 200:
            result = response.json()
            self.pending_scores = [
                s for s in self.pending_scores
                if s["session_id"] not in result["reconciled_sessions"]
            ]


async def reconcile_batch(player_id: str, scores: List[dict], device_id: str):
    # 1. Validate integrity of each score submission
    validated_scores = []
    for score_entry in scores:
        if not verify_score_integrity(score_entry):
            continue  # Skip tampered or invalid entries
        validated_scores.append(score_entry)

    # 2. Deduplicate by session_id to prevent double-counting
    deduped = {}
    for entry in validated_scores:
        session_id = entry["session_id"]
        if session_id not in deduped or entry["score"] > deduped[session_id]["score"]:
            deduped[session_id] = entry

    # 3. Apply reconciled scores via Redis pipeline
    redis_key = f"leaderboard:daily:{get_current_window_key()}"
    reconciled = []
    async with redis.pipeline(transaction=True) as pipe:
        for entry in deduped.values():
            await pipe.zadd(redis_key, {player_id: entry["score"]})
            reconciled.append(entry["session_id"])
        await pipe.execute()

    return {"reconciled_sessions": reconciled, "count": len(reconciled)}
```

This pattern ensures offline scores are not lost and the leaderboard remains consistent even with delayed submissions. The validation step is critical for preventing score injection attacks. The max batch size of 50 prevents a single client from overwhelming the server after a long offline period.

### Pattern 6: Tiered Leaderboard with Percentile Buckets

For games with millions of players, computing and displaying exact rank for every player is wasteful. Instead, group players into percentile buckets and show an approximate rank range. This reduces storage requirements and improves response times for the majority of players.

```python
# Tiered leaderboard with percentile buckets
class TieredLeaderboard:
    TIERS = [
        {"name": "legendary",  "min_percentile": 0.00, "max_percentile": 0.01,
         "label": "Top 1%"},
        {"name": "elite",      "min_percentile": 0.01, "max_percentile": 0.05,
         "label": "Top 5%"},
        {"name": "platinum",   "min_percentile": 0.05, "max_percentile": 0.10,
         "label": "Top 10%"},
        {"name": "gold",       "min_percentile": 0.10, "max_percentile": 0.25,
         "label": "Top 25%"},
        {"name": "silver",     "min_percentile": 0.25, "max_percentile": 0.50,
         "label": "Top 50%"},
        {"name": "bronze",     "min_percentile": 0.50, "max_percentile": 1.00,
         "label": "Bottom 50%"},
    ]

    def __init__(self, redis_client):
        self.redis = redis_client

    async def get_player_tier(self, player_id: str, window_type: str) -> dict:
        window_key = get_current_window_key(window_type)
        redis_key = f"leaderboard:{window_type}:{window_key}"
        total_players = await self.redis.zcard(redis_key)
        if total_players == 0:
            return {"tier": "unranked", "rank": None, "total": 0}

        rank = await self.redis.zrevrank(redis_key, player_id)
        if rank is None:
            return {"tier": "unranked", "rank": None, "total": total_players}

        percentile = (rank + 1) / total_players
        for tier in self.TIERS:
            if tier["min_percentile"] <= percentile < tier["max_percentile"]:
                return {
                    "tier": tier["name"],
                    "label": tier["label"],
                    "rank": rank + 1,
                    "total": total_players,
                    "percentile": round(percentile * 100, 1),
                }

        return {"tier": "bronze", "label": "Bottom 50%",
                "rank": rank + 1, "total": total_players}

    async def get_tier_thresholds(self, window_type: str) -> list:
        window_key = get_current_window_key(window_type)
        redis_key = f"leaderboard:{window_type}:{window_key}"
        total_players = await self.redis.zcard(redis_key)
        thresholds = []

        for tier in self.TIERS:
            if tier["name"] == "bronze":
                min_score = 0
            else:
                rank_index = int(total_players * tier["min_percentile"])
                entry = await self.redis.zrevrange(
                    redis_key, rank_index, rank_index, withscores=True
                )
                min_score = int(entry[0][1]) if entry else 0

            thresholds.append({
                "tier": tier["name"],
                "label": tier["label"],
                "min_score": min_score,
            })

        return thresholds
```

The tiered approach has three advantages: it reduces computation for rank lookups by an order of magnitude, it provides meaningful progress indicators for players in the lower half, and it allows the system to cache tier thresholds rather than individual ranks. For a game with 10 million players, the tiered leaderboard reduces storage from 64 bytes per player to 6 bytes per tier threshold -- a reduction of over 99.9 percent for the rank data.

### Active-Active Replication Strategy

Each region has a Redis primary that accepts writes locally. Writes are replicated to other regions asynchronously. Conflict resolution uses a last-writer-wins strategy based on server-side timestamps.

```python
# Multi-region write with conflict resolution
async def multi_region_write(player_id: str, score: int, region: str):
    local_key = f"leaderboard:daily:{get_current_window_key()}"
    local_ts = time.time()

    # Write to local primary with region timestamp
    await local_redis.zadd(local_key, {player_id: score})
    await local_redis.set(f"ts:{player_id}", local_ts)

    # Publish cross-region replication event
    replication_event = {
        "player_id": player_id,
        "score": score,
        "region": region,
        "timestamp": local_ts,
    }
    await replication_producer.send("leaderboard.replicate", replication_event)

    # Return immediately -- don't wait for cross-region replication
    return {"status": "accepted", "region": region, "timestamp": local_ts}
```

### Cross-Region Merge for Global Top-N

The global coordinator periodically fetches the top-N from each region and merges them into a single global leaderboard.

```python
# Global merge of regional top-N
async def merge_global_leaderboard(top_n: int = 100):
    regions = ["us-east", "eu-west", "ap-east"]
    all_candidates = []

    for region in regions:
        redis_client = get_region_redis(region)
        key = f"leaderboard:daily:{get_current_window_key()}"
        region_top = await redis_client.zrevrange(key, 0, top_n - 1, withscores=True)
        all_candidates.extend(region_top)

    # Sort and deduplicate by player_id (keep highest score)
    merged = {}
    for player_id, score in all_candidates:
        pid = player_id.decode() if isinstance(player_id, bytes) else player_id
        if pid not in merged or score > merged[pid]:
            merged[pid] = int(score)

    # Sort by score descending
    sorted_players = sorted(merged.items(), key=lambda x: -x[1])

    # Return top-N
    return [
        {"rank": i + 1, "player_id": pid, "score": score}
        for i, (pid, score) in enumerate(sorted_players[:top_n])
    ]
```

## Capacity Planning

### Estimating Redis Memory

The primary cost driver is Redis memory. Each entry in a sorted set consumes approximately 64 bytes plus the key and member size. For 10 million players:

- 10,000,000 players at 64 bytes each = 640 MB for the skip list
- Player IDs at 16 bytes average = 160 MB for the dictionary
- Key overhead = negligible
- Total: approximately 800 MB per sorted set

For a system with daily, weekly, monthly, and all-time leaderboards:
- 4 sorted sets at 800 MB each = 3.2 GB
- Cache tier: 500 MB for top-N cache entries
- Buffer for growth (20 percent): 750 MB
- Total Redis memory required: approximately 4.5 GB

### Estimating Throughput

For 10 million daily active users with peak concurrency of 20 percent:

```
Peak concurrent players: 10,000,000 x 0.20 = 2,000,000
Score submissions per second (1 per player per 60 seconds): 2,000,000 / 60 = 33,333
Read requests per second (each player checks leaderboard every 30 seconds): 66,667
Total requests per second: 100,000
```

Redis single-threaded throughput is approximately 100,000 to 200,000 operations per second. At 100,000 requests per second, a single Redis instance can handle the load. For higher throughput, shard the sorted set across multiple Redis instances.

Recovery Point Objective (RPO): 60 seconds

Recovery steps:

   - Time: under 10 seconds

5. Verify data integrity
   - Compare top-100 from Redis with top-100 from database
   - Check total player count matches expected
   - Alert if discrepancy exceeds 0.1 percent

## Service Level Objectives

Defining clear SLOs for the leaderboard system ensures you know when the system is healthy and when you need to intervene. Each SLO is backed by an SLI measured in production and a burn rate that triggers alerts.

### Read Latency SLO

The leaderboard read path must meet strict latency targets to avoid degrading the player experience.

| Operation | Target | SLO | Measurement Method |
|---|---|---|---|
| Top-100 read | p50 under 10ms, p99 under 50ms | 99.9 percent | Distributed tracing span duration |
| Player rank lookup | p50 under 5ms, p99 under 30ms | 99.9 percent | APM instrumentation on API handler |
| Leaderboard page load | p95 under 200ms (CDN cached) | 99.5 percent | Synthetic check every 60 seconds from 3 regions |

### Write Latency SLO

Score submissions follow an asynchronous path through the message queue. The staleness between submitting a score and seeing it on the leaderboard is the key metric.

| Operation | Target | SLO | Measurement Method |
|---|---|---|---|
| Score submission acknowledgment | p99 under 100ms | 99.95 percent | Application metric on ingest endpoint |
| End-to-end score visibility | p99 under 2 seconds | 99.5 percent | Trace from ingest to sorted-set update |
| Duplicate rejection | p99 under 50ms | 99.9 percent | Application metric on dedup logic |

### Availability SLO

Each component of the leaderboard system has its own availability target. The overall system availability is the product of the component SLOs along the critical path.

| Component | Monthly Target | Annual Downtime Budget | Dependencies |
|---|---|---|---|
| Leaderboard read API | 99.99 percent | 52.56 minutes | Redis replicas, cache tier |
| Score submission API | 99.95 percent | 4.38 hours | Kafka, Redis primary |
| Redis sorted set | 99.995 percent | 26.3 minutes | Primary and replica health |
| PostgreSQL | 99.99 percent | 52.56 minutes | Multi-AZ automatic failover |
| Kafka cluster | 99.95 percent | 4.38 hours | Broker count at or above minimum ISR |

The composite availability for the read path is 99.98 percent, which means approximately 1.75 hours of aggregate downtime per year. The write path composite availability is 99.90 percent, approximately 8.76 hours per year.

### Freshness SLO

Leaderboard data is eventually consistent. The system guarantees maximum staleness for each access pattern.

| Access Pattern | Maximum Staleness | Mechanism |
|---|---|---|
| Real-time WebSocket push | 500ms | Redis Pub/Sub to WebSocket bridge |
| Top-N leaderboard display | 60 seconds | Cache TTL with background refresh |
| Player rank display | 30 seconds | Cache TTL with lazy invalidation |
| Historical analytics | 5 minutes | Scheduled database snapshot job |

### Precision SLO

Not every leaderboard query requires exact precision. The system defines precision tiers to match the access pattern to the appropriate data source.

| Precision Level | Error Margin | Data Source | Use Case |
|---|---|---|---|
| Exact | Rank is 100 percent correct | Redis sorted set primary | Tournament qualification, prize distribution |
| Near-exact | Within 5 ranks | Redis sorted set replica | Player rank display in UI |
| Approximate | Within percentile bucket | Tier cache | Lower-half player rank display |
| Stale | Correct at snapshot time | Database | Analytics, historical reports |

### Error Budget

For a 99.99 percent availability SLO on the read path, the monthly error budget is calculated as follows:

- 99.99 percent availability equals a 0.01 percent error rate
- Monthly time budget: 30 x 24 x 60 x 0.0001 = 4.32 minutes of downtime
- Monthly request budget at 100,000 requests per second: 100,000 x 86,400 x 30 x 0.0001 = 25,920,000 failed requests

When the error budget falls below 20 percent remaining, the team enters a freeze period. No new features are deployed until reliability improvements reduce the error rate. Features that add capacity or improve resilience are exempt from the freeze.

### SLI Measurement

Each SLI is measured with a combination of three techniques:

- **RED metrics**: Rate (requests per second), Errors (failed requests), Duration (latency distribution). Collected by Prometheus with 15-second scrape intervals and retained for 30 days at full resolution.

- **Synthetic probes**: A probe running in each region submits a score and then reads the leaderboard every 60 seconds. The probe measures end-to-end latency, staleness, and correctness. If the probe detects a discrepancy exceeding the precision SLO, it fires a pager-level alert.

- **Trace sampling**: Every score submission is traced at a 1:100 sampling rate for normal traffic. Traces that exceed the latency SLO are sampled at 1:1 (full sampling) for troubleshooting. Traces are exported to the tracing backend with 30-second export intervals and retained for 14 days.

### Burn Rate Alerts

Burn rate alerts tell you how quickly you are consuming the error budget. Multiple burn rates trigger different response actions.

| Burn Rate | Duration to Deplete Budget | Alert Action |
|---|---|---|
| 2x (consumes budget 2x faster than expected) | 15 days | Page on-call engineer within 5 minutes |
| 6x | 5 days | Auto-scale read replicas, page on-call team |
| 30x | 1 day | Emergency traffic shift to secondary region, page engineering lead |
