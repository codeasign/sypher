## The Problem: It Is 2 AM and You Are Storing Passwords

Picture this scene. You are building a calendar application. A user signs up and immediately says, "Great, now import my Google Calendar events so I can see everything in one place."

Your first engineering instinct: "I will just ask for their Google password, log in as them, and fetch the data."

Stop right there.

That approach is a security nightmare. Let us count the ways.

**Full account access.** That single password unlocks their email, their Google Drive, their photos, their YouTube history, everything. Your application only needs calendar data. But the password grants the keys to the entire kingdom.

**No granularity.** There is no way to say "only calendar access." With a password, you get it all or nothing. The user has no control over what you can see or do.

**No revocation.** Once the user gives you their password, they cannot take back calendar access without changing their password entirely. Changing the password also logs them out of every other application they use.

**Massive liability.** If your database is ever breached, every stored Google password is stolen. You become the headline. Your users lose trust in you. Lawsuits follow.

**Zero user trust.** Would you hand your Google password to a random calendar application you found on the internet? No. Neither would your users. The moment you ask for a password, they close the tab and never come back.

Every user instinct screams no. And for good reason.

But your application does need to access their calendar data. There is a genuine user need here. What if there was a way to grant limited, revocable, auditable access without ever seeing their password?

There is.

## The Insight: OAuth2 Is a Valet Key

**OAuth2 (Open Authorization)** is a protocol that lets a user grant a third-party application limited access to their resources on another service without ever sharing their credentials with that application.

Think about a valet key for your car.

Your master key ring opens every door, starts the engine, unlocks the trunk, opens the glovebox, and releases the gas cap. You carry that key because you need full access to your own car.

Now imagine you pull up to a restaurant and hand your keys to a valet. You do not hand them your master key ring. You hand them a special valet key. That valet key only opens the driver's door and starts the engine. It cannot open the trunk. It cannot open the glovebox. It cannot access the storage compartments in the back seat.

If the valet decides to rummage through your car, they cannot. The valet key simply will not open those compartments. And if you decide you do not trust this valet anymore, you do not need to re-key the entire car. You just invalidate that one valet key.

**OAuth2 is that valet key for web APIs.**

Instead of giving an application your Google password (the master key), OAuth2 lets you issue a limited token (the valet key) that says, "This application can read my calendar events for the next hour. Nothing else. No email access. No Drive access. No password changes."

If you decide you no longer trust the application, you revoke the token. The application loses access. You never changed your password. Every other application you authorized keeps working.

This is the core insight. OAuth2 decouples *authentication* (who you are) from *authorization* (what an application is allowed to do on your behalf).

## Core Vocabulary

Before we go further, you need to speak the language of OAuth2. These terms appear in every specification, every API documentation page, and every system design interview question about authorization.

| Term | What It Means |
|------|---------------|
| **OAuth2** | The protocol itself. An authorization framework, not an authentication protocol (though it is often used for both). Version 2 is the current standard, published as RFC 6749. |
| **Resource Owner** | The user who owns the data and can grant access to it. In our example, the person with a Google Calendar who decides whether to let your application see their events. |
| **Client** | The application requesting access to the Resource Owner's data. This is your calendar application. Clients can be web applications, mobile apps, desktop apps, server-side services, or even smart devices. |
| **Authorization Server** | The server that authenticates the Resource Owner, obtains consent, and issues tokens. In our example, this is Google's accounts.google.com. It is the gatekeeper. |
| **Resource Server** | The server that hosts the protected data and accepts access tokens. In our example, this is Google Calendar's API server. It trusts the Authorization Server to validate tokens. |
| **Access Token** | A short-lived credential that the Client presents to the Resource Server to access data. Think of this as the valet key itself. Usually valid for minutes to hours. The format is often a JWT but can be an opaque string. |
| **Refresh Token** | A longer-lived credential used to obtain new access tokens without requiring the user to re-authenticate. Usually valid for days, months, or until explicitly revoked. |
| **Authorization Code** | An intermediate value that the Authorization Server returns after the Resource Owner grants consent. The Client exchanges this code for tokens. It is short-lived (typically minutes) and single-use. |
| **Client ID** | A public identifier for the Client application. It is not a secret. Think of it as a username for your application. It is visible in the authorization URL and can be shared. |
| **Client Secret** | A confidential secret known only to the Client and the Authorization Server. Used to authenticate the Client when exchanging codes or requesting tokens. Must never be exposed to browsers or mobile app binaries. |
| **Redirect URI** | The URL on the Client's domain where the Authorization Server sends the Resource Owner after authorization. Must be pre-registered with the Authorization Server to prevent open redirect attacks. |
| **Scope** | A set of permissions the Client is requesting. Each scope represents a specific action or resource. Examples: `calendar.readonly`, `profile.email`, `openid`. The user sees and approves these on the consent screen. |
| **Grant Type** | The specific OAuth2 flow being used. Each grant type is designed for a different scenario. The four main grant types are Authorization Code, Client Credentials, Device Code, and Refresh Token. |

## Big Picture: The Four Roles

Every OAuth2 interaction involves exactly four roles working together. Understanding how these roles relate to each other is the foundation of everything else.

The flow reads top to bottom. Here is how the roles interact:

1. The **Resource Owner** (your user) authenticates with the **Authorization Server** and grants permission for a specific **Client** (your application) to access their data.
2. The **Client** presents proof of this permission (the authorization code) to the **Authorization Server** and receives tokens in return.
3. The **Client** presents the access token to the **Resource Server** to fetch protected data.

Note that the **Authorization Server** and **Resource Server** can be the same physical server. Google runs both accounts.google.com (Authorization Server) and www.googleapis.com (Resource Server). But they are logically separate roles with distinct responsibilities. The Authorization Server issues tokens. The Resource Server validates and consumes them.

This separation is powerful. It means a single Authorization Server can serve many Resource Servers. Your Google login can give you access to Calendar, Drive, Gmail, Photos, and YouTube, each its own Resource Server, all using tokens issued by the same Authorization Server.

## Step by Step: The Authorization Code Flow

The Authorization Code grant type is the most common OAuth2 flow. It is what happens every time you click "Login with Google" or "Sign in with GitHub." This is the default choice for web applications that have a server-side backend.

Here is every step, from your user's click to your application displaying their data. We will trace the full journey.

### Step 1: User Initiates Login

Your user visits your calendar application and clicks a button labeled "Import from Google Calendar" or "Sign in with Google." This is the starting point. No tokens exist yet. No authorization has been granted.

### Step 2: Client Builds the Authorization URL

Your application's backend constructs a URL pointing to Google's Authorization Server. This URL encodes everything the Authorization Server needs to know: who is asking, what they want, where to send the user afterward.

The URL looks like this:

```text
https://accounts.google.com/o/oauth2/v2/auth?
  response_type=code&
  client_id=123456789.apps.googleusercontent.com&
  redirect_uri=https://yourapp.com/auth/callback&
  scope=https://www.googleapis.com/auth/calendar.readonly&
  state=random_csrf_token_abc123&
  access_type=offline
```

Each parameter has a specific purpose:

- **response_type=code**: Tells the Authorization Server you want an authorization code (not a token directly).
- **client_id**: Publicly identifies your application. This is registered with Google when you set up your project.
- **redirect_uri**: Where the user should be sent after they approve or deny. Must exactly match what you registered.
- **scope**: What permissions you are requesting. Here, read-only access to calendar data.
- **state**: A random, unguessable string you generate. This is your CSRF defense.
- **access_type=offline**: Requests a refresh token so your app can get new access tokens later without user interaction.

### Step 3: Client Redirects the User

Your application sends the user's browser to the constructed URL. The user leaves your domain and lands on Google's authentication page. Your application does not see the user's credentials at any point.

### Step 4: User Authenticates with the Authorization Server

Google displays a login page. The user enters their email and password. If they have two-factor authentication enabled, they complete that step too. The user is now authenticated on the Authorization Server.

Important: this authentication happens entirely on Google's domain. Your application never sees the password, never handles the 2FA code, and never stores any authentication credentials.

### Step 5: User Consents to Scopes

After authenticating, Google shows a consent screen. This is a critical user moment. The screen lists exactly what your application is requesting permission to do.

The consent screen looks roughly like this:

```text
YourApp wants to:

  [x] View your Google Calendar events

                    [Allow]    [Deny]
```

The user sees what they are granting. They click Allow.

This is the valet key contract made visible. The user is saying, "I give this application permission to read my calendar events, and only that." If your application later tries to access their email, Google will block it because the scope was not approved.

### Step 6: Authorization Server Issues an Authorization Code

Google generates a short-lived, single-use authorization code. Google appends this code to your application's registered redirect URI and sends the user's browser there.

The browser arrives at your application with a URL like this:

```text
https://yourapp.com/auth/callback?
  code=AUTH_CODE_XYZ&
  state=random_csrf_token_abc123
```

**Your application MUST verify the state parameter.** Compare the `state` value in the URL to the one you generated in Step 2. If they do not match, abort the request. This prevents CSRF attacks where an attacker tricks a user into completing an authorization flow they did not initiate.

### Step 7: Client Exchanges the Authorization Code

Now the critical exchange happens. Your application's backend makes a direct server-to-server HTTP POST to Google's token endpoint. No browser involved. This is important because the request includes your client secret.

```text
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTH_CODE_XYZ
&redirect_uri=https://yourapp.com/auth/callback
&client_id=123456789.apps.googleusercontent.com
&client_secret=YOUR_CLIENT_SECRET
```

Because this call happens server-to-server, the `client_secret` is never exposed to the browser. If an attacker intercepts the authorization code in Step 6 (possible in some scenarios), they still cannot exchange it for tokens because they do not have the client secret.

### Step 8: Authorization Server Validates and Returns Tokens

Google's token endpoint validates every piece of data:

1. The authorization code exists, has not expired, and has not been used before.
2. The client ID matches the one the code was issued to.
3. The client secret matches the client ID.
4. The redirect URI matches the one registered for this client.

If all checks pass, Google returns a JSON response:

```json
{
  "access_token": "ya29.a0AfH6S...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "1//0eAbCdEfGhIjK..."
}
```

- **access_token**: The valet key. Your app will send this with every API call.
- **token_type**: Always `Bearer` for OAuth2. This tells the Resource Server how to interpret the token.
- **expires_in**: Lifetime in seconds. 3600 seconds = 1 hour.
- **refresh_token**: A long-lived token for getting new access tokens later.

### Step 9: Client Uses the Access Token

Your application now has an access token. It can call Google Calendar's API. The access token goes in the HTTP `Authorization` header:

```text
GET https://www.googleapis.com/calendar/v3/calendars/primary/events
Authorization: Bearer ya29.a0AfH6S...
```

The format is always `Bearer ` followed by the token value. This is the standard OAuth2 convention for carrying tokens in HTTP requests.

### Step 10: Resource Server Validates the Token

Google Calendar's API server (the Resource Server) receives your request. It must determine whether the access token is valid. It performs these checks:

1. **Parses the token**: If the token is a JWT, it decodes the header and payload. If opaque (a random string), it calls the Authorization Server's introspection endpoint.
2. **Checks the signature**: For JWT tokens, verifies the cryptographic signature using the Authorization Server's public key. This ensures the token was genuinely issued by the Authorization Server and not forged.
3. **Checks expiration**: Verifies the token has not expired. If expired, returns HTTP 401.
4. **Checks scopes**: Verifies the token includes the scope required by the requested resource. Reading calendar events requires `calendar.readonly`. If the token only has `profile.email`, the request is denied.
5. **Checks revocation**: Optionally checks if the token has been revoked by the user.

If all checks pass, the request proceeds. If any check fails, the Resource Server returns HTTP 401 Unauthorized with an error description.

### Step 11: Resource Server Returns Data

The Calendar API authenticates the request and returns the user's calendar events as JSON. Your application receives this data and displays it to the user.

### Step 12: Token Expiry and Refresh

One hour later, the access token expires. Your application tries to fetch the user's calendar again and receives HTTP 401 Unauthorized.

Your application does not panic. It does not ask the user to log in again. It uses the refresh token from Step 8.

Your backend makes another server-to-server request:

```text
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=1//0eAbCdEfGhIjK...
&client_id=123456789.apps.googleusercontent.com
&client_secret=YOUR_CLIENT_SECRET
```

Google validates the refresh token, the client ID, and the client secret, then returns a fresh access token:

```json
{
  "access_token": "ya29.a0AfH6S.NEWTOKEN...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

Your application continues using the new access token. The user never knew the token expired. This is a seamless experience.

## Variants

The Authorization Code flow is the most common grant type, but OAuth2 defines others. Each variant exists to solve a specific constraint: no browser, no server-side backend, limited user input, or server-to-server communication.

### Authorization Code with PKCE

The standard Authorization Code flow relies on a client secret that only the Client and Authorization Server know. This works for server-side web applications where the secret stays on the backend.

But what about mobile apps and single-page applications (SPAs)?

A mobile app binary can be decompiled. Embedded secrets are extractable. No matter how well you obfuscate, a determined attacker can find the client secret. An SPA runs entirely in the browser. Every line of code is visible. There is no secure place to store a secret.

**PKCE (Proof Key for Code Exchange, pronounced "pixie")** solves this problem without requiring any client secret at all.

Here is how PKCE works:

1. The Client generates a random string called the `code_verifier`. This is a high-entropy secret, at least 43 characters long.
2. The Client creates a SHA-256 hash of the verifier. This hash is called the `code_challenge`.
3. In the initial authorization request, the Client sends the `code_challenge` (not the original verifier) and the hash method.
4. The Authorization Server stores the `code_challenge` and returns an authorization code as usual.
5. When exchanging the code for tokens, the Client sends the original `code_verifier`.
6. The Authorization Server applies SHA-256 to the received verifier and compares the result to the stored `code_challenge`. If they match, the exchange succeeds.

This prevents a specific attack: an attacker who intercepts the authorization code cannot exchange it for tokens because they do not know the `code_verifier`. The attacker sees the `code_challenge` in the initial request, but hashing is one-way. They cannot reverse the hash to find the original verifier.

### Client Credentials

Sometimes no user is involved. A backend cron job needs to fetch data from an internal analytics API. A microservice needs to authenticate to another microservice. A batch processing job needs to upload results to a storage service.

In these cases, the client acts on its own behalf, not on behalf of any user. The Client Credentials grant is designed for this exact scenario.

The flow is simple:

1. The Client authenticates to the Authorization Server using its `client_id` and `client_secret`.
2. The Authorization Server validates the credentials and returns an access token.
3. The Client uses the token to access the Resource Server.

There is no redirect. No user consent screen. No browser interaction of any kind. The entire exchange is a single HTTP request.

```text
POST https://auth.example.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
&scope=analytics.read
```

Response:

```json

  "token_type": "Bearer",
  "expires_in": 3600
}
```

Common use cases for Client Credentials:

- **Microservice-to-microservice authentication**: Service A needs to call Service B. Service A authenticates itself using its client credentials and receives a token scoped to the specific API it needs.
- **Cron jobs and batch processing**: A scheduled job that processes data from an internal API. No user is sitting at a keyboard approving this.
- **Admin operations**: An internal dashboard that performs administrative tasks. The dashboard authenticates as itself, not as any specific admin user.
- **Service accounts**: A non-human identity that needs programmatic access to resources. Cloud providers use this pattern extensively.

### Device Code

Some devices do not have a good way to enter credentials. A smart TV has a remote control with directional buttons. A gaming console has a controller. A CLI tool has no graphical interface at all.

The Device Code grant (also called the Device Authorization grant) solves this by splitting the flow across two devices.

Here is how it works:

1. The device requests authorization from the Authorization Server and receives a `device_code`, a `user_code` (a short, human-readable code), and a `verification_uri`.
2. The device displays the `verification_uri` and `user_code` on its screen. A smart TV shows: "Go to https://example.com/activate on your phone or computer and enter code ABC-1234."
3. The user opens the URL on their phone or laptop, enters the code, authenticates, and approves the scopes.
4. Meanwhile, the device polls the Authorization Server at regular intervals (every few seconds, as specified by the `interval` parameter) asking, "Is the user done yet?"
5. Once the user completes authorization, the polling response includes the access token (and refresh token).

```text
POST /device_authorization
Content-Type: application/x-www-form-urlencoded

client_id=YOUR_CLIENT_ID
&scope=streaming.read
```

Response:

```json
{
  "device_code": "DEVICE_CODE_XYZ",
  "user_code": "ABC-1234",
  "verification_uri": "https://example.com/activate",
  "interval": 5,
  "expires_in": 1800
}
```

The device then polls:

```text
POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=urn:ietf:params:oauth:grant-type:device_code
&device_code=DEVICE_CODE_XYZ
&client_id=YOUR_CLIENT_ID
```

The Authorization Server returns one of three responses:
- **authorization_pending**: The user has not yet finished. Try again after `interval` seconds.
- **success**: The user finished. Here is your access token.
- **expired_token**: The device code expired. Start over.

This is the flow behind every "Activate your device" experience. Netflix uses it. Amazon Prime uses it. Spotify uses it. You have almost certainly used it without knowing its name.

### Refresh Token Flow

The Refresh Token grant is not a standalone OAuth2 flow. It is a companion to the Authorization Code and Device Code grants. It exists because access tokens are intentionally short-lived.

A refresh token is a long-lived credential that the client uses to obtain new access tokens without requiring the user to re-authenticate or re-consent.

Key properties of refresh tokens:

- **Long-lived**: Valid for days, months, or until explicitly revoked. Some providers never expire refresh tokens.
- **Revocable**: The user can revoke a refresh token at any time, cutting off the application's access.
- **Rotatable**: Many providers issue a new refresh token each time one is used. The old refresh token is invalidated. This limits the damage if a refresh token is leaked.
- **Storage-sensitive**: A refresh token is as powerful as a password. It must be stored securely on the server side, never in browser storage or a mobile app's unprotected storage.

The refresh flow itself is simple. The client detects that the access token has expired (receives a 401 or checks the expiry time) and calls the token endpoint:

```text
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=1//0eAbCdEfGhIjK...
&client_id=123456789.apps.googleusercontent.com
&client_secret=YOUR_CLIENT_SECRET
```

The Authorization Server returns a fresh access token:

```json
{
  "access_token": "ya29.a0AfH6S.NEWLY_ISSUED_TOKEN...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "1//0eAbCdEfGhIjK.NEW_OR_SAME..."
}
```

Some providers issue the same refresh token on every refresh. Others issue a new one (refresh token rotation). If a refresh token is rotated and you lose the new one, you cannot recover. Store the new refresh token immediately upon receiving it.

The refresh flow makes long-lived application access possible. A user can authorize your application once, and your application can access their data for weeks or months without bothering them again.

## Mental Model Check

Answer these five questions to confirm you understand OAuth2 at a system design level. Cover the answer before reading it, then check yourself.

### Question 1

Your CEO says, "Our application needs to post tweets on behalf of users. Why can we not just ask for their Twitter password and store it?"

**Answer**: Storing user passwords for a third-party service is a security disaster. If your database is breached, every user's Twitter credentials are compromised. OAuth2 lets users grant your application limited access (just posting tweets, not reading DMs or changing settings) and revoke that access at any time without changing their password. Even if your database is breached, the attacker gets only access tokens (which are scoped and short-lived), not passwords that unlock everything.

### Question 2

Explain the valet key analogy in your own words. Why does it matter for system design?

**Answer**: Your master car key opens everything. A valet key opens only the driver's door and starts the engine. OAuth2 is the valet key for APIs. Instead of giving your password (the master key), the Authorization Server issues a limited token (the valet key) that grants only the specific permissions the user approved. For system design, this means you can design systems where third-party applications have precisely scoped access, users retain control, and the blast radius of any breach is limited to the scopes on the compromised token, not the user's full account.

### Question 3

What is the difference between an access token and a refresh token in terms of lifetime, usage, and security properties?

**Answer**: An access token is short-lived (typically 60 minutes or less) and is presented to the Resource Server with every API call. It is the credential that proves permission. If leaked, its damage is limited by its short expiration. A refresh token is long-lived (days to months) and is used only to obtain new access tokens, never to access resources directly. The refresh token is stored securely server-side and must never be exposed to the browser or sent to the Resource Server. If a refresh token is leaked, the attacker can obtain new access tokens indefinitely, which is why refresh tokens should be revocable and optionally rotated.

### Question 4

You are building a mobile application that needs to access a user's cloud storage. Why must you use PKCE instead of the standard Authorization Code flow?

**Answer**: In a mobile application, you cannot keep a client secret secret. Anyone can decompile the mobile app binary and extract any embedded secrets. The standard Authorization Code flow assumes the client can protect a secret, which is false for mobile apps. PKCE eliminates the need for a client secret by using a dynamically generated `code_verifier` that the attacker cannot guess, even if they intercept the authorization code. The verifier is generated fresh for each authorization request and is never stored in the application binary.

### Question 5

Your backend cron job needs to fetch data from an internal analytics API every hour. No user is involved. Which grant type should you use and why?

**Answer**: Client Credentials. No user exists in this scenario, so there is no Resource Owner to authenticate or obtain consent from. The cron job (your backend service) authenticates itself using its own `client_id` and `client_secret` and receives a token scoped to the service itself, not to any user. The Authorization Code flow would be inappropriate because there is no user to redirect to a login page. The Device Code flow would be inappropriate because there is no device for a user to authorize on a separate device.
