# OAuth2 in Production

OAuth2 is a specification, but every production implementation makes real-world trade-offs. This page walks through how major platforms deploy OAuth2, what open-source and cloud solutions exist, and the hard-won lessons engineers have learned operating these systems at scale.

---

## Case Study: Google OAuth2

Google's OAuth2 implementation powers sign-in for millions of third-party applications through Google Identity Platform. It follows the standard Authorization Code flow with PKCE (Proof Key for Code Exchange) and adds several production-grade extensions.

### How Google OAuth2 Works

Google's OAuth2 flow uses the Authorization Code grant type as its primary mechanism. When a user clicks "Sign in with Google" on a third-party app, the app redirects the user to Google's authorization endpoint. The user authenticates with Google (often using an existing session cookie) and is presented with a consent screen listing the requested scopes. After the user grants consent, Google issues an authorization code and redirects back to the application. The application then exchanges this code, along with its client secret and a PKCE verifier, for an access token and optionally a refresh token.

### Key Production Details

Google's implementation includes several features that separate it from a textbook OAuth2 flow:

- **ID Tokens**: Google returns a JWT-based ID token alongside the access token. The ID token contains user identity claims (sub, email, name, picture) that the client can verify locally without calling the userinfo endpoint. This reduces latency and backend load.

- **Cross-Origin Opener Policy (COOP)**: Google uses COOP headers to prevent cross-origin attacks during the redirect flow. This is transparent to the developer but critical for security.

- **Automatic Token Refresh**: Google's client libraries handle token refresh transparently. The refresh token is long-lived (unless revoked), and the library automatically exchanges it when the access token expires.

- **Granular Consent**: Google shows a consent screen that lists every requested scope. Users can revoke individual scopes from their Google Account settings page. Google also supports incremental authorization — apps can request additional scopes later without re-authenticating the user.

- **Rate Limiting**: Google enforces per-client and per-user rate limits. The /token endpoint is limited to approximately 100 requests per minute per client. Exceeding this triggers exponential backoff.

- **Revocation Endpoint**: Google provides a revocation endpoint at `https://oauth2.googleapis.com/revoke`. Clients call this when a user signs out or when tokens need to be invalidated.

### Security Considerations

Google uses PKCE (Proof Key for Code Exchange) even for server-side web apps, not just for mobile apps. This provides defense against authorization code interception attacks. Google also rotates the authorization code — each code can be exchanged only once, and it expires within 10 minutes.

## Case Study: GitHub OAuth2

GitHub uses OAuth2 primarily for API access rather than simple authentication. When a developer integrates with GitHub's API (for example, to read repositories, create issues, or trigger workflows), the OAuth2 flow grants the application a token that acts as a credential for the GitHub API.

### GitHub's Token-Centric Model

GitHub's OAuth2 implementation is notable for its emphasis on PATs (Personal Access Tokens) and fine-grained permissions. Unlike Google's focus on identity, GitHub treats OAuth2 as an API authorization mechanism.

### Key Production Details

GitHub's OAuth2 implementation has several distinctive characteristics:

- **Scopes Are Repository-Level**: GitHub scopes map to repository permissions (repo, repo:status, write:packages, admin:org, etc.) rather than user identity. This reflects GitHub's focus on resource access.

- **Fine-Grained PATs**: GitHub introduced fine-grained Personal Access Tokens that allow developers to specify exact repository access, expiration dates, and permission levels. This is a significant improvement over the earlier all-or-nothing PAT model.

- **Token Expiration and Rotation**: GitHub encourages short-lived tokens through the OAuth device flow and regularly rotates signing keys. The default token lifetime is configurable, but GitHub recommends 1 year maximum with regular rotation.

- **GitHub App Installation Tokens**: For GitHub Apps, the OAuth flow grants an installation token that is scoped to specific repositories. These tokens expire after 1 hour and must be refreshed via a JWT-authenticated API call. This is a significant departure from standard OAuth2 — the app authenticates itself with a JWT, then exchanges it for an installation access token.

- **Webhook Integration**: GitHub OAuth tokens can be used to subscribe to webhook events. When a token is revoked, all associated webhook subscriptions are automatically cleaned up.

### Security Considerations

GitHub implements several security layers beyond the OAuth2 specification:

- GitHub monitors for suspicious token usage patterns and can automatically revoke tokens that appear compromised.
- Repository access tokens are scoped to specific repositories, not the entire user account.
- GitHub's OAuth app review process requires third-party apps to undergo manual review before requesting sensitive scopes.
- GitHub supports request signing with `Signed_Headers` for enterprise customers.

## Case Study: Facebook Login

Facebook Login is one of the earliest and most widely deployed OAuth2 implementations. It popularized the "social login" pattern and introduced several innovations that became industry standards.

### Facebook's Hybrid Approach

Facebook Login uses a variant of the Authorization Code flow that also returns a signed user identity claim. This is conceptually similar to Google's ID token approach but predates it by several years.

### Key Production Details

Facebook's implementation includes several features that are worth understanding:

- **App Secret Proof**: Facebook requires a cryptographic proof of the app secret on every API call. The client computes an HMAC-SHA256 hash of the access token using the app secret as the key. This prevents token leakage from being useful to attackers who do not know the app secret.

- **Signed Request**: Facebook's login flow returns a `signed_request` parameter alongside the authorization code. This is a base64-encoded JSON payload signed with the app secret. It contains the user ID, issued-at timestamp, and OAuth code. The client can verify this payload without a server round-trip.

- **Access Token Types**: Facebook issues different token types:
  - **User Access Token**: Authenticates API calls on behalf of a user.
  - **Page Access Token**: Authenticates API calls on behalf of a Facebook Page.
  - **App Access Token**: Authenticates API calls on behalf of the app itself.
  - **Client Token**: An identifier for the app, used for analytics endpoints.

- **Token Introspection**: Facebook provides a debug endpoint at `/debug_token` that returns the token's scopes, expiration, and application ID. This is essential for debugging token issues.

- **Automatic Token Refresh**: Facebook automatically extends the lifetime of short-lived tokens (2 hours) to long-lived tokens (60 days) when the user interacts with the app. This is transparent to the developer.

### Security Considerations

Facebook's `appsecret_proof` mechanism is a notable security innovation. Even if an access token is intercepted, the attacker cannot use it without the app secret. This is a defense-in-depth measure that has been adopted by several other platforms.

Facebook also aggressively monitors for unusual login patterns. If a user logs in from a new device or location, Facebook may require additional verification (email, SMS, or biometric).

## Case Study: Slack OAuth2

Slack's OAuth2 implementation is notable for its granular scope system, bot token support, and multi-workspace architecture. Slack uses OAuth2 to authorize both user-facing apps and bot integrations that operate independently of any single user.

### Slack's Bot-Centric Model

Slack's OAuth2 flow is unique because it often produces multiple tokens: a user token and a bot token. The bot token allows the application to act as a Slack bot, posting messages and responding to events, even when no user is actively using the app.

### Key Production Details

Slack's OAuth2 implementation has several distinctive features:

- **Dual Token Response**: The OAuth response includes both an `access_token` (user-level) and a `bot_token` (bot-level). The bot token is scoped to the bot's permissions and persists independently of the user who installed the app. This allows the app to continue operating even if the installing user leaves the workspace.

- **Granular Scopes**: Slack's scope system is exceptionally granular. There are over 100 scopes covering channels, messages, files, users, user groups, reactions, stars, reminders, and more. Each scope follows the `resource:action` pattern (e.g., `channels:read`, `channels:write`, `chat:write`).

- **Workspace Installation**: Slack OAuth flows include workspace selection. The authorization code is tied to a specific workspace (`team_id`). The app must be installed in each workspace separately, receiving a distinct set of tokens for each one.

- **Events API Integration**: After the OAuth flow, apps typically subscribe to the Events API. Slack sends real-time events (message posted, user joined, app mentioned) to the app's configured Request URL. The app must verify the event's authenticity using Slack's signing secret.

- **Token Rotation**: Slack supports token rotation through the `oauth.v2.access` and `oauth.v2.exchange` endpoints. Apps can exchange an expiring access token for a new one, reducing the risk of long-lived token leakage.

### Security Considerations

Slack implements several security measures:

- All event payloads include a signed timestamp and a signing secret that the app must verify.
- Slack supports the `state` parameter for CSRF protection and requires it for all OAuth flows.
- Bot tokens have limited scope compared to user tokens. A bot token cannot, for example, change workspace settings or access billing information.
- Slack's OAuth token page allows users to revoke individual app tokens from any workspace.

## Open Source OAuth2 Solutions

Many organizations run their own OAuth2 servers rather than relying on third-party providers. This is common in enterprise environments, regulated industries, and organizations that need full control over their authentication infrastructure.

### Ory Hydra

Ory Hydra is an open-source OAuth2 and OpenID Connect server implemented in Go. It is designed to be a hardened, production-grade authorization server that can be integrated with any identity provider.

**Architecture**: Hydra is strictly an OAuth2 server — it does not handle user authentication. Instead, it delegates authentication to a separate identity provider through a "consent app" interface. This separation of concerns allows organizations to use their existing user store (LDAP, Active Directory, custom database) while benefiting from Hydra's OAuth2 implementation.

**Key Features**:
- Supports all OAuth2 grant types: authorization code, implicit, client credentials, resource owner password, refresh token, and device code.
- Full OpenID Connect support (discovery, userinfo, ID tokens).
- Cryptographic signing with RS256, RS384, RS512, ES256, ES384, ES512, PS256, PS384, PS512, and EdDSA.
- JSON Web Key Set (JWKS) rotation and management.
- OAuth2 token introspection and revocation endpoints.
- SQL, PostgreSQL, MySQL, CockroachDB, and in-memory storage backends.
- Horizontal scaling via stateless access tokens (JWT) or opaque tokens with Redis caching.

**Production Considerations**:
- Hydra is designed for Kubernetes deployment. The Helm chart is the recommended installation method.
- The consent app must be highly available; if it goes down, new OAuth2 flows cannot complete.
- Hydra's JWT access tokens can be validated without a database call, which is critical for high-throughput systems.
- The `hydra token introspect` command is useful for debugging but should not be in the critical path of production requests.

### Dex

Dex is a federated OpenID Connect identity provider written in Go. It is a CNCF (Cloud Native Computing Foundation) graduated project, originally created by CoreOS. Dex acts as a bridge between external identity providers and Kubernetes API server authentication.

**Architecture**: Dex implements the OpenID Connect protocol and provides connectors to various identity backends. It does not store users itself — it delegates authentication to the configured connector. This makes Dex a "pass-through" identity provider that translates external authentication into OIDC tokens.

**Supported Connectors**:
- LDAP / Active Directory
- SAML (via SAML 2.0)
- GitHub, GitLab, Google, Microsoft
- OpenID Connect (Dex can chain to another OIDC provider)
- Static users (for testing only)
- Okta (via SAML or OIDC)
- Atlassian Crowd

**Key Features**:
- Token refresh with offline access scope.
- Cross-site request forgery (CSRF) protection.
- Password grant type for non-interactive clients.
- Dynamic client registration.
- Token introspection for token validation.
- gRPC and HTTP API for management.

**Production Considerations**:
- Dex is commonly deployed alongside Kubernetes as a sidecar or in the same cluster. It is the standard way to authenticate kubectl with external identity providers.
- Dex does not include a user interface for user management. User provisioning and deprovisioning must be handled by the external identity provider.
- Connector configuration is loaded at startup. Changes to connector configuration require a Dex restart.
- Dex supports multiple simultaneous connectors, and users can choose which connector to use at login time.

### Keycloak

Keycloak is an open-source identity and access management solution developed by Red Hat. It is the most feature-complete open-source OAuth2/OIDC server and includes user management, role-based access control, and identity brokering.

**Architecture**: Keycloak is a full-stack IAM (Identity and Access Management) solution. Unlike Hydra (which separates authentication from authorization), Keycloak handles both user management and OAuth2 token issuance. It includes a web-based admin console, user self-service portal, and a comprehensive REST API.

**Key Features**:
- OAuth2 and OpenID Connect (certified by the OpenID Foundation).
- SAML 2.0 support.
- User federation with LDAP and Active Directory.
- Identity brokering and social login (Google, Facebook, GitHub, Twitter, etc.).
- Role-based access control (RBAC) with fine-grained permissions.
- User-managed access (UMA) for resource sharing.
- Authentication flows with step-up authentication, conditional authentication, and multi-factor authentication.
- Account management console for end users.
- Client-side session management with session timeouts and idle timeouts.
- Audit logging and event reporting.
- Custom themes and internationalization.
- Admin REST API for automation.

**Production Considerations**:
- Keycloak can run standalone or in a clustered configuration. For production, a minimum of two nodes behind a load balancer is recommended.
- Database-backed sessions are required for clustered deployments. Keycloak supports PostgreSQL, MySQL, MariaDB, and Oracle.
- Infinispan (distributed cache) is used for session replication. The cache configuration must be tuned for the expected user load.
- Keycloak can be heavy — it requires approximately 512 MB of heap per node for moderate workloads and can scale to multiple GB for large deployments.
- Version upgrades require careful migration planning. Keycloak provides migration tooling, but downtime is often required.
- The Quarkus-based distribution (Keycloak.X) is the recommended deployment option for new installations.

### Spring Authorization Server

Spring Authorization Server is a framework for building OAuth2 and OpenID Connect servers in the Java ecosystem. It is the successor to the deprecated Spring Security OAuth2 project.

**Architecture**: Spring Authorization Server is a library, not a standalone server. Developers embed it into a Spring Boot application and configure it programmatically or via properties. This provides maximum flexibility but requires more development effort than turnkey solutions.

**Key Features**:
- Authorization code, client credentials, refresh token, and device code grant types.
- PKCE support.
- OpenID Connect 1.0 with discovery and userinfo endpoints.
- JWT access tokens and opaque tokens.
- JWK set endpoint with key rotation.
- Client authentication with client secret (basic, post, JWT) and private key JWT.
- OAuth2 client and resource server configuration via Spring Security.
- Reactive (WebFlux) support.

**Production Considerations**:
- Spring Authorization Server is best suited for Java shops that already use Spring Boot. It integrates naturally with Spring Security.
- The registry pattern for clients and tokens is extensible. Implementations can store clients in a database, LDAP, or any other backend.
- Custom token customization is straightforward via the `OAuth2TokenCustomizer` interface.
- The framework is relatively new and less battle-tested than Keycloak or Hydra. It is suitable for production but requires thorough testing.
- There is no admin UI. All configuration is code-based or API-driven.

## Cloud OAuth2 Solutions

Cloud-based OAuth2 solutions abstract away the operational complexity of running authorization servers. They provide managed identity services with built-in scalability, security, and compliance features.

### Amazon Cognito

Amazon Cognito is AWS's managed identity service. It provides user pools (for authentication) and identity pools (for AWS resource authorization). Cognito implements OAuth2 and OpenID Connect for user pool authentication.

**Architecture**: Cognito User Pools act as an OAuth2 authorization server. They support the Authorization Code grant, implicit grant, and client credentials grant. Cognito also provides a hosted UI for login flows, or developers can use the Cognito SDK for custom UI implementations.

**Key Features**:
- User directory with sign-up, sign-in, and profile management.
- Multi-factor authentication (SMS, TOTP, email).
- Social identity provider federation (Google, Facebook, Apple, Amazon).
- SAML and OIDC federation with enterprise identity providers.
- JWT-based access and ID tokens.
- Custom authentication flows via Lambda triggers.
- Adaptive authentication with risk-based policies.
- User import and export via CSV.
- AWS WAF integration for DDoS protection.

**Production Considerations**:
- Cognito has a "soft limit" of 10,000 active users per user pool. For larger user bases, create multiple pools or use a custom user store with a Lambda trigger.
- The hosted UI is customizable but limited. For complex UI requirements, use the Cognito SDK with a custom frontend.
- Lambda triggers (pre sign-up, post confirmation, pre authentication, etc.) add latency to authentication flows. Keep trigger functions lightweight.
- Cognito Advanced Security Features incur additional costs but provide valuable protection against credential stuffing and account takeover.
- Cognito integrates with API Gateway and ALB for token-based authorization. This is the recommended pattern for serverless applications.

### Auth0

Auth0 is a cloud-based identity platform that provides OAuth2 and OpenID Connect as a service. It is one of the most popular identity-as-a-service (IDaaS) platforms and offers a generous free tier.

**Architecture**: Auth0 provides a fully managed authorization server. The platform handles token issuance, user management, password policies, MFA, and security monitoring. Auth0's architecture is multi-tenant, with each tenant isolated from others.

**Key Features**:
- OAuth2 and OpenID Connect certified by the OpenID Foundation.
- Social login with 50+ identity providers.
- Enterprise federation with SAML, WS-Federation, and OIDC.
- Passwordless authentication (email, SMS, magic links).
- Multi-factor authentication with push notifications, TOTP, SMS, and biometrics.
- Anomaly detection and brute-force protection.
- Custom database connections for existing user stores.
- Actions (custom logic via Node.js) for extending authentication flows.
- Organizations feature for multi-tenant B2B applications.
- Role-based access control with permissions.
- Audit logs with 30-day retention (extendable).

**Production Considerations**:
- Auth0's tenant limit is configurable. For large deployments, enterprise plans provide higher limits.
- Custom domains are available on paid plans. Without a custom domain, the Auth0 domain appears in the URL (e.g., `tenant.us.auth0.com`).
- The Actions engine runs custom Node.js code during the authentication pipeline. Actions are isolated and have a timeout of 20 seconds.
- Auth0 supports "delegated administration" — organizations can manage their own users and roles without access to the Auth0 dashboard.
- Rate limits vary by plan. The free tier allows 7 requests per second. Enterprise plans offer higher limits and burst capacity.
- Auth0's Management API allows full programmatic control over tenants, making it suitable for CI/CD-driven identity management.

### Okta

Okta is an enterprise-grade identity platform that provides OAuth2, OpenID Connect, and SAML services. It is the most widely adopted identity platform in the enterprise market.

**Architecture**: Okta's architecture is built around the concept of an "Okta tenant" — an isolated instance of the Okta service. Each tenant has its own OAuth2 authorization server, user directory, and policy configuration. Okta provides both a cloud service and on-premises deployment options.

**Key Features**:

- Okta's event hooks allow integration with external systems for real-time identity event processing.

## Lessons Learned: 5 Key Patterns

After examining how Google, GitHub, Facebook, Slack, and the major open-source and cloud solutions implement OAuth2, several recurring patterns emerge. These are the lessons that experienced teams apply when building or integrating OAuth2 systems.

### Pattern 1: Always Use PKCE

The Proof Key for Code Exchange (PKCE) extension was originally designed for mobile and native applications, but every production deployment should use it regardless of the client type. PKCE prevents authorization code interception attacks by binding the authorization code to the client that initiated the flow.

**Why it matters**: Without PKCE, an intercepted authorization code can be exchanged by an attacker. With PKCE, the attacker would need the original `code_verifier` value, which is never transmitted during the authorization request. Google, Auth0, and Okta all require PKCE for mobile and SPA clients. GitHub and Slack strongly recommend it.

**Implementation**: The client generates a random `code_verifier` (minimum 43 characters, maximum 128 characters, using unreserved characters). The `code_challenge` is the base64url-encoded SHA-256 hash of the verifier. The challenge is sent with the authorization request; the verifier is sent with the token request.

### Pattern 2: Token Rotation Limits Blast Radius

Short-lived access tokens combined with refresh token rotation significantly reduce the impact of token leakage. Every major platform implements some form of token lifetime management.

**The blast radius principle**: If an access token is compromised, the attacker can use it only until it expires. If the refresh token is also rotated on each use, a compromised refresh token becomes invalid after the next legitimate use. This creates a "race" between the attacker and the legitimate client.

**Implementation guidelines**:
- Access tokens: 1 hour lifetime (or less for high-security applications).
- Refresh tokens: Enable rotation, where each refresh produces a new refresh token and invalidates the old one.
- Refresh token reuse detection: If the authorization server detects that a previously rotated refresh token is being used, it immediately invalidates all tokens for that client and user.
- Set a maximum refresh token lifetime (e.g., 90 days) after which the user must re-authenticate.

**Real-world examples**: Auth0 enables refresh token rotation by default for single-page applications. Google's refresh tokens are long-lived but can be revoked by the user at any time. GitHub's installation tokens expire after 1 hour, forcing frequent rotation.

### Pattern 3: Separate Authentication from Authorization

The most successful OAuth2 deployments treat authentication (who the user is) and authorization (what the client can do) as separate concerns. This is the principle that Hydra and Dex are built on.

**The separation principle**:
- **Authentication**: Handled by an identity provider (IdP) that verifies user credentials. The IdP may use MFA, SSO, password policies, and other authentication mechanisms.
- **Authorization**: Handled by the OAuth2 authorization server. It issues tokens based on the authentication result and the client's requested scopes.

**Why this matters**: Separating the two concerns allows each system to evolve independently. The authentication system can be upgraded to support biometrics, passkeys, or hardware security keys without changing the OAuth2 flow. The authorization server can add new grant types, token formats, or scope models without affecting the authentication pipeline.

**Implementation patterns**:
- Ory Hydra uses a consent app as the bridge between authentication and authorization.
- Keycloak combines both but allows custom authentication flows.
- Cognito uses Lambda triggers to customize authentication behavior.
- Auth0 Actions allow custom logic at every stage of the authentication pipeline.

### Pattern 4: Design for Token Validation Without Network Calls

In production systems, the token validation path must be fast and reliable. The most common approach is to use JWTs (JSON Web Tokens) that can be validated locally without a network call to the authorization server.

**The validation trade-off**:
- **Opaque tokens**: Must be validated by calling the authorization server's introspection endpoint. This adds latency, increases load on the authorization server, and introduces a network dependency. However, opaque tokens can be instantly revoked.
- **JWT tokens**: Can be validated locally by verifying the signature, expiration, and claims. This is fast and does not require network access to the authorization server. However, JWTs cannot be revoked before their expiration time (unless a revocation list or short expiration is used).

**Production patterns**:
- Use JWTs for access tokens with a short expiration (15-60 minutes).
- Maintain a local cache of the authorization server's JWKS (JSON Web Key Set) and refresh it periodically.
- For token revocation, use a distributed revocation list (e.g., Redis) or accept the short window between token issuance and expiration.
- For high-security scenarios, combine JWTs with a pre-authorization check for the first request, then rely on local validation for subsequent requests.

**Real-world examples**: Google's ID tokens are JWTs that can be verified using Google's published JWKS. Auth0 provides both opaque and JWT tokens, with the JWT format being the default for new applications. Keycloak primarily uses JWTs and provides a JWKS endpoint at `/auth/realms/{realm}/protocol/openid-connect/certs`.

### Pattern 5: Graceful Degradation and Error Handling

OAuth2 introduces multiple points of failure in the authentication flow. Production systems must handle every failure mode gracefully.

**Common failure modes**:
1. **Authorization server unavailable**: The user cannot authenticate. The application should show a clear error message and offer alternatives (retry, contact support, or use a different authentication method).
2. **Token endpoint timeout**: The token exchange request times out. The client should retry with exponential backoff and jitter.
3. **Invalid redirect URI**: The authorization server rejects the redirect URI. This is usually a configuration error that should be caught during development, not production.
4. **Expired authorization code**: The user waited too long to complete the flow. The application should redirect the user to start the flow again.
5. **Scope consent denied**: The user denied some requested scopes. The application should handle this gracefully — either proceed with reduced functionality or explain why the scope is needed.
6. **Token revocation**: The user revoked the application's access. The application should detect this on the next API call and prompt the user to re-authenticate.

**Implementation patterns**:
- Implement retry logic with exponential backoff for all OAuth2 HTTP requests.
- Use the `state` parameter for CSRF protection and include a nonce that can be verified when the user returns.
- Validate the `iss` (issuer) claim in ID tokens to prevent token substitution attacks.
- Log all OAuth2 errors at appropriate levels — auth failures at INFO, token validation failures at WARN, and unexpected errors at ERROR.
- Monitor OAuth2 error rates with alerts for abnormal patterns (e.g., a sudden spike in token exchange failures may indicate a token harvesting attack).

## Bridge to Interview

Understanding OAuth2 in production is essential for system design interviews. When an interviewer asks you to design a system that involves authentication or authorization, OAuth2 is almost always the right answer — but knowing how to adapt it to the specific constraints of the problem is what separates strong candidates from average ones.

### Key Interview Questions to Practice

**Design a "Sign in with Google" for a social media platform**: Walk through the full OAuth2 flow, including PKCE, token handling, and session management. Discuss how you would handle token refresh, revocation, and cross-origin requests.

**Design an API gateway with OAuth2**: Discuss how the gateway would validate tokens, enforce scopes, and rate-limit requests. Compare JWT validation vs. token introspection. Discuss how to handle token caching and revocation.

**Design an enterprise SSO system**: Discuss how OAuth2 and SAML interact, how to federate with external identity providers, and how to handle user provisioning and deprovisioning. Consider the role of OpenID Connect and how it simplifies the SSO architecture.

**Design a multi-tenant SaaS platform with OAuth2**: Discuss tenant isolation, per-tenant authorization servers, and how to handle cross-tenant resource access. Consider the use of Auth0 Organizations or Okta's multi-tenancy features.

### Deeper Dives

For further study, explore these topics:

1. **OpenID Connect Discovery**: How clients discover the authorization server's endpoints, supported scopes, and signing keys via the `/.well-known/openid-configuration` endpoint.

2. **OAuth2 Token Exchange**: How to exchange one token for another with different scopes or audience, as defined in RFC 8693. This is essential for service-to-service delegation.

3. **OAuth2 Device Authorization Grant**: The "device code" flow for devices with limited input capabilities (smart TVs, CLI tools, IoT devices). This is how GitHub CLI authenticates users.

4. **OAuth2 for Machine-to-Machine**: The client credentials grant and how it differs from user-facing flows. Discuss when to use client credentials vs. JWT bearer tokens.

5. **OAuth2 Security Best Practices**: The OAuth2 Security Best Current Practice document (RFC 9700) covers the latest security recommendations, including why the implicit grant is deprecated and why resource owner password credentials grant is discouraged.
