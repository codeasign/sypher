## The Path From Naive to Production

OAuth2 looks simple on paper. The spec defines four grant types, a few endpoints, and a token format. Everything fits in a single RFC document. But when you deploy OAuth2 at internet scale -- millions of users, thousands of client applications, multiple regions, sub-millisecond p99 latency requirements -- the architecture transforms from a simple HTTP endpoint into a distributed system with its own data stores, replication strategy, and capacity planning model.

This page walks you through that evolution. You start with the naive approach that works for a weekend project. Then you build the production-grade architecture that powers the authorization systems of the largest platforms on the internet. By the end, you will understand the architectural decisions behind every major identity provider and how to scale OAuth2 from a single instance to a global multi-region deployment.

The journey covers five stages, each building on the previous one:
- **Naive:** a single shared secret, no introspection, no revocation, one process
- **Production:** dedicated auth server, per-client registration, introspection, revocation, data layer
- **Integration:** API gateway, web apps, mobile apps, server-to-server communication
- **Multi-region:** global deployment, token consistency, regional failover, cross-region replication
- **Capacity:** token issuance QPS, storage sizing, introspection throughput, Redis cluster sizing

## Stage 1: Naive Approach -- Embedded Client Credentials

The simplest OAuth2 deployment embeds client credentials directly into the client application. A mobile app or single-page application ships with a hardcoded `client_id` and `client_secret`. The web backend exposes a single `/token` endpoint. Every client shares the same secret. There is no client registration database, no introspection endpoint, no revocation list, and no audit trail.

### What Breaks at 1,000 Users

Nothing. The naive approach works fine. A single server can handle 1,000 users making occasional authentication requests. The token endpoint processes a few requests per second. The JWT signing is fast enough. Users are happy.

### What Breaks at 10,000 Users

**Secret leakage becomes inevitable.** With 10,000 users, the mobile app has been downloaded and decompiled by security researchers. The shared secret is on GitHub. The SPA bundle has been inspected. The secret is in the logs of every deployment. You cannot fix this without updating every client.

**No revocation starts to hurt.** A user changes their password but their old token remains valid for days. A compromised token is shared on a forum. You cannot revoke it. The only option is to rotate the signing key, which logs out every user.

### What Breaks at 100,000 Users

**Single point of failure causes outages.** The auth server crashes during a deployment. Every client, every service, every user is locked out. The entire platform is down for 30 minutes while the server is restarted. The incident is a P0 for the on-call engineer.

**No client differentiation causes security incidents.** A mobile app is compromised, but you cannot tell which requests are from the compromised app versus the web app versus the backend service. You cannot block the compromised client without blocking all clients.

**Audit trail gaps.** A security incident requires understanding which client accessed which resource. The naive implementation has no audit trail. You cannot trace the attack to its source.

### Scaling Limits

| Dimension | Naive Limit |
|---|---|
| Token issuance | ~1,000 tokens/second (single process) |
| Concurrent clients | 1 (one shared credential) |
| Token validation | 0 (no introspection) |
| Secrets | 1 shared secret |
| Revocation | Not supported |
| Audit trail | None |
| Client types | No differentiation |
| Uptime | Single point of failure |
| Rotation | Not possible without all-client update |
| Token expiry | Days or weeks (no refresh mechanism) |

### Key Architectural Decisions

**Per-client credential registration.** Every client application registers with a unique `client_id`. Confidential clients (backend services) also receive a `client_secret`, stored as a bcrypt hash. Public clients (mobile apps, SPAs) use PKCE instead of a secret. This means one client leaking credentials does not affect any other client. You can rotate a single client's secret without touching any other client. The client registration database stores the allowed redirect URIs, the allowed scopes, the client type, and the client secret hash.

**Token introspection.** Resource servers call the introspection endpoint before accepting a token. This is mandatory for opaque tokens (random strings) and recommended for JWT tokens when revocation is required. The introspection endpoint checks expiration, revocation status, and scope in a single atomic operation. The response includes the `active` boolean, the user identifier (`sub`), the scope, the client ID, and the expiration timestamp. The introspection endpoint is authenticated with a bearer token or basic auth to prevent unauthorized introspection queries.

**Token revocation.** The revocation endpoint adds a token identifier to a revocation list. The list is stored in Redis with a TTL equal to the maximum token lifetime. A background cron job removes expired entries. Resource servers check the revocation list during introspection or as a local cache lookup. The revocation endpoint is idempotent: calling it twice with the same token has the same effect as calling it once. The revocation endpoint accepts both access tokens and refresh tokens.

**Separate control plane and data plane.** The authorization server handles token issuance, introspection, and revocation (control plane). Resource servers validate tokens and enforce scopes (data plane). This separation allows each plane to scale independently. The control plane is CPU-bound (cryptographic operations) and I/O-bound (Redis writes). The data plane is I/O-bound (cache lookups) and can be scaled horizontally with the API gateway. The control plane runs on auto-scaled instances with CPU-based scaling policies. The data plane runs on the API gateway or on the resource server sidecar.

**Client secret storage.** Client secrets are never stored in plaintext. The registration endpoint receives the secret, hashes it with bcrypt (cost factor 10-12), and stores the hash. The token endpoint receives the secret, hashes it, and compares the hash. This means a database breach does not expose client secrets. The tradeoff is that bcrypt verification adds 10-50ms to each token request. For high-throughput systems, consider using Argon2id or a hardware security module (HSM) for client authentication.

**Authorization code lifecycle.** Authorization codes are single-use, short-lived (10 minutes), and stored in Redis with a TTL. The code is bound to the PKCE code challenge, the redirect URI, and the client. When the token endpoint exchanges the code, it atomically deletes the code from Redis. If the code is used twice, the second request fails because the code no longer exists. This prevents replay attacks. The code is a cryptographically random string (minimum 128 bits of entropy).

### The Authorization Code Flow Step by Step

Understanding the full flow helps you design the architecture correctly. Here is the authorization code flow with PKCE, the most common grant type for user-facing applications:

1. **Client initiates.** The mobile app or SPA generates a `code_verifier` (random string, 43-128 characters) and a `code_challenge` (SHA-256 hash of the verifier). The app opens the system browser to the `/authorize` endpoint with the `client_id`, `redirect_uri`, `scope`, `state`, `code_challenge`, and `code_challenge_method`.

2. **Authorization server authenticates the user.** The user logs in (or is already logged in via SSO). The authorization server presents a consent screen showing the requested scopes. The user approves.

3. **Authorization server redirects.** The authorization server redirects the browser to the `redirect_uri` with an `authorization_code` and the `state` parameter. The code is single-use and expires in 10 minutes.

4. **Client exchanges the code.** The app captures the redirect (the system browser redirects to a custom scheme or a universal link). The app sends the `authorization_code`, `code_verifier`, `client_id`, and `redirect_uri` to the `/token` endpoint.

5. **Authorization server validates.** The server looks up the stored authorization code, verifies the `code_verifier` against the `code_challenge` using SHA-256, validates the `client_id` and `redirect_uri`, and checks that the code has not expired and has not been used before.

6. **Authorization server issues tokens.** The server generates an `access_token` (JWT, 15-60 minute lifetime), a `refresh_token` (opaque, 7-30 day lifetime), and optionally an `id_token` (OpenID Connect). The server stores the refresh token and the authorization code (marked as used) in Redis.

7. **Client uses the access token.** The app includes the access token in the `Authorization: Bearer` header of every API request. The API gateway validates the token before forwarding the request.

8. **Client refreshes the token.** When the access token expires, the app sends the `refresh_token` to the `/token` endpoint with `grant_type=refresh_token`. The server validates the refresh token, invalidates the old one, and issues a new access token and a new refresh token (rotation).

### Token Format Decision: JWT vs Opaque

| Criterion | JWT | Opaque |
|---|---|---|
| Self-contained | Yes | No |
| Server-side storage | Not needed for validation | Required |
| Revocation | Requires revocation list | Inherent (delete from store) |
| Token size | ~1-2 KB | ~32 bytes |
| Debugging | Decode and inspect | Call introspection |
| Cross-region validation | Local (signature verify) | Requires introspection round trip |
| Information disclosure | Payload visible to holder | No payload |

The decision depends on your architecture. If you have a single region with a fast Redis cluster, opaque tokens are simpler and more secure. If you need multi-region validation without cross-region round trips, JWT tokens are the better choice. Many large-scale systems use both: JWTs for access tokens (validated locally) and opaque tokens for refresh tokens (always stored server-side).

### Signing Key Management

Production OAuth2 systems rotate signing keys on a schedule. The system maintains two active keys: the current signing key and the previous signing key. Tokens signed with the previous key remain valid until they expire. The JWKS endpoint exposes both keys.

- **Key rotation frequency:** weekly or daily, depending on the security posture
- **Key types:** RSA-2048 or ECDSA P-256 (ECDSA produces smaller signatures and is faster to verify)
- **JWKS endpoint:** well-known URL (`/.well-known/jwks.json`) that returns the public keys
- **Key identifiers:** each key has a `kid` (key ID) in the JWT header and the JWKS response
- **Key revocation:** if a key is compromised, the system generates a new key immediately and rejects all tokens signed with the compromised key
- **Key storage:** private keys are stored in a hardware security module (HSM) or a secrets manager (AWS KMS, HashiCorp Vault)

### Client Registration and Management

The client registration endpoint is the administrative interface for managing OAuth2 clients. It is a separate API from the token and authorization endpoints, with its own authentication and authorization model.

**Client registration flow:**

1. **Admin authenticates.** An administrator authenticates to the client registration API using a separate admin API key or an OAuth2 token with the `clients:admin` scope.

2. **Admin submits client details.** The admin provides the client name, type (`public` or `confidential`), redirect URIs, allowed scopes, and optionally a logo URI and privacy policy URL.

3. **Server generates credentials.** For confidential clients, the server generates a cryptographically random client secret (32 bytes encoded as base64, 128 bits of entropy). The server stores the bcrypt hash of the secret. The raw secret is returned to the admin exactly once, in the registration response.

4. **Server returns client metadata.** The response includes the `client_id`, `client_secret` (for confidential clients), `client_id_issued_at`, and `client_secret_expires_at`. The client secret can be set to never expire or to expire after a configurable period.

5. **Admin distributes credentials.** The admin securely distributes the `client_id` and `client_secret` to the client application. The secret is stored in a secrets manager, not in a configuration file.

**Client registration database schema:**

| Column | Type | Description |
|---|---|---|
| id | VARCHAR(128) | Unique client identifier (e.g., "mobile_ios_v3") |
| type | ENUM("public", "confidential") | Client type determines authentication method |
| client_secret_hash | VARCHAR(256) | Bcrypt hash of the client secret (NULL for public clients) |
| redirect_uris | JSONB | Array of allowed redirect URIs |
| scopes | JSONB | Array of allowed scopes |
| grant_types | JSONB | Array of allowed grant types |
| token_endpoint_auth_method | VARCHAR(32) | "client_secret_basic", "client_secret_post", "private_key_jwt", or "none" |
| client_name | VARCHAR(256) | Human-readable client name |
| client_uri | VARCHAR(512) | Link to the client's homepage |
| policy_uri | VARCHAR(512) | Link to the client's privacy policy |
| tos_uri | VARCHAR(512) | Link to the client's terms of service |
| created_at | TIMESTAMP | Registration timestamp |
| updated_at | TIMESTAMP | Last update timestamp |

**Client secret rotation.** The client registration endpoint supports secret rotation. The admin calls a `POST /clients/{id}/rotate-secret` endpoint. The server generates a new secret, updates the hash, and returns the new raw secret. The old secret remains valid for a configurable grace period (typically 24 hours) to allow clients to update their configuration without downtime.

### Redis Data Model and Key Design

The Redis cluster stores several types of data with different TTLs and access patterns. The key naming convention uses colons to create a hierarchy, which enables Redis key space scanning and prefix-based deletion.

**Authorization codes:**

```
KEY:   authcode:{code_hash}
VALUE: { "client_id": "mobile_ios_v3", "redirect_uri": "...", "code_challenge": "...", "code_challenge_method": "S256", "nonce": "...", "scope": "openid profile", "auth_time": 1711843200, "acr": "urn:mace:incommon:iap:silver" }
TTL:   600 seconds (10 minutes)
```

**Access tokens (opaque):**

```
KEY:   access:{token_hash}
VALUE: { "sub": "user_8421", "client_id": "mobile_ios_v3", "scope": "profile:read", "iss": "https://auth.example.com", "iat": 1711843200, "exp": 1711846800, "azp": "authorization_code" }
TTL:   3600 seconds (1 hour)
```

**Refresh tokens:**

```
KEY:   refresh:{token_hash}
VALUE: { "sub": "user_8421", "client_id": "mobile_ios_v3", "scope": "profile:read offline_access", "family_id": "fam_abc123", "rotation_count": 3, "exp": 1714435200 }
TTL:   2592000 seconds (30 days)
```

**Refresh token family.** When refresh token rotation is enabled, all rotated refresh tokens belong to the same family. The `family_id` links tokens across rotations. If a token from a family is reused (the old token is used after the new token has been issued), the server detects the anomaly and revokes all tokens in the family.

**Revocation list:**

```
KEY:   revoked:{token_jti}
VALUE: { "revoked_at": 1711843200, "reason": "user_logout", "revoked_by": "client_app" }
TTL:   max_token_lifetime (e.g., 86400 seconds for 24-hour max)
```

**Client registration cache:**

```
KEY:   client:{client_id}
VALUE: { "type": "confidential", "scopes": ["payments:write"], "redirect_uris": ["..."], "token_endpoint_auth_method": "client_secret_basic" }
TTL:   3600 seconds (1 hour, can be evicted early)
```

**Rate limiting:**

```
KEY:   ratelimit:{client_id}:{endpoint}:{window}
VALUE: counter (incremented via Redis INCR)
TTL:   window_size (e.g., 60 seconds for a 1-minute window)
```

## Stage 3: Integration Patterns

The production architecture above is the core. But OAuth2 does not live in isolation. You integrate it with your API gateway, your web applications, your mobile apps, and your server-to-server communication. Each integration pattern has different requirements for token validation, caching, and security.

The API gateway pattern has several advantages:

**Centralized policy enforcement.** All token validation logic lives in one place. You update the validation rules once, and every route benefits. Common policies include scope enforcement, rate limiting by client, IP allowlisting, and token lifetime checks. If you need to add a new policy (e.g., geo-fencing tokens to specific regions), you add it to the gateway middleware. The gateway is also the ideal place to implement token binding (binding a token to a specific TLS session or device fingerprint).

**Reduced backend complexity.** Backend services trust the gateway headers. They do not need to parse JWTs, fetch JWKS keys, or call introspection endpoints. This reduces the OAuth2 surface area across your service mesh. A backend service can focus on business logic. The gateway headers include the user ID, scope, client ID, and token ID. Backend services use these headers for authorization decisions, rate limiting, and audit logging.

**Performance optimization.** The gateway caches JWKS responses, token introspection results, and revocation list checks. A single gateway instance can validate 50,000+ tokens per second with sub-millisecond p99 latency for cached tokens. The introspection endpoint is called only on cache miss. With a 60-second cache TTL, the cache hit rate is 95%+ for most workloads. The gateway also maintains a local in-memory cache of the JWKS keys, which are rotated at most once per day.

**Audit trail.** The gateway logs every token validation attempt, including the client, user, scope, route, and result. This produces a complete audit trail of who accessed what resource, when, and whether access was granted or denied. The audit log is shipped to a centralized logging system (Elasticsearch, Splunk, or a data lake) for security analysis and incident response. The gateway also logs the token ID, which can be correlated with the authorization server's token issuance logs.

**Rate limiting by client.** The gateway enforces rate limits per client_id. If a mobile app with a leaked token is making 10,000 requests per second, the gateway detects the anomaly and rate-limits the client. The rate limit configuration is stored in the client registration database and distributed to the gateway via a configuration management system. Rate limits are enforced at the gateway level, before the request reaches the backend services.

### Integration with Web Applications

Web applications use the Authorization Code Grant with PKCE. The flow involves a browser redirect to the authorization server, a callback to the web application, and a token exchange.

The web application never sees the access token directly. The authorization server issues the token to the browser, which stores it in an HTTP-only cookie or a secure session store. The browser sends the token to the web application's backend, which validates it before proxying to the API gateway.

Key considerations for web applications:

**Session management.** The web application maintains its own session cookie (server-side session). The OAuth2 access token is bound to the session, not directly to the browser session cookie. This means the OAuth2 token can be refreshed or rotated without invalidating the user's login session. The session store maps the session ID to the user ID, the access token, and the refresh token. The session store is typically Redis or a database.

**Token refresh.** The web application uses a refresh token to obtain new access tokens without user interaction. The refresh token is stored in the server-side session, not in the browser. This prevents XSS attacks from stealing the refresh token. The refresh flow is triggered when the API gateway returns a 401 response, indicating the access token has expired. The web application intercepts the 401, calls the refresh endpoint, and retries the original request with the new access token.

**Logout propagation.** When a user logs out, the web application must: (1) clear its session cookie, (2) call the revocation endpoint to revoke the access token, and (3) call the authorization server's end-session endpoint to propagate logout to other applications. The OpenID Connect RP-Initiated Logout specification formalizes this flow. Without logout propagation, the user remains logged into other applications that share the same authorization server.

**CSRF protection.** The authorization flow includes a `state` parameter that is generated by the web application and verified after the redirect. This prevents CSRF attacks on the authorization callback. The state parameter is a cryptographically random string stored in the session. The authorization server echoes the state parameter back to the web application, which verifies that it matches the stored value.

### Integration with Mobile Applications

Mobile applications are the most challenging integration target. The app can be decompiled, the network can be intercepted, and the device can be compromised. The OAuth2 best practices for mobile apps are:

**Use Authorization Code Grant with PKCE.** The mobile app generates a code verifier (random string) and a code challenge (SHA-256 hash of the verifier). The authorization server stores the code challenge. When the app exchanges the authorization code, it sends the original verifier. The server verifies that the verifier matches the stored challenge. This prevents authorization code interception attacks.

**Use the system browser, not a WebView.** The authorization flow should open the device's default browser (or a Chrome Custom Tab / SFSafariViewController). A WebView cannot access the system's credential manager, cookies, or saved passwords. The system browser also provides Single Sign-On (SSO): if the user is already logged into the identity provider, the authorization flow proceeds without an additional login prompt.

**Use biometric authentication for token storage.** Access tokens and refresh tokens are stored in the platform's secure enclave (iOS Keychain, Android Keystore). The app requests biometric authentication before accessing the token store. This prevents token theft from a compromised device.

**Implement token rotation.** Refresh tokens are rotated on every use. Each refresh token is single-use. The authorization server issues a new refresh token with each refresh response. If a refresh token is stolen and used twice, the server detects the reuse and revokes all tokens for that user and client. This is called refresh token rotation and is a mandatory security practice for mobile apps.

**Handle token loss gracefully.** If the refresh token is lost (e.g., the user clears app data), the app must fall back to the full authorization flow. The app should detect a 401 response from the refresh endpoint and redirect the user to the authorization endpoint.

Key considerations for server-to-server:

**Token scope is the security boundary.** Each service registers with the minimum set of scopes it needs. The payment service gets `inventory:read` and `orders:write` but not `users:admin`. Scopes are enforced by the resource server, not the client. If the payment service is compromised, the blast radius is limited to its scopes. This is the principle of least privilege applied to service-to-service communication.

**Client credentials are not user credentials.** The access token represents the service, not a user. If you need to act on behalf of a user from a server-to-server flow, use the Token Exchange extension (RFC 8693) to exchange a user token for a service token with delegated authority. The token exchange endpoint accepts a subject token (user token) and returns an impersonation token that includes both the service identity and the user identity.

**Token caching.** Backend services should cache access tokens. The token is valid for the full `expires_in` duration. Caching avoids a token endpoint round trip on every request. Use a local cache with a TTL slightly shorter than the token lifetime (e.g., 55 minutes for a 60-minute token) to avoid edge-of-expiry race conditions. The cache is a simple in-memory map with a TTL eviction policy.

**Mutual TLS (mTLS).** For high-security environments, use mTLS instead of a client secret. The authorization server issues a client certificate to each service. The service presents the certificate during the token request. This eliminates the risk of client secret leakage in CI/CD pipelines or configuration files. mTLS is specified in RFC 8705 (OAuth 2.0 Mutual-TLS Client Authentication and Certificate-Bound Access Tokens). Certificate-bound access tokens (also called token binding) bind the token to the TLS session, preventing token replay.

**Token provenance.** The token includes claims that identify the calling service, the intended audience, and the scope. The resource server validates that the token's audience matches its own service name. This prevents token replay across services: a token for the inventory service cannot be used to call the payments service. The `aud` claim is a unique identifier for the target service.

### Security Considerations for Integration Patterns

Each integration pattern introduces specific security risks that you must address:

**API gateway security.** The gateway headers (X-User-Id, X-User-Scope, etc.) are trusted only if the backend service is not directly accessible from the internet. Ensure that backend services are only reachable through the gateway, not directly. Use network policies (security groups, network ACLs) to enforce this. The gateway should strip any incoming X-User-* headers from the client before injecting its own headers, preventing header injection attacks.

**Backend service identity.** Backend services that call the introspection endpoint must authenticate themselves. Use a separate service account token for each backend service, with the `introspect:read` scope. The introspection endpoint logs the caller's identity for audit purposes.

**Mobile app certificate pinning.** The mobile app should pin the authorization server's TLS certificate. This prevents man-in-the-middle attacks where an attacker intercepts the authorization code or the token response. Certificate pinning is configured in the app's network security configuration.

**Web app CSRF on the token exchange.** The token exchange endpoint must validate that the authorization code was issued to the same client that is exchanging it. The client authenticates with its client secret (confidential clients) or the PKCE verifier (public clients). The `redirect_uri` must match the one used in the authorization request.

**Server-to-server credential rotation.** Backend service client secrets should be rotated regularly. Automated rotation is possible with a secrets manager that supports dynamic secrets (e.g., HashiCorp Vault). The client secret is rotated every 30-90 days, with a grace period where both the old and new secrets are accepted.

### Monitoring and Observability

A production OAuth2 system requires comprehensive monitoring across all components. The following metrics should be collected and alerted on:

**Token issuance metrics:**

- `oauth2.token.issued.count` -- rate of token issuance per grant type
- `oauth2.token.issued.latency_p50` -- median latency of token requests
- `oauth2.token.issued.latency_p99` -- p99 latency of token requests
- `oauth2.token.issued.error_count` -- rate of token request errors (invalid client, invalid grant, expired code)
- `oauth2.token.issued.error_rate` -- error rate as a percentage of total requests

**Introspection metrics:**

- `oauth2.introspect.count` -- rate of introspection requests
- `oauth2.introspect.latency_p50` -- median latency of introspection requests
- `oauth2.introspect.latency_p99` -- p99 latency of introspection requests
- `oauth2.introspect.cache_hit_ratio` -- fraction of introspection requests served from cache
- `oauth2.introspect.active_ratio` -- fraction of tokens reported as active

**Revocation metrics:**

- `oauth2.revoke.count` -- rate of revocation requests
- `oauth2.revoke.reason_logout` -- revocations due to user logout
- `oauth2.revoke.reason_admin` -- revocations due to admin action
- `oauth2.revoke.reason_compromise` -- revocations due to suspected compromise
- `oauth2.revoke.reason_rotation` -- revocations due to refresh token rotation

**Client registration metrics:**

- `oauth2.client.register.count` -- rate of new client registrations
- `oauth2.client.rotate.count` -- rate of client secret rotations
- `oauth2.client.active_count` -- total number of active clients

**Redis cluster metrics:**

- `redis.memory.used_bytes` -- memory usage against the maxmemory limit
- `redis.keyspace.hits` -- cache hit count
- `redis.keyspace.misses` -- cache miss count
- `redis.keyspace.hit_ratio` -- cache hit ratio (should be above 95%)
- `redis.cpu.usage` -- CPU utilization of Redis nodes
- `redis.replication.lag` -- replication lag between master and replica
- `redis.connected_clients` -- number of connected clients

### Token Consistency Strategy

Multi-region OAuth2 introduces a consistency challenge. A token issued in US-East must be introspectable in AP-East. A token revoked in AP-East must be rejected in EU-West. The latency of cross-region replication determines the window of inconsistency.

**Option 1: JWT tokens with short expiry.** The simplest approach: use JWT access tokens with a short expiry (5-15 minutes). Each region validates the JWT locally using the public signing key and the JWKS endpoint. No cross-region introspection is needed. The tradeoff is that revocation takes effect only after the token expires. For most applications, a 5-minute revocation window is acceptable. The short expiry also limits the blast radius of a leaked token.

**Option 2: Global Redis with active-active replication.** Deploy Redis Enterprise with active-active cross-region replication. Each region writes to its local Redis cluster, and Redis Enterprise syncs the data across regions. Token cache entries and revocation list entries are available in all regions within seconds. This approach requires Redis Enterprise, which is expensive but provides near-synchronous consistency. The tradeoff is cost and operational complexity.

**Option 3: Regional token issuance with regional validation.** Each region issues its own tokens with a region-specific signing key. A token issued in US-East includes a `reg` claim with the value `us-east`. Resource servers in AP-East that receive a US-East token fetch the US-East JWKS (which is cached for 5 minutes) and validate locally. Revocation lists are replicated via a global message bus (Kafka or Pulsar). This is the approach used by large-scale identity providers.

**Option 4: Centralized introspection with regional caching.** All regions call the same introspection endpoint in the primary region. Regional caches reduce the latency of repeated calls. A cache miss in AP-East requires a cross-region round trip to US-East (200-300ms). This is the simplest approach but the highest latency on cache miss. It is suitable for read-heavy workloads with a high cache hit rate.

### Cross-Region Write Path

Writes (client registration, authorization code creation, token revocation) all go to the primary region's database. Read replicas in each region serve client lookups. The replication lag between primary and replicas is typically 50-500ms, depending on geographic distance.

If a client registers in EU-West, the write is forwarded to US-East. The EU-West replica sees the new client within 100-500ms. During that window, the EU-West auth server returns a stale result. This is acceptable for client registration (a rare operation) but not for token revocation.

For token revocation, the system uses a hybrid approach: the revocation is written to the primary database AND published to a global message bus (Kafka or Pulsar). Each region's Redis cluster subscribes to the revocation topic and updates its local revocation list immediately. This reduces the revocation propagation delay from seconds to milliseconds.

### Regional Failover

If the primary region goes down, the system must fail over to a secondary region. The failover process involves:

1. **Health check failure detection.** A global health checker (running in a third region or a cloud provider's health check service) detects that the primary region's auth server is unreachable. The health checker checks the `/health` endpoint every 5 seconds from three different vantage points.

2. **DNS update.** The global DNS routing updates the primary region's entry to point to the secondary region. DNS TTLs are set to 60 seconds or lower. The DNS update propagates within 60 seconds (the TTL).

3. **Database promotion.** The secondary region's read replica is promoted to primary. This operation takes 30-120 seconds, depending on the database engine and the replication lag. During this window, the secondary region's auth server can only read data (client registrations, authorization codes) but cannot write.

4. **Auth server restart.** The auth server instances in the secondary region are restarted with the new primary database endpoint. The local Redis cache is warm (populated from the previous read traffic). The cache hit rate is preserved during the failover.

5. **Traffic draining.** The primary region's traffic is gradually drained. Existing connections are allowed to complete within a grace period (typically 5 minutes). New connections are routed to the secondary region.

### Global Consistency Tradeoffs

| Consistency Model | Revocation Latency | Cross-Region Cost | Complexity |
|---|---|---|---|
| JWT + short expiry | 5-15 minutes | Low | Low |
| Global Redis active-active | 1-5 seconds | High | High |
| Regional signing + message bus | 100-500ms | Medium | Medium |
| Centralized introspection + cache | 200-300ms (cache miss) | Medium | Low |

## Stage 5: Capacity Planning

You need to plan for three dimensions of OAuth2 capacity: token issuance throughput, token storage, and introspection throughput. Each dimension has different scaling characteristics.

### Token Issuance Throughput

Token issuance is the rate at which the authorization server can produce new tokens. This includes the authorization code flow (two round trips: `/authorize` and `/token`) and the client credentials flow (one round trip: `/token`).

**Request volume estimation:**

| Client Type | Daily Active Users | Token Requests Per User Per Day | Total Token Requests Per Day |
|---|---|---|---|
| Mobile app | 10,000,000 | 3 (login + 2 refreshes) | 30,000,000 |
| Web app | 5,000,000 | 5 (login + 4 refreshes) | 25,000,000 |
| Server-to-server | 500 services | 1,000 (each service polls every 50s) | 500,000 |
| Total | | | 55,500,000 |

Peak QPS: Daily requests / 86,400 seconds multiplied by 3x peak factor = 55,500,000 / 86,400 multiplied by 3 = approximately 1,927 QPS.

**Sizing the token endpoint:**

Each token request involves:
- TLS termination (handled by the gateway)
- Client authentication (bcrypt verify on confidential clients, 10-50ms)
- Authorization code lookup (if applicable, 2-5ms)
- Token generation (JWT signing with RSA256 or ECDSA, 1-2ms)
- Token storage (Redis write, 1-3ms)
- Response serialization (less than 1ms)

Total: approximately 15-60ms per request, depending on the client authentication method.

A single modern server (8 vCPU, 16GB RAM) can handle approximately 500-1,000 token requests per second. For 2,000 peak QPS, you need 3-4 instances behind a load balancer.

**Scaling strategy:**

Token issuance is CPU-bound (bcrypt, JWT signing) and I/O-bound (Redis writes). Scale horizontally by adding more auth server instances behind a load balancer. The client registration database is read-heavy (reads on every request) and write-light (only when a client is registered or updated). Use a database connection pool with 20-50 connections per auth server instance.

### Token Storage

Token storage is the total volume of active tokens. Each token entry in Redis consumes approximately 500 bytes (key plus value plus metadata). The total storage requirement depends on the token lifetime and the issuance rate.

**Storage estimation:**

| Token Type | Issuance Per Day | Lifetime | Concurrent Tokens |
|---|---|---|---|
| Access tokens | 55,000,000 | 1 hour | 55,000,000 / 24 = 2,291,666 |
| Refresh tokens | 10,000,000 | 30 days | 10,000,000 x 30 = 300,000,000 |
| Authorization codes | 20,000,000 | 10 minutes | 20,000,000 / 144 = 138,888 |

Total concurrent tokens: approximately 302 million entries.

Storage: 302,000,000 x 500 bytes = 151 GB.

**Redis cluster sizing:**

For 151 GB of token data, use a Redis cluster with 6 nodes (each with 32 GB RAM). Configure 3 master nodes and 3 replica nodes. Each master handles approximately 50 GB of data. The replicas provide read capacity for introspection requests and failover in case of a master failure.

**Memory optimization:**

- Use short token lifetimes (15 minutes for access tokens, 24 hours for refresh tokens) to reduce the concurrent token count.
- Use JWT tokens instead of opaque tokens. JWTs are self-contained and do not need server-side storage for validation. Only the revocation list and refresh tokens need storage.
- Compress token entries in Redis. OAuth2 tokens are typically JSON strings with significant overhead. Compressing with LZ4 or Snappy reduces storage by 60-70%.

### Introspection Throughput

Introspection is the highest-volume operation in a production OAuth2 system. Every API request requires a token validation. The introspection endpoint must handle the full API traffic volume, not just the authentication traffic.

**API traffic estimation:**

Assume 10 million daily active users, each making 100 API requests per day:

Total API requests per day: 10,000,000 x 100 = 1,000,000,000.

Peak introspection QPS: 1,000,000,000 / 86,400 x 3 = approximately 34,722 QPS.

**Cache hit ratio:**

The introspection endpoint caches results in Redis. A token is introspected once per cache TTL (typically 60 seconds). If a user makes 100 API requests per day, and the average API request rate is 1 request per 86 seconds, each token introspected is cached for 60 seconds, so approximately 70% of requests hit the cache.

- Cache hit path: 34,722 x 0.7 = 24,305 QPS (Redis read, less than 1ms)
- Cache miss path: 34,722 x 0.3 = 10,417 QPS (full validation, 5-15ms)

**Sizing the introspection endpoint:**

Each introspection request involves:
- Token parsing (JWT decode, 1ms)
- Signature verification (RSA256 or ECDSA, 1-2ms)
- Expiration check (local, less than 1ms)
- Revocation list check (Redis read, 1-3ms)
- Scope validation (local, less than 1ms)
- Response serialization (less than 1ms)

Cache hit: total approximately 1ms.
Cache miss: total approximately 5-15ms.

A single server (8 vCPU, 16GB RAM) handles approximately 5,000-10,000 introspection requests per second on cache miss and 50,000+ on cache hit. For 35,000 peak QPS with a 70% cache hit rate, you need 4-6 introspection servers.

**Optimization: local introspection cache.**

The API gateway or resource server can maintain a local in-memory cache of introspection results. A token is introspected once per TTL (60 seconds) and the result is cached locally. This eliminates the introspection endpoint round trip for cache hits. The local cache reduces introspection endpoint load by an additional 50-80%.

With local caching:
- Local cache hit: 35,000 x 0.85 = 29,750 QPS (zero network cost)
- Introspection endpoint hit: 35,000 x 0.15 = 5,250 QPS (network round trip)
- Fraction that hits Redis cache: 5,250 x 0.7 = 3,675 QPS (Redis read)
- Fraction that hits full validation: 5,250 x 0.3 = 1,575 QPS (full validation)

The introspection endpoint can be scaled down to 2 instances instead of 6.

### Database Connection Pool Sizing

The client registration database is a shared resource across all auth server instances. Connection pool sizing follows the formula: `connections = (instances x threads_per_instance x (1 + reserve))`. For 6 auth server instances with 20 threads each and a 50% reserve:

connections = 6 x 20 x 1.5 = 180 connections.

The database should be configured to handle 200-300 concurrent connections. PostgreSQL handles this easily with proper `max_connections` configuration. For higher throughput, use a connection pooler like pgBouncer in transaction mode.

### Network Bandwidth Considerations

At scale, network bandwidth becomes a bottleneck for the introspection endpoint. Each introspection request and response is approximately 500 bytes. At 35,000 QPS, the introspection endpoint handles:

- Inbound: 35,000 x 500 bytes = 17.5 MB/s = 140 Mbps
- Outbound: 35,000 x 500 bytes = 17.5 MB/s = 140 Mbps
- Total: 280 Mbps

A single 1 Gbps NIC handles this easily. But if the introspection endpoint also handles token issuance (approximately 2,000 QPS with 2 KB responses), the additional bandwidth is:

- Token issuance: 2,000 x 2 KB = 4 MB/s = 32 Mbps

Total bandwidth: approximately 312 Mbps. A single 1 Gbps NIC is sufficient for the auth server, but you should plan for 10 Gbps NICs at large scale.

## Bridge to Trade-offs

You have seen the architecture of OAuth2 at scale. The production system described here handles millions of users, thousands of clients, and billions of API requests per day. But every architectural decision in this system comes with a tradeoff.

Why use JWT instead of opaque tokens? JWTs are self-contained and eliminate server-side storage for access tokens, but they cannot be revoked without a revocation list, and their payload is exposed to anyone who holds the token.

Why use short-lived tokens with refresh tokens? Short-lived tokens limit the blast radius of a leak, but they increase the load on the token endpoint and require refresh token rotation infrastructure.

Why use a centralized authorization server? Centralized policy enforcement ensures consistency, but it creates a dependency and a single point of failure for all authentication and authorization operations.

Why use PKCE for public clients? PKCE prevents authorization code interception attacks, but it adds complexity to the authorization flow and requires the client to generate and store a code verifier.

Why use multi-region deployment with regional token issuance? Regional issuance provides low latency and high availability, but it introduces consistency challenges and requires a global revocation strategy.

Why use bcrypt for client secrets? Bcrypt hashing protects against database breaches, but it adds 10-50ms of latency to each token request compared to plaintext comparison.

Why use the API gateway pattern? Centralized token validation reduces backend complexity, but it creates a choke point that must be scaled and monitored carefully.

Why use refresh token rotation? Rotation prevents a stolen refresh token from being used indefinitely, but it requires the client to handle the new refresh token on every refresh response, and the server to track the current refresh token version.

Why use JWKS endpoints? The JWKS endpoint allows resource servers to fetch public keys on demand, but it introduces a dependency on the JWKS endpoint being available and the keys being cached correctly.

Why use short authorization code lifetimes? Short lifetimes (10 minutes) limit the window for code interception, but they create a poor user experience if the authorization flow takes longer than expected.

These tradeoffs are not bugs. They are the design decisions you must make based on your system's requirements. The next page examines these tradeoffs in detail and helps you choose the right approach for your specific use case.
