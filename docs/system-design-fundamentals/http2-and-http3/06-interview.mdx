## How This Topic Appears

HTTP/2 and HTTP/3 appear across all stages of system design and infrastructure interviews. Expect questions in four categories:

### Networking and Protocol Questions

1. "Explain the difference between HTTP/1.1, HTTP/2, and HTTP/3. What specific problems does each solve?"
2. "What is head-of-line blocking? How does it manifest differently in HTTP/1.1, HTTP/2, and HTTP/3?"
3. "How does HTTP/2 multiplex work? Walk me through the frame structure and stream lifecycle."
4. "Explain QUIC. Why does it run over UDP instead of TCP? What are the trade-offs?"

### Performance and Optimization Questions

5. "Your web application loads 80 resources per page and takes 4 seconds to load. How would you use HTTP/2 or HTTP/3 to improve this?"
6. "A client reports that your API is fast on WiFi but slow on cellular. How would you diagnose and fix this?"
7. "Your boss asks you to estimate the latency improvement from switching to HTTP/3. Walk through your reasoning."

### Architecture and Design Questions

8. "Design a global CDN that supports both HTTP/2 and HTTP/3. How do you handle protocol negotiation, fallback, and connection migration?"
9. "You are building a real-time chat service that needs to handle 1 million concurrent connections. Would you use HTTP/2, WebSocket, or HTTP/3? Why?"
10. "How would you deploy HTTP/3 in a corporate environment where firewalls may block UDP?"

### Debugging and Troubleshooting Questions

11. "You notice that your HTTP/2 server's memory usage grows linearly with active connections. What is causing this, and how would you reduce it?"
12. "Some users report that their requests hang for 30 seconds before completing. Your HTTP/3 fallback is enabled. What is happening?"

## What the Interviewer Is Actually Testing

| Question Type | What They Test | Red Flags | Green Flags |
|---|---|---|---|
| Protocol comparison | Depth of understanding beyond definitions | Reciting RFC numbers without explaining real impact | Explaining HOL blocking and showing the timeline of a lost packet |
| Multiplexing | Understanding frame-level mechanics | "Multiple requests at once" without explaining streams | Walk through frame headers, stream IDs, flow control windows |
| QUIC design decisions | Trade-off thinking | "QUIC is better because UDP is faster" | "QUIC moves transport to userspace, enabling faster iteration, at the cost of CPU" |
| Performance diagnosis | Systematic problem-solving | Guessing without data | Starting with monitoring metrics, ruling out causes methodically |
| CDN architecture | System design at scale | Missing protocol negotiation, fallback, or monitoring | Describing Alt-Svc, ALPN, DNS HTTPS records, happy eyeballs |
| Real-time service | Protocol choice trade-offs | Picking one protocol without analyzing the workload | Discussing stream types, latency tolerances, head-of-line behavior per protocol |

## Level-by-Level Expectations

### L3 / L4 (Entry-Level / Junior)

**Expectation:** Explain HTTP/1.1, HTTP/2, and HTTP/3 at a high level. Define head-of-line blocking. Mention that HTTP/2 uses multiplexing and HTTP/3 uses QUIC over UDP.

**Sample answer to "What is the difference between HTTP/2 and HTTP/1.1?"**

"HTTP/1.1 processes one request at a time per TCP connection. If a page needs 80 resources, the browser opens 6 parallel connections and queues requests on each. This is called head-of-line blocking: the second request on a connection waits for the first to finish. HTTP/2 fixes this with multiplexing. It sends all 80 requests as streams on a single TCP connection. The server interleaves the response data, so small resources like CSS files can arrive before a large image finishes. HTTP/2 also compresses headers with HPACK, reducing overhead from 500 bytes per request to under 100 bytes."

**Sample answer to "What is head-of-line blocking?"**

"Head-of-line blocking is when one slow operation blocks subsequent operations that are queued behind it. In HTTP/1.1, it happens at the request level: if a connection sends a request for a large image, all subsequent requests on that connection wait for the image to finish. In HTTP/2, request-level HOL blocking is eliminated by multiplexing, but TCP-level HOL blocking remains: a lost TCP packet blocks delivery of data for all streams until retransmitted. HTTP/3 through QUIC eliminates transport-level HOL blocking because each stream recovers independently."

### L5 (Senior)

**Expectation:** Deep technical understanding of frame structure, HPACK, flow control mechanics. Can walk through a packet-level trace. Can design a simple HTTP/2 upgrade strategy.

**Sample answer to "How does HTTP/2 multiplexing actually work at the frame level?"**

"HTTP/2 breaks HTTP messages into binary frames. Each frame has a 9-byte header: length (3 bytes), type (1 byte), flags (1 byte), stream ID (4 bytes, 31-bit). Frames from different streams are interleaved on the wire. For example, a page loading HTML and CSS might send: [Stream 1 HEADERS] [Stream 3 HEADERS] [Stream 1 DATA fragment 1] [Stream 3 DATA fragment 1] [Stream 1 DATA fragment 2] [Stream 3 DATA fragment 2]. The receiver reassembles frames by stream ID. Flow control operates at two levels: each stream has its own window (default 65,535 bytes), and the connection has a combined window. WINDOW_UPDATE frames replenish these windows. If a stream's window is exhausted, the sender pauses that stream but continues sending on others. If the connection window is exhausted, all streams pause."

**Sample answer to "Your API is slow on cellular. How do you diagnose HTTP/2 vs. HTTP/3 issues?"**

"First, I would check the packet loss rate on the cellular connection, because HTTP/2 is sensitive to loss due to TCP HOL blocking. If loss is above 1%, HTTP/3 would likely improve performance. I would measure p50 and p99 request latency segmented by protocol version. If HTTP/3 is not enabled, I would enable it via Alt-Svc and compare. I would also look at the TCP retransmission rate and the QUIC loss rate. A key metric is the stream completion time: if all streams complete around the same time on HTTP/2, it suggests HOL blocking (all streams wait for the slowest packet). On HTTP/3, streams should complete independently. I would also check the QUIC handshake RTT. If the 0-RTT handshake is not working (the server rejects early data), we miss the latency benefit."

### L6 (Staff / Principal)

**Expectation:** Broad architectural thinking across services. Can design global HTTP/3 deployments. Understands failure modes and CAP trade-offs. Can articulate when NOT to use HTTP/2 or HTTP/3.

**Sample answer to "Design a global HTTP/3 deployment for a billion-user service."**

"The architecture has four layers. First, global DNS with SVCB (HTTPS) records advertising HTTP/3 support. Each region announces the same anycast IP. Second, edge nodes in each region terminate QUIC connections. Each edge node handles up to 200,000 concurrent QUIC connections. TLS is terminated at the edge using a centralized certificate management system. Third, a layer of Envoy proxies behind the edge nodes handle HTTP/2-to-backend routing. The edge-to-proxy link uses h2c (cleartext HTTP/2) to avoid double-TLS overhead. Fourth, stateless application servers behind the proxies.

Protocol negotiation uses Alt-Svc on the first HTTP/2 response: the server includes `Alt-Svc: h3=":443"; ma=86400`. The client caches this and attempts HTTP/3 on subsequent requests. If the QUIC handshake fails (UDP blocked), the client falls back to HTTP/2 within 200ms. The Alt-Svc max-age starts at 3600 seconds during rollout and increases to 86400 once stable.

Connection migration is critical for mobile users. The QUIC connection ID allows the server to reassociate the connection when the client's IP changes. But rapid roaming between WiFi access points can trigger too many migrations — the server path-validation timeout is set to 100ms to handle back-to-back migrations.

Key risks: UDP blocking by firewalls requires the fallback mechanism. Enterprise users may see consistent 200ms penalties if their firewall drops UDP. We monitor HTTP/3 adoption by ASN and adjust Alt-Svc max-age per network. Memory at the edge is the primary scaling constraint: 200,000 QUIC connections consume approximately 16 GB of RAM per edge node."

**Sample answer to "When would you choose HTTP/1.1 over HTTP/2 or HTTP/3?"**

"I would choose HTTP/1.1 in three scenarios. First, for internal microservice communication where services make one request at a time. HTTP/1.1 with keep-alive is simpler and the multiplexing benefit of HTTP/2 is irrelevant. Second, for environments with heavy UDP blocking, such as enterprise VPNs or government networks. The HTTP/3 fallback penalty adds complexity without benefit. Third, for extremely resource-constrained devices (IoT sensors, embedded systems). The HPACK table, TLS handshake, and stream state machine of HTTP/2 add memory and CPU overhead that a small microcontroller cannot afford. For these cases, a minimal HTTP/1.1 client with keep-alive is the right choice."

### L7 / L8 (Distinguished / Fellow)

**Expectation:** Industry perspective. Can discuss the evolution of protocols over decades. Understands the political and organizational implications of protocol changes. Can articulate why some ideas succeeded and others failed.

**Sample answer to "Why did server push fail while multiplexing succeeded?"**

"Server push failed for three reasons. First, it solved the wrong problem: the latency bottleneck is not discovering resources (which the browser can do via preload hints) but delivering them efficiently (which multiplexing handles). Second, push was cache-inefficient: the server does not know what the browser has cached, so it pushes duplicates. Chrome telemetry showed that 97% to 99% of pushed resources were wasted. Third, push created complexity at the protocol level that outweighed its benefit. The push promise frame, push ID namespace, and stream state transitions added implementation complexity that was not justified.

Multiplexing succeeded because it solved a real, measured problem. Akamai published a study showing that HTTP/2 reduced page load time by 15% to 50% depending on network conditions. The improvement came from connection reuse and eliminating request-level HOL blocking. Every browser vendor implemented it, every CDN deployed it, and the tooling ecosystem followed.

The lesson: protocol features that solve a measurable pain point survive. Features that seem like a good idea in a standards meeting but do not correspond to a real bottleneck get killed by the market."

## The FAANG Framework for Answering

### Step 1: Clarify the scope

Before answering, confirm what the interviewer is asking. Protocol questions can be narrow (frame structure) or broad (architecture).

"Before I dive in, can I clarify — are you asking about the protocol-level differences, or about how I would architect a service using these protocols?"

### Step 2: Start with the problem, not the solution

State the pain point that HTTP/2 or HTTP/3 solves. This shows you understand why the protocol exists, not just what it does.

"HTTP/1.1 had two fundamental problems for modern web applications. First, each TCP connection could only process one request at a time. Loading 80 resources required either serialization on single connections or multiple parallel connections. Second, HTTP headers were sent as uncompressed text on every request, adding 500 to 800 bytes per request."

### Step 3: Use concrete numbers

Every claim should have a number attached. "Multiplexing reduces latency" is weak. "On a mobile network with 2% packet loss, HTTP/3 reduces p99 latency from 3.8 seconds to 1.4 seconds compared to HTTP/2" shows depth.

### Step 4: Walk the lifecycle

Describe what happens from the moment the client initiates a connection. This demonstrates end-to-end understanding.

"A user types a URL and hits enter. The browser performs a DNS lookup, potentially discovering an HTTPS SVCB record advertising HTTP/3. It then attempts a QUIC handshake with the IP address on UDP port 443..."

### Step 5: Discuss trade-offs

Never present HTTP/3 as strictly better than HTTP/2, or HTTP/2 as strictly better than HTTP/1.1. Always discuss where each excels and where it falls short.

"HTTP/3 outperforms HTTP/2 on lossy networks due to independent stream recovery, but it adds 15-20% CPU overhead at the server for userspace QUIC processing, and it may be blocked by firewalls that drop UDP."

### Step 6: Connect to the business

End with the business impact: how does this affect users, revenue, or operational cost?

"Adopting HTTP/3 for a mobile-first audience would reduce median page load time by 30% on cellular connections, which correlates with a 12% improvement in conversion rate based on industry benchmarks."

## Phrase Bank

**Describing HOL blocking:**
- "HTTP/1.1 HOL blocking is like a single checkout lane where the customer with a full cart blocks everyone behind them."
- "HTTP/2 fixed request-level HOL but introduced transport-level HOL: a single lost TCP packet stalls all streams, like a multi-lane highway where a single accident blocks every lane because there is only one exit ramp."
- "HTTP/3 eliminates transport HOL entirely. Each stream has its own recovery mechanism, like independent ferry boats that cross the river separately."

**Describing QUIC vs. TCP:**
- "QUIC is TCP rebuilt in userspace with TLS built in. It replaces the kernel's TCP stack with an application-level implementation that can be updated without OS upgrades."
- "The 0-RTT handshake means a returning user can send data on the very first packet, saving one full round trip compared to TCP+TLS."
- "Connection migration means a user switching from WiFi to cellular does not lose their connection. The QUIC connection ID survives the IP address change."

**Describing HPACK/QPACK:**
- "HPACK is like a shared notebook between client and server. Once a header has been written down, both sides can just say 'see page 5, line 3' instead of repeating it."
- "QPACK adapts this notebook for out-of-order delivery by using a separate stream for table updates, trading some complexity for QUIC compatibility."

**Describing production deployments:**
- "We use a layered approach: terminate QUIC at the edge, maintain HTTP/2 between edge and proxy, and use gRPC for internal service-to-service communication."
- "Protocol negotiation uses a happy-eyeballs strategy: try QUIC and TCP simultaneously, use whichever succeeds first, cache the decision per origin."
- "Monitoring focuses on three metrics: HTTP/3 adoption rate (fraction of connections using QUIC), QUIC handshake success rate (UDP not blocked), and QUIC loss rate (network quality)."

**Describing failure scenarios:**
- "When a QUIC packet is lost, only the stream that owns the lost data stalls. Other streams continue delivering their responses independently."
- "If the HPACK dynamic table desynchronizes, the connection fails with COMPRESSION_ERROR. This happens when intermediaries modify headers without maintaining correct HPACK state."
- "Memory exhaustion from slow consumers is the most common HTTP/2 failure mode. Per-stream flow control prevents individual streams from consuming too much buffer, but connection-level flow control still allows one slow consumer to impact others."

## Red Flags

**Red Flag 1:** "HTTP/2 is just a faster version of HTTP/1.1."
- Why it is wrong: HTTP/2 is a fundamentally different protocol (binary framing, multiplexed streams, compressed headers), not a performance-tuned version of HTTP/1.1.
- Correct framing: "HTTP/2 replaces HTTP/1.1's text-based, serial protocol with a binary, multiplexed protocol."

**Red Flag 2:** "HTTP/3 is always faster than HTTP/2."
- Why it is wrong: On networks with zero packet loss and low latency, HTTP/2 and HTTP/3 perform identically. HTTP/3's advantage is conditional on network conditions.
- Correct framing: "HTTP/3 shows significant improvement over HTTP/2 on networks with packet loss above 0.5% or high latency."

**Red Flag 3:** "QUIC uses UDP, so it bypasses all TCP congestion control."
- Why it is wrong: QUIC implements its own congestion control (Cubic, BBR, or NewReno) within the userspace QUIC library. It respects congestion signals just like TCP.
- Correct framing: "QUIC implements congestion control in userspace, which allows faster iteration on congestion control algorithms without waiting for OS kernel updates."

**Red Flag 4:** "Server push eliminates round trips, so we should push all resources."
- Why it is wrong: Server push is deprecated by Chrome because push waste far exceeded its benefit. Pushed resources that the client already has cached waste bandwidth.
- Correct framing: "Server push is deprecated. The modern approach is 103 Early Hints, which gives the browser a head start on resource discovery without preemptively sending data."

**Red Flag 5:** "HTTP/2 connections should be kept open indefinitely."
- Why it is wrong: Long-lived connections consume server memory for HPACK tables, flow control windows, and stream state. Connections should have an idle timeout (typically 5 minutes).
- Correct framing: "HTTP/2 connections are long-lived but should have an idle timeout for resource management. The standard recommendation is 5 minutes of inactivity."

**Red Flag 6:** "We do not need HTTP/3 because we use a CDN."
- Why it is wrong: Many CDNs support HTTP/3 and handle the complexity for you. You gain the performance benefit without managing QUIC yourself.
- Correct framing: "Our CDN handles HTTP/3 at the edge. We get the latency benefit for our users without operating QUIC servers ourselves."

**Red Flag 7:** "All streams on an HTTP/2 connection have equal priority."
- Why it is wrong: Without priority signaling, the server treats all streams equally. The browser sends priority information (dependency tree + weights in HTTP/2, urgency values in HTTP/3). Servers that ignore priorities can make pages load slower than HTTP/1.1.
- Correct framing: "The browser signals stream priorities. The server should use this information to allocate bandwidth to critical resources first."

**Red Flag 8:** "We do not need to worry about HTTP/2 flow control because the defaults are fine."
- Why it is wrong: The default per-stream window (65,535 bytes) is too small for most applications. Most production servers raise it to 1 MB or higher to reduce WINDOW_UPDATE overhead.
- Correct framing: "The default HTTP/2 flow control window of 65 KB is too small for high-throughput services. We raise it to 1 MB and monitor connection-level window exhaustion."

## Practice Drills

### Drill 1: The Elevator Pitch (2 minutes)

**Prompt:** Explain HTTP/2 and HTTP/3 to a product manager who knows that "HTTP is how browsers load web pages."

**Time limit:** 2 minutes.

**Target:** Deliver a clear, non-technical explanation with the restaurant analogy and concrete impact numbers.

**Self-check:** Did you mention that HTTP/3 helps mobile users? Did you give a specific latency number?

### Drill 2: The Architecture Whiteboard (5 minutes)

**Prompt:** Whiteboard an HTTP/3 deployment architecture for an e-commerce site with 10 million monthly active users, 60% on mobile.

**Time limit:** 5 minutes.

**Target:** Draw the four layers (DNS with SVCB, edge QUIC termination, load balancers, application servers). Annotate protocol negotiation (Alt-Svc, ALPN). Label key numbers (200,000 connections per edge node, 200ms fallback timeout). Mark the monitoring points.

### Drill 3: The Debugging Scenario (5 minutes)

**Prompt:** Your team deployed HTTP/3 on a test server and measured 20% slower page loads on the office WiFi. Users in the office cannot use QUIC. What is happening?

**Time limit:** 5 minutes.

**Target answer:** The office firewall blocks UDP. The client tries QUIC, waits 200ms for a timeout, then falls back to HTTP/2. The 200ms penalty makes HTTP/3 slower than just using HTTP/2 directly. The fix: measure the QUIC failure rate from office IPs, confirm UDP blocking, and the 200ms wait is the fallback timeout. For offices that block UDP, skip the QUIC attempt and use HTTP/2 directly.

### Drill 4: The Design Decision (10 minutes)

**Prompt:** You are designing a real-time gaming platform that handles 500,000 concurrent players. Each player sends 20 state updates per second (small packets, under 100 bytes each). The platform needs to deliver state updates to other players in the same game session (broadcast to 10-50 players). Compare WebSocket over TCP, HTTP/2, and HTTP/3 for this use case.

**Time limit:** 10 minutes.

**Target answer:** WebSocket over TCP has head-of-line blocking: if one state update packet is lost, all subsequent updates wait for retransmission. HTTP/2 is worse because it multiplexes game updates with other streams (lobby chat, matchmaking, etc.), and TCP HOL blocks everything. HTTP/3 over QUIC is the best fit: each player's state update can be a separate QUIC stream or a datagram (unreliable, no retransmission delay). QUIC's independent stream recovery ensures one player's lost packet does not stall others. Use QUIC datagrams for state updates (loss-tolerant) and streams for reliable events (chat, inventory changes).

### Drill 5: The Capacity Plan (10 minutes)

**Prompt:** Estimate the server resources needed to terminate 1 million concurrent HTTP/3 connections for a messaging app that sends 1 KB messages at 10 messages per second per user. Compare to HTTP/2.

**Time limit:** 10 minutes.

**Target answer:**

| Aspect | HTTP/2 | HTTP/3 |
|---|---|---|
| Connection memory | 40 KB per conn × 1M = 40 GB | 80 KB per conn × 1M = 80 GB |
| TLS overhead | Per TCP handshake | Built into QUIC, less overhead |
| Bandwidth | 1M × 10 msg/s × 1 KB = 10 GB/s | Same |
| CPU (handshake) | 5000/s per core | 3000/s per core (UDP overhead) |
| TCP/UDP overhead | TCP segmentation offloaded to NIC | UDP GRO needed, less offloading |
| Node count (256 GB servers) | 256 GB / 40 GB = 7 nodes | 256 GB / 80 GB = 12 nodes |
| p99 latency (1% loss) | 50ms + retransmit spikes | Under 15ms |

HTTP/3 needs 60% more servers for CPU, but provides 3x better tail latency on lossy networks. Trade-off: 5 extra servers for dramatically better user experience on mobile.
