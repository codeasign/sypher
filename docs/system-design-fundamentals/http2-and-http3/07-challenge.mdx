## Tier 1 — Foundation

### Challenge 1.1: Explain Multiplexing (5 minutes)

**Prompt:** Describe HTTP/2 multiplexing to a junior engineer. Include what problem it solves, how streams and frames relate, and one concrete latency advantage over HTTP/1.1. Do not mention QUIC or HTTP/3.

**Evaluation criteria:**
- Clearly states that HTTP/1.1 processes one request per connection at a time (head-of-line blocking).
- Explains that multiplexing allows multiple streams on one TCP connection.
- Describes frames as the smallest unit, tagged with a stream ID for reassembly.
- Mentions that the server can interleave response fragments from different streams.
- Gives a concrete number: a page with 80 resources takes 4 seconds with HTTP/1.1 and drops to 2 seconds with HTTP/2.

**Model answer:** "HTTP/1.1 can only process one request at a time per TCP connection because of head-of-line blocking. If you have 80 resources, each connection serializes its requests — request 2 waits for response 1, request 3 waits for response 2, and so on. Browsers open 6 parallel connections, but 6 queues of serial requests still create substantial waiting. HTTP/2 solves this with multiplexing. It breaks each HTTP message into binary frames — HEADERS frames for headers, DATA frames for body. Each frame carries a stream ID. The client creates 80 streams, each tagged with a unique ID. The server interleaves DATA frames from different streams on the same TCP connection. Instead of waiting for an entire image to download before sending CSS, the server sends a fragment of the CSS between fragments of the image. The client reassembles by stream ID. This means the CSS can arrive in 500ms instead of waiting 3 seconds for the image to finish. The concrete impact: an HTTP/1.1 page loading 80 resources over 6 connections takes roughly 4 seconds. The same page over a single HTTP/2 connection completes in under 2 seconds."

### Challenge 1.2: Trace a Lost Packet (5 minutes)

**Prompt:** Draw a timeline showing what happens when a TCP packet carrying HTTP/2 stream data is lost on a connection with 5 active streams. Then draw the same scenario with QUIC (HTTP/3). Label the time delta.

**Evaluation criteria:**
- HTTP/2 timeline shows that the lost packet stalls all 5 streams.
- Marks the retransmission RTT (e.g., 80ms).
- Shows that no stream receives new data during retransmission.
- HTTP/3 timeline shows only the affected stream stalls.
- Labels the other 4 streams continuing to receive data.
- Correctly identifies the time saved: 80ms (one RTT) for 4 out of 5 streams.

**Model answer:** "In HTTP/2 over TCP, the connection has 5 active streams: S1, S3, S5, S7, S9. The sender sends TCP packets: P1 (S1 data), P2 (S3 data), P3 (S5 data), P4 (S7 data), P5 (S9 data). P3 is lost. The receiver sends duplicate ACKs for P2. The sender detects the loss after 3 duplicate ACKs and retransmits P3. During this time, P4 and P5 are buffered at the receiver but cannot be delivered to the HTTP/2 layer because TCP delivers bytes in order. All 5 streams stall for 80ms (one RTT). In HTTP/3 over QUIC, the connection has the same 5 streams. Packet 3 (QUIC frame for S5) is lost. The sender detects the loss via a packet number gap: it received ACKs for packets 1, 2, and 4, but not 3. The sender retransmits the S5 frame in a new packet (P6). Meanwhile, packets 4 and 5 are delivered immediately because QUIC reassembles streams independently. S7 and S9 continue without delay. Only S5 stalls for 80ms. The total time saved: 80ms × 4 streams = 320ms of cumulative wall time."

### Challenge 1.3: Compare Handshake Costs (5 minutes)

**Prompt:** Calculate the round-trip cost of establishing a connection for HTTP/1.1 with TLS, HTTP/2 with TLS, and HTTP/3 with QUIC (first visit, and returning visit). Assume 50ms RTT to the server.

**Evaluation criteria:**
- HTTP/1.1 TLS: 3 RTT = 150ms (TCP SYN/SYN-ACK/ACK = 1 RTT, TLS 1.3 = 1 RTT, first request = 1 RTT).
- HTTP/2 TLS: 3 RTT = 150ms (same as HTTP/1.1 — HTTP/2 does not change handshake).
- HTTP/3 first visit: 1 RTT = 50ms (QUIC handshake with TLS 1.3 integrated).
- HTTP/3 returning visit: 0 RTT = 0ms (0-RTT data sent in Initial packet).
- Correctly notes that HTTP/2 has same handshake as HTTP/1.1 because it requires TLS.

**Model answer:** "HTTP/1.1 with TLS requires 3 round trips for a new connection: 1 RTT for the TCP three-way handshake, 1 RTT for the TLS 1.3 handshake, and 1 RTT for the first HTTP request. At 50ms RTT, that is 150ms before any response data arrives. HTTP/2 with TLS has the exact same cost: 3 RTT (150ms) because HTTP/2 runs over TCP with TLS. HTTP/3 uses QUIC, which integrates the TLS handshake into the QUIC transport handshake. On the first visit, QUIC completes in 1 RTT: 50ms. On a returning visit, QUIC can send application data in the very first packet (0-RTT), reducing the cost to 0 RTT: the first response arrives within 50ms along with the handshake completion. This means HTTP/3 saves 100ms on first visits and 150ms on return visits compared to HTTP/2."

## Tier 2 — Design

### Challenge 2.1: Design an HTTP/2 Migration for an E-Commerce Site (15 minutes)

**Context:** A mid-size e-commerce site serves 500,000 daily active users. Currently using HTTP/1.1 with 6 parallel connections per client. Average page: 60 resources (HTML, CSS, JS, images, fonts). Median page load time: 4.2 seconds on desktop, 7.8 seconds on mobile. The CTO wants to cut load time by 50% and asked you to evaluate HTTP/2.

**Rubric:**

| Aspect | Poor (0 pts) | Fair (1 pt) | Good (2 pts) | Excellent (3 pts) |
|---|---|---|---|---|
| Problem analysis | Blames network latency | Blames request queuing | Identifies both connection overhead and request HOL blocking | Quantifies: 60 resources ÷ 6 connections = 10 requests per connection chain, each serialized |
| Solution design | "Enable HTTP/2" without specifics | Lists HTTP/2 features | Specifics: upgrade CDN and server libraries, enable HPACK, test with h2load | Includes fallback (ALPN negotiation), monitoring (active streams, flow control), and testing plan |
| Performance estimate | No numbers | "Faster" | Estimates: 40-60% reduction on desktop, 30-40% on mobile | Backed by published data: Akamai's HTTP/2 study showing 15-50% LCP reduction |
| Risk assessment | None | Mentions older equipment | Lists proxy transparency, HPACK state, and server push pitfalls | Prioritizes risks: (1) middlebox HPACK breakage → test with 3 proxies, (2) server CPU for HPACK, (3) monitoring gaps |

**Model answer:**

"First, let me quantify the problem. The site loads 60 resources across 6 parallel HTTP/1.1 connections. Each connection handles 10 resources serially. With a 50ms RTT and 40 KB average response, the total serial wait is (10 requests × 50ms RTT) + (10 responses × 40 KB / connection bandwidth). On a 10 Mbps connection, that is 500ms + 320ms per connection chain = 820ms per connection. With 6 parallel connections, the longest chain determines page load time. If resources are unevenly distributed, one connection may handle 12 requests while another handles 8. The page load time is roughly 1 second just for serial request overhead.

HTTP/2 eliminates this by multiplexing all 60 requests on a single connection. The 60 request-response pairs are interleaved. Instead of 10 serial requests on one connection, all 60 are in flight simultaneously. On a 50ms RTT, the total latency for a multiplexed load is roughly 50ms (negotiation) + 60 × 40 KB / 10 Mbps = 50ms + 1.92s ≈ 2 seconds. That is a 50% reduction from the HTTP/1.1 baseline.

For the migration: I would enable HTTP/2 on our CDN first (Cloudflare, Fastly, or Akamai all support it). The CDN terminates the client's HTTP/2 connection and forwards to our origin over HTTP/1.1 or HTTP/2. This gives us the client-side benefit without modifying our backend servers. Next, I would update our application server (Nginx or Envoy) to handle HTTP/2 if we want end-to-end. I would test with `h2load -n 10000 -c 100 -m 10 https://stage.example.com/` to verify throughput and monitor for any HPACK-related errors.

Key risks: (1) enterprise proxies that strip HTTP/2 upgrade headers — ALPN negotiation handles this transparently, falling back to HTTP/1.1. (2) HPACK state desynchronization if a proxy modifies headers — we test with our corporate proxy before rollout. (3) Server CPU for HPACK encoding — we estimate a 10-15% CPU increase and provision additional capacity. Expected result: median load time drops from 4.2s to under 2.5s on desktop, from 7.8s to under 4.5s on mobile."

### Challenge 2.2: Design a Low-Latency Mobile API with HTTP/3 (15 minutes)

**Context:** A ride-sharing company operates in 50 cities worldwide. The mobile app sends a ride request (POST /api/ride) with current location, destination, and preferences. The API must respond within 200ms (p99) to show nearby drivers. Currently using HTTP/1.1 with persistent connections. 60% of users are on 4G/5G cellular. Packet loss on cellular averages 1.5%. Current p99 is 850ms.

**Rubric:**

| Aspect | Poor (0 pts) | Fair (1 pt) | Good (2 pts) | Excellent (3 pts) |
|---|---|---|---|---|
| Problem diagnosis | Blames server speed | Notes network loss | Quantifies TCP HOL blocking contribution to p99 | "At 1.5% loss on 60 resources, there is a 60% chance of at least one packet loss per page view → each loss adds 80ms (cellular RTT) to all in-flight requests" |
| Protocol choice | HTTP/3 is faster | Picks HTTP/3 | Explains QUIC's independent stream recovery eliminates the 80ms penalty for non-affected streams | Discusses 0-RTT for returning users (saves 50-80ms per reconnection), connection migration for WiFi-to-cellular handoff |
| Architecture | Single server | CDN + backend | Edge-terminated QUIC with anycast routing | Server selection based on city proximity, QUIC connection migration for drivers moving between cells, 103 Early Hints for manifest discovery |
| Real-world trade-offs | None | Mentions UDP blocking | Estimates 5-10% of users on UDP-blocking networks → fallback strategy | Cost-benefit: 15% more edge CPU vs. 60% p99 reduction; monitoring HTTP/3 adoption by city; city-by-city rollout |

**Model answer:**

"The current p99 of 850ms is dominated by two factors: (1) TCP retransmission on cellular with 1.5% packet loss, and (2) the connection establishment cost for users who reconnect frequently (mobile apps are frequently background-killed).

For the ride request flow: a user opens the app, the app sends the ride request, which triggers several dependent API calls (driver search, pricing, ETA). Each call is a separate HTTP request. With HTTP/1.1 over TCP, a single lost packet on any response stalls everything. At 1.5% loss, approximately 1 in 67 packets is lost. With 60 resources, the probability of at least one loss is roughly 60%. Each loss adds one cellular RTT (80ms) to all in-flight requests.

Switching to HTTP/3 with QUIC gives us three improvements. First, independent stream recovery: a lost QUIC packet only affects its own stream. Other APIs complete without the 80ms penalty. Second, 0-RTT handshake: if the app was background-killed and reopened, the next request sends data immediately instead of waiting for a TCP+TLS handshake. Third, connection migration: drivers moving between cell towers do not lose their connection.

The architecture: deploy an HTTP/3-capable CDN edge in each city (or region). The edge terminates QUIC from the mobile app. Internal links use HTTP/2 or gRPC. We monitor HTTP/3 adoption by city and fall back to HTTP/2 for cities where UDP is blocked. We estimate p99 drops from 850ms to under 250ms based on Cloudflare's published data showing 30-40% latency reduction for mobile HTTP/3 traffic. The trade-off: 15-20% more CPU at the edge for QUIC processing."

### Challenge 2.3: Diagnose the Slow Proxy (15 minutes)

**Context:** Your team deployed an HTTP/2 proxy between clients and backend servers. After deployment, you notice that p50 response times increased by 40% but p99 response times increased by 200%. CPU and memory on the proxy are both under 50%. Network bandwidth is under 30%. What is happening?

**Rubric:**

| Category | Diagnosis | Evidence |
|---|---|---|
| Flow control | Connection-level flow control window exhausted by a few slow clients | Check per-connection flow control window utilization: look for connections where window is near zero while others have available window |
| HPACK CPU | HPACK compression overhead on the proxy | Check HPACK dynamic table hit rate. If below 60%, the proxy is re-encoding headers inefficiently |
| Stream prioritization | Server ignores priority, round-robins across all streams | Check stream completion order: if all streams complete around the same time, prioritization is wrong |
| Connection coalescing | Proxy coalescing connections incorrectly, routing all traffic through one backend | Check per-backend connection counts: one backend handles 80% of traffic? |
| TLS re-encryption | Proxy re-terminates TLS, adding double encryption overhead | Check TLS handshake count: proxy doing 2x the expected handshake rate |

**Model answer:** "The most likely cause is connection-level flow control exhaustion. The proxy maintains a single HTTP/2 connection per client. Each client connection has a flow control window (default 1 MB on the connection, 65 KB per stream). If a few clients are slow consumers (small TCP receive windows on their end, or high latency), the proxy's buffer for those clients fills up. The connection-level window applies across all streams, so when one slow client fills the window, all other streams on that connection stall waiting for WINDOW_UPDATE frames. The symptom is exactly what you see: p50 goes up moderately (many connections have no slow clients), but p99 spikes (the 1% of connections with slow clients suffer cascading stalls).

To confirm, I would check the monitoring metrics: `http2.flow_control_window_exhausted` count and per-connection buffered data. If any connection has over 10 MB of buffered data while others have under 1 MB, this confirms the diagnosis.

The fix: reduce `SETTINGS_INITIAL_WINDOW_SIZE` from 1 MB to 256 KB per stream. This limits the buffering per slow client. Implement per-stream backpressure in the application layer: do not generate response data until the stream window has available capacity. If the proxy is between client and backend, also increase the number of backend connections per proxy (from 10 to 50) to reduce the impact of any single client on backend bandwidth."

## Tier 3 — Expert

### Challenge 3.1: Design a Global HTTP/3 Infrastructure for Real-Time Collaboration (30 minutes)

**Context:** A company is building a real-time collaborative document editor (like Google Docs) that will serve 100 million active users worldwide. Users in the same document must see each other's edits within 100ms (p99). The platform must support:
- 10 million concurrent users across 1 million concurrent documents
- Each document has 2-50 collaborators
- Edits are small (under 1 KB per operation)
- Operations per second per document: 10 (active editing) to 0 (idle)

**Design requirements:**
- Protocol choice for client-to-server communication
- Multi-region architecture with conflict resolution
- UDP and firewall considerations
- Fallback strategy for blocked networks
- Capacity planning (servers, bandwidth, memory)

**Evaluation criteria:**
- Protocol choice: Justify HTTP/3 with QUIC datagrams for operational transformations (OT). Discuss why WebSocket over TCP introduces HOL blocking for edit operations. Consider WebTransport as an alternative.
- Architecture: Edge fan-out with operational transform (OT) servers per region. Each region has an OT server that holds the document state for documents whose active editors are in that region.
- Connection migration: Users switch networks while editing — QUIC connection migration allows seamless continuation without reconnection.
- Fallback: For UDP-blocked networks, fall back to WebSocket over HTTP/2. This adds 50-80ms latency but maintains functionality.
- Conflict resolution: Server-side operational transform or CRDT. Each operation is a QUIC datagram (unreliable, no retransmission delay). The server acknowledges operations reliably via a QUIC stream.

**No single model answer:** This is an open-ended design challenge. A strong answer addresses all five evaluation dimensions, provides concrete numbers (server count, bandwidth per server, memory per connection), discusses failure modes (region partition, server crash, UDP blocking), and acknowledges trade-offs (WebTransport vs. raw QUIC datagrams, OT vs. CRDT for conflict resolution).

### Challenge 3.2: Degraded Network Performance on a QUIC-Only Deployment (30 minutes)

**Context:** Your company deployed HTTP/3 as the primary protocol for all API traffic, removing HTTP/2 support entirely. Three months after deployment, the operations team notices:

1. Mobile users on two specific cellular carriers in Southeast Asia consistently experience p99 latency of 3 seconds (versus 400ms on other carriers).
2. Enterprise users at 50 large companies report that their API calls fail entirely after 30 seconds.
3. Server CPU at the edge has increased by 35% compared to the HTTP/2 baseline, but traffic volume is unchanged.
4. The customer support team has received no reports from home broadband users.

**Diagnose each symptom, propose a fix, and design a monitoring system to catch these issues before they affect users.**

**Evaluation criteria:**

**Symptom 1 (Southeast Asia carriers):**
- Diagnosis: These carriers likely throttle UDP traffic. Some cellular carriers rate-limit UDP to under 1 Mbps while allowing TCP at full bandwidth. QUIC packets are dropped or delayed after the first few KB. The QUIC connection degrades because congestion control reacts to the hidden rate limit.
- Fix: Detect the carrier (ASN) and fall back to HTTP/2 for affected carriers. Use Alt-Svc with a short max-age (60 seconds) for these users. Implement a per-carrier UDP throughput test on first connection.
- Verification: Deploy a version flag to force HTTP/2 for the specific ASN and measure the p99 change.

**Symptom 2 (Enterprise users):**
- Diagnosis: Enterprise firewalls block UDP entirely. The QUIC Initial packet is dropped silently. The client has a connection timeout set to 30 seconds (standard OS UDP timeout). After 30 seconds, the client falls back to... nothing, because HTTP/2 was removed. The connection attempt fails entirely.
- Fix: Restore HTTP/2 as a mandatory fallback. Maintain both stacks. For enterprise customers, advertise via Alt-Svc with `ma=60` so the client re-checks frequently but does not permanently stick to QUIC.
- Severity: This is a critical bug. Removing HTTP/2 was a mistake. HTTP/3 must always be deployed alongside HTTP/2 as a fallback, never as a replacement.

**Symptom 3 (35% CPU increase):**
- Diagnosis: QUIC's userspace processing is inherently more CPU-intensive than kernel TCP. TLS 1.3 per-packet encryption, stream reassembly, and loss detection all run in userspace. 35% is within the expected range (industry data shows 20-40% CPU overhead for QUIC vs. TCP).
- Fix: This may not need a fix if the CPU is still within capacity. Mitigations: (a) enable UDP GRO (Generic Receive Offload) to coalesce packets before delivery, (b) use a QUIC implementation with kernel bypass (DPDK), (c) add more edge nodes. The correct response is to right-size the edge cluster.
- Verification: Check if CPU correlates with QUIC handshake rate or data throughput. QUIC handshakes are especially CPU-heavy (asymmetric crypto). If handshake rate is high, enable 0-RTT more aggressively to reduce full handshakes.

**Symptom 4 (No reports from home broadband):**
- Diagnosis: Home broadband typically allows UDP. These users are likely the ones successfully using HTTP/3. No news is good news here.
- Caveat: Verify that the monitoring confirms HTTP/3 adoption on home broadband networks. If the adoption rate is under 30%, users may be falling back to HTTP/2 (which was removed — contradiction). If adoption rate is over 70%, home broadband users are benefiting from HTTP/3 as expected.

**Monitoring system design:**

```yaml
# Proposed monitoring additions
metrics:
  quic_adoption_rate:
    description: "Fraction of connections using QUIC vs. HTTP/2"
    grouped_by: [asn, country, carrier, isp_type]
    alert: > "Adoption under 20% on any major ASN (>100K requests/day)"

  quic_handshake_rt:
    description: "QUIC handshake round-trip time"
    alert: > "p99 > 500ms for any ASN (indicates UDP throttling)"

  quic_loss_rate_by_asn:
    description: "QUIC packet loss rate segmented by ASN"
    alert: > "Loss rate > 5% on any carrier"

  udp_blocked_rate:
    description: "Fraction of connections where QUIC Initial gets no response"
    alert: > "UDP blocked rate > 10% for enterprise ISPs"

  http2_fallback_rate:
    description: "How often clients fall back to HTTP/2"
    alert: > "Fallback rate > 50% indicates widespread UDP issues"
```

The challenge key insight: HTTP/3 must never be deployed without HTTP/2 as a fallback. UDP blocking, throttling, and middlebox interference are real and pervasive. A production-grade deployment monitors protocol adoption by network and dynamically adjusts per-network. Removing HTTP/2 to simplify operations creates exactly the failure mode described in this scenario.
