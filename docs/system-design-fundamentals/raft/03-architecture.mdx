## From Naive Implementation to Production System

A textbook Raft implementation works correctly on a single machine with simulated network delays. The production architecture requires significant engineering beyond the core algorithm.

### The Naive Approach

A minimal Raft implementation places every component on a single process per node: the state machine (key-value store), the Raft consensus module, the log storage, the RPC handlers, and the application API server. This approach works for small clusters (3-5 nodes) with low throughput (under 1,000 ops/sec). It fails at scale because:

- The API server competes with Raft RPCs for CPU and network bandwidth
- A single fsync thread blocks both log writes and snapshot operations
- The leader's network stack becomes a bottleneck under high throughput
- No isolation between the Raft-critical path and application-level processing

### Production Architecture (etcd / ZooKeeper Style)

Production Raft systems like etcd use a layered architecture that separates concerns and enables scaling.

**Transport layer:** Production Raft systems use gRPC streaming for Raft RPCs. This enables pipelining and reduces connection overhead. TLS termination is handled at this layer. Request multiplexing combines Raft RPCs and client API calls over the same connection.

**Consensus module:** Implements leader election, log replication, and safety checks. The log replicator uses a batcher that groups entries into configurable-sized batches. The voter logic handles RequestVote and pre-vote requests.

**MVCC backend:** Instead of a simple state machine, production systems use multi-version concurrency control (MVCC). Each read sees a consistent snapshot of the state machine. This enables reads to proceed without blocking writes.

**Client API layer:** Handles client connections, authentication, and request routing. The API layer forwards writes to the Raft leader and can serve reads locally using read-index optimization.

## Multi-Raft Sharding

A single Raft group cannot scale beyond approximately 10,000-30,000 writes per second because all writes go through a single leader node. Multi-Raft sharding solves this by partitioning the key space into multiple ranges, each managed by its own Raft group.

CockroachDB popularized this architecture. The key space is divided into contiguous ranges (default 64 MB each). Each range operates as an independent Raft group with its own leader and follower replicas. Ranges are automatically rebalanced across nodes to distribute load evenly.

### Raft Store per Node

In a multi-Raft system, each node hosts a Raft store that manages all Raft groups on that node. The store provides:

- A shared log storage pool (write-ahead log file that interleaves entries for all groups)
- A shared snapshot scheduler
- Per-group state tracking (commit index, last applied, current term)
- A shared transport for Raft RPCs

The store uses a **shared-nothing** approach for CPU: each Raft group runs on its own goroutine or thread, but the storage layer serializes writes to batch fsyncs across groups. This reduces the per-group fsync cost significantly.

## Integration with Databases and Caches

Raft is most commonly embedded in databases and coordination services.

### Embedded Raft in Databases

**etcd:** A key-value store that uses Raft for both consensus and replication. It serves as Kubernetes' primary datastore. etcd's Raft implementation logs every operation through the consensus layer. Writes achieve linearizability, and reads can be served with linearizable semantics using the read-index optimization.

**TiDB:** A distributed SQL database that uses Raft for consensus within each region (a shard of data). TiDB's Raft implementation includes a significant optimization: the Raft log is the same as the database's write-ahead log, eliminating redundant writes. This reduces write amplification from 2x to nearly 1x.

**CockroachDB:** Uses multi-Raft with per-range Raft groups. The Raft log is stored in a separate RocksDB instance per node. CockroachDB implements a "parallel commits" optimization that reduces write latency from two Raft rounds to one for non-conflicting transactions.

### Integration with Caches

Raft is used in distributed caching systems to maintain a consistent configuration store. For example, Redis Sentinel uses a Raft-like protocol for leader election, and some Redis Cluster deployments embed Raft for configuration consensus.

The key insight is that Raft is not used for cache data replication (which uses more efficient async replication) but for metadata consensus: who is the primary, what nodes are in the cluster, what is the shard layout.

## Multi-Region Raft

Operating Raft across multiple geographic regions introduces significant latency and failure mode challenges.

**Latency challenge:** A write in a US-East to US-West deployment requires one RTT for replication. At 60-80ms round trip, a single write takes 60-80ms minimum. This is acceptable for configuration data but not for high-throughput transactional workloads.

**Failure isolation:** A region-wide outage must not bring down the entire cluster. The typical deployment is 5 nodes spread across 3 regions: 2 in region A, 2 in region B, 1 in region C. Any single region can fail and the cluster continues (quorum remains in the other regions).

**Witness nodes:** To reduce cross-region replication cost, witness nodes (non-voting) can be placed in regions that need read locality but do not need to participate in quorum. These nodes receive the replicated log asynchronously.

## Capacity Planning

Estimating resources for a Raft-based system requires modeling the write path carefully.

**Write cost model:** Each write requires one fsync on the leader and one fsync on a majority of followers. For a 3-node cluster with 10,000 writes/sec:
- Each node performs approximately 6,700 fsyncs/sec (leader: 10,000, each follower contributes to majority roughly 3,350)
- Total write bandwidth: 10,000 entries * average entry size (e.g., 200 bytes) = 2 MB/sec per node

**Memory per node:** Each node maintains the full log in memory until compaction. For 100,000 entries at 200 bytes each: 20 MB for entries plus metadata overhead (term, index checksums) adds approximately 10-20% more.

**Snapshot cost:** Creating a snapshot of a 10 GB state machine takes 1-30 seconds depending on implementation (copy-on-write snapshots are faster but use more memory). During snapshot, the write throughput is typically reduced by 20-50%.

**Network bandwidth:** Each write generates N-1 AppendEntries RPCs (one per follower). With 3 nodes and 10,000 writes/sec: 20,000 outbound RPCs/sec from the leader, each containing the batch of entries plus metadata.

## Operations

Operating production Raft systems requires specific monitoring and maintenance practices.

**Monitoring:** Critical metrics include:
- Election rate (should be near zero after initial stabilization)
- Term changes (unexpected term increments indicate leader instability)
- Log lag per follower (divergence indicates a slow or failing follower)
- Snapshot duration (long snapshots block log compaction)
- Commit latency (p50 and p99)
- RPC failure rate (network issues)

**Member changes:** Adding or removing nodes requires joint consensus or a configuration change entry. The procedure is:
1. Propose the new configuration to the Raft cluster as a log entry
2. Once committed, the new configuration takes effect
3. Data transfer (for replaced nodes) proceeds through normal log replication

**Disaster recovery:** If a majority of nodes is lost permanently (e.g., entire data center destroyed), the cluster is unavailable. Recovery requires:
1. Identify the surviving node with the most up-to-date log
2. Force that node to become leader (skip election safety checks)
3. Add replacement nodes through the membership change mechanism
4. Accept the loss of any entries committed on the lost majority nodes
