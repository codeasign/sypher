## Company Case Studies

### Amazon DynamoDB — AP at Planet Scale

Amazon DynamoDB is the flagship AP system at Amazon. It was built after the 2004 holiday season, when Amazon's Dynamo paper documented the need for a highly available key-value store. The core insight: Amazon's shopping cart must never lose a write. Even if the network is partitioning, a customer's "add to cart" must succeed.

**Scale.** DynamoDB serves millions of requests per second across 30+ AWS regions. Each table can scale to petabytes of data. The system handles 10+ trillion requests per day.

**Problem.** Amazon's relational databases could not handle the peak load during the holiday shopping season. The databases would fail over, causing downtime. Every second of downtime cost millions of dollars in lost revenue. Amazon needed a database that never went down, even during network partitions.

**Solution.** DynamoDB is AP by design. It uses a shared-nothing architecture with automatic partitioning. Each partition is replicated across 3 Availability Zones. Writes are acknowledged as soon as one replica confirms (W=1 by default). Reads are eventually consistent by default, with strongly consistent reads available as an option.

During a partition, DynamoDB continues accepting writes on both sides. Hinted handoff stores writes for unreachable replicas. Conflict resolution uses last-write-wins by timestamp. Data converges when the partition heals.

**Result.** DynamoDB provides 99.999 percent availability for most workloads. The 2004 holiday season problem was solved. Amazon's shopping cart never loses writes, even during partitions. The cost: occasional stale reads and the rare silent data loss from last-write-wins conflict resolution.

### Google Spanner — CP with TrueTime

Google Spanner is Google's globally distributed relational database. It provides ACID transactions and external consistency across datacenters worldwide. It achieves this by using TrueTime, Google's synchronized clock infrastructure.

**Scale.** Spanner runs across hundreds of datacenters and millions of machines. It manages exabytes of data. Google's largest applications — Google Ads, Google Play, Google Search indexing — run on Spanner.

**Problem.** Google needed a database that provided strong consistency across global datacenters. Existing NoSQL systems (like Bigtable) were AP and could not provide the consistency guarantees required for financial applications. Traditional relational databases could not scale to Google's size.

**Solution.** Spanner is CP with a twist. It uses TrueTime, a distributed clock system based on GPS and atomic clocks, to provide bounded clock uncertainty. TrueTime exposes a time interval: TT.now() returns [earliest, latest], where the true time is guaranteed to be within that interval. The uncertainty window is typically 1-7 milliseconds.

During a partition, Spanner uses the TrueTime interval to ensure consistency without blocking reads. A read operation can be satisfied from any replica as long as the replica's data timestamp is within the uncertainty window. If the window has passed, the replica is guaranteed to be consistent.

**Result.** Spanner provides the strongest consistency model of any globally distributed database. It is CP in the CAP sense: during a partition, it prioritizes consistency over availability. The cost is higher latency (50-100ms for cross-region writes) and the complexity of TrueTime infrastructure.

### Apache Cassandra — AP with Tunable Consistency

Apache Cassandra is the most widely used AP system in the open source world. It was originally developed at Facebook for inbox search and later open sourced. Cassandra combines Dynamo's AP design with Bigtable's data model.

**Scale.** Cassandra powers Netflix, Apple, Uber, and thousands of other companies. Netflix runs over 2,500 Cassandra nodes across multiple regions. Apple manages over 100,000 Cassandra nodes globally.

**Problem.** Facebook needed a database that could handle billions of messages per day across global datacenters. The system had to be always available for writes, because users expect their messages to be delivered instantly. Stale data was acceptable because the inbox search could tolerate eventual consistency.

**Solution.** Cassandra is AP by default. It uses a Dynamo-style ring architecture with consistent hashing. Each node is responsible for a range of the hash ring. Data is replicated across N nodes (typically 3). Writes are acknowledged by the coordinator after W replicas respond.

Cassandra's tunable consistency is its key innovation. You can set W=R=QUORUM for strong consistency, or W=1, R=1 for maximum availability. You can change these settings per operation, allowing you to mix CP and AP behaviors in the same application.

During a partition, Cassandra's default behavior is AP. Both sides accept writes. Hinted handoff queues missed writes. Read repair fixes inconsistencies during reads. Anti-entropy (via Merkle trees) fixes inconsistencies in the background.

**Result.** Cassandra provides massive write throughput with linear scalability. Netflix achieves 1+ million writes per second on a single cluster. The cost: applications must handle eventual consistency, and concurrent writes may conflict.

### LinkedIn's Espresso — CP with Quorum

LinkedIn's Espresso is a distributed document database that provides strong consistency. It is used for LinkedIn's core data services, including member profiles, connections, and messaging.

**Scale.** Espresso runs across multiple LinkedIn datacenters, serving millions of requests per second. It stores billions of documents across thousands of partitions.

**Problem.** LinkedIn needed a database that provided strong consistency for member profiles. If a member updated their name, every subsequent read must return the updated name. Eventual consistency was not acceptable for LinkedIn's core member experience.

**Solution.** Espresso is CP. It uses a primary-secondary replication model with quorum-based writes. Each partition has a primary node and multiple secondary nodes. Writes go to the primary, which replicates to a quorum of secondaries before acknowledging. Reads can go to any secondary, but the system ensures that secondaries are not more than one version behind.

During a partition, Espresso uses a consensus-based leader election. The primary is elected by a majority of nodes in the partition's replica group. If the partition isolates the primary, a new primary is elected in the majority side. The minority side stops accepting writes.

**Result.** Espresso provides strong consistency with 99.99 percent availability. The cost: write throughput is limited by the primary node, and partition handling requires a consensus protocol that adds latency.

### ZooKeeper — CP

ZooKeeper is a distributed coordination service that provides strong consistency guarantees. It is CP in the CAP sense. ZooKeeper uses the Zab protocol (ZooKeeper Atomic Broadcast) to maintain a consistent state across all nodes.

ZooKeeper is used for: leader election, configuration management, service discovery, distributed locking, group membership. Any system that requires CP guarantees for coordination data.

Key characteristics: 3-5 node ensembles, all writes go through the leader, reads can be served by any node (with optional staleness), strict majority quorum, linearizable writes.

### etcd — CP

etcd is a distributed key-value store used by Kubernetes for cluster state management. It is CP, using the Raft consensus protocol. etcd is the most popular CP system for cloud-native applications.

Key characteristics: 3-5 node clusters, Raft-based consensus, watch API for real-time change notifications, TLS for security, backup and restore utilities. etcd is the storage backend for Kubernetes, so its CAP properties directly affect Kubernetes cluster behavior.

### Cassandra — AP with Tunable Consistency

Cassandra is the most popular open source AP system. It is designed for write-heavy workloads that require linear scalability. Cassandra's tunable consistency allows it to act as AP (W=1, R=1) or CP (W=QUORUM, R=QUORUM) depending on the operation.

Key characteristics: Ring architecture, consistent hashing, gossip protocol, hinted handoff, read repair, anti-entropy, CQL (Cassandra Query Language) similar to SQL.

### Riak — AP

Riak is an AP system based on the Dynamo paper. It focuses on high availability and operational simplicity. Riak uses consistent hashing, vector clocks for conflict detection, and allows application-level conflict resolution.

Key characteristics: Dynamo-inspired, vector clocks, CRDT support, multi-datacenter replication, HTTP API, operational simplicity.

## Cloud Provider Implementations

### AWS DynamoDB

DynamoDB is a fully managed AP service. It provides 99.999 percent availability SLA. Key features: on-demand capacity, auto-scaling, global tables, DAX (in-memory cache), streams, TTL.

DynamoDB's consistency model: eventually consistent reads (default, 1-second staleness window), strongly consistent reads (optional, may fail during partitions), transactional reads (ACID, across multiple items).

### Google Cloud Spanner

Spanner is a fully managed CP service with TrueTime. It provides 99.999 percent availability SLA. Key features: global distribution, ACID transactions, SQL interface, automatic replication, IAM security.

Spanner's consistency model: external consistency (strongest consistency model), globally consistent reads and writes, read-your-writes, monotonic reads.

### Azure Cosmos DB

Cosmos DB is a multi-model database that offers multiple consistency levels. It is the most flexible cloud database in terms of CAP trade-offs. You can choose from five consistency levels: strong, bounded staleness, session, consistent prefix, and eventual.

Strong consistency is CP (Cosmos DB uses a quorum-based protocol with multiple replicas). Eventual consistency is AP. Session consistency is a middle ground: it provides read-your-writes and monotonic reads within a client session.

## Lessons Learned

**Lesson 1: Know your consistency requirements before choosing a system.** Do not pick a database based on popularity. Pick based on the consistency guarantees you need. If you need strong consistency, do not pick a system that is AP by default. You will spend engineering time fighting the system's design.

**Lesson 2: Tunable consistency is a powerful tool.** Cassandra and Cosmos DB let you choose consistency per operation. This allows you to use strong consistency for critical operations and eventual consistency for high-throughput operations. The flexibility comes at the cost of complexity: you must understand which operations need which consistency level.

**Lesson 3: Partitions are not rare events.** Teams that design for the happy path discover the CAP theorem during their first production incident. Design for partitions from day one. Test with network failures. Use chaos engineering to validate your partition handling.

**Lesson 4: The cloud provider manages the infrastructure, not the CAP trade-off.** Moving to DynamoDB, Spanner, or Cosmos DB does not eliminate the CAP theorem. The cloud provider manages the infrastructure, but you still make the consistency vs availability choice. A misconfigured DynamoDB table with strongly consistent reads will fail during a partition just like a self-hosted CP system.

**Lesson 5: Consistency is a spectrum, not a binary.** The CAP theorem presents a binary choice during partitions. But between partitions, consistency is a spectrum: strong, read-your-writes, monotonic, bounded staleness, eventual. Each level provides different guarantees with different performance characteristics. Choose the weakest consistency level that meets your requirements.
