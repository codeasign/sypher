## Core Tension

The CAP theorem presents a hard constraint: during a network partition, you must choose between consistency and availability. When you choose consistency over availability, you are explicitly trading user-facing errors during partitions for strong data correctness. When you choose availability over consistency, you are trading data correctness guarantees for uninterrupted service.

This is not a one-time decision. It is a per-operation decision in many systems. Cassandra lets you set consistency level per query. DynamoDB lets you choose eventually consistent or strongly consistent reads per request. The system itself may mix CP and AP behaviors depending on the operation.

The core tension has operational consequences. A CP system that rejects requests during a partition must deal with client retry storms when the partition heals. An AP system that accepts diverging writes must deal with conflict resolution complexity. There is no free lunch.

The hardest part of the trade-off is that you make the decision at design time, but you feel the consequences only during failures. An AP system works perfectly during normal operation. When the partition hits, you discover the cost: lost updates, confused users, reconciliation nightmares. A CP system works perfectly during normal operation. When the partition hits, you discover the cost: the system is down for a subset of users.

## Decision Matrix

The following table captures the key decision points in a CAP-aware system. Each row represents a configuration knob, its available options, the trade-off it makes, and the operational impact.

| Decision Point | Options | Trade-off | Impact |

| Replication factor (N) | 1, 3, 5, 7 | Higher = better durability, higher storage cost, more nodes to coordinate. | N=3: standard for production. N=5: for multi-region, higher failure tolerance. N=1: never in production. |

### What Happens During a Partition

ZooKeeper (CP): The minority side stops serving all requests. Clients receive connection loss errors. Application code must handle this by retrying with exponential backoff, routing to a different ZooKeeper ensemble, or falling back to a cached configuration.

Cassandra (AP): Both sides continue serving requests. Writes are accepted and stored with hinted handoff. Reads may return stale data. When the partition heals, hinted handoff replays missed writes. Vector clocks or last-write-wins reconcile conflicts.

Spanner (CP with TrueTime): Spanner uses TrueTime, Google's synchronized clock service, to provide external consistency without blocking during partitions. TrueTime gives Spanner a bounded uncertainty window on clock skew. During a partition, Spanner delays writes until clock uncertainty is resolved, rather than blocking entirely. This is closer to CP than AP, but with better partition behavior than traditional CP systems.

## Failure Modes

### Failure Mode 1: Split-Brain

Split-brain occurs when two nodes both believe they are the leader. This is the nightmare scenario for CP systems. Two leaders accepting writes independently, each believing the other is dead.

The cause is a partition combined with an ambiguous leader election. Node A cannot reach Node B, declares itself leader, and starts accepting writes. Node B cannot reach Node A, declares itself leader, and starts accepting writes. Both sides accept writes. When the partition heals, the system has two divergent data sets.

Blast radius: total data loss for the diverged keys. Recovery requires manual reconciliation or snapshot restoration.

Detection: Monitor leader election events. Alert on any leader change that is not preceded by a confirmed node failure.

Recovery: Snapshot both sides. Determine which side has the authoritative data (by consensus log position, by timestamp, by business rules). Promote that side and discard the other. This is a manual process.

Prevention: Use a consensus protocol with strict quorum requirements. ZooKeeper requires a majority of nodes. With 3 nodes, a single node cannot declare itself leader alone. With 5 nodes, any node needs at least 3 votes.

### Failure Mode 2: Partition Recovery Storm

When a partition heals, the isolated nodes reconnect to the cluster. In a CP system, these nodes must catch up on all the writes they missed. The recovery process can trigger a storm.

The rejoining nodes request all missed writes from the quorum. The quorum nodes must serve these requests while also handling normal client traffic. The extra load can overwhelm the quorum. Latency spikes. Some nodes may crash under the load. The cluster enters a degraded state.

Blast radius: extended latency for all clients. Potential cascading failure if multiple nodes crash.

Detection: Monitor replication lag and the number of pending catch-up operations. Alert when lag exceeds 10 seconds.

Recovery: Throttle the catch-up rate. Prioritize the most recent writes first. Use backpressure from the quorum nodes.

Prevention: Provision extra capacity for the quorum to handle recovery workloads. Use a rate-limited recovery process.

### Failure Mode 3: Thundering Herd on Leader Failover

When a CP system loses its leader, it must elect a new one. All clients that were connected to the old leader reconnect to the new leader simultaneously. This is the thundering herd problem.

The new leader receives a burst of connection requests, read requests, and write requests. It may also need to catch up on replication if it was a follower. The combination of election overhead, client connection bursts, and catch-up replication can overwhelm the new leader.

Blast radius: 5-30 seconds of elevated latency or temporary unavailability.

Detection: Monitor leader election events and the latency spike that follows.

Recovery: Client-side connection pooling with exponential backoff. Pre-warm the new leader by limiting the rate of new connections.

Prevention: ZooKeeper uses a fast leader election algorithm (Zab) that completes in under a second. Raft uses randomized election timeouts to reduce the chance of concurrent elections.

### Failure Mode 4: Hint Queue Overflow

In an AP system, hinted handoff queues grow during a partition. Each write that cannot reach its target replica is stored locally. If the partition lasts long enough, the hint queue can overflow.

When the queue overflows, the oldest hints are discarded. The writes represented by those hints are lost. The system has silently lost data.

Blast radius: permanent data loss for writes made during the partition. The exact scope depends on how many hints were discarded.

Detection: Monitor hint queue depth. Alert when depth exceeds 50 percent of capacity.

Recovery: No automated recovery. The lost writes must be recreated from application-level logs or user retries.

Prevention: Increase hint queue capacity. Reduce hint TTL. Use a separate disk for the hint queue to avoid impacting normal operations.

### Failure Mode 5: Stale Read Cascade

In an AP system, a single stale read can cascade. Client A reads stale data, uses it to make a decision, and writes new data based on that stale read. Now the new data is derived from stale information.

This creates an inconsistency that propagates through the system. The stale read at the database level becomes stale state at the application level. Vector clocks cannot detect this because the application-level causality is invisible to the database.

Blast radius: widespread data corruption in the application layer. Hard to detect because the database looks consistent.

Detection: Application-level tracing and data lineage tracking.

Recovery: Application-specific reconciliation. May require manual intervention.

Prevention: Use strongly consistent reads for critical operations. Implement application-level idempotency checks. Use optimistic concurrency control with version numbers.

## Anti-Patterns

### Anti-Pattern 1: Ignoring Partitions

The most common anti-pattern is building a distributed system without thinking about partitions. Developers test only in a single-node environment. They assume the network is reliable. They discover the CAP theorem only when the system breaks in production.

Symptoms: unexplained data inconsistencies, mysterious write failures, split-brain in production.

Fix: Test with network failures. Use tools like Chaos Monkey or Jepsen to simulate partitions. Design for failure from day one.

### Anti-Pattern 2: Setting Consistency ALL Without Understanding the Cost

Consistency ALL means every write must be acknowledged by every replica. This seems like the safest choice. But ALL makes the system brittle. A single slow node or failed node blocks all writes. Availability drops to near zero.

Symptoms: Random write failures. High latency. Inability to survive any node failure.

Fix: Use QUORUM instead of ALL. QUORUM provides the same strong consistency during normal operation but allows the system to survive node failures.

### Anti-Pattern 3: Assuming CA Is Possible in a Distributed System

Teams sometimes design for CA, believing they can build a system that is both consistent and available without handling partitions. They use synchronous replication and assume the network never fails.

When the network does fail, they have no partition-handling logic. The system either becomes inconsistent (writes accepted on both sides) or unavailable (the application hangs waiting for acknowledgments from unreachable nodes).

Symptoms: The system works in testing but breaks catastrophically in production under network stress.

Fix: Accept that partitions happen. Decide in advance whether your system is CP or AP during partitions. Implement the corresponding partition-handling logic.

### Anti-Pattern 4: Using Eventual Consistency Where Strong Consistency Is Required

Teams sometimes choose an AP system for convenience, believing that eventual consistency is "good enough." Then they use that system for financial transactions, inventory management, or distributed locking.

Eventual consistency means the system explicitly allows stale reads. If you need read-your-writes consistency, a strictly eventually consistent system cannot provide it.

Symptoms: Duplicate charges, oversold inventory, conflicting locks.

Fix: Before choosing an AP system, audit every operation that requires strong consistency. Use strongly consistent reads for those operations, or use a CP system instead.

### Anti-Pattern 5: Blaming the CAP Theorem for Every Design Failure

The CAP theorem is sometimes used as an excuse for poor design. "We can't provide that feature because the CAP theorem says so." In many cases, the real constraint is not CAP but poor implementation, insufficient testing, or lack of understanding.

The CAP theorem applies only during network partitions. During normal operation, you can have both consistency and availability. Many systems that use CAP as an excuse are actually failing for other reasons: bad indexing, poor caching, inefficient queries.

## When Not to Use Distributed Systems

The CAP theorem applies only to distributed systems. If your system fits on a single machine, you do not need to make the CAP trade-off. A single PostgreSQL instance provides both consistency and availability (within the limits of that machine).

Consider a single-node solution when: your data fits on one machine, your throughput requirements are under 10,000 writes per second, your latency requirements are under 1 millisecond, and your availability requirements can tolerate single-node failure (or you use a managed service).

Consider a multi-node solution only when: your data exceeds a single machine's capacity, your throughput requires multiple machines, you need geographic distribution, or you need fault tolerance beyond what a single machine provides.

Even in a multi-node system, not every component needs to be distributed. Use a single-node cache for session data. Use a single-node job queue for low-throughput tasks. Only distribute components that actually need distribution.
