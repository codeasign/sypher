The introduction gave you the big picture. This section opens the hood and examines how requirements decompose, how they connect to system architecture, and how to manage their lifecycle in a real project.

## Internal Architecture of a Requirement

Every requirement, functional or non-functional, has an internal structure that determines how it is implemented, tested, and verified.

<AsciiDiagram id="functional-vs-non-functional-requirements/deepdive-requirement-anatomy" title="Anatomy of a Requirement" alt="Decomposition of a requirement into subject, action, object, condition, and acceptance criteria" caption="Every well-formed requirement decomposes into these five elements." content={`

│  │   [CONDITION], meeting [ACCEPTANCE CRITERIA]"│  │

│                                                    │
│  ┌────────────────────────────────────────────┐    │
│  │  Example FR                                │    │

A well-formed requirement has five components:

1. **Subject** — The entity or component the requirement applies to.
2. **Action or Property** — What the subject does (FR) or what quality is measured (NFR).
3. **Object** — The target of the action (FR) or scope of measurement (NFR).
4. **Condition** — The context in which the requirement applies.
5. **Acceptance Criteria or Measurement** — The testable threshold.

## Requirements Decomposition

A single high-level requirement decomposes into multiple lower-level requirements, each of which becomes an implementation task.

<AsciiDiagram id="functional-vs-non-functional-requirements/deepdive-decomposition" title="Requirements Decomposition Tree" alt="Tree diagram showing a top-level FR breaking down into sub-FRs and NFRs" caption="Each high-level requirement decomposes into sub-requirements across levels." content={`

│                                                    │

│   │Search│ │Filter│ │Detail│                     │
│   └──────┘ └──────┘ └──────┘                     │
│                           ┌──────┐               │
│                           │FR-3.1│               │
│                           │Auth  │               │
│                           │FR-3.2│               │

│                           │Chrg. │               │
│                           └──────┘               │
└───────────────────────────────────────────────────┘
  `} />

The decomposition follows a standard pattern:

- **Level 0** — Epic or business capability. "User purchases a product."
- **Level 1** — Major features. Product browsing, cart management, payment processing.
- **Level 2** — Sub-features. Search, filter, product detail, auth, validation, charge.
- **Level 3+** — Implementation tasks. API endpoint design, database schema, UI component.

At each level, both FRs and NFRs apply. The Level 0 epic has overall NFRs (checkout under 2 seconds). Each Level 1 feature has its own NFRs (search under 300ms). Each Level 2 sub-feature has implementation-specific NFRs (database query under 50ms).

└──────────────┴───────┴──────┴──────────┴─────────┴─────────┘
  `} />

Each row in the RTM links a requirement to:

- **Specification section** — Where the requirement is documented.
- **Design artifacts** — Which components, services, or modules implement it.
- **Test cases** — Which tests verify it.
- **Deployment configuration** — How it is maintained in production.

The RTM answers three critical questions:

1. **Coverage:** Are all requirements implemented and tested? (no orphan requirements)
2. **Impact analysis:** If this requirement changes, what breaks? (trace forward)
3. **Root cause:** If a test fails, which requirement does it trace back to? (trace backward)

## Prioritization Frameworks

Not all requirements are created equal. You need a systematic way to prioritize them.

### MoSCoW Method

The MoSCoW method categorizes requirements into four buckets:

<AsciiDiagram id="functional-vs-non-functional-requirements/deepdive-moscow" title="MoSCoW Prioritization" alt="Four-quadrant prioritization matrix with Must Have, Should Have, Could Have, Won't Have" caption="MoSCoW helps teams focus on what matters most for each release." content={`
┌─────────────────────────────────────────────────────────────┐
│                    MoSCoW PRIORITIZATION                    │
│                                                             │
│  ┌─────────────────────┐  ┌──────────────────────┐         │
│  │    MUST HAVE        │  │    SHOULD HAVE        │         │
│  │   (60-70%)          │  │    (15-25%)           │         │
│  │                     │  │                       │         │
│  │  Critical for MVP   │  │  Important but not    │         │
│  │  Without these,     │  │  critical. Can work   │         │
│  │  system has no      │  │  around if missing.   │         │
│  │  value.             │  │                       │         │
│  │                     │  │  FR: Order history    │         │
│  │  FR: User login     │  │  NFR: P99under 200ms      │         │
│  │  NFR: 99.9% uptime  │  │                       │         │
│  └─────────────────────┘  └──────────────────────┘         │
│                                                             │

│  │                     │  │                       │         │

│  │  and budget allow.  │  │  Prevents scope creep.│         │
│  │                     │  │                       │         │
│  │  FR: Social share   │  │  FR: Voice search     │         │
│  │  NFR: Dark mode     │  │  NFR: 99.999% uptime  │         │
│  └─────────────────────┘  └──────────────────────┘         │
└─────────────────────────────────────────────────────────────┘
  `} />

**Must Have** — If these requirements are not met, the system has no value. For an e-commerce site: user can browse products, add to cart, and check out. NFR: checkout works, data is secure.

**Should Have** — Important but not critical. A workaround exists. For e-commerce: order tracking, product recommendations.

**Could Have** — Nice to have. Include if resources permit. For e-commerce: social sharing, product comparison.

**Won't Have** — Explicitly excluded from this release. Prevents scope creep. For v1: voice search, AR try-on.

The percentages show typical allocation: 60-70% effort goes to Must Haves, 15-25% to Should Haves, 10-15% to Could Haves, and 5-10% explicitly deferred.

### Kano Model

The Kano model categorizes requirements by how they affect customer satisfaction:

<AsciiDiagram id="functional-vs-non-functional-requirements/deepdive-kano" title="Kano Model of Requirements" alt="Two-axis chart showing satisfaction vs functionality for basic, performance, and delight categories" caption="The Kano model explains which requirements drive customer satisfaction and which only prevent dissatisfaction." content={`
┌─────────────────────────────────────────────────────────┐
│                    KANO MODEL                           │
│                       ▲                                 │
│                  High │  ┌──────┐                       │
│              Satisf.  │  │DELT. │  ┌──────┐            │
│                       │  │ LIGHT│  │PERF. │            │
│                       │  └──────┘  │RMNCE │            │
│                       │            └──────┘            │
│                       │    ┌──┐                         │
│                       │    │  │                         │
│                  ─────┼────┴──┴──────────►              │
│                   Low │          Functionality   High   │
│                   Sat.│                                │
│                       │  ┌──────────┐                   │
│                       │  │  BASIC   │                   │
│                       │  │ EXPECTED │                   │
│                       │  └──────────┘                   │
│                       │  ▼                              │
│                       │  Low Satisfaction               │
└─────────────────────────────────────────────────────────┘
  `} />

- **Basic Expected (Threshold)** — Requirements customers assume will be there. A search bar on an e-commerce site. If missing, extreme dissatisfaction. If present, no increase in satisfaction. These are Must Haves.
- **Performance (Linear)** — The more you invest, the more satisfaction increases. Faster page load. Better search relevance. These correlate directly with satisfaction scores.
- **Delight (Excitement)** — Unexpected features that create high satisfaction. AR product preview, one-click reorder. Customers do not ask for these but love them when they appear. Over time, delighters become basic expectations.

For NFRs specifically: availability above 99.9% is a basic expectation for any serious service. Latency under 1 second is a performance attribute. "Works on an airplane with intermittent connectivity" could be a delight if done well.

### Stage Details

**Elicitation** is the most critical stage. Missing a requirement here costs 10x to 100x more to fix later. Techniques include:

- **Stakeholder interviews.** Ask "what does the system need to do?" and "how fast, how reliable, how secure?"
- **Domain analysis.** Study existing systems, competitor products, industry standards.
- **Document analysis.** Read RFPs, contracts, regulatory requirements.
- **Observation.** Watch users interact with current systems.
- **Workshops.** Bring multiple stakeholders together to negotiate and align.

**Analysis** is where raw stakeholder statements become structured requirements. For each statement:

1. Classify as FR or NFR.
2. For NFRs, determine the measurement and threshold.
3. Check for conflicts (e.g., "cheapest possible infrastructure" and "99.99% uptime" conflict).
4. Prioritize using MoSCoW or Kano.

**Specification** documents the requirements. In regulated environments, this follows IEEE 830 standard format. In agile environments, this means user stories and acceptance criteria.

**Validation** ensures the requirements match what stakeholders actually need. Walk through scenarios, prototype screens, simulate edge cases. The cost of fixing a misunderstood requirement at this stage is roughly 1x. After implementation, it is 10x. After release, 100x.

**Implementation** translates requirements into code. FRs become features. NFRs become design decisions and configuration.

**Verification** tests that requirements are met. FRs are verified by feature and integration tests. NFRs are verified by load tests, security scans, and chaos experiments.

**Operation** monitors that requirements remain met in production. FRs are tracked via business metrics. NFRs are tracked via SLIs, error budgets, and dashboards.

**Retirement** happens when a requirement is no longer relevant. Features are deprecated, NFR thresholds are adjusted, and the requirement is archived.

## Performance Characteristics of NFR Trade-offs

Different NFRs have different cost profiles. Understanding these profiles helps you make informed trade-off decisions.

| NFR | Cost Driver | Elasticity | Penalty for Violation |
|---|---|---|---|
| Latency P99 100ms → 50ms | 2x compute, edge caching | Low — linear cost improvement is expensive | User churn, conversion loss |
| Availability 99.9% → 99.99% | 3x infrastructure (redundancy) | Low — last 9 of availability is most expensive | Revenue loss, SLA penalties |
| Consistency strong vs eventual | 2x write latency, partition risk | Medium — depends on quorum size | Data anomalies, customer trust |
| Encryption at rest | 10-20% CPU overhead, key mgmt cost | High — scales with volume | Compliance fines, breach costs |
| Throughput 1K → 10K req/s | 2-3x compute + DB sharding | Medium — depends on architecture | Rate limiting, dropped requests |
| Durability 99.9999% → 99.999999% | 3x replication factor, cross-region | Low — last nines are exponential cost | Data loss liability |

The last few percentage points of any NFR cost exponentially more. Going from 99% to 99.9% availability requires redundancy. From 99.9% to 99.99% requires multi-region. From 99.99% to 99.999% requires fully active-active global deployment with automated failover.

## Configuration and Tuning

How you set NFR targets depends on system context:

<AsciiDiagram id="functional-vs-non-functional-requirements/deepdive-tuning" title="NFR Targets by System Context" alt="Table showing how NFR targets differ across startup, growth, enterprise, and regulated contexts" caption="NFR targets must be calibrated to the system's maturity and domain." content={`

├────────────────┼─────────────┼─────────┼──────────┼─────────┤

│ (10 users)     │             │         │          │         │
├────────────────┼─────────────┼─────────┼──────────┼─────────┤
│ Growth Stage   │ P99 under 500ms │ 99.9%   │ read-    │ moder.  │

│                │             │         │ write    │         │
├────────────────┼─────────────┼─────────┼──────────┼─────────┤
│ Enterprise     │ P99 under 200ms │ 99.99%  │ strong   │ higher  │
│ (1M users)     │             │         │          │         │
├────────────────┼─────────────┼─────────┼──────────┼─────────┤
│ Regulated      │ P99 under 500ms │ 99.995% │ strong + │ highest │

└────────────────┴─────────────┴─────────┴──────────┴─────────┘
  `} />

A startup building an MVP should not demand 99.99% availability. The cost would be 50% of their runway for 6 weeks of development. Instead, target 99.5% (3.6 hours downtime per month) and plan for higher availability in later stages.

A payment processing system, however, needs strong consistency and high durability from day one. The cost of a data inconsistency is higher than the infrastructure cost.

The key insight: **calibrate your NFRs to your business context, not to an abstract ideal.** The right availability target for a side project is different from the right target for a hospital's patient records system.
