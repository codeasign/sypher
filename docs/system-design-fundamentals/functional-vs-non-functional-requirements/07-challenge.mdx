This section contains hands-on challenges that test your ability to apply the FR vs NFR concepts from this topic. Each challenge presents a real-world scenario. Your job is to extract the requirements and assess their implications.

## Challenge 1: Historical Log Shipping Service

**Context:** A logistics company wants to build an internal system that tracks shipping containers across international routes. The system must record every location ping from GPS trackers on containers and make the data available for customer queries and internal analytics.

**Stakeholder input:**

- Operations VP: "We have 50,000 GPS trackers that ping every 5 minutes. Each ping is a small JSON payload. We need to know where every container is at any moment."
- Customer support manager: "Customers call asking where their shipment is. We need to answer in under 30 seconds."
- CFO: "This is a cost-sensitive project. Monthly infrastructure budget is $5,000. We cannot spend more than that."
- IT security: "Any data leaving our network must be encrypted end-to-end. We are subject to GDPR for European routes."

### Your Task

**Tier 1 — Extract the requirements:**

1. List 5 functional requirements for this system.
2. List 5 non-functional requirements with specific numbers.
3. Identify any conflicts between the requirements.
4. Propose a prioritization using MoSCoW.

**Tier 2 — Assess the architecture implications:**

1. Which NFR will most strongly shape the storage architecture?
2. Given the $5,000 budget, which NFRs must be relaxed from the ideal?
3. What is the monthly data volume? Show your calculation.
4. Recommend a database technology and explain your NFR-driven choice.

**Tier 3 — Handle a requirement change:**

1. After the system is built, Operations VP says: "We now need to update locations in real time. Every ping must be visible within 1 second." What changes and what breaks?
2. The CFO says: "Budget is now $2,000/month." Re-prioritize.

### Model Answer — Tier 1

**Functional requirements:**

| ID | Requirement |
|---|---|
| FR-1 | System receives GPS ping data from 50,000 trackers, each sending a JSON payload every 5 minutes. |
| FR-2 | System stores each ping with container ID, location (lat/lon), timestamp, and vessel ID. |
| FR-3 | Customer support can query a container's current location and recent history. |
| FR-4 | System generates a daily route summary for each active container. |
| FR-5 | System logs access to shipment data for GDPR compliance. |

**Non-functional requirements:**

| ID | Requirement |
|---|---|
| NFR-1 | Ingestion throughput: at least 167 pings/second (50,000 / (5 x 60)). With 2x peak buffer: 334 pings/second. |
| NFR-2 | Query latency: current location returned in under 5 seconds. |
| NFR-3 | Data durability: zero data loss — every ping must be persisted before acknowledgment. |
| NFR-4 | Security: all data encrypted at rest (AES-256) and in transit (TLS 1.3). |
| NFR-5 | Infrastructure cost: under $5,000/month total. |
| NFR-6 | Data retention: 90 days for raw pings, 2 years for daily summaries. |
| NFR-7 | GDPR compliance: data access logging, right to deletion, data residency in EU for European routes. |

**Conflicts identified:**

- **NFR-3 (zero data loss) vs NFR-5 ($5,000 budget):** Zero data loss requires synchronous replication and durable storage (SSD or EBS-backed). This adds significant cost. A $5,000 budget likely requires compromise: accept up to 1 minute of data loss and use cost-effective storage.
- **NFR-2 (5-second query) vs NFR-5 (budget):** Fast queries require indexed databases or search infrastructure. Hot data in fast storage + cold data in cheap storage can balance this.

**Prioritization (MoSCoW):**

- **Must Have:** FR-1, FR-2, FR-3, FR-5. NFR-1, NFR-4, NFR-7.
- **Should Have:** NFR-2 (relax to 10 seconds), NFR-3 (relax to 1 minute max data loss).
- **Could Have:** FR-4 (daily summaries can be batch processed).
- **Won't Have (v1):** Real-time alerts, anomaly detection, predictive ETA.

### Model Answer — Tier 2

**Most impactful NFR:** NFR-1 (334 pings/second ingestion). This drives a streaming ingestion architecture with a queue layer. Without a buffer, a burst of 1,000 trackers pinging simultaneously would overwhelm the database. A queue buffers the data and decouples ingestion from storage.

**Budget-driven relaxations:**

- Data retention: Store raw pings for 30 days in hot storage, archive to cold storage (S3 Glacier) for the remaining 60 days. This cuts storage cost by 60%.
- Replication: Use single-region with EBS snapshots every 5 minutes, not synchronous cross-region replication. Accept RPO of 5 minutes.
- Compute: Use spot instances for batch processing (daily summaries).

**Monthly data volume calculation:**

```
Each ping: 256 bytes (estimated JSON payload with metadata)
Trackers: 50,000
Frequency: every 5 minutes = 288 pings/day per tracker
Total daily: 50,000 x 288 = 14,400,000 pings
Daily data: 14,400,000 x 256 bytes = 3.69 GB/day
Monthly raw: 3.69 GB x 30 = 110.6 GB/month
With index and replication overhead: ~350 GB/month storage needed
```

**Database recommendation:** TimescaleDB (PostgreSQL extension for time-series data). It handles high-ingestion time-series data efficiently, supports compression (10:1 compression ratio for time-series data), has built-in partitioning by time, and supports SQL queries for ad-hoc lookups. Monthly storage cost for 350 GB on a single db.r6g.large instance: approximately $350. With managed TimescaleDB service: approximately $500/month. Fits the budget.

### Model Answer — Tier 3

**Change: real-time visibility (1 second latency).**

What changes:
- The ingestion pipeline must move from batch-flush to streaming. Kafka or Kinesis replaces batch SQS processing.
- The database must support real-time indexing. TimescaleDB's continuous aggregates handle this, but the write load increases, requiring a larger instance.
- Query latency NFR tightens from "5 seconds" to "1 second," requiring an in-memory cache (Redis) for current-location lookups.
- Cost increases: Redis cache approximately $100/month, larger DB instance approximately $200/month, Kafka approximately $150/month. Total adds $450/month.

What breaks:
- The $5,000 budget is now $5,450 — overshoot by 9%.
- The original architecture had 5-minute data windows, which simplified batch processing. Real-time breaks batch assumptions.
- Query patterns change: instead of "get current location from latest record," the system now needs a materialized current-location view updated within 1 second of each ping.

**Budget cut to $2,000/month:**

Re-prioritize:
- Reduce data retention: raw pings stored 14 days hot, 30 days warm, archive to S3 Glacier after 45 days.
- Remove Redis cache: current-location queries hit the database directly (accept 2-second latency).
- Downgrade DB instance from db.r6g.large to db.t3.large (burst vs dedicated CPU, save $150/month).
- Remove real-time requirement: revert to 5-minute visibility (saves Kafka cost).
- Batch daily summaries only (removes batch processing compute).
- Accept 10-minute RPO (snapshot frequency reduced).

## Challenge 2: Online Code Review Platform

**Context:** A startup is building a cloud-based code review platform similar to GitHub Pull Requests but simpler — focused on code review only, not source hosting. Developers push diffs, teammates review them inline, and the system tracks review status.

**Stakeholder input:**

- CTO: "We need to support 10,000 teams in the first year. Each team has 5-20 members. Reviews should feel instant."
- Lead developer: "The core workflow is: upload diff, assign reviewers, comment inline, approve or request changes, merge."
- Designer: "The UI must be responsive on desktop and mobile. No one reads documentation — the UX must be intuitive."
- Customer: "I am not uploading my proprietary code to a service that might leak it."

### Your Task

**Tier 1 — Extract and categorize:**

1. List 5 functional requirements for the platform.
2. List 5 non-functional requirements with specific targets.
3. Identify the single most critical NFR and explain why.
4. Estimate the storage and compute requirements for year 1.

**Tier 2 — Architecture impact:**

1. How does the security NFR affect the choice of hosted infrastructure?
2. How does the "instant review" NFR affect the data model?
3. What caching strategy supports the latency NFR?
4. What deployment topology supports the growth prediction?

**Tier 3 — Multi-tenant considerations:**

1. How should the system isolate code between teams?
2. What NFRs change if a large enterprise customer demands a dedicated instance?
3. How does the encryption NFR affect search and indexing?

### Model Answer — Tier 1

**Functional requirements:**

| ID | Requirement |
|---|---|
| FR-1 | Developer uploads a diff file or pastes code changes for review. |
| FR-2 | Developer assigns reviewers to a diff. |
| FR-3 | Reviewers see the diff with inline commenting on each changed line. |
| FR-4 | Reviewers can approve, request changes, or comment without a vote. |
| FR-5 | System tracks review status (open, changes requested, approved, merged) and notifies participants on status changes. |

**Non-functional requirements:**

| ID | Requirement |
|---|---|
| NFR-1 | Diff view loads in under 1 second at P95 for diffs up to 5,000 lines. |
| NFR-2 | All code data encrypted at rest (AES-256) and in transit (TLS 1.3). |
| NFR-3 | 99.9% availability for core workflow (diff upload, review, comment). |
| NFR-4 | Supports 200 concurrent diff uploads/second at peak (year 1 estimate). |
| NFR-5 | Notification delivery within 30 seconds of the triggering event. |

**Most critical NFR: NFR-2 (security).** Developers will not upload proprietary code if they do not trust the platform. A security breach on day one kills the company. All other NFRs are secondary to building trust through demonstrable security. This means dedicated encryption keys per tenant, SOC 2 compliance before launch, and a published security architecture.

**Year 1 storage estimate:**

```
Teams: 10,000
Avg team size: 10 developers = 100,000 users
Avg diff size: 50 KB (code changes) + 200 KB (comments and metadata) = 250 KB per review
Avg reviews per team per week: 20
Annual reviews: 10,000 x 20 x 52 = 10,400,000
Storage: 10,400,000 x 250 KB = 2.6 TB
With replication (3x) and backups: ~10 TB total
```

### Model Answer — Tier 2

**Security NFR on infrastructure:**
- Requires a cloud provider with HSM (Hardware Security Module) support for key management.
- All data must be encrypted per tenant, meaning each team's encryption key is separate.
- No shared database tenants — use schema-per-tenant or, at minimum, row-level security with encryption.
- Infrastructure must support VPC isolation with no cross-tenant network paths.

**Instant review on data model:**
- Diff content must be stored in a format optimized for rendering: structured line-by-line rather than as a blob.
- The diff view is read by the reviewer immediately on upload. The system must pre-render the diff and cache it.
- Comments are stored as references to specific file, line number, and commit hash. The view must merge comments with the diff in under 1 second.
- Solution: Store diff as an array of line objects (line number, type: added/removed/context, content). Cache the rendered diff in Redis keyed by diff_id and version.

**Caching strategy:**
- CDN for static assets (CSS, JS, logo — 0ms load for repeat visits).
- Application cache (Redis): rendered diffs, team member lists, review status. TTL of 5 minutes for status, 1 hour for rendered diffs (rarely changes after upload).
- Database query cache: team membership queries, frequently accessed review metadata.
- Estimated cache working set: 10,000 most recent rendered diffs at 50 KB each = 500 MB. Plus metadata: 1 GB total. Single Redis instance (cache.r6g.large, 13 GB) serves this easily.

**Deployment topology for year 1:**
- Single-region (us-east-1) with multi-AZ deployment.
- 4 application servers behind load balancer, auto-scaling to 12 at peak.
- PostgreSQL primary in us-east-1a with synchronous replica in us-east-1b and async replica in us-west-2 for disaster recovery.
- Redis cluster with 3 nodes for high availability.
- When team count reaches 100,000 in year 3: migrate to multi-region active-active.

### Model Answer — Tier 3

**Tenant isolation:** Schema-per-tenant approach. Each team gets its own PostgreSQL schema. This prevents cross-tenant data leaks (one team cannot accidentally query another team's diffs). The application routes requests to the correct schema based on team_id from the authentication token. Encryption keys are stored in AWS KMS with one key per team.

**Enterprise dedicated instance:** When a large enterprise demands a dedicated instance, two NFRs change:
- Availability must increase to 99.99% (dedicated infrastructure, SLA-backed).
- Security isolation becomes physical, not just logical (dedicated VPC, dedicated database cluster).
- Cost per customer increases 10x, but the enterprise pays for it.

**Encryption and search:** Encrypted data cannot be searched in the database. Two options:
- Search on the client side: download all encrypted diffs, decrypt locally, search. Only works for small sets.
- Use searchable encryption: maintain a separate encrypted index where search terms are hashed (for exact match) or use tokenization for full-text search.
- Recommendation: exclude diff content from search index at the database level. Search only on metadata (review title, author, status). For full-text diff search, use a dedicated search index with encryption at the application layer.

## Challenge 3: IoT Sensor Monitoring Platform

**Context:** An agricultural tech company wants to monitor soil moisture, temperature, and humidity sensors across 10,000 farms. Each farm has 50-200 sensors. Sensor data is used to generate irrigation recommendations.

**Stakeholder input:**

- Farm operations: "Sensors report every 15 minutes. We need a dashboard showing current conditions and 7-day trends."
- Data science: "We need 2 years of historical data to train ML models for drought prediction."
- Farm owner: "If the system goes down during growing season, I cannot water my crops properly. It must be available."
- Vendor: "Sensors run on batteries. Each transmission costs battery. Minimize acknowledgments."

### Your Task

**Tier 1 — Requirements:**

1. List the core functional requirements.
2. Calculate the peak write throughput.
3. Determine the data retention NFR and its storage cost implication.
4. Identify the conflict between sensor battery life and data reliability.

**Tier 2 — Architecture:**

1. Design the ingestion pipeline from sensor to database.
2. Choose a database and justify with NFRs.
3. Design the dashboard query path for under 3-second load time.
4. Estimate the monthly AWS cost for the solution.

### Model Answer — Tier 1

**Core FRs:**
- Sensor reports soil moisture, temperature, and humidity every 15 minutes.
- Dashboard displays current readings for a selected farm with 7-day trend charts.
- System generates irrigation recommendation based on current soil moisture.
- Data science team can export up to 2 years of historical data.
- System alerts farm operator when soil moisture drops below configurable threshold.

**Peak write throughput:**

```
Farms: 10,000
Sensors per farm: 100 (average)
Total sensors: 1,000,000
Report interval: 15 minutes = 4 reports/hour per sensor
Reports per second: 1,000,000 x 4 / 3600 = 1,111 writes/second
With 50% peak burst (all sensors report within same minute): ~16,667 writes/second peak
```

**Data retention:**
- Raw sensor data: retain 90 days in hot storage (fast queries)
- Aggregated hourly data: retain 2 years in warm storage (trend charts)
- ML training data: export to cold storage (S3 Glacier, 2+ years)

**Storage cost:**
```
Single reading: 50 bytes (sensor_id + farm_id + 3 metrics + timestamp)
Daily writes: 1,000,000 x 96 (reports/day) = 96,000,000 readings
Daily raw data: 96,000,000 x 50 = 4.8 GB/day
90 days hot: 4.8 GB x 90 = 432 GB
With indexing overhead: ~1 TB for hot storage
Warm storage (hourly aggregates): ~500 GB for 2 years
Glacier storage: ~50 GB for exported training data

Monthly storage cost: ~$300 for hot (SSD), ~$50 for warm (HDD), ~$5 for Glacier
```

**Conflict: battery life vs reliability.** Sensors that need to conserve battery should not wait for write acknowledgments. But the system needs reliable data ingestion. Resolution: use a fire-and-forget protocol from sensors to a gateway. The gateway acknowledges immediately and handles retries internally. The sensor does not need end-to-end acknowledgment.
