Requirements are abstract. Architecture is concrete. This section bridges the gap by showing how functional and non-functional requirements translate into specific design decisions, infrastructure choices, and deployment patterns.

## Naive Implementation: Only FRs Considered

The most common mistake in early-stage design is building for functional requirements alone. The result works at small scale but breaks under real-world conditions.

This architecture satisfies all functional requirements:

- Users can register and log in (auth module handles it)
- Users can search products (search queries the DB)
- Users can add to cart (cart module, single DB table)
- Users can check out (payment module, single DB transaction)
- Users receive confirmation email (email module, synchronous)

### Breaking Points

**At 1,000 users:** Works fine. Response times under 500ms. No issues.

**At 100,000 users:** The monolith starts struggling. The search query takes 3 seconds because the products table has 1 million rows with no dedicated index. The checkout flow blocks during payment processing. The email module slows down the entire response because it sends SMTP synchronously. Database connection pool exhausts at peak. Response times at P95 exceed 5 seconds.

**At 10 million users:** The architecture collapses. Database queries time out. The single server cannot handle the request volume. Vertical scaling tops out at 128 GB RAM and 64 vCPUs. Every deployment requires full system restart. A single query spike takes down the entire application. Availability drops to 97% (10 days of downtime per year). Users abandon the platform.

## Production-Grade Architecture: FRs + NFRs

Now consider the same functional requirements but with explicit non-functional requirements:

- **Latency:** P99 under 200ms for read operations, P99 under 1s for writes
- **Availability:** 99.99% uptime
- **Scalability:** 100x traffic spikes handled gracefully
- **Security:** PCI DSS compliant for payment processing
- **Durability:** Zero data loss

### How NFRs Reshape Each Layer

The production architecture addresses each NFR with specific design decisions:

**Latency NFR (P99 under 200ms) drives:**
- CDN at the edge for static assets and API response caching
- Redis caches in front of every service for frequently accessed data
- Database read replicas to offload primary from read traffic
- Connection pooling and query optimization at every database access

**Availability NFR (99.99%) drives:**
- Load balancer with health checks and automatic instance replacement
- Each service deployed with at least 2 replicas across availability zones
- Active-active multi-region deployment so one region can absorb all traffic if the other fails
- Circuit breakers on all inter-service calls to prevent cascade failures

**Scalability NFR drives:**
- Services decomposed so each can scale independently (search scales separately from checkout)
- Autoscaling groups with CPU and memory-based policies
- Queue-based async processing so traffic spikes are buffered
- Stateless service design so any instance can handle any request

**Security NFR (PCI DSS) drives:**
- Payment processing isolated in its own service with minimal data exposure
- Payment queue encrypts message payload at rest and in transit
- Payment workers have no direct access to user or order databases
- All traffic over TLS 1.3, all secrets in Vault

**Durability NFR (zero data loss) drives:**
- Database synchronous replication within region
- Automated backups every 5 minutes to S3 with cross-region copy
- Write-ahead log replication to stand-by replica in secondary region
- Payment queue with at-least-once delivery and deduplication on the consumer side

For the P99 under 200ms NFR, the typical request flow is:

1. Browser checks its cache for the resource. If fresh (Cache-Control: max-age=300), response is instant вҖ” 0ms.
2. CDN edge checks its cache. If warm, serves in under 20ms from the nearest edge location.
3. Application checks Redis. If the data is cached (80%+ hit rate for popular items), return in under 5ms.
4. Only on cache miss: query database (50-200ms depending on query complexity and indexing).

With this layered approach, P99 latency stays under 200ms even at 10x peak traffic. Without caching, P99 latency can exceed 2 seconds during traffic spikes.

## Integration Patterns: How NFRs Shape Service Design

Different NFR combinations produce different service interaction patterns.

### Pattern 1: Synchronous Request-Response

**When:** Latency NFR is tight (P99 under 100ms), consistency is required, and services are within the same data center.

**Architecture:** Service A calls Service B directly via gRPC or HTTP. Both services must be available. Circuit breakers prevent cascade failures.

**NFRs met:** Low latency, strong consistency.
**NFRs challenged:** Availability (if B is down, A fails).

### Pattern 2: Async with Queue

**When:** Latency NFR is moderate (P99 under 2s), reliability is critical, and processing can be deferred.

**Architecture:** Service A publishes to a queue (SQS, RabbitMQ, Kafka). Service B consumes when ready. Queue provides buffering, retry, and delivery guarantees.

**NFRs met:** High availability (queue buffers traffic during B outages), durability (messages persist until processed).
**NFRs challenged:** End-to-end latency (queuing adds delay).

### Pattern 3: Event-Driven with Pub/Sub

**When:** Loose coupling is required, multiple consumers need the same data, and consistency can be eventual.

**Architecture:** Service A publishes events to a topic. Multiple subscribers process independently. Each subscriber maintains its own view of the data.

**NFRs met:** Scalability (subscribers scale independently), maintainability (services are loosely coupled).
**NFRs challenged:** Consistency (eventual only, not strong).

### Pattern 4: Write-Ahead Log Replication

**When:** Zero data loss is required (durability NFR), and cross-region replication is needed.

**Architecture:** All writes go to a primary database. The write-ahead log (WAL) is streamed to replicas in other regions. Replicas apply the WAL in order.

**NFRs met:** Zero data loss (WAL is durable before acknowledgment), cross-region replication.
**NFRs challenged:** Write latency (every write must be logged to disk).

### NFR Translation to Multi-Region

**Latency NFR: P99 under 200ms for users in Europe, Asia, and North America.**

Without multi-region: a user in Singapore hitting a us-east-1 server sees 200-300ms network latency before any processing. This alone violates the NFR.

With multi-region: each user hits the nearest region. Cross-region network latency is eliminated. Processing time plus local network latency stays under 150ms for the P99.

**Availability NFR: 99.99% вҖ” maximum 52 minutes downtime per year.**

A single-region deployment in us-east-1 has had 3 significant outages in the past 5 years, each lasting 2-6 hours. Single-region availability is approximately 99.95% in practice.

With two active regions (us-east-1 and eu-west-1), if one region goes down, the global load balancer routes all traffic to the remaining region. Assuming both regions fail independently (they do not share power, cooling, or network), combined availability exceeds 99.995%.

**Five-nines availability (99.999%) requires three regions.** If any two regions fail simultaneously, the third absorbs all traffic. This requires fully active-active services in each region with zero dependency on a specific regional service.

### Multi-Region Database Strategy

| NFR | Strategy | Trade-off |
|---|---|---|
| P99 read under 100ms globally | Read replicas in each region with local cache | Replicas may lag by 1-5 seconds |
| P99 write under 200ms globally | Writes go to nearest region primary | Cross-region replication adds 50-200ms |
| No data loss | Synchronous WAL replication to 2nd region | Write latency increases by 50-100ms |
| 99.99%+ availability | Active-active with automatic failover | Application must handle conflict resolution |
| Strong consistency across regions | Single primary region for writes | Write latency varies by user location |

## Capacity Planning from NFRs

Non-functional requirements directly determine infrastructure sizing. Here is how to translate NFR targets into concrete resource estimates.

### Step 1: Estimate Peak Throughput

Given:
- Average daily active users: 1 million
- Average requests per user per session: 20
- Peak-to-average ratio: 3x (for e-commerce, typical peak is 3x the daily average)
- Peak happens over a 2-hour window

```
Daily requests = 1,000,000 x 20 = 20,000,000
Peak hourly requests = 20,000,000 / 24 x 3 = 2,500,000 per hour
Peak requests per second = 2,500,000 / 3600 = 695 RPS
```

Add 20% headroom: **835 RPS peak.**

### Step 2: Sizing from Latency NFR

Given P99 latency must be under 200ms:
- Average processing time per request on a single instance: 50ms
- That means one instance can handle 20 requests per second (1000ms / 50ms)
- P99 under 200ms means servers must not exceed 40% CPU utilization at peak (to avoid queuing)
- Effective capacity per instance at 40% CPU: 8 requests per second
- Instances needed: 835 / 8 = 105 instances
- Add zone redundancy (3 availability zones): 105 / 3 = 35 instances per zone
- Add N+1 per zone: 36 per zone, 108 total

### Step 3: Sizing from Durability NFR

Given zero data loss:
- All writes must be acknowledged by primary and at least one replica
- Each write operation takes 10ms on primary + 20ms for replica acknowledgment
- Effective write throughput: approximately 33 writes/second per DB node
- If the system has 100 writes/second peak: need 3 DB instances in the replication cluster

### Step 4: Sizing from Memory NFRs

Given cache hit rate must be 90%+ for popular content:
- Popular content size: 100 GB (product catalog top 20%)
- Working set must fit in Redis memory with 20% overhead: 120 GB
- Redis cluster: 6 nodes of cache.r6g.xlarge (20 GB each) = 120 GB
- Or 3 nodes of cache.r6g.2xlarge (40 GB each) = 120 GB

### Sample Capacity Plan

| Resource | FR-Driven | FR + NFR-Driven | Delta |
|---|---|---|---|
| App servers | 2 (t3.medium) | 108 (c6g.large) | 54x more |
| Database | 1 (db.t3.medium) | 6 (db.r6g.large, 1 primary, 5 replicas) | 6x more, plus multi-AZ |
| Cache | 0 | 3 (cache.r6g.xlarge) | New cost |
| CDN | 0 | Global (CloudFront, 10 TB/month) | New cost |
| Queue | 0 | 1 (SQS standard, 5M messages/day) | New cost |
| Monitoring | 0 | Full stack (Prometheus, Grafana, PagerDuty) | New cost |
| Monthly cost | $100 | $8,500 | 85x more |

This cost delta explains why startups build for FRs first and add NFRs later. But it also explains why mature systems require explicit NFR budgeting вҖ” the architectural cost is not visible until the NFRs are quantified.

## Deployment and Operations: NFRs in Production

Non-functional requirements must be monitored and enforced in production, not just verified in staging.

### SLI, SLO, and Error Budget

For each NFR, define:

- **SLI (Service Level Indicator):** The actual measurement. "P99 latency of the search endpoint over a 5-minute window."
- **SLO (Service Level Objective):** The target. "P99 latency under 200ms, 99.9% of the time over a 30-day rolling window."
- **Error Budget:** The allowed headroom. "With 99.9% SLO, we can have 43 minutes of P99 over 200ms per month."

### Alerting on NFR Breaches

Different NFRs trigger different response levels:

| SLO Breach | Severity | Response |
|---|---|---|
| P99 latency 2x over target for 5 minutes | P2 (high) | Investigate within 15 minutes. Check for recent deploy, traffic anomaly, or resource exhaustion. |
| P99 latency 3x over target for 2 minutes | P1 (critical) | Page on-call immediately. Roll back recent deploy or scale up compute. |
| Availability drops under 99.9% for 10 minutes | P1 (critical) | Page on-call. Activate incident response. Initiate failover if single-region issue. |
| Error rate exceeds 1% for 5 minutes | P1 (critical) | Page on-call. Check for code defect, dependency failure, or configuration change. |
| Cache hit rate drops under 70% for 15 minutes | P3 (low) | Investigate during business hours. Possible cache invalidation issue or traffic pattern change. |

### Deployment Strategy Aligned to NFRs

The deployment strategy must protect NFRs during code rollouts:

- **Canary deploys:** Route 1% of traffic to the new version first. Monitor SLIs for 10 minutes. If P99 stays under 200ms, increase to 10%, then 50%, then 100%.
- **Blue-green deploys:** Maintain two identical stacks. Deploy to the inactive stack, run smoke tests, then swap traffic. If NFR breaches occur, swap back.
- **Feature flags:** Toggle features on/off without deploy. If a new feature degrades P99 latency, disable it immediately.
- **Rollback automation:** If P99 exceeds 2x target or error rate exceeds 1% for 5 minutes after deploy, automatically roll back to the previous version.
