## How Companies Shard Data at Scale

The following case studies show how four major technology companies implemented sharding to solve specific scalability challenges. Each case study begins with the problem, describes the architecture, and explains the key design decisions based on publicly available engineering blog posts and conference talks.

---

## Case Study 1: Discord — From MongoDB to Cassandra Sharding

### The Problem

Discord launched in 2015 on a single MongoDB instance. By 2016, the platform had grown to 100 million messages per day. The MongoDB instance hit 50 GB of messages and began to show performance degradation. Write latency increased during peak hours. The single-instance architecture could not scale further.

Discord evaluated MySQL (relational model was a poor fit for chat messages), Cassandra (proven at scale for messaging), and ScyllaDB (faster Cassandra-compatible alternative written in C++). They chose Cassandra for its proven track record at Netflix and Apple.

### The Architecture

Discord sharded messages by guild ID (server ID). Each guild's messages are stored together because queries always filter by guild: "show me messages in the #general channel of the Gaming Server." The shard key is the guild ID, using Cassandra's built-in partitioner (Murmur3 consistent hashing).

### Key Design Decisions

**Guild ID as partition key.** Discord's analysis showed that every query filters by guild. A user opens a server and sees messages for that server. Guild-level sharding means that 99 percent of queries hit exactly one partition. This was the natural choice.

**Channel ID as clustering key.** Within each guild partition, messages are clustered by channel ID and ordered by message ID. This enables efficient range queries: "show messages in channel X from position Y to Z." Cassandra's wide-row model stores all messages for a channel in a single CQL row, which is efficient for ordered reads.

**Replication factor 3.** Each message is written to 3 nodes. If one node fails, the other two serve reads. Cassandra's hinted handoff and read repair ensure eventual consistency without a coordinator.

### Lessons Learned

**Large partitions are a problem.** Discord hit Cassandra's 2 GB partition size limit on the largest guilds. A guild with 500,000 members chatting in one channel created partitions exceeding 2 GB. Cassandra's performance degrades significantly when individual partitions exceed 100 MB. Discord's solution: split large guilds into multiple partitions using a "shard split" operation.

**Consistent hashing makes scaling manageable.** When Discord added nodes, Cassandra automatically rebalanced data with minimal manual intervention. Murmur3 consistent hashing ensured that only neighboring nodes on the hash ring exchanged data.

**Read repair is not free.** Discord found that read repair (compares replicas during reads) added 10-20ms of latency to read queries on nodes with inconsistent data. They tuned read repair consistency to LOCAL_QUORUM to balance latency and consistency.

## Case Study 2: Uber — Trip Data Sharded by City

### The Problem

Uber processes 25 million trips per day across 70 countries and 10,000 cities. By 2015, the trip database (a single PostgreSQL instance) had reached 50 TB. Queries for "show all trips in San Francisco on March 15" scanned the entire table and took over 30 seconds. Write throughput was capped at 10,000 writes per second, causing backpressure during peak hours.

### The Architecture

Uber built Schemaless, a partitioned document database on top of MySQL. Trips were partitioned first by city, then by day within each city. Uber used Ringpop, a consistent hashing library built on top of SWIM (Scalable Weakly-consistent Infection-style Process Group Membership), to assign city shard groups to physical nodes.

### Key Design Decisions

**City as the partition key.** Uber's query log analysis showed that 90 percent of queries include a city filter. A dispatcher viewing trips always specifies the city. This single decision made sharding effective for the dominant query pattern.

**Two-level partitioning.** Within each city, data is partitioned by day. This enables two critical operations: time-range queries ("show trips from yesterday") prune to exactly one daily partition, and data retention is implemented by dropping entire day partitions (a metadata operation).

**Ringpop for membership.** Uber's Ringpop library provided consistent hashing with virtual nodes (128 per physical node). When a node joined or left the cluster, only the affected city shard groups moved. This made cluster scaling operations routine rather than extraordinary.

**Schemaless storage.** Trip data was stored as JSON documents in MySQL. This avoided schema changes during the rapid evolution of the trip data model. The database had no fixed schema — the application interpreted the JSON.

### Results

From the published case study: query latency for dispatcher queries dropped from over 30 seconds to under 50ms. Table lock contention was eliminated. Write throughput scaled horizontally with city shard groups. Data retention (90-day rolling window) became a simple partition-drop operation.

## Case Study 3: Netflix — Sharding the Data Pipeline

### The Problem

Netflix's data pipeline in 2015 processed 500 billion events per day (streaming events, UI interactions, error logs). The pipeline used multiple Cassandra clusters, each sharded differently. The schema changed frequently as Netflix added new device types, UI experiments, and personalization features.

### The Architecture

Netflix sharded its event data by a composite key: `(date_hour, event_type)`. Events were written to a Cassandra cluster where the partition key was the hour bucket, and the clustering key was the event type. This enabled efficient range scans: "get all errors from the last hour."

### Key Design Decisions

**Time-based bucket as partition key.** Grouping events by hour provided bounded partition sizes. Each hour's events for a given type fit in a predictable partition size. Netflix tuned the bucket size (hour, 30 minutes, 10 minutes) based on event volume.

**Separate clusters for separate workloads.** Netflix maintained separate Cassandra clusters for real-time event processing, batch analytics, and operational monitoring. Each cluster had a different shard count, replication factor, and consistency level optimized for its workload. This prevented the analytics workload from impacting real-time event processing.

### Lessons Learned

**Time-based partitioning creates temporal hot spots.** The current hour's partition received all writes. Netflix mitigated this by writing to multiple "hot" buckets in parallel and using a compaction strategy optimized for time-series data.

**Rebalancing time-based data is complex.** Moving an hour bucket from one node to another requires replaying that hour's event stream. Netflix pre-created hour buckets with a TTL-based expiration to avoid rebalancing entirely — old buckets expired automatically.

## Case Study 4: Cloudflare — Sharding the DNS Database

### The Problem

Cloudflare's DNS resolution service handles 25 million DNS queries per second across 330 data centers worldwide. The authoritative DNS database stores zone configurations for 25 million domains. Every DNS query requires looking up the zone configuration.

### The Architecture

Cloudflare shards its DNS zone database by zone name (domain name). The shard key is a hash of the zone name modulo N shards. Each shard is a SQLite database stored on local SSD, replicated to multiple data centers via a custom replication protocol.

### Key Design Decisions

**SQLite as the database engine.** Cloudflare chose SQLite for its zero-configuration operation and excellent read performance. Each SQLite database is a single file, making replication, backup, and recovery trivial. SQLite handles the workload because each shard receives a modest number of writes (zone configuration changes are rare compared to reads).

**Per-data-center read replicas.** Cloudflare replicates each shard to multiple data centers. A DNS query in Tokyo reads from the local SQLite replica, not the source of truth in us-east-1. This provides sub-10ms read latency regardless of geographic location.

**Change Data Capture (CDC) for replication.** When a zone configuration changes, the change is captured from the source shard's SQLite WAL and propagated to all replicas via a streaming pipeline. Replication lag is under 1 second.

### Lessons Learned

**SQLite is surprisingly effective for sharded workloads.** The conventional wisdom is that SQLite does not scale. Cloudflare proved that sharding data across many SQLite instances (one per shard) scales well when writes are infrequent and reads dominate.

**Local storage is faster than network storage.** Cloudflare's architecture stores each shard on local SSD rather than a SAN or network filesystem. This eliminates network latency from the read path. The trade-off is that recovery from a failed node requires restoring from a replica rather than spinning up a new network-mounted instance.

## Open Source Implementations

### Vitess (CNCF Graduated)

Vitess shards MySQL using a proxy architecture. VTGate routers intercept SQL queries, extract the shard key, and route to the correct MySQL shard. Vitess supports both hash-based and range-based sharding with online resharding.

**Trade-offs:** Adds 3-8ms of latency per query through the proxy layer. Requires running etcd for metadata management. Online resharding is complex but proven at YouTube scale (where Vitess originated).

**Best for:** MySQL workloads that have outgrown a single instance. Organizations that want to keep MySQL compatibility with application drivers.

### Cassandra / ScyllaDB

Cassandra shards using consistent hashing with virtual nodes. Each node is responsible for a range of the hash ring. Writes are replicated to N nodes (configurable replication factor). ScyllaDB is a Cassandra-compatible database written in C++ with significantly better performance.

**Trade-offs:** AP system — eventual consistency by default. Strong consistency requires quorum reads/writes, which reduce availability. CQL is SQL-like but lacks JOINs, subqueries, and transactions.

**Best for:** Write-heavy workloads, time-series data, messaging systems. Any workload where availability and partition tolerance are more important than strong consistency.

### Citus (PostgreSQL Extension)

Citus shards PostgreSQL using a coordinator node. The coordinator stores metadata about which shard holds which key ranges. Queries go through the coordinator, which either routes to one shard or fans out.

**Trade-offs:** Single coordinator can become a bottleneck. Limited support for complex queries (subqueries, CTEs across shards). Provides PostgreSQL compatibility and ACID transactions within a shard.

**Best for:** Multi-tenant SaaS applications where each tenant fits in one shard. Real-time analytics on time-series data.

## Cloud Provider Implementations

### AWS: Amazon Aurora Global Database + Sharding

AWS does not offer a managed sharding service. Instead, Aurora Global Database provides multi-region replication. Sharding must be implemented at the application layer using a sharding library or a proxy. AWS recommends using Amazon RDS Proxy with application-level sharding.

**Cost/ops trade-off:** Low operational overhead (RDS manages the database), but sharding logic is the customer's responsibility. Aurora auto-scales storage to 128 TB, delaying the need for sharding for many workloads.

### GCP: Cloud Spanner (Auto-Sharded)

Google Cloud Spanner is a globally distributed, strongly consistent database that handles sharding automatically. Spanner splits data across nodes using a split algorithm based on key ranges. The application sets the primary key, and Spanner handles the rest.

**Cost/ops trade-off:** Premium pricing (approximately $10/hour per node). Zero sharding operational overhead. Strong consistency and global replication built in. Best for applications that need sharding's scalability without the operational burden.

### Azure: Cosmos DB (Partitioned by Design)

Azure Cosmos DB requires every collection to have a partition key. The partition key determines how data is distributed across physical partitions. Cosmos DB automatically splits partitions that exceed 20 GB.

**Cost/ops trade-off:** Pay-per-request pricing (RU/s). Automatic partition management. Limited to Cosmos DB's query model (SQL-like with restrictions). Best for greenfield applications that can design around Cosmos DB's query limitations.

## Lessons Learned

Five key lessons from the case studies, applicable to any sharding project.

**1. Choose the shard key based on the dominant query pattern, not data distribution.** Discord sharded by guild ID because every query filters by guild. Uber sharded by city because 90 percent of queries filter by city. Cloudflare sharded by domain name because every DNS query specifies a domain. The shard key must match the access pattern — uniform data distribution is a secondary concern.

**2. Two-level partitioning provides both scalability and operational convenience.** Uber's city + date scheme gave them both geographic query isolation and time-based retention. Discord's guild + channel scheme gave them guild-level routing and channel-level ordering. A single shard level forces a compromise between query performance and data management.

**3. Rebalancing is the hardest operation in distributed systems.** Every company that did rebalancing the wrong way lost data or caused outages. Use proven tools (Vitess reshard, Cassandra nodetool, custom tool with dual-writes and checksum verification). Test rebalancing in staging before running in production.

**4. Monitor per-shard, not per-cluster.** Cluster-wide averages hide hot spots, skew, and individual shard failures. Discord's large partition problem was invisible until individual partition sizes were monitored. Uber's hot city shard was invisible until per-city latency was tracked. Every metric that matters for the cluster matters more for individual shards.

**5. Sharding is a last resort, not a first choice.** Indexing, read replicas, connection pooling, vertical scaling, and caching solve 90 percent of database scaling problems with far less complexity. Cloudflare's SQLite sharding worked because the simpler alternatives (indexing, vertical scaling) had already been exhausted. Discord went to Cassandra because MongoDB's single-instance limit was a hard ceiling. Verify that simpler approaches have failed before committing to sharding.
