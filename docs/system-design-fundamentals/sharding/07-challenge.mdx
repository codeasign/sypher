## How to Approach These Challenges

Each challenge presents a concrete scenario. Solve it by identifying the shard key, choosing a strategy, estimating capacity, planning for skew, and explaining the tradeoffs. The three tiers progress from basic conceptual understanding to complex multi-region design.

All answers are expandable. Attempt each challenge before revealing the solution.

## Tier 1: Foundation

These challenges test your grasp of basic sharding concepts. Each should take 3-5 minutes.

### Challenge 1.1: Choose the Right Shard Key

A video streaming platform stores a `views` table with columns: `view_id` (auto-increment PK), `video_id`, `user_id`, `timestamp`, `duration_watched_seconds`, `device_type`, `country`, `isp`. The table receives 10 million new rows per day. The most common queries are:

- "Show total views for video X in the last 24 hours" (70 percent of queries)
- "Show a user's watch history" (20 percent of queries)
- "Aggregate views by country for a monthly report" (10 percent of queries)

**Question:** Choose a shard key and strategy. Justify why the other candidates are worse.

<details>
<summary>Solution</summary>

**Shard key:** `video_id` with hash-based sharding across 64 shards.

**Why video_id?** 70 percent of queries filter by `video_id`. The shard key should match the dominant access pattern. Hash sharding distributes videos uniformly — each shard holds roughly the same number of videos and receives roughly the same query load.

**Why not user_id?** Only 20 percent of queries filter by `user_id`. Partitioning by `user_id` would mean 80 percent of queries become scatter-gather — the opposite of what sharding should achieve.

**Why not timestamp?** Only the "last 24 hours" filter is time-constrained. But queries for different videos at the same time would hit the same partition, creating a temporal hot spot. All queries for the most popular video of the day would also be time-partitioned, creating a double hot spot.

**Why not composite (video_id + timestamp)?** Composite sharding by `video_id` (hash) then `timestamp` (range) would provide both video-level pruning and time-range pruning. This is a valid alternative if the storage system supports it. However, the added complexity is not justified when 70 percent of queries already prune to a single shard with `video_id` alone.

**Capacity:** 10M rows/day x 100 bytes = 1 GB/day. For 90-day retention: 90 GB. With hash sharding across 64 shards: approximately 1.4 GB per shard. Comfortably small.

</details>

### Challenge 1.2: Explain Resharding Without Downtime

**Question:** A social network uses hash-based sharding with 8 shards. The user base has grown to 100 million users, and each shard now holds 400 GB of data and handles 8,000 writes per second during peak. The team decides to double the shard count to 16. Walk through the resharding process without downtime. What are the critical failure points?

<details>
<summary>Solution</summary>

**Step 1: Provision new shards.** Deploy 8 new database instances (shards 8-15) with the same schema as the existing shards. Configure replicas and monitoring.

**Step 2: Start background data migration.** For each of the 8 old shards, compute which keys now map to new shards under `hash(key) % 16`. This is approximately 50 percent of keys per old shard. Stream those keys from old shards to the corresponding new shards using a consistent snapshot.

**Step 3: Enable dual-writes.** Once the initial sync completes, configure the application to write new data to both old and new shards. A write to key K goes to both `old_shard = hash(K) % 8` and `new_shard = hash(K) % 16`. The old shard write is synchronous (the application waits for it). The new shard write can be asynchronous (fire-and-forget or queued).

**Step 4: Verify consistency.** Run a checksum comparison between old and new shards for migrated keys. Any discrepancies must be resolved by replaying writes from the old shard's WAL.

**Step 5: Update the routing table.** Atomically flip the routing table from 8-shard to 16-shard topology. After the flip, all new reads and writes go to the new shards. The router must be updated atomically — either via a metadata service (etcd) push or a coordinated configuration rollout.

**Step 6: Remove old shards.** Keep old shards running in read-only mode for 48 hours to allow rollback if needed. After the verification period, decommission the old shards.

**Critical failure points:**
- **Dual-write race condition:** A write during the routing table flip could be written to the old shard but not the new shard. Use a cutoff timestamp: only route writes with a timestamp after the cutoff to the new shard.
- **Checksum mismatch:** Inconsistent data between old and new requires WAL replay. Have the replay script ready before starting the migration.
- **Connection storm:** After the flip, all application instances open connections to the new shards simultaneously. Warm the connection pools before the flip.
- **Rollback complexity:** If something goes wrong after the flip, the old shards must still have the latest data. Do not drop old shards until the verification period expires.

</details>

### Challenge 1.3: Spot the Shard Key Problem

**Question:** A team shards the `users` table by `country` column. The table has 200 million rows. The top 3 countries (US, India, Brazil) account for 65 percent of all users. The remaining 195 countries account for 35 percent. What problems will this sharding strategy cause?

<details>
<summary>Solution</summary>

**Three major problems:**

**1. Massive data skew.** The US partition holds approximately 65 million users (200M x 0.33). India approximately 45 million. Brazil approximately 32 million. Most other country partitions hold under 500,000 users. The largest partition (US) is 130x the size of the smallest partition.

**2. Hot spot on large country shards.** Queries for US users dominate traffic. The US shard handles 33 percent of all read and write traffic while handling only 1/200 of the shards. The US shard's CPU and I/O are saturated while most other shards sit idle.

**3. Query pattern mismatch.** Users query by `user_id`, not by `country`. Most queries would require `WHERE country = ? AND user_id = ?` to prune to a single shard. If the application queries by `user_id` without `country`, every query becomes a scatter-gather across all 200 shards.

**Fix:** Use `user_id` as the shard key with hash-based sharding. This distributes users uniformly regardless of country. If country-based queries are needed for compliance, add a secondary index on `country` within each shard and accept the scatter-gather cost for those queries. Alternatively, use composite partitioning: partition by `country` (list) at the top level, then by `user_id` (hash) within each country.

</details>

## Tier 2: Design

### Challenge 2.1: Design a Sharded Ride-Hailing Database

**Scenario:** A ride-hailing company stores trip data for 50 million users across 10,000 cities. The trip table receives 25 million new rows per day. Each row is approximately 1 KB. The dominant queries are:

- "Show my last 20 trips" — filtered by `rider_id` (55 percent of queries)
- "Show trips in city X in the last hour" — filtered by `city_id` and `timestamp` (25 percent)
- "Show trip details by trip_id" (15 percent)
- "Aggregate daily revenue by city" (5 percent)

**Task:** Design the sharding strategy. Specify:
- Shard key(s) and strategy (hash, range, composite)
- Estimated shard count with math
- Routing architecture (proxy-based or application-level)
- How queries without the shard key are handled

<details>
<summary>Solution</summary>

**Shard key: `rider_id` with hash-based sharding across 256 shards.**

55 percent of queries filter by `rider_id`, making it the dominant access pattern. Hash-based sharding distributes riders uniformly. Trips for the same rider are co-located on the same shard, enabling efficient "last 20 trips" queries.

**Capacity estimate:**
- Daily volume: 25M trips x 1 KB = 25 GB/day
- Yearly volume: 25 GB x 365 = 9.1 TB/year
- 3-year projection: 27.3 TB
- With index overhead (30 percent): 35.5 TB
- Target per-shard: 250 GB
- Shard count: 35.5 TB / 250 GB = 142 shards, round to 256 (power of 2)

**Write throughput:**
- Peak writes: 25M / 86,400 x 3 (peak ratio) = 868 writes/second
- Per shard at peak: 868 / 256 = 3.4 writes/second
- Well within a single PostgreSQL/MySQL instance's capacity

**Routing architecture:** Proxy-based using Vitess VTGate. Three VTGate instances behind an HAProxy load balancer. Routing table stored in etcd with 10-second refresh. Connection pool of 50 connections per shard per VTGate.

**Handling non-shard-key queries:**
- "Show trips in city X in the last hour" (25 percent): Create a materialized view partitioned by `city_id` + `hour`. The materialized view is populated asynchronously by a streaming pipeline (Kafka -> consumer -> materialized view). Queries against the materialized view prune to one shard by `city_id`.
- "Show trip details by trip_id" (15 percent): Store a secondary index on `trip_id` within each shard. Accept the scatter-gather cost for these queries — they represent only 15 percent of traffic and can use dedicated read replicas.
- "Aggregate daily revenue by city" (5 percent): Run this as a batch job during off-peak hours using scatter-gather. Cache the results for 24 hours.

</details>

### Challenge 2.2: Design a Multi-Tenant SaaS Sharding Strategy

**Scenario:** A SaaS platform serves 2,000 tenants. Three tenants have 50 million records each. The remaining 1,997 tenants have under 100,000 records each. Total database: 500 GB. The platform hosts 500 GB of data total. Growth rate: 50 percent per year.

**Task:** Design a sharding strategy that prevents large tenants from consuming all resources on shared shards. Specify the shard key, the number of shards, and how you handle tenant isolation.

<details>
<summary>Answer</summary>

**Strategy:** Directory-based sharding with tenant-to-shard mapping.

**Large tenant handling:** Assign each of the three large tenants to their own dedicated shards. Shard A: tenant_1 (50M records). Shard B: tenant_2 (50M records). Shard C: tenant_3 (50M records). Each large tenant gets its own database instance with predictable performance.

**Small tenant handling:** Distribute the remaining 1,997 small tenants across 8 shared shards. Each shared shard holds approximately 250 tenants x 100,000 records = 25 million records = approximately 2.5 GB. This is small enough that further sharding is unnecessary.

**Growth plan:** When any shard (dedicated or shared) exceeds 80 percent of its target capacity, add a new shard and move tenants. For shared shards, move the largest tenants first.

**Routing:** A directory service (etcd or a simple PostgreSQL table) maps `tenant_id` to `shard_id`. The application caches this mapping with a 60-second TTL. When a new tenant is created, the application assigns it to the least-loaded shared shard.

**Pros:** Large tenants get dedicated resources. Small tenants share cost-effectively. Adding capacity is straightforward.

**Cons:** Directory lookups add 1-5ms of latency per query (if not cached). Manual tenant placement decisions are needed when large tenants grow. Rebalancing a large tenant is expensive (50M records must move).

</details>

### Challenge 2.3: Design a Global Sharded Chat System

**Scenario:** Design a sharding strategy for a real-time chat application with 100 million daily active users across 500,000 guilds (servers). The system stores 500 million messages per day. Dominant queries: "show the last 50 messages in channel X" (90 percent) and "search messages in a channel" (10 percent).

**Task:** Specify the shard key, database technology, routing strategy, and how you handle large guilds that exceed partition size limits.

<details>
<summary>Answer</summary>

**Shard key:** `guild_id` with consistent hashing (Cassandra or ScyllaDB).

**Why consistent hashing:** Chat workloads have unpredictable guild growth. A gaming guild may grow from 100 to 500,000 members in a month. Consistent hashing allows adding nodes incrementally as guilds grow, without full cluster rebalancing.

**Database technology:** Cassandra (or ScyllaDB for better performance). Cassandra's wide-row model stores all messages for a channel in a single CQL row with clustering by `message_id`. Read queries for "last 50 messages" are efficient range scans within a single Cassandra partition.

**Schema design:**
```cql
CREATE TABLE messages (
    guild_id    bigint,
    channel_id  bigint,
    message_id  timeuuid,
    author_id   bigint,
    content     text,
    created_at  timestamp,
    PRIMARY KEY ((guild_id), channel_id, message_id)
) WITH CLUSTERING ORDER BY (channel_id ASC, message_id DESC);
```

The partition key is `guild_id`. Within each guild partition, messages are clustered by `channel_id` and ordered by `message_id` descending. The query "show last 50 messages in channel X" reads a single Cassandra partition with `WHERE guild_id = X AND channel_id = Y LIMIT 50`.

**Handling large guilds:** Cassandra has a practical limit of 100 MB per partition (the hard limit is 2 GB, but performance degrades beyond 100 MB). For a guild with 500,000 members and 1 million messages per day, a single guild partition exceeds 100 MB within weeks. Solution: implement a "guild split" that distributes a large guild across multiple physical partitions using a sub-shard key:
```cql
PRIMARY KEY ((guild_id, sub_shard), channel_id, message_id)
```
Where `sub_shard = channel_id % N` splits the guild's data across N sub-partitions. The application queries all sub-partitions in parallel and merges results.

**Routing:** Application-level routing using a Cassandra driver with token-aware load balancing. The driver computes the target node from the partition key hash and routes directly — no proxy needed.

</details>

## Tier 3: Expert

These challenges require deep system design knowledge and operational experience. Each should take 20-30 minutes.

### Challenge 3.1: Design an Online Resharding System

**Scenario:** Your team manages a sharded MySQL database with 32 shards using hash modulo (`hash(key) % 32`). The data volume has grown from 5 TB to 50 TB over two years. You need to scale to 128 shards. The system serves 100,000 requests per second during peak hours and tolerates zero downtime.

**Task:** Design the resharding pipeline. Specify the data migration strategy, the consistency verification mechanism, the routing table update protocol, and the rollback plan. Include the timeline estimate.

<details>
<summary>Solution</summary>

**Data migration strategy: Dual-write with consistent snapshot.**

**Phase 1: Preparation (week 1)**
- Deploy 96 new MySQL instances (shards 32-127) with the same schema and configuration.
- Set up replication from old shards to new shards for the initial data sync.
- Measure the network bandwidth between old and new shard groups.
- Deploy a staging environment that mimics the production topology for testing.

**Phase 2: Initial sync (week 2)**
- For each of the 32 old shards, take a consistent snapshot using `mysqldump --single-transaction --master-data=2`.
- Load the snapshots into the corresponding new shards. With 50 TB of data and 100 MB/s network bandwidth: 500,000 seconds = 5.8 days of continuous data transfer. Accelerate by running 32 parallel transfers (one per old shard). Parallel load: 32 streams x 100 MB/s = 3.2 GB/s. Total: 50 TB / 3.2 GB/s = 15,625 seconds = 4.3 hours.
- Record the binlog position from the snapshot for each old shard.

**Phase 3: Catch-up replication (week 2, after sync)**
- Start binlog-based replication from each old shard to its corresponding new shards. Replication streams all changes that occurred during the snapshot phase.
- Monitor replication lag: target lag under 5 seconds.
- If replication lag exceeds 30 seconds, increase replica resources or reduce concurrent traffic.

**Phase 4: Enable dual-writes (week 2-3)**
- Deploy a dual-write middleware layer. The middleware intercepts all write queries and sends them to both the old shard (`hash(key) % 32`) and the new shard (`hash(key) % 128`).
- Old shard writes are synchronous (application waits for acknowledgment). New shard writes are asynchronous (queued to a Kafka topic, processed by a consumer that writes to the new shard).
- Monitor dual-write latency: target under 5ms added latency.
- Verify that the new shards have consistent data by running periodic checksum comparisons.

**Phase 5: Routing table flip (week 3, go time)**
- Perform the routing table flip during a scheduled maintenance window (lowest traffic, typically 3-4 AM).
- Steps:
  1. Pause the dual-write middleware (queue remaining writes).
  2. Verify checksums one final time.
  3. Update the routing table in etcd from 32-shard to 128-shard topology.
  4. VTGate instances receive the push notification and update their in-memory routing table.
  5. Resume the middleware in read-only mode for old shards.
  6. Route all new reads and writes through the 128-shard routing table.
- Total flip time: under 30 seconds. Application impact: writes paused for 30 seconds.

**Phase 6: Verification and cleanup (week 3-4)**
- Monitor query latency, error rates, and data consistency for 48 hours.
- Keep old shards in read-only mode as a rollback option.
- After verification, decommission old shards: remove from the routing table, stop replication, release storage.

**Rollback plan:**
- If the new routing table causes errors, revert to the 32-shard routing table in etcd. VTGate instances pick up the rollback within 10 seconds.
- Old shards have all the latest data because dual-writes kept them in sync.
- Resume writes to old shards. The 30-second pause is the only data the new shards have that old shards do not — replay those 30 seconds from the dual-write queue.

**Timeline estimate:** 3-4 weeks including preparation, staging testing, migration, and verification. The data transfer itself takes approximately 4 hours for 50 TB with 32 parallel streams.

</details>

### Challenge 3.2: Diagnose and Fix a Production Sharding Incident

**Scenario:** You are the on-call engineer for a 64-shard Vitess cluster serving an e-commerce platform. At 2:15 PM on Black Friday, you receive a PagerDuty alert. The `orders` table is showing p99 latency of 4.2 seconds (baseline: 50ms). Error rate is 12 percent (baseline: 0.1 percent). The engineering team reports that the checkout flow is failing for some customers.

**Available data:**
- Cluster-wide average latency: 4.2 seconds
- Shard 41 p99 latency: 15ms (normal)
- Shard 42 p99 latency: 5.2 seconds (elevated)
- Shard 43 p99 latency: 18ms (normal)
- All other shards: under 25ms
- Shard 42 CPU: 98 percent (other shards: 25-35 percent)
- Shard 42 connections: 187 of 200 in use
- Shard 42 disk IOPS: 15,000 / 20,000 max
- Cross-shard query count: normal (no unusual scatter-gather activity)
- Application logs: errors are `connection refused` and `query timeout` for shard 42

**Questions:**
1. What is the likely root cause of the incident?
2. What immediate action do you take?
3. What is the long-term fix?
4. How do you prevent this from happening again?

<details>
<summary>Solution</summary>

**Root cause analysis:**

Shard 42 is a hot spot. The symptoms are classic: elevated latency on one shard, normal latency on all others; high CPU and connection utilization on the hot shard; errors are connection timeouts specific to the hot shard.

The likely cause is a popular product or promotional item whose `product_id` hash maps to shard 42. During Black Friday, a flash sale or promotion for this product causes all write traffic (orders, inventory updates) to concentrate on shard 42. A single product's order data is on one shard because orders are sharded by `order_id` or `product_id`.

**Immediate action (within 5 minutes):**

1. **Identify the hot key.** Query the VTGate slow query log to find the most frequent shard key value on shard 42 in the last 15 minutes. This identifies the product or promotion causing the hot spot.

2. **Throttle or redirect traffic.** If the hot key is a single product (e.g., product_id 48192): add an application-level rate limiter for that product. In parallel, add 2-3 read replicas behind shard 42 to absorb read queries. If the hot key is an entire promotion (range of product IDs): temporarily increase shard 42's instance resources (vertical scaling) to buy time.

3. **Enable caching for the hot key.** If the queries are read-heavy (product page views), add a Redis cache for the hot product's data. This reduces read load on shard 42 from tens of thousands of QPS to hundreds.

**Long-term fix (within 1 week):**

1. **Split the hot shard.** Split shard 42 into two shards. Use Vitess's `SplitClone` and `SplitDiff` tools to split the shard without downtime. After the split, the hot product's data is on one shard, and the rest of shard 42's data is on another. This halves the load on each shard.

2. **Implement query-aware routing.** For the hot product, route read queries to replicas and write queries to the primary. This prevents read load from competing with write load on the hot shard.

3. **Deploy a circuit breaker.** Configure VTGate to detect when shard 42's latency exceeds 100ms and automatically route read queries to replicas. Implement a fallback: if the shard is completely unavailable, serve cached data or a degraded experience.

**Prevention:**

1. **Add per-shard traffic anomaly detection.** Configure Prometheus to alert when any shard's QPS exceeds 2x the trailing 7-day average for that shard. This detects hot spots within minutes of onset.

2. **Pre-split known hot products.** Before major sales events (Black Friday, Cyber Monday), identify the top 10 products by expected traffic and ensure they are on separate shards or have dedicated resources.

3. **Implement application-level request hedging.** For queries to critical shards, send the query to both the primary and a replica in parallel. The first response wins. This reduces p99 latency when the primary is under load.

4. **Run a Black Friday game day.** Simulate the hot spot scenario in staging. Test the split operation, the circuit breaker, and the caching strategy. Measure the time to detect, diagnose, and mitigate.

</details>
