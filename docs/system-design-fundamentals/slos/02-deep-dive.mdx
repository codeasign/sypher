## Internal Architecture

An SLO computation system consists of four layers: instrumentation, aggregation, evaluation, and alerting. Each layer processes data at a different granularity and cadence. Understanding how these layers interact is essential for building a reliable SLO system.

### Instrumentation Layer

The instrumentation layer runs inside the application process. It intercepts every request, measures latency and status code, and emits a metric event. The instrumentation must be lightweight because it runs on the hot path of every request.

A typical instrumentation library implements a middleware that wraps the request handler. The middleware records the start time, calls the handler, records the end time, and emits a metric event with the latency in milliseconds, the status code, the endpoint name, and any relevant labels.

The metric event is emitted to a local buffer. The buffer is flushed to the aggregation layer every 10 seconds or when it reaches 10,000 events, whichever comes first. Batching improves throughput by reducing the number of network round trips.

### Aggregation Layer

The aggregation layer collects metric events from all service instances and groups them into time windows. The standard window size is 1 minute. Each window stores the total number of requests and the number of bad requests (requests that failed the SLI criteria).

The aggregation layer also computes histograms for latency SLIs. A histogram stores the count of requests in each latency bucket. Standard buckets are: 0-1ms, 1-5ms, 5-10ms, 10-25ms, 25-50ms, 50-100ms, 100-250ms, 250-500ms, 500-1000ms, 1000-2500ms, 2500-5000ms, and 5000ms+.

Histograms are more memory-efficient than storing every individual latency value. For a service processing 10,000 requests per second, storing every latency value would require 864 million data points per day. The histogram reduces this to 12 buckets per minute — 17,280 data points per day. This is a 50,000x reduction in storage.

### Evaluation Layer

The evaluation layer reads the aggregated windows and computes SLO compliance. For each SLO, it iterates over the windows in the measurement period, counts the number of good and bad windows, and computes the compliance percentage.

The evaluation layer also computes burn rates for multiple window sizes. For each window size (5 minutes, 30 minutes, 6 hours, 3 days), the evaluator computes the error rate over that window and normalizes it by the SLO target to produce the burn rate.

The evaluation is performed every minute for all active SLOs. For a system with 500 SLOs, this means 500 evaluations per minute, each reading 30 days of 1-minute windows (43,200 data points per SLO). The total data read is 21.6 million data points per minute. This is feasible with a time-series database that supports range queries.

### Alerting Layer

The alerting layer compares the computed burn rates against configured thresholds. Each threshold has a burn rate, a window size, and a notification channel. The alerting layer also implements hysteresis — once an alert fires, it will not fire again for a cooldown period (typically 30 minutes for page-level alerts).

The alerting layer must handle alert fatigue carefully. False positives are the leading cause of ignored alerts. Multi-window multi-burn-rate alerting reduces false positives by requiring both a high burn rate and a sustained window before alerting.

## Data Structures and Algorithms

### The Sliding Window

The fundamental data structure in SLO computation is the sliding window. The sliding window maintains a fixed-size buffer of the most recent N time windows. As each new minute elapses, the oldest minute is evicted and the newest minute is added.

The sliding window is implemented as a circular buffer. A circular buffer of size 43,200 (minutes in 30 days) stores the count of good and bad requests for each minute. The buffer has a head pointer that points to the current minute. When the current minute advances, the head pointer wraps around and overwrites the oldest entry.

```python
class SlidingWindow:
    def __init__(self, window_minutes: int = 43200):
        self.window_minutes = window_minutes
        self.buffer = [WindowEntry() for _ in range(window_minutes)]
        self.head = 0
        self.total_good = 0
        self.total_bad = 0

    def advance(self, new_minute: WindowEntry):
        # Subtract the evicted minute from totals
        evicted = self.buffer[self.head]
        self.total_good -= evicted.good
        self.total_bad -= evicted.bad

        # Add the new minute
        self.buffer[self.head] = new_minute
        self.total_good += new_minute.good
        self.total_bad += new_minute.bad

        # Advance the head pointer
        self.head = (self.head + 1) % self.window_minutes

    def compliance(self) -> float:
        total = self.total_good + self.total_bad
        if total == 0:
            return 1.0
        return self.total_good / total

    def error_budget_remaining(self, slo_target: float) -> float:
        total = self.total_good + self.total_bad
        if total == 0:
            return 1.0
        actual = self.total_good / total
        budget = 1.0 - slo_target
        consumed = 1.0 - (actual / slo_target)
        return max(0.0, 1.0 - consumed)
```

The sliding window supports O(1) insertion and O(1) compliance computation. The trade-off is memory: a 30-day window at 1-minute resolution requires 43,200 entries. Each entry stores two integers (good and bad count) plus a timestamp. At 24 bytes per entry, the total memory is approximately 1 megabyte per SLO.

### The Multi-Window Burn Rate

The burn rate is computed over multiple window sizes simultaneously. Each window size has its own sliding window buffer. The burn rate for a window of size W is:

```
burn_rate_W = error_rate_W / (1.0 - slo_target)
```

Where error_rate_W is the fraction of bad requests in the last W minutes.

The multi-window burn rate is computed using a cascade of sliding windows. The 5-minute window is a sliding window of size 5. The 30-minute window is a sliding window of size 30. The 6-hour window is a sliding window of size 360. The 3-day window is a sliding window of size 4,320.

```python
class MultiWindowBurnRate:
    def __init__(self, slo_target: float):
        self.slo_target = slo_target
        self.windows = {
            "5min": SlidingWindow(5),
            "30min": SlidingWindow(30),
            "6hr": SlidingWindow(360),
            "3day": SlidingWindow(4320),
        }

    def advance(self, entry: WindowEntry):
        for window in self.windows.values():
            window.advance(entry)

    def burn_rate(self, window_name: str) -> float:
        window = self.windows[window_name]
        total = window.total_good + window.total_bad
        if total == 0:
            return 0.0
        error_rate = window.total_bad / total
        allowed_rate = 1.0 - self.slo_target
        if allowed_rate == 0:
            return float("inf")
        return error_rate / allowed_rate

    def check_alerts(self) -> list:
        alerts = []
        burn_5min = self.burn_rate("5min")
        # Burn rate 14.4 over 5 minutes = page
        if burn_5min >= 14.4:
            alerts.append({"severity": "page", "burn_rate": burn_5min, "window": "5min"})
        # Burn rate 6.0 over 30 minutes = ticket
        burn_30min = self.burn_rate("30min")
        if burn_30min >= 6.0:
            alerts.append({"severity": "ticket", "burn_rate": burn_30min, "window": "30min"})
        # Burn rate 1.0 over 6 hours = notification
        burn_6hr = self.burn_rate("6hr")
        if burn_6hr >= 1.0:
            alerts.append({"severity": "notify", "burn_rate": burn_6hr, "window": "6hr"})
        return alerts
```

### The Histogram for Latency SLOs

Latency SLOs require tracking the distribution of latencies, not just the count of good and bad requests. The histogram data structure stores the count of requests in each latency bucket.

```python
class LatencyHistogram:
    def __init__(self, buckets: list):
        # buckets: [1, 5, 10, 25, 50, 100, 250, 500, 1000, 2500, 5000]
        self.buckets = sorted(buckets)
        self.counts = [0] * (len(buckets) + 1)

    def record(self, latency_ms: float):
        for i, bucket in enumerate(self.buckets):
            if latency_ms <= bucket:
                self.counts[i] += 1
                return
        self.counts[-1] += 1

    def percentile(self, p: float) -> float:
        # p is 0.0 to 1.0, e.g. 0.99 for p99
        total = sum(self.counts)
        target = total * p
        cumulative = 0
        for i, count in enumerate(self.counts):
            cumulative += count
            if cumulative >= target:
                if i == 0:
                    return 0.0
                if i > len(self.buckets):
                    return self.buckets[-1] * 2  # beyond last bucket
                return self.buckets[i - 1]
        return self.buckets[-1] * 2

    def merge(self, other: "LatencyHistogram"):
        for i in range(len(self.counts)):
            self.counts[i] += other.counts[i]
```

The histogram supports O(1) recording and O(B) percentile computation, where B is the number of buckets (typically 12). Merging two histograms is O(B). This makes histograms ideal for distributed aggregation — each service instance maintains its own histogram, and the aggregator merges them.

### The SLO Evaluator

The SLO evaluator ties everything together. It reads the aggregated windows, computes compliance, checks burn rates, and generates alerts. The evaluator runs once per minute on a scheduled basis.

```python
class SLOEvaluator:
    def __init__(self, slo_id: str, sli_type: str,
                 slo_target: float, window_days: int):
        self.slo_id = slo_id
        self.sli_type = sli_type  # "request" or "latency"
        self.slo_target = slo_target
        self.window_minutes = window_days * 24 * 60
        self.sliding_window = SlidingWindow(self.window_minutes)
        self.burn_rate = MultiWindowBurnRate(slo_target)

    def evaluate(self, minute_data: WindowEntry) -> dict:
        self.sliding_window.advance(minute_data)
        self.burn_rate.advance(minute_data)

        compliance = self.sliding_window.compliance()
        budget_remaining = self.sliding_window.error_budget_remaining(
            self.slo_target
        )
        alerts = self.burn_rate.check_alerts()

        return {
            "slo_id": self.slo_id,
            "compliance": compliance,
            "meeting_slo": compliance >= self.slo_target,
            "budget_remaining": budget_remaining,
            "alerts": alerts,
            "timestamp": minute_data.timestamp,
        }
```

## Request Lifecycle

The request lifecycle traces a single request from the client through the system and into the SLO computation pipeline. Each step adds information that contributes to the SLO evaluation.

### Step 1: Client Sends Request

The client sends an HTTP request to the service. The request enters the load balancer, which routes it to an application instance. The time the request enters the load balancer is the start of the latency measurement. This is the closest measurement to the true user experience.

### Step 2: Instrumentation Middleware Intercepts

The instrumentation middleware, running as part of the application framework, intercepts the request before it reaches the handler. The middleware records the current time with nanosecond precision using a monotonic clock. The monotonic clock is essential because wall-clock time can jump forward or backward due to NTP synchronization, which would corrupt latency measurements.

### Step 3: Handler Processes Request

The application handler processes the request. This may involve database queries, cache lookups, calls to downstream services, and computation. The handler eventually produces a response with a status code and response body.

### Step 4: Middleware Records Outcome

After the handler returns, the middleware calculates the elapsed time by subtracting the start time from the current time. It records the latency in milliseconds, the status code, and the endpoint name. It also records any error that occurred during processing.

### Step 5: Metric Event Is Buffered

The middleware writes the metric event to a local buffer. The buffer is a thread-safe queue that can hold up to 10,000 events. If the buffer is full, the middleware drops the event rather than blocking the request. Dropping metrics is acceptable because the SLO is computed over millions of requests — a few dropped events do not affect the result.

### Step 6: Buffer Flushes to Aggregator

Every 10 seconds, or when the buffer reaches 10,000 events, the buffer is flushed to the aggregation layer. The flush sends a batch of events in a single HTTP request or via a message queue. The aggregator acknowledges the batch and the buffer is cleared.

### Step 7: Aggregator Classifies the Minute

The aggregator groups events by the minute they occurred. For each minute, it counts the total number of requests and the number of requests that failed the SLI criteria. A failure is any request where the latency exceeds the threshold, the status code indicates an error, or the request timed out.

### Step 8: Window Entry Is Written

The aggregator writes a window entry for each minute. The window entry contains the timestamp, the total count, the bad count, and optionally the latency histogram. The entry is written to the time-series database and to the in-memory sliding window.

### Step 9: Evaluator Reads Window Entry

The SLO evaluator reads the new window entry and advances its sliding window. The evaluator computes the new compliance percentage, the remaining error budget, and the burn rates for all window sizes. If any burn rate exceeds its threshold, the evaluator generates an alert.

### Step 10: Alert Is Delivered

If an alert is generated, the alert manager determines the appropriate notification channel. Critical alerts are sent to PagerDuty, which pages the on-call engineer. Medium-severity alerts create tickets in the incident management system. Low-severity alerts update the dashboard.

## Consistency and State

### Crash Recovery

When the SLO evaluator process crashes, the in-memory sliding window is lost. On restart, the evaluator must reconstruct the window from the time-series database. The reconstruction reads the last 30 days of window entries from the database and replays them through the sliding window.

The reconstruction takes approximately 5 seconds for each 30-day window. For 500 SLOs, the total reconstruction time is about 42 minutes if done sequentially, or 2 minutes if done in parallel with 20 concurrent workers. The evaluator should prioritize critical SLOs during reconstruction.

During reconstruction, the evaluator cannot generate alerts. The system should degrade gracefully by suppressing alerts until the evaluator is fully initialized. The maximum window of vulnerability is the reconstruction time.

### Partition Recovery

In a distributed SLO system, the aggregation layer may be partitioned from the instrumentation layer. When the partition heals, the aggregator receives a burst of delayed metric events. The aggregator must handle these delayed events correctly.

The aggregator assigns each event to the minute it occurred, not the minute it was received. A 30-minute partition means the aggregator receives 30 minutes of backlogged data. The aggregator processes the backlog in chronological order, advancing the window for each minute.

The key insight is that the sliding window is advanced by wall-clock time, not by event time. If the aggregator was down for 30 minutes, the wall clock advanced by 30 minutes, but no window entries were created for those minutes. The evaluator treats missing minutes as having zero requests — they do not affect the compliance calculation because they contribute to neither the numerator nor the denominator.

This is a design choice. Some implementations treat missing minutes as bad minutes to be conservative. Others treat them as neutral. The conservative approach is safer for SLAs but generates more false alerts.

### Concurrent Writes

Multiple service instances may emit metrics for the same SLO simultaneously. The aggregation layer must merge these concurrent writes correctly. The standard approach is to use a convergent data structure — a counter that can be merged regardless of the order of operations.

The per-minute window entry is a convergent data structure. Each entry has two fields: total_requests and bad_requests. Both fields are counters that are merged by summing. If instance A reports 1,000 total and 2 bad, and instance B reports 2,000 total and 5 bad, the merged window is 3,000 total and 7 bad. The order of merging does not matter.

For latency histograms, the merge operation is also convergent. Two histograms are merged by summing the bucket counts. The merged histogram produces the same percentile computation regardless of the order of merging.

### Stale Reads

The time-series database may return stale data if replicas are behind the primary. A stale read means the evaluator sees data that is minutes old. The evaluator must handle stale reads gracefully.

The evaluator maintains a watermark — the timestamp of the most recent data it has seen. If a read returns data with a timestamp older than the watermark, the evaluator discards it as stale. If the evaluator cannot read fresh data for more than 5 minutes, it stops generating alerts and reports a data staleness error.

Stale reads are most dangerous during an incident. The database may be overloaded, causing replicas to fall behind. The evaluator reads stale data, computes a low burn rate, and does not alert. The incident continues undetected. To mitigate this, the evaluator should read from the primary database, not from replicas, for SLO evaluation.

## Performance Characteristics

### p50 Latency: The Normal Case

The p50 latency of SLO evaluation is the latency of the common case — a single minute of data is added to the sliding window, compliance is recomputed, and no alerts are generated. This path takes approximately 1-2 milliseconds for a single SLO.

The bottleneck is the time-series database read. The evaluator must read the latest window entry from the database. For a database that supports point lookups by key, this read takes 1-5 milliseconds. The O(1) sliding window update is in-memory and takes under 100 microseconds.

### p99 Latency: The Alerting Case

The p99 latency of SLO evaluation is dominated by the alerting path. When a burn rate exceeds a threshold, the evaluator must execute the alerting logic: check for existing alerts, apply hysteresis, format the notification, and send it to the alert manager.

The alerting path includes a network round trip to the alert manager. This adds 10-50 milliseconds to the evaluation time. The evaluator should use asynchronous alerting — the alert is queued and the evaluation continues. The alert manager processes the queue asynchronously.

### p999 Latency: The Reconstruction Case

The p999 latency of SLO evaluation is dominated by the reconstruction path. When the evaluator crashes and restarts, it must reconstruct the sliding window from the database. This path reads 43,200 window entries per SLO and replays them through the sliding window.

For 500 SLOs, the reconstruction reads 21.6 million entries. At 1,000 entries per second read throughput, the reconstruction takes 6 hours. This is unacceptable for a production system. The system must parallelize the reconstruction or use a snapshot-based approach.

The snapshot approach periodically writes the full sliding window state to disk. On restart, the evaluator reads the latest snapshot and then replays only the minutes since the snapshot. With hourly snapshots, the reconstruction replays at most 60 minutes of data, taking approximately 2 seconds.

## Memory Management and Optimization

### Memory Per SLO

Each SLO in the evaluation tier maintains a sliding window of 43,200 entries (30 days at 1-minute resolution). Each entry is a data structure containing:

```python
@dataclass
class WindowEntry:
    timestamp: int        # Unix timestamp, 8 bytes
    total_requests: int   # Total request count, 8 bytes
    bad_requests: int     # Bad request count, 8 bytes
    histogram: bytes      # Optional latency histogram, ~100 bytes
```

The total memory per SLO is:

```
Without histogram: 43,200 * (8 + 8 + 8) = 1,036,800 bytes ~= 1 MB
With histogram: 43,200 * (8 + 8 + 8 + 100) = 5,356,800 bytes ~= 5 MB
```

For a system with 500 SLOs, the total memory is 500 MB without histograms or 2.5 GB with histograms. This is feasible for a modern server with 16 GB or more of RAM.

### Memory Optimization Techniques

**Sparse storage:** Not all minutes have traffic. A service that processes 1 million requests per day does not necessarily have traffic every minute. If the service has zero traffic for 20% of minutes, those minutes can be stored as a sparse entry (just the timestamp with zero counts) instead of a full entry. Sparse storage reduces memory by 10-30% depending on the traffic pattern.

**Compressed histograms:** The latency histogram can be compressed using delta encoding. Instead of storing the raw bucket counts, store the differences between consecutive buckets. For a histogram with 12 buckets, delta encoding reduces the storage from 12 integers to 12 integers with smaller values, which compresses better.

**Sliding window sharing:** Multiple SLOs that share the same SLI can share the same sliding window. If a service has both a 99.9% and a 99.99% SLO for the same latency SLI, both SLOs can use the same window. The two SLOs differ only in the target threshold, not in the underlying data. This reduces memory by up to 50% for services with multiple SLOs per SLI.

**Memory-mapped files:** For very large SLO deployments (thousands of SLOs), the sliding windows can be stored in memory-mapped files instead of heap memory. The OS manages the paging of the memory-mapped file, allowing the SLO system to use disk as an extension of RAM. This trades performance for capacity.

### Garbage Collection Impact

In garbage-collected languages (Java, Go, Python), the sliding window creates significant GC pressure. Every minute, 43,200 entries are read and written. If the entries are heap-allocated objects, the GC must scan and compact them regularly.

To reduce GC pressure, use a single pre-allocated array of structs instead of a list of objects. The struct array is allocated once and reused — no new objects are created during normal operation. This eliminates GC pressure from the sliding window.

```python
# GC-friendly sliding window using arrays
class GcFriendlySlidingWindow:
    def __init__(self, window_minutes: int = 43200):
        self.timestamps = array("Q", [0] * window_minutes)  # unsigned long long
        self.totals = array("Q", [0] * window_minutes)
        self.bads = array("Q", [0] * window_minutes)
        self.head = 0
        self.total_sum = 0
        self.bad_sum = 0

    def advance(self, timestamp: int, total: int, bad: int):
        # Subtract evicted entry
        self.total_sum -= self.totals[self.head]
        self.bad_sum -= self.bads[self.head]

        # Store new entry
        self.timestamps[self.head] = timestamp
        self.totals[self.head] = total
        self.bads[self.head] = bad

        # Update sums
        self.total_sum += total
        self.bad_sum += bad

        # Advance head
        self.head = (self.head + 1) % len(self.totals)
```

## Data Serialization and Transport

### Metric Event Format

The metric event is the atomic unit of SLO data. Each event represents a single request outcome. The event format must be compact, self-describing, and efficiently serializable.

```protobuf
message MetricEvent {
    string service = 1;       // Service name, e.g. "user-api"
    string endpoint = 2;      // Endpoint name, e.g. "/users/search"
    double latency_ms = 3;    // Request latency in milliseconds
    int32 status_code = 4;    // HTTP status code
    int64 timestamp_ns = 5;   // Nanosecond precision timestamp
    string instance_id = 6;   // Unique instance identifier
    map<string, string> labels = 7;  // Optional key-value labels
}
```

The Protocol Buffers encoding of this event is approximately 50-80 bytes per event. With gzip compression, this drops to 30-50 bytes. For a service processing 10,000 requests per second, the metric data volume is 0.5-0.8 MB/s before compression.

### Batch Transport

Events are batched for transport. The batch format wraps multiple events in a single message:

```protobuf
message MetricBatch {
    string source_id = 1;            // Instance ID of the sender
    int64 batch_id = 2;              // Monotonically increasing batch ID
    int64 first_timestamp = 3;       // Timestamp of the first event
    int64 last_timestamp = 4;        // Timestamp of the last event
    int32 event_count = 5;           // Number of events in the batch
    bytes compressed_events = 6;     // Gzip-compressed serialized events
}
```

The batch format allows the receiver to quickly determine the batch's time range without deserializing individual events. The batch ID enables deduplication — if the receiver sees batch ID 42 from instance X, it can ignore batch 42 if it was already processed.

### Transport Protocols

Three transport protocols are commonly used for metric events:

**HTTP/2 with gRPC:** The most common transport for modern systems. gRPC supports streaming, bidirectional communication, and efficient binary serialization. The connection is persistent, which eliminates connection setup overhead. Throughput is approximately 100,000 events per second per connection.

**Kafka:** The preferred transport for high-throughput systems. Kafka provides durable storage, replay capability, and consumer group scaling. Throughput is limited by the Kafka cluster capacity, typically 1-10 million events per second per cluster.

**UDP:** The lowest-overhead transport. UDP has no connection setup, no acknowledgment, and no retransmission. Events can be lost without detection. Throughput is limited by the network interface, typically 1-10 million packets per second. UDP is suitable for systems where event loss is acceptable.

### Basic Error Budget Formula

The error budget is defined as:

```
error_budget = 1.0 - slo_target
```

For common SLO targets:

| SLO Target | Error Budget | Error Budget (30 days) |
|------------|-------------|----------------------|
| 99.9% | 0.1% | 43 minutes 12 seconds |
| 99.95% | 0.05% | 21 minutes 36 seconds |
| 99.99% | 0.01% | 4 minutes 19 seconds |
| 99.999% | 0.001% | 25.9 seconds |

The error budget is defined in terms of the measurement window. A 99.9% SLO over a 30-day window has an error budget of 43 minutes and 12 seconds. This means the service can be in a non-compliant state for a total of 43 minutes and 12 seconds over any 30-day rolling window.

### Budget Consumption Tracking

The budget consumption is tracked as a running count of bad time (minutes) or bad requests. The consumption is updated every minute as new data enters the window and old data exits.

```
budget_consumed = bad_minutes_in_window / total_error_budget_minutes
budget_remaining = 1.0 - budget_consumed
```

For a 30-day window with a 99.9% SLO:
- Total minutes: 30 * 24 * 60 = 43,200
- Error budget minutes: 43,200 * 0.001 = 43.2 minutes
- If 10 bad minutes have accumulated: consumption = 10 / 43.2 = 23.1%

### Recovery Time After Violation

When the service violates its SLO, the recovery time is the time required for the bad data to age out of the measurement window. If the violation occurs at day 15 of a 30-day window, the violation persists until day 45 — 30 days after the violation.

```
recovery_time = remaining_window_duration + measurement_window_duration
```

For a violation at day 15 of a 30-day window:
- Remaining window: 15 days
- Full window after violation clears: 30 days
- Total recovery time: 15 + 30 = 45 days from the start of the incident

For a 7-day window, the same violation recovers in 7 + 15 = 22 days. For a 90-day window, the recovery takes 90 + 15 = 105 days.

### Multi-SLO Budget Aggregation

When multiple SLOs are combined into a composite SLO, the combined error budget is not simply the sum of the individual budgets. The combined budget depends on the correlation between the SLOs.

For independent SLOs (no correlation):

```
P(all SLOs met) = P(SLO_1 met) * P(SLO_2 met) * ... * P(SLO_n met)
```

For perfectly correlated SLOs (same root cause):

```
P(all SLOs met) = min(P(SLO_1 met), P(SLO_2 met), ..., P(SLO_n met))
```

In practice, SLOs for the same service are partially correlated. A database slowdown affects both latency and availability SLOs simultaneously. The combined budget should be computed using historical correlation data.

```python
def composite_error_budget(slo_targets: list, correlation_matrix: list) -> float:
    # If all SLOs are independent, the composite is the product
    independent_composite = 1.0
    for target in slo_targets:
        independent_composite *= target

    # If all SLOs are perfectly correlated, the composite is the minimum
    perfectly_correlated = min(slo_targets)

    # The actual composite is somewhere between independent and perfectly correlated
    # Use the average pairwise correlation to interpolate
    avg_correlation = sum(
        correlation_matrix[i][j]
        for i in range(len(correlation_matrix))
        for j in range(i + 1, len(correlation_matrix))
    )
    num_pairs = len(slo_targets) * (len(slo_targets) - 1) / 2
    if num_pairs == 0:
        return independent_composite
    avg_correlation /= num_pairs

    # Interpolate between independent and perfectly correlated
    composite = independent_composite * (1 - avg_correlation) + \
                perfectly_correlated * avg_correlation
    return composite
```

### Budget Recovery After Mitigation

When a problem is fixed, the budget stops being consumed. The budget does not recover immediately — it recovers as the bad data ages out of the measurement window. The recovery rate depends on the window size and the amount of bad data.

```
budget_recovery_per_day = bad_data_amount / measurement_window_days
```

If the service had 30 bad minutes before mitigation and uses a 30-day window:
- Budget recovery per day: 30 / 30 = 1 minute per day
- Full recovery: 30 days after mitigation

If the same service uses a 7-day window:
- Budget recovery per day: 30 / 7 = 4.3 minutes per day
- Full recovery: 7 days after mitigation

Shorter windows recover faster but have less budget to begin with. The trade-off between window size and recovery time is a fundamental design decision in SLO systems.

## Configuration and Tuning

### SLO Target Selection

The SLO target is the most important configuration parameter. It determines the error budget, the burn rate thresholds, and the alerting sensitivity. The target should be set based on user requirements, not on current system performance.

The target must be achievable. If the system currently achieves 99.5% availability, setting a 99.99% target guarantees perpetual violation. The error budget is permanently negative, which means the SLO is useless. Set a target that is 0.1-0.5% above the current performance to force improvement without being impossible.

The target should also be meaningful. A 99.9% SLO for a service that actually achieves 99.999% is too loose. The error budget is never threatened, and the team becomes complacent. The target should be close enough to the actual performance that the team feels the pressure of budget consumption.

### Measurement Window Tuning

The measurement window determines how far back the SLO looks. Longer windows are more stable but slower to detect problems. Shorter windows detect problems faster but are noisier.

The standard window is 30 days. This is long enough to smooth out daily and weekly traffic patterns. It is short enough that a problem lasting several days is detected within a reasonable time. The 30-day window is the default for most SLO systems.

Consider using a 7-day window for services that change rapidly. A rapidly changing service has a shorter track record, and a 30-day window includes a lot of history that may not reflect the current state. A 7-day window resets faster and adapts to changes more quickly.

Use a 90-day window for services with SLAs. The longer window provides more data for compliance reporting and reduces the impact of brief outages on the compliance calculation. The trade-off is that a 90-day window takes longer to recover from a violation.

### Burn Rate Threshold Tuning

The burn rate thresholds determine when alerts fire. The standard thresholds are:

- **Page (critical):** Burn rate 14.4 over 5 minutes. This catches catastrophic failures immediately. The 14.4 threshold means the error budget would be exhausted in 2 days at this rate. The 5-minute window prevents false positives from brief blips.

- **Ticket (medium):** Burn rate 6.0 over 30 minutes. This catches sustained degradation. The 6.0 threshold means the budget would be exhausted in 5 days. The 30-minute window filters out noise from short bursts.

- **Notify (low):** Burn rate 1.0 over 6 hours. This catches slow degradation. The 1.0 threshold means the budget is being consumed at the nominal rate. The 6-hour window ensures the degradation is sustained.

- **Trend (info):** Burn rate 0.5 over 3 days. This catches long-term trends. The 0.5 threshold means the budget is being consumed at half the nominal rate. The 3-day window detects slow shifts that other windows miss.

### Hysteresis Configuration

Hysteresis prevents alerts from firing repeatedly. After an alert fires, the system enters a cooldown period. During the cooldown, the same alert condition does not trigger another notification.

The cooldown period should be proportional to the alert severity. Page-level alerts have a 30-minute cooldown. Ticket-level alerts have a 1-hour cooldown. Notification-level alerts have a 4-hour cooldown. Trend alerts have no cooldown because they are displayed on dashboards, not sent to notification channels.

### Snapshot Frequency

Snapshots accelerate the recovery of the sliding window after a crash. The snapshot frequency determines how much data must be replayed on restart.

Hourly snapshots are a good default. The evaluator writes the full sliding window state to disk every hour. On restart, the evaluator reads the latest snapshot and replays at most 60 minutes of data. The replay takes approximately 2 seconds.

Daily snapshots are sufficient for systems with low crash rates. The replay takes approximately 30 seconds for a daily snapshot. The trade-off is that a crash 23 hours after the last snapshot requires replaying 23 hours of data.

Real-time snapshots (every minute) provide the fastest recovery but add significant write overhead. Each snapshot writes 1 megabyte per SLO to disk. For 500 SLOs, this is 500 megabytes per minute — 720 gigabytes per day. This is too expensive for most systems.

### Alert Aggregation

Multiple alerts for the same root cause should be aggregated into a single notification. If the SLO violation is caused by a single deployment, the system should send one alert that says "SLO violation detected across services A, B, and C due to deployment v2.1.3" rather than three separate alerts.

Alert aggregation uses the concept of an alert group. Alerts that fire within 5 minutes of each other and share the same root cause label are grouped together. The group generates a single notification that lists all affected SLOs and the likely root cause.

### Alert Escalation

If an alert is not acknowledged within the configured time window, the alert manager escalates to the next tier of on-call engineers. The standard escalation policy has three tiers:

- **Tier 1:** Primary on-call engineer. Acknowledgment within 5 minutes. If not acknowledged, escalate to Tier 2.
- **Tier 2:** Secondary on-call engineer. Acknowledgment within 10 minutes. If not acknowledged, escalate to Tier 3.
- **Tier 3:** Engineering manager. Acknowledgment within 15 minutes. If not acknowledged, escalate to the incident commander.

The escalation policy ensures that alerts are always acknowledged by a human. The tiered approach gives each level time to respond before escalation.

## Advanced Tuning Strategies

### Dynamic Burn Rate Thresholds

The standard burn rate thresholds (14.4 over 5 minutes, 6.0 over 30 minutes) are based on a 99.9% SLO. For different SLO targets, the thresholds should be adjusted. The adjustment is based on the idea that the alert should fire when the error budget is at risk of exhaustion within a specific time window.

For a 99.99% SLO, the error budget is 10 times smaller than for a 99.9% SLO. The same absolute error rate produces a burn rate that is 10 times higher. The thresholds should be adjusted to account for this:

```python
def adjust_thresholds(slo_target: float) -> dict:
    # The standard thresholds are calibrated for 99.9% (0.999)
    # For other targets, adjust proportionally
    base_target = 0.999
    scale = (1.0 - slo_target) / (1.0 - base_target)

    return {
        "page_burn_rate": 14.4 * scale,
        "page_window_minutes": 5,
        "ticket_burn_rate": 6.0 * scale,
        "ticket_window_minutes": 30,
        "notify_burn_rate": 1.0 * scale,
        "notify_window_minutes": 360,
        "trend_burn_rate": 0.5 * scale,
        "trend_window_minutes": 4320,
    }
```

For a 99.99% SLO, the scaled thresholds are:
- Page: 144.0 burn rate over 5 minutes
- Ticket: 60.0 burn rate over 30 minutes
- Notify: 10.0 burn rate over 6 hours
- Trend: 5.0 burn rate over 3 days

These thresholds are much higher because the error budget is smaller. The same absolute error rate produces a higher burn rate, so the threshold must be higher to avoid false positives.

### Alert Fatigue Mitigation

Alert fatigue is the leading cause of SLO system failure. When engineers are paged too often, they start ignoring alerts. The SLO system must be tuned to minimize false positives while maintaining sensitivity.

The key mitigation techniques are:

**Multi-window confirmation:** An alert fires only if the burn rate exceeds the threshold in two consecutive windows. This prevents brief blips from triggering alerts. A single 5-minute window with a high burn rate does not trigger an alert. Two consecutive 5-minute windows with high burn rates do.

**Time-of-day sensitivity:** The burn rate threshold is adjusted based on the time of day. During peak hours, the threshold is higher because the system is under more stress. During off-peak hours, the threshold is lower because the system should be more stable. This prevents false alerts during expected peak-hour degradation.

**Day-of-week adjustment:** The burn rate threshold is adjusted based on the day of the week. Weekdays have different traffic patterns than weekends. The threshold should be calibrated for each day's typical traffic pattern. A burn rate that is normal for a weekday may be abnormal for a weekend.

**Holiday calendaring:** The SLO system should know about holidays. Traffic patterns on holidays are different from normal days. The burn rate thresholds should be adjusted for holiday traffic. The holiday calendar is configured in the SLO system and updated annually.

### Performance Budget for SLO Evaluation

The SLO evaluation itself consumes resources. The team should define a performance budget for the SLO evaluation system:

- **CPU budget:** The SLO evaluator should use no more than 5% of the available CPU on the evaluation instance.
- **Memory budget:** The SLO evaluator should use no more than 80% of the available memory on the evaluation instance.
- **Network budget:** The SLO evaluator should generate no more than 10 MB/s of network traffic for alerting and data transfer.
- **Storage budget:** The SLO evaluator should write no more than 1 GB per day of snapshot data.

If the SLO evaluation exceeds any of these budgets, the system should alert the operations team. The SLO system's performance is a service level indicator for the SLO system itself.
