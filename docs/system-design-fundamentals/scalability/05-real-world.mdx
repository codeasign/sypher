# Scalability in the Real World

## Netflix: Auto-Scaling at Global Scale

Netflix operates in 190+ countries and serves 260 million subscribers. Their streaming traffic follows predictable daily and weekly patterns — Friday night is peak, 3 AM is trough.

### The Problem

Netflix could not provision servers for peak traffic. A single Friday night spike would require 10x the capacity of a Tuesday morning. At Netflix's scale, that meant millions of dollars in idle hardware.

### The Solution

Netflix built auto-scaling into every tier using a system called **Scryer**.

### AWS Instance Types and Cost Breakdown

Scryer orchestrates instance types per workload: c5/c6i (compute-optimized) for transcoding, r5/r6i (memory-optimized) for Cassandra and EVCache, i3/i4i (storage-optimized with NVMe) for write-heavy tables, g4dn/g5 (GPU) for ML inference, and Graviton (arm-based) for stateless API gateways. Netflix reported up to 40% better price-performance on Graviton2 for their Zuul gateway tier.

Netflix's savings from Scryer include: 90% of compute running on spot instances (70-90% discount), reserved instances covering 40% of baseline capacity (30-50% savings), region arbitrage shifting load away from expensive spot zones (20-35% variable), and right-sizing (25% fleet-wide). Combined: approximately \$10M saved per year per major region.

### What Scryer Monitors

Scryer ingests metrics every 30 seconds from every tier: traffic metrics (RPS, bytes streamed, concurrent connections), encoding pipeline (queue depth, completion latency), CDN edge (node utilization, cache hit ratio), infrastructure (EC2 CPU, network throughput, EBS burst balance, Elasticache replication lag), and cost signals (spot interruption rate, price delta, reserved utilization). Scryer retrains its prediction model daily, accounting for seasonal spikes (4x on Christmas Day) and content release patterns (3x burst per Netflix Original debut).

**Key lessons:**

- **Predict, do not react.** Reactive auto-scaling (triggered by CPU) is too slow for Netflix's traffic velocity. They predict demand 10 minutes ahead using ML models.
- **Spot instances as a scaling mechanism.** 90% of compute runs on spot at 70-90% discount, blended with reserved and on-demand to maintain capacity.
- **Regional diversity.** When one region has constrained spot capacity, traffic shifts to other AWS regions.

## Twitter (X): Timeline Sharding

Twitter faced one of the hardest scalability problems in the industry: delivering a personalized timeline to each user while ingesting 6,000 tweets per second (peak).

### The Problem

Naively, generating a timeline means: for each user, scan all tweets from followed accounts, merge them by timestamp, and return the top 200. When a user follows 1,000 accounts and the database has 1 trillion tweets, this scan takes minutes.

### The Solution

Twitter uses **fan-out on write** with sharded timeline caches.

### Redis Timeline Cache Sharding in Depth

The timeline cache uses consistent hashing: `hash(user_id) % 256` mapping to 256 Redis masters (each with 2 replicas). Each node stores a sorted set per user (`ZADD timeline:{user_id} score={tweet_id} member={tweet}`) capped at 200 entries via `ZREMRANGEBYRANK`. TTL is 24 hours from last write. Hot users (100+ reads/hour) are detected by a monitoring pipeline and promoted to dedicated shards.

**Cache warming for new users:**

Twitter uses deferred warm-up: the first timeline request hits the fan-out read path (cache miss). A background job then pulls the last 200 tweets from each followed account and populates the sorted set. The second request (10-30 seconds later) hits a warm cache. High-profile users (celebrities, journalists) get priority pre-warming before their first login. This avoids wasting Redis memory on the approximately 30% of signups who follow accounts and never return.

**User deletion and cache invalidation:**

A deletion event is published to Kafka. Invalidation workers consume the topic and modify each follower's sorted set. For users with more than 10,000 followers, workers set a tombstone flag: `SET tombstone:{user_id}:{timestamp} 1 EX 86400`. Timeline rendering skips tweets whose author_id matches an active tombstone — this avoids 600M individual ZREM operations for a celebrity deletion.

General invalidations: unfollow triggers `ZREMRANGEBYSCORE` on a single Redis node (instant). Account suspension sets a short-lived tombstone (TTL = suspension duration + 24 hours). Content moderation iterates over the first 100,000 followers and issues ZREM; the remaining followers see the tweet until the next eviction cycle — a deliberate consistency-vs-volume trade-off.

**Sharding details:**
- Timeline cache sharded by `user_id % 256` across Redis clusters
- Each shard: 200 tweets for 5,000-10,000 users
- Celebrities bypass fan-out writes — pulled from a separate celebrity cache on read

**Key lessons:**

- Fan-out write is elegant until a celebrity posts. Hybrid approaches handle both cases.
- Cache sharding separates data by access patterns. Hot users move to dedicated shards.
- Consistent hashing enables adding shards without full rebalancing.

## Discord: Channel Sharding

Discord stores every message ever sent in every channel, indexed by channel ID. With 150 million monthly users and millions of channels, the message table is one of the largest in the world.

### The Problem

A naive PostgreSQL schema with a `messages` table indexed by `channel_id` works until a channel has 1 million messages. Then queries slow. At 100 million messages, the index does not fit in memory. Queries take seconds.

### The Solution

### PostgreSQL Tuning for Sharded Channels

**Connection pooling:** With 250 PostgreSQL shards, default `max_connections = 100` would exhaust memory. Discord reduced it to 15 per shard and wrapped access behind PgBouncer in transaction-pooling mode, multiplexing 1,000 application connections into 15 database connections. A custom middleware rewrites ORM queries to parameterized SQL instead of prepared statements — this alone cut connection churn by 40%.

**Query optimization:** Discord replaced `OFFSET` pagination with cursor-based pagination using a composite B-tree index on `(channel_id, created_at DESC, id DESC)`:

```sql
SELECT * FROM messages
WHERE channel_id = $1
  AND (created_at, id) < ($2, $3)
ORDER BY created_at DESC, id DESC
LIMIT 50;
```

For the "jump to message" feature, Discord finds the rank via `COUNT(*) WHERE channel_id = $1 AND created_at <= $2` (index-only scan), then fetches 50 messages around that rank. Both phases complete under 5ms even on shards with 100M+ messages.

**Partition sizing:** Each shard stores 50M-200M messages. The primary B-tree index is approximately 12 GB at 200M rows — kept entirely in RAM so index-only scans never touch disk. Write throughput is kept under 5,000 INSERT/sec per shard (beyond this, WAL write-ahead creates I/O contention). Channels exceeding 100M messages get promoted to dedicated shards via logical replication (pglogical), completing in under 5 minutes with zero downtime.

Discord also uses monthly table partitioning within each shard:

```sql
CREATE TABLE messages (
  id BIGSERIAL, channel_id BIGINT NOT NULL, author_id BIGINT NOT NULL,
  content TEXT, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (channel_id, id, created_at)
) PARTITION BY RANGE (created_at);

CREATE TABLE messages_2026_01
  PARTITION OF messages
  FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
```

Partitions stay under 10 GB each. Partitions older than 6 months are archived to S3.

**Key lessons:**

- A natural sharding boundary (channel ID) eliminates cross-shard queries entirely.
- Heavy channels (100M+ messages) get their own shard. Light channels share.
- Shard rebalancing moves entire channels, not individual messages.

## Pinterest: Sharded MySQL to Cassandra Migration

Pinterest grew from 10,000 to 100 million users in four years. Their MySQL sharding architecture could not keep up.

### The Problem

Pinterest initially sharded MySQL by user ID. As the user base grew, the number of shards grew too. Managing 1,000+ MySQL shards with manual rebalancing became operationally unsustainable.

- Adding a shard meant migrating data
- Schema changes required updating 1,000+ databases
- Hot users (celebrities) overwhelmed their assigned shards

### The Migration

Pinterest migrated to Cassandra, which provides automatic sharding (consistent hashing) and no single point of failure.

```
MySQL sharding (old):
  - 1,000+ logical shards across 100 physical hosts
  - Manual shard assignment: "User X goes to Shard Y"
  - Rebalancing required: dump, copy, verify, switch
  - Schema changes: ALTER TABLE on 1,000 databases

Cassandra (new):
  - Consistent hashing: automatic distribution
  - No manual shard management
  - Schema changes: applied once, propagated automatically
  - Linear scalability: add nodes, throughput increases
```

**Key lesson:** Build sharding that is operationally sustainable.

## Amazon: DynamoDB Scaling

Amazon's DynamoDB is a fully managed NoSQL database designed for infinite scale.

### Partition and Throughput

DynamoDB distributes data across partitions. Each partition handles up to 3,000 read capacity units (RCUs) or 1,000 write capacity units (WCUs).

```
Partition calculation: max(size / 10 GB, throughput / 3000 RCU)

  100 GB table → 10 partitions minimum
  50,000 RCU → 17 partitions required
  Final: 17 partitions (throughput dominates)
```

**Hot partition fix:** Use composite partition keys. `customer_id` alone creates a hot partition for customers with 10M orders. `customer_id + order_date_ym` spreads across 24 monthly partitions.

**Key lessons:** Partition design determines scalability ceiling. Bad keys create hot partitions no amount of auto-scaling can fix.

Uber handles over 15 million trips per day across 70+ countries. The **dispatch engine** pairs riders with nearby drivers in under 200 milliseconds.

### The Problem

Every second, the dispatch engine must ingest GPS from 5 million active drivers (each reporting every 4 seconds), match ride requests against nearby drivers geospatially, handle surge pricing without race conditions, and scale through daily spikes (5x during bar-close on weekends).

### Sharding by City

Each city runs an independent dispatch cluster — no cross-city coordination needed. City-specific tuning: Manhattan uses 200m hex cells (high density), suburban New Jersey uses 500m cells. Each cluster has: (1) a Redis geospatial index layer — one sorted set per H3 cell, updated every 4 seconds per GPS ping; (2) a Ringpop consistent-hashing state machine assigning requests to workers; (3) a MySQL database for trip history (not queried on the hot path).

### Redis and Surge Handling

Uber's Redis deployment handles 1.25 million writes/second from 5 million drivers. Pipeline batching groups 50 updates per muxed command across a 400-node cluster. When demand surge is detected (`demand_delta = requests / available drivers > 3.0` in any H3 cell), the engine activates surge pricing, expands the search radius from 7 to 19 cells, and pre-scales the city's dispatch cluster by 2x within 60 seconds.

**Key lessons:** Geospatial sharding with a hierarchical hex grid (H3) is more efficient than lat/lng range queries. Sharding by city eliminates cross-shard coordination. The hot dispatch path never touches the primary database — Redis handles the 200ms SLA.

## YouTube: CDN Strategy

YouTube serves over 2 billion monthly active users and processes 500 hours of video uploaded every minute.

### The Problem

A single 4K video at 60 fps can be 20 GB. Centralized serving from one data center is physically impossible — network links would saturate within seconds.

### Edge Caching and Predictive Pre-Warming

YouTube splits each video into 2-10 second segments stored in multiple bitrate variants (144p through 8K). The edge cache stores only the segments users in its region actually request. When playback starts, the edge proactively fetches the next 30 segments; if the user abandons, unpulled segments are discarded.

YouTube's ML system, **Thermal**, predicts viral content: within 60 seconds of upload it analyzes title, description, thumbnail engagement rate, and uploader history. The first 1,000 views in 10 minutes trigger a prediction of 1M+ views within 24 hours. Predicted viral videos are pre-populated to all Tier 2 regional caches before demand hits, preventing a thundering herd on origin. Geographic prediction pre-warms content in likely regions (e.g., K-pop pre-warmed in South Korea and Japan, not Brazil).

### Upload Pipeline

The pipeline processes 500 hours/minute: the client splits videos into 10 MB chunks for resumable concurrent upload. Each video is transcoded into 12+ bitrate variants on Google's Borg cluster. Frames are extracted for thumbnail generation (blur detection, composition score). Audio/video fingerprints check against the Content ID database (100+ petabyte hash table). Results are pushed to Tier 3 origin (Colossus), then Tier 2 regional caches. Total pipeline: under 2 minutes for 95% of uploads.

**Key lessons:** Segment-based caching maximizes utilization for partially-watched content. Predictive pre-warming prevents thundering herds. CDN tiering reduces origin load by 90%.

## Open Source Tools for Scaling

### Kubernetes Horizontal Pod Autoscaler (HPA)

Kubernetes HPA adjusts pod replicas based on observed metrics.

```
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: dishfind-api
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: dishfind-api
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Pods
        value: 2
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
```

**Key lesson:** HPA scales up fast (100% per 15 seconds) and down slow (5-minute stabilization) — matching real traffic patterns.

### Redis Cluster

Redis Cluster provides automatic sharding across Redis nodes.

```
Requirements:
  - Minimum 3 master nodes (for partition tolerance)
  - Each master needs at least 1 replica (for failover)
  - Total minimum: 6 nodes

Operations:
  - Add node: CLUSTER MEET + CLUSTER ADDSLOTS
  - Remove node: CLUSTER FORGET (after migrating slots)
  - Failover: automatic if majority detects master failure
  - Rebalancing: redis-cli --cluster rebalance

Limits:
  - Maximum 1,000 nodes per cluster
  - Cross-node operations limited (no multi-key commands across slots)
  - No automatic resharding (triggers must be manual)
```

### Kafka Partitioning

Kafka's scaling unit is the partition. More partitions = more parallelism.

```
Scaling guidance:
  Partitions per topic = max(consumer_throughput, producer_throughput)
  Partitions per broker = min(100, disk_throughput / replication_overhead)

Practical limits:
  - Maximum 4,000 partitions per broker (Apache recommendation)
  - Maximum 200,000 partitions per cluster
  - Each partition adds ~2MB of memory overhead on the controller

Throughput scaling:
  1 broker, 3 partitions: ~30,000 msg/sec
  3 brokers, 9 partitions: ~90,000 msg/sec
  10 brokers, 30 partitions: ~300,000 msg/sec
```

## Lessons from Scaling Failures

The most valuable lessons in system design come from failures.

### Twitter: The Fail Whale Era (2007-2012)

Twitter's downtime was legendary. A single MySQL database with Ruby on Rails — every page load hit the database directly, the ORM generated N+1 queries, and synchronous timeline generation tied up worker threads for 30 seconds. When the database crashed (hourly during peak), the entire site went down.

**What was learned:** Introduce caching before the pain is felt. Fan-out on write (with all its complexity) beats synchronous read-time fan-out. Database connection pooling and query timeouts prevent most cascading failures.

### Reddit: The 2008 Outage

Reddit ran on a single PostgreSQL database. No connection pooling — traffic spikes hit the connection limit and rejected all new connections. No read replicas — the home page query scanned the entire posts table, taking 12 seconds. Adding a composite index reduced it to 200ms.

**What was learned:** Read replicas are not optional. Run `EXPLAIN ANALYZE` on every query before production — queries on 100-row dev datasets collapse on 10M-row production datasets. Keep peak database CPU under 30% to absorb traffic spikes.

### GitHub: Database Overload (2018)

A 24-hour degradation where pull requests, issues, and code search were unusable. MySQL replicas fell 30+ seconds behind the master — users created a pull request and saw nothing on refresh. InnoDB deadlocks on the `repository_metadata` row (updated on every star and push) caused cascading lock contention. No circuit breakers: increased lag caused more queued writes, creating a positive feedback loop.

**What was learned:** Strong consistency on hot rows is a scalability antipattern — denormalize star counts into Redis with async updates. Circuit breakers prevent cascading failures (route reads to master when lag exceeds 5 seconds). Never run database migrations during peak hours.

| Detection | Immediate (site down) | Immediate (site down) | 6+ hours (degradation) |

## Seven Lessons from Production

**Lesson 1: Start simple, scale surgically.** Pinterest started with single MySQL. Twitter started with a single Rails app. Netflix shipped DVDs. Postpone scalability as long as possible, then apply it only where needed.

**Lesson 2: Measure before you scale.** Every company here had extensive monitoring before they scaled. Without measurements, you are guessing.

**Lesson 3: Natural sharding beats artificial sharding.** Discord's channels, Twitter's user IDs, and Pinterest's pins provide natural boundaries. Hash modulo N creates rebalancing pain.

**Lesson 4: Auto-scaling needs prediction, not reaction.** Netflix predicts 10 minutes ahead. HPA reacts in seconds. Blend both: predict for capacity, react for spikes.

**Lesson 5: Cost-aware scaling matters.** Netflix runs 90% on spot. Twitter caches only 200 tweets per user. Pinterest migrated because MySQL shard management cost more than a new database.

**Lesson 6: Monitoring is a first-class system component, not an afterthought.** Every failure above was preceded by a lack of actionable monitoring. Twitter had no cache hit ratio visibility. Reddit had no query latency dashboard. GitHub had alert thresholds set too high. Define SLOs, set alert thresholds, run load tests, practice incident response. If you cannot see it, you cannot fix it.

**Lesson 7: Plan for failure, not just success.** Netflix runs Chaos Monkey. Twitter's Fail Whale taught them to handle 10x traffic. GitHub learned to deny rather than degrade. Circuit breakers, bulkheads, graceful degradation, and load shedding are what separates systems that survive from systems that collapse.

The patterns from these companies — predictive auto-scaling, hybrid fan-out, natural shard boundaries, cost-aware planning, deep monitoring, and failure-aware architecture — form the playbook for scaling from prototype to global service.
