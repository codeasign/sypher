# Scalability Concepts

## The Startup That Couldn't Scale

Elara launched her food-discovery app, DishFind, to 100 beta users. A single $20/month VPS running PostgreSQL handled everything. Queries returned in 5 milliseconds. The app felt instant.

Word spread. A local food blog featured DishFind. Signups jumped to 5,000 users. The VPS started gasping — queries took 200ms. Elara upgraded to a $80/month server with 4 vCPUs and 8 GB RAM. Problem solved for the moment.

Then a national TV segment aired during prime time. Signups exploded to 50,000 users overnight. The database connection pool (max 100 connections) saturated at 300 concurrent users. New connections queued. Pages took 30 seconds to load. Users saw "500 Internal Server Error." Within 48 hours, 40% of new users churned. The app was mentioned on Twitter as "that slow food thing."

Elara scrambled. She doubled the database connection limit to 200 and added a second app server behind a simple round-robin load balancer. The system stabilized at 3-second response times. DishFind survived, but barely.

Three months later, DishFind hit 200,000 users. The database primary was at 95% CPU. Read replicas had been added, but writes were still bottlenecked on a single server. Queries against the `reviews` table — now 12 million rows — took 4 seconds even with indexes. Elara added Redis caching. Review pages loaded in 200ms again. Another temporary fix.

By 1 million users, the problems compound. The `user_id` modulo 4 sharding scheme left one shard holding 40% of the data because power users from San Francisco signed up in a wave. The Kafka queue for email notifications backed up to 2 hours of delay. Auto-scaling spun up instances, but the database connection storm when instances restarted caused cascading failures. The CDN was serving stale images for hours because cache invalidation headers were set wrong.

Elara's original mistake: she built for 100 users, not 50,000, and she never planned for the cascade of bottlenecks that appear at each order-of-magnitude growth milestone.

This is the scalability problem. Every successful system faces it. The question is not *if* you need to scale, but *when* and *how*. The pattern is always the same: growth exposes the next weakest link. You fix it. Something else breaks. You fix that. The skill is knowing which link will break next, and preemptively reinforcing it before the traffic spike arrives.

## What Is Scalability?

Scalability is the ability of a system to handle increased load — more users, more requests, more data — without degrading performance. A scalable system maintains consistent response times and throughput as load grows by an order of magnitude, then another, then another.

### The Food Truck Analogy

A single server is a food truck. It serves 20 customers per hour with one cook. When 200 customers show up, the line wraps around the block. You have two options:

- **Vertical scaling**: Buy a bigger food truck with two grills and a faster cook. Still one truck, but serves 50 per hour. Simpler, but you eventually hit the ceiling of what one truck can do.
- **Horizontal scaling**: Open 10 food trucks across town. Each serves 20 per hour. Now you serve 200 per hour. But now you need signage so customers find the right truck, a coordinator to restock ingredients, and a way to handle a burned-out truck without losing all your business.

A food truck chain that operates in 50 cities with central kitchens, regional distribution, and dynamic staffing based on lunch-rush demand — that is a globally scalable system. Each city is a region. The central kitchen is a data warehouse. The dynamic staffing is auto-scaling. And the signage directing customers to the nearest open truck is a global load balancer with health checks.

These concepts form the vocabulary you need to describe any scalable system. What follows are the essential terms, each one a tool you will reach for when a bottleneck appears.

## Core Vocabulary

**Vertical scaling (scale up)**: Adding more power (CPU, RAM, disk) to a single machine. Limits at hardware maximums. AWS offers instances from t2.nano (1 vCPU, 0.5 GB) up to u-24tb1.metal (448 vCPUs, 24 TB RAM) at over $100,000/month. At some point, you run out of bigger machine.

**Horizontal scaling (scale out)**: Adding more machines to distribute load. Theoretically limitless, but adds coordination complexity. You trade simpler operations for near-unbounded capacity.

**Load balancer**: A traffic cop that distributes incoming requests across multiple servers. Prevents any single server from being overwhelmed. Common algorithms: round robin, least connections, IP hash, weighted distribution. Tools include HAProxy, Nginx, AWS ALB/NLB, and Google Cloud Load Balancing.

**Auto-scaling**: Automatically adding or removing servers based on metrics like CPU usage, memory, or request queue depth. AWS Auto Scaling Groups, GCP Instance Groups, and Kubernetes Horizontal Pod Autoscaler all implement this pattern with configurable cooldown periods to prevent thrashing.

**Sharding**: Splitting a database across multiple machines by some key (user ID, geographic region). Each shard holds a subset of data. The shard key choice is critical — a poor choice creates hot spots (one shard handling 80% of traffic). Consistent hashing mitigates this by distributing keys more evenly and minimizing rebalancing when nodes change.

**Replication**: Copying data from one database server to others. Read replicas handle query traffic; the primary handles writes. PostgreSQL streaming replication, MySQL Group Replication, and MongoDB replica sets all follow this pattern. Replication lag — the delay between a write on the primary and its appearance on a replica — is a common source of subtle bugs.

**Partition**: A division of data within a system. Can be horizontal (sharding by key range) or vertical (splitting columns across tables). Range partitioning splits by value ranges (users A-M on shard 1, N-Z on shard 2). List partitioning groups by discrete values (US users on shard 1, EU users on shard 2). Hash partitioning applies a hash function for even distribution.

**Throughput**: The number of requests a system can process per unit time, measured in requests per second (RPS) or queries per second (QPS). Throughput is the system's capacity. If your database handles 1,000 QPS and you need 2,000, you scale.

**Connection pool**: A cache of database connections maintained by the application, avoiding the cost of opening a new TCP connection for every request. Python's psycopg2 `ThreadedConnectionPool`, Java's HikariCP, and Go's `database/sql` package all implement pooling. Pool size tuning is a science: too small and requests queue, too large and the database runs out of memory for connection overhead.

**Connection limit**: The maximum number of simultaneous connections a database server accepts. PostgreSQL defaults to 100. MySQL default is 151. Exceeding this causes "too many connections" errors. Connection pooling keeps your active connections under this ceiling while serving many more concurrent requests.

**Database bottleneck**: The situation where the database is the limiting factor in system throughput. The most common bottleneck in web applications. Symptoms: high CPU on the database server, slow queries in the slow-query log, connection saturation, and application timeouts.

**Stateless**: A server that does not store session data locally. Any request can go to any server. Enables horizontal scaling by simply adding more servers. Session state lives in a shared store like Redis or a database.

**Stateful**: A server that stores session data locally. Requests from the same user must route to the same server (sticky sessions). Harder to scale horizontally because adding or removing servers requires rebalancing sessions. WebSocket servers, game servers, and some legacy applications are stateful by nature.

**CDN (Content Delivery Network)**: A globally distributed network of proxy servers that cache static content (images, CSS, JS) close to users. Shifts load off your origin servers and reduces latency. CloudFront, Cloudflare, Fastly, and Akamai are major providers. CDNs also provide DDoS absorption and SSL termination at the edge.

**Backpressure**: A mechanism where a system signals its upstream consumers to slow down when it cannot keep up with the load. Like a busy restaurant asking the host to stop seating new tables. Implemented via HTTP 429 (Too Many Requests), TCP backpressure in stream processing (Kafka, Akka Streams), or explicit flow control in gRPC. Without backpressure, overwhelmed services crash under cascading failures.

**Connection pooling**: The technique of reusing database connections rather than opening and closing them per request. A typical web app opens a connection at startup, draws from the pool during requests, and returns the connection after the request completes. Python's SQLAlchemy defaults to a pool size of 5, max overflow of 10:

```python
from sqlalchemy import create_engine

engine = create_engine(
    "postgresql://user:pass@host/db",
    pool_size=10,        # Keep 10 connections ready
    max_overflow=20,     # Allow up to 20 additional connections under load
    pool_timeout=30,     # Wait 30 seconds before giving up on a connection
    pool_pre_ping=True,  # Test connections before using them
)
```

Set pool size too low and requests queue. Set it too high and the database spends all its resources managing connections instead of running queries. A good starting point: `2 * number_of_CPU_cores + effective_spindle_count` for PostgreSQL.

**Denormalization**: The practice of adding redundant data to a database to improve read performance. In a normalized schema, you join across tables. Denormalization stores pre-joined data in a single table. An order summary might store the customer name and shipping address directly in the `orders` table, duplicating what lives in `customers`, to avoid a join on every order page load:

```sql
-- Normalized: requires JOIN
SELECT o.id, c.name, c.address
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.id = 12345;

-- Denormalized: single table read
SELECT id, customer_name, shipping_address
FROM orders
WHERE id = 12345;
```

Denormalization trades write complexity (now you must update `orders` when a customer changes their address) for read speed. Use it when read-to-write ratios exceed 20:1 on a critical query path.

**Eventual consistency**: A consistency model where updates propagate to all replicas eventually, but reads may return stale data for a bounded window. DNS, Amazon DynamoDB, and Cassandra use this model. The window of inconsistency is typically measured in milliseconds to seconds. Compare with strong consistency, where every read returns the latest write. The trade-off: eventual consistency allows higher availability and lower latency, but your application code must tolerate staleness, duplicate data, or conflicting updates.

**Fault tolerance**: The ability of a system to continue operating correctly after a component failure. A fault-tolerant system has no single point of failure. Every component has a redundant counterpart. If a server crashes, a load balancer routes around it. If an AZ goes down, traffic shifts to another region. Fault tolerance is achieved through redundancy, health checks, graceful degradation, and circuit breakers.

**Latency**: The time between sending a request and receiving a response, measured in milliseconds. Also called response time or P95/P99 latency (the worst 5% or 1% of requests). Users notice latency above 100ms. Applications must distinguish between mean latency (misleadingly low because fast requests dominate) and tail latency (the slow requests that cause user frustration). A system with mean latency of 50ms but P99 latency of 5 seconds has a serious tail-latency problem.

**Throughput vs Bandwidth**: Throughput is the actual rate of successful data transfer. Bandwidth is the theoretical maximum rate. Bandwidth is the width of the pipe; throughput is how much water actually flows through it. Protocol overhead, packet loss, and congestion reduce throughput below bandwidth. A 1 Gbps network link might deliver only 600 Mbps throughput for a chatty application due to TCP handshake overhead per request.

**Timeouts**: A mechanism that limits how long a system waits for an operation to complete. Timeouts prevent a single slow dependency from holding an entire request thread. Every network call needs a timeout — connect timeout, read timeout, and total request timeout. A common mistake is setting timeouts too long (30+ seconds), which causes thread pool exhaustion under load. Set connect timeouts at 500-1000ms and read timeouts at 2-5 seconds for internal services. For external APIs, set 5-10 seconds and implement retry with exponential backoff.

```python
import requests

# Timeouts in requests library — always set them
response = requests.get(
    "https://api.example.com/data",
    timeout=(3.0, 5.0),  # (connect timeout, read timeout)
)
```

**Circuit breaker**: A pattern that prevents a system from repeatedly trying an operation that is likely to fail. Modeled after electrical circuit breakers, it has three states:

- **CLOSED**: Normal operation. Requests pass through. Failures are counted.
- **OPEN**: Failure threshold exceeded. Requests fail immediately without attempting the operation. After a timeout period, transitions to HALF-OPEN.
- **HALF-OPEN**: A limited number of test requests are allowed through. If they succeed, the circuit closes. If they fail, it opens again.

```python
class CircuitBreaker:
    def __init__(self, threshold=5, recovery_timeout=30):
        self.threshold = threshold
        self.recovery_timeout = recovery_timeout
        self.failure_count = 0
        self.state = "CLOSED"
        self.last_failure_time = None

    def call(self, fn, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "HALF_OPEN"
            else:
                raise CircuitBreakerOpenError("Circuit is open")

        try:
            result = fn(*args, **kwargs)
            self.failure_count = 0
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"
            return result
        except Exception:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.threshold:
                self.state = "OPEN"
            raise
```

Python libraries like `pybreaker` and `circuitbreaker` provide production-ready implementations. Hystrix (Java) and `circuit-breaker-js` (Node.js) are popular alternatives.

**Bulkhead**: An isolation pattern named after the compartments on a ship. If one compartment floods, the ship stays afloat. In software, a bulkhead partitions resources — thread pools, connections, memory — so that a failure in one partition cannot consume resources from another. For example, allocate separate connection pools for user-facing queries and analytics queries. If analytics runs a slow query that exhausts its pool, the user-facing pool remains unaffected:

```python
import sqlalchemy

# Separate pools for different workloads
user_engine = create_engine(DB_URL, pool_size=10, max_overflow=5)
analytics_engine = create_engine(DB_URL, pool_size=3, max_overflow=2)

def get_user(user_id):
    with user_engine.connect() as conn:  # Uses its own pool
        return conn.execute("SELECT * FROM users WHERE id = ?", user_id)

def run_report(report_id):
    with analytics_engine.connect() as conn:  # Separate pool — failures isolated
        return conn.execute("SELECT ... heavy query ...")
```

**Leader election**: A process by which distributed nodes select one node as the leader to coordinate work, manage writes, or assign tasks. The leader handles all writes to a replicated log (Raft) or manages shard assignments (Kubernetes). If the leader fails, the remaining nodes detect the failure (via heartbeat timeout) and elect a new leader. ZooKeeper, etcd, and Consul implement leader election for distributed systems. The Raft consensus algorithm is the most widely used protocol.

**Gossip protocol**: A decentralized communication protocol where each node periodically exchanges state information with a random set of peers. Information spreads through the cluster like a rumor — exponential propagation. Cassandra, Consul, and Redis Sentinel use gossip for membership detection and failure discovery. A node that is unreported for too long (configurable, typically 30-60 seconds) is marked as failed. Gossip scales well because each node talks to only a few peers per cycle regardless of cluster size.

**CAP theorem**: A theorem stating that a distributed data store can provide at most two of three guarantees simultaneously: Consistency (every read returns the most recent write), Availability (every request receives a response, even if it is stale), and Partition Tolerance (the system continues operating despite network partitions). Network partitions are inevitable in distributed systems, so you must choose between CP (sacrifice availability) and AP (sacrifice consistency). CA systems effectively do not exist because partitions always happen.

**PACELC**: An extension of CAP that adds a dimension for normal operation (no partition). PACELC states: if a partition occurs (P), trade off Availability vs Consistency (A vs C). Else (E, when no partition), trade off Latency vs Consistency (L vs C). This explains why DynamoDB and Cassandra choose availability + low latency over strong consistency: during normal operation, they prefer lower latency, which requires relaxing consistency. PostgreSQL with synchronous replication chooses consistency over latency.

## Step by Step: Scaling Through Growth Stages

Every system follows a similar arc as it grows. At each stage, a new bottleneck emerges and forces the next architectural change. Walking through these stages reveals which lever to pull when.

### Stage 1: The Monolith (0 to 1,000 users)

A single application server and a single database. Everything lives on one machine or two. This is correct for this scale. Adding complexity now would waste time and money.

At this stage, you can use SQLite or a small PostgreSQL instance. Deployments are manual — copy files, restart the server. Backups are a cron job running `pg_dump`. Response times are under 50ms. Everything fits in memory. If you process a request synchronously, nobody notices the 200ms email-send delay during the response.

**What breaks next**: As users grow past 1,000, the database and app server compete for CPU and memory. The app process and PostgreSQL both want RAM. One `SELECT *` against a table with 10,000 rows forces PostgreSQL to spill to disk, and the app server GC-pauses at the same moment. Response time jumps from 50ms to 2 seconds.

### Stage 2: Separate Database (1,000 to 10,000 users)

The database moves to its own server. The app server can now use all its resources for request processing. The database server gets dedicated disk I/O.

```
Architecture:
  App Server (CPU-optimized) ──> Database Server (IO-optimized)

Impact:
  - App server handles 3x more requests (no DB process competing for CPU)
  - Database dedicated disk means 50% faster queries
  - Cost: 2 machines instead of 1
```

The two-server architecture is the first step toward distributed systems. You now have a network hop between app and database. This means you must handle connection errors, DNS resolution, and TCP timeouts — concerns that did not exist when everything ran on localhost. At this scale, a single well-indexed database handles both reads and writes comfortably.

**What breaks next**: As active users approach 10,000, every page load hits the database. With a 10:1 read-to-write ratio, that single database saturates at around 1,000 queries per second. The `EXPLAIN ANALYZE` output starts showing sequential scans on tables that used to fit in memory. Read replicas become necessary.

### Stage 3: Read Replicas (10,000 to 100,000 users)

The database read load exceeds what one server can handle. Most applications have a 10:1 read-to-write ratio. Add read replicas.

Configuring read/write splitting in your application:

```python
import sqlalchemy
from sqlalchemy.orm import Session

# Primary for writes
write_engine = create_engine(WRITE_DATABASE_URL, pool_size=10)
# Replicas for reads
read_engine = create_engine(READ_DATABASE_URL, pool_size=20)

class ReadWriteSession:
    def __init__(self):
        self.read = Session(read_engine)
        self.write = Session(write_engine)

    def commit(self):
        self.write.commit()

# Usage — route intelligently
def get_reviews(restaurant_id):
    # Read from replica — stale by milliseconds is acceptable
    return session.read.execute(
        "SELECT * FROM reviews WHERE restaurant_id = ?", restaurant_id
    )

def submit_review(user_id, restaurant_id, rating):
    # Write to primary — must be authoritative
    session.write.execute(
        "INSERT INTO reviews (user_id, restaurant_id, rating) VALUES (?, ?, ?)",
        user_id, restaurant_id, rating,
    )
    session.commit()
```

**What breaks next**: A single blog post featuring DishFind drives 50,000 visitors. Most visit the same restaurant page — the one with the feature image, 200 reviews, and restaurant details. The database is queried for every visit, even though the content has not changed. Response time for that page hits 1 second under the spike. Caching would eliminate 95% of those database calls.

### Stage 4: Caching (50,000 to 500,000 users)

A cache layer (Redis or Memcached) sits between app servers and the database. Frequently accessed data — user sessions, product listings, feed content, restaurant menus — is served from memory in under 1 millisecond instead of querying the database in 10-50 milliseconds.

```
Cache hit rate target: 90%+
Memory sizing: Working set x 1.5
Eviction policy: LRU (Least Recently Used)
```

Typical cache saves 80-95% of database queries for read-heavy workloads. The cache-aside pattern is the most common strategy:

```python
import redis
import json

cache = redis.Redis(host="redis-cluster", port=6379, decode_responses=True)

def get_restaurant_menu(restaurant_id):
    cache_key = f"restaurant:menu:{restaurant_id}"

    # Try cache first
    cached = cache.get(cache_key)
    if cached:
        return json.loads(cached)

    # Cache miss — query database
    menu = db.execute(
        "SELECT * FROM menu_items WHERE restaurant_id = ?", restaurant_id
    ).fetchall()

    # Store in cache with TTL
    cache.setex(cache_key, 300, json.dumps(menu))  # 5 minute TTL
    return menu
```

Cache invalidation is the hard part. Setting TTLs (time-to-live) is the simplest approach — data becomes stale predictably. Write-through caches update on every write but add latency. Write-behind caches batch updates for higher throughput but risk data loss if the cache fails.

**What breaks next**: At 500,000 users, the database itself grows too large. The `reviews` table hits 50 million rows. Even with replicas and caching, the primary handles all writes and the write throughput saturates at 500 writes/second. You need to split the data across multiple database servers.

### Stage 5: Content Delivery Network (100,000 to 500,000 users)

Between caching and sharding, there is another lever: offload traffic to a CDN. Static assets — images of restaurant dishes, CSS, JavaScript bundles — should never hit your application servers. A CDN serves these from edge locations close to the user.

CDN configuration is straightforward — set Cache-Control headers on your origin responses:

```
# Static assets — cache aggressively
Cache-Control: public, max-age=31536000, immutable

# API responses — short cache
Cache-Control: public, max-age=60, stale-while-revalidate=300
```

CloudFront, Cloudflare, Fastly, and Akamai all implement the same core contract: obey your cache headers, serve from the edge when fresh, fetch from origin when stale.

### Stage 6: Sharding (500,000 to 5,000,000 users)

The database itself grows too large for a single server. Even with replicas, the primary handles all writes and can become a bottleneck. Sharding splits data across multiple database servers by a shard key.

  `}

Choosing a shard key is the most important decision in sharding. A poor shard key creates hot spots. Elara's modulo-4 sharding by `user_id` worked until 40% of power users ended up in shard 1. Consistent hashing solves this by mapping keys to a ring and assigning each node responsibility for a range of the ring:

```python
import hashlib
import bisect

class ConsistentHashRing:
    def __init__(self, nodes, replicas=3):
        self.replicas = replicas
        self.ring = {}
        self.sorted_keys = []
        for node in nodes:
            for i in range(replicas):
                key = self._hash(f"{node}:{i}")
                self.ring[key] = node
                bisect.insort(self.sorted_keys, key)

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def get_node(self, key):
        if not self.ring:
            return None
        hash_value = self._hash(key)
        idx = bisect.bisect(self.sorted_keys, hash_value) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]
```

When a shard is added or removed, consistent hashing reassigns only `N / total_nodes` fraction of keys, not all of them. This makes rebalancing practical at scale.

**What breaks next**: Synchronous processing of every request becomes impossible. Image resizing for user-uploaded dish photos takes 500ms per image. Email notifications for new reviews must be sent. Analytics aggregations run for minutes. All of these block the request-response cycle. A queue decouples them.

### Stage 7: Queue for Async Work (1,000,000+ users)

Synchronous processing of every request becomes impossible. Background jobs — email notifications, image processing, analytics — move to a queue.

```
Synchronous: Request → Process image → Return → 5 second latency
Asynchronous: Request → Enqueue → Return immediately → Worker processes in background
```

Kafka or RabbitMQ handle millions of messages per second. Workers scale independently of the web tier. If image processing takes 500ms per image and you have 50 images/second, one worker handles it. At 500 images/second, add nine more workers. The queue absorbs the backlog if workers fall behind.

```python
import json
from kafka import KafkaProducer

producer = KafkaProducer(
    bootstrap_servers=["kafka-1:9092", "kafka-2:9092"],
    value_serializer=lambda v: json.dumps(v).encode(),
    acks="all",           # Wait for all replicas to acknowledge
    retries=3,            # Retry transient failures
)

def submit_review(user_id, restaurant_id, rating, photo):
    # Return immediately — don't process the photo here
    producer.send(
        "review-processing",
        {
            "user_id": user_id,
            "restaurant_id": restaurant_id,
            "rating": rating,
            "photo_s3_key": upload_to_s3(photo),
        },
    )
    return {"status": "accepted", "message": "Review submitted"}
```

Workers consume from the queue, process each message, and write results back to the database:

```python
from kafka import KafkaConsumer

consumer = KafkaConsumer(
    "review-processing",
    bootstrap_servers=["kafka-1:9092"],
    group_id="review-workers",
    auto_offset_reset="earliest",
    enable_auto_commit=False,  # Manual commit for exactly-once processing
)

for message in consumer:
    data = json.loads(message.value)
    resized = resize_image(data["photo_s3_key"], width=800)
    store_resized(data["review_id"], resized)
    send_notification(data["user_id"], "Your review is live!")
    consumer.commit()  # Acknowledge only after successful processing
```

**What breaks next**: Traffic follows the sun. DishFind users in Tokyo peak while San Francisco sleeps, and vice versa. Running servers in a single region means either over-provisioning for peak global traffic or accepting high latency for users far from the server. Auto-scaling adjusts capacity within a region but cannot move capacity between continents.

### Stage 9: Multi-Region Deployment (5,000,000+ users)

Key decisions for multi-region:
- **Active-Passive**: One region serves traffic. The other stands by, replicating data asynchronously. Failover takes minutes but is simpler to implement.
- **Active-Active**: Both regions serve traffic simultaneously. Users in Europe hit EU-West. Users in the US hit US-East. This requires handling cross-region data conflicts (last-write-wins, CRDTs, or custom merge logic).

Cross-region replication introduces the central trade-off of distributed systems: latency vs consistency. Asynchronous replication keeps writes fast (single-digit milliseconds) but risks data loss if the primary region fails before replication completes. Synchronous replication across regions adds 50-100ms of write latency but guarantees no data loss.

### Stage 10: Global Load Balancing and Traffic Shaping (10,000,000+ users)

At tens of millions of users, traffic patterns diverge by region, device type, and time of day. A global traffic manager routes requests based on latency, cost, and capacity:

```typescript
// Simplified global routing policy
type Region = "us-east" | "eu-west" | "ap-southeast";

interface RouteDecision {
  region: Region;
  reason: "latency" | "cost" | "capacity" | "failover";
}

function routeRequest(userIp: string): RouteDecision {
  const geo = geoIpLookup(userIp);

  // Route to nearest region by default
  const nearest = nearestRegion(geo);

  // Override: shift traffic away from degraded regions
  if (!isHealthy(nearest)) {
    return { region: nextBestRegion(nearest), reason: "failover" };
  }

  // Override: route premium users to best-performing region
  if (isPremiumUser(userIp)) {
    return { region: lowestLatencyRegion(userIp), reason: "latency" };
  }

  // Override: gradual rollout of new deployments
  if (isInCanary(userIp)) {
    return { region: "eu-west", reason: "capacity" };
  }

  return { region: nearest, reason: "latency" };
}
```

Global traffic shaping also includes gradual rollout (canary deployments), A/B testing across regions, and cost-based routing (route non-critical traffic to cheaper regions).

### Stage 11: Observability and Resilience Testing (ongoing)

No scaling strategy survives first contact with production without measurement. Observability — metrics, logs, and traces — tells you which bottleneck will break next. Resilience testing — chaos engineering — validates that your scaling works when it matters.

**Four Golden Signals** for scalability:
1. **Latency**: Time to serve a request. Track P50, P95, P99 separately. A P99 increase from 200ms to 2 seconds means something is breaking for your slowest users.
2. **Traffic**: Requests per second. Sudden increases precede bottlenecks.
3. **Errors**: Rate of failed requests. Rising errors under increasing traffic indicate a capacity ceiling.
4. **Saturation**: How close each resource is to its limit. CPU at 95% means the next spike triggers auto-scaling — or a crash if auto-scaling cannot keep up.

Netflix's Chaos Monkey terminates random production instances during business hours to ensure engineers build systems that survive failures. You need not go that far, but simulating a database replica failure, a cache node outage, or a traffic spike in a staging environment reveals weaknesses before they hit production.

## Variants of Scaling

Beyond the step-by-step growth path, three primary scaling strategies serve different constraints. Each involves specific trade-offs along the dimensions of cost, complexity, and capacity.

### Vertical Scaling (Scale Up)

Adding resources to a single machine: more CPU cores, more RAM, faster SSD, larger instance type.

**Cloud provider specifics:**

- **AWS**: Instance families range from `t4g.nano` (2 vCPUs, 0.5 GB, $4.20/month) to `u-24tb1.metal` (448 vCPUs, 24 TB RAM, ~$100,000/month). The `r7g` family (memory-optimized, Graviton) offers up to 64 vCPUs and 512 GB RAM at approximately $4.50/hour on-demand. The `c7g` family (compute-optimized) provides up to 64 vCPUs at approximately $3.50/hour.
- **GCP**: Machine families include `n2` (standard), `c2` (compute-optimized), and `m2` (memory-optimized). `c2-standard-60` provides 60 vCPUs and 240 GB RAM at approximately $3.20/hour. The `m2-megamem-416` offers up to 416 vCPUs and 12 TB RAM at approximately $80/hour.

Vertical scaling has hard limits. AWS's largest general-purpose instance (`m7g.metal-48xl`) provides 192 vCPUs and 768 GB RAM. Beyond that, you cannot scale up. The cost curve is super-linear: a 16x machine (from `t3.nano` to `r7g.16xlarge`) costs roughly 100x more, not 16x more.

**When to use:**
- Traffic is predictable and stays under single-machine limits
- The application is stateful and hard to distribute
- Database writes (harder to scale horizontally than reads)
- Compliance requires data to stay on one machine

**When not to use:**
- There is a hardware ceiling (largest instance type)
- Downtime during upgrades requires a maintenance window
- Cost grows super-linearly (a 2x machine costs more than 2x)
- Single point of failure risk is unacceptable

Adding more machines to distribute the load.

**Cloud provider specifics:**

- **AWS Auto Scaling Groups**: Define a launch template (AMI, instance type), min/max/desired capacity, and scaling policies. Combined with an Application Load Balancer, ASGs handle instance registration and health-check draining automatically. A typical web tier uses `t3a.medium` ($0.0376/hour) instances behind an ALB ($0.0225/hour + $0.008 per GB processed).
- **GCP Managed Instance Groups**: Similar to ASGs with additional features like predictive autoscaling (uses ML to forecast traffic 15 minutes ahead). An `e2-standard-2` ($0.067/hour) in a MIG behind GCP's HTTP Load Balancer ($0.025/hour + $0.008 per million requests) forms a horizontally scalable tier.
- **Kubernetes Horizontal Pod Autoscaler**: Configures target CPU or memory utilization, and HPA adjusts replica counts accordingly. HPA works with custom metrics and external metrics (e.g., queue depth in RabbitMQ):

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: dishfind-web
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Pods
    pods:
      metric:
        name: requests_per_second
      target:
        type: AverageValue
        averageValue: 500
```

**Real-world cost comparison:** Consider 10,000 concurrent users with a stateless Python web application:

| Approach | Instance type | Nodes | Monthly cost | Complexity |
|---|---|---|---|---|
| Vertical | `c5.4xlarge` | 1 | $680 | Low |
| Horizontal | `t3a.medium` | 10 | $270 + ALB $100 = $370 | Medium |
| Horizontal (spot) | `t3a.medium` Spot | 10 | ~$80 + ALB = $180 | Medium+ |
| K8s (spot) | `t3a.medium` Spot | 10-20 autoscaled | ~$100 + overhead | High |

Horizontal scaling with spot instances costs about 75% less than vertical scaling for the same capacity, at the cost of operational complexity and the risk of spot-instance termination.

**When to use:**
- Traffic grows unpredictably or has spikes
- You need high availability (no single point of failure)
- You want to use commodity hardware
- The application is stateless

**When not to use:**
- The application cannot run multiple instances (stateful, session-dependent)
- Database cannot be sharded or replicated
- Coordination overhead outweighs benefits (very small scale)
- Regulatory constraints limit data distribution

Dynamic horizontal scaling based on real-time metrics.

**Predictive auto-scaling** uses historical patterns to provision capacity before a spike arrives. AWS Predictive Scaling analyzes the last 14 days of traffic and generates a forecast. GCP Predictive Autoscaler uses a lookahead of 15 minutes. Both claim to reduce scale-out lag from 8-10 minutes to near zero for predictable patterns like lunch rushes.

**Scheduled auto-scaling** is simpler and sufficient when traffic follows a known schedule:

```bash
# AWS CLI: Schedule 20 instances for lunch rush
aws autoscaling put-scheduled-update-group-action \
  --auto-scaling-group-name dishfind-web \
  --scheduled-action-name lunch-rush \
  --recurrence "0 11 * * *" \
  --desired-capacity 20

aws autoscaling put-scheduled-update-group-action \
  --auto-scaling-group-name dishfind-web \
  --scheduled-action-name lunch-rush-end \
  --recurrence "0 14 * * *" \
  --desired-capacity 5
```

**When to use:**
- Traffic has unpredictable spikes
- Operating cost optimization (pay only for what you use)
- Variable load patterns (diurnal cycles, promotional events)

**When not to use:**
- Applications that take 10+ minutes to start (boot time too slow for spikes)
- Stateful applications (new instances need data replication)
- Very consistent traffic patterns (fixed capacity is cheaper)
- Starting instances requires manual approval (regulated industries)

## Mental Model Check

1. Your database handles 1,000 writes/second and the primary CPU is at 90%. The read replicas are at 20%. Do you add more replicas or fix the write bottleneck?

2. A cache hit rate drops from 95% to 40% after a deployment. What changed? How do you diagnose?

3. You add 10 app servers but throughput only increases by 30%. What is the bottleneck? How do you find it?

4. An auto-scaling rule adds instances at 70% CPU, but each new instance takes 8 minutes to initialize. Traffic spikes from 100 RPS to 10,000 RPS in 2 minutes. What happens?

5. You shard a database by user_id modulo 16. Users 1-10% of the population generate 80% of the traffic. What problem does this cause, and how would you fix it?

6. A circuit breaker for a payment service transitions to OPEN during a traffic spike. A retry storm from 50 app servers continues hammering the payment service. What additional mechanism would prevent this?

7. Your API has a mean latency of 45ms but P99 latency of 4.2 seconds. What does this gap tell you about your system, and where do you look first?

8. A system uses eventual consistency with a 500ms replication lag. A user submits a review, then immediately refreshes the page and does not see it. How do you decide whether to add strong consistency or change the user experience?

9. You are designing a service with a 10:1 read-to-write ratio, and the database is on a single machine. The primary has a 256 GB SSD. The working set is 80 GB. A new feature requires scanning 200 GB of historical data daily. Do you add read replicas, increase disk, or restructure the query?

10. A two-node leader-election setup splits when network latency between the nodes exceeds 500ms. Both nodes think the other is dead and each becomes leader. What is this called, and how do you prevent it?

11. Your bulkhead design separates user-facing queries and batch-report queries into two connection pools. A misconfigured ORM starts opening connections in the user pool for report queries. What metric would alert you to this leak before users are impacted?

12. A gossip-protocol-based cluster has 50 nodes. Each node gossips with 3 random peers every second. How many seconds until every node knows about a new node joining? What happens at 500 nodes?

These questions test the depth of your intuition, not memorization. For each one, trace through the failure scenario step by step. A scalable system is one where your mental model of what should happen matches what actually happens when the load test runs. The gap between those two is where the learning lives.

This foundation leads into the mechanisms that make scaling work — consistent hashing, connection pooling, partition strategies, and the internal architecture that turns a single-server prototype into a global system. The deep-dive on each mechanism follows in the next article.
