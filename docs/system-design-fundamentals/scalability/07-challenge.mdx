## Instructions

Solve each exercise within the stated time limit. Write your solution first, then compare with the model answer.

**Grading rubric:**
- Does your solution state concrete numbers (QPS, storage, server count)?
- Does it identify the bottleneck at each growth stage?
- Does it propose a specific scaling mechanism (not just "add more servers")?
- Does it discuss trade-offs and failure modes?

## Tier 1: Foundation (15 minutes each)

### Problem 1: Database Connection Pool Sizing

You have 20 application servers. Each server processes 500 requests per second. Each request makes 2 database queries. The database is PostgreSQL with `max_connections = 300`. Your current connection pool is set to `max_size = 50` per app server.

**What happens under peak load? Calculate the actual connection demand vs supply, then recommend a fix.**

<details>
<summary>Model Answer</summary>

```
Connection demand calculation:
  - 20 app servers x 50 pool connections = 1,000 potential connections
  - PostgreSQL max_connections = 300
  - Connection excess = 700 connections that cannot be established

What actually happens:
  - App servers open connections until PostgreSQL hits 300
  - The 301st connection attempt fails
  - App servers: 15 get full pools (50 ea = 750), but only 300 succeed
  - Each server average: 300 / 20 = 15 connections per server
  - When those 15 are busy, remaining 35 workers per server queue
  - Queue timeout (default 30 seconds) — requests timeout
  - Error rate: 50-70% of requests fail

Recommended fix:
  pool_max_size = max_connections * 0.8 / app_servers
  300 * 0.8 / 20 = 12 connections per server

  Leave 20% headroom for admin connections, reporting tools, and batch jobs:
  Actual pool = 12 per server (leaves 60 connections admin buffer)

  Verify:
  20 servers x 12 connections = 240 total (under 300 max)
  Each server can process 12 concurrent DB queries
  At 500 RPS x 2 queries = 1000 QPS per server
  Each query takes ~12ms to complete (1000 QPS x 0.012s = 12 concurrent)
  This fits within 12 pool connections.

  Safety margin: 240 / 300 = 80% utilization
  Alert when pool utilization exceeds 70% for 5 minutes.
```
</details>

### Problem 2: Cache Hit Rate Crisis

Your Redis cache serves user profile data. It has 16 GB of memory. The application has 5 million users. Each user profile is 3 KB. The working set (users accessed in the last hour) is 2 million users. Cache hit rate is currently 94%. Usage is growing 10% per month.

**When will the cache hit rate collapse? What do you do?**

<details>
<summary>Model Answer</summary>

```
Working set growth:
  - Current: 2 million users x 3 KB = 6 GB (fits in 16 GB)
  - Next month: 2.2 million users x 3 KB = 6.6 GB
  - In 6 months: 3.2 million users x 3 KB = 9.6 GB
  - In 12 months: 6.2 million users x 3 KB = 18.6 GB

Cache capacity breach: Month 10
  - Working set reaches 5.2M users x 3 KB = 15.6 GB
  - At 90% memory utilization, Redis starts evicting (LRU)
  - Eviction rate increases as working set exceeds cache size

Cache hit rate decline:
  - Month 10: hit rate drops from 94% to 85% (5% eviction rate)
  - Month 11: hit rate drops to 70% (15% eviction rate)
  - Month 12: hit rate drops to 50% (users accessing evicted data)
  - Database load increases 3x, causing 200ms+ query latency

Remediation options:

Option A: Increase Redis memory (vertical scale)
  - Upgrade to 32 GB node: $500/month more
  - Buys 12 more months of growth
  - Simple, zero code changes

Option B: Shard Redis (horizontal scale)
  - Split cache across 2 Redis nodes (16 GB each)
  - Shard by user_id % 2
  - More complex but no single-node memory limit
  - Supports future growth beyond 32 GB

Option C: Selective caching
  - Cache only profiles accessed in the last 15 minutes (hot set)
  - Reduce working set from 2M to 500K (1.5 GB)
  - Frees 14 GB for other data
  - Requires cache invalidation logic for cold-start users

Recommended: Option A now (buy time) + plan Option B for Month 9.
```
</details>

### Problem 3: Auto-Scaling Oscillation

Your auto-scaling group adds instances when CPU exceeds 70% and removes them when CPU drops below 30%. The scale-out cooldown is 60 seconds. During a traffic spike, the system adds 5 instances. CPU drops to 40%. After 60 seconds, the scale-in policy triggers and removes 2 instances. CPU jumps back to 75%. The group adds 2 instances. CPU drops to 45%. This cycle repeats every 2 minutes.

**Identify the problem and fix the configuration.**

<details>
<summary>Model Answer</summary>

```
Problem: Auto-scaling oscillation (flapping)

Root cause:
  - Scale-in cooldown (60 seconds) is too short — new instances
    are removed before traffic naturally fluctuates
  - Scale-in threshold (30% CPU) is too close to the post-scale CPU
    level (40%), so a small traffic dip triggers removal
  - Scale-out step (5 instances) is too aggressive — overshoots target

Fix:

Scale-out:
  - Step size: 2 instances (not 5)
  - Threshold: CPU > 70% for 3 consecutive evaluations (not immediate)
  - Cooldown: 180 seconds (let metrics stabilize)

Scale-in:
  - Threshold: CPU under 20% for 10 consecutive minutes (not 30% for 60s)
  - Cooldown: 300 seconds
  - Step size: 1 instance at a time

Resulting behavior:
  - Spike hits: adds 2 instances, CPU drops to 55% (within range)
  - Evaluates again in 180s: if CPU still > 70%, adds 2 more
  - Gradual scaling: never overshoots
  - Scale-in only happens when CPU has been low for 10 minutes
  - No oscillation

Target tracking alternative:
  - Set target CPU = 55% (auto-scaling adjusts to maintain this)
  - AWS target tracking handles the math internally
  - Less flexible but eliminates oscillation.
```
</details>

### Problem 4: DNS Resolution at Scale

Your application serves users across three regions: US-East, EU-West, and Asia-SE. You need a DNS-based load balancing strategy that routes users to the nearest healthy region. Each region runs 20 application instances behind a regional load balancer. You expect 50,000 QPS globally. Traffic distribution is 50% US, 30% EU, 20% Asia. You must handle an entire region failing over to the next closest region within 60 seconds. Users cache DNS records, so TTL decisions directly affect failover speed.

**Design the DNS routing architecture with specific TTL values, health check configurations, and failover logic.**

<details>
<summary>Model Answer</summary>

```
DNS architecture: Latency-based routing with geo-proximity fallback
  US-East: app-us.example.com  A → 203.0.113.10  TTL: 30s
  EU-West: app-eu.example.com  A → 203.0.113.20  TTL: 30s
  Asia-SE: app-asia.example.com A → 203.0.113.30 TTL: 30s

Health checks: HTTPS GET /healthz every 10s from 3 locations
  Unhealthy: 3 consecutive failures (30s), 2-of-3 checkers agree

Failover (US-East down): T+30 removed, T+120 full propagation. Exceeds 60s target. Mitigate with client-side pre-fetch: resolve all 3 regions every 10s, cache locally, switch in ~5s.
Capacity: EU-West absorbs 25K+15K=40K QPS. Scale from 15 to 40 instances.
```
</details>

## Tier 2: Applied (20 minutes each)

### Problem 5: Multi-Region Write Conflicts

You operate a global note-taking app with 25 million users across 3 regions (US, EU, Asia). You deploy active-active in all 3 regions. A user in Tokyo edits a note at 14:00 UTC. A user in New York edits the same note at 14:01 UTC. Both changes replicate to the other regions.

**Design a conflict resolution strategy that minimizes data loss and keeps latency under 200ms for writes.**

<details>
<summary>Model Answer</summary>

```
Design: Last-Writer-Wins with per-field granularity

Approach:
  Each note stored as a set of fields (title, body, tags).
  Each field has a version vector: (region_id, timestamp, crdt_id).
  Conflicts resolved independently per field.

Write path:
  User edits body in Tokyo at T1:
    - Store: note_123.body = "new body text"
    - Version: {region: "ap-southeast-1", ts: T1, crdt_id: UUID}
    - Response: 200 OK in 50ms (local write)

  User edits title in New York at T2:
    - Store: note_123.title = "new title"
    - Version: {region: "us-east-1", ts: T2, crdt_id: UUID}
    - Response: 200 OK in 50ms (local write)

Conflict resolution:
  Cross-region replication (async, 500ms-2s lag):
    Tokyo receives New York's title update:
      - title field: version (US, T2, UUID) is newer than (Tokyo, T0, UUID)
      - Accept: "new title"

    New York receives Tokyo's body update:
      - body field: version (Tokyo, T1, UUID) is newer than (US, T0, UUID)
      - Accept: "new body text"

  Same-field conflict (rare):
    Tokyo: body = "alpha" at T1
    New York: body = "beta" at T1 (same microsecond)
    - LWW by (timestamp, region_id): tiebreak on region priority
    - US has higher priority than AP → body = "beta"
    - Alternative: CRDT merge (both values preserved, merge algorithm)

Limitations:
  - 2-second replication window for potential conflicts
  - Same-field conflicts lose one edit (LWW)
  - Cannot enforce uniqueness constraints globally (email, username)

Detection:
  - Log conflict rate per region: alert if > 0.1% of writes
  - Track replication lag: alert if > 5 seconds
  - Monitor "lost edits" ratio: target under 0.01%
```
</details>

### Problem 6: Read-Heavy News Feed at 100 Million Users

Design a news feed system that supports 100 million daily active users. Each user follows 200 accounts on average. Each follow produces 5 posts per day. Users check their feed 10 times per day.

**Calculate the read/write ratio and design the feed generation pipeline.**

<details>
<summary>Model Answer</summary>

```
Capacity estimation:

Writes (posts created):
  100M users / 5 (active posters) = 20M daily posts
  Average: 20M / 86400 = 231 posts/second peak: 2,000 posts/second

Reads (feed views):
  100M users x 10 feed checks/day = 1B daily feed views
  Average: 1B / 86400 = 11,574 reads/second
  Peak: 50,000 reads/second

Read-to-write ratio:
  50,000 reads / 2,000 writes = 25:1 (read-heavy)

Feed generation: Hybrid fan-out

Fan-out on write (for users with under 10K followers):
  When a user posts:
    - Look up all followers from Redis (sharded by user_id)
    - Insert post ID into each follower's timeline (Redis sorted set)
    - Timeline key: feed:{follower_id}, max 500 posts, TTL 24h
  Writing to 200 followers (average user): 200 Redis ZADD ops
  Total: 200 x 2,000 = 400,000 ops/sec — feasible on Redis cluster

Fan-out on read (for users with 10K+ followers):
  When a user views feed:
    - Fetch posts from followed accounts (last 24 hours)
    - Merge by timestamp
    - Return top 50
  Only needed for 0.1% of users (celebrities)
  Cache their recent posts (last 50) — regenerate every minute

Database:
  - posts table sharded by post_id (32 shards)
  - follower graph in Redis (1KB per user, 100GB total)
  - Timeline cache: 100M users x 500 posts x 8 bytes = 400GB
  - Redis cluster: 16 nodes x 32GB = 512GB total
```
</details>

### Problem 7: Queue Backpressure

Your Kafka cluster processes order events. A flash sale causes 1 million orders in 1 minute (16,667 orders/second). Your normal throughput is 500 orders/second. The Kafka cluster has 6 partitions across 3 brokers. Each partition handles 100 orders/second before consumer lag grows.

**Design the system to handle 30x traffic spikes without losing data.**

<details>
<summary>Model Answer</summary>

```
Problem analysis:
  - Normal throughput: 500 orders/sec (fits in 6 partitions)
  - Flash sale spike: 16,667 orders/sec (33x normal)
  - Partition capacity: 100 orders/sec each = 600 orders/sec total
  - Excess: 16,067 orders/sec accumulate in Kafka

Time to consumer lag:
  - Each partition has 1GB retention (default: 7 days, but let us
    assume 1GB practical headroom)
  - Each order: 2KB message
  - Capacity per partition: 1GB / 2KB = 500,000 messages
  - 6 partitions: 3M messages buffer
  - At 16,167 excess/sec: 3M / 16,167 = 185 seconds (~3 minutes)
  - After 3 minutes: consumer lag exceeds disk, messages dropped

Solutions (apply in combination):

1. Increase partitions (BEFORE the event):
   - Scale from 6 to 60 partitions (10 per broker)
   - Partition capacity: 60 x 100 = 6,000 orders/sec
   - 15x normal — still not enough for 33x

2. Queue-based load leveling:
   - Accept that consumers cannot keep up
   - Let Kafka buffer messages
   - Increase retention to 72 hours (large disks: 10TB per broker)
   - Buffer: 10TB / 2KB = 5B messages — handles any spike duration
   - Consumers process backlog at 600/sec after spike ends
   - Backlog of 1M orders / 600 = 28 minutes to catch up

3. Client-side throttling (admission control):
   - Enqueue order request, return "pending" immediately
   - Client polls for order confirmation
   - Server processes from Kafka at capacity
   - User experience: order submitted (pending), confirmed in 5-30 min
   - Better than "500 Internal Server Error"

4. Prioritized processing:
   - High-priority topic: first 10,000 orders (VIP customers)
   - Normal topic: remaining orders
   - Dedicated consumer group for high-priority: 50 partitions
   - VIP orders processed within 30 seconds
```
</details>

### Problem 8: Session Store Scaling

You operate an online platform with 100 million users. Each user maintains an active session. Sessions expire after 30 minutes of inactivity. Each session stores 2 KB of data (user_id, roles, preferences, cart). Peak concurrency is 60 million simultaneous sessions. You must support session reads in under 5ms and session writes in under 10ms. You need a failover strategy that loses no more than 1% of sessions during a node failure.

**Design the Redis cluster architecture: calculate memory requirements, determine node count, define session TTL strategy, design reconnection handling, and specify the failover approach.**

<details>
<summary>Model Answer</summary>

```
Capacity: 60M x 2 KB = 120 GB, +9 GB overhead + 26 GB buffer = 155 GB (round to 200 GB)
Topology: 16 primaries x 16 GB (256 GB), CRC32(session_id) % 16384, 16 replicas
  Instance: r6g.xlarge (4 vCPU, 32 GB), ~3.75M sessions per primary

TTL: 1800s, refresh on read via EXPIRE, allkeys-lru eviction
  Read p99 ~3.5ms, Write p99 ~5.5ms (both under target)

Reconnection: pool 10 conns/primary, TCP timeout 2s, backoff 50→800ms
Failover: 3 Sentinels (Q:2), SDOWN 5s, ODOWN 2-of-3, promotion ~15s
  Data loss ~1% (async lag 50-500ms), WAIT command mitigates (+1-2ms)
  Cross-region DR: async, RPO 5s, RTO 60s
```
</details>

## Tier 3: System Design (30 minutes each)

### Problem 9: Design a Scalable URL Shortener

Design a service like bit.ly for 10 billion URL creations per year. Each short URL is viewed 100 times on average. URLs persist forever.

**Include: capacity estimates, data model, key generation, caching strategy, database scaling plan, and a failure mode analysis.**

<details>
<summary>Model Answer</summary>

```
Capacity estimates:

Creates: 10B / year = 27M/day = 317 creates/second (peak 1,000/s)
Reads: 10B x 100 = 1T views/year = 2.7B/day = 31,700 reads/second
  Peak reads: 100,000/second
Storage: 10B URLs x (short_key 7 bytes + long_url 500 bytes + meta 100 bytes)
  = 10B x 607 bytes = 6.07 TB total
  10 years: 60 TB

Data model:

short_urls table (sharded by short_key):
  short_key: VARCHAR(7) PRIMARY KEY
  long_url: TEXT (URL encoded, max 2048 chars)
  created_at: TIMESTAMP
  expires_at: TIMESTAMP (NULL for permanent)
  user_id: BIGINT (NULL for anonymous)

click_events table (separate database, append-only):
  short_key: VARCHAR(7)
  clicked_at: TIMESTAMP
  referer: TEXT
  user_agent: TEXT
  country: VARCHAR(2)
  (Partitioned by clicked_at month)

Key generation (base62 encoding):
  7 characters of [a-zA-Z0-9] = 62^7 = 3.5 trillion combinations
  Enough for 350 years at 10B/year

Generation strategies:
  Option A: Pre-generate keys in batches of 1M
    - Batch service generates keys, stores in Redis set
    - App servers pop keys from Redis (O(1))
    - Batch generates new batch when count under 100K
    - No ID collisions, no blocking on write path

  Option B: Counter + hash
    - Centralized counter (Redis atomic INCR)
    - Hash counter to base62: base62_encode(INCR counter + random_offset)
    - Single point of failure — use Redis Cluster
    - OK for 317 creates/sec (Redis handles 100K+ INCR/sec)

Caching:
  - Cache short_key -> long_url in Redis
  - TTL: 24 hours (URLs rarely expire)
  - Working set: top 1% of short URLs = 10B x 1% = 100M keys
  - Memory: 100M x (7 + 500 + 32 overhead) = 54 GB
  - Redis cluster: 6 nodes x 16 GB = 96 GB
  - Cache hit target: 99% (top 1% URLs serve 99% of traffic)

Database scaling:
  Phase 1 (0-100M URLs): Single PostgreSQL, read replicas
  Phase 2 (100M-1B): Shard by short_key (32 shards)
  Phase 3 (1B+): Consistent hashing across 128+ shards

Failure modes:
  1. Key collision: In 3.5T combinations, collision probability at 10B
     is negligible (birthday paradox: 10B^2 / 2 x 3.5T = 1.4%)
     Mitigation: check for collision on insert, retry with new key
  2. Cache stampede: Redis master fails, 100K reads hit DB
     Mitigation: Redis Cluster with replica failover, circuit breaker at 10K QPS
  3. Storage cost: 60TB in 10 years at $0.02/GB/month = $14,400/month
     Mitigation: expire unused URLs after 12 months, archive to S3 Glacier
```
</details>

### Problem 10: Design a Global File Upload Service

Design a service like Google Drive or Dropbox for 200 million users. Each user stores 5 GB of files. Users upload 100 MB per day on average.

**Calculate storage, design the upload pipeline, and plan for geographic distribution.**

<details>
<summary>Model Answer</summary>

```
Capacity estimates:

Total storage: 200M x 5GB = 1,000 PB (1 exabyte)
Daily upload: 200M x 100MB = 20 PB/day
Upload QPS: 20 PB / 86400 / 100MB average = 2,315 uploads/sec
  Peak: 10,000 uploads/sec
Download QPS: 10x upload = 100,000 downloads/sec peak

Storage tier strategy:
  Hot tier (accessed in last 7 days): 140 PB
    - SSD-backed object store (S3 Express or equivalent): $0.02/GB/month = $2.8M/month
  Warm tier (accessed in last 90 days): 400 PB
    - Standard object store (S3 Standard): $0.023/GB/month = $9.2M/month
  Cold tier (accessed in last 12 months): 460 PB
    - Infrequent access (S3 IA): $0.0125/GB/month = $5.75M/month
  Archive tier (older than 12 months): Free tier or Glacier: $0.004/GB/month = $1.84M/month

Upload pipeline (async):

  Step 1: Client uploads to nearest edge (CDN with upload support)
    - Chunked upload: 5MB chunks
    - Parallel upload: 4 concurrent chunks
    - Resumable: track progress per chunk

  Step 2: Edge receives chunks, returns receipt
    - Chunk stored in edge cache (24-hour TTL)
    - Chunk metadata written to local queue

  Step 3: Async processing (Kafka + workers)
    - Reassemble chunks into file
    - Compute SHA-256 checksum (verify integrity)
    - Virus scan
    - Deduplicate (content-addressed storage: dedup by SHA-256)
    - Generate thumbnail (for images/videos)
    - Move to hot tier storage

  Step 4: Replication
    - Replicate to 2nd region (async, within 5 minutes)
    - Update metadata database (sharded by user_id)
    - Notify client: upload complete

Geographic distribution:
  - 8 regions: US-East, US-West, EU-West, EU-North, Asia-SE, Asia-NE, SA-East, AU-SE
  - Write: user uploads to nearest region
  - Read: user downloads from nearest region (CDN for frequent files)
  - Cross-region replication: each region replicates to 1 other region
  - Metadata: per-region database, global metadata in DynamoDB (global tables)

Failure modes:
  1. Upload fails mid-stream: resumable upload resumes from last chunk
  2. Disk failure in hot tier: replicated to 2nd region, failover in 5 minutes
  3. CDN edge saturated: redirect to closest available edge
  4. Virus detected: quarantine file, notify user, restore from backup
```
</details>

## Common Mistakes
1. **Ignoring headroom.** You forget admin connections and replication overhead. Always leave 15-20%.
2. **Setting TTL too long for failover.** DNS TTL of 300s means 5+ min failover. For &lt;60s failover, use 30s or less.
3. **Using defaults.** Redis `noeviction`, PostgreSQL `max_connections=100` — defaults are for development. Configure explicitly.
4. **Forgetting cache miss cascades.** Hit rate 94% to 89% means DB queries go from 6% to 11% of traffic — an 83% jump. Calculate relative change.
5. **Single-region bottleneck.** Every component must be distributed across regions.
6. **Overlooking reconnection storms.** Use exponential backoff, never fixed retries.
7. **Confusing vertical vs horizontal timelines.** Vertical takes minutes. Horizontal may take hours.
