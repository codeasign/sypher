# Scalability Architecture Patterns

## Naive Architecture: Single Server

**Limitations at this scale:**
- Any deployment requires downtime (stop server, update code, restart)
- Database backup blocks requests (no replicas for failover)
- No redundancy — if the server dies, the app is down until restored
- Response time degrades linearly with traffic

### At 100,000 Users

The single server cannot handle 100,000 users. The database connection pool exhausts. CPU runs at 100%. Requests time out.

```
Predicted performance at 100K users (single server):
  - 50 concurrent users peak (0.15% of daily active users)
  - Database connection pool: 100 connections (maxed at 20 concurrent)
  - Average query time: 800ms (when pool not exhausted)
  - Error rate: 15% (connection timeouts)
  - Cost per user: $0.003 per request (vs $0.0001 at 1K)
```

You must split the application across multiple machines.

### At 10,000,000 Users

A single server for 10 million users is physically impossible. Even the largest cloud instance (AWS u-12tb1.metal: 448 vCPUs, 12TB RAM at $150/hour) would fail because the database alone needs more than 12TB for a user-generated content platform — and the single point of failure is unacceptable for a business at this scale.

### Per-Stage Cost Breakdown and Instance Types

Each growth stage demands a different server profile and budget.

```
Stage 1: 0-100 users — t3.nano app + db.t3.micro DB — $23/mo — all-in-one
Stage 2: 100-5K users — t3.medium × 1 + db.t3.small — $75/mo — separate app/DB
Stage 3: 5K-50K users — c6i.large × 2 + db.r6g.large primary+replica + Redis — $642/mo
Stage 4: 50K-500K users — c6i.xlarge × 10 + 4 shards + Kafka — $7,025/mo
Stage 5: 500K-10M users — c6i.2xlarge × 100 + 12 shards + multi-region — $63,550/mo
```

## Database Scaling Patterns

### Pattern 1: Read Replicas

Read replicas copy data from the primary database and serve read-only queries. Most applications are read-heavy (10:1 ratio).

```
Configuration:
  Primary: 1 node (all writes, replication log)
  Replicas: 2-5 nodes (read queries, reporting, backups)
  Replication: Asynchronous (default) or synchronous

Routing:
  Write queries ──> Primary database
  Read queries ──> Random replica (round-robin)
  Reporting queries ──> Dedicated replica (isolated from traffic)

Limitation:
  - Asynchronous replication lag: 10-100ms typical
  - A user writes data then reads immediately — may not see their own write
  - Solution: Read-after-write consistency ("read your own writes" pattern)

Scaling ceiling:
  - Primary still handles all writes (single bottleneck)
  - ~5,000 writes/second before the primary becomes saturated
```

### Pattern 2: Database Sharding

Sharding splits data across multiple databases by a shard key. Each shard is an independent database with its own primary and replicas.

<AsciiDiagram
  title="Sharding Strategies"
  content={`
Strategy A: Range-Based Sharding
  Shard 0: user_id 1-1,000,000
  Shard 1: user_id 1,000,001-2,000,000
  Shard 2: user_id 2,000,001-3,000,000

  Problem: Hot spot on the latest shard (new users write there)
  Benefit: Range queries are efficient, easy to understand

Strategy B: Hash-Based Sharding
  Shard = hash(user_id) % 4

  0 ──> Shard 0    2 ──> Shard 2
  1 ──> Shard 1    3 ──> Shard 3

  Benefit: Uniform distribution (no hot shards)
  Problem: Adding shards changes N, remaps everything
  Solution: Consistent hashing (see Deep Dive)

Strategy C: Directory-Based Sharding
  Lookup table: user_id ──> shard_id

  ┌────────────────────────────┐
  │ user_id  │ shard_id        │
  ├──────────┼─────────────────┤
  │ 1        │ shard-0         │
  │ 2        │ shard-1         │
  │ 3        │ shard-0         │
  │ 1000001  │ shard-5         │
  └────────────────────────────┘

  Benefit: Flexible, can move data between shards
  Problem: Lookup table is a single point of failure and bottleneck
  Solution: Cache lookup table in Redis
  `}
  alt="Three database sharding strategies: range-based with hot spot problem, hash-based with uniform distribution, and directory-based with lookup table"
  caption="Choose a sharding strategy based on query patterns and growth expectations"
/>

**Shard key selection rules:**

```
Good shard keys:
  - High cardinality (many distinct values)
  - Even distribution across values
  - Natural to application queries
  - Immutable (or rarely changes)

Bad shard keys:
  - Boolean values (max 2 shards)
  - Creation date (all writes hit one shard)
  - Country (US dominates, creates hot shard)
  - Null values (all nulls go to one shard)

Examples:
  ✓ user_id (UUID or auto-increment) — excellent
  ✓ order_id — excellent
  ✓ restaurant_id — excellent for restaurant-centric apps
  ✗ is_premium (true/false) — terrible
  ✗ created_at — terrible for write-heavy workloads
```

### Pattern 3: Cache Scaling (Redis Cluster)

Redis Cluster distributes data across multiple nodes using hash slots.

```
16384 hash slots total (fixed by Redis protocol)
Each node owns a range of hash slots

3-node cluster:
  Node A: slots 0-5460
  Node B: slots 5461-10922
  Node C: slots 10923-16383

Adding Node D:
  Migrates slots from A, B, C to D
  Continuous operation during migration
  Clients handle MOVED redirects

Scaling limits:
  - Maximum 1000 nodes in a cluster
  - Each node: ~25GB memory (practical limit for replication)
  - Total cluster: ~25TB
  - Network bandwidth becomes bottleneck before CPU
```

### Cache Scaling Patterns Beyond Redis Cluster

Production systems layer multiple caching strategies across the request path before reaching Redis.

**Memcached pool:** Simpler than Redis Cluster when you only need key-value caching. No replication, no persistence, no data structures. Client-side consistent hashing distributes keys across nodes. Adding a server remaps a fraction of keys rather than all of them. Each r6g.large node handles ~200,000 get/set operations per second at sub-1ms latency within the same availability zone. Maximum practical pool size is about 50 nodes before client-side connection management becomes complex.

```
Memcached vs Redis Cluster:
┌────────────────────────────┬─────────────────────┬─────────────────────┐
│ Feature                   │ Memcached Pool      │ Redis Cluster       │
├────────────────────────────┼─────────────────────┼─────────────────────┤
│ Data structures           │ Strings only        │ Strings, hashes,    │
│                           │                     │ lists, sets, sorted │
│ Replication               │ None                │ Master-replica      │
│ Persistence               │ None                │ RDB / AOF           │
│ Multi-key operations      │ Not supported       │ Supported           │
│ Memory overhead per key   │ ~50 bytes           │ ~100+ bytes         │
│ Cluster coordination      │ Client-managed hash │ Server-managed      │
│                           │ ring                │ 16384 hash slots    │
│ Best for                  │ Simple KV cache     │ Complex caching     │
│                           │                     │ + sessions + rate   │
│                           │                     │ limiting            │
└────────────────────────────┴─────────────────────┴─────────────────────┘
```

**CDN multi-tier caching:** Three layers protect your origin. Browser cache (Cache-Control headers) serves ~40% of repeat requests from local disk. Edge POPs (200 global locations) serve another ~60% of what misses the browser. Regional hubs (10 locations) catch ~85% of remaining misses. Net result: over 96% of requests never reach your origin servers.

**Browser caching headers:**

| Directive | Use case | TTL | CDN behavior |
|-----------|----------|-----|-------------|
| public, max-age=31536000, immutable | Static assets (images, fonts) | 1 year | Cache at CDN and browser |
| no-cache | HTML pages, API responses | Always revalidate | Must check origin |
| private, max-age=300 | User-specific data | 5 minutes | CDN must NOT cache |
| no-store | Auth tokens, payments | Never | No caching anywhere |
| stale-while-revalidate=86400 | Semi-static content | Serve stale + refresh | Background fetch on hit |

**ETag/If-None-Match:** Server sends a content hash. Client includes it on re-request. If unchanged: 304 Not Modified (zero bytes transferred). If changed: 200 with new body and ETag.

**Cache invalidation strategies:** Three approaches to purge stale content. Time-based invalidation (TTL expiry) is simplest but can serve stale data. Key-based invalidation (purge specific keys by pattern) gives precision but requires knowing exactly what to invalidate. Versioned resources (/static/style.v2.css) shift the invalidation burden to the deployment pipeline — old versions naturally fall out of cache over time. Most production systems combine all three: TTL for safety, key-based purge for emergencies, versioning for deployments.

### Pattern 4: Queue Scaling (Kafka Partitions)

Kafka achieves scalability through partitions. A topic is split into partitions, and each partition can be hosted on a different broker.

```
Topic: "order_events"
Partitions: 12
Replication factor: 3
Brokers: 6

Each broker hosts 6 partition leaders (12 partitions / 6 brokers * 3 replicas)
Each partition leader: handles all reads and writes for that partition

Consumer groups:
  Group A: 6 consumers, each reads 2 partitions (balanced load)
  Group B: 12 consumers, each reads 1 partition (maximum parallelism)
  Group C: 1 consumer, reads all 12 partitions (sequential processing)

To scale:
  Consumers: add consumer instances up to number of partitions
  Partitions: increase partition count (requires careful migration)
  Brokers: add brokers and reassign partition leadership
```

### Pattern 5: NoSQL vs SQL Scaling Comparison

| Feature | PostgreSQL | DynamoDB | Cassandra | MongoDB | CockroachDB |
|---------|-----------|---------|-----------|---------|------------|
| Scaling model | Read replicas + manual shard | Auto-partitioned | Peer-to-peer add nodes | Sharded cluster | Auto-sharded |
| Consistency | Strong ACID | Eventual (strong read-your-writes) | Tunable (eventual to strong) | Configurable (strong/eventual) | Serializable (Raft) |
| Max practical nodes | ~50 shards | Unlimited | 300+ | 100+ shards | Unlimited (cloud) |
| Write throughput/node | ~5K writes/sec | Unlimited | 1M+ writes/cluster | ~100K writes/cluster | ~100K writes/cluster |
| Query expressiveness | Full SQL, joins, CTEs | Single-table only | Partition + sort key (CQL) | Rich queries, indexes | Full SQL + geo-distributed |
| Operational cost | 1x baseline | 2-3x | ~1x (self-managed) | 1.5-2x | 2-4x |

**When to pick each:** PostgreSQL for ACID and complex queries, scale with replicas then manual sharding at 10TB+. DynamoDB for zero-ops auto-scaling KV workloads where query flexibility is not needed. Cassandra for massive write throughput (IoT, time-series) with eventual consistency tolerance. MongoDB for evolving document schemas with auto-sharding. CockroachDB for SQL semantics with global distribution — but at higher latency and cost.

### Database Migration Strategies During Scaling

Scaling your database from a single node to a sharded cluster is one of the riskiest operations in system design.

**Online schema migrations (gh-ost):** Create a shadow table with the new schema, apply binlog replication from the original table, then atomically RENAME the tables. Lock wait: 0 milliseconds. Downside: requires MySQL binlog support.

**Zero-downtime sharding:** A proxy layer sits between your app and the database. Phase 1: dual-write to old DB and new shards. Phase 2: backfill historical data via CDC pipeline. Phase 3: cutover reads to new shards. Phase 4: decommission old DB. Tools: ProxySQL, Vitess (YouTube-scale), pg_shardman.

            └──> Shard 0          stream)    └──> Shard 2            └──> Shard 2

**Blue-green database deployments:** Run two complete database stacks (different versions, sizes, or shard counts) side by side. Provision Green, replicate Blue to Green via CDC, validate with canary queries, switch DNS, retain Blue for 72-hour emergency rollback. Typical switch time: under 30 seconds. Rollback time: 5-15 minutes.

**Multi-region considerations:**

| Factor | Single region | Multi-region |
|--------|--------------|--------------|
| Latency | 50ms (same coast) | 5ms (edge) to 200ms (global) |

| Complexity | Low | High (conflict resolution, sync) |

| Data consistency | Strong | Eventual (cross-region) |

### Real Cloud Provider Latency

Measured round-trip p99 latency between AWS regions for a 1KB payload:

| Region pair | Distance | p50 | p99 | Bandwidth |
|-------------|----------|-----|-----|-----------|
| us-east-1 ↔ us-west-2 | 3,900 km | 32ms | 64ms | 25 Gbps |
| us-east-1 ↔ eu-west-1 | 5,600 km | 45ms | 87ms | 10 Gbps |
| us-east-1 ↔ ap-southeast-1 | 15,500 km | 125ms | 210ms | 5 Gbps |
| us-east-1 ↔ sa-east-1 | 7,500 km | 80ms | 145ms | 2.5 Gbps |
| eu-west-1 ↔ eu-central-1 | 900 km | 13ms | 25ms | 25 Gbps |
| us-west-2 ↔ ap-northeast-1 | 8,100 km | 85ms | 150ms | 5 Gbps |

**Region triads for coverage:**

- **Tier 1 (3 regions, 80% of users):** us-east-1 + eu-west-1 + ap-southeast-1. Cost: ~3.2x single region. Covers 75% of global users within 50ms.
- **Tier 2 (6 regions, 95%):** Add us-west-2 + sa-east-1 + ap-northeast-1. Cost: ~5.8x. Covers 92% within 50ms.
- **Tier 3 (12 regions, 99%):** Add eu-central-1 + me-south-1 + af-south-1 + ap-south-1 + ap-southeast-2 + ca-central-1. Cost: ~11x. Covers 99% within 80ms.

**Cross-region data transfer cost warning:** At 10TB/day replication, us-east-1 to ap-southeast-1 costs approximately $27,000/month in transfer fees alone. This is often the single largest line item in a global cloud bill.

## Capacity Planning

### QPS to Server Count Calculation

```
Given:
  - Average request processing time: 50ms per request
  - Server capacity: 4 vCPU, 8GB RAM
  - Target CPU utilization: 60% (leaves headroom for spikes)
  - Concurrent capacity per server: 50 requests (at 50ms each, 4 CPUs)
  - Peak traffic: 50,000 requests per second

Calculation:
  Server capacity at 60% CPU = 50 * 0.60 = 30 concurrent requests
  Servers needed = 50,000 / 30 = 1,667 servers

  But most requests don't need a full 50ms (cache hits are faster).
  With 90% cache hit rate:
    Cache hit: 5ms processing (read from Redis, return)
    Cache miss: 50ms (query database)
    Average: 0.9 * 5ms + 0.1 * 50ms = 9.5ms

  Recalculate:
    Server capacity at 60% CPU = 50 * (50/9.5) ≈ 263 concurrent requests
    Servers needed = 50,000 / 263 ≈ 190 servers
```

## Infrastructure as Code: Auto-Scaling Configurations

### Kubernetes HPA

```
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: app-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: app-server
  minReplicas: 10
  maxReplicas: 200
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60
    - type: Pods
      pods:
        metric:
          name: requests_per_second
        target:
          type: AverageValue
          averageValue: 500
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Percent
          value: 100
          periodSeconds: 15
      selectPolicy: Max
```

Key tuning: aggressive scale-up (doubles capacity every 15 seconds), conservative scale-down (10% per minute after 5-minute stabilization) to prevent thrashing.

## Deployment: Zero-Downtime Scaling

Scaling events (adding or removing servers) should not cause downtime.

```
Zero-downtime deployment during scale-out:

Phase 1 (Prepare):
  - Pre-provision new instances with latest AMI
  - Warm up instance (install packages, configure)
  - Add to load balancer target group (health checks disabled)

Phase 2 (Warm up):
  - Enable health checks (endpoint: /health)
  - Wait for 2 consecutive healthy checks (10 seconds)
  - Instance begins receiving traffic

Phase 3 (Observe):
  - Monitor error rates and latency for 5 minutes
  - If errors spike: drain and remove instance
  - If stable: mark as active

Phase 4 (Scale-in):
  - Remove instance from load balancer (connection draining)
  - Wait for existing requests to complete (30-second drain timeout)
  - Send SIGTERM to application (graceful shutdown)
  - Wait 30 seconds for graceful shutdown
  - Send SIGKILL if process still running
  - Terminate instance
```

**Monitoring scale events:**

| Metric | Warning | Critical | Action |
|--------|---------|----------|--------|
| New instance health check failures | >5% | >20% | Roll back deployment |
| Post-scale latency increase | >20% | >50% | Scale down, investigate |
| Replication lag (during DB scale) | >10 seconds | >60 seconds | Pause migration |
| Cache miss rate (post scale) | >30% | >50% | Pre-warm cache on new nodes |

**Parallel test execution:** Shard 16,000 tests across 16 workers. Serial: 60 minutes. 16-way: 4 minutes 30 seconds (13x improvement). Use timing-based shard allocation (pytest-split, jest --shard, JUnit parallel) so each shard takes equal time.

**Build caching:** Cache Docker layers (base image → deps layer hit 85% of the time), dependency artifacts (Maven/npm/pip, 80% hit rate), and intermediate build outputs. Structure your Dockerfile to maximize cache reuse: install dependencies before copying source code so the expensive pip/npm/mvn install step only re-runs when requirements change. Store caches in S3/ECR with a checksum-based cache key.

```
  Dockerfile order for cache efficiency:
  FROM python:3.11-slim           → cached (weekly rebuild)
  COPY requirements.txt .         → cached if hash unchanged
  RUN pip install -r requirements → 15 min → 0 min on cache hit
  COPY . .                        → always invalidated
  RUN pytest --shard              → 60 min → 4 min (16-way)
```

Reduces total build time from 75 minutes to 6 minutes — a 12.5x improvement.

**Artifact distribution:** Push once to us-east-1 ECR with automatic cross-region replication. Each region pulls from its local replica (500MB in 35 seconds) rather than cross-region (same image in 4 minutes, throttled).

The architecture patterns in this chapter — read replicas, sharding, caching layers, queues, multi-region deployment, online migrations, scaled CI/CD, and infrastructure-as-code — are the building blocks of every system that serves millions of users. Each pattern trades simplicity for scale. The next section, Tradeoffs and When to Use Each Pattern, helps you decide which combination fits your specific growth stage.
