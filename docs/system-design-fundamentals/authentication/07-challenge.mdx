# Authentication — Challenges

This page contains eight challenges organized across three tiers. Each challenge
simulates a real interview scenario. You should attempt each challenge under the
stated time limit before reading the model answer.

---

## Tier 1 — Foundation

These challenges test your understanding of core authentication concepts. Each
should take no more than five minutes to complete.

---

### Challenge 1: Password Hashing

**Time limit:** 5 minutes

**Prompt:**

A junior engineer on your team proposes storing user passwords by applying
SHA-256 directly to the password and storing the resulting hash. They argue that
"hashing is one-way, so it is secure." Explain to them why this approach is
dangerous, what a salt is and why it is necessary, and how hashing differs from
encryption.

**Evaluation criteria:**

- You identify that raw SHA-256 is vulnerable to rainbow table attacks
- You explain that a salt is a random per-user value appended to the password
  before hashing, and that it defeats precomputation attacks
- You describe the difference between hashing (one-way, no key) and encryption
  (two-way, requires a key)
- You mention that dedicated password hashing functions (bcrypt, argon2,
  scrypt) are preferred because they are computationally expensive and
  configurably slow
- Bonus: you discuss pepper (a secret salt stored server-side) as an additional
  layer

**Common mistakes:**

- Claiming that SHA-256 is sufficient if you "hash twice"
- Confusing salting with encryption
- Suggesting that you should encrypt passwords rather than hash them
- Omitting the threat model (database leak is the assumed breach)

**Model answer:**

Password hashing converts a plaintext password into a fixed-size digest using a
one-way function. The key property is that it is computationally infeasible to
reverse the operation.

Raw SHA-256 is unsuitable because it is fast and deterministic. Attackers can
precompute SHA-256 for every common password and store the results in a rainbow
table. Given a leaked hash, they simply look it up. A salt solves this: a
random value (typically 16-32 bytes) is generated per user, concatenated with
the password, and the combination is hashed. The salt is stored alongside the
hash in plaintext. Even if the attacker has the salt, they must compute a new
rainbow table per salt — infeasible at scale.

Hashing and encryption are fundamentally different. Hashing is one-way: you
cannot recover the original input from the output. Encryption is two-way: you
can decrypt ciphertext back to plaintext if you possess the key. Encrypting
passwords would mean storing a key, and if that key is compromised, all
passwords are recoverable. Hashing avoids this risk entirely.

Modern password hashing functions like bcrypt, argon2, and scrypt add a work
factor that makes each hash computation intentionally expensive. This slows
down brute-force attacks even when the hashes and salts are leaked.

### Challenge 2: Multi-Factor Authentication

**Time limit:** 5 minutes

**Prompt:**

Explain multi-factor authentication (MFA) to a product manager who is not
technical. Cover what factors exist, how TOTP (Time-Based One-Time Password)
works at a protocol level, and why MFA is considered essential for modern
applications.

**Evaluation criteria:**

- You enumerate the three factor categories: something you know, something you
  have, something you are
- You explain that MFA requires factors from at least two different categories
- You describe TOTP as deriving a code from a shared secret and the current
  time using HMAC, typically with a 30-second window
- You explain that MFA mitigates credential stuffing and phishing because a
  stolen password alone is insufficient
- Bonus: you discuss recovery codes and the trade-off between security and
  usability

**Common mistakes:**

- Calling two instances of the same factor "MFA" (e.g., password plus security
  question is still single-factor)
- Confusing TOTP with SMS-based codes and ignoring that SMS is vulnerable to
  SIM-swapping
- Stating that MFA prevents all account compromise
- Omitting the shared secret provisioning step (QR code scanning)

**Model answer:**

MFA requires a user to present evidence from at least two of three categories:

- **Knowledge factors** — something you know (password, PIN)
- **Possession factors** — something you have (phone, hardware token)
- **Inherence factors** — something you are (fingerprint, face)

Using two factors from the same category (password + security question) is not
true MFA because both are knowledge factors that can be compromised in a single
breach.

TOTP works as follows. During enrollment, the server generates a shared secret
and displays it as a QR code. The authenticator app stores this secret. To
generate a code, the app takes the current Unix time divided by 30 (the time
step), converts it to an 8-byte value, and computes an HMAC-SHA1 using the
shared secret as the key. A portion of the HMAC output is truncated to 6-8
digits and presented to the user. The server performs the same computation and
compares the result. Because time moves forward, the code changes every 30
seconds and remains valid only within a small window (typically ±1 step to
account for clock drift).

MFA is essential because passwords alone are vulnerable to phishing, credential
stuffing, and database leaks. Even if an attacker obtains your password, they
cannot authenticate without the second factor. This dramatically raises the
cost of account compromise.

### Challenge 3: Single Sign-On

**Time limit:** 5 minutes

**Prompt:**

Your company is adopting single sign-on (SSO) to allow employees to access
multiple SaaS tools with one corporate login. Explain to your team how SSO
works, what SAML and OIDC are and how they differ, and what security risks SSO
introduces.

**Evaluation criteria:**

- You identify the three roles: identity provider (IdP), service provider (SP),
  and user
- You explain that SSO delegates authentication to the IdP, which issues a
  token that the SP trusts
- You describe SAML as using XML assertions typically exchanged via browser
  redirects (HTTP POST or redirect bindings)
- You describe OIDC as an authentication layer built on top of OAuth 2.0,
  using JWTs and REST APIs
- You enumerate the risks: a compromised IdP grants access to all SPs,
  session hijacking at the IdP, and the blast radius of a single credential
- Bonus: you discuss Just-In-Time provisioning and SCIM for user lifecycle
  management

**Common mistakes:**

- Using SAML and OIDC interchangeably without acknowledging OIDC is better
  suited for mobile and single-page applications
- Ignoring the trust establishment step (metadata exchange or certificate
  registration)
- Forgetting to mention that SSO does not solve authorization, only
  authentication
- Omitting the logout problem (single logout or SP-initiated logout
  coordination)

**Model answer:**

SSO allows a user to authenticate once with an identity provider (IdP) and
access multiple service providers (SPs) without re-entering credentials. The
IdP is the authoritative source for authentication; the SP trusts the IdP's
assertions rather than managing its own credential store.

**SAML** (Security Assertion Markup Language) is an XML-based protocol. The IdP
produces a signed SAML assertion containing the user's identity and attributes.
This assertion is typically delivered via an HTTP POST from the user's browser
to the SP (POST binding) or via a redirect (redirect binding). SAML is mature
and widely used in enterprise environments, but its XML payloads are verbose
and its bindings can be complex to implement on mobile clients.

**OIDC** (OpenID Connect) is a modern authentication protocol built on OAuth
2.0. Instead of XML assertions, it uses JSON Web Tokens (JWTs) that contain
identity claims. OIDC flows use standard REST API calls and are significantly
simpler to implement on mobile and single-page applications. OIDC also supports
discovery (well-known OpenID configuration) and dynamic client registration.

**Key difference:** SAML handles authentication (who you are) and can carry
attribute data for authorization. OIDC authenticates the user via the ID token
(JWT) and can separately authorize access via OAuth 2.0 access tokens. OIDC is
the preferred choice for new implementations unless legacy enterprise
requirements dictate SAML.

**Risks:** SSO creates a single point of failure. If the IdP is compromised,
every SP that trusts it is vulnerable. Session hijacking at the IdP grants the
attacker access to all connected services. You also face the logout challenge:
signing out of the IdP does not guarantee the SP terminates its local session
unless single logout (SLO) is implemented. Finally, SSO covers authentication
only — authorization decisions remain the responsibility of each SP.

## Tier 2 — Design

These challenges ask you to design an authentication system for a specific
scenario. Each should take no more than 15 minutes. Start by clarifying
requirements, then scope the system, make key decisions, and sketch a design.

### Challenge 4: Auth for a 10 Million User Social App

**Time limit:** 15 minutes

**Prompt:**

Design the authentication system for a social media app targeting 10 million
monthly active users. The system must support password-based authentication
with MFA. Users access the app via mobile and web. The team has three
engineers.

**Scale estimates:**

- 10M MAU, approximately 3M daily active users
- Peak login rate: ~500 logins per second (assuming each user logs in once per
  day, concentrated in two peak hours)
- Each login involves credential verification, MFA challenge (if enabled), and
  session creation
- Token validation requests from other services: ~5,000 requests per second

**Key decisions you must make:**

- Password storage: bcrypt with cost factor 10-12 is appropriate. Argon2id is
  the modern alternative.
- Session format: JWT for stateless verification or opaque session ID with a
  Redis-backed session store. JWT reduces database lookups but complicates
  revocation.
- MFA approach: TOTP is the baseline. Push notifications (FCM/APNs) provide a
  better user experience. SMS should be a fallback only.
- Rate limiting: implement per-IP and per-account rate limiting on the login
  endpoint. Use a sliding window counter in Redis.
- Token refresh: issue short-lived access tokens (15 minutes) with longer-lived
  refresh tokens (7 days). Refresh tokens are stored and can be revoked.

**Model answer:**

The architecture follows a layered approach:

```
Client ──> Load Balancer ──> API Gateway ──> Auth Service
                              │
                              ├──> User Service (read user data)
                              ├──> Redis (rate limits, session cache)
                              └──> Database (user credentials, MFA secrets)
```

The auth service exposes three primary endpoints:

1.  `POST /auth/login` — accepts email and password, verifies the bcrypt hash,
    checks if MFA is enrolled, and if so, returns a challenge identifier. The
    client completes the challenge at `POST /auth/mfa`. On success, the server
    returns an access token (JWT) and a refresh token (opaque, stored in
    Redis).

2.  `POST /auth/refresh` — accepts a refresh token, validates it against Redis,
    and issues a new access token. The old refresh token is rotated
    (token rotation prevents replay).

3.  `POST /auth/logout` — invalidates the refresh token in Redis and adds the
    access token's JTI (JWT ID) to a denylist until its natural expiry.

For MFA enrollment, the user scans a QR code via the mobile or web client. The
provisioning URI follows the `otpauth://` scheme, and the server stores the
TOTP secret encrypted at rest (AES-256-GCM, key managed by a KMS).

Rate limiting is applied at the API Gateway using a token bucket per IP and a
separate bucket per account. After five failed attempts on one account, further
attempts are delayed by an exponential backoff.

**Why this works for 10M users:** bcrypt with a cost factor of 12 takes about
250ms per hash. With 500 logins per second and 10 auth service replicas, each
replica validates 50 logins per second, requiring ~12.5 seconds of CPU per
second — sustainable with modern hardware. Redis handles token validation at
microsecond latency.

### Challenge 5: Passwordless Login for a Mobile App

**Time limit:** 15 minutes

**Prompt:**

Design a passwordless authentication system for a mobile-only application. The
app targets 1 million users. Users authenticate using magic links sent via
email or biometrics (fingerprint/face) on the device. There is no password.

**Scale estimates:**

- 1M MAU, approximately 300K DAU
- Magic link requests: ~100 per second at peak (one per session start for
  email users)
- Biometric authentication is entirely client-side; the server only manages
  the cryptographic key pairs
- Token validation: ~2,000 requests per second

**Key decisions you must make:**

- Magic link flow: generate a cryptographically random token, store its hash in
  Redis with a 15-minute TTL, and include it in a deep link. When the user
  clicks the link, the client sends the token to the server, which verifies the
  hash and issues a session. The token is single-use.
- Biometric flow: use the WebAuthn or platform attestation (iOS App Attest /
  Android SafetyNet). The device generates a key pair. The private key never
  leaves the device's secure enclave. The public key is registered with the
  server. Authentication is a cryptographic challenge-response.
- Session tokens: use opaque bearer tokens stored in the iOS Keychain or
  Android Keystore.
- Account recovery: since there is no password, recovery relies on the user's
  email. A recovery magic link is sent, and the user re-enrolls biometrics on
  the new device.

**Model answer:**

**Magic link flow:**

1.  User enters their email address. The client calls `POST /auth/magic-link`
    with the email.
2.  The server generates a 32-byte random token using a CSPRNG. It stores
    `SHA-256(token)` in Redis with key `magic:{hash}` and a 15-minute TTL.
    The email contains a deep link: `myapp://auth/callback?token={token}`.
3.  The client intercepts the deep link, extracts the token, and calls
    `POST /auth/verify-magic-link` with the token.
4.  The server hashes the received token and looks it up in Redis. If found
    and not expired, it deletes the key (single-use), creates a user session,
    and returns an opaque bearer token.
5.  If the token is invalid or expired, the server returns an error and the
    client prompts the user to request a new link.

**Biometric flow:**

1.  During initial enrollment, the client calls `POST /auth/webauthn/register`
    to get a challenge from the server.
2.  The device's platform authenticator creates a key pair and returns the
    public key, key ID, and attestation to the server.
3.  The server stores the public key associated with the user's account.
4.  For subsequent logins, the client calls `POST /auth/webauthn/authenticate`
    to get a challenge. The platform authenticator signs the challenge with the
    private key (prompting the user for biometric verification). The signature
    is sent to the server, which verifies it against the stored public key.

**Why passwordless:** Eliminating passwords removes the risk of credential
stuffing, phishing (magic links contain the token in the URL itself, so
phishing pages cannot reuse them), and password reuse across services. The
trade-off is reliance on email delivery and device availability.

### Challenge 6: Enterprise SSO Integration

**Time limit:** 15 minutes

**Prompt:**

Design an enterprise SSO integration system that allows your B2B SaaS product
to connect with customer identity providers. The system must support both OIDC
and SAML 2.0, directory synchronization (SCIM), and Just-In-Time (JIT)
provisioning. Your platform has 500 enterprise tenants.

**Scale estimates:**

- 500 tenants, each with 100-10,000 users
- Peak authentication: ~200 SSO logins per second
- SCIM sync events: ~50 events per second (user creation, update, deactivation)
- Metadata storage: 500 IdP configurations (~10 KB each)

**Key decisions you must make:**

- Multi-tenant metadata store: each tenant has an IdP configuration record
  containing the issuer URL, certificate, SSO endpoint, and SCIM base URL.
  Store in a relational database with tenant ID as the partition key.
- IdP-initiated vs. SP-initiated SSO: support both. SP-initiated is more
  common for user-facing login; IdP-initiated is needed for app launcher
  portals.
- JIT provisioning: when a user authenticates via SSO for the first time,
  create a local user record automatically. Map claims from the IdP (email,
  groups, roles) to local roles.
- SCIM lifecycle: use SCIM 2.0 for push provisioning (IdP pushes user changes
  to your service). Implement idempotency keys to handle retries.
- Session mapping: the SSO session at the IdP may expire while the SP session
  is still active. Implement session lifetime coordination or accept that SP
  sessions are independent.

**Model answer:**

**Tenant onboarding flow:**

1.  Admin configures SSO via a setup wizard. Upload metadata XML (SAML) or
    enter the discovery URL (OIDC). The system validates the metadata and
    stores it under the tenant's ID.
2.  Admin configures attribute mapping: map IdP claims (e.g.,
    `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`) to
    local user fields. Provide a default mapping that covers most providers.
3.  Admin configures SCIM: provide your SCIM endpoint URL and bearer token to
    the IdP. Your SCIM implementation supports `/Users` and `/Groups`
    endpoints with standard CRUD operations.

**Authentication flow (SP-initiated, SAML):**

1.  User visits your login page and enters their email domain.
2.  The system looks up the tenant by domain and determines the IdP
    configuration.
3.  The system generates a SAML authentication request and redirects the user
    to the IdP's SSO URL.
4.  The user authenticates at the IdP. The IdP posts a SAML response (signed
    XML assertion) to your ACS (Assertion Consumer Service) endpoint.
5.  Your system validates the signature against the stored certificate, checks
    the audience and conditions, extracts the NameID and attributes, and either
    finds or JIT-provisions the user.
6.  A local session is created, and the user is redirected to the application.

**OIDC variant (steps 3-5):**

- Redirect the user to the IdP's authorization endpoint with your
  client_id and redirect_uri.
- The IdP redirects back with an authorization code.
- Your system exchanges the code for an ID token and access token at the
  token endpoint.
- Validate the ID token's signature (using JWKS from the discovery endpoint),
  extract the `sub` claim and other claims, and JIT-provision if needed.

**Security considerations:** Validate all signatures. Enforce audience
restrictions. Rotate your SP certificate regularly. Log all SSO attempts for
audit. Implement a kill switch per tenant to disable SSO if the IdP is
compromised.

### Challenge 7: Banking API Authentication with Regulatory Compliance

**Time limit:** 30 minutes

**Prompt:**

Design an authentication system for a banking API that must comply with
PCI-DSS, PSD2, and Strong Customer Authentication (SCA) regulations. The API
serves both retail customers and third-party providers (TPPs). Transactions
range from micro-payments to high-value wire transfers. The system must support
both human-initiated and automated (recurring) payments.

**Dimensions to explore:**

- **Multi-level assurance:** SCA requires at least two factors from different
  categories. For low-value transactions, you may use simplified
  authentication. For high-value transactions, require step-up authentication
  with additional verification.
- **TPP access:** PSD2 requires that third-party providers can access customer
  accounts via APIs. Authentication uses eIDAS certificates issued by National
  Competent Authorities. TPPs authenticate at the legal entity level, then
  request customer consent via a redirect flow.
- **PCI-DSS compliance:** The cardholder data environment (CDE) must be
  isolated. Authentication tokens used for API access must not expose PAN or
  other sensitive data. Use tokenization: replace the PAN with a payment token
  scoped to a specific merchant or transaction.
- **Session binding:** SCA requires that the authentication session is
  cryptographically bound to the transaction. The generated authentication code
  must include the transaction amount and the payee. This prevents a
  man-in-the-middle from modifying transaction details after authentication.
- **Recurring payments:** For standing orders and subscriptions, SCA is
  required only for the first transaction. Subsequent payments use a
  recurring payment reference tied to the initial SCA.

**Anti-patterns to avoid:**

- Treating all transactions with the same authentication level — this causes
  friction for micro-payments and insufficient security for wire transfers
- Storing raw PANs in the token store — violates PCI-DSS scope reduction
  requirements
- Using short OTP codes for high-value transactions without transaction
  binding — the code alone does not prove the user authorized the specific
  amount
- Allowing TPPs unlimited API access without per-consent scoping — violates
  PSD2's explicit consent requirement

**Model answer (one valid approach):**

The system uses a tiered authentication model:

**Tier 1 — Low value (up to 30 EUR):**
- Simplified authentication. A single factor (password or biometric) plus
  device recognition is sufficient.
- No SCA required under PSD2's exemptions for low-value payments.
- API access with a standard OAuth 2.0 bearer token, scoped to the
  transaction amount.

**Tier 2 — Standard value (30-500 EUR):**
- Full SCA required: possession factor (phone with TOTP or push approval) plus
  knowledge factor (PIN) or inherence factor (biometric).
- Transaction binding: the push notification displays the amount and payee.
  The user's approval includes a cryptographic hash of the transaction details.
- The API token is a JWT containing the `transaction_hash`, `amount`, and
  `payee` claims. The resource server verifies the binding before processing.

**Tier 3 — High value (over 500 EUR) or new payee:**
- Step-up authentication: all of Tier 2 plus a hardware security key (FIDO2)
  or a video call verification for very high amounts.
- Separate approval workflow: a second authorized user must confirm the
  transaction.
- Audit logging at the individual keystroke level for forensic analysis.

**TPP access flow:**

1.  TPP registers its eIDAS certificate with the bank. The bank validates the
    certificate chain against the NCA's root.
2.  Customer initiates payment via the TPP's app. The TPP redirects the
    customer to the bank's authorization page.
3.  Customer authenticates with SCA (Tier 2 or 3 depending on amount) and
    explicitly consents to the specific transaction.
4.  Bank generates a consent token scoped to the exact transaction. The TPP
    uses this token to call the payment initiation API.
5.  The token is single-use and expires after the transaction completes.

**PCI-DSS tokenization:**

- When a card is first used, the system generates a payment token via a
  tokenization service. The PAN is stored in a Hardware Security Module (HSM)
  and never touches the application servers.
- The token is scoped to a merchant + device pair. If the token is leaked, it
  cannot be used from a different device or for a different merchant.
- All authentication and authorization decisions happen outside the CDE. The
  CDE only receives the validated token and the transaction amount.

### Challenge 8: Global Auth System with Offline Support

**Time limit:** 30 minutes

**Prompt:**

Design an authentication system that works reliably in regions with
intermittent or no internet connectivity. The system must support user
authentication on a mobile device even when the server is unreachable, and must
handle conflicts when the device comes back online. Consider a scenario where
the system is used by field workers in remote areas who authenticate once in
the morning and work offline all day.

**Dimensions to explore:**

- **Offline credential storage:** The device must store credentials locally in
  a secure enclave (iOS Keychain, Android Keystore, or TEE). The private key
  never leaves the secure hardware. Biometric unlock gates access to the key.
- **Offline session tokens:** Issue a token with a long expiry (24 hours) that
  can be verified locally. This token is a signed JWT or a platform-signed
  attestation. The device caches the token and presents it to local services.
- **Conflict resolution:** When the device comes online, the server and device
  may have divergent state. Examples include remote credential revocation or
  privilege changes. The system must detect and reconcile these conflicts.
- **Clock skew:** Offline devices may have incorrect clocks. TOTP and JWT
  validation both depend on accurate time. Consider using a device's local
  clock drift estimate or GPS time signals.
- **Replay protection:** Offline tokens could be intercepted. Use
  challenge-response between local services and the auth token. Bind the token
  to the device ID during issuance so it cannot be used on another device.
- **Graceful degradation:** Define what operations are permitted offline vs.
  requiring a connection. High-risk operations (large money transfers, password
  changes) should require online verification.

**Anti-patterns to avoid:**

- Storing a password hash locally on the device — if the device is
  compromised, the attacker can brute-force the password offline
- Using long-lived static tokens with no mechanism for revocation — if a
  device is lost, the token remains valid until expiry
- Ignoring clock sync issues and having authentication fail for users in areas
  with poor network connectivity to NTP servers
- Implementing conflict resolution as "last-write-wins" without considering
  security implications — a revoked credential could be reinstated

**Model answer (one valid approach):**

**Registration and credential provisioning (online):**

1.  During initial setup with network access, the device generates a key pair
    inside the secure enclave. The public key is registered with the server.
2.  The server issues a signed certificate binding the public key to the user's
    identity and role. This certificate has a 24-hour validity and includes
    the device ID, user ID, role claims, and a timestamp.
3.  The device stores the certificate and its private key in the secure
    enclave. The enclave requires biometric verification before releasing a
    signature.

**Offline authentication flow:**

1.  User unlocks the device with biometrics. The secure enclave signs a local
    authentication challenge (a random nonce generated by the local service)
    using the stored private key.
2.  The local service (running on the device or a local gateway) verifies the
    signature against the cached certificate. If the signature is valid and the
    certificate is not expired, the user is authenticated.
3.  The local service issues a short-lived (5 minute) local session token to
    the requesting app. This token is scoped to the user's cached role.

**Online sync and conflict resolution:**

1.  When connectivity is restored, the device calls `POST /auth/sync` with its
    current state: the certificate ID, last-seen revocation list version, and
    any pending operations (e.g., a privilege escalation request made offline).
2.  The server computes the diff. If the user's credential was revoked while
    offline, the server returns a revocation record signed by the server's key.
    The device must accept this record and lock the user out.
3.  For privilege changes, the server's version always wins for security
    decisions. If an admin demoted a user's role while they were offline, the
    server returns the updated role claims, and the device must honor them.
4.  For non-security state (e.g., a list of transactions cached locally), use
    CRDT (Conflict-Free Replicated Data Types) or a version-vector approach.
    Each record has a vector clock. The server merges concurrent updates using
    last-writer-wins with an authoritative tiebreaker (the server wins for
    transactional data, the device wins for user-created data).

**Clock drift mitigation:**

- During online sync, the server includes its current timestamp. The device
  maintains a `skew = server_time - device_time` offset and applies it to all
  JWT validity checks.
- If GPS is available, use the GPS timestamp to calibrate. GPS satellites have
  atomic clocks and broadcast UTC time.
- If the device detects that its clock has been manually changed by more than a
  threshold (5 minutes), invalidate the local certificate and require online
  re-authentication.

**Revocation strategy:**

- The server maintains a certificate revocation list (CRL). Each CRL entry
  includes the certificate ID and a monotonically increasing version number.
- During sync, the device presents its last-seen CRL version. The server sends
  only entries newer than that version.
- For emergency revocation (stolen device), the server sets the CRL version to
  a high value and marks the certificate ID as revoked. On next sync, the
  device locks itself.

**Discussion of alternative approaches:**

An alternative is to avoid local certificates entirely and use an
asynchronous authentication model. The user authenticates once while online and
receives a batch of one-time-use tokens (like a ticket book). The device
consumes one token per local operation. This avoids the complexity of
certificate management and revocation but requires planning the number of
tokens needed and makes it harder to change privileges mid-batch.

Another approach uses a local authentication proxy that caches credentials and
replicates authentication decisions. This works well for field offices with a
local network but no internet connection. A local gateway handles
authentication within the LAN and syncs with the central server when
connectivity is available. This reduces the attack surface compared to
individual devices holding certificates but introduces a single point of
failure in the local gateway.

Each approach involves trade-offs between security, availability, and
complexity. The certificate-based model above favors strong security and
fine-grained revocation at the cost of implementing certificate lifecycle
management. The token-book model favors simplicity at the cost of flexibility.
The local proxy model favors centralized control at the cost of a
hardware-dependent single point of failure.
