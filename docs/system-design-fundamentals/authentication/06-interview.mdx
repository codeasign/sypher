## How This Topic Appears

Authentication questions in FAANG system design interviews test whether you understand identity management at scale. The topic can surface as the primary interview question, as a sub-component of a larger design, or as a debugging scenario embedded in a coding interview. Roughly one in four system design interviews includes an authentication-specific component, and dedicated authentication design questions are common at the senior staff level and above.

### Direct Questions

These questions test foundational knowledge. They often open an interview or appear as a quick screening question before the main design discussion.

1. "Explain how password hashing works. What makes bcrypt better than SHA-256 for storing passwords?"
   - Tests whether you understand the difference between cryptographic hashing and key derivation. The interviewer wants to hear about salting, work factor, and why fast hashes are dangerous for password storage.

2. "Walk me through the OAuth 2.0 authorization code flow with PKCE."
   - Tests protocol-level precision. Strong candidates draw the message sequence, name each endpoint (authorization, token, redirect), and explain why PKCE exists for public clients.

3. "What is the difference between authentication and authorization?"
   - A gate-check question. The candidate who cannot cleanly separate these two concepts will not pass the interview. Authentication verifies identity. Authorization grants permissions.

4. "How does a JSON Web Token work? What are its security properties?"
   - Tests understanding of token-based auth. The interviewer wants to hear about signature verification, expiry, the difference between JWS and JWE, and why JWTs should not store sensitive claims without encryption.

### Design Questions

These questions ask you to design an authentication system from scratch. They test architecture, scalability, and security reasoning.

5. "Design an authentication system for 100 million users."
   - The classic authentication system design question. Tests everything: credential storage, session management, token issuance, rate limiting, database schema, scaling strategy. The interviewer expects a complete architecture with specific technology choices.

6. "Design a single sign-on (SSO) system for a company with 50 internal applications."
   - Tests federated identity concepts. Strong candidates discuss SAML or OIDC, identity provider (IdP) architecture, session propagation, and the trade-offs between different SSO protocols.

7. "Design a multi-factor authentication (MFA) system. How do you handle TOTP and push notifications?"
   - Tests secondary factor design. The interviewer wants to hear about TOTP seed storage, push notification reliability, backup codes, and the user experience trade-offs of each MFA method.

8. "Design an API authentication system for a developer platform."
   - Tests API key management, HMAC signing, rate limiting per key, key rotation, and permission scoping. The interviewer evaluates whether you understand the difference between first-party and third-party API auth.

### Debugging Questions

These questions present a broken authentication scenario and ask you to diagnose and fix it.

9. "Users report that they are sometimes logged out unexpectedly. Some sessions work, some do not. What do you investigate?"
   - Tests operational reasoning. Strong candidates walk through session token expiration, clock skew between services, database failover, load balancer stickiness, and token refresh logic.

10. "Our authentication service is returning 5xx errors during peak traffic. Tracing shows database connection pool exhaustion. The auth database is a single PostgreSQL instance. How do you fix it?"
    - Tests capacity planning and caching strategy for auth data. The interviewer wants to hear about read replicas for session validation, caching with Redis, connection pooling tuning, and read-after-write consistency concerns.

11. "A security audit found that our password reset tokens are predictable. How do you fix the design?"
    - Tests secure token generation. Strong candidates discuss cryptographically secure random number generators, token entropy (minimum 128 bits), short expiry (under 15 minutes), single-use enforcement, and rate limiting on the reset endpoint.

### Trade-off Questions

These questions present two approaches and ask you to choose. They test your ability to reason about trade-offs under uncertainty.

12. "Session tokens versus JWTs — which do you choose for a new web application?"
    - The most common authentication trade-off question. The interviewer wants structured comparison: session tokens are simpler, revocable server-side, but require a session store. JWTs are self-contained, scale read operations, but cannot be revoked without a blocklist.

13. "Should we build our own authentication system or use Auth0 / Okta / Cognito?"
    - Tests build-vs-buy reasoning. Strong candidates analyze cost, compliance requirements, engineering team size, and time-to-market. At FAANG scale, most teams build their own. At startup scale, buy is the default.

14. "Passwordless authentication (magic links, passkeys) versus passwords — which is more secure?"
    - Tests understanding of modern auth trends. The interviewer wants to hear about phishing resistance (passkeys), device binding, user experience, and the migration challenge of moving existing users from passwords to passwordless.

15. "API keys versus OAuth 2.0 client credentials — which should a machine-to-machine service use?"
    - Tests understanding of service-to-service authentication. API keys are simpler but shared secrets. OAuth 2.0 client credentials grant provide scoped access with token expiry and rotation. The interviewer wants to hear about the trade-off between operational simplicity and security granularity.

16. "Password rotation policies reduce security. Defend or refute."
    - Tests awareness of current NIST guidelines. NIST SP 800-63B no longer recommends periodic password rotation. The candidate should explain that rotation encourages predictable patterns (Password1!, Password2!) and imposes user friction without measurable security benefit. Say "I enforce password rotation only when there is evidence of compromise, not on a fixed schedule."

## What the Interviewer Is Actually Testing

Each question type measures different signals. Understanding what the interviewer evaluates helps you prioritize your answer.

### Direct Questions: Precision and Depth

The interviewer tests whether you can explain authentication primitives with technical precision. For a question about password hashing, a weak candidate says "bcrypt is better because it is slower." A strong candidate says:

"bcrypt uses a Blowfish-based key derivation function with a configurable cost factor. Each hash includes a 16-byte salt and the cost factor, so the output is self-describing. The default cost of 10 means 2^10 = 1024 iterations of the Blowfish key schedule, which takes roughly 80ms on modern hardware. SHA-256 is designed for speed — it hashes at gigabyte-per-second rates on GPUs — which makes it trivial to brute force at scale. Password hashing algorithms must be slow. Bcrypt, scrypt, Argon2 all enforce this property."

The signal is depth: can you name the algorithm family, explain why it is slow, cite a concrete iteration count, and contrast it with the wrong choice.

For protocol questions (OAuth 2.0, OpenID Connect, SAML), the interviewer tests whether you understand the message flow and the security properties of each step. Walk the flow step by step. Name each endpoint. Explain why each message exists. If you skip the authorization code exchange and jump to the token, the interviewer notes the gap.

### Design Questions: Architecture and Scale

The interviewer tests whether you can design an authentication system that handles FAANG-level traffic. The key signals are:

- **Separation of concerns.** You do not put auth logic in the same service as business logic. You design a dedicated authentication service with its own database, its own caching layer, and its own rate limiting.
- **Session versus token strategy.** You state your choice and justify it with concrete reasoning about revocability, read load, and consistency requirements.
- **Credential storage.** You specify the hashing algorithm, the salt strategy, and the pepper (if any). You do not say "store passwords in a database" without describing how they are protected.
- **Failure modes.** You discuss what happens when the auth database is unreachable, when the session cache is cold, when a token expires during a multi-step operation.
- **Security boundaries.** You distinguish between requests that travel over the public internet and requests that stay within the data center. You do not use the same token type for both.

### Debugging Questions: Systematic Reasoning

The interviewer tests how you debug a production authentication issue. They do not expect you to guess the root cause from the description — they expect you to walk through a systematic investigation.

A strong debugging answer follows this structure:

1. "Let me start by confirming the symptoms. When does the logout happen? Is it session-wide or specific to certain features? Let me check the logs."
2. "I would check the session token expiry. If tokens expire at 24 hours and users are seeing logouts at 23 hours 55 minutes, clock skew between the application server and the auth server is the likely cause."
3. "Next, I would check whether the session cache is populated. If the Redis cluster that stores sessions has a failover event, the new primary might have cold cache. I would check the cache hit rate at the time of the reported issues."
4. "If the cache looks healthy, I would check whether the database is reachable. If the auth database is under load, session validation queries might time out, causing the auth service to reject valid tokens."

The signal is the structure, not the specific root cause. You must demonstrate that you know what to check and in what order.

### Most Common Mistakes Across All Question Types

Beyond the per-category signals, there are mistakes that candidates make consistently across all authentication interview questions. Avoid these.

**Mistake 1: Focusing only on the happy path.** You describe the login flow but never discuss what happens when the database is down, the cache is cold, the token expires mid-request, or the user enters the wrong password five times. The interviewer needs to see that you design for failure. Every security boundary you describe should include the failure case.

**Mistake 2: Hand-waving security properties.** You say "I will encrypt the passwords" instead of "I will hash them with bcrypt." You say "I will use a secure token" without specifying the bit length, the random source, or the encoding. The interviewer cannot evaluate whether your design is secure if you skip the implementation details. Security is in the specifics.

**Mistake 3: Not separating the read path from the write path.** Authentication has two distinct performance profiles: the write path (login, registration, password change) is CPU-bound and low-volume. The read path (session validation, token verification) is I/O-bound and high-volume. You must design and scale each path independently. A candidate who says "I will put the auth service behind a load balancer" without distinguishing these paths misses the central architectural insight.

**Mistake 4: Ignoring the user experience.** You design a perfectly secure system that requires users to authenticate every 5 minutes, remember 20-character passwords with special characters, and enroll MFA on every device. The system is secure but nobody uses it. The interviewer evaluates whether you balance security requirements with user experience. MFA should be required on new devices but trusted on known devices. Password policies should encourage length, not complexity. Session lifetime should match the application's usage pattern.

**Mistake 5: Treating auth as a binary (authenticated or not).** Real authentication systems have multiple states: unauthenticated, partially authenticated (password provided but MFA pending), fully authenticated, session expired, token revoked, credentials expired, account locked. Your design must handle each state. A candidate who models auth as a boolean flag misses the complexity of real-world auth flows.

### Trade-off Questions: Structured Comparison

The interviewer tests whether you compare options on the same dimensions. A strong trade-off answer defines the dimensions first, then compares each option on every dimension.

For session tokens versus JWTs:

"I evaluate this choice on four dimensions: revocability, read path latency, scaling cost, and client compatibility."

Then compare each dimension:

- "Session tokens are revocable by default — delete the session from the database and the token is useless. JWTs are not revocable in their standard form. You need a blocklist or short expiry with refresh tokens, which adds complexity."
- "For read path latency, JWTs win. The service validates the token signature locally without a database or cache lookup. Session tokens require a database or cache query on every request."
- "For scaling cost, session tokens require a shared session store, which becomes a bottleneck at very high throughput. JWTs are stateless — any server can validate any token without coordination."
- "For client compatibility, JWTs have broad library support across platforms. Session tokens require cookie support, which can be problematic for mobile apps and third-party integrations."

The signal is the dimensional framework itself. Candidates who jump to a conclusion without a structured comparison score lower even if they make the right choice.

## Level-by-Level Expectations

FAANG leveling determines the depth and scope expected in your answer. Here is what each level must demonstrate for authentication questions.

### L3 / L4 (Entry Level / Mid-Level)

**Expectation:** You understand the vocabulary. You can explain password hashing, session management, and basic token concepts. You know why authentication matters in system design but have not implemented an auth system at scale.

**What you must show:**
- Define authentication and explain why it is not the same as authorization.
- Explain bcrypt, salt, and work factor. Contrast with SHA-256.
- Describe the difference between session cookies and tokens.
- Draw a basic auth flow: user submits credentials, server validates, server issues a cookie or token, client includes it on subsequent requests.
- Know the difference between HTTP Basic Auth, Bearer tokens, and cookies.

**Sample answer to "How do you store passwords securely?" at L3/L4:**

"Passwords must never be stored in plain text. I would hash each password using bcrypt with a unique salt per user. The salt prevents rainbow table attacks, and bcrypt's configurable work factor makes brute force computationally expensive. When a user logs in, I retrieve the stored hash and salt, hash the provided password with the same salt, and compare. If they match, the password is correct."

**What is not expected at L3/L4:**
- You do not need to know OAuth 2.0 flow details.
- You do not need to design a multi-region auth system.
- You do not need to discuss federated identity protocols like SAML or OpenID Connect.
- You are not expected to quantify latency or throughput for auth operations.

### L5 (Senior)

**Expectation:** You can design an authentication system for moderate scale (millions of users) with MFA, SSO, and session management. You understand trade-offs between different approaches and can justify your choices with concrete reasoning.

**What you must show:**
- Design a complete auth flow with credential validation, session creation, token issuance, and session recovery.
- Explain how to add MFA (TOTP, SMS, push notification) to an existing auth flow.
- Describe SSO architecture and the role of the identity provider.
- Discuss session token expiry, refresh tokens, and token rotation.
- Compare session stores: PostgreSQL, Redis, Memcached. Choose and justify.
- Understand passwordless auth (magic links, WebAuthn) at a conceptual level.
- Identify the bottleneck in a naive auth design and propose a fix.

**Sample answer to "Design authentication for 10 million users" at L5:**

"I would design a dedicated authentication service with three storage layers. First, a PostgreSQL database stores user records and credential hashes. Second, a Redis cluster caches active sessions with a TTL of 24 hours. Third, a local in-memory cache on each auth service node caches the most recently validated sessions for sub-millisecond validation.

For the login flow: the client sends credentials to the auth service over TLS. The auth service verifies the password hash using bcrypt with cost factor 12. On success, it generates a session token — a 128-bit random value from a CSPRNG — and stores (session_id, user_id, expiry) in Redis. It returns the session token to the client as an HTTP-only, Secure, SameSite=Strict cookie.

If the user has MFA enabled, the auth service returns a signal requesting the second factor. The client submits the MFA code, the auth service validates it against the TOTP seed stored in the user record, then issues the session token.

For session validation on subsequent requests: the auth service or API gateway reads the session cookie, checks the local cache (hit in under 1ms), then checks Redis (hit in under 5ms), then falls back to PostgreSQL (miss in under 50ms). If the session is valid, the request proceeds. If invalid, the client must re-authenticate.

For refresh: the session token has a 24-hour TTL. The client can refresh before expiry by calling a refresh endpoint that extends the session by another 24 hours, up to a maximum of 7 days of continuous activity."

**What is not expected at L5:**
- You do not need to design a federated identity protocol from scratch.
- You do not need to handle multi-region active-active auth with global session replication.
- You are not expected to quantify the cost of auth operations in terms of dollars per authentication.

### L6 (Staff)

**Expectation:** You can design an identity provider (IdP) architecture. You handle federated authentication (OAuth 2.0, OIDC, SAML) as a first-class capability. You quantify throughput, latency, and cost. You design for multi-region deployment and zero-downtime credential rotation.

**What you must show:**
- Design the full OAuth 2.0 authorization server: authorization endpoint, token endpoint, redirect endpoint, refresh endpoint. Describe each with request and response structure.
- Support OpenID Connect on top of OAuth 2.0 for federated identity: the ID token as a JWT, the UserInfo endpoint, the discovery document.
- Handle multi-region session replication with trade-off analysis: asynchronous replication with stale reads versus synchronous replication with higher latency but no staleness.
- Quantify the cost of each auth operation: password verification (bcrypt cost factor 12 = 250ms per hash), session token generation (CSPRNG = under 1ms), JWT signing (RSA 2048 = 5ms), JWT verification (RSA 2048 = under 1ms).
- Design credential rotation with zero downtime: dual-key strategy where the new key signs new tokens while the old key still validates existing tokens.
- Discuss compliance constraints: SOC 2, GDPR right-to-deletion, FedRAMP for government customers, PCI-DSS for payment-adjacent auth.

**Sample answer to "Design an identity provider for a platform with 50 million users" at L6:**

"I would design a multi-region IdP with three planes: the control plane for configuration, the data plane for authentication, and the audit plane for compliance.

The control plane stores OAuth 2.0 client registrations, redirect URIs, allowed scopes, and signing keys in a globally replicated database. Changes propagate via change data capture to a configuration cache in each region within 5 seconds.

The data plane runs in each region independently. Each region has:
- Authorization servers that handle the authorization code flow, token issuance, and token refresh. These are stateless for authorization codes (stored in regional Redis with 10-minute TTL) and stateless for JWTs (no session store needed for self-encoded tokens).
- A credential verification service that validates passwords, TOTP codes, and WebAuthn assertions. This is the performance-critical path — password hashing at bcrypt cost 12 takes 250ms per verification, limiting each node to 4 verifications per second per CPU core. I would provision 50 nodes per region to handle 200 verifications per second with headroom.
- A session service that stores session metadata for long-lived sessions that need server-side revocability. Sessions are stored in regional Redis with async replication to a cross-region aggregate for global session views.

For multi-region operation: authorization codes are region-scoped with a 10-minute TTL. Tokens are self-encoded JWTs signed with a region-specific key. A global JWKS endpoint serves all region keys so any service in any region can verify any token. If a user authenticates in us-east-1 and their next request routes to eu-west-1, the JWT is verified locally using the JWKS cache refreshed every 5 minutes. No cross-region database call is needed.

For the quantifiable cost: each password verification costs 250ms of CPU time at bcrypt cost 12, translating to roughly 4 verifications per second per core. If we expect 1,000 logins per second at peak, we need 250 cores dedicated to password verification. At $0.10 per vCPU-hour on AWS, that is $25 per hour for the verification layer alone. JWT verification is nearly free by comparison — under 1ms per verification, so 1,000 verifications per second consumes roughly 1 vCPU.

For compliance: every authentication event — login attempt, token issuance, token refresh, token revocation, MFA enrollment — is written to an append-only audit log in a separate database. The audit log is immutable and retained for 7 years per SOC 2 requirements. GDPR deletion requests require deleting the user record from the credential store and tokenizing the audit log entries to remove personal data while preserving the event metadata."

**What is not expected at L6:**
- You are not expected to design the org-level auth strategy across an entire company.
- You are not expected to define auth policy as code at the executive level.

### L7 / L8 (Senior Staff / Principal)

**Expectation:** You define organizational authentication strategy. You evaluate auth infrastructure against business outcomes: user conversion rate, account takeover rate, compliance audit findings, engineering maintenance cost. You design auth systems that serve not just one product but the entire company.

**What you must show:**
- Define an auth strategy across all products in the organization. Centralized IdP versus per-product auth versus hybrid. Justify the choice with cost analysis and engineering productivity metrics.
- Design for zero-trust architecture. Every request from every source — internal or external — must authenticate. No implicit trust within the network boundary.
- Evaluate passkeys and WebAuthn as a replacement for passwords. Quantify the reduction in account takeover rate. Analyze the migration cost and user experience impact.
- Discuss regulatory compliance at the organizational level: FedRAMP, SOC 2 Type II, PCI-DSS, GDPR, CCPA. Design auth controls that satisfy the strictest applicable regulation.
- Analyze the cost of auth failures: account takeover costs (fraud losses, customer support hours, brand damage), authentication friction costs (users abandon signup if it takes more than 30 seconds), engineering maintenance costs (every authentication library version upgrade, every protocol change).
- Define SLAs for the auth system: 99.99% availability for authentication, 99.999% for token validation (read path), under 100ms p99 for login, under 5ms p99 for token validation.
- Design an auth migration plan: migrating from session tokens to JWTs, from passwords to passkeys, from a homegrown IdP to a commercial solution, all with zero downtime and no user-facing impact.

**Sample answer to "Define our company's authentication strategy for the next 3 years" at L7/L8:**

"I would organize the strategy around three pillars: user experience, security posture, and engineering efficiency.

For user experience: passwords are the largest source of authentication friction. Our signup conversion drops 15% when users must create a password, and our account recovery volume costs $2 million per year in support costs. I would prioritize passkey adoption: define a phased rollout starting with new users (passkey-only signup, password optional), then existing users (offer passkey enrollment during login), then password deprecation (two-year timeline to retire password authentication entirely). I would quantify the success metric as a 10% improvement in signup conversion and a 50% reduction in account recovery requests.

For security posture: we currently have 12 different authentication systems across 47 products. Each has different credential storage, session management, and MFA policies. This fragmented approach creates a large attack surface. I would consolidate onto a single corporate identity platform: one IdP, one credential store, one session management layer, one audit pipeline. The consolidation reduces our attack surface by eliminating 11 authentication implementations. The business case is: we spend $3 million per year maintaining 12 auth systems. Consolidation costs $5 million over two years but reduces maintenance cost to $500,000 per year after year two, for a three-year net savings of $4.5 million.

For engineering efficiency: our current authentication middleware is a monolith that crosses every product team's boundaries. Changes require coordinated releases across 12 teams. I would extract authentication into a dedicated platform team with four pillars: a credential service, a token service, an MFA service, and an audit service. Each service has a well-defined API, independent deployment, and independent scaling. Product teams integrate through the API and never touch auth infrastructure directly.

The three-year roadmap is: year one — consolidate onto a single IdP and begin passkey rollout. Year two — complete passkey adoption, deprecate passwords for 80% of users, begin zero-trust migration. Year three — achieve passwordless authentication across all products, full zero-trust architecture, FedRAMP Moderate certification."

## FAANG Framework for Authentication Questions

Use this six-step framework when answering any authentication system design question. It mirrors the general system design process but is specifically tuned for authentication.

### Step 1: Scope the Identity Requirements

Before designing anything, clarify who needs to authenticate, how many of them there are, and what security guarantees are needed.

**Questions to ask:**
- "How many users? What is the expected growth rate over the next 3 years?"
- "Is this a consumer-facing service, an enterprise service, or both?"
- "Do we need MFA? If yes, which factors — TOTP, SMS, push, WebAuthn?"
- "Do we support single sign-on with third-party identity providers (Google, Microsoft, Apple, SAML-based enterprise IdPs)?"
- "What is the password policy? Minimum length? Complexity requirements? Rotation policy?"
- "What compliance requirements apply? SOC 2, PCI-DSS, HIPAA, FedRAMP, GDPR?"
- "What is the session timeout? Can users stay logged in indefinitely on trusted devices?"
- "Do we need API access for third-party developers? If yes, are we building an OAuth 2.0 authorization server?"

**Output:** A written list of 5-7 clear assumptions. For example: "50 million users, growing to 200 million in 3 years. Consumer-facing with enterprise SSO as a future requirement. MFA optional but supported. Passwords required with bcrypt. Compliance: SOC 2 Type II. Session timeout: 24 hours with 7-day max refresh window."

### Step 2: Design the Credential Store

Design how credentials are stored, hashed, and verified. This is the most security-critical decision in the entire design.

**Key decisions:**
- Hashing algorithm: bcrypt with cost factor 12 as default. Argon2id if the team has experience with it. Never SHA-256, never MD5, never plain text.
- Salt: unique per user. Generate 16+ bytes from a CSPRNG. Store alongside the hash.
- Pepper (optional): a shared secret added to the password before hashing. Stored separately from the database — in a secret manager like AWS Secrets Manager or HashiCorp Vault. If the database is breached but the secret manager is not, the attacker cannot crack the hashes.
- Database schema: users table with (user_id, email, password_hash, salt, mfa_secret, created_at, updated_at). Index on email for login lookups.

**Authentication flow:**
1. Client sends email and password to auth service over TLS.
2. Auth service looks up the user by email. If not found, return generic error ("Invalid credentials") to avoid user enumeration.
3. Auth service retrieves the salt and pepper, re-computes the hash with the provided password.
4. Auth service compares the computed hash to the stored hash using a constant-time comparison function. This prevents timing attacks.
5. If match: proceed to session/token creation. If no match: increment failed attempt counter. Lock account after N consecutive failures (typically 5-10).

**Quantify the cost:**
- bcrypt cost 12: approximately 250ms per hash on modern CPU. One core handles ~4 verifications per second.
- At 1,000 logins per second peak: 250 cores dedicated to password verification.
- Optimization: hash verification is CPU-bound and embarrassingly parallel. Use a pool of worker threads or offload to a dedicated verification service that can scale independently.

### Step 3: Choose the Session and Token Strategy

Design the session management and token issuance system. This is where the trade-off decisions live.

**Decision tree:**

Is server-side revocation needed?
- Yes: Use session tokens (opaque random string stored in a server-side session store). The session is revocable by deleting the session record.
- No: Use JWTs (self-encoded tokens with claims). The token is verified locally without a server-side lookup.

Does the system need to support offline validation?
- Yes: Use JWTs. The token contains all information needed for validation. Any service in any region can verify it without calling home.
- No: Session tokens are simpler and safer for most use cases.

Is cross-region latency a concern?
- Yes for session tokens: Deploy regional session stores with local replication. Accept stale reads within a configurable window. Use a global session view for compliance use cases.
- Yes for JWTs: No concern. JWTs are validated locally. JWKS is distributed via a global CDN with 5-minute cache TTL.

**For session tokens:**
- Token format: 128-bit random value encoded as hex or base64url. Generated from a CSPRNG (crypto.randomBytes in Node.js, secrets.token_hex in Python, java.security.SecureRandom in Java).
- Session store: Redis cluster with replication factor 2 and 24-hour TTL. Local in-memory cache on each auth node as an L1 cache (1-minute TTL, LRU eviction, 10,000 entry limit).
- Session record: (session_id, user_id, created_at, expires_at, metadata_json).

**For JWTs:**
- Claims: sub (user_id), iss (issuer URL), aud (audience, the service that should accept this token), exp (expiry timestamp), iat (issued-at timestamp), jti (unique token ID for replay detection).
- Signing: RS256 (RSA 2048-bit key pair) for public-key verification. HS256 (HMAC-SHA256 with shared secret) for internal services that share a trust boundary.
- Key rotation: dual-key strategy. Active key signs new tokens. Previous key still validates existing tokens until expiry. Rotate keys every 90 days.
- JWKS endpoint: /.well-known/jwks.json serves the public keys. Cache by clients with a 5-minute TTL.

### Step 4: Design the MFA and Recovery Path

Design the multi-factor authentication system and account recovery. These are often the weakest points in an auth design.

**MFA options ranked by security:**
1. WebAuthn / Passkeys — phishing-resistant, device-bound. Best security, best user experience (biometric unlock).
2. TOTP — time-based one-time passwords. Good security. User experience: must scan QR code and enter 6-digit code.
3. Push notification — user taps approve on their phone. Good UX risk: push fatigue (user taps approve reflexively).
4. SMS — least secure (SIM swap attacks, SS7 vulnerabilities). Use only if no other option is available.

**MFA enrollment flow:**
1. User authenticates with password. Auth service checks MFA status.
2. If MFA not enrolled: redirect to enrollment page. Generate TOTP seed (16 bytes from CSPRNG, base32-encoded for QR display).
3. User scans QR code into authenticator app. User enters one TOTP code to confirm setup.
4. Auth service stores the TOTP seed encrypted at rest. Generates 5 backup codes (each 10 digits, single-use).
5. User confirms backup codes are saved. MFA is active.

**Account recovery flow:**
1. User requests password reset. Auth service sends email with a recovery token.
2. Recovery token: 128-bit random value from CSPRNG. Single-use. Expires in 15 minutes.
3. User clicks link, provides new password. Auth service verifies the token is valid, not expired, and not already used.
4. Auth service invalidates all existing sessions for this user. All devices must re-authenticate.
5. If the user had MFA: MFA remains enrolled. The user does not need to re-enroll.

**Rate limiting on recovery:**
- Maximum 3 recovery attempts per hour per user.
- Maximum 10 recovery emails per hour per IP address.
- Lockout after 5 failed recovery attempts across the entire user base within 5 minutes (possible brute force against recovery tokens).

### Step 5: Design for Scale and Failure

Design the auth system to handle load spikes, region failures, and security incidents.

**Scaling the credential verification path:**
- Password verification is CPU-bound. Scale horizontally by adding more verification nodes behind a load balancer.
- Each verification node uses a thread pool sized to the number of CPU cores. Async I/O for database lookups; synchronous blocking for bcrypt computation.
- If cost factor 12 makes verification too slow, reduce to cost factor 10 (approximately 60ms per hash) and accept a 3x increase in brute force efficiency. Quantify the trade-off.

**Scaling the session validation path:**
- Session validation is I/O-bound (cache lookup) or CPU-bound (JWT signature verification). Both scale horizontally.
- For session tokens: deploy Redis replicas in each availability zone. Prefer local reads over quorum reads for latency.
- For JWTs: no scaling concern. Signature verification is under 1ms on modern hardware. 1,000 verifications per second per core.

**Failure modes and mitigations:**
- Auth database is unreachable: authentication of existing users fails. New user registration fails. Active sessions continue to work if session cache is healthy. Mitigation: degrade to read-only mode where cached sessions still pass validation but no new logins or registrations are accepted.
- Session cache is cold: all requests must validate against the database. Mitigation: pre-warm the cache during deployment. Use a session cache with replication and automatic failover.
- IdP is unreachable during SSO: users cannot log in through third-party providers. Mitigation: offer direct credential authentication as a fallback for SSO-enabled accounts.
- Credential verification is slow under load: login latency spikes from 250ms to 2 seconds. Mitigation: queue verification requests with priority. Authentication requests from recently active users get higher priority than first-time logins.

**Security incident response:**
- Credential database breach: force password reset for all users. Invalidate all sessions. Notify affected users within regulatory timelines.
- Token signing key compromise: rotate keys immediately. Accept that tokens signed with the compromised key are valid until their expiry. If the risk is severe, invalidate all sessions and force re-authentication.
- Account takeover detected: lock the compromised account. Notify the user. Initiate recovery flow with additional identity verification (government ID upload, knowledge-based verification questions).

### Step 6: Wrap with Trade-offs and Future Work

Summarize the design in 30 seconds. Discuss what you would add with more time or at larger scale.

**Sample wrap:**

"To summarize, the authentication system uses bcrypt with cost factor 12 for password hashing, session tokens stored in a regional Redis cluster with a local in-memory cache for sub-millisecond validation, TOTP MFA with backup codes, and a dedicated authentication service that scales independently from application logic. The design handles 50 million users with 1,000 logins per second and 100,000 session validations per second.

Key trade-offs I chose: session tokens over JWTs because server-side revocation is a requirement for compliance. bcrypt over Argon2id because the team has more operational experience with bcrypt. Redis over DynamoDB for session storage because sub-millisecond read latency is critical for the session validation path, and Redis's single-threaded architecture is not a bottleneck at our session count.

For future work: add FIDO2/WebAuthn support for phishing-resistant authentication. Implement a session management dashboard for users to view and revoke active sessions. Design a federation layer for enterprise SSO via SAML and OIDC. Migrate from bcrypt to Argon2id as the team gains operational confidence."

## Common Interview Scenarios

Authentication questions often take unexpected turns during the interview. Here are four common scenarios and how to handle them.

### Scenario 1: The Interviewer Pushes Back on Your Token Choice

You have chosen session tokens. The interviewer says: "Are you sure about that? JWTs seem like the better choice here."

How to handle it: The interviewer is testing whether you can defend your choice or update your design based on new information. They are not telling you that you are wrong — they are probing your reasoning.

**Option A — Defend with reasoning:** "I chose session tokens because server-side revocation is a hard requirement. If a user's account is compromised, I need to invalidate all active sessions immediately. With session tokens, I delete the session record from Redis and the token is dead. With JWTs, even with short expiry, a compromised token remains valid for up to 15 minutes — and in a security incident, 15 minutes is enough for an attacker to exfiltrate significant data. If the revocation requirement can be relaxed to a 15-minute window, then JWTs become viable, and I would re-evaluate."

**Option B — Acknowledge and adjust:** "You raise a valid point. JWTs would reduce session store read load and simplify cross-region validation. If the business requirement is that session revocation within 15 minutes is acceptable rather than immediate, I would switch to JWTs. The design changes would be: issue RS256-signed JWTs with a 15-minute expiry, add a JWKS endpoint for key distribution, implement a refresh token with 30-day expiry and rotation, and add a blocklist for emergency revocation with a 15-second propagation delay across regions."

The strongest signal is that you do not view your design as fixed. You have a preferred approach but can pivot cleanly when the requirements change.

### Scenario 2: The Interviewer Asks You to Add a Feature Mid-Design

You are halfway through designing the authentication system. The interviewer says: "Actually, we also need to support enterprise SSO through SAML. How does that change your design?"

How to handle it: Do not panic. This is a positive signal — the interviewer is testing whether your design is extensible. A good design already has natural extension points for federation.

"I would add a federation layer on top of the existing credential authentication. The architecture becomes:

1. A federation gateway that accepts SAML assertions from enterprise identity providers. The gateway validates the SAML response (checks the assertion signature against the IdP's public certificate, validates the audience and recipient, checks the timestamp for freshness, and extracts the user's email and attributes).

2. A user linking service that maps the SAML assertion's NameID or email to an existing user account or creates a new account on first login. The mapping is stored in a new table: federated_identities (idp_entity_id, name_id_format, name_id_value, user_id).

3. The existing session and token infrastructure remains unchanged. After the federation gateway validates the SAML assertion and links the user, it issues the same session token or JWT as the credential-based flow. The downstream services do not need to know whether the user authenticated via password or SSO.

4. For the user experience: the login page shows a 'Sign in with SSO' button that redirects to the enterprise IdP. After the SAML response is received, the user is redirected back to the application with the same session cookie as any other login.

The key design constraint is that the federation gateway must be stateless for SAML responses (the SAML response contains all needed information) and must handle IdP-initiated SSO (where the user starts from the IdP's portal, not from the application)."

### Scenario 3: The Interviewer Challenges Your Security Posture

You have described your credential storage design. The interviewer says: "What if the database is breached? How do you protect users?"

How to handle it: This is a stress test. The interviewer wants to see whether you have thought about defense in depth, not just the happy path.

"I design for database breach as a known threat model, not a hypothetical. The defenses are layered:

Layer 1 — Password hashing: bcrypt with cost factor 12 and a unique salt per user. If the database is breached, the attacker has 250ms-per-hash work to crack each password. At 1,000 hashes per second per GPU, cracking a 12-character random password takes centuries. Weak passwords (dictionary words, common patterns) are still vulnerable, which is why we enforce a minimum length of 12 characters and check against a common password dictionary at registration.

Layer 2 — Pepper: a shared secret stored in the secrets manager, not in the database. The pepper is concatenated with the password before hashing. If the database is breached but the secrets manager is not, the attacker cannot compute hashes at all — they have the hash output but not the input that produced it.

Layer 3 — Encryption at rest: the credential database uses AES-256 encryption. Even if an attacker obtains a database backup or a disk snapshot, they cannot read the data without the encryption key.

Layer 4 — Breach response plan: if we detect a breach, we immediately force password reset for all users. We invalidate all active sessions. We notify affected users within the regulatory timeline (72 hours for GDPR). We rotate the pepper. We rotate the database encryption key. We audit the access logs to determine the breach vector.

Layer 5 — Monitoring and alerting: we monitor for anomalous database access patterns. A query that reads the entire password_hash column is an alert triggers immediate investigation. Read replicas have restricted access — only the auth service should read credential hashes, and only during login."

### Scenario 4: You Are Running Out of Time

You have 10 minutes left and you are still describing the OAuth 2.0 authorization server design. The interviewer has not seen your trade-offs or wrap.

How to handle it: Stop the current discussion immediately. Say: "I want to make sure I have time to cover the key trade-offs and a summary. Let me note the remaining design points for the authorization server and move to the trade-off analysis."

Then present 2-3 trade-offs in 3 minutes:

"First trade-off: authorization code vs implicit flow. I chose the authorization code flow because the implicit flow exposes the access token in the URL fragment and has no client authentication at the token endpoint. The code flow with PKCE is the current best practice for all client types, including single-page applications.

Second trade-off: opaque refresh tokens vs rotating JWTs. I chose opaque tokens stored server-side because they are revocable. The trade-off is that each refresh requires a database lookup, but at 10,000 refreshes per second, that is a manageable load on PostgreSQL.

Third trade-off: per-region signing keys vs a global signing key. I chose per-region keys to limit the blast radius of a key compromise. The trade-off is that the JWKS endpoint must aggregate keys from all regions, adding a 5-minute propagation delay for new keys."

Then deliver a 2-minute wrap summarizing the entire design.

A partial design with trade-offs and wrap scores higher than a complete design description with no conclusion.

## Red Flags

These statements reveal gaps in your authentication knowledge. Avoid them in interviews.

**Red Flag 1:** "I store passwords encrypted in the database."

Encryption is reversible. If the encryption key is compromised, all passwords are exposed. Authentication requires hashing, not encryption. Hashing is one-way: you cannot recover the original password from the hash. If the database is breached, hashed passwords are not immediately useful to the attacker. Say "I hash passwords with bcrypt and a unique salt per user" instead.

**Red Flag 2:** "I use SHA-256 for password hashing because it is secure."

SHA-256 is a cryptographic hash function, not a password hashing function. It is designed for speed — modern GPUs compute billions of SHA-256 hashes per second. Password hashing algorithms must be deliberately slow. Bcrypt's cost factor, scrypt's memory hardness, and Argon2's CPU/memory parameters enforce this property. Say "I use a key derivation function designed for password hashing, such as bcrypt, scrypt, or Argon2id."

**Red Flag 3:** "I will store the JWT in localStorage."

JWTs in localStorage are accessible to any JavaScript running on the same origin. If the application has any XSS vulnerability, the attacker reads the token from localStorage and gains full access. HTTP-only cookies are not accessible to JavaScript, preventing XSS-based token theft. Say "I store session tokens or JWTs in HTTP-only, Secure, SameSite=Strict cookies. For mobile apps, I use the platform's secure storage (iOS Keychain, Android Keystore)."

**Red Flag 4:** "I will use JWTs because they are stateless and scale perfectly."

JWTs are stateless in theory but not in practice for most real-world applications. You need a blocklist for revoked tokens, a refresh token that is stateful, or very short token lifetimes (under 5 minutes) that degrade user experience. Additionally, JWT verification is not free — asymmetric signature verification adds 1-5ms per request. And JWTs cannot be revoked unless you introduce server-side state, which defeats the stateless purpose. Say "JWTs reduce session store load but require careful design around revocation, key rotation, and token lifetime."

**Red Flag 5:** "I will not implement MFA because it adds friction."

MFA is not optional for any production auth system. It is the single most effective control against account takeover. The user experience impact is manageable: most users enroll MFA once and authenticate with the second factor only on new devices or after session expiry. At FAANG, MFA is a baseline requirement, not a feature toggle. Say "I will support TOTP as the default MFA method and WebAuthn for users who want the strongest protection. The user experience overhead is one additional step on first login from an unrecognized device."

**Red Flag 6:** "I use the same token for API access and web session."

API tokens and web session tokens have different security properties and should be different token types. Web session tokens are short-lived (hours), associated with a browser session, and should not have access to sensitive API operations. API tokens are long-lived (months), associated with a developer or service account, and scoped to specific permissions. Say "Web sessions use cookies with session tokens. API access uses bearer tokens with scoped permissions and longer lifetimes. They are issued from different endpoints with different security policies."

**Red Flag 7:** "I will build password complexity requirements (uppercase, lowercase, digit, special character)."

Password complexity rules are counterproductive. NIST SP 800-63B recommends password length over complexity. A 16-character passphrase is more secure and more memorable than "P@ssw0rd!". Complexity rules encourage users to choose predictable patterns (capitalize the first letter, add a digit and an exclamation mark). Say "I enforce a minimum password length of 12 characters and check against a dictionary of common passwords. I do not require character class diversity. Instead, I encourage passphrases and use password strength meters to guide users."

**Red Flag 8:** "I will not rate limit the login endpoint because legitimate users might be blocked."

Rate limiting is non-negotiable for login endpoints. Without rate limiting, credential stuffing attacks run indefinitely. At FAANG scale, credential stuffing is a constant, automated threat — millions of login attempts per hour from botnets. The design must absorb these attacks without affecting legitimate users. Say "I rate limit per user (10 attempts per minute), per IP (100 attempts per minute), and globally (50,000 attempts per minute). I use a sliding window algorithm in Redis for accurate counting. Legitimate users rarely exceed 2-3 login attempts, so these limits have minimal impact on the normal flow."

**Red Flag 9:** "I will not build password reset because it is just an email link."

Password reset is one of the most attacked flows in any authentication system. If done incorrectly, it bypasses all other security controls. The reset token must be cryptographically random (128+ bits from a CSPRNG), single-use, time-limited (under 15 minutes), and bound to a specific user account. The reset email must not reveal whether the account exists. The reset endpoint must be rate limited to prevent brute force of the reset token. Say "Password reset requires a cryptographically secure token, single-use enforcement, 15-minute expiry, rate limiting on the reset endpoint, and a generic email message that does not reveal whether the account exists."

**Red Flag 10:** "I will let users change their email without re-authentication."

Email change is a security-sensitive operation. If an attacker gains access to a session, they could change the email and initiate a password reset to permanently lock the legitimate user out. Email changes must require the user's current password, and if MFA is enabled, require an MFA challenge as well. The old email must be notified of the change. Say "Email changes require password confirmation, MFA challenge if MFA is enabled, and email notification to both the old and new email addresses. The change takes effect immediately but the user can revert it within 24 hours by contacting support."

**Red Flag 11:** "I will not worry about timing attacks because the network latency is higher than the computation time."

Timing attacks are not theoretical. While network jitter adds noise, an attacker can average thousands of measurements to extract information. Constant-time comparison removes the signal entirely. The difference between a constant-time comparison and a naive string comparison is one line of code. There is no excuse to skip it. Say "I use a constant-time comparison function for all security-sensitive comparisons: password hashes, session tokens, recovery tokens, API keys. The implementation is `crypto.timingSafeEqual` in Node.js, `hmac.compare_digest` in Python, or `MessageDigest.isEqual` in Java."

**Red Flag 12:** "I will validate sessions on every request by querying the database."

Session validation on every request against the database does not scale. At 100,000 requests per second, the database connection pool is exhausted within seconds. Session validation must be a cache-hit operation: in-memory cache first, Redis second, database only as the last resort. Or use JWTs that require no server-side state. Say "Session validation uses a two-tier cache: local in-memory LRU cache (under 1ms) with Redis as the shared cache (under 3ms). The database is the fallback for cache misses and should only serve a small fraction of requests — under 1% cache miss rate at steady state."

**Red Flag 13:** "I will use the same session store for web sessions and MFA challenge state."

MFA challenge state is short-lived (under 10 minutes), high-turnover (each code attempt creates and destroys state), and sensitive (if compromised, the attacker can bypass the second factor). Intermixing it with long-lived session data in the same Redis instance creates cross-contamination risk: a cache eviction caused by MFA state churn could prematurely expire a session, or a session cache miss during MFA could cause the wrong user to be authenticated. Say "I separate MFA challenge state into a dedicated Redis namespace with its own TTL (10 minutes). It never shares a cache key prefix or an eviction policy with the session store."

**Red Flag 14:** "I will not encrypt the database because it is behind our firewall."

Database encryption at rest is not optional for any system that stores credentials or session data. The firewall protects against external access but does not protect against a compromised internal service, a backup tape that is lost in transit, or a database snapshot that is copied for debugging. Most compliance frameworks (SOC 2, PCI-DSS, HIPAA) require encryption at rest. Say "The credential database uses AES-256 encryption at rest with automated key rotation through the cloud provider's key management service. Backups are also encrypted. The encryption keys are stored in a separate hardware security module with access logging and audit trails."

### Drill 5: The 10-Minute Deep Dive — "Design an OAuth 2.0 Authorization Server"

Set a timer for 10 minutes. Design an OAuth 2.0 authorization server for a developer platform. This is the most common advanced authentication design question at the Staff level and above.

**Your answer should cover:**

**Step 1 — Requirements (1 minute):**
- The platform has 10,000 third-party developer applications.
- Each application has 1,000 to 10 million end users.
- Supports the authorization code flow. PKCE required for mobile apps.
- Token types: access token (JWT, 1-hour expiry), refresh token (opaque, 30-day expiry, rotation).
- Scopes: read, write, admin. Each application requests specific scopes during authorization.
- Compliance: the authorization server must log every authorization and token issuance for SOC 2 audit.

**Step 2 — Client registration (1 minute):**
- Developer registers their application and receives a client_id and client_secret.
- Client_id is public (used in the authorization redirect). Client_secret is confidential (stored server-side by the application).
- Registration includes: redirect_uris (allowlist), allowed scopes, application type (public or confidential).
- Storage: client_registrations table in PostgreSQL with (client_id, client_secret_hash, redirect_uris, allowed_scopes, created_at, updated_at).

**Step 3 — Authorization code flow (3 minutes):**
1. Client redirects user to authorization endpoint: GET /authorize?response_type=code&client_id=APP_ID&redirect_uri=https://app.com/callback&scope=read&state=XYZ.
2. Authorization server authenticates the user (redirect to login page, standard auth flow).
3. After authentication, user sees consent screen: "App XYZ wants read access to your profile. Allow?"
4. User approves. Authorization server generates an authorization code: 128-bit random value from CSPRNG, 10-minute TTL, stored in Redis with (code, client_id, redirect_uri, user_id, scopes, code_challenge, code_challenge_method).
5. Authorization server redirects to the application's redirect_uri: https://app.com/callback?code=AUTH_CODE&state=XYZ.
6. Application sends the authorization code to the token endpoint: POST /token with grant_type=authorization_code, code=AUTH_CODE, redirect_uri=https://app.com/callback, client_id=APP_ID, client_secret=APP_SECRET (for confidential clients), or code_verifier=VERIFIER (for PKCE).
7. Authorization server validates: code exists, code is not expired, redirect_uri matches, client_id matches, client_secret matches (if applicable), code_verifier matches code_challenge (if PKCE). If valid, deletes the code (single-use), generates tokens, returns: {access_token, token_type, expires_in, refresh_token, scope}.

**Step 4 — Token structure (1 minute):**
- Access token: JWT signed with RS256. Claims: sub (user_id), client_id, scope, exp (1 hour), iat, jti, iss.
- Refresh token: opaque random string, 128 bits, stored in PostgreSQL with (refresh_token_hash, user_id, client_id, scope, expires_at, rotation_count). Hashed on storage. 30-day expiry.
- Token revocation: access tokens self-expire (1 hour, no server-side state). Refresh tokens are revocable by deleting the database record.

**Step 5 — Token refresh (1 minute):**
- When access token expires, application calls POST /token with grant_type=refresh_token, refresh_token=REFRESH_TOKEN.
- Authorization server looks up the refresh token hash in database. Validates expiry, checks if token has been used (rotation_count).
- If valid: issues new access token and new refresh token. Increments rotation_count. Old refresh token is invalidated (rotation prevents replay).
- If invalid (token already rotated): the previous refresh token might have been stolen. Invalidate all refresh tokens for this client+user combination. Require re-authorization.

**Step 6 — JWKS endpoint and key rotation (1 minute):**
- Authorization server exposes GET /.well-known/jwks.json returning the public keys for token verification.
- Two keys always present: current (use for signing) and previous (use for verification of existing tokens).
- Keys are 2048-bit RSA pairs. Current key rotates every 90 days. On rotation: new key becomes current, old key moves to previous, the oldest key is removed from the JWKS response.
- Private keys stored in a hardware security module (HSM) or a secrets manager with access logging.

**Step 7 — Scale and failure (1 minute):**
- Authorization code storage: Redis cluster with replication. Codes have 10-minute TTL and are deleted on use. At 10,000 codes per minute, storage requirement is under 10 MB.
- Token signing: RSA 2048 signing takes ~5ms per token. If we issue 1,000 tokens per second, we need ~5 cores for signing. Scale by adding more signing nodes behind a load balancer.
- Token verification: RSA 2048 verification takes ~0.5ms. Third-party services cache the JWKS response. Verification is not a bottleneck.
- Refresh token storage: 10 million active refresh tokens × 256 bytes per record = 2.5 GB. PostgreSQL handles this comfortably.
- Failure mode: if the authorization server is unreachable, existing access tokens continue to work until expiry. Users cannot authorize new applications. Mitigation: deploy the authorization server in multiple regions with active-active configuration. Each region has its own key pair. Tokens verify globally via the aggregated JWKS endpoint.

**After the drill, check:** Did you cover all seven steps? Did you explain the PKCE flow? Did you mention refresh token rotation? Did you discuss the JWKS endpoint and key rotation? Did you handle failure modes? Did you quantify storage and throughput? Did you finish in 10 minutes?
