# Durability in System Design Interviews

Durability questions test whether you understand how data survives failures — not just in theory, but in the concrete decisions you make during system design. Interviewers probe durability to distinguish candidates who vaguely know about "replication" from candidates who can explain exactly when data is at risk and what prevents loss.

---

### Additional High-Signal Phrases

These phrases demonstrate deeper expertise during interviews:

**For WAL and fsync depth:**
- "PostgreSQL's WAL records are stored in 16MB segments named by timeline ID and LSN. The `pg_wal` directory can be inspected to find exactly which transactions are committed. During recovery, PostgreSQL replays from the last checkpoint's redo point. The typical recovery speed is about 100MB of WAL per second on modern hardware."

**For distributed durability:**
- "DynamoDB's synchronous replication across three AZs means every write is durably stored on three separate facilities before the client receives a success response. The write happens in parallel — the leader sends to all followers simultaneously and waits for the slowest one. This is why DynamoDB write latency is typically 5-15ms."

**For backup strategy:**
- "We follow the 3-2-1 rule with an extension: the third copy should be in a different geographic region, ideally managed by a different team with different credentials. This protects against both natural disasters and insider threats."

**For operational verification:**
- "We measure three metrics for backup health: backup success rate (must be 100%), restore success rate (tested quarterly), and RTO achievement rate (restore completes within the target window). A backup that succeeds but takes 8 hours to restore fails the RTO requirement."

**For edge cases:**
- "The most dangerous durability scenario is not a catastrophic failure — it is partial failure combined with silent corruption. A single bit flip in a backup file that is not detected until restore time can cause days of data loss. This is why we do scrubbing with checksums and automated restore testing."

### Additional Red Flags

Beyond the eight core red flags, interviewers also notice these subtler issues:

**Red Flag 9: Treating all data the same**
Candidate proposes the same durability strategy for all data, regardless of value. A system that applies synchronous replication to session cache data wastes resources. A system that uses the same backup frequency for audit logs and analytics data may miss compliance requirements.

**Red Flag 10: No recovery time estimate**
Candidate describes the backup strategy but cannot estimate how long recovery would take. If you cannot say whether recovery takes 15 minutes or 15 hours, you have not designed for operations.

**Red Flag 11: Ignoring the human factor**
Candidate designs an elaborate automated durability system but provides no process for what happens when an alert fires at 3 AM. Who gets paged? What do they check? What is the escalation path?

**Red Flag 12: No cost awareness**
Candidate proposes a multi-region, multi-cloud, synchronous replication architecture for a system that stores 10GB of data with a USD 500/month budget. Durability decisions must account for business constraints.

## The Durability Question Spectrum

Durability appears in three forms during system design interviews.

**Form 1 — Implicit Durability Requirement (most common)**

The interviewer does not mention durability. You must infer it from the use case. A payment system, a medical records system, a file storage service — these all require strong durability. Identify this early and state your durability assumptions explicitly.

```
"How would you design a photo-sharing application?"

Candidate who misses durability: "We store photos on S3 and use
a PostgreSQL database for metadata. We use Redis for caching."

Candidate who covers durability: "For the photo storage layer,
we use S3 which provides 11 nines durability through automatic
replication across three availability zones. For metadata, we
use PostgreSQL with synchronous replication to a cross-AZ replica
and continuous WAL archiving to S3 for point-in-time recovery."
```

**Form 2 — Explicit Durability Discussion**

The interviewer asks directly: "How do you ensure data is not lost?" Or: "What happens if the database crashes mid-write?"

```
Expected structure:
1. Write-ahead log + fsync for crash recovery
2. Replication strategy (sync vs async, replication factor)
3. Backup strategy (frequency, type, retention)
4. Consistency during failures (torn pages, partial writes)
5. Monitoring and verification (checksums, restore tests)
```

**Form 3 — Durability vs Availability Tradeoff**

The interviewer introduces a constraint: "We need high write throughput but cannot lose data." Or: "Write latency must be under 10ms."

```
The tradeoff framework:
- Durability forces synchronous writes (fsync, quorum)
- Synchronous writes add latency
- Latency reduces throughput (connections blocked on fsync)
- Mitigations: group commit, batching, durable hardware

Signal of seniority: explaining the specific latency numbers
and knowing which configurations work for each scenario.
```

## The Durability Interview Framework

Use this framework when durability comes up in any system design interview.

### Step 1: Identify Durability Requirements

```
Questions to ask the interviewer:
- "What is the data loss tolerance? Can we lose any data, or does
   every write need to survive?"
- "Are there regulatory requirements for data retention or backup?"
- "What is the expected RPO (Recovery Point Objective)?"
- "What is the expected RTO (Recovery Time Objective)?"
- "Is this a new system or replacing an existing one with data?"
```

### Step 2: Map Requirements to Mechanisms

```
Zero data loss (RPO = 0):
  - WAL + fsync per transaction
  - Synchronous replication (min 1 replica)
  - Battery-backed cache on storage hardware
  - Cross-region point-in-time recovery

Sub-second data loss (RPO under 1s):
  - WAL with group commit (fsync every 5-10ms)
  - Asynchronous replication
  - Continuous WAL archiving

Minute-level data loss (RPO under 1 hour):
  - Periodic fsync (every 30s)
  - Daily snapshots
  - Regular backups

Hour+ data loss:
  - No synchronous write guarantee
  - Periodic snapshots only
  - Data ingested via batch process
```

### Step 3: Describe the Write Path

Walk through a single write from client to durable storage.

```
"I'll walk through the write path for a key-value store:

1. Client sends PUT(key=photo_123, value=image_bytes) to the API server
2. API server validates and forwards to the database primary
3. Database begins a transaction and appends to the write-ahead log:
   WAL record: {LSN=1000, TID=42, op=PUT, key=photo_123, value=...}
4. Database calls fsync() on the WAL file descriptor
   - This blocks until the storage device confirms data is on stable media
   - On enterprise SSD: 2ms average fsync latency
5. Database writes the key-value pair to the memtable or data page
6. Database writes COMMIT record to WAL and fsyncs again
7. ACK returns to the client: "your data is permanently stored"
8. WAL entry is shipped to replicas via streaming replication
9. At the replica, the same WAL entry is replayed locally
10. Replica fsyncs and sends ACK back to primary

If any step fails before fsync: the transaction is rolled back
If any step fails after fsync: the data survives crash recovery
"
```

### Step 4: Address Failure Scenarios

```
"If the database crashes just after step 4 (WAL fsync) but before
step 7 (ACK to client):

On restart, the database replays the WAL:
- It sees TID=42 with a PREPARE record but no COMMIT record
- The transaction was NOT committed (no COMMIT in WAL)
- The data pages may or may not have the write applied
- Recovery: the transaction is rolled back (UNDO processing)
- Result: client never received ACK, transaction did not commit,
  no data was lost — the client can safely retry

If the crash happens after step 7 (ACK sent but not received):
- Database has committed the transaction
- WAL contains COMMIT for TID=42
- Recovery: REDO phase replays the COMMIT
- Data is present in the database
- Client never received ACK due to network failure
- Client retries with idempotency key → database detects duplicate
  and returns the stored result without reapplying the mutation"
```

### Step 5: Discuss Monitoring and Verification

```
"How do we know our durability mechanisms are working?

1. WAL monitoring:
   - WAL write latency (p50, p99) — alert if > 10ms
   - WAL volume growth rate — alert if exceeds projection by 50%

2. Replication monitoring:
   - Replication lag — alert if > 5 seconds
   - Number of in-sync replicas — alert if below replication factor
   - Replica connection status — alert on disconnect

3. Backup monitoring:
   - Backup success/failure — alert on any failure
   - Backup age — alert if most recent backup is older than 24 hours
   - Restore test success — quarterly automated validation

4. Integrity monitoring:
   - Checksum errors — alert on any checksum mismatch
   - Scrub completion — ensure every data page is checked within 30 days
   - Hardware health — disk SMART metrics, controller errors"
```

## Phrase Bank: 15+ Interview Phrases

Use these phrases to demonstrate depth during interviews.

### For Explaining Durability Mechanisms

1. "The WAL is written sequentially, which is an order of magnitude faster than random I/O. The fsync at commit time is the point of no return — before fsync, data is at risk; after fsync, crash recovery can reconstruct any committed transaction."

2. "PostgreSQL's `full_page_writes = on` ensures that after a checkpoint, the first modification to each data page is recorded as a full page image in the WAL. This prevents torn pages during crash recovery."

3. "Group commit batches multiple transactions into a single fsync call. Instead of 5ms per transaction, you spend 5ms total for a batch of 50 transactions — a 50x throughput improvement."

4. "Enterprise SSDs with power-loss protection have capacitors that provide enough energy to flush the write cache during a power interruption. Consumer SSDs without this protection can lose data even after fsync returns success."

5. "In a distributed system, quorum writes provide durability without requiring all replicas. With replication factor 3 and write quorum 2, you can lose one node and still have a durable copy of every write."

### For Discussing Tradeoffs

6. "The fundamental tension is that every durability guarantee adds latency. fsync adds 1-15ms. Synchronous replication adds another 3-10ms of network round trip. The art is choosing the minimum durability that meets your requirements."

7. "We use `synchronous_commit = remote_write` for transactions that need cross-AZ durability but can tolerate the replica not having applied the change yet — only having received it into the OS page cache."

8. "For this use case, data loss tolerance is 1 second. We configure `synchronous_commit = off` with `wal_writer_delay = 200ms`, giving us sub-second loss while achieving 50,000 writes per second."

### For Explaining Recovery

9. "ARIES recovery has three phases: analysis to determine which transactions were active at crash time, REDO to replay all changes from the last checkpoint, and UNDO to roll back uncommitted transactions."

10. "Point-in-time recovery uses continuous WAL archiving. You restore the last full backup, then replay WAL segments up to the exact second before the incident. The typical RPO is under 1 minute with continuous archiving."

### For Operational Excellence

11. "A backup you have never restored is a hope, not a plan. We run automated restore tests quarterly that validate row counts, checksums, and application functionality against the restored database."

12. "We use checksums at the page level in PostgreSQL and at the application level for critical records. This ensures that corruption anywhere in the storage stack — disk, controller, cable, OS — is detected before it produces incorrect query results."

13. "The 3-2-1 rule: three copies of your data, on two different types of storage media, with one copy offsite. This gives you resilience against device failure, media-specific corruption, and site-level disasters."

### For Quantifying Durability

14. "With replication factor 3 and synchronous replication, the probability of data loss is the probability that all 3 nodes fail simultaneously within the fsync window. For AWS instances with 99.9% availability each, the joint probability is roughly 1 in a billion — effectively zero."

15. "A full backup of a 1TB database with pg_dump takes approximately 90 minutes. The backup consumes roughly 20% CPU and causes buffer cache churn. WAL archiving avoids this window entirely by capturing changes continuously rather than in a bulk snapshot."

16. "The cost of cross-region replication for a 1TB database that writes 100GB/day is approximately USD 6,000/month in data transfer alone. For data that does not need sub-second cross-region durability, periodic WAL archiving to another region at USD 100/month is a better tradeoff."

### For Handling Edge Cases

14. "Idempotency keys prevent data loss from ambiguous write outcomes. If the client sends a write, the acknowledgment is lost, and the client retries — the server detects the duplicate and returns the saved result without reapplying the mutation."

15. "A torn page occurs when a power failure interrupts a multi-sector write. InnoDB's double-write buffer prevents this by writing a full page copy before writing in place. PostgreSQL uses full-page writes in the WAL after each checkpoint."

## Red Flags: 8+ Things That Lose Points

### Red Flag 1: Confusing Durability with Availability

Candidate says: "We have three replicas, so data is safe."

The problem: Replication provides durability only if replicas have the same data. Asynchronous replication means the primary can have data that replicas have not received yet. Three replicas do not guarantee durability if you lose the primary before replication completes.

Fix: Always specify replication mode (sync vs async) and what happens during failover.

### Red Flag 2: Assuming Backups Work

Candidate says: "We take daily backups to S3."

The problem: A backup that exists is not a backup that restores. Backup corruption, missing WAL segments, and configuration drift all cause restore failures.

Fix: Mention automated restore testing, checksum verification, and the specific RPO/RTO numbers your backup strategy achieves.

### Red Flag 3: No Recovery Plan

Candidate says: "If data is lost, we can restore from backup."

The problem: Restoring from backup means recreating the infrastructure, applying backups, replaying WAL, reconfiguring networking, and validating data — all under pressure during an active incident.

Fix: Describe the specific restore process, including time estimates for each phase.

### Red Flag 4: Ignoring Human Error

Candidate says: "Our automated systems prevent data loss."

The problem: The most common cause of data loss is accidental deletion by a human. No automated system can protect against a DROP TABLE with full permissions.

Fix: Mention point-in-time recovery, database roles, 4-eyes approvals, and read-only replicas for reporting.

### Red Flag 5: No Checksums

Candidate says: "Checksums add overhead, so we skip them."

The problem: Without checksums, silent data corruption goes undetected. A single corrupted page can produce wrong results for months.

Fix: Enable data checksums, explain the 2-3% CPU cost, and describe how scrubbing detects issues.

### Red Flag 6: Believing fsync Always Works

Candidate says: "We call fsync after every write, so data is safe."

The problem: Consumer SSDs with DRAM write caches can report fsync success before data is on non-volatile media. Power loss in this window loses data.

Fix: Discuss battery-backed/capacitor-backed caches and specify enterprise hardware with power-loss protection.

### Red Flag 7: No Cross-Region Backup

Candidate says: "We backup to S3 in the same region."

The problem: Region-level outages are rare but catastrophic. An AWS us-east-1 outage takes both primary and backup.

Fix: Cross-region replication for backups, ideally to a different cloud provider for critical data.

### Red Flag 8: Testing Restores Only Once

Candidate says: "We tested the restore once when we set it up."

The problem: Database versions change, infrastructure changes, personnel changes. A restore that worked six months ago will have undocumented dependencies.

Fix: Quarterly automated restore drills with documented results.

## Drills: 5 Practice Scenarios

### Drill 1: Design the Write Path

Describe the exact sequence of events when a user uploads a profile photo in a social media application. Cover: which components are involved, what writes are durably stored, where fsync happens, how replication propagates the data, and what happens if the server crashes at each step.

### Drill 2: Choose a Database Configuration

A financial trading system processes 5,000 transactions per second. Each transaction involves updating an account balance. Data loss is not acceptable. Write latency must stay under 50ms. Choose a database, replication strategy, backup strategy, and hardware configuration. Justify each choice with numbers.

### Drill 3: Diagnose a Durability Incident

A SaaS company discovers that 0.1% of customer records inserted during the past week are missing from the database. The application code uses standard INSERT statements with default transaction settings. The database is PostgreSQL with `synchronous_commit = on` and one async replica. What happened? How do you investigate? How do you fix it?

### Drill 4: Plan a Multi-Region Deployment

An e-commerce platform expanding from one region to three regions (US, EU, APAC). Customer account data must be durable across region failures. Order data must be consistent within a region. Describe the durability architecture, including replication topology, backup strategy, and failover process for each data type.

### Drill 5: Reduce Cost Without Reducing Durability

Drill 4 answer sketch (for reference):

```
Drill 4 — Multi-Region Durability for E-Commerce Platform:

Customer account data (must survive region failure):
  - Primary database with synchronous replication to 3 AZs within region
  - Cross-region async replication to each other region
  - Cross-region replicas are synced: data exists in 2+ regions at all times
  - WAL archiving from each primary to S3 in its own region
  - Cross-region backup: copy WAL archive to other regions' S3 buckets

Order data (must be consistent within region):
  - Primary in each region for local orders
  - Synchronous replication within region (3 AZs)
  - Orders are not replicated across regions (different order IDs)
  - WAL archiving for point-in-time recovery
  - Cross-region backup for disaster recovery only

Failover process:
  - Region outage detected by health check (30-second timeout)
  - Cross-region replica promoted to primary
  - DNS updated to point to new region's load balancer
  - Remaining orders from the failed region can be reconstructed
    from WAL archive (RPO: under 5 seconds)
```

### Model Answer for Drill 1 — Photo Upload Write Path

```
Drill 1: Design the write path for a user uploading a profile photo.

Components involved:
  1. Client (browser or mobile app)
  2. API server (stateless, behind load balancer)
  3. Object storage (S3)
  4. PostgreSQL database (metadata)
  5. Cache (Redis, for profile reads)

Write path:

Step 1 — Client sends HTTP POST /profile/photo with multipart upload
Step 2 — API server receives the file bytes (typically 10KB-10MB)
Step 3 — API server uploads file to S3 (presigned URL or server-side SDK)
          - S3 returns HTTP 200 with ETag (MD5 checksum of uploaded object)
          - At this point, the file is durable (S3 11 nines)
          - S3 internally replicates to 3 AZs

Step 4 — API server begins database transaction
          BEGIN;
          INSERT INTO photos (user_id, photo_url, uploaded_at)
          VALUES (42, 'https://s3...', NOW());
          COMMIT;

          Inside the COMMIT:
            a. PostgreSQL writes WAL record to WAL buffer
            b. WAL buffer is flushed to WAL file on disk
            c. fsync() is called — blocks until confirmed on stable storage
            d. COMMIT record is written to WAL
            e. fsync() again for COMMIT record
            f. ACK returned to API server

Step 5 — API server invalidates Redis cache for user 42's profile
          (DEL user:42:profile)

Step 6 — API server returns HTTP 201 to client
          {"photo_url": "https://s3.amazonaws.com/...", "status": "created"}

Crash scenarios:
  - Crash after Step 3, before Step 4: photo is in S3, DB has no record,
    user can retry upload, S3 cleanup runs for orphaned objects
  - Crash during Step 4 COMMIT: WAL recovery handles it —
    no data loss, user retries with idempotency key
  - Crash after Step 4, before Step 5: data is in DB and S3,
    cache is stale, next read hits DB and refreshes cache
  - Crash after Step 5, before Step 6: all data durable, client
    times out, retries with idempotency key, gets existing data

Durability guarantee:
  - Photo file: 11 nines (S3 standard)
  - Photo metadata: zero data loss (PostgreSQL synchronous commit)
  - Cache: ephemeral (rebuildable from DB)
```

### Model Answer for Drill 6 — Worst-Case Migration Plan

```
Drill 6: Single SQLite file on a single server with no backups.

Priority-ordered migration plan:

Step 1 — Enable WAL mode on SQLite (cost: free, downtime: 0)
  PRAGMA journal_mode=WAL;
  - Protects against: crash during write (torn pages, partial writes)
  - Durability improvement: from "data lost on crash" to "last transaction safe"
  - Risk: none

Step 2 — Add daily backup to S3 (cost: USD 5/month, downtime: 5 minutes)
  - Backup script: sqlite3 data.db .backup /tmp/backup.db
  - Upload to S3 with versioning enabled
  - Protects against: disk failure, accidental deletion, ransomware
  - Durability improvement: from "single point of failure" to "24h RPO"
  - Risk: low

Step 3 — Add cron monitoring for database integrity (cost: free, downtime: 0)
  - Weekly: sqlite3 data.db "PRAGMA integrity_check;"
  - Alert on any error
  - Protects against: silent corruption
  - Durability improvement: detection of bit rot within 7 days

Step 4 — Migrate to PostgreSQL (cost: USD 50/month, downtime: 2 hours)
  - Move from embedded SQLite to client-server PostgreSQL
  - Enables: WAL archiving, replication, point-in-time recovery
  - Durability improvement: from "file-level durability" to
    "transaction-level durability with crash recovery"
  - Risk: moderate (schema migration, application changes)

Step 5 — Add synchronous replica (cost: USD 100/month, downtime: 1 hour)
  - Second PostgreSQL instance in different AZ
  - synchronous_commit = on
  - Protects against: single-AZ failure, primary disk failure
  - Durability improvement: zero data loss on single-host failure

Step 6 — Cross-region backup (cost: USD 15/month, downtime: 0)
  - WAL archiving to S3 with cross-region replication
  - Protects against: region-level disaster
  - Durability improvement: RPO under 1 minute across regions

Total cost after migration: ~USD 170/month
Total downtime: ~3.5 hours (one-time migration, not recurring)
Durability: from "single point of failure, no recovery" to
  "12 nines effective durability, under 1 minute RPO, under 1 hour RTO"
```

### Sample Answer: Drill 3 Diagnosis

```
Diagnosis:

Most likely cause: The async replica failed and was promoted,
but replication lag caused data loss.

Scenario:
1. Application writes to primary PostgreSQL
2. synchronous_commit = on, so fsync happens on primary
3. Client receives ACK
4. Async replica is 2 seconds behind due to load
5. Primary crashes (disk failure, power, etc.)
6. Replica is promoted to primary
7. Writes from the last 2 seconds before crash are missing

The 0.1% figure matches the proportion of writes that happened
in the 2-second window before the crash, assuming ~50ms per write.

Investigation:
1. Check PostgreSQL logs for time of crash and failover
2. Check replication lag metrics from monitoring
3. Compare WAL end position on failed primary vs promoted replica
4. Calculate the number of committed transactions in the gap window

Fix:
1. For zero data loss: change to synchronous replication
   (synchronous_standby_names = '*')
2. Tradeoff: write latency increases from ~3ms to ~6ms
3. Alternative: keep async but add WAL archiving to S3
   (if primary's WAL is archivable before disk failure)
```
