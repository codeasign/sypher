# Durability in the Real World

The largest storage systems in the world — Amazon S3, Apache Kafka, Google Spanner, Apache Cassandra — achieve durability at scales that would be unimaginable on a single machine. Each system takes a different approach, optimized for its specific workload patterns and guarantees.

---

## Amazon S3: 11 Nines of Durability

Amazon S3 is designed for 99.999999999% durability. For 10 trillion objects, an expected average loss of 0.1 objects per year.

### How S3 Achieves This

S3 stores each object across a minimum of three availability zones within a region. But the mechanism goes far beyond simple replication.

### S3 Durability Failure Modes

Even with 11 nines, S3 has specific failure scenarios you must plan for.

```
S3 failure modes:

1. Object overwrite (accidental):
   - Without versioning: overwrite is permanent
   - With versioning: previous version exists, recoverable
   - Mitigation: enable versioning on all buckets

2. Object deletion (accidental):
   - Without MFA Delete: any IAM user with s3:DeleteObject can delete
   - With MFA Delete: deletion requires multi-factor authentication
   - Mitigation: enable MFA Delete + versioning + S3 Object Lock

3. Cross-region replication lag:
   - CRR is asynchronous, typical RPO under 15 minutes
   - A region-level outage during CRR lag = data not yet replicated
   - Mitigation: monitor replication metrics (bytes pending, operations pending)

4. S3 service disruption (region-level):
   - us-east-1 had a multi-hour outage in 2017 affecting all services
   - Data was not lost (S3 is durable within the region), but
     objects were inaccessible during the outage
   - Mitigation: cross-region replication + DR plan with DNS failover

5. Ransomware / malicious deletion:
   - If an attacker gains IAM credentials, they can delete objects
   - Without versioning + MFA Delete: permanent data loss
   - Mitigation: S3 Object Lock (write-once-read-many), MFA Delete,
     separate backup account with different credentials
```

### S3 Versioning Cost Impact

```
Scenario: 100TB bucket, 10% of objects updated daily

Without versioning:
  - 100TB × USD 23 = USD 2,300/month

With versioning (30-day retention):
  - Storage: 100TB current + 10% × 100TB × 30 days = 130TB average
  - Cost: 130TB × USD 23 = USD 2,990/month
  - Overhead: 30% cost increase for versioning protection

With versioning + cross-region replication:
  - Primary: 130TB × USD 23 = USD 2,990
  - CRR destination: 130TB × USD 23 = USD 2,990
  - Data transfer: 300TB × USD 0.02/GB = USD 6,000
  - Total: USD 11,980/month
  - Overhead: 5x cost for multi-region durability
```

### Key Durability Mechanisms

**Erasure coding**: Instead of storing 3 full copies of each object (3x overhead), S3 uses Reed-Solomon erasure coding with approximately 1.5x overhead. For a 256MB object, 12 shards of ~21MB each are distributed across AZs. Any 8 shards reconstruct the original.

**Background scrubbing**: S3 continuously reads and verifies checksums on all stored data. When a shard degrades (due to media failure), it is reconstructed from the remaining shards and re-replicated.

**Versioning**: When enabled, S3 never overwrites object data. Each PUT creates a new version. A DELETE creates a delete marker without destroying the underlying data. Previous versions are recoverable.

**Multi-region durability**: With Cross-Region Replication (CRR), objects are asynchronously copied to a bucket in another region. The replication process itself is durable — each replicated object goes through the same erasure coding and integrity checks in the destination region.

## Apache Kafka: Durable Commit Log

Kafka's durability model is built around its append-only commit log. Every message written to a Kafka topic is appended to a log segment on the broker's disk.

### How Kafka Achieves Durability

### Key Durability Mechanisms

**Replication factor**: Each partition is replicated across N brokers. A replication factor of 3 survives 2 broker failures.

**ISR (In-Sync Replicas)**: The leader maintains a set of replicas that are fully caught up. Only ISR members are eligible to become leaders during failover. This prevents data loss from electing a lagging replica.

**min.insync.replicas**: The minimum number of replicas that must acknowledge a write before the producer receives a success response. With `min.isr=2` and `replication.factor=3`, the system can tolerate 1 broker failure without data loss.

**Log flushing**: Kafka flushes the log to disk based on either message count or time interval. Default is no periodic flush (relies on OS page cache). Production configurations typically flush every 5,000 messages or every 1 second.

**Segment compaction**: Kafka's log compaction retains the latest value for each key, reducing log size while preserving the ability to reconstruct state.

### Key Durability Mechanisms

**Synchronous replication**: Within a region, every write must be acknowledged by all three replicas before the client receives a success response. This provides zero data loss within a region.

**WAL on SSD**: Each replica writes to a write-ahead log on SSD with battery-backed cache. The SSDs are enterprise-grade with power-loss protection.

**Global Tables**: For multi-region durability, DynamoDB Global Tables replicate data to additional regions asynchronously. The replication process is conflict-free: DynamoDB uses last-writer-wins (LWW) based on timestamps.

**On-Demand backup**: DynamoDB offers continuous backups with point-in-time recovery (PITR) within the previous 35 days. Restore to any second within the window.

### Durability Numbers

```
Durability: 99.999999999% (11 nines) within a region
Availability SLA: 99.99% within a region
Global Tables: RPO typically under 1 second, RTO under 1 minute

Storage: unlimited (auto-scaling)
Throughput: virtually unlimited (auto-scaling partitions)
```

### Key Durability Mechanisms

**Paxos consensus**: Each Spanner tablet (a range of rows) is replicated to 5 datacenters across the globe as a Paxos group. Writes require a majority (3 out of 5) to commit.

**TrueTime**: Spanner uses GPS and atomic clocks to synchronize time across datacenters with bounded clock uncertainty (typically 1-7ms). This enables external consistency without sacrificing durability.

**Automatic failover**: When the Paxos leader fails, a new leader is elected from the remaining replicas. The new leader has all committed data because consensus ensures the majority has the latest state.

**Distributed backups**: Spanner continuously exports data to Google Cloud Storage using a distributed backup process that parallelizes across tablets. Backups are encrypted and stored across multiple regions.

## Apache Cassandra: Tunable Durability

Cassandra offers configurable consistency levels that let you set the exact tradeoff between durability, latency, and availability.

### How Cassandra Achieves Durability

```
Write consistency levels and their durability implications:

CL = ONE:
  - 1 replica acknowledges (any replica)
  - Lowest durability, lowest latency
  - Data loss if that replica fails before replication completes
  - Typical use: metrics, time-series data, analytics

CL = QUORUM:
  - Majority of replicas (N/2 + 1) acknowledge
  - Survives failure of floor(N/2) nodes
  - Typical use: production applications with RF=3, CL=QUORUM

CL = ALL:
  - Every replica acknowledges
  - Highest durability, highest latency
  - Writes fail if any replica is unavailable
  - Typical use: writes that must be in every copy immediately
```

Cassandra's commit log serves as the WAL. Each write goes to both the commit log (sequential) and the memtable (in-memory sorted structure). The commit log is fsynced at a configurable interval (default 10ms).

### Durability Limitations

Cassandra's durability model has a well-known gap: the commit log is fsynced periodically, not per write. If a node crashes between fsyncs, writes in the current commit log buffer are lost. For this reason, application-level consistency levels (QUORUM or ALL) are essential for durability in production.

### Cassandra Anti-Entropy Repair

Cassandra nodes can diverge over time due to missed writes, failed repairs, or hinted handoff expiration. Anti-entropy repair fixes this divergence.

```
Repair process:
1. A repair session is initiated for a token range
2. All replicas in the range compare their data using Merkle trees
   (each node builds a hash tree of its SSTables)
3. Nodes exchange Merkle tree roots and compare
4. Mismatched ranges are identified and data is streamed
   from the authoritative replica to the diverged replicas
5. Consistency is restored within the token range

Repair types:
  - Incremental repair: repairs only the data that changed
    since the last repair. Faster, but requires tracking state.
  - Full repair: rebuilds entire Merkle trees for the range.
    Thorough, but can take days for large datasets.

Repair frequency:
  - Default: every 7 days (nodetool repair)
  - Best practice: run incremental repair daily
  - For RF=3, running repair every 7 days means up to 7 days
    of divergence before detection. This is the durability gap.
  - With daily repair: under 24 hours of potential divergence

Repair cost (for 1TB per node, 3 nodes, RF=3):
  - Full repair: 6-12 hours, 50-80% CPU utilization
  - Incremental repair: 30-60 minutes, 10-20% CPU
  - Network transfer during repair: 100-500MB per cycle
```

### DynamoDB Global Tables Conflict Resolution

DynamoDB Global Tables use last-writer-wins (LWW) conflict resolution. When the same item is updated concurrently in two regions, the most recent write (by timestamp) wins.

```
Conflict scenario:
  Region 1 (us-east-1): Update item A with timestamp T1
  Region 2 (eu-west-1): Update item A with timestamp T2

  If T2 > T1: eu-west-1's update wins globally
  If T1 > T2: us-east-1's update wins globally
  If T1 == T2 (rare): system uses a deterministic tiebreaker
    (the region with the higher sortable ID wins)

Durability implication:
  - A write that is newer but not yet replicated can be overwritten
    by an older write from another region if clocks are out of sync
  - Mitigation: DynamoDB uses MSI (multi-site coordinator) with
    clock synchronization to minimize this window
  - For financial data: use strongly consistent reads and single-region
    writes with Global Tables as read-only replicas
```

### Spanner TrueTime and External Consistency

Spanner's TrueTime is the key innovation that makes its durability model unique. By exposing clock uncertainty, Spanner provides external consistency — the strongest consistency model in distributed databases.

```
TrueTime mechanism:
  - Each datacenter has GPS receivers and atomic clocks
  - Servers synchronize time within their datacenter
  - TrueTime exposes an interval [earliest, latest] for the
    current time, with bounded uncertainty (typically 1-7ms)
  - During a write, Spanner waits out the uncertainty interval
    to ensure all observers see a consistent ordering

External consistency guarantee:
  Transaction A commits before Transaction B starts
  → A's timestamp is guaranteed to be less than B's timestamp
  → All observers see A's effects before B's effects

This is stronger than linearizability: it guarantees that
real-time ordering is preserved across all operations,
even if they touch different data in different datacenters.

Durability + consistency cost:
  - Write latency: typically 20-100ms (includes TrueTime wait)
  - Read latency: 2-10ms (can read from local replica with
    timestamp-based consistency)
  - Throughput: millions of writes per second (auto-sharding)
```

## Five Lessons from the Real World

### Lesson 1: No Single Mechanism Is Sufficient

Every major system layers multiple durability mechanisms. S3 uses erasure coding + multi-AZ + continuous scrubbing + cross-region replication. Kafka uses replication + ISR + configurable flushing. No system relies on a single mechanism.

### Lesson 2: Tunability Is Essential

Kafka's `acks`, Cassandra's consistency levels, and PostgreSQL's `synchronous_commit` all let the operator choose the right durability point. A system that forces maximum durability on all writes loses to systems that let less-important data skip expensive safety checks.

### Lesson 3: Checksums Everywhere

S3 checksums every shard on every read. Spanner checksums every Paxos log entry. PostgreSQL checksums every data page. The cost is low (2-5% CPU), and the benefit is detection of corruption that would otherwise go undetected until it causes data loss.

### Lesson 4: Automate Everything

S3's background scrubbing, Spanner's automated failover, Kafka's ISR management, Cassandra's hinted handoff — all are automated. Human-mediated durability processes (manual backups, manual failover, manual integrity checks) are too slow and too error-prone for production systems.

### Lesson 5: Test the Recovery Path

Google runs "DiRT" (Disaster Recovery Testing) drills across production systems. Amazon conducts GameDays simulating failures. Every system that claims durability must prove it by actually surviving failures under controlled conditions.

These real-world implementations show that planetary-scale durability requires deep integration of WAL internals, replication protocols, consensus algorithms, and automated recovery — all working together. The next page translates these patterns into interview-ready knowledge.
