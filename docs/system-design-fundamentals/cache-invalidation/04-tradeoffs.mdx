## The Core Tension

Cache invalidation trades cache freshness for complexity. Every invalidation mechanism you add improves the freshness of cached data but introduces operational overhead, potential failure modes, and architectural constraints.

The simplest invalidation strategy — TTL only — has zero operational complexity but provides the worst freshness guarantee. The most sophisticated strategy — event-driven, tag-based, multi-region invalidation — provides near-instant freshness but requires a CDC pipeline, a message broker, a tag registry, a consistency checker, and dedicated operations expertise.

The decision is never "should we invalidate?" but rather "how stale is too stale, and how much complexity are we willing to accept to achieve that freshness target?"

## Decision Matrix

The following matrix helps you choose an invalidation strategy based on your requirements.

| Factor | TTL Only | Explicit Purge | Write-Update | Event-Driven |
|---|---|---|---|---|
| **Implementation complexity** | None | Low | Low | High |
| **Freshness guarantee** | TTL bound | Immediate | Immediate | Near-instant |
| **Staleness window** | 0 to TTL | Propagation | Propagation | Propagation |
| **Operational overhead** | None | Low | Low | High |
| **Bandwidth per invalidation** | 0 | Key only | Key + value | Event + keys |
| **Cache miss after invalidation** | Yes (TTL expiry) | Yes (first read) | No | Yes (first read) |
| **Idempotent** | Yes | Yes | Yes | Yes |
| **Works with any cache** | Yes | Yes | Yes | Yes |
| **Requires app changes** | No | Yes | Yes | Depends |
| **Recovery from failure** | Auto (time) | Manual retry | Manual retry | Log replay |

### When to Use Each Strategy

**Choose TTL only when:**
- Data changes at predictable intervals.
- Bounded staleness is acceptable (for example, a leaderboard updated every 5 minutes).
- You have no infrastructure for event pipelines.
- You are caching immutable or append-only data (see below).

**Choose explicit purge when:**
- You are building a monolith or small service where the writer knows the cache keys.
- The number of cache entries per write is small (1-3 keys).
- You want minimal infrastructure.

**Choose write-update when:**
- The cached value is small (under 1KB).
- You want zero cache misses after a write.
- The read-to-write ratio is very high.
- The write path can tolerate the extra latency of updating the cache.

**Choose event-driven when:**
- You have a microservice architecture with multiple writers.
- The invalidation logic is complex (many keys per write, many cache entry types).
- You need near-instant invalidation with bounded staleness.
- You already have a CDC pipeline or event bus for other purposes.

## CAP Theorem and Cache Invalidation

The CAP theorem states that a distributed data system can provide at most two of three properties: consistency, availability, and partition tolerance. Cache invalidation in a distributed cache is subject to the same trade-off.

### CP Invalidation

In a CP strategy, reads to cached entries are blocked until the invalidation is confirmed across all cache nodes. This ensures that no stale data is served, but it means the cache becomes unavailable for those keys during the invalidation propagation window.

CP is appropriate for data where stale values cause immediate harm: financial balances, medical records, access control permissions.

Implementation: before serving a cached value, check with a consensus protocol (like Raft or Paxos) that the value version is the latest. If the value was invalidated but the invalidation has not propagated, block the read or return an error.

### AP Invalidation

In an AP strategy, cache nodes serve whatever value they have, even if the value may be stale. Invalidation is applied asynchronously and converges over time.

AP is appropriate for most web applications: social media feeds, product catalogs, article content. The cost of serving stale data is bounded (the user sees out-of-date information) and acceptable compared to the cost of failing the request.

Implementation: the cache node checks the entry's validity time (TTL) and version. If the entry is within its TTL, it is served. If a more recent version exists, the entry is marked stale and refreshed asynchronously while the stale value is returned to the client (stale-while-revalidate pattern).

## Failure Modes

Cache invalidation introduces specific failure modes that must be understood and mitigated.

### Failure Mode 1: Invalidation Storm

An invalidation storm occurs when a single data change triggers the invalidation of many cache entries, and the subsequent cache misses all hit the database simultaneously.

**Mitigation.** Rate-limit invalidation-triggered re-population. Invalidate in batches over time rather than all at once. Use a "re-population delay" where cache misses after invalidation are served stale data for a brief period while fresh data is loaded lazily. Use write-update instead of write-invalidate for storm-prone data: push the new values directly to the cache instead of deleting and re-populating.

### Failure Mode 2: Silent Invalidation Failure

An invalidation event is published but never reaches the cache, and no error is raised. The cache continues serving stale data unboundedly.

**Causes.** (1) The invalidation consumer crashes after acknowledging the event but before processing it. (2) The event is published to the wrong topic or partition. (3) The event is filtered out by a misconfigured consumer filter. (4) The cache node is unreachable but the consumer silently retries and eventually gives up.

**Detection.** Monitor the percentage of invalidation events that result in cache modifications. If the percentage drops below expected, investigate. Use the consistency checker to sample cache entries and compare with source data.

**Mitigation.** Never disable TTL entirely. Always set a TTL that serves as a safety net. Make invalidation processing idempotent so retries are safe. Implement dead-letter queues for events that fail processing after the maximum retry count.

### Failure Mode 3: Stale Data Cascade

One uncorrected stale entry causes other entries to become stale. For example, a cached user profile contains a reference to a cached avatar image. The avatar image is updated and invalidated correctly, but the user profile cache entry is not invalidated. When a reader fetches the user profile, it gets a stale reference to the old avatar URL. Even though the avatar was correctly invalidated, the stale profile entry keeps serving the old avatar URL.

### Failure Mode 4: Invalidation Loop

An invalidation event triggers a write to the database, which triggers another invalidation event, which triggers another write, repeating indefinitely.

This typically happens when:

- The invalidation consumer updates the database as part of its processing (for example, marking an item as "invalidated").
- The update triggers the CDC pipeline, generating a new invalidation event.
- The cycle repeats.

### Failure Mode 5: Partial Cluster Invalidation

An invalidation event reaches some cache nodes but not others. This creates a window where clients routed to different nodes see different data.

**Causes.** (1) The invalidation consumer sends the command to a subset of nodes due to misconfiguration. (2) A network partition isolates some nodes. (3) Some nodes are under load and the invalidation request times out.

**Mitigation.** Implement a two-phase commit for critical invalidations: phase one sends the command to all nodes and waits for acknowledgement; phase two only commits if all nodes acknowledged. For non-critical invalidations, accept the inconsistency and rely on TTL safety nets. Use consistent hashing for both request routing and invalidation targeting so the same nodes that serve a key also receive its invalidation.

## Anti-Patterns

These anti-patterns appear regularly in production systems and should be avoided.

### Anti-Pattern 1: Relying Solely on TTL

The simplest invalidation strategy is also the most dangerous when used as the sole mechanism. TTL alone provides no protection against stale data within the TTL window.

**Why it happens.** Developers are in a hurry. TTL is one line of code. Setting up a CDC pipeline is a project.

**The fix.** Always pair TTL with at least one explicit invalidation mechanism for mutable data. Use TTL as a safety net, not as the primary strategy. If you must use TTL only, set the TTL to the maximum acceptable staleness for your application.

### Anti-Pattern 2: No Fallback for Invalidation Failure

The invalidation system fails (consumer crash, network partition, broker outage), and there is no mechanism to recover. Stale data accumulates until the system is manually restarted.

**Why it happens.** The invalidation system was designed as an afterthought without considering failure modes.

**The fix.** Always have a fallback: TTL safety net, a dead-letter queue for failed events with alerting, an invalidation log for replay, or a consistency checker that detects and repairs stale entries.

### Anti-Pattern 3: Invalidating Too Broadly

A single data change triggers invalidation of the entire cache or large sections of it. This causes a massive cache miss event that overwhelms the database.

**Why it happens.** It is easier to flush the entire cache than to track exactly which entries changed. Some systems ship with built-in "flush all" operations that become habit.

**The fix.** Implement fine-grained invalidation. Use tags to invalidate only the affected subset. If you must do bulk invalidations, rate-limit them and stagger the re-population. Consider write-update for bulk changes instead of delete-then-reload.

### Anti-Pattern 4: Ignoring Invalidation Ordering

Invalidation events for the same key arrive out of order. An old invalidation arrives after a newer one, causing the cache to delete a fresh entry.

**Why it happens.** Multiple invalidation sources (multiple writers, multiple consumers) do not coordinate their events. The event bus has multiple partitions, and events for the same key end up in different partitions.

**The fix.** Partition the event bus by cache key so that all events for the same key go to the same partition and arrive in order. Include a monotonically increasing version number with each event. The cache should reject invalidation events with a version number equal to or older than the current entry's version.

### Anti-Pattern 5: Infinite TTL

A cache entry is created with no TTL (or a very large TTL like 10 years). The entry becomes stale and is never refreshed. The only way to fix it is to restart the cache or manually purge the entry.

**Why it happens.** Developers set TTL to zero (which Redis treats as "no expiry") or forget to set a TTL entirely. The entry accumulates in the cache and is never revisited.

**The fix.** Always set a TTL on every cache entry. Use a cache middleware that requires a TTL parameter. Set a global maximum TTL in your cache configuration. Periodically audit cache entries for missing or excessive TTLs.

## When NOT to Use Cache Invalidation

There are data patterns where cache invalidation adds complexity without benefit.

**Immutable data.** Data that never changes after creation does not need invalidation. A blog post, once published and corrected, is effectively immutable. Set a long TTL and never invalidate. The TTL is only needed for memory management.

**Append-only logs.** Data that is only appended (never updated or deleted) does not need invalidation for existing entries. New entries can be cached independently without affecting old ones. A streaming timeline or event log is a good example.

**Data with a very short useful lifetime.** If data becomes irrelevant in under 10 seconds, the overhead of an invalidation pipeline exceeds the benefit. A real-time stock ticker that is refreshed every 100ms does not need invalidation: just set a 100ms TTL and let it expire naturally.

**Data that is always read from the source.** If the cache hit rate is under 30%, the cache is likely adding complexity without value. Drop the cache entirely rather than adding invalidation infrastructure.

In these cases, the cache is still useful for read performance. But the invalidation mechanism should be minimal or nonexistent. A well-chosen TTL is sufficient.
