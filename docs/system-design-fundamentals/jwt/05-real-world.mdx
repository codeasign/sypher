## Production JWT

The previous pages covered the theory: what JWT is, how signing and verification work, and the tradeoffs between algorithms and architectures. Now you need to see how JWT is actually used at scale. Every major identity platform relies on JWT. Each one made different design decisions that reveal important production patterns.

This page covers four real-world case studies from identity providers, the open-source libraries you will use, cloud provider support, and five lessons learned from production incidents.

---

## Case Study 1: Auth0 — JWKS and Key Rotation

Auth0 is one of the largest identity-as-a-service platforms, handling billions of token requests per day across tens of thousands of tenants. Auth0 issues JWT access tokens and ID tokens using RS256 (RSA with SHA-256) and publishes verification keys through a JWKS (JSON Web Key Set) endpoint.

### The JWKS Architecture

Instead of sharing a static public key, Auth0 publishes a set of public keys at a well-known URL: `https://YOUR_TENANT.auth0.com/.well-known/jwks.json`. Each key in the set has a unique Key ID (kid) that corresponds to the kid in the JWT header.

### How Key Rotation Works

Auth0 maintains two signing keys: the **current** key and the **previous** key. When a new key is generated (every 90 days by default), the previous key becomes the "previous" key and the oldest key is removed from the JWKS. The JWKS endpoint always returns both the current and previous keys.

Your application must cache the JWKS response and refresh it periodically. A common pattern is to cache for 24 hours but refresh on a 1-hour interval. If verification fails with a cached key (the kid is not found in the cached set), your application should fetch the JWKS fresh and retry. This handles the case where a key was rotated between your cache refresh intervals.

**Critical production rule:** Never hardcode a public key. Always fetch from JWKS. Companies that hardcode keys suffer outages during key rotation.

### Auth0's Multi-Tenant Architecture

Auth0 serves thousands of tenants from a shared infrastructure. Each tenant has its own signing key pair. The tenant identifier is embedded in the JWT's `iss` (issuer) claim.

```json
{
  "iss": "https://my-tenant.auth0.com/",
  "sub": "auth0|123456",
  "aud": "https://my-api.example.com",
  "exp": 1719878400,
  "iat": 1719792000,
  "scope": "read:orders write:orders"
}
```

The verification flow is:

1. Extract the `iss` claim to determine which tenant issued the token.
2. Fetch the JWKS from that tenant's well-known endpoint.
3. Find the key by `kid` and verify the signature.
4. Validate `aud`, `exp`, and `iat` claims.
5. If the token is an ID token, also validate `nonce` and `azp` (authorized party).

### What Broke

In 2020, Auth0 experienced a partial outage when a JWKS endpoint returned an incomplete key set during a deployment. Some applications received JWKS responses missing the current signing key. Applications that cached aggressively (24+ hour TTL) were unaffected. Applications that fetched on every request failed because they got the incomplete set. The lesson: cache JWKS responses with a reasonable TTL (1-6 hours) and implement a retry-with-fresh-fetch pattern when verification fails.

## Case Study 2: Okta — JWT as Access Tokens

Okta is another major identity platform used by enterprises for workforce identity and customer identity. Okta issues JWT access tokens by default in its OAuth 2.0 and OpenID Connect flows. Okta's design emphasizes enterprise requirements: custom claims, group membership, and tenant isolation.

### Okta's Access Token Structure

Okta access tokens contain claims that reflect enterprise directory data:

```json
{
  "iss": "https://dev-123456.okta.com/oauth2/default",
  "sub": "00u1abcdef1234567890",
  "aud": "https://api.example.com",
  "ver": 1,
  "iat": 1719792000,
  "exp": 1719878400,
  "jti": "AT.1234567890abcdef",
  "cid": "0oa1abcdef1234567890",
  "uid": "00u1abcdef1234567890",
  "scp": ["openid", "profile", "email", "orders:read"],
  "auth_time": 1719791800,
  "groups": ["Everyone", "Engineering", "Admin"],
  "email": "user@example.com"
}
```

Key differences from Auth0:

- **`ver` claim**: Okta includes a version number for the token format itself. This allows Okta to evolve the token structure without breaking existing verifiers.
- **`jti` claim**: A unique JWT ID that Okta uses for token revocation. When a user logs out, Okta invalidates the `jti` so that subsequent requests with that token are rejected.
- **`groups` claim**: Okta embeds group membership directly in the token. This is convenient but increases token size. An enterprise user in 500 groups would have a token exceeding 10KB.
- **`cid` claim**: The client ID of the application that requested the token. This allows the resource server to verify which client application originated the request.

### Token Size Management

Enterprise users belong to dozens or hundreds of groups. Embedding all group names in the JWT can cause the token to exceed HTTP header size limits (typically 8KB for most web servers, 16KB for NGINX). Okta recommends:

1. Use group IDs instead of group names to reduce claim size (UUIDs are shorter than "Engineering-West-Coast-Security-Team").
2. Set a maximum group claim size and truncate if exceeded. The token will contain the first N groups that fit within the limit.
3. For applications that need full group membership, use Okta's Groups API instead of relying on the token claim.

### Token Revocation Strategy

Okta supports token revocation through the `jti` claim. When a user logs out or an admin revokes a session:

1. Okta adds the `jti` to a revocation list in its database.
2. Your resource server must check the `jti` against the revocation list on every request, OR accept the access token's short TTL (typically 1 hour) and let it expire naturally.
3. Okta provides a token introspection endpoint for real-time validation: `POST /oauth2/default/v1/introspect`.

**Production pattern:** Most Okta deployments use short-lived access tokens (15-60 minutes) and skip real-time introspection for performance. For critical operations (password change, financial transactions), they perform introspection or require a fresh token.

### What Broke

Enterprise Okta customers have reported incidents involving token size. In one case, a user was added to 300 groups through an automated HR integration. The next token issued exceeded 12KB. The application's web server (Tomcat with default 8KB header limit) silently truncated the Authorization header, causing signature verification to fail. The user was locked out until the token expired and a new one was issued (without truncation because the error had been fixed). Okta responded by adding configurable group claim limits.

## Case Study 3: Google Identity Platform — JWT for Service Accounts

Google Cloud Platform (GCP) uses JWT extensively for service-to-service authentication. Google's approach is notable because it demonstrates JWT used in a **non-human context**: machine identities communicating with other machines.

### Service Account JWT Flow

A GCP service account is a machine identity with its own RSA key pair. When a service (for example, a compute engine VM) needs to call another Google API, it creates and signs a JWT using the service account's private key.

### The Assertion Grant Pattern

Google's flow uses a standard OAuth 2.0 extension called **JWT Bearer Token for Client Authentication** (RFC 7523). The service creates a JWT with these critical claims:

- **`iss`**: The service account email address, which identifies the issuer.
- **`scope`**: The Google API scopes being requested (for example, `https://www.googleapis.com/auth/cloud-platform`).
- **`aud`**: The token exchange endpoint URL (`https://oauth2.googleapis.com/token`).
- **`iat`** and **`exp`**: The token is typically valid for 3600 seconds (1 hour).
- **`sub`**: Optional. If the service account is impersonating a user, this is the user's email.

The service signs this JWT with its RSA private key (RS256) and sends it to Google's token exchange endpoint. Google verifies the signature using the service account's public key, checks that the JWT has not expired, and issues a Google-signed access token. The access token has a shorter lifetime (typically 3600 seconds or less) and is used to call the target API.

### Why Two Tokens?

You might wonder: why not just use the self-signed JWT directly? Google's design has two advantages:

1. **Centralized revocation.** The Google token exchange endpoint can refuse to issue access tokens even if the self-signed JWT is valid. If a service account is deactivated, Google revokes it centrally. The self-signed JWT becomes useless because the token exchange will reject it.
2. **Token format standardization.** The access token that Google issues is a standard opaque token or Google-format JWT. Downstream services do not need to understand service account JWTs. They only need to verify Google's signature, which they can do with Google's public JWKS endpoint.

### What Broke

A common incident pattern with Google service account JWT: a developer exports the service account's private key as a JSON file and checks it into a GitHub repository. The key is then compromised. The attacker uses the key to sign JWTs and impersonate the service account. Google's incident response playbook for this: revoke the compromised key and rotate all service account keys. The lesson: service account private keys are as sensitive as passwords. Use a secrets manager (GCP Secret Manager, HashiCorp Vault) to store them. Never check them into source control.

## Case Study 4: AWS Cognito — JWT for User Pools

AWS Cognito is Amazon's identity platform for web and mobile applications. Cognito issues JWT tokens in three varieties: access token, ID token, and refresh token. Cognito's implementation is representative of how JWT is used in the AWS ecosystem.

### Cognito Token Structure

Cognito issues three tokens after a successful authentication:

**Access token (JWT):** Contains scopes and the user's username. Used to authorize API calls.

```json
{
  "sub": "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee",
  "iss": "https://cognito-idp.us-east-1.amazonaws.com/us-east-1_ExAmPlE",
  "client_id": "1example23456789",
  "origin_jti": "unique-token-id",
  "event_id": "event-unique-id",
  "token_use": "access",
  "scope": "aws.cognito.signin.user.admin openid profile email",
  "auth_time": 1719791800,
  "exp": 1719878400,
  "iat": 1719792000,
  "jti": "unique-jwt-id",
  "username": "johndoe"
}
```

**ID token (JWT):** Contains user attributes like email, phone number, and custom attributes.

```json
{
  "sub": "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee",
  "iss": "https://cognito-idp.us-east-1.amazonaws.com/us-east-1_ExAmPlE",
  "aud": "1example23456789",
  "token_use": "id",
  "auth_time": 1719791800,
  "exp": 1719878400,
  "iat": 1719792000,
  "email": "user@example.com",
  "email_verified": true,
  "phone_number": "+15551234567",
  "cognito:username": "johndoe"
}
```

**Refresh token (opaque):** Not a JWT. A long-lived opaque string stored client-side to obtain new access and ID tokens without re-authentication.

### Cognito Authorizer Integration

AWS API Gateway has a built-in Cognito authorizer that handles JWT verification without any custom code. When you configure the authorizer:

1. API Gateway fetches the Cognito user pool's JWKS URL automatically.
2. On each request, API Gateway extracts the JWT from the Authorization header.
3. It verifies the signature, expiration, issuer, and token_use.
4. It maps claims from the verified JWT to request context variables that your Lambda function can access.

This is the easiest JWT verification setup you will encounter. No library installation, no manual JWKS caching, no signature verification code. However, the tradeoff is that you are limited to Cognito as your identity provider. If you need to support multiple identity providers (Auth0, Okta, custom IdP), you must use a custom Lambda authorizer.

### Cognito Groups and Roles

Cognito supports user groups that map to IAM roles. When a user belongs to a group, Cognito includes a `cognito:groups` claim in the access token:

```json
{
  "cognito:groups": ["Admins", "PremiumUsers"],
  "cognito:preferred_role": "arn:aws:iam::123456789012:role/AdminRole",
  "cognito:roles": [
    "arn:aws:iam::123456789012:role/AdminRole",
    "arn:aws:iam::123456789012:role/PremiumRole"
  ]
}
```

Your backend can use these claims for fine-grained authorization. For example, only users with the "Admins" group in their JWT can call the admin API.

### What Broke

Cognito had a widely reported incident in 2022 where its JWKS endpoint returned a 503 error for approximately 45 minutes. Applications using API Gateway's built-in Cognito authorizer were unaffected because API Gateway caches the JWKS response internally. However, applications that validated Cognito tokens manually (using a third-party JWT library) and fetched the JWKS on every request experienced a cascading outage: every verification attempt failed, every request was rejected, and the application became completely unavailable.

The lesson: the AWS well-architected framework recommends caching the JWKS response for at least 1 hour. The incident showed that applications following this guidance survived the outage. Applications that did not cache went down.

## Open-Source JWT Libraries

You will likely use an open-source library to verify JWTs in your application rather than implementing RS256 verification from scratch. Each major language has a dominant library. Here is a comparison of the most widely used ones.

### jose (JavaScript / TypeScript)

The `jose` library is the modern standard for JWT operations in JavaScript, supporting the JOSE (JSON Object Signing and Encryption) standards family.

**Strengths:**
- Supports all JWT algorithms: RS256, RS384, RS512, ES256, ES384, ES512, EdDSA, and symmetric HS256/HS384/HS512.
- Built-in JWKS support: `createRemoteJWKSet` fetches and caches JWKS from a URL automatically.
- Zero runtime dependencies.
- Works in Node.js, browsers, Cloudflare Workers, and Deno.
- Supports JWE (JWT encryption) for payload confidentiality.

**Weaknesses:**
- API surface is large and can be confusing. The library covers JWS, JWE, JWK, JWKS, and JWT in one package.
- Documentation examples sometimes lag behind the API.

```javascript
import { createRemoteJWKSet, jwtVerify } from 'jose';

const JWKS = createRemoteJWKSet(
  new URL('https://example.com/.well-known/jwks.json')
);

async function verifyToken(token) {
  const { payload, protectedHeader } = await jwtVerify(token, JWKS, {
    issuer: 'https://my-tenant.auth0.com/',
    audience: 'https://api.example.com',
  });
  return payload;
}
```

### jsonwebtoken (Node.js)

The `jsonwebtoken` library (often abbreviated `jwt`) is the older, simpler alternative for Node.js.

**Strengths:**
- Extremely simple API: `jwt.verify(token, secretOrKey, options)`.
- Wide adoption and extensive documentation.

**Weaknesses:**
- No built-in JWKS support. You must fetch and cache keys yourself.
- Synchronous API by default. Async version requires callbacks or manual promisification.
- No browser support (depends on Node.js `crypto` module).
- Not actively maintained as of 2024. The `jose` library is the recommended successor.

```javascript
const jwt = require('jsonwebtoken');

// You must fetch and cache the JWKS yourself
const publicKey = await fetchPublicKeyFromJWKS(token);
const decoded = jwt.verify(token, publicKey, {
  algorithms: ['RS256'],
  issuer: 'https://my-tenant.auth0.com/',
  audience: 'https://api.example.com',
});
```

### PyJWT (Python)

PyJWT is the most widely used JWT library in the Python ecosystem.

**Strengths:**
- Simple API: `jwt.decode(token, key, algorithms=[...], options=...)`.
- Built-in JWKS support via `PyJWKClient`.
- Supports async with `PyJWKClient` in async contexts.
- Comprehensive algorithm support.

**Weaknesses:**
- The `algorithms` parameter is required (a good thing — prevents algorithm confusion attacks).
- The `options` dict for verification settings is verbose. You must explicitly disable leeway or set accepted algorithms.

```python
from jwt import PyJWKClient, decode

jwks_client = PyJWKClient("https://example.com/.well-known/jwks.json")

def verify_token(token: str) -> dict:
    signing_key = jwks_client.get_signing_key_from_jwt(token)
    data = decode(
        token,
        signing_key.key,
        algorithms=["RS256"],
        issuer="https://my-tenant.auth0.com/",
        audience="https://api.example.com",
    )
    return data
```

### java-jwt (Java)

Auth0's official Java library for JWT verification.

**Strengths:**
- Fluent builder API: `JWT.require(algorithm).withIssuer(...).build().verify(token)`.
- Built-in JWKS support: `JwkProviderBuilder` with configurable caching.
- Type-safe claim verification.

**Weaknesses:**
- Verbose compared to Python or JavaScript libraries.
- Requires careful exception handling for `JWTVerificationException` subtypes.

```java
import com.auth0.jwk.JwkProvider;
import com.auth0.jwk.JwkProviderBuilder;
import com.auth0.jwt.JWT;
import com.auth0.jwt.algorithms.Algorithm;
import com.auth0.jwt.interfaces.DecodedJWT;
import com.auth0.jwt.interfaces.JWTVerifier;

JwkProvider provider = new JwkProviderBuilder("https://example.com/.well-known/jwks.json")
    .cached(true)
    .cacheSize(10)
    .cacheTime(24, TimeUnit.HOURS)
    .rateLimited(false)
    .build();

DecodedJWT jwt = JWT.decode(token);
RSAPublicKey publicKey = (RSAPublicKey) provider.get(jwt.getKeyId()).getPublicKey();
Algorithm algorithm = Algorithm.RSA256(publicKey, null);
JWTVerifier verifier = JWT.require(algorithm)
    .withIssuer("https://my-tenant.auth0.com/")
    .withAudience("https://api.example.com")
    .build();

DecodedJWT verified = verifier.verify(token);
```

| Criteria | jose (JS) | jsonwebtoken (JS) | PyJWT (Python) | java-jwt (Java) |
|---|---|---|---|---|
| JWKS built-in | Yes | No | Yes | Yes |
| Cache support | Built-in (HTTP cache) | Manual | Built-in (PyJWKClient) | Built-in (JwkProvider) |
| Browser support | Yes | No | N/A | N/A |
| Active maintenance | Active | Maintained | Active | Active |
| Algorithm coverage | Full JOSE | Common | Full JOSE | Common |
| JWE support | Yes | No | No | No |
| API complexity | Medium | Low | Low | Medium |

**Recommendation:** For new projects, prefer `jose` (Node.js) or `PyJWT` (Python). Both have built-in JWKS support, active maintenance, and comprehensive algorithm coverage. Avoid `jsonwebtoken` for new projects unless you need minimal API surface and are comfortable managing JWKS caching yourself.

## Cloud Provider JWT Support

Each major cloud provider has its own approach to JWT. Here is how AWS, Azure, and GCP handle JWT for their identity systems.

### AWS Cognito

Cognito is the primary JWT issuer in the AWS ecosystem. It integrates natively with API Gateway, ALB (Application Load Balancer), and AppSync.

- **API Gateway Cognito authorizer:** Built-in JWT verification. No code required.
- **ALB authentication:** Application Load Balancers can authenticate users via Cognito and pass the JWT to backend targets in the `x-amzn-oidc-accesstoken` header.
- **AppSync:** Cognito user pools are a built-in authentication provider for GraphQL APIs.
- **Lambda authorizer (custom):** For complex rules or multi-IdP support, write a Lambda function that verifies tokens against any JWKS endpoint.

Cognito's token lifetime configuration:
- Access token: 1 hour (configurable, 5 minutes to 24 hours).
- ID token: 1 hour (configurable, 5 minutes to 24 hours).
- Refresh token: 30 days (configurable, 1 hour to 3650 days).

### Azure AD (Microsoft Entra ID)

Azure AD issues JWT access tokens for Microsoft APIs and custom APIs. Azure AD's tokens include Microsoft-specific claims.

- **`tid`:** Tenant ID — identifies which Azure AD tenant issued the token.
- **`oid`:** Object ID — the unique identifier for the user or service principal in Azure AD.
- **`appid`:** Application ID — the client application that requested the token.
- **`roles`:** Application roles assigned to the user (Azure AD's equivalent of groups).
- **`scp`:** Scopes (same as OAuth 2.0 scopes).

Azure AD publishes its JWKS endpoint at `https://login.microsoftonline.com/common/discovery/v2.0/keys`. The `common` tenant endpoint returns keys for all Azure AD tenants, or you can scope it to a specific tenant.

Azure AD also supports **managed identities** for Azure resources (VMs, App Services, Functions). The managed identity endpoint provides JWT access tokens for the resource to authenticate to other Azure services, similar to Google's service account flow.

### GCP IAM

GCP's approach to JWT is more decentralized than AWS's. Three different JWT flows exist:

1. **Service account self-signed JWTs:** As described in the Google case study. The service account creates and signs a JWT to prove its identity to Google's token exchange.

2. **Identity-Aware Proxy (IAP):** GCP's IAP issues JWT tokens to verified users. The JWT is included in the `x-goog-iap-jwt-assertion` header that your backend can verify using Google's JWKS endpoint. The token includes the user's email and a unique identifier.

3. **Workload Identity Federation:** Allows workloads running outside GCP (AWS, Azure, on-premises) to exchange external credentials for GCP JWTs without a service account key file.

## Five Lessons Learned from Production

After reviewing these case studies and the open-source ecosystem, five patterns emerge that you should apply in every JWT-based system.

### Lesson 1: Always Cache JWKS Responses

Every case study in this section demonstrated the same failure mode: not caching the JWKS response caused an outage when the JWKS endpoint was temporarily unavailable.

**What to do:**

- Cache the JWKS response for 1 to 24 hours.
- Set a background refresh interval shorter than the cache TTL (for example, cache for 24 hours, refresh every hour).
- When verification fails with a cached key (kid not found), fetch a fresh JWKS and retry before rejecting the token.
- Use a dedicated HTTP client with connection pooling and timeout settings for JWKS fetches.

**Why it matters:** The JWKS endpoint is a single point of failure for your entire authentication system. Every request that requires JWT verification depends on it. If the endpoint is down and you have no cache, every single request fails.

### Lesson 2: Validate All Standard Claims

Do not skip claim validation. Every claim in the JWT standard exists because production systems needed it.

**Minimum validation set:**

- **`exp` (expiration):** Must be in the future. Reject expired tokens.
- **`nbf` (not before):** Must be in the past. Prevents tokens from being used before they are active.
- **`iss` (issuer):** Must match your expected issuer. Prevents token injection attacks where an attacker uses a token from a different identity provider.
- **`aud` (audience):** Must match your API's identifier. Prevents a token intended for Service A from being used to access Service B.
- **`iat` (issued at):** Optional, but useful for detecting tokens that are suspiciously old.
- **`token_use` or `typ`:** Validate that the token is used for its intended purpose (access token vs. ID token).

**Why it matters:** Without audience validation, any JWT issued by your identity provider can be used against any of your APIs. A token issued for a photo upload API could be used to call the admin API. This is a common finding in security audits.

### Lesson 3: Use Short Expiration and Implement Refresh Tokens

Short-lived access tokens reduce the blast radius of token theft. If an access token is compromised, it is only useful for a short window. The tradeoff is that your users must re-authenticate frequently — unless you implement refresh tokens.

**Token lifetime guidelines:**

- Access tokens: 15 minutes to 1 hour. Use 15 minutes for high-security APIs, 1 hour for standard APIs.
- Refresh tokens: 7 to 30 days. Rotate refresh tokens on each use (refresh token rotation).
- ID tokens: Same as access tokens. They convey user identity and should not outlive the access token.

**Refresh token rotation:** Each time the client uses a refresh token to obtain new tokens, the server issues a new refresh token and invalidates the old one. This limits the reuse window if a refresh token is stolen. Auth0 and Okta both recommend this pattern.

### Lesson 4: Plan for Key Rotation

Signing keys must be rotated periodically. When they are, your application must handle the transition gracefully.

**Key rotation patterns:**

1. **Dual-key overlap (recommended):** Keep the old key in the JWKS for at least the maximum token lifetime after rotation. If your tokens live for 1 hour, keep the old key for 2 hours. This ensures tokens signed with the old key can still be verified until they expire.

2. **Graceful verification:** When verification fails, first check if the kid was found in the JWKS. If the kid matches a known key but the signature is invalid, the token is definitely invalid. If the kid is not found in the JWKS, fetch fresh keys and retry. The key may have been rotated between fetches.

3. **Monitor rotation events:** Set up monitoring for JWKS changes. If keys rotate more frequently than expected, it may indicate a security incident (compromised key being replaced).

### Lesson 5: Handle Token Leakage and Revocation

JWT's stateless nature means you cannot revoke a token once it is issued. The token remains valid until it expires. This creates a security gap: if a token is leaked, you must wait for it to expire or change your signing key (which invalidates all tokens).

**Mitigation strategies:**

1. **Short TTL (primary defense):** The simplest approach. A 15-minute access token window means leaked tokens are useful for at most 15 minutes.

2. **Token blocklist (conditional):** Maintain a blocklist of revoked `jti` values in a fast database (Redis with TTL matching the token's expiration). Check the blocklist on every request. This works for small blocklists (hundreds of entries) but does not scale to millions of concurrent revoked tokens.

3. **Token introspection (enterprise):** Call the identity provider's introspection endpoint for high-value operations. This is slow (adds a network round trip) but provides real-time revocation. Use only for critical operations like password changes or financial transactions.

4. **Rotate keys on breach (last resort):** If you suspect a widespread token leak, rotate your signing key. This invalidates all tokens, forcing all users to re-authenticate. Plan this as an emergency procedure with a runbook.
