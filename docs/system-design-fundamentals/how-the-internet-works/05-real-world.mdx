## Case Study 1: Facebook's BGP Outage (2021)

On October 4, 2021, Facebook disappeared from the internet for 6 hours. The outage affected Facebook, Instagram, WhatsApp, Messenger, and Oculus — 3.5 billion users. Internal engineers could not reach their own servers. Even the physical badge readers at Facebook's data centers stopped working.

### What Happened

A routine BGP configuration change went wrong. The Facebook engineer intended to withdraw a specific BGP route for maintenance but instead withdrew all BGP routes for Facebook's entire DNS prefix. The withdrawal propagated through the global internet routing system in under 30 seconds.

### Root Cause Analysis

Three independent failures compounded into a complete outage:

1. **BGP configuration bug:** The automated tool did not validate the scope of the route withdrawal. It withdrew all prefixes instead of the intended one. There was no "human-in-the-loop" approval for the change.

2. **No out-of-band management:** Facebook had no independent network path for emergency access. All remote access depended on the production DNS system, which was unreachable because the BGP routes were withdrawn. The engineers could not log in to fix the problem because the login process required DNS resolution.

3. **Single point of failure in DNS:** Facebook's DNS authoritative servers were in the same AS as the withdrawn prefixes. There was no secondary DNS provider, no anycast DNS with independent connectivity, and no DNS failover to a different network.

### Lessons Learned

**BGP changes must be validated** before deployment. Use a staging environment that simulates the global routing table. Require two-person approval for route withdrawals. Implement automatic rollback if reachability drops below a threshold.

**Out-of-band management is essential** for critical infrastructure. An independent network path (separate ISP, separate DNS, separate authentication) that does not depend on the production network. This is a standard practice for data centers but often overlooked for cloud infrastructure.

**DNS should be on a separate network** from the services it resolves. Use a managed DNS provider with anycast routing and independent connectivity. Never run your authoritative DNS servers on the same network as your web servers.

## Case Study 2: Cloudflare's Anycast Network

Cloudflare operates one of the largest anycast networks on the internet, serving approximately 20% of all web traffic. Their architecture demonstrates how anycast routing, CDN caching, and DDoS protection work at planetary scale.

### Architecture

Cloudflare uses **anycast** — the same IP address is announced from 330+ data centers worldwide. BGP routes each user to the nearest data center. Within each data center, a consistent hash distributes traffic across servers.

### Key Numbers

- **330+ data centers** in 120+ countries
- **20% of all web traffic** passes through Cloudflare
- **10+ Tbps DDoS scrubbing capacity** — the largest attack mitigated was 2.5 Tbps
- **99.999% uptime** for the core network over the last 5 years
- **1-5ms cache hit latency** within the nearest data center
- **50-200ms origin fetch latency** if the cache misses

### Anycast Design Decisions

**Why not DNS-based routing?** DNS-based routing (Route53 latency-based) requires a DNS query per region change and cannot handle sub-second failover. Anycast failover is instant — as soon as BGP withdraws the route, traffic stops going to that location.

**Why not a single global load balancer?** A single load balancer would be a bottleneck, a single point of failure, and would add 50-200ms of latency for users far from it. Anycast distributes traffic to the nearest data center, minimizing latency and eliminating single points of failure.

**How does caching work with anycast?** Each data center has its own cache. If the same content is popular in both Tokyo and London, both caches store a copy. This is less efficient than a single central cache (duplicate storage) but much faster (no cross-continent fetches).

## Case Study 3: Google's Private Backbone

Google operates one of the largest private networks in the world, connecting its data centers, edge POPs, and offices. This private backbone carries most of Google's internal traffic — including search, YouTube, Gmail, and Cloud — and is separate from the public internet.

### Why a Private Backbone?

The public internet is a best-effort network with no quality-of-service guarantees. Google's traffic is too valuable and too latency-sensitive to rely on the public internet. The private backbone provides:

- **Dedicated bandwidth:** 100-400 Gbps links between data centers. No contention with other traffic.
- **Predictable latency:** No queueing delay from other users' traffic. Latency is determined by fiber distance alone.
- **Traffic engineering:** Google can optimize the path for each traffic type — bulk replication takes a different path than user-facing search.
- **No BGP hijacking risk:** Internal routes are not advertised to the public internet. No risk of route hijacking.

### Key Numbers

- **100+ Tbps total backbone capacity**
- **100-400 Gbps per link** using DWDM (Dense Wavelength Division Multiplexing)
- **99.999% availability** on the backbone
- **~10ms latency** between US-East and US-West data centers (vs 64ms on the public internet)
- **SDN-based traffic engineering** reallocates bandwidth in under 10 seconds

### Traffic Engineering

Google's B4 network uses a Software-Defined Networking (SDN) controller that centrally manages traffic flows. The controller:

1. Collects real-time link utilization from every router
2. Computes the optimal traffic allocation using a global optimization algorithm
3. Pushes the forwarding rules to each router via OpenFlow

This is fundamentally different from BGP, where each router makes independent decisions based on local information. The SDN controller has a global view and can optimize across the entire network. B4's traffic engineering is described in Google's 2013 SIGCOMM paper — the first large-scale deployment of SDN in production.

## Case Study 4: Netflix Open Connect CDN

Netflix delivers over 15% of global internet traffic. To handle this scale, Netflix built its own CDN — **Open Connect** — a network of caching appliances deployed inside ISP networks worldwide.

### Architecture

Instead of serving content from a few central data centers, Netflix places caching servers directly inside ISP data centers. When a user presses play, the content is served from a server that is within the ISP's network — often only 1-3 network hops away.

### Key Numbers

- **15%+ of global internet traffic** is Netflix
- **95%+ of traffic** served from Open Connect appliances (not from AWS)
- **100 TB SSD storage** per appliance, 40 Gbps throughput
- **Hundreds of appliances** deployed in ISP networks globally
- **under 5ms latency** from appliance to user (1-3 network hops)
- **Billions of dollars saved** in bandwidth costs vs. serving from central cloud

### Why Not Just Use a Commercial CDN?

At Netflix's scale (15% of global traffic), commercial CDN pricing would be prohibitively expensive. Building their own CDN gave Netflix:

**Cost control:** The hardware cost of an OCA is a one-time expense. Commercial CDN bandwidth pricing is recurring. At Netflix's scale, the OCA model pays for itself in under 6 months.

**Performance control:** Netflix controls the hardware, the software, and the deployment. They can optimize the caching algorithm for their specific content (video streaming) without being constrained by a CDN provider's general-purpose cache.

**ISP relationship:** Deploying inside ISP networks gives Netflix a direct relationship with ISPs. This eliminates the need for transit agreements and peering disputes. The ISP benefits because Netflix traffic stays within their network — no transit costs for carrying Netflix traffic to a third-party CDN.
