### What Breaks at Each Scale

**At 1,000 users:** The server handles everything. DNS is a single A record with a 24-hour TTL. Response time is under 50ms. Everything works fine.

**At 10,000 users:** The server CPU hits 80%. The database connection pool is nearly exhausted. Response time climbs to 200ms. The server starts dropping connections during peak hours. You cannot add a second server because the DNS TTL is 24 hours — adding a second IP address would not take effect until tomorrow.

**At 100,000 users:** The server is overwhelmed. The database connection pool is exhausted. The web server's thread pool is saturated. The server crashes multiple times per day. You need to redesign the architecture immediately, but you cannot change the DNS record quickly because the TTL is 24 hours. Every minute of downtime costs measurable revenue.

### Key Architectural Decisions

**DNS TTL of 60 seconds:** A short TTL means you can change the IP address of your service and have the change propagate within one minute. The trade-off is higher query load on your authoritative DNS servers (1,440 queries per day per resolver instead of 60). At 1 million users, this is approximately 1,440 DNS queries per second — trivial for managed DNS.

**CDN for TLS termination:** Terminating TLS at the CDN edge reduces the CPU load on your origin servers by 90% (TLS handshake is computationally expensive). The CDN connects to the origin over a private encrypted connection (origin pull). The trade-off is that the CDN can see decrypted traffic — ensure your CDN provider has a SOC 2 certification and a data processing agreement.

**Multi-AZ deployment:** Deploying across three availability zones (data centers within a region) provides redundancy against a single data center failure. AWS us-east-1 has had multi-hour AZ-specific outages. With multi-AZ, if one AZ goes down, the load balancer routes traffic to the remaining AZs.

### DNS-Based Routing Strategies

**Latency-based routing:** Route53 measures latency between the user's DNS resolver and each region. Users in Sydney connect to ap-southeast-1 (15ms) instead of us-east-1 (210ms). This is the standard approach for global services.

**Geolocation routing:** Route53 routes based on the user's geographic location. EU users go to eu-west-1, US users go to us-east-1. Useful for data sovereignty compliance (GDPR requires EU user data to stay in EU).

**Weighted routing:** Distribute traffic across regions in a specific ratio. 80% to us-east-1, 20% to eu-west-1. Useful for canary deployments and gradual rollouts.

**Failover routing:** Primary region is us-east-1. If health checks fail, all traffic goes to eu-west-1. Simple and reliable for active-passive deployments.

### Cross-Region Data Considerations

**Asynchronous replication:** The standard for most web applications. The primary in us-east-1 accepts writes and asynchronously replicates to eu-west-1. The replica is typically 50-200ms behind. If the primary fails, the replica may be missing the last few writes.

**Synchronous replication:** The primary waits for a replica in another region to acknowledge the write. This guarantees zero data loss but adds 50-200ms to every write. Used for financial transactions and critical metadata.

**Conflict resolution:** In an active-active setup where both regions accept writes, conflicts must be resolved. Last-writer-wins (LWW) is the simplest approach but loses data. CRDTs or application-level conflict resolution provide stronger guarantees at higher complexity.

## Integration Patterns

### Pattern 1: CDN Origin Shield

The CDN does not fetch every cache miss from the origin server. An **origin shield** layer sits between the edge POPs and the origin. Edge POPs that miss fetch from the shield (a regional cache layer), and only the shield fetches from the origin. This reduces origin load by 10x because multiple edge POPs that miss on the same content all hit the shield, which serves the cached copy.

```
User ──> Edge POP (Tokyo) ──miss──> Origin Shield (Singapore) ──miss──> Origin (us-east-1)
User ──> Edge POP (Sydney) ──miss──> Origin Shield (Singapore) ──hit──> Return cached
User ──> Edge POP (Mumbai) ──miss──> Origin Shield (Singapore) ──hit──> Return cached
```

Without origin shield: 3 origin requests. With origin shield: 1 origin request.

### Pattern 2: DNS-Based Service Discovery

Internal services (microservices) use DNS for service discovery. When Service A needs to call Service B, it resolves `service-b.prod.svc.cluster.local` to an IP address. DNS-based discovery:

- **Simple:** No external service registry needed. Every language has a DNS resolver.
- **Load balancing:** DNS returns multiple IPs in round-robin order (DNS RR).
- **Health checking:** Unhealthy instances are removed from DNS (via service mesh or consul).
- **Limitation:** DNS caching means slow failover. TTL of 5 seconds is the practical minimum. For faster failover, use a service mesh with client-side load balancing.

### Pattern 3: Global Load Balancing with Anycast

Anycast gives the same IP address to multiple servers in different locations. BGP routes each user to the nearest server. Anycast provides:

- **Automatic failover:** If one server goes down, BGP withdraws the route, and traffic goes to the next nearest server.
- **DDoS mitigation:** Traffic is distributed across all anycast locations. A 1 Tbps DDoS attack hitting 10 anycast locations sends 100 Gbps to each location — manageable.
- **Simplified DNS:** One IP address for the entire service. No need for latency-based DNS routing.

Anycast is used by DNS root servers, Cloudflare, and critical internet infrastructure. The trade-off is that stateful connections (TCP, TLS) may break if different packets from the same connection reach different servers. Anycast works best for stateless protocols (DNS, HTTP with shared session state).

## Capacity Planning

### Bandwidth Calculation for a Global Service

```
For a video streaming service serving 1 million concurrent viewers at 5 Mbps:

Total bandwidth: 1,000,000 × 5 Mbps = 5,000,000 Mbps = 5 Tbps

With CDN offload (95% cache hit rate):
  Origin bandwidth: 5 Tbps × 5% = 250 Gbps
  CDN edge bandwidth: 5 Tbps × 95% = 4.75 Tbps (handled by CDN provider)

Cross-region replication:
  Per-region write traffic: 100 Mbps
  Replication to 3 regions: 300 Mbps total
  Bandwidth cost: $0.02/GB × 300 Mbps × 86,400s/day = $51,840/day

Without CDN: origin must handle 5 Tbps = ~1,250 servers at 4 Gbps each
With CDN: origin handles 250 Gbps = ~63 servers at 4 Gbps each
```

### DNS Query Volume

```
For a global service with 100 million daily active users:

Each user triggers 10 DNS queries per day (different subdomains, CDN fetches)
Total DNS queries: 1 billion per day = 11,574 queries/second at peak

With 60-second TTL:
  Each resolver caches for 60 seconds = 1,440 queries/day per resolver
  Number of unique resolvers: 100M / 1,440 = ~70,000 resolvers
  QPS to authoritative nameserver: 1B / 86,400 = 11,574 QPS

Managed DNS (Route53, CloudDNS, Dyn): handles 10M+ QPS, cost ~$1,000/month
Running your own DNS (BIND, PowerDNS): 10,000 QPS per server, 2 servers needed
```

### Server Count for a Global Web Service

```
For an API service handling 100,000 RPS peak:

Per-server capacity: 5,000 RPS (modern Go/Java app, stateless)
Servers needed: 100,000 / 5,000 = 20 servers
With 2x headroom for failover: 40 servers
With 3 regions: 40 × 3 = 120 servers total

Monthly cost estimate:
  120 servers × $150/month (c6i.xlarge reserved) = $18,000
  3 load balancers × $22/month = $66
  DNS (Route53) = $1,000/month
  CDN (CloudFront) = $5,000/month
  Total: ~$24,000/month

Cost per million requests: $24,000 / (100,000 RPS × 86,400 × 30) = $0.00009
```
