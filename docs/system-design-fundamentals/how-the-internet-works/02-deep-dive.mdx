## Internal Architecture of Packet Delivery

The previous section showed the surface-level path of a packet. This section opens the box and examines the data structures, algorithms, and protocols that make the internet work at scale — from the structure of a single packet to the global BGP routing table that contains nearly a million routes.

### The Five-Layer TCP/IP Model

Internet communication is organized into layers. Each layer provides services to the layer above and consumes services from the layer below. This separation of concerns means a web browser does not need to know whether the underlying network is fiber, copper, or satellite — each layer handles its own responsibilities.

### Layer 1: Physical

The physical layer defines how raw bits are transmitted over a medium. Fiber optic cables carry light pulses at 400 Gbps per wavelength using Dense Wavelength Division Multiplexing (DWDM) — 80+ wavelengths per fiber, for a total of 32 Tbps per fiber pair. Copper Ethernet (Cat 6A) carries electrical signals at 10 Gbps over 100 meters. Wi-Fi 6 uses Orthogonal Frequency-Division Multiple Access (OFDMA) to serve multiple devices simultaneously on the same channel.

The physical layer has no concept of packets, addresses, or protocols. It transmits bits. Whether those bits represent a cat video or a bank transfer is invisible at this layer.

### Layer 2: Data Link

The data link layer handles communication between directly connected devices on the same network segment. Each device has a **MAC address** (48-bit, burned into the network interface card). An Ethernet frame carries the source MAC, destination MAC, the encapsulated network layer packet, and a frame check sequence for error detection.

**ARP (Address Resolution Protocol)** maps IP addresses to MAC addresses within a local network. When a router needs to send a packet to the next hop, it broadcasts an ARP request: "Who has IP 203.0.113.1? Tell me your MAC address." The device with that IP responds with its MAC address. The router caches this mapping in its ARP table (typically 5-15 minute timeout).

**Switches** operate at Layer 2. A switch learns which MAC addresses are on each port by inspecting source MAC addresses in incoming frames. When it receives a frame destined for a known MAC, it forwards the frame only to the correct port. For unknown MACs, it floods the frame to all ports except the source.

### Layer 3: Network

The network layer (IP) handles end-to-end delivery across multiple networks. This is where the internet's global addressing and routing happens.

**IPv4 packet structure:**
- Version (4 bits): always 4 for IPv4
- IHL (4 bits): header length in 32-bit words (minimum 5 = 20 bytes)
- DSCP (6 bits): differentiated services code point (QoS marking)
- Total Length (16 bits): entire packet size, maximum 65,535 bytes
- Identification (16 bits) + Flags (3 bits) + Fragment Offset (13 bits): used when a packet must be fragmented to fit the MTU of the next link
- TTL (8 bits): decremented by every router. When it reaches 0, the packet is dropped and an ICMP "Time Exceeded" is sent back. Prevents routing loops.
- Protocol (8 bits): identifies the transport layer protocol (6 = TCP, 17 = UDP)
- Header Checksum (16 bits): verifies the header integrity (not the payload)
- Source IP (32 bits) and Destination IP (32 bits)

**Routing table lookup:** Each entry in a routing table consists of a destination prefix (network address and subnet mask), a next hop IP, an interface, and a metric. When a packet arrives, the router performs a **longest prefix match**: it finds the routing table entry with the most specific (longest) subnet mask that matches the destination IP. If the destination is 93.184.216.34, an entry for 93.184.216.0/24 matches more specifically than an entry for 93.0.0.0/8.

### Layer 4: Transport

**TCP segment structure:**
- Source Port (16 bits) and Destination Port (16 bits)
- Sequence Number (32 bits): the byte offset of the first byte in this segment
- Acknowledgment Number (32 bits): the next byte the receiver expects
- Data Offset (4 bits): TCP header length in 32-bit words
- Flags (9 bits): URG, ACK, PSH, RST, SYN, FIN (plus CWR, ECE, NS)
- Window Size (16 bits): how many bytes the receiver can accept (flow control)
- Checksum (16 bits): covers the TCP header, payload, and a pseudo-IP-header
- Urgent Pointer (16 bits): offset to urgent data (rarely used)
- Options (variable): maximum segment size, window scaling, selective ACK, timestamps

**TCP congestion control** is the most important algorithm most engineers never see. Congestion control prevents the sender from sending data faster than the network can handle. The default algorithm (CUBIC) uses a three-phase approach:

1. **Slow start:** Double the sending rate every round trip until packet loss is detected or a threshold (ssthresh) is reached.
2. **Congestion avoidance:** Increase the sending rate linearly (additive increase) until loss occurs, then halve it (multiplicative decrease). This is AIMD — Additive Increase, Multiplicative Decrease.
3. **Fast recovery:** When duplicate ACKs indicate a lost packet, retransmit the lost packet without waiting for a retransmission timeout.

The congestion window (cwnd) is the number of bytes the sender can transmit without waiting for an ACK. A typical starting cwnd is 10 segments (~14,460 bytes). After one RTT without loss, it doubles to 20 segments. After 4 RTTs, it reaches 160 segments. Slow start is why short web requests never achieve high throughput — the connection finishes before the window opens.

### Layer 5: Application

The application layer implements specific protocols for specific use cases. Each protocol uses the transport layer (TCP or UDP) and defines its own message format, state machine, and semantics. Common application protocols: HTTP (web), DNS (name resolution), SMTP (email), TLS (encryption — technically sits between transport and application), WebSocket (real-time bidirectional), and gRPC (RPC over HTTP/2).

## BGP — The Internet's Routing Protocol

BGP is the protocol that makes the internet work as a network of networks. Each autonomous system (AS) announces the IP prefixes it can reach and learns about other prefixes from its neighbors.

### BGP Message Types

1. **OPEN:** Establishes a BGP session between two routers. Includes the AS number, hold time, and BGP identifier.
2. **UPDATE:** Advertises new routes or withdraws previously advertised routes. Each UPDATE contains one prefix and its path attributes.
3. **NOTIFICATION:** Reports an error condition and closes the session.
4. **KEEPALIVE:** Sent periodically (typically every 30 seconds) to maintain the session. If no KEEPALIVE is received within the hold time (typically 90 seconds), the session is terminated.

### BGP Path Selection

When a BGP router receives multiple routes to the same prefix, it selects the best route using this ordered decision process:

1. Highest **Local Preference** (set by policy — local admin control)
2. Shortest **AS Path** (number of AS hops — the default metric)
3. Lowest **Origin type** (IGP < EGP < incomplete)
4. Lowest **MED** (Multi-Exit Discriminator — suggests preference to external AS)
5. Prefer **eBGP over iBGP** (external routes over internal)
6. Lowest **IGP metric** (shortest path to the next hop)
7. **Router ID** (tiebreaker — lowest wins)

### BGP Security

BGP was designed when the internet was a small, trusted network. It has no built-in authentication: any AS can advertise any prefix. This leads to **route hijacking** — an AS advertises a prefix it does not own, and traffic destined for that prefix is rerouted through the hijacker.

**RPKI (Resource Public Key Infrastructure)** is the primary defense. Route Origin Authorizations (ROAs) cryptographically bind a prefix to an AS number. Routers validate BGP announcements against ROAs. A route that does not match any ROA is treated as invalid. As of 2025, approximately 35% of the internet's prefixes are covered by ROAs. Deployment is growing but slow, and RPKI does not prevent AS path forgery — an attacker can still advertise a valid prefix with a fabricated AS path.

| Type | Name | Function | Example |
|------|------|----------|---------|

## Performance Characteristics

### Latency Budget for a Single Web Page Load

| Step | Time | Cumulative |
|------|------|-----------|
| DNS resolution (cache miss) | 50ms | 50ms |
| TCP handshake (1 RTT) | 40ms | 90ms |
| TLS handshake (2 RTTs) | 80ms | 170ms |
| HTTP request (1 RTT) | 40ms | 210ms |
| Server processing | 100ms | 310ms |
| Response download | 20ms | 330ms |
| Browser rendering | 150ms | 480ms |

A page load that takes 480ms has spent 210ms (44%) on network round trips before the server even processes the request. This is why HTTP/2 multiplexing, connection reuse, and CDN edge caching are so effective — they attack the network overhead, not the server processing time.

### Bandwidth vs. Latency Trade-off

The product of bandwidth × RTT determines how much data can be in flight at any moment. This is the **bandwidth-delay product** (BDP). For a 100 Mbps connection with 80ms RTT: BDP = 100 Mbps × 0.08s = 10 Mb = 1.25 MB. The TCP congestion window must grow to at least this size to fully utilize the link. For a 10 Gbps link with 200ms RTT (satellite): BDP = 10 Gbps × 0.2s = 2 Gb = 250 MB. The receiver's window must allow 250 MB of unacknowledged data — requiring TCP Window Scaling to extend the 16-bit window field to 1 GB.

## Configuration and Tuning

### Linux Network Stack Tuning

```
# /etc/sysctl.conf — Network tuning for high-throughput servers

# Increase TCP buffer limits (default: 128K, max: 16MB)
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# Enable TCP BBR congestion control (better than CUBIC for long-fat pipes)
net.core.default_qdisc = fq
net.ipv4.tcp_congestion_control = bbr

# Increase the local port range for outbound connections
net.ipv4.ip_local_port_range = 1024 65535

# Enable TCP Fast Open (saves 1 RTT on repeat connections)
net.ipv4.tcp_fastopen = 3

# Enable window scaling (needed for BDP > 64KB)
net.ipv4.tcp_window_scaling = 1

# Reduce TIME_WAIT sockets (can exhaust ports under high connection rates)
net.ipv4.tcp_fin_timeout = 15
net.ipv4.tcp_tw_reuse = 1
```

Without these settings, a server handling 10,000 outbound connections per second will exhaust the ephemeral port range in under 6 seconds. New connections fail. The application appears down. The root cause is not the application — it is the kernel refusing to allocate ports.

### Tuning for Different Workloads

**Latency-sensitive (stock trading, real-time gaming):** Use BBR congestion control, reduce the initial RTO (retransmission timeout) from 1 second to 200ms, and enable TCP_NODELAY to disable Nagle's algorithm. For ultra-low latency, consider kernel bypass (DPDK) or RDMA.

**Throughput-sensitive (video streaming, large file transfers):** Increase the TCP buffer sizes to accommodate high bandwidth-delay products. Use CUBIC or BBR congestion control. Disable TCP Segmentation Offload at the NIC if it causes jitter.

**Connection-heavy (web servers, APIs):** Enable TCP Fast Open, TCP_DEFER_ACCEPT, and increase the connection backlog (net.core.somaxconn = 65535). Use a connection pool to avoid per-request TCP handshake overhead.
