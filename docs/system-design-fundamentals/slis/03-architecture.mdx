Building a system that collects, aggregates, and evaluates SLIs is straightforward at small scale. At large scale — millions of requests per second, thousands of services, global user populations — the architecture must change fundamentally. This section traces the evolution from a naive implementation to a production-grade SLI platform.

## The Naive Implementation

Most teams start with the simplest possible approach: log every request, store it in a database, and query the database to compute SLI values. This works at small scale and fails catastrophically at larger scales.

### At 1,000 Requests Per Second

A single application server processes requests and writes each one to a relational database. The database has a table called `request_log` with columns for timestamp, latency, status code, and service name.

```
request_log table:
  id: BIGINT AUTO_INCREMENT
  timestamp: DATETIME(6)
  service: VARCHAR(64)
  latency_ms: INT
  status_code: INT
  path: VARCHAR(256)

SELECT COUNT(*) as total,
       SUM(CASE WHEN status_code < 500 THEN 1 ELSE 0 END) as good
FROM request_log
WHERE service = 'api'
  AND timestamp > NOW() - INTERVAL 30 DAY;
```

At 1,000 requests per second, this table grows by 86 million rows per day. A query over 30 days scans 2.6 billion rows. Even with indexes, this query takes minutes. The team accepts this because they run it once per day for a manual report.

This is the first breakpoint. The naive approach works for reporting but not for real-time alerting.

### At 100,000 Requests Per Second

The table grows by 8.6 billion rows per day. The database disk fills up within hours. The team adds a retention policy that deletes rows older than 30 days, but the delete operation itself causes massive I/O and blocks writes.

The query over 30 days now scans 260 billion rows. It runs for hours and consumes all available database CPU. Other queries — including user-facing ones — are starved.

The team realizes the naive approach is dead. They need a fundamentally different architecture.

### At 10 Million Requests Per Second

At this scale, the naive approach is not just slow — it is impossible. Writing 10 million rows per second to a single database would require approximately 500 GB per second of write throughput. No single database instance can handle this.

The request log table would grow by 864 billion rows per day. A 30-day query would scan 26 trillion rows. The query would never complete.

## Production-Grade Architecture

A production SLI system uses a fundamentally different approach: batch aggregation, pre-computed histograms, and horizontal sharding.

### Key Architectural Decisions

**Aggregate, do not store raw data.** Instead of writing every request to a database, aggregate counts in memory and write summaries to the database. This reduces data volume by a factor equal to the number of requests per second.

**Shard by service.** Each service's SLI data is independent. Sharding by service gives you natural isolation and allows you to add capacity by adding more shards.

**Pre-compute percentiles.** Do not compute p50/p99/p999 from raw data at query time. Pre-compute them during aggregation and store the results. Queries become simple lookups.

**Use multiple time scales.** Store high-resolution data for the recent past and low-resolution data for the distant past. This keeps query performance fast while still supporting long windows.

## Integration Patterns

SLI collection must integrate with the existing infrastructure. The choice of integration pattern depends on the existing infrastructure, the latency requirements, and the team's tolerance for modifying application code. There is no single best pattern — most organizations use a combination of patterns for different services.

The key tradeoff across all patterns is between control and convenience. Patterns that require modifying application code (cache integration, queue integration) give you more control over the SLI data quality. Patterns that consume existing data (database integration, CDN integration) require no code changes but have less control over data quality.

Each pattern has different latency characteristics, durability guarantees, and operational overhead. The following sections describe the four most common patterns in detail, including when to use each one and the tradeoffs involved.

### Pattern 1: Database Integration

For systems that already write request data to a database, the SLI system can consume this data directly using a change data capture (CDC) stream or a periodic query.

**How it works**: The SLI system subscribes to the database's write-ahead log (WAL). Every time a new row is inserted into the request log, the CDC stream notifies the SLI collector, which reads the row and updates the appropriate aggregate counters.

**When to use**: When you already have a request log database and cannot modify the application code to emit metrics directly. This is common in legacy systems or when using third-party software.

**Tradeoffs**: The CDC stream adds latency — typically 100 milliseconds to 1 second between the request and the SLI update. The database must be configured to retain the WAL for long enough to prevent the CDC consumer from falling behind.

### Pattern 2: Cache Integration

For systems that use a distributed cache like Redis or Memcached, the SLI aggregation can use the cache as a shared counter store.

**How it works**: Each service instance atomically increments counters in Redis using INCR commands. The counters are keyed by service name and time bucket. The aggregation layer reads these counters periodically and computes SLI values.

```
INCR sli:api:good:2026-07-02T10:00
INCR sli:api:total:2026-07-02T10:00
```

**When to use**: When you need minimal latency between request and SLI update. Redis INCR is single-digit milliseconds, making this the fastest integration pattern.

**Tradeoffs**: Redis is single-threaded for write operations, so at very high throughput (above 1 million INCR per second), Redis becomes the bottleneck. Shard the Redis keys across multiple instances to handle higher throughput.

### Pattern 3: Queue Integration

For systems that already use a message queue (Kafka, RabbitMQ, Kinesis) for event streaming, the SLI data can be published as a separate topic.

**How it works**: Each service instance publishes a metrics event to a dedicated Kafka topic. The aggregation layer consumes from this topic and updates its counters. The Kafka topic is partitioned by service name, so all events for a given service go to the same partition, ensuring ordered processing.

**When to use**: When you already have Kafka in your infrastructure. This is the most common pattern at large companies because it leverages existing infrastructure and provides strong durability guarantees.

**Tradeoffs**: Kafka adds 10 to 100 milliseconds of latency between the request and the SLI update. The Kafka cluster must be sized to handle the metrics volume, which can be significant at scale.

### Pattern 4: CDN Integration

For systems that serve content through a CDN, the SLI data must be collected from edge locations that you do not directly control.

**How it works**: The CDN provider exposes a metrics API or sends log files to a cloud storage bucket. The SLI system periodically fetches this data (every 5 to 15 minutes) and computes SLI values from the edge logs.

**When to use**: When your service is fronted by a CDN and you cannot instrument the application layer directly. This is common for static content delivery, video streaming, and global API gateways.

**Tradeoffs**: CDN metrics are delayed. The data might be 5 to 30 minutes stale by the time it reaches the SLI system. Burn rate alerts based on CDN data cannot detect fast failures. You need a separate, faster SLI pipeline for the backend services that the CDN proxies.

## Multi-Region Architecture

A global service requires SLI collection across multiple geographic regions. The architecture must handle cross-region data consistency, different regional traffic patterns, and regional outages. The key challenge is that the SLI system must be at least as reliable as the service it monitors. If the service runs in 3 regions, the SLI system must also run in 3 regions — and it must continue operating if one region becomes unavailable.

The standard approach is to deploy the SLI system in each region alongside the service. Each region's SLI pipeline is self-contained: it collects metrics from services in that region, aggregates them, and evaluates SLOs independently. The regional SLI data is then replicated to a central aggregator for global visibility.

This design means that a regional outage does not affect SLI collection in other regions. If the US-East region goes down, the EU-West region continues collecting and evaluating SLIs for its services. The central aggregator may show missing data for US-East, but the EU-West SLI data remains accurate.

### Regional Isolation

Each region runs its own SLI pipeline — instrumentation, collection, aggregation, and evaluation. The SLO evaluator in each region computes SLI values independently for that region's traffic.

**Why regional isolation?** If one region's SLI pipeline goes down, the other regions continue operating. A network partition between regions does not affect SLI computation within each region.

### Global Aggregation

For services that serve global audiences, you need a global SLI that aggregates across all regions. The global aggregator runs in a separate location and pulls data from each regional aggregator every 60 seconds.

The global aggregator computes:
- Global SLI (average across all regions, weighted by traffic volume)
- Worst-region SLI (the region with the lowest SLI value)
- Regional SLI diffs (how each region is performing relative to the global average)

### Cross-Region Traffic

When a user in Europe connects to a US region, the latency SLI for that request is measured by the US region. The SLI must account for the fact that cross-region latency is inherently higher than in-region latency.

The standard approach is to tag each request with the user's geographic region and compute separate SLIs for in-region and cross-region traffic. The SLO for cross-region traffic can be more lenient — for example, p99 under 500 milliseconds instead of p99 under 200 milliseconds.

## Capacity Planning

Let us walk through a concrete capacity planning exercise for an SLI system serving 10 million daily active users.

### Assumptions

- 10 million DAU
- Each user makes 50 requests per day on average
- Total requests per day: 500 million
- Average requests per second: 5,787 (500 million / 86,400 seconds)
- Peak requests per second: 17,361 (3x the average)
- 200 services to monitor
- 30-day sliding window
- 1-minute bucket resolution

### Collection Layer

Each second, the collection layer receives 17,361 batches from the instrumented services. Each batch is approximately 1 KB (containing the good count, total count, and histogram data).

```
Inbound data rate: 17,361 KB/s ≈ 17 MB/s
Hourly data volume: 17 MB/s x 3,600 s = 61 GB
Daily data volume: 17 MB/s x 86,400 s = 1.47 TB
```

The collection layer needs enough capacity to handle the peak rate. With 10 collector nodes, each handles 1.7 MB/s of inbound data — well within the capabilities of a modern server.

### Aggregation Layer

Each aggregation shard handles a subset of services. With 200 services across 10 shards, each shard handles 20 services.

```
Per-shard memory:
  Sliding window: 20 services x 43,200 buckets x 16 bytes = 13.8 MB
  Histograms: 20 services x 256 KB = 5.1 MB
  Burn rate counters: 20 services x 4 windows x 16 bytes = 1.3 KB
  Total per shard: approximately 19 MB

Per-shard CPU:
  Batch processing: 17,361 / 10 = 1,736 batches/second
  Batch merge time: 1,736 x 10 microseconds = 17.4 ms/s
  Query processing: 10 queries/second x 50 microseconds = 0.5 ms/s
  Total CPU: approximately 2 percent of one core
```

The aggregation layer is not the bottleneck. Even at 10x the peak traffic, the aggregation layer uses less than 20 percent of one core per shard.

However, the aggregation layer is memory-bound, not CPU-bound. Each shard must hold the full 30-day sliding window in memory for all assigned services. If the number of services per shard grows to 200 instead of 20, the memory required per shard grows from 19 MB to 190 MB. This is still manageable, but the histogram merge time also grows because each service's histogram must be updated separately.

The key scaling limit for the aggregation layer is the number of concurrent histogram updates. Each batch triggers a histogram merge operation for one service. If 20 services are each sending 1 batch per second, the shard performs 20 histogram merges per second. At 200 services, it performs 200 histogram merges per second. The histogram merge operation takes approximately 10 microseconds, so 200 merges per second consumes 2 milliseconds of CPU time per second — still negligible. The real limit is the memory bandwidth: each merge reads and writes the histogram data structure, which is 256 KB per service. At 200 services, the memory bandwidth consumed is 200 x 256 KB = 51 MB per second, which is within the capabilities of modern DDR5 memory (approximately 50 GB/s).

### Storage Layer

The time-series database stores aggregated data at multiple resolutions:

```
High-resolution (1-minute buckets, retained 7 days):
  200 services x 10,080 buckets x 16 bytes = 32 MB

Medium-resolution (5-minute buckets, retained 30 days):
  200 services x 8,640 buckets x 16 bytes = 28 MB

Low-resolution (1-hour buckets, retained 365 days):
  200 services x 8,760 buckets x 16 bytes = 28 MB

Total storage for 1 year of SLI data: approximately 100 MB
```

The storage requirements are minimal because the system stores only aggregated counts, not raw request data. The raw request data would be 100,000 times larger.

### Network

```
Per-service outbound bandwidth: 17 KB/s (one batch per second)
Total outbound from 200 services: 3.4 MB/s
Total inbound to 10 collectors: 3.4 MB/s
Total inter-region traffic (for global aggregation): 3.4 MB/s x 3 regions = 10.2 MB/s
```

Network is not a concern. The SLI pipeline uses approximately 0.1 percent of the bandwidth that the actual service traffic uses.

However, network latency matters for global aggregation. If the global aggregator is in US-East and it pulls data from the APAC region, the cross-region link adds 150 to 200 milliseconds of latency. The global aggregator must account for this by using a longer polling interval (60 seconds for global aggregation vs. 10 seconds for regional aggregation).

### High Availability Design

The SLI system must be highly available because it is the system that detects failures in other systems. If the SLI system itself goes down, the entire organization is flying blind.

The standard approach is a 2N+1 redundancy model for the aggregation shards. If you need 10 shards to handle the load, you deploy 21 shards (10 active + 10 standby + 1 extra for failover capacity). Each active shard has a dedicated standby that receives the same data stream but does not serve queries. If the active shard fails, the standby takes over within 5 seconds.

The collection layer is stateless and can be load-balanced behind a single endpoint. If one collector fails, the load balancer routes traffic to the remaining collectors. The collectors are deployed with a headroom of 50 percent — each collector operates at 50 percent capacity during normal operation, so it can absorb the traffic from a failed neighbor.

The SLO evaluator is the most critical component. It is deployed in an active-active configuration with a minimum of 3 replicas. Each replica independently evaluates SLOs and fires alerts. A consensus mechanism (majority vote) prevents duplicate alerts: if at least 2 of 3 replicas agree that a burn rate threshold has been exceeded, the alert fires. This prevents a single misconfigured evaluator from causing a false alert storm.

**Failure scenarios and recovery times:**

| Failure | Detection | Recovery | Data Loss |
|---------|-----------|----------|-----------|
| Single aggregator shard crash | 5 seconds | 60 seconds | 0-60 seconds |
| Collector node failure | 10 seconds | 30 seconds | None |
| SLO evaluator crash | 10 seconds | 20 seconds | None |
| Kafka broker failure | 30 seconds | 5 minutes | None |
| Full region outage | 60 seconds | Manual | Partial |

The recovery times are based on automated failover. Manual recovery for a full region outage involves updating DNS records to route traffic to a healthy region. This process is practiced quarterly in a chaos engineering exercise.

### Cost Analysis

The SLI system costs approximately 0.1 percent of the infrastructure cost of the services it monitors. For a company spending $10 million per year on production infrastructure, the SLI system costs roughly $10,000 per year.

The cost breakdown:

| Component | Monthly Cost | Notes |
|-----------|-------------|-------|
| Compute (collectors) | $500 | 10 nodes at $50/month each |
| Compute (aggregators) | $1,000 | 20 nodes at $50/month each |
| Compute (evaluators) | $300 | 3 nodes at $100/month each |

The cost is justified by the value. A single prevented outage that would have cost $100,000 in lost revenue pays for 2.5 years of the SLI system.

## Deployment and Operations

Running an SLI system in production requires attention to deployment strategy, monitoring, and operational procedures.

### Deployment Strategy

Deploy the SLI system as a separate infrastructure from the services it monitors. This prevents a cascading failure where a service outage brings down the SLI system, which then cannot alert on the outage.

Deploy in a canary pattern. Deploy the new version to one shard, let it run for 30 minutes, and verify that the SLI values match the expected values before deploying to the remaining shards.

### Monitoring the Monitor

The SLI system itself must be monitored. Key metrics for the SLI system:

- **Batch lag**: The time between a batch being emitted by the service and being processed by the aggregator. Alert if lag exceeds 60 seconds.
- **Deduplication rate**: The percentage of batches that are duplicates. A sudden increase in duplicates indicates a network issue or a retry storm.
- **Data loss rate**: The percentage of batches that are dropped. Alert if data loss exceeds 0.1 percent.
- **SLO evaluation freshness**: The time since the last SLO evaluation. Alert if no evaluation has occurred in the last 120 seconds.

### Operational Runbooks

**Runbook: Aggregation shard is down**

1. The remaining shards continue processing their own services. Unaffected services continue to have SLI data.
2. For the affected shard, incoming batches are buffered in the Kafka queue. The queue has approximately 5 minutes of capacity.
3. If the shard does not recover within 5 minutes, a new shard instance is provisioned.
4. The new shard loads its state from the time-series database and the Kafka queue offset.
5. SLI data for the affected services is restored within 1 minute of the new shard starting.

**Runbook: SLO evaluator is reporting incorrect values**

1. The evaluator is put into "read-only" mode. Alerts are suppressed.
2. The evaluator's state is dumped and compared against a known-good evaluator in a staging environment.
3. If the state is corrupted, the evaluator is rebuilt from the time-series database.
4. The evaluator is restarted in "active" mode, and alerts are re-enabled.
