## How This Topic Appears

The URL shortener problem is one of the most frequently asked system design questions at FAANG companies. It tests your ability to decompose a familiar problem into subsystems, make design trade-offs under constraints, and reason about distributed system behavior. You will encounter the topic in four categories.

### Direct Questions

The interviewer names the system explicitly. These questions test breadth: can you identify the subsystems, describe data flow, and justify design choices across the entire stack.

1. "Design a URL shortener like TinyURL or Bitly that handles 100 million redirects per day and 10 million new URLs per day." — **Amazon, Google, Meta, Stripe.** The interviewer expects a complete architecture: key generation, database schema, caching, analytics, and scaling strategy.

2. "How would you generate unique short codes for a URL shortener? Compare three approaches with their trade-offs." — **Microsoft, Uber, LinkedIn.** This question isolates the key generation subsystem. The interviewer tests your understanding of hash-based, sequential, and Snowflake strategies, including collision probability math and throughput limits.

3. "Design the database schema for a URL shortener that supports custom slugs, expiration, and click tracking." — **DoorDash, Pinterest, Snap.** The interviewer tests data modeling: column types, indexes, partitioning, and the trade-off between a single table and separate tables for different concerns.

4. "How does a URL shortener handle analytics? Design the click tracking pipeline from redirect to dashboard." — **Twitter, Reddit, Spotify.** This tests stream processing knowledge: Kafka partitioning, deduplication, enrichment, aggregation, and time-series storage.

### Design Questions (topic is a sub-problem)

The shortener appears inside a larger system design. You must discover the constraints by asking questions about the context.

5. "Design a social media platform with a 140-character post limit. Users must be able to share links without breaking the character limit." — **Twitter (real interview).** The shortener is one component of a larger feed system. The interviewer watches whether you identify that links consume characters and that the shortener must be transparent to the user.

6. "Design an SMS marketing platform. Each message is limited to 160 characters and must include a trackable link for click-through analysis." — **Twilio, Braze.** The constraint is extreme: a 180-character URL with tracking parameters cannot fit in a single SMS segment. You must shorten the URL and choose between 301 and 302 based on analytics needs.

7. "Design a QR code menu system for restaurants. The QR code must be scannable at 10mm x 10mm and the link must never expire." — **Toast, Square.** The interviewer tests constraint-driven design: the QR code size drives the short code length. A version 2 QR code (25x25 cells) fits a 24-character URL, so the short domain plus code must stay under that limit.

8. "Design a link preview system for a messaging app. When a user shares a URL, the app must display the title, description, and image from the destination page." — **WhatsApp, Telegram, Slack.** The shortener is part of a larger pipeline: URL detection, content fetching, Open Graph parsing, and cache management. The interviewer tests your understanding of prefetch behavior and how it affects click analytics.

### Debugging Questions

The interviewer describes symptoms and asks you to diagnose the root cause. These test your systematic debugging approach and understanding of failure modes.

9. "Our URL shortener returns 404 for short codes that were created 5 seconds ago. The database row exists. The cache entry was set at creation time. What is wrong?" — **Meta (real interview).** Root cause: DNS propagation delay, CDN edge cache not invalidated, or cross-region replication lag. The interviewer tests whether you check the network path before the application path.

10. "Click counts in the analytics dashboard are 8x higher than expected for one short code. The redirect endpoint shows a normal number of 302 responses. What is happening?" — **Stripe.** Root cause: bot traffic from link preview fetch (Twitterbot, Facebook crawler, Slack fetcher) inflating counts. The interviewer tests whether you know that prefetch events generate redirects but should be filtered from human click counts.

11. "During a Black Friday campaign, P99 redirect latency jumped from 8ms to 3 seconds. Redis CPU is at 100 percent on one node. Other Redis nodes are at 40 percent. What is the root cause?" — **Amazon (real interview).** Root cause: hot key meltdown. A single short code on one Redis shard receives all the traffic. The interviewer tests whether you identify the hot key pattern and propose shard-level mitigation.

### Trade-off Questions

The interviewer presents conflicting requirements and asks for a decision with justification.

12. "Your CMO wants accurate click analytics. Your CTO wants to minimize infrastructure costs. You must choose between 301 and 302 redirects for a URL shortener with 15 billion redirects per month. What do you recommend?" — **Google.** The answer is not one or the other. It is a hybrid: 302 for users who need analytics (authenticated users, marketing campaigns), 301 for anonymous users. The trade-off is 3x origin traffic for 302 versus 90 percent traffic reduction for 301.

13. "Should we use a sequential counter or Snowflake IDs for key generation? We expect 50,000 new URLs per second at peak. We also need time-ordered short codes for debugging." — **Uber.** The answer is Snowflake. A sequential counter in Redis supports approximately 100,000 INCR operations per second, so technically it works. But Snowflake provides time-ordered IDs without the coordination overhead and without the single point of failure.

14. "Should we use MySQL, Cassandra, or DynamoDB for a URL shortener that spans 5 regions?" — **Amazon.** The answer depends on consistency requirements. If you need strong consistency on writes (prevent duplicate short codes), MySQL or a NewSQL database is appropriate. If you can tolerate eventual consistency with conflict resolution, Cassandra or DynamoDB provides better multi-region write performance.

## What the Interviewer Is Actually Testing

Each question category evaluates specific competencies. Understanding what the interviewer is looking for per category helps you calibrate your answer.

### Direct Questions: Breadth and Decomposition

The interviewer tests whether you can decompose a familiar problem into subsystems and describe how they interact. The signal for a 4/4 answer is that you identify the six subsystems — key generation, storage, caching, redirect engine, analytics pipeline, and administration — and explain the data flow between them before diving into any single component.

A 3/4 answer names the subsystems but dives into one (usually key generation) without establishing the full architecture. The interviewer has to pull you back to the big picture.

A 4/4 answer says: "This system has six logical subsystems. Let me describe the high-level data flow first, then I will deep-dive into each one. The read path is separate from the write path. The read path handles 99 percent of traffic and requires sub-10ms latency. The write path handles 1 percent of traffic and can tolerate up to 200ms."

### Design-as-Sub-Problem: Constraint Discovery

The interviewer tests your ability to discover hidden constraints by asking questions. The signal for a 4/4 answer is that you identify constraints before making design decisions.

A 3/4 answer assumes the shortener requirements based on the public TinyURL model: permanent links, no analytics, sequential keys. If the actual context is SMS marketing, this design misses the 160-character limit constraint and the need for per-campaign analytics.

A 4/4 answer asks: "Is this for SMS, email, or social media? What is the character limit? Do we need per-link analytics? Do links expire? Can users choose custom slugs?" Each answer changes the design. For SMS, the short code must be 6 characters max to stay under the segment limit. For email, prefetch behavior from email clients inflates click counts and must be filtered.

### Debugging Questions: Systematic Traversal

The interviewer tests your ability to traverse the system layer by layer, eliminating possibilities until the root cause emerges. The signal for a 4/4 answer is that you start with the most likely cause (cache) before moving to less likely causes (data corruption).

A 3/4 answer jumps to a conclusion: "The database must have lost the data." This shows no systematic debugging process.

A 4/4 answer walks through the layers methodically: "First, I would check whether the short code exists in the database by running a direct query. If it exists, I would check the cache entry for the same code. If the cache is missing, I would examine the cache write at creation time — was it a write-through or a write-behind? I would also check DNS propagation: is the request reaching the right region? Finally, I would check the CDN cache invalidation: was the CDN edge serving a stale 404 from a previous failed lookup?"

### Trade-off Questions: Judgment and Quantification

The interviewer tests your ability to quantify trade-offs and defend a position. The signal for a 4/4 answer is that you provide concrete numbers for both sides of the trade-off and make a recommendation based on the stated priorities.

A 3/4 answer states an opinion without numbers: "Use 302 because we need analytics." This is correct but shallow.

A 4/4 answer provides the full analysis: "With 15 billion redirects per month, 301 reduces origin traffic by 90 percent, saving 150 application server hours per month. That is approximately $3,000 in compute cost. The trade-off is that repeat visitors are never counted. If the marketing team needs accurate click data to optimize campaigns, the $3,000 monthly cost is easily justified by the campaign optimization value. I recommend 302 for links with analytics requirements and 301 for default links. The user specifies which type when creating the link."

## Level-by-Level Expectations

### L3 / L4 (Early Career)

**What you must know:**
- The core flow: POST /shorten creates a mapping, GET /{code} redirects to the original URL
- At least one key generation strategy (hash-based is the most common choice at this level)
- The difference between 301 and 302 redirects
- Why a cache helps: fewer database queries, lower latency
- A basic database schema: short_code, long_url, created_at
- What base-62 encoding is and why it is used

**Good answer:** "I would build a URL shortener with a hash-based key generator. When a user submits a URL, I hash it with MD5, take the first 7 bytes, encode them in base-62 to get a 7-character short code, and store the mapping in a MySQL table. The short_code column is the primary key. When a visitor accesses the short URL, I look up the code in the database and return a 302 redirect. I add a Redis cache in front of the database. On a redirect request, I check Redis first. If the code is in Redis, I return the redirect without querying the database. The cache uses an LRU eviction policy with a 24-hour TTL."

**What distinguishes 3/4 from 4/4:** The 4/4 candidate explains why they chose 7 characters (62^7 = 3.5 trillion, enough for the expected scale). They also mention collision handling: "If two URLs produce the same hash, I check whether the existing mapping matches. If it matches a different URL, I append a counter to the URL and rehash. This adds a retry loop but collisions are rare at our scale." The candidate also mentions that the cache is populated at creation time (write-through) so the first visitor does not miss.

**Common L3/L4 mistakes:** Proposing a single server with no cache. Not specifying the redirect type. Not calculating namespace capacity. Proposing a random short code without explaining collision handling.

### L5 (Senior Engineer)

**What you must know:**
- All three key generation strategies (hash, sequential, Snowflake) with trade-offs and specific numbers
- Multi-tier cache design: L1 (local in-memory), L2 (Redis cluster), L3 (CDN edge)
- Database indexing strategy and query optimization for primary key lookups
- Asynchronous analytics pipeline with Kafka or Kinesis
- Rate limiting, URL validation, and abuse prevention
- Capacity estimation for storage, throughput, and cost at the stated scale

**Good answer:** "I would use Snowflake IDs for key generation because we expect 115 URL creations per second at peak and want time-ordered codes. The 64-bit Snowflake ID uses 41 bits for a millisecond timestamp, 10 bits for a worker ID supporting 1,024 workers, and 12 bits for a sequence giving 4,096 IDs per millisecond per worker. Each worker generates IDs independently with zero network coordination. The ID is encoded in base-62 to produce a 7-character short code."

"On the caching side, I use a three-tier hierarchy. L1 is a local LRU cache on each application server holding 10,000 entries with a 1-minute TTL, handling 60 percent of reads at 0.05ms. L2 is a Redis cluster with 8 nodes and LRU eviction at a 24-hour TTL, handling 35 percent of reads at 0.5ms. L3 is the database read replicas, handling 5 percent of reads at 5ms."

"For the database, I use MySQL with one primary for writes and three read replicas. The table is hash-partitioned by short_code prefix into 64 partitions for even data distribution. The primary key is the short_code, enabling single-row lookups through the B-tree index."

"The analytics pipeline uses Kafka with 16 partitions. The redirect engine emits a click event asynchronously — fire-and-forget to an in-memory buffer that flushes to Kafka every 100ms. The stream processor aggregates counts into a time-series database and stores raw events in ClickHouse for 90 days."

**What distinguishes 3/4 from 4/4:** The 4/4 candidate provides specific numbers for every claim and raises failure modes proactively. "The Redis cluster needs 8 primary nodes with 16 GB each to maintain a 95 percent cache hit rate for the 10 million hot entries. If a node fails, consistent hashing redistributes the keys. During redistribution, the database must handle 5x the normal query load. I configure the database connection pool with 75 connections and implement a circuit breaker that returns 503 if latency exceeds 100ms." The candidate also compares alternatives: "I chose Snowflake over sequential because the Redis counter becomes a throughput bottleneck above 100,000 creates per second. I chose Snowflake over hash-based because collision retries at 115 creates per second with 10 million existing entries have a 14 percent probability per new entry."

**Common L5 mistakes:** Not mentioning the analytics pipeline. Not discussing failure modes (hot key, cache stampede). Proposing synchronous click counting in the database. Not addressing cross-region deployment.

### L6 (Staff Engineer)

**What you must know:**
- Everything at L5 level, plus:
- Multi-region active-active deployment with code prefix routing
- Cross-region consistency models and replication lag management
- Hot key detection and automated mitigation strategies
- Cost optimization across infrastructure layers with per-unit cost analysis
- SLA and SLO definition with error budget management for each endpoint
- Capacity planning projecting 3-5 years of growth

**Good answer:** "I would design a multi-region active-active deployment with three regions: us-east-1, eu-west-1, and ap-southeast-1. Each region owns a prefix range of short codes determined by the first base-62 character of the code. For example, codes starting with 0-3 belong to us-east-1, codes starting with 4-7 belong to eu-west-1, and codes starting with 8-9 and a-g belong to ap-southeast-1. The CDN routes redirect requests to the owning region based on the first character of the path."

"Each region has its own Redis cluster and database primary. Writes go to the local primary. Asynchronous MySQL binlog replication streams writes to the other regions with 1-3 seconds of lag. Read-after-write consistency within 5 seconds is guaranteed by a write-forward mechanism: if a region receives a redirect for a code it does not own and the code was created within the last 5 seconds, it proxies the request to the owning region."

"Hot key mitigation uses a three-tier approach. L1: each application server caches the top 10,000 codes in local memory, so a hot key is cached on every server that serves it. L2: each Redis shard has 3 read replicas that distribute read load. L3: if a single short code exceeds 50,000 requests per second, the application returns HTTP 429 with a 1-second Retry-After header, which the client respects."

"Cost analysis: the per-region infrastructure is approximately $9,200 per month. Three regions cost $27,600 per month. The cost per 1,000 redirects is approximately $0.0006, which is well under the $0.001 target for a free consumer service. If we monetize through enterprise analytics, the per-customer cost is negligible."

**What distinguishes 3/4 from 4/4:** The 4/4 candidate drives the conversation proactively, raising multi-region architecture before the interviewer asks. They also discuss operational maturity: "The multi-region design requires an SRE team on-call for each region. The database team owns the replication pipeline. I would structure the teams around these ownership boundaries and set up a weekly cross-region replication health review."

**Common L6 mistakes:** Designing for the average case without discussing cost. Not addressing how the system recovers from partial failures (one region goes down). Not discussing the migration path from single-region to multi-region.

### L7 / L8 (Principal / Distinguished Engineer)

**What you must know:**
- Everything at L6 level, plus:
- Build-versus-buy analysis with total cost of ownership over 5 years
- Organizational impact: team structure, API governance, cross-team dependencies
- Long-term technical debt management: namespace exhaustion, schema evolution, deprecation
- Security architecture: audit trails, compliance (GDPR right to deletion, CCPA), penetration testing
- Business-level strategy: should the company commoditize the shortener or make it a product?

**Good answer:** "Before discussing the architecture, I would evaluate whether building a custom URL shortener is the right decision. Bitly Enterprise costs approximately $1,000 per month for 100,000 links with analytics. Building internally costs $28,000 per month in infrastructure plus 4 engineers at $320,000 per year. The build-versus-buy break-even point is 18 months. If the business needs custom analytics, private-label domains, and HIPAA compliance, building makes sense."

"If we build, I would integrate the shortener into the company's existing data platform. Rather than operating independent infrastructure, the shortener shares the company MySQL cluster and Kafka infrastructure. This reduces infrastructure cost from $28,000 to $8,000 per month. The shortener team sits in the platform engineering org, not a product team, because every product team that shares links depends on it."

"For the API design, I version the API from day one. The v1 API is stable for 3 years. Breaking changes go in v2 with a 2-year deprecation window for v1. This lets product teams integrate without worrying about breaking changes."

"Long-term, I plan for code exhaustion. At 10 million new URLs per day, the 7-character base-62 namespace (3.5 trillion codes) lasts 350 years. But as short code length approaches exhaustion, I prepare a migration from 7-character to 8-character codes with a 6-month dual-read and dual-write migration window."

**What distinguishes pass from fail at this level:** A failing answer dives into technical architecture without questioning whether to build. A passing L7 answer starts with business analysis: "Let me evaluate whether building is the right decision given the company's strategic priorities." The candidate also discusses organizational impact: "This system touches every product team that shares links. The API contract must be stable. The on-call rotation must be staffed 24/7. The cost of operating the shortener is dominated by people, not infrastructure."

## The FAANG Framework

Use this five-step framework for any URL shortener design question.

### Step 1: Clarify Requirements (3 minutes)

Ask clarifying questions in this order to establish scope before designing.

**Functional requirements:**
- "What operations does the system support? Create, redirect, delete, update, analytics?"
- "Do users need custom slugs, or are auto-generated codes sufficient?"
- "Do short URLs expire, or are they permanent?"
- "What analytics data do we need? Click counts, referrer, geographic, device type?"
- "Do we need authentication? API keys for programmatic access?"

**Non-functional requirements:**
- "How many new URLs are created per day? Per second at peak?"
- "How many redirect requests per second at peak?"
- "What is the latency target for redirects? What about P50 and P99?"
- "What is the availability target: 99.9 percent, 99.99 percent?"
- "How much data do we store? Retention period for analytics?"

**Confirm by repeating:** "So the system handles 10 million new URLs per day, 15,000 redirects per second at peak, with 99.99 percent availability and 20ms P99 redirect latency. We also need click analytics with geographic data. Did I miss anything?"

### Step 2: Estimate Scale (2 minutes)

Calculate the scale requirements using the answers from Step 1.

```
New URLs:     10 million per day = 115 per second
Redirects:    500 million per day = 5,800 per second average
                                      15,000 per second peak
Storage:      10 million * 300 bytes = 3 GB per day
              3 GB * 365 = 1.1 TB per year
Cache:        10 million hot entries * 344 bytes = 3.4 GB
              With 3 replicas: 10.3 GB total
Bandwidth:    15,000 redirects * 300 bytes = 4.5 MB/s peak
              Monthly: 500 million * 300 bytes = 150 GB/month
```

State what each number means: "At 15,000 redirects per second peak, the database cannot serve every request directly. We need a cache layer that absorbs at least 95 percent of reads. The storage grows at 1.1 TB per year, so we need to partition the database within 2 years."

### Step 3: High-Level Design (5 minutes)

Present the architecture at a high level. Identify the components and the data flow.

**Components:**
- Edge layer: CDN + DNS geo-routing
- API gateway: rate limiting, authentication, request validation
- Application servers: stateless, horizontally scalable
- Cache layer: Redis cluster (primary read path)
- Database: MySQL with primary and read replicas
- Analytics pipeline: Kafka + stream processor + data warehouse

**Read path:** Browser -> CDN -> API gateway -> App server -> Redis cache (hit: 0.5ms) or database (miss: 5ms) -> 302 redirect

**Write path:** Client -> API gateway -> App server (validate, generate code, check blocklist) -> database primary -> Redis (write-through) -> response

State the key choices and why: "I am using 302 redirects for accurate click analytics. I am using Snowflake IDs for key generation because we need time-ordered codes without coordination overhead. I am using write-through caching so the first visitor never misses the cache."

### Step 4: Deep Dive (10 minutes)

The interviewer will choose one or two subsystems. Be ready for these common deep dives.

**Key generation deep dive:** "The Snowflake ID has 41 bits for the timestamp, 10 bits for the worker ID, and 12 bits for the sequence number. Each worker generates 4,096 IDs per millisecond. With 1,024 workers, total throughput is 4 million IDs per second. Clock skew is handled by blocking generation until the clock catches up. Sequence overflow pauses for 1ms until the next millisecond tick. Worker IDs are assigned via a ZooKeeper coordination service with a 60-second TTL lease."

**Cache deep dive:** "The three-tier cache hierarchy is L1 (local LRU, 10K entries, 0.05ms, handles 60 percent), L2 (Redis cluster, 50M entries, 0.5ms, handles 35 percent), L3 (database replicas, 5ms, handles 5 percent). Cache stampede is prevented by jittered TTLs and probabilistic early expiration. Cache invalidation on updates uses an explicit delete command sent to Redis before the database write commits."

**Analytics deep dive:** "The redirect engine emits a click event to an in-memory buffer. Every 100ms, the buffer flushes 1,000 events to Kafka. A Kafka Streams consumer deduplicates by event_id using a 5-minute window, enriches with GeoIP data and device classification, and aggregates counts into 5-minute time buckets in a time-series database. Raw events are stored in ClickHouse for 90 days. The dashboard queries the time-series database for sub-second responses."

### Step 5: Trade-offs Discussion (3 minutes)

Summarize the key trade-offs and justify your choices.

"First, I chose 302 redirects over 301. This gives us per-visit analytics but costs 3x more origin traffic. The additional infrastructure cost is approximately $2,000 per month at our scale. The marketing team needs accurate click data to attribute campaigns, so the cost is justified."

"Second, I chose Snowflake IDs over a sequential counter. Snowflake eliminates the coordination round-trip, reducing create latency by 5ms. The trade-off is clock skew risk, which I mitigate with a clock monitoring daemon that alerts at 10ms of skew."

"Third, I chose async analytics over synchronous event processing. The analytics pipeline never blocks the redirect path. The trade-off is that 0.1 percent of events may be lost if the application server crashes before the Kafka buffer flushes. We accept this loss because 99.9 percent analytics accuracy is sufficient."

### Step 6: Wrap Up (1 minute)

Summarize the design and identify next steps.

"This system handles 15,000 redirects per second with 10ms P99 latency. It uses Snowflake IDs for key generation, a three-tier cache hierarchy, MySQL with read replicas and hash partitioning, and an async analytics pipeline with Kafka. The architecture supports multi-region expansion through code prefix routing."

"Next steps: I would add CDN caching for anonymous 301 redirects to reduce origin load. I would implement a Bloom filter for faster URL deduplication. I would add anomaly detection on the analytics pipeline to detect click fraud. Given more time, I would design a full multi-region deployment with regional code ownership and cross-region replication."

## Phrase Bank

Use these phrases in your interview to demonstrate depth and precision.

1. **"The 7-character base-62 namespace gives us 3.5 trillion combinations, which is sufficient for 350 years at our creation rate."** — Context: explaining short code length choice. Shows you calculated the namespace against the growth rate.

2. **"The cache is the primary read path, not an optimization."** — Context: describing the architecture. Shows you understand that 95 percent of reads must come from the cache for the system to meet latency targets.

3. **"Snowflake IDs eliminate the coordination round-trip at the cost of clock dependency."** — Context: comparing key generation strategies. Shows you understand the fundamental trade-off between the two approaches.

4. **"The write path is 1 percent of traffic and can tolerate 200ms latency. The read path is 99 percent of traffic and must complete in 10ms."** — Context: explaining why you separate the read and write paths. Shows you derive latency targets from the traffic distribution.

5. **"I use write-through caching on create so the first visitor never experiences a cache miss."** — Context: cache population strategy. Shows you considered the first-request experience.

6. **"Hot key meltdown is the most dangerous failure mode for a URL shortener. I prevent it with three tiers of defense: local L1 cache, replicated Redis reads, and per-code rate limiting at 50,000 requests per second."** — Context: discussing failure modes. Shows you identified the highest-risk scenario and have a layered mitigation.

7. **"A 301 redirect saves 90 percent of origin traffic but loses per-visit analytics. I would use 302 for users who need accurate counts and 301 for anonymous visitors."** — Context: the redirect type decision. Shows you split the decision by user type rather than making a blanket choice.

8. **"The database connection pool must survive a full cache failure. With 75 connections and 15,000 queries per second, the pool saturates at 5ms per query. I add a circuit breaker that returns 503 if database latency exceeds 100ms."** — Context: database resilience. Shows you can calculate pool requirements and design protection mechanisms.

9. **"Prefix-based code ownership enables multi-region routing without a global lookup. The first character of the short code determines which region owns the mapping."** — Context: multi-region architecture. Shows you understand how to distribute data across regions without a coordination service.

10. **"The Bloom filter for URL deduplication consumes 120 MB and provides sub-microsecond existence checks. The 1 percent false positive rate means 1 in 100 URLs triggers an unnecessary database lookup."** — Context: optimization. Shows you know the specific cost and accuracy trade-off of the data structure.

11. **"Cache stampede probability increases with batch-created links. Marketing campaigns create 500,000 links at midnight with identical TTLs. I add jittered TTLs to spread expiration across a 2-hour window."** — Context: cache strategy. Shows you considered real-world usage patterns.

12. **"The short code namespace is shared between generated codes and custom slugs. I reserve a prefix range for custom slugs to prevent collision with auto-generated codes."** — Context: custom slug support. Shows you identified the namespace collision problem.

13. **"Click deduplication removes prefetch traffic from analytics. Twitterbot and Facebook crawler user agents are filtered out before counting human clicks."** — Context: analytics accuracy. Shows you understand that link previews inflate click counts.

14. **"The redirect path must handle 15,000 requests per second with a P99 latency of 20ms. Every millisecond of database query time beyond 5ms is a design failure."** — Context: performance targets. Shows you define measurable targets and hold the design accountable to them.

15. **"Capacity planning for 5 years: we need 5.5 TB of database storage, 64 GB of cache memory distributed across 8 Redis nodes, and 20 application servers at 5,000 req/s each. The cost is dominated by storage, not compute."** — Context: capacity planning. Shows you project growth and identify the real cost driver.

16. **"The coordinatorless design of Snowflake IDs means the key generation system has no single point of failure."** — Context: distributed system design. Shows you understand how the architecture supports high availability.

17. **"The create endpoint uses a database transaction to ensure atomicity of the insert and the cache write. If the cache write fails, the database still holds the mapping."** — Context: write path reliability. Shows you design for partial failures.

18. **"The analytics system uses a separate database from the URL mapping store so analytics queries never compete with redirect queries."** — Context: workload isolation. Shows you separate hot and cold paths.

## Red Flags

Avoid these mistakes. Each one signals a gap in understanding that can affect your rating.

**Red flag 1: Proposing random short codes without discussing collision probability.** Saying "I would generate a random 7-character string" without calculating the birthday paradox shows you do not understand collision risk. At 10 million entries, the collision probability for random 7-character codes is approximately 14 percent. You must explain how you handle collisions or why you chose a deterministic strategy.

**Red flag 2: Using a single database node without discussing scaling.** A single MySQL instance handles approximately 10,000 simple queries per second. At 15,000 redirects per second, it fails. You must discuss caching, read replicas, or partitioning. Not mentioning scaling shows you only designed the happy path.

**Red flag 3: Forgetting about analytics entirely.** A production URL shortener without analytics is not viable. Every company needs to track clicks, referrers, and geographic data. If the requirements do not mention analytics, raise it yourself. Saying "we do not need analytics" misses a major subsystem.

**Red flag 4: Saying "I would use 301 redirects because they are faster" without discussing the analytics trade-off.** 301 redirects are faster for repeat visitors because browsers cache them. But every cached redirect bypasses the shortener, losing analytics. The interviewer expects you to identify this trade-off, not just pick the fast option.

**Red flag 5: Not discussing cache stampede or hot key scenarios.** These are the two most common production incidents for URL shorteners. If you design a cache layer but do not discuss what happens when 500,000 links expire simultaneously or when one link goes viral, you miss the hardest operational problems.

**Red flag 6: Proposing MySQL without discussing the schema in detail.** "I would use MySQL" is not enough. The interviewer expects the DDL: column types, primary key, indexes, character set, and partitioning strategy. A candidate who cannot articulate the schema has not designed past the conceptual level.

**Red flag 7: Ignoring security.** A URL shortener can host malicious links behind an opaque redirect. If you do not mention URL blocklists, abuse detection, or rate limiting on creation, you miss critical non-functional requirements. Mention domain blocklists, Safe Browsing API integration, and rate limiting.

**Red flag 8: Designing for the average case without mentioning the peak.** "We handle 10 million URLs per day" is the average. The system must survive the peak: 15,000 redirects per second during a viral event. Size infrastructure for 2-3x the average as a safety margin. If you design for the average, the system fails during the peak.

**Red flag 9: Not distinguishing between create and redirect as different performance paths.** The create path handles 1 percent of traffic and can tolerate 200ms. The redirect path handles 99 percent of traffic and needs 10ms. If you design the same infrastructure for both, the create path is over-provisioned and the redirect path is under-provisioned. Call out the separation explicitly.

**Red flag 10: Saying "the cache is optional."** A URL shortener at any production scale cannot operate without a cache. 15,000 redirects per second hitting the database directly requires 150+ database connections and results in 50ms+ latency. The cache is the primary read path, not an optional optimization.

**Red flag 11: Not mentioning how you handle expired or deleted links.** The redirect engine must check whether a link is still active before redirecting. A deleted link should return 410 Gone, not 302 to a destination that no longer exists. Forgetting this check means deleted links continue to redirect.

**Red flag 12: Proposing a single-region design without discussing cross-region latency.** A single region adds 100-300ms of latency for users on other continents. At a global scale, this is unacceptable. Even if the interviewer does not ask about multi-region, mention that you would expand to multiple regions and describe the routing approach.

## Practice Drills

### Drill 1: Hash-Based Collision Handling (5 minutes)

**Prompt:** You use hash-based key generation with MD5 and base-62 encoding for a 7-character short code. The system has 50 million existing mappings. Calculate the collision probability for the next 1 million new URLs. Design the collision handling loop including the expected number of retries per new URL.

**Key points to hit:**
- Birthday paradox formula: P(collision) = N^2 / (2 * 62^7) where N = 50 million
- P(collision) = (5e7)^2 / (2 * 3.5e12) = 2.5e15 / 7e12 = 357 collisions expected in the next million creates
- Collision handling: append a salt ("url:1", "url:2", etc.) and rehash until a unique code is found
- Expected retries per collision: 1 (first retry succeeds because the salt changes the hash)
- Average extra latency per create: 357 collisions per 1 million creates * 1 retry * 5ms per hash = 0.0018ms average, negligible

**Common mistake:** Not calculating the collision probability and assuming "hash collisions never happen." The birthday paradox guarantees collisions at scale. Design for them.

### Drill 2: Write-Through vs Write-Behind Cache (10 minutes)

**Prompt:** Compare write-through and write-behind cache strategies for the create path of a URL shortener. The system creates 115 new URLs per second and serves 15,000 redirects per second. Which strategy do you choose and why? What happens to data consistency in each approach? How does each affect the latency of the create endpoint?

**Key points to hit:**
- Write-through: the create endpoint writes to both database and cache before returning. This adds 1ms of latency to the create path (the cache round-trip) but guarantees the first visitor never misses. This is the standard choice for URL shorteners.
- Write-behind: the create endpoint writes to the database only. A background process writes to the cache. The first visitor may miss the cache and hit the database. This saves 1ms on the create path but degrades the read path for new links.
- Consistency: write-through provides read-your-writes consistency for the cache. Write-behind has a window where the database has the mapping but the cache does not.
- Recommendation: write-through. The create path (115 req/s) can tolerate 1ms of additional latency. The redirect path (15,000 req/s) benefits from the cache being populated immediately.

**Common mistake:** Choosing write-behind for "better performance" without calculating that the create path is 1 percent of traffic and can afford the extra latency. Optimize for the 99 percent path, not the 1 percent path.

### Drill 3: Identify the Anti-Pattern (10 minutes)

**Prompt:** A colleague proposes the following database schema for a URL shortener. Identify every anti-pattern and explain the production impact of each.

```sql
CREATE TABLE url_mappings (
    id INT AUTO_INCREMENT PRIMARY KEY,
    short_code VARCHAR(8),
    long_url TEXT,
    click_count INT,
    created_at DATETIME
);
```

**Key points to hit:**
- `short_code` has no UNIQUE index: duplicate short codes are possible. This breaks the core invariant of a URL shortener: one code maps to one URL.
- `long_url` uses TEXT instead of VARCHAR(2048): TEXT has a different storage engine behavior in MySQL. VARCHAR(2048) is the correct choice because URLs have a known maximum length.
- `click_count` is in the mapping table: every redirect requires an UPDATE on this column, causing row-level lock contention. At 15,000 req/s, requests queue behind the UPDATE lock and latency spikes to 500ms.
- `click_count` uses INT instead of BIGINT: 2.1 billion max. A popular short URL can exceed this in a month. Use BIGINT.
- `created_at` uses DATETIME instead of TIMESTAMP: TIMESTAMP supports timezone conversion and has a smaller storage footprint (4 bytes vs 8 bytes).
- No index on `short_code`: every lookup requires a full table scan. At 50 million rows, a table scan takes seconds. Every redirect would time out.

**Common mistake:** Identifying only one or two anti-patterns. The schema has six distinct issues. A strong answer lists all six and explains each one's production impact with concrete numbers.

### Drill 4: Multi-Region Recovery (15 minutes)

**Prompt:** Your URL shortener operates in three regions. A major incident occurs: the us-east-1 data center loses power for 4 hours. The database primary in us-east-1 goes offline. During the outage, users in North America cannot create short URLs or access existing URLs owned by us-east-1. Walk through the incident response procedure. What do you do in the first 5 minutes? The first 30 minutes? The first 2 hours? How do you restore service? How do you ensure data integrity when the data center comes back online?

**Key points to hit:**
- First 5 minutes: detect the outage via health check alerts. Verify that the outage is regional (eu-west-1 and ap-southeast-1 are healthy). Update the status page.
- First 30 minutes: promote the us-east-1 read replica in eu-west-1 to primary. Update DNS routing to redirect us-east-1 code prefixes to eu-west-1. Verify that redirects for us-east-1 codes now resolve from eu-west-1.
- First 2 hours: scale up the eu-west-1 database and cache to absorb the additional load from us-east-1 codes. Monitor replication from the original us-east-1 primary (it is offline, so no replication flows). Accept that codes created in the last 5 minutes before the outage may be lost (RPO of 5 minutes).
- When us-east-1 comes back: restore the us-east-1 database from the last backup and apply binlog up to 5 minutes before the outage. Manually reconcile any codes created in us-east-1 during the outage (if writes were re-routed to eu-west-1, those codes now have duplicate entries). Re-replicate data from eu-west-1 back to us-east-1. Fail back traffic to us-east-1.
- Data integrity check: run a reconciliation query comparing us-east-1 and eu-west-1 databases. Sample 10,000 codes from each prefix range and verify they resolve to the same URL. Fix any discrepancies manually.

**Common mistake:** Assuming the system recovers automatically. Multi-region failover requires manual verification at every step. The RPO and RTO targets must be validated through quarterly drills, not assumed from the architecture alone.

### Drill 5: URL Validation and Abuse Prevention (5 minutes)

**Prompt:** Design the URL validation and abuse prevention system for a public URL shortener. The system receives 10 million new URLs per day from 500,000 users. List every validation check you perform at creation time and every abuse detection mechanism you operate at redirect time. Order them by priority (most important first).

**Key points to hit:**
Creation-time checks:
1. URL format validation: must be a valid HTTP or HTTPS URL. Reject javascript:, data:, file: schemes.
2. Length limit: maximum 2048 characters.
3. Internal IP check: reject URLs pointing to 10.x.x.x, 172.16-31.x.x, 192.168.x.x, 127.0.0.1, and 169.254.x.x.
4. Domain blocklist: check against known phishing, malware, and spam domains using a Bloom filter with 100 million entries.
5. Safe Browsing API: submit the URL to Google Safe Browsing for real-time classification.
6. Rate limiting: maximum 100 creations per minute per user. Token bucket algorithm with burst capacity of 500.

Redirect-time checks:
1. Active flag: if the link was deactivated since creation, return 410 Gone.
2. Re-blocklist check: the destination domain may have been compromised since creation. Check the blocklist again.
3. Abuse report flag: if the link received 10 or more abuse reports, automatically deactivate and return 410.
4. Click rate anomaly: if a link receives more than 100,000 clicks in 1 hour, flag it for manual review as potential malware distribution.

**Common mistake:** Only checking at creation time and never at redirect time. A URL that was safe at creation can be compromised later. Redirect-time checks catch compromised links that creation-time checks missed.

### Drill 7: Follow-Up Question Practice (10 minutes)

Practice answering these follow-up questions. Each question tests a specific competency that the interviewer wants to verify.

**Q: How do you handle cache miss stampede?**
A: I use probabilistic early expiration. Before a cache entry expires, I refresh it with a probability that increases as the entry approaches its TTL. This smooths the refresh load across time rather than having all entries expire at once. I also add jittered TTL values (random offset of plus or minus 10 percent of the base TTL) to prevent batches of entries from expiring simultaneously.

**Q: How do you handle the case where two users request the same custom slug at the same time?**
A: The database's unique constraint on the short_code column prevents duplicate inserts. I use INSERT ... ON CONFLICT DO NOTHING in PostgreSQL. The second transaction receives a 409 Conflict response. The client must retry with a different slug.

**Q: Can we use a CDN for 302 redirects?**
A: CDNs can cache 302 responses, but the Cache-Control header must explicitly allow it. However, caching 302 redirects defeats the purpose of using them -- you lose per-visit analytics. For 302 redirects, I set Cache-Control: no-cache, no-store to prevent all caching. This ensures every redirect reaches the origin for accurate click tracking.

**Q: What is your strategy for short code exhaustion?**
A: I monitor namespace utilization as a percentage of the total addressable space. For 7-character base-62 codes, that is 3.5 trillion values. When utilization reaches 60 percent, I trigger planning for code length extension. The migration from 7 characters to 8 characters follows a dual-read and dual-write process with a background backfill job.

**Q: How do you detect and prevent malicious URL shortening?**
A: At creation time, I check the destination URL against a Bloom filter of known malicious domains. The Bloom filter has a 1 percent false positive rate and is rebuilt weekly. At redirect time, I check the destination again because a URL that was safe at creation may have been compromised. For suspicious URLs, I require additional authentication or manual review.

**Q: What happens when Redis runs out of memory?**
A: The allkeys-lru eviction policy removes the least recently accessed entries. The next request for an evicted entry fetches it from the database and repopulates the cache. This is self-healing -- no data is lost, only latency increases for cold entries. I monitor the eviction rate and alert when it exceeds 100 entries per minute.

**Q: Can you use a distributed database like CockroachDB instead of MySQL?**
A: CockroachDB provides automatic sharding and multi-region consistency that would simplify the database layer. The trade-off is higher per-query latency (3-5ms for CockroachDB versus 1-2ms for MySQL in the same region) and higher cost. For a URL shortener where the cache absorbs 95 percent of reads, the database latency is rarely the bottleneck, so MySQL's lower cost and simpler operations are preferable.

**Q: How do you support link previews in social media?**
A: When a short URL is created, I fetch the Open Graph metadata (title, description, image) from the destination URL. This metadata is cached alongside the mapping. When a social media crawler accesses the short URL, the system returns a small HTML page with the Open Graph tags and a meta refresh redirect. This lets the crawler extract the preview information without following the redirect.

**Q: How does your design change for a team with 5 engineers instead of 50?**
A: With a smaller team, I simplify the architecture significantly. I use a single MySQL database with no read replicas and a single Redis instance instead of a cluster. I skip the Kafka-based analytics pipeline in favor of simple database click counting with rate limiting to prevent lock contention. I trade throughput for operational simplicity. The architecture scales vertically until the team has the bandwidth to manage distributed systems.

**"What monitoring metrics are most important for a URL shortener?"**

The five most critical metrics: cache hit rate (target 95 percent+, alert at 90 percent), redirect P99 latency (target under 20ms, alert at 50ms), database query throughput (alert at 80 percent of connection pool capacity), Kafka consumer lag (alert at 100,000 events), and error rate by status code (4xx alert at 1 percent, 5xx alert at 0.1 percent). **"How would you design a read-after-write consistency guarantee?"**

When a user creates a short URL and immediately visits it, the redirect must succeed. Guarantee this by: populating the cache in the same request as the database write (write-through), returning the short code to the client only after the cache write succeeds, and routing the user's own redirect to the same region where the link was created. For cross-region consistency, use a session cookie that pins the user to the creating region for the first 5 seconds after creation.

## Follow-Up Questions to Expect

Interviewers often ask follow-up questions after you present your initial design. These test your depth of knowledge and ability to adapt under pressure.

**"How would your design change if the requirement was 1 billion redirects per day instead of 100 million?"**

Your response should focus on: adding a CDN layer for 301 redirects, increasing Redis cluster size from 4 to 16 nodes, adding a local L1 cache on each application server, sharding the database by short code prefix, and deploying to at least 2 regions for geographic distribution. The key insight is that the read path scales by adding cache layers and read replicas, while the write path scales by sharding the database and using decentralized key generation.

**"How would your design change if all links must expire after 30 days?"**

This simplifies the design significantly. The database grows at a bounded rate (30 days of links). Old data is automatically removed. The namespace is recycled. You can use shorter short codes (6 characters instead of 7) because the namespace never reaches capacity. The analytics pipeline retains only 30 days of data, reducing storage costs. Cache TTLs match the link TTL, so no manual cache invalidation is needed.

**"How would you handle a migration from 6-character to 7-character short codes?"**

The migration follows a multi-phase rollout: Phase 1 (dual-read): all application servers accept both 6 and 7 character codes. Read requests check both lengths. Phase 2 (dual-write): new links are created with 7-character codes. All existing links get 7-character equivalents assigned. Phase 3 (backfill): a background job generates 7-character codes for all existing mappings. Phase 4 (cutover): the system stops reading 6-character codes. The old codes are archived but not deleted, in case of rollback.

**"What if your shortener needs to support 50 million custom slugs from enterprise customers?"**

Custom slugs require a uniqueness check before creation. With 50 million slugs, the uniqueness check must be efficient. Use a Redis set for custom slug reservation with a Bloom filter for fast existence checks. The Bloom filter supports 50 million entries with 1 percent false positive rate consuming 60 MB of memory. When a user requests a custom slug, check the Bloom filter first (microseconds), then verify with Redis if the filter returns positive. The database is the authoritative source for the reservation.

**"How do you prevent abuse and spam in a public URL shortener?"**

Abuse prevention requires multiple layers. At creation time: check the destination URL against blocklists (Google Safe Browsing, VirusTotal), rate limit per user (100 creates per minute), require email verification for accounts, and scan URLs with an ML model trained on known spam patterns. At redirect time: re-check the destination if it has not been scanned in the last 24 hours, block redirects to newly registered domains (less than 7 days old), and flag links that receive an unusually high number of clicks for manual review. Post-processing: run daily scans of all active links, deactivate links that now point to malicious destinations, and publish a transparency report showing enforcement actions.

**"How would you implement a link preview feature that shows the destination URL before redirecting?"**

The link preview feature requires the shortener to fetch the destination URL's title, description, and Open Graph metadata before showing a preview page. This adds 200-500 milliseconds of latency to the first visit. Cache the preview metadata aggressively: store title, description, image URL, and fetch timestamp in the cache with a 7-day TTL. Generate the preview page as HTML and cache it on the CDN. Only show the preview for first-time visitors or links that have not been visited in the last 24 hours. Return visitors get a direct redirect without the preview.
