# Reliability — Tradeoffs and Decision Making

Every reliability mechanism has a cost. Retries add latency. Circuit breakers increase complexity. Redundancy doubles infrastructure spend. This page helps you make informed decisions by mapping the tradeoffs, defining when each mechanism applies, and identifying common anti-patterns.

## Core Tension: Reliability vs Latency

The fundamental tradeoff: reliability mechanisms add latency to the happy path so the system survives the unhappy path. Every retry, every circuit breaker check, every bulkhead acquire adds time to every request.

</AsciiDiagram>

The happy path adds ~1 microsecond per request for circuit breaker checks. Negligible. The failure path adds the full retry latency: 200ms backoff plus 50ms call time. If the failure rate is 0.1%, the expected latency increase is `0.001 * 250ms = 0.25ms` — invisible.

But if the failure rate is 10%, the expected latency increase is `0.1 * 250ms = 25ms` — noticeable. At a 50% failure rate, the expected increase is `0.5 * 250ms = 125ms` — users see degraded performance before the circuit breaker opens. After the circuit breaker opens (10 seconds at 50% failure rate), latency drops to near zero for rejected calls.

This is the circuit breaker's magic: it trades latency for protection. During the outage detection window (10 seconds), users experience higher latency. After the window closes, rejected calls return instantly.

## Decision Matrix: When to Use Each Mechanism

| Condition | Retry | Circuit Breaker | Bulkhead | Fallback |
|---|---|---|---|---|
| Transient failure (network blip) | Yes — 3 attempts, 200ms backoff | No — circuit does not open | No — pool short-term blip fine | Optional — if stale data acceptable |
| Persistent failure (service down) | No — waste of resources | Yes — open after 50% failure rate | Yes — protect other dependencies | Yes — serve cached or default |
| Slow dependency (high latency) | Conditional — use timeout | Yes — threshold on timeout count | Yes — separate pool so slow calls do not block fast ones | Conditional — if degraded data acceptable |
| Dependency returning errors (5xx) | Yes — retry up to max_attempts | Yes — open on error threshold | Yes — limit concurrent calls | Yes — fall back to secondary |
| Dependency returning garbage data (200 with wrong data) | No — retry does not fix | No — error count not incremented | No — wrong mechanism | No — need validation layer |
| Dependency completely unreachable | No — immediate fail | Yes — open fast | Yes — pool releases immediately | Yes — fall back to alternative |
| Cascading failure | No — retries amplify | Yes — first line of defense | Yes — second line of defense | Yes — keep the system alive |

### Reading the Matrix

**When to retry:** only for transient, idempotent, safe-to-repeat operations. Retry on network timeouts, 503 Service Unavailable, 429 Too Many Requests (with rate-limit-aware backoff). Do not retry on 400 Bad Request, 404 Not Found, or 409 Conflict — these are client errors that retrying will not fix.

**When to circuit break:** whenever a dependency consistently fails. The circuit breaker turns off the spigot so the dependency can recover. Without it, your retries become a denial-of-service attack against the failing service.

**When to bulkhead:** whenever you depend on multiple downstream services with different latency and reliability profiles. The payment service needs isolation from the logging service — you do not want a logging outage to block payments.

**When to fall back:** whenever a degraded user experience is better than no user experience. Serve stale data from cache. Show "temporarily unavailable" instead of a crash. Return safe defaults.

## CAP Theorem and Reliability

The CAP theorem states that a distributed data system can provide at most two of three properties: Consistency, Availability, and Partition Tolerance.

In a distributed system, partitions happen. The CAP theorem forces a choice between consistency and availability during a partition.

**CP systems (Consistency + Partition Tolerance):** When a partition occurs, CP systems stop accepting writes on the minority side to maintain consistency. This means the system is unavailable for writes on that side. Example: ZooKeeper, etcd, HBase.

**AP systems (Availability + Partition Tolerance):** When a partition occurs, AP systems continue accepting writes on both sides. This means the two sides may diverge, and consistency is eventual. Example: Cassandra, DynamoDB, Redis Cluster.

Reliability means different things for CP and AP systems:

- **CP system reliability** means the system correctly rejects writes it cannot make consistent. A reliable CP system has fast, reliable partition detection and fast failover to the majority side. The unavailability window is minimized.
- **AP system reliability** means the system successfully accepts and stores every write, even during a partition. A reliable AP system has robust conflict resolution and bounded divergence.

## Five Failure Modes of Reliability Mechanisms

### Failure Mode 1: Retry Storm

A retry storm occurs when a large number of clients retry simultaneously, overwhelming the failing service. This is the "thundering herd" problem amplified by retry logic.

**Symptoms:** the dependency's request rate spikes to 3-10x normal immediately after it begins failing. The dependency, already struggling, dies under the extra load.

**Prevention:** jitter in backoff, circuit breakers, retry budgets. Without jitter, all clients retry in lockstep. Without circuit breakers, unlimited retries continue indefinitely. Without retry budgets, there is no ceiling on retry volume.

### Failure Mode 2: Cascading Circuit Breakers

One service's circuit breaker opening triggers a wave of circuit breakers opening in upstream services. Service A call Service B. Service B's circuit breaker opens. Service A's call to B fails fast. The failure rate for Service A increases. Service A's circuit breaker opens. Now Service C, which calls A, sees failures. Service C's circuit breaker opens.

**Prevention:** bulkheads isolate dependencies. Graceful degradation absorbs failures without propagating them. Timeouts at each layer prevent chain reactions. Most importantly: a circuit breaker open on an upstream service should not open the circuit breaker on the downstream — the downstream is already at capacity, and upstream retries make it worse.

### Failure Mode 3: Fallback Avalanche

A fallback that calls another service (cache, secondary database) can shift the load from the failing service to the fallback service, overwhelming it too.

**Scenario:** the database is slow. The service falls back to Redis. But Redis handles 1,000 ops/second at the primary database's latency. When the database fails, all 10,000 requests/second hit Redis. Redis becomes the new bottleneck. Now both the primary and fallback are unavailable.

**Prevention:** rate-limit fallback calls. Reserve capacity in fallback systems. Monitor fallback utilization continuously. A fallback that runs at 90% capacity under normal load has no headroom for failover.

### Failure Mode 4: Phantom Retries

A retry succeeds but the caller does not know it succeeded (the response was lost). The caller retries, and the operation runs again. Without idempotency, the operation executes twice.

**Scenario:** `POST /charge` times out. The payment gateway processed the charge but the response packet was dropped. The client retries. The payment gateway charges the credit card again. The customer gets two charges.

**Prevention:** always use idempotency keys for mutation operations. The server deduplicates based on the key. The second call returns the result of the first charge without executing the charge again.

### Failure Mode 5: Self-Inflicted Outage

A misconfigured reliability mechanism causes the outage it was designed to prevent.

**Scenario:** You set the circuit breaker threshold too low (10%). A brief network blip opens the circuit. The circuit stays open for 60 seconds. Millions of requests fail unnecessarily during those 60 seconds. The brief blip becomes a full outage because the circuit breaker was too sensitive.

**Prevention:** set thresholds based on historical failure rates. Start conservative (50% threshold, 30-second window) and tune down as you understand your system's normal failure characteristics. Use gradual threshold reductions over weeks, not days.

## Five Anti-Patterns

### Anti-Pattern 1: Infinite Retries

Retrying forever without a maximum attempt count.

```python
# Bad: infinite retries
while True:
    try:
        result = call_service()
        return result
    except Exception:
        pass  # retry forever
```

This retries indefinitely, consuming threads, connections, and memory. When the service is down, the retries never stop. The system leaks resources until it crashes.

**Fix:** always set `max_attempts` (3-5 for most operations). Move failed messages to a dead-letter queue after exhausting attempts.

### Anti-Pattern 2: Retrying Non-Idempotent Operations Without Idempotency Keys

```python
# Bad: retrying without idempotency
def charge_customer(amount):
    for attempt in range(3):
        try:
            return payment_gateway.charge(amount)
        except Timeout:
            pass  # Already charged! Retry charges again!
```

Without an idempotency key, each retry creates a new charge. The customer is charged 3x.

**Fix:** generate an idempotency key for each operation. Send it with every request. The server checks the key before processing.

### Anti-Pattern 3: Retrying Every Error Type

```python
# Bad: retry everything
@retry(max_attempts=3)
def get_user(user_id):
    return db.query(User).get(user_id)  # Retries on 404!
```

Retrying a 404 will never succeed — the user does not exist. The retries waste time and resources.

**Fix:** only retry transient errors: timeout, connection reset, 503, 429. Do not retry 400, 401, 403, 404, 409, 422 — these are client errors that retrying will not fix.

### Anti-Pattern 4: No Fallback Degradation

Failing completely when a non-critical dependency is unavailable.

```python
# Bad: hard failure on non-critical dependency
def get_dashboard(user_id):
    recommendations = recommendation_service.get(user_id)  # This fails
    # Entire dashboard is blank because recommendations are down
    return render_dashboard(recommendations, ...)
```

**Fix:** every non-critical dependency should have a fallback. If recommendations are down, show popular items. If the personalization service is slow, show generic content.

### Anti-Pattern 5: Trusting External Systems Implicitly

Assuming external APIs always return data in the expected format and within the expected latency.

```python
# Bad: no validation
def process_payment(data):
    result = payment_gateway.charge(data)
    # result assumed to have 'transaction_id' and 'status'
    return result['transaction_id']
```

If the payment gateway returns an unexpected response (schema change, error format, partial data), the code crashes with a `KeyError` or `TypeError`.

**Fix:** validate external responses. Check the structure before accessing fields. Set strict timeouts. Treat external APIs as untrusted.

### Anti-Pattern 6: Shared Thread Pools Across Dependencies

Using a single thread pool for all outbound calls. A slow or failing dependency consumes all threads, blocking calls to healthy dependencies.

```python
# Bad: shared thread pool
thread_pool = ThreadPoolExecutor(max_workers=20)

def call_service_a():
    thread_pool.submit(service_a.call)

def call_service_b():
    thread_pool.submit(service_b.call)
    # Service A hanging consumes all 20 threads
    # Service B cannot get a thread
```

**Fix:** use separate thread pools per dependency. Each pool has its own `max_workers`. A hanging Service A consumes only its own pool's threads.

### Anti-Pattern 7: Ignoring Tail Latency

Optimizing for p50 latency while ignoring p99 and p999. A service that responds in 10ms median but 5 seconds at p999 has a reliability problem — 1 in 1000 requests fails the user's patience threshold.

```python
# P50: 10ms, P95: 50ms, P99: 500ms, P999: 5000ms
# The p999 is 500x the p50 — something is very wrong
```

**Fix:** set timeouts based on p99, not p50. Monitor the ratio of p99 to p50. If p99 exceeds p50 by more than 10x, investigate the root cause (slow queries, garbage collection pauses, lock contention).

### Anti-Pattern 8: Over-Retrying on the Client Side

The client retries 3 times, the API gateway retries 3 times, and the service retries 3 times. Total amplification: 27x. Nine out of ten calls are retries.

```python
# Client retries: 3 attempts
# Gateway retries: 3 attempts
# Service retries: 3 attempts
# Total: 3 * 3 * 3 = 27 attempts for one user request
```

**Fix:** each layer must know the total retry budget. Use a header (`X-Retry-Count`) to track retries across layers. The client retries 3 times. The gateway and service do not retry — they pass the failure up to the client. Only the edge layer retries.

### Anti-Pattern 9: Synchronous Fallback That Calls the Same Dependency

The fallback for a failing database call is another database call. This provides no protection — the fallback fails with the same error.

```python
# Bad: fallback calls the same failing dependency
def get_user(user_id):
    try:
        return primary_db.query(user_id)
    except DatabaseError:
        return secondary_db.query(user_id)  # Same database, same failure
```

**Fix:** the fallback must use a different mechanism. If the primary database is failing, fall back to cache, not to another database replica. If the primary service is failing, fall back to a different service, not to the same service in a different region.

### Anti-Pattern 10: No Backpressure in Async Pipelines

An async pipeline accepts events faster than the consumer can process them. The queue grows unbounded. When the queue fills memory, the process crashes.

**Fix:** every queue must have a maximum size. When the queue is full, the producer must either block, drop the event, or apply backpressure to the upstream source. Use flow control mechanisms: Kafka's `max.in.flight.requests`, TCP's receive window, or application-level credit-based flow control.

## The Cost of Reliability

Reliability mechanisms have direct and indirect costs:

**Direct costs:**
- **Infrastructure cost.** Multi-region deployment doubles infrastructure spend. Redundant instances add 50-100% overhead. Separate circuit breaker and bulkhead pools consume memory and threads.
- **Latency cost.** Retries add 200ms-10s to the failure path. Circuit breaker checks add ~1 microsecond per call. Bulkhead acquires add 100ns-200 microseconds per call.
- **Complexity cost.** Each mechanism adds code, configuration, testing, and debugging overhead. A system with timeouts, retries, circuit breakers, bulkheads, and fallbacks is harder to understand and maintain than a system without them.

**Indirect costs:**
- **Operational cost.** Dashboards, alerts, runbooks, on-call rotations for reliability mechanisms. A circuit breaker opening requires investigation — was it a real failure or a false positive?
- **Development cost.** Implementing and testing reliability mechanisms takes developer time that could be spent on features.
- **Cognitive cost.** Developers must understand how reliability mechanisms interact. A retry policy + circuit breaker + bulkhead combination has complex behavior during edge cases.

## When NOT to Add Reliability Mechanisms

Reliability mechanisms are not free. Sometimes the cost exceeds the benefit:

**When failures are extremely rare (under 0.01%).** If a dependency has a 99.99% uptime track record, adding circuit breakers and bulkheads adds complexity that may never pay off. A simple timeout is sufficient.

**When the operation is non-critical and stateless.** A logging service that occasionally drops events is acceptable. Adding retries and fallbacks to a logging call adds latency and complexity for marginal benefit.

**When the system must be simple.** A startup with 1,000 users and a single database does not need circuit breakers. A single database with daily backups and a simple restart is sufficient. Add reliability mechanisms as the system scales.

**When the failure is non-recoverable anyway.** If the primary region fails completely and you have no multi-region setup, retries and circuit breakers do not help. You need a database replica in another region before you can add multi-region reliability.

## The 80/20 Rule of Reliability

80% of the reliability benefit comes from 20% of the mechanisms. The baseline recommendation for most systems:

1. **Timeouts on every outbound call** — single biggest reliability improvement per line of code. Start here.
2. **Exponential backoff with jitter on retries** — prevents retry storms. Add second.
3. **Circuit breaker on critical dependencies** — protects against cascading failures. Add when you have three or more services.
4. **Bulkheads for different latency profiles** — isolates fast paths from slow paths. Add when you notice latency correlation between different endpoints.
5. **Fallbacks for read operations** — keeps users working during partial outages. Add when you have cached or static alternatives.

These five mechanisms cover 90% of reliability scenarios. The remaining 10% (Byzantine fault tolerance, multi-region consistency, CRDTs) are necessary only for systems that cannot tolerate any data loss or inconsistency.
