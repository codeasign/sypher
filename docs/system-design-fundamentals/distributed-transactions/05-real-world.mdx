# Distributed Transactions in the Real World

## Case Study 1: Amazon DynamoDB Transactions

### The Problem

Amazon operates one of the largest e-commerce platforms in the world. An order on Amazon touches multiple DynamoDB tables: the customer account table, the inventory table, the order table, and the payment table. Before 2018, coordinating updates across these tables required application-level logic with manual rollback handling.

An order flow that deducted inventory, charged the customer, and created the order record had no atomic guarantee. If the application crashed after deducting inventory but before charging the customer, the inventory was lost and the customer was not charged. Amazon reported that a significant fraction of operational incidents involved data inconsistency from uncoordinated multi-table updates.

### The Solution

Amazon introduced DynamoDB Transactions in November 2018. DynamoDB Transactions use a variant of 2PC optimized for DynamoDB distributed storage. The key design decisions were:

1. **Coordinator is embedded in DynamoDB**: The transaction coordinator runs inside the DynamoDB service itself. Applications do not deploy a separate coordinator.

2. **Optimistic concurrency**: DynamoDB Transactions use optimistic locking rather than pessimistic locking. If a transaction conflicts with another transaction, it is aborted and retried. This avoids the blocking problem of traditional 2PC.

3. **Synchronous replication**: The transaction log is synchronously replicated across three availability zones within a region. A transaction is committed only after the log write is confirmed by all three zones.

4. **25-item limit per transaction**: DynamoDB Transactions support up to 25 items per transaction to limit the transaction duration and reduce lock contention.

### Performance

DynamoDB Transactions add approximately 2x latency compared to single-item writes. A single-item write takes 5-10ms. A transactional write with 3 items takes 15-25ms. The throughput is limited by the write capacity units of the tables involved.

### Lessons Learned

1. **Embedding the coordinator in the database** eliminates the operational burden of running a separate coordinator service.
2. **Optimistic concurrency** avoids the blocking problem at the cost of retries under contention.
3. **Transaction size limits** prevent long-running transactions from blocking the system.

## Case Study 2: Google Spanner

### The Problem

Google needed a globally distributed database that could provide strong consistency across data centers. Google AdWords, Google Search indexing, and Google Cloud Platform all required the ability to read and write data consistently across multiple continents.

Traditional 2PC does not work well across continents because the latency is too high (150-300ms per round trip). A 3-RTT 2PC transaction between US and Europe would take 600-900ms, which is unacceptable for interactive applications.

### The Solution

Google Spanner, announced in 2012 and made public in 2017, uses a combination of TrueTime (Google atomic clock infrastructure) and 2PC to provide strong consistency across global regions.

The key innovations are:

1. **TrueTime**: Spanner uses GPS and atomic clocks to provide a globally synchronized clock with bounded uncertainty. The TrueTime API returns a time interval [earliest, latest] that is guaranteed to contain the actual time. Typical uncertainty is 1-7ms.

2. **Commit Wait**: After a 2PC commit, Spanner waits for the TrueTime uncertainty interval to pass before making the commit visible. This ensures that all nodes see the same timestamp ordering.

3. **Paxos-based replication**: Each shard within Spanner uses Paxos consensus to replicate the transaction log. Writes are acknowledged only after a majority of replicas confirm.

4. **Read-Write transactions use 2PC**: Spanner read-write transactions use 2PC with the coordinator role assigned to the leader Paxos replica of the first group involved in the transaction.

### Performance

Spanner read-write transactions commit in 10-50ms within a single region and 100-300ms across regions. Read-only transactions are lock-free and execute in 1-5ms using TrueTime timestamps.

### Lessons Learned

1. **TrueTime eliminates the need for distributed locking**: Because Spanner has globally synchronized time, it can assign monotonically increasing timestamps to transactions without coordination. This reduces the overhead of 2PC.

2. **2PC still works across regions when combined with consensus**: The Paxos layer handles the coordination latency, and 2PC provides the atomicity. Spanner combines both protocols.

3. **External consistency**: Spanner provides the strongest consistency guarantee available in a distributed database, formalized as "external consistency" -- equivalent to linearizability.

## Case Study 3: Uber

### The Problem

Uber operates a real-time marketplace connecting riders and drivers. A trip lifecycle involves multiple services: the trip service, the payment service, the driver service, and the pricing service. Before 2017, Uber used a monolithic database for trip data. As Uber scaled globally, the database could not keep up with the volume.

Uber needed to shard its database. But sharding introduced the distributed transaction problem: a trip update might need to update the trip record, the driver earnings, and the payment record across different shards.

### The Solution

Uber designed its sharding strategy to minimize distributed transactions. The core insight was that most trip operations involve a single trip ID, and the trip ID can be used as the shard key. By placing all data for a single trip on the same shard, Uber avoided distributed transactions for the critical path.

For the remaining cases where cross-shard operations were unavoidable, Uber used:

1. **Saga pattern**: The trip lifecycle is modeled as a saga. Each step (driver assigned, rider picked up, trip completed, payment processed) is a local transaction on the trip shard. If a step fails, compensating transactions are executed.

2. **After-image queues**: For cross-shard updates that do not need immediate consistency, Uber uses after-image queues. A change is written to the local shard and a message is enqueued for async processing on the target shard.

3. **Idempotent operations**: All cross-shard operations are idempotent. If a message is processed twice, the second execution is a no-op.

### Lessons Learned

1. **Design your data model to avoid distributed transactions**: The most effective strategy is to co-locate related data on the same shard. Shard by entity ID, not by entity type.

2. **Saga works well for long-lived business processes**: The trip lifecycle spans 10-30 minutes. A saga with compensating transactions handles this naturally.

3. **Async queues reduce the need for 2PC**: Most cross-shard operations can tolerate seconds of delay. An async queue with idempotent consumers is simpler and more reliable than 2PC.

## Case Study 4: Airbnb

### The Problem

Airbnb operates a marketplace for short-term rentals. A booking involves multiple services: the listing service, the reservation service, the payment service, and the notification service. Before 2016, Airbnb used a monolithic Rails application with a single MySQL database. As the platform grew, Airbnb decomposed the monolith into microservices, each with its own database.

The booking flow became a distributed transaction: reserve the listing, charge the guest, create the reservation, credit the host, and send notifications. If any step failed, the system had to undo all previous steps.

### The Solution

Airbnb chose the Saga pattern with orchestration. The key design decisions were:

1. **Event-driven saga**: Each service publishes events when its local transaction completes. The saga orchestrator listens for events and triggers the next step or executes compensation.

2. **Reservation with expiration**: Listings are not locked during the booking flow. Instead, inventory is reserved with an expiration time. If the saga does not complete within the reservation window, the reservation expires and the inventory is released.

3. **Compensation as business logic**: Compensating transactions are not invisible rollbacks. They are business operations that the user can see. If a booking fails, the guest receives a cancellation email and the host receives a notification.

4. **Monitoring dashboard**: Airbnb built a dedicated saga monitoring dashboard that shows the status of every in-flight booking. Operations teams can see which sagas are stuck and manually intervene.

### Lessons Learned

1. **Reservation with expiration avoids locking**: By using time-bound reservations instead of locks, Airbnb avoids the blocking problem of 2PC.

2. **Compensations are visible to users**: Unlike 2PC rollbacks, which are invisible to users, saga compensations are business operations. Users see "Booking cancelled" and "Refund issued" rather than a silent failure.

3. **Monitoring is critical for saga reliability**: Without a coordinator to enforce atomicity, sagas can get stuck in partial states. A monitoring dashboard is essential for operational visibility.

## Open Source Frameworks

### Atomikos

Atomikos is a Java-based transaction manager that provides both 2PC and TCC support. It is one of the most widely used transaction managers in the Java ecosystem.

**Key Features**:
- Full JTA (Java Transaction API) support.
- Supports 2PC across JDBC connections, JMS queues, and JCA resources.
- Built-in recovery manager with automatic crash recovery.
- TCC support for microservices environments.

**Best For**: Java monoliths and microservices that need 2PC across databases and message queues.

### Narayana

Narayana is a JBoss transaction manager, now part of the WildFly ecosystem. It supports 2PC, TCC, and Sagas.

**Key Features**:
- Supports JTA, JTS (Java Transaction Service), and REST-AT.
- Long-running action (LRA) support for sagas.
- Distributed transaction recovery service.
- Integration with WildFly, Spring Boot, and Quarkus.

**Best For**: JBoss and WildFly deployments, Spring Boot applications with LRA saga support.

### Apache Seata

Apache Seata is an open-source distributed transaction framework for microservices. It supports AT (Automatic Transaction), TCC, Saga, and XA modes.

**Key Features**:
- AT mode: Automatic 2PC with undo log generation. The framework generates undo logs automatically from SQL statements.
- TCC mode: Try-Confirm-Cancel with user-defined business logic.
- Saga mode: State machine-based saga orchestration.
- XA mode: Standard XA protocol support for databases that support XA.

**Best For**: Microservices environments where automatic rollback generation is valuable.

## Cloud Services

### AWS DynamoDB Transactions

DynamoDB Transactions provide ACID guarantees across up to 25 items within a single DynamoDB table or across multiple tables in the same region. The API is `TransactWriteItems` and `TransactGetItems`.

**Pricing**: You pay for each read or write unit consumed by the transaction, plus an additional 2x write cost for the transaction log.

**Limitations**: Transactions cannot span regions. Maximum 25 items per transaction. Maximum 4 MB of data per transaction.

### GCP Cloud Spanner

Spanner provides full ACID transactions across rows, tables, and databases in a globally distributed deployment. Spanner supports both read-write transactions (using 2PC with TrueTime) and read-only transactions (using lock-free consistent reads).

**Pricing**: You pay for the processing nodes, storage, and data transfer. There is no per-transaction cost.

**Limitations**: Higher latency than single-region databases. Higher cost than DynamoDB for simple workloads.

### Azure Cosmos DB

Cosmos DB supports multi-item transactions within a single logical partition using stored procedures. For cross-partition transactions, Cosmos DB uses the transactional batch API, which provides atomicity across multiple items within the same partition key.

**Pricing**: You pay for request units (RU) consumed by the transaction. No additional transaction log cost.

**Limitations**: Transactions are limited to a single partition key. Cross-partition distributed transactions are not supported.

### Cloud Service Comparison Summary

| Cloud Service | Transaction Type | Consistency | Max Items | Max Data | Cross-Region | Pricing |
|---|---|---|---|---|---|---|
| DynamoDB Transactions | 2PC (optimistic) | Strong | 25 | 4 MB | No | 2x write cost |
| Cloud Spanner | 2PC + TrueTime | External (linearizable) | Unlimited | 1 GB | Yes (10-300ms) | Node + storage |
| Cosmos DB | Stored procedure | Strong per partition | 10,000 RU | 2 MB | No | Normal RU cost |

## Additional Case Study: Netflix

### The Problem

Netflix operates a streaming platform with millions of concurrent users. The content ingestion pipeline involves multiple services: content metadata, encoding, subtitle generation, thumbnail creation, and content distribution. Content updates must be atomic across these services.

If a content metadata update is applied but encoding fails, the content appears in the catalog but cannot be streamed. If encoding completes but thumbnails are not generated, the content appears in the catalog without a browse image.

### The Solution

Netflix uses an event-driven architecture with choreography-based sagas for content ingestion. Each step in the ingestion pipeline publishes events that trigger the next step. If a step fails, the failure event triggers compensating actions.

The key design decisions were:

1. **Event sourcing**: Each content update is recorded as an event in an event store. The current state of a content item is derived by replaying its events.

2. **Idempotent consumers**: Each content processing service is idempotent. If an event is delivered twice, the second processing is a no-op.

3. **Compensation via reversal events**: If encoding fails, a "EncodingFailed" event is published. Services that processed earlier steps listen for this event and reverse their changes. For example, the metadata service reverts the content to "draft" status.

4. **Eventual consistency with monitoring**: Netflix accepts that content may be briefly inconsistent during ingestion. A monitoring dashboard tracks the state of every content item. If a content item remains in an intermediate state for more than 30 minutes, an alert is triggered.

### Lessons Learned

1. **Event-driven sagas scale well**: Netflix ingests thousands of content items per day without a central coordinator bottleneck.

2. **Monitoring compensates for eventual consistency**: The monitoring dashboard provides visibility into ingestion pipeline health. Operations teams can detect and fix stuck sagas before they impact users.

3. **Event sourcing provides a built-in audit trail**: Every content change is recorded in the event store, providing a complete history of content ingestion.

## Additional Case Study: LinkedIn

### The Problem

LinkedIn operates a social network with 700 million members. When a member updates their profile, the update must propagate to multiple services: the member profile service, the search index, the feed service, the notification service, and the analytics service.

The profile update must be atomic: if the profile service confirms the update but the search index update fails, members searching for the updated profile cannot find it. LinkedIn cannot use 2PC for this because the transaction spans 5 services with different data models and storage technologies.

### The Solution

LinkedIn uses an event-driven architecture with at-least-once delivery and idempotent consumers. The key components are:

1. **Kafka as the transaction log**: Profile updates are written to a Kafka topic. Each service consumes from its own partition of the topic. The Kafka log serves as the durable transaction record.

2. **Local transactions per service**: Each service processes the update in its own database using local transactions. If a service fails, it resumes processing from its last committed Kafka offset.

3. **Compensation via tombstone events**: If a profile update is rolled back (for example, due to a policy violation), a tombstone event is published. Each service consumes the tombstone event and reverts the update.

4. **Reconciliation job**: A daily reconciliation job compares the profile service database with the search index and feed database. Any discrepancies are flagged and automatically corrected.

### Lessons Learned

1. **Kafka can serve as a distributed transaction log**: The Kafka log provides durability, ordering, and replay capabilities that serve the same role as a 2PC transaction log.

2. **Reconciliation is a practical alternative to distributed transactions**: Rather than preventing inconsistencies through protocol, detect and fix them through reconciliation.

3. **Event-driven architectures reduce coupling**: Each service processes profile updates independently without a shared coordinator.

## Cloud Service Best Practices

### When to Use DynamoDB Transactions

Use DynamoDB Transactions when:
- You are already using DynamoDB as your primary database.
- Your transaction involves 25 items or fewer.
- You need strong consistency within a single region.
- You can tolerate the 2x write cost.

Do not use DynamoDB Transactions when:
- Your transaction spans multiple regions.
- Your transaction involves more than 25 items.
- You need sub-millisecond commit latency.

### When to Use Cloud Spanner

Use Cloud Spanner when:
- You need strong consistency across multiple regions.
- Your transaction involves a large number of rows or tables.
- You need external consistency (linearizability).
- You are willing to accept higher latency for consistency.

Do not use Cloud Spanner when:
- Your workload is read-heavy and eventual consistency is acceptable.
- Your data volume is small (under 10 GB).
- You are optimizing for cost rather than consistency.

### When to Use Event-Driven Saga

Use event-driven sagas when:
- Your transaction spans more than 5 services.
- Your transaction takes longer than 30 seconds.
- You can tolerate eventual consistency.
- Your services are independently deployed and owned by different teams.

Do not use event-driven sagas when:
- Your transaction involves money or inventory where inconsistency is unacceptable.
- You cannot implement idempotent consumers.
- Your monitoring infrastructure is immature.
