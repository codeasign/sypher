## Foundation Challenges

### Challenge 1: Explain the 2PC Protocol to a Non-Technical Stakeholder

**Problem**: Your product manager asks you to explain why the checkout flow sometimes fails with the message "Order could not be processed. Please try again." They want to understand what is happening behind the scenes.

**Task**: Write a 150-word explanation that a non-technical person can understand. Use an analogy. Do not use terms like "transaction log" or "participant."

**Model Answer**:

"When you place an order, three separate systems must agree to work together: our inventory system must set aside the item, our payment system must charge your card, and our shipping system must prepare the package. Think of it like three colleagues who each need to sign a document before the deal is final. The order system sends the document to all three at once. Each colleague reviews their part and says 'I am ready to sign.' Only after all three say they are ready does the system say 'Everyone sign now.' If one colleague cannot complete their part, the system tells the other two to cancel what they started, and you see the failure message. You can try again, and the whole process restarts. This ensures your card is never charged without an item being set aside, and an item is never set aside without a shipping label being prepared."

### Challenge 2: Identify the Problem

**Problem**: Your team has been running a checkout service using a naive distributed transaction implementation. The service code looks like this:

```java
public void checkout(String customerId, String sku, int quantity) {
    try {
        inventoryClient.decrement(sku, quantity);
        paymentClient.charge(customerId, getPrice(sku, quantity));
        shippingClient.createLabel(customerId, sku, quantity);
    } catch (Exception e) {
        if (e instanceof ShippingException) {
            inventoryClient.increment(sku, quantity);
            paymentClient.refund(customerId, getPrice(sku, quantity));
        } else if (e instanceof PaymentException) {
            inventoryClient.increment(sku, quantity);
        }
    }
}
```

**Task**: Identify all the problems with this approach. List at least four distinct issues.

**Model Answer**:

1. **No atomicity guarantee**: If the application crashes after `inventoryClient.decrement` but before `paymentClient.charge`, the inventory is decremented but the customer is not charged. There is no compensation mechanism for application crashes.

2. **Compensation failures**: If `inventoryClient.decrement` succeeds and `paymentClient.charge` fails, the catch block calls `inventoryClient.increment`. But if the increment call also fails (network error, database error), the inventory remains decremented with no further recovery attempt.

3. **No idempotency**: If `paymentClient.charge` succeeds but the HTTP response is lost due to a network timeout, the exception handler calls `paymentClient.refund` even though the charge was actually successful. The customer is both charged and refunded, resulting in a free item.

4. **No retry mechanism**: There is no retry logic for transient failures. If the shipping service is temporarily unavailable, the entire checkout fails. The customer must try again from the beginning.

5. **No monitoring**: There is no logging of partial states. If the compensation calls also fail, the system enters an inconsistent state with no record of what happened, making manual reconciliation impossible.

### Challenge 3: Design a Recovery Strategy

**Problem**: A coordinator processes a 2PC transaction across three participants. The coordinator receives YES votes from all three participants. The coordinator crashes after writing the COMMIT_DECISION record to its log but before sending any COMMIT messages. The coordinator restarts 30 seconds later.

**Task**: Describe exactly what happens during recovery, step by step.

**Model Answer**:

1. **Log Scan**: The recovery manager reads the transaction log from the last checkpoint forward. It finds the COMMIT_DECISION record for the unresolved transaction.

2. **Decision Determination**: Because the COMMIT_DECISION record exists, the recovery manager determines that the coordinator decided to commit. This decision is final and must be honored.

3. **Resume Protocol**: The recovery manager resumes the commit phase. It begins sending COMMIT messages to all three participants.

4. **Phase 2 (Commit) Execution**: The coordinator sends COMMIT to participant 1, participant 2, and participant 3. Each participant applies the committed changes, releases locks, and sends an ACK.

5. **Completion**: The coordinator receives ACKs from all participants. It writes the COMPLETED record to the transaction log.

**Fallout During 30-second Downtime**: During the 30 seconds the coordinator was down, all three participants were in the PREPARED state, holding locks on the resources involved in the transaction. Other transactions that needed those resources were blocked. After recovery, the COMMIT is sent and the locks are released.

## Design Challenges

### Challenge 4: Design a Distributed Transaction System for a Global Hotel Booking Platform

**Problem**: A hotel booking platform operates in 50 countries. A customer books a hotel stay, which involves:
- Reserving the room in the hotel inventory system.
- Charging the customer payment.
- Creating a booking record.
- Crediting the travel agent commission.
- Triggering a notification to the hotel.

The system processes 5,000 bookings per minute at peak. The booking must complete within 10 seconds. If any step fails, all previous steps must be undone. The system must handle the case where the payment service processes in a different currency than the hotel inventory service.

**Model Answer**:

**Protocol Selection**: Use TCC (Try-Confirm-Cancel).

**Rationale**: 

1. 2PC would hold locks across all five services for the booking duration. With 5,000 bookings per minute (83 per second), lock contention would be extreme.

2. A Saga would leave a window where the booking is partially completed. If the payment succeeds but the notification fails, the hotel is paid but never notified.

3. TCC reserves resources during Try. If any Try fails, all reservations are released. The Confirm phase applies all changes atomically. Resources are held only during the Try phase, which completes in under 2 seconds.

**Architecture**:

1. **Regional Coordinators**: Deploy a TCC coordinator cluster in each region. Bookings are processed within a single region. Cross-region bookings use a global event bus.

2. **Try Phase (under 2 seconds)**:
   - Hotel inventory: Reserve the room for 60 seconds.
   - Payment: Authorize the charge (no capture).
   - Booking record: Create a pending booking record.
   - Travel agent: Reserve the commission amount.
   - Notification: Validate the hotel contact details.

3. **Confirm Phase (under 1 second)**:
   - Hotel inventory: Block the room permanently.
   - Payment: Capture the authorized charge.
   - Booking record: Mark as CONFIRMED.
   - Travel agent: Record commission.
   - Notification: Send booking confirmation.

4. **Cancel Phase**: Release all reservations and void the payment authorization.

**Failure Handling**:
- If any Try fails: Coordinator calls Cancel on all successful Try participants.
- If Confirm on payment succeeds but Confirm on inventory fails: Coordinator retries Confirm on inventory every second for 30 seconds, then moves to a dead letter queue.

**Currency Conversion**: The payment service handles currency conversion during the Authorize phase using current exchange rates. The exchange rate is recorded in the transaction log. If the rate changes between Try and Confirm, the Confirm uses the rate from Try.

### Challenge 5: Design a Reconciliation System for a Saga-Based Order Platform

**Problem**: An e-commerce platform uses Sagas for order processing. The platform processes 100,000 orders per day. You need to design a reconciliation system that detects and fixes inconsistencies in saga state.

**Task**: Design the reconciliation system. It must:
- Detect orders that are stuck in inconsistent states.
- Automatically fix the most common inconsistencies.
- Escalate unresolved issues to the operations team.
- Complete within 15 minutes during off-peak hours.

**Model Answer**:

**System Components**:

1. **Saga State Store**: Each saga step records its result in a saga state table. The table contains: `saga_id`, `step_name`, `status` (PENDING, SUCCEEDED, FAILED, COMPENSATED), `timestamp`, and `details`.

2. **Reconciliation Job** (runs every 5 minutes):
   a. Query the saga state store for sagas that are in an incomplete state for more than 10 minutes.
   b. For each incomplete saga, check the current state of each participant by calling the participant status API.
   c. Compare the expected state (from the saga definition) with the actual state (from the participant APIs).
   d. If there is a mismatch, fix it automatically if the fix is known to be safe.
   e. If the fix is not safe, create an escalation ticket.

**Auto-Fix Rules**:

| Scenario | Expected | Actual | Auto-Fix |
|---|---|---|---|
| Saga step SUCCEEDED but compensation never ran | Step should be COMPENSATED | Step is SUCCEEDED | Run compensation |
| Saga step reported as PENDING but participant says SUCCEEDED | Step should be SUCCEEDED | Step is PENDING | Mark as SUCCEEDED, continue saga |
| Saga step reported as FAILED but participant says SUCCEEDED | Step should be COMPENSATED | Step is FAILED with no compensation | Run compensation for this step |
| Saga with no updates for 24 hours | Saga should be COMPLETED or FAILED | Saga is PENDING | Escalate to operations |

**Escalation**: Sagas that cannot be auto-fixed are written to an escalation queue. The operations team receives a notification with the saga ID, the expected state, the actual state, and the participant API responses. The team uses a manual resolution tool that provides a safe "force complete" or "force rollback" option.

**Performance**: The reconciliation job scans 10,000 saga records per minute. At 100,000 orders per day, the average scan completes in 10 minutes. Sagas older than 30 days are archived to reduce scan time.

## Expert Challenges

### Challenge 6: Design a Protocol That Combines the Strengths of 2PC and Saga

**Problem**: 2PC provides strong consistency but blocks during failures. Saga provides high availability but only eventual consistency. Design a hybrid protocol that provides strong consistency for a configurable timeout period, and degrades to eventual consistency if the timeout expires.

**Model Answer**:

**Protocol Name**: TCC with Timeout Escalation (TCC-TE)

**Design**:

The protocol has three phases: Try, Confirm, and Timeout Escalation.

**Phase 1: Try (0-500ms)**
Each participant reserves resources and returns a reservation token. The reservation has a TTL of 60 seconds. Participants write a prepare record to their local log. This is identical to TCC Try.

**Phase 2: Confirm (500ms-5s)**
The coordinator sends Confirm to all participants. Participants make the reservations permanent. If the coordinator receives all ACKs, the transaction is committed with strong consistency.

**Phase 3: Timeout Escalation (5-60s)**
If the coordinator does not receive all Confirm ACKs within 5 seconds, it enters timeout escalation:
1. The coordinator broadcasts a PROBE message to all participants asking for their current state.
2. Each participant responds with one of: COMMITTED, PREPARED, or UNKNOWN.
3. If all participants respond with COMMITTED, the transaction is resolved.
4. If any participant responds with PREPARED, the coordinator retries Confirm.
5. If a participant is UNKNOWN (crashed), the coordinator waits for it to recover.

**Fallback to Saga (60s+)**
If the transaction is not resolved within 60 seconds (the reservation TTL):
1. The coordinator declares the transaction as "indeterminate" and writes this to the log.
2. Participants auto-release reservations when the TTL expires.
3. A reconciliation job runs to detect and fix any inconsistencies.
4. The system degrades to eventual consistency.

**Guarantees**:
- Within 5 seconds: Strong consistency (equivalent to 2PC).
- Within 60 seconds: Medium consistency (equivalent to TCC with known state).
- After 60 seconds: Eventual consistency (equivalent to Saga with reconciliation).

This protocol provides the best of both worlds: strong consistency for the common case and graceful degradation under failure.

### Challenge 7: Design a Global Distributed Transaction System for a Digital Wallet

**Problem**: A digital wallet platform operates in 20 countries. Users can transfer money between wallets in different countries. Each country has its own database for regulatory compliance. You need to design a transaction system that guarantees:

- Exactly-once transfer semantics.
- No money creation or destruction across all wallets.
- 99.99% availability.
- Transfer completion within 5 seconds for same-country transfers and 30 seconds for cross-country transfers.
- Audit trail for every transfer.

**Model Answer**:

**Architecture**:

1. **Regional Transaction Coordinator (RTC)**: Each country has an RTC that manages transactions involving wallets in that country. Cross-country transactions involve two RTCs.

2. **Global Transaction Coordinator (GTC)**: A single coordinator that manages cross-country transactions using a replicated transaction log stored in a global consensus system (Spanner or CockroachDB).

3. **Wallet Service**: Each country has its own wallet database. The wallet service exposes idempotent debit and credit operations.

**Same-Country Transfer (under 5 seconds)**:

1. Client sends transfer request to RTC.
2. RTC uses 2PC: Debit sender wallet, credit receiver wallet.
3. If either fails, both are rolled back.
4. Transaction log records the transfer with unique transfer ID.

**Cross-Country Transfer (under 30 seconds)**:

1. Client sends transfer request to either RTC.
2. RTC forwards to GTC.
3. GTC uses TCC with each country RTC:
   - Try: RTC reserves the debit amount. RTC validates that the receiver wallet exists and reserves the credit.
   - Confirm: RTC applies the debit. RTC applies the credit.
   - Cancel: RTC releases the reservation.
4. Each RTC uses 2PC within its country to update its wallet database.
5. GTC transaction log records every state transition for audit.

**Exactly-Once Guarantee**:

1. Each transfer has a unique, idempotent transfer ID.
2. Each debit and credit operation checks the transfer ID. If the operation was already performed, it returns the existing result.
3. The GTC retries Confirm until all participants acknowledge.
4. A reconciliation job runs every minute and checks for stuck transfers.

**No Money Creation or Destruction**:

1. The total of all debits must equal the total of all credits for each transfer.
2. The system enforces: `SUM(debits) = SUM(credits)` at the GTC level.
3. A daily reconciliation job verifies that the total wallet balance across all countries equals the expected total.

**99.99% Availability**:

1. Each RTC is deployed across three availability zones.
2. The GTC is deployed across three global regions with Paxos consensus.
3. Wire transfers against a single region RTC are unaffected if another region is down.
4. Cross-country transfers may experience a few seconds of delay during regional failures.

**Audit Trail**:

1. Every state transition is written to the GTC transaction log.
2. The log is immutable and append-only.
3. Each transfer record includes: transfer ID, sender wallet, receiver wallet, amount, currency, exchange rate, timestamp, and participant acknowledgments.
4. A separate audit service indexes the log for query by compliance teams.

**Performance at 1,000 transfers per second globally**:

| Phase | Latency | Throughput |
|---|---|---|
| Same-country | 3-5 seconds | 800 per second |
| Cross-country | 10-25 seconds | 200 per second |
| Reconciliation | 1-minute cycle | 60,000 checked |

## Additional Foundation Challenge

### Challenge 8: Design a Compensation Strategy for a Multi-Step Booking Flow

**Problem**: A vacation booking platform has a 5-step booking flow:
1. Reserve flight (Airline Service).
2. Reserve hotel (Hotel Service).
3. Reserve car rental (Car Rental Service).
4. Charge customer (Payment Service).
5. Send confirmation (Notification Service).

The reserving steps are time-limited: the flight reservation expires in 30 minutes, the hotel reservation expires in 24 hours, and the car rental reservation expires in 1 hour. The payment step is the final commitment.

**Task**: Design the compensation strategy for this flow. Specify the exact compensation for each step, the order of compensation execution, and what happens if a compensation itself fails.

**Model Answer**:

**Compensation Strategy**:

| Step | Forward Operation | Compensation | Expiry |
|---|---|---|---|
| 1. Reserve flight | Book flight with hold | Cancel flight reservation | 30 min |
| 2. Reserve hotel | Book hotel with hold | Cancel hotel reservation | 24 hours |
| 3. Reserve car | Book car with hold | Cancel car reservation | 1 hour |
| 4. Charge customer | Charge credit card | Refund credit card | N/A |
| 5. Send confirmation | Send email | Send cancellation email | N/A |

**Compensation Order**: Execute compensations in reverse order of the forward steps. If step 4 (charge customer) fails after step 3 succeeded:

1. Cancel car reservation (step 3 compensation).
2. Cancel hotel reservation (step 2 compensation).
3. Cancel flight reservation (step 1 compensation).

**Compensation Failure Handling**:

Case A: Flight reservation cancellation fails. The flight hold will expire in 30 minutes automatically. The system schedules a retry of the cancellation. If the retry also fails, the flight reservation is allowed to expire naturally.

Case B: Hotel reservation cancellation fails. The hotel hold expires in 24 hours. The system retries the cancellation every 5 minutes for 30 minutes. If all retries fail, an escalation ticket is created.

Case C: Payment refund fails. This is the most critical failure. The system retries the refund every minute for 10 minutes. If all retries fail, the payment service is paged and the incident is escalated to the financial operations team.

**Time-bound Expiry Handling**: The compensation strategy must account for reservation expiries:

- If step 1 (flight) fails after 35 minutes, the flight reservation has already expired. The compensation is a no-op.
- If step 3 (car) fails after 65 minutes, the car reservation has already expired. The compensation is a no-op.
- The system checks the reservation expiry time before executing compensation. If the reservation has already expired, the compensation is skipped.

## Additional Design Challenge

### Challenge 9: Design a Distributed Transaction System for a Stock Trading Platform

**Problem**: A stock trading platform executes 10,000 trades per second. Each trade involves:
1. Validate the order (Order Service).
2. Check account balance (Account Service).
3. Check stock availability (Market Service).
4. Execute the trade (Execution Service).
5. Update account balance (Account Service).
6. Update stock holdings (Portfolio Service).
7. Send trade confirmation (Notification Service).

The system must guarantee that:
- No trade executes without sufficient funds.
- No trade executes without sufficient stock available.
- Account balances are updated atomically with trade execution.
- The system can handle a single trade failing without affecting other trades.

**Model Answer**:

**Architecture**:

Use a two-tier approach: Execute trades within a single shard where possible, and use TCC for cross-shard trades.

**Tier 1: Single-Shard Trading**

Shard accounts by account ID. All accounts on the same shard use local database transactions. The trade execution flow on a single shard:

1. Begin local transaction.
2. Validate order.
3. Check account balance.
4. Execute trade (debit account, credit stock holdings).
5. Record trade in trade history.
6. Commit local transaction.
7. Send event for trade confirmation (async).

This handles 90% of trades (those where both buyer and seller are on the same shard).

**Tier 2: Cross-Shard TCC Trading**

For trades where buyer and seller are on different shards:

1. **Try Phase**:
   - Reserve funds on shard A (buyer account).
   - Reserve stock on shard B (seller account).
   - Both reservations expire in 5 seconds.

2. **Confirm Phase**:
   - Deduct funds on shard A.
   - Add stock holdings to shard A.
   - Deduct stock holdings on shard B.
   - Add funds to shard B.
   - Mark trade as COMPLETED.

3. **Cancel Phase**:
   - Release fund reservation on shard A.
   - Release stock reservation on shard B.
   - Mark trade as CANCELLED.

**Performance at 10,000 TPS**:

| Tier | Trades/sec | Latency | Failure Rate |
|---|---|---|---|
| Single-shard | 9,000 | 2-5ms | 0.1% |
| Cross-shard | 1,000 | 10-25ms | 0.5% |
| Overall | 10,000 | 2-25ms | 0.14% |

**Failure Handling**:

- Try phase timeout: Cancel the trade. The reservation expires automatically in 5 seconds.
- Confirm phase failure: Retry the confirm for up to 10 seconds. If all retries fail, the trade is marked as "Indeterminate" and a reconciliation job resolves it.
- Reconciliation job: Runs every 10 seconds, checks for trades in "Indeterminate" state, queries both shards, and resolves the state.

## Additional Expert Challenge

### Challenge 10: Design a Distributed Transaction Protocol for a Multi-Cloud Deployment

**Problem**: A multinational corporation operates across AWS, GCP, and Azure. Each cloud provider hosts a set of services that must participate in distributed transactions. The network latency between clouds is 100-200ms. The system must provide strong consistency for financial transactions across all three clouds.

**Design constraints**:
- No single cloud can be the coordinator (to avoid vendor lock-in).
- The system must survive the failure of one entire cloud provider.
- The maximum acceptable transaction latency is 5 seconds.
- The system processes 500 transactions per second.

**Model Answer**:

**Architecture**: Federated Transaction Coordinator (FTC)

**Components**:

1. **Regional Coordinators**: Each cloud provider runs a coordinator instance. Each coordinator manages participants within its own cloud.

2. **Global Coordinator**: A consensus group (using Raft) is deployed across all three clouds. The global coordinator is not a single process -- it is a distributed consensus group that elects a leader per term.

3. **Cross-Cloud Log**: A replicated log is stored in each cloud (DynamoDB on AWS, Spanner on GCP, Cosmos DB on Azure). Log writes require consensus from at least two clouds.

**Protocol Modified 3PC (M3PC)**:

Phase 1 -- Intra-Cloud Prepare (0-500ms):
1. Each regional coordinator sends PREPARE to participants within its cloud.
2. Participants vote YES or NO.
3. Regional coordinator aggregates votes.

Phase 2 -- Cross-Cloud Prepare (500-1500ms):
1. Global coordinator receives aggregated votes from all regional coordinators.
2. Global coordinator writes the prepare status to the cross-cloud log.
3. If all regions voted YES, proceed. Otherwise, abort.

Phase 3 -- Cross-Cloud Pre-Commit (1500-2500ms):
1. Global coordinator sends PRE-COMMIT to all regional coordinators.
2. Regional coordinators propagate PRE-COMMIT to participants.
3. Participants acknowledge PRE-COMMIT.

Phase 4 -- Cross-Cloud Commit (2500-3500ms):
1. Global coordinator sends COMMIT to all regional coordinators.
2. Regional coordinators propagate COMMIT to participants.
3. Participants acknowledge COMMIT.

**Surviving Cloud Failure**:

- If AWS fails: The global coordinator continues with GCP and Azure. The consensus quorum requires 2 of 3 clouds, so GCP and Azure can continue.
- Transactions that involve AWS participants are blocked until AWS recovers.
- New transactions that do not involve AWS can proceed normally.

**Performance**:

| Phase | Latency | Description |
|---|---|---|
| Intra-cloud prepare | 200ms | Prepare participants within each cloud |
| Cross-cloud prepare | 600ms | Aggregate votes across clouds (2 RTTs at 150ms) |
| Cross-cloud pre-commit | 600ms | Send pre-commit across clouds |
| Cross-cloud commit | 600ms | Send commit across clouds |
| Total | 2000ms | Well within the 5-second limit |

**Throughput**: 500 TPS with three participants per transaction. The bottleneck is the cross-cloud log write, which handles approximately 2,000 writes per second with group commit.
