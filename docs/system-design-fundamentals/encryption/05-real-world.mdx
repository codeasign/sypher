# Encryption in Production

Theory is necessary, but production encryption is where design decisions meet real-world constraints — latency budgets, key rotation, compliance audits, cross-region replication, and hardware security modules. This page examines how five major systems deploy encryption at scale, surveys the open-source and cloud tooling landscape, and distills the patterns that separate production-grade designs from textbook ones.

---

## 1. WhatsApp — End-to-End with the Signal Protocol

WhatsApp encrypts every message, call, and media transfer end-to-end using the Signal Protocol. No server — including WhatsApp itself — can read plaintext content.

**Key design decisions:**

- **Per-conversation keys.** Each message is encrypted with a unique AES-256 key derived from a per-session shared secret (established via X3DH — Extended Triple Diffie-Hellman).
- **Double Ratchet.** Forward secrecy and future secrecy are achieved through a ratcheting mechanism that derives new keys after every message. If a session key is compromised, past and future messages remain safe.
- **Pre-keys.** Clients upload one-time pre-key bundles to the server when online. When Alice sends Bob a first message, the protocol consumes a one-time pre-key from Bob's bundle to establish the initial shared secret.
- **Authentication via public-key fingerprints.** Users compare a 60-digit numeric fingerprint or a QR code to verify they are talking to the real recipient (out-of-band verification).

- Pre-key bundles let the server facilitate first-contact messages without ever seeing the key material — the server is a dumb relay.
- The Double Ratchet means you never encrypt two messages with the same key. This limits the blast radius of any single key compromise.
- Out-of-band fingerprint verification is the weakest link in practice. Most users never verify, making the protocol rely on the server honestly delivering the correct public keys during the initial bundle fetch.

## 2. Google — AES-GCM with the Tink Library

Google encrypts data at rest and in transit across virtually every service — BigQuery, Spanner, Gmail, Drive, YouTube — using a uniform cryptographic layer built on AES-GCM and managed through the Tink library.

**Key design decisions:**

- **AES-256-GCM as the single primitive.** Google standardizes on one authenticated encryption mode across all services. This reduces the audit surface, eliminates configuration errors from choosing between modes, and lets teams build shared infrastructure (key management, monitoring, rotation) around a single cipher.
- **Tink library.** Tink is Google's open-source cryptography library designed to make it hard to misuse. It exposes high-level APIs (`Aead`, `DeterministicAead`, `HybridEncrypt`, `DigitalSignature`) that do not let developers choose raw key sizes, IVs, or padding modes. The library enforces minimum key lengths, validates ciphertext integrity automatically, and rotates keys based on a key-management configuration.
- **Envelope encryption with KMS.** Google uses a tiered key hierarchy similar to AWS KMS (see case study 3). Data is encrypted with DEKs (data encryption keys), and DEKs are wrapped by KEKs (key encryption keys) stored in Cloud KMS or a hardware-backed key store.
- **Key rotation via key versioning.** Tink supports key sets with multiple versions. A new key version is added on rotation; old keys are retired for decryption only. The library handles versioning transparently — applications never change their code.

**Production takeaways:**

- Standardising on one cipher and one library eliminates the most common cause of encryption bugs: misconfiguration and developer choice. Tink's `Aead` interface is a single function call — `encrypt(plaintext, aad)` — with no parameters for key size, mode, or padding.
- Envelope encryption with a two-tier key hierarchy is the industry standard for a reason: the KEK lives in a small, tightly controlled HSM-backed store, while DEKs are generated and used at massive scale.
- Key versioning in Tink decouples key rotation from application deployment. A new key version can be added to the key set without any code change or restart.

## 3. AWS — KMS Envelope Encryption

AWS Key Management Service (KMS) is the backbone of encryption across the AWS ecosystem. Every AWS service that encrypts data — S3, EBS, RDS, DynamoDB, Lambda environment variables — uses KMS at some level.

**Key design decisions:**

- **Envelope encryption.** Data is encrypted with a customer data key (DEK) under the application's control. The DEK itself is encrypted by a KMS key (KEK) that never leaves the KMS service. This means the KEK is never exposed to the application process, limiting the blast radius of an application compromise.
- **Regional isolation.** KMS keys are scoped to a single AWS Region. Cross-region access requires explicit key policy grants. This aligns with compliance frameworks (GDPR, SOC, PCI) that mandate data residency.
- **Automatic key rotation.** AWS can rotate KMS keys annually (or on a custom schedule). Old key material is retained for decryption of existing data; new data uses the new key material.
- **FIPS 140-2 / 140-3 validated HSM.** KMS uses dedicated hardware security modules for key generation and cryptographic operations. Keys are generated and used inside the HSM boundary — the software layer never sees the plaintext KEK.
- **Integration with CloudTrail.** Every KMS operation — `GenerateDataKey`, `Encrypt`, `Decrypt`, `CreateKey`, `RotateKey` — is logged to CloudTrail, providing an audit trail of all key usage.

- Envelope encryption solves the scaling problem: you never send the large data payload to the KMS — only the small DEK. The data is encrypted locally with the DEK, which is fast and parallelizable.
- Storing the wrapped DEK alongside the ciphertext (the "encrypted data envelope") makes the data self-describing. Any system that reads the ciphertext also has the wrapped DEK and can decrypt it — no separate metadata lookup needed.
- KMS rate limits (`GenerateDataKey` and `Decrypt` calls per second per key) are a real constraint. For high-throughput systems, cache the plaintext DEK for the lifetime of the request and re-use it across records. Never call `Decrypt` per row in a database query.

## 4. Apple — iCloud End-to-End Encryption

Apple provides end-to-end encryption for iCloud data — including photos, device backups, messages, health data, and passwords — under a model it calls Advanced Data Protection (ADP). Apple's approach is notable because it operates at a scale of over a billion devices and must handle the complexity of cross-device sync, key recovery, and government compliance requests.

**Key design decisions:**

- **Per-service key domains.** Each iCloud service type (CloudKit, iMessage, iCloud Backup, Health, Keychain) has its own key hierarchy. A compromise of one service's keys does not affect others.
- **Key escrow via iCloud Keychain (with a twist).** Apple uses a key escrow system where the user's encryption keys are stored securely on Apple's servers — but Apple cannot read them. The keys are wrapped with a passcode-derived key (using PBKDF2 and a hardware-backed Secure Enclave) that only the user knows. The server stores the wrapped key, not the passcode.
- **Account Recovery Contact.** If a user loses their device, a trusted contact can help recover access. The recovery key is split using Shamir's Secret Sharing (SSS) — the user's device holds one share, the recovery contact holds another, and Apple holds a third. No single party can reconstruct the key.
- **iMessage with PQ3.** Apple's iMessage protocol recently added PQ3 — a post-quantum cryptographic extension to the Signal Protocol. This adds a Kyber-1024 encapsulation layer alongside the existing elliptic-curve Diffie-Hellman, protecting against future quantum decryption of recorded traffic.

**Production takeaways:**

- **Per-service key isolation** is a critical pattern: if a vulnerability in the CloudKit encryption code is discovered, iMessage and Health data remain encrypted under different keys. Never use a single key for all data.
- **Hardware-backed key storage** (Secure Enclave, TEE) is the gold standard for protecting keys at rest on the client. The CPU never has direct access to the plaintext key material.
- **Shamir's Secret Sharing for recovery** strikes a balance between security and usability. The user can recover their data without a central authority, but no single party (including Apple) can reconstruct the key.
- **Post-quantum readiness** is a production concern today, not a future one. Harvest-now-decrypt-later attacks mean that data encrypted today with classical cryptography alone may be decryptable in the future. Apple's PQ3 hybrid approach is the pattern to follow.

## 5. Open Source Libraries

Production encryption is rarely written from scratch. Almost every team builds on top of one or more of these battle-tested libraries.

| Library | Maintainer | Primary Use Case | Languages | Cipher Suite |
|---------|-----------|-----------------|-----------|-------------|
| libsodium | Community (Frank Denis) | General-purpose crypto | C, bindings for 30+ languages | XChaCha20-Poly1305, X25519, Argon2 |
| OpenSSL | OpenSSL Project | TLS, certificate handling, general crypto | C (via CLI, libssl, libcrypto) | AES-GCM, ChaCha20-Poly1305, ECDH, RSA |
| Tink | Google | Application-layer AEAD, key management | Java, C++, Go, Python, Obj-C | AES-GCM, XChaCha20-Poly1305, HPKE |
| BoringSSL | Google | Fork of OpenSSL for Chrome/Android | C (libssl, libcrypto) | Subset of OpenSSL, modern ciphers only |

### 5.1 libsodium

libsodium is the most widely recommended library for application-layer cryptography. It is a fork of NaCl (Networking and Cryptography library) by Daniel J. Bernstein, designed to be hard to misuse.

**Key features:**

- **High-level API.** `crypto_secretbox()` and `crypto_box()` handle key generation, nonce management, and authentication transparently. The developer does not choose between AES, ChaCha20, GCM, or Poly1305 — the library picks the best available primitive.
- **XChaCha20-Poly1305 as the default.** XChaCha20 extends the ChaCha20 stream cipher with a 192-bit nonce, enabling random nonces without collision risk. This eliminates the "nonce reuse" vulnerability that plagues AES-GCM deployments.
- **Argon2 for password hashing.** libsodium's `crypto_pwhash()` uses Argon2id, the winner of the Password Hashing Competition, with configurable memory cost (up to 2 GB) and time cost.
- **Constant-time operations.** All critical code paths are constant-time, preventing timing side-channel attacks.
- **Key exchange.** `crypto_kx()` provides a simple, authenticated key exchange based on X25519.

### 5.2 OpenSSL

OpenSSL is the de facto standard for TLS in the open-source ecosystem. It is used by Nginx, Apache, curl, PostgreSQL, Python, Ruby, and most Linux distributions. It is also the most audited open-source crypto library — and the most common source of encryption bugs when misused.

**Key features:**

- **TLS 1.3 support.** OpenSSL 1.1.1+ implements TLS 1.3 with 0-RTT, 1-RTT, and the full cipher suite. TLS 1.3 removes weak ciphers (RC4, 3DES, CBC-mode static RSA) and mandates forward secrecy.
- **Engine and provider architecture.** OpenSSL supports pluggable cryptographic engines. Hardware-backed engines (Intel QAT, IBM Crypto Express, PKCS#11) offload crypto operations to dedicated hardware.
- **Extensive but dangerous API.** The OpenSSL API exposes hundreds of functions with many valid-but-unsafe parameter combinations. The `EVP_*` "high-level" interface (EVP_EncryptInit_ex, EVP_EncryptUpdate, EVP_EncryptFinal_ex) is the recommended approach, but it is still easy to call incorrectly — omitting `EVP_CIPHER_CTX_set_padding` or using `EVP_aes_256_ecb()` instead of `EVP_aes_256_gcm()`.

### 5.3 Tink

Tink, covered in the Google case study above, is Google's open-source cryptography library. It is designed to be the safe default for any team building a new system.

**Key features:**

- **Misuse resistance.** Tink's API does not expose raw key material, nonces, or IVs. The `Aead` interface has exactly two methods: `encrypt(plaintext, associatedData)` and `decrypt(ciphertext, associatedData)`. Nonces are generated internally and stored in the ciphertext.
- **Key management integration.** Tink reads keysets from JSON or binary format and supports key rotation. The keyset defines which key is the primary (used for encryption) and which keys are secondary (used for decryption only).
- **Multi-language.** Tink provides the same API surface in Java, C++, Go, Python, and Objective-C. A team can use the same key management patterns across a polyglot microservice architecture.
- **Deterministic AEAD.** For applications that need deterministic encryption (e.g., database indexes), Tink provides `DeterministicAead` using AES-SIV, which preserves the authenticated encryption property while being deterministic.

### 5.4 BoringSSL

BoringSSL is Google's fork of OpenSSL, created to strip out the parts of OpenSSL that Chrome and Android do not need. It is maintained by Google and used by Chrome, Android, and many Google-internal services.

**Key features:**

- **API compatibility with OpenSSL.** BoringSSL maintains API compatibility with a subset of OpenSSL's `libssl` and `libcrypto` APIs. Most code that compiles against OpenSSL compiles against BoringSSL without changes.
- **Removed legacy cruft.** BoringSSL removes support for SSL 3.0, DTLS 1.0, RC4, 3DES, and dozens of obsolete cipher suites. This reduces the attack surface and makes the library significantly smaller.
- **Faster review cycle.** Because BoringSSL does not promise to support every platform and every cipher, Google can review and merge changes faster than the OpenSSL project.
- **CECPQ2 and Kyber.** BoringSSL was the first major TLS library to ship post-quantum key agreement in production (CECPQ2 in Chrome, later Kyber).

## 6. Cloud Key Management Services

Every major cloud provider offers a managed key management service. These services handle key generation, storage, rotation, audit logging, and HSM backing — all the operational burden that teams would otherwise need to build themselves.

### 6.1 AWS KMS

AWS Key Management Service is the most widely deployed cloud KMS. It is integrated with over 100 AWS services and supports both symmetric and asymmetric keys.

**Key features:**

- **Customer managed keys (CMK).** You create and manage the key material, including rotation schedule, key policy, and alias.
- **AWS managed keys.** AWS creates and manages keys on your behalf for integrated services (S3, EBS, RDS, etc.). These keys are automatically rotated yearly.
- **Custom key store (CloudHSM).** For compliance requirements that mandate your own HSM, AWS KMS can use a CloudHSM cluster as the key store instead of the shared KMS HSM.
- **Key policies.** Resource-based policies that control who can use a key and under what conditions. Combine with IAM policies for fine-grained access control.
- **Multi-Region keys.** Replicate key material across regions for disaster recovery, with the same key ID and key material in each region.
- **Automatic key rotation.** Once per year by default, or on a custom schedule. Old key material is retained for decryption of existing data.

**Limitations to know:**

- KMS API rate limits (typically 5,500 to 55,000 requests per second per Region, depending on the key type). Exceeding this throttles requests.
- Key material is regional by default. Cross-region access requires a multi-Region key or a cross-Region key import.
- Costs: $1/month per key + $0.03 per 10,000 `GenerateDataKey` or `Decrypt` calls. At scale, this adds up.

### 6.2 GCP Cloud KMS

Google Cloud's KMS shares many features with AWS KMS but has some distinctive capabilities.

**Key features:**

- **Global key rings and regional keys.** Key rings are global resources, but key material is stored in a single region. You can create keys in any region and reference them from the global key ring.
- **Key import.** You can import your own key material (BYOK) into Cloud KMS. The key material must be wrapped using a Google-provided public key.
- **HSM key tier.** Cloud KMS offers a software-backed tier (default) and an HSM-backed tier (Cloud HSM). The HSM tier is FIPS 140-2 Level 3 validated.
- **Key rotation with rotation period.** Set a rotation period (e.g., 90 days) and Cloud KMS automatically rotates the key material. Old key material is retained for decryption.
- **Cloud External Key Manager (Cloud EKM).** For customers who need to manage key material outside of Google's infrastructure, Cloud EKM lets you use a supported external key management partner (Thales, Fortanix, etc.) while still using Cloud KMS APIs.
- **Audit logging.** All Cloud KMS operations are logged to Cloud Audit Logs, with retention configurable via log sinks.

**GCP-specific patterns:**

- **CMEK (Customer-Managed Encryption Keys).** You can use a Cloud KMS key to protect data in any GCP service that supports CMEK (BigQuery, Cloud Storage, Cloud SQL, Spanner, etc.). The key is used for envelope encryption: GCP services generate and use DEKs internally, wrapping them with your CMEK.
- **CSEK (Customer-Supplied Encryption Keys).** For Compute Engine and Cloud Storage, you can supply your own AES-256 keys directly. Google stores only a cryptographic hash of the key, not the key itself.

### 6.3 Azure Key Vault

Azure Key Vault provides a different model from AWS KMS and GCP Cloud KMS, with a stronger emphasis on certificate management and secret storage alongside key management.

**Key features:**

- **Three vault types.** Azure offers Key Vault (software-backed, for keys, secrets, and certificates) and Managed HSM (FIPS 140-2 Level 3 validated, for keys only). A third offering, Azure Dedicated HSM, provides FIPS 140-2 Level 3 validated single-tenant HSM appliances.
- **Key Vault vs. Managed HSM.** Key Vault is multi-tenant with software-backed key operations. Managed HSM is single-tenant with hardware-backed key operations. Managed HSM supports higher throughput (up to 10,000 operations per second per partition) and is required for compliance standards that mandate HSM-backed key material.
- **RBAC (Role-Based Access Control).** Azure Key Vault uses Azure RBAC to control access to keys, secrets, and certificates. This is separate from the vault's access policy — both can be used together.
- **Key rotation.** Key Vault supports automatic key rotation through a rotation policy set on the key. You can also rotate keys manually.
- **Soft-delete and purge protection.** Deleted keys and secrets are retained for a configurable period (default 90 days), allowing recovery. Purge protection prevents permanent deletion before the retention period expires.
- **Certificate lifecycle management.** Key Vault can issue, renew, and rotate certificates from public CAs (DigiCert, GlobalSign) or private CAs, with automatic expiration notifications.

**Azure-specific patterns:**

- **Azure Disk Encryption** uses Key Vault to protect VM disks. It uses BitLocker on Windows and DM-Crypt on Linux, with the encryption keys stored in Key Vault.
- **Azure Storage Service Encryption** (SSE) integrates with Key Vault for customer-managed encryption keys. Azure Storage handles the envelope encryption transparently.
- **Transparent Data Encryption (TDE)** for Azure SQL Database and SQL Managed Instance uses Key Vault as the key store for TDE protectors.

## 7. Lessons Learned — Five Key Patterns

Across these case studies, five patterns emerge consistently. They are the closest thing to universal truths in production encryption.

### 7.1 Envelope Encryption Is the Universal Primitive

Every system at scale — WhatsApp, Google, AWS, Apple, and every cloud KMS — encrypts data with a data key (DEK) and then wraps the DEK with a key encryption key (KEK). This is not a coincidence.

**Why it works:**

- The KEK is small and lives in a tightly controlled store (HSM, key management service, Secure Enclave).
- The DEK is ephemeral and generated per-request or per-session. Compromise of a DEK exposes only the data encrypted with that single DEK.
- The wrapped DEK is stored alongside the ciphertext, making the data self-describing.
- Key rotation requires only re-wrapping the DEK (if the KEK changes), not re-encrypting the data.

**Rule of thumb:** If you are designing a system that encrypts data, always use a two-tier key hierarchy. Never use the KEK directly to encrypt data.

### 7.2 Key Management Is Harder Than Cryptography

The most common encryption failures in production are not cipher breaks — they are key management failures. When a key is lost, data is permanently inaccessible. When a key is compromised, all data encrypted under it is exposed. When a key rotation policy is wrong, data may be encrypted with a retired key that cannot be decrypted.

**Production practices:**

- **Key rotation must be automatic.** Manual rotation is forgotten, delayed, or done incorrectly. AWS KMS, Cloud KMS, and Azure Key Vault all support automatic rotation. Use it.
- **Key rotation is a key version change, not a key replacement.** Old key material must be retained for decryption of existing data. Use a key versioning scheme (Tink's keyset, AWS multi-Region keys, Cloud KMS key versions).
- **Key backup is essential.** If you use your own HSM or BYOK, back up the key material in a secure, offline location. A lost key means permanently lost data.
- **Key access must be audited.** Every key operation should be logged with the principal, timestamp, key ID, and operation. CloudTrail, Cloud Audit Logs, and Azure Monitor all support this. Monitor for unexpected `Decrypt` calls — they are a sign of data exfiltration.

### 7.3 Use Authenticated Encryption, Always

Every major production system uses authenticated encryption — cipher modes that provide both confidentiality and integrity. AES-GCM (Google, AWS, Tink) and XChaCha20-Poly1305 (libsodium, WhatsApp) are the two dominant choices.

**Why authenticated encryption matters:**

- Without authentication, an attacker can modify ciphertext in transit. Even if the attacker cannot decrypt, they can flip bits in the ciphertext, causing the decrypted plaintext to be corrupted in a controlled way (bit-flipping attack).
- Authenticated encryption detects any modification. The tag check at decryption time produces a failure instead of corrupted data.
- AES-GCM and ChaCha20-Poly1305 are the only two authenticated encryption modes recommended by NIST (GCM) and the IETF (ChaCha20-Poly1305).

**Rule of thumb:** Never use AES-CBC, AES-CTR, or AES-ECB for new systems. They are not authenticated. AES-GCM or XChaCha20-Poly1305 are the only acceptable choices for general-purpose encryption.

### 7.4 Hardware-Backed Key Storage Is the Standard

At the infrastructure level, keys are stored in HSMs. At the client level, keys are stored in Secure Enclaves, TEEs, or trusted platform modules. Software-only key storage is a risk for any system that must protect data against a compromised host.

**Hardware security tiers:**

| Tier | Technology | Use Case | Examples |
|------|-----------|---------|---------|
| Dedicated HSM | FIPS 140-2 Level 3 appliance | Root KEK, CA key, high-value key | AWS CloudHSM, Azure Dedicated HSM |
| Shared KMS HSM | Multi-tenant HSM | Cloud KMS key material | AWS KMS, GCP Cloud KMS HSM, Azure Managed HSM |
| Secure Enclave | On-CPU secure co-processor | Client-side keys, biometrics | Apple Secure Enclave, Android TEE, Intel SGX |
| TPM | Discrete or firmware TPM | Device identity, disk encryption | TPM 2.0 in Windows, Linux, Chromebook |

### 7.5 Plan for the Post-Quantum Future

The transition to post-quantum cryptography is underway. NIST selected Kyber (KEM) and Dilithium (signatures) as the first standards in 2024. Apple's PQ3 in iMessage, Chrome's CECPQ2/Kyber in TLS, and Signal's PQXDH are production deployments.

**What to do now:**

- **Use hybrid key agreement** where possible: combine a classical curve (X25519) with a post-quantum KEM (Kyber, ML-KEM). This ensures that even if the classical algorithm is broken, the hybrid key remains secure — and vice versa. This is what Apple's PQ3 and Signal's PQXDH do.
- **Prefer libraries that are PQ-ready.** BoringSSL, libsodium (with `crypto_kx`), and Tink are all actively adding post-quantum support. Avoid libraries that are not maintained.
- **Assume harvest-now-decrypt-later.** Data encrypted today with classical cryptography alone will be decryptable when a sufficiently large quantum computer exists. For data with a long secrecy lifetime (10+ years), use a PQ-hybrid cipher today.
- **Monitor NIST standards.** The NIST post-quantum standardization process is ongoing. ML-KEM (FIPS 203), ML-DSA (FIPS 204), and SLH-DSA (FIPS 205) are the current standards. Libraries will adopt these over the next few years.

## 8. Bridge to the Interview

When you encounter an encryption question in a system design interview, you will not be expected to design the Signal Protocol from scratch. But you will be expected to make informed, defensible choices about key management, cipher selection, and operational security.

**Five questions to ask yourself in every design that involves encryption:**

1. **Where are the keys stored?** If keys are in the same system as the data they protect, you have a single point of compromise. Use a separate key store (KMS, HSM, Secure Enclave).
2. **What happens when a key is compromised?** Can you rotate the key without re-encrypting all data? (Envelope encryption makes this possible.)
3. **What happens when a key is lost?** Is there a recovery mechanism, or is the data permanently lost? (Key escrow, Shamir's Secret Sharing, multi-region replication.)
4. **What is the threat model?** Who are you protecting against? An external attacker? A malicious insider? A compromised server? A government subpoena? Each threat model leads to different encryption choices.
5. **What is the performance budget?** How many encryption operations per second? How much latency per operation? Can you cache DEKs? Can you use hardware acceleration?

**Sample interview frameworks to reference:**

- **Data at rest:** "We will use envelope encryption with AES-256-GCM. The KEK lives in AWS KMS with automatic rotation. A DEK is generated per file and stored alongside the file, wrapped by the KEK."
- **Data in transit:** "We will use TLS 1.3 with AEAD cipher suites (TLS_AES_256_GCM_SHA384 or TLS_CHACHA20_POLY1305_SHA256). Certificates are managed by a public CA, with mutual TLS for service-to-service communication."
- **End-to-end encryption:** "We will use a hybrid encryption model: a per-conversation shared secret established via X25519 key agreement, with a Double Ratchet for forward secrecy. Message keys are derived from the ratchet state and never reused."

**Recommended further reading:**

- *Applied Cryptography* by Bruce Schneier — the classic reference, dated but foundational.
- *Serious Cryptography* by Jean-Philippe Aumasson — a modern, practical introduction with a focus on real-world primitives.
- *Cryptography Engineering* by Niels Ferguson, Bruce Schneier, and Tadayoshi Kohno — design principles for building secure systems.
- NIST SP 800-175B — "Guideline for Cryptographic Algorithm Selection and Key Management."
- The Signal Protocol Specification — the canonical reference for end-to-end encryption.
- Tink documentation — Google's cryptographic library, with excellent usage guides.
- AWS KMS Developer Guide, GCP Cloud KMS documentation, Azure Key Vault documentation — each contains detailed architecture and best-practice guidance.
