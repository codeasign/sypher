# Encryption — Under the Hood

You have already seen the broad strokes of encryption in system design. Now you will open the cryptographic engine and examine every gear. This page covers symmetric cipher modes, asymmetric key mechanics, the TLS 1.3 handshake, perfect forward secrecy, key derivation, authenticated encryption, and the nonce. By the end you will know exactly what happens when data moves through an encrypted channel.

---

## 1. Symmetric Encryption — AES Modes

AES operates on 16-byte blocks. How you chain those blocks together determines the mode. Each mode has different security properties, parallelism, and error characteristics.

**CBC (Cipher Block Chaining):** Each ciphertext block feeds into the XOR of the next plaintext block. This makes CBC sequential — you cannot encrypt block 2 until block 1 finishes. Decryption is parallelizable, but encryption is not. CBC requires padding (PKCS#7) because plaintext may not be a multiple of 16 bytes. Padding oracle attacks exploit error messages to recover plaintext byte by byte.

**CTR (Counter):** Encrypts incrementing counter values and XORs the result with plaintext. This is fully parallelizable — you can encrypt all blocks simultaneously. CTR does not need padding; it turns AES into a stream cipher. The nonce-plus-counter must never repeat across messages under the same key.

**GCM (Galois/Counter):** Built on CTR mode. It adds a GMAC authentication tag computed over the ciphertext and any associated data (AAD). GCM is an AEAD (Authenticated Encryption with Associated Data) mode. The tag is computed using GHASH, a polynomial evaluation in GF(2^128). On decryption, the receiver recomputes the tag and compares — if they differ, the data is rejected. GCM is the most widely used mode in TLS 1.2 and TLS 1.3.

## 2. Asymmetric Encryption — RSA vs ECC

Asymmetric encryption uses a key pair: a public key you share and a private key you keep secret. The two dominant families are RSA and Elliptic Curve Cryptography (ECC).

  └─────────────────────────────────────────────────────────────────────┘

/>

**RSA** relies on the difficulty of factoring large semiprimes. Generating a 2048-bit RSA keypair requires finding two ~1024-bit primes — a probabilistic process that can take seconds. RSA encryption and signature verification are fast (small exponent, often e=65537), but decryption and signing are slow (exponentiation with the private exponent). RSA is not suitable for bulk encryption; you use it to encrypt an AES key (hybrid encryption).

**ECC** relies on the elliptic curve discrete logarithm problem. For curve P-256, a 256-bit key provides 128-bit security. ECC is faster than RSA at equivalent security levels, especially for key generation and signing. The most common curves are P-256 (secp256r1), P-384, X25519 (Curve25519 for Diffie-Hellman), and Ed25519 (Edwards curve for signatures).

In practice, you use ECDHE (Elliptic Curve Diffie-Hellman Ephemeral) for key exchange in TLS 1.3, and RSA or ECDSA for certificate signatures. ECDHE generates ephemeral key pairs per session, ensuring perfect forward secrecy.

## 3. TLS 1.3 Handshake — Step by Step

TLS 1.3 is the current standard. It reduces the handshake from two round trips (TLS 1.2) to one round trip on first connection, and zero round trips on resumption.

The handshake proceeds as follows:

**Step 1 — ClientHello:** You send your supported cipher suites, key exchange groups (X25519, P-256), and a public key share for each group. This is the "key_share" extension that makes TLS 1.3 faster than 1.2 — the server already has your ECDHE share and can derive the shared secret immediately.

**Step 2 — Server derives handshake secret:** The server picks its own ECDHE private key, computes the ECDHE shared secret, then runs HKDF-Extract to create the handshake secret. This is done with the `early_secret` derivation.

**Step 3 — ServerHello:** The server responds with its chosen cipher suite, its ECDHE public key share, and a random nonce. At this point, both sides can compute the handshake traffic key.

**Steps 4-7 — Encrypted flight:** The server sends its certificate, a CertificateVerify message (signed with its private key to prove possession), and a Finished message (a MAC of the entire handshake transcript). All of these are encrypted with the handshake traffic key.

**Steps 8-10 — You verify:** You validate the certificate chain, compute the handshake secret from the ECDHE shared secret, and verify the Finished MAC. This proves the handshake was not tampered with.

**Step 11-13 — Your Finished and traffic secret:** You send your Finished message, encrypted under the handshake traffic key. The server verifies it. Both sides then derive the application traffic secret using HKDF-Expand-Label. Application data flows encrypted under this key.

**Key derivation sequence in TLS 1.3:**

```
PSK (or 0) ──> HKDF-Extract ──> early_secret
                                     │
                                     ├─ HKDF-Expand-Label ──> binder_key
                                     │
                              (EC)DHE input ──> HKDF-Extract ──> handshake_secret
                                                                       │
                                                                       ├─ HKDF-Expand-Label ──> handshake_traffic_secret
                                                                       │
                                                                 0 ──> HKDF-Extract ──> master_secret
                                                                                           │
                                                                                           ├─ HKDF-Expand-Label ──> application_traffic_secret
```

Each stage uses the output of the previous Extract as the salt for the next. This is a "salt chain" that cryptographically binds every stage of the handshake together.

## 4. Perfect Forward Secrecy

Perfect Forward Secrecy (PFS) ensures that compromising the server's long-term private key does not compromise past session keys. PFS is achieved through ephemeral Diffie-Hellman key exchange.

In TLS 1.3, every key exchange is ephemeral — there is no static RSA key exchange option. The server's certificate is used only for authentication (signing the CertificateVerify message), not for encrypting the pre-master secret. This means that even if an attacker records every encrypted byte of your traffic and later steals the server's private key, they cannot decrypt any past session. They would need one of the ephemeral private keys, which were deleted after the handshake.

## 5. Key Derivation — PBKDF2 and HKDF

Raw key material from a Diffie-Hellman exchange or a user password is not uniformly random. Key derivation functions (KDFs) transform it into cryptographically strong key material.

**PBKDF2** applies a pseudorandom function (usually HMAC-SHA256) many thousands of times. Each iteration depends on the previous one, making it inherently sequential — you cannot parallelize it. This is intentional: it forces legitimate users to do N iterations (fast enough) but also forces attackers to do N iterations per password guess (slow enough). For password storage in 2026, you should prefer Argon2id over PBKDF2, but PBKDF2 remains important because it is the only KDF supported by many hardware security modules (HSMs).

**HKDF** (HMAC-based Key Derivation Function) is used in TLS 1.3, IPsec, and WireGuard. It has two phases:

- **Extract:** Takes the input key material (IKM) and a salt, produces a pseudorandom key (PRK). This step "smooths" any non-uniformity in the IKM.
- **Expand:** Takes the PRK and a context string (info), produces the desired length of key material in a streaming fashion. Each block is chained from the previous one.

HKDF is designed to be fast and efficient. It is not suitable for password-based derivation because it lacks a work factor — you would use PBKDF2 or Argon2 on top of HKDF for password hashing.

## 6. AEAD — Authenticated Encryption with Associated Data

AEAD provides confidentiality, integrity, and authenticity in a single operation. GCM and ChaCha20-Poly1305 are the two dominant AEAD constructions.

The key property of AEAD is that you cannot accidentally encrypt without authenticating. The ciphertext is always paired with a tag. If an attacker modifies even one bit of the ciphertext, the tag will not verify, and the receiver rejects the message. This prevents chosen-ciphertext attacks (CCA) where an attacker manipulates ciphertext blocks and observes the response.

**GCM** uses GHASH, a polynomial evaluation in the Galois field GF(2^128). The field is defined by the irreducible polynomial x^128 + x^7 + x^2 + x + 1. Multiplication in this field is the core operation — it is fast in hardware but slow in software without AES-NI and PCLMULQDQ instructions.

**ChaCha20-Poly1305** uses the ChaCha20 stream cipher (a variant of Salsa20) with the Poly1305 MAC. It is faster than AES-GCM in software (no hardware acceleration needed), making it ideal for mobile devices. ChaCha20-Poly1305 is the mandatory cipher suite in TLS 1.3 for devices that lack AES-NI.

## 7. Data Structures

### 7.1 Cipher Block Chaining — The State Machine

AES-CBC encryption can be modeled as a stateful process. Each block depends on the previous ciphertext block.

```
CBC Encryption State:
  state = IV (16 bytes, random)
  for each plaintext block P[i]:
    block = P[i] XOR state
    C[i] = AES_encrypt(block, key)
    state = C[i]
  output: C[0] || C[1] || ... || C[n-1]

CBC Decryption State:
  for each ciphertext block C[i]:
    block = AES_decrypt(C[i], key)
    P[i] = block XOR state
    state = C[i]
  output: P[0] || P[1] || ... || P[n-1]
  remove PKCS#7 padding from P[n-1]
```

### 7.2 Galois Field Multiplication — GHASH Core

GHASH operates on 128-bit blocks in GF(2^128). The field is defined by the polynomial x^128 + x^7 + x^2 + x + 1. Multiplication uses a carry-less multiplication instruction (PCLMULQDQ on x86) followed by reduction.

```
GHASH(H, A, C):
  Let H = AES_encrypt(zeros_128, key)   // hash subkey
  Let X[0] = 0                           // accumulator
  For each AAD block A[i]:
    X[i] = (X[i-1] XOR A[i]) * H        // in GF(2^128)
  For each ciphertext block C[j]:
    X[m+j] = (X[m+j-1] XOR C[j]) * H
  Let len_block = len(A) || len(C)       // 64-bit each, 128-bit total
  tag = (X[last] XOR len_block) * H
  return tag
```

The `*` operation is multiplication in GF(2^128), not integer multiplication. Each multiply involves a carry-less multiply (64 x 64 → 128, or 128 x 128 → 256) followed by reduction modulo the irreducible polynomial.

### 7.3 Elliptic Curve — Point Addition and Scalar Multiplication

An elliptic curve over a finite field is defined by the equation y^2 = x^3 + ax + b (Weierstrass form). For P-256, a = -3 and b = 0x5ac635d8aa3a93e7b3ebbd55769886bc651d06b0cc53b0f63bce3c3e27d2604b.

```
Point Addition:
  Given P = (x1, y1), Q = (x2, y2):
  If P == Q:  // point doubling
    s = (3 * x1^2 + a) / (2 * y1)    mod p
  Else:
    s = (y2 - y1) / (x2 - x1)        mod p

  x3 = s^2 - x1 - x2                  mod p
  y3 = s * (x1 - x3) - y1             mod p

Scalar Multiplication k * G:
  Double-and-add algorithm:
    result = point_at_infinity
    for each bit of k from MSB to LSB:
      result = double(result)
      if bit == 1:
        result = add(result, G)
    return result
```

The scalar multiplication k * G is the core operation in ECDHE. The security comes from the fact that given G and k * G, finding k is computationally infeasible (the elliptic curve discrete logarithm problem).

### 7.4 AES Key Schedule

The AES key schedule expands the original key (128, 192, or 256 bits) into round keys (10, 12, or 14 rounds respectively).

```
AES-128 Key Schedule:
  w[0..3] = original key (4 x 32-bit words)
  for i = 4 to 43:
    temp = w[i-1]
    if i % 4 == 0:
      temp = SubWord(RotWord(temp)) XOR Rcon[i/4]
    w[i] = w[i-4] XOR temp

  SubWord: apply AES S-box to each byte
  RotWord: circular left shift by one byte
  Rcon[i]: round constant = [2^(i-1), 0, 0, 0] in GF(2^8)
```

The round constants break symmetry. Without them, identical round keys would make the cipher vulnerable to slide attacks. The S-box provides non-linearity — it is the only non-linear operation in AES.

## 9. Consistency — What Can Go Wrong

Encryption is precise. Small mistakes lead to catastrophic failures.

### 9.1 Nonce Reuse Disaster

If you encrypt two messages with the same key and the same nonce in GCM or CTR mode, the encryption is broken:

```
C1 = P1 XOR AES_k(nonce || counter)
C2 = P2 XOR AES_k(nonce || counter)  // same keystream!

C1 XOR C2 = P1 XOR P2
```

An attacker who observes C1 and C2 can compute P1 XOR P2. With known plaintext in one message (e.g., HTTP headers), the attacker can recover the other plaintext entirely. Worse, in GCM, nonce reuse also reveals the authentication key H, allowing the attacker to forge arbitrary messages.

**Prevention:** Use a deterministic nonce construction (e.g., 96-bit nonce = 64-bit counter + 32-bit random) or use XChaCha20-Poly1305 with a 192-bit nonce, which allows random nonces safely.

### 9.2 Padding Oracle Attack

When a server decrypts data and reveals whether padding was valid (through error messages, timing, or status codes), an attacker can recover plaintext byte by byte:

```
CBC padding oracle attack:
  1. Attacker has C[n-1] || C[n] (last two ciphertext blocks)
  2. Attacker modifies the last byte of C[n-1]
  3. Sends modified ciphertext to server
  4. If server says "padding OK" → the last byte of P[n] is 0x01
  5. If server says "padding error" → try next value
  6. Repeat for all bytes of P[n], then move to previous block
```

**Prevention:** Use AEAD modes (GCM, ChaCha20-Poly1305) that do not use padding. If you must use CBC, implement "encrypt-then-MAC" with a constant-time MAC comparison.

### 9.3 Key Rotation

Keys must be rotated periodically to limit the damage of a compromise and to stay within the "safe usage limits" of the cipher.

## 10. Performance Considerations

### 10.1 AES-NI Hardware Acceleration

Modern x86 CPUs include the AES-NI instruction set, which provides hardware-accelerated AES operations:

```
Instruction    Operation                    Latency  Throughput
─────────────  ───────────────────────────  ───────  ──────────
AESENC         One round of AES encryption  4-8 cyc  1/cycle
AESENCLAST     Final round of AES enc.      4-8 cyc  1/cycle
AESDEC         One round of AES decryption  4-8 cyc  1/cycle
AESDECLAST     Final round of AES dec.      4-8 cyc  1/cycle
AESKEYGENASSIST Generate round key          8-12 cyc 1/cycle
PCLMULQDQ      Carry-less multiply (GHASH)  8-14 cyc 1/cycle
```

With AES-NI, AES-128-GCM can encrypt at 10+ Gbps per core. Without AES-NI, software AES runs at 200-400 Mbps — a 25-50x slowdown. You should assume AES-NI is available on server hardware, but never assume it on mobile devices.

### 10.3 ChaCha20 on Mobile

On mobile devices (ARM without ARMv8 Crypto Extensions), AES-GCM can be 3-5x slower than ChaCha20-Poly1305. This is why TLS 1.3 mandates ChaCha20-Poly1305 as a mandatory cipher suite:

```
TLS 1.3 mandatory cipher suites:
  TLS_AES_128_GCM_SHA256
  TLS_AES_256_GCM_SHA384
  TLS_CHACHA20_POLY1305_SHA256
```

On Apple devices with the Secure Enclave, or on ARMv8.2+ with cryptographic extensions, AES-GCM is hardware-accelerated and matches or exceeds ChaCha20. But on older ARM devices, or devices without crypto extensions, ChaCha20-Poly1305 is the performance winner.

## 11. Configuration — Choosing Your Cryptography

When you configure encryption for a system, you make several decisions that affect security, performance, and compatibility.

### 11.1 Cipher Suite Selection

For TLS 1.3, the cipher suite determines the AEAD cipher and the hash function:

```
TLS_AES_128_GCM_SHA256    │ AES-128  │ GCM     │ SHA-256  │ Best perf
TLS_AES_256_GCM_SHA384    │ AES-256  │ GCM     │ SHA-384  │ Highest sec
TLS_CHACHA20_POLY1305_    │ ChaCha20 │ Poly1305│ SHA-256  │ Mobile
  SHA256                  │          │         │          │
```

**Recommendation:** Offer all three, let the client and server negotiate. Use TLS_AES_128_GCM_SHA256 as the default for server-to-server (AES-NI available). Add TLS_CHACHA20_POLY1305_SHA256 for mobile clients.

### 11.2 Key Size Selection

| Symmetric | Security | Recommendation |
|-----------|----------|---------------|
| AES-128   | 128-bit  | Sufficient for all data until 2030+ |
| AES-256   | 256-bit  | Use for highly sensitive or long-term secrets |
| ChaCha20  | 256-bit  | Always 256-bit (no 128-bit variant) |

| Asymmetric | Security | Recommendation |
|------------|----------|---------------|
| RSA-2048   | 112-bit  | Legacy, avoid for new systems |
| RSA-3072   | 128-bit  | Acceptable, but prefer ECC |
| ECC P-256  | 128-bit  | Default for new systems |
| ECC P-384  | 192-bit  | For high-security / FIPS |
| X25519     | 128-bit  | Best for key exchange (libsodium) |
| Ed25519    | 128-bit  | Best for signatures (libsodium) |

### 11.3 TLS Version

| Version | Status | Notes |
|---------|--------|-------|
| TLS 1.0 | Deprecated | No PFS, weak CBC ciphers |
| TLS 1.1 | Deprecated | No PFS, weak CBC ciphers |
| TLS 1.2 | Transitional | Supports PFS, but complex cipher negotiation |
| TLS 1.3 | Current | Mandatory PFS, 1-RTT handshake, simpler ciphers |

**You should disable TLS 1.0, 1.1, and 1.2 on all new services.** TLS 1.2 is acceptable only when clients do not support TLS 1.3 (legacy systems), and even then only with PFS cipher suites (ECDHE).

### 11.4 HSM Integration

A Hardware Security Module (HSM) stores keys in tamper-resistant hardware. Keys never leave the HSM in plaintext form.

```
HSM-backed encryption flow:

  1. Application sends plaintext to HSM with key ID
  2. HSM encrypts using the key stored in its secure memory
  3. HSM returns ciphertext and (optionally) a wrapped data key
  4. Application stores ciphertext in database

  Key hierarchy:
  ┌──────────────┐
  │ Master Key   │ ← stored in HSM, never leaves
  │ (HSM-backed) │
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │ Data Key     │ ← encrypted by master key, stored alongside data
  │ (per-record) │
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │ Plaintext    │
  │ (encrypted)  │
  └──────────────┘
```

**AWS KMS** uses this pattern: a Customer Master Key (CMK) stays in the HSM, and you call GenerateDataKey to get a data key encrypted under the CMK. You use the plaintext data key for local encryption and store the encrypted data key with the ciphertext. On decryption, you send the encrypted data key to KMS, which returns the plaintext data key.

Here is a representative TLS 1.3 server configuration for a production service:

**Cipher suites (in order of preference):**
```
TLS_AES_128_GCM_SHA256
TLS_CHACHA20_POLY1305_SHA256
TLS_AES_256_GCM_SHA384
```

**Supported groups (key exchange):**
```

**Signature algorithms:**
```
