### Challenge 1: Symmetric vs Asymmetric Encryption

**Question.** A teammate proposes using AES-256 to exchange session keys over the public internet. Explain why this does not work, then compare symmetric and asymmetric encryption across five dimensions: key management, performance, use cases, security properties, and scalability.

**Model Answer.**

Symmetric encryption (AES, ChaCha20) uses a single shared secret to both encrypt and decrypt. Asymmetric encryption (RSA, ECDH, ECDSA) uses a public-private key pair.

**Why AES-256 alone fails for key exchange over the internet:** Both parties need the same secret to communicate. If no secure channel exists, transmitting the AES key in plaintext defeats encryption. The key-distribution problem is the exact problem asymmetric cryptography solves — one party publishes a public key, and the other encrypts the session key with it.

Key exchange protocols like ECDH (Elliptic Curve Diffie-Hellman) or RSA-KEM bridge the gap: asymmetric crypto establishes a shared secret over an untrusted channel, then both sides derive symmetric keys for bulk encryption.

**Five-dimension comparison:**

| Dimension | Symmetric | Asymmetric |
|---|---|---|
| Key management | Single shared secret; distribution is the hard problem | Public key published freely; private key kept secret |
| Performance | Fast (hardware-accelerated AES-NI); 1+ GB/s throughput | Slow (1000-10000x slower); limited to small payloads |
| Use cases | Bulk encryption: files, disk, TLS record layer, database columns | Key exchange, digital signatures, certificates, identity |
| Security properties | Confidentiality + integrity (with AEAD mode like GCM) | Confidentiality (public-key encryption), non-repudiation (signatures) |
| Scalability | O(n²) key pairs for n parties needing pairwise secrets | O(n) — each party publishes one public key |

**Takeaway:** Use asymmetric crypto for key exchange and signatures; use symmetric crypto for everything else. TLS, Signal, and PGP all follow this hybrid pattern.

### Challenge 2: Explain the TLS Handshake

**Question.** Walk through the TLS 1.3 handshake step by step. Cover how the client and server negotiate cipher suites, authenticate the server, establish a shared secret, and begin encrypted communication. Include the messages exchanged and the cryptographic operations each side performs.

**Model Answer.**

The TLS 1.3 handshake completes in one round trip (1-RTT) for new connections and 0-RTT for resumed ones. Here is the full flow:

**Step 1 — ClientHello.** The client sends:
- Supported protocol versions (e.g., TLS 1.3, 1.2)
- Cipher suites (e.g., TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256)
- Key share: an ephemeral X25519 or P-256 public key (via ECDHE)
- Random nonce

**Step 2 — ServerHello.** The server responds with:
- Selected protocol version and cipher suite
- Its own ephemeral ECDHE public key
- Server certificate chain (X.509, signed by a CA)
- CertificateVerify: signature over the handshake transcript proving possession of the private key
- Finished: MAC of the handshake using the derived symmetric key

At this point, both sides compute the shared secret via ECDHE, then derive the symmetric session keys using a key-derivation function (HKDF).

**Step 3 — Client Finished.** The client sends its own Finished message: a MAC of the handshake transcript. All subsequent data is encrypted under the session keys using an AEAD cipher (AES-GCM or ChaCha20-Poly1305).

**Cryptographic operations performed:**
- Client: ECDHE key generation, shared-secret computation, HKDF expansion, AEAD encryption
- Server: ECDHE key generation, shared-secret computation, HKDF expansion, signature verification, AEAD encryption
- Both: transcript hash computation (SHA-256 or SHA-384)

**Security guarantees established:**
- Server authentication (via certificate + CertificateVerify signature)
- Forward secrecy (ephemeral keys are discarded after the session — compromise of the long-term private key does not reveal past sessions)
- Confidentiality and integrity (AEAD encryption of all records)

### Challenge 3: Key Exchange — Diffie-Hellman and ECDH

**Question.** Explain how Diffie-Hellman key exchange allows two parties to establish a shared secret over an untrusted channel. Then describe elliptic-curve Diffie-Hellman (ECDH) and why it is preferred over classical DH in modern systems. Include the mathematical intuition without requiring a full number-theory proof.

**Model Answer.**

**Classic Diffie-Hellman (DH).** Alice and Bob agree on public parameters: a prime `p` and a generator `g`. Each picks a private random integer (`a` and `b`), computes a public value (`g^a mod p` and `g^b mod p`), and transmits it. The shared secret is `g^(ab) mod p`, which both compute independently but an eavesdropper cannot derive because the discrete-log problem is computationally hard.

```
Public: g, p
Alice:  a (private)  →  g^a mod p (public)     Shared: (g^b)^a = g^(ab)
Bob:    b (private)  →  g^b mod p (public)     Shared: (g^a)^b = g^(ab)
Eve:    sees g^a, g^b  →  cannot compute g^(ab) without a or b
```

**Elliptic-Curve Diffie-Hellman (ECDH).** Instead of exponentiation in a multiplicative group, ECDH works over points on an elliptic curve. Alice and Bob agree on a curve (e.g., Curve25519 or P-256) and a generator point `G`. Each multiplies `G` by their private scalar to produce a public point: `A = aG`, `B = bG`. The shared secret is the x-coordinate of `aB = bA = abG`.

**Why ECDH is preferred:**
- **Smaller keys:** ECDH-256 provides 128-bit security (equivalent to RSA-3072 or DH-3072). A 256-bit ECDH key fits in 32 bytes; DH-3072 requires 384 bytes.
- **Faster computation:** Elliptic-curve point multiplication is faster than modular exponentiation on equivalent hardware. ECDH performs 3-5x faster than classical DH at the same security level.
- **Lower bandwidth:** Smaller public keys reduce handshake size — critical for mobile and IoT.
- **Side-channel resistance:** Constant-time implementations of X25519 are simpler and more auditable than classical DH.

**Real-world usage:** TLS 1.3 mandates (EC)DHE key exchange. Signal protocol uses X25519 (Curve25519 ECDH) for all initial key agreements. WireGuard uses Curve25519 for session key establishment.

### Challenge 4: Design Encryption for a Chat Application

**Question.** Design the encryption architecture for a real-time chat application that supports one-on-one messages, group chats with up to 256 participants, and message history synced across multiple devices. Requirements: messages must be encrypted end-to-end; the server should never have access to plaintext; users authenticate with a password; and the system must support offline message delivery.

**Model Answer.**

**Core protocol — Signal Protocol double ratchet.**
Each conversation uses a chain of symmetric keys that ratchet forward with every message. This provides:
- **Forward secrecy:** Compromising a message key does not reveal past messages
- **Future secrecy (self-healing):** A compromised ratchet state recovers after one message from the other party
- **Post-compromise security:** A new DH ratchet step heals the session after key compromise

**One-on-one chat flow:**

1. **Registration:** Each client generates a long-term identity key pair (Ed25519), a signed pre-key (signed by the identity key), and a bundle of one-time pre-keys. These are uploaded to the server.

2. **Session setup (X3DH):** When Alice wants to message Bob for the first time, she fetches Bob's pre-key bundle from the server, performs three Diffie-Hellman computations (identity-to-identity, identity-to-ephemeral, ephemeral-to-signed-pre-key), and derives an initial root key.

3. **Message encryption:** Each message extends the key chain:
   - Sender derives a new message key from the current ratchet state
   - Encrypts the message with AES-256-GCM (or ChaCha20-Poly1305)
   - Attaches the new ratchet public key
   - Sender ratchets the key chain forward

4. **Receiver decrypts:**
   - Uses the attached ratchet public key to advance its own ratchet
   - Derives the same message key
   - Decrypts and verifies the AEAD tag

**Group chats — sender keys.**
For groups with up to 256 participants, each member generates a symmetric sender key and encrypts it individually for each group member using their pairwise session. All subsequent group messages use the sender key with a per-message ratchet. This avoids O(n²) encryption operations per message.

**Multi-device sync.**
Each device maintains its own identity key. The server stores encrypted message payloads keyed by (sender_id, recipient_device_id, message_id) and delivers them on demand. When a device comes online, it:
1. Fetches its pending message queue
2. Re-establishes pairwise sessions with each sender if needed (using the stored one-time pre-keys)
3. Syncs group sender keys via an encrypted device-to-device channel

**Offline delivery.**
Messages are stored server-side encrypted (payload encrypted with the per-message key; the server only holds ciphertext). The server delivers them when the recipient's device connects. The message key header contains enough information (ratchet public key, chain index) for the recipient to decrypt after receiving.

**Authentication.**
Password-based authentication uses SRP (Secure Remote Password) or PAKE (Password-Authenticated Key Exchange) so the password never reaches the server in plaintext. The identity key is derived from a seed encrypted with a key-stretching function (Argon2id) applied to the user's password — this binds the identity to the credential without server-side key storage.

### Challenge 5: Design Encryption at Rest for a Database

**Question.** Design an encryption-at-rest system for a relational database storing personally identifiable information (PII) including names, email addresses, Social Security numbers, and medical records. Requirements: column-level encryption for sensitive fields; application-layer encryption so the database operator cannot read plaintext; support for search and equality queries on encrypted data; key rotation without re-encrypting the entire dataset.

**Model Answer.**

**Architecture overview.**
The system uses a three-tier key hierarchy, application-layer encryption, and deterministic encryption for search.

**Key hierarchy:**

1. **Master Key (MK):** Stored in a hardware security module (HSM) or cloud KMS (AWS KMS, Azure Key Vault, GCP Cloud KMS). Never leaves the HSM. Rotated manually on a schedule (e.g., annually).

2. **Key Encryption Keys (KEKs):** One per tenant or application module. Encrypted under the master key. Stored in a key management database alongside the encrypted data.

3. **Data Encryption Keys (DEKs):** One per column or per row (for fine-grained isolation). Encrypted under the relevant KEK. The encrypted DEK is stored alongside the data — either in a separate `_keys` table or in a column header.

**Encryption operations:**

- **Column-level encryption:** Each sensitive column has an associated DEK. On write, the application fetches the DEK from the key store, decrypts it using the KEK (which requires a call to the HSM/KMS), encrypts the plaintext value, and stores the ciphertext.

- **Deterministic encryption for search:** For columns that require exact-match queries (e.g., email lookup for login), use AES-SIV (RFC 5297) — a deterministic authenticated encryption mode. SIV is keyed, so the same plaintext always produces the same ciphertext under the same key. This supports index lookups, JOINs, and UNIQUE constraints on ciphertext.

- **Search on encrypted data:**
  - *Exact match:* Deterministic encryption with database index on the ciphertext column
  - *Partial match / LIKE queries:* Blind indexing — store a hash of substrings (e.g., trigrams) in a separate index table. The application computes the substring hashes and queries them.
  - *Range queries for encrypted numbers:* Order-preserving encryption (OPE) is theoretically available but is **not recommended** — it leaks ordering information. Instead, perform range filtering on the application side after fetching a bounded set.

**Key rotation without full re-encryption:**

The key hierarchy makes rotation efficient:
- To rotate the master key: re-wrap all KEKs under the new MK. DEKs remain unchanged and are not re-encrypted.
- To rotate a KEK: re-wrap the DEKs under the new KEK. The underlying data is unaffected.
- To rotate a DEK: the application reads each row, decrypts with the old DEK, re-encrypts with the new DEK, and writes back. This is a background process that processes batches of rows.

The key store maintains a version number. Each DEK is stored with its version, allowing multiple versions to coexist during rotation.

**Threat model and access control:**

| Threat | Mitigation |
|---|---|
| DB admin reads raw data | Application-layer encryption; DB sees only ciphertext |
| KMS compromise | Master key in HSM with automatic key rotation; audit logging |
| Side-channel through query patterns | Pad queries to fixed size; use constant-time comparison |
| Rollback attack | Version counter in key store; monotonic sequence |

### Challenge 6: Design Key Rotation for a Microservices Platform

**Question.** Design a key rotation system for a platform with 50+ microservices, each using symmetric keys to sign internal JWTs, encrypt inter-service message bus payloads, and authenticate with external APIs. Requirements: zero downtime during rotation; automatic rotation every 90 days; ability to revoke a compromised key within 5 minutes; backward compatibility during the rotation window.

**Model Answer.**

**Key version strategy — N-1 backward compatibility.**

Every key is versioned. At any point, two key versions exist: the **active** version (used for signing/encrypting new data) and the **previous** version (accepted for verification/decryption of data already in flight). After the rotation window expires, the previous version is retired.

**Key states:**
- `ACTIVE`: Used for all new cryptographic operations
- `PENDING_ROTATION`: Previously active; accepted for verification only
- `RETIRED`: No longer accepted for any operation; data encrypted under this key must be re-encrypted
- `COMPROMISED`: Emergency revoke — immediately rejected by all services

**Architecture components:**

1. **Key Management Service (KMS):** A centralized service that stores keys, handles rotation, and serves key metadata to all microservices. The KMS itself stores keys in a secure vault (Vault by HashiCorp, AWS KMS, or Azure Key Vault).

2. **Key distribution — push vs pull:**
   - **Pull model (preferred):** Each service fetches key metadata (key_id, version, public key or encrypted DEK) on startup and caches it with a 60-second TTL. When a key is rotated, services pick up the change within one TTL period.
   - **Push model (optional):** KMS publishes rotation events to a message bus. Services subscribe and update their local cache immediately.

3. **Rotation workflow — scheduled (90-day):**

   a. **Pre-rotation (Day 89):** KMS generates a new key version. It is marked `PENDING_ACTIVE` but not used yet. Services prefetch it.
   b. **Activation (Day 90):** KMS transitions the new version to `ACTIVE` and the old version to `PENDING_ROTATION`. Services begin using the new key for signing.
   c. **Verification window (Days 90-97):** Tokens signed with either version are accepted. The previous version remains available for decryption.
   d. **Retirement (Day 97):** KMS transitions the old version to `RETIRED`. Services reject tokens signed with it. Data encrypted under the retired key is re-encrypted in a background job.
   e. **Cleanup (Day 98):** The retired key material is destroyed (unless required for audit).

4. **Emergency revocation (5-minute target):**

   a. Operator flags a key as `COMPROMISED` via the KMS admin API.
   b. KMS updates the key state and increments a global version counter.
   c. Each service polls the KMS every 60 seconds. On detecting a change in the version counter, it fetches the full key manifest.
   d. Services immediately stop accepting any operation referencing the compromised key ID.
   e. A `compromised_key` event fires to the incident-response system.
   f. All tokens signed with the compromised key are re-issued within 5 minutes using a rotation-specific endpoint.

**Key naming and uniqueness:**

Each key is identified by `{service_name}:{purpose}:{version}`. Example: `payment-service:jwt-signing:v3`.

**Audit and observability:**

- Every key operation (generation, rotation, revocation, usage) is logged to an immutable audit log.
- Prometheus metrics: `key_version_count`, `key_rotation_duration_seconds`, `key_compromise_events_total`.
- Services export the key version they are currently using as a metric for dashboards.

### Challenge 7: Design End-to-End Encryption for a Collaboration Platform

**Question.** Design end-to-end encryption for a real-time collaboration platform similar to Google Docs or Notion. Requirements: multiple users edit the same document simultaneously; changes sync via operational transforms (OT) or conflict-free replicated data types (CRDTs); the server facilitates sync but must never see document plaintext; users join and leave shared documents dynamically; the system supports 100+ concurrent editors per document.

**Model Answer.**

**Core challenge:** Operational transforms and CRDTs require the server to apply, merge, and broadcast edits. If edits are encrypted individually, the server cannot merge them. The solution is to encrypt at a higher granularity — per operation, with server-side operation ordering but no plaintext access.

**Architecture — encrypted CRDT with key server.**

**Key insight:** Split the document into chunks (paragraphs, blocks, or cells). Each chunk has its own encryption key. The server syncs encrypted CRDT operations without understanding their content.

**Component overview:**

1. **Document key hierarchy:**
   - **Document root key (DRK):** A symmetric key generated when the document is created. Known only to collaborators.
   - **Chunk keys:** Derived from the DRK via HKDF with the chunk index as context. Each chunk (paragraph, table cell, embedded block) has its own key.
   - **Per-operation nonce:** Each edit uses a unique nonce (counter + device ID) to ensure AEAD security.

2. **Client-side operations:**
   - When a user edits a chunk, the client:
     a. Encrypts the edit operation (insert, delete, format) with the chunk key using AES-256-GCM
     b. Wraps the encrypted operation in a CRDT envelope with a Lamport timestamp and the chunk ID
     c. Sends the encrypted envelope to the server
   - The client stores the chunk keys in memory, derived from the DRK on join

3. **Server role (encryption-agnostic):**
   - Receives encrypted CRDT operations
   - Orders operations by timestamp (server acts as the total-order broadcast sequencer)
   - Stores the ordered operation log — never decrypts
   - Broadcasts ordered operations to all connected clients

4. **New collaborator join flow:**

   a. A user with access generates a **join token**: a signed JWT containing the document ID, the user's identity, and an ephemeral ECDH public key. The token is encrypted with the document root key (DRK) by an existing collaborator.

   b. The new user sends the join token to the key server (a sidecar service that holds the DRK encrypted under each collaborator's public key).

   c. The key server verifies the token signature, then encrypts the DRK with the new user's public key and returns it.

   d. The new user decrypts the DRK and derives all chunk keys.

5. **Dynamic membership — access revocation:**
   - When a collaborator is removed, the DRK is rotated:
     a. The client that performs the revocation generates a new DRK
     b. Re-encrypts all chunk keys under the new DRK
     c. Distributes the new DRK to remaining collaborators (encrypted under their public keys)
     d. The server stores a versioned key manifest: `{doc_id, drk_version, encrypted_drk_per_user}`
   - The removed user's old DRK version is still valid for history they already have, but they cannot decrypt future edits.

**Concurrent edit handling:**
- Each client maintains a local CRDT replica. When two clients edit the same chunk simultaneously, the CRDT merge logic converges both replicas to the same state — even though neither client (nor the server) decrypts the other's operation.
- The server witnesses only encrypted operations with Lamport timestamps. It detects conflicts by chunk ID and timestamp ordering. Clients resolve conflicts upon receiving and decrypting the broadcast.
- Chunk-level granularity means two users can edit different paragraphs without any conflict; editing the same paragraph produces a CRDT merge on the client.

**Threat model:**

| Threat | Mitigation |
|---|---|
| Server compromise | Server holds only ciphertext and ordering metadata, never plaintext |
| Compromised collaborator | DRK rotation on revocation; audit log of key access |
| Replay attack | Lamport timestamps; server rejects stale operations |
| Malicious key server | Key server is a pinned service; all key transactions are signed and logged |
| Side-channel via edit timing | Add random padding to encrypted operation sizes |

### Challenge 8: Design Encryption for a Global File Storage System

**Question.** Design the encryption architecture for a globally distributed file storage system similar to Dropbox, Google Drive, or S3. Requirements: client-side encryption so the storage provider cannot read file contents; support for sharing individual files and folders with specific users; efficient access to large files (multi-GB); incremental sync of partial file changes; the system must work across continents with high latency between regions.

**Model Answer.**

**Hybrid encryption with file-level key hierarchy.**

Each file has its own unique encryption key. Sharing is implemented by encrypting the file key for each authorized user. Large files are split into fixed-size blocks (4 MB), each encrypted independently, enabling partial reads, parallel upload/download, and incremental sync.

**Key hierarchy:**

1. **User Master Key (UMK):** An asymmetric key pair (Ed25519) generated client-side. The private key never leaves the user's device. The public key is uploaded to the server for sharing operations.

2. **File Key (FK):** A random 256-bit symmetric key (AES-256-GCM) generated per file. Used to encrypt the file's block keys. Stored in a metadata envelope alongside the file.

3. **Block Keys (BK):** Derived from the FK using HKDF-expand with the block index as context. Each 4 MB block is encrypted with its block key and a unique nonce (block_index || chunk_offset).

**Encryption layout on disk:**

```
File metadata (encrypted under FK):
  ├── file_id, version, block_count
  ├── FK (encrypted for each authorized user)
  │   ├── user_alice: RSA-OAEP(FK)
  │   ├── user_bob:   RSA-OAEP(FK)
  │   └── ...
  └── block_hashes (SHA-256 of each encrypted block)

Block storage:
  ├── block_0000.bin: AES-256-GCM(block_data, key=BK_0, nonce=0)
  ├── block_0001.bin: AES-256-GCM(block_data, key=BK_1, nonce=1)
  └── ...
```

**Client operations:**

**Upload flow:**
1. Client splits the file into 4 MB blocks.
2. For each block, generates BK_i = HKDF-expand(FK, "block", i), then encrypts with AES-256-GCM.
3. Computes SHA-256 of each encrypted block (for integrity verification).
4. Uploads encrypted blocks in parallel (configurable concurrency, e.g., 8 threads).
5. Uploads the metadata envelope (encrypted FKs per user, block hashes).
6. The server stores blocks in its blob store (S3, GCS, Azure Blob) keyed by content hash for deduplication.

**Download flow:**
1. Client fetches the metadata envelope.
2. Decrypts the FK using the user's private key (RSA-OAEP decrypt).
3. Derives block keys for the required range (useful for partial reads — e.g., a video player seeking to minute 30).
4. Fetches only the needed encrypted blocks (chunked range requests).
5. Verifies each block's SHA-256 before decrypting.
6. Decrypts blocks in parallel and assembles the plaintext.

**Sharing:**
- When Alice shares a folder with Bob, the client:
  1. Fetches Bob's public key from the directory service.
  2. For each file in the folder, re-encrypts the FK under Bob's public key: `enc_FK_for_Bob = RSA-OAEP(Bob.pub, FK)`.
  3. Uploads the updated metadata envelope containing both Alice's and Bob's encrypted FKs.
  4. Bob's client picks up the new envelope on next sync and decrypts the FK.
- Sharing is transitive: if Bob shares with Carol, Bob's client re-encrypts the FK under Carol's public key.
- **Revocation:** When Alice revokes Bob's access, a new FK is generated for each file. All blocks are re-encrypted with the new FK. The new FK is distributed to remaining authorized users. (This is expensive but necessary — client-side encryption offers no way to selectively exclude without key rotation.)

**Global distribution and latency:**
- Encrypted blocks are stored in the nearest regional blob store (e.g., us-east-1, eu-west-1, ap-southeast-1).
- Block content-hash addressing enables deduplication across regions without cross-region plaintext transfer.
- Metadata (encrypted FKs, block hashes) is replicated to all regions via a global database (e.g., DynamoDB Global Tables, Spanner).
- Clients connect to the nearest region for both metadata and block operations.
- Block-level encryption enables clients to fetch blocks in parallel from multiple regions using byte-range requests.

**Incremental sync:**
- When a user modifies a large file (e.g., a 2 GB video), only the blocks that changed are re-encrypted and uploaded.
- The client computes block-level hashes (Merkle tree over blocks) and compares with the server's block manifest.
- Changed blocks are encrypted with the same FK (no key rotation for edits — only for sharing changes).
- Unchanged blocks remain on the server; their hash entries in the metadata are untouched.

## Summary

| Tier | Challenge | Core Concept | Key Trade-off |
|---|---|---|---|
| 1 | Symmetric vs asymmetric | Key distribution problem | Performance vs key management complexity |
| 1 | TLS handshake | 1-RTT authenticated key exchange | Certificate management overhead |
| 1 | Diffie-Hellman key exchange | Computational hardness assumptions | Post-quantum vulnerability |
| 2 | Chat encryption | Double ratchet + X3DH | Forward secrecy vs computation per message |
| 2 | Database encryption at rest | Key hierarchy with deterministic search | Search capability vs information leakage |
| 2 | Key rotation lifecycle | N-1 backward compatibility | Security window vs re-encryption cost |
| 3 | Collaborative editing E2E | Encrypted CRDT with chunk-level keys | Collision resolution complexity |
| 3 | Global file storage | File-level keys with block encryption | Re-encryption cost on revocation |
