## Company Case Studies

### Twitter: Timeline Cache at 500 Million Tweets Per Day

Twitter faced one of the most demanding caching problems in the industry. Every user's timeline must be assembled from tweets of accounts they follow. With 500 million tweets per day and 330 million monthly active users, generating timelines from the database for every request would require hundreds of database servers.

**The Challenge**: Timeline generation requires fetching the 800 most recent tweets from each followed account, then ranking and merging them. For a user following 200 accounts, this is 200 database queries per timeline request. At 10,000 timeline requests per second, that is 2 million queries per second, far beyond any reasonable database setup.

**The Solution**: Twitter uses a fanout-on-write approach with Redis as the distributed cache.

When a user tweets, the system looks up the user's followers, retrieves their timeline cache keys, and prepends the new tweet to each follower's timeline in Redis. Each follower's timeline is a Redis list with a capped length of 800 entries. When a follower requests their timeline, Redis returns the list in O(1) time.

**Fanout details**:
- Users with under 32,000 followers: fanout to all followers synchronously
- Users with over 32,000 followers (celebrities): fanout on read instead. The timeline is assembled by merging the celebrity's tweets with the cached fanout timeline
- Timeline list: Redis `LPUSH` + `LTRIM` to maintain 800 entries
- TTL: 24 hours with sliding expiry (renewed on each access)

**Key metrics**: Twitter's Redis cluster handles millions of timeline writes per second. Each tweet from a user with 12,000 followers generates 12,000 `LPUSH` operations. **Lesson learned**: The fanout-on-write model required careful handling of celebrities. Without the celebrity exception, a single tweet from a user with 100 million followers would generate 100 million Redis writes, saturating the cluster. The hybrid approach where high-follower accounts use fanout-on-read solved this.

### Meta: Cache Infrastructure Serving Billions

Meta operates one of the largest distributed cache systems in the world. Their cache infrastructure, built on Memcached (and later their own TAO system), serves billions of objects for Facebook, Instagram, and WhatsApp.

**The Challenge**: Meta's data graph contains trillions of edges (friendships, likes, comments, shares). Each page load requires fetching hundreds of edges. Database reads are too slow. Meta needed a cache that could handle 10 billion requests per second at peak.

**The Solution**: Meta built a multi-tier cache architecture called TAO (The Association Object).

**TAO architecture**:
- Tier 1: Memcached-based front-end cache, co-located on each application server (local memory)
- Tier 2: Regional Memcached clusters
- Tier 3: TAO service layer that handles cache misses by fetching from MySQL

TAO stores associations as typed edges: User A "likes" Post B, User C "friends" User D. Each edge is a cache entry. The cache key is a combination of the source object and the association type.

**Edge caching details**:
- Cache key format: `(source_id, association_type, target_id)`
- Range queries: `GET (source_id, association_type)` returns all edges of that type
- Write path: Writes go to MySQL first, then to cache
- Read path: Reads go to TAO, which checks cache, then MySQL

**Key metrics**: TAO handles billions of requests per second across Meta's data centers. The multi-tier approach achieves a 90 percent overall hit rate. The local cache tier alone absorbs 60 percent of reads, reducing network traffic to the regional cluster by 60 percent.

**Lesson learned**: Meta found that a single-tier cache was insufficient for their scale. The local cache tier reduced network bandwidth by 60 percent, dramatically lowering infrastructure costs. The tiered approach also provides natural isolation: a regional cluster failure degrades performance but does not cause a complete outage.

### Amazon: DynamoDB DAX Caching Layer

Amazon DynamoDB offers DAX (DynamoDB Accelerator), a fully managed, highly available, in-memory cache for DynamoDB tables.

**The Challenge**: DynamoDB provides single-digit millisecond latency for most workloads. But for read-heavy applications with hot keys, even 5 milliseconds per read adds up. High-traffic applications like Amazon's own product catalog needed single-digit microsecond latency.

**The Solution**: DAX is a write-through cache that sits between the application and DynamoDB. It intercepts reads and writes, serving reads from memory and propagating writes to DynamoDB synchronously.

**DAX architecture**:
- Primary node (writable) with multiple read replicas
- Consistent hashing for key distribution
- Write-through: writes go to primary node, which writes to DynamoDB synchronously
- Read replicas: serve read traffic, reducing load on the primary
- TTL: configurable, default 5 minutes

**Performance**: DAX reduces read latency from 5 milliseconds (DynamoDB) to under 1 millisecond. For hot keys, latency is under 100 microseconds. Write latency increases from 5ms to 8ms due to the write-through overhead.

### Netflix: EVCache for Real-Time Personalization

Netflix uses EVCache (Ephemeral Volatile Cache), a distributed Memcached-based caching layer, to serve personalized recommendations to 200 million subscribers.

**The Challenge**: Netflix's recommendation engine computes personalized rankings for each subscriber. The computation involves collaborative filtering, content-based filtering, and deep learning models. Running this computation for every page load would require hundreds of thousands of GPU hours per day.

**The Solution**: EVCache stores pre-computed recommendation results. When a subscriber opens Netflix, the application fetches their personalized recommendations from EVCache. The recommendations are computed asynchronously when the subscriber's viewing history changes.

**EVCache architecture**:
- Memcached-based, 200+ node cluster
- Consistent hashing with virtual nodes
- Zone-replicated: each cache key is replicated across three availability zones
- Auto-discovery: new nodes are discovered via Netflix's service discovery system
- TTL-based invalidation: most entries have a TTL of 2 to 4 hours

## Open Source Implementations

### Redis

Redis is the most widely used open-source distributed cache. It started as a simple key-value store and evolved into a multi-model data structure server.

**Key features for caching**:
- Data structures: strings, hashes, lists, sets, sorted sets, bitmaps, hyperloglogs, streams
- Eviction policies: LRU, LFU, TTL, random, noeviction
- Persistence: RDB snapshots, AOF logs, or both
- Cluster: Redis Cluster with automatic sharding and failover
- Replication: async master-replica with optional read-replica
- Lua scripting: server-side logic for atomic operations
- Pub/Sub: real-time messaging channels

**Typical deployment**: 3 master nodes, 3 replica nodes, each with 8 GB RAM. Redis Cluster distributes 16,384 hash slots across masters. Client library (redis-py, ioredis, jedis) handles routing.

**Limitations**: Single-threaded command processing limits CPU utilization. Multi-key operations only work within the same hash slot. Cluster mode requires client awareness of slot mapping.

### Memcached

Memcached is the older, simpler distributed cache. It is a pure key-value store with no data structures, no persistence, and no clustering.

**Key features**:
- Simple key-value with TTL
- Slab allocator for memory efficiency
- Multithreaded (unlike Redis) -- uses multiple cores
- ASCII and binary protocols
- UDP support for high-throughput, low-reliability use cases

**Where Memcached wins**: Memcached is faster than Redis for simple GET/SET operations because it is multithreaded and has a simpler codebase. For workloads that only need key-value storage with TTL, Memcached is the right choice.

**Where Memcached loses**: No data structures, no persistence, no clustering. Cache misses require application-level database fallback. No atomic operations like INCR, which makes rate limiting and counters harder.

### Comparison: Redis vs. Memcached

| Aspect | Redis | Memcached |
|--------|-------|-----------|
| Data structures | Strings, lists, sets, hashes, sorted sets, streams | Strings only |
| Persistence | RDB + AOF | None (ephemeral) |
| Clustering | Built-in Redis Cluster | Client-side sharding |
| Threading | Single-threaded event loop | Multi-threaded (4-16 threads) |
| Eviction | 6 policies | LRU only |
| Memory efficiency | Moderate (overhead per key) | High (slab allocator) |
| Atomic operations | INCR, DECR, Lua scripting | INCR, DECR only |
| Pub/Sub | Yes | No |
| Replication | Master-replica | No built-in replication |
| Latency (p50 GET) | 0.5-1ms | 0.3-0.8ms |
| Throughput | 100K ops/sec (single-threaded) | 500K ops/sec (multi-threaded) |

### Hazelcast and Apache Ignite

For Java ecosystems, Hazelcast and Apache Ignite provide distributed caches with JVM integration, compute capabilities, and strong consistency options.

**Hazelcast**: In-memory data grid with distributed maps, topics, queues, and locks. Supports JCache (JSR 107) standard. Provides near-cache (local cache in front of distributed cache) for ultra-low latency.

**Apache Ignite**: Similar to Hazelcast but with SQL support (ANSI-99), ACID transactions, and machine learning integration. Supports Durable Memory (data stored off-heap with optional persistence to disk).

**Use case**: These are overkill for simple caching but powerful for applications that need a unified platform for caching, compute, and data processing.

## Cloud Provider Implementations

### Amazon ElastiCache

AWS ElastiCache provides managed Redis and Memcached clusters. It handles provisioning, patching, monitoring, and failover.

**Redis Cluster configuration**:
- Node types: cache.r6g.large (13 GB) to cache.r6g.16xlarge (400 GB)
- Cluster mode: enabled (sharded) or disabled (single node with replicas)
- Multi-AZ: automatic failover across availability zones
- Encryption: TLS at rest and in transit
- Backup: automated snapshots with configurable retention
- Monitoring: CloudWatch metrics (CPUUtilization, CacheHits, CacheMisses, CurrConnections)

**Cost**: $0.15/hour for cache.t3.micro (dev) to $5.00/hour for cache.r6g.4xlarge (production). A 3-node Redis Cluster with 1 replica per node costs approximately $800/month.

**Best practice**: Enable auto-failover with Multi-AZ. Set `reserved-memory` to 25 percent of instance memory for Redis overhead. Monitor `Evictions` and `CurrConnections` CloudWatch metrics.

### Google Cloud Memorystore

GCP Memorystore provides managed Redis and Memcached services.

**Redis configuration**:
- Tier: Basic (single node, no replication) or Standard (multi-AZ with replication)
- Capacity: 1 GB to 300 GB
- Scaling: scale up (vertical) or scale out (horizontal with Redis Cluster)
- Security: VPC-only, IAM integration, encryption in transit
- Maintenance: automatic patching with configurable window

**Key difference**: Memorystore does not support Redis Cluster mode in all regions. Check regional availability before designing a multi-region architecture.

### Azure Cache for Redis

Azure Cache for Redis provides managed Redis across three tiers.

**Tiers**:
- Basic: single node, no SLA, dev/test
- Standard: two nodes (master/replica), 99.9 percent SLA
- Premium: cluster support, persistence, data replication, 99.9 percent SLA
- Enterprise: Redis Enterprise with active geo-replication, CRDT-based conflict resolution

**Enterprise tier feature**: Active geo-replication allows writes in multiple regions with conflict resolution. This is unique among cloud providers -- AWS and GCP require third-party tools for active-active Redis.

## Lessons Learned

### Lesson 1: Instrument Everything

The most common failure pattern in cache deployments is flying blind. Teams deploy a cache, see improved latency, and stop monitoring. Then a hit rate drop goes unnoticed for days.

**What to monitor**:
- Hit rate per node (not just cluster-wide)
- Eviction rate per node
- Command latency per operation type
- Memory fragmentation ratio
- Network throughput and connection count

**Alert thresholds**:
- Hit rate below 85 percent: check for key format changes
- Eviction rate above 100/s: memory pressure, scale out
- p99 latency above 10ms: check for slow commands or network issues
- Fragmentation ratio above 1.5: schedule restart or defragmentation

### Lesson 2: Design for Failure

Caches are ephemeral. Every cache node will fail eventually. Design your application to handle cache failure gracefully.

**Graceful degradation pattern**:
```python
def get_user_profile(user_id):
    try:
        cached = cache.get(f"user:{user_id}")
        if cached is not None:
            return deserialize(cached)
    except (ConnectionError, TimeoutError):
        pass  # Cache unavailable, fall through to database

    # Fallback to database
    profile = db.query("SELECT * FROM users WHERE id = %s", user_id)
    return profile
```

**Circuit breaker**: If the cache cluster is down, stop trying to connect for a configurable period. Use a circuit breaker library (Hystrix, resilience4j) to avoid connection storms.

### Lesson 3: Cache Key Hygiene

Cache keys are a design concern. Poor key design leads to collisions, hot spots, and hard-to-debug behavior.

**Key naming conventions**:
- `{entity}:{id}:{field}` -- `user:7421:profile`
- `{scope}:{entity}:{id}:{variant}` -- `us:product:993:details_v2`
- Include version number in keys when data format changes

**Key collision prevention**: Use a namespace prefix per service. The search service uses `search:*`, the recommendation service uses `rec:*`. This prevents accidental key collisions in shared cache clusters.

### Lesson 4: TTL Is Your Safety Net

TTL is the most important cache configuration. It bounds staleness, prevents memory leaks, and enables self-healing.

**TTL guidelines**:
- Reference data: 24 hours or explicit invalidation only
- Aggregated data (reports, analytics): 1 to 6 hours
- User data: 5 to 15 minutes
- Session data: 24 hours with sliding expiry (touch on each access)
- Real-time data: 30 seconds to 2 minutes
- Error responses (null caching): 30 to 60 seconds

**TTL jitter**: Add 10 to 20 percent randomness to TTLs to prevent synchronized expiration:
```python
ttl = base_ttl + random.randint(-int(base_ttl * 0.1), int(base_ttl * 0.1))
```

### Lesson 5: Plan for Scale

Cache clusters grow. What starts as a single node inevitably becomes a 20-node cluster. Plan for growth from day one.

**Design for scaling**:
- Use consistent hashing from the start (not fixed partitioning)
- Use virtual nodes (150 per physical node) for balanced load
- Use a proxy layer (Twemproxy, HAProxy) to decouple clients from topology
- Design keys to be shardable: avoid multi-key operations that span shards

**Capacity planning cadence**: Review cache capacity monthly. Calculate growth rate of keys, memory, and throughput. Project the date when you will hit 70 percent utilization. Order new nodes 2 weeks before that date.

### Lesson 6: Cache Invalidation Is Hard

Invalidation is the hardest part of caching. TTL-based invalidation is simple but introduces staleness. Event-driven invalidation is low-latency but complex. Write-through avoids staleness but adds write latency.

**Invalidation strategy selection**:
- For data that changes rarely and is not time-sensitive: TTL only
- For data that must be fresh within seconds: event-driven invalidation with CDC
- For data that must be fresh always: write-through cache
- For user-facing data that users modify: write-through or immediate invalidation on write

**The invalidation golden rule**: When in doubt, invalidate. Serving a stale value is worse than fetching from the database. A cache miss is a mild performance penalty. A stale value is a correctness bug.
