## How This Topic Appears

The distributed cache pattern appears in system design interviews in two forms: as the primary topic and as a component within a larger design.

**Primary topic questions**:
- Design a distributed caching system.
- Design Redis or Memcached from scratch.
- How would you implement a cache that handles 10 million requests per second?

**Component questions** (cache is part of a larger design):
- Design Twitter. How do you cache timelines for 330 million users?
- Design YouTube. How do you cache video metadata for billions of views per day?
- Design an e-commerce platform. How do you cache product catalog and pricing for 100 million products?
- Design a rate limiter. How do you cache rate limit counters for 1 billion API requests per day?
- Design a URL shortener. How do you cache the mapping from short URL to long URL?

**Design deep-dive questions**:
- Compare cache-aside and read-through caching. When would you choose each?
- How does consistent hashing work and why is it important for distributed caches?
- Describe the eviction policies Redis supports and when you would use each.
- How do you handle a hot key that receives 1 million requests per second?
- What happens to a distributed cache when a node fails? Walk through the recovery process.

**Troubleshooting questions**:
- Your cache hit rate dropped from 95 percent to 60 percent overnight. What do you check?
- Your database is overloaded even though you have a cache with 90 percent hit rate. What could be wrong?
- Users report seeing stale data up to 30 minutes after updating their profile. How do you fix this?

## What the Interviewer Tests

Interviewers asking about distributed caching are evaluating specific skills.

**Capacity to reason about trade-offs**: Caching is fundamentally about trade-offs. The interviewer wants to see you identify the tensions between speed, consistency, cost, and complexity. Strong candidates say "it depends" and explain the conditions.

**Understanding of distributed systems fundamentals**: Consistent hashing, replication, failure detection, and partition tolerance are core distributed systems concepts. The interviewer tests whether you understand these in the context of caching specifically.

**Operational awareness**: Caching in production fails in predictable ways. The interviewer wants to hear about cache stampedes, hot keys, cache penetration, and memory pressure. Candidates who mention monitoring, alerting, and graceful degradation signal production experience.

**Ability to simplify complex systems**: A distributed cache is itself a distributed system. The interviewer tests whether you can explain Redis Cluster or Memcached architecture in clear, structured terms without diving into irrelevant detail.

**Design judgment**: Not everything should be cached. The interviewer wants to see you evaluate when caching helps and when it adds unnecessary complexity.

## Level-by-Level Expectations

### L3 / Entry Level

**What they test**: Basic understanding of caching concepts. Can you explain what a cache is and why it is useful?

**Sample question**: "What is caching and when would you use it?"

**Target answer**: "Caching stores the results of expensive operations so future requests can serve data faster. I would use caching when the same data is read many times, when generating the data is expensive in terms of time or computation, and when some staleness is acceptable. The key trade-off is that cached data can become stale, so I need to decide how much staleness is acceptable and set appropriate TTLs. For example, caching database query results can reduce read latency from 50 milliseconds to under 1 millisecond, but the cached data might be up to the TTL duration out of date."

**What lifts answer to strong**: Mentioning specific numbers (50ms vs 1ms), naming the trade-off explicitly (staleness vs speed), and showing awareness that caching is not always the answer.

### L4 / Mid-Level

**What they test**: Can you design a caching layer for a moderate-scale application? Do you understand different caching patterns?

**Sample question**: "Design a caching layer for a news website that serves 1 million visitors per day."

**Target answer**: "I would start with the cache-aside pattern because it gives the application control over what to cache. For a news site, article pages and author bios are good cache candidates because they are read far more often than written. I would use Redis as the cache backend, deployed as a 3-node cluster for availability. My plan has four parts:"

"One, cache key design: `article:{id}:html` for rendered article HTML with a TTL of 10 minutes. `author:{id}:bio` for author bios with a TTL of 1 hour. `section:{slug}:top_articles` for section listings with a TTL of 5 minutes."

"Two, the application checks the cache for every article request. On a hit, return the cached HTML directly. On a miss, render the article page from the database, store it in the cache, and return it."

"Three, invalidation: when an editor updates an article, the CMS publishes an invalidation event to a message queue. A worker consumes the event and deletes the cache key. The next request re-renders the article."

"Four, for hot articles going viral, I would add request coalescing. If 1,000 requests arrive for a cold article simultaneously, only the first request renders the page and the other 999 wait for the cache to populate. This prevents a cache stampede on the rendering service."

**What lifts answer to strong**: Mentioning request coalescing and cache stampede protection shows production awareness. Distinguishing multiple cache entry types with different TTLs shows nuanced thinking.

### L5 / Senior

**What they test**: Can you design a distributed cache system from scratch? Do you understand the internals of distributed caching?

**Sample question**: "Design a distributed cache from scratch that handles 10 million operations per second across 50 nodes."

**Target answer**: "I will design a distributed cache with client-side routing, consistent hashing, replication, and an LRU eviction policy. Here is my design:"

"**Data model**: Key-value store. Keys are arbitrary byte strings up to 1 KB. Values are arbitrary byte strings up to 1 MB. Each key-value pair has a configurable TTL."

"**Partitioning**: Consistent hashing with virtual nodes. The keyspace is hashed using SHA-256 truncated to 64 bits. Each physical node is represented by 200 virtual points on the hash ring. When a node joins or leaves, only the keys on its adjacent virtual nodes are remapped, which is approximately 1/N of the keyspace."

"**Replication**: Each key is replicated to 3 nodes for fault tolerance. The primary node (first clockwise node on the hash ring) handles all writes. Two secondary nodes receive asynchronous replication. Reads can go to any of the 3 replicas."

"**Eviction**: Approximate LRU with sampling. Each node tracks the last access time of every key (24-bit timestamp in the key metadata). When memory is full, the node samples 20 random keys and evicts the one with the oldest access time. This provides nearly identical hit rate to true LRU with minimal overhead."

"**Request handling**: The client library computes the hash of the key, finds the primary node on the ring, and sends the request over a persistent TCP connection. The node processes commands in a single-threaded event loop using epoll for I/O multiplexing. Responses are sent back on the same connection."

"**Failure detection**: Nodes gossip their health status every 100 milliseconds using a PING-PONG protocol. If a node is unresponsive for 5 seconds, the cluster marks it as failing. The remaining nodes update their hash ring to exclude the failed node. Its replicas promote themselves to primary."

"**Capacity**: With 50 nodes, each node handling 200,000 operations per second, the cluster handles 10 million operations per second. Consistent hashing ensures 98 percent load balance across nodes. Virtual nodes prevent any single physical node from becoming overloaded."

**What lifts answer to strong**: Mentioning specific algorithms (SHA-256 truncation, 200 virtual nodes, 24-bit timestamp), graceful degradation (failure detection protocol), and providing concrete capacity numbers.

### L6 / Staff

**What they test**: Can you design a globally distributed, multi-region cache? Can you reason about consistency models and conflict resolution?

**Sample question**: "Design a globally distributed cache that serves users in North America, Europe, and Asia with under 10ms p99 latency."

**Target answer**: "I will design a multi-region, active-passive cache hierarchy. Each region has an independent cache cluster serving local traffic. Cross-region replication propagates writes asynchronously."

"**Local cluster design**: Each region has a Redis Cluster with 10 shards and 2 replicas per shard. This provides 80 GB of usable cache per region with 2x failover coverage."

"**Cache hierarchy within a region**: Tier 1 is a local memory cache on each application server (2 GB, 5-second TTL). Tier 2 is the regional Redis Cluster (80 GB, configurable TTL up to 1 hour). The local cache absorbs 60 percent of reads, reducing load on the regional cluster."

"**Cross-region replication**: The primary region (US-East) accepts writes. Writes are replicated to EU-West and Asia-Pacific via Redis replication over dedicated cross-region links. Replication latency is 50ms to EU and 120ms to Asia. Reads in secondary regions serve data that is 50-120ms stale."

"**Cache invalidation across regions**: When a write occurs in the primary region, we use a global invalidation bus built on Kafka with producers in the primary region and consumers in each secondary region. Invalidation messages propagate in under 200 milliseconds. Upon receiving an invalidation, a region deletes the cache key, and the next read fetches fresh data."

"**Handling regional failover**: If the primary region fails, we promote EU-West to primary. This involves redirecting writes to EU-West, updating DNS records, and establishing new replication links from EU-West to the remaining regions. The stale data in Asia-Pacific converges as replication from EU-West completes."

"**Consistency guarantees**: We provide eventual consistency across regions with a maximum staleness bound of 5 minutes (the TTL). For operations requiring strong consistency, the application must read from the primary region. We expose this via an API with a `consistency:strong` parameter that routes to the primary region."

**What lifts answer to strong**: Multi-tier design (local + regional), concrete inter-region latency numbers, invalidation bus design, and clear consistency guarantees.

### L7 / Principal

**What they test**: Strategic architecture decisions, cost optimization, and organization-level impact. Can you evaluate whether a cache is the right solution compared to alternatives?

**Sample question**: "Our e-commerce platform has 100 million products and 500 million page views per day. We currently use a Redis cluster with 50 nodes costing $50,000/month. Our team wants to double the cache size. Evaluate the proposal."

**Target answer**: "Before approving a 2x cache expansion, I need data on current cache effectiveness and an analysis of alternatives."

"**Current state analysis**: What is the current hit rate? If it is 95 percent or higher, doubling the cache will improve hit rate to maybe 96 or 97 percent. The marginal improvement does not justify $50,000/month. If the hit rate is 80 percent, doubling the cache might improve it to 92 percent, which could be worth the cost."

"**Access pattern analysis**: If the 80/20 rule applies (80 percent of requests hit 20 percent of products), then doubling the cache to cover 40 percent of products increases hit rate significantly. If access is uniformly distributed, the hit rate improvement from doubling cache is minimal."

"**Alternatives to evaluate**:"

"One, implement a local L1 cache on each application server. This often improves effective hit rate by 20 to 30 percent at zero infrastructure cost. Only the remaining 70 percent of requests need to hit the Redis cluster."

"Two, optimize TTLs. Are we caching data with TTLs longer than necessary? Shortening TTLs for data that changes frequently would increase hit rate by reducing stale entries that get invalidated early."

"Three, review the eviction policy. If we use noeviction or allkeys-random, switching to allkeys-lru keeps popular entries longer."

"Four, implement compression. If we store serialized JSON as-is, compressing with zstd at compression level 3 reduces entry size by 40 to 60 percent. This effectively doubles cache capacity without adding nodes."

"**Recommendation**: Implement L1 local caching and zstd compression first. Measure the impact over 2 weeks. If the effective hit rate after these changes is still under 90 percent, then and only then approve the cache expansion. This approach saves $500,000 in the first year while achieving better performance."

**What lifts answer to strong**: Evaluating marginal returns, quantifying cost-benefit trade-offs, proposing operational optimizations before capacity expansion, and suggesting measurable experiments.

### L8 / Distinguished

**What they test**: Cross-organizational impact and industry leadership. Can you design a caching platform used by dozens of teams?

**Sample question**: "Design a company-wide caching platform that 50 product teams will use for their services."

**Target answer**: "I would design a multi-tenant caching platform with strong isolation, self-service provisioning, and centralized monitoring."

"**Multi-tenancy model**: Each team gets a logically isolated cache instance within a shared physical cluster. Isolation is achieved through key prefixing (`teamA:*`), separate Redis databases (DB 0-15), and per-team memory quotas enforced by `maxmemory` settings."

"**Self-service provisioning**: Teams provision cache instances through a REST API or web UI. They specify capacity (2GB to 200GB), TTL defaults, eviction policy, and region. The platform automatically deploys the cluster, configures monitoring, and provides connection endpoints."

"**Service level objectives**: We offer three tiers: Platinum (sub-1ms p99 latency, 99.99 percent availability, automatic failover), Gold (sub-5ms p99, 99.9 percent, manual failover), and Silver (sub-10ms p99, 99.5 percent, no failover). Each tier has a cost model that teams pay for from their budget."

"**Centralized monitoring and debugging**: A Grafana dashboard per cache instance showing hit rate, latency percentiles, eviction rate, and memory usage. A slow-log explorer for diagnosing slow commands. A hot-key detection tool that alerts when a single key exceeds 10 percent of node throughput."

"**Platform-level optimization**: We pool resource across teams. Not all teams use peak capacity simultaneously. Over-provisioning 20 percent across the pool with 30 percent buffer is cheaper than each team provisioning separately with 50 percent buffer."

"**Governance**: A monthly review of cache utilization across the platform. Teams with hit rates under 80 percent or memory utilization under 50 percent are asked to right-size. This prevents cache sprawl and controls costs."

## Phrase Bank

Use these phrases to articulate cache concepts clearly and precisely during your interview.

**On cache placement**: "The cache sits between the application and the database, intercepting read requests to serve data from memory."

**On hit rate**: "Hit rate is the percentage of requests served from cache. A 95 percent hit rate means only 5 percent of requests reach the database."

**On TTL**: "TTL bounds maximum staleness. After the TTL expires, the entry is automatically removed and the next read fetches fresh data."

**On eviction policies**: "LRU evicts the least recently accessed keys when memory is full. LFU evicts the least frequently accessed keys. TTL evicts keys that have expired based on their time-to-live."

**On consistent hashing**: "Consistent hashing maps keys and nodes onto a hash ring. When a node is added or removed, only the keys on adjacent nodes need to be reassigned, not the entire keyset."

**On cache stampede**: "A cache stampede happens when many concurrent requests miss the cache simultaneously, all hitting the origin at once. We prevent this with request coalescing: one request fetches the data while others wait."

**On hot keys**: "A hot key receives a disproportionate share of traffic and can overwhelm a single cache node. Mitigations include local caching, key replication across nodes, and rate limiting at the application layer."

**On cache penetration**: "Cache penetration occurs when requests target keys that do not exist in the database. Without protection, every such request causes a cache miss and a database miss. We mitigate this by caching null results and using Bloom filters."

**On write-through**: "Write-through caching ensures the cache is always consistent with the database by writing to both synchronously. The trade-off is increased write latency."

**On write-behind**: "Write-behind caching acknowledges writes immediately and persists to the database asynchronously. This reduces write latency but risks data loss if the cache fails before persisting."

**On the reader-writer problem**: "If a cache is updated through a slow asynchronous process, readers may see stale data. Solutions include read-your-write consistency, invalidation queues, and short TTLs for mutable data."

**On stale reads**: "A stale read occurs when the cache returns an outdated value. The impact depends on the use case: stale product inventory is acceptable for a few seconds, stale authentication tokens are not."

**On cache warming**: "A cold cache has zero hit rate. We warm it by pre-populating hot keys before directing production traffic to it."

**On multi-tier caching**: "A multi-tier cache uses a small, fast local cache in front of a larger, slightly slower distributed cache. The local cache absorbs the highest-frequency reads."

**On cache-as-a-pass-through**: "If eviction happens too quickly, entries are removed before they serve any hits. This turns the cache into a pass-through with all the overhead and none of the benefit."

## Red Flags

Avoid these mistakes in your interview. They signal inexperience with production caching.

**Red flag 1: Claiming a cache guarantees consistency**. No distributed cache guarantees strong consistency by default. Explicitly state "eventual consistency within the TTL window" or "write-through achieves strong consistency at the cost of write latency."

**Red flag 2: Proposing a cache without mentioning eviction**. If you do not mention what happens when memory fills up, the interviewer will probe your understanding of eviction policies. Always discuss the eviction policy.

**Red flag 3: Ignoring failure modes**. If your design assumes the cache never fails, you miss the point. Discuss what happens when a cache node goes down: how does the application degrade gracefully?

**Red flag 4: Using the same TTL for everything**. A single TTL for all cached data signals lack of nuanced thinking. Distinguish between reference data (long TTL), user data (medium TTL), and real-time data (short TTL).

**Red flag 5: Caching everything**. If you propose caching every query result including rarely accessed data, the interviewer will question your judgment. Discuss the criteria for what to cache: read-to-write ratio, computation cost, access frequency.

**Red flag 6: Forgetting about serialization**. Caches store bytes, not objects. Mention serialization format (JSON, Protocol Buffers, MessagePack) and its impact on memory and CPU.

**Red flag 7: Ignoring the network**. The cache is a remote service accessed over the network. Mention connection pooling, network latency (0.5ms within AZ, 2-5ms across AZ), and the impact of network partitions.

**Red flag 8: No mention of monitoring**. If your system design does not include what you would monitor (hit rate, eviction rate, latency percentiles), it signals operational inexperience.

## Practice Drills

### Drill 1: Two-Minute Explanation

Set a timer for 2 minutes. Explain to an imaginary colleague what a distributed cache is and when to use it. Cover: problem (database bottleneck), solution (in-memory cache), trade-off (staleness), and one real-world example.

### Drill 2: Whiteboard Consistent Hashing

Draw a hash ring with 4 nodes and 8 keys. Trace what happens when a node fails. Trace what happens when a node is added. Calculate how many keys are remapped in each case.

### Drill 3: Design the Cache for Twitter Timelines

Take 10 minutes to design the cache layer for Twitter. Write down: the cache key structure, the data flow for tweeting, the data flow for reading a timeline, the hot key problem (celebrities), and the solution (fanout-on-read for high-follower users). Compare your design to the real Twitter approach.

### Drill 4: Evaluate a Cache Proposal

Scenario: "A team proposes adding a Redis cache with 100 GB to cache all database queries for an e-commerce site serving 10 million daily active users. The average query takes 20ms. The cache would cost $2,000 per month."

Evaluate: Is this a good investment? What data would you need to decide? What alternatives should you consider?

Answer structure: Calculate current database load (queries per second), estimate hit rate based on access patterns, calculate cost savings in database resources, compare to alternatives (query optimization, read replicas, materialized views).

### Drill 5: Triage a Cache Incident

Scenario: "The on-call engineer reports that the cache hit rate dropped from 94 percent to 55 percent at 2:00 AM. Database CPU is at 90 percent."

Walk through your triage process: Check deployment history (new code pushed?), check key format changes, check TTL changes, check cluster topology changes, check for traffic pattern shifts, check for mass evictions due to memory pressure. Propose a rollback or mitigation.

### Drill 6: Compare Caching Patterns

Write down the pros and cons of cache-aside, read-through, write-through, write-behind, and refresh-ahead in 5 minutes. Be ready to recommend one for a given scenario:
- A social media feed: refresh-ahead for hot feeds, cache-aside for cold feeds
- A user preferences service: write-through for strong consistency
- An analytics counter: write-behind for write throughput
- A product catalog: cache-aside with TTL-based invalidation

### Drill 7: Design a Bloom Filter for Cache Penetration

Describe how you would use a Bloom filter to prevent cache penetration. Calculate the memory for 100 million keys with a 1 percent false positive rate: `m = n * ln(p) / (ln(2)^2)` = 100 million * ln(0.01) / (0.48) = approximately 958 million bits = approximately 114 MB. With 7 hash functions, each lookup takes approximately 50 nanoseconds.
