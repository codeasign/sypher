## The Problem

Imagine your application serves 50,000 requests per second. Each request fetches user profiles, product catalogs, and session data from a relational database. The database, a powerful Postgres cluster, handles 10,000 reads per second comfortably. But at 50,000 requests per second, every query cascades. Connection pools saturate. Disk I/O spikes to 800 MB per second. Query latency climbs from 5 milliseconds to 200 milliseconds. Pages take seconds to load. Users refresh and resubmit, making the situation worse. The database becomes the bottleneck, and your application grinds to a halt.

This is not a hypothetical scenario. Every major internet service has faced this wall. The naive solution, throwing more database replicas at the problem, works only to a point. Each replica doubles your storage cost and adds replication lag. At 100 replicas, you are spending millions of dollars per year on hardware that sits mostly idle, waiting for the occasional spike. You need a way to serve data that does not require hitting the full database stack for every request.

The solution is caching: store the result of an expensive operation so subsequent requests can skip the computation. A single-server cache, like a local `HashMap` or an in-process Redis instance, works well for a single-node application. But modern systems run across dozens or hundreds of servers. A cache on server A is invisible to server B. If each server has its own local cache, they duplicate storage and drift out of sync. You need a cache that every server can reach, that grows with your data, and that survives server failures. That is a distributed cache.

Consider a social media feed. When 10 million users log in each morning, each one expects to see their personalized feed. Generating a feed requires fetching the last 200 posts from 150 followed accounts, ranking them by relevance, filtering blocked content, and rendering snippets. Doing this from the database for 10 million users would require 1.5 billion follow-relationship queries and 2 billion post fetches. A distributed cache stores the rendered feed keyed by user ID. The first request for user 42 computes the feed and stores it. The next 99 requests read from cache in under 5 milliseconds each. Your database load drops by 99 percent.

## The Insight

A distributed cache is a shared, in-memory data store that sits between your application servers and your persistent storage. It trades durability for speed: data lives in RAM, not on disk, so reads take microseconds instead of milliseconds. It trades consistency for availability: the cache might return stale data, but it will almost never refuse a request. And it trades simplicity for scale: the cache itself is a distributed system, with all the complexity that brings.

The real-world analogy is a library's reserve desk. The library holds millions of books in the stacks (your database). Fetching a book from the stacks takes 10 to 15 minutes (a database query). The reserve desk keeps high-demand books on a shelf behind the counter. When a student asks for "Introduction to Algorithms," the librarian checks the reserve shelf first. If it is there, the student gets the book in 30 seconds. If not, the librarian fetches it from the stacks, puts a copy on reserve, and hands it over. The reserve desk is small, holding only 200 books, but those 200 books serve 80 percent of requests. That is a cache: small, fast, and targeted at the most popular data.

## Core Vocabulary

Before diving deeper, establish these terms. They appear in every distributed cache discussion, from system design interviews to production postmortems.

**Cache Hit** -- A request finds the requested data in the cache. Measured in hits per second. A 95 percent hit rate means 95 of every 100 requests are served from cache.

**Cache Miss** -- A request does not find the requested data in the cache. The system must fetch the data from the origin (database, API, or computation) and typically populates the cache for future requests.

**Hit Rate / Miss Rate** -- The ratio of hits to total requests. Expressed as a percentage. Improving hit rate from 90 percent to 99 percent reduces origin load by 10x. This is the single most important operational metric for any cache system.

**Eviction Policy** -- The algorithm that decides which data to remove when the cache is full. Common policies include Least Recently Used (LRU), Least Frequently Used (LFU), First In First Out (FIFO), and Time-To-Live (TTL).

**Time-To-Live (TTL)** -- A duration assigned to each cached entry. After the TTL expires, the entry is automatically removed or marked stale. TTL is the primary mechanism for bounding staleness. Typical values range from 30 seconds for real-time data to 24 hours for reference data.

**Cache Stampede / Thundering Herd** -- A scenario where many concurrent requests miss the cache simultaneously, all hitting the origin at once. This can overload the database. Prevention strategies include request coalescing, early recomputation, and jittered TTLs.

**Write-Through Cache** -- A strategy where writes go to the cache first, and the cache synchronously writes to the database. Ensures cache and database are always consistent, but increases write latency.

**Write-Behind Cache** -- A strategy where writes go to the cache, and the cache asynchronously writes to the database. Reduces write latency but risks data loss if the cache fails before the write is persisted.

**Cache-Aside** -- The application explicitly checks the cache before reading from the database and writes to the cache after a miss. The cache does not interact with the database directly. This is the most common pattern in web applications.

**Read-Through Cache** -- The cache itself fetches data from the database on a miss. The application only talks to the cache. Simplifies application code but requires the cache to understand the database schema.

**Cache Invalidation** -- The process of removing or updating cached data when the source data changes. Invalidation is one of the two hard problems in computer science (along with naming and off-by-one errors). Strategies include TTL-based expiry, event-driven invalidation, and explicit eviction.

**Distributed Hash Table (DHT)** -- The underlying data structure for most distributed caches. Keys are mapped to nodes using consistent hashing, which minimizes redistribution when nodes are added or removed.

**Consistent Hashing** -- A hashing technique where each cache node is assigned positions on a hash ring. Data keys are hashed to positions on the same ring, and each key is stored on the nearest clockwise node. When a node is added or removed, only the keys on adjacent nodes need to be remapped, not the entire dataset.

**Replication Factor** -- The number of cache nodes that store a copy of each key-value pair. A replication factor of 3 means each piece of data exists on three different nodes, providing fault tolerance at the cost of storage and write overhead.

**Sharding / Partitioning** -- Dividing the cache dataset across multiple nodes so that each node stores a subset of the data. Sharding enables horizontal scaling but complicates queries that span multiple shards.

**Hot Key** -- A single cache key that receives a disproportionate share of traffic. A key accessed 100,000 times per second while other keys average 100 times per second. Hot keys can overload a single cache node and require special handling like local caching or replication.

**Cache Penetration** -- Requests for keys that are known not to exist in the database. Without protection, every request for a non-existent key causes a cache miss, then a database miss, and does not populate the cache. Attackers can exploit this to bypass the cache entirely. Mitigations include caching the null result for a short duration and using Bloom filters.

## How It Works Step by Step

The life of a cache request from the moment it reaches the application server until the response returns to the client.

### Step 1: Request Arrives

A client sends a GET request for a specific resource, such as the profile for user ID 74321. The load balancer routes the request to one of 50 application servers.

### Step 2: Application Generates Cache Key

The application constructs a cache key from the request parameters. For a user profile, the key might be `user:profile:74321`. The key must be deterministic: the same request always produces the same key. The application uses a hash of the key to determine which cache node holds the data.

### Step 3: Cache Node Lookup (Consistent Hashing)

The application hashes the key using a consistent hashing algorithm. It maps the hash value to a position on a hash ring and finds the nearest cache node in clockwise order. For a 6-node cluster, this operation completes in O(log n) using a sorted list of node positions.

### Step 4: Network Round Trip

The application sends the GET command to the target cache node over TCP. Typical latency within a data center is 0.5 to 2 milliseconds. The cache node receives the command, parses it, and looks up the key in its in-memory hash table.

### Step 5: Cache Hit Path

If the key exists and has not expired, the cache node returns the serialized value (typically JSON, Protocol Buffers, or a custom binary format). The application deserializes the value and returns it to the client. Total time: 1 to 5 milliseconds. This is the happy path.

### Step 6: Cache Miss Path

If the key does not exist, the cache node returns a miss indicator (nil, not found). The application must fetch the data from the origin database.

### Step 7: Origin Fetch

The application queries the database for the data. For a user profile, this is a single indexed query: `SELECT * FROM users WHERE id = 74321`. The database returns the row in 5 to 50 milliseconds, depending on index depth, buffer cache state, and disk I/O.

### Step 8: Cache Population

The application stores the fetched data in the cache with a TTL. The SET operation includes the key, value, and expiration time. For user profiles, a typical TTL is 300 seconds (5 minutes). The cache node stores the key-value pair in memory and sets a timer for eviction.

### Step 9: Request Coalescing (Stampede Protection)

Under high concurrency, 50 application servers might miss the cache simultaneously for the same key. Without protection, all 50 hit the database at once. Request coalescing ensures only one request fetches from the database while the others wait. This is implemented with a mutex or distributed lock keyed on the cache key. The first request acquires the lock and fetches the data. Subsequent requests see the lock and either wait briefly or receive a stale cached value.

### Step 10: Response

The application returns the data to the client. If the response came from cache, it might include a header `X-Cache: HIT` or `X-Cache: MISS` for debugging. The client sees sub-10-millisecond latency regardless of the hit or miss outcome, because the cache miss penalty is amortized across subsequent requests.

### Step 11: Eviction

The cache has finite memory, typically 512 MB to 64 GB per node. When memory is full and a new entry must be stored, the eviction policy selects victims. LRU eviction tracks access times and removes the oldest accessed entries. LFU eviction tracks access frequency and removes the least popular entries. In practice, many distributed caches use a hybrid approach: approximate LRU (Redis) or a combination of recency and frequency.

### Step 12: TTL Expiry

Even if the cache has free memory, entries expire after their TTL. Expired entries are removed lazily (on access) or actively (background sweeper). Active expiry scans a portion of the keyspace every second to remove expired entries proactively. TTL is the simplest and most reliable invalidation mechanism.

## Variants and Flavors

### Cache-Aside Pattern

Cache-aside is the most widely used caching pattern. The application is responsible for both checking the cache and populating it. The cache is passive: it never initiates communication with the database.

    │         WHERE id = key ──────────────────────────>│
    │<── result row ◄───────────────────────────────────│
    │                                                   │
    │──> SET key = result                               │
    │    TTL = 3600 ──────────>│                       │
    │<── OK                    │                       │
    │                                                   │
    │──> GET key                │                       │
    │<── result (hit)           │                       │
    │                           │                       │
  `}
/>

The application controls caching logic entirely. You decide which queries to cache and for how long. You can skip caching for write-heavy endpoints or sensitive data. The downside is code duplication: every read path needs the check-and-populate logic. Most web frameworks provide middleware or decorators to abstract this pattern.

**Implementation sketch** in Python-like pseudocode:

```python
def get_user_profile(user_id):
    cache_key = f"user:profile:{user_id}"
    result = cache.get(cache_key)
    if result is not None:
        return deserialize(result)

    profile = db.query("SELECT * FROM users WHERE id = %s", user_id)
    cache.set(cache_key, serialize(profile), ttl=300)
    return profile
```

Cache-aside works best when reads significantly outnumber writes, and when stale data is acceptable within the TTL window.

### Read-Through / Write-Through Pattern

In read-through caching, the cache layer handles misses automatically. The application sends all reads to the cache. When the cache misses, it fetches the data from the database, populates itself, and returns the result. Write-through caching extends this to writes: every write goes through the cache to the database synchronously.

Write-through guarantees consistency: the cache is always fresh. But it adds write latency because the write path includes both cache and database round trips. Write-through is suitable for data that is read immediately after writing, such as session state or user preferences.

**Write-behind** is a variant where the cache acknowledges the write immediately and persists to the database asynchronously. This reduces write latency from 10ms to 1ms but introduces a window of data loss if the cache fails before the database write completes.

### Write-Behind / Write-Back Pattern

Write-behind caching acknowledges writes immediately and persists to the database asynchronously. This is also called write-back caching.

**Trade-offs**: Write-behind can batch multiple writes to the same key, reducing database load. A counter incremented 1,000 times per second becomes one database update per second. But if the cache node fails before flushing, all buffered writes are lost. Write-behind also complicates read-after-write consistency: a client that writes and then immediately reads might see stale data if the read goes to a different cache node that has not received the update.

### Refresh-Ahead Pattern

Refresh-ahead caching proactively refreshes cache entries before they expire. When a cache entry is accessed and is within a configurable threshold of its TTL, the cache triggers an asynchronous refresh.

```python
def get_with_refresh_ahead(key):
    entry = cache.get(key)
    if entry is None:
        return load_from_db(key)  # cold miss

    value, ttl_remaining = entry
    if ttl_remaining < REFRESH_THRESHOLD:  # 30 seconds
        async_refresh(key)  # non-blocking
    return value
```

Refresh-ahead prevents synchronous misses for high-traffic keys. A hot key with a 5-minute TTL never causes a miss, because the cache refreshes it proactively. The cost is extra load on the database for refreshes that might be unnecessary if the key stops being accessed.

**Comparison summary**:

| Pattern | Read Strategy | Write Strategy | Consistency | Best For |
|---------|--------------|---------------|-------------|----------|
| Cache-aside | App checks cache | App writes to DB | Eventually consistent | Most web apps |
| Read-through | Cache loads on miss | App writes to DB | Eventually consistent | Apps wanting simplicity |
| Write-through | Cache loads on miss | Cache writes to DB synchronously | Strong | Session state, config |
| Write-behind | Cache loads on miss | Cache writes to DB async | Weak (loss risk) | High-throughput writes |
| Refresh-ahead | Cache pre-fetches | App writes to DB | Eventually consistent | Read-heavy hot keys |

## Mental Model Check

Test your understanding of the distributed cache pattern with these questions.

**Q1**: Your cache hit rate drops from 95 percent to 60 percent after deploying new code. What changed?

**A1**: The most likely cause is a change in the cache key format. If the new code uses a different key structure (e.g., `user:{id}:profile` instead of `profile:user:{id}`), existing cache entries are never found. Check the key generation code. Other possibilities: the TTL was reduced (entries expire faster), the cache cluster was resized (consistent hashing remapped keys), or the data access pattern shifted significantly.

**Q2**: Under what conditions would you choose a 30-second TTL instead of 5 minutes?

**A2**: Choose a short TTL when data changes frequently and freshness matters more than hit rate. Examples: stock prices, leaderboard rankings, availability status. Short TTLs increase database load but bound staleness. For reference data like country codes or product categories that change weekly, use a 24-hour TTL or no TTL with explicit invalidation.

**Q3**: Why does Redis use single-threaded event loop for command processing?

**A3**: Single-threaded execution eliminates locking overhead. All data structures are thread-safe by design because only one command executes at a time. This simplifies the codebase dramatically and avoids context-switching costs. Throughput comes from non-blocking I/O: Redis uses epoll/kqueue to handle thousands of connections while processing commands sequentially. The trade-off is that a single slow command (e.g., `KEYS *`) blocks all other operations.

**Q4**: Your cache cluster has 10 nodes, each with 8 GB RAM. You store 1 KB entries. What is the maximum number of entries you can store assuming 50 percent memory utilization for overhead?

**A4**: Each node has 8 GB, but overhead (Redis data structures, connection buffers, replication buffers) consumes approximately 50 percent. Usable memory per node: 4 GB. Entries per node: 4 GB / 1 KB = approximately 4 million. Total cluster capacity: 10 nodes * 4 million = 40 million entries. In practice, compression and serialization overhead make each entry larger than 1 KB. Measure actual memory usage with production data.

**Q5**: How do you handle a hot key in a distributed cache that receives 500,000 requests per second?

**A5**: A single hot key can overwhelm one cache node even if the cluster has 50 nodes. Mitigations: (1) Replicate the hot key to multiple nodes and distribute reads across replicas. (2) Use local in-process caching on each application server for the hot key, with a short TTL. (3) Use consistent hashing with virtual nodes to spread the key's replicas across physical nodes. (4) Apply rate limiting at the application layer to protect the cache node. (5) Consider caching the hot key data in CDN or edge cache for an additional layer.
