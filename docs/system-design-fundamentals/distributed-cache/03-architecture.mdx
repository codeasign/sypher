## Naive Implementation

The simplest distributed cache is a hash table across multiple servers with a fixed partitioning scheme. Here is how it works.

### Fixed Partitioning

Divide the keyspace into N fixed buckets. Assign server i to handle all keys where `hash(key) % N == i`. Server 1 handles keys 0, N, 2N, 3N. Server 2 handles keys 1, N+1, 2N+1.

```python
def get_cache_node(key, num_nodes):
    return hash(key) % num_nodes
```

This works perfectly until you need to add or remove a node. When N changes from 10 to 11, nearly 90 percent of keys are remapped. Every application server must update its node mapping. Every data key must be re-cached or migrated. The entire cache is effectively invalidated.

### Request Routing via Proxy

The naive approach routes requests at the application level. Every application server knows the full cluster topology. This coupling makes topology changes dangerous: all application servers must be updated or restarted when the cluster changes.

### Failure Handling

In a fixed-partition setup, when node i fails, every key hashed to node i is unreachable. There is no automatic failover. The application must detect the failure, exclude the node, and accept the temporary hit rate loss until a replacement node warms up.

These limitations motivated the development of production-grade distributed cache systems with consistent hashing, automatic failover, and proxy-based routing.

## Production-Grade Architecture

A production-grade distributed cache architecture has multiple layers, each solving a specific problem.

### Cluster Topology

The production architecture has three layers: the cache client library, a proxy layer (optional), and the cache node cluster.

**Cache Client Library**: Embedded in each application server. The client maintains a view of the cluster topology, implements consistent hashing for key-to-node mapping, handles connection pooling, and detects node failures.

**Proxy Layer (Redis Enterprise, Twemproxy, HAProxy)**: Sits between clients and cache nodes. Proxies provide a single endpoint for clients, hide topology changes, and can implement advanced routing logic. The trade-off is an extra network hop and increased latency by 0.5 to 2 milliseconds.

**Cache Node Cluster**: The actual Redis or Memcached instances. For Redis Cluster, nodes communicate via the cluster bus (port + 10000) to detect failures and coordinate failover.

### Redis Cluster Architecture

Redis Cluster uses a decentralized approach. Each node knows the full cluster topology and can route requests to the correct node. The client library handles the routing logic.

**Hash slots**: The keyspace is divided into 16,384 slots. Slot assignment: `CRC16(key) % 16384`. Each master node owns a contiguous range of slots.

**Gossip protocol**: Nodes exchange cluster state via the cluster bus using a gossip protocol. Every 100 milliseconds, each node pings a random peer. Within 5 seconds of an unresponsive node, the cluster marks it as failing. If a master fails and its replica exists, the replica promotes itself after `cluster-node-timeout` milliseconds.

**Failover**: When a replica detects its master as failing, it checks with other masters for confirmation. If a majority agrees, the replica increments its configuration epoch, becomes the new master, and starts serving the hash slots.

**Resharding**: Slots can be moved between nodes without downtime. The cluster orchestrator marks a slot as `MIGRATING` on the source node and `IMPORTING` on the destination node. Clients attempt the source first, and if the key is not found, retry the destination.

### Client-Side Routing

Clients directly connect to cache nodes without a proxy. The client maintains a view of the slot-to-node mapping.

```
Application                   Cache Nodes
   │                              │
   │── GET key                    │
   │                              │
   │ Compute slot = CRC16(key)    │
   │ % 16384 = 7281               │
   │                              │
   │ Map slot 7281 ────────────► Node B
   │                              │
   │◄── value ───────────────────│
```

When a node fails, the client detects the failure through connection errors or timeouts. It updates its slot map by fetching the latest cluster configuration. The client library handles this transparently.

### Replication

Each master node has one or more replicas. Replicas are read-only by default. They can serve read traffic to reduce load on masters (read-replica pattern).

```
Write path: app ──SET──> master ──replicate──> replica1 ──> replica2
Read path:  app ──GET──> master (or replica if read-replica enabled)
```

Replication is asynchronous by default. After a master acknowledges a write to the client, it propagates the write to replicas in the background. This means a master crash before replication completion loses acknowledged writes.

For stronger consistency, use the `WAIT` command to block until N replicas have acknowledged the write:

```python
redis.set("key", "value")
redis.execute_command("WAIT", 1, 5000)  # Wait up to 5s for 1 replica
```

## Integration Patterns

### Pattern 1: Database Query Cache

Cache the results of expensive database queries. This is the most common integration pattern.

**Flow**: Application receives a query for user profiles under department "engineering." It constructs a cache key from the query parameters: `"dept:engineering:users:paginated:50:0"`. It checks the cache. On miss, it runs the database query, serializes the result to JSON, stores it with a TTL of 300 seconds, and returns the result.

```python
def get_users_by_department(dept, limit=50, offset=0):
    cache_key = f"dept:{dept}:users:paginated:{limit}:{offset}"
    cached = cache.get(cache_key)
    if cached is not None:
        return json.loads(cached)

    result = db.query(
        "SELECT * FROM users WHERE department = %s "
        "ORDER BY name LIMIT %s OFFSET %s",
        dept, limit, offset
    )
    cache.set(cache_key, json.dumps(result), ex=300)
    return result
```

**Considerations**: Cache key must include all query parameters. Pagination offset matters. A TTL of 300 seconds means users see stale data for up to 5 minutes. For active directories, use event-driven invalidation instead of TTL.

### Pattern 2: Session Store

Store user session data in the distributed cache. Session data has high read-write ratio (read on every request, write on session mutation) and tolerates loss (user re-logs in).

**Flow**: User authenticates. Application creates a session with unique ID and stores session data (user ID, roles, permissions, expiry) in the cache. The session ID is set as a cookie on the response. On subsequent requests, the application reads the session ID from the cookie and fetches session data from the cache.

```
Request 1: POST /login
  └─> Authenticate user ──> Create session {user_id, roles, exp}
  └─> SET session:abc123 ──> {data, ttl=86400}
  └─> Set-Cookie: session_id=abc123

Request 2: GET /dashboard
  └─> Read cookie: session_id=abc123
  └─> GET session:abc123
  └─> HIT ──> {user_id, roles, exp} ──> Authorize request
```

**Session data specifications**: Each session is 200 to 500 bytes. With 10 million active sessions at 400 bytes each, total cache memory needed is 4 GB plus overhead (approximately 6 GB total). A TTL of 24 hours with automatic re-expiry on each access ensures sessions expire after inactivity.

**Failover considerations**: Session data is ephemeral. If a cache node fails, users routed to that node lose their sessions. Use replica nodes and session retry to mitigate this. For zero session loss, use a database-backed session store with the cache as an accelerator.

### Pattern 3: Rate Limiter

Use the cache's atomic counter operations to implement distributed rate limiting. Redis atomic increments with TTL are a natural fit.

**Flow**: Each API request consumes one unit of a per-user rate limit. The rate limit key is `rate_limit:{user_id}:{window_start}`. The application atomically increments the counter and checks if it exceeds the limit.

```python
def check_rate_limit(user_id, max_requests=100, window_seconds=60):
    now = int(time.time())
    window_key = f"rate_limit:{user_id}:{now // window_seconds}"

    current = cache.incr(window_key)
    if current == 1:
        cache.expire(window_key, window_seconds)

    return current <= max_requests
```

This atomic pattern rates up to 100,000 users per second on a single Redis instance. For higher throughput, shard by user ID.

**Sliding window variant**: Use a sorted set to track individual request timestamps. ZREMOVE out-of-window entries, ZCOUNT the window, and ZADD the current timestamp.

```python
def check_sliding_window(user_id, max_requests=100, window_seconds=60):
    key = f"rate_limit:{user_id}:sliding"
    now = time.time()
    window_start = now - window_seconds

    pipe = redis.pipeline()
    pipe.zremrangebyscore(key, 0, window_start)
    pipe.zcard(key)
    pipe.zadd(key, {now: now})
    pipe.expire(key, window_seconds)
    _, count, _, _ = pipe.execute()

    return count <= max_requests
```

### Pattern 4: Content Cache

Cache rendered HTML fragments, API responses, or media metadata. Content caching reduces compute as well as database load.

**Flow**: A news application renders article pages. Each article page includes the article body, author bio, related articles, and ad placements. Render time is 200 milliseconds. Cache the rendered HTML for each article, keyed by `article:{id}:html`. Serve 50,000 requests per second from cache instead of rendering each one.

```python
def get_article_page(article_id):
    cache_key = f"article:{article_id}:html"
    cached_html = cache.get(cache_key)
    if cached_html is not None:
        return cached_html  # Return as bytes, no deserialization

    article = db.query("SELECT * FROM articles WHERE id = %s", article_id)
    author = db.query("SELECT * FROM authors WHERE id = %s", article.author_id)
    html = render_template("article.html", article=article, author=author)
    cache.set(cache_key, html, ex=600)  # Cache for 10 minutes
    return html
```

**Cache invalidation**: When an article is updated, the publishing system deletes the cache key. The next request triggers a miss, re-renders the page, and repopulates the cache. Use a content management system webhook or change-data-capture pipeline for invalidation.

**Edge caching**: For global audiences, combine the distributed cache with a CDN (e.g., CloudFront, Cloudflare). The CDN caches at edge locations closer to users. The CDN's origin is the distributed cache, which in turn has the database as its origin. This creates a cache hierarchy: L1 (browser), L2 (CDN), L3 (distributed cache), L4 (database).

### Pattern 5: Object and Relation Cache

Cache composite objects assembled from multiple database queries. Social feeds, product detail pages, and dashboards all construct complex objects from many source queries.

**Flow**: A social network timeline requires 150 followee queries, 200 post fetches, and 50 engagement score lookups. Instead of caching individual queries, cache the fully assembled timeline object.

```python
def get_timeline(user_id):
    cache_key = f"timeline:{user_id}:v2"
    cached = cache.get(cache_key)
    if cached is not None:
        return deserialize_timeline(cached)

    followees = db.query("SELECT followee_id FROM follows WHERE follower_id = %s", user_id)
    posts = db.query(f"SELECT * FROM posts WHERE author_id IN ({','.join(followees)}) ORDER BY created_at DESC LIMIT 200")
    enriched = enrich_with_engagements(posts)
    serialized = serialize_timeline(enriched)

    cache.set(cache_key, serialized, ex=60)  # Cache for 60 seconds
    return enriched
```

The trade-off is granularity. If any underlying data changes, the entire timeline object must be regenerated. A post's engagement score changing triggers a full recompute for all timelines containing that post.

## Multi-Region Architecture

### Active-Passive Multi-Region

One region has a writable cache cluster. Other regions have read-only replicas that replicate from the primary region via Redis replication across regions.

```
Region US-East (Active)          Region EU-West (Passive)
┌─────────────────────┐          ┌─────────────────────┐
│   Cache Cluster     │          │   Cache Cluster     │
│ ┌──────────────────┐│          │ ┌──────────────────┐│
│ │ Master A ── ReplA││────WAN──┼─►│ Replica A (RO)   ││
│ │ Master B ── ReplB││          │ │ Replica B (RO)   ││
│ │ Master C ── ReplC││          │ │ Replica C (RO)   ││
│ └──────────────────┘│          │ └──────────────────┘│
│   Application       │          │   Application       │
│   (read + write)    │          │   (read only)       │
└─────────────────────┘          └─────────────────────┘
```

Cross-region replication latency is typically 30 to 100 milliseconds. The passive region serves stale data within that window. On failover, the passive region promotes its replicas to masters and accepts writes.

### Active-Active Multi-Region

Both regions accept writes. Conflict resolution is required. Redis uses CRDT (Conflict-Free Replicated Data Type) techniques: last-writer-wins for simple values, set semantics for collections.

```
Region US-East                    Region EU-West
┌─────────────────────┐          ┌─────────────────────┐
│   Cache Cluster     │          │   Cache Cluster     │
│ ┌──────────────────┐│─bidirec─┼►┌──────────────────┐│
│ │ Master A ── ReplA││─ctional││ Master A ── ReplA ││
│ │ Master B ── ReplB││─sync───┼─│ Master B ── ReplB ││
│ │ Master C ── ReplC││        ││ Master C ── ReplC ││
│ └──────────────────┘│        │└──────────────────┘│
└─────────────────────┘        └─────────────────────┘

Write conflict: key "promo_banner" set to "spring_sale" in US-East
                               set to "summer_sale" in EU-West simultaneously
Result (CRDT): Last writer wins using wall-clock timestamps
```

Active-active is complex and risky. Most applications use active-passive for cache because cache data is ephemeral and active-active adds operational complexity without proportional benefit.

## Capacity Planning

### Estimating Cache Size

Calculate the cache memory needed based on data size, overhead, and replication.

**Formula**: Total Memory = Data Size * (1 + Overhead Factor) * Replication Factor

| Component | Calculation | Example |
|-----------|-------------|---------|
| Average entry size | 500 bytes | 500 bytes |
| Number of entries | 10 million | 10,000,000 |
| Raw data size | entries * entry size | 5 GB |
| Overhead factor | 1.2 to 1.5 (20-50 percent) | 1.3x |
| Data with overhead | raw data * overhead | 6.5 GB |
| Replication factor | 2 (one replica) | 2 |
| **Total memory needed** | data * overhead * replication | **13 GB** |

Production rule: provision 30 percent headroom above the calculated total to handle traffic spikes and data growth. In this example, provision 17 GB.

### Determining Node Count

Node count = Total Memory / Memory per Node

Using 8 GB instances with a maxmemory of 6 GB (75 percent utilization):

```
Total memory needed: 17 GB
Memory per node: 6 GB
Minimum node count: 17 / 6 = 3 (round up)
```

With replication (3 masters, 3 replicas = 6 nodes), the memory capacity of masters must cover the data size. Replicas double memory consumption but provide failover.

For production, use at least 3 master nodes for fault tolerance. With 6 nodes total (3 masters, 3 replicas) at 6 GB each, total cluster memory is 18 GB, of which 9 GB is usable (masters only).

### Throughput Planning

Estimate operations per second based on workload.

**Workload assumptions**:
- 500,000 read requests per second
- 50,000 write requests per second
- Read: 70 percent pipeline batch size 10, 30 percent individual
- Write: 100 percent individual

**Single Redis instance capacity**:
- Pipeline reads: 800,000 ops/sec (one batch of 10 counts as 10 operations)
- Individual reads: 120,000 ops/sec
- Individual writes: 60,000 ops/sec

**Nodes needed for reads**:
- Pipeline reads: 500,000 * 0.7 = 350,000 ops/sec / 800,000 = 0.44 nodes
- Individual reads: 500,000 * 0.3 = 150,000 ops/sec / 120,000 = 1.25 nodes
- Total read nodes: 2 (rounded up)

**Nodes needed for writes**: 50,000 / 60,000 = 0.83 rounded to 1 node

**Total master nodes**: max(2, 1) = 3 (with safety margin)

The throughput and memory estimates converge on 3 master nodes. With 1 replica each, total nodes = 6.

### Rolling Upgrades

Upgrade Redis versions without downtime using rolling updates.

1. Upgrade a replica node first. Wait for it to reconnect and sync.
2. Trigger manual failover: `CLUSTER FAILOVER`. The replica becomes master.
3. Upgrade the former master (now replica).
4. Repeat for each node pair.

For major version upgrades, build a new cluster in parallel and switch traffic using DNS or proxy reconfiguration.

### Backup and Restore

Backup strategy for cache clusters:

**RDB snapshots**: Schedule RDB saves every 6 hours. Store snapshots in S3 with 30-day retention.

```bash
redis-cli -h <node> --rdb /tmp/backup.rdb
aws s3 cp /tmp/backup.rdb s3://cache-backups/$(date +%Y-%m-%d-%H).rdb
```

**Restore procedure**: Spin up a new cluster with the same node count. Load the RDB file on each node. Start the cluster and verify key counts match pre-backup.

**Pitfall**: RDB files are node-specific. A backup from a 6-node cluster must be loaded onto a 6-node cluster with the same slot assignment. If the slot map differs, keys end up on wrong nodes. Use the cluster's own export mechanism when available.

### Operational Runbooks

**Node failure**:
1. Alert: Cluster node timeout exceeded.
2. Verify: Check node process, network, and disk.
3. If unrecoverable: Mark node as dead. Remaining nodes redistribute slots if using cluster mode.
4. Spin up replacement node. Add to cluster using `CLUSTER MEET`.
5. Migrate slots from replicas or wait for auto-rebalancing.

**High eviction rate**:
1. Alert: Eviction rate exceeds 100 keys per second.
2. Investigate: Check `INFO stats` for evicted_keys. Identify whether a hot key is causing mass evictions.
3. Mitigation: Increase `maxmemory` on affected nodes. Add more shards to distribute load.
4. Root cause: Check for memory leak (no TTL), data size increase, or traffic surge.

**Cache stampede**:
1. Alert: Hit rate drops below 80 percent for 5 minutes.
2. Investigate: Check application logs for error spikes. Identify which keys are missing.
3. Mitigation: Enable request coalescing. Add jitter to TTLs. Scale database read replicas temporarily.
4. Root cause: Check deployment history for key format changes, TTL changes, or cluster topology changes.

**Split brain**:
1. Alert: Cluster state shows two masters for the same slot.
2. Verify: Check `CLUSTER NODES` for duplicate master entries.
3. Mitigation: Shut down one side. Force the healthy side to accept all slots.
4. Root cause: Network partition between cluster nodes. Strengthen `cluster-node-timeout` and network reliability.
