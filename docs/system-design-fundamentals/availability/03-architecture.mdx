# Availability Architecture

## From Naive to Production

Availability architecture is a progression. You start simple, add redundancy at the most critical points, and iterate toward a design that tolerates failures at every layer.

### Naive: Single Server

```
┌──────────────┐
│   Browser    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  [Web/App/   │
│   DB Server] │  SPOF for everything
└──────────────┘
```

A single server handles HTTP, application logic, and database. If the server fails, everything fails. If the network switch fails, everything fails. If a power outage hits the data center, everything fails. Availability: ~95-98%.

### Step 1: Redundant App Servers

```
         ┌────────────────┐
         │  [Load         │
         │   Balancer]    │
         └────┬──────┬────┘
              │      │
              ▼      ▼
         ┌──────┐┌──────┐
         │[App  ││[App  │
         │Svr 1]││Svr 2]│
         └──┬───┘└──┬───┘
            └───┬───┘
                ▼
         ┌──────────────┐
         │   ((DB))     │  Still a SPOF
         └──────────────┘
```

Adding a load balancer and redundant application servers eliminates the app tier as a SPOF. If App Svr 1 fails, the load balancer routes traffic to App Svr 2. But the database is still a single point of failure. Availability: ~99-99.5%.

### Step 2: Database Replication

```
         ┌────────────────┐
         │  [Load         │
         │   Balancer]    │
         └────┬──────┬────┘
              │      │
              ▼      ▼
         ┌──────┐┌──────┐
         │[App  ││[App  │
         │Svr 1]││Svr 2]│
         └──┬───┘└──┬───┘
            └───┬───┘
                ▼
         ┌──────────────┐
         │  ((Primary   │
         │   Database)) │
         └──────┬───────┘
                │
                │ async replication
                ▼
         ┌──────────────┐
         │  ((Replica   │
         │   Database)) │
         └──────────────┘
```

A primary database with an async replica. If the primary dies, the replica can be promoted. Reads can be served from the replica to reduce load on the primary. Availability: ~99.5-99.9%.

### Step 3: Caching Layer

```
         ┌────────────────┐
         │  [Load         │
         │   Balancer]    │
         └────┬──────┬────┘
              │      │
              ▼      ▼
         ┌──────┐┌──────┐
         │[App  ││[App  │
         │Svr 1]││Svr 2]│
         └──┬───┘└──┬───┘
            │       │
            └───┬───┘
                │
         ┌──────▼──────┐
         │  {Cache}    │  Redis/Memcached
         └──────┬──────┘
                │
         ┌──────▼──────┐
         │  ((Primary  │
         │   Database))│
         └─────────────┘
```

A cache in front of the database improves both performance and availability. If the database is slow or unreachable, the cache serves stale data. The cache itself should be replicated (Redis Sentinel or Cluster) to avoid introducing a new SPOF. Availability: ~99.9%.

### Step 4: Multi-Region

```
         ┌────── US-EAST-1 ──────┐    ┌────── US-WEST-2 ─────┐
         │  [Global LB]          │    │  [Global LB]          │
         │  ┌────────────────┐   │    │  ┌────────────────┐   │
         │  │  [Regional LB] │   │    │  │  [Regional LB] │   │
         │  └────┬──────┬────┘   │    │  └────┬──────┬────┘   │
         │       │      │        │    │       │      │        │
         │  ┌────┐┌────┐│        │    │  ┌────┐┌────┐│        │
         │  │App ││App ││        │    │  │App ││App ││        │
         │  │[1] ││[2] ││        │    │  │[1] ││[2] ││        │
         │  └─┬──┘└─┬──┘│        │    │  └─┬──┘└─┬──┘│        │
         │    │     │   │        │    │    │     │   │        │
         │    └──┬──┘   │        │    │    └──┬──┘   │        │
         │       ▼      │        │    │       ▼      │        │
         │  {Cache}     │        │    │  {Cache}     │        │
         │       │      │        │    │       │      │        │
         │       ▼      │        │    │       ▼      │        │
         │  ((Primary   │        │    │  ((Replica   │        │
         │   Database)) │        │    │   Database)) │        │
         │              │        │    │              │        │
         └──────────────┘        │    └──────────────┘        │
                │                 │          │                │
                │  async replication                    │
                └───────────────────┼──────────────────────┘
                                   │
                                   ▼
                            ┌──────────────────┐
                            │    Monitoring    │
                            │  + Orchestration │
                            └──────────────────┘
```

Multi-region is the gold standard for high availability. A region-level outage (rare but documented — AWS us-east-1 had a multi-hour outage in 2023) is survivable. The secondary region must have enough capacity to handle 100% of traffic. Availability: ~99.99%.

Key availability patterns for databases:

- **Read-write split**: Route writes to primary, reads to replicas. Implement with a custom routing data source.
- **Replica lag monitoring**: If replica lag exceeds a threshold (e.g., 1 second), route reads to primary to avoid serving stale data.
- **Automated failover**: Use a tool like Orchestrator (for MySQL) or Patroni (for PostgreSQL) to handle automatic promotion.
- **Connection pool resilience**: Drain connections gracefully. Configure minimum idle connections so the pool is always warm.

Key cache availability patterns:
- **Redis Sentinel**: Monitor the primary. If it fails, automatically promote a replica. Client uses Sentinel to discover the current primary.
- **Redis Cluster**: Data sharded across nodes. Each shard has a primary and replicas. Failover is per-shard.
- **Cache fallback**: When the cache is unavailable, fall through to the database. The database must handle the extra load — ensure it has capacity.
- **Gradual warmup**: After a cache failure and recovery, do not flood the cache with all keys simultaneously. Use rate-limited population to avoid the thundering herd problem.

Key queue availability patterns:
- **Local buffering**: When the queue is unreachable, the producer writes messages to a local file or database table. When the queue recovers, the buffered messages are forwarded.
- **Dead-letter queues**: Messages that fail processing (after N retries) are moved to a dead-letter queue. An operator reviews and re-queues or discards them. This prevents poison-pill messages from blocking the queue.
- **Visibility timeout**: In SQS, a message becomes visible to other consumers if the original consumer does not delete it within the visibility timeout. This handles consumer crashes without message loss.

- **Client-side timeout**: Always set both connection timeout (100ms) and read timeout (500ms). An external API that hangs indefinitely keeps your threads blocked.

- **Stale fallback**: Cache the last successful response. When the external API is down, serve the cached response. This provides degraded but functional behavior.

## Multi-Region Architecture: Detailed Design

A production-grade multi-region architecture must handle three critical scenarios:

1. **Regional failure**: One region becomes completely unreachable.
2. **Cross-region latency**: Normal inter-region latency (e.g., US-East to EU-West: ~80ms).
3. **Split-brain**: Network partition between regions.

Key decisions in multi-region design:

- **Active-passive vs active-active**: Active-passive is simpler but wastes infrastructure. Active-active shares load but requires conflict resolution for writes.
- **Replication mode**: Synchronous replication provides zero data loss but increases write latency across regions. Asynchronous replication is faster but risks data loss on primary failure.
- **Failover automation**: Fully automated failover risks false positives. A common pattern is semi-automated: the system detects the failure and alerts, but a human must confirm the failover. The confirmation must happen within minutes to meet the SLO.

## Capacity Planning for Availability

Capacity planning for availability means running enough redundant capacity to handle failures. The math is straightforward:

**N+1**: N instances for peak load, 1 spare. If peak load is 1,000 RPS and each instance handles 250 RPS, you need 4 + 1 = 5 instances. Cost: 25% overhead.

**N+2**: N instances for peak load, 2 spares. Same math: 4 + 2 = 6 instances. Cost: 50% overhead.

**2N**: Twice the peak capacity. 8 instances for 1,000 RPS. Cost: 100% overhead.

Redundancy cost calculation:

```
Monthly server cost per instance:      $500
N instances at peak:                   4
N+1 cost:   5 * $500 = $2,500          (25% overhead)
N+2 cost:   6 * $500 = $3,000          (50% overhead)
2N cost:    8 * $500 = $4,000          (100% overhead)

Annualized cost difference:
N+1 vs 2N:  (8 - 5) * $500 * 12 = $18,000/year

Expected downtime reduction:
N+1: tolerates 1 server failure
N+2: tolerates 2 simultaneous server failures
2N:  tolerates 4 simultaneous server failures (half the fleet)

Probability of exceeding each level (assuming 1% failure rate per server):
P(2 failures) = C(5,2) * 0.01^2 * 0.99^3 = 0.097% 
P(3 failures) = C(6,3) * 0.01^3 * 0.99^3 = 0.00194%
P(5 failures) = C(8,5) * 0.01^5 * 0.99^3 = 0.00000034%
```

The diminishing returns are clear: going from N+1 to 2N costs 60% more but reduces failure probability by five orders of magnitude. Whether this is worth it depends on the cost of downtime per minute.

## Deployment Strategies for Availability

## Database Migration Strategies for High Availability

Database schema changes are one of the riskiest operations for availability. A poorly executed migration can lock tables, block writes, and cause downtime.

### Expand-Contract (Additive) Migrations

The safest pattern for zero-downtime schema changes:

1. **Expand**: Add the new column, table, or index. The old schema remains fully functional.
2. **Migrate**: Backfill data into the new structure. This runs in the background over hours or days.
3. **Contract**: Remove the old structure in a subsequent deployment.

```
Phase 1 (Expand):
   Application v1 writes to both old_col and new_col
   ┌──────────────────┐
   │  ((DB))           │
   │  old_col VARCHAR  │
   │  new_col TEXT     │  ← new column added
   └──────────────────┘

Phase 2 (Migrate):
   Background job copies old_col → new_col
   Application reads from new_col
   ┌──────────────────┐
   │  ((DB))           │
   │  old_col VARCHAR  │
   │  new_col TEXT     │  ← populated
   └──────────────────┘

Phase 3 (Contract):
   Application no longer references old_col
   ALTER TABLE DROP COLUMN old_col
   ┌──────────────────┐
   │  ((DB))           │
   │  new_col TEXT     │  ← only column
   └──────────────────┘
```

This pattern works with blue-green deployments because both the old and new application versions can operate on the database schema during the transition.

### Online Schema Change Tools

Tools like `gh-ost` (GitHub's Online Schema Migration for MySQL) and `pgroll` (for PostgreSQL) perform changes without locking tables:

- Create a shadow table with the new schema
- Apply ongoing binlog changes to the shadow table
- Cut over atomically when the migration is complete
- Keep the original table as rollback target

GitHub uses `gh-ost` to migrate their production MySQL schema hundreds of times per year without downtime. The secret is making the operation asynchronous and continuously applying incremental changes rather than blocking for a bulk operation.

### Backward-Compatible API Changes

For API-level changes that affect availability:

- Never remove a required field in a single version. Deprecate it first, keep it optional for N versions, then remove it.
- Always support at least one previous API version alongside the current version.
- Use feature flags to gate new behavior. If the new code path has a bug, flip the flag off without a deploy.

## Graceful Shutdown Patterns

A server that is being terminated (scaling in, rolling deploy, node failure) must handle existing requests properly.

### The Graceful Shutdown Sequence

```
1. Load balancer removes instance from pool (deregistration delay)
   ── No new connections arrive ──
2. OS sends SIGTERM to application process
3. Application stops accepting new connections
4. Application waits for in-flight requests to complete (drain timeout)
5. Application flushes logs, metrics, and any buffered data
6. Process exits

Timeline:
t=0:   LB starts draining (no new connections)
t=10s: SIGTERM sent to process
t=10s: Application stops accepting, starts draining
t=20s: All in-flight requests complete (5s average * 2 safety factor)
t=20s: Process exits
t=30s: LB deregistration delay ends, instance removed from target group
```

### Key Configuration

- **Deregistration delay (ELB)**: 30-60 seconds. Gives the instance time to drain connections before being forcibly removed from the target group.
- **preStop hook (Kubernetes)**: A command or HTTP request that signals the application to drain before the SIGTERM is sent.
- **terminationGracePeriodSeconds (Kubernetes)**: 60-120 seconds. Must be longer than the expected drain time.
- **Connection draining timeout**: 30 seconds for typical apps, 60 seconds for long-poll or streaming apps.

```yaml
# Kubernetes pod spec with proper graceful shutdown
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  terminationGracePeriodSeconds: 90
  containers:
  - name: app
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 30"]
```

The `preStop` sleep gives the load balancer time to remove the pod from its routing table before the application stops accepting connections. Without it, the SIGTERM arrives while the load balancer is still sending traffic.

## Stateful Service Availability

Not all services are stateless. Stateful services — databases, caches, session stores — require special availability patterns.

### Session Affinity (Sticky Sessions)

Some applications store session state in memory. If a user's request goes to a different server, the session data is lost.

```
Without sticky sessions:         With sticky sessions:
Request 1 → Server A (session)   Request 1 → Server A (session)
Request 2 → Server B (lost!)     Request 2 → Server A (same)
Request 3 → Server A (diff)      Request 3 → Server A (same)
```

Sticky sessions introduce an availability problem: if Server A fails, the user's session is lost anyway, and traffic to Server A has nowhere to go.

**Better approach**: Externalize session state to a shared data store (Redis, DynamoDB, Memcached). Any server can serve any request because session data is fetched from the shared store.

```
All servers read/write session state to shared Redis:
┌────────────┐  ┌────────────┐  ┌────────────┐
│  Server A  │  │  Server B  │  │  Server C  │
└─────┬──────┘  └─────┬──────┘  └─────┬──────┘
      │               │               │
      └───────────────┼───────────────┘
                      │
               ┌──────▼──────┐
               │  {Redis     │
               │   Session}  │
               └─────────────┘
```

Redis itself must be highly available (Redis Sentinel or Cluster) to avoid introducing a new SPOF.

## Infrastructure as Code for Availability

Manual infrastructure management is the enemy of availability. Every manual step is a failure point during incident response. Infrastructure as Code (IaC) ensures that environments are reproducible, version-controlled, and auditable.

### Immutable Infrastructure

Treat servers as disposable. Never SSH into a running server to fix a problem — deploy a new version instead.

- **Gold images**: Pre-baked AMIs or container images with all dependencies included. No "it works on my machine" problems.
- **No configuration drift**: Every deploy replaces the entire server. No lingering configuration changes from 6 months ago.
- **Rollback is a redeploy**: If the new version has a problem, deploy the previous image.

### Auto Scaling Policies for Availability

Auto Scaling replaces failed instances automatically. Key configuration:

```json
{
  "AutoScalingGroupName": "web-app-asg",
  "MinSize": 3,
  "MaxSize": 10,
  "DesiredCapacity": 5,
  "AvailabilityZones": ["us-east-1a", "us-east-1b", "us-east-1c"],
  "HealthCheckType": "ELB",
  "HealthCheckGracePeriod": 300,
  "TerminationPolicies": ["OldestLaunchConfiguration"],
  "Metrics": [
    {"MetricName": "CPUUtilization", "Statistic": "Average", "Threshold": 70},
    {"MetricName": "RequestCountPerTarget", "Statistic": "Sum", "Threshold": 5000}
  ]
}
```

The auto scaling group distributes instances across availability zones evenly. If one AZ fails, the group replaces instances in the remaining AZs.

A common mistake: setting `MinSize` equal to `DesiredCapacity`. If one instance fails, the group cannot replace it until the desired count drops — but auto scaling does not automatically reduce desired count. Always set MinSize lower than DesiredCapacity (e.g., Min=3, Desired=5). This gives the group room to scale down during a partial failure and scale back up.
