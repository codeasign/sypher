## Internal Architecture

The LLM serving system is not a single process running on a single GPU. It is a distributed system with specialized components for request handling, batching, memory management, and inference execution. Understanding the internals of each component is what separates a candidate who can describe LLM serving from one who can build it.

## Data Structures and Algorithms

### The Paged KV Cache (vLLM)

The KV cache is the most important data structure in LLM serving. Traditional implementations allocate a contiguous block of GPU memory for each request's KV cache. This causes fragmentation — a request that needs 1,000 tokens of cache gets a block sized for the maximum (2,048 tokens), wasting 50% of GPU memory.

**What it is:** A paged KV cache divides GPU memory into fixed-size blocks (typically 16 tokens per block). Each request's KV cache is stored in a list of blocks, similar to how virtual memory pages map to physical frames. Only the blocks actually needed are allocated. When a request finishes, its blocks are freed and immediately reusable.

**Why this structure:** Paging reduces memory fragmentation from 50% to under 5%. With contiguous allocation, a 70B model with batch size 32 needs roughly 40 GB of KV cache memory. With paged allocation, the same workload fits in 22 GB — freeing up 18 GB for larger batches or bigger models.

**Time complexity:** Allocation: O(1) per block (free list pop). Lookup: O(1) per block (block table index). The block table maps logical block IDs to physical block addresses using an array index.

### The Continuous Batching Queue

The continuous batching algorithm manages which requests are in the GPU at any moment. It maintains three queues: waiting (requests not yet started), running (requests currently being processed), and done (requests that have finished generating).

**What it is:** A multi-queue scheduler that moves requests between states. When a slot opens in the running queue, the scheduler selects the highest-priority request from the waiting queue and adds it to the next batch. When a request finishes (end-of-sequence token or max tokens reached), it moves to done and its slot is freed.

**Why this structure:** A single FIFO queue would block short requests behind long ones. A priority queue with preemption would starve large requests. The three-queue structure provides flexibility — short requests can be prioritized without starving long ones, and the batch can be dynamically resized.

**Time complexity:** Schedule decision: O(log n) for priority queue operations. Batch rebuild: O(k) where k is the number of slots freed since last rebuild.

### FlashAttention

FlashAttention is an algorithm that computes attention without materializing the full attention matrix in GPU high-bandwidth memory (HBM). Instead, it uses tiling to compute attention in the faster SRAM.

**What it is:** FlashAttention divides the query, key, and value matrices into tiles that fit in GPU SRAM (typically 192 KB per streaming multiprocessor). It computes attention on each tile, accumulating the result in SRAM, and writes the final result to HBM. This avoids the O(n²) memory cost of the full attention matrix.

**Why this structure:** The standard attention implementation materializes an n × n attention matrix in GPU HBM, which requires O(n²) memory. For a 4,096-token sequence, that is 16 million floating-point values = 64 MB. For a batch of 32 sequences, that is 2 GB — larger than the model itself. FlashAttention eliminates this by 10-20x, processing each tile in fast on-chip memory.

**Time complexity:** Standard attention: O(n² × d) compute, O(n²) memory. FlashAttention: O(n² × d) compute, O(n × d) memory. The compute is the same, but FlashAttention is 2-3x faster in practice because it avoids HBM bandwidth bottlenecks.

### Tensor Parallelism (Megatron-LM Style)

Tensor parallelism splits each layer of the model across multiple GPUs. The attention heads are distributed across GPUs, and the feed-forward network is split column-wise and row-wise.

**What it is:** For a model with 80 layers and 4 GPUs, each GPU holds 20 layers' worth of parameters. During forward pass, GPU 0 sends its attention output to GPU 1, GPU 1 sends to GPU 2, and so on. All GPUs compute in parallel but must synchronize at layer boundaries via all-reduce.

**Why this structure:** Without tensor parallelism, a 70B model requires 140 GB of GPU memory (16-bit precision). The largest H100 GPU has 80 GB. You need at least 2 GPUs just to hold the model. With tensor parallelism across 4 GPUs, each GPU holds 35 GB of parameters, leaving 45 GB for the KV cache — enough to serve multiple concurrent requests.

**Time complexity:** Forward pass: O(L × (n² × d + d² × n) / P) + O(L × P × B) where P = number of GPUs, L = layers, d = hidden dimension, n = sequence length, B = all-reduce bandwidth. The first term is the compute (scales down with P). The second term is the communication overhead (scales up with P). At P=4 or P=8, the compute savings outweigh the communication cost.

## The Request/Write/Read Lifecycle

### Full Lifecycle: A Chat Completion Request

Walk through exactly what happens when a user sends a chat message to an LLM serving system.

#### Step 1: HTTP Request Arrives

The client sends `POST /v1/chat/completions` with a JSON body containing the messages array, model name, and generation parameters. The HTTP server (uvicorn/gunicorn) parses the request and creates a request object.

#### Step 2: Tokenization

The request's messages are concatenated into a single prompt string using the model's chat template. The tokenizer converts this string into token IDs. For a chat history with 5 messages, the prompt is roughly 500-1,500 tokens.

#### Step 3: Rate Limit Check

The scheduler checks the request's API key against the rate limiter. If the client has exceeded their quota (e.g., 100 requests per minute), the request is rejected with HTTP 429.

#### Step 4: Queue Admission

The request enters the waiting queue. The scheduler assigns it a priority based on the client's tier (paying customers get priority over free tier). The request waits until a slot in the running batch opens.

#### Step 5: Batch Assignment

A slot opens (another request finished). The scheduler pops the highest-priority request from the waiting queue. It allocates KV cache blocks for the prefill phase. The request joins the next iteration of the running batch.

#### Step 6: Prefill on GPU

The GPU runs the prefill phase for the new request alongside the ongoing decode phase for other requests. FlashAttention computes attention over the full prompt. The first output token and the initial KV cache are produced.

#### Step 7: First Token Streaming

The first token is streamed to the client via SSE. The client's UI displays the first word. The perceived latency so far is the sum of queue wait + prefill time + first decode step — typically 200-1,000ms.

#### Step 8: Iterative Decode

The model enters the decode loop. For each iteration:
1. Read the latest token embedding
2. Compute attention using the KV cache (no recomputation)
3. Run the feed-forward network
4. Sample the next token
5. Append the new token's KV to the cache
6. Stream the token to the client
7. Check for end-of-sequence or token limit

Each iteration takes 10-50ms. For a 500-token response, the total decode time is 5-25 seconds.

#### Step 9: Memory Pressure Handling

If GPU memory is full, the KV cache manager offloads the least-recently-used blocks to CPU memory. The scheduler pauses allocation for low-priority requests. When memory pressure eases, blocks are swapped back to GPU.

#### Step 10: Response Completion

The model generates the end-of-sequence token, or the token limit is reached. The final token is streamed. The scheduler moves the request to the done queue. The KV cache blocks are freed. Metrics are recorded: time to first token (TTFT), tokens per output token (TPOT), total generation time, tokens generated.

## Consistency and State

### Scenario 1: GPU Crashes Mid-Decode

**What happens:** The GPU encounters a hardware error (ECC memory error, power fault) during the decode phase. The CUDA kernel returns an error. All in-flight requests on that GPU are lost.

**Resolution:** The scheduler detects the GPU failure via a heartbeat timeout (default 5 seconds). It marks the GPU as unhealthy and removes it from the active pool. In-flight requests are re-queued to the waiting queue (up to max-retry limit, typically 3). The GPU instance is replaced by the orchestrator (Kubernetes auto-recovery). The model weights are reloaded on the new GPU.

**Impact:** In-flight requests experience a 5-30 second delay (detection + re-queue + re-prefill). Completed requests are unaffected. The system continues serving at reduced capacity until the GPU is replaced.

### Scenario 2: Out-of-Memory

**What happens:** The KV cache grows beyond available GPU memory because requests generate too many tokens or the batch is too large.

**Resolution:** The memory monitor triggers pre-emptive action. First, it reduces batch size by pausing admission of new requests. Second, it offloads KV cache blocks for the longest-running requests to CPU memory (swap). Third, if memory is critically low, the scheduler sends backpressure signals to the rate limiter to reject new requests.

**Impact:** Long-running requests experience higher latency due to KV cache swapping (CPU memory is 10-100x slower than GPU memory). New requests are queued. The system never crashes — it degrades gracefully.

### Scenario 3: Two Requests Arrive Simultaneously

**What happens:** Two requests arrive at the exact same microsecond. Both enter the waiting queue at the same time. The scheduler must decide which to process first.

**Resolution:** The scheduler uses a deterministic tiebreaker: request ID (UUID) lexicographic ordering. The first request in sort order gets the next available slot. The second request waits for the following slot. This ensures no starvation — both requests are processed within one batch cycle.

### Scenario 4: Model Weights Update During Serving

**What happens:** A new version of the model is deployed while requests are being served. The old model weights are still in GPU memory. The new weights are being loaded.

**Resolution:** The deployment uses a blue-green strategy. The old model (blue) continues serving until the new model (green) is fully loaded and passes health checks. When green is ready, new requests are routed to green. In-flight requests on blue are allowed to complete (within a timeout, typically 5 minutes). Blue is decommissioned when all in-flight requests finish or the timeout expires.

## Configuration and Tuning

### Batch Configuration

**max_num_seqs** — Maximum number of sequences processed simultaneously. Default: 256 for vLLM. Increasing this improves throughput but increases KV cache memory usage and latency. Set this based on your latency SLA: for under 500ms TTFT, keep it under 64. For maximum throughput, set it as high as GPU memory allows.

**max_num_batched_tokens** — Maximum tokens in a single batch iteration. Default: 4,096 for vLLM. Controls how many tokens are processed in one forward pass. Higher values improve GPU utilization but increase latency. The sweet spot is typically 2,048-8,192 depending on model size.

### KV Cache Configuration

**block_size** — Size of each KV cache block in tokens. Default: 16 for vLLM. Smaller blocks (8) reduce fragmentation but increase block table overhead. Larger blocks (32) reduce overhead but increase fragmentation. The 16-token default is optimal for most workloads.

**max_model_len** — Maximum sequence length the model can handle. Default: 4,096 for most models. Longer sequences (32,768 for GPT-4 class models) require more KV cache memory per sequence but enable longer context windows. The KV cache scales linearly with this value.

**gpu_memory_utilization** — Fraction of GPU memory allocated to the KV cache. Default: 0.90 for vLLM. Higher values (0.95) increase batch capacity but risk OOM. Lower values (0.80) provide safety margin but reduce throughput. The remaining memory is used for model weights, activations, and CUDA overhead.

### Scheduling Configuration

**scheduling_policy** — How the scheduler selects requests. Default: "fcfs" (first-come-first-served). Alternative: "priority" — requests are ordered by client priority tier. Priority scheduling can starve low-priority requests during high load. Use fair scheduling to prevent starvation in multi-tenant deployments.

**max_waiting_requests** — Maximum number of requests in the waiting queue. Default: 10,000 for vLLM. When exceeded, new requests are rejected with HTTP 503. Set this based on your latency SLA: at 100 req/s with 10k queue capacity, requests wait up to 100 seconds.

### Sampling Configuration

**temperature** — Controls output randomness. Default: 0.7 for chat models. Lower values (0.1) make output deterministic. Higher values (1.0) increase creativity. Temperature does not affect serving performance directly, but lower temperatures can increase cache hit rates in speculative decoding.

**top_p** — Nucleus sampling threshold. Default: 0.9. The model only considers tokens whose cumulative probability exceeds top_p. Lower values reduce the vocabulary search space slightly, improving GPU utilization by 1-2%.

**max_tokens** — Maximum tokens to generate per request. Default: 1,024 for most APIs. This is the most important parameter for capacity planning. A request that generates 4,096 tokens consumes 4x the KV cache and 4x the decode time of a 1,024-token request. Set per-client limits to prevent a single request from dominating GPU capacity.
