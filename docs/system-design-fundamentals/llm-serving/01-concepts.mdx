## The Problem Story

You are building a chatbot for a customer support platform. Your team spent six months fine-tuning a 70-billion-parameter language model. It can answer questions, summarize tickets, and escalate issues. The model is the best you have ever built. Your CEO loves the demo.

Then you put it in production.

The first user sends a message. Your server loads the model into GPU memory — 140 GB of model weights. That takes 45 seconds. The user times out and refreshes. The model loads again. Another 45 seconds. The user leaves.

You optimize: keep the model loaded in memory. Now the first request takes 3 seconds — still too slow. You add a second GPU. Now you can handle two requests at a time. But your users are sending 50 requests per second. The queue grows. Requests sit in memory for 30 seconds before the model even sees them. GPU memory is overflowing. The inference server crashes. Restart. Repeat.

You check your cloud bill. Four A100 GPUs running 24/7 cost $160,000 per month. Your startup has six months of runway. At this rate, you have three. Your CTO asks: "Why can't we just call the OpenAI API?" But your model is custom — fine-tuned on your proprietary data. You cannot use a third-party API. You need to serve this model yourself, at scale, under budget.

This is the cold reality of LLM serving. Building the model is hard. Serving it at scale is a different engineering discipline entirely. Every layer of the stack — hardware, memory management, request scheduling, batching, caching, networking — must be optimized for the unique characteristics of large language models: their massive memory footprint, their unpredictable latency, their quadratic attention costs, and their voracious appetite for GPU compute.

## The Insight

An LLM serving system is a **factory assembly line for text generation**. The model is the factory machinery. Raw tokens come in one end. Processed tokens come out the other. The factory's job is to keep the machinery running at full capacity, never idle, never overloaded, never wasting a cycle.

Think of a restaurant kitchen. The chef (the GPU) can cook one dish at a time, but it takes 30 seconds per dish. If 50 customers order at once, the kitchen does not start cooking each order from scratch — it preps ingredients in bulk, batches similar orders together, and keeps the stove running at full capacity. The waiter (the serving system) queues orders, groups them by dish type, and hands the chef a batch of 8 orders at once. The chef cooks them all together in 60 seconds instead of 8 separate 30-second cooks.

This is exactly how LLM serving works. The GPU is the chef. The model weights are the recipe book. The tokens are the ingredients. The serving system batches multiple requests together so the GPU processes them in parallel, maximizing throughput and minimizing idle time. The difference is that in an LLM, each "dish" takes a different amount of time depending on how many words it generates — and the chef cannot know how long each dish will take until it is finished.

## Core Vocabulary

**Inference** — The process of running a trained model on new input to produce output. For LLMs, this means feeding in a prompt and getting back generated text. Each inference run consumes GPU compute and memory. Inference is the fundamental operation of the serving system.

**Latency** — The time from when a user sends a request to when they see the first token of the response. For chat applications, latency under 500ms is expected. For document processing, latencies of 5-30 seconds are acceptable. LLM latency is dominated by the time to generate each token sequentially.

**Throughput** — The number of tokens the system can generate per second, across all requests. High throughput means the GPU is fully utilized. Low throughput means the GPU is idle, waiting for data or synchronization. The tension between latency and throughput is the central challenge of LLM serving.

**KV cache** — The key-value cache that stores intermediate attention computations from previously generated tokens. Without it, every new token would require recomputing attention over the entire prompt. The KV cache grows linearly with sequence length and batch size, and it is the primary memory bottleneck in LLM serving.

**Prefill phase** — The first phase of LLM inference, where the model processes the input prompt and computes the first output token. This phase is compute-bound — it requires a large matrix multiplication over the entire prompt. The prefill phase typically takes 10-100ms for a 1,000-token prompt.

**Decode phase** — The second phase, where the model generates one token at a time, using the KV cache from the prefill phase. Each token takes 5-50ms depending on model size and hardware. The decode phase is memory-bound — the bottleneck is reading the KV cache and model weights from GPU memory.

**Continuous batching** — A technique where the serving system dynamically adds new requests to the batch as existing requests finish, instead of waiting for all requests in a batch to complete. This keeps the GPU busy and prevents idle time. Continuous batching is the key innovation that made high-throughput LLM serving practical.

**Prompt** — The input text provided by the user. Prompts can range from a few words to thousands of tokens. Longer prompts increase the prefill time and KV cache memory. Prompt engineering is the art of crafting inputs that produce the desired output.

**Temperature** — A parameter that controls the randomness of the model's output. Lower temperature (0.1) makes output deterministic and focused. Higher temperature (1.0) makes output creative and diverse. Temperature affects how the model selects tokens during generation, but does not affect serving performance.

**Streaming** — The technique of sending generated tokens to the client one at a time as they are produced, instead of waiting for the full response. Streaming improves perceived latency dramatically — the user sees the first word in 200ms instead of waiting 5 seconds for the complete response. Most LLM serving systems support Server-Sent Events (SSE) for streaming.

**Model parallelism** — Distributing a single model across multiple GPUs because it does not fit on one. Tensor parallelism splits each layer across GPUs (for models up to 175B parameters). Pipeline parallelism splits layers across GPUs (for very deep models). Both are required for models larger than 13B parameters on consumer GPUs or 70B parameters on professional GPUs.

**Speculative decoding** — A technique where a small, fast draft model generates candidate tokens, and the large model verifies them in parallel. This can speed up generation by 2-3x because the large model processes multiple tokens at once instead of one at a time. The speedup comes from better GPU utilization during the decode phase.

## How It Actually Works — Step by Step

### Step 1: Request Arrival

A client sends an HTTP POST request to the serving endpoint. The request includes the prompt text, generation parameters (temperature, max tokens, top-p), and an API key. The request is typically 1-10 KB of JSON.

**Why this step exists:** The request is the contract between the client and the serving system. Without a structured request format, the serving system cannot validate inputs, enforce policies, or schedule efficiently.

### Step 2: Authentication and Rate Limiting

The gateway validates the API key against a key store (Redis or database). It checks the rate limit — how many requests per minute this client is allowed. If the client exceeds the limit, the gateway returns HTTP 429 with a Retry-After header.

**Why this step exists:** LLM inference is expensive. A single request can cost $0.01 to $0.10 in GPU compute. Without rate limiting, one client's traffic spike can consume all GPU capacity, starving other clients and running up the cloud bill.

### Step 3: Tokenization

The prompt text is converted into a sequence of token IDs using the model's tokenizer. Tokenization splits text into subword units: "Hello world" becomes ["Hel", "lo", " world"] → [182, 562, 127]. A 100-word prompt typically produces 130-150 tokens.

**Why this step exists:** LLMs do not process text directly. They process token IDs — integers that map to entries in the model's embedding table. Tokenization is the bridge between human language and model representations.

### Step 4: Prefill Phase

The tokenized input is passed to the GPU. The model processes all tokens in parallel, computing attention across the entire sequence. This produces the first output token and initializes the KV cache. The prefill phase is compute-bound — it uses 100% of GPU compute units.

**Why this step exists:** The model must understand the full context of the prompt before it can generate a response. The prefill phase computes the attention scores for every pair of tokens in the prompt, building the initial state from which generation proceeds.

### Step 5: KV Cache Allocation

The KV cache is allocated in GPU memory. For a 70B parameter model with a 2,048-token prompt and batch size 4, the KV cache needs roughly 2 GB per layer × 80 layers = 160 GB total. The KV cache manager tracks which cache slots are in use and reuses slots from completed requests.

**Why this step exists:** Without the KV cache, each new token would require recomputing attention over the entire prompt. For a 1,000-token prompt generating 500 tokens, the KV cache reduces compute by 500x. The cache is the difference between a 3-second response and a 25-minute wait.

### Step 6: Decode Phase — First Token

The model generates the first output token. It reads the KV cache, computes attention over all cached keys and values, and produces a probability distribution over the vocabulary. The highest-probability token (or a sampled token, depending on temperature) is selected. This takes 10-50ms depending on model size.

**Why this step exists:** The first token is the start of the response. The user sees the first word of the response after the prefill time plus the first decode step. For a 1,000-token prompt, this is typically 100-500ms total.

### Step 7: Streaming Response

The first token is sent to the client via Server-Sent Events (SSE) or WebSocket. The client receives the token and displays it immediately. The user sees the response appearing word by word, which feels much faster than waiting for the complete response.

**Why this step exists:** Perceived latency matters more than raw latency. A user who sees the first word in 200ms feels like the system is fast, even if the full response takes 10 seconds. Without streaming, the user stares at a spinner for 10 seconds, which feels like an eternity.

### Step 8: Continuous Batching

While the model generates tokens for one request, the scheduler adds new requests to the batch. As requests finish, they are removed from the batch. The GPU never sits idle — it always has the maximum number of requests that fit in memory.

**Why this step exists:** Without continuous batching, the GPU is idle while waiting for the slowest request in a batch to finish. With continuous batching, new requests fill the slots vacated by finished ones. This increases GPU utilization from 30% to 90%+.

### Step 9: Memory Management and KV Cache Eviction

As the KV cache grows, the system must manage memory. Finished requests have their KV cache freed. Long-running requests may have their KV cache offloaded to CPU memory if GPU memory is full. The KV cache manager uses a paging algorithm (similar to virtual memory) to handle oversubscription.

**Why this step exists:** GPU memory is the most expensive and scarce resource in LLM serving. Without efficient memory management, the system runs out of GPU memory, crashes, or must reject requests. Paged attention (vLLM's innovation) reduced memory fragmentation by 95% compared to naive caching.

### Step 10: Response Completion and Cleanup

The model generates the end-of-sequence token or reaches the maximum token limit. The final token is streamed to the client. The KV cache slot is freed. The request is logged with timing data (prefill time, decode time, tokens generated, throughput). The metrics are sent to the monitoring system.

**Why this step exists:** Without cleanup, GPU memory fills up with stale KV caches, and the system grinds to a halt. Without logging, you cannot measure latency, throughput, or cost per request — making it impossible to optimize the system.

## Mental Model Check

**Q1:** Why does the KV cache grow as the model generates more tokens?

**A1:** The KV cache stores the attention keys and values for every token the model has seen so far. Each new token requires attention over all previous tokens, so the cache must grow to include the new token's key-value pairs. Without the cache, the model would recompute attention over the entire history for every token, which is 500x slower for a 500-token response.

**Q2:** What is the difference between the prefill phase and the decode phase?

**A2:** The prefill phase processes the input prompt in parallel, computing attention over all prompt tokens at once. The decode phase generates tokens one at a time, using the KV cache from the prefill phase. The prefill phase is compute-bound (large matrix multiply). The decode phase is memory-bound (reading the KV cache).

**Q3:** Why does continuous batching improve GPU utilization?

**A3:** Without continuous batching, the GPU waits for all requests in a batch to finish before starting a new batch. If one request generates 1,000 tokens and another generates 10, the GPU is idle for 990 tokens worth of time. Continuous batching adds new requests to the batch as old ones finish, keeping the GPU busy at all times.

**Q4:** What is the trade-off between batch size and latency?

**A4:** Larger batches increase throughput (tokens per second) because the GPU can process multiple sequences in parallel. But larger batches also increase latency because each request spends more time in the queue and the GPU takes slightly longer to process each token when handling more sequences. The optimal batch size balances throughput requirements against latency SLAs.

**Q5:** Why does model parallelism matter for LLM serving?

**A5:** A 70B parameter model in 16-bit precision requires 140 GB of GPU memory. The largest consumer GPU (NVIDIA RTX 4090) has 24 GB. The largest professional GPU (NVIDIA H100) has 80 GB. A single GPU cannot hold the full model. Model parallelism splits the model across multiple GPUs, enabling serving of models that would otherwise be impossible to deploy.
