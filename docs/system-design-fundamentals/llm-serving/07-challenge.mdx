## Tier 1 — Foundation

### Challenge 1: Explain the LLM Serving Pipeline

**Prompt:** You are interviewing a junior engineer who just joined your team. Explain the LLM serving pipeline in 5 minutes. Include the purpose of each stage, the data flow between stages, and what breaks if a stage is missing.

**Time limit:** 5 minutes

**Evaluation criteria:**
- Names all major stages: tokenization, prefill, KV cache allocation, decode, streaming
- Explains the purpose of each stage in one sentence
- Describes the data flow: text → tokens → embeddings → attention → logits → tokens → text
- Identifies what breaks without each stage
- Provides a concrete example (a chat request flowing through the pipeline)

**Common mistakes:**
- Forgetting the KV cache (most common omission — without it, the model recomputes attention for every token)
- Confusing prefill and decode (prefill processes the prompt in parallel, decode generates tokens one at a time)
- Not explaining why streaming matters (perceived latency, not raw latency)
- Not addressing GPU memory constraints (models must fit in GPU memory)

**Model answer:**

"The LLM serving pipeline has 6 stages. First, tokenization converts the prompt text into token IDs. Without this, the model cannot process text — it works with integers, not characters.

Second, the prefill phase processes all prompt tokens in parallel through the model's attention layers. This computes the initial state and generates the first output token. Without this, the model has no context for generation.

Third, the KV cache stores the attention keys and values from the prefill phase. Without the cache, each new token would recompute attention over the entire prompt history, which is 500x slower for a 500-token response.

Fourth, the decode phase generates tokens one at a time, using the KV cache for efficiency. Each token takes 10-50ms. Without this, you cannot generate a response — the model only produces one token per forward pass.

Fifth, streaming sends each token to the client as it is generated. Without streaming, the user waits for the complete response, which feels 5x slower than seeing tokens appear.

Sixth, the end-of-sequence check stops generation when the model outputs a special stop token or reaches the maximum token limit. Without this, the model would generate tokens forever, consuming GPU memory until it crashes."

### Challenge 2: KV Cache Sizing

**Prompt:** You are deploying a 70B parameter Llama 3 model with Grouped-Query Attention (8 KV heads, 64 query heads). The model has 80 layers, a hidden dimension of 8,192, and supports up to 8,192 tokens. You use FP16 precision for the KV cache. Calculate the KV cache memory per sequence and determine how many concurrent sequences you can fit on a single H100 GPU (80 GB) along with the model weights.

**Time limit:** 5 minutes

**Evaluation criteria:**
- Calculates KV cache memory correctly: 2 (bytes per FP16) × 8 (KV heads) × 128 (head dimension) × 80 (layers) × 8,192 (max tokens) = 1.34 GB per sequence
- Accounts for model weights: 70B × 2 bytes = 140 GB → needs at least 2 GPUs for tensor parallelism
- With 4 GPUs (tensor parallel): each GPU holds 35 GB of weights, leaving 45 GB for KV cache
- Calculates concurrent sequences: 45 GB / 1.34 GB = 33 sequences per GPU (practical limit lower due to overhead)

**Common mistakes:**
- Forgetting to divide by the number of GPUs (tensor parallelism splits both weights and KV cache)
- Using the wrong precision assumption (8-bit KV cache would be half the memory)
- Not accounting for CUDA context and activation memory (4 GB + 8 GB overhead)

**Model answer:**

"A 70B model in FP16 requires 70B × 2 bytes = 140 GB for weights. An H100 has 80 GB, so at least 2 GPUs are needed. With 4 GPUs in tensor parallelism, each GPU holds 140 GB / 4 = 35 GB of weights.

The KV cache per sequence: each of the 80 layers stores 8 KV heads × 128 dimensions = 1,024 values per token in FP16 = 2,048 bytes = 2 KB per token. For 8,192 max tokens: 2 KB × 8,192 = 16 MB per layer. Across 80 layers: 16 MB × 80 = 1.28 GB. With tensor parallelism across 4 GPUs: 1.28 GB / 4 = 320 MB per GPU per sequence.

Available GPU memory after weights: 80 GB - 35 GB (weights) - 4 GB (CUDA) - 8 GB (activations) = 33 GB. Concurrent sequences per GPU: 33 GB / 320 MB = 103. With overhead and fragmentation, the practical limit is roughly 64 sequences per GPU for stable serving."

### Challenge 3: Latency vs Throughput Trade-off

**Prompt:** Your LLM serving system has an average response time of 8 seconds and handles 100 requests per second. The product team wants to reduce response time to under 2 seconds. Your GPU utilization is 85% with batch size 64. How do you achieve the latency target? What is the impact on throughput and cost?

**Time limit:** 5 minutes

**Evaluation criteria:**
- Identifies that latency and throughput are inversely related
- Explains that reducing batch size reduces latency but also reduces throughput
- Calculates the required batch size change: 2s / 8s = 25% of current, so batch size should be roughly 64 × 0.25 = 16
- Calculates throughput impact: batch 64 → 100 req/s, batch 16 → 25 req/s per GPU → needs 4x more GPUs
- Suggests alternatives: model quantization, speculative decoding, response caching

**Common mistakes:**
- Suggesting adding GPUs without considering cost (4x GPUs = 4x cost)
- Not considering that response time includes both queue wait and decode time
- Ignoring the impact on time-to-first-token vs total response time

**Model answer:**

"Latency and throughput are in direct tension. Current response time is 8 seconds at batch size 64 and 85% utilization. To reduce to 2 seconds, I need to reduce the batch size by roughly 4x — from 64 to 16. Smaller batches mean less queue wait and shorter decode per iteration.

At batch 16, throughput per GPU drops from 100 req/s to roughly 25 req/s. To maintain 100 req/s, I need 4x more GPUs: from 1 GPU to 4 GPUs. This increases cost from $30,000/month to $120,000/month.

Before buying more GPUs, I explore alternatives. First, speculative decoding — use a small draft model to generate candidate tokens, verified by the large model in parallel. This can reduce latency by 40-50% without changing batch size. Second, response caching — cache frequently asked questions to skip inference entirely. A 20% cache hit rate reduces load by 20%. Third, model quantization — INT4 quantization reduces memory and can improve latency by 20% through better memory bandwidth utilization.

If these optimizations achieve 50% latency reduction (from 8s to 4s), I need only 2x more GPUs instead of 4x."

## Tier 2 — Design

### Challenge 1: Chat Application Serving

**Prompt:** Design an LLM serving system for a customer support chatbot. The chatbot serves 1 million users, with each user sending 5 messages per day. The messages are short (50-200 tokens prompt, 100-300 tokens response). The latency requirement is under 1 second for time to first token and under 5 seconds total. The system must be cost-effective — the budget is $200,000 per month for GPU infrastructure.

**Time limit:** 15 minutes

**Scale estimates:**
- 1M users × 5 messages/day = 5 million requests/day
- Peak: 5x average = 5M / 24 / 3600 × 5 = 289 requests/second
- Average tokens per request: 150 prompt, 200 response
- Peak tokens per second: 289 × 350 = 101,150
- Budget: $200,000/month for GPUs

**Key decisions:**
- Use a 7B model (sufficient for customer support, fits on one H100 GPU)
- INT4 quantization (reduces model to 3.5 GB, fits on any modern GPU)
- Single GPU per instance, no model parallelism needed
- Response caching for frequently asked questions (30% hit rate expected)
- Continuous batching with a max batch size of 32 (latency priority)

**What a strong answer covers:**
- Model selection: 7B is sufficient for customer support; the 70B model is overkill and exceeds the budget
- GPU estimate: 7B model at 2,000 tok/s per GPU → 101,150 / 2,000 = 51 GPUs. With 30% headroom: 66 GPUs = 9 instances (8× GPU). Cost: 9 × $30,000 = $270,000 → over budget
- Optimization: INT4 quantization reduces memory, enabling larger batch and more throughput per GPU. 7B INT4: roughly 3,000 tok/s per GPU → 101,150 / 3,000 = 34 GPUs. With headroom: 44 GPUs = 6 instances. Cost: 6 × $30,000 = $180,000 → within budget
- Cache strategy: Redis response cache, 30% hit rate reduces effective load to 70%. GPUs needed: 44 × 0.7 = 31 GPUs = 4 instances. Cost: 4 × $30,000 = $120,000
- Final cost: $120,000/month for GPU + $5,000 for Redis + $5,000 for load balancer + $10,000 for networking = $140,000/month

**What a weak answer misses:**
- No response caching (without it, 30% more GPUs needed — blows the budget)
- No quantization (FP16 7B costs more than INT4 7B for the same latency)
- No batching strategy (continuous batching is essential for GPU utilization)
- No cost calculation (the answer must demonstrate that the design fits within $200K/month)

**Model answer:**

"With 1 million users and 5 messages per day, I need to handle 5 million requests daily. At peak, roughly 289 requests per second. Each request has 150 prompt tokens and 200 response tokens — relatively short.

I use a 7B model in INT4 quantization. The quantized model is 3.5 GB, fitting easily on any GPU with room for KV cache (under 100 MB per sequence for 500 total tokens). The model generates roughly 3,000 tokens per second on an H100 in INT4. At 289 requests per second × 200 response tokens = 57,800 decode tokens per second. I need 57,800 / 3,000 = 20 GPUs for decode, plus 5 GPUs for prefill overhead. With 30% headroom: 33 GPUs.

But I also cache responses. Customer support chatbots handle many repeat questions. With a Redis cache and 30% hit rate, only 70% of requests reach the GPU. Effective load: 33 × 0.7 = 23 GPUs = 3 instances (8× H100). Monthly cost: 3 × $30,000 = $90,000 for GPUs, plus $15,000 for supporting infrastructure = $105,000 total — well under the $200K budget.

For latency, I use continuous batching with max batch size 16 to keep TTFT under 300ms. The short response length (200 tokens) keeps total response time under 3 seconds. Streaming ensures tokens appear immediately."

### Challenge 2: Multi-Model Serving Platform

**Prompt:** Design a serving platform for a company that needs to serve 10 different fine-tuned models. The models range from 1B to 70B parameters. Different product teams own different models. Usage is unpredictable — one model might receive 1,000 requests per second while another receives 1 request per second. The platform must minimize GPU costs while serving all models with acceptable latency (under 3 seconds TTFT for models under 13B, under 10 seconds for 70B).

**Time limit:** 15 minutes

**Scale estimates:**
- 10 models: 2 × 1B, 3 × 7B, 3 × 13B, 2 × 70B
- Total peak load: 2,000 requests/second across all models
- Most requests (60%) go to the 7B models
- GPU memory per model: 1B (2 GB FP16), 7B (14 GB), 13B (26 GB), 70B (140 GB)

**Key decisions:**
- Use a shared GPU pool with hot/cold model swapping
- Keep 7B and 13B models hot (always loaded, 30-second idle timeout for cold)
- Swap 1B models aggressively (1-minute idle timeout — they load fast)
- For 70B models, keep dedicated instances (model parallelism makes swapping expensive)
- Model router classifies and routes requests to the right model instance

**What a strong answer covers:**
- Hot/cold pool with different swap policies by model size
- Shared GPU instances for small models (multiple models per GPU using model isolation)
- Dedicated instances for large models (70B needs 4 GPUs with tensor parallelism)
- Model router with request queuing per model
- Auto-scaling based on queue depth per model
- Cost optimization: 3 hot instances (8× GPU) for 7B/13B, 2 dedicated instances for 70B

**What a weak answer misses:**
- No model isolation strategy (models on the same GPU must be isolated — CUDA context switching is expensive)
- No appropriate swap policy (swapping a 70B model takes 45 seconds — too expensive to do frequently)
- No traffic routing priority (a 1B model should not block a 70B request)
- No cost optimization for low-usage models

**Model answer:**

"I design a multi-model serving platform with a hot/cold pool strategy. The key insight is that model swap cost scales with model size — a 1B model loads in 5 seconds, but a 70B model takes 45 seconds.

Small models (1B, 7B): 5 GPU instances, each running 2-3 models concurrently. Models are swapped on demand with a 60-second idle timeout. The 1B models load in 5 seconds — users accept a 5-second cold start for rarely-used models.

Medium models (13B): 3 dedicated GPU instances (one per model). Each 13B model stays loaded in memory. The low usage (under 50 req/s per model) means GPU utilization is low, but the swap cost is too high to swap aggressively.

Large models (70B): 2 dedicated instances with 4 GPUs each (tensor parallelism). These models are always hot. The cost is $60,000/month per 70B model = $120,000/month total. At 200 req/s total for both models, that is $600 per req/s — expensive but necessary.

The model router sits in front of all instances. It routes requests to the correct instance based on model name. If the requested model is not loaded, the router triggers a load and returns a 202 (Accepted) with a callback URL for async completion.

Total GPU instances: 5 (small) + 3 (medium) + 2 × 4 (large) = 16 instances. Monthly cost: $400,000/month for GPUs. At 2,000 req/s total, that is $200 per req/s — efficient for a multi-model platform."

### Challenge 3: Cost-Optimized Serving

**Prompt:** Your company's LLM serving costs have doubled after launching a new product feature. The monthly GPU bill is $2.5 million. The CTO wants to reduce costs by 35% while maintaining a p99 latency under 3 seconds for interactive use cases. The system serves 3 models: a 7B model (80% of traffic), a 70B model (15% of traffic), and a 405B model (5% of traffic). Design the cost optimization strategy.

**Time limit:** 15 minutes

**Cost breakdown (assumptions):**
- 7B model: 40 H100 instances (8×GPU), $1.2M/month — 80% of traffic at low cost per token
- 70B model: 20 instances (4×GPU each), $600K/month — 15% of traffic at medium cost
- 405B model: 10 instances (8×GPU each), $700K/month — 5% of traffic at high cost per token
- Total: $2.5M/month

**Key decisions:**
- Quantize the 7B model to INT4 (2x throughput per GPU, reduces instances from 40 to 20)
- Implement model routing for the 70B model (route simple queries to 7B, saving 30% of 70B traffic)
- Use speculative decoding for the 405B model (2x throughput, reduces instances from 10 to 5)
- Implement response caching (20% cache hit rate across all models)
- Use spot instances for 30% of the 7B fleet (30% discount)

**What a strong answer covers:**
- Quantization impact: INT4 7B has negligible accuracy loss for most use cases
- Model routing: a small classifier determines query complexity, routes simple queries to 7B
- Caching: exact match + semantic similarity cache with 20% hit rate
- Spot instances: balance cost vs availability (spot termination handling)
- Total savings: 50% from quantization + 15% from routing + 10% from caching + 10% from spot = 50% reduction

**What a weak answer misses:**
- Accuracy concerns with quantization (needs evaluation before production deployment)
- Spot instance termination handling (models must reload on a new instance when spot is reclaimed)
- Cache invalidation (cached responses for dynamic queries are stale)
- Monitoring quality post-optimization (quantization and routing can degrade output)

**Model answer:**

"The current cost is $2.5M/month. I need to reduce by 35% to $1.625M/month. I attack the largest cost first — the 7B fleet ($1.2M — 48% of total).

Step 1: Quantize the 7B model to INT4. This doubles throughput per GPU from 3,000 tok/s to 6,000 tok/s. The 7B fleet drops from 40 instances to 20. Savings: $600K/month. Accuracy impact: INT4 quantization preserves over 99% of FP16 quality for 7B models — acceptable for most use cases.

Step 2: Implement model routing for the 70B model. A small classifier (under 1B, runs on CPU) determines whether each request to the 70B model is complex enough to need 70B. 30% of 70B traffic goes to 7B instead. 70B instances drop from 20 to 14. Savings: $180K/month.

Step 3: Speculative decoding for the 405B model. A 7B draft model generates candidate tokens, verified by the 405B model in parallel. Throughput doubles. 405B instances drop from 10 to 5. Savings: $350K/month.

Step 4: Response caching across all models. A Redis cache with exact match and semantic similarity (embedding-based) achieves 20% cache hit rate. All fleets scale down by 20%. Savings: $500K/month.

Step 5: Spot instances for 30% of the 7B fleet. 6 of the 20 7B instances run on spot GPUs (30% discount). These handle non-critical traffic. If spot instances are reclaimed, traffic shifts to on-demand instances. Savings: $54K/month.

Total savings: $600K + $180K + $350K + $500K + $54K = $1.684M. New total: $2.5M - $1.684M = $816K/month — a 67% reduction, exceeding the 35% target.

Monitoring: I track output quality (LLM-as-judge on 1% of sampled responses) to detect degradation from quantization and routing. If quality drops, I adjust the routing threshold or quantization level."

## Tier 3 — Expert

### Challenge 1: Global LLM Serving at Planetary Scale

**Prompt:** You are the principal engineer at a global AI company with 500 million users. The company needs a serving system that delivers inference responses within 200ms for small models and 2 seconds for large models, supports 50+ models including a flagship 1 trillion parameter model, operates in 100+ countries with data residency requirements in the EU (GDPR), China (data localization), and India (data localization), and must handle 10x traffic spikes during major product launches. The infrastructure budget is $50 million per month. The team has 200 engineers across 5 timezones.

Design the LLM serving system architecture. Discuss the technical challenges, regulatory constraints, and organizational approach.

**Time limit:** 30 minutes

**Dimensions a strong answer explores:**

**Technical:**
- Edge inference for small models: deploy 7B models on edge GPU instances in 50+ locations around the world. The edge node closest to the user handles the request. 200ms latency target requires the inference to happen within 500 km of the user.
- Regional hubs for large models: deploy 70B-1T models in 10 regional hubs (one per major geography). Each hub has 1,000+ GPUs. Regional hubs communicate via high-bandwidth interconnects.
- Hierarchical model routing: edge node tries small model first. If the query is complex, the edge node forwards to the regional hub for the large model. If the query exceeds the regional hub's capacity, it routes to the global cluster.
- Specialized hardware: use a mix of GPUs (NVIDIA H100) for training-compatible inference and TPUs (Google TPU v5p) for optimized inference of Google-born models. Use LPUs (Groq) for ultra-low-latency small model inference.

**Regulatory:**
- EU: all inference happens on EU-based GPU clusters. No user data leaves the EU. Model weights can be replicated from a central store, but prompts and KV caches stay in the EU.
- China: a separate deployment with a Chinese cloud partner (Alibaba Cloud, Tencent Cloud). The model weights are a distilled version that complies with Chinese AI regulations.
- India: inference happens on India-based GPU clusters. Data localization requirements mean KV caches cannot leave India.
- The solution: region-specific deployments with no cross-region data movement for user data. Aggregated analytics (no PII) can cross regions.

**Organizational:**
- Team structure: 4 sub-teams (edge inference, core inference, regulatory compliance, developer experience)
- The edge inference team owns the 50+ edge locations and small model serving
- The core inference team owns the 10 regional hubs and large model serving
- The regulatory compliance team owns the region-specific deployments and data handling
- The developer experience team owns the unified API, SDK, documentation, and model onboarding
- The platform must support self-service model deployment — any team can deploy a model with a YAML configuration file

**Anti-patterns a weak answer shows:**
- Suggesting a single global deployment (violates data residency regulations)
- Ignoring the 200ms latency requirement for small models (edge nodes are required)
- Not addressing the 1T parameter model (requires specialized model parallelism beyond standard tensor/pipeline)
- Proposing a single hardware type (mix of GPU, TPU, LPU is optimal)
- Not addressing the organizational complexity (200 engineers across 5 timezones need clear ownership)

**Discussion of multiple valid approaches:**

**Approach A: Edge-first (Cloudflare-style)**
- Deploy small models on GPU instances at 100+ edge locations
- Deploy large models at 10 regional hubs
- Edge node handles 80% of requests without forwarding to regional hub
- Pros: lowest latency for most requests (under 100ms), natural data residency (edge node is in the user's country)
- Cons: complex operational model (100+ deployments), higher infrastructure cost (many small GPU instances)

**Approach B: Regional hub (AWS-style)**
- Deploy all models at 10 regional hubs
- Users connect to the nearest hub via DNS-based routing
- No edge inference — all requests go to a regional hub
- Pros: simpler operations (10 deployments), better GPU utilization (larger pools)
- Cons: higher latency for edge users (a user in Argentina connecting to US-East: 150ms RTT), harder data residency (serving multiple countries from one hub)

**Approach C: Hybrid (Google-style)**
- Use edge nodes for small model inference and response caching
- Use regional hubs for large model inference and training
- Edge nodes pre-process requests (tokenization, safety filtering, cache lookup)
- Regional hubs handle the heavy inference lifting
- Pros: best of both worlds, moderate operational complexity
- Cons: requires coordination between edge and hub layers, consistent routing logic

### Challenge 2: The 10x Traffic Spike

**Prompt:** Your LLM-powered application is about to launch a major marketing campaign. The campaign is expected to increase traffic by 10x for 48 hours. The system currently serves 1,000 requests per second with 500ms p50 latency. The campaign starts at 9:00 AM on a Monday — peak business hours.

Your infrastructure team is worried the system will not handle the load. The CTO says "just add more GPUs." But provisioning GPUs takes 24 hours with your cloud provider. You have 72 hours before the campaign launches.

Design the strategy to handle the 10x traffic spike. Include: (1) what you do in the 72 hours before the campaign, (2) what happens during the spike, and (3) post-campaign analysis.

**Time limit:** 30 minutes

**Dimensions a strong answer explores:**

**Pre-campaign (72 hours):**
- Pre-warm additional GPU instances: provision 50% more capacity and load model weights (45 minutes per instance). This ensures the GPUs are ready when traffic hits.
- Implement response caching: a 30% cache hit rate reduces effective load by 30%. Configure Redis cache with pre-warming for expected popular queries.
- Reduce max token limit: lower max_tokens from 2,048 to 512. Shorter responses mean less KV cache per request and higher throughput.
- Increase max batch size: from 32 to 64 (higher throughput at the cost of higher latency — acceptable during the spike).
- Notify customers: send rate limit increase requests to the cloud provider (many providers have default GPU quotas that must be increased manually).

**During the spike:**
- Rate limit non-critical traffic: batch workloads (offline processing, analytics) are paused. Only interactive traffic gets GPU resources.
- Model routing: switch from the 70B model to the 7B model for all non-critical requests. Only the most important requests (paying customers) get the 70B model.
- KV cache offloading: reduce the KV cache allocation per request by offloading to CPU memory. Each request gets 50% of the normal KV cache — causes some recomputation for very long sequences but keeps the system running.
- Dynamic batch sizing: the scheduler increases batch size as queue depth grows. At 10x traffic, batch size goes from 32 to 128. TTFT increases from 500ms to 2 seconds, but the system stays up.
- Graceful degradation: if GPU memory reaches 95%, the system switches to queue-only mode. New requests are queued with an estimated wait time. Users see "All our agents are busy" messages instead of timeouts.

**Post-campaign:**
- Analyze traffic patterns: which models got the most requests? Which queries were cached?
- Scale down: deprovision additional GPU instances (save cost)
- Update the capacity plan: the campaign proved the system can handle 10x spikes with preparation
- Implement auto-scaling pre-warming: pre-warm GPUs before future campaigns based on calendar events

**Anti-patterns a weak answer shows:**
- Relying on cloud auto-scaling (GPU provisioning takes 24 hours — too slow for a 72-hour prep window)
- Not pre-warming (if GPUs are provisioned but model weights are not loaded, the system still fails)
- No graceful degradation (the system should degrade before it crashes)
- No caching (caching is the fastest way to reduce GPU load)
- Not notifying customers (API users should know about potential rate limit changes)

**Model answer:**

"72 hours before the campaign, I have three days to prepare. My strategy has four phases.

Phase 1 (hours 72-48): Capacity estimation. I calculate that 10x traffic means 10,000 req/s. Current capacity is 1,000 req/s. Each GPU handles 100 req/s for the 70B model. I need 100 GPUs for the spike — 90 more than my current 10. I request a GPU quota increase from my cloud provider immediately (24-hour approval). I also provision 50 additional instances (400 GPUs) but keep them in a stopped state (saves cost, but the model weights are already loaded).

Phase 2 (hours 48-24): Optimization. I implement response caching. My team pre-computes cached responses for the 1,000 most common queries (expected to cover 30% of traffic). I reduce the default max_tokens from 2,048 to 512. I switch from the 70B model to a 7B model for the free tier. Only paid users get the 70B model during the spike.

Phase 3 (hours 24-0): Pre-warming. I start the 50 additional instances. Model weights load in 45 minutes, but I start them 24 hours ahead to ensure they are ready. I run a load test at 5x traffic to verify the system holds.

Phase 4 (during the spike): I monitor queue depth and GPU memory in real-time. If queue depth exceeds 5,000, I increase batch size from 32 to 64. If GPU memory exceeds 95%, I reduce per-request KV cache allocation. If all else fails, I start queuing non-critical requests with an estimated wait time — users see a progress bar instead of a timeout.

Post-campaign: I analyze the cache hit rate, peak throughput, and bottleneck GPU. The learning becomes part of the capacity plan for future campaigns.
