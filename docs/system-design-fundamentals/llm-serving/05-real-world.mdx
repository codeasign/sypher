## Company Case Studies

### OpenAI — The Production API

**Scale:** Millions of users. Billions of tokens per day. 200+ countries. Models ranging from GPT-3.5 to GPT-4 to GPT-4o.

**The problem:** OpenAI needed to serve multiple model sizes across a global user base with sub-second latency for chat and high throughput for batch processing. The system had to handle unpredictable traffic spikes (a viral ChatGPT feature, a new model release) without degrading quality.

**What they built:** OpenAI built a distributed inference system with three key innovations: (1) a model router that directs requests to the appropriate model size based on the user's subscription and the request's complexity, (2) a continuous batching system with dynamic batch sizing that adjusts based on real-time latency and throughput metrics, and (3) a multi-region deployment with regional inference clusters and a global load balancer.

**The result:** GPT-4 generates roughly 15-20 tokens per second for most queries. The system handles 10x traffic spikes without degradation by dynamically allocating GPU capacity from a shared pool. The model router reduced average cost per request by 40% by routing simple queries to smaller models.

### Anthropic — Safety-First Serving

**Scale:** Millions of users across Claude.ai and the API. Focus on safety, reliability, and constitutional AI.

**The problem:** Anthropic needed a serving system that could enforce constitutional AI constraints during inference, not just at training time. The system had to run safety checks on every input and output without adding noticeable latency. Additionally, Claude's long-context capabilities (100K+ tokens) required massive KV cache management.

**What they built:** Anthropic built a serving system with a safety inference layer: every prompt passes through a classifier before reaching the model, and every response is checked by a separate safety model before streaming to the user. The KV cache uses a hierarchical paging system that handles 100K+ token contexts by swapping rarely-accessed cache blocks to CPU memory. The system uses a "constitution" — a set of rules that the model checks its own output against before responding.

**The result:** Claude maintains safety guardrails with under 100ms added latency for safety checks. The hierarchical KV cache enables 100K token contexts without requiring 80 GB of GPU memory per request. The constitutional AI approach reduced harmful outputs by 75% compared to baseline.

### Google — TPU-Based Serving

**Scale:** Serving Gemini across Google's product suite (Search, Workspace, Cloud). Hundreds of millions of users.

**The problem:** Google needed to serve increasingly large models (Gemini Ultra, Gemini 1.5 Pro) with 1M+ token context windows. Traditional GPU-based serving could not handle this scale efficiently. The attention mechanism for 1M tokens requires O(n²) compute, which is prohibitive at inference time.

**What they built:** Google deployed Gemini on TPU v5p pods — custom ASICs designed specifically for neural network inference. A single TPU pod has 8,960 TPU chips with 95 TB of HBM memory. The serving system uses a customized attention mechanism (Multi-Query Attention) that reduces KV cache size by sharing keys and values across attention heads.

**The result:** Gemini 1.5 Pro achieves 1M token context at under 30 seconds prefill time, with token generation at roughly 50 tokens/second. The TPU architecture provides 2.3x better performance per watt compared to H100 GPUs for inference workloads. The shared KV cache mechanism reduces memory usage by 80% compared to standard multi-head attention.

### Meta — Open Source Serving (Llama)

**Scale:** Llama models (7B, 13B, 70B, 405B) downloaded hundreds of millions of times. The models are open source, so the serving infrastructure is distributed across thousands of independent deployments.

**The problem:** Meta needed to ensure that Llama models could be served efficiently by anyone — from a single developer on a laptop to a large company with a GPU cluster. The models had to work with existing open source serving frameworks (vLLM, Ollama, llama.cpp).

**What they built:** Meta optimized Llama for serving efficiency. The architecture uses Grouped-Query Attention (GQA) — a middle ground between standard multi-head attention and Google's multi-query attention. Llama 3 uses 8 KV heads and 32 query heads, reducing KV cache by 75% compared to standard attention. The models are released in multiple quantization levels (FP16, FP8, INT4) to support different hardware.

**The result:** Llama 3 70B runs on a single H100-80GB GPU with INT4 quantization (35 GB model size + 12 GB KV cache = 47 GB total). The open source serving ecosystem (vLLM, Ollama) achieved within 10% of Meta's internal serving performance. The GQA architecture reduced KV cache memory by 75%, enabling 4x larger batches on the same hardware.

### Replicate — Cloud GPU Marketplace

**Scale:** Thousands of models served across a shared GPU pool. Models include open source LLMs, image generation models, audio models, and video models.

**The problem:** Replicate needed to serve thousands of different models on a shared GPU pool, with each model having different hardware requirements (GPU memory, CUDA compatibility) and usage patterns (spiky, unpredictable).

**What they built:** Replicate built a hot-cold GPU pool. The hot pool keeps frequently-used models loaded in GPU memory (no cold start). The cold pool loads models on demand. When a request arrives, the scheduler checks if the model is already loaded on a hot GPU. If yes, the request is routed there. If no, the scheduler finds the cheapest GPU that can fit the model and loads it (cold start: 10-60 seconds). After 15 minutes of inactivity, the model is unloaded and the GPU is returned to the pool.

**The result:** The hot-cold pool model achieved 80% cache hit rate for popular models. Cold start latency (10-60 seconds) is acceptable for batch workloads. The shared pool reduced GPU costs by 60% compared to dedicated instances for each model.

## Open Source Implementations

### vLLM

vLLM is the most widely used open source LLM serving framework. It was developed at UC Berkeley and is now maintained by the community.

**How it implements the concept:** vLLM's key innovation is PagedAttention — a paged KV cache that eliminates memory fragmentation. It also implements continuous batching, efficient tensor parallelism, and a scheduler that handles waiting/running/done queues. vLLM supports all major model architectures (Llama, Mistral, Falcon, GPT-NeoX, etc.) through a unified interface.

**Specific trade-offs vs alternatives:** vLLM has lower latency than Hugging Face TGI for most workloads (20-30% faster). It uses less GPU memory due to paged KV cache (up to 50% savings). Compared to TensorRT-LLM, vLLM is easier to use (no compilation step, dynamic batching) but achieves 5-10% lower peak throughput.

**When engineers choose it:** vLLM is the default choice for most LLM serving deployments. It is easy to set up, supports most models, and provides excellent performance. Choose vLLM when you need a production-ready serving system without vendor lock-in.

### TensorRT-LLM (NVIDIA)

TensorRT-LLM is NVIDIA's optimized inference engine. It compiles models into optimized CUDA kernels and provides graph optimization for maximum throughput.

**How it implements the concept:** TensorRT-LLM uses graph compilation — it analyzes the model's computation graph and fuses operations, eliminates redundant computations, and generates custom CUDA kernels. The compilation step takes 10-60 minutes per model but produces the fastest possible inference. It supports INT4, FP8, and FP4 quantization through NVIDIA's tensor cores.

**Specific trade-offs vs alternatives:** TensorRT-LLM achieves 10-20% higher throughput than vLLM on NVIDIA GPUs. However, it requires longer setup time (model compilation), has less model coverage (NVIDIA GPUs only), and is harder to extend (custom kernels vs Python).

**When engineers choose it:** Choose TensorRT-LLM when you need maximum throughput on NVIDIA hardware, you have a single model that you serve at high volume, and you can invest in the compilation and tuning process.

### llama.cpp

llama.cpp is a C++ implementation of LLM inference optimized for CPU and consumer GPUs. It supports quantization down to 2 bits, enabling large models on modest hardware.

**How it implements the concept:** llama.cpp uses memory-mapped model loading and optimized CPU kernels (MMLA, ARM NEON). It supports various quantization schemes (Q2_K through Q8_0) that trade accuracy for memory. A 70B model in Q4_K_M quantization uses 40 GB — fitting on a single consumer GPU or high-end CPU with enough RAM.

**Specific trade-offs vs alternatives:** llama.cpp runs on CPUs and consumer GPUs, making it accessible to anyone. It has lower throughput than GPU-native solutions (3-5x slower for batch workloads). However, for single-user interactive use (one request at a time), the performance difference is minimal.

**When engineers choose it:** Choose llama.cpp for local development, edge devices, or when GPU access is limited. It is also the foundation for Ollama, which provides a Docker-like experience for running LLMs locally.

## Cloud Provider Implementations

### AWS

**AWS Bedrock** — Managed API for foundation models. Supports models from AI21, Anthropic, Cohere, Meta, Mistral, and Stability AI. Bedrock handles infrastructure, scaling, and model updates. You pay per token with no GPU provisioning.

**AWS SageMaker** — Managed ML platform that supports custom model deployment. You provision GPU instances and deploy your model using SageMaker's inference endpoints. SageMaker handles auto-scaling, health checks, and model versioning.

**Cost trade-off:** Bedrock costs $0.01-$0.10 per 1,000 tokens for popular models. SageMaker self-hosted costs roughly $30,000/month per 8×H100 instance. At 100 million tokens per day, Bedrock costs $1,000-$10,000/day ($30,000-$300,000/month). SageMaker costs $30,000/month for the GPU instance — 10x cheaper at high volume.

### GCP

**Vertex AI** — Google's managed ML platform. Supports Gemini models, Claude (via Anthropic partnership), Llama, and custom models on TPUs or GPUs. Vertex AI provides model garden with pre-built models and deployment templates.

**Cloud TPU** — Google's custom TPU v5p chips for inference. Available through Vertex AI. TPUs provide better performance per watt than GPUs for LLM inference but require model adaptation (JAX or TensorFlow).

**Cost trade-off:** Vertex AI charges per token for managed models ($0.0025-$0.05 per 1,000 tokens for Gemini). Self-hosted TPU pods cost $100-$200/hour for a full pod. At high volume (1B+ tokens/day), self-hosted is more cost-effective.

### Azure

**Azure OpenAI Service** — Microsoft's managed API for OpenAI models (GPT-4, GPT-4o). Provides the same models as OpenAI's API but with Azure's enterprise features (data residency, private networking, compliance certifications).

**Azure Machine Learning** — Managed ML platform for custom model deployment. Supports GPU instances (ND-series with H100). Provides auto-scaling, monitoring, and model registry.

**Cost trade-off:** Azure OpenAI Service costs the same as OpenAI's API (roughly $0.03-$0.06 per 1,000 tokens for GPT-4). Azure ML self-hosted costs are similar to AWS SageMaker. The advantage of Azure is enterprise integration: Active Directory authentication, private endpoints, and compliance certifications (HIPAA, FedRAMP, SOC 2).

## Lessons Learned

### Lesson 1: Memory Is the Bottleneck, Not Compute

Every team reported the same lesson: GPU memory limits batch size more than GPU compute. The KV cache is the dominant memory consumer, growing linearly with sequence length and batch size. Optimizing the KV cache (paging, quantization, multi-query attention) has the highest impact on serving throughput.

**The pattern:** Before optimizing compute kernels, measure memory utilization. The bottleneck is usually the KV cache, not the matrix multiplication. Solutions: use paged KV cache (vLLM), reduce KV heads (GQA, MQA), quantize KV cache to FP8, and offload infrequently used blocks to CPU memory.

### Lesson 2: Continuous Batching Is Not Optional

Every production system uses continuous batching. Static batching wastes GPU cycles waiting for slow requests. Continuous batching fills every idle slot with a new request. Teams that started with static batching saw 3-5x throughput improvements after switching to continuous batching.

**The pattern:** Even with a single concurrent user, continuous batching helps by pipelining the prefill and decode phases. Always implement continuous batching from day one.

### Lesson 3: Model Routing Saves 40-60% on Cost

Not all requests need the largest model. Simple requests (translation, classification, summarization) can be handled by a 7B model. Complex reasoning tasks need 70B+. A model router that classifies request complexity and routes to the appropriate model size reduces costs by 40-60% without degrading quality.

**The pattern:** Implement a classifier (small, fast model) that predicts the optimal model size for each request. The classifier runs in under 10ms. Route simple queries to a small model and complex queries to the large model.

### Lesson 4: Cold Starts Matter More Than You Think

Loading model weights takes 45-90 seconds. Every cold start means 45 seconds of unavailability for that GPU instance. Teams that ignored cold starts ended up with 5-10 minute deployment times and poor auto-scaling behavior.

**The pattern:** Always pre-warm. Keep the model loaded in GPU memory on standby instances. Use a hot pool for frequently-used models. Use blue-green deployments so the new version is loaded before the old version is drained.

### Lesson 5: Monitor Quality, Not Just Performance

The serving system monitors latency, throughput, and error rates. But the most important metric — output quality — is invisible. Teams discovered quality degradation only through user reports, days or weeks after the regression was introduced.

**The pattern:** Deploy a quality monitoring system that runs alongside the serving system. Use an LLM-as-judge to score response quality on a sample of requests (1% of traffic). Track quality scores over time and alert on regressions. Add regression tests with known expected outputs to the deployment pipeline.
