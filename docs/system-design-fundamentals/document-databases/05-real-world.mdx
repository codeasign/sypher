# Document Databases in Production

## Company Case Studies

### EA Games: Game Client Data Platform

Electronic Arts runs one of the largest MongoDB deployments in the world.
Every EA game -- FIFA, Battlefield, Madden, The Sims -- uses MongoDB as the
backend for player profiles, game state, leaderboards, and progression data.
The platform serves over 300 million players globally.

**The challenge**: Each game has a different data model. FIFA tracks player
cards, chemistry styles, and transfer market listings. Battlefield tracks
weapon loadouts, kill/death statistics, and map progress. The Sims tracks
furniture placement, character skills, and relationship scores. A relational
schema would require separate databases or hundreds of migration scripts per
game release.

**The solution**: A shared document database platform. Each game stores its
data as JSON documents in a collection. New game features add new fields
without schema changes. EA runs MongoDB sharded clusters with 50+ shards per
cluster, handling millions of writes per second during peak launch days.

**Performance numbers**: FIFA Ultimate Team, the most data-intensive mode,
processes over 3 million concurrent players during major tournaments. Average
write latency is under 3ms. Read latency for player profile loads is under
5ms at p99.

### Discord: Message Storage at Billion-Message Scale

Discord stores chat messages in MongoDB. Every message is a document:
`{id, channelId, authorId, content, timestamp, attachments, reactions}`.
With over 500 million messages sent daily, the message collection grows by
hundreds of gigabytes per day.

**The challenge**: Messages are written once and read for a short window (the
last 100 messages per channel). Old messages are rarely accessed but must be
available for search. The write volume is massive and evenly distributed
across channels.

**The solution**: Discord shards messages by `channelId` using a hashed shard
key. Each channel's messages are stored in time order via a compound index
`{channelId: 1, timestamp: -1}`. Old messages are migrated to colder storage
after 30 days using a TTL-based archival pipeline. Discord uses MongoDB
change streams to emit new messages to their real-time WebSocket servers.

**Key numbers**: Peak writes: 15,000 messages/second. Message documents:
under 1KB each. Cache hit rate for recent messages: 98%. Search latency for
messages under 30 days: under 50ms.

### Uber: Trip Data and Event Storage

Uber's early architecture used MongoDB for trip data: every ride generates a
trip document that is updated as the ride progresses through states (requested,
driver-assigned, picked-up, in-progress, completed, paid, rated).

**The challenge**: Trip documents are updated 10-15 times during a ride.
Updates come from different services (dispatch, payments, ratings). The
document needs atomic updates to individual fields without conflicting.

**The solution**: Uber used MongoDB's `$set` operator for atomic field updates.
Each update touched only the changed fields. The trip document contained
embedded subdocuments for fare breakdown, route waypoints, and driver
information -- all read together on trip completion. After scaling past
100 million trips, Uber migrated trip storage to its own document store
(Schemaless, built on MySQL) but the document model philosophy remained.

**Lesson learned**: Uber's Schemaless system is essentially a document
database on top of MySQL: each row stores a JSON blob with a schema version
number. The application reads the version and interprets the JSON accordingly.
This is the document model with deliberate schema versioning -- a hybrid
that gives schema flexibility without losing the battle-tested MySQL storage
engine.

### Google Firebase / Firestore: Serverless Document Database

Google Firestore is a serverless document database used by startups and
mobile apps worldwide. It is the backend for Firebase, Google's mobile
development platform. Firestore stores data as documents organized into
collections and subcollections.

**The challenge**: As a serverless database, Firestore must handle tenants
ranging from hobby projects (10 documents) to enterprise applications (1
billion documents) on the same infrastructure, with no server provisioning.

**The solution**: Firestore uses Google Spanner as the underlying storage
engine. Spanner provides global strong consistency, automatic sharding, and
replication across data centers. Firestore layers a document API on top:
collections, documents, subcollections, real-time listeners, and security
rules. The Spanner layer handles the sharding, replication, and transactions.

**Key properties**: Strong consistency by default (unlike MongoDB's eventual
consistency for secondary reads). Real-time listeners via WebSocket -- your
app subscribes to a document and gets push updates when it changes.
Serverless pricing: you pay per document read/write/delete, not per provisioned
server. Autoscaling is transparent.

### Cloudflare: D1 and Workers KV

Cloudflare uses document databases in multiple products. D1 is a serverless
SQLite database that stores JSON documents in BLOB columns. Workers KV is a
global key-value store where values are JSON documents replicated to 300+
edge locations.

**The challenge**: Workers KV must serve documents with single-digit
millisecond latency from any of 300 global locations. Each location needs a
local copy. Writes must propagate globally within 60 seconds.

**The solution**: KV stores documents as JSON blobs in a replicated log-
structured merge-tree. Each edge location caches frequently-read documents.
Writes go to a central coordinator and are broadcast to all edges via a
gossip protocol. Documents have a TTL (default 60s for reads, unlimited for
writes). The cache-aside pattern: each edge checks its local cache first,
then reads from the central store on miss.

## Open Source Implementations

**MongoDB** is the dominant open-source document database. Community Server
is free with the SSPL license. Features: replica sets, sharding, aggregation
pipeline, change streams, Atlas search (Lucene-based), transactions.

**CouchDB** takes a different approach: documents are stored as JSON, accessed
via HTTP REST API. Every document has a `_rev` field for optimistic
concurrency control. Primary use case: offline-first applications and
peer-to-peer sync (PouchDB on the browser syncs with CouchDB on the server).

**Couchbase** combines document storage with a distributed key-value cache
layer. Documents are accessed via the memcached protocol for sub-millisecond
reads or via N1QL (SQL for JSON) for complex queries. Couchbase uses a
full-text search service built on Bleve.

**RavenDB** is a .NET-native document database with ACID transactions across
the cluster. Designed for the Microsoft ecosystem with LINQ query support.

## Cloud Provider Implementations

**AWS DocumentDB**: MongoDB-compatible document database. Not truly MongoDB --
it implements the MongoDB 4.0 wire protocol but lacks features like change
streams, aggregation pipeline stages ($lookup), and MongoDB 5.0+ features.
Best for lift-and-shift of existing MongoDB workloads that stay within the
AWS ecosystem. Use native MongoDB Atlas for full feature parity.

**Azure Cosmos DB for MongoDB**: Azure's managed MongoDB-compatible service.
Supports the MongoDB 4.0 wire protocol with some 5.0 features. Cosmos DB
provides turnkey global distribution -- enable any number of Azure regions
with a button click. Each region has a local replica and writes can be
multi-master. The cost: RU/s pricing is different from MongoDB Atlas's
instance-based pricing.

**Google Firestore**: Described above. Serverless, strongly consistent,
real-time. Best for mobile apps and serverless architectures. Not MongoDB-
compatible -- you use the Firestore SDK directly.

## Lessons Learned

### Lesson 1: Design Your Documents Around Access Patterns

EA Games succeeded because each game's document schema was designed around
how that game accesses data. FIFA Ultimate Team reads a player's squad
(lineup) and chemistry in one query. The squad document embeds all 11
players with their chemistry values. A relational approach would need 3
JOINs. The document approach is one read.

**The pattern**: Before writing any code, list every query your application
makes. Design one document type per query type. If a query needs data from
two documents, consider embedding the second document's data into the first.

### Lesson 2: Schema Version Your Documents

Uber's Schemaless system taught the industry a critical lesson: embedding a
`schema_version` field in every document enables safe schema evolution.
When you need to change a document's structure, your application writes new
documents with `schema_version: 2` and handles both version 1 and version 2
on reads. After all old documents are migrated (lazy migration on read), you
can drop the version 1 code path.

### Lesson 3: Monitor the Oplog Window

Every MongoDB outage at scale traces back to an oplog problem: too small,
too full, or a stuck secondary. EA's SRE team monitors `rs.printReplicationInfo()`
as a first-line health check. The oplog window (time between oldest and
newest entry) should never drop below 2 hours. If it does, either the oplog
size is too small or a secondary is lagging.

### Lesson 4: Shard Keys Cannot Change After Deployment

Choosing a shard key is a permanent decision in MongoDB. You cannot change a
collection's shard key after data is inserted. Discord picked `channelId`
(hash) and has lived with it. Uber's early MongoDB deployment chose
`tripId` as the shard key and could not change it when access patterns
shifted. Either pick a shard key that will work for years, or choose a
database that supports online shard key changes.

### Lesson 5: Read Your Writes with Majority Concern for Critical Paths

Discord's message system uses `writeConcern: majority` for sending messages
(the user needs to see their message immediately) and `writeConcern: 1` for
reactions and edits (fast, eventual consistency acceptable). This tiered
approach gives durability where it matters and speed where it does not.
