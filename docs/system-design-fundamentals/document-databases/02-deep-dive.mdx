# Document Databases — Under the Hood

MongoDB is the most widely deployed document database in production today. This deep dive takes you inside the storage engine, query execution layer, replication protocol, and performance tuning knobs that make MongoDB tick. By the end, you will understand exactly what happens when you insert a document, run a query, or lose a node in a replica set.

**Note:** This page uses explicit line-break escaping in diagram code blocks to preserve ASCII art in the rendered output.

## 1. Internal Architecture

At the highest level, a MongoDB `mongod` process delegates all persistent storage responsibilities to the **WiredTiger storage engine**. WiredTiger handles B-tree management, caching, compression, transactions, and checkpointing. The `mongod` layer supplies the query language (MQL), authentication, replication, and sharding logic on top.

### WiredTiger in Detail

WiredTiger is an **LSM-free, B-tree-based** storage engine. It stores each collection as two B-trees: one for the document records (the "data handle") and one for each index. Every B-tree splits its pages into an internal (non-leaf) tree and leaf pages that hold the actual key-value pairs.

The key-value format inside WiredTiger for a document store is:

- **Key**: the `_id` value (or a user-defined shard key combined with `_id`) encoded as a BSON binary.
- **Value**: the full BSON document bytes.

For secondary indexes the key is the indexed field value(s) and the value is the `_id` of the document, which WiredTiger then uses to look up the full document in the record store.

WiredTiger supports **row-oriented storage** (default) and **column-oriented storage** for specific workloads. MongoDB always uses row-oriented (key-value) mode.

### Snapshot Isolation and Checkpoints

WiredTiger provides **snapshot isolation** using a multiversion concurrency control (MVCC) scheme. Every transaction sees a consistent snapshot of the data as of the transaction's start time. When a transaction commits, its changes become visible to new transactions.

A **checkpoint** is a consistent, on-disk snapshot of all data at a point in time. WiredTiger takes a checkpoint every 60 seconds or when the journal exceeds 2 GB. During a checkpoint, WiredTiger flushes all dirty pages to disk and records a checkpoint marker in the journal. On recovery, MongoDB replays the journal from the last checkpoint forward.

## 2. Data Structures and Algorithms

### BSON — Binary JSON

MongoDB stores every document internally as **BSON** (Binary JSON). BSON is a binary-encoded serialization of JSON-like documents that extends JSON with additional data types.

The BSON specification defines a type system where each field is encoded as a **type byte**, a **field name string** (null-terminated), and a **value**. The document itself is wrapped in a 4-byte length prefix and a trailing `\\x00`.

BSON supports these additional types beyond standard JSON:

- **ObjectId** (`\\x07`): 12-byte identifier, unique per document.
- **Date** (`\\x09`): 64-bit integer milliseconds since epoch.
- **Binary** (`\\x05`): raw byte arrays with a subtype tag.
- **Decimal128** (`\\x13`): IEEE 754-2008 decimal floating-point.
- **Regular Expression** (`\\x0B`): stores regex pattern and flags.
- **JavaScript** (`\\x0D` / `\\x0F`): stores JS code, optionally with scope.
- **Timestamp** (`\\x11`): 64-bit value used internally for the oplog.

When you query a collection, MongoDB reads BSON from WiredTiger leaf pages, deserializes only the fields requested by your projection, and returns a BSON document over the wire. This zero-copy, field-level deserialization is one reason MongoDB can stream large result sets efficiently.

### B-Tree Indexes

MongoDB indexes are **B-Trees** managed inside WiredTiger. Every index — whether it is a single-field index, a compound index, a multikey index, a text index, or a geospatial index — is stored as a separate B-tree in its own `.wt` file.

A B-tree in WiredTiger is a **page-oriented** tree. Each page stores a sorted array of key-value pairs plus pointers to child pages. The fanout (number of children per page) is typically in the hundreds, so a tree storing billions of documents is only 3-5 levels deep.

Key algorithmic properties of a MongoDB B-tree index:

- **Search**: O(log n) comparisons. At each internal node a binary search identifies the child pointer to follow.
- **Insert**: O(log n). WiredTiger locates the leaf page, inserts the key-value pair in sorted order, and splits the page if it exceeds the maximum page size (default 16 KB).
- **Range Scan**: O(log n + m) where m is the number of matching entries. The tree is traversed to the first matching key, then leaf pages are followed via the next-page sibling pointer.
- **Unique Index**: O(log n) for insert with an additional uniqueness check against the existing keys on the leaf page.
- **Compound Index**: Keys are concatenated field values. The B-tree sorts by the first field, then the second, and so on. Only prefix queries can use the index efficiently.

### MVCC — Multiversion Concurrency Control

WiredTiger implements MVCC to allow concurrent reads and writes without blocking. Each update to a B-tree page does not overwrite the existing value in place. Instead, WiredTiger creates a **new version** of the affected row or page, tagged with the transaction ID that created it.

Here is how MVCC works at the page level:

When a read operation arrives, WiredTiger computes a **snapshot** that contains:

- `snapshot_min`: the oldest active transaction ID at snapshot time.
- `snapshot_max`: the newest committed transaction ID visible to this snapshot.
- An array of **snapshots** (active transaction IDs between min and max that are not yet committed).

During a page read, WiredTiger walks the version chain backward from the most recent version until it finds one whose transaction ID is committed and within the snapshot window. Old versions are eventually garbage-collected by a background **eviction thread** once no active transaction can still reference them.

This design gives you **snapshot isolation**: every query sees a consistent view of the database as of the moment the query began, regardless of concurrent writes. It also means that a long-running query does not block writers — they each create new versions, and the query simply walks past them.

Let us walk through each stage in more detail.

### Step-by-Step Breakdown

**Step 1 — TCP Connection.** The driver opens a TCP socket to the `mongod` process. MongoDB uses a custom wire protocol over TCP, default port 27017. Modern drivers maintain a connection pool (default size varies, typically 10-100 connections per client).

**Step 2 — Wire Protocol Message.** The driver serializes the query as an `OP_MSG` message. The wire protocol frame includes a 16-byte standard message header (message length, request ID, response to, opcode) followed by the BSON-encoded query document. The message is written to the socket.

**Step 3 — Session and Auth Check.** The `mongod` networking layer deserializes the message and associates it with a client session. If authentication is enabled, the server verifies that the client has the `find` action on the `users` collection. If authorization fails, the server replies with an error and never reaches the query engine.

**Step 4 — Query Planning.** The query planner enumerates candidate indexes using a combination of the query filter, sort, and projection. For each candidate index, the planner computes whether the index can satisfy:

- **The filter**: can we use the index to bound the scan (e.g., `age >= 18`)?
- **The sort**: can we return results in the requested sort order without an in-memory sort?
- **The projection**: can we return all required fields from the index alone (a **covered query**)?

The planner selects the index with the lowest estimated cost. In the example, the compound index `{age: 1, name: 1}` wins because it satisfies both the filter prefix (`age`) and the sort (`name`) in one traversal.

**Step 5 — Open Cursor.** The storage engine opens a `WT_CURSOR` on the selected index. A cursor is WiredTiger's abstraction for a position in a B-tree. It holds a snapshot and a reference to the current key-value pair.

**Step 6 — B-Tree Search.** The cursor searches the index B-tree for the first key >= the lower bound (`age=18`). Starting from the root page, it performs a binary search at each internal page to choose the correct child pointer, descending until it reaches the leaf page that could contain the first matching entry.

**Step 7 — Read and Copy Page.** Once the leaf page is located, WiredTiger checks whether the page is already in the in-memory cache. If not, it issues a **page-in** from disk:

1. Read the compressed page bytes from the `.wt` file.
2. Decompress the page (snappy, zlib, or zstd, depending on configuration).
3. Rebuild the page's internal key-value array from the packed format.
4. Insert the page into the cache's page hash table.
5. Pin the page in memory so it is not evicted during the cursor iteration.

If the page was already cached but dirty (modified by a write that has not been checkpointed), WiredTiger returns the clean, snapshot-consistent version by walking the MVCC chain.

**Step 8 — Iterate and Apply Filter.** The cursor advances through each key-value pair on the leaf page. For each pair:

- The key (the indexed fields) is compared against the filter. Because the scan began at `age=18`, the first key is guaranteed to satisfy the `$gte` condition. The cursor continues until it encounters a key where `age > 18` evaluates to false (though in practice, the scan stops at the end of the qualifying range).
- The sort order is maintained because the compound index stores keys sorted by `age` first, then `name`.
- Because the index contains both `age` and `name`, the query is **covered** — no document lookup is needed. The fields are read directly from the index key.
- The matching document fields are collected into a result batch. The batch size is 101 documents by default, or smaller if the total batch exceeds 16 MB.

**Step 9 — Follow Sibling Pointers.** When the current leaf page is exhausted, the cursor follows the `next` sibling pointer to the adjacent leaf page at the same B-tree level. WiredTiger leaf pages are linked in a singly linked list, enabling efficient forward range scans without revisiting internal pages.

**Step 10 — Reply.** The server serializes the batch of results into an `OP_REPLY` message and writes it to the TCP socket. The reply includes a cursor ID that the client can use in subsequent `getMore` calls.

**Step 11 — getMore.** If the query matched more documents than fit in the first batch, the driver sends a `getMore` message with the cursor ID. The server locates the still-open cursor (cursors expire after 10 minutes of inactivity by default) and resumes iteration from where it left off.

**Step 12 — Continue.** Steps 8-11 repeat until either the cursor is exhausted (no more matching documents) or the query limit is reached. The server then closes the cursor and frees its resources.

### Write Operation Lifecycle

Writes follow a similar path but with three critical additions: journaling, replication, and MVCC version creation.

1. The write command reaches the storage engine interface.
2. WiredTiger begins a transaction and obtains a transaction ID.
3. The B-tree leaf page is modified in memory: a new version is appended to the version chain (no in-place overwrite).
4. The change is appended to the **journal** (write-ahead log) before the transaction is acknowledged to the client.
5. On transaction commit, the journal entry is synced to disk (configurable via `writeConcern: {j: true}`).
6. The write is recorded in the **oplog** — a capped collection that stores every write operation in an idempotent format.
7. Secondary members pull the oplog entry and replay it against their own WiredTiger instance.
8. The client receives an acknowledgment only after the configured write concern is satisfied.

## 4. Consistency and State

### Replica Sets

A MongoDB **replica set** is a group of `mongod` processes that maintain the same data set. One node is the **primary** (accepts writes), and the rest are **secondaries** (apply writes from the primary's oplog). If the primary fails, the remaining secondaries elect a new primary.

  6. A checks: S1's optime >= my (no data) OK
  7. Majority reached: S1 (self) + S2 + A = 3 out of 3 votes
  8. S1 transitions to PRIMARY, opens new connections
  9. All writes redirected to new primary
  10. Old primary (P) re-enters as SECONDARY on reconnection`}

### Rollback

If a primary goes down after accepting writes but before those writes have propagated to secondaries, a **rollback** occurs when the old primary re-joins the replica set. The rolled-back writes are written to a rollback directory in `dbpath` as BSON files so you can manually recover them if needed.

MongoDB can avoid rollbacks entirely by using `writeConcern: "majority"`. With majority write concern, a write is acknowledged only after it has been committed on a majority of voting members. This guarantees that if a primary fails, the write survives on at least one node that participated in the next election.

### Read Concerns and Write Concerns

MongoDB provides tunable consistency levels.

**Read Concern** controls the consistency and isolation of read operations:

| Concern         | Guarantee                                                  | Use Case                              |
|-----------------|------------------------------------------------------------|---------------------------------------|
| `local`         | Returns the most recent data on that node (may be stale)   | Default, lowest latency               |
| `available`     | Same as `local` for non-sharded clusters                   | Sharded clusters, fastest             |
| `majority`      | Returns data that has been committed to a majority         | Strong consistency, avoid rollbacks   |
| `linearizable`  | Reads the most recent majority-committed write (slowest)   | Single-document linearizability       |
| `snapshot`      | Returns data from a specific majority-committed timestamp   | Causal consistency, transactions      |

**Write Concern** controls the durability and acknowledgment of write operations:

| `w` value       | Behavior                                                     |
|-----------------|--------------------------------------------------------------|
| `1`             | Acknowledge after the primary commits the write (default)    |
| `"majority"`    | Acknowledge after a majority of voting members commit        |
| `N`            | Acknowledge after N members commit (N &lt;= set size)          |
| `0`             | Fire-and-forget, no acknowledgment (fastest, no guarantee)   |

Combined with `j: true` (journal sync), write concern lets you dial between maximum throughput (w=0, j=false) and maximum durability (w="majority", j=true).

## 5. Performance Characteristics

### Latency

Read and write latency in MongoDB is dominated by three factors:

1. **WiredTiger Cache Hit Ratio.** If the working set fits in the cache, reads are sub-millisecond (memory-only). Once the working set exceeds the cache, reads require disk I/O, adding 2-10 ms per page fault depending on your storage medium (NVMe SSD vs. spinning disk).

2. **Journal Sync Frequency.** With the default journal commit interval (100 ms), a write that requires journal sync (`j: true`) takes at minimum 100 ms wall-clock. If you can tolerate losing up to 100 ms of writes on a crash, set `j: false` and gain significant write throughput.

3. **Network Round Trip.** MongoDB drivers serialize BSON over TCP. For a single-document find, expect roughly 1 RTT + server processing. In a geographically distributed replica set, the round-trip to the primary dominates.

### Throughput

Write throughput is limited by:

- **WiredTiger checkpoint contention.** During a checkpoint (every 60 s or 2 GB of journal), WiredTiger flushes dirty pages to disk. Write throughput can drop by 10-20% during checkpoint I/O.
- **Oplog apply rate on secondaries.** Each secondary must replay every write from the oplog. If writes on the primary exceed a single secondary's replay capacity, replication lag grows unbounded.
- **Document size.** Larger documents mean more BSON bytes to serialize, write to the journal, compress, and replicate. MongoDB enforces a 16 MB document size limit.

Read throughput scales with the number of secondary nodes you add (for read-only workloads that tolerate staleness). Each secondary can serve `readPreference: secondary` queries independently.

### Memory

WiredTiger uses both the **internal cache** (configurable, default 50% of RAM - 1 GB) and the operating system's **filesystem cache** (page cache). Together, they cache recently accessed B-tree pages.

Key memory considerations:

- The internal cache stores **uncompressed** pages. Compression savings apply at the disk level; pages are decompressed when read into cache.
- Indexes consume memory proportionate to the number of unique keys. A 10 million-document collection with a 12-byte key and 8-byte value per index entry uses roughly 200 MB of cache for that index alone.
- The **eviction thread** runs continuously, scanning for cold pages to evict. In high-throughput workloads, eviction contention is the most common bottleneck.

### Query Execution Stats

You can inspect query performance using `explain()`:

```
db.users.find({ age: { $gte: 18 } }).sort({ name: 1 }).explain("executionStats")
```

Key metrics to examine:

- **totalKeysExamined**: number of index keys scanned. Should be close to `nReturned`. A large gap means an inefficient index scan.
- **totalDocsExamined**: number of documents fetched from the record store. Zero means the query was covered by the index.
- **executionTimeMillis**: total wall-clock time for query execution.
- **stage**: the query plan stage (e.g., `IXSCAN`, `FETCH`, `SORT`, `SHARD_MERGE`).
- **rejectedPlans**: if more than zero, the query planner considered and rejected other index plans.

## 6. Configuration and Tuning

### WiredTiger Cache Size

The single most impactful configuration parameter is `wiredTigerCacheSizeGB`. It controls how much memory WiredTiger dedicates to its internal page cache.

Default formula: `min(50% of (RAM - 1 GB), 256 GB)`.

Tuning guidelines:

- Set it high enough to hold your **working set** (the set of documents and indexes accessed frequently). A working set that fits in cache avoids disk reads.
- Leave enough memory for the OS filesystem cache (the OS also caches `.wt` and journal files). A common production setting is 60-70% of RAM for WiredTiger on a dedicated database server.
- Monitor `wiredTiger.cache.bytes currently in the cache` and `wiredTiger.cache.tracked dirty bytes in the cache` via `serverStatus()`. If the dirty ratio consistently exceeds 20%, your cache is too small or writes are too bursty.

### Oplog Size

The oplog is a capped collection (`local.oplog.rs`) on each replica set member. Its default size is:

- 5% of free disk space on 64-bit systems (min 990 MB, max 50 GB).

A small oplog causes secondaries to fall off the oplog window if they lag too long. Increase it if you anticipate prolonged secondary downtime, large bulk writes, or initial sync of a new member.

Production rule: set oplog size to allow at least **24 hours of writes** at peak throughput. Calculate:

```
oplogSizeMB = (avgWriteBytesPerSecond * 86400) / (1024 * 1024)
```

On a system doing 10 MB/s of writes: `(10 * 86400) / 1.05` = about 823 GB. In practice, 50-100 GB is typical.

### Max Connections

The `maxIncomingConnections` setting (default 65536) limits the number of concurrent client connections. Each connection consumes a thread and approximately 1 MB of RAM for its stack.

If your application uses a connection pool of 100 connections per instance and you scale to 100 instances, you need 10,000 connections. Monitor `connections.current` via `serverStatus()` and ensure it stays below 80% of the limit.

### Practical Tuning Workflow

1. **Establish a baseline.** Run your workload with default settings and collect metrics: cache hit ratio, average query latency, write throughput, replication lag.
2. **Identify the bottleneck.** Is it CPU, disk IOPS, memory pressure, or network? Use OS tools (`iostat`, `vmstat`, `top`) and MongoDB metrics in tandem.
3. **Tune one parameter at a time.** Change the cache size, journal commit interval, or oplog size independently. Re-run the workload and compare.
4. **Iterate.** Database tuning is workload-specific. What works for a time-series workload (high write throughput, append-only) differs from what works for a catalog workload (read-heavy, many indexes).

### Common Pitfalls and Anti-Patterns

Even with a solid understanding of MongoDB internals, teams often fall into these traps. Knowing them helps you avoid expensive remediation later.

**Oversized Documents.** Embedding everything into a single document because "MongoDB is document-oriented" leads to documents that exceed 1 MB. Large documents consume more cache space, slow down replication (the entire document is written to the oplog on each update), and make the storage engine page-out more aggressive. Rule of thumb: keep documents under 100 KB in the general case.

**Missing Index for Sort.** If you run `db.collection.find(query).sort({ field: -1 })` without an index that supports the sort, MongoDB performs a 32 MB in-memory sort. When the sort exceeds 32 MB, it either spills to disk (slow) or errors out. Always check `explain()` for a `SORT` stage — if present, add an index.

**Unbounded Array Push.** Pushing elements into an array without limit causes the document to grow without bound. Each array append rewrites the entire document, causing write amplification. Worse, multikey indexes on the array grow linearly with array size. Cap arrays at a reasonable size or use a separate collection.

**Not Using Write Concern.** Running with default write concern (`w: 1`) means a primary can acknowledge a write, crash, and that write can be rolled back. For critical data, use `w: "majority"`. For firehose ingestion, use `w: 1` and accept the risk.

**Too Many Indexes.** Each index adds write overhead — every insert or update must update every index. A collection with 10 indexes runs 10 B-tree modifications per write. Index only the query patterns you actually use, and drop unused indexes (check `$indexStats`).

### Putting It All Together

A MongoDB production deployment is the interplay of the storage engine's B-tree internals, the MVCC concurrency model, the replication protocol, and your configuration choices. When you write a document:

- WiredTiger appends a new version to the B-tree leaf page in memory.
- The change is serialized to the journal.
- On checkpoint, the in-memory tree is flushed to the `.wt` files on disk.
- The oplog captures the write for replication to secondaries.
- The client receives an acknowledgment based on the configured write concern.

When you read a document:

- The query planner selects an index.
- WiredTiger navigates the B-tree, reads the leaf page from cache or disk, and walks the MVCC chain to the correct snapshot version.
- The matching documents are serialized as BSON and returned over the wire.

Every layer — from B-tree page layout to replica set elections — is designed to give you tunable tradeoffs between consistency, availability, and performance. Understanding these internals lets you diagnose slow queries, configure your cluster correctly, and design schemas that take full advantage of MongoDB's architecture.
