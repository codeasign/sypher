# Understanding Document Databases

## The Startup Pain

You are building the next big social platform. Users create profiles, write
posts, comment on each other's content, and tag friends. Day one, you reach
for a relational database because that is what you learned in school. You
design tables: `users`, `posts`, `comments`, `tags`, `post_tags`. The schema
looks clean. Foreign keys connect everything. Normalization feels right.

Week two arrives and something is off. A single user profile page needs seven
JOINs. A feed query that shows the last 20 posts with their authors, comment
counts, and like status runs over 200 milliseconds. You add indexes. You
optimize queries. You denormalize a few columns. The pain shifts — now you are
running migrations every time a feature changes. Adding an "avatar URL" field
requires an ALTER TABLE that locks production for minutes. Every schema change
is a deployment risk.

A friend at a YC startup tells you they switched to MongoDB last year. They
store a user profile as one JSON document — name, email, avatar, preferences,
address history, all nested together. A post is another document with comments
embedded inside it. No JOINs. No migrations for new fields. Adding a field
means just writing it. Their query time for a feed dropped from 200ms to
5ms. You are skeptical. You ask about data duplication, consistency, and what
happens when a comment updates across a thousand copies. Your friend smiles.
"Trade-offs," they say. "But for 80% of our reads, we never touch more than
one document."

This is the moment document databases stop being an academic curiosity and
become a practical weapon. They trade storage efficiency for speed and
developer velocity. They let you think about your data the way your code
already does — as objects, not as rows in a spreadsheet.

## The Insight

Imagine a restaurant kitchen. In a relational kitchen, every ingredient lives
in its own labeled bin: one bin for tomatoes, one for onions, one for spices.
When the chef wants to make a pasta dish, they walk to six different bins,
gather components, and assemble. The kitchen is organized but slow. Every dish
needs a fetch from many bins.

A document-database kitchen works differently. Each recipe has its own
container. The pasta recipe container holds the tomatoes, onions, spices, and
instructions all together in one box. The chef grabs one container and starts
cooking. If tomorrow the recipe adds a new ingredient — mushroom — you just
drop a mushroom into that container. You do not need to build a new bin for
mushrooms across the entire kitchen.

This is the document model. You store data the way your application uses it —
as self-contained units that bundle related information together. The unit
is a document: a JSON-like structure of nested fields and arrays. A customer
is not rows spread across `customers`, `addresses`, and `orders` tables. A
customer is one document with addresses nested inside and order references
embedded or linked.

## Core Vocabulary

**Document** — The fundamental unit of data. A self-contained, self-describing
structure (usually JSON or BSON) that stores all related information for a
single entity. Contrast with a relational row, which stores only the data
for one table and relies on JOINs to assemble related data.

**Collection** — A grouping of documents, analogous to a table in relational
databases. Unlike tables, collections do not enforce a uniform schema across
documents — each document can have different fields.

**BSON** — Binary JSON, the data format used by MongoDB. Extends JSON with
additional types (ObjectId, Date, Binary, Decimal128). More compact than text
JSON and supports native comparison operators for types like dates.

**Embedded Document** — A document nested inside another document as a field
value. Represents a one-to-one or one-to-few relationship stored inline.
Example: an `address` object inside a `user` document.

**Reference** — A pointer from one document to another, typically using an ID
field. Used for one-to-many and many-to-many relationships. Requires a second
query or a $lookup operation to resolve.

**Aggregation Pipeline** — A multi-stage data processing framework that
filters, transforms, groups, sorts, and projects documents through a sequence
of stages ($match, $group, $sort, $project, $lookup). The document equivalent
of SQL GROUP BY and JOIN combined.

**Index** — A data structure (B-tree by default) that speeds up document
lookups by field value. MongoDB supports single-field, compound, multikey,
text, geospatial, and hashed indexes. Indexes consume memory and slow writes.

**Sharding** — Horizontal partitioning of a collection across multiple servers.
Documents are distributed by a shard key. Each shard holds a subset of the
data. Enables writes and reads to scale beyond a single machine.

**Replica Set** — A group of MongoDB servers that maintain the same data set
for redundancy and high availability. One primary accepts writes; secondaries
replicate data and can serve reads if configured.

**Schema Flexibility** — The property that documents in the same collection
can have different fields. Contrast with rigid relational schemas. Enables
rapid iteration but shifts schema management to the application layer.

**Embedded vs Referenced** — The fundamental design decision in document
databases. Embedding stores related data together (fast reads, no JOINs, data
duplication). Referencing keeps data normalized (no duplication, requires
multiple queries or $lookup). Picking correctly is the difference between a
fast system and a slow one.

**TTL Index** — A special index that automatically deletes documents after a
configurable time period. Used for session data, logs, and time-series data.

## How It Actually Works

Every operation in a document database flows through a well-defined sequence
of stages. Understanding these stages is the difference between treating the
database as a magic black box and knowing exactly why your query took 300ms.

### Step 1: Client sends a write request

Your application sends a BSON document to the database driver. The driver
serializes your application's data structure — a Python dict, a JavaScript
object, a Go struct — into the BSON wire format. This serialization happens
on the client side, in your application process.

### Step 2: Server receives and parses the BSON

The MongoDB server receives the raw bytes on its wire protocol port (27017
by default). The networking layer hands the bytes to the message parser,
which validates the BSON structure and extracts the operation type (insert,
update, delete, query), the target collection, and the document payload.
The server assigns a unique connection thread from its thread pool.

### Step 3: Authorization and validation

The server checks whether the authenticated user has permission to write to
the target collection. Role-based access control (RBAC) is evaluated. If the
user lacks `insert` or `update` privileges, the server returns an error
immediately — no data structures are touched. For authorized writes, the
server proceeds to the storage engine layer.

### Step 4: WiredTiger writes to the document store

WiredTiger is MongoDB's default storage engine. It maintains a B-tree of
documents organized by `_id`. The write goes to two places simultaneously:
the in-memory cache (for fast recent-read access) and the write-ahead log
(journal). The journal is written first — this guarantees durability even if
the server crashes before the data reaches disk.

### Step 5: Index is updated

If the collection has indexes on any fields (beyond the mandatory `_id`
index), WiredTiger updates each index B-tree. A compound index on
`{lastName: 1, city: 1}` creates a single B-tree where keys are
`("Smith", "NYC")` pairs pointing to the document location. Adding an index
entry is an additional B-tree insert — fast for one index, costly for ten.

### Step 6: Oplog entry is created

MongoDB's replication relies on the oplog (operations log). The primary node
creates an oplog entry describing the write: namespace, operation type,
document ID, and the new document state (for idempotent replay). The oplog
is a capped collection — it has a maximum size (default: 5% of free disk
space, minimum 1GB, maximum 50GB). When the oplog fills, old entries are
overwritten.

### Step 7: Response to client

The server sends a write acknowledgement back to the client. If write concern
is `{w: 1}` (the default), the server acknowledges after the journal write
succeeds. If write concern is `{w: "majority"}`, the server waits until a
majority of replica set members have acknowledged before responding. Each
replica set member independently applies the oplog entry.

### Step 8: Read path — query dispatch

When you issue a read (`db.users.find({name: "Al"})`), the server consults
the query planner. The planner examines available indexes, evaluates their
selectivity, and picks the winning plan. If a matching index exists, the
server traverses the B-tree to locate documents. If no index matches, the
server performs a collection scan — reading every document in the collection
and filtering by the query predicate. A collection scan on 10 million
documents means touching every B-tree leaf.

## Variants and Flavors

### Pure Document Stores

MongoDB and Couchbase are the dominant pure document databases. Every
interaction is with JSON/BSON documents. There is no separate query language
like SQL — you use a document-oriented API (find, insert, update, aggregate).
MongoDB uses a JSON-like query syntax. Couchbase uses N1QL, which feels like
SQL but operates on JSON documents.

### Key-Value with Document Features

Databases like Amazon DynamoDB and RedisJSON blur the line between key-value
stores and document databases. DynamoDB stores items (documents) accessed by
a primary key, with optional secondary indexes. It supports document queries
on nested attributes but lacks MongoDB's rich aggregation pipeline and ad-hoc
query capabilities. RedisJSON adds document operations on top of Redis's
in-memory key-value store, giving you document reads at microsecond latency
for datasets that fit in RAM.

### NewSQL Document Databases

Databases like CockroachDB and Google Spanner store JSON documents but
provide SQL access and strong consistency across regions. They combine the
developer experience of documents with the transactional guarantees of
traditional relational databases. The trade-off: higher latency for cross-
region writes due to consensus protocols (Paxos, Raft). These are document
databases for when you absolutely need serializable transactions but want
JSON flexibility.

### Cloud-Managed Document Databases

Amazon DocumentDB, Azure Cosmos DB, and Google Firestore are fully managed
document databases. Firestore is serverless — you pay per operation, not per
provisioned server. It offers real-time listeners: your application subscribes
to document changes and gets push notifications when data updates. Cosmos DB
supports multiple API surfaces (MongoDB-compatible, SQL, Gremlin, Cassandra)
on the same underlying engine.

## Mental Model Check

**Q1: When should you embed a related document instead of referencing it?**

Embed when the related data is always accessed together with the parent and
does not grow unboundedly. A user's shipping address: embed it (one or two
addresses, always read with the user). A user's 10,000 orders: reference them
(too large to embed, often queried independently).

**Q2: What happens to write performance when you add 10 new indexes?**

Each index adds a B-tree insert per write. Writes become 10x slower for
index maintenance. Indexes optimize reads at a direct cost to write speed.
Rule of thumb: no more than 5-8 indexes per collection for write-heavy
workloads.

**Q3: Can you run JOINs in MongoDB?**

Yes, using the `$lookup` aggregation stage. It performs a left outer join
with another collection. However, `$lookup` is slower than relational JOINs
because MongoDB was not designed for it. Use `$lookup` sparingly — if you
need many JOINs, consider whether a document database is the right choice,
or redesign your document schema to embed the data.

**Q4: How does schema flexibility actually work day-to-day?**

Your application code defines the schema that documents should follow. If a
new version of your app writes a new field, existing documents simply lack
that field. Your code must handle missing fields gracefully. This shifts
schema enforcement from the database to your application — a trade-off that
gives you zero-downtime deployments but requires defensive coding.

**Q5: What is the max document size in MongoDB?**

16 megabytes. This is a hard limit. If you need to store larger data, use
GridFS, which splits the data into chunks stored as separate documents. Most
applications should keep documents under 100KB for performance — large
documents waste cache space and make every read more expensive.
