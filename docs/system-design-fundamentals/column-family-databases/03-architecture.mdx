# Column-Family Databases at Scale

This page shows how column-family databases like Apache Cassandra and ScyllaDB are deployed and operated in real production environments — from a single node all the way to multi-region clusters handling millions of writes per second.

## Naive Implementation

The simplest Cassandra deployment is a single node running with default settings. You create a keyspace, add a table, and start writing data.

### What Breaks at 1,000 Users

At low scale, the single-node setup works fine. The database handles a few thousand writes per second with sub-10ms latency. The commit log provides crash recovery, and the memtable flushes produce a manageable number of SSTables.

### What Breaks at 100,000 Users

- **Single point of failure**: The node goes down, all data is inaccessible until it restarts. With default settings (RF=1), any disk failure causes permanent data loss.
- **Read pressure**: All queries hit one node. Cache efficiency drops as the working set grows beyond available memory.
- **Write throughput**: A single node can handle about 10,000-50,000 writes per second. Beyond that, compaction storms overwhelm the I/O subsystem and latency spikes.

### What Breaks at 10,000,000 Users

- **Impossible**: A single Cassandra node cannot handle this scale under any configuration. The storage engine, compaction, and garbage collection all reach fundamental limits.
- **Symptoms**: Request timeouts (Cassandra defaults to 10-second timeout), heap pressure (OutOfMemoryErrors in JVM), compaction falling behind, hinted handoff queue growing unboundedly, clients experiencing connection failures.

## Production-Grade Architecture

A production Cassandra or ScyllaDB deployment uses multiple nodes in a ring topology with data partitioned across all nodes using consistent hashing.

### Key Components

**Virtual nodes (vnodes)**: Each physical node handles multiple token ranges, improving load distribution and reducing the impact of node additions and removals.

**Gossip protocol**: Every second, each node exchanges state information with 1-3 other nodes. This provides eventual cluster-wide membership awareness without a centralized registry.

**Snitch**: Maps nodes to physical locations (racks, datacenters). The snitch tells the partitioner which replicas are closest, minimizing cross-datacenter traffic.

**Replication factor**: Defines how many copies of each row exist. RF=3 is standard for production, tolerating up to 2 node failures in a single cluster and providing read consistency with QUORUM.

## Integration Patterns

### Pattern 1: Time-Series Event Ingestion

Cassandra excels at write-heavy time-series workloads. Each event gets a partition key (e.g., device_id + hour) and a clustering key (timestamp).

**Key characteristics**: Fast point lookups by session ID; automatic TTL expiration cleans up stale sessions; no single point of failure.

**Trade-off**: Cannot query by session attribute-for that you would need a secondary index or a separate table.

### Pattern 2: Session Store

Cassandra serves as a distributed session store for web applications. The session ID is the partition key, and the entire session is stored as a single wide row with map columns for session attributes.

### Pattern 3: Shopping Cart

Cassandra's lightweight transactions (LWT) with IF NOT EXISTS / IF conditions enable compare-and-set operations that work well for shopping cart scenarios.

**Important**: LWTs are expensive (4 round-trips instead of 1). Limit LWT usage to fewer than 100 operations per second per partition.

### Pattern 4: Multi-Region Active-Active

Cassandra supports multi-datacenter replication natively. Each datacenter operates independently with its own RF setting, and data replicates asynchronously between datacenters.

## Multi-Region and Global Scale

### Consistency in Multi-Region

Cross-datacenter replication is asynchronous by default. If you write to US-East and immediately read from EU-West, you may get stale data. Use CL=EACH_QUORUM for writes that must be visible globally, but expect 100-300 ms additional latency per write.

### Failover

If one datacenter becomes unavailable, the application configures the driver to route to another datacenter. Since each datacenter has a full copy of the data (with RF=3), reads continue with CL=LOCAL_QUORUM in the healthy datacenter.

## Capacity Planning

### Sizing Formula

Nodes = max(writes_per_sec / 30000, reads_per_sec / 10000, disk_per_node / 4000)

### Example: 10M DAU Application

Assume each user causes 1 read + 0.5 writes per active session:
- Peak writes: 5,000,000 users x 0.5 writes x 2x peak factor = 5,000,000 writes/sec
- Peak reads: 5,000,000 x 1 read x 2x peak factor = 10,000,000 reads/sec
- Data per day: 5,000,000 writes x 1 KB x 86400 sec = roughly 430 GB/day
- Data retention: 30 days active + 90 days cold = roughly 52 TB

**Result**: 50-80 nodes across 2 datacenters, each with 2 TB SSD, 32 GB RAM, RF=3.

## Deployment and Operations

### Zero-Downtime Deployment

Cassandra supports rolling restarts. To upgrade the cluster without downtime:
1. Set nodetool disablebinary on one node
2. Drain the node: nodetool drain
3. Upgrade the software and restart
4. Wait for the node to rejoin the ring (check nodetool status)
5. Run repair: nodetool repair -pr
6. Repeat for the next node

### Monitoring

| Metric | What It Tells You | Alert Threshold |
|---|---|---|
| Pending compactions | Compaction is falling behind | over 10 pending |
| Read timeout count | Queries are timing out | over 0.1 percent of reads |
| Hinted handoff count | Nodes are down for extended periods | over 10,000 |
| Cache hit ratio | Working set doesn't fit in memory | under 80 percent |
| SSTable count per table | Compaction is struggling | over 50 per shard |

### Backup and Restore

Cassandra snapshots are file-system level. Run nodetool snapshot to create a snapshot of all SSTables. Snapshots are incremental-they share unchanged files with the live data directory.

Restore procedure:
1. Truncate the table to remove existing data
2. Copy snapshot SSTables to the data directory
3. Run nodetool refresh to load the SSTables
4. Run repair if restoring to multiple nodes

### Common Pitfalls

- Too many tokens or vnodes over 256 increases gossip overhead and repair time
- RF mismatch across keyspaces can cause unexpected data loss
- No regular repairs: anti-entropy repair should run at least once every gc_grace_seconds period (default 10 days)
- Overloaded partitions: a single partition key that gets 100K+ writes per second creates a hot spot
- Too many secondary indexes: each secondary index query hits all nodes

## Summary

Column-family databases trade rich query capabilities for write scalability and availability under partition. A production cluster distributes data via consistent hashing, replicates across nodes and datacenters, and uses gossip-based membership for decentralized operation. The key to success is careful partition key design, regular maintenance (repairs, compaction monitoring), and sizing for write throughput rather than raw storage.
