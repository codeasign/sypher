# Column-Family Databases -- Interview Playbook

System design interviews frequently touch on column-family databases. You might be asked to design a system that uses Cassandra, compare it to other databases, or explain when you would use a wide-column store over a relational database or a document store. This page prepares you for each of these scenarios.

## How This Topic Appears

Interview questions about column-family databases fall into eight categories.

**Category 1: Database Selection.** "You are designing a time-series metrics system. Which database would you choose and why?" The interviewer wants to hear you reason about the trade-off between consistency, availability, and schema flexibility.

**Category 2: Cassandra-Specific Design.** "Design a messaging system like Discord. How would you model the data in Cassandra?" This tests your ability to design schemas for specific query patterns.

**Category 3: Comparison.** "Compare Cassandra to MongoDB for an event-logging workload." Tests your understanding of the fundamental differences between document stores and wide-column stores.

**Category 4: CAP Theorem.** "Explain how Cassandra handles a network partition. What happens to consistency?" Tests your understanding of tunable consistency and the AP trade-off.

**Category 5: Internals.** "How does Cassandra achieve high write throughput? Walk me through the write path." Tests depth of knowledge about commit logs, memtables, SSTables, and compaction.

**Category 6: Schema Design.** "You need to store user sessions that expire after 24 hours. Design the Cassandra schema." Tests your ability to use partition keys, clustering keys, TTLs, and composite keys.

**Category 7: Operations.** "You are running a Cassandra cluster and read latency spikes every hour. How do you debug it?" Tests operational knowledge: compaction, GC pauses, hot partitions, tombstone accumulation.

**Category 8: Scaling.** "Your Cassandra cluster handles 50K writes/second but needs to handle 500K writes/second. How do you scale?" Tests your understanding of horizontal scaling, ring management, and capacity planning.

## What the Interviewer Is Actually Testing

Interviewers ask about column-family databases to evaluate four things:

**Depth of understanding.** Can you explain the architecture at multiple levels of abstraction? Do you know the write path step by step? Can you talk about Bloom filters, compaction strategies, and gossip protocol with specificity? Vague answers ("Cassandra is fast because it writes to memory") reveal surface-level knowledge.

**Trade-off awareness.** Do you understand that every database choice involves trade-offs? A candidate who says "Cassandra is always the right choice for high-throughput systems" fails this test. A candidate who says "Cassandra is the right choice for this workload because of X, but the trade-off is Y" passes.

**Schema design skill.** The ability to design a Cassandra schema from query patterns is the strongest signal of real experience. If you cannot explain why your partition key choice avoids hot spots and your clustering key choice enables efficient range scans, you do not understand the storage model.

**Operational maturity.** Interviewers at senior levels (L5+) expect you to discuss operations: repair scheduling, compaction tuning, monitoring, backup strategies. Junior candidates can focus on features; senior candidates must focus on operating at scale.

## Level-by-Level Expectations

### L3/L4 (Early Career)

At this level, you should be able to describe the high-level architecture and name the core components. You should understand when to use a column-family database versus a relational database.

Sample answer for "Compare Cassandra and PostgreSQL for an event-logging system":

"PostgreSQL stores data in fixed-schema rows. Each event would need the same columns, and adding a new event type requires an ALTER TABLE. Cassandra stores data in wide columns: each row can have different columns, and the schema is flexible. For event logging where different event types have different attributes, Cassandra avoids null columns and schema migrations. The trade-off is that Cassandra does not support joins or ACID transactions across partitions -- if you need to join events with user profiles in the same query, PostgreSQL is the better choice."

### L5 (Senior)

At this level, you should discuss internals, discuss specific trade-offs, and propose concrete schema designs. You should reference specific features like compaction strategies, Bloom filters, and tunable consistency.

Sample answer for "Design the schema for a time-series metrics system with 10-second resolution, 100,000 metrics, 7-day retention":

"I would use a composite partition key: `(metric_name, date_bucket)` where `date_bucket` is the date in YYYYMMDD format. The clustering key would be the timestamp. This distributes writes across 7 partitions per metric (one per day), avoiding a hot partition on a single metric. For reads of the last hour, the client queries the current day's partition with a timestamp range. I would use TimeWindowCompactionStrategy with a 7-day TTL. The TTL automatically cleans data older than 7 days, and TWCS only compacts each day's window once, minimizing write amplification. For consistency, I would use LOCAL_QUORUM for both reads and writes so that each datacenter is consistent independently."

### L6 (Staff)

At this level, you should discuss multi-datacenter deployment, failure scenarios, and capacity planning in detail. You should reference real-world incidents or patterns from known deployments.

Sample answer for "How would you operate a Cassandra cluster across three regions with 200 nodes?":

"I would deploy 6 nodes per availability zone, 3 AZs per region, for 18 nodes per region across multiple regions. Replication factor of 3 within each region. Cross-region replication is asynchronous. For consistency, I would use LOCAL_QUORUM for writes and reads to avoid cross-region latency. For conflict resolution, last-write-wins with client-provided timestamps -- I would ensure all application servers run NTP with sub-10 ms drift. Weekly repair via Cassandra Reaper, scheduled during low traffic. Monitoring would focus on pending compactions, p99 latency, hint counts, and SSTable counts per read. I would use ScyllaDB if JVM GC pauses become problematic -- ScyllaDB's seastar framework eliminates GC and provides more predictable latency."

### L7/L8 (Principal/Director)

At this level, you should discuss organizational impact, cost analysis, and when NOT to use the technology. You should show that you evaluate databases based on business requirements, not technical novelty.

Sample answer for "Should we standardize on Cassandra for all new services?":

"No. Cassandra is excellent for specific workloads: high-throughput writes, time-series data, event logging, and data with flexible schemas. But it is a poor choice for workloads that need ACID transactions, complex joins, ad-hoc analytical queries, or strong global consistency with zero staleness. I would establish a decision framework: services with write-heavy workloads, partition-tolerant consistency requirements, and simple query patterns use Cassandra. Services with complex relational data use PostgreSQL. Services with document-oriented data use DynamoDB or MongoDB. The key is to match the database to the workload, not to standardize on a single database for all workloads."

## FAANG Framework for Answering

When you encounter a column-family database question in an interview, use this six-step framework.

**Step 1: Clarify requirements.** "What is the read-to-write ratio? What are the query patterns? Are there consistency requirements? What is the data volume and TTL? Is this a global system with multiple regions?"

**Step 2: State the core question.** "So the problem is: design a storage system that handles 100,000 events per second from IoT devices, with 30-day retention, queryable by device ID and timestamp range. The client needs fast writes and eventually consistent reads."

**Step 3: Propose the database.** "I would use Cassandra or ScyllaDB because the workload is write-heavy, the schema is flexible (different device types produce different data), and eventual consistency is acceptable."

**Step 4: Design the schema.** "The partition key would be `(device_id, date)` to distribute writes across days. The clustering key would be `timestamp DESC` so the most recent events are read first. I would use TWCS for compaction and a 30-day TTL."

**Step 5: Discuss trade-offs.** "The trade-off is that range queries across devices require multiple queries. If the product team needs to query 'all devices in a factory' as a single partition, I would need a different partition key. The TTL-based cleanup means queries cannot access data older than 30 days without a separate archival strategy."

**Step 6: Describe operations at scale.** "At 100K writes/second, I would start with 12 nodes (each handling approximately 25K writes/second at RF=3). I would monitor compaction backlog and add nodes when utilization exceeds 60%. I would configure Grafana dashboards for p50/p99/p999 latency, pending compactions, and SSTable counts. I would schedule weekly repair and automated backups via SSTable snapshots."

## Phrase Bank

Use these phrases to demonstrate expertise. Each one signals a specific dimension of knowledge.

1. "The write path is about sequential I/O -- commit log append is sequential, memtable flush is sequential, compaction reads and writes are sequential. The read path requires random I/O, but Bloom filters minimize the number of SSTables that need scanning."

2. "Tunable consistency means you choose your trade-off at the query level, not at the database level. You can use ONE for high-throughput logging and QUORUM for user-facing reads, all on the same cluster."

3. "The partition key is the single most important design decision. A hot partition is the most common production failure mode, and it is almost always caused by a partition key with insufficient cardinality."

4. "Bloom filters let you skip SSTables that cannot contain the requested key. They are why Cassandra can maintain good read performance even with dozens of SSTables per column family."

5. "Compaction is not optional maintenance. It is a direct function of write throughput. More writes means more SSTable flushes means more compaction. You must size your cluster for compaction I/O, not just for client I/O."

6. "In a multi-datacenter deployment, use LOCAL_QUORUM to avoid cross-DC latency on every request. Asynchronous cross-DC replication handles global distribution, but you accept a replication lag of 100-500 ms."

7. "Hinted handoff improves write availability but is not a substitute for repair. Hints are only stored for a limited window (default 3 hours). After that, you need full repair."

8. "TimeWindowCompactionStrategy is the right choice for time-series data. It groups SSTables by time window and compacts each window once. This minimizes write amplification compared to size-tiered or leveled compaction."

9. "ScyllaDB's shared-nothing architecture gives each vCPU its own memory, storage, and networking stack. This eliminates lock contention and makes latency predictable, which is especially important for multi-tenant workloads."

10. "The gossip protocol handles failure detection and cluster membership without a single point of failure. But gossip is slow to converge in large clusters -- plan for a 30-60 second delay before all nodes agree on a topology change."

11. "Read repair brings replicas into consistency during reads, but only for rows that are actively read. You still need full anti-entropy repair to fix rows that are never read."

12. "Cassandra's primary key is more than a unique identifier. It is a physical data placement directive. The partition key determines which node stores the data. The clustering key determines the sort order within that node."

13. "When you design a Cassandra schema, start with your queries. List every query your application needs. Then design one table per query pattern. Denormalization is not a bug; it is the expected design pattern."

14. "Last-write-wins is simple and fast, but it assumes clocks are synchronized. In practice, clock skew causes data loss. Run NTP with tight configuration and monitor clock drift across your nodes."

15. "Cassandra's batch statement is not a performance optimization. It is an atomicity guarantee for writes within a single partition. Using batches across partitions actually hurts performance because the coordinator must coordinate with multiple replicas."

## Red Flags

Avoid these specific mistakes in your answer. Each one signals inexperience or misunderstanding.

**1. Saying "Cassandra is eventually consistent" without qualification.** Cassandra is tunably consistent. You can make it as consistent as a relational database (ALL level) or as fast as possible (ONE level). The ability to choose per-query is the whole point.

**2. Proposing a Cassandra schema without discussing partition key design.** The partition key determines everything about performance and scalability. If you do not mention partition key cardinality and distribution, the interviewer knows you have not deployed Cassandra at scale.

**3. Using "NoSQL" as a complete answer.** "Cassandra is a NoSQL database" explains nothing. Be specific: wide-column store, peer-to-peer architecture, tunable consistency, flexible schema.

**4. Recommending Cassandra for OLAP/analytics workloads.** Cassandra is an OLTP database. It is designed for fast point queries and partition scans, not for full-table aggregations. If you need to run SUM, COUNT, GROUP BY across all partitions, you need a different tool.

**5. Forgetting to mention repair.** Repair is the most commonly skipped operational task and the most common cause of data inconsistency. If you do not mention repair in your operational plan, you appear inexperienced.

**6. Treating compaction as a one-time setup.** Compaction is an ongoing background process that affects read and write latency continuously. You must monitor it, tune it, and plan capacity for it.

**7. Confusing column-family with columnar (OLAP).** Column-family databases store rows grouped by row key. Columnar databases (like Redshift, ClickHouse) store columns grouped by column. Different workloads, different architectures. This is the most common conceptual error.

**8. Overlooking the importance of client-provided timestamps.** Cassandra's conflict resolution depends on timestamps. If you do not control timestamps from the client, clocks on different nodes can produce different orderings. You should always send timestamps from the application server.

## Practice Drills

Use these five timed prompts to practice answering column-family database questions under pressure.

**Drill 1 (5 minutes).** "Your team is building an IoT platform that ingests 50,000 sensor readings per second. Each reading is 500 bytes. Sensors send data every 10 seconds. Queries are: 'get the last 100 readings from sensor X' and 'get readings from sensor X between time A and B.' Data must be retained for 90 days. Design the storage solution."

**Drill 2 (5 minutes).** "Explain the difference between SizeTieredCompaction, LeveledCompaction, and TimeWindowCompaction. When would you choose each one?"

**Drill 3 (5 minutes).** "Your Cassandra read latency has been climbing over the past week. You notice that SSTable count per read is averaging 25, up from 4 a week ago. What is happening and what do you do?"

**Drill 4 (10 minutes).** "Design a multi-region active-active Cassandra deployment for a global e-commerce platform. Users should be able to read and write from any region with low latency. Address conflict resolution, failover, and consistency."

**Drill 5 (10 minutes).** "You need to migrate a 5 TB PostgreSQL table to Cassandra. The table has 80 columns and 200,000,000 rows. Reads filter on user_id, date range, and event_type. Only 50% of columns are populated per row. Design the migration plan, the new schema, and the rollback strategy."
