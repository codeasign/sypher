Docker and containerization are a staple topic in system design interviews. Companies ask about Docker to evaluate your understanding of deployment, infrastructure, and operational practices. This playbook covers Docker-specific interview patterns, level expectations, and practice drills.

## How This Topic Appears

Docker questions appear in several forms across the interview process. Interviewers may ask directly about containers or embed containerization concepts within broader system design scenarios.

### Direct Questions (Design-Focused)

1. "Design a container orchestration system for a microservice architecture."
2. "How would you deploy and manage 100 microservices using Docker?"
3. "Design a system to build, store, and deploy Docker images at global scale."
4. "How would you design a CI/CD pipeline that uses Docker for testing and deployment?"

### Embedded Questions (Scenario-Focused)

5. "You are deploying a distributed system across three cloud regions. How do you ensure consistency in the deployment process?"
6. "Your production deployment takes 30 minutes because images are too large. How do you optimize it?"
7. "A containerized database keeps losing data on restart. What is wrong and how do you fix it?"
8. "Your Docker host runs out of disk space every two weeks. What is causing it and how do you prevent it?"
9. "Design a deployment strategy that allows you to roll back a bad release in under 60 seconds."
10. "You need to run untrusted third-party code in your infrastructure. Can Docker containers provide sufficient isolation?"

### Behavioral Questions (Experience-Focused)

11. "Tell me about a time you used Docker to solve a deployment problem."
12. "What was the most significant production issue you encountered with Docker containers?"

## What Interviewer Is Actually Testing

Interviewers ask Docker questions to evaluate specific competencies:

**Operational maturity**: Do you understand what it takes to run containers in production? Candidates who only know `docker run` and `docker compose` from local development reveal a gap between development and production experience.

**Security awareness**: Do you consider the security implications of containers? Strong candidates discuss image scanning, user namespaces, seccomp profiles, capability dropping, and read-only root filesystems.

**Resource management**: Do you understand CPU, memory, and I/O constraints for containers? Can you reason about capacity planning and bin-packing?

**Deployment strategy**: Do you think about zero-downtime deployments, rollbacks, canary releases, and blue-green deployments?

**Trade-off recognition**: Do you understand when containers are the right choice and when they are not? Can you articulate the differences between containers and VMs?

**Debugging skill**: Can you diagnose common container failures — disk space, zombie processes, permission issues, network conflicts?

## Level-by-Level Expectations

### L3 / Entry Level

**Expectation**: You can describe what Docker is and demonstrate basic usage. You understand the difference between images and containers. You can write a simple Dockerfile.

**Sample question**: "What is the difference between a Docker image and a Docker container?"

**Model answer**: "A Docker image is a read-only template that contains the application, its dependencies, and configuration. It is built from a Dockerfile and stored in a registry. A container is a running instance of an image. The image is the blueprint; the container is the running process. You can have multiple containers running from the same image, each with its own isolated filesystem, network stack, and process tree. Containers add a thin writable layer on top of the image layers for any runtime modifications."

**Sample question**: "Write a Dockerfile for a Node.js application."

**Model answer**: "I would write a Dockerfile that uses a small base image, separates dependency installation from source code copy for layer caching, and runs as a non-root user for security:

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --production
COPY src/ ./src/
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup
USER appuser
EXPOSE 3000
CMD [\"node\", \"src/server.js\"]
```
"

### L4 / Mid Level

**Expectation**: You can design multi-service Docker deployments using Compose or an orchestrator. You understand Docker networking, volumes, and resource limits. You can debug common container failures.

**Sample question**: "How would you deploy a web application with a database and cache using Docker Compose?"

**Model answer**: "I would define three services in a docker-compose.yml file — web, db, and cache — connected by a bridge network. The database would use a named volume for persistence and include a health check. The web service would depend on the database being healthy rather than just started. Environment variables configure connection strings:

```yaml
version: \"3.8\"
services:
  web:
    image: my-app:1.0
    ports:
      - \"80:3000\"
    environment:
      - DB_URL=postgres://db:5432/myapp
      - REDIS_URL=redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
  db:
    image: postgres:15
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: [\"CMD\", \"pg_isready\"]
      interval: 5s
  cache:
    image: redis:7-alpine
volumes:
  pgdata:
```

For production, I would add resource limits, logging configuration, and a restart policy. I would also ensure the database is backed up regularly."

**Sample question**: "A container keeps restarting and losing its data. What could be wrong?"

**Model answer**: "There are several possible causes. First, the container might be crashing due to an application error — check `docker logs` for stack traces. Second, it might be exceeding its memory limit — check `docker inspect` for OOM kill signals. Third, data loss suggests the application is writing to the container's writable layer instead of a mounted volume. When the container restarts, Docker creates a fresh writable layer. The fix is to ensure the application writes persistent data to a volume mount path. Fourth, if the restart policy is set to `--restart always`, Docker restarts the container with a fresh writable layer every time the process exits."

### L5 / Senior Level

**Expectation**: You can design a production-grade container platform. You understand orchestration trade-offs, image optimization, multi-region deployment, and security hardening. You can articulate the relationship between Docker, containerd, Kubernetes, and other tools.

**Sample question**: "Design a container platform to run 500 microservices across multiple cloud regions."

**Model answer**: "I would design the platform around four layers: the container runtime, the orchestrator, the networking layer, and the delivery pipeline.

For the container runtime, I would use containerd directly rather than the full Docker engine, since we do not need the Docker API or build capabilities in production. This reduces the attack surface and resource overhead.

For orchestration, I would use Kubernetes on a managed service — EKS on AWS, GKE on GCP, or AKS on Azure. The orchestrator handles scheduling, health monitoring, rolling updates, and resource management. I would use node auto-scaling to adjust cluster size based on demand.

For networking, I would use a CNI plugin like Calico for per-pod IP addresses and network policies. This allows fine-grained network security policies between services.

For the delivery pipeline, I would build images using a multi-stage Dockerfile to keep images under 200 MB. Images are pushed to a geo-replicated registry (ECR or Artifact Registry) so each region pulls from a local endpoint. A CI/CD pipeline builds, scans, and signs images before deployment.

For multi-region deployment, each region has its own Kubernetes cluster. A global load balancer routes traffic to the nearest region. Images are replicated across regions asynchronously. Configuration differences between regions (database endpoints, feature flags) are injected through environment variables, not baked into images.

Security hardening includes running all containers as non-root, dropping all Linux capabilities, using read-only root filesystems, scanning images for vulnerabilities, and enforcing network policies."

### L6 / Staff Level

**Expectation**: You can design container infrastructure that serves the entire engineering organization. You understand the full ecosystem including service mesh, image signing, vulnerability management, and cost optimization. You can evaluate build vs. buy decisions for container infrastructure components.

**Sample question**: "Design a company-wide containerization strategy for a 500-engineer organization migrating from EC2 instances. Should you build on Kubernetes or use a serverless container platform?"

**Model answer**: "The choice between Kubernetes and serverless containers depends on our workload characteristics and organizational maturity.

Kubernetes (EKS/GKE/AKS) is the right choice if we have workloads that require fine-grained control over scheduling, networking, and storage. Kubernetes supports GPU workloads, stateful applications, and custom scheduling policies. However, Kubernetes requires significant operational expertise — we need a dedicated platform team of at least 3-5 engineers to manage the control plane, upgrade cycles, and cluster operations.

Serverless containers (AWS Fargate, Google Cloud Run, Azure Container Instances) are better if our workloads are predominantly stateless HTTP services. Teams define their containers, and the platform handles scaling, patching, and capacity management. The cost per request is higher than Kubernetes, but the operational cost is near zero.

My recommendation for a 500-engineer organization would be a hybrid approach. Run Kubernetes for stateful workloads, batch processing, and services that need specific infrastructure configurations. Use Cloud Run or Fargate for stateless API services, web applications, and job processors. This gives teams flexibility while keeping operational complexity manageable.

The migration strategy would follow a phased approach. Phase 1: containerize five non-critical services to build expertise. Phase 2: establish the CI/CD pipeline, image registry, and security scanning. Phase 3: migrate 20 services with well-defined interfaces. Phase 4: open the platform to all teams with documentation, templates, and training."

### L7-L8 / Principal to Distinguished

**Expectation**: You drive the technical strategy for containerization across the company or industry. You influence open-source container projects. You design novel approaches to container security, networking, or resource management.

**Sample question**: "How would you design a container runtime for a multi-tenant edge computing platform where each tenant runs arbitrary code?"

**Model answer**: "For multi-tenant edge computing with untrusted code, standard Docker containers do not provide sufficient isolation. I would design a runtime stack that uses hardware-level isolation for security combined with container semantics for developer experience.

The architecture would use micro-VMs (Firecracker or gVisor) rather than namespaced containers. Each tenant workload runs in a lightweight VM that boots in under 125 milliseconds and uses 5 MB of memory overhead — comparable to containers but with hardware-level isolation. The micro-VM includes a minimal kernel with only the syscalls needed by common runtimes.

I would implement a custom OCI runtime that wraps Firecracker, so developers use standard Dockerfiles and `docker build`, but the runtime executes their image in a micro-VM instead of a container. The runtime handles image unpacking, VM creation, network setup, and resource limits.

For networking, each micro-VM gets a dedicated network interface with bandwidth limits and packet filtering enforced at the hypervisor level, not through iptables. This prevents noisy-neighbor problems from affecting network performance.

For resource management, I would use a two-level scheduler. The first level assigns micro-VMs to physical hosts based on resource requirements and data locality. The second level handles CPU and memory allocation within each host using cgroups, with hard limits per micro-VM.

This approach provides container-like developer experience with VM-level isolation, suitable for running untrusted code from multiple tenants on shared infrastructure."

## FAANG Framework for Answering

When you encounter a Docker system design question, use this six-step framework.

### Step 1: Clarify the Scope

Understand what the interviewer is asking. Docker questions can span the entire lifecycle or focus on a specific aspect.

- "Are we designing the full container platform from image build through production deployment, or focusing on a specific aspect like the scheduling layer?"
- "How many services are we supporting and at what scale?"
- "Do we need to support stateful workloads or is everything stateless?"
- "What is the security model — single-tenant or multi-tenant?"

### Step 2: State the Core Requirements

List the non-negotiable requirements derived from the scope.

- All services run in containers with consistent environments across dev, staging, and production.
- Deployments must support zero-downtime updates and instant rollbacks.
- Resource limits prevent any single service from starving others.
- Container images are built once, signed, and deployed across environments.

### Step 3: Propose the Architecture

Describe the high-level architecture before diving into details. Use a whiteboard (or mental diagram) with the key components.

- **Build pipeline**: Source code, Dockerfile, image build, vulnerability scan, signing, registry push.
- **Orchestrator**: Kubernetes or managed alternative that handles scheduling, health, and scaling.
- **Container runtime**: containerd or Docker engine on each host.
- **Networking**: CNI plugin for per-container IP, network policies for isolation.
- **Storage**: Volumes for stateful workloads, with backup and restore procedures.
- **Registry**: Geo-replicated for global deployments.

### Step 4: Walk Through a Deployment

Show the complete lifecycle of a deployment to demonstrate end-to-end understanding.

"A developer pushes code to the main branch. The CI pipeline checks out the code, runs unit tests, and builds the Docker image using a multi-stage Dockerfile. The image is pushed to the registry with a unique tag (the Git commit hash). The pipeline scans the image for vulnerabilities — if critical vulnerabilities are found, the build fails. If the scan passes, the image is signed. The CD pipeline deploys the new image to staging. After smoke tests pass, the pipeline promotes the image to production. The orchestrator performs a rolling update, replacing containers one at a time while monitoring health checks. If health checks fail, the deployment is automatically rolled back."

### Step 5: Address Failure Modes

Show you have considered what can go wrong.

- "If a container is OOM-killed, the orchestrator restarts it. If it keeps failing, the orchestrator backs off with exponential delay."
- "If the registry is unavailable, new deployments are blocked, but running containers continue. The registry should be multi-region to avoid single points of failure."
- "If the orchestrator control plane is down, running containers continue, but no new deployments or scaling changes happen until it recovers."
- "If a host fails, the orchestrator reschedules its containers on other hosts, pulling cached images from the local registry."

### Step 6: Discuss Trade-offs

Show that you understand when Docker is the right choice and when it is not.

- "Containers provide strong deployment consistency but weaker security isolation than VMs. For multi-tenant workloads, I would evaluate Firecracker or gVisor."
- "Kubernetes provides powerful orchestration but adds operational complexity. For simpler deployments, Docker Swarm or managed container services may be more appropriate."
- "Image layer caching speeds up builds but large images increase pull times. Multi-stage builds and lightweight base images help."
- "Containers are excellent for stateless microservices but more challenging for stateful workloads. Managed database services are often a better choice."

## Phrase Bank

Using the right terminology demonstrates depth of experience. Here are phrases that signal senior-level Docker knowledge.

### Architecture and Design

- "I would use a union filesystem with copy-on-write to efficiently share image layers across containers."
- "The container runtime implements the OCI runtime specification, which defines the standard for creating and running containers."
- "Image manifests use content-addressable storage where each layer is identified by its SHA-256 digest."
- "The orchestration layer reconciles the desired state with the actual state through a control loop."
- "I would use a sidecar container pattern for cross-cutting concerns like logging and monitoring without modifying the application container."

### Operations

- "I configure log rotation at the Docker daemon level to prevent disk space exhaustion."
- "Running with `--init` ensures zombie processes are properly reaped inside the container."
- "The live-restore configuration allows the Docker daemon to restart without stopping running containers."
- "I use health checks with a start period to give the application time to initialize before Docker checks its status."
- "Read-only root filesystems with tmpfs mounts for write paths provide defense in depth against container escapes."

### Security

- "I drop all Linux capabilities and add back only the specific ones the application needs."
- "User namespace remapping maps container root to a non-root user on the host."
- "Custom seccomp profiles restrict the syscalls available inside the container."
- "Each image is signed with cosign before deployment, and admission controllers verify the signature."
- "Vulnerability scanning runs at build time and blocks images with critical CVEs."

### Performance

- "Docker adds under 3% CPU overhead for compute-bound workloads because containers share the host kernel."
- "The copy-on-write filesystem triggers a copy-up penalty on the first write to a file in a read-only layer."
- "Bridge networking adds 10-50 microseconds of latency per packet compared to host networking."
- "Container startup from a cached image takes 100-500 milliseconds, dominated by namespace and cgroup creation."
- "I would use host networking for latency-sensitive workloads to avoid the veth pair and iptables overhead."

## Red Flags

Interviewers watch for these common mistakes that indicate shallow Docker knowledge.

**Red Flag 1**: Claiming containers provide VM-level security isolation. They do not. If you say "containers are just like lightweight VMs," the interviewer will probe your understanding of namespaces, cgroups, and the shared kernel.

**Red Flag 2**: Suggesting containers are always better than VMs. Strong candidates articulate the trade-offs — isolation vs. density, compatibility (Linux containers require Linux hosts), and the security boundary differences.

**Red Flag 3**: Not mentioning resource limits. If you deploy containers without specifying memory and CPU limits, you are not production-ready. Every production deployment should constrain every container.

**Red Flag 4**: Describing Docker as a virtualization technology. Docker does not virtualize hardware — it uses kernel features for isolation. Calling it a "lightweight VM" reveals a misunderstanding of the technology.

**Red Flag 5**: Ignoring image optimization. If you do not mention multi-stage builds, image size reduction, or layer caching, the interviewer will question your production experience.

**Red Flag 6**: Suggesting databases are easy to containerize. Running stateful containers in production is significantly harder than stateless ones. Admitting this complexity shows experience.

**Red Flag 7**: Not knowing about container runtimes beyond Docker. If you do not know containerd, runc, or the OCI specification, you lack depth in the container ecosystem.

**Red Flag 8**: Overlooking security hardening. Running containers as root with default capabilities, no seccomp profile, and a writable filesystem is a common security mistake made by beginners.

## Practice Drills

Use these timed prompts to prepare for Docker interview questions. Set a 5-minute timer for each.

### Drill 1: Dockerfile Optimization (5 minutes)

You are given this Dockerfile:

```dockerfile
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y build-essential curl wget nodejs npm
COPY . /app
WORKDIR /app
RUN npm install
RUN npm run build
EXPOSE 3000
CMD ["npm", "start"]
```

Identify all issues and rewrite it for production use. Focus on image size, layer caching, security, and build efficiency.

### Drill 2: Production Deployment Design (5 minutes)

Design a Docker-based deployment for a team of 20 engineers deploying 15 microservices. There are 5,000 requests per second at peak. You have AWS available. What services do you use? How do you handle rollbacks? How do you ensure zero-downtime deployments?

### Drill 3: Debugging a Container Failure (5 minutes)

A containerized web application crashes every 4-6 hours. The orchestrator restarts it automatically, but each restart causes 30 seconds of downtime. The application logs show an OOM error. Walk through your debugging process and propose solutions.

### Drill 4: Multi-Tenant Isolation (5 minutes)

Your company wants to run customer workloads in Docker containers on shared infrastructure. Customers deploy their own code (Python, Node.js, Java). You need to prevent any customer workload from affecting others. What isolation mechanisms do you use? Is plain Docker sufficient?

### Drill 5: Global Container Registry (5 minutes)

Design a container registry that serves 10,000 developers across 6 continents. Images range from 50 MB to 2 GB. Developers push 500 new images per day. The registry must be highly available and fast for pulls worldwide. Consider storage, replication, authentication, and caching.
