## The Core Tension

**DNS trades consistency for availability and performance.**

Every design decision in DNS is a negotiation between how quickly you can propagate changes (consistency) and how reliably you can answer queries (availability). A short TTL gives you fast propagation but increases load on authoritative servers and makes your service dependent on their uptime. A long TTL makes your service resilient to authoritative server outages but means changes take hours to reach all clients.

This is not a bug. DNS was designed in 1983 for a research network where domain names changed rarely (weeks or months) and availability mattered more than freshness. Forty years later, the internet runs on microservices that change IPs every deployment, CDNs that reroute traffic in seconds, and global routing that shifts with every failure. The tension between DNS's original design and modern operational needs drives every trade-off in this chapter.

## Decision Matrix

| Decision | Option A | Option B | Choose A (Low TTL) | Choose B (High TTL) |
|---|---|---|---|---|
| **TTL for A/AAAA records** | 60 seconds | 3600 seconds | You deploy IP changes multiple times per day. Rollback speed matters more than auth NS load. Example: CDN edge, auto-scaling group with new instances. | Your IPs are stable for weeks. You want resilience against auth NS outages. Reducing DNS query cost matters. Example: corporate website, API with stable ELB DNS name. |
| **Nameserver count** | 2 (minimum redundancy) | 4+ (maximum resilience) | Your DNS provider manages the server fleet. You have limited budget or administrative overhead. Example: SaaS subdomain on Route53 with SLA. | Your domain is business-critical at global scale. You operate anycast across 4+ regions. Every 9 of uptime matters. Example: google.com, amazon.com, netflix.com. |
| **DNSSEC enabled** | Yes | No | You need cryptographic assurance that responses are authentic. Compliance (PCI DSS, HIPAA) demands it. Your clients verify signatures. Example: banking domains, government services. | You do not want the operational burden of key rotation and signature management. The 2-5% query latency overhead matters. Your recursion infrastructure does not verify. Example: internal corporate domain, small business. |
| **DNS provider** | Single managed provider (Route53) | Multi-provider (Route53 + NS1 + UltraDNS) | You accept the provider's uptime SLA. Single API to learn and maintain. Lower complexity. Example: startup scaling to 1M DAU. | You cannot tolerate any single-provider outage. You pay for redundancy and complexity. Example: top 100 global websites, financial exchanges. |
| **Recursive resolver** | Use public resolver (8.8.8.8) | Operate own resolver fleet | You have a small team or low traffic volume. Public resolvers have high cache hit rates across millions of domains. Example: early-stage startup, personal projects. | You need control over caching policy, logging, and security. You serve latency-sensitive traffic. You want to avoid resolver-level logging by third parties. Example: large-scale e-commerce, streaming platform. |
| **Zone update method** | Manual (registrar web UI) | Automated (API + IaC) | You update records a few times per year. The domain is non-critical. Example: personal blog, static brochure site. | You update records multiple times per day. Mistakes must be preventable, auditable, and revertible. Example: 500-microservice architecture, global CDN. |
| **CNAME flattening (ALIAS/ANAME)** | Enabled | Disabled | Your root domain needs CDN or PaaS hosting. You want the root to behave like a CNAME without violating the RFC. Example: example.com pointed to CloudFront. | You do not need root domain aliasing. You want strict RFC compliance. Your DNS provider charges extra for ALIAS records. Example: simple zone with only A records at root. |
| **EDNS0 client subnet (ECS)** | Enabled | Disabled | You use GeoDNS and need accurate client location even when queries come from public resolvers. Example: CDN routing for global user base. | You prioritize user privacy. ECS leaks the client's IP subnet to authoritative servers. Your traffic is mostly regional. Example: GDPR-sensitive application, local service. |

## CAP Theorem Positioning

DNS is an **AP (Available + Partition-tolerant)** system. When a network partition occurs, DNS servers continue to serve whatever data they have — even if it is stale. Consistency is sacrificed.

### What This Means in Practice

In a partition, DNS makes the following guarantees:

- **Every query gets an answer** — never a timeout or hang (within the server's capacity)
- **The answer may be stale** — up to the TTL value configured on the record
- **Stale is bounded** — the TTL caps how long a resolver can serve outdated data
- **After TTL expiry, SERVFAIL is preferred over unbounded staleness** — DNS chooses to stop answering rather than serve indefinitely old data

This is a clean AP design. Compare this to a CP system (like ZooKeeper or etcd), which would refuse to answer during a partition until quorum is restored. DNS chooses the opposite: answer always, even if the answer is slightly wrong.

## Failure Modes

### Failure 1: DDoS Attack on Authoritative Nameservers

**What happens**: An attacker sends 100 million queries per second to your authoritative nameservers from a botnet of compromised IoT devices. The nameservers are overwhelmed. Legitimate queries are dropped.

**Root cause**: UDP amplification attack vectors. The attacker spoofs source IPs so responses flood the victim. Or the attacker targets the authoritative server directly with query volume exceeding its capacity.

**Blast radius**: Complete domain unavailability for users whose cache has expired. Users with cached records (within TTL) remain unaffected. All uncached queries fail with SERVFAIL or timeout.

**Detection**: Query error rate spikes from 0.1% to 50%+. Authoritative server CPU and bandwidth saturates at 100%. Traffic graph shows a sudden step-change to 10x normal volume.

**Recovery**: Activate DDoS mitigation (Cloudflare, AWS Shield, Akamai). Enable rate limiting at the DNS provider level. Switch to a DDoS-protected DNS provider. The recovery time depends on how quickly you can change NS records at the registrar — which may be slow if the registrar's own systems are under load.

### Failure 2: Accidental Zone Deletion

**What happens**: An operator runs `terraform destroy` on the wrong workspace, deleting all DNS records for `production.example.com`. The entire zone disappears.

**Root cause**: Human error combined with insufficient access control and no deployment approval gates.

**Blast radius**: Complete domain unavailability. All subdomains, MX records, TXT records — everything resolves to NXDOMAIN. Email delivery fails within minutes. All web properties go offline.

**Detection**: Immediate. Synthetic monitoring detects all endpoints unreachable. NXDOMAIN rate jumps to 100%. Customer support tickets flood in.

**Recovery**: Restore from the last zone backup. Most DNS providers maintain a soft-delete window (5-30 minutes) where deleted zones can be recovered. Without a backup, you must reconstruct the zone from memory, version control, or secondary nameserver caches. Recovery time: 5 minutes (backup restore) to 2 hours (reconstruction).

**Prevention**: Infrastructure-as-code with mandatory approval gates. Read-only access for normal operations. Point-in-time backups with automated restore testing. Multiple environment isolation (production vs staging zones cannot be confused).

### Failure 3: TTL-Based Cascading Failure

**What happens**: During a traffic spike, the authoritative nameservers become overloaded. Queries time out. Recursive resolvers that would normally refresh their cache cannot reach the authoritative servers. The cache hit rate drops from 85% to 30% because the spike itself is driven by new users who have no cached records.

**Root cause**: The TTL was set too low (30 seconds) on a high-traffic domain. Under normal load (100 QPS), the authoritative server handles the 15 uncached queries per second easily. Under a 10x spike (1000 QPS), the uncached queries jump to 150 QPS, and the spike's novelty (new domains users have never visited) means 85% cache miss instead of 15%. The resulting 850 QPS overwhelms the authoritative server.

**Blast radius**: Self-reinforcing. More queries fail → more retries → more load → more failures. The failure cascades to all domains served by the same authoritative server (shared infrastructure).

**Detection**: Resolver error rate climbs exponentially. Authoritative server CPU hits 100% within 30 seconds of the spike.

**Recovery**: Increase TTL at the authoritative server (immediate effect on queries that reach the server). Throttle query rate at the resolver (drop excess queries). Activate stale cache serving (serve cached records even past TTL when refresh fails).

### Failure 4: Registrar Lock or Expiry

**What happens**: The domain registration expires because the billing credit card was replaced. The registrar suspends DNS resolution. The domain enters "registrar hold" and stops resolving entirely.

**Root cause**: Expired payment method, missed renewal email, or process failure in the domain management team.

**Blast radius**: Complete loss of the domain. ALL services stop: website, email, APIs, subdomains. This is the nuclear option of DNS failures.

**Detection**: All synthetic checks fail globally. The domain returns NXDOMAIN even for cached queries (registrar hold is enforced at the registry level, which is upstream of DNS).

**Recovery**: Pay the renewal fee and wait for registry propagation (1-24 hours for most TLDs). If the domain entered the redemption period (30+ days past expiry), recovery costs USD 100-400 and takes 1-7 days.

**Prevention**: Auto-renew with 30-day grace period credit card on file. Monitor domain expiry dates with alerts at 60, 30, 14, 7, and 3 days before expiry. Use multi-year registration (up to 10 years for most TLDs).

### Failure 5: DNS Poisoning / Cache Pollution

**What happens**: An attacker successfully injects a fake DNS record into a recursive resolver's cache. The resolver now returns the attacker's IP address for your domain. Users connect to the attacker's server, which presents a fake login page and captures credentials.

**Root cause**: The recursive resolver accepted a response containing additional records that were not requested. Without DNSSEC or source port randomization, an attacker can forge responses that the resolver accepts as authoritative.

## Anti-Patterns Hall of Shame

### Anti-Pattern 1: The 30-Second TTL

**Mistake**: Setting TTL to 30 seconds across all records because "we want changes to propagate fast."

**Why it seemed reasonable**: In a CI/CD world where IPs change every deploy, shorter TTLs mean faster cutover. Thirty seconds seems aggressive but not crazy.

**Production impact**: Every recursive resolver re-queries every 30 seconds. A site with 10,000 resolvers worldwide generates 333 queries per second to the authoritative server for a single record. With 50 records, that is 16,650 QPS of unnecessary load. A traffic spike that originally triggered 1000 QPS now triggers 16,650 QPS. The authoritative server melts down.

**Correct pattern**: TTL of 60-300 seconds for production records. For migration windows, pre-reduce TTL to 60 seconds 48 hours before the change. After the change, increase TTL back to 300+ seconds.

### Anti-Pattern 2: CNAME at the Zone Apex

**Mistake**: Setting a CNAME record on the bare domain `example.com` (the zone apex) instead of on `www.example.com`.

**Why it seemed reasonable**: You want `example.com` to point to the same CDN as `www.example.com`. The CDN documentation says "point a CNAME to our endpoint." So you set `example.com CNAME cdn.example.net`.

**Production impact**: The DNS specification (RFC 1034) explicitly forbids CNAME records at the zone apex. A CNAME at apex means no other records can exist for the domain — no MX records, no NS records, no TXT records. Email delivery breaks. Some resolvers ignore the CNAME entirely and return NXDOMAIN. Others follow it but drop the additional records.

**Correct pattern**: Use an ALIAS or ANAME record (provider-specific pseudo-record that behaves like CNAME but resolves to an A record at the DNS server). Or configure the CDN to accept traffic at the apex IP and use A records.

### Anti-Pattern 3: DNS as a Health Check

**Mistake**: Using DNS return-code-based load balancing (removing unhealthy backend IPs from DNS responses) as the sole health-check mechanism.

**Why it seemed reasonable**: Fewer components. Let DNS handle routing. If a server dies, stop returning its IP. Clients automatically go to the remaining servers.

**Production impact**: DNS is slow to react. With a 60-second TTL and a 30-second health check interval, a server can be down for 90 seconds before all resolvers stop returning its IP. And even then, client-side DNS caching may keep the old IP alive for the full TTL. During that time, 30% of users still hit the dead server.

**Correct pattern**: Use a load balancer (ELB, HAProxy, NGINX) for health-based routing. DNS points to the load balancer. The load balancer detects backend health in real-time (sub-second failover). DNS handles only geographic routing, not backend health.

### Anti-Pattern 4: Wildcard Everything

**Mistake**: Using a wildcard record `*.example.com A 203.0.113.10` instead of explicit records for each subdomain.

**Why it seemed reasonable**: You have dozens of internal subdomains that all point to the same load balancer. A wildcard saves 50 lines in the zone file. If you add a new subdomain, it just works.

**Production impact**: Wildcard records inappropriately match queries for domains that should not exist. A typo like `wadmin.example.com` instead of `admin.example.com` resolves instead of failing with NXDOMAIN. This breaks email security (SPF/DKIM/DMARC checks on non-existent mail subdomains), enables phishing (attackers register `fake.example.com`), and makes your zone harder to audit.

**Correct pattern**: Explicit records for every subdomain. Use infrastructure-as-code to automate record creation (Terraform, Pulumi, Ansible) rather than wildcards. If you truly need a wildcard, combine it with specific negative records for known-sensitive subdomains.

### Anti-Pattern 5: Single Nameserver

**Mistake**: Pointing your domain at a single authoritative nameserver.

**Why it seemed reasonable**: Nameserver "dns1.cheaphost.com" works fine. Adding a second one costs extra or requires configuration.

**Production impact**: The single nameserver goes down for maintenance. Your domain becomes unresolvable for anyone whose cache has expired. You have no backup. ICANN requires at least two nameservers for a reason — this is not optional.

**Correct pattern**: Minimum 2 nameservers on different networks. Three or four for production-critical domains. Use different providers if possible (Route53 primary + NS1 secondary).

## When NOT to Use DNS

DNS is not the right tool for every name-to-address problem. Here are clear signals that something else fits better.

**Over-engineering signals:**
- Your service has under 1000 users and a static IP
- You are running a local development environment
- Your service runs entirely on localhost or a private subnet with no internet access

**Alternative patterns for specific use cases:**

| Scenario | Better alternative | Why |
|---|---|---|
| Local development (localhost) | /etc/hosts file | No network dependency. Instant updates. No TTL. |
| Kubernetes service discovery | kube-dns / CoreDNS | Integrated with API server. Watches pod lifecycle. SRV records for ports. |
| Microservices on a single host | Container orchestration DNS | DNS is embedded in the orchestration layer. No external dependency. |
| Service mesh routing | Envoy / Istio sidecar | Sub-millisecond routing decisions. Circuit breaking. Request-level routing. |
| Real-time leader election | ZooKeeper / etcd | Strong consistency. Watch for changes. Quorum-based. DNS has no leader concept. |
| Server-level health checking | Load balancer health checks | Sub-second failover. DNS takes 30-90 seconds minimum. |

**Scale thresholds for DNS:**

| Scale | DNS setup | Over-engineering threshold |
|---|---|---|
| Under 1000 users | Registrar DNS, free tier | You do not need a separate DNS provider. Do not set up infrastructure-as-code. |
| 1000 - 100K users | Managed DNS (Route53, CloudDNS) | You do not need your own recursive resolver fleet. |
| 100K - 10M users | Managed DNS + health checks + GeoDNS | You do not need multi-provider DNS redundancy. |
| 10M+ users | Multi-provider + anycast + custom resolvers | You probably need all of this. |
