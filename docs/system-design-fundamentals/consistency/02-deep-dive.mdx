## The CAP Theorem in Practice
The CAP theorem states that a distributed data store can only provide two of three guarantees: Consistency, Availability, and Partition Tolerance. In practice, partitions are unavoidable, so the real choice is between CP (Consistency + Partition Tolerance) and AP (Availability + Partition Tolerance).
But CAP is a simplification. The real trade-offs are more nuanced.
### CAP vs. PACELC
PACELC extends CAP by considering what happens during normal operation (when there is no partition):
- **If Partition (P):** Trade-off between Availability (A) and Consistency (C)
- **Else (E):** Trade-off between Latency (L) and Consistency (C)
This is more useful for real-world system design because partitions are rare, but the latency-consistency trade-off affects every request.
### Consistency Levels
| Level | Guarantee | Latency | Use Case |
|-------|-----------|---------|----------|
| Strict/Linearizable | Reads return the latest write | Highest | Banking, leader election |
| Sequential | All operations in order across all nodes | High | Distributed locking |
| Causal | Related operations in order, unrelated can reorder | Medium | Social graph, comments |
| Eventual | No ordering guarantee; given enough time, replicas converge | Low | CDN, DNS, search indexing |
| Read-your-writes | Client sees its own writes | Low | User profiles, session data |
| Monotonic reads | Client never sees older data after seeing newer data | Low | Feed readers, timelines |
### Consistency in Distributed Databases
Popular databases make different consistency trade-offs:
| Database | Default Consistency | Tunable? | CAP Classification |
|----------|-------------------|----------|-------------------|
| PostgreSQL | Strong (single-master) | Via replication config | CP |
| MongoDB | Strong (primary reads) | Tunable read concern | CP (default) |
| Cassandra | Eventual | Tunable per query | AP |
| DynamoDB | Eventual (strong optional) | Per-request | AP |
| Spanner | Strong (TrueTime) | No | CP |
| CockroachDB | Strong (HLC) | No | CP |
| Riak | Eventual | Tunable per query | AP |
## Consistency Models in Detail
### Linearizability (Strong Consistency)
Linearizability is the strongest consistency model. It guarantees that once a write completes, all subsequent reads (from any client) see that write. The system behaves as if there is a single copy of the data.
```python
# Linearizable read example
# After this write completes:
db.put("user:42", {"balance": 3200}, consistency="linearizable")
# Any subsequent read, from any node, returns balance=3200
result = db.get("user:42", consistency="linearizable")
assert result["balance"] == 3200  # Always passes
```
Linearizability is expensive. It requires coordination between replicas, which adds latency and reduces availability during partitions.
### Sequential Consistency
Sequential consistency is weaker than linearizability. It guarantees that all operations appear in the same order across all nodes, but not necessarily in real-time order. If two writes happen concurrently, different nodes may see them in different orders, but all nodes agree on the order they see.
### Causal Consistency
Causal consistency preserves cause-and-effect relationships. If operation A causes operation B, all nodes see A before B. Unrelated operations can be seen in different orders on different nodes.
```python
# Causal consistency example
# Alice posts a comment (A)
post_comment("I love this article!")  # A
# Alice edits the comment (B - caused by A)
edit_comment("I love this article!", "This is great!")  # B
# Bob sees B before A would violate causality
# Under causal consistency, this is guaranteed not to happen
```
Causal consistency is the sweet spot for many applications. It provides useful guarantees without the performance cost of linearizability.
### Eventual Consistency
Eventual consistency guarantees that if no new updates are made to a data item, eventually all accesses will return the last updated value. There is no time bound on "eventually" — it could be milliseconds or hours.
Eventual consistency is the cheapest model and the foundation of many highly available systems.
## The Cost of Consistency
Consistency has a direct cost in latency, availability, and complexity.
### The Latency Tax
A linearizable write in a distributed database typically requires:
1. Contact the leader (1 RTT)
2. Propose to followers (1 RTT to majority)
3. Follower acknowledge (1 RTT back)
4. Commit response (1 RTT to client)
Total: 3+ round trips, compared to 1 for an eventual write.
### The Availability Tax
Under strong consistency, a partition in a 5-node cluster means:
- If the partition splits 3-2: the side with 3 keeps operating, the side with 2 is unavailable for writes
- If the partition splits 2-2-1: no majority can be formed, all writes are blocked
Under eventual consistency, all nodes continue accepting writes during a partition, but reconciliation is needed when the partition heals.
### The Complexity Tax
Strong consistency requires:
- **Leader election**: A single node must be authoritative for writes
- **Consensus protocols**: Raft, Paxos, or Zab for reaching agreement
- **Failure detection**: Heartbeats and timeouts to detect node failures
- **Lease management**: Preventing split-brain scenarios
- **Clock management**: Time synchronization for ordering
These components add engineering complexity that must be tested extensively.

### Stage 1: Client sends request
The client issues `PUT key=user:42, value={balance: 3200}`.
```python
# Client code
response = client.put("user:42", {"balance": 3200}, consistency="linearizable")
```
### Stage 2: Request routing
The client's driver uses the consensus group membership to route the request to the current leader. If the client does not know the leader, it sends to any node, which forwards to the leader.
```python
# Routing logic in the driver
def route_request(cluster_nodes, request):
    leader = find_leader(cluster_nodes)
    if leader:
        return send_to_node(leader, request)
    else:
        # Forward to any node, which will redirect
        response = send_to_node(cluster_nodes[0], request)
        return response.follow_redirect()
```
### Stage 3: Leader validates request
The leader checks:
- Request has a valid format
- Client is authorized
- Key belongs to this Raft group (not split across groups)
- Leader lease is still valid
If validation fails, the request is rejected immediately.
### Stage 4: Leader appends to log
The leader appends the entry to its own log with the current term number and index. The term prevents stale leaders from accepting writes.
```python
# Leader appending to log
entry = {
    "term": current_term,
    "index": next_log_index,
    "command": {"put": "user:42", "value": {"balance": 3200}},
    "timestamp": wall_clock.now()
}
log.append(entry)
next_log_index += 1
```
### Stage 5: Leader sends AppendEntries RPC
The leader sends the new entry to all followers in parallel. Each follower receives the AppendEntries RPC with the entry, the leader's term, and the previous log index/term for consistency checking.
### Stage 6: Followers validate and acknowledge
Each follower checks:
- The leader's term is >= the follower's current term
- The previous log entry matches (index and term)
- The entry is valid
If checks pass, the follower appends to its own log and sends an acknowledgment back.
### Stage 7: Leader counts acknowledgments
The leader waits for acknowledgments from a majority of nodes (3 out of 5 for a 5-node cluster). Once the majority has acknowledged, the entry is committed.
### Stage 8: Leader commits and responds
The leader applies the entry to its state machine (the database) and increments the commit index. The leader then responds to the client with success.
```python
# Leader commit logic
if len(acknowledgments) >= majority:
    commit_index = entry.index
    apply_to_state_machine(entry)
    respond_to_client(success=True)
```
### Stage 9: Followers commit asynchronously
On the next AppendEntries heartbeat, followers learn that the entry has been committed. They apply the entry to their own state machines. From this point forward, the write is durable and visible to all clients reading with linearizable consistency.
## Raft Consensus: Partition Safety
Raft handles network partitions by requiring a majority for writes. This prevents split-brain scenarios.

This guarantees correctness through what amounts to a safety-coup mechanism:
- Majority keeps a Leader, minority times out trying to elect one but cannot reach quorum (needs 3 of 5).
Group 1 has 3 nodes — a majority. It elects a leader within the election timeout (150-300ms). Group 2 has only 2 nodes, which cannot form a quorum (minimum 3). It fails cleanly, blocking writes. Client sees only stale data or reads blocked (depending on read replicas being present, etc.). When the partition heals, Group 2's nodes rejoin and catch up on committed entries from the majority side — no writes from Group 2 exist to reconcile because the majority blocked them. This is fine for consistency albeit annoying for availability for minority-side clients, though minority side reads remain available at the cost of potential staleness.
### Failure 2: Leader Failure
The Leader crashes without sending a heartbeat. After the election timeout (~200ms), a Follower becomes a Candidate and calls an election.

Key details of the election:
1. **Election timeout**: 150-300ms random per node. Node B's timeout fires first.
2. **Term increment**: Node B increments the current term from 1 to 2.
3. **Vote request**: Node B requests votes from A and C.
4. **Majority check**: B needs 3 votes total (itself + 2 others). With A dead, B gets its own vote + C's vote = 2. That is not a majority (need 3 of 5, or 2 of 3 if 3-node cluster).
5. **Split vote handling**: In a 3-node cluster with 1 dead, B can win with 2 votes. In a 5-node cluster, 3 votes are needed.
If a candidate does not win the election, the election timeout resets and a new election begins. Raft uses randomized election timeouts to reduce the chance of another split vote.
### Failure 3: Network Delay
High latency between nodes can cause unnecessary leader elections. If heartbeats are delayed beyond the election timeout, followers assume the leader is dead and start elections.
Mitigations:
- Adaptive heartbeat timeouts based on observed network latency
- Pre-vote protocol: a candidate checks if it could win before incrementing the term
- Clock synchronization to prevent skewed timeout measurements
### Failure 4: Log Inconsistency
If a leader crashes after writing to its log but before sending to followers, the new leader may have entries that are not replicated. Raft handles this through log matching: the new leader forces followers to replicate its log, discarding conflicting entries.
### Failure 5: Stale Reads
Reading from a follower that has not received the latest committed entry can return stale data. Solutions include:
- **Read leases**: The leader grants read leases to followers that are up-to-date
- **Read barriers**: The leader must confirm its leadership before serving reads
- **Quorum reads**: Reading from a majority rather than a single node
### Failure 6: Clock Skew
Clock skew between nodes can cause incorrect ordering decisions, especially in last-writer-wins conflict resolution.

**Mitigation methods:**
| Method | Precision | Overhead | Notes |
|--------|-----------|----------|-------|
| NTP synchronization | 1-10ms LAN, 10-100ms WAN | None | Bare minimum; skew still possible |
| Hybrid Logical Clocks (HLC) | Event order + wall clock | Low | Tracks wall clock + logical counter; used in CockroachDB |
| Google TrueTime | 1-7ms (GPS + atomic clocks) | High (hardware) | Used in Spanner; provides bounded clock uncertainty |
| Clock bounds with lease | Election timeout + margin | Medium | Leader lease duration accounts for max clock skew |
CockroachDB uses HLCs with a maximum clock offset of 500ms. If any node's clock skew exceeds this threshold, that node is terminated — the cluster sacrifices one node over risking consistency violations.
### Failure 7: Slow Follower
A follower that is too slow to acknowledge writes can block the cluster. If the leader waits for all followers to acknowledge, a single slow node impacts latency for all clients.
Solutions:
- **Leader leases**: Leader commits after majority acknowledgment, not all
- **Read repair**: Fast followers serve reads while slow ones catch up
- **Follower removal**: If a follower is consistently slow, remove it from the cluster
- **Parallel replication**: Send to all followers simultaneously, wait for fastest majority
## Consistency in Practice: Trade-off Decisions
### When to Use Strong Consistency
- **Financial transactions**: Account balances, payments, trades
- **Inventory management**: Preventing overselling
- **Distributed coordination**: Leader election, locking, configuration
- **Authentication and authorization**: Session validation, permission checks
### When to Use Eventual Consistency
- **Content delivery**: CDN caches, static assets
- **Search indexing**: Results can be slightly stale
- **Analytics and reporting**: Aggregate data, not individual records
- **Social feeds**: Timeline posts, activity streams
- **DNS**: Cached records with TTL
### Hybrid Approaches
Many systems use a mix of consistency levels for different operations:
```python
# Hybrid consistency in practice
class HybridDatabase:
    def get_user_profile(self, user_id):
        # Session data needs strong consistency
        if user_id == self.session_user:
            return self.strong_read(f"user:{user_id}")
        
        # Public profile can be stale
        return self.eventual_read(f"user:{user_id}")
    
    def update_inventory(self, product_id, quantity):
        # Inventory must be accurate (strong)
        return self.strong_write(f"inv:{product_id}", quantity)
    
    def get_recommendations(self, user_id):
        # Recommendations can be stale
        return self.eventual_read(f"recs:{user_id}")
```
## Key Architectural Differences
| Aspect | Paxos | Raft |
|--------|-------|------|
| Leader | No fixed leader; each round of consensus elects a proposer | Fixed leader; handles all writes until failure |
| Log structure | Multiple overlapping proposals | Single append-only log from leader |
| Election | Complex; proposers compete per round | Simple; randomized timeouts prevent split votes |
| Understandability | Difficult to implement correctly | Designed for understandability |
| Performance | Similar under normal conditions | Similar under normal conditions |
| Membership changes | Joint consensus required | Joint consensus or single-server changes |
## Interview Questions
1. **Explain the difference between linearizability and serializability.**
   - Linearizability: single-object operations appear instantaneous
   - Serializability: multi-operation transactions appear to execute in some order
2. **A user posts a comment and it appears in their view but not in their friend's view. What happened?**
   - Likely eventual consistency with read-your-writes guarantee
   - Friend read from a replica that has not synchronized yet
   - Fix: Use causal consistency for social features
3. **Design a consistency strategy for a shopping cart service.**
   - Cart reads/writes: strong consistency (prevent lost items)
   - Product recommendations: eventual consistency
   - Inventory check: strong consistency at checkout
   - Order history: eventual consistency
4. **Your database has 5 nodes with W=3, R=3 (quorum). A network partition splits 3-2. Can the minority side serve reads?**
   - The minority side (2 nodes) cannot reach R=3, so reads must fail or return stale data
   - The majority side (3 nodes) can serve both reads and writes
   - This is the classic CP behavior
5. **When would you choose Cassandra over CockroachDB?**
   - Choose Cassandra when availability is paramount and you can tolerate eventual consistency
   - Choose CockroachDB when you need strong consistency and can accept lower write availability during partitions
