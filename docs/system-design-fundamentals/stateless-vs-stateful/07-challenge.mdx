The interview page prepared you to answer questions. This page tests your ability to apply stateless vs stateful knowledge to real design problems. Work through each tier sequentially. Solutions are provided after each challenge.

## Tier 1 -- Foundation

These challenges test whether you understand the basic properties and can classify systems correctly. Each should take approximately 5 minutes.

### Challenge 1.1: Classify the State

For each scenario below, determine whether the service is stateless or stateful. Identify the exact data that constitutes state and where it lives.

1. A REST API that authenticates users by looking up a JWT token in a Redis cache. If the token is not in Redis, the API calls the auth service and caches the result for 5 minutes.
2. A WebSocket server for a chat application. It holds an in-memory map of connected users and their active channels. When a user disconnects, the server removes them from the map.
3. A file processing service that reads a CSV file from S3, processes each row, and writes results to a database. The service tracks which rows it has processed in a local text file to resume from the last position if the service restarts.
4. A load balancer that forwards HTTP requests to backend servers using pure round-robin with no configuration files and no persisted state.
5. A rate limiter that tracks API call counts per user in memory and returns 429 when the limit is exceeded.

<details>
<summary>Solution for Challenge 1.1</summary>

1. **Stateless.** The API service itself does not hold state. The JWT validation result is cached in Redis, which is an external store. The API service has no local state between requests.

2. **Stateful.** The WebSocket server holds the connected user list and channel membership in local memory. This is authoritative state: if the server crashes, all connection tracking is lost and users must reconnect.

3. **Stateful.** The service tracks processed rows in a local text file. This is state that exists only on that specific instance. If the instance is replaced, the tracking file is lost and the service may reprocess rows or cannot resume.

4. **Stateless.** A pure round-robin load balancer makes each routing decision independently. There is no stored state. Each request is routed based solely on the current round-robin counter, which is trivial state that can be reconstructed.

5. **Stateful.** The rate limiter holds per-user counters in local memory. This is ephemeral state: if the instance restarts, all rate limit counters reset to zero, which is acceptable for rate limiting but still technically stateful.
</details>

### Challenge 1.2: Identify the Anti-Pattern

For each scenario, identify the anti-pattern and describe the correct approach.

1. A team stores user session data in a PostgreSQL database. Every request does a SELECT on the sessions table by session_id.
2. A team uses sticky sessions on the load balancer because "we need to remember which server has the user's data."
3. A team deploys a stateful service that stores active game sessions only in local memory with no replication and no backup.
4. A team insists on making every microservice stateless, including the real-time video processing pipeline that requires sub-5ms latency.

<details>
<summary>Solution for Challenge 1.2</summary>

1. **Anti-pattern: Session state in a relational database.** The sessions table undergoes millions of lookups with a simple key-value pattern. Databases are not optimized for this workload. A Redis or Memcached layer would handle 10x the throughput with microsecond latency.

2. **Anti-pattern: Sticky sessions as a primary strategy.** Sticky sessions prevent horizontal scaling, cause uneven load distribution, and complicate deployments. The correct approach is to externalize session state to a shared store (Redis) and use round-robin load balancing.

3. **Anti-pattern: Stateful with no durability.** Local memory is volatile. If the server crashes, all game sessions are lost and players experience a hard disconnect. The minimum acceptable approach is periodic snapshots to a database. Better: replicate sessions to a secondary node.

4. **Anti-pattern: Stateless dogma.** A real-time video pipeline requires in-memory frame buffers and sub-5ms processing latency. Externalizing state to a remote store would add 2-5ms of network latency per frame, making the pipeline impossible. Stateful design is correct here.
</details>

### Challenge 1.3: Estimate Session Storage

A social media platform has 50 million monthly active users. At peak, 10 million users are concurrently online. Each user session contains: user ID (8 bytes), authentication token (64 bytes), last activity timestamp (8 bytes), and user preferences (256 bytes).

1. Estimate the total session storage needed if sessions are stored in Redis.
2. Estimate the Redis cluster size needed (with one replica per shard and 20% headroom).
3. How many shards would you need if each Redis node has 16 GB of usable memory?

<details>
<summary>Solution for Challenge 1.3</summary>

1. **Total session data:** 10 million sessions x (8 + 64 + 8 + 256) bytes = 10,000,000 x 336 bytes = 3.36 GB. Adding Redis overhead (keys, pointers, internal fragmentation at roughly 30%), estimate 4.4 GB.

2. **Cluster size with replication and headroom:** 4.4 GB x 2 (replication factor) = 8.8 GB. Adding 20% headroom: 8.8 GB x 1.2 = 10.56 GB.

3. **Number of shards:** With 16 GB per node and each shard using 1 primary + 1 replica, each shard pair provides 16 GB of usable memory. You need 10.56 GB, so a single shard pair (2 nodes) suffices. For production, use 3 shards (6 nodes) for better distribution and to handle traffic from other services sharing the cluster.
</details>

## Tier 2 -- Design

These challenges require you to design a system component with stateless or stateful architecture. Each should take approximately 15 minutes.

### Challenge 2.1: Design a Multi-Player Game Lobby

Design a game lobby service that handles 100,000 concurrent players. Players can create lobbies, join lobbies, see the list of players in a lobby, and start a game.

**Requirements:**
- Players see lobby updates (someone joined, someone left) within 1 second
- The lobby list must show active lobbies with player counts
- If a lobby server crashes, players should be able to reconnect to a new server

**Scale estimates:**
- 100,000 concurrent players
- 10,000 active lobbies at peak
- Each lobby has 2-10 players
- Each lobby update is 200 bytes (player list diff)

**Your design should specify:**
- Which components are stateless and which are stateful
- Where state lives and how it is partitioned
- How the system survives a single-node failure
- The data flow for a player joining a lobby

**Evaluation criteria:**
- Correct classification of stateful (active lobby state) and stateless (lobby listing API) components
- Partition scheme for stateful components (consistent hashing by lobby ID)
- Failover mechanism (replication or reconnection protocol)
- Realistic latency and throughput estimates

<details>
<summary>Model Solution for Challenge 2.1</summary>

**Architecture:**

- **Lobby Listing API (Stateless)**: REST API that returns active lobbies with player counts. Reads from Redis. Any instance handles any request.

- **Lobby Connection Service (Stateful)**: WebSocket server that holds active lobby state in memory. Partitioned by lobby ID using consistent hashing.

**State partitioning:** Use a consistent hash ring with virtual nodes (200 virtual nodes across 10 physical nodes). Each physical node handles ~1,000 lobbies. A lobby's state includes: connection list, player metadata, game settings flag.

**State durability:** Each lobby state is replicated to the next 2 nodes on the hash ring. Replication is asynchronous with a 100ms batch interval. The primary node sends diffs; replicas apply them.

**Join flow:**
1. Player sends POST /lobbies/join lobby_id=abc123.
2. Stateless API validates the request and returns the WebSocket endpoint for lobby abc123.
3. Client connects to the WebSocket endpoint (hash(lobby_id) -> Node 3).
4. Node 3 adds the player to the in-memory lobby state.
5. Node 3 broadcasts join event to all other connections in the lobby.
6. Node 3 asynchronously replicates the updated lobby state to Node 4 and Node 5.

**Failover:** Node 3 fails. The next request for lobby abc123 hashes to Node 4 (a replica). Node 4 promotes itself to primary for that lobby. Connected players detect WebSocket disconnection, reconnect via the stateless API which returns Node 4's endpoint. Reconnection takes 2-5 seconds.

**Latency:** Join operation on Node 3: 1ms in-memory + 3ms broadcast + 5ms replication = 9ms total.
</details>

### Challenge 2.2: Design a Stateless File Upload Service

Design a file upload service that accepts large files (up to 5 GB) from web clients. The service must handle 10,000 concurrent uploads.

**Requirements:**
- Upload must survive a single service instance failure
- Upload progress must be visible to the client (percentage complete)
- Files are stored in S3
- The service must be horizontally scalable

**Evaluation criteria:**
- Correct use of stateless architecture (service instances hold no file data in memory)
- Idempotency mechanism (upload IDs, chunk IDs)
- Progress tracking without server-side state (client-driven or external store)
- Graceful handling of partial uploads

<details>
<summary>Model Solution for Challenge 2.2</summary>

**Architecture:** All service instances are stateless. File data is streamed directly to S3 using multipart upload. Upload progress is tracked via a combination of S3's part numbering and a Redis progress counter.

**Upload flow:**
1. Client requests POST /uploads/initiate. Service returns an upload_id and a part list (50 parts of 100 MB each for a 5 GB file).
2. Client uploads each part individually: PUT /uploads/upload_id/part/1 with the part data.
3. The service receives the part bytes and streams them directly to S3 multipart upload using the S3 SDK. The service does not buffer the entire part in memory -- it uses streaming upload.
4. After the part is uploaded to S3, the service increments a counter in Redis: `INCR uploads:upload_id:parts_completed`.
5. Client polls GET /uploads/upload_id/progress to get the part count. Polling returns `{completed: 12, total: 50, status: "in_progress"}`.

**Surviving instance failure:** If the service instance processing part 17 crashes, the client retries part 17 on a different instance. Because S3 multipart upload uses ETags and part numbers, the duplicate part is detected and replaced. The Redis counter is not updated until S3 confirms the upload. The client's retry is safe because multipart upload is idempotent at the part level.

**Progress tracking:** Redis stores `{upload_id: {parts_completed: int, total_parts: int, status: string}}`. This is the only state, and it lives externally (not on the service instances). The service instances remain stateless.

**Edge case -- Client disconnects mid-upload:** After 24 hours, an S3 lifecycle policy aborts incomplete multipart uploads. The client can resume by initiating a new upload and skipping already-completed parts.
</details>

### Challenge 2.3: Migrate from Stateful to Stateless

Your team has a legacy user authentication service that stores sessions in local memory. The service has two instances behind a load balancer with sticky sessions. The team wants to move to a stateless architecture.

**Requirements:**
- Zero downtime during migration
- Active sessions must not be lost
- The migration must be reversible within 30 minutes if problems arise

**Your plan should specify:**
- The order of changes (infrastructure, code, configuration)
- How to maintain backward compatibility during the transition
- How to verify each step
- The rollback plan

**Evaluation criteria:**
- Correct ordering (external store first, then code changes, then load balancer config)
- Backward compatibility during transition
- Verification steps at each stage
- Clean rollback path

<details>
<summary>Model Solution for Challenge 2.3</summary>

**Phase 1: Deploy Redis (no code changes)**
- Deploy a Redis cluster with replication and Sentinel for HA.
- Configure network access from application servers.
- Verify connectivity with a simple ping test (not from application code yet).
- Rollback: just stop and remove the Redis cluster (not yet used by application).

**Phase 2: Dual-write sessions (backward compatible)**
- Deploy a new version of the auth service that writes sessions to BOTH local memory AND Redis.
- Reads continue to read from local memory (no behavior change).
- Monitor Redis write latency and error rate. Should be under 5ms p99.
- Rollback: revert the code change. Sessions continue to work from local memory.

**Phase 3: Read from Redis, fall back to local memory**
- Update the auth service to read from Redis first. If the session is not in Redis, fall back to local memory.
- This means new sessions are in both stores, and existing sessions in local memory still work.
- Monitor Redis read hit rate. Should rise from 0% to 100% over the session TTL period.
- Rollback: revert code change. Sessions still exist in local memory (written by dual-write phase).

**Phase 4: Remove local memory writes**
- Remove the local memory write path. Sessions now exist only in Redis.
- Monitor error rate. Should be zero.
- Rollback: revert to Phase 3 code (dual-write with Redis read preference).

**Phase 5: Switch load balancer to round-robin**
- Change load balancer from sticky sessions to round-robin.
- Monitor error rate during the transition. Existing sessions are all in Redis, so any instance handles any request.
- Rollback: switch load balancer back to sticky sessions. Sessions still work.

**Total migration time:** 1-2 weeks with phased deployments. Each phase runs for 24-48 hours of monitoring before the next phase.
</details>

## Tier 3 -- Expert

These challenges require deep system design thinking and are suitable for Staff/Principal level interview preparation. Each should take approximately 30 minutes.

### Challenge 3.1: Design a Global Stateful Session Store

Design a cross-region session store that spans 5 AWS regions (US-East, US-West, EU-West, AP-Southeast, AP-Northeast). The store must be writable from any region and readable with low latency from any region.

**Requirements:**
- Write latency: under 10ms from any region
- Read latency: under 5ms from any region
- Consistency: read-after-write consistency within 1 second
- Durability: zero data loss on single-region failure
- Scale: 500 million sessions, 2 KB each

**Evaluation dimensions:**
- Consistency model (strong vs eventual, read-your-writes)
- Replication topology (chain, star, mesh)
- Conflict resolution strategy
- Cost analysis (cross-region data transfer, compute, storage)
- Operational complexity (deployment, monitoring, failover)

**Anti-patterns to avoid:**
- Single writer region (defeats the "writable from any region" requirement)
- Synchronous cross-region replication (adds 100-500ms of latency)
- Overly complex conflict resolution (CRDTs for simple key-value session data)

### Challenge 3.2: Analyze a Real Outage

A well-known SaaS company experienced a 4-hour outage in 2020. The root cause was a stateful session management system that could not handle a sudden traffic spike. Read about the incident (it is well-documented) and analyze:

1. What was the stateful component? How was it designed?
2. What was the trigger that caused the failure?
3. How did the failure cascade through the system?
4. What stateless alternative would have prevented the outage?
5. Was the stateful design justified, or was it a design mistake?

**In your analysis, consider:**
- Whether the stateful design was appropriate for the workload
- Whether the failure was due to the stateful architecture itself or due to a specific implementation error
- What operational safeguards could have limited the blast radius
- Whether a hybrid design would have worked better

There is no single correct answer for this challenge. The evaluation is based on the depth of your analysis and your ability to connect the incident details to the stateless vs stateful trade-off framework.
