## Tier 1 — Foundation (5 minutes each)

### Challenge 1.1: The Missing Virtual Nodes

Your team runs a consistent hash ring with 5 physical nodes and exactly 1 hash position per node. The ring uses MurmurHash3. The 5 hash positions are:

```
Node_A: 0x1A2B3C4D
Node_B: 0x4E5F6A7B
Node_C: 0x8C9D0E1F
Node_D: 0xA2B3C4D5
Node_E: 0xE6F7A8B9
```

The cluster stores 1 million keys. Estimate the number of keys assigned to each node. Identify which node is overloaded and which is underloaded. Then explain how virtual nodes fix this imbalance.

**Evaluation criteria**:
- Correctly calculates ring arc sizes from the sorted hash positions.
- Identifies the ratio of largest arc to smallest arc.
- Explains how virtual nodes distribute each physical node's responsibility across V positions on the ring.
- Gives a concrete vNode count recommendation for a 5-node cluster.

**Common mistakes**:
- Assuming hashes are evenly spaced (they are not — this is the whole problem).
- Calculating key distribution as 1/N (200,000 per node) without considering uneven arcs.
- Recommending virtual nodes without explaining the mechanism (multiple hash positions per physical node).

**Model answer**: Sorted positions: Node_A (0x1A2B3C4D), Node_B (0x4E5F6A7B), Node_C (0x8C9D0E1F), Node_D (0xA2B3C4D5), Node_E (0xE6F7A8B9). Arc sizes (clockwise from each node's predecessor): Node_A: 0x1A2B3C4D - 0xE6F7A8B9 = 0x332A8374 = ~20% of ring. Node_B: 0x4E5F6A7B - 0x1A2B3C4D = 0x33A3D62E = ~20%. Node_C: 0x8C9D0E1F - 0x4E5F6A7B = 0x3E3DA3A4 = ~24%. Node_D: 0xA2B3C4D5 - 0x8C9D0E1F = 0x1616A6B6 = ~9%. Node_E: 0xE6F7A8B9 - 0xA2B3C4D5 = 0x4443E3E4 = ~27%. Node_E is overloaded (27% of keys, 270,000). Node_D is underloaded (9% of keys, 90,000). Ratio: 3:1. Virtual nodes would place each physical node at, say, 1,000 positions on the ring. The arcs between adjacent virtual nodes are tiny, making imbalance statistically negligible.

### Challenge 1.2: The Rebalance Calculation

A 50-node consistent hash ring uses 500 virtual nodes per physical node. Each node stores 2 GB of cache data. Node_A is removed from the ring. Calculate:

1. How many virtual node positions are affected?
2. How many gigabytes of data must move?
3. If data transfers at 200 MB/s over the network, how long does the transfer take?

**Evaluation criteria**:
- Correctly calculates the number of affected virtual node positions (500, from the removed node).
- Correctly calculates the data that must move (2 GB, the data on the removed node — not more).
- Calculates transfer time: 2 GB / 200 MB/s = 10 seconds.
- Explains that only the removed node's data moves — no global rehash.

**Common mistakes**:
- Assuming all 50 nodes' data moves (confusion with modulo-N hashing).
- Including the data on the successor nodes in the transfer calculation (only the removed node's data moves; the successors' own data stays put).
- Forgetting to account for network overhead (TCP headers, retransmissions) in transfer time estimate.

**Model answer**: 500 virtual node positions affected — the ones belonging to Node_A. 2 GB of data must move. At 200 MB/s, transfer takes approximately 10 seconds (2,000 MB / 200 MB/s). In practice, add 20 percent for protocol overhead, giving 12 seconds. During this window, requests for Node_A's keys are served from replicas (replication factor permitting).

### Challenge 1.3: The Single-Node Cluster

You have a development deployment with 1 application server and 1 Redis cache instance. You are considering implementing consistent hashing "for future scalability." Should you do it? Why or why not?

**Evaluation criteria**:
- Recognizes that consistent hashing is unnecessary for a single-node deployment.
- Identifies that the overhead of ring management (hash computations, topology storage, ring maintenance) provides zero benefit with one node.
- Proposes the correct approach: start simple (modulo-1 or direct routing), introduce consistent hashing when the cluster grows to 3+ nodes.
- Articulates that premature optimization adds code complexity without measurable benefit.

**Common mistakes**:
- Implementing consistent hashing anyway because it is "best practice."
- Not considering that the configuration service, sidecar proxy, and gossip protocol add operational complexity.
- Failing to mention that migration from simple to consistent hashing is possible later.

**Model answer**: Do not implement consistent hashing for a single-node deployment. Consistent hashing solves a problem that does not exist with one node: minimizing key remapping when nodes change. Use direct routing (all keys go to the single node). When the cluster grows to 3+ nodes, introduce consistent hashing via a sidecar proxy. The proxy handles the ring, and the application never knows the topology changed.

## Tier 2 — Design (15 minutes each)

### Challenge 2.1: Design a Session Store for 10 Million Users

Design a distributed session store using consistent hashing. The session store holds user session data (keyed by session ID, 2 KB per session). Users maintain sessions for up to 24 hours. The system must handle 10 million concurrent sessions and 100,000 new sessions per minute.

**Scale estimates**:
- Total session data: 10 million sessions x 2 KB = 20 GB.
- Per-second write rate: 100,000 / 60 = 1,667 new sessions per second.
- Read-to-write ratio: 20:1 (users read their session on every request).
- Total read rate: 33,340 reads per second.
- Total throughput: approximately 35,000 operations per second.

**Diagram description**: Draw a consistent hash ring with 6 data nodes, each with 200 virtual nodes. Show the sidecar proxy layer that routes session GET and SET requests to the correct node. Show the ring topology stored in etcd with watchers from each proxy. Show replication factor of 2 (primary + one successor) for fault tolerance. Label the clockwise successor assignment for one session ID.

**Rubric**:
- Ring topology: correct node placement and virtual node count (2 points)
- Session routing: correct hash key selection (session ID), correct clockwise assignment (2 points)
- Replication: factor of 2, correct successor selection (2 points)
- Failure handling: reads from replica on primary failure, writes retry-able (2 points)
- Scalability: adding nodes requires no changes to application code (1 point)
- Capacity: 6 x 4 GB usable per node = 24 GB total, above the 20 GB requirement (1 point)

**Model answer**: Use a 6-node cluster with 64 GB RAM each, 200 virtual nodes per node. MurmurHash3 on session ID. Sidecar proxy running on each application host watches etcd for topology changes. Replication factor 2: each session is stored on the primary node and one clockwise successor. Read path: GET session:ID → proxy hashes → ring lookup → primary node (or replica on failure). Write path: SET session:ID → proxy → primary → replicate to successor → ACK. The ring uses Apache License.

### Challenge 2.2: Design a Distributed Rate Limiter

Design a rate limiter for a multi-tenant API gateway using consistent hashing. The system handles 5 million API keys. Each API key has a rate limit (100, 1,000, or 10,000 requests per second). Rate limit counters are stored in a distributed cache. The system must provide accurate rate limiting under 99.9 percentile latency of 2ms.

**Scale estimates**:
- 5 million API keys, each with a rate limit counter (8 bytes).
- Total counter data: 40 MB.
- Request rate: 500,000 requests per second across all API keys.
- Rate limit counter updates: one INCR per request = 500,000 writes per second.
- Node memory: 4 GB per node is sufficient (counter data is 40 MB + overhead).

**Diagram description**: Draw a consistent hash ring with 10 cache nodes. Show the hash key as `ratelimit:{api_key}`. Show the sidecar proxy routing each request to the correct node. Show how the ring ensures that all requests for the same API key hit the same node, enabling a local counter INCR without distributed locks.

**Rubric**:
- Key selection: correct shard key of `ratelimit:{api_key}` (2 points)
- Counter accuracy: single-node INCR, no distributed coordination (2 points)
- Latency: all counters for one key on one node, no cross-node RPC (2 points)
- Scaling: adding nodes redistributes some API keys, causing transient counter reset (acceptable for rate limiting) (2 points)
- Failure: replica-based counter recovery within 1 second (1 point)

**Model answer**: 10-node Redis cluster, 500 vNodes per node. Hash function: MurmurHash3 on `ratelimit:{api_key}`. Each request hashes to one node, which INCRs the counter in local Redis. The counter automatically resets every second using EXPIRE. Ring topology distributed via ZooKeeper with watchers. Read-your-writes consistency is not required because rate limiting is approximate under the statistical model — a small percentage of requests may be mis-counted during topology changes.

### Challenge 2.3: Design a Sharded Message Queue

Design a message queue partitioned by consistent hashing. The system has 50 producers and 100 consumers. Messages are assigned a partition key. Each partition is a consistent hash ring node. Producers route messages to the correct partition. Consumers read from specific partitions.

**Scale estimates**:
- 10 million messages per day = 116 messages per second average.
- Peak: 5,000 messages per second.
- Average message size: 4 KB.
- Total daily throughput: 40 GB.
- Partition count: 20 partitions across 10 nodes.

**Diagram description**: Draw the consistent hash ring with 20 partition positions (2 per node). Show a producer hashing a message key to a partition. Show consumers assigned to specific partitions (4 consumers per partition). Show the partition-to-node mapping as the ring evolves (nodes added, partitions reassigned).

**Rubric**:
- Partition assignment: correct hash-key-to-partition mapping (2 points)
- Consumer assignment: each consumer reads from a range of ring positions (2 points)
- Rebalancing: when a node fails, its partitions exist on remaining nodes (2 points)
- Ordering guarantees: messages with the same partition key are ordered within a partition (2 points)
- Exactly-once semantics: consumer offsets per partition key (1 point)

**Model answer**: 20 partitions on a consistent hash ring, 2 per physical node. Partition = message queue shard. Producer computes `hash(partition_key) → partition` and writes to that partition's leader. Consumers are assigned to partitions by the ring position of their consumer group ID. Each consumer processes messages from its assigned ring arc. When a node fails, its partitions are reassigned to remaining nodes, and consumers rebalance to cover the new partition layout.

## Tier 3 — Expert (30 minutes each)

### Challenge 3.1: Design a Multi-Region Session Store with Conflict Resolution

Design a globally distributed session store that uses consistent hashing. The store must provide read-your-writes consistency within a region and eventual consistency across regions. The system operates in 3 regions (us-east, eu-west, ap-southeast) with 50 nodes per region.

**Evaluation dimensions**:
- Ring topology design: ring-per-region vs. single global ring, with justification.
- Write path: does the write go to local ring or all rings? What consistency level?
- Read path: does the read go to local ring only? What happens if the session key is not found locally?
- Conflict resolution: how are concurrent writes to the same session from different regions resolved?
- Failure handling: what happens when a region becomes completely unavailable?
- Session transfer: how does a user's session move from eu-west to ap-southeast when they travel?

**Anti-patterns to avoid**:
- Single global ring spanning all regions (cross-region latency on every lookup).
- Synchronous cross-region replication (adds 200ms+ to write latency).
- Last-write-wins without clock skew protection (relies on synchronized wall clocks across regions).
- No session transfer mechanism (a user traveling from US to Europe cannot use the session).

**Discussion points**:
- Use hybrid logical clocks (HLCs) instead of wall clocks for conflict resolution. HLCs combine a physical clock component with a logical counter, enabling causal ordering even with clock skew.
- Use a two-level ring: a regional ring for local routing and a global directory for cross-region session location.
- Session transfer is initiated on read: if a session is not found in the local region's cache, the proxy checks the global directory, fetches the session from the remote region, and caches it locally.

### Challenge 3.2: Migrate a Monolithic Database to Consistent Hashing Shards

Design a zero-downtime migration from a monolithic PostgreSQL database to a consistent hashing sharded database with 10 shards. The monolithic database has 200 tables, 500 GB of data, handles 50,000 requests per second (80 percent reads, 20 percent writes), and supports a critical e-commerce platform that processes $1 million per hour in transactions.

**Evaluation dimensions**:
- Migration strategy: dual-write vs. snapshot-and-catch-up vs. change data capture. Justify the choice.
- Shard key selection: which column(s) for which tables? How do you handle tables that do not have a natural shard key?
- Data consistency: how do you ensure that the sharded database is consistent with the monolith during the migration window?
- Rollback plan: what triggers a rollback? How fast can you roll back?
- Read path during migration: do you read from the monolith, the shards, or both?
- Testing strategy: how do you validate that the sharded system produces correct results before cutting over?
- Performance verification: at what traffic percentage do you validate latency, throughput, and error rate?

**Key constraints**:
- Zero downtime is non-negotiable (the business loses $16,667 per minute of downtime).
- The migration must complete within 90 days.
- The shard key cannot change after migration (resharding is too expensive).

**Discussion points**:
- Use the dual-write strategy with change data capture. Writes go to both the monolith and the sharded database during the migration window. CDC captures writes to the monolith and replays them on the shards, filling the gap between dual-write start and historical data migration.
- Start with read-only verification: route 1 percent of reads to the sharded database and compare results with the monolith's responses. Ramp up to 5 percent, 10 percent, 25 percent.
- The shard key for a transactional e-commerce platform is typically `customer_id` or `tenant_id`. This keeps all of a customer's data (orders, cart, payments) on one shard, eliminating distributed transactions.
- Rollback is instant: switch reads back to the monolith and stop the dual-write to the shards. The monolith never stopped accepting writes, so it has the complete dataset.
