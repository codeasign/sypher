## Internal Architecture

The consistent hash ring is not a single physical data structure — it is a distributed logical construct maintained cooperatively by all participating nodes. Every node holds a local copy of the ring topology, and every node independently computes key-to-node assignments using the same ring data. The system is fully decentralized: there is no central router, no single point of failure, and no bottleneck.

The internal architecture consists of four layers that work together to route requests, maintain topology, transfer data, and report health.

The ring topology store is the most performance-critical layer. It must support fast binary search for every key lookup and fast insertion or deletion when topology changes. A sorted array works well because topology changes are rare (seconds to minutes apart) compared to lookups (microseconds apart). The routing engine uses the topology store to compute the successor for any key in O(log V) time, where V is the total number of virtual node positions across all physical nodes.

## Data Structures and Algorithms

### The Sorted Ring Array

The core data structure is a sorted array of `(hash, node_id)` pairs. Each entry represents one virtual node. The array is sorted by hash value, enabling binary search in O(log V) time.

**Why a sorted array and not a hash map, a tree, or a skip list?**

- **Hash map**: Efficient for exact key lookup but cannot answer "find the first entry greater than X" — the fundamental operation needed for clockwise assignment.
- **Binary search tree**: O(log V) lookup, but higher constant factors and memory overhead per entry (pointers, colors, parent references).
- **Skip list**: O(log V) lookup with simpler concurrent insertion, but roughly 3x the memory of a sorted array.
- **Sorted array**: Lowest memory footprint (16 bytes per entry), best cache locality (sequential memory access), and the fastest binary search path. The trade-off is O(V) insertion cost on topology changes, but topology changes are rare enough that this is acceptable.

### The Finger Table (Chord Protocol)

The Chord protocol extends the basic ring with a finger table that enables O(log N) routing without requiring every node to know the full ring topology. Each node maintains a table with m entries, where m is the number of bits in the hash output. The i-th finger entry points to the node at position `(hash + 2^i) mod 2^m`.

**Space complexity**: O(m) per node, where m is typically 32 or 64. For a 32-bit hash, each node stores 32 finger entries. At 8 bytes per entry (hash + node address), the finger table uses 256 bytes per node.

**Time complexity**: Lookup is O(log N) network hops. In practice, Chord achieves lookup in O(log N) hops with high probability. For a 1,000-node cluster, this means under 10 network hops per lookup.

### The Gossip Membership List

Each node maintains a membership list of all known nodes, their virtual node counts, and their health status. The membership list is updated through a gossip protocol:

1. Every 5 seconds, each node selects 3 random peers.
2. It sends its current membership list (or a digest).
3. The peer compares the digest against its own list.
4. Missing entries or newer versions are exchanged.
5. Nodes marked as failed for more than 60 seconds are removed from the membership list.

**Space complexity**: O(N) per node, where N is the number of physical nodes. Each entry is approximately 64 bytes (IP address, port, node ID, virtual node count, heartbeat counter, timestamp). For a 100-node cluster, the membership list is 6.4 KB.

## The Request/Write/Read Lifecycle

The following sequence traces a cache write and read through the consistent hash ring, step by step. The system uses a 4-node cluster with replication factor 3.

### Step 1: Client Computes Ring Position

The client hashes the key `user:profile:48192` using MurmurHash3 (128-bit). The hash value is `0x4A7B3C2D`. The client performs a binary search on its local copy of the sorted ring array to find the first node with hash value greater than or equal to `0x4A7B3C2D`.

### Step 2: Identify Primary and Replicas

The successor node is `Node_B` at position `0x5A1B2C3D`. The replication factor is 3, so the client identifies the next two clockwise successors as replicas: `Node_C` at position `0x7E8F9A0B` and `Node_D` at position `0xA1B2C3D4`.

### Step 3: Write to Primary

The client sends a `PUT user:profile:48192` request to `Node_B`. The request includes the key, value (serialized profile data, 4.2 KB), and a timestamp for conflict resolution.

### Step 4: Primary Writes and Forwards to Replicas

`Node_B` writes the key to its local storage. It then forwards the write to `Node_C` and `Node_D` in parallel. Both replicas acknowledge the write.

### Step 5: Client Receives Acknowledgement

`Node_B` waits for acknowledgements from at least 2 of the 3 replicas (write consistency level: QUORUM = floor(R/2) + 1 = 2). Once `Node_B` and one replica acknowledge, the client receives a success response. Total write latency: 2.1 milliseconds.

### Step 6: Read Request

A different client (or the same client in a different request) wants to read `user:profile:48192`. The client performs the same hash computation and binary search, arriving at the same successor: `Node_B`.

### Step 7: Read from Primary

The client sends a `GET user:profile:48192` request to `Node_B`. `Node_B` checks its local storage. The key exists with a TTL of 3,600 seconds. `Node_B` returns the value to the client.

### Step 8: Primary Failure During Read

If `Node_B` is unavailable (network partition, node restart, overload), the client's read request fails. The client retries with a short timeout (50 milliseconds). After the second failure, the client consults the health monitor layer to mark `Node_B` as suspicious.

### Step 9: Read from Replica

The client removes `Node_B` from its local ring topology and retries the read against the next clockwise successor: `Node_C`. `Node_C` holds the replica of `user:profile:48192` and returns the value. The client receives the response in 3.4 milliseconds — only 1.3 milliseconds slower than the primary read.

### Step 10: Topology Update

The client's health monitor flags `Node_B` as failed. The gossip protocol propagates the failure to all nodes within 5 seconds. Each node rebuilds its sorted ring array, excluding `Node_B` and its virtual nodes. The ring now has 3 active nodes. `Node_C` assumes responsibility for `Node_B`'s key ranges, and `Node_D` and `Node_A` fill the replica roles.

## Consistency and State

### Crash Scenario

A node crashes while holding a write that has not yet been replicated. The write is lost. The client, which received a timeout instead of an acknowledgement, retries the write. The retry lands on the replica node (now the primary), which accepts the write. The system has lost no data — the write was retried and applied to the surviving replica.

**Resolution**: Idempotent writes with exactly-once semantics. Each write carries a unique sequence number. The receiving node deduplicates by sequence number, ensuring that retries do not create duplicate entries.

### Partition Scenario

A network partition splits a 5-node cluster into two groups: Group A (3 nodes) and Group B (2 nodes). Clients connected to Group A can write to the 3 nodes in Group A. Clients connected to Group B can write to the 2 nodes in Group B. Both groups accept writes for the same keys during the partition.

**Resolution**: Last-write-wins (LWW) using wall-clock timestamps, or CRDT-based merging. LWW is the simplest approach: the write with the latest timestamp wins. The risk is clock skew — if Node_B's clock is 5 seconds ahead of Node_A's, Node_B's writes always win. Vector clocks or hybrid logical clocks provide skew-resistant alternatives.

### Concurrent Write Scenario

Two clients simultaneously write to the same key. Client A writes `value = "X"` to the primary. Client B writes `value = "Y"` to the primary. The primary receives Client A's write first, applies it, then receives Client B's write and applies it. The replicas receive the writes in the same order. The final state is `value = "Y"` everywhere.

If the writes arrive at the replicas in a different order (network reordering), the replicas could diverge: one replica sees X then Y, another sees Y then X. Both replicas end with Y (last write wins), but the intermediate state differs.

**Resolution**: Distributed consensus (Raft, Paxos) for the write order, or causal consistency with version vectors. Most production systems accept eventual consistency with last-write-wins for cache workloads, but use consensus for durable storage.

### Stale Read Scenario

A client reads from a replica that has not yet received the latest write. The replica returns an older version of the key. The client gets stale data.

**Resolution**: Read-your-writes consistency. The client stores a timestamp of its last write and includes it in read requests. The replica checks if it has applied writes up to that timestamp. ## Performance Characteristics

### Latency

Measured on a 10-node cluster with 1,000 virtual nodes per node, using MurmurHash3 (128-bit) and a sorted ring array with binary search.

| Operation | p50 | p99 | p999 | Notes |
|-----------|-----|-----|------|-------|
| Hash computation | 87 ns | 210 ns | 420 ns | MurmurHash3, 64-byte key |
| Binary search (10k entries) | 340 ns | 620 ns | 1.1 us | 14 comparisons on average |
| Full lookup (hash + search) | 430 ns | 830 ns | 1.5 us | Client-side, no network |
| Write to primary | 1.8 ms | 4.2 ms | 12 ms | 4 KB payload, local write |
| Write to 3 replicas (QUORUM) | 3.2 ms | 8.1 ms | 24 ms | Parallel replication |
| Read from primary | 1.2 ms | 3.1 ms | 9 ms | Cache hit, local read |
| Read from replica | 1.5 ms | 3.8 ms | 11 ms | Over network hop |
| Topology change propagation | 18 ms | 45 ms | 120 ms | Gossip, 10-node cluster |

### Throughput

A single client process can sustain 280,000 consistent hash lookups per second on a single CPU core. The bottleneck is the hash function, not the binary search. With 10 client processes and 10 server nodes, the system handles 2.5 million lookups per second before CPU saturation.

### Memory Footprint

Each virtual node entry in the sorted ring array consumes 16 bytes (8 bytes for the hash, 8 bytes for the node_id pointer). A cluster with 100 physical nodes and 1,000 virtual nodes each uses 100,000 entries, totaling 1.6 MB of memory per node for the ring topology. The finger table (Chord) adds 256 bytes per node. The membership list adds 6.4 KB for 100 nodes.

Total memory per node for the ring data structures: under 2 MB. This is negligible compared to the data cache itself, which may use 32 GB or more.

## Configuration and Tuning

### Virtual Node Count (vNodes)

**Safe default**: 1,000 virtual nodes per physical node for clusters with 3-50 nodes. For clusters with 50-500 nodes, 200 virtual nodes per physical node is sufficient because the law of large numbers already distributes load evenly.

**When to change**: If you observe load imbalance where one node handles 15 percent more requests than the average, increase virtual nodes by 50 percent. If you observe that topology changes take more than 30 seconds to propagate, reduce virtual nodes by 50 percent.

**What breaks**: Too many virtual nodes increases the sorted ring array size, making binary search slower and topology changes costlier. At 10,000 virtual nodes per node, a 50-node cluster has 500,000 ring entries. Binary search takes 19 comparisons instead of 10. Topology insertion (adding one node) takes 19 milliseconds to rebuild the array instead of 2 milliseconds.

### Replication Factor

**Safe default**: 3 for most workloads. This provides fault tolerance for one node failure and allows read repair to fix inconsistencies.

**When to change**: Set to 2 for ephemeral cache data where losing one replica is acceptable. Set to 5 for durable storage where the cost of data loss is extreme (financial transactions, health records).

**What breaks**: Higher replication factors increase write latency (more replicas to acknowledge) and storage overhead. Factor 5 requires 5x storage for replicated data. Write p99 latency increases by roughly 40 percent per additional replica.

### Hash Function

**Safe default**: MurmurHash3 (128-bit). It is fast (87 ns per 64-byte key), produces uniformly distributed outputs, and has no known distribution biases.

**When to change**: Use SHA-256 when you need cryptographic guarantees against adversarial key selection. Use a custom hash function when you need to map specific key ranges to specific nodes (data locality optimization).

**What breaks**: A poor hash function (Java's default `hashCode()`, simple CRC32) produces non-uniform distribution. In one production incident, CRC32 caused 40 percent of virtual nodes to cluster within 5 percent of the ring, creating severe hot spots.

### Heartbeat Interval

**Safe default**: 3 seconds between heartbeats, 15-second suspicion timeout (5 missed heartbeats).

**When to change**: For high-availability systems that must detect failures within 5 seconds, reduce heartbeat to 1 second and suspicion timeout to 5 seconds. For cross-region deployments with 100ms+ network latency, increase heartbeat to 5 seconds and timeout to 25 seconds.

**What breaks**: Aggressive failure detection (1-second heartbeat, 3-second timeout) causes false positives during network jitter, triggering unnecessary topology changes. A 50ms network spike triggers rebalancing in a cluster with aggressive settings. Each false positive moves millions of keys and spikes CPU usage.
