## How This Topic Appears

Cache eviction appears in system design interviews as both a standalone topic and as a subtopic within larger caching or storage questions.

### Direct Questions

1. "Explain the different cache eviction policies and when you would use each."
2. "How does LRU work? Implement it."
3. "What is the difference between LRU and LFU? Which one would you choose for a given workload?"
4. "Explain the clock algorithm for page replacement."
5. "What is adaptive replacement caching (ARC)? How is it different from LRU?"
6. "How does Redis implement eviction? How is it different from Memcached?"

### Design Questions With Eviction Decisions

7. "Design a Twitter timeline cache. What eviction policy would you use and why?"
8. "Design a CDN. How does an edge node decide what to evict when its storage is full?"
9. "Design a database buffer pool. How do you choose which pages to evict?"
10. "Design a recommendation system cache. Users have personalized recommendations that change hourly. What eviction policy fits?"
11. "Design an API gateway cache for 10,000 upstream APIs. Each API has different traffic patterns. How do you handle eviction?"
12. "Design a distributed rate limiter that uses a cache. What eviction policy prevents stale data from breaking rate limits?"

## What the Interviewer Is Testing

When an interviewer asks about cache eviction, they are assessing:

**1. Do you understand that eviction is a trade-off?** The interviewer wants to hear that no single policy is best for all workloads. You should discuss access patterns, scan resistance, and the recency-vs-frequency trade-off.

**2. Can you implement the core data structures?** LRU and LFU implementation questions are common coding problems. The interviewer wants to see that you understand doubly-linked lists, hash maps, and frequency partitioning.

**3. Do you know production caching systems?** Mentioning Redis, Memcached, Caffeine, and their eviction policies signals real-world experience. Discussing Redis's approximate LRU versus exact LRU shows depth.

**4. Can you design around eviction failures?** The interviewer wants to hear about cache thrashing, scan attacks, LFU starvation, and how to mitigate them. Simply knowing the policies is not enough; you must know when they fail.

**5. Can you handle scale?** Eviction at 10K entries is trivial. Eviction at 10M entries per node, across 100 nodes, with 100K writes per second, is hard. The interviewer wants to hear about eviction rate, memory overhead, and batching.

## Level Expectations

### L3 / Entry Level

**What you must know:**
- What an eviction policy is and why it is needed.
- Basic LRU concept: the least recently used item is evicted.
- Basic FIFO concept: the oldest item is evicted.
- How TTL-based eviction works.

**Example question:** "Describe the LRU eviction policy."

**Expected answer (30 seconds):** "LRU stands for Least Recently Used. It evicts the entry that was accessed longest ago. The idea is that if you accessed data recently, you are likely to access it again soon. LRU is the most common eviction policy and works well for workloads with temporal locality."

### L4 / Mid Level

**What you must know:**
- LRU and LFU implementation details (data structures, complexity).
- When to use each policy.
- Basic understanding of Redis eviction configuration.
- How to implement an LRU cache.

**Example question:** "Design an LRU cache. What data structures do you use and why?"

**Expected answer (3-4 minutes):** "An LRU cache requires two data structures: a hash map for O(1) lookup by key, and a doubly-linked list for O(1) eviction. The hash map maps each key to a node in the linked list. The linked list maintains entries in access order: the head is the most recently used, the tail is the least recently used. On access, we move the node to the head (O(1) pointer updates). On eviction, we remove the tail node (O(1)). The total memory overhead is approximately 52 bytes per entry for the pointers and hash map entry."

### L5 / Senior Level

**What you must know:**
- ARC, 2Q, SLRU, and their trade-offs.
- Scan resistance and how to achieve it.
- Eviction in distributed caches (per-shard, no cross-node coordination).
- Cache thrashing, scan attacks, and their mitigations.
- Redis eviction policy configuration in detail.

**Example question:** "Your cache hit ratio dropped from 92% to 40% after a traffic change. What could be wrong and how do you fix it?"

**Expected answer (5-6 minutes):** "A sudden hit ratio drop suggests the eviction policy is mismatched to the new access pattern. I would check three things: First, check the eviction rate. If it spiked, the working set may have grown beyond cache capacity. Second, check what is being evicted. If hot data is being evicted, a scan attack may be in progress -- a sequential scan of unique keys is polluting the cache. Third, check the access pattern. If the workload shifted from session-oriented (good for LRU) to scan-oriented (bad for LRU), I would switch to a scan-resistant policy like 2Q or ARC. I would also check whether a batch job started running during this period, scanning data that does not need to be cached."

### L6 / Staff Level

**What you must know:**
- All of L5 plus:
- Adaptive policies in depth (ARC internals, W-TinyLFU).
- Eviction in the context of operating system page replacement (clock algorithm, working set model).
- Capacity planning for eviction (working set estimation, eviction rate budgeting).
- Multi-region eviction strategies.
- Eviction event buses and cache warming.

**Example question:** "Design the caching layer for a global e-commerce platform with 100M DAU across 5 regions. Each region has different popular products. How do you handle eviction?"

**Expected answer (7-8 minutes):** "I would use a per-region, per-shard eviction architecture. Each region has its own cache cluster, and each shard within the cluster manages its own eviction independently. There is no cross-region coordination for eviction -- that would add unacceptable latency. For the eviction policy, I would use ARC because it adapts to each region's access pattern without manual tuning. In region A, users might browse with strong temporal locality (LRU pattern). In region B, users might have stable popularity (LFU pattern). ARC handles both. I would size each region's cache to the region's working set plus 50% headroom. I would set TTLs on all data (5-60 minutes) to reduce eviction pressure. I would monitor eviction rate and hit ratio per shard, and alert if any shard's eviction rate exceeds 2x its baseline. For cold starts (new regions, cache restarts), I would pre-warm with the top 10K items from the nearest region."

### L7 / Principal Level

**What you must know:**
- All of L6 plus:
- Research-grade eviction algorithms (LIRS, Belady's approximation, learning-based eviction).
- Eviction in non-volatile memory (persistent memory, NVMe SSDs, tiered storage).
- Formal analysis of eviction policy performance (competitive ratio, asymptotic optimality).
- Trade-offs in custom eviction policy design.

**Example question:** "You are building a new caching system from scratch for a workload with access patterns that change every 15 minutes. The workload includes sequential scans, random access, and bursts. How do you design the eviction policy?"

**Expected answer (8-10 minutes):** "For this workload, no single static policy will work. I would design a two-tier policy: a small window cache (1-5% of total size) using LRU for immediate recency, and a frequency-based main cache using a Count-Min Sketch for approximate frequency tracking. This is similar to Caffeine's W-TinyLFU design. The window cache handles burst traffic and captures recent access patterns. The main cache handles stable, popular data. The frequency sketch periodically resets to adapt to changing patterns. I would also add scan detection: if a sequence of unique keys arrives faster than a threshold, route them through a probationary bucket rather than the main cache. This provides scan resistance. For the main eviction decision, I would use a hybrid score: frequency * recency_decay, where recency_decay is a time-based factor that reduces the score for entries not accessed in the last N minutes."

## The Framework

When answering a cache eviction question in an interview, use this framework:

### Step 1: Identify the Access Pattern

Before choosing a policy, understand the workload:
- Is access sequential or random?
- Does recent access predict future access (temporal locality)?
- Does frequency predict future access (stable popularity)?
- Are there scans (sequential read of many unique items)?
- What is the write-to-read ratio?

### Step 2: State the Trade-Off

Every policy has strengths and weaknesses. Do not present one policy as universally superior. Say: "For this workload, LRU works well because X, but we must watch out for Y."

### Step 3: Propose the Policy

State the policy and explain why it fits:
- "I recommend LRU because user sessions have strong temporal locality."
- "I recommend LFU because reference data has stable popularity."
- "I recommend ARC because we do not know the access pattern and need adaptivity."

### Step 4: Address Failure Modes

Acknowledge the failure modes of your chosen policy and propose mitigations:
- "LRU is vulnerable to scans, so I will add a probationary queue to prevent one-hit-wonders from polluting the cache."
- "LFU cannot adapt to changing patterns, so I will add periodic frequency decay."
- "ARC adds ghost list overhead, so I will size the ghost lists to 10% of the cache."

### Step 5: Discuss Scale

Describe how the policy behaves at scale:
- Per-entry memory overhead.
- Eviction time complexity.
- Concurrency model.
- Distributed coordination (or lack thereof).

## Phrase Bank

Use these phrases in your interview answers:

**On LRU:**
"LRU is the default for a reason: most workloads have strong temporal locality. Data accessed recently tends to be accessed again soon. The implementation is straightforward with a doubly-linked list and a hash map."

**On LFU:**
"LFU works well for workloads with stable popularity distributions. The risk is LFU starvation: new data cannot enter the cache because old entries have accumulated insurmountable frequency counts. This requires periodic frequency decay."

**On ARC:**
"ARC is my recommendation for production systems with unknown or changing access patterns. It is self-tuning: it balances between recency and frequency based on recent eviction behavior. The ghost lists add memory overhead, but the adaptivity is worth the cost."

**On scan resistance:**
"Scan resistance is critical for any cache that may receive sequential access. Without it, a single sequential scan can evict the entire hot set. 2Q and SLRU provide scan resistance by requiring two accesses before promoting a entry to the protected segment."

**On cache thrashing:**
"Cache thrashing occurs when every write triggers an eviction. The symptom is that the eviction rate equals the write rate. This means the cache is too small for the working set, or the write rate is too high. The solution is to increase cache size, reduce write rate, or add TTLs."

**On approximate LRU:**
"Exact LRU is rarely necessary. Redis achieves within 5% of exact LRU by sampling 5 random keys and evicting the oldest among them. This avoids the overhead of a globally sorted list while providing most of the benefit."

**On per-class eviction:**
"Memcached's per-slab-class LRU is an elegant design. Each item size class has its own LRU list. This prevents large items from disproportionately evicting small items."

**On eviction event buses:**
"In a distributed system, eviction events should be published to an event bus for monitoring and cache warming. This allows the system to react to eviction patterns without tight coupling."

**On TTL:**
"TTL is the best eviction policy because it prevents data from living in the cache indefinitely. I always set TTLs on every cache entry. The TTL should be long enough to absorb bursts of traffic but short enough to prevent stale data accumulation."

**On monitoring:**
"The eviction rate is the canary in the coal mine for cache health. A sustained increase in eviction rate signals that the working set has grown beyond the cache size, or the access pattern has changed."

**On cold starts:**
"A cold cache is the worst case for eviction because every request is a miss. Pre-warming the cache with the top 100 items by access frequency reduces the cold start window."

**On write-to-read ratio:**
"A high write-to-read ratio (more writes than reads) indicates cache pollution. The cache is being filled with data that nobody reads. This is a sign of a misuse of caching, not an eviction policy problem."

**On multi-tenant caches:**
"In a multi-tenant cache, each tenant should have its own memory quota and eviction policy. Otherwise, a write-heavy tenant can evict a read-heavy tenant's data."

## Red Flags

Avoid these mistakes in interviews:

### 1. Claiming One Policy Is Always Best
Never say "LRU is always the best policy" or "LFU is better than LRU." The correct answer is "it depends on the workload." Say: "LRU works well for workloads with temporal locality, but for workloads with stable popularity, LFU may be better."

### 2. Ignoring Failure Modes
Never propose a policy without discussing its failure modes. Every policy fails on some workload. Acknowledging the failure mode and proposing a mitigation shows depth.

### 3. Forgetting About Memory Overhead
Never propose ARC for a memory-constrained environment without discussing the ghost list overhead. ARC's per-entry overhead is approximately 2x that of LRU. In some cases, the extra memory could be better used for data.

### 4. Assuming Perfect LRU Is Necessary
Redis uses approximate LRU and achieves within 5% of exact LRU. Claiming that your design needs exact LRU without justification shows inexperience.

### 5. Overcomplicating for Simple Workloads
If the cache holds 100 configuration values that change once a day, do not propose ARC with ghost lists. A simple FIFO or random eviction is sufficient. Over-engineering is a red flag.

### 6. Ignoring TTL
Suggesting a cache without TTLs on every entry is a red flag. TTL is the simplest and most effective eviction mechanism. Without TTL, data lives in the cache indefinitely, and the eviction policy must handle all memory pressure.

### 7. Proposing Distributed Eviction Coordination
Never propose a distributed eviction protocol that requires cross-node coordination (locking, consensus). Cache eviction must be a local decision. Cross-node coordination adds latency that defeats the purpose of caching.

### 8. Ignoring Concurrency
When discussing LRU implementation, never omit thread safety. A naive LRU implementation with a single global lock will not scale. Mention per-shard locking, RCU, or lock-free data structures.

## Practice Drills

### Drill 1: LRU Cache Implementation

Implement an LRU cache with `get(key)` and `put(key, value)` operations in O(1) time.

**Key insight:** Use a hash map from key to node, and a doubly-linked list to maintain access order.

**Test case:**
```
cache = LRUCache(2)   // capacity = 2
cache.put(1, 1)
cache.put(2, 2)
cache.get(1)          // returns 1, moves key 1 to head
cache.put(3, 3)       // evicts key 2 (least recently used)
cache.get(2)          // returns -1 (not found)
```

### Drill 2: LFU Cache Implementation

Implement an LFU cache with `get(key)` and `put(key, value)` operations in O(1) time. Use frequency partitioning (array of doubly-linked lists indexed by frequency).

**Key insight:** Maintain a `minFreq` pointer that tracks the current minimum frequency. On eviction, remove from the `minFreq` bucket.

**Test case:**
```
cache = LFUCache(2)   // capacity = 2
cache.put(1, 1)       // freq(1)=1
cache.put(2, 2)       // freq(2)=1, minFreq=1
cache.get(1)          // freq(1)=2, minFreq=1
cache.put(3, 3)       // evicts key 2 (freq=1, LRU within bucket)
cache.get(2)          // returns -1 (not found)
```

### Drill 3: Eviction Policy Selection

You are designing a caching layer for three different services:
1. User session store (10M sessions, accessed within 30 minutes of creation)
2. Product catalog (100K products, some accessed 1M times/day, some once/week)
3. API response cache (bursty traffic, high one-hit-wonder rate)

For each, state which eviction policy you would use and explain why.

**Answer:**
1. Session store: LRU. Sessions have strong temporal locality. An active session is accessed every few seconds. An expired session is never accessed again.
2. Product catalog: LFU with time-decay. Product popularity is stable. The top-1% products get 90% of traffic. LFU retains them. Time-decay accounts for seasonal changes.
3. API response: 2Q or SLRU. Bursty traffic with high one-hit-wonder rate means scan resistance is critical. The FIFO probation queue prevents one-hit-wonders from polluting the hot set.

### Drill 4: Hit Ratio Estimation

Your cache has 100 GB memory. The working set is 50 GB. The eviction rate is 100 evictions/second. You increase the cache to 200 GB. The working set stays at 50 GB. What happens to the eviction rate and hit ratio?

**Answer:** The eviction rate drops to near zero because the cache is now 4x the working set size with generous headroom. The hit ratio may increase slightly (from 95% to 97-98%) because entries that were evicted under memory pressure are now retained. The main effect is on eviction rate: from 100/s to 0-1/s. This dramatically reduces the database load from cache misses.

### Drill 5: Scan Attack Defense

Your cache is using LRU. You notice a sudden drop in hit ratio from 90% to 30%. The eviction rate has increased 10x. What happened and what do you do?

**Answer:** This is a classic scan attack. A sequential scan of unique keys is polluting the cache. Each new key evicts one hot entry from the LRU tail. After enough sequential requests, the entire hot set is replaced with cold data.

**Immediate action:** Switch the eviction policy to a scan-resistant policy (2Q or ARC if available). If neither is available, add a probationary queue: new entries go to a FIFO queue, and only entries accessed twice enter the main LRU cache.

**Long-term fix:** Implement scan detection. If the number of unique keys requested per second exceeds a threshold, route them through a separate cache or bypass the cache entirely.
