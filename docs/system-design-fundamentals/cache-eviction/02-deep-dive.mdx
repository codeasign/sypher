## Internal Architecture

A production-grade cache with eviction support is not a simple hash table. It is a composition of several data structures that work together to provide fast lookups, fast eviction, and low memory overhead.

### Component Responsibilities

**Hash table.** Provides O(1) lookup by key. Maps each key to an entry node that contains the value, metadata (size, insertion time, last access time), and pointers into the other structures (list pointers for LRU, bucket pointers for LFU).

**LRU doubly-linked list.** Maintains entries in access-time order. The head is the most recently accessed entry. The tail is the least recently accessed entry. On eviction with LRU policy, the tail is removed. On access, the entry is moved to the head. Both operations are O(1) with the doubly-linked list.

**LFU frequency structure.** Maintains frequency buckets for LFU eviction. Each bucket holds all entries with the same access count. On access, the entry's count is incremented and it moves to the next bucket. On eviction, the least-frequently-used bucket is selected, and an entry from that bucket (usually the LRU within that bucket) is removed.

**Eviction policy selector.** The switchboard that selects which eviction algorithm to run. Common implementations use a function pointer or strategy pattern. The selector reads the configured policy and calls the corresponding evict function.

**Memory monitor.** Tracks total memory usage and triggers eviction when the limit is reached. Most implementations use atomic counters for thread safety. The monitor runs on every write (synchronous) or on a periodic timer (asynchronous).

Each eviction policy uses specific data structures to achieve its performance guarantees. Understanding these data structures is essential for implementing your own cache or tuning an existing one.

The doubly-linked list is the heart of LRU. Each entry node has `prev` and `next` pointers. When an entry is accessed:

1. Remove the node from its current position (O(1) pointer updates).
2. Insert the node at the head (O(1) pointer updates).
3. Update the `head` pointer to the new node.

When eviction is needed:

1. The tail node is the victim (least recently used).
2. Remove the tail node from the list (O(1) pointer updates).
3. Remove the entry from the hash map (O(1)).
4. Free the memory.

The hash map provides the mapping from key to node. Without it, finding an entry by key would require O(n) list traversal. With it, every operation is O(1).

The naive LFU implementation uses a min-heap keyed by frequency. On every access, the entry's frequency is incremented and the heap is sifted. On eviction, the root of the heap (lowest frequency) is removed. This gives O(log n) per operation.

The optimized implementation uses frequency partitioning: an array of doubly-linked lists, where index i holds all entries with frequency i. A `minFreq` pointer tracks the current minimum frequency. On eviction, the entry at `minFreq` is removed. On access, the entry is moved to the `freq+1` list, and `minFreq` may be updated. This gives O(1) per operation.

The clock algorithm provides an approximation of LRU with minimal overhead. Instead of maintaining a sorted list, it uses a circular buffer and reference bits. Each entry has one reference bit that is set to 1 on access. The clock hand sweeps through the buffer: entries with R=1 get a second chance (R is cleared and the hand advances), entries with R=0 are evicted.

## Eviction Lifecycle

The eviction lifecycle differs by policy. This section shows the exact steps for LRU, LFU, and ARC side by side.

### LRU Eviction Lifecycle

1. **Write received.** `SET key="user:42" value="{...}"`. The cache engine receives the write command.

2. **Hash table lookup.** The engine checks whether the key already exists. If it does, this is an update rather than an insert.

3. **Memory check.** The engine adds the new entry's size to the current memory usage. If the total exceeds `maxmemory`, eviction begins.

4. **Find LRU tail.** The engine reads the `tail` pointer of the doubly-linked list. This is the least recently used entry.

5. **Remove from list.** The tail entry is unlinked from the doubly-linked list. Its `prev` entry becomes the new tail.

6. **Remove from hash table.** The engine deletes the key from the hash table. The memory for the key and value is freed.

7. **Update counters.** The memory usage counter is decremented by the evicted entry's size. The eviction counter is incremented.

8. **Post-eviction check.** If memory is still above the limit, steps 4-7 are repeated until memory drops below the watermark.

9. **Insert new entry.** The new entry is created, inserted into the hash table, and added to the head of the linked list.

10. **Response.** The cache returns OK to the caller.

### LFU Eviction Lifecycle

1. **Write received.** Same as LRU.

2. **Hash table lookup.** Same as LRU.

3. **Memory check.** Same as LRU.

4. **Find minimum frequency.** The engine checks the `minFreq` pointer (in optimized implementation) or peeks at the heap root. The minimum frequency bucket is identified.

5. **Select victim within bucket.** If multiple entries share the minimum frequency, a tiebreaker is applied. Common tiebreakers: LRU within the bucket (evict the entry accessed longest ago within the same frequency group), or random selection.

6. **Remove from frequency bucket.** The victim is unlinked from its frequency bucket's doubly-linked list.

7. **Remove from hash table.** The entry is deleted from the hash table.

8. **Update minFreq.** If the frequency bucket is now empty, `minFreq` is incremented to the next non-empty bucket. If `minFreq` was the only bucket, it may need to be recalculated by scanning all buckets.

9. **Insert new entry.** The new entry is created with frequency=1 and inserted into the freq=1 bucket. If no freq=1 bucket exists, it is created. `minFreq` is set to 1.

10. **Response.** The engine returns OK.

### ARC Eviction Lifecycle

1. **Write received.** Same as LRU.

2. **Hash table lookup.** The engine checks all four lists (T1, T2, B1, B2) for the key. If found in a ghost list, a ghost hit is recorded.

3. **Ghost hit adaptation.** If the key is found in B1 (recent ghost), the target size `p` is adjusted upward (more room for recent entries). If in B2 (frequent ghost), `p` is adjusted downward.

4. **Memory check.** Same as LRU.

5. **Eviction decision.** If T1 (recent list) exceeds target size `p`, evict from T1. Otherwise, evict from T2.

6. **Move to ghost.** The evicted entry's key and metadata are moved to the appropriate ghost list (B1 for T1 evictions, B2 for B2 evictions).

7. **Ghost list pruning.** If the ghost list exceeds capacity, the oldest ghost entry is removed (the ghost list should be bounded, typically to the same size as the cache).

8. **Insert new entry.** The new entry is inserted into T1 (recent list) at the MRU position.

9. **Response.** The engine returns OK.

## Consistency and State

Cache eviction becomes significantly more complex in concurrent and distributed environments. This section covers the failure modes that appear under real-world conditions.

### Concurrent Access During Eviction

When multiple threads can read and write the cache simultaneously, eviction must be thread-safe. Consider this scenario:

- Thread A reads entry X. X is hot, accessed frequently.
- Thread B writes entry Y. The cache is full, so B triggers eviction.
- Thread B selects entry X as the victim (LRU policy, X is at the tail).
- Thread A is still reading X's value.
- Thread B frees X's memory.

Thread A now has a dangling pointer. The next read crashes the application (segfault) or reads corrupted data.

**Solutions:**

- **Read-copy-update (RCU).** Readers do not lock. Writers create a new copy of the entry, update pointers atomically, and defer the free until all readers have finished. Used in the Linux kernel.

- **Reference counting.** Each entry has a reference count. Readers increment the count before reading and decrement after. Eviction skips entries with non-zero reference counts. This prevents freeing memory that is in use, but it can delay eviction under high concurrency.

- **Sharded locking.** The cache is divided into shards, each with its own lock. Eviction operates on one shard at a time. This provides fine-grained concurrency without a global lock.

### Priority Inversion: Frequency vs Recency

Priority inversion occurs when a policy optimizes for the wrong dimension:

- LFU retains the frequency-3 entry that was last accessed 5 minutes ago (it is still "frequently used").
- LFU evicts the frequency-2 entry that was accessed 1 second ago (it is trending but has not accumulated enough frequency).
- The frequency-2 entry was the future popular content. The cache now misses on it.

This is LFU's fundamental weakness: it cannot distinguish between a entry that was popular yesterday and a entry that is popular today. The frequency counter is a lagging indicator.

**Mitigation:** Use time-decayed frequency (LFU with aging). Periodically halve all frequency counters, or use a sliding window that considers only accesses within the last N minutes. Redis implements this with its `lfu-log-factor` parameter.

### Scan Attacks: Polluting LRU

A scan attack floods the cache with one-hit-wonders, intentionally evicting the hot set.

```
Scenario:
  - Cache holds 100 entries (the hot set).
  - Attacker sends 100 sequential requests for 100 new unique keys.
  - Each new key is inserted, evicting one hot entry.
  - After 100 requests, the entire hot set is replaced with one-hit-wonder data.
  - All subsequent requests for the hot set miss and hit the database.
```

**Mitigation:** Scan-resistant policies (ARC, 2Q, SLRU). With 2Q, an attacker's sequential scan fills the FIFO queue but never promotes to the LRU queue. The hot entries in the LRU queue are protected.

### Memory Pressure Cascade

When the cache runs out of memory and eviction cannot keep up, the system enters a pressure cascade:

1. Eviction falls behind (too many writes, too few evictions).
2. Memory usage exceeds `maxmemory`.
3. The engine triggers aggressive eviction, evicting multiple entries per write.
4. High eviction rate causes cache miss rate to spike.
5. Missed requests hit the database, increasing database load.
6. Database response time increases.
7. Application threads block waiting for database responses.
8. Application thread pool saturates.
9. New requests queue up, increasing latency.
10. Health checks fail, load balancer marks the instance as unhealthy.
11. Traffic is redirected to remaining instances, which also hit the cascade.

## Performance

### Eviction Complexity

| Policy | Eviction | Access | Insert | Notes |
|--------|----------|--------|--------|-------|
| LRU    | O(1)     | O(1)   | O(1)   | Doubly-linked list + hash map |
| LFU (naive) | O(log n) | O(log n) | O(log n) | Min-heap per access |
| LFU (optimized) | O(1) | O(1) | O(1) | Frequency partitioning |
| FIFO   | O(1)     | O(1)   | O(1)   | Simple queue |
| MRU    | O(1)     | O(1)   | O(1)   | Same as LRU, reverse direction |
| Random | O(1)     | O(1)   | O(1)   | Random selection |
| ARC    | O(1)     | O(1)   | O(1)   | Dual lists + ghost management |
| Clock  | O(n)*    | O(1)   | O(1)   | *Worst-case full rotation |
| 2Q     | O(1)     | O(1)   | O(1)   | Two queues + promotion logic |
| SLRU   | O(1)     | O(1)   | O(1)   | Two segments + promotion |

The clock algorithm has O(1) amortized eviction but O(n) worst-case when all reference bits are set and the hand must sweep the entire buffer.

## Configuration

### Memory Limits

Every production cache requires a configured memory limit. Common configuration parameters:

**maxmemory.** The absolute memory limit for the cache. When total memory usage exceeds this value, eviction is triggered. In Redis: `maxmemory 4gb`. In Memcached: `-m 4096` (megabytes).

**Eviction policy selection**. The algorithm to use when evicting. In Redis: `maxmemory-policy allkeys-lru`. Six options across three categories:
- `allkeys-*`: evicts from all keys (no TTL filter)
- `volatile-*`: evicts only keys with TTL set
- `*-lru`: LRU eviction
- `*-lfu`: LFU eviction
- `*-ttl`: Evict keys with shortest TTL
- `*-random`: Random eviction
- `noeviction`: Return errors when memory is full (writes fail)

**Sample size (for approximate LRU).** Redis uses an approximate LRU that samples N keys and evicts the oldest among them. Larger samples produce more accurate LRU behavior but cost more CPU per eviction. Default: 5 entries. Increasing to 10 improves accuracy by approximately 5% at the cost of doubling eviction CPU time.

```
eviction-policy-samples: 5    # Recommended for most workloads
eviction-policy-samples: 10   # For high hit-ratio requirements
eviction-policy-samples: 20   # For maximum accuracy, higher CPU cost
```

**Watermark thresholds.** Some caches use two thresholds: a high watermark (trigger eviction) and a low watermark (stop eviction). This prevents eviction thrashing around the memory limit.

```
maxmemory: 4gb
eviction-high-watermark: 0.95  # Trigger at 3.8 GB
eviction-low-watermark: 0.85   # Stop at 3.4 GB
```

**Eviction batch size.** The number of entries to evict per eviction cycle. Higher batch sizes free memory faster but increase latency per write.

```
eviction-batch-size: 1    # Conservative: 1 entry per eviction
eviction-batch-size: 10   # Aggressive: 10 entries per eviction
eviction-batch-size: 100  # Emergency: used when maxmemory is breached
```

**Maxmemory-samples (Redis).** Controls the size of the sample pool for approximate LRU/LFU/TTL eviction. The engine samples N random keys and applies the policy to select the victim. Larger samples improve accuracy.

```
maxmemory-samples 5    # Default
maxmemory-samples 10   # Better accuracy, slightly higher CPU
```

**LFU configuration (Redis).** Two parameters control LFU behavior:

- `lfu-log-factor`: Logarithmic counter factor. Reduces the rate at which the frequency counter increments. Higher values make the counter grow more slowly, favoring older entries. Default: 10.

- `lfu-decay-time`: The number of minutes between counter halvings. Higher values make the counter decay more slowly, favoring entries with sustained popularity. Default: 1 (halve the counter every minute).

### Tuning for Different Workloads

**Session cache (high temporal locality, fixed TTL).** Use `allkeys-lru` in Redis. Session data has strong recency: active sessions are accessed frequently, inactive sessions age out naturally. Set TTL equal to the session timeout (30-60 minutes). The LRU policy handles eviction when memory fills, while TTL handles expiration.

**Database query cache (mixed access, moderate locality).** Use `allkeys-lfu` if query patterns are stable (most queries target a small set of popular rows). Use `allkeys-lru` if the workload is dynamic. The choice depends on whether query popularity is stable or changing.

**API response cache (bursty, high one-hit-wonder rate).** Use `volatile-lru` with TTL set to the API response's cache-control max-age. The TTL handles most eviction; LRU handles overflow. Consider adding 2Q or SLRU for scan resistance.

**Static content cache (CDN, moderate object lifetime).** Use `allkeys-lfu` for content with stable popularity (most-visited pages, popular images). Use `allkeys-lru` for content with temporal bursts (news articles, trending videos). Monitor the eviction rate and switch policies if the hit ratio drops below 80%.
