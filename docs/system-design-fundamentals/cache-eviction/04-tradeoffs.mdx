## The Core Tension

Every eviction policy optimizes for a specific access pattern and performs poorly on others.

This is the central trade-off in cache eviction: there is no universal best policy. The best policy for your workload depends on your access pattern, and your access pattern is determined by your application's data access behavior.

The trade-off matrix:

- **Recency vs frequency.** LRU optimizes for recent access. LFU optimizes for frequent access. A workload with rapid popularity changes needs recency. A workload with stable popularity needs frequency. No single policy can optimize both simultaneously.

- **Scan resistance vs hot retention.** Policies that are strongly scan-resistant (2Q, SLRU) sacrifice some hot-set retention because they require a probationary period before promoting entries. Policies with strong hot retention (LRU) are vulnerable to scan attacks.

- **Simplicity vs adaptivity.** Simple policies (FIFO, random) are easy to implement and reason about, but they perform poorly on most workloads. Adaptive policies (ARC) perform well across workloads but are complex to implement and hard to debug.

- **Memory overhead vs policy accuracy.** LRU with a full doubly-linked list gives perfect LRU ordering but costs 52 bytes per entry. The clock algorithm costs 16 bytes per entry but gives only approximate LRU ordering. The trade-off: per-entry memory for policy accuracy.

- **Eviction speed vs hit ratio.** Fast eviction (random, FIFO) minimizes write latency but may evict the wrong entries, reducing hit ratio. Accurate eviction (LRU, LFU) maximizes hit ratio but adds overhead per write.

### When Each Policy Wins

**LRU wins when:** Access has strong temporal locality. User sessions, API response caches, and most web application caches benefit from LRU. It is the safe default for workloads without extreme scan behavior.

**LFU wins when:** Access frequency is stable and highly skewed. Reference data, configuration values, and top-N lists benefit from LFU. An entry accessed 1,000 times per day should not be evicted for an entry accessed twice.

**FIFO wins when:** Access order equals access importance. Event logs, time-series data, and request tracing benefit from FIFO. The oldest data is the least valuable.

**ARC wins when:** The access pattern is unknown or changes over time. Production systems that cannot be manually tuned benefit from ARC's self-tuning behavior. ARC is the closest practical approximation to Belady's optimal algorithm.

**Random wins when:** The workload is unpredictable or when simplicity is paramount. Random eviction with a large cache performs surprisingly well -- within 10% of LRU on most workloads.

**2Q and SLRU win when:** The workload includes occasional scans that would pollute an LRU cache. Proxy caches, CDN edge caches, and any cache serving mixed traffic types benefit from scan resistance.

## CAP Theorem and Eviction

Cache eviction does not directly relate to the CAP theorem (Consistency, Availability, Partition Tolerance). Eviction is a local decision within a single cache node, not a distributed consensus problem.

However, distributed eviction coordination adds complexity that touches on CAP-like trade-offs:

- **Consistency of eviction decisions.** In a distributed cache with replication, should all replicas evict the same entries at the same time? If a primary evicts key X but a replica still holds X, a read from the replica returns stale data. This is an intentional trade-off: cache replicas are eventually consistent by design. Coordinated eviction would require distributed consensus (Paxos/Raft), which adds unacceptable latency.

- **Availability during eviction.** Eviction must not block reads. If a cache node is evicting a batch of entries, reads for non-evicted keys must still succeed. Production caches achieve this through per-shard locking or lock-free data structures.

- **Partition tolerance.** If a cache node is partitioned from the network, it continues operating independently. Its eviction decisions are local and may diverge from the rest of the cluster. When the partition heals, the node's cache may contain entries that were evicted elsewhere. This is acceptable -- the node will serve slightly stale data until its entries are naturally evicted or its TTLs expire.

The practical implication: do not try to coordinate eviction across cache nodes. Eviction should be a per-node decision, optimized for the node's own access pattern, without cross-node coordination.

## Failure Modes

### 1. Cache Thrashing

Cache thrashing occurs when the eviction rate is so high that the cache spends more time evicting than serving data. Every insert triggers an eviction, and every eviction removes data that is immediately re-requested.

**Causes:** Cache too small for the working set, or write rate exceeds the cache's ability to retain hot data.

**Detection:** Eviction rate equals write rate. Hit ratio below 20%.

**Mitigation:** Increase cache size. If that is not possible, use a scan-resistant policy (2Q, ARC) or add TTLs so data self-evicts.

### 2. Scan Attack (LRU Pollution)

A scan attack sends a large number of sequential requests for unique keys, intentionally evicting the hot set from an LRU cache.

```
Before scan attack:
  Cache: [hot_1, hot_2, hot_3, ..., hot_100]  (all frequently accessed)

During scan attack (100 sequential requests for unique keys):
  Request key_scan_1 -> Insert -> Evict hot_100 (now 99 hot entries)
  Request key_scan_2 -> Insert -> Evict hot_99  (now 98 hot entries)
  ...
  Request key_scan_100 -> Insert -> Evict hot_1  (0 hot entries remain)

After scan attack:
  Cache: [key_scan_1, key_scan_2, ..., key_scan_100] (all cold data)
  Next request for hot_1 -> Cache miss -> Database hit
```

**Causes:** LRU's temporal locality assumption is violated by sequential access.

**Detection:** Eviction rate spikes. Hit ratio drops. All evicted entries are from the hot set.

**Mitigation:** Use a scan-resistant policy (ARC, 2Q, SLRU). These policies require two accesses before considering a entry for retention, protecting against one-hit-wonder floods.

### 3. LFU Starvation

LFU starvation occurs when new data cannot enter the cache because existing entries have accumulated high frequency counts.

```
Scenario:
  Cache holds 100 entries, all with frequency >= 50.
  A new entry arrives (frequency = 1).
  All existing entries have higher frequency.
  LFU evicts nothing (all entries have nonzero frequency).
  Actually, LFU would still evict from the lowest frequency bucket.
  
  But the problem is more nuanced:
  Entry A: inserted day 1, accessed 100 times since, freq=100
  Entry B: inserted day 2, accessed 99 times since, freq=99
  ...
  Today: New hot topic "C" emerges.
  Entry C: inserted now, freq=1
  
  Even if "C" is accessed 10 times in the next minute (freq=10),
  it still ranks below all the old entries. The old entries
  have accumulated frequency over days and cannot be overtaken.
```

**Causes:** LFU has no decay mechanism. Frequency counters only increment. Old entries accumulate insurmountable frequency leads.

**Detection:** New entries have very short lifespans (evicted quickly). The hit ratio drops for new content but remains high for old content.

**Mitigation:** Use LFU with aging (time-decayed frequency counters). Redis's `lfu-decay-time` configuration controls how quickly counters decay. Set it to 1 (decay every minute) for workloads with changing popularity.

### 4. Memory Leak from Eviction Failure

In some implementations, the eviction process itself can fail:

- **Eviction target is pinned.** The entry selected for eviction is locked by another operation (transaction, streaming read). The eviction engine cannot remove it. It skips the entry and selects the next candidate. If enough entries are pinned, eviction falls behind.

- **Eviction cascades.** Writing a single large entry (1 MB) may require evicting 100 small entries (10 KB each). Each eviction involves hash table operations, memory freeing, and list updates. If eviction is too slow, memory usage grows past the limit.

- **Memory not returned to OS.** In languages with GC (Java, Go), evicting an entry frees the object reference, but the garbage collector may not immediately return memory to the OS. The OS sees the process's RSS remain high even though the cache has freed entries. This can trigger OOM kills even when the cache believes it is under the limit.

**Mitigation:** Use jemalloc or tcmalloc (better memory reclamation than glibc malloc). In Java, tune the GC for prompt memory reclamation. Monitor RSS separately from the cache's internal memory counter.

## Anti-Patterns

### 1. No Eviction Policy (OOM Risk)

Running a cache without an eviction policy -- relying only on TTL expiration -- is dangerous. If the write rate exceeds the TTL expiration rate, memory grows without bound until the process is killed.

Redis defaults to `noeviction` policy, which returns errors on writes when memory is full. This is safe for some workloads (if memory is sized to handle peak writes), but it must be explicitly configured. Many production incidents are caused by Redis instances reaching `maxmemory` with `noeviction` policy, causing write failures that cascade to the application.

**Fix:** Always set an eviction policy. Use `allkeys-lru` as the safe default. Only use `noeviction` if you have verified that the working set fits in memory with sufficient headroom.

### 2. Same Policy for All Data Patterns

A multi-tenant cache or a cache serving heterogeneous data types should not use a single eviction policy. Session data needs LRU. Configuration data needs LFU. Event logs need FIFO.

Redis addresses this with database-level isolation (multiple databases per instance), each with its own eviction policy. Memcached does not support per-namespace policies.

**Fix:** Separate caches by data type, each with its own policy. Use Redis databases or separate Redis instances.

### 3. Too Small Cache

A cache sized too small for the working set evicts constantly. Every write triggers an eviction, and every eviction creates a future cache miss. The cache becomes a performance liability rather than an asset.

Common sizing mistake: using a fixed memory limit (e.g., 1 GB) without measuring the working set. The working set may be 5 GB, and the 1 GB cache is thrashing at 90% write rate.

**Fix:** Measure the working set. Size the cache to at least 100% of the working set, ideally 150-200% for headroom. Monitor the eviction rate to validate the sizing.

### 4. Ignoring Access Pattern Changes

A cache configured with LFU for a workload that suddenly shifts to rapid popularity changes will degrade. The LFU cache retains yesterday's hot entries, and the hit ratio drops as users request today's content.

Similarly, a cache configured with LRU for a workload that shifts to stable popularity will underperform. The LRU cache evicts stable high-value entries that are accessed infrequently but consistently.

**Fix:** Monitor the hit ratio over time. A sustained drop signals that the access pattern has changed and the eviction policy may need adjustment. Consider ARC for workloads with changing patterns.

### 5. Over-Provisioning Memory

Giving a cache too much memory is not always safe. A cache that never evicts holds stale data indefinitely. Entries that are no longer relevant (expired sessions, deleted products, deprecated configurations) consume memory unnecessarily.

Worse, an over-provisioned cache masks the need for a proper eviction policy. The system appears to work, but only because memory is abundant. When memory pressure does occur (traffic spike, data growth), the eviction policy may be poorly tuned for the actual workload.

**Fix:** Set `maxmemory` to a reasonable value based on the working set. Use TTLs liberally. Test eviction behavior under expected memory pressure.

## When NOT to Use

Cache eviction -- as a concept -- is not needed in all scenarios. Here are cases where eviction is irrelevant or harmful.

### When You Can Size the Cache for the Working Set

If the entire working set fits comfortably in memory, eviction never triggers. This is the ideal scenario: 100% hit ratio, zero evictions.

This is achievable for reference data, configuration databases, small user bases, or any workload where the total data footprint is known and bounded.

**Example:** An application with 10,000 users and 100 KB of cached data per user has a 1 GB working set. A 4 GB cache never evicts. No eviction policy needed.

### When Data Has Natural TTL

If every cached entry has a short TTL (under 60 seconds), eviction may never trigger because entries expire before the cache fills.

**Example:** A real-time dashboard that caches data for 30 seconds. With 10,000 entries and 30-second TTL, the cache reaches a steady state where the expiration rate matches the write rate. Eviction never fires.

In this scenario, TTL acts as a soft eviction policy. You do not need an explicit eviction policy, but you should still configure one as a safety net for TTL expiration failures.

### When Access Is Purely Sequential

If every piece of data is accessed exactly once (sequential processing, batch jobs), eviction is irrelevant. Nothing is worth retaining because nothing is accessed twice.

**Example:** A batch processing pipeline that reads records from a database, processes them, and never needs them again. There is no benefit to caching the records, and eviction is a waste of CPU.

In this scenario, do not use a cache at all. Eviction is the symptom of a meaningless caching strategy.

### When the Backing Store Is Faster Than the Cache

This is rare but possible: if the database uses SSDs and the cache uses a network-based service with high latency, the database may be faster than the cache. In this scenario, caching degrades performance, and eviction is irrelevant.

**Example:** A local SQLite database (microsecond access) used to populate a remote Redis cache (millisecond network round-trip). Cache misses are faster than cache hits.

Of course, modern caching systems (Redis on localhost, Memcached on the same rack) are almost always faster than databases. But this assumption deserves verification.
