## The Problem Story

A financial analytics platform deploys a single LLM to answer complex investment questions. The model receives a query: "Analyze the Q3 earnings of our top 10 holdings, compare their P/E ratios against industry averages, flag any that dropped below their 52-week low, and write a summary memo for the CIO." The LLM produces a response. It looks plausible. It cites "Microsoft's P/E of 45.2" — but Microsoft's actual P/E is 34.8. It mentions "Apple's 52-week low of $145" — two quarters out of date. It misses the warning about a holding that just announced a dividend cut, because the LLM's training data stopped nine months before the announcement. The analyst who receives the report trusts the output, forwards it to the CIO, and the firm makes a buy decision based on stale data. The error cost: 2.3 million dollars.

A customer support chatbot for a SaaS platform receives a request: "I lost access to my account after changing my email. I need to reset my password, restore my billing history, and reactivate my team workspace." The chatbot tries to handle everything in one prompt. It generates a password reset link but does not check whether the user's session is still valid. It promises to restore billing history but has no access to the billing database. It attempts to reactivate the workspace without verifying the user's role. The result: the user goes through three incorrect password resets, gets locked out of their email, and calls support in frustration. The chatbot attempted to solve three problems simultaneously and solved none of them.

A code-generation tool receives a task: "Refactor the authentication module to use OAuth 2.0 with PKCE, update all downstream consumers, add unit tests, and update the API documentation." The LLM generates a single file that attempts to do everything. It rewrites the authentication module, but the new code does not match the existing interfaces. The downstream consumers break. The unit tests reference methods that no longer exist. The API documentation describes a flow that was never implemented. The developer spends two days debugging the generated code, then reverts everything and writes it manually.

These failures share a single root cause: a monolithic LLM, called once, cannot decompose complex tasks into steps, cannot maintain state across operations, cannot call external tools to verify facts, and cannot recover from its own mistakes. An LLM generates text. An agent reasons, plans, acts, observes, and iterates. The difference is architecture.

## The Insight

AI Agent Architecture is the system design pattern that structures autonomous LLM-powered agents as a loop of reasoning, action, and observation, connected to tools, memory, and orchestration layers. Instead of a single LLM call that produces a final answer, you get a controlled loop where the agent thinks, acts, checks the result, and decides what to do next.

Think of it as a restaurant kitchen brigade. A single chef trying to cook a 12-course tasting menu for 200 guests alone would fail — they would burn the sauce while butchering the fish, forget the garnish while plating the entree, and run out of prep time for dessert. A well-designed kitchen assigns a sous-chef to coordinate, a saucier to handle sauces, a poissonnier to prepare fish, a patissier for desserts, and a commis to handle prep work. Each station has its own tools, its own workspace, and its own expertise. The sous-chef calls out the order, each station executes its part, and the finished plates come together at the pass.

An AI agent architecture works the same way. The orchestrator is the sous-chef. The planner decomposes the order into tasks. The executor runs each task with the right tool. The memory system stores the recipe and the state of each dish. Response is the pass where the finished order comes together.

## Core Vocabulary

**Agent:** An autonomous system that perceives its environment, reasons about a goal, selects actions, executes them, and observes the results. An agent is not a single LLM call — it is a loop that runs until the goal is achieved or a stopping condition is met.

**Orchestrator:** The central coordinator that manages the agent lifecycle. The orchestrator receives the user request, initializes the agent loop, routes messages between components, and returns the final response. It decides when to plan, when to execute, when to stop, and what to do when something goes wrong.

**Tool:** A function or API that the agent can call to interact with the external world. Tools include database queries, web APIs, file system operations, code execution sandboxes, search engines, and calculators. Each tool has a name, description, input schema, and output schema that the LLM uses to decide when and how to call it.

**Planner:** The component that decomposes a complex goal into a sequence of steps or a dependency graph. The planner takes the user's request and the current system state, then produces a plan: "Step 1: look up user by email. Step 2: reset password. Step 3: fetch billing history. Step 4: verify team role. Step 5: reactivate workspace."

**Executor:** The component that runs a single plan step by selecting the appropriate tool, constructing the tool call, executing it, and capturing the result. The executor may retry on failure, handle timeouts, and validate the tool output before passing it to the next step.

**Memory:** The storage system that maintains the agent's state across loop iterations. Memory includes the conversation history, tool call results, intermediate reasoning steps, user preferences, and any long-term knowledge the agent needs to persist across sessions.

**State:** The complete snapshot of the agent's current situation at any point in the loop. State includes the original goal, the current plan, completed steps, pending steps, accumulated observations, and the agent's internal reasoning. State must be serializable, inspectable, and recoverable.

**Reasoning Loop:** The iterative cycle of think-act-observe that drives agent behavior. The agent receives the current state, reasons about what to do next, produces an action (a tool call or a response), executes the action, observes the result, and feeds the observation back into the next reasoning step.

**Tool Call:** A structured invocation of an external function by the agent. A tool call includes the tool name, the input parameters, and a unique identifier. The LLM outputs a structured tool call request, the system executes it, and the result is returned to the LLM as a tool response message.

**Supervisor:** An agent that manages other agents. A supervisor receives tasks, assigns them to sub-agents, monitors their progress, aggregates their results, and handles failures. The supervisor pattern is used in multi-agent systems where different agents have different capabilities or knowledge domains.

## How It Works Step by Step

The agent loop is the core runtime of any AI agent system. Below is the complete sequence of steps that execute for every user request. Understanding this sequence is essential for designing, debugging, and optimizing agent systems.

### Step 1: Receive the User Request

The orchestrator receives the user's query and creates a session context. The session includes the user ID, conversation history, and any relevant metadata. The orchestrator also validates the user's authentication, checks rate limits, and determines whether the user has permission to use the requested agent capabilities.

### Step 2: Analyze and Classify the Request

The orchestrator analyzes the request to determine complexity, required tools, and expected effort. Simple requests bypass the planner and go directly to execution. Complex requests enter the planning phase. The orchestrator may tag the request with metadata: priority level, expected response time, required security clearance, and supported languages. This classification step helps the system decide how to allocate resources — a simple "what is the weather" request gets a fast, cheap LLM call, while a complex "analyze our quarterly financials" request gets the full agent pipeline with a powerful reasoning model.

### Step 3: Decompose into a Plan

The planner receives the classified request and produces a step-by-step plan. Each step has a description, required tool, expected output, and dependency ordering. The planner may produce multiple plan variants and select the one with the lowest estimated cost or highest expected success rate. The plan is stored in the session state and serves as the roadmap for the execution phase.

### Step 4: Enqueue Tasks

The plan is converted into a task queue. Each task is a self-contained unit of work with a tool assignment, input parameters, and a status field. The queue supports parallel execution of independent tasks. The queue also supports priority ordering — critical tasks jump to the front of the queue, while background tasks wait their turn. Task assignments include a timeout value derived from the tool's expected execution time and the overall agent time budget.

### Step 5: Execute Task — Tool Selection

The executor picks the next ready task from the queue. It selects the appropriate tool based on the task definition and constructs the tool call with the required parameters. The executor validates the parameters against the tool's input schema before making the call. If validation fails, the executor returns a validation error without executing the tool, saving time and resources. The validator checks for missing required fields, type mismatches, value range violations, and format errors.

### Step 6: Execute Task — Tool Call

The executor invokes the selected tool. For a database tool, this means running a SQL query. For an API tool, this means making an HTTP request. The tool executes and returns a result. The executor enforces a timeout on every tool call — if the tool does not respond within the configured timeout, the executor returns a timeout error. The timeout is tool-specific: a simple calculator might time out after 2 seconds, while a web search might time out after 15 seconds.

### Step 7: Observe and Store Result

The tool result is stored in the memory store along with metadata: execution time, success status, token count, and any error codes. The observation becomes part of the agent's state for subsequent reasoning steps. The executor also checks the result for data quality issues — if the result is empty, malformed, or contains error codes, the executor flags the observation as potentially unreliable. This flag informs the LLM that it should treat the result with caution.

### Step 8: Update Plan State

The executor marks the task as completed in the plan state. If the task succeeded, the result is attached to the task record. If the task failed, the error is recorded and the planner may be invoked to revise the plan. Dependencies are recalculated. Newly unblocked tasks become ready for execution.

### Step 9: Iterate — Next Action

The loop continues. The executor picks the next ready task, calls the tool, observes the result, and updates state. This cycle repeats until all tasks are complete, a maximum iteration limit is reached, or a critical error occurs. The loop controller tracks the number of iterations, total tokens consumed, and elapsed time. If any of these exceed configured limits, the loop terminates with a partial result. The loop controller also monitors for stagnation — if the agent has not made progress toward the goal for several iterations, it triggers a replanning step.

### Step 10: Handle Tool Failures

If a tool call fails, the executor checks the retry policy. If retries are exhausted, the error is escalated to the orchestrator. The orchestrator may ask the planner to generate an alternative approach, ask the user for guidance, or gracefully degrade by skipping the failed step. The orchestrator records the failure in the session state for post-hoc analysis. The failure is also fed back to the LLM so the agent can reason about what went wrong and adjust its approach.

### Step 11: Synthesize Final Response

When all tasks are complete, the orchestrator collects the accumulated results and passes them to the LLM for final response generation. The LLM synthesizes the observations into a coherent answer, citing sources and caveats as appropriate. The orchestrator may also run a quality check on the response — checking for hallucination, ensuring all user questions were answered, and verifying that the response does not contain sensitive information.

### Step 12: Return and Clean Up

The orchestrator returns the final response to the user. Session state is persisted to long-term storage for future conversations. Resources are released. The agent loop terminates. The orchestrator emits metrics for the completed run: total iterations, total tokens, total cost, tool call breakdown, and success or failure status. These metrics feed into dashboards for monitoring agent health and cost.

## Variants and Flavors

Different use cases call for different agent architectures. Below are the four most common patterns.

### ReAct Agent

The ReAct (Reasoning + Acting) pattern interleaves reasoning and action at every step. The agent thinks, acts, observes, and repeats. This is the most common pattern for single-agent systems.

**When to use:** Single-step tool use, question-answering with retrieval, simple task automation. The ReAct pattern is effective when each step depends on the result of the previous step and the reasoning is straightforward.

### Plan-and-Execute Agent

The Plan-and-Execute pattern separates planning from execution. The planner produces a complete plan upfront, then the executor runs through the steps. This is also called the "plan-then-act" pattern.

### Multi-Agent Supervisor Pattern

A supervisor agent coordinates multiple sub-agents, each with specialized capabilities. The supervisor assigns tasks, monitors progress, handles conflicts, and aggregates results.

### Variant Comparison

| Dimension | ReAct | Plan-and-Execute | Multi-Agent Supervisor | Tool-Use |
|-----------|-------|-----------------|----------------------|----------|
| Planning | Per-step (dynamic) | Upfront (static) | Per-agent (delegated) | None (implicit) |
| Execution order | Sequential based on observations | Sequential or parallel per plan | Parallel by sub-agent | Sequential by LLM choice |
| Error recovery | Good (replan per step) | Limited (replan whole plan) | Good (reassign between agents) | Limited (retry same tool) |
| Scalability | Single agent | Single agent with parallel tasks | Multiple agents | Single agent |
| Complexity | Medium | Low | High | Low |
| Best use case | Research, debugging | Reports, pipelines | Multi-domain tasks | Simple Q&A |
| Cost per task | Medium | Low | High | Low |
| Predictability | Low (dynamic path) | High (fixed path) | Medium (per-agent variation) | Low (LLM decides) |

### How to Choose Between Variants

The choice of agent variant depends on three factors: the nature of the task, the operating environment, and the risk tolerance.

For tasks where the path is unknown and depends on intermediate results, use ReAct. The classic example is a research agent that searches for information and decides what to search for next based on what it finds. The path cannot be planned upfront because you do not know what you will find.

For tasks where the path is known and the dependencies are fixed, use Plan-and-Execute. The classic example is a report generator that always collects data from the same sources and formats it the same way. The path is known upfront and every instance follows the same sequence.

For tasks that require multiple domains of expertise, use the Multi-Agent Supervisor pattern. The classic example is a system that needs a legal expert, a financial expert, and a technical expert to answer a single question. No single agent can have expertise in all three domains.

For tasks that require simple tool access without complex reasoning, use the Tool-Use pattern. The classic example is a chatbot that can look up weather, set reminders, and send emails. These are independent tool calls that do not require multi-step reasoning.

## Mental Model Check

Test your understanding of AI Agent Architecture concepts with these questions.

**Question 1:** What is the fundamental difference between a single LLM call and an AI agent?

**Answer:** A single LLM call generates text based on the input and the model's training data. It has no ability to reason iteratively, call external tools, maintain state across steps, or recover from errors. An AI agent operates as a loop: it receives a goal, reasons about what to do, calls tools to gather information or take actions, observes the results, and iterates until the goal is achieved. The agent can adapt its approach based on observations, recover from tool failures, and maintain state across multiple reasoning steps.

**Question 2:** Why does the ReAct pattern interleave reasoning and action instead of planning everything upfront?

**Answer:** ReAct is designed for situations where the next step depends on the result of the previous step. If you are searching for information, you cannot know what to search for next until you see the results of the first search. ReAct allows the agent to adapt dynamically. Plan-and-Execute is better when the full sequence of steps is known in advance and each step is independent of the others.

**Question 3:** What problem does the supervisor agent pattern solve?

**Answer:** The supervisor pattern solves the problems of scope, specialization, and scale. A single agent with all tools and all knowledge becomes unwieldy — the system prompt grows too long, the LLM struggles to choose among hundreds of tools, and one agent cannot have expertise in everything. The supervisor delegates to specialized sub-agents, each with a focused system prompt, a limited tool set, and deep expertise in one domain. The supervisor also handles conflict resolution when sub-agents produce contradictory results.

**Question 4:** What happens when a tool call fails in an agent system?

**Answer:** The agent system has several options depending on the retry policy: retry the same tool call with exponential backoff, try an alternative tool that provides similar functionality, ask the planner to generate a different approach, ask the user for guidance, or gracefully degrade by skipping the failed step and continuing with the remaining steps. The choice depends on the criticality of the failed step and the availability of alternatives.

**Question 5:** How does the agent manage its context window when working through a multi-step task?

**Answer:** The agent uses memory management strategies to stay within the context window. Common approaches include: summarization of older conversation turns, sliding window (keep the last N turns), key-value retrieval (store observations in a vector store and retrieve relevant ones), and structured state serialization (maintain a compact JSON representation of the plan and completed steps). The memory store is external to the LLM context and only the most relevant information is injected into the prompt at each step.
