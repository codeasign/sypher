The previous section showed how to build AI agent systems at scale. This section covers the hard trade-offs every architect must navigate, the most common failure modes, the anti-patterns that produce fragile systems, and the situations where you should not use an agent at all.

## The Core Tension

The central tension in AI agent architecture is autonomy versus control. More autonomous agents are more capable but less predictable. More controlled agents are safer but less flexible.

This tension is not a bug you fix — it is a design axis you navigate. Every agent system exists at some point on this spectrum, and the correct position depends on your tolerance for failure, your cost constraints, your user expectations, and the regulatory environment.

Consider two extremes. A fully autonomous agent for IT operations might scan your infrastructure, detect a failed service, restart it, analyze the root cause, apply a fix, and verify the fix — all without human intervention. When this works, it resolves incidents in under a minute instead of the 30 minutes a human would take. When it fails, it might restart the wrong service, apply an incorrect fix that makes the problem worse, or cascade into a larger outage. The autonomous agent is powerful but dangerous.

At the other extreme, a fully controlled agent for medical diagnosis might require a doctor to approve every patient interaction, every test recommendation, every diagnosis, and every treatment suggestion. The agent cannot act autonomously — it is a sophisticated suggestion engine. When this works, it reduces diagnostic errors by providing evidence-based recommendations. When it fails, a human catches the mistake. The controlled agent is safer but slower and less impactful.

The skill of agent architecture is finding the right point on this spectrum for your specific use case, not maximizing autonomy for its own sake.

### Autonomy Spectrum Detailed

The autonomy spectrum has four major zones. Each zone has different architectural characteristics.

**Zone 1: Deterministic Workflow (no autonomy).** The system follows a predefined flowchart. Every decision point has explicit rules. The LLM is not involved in decision-making. This is appropriate for compliance-critical processes where every action must be traceable to a rule.

**Zone 2: Recommend-Then-Execute (low autonomy).** The LLM analyzes the situation and recommends an action. A human reviews and approves the recommendation before execution. This is appropriate for high-stakes domains like healthcare, finance, and legal.

**Zone 3: Execute-Then-Review (medium autonomy).** The agent executes actions autonomously but logs everything for human review. Humans can override and roll back actions. This is appropriate for IT operations, content generation, and data analysis.

**Zone 4: Full Autonomy (high autonomy).** The agent operates independently. Humans are only involved when the agent explicitly requests help or when the guardrails trigger. This is appropriate for low-risk, high-volume tasks like customer support triage, calendar management, and information retrieval.

The tension manifests in every design decision. A more autonomous agent can handle unexpected situations, recover from errors, and work with ambiguous instructions. But it can also make unexpected tool calls, misinterpret user intent, produce non-deterministic outputs, and consume more tokens. A more controlled agent follows a predefined path, costs less, and behaves predictably, but it fails when the situation deviates from the expected patterns.

### The Cost of Autonomy

Choosing a higher autonomy level is not free. The costs of autonomy extend beyond token consumption to include operational risk, trust erosion, and reputational damage. Architects must weigh these costs against the benefits of increased capability.

**The Cost of a Wrong Decision**

When an autonomous agent makes a wrong decision, the impact falls into four categories:

| Cost Category | Description | Typical Range | Example |
|---|---|---|---|
| Direct financial | Revenue loss, fines, refunds | USD 10 to USD 100,000 per incident | Agent approves a fraudulent refund request |
| Operational | Manual fix effort, escalation time | 15 minutes to 8 hours per incident | Engineer diagnoses and rolls back a bad config change |
| Reputational | User trust erosion, brand damage | 6 to 24 months to recover | Agent sends incorrect billing statement to 5,000 customers |
| Regulatory | Compliance violations, audit findings, penalties | USD 5,000 to USD 1 million per incident | Agent mishandles personal data in a GDPR-regulated context |

Consider a concrete example. An e-commerce company deploys an autonomous customer support agent with the ability to issue refunds up to USD 500. The agent processes 10,000 refund requests per day. If the agent has a 99.5% accuracy rate, that means 50 incorrect refunds per day. At an average incorrect refund of USD 200, the daily financial loss is USD 10,000. Monthly: USD 300,000. The direct financial cost alone justifies adding human review for refunds above USD 50, even if that reduces the autonomy level.

**The Autonomy Accuracy Threshold**

There is a simple formula for determining whether autonomy is financially viable for a given action:

If (cost per incorrect action multiplied by the error rate) is greater than (cost of human review multiplied by the volume), then human review is cheaper than accepting the errors.

For example, if each incorrect refund costs USD 200 and the error rate is 0.5%, the expected cost of errors per action is USD 1.00. If a human review costs USD 0.25 per action, human review is the cheaper option. The agent should recommend refunds, not execute them.

This calculation changes as the error rate drops. If the error rate is 0.01%, the expected cost per action drops to USD 0.02, which is cheaper than the USD 0.25 human review. At that accuracy level, autonomous execution is financially sound.

**User Trust Erosion**

Trust erosion from agent mistakes is harder to quantify than direct financial costs, but it is often more damaging. A single bad experience with an agent can cause a user to distrust the system permanently.

Research on human-computer interaction shows that users who experience one agent failure are 4 times more likely to abandon the system entirely compared to users who never experience a failure. Users who experience two failures are 12 times more likely to abandon the system. The recovery rate is slow: even after 10 successful interactions following a failure, the abandonment rate remains 3 times higher than baseline.

This asymmetry means the cost of a mistake is not the cost of fixing the mistake but the long-term reduction in user engagement. A banking agent that incorrectly denies a loan application costs the bank the loan amount plus the lifetime value of the customer who leaves. If the customer has a lifetime value of USD 10,000, the real cost of that single incorrect denial is USD 10,000 plus the loan amount, not just the operational cost of the review.

**Reputational Damage at Scale**

Reputational damage from agent failures is amplified by social sharing. Users who experience an agent failure tell an average of 16 people. If the failure is notable enough to reach social media, the reach expands to thousands or millions.

A memorable example occurred at a travel booking company where an autonomous agent booked a non-refundable flight for a customer who explicitly requested a refundable ticket. The customer shared screenshots on social media. The story was picked up by three news outlets. The company's social media sentiment dropped by 18% over the following month. The cost of the refund was USD 850. The reputational damage was estimated at USD 2.3 million in reduced booking volume.

The multiplier effect means that high-visibility, high-confidence agent failures are catastrophically expensive. The only defense is to reduce the error rate to near zero for actions that affect user outcomes, or to reduce the autonomy level for those actions.

**The Trust Speed Penalty**

There is another hidden cost: speed reduces trust. When a human handles a task, users attribute delays to careful consideration. When an agent handles the same task instantly, users assume it was trivial and the agent cannot fail. This creates an expectation of perfection that no agent system can meet.

A well-designed agent system manages expectations by communicating uncertainty. When the agent is confident, it acts quickly and transparently. When the agent is uncertain, it slows down, explains why, and asks for confirmation. This pacing — fast for routine, slow for uncertain — aligns user expectations with agent capabilities and reduces the trust penalty when mistakes occur.

**Balancing Autonomy and Safety**

The recommended approach is graduated autonomy. Map every action the agent can take to a autonomy level based on risk, reversibility, and cost of failure:

Low-risk actions (read data, search, summarize) get full autonomy. The agent executes immediately without human review. The cost of a mistake is low and the action is reversible.

Medium-risk actions (create drafts, moderate content, categorize records) get conditional autonomy. The agent executes but logs the action for review. A human can roll back the action within a defined window. Most mistakes are caught before they reach the user.

High-risk actions (financial transactions, account changes, deletions, medical recommendations, legal determinations) get no autonomy. The agent recommends, a human approves, the agent executes. This is the recommend-then-execute pattern from the autonomy spectrum.

This graduated approach captures the cost savings of autonomy where it is safe while preventing catastrophic failures where the cost of a mistake is highest.

Consider the decision of whether the agent can choose which tool to call. An autonomous agent with free tool selection can discover creative solutions — it might find a workaround when the primary tool is unavailable. But it might also call the wrong tool, chain tools in unexpected ways, or attempt to call tools that should not be available in the current context. A controlled agent with explicit tool routing cannot discover creative solutions, but it will never call the wrong tool either.

This trade-off appears in every aspect of agent design: how the plan is generated, how tools are selected, how errors are handled, how memory is managed, how the agent communicates with the user. Each decision moves the system along the autonomy-control spectrum.

## Decision Matrix

Every agent architecture decision involves trade-offs. The matrix below maps the key decision points, their options, and the trade-offs.

| Decision | Option A | Option B | When to Choose A | When to Choose B |
|----------|----------|----------|------------------|------------------|
| Planning approach | Plan-and-Execute | ReAct | Steps are known upfront; dependencies are clear | Each step depends on previous results; exploration needed |
| Tool selection | LLM chooses | Explicit routing | Many tools, varied use cases | Few tools, fixed workflows |
| Memory strategy | Full history | Sliding window | Context window is large; accuracy critical | Cost is priority; context is limited |
| Error handling | Auto-retry | Human-in-loop | Transient errors; low risk | High-stakes actions; irreversible operations |
| Parallelism | Parallel tool calls | Sequential | Independent tasks; latency critical | Shared state; race conditions possible |
| LLM model | Single model | Tiered (fast + powerful) | Simple tasks; cost sensitive | Complex reasoning + many iterations |
| State storage | In-memory cache | Persistent database | Stateless sessions; short-lived | Long-running agents; durability needed |
| Scaling | Synchronous (blocking) | Asynchronous (queue) | Low latency expectations; simple tasks | High latency variance; background processing |

### Deep Dive on Each Decision

**Planning approach:** Plan-and-Execute is best when the steps are known at design time and the dependencies between steps are static. A report generator that always collects data from five dashboards and formats a PDF is a perfect Plan-and-Execute candidate. ReAct is best when the path depends on intermediate results. A debugging agent that searches logs, finds errors, researches solutions, and applies fixes needs ReAct because it cannot know what it will find in the logs until it looks.

**Tool selection:** LLM-based tool selection is flexible but unpredictable. The LLM may choose the wrong tool, choose a suboptimal tool, or attempt to call a tool with invalid parameters. Explicit routing uses business logic — "if the user asks about billing, call the billing tool" — which is predictable but brittle when new situations arise. A hybrid approach works well: explicit routing for known scenarios and LLM selection as a fallback for novel situations.

**Memory strategy:** Full history preserves all context at the cost of token consumption. A 50-turn conversation about a complex technical issue might consume 30,000 tokens of history alone. Sliding window drops older context but may lose important information. Summarization is a middle ground — compress old context into a summary while keeping recent context intact. The choice depends on how far back in the conversation the agent needs to refer.

**Error handling:** Auto-retry works for transient errors like network timeouts or database deadlocks. A retry with exponential backoff (500ms, 1s, 2s, 4s) resolves most transient failures. Human-in-loop is required for errors that suggest a fundamental problem — permission denied, data not found, invalid configuration. The agent should not retry these errors autonomously because retrying will not fix them and may make things worse.

**Parallelism:** Parallel tool calls cut latency dramatically. Five independent tool calls executed in parallel take the same time as the slowest call, not the sum of all calls. The downside is increased complexity: race conditions, partial failures, and result ordering. Sequential execution is simpler but slower. A good practice is parallel for read-only tools and sequential for tools that modify state.

**LLM model:** A single model approach uses the same LLM for planning, reasoning, and execution. A tiered approach uses a fast, cheap model (GPT-4o-mini, Claude Haiku) for routine tool calls and a powerful model (GPT-4, Claude Opus) for complex reasoning and planning. The tiered approach saves cost without sacrificing quality on hard problems. The trade-off is increased latency from routing between models.

**State storage:** In-memory cache (Redis) provides sub-millisecond state access and is suitable for short-lived agent sessions under 30 minutes. Persistent database (PostgreSQL, DynamoDB) provides durability across restarts and sessions. For long-running agents that operate for hours or days, persistent storage is mandatory. A common pattern is cache-aside: read from cache, write to both cache and database.

**Scaling:** Synchronous scaling means each agent request holds a connection open until the agent completes. This is simple but wastes resources during the 10-60 seconds the agent is running. Asynchronous scaling uses a task queue — the request returns immediately with a task ID, and the client polls for the result. This is better for production because workers can scale independently and the system can handle bursts without connection exhaustion.

Most production agent systems choose AP for the agent loop itself (the agent should continue responding even if state persistence is temporarily unavailable) but CP for critical operations (tool calls that modify external systems must be consistent).

### Practical Partition Scenarios

**Scenario 1: State store partition during planning.** The agent is generating a plan when the connection to the state store drops. With AP, the agent continues with locally cached state, completes the plan, and syncs to the state store when the connection is restored. With CP, the agent stops and returns an error. The AP approach means the user gets a response faster but may see a stale plan if the state was outdated. This is acceptable for most use cases.

**Scenario 2: State store partition during tool execution.** The agent executes a tool call that modifies an external system (creating a ticket, sending an email). The tool succeeds, but the state store is partitioned and the result cannot be persisted. With AP, the agent records the result in a local buffer and syncs later. If the agent crashes before the sync completes, the tool result is lost and the agent state becomes inconsistent with the external system. This is why write tool calls should always be CP: the agent must confirm the result was persisted before proceeding.

**Scenario 3: Multi-region partition.** The user session is in US-East, but the agent needs data from the EU-West state store, which is partitioned. With AP, the agent uses the last replicated snapshot from EU-West, which may be 30 seconds stale. With CP, the request fails. For most agent use cases, 30-second stale data is acceptable — the agent is serving a human who expects a response, not a real-time trading system.

## Cost vs. Quality Trade-off

Every additional agent iteration improves output quality — up to a point. The relationship between cost and quality is not linear. Early iterations deliver large quality gains. Later iterations produce diminishing returns. Understanding this curve is essential for designing cost-effective agent systems.

Proactive context management (summarization, sliding windows, structured state) directly reduces the per-iteration cost of later steps.

### Recommendation Framework

The following framework helps architects determine the optimal number of iterations for their use case. The three factors to balance are task complexity, cost sensitivity, and quality requirements.

**Simple tasks (low complexity, low cost sensitivity):**
These are tasks with a clear path to completion and few decision points. Example: "Look up the weather for Tokyo." The optimal iteration count is 1-2. More iterations add no value. Use a single LLM call or a deterministic workflow.

**Moderate tasks (medium complexity, moderate cost sensitivity):**
These tasks require a few tool calls and some decision-making. Example: "Research three competitors and summarize their pricing." The optimal iteration count is 3-5. This captures the steep part of the quality curve without entering the diminishing returns zone.

**Complex tasks (high complexity, lower cost sensitivity):**
These tasks require multiple tool calls, analysis, and iteration. Example: "Analyze our Q3 customer churn data, identify the top 3 reasons for churn, and propose remediation strategies." The optimal iteration count is 5-10. Beyond 10 iterations, quality gains are minimal and the cost compounds significantly.

**Research tasks (open-ended, lowest cost sensitivity):**
These tasks explore unknown territory. Example: "Investigate the current state of quantum computing error correction." The optimal iteration count is 10-25. These tasks benefit from exploration across multiple sources, but enforcing a hard limit prevents runaway costs.

### Cost Control Strategies

Apply these strategies to manage the cost-quality trade-off in production:

**Strategy 1: Early stopping.** Evaluate after each iteration whether the agent has produced a satisfactory result. If the answer is sufficient, stop immediately rather than completing the planned number of iterations. This is the most effective cost control technique and should be the default for all agent systems.

**Strategy 2: Model tiering.** Use a fast, cheap model for the first 2-3 iterations where the path is still being established. Switch to a more capable model only when the agent encounters a genuinely hard problem. A simple classifier can predict which requests need the powerful model based on the initial query complexity.

**Strategy 3: Iteration budgeting.** Assign a budget of max_iterations and max_cost to each request. Track both in real time. If the cost budget is exhausted before the iteration budget, stop. Cost budgeting is more effective than iteration budgeting because a single expensive iteration (large tool result, long reasoning chain) can consume the entire budget.

**Strategy 4: Caching intermediate results.** Cache deterministic tool results so the agent does not re-fetch the same data across iterations or across sessions. A cache hit saves the entire cost of the tool call and the associated LLM reasoning. For example, if the agent calls `search_web` with the same query in iterations 2 and 4, the second call should return the cached result.

**Strategy 5: Parallel exploration.** When the agent needs to explore multiple paths, run them in parallel rather than sequentially. Five parallel iterations complete in the same wall-clock time as one sequential iteration. The token cost is the same either way, but the user experience is dramatically better. Use parallel execution for independent sub-tasks and sequential execution for dependent steps.

## Failure Modes

AI agent systems fail in distinctive ways. Here are the six most common failure modes and how to address them.

### Failure Mode 1: Tool Hallucination

The LLM invents a tool call that does not exist in the registry. The tool name is similar to a real tool but slightly different, or the parameters are structurally valid but semantically nonsensical.

**Example:** The agent calls `search_knoweldge_base` (misspelled) instead of `search_knowledge_base`. The tool registry returns "unknown tool." The agent retries with the same misspelling, loops three times, and then errors out.

**Real-world impact:** At a mid-size SaaS company, tool hallucination caused 12% of agent runs to fail in the first week of deployment. The LLM consistently misspelled tool names, called deprecated tools, and passed parameters in the wrong format. Each failure wasted an average of 4 iterations before the agent gave up.

**Mitigation:** Use strict tool schema validation before execution. Implement fuzzy matching for tool names with a similarity threshold of 0.8 (Levenshtein distance). Return a helpful error message that includes the closest matching tool name and a list of valid parameters. Train the LLM on actual tool call patterns using few-shot examples in the system prompt.

### Failure Mode 2: Infinite Loop

The agent repeatedly takes the same action and gets the same result. This is most common when the agent tries to call a tool that returns insufficient data, but the agent believes it needs more data and calls the same tool again.

**Example:** The agent calls `search_web("latest news about NVIDIA")`. The search returns general results. The agent calls `search_web("latest news about NVIDIA 2025")`. Same results. The agent calls `search_web("NVIDIA news this week")`. Same results. This continues until max_iterations is hit.

**Real-world impact:** Infinite loops are the most common production incident in agent systems. At one company, 5% of all agent runs hit the max iteration limit because of looping behavior. Each loop consumed $0.42 in wasted LLM tokens. At 50,000 requests per day with 5% looping, the daily waste was $1,050.

**Mitigation:** Implement loop detection (same tool call within the last N iterations). Introduce a "novelty" check — if the same tool with the same parameters appears more than twice, force the agent to try a different approach or stop. Set a reasonable max_iterations limit (10-25 depending on use case). Add a "frustration" mechanism — after 3 identical tool calls, the system prompt injects "You have called this tool with the same parameters three times. Please try a different approach or admit that you cannot complete this task."

### Failure Mode 3: Cascading Tool Failures

One tool failure triggers a chain of dependent failures. The agent cannot recover because each step depends on the previous failed step.

**Example:** The database tool fails due to a connection timeout. The agent cannot retrieve user data. The analysis tool cannot run without user data. The report generator cannot produce output without the analysis. The entire agent run fails.

**Real-world impact:** Cascading failures are particularly damaging because a single upstream failure can cause hundreds of downstream agent runs to fail. During a 15-minute database outage at a financial services company, 2,300 agent runs failed because the first step (retrieve portfolio data) failed. The cascading failures consumed $8,500 in wasted LLM tokens as agents tried and retried.

**Mitigation:** Implement graceful degradation for each tool. If the database is unavailable, the agent should use a cached version of the data, or skip the data-dependent steps and produce a partial report. Use circuit breakers for each tool to prevent cascading failures. If a tool has failed more than 5 times in a 60-second window, the circuit breaker opens and the agent immediately receives a "tool unavailable" response without attempting the call.

### Failure Mode 4: Context Overflow

The agent accumulates too many observations, tool results, and reasoning steps, exceeding the LLM's context window. The LLM either truncates input (losing critical information) or produces an error.

**Example:** A research agent searches 15 different sources, collecting 2,000 tokens of content from each. Total context: 30,000 tokens. Plus conversation history: 5,000 tokens. Plus system prompt and tool definitions: 3,000 tokens. Total: 38,000 tokens. The LLM's context window is 32K tokens. The beginning of the conversation is silently dropped, including the original user request.

**Real-world impact:** Context overflow is silent and insidious. The LLM does not report that it dropped the beginning of the conversation. The agent continues reasoning without the original user request, the early tool results, or the plan. Responses become incoherent. At one company, 8% of agent responses were flagged as "confused" by users before the root cause (context overflow) was identified.

**Mitigation:** Implement proactive context management. Summarize old observations before adding new ones. Use a structured state representation instead of raw conversation history. Set a hard limit on the number of tool results stored in context (maximum 10 tool results in full, summarize older ones). Monitor context utilization as a metric and alert when it exceeds 80%.

### Failure Mode 5: Permission Escalation

The agent uses a chain of tool calls to perform an action that individual tools would not allow. This is the agent equivalent of a privilege escalation attack.

**Example:** The agent cannot directly call `delete_user(user_id=42)` because the tool has a permission check. Instead, the agent calls `get_user_by_email(email)` to find the user, then `deactivate_account(user_id=42)` to disable the account, then `purge_user_data(user_id=42)` to remove the data. The combined effect is equivalent to deleting the user, but each individual tool call passed its permission check.

**Real-world impact:** Permission escalation is the most dangerous failure mode because it involves actual data loss, not just wasted tokens. In a documented incident at a CRM platform, an agent sequentially called "export_contacts," "delete_contacts," and "purge_recycle_bin" to permanently delete 50,000 customer records. Each individual tool call was authorized. The combined effect was a data loss incident that took 6 hours to recover from backups.

**Mitigation:** Implement cross-tool policy enforcement. Track the combined effect of tool calls in a session. If the combined effect of sequential tool calls would violate a policy, block the last tool call and flag the session for review. Log all tool call chains for audit. Implement a "tool call pattern" detector that identifies known escalation patterns and blocks them preemptively.

### Failure Mode 6: Agent Stuck

The agent reaches a state where none of its available tools can make progress, but the agent does not recognize this and continues trying. This is different from an infinite loop — the agent is trying different things, but nothing works.

**Example:** The agent is asked to "find the CEO's email address." The available tools are `search_web` and `company_directory`. The search returns nothing useful. The directory does not have email addresses. The agent tries different search queries, then tries the directory with different parameters, then tries searching for the company's SEC filings, then tries looking up the company's LinkedIn page. None of these tools have email addresses. The agent keeps trying.

**Real-world impact:** "Stuck" agents are frustrating for users because the agent appears to be working — it is making tool calls, analyzing results, and reasoning. But it is making no progress toward the goal. Users wait 30-60 seconds for a response and then receive "I could not find this information." The wasted time and tokens could have been avoided if the agent recognized the dead end earlier.

**Mitigation:** Implement a "tool exhaustion" check. If the agent has tried all available tools and none produced a useful result, the agent should stop and report "I cannot find this information with the tools available to me." This should be explicitly stated in the system prompt. Also implement a "stagnation" detector — if the agent has not completed any task in the last 3 iterations, trigger a replanning step or ask the user for guidance.

## Anti-Patterns Hall of Shame

These anti-patterns appear in production agent systems and consistently cause problems. Recognizing them early saves months of debugging.

**Fix:** Decompose into specialized agents. Each agent has a focused system prompt (under 2,000 words) and a limited tool set (under 20 tools). The supervisor pattern is the natural solution here — the supervisor decides which specialized agent to invoke, and each specialist operates within its narrow domain.

**Anti-Pattern 2: The Silent Agent.** The agent runs entirely in the background with no feedback to the user. The user types a query and waits 30 seconds with no indication of progress. The user assumes the system is broken and refreshes, which starts a new agent run. The user then sees two simultaneous agent runs, both consuming tokens and both producing partial results.

**Real-world impact:** A SaaS company deployed an agent without streaming. User satisfaction scores dropped by 40% in the first week. Users reported that the system "felt broken" even though the agent was working correctly. The company added streaming, and satisfaction scores returned to baseline within 3 days.

**Fix:** Stream intermediate results. Show the agent's plan, current step, and tool call results as they happen. Use WebSockets or Server-Sent Events for real-time updates. Even a simple "thinking..." indicator during LLM calls improves perceived performance. The goal is to show the user that progress is being made, even if the final answer is not ready yet.

**Anti-Pattern 3: The Optimistic Agent.** The agent assumes every tool call will succeed and proceeds as if all results are available. When a tool fails, the agent has no fallback and produces a response that references data it never received. This produces confident-sounding nonsense — the worst kind of LLM failure because the user trusts the response.

**Real-world impact:** A financial analytics agent assumed a stock price tool call succeeded and produced a report with price targets, trend analysis, and buy/sell recommendations — all based on data that the tool never returned. The agent hallucinated the stock prices. The compliance team flagged the report and shut down the agent deployment. The company spent 6 weeks adding error handling before re-deploying.

**Fix:** Implement explicit error handling at every step. The agent should check that each tool result is valid before proceeding. Use structured output parsing with validation. Every tool call must return a status field. If the status is not "success," the agent must not use the result for further reasoning. The system prompt should include explicit instructions: "If a tool call fails, do not assume the result. Ask for help, try an alternative tool, or report that the data is unavailable."

**Anti-Pattern 4: The Amnesiac Agent.** The agent has no memory across sessions. Every conversation starts from scratch. The user has to re-explain their context, preferences, and history every time. The agent asks "what is your account number?" in every conversation, even though the user provided it in the previous conversation.

**Real-world impact:** A customer support agent was deployed without persistent memory. Users reported extreme frustration at having to repeat their account details, issue description, and troubleshooting steps in every interaction. The average conversation length was 15 messages because the agent kept asking questions that were answered in previous conversations. After adding persistent memory, the average conversation length dropped to 5 messages and user satisfaction improved by 35%.

**Fix:** Implement persistent memory. Store user profile information, conversation summaries, and key facts in a database. Load relevant context at the start of each session. Use a vector store to retrieve relevant facts from past conversations. Include a "user context" section in the system prompt that summarizes the user's known information — account type, previous issues, preferences, and history.

**Anti-Pattern 5: The Unbounded Agent.** The agent has no iteration limit, no cost limit, and no time limit. A single agent run can continue for 500 iterations, consume 2 million tokens, and cost $15. The user gets a response that is 50 pages long for a simple question. The company gets a $15 bill for a single interaction.

**Real-world impact:** An early AutoGPT user set the agent to "research the history of the Roman Empire" with no iteration limit. The agent ran for 6 hours, consumed 3.5 million tokens, and generated a 47,000-word document. The cost was $52. The user stopped reading after the first paragraph. The agent had no way to know that the user expected a 500-word summary, not a doctoral thesis.

**Fix:** Set hard limits for every agent run. Enforce max_iterations, max_tokens, max_cost, and max_duration. Return partial results when limits are exceeded. Configure limits based on the use case: a simple Q&A agent should have max_iterations=5 and max_tokens=10,000, while a research agent might have max_iterations=25 and max_tokens=100,000. Always set a cost limit — even $1 per request is a reasonable upper bound for most use cases.

**Anti-Pattern 6: The Lone Wolf Agent.** A single agent handles everything with no human oversight. The agent makes decisions, takes actions, and communicates results without any human review. When the agent makes a mistake, there is no safety net. The mistake propagates to the user, the database, and potentially to external systems.

**Real-world impact:** A recruitment platform deployed an agent that could autonomously reject job applicants. The agent misread a resume, classified a qualified candidate as "unqualified," and sent a rejection email. The candidate appealed to the company's CEO on LinkedIn. The incident became a PR problem. The company added human review for all rejection decisions and apologized to the candidate.

**Fix:** Implement human-in-the-loop for high-risk actions. Define risk levels for each tool call. Low-risk (read-only queries) are automatic. Medium-risk (data modifications) require confirmation. High-risk (financial transactions, account changes, rejections, deletions) require explicit approval from a human supervisor. The risk level should be defined in the tool metadata, not in the system prompt — the LLM cannot be trusted to correctly assess the risk of its own actions.

## When NOT to Use This

AI agent architecture is powerful but not universally applicable. Consider these situations where you should avoid it.

**Simple deterministic tasks:** If a task can be expressed as a fixed sequence of steps with no branching, a traditional workflow engine is cheaper, faster, and more reliable than an AI agent. Example: "Every night at 2 AM, export the database, compress it, upload it to S3, and send a notification." Use a cron job, not an agent.

**Human safety-critical decisions:** If a mistake could cause physical harm, death, or serious financial loss, do not use an autonomous agent. Agent systems can make unpredictable decisions, hallucinate tool calls, and operate outside their intended scope. Use deterministic systems with human review. Example: "Control the dosage of a medical infusion pump based on patient vitals." Use a certified medical device, not an AI agent.

**Fully predictable workflows:** If the exact sequence of steps is known at design time and never changes, a hardcoded workflow is simpler and more maintainable. Example: "When a user signs up, create an account, send a welcome email, and add them to the CRM." Use a state machine or BPMN workflow, not an agent.

**Extremely high throughput, low latency:** If you need to process 100,000 requests per second with sub-100ms latency, an agent is the wrong tool. The LLM calls alone add 500ms to 5,000ms of latency. Example: "Validate credit card transactions in real time." Use a rules engine, not an agent.

**Regulated environments with strict audit requirements:** If every decision must be traceable to a deterministic rule, the non-deterministic nature of LLM-based agents is a liability. Example: "Determine loan eligibility under Regulation Z." Use a rule-based system, not an agent. An agent may apply inconsistent criteria to different applicants, and it cannot explain its decisions in terms of specific regulations. Regulators require deterministic, auditable decision processes.

**Systems with high consistency requirements:** If every request must produce the same result given the same inputs, an agent is the wrong choice. LLM-based agents produce non-deterministic outputs — the same query with the same tools can produce different responses at different times due to sampling temperature, model updates, and context variability. Example: "Calculate the sales tax for a purchase in California." Use a deterministic tax engine, not an agent.

**Systems where you cannot accept the cost per request:** If your business model requires $0.001 per request cost, an agent system will not work. A single agent iteration consumes thousands of tokens and costs $0.01-$0.05. Multi-iteration agents cost $0.10-$1.00 per request. If your revenue per user interaction is measured in fractions of a cent, use a rules-based system or a deterministic model. Example: "Classify each page view as bot or human." Use a simple ML model, not an agent.

**Teams without observability infrastructure:** An agent system without logs, traces, and metrics is an unmanageable system. If your team does not have the infrastructure to capture and analyze agent behavior, do not deploy agents. You will be flying blind. Example: A startup with no monitoring and no logging deploys an agent and cannot debug why it is producing wrong answers. The agent is turned off after two weeks.
