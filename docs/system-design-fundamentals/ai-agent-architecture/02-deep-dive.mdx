## Data Structures and Algorithms

AI agent systems rely on several well-defined data structures and algorithms. Understanding these is essential for designing, debugging, and scaling agent systems.

### The ReAct Loop

The ReAct loop is the most fundamental algorithm in agent systems. It is a state machine that cycles through thought, action, and observation until a stopping condition is met.

```
Algorithm: ReAct Loop

Input:  user_query, tool_registry, max_iterations
Output: final_response

state = {
    goal: user_query,
    history: [],
    plan: null,
    iteration: 0,
    is_done: false
}

while not state.is_done and state.iteration < max_iterations:
    state.iteration += 1

    // Step 1: Think
    thought = llm.generate(
        system_prompt = build_system_prompt(tool_registry),
        messages = state.history + user_query,
        temperature = 0.2
    )

    // Step 2: Act
    if thought.contains_tool_call():
        tool_name = extract_tool_name(thought)
        tool_args = extract_tool_args(thought)
        tool_fn = tool_registry.get(tool_name)

        // Step 3: Observe
        try:
            result = tool_fn.execute(tool_args)
            observation = {tool: tool_name, status: "success", output: result}
        except Exception as e:
            observation = {tool: tool_name, status: "error", error: str(e)}

        state.history.append(observation)

    else:
        // No tool call — produce final response
        state.is_done = true
        final_response = thought

    // Check for loop safety
    if detect_infinite_loop(state.history):
        state.is_done = true
        final_response = "I was unable to complete this task. Please try rephrasing."

return final_response
```

### Tool Registry

The tool registry is a lookup table that maps tool names to their implementations, schemas, and metadata. The registry is the single source of truth for all tool-related information in the agent system. Every tool must be registered before the agent can call it, and every tool call is validated against the registry.

The registry supports tool discovery — agents can query the registry to find tools that match a specific capability. This enables dynamic tool selection where the agent discovers available tools at runtime rather than receiving a fixed list in the system prompt. Tool discovery is particularly useful in multi-agent systems where different agents have access to different tool sets.

The registry also supports tool versioning. When a tool's implementation changes, the registry maintains the previous version for a configurable period. This allows the agent to continue using the old version while the new version is tested. Tool versioning is essential for A/B testing tool changes and for rolling back problematic updates.

Each tool entry in the registry includes:

### Message History Buffer

The message buffer maintains the conversation history for the LLM context. Since LLMs have finite context windows, the buffer must be managed carefully.

```python
class MessageBuffer:
    def __init__(self, max_tokens=128000):
        self.max_tokens = max_tokens
        self.messages = []
        self.total_tokens = 0

    def add_message(self, role, content):
        tokens = count_tokens(content)
        self.messages.append({"role": role, "content": content, "tokens": tokens})
        self.total_tokens += tokens

        while self.total_tokens > self.max_tokens:
            removed = self.messages.pop(0)
            self.total_tokens -= removed["tokens"]

    def summarize(self):
        """Compress old messages into a summary, keeping recent messages intact."""
        if len(self.messages) < 10:
            return

        old_messages = self.messages[:-6]
        recent_messages = self.messages[-6:]

        summary = llm.summarize(old_messages)

        self.messages = [
            {"role": "system", "content": f"Summary: {summary}", "tokens": count_tokens(summary)}
        ] + recent_messages
        self.total_tokens = sum(m["tokens"] for m in self.messages)
```

## The Request Lifecycle

A complete agent request goes through 12 distinct stages. Each stage has specific inputs, outputs, error modes, and observability hooks.

### Stage 1: Request Ingestion

The orchestrator receives the HTTP request, validates the user authentication, checks rate limits, and creates a session. The session ID is returned to the caller for asynchronous polling.

### Stage 2: Context Assembly

The agent assembles the context for the first LLM call. This includes the system prompt, tool definitions, conversation history from the memory store, and any relevant documents retrieved from a knowledge base.

```
Context Assembly:
  1. Load system prompt template
  2. Inject tool descriptions (from tool registry)
  3. Load conversation history (from memory store)
  4. Retrieve relevant documents (from vector store)
  5. Assemble into a single prompt
  6. Token count check (truncate if necessary)
```

### Stage 3: Goal Decomposition

The planner receives the assembled context and generates a plan. The plan may be a simple sequence of steps or a complex DAG of interdependent tasks.

### Stage 4: Task Scheduling

The plan is converted into a schedule. Independent tasks are assigned to parallel execution slots. Dependent tasks are queued behind their prerequisites. The scheduler produces an execution plan:

```
Batch 0: [Task 1: lookup_holdings, Task 3: search_industry_benchmarks]
Batch 1: [Task 2: fetch_earnings(depends on Task 1)]
Batch 2: [Task 4: calculate_pe_ratios(depends on Tasks 2, 3)]
Batch 3: [Task 5: flag_52_week_lows(depends on Task 4)]
Batch 4: [Task 6: generate_summary(depends on Tasks 2, 5)]
```

### Stage 5: Tool Selection

For each task, the executor selects the appropriate tool from the registry. The tool is chosen based on the task description, the tool's capabilities, and any constraints (cost, latency, availability).

### Stage 6: Tool Invocation

The executor constructs the tool call payload, validates the parameters against the tool's input schema, and invokes the tool. The tool execution may be synchronous (blocking) or asynchronous (fire-and-forget with callback).

### Stage 7: Observation Capture

The tool result is parsed and stored as an observation. The observation includes the tool name, the parameters used, the raw output, a success or error status, execution duration, and token count if applicable.

### Stage 8: State Update

The agent state is updated with the new observation. The task status is set to complete. Dependencies are recalculated. Newly unblocked tasks are added to the ready queue.

### Stage 9: Reasoning Step

The LLM receives the accumulated context — the original goal, the plan, completed task results, and the current state — and produces the next reasoning step. This may be a tool call for the next task or a final response.

### Stage 10: Loop Decision

The loop controller decides whether to continue or stop. Conditions for stopping include: all tasks complete, maximum iteration limit reached, a critical error that cannot be recovered, or a user cancellation request.

### Stage 11: Response Synthesis

If all tasks are complete, the orchestrator collects the accumulated observations and passes them to the LLM for final response synthesis. The LLM produces a natural language answer that incorporates the tool results. The orchestrator may also inject a quality assurance step — a second LLM call that reviews the response for accuracy, completeness, and safety before returning it to the user.

### Stage 12: Session Persistence

The final response is returned to the user. The session state, including the plan, all observations, and the final response, is persisted to long-term storage. Metrics are emitted for observability. The persistence step is critical for auditability — if a user later disputes the agent's response, the full session trace must be available for review.

## Consistency and State

AI agent systems face several failure modes related to consistency and state management. Understanding these is critical for building reliable systems.

### Tool Call Failure

When a tool call fails, the agent must decide how to proceed. The options are:

1. **Retry:** Retry the same tool call with exponential backoff. Use this for transient failures like network timeouts or rate limits.
2. **Fallback:** Try an alternative tool that provides similar functionality. Use this when a specific data source is unavailable but an alternative exists.
3. **Replan:** Ask the planner to generate a different approach. Use this when the failed step is critical and no alternative tool exists.
4. **Skip:** Mark the task as skipped and continue. Use this for non-critical steps where the agent can still produce a useful response.
5. **Abort:** Stop the entire agent run and return an error. Use this for security failures, permission errors, or data corruption.

### Infinite Loop Detection

An agent can get stuck in a loop if it repeatedly takes the same action and gets the same result. Detection strategies include:

```
def detect_infinite_loop(history, window_size=5):
    """Detect if the agent is repeating the same tool call pattern."""
    if len(history) < window_size * 2:
        return False

    recent = history[-window_size:]
    previous = history[-window_size*2:-window_size]

    # Check if the same sequence of tool calls repeats
    recent_tools = [h.get("tool") for h in recent if h.get("type") == "tool_call"]
    previous_tools = [h.get("tool") for h in previous if h.get("type") == "tool_call"]

    if recent_tools == previous_tools:
        return True

    # Check for identical consecutive tool call results
    if len(history) >= 3:
        last_three = history[-3:]
        if all(h.get("tool") == last_three[0].get("tool") for h in last_three):
            return True

    return False
```

### Parallel Tool Call Racing

When multiple tool calls execute in parallel, the agent must handle race conditions. If two tools both update the same state, the results may be inconsistent. Solutions include:

- **Immutable state:** Each tool call produces a new version of state rather than modifying the existing state.
- **Merge strategy:** Define a deterministic merge function for parallel results.
- **Sequential critical sections:** Force serial execution for tools that modify shared state.

## Performance Characteristics

Understanding the performance characteristics of AI agent systems is essential for capacity planning and cost management.

### Tokens per Agent Turn

Each agent loop iteration consumes tokens for the prompt (input) and the generated response (output). Typical consumption:

| Component | Input Tokens | Output Tokens | Total |
|-----------|-------------|---------------|-------|
| System prompt | 500 - 2,000 | 0 | 500 - 2,000 |
| Conversation history | 500 - 10,000 | 0 | 500 - 10,000 |
| Tool definitions | 200 - 1,000 | 0 | 200 - 1,000 |
| LLM response | 0 | 100 - 500 | 100 - 500 |
| Tool result (next iteration) | 100 - 2,000 | 0 | 100 - 2,000 |
| **Per iteration total** | **1,300 - 15,000** | **100 - 500** | **1,400 - 15,500** |

### Latency Breakdown

The latency of an agent request is dominated by three components: planning time, tool execution time, and LLM generation time.

```
Total Latency = Planning + Sum of Loop Iterations
  where each iteration = LLM Generation + Tool Execution + Observation Parse

Typical latencies:
  Planning: 500ms - 3,000ms (single LLM call)
  LLM Generation: 500ms - 5,000ms per iteration
  Tool Execution: 50ms - 10,000ms per call (varies by tool)
  Observation Parse: 5ms - 50ms

  Total for 5-iteration agent: 5s - 60s
  Total for 25-iteration agent: 25s - 300s
```

### Loop Iteration Count Distribution

The number of iterations an agent requires depends on task complexity. Real-world distributions from production systems:

```
Simple tasks (Q&A, single tool call):     1 - 2 iterations
Moderate tasks (multi-step reasoning):    3 - 8 iterations
Complex tasks (research, analysis):       8 - 20 iterations
Very complex tasks (report generation):   20 - 50 iterations
```

Approximately 80% of production agent tasks complete in 10 or fewer iterations. Long-running agents (over 25 iterations) should be designed for asynchronous execution with polling or webhook callbacks.

## Configuration and Tuning

AI agent systems expose several configuration parameters that control behavior, cost, and reliability.

### Core Parameters

**max_iterations:** The maximum number of loop iterations before the agent stops. Setting this too low causes incomplete responses. Setting it too high wastes tokens and increases latency. Recommended range: 10 for simple agents, 25 for moderate agents, 50 for complex agents.

**temperature:** The LLM sampling temperature. Lower values (0.0 - 0.2) produce more deterministic, focused reasoning. Higher values (0.3 - 0.7) produce more creative, exploratory behavior. For agent systems, low temperatures are generally preferred for reliability.

**tool_timeout:** The maximum time to wait for a tool call to complete. Individual tool calls should have timeouts independent of the agent loop timeout. Recommended range: 5,000ms for fast tools (calculators, simple queries) to 60,000ms for slow tools (web searches, file processing).

**retry_policy:** Controls how the agent handles tool call failures. Typical configuration:

```json
{
  "retry_policy": {
    "max_retries": 3,
    "base_delay_ms": 500,
    "max_delay_ms": 10000,
    "backoff_factor": 2.0,
    "retryable_errors": ["timeout", "rate_limit", "service_unavailable"],
    "non_retryable_errors": ["permission_denied", "invalid_input", "not_found"]
  }
}
```

**memory_window_size:** The number of recent conversation turns to keep in the LLM context. Older turns are either discarded or summarized. Recommended range: 6 - 20 turns for most use cases.

### Advanced Parameters

**parallel_tool_calls:** The maximum number of tool calls that can execute in parallel. Setting this to 1 forces sequential execution. Higher values enable parallelism but increase complexity. Recommended range: 1 - 5. The optimal value depends on the independence of the tool calls and the rate limits of the external APIs. For a system that searches three databases simultaneously, 3 parallel calls cut total latency by 66%.

**stop_on_error:** When true, the agent stops immediately on any tool call error. When false, the agent attempts recovery. Production systems typically set this to false with a circuit breaker pattern. The circuit breaker prevents cascading failures — if 50% of tool calls in a 60-second window fail, the circuit opens and no new agent runs are started until the issue is resolved.

**cost_limit:** The maximum cost (in tokens or dollars) for a single agent run. When exceeded, the agent stops and returns partial results. This prevents runaway costs from buggy agent loops. For a production system processing 500,000 requests per day, a single buggy agent that hits a $10 cost limit instead of the expected $0.35 costs $4,800 per day in excess LLM spend.

**model_selection:** Different LLMs have different capabilities for agent tasks. Fast models (GPT-4o-mini, Claude Haiku) are suitable for simple tool calls. Powerful models (GPT-4, Claude Opus) are needed for complex reasoning and planning. Some systems use a tiered approach: a fast model for execution and a powerful model for planning.

### Tuning Methodology

Tuning an agent system is an empirical process. Follow these steps:

1. **Establish a baseline:** Deploy with conservative settings (max_iterations=10, temperature=0.1, tool_timeout=10s).
2. **Collect data for one week:** Gather metrics on completion rate, iteration count, cost, latency, and user satisfaction.
3. **Identify the bottleneck:** Are agents hitting max_iterations? Are tool calls timing out? Are users dissatisfied with response quality?
4. **Adjust one parameter at a time:** Change max_iterations from 10 to 15 and measure the impact on completion rate and cost. If completion rate improves by 10% but cost increases by 30%, the trade-off may not be worth it.
5. **A/B test each change:** Run the old and new configurations side by side for at least 1,000 requests per variant.
6. **Rinse and repeat:** Tuning is ongoing. As the LLM models improve, the optimal parameters change.
