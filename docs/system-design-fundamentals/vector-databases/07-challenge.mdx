## Tier 1 — Foundation

### Challenge 1: Explain ANN to a Non-Technical Stakeholder

Your product manager asks: "Why can't we just use our existing Postgres database for the new search feature? It already has all our products. I don't want to add another database."

**Time limit:** 5 minutes.

**Write a 200-word explanation** that the PM can understand — no jargon without explanation, no algorithms without context. Focus on the "why" not the "how."

**Evaluation criteria:**
- Explains the difference between keyword matching and semantic matching in plain language
- Uses a clear analogy (library, filing cabinet, etc.)
- Gives a concrete example of what breaks without a vector database
- Doesn't use terms like "HNSW", "ANN", "embedding dimension" without explanation

**Model answer:**
"Imagine you have a filing cabinet with 10 million product descriptions. Right now, when someone searches for 'cozy winter outfit,' the cabinet can only find files that literally contain those exact words — 'cozy,' 'winter,' and 'outfit.' It can't find anything that says 'warm wool coat' or 'insulated jacket' because those are different words, even though they mean the same thing.

A vector search system is like organizing the filing cabinet by meaning instead of by exact words. Every product gets a 'meaning fingerprint' — a mathematical representation of what it actually is. When someone searches, the system converts their search into a meaning fingerprint too, then finds all the products with matching fingerprints.

Postgres is great at finding exact matches, filtering by price, and joining tables. But it can't understand meaning. The new system we're adding works alongside Postgres — Postgres stays the source of truth, and the vector system just handles the 'find things that mean the same thing' part. Together they give us both exact precision and semantic understanding."

### Challenge 2: Spot the Index Type

For each scenario, identify which index type (HNSW, IVF, PQ, or brute-force) is the best choice and explain why.

**Scenario A:** An image similarity search for an art museum catalog. 50,000 images. Users expect results in under 50ms. The catalog runs on a single server with 16GB RAM. Maximum recall is required.

**Time limit:** 5 minutes.

**Model answer:** Brute-force. 50,000 vectors × 512 dims × 4 bytes = 100MB. Fits entirely in RAM. Brute-force gives exact results (100% recall) with ~5ms latency at this scale. Any ANN index would add complexity and approximation with no performance benefit.

**Scenario B:** A product recommendation engine for an e-commerce site. 100 million product vectors. 768 dimensions. 5,000 QPS. p99 latency must be under 20ms. You have a budget for 10 servers with 64GB RAM each.

**Time limit:** 5 minutes.

**Model answer:** HNSW. 100M vectors × 768 dims × 4 bytes = 300GB raw, approximately 470GB with HNSW overhead. 10 servers × 64GB = 640GB total, which fits. HNSW gives the lowest latency (2-10ms) and highest recall (0.95+) of any ANN index at this scale. IVF would be slower, and PQ would lose too much recall for recommendations.

**Scenario C:** A code search index for 5 billion code snippets. Must fit on a server cluster with 512GB total RAM. Recall above 0.85 is acceptable.

**Time limit:** 5 minutes.

**Model answer:** IVF with PQ (compressed). 5B vectors × 768 dims × 4 bytes = 15TB raw — impossible to fit uncompressed. PQ compression at 32 bytes per vector = 160GB. IVF provides cluster-based search that works well with compressed vectors. Recall of 0.85-0.90 is acceptable for code search where the application typically returns multiple candidates.

### Challenge 3: Trace the Query

A user searches for "affordable noise-canceling headphones." Write out every step of the query lifecycle from the user typing to results being displayed. Include:
- What happens in the application layer
- What the embedding service does
- What the vector database does internally
- How the metadata filter is applied
- How the result is returned to the user

**Time limit:** 5 minutes.

**Model answer:**
1. User types "affordable noise-canceling headphones" in search bar and hits Enter.
2. Application sends the query text to the embedding service (REST endpoint).
3. Embedding service tokenizes the query, runs it through the sentence-transformer model, returns a 768-dimension vector: [0.31, -0.52, 0.78, ..., 0.14].
4. Application packages the query vector with filters: `{ max_price: 200, category: "electronics" }` and sends to the vector database search API.
5. Vector database query planner receives the request. It determines which shards to search (all active shards) and which filter strategy to use (pre-filter by category then search, because "electronics" eliminates 80% of products).
6. Each shard's HNSW index is searched: entry point → greedy navigation through top layer → descend to lower layers with ef=300 → collect 100 candidates.
7. Metadata filter applied: of 100 candidates, 72 are in the "electronics" category with price under $200. The rest are discarded.
8. Each shard returns its top 20 filtered results to the coordinator.
9. Coordinator merges results from all shards, sorts by distance score, takes top 10.
10. Application receives IDs: [prod_2384, prod_9102, ...] with scores: [0.94, 0.91, ...].
11. Application fetches full product records from PostgreSQL by ID.
12. Results rendered as search results page. Total time: 85ms (12ms embedding + 68ms vector search + 5ms fetching).

## Tier 2 — Design

### Challenge 1: Design a Vector Search System for a Podcast Transcript Search

**Scenario:** You're building a podcast platform with 5 million episodes, each 30-60 minutes long. Users want to search for specific topics discussed within episodes — "find me episodes where they talk about Kubernetes security best practices."

**Constraints:**
- 5 million episodes (average 10,000 words each = ~1 trillion total words)
- 100,000 daily active users
- Each user searches 5 times per day
- p99 latency under 200ms
- Must support filtering by date, podcast name, and episode duration
- 5 servers available (each 32GB RAM, 8 cores)

**Time limit:** 15 minutes.

**Your design must include:**
- How you chunk the transcripts for embedding
- Your embedding model choice and why
- Index type and rationale
- Query flow
- Filtering strategy
- How you handle episodes being added daily
- What breaks at this scale and how you mitigate it

**Model answer:**

**Chunking:** Each episode transcript is split into 256-token chunks with 64-token overlap. At 10,000 words per episode (approximately 13,000 tokens), that's ~65 chunks per episode. Total: 5M × 65 = 325 million chunks.

**Embedding model:** Use a sentence-transformer model (all-MiniLM-L6-v2, 384 dimensions) because it's fast enough for real-time inference and requires less memory than 768-dim models. The smaller dimension means our index is half the size.

**Index type:** IVF with PQ compression. 325M vectors × 384 dims × 4 bytes = ~500GB raw. HNSW would need ~750GB — doesn't fit in 5 × 32GB = 160GB of RAM. IVF + 32-byte PQ compression reduces each vector to 32 bytes: 325M × 32 = 10.4GB. With IVF overhead, ~15GB. Fits in remaining RAM after OS overhead.

**Query flow:**
1. User query embedded with same model (384-dim vector).
2. Query sent to coordinator → broadcast to all 5 nodes.
3. Each node searches IVF index (nprobe=20) + PQ reconstructed distances.
4. Each node applies filters (date range, podcast name, duration) on top 50 candidates.
5. Coordinator merges results from 5 nodes → return top 20.

**Filtering strategy:** Pre-filter by podcast name (hash index — fast exact match). Post-filter by date range and duration (slower range checks applied after search). Rationale: podcast name filter is highly selective; date and duration filters are less selective.

**New episodes:** Daily ingestion pipeline: new transcripts → chunk → embed → insert into a "pending" buffer. Every 6 hours, merge pending buffers into the main index and trigger IVF cluster reassignment.

**What breaks at this scale:**
- IVF cluster drift: As new vectors are added, the k-means clusters become stale. Mitigation: reassign clusters weekly.
- PQ quality degradation: Compression becomes lossy. Mitigation: monitor recall; if below 0.85, rebuild index with updated centroids.
- Embedding latency at peak: 100K DAU × 5 queries = 500K queries/day = ~6 QPS average. Peak: 30 QPS. Embedding model on a single GPU handles this (300+ QPS).

### Challenge 2: Design a RAG Pipeline for a Legal Document Review Platform

**Scenario:** A legal tech platform helps lawyers review millions of documents for discovery. Lawyers search by natural language descriptions of what they're looking for: "Find all documents related to the merger agreement between Company A and Company B from Q3 2023 that discuss earn-out provisions."

**Constraints:**
- 10 million legal documents (PDFs, emails, contracts)
- 500 power users (lawyers), each performing 50 searches per day
- p99 latency under 500ms (lawyers are patient but not that patient)
- 100% result accuracy required (legal discovery has zero tolerance for missing relevant documents)
- Must support filtering by date range, document type, parties involved, and privilege status

**Time limit:** 15 minutes.

**Model answer:**

**Architecture:** This is a recall-critical system. I'd use a two-tier approach: vector search for broad candidate retrieval, then a cross-encoder re-ranker for precision. The vector database casts a wide net; the re-ranker ensures nothing relevant is missed.

**Chunking with overlap:** Documents are chunked at 512 tokens with 128-token overlap. Why the overlap? Legal terms often span chunk boundaries ("earn-out provisions" in the first chunk, "as defined in section 4.2" in the next). Overlap ensures the re-ranker sees complete contexts.

**Embedding:** Use a legal-domain fine-tuned model (Legal-BERT or similar). General text embeddings miss legal terminology — "consideration" means something very different in a contract than in everyday language.

**Index:** HNSW with recall optimized settings (M=32, ef_construction=800, ef=500). This increases memory but recall approaches 0.99. At 10M documents × 65 chunks × 768 dims = ~2TB vectors. With 3x replication, that's 6TB. Need ~60 nodes with 64GB RAM each. Expensive, but the requirement demands it.

**Two-stage retrieval:**
1. Vector search with ef=500 returns top 200 candidates per query.
2. A cross-encoder model (slower but more accurate) re-ranks the 200 candidates, producing calibrated relevance scores.
3. Top 50 re-ranked results returned to the lawyer.

**Filtering:** Date range and document type filters are applied at the metadata level during vector search (pre-filter because they're highly selective). Party name and privilege filters are applied after re-ranking (post-filter because they require document-level context that cross-encoders provide).

**Result accuracy guarantee:** Run 100 benchmark queries from the existing document review system weekly. Compare vector search + re-ranker results to known ground truth. Alert if recall@100 drops below 0.99.

**Fallback:** If the embedding service or vector database is unavailable, fall back to Elasticsearch keyword search. The lawyers get lower-quality results but still get results.

### Challenge 3: Design a Feed Recommendation System for a Social Media Platform

**Scenario:** You're building the content recommendation feed for a social media platform. Each post needs to be matched to users based on semantic similarity to their interests. Posts include text, images, and video.

**Constraints:**
- 500 million monthly active users
- 100 million new posts per day
- Each user sees 100 posts per feed load
- p99 latency under 100ms
- Must handle personalization per user (each user gets different results for the same query)
- Budget: must not exceed $50,000/month in infrastructure

**Time limit:** 15 minutes.

**Model answer:**

**Multimodal embeddings:** Use CLIP for images and a text embedding model for captions. Fuse them into a single 512-dim vector per post. Text-only posts get only the text embedding. Video posts get frame embeddings averaged.

**Two-tier personalization:**
1. A lightweight collaborative filtering step identifies the user's interest clusters based on past engagement.
2. Vector search then finds posts close to those interest cluster centroids.

This avoids per-user embedding generation — we don't embed each user as a vector, we embed content and cluster users by their engagement history.

**Index:** HNSW for each content cluster. 100M posts per day × 30 days = 3 billion active vectors. With 512 dims: 3B × 512 × 4 = 6TB raw. With HNSW overhead: ~9TB. Doesn't fit in $50K/month (9TB RAM costs ~$27K/month at $3/GB on cloud providers, plus compute). Need PQ compression.

PQ at 64 bytes/vector: 3B × 64 = 192GB. With HNSW overhead: ~250GB. $750/month for RAM. This is the right trade-off for a feed recommendation system where 0.95 vs 0.90 recall is unnoticeable.

**Query flow:**
1. User opens app.
2. Server retrieves user's interest clusters from Redis (pre-computed daily).
3. Query vector = mean of user's top 3 interest cluster centroids.
4. Vector database returns 500 candidate post IDs from relevant clusters.
5. Server applies engagement-based re-ranking (boost posts from followed accounts, demote posts already seen).
6. Top 100 posts returned in feed.

**Cost breakdown:**
- 3 node cluster (128GB RAM each): $1,500/month
- PQ-compressed index on SSD: 250GB × $0.08/GB = $20/month
- Embedding pipeline (batch, GPU instance): $1,000/month
- Redis cluster for user interests: $800/month
- Remaining budget for Kubernetes, networking, monitoring: ~$1,680/month
- Total: ~$5,000/month — well under $50K budget.

## Tier 3 — Expert

### Challenge 1: Multi-Tenant Vector Search at Global Scale

**Scenario:** You're building a vector search platform that serves 500 tenant companies. Each tenant has their own data, their own embedding model preferences, their own metadata schemas, and their own performance requirements. One tenant might have 5 vectors, another might have 5 billion. You need a shared infrastructure that isolates tenants from each other and lets them configure their own index parameters.

**Constraints:**
- 500 tenants
- Tenant sizes range from 100 vectors to 5 billion vectors
- Each tenant can choose their own embedding model and vector dimensionality
- Each tenant needs their own metadata schema
- A noisy tenant (high QPS, bad queries) must not impact other tenants
- Tenant A must not be able to access Tenant B's data
- Must support multiple cloud regions
- You have a startup-level team of 5 engineers

**Time limit:** 30 minutes.

**Evaluation dimensions:**
- Multi-tenancy model: per-tenant isolated index vs shared index with tenant ID partitioning
- Isolation strategy: noisy neighbor handling, resource quotas, QoS
- Schema management: how you handle 500 different metadata schemas
- Cost allocation: how you bill tenants
- Operational complexity: what your 5-person team can actually maintain
- Security: tenant data isolation at every layer

**Multiple valid approaches with trade-offs:**

**Approach A — Per-tenant isolated indexes:** Each tenant gets their own index, potentially on their own dedicated nodes. Pros: perfect isolation, per-tenant indexing tuning, easy billing. Cons: small tenants waste resources (an HNSW index with 100 vectors has terrible recall), operational complexity of managing 500 indexes.

**Approach B — Shared index with tenant ID partition:** All tenants share one index. Tenant ID is the first filter. Pros: efficient for small tenants (vectors from many tenants build a better graph), simpler operations. Cons: noisy tenant impact, harder billing, metadata schema must be unified.

**Model answer — Hybrid approach:**
- Small tenants (under 100K vectors) share a "small tenant cluster" with tenant ID as a required filter dimension. Their vectors combined produce a high-quality HNSW graph.
- Large tenants (over 100K vectors) get dedicated clusters. They pay for the infrastructure.
- Tenant isolation at the query layer: coordinator enforces tenant ID from authentication token, not from query parameters (can't be spoofed).
- Noisy neighbor protection: per-tenant QPS quotas enforced at the router. If a tenant exceeds their quota, queries are queued with a 200ms max wait, then rejected.
- Schema management: Use a document-oriented metadata store (one per tenant) rather than a fixed schema. Tenants send JSON metadata, queries use a simple filter grammar (field:value, field:range). No schema enforcement at the storage layer — enforce at the application layer instead.

**Implementation plan for a 5-person team:**
- Month 1-2: Single-tenant prototype with FAISS (one engineer)
- Month 3-4: Multi-tenant coordinator with quotas (two engineers)
- Month 5-6: Tenant management API, billing, dashboard (one engineer)
- Month 7-8: Multi-region support (one engineer)
- Avoid building your own storage — use S3 for persistent vector storage. Avoid building your own scheduling — use Kubernetes. Focus engineering effort on the multi-tenant coordination layer.

### Challenge 2: Real-Time Vector Search with Streaming Updates

**Scenario:** You're building a real-time content moderation system. Videos are uploaded, live-streamed, and must be checked against a database of known violating content (copyrighted material, violent content, spam). New violating content is identified every minute and must be searchable immediately — a 1-minute delay in being able to detect a new violating video could mean 10,000 more views.

**Constraints:**
- 1 billion known violating vectors (growing at 1,000/minute)
- 10,000 new videos uploaded per second
- Each new video must be checked within 500ms of upload
- New violating vectors must be searchable within 30 seconds of identification
- The system must maintain high recall (over 0.95)
- Must operate 24/7 with no downtime for indexing
- Running on a fixed cluster of 100 nodes

**Time limit:** 30 minutes.

**Evaluation dimensions:**
- Real-time index updates: how to insert 1,000 vectors/minute without stopping queries
- Delta index strategy: how to merge incremental updates into the main index
- Query scaling: 10,000 checks/second on a fixed cluster
- Recall maintenance: how to prevent index drift with continuous streaming updates
- Detection latency: how to minimize the time between a vector being added and it being searchable

**Model answer — Two-tier streaming architecture:**

**Tier 1 (Hot):** An in-memory buffer holding the last 5 minutes of new violating vectors (5,000 vectors). This buffer is searched FIRST — brute-force O(n) because 5,000 vectors × 10K QPS is trivially fast (under 1ms per query). Vectors in the hot tier are searchable within 1 second of ingestion.

**Tier 2 (Warm):** The main HNSW index holding all 1 billion known violating vectors. Updated every 5 minutes via background index rebuild. Daily updates are batched and applied in bulk.

**Query flow:**
1. New video uploaded → extract fingerprint vector.
2. Search hot buffer (brute-force, 5,000 vectors, 0.5ms).
3. Search warm index (HNSW, 1B vectors, 50ms).
4. Merge results from both, check if any violate.
5. If match found above threshold → flag for review / auto-block.
6. Total latency: ~55ms — well under 500ms.

**Delta indexing:**
- Every 5 minutes: merge the hot buffer (5,000 vectors) into the warm index.
- The warm index rebuilds in the background on a separate set of nodes.
- While rebuilding, the old warm index continues serving queries.
- After rebuild completes (5-10 minutes for 1B vectors), swap the old index for the new one.
- Result: continuous search availability, no downtime, vectors are searchable in under 1 minute from identification.

**Cluster optimization:**
- 80 nodes for the warm HNSW index (sharded, each holding ~12.5M vectors).
- 10 nodes for the hot buffer (each fully replicating the hot buffer for throughput).
- 10 nodes for the index rebuild pipeline (rotating — 5 building, 5 ready for swap).
- Query routing: each query goes to 1 hot node (random) and all warm shards in parallel.

**Handling duplicate checks:** A video being checked triggers hash-based dedup at the router. If the same video ID is checked twice within 1 second, the second check is queued with the first result.

**Recovery from cold start:** If a warm index node crashes, the remaining nodes redistribute the shard load (reduced throughput but still operational). The crashed node's shard is rebuilt from the last snapshot in S3 and joins the cluster within 2 minutes.
