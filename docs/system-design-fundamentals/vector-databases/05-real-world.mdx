## Company Case Studies

### Case Study 1: Notion — AI-Powered Search (2023)

**The scale:** 100 million+ documents across 20 million workspaces. 50,000 new documents created per hour.

**The problem:** Notion's keyword-based search couldn't find semantically related content. A user searching for "Q4 marketing strategy" wouldn't find a document titled "Holiday Campaign Plans" because they shared no keywords. Users complained that "Notion search is broken" on Twitter and Reddit daily.

**What they built:** Notion added an AI-powered search feature using vector embeddings. Every document is embedded using a sentence-transformer model (768-dimensional vectors) when created or updated. The embeddings are stored in a vector database (Pinecone) alongside the document metadata (workspace ID, permissions, tags).

**Key lessons:** Notion had to handle per-user access control filtering at query time. Each search query includes a filter for the user's workspace ID, which means the vector database must support efficient filtered search. They chose Pinecone for its managed filtering capabilities and built-in embedding pipeline.

### Case Study 2: Uber — Marketplace Fraud Detection (2022)

**The scale:** 25 million trips per day. 5 million driver-partners. 100+ cities.

**The problem:** Fraudsters were creating fake accounts and fake trips to launder money. Traditional rule-based fraud detection was too slow to adapt — fraudsters would find a new pattern, exploit it for 48 hours before the rule was updated, and cost Uber millions. The patterns were too diverse for simple rules: fake GPS locations, fake trip routes, fake identities.

**What they built:** Uber's fraud detection team built a vector-based similarity system. Each trip is embedded into a 128-dimensional vector that captures its "behavioral fingerprint" — route, timing, payment method, device fingerprints, GPS trajectory. New trips are compared against known fraud patterns using vector similarity.

### Case Study 3: Netflix — Content Recommendation (2023)

**The scale:** 260 million subscribers. 18,000+ titles. 1 billion+ recommendation requests per day.

**The problem:** Netflix's recommendation system was a complex ensemble of collaborative filtering, content-based filtering, and trending signals. But it struggled with the "cold start" problem — new titles with no watch history couldn't be recommended effectively. Users browsing "dark comedies" were shown popular titles instead of semantically similar ones.

**What they built:** Netflix added a vector-based similarity layer that computes semantic embeddings for every title using its plot description, genre tags, user reviews, and visual features from the trailer. The vector database powers the "More Like This" and "Because You Watched" sections.

**Key lessons:** Netflix's dataset is small (18,000 titles) but the embedding quality is paramount. They spent more engineering effort on the embedding fusion pipeline than on the vector database itself. The vector database is the simplest part of the system — the hard part is generating good vectors.

### Case Study 4: GitHub — Code Search with Copilot (2024)

**The scale:** 100 million+ repositories. 50 million developers. 1 billion+ code search queries per month.

**The problem:** GitHub's existing code search was keyword-based — it could find exact matches for function names and variables, but couldn't find code that *does the same thing* with different implementation. A developer searching for "parse JSON from HTTP response" wouldn't find code using different libraries or variable names.

**What they built:** GitHub Copilot Code Search uses a code-specific embedding model (CodeBERT-based) to embed code snippets. Developers can search by natural language description and find semantically similar code across the entire public repository index.

  │  4. Top 50 results returned                                    │

  ┌──────────────────────────────────────────────────────────────────┐
  │  - Semantic code search quality: 89% recall@10 over keyword     │

  │  - p99 latency: 200ms (including embedding + search + re-rank) │
  └──────────────────────────────────────────────────────────────────┘
`} />

**Key lessons:** GitHub's scale (5 billion vectors) forced them to use PQ compression — the uncompressed index would be 6TB, far too expensive for in-memory storage. They also use a two-stage ranking: vector search for candidate retrieval, then a more expensive cross-encoder model for re-ranking the top 50 results.

## Open Source Implementations

| System | Index Type | Language | Notable Feature | Best For |
|--------|-----------|----------|----------------|----------|
| **FAISS** (Facebook/Meta) | HNSW, IVF, PQ | C++/Python | GPU-accelerated search. SIMD-optimized distance computation. | Research, offline batch processing, building custom solutions |
| **Milvus** | HNSW, IVF, DiskANN | Go/C++ | Distributed architecture. Cloud-native. Supports GPU. | Production deployments needing horizontal scaling |
| **Qdrant** | HNSW | Rust | Extremely fast single-node performance. Advanced filtering. | Applications needing high throughput per node |
| **Weaviate** | HNSW | Go | Built-in embedding modules. GraphQL API. | Teams wanting an all-in-one solution with minimal setup |
| **pgvector** | IVFFlat, HNSW | C (PostgreSQL extension) | ACID transactions. SQL queries. | Already using PostgreSQL, smaller datasets (under 10M vectors) |

## Cloud Provider Implementations

| Provider | Service | How It Differs From Self-Hosted | Cost/Ops | Lock-In |
|----------|---------|--------------------------------|----------|---------|
| **AWS** | OpenSearch (k-NN plugin) | Runs on existing OpenSearch clusters. Limited to IVF and NMSLIB. | No separate infrastructure. Pay for OpenSearch nodes. | Moderate — OpenSearch is widely used, but you're tied to AWS for the managed service. |
| **GCP** | Vertex AI Vector Search (formerly Matching Engine) | Fully managed. Sharded by default. Supports billion-scale. | Expensive ($10K+/month for production). Zero ops overhead. | High — uses Google's proprietary infrastructure. |
| **Azure** | Azure Cognitive Search (vector) | Integrated with Azure AI. Built-in chunking and embedding. | Pay per query plus storage. Good for Azure-native stacks. | Moderate — if you're already on Azure, the integration is valuable. |
| **Pinecone** | Managed Vector DB | Serverless. No index management. Built-in embedding via integration. | Pay per pod + storage. Most expensive per vector. | High — proprietary format. No self-hosted option. |

## Lessons Learned

1. **The embedding model matters more than the vector database.** Every company we studied spent more engineering effort on getting good embeddings than on optimizing the vector database. A bad embedding model + a perfectly tuned index still gives bad results.

2. **Filters are the hardest problem.** Every production use case needs metadata filtering (by workspace, region, language, etc.). Filtering is where most vector databases struggle. Test your filtering patterns before committing to a vendor.

3. **Hybrid search is almost always necessary.** Pure vector search misses exact matches (that keyword search finds easily). The best results come from combining vector and keyword search with a fusion layer.

4. **Re-indexing is not optional.** Index drift is real and gradual. Without scheduled re-indexing, recall degrades silently. Schedule re-indexing based on insert volume, not calendar time.

5. **Start with pgvector, graduate to a vector database.** For most teams, pgvector in PostgreSQL is sufficient for the first 6-12 months. Only move to a dedicated vector database when you hit the limits — and by then you'll have a much better understanding of your requirements.
