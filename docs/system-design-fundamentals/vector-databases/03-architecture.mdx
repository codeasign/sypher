## Naive Implementation

The simplest possible vector database is a flat list of vectors stored in memory, searched with brute force:

```
vectors = [(id1, vec1), (id2, vec2), ..., (idN, vecN)]
search(query):
  for each (id, vec) in vectors:
    score = cosine_similarity(query, vec)
  return top_k by score
```

This works for 10,000 vectors. At 100,000 vectors, it starts to feel slow (50-100ms per query). At 1 million, it's unusable for real-time applications (500ms+ per query).

### Breakdown Points

**At 1,000 users (10,000 vectors, 1 search/sec):**
- Brute force works fine. Latency: 5-10ms. Memory: 50MB.
- No indexing needed. A simple Python script with numpy is sufficient.

**At 100,000 users (1M vectors, 100 searches/sec):**
- Brute force latency: 500ms — unacceptable for real-time.
- Memory: 4.7GB for HNSW — still fits on one machine.
- Without an index: searches take 100x longer than needed, users time out.
- Without batching: 100 concurrent searches overwhelm a single thread.

**At 10 million users (100M vectors, 10,000 searches/sec):**
- Single machine can't hold the HNSW index in memory (470GB+).
- Need distributed architecture: sharding, replication, partitioning.
- Without sharding: you hit the memory ceiling of a single machine.
- Without replication: one machine failure takes down the entire search service.

## Integration Patterns

### Integrating with a Relational Database

The most common pattern: use PostgreSQL for transactional data and a vector database for semantic search.

<AsciiDiagram title="Dual Database Pattern — PostgreSQL + Vector DB" content={`
  Application
  ───────────
       │
       ├─── Write: INSERT INTO products (id, name, price, category, vector)
       │           → PostgreSQL stores the row
       │           → PostgreSQL also stores the vector (pgvector extension)
       │             OR → Application sends vector to separate vector DB
       │
       └─── Search: POST /search { query: "warm winter coat", filters: { price: { $lt: 200 } } }
                    → Application embeds query text → vector
                    → Application sends vector + filters to vector DB
                    → Vector DB returns top 100 matching IDs
                    → Application queries PostgreSQL WHERE id IN (ids)
                    → Returns full records to user
  
  Watch out for:
  ┌──────────────────────────────────────────────────────────────────┐
  │  - Consistency: Vector DB and PostgreSQL can get out of sync     │
  │  - Solution: idempotent upserts, periodic reconciliation job     │
  │  - Latency: The extra round-trip to PostgreSQL adds 2-5ms        │
  │  - Solution: cache full records in Redis for hot items           │
  └──────────────────────────────────────────────────────────────────┘
`} />

### Integrating with a Cache Layer

For high-throughput applications, add Redis between the vector DB and the application.

         │

  Watch out for:
  ┌──────────────────────────────────────────────────────────────────┐

**Consistency trade-off:** Cross-region replication is always asynchronous. You can't have synchronous replication across continents because the speed of light imposes a minimum 60ms round-trip time between US and Europe. Every write committed to the primary is visible in the replica only after 1-2 seconds. Applications must tolerate this lag or route writes to the primary region.

## Capacity Planning

**Scenario:** Design a vector database for 10 million daily active users, each performing 10 searches per day. Each search needs to scan 1,000 product descriptions.

### Step 1: Estimate QPS

```
Daily searches = 10M users × 10 searches/day = 100M searches/day
Peak QPS = 100M / (24 × 3600) × 5 (peak factor) = 100M / 86400 × 5 ≈ 5,800 QPS
```

### Step 2: Calculate Storage Requirements

```
Vectors: 1,000 products × 768 dims × 4 bytes = 3MB per user × 10M = 30TB
Wait — that's wrong. You're not storing per-user vectors. You're storing 1,000 product vectors shared by everyone.

Total vectors: 1,000 (products) × 1 million (product variations) = 1 billion vectors
Each vector: 768 dims × 4 bytes = 3,072 bytes
Raw storage: 1B × 3KB = 3TB
HNSW index overhead: 3TB × 1.5 (index overhead) = 4.5TB
Metadata: 1B × 200 bytes = 200GB
Total: ~4.7TB

Memory (HNSW in RAM): 4.7TB — cannot fit on a single machine.
With 64GB RAM per node: 4.7TB / 64GB = 74 nodes
With 3x replication: 74 × 3 = 222 nodes
```

### Step 3: Determine Network Bandwidth

```
Query vector: 768 dims × 4 bytes = 3KB per query
Peak traffic: 5,800 QPS × 3KB = 17MB/s inbound
Result size: 10 results × 1KB each = 10KB per response
Outbound traffic: 5,800 × 10KB = 58MB/s
```

### Step 4: Size the Compute

```
Each query requires:
- HNSW graph traversal: 4ms
- Distance computation: 4ms
- Metadata filtering: 1ms
- Total per query: ~10ms on a single core
- With 5,800 QPS: need 58 cores minimum
- With 4x safety margin: 232 cores
- 16-core nodes: 15 nodes (plus 222 storage nodes)
```

## Deployment and Operations

### Zero-Downtime Deployment

1. **Rolling updates:** Update one replica at a time. The coordinator routes traffic away from the being-updated node.
2. **Blue-green deployment:** Stand up a new cluster, build the index, switch traffic via DNS or load balancer. Takes 1-2 hours for index build but is completely safe.
3. **Canary deployment:** Route 1% of traffic to the new version, monitor for 15 minutes, then ramp to 100%.

### Monitoring

| Metric | What It Tells You | Alert Threshold | Action |
|--------|-------------------|-----------------|--------|
| p50/p99 search latency | Overall system health | p99 > 100ms | Check index fragmentation, increase nodes |
| Recall rate | Index quality | < 0.90 | Trigger re-index |
| QPS per node | Load distribution | > 80% of max | Add shards or replicas |
| Memory usage | Index fit | > 90% of RAM | Add nodes, or switch to PQ compression |
| WAL lag | Write throughput | > 30s | Add write capacity, optimize batching |
| Segment count | Index fragmentation | > 100 per node | Trigger compaction |

### Debugging Common Issues

**High latency, low QPS:** Check if the index is fragmented — too many small segments. Run compaction to merge segments.

**Low recall:** Check if ef (query parameter) is set too low. Increase from 200 to 500. If that doesn't help, the embedding model may be producing poor quality vectors.

**OOM (out of memory):** The HNSW index is too large for the node. Either add more nodes (re-shard) or switch to a memory-compressed index type (IVF with PQ).

**Write amplification:** High write throughput is causing constant index rebuilds. Batch writes and use a larger segment size to reduce rebuild frequency.

Proceed to the Trade-offs page to understand the hard choices — what you're giving up when you choose a vector database over alternatives.
