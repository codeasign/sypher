# Rate Limiting in Production

Production rate limiting is where theory meets reality. Every major platform has built custom rate limiting systems, and the patterns they've converged on reveal the hard-won lessons of operating at scale. This page examines real-world implementations from Twitter, GitHub, Stripe, and Cloudflare, then surveys open-source tooling and cloud-native solutions.

---

## 1. Twitter (X) — Per-Endpoint, Per-User Tiered Limits

Twitter rate limits every API endpoint independently, and user tiers determine the ceiling. The core design separates read limits from write limits and applies sliding windows at the user level.

### Key Design Decisions

**Read vs. Write separation.** Twitter assigns different quota pools to read and write operations. A user can fetch 1000 timeline entries but only post 300 tweets in the same window. This prevents a write-heavy user from starving their own read capacity.

**Endpoint-specific counters.** The search endpoint has a lower limit than the timeline endpoint because search is computationally more expensive. This granularity lets Twitter maximize throughput on cheap operations while protecting expensive ones.

**Tier multipliers as a scaler.** Rather than hard-coding separate limits for every tier, Twitter applies a multiplier to a base limit. Free tier is 1x, standard is 2x, premium is 5x. New endpoints get a single base limit; the tier system scales automatically.

**Rate limit headers in every response.** Every API response carries `x-rate-limit-limit`, `x-rate-limit-remaining`, and `x-rate-limit-reset`. Clients can introspect their remaining quota without speculative retries, which dramatically reduces 429 traffic.

## 2. GitHub API — Unauthenticated vs Authenticated Limits

GitHub enforces two distinct rate limit regimes: one for unauthenticated requests and a much higher one for authenticated requests. This is the simplest and most common pattern in production APIs.

### Key Design Decisions

**Authentication as the primary discriminator.** GitHub's rate limit is fundamentally a lever to encourage authentication. Unauthenticated access is intentionally low (60 req/h) — enough for exploration, unusable for production. This drives API consumers to register apps and use tokens.

**Conditional request optimization.** GitHub pioneered the use of HTTP conditional requests (`If-None-Match`, `If-Modified-Since`) as a rate-limit escape hatch. When a resource hasn't changed, GitHub returns a `304 Not Modified` that does not count against the rate limit. For polling clients, this can reduce rate-limit consumption by 90% or more.

**Three-tier authentication model.** Personal access tokens (5,000 req/h), OAuth tokens (15,000 req/h), and GitHub App installation tokens (30,000 req/h) form a clear upgrade path. Each tier corresponds to a higher level of integration and trust.

**GraphQL cost model.** Unlike REST endpoints that count each request as one, GitHub's GraphQL API uses a point system. Each query specifies a cost, and the rate limit is expressed in points per hour. This allows expensive queries (deeply nested, many nodes) to consume more quota than simple queries, preventing abuse while rewarding efficient queries.

## 3. Stripe — Per-API-Key and Per-Account Rate Limits

### Key Design Decisions

**Dual-layer defense.** A single compromised API key hitting Stripe at 100 req/s can exhaust the key's limit without affecting other keys on the same account — because the per-key layer catches it. But if every key is operating near its limit, the per-account layer provides a circuit breaker for the entire account. This layered approach is especially important for platforms where multiple services or developers share the same Stripe account.

**Uneven rate allocation.** Not all API endpoints share the same limit. Stripe uses a "rate limit cost" model similar to GitHub's GraphQL approach: creating charges costs more than listing charges, and listing charges with expanded objects costs more than a simple list. The cost is documented in the response so clients can observe and adapt.

**No sliding window — token bucket.** Stripe explicitly uses a token bucket rather than a sliding window because it is simpler to reason about and implements easily in a distributed counter. The trade-off is that burst behavior is less smooth, but for payment APIs, predictable limit enforcement is more important than smoothness.

**Error body with actionable guidance.** Unlike minimal 429 responses, Stripe returns a structured JSON error body with a `"type": "rate_limit_error"` and a `"doc_url"` pointing to rate-limit best practices. This reduces support tickets by giving developers self-service troubleshooting.

### Key Design Decisions

**Local counters with periodic sync.** Each edge data center maintains a local counter for the current window. When a request arrives, the edge checks its local counter first — if it is below the limit, the request passes immediately without RPC overhead. Local counters sync to a central aggregator every few seconds. This design achieves sub-millisecond per-request overhead while maintaining global consistency within a few seconds.

**"Rate Trail" auto-learning.** Cloudflare's Rate Trail mode observes traffic patterns over a configurable period (usually 24 hours) and automatically selects a rate limit threshold based on the 95th percentile of legitimate traffic. This eliminates the need for operators to guess the "right" limit — Cloudflare learns it.

**Multiple enforcement actions.** Rate-limited requests don't have to be dropped. Cloudflare supports: block (return 429), JS challenge (browser must solve a challenge), captcha, log-only (observe without enforcing), and dynamic throttling (reject only if origin is already overloaded). For API traffic, block is typical; for browser traffic, a challenge weeds out bots without blocking real users.

**Automatic origin protection.** Cloudflare monitors the origin server's response rate. If the origin starts returning a high rate of 429s or 5xx errors, Cloudflare automatically activates rate limiting — even if no explicit rule was configured — to protect the origin from collapse. This is a circuit breaker at the CDN level.

│  │  Key: {consumer_id}:{route_id}:{window_type}:{start_time}    │      │
│  │  Value: counter (INCR + EXPIRE in Lua)                       │      │
│  └────────────────────────────────────────────────────────────────┘      │
└──────────────────────────────────────────────────────────────────────────┘
  `}
/>

Kong's plugin supports six time windows simultaneously (second, minute, hour, day, month, year) and enforces the most restrictive. The default policy is `local` (in-memory counters per Kong node), but production deployments use `redis` for cluster-wide consistency.

### Envoy — Local and Global Rate Limiting

Envoy Proxy offers two rate limiting modes: local (in-process token bucket per worker thread) and global (external rate limit service via gRPC).

**Local rate limiting** is configured via the `envoy.extensions.filters.http.local_ratelimit` filter. It uses a per-thread token bucket for low-latency enforcement — no network calls, sub-microsecond overhead. Suitable for per-connection or per-route limits where approximate enforcement is acceptable.

**Global rate limiting** delegates the decision to an external rate limit service (Lyft's `envoyproxy/ratelimit`) over gRPC. The service is typically backed by Redis and uses a fixed-window algorithm. The request is paused, the service is queried, and the response (OK or OVER_LIMIT) determines whether the request proceeds or is rejected. The added latency is typically 5-15ms.

The two modes compose naturally: local limits work as a first-pass check (cheap, fast, approximate), and global limits provide cross-cluster enforcement (expensive, slower, exact).

### Redis at the Core

Redis is the backbone of virtually every production rate limiting system. The `INCR` + `EXPIRE` pattern is the simplest and most common approach:

```redis
MULTI
  INCR rate_limit:{user_id}:{endpoint}:{window_start}
  EXPIRE rate_limit:{user_id}:{endpoint}:{window_start} {window_size}
EXEC
```

If the returned value exceeds the threshold, the request is denied. This Lua-scripted or pipelined approach runs in under 1ms for most deployments.

For sliding windows, a more sophisticated approach uses sorted sets:

```redis
ZREMRANGEBYSCORE rate_limit:{user_id} 0 {now - window}
ZCARD rate_limit:{user_id}
ZADD rate_limit:{user_id} {now} {unique_id}
EXPIRE rate_limit:{user_id} {window}
```

The `ZCARD` result gives the count of requests in the current sliding window. This is more memory-intensive (stores one element per request) but produces exact sliding-window enforcement.

## 6. Cloud Rate Limiting Services

Every major cloud provider offers managed rate limiting. The designs differ in placement and granularity.

### AWS WAF Rate Limiting

AWS WAF rate limits at the Application Load Balancer (ALB) or CloudFront level, before traffic reaches application servers.

- **Rate-based rules**: Define a threshold (e.g., 2,000 requests per 5-minute window) matching a specific web request condition.
- **Source IP aggregation**: Counts are grouped by source IP by default, or by a custom request forwarding header.
- **Auto-blocking duration**: Once a rate limit is breached, the source is blocked for the remainder of the window. No gradual backoff — hard cutoff.
- **ACL integration**: Rate limit rules live inside WAF ACLs alongside other rules (SQL injection, XSS, IP reputation). Evaluation order is user-defined.

The primary limitation of AWS WAF rate limiting is the 5-minute minimum window — you cannot enforce per-second or per-minute limits through WAF alone. For sub-minute granularity, you need an application-level rate limiter.

### GCP Cloud Armor

Cloud Armor provides rate limiting at the Google Cloud HTTP Load Balancer edge, with per-second granularity.

- **Rate limiting rules**: A rule can specify `rate_limit_threshold` as a count per client per second.
- **Enforcement action**: Enforce on the client (return 429) or enforce on the entire backend (concurrency-based backpressure).
- **Exceed action**: On exceeding, Cloud Armor can redirect to a custom error page, reCAPTCHA challenge, or simply block.
- **Adaptive protection**: Machine learning-based anomaly detection that learns normal traffic patterns and auto-generates rate limiting rules when deviations are detected.

Cloud Armor's per-second granularity makes it suitable for API abuse protection, whereas AWS WAF's 5-minute minimum is better suited for volumetric DDoS defense.

### Azure Rate Limiting

Azure rate limiting is primarily implemented in API Management (APIM) rather than at the network edge.

- **Product-level limits**: An Azure API Management product can have a rate limit (requests per time period) and a quota (total requests).
- **Per-subscription key**: Limits are enforced per subscription key, not per IP. This aligns with Azure's API-commerce model where different subscription tiers map to different limits.
- **Policy-based**: Rate limiting is configured as an XML policy injected into the API pipeline:

```xml
<rate-limit calls="100" renewal-period="60" remaining-calls-variable-name="remainingCalls" />
```

- **Concurrency limit**: Azure APIM also supports a concurrent request limit (`<limit-concurrency>`) which caps the number of in-flight requests from a single subscription — useful for preventing connection pool exhaustion.

## 7. Lessons Learned — 5 Key Patterns

Across these real-world systems, five patterns consistently emerge as the most important to get right.

### Pattern 1: Layered Defense

No single rate limiting layer is sufficient. The most robust systems enforce limits at multiple levels:

| Layer | Example | What It Protects |
|-------|---------|-----------------|
| Edge/CDN | Cloudflare, Cloud Armor | DDoS, volumetric abuse |
| API Gateway | Kong, AWS WAF | Endpoint-level abuse |
| Application | Stripe per-key | Fine-grained per-user |
| Database | Redis INCR+EXPIRE | Traffic surges to storage |

A request that passes through all layers unscathed is a legitimate request. A request blocked at the edge never touches the application server — saving CPU, memory, and database connections.

### Pattern 2: Informative Rate Limit Responses

Every major platform returns rate limit state in response headers. The minimum viable set:

- **`x-rate-limit-limit`** — the quota for the current window
- **`x-rate-limit-remaining`** — how many requests remain
- **`x-rate-limit-reset`** — the Unix timestamp when the window resets
- **`Retry-After`** — how many seconds to wait before retrying

Without these headers, clients must guess when to retry. With them, clients can implement intelligent backoff, display remaining quota to users, and avoid the thundering herd that follows a window reset.

### Pattern 3: Different Algorithms for Different Constraints

No single rate limiting algorithm is universally best. The trade-offs matter:

| Algorithm | Use Case | Key Trade-Off |
|-----------|----------|---------------|
| Token Bucket | Bursty traffic (social media APIs) | Simple, allows short bursts |
| Sliding Window | Consistent rate enforcement (Stripe) | Smooth but memory-intensive |
| Fixed Window | Simple per-minute limits (Kong) | Edge burst at window boundary |
| Sliding Window Log | Exact enforcement (Redis sorted set) | O(n) memory per request |
| GCRA | Carrier-grade precision (Cell rate algorithm) | Complex implementation |

Match the algorithm to the traffic pattern. Social feeds tolerate short bursts; payment APIs do not.

### Pattern 4: Graceful Degradation and Circuit Breakers

Rate limiting is itself a system that can fail. What happens when the rate limiter goes down?

- **Fail open** — allow all requests through (risk: backends get overwhelmed)
- **Fail closed** — deny all requests until the limiter recovers (risk: total outage)

The correct choice depends on the business context. Cloudflare fails open at the edge — better to serve traffic than to drop it. Stripe fails open for per-key checks but fail-closed for account-level checks — better to temporarily risk an overloaded account than to block legitimate payments.

A circuit breaker pattern should also protect the rate limiter itself. If the Redis cluster backing your rate limit counters is under strain, the rate limiter should fall back to local counters rather than compounding the Redis load.

### Pattern 5: Observability and Feedback Loops

Rate limiting must be observable end-to-end. The most successful teams track:

- **Rate limit hit rate** — what percentage of requests are being rate-limited? A sudden spike may indicate a misconfigured rule, not a real attack.
- **Rate limit by rule/endpoint** — which rules are triggering most often? Low-value rules should be removed or tuned.
- **False positive rate** — how often are legitimate users rate-limited? This is the most critical metric for customer satisfaction.
- **Time to rate limit** — how long does a user stay blocked after the limit is exceeded? Too long causes frustration; too short defeats the purpose.
- **Cost of rate limiting** — how much CPU/memory does the rate limiter consume? At scale, a rate limiter that costs more than the requests it blocks is counterproductive.

Dashboards should surface these metrics alongside application health so teams can correlate rate limiting events with deployment changes, traffic spikes, and customer complaints.

## 8. Real-World Failure Mode: The Thundering Herd at Window Reset

One subtle failure mode recurs across every system discussed here: the thundering herd that forms at the exact second a rate limit window resets.

When a rate limit window expires at a known time (e.g., the top of the hour for a fixed-window limit), all rate-limited clients retry simultaneously. This creates a spike that often exceeds the limit several times over before the limiter catches up, leading to overlapping windows of overload.

**Mitigations** in production:

1. **Jitter**: Add a random offset (0 to 10% of the window) to each client's retry time. This spreads the herd across the window boundary.
2. **Sliding windows**: Avoid hard edges entirely. A sliding window (or sliding window log) does not have a single reset point.
3. **Token bucket with refill**: Tokens refill continuously (e.g., 1 token per 10ms) rather than dumping a full bucket at window boundaries.
4. **Server-side smoothing**: The rate limiter itself can queue and release requests gradually rather than hard-rejecting at the boundary — Stripe's approach.
