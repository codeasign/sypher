# Rate Limiting — Challenges

A curated set of FAANG-level challenges organized into three tiers of difficulty. Tier 1 validates conceptual fluency, Tier 2 exercises applied design reasoning, and Tier 3 tests architectural depth at global scale. Each challenge includes a prompt and a model answer.

---

## Tier 1 — Core Concepts

These challenges test whether you can explain fundamental rate-limiting algorithms and protocols clearly. Expect these in phone screens, L4 on-sites, and as warm-up questions in senior interviews.

---

### Challenge 1.1 — Explain the Token Bucket Algorithm

**Prompt:** "Explain the token bucket algorithm to a junior engineer. Cover the data structures, refill mechanics, burst behavior, and space-time complexity. Provide an implementation sketch."

**Model answer:**

The token bucket models rate limiting as a physical bucket that holds tokens. Two configuration parameters control its behavior: **capacity** (maximum tokens the bucket can hold) and **refill rate** (tokens added per second). Each request consumes one token. If the bucket has tokens, the request passes. If the bucket is empty, the request is rejected with HTTP 429.

**Data structures:**

Two variables per bucket and two configuration parameters:

| Variable | Type | Purpose |
|---|---|---|
| `tokens` | float | Current token count |
| `last_refill_time` | integer | Unix timestamp of last refill |
| `capacity` | integer (config) | Maximum burst size |
| `refill_rate` | float (config) | Tokens added per second |

**Lazy refill mechanic:**

Instead of running a background timer, compute tokens on every request:

```
elapsed = now - last_refill_time
tokens = min(capacity, tokens + elapsed * refill_rate)
last_refill_time = now
if tokens >= 1.0:
    tokens -= 1.0
    allow
else:
    deny
```

The `min` function clamps tokens to capacity so unused tokens never accumulate beyond the burst limit. The `last_refill_time` is updated on every request regardless of whether the request is allowed — skipping this causes the refill calculation to drift over time.

**Implementation sketch:**

```python
class TokenBucket:
    def __init__(self, capacity: int, refill_rate: float):
        self.capacity = capacity
        self.refill_rate = refill_rate
        self.tokens = float(capacity)
        self.last_refill = time.monotonic()

    def allow(self) -> bool:
        now = time.monotonic()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity,
                          self.tokens + elapsed * self.refill_rate)
        self.last_refill = now
        if self.tokens >= 1.0:
            self.tokens -= 1.0
            return True
        return False
```

**Burst behavior:**

A client that has been idle for 10 seconds with a refill rate of 10 tokens/second and capacity of 100 accumulates 100 tokens. The client can then send 100 requests instantly — a 10x burst over the steady-state rate. After the burst, the bucket is empty and refills at 10 tokens/second.

**Space-time complexity:**
- Time: O(1) per request.
- Space: O(1) per user (two floats + two config ints). For 1 million users: approximately 32 MB.

**Trade-offs:**

| Strength | Limitation |
|---|---|
| Natural burst absorption | Tokens accumulated during idle periods can enable sudden spikes |
| O(1) per request — no scanning | Does not guarantee uniform request spacing |
| Intuitive, widely documented | Not suitable when exact per-window enforcement is required |
| Redis has native `CL.THROTTLE` | Configuring capacity and rate requires understanding traffic patterns |

**Follow-up: "When would you NOT use token bucket?"**

When you need exact rate enforcement regardless of prior idle time. If an attacker waits 10 minutes and accumulates 6,000 tokens, token bucket allows 6,000 requests at once. Use sliding window algorithms when hard per-second limits are required.

### Challenge 1.2 — Explain the Fixed Window Problem

**Prompt:** "What is the fixed window boundary problem? Why is it dangerous in production, and how do sliding window algorithms solve it?"

**Model answer:**

**The problem:**

Fixed window divides time into discrete, calendar-aligned intervals (00:00-00:59, 01:00-01:59, etc.). Each window has a counter that resets at the boundary. A client can send their full limit at the very end of one window and again at the very start of the next, achieving twice the allowed rate in a few seconds.

**Why this is dangerous:**

Attackers specifically target window boundaries. Web scrapers, credential-stuffing bots, and DDoS tools synchronize on clock boundaries. In production, this pattern causes 3-5x traffic spikes at every window turn. Auto-scaling systems react to these spikes with unnecessary scale-up events followed by scale-down, causing cost and stability churn. A rate limiter that allows 2x traffic defeats its own purpose.

**How sliding window algorithms solve it:**

**Sliding Window Log:** Stores every request timestamp. The window is defined as "all requests in the last 60 seconds," computed on every request by scanning the log and removing entries older than 60 seconds. No boundaries to exploit. Memory cost: O(N) per user where N is the limit per window.

**Sliding Window Counter:** Uses two counters — one for the current fixed window and one for the previous window — and computes a weighted estimate:

```
weight = (window_duration - elapsed_into_current) / window_duration
estimated = current_counter + previous_counter * weight
```

If a request arrives 10 seconds into a 60-second window, weight = 50/60 = 0.83. The previous window's contribution decays linearly as the current window progresses. Memory cost: O(1) per user, approximately 25x less than sliding window log while providing ~98% precision.

**Recommendation:** Sliding window counter is the sweet spot for most systems — near-exact enforcement with O(1) memory.

### Challenge 1.3 — Explain HTTP 429 Responses

**Prompt:** "What headers must a 429 response include? What body format should it use? How should clients handle it? Cover RFC specifications and production best practices."

**Model answer:**

**The status code:**

HTTP 429 Too Many Requests (RFC 6585) signals that the client has exceeded the server's rate limit. Unlike 401 (unauthorized) or 403 (forbidden), 429 is a temporary condition — the client may retry after a delay.

**Required headers:**

Only `Retry-After` is mandatory per the specification:

```text
Retry-After: 12                       # seconds (integer)
Retry-After: Wed, 02 Jul 2026 12:00:00 GMT  # HTTP-date
```

The integer format is preferred for programmatic clients.

**Industry-standard headers (best practice):**

```text
RateLimit-Limit: 100                 # maximum requests per window
RateLimit-Remaining: 0               # requests remaining in current window
RateLimit-Reset: 1740788040          # Unix timestamp when the window resets
Retry-After: 12                      # suggested wait in seconds
```

These headers (inspired by GitHub's API) enable clients to self-regulate without polling.

**Response body (RFC 9457 Problem Details):**

```json
HTTP/1.1 429 Too Many Requests
Content-Type: application/problem+json
Retry-After: 12
RateLimit-Limit: 100
RateLimit-Remaining: 0
RateLimit-Reset: 1740788040

{
  "type": "https://api.example.com/errors/rate-limit-exceeded",
  "title": "Too Many Requests",
  "status": 429,
  "detail": "Exceeded limit of 100 requests per minute. Retry in 12 seconds.",
  "instance": "/api/v1/search",
  "retry_after_seconds": 12
}
```

**Client handling with exponential backoff and jitter:**

```python
import random, time

def call_with_backoff(request_fn, max_retries=5):
    delay = 1.0
    for attempt in range(max_retries):
        resp = request_fn()
        if resp.status_code != 429:
            return resp
        retry = resp.headers.get("Retry-After")
        sleep_time = int(retry) if retry and retry.isdigit() else delay
        sleep_time += random.uniform(0, sleep_time * 0.5)
        time.sleep(sleep_time)
        delay = min(delay * 2, 60)
    raise Exception("Rate limit exceeded after max retries")
```

**Interviewer signal check:** Interviewers listen for knowledge of RFC 6585 and RFC 9457, the mandatory `Retry-After` header, `RateLimit-*` headers for self-regulation, and the distinction between a hard block and a soft throttle.

### Challenge 2.1 — Design Rate Limiting for a Public API

**Prompt:** "Design rate limiting for a public REST API serving 50,000 req/s across 1,000 registered developers. Free tier: 10 req/s. Paid tier: 100 req/s. Rate limiter must add less than 5 ms overhead and survive a Redis failover."

**Model answer:**

**Algorithm:** Sliding window counter with Redis Lua scripts. Provides 98% precision with O(1) memory per key. Token bucket is close, but sliding window counter gives more predictable per-second enforcement for a public API.

**Architecture:**

**Redis cluster sizing:**

```
Total RPS: 50,000
Rate check ops per request: 1 (EVALSHA)
Total Redis ops: 50,000 writes/sec
Redis node capacity: ~100,000 ops/sec
Shards needed: 1 (but 3 for HA with replication)
Replication factor: 2
Total nodes: 6 (3 masters, 3 replicas)
```

**Local cache layer:**

With only 1,000 API keys, the entire dataset fits in a local LRU cache on each gateway node. Each key's rate limit decision is cached with a 500 ms TTL. This reduces Redis load by over 90% — only the first request per key every 500 ms hits Redis. The stale window trade-off is acceptable: a key may get 1-2 extra requests every 500 ms, well within the sliding window counter's precision budget.

**Redis key structure:**

```
rate_limit:free:{api_key}:{window_timestamp}   -> counter value
rate_limit:paid:{api_key}:{window_timestamp}   -> counter value
```

**Lua script:**

```lua
-- KEYS[1] = current window key   KEYS[2] = previous window key
-- ARGV[1] = previous window weight   ARGV[2] = tier limit   ARGV[3] = TTL

local current = redis.call("GET", KEYS[1])
local previous = redis.call("GET", KEYS[2])
local count = (tonumber(current) or 0) * 1.0 +
              (tonumber(previous) or 0) * tonumber(ARGV[1])

if count >= tonumber(ARGV[2]) then
    return {0, math.floor(count)}
end

redis.call("INCR", KEYS[1])
redis.call("EXPIRE", KEYS[1], tonumber(ARGV[3]))
return {1, math.floor(count) + 1}
```

**Failover handling:**

When Redis is unreachable, the rate limiter switches to local in-memory counters per key with a 20% reduced limit (8 req/s for free, 80 req/s for paid). This safety margin ensures that even if local counters drift from the global Redis state, the effective limit stays below the advertised limit. On Redis recovery, local counters are discarded and Redis counters resume. A `rate_limiter_degraded_mode` metric tracks time spent in fallback and alerts if it exceeds 30 seconds.

### Challenge 2.2 — Design Rate Limiting for a Chat App

**Prompt:** "Design rate limiting for a real-time chat app with 10 million DAU. Users send messages in channels with 2-500 members. The system must prevent spam without blocking natural conversation. Consider DMs, group chats, and large channels."

**Model answer:**

**Unique challenges for chat:**

Chat rate limiting is fundamentally different from API rate limiting. A user typing a long message sends nothing for 30 seconds then sends one message — legitimate. A user copy-pasting three paragraphs sends three messages in one second — also legitimate. The rate limiter must distinguish these from a bot sending 30 messages in 3 seconds. Additionally, dropping messages silently causes confusion; users must see a clear error indicator.

**Multi-layer design:**

**Algorithm selection per layer:**

| Layer | Algorithm | Rationale |
|---|---|---|
| L1: Per-user global | Token bucket | Natural burst absorption for conversation |
| L2: Per-channel | Token bucket | Different capacities per channel size |
| L3: Content dedup | Fixed window with content hash | Simple, memory-efficient for duplicate detection |

**WebSocket-specific considerations:**

- Use **backpressure** instead of hard rejection. When a user hits limits, the server sends a `rate_limit: slow_down` frame. Most WebSocket client libraries interpret this and reduce send rate automatically.
- Typing indicators bypass rate limiting entirely. They are ephemeral and low-cost, but blocking them creates terrible UX.
- On WebSocket reconnect, the new connection inherits the user's L1 bucket state from Redis. Do not reset counters on reconnect — this would allow bypassing limits by cycling connections.

**Scaling to 10M DAU:**

Shard Redis by user ID hash. Co-locate the rate limiter with the WebSocket server to minimize network hops. Each server node maintains a local write-behind cache of the most active users' token buckets and syncs to Redis asynchronously.

### Challenge 2.3 — Design Tiered Rate Limits

**Prompt:** "Design a tiered rate limiting system where accounts and API keys have different limits, some endpoints share common limits, and overage behavior varies by customer. Configuration must be hot-reloadable."

**Model answer:**

**Hierarchical limit tree:**

Model the configuration as a tree of scopes. Each scope node defines a limit. A request must pass through every node from root to leaf. The effective rate is the minimum of all applicable limits along the path.

**Configuration model (stored in a config service, not hard-coded):**

```yaml
rate_limits:
  - scope: "account:<account_id>"
    limits:
      - window: "hour"  max: 10000
      - window: "minute" max: 1000
    overage: "hard"

  - scope: "key:<api_key>"
    limits:
      - window: "minute" max: 100
    overage: "soft"

  - scope: "endpoint_group:reports"
    match: "/reports/**"
    limits:
      - window: "minute" max: 10
    overage: "hard"
```

**Evaluation algorithm:**

When a request arrives, the rate limiter:

1. Collects all scopes that apply (account, key, endpoint groups).
2. For each scope, checks the current count against the scope's limit per window.
3. If any scope with `overage: hard` would be exceeded, the request is denied.
4. If only `overage: soft` scopes would be exceeded, the request is allowed with a warning header.
5. If `overage: burst`, the request consumes from a monthly burst pool.

```python
def check_rate_limit(request, scopes):
    for scope in scopes:
        for window, limit in scope.limits:
            current = get_counter(scope.id, window)
            if current + request.cost > limit:
                if scope.overage == "hard":
                    return False
                elif scope.overage == "burst":
                    if not consume_burst(scope.id, request.cost):
                        return False
    return True
```

**Hot-reloadable configuration:**

- Store rate limit rules in etcd or Consul.
- Each rate limiter node subscribes to changes via a long-poll watch.
- On config change, atomically swap the in-memory scope tree reference. In-flight requests use the old tree; new requests use the new one.
- Use a generation counter to detect stale configs and trigger re-evaluation.

**Testing tiered configurations:**

| Scenario | Expected |
|---|---|
| Free account, 1 key, 15 req/min sent | Denied at account level |
| Pro account, 3 keys, 50 req/min each | Allowed (each key < 100, total 150/min < 10K/hr) |
| Enterprise, `/reports` endpoint, 15 req/min | Denied at endpoint group (10/min hard limit) |

### Challenge 3.1 — Design Distributed Rate Limiting for a Global Platform

**Prompt:** "Design rate limiting for a global SaaS platform with 12 geographic regions. Cross-region latency is unacceptable — requests from Australia must not wait for a Redis write in US-East. Global limits must be enforced even though each region operates independently."

**Model answer:**

**Architecture: Regional primary with global coordination.**

Each region operates an independent rate limiter backed by a local Redis cluster for sub-millisecond latency. A background synchronization layer periodically aggregates counters across regions and propagates adjustments. The system is eventually consistent for global limits but exact within each region.

**Core algorithm: Regional quota with global sync.**

Each region receives a `base_quota` proportional to its historical traffic share. US-East gets 40%, EU-West 30%, AP-Southeast 15%, etc. Quotas are stored in a configuration service and updated periodically.

**Request flow:**

1. Client request hits the nearest regional endpoint.
2. Local rate limiter checks the region's quota using a sliding window counter in the local Redis cluster. The check completes in under 2 ms.
3. If the region's quota is not exhausted, the request is allowed and the local counter is incremented.
4. If the region's quota is approaching exhaustion (80%+), the rate limiter pre-emptively requests a top-up from the global coordinator.
5. The coordinator reallocates unused quota from quieter regions or denies the top-up if the global limit is reached.

**Global reconciliation:**

Every 30 seconds, each region emits its counter values to a Kafka topic. A coordinator process consumes these logs, sums the counters per user, and compares against the global limit. If the sum exceeds the limit, the coordinator emits **correction events** that instruct regions to reduce their quotas for the next window:

```python
class GlobalCoordinator:
    def reconcile(self, regional_counters):
        for user_id in all_users:
            global_total = sum(regional_counters[user_id].values())
            if global_total > self.global_limits[user_id]:
                overage = global_total - self.global_limits[user_id]
                self.emit_correction(user_id, self.compute_correction(user_id, overage))

    def compute_correction(self, user_id, overage):
        corrections = {}
        for region, count in self.regional_breakdown[user_id].items():
            ratio = count / max(self.global_totals[user_id], 1)
            reduction = int(overage * ratio)
            corrections[region] = max(0, self.base_quotas[region] - reduction)
        return corrections
```

**Handling partitions:**

During a 5-minute network partition:
- Each region continues serving from its local quota allocation. A region might allow up to `base_quota * 1.2` requests before the global limit is reached.
- The 20% buffer ensures that even if all traffic shifts to one region during a partition, the global limit is not exceeded by more than 20%.
- When the partition heals, the coordinator detects the overage from backlogged events and reduces quotas aggressively to compensate.

**Consistency trade-offs:**

| Approach | Latency | Global Precision | Partition Tolerance |
|---|---|---|---|
| Synchronous global check | +50-300 ms | Exact | Low |
| Regional quota with sync (chosen) | +0.5-2 ms | ~95% | High |
| Local-only (no global sync) | +0 ms | None | N/A |

**Observability signals:**

- `quota_utilization` per region — alert if over 90%.
- `correction_magnitude` — large corrections indicate mis-calibrated quotas.
- `topup_failure_rate` — high rate means global limit is being hit.
- `cross_region_drift` — sum of regional counters vs. global counter. Investigate over 5%.

### Challenge 3.2 — Design Rate Limiting for a Real-Time Bidding System

**Prompt:** "Design rate limiting for a real-time bidding (RTB) system processing 500,000 bid requests per second globally. DSPs must respond within 50 ms. Rate limiting must cap spend per advertiser, limit QPS per campaign, and prevent a misconfigured DSP from flooding the exchange. Every microsecond counts."

**Model answer:**

**The constraints are extreme:**

- 500,000 req/s with a 50 ms deadline means rate limiting overhead must be under 1 ms.
- Spend limits must be globally consistent (advertisers cannot exceed daily budgets).
- Traffic arrives in bursts of 100,000+ requests per second.
- False negatives (rejecting a winning bid) cost real money.

**Algorithm selection:**

| Constraint | Algorithm | Why |
|---|---|---|
| Campaign QPS | Leaky bucket (in-memory) | Smooths bursts, no network call, no cross-node coordination |
| Advertiser spend | Token bucket (async Redis) | Cumulative global limit, expensive to check synchronously |
| Pre-filter | Bloom filter | Sub-microsecond rejection of uninteresting requests |

**Stage 1: Pre-filter (Bloom filter)**

Before any rate limit check, determine if the DSP is interested. Use a Bloom filter keyed on `(campaign_id, geo, domain, device_type)`. Requests that match no active campaign are dropped immediately.

- Memory: 10 MB Bloom filter handles 1 million campaign combinations with 0.1% false-positive rate.
- Latency: ~200 nanoseconds per check.
- Effect: Eliminates 60-80% of incoming requests before they reach the rate limiter.

**Stage 2: Campaign QPS (leaky bucket, in-memory)**

Each application server maintains its own leaky bucket per campaign. No cross-node coordination needed — rate enforcement is approximate within the burst tolerance.

```python
class LeakyBucket:
    def __init__(self, rate_per_sec, burst_size):
        self.rate = rate_per_sec
        self.burst = burst_size
        self.queue = 0.0
        self.last_drain = time.monotonic_ns()

    def allow(self):
        now = time.monotonic_ns()
        elapsed = (now - self.last_drain) / 1_000_000_000
        self.queue = max(0.0, self.queue - elapsed * self.rate)
        self.last_drain = now
        if self.queue < self.burst:
            self.queue += 1.0
            return True
        return False
```

Per-server burst is set to `global_burst / num_servers`. Total throughput will not exceed `global_rate + global_burst`.

**Stage 3: Advertiser spend (token bucket with async Redis)**

Spend limits must be globally consistent. Synchronous Redis on every request adds 1-3 ms — too slow for the 50 ms deadline. The solution is a local estimate with asynchronous reconciliation:

```python
class AsyncSpendLimiter:
    def __init__(self, redis, local_size=10000):
        self.redis = redis
        self.local = {}
        self.sync_interval = 0.1
        self.last_sync = {}

    def try_spend(self, advertiser_id, bid_price_cents):
        estimate = self.local.get(advertiser_id, 0)
        if estimate + bid_price_cents > DAILY_BUDGET:
            return False
        self.local[advertiser_id] = estimate + bid_price_cents
        if time.monotonic() - self.last_sync.get(advertiser_id, 0) > self.sync_interval:
            self.redis.hincrby(f"spend:{advertiser_id}", "spent", self.local[advertiser_id])
            self.local[advertiser_id] = 0
            self.last_sync[advertiser_id] = time.monotonic()
        return True
```

**Precision vs. latency trade-off:**

| Strategy | Per-Request Latency | Global Precision |
|---|---|---|
| Synchronous Redis | +1-3 ms | Exact |
| Local estimate + async sync (chosen) | +0-10 µs | ~99% within 100 ms |
| Batch periodic flush | +0-5 µs | ~95% within 1 s |

The DSP can over-spend by 1-2% because the cost of a false negative (rejecting a winning bid) exceeds the cost of a small overage. At the end of the budget window, a final reconciliation query confirms the exact total.

**Handling the 100,000 req/s burst:**

1. Pre-filter checks 100K requests in ~20 ms total (200 ns each).
2. Leaky bucket admits only 5,000 req/s (configured campaign QPS). The rest are rejected or buffered.
3. Of the 5,000 that pass, async spend limiter performs a local memory check (~500 ns each) and a batched Redis sync every 100 ms.
4. Total per-request overhead: under 2 µs.

**Failure modes:**

| Failure | Impact | Mitigation |
|---|---|---|
| Redis unavailable | Spend sync fails | Fall back to local-only mode (overspend risk). Alert immediately. |
| Server crash | Lost local estimates (up to 100 ms) | Acceptable — at most $0.01 in ad spend lost per crash window. |
| Clock skew | Leaky bucket drain rate drifts | Use `time.monotonic_ns()` instead of wall clock. |
| Traffic spike exceeds capacity | Bid requests queued | Pre-scale based on historical patterns. Adaptive QPS that reduces when CPU > 80%. |
