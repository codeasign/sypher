# Rate Limiting — Interview Playbook

A dedicated guide to acing rate-limiting questions in system design interviews. This playbook covers question taxonomy, level-appropriate expectations, a structured response framework, a curated phrase bank, common red flags, and high-signal practice drills.

---

## Question Taxonomy

Rate-limiting questions fall into four archetypes. Each tests a different facet of your design thinking.

### Type 1 — Greenfield Design (most common)

You are asked to design a rate limiter from scratch. The interviewer provides a high-level requirement, and you must scope, model, and defend your choices.

**Typical prompts:**
- "Design a rate limiter for a public API."
- "Design a distributed rate-limiting system."
- "How would you implement API rate limiting at scale?"

**What is tested:** Requirements gathering, algorithmic trade-offs, data store selection, scale reasoning, and failure-mode thinking.

### Type 2 — Integration Design

You are given an existing system and asked to add rate limiting to it. These questions test your ability to retrofit without large-scale refactoring.

**Typical prompts:**
- "Add rate limiting to our existing chat service."
- "How would you introduce rate limiting into a legacy monolith?"
- "We need per-user rate limits on our payment gateway — where do you put the logic?"

**What is tested:** Incremental architecture, reverse proxies and middleware patterns, configuration-driven design, and operational safety.

### Type 3 — Algorithm Deep Dive

You are asked to compare, implement, or optimize a specific rate-limiting algorithm in detail.

**Typical prompts:**
- "Compare the token bucket and sliding window log algorithms."
- "Implement a leaky bucket with a priority queue."
- "How would you implement sliding window counters without storing every request?"

**What is tested:** Algorithmic fluency, space-time complexity analysis, precision versus cost trade-offs, and implementation skill (often with live coding).

### Type 4 — Failure Mode & Recovery

You are presented with a broken or overloaded system and asked to diagnose and fix rate-limiting failures.

**Typical prompts:**
- "Our rate limiter is allowing too many requests under load — debug it."
- "What happens when your Redis cluster goes down during a traffic spike?"
- "Users are complaining about unfair throttling — how do you investigate?"

**What is tested:** Observability awareness, graceful degradation, fail-open versus fail-closed reasoning, and operational debugging discipline.

## Level Expectations

Your depth and framing should match the seniority level the interviewer is probing.

### L4 / New Grad — Implementation Fluency

| Expectation | Detail |
|---|---|
| Algorithm choice | Explain token bucket with clear reasoning. |
| Mental model | Single-node, in-memory rate limiter. |
| Data structures | HashMap of counters or token buckets per user. |
| Concurrency | Basic awareness of race conditions (synchronized, locks). |
| Complexity | O(1) per request, O(N) memory for N users. |

**Typical question:** "Write a rate limiter that allows 10 requests per second per user."

### L5 / Senior — Distributed Design

| Expectation | Detail |
|---|---|
| Algorithm choice | Compare 2-3 algorithms and pick with justification. |
| Mental model | Distributed rate limiter with a shared Redis backend. |
| Data structures | Sorted sets (for sliding window log), or Lua scripts for atomicity. |
| Consistency | Discuss eventual consistency, clock skew, and Redis cluster semantics. |
| Failure modes | Graceful degradation when the backing store is unavailable. |

**Typical question:** "Design a distributed rate limiter that serves 1M requests per second across three data centers."

### L6 / Staff — System-Wide Ownership

| Expectation | Detail |
|---|---|
| Algorithm choice | Hybrid approach — different algorithms per route, user tier, or risk level. |
| Mental model | Multi-layer rate limiting: edge proxy, application middleware, and data-layer guard. |
| Data structures | Custom sliding-window counters with probabilistic data structures (Bloom filters, Count-Min Sketch) for hot shards. |
| Consistency | Configurable consistency per route — strong for billing, eventual for search. |
| Failure modes | Circuit breakers, fallback to local counters, audit trails, and automated config rollback. |
| Cross-cutting | Cost attribution, quota management UI, proactive alerting, and integration with billing systems. |

**Typical question:** "Design a global rate-limiting platform that product teams across the company can configure and monitor."

### L7 / Principal — Strategy & Platform

| Expectation | Detail |
|---|---|
| Algorithm choice | Abstraction layer that teams can plug custom algorithms into. |
| Mental model | Rate limiting as a platform service with SLAs, SDKs, and a governance model. |
| Data structures | Pluggable backend: Redis, local memory, or custom sharded stores depending on criticality. |
| Consistency | Business-driven: banking routes use strong consistency, analytics routes use best-effort. |
| Failure modes | Chaos-engineering validated, self-healing config, canary deployments for rule changes. |
| Cross-cutting | Developer portal, self-service quota management, usage analytics, and chargeback. |

**Typical question:** "Design a company-wide rate-limiting platform that serves 10,000+ microservices with diverse latency and consistency requirements."

## Structured Response Framework — The RATE Acronym

Use this five-step framework to organize any rate-limiting answer. It ensures you cover breadth before depth and never miss a critical section.

### R — Requirements (2 minutes)

State both functional and non-functional requirements before proposing solutions.

**Functional requirements:**
- What is being rate limited? (endpoint, user, IP, API key, tenant)
- What is the limit? (X requests per Y seconds, or concurrency limits)
- Scope: single node, multi-region, or global?
- Differentiation: do all users have the same limit?

**Non-functional requirements:**
- **Latency:** The rate limiter should add less than 1-5 ms of overhead.
- **Accuracy:** How precise must the limit be? Exact (wall-clock hard stop) or approximate (average over window)?
- **Availability:** Fail-closed (deny when store is down) or fail-open (allow when store is down)?
- **Memory efficiency:** Can we hold a bucket per user, or do we need probabilistic structures?
- **Scalability:** How many rules, routes, and tenants must it support?

**Anti-pattern:** Jumping straight to "I will use Redis" without enumerating requirements.

**Sample clarifying questions to ask:**
- "Are we rate limiting at the API gateway layer or within a specific service?"

### A — Algorithm & Data Store (5 minutes)

Choose the algorithm that best fits your requirements. Do not commit immediately — survey the landscape and narrow down.

**Algorithm Comparison Table:**

| Algorithm | Precision | Memory | Burst Handling | Distributed Complexity | Best For |
|---|---|---|---|---|---|
| Token Bucket | Loose | Low (2 vars per user) | Yes (natural) | Medium (sync refill rates) | APIs with bursty traffic |
| Leaky Bucket | Strict (process rate) | Low (queue pointer + timestamp) | Smooths bursts | Medium (shared queue) | Queue-based processing |
| Fixed Window | Coarse (edge-of-window spikes) | Very low (1 counter + reset) | Poor (double at boundary) | Low | Low-precision, high-throughput |
| Sliding Window Log | Exact | High (stores every timestamp) | No | High (store per request) | When exact limits are required |
| Sliding Window Counter | Good (1-2% error) | Low (2 counters, 2 timestamps) | Low | Medium | Best balance for most systems |

**Decision flow:**
1. Need exact limits? → Sliding Window Log (but estimate memory cost).
2. Need memory efficiency? → Token Bucket or Sliding Window Counter.
3. Need burst absorption? → Token Bucket (preferred) or Leaky Bucket.
4. Operating at extreme scale (>1M RPS per shard)? → Count-Min Sketch or Token Bucket with local counters and periodic sync.
5. Fixed Window is rarely the right answer — name it only to explain why you are not using it.

**Data store options:**

| Store | Latency | Persistence | Consistency | Atomicity | Best For |
|---|---|---|---|---|---|
| Local memory (HashMap) | under 0.1 ms | None | N/A | Need locks | Single-node, low scale |
| Redis (single) | 1-5 ms | Optional (RDB/AOF) | Strong (single thread) | Lua scripts | Most distributed designs |
| Redis Cluster | 2-10 ms | Optional | Eventual (cross-shard) | Lua per shard | Multi-region, high scale |
| Memcached | 1-4 ms | None | Eventual | CAS (limited) | Caching counters only |
| Cassandra | 5-20 ms | Yes | Tunable (weak) | Lightweight transactions | Global rate limiting at scale |

**Default recommendation for most interviews:** Sliding Window Counter with Lua scripts in Redis. It provides good precision, low memory, and atomic operations.

### T — Traffic & Capacity Planning (3 minutes)

Estimate throughput, memory, and Redis cluster sizing.

**Throughput estimation:**
```
Requests per second (RPS): 100,000
  └── Rate limiter checks: 1 per request
  └── Redis ops per check: 2 (GET + SET or EVALSHA)
  └── Redis ops per second: 200,000
  └── Redis single-node throughput: ~100K - 200K ops/sec
  └── Redis cluster shards needed: 2-4 shards (with replication)
```

**Memory estimation (sliding window log example):**
```
Per-user limit: 100 requests per minute
Per-request storage: ~100 bytes (user_id + timestamp)
Active users per minute: 1,000,000
Memory: 1,000,000 × 100 × 100 bytes = 10 GB
  └── Can be reduced with expiration (TTL = 60s)
  └── Realistic peak memory: ~1-2 GB with TTL pruning
```

**Memory estimation (token bucket example):**
```
Per-user state: 2 integers (tokens, last_refill) = ~32 bytes
Active users per minute: 1,000,000
Memory: 1,000,000 × 32 bytes ≈ 32 MB
  └── Plus HashMap overhead: ~100 MB total
```

**Redis cluster sizing:**
```
Desired throughput: 500,000 checks/sec
Redis node capacity: 100,000 ops/sec (p99 < 5ms)
Minimum shards: 5
With replication factor 2: 10 nodes total
With one hot shard: 10-15 nodes (for skew headroom)
```

**Key formula to have ready:**
```
Total Redis ops = RPS × (1 rate check + write rate per check)
  └── Read-only for sliding window counter (compare + increment in Lua)
  └── 2 ops per check for naive GET + SET approach
```

**Request flow:**
1. Client sends request to the load balancer.
2. Load balancer forwards to an API gateway node.
3. Rate limiter middleware extracts the entity identifier (user ID, API key, or IP address) from the request.
4. Middleware constructs the Redis key: `rate_limit:{entity}:{route}:{window}`.
5. Middleware sends `EVALSHA` to Redis with a Lua script that atomically checks and increments the counter.
6. Lua script returns `1` (allow) or `0` (deny). If deny, middleware responds with `429 Too Many Requests`.
7. On allow, the request proceeds to the backend service.
8. Response headers are added: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`.

**Lua script example (sliding window counter):**
```lua
-- KEYS[1] = current window key       e.g. rate_limit:user42:api:1740788040
-- KEYS[2] = previous window key      e.g. rate_limit:user42:api:1740788010
-- ARGV[1] = current window weight    (0.0 - 1.0, based on time elapsed)
-- ARGV[2] = limit                    e.g. 100
-- ARGV[3] = ttl                      e.g. 12 (seconds)

local current = redis.call("GET", KEYS[1])
local previous = redis.call("GET", KEYS[2])

local count = (tonumber(current) or 0) * 1.0 +
              (tonumber(previous) or 0) * tonumber(ARGV[1])

if count >= tonumber(ARGV[2]) then
    return 0  -- deny
end

redis.call("INCR", KEYS[1])
redis.call("EXPIRE", KEYS[1], tonumber(ARGV[3]))
return 1  -- allow
```

**Local cache layer:**
- Cache rate limit decisions for hot keys (tier-1 users, popular endpoints).
- Short TTL: 100-500 ms. Long enough to reduce Redis load, short enough to avoid stale approvals.
- Use LRU eviction with ~10,000 entries.
- Trade-off: some requests may pass when they should have been denied (false positives). Acceptable for fail-open tolerance.

**Response headers and client backoff:**
```text
HTTP/1.1 200 OK
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1740788040

HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1740788040
Retry-After: 12
Content-Type: application/problem+json

{
  "type": "https://api.example.com/errors/rate-limit",
  "title": "Too Many Requests",
  "detail": "You have exceeded the rate limit of 100 requests per minute.",
  "instance": "/api/v1/search",
  "retry_after_seconds": 12
}
```

**Database integration — atomic counters with PostgreSQL:**

If Redis is unavailable and the system already uses PostgreSQL, rate limit counters can be stored in a dedicated table with atomic increments:

```sql
-- Rate limit counters table
CREATE TABLE rate_limit_counters (
    entity_id    TEXT NOT NULL,      -- user ID, API key, or IP
    route        TEXT NOT NULL,      -- API route or wildcard
    window_start TIMESTAMPTZ NOT NULL,
    count        INTEGER NOT NULL DEFAULT 0,
    limit_value  INTEGER NOT NULL,
    PRIMARY KEY (entity_id, route, window_start)
);

-- Atomic increment with upsert
INSERT INTO rate_limit_counters (entity_id, route, window_start, count, limit_value)
VALUES ('user42', '/api/v1/search', '2026-07-02T10:00:00Z', 1, 100)
ON CONFLICT (entity_id, route, window_start)
DO UPDATE SET count = rate_limit_counters.count + 1
WHERE rate_limit_counters.count < rate_limit_counters.limit_value
RETURNING count <= limit_value AS allowed;
```

PostgreSQL-based rate limiting adds 5-20 ms per check and increases database connection pool pressure. It is a viable fallback but not the primary store for high-throughput systems.

### E — Evaluation & Trade-offs (5 minutes)

Close with a candid evaluation. No design is perfect — showing you understand the weaknesses is a strength.

**Strengths of the sliding window counter approach:**
- Good precision (~1-2% error vs. exact sliding window log).
- Low memory footprint (two counters and two timestamps per bucket).
- Atomic operations via Lua scripts eliminate race conditions.
- Redis cluster scales horizontally for high throughput.
- Local cache reduces Redis load for hot keys.

**Limitations and mitigations:**

| Limitation | Impact | Mitigation |
|---|---|---|
| Clock skew across data centers | Incorrect window alignment for globally-distributed requesters | Use NTP with < 10 ms skew; acceptable error budget; use logical time (Redis TIME) |
| Redis outage | Rate limiter unavailable | Fail-open mode: allow requests, log denies-only for audit, alert on fallback activation |
| Hot key shard imbalance | One Redis shard handles 10x traffic | Use hash tags for key grouping; pre-shard by entity ID hash; consider CRDT-based local counters with periodic sync for extreme hot keys |
| Memory leak on counters | Counters grow unbounded for inactive users | Always set TTL on counters (window + 5 seconds grace); monitor Redis memory with `INFO memory` |
| Race condition with naive approach | Double-count during concurrent requests | Always use Lua scripts or Redis transactions — never GET + SET separately |
| Cross-region latency | 50-200 ms for global checks against a single Redis region | Deploy Redis per region with eventual consistency or CRDT counters; accept bounded staleness |

**Fail-open vs. fail-closed decision framework:**
```text
Is the rate limiter a security control (auth, payment)?
    ├── Yes → Fail-closed (deny on outage). Losing revenue is better than fraud.
    └── No  → Is the backend capable of absorbing the extra load?
        ├── Yes → Fail-open (allow on outage). Availability over enforcement.
        └── No  → Fail-closed with degraded throughput limit (static local counter).
```

## Phrase Bank — 15+ High-Signal Expressions

Use these phrases strategically. They signal depth, experience, and system thinking.

### Requirements & Scoping

1. **"Let me first clarify the scope. Are we rate limiting at the API gateway layer or deep within a specific service?"** — Shows you distinguish edge vs. application-layer rate limiting.

2. **"What is the blast radius if this rate limiter fails? That determines our fail-open versus fail-closed stance."** — Shows operational maturity and business impact awareness.

3. **"Before picking an algorithm, I need to understand the traffic pattern: is it steady-state or bursty? Bursty traffic favors token bucket."** — Shows you match algorithms to workload, not the reverse.

### Algorithm Reasoning

4. **"Fixed window is simple but suffers from the boundary problem — a user can send 2x the limit if they time requests across the window edge. I'd avoid it for any system where precise enforcement matters."** — Shows you know the common pitfalls, not just the textbook definitions.

5. **"For this use case, I'd recommend the sliding window counter. It gives us 98-99% precision with only two counters per bucket — roughly one-tenth the memory of a sliding window log."** — Shows you can quantify the trade-off, not just name the algorithm.

6. **"Token bucket naturally handles bursts: unused tokens accumulate up to the bucket size, so a client that has been idle can catch up quickly without violating the average rate."** — Shows you understand the behavioral properties, not just the mechanics.

7. **"At this scale, storing every request timestamp in a sorted set becomes expensive. For 1M active users with 100 req/min each, that's 100M sorted-set entries — roughly 10 GB of memory just for the log."** — Shows you can do back-of-envelope memory math under pressure.

### Distributed Systems Depth

8. **"I'd use Lua scripts in Redis for atomicity. A naive GET-then-SET approach has a race condition — two concurrent requests could both read N and both increment to N+1, allowing N+2 requests through. Lua prevents that."** — Shows you understand concurrency at the protocol level.

9. **"In a multi-region deployment, clock skew between data centers can cause over- or under-counting. I'd use Redis's TIME command for a consistent clock rather than relying on application-server clocks."** — Shows you know how to build distributed agreement around time.

10. **"For the hot-key problem — say a tier-1 customer on a single shard — I'd consider local counters with periodic batch sync to Redis. We lose some precision but avoid burning one Redis shard to 100% CPU."** — Shows you can design around scaling bottlenecks.

11. **"If the Redis cluster goes down, I'd fail open for non-critical routes and fail closed for payment or authentication routes. The key is having a per-route configuration flag for availability mode."** — Shows you think in terms of configuration-driven architecture, not hard-coded behavior.

### Operational & Production Mindset

12. **"The rate limiter should emit metrics on every check: allow count, deny count, Redis latency, and local cache hit rate. These feed into a dashboard and alert on anomaly — for example, if the deny rate drops suddenly, our Redis cluster might be down."** — Shows observability-first thinking.

13. **"I'd use a canary deployment for rate limit rule changes. Roll out the new rule to 1% of traffic, verify the deny rate matches expectations, then ramp to 100%. This prevents a misconfigured limit from taking down the entire API."** — Shows safe change management practices.

14. **"Rate limit configuration should live in a version-controlled service, not hard-coded. Teams should be able to update limits via a config API or GitOps workflow, with an audit trail and automatic rollback if the error rate spikes."** — Shows platform-level thinking.

### Testing & Validation

15. **"I'd test the rate limiter with three scenarios: normal traffic (verify limits are enforced), burst traffic (verify burst behavior matches the algorithm), and failure injection (kill the Redis cluster and verify the fail-open or fail-closed behavior)."** — Shows disciplined testing methodology.

16. **"The hardest bugs in rate limiters are off-by-one errors in the window boundary logic. I'd write property-based tests that generate random timestamps and verify that the sliding window counter never exceeds the configured limit."** — Shows deep testing sophistication.

### Trade-off & Decision Saturation

17. **"This design has four knobs — algorithm, window size, store backend, and fail mode — and they interact. For example, picking token bucket with Redis and fail-closed gives us strong enforcement but higher operational cost. Picking sliding window counter with local memory and fail-open gives us low cost but only eventual consistency. The right combination depends on the route's criticality."** — Shows you think of design as a multi-dimensional space, not a single answer.

18. **"I want to make sure we have a throttle, not a hard block. A 429 response should include a Retry-After header and a structured error body so clients can implement exponential backoff without polling. We should also publish a well-documented rate limit policy."** — Shows you care about developer experience and client-side integration.

19. **"At Amazon, we found that adding a small random jitter to the window alignment prevented the thundering-herd problem when millions of devices synchronized to the same clock tick. I'd apply the same principle here."** — Shows real-world experience and an understanding of emergent behavior at scale (substitute your own company or a well-known industry example).

## Red Flags — 8+ Signs of Weakness

These are signals that interviewers watch for. Avoid them at all cost.

### 1. Jumping Straight to Redis

**Weak:** "I'll use Redis with a sorted set."
**Strong:** "Let me clarify the requirements first — then I can choose the right algorithm and backend."

Redis is almost always the right store, but naming it before you understand the requirements signals that you are following a script rather than designing.

### 2. Ignoring the Boundary Problem

**Weak:** "Fixed window is fine."
**Strong:** "Fixed window has a well-known boundary problem where traffic can spike to 2x at window edges. For this system, I'd prefer..."

If you propose fixed window and do not address the boundary problem, the interviewer will probe it. Get ahead of it.

### 3. No Memory Estimate

**Weak:** "We'll store it in Redis."
**Strong:** "With 1M active users and 100 req/min each, the sliding window log approach needs roughly 10 GB of memory. The sliding window counter needs about 50 MB. Given that, I recommend..."

Interviewers expect back-of-envelope calculations. If you cannot estimate memory, you cannot design for scale.

### 4. Single-Point-of-Failure Blindness

**Weak:** "The rate limiter checks Redis."
**Strong:** "If Redis goes down, we fail open for non-critical routes and fall back to a local in-memory token bucket with degraded limits for critical routes. We alert immediately."

A single Redis instance is a SPOF. A Redis cluster with replicas reduces the risk but does not eliminate it. Always discuss the failure mode.

### 5. No Race Condition Awareness

**Weak:** (GET then SET without comment)
**Strong:** "I need to ensure atomicity — GET followed by SET is racy. I'll use a Lua script or Redis INCR with expiration to avoid double-counting."

Race conditions in rate limiters are subtle and common. If you gloss over atomicity, the interviewer will assume you have never debugged one.

### 6. No Differentiation of Limits

**Weak:** "All users have the same limit: 100 requests per minute."
**Strong:** "Free-tier users get 10 req/min, pro-tier get 100, and enterprise get 10,000. The rate limiter configuration should support tier-based limits, and we should publish the overage pricing for enterprise customers."

Flat limits signal that you have not thought about business requirements or multi-tenancy.

### 7. Ignoring Distributed Clock Skew

**Weak:** "We'll use the server timestamp for all rate limit windows."
**Strong:** "In a distributed system, clock skew between data centers can cause up to 50 ms of window misalignment. I'd centralize time using Redis's TIME command, or accept a small error budget as a trade-off for simpler architecture."

Cross-datacenter clock skew is the sort of detail that distinguishes L5+ candidates from L4.

### 8. No Retry-After Header

**Weak:** (returns 429 with empty body)
**Strong:** "A 429 response must include the Retry-After header and a structured JSON body with the reset timestamp. This allows clients to implement proper backoff without guessing."

Missing the Retry-After header is a common oversight that signals insufficient API design experience.

### 9. No Configuration Management

**Weak:** "The rate limit is hard-coded to 100."
**Strong:** "Rate limits should be externally configurable — per route, per tier, per entity. We can use a config service, environment variables, or a GitOps workflow. Changes should be versioned, reviewed, and deployable via canary."

Hard-coded limits signal that you are building a prototype, not a production system.

### 10. Overlooking the Hot-Key Problem

**Weak:** "Redis cluster will handle the load."
**Strong:** "If one customer generates 50% of the traffic, they all hash to the same Redis shard. That shard becomes a bottleneck. I'd either use hash tags to split the hot key across multiple slots, or use local counters with eventual sync for extreme cases."

Hot-key shard imbalance is a classic scaling failure. If you do not anticipate it, the interviewer will raise it — and it is much better to raise it yourself first.

## Practice Drills — 5 High-Signal Exercises

These drills simulate real interview conditions. Each targets a specific skill dimension.

### Drill 1 — Algorithm Speed Run (5 minutes)

**Task:** Without pausing, name all five rate-limiting algorithms, their core data structures, and the space and time complexity of each.

**Expected output:**
| Algorithm | Data Structures | Per-Request Time | Per-User Memory |
|---|---|---|---|
| Token Bucket | 2 counters | O(1) | O(1) |
| Leaky Bucket | Queue + counter | O(1) amortized | O(burst size) |
| Fixed Window | 1 counter + timestamp | O(1) | O(1) |
| Sliding Window Log | Sorted set | O(log N) | O(N per window) |
| Sliding Window Counter | 2 counters + 2 timestamps | O(1) | O(1) |

**Why this matters:** Speed fluency signals deep comfort. Hesitation on basic facts undermines confidence.

---

### Drill 2 — Requirements Gathering (10 minutes)

**Task:** Take the vague prompt "Design a rate limiter for our API" and list every clarifying question you would ask. Group them by category.

**Create your own list, then compare:**

- **Scope:** Where does the rate limiter live (gateway, service, client SDK)? Is it new or retrofitted?
- **Entities:** Limited by user, API key, IP, tenant? Hierarchical (user in org)?
- **Limits:** Specific values per tier? Per-endpoint or global? Are limits time-based or concurrency-based?
- **Traffic:** Current RPS? Expected growth? Read-heavy or write-heavy? Bursty or steady?
- **Latency:** Acceptable overhead in ms? Synchronous or async enforcement?
- **Persistence:** Do limits reset after the window, or do counters persist? Need audit trail?
- **Availability:** Fail-open or fail-closed? What is the cost of a false positive (allow when should deny) vs. false negative (deny when should allow)?
- **Observability:** What metrics, dashboards, and alerts are needed? Any compliance requirements?

**Why this matters:** Asking the right questions before designing separates senior engineers from junior ones.

---

### Drill 3 — Memory Estimation (10 minutes)

**Task:** Estimate the memory required for a sliding window log rate limiter with 500,000 active users, each limited to 200 requests per minute, storing each request as an 80-byte Redis sorted set member.

**Solution walkthrough:**
```text
Requests stored per user per minute: 200
Bytes per request: 80
Active users: 500,000

Total at peak: 500,000 × 200 × 80 bytes
             = 100,000,000 × 80 bytes
             = 8,000,000,000 bytes
             ≈ 7.45 GB

With Redis sorted set overhead (~50%): ~11 GB
With TTL (entries auto-expire after 60s): peak is bounded
But: RAM cost at $0.15/GB/hr on AWS ElastiCache = ~$1.65/hr, ~$1,188/mo

Alternative: Sliding window counter
Memory per user: ~50 bytes (2 counters, 2 timestamps, key overhead)
Total: 500,000 × 50 bytes = 25 MB (vs. 11 GB for sliding window log)
```

**Why this matters:** These calculations come up in every mid-to-senior system design interview. Practice speed math with rates, memory, and cost.

---

### Drill 4 — Failure Mode Walkthrough (15 minutes)

**Task:** Your company's main API is serving 429 errors to 30% of valid requests. Walk through your debugging process step by step.

**Structured walkthrough:**

**Step 1 — Validate the symptom:**
- Is the 30% consistent across all routes or concentrated on specific endpoints?
- Are specific users or tiers affected disproportionately?
- Did this start after a deployment or configuration change?

**Step 2 — Check the rate limiter's view of the world:**
- Query the Redis cluster: are counters growing unexpectedly?
- Check Redis CPU and memory: is one shard hot?
- Check local cache hit rate: is it too high (stale allow decisions) or too low (all traffic hitting Redis)?

**Step 3 — Common root causes:**
| Root Cause | Signature | Fix |
|---|---|---|
| Misconfigured limit | All users affected, uniform error rate | Check config service or environment variable; roll back or correct the limit value |
| Redis shard hot key | Errors concentrated on one customer or route | Add hash tag splitting or increase local cache TTL for that key |
| Clock skew | Errors correlate with node startup time | Verify NTP sync; move to Redis TIME for window alignment |
| Lua script bug | Errors reproducible with specific payloads | Review Lua script; add logging of input keys and arguments |
| Memory pressure (Redis) | Redis evicts keys, counters reset | Increase Redis memory; reduce TTL; switch to lower-memory algorithm |
| Network latency spike | Errors correlate with p99 Redis latency > 50ms | Add connection pooling; increase timeout; ensure Redis cluster is co-located |

**Step 4 — Mitigation:**
- If the root cause is not immediately clear, switch to fail-open mode for all non-critical routes.
- For critical routes, deploy a hotfix with a known-good config version.
- After mitigation, root-cause analysis: add monitoring gaps, update runbook, schedule chaos engineering test.

**Why this matters:** Debugging rate limiter failures under pressure is a common interview scenario. Having a structured investigation process signals production experience.

### Drill 5 — Design Defense (20 minutes)

**Task:** You propose a sliding window counter with Redis. The interviewer challenges you on each of the following. Prepare a calm, reasoned defense or concession for each.

**Challenge 1: "Why not use a simpler fixed window? It uses less memory."**
**Defense:** "Fixed window uses slightly less memory — one counter instead of two — but it has the boundary problem. A user who sends 100 requests at 10:59:59 and another 100 at 11:00:00 can send 200 requests in a 2-second span, effectively doubling the limit. For our use case, where we guarantee fair resource allocation, this is unacceptable. The sliding window counter uses one additional counter per bucket and completely eliminates the boundary spike. The memory cost is negligible — for 500K users, about 25 MB instead of 12 MB."

**Challenge 2: "Redis is too slow. We can do this faster in local memory."**
**Concession and counter:** "You are right that local memory is faster — sub-millisecond versus 1-5 ms for Redis. However, local memory cannot coordinate across nodes. If we have 10 API gateway nodes, each with a local counter set to 100 req/min, a user can send 1,000 requests across the cluster before any single node blocks them. If the business requirement allows 1,000 req/min per user, then local memory is fine and simpler. If the requirement is 100 req/min globally, we need a shared store. Redis is the standard choice, but we can reduce latency by adding a local cache layer with a 200 ms TTL for hot keys."

**Challenge 3: "All the logic is in Lua scripts — that's hard to test."**
**Concession and counter:** "That is a valid concern. Lua scripts can be difficult to unit test and debug. Mitigations include: (1) writing the script logic in a testable language first (Python or JavaScript) with a Redis mock, then translating to Lua; (2) using Redis's `SCRIPT LOAD` and `EVALSHA` for consistent execution; (3) adding structured logging inside the Lua script using `redis.log`; (4) having a dedicated integration test suite that runs Lua scripts against a real Redis instance in CI. I would also add a feature flag to bypass the Lua script and use a slower-but-simpler approach for debugging purposes."

**Challenge 4: "What happens when we need to enforce limits across data centers in different continents?"**
**Defense:** "Cross-continent rate limiting introduces latency challenges. A request hitting our US-East gateway needs to coordinate with a counter that may also be updated in EU-West. I see three approaches: (1) Single Redis cluster in one region, accepting 100-200 ms latency for cross-region writes — suitable for low-throughput global limits. (2) Per-region Redis clusters with asynchronous counter sync via a message queue — suitable for eventually-consistent limits with bounded staleness. (3) CRDT-based counters (like Redis with CRDTs or a custom solution) that merge concurrent increments without conflict — the most complex but most correct approach. For most systems, approach (2) strikes the best balance between consistency and latency."

**Challenge 5: "How do you handle tier-1 enterprise customers who need higher limits?"**
**Defense:** "The rate limiter should be configuration-driven. Each customer has a tier attribute, and the rate limit middleware loads the limit from a configuration source — a config service, a database, or a local file. Enterprise customers have a higher limit but may also have a different algorithm: token bucket with a larger burst capacity, for example. The configuration should support per-customer overrides, and we should publish a self-service dashboard where enterprise admins can request limit increases (subject to approval and billing)."

**Why this matters:** The best way to prepare for a system design interview is to have someone push back on your decisions. This drill trains you to defend your choices without becoming defensive, and to concede gracefully when a trade-off is genuinely better.
