Code review is one of the most valuable quality gates in software development, yet it is also one of the slowest. Pull requests routinely sit unmerged for days waiting for a reviewer to find a free block of focused time. When reviews finally happen, reviewers juggle multiple files, context-switch between tasks, and miss bugs that a fresh pair of eyes would catch. For small teams without a dedicated reviewer, the bottleneck is even worse — everyone is stretched thin and code quality suffers as a result.

An automated code review tool does not replace human judgment, but it handles the first pass: scanning every changed line, flagging common issues, and surfacing problems before a human ever opens the PR. This frees your team to focus on architecture, design decisions, and the subtle bugs that only come from experience.

## What You'll Build

You will build a CLI tool called `pr_review.py` that analyzes a git branch's diff and generates structured code review comments using Claude Haiku. The tool will:

- Parse the git diff of your current branch against the base branch (main or master)
- Send the diff to Claude Haiku with a structured review prompt
- Check for common issues: dead code, missing error handling, hardcoded values, logic errors, style violations
- Generate inline-style comments per file with line numbers
- Rate the confidence of each finding (high / medium / low)
- Output a formatted Markdown summary you can paste into a PR comment or share with your team

## Concept in 5 Minutes

Think of the LLM as a tireless junior reviewer who reads every changed line, catches patterns, never gets tired, and never rushes. You feed it the diff — the exact lines added, removed, and modified — and it applies a checklist of review criteria you define. The result is a structured report: per-file findings with line numbers, a confidence rating for each issue, and a summary table.

The entire pipeline is three stages: extract the diff from git, send it to Claude, and format the response. The hard part — actually understanding the code — is handled by the LLM. Your job is to write a prompt that tells it what to look for and how to structure the answer.

## Prerequisites

- Python 3.10 or later installed on your system
- `pip install anthropic` (version 0.49 or later)
- An Anthropic API key with access to Claude Haiku
- A git repository with at least one branch diverged from main to test against

## Build It

### Step 3: Capture the Git Diff

The core of any code review tool is the diff — the set of changes between your branch and the base branch. Build a function that runs `git diff` and captures the output.

```python
# diff_utils.py
import subprocess
import sys


def get_current_branch() -> str:
    """Return the name of the current git branch."""
    result = subprocess.run(
        ["git", "rev-parse", "--abbrev-ref", "HEAD"],
        capture_output=True,
        text=True,
        check=True,
    )
    return result.stdout.strip()


def get_base_branch() -> str:
    """Detect the base branch: prefer main, fall back to master."""
    for candidate in ["main", "master"]:
        result = subprocess.run(
            ["git", "rev-parse", "--verify", candidate],
            capture_output=True,
            text=True,
        )
        if result.returncode == 0:
            return candidate
    sys.exit("Error: no base branch (main or master) found.")


def get_diff(base_branch: str) -> str:
    """Return the git diff between base_branch and the current branch."""
    result = subprocess.run(
        ["git", "diff", base_branch + "...HEAD"],
        capture_output=True,
        text=True,
        check=True,
    )
    return result.stdout
```

Test it from the command line. In any git repository with a branch, run:

```bash

branch = get_current_branch()
base = get_base_branch()
diff = get_diff(base)
print(f'Branch: {branch}')
print(f'Base:   {base}')
print(f'Diff size: {len(diff)} characters')

"
```

```
+++ b/src/app.py
@@ -10,6 +10,8 @@ def handle_request(data):
+    if data is None:
+        raise ValueError("data cannot be None")
...
```

**What happened:** The `diff_utils.py` module wraps three git commands. `rev-parse --abbrev-ref HEAD` returns the current branch name. `rev-parse --verify` tests whether a branch exists without failing on missing branches. The `git diff base...HEAD` command uses the three-dot syntax to show only the changes introduced by your branch compared to the base branch — this is the standard comparison for pull requests.

### Step 4: Write the Review Prompt

The quality of the review depends almost entirely on the prompt. Design a prompt that asks Claude to analyze the diff file by file, with specific review categories and a structured output format.

```python
# prompt.py

REVIEW_PROMPT = """You are a senior code reviewer. Review the following git diff and provide structured feedback.

For each file in the diff, analyze it for these categories:
1. LOGIC_ERRORS — potential bugs or incorrect assumptions
2. ERROR_HANDLING — missing try/except, unchecked edge cases
3. HARDCODED_VALUES — magic numbers or strings that should be configurable
4. DEAD_CODE — unused imports, unreachable branches, commented-out code
5. STYLE_ISSUES — violations of PEP 8 or Pythonic conventions

Return your analysis as valid JSON with this exact structure:
{
  "files": [
    {
      "file_path": "src/app.py",
      "findings": [
        {
          "line": 12,
          "category": "ERROR_HANDLING",
          "severity": "high",
          "message": "raise ValueError(\"data cannot be None\") on line 12 lacks a try/except wrapper — a None value from upstream will crash the request handler.",
          "confidence": 0.85
        }
      ]
    }
  ],
  "summary": {
    "total_files": 1,
    "total_findings": 1,
    "high_severity_count": 1,
    "medium_severity_count": 0,
    "low_severity_count": 0
  }
}

Guidelines:
- line must be an integer (0 if the finding applies to the whole file)
- severity must be "high", "medium", or "low"
- confidence must be a float between 0.0 and 1.0
- Only report actual issues — do not invent problems
- If the file has no issues, include it with an empty findings list

Here is the diff:

```
{diff_content}
```
"""
```

The `{diff_content}` placeholder will be filled with the actual diff at runtime using Python string formatting.

**What happened:** The prompt defines five review categories that cover the most common code review findings. It uses a JSON output schema so you can parse the response programmatically. Each finding includes a line number, severity level, and confidence score — three dimensions that help you triage which issues matter most. The structured format makes it straightforward to generate a Markdown report in the next step.

### Step 5: Parse the Structured Response

Claude returns the review as a JSON text block. Write a parser that extracts the JSON, validates it, and converts it into a Markdown summary.

```python
# report.py
import json
import re
from typing import Any


def parse_review_response(response_text: str) -> dict[str, Any]:
    """Extract and parse the JSON review from Claude's response."""
    # Try to extract JSON from code fence or raw text
    json_match = re.search(
        r"```(?:json)?\s*\n?(\{.*?\})\n?\s*```",
        response_text,
        re.DOTALL,
    )
    if json_match:
        raw = json_match.group(1)
    else:
        raw = response_text.strip()

    return json.loads(raw)


def generate_report(data: dict[str, Any]) -> str:
    """Convert the parsed review data into a Markdown report."""
    severity_colors = {
        "high": "🔴",
        "medium": "🟡",
        "low": "🟢",
    }

    lines = ["# Pull Request Review Report\n"]
    summary = data.get("summary", {})
    lines.append("## Summary")
    lines.append(f"- Files reviewed: {summary.get('total_files', 0)}")
    lines.append(f"- Total findings: {summary.get('total_findings', 0)}")
    lines.append(f"- High severity:   {summary.get('high_severity_count', 0)}")
    lines.append(f"- Medium severity: {summary.get('medium_severity_count', 0)}")
    lines.append(f"- Low severity:    {summary.get('low_severity_count', 0)}")
    lines.append("")

    for file_entry in data.get("files", []):
        file_path = file_entry["file_path"]
        findings = file_entry.get("findings", [])
        lines.append(f"## {file_path}")
        if not findings:
            lines.append("No issues found.")
            lines.append("")
            continue
        for f in findings:
            icon = severity_colors.get(f["severity"], "⚪")
            line_num = f.get("line", 0)
            category = f["category"]
            message = f["message"]
            confidence = f.get("confidence", 0.0)
            confidence_label = (
                "High" if confidence >= 0.7
                else "Medium" if confidence >= 0.4
                else "Low"
            )
            lines.append(
                f'{icon} **Line {line_num}** — {category} '
                f'(Severity: {f["severity"]}, Confidence: {confidence_label})'
            )
            lines.append(f"  {message}")
            lines.append("")
        lines.append("---")

    return "\n".join(lines)
```

**What happened:** The `parse_review_response` function handles both fenced and unfenced JSON responses from Claude, making it robust to small variations in output format. The `generate_report` function converts the structured JSON into a readable Markdown document with color-coded severity icons, line numbers, categories, and confidence levels. This report can be pasted directly into a PR comment or saved as a file.

### Step 6: Full Pipeline

Combine all the pieces into a single command-line application that detects the base branch, captures the diff, sends it for review, and outputs the report.

```python
# pr_review.py
import os
import sys

from anthropic import Anthropic

from config import API_KEY, MODEL, MAX_TOKENS_PER_RUN, MAX_ITERATIONS
from diff_utils import get_current_branch, get_base_branch, get_diff
from prompt import REVIEW_PROMPT
from report import parse_review_response, generate_report


def review_diff(diff_content: str) -> str:
    """Send the diff to Claude and return the raw response text."""
    if not diff_content.strip():
        return '{"files": [], "summary": {"total_files": 0, "total_findings": 0, '
        '"high_severity_count": 0, "medium_severity_count": 0, '
        '"low_severity_count": 0}}'

    client = Anthropic(api_key=API_KEY)
    prompt = REVIEW_PROMPT.format(diff_content=diff_content)

    response = client.messages.create(
        model=MODEL,
        max_tokens=MAX_TOKENS_PER_RUN,
        messages=[{"role": "user", "content": prompt}],
    )
    return response.content[0].text


def main() -> None:
    iteration = 0
    while iteration < MAX_ITERATIONS:
        try:
            current_branch = get_current_branch()
            base_branch = get_base_branch()
            print(f"Reviewing branch: {current_branch}")
            print(f"Against base:     {base_branch}")
            print("Fetching diff...")
            diff = get_diff(base_branch)
            print(f"Diff is {len(diff)} characters.")
            if not diff:
                print("No changes found. The branch is up to date with "
                      f"{base_branch}.")
                return
            print("Sending for review...")
            raw_response = review_diff(diff)
            print("Parsing response...")
            review_data = parse_review_response(raw_response)
            report = generate_report(review_data)
            output_path = "review_report.md"
            with open(output_path, "w", encoding="utf-8") as f:
                f.write(report)
            print(f"Review report written to {output_path}")

            # Print a quick summary to stdout
            summary = review_data.get("summary", {})
            print(f"\nFindings: {summary.get('total_findings', 0)} "
                  f"(High: {summary.get('high_severity_count', 0)}, "
                  f"Medium: {summary.get('medium_severity_count', 0)}, "
                  f"Low: {summary.get('low_severity_count', 0)})")
            return
        except Exception as e:
            iteration += 1
            print(f"Error (attempt {iteration}/{MAX_ITERATIONS}): {e}",
                  file=sys.stderr)
            if iteration >= MAX_ITERATIONS:
                print("Max retries reached. Exiting.", file=sys.stderr)
                sys.exit(1)


if __name__ == "__main__":
    main()
```

**What happened:** This is the complete pipeline. The `main` function orchestrates every stage: it detects the current branch and base branch, fetches the diff, sends it to Claude Haiku for review, parses the structured JSON response, generates a Markdown report, and writes it to `review_report.md`. The retry loop with `MAX_ITERATIONS` handles transient API errors gracefully without infinite retries.

Now run the pipeline against any git repository with unmerged changes:

```bash

```

Expected output:

```

Against base:     main
Fetching diff...
Diff is 2431 characters.
Sending for review...
Parsing response...
Review report written to review_report.md

```

## Make It Yours

The base implementation is deliberately simple so you can extend it to match your team's workflow.

**Different review depth.** Add a `--depth` flag that switches between quick and thorough modes. Quick mode uses `max_tokens=2048` and a shorter prompt focused on logic errors and crashes. Thorough mode uses `max_tokens=8192` and includes style guidelines, performance concerns, and test coverage suggestions. The model stays `claude-haiku-3-5` for both — only the prompt and token budget change.

```python
import argparse

parser = argparse.ArgumentParser()
parser.add_argument("--depth", choices=["quick", "thorough"], default="thorough")
args = parser.parse_args()
```

**CI integration.** To run the review automatically on every pull request, add a GitHub Actions step:

```yaml
# .github/workflows/pr-review.yml
name: AI Code Review
on: pull_request
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install anthropic python-dotenv
      - run: python3 pr_review.py
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

The `fetch-depth: 0` is important — without it, GitHub Actions only checks out a shallow clone with no branch history, and `git diff` will return nothing.

**Custom review checklist.** Modify the `REVIEW_PROMPT` in `prompt.py` to add categories specific to your project. For example, if you work with Django, add a `SECURITY` category that checks for SQL injection via raw queries, missing CSRF tokens, or exposed `DEBUG=True`. If you work with data pipelines, add a `DATA_VALIDATION` category that checks for unvalidated inputs and schema assumptions.
