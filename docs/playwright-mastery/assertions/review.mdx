# Assertions — Review

## Key Takeaways

1. **Prefer auto-retrying matchers.** Always use `expect(locator).toBeVisible()` over `expect(await locator.textContent()).toBe('...')`. Auto-retrying matchers poll the page and handle timing issues automatically.
2. **Use soft assertions for comprehensive test feedback.** When testing multiple aspects of a page, use `expect.soft()` so that all failures are reported in a single run instead of stopping at the first failure.
3. **Page-level assertions simplify URL and title checks.** Use `expect(page).toHaveURL()` and `expect(page).toHaveTitle()` instead of reading values manually.
4. **Custom matchers extend the assertion vocabulary.** Use `expect.extend()` to create domain-specific matchers that make your tests more expressive.
5. **Timeout configuration matters.** Default 5-second timeouts work for most cases, but increase them for slow-loading content or slow environments like CI.

## Recap

- Auto-retrying assertions poll every 500ms up to the configured timeout (default 5 seconds).
- Hard assertions stop the test on failure; soft assertions let the test continue and collect all failures.
- Use `toHaveText` for exact/regex text matches, `toContainText` for substring checks.
- Page-level assertions (`toHaveURL`, `toHaveTitle`) operate on the page object, not on locators.
- Custom matchers let you create reusable, descriptive assertions for your application.

## Quick Quiz

### Multiple Choice Questions

**1. Which of the following assertions is auto-retrying?**

A. `expect(await page.textContent('.title')).toBe('Hello')`
B. `expect(page.locator('.title')).toHaveText('Hello')`
C. `expect(page.locator('.title').textContent()).toBe('Hello')`
D. `expect(page.getByRole('heading')).toBeInstanceOf(Locator)`

**Answer: B.** `expect(locator).toHaveText()` is an auto-retrying matcher. Options A and C use manual text extraction which is not auto-retrying.

---

**2. How often does Playwright poll the page during an auto-retrying assertion?**

A. Every 100ms
B. Every 500ms
C. Every 1 second
D. Every 5 seconds

**Answer: B.** Playwright polls every 500ms during auto-retrying assertions.

---

**3. What is the default timeout for auto-retrying assertions?**

A. 2 seconds
B. 5 seconds
C. 10 seconds
D. 30 seconds

**Answer: B.** The default timeout is 5 seconds. You can configure it globally or per-assertion.

---

**4. What is the effect of using `expect.soft()`?**

A. The test runs faster
B. The test skips the assertion
C. The test continues after the assertion fails and reports the failure along with others
D. The assertion is retried indefinitely

**Answer: C.** Soft assertions allow the test to continue after a failure. All soft assertion failures are collected and reported together at the end.

---

**5. Which assertion would you use to check that a page navigated to `/dashboard` after clicking a button?**

A. `await expect(page.getByText('/dashboard')).toBeVisible()`
B. `await expect(page).toHaveURL('/dashboard')`
C. `await expect(page.locator('url')).toContainText('/dashboard')`
D. `await expect(page.url()).toBe('/dashboard')`

**Answer: B.** `expect(page).toHaveURL()` is the correct page-level assertion for checking the current URL. It is auto-retrying, so it waits for navigation to complete.

### Scenario Questions

**Scenario 1:** You are testing a search feature. After typing a query and pressing Enter, the results appear after a 3-second API call. Write assertions to verify the search results loaded correctly. Use appropriate timeouts.

```typescript
// Answer
await page.getByRole('searchbox').fill('playwright');
await page.keyboard.press('Enter');

// Wait for results with increased timeout due to slow API
await expect(page.getByText('Search results')).toBeVisible({ timeout: 10000 });
await expect(page.getByRole('listitem')).toHaveCount(10, { timeout: 10000 });

// Soft assertion for the first result's text
await expect.soft(page.getByRole('listitem').first()).toContainText('playwright');
```

---

**Scenario 2:** A test checks five elements on a page, but the first one is often missing due to a known issue. You want to see all failures in one run, not just the first. How do you structure the assertions?

```typescript
// Answer — use soft assertions for all five checks
await expect.soft(page.getByTestId('element-1')).toBeVisible();
await expect.soft(page.getByTestId('element-2')).toBeVisible();
await expect.soft(page.getByTestId('element-3')).toBeVisible();
await expect.soft(page.getByTestId('element-4')).toBeVisible();
await expect.soft(page.getByTestId('element-5')).toBeVisible();

// If you need to check if any failed after all soft assertions:
// test.info().errors will contain the collected failures
```
