# Assertions Overview

Difficulty: Beginner

## What You Will Learn

- How to use soft and hard assertions to control test flow
- How to check element text, visibility, count, and attributes
- How Playwright's polling auto-retry mechanism works for assertions
- How to write custom expect matchers using `expect.extend()`

## Prerequisites

You should understand how to locate elements with Playwright locators and perform actions like click, fill, and select.

## Why This Matters

Assertions are how your test decides if a feature is working correctly. Traditional testing frameworks check a condition once and fail immediately if it is not met. Playwright's auto-retrying assertions poll the page every 500ms until the condition is met or the timeout expires. This eliminates flaky assertions caused by timing issues, lazy-loading content, or animation delays.

## Quick Reference

| Method | Description | Auto-retrying |
|---|---|---|
| `toBeVisible()` | Element is visible in the viewport | Yes |
| `toHaveText(text)` | Element's text content matches exactly or by regex | Yes |
| `toContainText(text)` | Element's text contains the given string | Yes |
| `toHaveValue(value)` | Input/select has the given value | Yes |
| `toHaveCount(count)` | Locator matches exactly N elements | Yes |
| `toHaveAttribute(name, value)` | Element has the given attribute with optional value | Yes |
| `toBeEnabled()` | Element is enabled (not disabled) | Yes |
| `toBeDisabled()` | Element is disabled | Yes |
| `toBeChecked()` | Checkbox or radio is checked | Yes |
| `toBeFocused()` | Element has focus | Yes |
| `toHaveURL(url)` | Page URL matches (partial or exact, string or regex) | Yes |
| `toHaveTitle(title)` | Page title matches | Yes |
| `toHaveScreenshot()` | Visual comparison against a stored screenshot | Yes |
| `toBe(value)` | Generic expect (not auto-retrying) | No |
| `toEqual(value)` | Deep equality (not auto-retrying) | No |

## The Big Picture

<AsciiDiagram id="playwright-mastery/assertions/overview" alt="Auto-retrying assertion polling flow" caption="How Playwright assertions retry automatically" content={`
  expect(locator).toBeVisible()
       │
       ▼
  ┌─────────────────────────────────┐
  │  Poll every 500ms               │
  │  until: element visible OR      │
  │        timeout (5s default)     │
  └─────────────────────────────────┘
       │                    │
       ▼                    ▼
    PASS                  FAIL
                     (with details)
`} />

When you call an auto-retrying assertion like `expect(locator).toBeVisible()`, Playwright polls the locator every 500ms. If the condition is met within the timeout (default 5 seconds), the assertion passes. If the timeout expires, the assertion fails with a detailed error message that includes the actual state of the element.
