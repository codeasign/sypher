# Assertions — Avoid Mistakes

## When NOT to Use This

- **Do not use `toHaveText` with exact whitespace expectations.** The `toHaveText` method normalizes whitespace by default. If you need to verify exact whitespace, use `toContainText` with a specific substring or use the `useInnerText` option.
- **Do not assert element counts before the list finishes rendering.** Even though `toHaveCount` is auto-retrying, if you check the count too early in a lazy-loaded list, it might pass with the wrong initial count. Combine with a visibility assertion first.

## The Pattern to Remember

Always use auto-retrying matchers (`expect(locator).toBeVisible()`, `expect(locator).toHaveText()`) over manual checks like `expect(await locator.textContent()).toBe('text')`. Auto-retrying matchers handle timing issues automatically and provide better error messages.

## Common Mistakes

### Mistake 1: Using manual textContent instead of auto-retrying matchers

```typescript
// BAD — manual check, not auto-retrying, brittle
const text = await page.locator('.message').textContent();
expect(text).toBe('Success');

// GOOD — auto-retrying, handles timing
await expect(page.locator('.message')).toHaveText('Success');
```

### Mistake 2: Forgetting to await page-level assertions

```typescript
// BAD — missing await, assertion never runs
expect(page).toHaveURL('/dashboard');

// GOOD — awaited, assertion runs correctly
await expect(page).toHaveURL('/dashboard');
```

### Mistake 3: Not using .soft() for non-critical checks

```typescript
// BAD — first failure stops the test, hiding other issues
await expect(page.getByText('Welcome')).toBeVisible();
await expect(page.getByText('User name')).toBeVisible();
await expect(page.getByText('Email')).toBeVisible();

// GOOD — all three are checked, all failures reported
await expect.soft(page.getByText('Welcome')).toBeVisible();
await expect.soft(page.getByText('User name')).toBeVisible();
await expect.soft(page.getByText('Email')).toBeVisible();
```

### Mistake 4: Asserting exact text when contains is sufficient

```typescript
// BAD — fails if there is any extra text, whitespace, or line breaks
await expect(page.locator('.status')).toHaveText('Payment processed successfully');

// GOOD — more flexible, passes even with extra text
await expect(page.locator('.status')).toContainText('Payment processed');

// BAD — using toBe (non-retrying) instead of toHaveText
expect(await page.locator('.status').textContent()).toBe('Payment processed successfully');
```

### Mistake 5: Setting timeout too short for slow-loading content

```typescript
// BAD — 2 seconds might not be enough for a slow API response
await expect(page.getByText('Data loaded')).toBeVisible({ timeout: 2000 });

// GOOD — increase timeout for slow-loading content
await expect(page.getByText('Data loaded')).toBeVisible({ timeout: 10000 });

// BETTER — set a global timeout in playwright.config.ts
// expect: { timeout: 10000 },
```

## Troubleshooting

| Error | Cause | Solution |
|---|---|---|
| `toHaveText` fails on whitespace | Text contains extra spaces, tabs, or line breaks | Use `toContainText` instead, or pass `{ useInnerText: true }` to `toHaveText` |
| `toBeVisible` times out | Element is hidden (display:none, opacity:0, offscreen) | Check CSS properties in devtools; use `toBeInViewport` if element is technically visible but scrolled away |
| `toHaveCount` mismatch with lazy-loaded items | Items are loaded after the assertion timeout | Increase the timeout, or assert visibility of a parent container first |
| `expect(locator)` — received value must be a Locator or Page | You passed a raw element handle or a string | Wrap the locator in `expect()` correctly: `expect(page.locator(...))` |
