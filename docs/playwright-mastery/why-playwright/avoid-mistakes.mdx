# Why Playwright — Avoid Mistakes

**Reading time:** 6 minutes · **Difficulty:** Beginner

## When NOT to Use This

Playwright is a browser automation framework. It excels at testing web applications in real browsers. But it is not the right tool for every job.

| Use Playwright for | Do NOT use Playwright for |
|---|---|
| End-to-end browser testing | Simple API-only endpoint checks |
| Cross-browser compatibility testing | Testing native mobile apps (iOS/Android) |
| Visual regression testing | Performance benchmarking or load testing |
| Component testing in browser context | Command-line tool testing |
| Screenshot and PDF generation | Database query validation |

## The Pattern to Remember

Choose the right framework for the job:

- **Need to verify a REST API response?** Use `curl`, `httpie`, or a lightweight HTTP client. A full browser is overkill.
- **Need to test an iOS or Android native app?** Use Appium, XCUITest, or Espresso. Playwright controls browsers, not native UI components.
- **Need to benchmark page load performance?** Use Lighthouse, Sitespeed.io, or WebPageTest. Playwright has performance APIs, but it is not a dedicated benchmarking tool.
- **Need to test a web application in a real browser?** Playwright is the best choice. That is what it was built for.

## Common Mistakes

### Mistake 1: Choosing Playwright for simple API-only checks

Engineers sometimes reach for Playwright to test a REST API endpoint that returns JSON. Playwright launches a browser, creates a context, and opens a page — all to make a single HTTP request. Instead, use a lightweight HTTP tool or Playwright's built-in API testing context (which does not need a browser page), but for pure API validation, a dedicated tool is simpler.

```typescript
// Overkill — launching a browser just to check an API response
import { test } from '@playwright/test';
test('check API', async ({ page }) => {
  const response = await page.evaluate(async () => {
    const res = await fetch('https://api.example.com/health');
    return res.json();
  });
  expect(response.status).toBe('ok');
});
```

### Mistake 2: Assuming Playwright works on native mobile apps

Playwright can emulate mobile browsers — Safari on iOS, Chrome on Android — but it cannot interact with native UI elements like push notifications, camera controls, or native dialogs. For native mobile testing, you need a platform-specific framework.

### Mistake 3: Not using auto-wait and adding manual waits

Playwright automatically waits for elements to be actionable before interacting. Adding `page.waitForTimeout(2000)` is almost always a sign you are fighting a flaky condition that Playwright can handle better with locator-based waiting.

```typescript
// Bad — brittle manual wait
await page.waitForTimeout(3000);
await page.click('.submit-button');

// Good — Playwright auto-waits for the button to be visible and enabled
await page.click('.submit-button');
```

### Mistake 4: Running tests headed when they should be headless in CI

Running headed mode (visible browser window) in CI is wasteful. CI environments rarely have a display server, and headed mode consumes extra memory. Use `headless: true` in CI and headed mode only during local debugging.

```typescript
// playwright.config.ts
export default defineConfig({
  use: {
    headless: process.env.CI ? true : false,
  },
});
```

### Mistake 5: Not using the test generator for initial exploration

Playwright's code generator (`npx playwright codegen`) records your interactions as you click through an application and generates the test code. Many engineers write locators by hand when a single pass through the app with the generator would produce accurate, working selectors instantly.

```bash
npx playwright codegen https://example.com
```

## Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| `npm init playwright@latest` fails | Network issue or npm registry problem | Run `npm config set registry https://registry.npmjs.org` and retry |
| Browser download fails | Slow network or disk space | Run `npx playwright install --with-deps` or install browsers individually: `npx playwright install chromium` |
| Test won't launch browser | Headless mode in environment without display | Install system dependencies: `npx playwright install-deps` |
| Timeout on first test | Page is slow to load or URL is wrong | Increase timeout in config or use a local test server |
| Permissions error in CI | Missing system libraries | Use `mcr.microsoft.com/playwright` Docker image in CI |
