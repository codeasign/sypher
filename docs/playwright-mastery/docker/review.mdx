# Docker — Review

**Reading time:** 5 minutes · **Difficulty:** Intermediate

## Key Takeaways

1. The official Playwright Docker image `mcr.microsoft.com/playwright:v1.52.0-jammy` includes Node.js, Playwright, all three browsers, and all system dependencies — no additional installation needed.
2. Always pin the Docker image tag to the exact Playwright version in your `package.json` to avoid version mismatch failures.
3. Layer your Dockerfile for cache efficiency: copy `package*.json` first, run `npm ci`, then copy the rest of the source code.
4. Mount volumes for `test-results/` and `playwright-report/` when running containers so that test artifacts survive container cleanup.
5. Use Docker Compose for integration tests that depend on other services (web app, database, mock API), with `depends_on` and `condition: service_healthy`.

## Recap in 60 Seconds

1. Pull `mcr.microsoft.com/playwright:v1.52.0-jammy` — everything is pre-installed.
2. Write a Dockerfile that copies code, installs deps, and runs tests.
3. Add `.dockerignore` to exclude `node_modules/`.
4. Use `docker compose` with `depends_on` for multi-service tests.
5. Mount volumes so test results persist after the container exits.

### Scenario Questions

**Scenario 1:** Your Playwright tests pass locally on macOS but fail in your Docker container with "browserType.launch: Host system missing dependencies!" You are using `FROM node:20` as your base image. What is the most likely cause and how do you fix it?

**Scenario 2:** Your team's CI pipeline builds a new Docker image for every commit using `docker build -t playwright-tests . && docker run playwright-tests`. The pipeline takes 7 minutes, of which 4 minutes is the Docker build step. The source code changes frequently but `package.json` rarely changes. How would you restructure the Dockerfile to reduce build time?

### Answers

**Multiple Choice:** 1-C, 2-B, 3-B, 4-A, 5-B

**Scenario 1:** The `node:20` base image does not include the system libraries that Playwright browsers need (libnss3, libnspr4, libatk-bridge, etc.). The fix is to switch the base image to `mcr.microsoft.com/playwright:v1.52.0-jammy`, which has all system dependencies pre-installed. Alternatively, you could keep `node:20` and manually install the dependencies, but the official image is simpler and more reliable.

**Scenario 2:** Restructure the Dockerfile to separate dependency installation from source code copying. Copy `package*.json` and run `npm ci` first, then copy the rest of the source code. Docker caches the `npm ci` layer and only re-executes it when `package.json` or `package-lock.json` changes. This means most builds (where only source code changed) will skip the 4-minute `npm ci` step and only execute the fast `COPY . .` step:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.52.0-jammy
WORKDIR /app
# These layers are cached unless package.json changes
COPY package*.json ./
RUN npm ci --only=production
# This layer rebuilds every time source code changes
COPY . .
CMD ["npx", "playwright", "test"]
```
