# Docker — Build It

**Reading time:** 14 minutes · **Difficulty:** Intermediate

## Step by Step

### 1. Pull the official Playwright image

The official Playwright Docker image includes Node.js, Playwright, all three browser binaries, and all system dependencies pre-installed. Start by pulling the image that matches your Playwright version:

```bash
docker pull mcr.microsoft.com/playwright:v1.52.0-jammy
```

The `jammy` tag refers to Ubuntu 22.04 LTS, which is the recommended base. If your Playwright version is different, check the [available tags on Microsoft Artifact Registry](https://mcr.microsoft.com/en-us/product/playwright/about).

### 2. Run a quick smoke test

Verify the image works by running a container interactively:

```bash
docker run --rm -it mcr.microsoft.com/playwright:v1.52.0-jammy /bin/bash
```

Inside the container, verify Playwright can launch a browser:

```bash
npx playwright install
node -e "const { chromium } = require('playwright'); (async () => { const b = await chromium.launch(); console.log('Chromium works!'); await b.close(); })();"
```

Type `exit` to leave the container.

### 3. Create a Dockerfile

Create a `Dockerfile` in your project root that copies your test code into the Playwright image and runs your tests:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.52.0-jammy

# Set the working directory inside the container
WORKDIR /app

# Copy package files first (they change less often than source code)
COPY package*.json ./

# Install production dependencies only
RUN npm ci --only=production

# Copy the rest of the application code
COPY . .

# Set environment variables for CI-like behaviour
ENV CI=true

# Default command: run Playwright tests
CMD ["npx", "playwright", "test"]
```

The Dockerfile uses a multi-stage approach to layer caching. The `COPY package*.json ./` followed by `RUN npm ci` means Docker caches the `npm ci` layer. As long as `package*.json` does not change, Docker reuses the cached layer instead of re-installing dependencies on every build.

### 4. Create a .dockerignore

Create `.dockerignore` to prevent unnecessary files from being copied into the Docker image. This keeps the image small and avoids invalidating caches:

```
node_modules/
.git/
.gitignore
README.md
test-results/
playwright-report/
screenshots/
.DS_Store
.vscode/
*.log
```

The most important entry is `node_modules/`. Without it, `COPY . .` copies your local `node_modules/` into the container, overwriting the fresh `npm ci` installation and potentially mixing host and container packages.

### 5. Build the Docker image

Build the image with a tag for easy reference:

```bash
docker build -t playwright-tests .
```

This reads the `Dockerfile` and `.dockerignore` from the current directory and produces an image tagged `playwright-tests:latest`.

### 6. Run tests in the container

Run the container and execute the tests:

```bash
docker run --rm -v "$(pwd)/test-results:/app/test-results" -v "$(pwd)/playwright-report:/app/playwright-report" playwright-tests
```

The `-v` flags mount host directories into the container so that test results, screenshots, and the HTML report persist on your host machine after the container exits. The `--rm` flag removes the container after it finishes.

### 7. Set up Docker Compose for multi-service testing

Many web applications depend on backend services — a database, a mock API, or a separate frontend dev server. Docker Compose lets you define multiple services and network them together.

Create `docker-compose.yml` in your project root:

```yaml
services:
  app:
    build:
      context: ./app
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=test
    healthcheck:
      test: ["CMD", "node", "-e", "require('http').get('http://localhost:3000/health', r => process.exit(r.statusCode === 200 ? 0 : 1))"]
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 15s

  tests:
    build:
      context: .
      dockerfile: Dockerfile
    depends_on:
      app:
        condition: service_healthy
    environment:
      - BASE_URL=http://app:3000
      - CI=true
    volumes:
      - ./test-results:/app/test-results
      - ./playwright-report:/app/playwright-report
    command: ["npx", "playwright", "test"]
```

This Compose file defines two services:

- **app**: Your web application, built from the `./app` subdirectory. It exposes port 3000 and has a health check that waits for the `/health` endpoint to return 200.
- **tests**: Your Playwright test runner. It depends on `app` being healthy, receives the app's URL via the `BASE_URL` environment variable, and mounts volumes for test output.

Run both services together:

```bash
docker compose run --rm tests
```

The `run --rm` runs the `tests` service, which starts the `app` service first (because of `depends_on`), waits for it to be healthy, then executes the Playwright tests.

### 8. (Optional) Configure VS Code Dev Containers

For teams where every developer should have an identical environment, add a Dev Container configuration. Create `.devcontainer/devcontainer.json`:

```json
{
  "name": "Playwright Tests",
  "image": "mcr.microsoft.com/playwright:v1.52.0-jammy",
  "forwardPorts": [3000],
  "postCreateCommand": "npm ci",
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-playwright.playwright"
      ]
    }
  }
}
```

With this file, VS Code prompts you to "Reopen in Container" when you open the project. Inside the container, you have Node.js, Playwright, all browsers, and the Playwright VS Code extension — no local setup required.

## Build Something Real

Now create a production-ready Docker setup for a Playwright test project that tests a web application. You will need four files.

### File 1: Dockerfile

Create `Dockerfile`:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.52.0-jammy

WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

COPY . .

ENV CI=true

CMD ["npx", "playwright", "test"]
```

### File 2: docker-compose.yml

Create `docker-compose.yml`:

```yaml
services:
  app:
    build:
      context: ./app
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=test
    healthcheck:
      test: ["CMD", "node", "-e", "require('http').get('http://localhost:3000/health', r => process.exit(r.statusCode === 200 ? 0 : 1))"]
      interval: 5s
      timeout: 5s
      retries: 10
      start_period: 15s

  tests:
    build:
      context: .
      dockerfile: Dockerfile
    depends_on:
      app:
        condition: service_healthy
    environment:
      - BASE_URL=http://app:3000
      - CI=true
    volumes:
      - ./test-results:/app/test-results
      - ./playwright-report:/app/playwright-report
    command: ["npx", "playwright", "test"]
```

### File 3: .dockerignore

Create `.dockerignore`:

```
node_modules/
.git/
.gitignore
README.md
test-results/
playwright-report/
screenshots/
.DS_Store
.vscode/
*.log
```

### File 4: Configure Playwright for Docker

Make sure your `playwright.config.ts` reads the `BASE_URL` environment variable set by Compose:

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 4 : undefined,
  reporter: [
    ['html', { open: 'never' }],
    ['junit', { outputFile: 'test-results/junit.xml' }],
  ],
  use: {
    baseURL: process.env.BASE_URL || 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
  projects: [
    { name: 'chromium', use: { browserName: 'chromium' } },
    { name: 'firefox', use: { browserName: 'firefox' } },
    { name: 'webkit', use: { browserName: 'webkit' } },
  ],
});
```

### Run it

```bash
# Build the images and run tests
docker compose run --rm tests

# To rebuild the test image after code changes
docker compose build tests
docker compose run --rm tests
```
