# Docker — Avoid Mistakes

**Reading time:** 6 minutes · **Difficulty:** Intermediate

## When NOT to Use This

Docker is excellent for reproducible environments, but it is not always the right choice.

| Use Docker for | Do NOT use Docker for |
|---|---|
| CI/CD pipelines where environment consistency is critical | Quick local debugging where container startup latency is unacceptable |
| Teams with heterogeneous development machines | Single-developer projects on a well-maintained local environment |
| Multi-service integration tests (app + DB + mock services) | Tests that need GPU acceleration or direct hardware access |

Do not build a new Docker image for every CI run if you can use the official Playwright image directly. Instead of `docker build` on every CI run, reference `mcr.microsoft.com/playwright:v1.52.0-jammy` as the `runs-on` container or use the image in GitHub Actions with the `container:` option:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    container:
      image: mcr.microsoft.com/playwright:v1.52.0-jammy
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx playwright test
```

This avoids the Docker build step entirely and uses the pre-built Microsoft image.

## The Pattern to Remember

1. **Use the official Playwright image** — `mcr.microsoft.com/playwright:v1.52.0-jammy` has everything pre-installed
2. **Layer your Dockerfile** — Copy `package*.json` first, run `npm ci`, then copy source code. This maximises cache reuse.
3. **Mount volumes for test output** — Without volume mounts, test results disappear when the container exits
4. **Use Docker Compose for multi-service tests** — Define your app service, test service, and any dependencies in one file
5. **Match Playwright versions** — The Docker image tag must match the Playwright version in your `package.json`

## Common Mistakes

### Mistake 1: Not matching the Playwright version to the browser version tag

The official Docker image is tagged with a specific Playwright version, e.g., `v1.52.0-jammy`. If your `package.json` specifies `@playwright/test: "1.48.0"` but you use the `v1.52.0` image, the Playwright CLI inside the container may fail with a version mismatch.

```dockerfile
# Bad — uses a floating or mismatched tag
FROM mcr.microsoft.com/playwright:latest

# Good — pins to the version matching your package.json
FROM mcr.microsoft.com/playwright:v1.52.0-jammy
```

Check your Playwright version with `npx playwright --version` and use the matching Docker image tag.

### Mistake 2: Installing all devDependencies in the production image

The official Playwright image already includes Playwright and all browsers. Running `npm ci` (without `--only=production`) installs Playwright again from npm, which may overwrite the pre-installed version and increase the image size unnecessarily.

```dockerfile
# Bad — installs everything, including devDependencies that are already in the image
RUN npm ci

# Good — installs only production dependencies
RUN npm ci --only=production
```

If your tests depend on TypeScript or other dev tools, add them explicitly to `dependencies` in `package.json`, or run `npm ci` without `--only=production` and accept the slightly larger image.

### Mistake 3: Running as root inside the container

The official Playwright image includes a non-root user `pwuser` for security best practices. Running as root can cause permission issues with mounted volumes: files created by the container (screenshots, traces) are owned by root and cannot be deleted or modified by your host user.

```dockerfile
# Bad — runs as root
FROM mcr.microsoft.com/playwright:v1.52.0-jammy
WORKDIR /app
COPY . .
RUN npm ci --only=production
CMD ["npx", "playwright", "test"]

# Good — switches to pwuser
FROM mcr.microsoft.com/playwright:v1.52.0-jammy
WORKDIR /app
COPY . .
RUN npm ci --only=production
USER pwuser
CMD ["npx", "playwright", "test"]
```

If you need root access for installation steps, do the installations first, then switch to `pwuser` for the runtime command.

### Mistake 4: Forgetting .dockerignore (copying node_modules)

Without a `.dockerignore` file, the `COPY . .` instruction copies the entire project directory, including `node_modules/`. This causes two problems:

1. The image becomes very large because it includes all npm packages twice (once from `node_modules/` and once from `npm ci`)
2. If your local `node_modules/` was installed on a different OS or Node version, the copied packages may be incompatible with the container

```dockerignore
# Always include these in your .dockerignore
node_modules/
.git/
.gitignore
test-results/
playwright-report/
.DS_Store
```

### Mistake 5: Not handling screenshots and traces with volume mounts

When Playwright tests run inside a container, any screenshots, traces, or reports written to disk are stored inside the container's ephemeral filesystem. When the container exits (and is removed with `--rm`), those files are lost.

```bash
# Bad — no volume mount, test results vanish after container exits
docker run --rm playwright-tests

# Good — mounts host directories, results survive container removal
docker run --rm \
  -v "$(pwd)/test-results:/app/test-results" \
  -v "$(pwd)/playwright-report:/app/playwright-report" \
  playwright-tests
```

## Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| `browser crashes: missing system libraries` | Using a non-Playwright base image | Use `mcr.microsoft.com/playwright` instead of a plain Node image |
| `cannot open display` in headless mode | `DISPLAY` environment variable set in container | Run in headless mode: `npx playwright test` (headless is default) |
| `docker: command not found` in CI | Docker not installed on runner | Use the official GitHub Actions container feature: `container: mcr.microsoft.com/playwright:v1.52.0-jammy` |
| `permission denied on test-results` | Files created by root user in container | Use `USER pwuser` in Dockerfile, or set user with `--user pwuser` in `docker run` |
| Different Playwright version in image vs package.json | Version mismatch | Run `npx playwright --version` locally and use the matching Docker image tag |

## Production Checklist

Before putting your Dockerised Playwright tests into production, verify these items:

- [ ] Docker image tag pinned to a specific Playwright version (not `latest`)
- [ ] `.dockerignore` file exists and excludes `node_modules/` and `test-results/`
- [ ] Volume mounts configured for `test-results/` and `playwright-report/`
- [ ] Docker Compose defined for multi-service integration tests
- [ ] Image rebuilt only when `package*.json` changes (leveraging Docker layer caching)
- [ ] Non-root user (`pwuser`) used in Dockerfile
- [ ] `npm ci --only=production` used to avoid duplicating Playwright binaries
