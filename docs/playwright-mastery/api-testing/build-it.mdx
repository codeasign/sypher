# Build It — API Testing in Practice

## Step 1: Create an API Request Context

The `request.newContext()` method creates an isolated API context with its own base URL, headers, and authentication state.

```typescript
import { test, expect } from '@playwright/test';

test('create API context with base URL', async ({ request }) => {
  // The test fixture `request` is already a pre-configured APIRequestContext.
  // For custom configuration, create a new context:
  const apiContext = await request.newContext({
    baseURL: 'https://api.example.com',
    extraHTTPHeaders: {
      'Accept': 'application/json',
      'X-API-Key': 'test-api-key'
    }
  });

  // Any request made with this context automatically prefixes the baseURL
  // so you only need to provide the path
  const response = await apiContext.get('/health');
  expect(response.status()).toBe(200);

  // Clean up the context when done
  await apiContext.dispose();
});
```

Using a base URL keeps your tests DRY and makes it easy to switch between environments (staging, production) by changing a single configuration value.

## Step 2: Send a POST Request to Create Data

Use `apiContext.post()` to create resources via your API. This is the foundation of API-driven test setup.

```typescript
import { test, expect } from '@playwright/test';

test('create a user via API', async ({ request }) => {
  const response = await request.post('https://api.example.com/users', {
    data: {
      name: 'Test User',
      email: 'testuser@example.com',
      role: 'editor'
    }
  });

  // Assert the response status
  expect(response.status()).toBe(201);

  // Parse the response body
  const body = await response.json();
  expect(body).toHaveProperty('id');
  expect(body.name).toBe('Test User');
  expect(body.email).toBe('testuser@example.com');

  // The created user ID can be used in subsequent requests
  const userId = body.id;
  console.log(`Created user with ID: ${userId}`);
});
```

## Step 3: Use API for Test Setup, Then Verify with UI

The most powerful pattern in Playwright API testing is seeding data via API and then verifying it appears correctly in the browser.

```typescript
import { test, expect } from '@playwright/test';

test('seed user via API and verify in UI', async ({ page, request }) => {
  // --- API Setup: Create a user ---
  const createResponse = await request.post('https://api.example.com/users', {
    data: {
      name: 'Alice',
      email: 'alice@example.com'
    }
  });
  expect(createResponse.status()).toBe(201);
  const { id: userId } = await createResponse.json();

  // --- UI Test: Navigate and verify ---
  await page.goto('https://app.example.com/users');

  // The user created via API should appear in the UI table
  await expect(page.locator(`tr[data-user-id="${userId}"]`)).toBeVisible();
  await expect(page.locator(`tr[data-user-id="${userId}"] td.name`)).toHaveText('Alice');
  await expect(page.locator(`tr[data-user-id="${userId}"] td.email`)).toHaveText('alice@example.com');

  // --- Cleanup: Remove the created user via API ---
  const deleteResponse = await request.delete(`https://api.example.com/users/${userId}`);
  expect(deleteResponse.status()).toBe(200);
});
```

This three-phase pattern (setup, act, cleanup) keeps your tests independent and idempotent. Each test creates its own data, validates it, and removes it, so tests can run in any order without interfering with each other.

## Step 4: Mock Backend Responses with Route Interception

Use `page.route()` to intercept network requests and return mock responses. This lets you test how your UI handles error states, empty results, and edge cases without needing a real backend.

```typescript
import { test, expect } from '@playwright/test';

test('mock API returns empty list and UI shows empty state', async ({ page }) => {
  // Register a route handler BEFORE navigating
  await page.route('**/api/users', async route => {
    // Fulfill the request with mock data
    await route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify([]) // Empty user list
    });
  });

  await page.goto('https://app.example.com/users');

  // The UI should show an empty state message
  await expect(page.getByText('No users found')).toBeVisible();
  await expect(page.getByText('Create your first user')).toBeVisible();
});

test('mock API error and UI shows error message', async ({ page }) => {
  await page.route('**/api/users', async route => {
    await route.fulfill({
      status: 500,
      contentType: 'application/json',
      body: JSON.stringify({ error: 'Internal Server Error' })
    });
  });

  await page.goto('https://app.example.com/users');

  // The UI should display an error banner
  await expect(page.getByText('Something went wrong')).toBeVisible();
  await expect(page.getByRole('button', { name: 'Retry' })).toBeVisible();
});
```

Route mocking is especially valuable for testing error states that are difficult to reproduce with a real backend, such as 500 errors, timeouts, and malformed responses.

## Step 5: Test Authentication with API and Browser Shared State

You can authenticate via API and then share the session with the browser context using `storageState`.

```typescript
import { test, expect } from '@playwright/test';
import path from 'path';

test('authenticate via API and share session with browser', async ({ page, request }) => {
  // --- API Authentication ---
  const authResponse = await request.post('https://api.example.com/auth/login', {
    data: {
      email: 'admin@example.com',
      password: 'admin-password'
    }
  });
  expect(authResponse.status()).toBe(200);
  const { token } = await authResponse.json();

  // Use the token for subsequent API requests
  const authedContext = await request.newContext({
    baseURL: 'https://api.example.com',
    extraHTTPHeaders: {
      'Authorization': `Bearer ${token}`
    }
  });

  // --- Share auth state with the browser ---
  // Set the token in localStorage before navigating
  await page.goto('https://app.example.com/login');
  await page.evaluate((authToken) => {
    localStorage.setItem('auth_token', authToken);
  }, token);

  // Now navigate to the protected page — the browser is already authenticated
  await page.goto('https://app.example.com/dashboard');
  await expect(page.getByText('Welcome, Admin')).toBeVisible();

  // Clean up the API context
  await authedContext.dispose();
});
```

For larger test suites, create a global setup that authenticates once and saves the storage state to a file. Then reuse it across all tests to avoid logging in before every test.

## Step 6: Build Something Real

Combine API setup, UI interaction, API verification, and route mocking into a complete end-to-end test.

```typescript
import { test, expect } from '@playwright/test';

test('full API + UI workflow with error state', async ({ page, request }) => {
  // --- Phase 1: API Setup — Create 5 users ---
  const userIds: number[] = [];
  for (let i = 1; i <= 5; i++) {
    const res = await request.post('https://api.example.com/users', {
      data: {
        name: `User ${i}`,
        email: `user${i}@example.com`
      }
    });
    expect(res.status()).toBe(201);
    const { id } = await res.json();
    userIds.push(id);
  }

  // --- Phase 2: UI — Navigate and verify user list ---
  await page.goto('https://app.example.com/users');

  // Assert the user table shows exactly 5 rows
  const userRows = page.locator('table#user-table tbody tr');
  await expect(userRows).toHaveCount(5);

  // Verify each user name is visible
  for (let i = 1; i <= 5; i++) {
    await expect(page.getByText(`User ${i}`)).toBeVisible();
  }

  // --- Phase 3: UI — Delete a user ---
  const deleteButton = page.locator('tr[data-user-id="' + userIds[0] + '"] button.delete');
  await deleteButton.click();

  // Wait for the delete modal to appear and confirm
  await page.getByRole('button', { name: 'Confirm Delete' }).click();

  // Wait for the table to update (auto-wait handles the DOM change)
  await expect(userRows).toHaveCount(4);

  // --- Phase 4: API — Verify backend state ---
  const verifyResponse = await request.get('https://api.example.com/users');
  const allUsers = await verifyResponse.json();
  expect(allUsers).toHaveLength(4);

  // The deleted user should not be in the list
  const deletedUserStillExists = allUsers.some(
    (u: { id: number }) => u.id === userIds[0]
  );
  expect(deletedUserStillExists).toBe(false);

  // --- Phase 5: Test error state with route mocking ---
  // Register a mock that returns a 500 error on the next API call
  await page.route('**/api/users', async route => {
    // Only intercept if the request method is GET (the list endpoint)
    if (route.request().method() === 'GET') {
      await route.fulfill({
        status: 500,
        contentType: 'application/json',
        body: JSON.stringify({ error: 'Database connection failed' })
      });
    } else {
      await route.continue();
    }
  });

  // Trigger a reload of the user list
  await page.getByRole('button', { name: 'Refresh' }).click();

  // The UI should show the error message
  await expect(page.getByText('Database connection failed')).toBeVisible();
  await expect(page.getByRole('button', { name: 'Retry' })).toBeVisible();

  // --- Phase 6: Cleanup — Remove all created users ---
  // Unregister the route mock first
  await page.unroute('**/api/users');

  for (const id of userIds) {
    const delRes = await request.delete(`https://api.example.com/users/${id}`);
    expect(delRes.status()).toBe(200);
  }

  // Final verification that cleanup worked
  const finalCheck = await request.get('https://api.example.com/users');
  const remainingUsers = await finalCheck.json();
  expect(remainingUsers).toHaveLength(4); // The 4 users that weren't deleted via UI

  console.log('Full API + UI workflow test passed!');
});
```

This comprehensive test demonstrates all the key patterns:

- **API for setup**: Creates 5 users quickly without browser interaction
- **API for teardown**: Cleans up test data after the test completes
- **API for verification**: Confirms the UI deletion actually removed the record from the backend
- **Route mocking**: Tests the error UI by intercepting the API response
- **Route cleanup**: Unregisters mocks so they do not affect other tests
