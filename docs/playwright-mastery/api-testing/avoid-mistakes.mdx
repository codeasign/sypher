# Avoid Common API Testing Mistakes

## When NOT to Use This

Avoid API shortcuts in these scenarios:

- **Do not bypass critical UI flows with API calls.** If you are testing a checkout process, always complete the purchase through the browser UI — do not call the checkout API directly and then verify on the confirmation page. You would miss real-world issues like button rendering, form validation, and payment gateway interactions.
- **Do not mock every API call.** Over-mocking makes your tests isolated but unrealistic. A test that mocks every backend response might pass while the real application is broken. Use real API calls for the primary flow you are testing, and mock only for error states, edge cases, and third-party services you cannot control.
- **Do not rely on shared persistent test data.** Tests that assume specific data already exists in the database are fragile — they break when another test modifies that data or when the database is reset. Each test should create its own test data and clean it up afterward.

## The Pattern to Remember

Use APIs for what they do best and the browser for what it does best:

- **API for setup and teardown** — create test data before your test runs and clean it up after. This is faster than navigating UI forms and gives you precise control over the data state.
- **Browser for user-facing behavior** — verify that the UI renders correctly, user interactions work, and the application responds as expected.
- **API for backend state verification** — after a UI action (like deleting an item), call the API to confirm the backend reflects the change. This catches issues where the UI appears to succeed but the API call actually failed.

## Common Mistakes

Here are the five most common mistakes in Playwright API testing.

### Mistake 1: Not Using `request.newContext` Isolation Between Tests

```typescript
import { test, expect } from '@playwright/test';

// BAD — tests share state through a global API context
let sharedApiContext: any;

test.beforeAll(async ({ request }) => {
  sharedApiContext = await request.newContext({
    baseURL: 'https://api.example.com',
  });
});

test('bad test 1', async () => {
  await sharedApiContext.post('/users', { data: { name: 'User A' } });
});

test('bad test 2', async () => {
  // This test inherits the auth headers and cookies from test 1
  // If test 1 left the context in an invalid state, this test fails
});
```

```typescript
// GOOD — each test gets its own isolated API context
import { test, expect } from '@playwright/test';

test('good test 1', async ({ request }) => {
  // The `request` fixture is automatically scoped per test
  const res = await request.post('https://api.example.com/users', {
    data: { name: 'User A' }
  });
  expect(res.status()).toBe(201);
});

test('good test 2', async ({ request }) => {
  // Clean state — no interference from test 1
  const res = await request.post('https://api.example.com/users', {
    data: { name: 'User B' }
  });
  expect(res.status()).toBe(201);
});
```

The `request` fixture in Playwright creates a fresh `APIRequestContext` for each test. Always use this fixture rather than creating a shared context in `beforeAll`.

### Mistake 2: Mocking APIs and Forgetting to Unmock

```typescript
import { test, expect } from '@playwright/test';

// BAD — route mock leaks into subsequent tests
test('bad test: mock not cleaned up', async ({ page }) => {
  await page.route('**/api/users', async route => {
    await route.fulfill({ status: 200, body: '[]' });
  });
  await page.goto('https://app.example.com/users');
  // ... assertions ...
  // No page.unroute() call — the mock persists!
});

// GOOD — register mock in beforeEach, unregister in afterEach
test.describe('mocked API tests', () => {
  test.beforeEach(async ({ page }) => {
    await page.route('**/api/users', async route => {
      await route.fulfill({
        status: 200,
        contentType: 'application/json',
        body: JSON.stringify([{ id: 1, name: 'Mocked User' }])
      });
    });
  });

  test.afterEach(async ({ page }) => {
    await page.unroute('**/api/users');
  });

  test('users page shows mocked data', async ({ page }) => {
    await page.goto('https://app.example.com/users');
    await expect(page.getByText('Mocked User')).toBeVisible();
  });
});
```

If you forget to call `page.unroute()`, the mock intercepts requests in all subsequent tests in the same file and can cause confusing failures when those tests expect real data.

### Mistake 3: Not Cleaning Up Test Data After Tests

```typescript
import { test, expect } from '@playwright/test';

// BAD — test data accumulates in the database
test('create user and verify', async ({ page, request }) => {
  const res = await request.post('https://api.example.com/users', {
    data: { name: 'Temp User', email: 'temp@example.com' }
  });
  const { id } = await res.json();

  await page.goto('https://app.example.com/users');
  await expect(page.locator(`tr[data-user-id="${id}"]`)).toBeVisible();

  // No cleanup — the user is left in the database!
});

// GOOD — cleanup in afterEach
test.describe('user CRUD tests', () => {
  let createdUserId: number;

  test.afterEach(async ({ request }) => {
    if (createdUserId) {
      await request.delete(`https://api.example.com/users/${createdUserId}`);
    }
  });

  test('create user and verify in UI', async ({ page, request }) => {
    const res = await request.post('https://api.example.com/users', {
      data: { name: 'Cleanup User', email: 'cleanup@example.com' }
    });
    expect(res.status()).toBe(201);
    const { id } = await res.json();
    createdUserId = id;

    await page.goto('https://app.example.com/users');
    await expect(page.locator(`tr[data-user-id="${id}"]`)).toBeVisible();
  });
});
```

Leaving test data in the database causes three problems: it pollutes the test environment, it may cause unique constraint violations on subsequent test runs, and it degrades performance over time as the test data accumulates.

### Mistake 4: Mixing API Authentication with Browser Authentication Incorrectly

```typescript
import { test, expect } from '@playwright/test';

// BAD — assumes the browser automatically uses the API session
test('mixing auth contexts incorrectly', async ({ page, request }) => {
  // Authenticate via API
  const authRes = await request.post('https://api.example.com/auth/login', {
    data: { email: 'admin@test.com', password: 'password' }
  });
  const { token } = await authRes.json();

  // Navigate — the browser does NOT automatically use the API session!
  await page.goto('https://app.example.com/dashboard');
  // The user is redirected to login because the browser has no auth token
  await expect(page).toHaveURL('**/login');
});
```

```typescript
// GOOD — explicitly share the auth state with the browser
test('sharing auth state correctly', async ({ page, request }) => {
  const authRes = await request.post('https://api.example.com/auth/login', {
    data: { email: 'admin@test.com', password: 'password' }
  });
  const { token } = await authRes.json();

  // Set the token in the browser's localStorage before navigation
  await page.goto('https://app.example.com');
  await page.evaluate((t) => {
    localStorage.setItem('auth_token', t);
  }, token);

  // Now the navigation will use the stored token
  await page.goto('https://app.example.com/dashboard');
  await expect(page.getByText('Welcome')).toBeVisible();
});
```

The API context and browser context are completely separate. Cookies, tokens, and session data from your API calls are not automatically shared with the browser. You must explicitly transfer the authentication state.

### Mistake 5: Not Handling Non-JSON Responses

```typescript
import { test, expect } from '@playwright/test';

// BAD — assumes all responses are JSON
test('handle non-JSON response badly', async ({ request }) => {
  const res = await request.get('https://api.example.com/export/csv');
  // This throws if the response body is not valid JSON!
  const data = await res.json();
  expect(data.rows).toBeDefined();
});

// GOOD — check content type before parsing
test('handle non-JSON response correctly', async ({ request }) => {
  const res = await request.get('https://api.example.com/export/csv');
  expect(res.status()).toBe(200);

  const contentType = res.headers()['content-type'] || '';

  if (contentType.includes('application/json')) {
    const data = await res.json();
    expect(data.rows).toBeDefined();
  } else if (contentType.includes('text/csv')) {
    const csvText = await res.text();
    expect(csvText).toContain('Name,Email,Role');
  } else {
    // For binary responses like PDF exports
    const buffer = await res.body();
    expect(buffer.length).toBeGreaterThan(0);
  }
});
```

The `res.json()` method throws a `SyntaxError` if the response body is not valid JSON. Always check the `Content-Type` header or use `res.text()` / `res.body()` for non-JSON responses.

## Troubleshooting

| Symptom | Likely Cause | Solution |
|---|---|---|
| API context unauthorized | No auth headers set on the context | Check your `extraHTTPHeaders` configuration; authenticate first and pass the token |
| `route.fulfill` gets ignored | Route pattern does not match the actual request URL | Turn on `page.on('request')` logging to see the exact URL; check the request method in the route handler |
| JSON parse error on empty body | API returned 204 No Content with an empty body | Check the response status before calling `res.json()`; use `res.text()` and handle empty strings |
| CORS errors in mock | Browser enforces CORS even for mocked routes | Set the correct `Access-Control-Allow-Origin` header in `route.fulfill({ headers: ... })` |
| API data persists across tests | Tests do not clean up created resources | Use `test.afterEach` to delete test data; use unique identifiers per test run such as timestamps or UUIDs |

## Production Checklist

Before considering your API testing approach ready for CI, verify these items:

- [ ] All tests use the per-test `request` fixture rather than a shared `APIRequestContext`
- [ ] Route mocks are registered in `beforeEach` and unregistered in `afterEach` (or use `page.unrouteAll()`)
- [ ] Every test cleans up any data it creates, or uses isolated test environments that auto-reset
- [ ] API tests that share authentication with the browser explicitly transfer tokens or cookies via `page.evaluate` or `storageState`
- [ ] Response body parsing handles non-JSON content types (HTML, CSV, binary) without crashing
- [ ] Tests that combine API and UI do not bypass critical user-facing flows — the browser still performs the actions you want to validate
