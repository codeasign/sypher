# Review — API Testing

## Key Takeaways

1. **Use the `request` fixture for per-test API isolation** — Playwright provides an `APIRequestContext` via the `request` fixture that is scoped to each test, so test data and authentication state never leak between tests.
2. **Combine API calls with browser testing in a three-phase pattern** — use API for setup (create data) and teardown (clean up data), and use the browser for the actual user-facing interaction you want to validate.
3. **Mock backend responses with `page.route()` and `route.fulfill()`** — route interception lets you simulate error states (500 errors, empty responses, timeouts) that are difficult to reproduce with a real backend.
4. **Always register mocks in `beforeEach` and unregister in `afterEach`** — forgotten route mocks leak into sibling tests and cause confusing failures when tests expect real data but receive mocked responses.
5. **API authentication does not automatically transfer to the browser** — cookies and tokens obtained from API calls must be explicitly set in the browser context via `page.evaluate()` or `storageState()`.

## Recap

In this lesson you learned how to use Playwright's `APIRequestContext` to send HTTP requests directly from your tests. You discovered the three-phase pattern (API setup, browser actions, API verification) and saw how to mock backend responses with route interception. You also learned how to share authentication state between API calls and the browser, and how to handle non-JSON responses safely. These patterns let you write faster, more reliable tests that cover both the frontend and backend of your application.

### Scenario Questions

**Scenario 1:** You have a test suite for a task management application. Each test creates tasks via API, performs UI actions on the task list, and then checks results. After running the full suite several times, you notice that the database has thousands of leftover test tasks and some tests fail with duplicate key errors. What is wrong and how do you fix it?

**Answer:** The tests are not cleaning up the task data they create. Over multiple runs, test tasks accumulate in the database. When a test tries to create a task with a name or identifier that already exists (from a previous run), the API returns a 409 Conflict or the test fails on a uniqueness constraint. The fix is to add cleanup in `test.afterEach` or `test.afterAll` that deletes all tasks created by the test. Use unique identifiers (e.g., `task-${Date.now()}-${Math.random()}`) for task names to avoid collisions, and add an `afterEach` hook that calls the DELETE endpoint for each created task.

**Scenario 2:** You need to test that your application's user list page shows an appropriate message when the API returns an empty array, when it returns a 500 error, and when it returns 100 users. You do not have a test database that you can load with different data sets. How would you structure your tests to cover all three cases?

**Answer:** Use Playwright's `page.route()` to mock the API endpoint for each test case:

1. **Empty array test:** Register a route for `**/api/users` that fulfills with `{ status: 200, body: '[]' }`. Navigate to the page and assert the empty state message is visible.
2. **500 error test:** Register the same route but fulfill with `{ status: 500, body: JSON.stringify({ error: 'Server error' }) }`. Assert the error banner and retry button are visible.
3. **100 users test:** Generate an array of 100 mock user objects, register the route to return that array, and assert the UI shows all 100 rows (e.g., `await expect(page.locator('table tbody tr')).toHaveCount(100)`).

Use `test.describe` to group these tests and register/unregister the mock in `beforeEach`/`afterEach` hooks so each test starts with a clean routing table.
