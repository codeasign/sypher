# Avoid Mistakes with Visual Testing

## When NOT to Use This

- **Static third-party content**: Don't screenshot embedded iframes, maps, or external widgets you don't control. Their content changes unpredictably and will break your tests.
- **Text-heavy content**: Use `toHaveText()` or `toContainText()` for text assertions — they are faster, more precise, and produce actionable error messages. Visual testing for text content gives you a 1000-pixel diff that says "something changed" instead of "expected 'Submit' but found 'Send'".
- **Dynamic landing pages**: Hero sections with rotating testimonials, random background images, or A/B-tested copy are poor candidates for visual testing. Mask or skip these sections.
- **User-specific dashboards**: Screenshots of user profiles or personalized dashboards vary per test account. Test the layout with a mock user and mask account-specific data.

## The Pattern to Remember

- Mask every dynamic element: timestamps, badges, ads, avatars, notifiers.
- Set per-component thresholds: a full-page screenshot needs higher tolerance than a single button screenshot.
- Update baselines intentionally — never auto-accept new baselines without human review.

## Common Mistakes

**1. Not masking dynamic content**

Timestamps, loading spinners, and ads change every run. Without masks, each test run produces different pixels and your tests fail constantly.

```typescript
// BAD: dynamic timestamp causes flaky comparison
await expect(page).toHaveScreenshot('dashboard.png');

// GOOD: mask the timestamp
await expect(page).toHaveScreenshot('dashboard.png', {
  mask: [page.getByTestId('current-time')],
});
```

**2. Using the same threshold for all screenshots**

A full-page screenshot with charts and gradients needs a higher threshold than a small button screenshot. Using one threshold everywhere either misses subtle regressions or causes false positives.

```typescript
// BAD: same threshold for everything
const THRESHOLD = 0.1;
await expect(page).toHaveScreenshot('chart.png', { threshold: THRESHOLD });
await expect(page).toHaveScreenshot('button.png', { threshold: THRESHOLD });
```

**3. Not updating baselines after intentional UI changes**

A designer updates the color scheme. You see the visual test fail and — instead of updating the baseline — you increase the threshold to silence it. Now real regressions slip through.

```typescript
// BAD: silencing instead of updating
await expect(page).toHaveScreenshot('dashboard.png', {
  maxDiffPixelRatio: 0.5, // Hides regressions
});
```

**4. Cross-browser rendering differences cause spurious failures**

Font rendering, subpixel positioning, and anti-aliasing differ across Chrome, Firefox, and Safari. A test that passes on macOS Chrome may fail on Windows Firefox.

```typescript
// Each browser needs its own baseline
// Playwright auto-names: name-chromium-win32.png, name-firefox-win32.png
// But if thresholds are too tight, valid cross-browser diffs fail
```

**5. Using `fullPage: true` on infinite-scroll pages**

An infinite-scroll page never finishes loading. `fullPage: true` captures whatever is in the viewport at the time, producing inconsistent screenshots.

```typescript
// BAD: infinite scroll never ends
await expect(page).toHaveScreenshot('feed.png', { fullPage: true });
```

## Troubleshooting

| Symptom | Likely Cause | Solution |
|---|---|---|
| Screenshot comparison: 5000 different pixels | Antialiasing, font rendering, or subpixel alignment differences | Increase `threshold` to 0.2 and `maxDiffPixels` to 200; ensure same OS screenshots |
| Baseline not found on CI | `snapshotDir` misconfigured or baselines not committed | Verify snapshotDir path; commit `__snapshots__/` folder |
| Animations cause flaky screenshots | CSS animations or transitions run during capture | Set `animations: 'disabled'` or wait for animation end |
| Font rendering differs between OS | Different font engines (ClearType vs CoreText) | Use separate baselines per OS (Playwright does this by default) |
| Test passes locally, fails in CI | Different screen resolution, OS font rendering, or browser version | Run CI with same browser versions; use Docker for consistency |
| Diff image is entirely one color | Page failed to load (blank/error page compared to baseline) | Check if page.goto succeeded; add waitForSelector before screenshot |

## Production Checklist

- [ ] All dynamic content is masked (timestamps, ads, notifications, avatars)
- [ ] Animations are set to `'disabled'` or handled with proper waits
- [ ] Different thresholds are configured per component type
- [ ] Baselines are committed to the repository and reviewed in PRs
- [ ] CI snapshot storage is configured (artifact storage for diff images)
- [ ] Team has a documented process for updating baselines (PR with side-by-side comparison)
- [ ] Cross-browser and cross-platform baselines are managed (separate baselines per platform)
- [ ] Screenshots are captured after the page is fully loaded (not during rendering)
