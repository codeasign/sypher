# Visual Testing Review

## Key Takeaways

1. **Mask dynamic content**: Every timestamp, ad, badge, and notification that changes between runs must be masked to keep visual tests reliable.
2. **Set per-component thresholds**: A full-page chart needs higher tolerance than a single icon. Tune thresholds per test, not globally.
3. **Update baselines intentionally**: Never auto-accept new baselines. Review visual diffs in a PR to confirm the change is intentional.
4. **Separate baselines per platform**: Playwright auto-names snapshots by browser and OS. Commit them all so cross-platform rendering differences don't cause failures.
5. **Use visual testing for layout, not text**: Catch CSS shifts, overlapping elements, and broken layouts. Use text assertions for content verification.

### Scenario Questions

**Scenario 1**: Your visual test for a dashboard page passes locally on macOS Chrome but fails on CI running Linux Chrome with 3000 different pixels. The diff image shows the chart area is slightly shifted. What are the likely causes? How would you fix it?

**Scenario 2**: A teammate updated the button styles from blue to green and the visual test failed. They increased `maxDiffPixelRatio` to 0.5 to make the test pass. Two weeks later, a CSS change accidentally shifts the button off-screen and the test still passes. What went wrong? What should the process be?

### Answer Key

**MCQ Answers**: 1-C, 2-B, 3-B, 4-B, 5-B

**Scenario 1**: Likely causes: antialiasing differences between macOS and Linux font renderers, different browser versions, or subpixel positioning differences in the chart library. Fix: run tests in Docker with consistent rendering, increase threshold to 0.2-0.3 for the chart test, or use separate baselines per platform (Playwright does this automatically — make sure Linux baselines are committed).

**Scenario 2**: The teammate used threshold inflation to paper over a real visual change instead of updating the baseline. This hid the subsequent layout regression. The correct process: when a visual test fails due to intentional changes, the baseline should be updated and committed as part of the same PR. The PR reviewer should inspect the before/after diff to confirm the change is intentional. Never increase thresholds to silence failures — fix the root cause or update the baseline.
