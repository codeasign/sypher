# Locators — Review

## Key Takeaways

1. **Prefer `getByRole` first.** ARIA role locators are the most resilient because they match how users and assistive technology interact with the page. Always start here before falling back to other strategies.
2. **Chain locators for precision.** When a page has multiple sections with similar elements, chain a parent locator to a child locator instead of writing deep CSS selectors.
3. **Use `.filter()` to disambiguate.** When a locator matches multiple elements, narrow it down with `.filter({ hasText, has, hasNotText, hasNot })` rather than using brittle nth-child selectors.
4. **Use `getByTestId` as your fallback for custom components.** When semantic roles and labels do not apply, add a `data-testid` attribute. It is explicit and will not change with styling updates.
5. **CSS and XPath are last resorts.** They are brittle, tied to DOM structure, and harder to maintain. Use them only when no semantic locator works.

## Recap

- Playwright provides multiple locator strategies: role, label, placeholder, text, testid, and CSS/XPath.
- The recommended priority is role > label > placeholder > testid > text > CSS/XPath.
- Chaining and filtering give you precise targeting without deep CSS nesting.

## Quick Quiz

### Multiple Choice Questions

**1. Which locator method should you prefer first when writing a Playwright test?**

A. `page.locator('button.submit')`
B. `page.getByRole('button', { name: /submit/i })`
C. `page.$('button')`
D. `page.getByXPath('//button')`

**Answer: B.** `getByRole` is the most preferred locator strategy because it uses accessible names and is resilient to DOM changes.

---

**2. What happens when `page.getByText('Login')` matches two elements?**

A. The test clicks the first one automatically
B. Playwright throws a strict mode violation error
C. Both elements are clicked
D. The test waits indefinitely

**Answer: B.** Playwright enforces strict mode by default. If a locator matches multiple elements, it throws an error. Use `.first()` or `.filter()` to narrow the match.

---

**3. Which locator is best for a form input that has an associated `<label>` element?**

A. `page.getByPlaceholder()`
B. `page.getByTestId()`
C. `page.getByLabel()`
D. `page.locator('input')`

**Answer: C.** `getByLabel` finds the input by its associated `<label>` text, which is both semantic and resilient.

---

**4. How do you configure a custom test ID attribute in Playwright?**

A. `page.setTestIdAttribute('data-test')`
B. Set `testIdAttribute: 'data-test'` in `playwright.config.ts` under `use`
C. Pass it to each `getByTestId` call
D. It is not configurable

**Answer: B.** The `testIdAttribute` option in `playwright.config.ts` under the `use` section lets you change the attribute that `getByTestId` looks for.

---

**5. Which of the following is NOT a valid way to narrow a locator that matches multiple elements?**

A. `.first()`
B. `.nth(2)`
C. `.filter({ hasText: 'Active' })`
D. `.last()` — wait, that is valid too

Wait — all of `.first()`, `.nth()`, and `.filter()` are valid. Let me rephrase.

**5. What does `.filter({ hasText: 'Active' })` return when applied to a locator that matches 5 list items, 2 of which contain the text "Active"?**

A. The first matching list item
B. A locator matching exactly the 2 items with "Active"
C. An error because of strict mode
D. A locator matching all 5 items

**Answer: B.** `.filter()` returns a new locator that matches only the elements that satisfy the filter condition. It does not throw a strict mode error.

### Scenario Questions

**Scenario 1:** You are testing a navigation menu with three sections: "Home", "Products", "Contact". Each section has its own list of sub-items, and the "Products" section contains a sub-item called "Pricing". Write a locator that finds the "Pricing" link only inside the "Products" section.

```typescript
// Answer
const pricingLink = page
  .getByRole('navigation')
  .filter({ hasText: 'Products' })
  .getByRole('link', { name: 'Pricing' });
```

---

**Scenario 2:** A test fails with "strict mode violation" on `page.getByRole('button').click()`. The page has a "Submit" button at the top and a "Cancel" button at the bottom. Both have the same role. How do you fix the locator so it only clicks "Submit"?

```typescript
// Answer — use the name option to disambiguate
await page.getByRole('button', { name: /submit/i }).click();

// Alternatively, if the buttons are in separate sections:
await page.locator('.form-footer').getByRole('button', { name: /submit/i }).click();
```
