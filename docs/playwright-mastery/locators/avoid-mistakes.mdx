# Locators — Avoid Mistakes

## When NOT to Use This

- **Do not use CSS selectors for dynamic content.** If your CSS selector relies on classes, IDs, or data attributes that change between builds or environments, your tests will break. Prefer role, label, or text locators that match what the user sees.
- **Do not use XPath unless you have no other option.** XPath is brittle, harder to read, and not supported for all Playwright actions. Modern web apps rarely need it.
- **Do not use `page.locator()` with a complex CSS chain like `div > .form > div > input[type="text"]:nth-child(2)`.** A single class rename or DOM restructure will break it. Use chaining with semantic methods instead.

## The Pattern to Remember

Follow the priority hierarchy from the overview every time you write a locator:

1. Start with `getByRole()` and an accessible name
2. If no role fits, try `getByLabel()` or `getByPlaceholder()` for form fields
3. Fall back to `getByTestId()` for custom components
4. Use `getByText()` for text-based elements
5. Use `.locator()` with CSS only as a last resort

When you chain locators, keep the chain short. A chain of two is usually enough: `parentLocator.childLocator`. Three at most.

## Common Mistakes

### Mistake 1: Using brittle CSS selectors with deep nesting

```typescript
// BAD — brittle and tied to DOM structure
await page.locator('div.container > div.row > div.col-md-6 > button.btn-primary').click();

// GOOD — semantic and resilient
await page.getByRole('button', { name: /submit/i }).click();
```

### Mistake 2: Matching multiple elements with getByText

```typescript
// BAD — if "Login" appears in both a heading and a button, this fails
await page.getByText('Login').click();

// GOOD — be specific or use .first()
await page.getByRole('button', { name: /login/i }).click();

// Alternatively, if you need the first matching element
await page.getByText('Login').first().click();
```

### Mistake 3: Not using .filter() when multiple elements match

```typescript
// BAD — finds all list items including ones you don't want
await page.getByRole('listitem').click();

// GOOD — filter to the specific one you need
await page.getByRole('listitem').filter({ hasText: 'Submit' }).click();
```

### Mistake 4: Relying on XPath

```typescript
// BAD — XPath is brittle and hard to read
await page.locator('//div[@class="form"]//button[text()="Login"]').click();

// GOOD — use Playwright's built-in methods
await page.getByRole('button', { name: /login/i }).click();
```

### Mistake 5: Forgetting to escape special characters in text locators

```typescript
// BAD — parentheses and brackets can cause unexpected matches
await page.getByText('Total ($100.00)').click();

// GOOD — use getByRole or escape properly
// If this is a button:
await page.getByRole('button', { name: /total.*100\.00/i }).click();
// Or use getByTestId for complex text
await page.getByTestId('total-amount').click();
```

## Troubleshooting

| Error | Cause | Solution |
|---|---|---|
| `locator.hover: Target closed` | The page navigated or a dialog closed the element | Ensure no navigation happens before the action; use `await` properly |
| Strict mode violation | The locator matches more than one element | Use `.first()`, `.nth(N)`, or add `.filter()` to narrow the match |
| `getByRole` can't find element | The element does not have the expected ARIA role or accessible name | Inspect the element in devtools; check if it has `role` or `aria-label` |
| Element not attached to DOM | The element was removed and re-added (e.g., SPA navigation) | Use `page.waitForSelector()` or rely on Playwright's auto-waiting |
