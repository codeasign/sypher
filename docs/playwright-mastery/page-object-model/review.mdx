# Page Object Model — Review

Review the Page Object Model best practices, test your knowledge with a quiz, and prepare for interview questions.

## Key Takeaways

1. Page Object Model encapsulates page structure and behavior into classes, separating what the test does (business flow) from how the browser does it (Playwright API calls).
2. Locators must be defined once per class, never duplicated across methods. This ensures a single point of change when the UI updates.
3. Playwright fixtures provide dependency injection for page objects, keeping test code clean and eliminating repetitive instantiation.
4. Page objects should be stateless — read values from the DOM on demand rather than caching them in class fields.
5. Composition over inheritance: page objects hold a reference to `Page` and compose behavior, rather than extending a base class.

## Recap

- **Encapsulation** — each page or view gets one class with all its locators and interaction methods
- **Fixtures** — Playwright's fixture system injects page objects into tests without manual construction
- **Maintainability** — when the UI changes, you update the page object class, not the tests

## Quiz

### Multiple Choice Questions

**1. What is the primary benefit of defining locators as class properties in a page object?**

- A) They run faster than inline locators
- B) The selector string lives in one place, so UI changes require one edit
- C) It reduces the number of imports needed
- D) Locators defined in constructors are automatically lazy

<details>
<summary>Answer</summary>
B — A single source of truth for each selector. When the UI changes, update the locator property, not every method.
</details>

**2. Which Playwright feature is the recommended way to provide page objects to tests?**

- A) Global `beforeEach` hooks that assign to `globalThis`
- B) Fixtures created with `test.extend()`
- C) Static factory methods on page objects
- D) Dependency injection via constructor parameters in test files

<details>
<summary>Answer</summary>
B — `test.extend()` creates typed fixtures that inject page objects into tests cleanly.
</details>

**3. What should a page object method that navigates to a new page return?**

- A) `void`
- B) `this`
- C) A new page object instance for the destination page
- D) A boolean indicating success

<details>
<summary>Answer</summary>
C — Returning the destination page object enables fluent navigation chains.
</details>

**4. Why should page objects avoid storing state in class fields?**

- A) Class fields are slower than DOM queries
- B) The stored value can drift from the actual DOM state, causing flaky tests
- C) TypeScript does not allow class fields on page objects
- D) Playwright fixtures clear class fields between tests automatically

<details>
<summary>Answer</summary>
B — DOM state changes independently of the page object. Reading fresh from the DOM guarantees accuracy.
</details>

**5. What is the best approach when two page objects share a common UI component (e.g., a header with navigation)?**

- A) Create a base class that both page objects extend
- B) Create a separate component class and compose it into both page objects
- C) Duplicate the locators in each page object
- D) Write the shared locators as standalone functions

<details>
<summary>Answer</summary>
B — Composition over inheritance. A component class (like `Header`) can be instantiated inside multiple page objects without the coupling of inheritance.
</details>

### Scenario Questions

**Scenario 1:** Your test suite has 200 tests across 15 page objects. The product team changed the navigation bar, moving the "Settings" link from a top menu to a sidebar. The locator for the settings link was defined once in a shared `Header` component class. How many locator changes do you need to make?

<details>
<summary>Answer</summary>
One change — in the `Header` component class's locator property. Every page object that composes the `Header` class automatically uses the updated locator. This is exactly why composition and single-locator-definition save maintenance time at scale.
</details>

**Scenario 2:** A junior developer created a page object where every method accepts a `page` parameter instead of using the `page` instance stored in the constructor. The tests look like `await loginPage.login(page, 'user', 'pass')`. What is wrong with this approach, and how do you refactor it?

<details>
<summary>Answer</summary>
Passing `page` as a parameter defeats the purpose of encapsulation. The page object already holds a `page` reference from its constructor. The refactored version removes the redundant parameter: `await loginPage.login('user', 'pass')`. The method uses `this.page` internally. This also simplifies the test code and makes the test's intent clearer.
</details>
