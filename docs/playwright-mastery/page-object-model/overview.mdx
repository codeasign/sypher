# Page Object Model

Difficulty: Intermediate

Reading time: 10 minutes

## What You Will Learn

- Design Page Object Model classes that encapsulate page structure and behavior
- Compose page objects to model multi-page workflows
- Use Playwright fixtures for dependency injection of page objects
- Structure test suites with POM for long-term maintainability

## Prerequisites

- TypeScript basics for Playwright (see [TypeScript Basics](../typescript-basics/overview))
- Playwright project set up with `npx playwright init`
- Understanding of test fixtures in Playwright

## Why This Matters

The Page Object Model (POM) is the industry-standard pattern for maintainable test suites. Without it, a 50-test file becomes an unreadable mess of duplicated locators, inline URL strings, and copy-pasted interaction logic. POM turns each page or view into a class with meaningful methods — `loginPage.login('user', 'pass')` instead of `page.locator('#username').fill('user')` spread across twenty tests. When the login form changes, you update one class, not every test.

## Quick Reference

| Concept | Example | Purpose |
|---|---|---|
| POM class | `class LoginPage` | Encapsulate one page's locators and actions |
| Method | `async login(user: string, pass: string)` | Expose meaningful business actions |
| Fixture injection | `test.use({ loginPage: new LoginPage(page) })` | Provide page objects to tests via DI |
| Composition | `DashboardPage` accepts `Page` | Page objects hold a reference to `Page`, not extend it |
| Return self | `return this` | Enable method chaining on page objects |

## The Big Picture

A test uses fixtures to obtain page objects. Each page object wraps a single view or page, exposing business-level methods. The page objects interact with the browser through Playwright's `Page` API. This creates clean separation: tests describe what the user does, page objects describe how the browser does it.
