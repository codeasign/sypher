# Page Object Model — Avoid Mistakes

Page Object Model is powerful, but misapplied it turns into a maintenance burden. This lesson covers when NOT to use POM, the core pattern to always follow, and the five most common mistakes.

## When NOT to Use This

Do not force Page Object Model on a 2-page application. If your app has a login page and one dashboard view, inline locators in the test may be simpler and more readable. POM shines when you have 5, 10, or 50 pages — not 1 or 2.

Do not over-abstract single-use interactions. If a particular button is clicked in exactly one test, creating a dedicated page object method for it adds indirection without benefit. Extract it when a third test needs it, not before.

## The Pattern to Remember

One class per page or view. Methods return `this` (for chaining) or another page object (for navigation). Locators are defined once as class properties, never duplicated across methods.

```typescript
// Correct pattern — one class per view, locators at the top, methods return self or another page
export class CheckoutPage {
  constructor(private page: Page) {
    // All locators defined in constructor
  }

  async fillShipping(address: string): Promise<this> { /* ... */ return this; }
  async submitOrder(): Promise<ConfirmationPage> { /* ... */ return new ConfirmationPage(this.page); }
}
```

## Common Mistakes

### 1. Storing Page-Scoped State in Class Fields

Page objects should be stateless. Storing values like `this.currentCount` or `this.lastError` introduces subtle test-ordering bugs.

```typescript
// Wrong — stateful page object
class TodoPage {
  private itemCount = 0;

  async addTodo(text: string): Promise<void> {
    await this.input.fill(text);
    await this.button.click();
    this.itemCount++; // This does not reflect the actual DOM state
  }
}

// Correct — stateless, always reads from the DOM
class TodoPage {
  async getItemCount(): Promise<number> {
    return this.todoItems.count();
  }
}
```

### 2. Making Methods Not Return Anything

Methods that perform an action should return `this` for chaining or a page object for the next view. Methods that read data should return the value. Void methods break the fluent pattern.

```typescript
// Wrong — void method breaks chaining
class LoginPage {
  async login(username: string, password: string): Promise<void> {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }
}

// Correct — returns void intentionally, but accepts this tradeoff for clarity
class LoginPage {
  async login(username: string, password: string): Promise<void> {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
    // Login transitions to dashboard but the fixture provides DashboardPage separately
  }
}
```

### 3. Not Using Fixtures for Page Objects

Directly instantiating page objects in every test creates duplication and makes setup changes painful.

```typescript
// Wrong — manual instantiation in every test
test('login', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('user', 'pass');
});

test('other test', async ({ page }) => {
  const loginPage = new LoginPage(page); // Duplicated
  await loginPage.goto();
  // ...
});

// Correct — fixture provides the page object
test('login', async ({ loginPage }) => {
  await loginPage.goto();
  await loginPage.login('user', 'pass');
});
```

### 4. Locator Duplication Across Methods

Repeating the same selector string in every method makes the page object fragile. If the selector changes, you must update every method.

```typescript
// Wrong — locator string duplicated
class ProfilePage {
  async getName(): Promise<string> {
    return (await page.locator('.profile-name').textContent()) ?? '';
  }
  async setName(name: string): Promise<void> {
    await page.locator('.profile-name').fill(name); // Duplicated string
  }
}

// Correct — locator defined once
class ProfilePage {
  readonly nameInput = page.locator('.profile-name');

  async getName(): Promise<string> {
    return (await this.nameInput.textContent()) ?? '';
  }
  async setName(name: string): Promise<void> {
    await this.nameInput.fill(name);
  }
}
```

### 5. Tight Coupling Between Page Objects

Page objects that call each other's methods or depend on each other's internal state create a brittle web of dependencies.

```typescript
// Wrong — LoginPage knows about DashboardPage internals
class LoginPage {
  async login(username: string, password: string): Promise<void> {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
    await this.page.waitForSelector('.dashboard-stats'); // Coupling!
  }
}

// Correct — LoginPage only handles login; tests compose pages
class LoginPage {
  async login(username: string, password: string): Promise<void> {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
    // The test waits for the next page using DashboardPage methods
  }
}
```

## Troubleshooting

| Error | Cause | Fix |
|---|---|---|
| `fixture "loginPage" not found` | The fixture is not exported or the test imports from the wrong file | Verify the test imports `test` from your `fixtures.ts`, not from `@playwright/test` |
| `locator chain breaks with "strict mode violation"` | Multiple elements match a locator | Make the locator more specific using `.first()`, `.nth()`, or `.filter({ hasText: ... })` |
| `target closed` during page object method | The browser context or page was closed before the action completed | Check that all async methods are awaited and that fixtures manage the page lifecycle |
| `Cannot read properties of undefined` on locator | The locator was defined before the page navigated or the DOM was ready | Use `waitFor()` or define locators after navigation in the test flow |

## Production Checklist

- [ ] All locators are defined once in the constructor or as class-level properties
- [ ] Methods return typed Promises (`Promise<void>`, `Promise<string>`, or `Promise<this>`)
- [ ] Fixtures are scoped properly (use `scope: 'test'` — the default — for page objects)
- [ ] No page object stores test-state data in class fields
- [ ] No page object references another page object's internal locators
- [ ] Locator selectors are resilient — prefer `getByRole`, `getByText`, `getByTestId` over CSS class names
