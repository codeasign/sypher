# Build It: AI Test Generation

Follow these steps to generate Playwright tests using AI prompts.

## Step 1: Write a Prompt for User Registration

Open your AI tool (Claude, ChatGPT, or Copilot) and paste this prompt:

```
Generate a complete Playwright test file for a user registration page with the following fields: email, password, confirm password, and a "Sign Up" button. Include these test cases:

1. Valid registration: fill all fields correctly and click Sign Up, then verify a success message appears
2. Password mismatch: enter different passwords in the password and confirm password fields, verify an error message is shown
3. Invalid email format: enter an invalid email like "notanemail" in the email field, verify a validation error
4. Empty required fields: click Sign Up with all fields empty, verify validation messages

Use Playwright best practices: getByRole and getByLabel locators, proper assertions, and a describe block. Assume the registration form is at the route /register on http://localhost:3000.
```

The AI will generate something like this:

```typescript
import { test, expect } from '@playwright/test';

test.describe('User Registration', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('http://localhost:3000/register');
  });

  test('should register successfully with valid data', async ({ page }) => {
    await page.getByLabel('Email').fill('user@example.com');
    await page.getByLabel('Password').fill('SecurePass123!');
    await page.getByLabel('Confirm Password').fill('SecurePass123!');
    await page.getByRole('button', { name: 'Sign Up' }).click();

    await expect(page.getByText('Registration successful')).toBeVisible();
  });

  test('should show error when passwords do not match', async ({ page }) => {
    await page.getByLabel('Email').fill('user@example.com');
    await page.getByLabel('Password').fill('SecurePass123!');
    await page.getByLabel('Confirm Password').fill('DifferentPass456!');
    await page.getByRole('button', { name: 'Sign Up' }).click();

    await expect(page.getByText('Passwords do not match')).toBeVisible();
  });

  test('should show error for invalid email format', async ({ page }) => {
    await page.getByLabel('Email').fill('notanemail');
    await page.getByLabel('Password').fill('SecurePass123!');
    await page.getByLabel('Confirm Password').fill('SecurePass123!');
    await page.getByRole('button', { name: 'Sign Up' }).click();

    await expect(page.getByText('Please enter a valid email')).toBeVisible();
  });

  test('should show validation errors for empty fields', async ({ page }) => {
    await page.getByRole('button', { name: 'Sign Up' }).click();

    await expect(page.getByText('Email is required')).toBeVisible();
    await expect(page.getByText('Password is required')).toBeVisible();
    await expect(page.getByText('Confirm password is required')).toBeVisible();
  });
});
```

Review the generated code. Check for these quality indicators:

- Are all locators using best practices (getByRole, getByLabel)?
- Are there meaningful assertions (checking for visible text, not just presence)?
- Are all four requested test cases covered?
- Is the beforeEach hook used to avoid repeating the goto?

## Step 2: Iterate with a Follow-Up Prompt

Now refine the generated tests with this prompt:

```
Add a test to the registration suite that mocks the API response for an already-registered email. Use page.route to intercept the registration POST request and return a 409 Conflict response with a JSON body: { "error": "Email already in use" }. Verify the error message is shown to the user.
```

The AI should generate an additional test like this:

```typescript
test('should show error for already-registered email', async ({ page }) => {
  await page.route('**/api/register', async (route) => {
    await route.fulfill({
      status: 409,
      contentType: 'application/json',
      body: JSON.stringify({ error: 'Email already in use' }),
    });
  });

  await page.getByLabel('Email').fill('existing@example.com');
  await page.getByLabel('Password').fill('SecurePass123!');
  await page.getByLabel('Confirm Password').fill('SecurePass123!');
  await page.getByRole('button', { name: 'Sign Up' }).click();

  await expect(page.getByText('Email already in use')).toBeVisible();
});
```

## Step 3: Generate Test Data

Use AI to create test data for data-driven testing:

```
Generate a JSON array of 5 user objects for data-driven registration tests. Each object should have: name (string), email (string, must be unique), password (string, at least 8 chars with 1 number), role (one of "admin", "user", "viewer"). Include one admin, two users, and two viewers.
```

The AI should produce something like:

```json
[
  { "name": "Alice Admin", "email": "alice@test.com", "password": "Admin1234", "role": "admin" },
  { "name": "Bob User", "email": "bob@test.com", "password": "UserPass1", "role": "user" },
  { "name": "Carol User", "email": "carol@test.com", "password": "CarolPass2", "role": "user" },
  { "name": "Dave Viewer", "email": "dave@test.com", "password": "Viewer123", "role": "viewer" },
  { "name": "Eve Viewer", "email": "eve@test.com", "password": "EvePass456", "role": "viewer" }
]
```

Use this data in a data-driven Playwright test:

```typescript
import { test, expect } from '@playwright/test';

const users = [
  { name: 'Alice Admin', email: 'alice@test.com', password: 'Admin1234', role: 'admin' },
  { name: 'Bob User', email: 'bob@test.com', password: 'UserPass1', role: 'user' },
  { name: 'Carol User', email: 'carol@test.com', password: 'CarolPass2', role: 'user' },
  { name: 'Dave Viewer', email: 'dave@test.com', password: 'Viewer123', role: 'viewer' },
  { name: 'Eve Viewer', email: 'eve@test.com', password: 'EvePass456', role: 'viewer' },
];

test.describe('Data-driven registration', () => {
  users.forEach(({ name, email, password, role }) => {
    test(`should register ${name} as ${role}`, async ({ page }) => {
      await page.goto('http://localhost:3000/register');

      await page.getByLabel('Name').fill(name);
      await page.getByLabel('Email').fill(email);
      await page.getByLabel('Password').fill(password);
      await page.getByLabel('Confirm Password').fill(password);
      await page.getByRole('combobox', { name: 'Role' }).selectOption(role);
      await page.getByRole('button', { name: 'Sign Up' }).click();

      await expect(page.getByText(`Welcome, ${name}!`)).toBeVisible();
    });
  });
});
```

## Step 4: Build Something Real

Now build a complete test suite for a todo application from scratch.

Write this feature description for your AI:

```
Generate a complete Playwright test suite for a todo application with the following features:
- Users can add a todo with a title and due date
- Users can mark a todo as complete
- Users can delete a todo
- Todos are displayed in a list

Generate tests for:
1. Add a new todo and verify it appears in the list
2. Mark a todo as complete and verify it shows as completed
3. Delete a todo and verify it is removed from the list
4. Try to add a todo with an empty title and verify an error
5. Add 3 todos and verify all appear in the correct order

Assume the app runs at http://localhost:3000 with a simple HTML/JS frontend.
Use getByRole and getByLabel locators. Add proper assertions with expect.
```

After the AI generates the initial suite, refine it with these follow-up prompts:

**Follow-up 1: Page Object Model**

```
Refactor the todo test suite to use the Page Object Model pattern. Create a TodoPage class with methods: addTodo(title, dueDate), completeTodo(title), deleteTodo(title), getTodoCount(). Keep the same test scenarios.
```

**Follow-up 2: Visual regression**

```
Add visual regression tests to the todo suite using Playwright's toHaveScreenshot(). Take screenshots of the initial empty state, a list with 3 todos, a completed todo, and an empty todo list after deleting all items.
```

**Follow-up 3: API mocking for offline mode**

```
Add a test that uses page.route to simulate an offline scenario. When the user tries to add a todo while offline, show an error message "Unable to save. You are offline." Use route.abort() to simulate the network failure.
```

**Follow-up 4: Edge cases**

```
Add the following edge case tests to the todo suite:
1. Try adding a todo with a past due date — verify a warning is shown
2. Try adding 1000 todos rapidly — verify performance (the list should render within 2 seconds)
3. Try completing a todo that is already completed — verify it remains completed
4. Try deleting a todo while another user is interacting with the same list — simulate with a second page context
```

After each prompt, review the AI's output. Run the generated tests. Fix any issues. The final result should be a complete, reviewed, production-ready test suite covering happy paths, error states, edge cases, visual regression, and API mocking.
