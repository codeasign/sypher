# Parallel Execution — Review

## Key Takeaways

1. **Workers are isolated browser processes** — each worker gets its own browser, context, and page. No state leaks between workers when you use Playwright fixtures correctly.
2. **Sharding distributes tests across CI machines** — use `--shard=<N>/<M>` to split your suite into M slices. Each CI machine runs one slice independently.
3. **Worker count matters** — oversubscribing (workers > CPU cores) causes memory thrashing. The sweet spot for browser tests is 0.5-0.75 workers per CPU core.
4. **Serial mode keeps dependent tests in one worker** — use `test.describe.configure({ mode: 'serial' })` for tests that build on each other's state, but keep the group small to avoid long worker lock times.
5. **Projects multiply parallelism** — each project (chromium, firefox, webkit) runs your test files against a different configuration. Workers are shared across all projects in the pool.

### Scenario Questions

**Scenario 1:** Your team has 120 test files, 3 browser projects (chromium, firefox, webkit), and you want CI feedback under 5 minutes. Each test takes 6 seconds on average. Your CI runners have 8 CPU cores and 8 GB RAM.

**Question:** Design the worker count, shard count, and project configuration. Calculate the approximate completion time per shard.

**Answer:** 120 files x 3 projects = 360 total runs. With 8 cores, use 4-6 workers per shard. Pick 6 shards: 360 / 6 = 60 tests per shard. 60 tests / 5 workers = 12 batches. 12 x 6 seconds = 72 seconds. The suite finishes in about 1.2 minutes per shard, well under the 5-minute target.

**Scenario 2:** A test suite with 8 test files consistently fails in CI when all files run together, but every file passes when run individually with `npx playwright test tests/single-file.spec.ts`.

**Question:** What three things should you check first? How would you fix each?

**Answer:** (1) Check for shared global state — move cookies/localStorage to per-test fixtures or use `storageState`. (2) Check for DB record collisions — use unique test data per file (e.g., UUID suffixes on usernames). (3) Check for file-level variable pollution — ensure variables are declared inside `test()` blocks, not at module scope.
