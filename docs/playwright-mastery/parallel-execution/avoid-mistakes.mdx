# Parallel Execution — Avoid Mistakes

## When NOT to Use This

Parallel execution is not always the right choice. Avoid it in these scenarios:

- **Tests share state** — if your tests read and write to the same database records, files, or in-memory data, parallel runs will collide. Use serial execution or isolate state per test.
- **Shard count exceeds test files** — sharding into more pieces than you have test files produces empty shards. Each shard receives at most 1 test file when shard count >= file count. This wastes CI machines.
- **Setup overhead dominates** — if each test takes under 1 second and the browser launch time (3-5 seconds) is the majority of the runtime, parallelism adds complexity for minimal gain. Consider batching or increasing test complexity.

## The Pattern to Remember

Isolate test state per file. Use Playwright test fixtures to create fresh state for each test. Never rely on global variables or shared browser state across parallel workers. Each worker gets its own browser, context, and page — leverage that.

```typescript
// Good: fixture-based isolation
import { test as base, expect } from '@playwright/test';

const test = base.extend({
  // Each test gets a fresh authenticated context
  authenticatedPage: async ({ browser }, use) => {
    const context = await browser.newContext({
      storageState: '.auth/user.json',
    });
    const page = await context.newPage();
    await use(page);
    await context.close();
  },
});

test('works in isolation', async ({ authenticatedPage }) => {
  await authenticatedPage.goto('/dashboard');
  // ...
});
```

## Common Mistakes

### Mistake 1: Workers Exceeding CPU Cores

Setting `workers` higher than your CPU count causes context switching overhead and memory thrashing. Browser processes are heavy — each one takes 200-500 MB.

```typescript
// BAD — 8 workers on a 4-core machine
export default defineConfig({ workers: 8 });

// GOOD — 2-4 workers on a 4-core machine
export default defineConfig({ workers: 2 });
```

For browser tests, the sweet spot is 0.5 to 0.75 workers per CPU core. On a 4-core machine, 2-3 workers. On an 8-core CI runner, 4-6 workers.

### Mistake 2: Tests Sharing Global State

Cookies, localStorage, authentication tokens, and in-memory counters set by one test can leak to another in parallel runs. Even if your tests pass in serial, they may fail intermittently in parallel.

```typescript
// BAD — auth token leaks between tests
let authToken = '';
test('login', async ({ page }) => {
  const response = await page.request.post('/api/login');
  authToken = await response.text(); // global variable — collision!
});

// GOOD — use storageState or per-test fixtures
import { test, expect } from '@playwright/test';

test.use({ storageState: '.auth/user.json' });
test('dashboard loads', async ({ page }) => {
  await page.goto('/dashboard');
  await expect(page.locator('h1')).toHaveText('Dashboard');
});
```

### Mistake 3: Shard Count Exceeding Test Files

If you have 10 test files and set `--shard=12/12`, shards 11 and 12 will report "No tests found." Each shard receives at most one test file when the shard count equals or exceeds the file count.

```
10 test files, 4 shards → shards get 3, 3, 2, 2 tests (balanced)
10 test files, 10 shards → shards get 1 test each (okay)
10 test files, 12 shards → shards 11 and 12 get 0 tests (wasted)
```

### Mistake 4: Not Adjusting Retries for Parallel Runs

Retries multiply by your worker count in parallel runs. If you have 4 workers and `retries: 3`, a failing test can consume up to 4 workers x 4 runs (1 original + 3 retries) = 16 worker slots. This delays other tests.

```typescript
// For parallel runs, keep retries low
export default defineConfig({
  fullyParallel: true,
  retries: process.env.CI ? 2 : 0, // 2 max in CI
  // Workers bound to prevent retry explosion
  workers: process.env.CI ? 4 : 2,
});
```

### Mistake 5: Using test.describe.serial Incorrectly with Workers > 1

`test.describe.configure({ mode: 'serial' })` keeps tests in the same describe block on the same worker — but other describe blocks in the same file can still run on different workers. If you intend the entire file to be serial, wrap everything in one describe block.

```typescript
// BAD — two describe blocks, both serial but on different workers possible
test.describe.configure({ mode: 'serial' });
test('one', async () => { /* ... */ });

test.describe.configure({ mode: 'serial' });
test('two', async () => { /* depends on one */ });

// GOOD — single serial describe block
test.describe.configure({ mode: 'serial' });
test('one', async () => { /* ... */ });
test('two', async () => { /* depends on one */ });
```

## Troubleshooting

| Symptom | Likely Cause | Solution |
|---|---|---|
| Test times out in parallel but passes in serial | Shared state collision | Isolate cookies, localStorage, and DB records per test |
| Shard 4/4 reports "No tests found" | Shard count > test file count | Reduce shard count or add more test files |
| Out of memory with N workers | Too many browser processes | Reduce workers to 50-75% of CPU cores |
| Flaky test only fails in parallel | Race condition on shared resource | Use test fixtures, not globals |
| Worker count 0 error | Invalid workers config | Ensure workers >= 1 or use `workers: undefined` for default |

## Production Checklist

- [ ] Worker count set to 50-75% of CPU cores
- [ ] Test isolation verified (no shared cookies, localStorage, or global state)
- [ ] Sharding configured in CI matrix with correct total count
- [ ] Retries adjusted for parallel runs (`maxRetries: 2` or lower)
- [ ] `storageState` used for authenticated sessions to avoid repeated login
- [ ] `fullyParallel: true` explicitly set (not relying on defaults)
- [ ] CI artifact strategy per shard for debugging failures
