# Parallel Execution — Build It

Follow these steps to configure parallel execution in your Playwright test suite.

---

## Step 1: Enable fullyParallel

Open your `playwright.config.ts` and set `fullyParallel: true`. This tells Playwright to run all test files in parallel across available workers. Tests within a single file still run in order by default.

```typescript
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  // Each test file runs in its own worker when a worker is free
});
```

With `fullyParallel: true`, if you have 4 workers and 12 test files, up to 4 files execute at once. When one finishes, the next file starts immediately.

---

## Step 2: Set Worker Count Based on Environment

You almost always want different worker counts locally versus in CI. Use an environment variable to adapt.

```typescript
// playwright.config.ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  // CI gets 4 workers, local gets half your CPU cores
  workers: process.env.CI ? 4 : 2,
});
```

**Rule of thumb for browser tests:** set workers to 50–75% of your CPU cores. Each browser process is heavy (200–500 MB), so oversubscribing causes memory pressure and thrashing.

---

## Step 3: Configure Sharding for CI

Sharding splits your test suite across multiple CI machines. Each machine runs a slice and you sum the results.

Run this on each CI machine:

```bash
# Machine 1 of 4
npx playwright test --shard=1/4

# Machine 2 of 4
npx playwright test --shard=2/4

# Machine 3 of 4
npx playwright test --shard=3/4

# Machine 4 of 4
npx playwright test --shard=4/4
```

### GitHub Actions Matrix Example

```yaml
# .github/workflows/playwright.yml
name: Playwright Tests
on: [push]
jobs:
  test:
    strategy:
      matrix:
        shard: [1/4, 2/4, 3/4, 4/4]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test --shard=${{ matrix.shard }}
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report-${{ matrix.shard }}
          path: playwright-report/
          retention-days: 7
```

Each shard runs independently and you view the combined report via the HTML report aggregator or each shard's artifact separately.

## Step 4: Use Serial Mode for Dependent Tests

Some tests depend on state set by earlier tests (for example, creating a resource and then editing it). Use `test.describe.configure({ mode: 'serial' })` to keep those tests in the same worker and run them in order.

```typescript
// tests/cart.spec.ts
import { test, expect } from '@playwright/test';

test.describe.configure({ mode: 'serial' });

let orderId: string;

test('create order', async ({ page }) => {
  await page.goto('/products');
  await page.click('[data-testid="buy-now"]');
  await page.fill('[name="address"]', '123 Main St');
  await page.click('[data-testid="submit-order"]');
  orderId = await page.locator('[data-testid="order-id"]').textContent();
  expect(orderId).toBeTruthy();
});

test('cancel order', async ({ page }) => {
  await page.goto(`/orders/${orderId}`);
  await page.click('[data-testid="cancel-order"]');
  await expect(page.locator('[data-testid="status"]')).toHaveText('Cancelled');
});
```

When mode is `'serial'`, if the first test fails, Playwright skips the remaining tests in that describe block. This prevents wasted runs on already-broken state.

## Step 5: Use Projects for Cross-Browser Parallelism

Projects let you run the same test files against multiple browsers or device configurations in parallel.

```typescript
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  workers: process.env.CI ? 4 : 2,
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
    },
    {
      name: 'webkit',
      use: { ...devices['Desktop Safari'] },
    },
  ],
});
```

With 4 workers and 3 projects, the workers are shared across all projects. A chromium test and a firefox test could run in the same pool simultaneously.

## Step 6: Build Something Real

Create a production-ready config that combines everything: 3 projects (chromium, firefox, webkit), fullyParallel across test files, each project limited to 2 workers in CI, sharded across 4 CI machines.

```typescript
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  retries: 2,
  workers: process.env.CI ? 2 : 1,
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
    },
    {
      name: 'webkit',
      use: { ...devices['Desktop Safari'] },
    },
  ],
  report: process.env.CI
    ? [['html', { outputFolder: 'playwright-report' }], ['github']]
    : [['html']],
});
```

**The Math Behind This Configuration**

Assume you have 60 test files and each runs against all 3 browsers.

- Total test runs: 60 files x 3 browsers = 180 runs
- CI shards: 4 machines
- Tests per shard: 180 / 4 = 45 tests
- Workers per shard: 2
- Tests running simultaneously per shard: up to 2
- Slots needed per shard: 45 / 2 = 22.5 (rounded up to 23 batches)
- Estimated time: 23 batches x average test time per batch

If each test takes 5 seconds on average, a single shard completes in roughly 23 x 5 = 115 seconds (under 2 minutes). Without parallelism (1 worker, no sharding), 180 tests x 5 seconds = 900 seconds (15 minutes). That is a 7x speedup.

The math in short: 180 tests / 4 shards = 45 tests per shard. 2 workers per shard means up to 2 tests run at once. 45 / 2 = 23 slots per shard. Multiply by average test time.

```yaml
# .github/workflows/playwright-ci.yml
name: Playwright CI
on: [push]
jobs:
  test:
    timeout-minutes: 30
    strategy:
      fail-fast: false
      matrix:
        shard: [1/4, 2/4, 3/4, 4/4]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx playwright install --with-deps
      - name: Run Playwright tests
        run: npx playwright test --shard=${{ matrix.shard }}
        env:
          CI: 'true'
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report-shard-${{ matrix.shard }}
          path: playwright-report/
          retention-days: 7
```

This configuration gives you:
- 3 browsers tested per shard
- 2 concurrent workers per shard (bounded to prevent OOM)
- 4-way sharding for fast CI feedback
- Retries for flaky tests in parallel environments
- Artifact uploads per shard on failure
