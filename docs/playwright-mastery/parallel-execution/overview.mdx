# Parallel Execution — Overview

Difficulty: Advanced

## What You Will Learn

- Configure sharding across CI workers to distribute test load evenly
- Use project-level parallelism for cross-browser and cross-device testing
- Manage test isolation in parallel runs to prevent flaky collisions
- Optimize worker count for your hardware and CI environment

## Prerequisites

- Playwright configuration fundamentals (playwright.config.ts structure)
- CI/CD basics (environment variables, matrix builds)
- Understanding of test fixtures and BrowserContext isolation

## Why This Matters

Parallel execution is how you scale from a 5-minute test suite to a 30-second one. A single worker running 60 tests sequentially wastes CPU cores and CI minutes. But wrong configuration — too many workers, shared state between tests, misconfigured sharding — causes flaky tests that erode trust in your suite. Mastering parallelism means you ship faster without sacrificing reliability.

## Quick Reference

| Config / Flag | Purpose | Example |
|---|---|---|
| `fullyParallel: true` | Run all tests in parallel across files | `fullyParallel: true` |
| `workers: <number>` | Limit concurrent workers | `workers: 4` |
| `projects` | Group tests by browser / device | `projects: [{ name: 'chromium' }, ...]` |
| `--workers=<N>` | CLI override for worker count | `npx playwright test --workers=2` |
| `--shard=<N>/<M>` | Run a slice of the test suite | `npx playwright test --shard=1/4` |
| `--repeat-each=<N>` | Repeat each test N times (stress test) | `--repeat-each=3` |
| `maxFailures: <N>` | Stop after N failures | `maxFailures: 5` |

Each worker spawns its own browser process with an isolated BrowserContext. Tests are dequeued from the queue and assigned to the next available worker. Worker count is capped by the `workers` setting in your config or CLI.
