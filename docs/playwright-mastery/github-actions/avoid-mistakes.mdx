# GitHub Actions — Avoid Mistakes

**Reading time:** 6 minutes · **Difficulty:** Intermediate

## When NOT to Use This

GitHub Actions is an excellent CI platform for Playwright, but it is not always the right choice.

| Use GitHub Actions for | Do NOT use for |
|---|---|
| Small to medium Playwright test suites | Very large monorepos where a self-hosted runner is faster |
| Open-source projects (free for public repos) | Projects with strict compliance requirements (on-premises CI) |
| Standard Linux/macOS/Windows runners | Tests requiring GPU acceleration or specialized hardware |

Do not use the `microsoft/playwright-github-action` marketplace action if you need full control of sharding, caching, or browser versions. The official action bundles its own Playwright version, which can conflict with your project's Playwright dependency. The recommended approach — direct npm commands — gives you full control.

Do not schedule Playwright tests on every commit in a monorepo where only backend code changed. Schedule tests only when frontend or test files actually change, or use path filters in your workflow's `on` triggers.

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'src/**'
      - 'tests/**'
      - 'package.json'
      - 'playwright.config.ts'
```

## The Pattern to Remember

A production-grade Playwright CI pipeline has four essential components:

1. **Cache dependencies and browsers** — Use `actions/cache@v4` with a hash of `package-lock.json` to save 2-5 minutes per run
2. **Shard tests across multiple runners** — Use a matrix strategy with `--shard` to parallelize execution
3. **Upload traces and screenshots on failure** — Always set `if: always()` or `if: failure()` on artifact upload actions
4. **Merge blob reports** — Use `npx playwright merge-reports` to combine results from all shards into a single HTML report

## Common Mistakes

### Mistake 1: Not caching Playwright browsers

Without caching, every CI run re-downloads Chromium, Firefox, and WebKit — roughly 500 MB of browser binaries per run. This adds 3-5 minutes to each execution.

```yaml
# Bad — no caching, installs browsers every time
- name: Install Playwright
  run: npx playwright install --with-deps

# Good — caches browsers based on package-lock.json hash
- name: Cache Playwright browsers
  uses: actions/cache@v4
  id: playwright-cache
  with:
    path: ~/.cache/ms-playwright
    key: ${{ runner.os }}-playwright-${{ hashFiles('**/package-lock.json') }}

- name: Install Playwright browsers
  if: steps.playwright-cache.outputs.cache-hit != 'true'
  run: npx playwright install --with-deps
```

### Mistake 2: Setting shard count higher than the number of test files

If you have 3 test files and set `matrix.shard: [1/4, 2/4, 3/4, 4/4]`, one shard will have 1 test file and three shards will have 0 test files. The empty shards waste CI minutes and slow down your pipeline.

Calculate your shard count based on your test file count and average test duration. A good rule of thumb is 10-20 test files per shard, or target 2-5 minutes of execution time per shard.

### Mistake 3: Not setting `retention-days` on artifacts

GitHub's default artifact retention is 90 days. If your CI runs 50 times per day, each run uploading a 10 MB report, that is 45 GB of storage per quarter — which counts toward your GitHub Actions storage limit.

```yaml
# Bad — uses default 90-day retention
- uses: actions/upload-artifact@v4
  with:
    name: playwright-report
    path: playwright-report/

# Good — explicit retention period
- uses: actions/upload-artifact@v4
  if: always()
  with:
    name: playwright-report
    path: playwright-report/
    retention-days: 30
```

### Mistake 4: Forgetting `--with-deps` in CI

The `npx playwright install --with-deps` flag installs both the browser binaries and the system libraries they depend on (libnss3, libnspr4, libatk-bridge, and dozens more). Running `npx playwright install` without `--with-deps` in a fresh CI environment causes failures like:

```
browserType.launch: Host system missing dependencies!
```

Always use `--with-deps` in CI:

```bash
npx playwright install --with-deps
```

If you use the `mcr.microsoft.com/playwright` Docker image, system dependencies are pre-installed and you only need `npx playwright install` (without `--with-deps`).

### Mistake 5: Not using blob reports for sharded results

When you shard tests across multiple runners without blob reports, each runner generates its own HTML report. The final `report` job only has access to the last uploaded report, meaning you lose all trace data, screenshots, and failure details from the other shards.

```yaml
# Bad — each shard generates its own HTML report, only the last one survives
reporter: [
  ['html', { open: 'never' }],
]

# Good — each shard generates a blob, merge step combines all blobs
reporter: [
  ['blob'],
  ['html', { open: 'never' }],
]
```

Then merge them in your report job:

```bash
npx playwright merge-reports --reporter html ./all-blob-reports
```

## Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| `browser not found in CI` | Browsers not installed or cached before test run | Ensure `npx playwright install --with-deps` runs before `npx playwright test` |
| `disk space exhausted during tests` | Traces and videos accumulate in long runs | Set `trace: 'on-first-retry'` and `video: 'retain-on-failure'` instead of `'on'` |
| `timed out waiting for browser to launch` | Missing system libraries | Use `--with-deps` flag, or switch to the `mcr.microsoft.com/playwright` Docker image |
| `artifact upload failed` | Free tier limit is 10 GB total; you exceeded it | Reduce retention-days, delete old artifacts, or upgrade to a paid plan |
| `shard 4/4 reports 0 tests` | Shard count exceeds available test files | Reduce shard count or add more test files |

## Production Checklist

Before considering your CI pipeline production-ready, verify these items:

- [ ] Node and OS versions pinned in the workflow (not using `latest` tags)
- [ ] Cache configured for both npm dependencies and Playwright browsers
- [ ] `npx playwright install --with-deps` used in every test job
- [ ] Blob reporter configured in `playwright.config.ts`
- [ ] Artifact retention-days set to a reasonable value (14-30 days)
- [ ] Secrets (not env vars) used for credentials and API keys
- [ ] `fail-fast: false` set in matrix strategy
- [ ] Report merge job with `if: always()` so results are reported even on failure
- [ ] Path filters or conditional triggers to skip irrelevant commits
