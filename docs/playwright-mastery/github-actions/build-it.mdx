# GitHub Actions — Build It

**Reading time:** 12 minutes · **Difficulty:** Intermediate

## Step by Step

### 1. Create the workflow directory

GitHub Actions looks for workflow files in the `.github/workflows/` directory at the root of your repository. Create this directory:

```bash
mkdir -p .github/workflows
```

### 2. Understand the workflow anatomy

Every GitHub Actions workflow is a YAML file with these sections:

- **name**: A human-readable label for the workflow
- **on**: The events that trigger the workflow (push, pull_request, schedule, workflow_dispatch)
- **jobs**: One or more jobs that run in parallel or sequentially
- **steps**: Individual commands within a job (checkout, install, test, upload)

### 3. Write a basic Playwright workflow

Create `.github/workflows/playwright.yml` with a simple single-job workflow that runs Playwright tests on Chromium:

```yaml
name: Playwright Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Install dependencies
        run: npm ci
      - name: Install Playwright browsers
        run: npx playwright install --with-deps chromium
      - name: Run Playwright tests
        run: npx playwright test --project=chromium
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
```

This workflow triggers on every push and pull request to `main`. It checks out the code, installs dependencies, installs Chromium with its system dependencies, runs tests against Chromium only, and uploads the HTML report as an artifact — even if tests fail (note `if: always()`).

### 4. Add caching

Without caching, every CI run re-downloads all npm packages and Playwright browsers, which adds 2-5 minutes per run. GitHub Actions supports caching via `actions/cache@v4`:

```yaml
name: Playwright Tests with Caching

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Cache npm dependencies
        uses: actions/cache@v4
        id: npm-cache
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-node-
      - name: Cache Playwright browsers
        uses: actions/cache@v4
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: ${{ runner.os }}-playwright-${{ hashFiles('**/package-lock.json') }}
      - name: Install dependencies
        run: npm ci
      - name: Install Playwright browsers
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: npx playwright install --with-deps
      - name: Run Playwright tests
        run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
```

The `actions/cache` step creates a cache key based on the operating system and the hash of `package-lock.json`. If the hash matches a previous run, the cache is restored and the `npm ci` and `npx playwright install` steps use the cached files instead of downloading fresh ones. The `cache-hit` output lets you skip browser installation when the cache is intact.

### 5. Add matrix sharding

For larger test suites, running everything sequentially on one runner is too slow. Matrix sharding splits your tests across multiple parallel runners. Each runner gets a `shard` identifier and runs only its portion of the test suite.

First, configure blob reporting in `playwright.config.ts` so sharded results can be merged:

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 1 : undefined,
  reporter: [
    ['blob'],
    ['html', { open: 'never' }],
  ],
  use: {
    baseURL: process.env.BASE_URL || 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
  },
  projects: [
    {
      name: 'chromium',
      use: { browserName: 'chromium' },
    },
    {
      name: 'firefox',
      use: { browserName: 'firefox' },
    },
    {
      name: 'webkit',
      use: { browserName: 'webkit' },
    },
  ],
});
```

The `blob` reporter generates a binary blob report file in `blob-report/` that contains all test results, traces, screenshots, and metadata. This blob can later be merged with blobs from other shards.

### 6. Add schedule trigger

To run tests on a schedule — for example, every Monday at 6 AM to catch regressions introduced by upstream dependency changes — add a `schedule` event:

```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'
  workflow_dispatch:
```

The cron syntax follows standard Unix cron: minute, hour, day of month, month, day of week. `0 6 * * 1` means "at 06:00 UTC every Monday."

## Build Something Real

Now combine everything into a production-grade CI pipeline. This workflow:

- Runs on push to `main`, pull requests to `main`, and a scheduled Monday run
- Spreads tests across 4 shards and 3 browser projects (12 total jobs)
- Caches npm packages and Playwright browsers
- Uses blob reports to merge results from all shards
- Has a final `report` job that merges all blob reports into a single HTML report
- Uploads the merged HTML report with 30-day retention

Create `.github/workflows/playwright.yml` with the following complete YAML:

```yaml
name: Playwright Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'
  workflow_dispatch:

jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        shard: [1/4, 2/4, 3/4, 4/4]
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Cache npm dependencies
        uses: actions/cache@v4
        id: npm-cache
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-node-

      - name: Cache Playwright browsers
        uses: actions/cache@v4
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: ${{ runner.os }}-playwright-${{ hashFiles('**/package-lock.json') }}

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright browsers
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: npx playwright install --with-deps

      - name: Run Playwright tests
        run: npx playwright test --shard=${{ matrix.shard }}

      - name: Upload blob report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: blob-report-${{ matrix.shard }}
          path: blob-report/
          retention-days: 7
          if-no-files-found: ignore

  report:
    needs: [test]
    if: always()
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Download all blob reports
        uses: actions/download-artifact@v4
        with:
          pattern: blob-report-*
          merge-multiple: true
          path: all-blob-reports

      - name: Merge reports
        run: npx playwright merge-reports --reporter html ./all-blob-reports

      - name: Upload merged HTML report
        uses: actions/upload-artifact@v4
        with:
          name: playwright-html-report
          path: playwright-report/
          retention-days: 30

      - name: Upload test results on failure
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: test-results/
          retention-days: 14
```

Key design decisions in this workflow:

- **`fail-fast: false`** — If one shard fails, other shards continue running. This gives you the full picture of all failures across all browsers and shards.
- **`if: always()` on the report job** — The report job runs even if some test shards failed, so you always get a merged report.
- **Blob reports uploaded per shard** — Each shard uploads its blob separately. The report job downloads all of them using the `pattern` filter and `merge-multiple: true`.
- **Separate retention for blobs (7 days) vs final report (30 days)** — Blob reports are intermediate artifacts; you only need the final HTML report for long-term storage.
- **`if-no-files-found: ignore`** — Prevents the workflow from failing if a shard produced no blob report (e.g., all tests were skipped).

### Configure your Playwright config

Make sure your `playwright.config.ts` includes the `blob` reporter for sharded runs:

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 1 : undefined,
  reporter: [
    ['blob'],
    ['html', { open: 'never' }],
  ],
  use: {
    baseURL: process.env.BASE_URL || 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
  projects: [
    { name: 'chromium', use: { browserName: 'chromium' } },
    { name: 'firefox', use: { browserName: 'firefox' } },
    { name: 'webkit', use: { browserName: 'webkit' } },
  ],
});
```

### Verify the workflow

Commit and push the workflow file to your repository:

```bash
git add .github/workflows/playwright.yml playwright.config.ts
git commit -m "Add Playwright CI pipeline with matrix sharding and caching"
git push origin main
```

Navigate to the Actions tab in your GitHub repository. You should see the workflow running with 4 shard jobs in the `test` matrix and a final `report` job.
