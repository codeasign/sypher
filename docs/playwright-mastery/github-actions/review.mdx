# GitHub Actions — Review

**Reading time:** 5 minutes · **Difficulty:** Intermediate

## Key Takeaways

1. GitHub Actions workflows are YAML files in `.github/workflows/` that trigger on events like `push`, `pull_request`, `schedule`, and `workflow_dispatch`.
2. Matrix sharding with `--shard=${{ matrix.shard }}` splits your test suite across parallel runners, reducing total execution time proportionally to the number of shards.
3. Cache both npm dependencies (`~/.npm`) and Playwright browsers (`~/.cache/ms-playwright`) to save 3-5 minutes per CI run by avoiding re-downloads.
4. Always use blob reports when sharding: each shard produces a blob, and a final merge job combines them into a single HTML report with all traces, screenshots, and failure data intact.
5. Set `retention-days` on artifacts, use `--with-deps` for browser installation, and set `fail-fast: false` so you see the full picture of all failures across all shards.

## Recap in 60 Seconds

1. Create `.github/workflows/playwright.yml` with `on: [push, pull_request]`.
2. Add a matrix strategy with 4 shards and a `report` merge job.
3. Cache npm and Playwright browsers. Install with `--with-deps`.
4. Use blob reporter in config, merge in CI with `npx playwright merge-reports`.
5. Always upload artifacts with explicit `retention-days`.

### Scenario Questions

**Scenario 1:** Your Playwright CI pipeline takes 12 minutes to run on a single runner. You have 120 test files spread across 3 browser projects. The team wants the pipeline to finish in under 3 minutes. The tests are independent and can run in any order. How would you configure the matrix strategy?

**Scenario 2:** A colleague set up a GitHub Actions workflow that runs `npx playwright test` without blob reports. When a test fails on shard 3 of 4, the team cannot see the trace or screenshot for that failure in the final report. What is the root cause and how do you fix it?

### Answers

**Multiple Choice:** 1-B, 2-B, 3-C, 4-B, 5-B

**Scenario 1:** Configure a matrix with shards and browser projects. For example, 4 shards across 3 browsers gives 12 parallel jobs. Each job runs `npx playwright test --shard=${{ matrix.shard }}` against its assigned browser project. With 120 files across 12 jobs, each job runs roughly 10 test files. If each file takes 15 seconds, a shard finishes in about 2.5 minutes. Configure blob reports and a merge step to combine results. Adjust the shard count based on actual test durations.

**Scenario 2:** Without blob reports, each shard generates its own HTML report in `playwright-report/`. When the CI jobs finish, only the last shard's report is uploaded or accessible — the traces and screenshots from shards 1, 2, and 3 are lost. The fix is to configure `['blob']` as one of the reporters in `playwright.config.ts`, upload each shard's blob report as a separate artifact, and add a merge job that runs `npx playwright merge-reports --reporter html ./all-blob-reports` to combine all blobs into a single complete report.
