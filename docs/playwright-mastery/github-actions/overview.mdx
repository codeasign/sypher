# GitHub Actions

**Reading time:** 6 minutes · **Difficulty:** Intermediate

## What You Will Learn

- Set up a Playwright GitHub Actions workflow that runs on every push and pull request
- Configure matrix builds to run tests across multiple browsers simultaneously
- Integrate sharding across CI runners to cut total execution time proportionally
- Cache npm dependencies and Playwright browsers to avoid re-downloading on every run

## Prerequisites

This lesson assumes you have a Playwright project with at least a few tests, basic Git and GitHub knowledge (commits, pushes, pull requests), and a GitHub account. No prior CI/CD experience is required, but familiarity with YAML syntax is helpful.

## Why This Matters

A test suite that only runs locally is not a test suite — it is a collection of scripts you hope work. CI is where tests earn their keep: every push, every pull request, every merge should automatically validate that nothing broke.

But running Playwright tests in CI is harder than running them locally. You need browsers installed, system dependencies available, enough parallelism to keep the pipeline fast, and a way to collect test artifacts (traces, screenshots, reports) when something fails. Misconfigured CI gives false confidence: tests that pass because they never actually ran, or results that are missing because artifacts were not uploaded.

GitHub Actions solves this with a mature ecosystem of actions, matrix strategies, and artifact management. When you wire it up correctly, your CI pipeline becomes a reliable gatekeeper that blocks bad code before it reaches production.

## Quick Reference

| Concept | Details |
|---|---|
| Trigger events | `push`, `pull_request`, `schedule` (cron), `workflow_dispatch` (manual) |
| Playwright GH Action | `microsoft/playwright-github-action@v1` — optional, official approach uses direct npm commands |
| Matrix strategy keys | `matrix.browser` (chromium/firefox/webkit), `matrix.shard` (1/4, 2/4, ...) |
| Caching strategies | npm cache (`~/.npm`), Playwright browsers (`~/.cache/ms-playwright`) |
| Artifact upload actions | `actions/upload-artifact@v4` with `retention-days` parameter |
| Blob report merge | `npx playwright merge-reports --reporter html ./all-blob-reports` |
| Report retention | Default 90 days; set `retention-days: 30` to reduce storage costs |

The key insight is the **matrix strategy**: instead of running all tests on a single runner (which takes as long as the slowest browser), you split tests across browsers AND shards. With 3 browsers and 4 shards, you get 12 parallel jobs that finish in approximately 1/4 of the sequential time.
