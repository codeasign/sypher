# Avoid Common Mistakes: Playwright MCP Server

## When NOT to Use This

Do not use the Playwright MCP server for headless CI runs where no AI assistant is involved. MCP is designed for AI-assisted development, exploration, debugging, and rapid test creation. Running it in a production CI pipeline adds unnecessary complexity — use Playwright's standard `playwright test` command instead.

Also avoid using MCP when you need fine-grained control over Playwright's internal behavior, such as custom network interception policies or bespoke browser context configurations. In those cases, writing Playwright code directly gives you more control.

## The Pattern to Remember

MCP bridges natural language to browser automation. Use it when you want to explore a page, debug a failing interaction, or rapidly prototype a test flow. The pattern is always: you describe what you want, the AI translates it to Playwright commands, and the MCP server executes them in the browser.

## Common Mistakes

**1. MCP server not running when the AI tool attempts to connect**

The AI assistant cannot call any browser tools if the MCP server is not running. Always start the MCP server in a separate terminal window before launching your AI tool. Verify it is listening by checking for the "listening on port" log message.

**2. Not understanding that the MCP browser context is separate from Playwright test context**

The MCP server opens its own browser context. If you run a Playwright test in another terminal, the MCP session and the test session do not share state, cookies, or localStorage. Do not expect login state from one to carry over to the other.

**3. Relying on MCP for production test execution**

MCP is a development-time tool. It adds latency (the AI round trip), relies on a network connection to the AI assistant, and introduces non-determinism from LLM output variability. Use it for prototyping, exploration, and debugging. Run your finalized tests with `npx playwright test` in CI.

**4. Forgetting to close browser sessions between MCP interactions**

Each MCP session creates a browser context that holds memory and processes. If you run many sessions without cleanup, you may exhaust system resources. Restart the MCP server periodically or close idle browser windows. Some MCP configurations auto-close on disconnect, but you should not rely on this.

**5. MCP server version mismatch with Playwright browser versions**

The Playwright MCP server depends on Playwright under the hood. If your MCP server version and your Playwright browser versions drift apart, you may see cryptic errors or silent failures. Pin the MCP server version and keep it in sync with the Playwright version you use in your project.

## Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| MCP server fails to start | Port 3100 already in use | Kill the existing process or start the server on a different port using `--port` |
| Browser does not launch from MCP | Playwright browsers not installed | Run `npx playwright install` to download browser binaries |
| Snapshot returns an empty page | No page has been navigated to yet | Call `navigate(url)` before `snapshot()` |
| Click action does nothing | Selector is wrong or element is not visible | Use `snapshot()` first to see the actual DOM structure, then craft a more specific selector |
| MCP connection timeout | AI tool cannot reach the MCP server | Verify the server is running and the port is correct; check firewall settings |

## Production Checklist

- [ ] MCP server version is pinned in your project configuration
- [ ] Playwright browsers are installed and the path is configured explicitly
- [ ] Error handling is in place for session timeouts and disconnections
- [ ] No CI pipeline depends on the MCP server — use it only in development
