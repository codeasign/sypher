# Advanced Practice Exercises

## Exercise 1: CI Pipeline Design

Design and implement a complete production CI pipeline for a Playwright test suite that runs in GitHub Actions.

### Problem

You are responsible for the test infrastructure of a large web application. The team needs a CI pipeline with the following requirements:

- **4 browser projects**: Chromium, Firefox, WebKit, and a mobile Chrome (Pixel 5 emulation)
- **Sharding**: The Chromium project must be sharded across 4 parallel runners for speed
- **Blob reports**: All shard results must be merged into a single blob report for unified reporting
- **Traces**: Traces must be captured on failure only (not for passing tests)
- **HTML report**: The merged HTML report must be uploaded as a CI artifact
- **Caching**: Playwright browsers and dependencies must be cached to speed up install
- **Scheduled run**: A nightly run at 2 AM that includes all browsers with full trace capture
- **PR checks**: On pull request, only run Chromium (fast feedback) without traces
- **Conditional**: The full multi-browser suite runs only on pushes to `main` and the scheduled trigger

### Deliverables

Provide the following:

1. **`playwright.config.ts`** — Full config with 4 projects, sharding config, trace mode set to `on-first-retry`, blob reporter configured
2. **`.github/workflows/playwright.yml`** — A single workflow file with:
   - PR trigger (only Chromium, no traces)
   - Push to main trigger (all 4 projects, chromium sharded 4x, blob reports, upload artifacts)
   - Scheduled trigger at 2 AM daily (all browsers, traces on all failures)
   - Caching step for Playwright browsers and npm dependencies
   - A matrix strategy for Chromium shards
   - A merge step that collects all blob reports and generates the HTML report
3. **`package.json` scripts** — Add `test:chromium`, `test:all`, `test:ci`, `merge-reports` scripts
4. **Sharding equation**: Explain how many total containers are used, how tests are distributed, and the expected time savings

## Exercise 2: Cross-Role Auth System

Build a comprehensive test suite for a multi-role application using Playwright's authentication system.

### Problem

A web application has three user roles, each with different access levels:

| Role | Can Access | Cannot Access |
|------|-----------|---------------|
| Admin | `/admin`, `/dashboard`, `/profile`, `/settings` | — |
| User | `/dashboard`, `/profile` | `/admin`, `/settings` |
| Viewer | `/dashboard` | `/admin`, `/profile`, `/settings` |

You need to build a test suite that:

1. **Setup projects** — Create three setup projects (one per role) that:
   - Log in via the API using `page.request.post()`
   - Save `storageState` to a unique file per role
   - Run before the main tests

2. **Login flows** — Write tests that verify each role can log in successfully and reach their default landing page

3. **Access verification** — For each role, verify:
   - Pages they should access load correctly (HTTP 200, expected content visible)
   - Pages they should NOT access either return HTTP 403 or redirect to the login page

4. **Session expiry** — Write a test that:
   - Clears the storage state (simulating session expiry)
   - Attempts to access a protected page
   - Verifies the user is redirected to login

### Deliverables

Provide the following:

1. **`.github/workflows/playwright.yml`** or **`playwright.config.ts`** — Config with 3 setup projects and 3 test projects that depend on them
2. **`auth.setup.ts`** — A single setup file parameterized by role that logs in and saves storage state
3. **`tests/admin.spec.ts`** — Admin tests
4. **`tests/user.spec.ts`** — User tests
5. **`tests/viewer.spec.ts`** — Viewer tests
6. **`tests/session.spec.ts`** — Session expiry test
7. **`helpers/mock-server.ts`** or inline HTML — A simulated app with routes that enforce role-based access, created via `page.route()` or `page.setContent()` with client-side routing

The entire suite must be self-contained — no external server needed. Use `page.route()` to intercept network requests and simulate role-based access control.

### Starter Structure

```
tests/
  auth.setup.ts
  admin.spec.ts
  user.spec.ts
  viewer.spec.ts
  session.spec.ts
helpers/
  mock-server.ts
playwright.config.ts
```
