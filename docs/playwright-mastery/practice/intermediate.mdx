# Intermediate Practice Exercises

## Exercise 1: Combine API and UI

Bridge the gap between API testing and UI testing by seeding data through API calls and verifying it renders correctly in the browser.

### Problem

Use `https://jsonplaceholder.typicode.com` — a free fake REST API — to:

1. **Seed data via API**: Make a POST request to `/posts` to create a new blog post. Capture the returned `id` from the response.
2. **Verify via API**: Make a GET request to `/posts/{id}` to confirm the post was created with the correct title.
3. **UI verification**: Navigate to a custom HTML page (created via `page.setContent()`) that fetches and renders posts from the API. Simulate this with a pre-populated table that includes the post you created, then verify the table contains your post's title.

### Requirements

- Use `page.request.post()` and `page.request.get()` for API calls
- Extract the response `id` and use it in subsequent steps
- Create a table via `page.setContent()` that contains the post data
- Use `getByRole('row')` or `getByText()` to assert the data appears in the table
- Clean up by deleting the post via `page.request.delete()` after the test

### Starter Code

```typescript
import { test, expect } from "@playwright/test";

test("API seed then UI verify", async ({ page }) => {
  const BASE = "https://jsonplaceholder.typicode.com";

  // 1. POST to create a post
  // 2. GET to verify the post
  // 3. Create HTML table with the data and verify in UI
  // 4. DELETE to clean up
});
```

## Exercise 2: POM with Fixtures

Apply the Page Object Model pattern to a multi-page checkout flow, using Playwright fixtures for dependency injection.

### Problem

Build a checkout flow test using three page objects:

1. **CartPage** — shows cart items, has a "Checkout" button
2. **CheckoutPage** — has shipping form fields (name, address, city, zip) and a "Continue to Payment" button
3. **PaymentPage** — has card number, expiry, CVV fields and a "Place Order" button with a success confirmation message

### Requirements

- Create each page object as a class with typed locators and methods
- Use a custom fixture that composes all three page objects into a single `checkout` fixture
- The fixture should navigate to the cart page on creation
- Tests use the fixture directly without manually instantiating page objects
- Use `page.setContent()` to create the HTML for each page so the test is self-contained
- Test the full flow: cart -> checkout -> payment -> success

### Starter Code

**pages/cart-page.ts:**
```typescript
import { type Page, type Locator } from "@playwright/test";

export class CartPage {
  readonly page: Page;
  readonly checkoutButton: Locator;

  constructor(page: Page) {
    this.page = page;
    this.checkoutButton = page.getByRole("button", { name: "Checkout" });
  }

  async proceedToCheckout() {
    // Click checkout and wait for navigation
  }
}
```

**pages/checkout-page.ts** and **pages/payment-page.ts** follow the same pattern.

**fixtures.ts:**
```typescript
import { test as base } from "@playwright/test";
import { CartPage } from "./pages/cart-page";
import { CheckoutPage } from "./pages/checkout-page";
import { PaymentPage } from "./pages/payment-page";

type CheckoutFixtures = {
  cartPage: CartPage;
  checkoutPage: CheckoutPage;
  paymentPage: PaymentPage;
  checkout: {
    goToCart: () => Promise<CartPage>;
    checkoutPage: CheckoutPage;
    paymentPage: PaymentPage;
    completePurchase: (shipping: {}, payment: {}) => Promise<string>;
  };
};

export const test = base.extend<CheckoutFixtures>({ ... });
```

## Exercise 3: Visual and Trace

Combine visual regression testing with trace viewer analysis to simulate a real debugging workflow.

### Problem

1. **Baseline**: Write a test that navigates to a styled HTML page (created via `page.setContent()`) with a heading, a button, and a colored box. Take a visual snapshot and save it as the baseline.
2. **Break**: Modify the HTML so the colored box has a different background color and the button text changes.
3. **Detect**: Re-run the test. The visual diff should fail. Configure the test to capture a trace on failure.
4. **Analyze**: Use the trace viewer to inspect the before/after DOM, network requests, and console output. Document what the trace reveals about the failure.

### Requirements

- Use `expect(page).toHaveScreenshot()` for visual comparison
- Configure `playwright.config.ts` to enable traces `on-first-retry` or `on-all-retries`
- Set `snapshotPathTemplate` in config to organize screenshots
- Use `test.describe` with `test.fail` or `test.only` to isolate the failing test
- After running, instruct how to view the trace: `npx playwright show-trace test-results/.../trace.zip`
- Manually inspect the trace diff viewer to spot the visual difference
- Write a brief summary of what the trace revealed

### HTML Template for the Test Page

```html
<div style="padding: 2rem; font-family: sans-serif;">
  <h1>Visual Test Page</h1>
  <div id="box" style="width: 200px; height: 200px; background-color: blue;"></div>
  <button id="action-btn" style="margin-top: 1rem; padding: 0.5rem 1rem;">Submit</button>
</div>
```

For the broken version, change `background-color: blue` to `background-color: red` and "Submit" to "Send".
