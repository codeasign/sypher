# Beginner Practice Exercises

## Exercise 1: Your First Playwright Test

Write your very first Playwright test from scratch. This exercise walks through the complete setup and execution of a basic test.

### Problem

Create a Playwright test that:
1. Launches a Chromium browser
2. Navigates to `https://playwright.dev`
3. Takes a screenshot of the homepage
4. Asserts the page URL contains "playwright"
5. Closes the browser

### Requirements

- Use the Playwright test runner (not raw `playwright` library calls)
- Save the screenshot as `homepage.png` in a `screenshots/` directory
- Use the built-in `page` fixture from `@playwright/test`
- Assert on both the URL and the page title

### Starter Code

```typescript
import { test, expect } from "@playwright/test";

test("Playwright dev homepage", async ({ page }) => {
  // Your code here
});
```

### Hints

- Use `page.goto()` to navigate
- `page.screenshot()` accepts a `{ path }` option
- `expect(page).toHaveURL()` tests URL patterns
- `expect(page).toHaveTitle()` tests the page title
- The `./screenshots/` directory must exist or be created

## Exercise 2: Use Locators

Practice finding elements using all five major locator strategies.

### Problem

On `https://playwright.dev`, locate and interact with five different elements, each using a different locator strategy:

1. **By role**: Find the "Get started" link using `getByRole` and click it
2. **By label**: Find the search input using `getByLabel` (the magnifying glass icon opens search), type "locators", and press Enter
3. **By test ID**: Find an element using `getByTestId` — inspect the page to find an element with a `data-testid` attribute, or add one yourself via `page.setContent()`
4. **By text**: Find a heading or paragraph using `getByText` that contains the word "Playwright" and verify its text
5. **By CSS**: Find all elements matching a CSS selector (e.g., all navigation links in the top navbar) and count them

### Requirements

- Each interaction must use the specified locator strategy
- After each interaction, add a small assertion to confirm the expected state
- Use `page.setContent()` for test ID practice if the live page lacks `data-testid` attributes
- Handle any waits implicitly — Playwright auto-waits

### Starter Code

```typescript
import { test, expect } from "@playwright/test";

test("five locator strategies", async ({ page }) => {
  // 1. Role locator
  // 2. Label locator
  // 3. Test ID locator
  // 4. Text locator
  // 5. CSS locator
});
```

## Exercise 3: Form Test

Write a test that fills out a form, submits it, and validates both success and error states.

### Problem

On a form page (you will create one using `page.setContent()` or use a real form such as `https://demo.playwright.dev/todomvc`), write tests that:

1. Fill input fields (name, email, message) with valid data and submit
2. Assert a success message is visible after submission
3. Leave the name field empty and submit again
4. Assert an error message appears for the required name field

### Requirements

- Create the form inline using `page.setContent()` with HTML that includes:
  - A form with fields: name (required), email, message (textarea)
  - A submit button
  - A success message div (hidden initially, shown after valid submit)
  - Error message spans next to each required field
- Use `page.getByLabel()` to locate form fields
- Use `page.getByRole('button')` to find and click the submit button
- Use `page.getByText()` to locate success and error messages
- Test both happy path and validation path in the same test or separate tests

### Starter Code

```typescript
import { test, expect } from "@playwright/test";

test.describe("Form tests", () => {
  test.beforeEach(async ({ page }) => {
    await page.setContent(`
      <!-- Your form HTML here -->
    `);
  });

  test("successful form submission", async ({ page }) => {
    // Fill and submit
  });

  test("empty name shows error", async ({ page }) => {
    // Submit without filling name
  });
});
```
