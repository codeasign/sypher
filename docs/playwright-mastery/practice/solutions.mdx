# Practice Solutions

## Beginner Exercise 1: Your First Playwright Test

### Problem Restated

Create a Playwright test that navigates to `https://playwright.dev`, takes a screenshot, and asserts the URL and title.

### Key Insight

Playwright's test runner provides built-in fixtures (`page`, `browser`, `context`) that handle browser lifecycle automatically. You never need to manually call `browser.newPage()` or `browser.close()` in test runner mode.

### Thinking Process

1. Start with the `test` and `expect` imports from `@playwright/test`
2. Use the `page` fixture — it creates a new browser context and page for each test
3. Navigate with `goto` which auto-waits for the page to load
4. Take a screenshot to a relative path — Playwright creates the file relative to the project root
5. Assert the URL contains "playwright" using a regex or string
6. Assert the title so you know the page rendered correctly

### Complete Code

```typescript
import { test, expect } from "@playwright/test";
import * as fs from "fs";
import * as path from "path";

test("Playwright dev homepage", async ({ page }) => {
  // Ensure the screenshots directory exists before taking a screenshot
  const screenshotDir = path.join(__dirname, "..", "screenshots");
  if (!fs.existsSync(screenshotDir)) {
    fs.mkdirSync(screenshotDir, { recursive: true });
  }

  // Navigate to the Playwright website.
  // page.goto() auto-waits until the 'load' event fires.
  await page.goto("https://playwright.dev");

  // Take a full-page screenshot and save it to the screenshots directory.
  // The fullPage: true option captures the entire scrollable page.
  await page.screenshot({
    path: path.join(screenshotDir, "homepage.png"),
    fullPage: true,
  });

  // Assert the current URL contains "playwright".
  // toHaveURL accepts a string (exact match), regex, or string pattern.
  await expect(page).toHaveURL(/.*playwright.*/);

  // Assert the page title is what you expect.
  // This confirms the page loaded and rendered its <title> tag.
  await expect(page).toHaveTitle(/Playwright/);
});
```

### Why This Works

- The `page` fixture is automatically created and torn down by Playwright. Each test gets a clean isolated page in its own browser context.
- `page.goto()` waits for the `load` event, so the page is fully rendered before you take the screenshot.
- `page.screenshot()` captures the visual state at the moment it is called, which is right after navigation completes.
- The `expect` auto-retries assertions for a configurable timeout (default 5 seconds), making the test resilient to slow page loads.

### Edge Cases and Gotchas

- If `screenshots/` does not exist, `page.screenshot()` throws an error. Either create it beforehand or use the `fullPage: true` option and handle the directory creation in a `beforeAll` hook.
- Screenshots taken in headless mode may differ visually from headed mode due to font rendering differences.
- On CI, screenshots should be uploaded as artifacts if you want to inspect them later.
- The `toHaveURL` matcher with a regex like `/.*playwright.*/` will pass for any URL containing "playwright" — including query parameters or hash fragments. Use an exact string or a more precise regex if you need an exact match.

### Alternative Approaches

- Use `test.beforeAll` to create the screenshots directory once for the entire test file, instead of creating it inside each test.
- Use Playwright's built-in `screenshot` fixture via `testInfo.snapshotSuffix` if you plan to compare screenshots against baselines.
- Combine the URL and title assertions into a single `expect` chain if you prefer a more fluent style.

## Beginner Exercise 2: Use Locators

### Problem Restated

Find and interact with five elements on `https://playwright.dev` using five different locator strategies: role, label, test ID, text, and CSS.

### Key Insight

Each locator strategy serves a specific purpose. `getByRole` is the most accessible and preferred approach. `getByTestId` is the most resilient for elements without semantic roles. CSS selectors are the most flexible but most brittle.

### Thinking Process

1. Start on the Playwright homepage.
2. For role: the "Get started" link is the most prominent semantic element.
3. For label: the search input on Playwright.dev is accessible via `getByLabel` once you open the search overlay.
4. For test ID: since the live page may not have `data-testid` attributes, use `page.setContent` to inject a test element.
5. For text: headings and paragraphs are easy targets for `getByText`.
6. For CSS: navbar links are a common pattern that works well with CSS.

### Complete Code

```typescript
import { test, expect } from "@playwright/test";

test("five locator strategies", async ({ page }) => {
  await page.goto("https://playwright.dev");

  // 1. Role locator — find the "Get started" link by its accessible role and name.
  // getByRole('link', { name: ... }) matches the ARIA role and accessible name.
  const getStartedLink = page.getByRole("link", { name: "Get started" });
  await expect(getStartedLink).toBeVisible();
  // Click the link and wait for navigation to complete.
  await getStartedLink.click();
  await expect(page).toHaveURL(/.*docs.*/);

  // Navigate back to the homepage for the next locator exercise.
  await page.goBack();
  await expect(page).toHaveURL("https://playwright.dev/");

  // 2. Label locator — open the search dialog and find the search input by label.
  // The search button (magnifying glass) opens an overlay with a labeled input.
  const searchButton = page.getByRole("button", { name: "Search" });
  await searchButton.click();

  // The search input has an accessible label — use getByLabel to locate it.
  const searchInput = page.getByLabel("Search");
  await expect(searchInput).toBeVisible();
  // Type a query and press Enter to search.
  await searchInput.fill("locators");
  await searchInput.press("Enter");
  // Wait for search results page to load.
  await expect(page).toHaveURL(/.*locators.*/);

  // Navigate back again for the next locator exercise.
  await page.goBack();

  // 3. Test ID locator — inject a custom element with a data-testid attribute.
  // This simulates how many SPAs attach data-testid for testing.
  await page.setContent(`
    <div>
      <h1>Test ID Demo</h1>
      <button data-testid="submit-btn">Submit</button>
    </div>
  `);

  // getByTestId locates the element by the testid attribute.
  const submitButton = page.getByTestId("submit-btn");
  await expect(submitButton).toBeVisible();
  await expect(submitButton).toHaveText("Submit");

  // 4. Text locator — find an element by its visible text content.
  // Navigate back to the Playwright homepage first.
  await page.goto("https://playwright.dev");
  const heading = page.getByText("Playwright enables reliable");
  await expect(heading).toBeVisible();

  // 5. CSS locator — count all navigation links in the top navbar.
  // CSS selectors let you use any valid CSS selector syntax.
  const navLinks = page.locator("nav a");
  const count = await navLinks.count();
  // Expect at least 3 navigation links in the navbar.
  expect(count).toBeGreaterThanOrEqual(3);
});
```

### Why This Works

- Each locator uses the most appropriate strategy for the element type: semantic elements get `getByRole`, form inputs get `getByLabel`, test hooks get `getByTestId`, visible text gets `getByText`, and structural queries get CSS.
- `getByRole` aligns with accessibility best practices — elements are found the same way screen readers do.
- Using `page.setContent()` for test ID practice makes the exercise self-contained and not dependent on the live page having test IDs.

### Edge Cases and Gotchas

- On Playwright.dev, the search dialog may not be immediately available. Add a short wait before clicking the search button if the button is animated.
- `getByText` matches text in child elements as well, so a parent element might match if its text includes the search string. Use `{ exact: true }` for exact matching.
- CSS locators are the most brittle — a class rename or structural change in the navbar will break the test.
- The search input may have a `placeholder` attribute instead of a label. Playwright's `getByLabel` does not match placeholder text — use `getByPlaceholder` for that.

### Alternative Approaches

- Use `locator('role=link[name="Get started"]')` syntax for an inline role-based selector.
- Use `page.locator('nav >> a')` for a chained CSS selector.
- For counting elements, use `await page.locator('nav a').all()` and check the array length instead of `count()`.

## Beginner Exercise 3: Form Test

### Problem Restated

Create a self-contained form with name, email, and message fields. Test valid submission shows success. Test empty name shows error.

### Key Insight

You can embed the entire application HTML inside the test using `page.setContent()`. This makes the test fully self-contained — no external server needed. The form's JavaScript handles validation and success message display.

### Thinking Process

1. Create HTML with a form, validation logic, success div, and error spans.
2. Use `page.getByLabel` to fill each field by its label text — this mirrors real user behavior.
3. Submit via `getByRole('button')` to use the accessible role.
4. After submission, check for the success message.
5. For the error test, skip filling the name field and submit again.
6. Assert the error message appears.

### Complete Code

```typescript
import { test, expect } from "@playwright/test";

test.describe("Form tests", () => {
  test.beforeEach(async ({ page }) => {
    // Set up a complete form with client-side validation.
    // The form has name (required), email, and message fields.
    // A success message div is hidden by default and shown on valid submission.
    // Error spans next to required fields display validation messages.
    await page.setContent(`
      <div id="app" style="font-family: sans-serif; max-width: 500px; margin: 2rem auto;">
        <h1>Contact Us</h1>
        <form id="contact-form">
          <div style="margin-bottom: 1rem;">
            <label for="name">Name *</label>
            <input type="text" id="name" name="name" required
                   style="display: block; width: 100%; padding: 0.5rem;" />
            <span id="name-error" style="color: red; display: none;">Name is required</span>
          </div>
          <div style="margin-bottom: 1rem;">
            <label for="email">Email</label>
            <input type="email" id="email" name="email"
                   style="display: block; width: 100%; padding: 0.5rem;" />
          </div>
          <div style="margin-bottom: 1rem;">
            <label for="message">Message</label>
            <textarea id="message" name="message" rows="4"
                      style="display: block; width: 100%; padding: 0.5rem;"></textarea>
          </div>
          <button type="submit" style="padding: 0.5rem 2rem;">Submit</button>
        </form>
        <div id="success" style="display: none; color: green; margin-top: 1rem;">
          Thank you! Your message has been sent.
        </div>
      </div>
      <script>
        document.getElementById('contact-form').addEventListener('submit', function(e) {
          e.preventDefault();
          const name = document.getElementById('name').value.trim();
          const nameError = document.getElementById('name-error');
          const success = document.getElementById('success');
          if (!name) {
            nameError.style.display = 'block';
            success.style.display = 'none';
            return;
          }
          nameError.style.display = 'none';
          success.style.display = 'block';
        });
      </script>
    `);
  });

  test("successful form submission", async ({ page }) => {
    // Fill each form field using getByLabel — finds the label element and
    // targets the associated input by the 'for' attribute.
    await page.getByLabel("Name *").fill("Jane Doe");
    await page.getByLabel("Email").fill("jane@example.com");
    await page.getByLabel("Message").fill("Hello, I would like to learn more about Playwright.");

    // Click the submit button using getByRole — finds the <button> with text "Submit".
    await page.getByRole("button", { name: "Submit" }).click();

    // Assert the success message is visible after submission.
    // getByText finds the element by its visible text content.
    await expect(page.getByText("Thank you! Your message has been sent.")).toBeVisible();

    // Assert the error message is NOT visible (happy path).
    await expect(page.getByText("Name is required")).not.toBeVisible();
  });

  test("empty name shows error", async ({ page }) => {
    // Fill only the email and message fields, leaving name empty.
    await page.getByLabel("Email").fill("jane@example.com");
    await page.getByLabel("Message").fill("Hello from Playwright!");

    // Click submit — form validation should catch the empty name.
    await page.getByRole("button", { name: "Submit" }).click();

    // Assert the error message for the name field is now visible.
    await expect(page.getByText("Name is required")).toBeVisible();

    // Assert the success message is NOT visible.
    await expect(page.getByText("Thank you! Your message has been sent.")).not.toBeVisible();
  });
});
```

### Why This Works

- `page.setContent()` loads the HTML immediately without any network request, making the test fast and reliable.
- `getByLabel` automatically resolves the label-input association via the `for` attribute, so it types into the correct field.
- The form's JavaScript handles both cases: showing the success div on valid input and showing the error span on invalid input.
- Playwright's auto-waiting ensures the form's DOM updates are visible before the assertions run.

### Edge Cases and Gotchas

- If the label text contains special characters (like the asterisk in "Name *"), `getByLabel` still matches because it uses the full text content.
- The `fill` method clears the field before typing, so you do not need to clear it manually.
- Client-side validation can interfere — if the browser's built-in `required` validation fires, it may show a browser-native tooltip instead of your custom error. The example uses `e.preventDefault()` in the submit handler to bypass browser-native validation.
- If multiple elements match `getByText("Name")`, the assertion may fail. Use `{ exact: true }` or a more specific locator.

### Alternative Approaches

- Use `page.locator('#name')` for direct CSS-based field selection, but this ties the test to internal IDs.
- Use `page.getByRole('textbox', { name: 'Name *' })` instead of `getByLabel` for a role-based approach.
- Write a single test that checks both paths using `test.step` for better reporting.

## Intermediate Exercise 1: Combine API and UI

### Problem Restated

Use the JSONPlaceholder API to create a post via POST, verify it via GET, render it in a simulated UI table, and clean up via DELETE.

### Key Insight

Playwright's `page.request` API provides a full HTTP client that reuses the browser's cookies and storage context but is also available for isolated API calls. This allows you to mix API data seeding with UI verification in a single test.

### Thinking Process

1. POST to create a new blog post. Capture the returned `id`.
2. GET the post by `id` to verify it was created correctly.
3. Build an HTML table that includes this post data, load it via `setContent`.
4. Assert the post title appears in the table.
5. DELETE the post to clean up.

### Complete Code

```typescript
import { test, expect } from "@playwright/test";

test("API seed then UI verify", async ({ page }) => {
  const BASE = "https://jsonplaceholder.typicode.com";

  // Step 1: Seed data via POST request.
  // page.request.post() sends a POST request through the browser's network stack.
  // The response is a standard Response object — use .json() to parse the body.
  const postResponse = await page.request.post(`${BASE}/posts`, {
    data: {
      title: "Playwright API Testing",
      body: "Combining API calls with UI verification ensures end-to-end data integrity.",
      userId: 1,
    },
  });

  // Verify the POST request succeeded (HTTP 201 Created).
  expect(postResponse.status()).toBe(201);

  // Parse the response body to get the created post's ID.
  const createdPost = await postResponse.json();
  const postId = createdPost.id;
  console.log(`Created post with ID: ${postId}`);

  // Step 2: Verify the post exists via GET request.
  const getResponse = await page.request.get(`${BASE}/posts/${postId}`);
  expect(getResponse.status()).toBe(200);
  const fetchedPost = await getResponse.json();
  expect(fetchedPost.title).toBe("Playwright API Testing");
  expect(fetchedPost.id).toBe(postId);

  // Step 3: Create a simulated UI table with the post data.
  // In a real app, a React/Vue component would fetch and render this data.
  // Here we simulate the rendered output to test the UI layer.
  await page.setContent(`
    <html>
      <head><title>Blog Posts</title></head>
      <body style="font-family: sans-serif; padding: 2rem;">
        <h1>Recent Posts</h1>
        <table border="1" cellpadding="8" style="width: 100%; border-collapse: collapse;">
          <thead>
            <tr>
              <th>ID</th>
              <th>Title</th>
              <th>Body</th>
              <th>User</th>
            </tr>
          </thead>
          <tbody>
            <tr>
              <td>${fetchedPost.id}</td>
              <td>${fetchedPost.title}</td>
              <td>${fetchedPost.body}</td>
              <td>${fetchedPost.userId}</td>
            </tr>
          </tbody>
        </table>
      </body>
    </html>
  `);

  // Step 4: Assert the post title appears in the table.
  // getByText with exact: false (default) finds partial matches.
  await expect(page.getByText("Playwright API Testing")).toBeVisible();

  // Also verify the post ID appears in the table.
  await expect(page.getByText(`${postId}`)).toBeVisible();

  // Step 5: Clean up by deleting the post.
  const deleteResponse = await page.request.delete(`${BASE}/posts/${postId}`);
  expect(deleteResponse.status()).toBe(200);
  console.log(`Deleted post with ID: ${postId}`);
});
```

### Why This Works

- `page.request.post()`, `.get()`, and `.delete()` are thin wrappers around the standard Fetch API, so they behave consistently with the browser's network stack.
- By seeding data via the API, you create deterministic test data without relying on manual UI workflows.
- The UI verification step confirms that the data pipeline (API -> frontend -> render) works correctly end-to-end.
- The cleanup step prevents test data pollution in shared environments.

### Edge Cases and Gotchas

- JSONPlaceholder is a fake API — POST requests return a mock response with `id: 101` (the next available mock ID), not a real database entry. GET requests for that ID will return a 200 with the mock data. DELETE always returns 200. The test logic is correct for the API simulation.
- In a real application, you would use your own backend. Always check the response status code to catch API changes.
- The HTML template uses template literals with `${}` — be careful about XSS if the fetched data contains user-generated content. Use `textContent` or a templating engine for production tests.
- Some APIs return `201` for create, some return `200`. Check your target API's convention.

### Alternative Approaches

- Use `apiRequestContext` from a fixture for reusable API clients with authentication.
- Extract the API calls into a helper class (e.g., `ApiClient.post(title, body)`) for reuse across tests.
- Use `test.use({ storageState: 'auth.json' })` if the API requires authentication cookies.

## Intermediate Exercise 2: POM with Fixtures

### Problem Restated

Build a three-page checkout flow with page objects and a custom fixture that injects all page objects into tests.

### Key Insight

Custom fixtures give you dependency injection for page objects. Instead of manually constructing page objects in every test, you define them once in a fixture file and use them directly with `test.use` or by destructuring from the extended `test`.

### Thinking Process

1. Create three page objects: CartPage, CheckoutPage, PaymentPage.
2. Each page object exposes typed locators and action methods.
3. Create a custom fixture `checkout` that composes all three pages and provides flow-level methods.
4. The fixture navigates to the cart on creation.
5. Tests use the `checkout` fixture to run through the entire flow.

### Complete Code

**pages/cart-page.ts:**

```typescript
import { type Page, type Locator, expect } from "@playwright/test";

export class CartPage {
  // Mark fields as readonly to prevent accidental reassignment.
  readonly page: Page;
  readonly checkoutButton: Locator;
  readonly cartItems: Locator;

  constructor(page: Page) {
    this.page = page;
    // Locate the checkout button by its accessible role and name.
    this.checkoutButton = page.getByRole("button", { name: "Checkout" });
    // Locate all cart item rows by their test ID.
    this.cartItems = page.locator("[data-testid='cart-item']");
  }

  async goto() {
    await this.page.goto("https://example.com/cart");
  }

  async getItemCount(): Promise<number> {
    return this.cartItems.count();
  }

  async proceedToCheckout() {
    // Click the checkout button and wait for navigation.
    await this.checkoutButton.click();
    await this.page.waitForURL(/.*checkout.*/);
  }
}
```

**pages/checkout-page.ts:**

```typescript
import { type Page, type Locator } from "@playwright/test";

export class CheckoutPage {
  readonly page: Page;
  readonly nameInput: Locator;
  readonly addressInput: Locator;
  readonly cityInput: Locator;
  readonly zipInput: Locator;
  readonly continueButton: Locator;
  readonly form: Locator;

  constructor(page: Page) {
    this.page = page;
    this.form = page.locator("form");
    // Use getByLabel to find fields by their associated label text.
    this.nameInput = page.getByLabel("Full Name");
    this.addressInput = page.getByLabel("Address");
    this.cityInput = page.getByLabel("City");
    this.zipInput = page.getByLabel("ZIP Code");
    this.continueButton = page.getByRole("button", { name: "Continue to Payment" });
  }

  async fillShippingInfo(name: string, address: string, city: string, zip: string) {
    await this.nameInput.fill(name);
    await this.addressInput.fill(address);
    await this.cityInput.fill(city);
    await this.zipInput.fill(zip);
  }

  async continueToPayment() {
    await this.continueButton.click();
    await this.page.waitForURL(/.*payment.*/);
  }
}
```

```typescript
import { type Page, type Locator } from "@playwright/test";

export class PaymentPage {
  readonly page: Page;
  readonly cardNumberInput: Locator;
  readonly expiryInput: Locator;
  readonly cvvInput: Locator;
  readonly placeOrderButton: Locator;
  readonly successMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.cardNumberInput = page.getByLabel("Card Number");
    this.expiryInput = page.getByLabel("Expiry Date");
    this.cvvInput = page.getByLabel("CVV");
    this.placeOrderButton = page.getByRole("button", { name: "Place Order" });
    this.successMessage = page.getByText("Order placed successfully!");
  }

  async fillPaymentDetails(cardNumber: string, expiry: string, cvv: string) {
    await this.cardNumberInput.fill(cardNumber);
    await this.expiryInput.fill(expiry);
    await this.cvvInput.fill(cvv);
  }

  async placeOrder() {
    await this.placeOrderButton.click();
  }

  async isOrderSuccessful(): Promise<boolean> {
    return this.successMessage.isVisible();
  }
}
```

```typescript
    );

    // Place the order.
    await paymentPage.placeOrder();

    // Verify the success message is visible.
    expect(await paymentPage.isOrderSuccessful()).toBe(true);
  });
});
```

### Why This Works

- Page objects encapsulate selectors and interaction logic, making tests readable and maintainable.
- Custom fixtures provide dependency injection — page objects are created once and reused across tests.
- The fixture's `use` callback pattern ensures proper cleanup (Playwright closes the page after each test).
- Route interception makes the test fully self-contained, no external server required.

### Edge Cases and Gotchas

- The `use` callback in a fixture is async. You must `await use(...)` to pass control to the test. Nothing after `use()` runs until the test completes.
- If a page object method throws, the fixture's teardown (code after `use()`) still runs, so you can add cleanup logic there.
- Multiple fixtures that depend on the same `page` fixture share the same page instance, which is what you want here.
- `page.route()` interceptors must be registered before navigation. Using `beforeEach` ensures they are in place.

### Alternative Approaches

- Use a single `checkoutApp` fixture that exposes all three page objects as properties instead of three separate fixtures.
- Use Playwright's `PageObject` pattern with `pageObjects` config option (experimental in newer versions).
- Extract route interceptors into a separate `mock-server.ts` helper module for reuse across test files.

### Analyzing the Trace

After running the failed test, follow these steps to inspect the trace:

1. **Locate the trace file**:
   The trace is saved in `test-results/visual-trace-spec-broken-ui-detected-by-visual-diff/` (the folder name depends on your project name and test title). Look for a `trace.zip` file.

2. **Open the trace viewer**:
   ```bash
   npx playwright show-trace test-results/.../trace.zip
   ```

3. **What to look for in the trace**:
   - **DOM Snapshot tab**: Compare the page DOM from the passing vs failing snapshot. You will see the `background-color: blue` changed to `background-color: red` and the button text changed from "Submit" to "Send".
   - **Console tab**: Check for any console errors or warnings.
   - **Network tab**: Verify no unexpected network requests occurred.
   - **Timeline tab**: See the sequence of actions and the exact moment the visual comparison failed.

4. **The diff image**:
   Playwright generates `baseline.png.actual.png` (the actual screenshot), `baseline.png.expected.png` (the baseline), and `baseline.png.diff.png` (pixel differences highlighted in pink). The diff image clearly shows the box area and button text as changed regions.

5. **Summary of findings**:
   The trace reveals that the DOM structure is identical except for:
   - The `#box` element's `background-color` changed from `blue` to `red`
   - The `#action-btn` element's text changed from "Submit" to "Send"
   - No console errors or network issues — the failure is purely visual
   - The visual diff highlights both changes in the diff overlay

### Why This Works

- `toHaveScreenshot()` compares the current pixel rendering against a stored baseline image.
- When the comparison fails, Playwright generates `actual`, `expected`, and `diff` images for manual inspection.
- Traces capture a complete recording of the test: DOM snapshots, network requests, console logs, and test steps.
- The trace viewer's "DOM snapshot" tab lets you inspect the page state at any action, showing the actual rendered HTML.

### Edge Cases and Gotchas

- Visual snapshots are fragile — font rendering, OS differences, and anti-aliasing can cause false positives. Use Docker for consistent rendering across environments.
- The `snapshotPathTemplate` must be configured correctly, or snapshots may be stored in unexpected locations.
- Traces on failure consume disk space. In CI, configure trace retention to keep only failed test traces.
- Diff images are only generated if the snapshot comparison fails. Passing tests do not produce diff images.
- Anti-aliasing around borders and text can cause minor pixel differences. Use `maxDiffPixels` or `maxDiffPixelRatio` options to tolerate small variations.

### Alternative Approaches

- Use `toMatchSnapshot()` for non-visual snapshot comparisons (text, JSON).
- Use Playwright's Component Testing mode for visual snapshots of individual components.
- Integrate with Percy or Applitools for cloud-based visual regression with AI-driven diff analysis.

```yaml
name: Playwright Tests

# Define environment variables shared across all jobs.
env:
  BASE_URL: https://staging.example.com
  PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD: 0

on:
  # PR trigger — fast feedback with only Chromium, no traces.
  pull_request:
    branches: [main]

  push:
    branches: [main]
  # Scheduled nightly run — full suite with full traces.
  schedule:
    - cron: '0 2 * * *'  # Every day at 2 AM UTC

jobs:
  # PR check job: quick Chromium-only run for fast feedback.
  pr-check:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      # Cache npm dependencies using the lock file hash.
      - name: Cache npm dependencies
        uses: actions/cache@v3
        id: npm-cache
        with:
          path: ~/.npm
          key: npm-${{ hashFiles('package-lock.json') }}
          restore-keys: |
            npm-

      - name: Install dependencies
        run: npm ci

      # Cache Playwright browsers to avoid re-download on every run.
      - name: Cache Playwright browsers
        uses: actions/cache@v3
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: playwright-${{ hashFiles('package-lock.json') }}

      - name: Install Playwright browsers (Chromium only)
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: npx playwright install chromium

      # Run only Chromium tests for PR checks — fast and no traces.
      - name: Run Chromium tests
        run: npx playwright test --project=chromium
        env:
          PWTEST_TRACE: "off"  # Explicitly disable traces for speed.

      # Upload the HTML report as an artifact for review.
      - name: Upload HTML report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report-pr
          path: playwright-report/
          retention-days: 7

  # Main CI job: full suite across all browsers with sharding.
  ci-tests:
    if: github.event_name != 'pull_request'
    strategy:
      fail-fast: false  # Don't cancel other shards if one fails.
      matrix:
        # Four shards for the Chromium project.
        shard: [1, 2, 3, 4]
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Cache npm dependencies
        uses: actions/cache@v3
        id: npm-cache
        with:
          path: ~/.npm
          key: npm-${{ hashFiles('package-lock.json') }}
          restore-keys: |
            npm-

      - name: Install dependencies
        run: npm ci

      - name: Cache Playwright browsers
        uses: actions/cache@v3
        id: playwright-cache
        with:
          path: ~/.cache/ms-playwright
          key: playwright-${{ hashFiles('package-lock.json') }}

      - name: Install all Playwright browsers
        if: steps.playwright-cache.outputs.cache-hit != 'true'
        run: npx playwright install --with-deps

      # Run tests with sharding for Chromium, full run for other projects.
      # The --shard flag splits tests across shards.
      - name: Run Playwright tests
        run: |
          npx playwright test \
            --project=chromium \
            --shard=${{ matrix.shard }}/4
        env:
          # On scheduled runs, capture traces on all failures.
          PWTEST_TRACE: ${{ github.event_name == 'schedule' && 'on' || 'on-first-retry' }}

      # Upload blob reports for the merge step.
      # Each shard uploads its own blob report.
      - name: Upload blob report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: blob-report-chromium-shard-${{ matrix.shard }}
          path: blob-report/
          retention-days: 3

  # Run Firefox, WebKit, and Mobile Chrome on separate runners.
  # These do not need sharding because they have fewer tests or run shorter.
  other-browsers:
    if: github.event_name != 'pull_request'
    strategy:
      matrix:
        browser: [firefox, webkit, mobile-chrome]
    runs-on: ubuntu-latest
    timeout-minutes: 20
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Install dependencies
        run: npm ci
      - name: Install Playwright browsers
        run: npx playwright install --with-deps ${{ matrix.browser }}
      - name: Run ${{ matrix.browser }} tests
        run: npx playwright test --project=${{ matrix.browser }}
      - name: Upload blob report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: blob-report-${{ matrix.browser }}
          path: blob-report/
          retention-days: 3

  # Merge job: collects all blob reports from all shards and browsers,
  # then generates a single unified HTML report.
  merge-reports:
    if: github.event_name != 'pull_request'
    needs: [ci-tests, other-browsers]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Install dependencies
        run: npm ci

      # Download all blob reports.
      - name: Download all blob reports
        uses: actions/download-artifact@v4
        with:
          pattern: blob-report-*
          path: blob-report/
          merge-multiple: true

      # Merge all blob reports into a single HTML report.
      - name: Merge reports
        run: npx playwright merge-reports --reporter html ./blob-report

      - name: Upload merged HTML report
        uses: actions/upload-artifact@v4
        with:
          name: merged-playwright-report
          path: playwright-report/
          retention-days: 30
```

### Sharding Equation

The Chromium test suite is sharded across 4 parallel runners. Each runner receives roughly 25% of the tests.

**Total containers used:**
- PR trigger: 1 runner (Chromium only)
- Push to main trigger: 4 Chromium shards + 3 other browser runners + 1 merge runner = 8 total containers
- Scheduled nightly: same as push (8 containers)

**Distribution:**
- Chromium tests are split by file path: shard 1 gets files A-D, shard 2 gets E-H, etc.
- Firefox, WebKit, and Mobile Chrome each run their full suite on a single runner.

**Expected time savings:**
- Without sharding: 1 Chromium runner takes ~20 minutes
- With 4 shards: each Chromium runner takes ~5 minutes (not exactly 25% due to uneven test distribution)
- Total wall clock time for push: max(5 min for chromium shard, ~8 min for other browsers) + 1 min for merge = about 9 minutes total
- Without sharding and parallelization: ~30 minutes (running all browsers sequentially on one runner)

### Why This Works

- Blob reports are independent per shard — they contain all test data including traces, screenshots, and errors.
- The merge step (`playwright merge-reports`) combines all blob reports into a single HTML report with all test results visible in one place.
- Caching Playwright browsers saves 30-60 seconds per runner on install.
- Conditional triggers ensure PRs get fast feedback while main pushes get full coverage.

### Edge Cases and Gotchas

- Blob report file names must be unique across shards. Use the PLAYWRIGHT_SHARD env variable or a unique suffix per shard.
- The `merge-multiple: true` option in download-artifact merges all matching artifacts into one directory.
- If a shard produces no blob report (e.g., it had zero tests assigned), the merge step may fail. Use `if: always()` on the upload step to ensure blobs are uploaded even on failure.
- The `retention-days` setting on artifacts should align with your team's debugging window. 30 days for merged reports, 3 days for individual blob reports.
- The `fail-fast: false` on the matrix ensures other shards continue running even if one shard fails, giving you complete test results.

### Alternative Approaches

- Use GitHub Actions `matrix` for all browsers, not just Chromium, to parallelize Firefox/WebKit/Mobile as well.
- Use a self-hosted runner with larger disk for trace storage.
- Replace blob reports with HTML reports per shard and use a custom merge script.
- Add a Slack notification step in the merge job to alert the team on failure.

**helpers/mock-server.ts:**

```typescript
import type { Page, Route } from "@playwright/test";

// Define the user database and role mappings.
// In a real app, these would come from your backend.
const USERS: Record<string, { password: string; role: "admin" | "user" | "viewer" }> = {
  "admin@test.com": { password: "admin123", role: "admin" },
  "user@test.com": { password: "user123", role: "user" },
  "viewer@test.com": { password: "viewer123", role: "viewer" },
};

// Define which pages each role can access.
const ROLE_ACCESS: Record<string, string[]> = {
  admin: ["/admin", "/dashboard", "/profile", "/settings"],
  user: ["/dashboard", "/profile"],
  viewer: ["/dashboard"],
};

// All protected pages.
const PROTECTED_PAGES = ["/admin", "/dashboard", "/profile", "/settings"];

/**
 * Set up route interception to simulate a multi-role web application.
 * Call this in the test's beforeEach or in the setup project.
 *
 * @param page - The Playwright Page object
 * @param role - The role to simulate (admin, user, viewer), or null for unauthenticated
 */
export async function setupMockServer(page: Page, role: string | null) {
  // Intercept all requests to the base URL.
  await page.route("**/*", async (route: Route) => {
    const url = new URL(route.request().url());
    const path = url.pathname;

    // Handle login API endpoint.
    if (path === "/api/login" && route.request().method() === "POST") {
      const body = JSON.parse(route.request().postData() || "{}");
      const user = USERS[body.email];

      if (user && body.password === user.password) {
        // Successful login — return user data and a fake auth token.
        return route.fulfill({
          status: 200,
          contentType: "application/json",
          body: JSON.stringify({
            token: `mock-token-${user.role}`,
            role: user.role,
            email: body.email,
          }),
        });
      } else {
        // Invalid credentials.
        return route.fulfill({
          status: 401,
          contentType: "application/json",
          body: JSON.stringify({ error: "Invalid email or password" }),
        });
      }
    }

    // Handle page access for protected pages.
    if (PROTECTED_PAGES.includes(path)) {
      if (!role) {
        // Unauthenticated — redirect to login.
        return route.fulfill({
          status: 200,
          contentType: "text/html",
          body: `
            <html><body>
              <h1>Login Page</h1>
              <p>You must log in to access this page.</p>
              <a href="/login">Login</a>
              <script>
                // Store the intended redirect in sessionStorage.
                sessionStorage.setItem("redirect", "${path}");
              </script>
            </body></html>
          `,
        });
      }

      const allowedPages = ROLE_ACCESS[role] || [];
      if (allowedPages.includes(path)) {
        // Role has access — render the page.
        return route.fulfill({
          status: 200,
          contentType: "text/html",
          body: `
            <html><body>
              <h1>${path.slice(1).charAt(0).toUpperCase() + path.slice(2)} Page</h1>
              <p>Welcome, ${role} role!</p>
              <p>You are viewing: ${path}</p>
              <nav>
                <a href="/dashboard">Dashboard</a>
                ${role === "admin" ? '<a href="/admin">Admin</a>' : ""}
                ${role !== "viewer" ? '<a href="/profile">Profile</a>' : ""}
                ${role === "admin" ? '<a href="/settings">Settings</a>' : ""}
                <a href="/logout">Logout</a>
              </nav>
              <div id="role-badge" data-testid="role-badge">${role}</div>
            </body></html>
          `,
        });
      } else {
        // Role does not have access — return 403 Forbidden.
        return route.fulfill({
          status: 403,
          contentType: "text/html",
          body: `
            <html><body>
              <h1>403 Forbidden</h1>
              <p>Your role (${role}) does not have access to ${path}.</p>
            </body></html>
          `,
        });
      }
    }

    // Handle the login page itself.
    if (path === "/login") {
      return route.fulfill({
        status: 200,
        contentType: "text/html",
        body: `
          <html><body>
            <h1>Login</h1>
            <form id="login-form" action="/api/login" method="POST">
              <label>Email: <input type="email" name="email" id="email" /></label><br/>
              <label>Password: <input type="password" name="password" id="password" /></label><br/>
              <button type="submit" id="login-btn">Log In</button>
            </form>
            <div id="error" style="display:none; color:red;"></div>
            <script>
              document.getElementById("login-form").addEventListener("submit", async (e) => {
                e.preventDefault();
                const email = document.getElementById("email").value;
                const password = document.getElementById("password").value;
                const res = await fetch("/api/login", {
                  method: "POST",
                  headers: { "Content-Type": "application/json" },
                  body: JSON.stringify({ email, password }),
                });
                if (res.ok) {
                  const data = await res.json();
                  // Store auth data in localStorage to simulate session.
                  localStorage.setItem("auth", JSON.stringify(data));
                  const redirect = sessionStorage.getItem("redirect") || "/dashboard";
                  window.location.href = redirect;
                } else {
                  document.getElementById("error").style.display = "block";
                  document.getElementById("error").textContent = "Invalid credentials";
                }
              });
            </script>
          </body></html>
        `,
      });
    }

    // Handle logout.
    if (path === "/logout") {
      return route.fulfill({
        status: 200,
        contentType: "text/html",
        body: `
          <html><body>
            <h1>Logged Out</h1>
            <p>You have been logged out.</p>
            <a href="/login">Log in again</a>
            <script>
              localStorage.removeItem("auth");
            </script>
          </body></html>
        `,
      });
    }

    // Default: render a simple home page.
    return route.fulfill({
      status: 200,
      contentType: "text/html",
      body: `
        <html><body>
          <h1>Home Page</h1>
          <nav>
            <a href="/login">Login</a>
            <a href="/dashboard">Dashboard</a>
            <a href="/admin">Admin</a>
            <a href="/profile">Profile</a>
            <a href="/settings">Settings</a>
          </nav>
        </body></html>
      `,
    });
  });
}
```

```typescript
import { test, expect } from "@playwright/test";
import { setupMockServer } from "../helpers/mock-server";

test.describe("Session expiry tests", () => {
  test.beforeEach(async ({ page }) => {
    // Set up the mock server without a role (unauthenticated).
    // The mock server will redirect protected pages to login.
    await setupMockServer(page, null);
  });

  test("unauthenticated user is redirected to login", async ({ page }) => {
    // Try to access a protected page without being logged in.
    await page.goto("/dashboard");

    // The mock server should render the login page for protected pages
    // when no role is set.
    await expect(page.getByText("Login Page")).toBeVisible();
    await expect(page.getByText("You must log in to access this page.")).toBeVisible();
  });

  test("session expiry after logout redirects to login", async ({ page }) => {
    // First, simulate an authenticated session by logging in as a user.
    await setupMockServer(page, "user");
    await page.goto("/dashboard");
    await expect(page.getByText("Dashboard Page")).toBeVisible();

    // Now simulate session expiry by clearing storage state.
    // In a real app, this could be a token timeout, server-side logout, etc.
    // We clear the page's storage to simulate the session ending.
    await page.evaluate(() => {
      localStorage.clear();
      sessionStorage.clear();
    });

    // Reload the page — the mock server sees no role, so it shows the login form.
    await page.reload();

    // After clearing storage and reloading, the mock server sees no auth data.
    // The protected page route handler returns the login page for unauthenticated users.
    await expect(page.getByText("Login Page")).toBeVisible();
  });

  test("accessing protected page after logout redirects", async ({ page }) => {
    // Log in as admin first.
    await setupMockServer(page, "admin");
    await page.goto("/dashboard");
    await expect(page.getByText("Dashboard Page")).toBeVisible();

    // Simulate session expiry.
    await page.evaluate(() => {
      localStorage.clear();
    });

    // Try to access a protected page.
    await setupMockServer(page, null);  // Set role to null for unauthenticated.
    await page.goto("/admin");

    // Should see the login page.
    await expect(page.getByText("Login Page")).toBeVisible();
    // The redirect URL should be stored in sessionStorage for post-login redirect.
  });
});
```

### Why This Works

- **Setup projects** run before test projects and save authenticated browser state to JSON files. Test projects declare a dependency on their setup project and load the saved state automatically.
- **Route interception** (`page.route()`) simulates a real backend without requiring a server. The mock server checks roles and returns appropriate responses (200, 403, or login page).
- **storageState** captures cookies, localStorage, and sessionStorage. When loaded, the browser starts in the authenticated state as if the user just logged in.
- **Each test project** is fully isolated by role — a bug in the admin flow does not affect user or viewer tests.

### Edge Cases and Gotchas

- The setup project's `testMatch` pattern must match only the auth setup file. If it matches other test files, they will run as setup projects too.
- storageState files are written during setup and read during test projects. They must be in `.gitignore` (they are machine-generated).
- The `dependencies` array in config ensures setup runs before tests. Without this, tests may run before the storage state is created.
- The mock server must handle all routes the tests visit. Any unhandled route will cause a real network request that may fail.
- Session expiry tests must be in a separate project without storageState, or they would reuse the stored authenticated state.

### Alternative Approaches

- Use `globalSetup` instead of setup projects for a single global auth flow. Setup projects are better when you need multiple auth states.
- Use `test.use({ storageState: roleStorageMap[test.info().project.name] })` inside tests for dynamic role assignment.
- Store auth tokens in environment variables for CI instead of having setup projects log in every time.
- Use Playwright's `apiRequestContext` for quicker API-based login in setup projects instead of UI-based login.
