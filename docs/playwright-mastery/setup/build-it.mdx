# Setup — Build It

**Reading time:** 12 minutes · **Difficulty:** Beginner

## Step by Step

### 1. Initialize Playwright in your project

If you are continuing from the previous lesson, you already have a project. Otherwise, create a new directory and run:

```bash
mkdir playwright-workspace
cd playwright-workspace
npm init playwright@latest
```

When prompted, choose:

- **TypeScript** — for type safety and better IDE support
- **`./tests`** — for the test file location
- **No** to GitHub Actions workflow (you will set that up later in the course)
- **Yes** to install browsers

### 2. Understand the generated `playwright.config.ts`

Open the generated `playwright.config.ts` file. It looks approximately like this:

```typescript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 1 : undefined,
  reporter: 'html',
  use: {
    trace: 'on-first-retry',
  },
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
    },
    {
      name: 'webkit',
      use: { ...devices['Desktop Safari'] },
    },
  ],
});
```

Here is what each property controls:

- **`testDir`** — Playwright looks for test files in this directory. Any file matching `*.spec.ts` or `*.spec.js` inside this folder is treated as a test file.
- **`fullyParallel`** — When `true`, all tests across all files run in parallel. Set to `false` if tests share state and must run sequentially.
- **`forbidOnly`** — If `true` and `test.only` is used, the test run fails. Useful in CI to prevent accidentally committing focused tests.
- **`retries`** — Number of times to retry a failed test. In CI, 2 retries is a good starting point to handle flaky infrastructure. Locally, 0 retries is fine.
- **`workers`** — Maximum number of parallel worker processes. `undefined` means Playwright uses 50% of available CPU cores. Set to `1` in CI if you want deterministic execution.
- **`reporter`** — Output format. `'html'` generates a rich HTML report. `'list'` prints test results to the terminal. You can use multiple reporters: `reporter: [['html'], ['list']]`.
- **`use.trace`** — When to capture a trace (a detailed execution log with DOM snapshots, network logs, and console output). `'on-first-retry'` captures a trace only when a test fails on the first attempt and is being retried.
- **`projects`** — Each project runs the same tests against a different browser configuration. The example above runs every test in Chromium, Firefox, and WebKit.

### 3. Configure for three browsers

The generated config already includes Chromium, Firefox, and WebKit projects. If your config file is missing any, add them:

```typescript
projects: [
  {
    name: 'chromium',
    use: { ...devices['Desktop Chrome'] },
  },
  {
    name: 'firefox',
    use: { ...devices['Desktop Firefox'] },
  },
  {
    name: 'webkit',
    use: { ...devices['Desktop Safari'] },
  },
],
```

Run tests against a single browser:

```bash
npx playwright test --project=chromium
```

Run against all three:

```bash
npx playwright test
```

### 4. Install the VS Code extension

Open VS Code, go to the Extensions panel (Ctrl+Shift+X), and search for **Playwright Test for VSCode** by Microsoft. Install it.

After installation, you will see a testing icon in the sidebar. It shows:

- All discovered test files and individual tests
- Pass/fail status with inline error messages
- Options to run, debug, or record tests
- A "pick locator" button for interactively selecting elements

Click the play button next to any test to run it directly from the editor.

### 5. Set up `.gitignore`

Create a `.gitignore` file in your project root:

```bash
node_modules/
test-results/
playwright-report/
playwright/.cache/
```

The key entries are:

- **`node_modules/`** — Dependencies are not committed to version control
- **`test-results/`** — Test artifacts (screenshots, videos, traces) generated during test runs
- **`playwright-report/`** — The generated HTML report
- **`playwright/.cache/`** — Playwright's internal cache files

## Build Something Real

Now configure Playwright for a real project. Replace the contents of your `playwright.config.ts` with this production-ready configuration:

```typescript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 4 : undefined,
  reporter: [
    ['html', { outputFolder: 'playwright-report' }],
    ['list'],
  ],
  use: {
    baseURL: 'https://example.com',
    screenshot: 'only-on-failure',
    trace: 'retain-on-failure',
    video: 'retain-on-failure',
  },
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
    },
    {
      name: 'webkit',
      use: { ...devices['Desktop Safari'] },
    },
  ],
});
```

Key additions from the default config:

- **`baseURL`** — All test URLs become relative: `await page.goto('/login')` navigates to `https://example.com/login`
- **`screenshot: 'only-on-failure'`** — Automatically captures a screenshot when a test fails, invaluable for debugging
- **`trace: 'retain-on-failure'`** — Captures a full trace (network, console, DOM snapshots) only on failure
- **`video: 'retain-on-failure'`** — Records a video of the test run, saved only when the test fails
- **`workers: 4`** — In CI, use exactly 4 parallel workers (tune this to your CI machine's CPU count)

Create a test to verify the configuration works. Save this as `tests/config-check.spec.ts`:

```typescript
import { test, expect } from '@playwright/test';

test('configuration is working', async ({ page }) => {
  // Uses baseURL from config — this navigates to https://example.com
  await page.goto('/');
  await expect(page).toHaveTitle(/Example/);
  console.log('Base URL:', page.url());
});

test('screenshot on failure demo', async ({ page }) => {
  await page.goto('/');
  // This will pass — the h1 exists on example.com
  await expect(page.locator('h1')).toBeVisible();
});

test('trace captures network activity', async ({ page }) => {
  await page.goto('/');
  // Verify the page loaded with a network call
  const response = await page.goto('/');
  expect(response?.status()).toBe(200);
});
```

Run the test:

```bash
npx playwright test tests/config-check.spec.ts
```

Since all tests pass, no screenshots, traces, or videos are saved. To see the failure artifacts in action, intentionally break a test and run again:

```typescript
test('this will fail and generate artifacts', async ({ page }) => {
  await page.goto('/');
  await expect(page.locator('h1')).toHaveText('Nonexistent Text');
});
```

After the failed run, check the `test-results/` directory. You will find:

- A screenshot of the page at the moment of failure
- A trace file you can open with `npx playwright show-trace path/to/trace.zip`
- A video recording of the test execution
