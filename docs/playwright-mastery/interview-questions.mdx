# Interview Questions

## Why Playwright

### Easy

**Q: What is Playwright and how does it differ from Selenium WebDriver?**

Playwright is an open-source end-to-end testing framework developed by Microsoft. Unlike Selenium WebDriver, which communicates with browsers via the WebDriver protocol (requiring a separate driver binary per browser), Playwright uses the Chrome DevTools Protocol (CDP) for Chromium and proprietary protocols for Firefox and WebKit via a single `playwright` binary. Playwright also provides auto-waiting, browser context isolation, and network interception out of the box — features that require third-party libraries in Selenium.

**Q: Which browsers does Playwright support?**

Playwright supports three browser engines: Chromium (Google Chrome, Microsoft Edge, Opera), Firefox (Mozilla Firefox), and WebKit (Apple Safari). Each browser can run in both headless and headed mode. Playwright also supports mobile device emulation for Chrome on Android and Safari on iOS.

**Q: What is the difference between headed and headless mode in Playwright?**

In headless mode, the browser runs without a visible UI window. It is faster and uses fewer system resources, making it ideal for CI environments. In headed mode, the browser UI is visible, which helps with debugging by letting you see what the test is doing. You control the mode with the `headless` option in the config: `use: { headless: true }` (default) or `use: { headless: false }`.

### Medium

**Q: What are browser contexts in Playwright and why are they useful?**

A browser context is an isolated browser session akin to an incognito window. Each context has its own cookies, localStorage, sessionStorage, and browsing history. Contexts are useful for testing scenarios involving multiple users simultaneously — you can create one context per user and switch between them without logging out. They also enable parallel test execution without shared state. You create a context with `browser.newContext()` or use the default context from the `page` fixture in the test runner.

**Q: How does Playwright handle iframes and multi-page scenarios differently from other tools?**

Playwright treats iframes and popup pages as first-class citizens. For iframes, you use `page.frameLocator()` to get a `FrameLocator` and then chain locators on it — the API is identical to regular locators. For popups, Playwright provides a `page.on('popup')` event and `page.waitForEvent('popup')` pattern. You can also use `page.context().waitForEvent('page')` to capture new tabs. This contrasts with Selenium, where switching between frames, windows, and tabs requires explicit `switchTo()` calls.

**Q: What is the Playwright test runner and what benefits does it provide over using the library directly?**

The Playwright test runner (`@playwright/test`) is a full-featured test framework built on top of the Playwright library. It provides test fixtures (automatic page, context, browser creation and teardown), project configuration (multiple browsers, device emulation), parameterized tests, parallel execution, sharding, built-in reporters, and automatic retries. Using the library directly (`playwright` package) gives you lower-level control but requires you to manage browser lifecycle, parallelism, and reporting yourself.

### Hard

**Q: Explain how Playwright achieves cross-browser consistency despite browser differences in rendering and behavior.**

Playwright normalizes browser behavior by patching browser internals. For example, it standardizes how different browsers handle focus, scroll, and visibility checks. It injects its own Playwright-specific JavaScript into pages to ensure consistent selectors and event handling. The `expect` assertions retry until the condition is met, smoothing over timing differences between browsers. Playwright's team maintains the browsers (they fork the WebKit and Firefox builds) to ensure consistent API behavior across engines. This means the same Playwright script produces the same result on Chromium, Firefox, and WebKit.

**Q: Describe the architecture of Playwright's communication with browsers and how it avoids the "send-command-wait-for-result" bottleneck.**

Playwright communicates with browser processes through a single WebSocket connection using CDP (Chromium) or custom protocols (Firefox, WebKit). Instead of the traditional "send command, wait for response, send next command" RPC model, Playwright sends commands asynchronously over the WebSocket and uses a dispatcher pattern. The browser process maintains an event loop that processes commands and sends events back. This architecture allows Playwright to perform multiple operations concurrently (e.g., clicking a button while monitoring network requests) without blocking. The auto-waiting mechanism runs in the browser process itself, eliminating the round-trip latency of waiting for the client to check element state.

## Setup

### Easy

**Q: How do you install Playwright and its browsers?**

You install Playwright via npm: `npm init playwright@latest` creates a new project with all configuration files. You can also install manually: `npm install @playwright/test` then `npx playwright install` to download the browser binaries. The `npx playwright install` command downloads Chromium, Firefox, and WebKit into a cache directory. You can install individual browsers with `npx playwright install chromium`.

**Q: What configuration options go in `playwright.config.ts`?**

The config file controls test directory, timeouts, retries, projects (browser configurations), reporters, and shared `use` options. Common options include `testDir` (where tests live), `timeout` (per-test timeout), `retries` (number of retry attempts), `workers` (parallelism), `reporter` (report format), `projects` (array of browser device configurations), `use.baseURL` (base URL for relative URL navigation), and `use.viewport` (window size).

**Q: What is the purpose of the `fixtures` folder in a Playwright project?**

The `fixtures` folder (or `fixtures.ts` file) contains custom test fixtures — reusable components that inject dependencies into tests. Fixtures can be page objects, API clients, database connections, or any setup/teardown logic. By extending the base `test` with custom fixtures, you get dependency injection without manual construction in every test.

### Medium

**Q: Explain the role of `test.use()` and `test.describe.configure()` in test configuration.**

`test.use()` overrides configuration options for a single test or a group of tests inside a `describe` block. For example, `test.use({ storageState: 'admin.json' })` sets the authenticated state for one test. `test.describe.configure()` configures test-runner behavior for a group, such as `test.describe.configure({ mode: 'parallel' })` to run tests in parallel or `{ retries: 3 }` to override the retry count for a flaky group of tests.

**Q: How do you set up Visual Studio Code for Playwright development?**

Install the Playwright extension for VS Code, which provides a test explorer, a "Run Test" code lens above each test, and the ability to run/debug tests directly in the IDE. You also set up the `launch.json` for debugging by adding a Playwright configuration. The extension integrates with `playwright.config.ts` and shows project selectors, test statuses, and trace viewer access.

**Q: What is the difference between `npm install @playwright/test` and `npm install playwright`?**

`@playwright/test` is the full test runner package that includes the test framework, fixtures, reporters, and the Playwright library. It is the recommended package for end-to-end testing. `playwright` is the lower-level automation library without the test runner. Use `playwright` when you need browser automation outside of a testing context (e.g., web scraping, PDF generation) or when integrating with a different test framework like Jest or Mocha.

### Hard

**Q: How would you configure Playwright to run in a Docker container and what are the key considerations?**

Use `mcr.microsoft.com/playwright` as the base Docker image, which includes the Playwright browsers and their system dependencies. Your Dockerfile might look like:

```dockerfile
FROM mcr.microsoft.com/playwright:v1.45.0
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
CMD ["npx", "playwright", "test"]
```

Key considerations: the container must have the correct library dependencies for each browser, the `--ipc=host` flag is needed for Chromium to avoid shared memory issues, and you must set `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1` in the Node image before copying in the Playwright browsers from the base image. Also, use `-e CI=true` to ensure CI-specific config paths.

**Q: Explain the complete project structure for a large-scale Playwright test suite with multiple teams contributing.**

A large-scale Playwright project should have:

```
tests/
  auth/
    auth.setup.ts
  checkout/
    checkout.spec.ts
  dashboard/
    dashboard.spec.ts
  admin/
    admin.spec.ts
  helpers/
    mock-server.ts
    test-data.ts
  pages/
    login-page.ts
    checkout-page.ts
  fixtures.ts
playwright.config.ts
package.json
```

Each team owns a subdirectory under `tests/`. Shared helpers live in `helpers/`, page objects in `pages/`, and custom fixtures in `fixtures.ts`. The `playwright.config.ts` uses project filtering to include or exclude directories per run. The `testDir: './tests'` keeps all tests under one root, while projects use `testIgnore` or `testMatch` to filter. The `.auth/` directory (used for storageState) is added to `.gitignore`.

## TypeScript Basics

### Easy

**Q: Why does Playwright recommend TypeScript over JavaScript?**

TypeScript provides type safety — the Playwright API exposes complex locator chains, assertion matchers, and configuration types that benefit from autocompletion and compile-time error checking. The `@playwright/test` package ships with TypeScript definitions, so your IDE gives you inline documentation and type hints. TypeScript also catches common mistakes like calling methods with wrong argument types or accessing nonexistent properties.

**Q: What does the `import { test, expect } from "@playwright/test"` statement do?**

It imports the `test` function and `expect` assertion library from the Playwright test runner. `test` is used to define test cases (`test('title', async ({ page }) => {...})`) and `expect` provides the assertion matchers (`expect(element).toBeVisible()`). The import statement is the standard entry point for every Playwright test file.

**Q: What is the `Page` type in Playwright and how do you use it with TypeScript?**

`Page` is a TypeScript interface representing a single browser tab. It exposes methods for navigation (`goto`, `reload`, `goBack`), interaction (`click`, `fill`, `selectOption`), and information gathering (`url()`, `title()`, `content()`). In TypeScript, you type annotate parameters that receive a `Page` object, for example in page object constructors: `constructor(page: Page)`. The type is imported from `@playwright/test`.

### Medium

**Q: What is the benefit of using interfaces for page objects in TypeScript?**

Interfaces define the contract a page object must fulfill. They enforce type checking at compile time, making it clear what methods and properties each page object exposes. For example:

```typescript
interface ILoginPage {
  login(email: string, password: string): Promise<void>;
  readonly loginButton: Locator;
}
```

When you implement this interface in a class, TypeScript ensures your `LoginPage` has a `login` method and a `loginButton` property. This is valuable in large teams where multiple people contribute to page objects, as it prevents method signature drift.

**Q: How do you define a type for fixture objects in Playwright?**

You define a type for your custom fixtures using a TypeScript interface or type alias:

```typescript
import { test as base } from "@playwright/test";

type MyFixtures = {
  adminPage: AdminPage;
  apiClient: ApiClient;
};

const test = base.extend<MyFixtures>({
  adminPage: async ({ page }, use) => { ... },
  apiClient: async ({ request }, use) => { ... },
});
```

The type parameter `MyFixtures` tells TypeScript what fixture names and types are available. When you destructure `test.use({ adminPage })` in a test, TypeScript knows `adminPage` is of type `AdminPage`.

**Q: What is the `as` keyword used for in Playwright TypeScript tests?**

The `as` keyword is used for type assertions when you need to override TypeScript's inferred type. For example, if you parse a JSON response and TypeScript infers `any`, you use `as` to narrow the type: `const data = await response.json() as MyCustomType`. You might also use it when casting locator results: `const element = page.locator('.item').first() as Locator`. Use type assertions sparingly — they bypass TypeScript's type checking and can mask real errors.

### Hard

**Q: How do you handle TypeScript strict mode and its implications for Playwright tests?**

With `strict: true` in `tsconfig.json`, TypeScript enables `noImplicitAny`, `strictNullChecks`, `strictFunctionTypes`, and more. This means locator results must be typed explicitly, `null` and `undefined` values must be handled, and async functions must have proper return types. For example, `page.$('.selector')` returns `Locator | null` under strict mode, so you cannot call methods on it without a null check. Use the Playwright API's non-nullable alternatives like `page.locator()` or `page.getByRole()` instead.

**Q: Compare using classes vs functions for page objects in TypeScript and when each is preferable.**

Classes provide encapsulation with private fields, public methods, and constructor-based dependency injection. They work well for complex page objects with many locators and methods. Functions (factory functions or plain objects) are simpler and use less boilerplate, making them better for simple page objects. Classes allow you to define private locators with `#` private fields, while functions use closures for privacy. Classes are more familiar to developers coming from Java or C#; functions align better with functional programming styles. For a large test suite with complex pages, classes are generally preferred for their structure and maintainability.

## Page Object Model

### Easy

**Q: What is the Page Object Model pattern?**

The Page Object Model (POM) is a design pattern where each page of your application is represented by a class. The class encapsulates the page's locators, interaction methods, and assertions. Tests use these page objects instead of directly interacting with page selectors, which centralizes selector management and makes tests more readable and maintainable.

**Q: What is the difference between a page object and a component object?**

A page object represents an entire page (e.g., `LoginPage`, `DashboardPage`). A component object represents a reusable UI component that appears on multiple pages (e.g., `NavBar`, `Footer`, `ModalDialog`). Page objects may compose component objects. For example, `DashboardPage` may have a `NavBar` component that is shared with `SettingsPage`. Component objects follow the same pattern as page objects but are smaller in scope.

**Q: What should a page object's methods return?**

A page object method should return either the page object itself (for method chaining), a new page object representing the page you navigated to, or a value extracted from the page. For example, `loginPage.login('user', 'pass')` should return a `DashboardPage` instance. `cartPage.getItemCount()` should return a number. Returning the same page object enables fluent chaining: `loginPage.enterUsername('user').enterPassword('pass').clickLogin()`.

### Medium

**Q: How do you handle page objects that share common components like headers and footers?**

Use inheritance or composition. With inheritance, create a `BasePage` class with common locators and methods:

```typescript
class BasePage {
  constructor(protected page: Page) {}
  async logout() { ... }
  get navBar() { return new NavBar(this.page); }
}

class DashboardPage extends BasePage { ... }
```

With composition, each page object takes optional component objects:

```typescript
class DashboardPage {
  readonly navBar: NavBar;
  constructor(page: Page) {
    this.navBar = new NavBar(page);
  }
}
```

Composition is more flexible and avoids deep inheritance hierarchies.

**Q: How do you handle dynamic content in page objects (e.g., items loaded from an API)?**

Use locators that match the dynamic content pattern, not the static content. For example, use `page.getByRole('row')` instead of `page.getByText('Item #123')`. For data-dependent tests, use API calls to seed the data before navigating to the page, then use role-based or testid-based locators to find the rendered data. If the content structure varies, use methods that accept parameters: `searchResults.getItemByIndex(n)` or `searchResults.findItemByName(name)`.

**Q: Should page objects contain assertions? Why or why not?**

Page objects should not contain assertions for general navigation and interaction methods. Assertions belong in the test — the test knows what the expected state should be. However, page objects may contain verification methods that return page state for the test to assert on. For example, `checkoutPage.getOrderConfirmationText()` returns a string for the test to assert. The exception is for "guard" or "wait" methods that implicitly assert a state before proceeding, like `loginPage.waitForDashboard()` which asserts navigation completed before returning.

### Hard

**Q: Design a page object strategy for an application with 200+ pages that must be maintained by a team of 10 engineers.**

Use a layered architecture:

1. **BasePage** — contains shared components (navbar, footer, toast notifications, error modals) and utility methods (waitForLoading, getPageTitle).
2. **Domain pages** — grouped by feature domain (Checkout, Admin, UserProfile). Each domain has a base class for shared domain-specific patterns.
3. **Fragment objects** — reusable UI fragments that appear on multiple pages (DataTable, FilterPanel, Paginator).
4. **Factory pattern** — a `PageFactory` that creates page objects and handles navigation. Tests use `pageFactory.getDashboardPage()` rather than `new DashboardPage(page)`.
5. **Static analysis** — lint rules enforce page object standards: locators must be readonly, methods must return typed values, navigation must use baseURL.

This structure allows team members to work on different domains without merge conflicts and ensures consistent patterns across the suite.

**Q: How do you handle page objects for Single Page Applications (SPAs) where the URL does not change between views?**

SPAs use client-side routing, so the URL stays the same but the DOM changes. Page objects for SPAs should not rely on URL patterns for verification. Instead, use content-based waiting — wait for a specific element to appear:

```typescript
class SPAApp {
  constructor(private page: Page) {}

  async navigateToDashboard() {
    await this.page.getByRole('link', { name: 'Dashboard' }).click();
    await this.page.getByRole('heading', { name: 'Dashboard' }).waitFor();
  }
}
```

Use `waitForURL` only if the SPA changes the URL hash or path. Otherwise, wait for the distinct content of each view to appear. SPAs also require careful handling of loading spinners and skeleton screens — add a "wait for loading to complete" step before interacting with page content.

## Locators

### Easy

**Q: What is the difference between `page.locator()` and `page.getByRole()`?**

`page.locator()` accepts any CSS selector or Playwright-specific selector string. It is the most flexible option — you can use CSS, XPath, or text selectors. `page.getByRole()` specifically targets elements by their ARIA role and accessible name, which aligns with accessibility best practices. `getByRole` is preferred because it tests the same way users and assistive technologies interact with the page.

**Q: What does `getByTestId` look for and how do you configure the test ID attribute?**

`getByTestId` searches for elements with a `data-testid` attribute by default. You can configure the attribute name in `playwright.config.ts`:

```typescript
use: {
  testIdAttribute: 'data-test-id'
}
```

This is useful when your application uses a different convention like `data-test`, `data-qa`, or `data-automation-id`.

**Q: How do you locate an element by its visible text?**

Use `page.getByText('text content')`. This matches elements that contain the specified text, including child text nodes. For exact matching, use `page.getByText('text content', { exact: true })`, which matches only elements whose full text content equals the string.

### Medium

**Q: What happens when multiple elements match a locator?**

Playwright throws a "strict mode violation" error if a locator matches multiple elements and you try to perform an action on it. This is a safety feature to prevent accidental interactions with the wrong element. To handle multiple matches, use `.first()`, `.last()`, `.nth(index)`, or `.filter()` to narrow the selection. For example: `page.getByRole('button').filter({ hasText: 'Submit' }).click()`.

**Q: How do you chain locators to narrow down to a specific element within a section?**

Chain locators by calling one locator method after another:

```typescript
// Find the row in the table that contains "Playwright" and then click the edit button in that row.
const editButton = page
  .getByRole('table')
  .getByRole('row')
  .filter({ hasText: 'Playwright' })
  .getByRole('button', { name: 'Edit' });
```

You can also use the `locator()` method on a previously created locator to scope the search: `row.locator('button.edit')`.

**Q: What is the difference between `getByLabel` and `getByPlaceholder`?**

`getByLabel` finds elements by their associated `<label>` element using the `for` attribute or by wrapping the input. It is the accessible way to find form fields. `getByPlaceholder` finds elements by their `placeholder` attribute, which is placeholder text shown inside the input when it is empty. Placeholder text is not a label — it disappears when the user starts typing, making `getByPlaceholder` less reliable.

### Hard

**Q: Explain how Playwright's internal locator resolution algorithm works and how it handles dynamic content.**

Playwright's locator resolution uses a multi-step process. First, it resolves the locator to a list of matching elements in the DOM. If the locator returns multiple elements, it enforces strict mode (throws error). If it returns a single element, Playwright performs actionability checks: the element must be attached to the DOM, visible, stable (not animating), receives events (not obscured by another element), and enabled. Playwright retries these checks until the timeout expires or the checks pass. For dynamic content (single-page apps adding/removing elements), Playwright re-queries the locator on each action attempt, so it handles DOM updates automatically.

**Q: How do you create custom locator strategies in Playwright, and when would you need them?**

Use `page.locator().and()`, `.or()`, and custom selectors via `page.selectors.register()`. The `and` and `or` methods combine locators:

```typescript
// Button that is either "Save" or "Update"
const saveButton = page.getByRole('button', { name: 'Save' })
  .or(page.getByRole('button', { name: 'Update' }));

// Checkbox that is both visible and checked
const checkedBox = page.getByRole('checkbox')
  .and(page.locator(':checked'));
```

For truly custom strategies, use `selectors.register()` with a custom selector engine that matches elements by your own criteria (e.g., data attributes from a CMS). Custom strategies are rarely needed — the built-in ones handle nearly all cases.

## Actions

### Easy

**Q: What does `page.fill()` do that makes it different from `page.type()`?**

`page.fill()` clears any existing content in the input field and types the new value in a single operation. It is fast and reliable. `page.type()` types each character individually with a configurable delay (`type('text', { delay: 100 })`), which is slower and can trigger per-character event handlers. Use `fill` for most form inputs and `type` only when you need to test actual keyboard input (e.g., for live search suggestions that trigger on each keystroke).

**Q: How do you click an element in Playwright and what happens if the element is not visible?**

Use `element.click()` or `page.getByRole('button').click()`. Playwright automatically waits for the element to be visible, enabled, stable, and not obscured before clicking (the actionability checks). If the element does not become actionable within the timeout, the click throws a `TimeoutError`. If you want to bypass these checks, use `{ force: true }` — but this is almost never the right approach.

**Q: How do you press keyboard keys in Playwright?**

Use `page.keyboard.press('Enter')` for a single key press, `page.keyboard.down('Shift')` / `page.keyboard.up('Shift')` for modifier key combinations, and `page.keyboard.type('hello')` to type text. You can also use locator methods: `input.press('Enter')` or `input.press('Control+a')` (select all). Special key names include 'Enter', 'Tab', 'Escape', 'ArrowDown', 'Backspace', and 'Control'.

### Medium

**Q: What is the difference between `click()`, `dblclick()`, and `dispatchEvent('click')`?**

`click()` performs a full actionability check — waits for element visibility, stability, and enabled state, then fires `mousedown`, `mouseup`, and `click` events with the correct coordinates. `dblclick()` is the same but fires two sequences with a short delay between. `dispatchEvent('click')` bypasses all actionability checks and fires a synthetic `click` event directly on the element, even if it is invisible, disabled, or obscured. Use `dispatchEvent` only for edge cases where the element is intentionally non-interactive but you need to trigger its event handler.

**Q: How do you upload a file in Playwright?**

Use `page.getByLabel('Choose file').setInputFiles('path/to/file.pdf')` or `locator.setInputFiles('path/to/file.pdf')`. For multiple files: `setInputFiles(['file1.pdf', 'file2.pdf'])`. You can also pass a `FilePayload` object with name, mimeType, and buffer for dynamically generated files:

```typescript
await page.getByLabel('Upload').setInputFiles({
  name: 'test.txt',
  mimeType: 'text/plain',
  buffer: Buffer.from('test content'),
});
```

**Q: How do you handle drag-and-drop in Playwright?**

Use `page.locator('#source').dragTo(page.locator('#target'))`. Playwright performs a native HTML5 drag-and-drop by dispatching `dragstart`, `dragenter`, `dragover`, `drop`, and `dragend` events. If your application uses a custom drag implementation (like React DnD or a canvas-based drag), you may need to use `page.mouse.move()`, `page.mouse.down()`, and `page.mouse.up()` to simulate mouse-based dragging step by step.

### Hard

**Q: Explain how Playwright implements its actionability checks internally and what happens during a `click()` call.**

When you call `element.click()`, Playwright executes the following sequence internally:

1. **Wait for element** — Repeatedly query the locator until it returns a matching element (or timeout).
2. **Scroll into view** — `element.scrollIntoViewIfNeeded()` is called to make the element visible in the viewport.
3. **Actionability checks**:
   - **Attached**: The element exists in the DOM (not detached).
   - **Visible**: The element is not hidden (`display: none`, `visibility: hidden`, zero size, or opacity: 0).
   - **Stable**: The element's bounding box has not changed for at least 500ms (prevents clicking during animations).
   - **Enabled**: The element is not disabled (`<button disabled>` or `aria-disabled="true"`).
   - **Receives events**: Hit-test at the click point returns the element (not obscured by a covering element).
4. **Perform click** — Fire pointer events (`pointerdown`, `mousedown`, `pointerup`, `mouseup`, `click`) at the element's center point by default.
5. **Wait for navigation** — If the click triggers a navigation, Playwright waits for the new page to load.

All checks are performed in the browser process itself, not via round-trips to the Node.js client.

**Q: How would you test a file download in Playwright?**

Use `page.waitForEvent('download')` to capture the download event:

```typescript
const [download] = await Promise.all([
  page.waitForEvent('download'),  // Start waiting for the download
  page.getByText('Download Report').click(),  // Trigger the download
]);

// Wait for the download to complete.
const filePath = await download.path();

// Assert the downloaded file.
const content = fs.readFileSync(filePath, 'utf-8');
expect(content).toContain('expected data');

// Save to a permanent location (download.path() is temporary).
await download.saveAs('./downloads/report.csv');
```

Always use `Promise.all` to handle the race condition between clicking and the download starting.

## Assertions

### Easy

**Q: What is the difference between `toBeVisible` and `toBeInViewport`?**

`toBeVisible` checks that the element exists in the DOM, is not hidden (`display: none`, `visibility: hidden`, or `opacity: 0`), and has a non-zero bounding box. `toBeInViewport` checks that the element is visible AND its bounding box intersects the current viewport. An element can be visible but off-screen (scrolled down), which would pass `toBeVisible` but fail `toBeInViewport`.

**Q: What does `toHaveText('value')` assert?**

It asserts that the element's text content equals the exact string `'value'`, including whitespace and child text nodes. For partial matching, use `toContainText('value')`. Both are auto-retrying — they keep checking until the timeout or the condition is met.

**Q: What happens when an assertion fails?**

When `expect(element).toBeVisible()` fails, it throws an `AssertionError` with a detailed message showing the expected condition, the actual state, and the locator that was used. The test is marked as failed. If `retries` is configured, Playwright retries the entire test. Trace and screenshot options (if enabled) capture the page state at the time of failure for debugging.

### Medium

**Q: What is the difference between `toHaveURL` and `toContainURL`?**

`expect(page).toHaveURL('https://example.com/dashboard')` asserts the full URL exactly matches the string or regex. `expect(page).toHaveURL(/dashboard/)` is the common pattern for partial matching — technically there is no `toContainURL` matcher; you use a regex in `toHaveURL` for partial matching. `expect(page).toHaveURL(/dashboard/)` passes if the URL contains "dashboard" anywhere.

**Q: How does Playwright's `expect` auto-retry mechanism work?**

Playwright's `expect` uses a polling loop. When you call `expect(locator).toBeVisible()`, Playwright repeatedly checks the condition (is the element visible?) every few hundred milliseconds until either the condition passes or the timeout expires (default 5 seconds, configurable with `expect({ timeout: 10000 })`). This eliminates the need for manual `waitFor` calls in most cases. The polling happens in the browser process, so it is efficient and does not block the Node.js event loop.

**Q: What is the difference between soft assertions and hard assertions?**

Hard assertions (`expect(value).toBe(x)`) immediately fail the test when the condition is not met. Soft assertions (`expect.soft(value).toBe(x)`) record the failure but continue executing the test. After the test finishes, Playwright reports all collected soft assertion failures. Soft assertions are useful when you want to check multiple independent conditions in a single test run, such as verifying all fields in a form render correctly.

### Hard

**Q: How would you write a custom assertion matcher in Playwright?**

Extend `expect` using `expect.extend()`:

```typescript
import { expect as baseExpect } from "@playwright/test";

const customExpect = baseExpect.extend({
  async toBeWithinRange(received: number, floor: number, ceiling: number) {
    const pass = received >= floor && received <= ceiling;
    return {
      message: () => `expected ${received} to be within range [${floor}, ${ceiling}]`,
      pass,
    };
  },
});
```

For locator-based custom matchers, implement the `LocatorAssertions` interface. Custom matchers are useful for domain-specific checks like "has valid CSS class combination" or "is sorted correctly."

**Q: Explain the relationship between `expect.poll` and global auto-retrying assertions, and when you would use `poll` instead.**

`expect.poll` is a lower-level retry mechanism that repeatedly evaluates a function until it passes or times out:

```typescript
await expect.poll(async () => {
  const response = await page.request.get('/api/status');
  return response.status();
}, { timeout: 10000 }).toBe(200);
```

Unlike locator-based assertions that auto-retry because Playwright knows the element is being tracked, `expect.poll` is for non-locator conditions like API responses, page variable values, or network states. Use `expect.poll` when you need to retry a custom condition that is not tied to a specific locator.

## Waiting

### Easy

**Q: Does Playwright require explicit `wait` calls like `Thread.sleep` in other frameworks?**

No. Playwright's auto-waiting mechanism handles most waiting scenarios automatically. When you call `click()`, Playwright waits for the element to be visible, stable, and enabled. When you call `goto()`, it waits for the page to load. Most `expect` assertions auto-retry until the condition is met. You rarely need explicit waits, unlike Selenium where implicit/explicit waits are a daily necessity.

**Q: What does `page.waitForSelector()` do?**

`page.waitForSelector('#my-element')` pauses test execution until an element matching the CSS selector appears in the DOM. You can specify the element state: `{ state: 'attached' }` (element exists in DOM, may be hidden), `{ state: 'visible' }` (element is visible), `{ state: 'hidden' }` (element disappears), or `{ state: 'detached' }` (element leaves DOM). With auto-waiting assertions, you rarely need `waitForSelector` — use `expect(locator).toBeVisible()` instead.

**Q: What is a reasonable default timeout for Playwright operations?**

The default timeout for assertions is 5 seconds. The default timeout for actions (click, fill) is 30 seconds. The default test timeout is 30 seconds. These are reasonable for most applications. For slow-loading pages, increase the test timeout to 60 or 120 seconds. For flaky assertions on slow CI, increase the assertion timeout to 10 seconds.

### Medium

**Q: When would you use `page.waitForTimeout()` instead of a proper wait?**

`page.waitForTimeout(1000)` pauses execution for exactly 1 second. You should almost never use it. It introduces flakiness (the needed condition may take 900ms or 1100ms) and slows down tests. Use it only in two scenarios: debugging (temporarily to inspect page state) and testing animations where you must wait a specific duration for the animation to complete. Never use `waitForTimeout` to wait for a DOM element — use `waitForSelector`, `toBeVisible`, or `waitForResponse` instead.

**Q: How do you wait for a network request to complete before continuing?**

Use `page.waitForResponse()`:

```typescript
// Trigger an action that causes a network request.
const [response] = await Promise.all([
  page.waitForResponse(resp => resp.url().includes('/api/data') && resp.status() === 200),
  page.getByRole('button', { name: 'Load Data' }).click(),
]);

// Now the API response is available.
const data = await response.json();
```

Use `Promise.all` to avoid race conditions. The `waitForResponse` must be registered before the action that triggers the request.

**Q: What does `page.waitForLoadState('networkidle')` do and when should you use it?**

`waitForLoadState('networkidle')` waits until there are no more than 0 active network connections for at least 500ms. It is useful for single-page applications that make background API calls after the initial page load. However, it can be slow on pages with long-polling or WebSocket connections that never truly become idle. Prefer waiting for specific elements or API responses rather than the generic `networkidle` state.

### Hard

**Q: Explain Playwright's auto-waiting algorithm and how it decides when an element is "ready" to be interacted with.**

Playwright's auto-waiting algorithm performs four actionability checks in sequence, all within the browser process:

1. **Attached**: The element's parent node is not `null` and the element is reachable in the DOM tree.
2. **Visible**: The element has `offsetWidth > 0` and `offsetHeight > 0`, `opacity > 0`, `visibility !== 'hidden'`, `display !== 'none'`, and no ancestor hides it. The element must also have a computed style that does not collapse it.
3. **Stable**: The element's bounding box has not changed for two consecutive animation frames (~500ms). This prevents clicking during CSS transitions, JavaScript animations, or loading spinners.
4. **Receives events**: A hit-test at the element's center point returns the element itself or a descendant. If another element (like a modal overlay or a fixed-position banner) covers the element, Playwright waits for the covering element to move or disappear.

These checks repeat every ~100ms until they all pass or the timeout expires. The checks run entirely in the browser's rendering pipeline, so they accurately reflect what a user would experience.

**Q: How do you handle pages with WebSocket connections or Server-Sent Events that never reach `networkidle`?**

Pages with persistent WebSocket connections never reach `networkidle` because there is always an active connection. For such pages:

- Do not use `waitForLoadState('networkidle')` — it will time out.
- Use `page.goto(url, { waitUntil: 'domcontentloaded' })` to stop waiting after the HTML is parsed, ignoring ongoing WebSocket traffic.
- Wait for specific elements: `await page.getByText('Dashboard loaded').waitFor()`.
- Use `page.waitForURL()` to wait for URL changes after navigation.
- Use `page.waitForResponse()` for specific API calls.

The general rule is: wait for what matters (specific content or API responses), not for generic page states.

## API Testing

### Easy

**Q: How do you make a GET request with Playwright?**

Use `page.request.get()`:

```typescript
const response = await page.request.get('https://api.example.com/users');
expect(response.status()).toBe(200);
const data = await response.json();
expect(data.length).toBeGreaterThan(0);
```

`page.request` provides a full HTTP client that shares cookies and storage with the browser page but can also be used independently in tests that do not have a page.

**Q: What HTTP methods does Playwright's API testing support?**

Playwright's `page.request` object supports all standard HTTP methods: `get()`, `post()`, `put()`, `patch()`, `delete()`, `head()`, and `options()`. Each method accepts the URL, an optional `params` object for query parameters, and an optional `data` or `form` object for the request body.

**Q: How do you send JSON data in a POST request with Playwright?**

```typescript
const response = await page.request.post('https://api.example.com/users', {
  data: {
    name: 'Jane Doe',
    email: 'jane@example.com',
  },
});
expect(response.status()).toBe(201);
```

The `data` field automatically sets the `Content-Type: application/json` header and serializes the object.

### Medium

**Q: What is the difference between `page.request` and `request` fixture in Playwright?**

`page.request` is scoped to a specific page and shares its cookies, storage, and authentication state. It is useful when your API calls need to be authenticated with the same session as the page. The `request` fixture (available in the test runner) is a standalone API client that does not share any browser state. It is useful for requests that should be independent of the browser's session, such as seeding test data before logging in. Choose `page.request` when the API call depends on the current user session; choose `request` when setting up preconditions.

**Q: How do you set custom headers in a Playwright API request?**

Pass the `headers` option:

```typescript
const response = await page.request.get('https://api.example.com/users', {
  headers: {
    'Authorization': 'Bearer token123',
    'X-API-Key': 'my-api-key',
  },
});
```

You can also set default headers for all requests using `page.context().setExtraHTTPHeaders()` or by creating a custom `APIRequestContext` with pre-configured headers.

**Q: How do you handle paginated API responses in Playwright?**

Iterate through pages by checking for a `next` link or using query parameters:

```typescript
let pageNum = 1;
let allUsers: User[] = [];
let hasMore = true;

while (hasMore) {
  const response = await page.request.get('https://api.example.com/users', {
    params: { page: pageNum, limit: 100 },
  });
  const data = await response.json();
  allUsers = allUsers.concat(data.users);
  hasMore = data.hasNextPage;
  pageNum++;
}
```

This pattern works both for REST APIs that return `hasNextPage` and for those that return a `next` URL link.

### Hard

**Q: How can you use `page.route()` to intercept and modify API responses for testing edge cases?**

`page.route()` intercepts network requests and lets you modify the response:

```typescript
// Intercept the users API and return an empty array to test the "no users" state.

  await route.fulfill({
    status: 200,

```

This is invaluable for testing error states, empty states, and rare edge cases without needing to manipulate the real backend.

```typescript
  await expect(page.getByText('Test User')).toBeVisible();
});
```

This pattern reuses the same authentication across both API and UI tests, ensuring consistency.

## Self-Healing Locators

### Easy

**Q: What does "self-healing locator" mean in the context of Playwright?**

A self-healing locator is a locator strategy that is resilient to minor changes in the DOM structure, such as CSS class renames, attribute changes, or DOM restructuring. Playwright's built-in locators (especially `getByRole`, `getByLabel`, `getByTestId`) are inherently self-healing because they rely on semantic and accessible attributes that are less likely to change than CSS classes or DOM structure.

**Q: Are self-healing locators built into Playwright or do you need a plugin?**

Playwright's locator API is inherently designed for resilience. `getByRole`, `getByLabel`, `getByText`, and `getByTestId` are all stable across DOM changes because they use semantic attributes rather than DOM structure. There is no separate "self-healing" plugin needed — using the right locator strategy is the self-healing mechanism.

**Q: Why is `getByTestId` considered a self-healing locator?**

`getByTestId` uses a dedicated `data-testid` attribute that is intentionally placed on elements for testing. This attribute remains stable across CSS changes, DOM restructuring, and component renames. As long as the developer keeps the `data-testid` attribute on the element, the locator will never break. This makes it the most self-healing of all locators — it is immune to all DOM changes except the removal of the attribute itself.

### Medium

**Q: How do you implement a fallback locator strategy for elements whose selectors change frequently?**

Use a locator chaining approach with `or()` to provide fallback selectors:

```typescript
// Try the test ID first, then role, then text as a fallback.
const submitButton = page.getByTestId('submit-btn')
  .or(page.getByRole('button', { name: 'Submit' }))
  .or(page.getByText('Submit'));

await submitButton.click();
```

The `or()` method returns the first locator that matches. If none match, it throws a timeout error. This gives you multiple chances to find the element.

**Q: What are the limitations of self-healing locators and when do they fail?**

Self-healing locators fail when the application's semantic structure changes fundamentally — when a button becomes a link, when an input's label text changes, or when the developer removes `data-testid` attributes. They also fail when the page uses non-semantic HTML (a `<div>` with an `onclick` handler instead of a `<button>`) because `getByRole` cannot infer roles from non-semantic elements. In such cases, developers need to update locators or add proper ARIA roles.

**Q: How do you handle locators that need to work across multiple versions of a component?**

Version the `data-testid` or add feature detection:

```typescript
// Detect which version of the component is rendered.
const v2Submit = page.getByTestId('submit-btn-v2');
const v1Submit = page.getByTestId('submit-btn-v1');

// Wait for whichever version appears.
const submitBtn = await Promise.race([
  v2Submit.waitFor({ timeout: 5000 }).then(() => v2Submit),
  v1Submit.waitFor({ timeout: 5000 }).then(() => v1Submit),
]);

await submitBtn.click();
```

Alternatively, use a single `data-testid` that stays the same across versions and only changes the underlying implementation.

### Hard

**Q: Design a custom locator strategy that automatically tries alternative selectors if the primary one fails.**

Implement a wrapper around Playwright's locator that uses a retry-with-fallback pattern:

```typescript
class ResilientLocator {
  private strategies: (() => Locator)[];

  constructor(page: Page, selectors: { testId?: string; role?: { role: string; name?: string }; text?: string }) {
    this.strategies = [];
    if (selectors.testId) {
      this.strategies.push(() => page.getByTestId(selectors.testId!));
    }
    if (selectors.role) {
      this.strategies.push(() => page.getByRole(selectors.role.role as any, { name: selectors.role.name }));
    }
    if (selectors.text) {
      this.strategies.push(() => page.getByText(selectors.text!));
    }
  }

  async click(options?: { timeout?: number }): Promise<void> {
    const timeout = options?.timeout ?? 5000;
    const start = Date.now();

    for (const strategy of this.strategies) {
      const remaining = timeout - (Date.now() - start);
      if (remaining <= 0) break;

      try {
        const locator = strategy();
        await locator.click({ timeout: remaining });
        return;  // Success
      } catch {
        // Fall through to next strategy.
      }
    }

    throw new Error(`All locator strategies failed within ${timeout}ms`);
  }
}
```

This pattern tries each strategy in order, falling through on failure. Log the successful strategy to track which one works in production.

**Q: Compare Playwright's locator resilience to Selenium's and Cypress's approaches.**

Selenium relies primarily on CSS selectors and XPath, both of which are brittle — a CSS class rename or a wrappers `<div>` insertion breaks them. Selenium has no built-in self-healing and requires custom libraries or frameworks (like Healenium or Selenium's Fluent Wait) to add resilience. Cypress uses CSS selectors by default but encourages `data-cy` attributes and offers `contains()`, `getByRole`-like selectors through its `cy.findByRole` (from `@testing-library/cypress`). Playwright is the only framework that bakes semantic locators (`getByRole`, `getByLabel`, `getByTestId`) into the core API with first-class support, making resilience the default rather than an add-on.

## Visual Testing

### Easy

**Q: How do you take a screenshot in Playwright?**

Use `page.screenshot()`:

```typescript
await page.screenshot({ path: 'screenshot.png' });
await page.screenshot({ path: 'fullpage.png', fullPage: true }); // Full scrollable page.
await element.screenshot({ path: 'element.png' }); // Screenshot of a specific element.
```

**Q: What does `toHaveScreenshot()` do?**

`expect(page).toHaveScreenshot('baseline.png')` captures a screenshot of the page and compares it pixel-by-pixel to a stored baseline image. On the first run, it creates the baseline. On subsequent runs, it compares and generates a diff image if they differ. The diff image highlights changed pixels in pink, along with the actual and expected images.

**Q: How do you update visual baselines when the UI intentionally changes?**

Run the tests with the `--update-snapshots` flag:

```bash
npx playwright test --update-snapshots
```

This overwrites all baseline images with the current screenshots. You commit the updated baselines to version control. You can also update individual snapshots by passing a test filter like `--grep "test name"`.

### Medium

**Q: How do you handle visual differences caused by different operating systems or rendering engines?**

Use project-specific snapshot directories or snapshot suffixes. In `playwright.config.ts`:

```typescript
snapshotPathTemplate: '{testDir}/__snapshots__/{testFilePath}/{arg}-{projectName}{ext}',
```

This stores separate baselines per project (chromium, firefox, webkit). The `-` flag ensures they do not collide. When a new browser is added, run `--update-snapshots` to generate its baselines.

**Q: What options does `toHaveScreenshot` accept to reduce false positives?**

Common options include:

```typescript
await expect(page).toHaveScreenshot({
  maxDiffPixels: 100,           // Allow up to 100 different pixels.
  maxDiffPixelRatio: 0.01,     // Allow 1% of pixels to differ.
  threshold: 0.2,              // Color threshold (0 = exact, 1 = any).
  animations: 'disabled',      // Disable animations for consistent screenshots.
  caret: 'hide',               // Hide the text cursor.
  mask: [page.locator('.timestamp')],  // Mask dynamic content regions.
  maskColor: '#FF0000',        // Color for masked regions.
});
```

These options help you ignore tiny rendering differences (anti-aliasing), dynamic content (timestamps, ads), and animated elements.

**Q: How do you take a screenshot of a specific element rather than the full page?**

Use `element.screenshot()`:

```typescript
const header = page.locator('header');
await header.screenshot({ path: 'header.png' });
```

This captures only the bounding box of the selected element, excluding everything else on the page. This is useful for testing visual regressions in specific components without being affected by other parts of the page.

### Hard

**Q: How would you implement a visual regression pipeline that supports multiple environments (dev, staging, production)?**

Create environment-specific snapshot directories in the config:

```typescript
snapshotPathTemplate: '{testDir}/__snapshots__/{env}/{testFilePath}/{arg}-{projectName}{ext}',
```

Set the `env` value from an environment variable in CI:

```bash
# In CI, run tests with a specific environment.
ENV=staging npx playwright test --update-snapshots
ENV=production npx playwright test
```

Store all environment baselines in version control. Add a CI job that runs on each environment deploy and uploads diff reports. Use a custom reporter to aggregate visual failures across environments and notify the team when production differs from staging.

**Q: Explain how Playwright handles anti-aliased text rendering and font differences across machines.**

Playwright cannot eliminate anti-aliasing entirely — it is a rendering engine feature. However, it provides workarounds:

1. **`threshold` option**: The color comparison threshold (default 0.2) allows small RGB differences caused by anti-aliasing.
2. **`animations: 'disabled'`**: Disables CSS animations and transitions, but does not affect font rendering.
3. **Consistent rendering in Docker**: Run tests in `mcr.microsoft.com/playwright` Docker images, which have fixed font and rendering configurations.
4. **`maxDiffPixelRatio`**: Set to 0.001 (0.1%) to tolerate tiny anti-aliasing differences without allowing real regressions.
5. **Font matching**: Install the same fonts on all machines. Docker images include standard web fonts.

If antialiasing still causes issues, combine multiple tolerance options for a pragmatic balance between catching real bugs and avoiding false positives.

## Trace Viewer

### Easy

**Q: What is the Playwright Trace Viewer?**

The Trace Viewer is a graphical tool that displays a recorded timeline of a Playwright test run. It shows DOM snapshots at each action, console messages, network requests, and test steps. It is invaluable for debugging failed tests — you can scrub through the timeline to see exactly what the page looked like at each moment.

**Q: How do you enable trace capture in Playwright?**

Configure in `playwright.config.ts`:

```typescript
use: {
  trace: 'on-first-retry',   // Capture trace on the first retry of a failed test.
  // Other options: 'on', 'off', 'retain-on-failure', 'on-all-retries'
}
```

- `'on'`: Capture a trace for every test.
- `'off'`: Never capture traces (default).
- `'on-first-retry'`: Capture trace only when a test is retried (useful for CI).
- `'retain-on-failure'`: Capture trace on every test run but only keep it if the test failed.
- `'on-all-retries'`: Capture trace on every retry attempt.

**Q: How do you open a trace file for viewing?**

```bash
npx playwright show-trace path/to/trace.zip
```

This opens an interactive web-based viewer in your browser. You can also drag and drop a `trace.zip` file onto the Trace Viewer hosted at `https://trace.playwright.dev`.

### Medium

**Q: What information does the Trace Viewer show for each test step?**

For each action (click, fill, navigation, etc.), the Trace Viewer shows:

- **DOM snapshot**: The page's HTML at the moment before and after the action.
- **Action details**: The locator used, the element's bounding box, and the action parameters.
- **Network activity**: All network requests and responses that occurred, including headers, status codes, and timing.
- **Console messages**: `console.log`, `console.error`, and other console output.
- **Source map**: The exact line of the test code that performed the action, with file and line number.
- **Timeline**: A visual timeline showing test steps, network requests, and page events.

**Q: How do you capture a trace for a specific test without changing the config?**

Override the trace setting at the test or describe level:

```typescript
test.describe('flaky-suite', () => {
  test.use({ trace: 'on' });  // Enable traces for all tests in this describe.

  test('flaky test', async ({ page }) => {
    // This test always captures a trace.
  });
});
```

Or for a single test:

```typescript
test('debug this test', async ({ page }, testInfo) => {
  // Manually start trace capture.
  await page.context().tracing.start({ screenshots: true, snapshots: true });

  // ... test steps ...

  // Stop and save the trace.
  await page.context().tracing.stop({ path: './traces/debug-trace.zip' });
});
```

**Q: What is the difference between screenshots and snapshots in a trace?**

Screenshots capture the visual rendering of the page at a point in time (a PNG image). Snapshots capture the DOM structure (the HTML). Screenshots are useful for visual inspection — you can see what the page looked like. Snapshots are useful for DOM inspection — you can see the HTML, CSS classes, attributes, and the full element tree. The Trace Viewer shows both: screenshots on the timeline and snapshots in the "DOM snapshot" panel.

### Hard

**Q: How would you build a CI pipeline that automatically uploads traces from failed tests and makes them accessible via a shareable link?**

```yaml

  if: failure()
  uses: actions/upload-artifact@v4
  with:
    name: traces-${{ matrix.shard }}
    path: test-results/**/trace.zip

```

Then, set up a custom script or use a service like `trace.playwright.dev` to generate shareable links. You can also integrate with Slack or email notification:

```typescript

```

For ephemeral CI environments, upload traces to cloud storage (S3, GCS) with a short-lived URL that is posted to your team chat.

**Q: Explain the performance impact of capturing traces and how to minimize it in production CI.**

Traces capture screenshots (a PNG per action), DOM snapshots (serialized HTML), and network request bodies. This can increase test runtime by 2-3x and storage by 10-100MB per test. To minimize impact:

1. Use `trace: 'on-first-retry'` instead of `trace: 'on'` — only capture when a test is retried.
2. In your config, set `screenshots: false` in the tracing options if you only need DOM snapshots.
3. For CI PR checks, disable traces entirely — run them only on main/scheduled runs.
4. Use `testInfo.attachments` to selectively attach only the trace from the last retry.
5. Set an artifact retention policy (3-7 days) to avoid filling storage.

The trade-off is debugging speed vs CI speed. For most teams, `on-first-retry` provides the best balance.

## Parallel Execution

### Easy

**Q: How do you run tests in parallel with Playwright?**

By default, Playwright runs tests in parallel within a file and across files. The `workers` config controls the number of parallel worker processes:

```typescript
// playwright.config.ts
workers: 4,  // Run 4 test workers in parallel.
```

You can also set it to a percentage: `workers: '50%'` (half of CPU cores). Each worker runs in its own process with its own browser instance.

**Q: What is the difference between `fullyParallel` and `workers` in the config?**

`workers` controls how many worker processes run at the same time (parallelism level). `fullyParallel: true` (default) tells Playwright to run all tests within a single file in parallel. When set to `false`, tests in a single file run sequentially but multiple files can still run across workers. Use `fullyParallel: false` when tests in the same file share state (e.g., they are in the same describe block with shared variables).

**Q: How do you run tests sequentially within a single file?**

Use `test.describe.configure({ mode: 'serial' })`:

```typescript
test.describe('sequential tests', () => {
  test.describe.configure({ mode: 'serial' });

  let sharedState: number;

  test('first', async ({ page }) => {
    sharedState = 42;
  });

  test('second', async ({ page }) => {
    expect(sharedState).toBe(42);  // Safe because tests run sequentially.
  });
});
```

Serial mode ensures the tests run one after the other in the same worker.

### Medium

**Q: What are the risks of shared state in parallel tests and how do you mitigate them?**

Shared state (files, database records, global variables) is the primary risk in parallel tests. If two tests write to the same file or modify the same database record, they interfere with each other. Mitigations:

- **Isolation**: Each test creates its own data and cleans up after itself.
- **Local scoping**: Use `test.describe()` blocks with `beforeEach` for setup, not `beforeAll`.
- **Unique test data**: Use random suffixes or UUIDs in test data names.
- **File isolation**: Write test output to unique temporary directories.
- **Database**: Use transactions that roll back after each test, or use separate schemas per worker.

**Q: How does Playwright's sharding work and when would you use it?**

Sharding splits your test suite across multiple machines. Each machine runs a subset of tests:

```bash
npx playwright test --shard=1/4  # Machine 1 runs 25% of tests
npx playwright test --shard=2/4  # Machine 2 runs 25% of tests
npx playwright test --shard=3/4  # Machine 3 runs 25% of tests
npx playwright test --shard=4/4  # Machine 4 runs 25% of tests
```

Use sharding when your test suite is too large for a single CI runner (e.g., 1000+ tests taking 30+ minutes). Each shard runs on a separate CI machine, reducing total wall-clock time.

**Q: How do you configure test retries and how do they interact with parallel execution?**

Set `retries` in the config:

```typescript
retries: 2,  // Retry each failed test up to 2 times.
```

When a test fails, it is retried in the same worker (same process). Retries do NOT increase parallelism — a retried test occupies a worker that could otherwise run a new test. Playwright uses a "retry fail" heuristic: if a test passes on retry, the overall result is "flaky" (passed after retry). If it fails on all retries, the result is "failed". Retries are good for flaky tests but increase total test execution time.

### Hard

**Q: Explain how Playwright's worker allocation algorithm distributes tests and how it handles test file size imbalance.**

Playwright uses a greedy load-balancing algorithm. When workers start, each worker picks up one test file at a time. When a worker finishes a file, it picks up another pending file. This naturally balances load — a worker that picks a small file finishes quickly and picks up another, while a worker stuck on a large file continues without interruption. This is more efficient than pre-splitting tests (where a large file could slow down one shard). For extremely large test files (100+ tests), consider splitting into smaller files or using `test.describe.configure({ mode: 'parallel' })` to parallelize within the file.

**Q: How do you troubleshoot flaky tests that only fail in parallel execution?**

Flaky tests in parallel execution often indicate a shared-state issue. Use these debugging steps:

1. **Run with 1 worker**: If the test always passes with 1 worker but fails with multiple, it is a shared-state issue.
2. **Check for race conditions**: Look for tests that write to files, databases, or localStorage without cleanup.
3. **Enable traces**: Set `trace: 'on-first-retry'` and inspect the trace to see what other pages were doing.
4. **Add logging**: Use `testInfo.attachments` to attach console output and page state at key moments.
5. **Isolate with `only`**: Run the failing test in isolation to confirm it passes alone.
6. **Check test order dependencies**: If test A sets up state that test B depends on, they must run sequentially or use `beforeEach` for independent setup.

The root cause is almost always tests that mutate shared resources without proper isolation.

## Authentication

### Easy

**Q: How do you log in to a website in a Playwright test?**

```typescript
await page.goto('https://example.com/login');
await page.getByLabel('Email').fill('user@example.com');
await page.getByLabel('Password').fill('password123');
await page.getByRole('button', { name: 'Log In' }).click();
await expect(page.getByText('Welcome, User!')).toBeVisible();
```

Playwright automatically handles cookies and session storage — after login, subsequent requests within the same context are authenticated.

**Q: What is `storageState` in Playwright?**

`storageState` is a JSON-serialized snapshot of a browser context's cookies, localStorage, and sessionStorage. You can save it after login and reuse it in other tests to skip the login step. This is a critical performance optimization — every test does not need to re-login.

**Q: How do you save and load authentication state?**

Save after logging in:

```typescript
await page.context().storageState({ path: '.auth/user.json' });
```

Load in the config:

```typescript
use: {
  storageState: '.auth/user.json',
}
```

All tests that use this config start with the saved authentication state, exactly as if they had just logged in.

### Medium

**Q: What are setup projects and how do they work with authentication?**

Setup projects are special Playwright projects that run before your main test projects. They typically perform login operations and save storageState files. The main test projects declare a dependency on the setup project, ensuring the state file exists before any test runs.

**Q: How do you handle multi-factor authentication (MFA) in Playwright tests?**

Strategies for MFA:

1. **Bypass MFA in test environments**: Configure your application to skip MFA for test accounts.
2. **Use an API-based login**: Log in via API (which may not require MFA) and then set cookies on the page.
3. **Persist storageState**: Perform MFA once manually, save the storageState, and reuse it across test runs.
4. **Use TOTP secret**: If your app uses TOTP (time-based one-time passwords), use a library like `otplib` to generate codes in your setup:

```typescript
import { authenticator } from 'otplib';
authenticator.options = { step: 30 };
const token = authenticator.generate(TOTP_SECRET);
await page.getByLabel('Authentication Code').fill(token);
```

**Q: How do you test with multiple authenticated users simultaneously?**

Create multiple browser contexts, each with its own authentication state:

```typescript
test('admin and user interact', async ({ browser }) => {
  const adminContext = await browser.newContext({ storageState: '.auth/admin.json' });
  const userContext = await browser.newContext({ storageState: '.auth/user.json' });

  const adminPage = await adminContext.newPage();
  const userPage = await userContext.newPage();

  // adminPage is logged in as admin.
  // userPage is logged in as user.
  // They have independent sessions.
});
```

The `browser` fixture gives you the browser instance without the default context, letting you create your own.

### Hard

**Q: Design a token refresh strategy for long-running test suites.**

Long-running test suites may experience token expiry mid-suite. Mitigation strategies:

1. **Short suites (< 1 hour)**: Use long-lived tokens in storageState. No refresh needed.
2. **Long suites (1-4 hours)**: Add a `beforeEach` hook that checks if the token is still valid:

```typescript
test.beforeEach(async ({ page }) => {
  const isValid = await page.evaluate(async () => {
    const token = localStorage.getItem('authToken');
    if (!token) return false;
    const payload = JSON.parse(atob(token.split('.')[1]));
    return payload.exp > Date.now() / 1000 + 60;  // At least 60 seconds remaining.
  });
  if (!isValid) {
    // Refresh token or re-login.
    await refreshToken(page);
  }
});
```

3. **Using API refresh flows**: Implement a custom fixture that periodically refreshes the token using the app's refresh-token endpoint.

```typescript
const test = base.extend({
  autoRefreshToken: [async ({ page }, use) => {
    const interval = setInterval(async () => {
      await page.evaluate(async () => {
        const refreshToken = localStorage.getItem('refreshToken');
        const response = await fetch('/api/refresh', {
          method: 'POST',
          body: JSON.stringify({ refreshToken }),
        });
        const { token } = await response.json();
        localStorage.setItem('authToken', token);
      });
    }, 10 * 60 * 1000);  // Refresh every 10 minutes.

    await use();
    clearInterval(interval);
  }, { auto: true }],
});
```

**Q: How do you handle SSO (Single Sign-On) authentication in Playwright?**

SSO typically involves redirecting to an identity provider (Okta, Auth0, Azure AD) and back. Strategies:

1. **Persist the session**: Log in once (interactively or via API) and save storageState. Reuse for all subsequent runs.
2. **Use the SSO provider's test API**: Auth0, Okta, and Azure AD have APIs for programmatic login (OAuth client credentials flow). Use `page.request.post()` to get tokens directly.
3. **Bypass SSO in test environments**: Add a "magic link" or test-only login endpoint that skips the SSO redirect.
4. **Mock the SSO response**: Use `page.route()` to intercept the SSO callback URL and inject a mock token.

```typescript
await page.route('**/oauth/callback*', async (route) => {
  // Redirect back to the app with a mock token.
  await route.fulfill({
    status: 302,
    headers: {
      Location: '/dashboard?token=mock-sso-token',
    },
  });
});
```

For most teams, option 1 (persist and reuse) is the most practical approach for SSO.

**Q: How would you implement a rollback mechanism that triggers when Playwright tests fail on production deployment?**

```yaml
name: Production Deploy
on:
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploy to production..."
      # Store the previous version for rollback.
      - run: echo "Current version: $(cat VERSION)" > previous-version.txt

  smoke-test:
    needs: deploy
    runs-on: ubuntu-latest
    steps:
      - run: npx playwright test --grep "@smoke"
      # If smoke tests pass, continue. If they fail, the workflow fails.

  rollback:
    needs: [deploy, smoke-test]
    if: failure()
    runs-on: ubuntu-latest
    steps:
      - run: echo "Rolling back to previous version..."
      # Deploy the previous version stored in artifacts.
      - name: Notify team
        run: |
          curl -X POST -H "Content-Type: application/json" \
            -d '{"text":"Deploy failed — rolling back"}' \
            ${{ secrets.SLACK_WEBHOOK_URL }}
```

The `needs` with `if: failure()` creates an automatic rollback when smoke tests fail. The key is that the `smoke-test` job must be a critical path — if it fails, the deployment is considered failed.

## Docker

### Easy

**Q: What is the Playwright Docker image and why would you use it?**

The Playwright Docker image (`mcr.microsoft.com/playwright`) is a pre-built Docker image that contains the Playwright CLI and all three browser engines (Chromium, Firefox, WebKit) with their system dependencies. You use it to run Playwright tests in a consistent, reproducible environment — eliminating "it works on my machine" issues caused by OS differences, font variations, and missing system libraries.

**Q: How do you pull and run the Playwright Docker image?**

```bash
# Pull the image.
docker pull mcr.microsoft.com/playwright:v1.45.0

# Run tests inside the container.
docker run -it --rm --ipc=host -v $(pwd):/app -w /app mcr.microsoft.com/playwright:v1.45.0 \
  /bin/bash -c "npm ci && npx playwright test"
```

The `--ipc=host` flag is critical for Chromium — it enables shared memory communication between Chromium processes.

**Q: What does the `--ipc=host` flag do and why is it important for Playwright in Docker?**

`--ipc=host` gives the container access to the host's IPC (Inter-Process Communication) namespace. Chromium uses shared memory for multiprocess architecture (one main process + multiple renderer processes). Without sufficient shared memory (`/dev/shm`), Chromium either crashes or runs extremely slowly. Docker containers have a small default `/dev/shm` (64MB), which is insufficient for Chromium. The `--ipc=host` flag bypasses this limitation by using the host's `/dev/shm`.

### Medium

**Q: How do you create a custom Dockerfile for a Playwright project?**

```dockerfile
FROM mcr.microsoft.com/playwright:v1.45.0 AS base
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
CMD ["npx", "playwright", "test"]

# For running tests and also starting the application.

# Copy the application code and start it.

```

**Q: How do you manage Playwright browser versions in Docker to match the local version?**

Use the same Playwright version tag for the Docker image and the npm package:

```bash
# Local package.json
"@playwright/test": "^1.45.0"

# Docker image tag
mcr.microsoft.com/playwright:v1.45.0
```

The `v` prefix matches your npm version. The Docker image contains the matching browser builds. If versions mismatch, you get "Browser not found" or browser launch errors. Pin the exact minor version in CI: `playwright:v1.45.0` not `playwright:latest`.

**Q: How do you run Playwright tests on a remote Docker host or Kubernetes?**

```bash
# On a remote Docker host:
docker -H tcp://remote-host:2375 run --rm -v $(pwd):/app mcr.microsoft.com/playwright ...

# On Kubernetes:
kubectl run playwright-test --image=mcr.microsoft.com/playwright:v1.45.0 \
  --restart=Never \
  -- /bin/bash -c "cd /app && npm ci && npx playwright test"
```

For Kubernetes, you must configure the pod to share the host IPC namespace and set `securityContext` to allow the Chrome sandbox:

```yaml

  containers:
  - name: playwright
    image: mcr.microsoft.com/playwright:v1.45.0
    securityContext:
      privileged: true  # Or configure seccomp properly.
  hostIPC: true
```

### Hard

**Q: How would you optimize a Playwright Docker image for CI to minimize build time and image size?**

Multi-stage builds and caching strategy:

```dockerfile
# Stage 1: Install production dependencies.
FROM node:20-slim AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --only=production

# Stage 2: Install Playwright browsers (only Chromium for PR checks).
FROM mcr.microsoft.com/playwright:v1.45.0 AS browsers
RUN npx playwright install chromium

# Stage 3: Final image.
FROM mcr.microsoft.com/playwright:v1.45.0
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
# Remove browsers we do not need (saves ~500MB).

CMD ["npx", "playwright", "test"]
```

Optimizations:
- Separate dependency install from application code (cache `npm ci` step).
- Use Playwright's official image as base (avoids installing system dependencies manually).
- Remove unused browsers for project-specific images (reduces image size).
- Use `.dockerignore` to exclude `node_modules`, `.git`, and test output directories.

**Q: Compare running Playwright in Docker vs running it with a local application server.**

| Aspect | Docker | Local Server |
|--------|--------|-------------|
| Consistency | High — identical environment everywhere | Lower — OS differences affect fonts, rendering |
| Setup time | Longer — build/run Docker | Quicker — start local server |
| Debugging | Harder — limited access to browser UI | Easier — headed mode, screenshots |
| CI integration | Natural — Docker is CI standard | May need additional setup |
| Multi-browser | Built-in — all browsers in one image | Must install all browsers manually |
| Resource usage | Higher — container overhead | Lower — runs natively |

Most teams use Docker in CI for consistency and run locally with a local server for development speed. Hybrid approach: Docker for CI, local for development.

### Easy

**Q: What reporters does Playwright support out of the box?**

Playwright ships with several built-in reporters: `list` (terminal output with pass/fail per test), `line` (compact one-line-per-test), `dot` (dots for fast CI feedback), `html` (interactive HTML report with test details, screenshots, and traces), `json` (machine-readable JSON output), `junit` (XML for CI tools like Jenkins), `blob` (for merging reports across shards), and `null` (no output). You configure reporters in the config.

**Q: How do you generate and open the HTML report?**

```typescript
reporter: [['html', { outputFolder: 'playwright-report', open: 'never' }]],
```

After running tests, open the report:

```bash
npx playwright show-report
```

Or open `playwright-report/index.html` directly in a browser. The default `open: 'on-failure'` auto-opens the report only when tests fail.

**Q: What information is included in the HTML report?**

The HTML report includes: a summary (total tests, passed, failed, skipped, flaky, duration), a per-test listing with project name and status, test error messages and stack traces, attached screenshots (on failure), trace file links, test steps (with timing), and project metadata (browser, viewport, base URL).

**Q: How do you configure multiple reporters in Playwright?**

Pass an array of reporter configurations:

Multiple reporters output simultaneously — the HTML report for human viewing, JSON for machine processing, and JUnit for CI integration.

**Q: How do you use the blob reporter and why would you need it?**

The blob reporter writes test results to a `.zip` file that can be merged later:

```typescript
reporter: [['blob', { outputDir: './blob-report' }]],
```

Use blob reports when running sharded tests. Each shard produces its own blob report. After all shards complete, you merge them:

```bash

```

This produces a single HTML report with all shard results combined. Without blob reports, each shard's HTML report only shows its subset of tests.

**Q: How do you create a custom reporter in Playwright?**

Implement the `Reporter` interface:

```typescript

    console.log(`Starting test suite: ${suite.title}`);
  }

  onTestEnd(test: TestCase, result: TestResult) {
    if (result.status !== 'passed') {
      console.log(`FAILED: ${test.title}`);
      console.log(`  Error: ${result.error?.message}`);
      // Post to Slack webhook.
    }
  }

  onEnd(result: { status: string }) {
    console.log(`Suite finished with status: ${result.status}`);
  }
}

export default SlackReporter;
```

Use it in config: `reporter: [['./slack-reporter.ts', { webhookUrl: '...' }]]`.

## Playwright MCP Server

### Easy

**Q: What is the Playwright MCP Server?**

The Playwright MCP (Model Context Protocol) Server is a bridge that allows AI assistants (like Claude) to interact with web browsers through Playwright. It implements the MCP standard, giving AI models capabilities like navigating pages, clicking elements, extracting content, and taking screenshots. It enables AI-driven browser automation where the AI can observe and interact with web pages in real-time.

**Q: How do you start the Playwright MCP Server?**

```bash
npx @playwright/mcp@latest
```

This starts an MCP server on stdio by default. You configure it with your AI tool's MCP configuration (e.g., in Claude Desktop's `claude_desktop_config.json` or VS Code's settings). The server starts a headless browser that the AI assistant can control.

**Q: What can the Playwright MCP Server do?**

The MCP server exposes tools and prompts to the AI assistant. Tools include: `browser_navigate` (go to a URL), `browser_click` (click an element), `browser_snapshot` (get a textual snapshot of the page), `browser_take_screenshot`, `browser_fill` (fill form fields), `browser_select_option`, `browser_hover`, and `browser_press_key`. The server maintains a single browser session that persists across the conversation.

### Medium

**Q: How does the MCP Server differ from using Playwright directly?**

The MCP Server is designed for AI-driven interaction — the AI assistant controls the browser through natural language or tool calls. Using Playwright directly requires programming (TypeScript/JavaScript) and the test runner. The MCP Server is session-based (the browser stays open across multiple commands), while Playwright tests are scripted and run sequentially. The MCP Server's tools are higher-level and designed for AI consumption, while Playwright's API is designed for programmatic control with fine-grained options.

**Q: What is the Model Context Protocol (MCP) in the context of Playwright?**

MCP is an open standard for connecting AI models with external tools and data sources. In Playwright's case, it defines a protocol where the AI assistant (the "client") sends tool requests to the Playwright MCP Server (the "server"), which executes browser automation and returns results. The protocol uses JSON-RPC messages over stdin/stdout or HTTP. This standardized interface means any MCP-compatible AI tool can use Playwright's browser automation.

**Q: How do you configure the Playwright MCP Server with Claude Desktop?**

Add this to your `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}
```

Claude Desktop then has access to browser automation tools in every conversation. The server launches a Chromium browser in the background that the AI can control.

### Hard

**Q: What are the security implications of running the Playwright MCP Server?**

The MCP Server gives an AI assistant full control over a web browser. Security concerns include:

1. **Data exfiltration**: The AI could navigate to internal services, read data, and send it externally.
2. **Unintended actions**: The AI could accidentally submit forms, delete data, or perform destructive actions.
3. **Credential exposure**: If the browser has saved passwords, the AI could access authenticated sessions.
4. **Network access**: The browser has the same network access as the host, including internal services.

Mitigations:
- Run the MCP Server in a sandboxed environment.
- Use a dedicated browser profile with no saved credentials.
- Limit network access to the specific application under test.
- Review all AI actions before the browser executes them.

**Q: How would you extend the Playwright MCP Server with custom tools?**

Create a wrapper that adds custom tools to the base server:

```typescript
import { MCPServer } from '@playwright/mcp';

const server = new MCPServer();

// Add a custom tool for taking full-page screenshots.
```

## Playwright CLI Agent

### Easy

**Q: What is the Playwright CLI Agent?**

The Playwright CLI Agent (also known as the Playwright MCP Agent or `playwright-cli-agent`) is a command-line tool that uses AI to generate and run Playwright tests from natural language instructions. You describe what you want to test in plain English, and the AI agent writes the Playwright code, runs it, analyzes failures, and fixes them — all without you writing a single line of code.

**Q: How do you install and use the Playwright CLI Agent?**

```bash
# Install the agent.
npm install -g @playwright/cli-agent

# Set your API key for the AI provider.
export ANTHROPIC_API_KEY=your-key

# Run a test from a natural language prompt.
playwright-agent "Go to google.com, search for 'Playwright', and verify the first result is playwright.dev"
```

The agent requires an API key for an LLM provider (Anthropic, OpenAI, etc.) and Playwright browsers installed on the system.

**Q: What kind of tests can the CLI Agent generate?**

The CLI Agent can generate tests for: navigation flows (login, signup, checkout), form validation, data extraction, visual verification, API interaction, and multi-step user journeys. It handles both UI-only flows and mixed API/UI scenarios. The generated output is standard Playwright test code that can be run with the normal test runner.

### Medium

**Q: How does the CLI Agent handle test failures during generation?**

The CLI Agent uses a loop: generate code, run it, examine the output, and iteratively fix any failures. If a locator fails to find an element, the agent examines the page DOM and adjusts the locator. If an assertion fails, the agent re-examines the expected behavior and corrects the assertion. This cycle continues until the test passes or the agent determines the task is impossible. The agent logs each iteration so you can see what changed between attempts.

**Q: What are the limitations of AI-generated Playwright tests?**

Generated tests may use brittle selectors (CSS classes, nested selectors) instead of the preferred semantic locators (`getByRole`, `getByLabel`). The AI may struggle with dynamic content (animations, live updates), complex authentication flows (OAuth, SSO), and application-specific logic (custom widgets, canvas elements). Generated tests also lack the structural organization (page objects, fixtures) that hand-written tests benefit from. Always review and refactor AI-generated tests before committing them to your repository.

**Q: How do you integrate the CLI Agent into an existing Playwright project?**

The CLI Agent can work with your existing `playwright.config.ts`:

```bash
playwright-agent \
  --config playwright.config.ts \
  --project chromium \
  "Test the login flow: go to /login, fill credentials, submit, verify redirect to /dashboard"
```

The agent respects your project's configuration (timeouts, viewports, baseURL, projects). Generated tests are saved to your configured test directory and can be run alongside your existing tests with `npx playwright test`.

### Hard

**Q: How would you build a CI pipeline that uses the CLI Agent to auto-heal broken locators?**

1. **Detect broken tests**: Run the test suite in CI. When tests fail, check if the failure is locator-related (element not found).
2. **Extract failure context**: Capture the failing locator, page DOM snapshot, and error message.
3. **Auto-heal with CLI Agent**: For locator-only failures, invoke the CLI Agent with the failure context:

```bash
playwright-agent \
  --fix locator \
  --test tests/dashboard.spec.ts \
  --context "Test failed because locator '.submit-btn' not found. Page now has <button class='send-btn'>Submit</button>"
```

4. **Review and accept**: Create a pull request with the agent's changes. A human reviews and accepts.
5. **Fallback on failure**: If the agent cannot fix the test, mark it as "needs human review" and alert the team.

This approach reduces maintenance burden but requires guardrails — the agent should never modify production code or change test assertions without verification.

**Q: Compare the Playwright MCP Server vs the CLI Agent for AI-driven browser automation.**

| MCP Server | CLI Agent |
|---|---|
| Real-time AI-browser interaction | Batch test generation |
| AI drives the browser step-by-step | AI generates code, then code runs |
| One-shot exploratory tasks | Multi-shot iterative refinement |
| No test artifacts produced | Produces test files |
| Session-based (browser persists) | Script-based (browser per run) |
| Best for: research, debugging, data extraction | Best for: test creation, maintenance, auto-healing |

The MCP Server is ideal when you want the AI to "drive" the browser in real-time (e.g., "find the price of this product"). The CLI Agent is ideal when you want to generate automated tests that run in CI (e.g., "write a test for the checkout flow").

## AI Test Generation

### Easy

**Q: How can AI assist in writing Playwright tests?**

AI can help in several ways: generating test code from natural language descriptions, suggesting locators for elements you describe, detecting flaky tests and suggesting fixes, generating test data (edge cases, boundary values), explaining test failures in plain English, and refactoring test code (e.g., extracting page objects). AI assistants like Claude and GitHub Copilot can inline-suggest test code as you type.

**Q: What prompts work well for generating Playwright tests with an AI assistant?**

Good prompts are specific and include context:

- "Write a Playwright test that navigates to /products, selects 'Sort by Price', and verifies items are sorted ascending."
- "Generate a page object for the login page with methods for entering email, entering password, and clicking submit."
- "Create a test that uses page.route() to mock the /api/users endpoint and verifies the empty state UI."
- "Refactor this test to use getByRole instead of CSS selectors: [paste code]"

Include your app's URL, the elements' accessible labels, and expected behaviors for the best results.

**Q: What is the risk of AI-generated locators?**

AI models may generate brittle CSS selectors like `.container > div:nth-child(3) > button.submit-btn` instead of the more robust `page.getByRole('button', { name: 'Submit' })`. The AI may also generate selectors based on the training data's DOM patterns rather than your actual application. Always review and replace AI-generated locators with semantic alternatives.

### Medium

**Q: How do you use AI to detect and fix flaky tests?**

Provide the AI with a flaky test's failure output, the test code, and the page state (DOM, console logs). A good prompt:

```
This Playwright test is flaky — it fails about 20% of the time. Here is the test code:
[code]
Here is the failure output:
[error]
Here is the DOM snapshot when it fails:
[DOM]

Why is this test flaky and how would you fix it?
```

The AI can identify common flakiness causes: race conditions (missing `Promise.all` for navigation/click), element instability (animations, transitions), network timing (data not loaded before assertion), or shared state between tests.

**Q: How do you validate AI-generated Playwright tests before adding them to the test suite?**

Validation steps:

1. **Code review**: Check for proper locators (prefer `getByRole`, `getByLabel` over CSS/XPath).
2. **Dry run**: Run with `--headed --slowmo 500` to visually confirm interactions.
3. **Assertion review**: Ensure assertions are specific enough (not just `toBeVisible` for every check).
4. **Isolation check**: Verify the test does not depend on other tests or shared state.
5. **Cross-browser test**: Run on at least two browsers to catch browser-specific issues.
6. **CI integration**: Run the generated test in a PR CI pipeline before merging.

**Q: Can AI generate tests for accessibility verification?**

Yes. Prompt the AI to generate tests that use accessibility locators (`getByRole`), check for proper ARIA attributes, validate keyboard navigation, and run axe-core integration:

```typescript
test('accessibility check', async ({ page }) => {
  await page.goto('/');
  // Run axe-core.
  const violations = await page.evaluate(async () => {
    const axe = await import('axe-core');
    const result = await axe.default.run();
    return result.violations;
  });
  expect(violations.length).toBe(0);
});
```

AI can also suggest improvements to your application based on accessibility test results.

### Hard

**Q: Design a system that uses AI to automatically triage and categorize Playwright test failures.**

Architecture:

1. **Failure collection**: Custom Playwright reporter captures failures with full context (error, screenshot, DOM snapshot, trace, console logs).
2. **AI categorization**: Send failure details to an LLM with a prompt:

```
Categorize this Playwright test failure into one of:
- LOCATOR_STALE: element selector no longer matches
- TIMEOUT: page or element load timeout
- ASSERTION: expected value does not match actual
- NETWORK: API call failed or returned unexpected data
- VISUAL: snapshot mismatch (intentional UI change)
- FLAKY: intermittent failure
- UNKNOWN: cannot determine

Provide: category, confidence (1-10), suggested fix (1-2 sentences), related tests.

Failure details:
[error, DOM snapshot, test code]
```

3. **Dashboard**: Store categorized failures and track trends (which categories increase over time).
4. **Auto-heal (for LOCATOR_STALE)**: For high-confidence stale locator failures, automatically trigger the CLI Agent to fix the locator.
5. **Alerting**: Notify the team when UNKNOWN or ASSERTION failures spike.

This system reduces manual triage time and helps identify systemic issues (e.g., "all our tests use brittle CSS selectors that keep breaking").

**Q: How would you use AI to generate comprehensive test data covering edge cases that human testers often miss?**

Use AI to analyze your application's schemas and generate edge-case test data:

```
Given this form schema:
- Name: string, required, max 100 chars
- Email: string, required, valid email format
- Age: number, optional, min 18, max 120
- Role: enum [admin, user, viewer], required

Generate Playwright test code that tests ALL validation rules with edge cases:
- Boundary values (0, 1, 17, 18, 119, 120, 121)
- Invalid formats (malformed emails, negative ages)
- SQL injection and XSS payloads in name field
- Empty required fields
- Maximum length strings (100-char name)
- Special characters (Unicode, emoji in name)
- Reserved keywords as email addresses
```

The AI can generate test code for each scenario and organize them into `describe` blocks. This catches edge cases that human testers might not think of, such as SQL injection in a name field or Unicode normalization issues in email addresses.
