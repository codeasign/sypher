# Reporting — Build It

**Reading time:** 14 minutes · **Difficulty:** Intermediate

## Step by Step

### 1. Understand the reporter configuration

Playwright reporters are configured in the `reporter` field of `playwright.config.ts`. You can specify a single reporter as a string, or multiple reporters as an array of tuples:

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // Single reporter
  reporter: 'html',
});
```

For multiple reporters, each entry is a tuple of `[reporterName, optionsObject]`:

```typescript
export default defineConfig({
  reporter: [
    ['html', { open: 'never' }],
    ['junit', { outputFile: 'test-results/junit.xml' }],
  ],
});
```

### 2. Configure the HTML reporter (default)

The HTML reporter generates a standalone interactive report with a filterable UI, test steps, trace viewer integration, and failure screenshots.

```typescript
export default defineConfig({
  reporter: [
    ['html', {
      open: process.env.CI ? 'never' : 'on-failure',
      outputFolder: 'playwright-report',
      host: '0.0.0.0',
      port: 9323,
    }],
  ],
});
```

The `open` option controls when the report opens in a browser:
- `'always'` — opens after every test run
- `'on-failure'` — opens only when a test fails
- `'never'` — never opens (use in CI)

The `outputFolder` defaults to `playwright-report`. You can change it if you need multiple reports or want to avoid conflicts.

### 3. Add the JUnit reporter for CI

JUnit XML is the standard format for CI systems. GitHub Actions, Jenkins, CircleCI, and GitLab CI all parse JUnit XML to annotate pull requests with test results.

```typescript
export default defineConfig({
  reporter: [
    ['html', { open: 'never' }],
    ['junit', {
      outputFile: 'test-results/junit.xml',
      stripANSIControlCodes: true,
      includeProjectInTestName: true,
    }],
  ],
});
```

The `includeProjectInTestName` option prepends the project name (e.g., `chromium`, `firefox`) to each test name, which is useful when the same test runs across multiple browsers.

For GitHub Actions, the JUnit XML file is automatically parsed and test failures appear as annotations on the pull request if you add this step to your workflow:

```yaml
- name: Publish test results
  uses: dorny/test-reporter@v1
  if: always()
  with:
    name: Playwright Tests
    path: test-results/junit.xml
    reporter: java-junit
```

### 4. Add the JSON reporter for programmatic access

The JSON reporter writes test results as structured JSON that you can feed into a dashboard, a Slack bot, or a custom analysis tool.

```typescript
export default defineConfig({
  reporter: [
    ['html', { open: 'never' }],
    ['junit', { outputFile: 'test-results/junit.xml' }],
    ['json', { outputFile: 'test-results/test-results.json' }],
  ],
});
```

The JSON output includes every test's full title, status, duration, error message, and metadata about retries.

### 5. Add the blob reporter for sharding

When you shard tests across multiple CI runners, each shard needs to produce a blob report that can be merged later. The blob reporter is designed specifically for this use case.

```typescript
export default defineConfig({
  reporter: [
    ['blob'],
    ['html', { open: 'never' }],
  ],
});
```

The blob reporter outputs a binary file (with a name like `report-{uuid}.zip`) into the `blob-report/` directory. After all shards complete, merge them with:

```bash
npx playwright merge-reports --reporter html ./all-blob-reports
```

This command reads all blob files from `./all-blob-reports/`, merges them, and generates a single unified HTML report with all traces, screenshots, and results from every shard.

### 6. Override reporters with environment variables

You can override the configured reporter at runtime using the `PLAYWRIGHT_REPORTER` environment variable. This is useful for quick debugging without changing your config file:

```bash
# Override to dot reporter for a quick overview
PLAYWRIGHT_REPORTER=dot npx playwright test

# Override to list reporter to watch test progress
PLAYWRIGHT_REPORTER=list npx playwright test

# Override to a specific combination
PLAYWRIGHT_REPORTER="[{\"reporter\":\"line\"},{\"reporter\":\"json\",\"outputFile\":\"quick.json\"}]" npx playwright test
```

## Create a Custom Reporter

Built-in reporters cover the most common use cases, but sometimes you need something specific — a Slack notification, a JIRA ticket for every failure, or a custom summary format.

### Step 1: Create the reporter class

Create `my-reporter.ts` in your project root:

```typescript
import type {
  FullConfig,
  FullResult,
  TestCase,
  TestResult,
  Reporter,
} from '@playwright/test/reporter';

class MyReporter implements Reporter {
  private results: Array<{
    file: string;
    title: string;
    status: string;
    duration: number;
  }> = [];

  onBegin(config: FullConfig, suite: any) {
    console.log(`Starting test run with ${suite.allTests().length} tests`);
  }

  onTestEnd(test: TestCase, result: TestResult) {
    const filePath = test.location.file;
    const fileName = filePath.split(/[/\\]/).pop() || filePath;
    this.results.push({
      file: fileName,
      title: test.title,
      status: result.status || 'unknown',
      duration: result.duration,
    });
  }

  onEnd(result: FullResult) {
    // Group results by file
    const grouped: Record<string, typeof this.results> = {};
    for (const r of this.results) {
      if (!grouped[r.file]) {
        grouped[r.file] = [];
      }
      grouped[r.file].push(r);
    }

    // Print summary table
    console.log('\n=== Test Summary ===');
    console.log(`Status: ${result.status}`);
    console.log(`Total: ${this.results.length} tests`);
    console.log('');

    for (const [file, tests] of Object.entries(grouped)) {
      const passed = tests.filter((t) => t.status === 'passed').length;
      const failed = tests.filter((t) => t.status === 'failed').length;
      const skipped = tests.filter((t) => t.status === 'skipped').length;
      const totalTime = tests.reduce((sum, t) => sum + t.duration, 0);

      console.log(`File: ${file}`);
      console.log(`  Passed: ${passed} | Failed: ${failed} | Skipped: ${skipped}`);
      console.log(`  Total time: ${(totalTime / 1000).toFixed(1)}s`);

      for (const t of tests) {
        if (t.status !== 'passed') {
          console.log(`    ${t.status === 'failed' ? 'X' : '-'} ${t.title}`);
        }
      }
      console.log('');
    }
  }
}

export default MyReporter;
```

### Step 2: Register the custom reporter

In `playwright.config.ts`, add your custom reporter to the reporter array:

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  reporter: [
    ['html', { open: 'never' }],
    ['junit', { outputFile: 'test-results/junit.xml' }],
    ['blob'],
    ['./my-reporter.ts'],
  ],
  // ...
});
```

### Step 3: Run tests and see custom output

```bash
npx playwright test
```

You will see the custom summary printed after the test run:

```
=== Test Summary ===
Status: passed
Total: 24 tests

File: login.spec.ts
  Passed: 8 | Failed: 0 | Skipped: 0
  Total time: 12.3s

File: checkout.spec.ts
  Passed: 6 | Failed: 1 | Skipped: 0
  Total time: 8.7s
    X should validate shipping address
```

## Build Something Real

Now combine everything into a production-grade reporting configuration.

### playwright.config.ts

```typescript
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 1 : undefined,
  reporter: [
    ['html', {
      open: process.env.CI ? 'never' : 'on-failure',
      outputFolder: 'playwright-report',
    }],
    ['junit', {
      outputFile: 'test-results/junit.xml',
      stripANSIControlCodes: true,
      includeProjectInTestName: true,
    }],
    ['blob'],
    ['./my-reporter.ts'],
  ],
  use: {
    baseURL: process.env.BASE_URL || 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
  projects: [
    { name: 'chromium', use: { browserName: 'chromium' } },
    { name: 'firefox', use: { browserName: 'firefox' } },
    { name: 'webkit', use: { browserName: 'webkit' } },
  ],
});
```

### CI merge step for blob reports

If you use this configuration with sharded CI runners, add a merge step to your GitHub Actions workflow:

```yaml
- name: Download all blob reports
  uses: actions/download-artifact@v4
  with:
    pattern: blob-report-*
    merge-multiple: true
    path: all-blob-reports

- name: Merge reports
  run: npx playwright merge-reports --reporter html ./all-blob-reports
```

The `merge-reports` command processes all blob files in the `all-blob-reports/` directory and generates a unified HTML report with complete trace data from every shard.

### Verify it works

Run your tests and verify each output is generated:

```bash
npx playwright test

# Check that all reports exist
ls -la playwright-report/   # HTML report
ls -la test-results/        # JUnit XML + custom report artifacts
ls -la blob-report/         # Blob report for shard merging
```
