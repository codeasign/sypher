# Reporting — Avoid Mistakes

**Reading time:** 6 minutes · **Difficulty:** Intermediate

## When NOT to Use This

Playwright's reporter system is flexible, but not every reporter is appropriate for every situation.

| Use this reporter | Do NOT use this reporter for |
|---|---|
| HTML reporter for local debugging | Using HTML reporter in CI without uploading the report as an artifact — it is invisible |
| JUnit reporter for CI integration | Using line or dot reporter in CI when JUnit is available — you lose PR annotations |
| Blob reporter for sharded CI runs | Using HTML reporter alone for sharded runs — each shard overwrites the previous report |
| Custom reporter for team-specific output | Building a custom reporter when the built-in JSON reporter plus a post-processing script would work |

Do not use the HTML reporter with `open: 'on-failure'` or `open: 'always'` in CI. The browser launch will fail because CI environments typically do not have a display server, and even if it succeeds, no one is watching the screen.

## The Pattern to Remember

1. **HTML for local debugging** — with `open: 'on-failure'` so you only see it when something breaks
2. **JUnit for CI** — GitHub Actions, Jenkins, and CircleCI all parse JUnit XML natively
3. **Blob for shards** — each shard produces a blob, merge step combines them
4. **Custom for alerts** — send notifications to Slack, email, or JIRA when critical tests fail
5. **Combine them** — you can use all four in a single test run

## Common Mistakes

### Mistake 1: HTML report not uploaded as a CI artifact

The HTML reporter generates `playwright-report/index.html` and all its supporting files, but these files exist only on the CI runner's filesystem. If you do not upload them as an artifact, no one can access them after the run completes.

```yaml
# Bad — HTML report generated but never uploaded
steps:
  - run: npx playwright test

# Good — upload the report as an artifact
steps:
  - run: npx playwright test
  - uses: actions/upload-artifact@v4
    if: always()
    with:
      name: playwright-report
      path: playwright-report/
      retention-days: 30
```

Always use `if: always()` on the upload step so the report is available even when tests fail.

### Mistake 2: Using multiple reporters that conflict

Some reporters write to the same file or directory path by default. For example, running two instances of the HTML reporter with the same `outputFolder` causes one to overwrite the other.

```typescript
// Bad — both write to the default playwright-report/ directory
reporter: [
  ['html'],           // outputFolder defaults to 'playwright-report'
  ['html'],           // overwrites the first reporter's output
]

// Good — give each reporter a unique output path
reporter: [
  ['html', { outputFolder: 'playwright-report/summary' }],
  ['html', { outputFolder: 'playwright-report/detailed' }],
]
```

In practice, you rarely need two instances of the same reporter. Use different reporter types instead.

### Mistake 3: Forgetting blob reporter for sharded runs

When you shard tests across multiple CI runners without the blob reporter, each shard generates its own HTML report. The final merge step only has access to one shard's data — all traces, screenshots, and failure details from other shards are lost.

```typescript
// Bad — each shard overwrites the HTML report
reporter: [
  ['html', { open: 'never' }],
]

// Good — each shard produces a blob, merge step combines them
reporter: [
  ['blob'],
  ['html', { open: 'never' }],
]
```

Then merge in CI:

```bash
npx playwright merge-reports --reporter html ./all-blob-reports
```

### Mistake 4: Not setting `open: 'never'` in CI

The HTML reporter tries to open a browser to display the report after the test run. In CI, there is no browser display available, so this either fails with an error or hangs waiting for a display connection.

```typescript
// Bad — tries to open browser in CI
reporter: [
  ['html'],    // default: open: 'on-failure'
]

// Good — never opens browser in CI
reporter: [
  ['html', { open: process.env.CI ? 'never' : 'on-failure' }],
]
```

### Mistake 5: JUnit without outputFile — writes to stdout

If you configure the JUnit reporter without an `outputFile`, it writes the XML to stdout instead of a file. CI systems looking for a `junit.xml` file will not find it.

```typescript
// Bad — JUnit XML goes to stdout, not a file
reporter: [
  ['junit'],
]

// Good — writes to a specific file
reporter: [
  ['junit', { outputFile: 'test-results/junit.xml' }],
]
```

Always specify an output file path for the JUnit reporter.

## Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| HTML report opens browser in CI | `open` option not set to `never` | Set `open: 'never'` or use `process.env.CI ? 'never' : 'on-failure'` |
| JUnit XML files overwritten in shards | All shards write to the same `outputFile` path | Use blob reporter for shards, generate JUnit in the merge step |
| Blob report directory not found | No blob reporter configured | Add `['blob']` to the reporter list |
| Custom reporter's `onEnd` not called | Syntax error in custom reporter | Check the reporter class implements the `Reporter` interface and is exported as default |
| Report missing after merge | Different Playwright versions across shards | Ensure all CI runners use the same Playwright version |
| Blob report merge fails with version mismatch | Blob files from different Playwright versions are incompatible | Pin Playwright version across all CI steps |

## Production Checklist

Before relying on your reporting setup, verify these items:

- [ ] Multiple reporters configured (HTML + JUnit + blob at minimum)
- [ ] HTML report `open` set to `'never'` in CI or based on `process.env.CI`
- [ ] Report artifacts uploaded with `if: always()` and `retention-days` set
- [ ] JUnit XML `outputFile` path specified and unique per project/shard
- [ ] Blob reporter configured for any sharded CI runs
- [ ] Blob reports directory structured for merge (each shard's blob in its own artifact)
- [ ] `merge-reports` step present in CI pipeline for sharded setups
- [ ] Custom reporter tested with both passing and failing tests
