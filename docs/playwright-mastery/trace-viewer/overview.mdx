# Trace Viewer

Difficulty: Intermediate

## What You Will Learn

- Enable tracing in Playwright tests with the correct mode for each environment
- Open and navigate trace files in the Trace Viewer UI
- Inspect action timelines, network requests, console logs, and DOM snapshots
- Use trace metadata and filtering to isolate failure root causes

## Prerequisites

- Playwright core lessons completed (writing and running tests)
- Familiarity with `playwright.config.ts`
- Basic understanding of browser DevTools concepts (network tab, console)

## Why This Matters

When a test fails in CI, you don't have a browser window to watch. Without traces, you guess what happened — was the API down? Did a selector fail? Was the page still loading? Traces capture every action, network request, console message, and DOM snapshot at each step. You replay the exact failure in the Trace Viewer, seeing exactly what the browser saw at every moment. This turns hours of debugging into minutes.

## Quick Reference

| Config Value | Behavior | Best For |
|---|---|---|
| `trace: 'off'` | No traces recorded | Stable, full test suites in CI |
| `trace: 'on'` | Trace every test | Local debugging, small suites |
| `trace: 'retain-on-failure'` | Trace only failed tests | CI — best balance of cost and debuggability |
| `trace: 'on-first-retry'` | Trace on the first retry after failure | Flaky test investigation |
| `trace: { mode: 'on', snapshots: true, screenshots: true }` | Full trace with all snapshots | Deep debugging of complex failures |

| Trace Viewer Feature | Description |
|---|---|
| Action Timeline | Every Playwright action in sequence with timing |
| Network Log | All HTTP requests with status, timing, and response body |
| Console | Browser console.log, warn, error messages |
| DOM Snapshot | Full DOM before and after each action |
| Source | Source file and line for each action |

## The Big Picture
