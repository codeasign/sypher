# Trace Viewer Review

## Key Takeaways

1. **Use `retain-on-failure` in CI**: Get traces for failures without paying storage costs for every passing test. This is the single best configuration for production test suites.
2. **Traces capture everything**: Actions, network requests, console logs, and DOM snapshots. When a test fails, the trace is your time machine — you see exactly what the browser saw.
3. **Look backward from the failure**: The assertion that failed is rarely the root cause. Scroll back through the action timeline to find the first unexpected state — that's where the bug is.
4. **Check the Network tab first**: When an expected element doesn't appear, the answer is usually in the network tab — a failed API call, a timeout, or a 500 error.
5. **Match trace mode to your environment**: `off` for large stable CI suites, `retain-on-failure` for normal CI, `on` for local debugging, `on-first-retry` for flaky tests.

### Scenario Questions

**Scenario 1**: Your team's Playwright CI suite has 300 tests. You set `trace: 'on'` to debug a flaky test. Now CI runs take 3x longer and the artifact storage is full. What changes do you make to the trace configuration, and how do you still keep the ability to debug the flaky test?

**Scenario 2**: A test clicks a "Save" button and asserts that a "Saved successfully" message appears. The test fails with a timeout. You open the trace. The DOM snapshot shows the "Save" button is visible and clickable. The action timeline shows the click succeeded. But the success message never appeared. Walk through your debugging steps in the Trace Viewer to find the root cause.

### Answer Key

**MCQ Answers**: 1-C, 2-C, 3-C, 4-B, 5-B

**Scenario 1**: Change from `trace: 'on'` to `trace: 'retain-on-failure'` for the main configuration. Create a separate project or test-specific override for the flaky test with `trace: 'on'` so you only pay the cost for that one test. Also add `trace: 'on-first-retry'` as a middle ground — it only records a trace after the first failure triggers a retry.

**Scenario 2**: Debugging steps:
1. Check the Network tab — look for the POST/PUT request to save the data. If it shows a 500 or (failed), that's the root cause.
2. Check the Console tab — the page might have logged an error when the save failed.
3. Look at the DOM snapshot "After" the click — compare it with "Before". Did the button state change? Did any element appear?
4. Check the Action Timeline timing — was the click very fast (maybe it didn't register) or did it wait for something?
5. Scroll back before the click — was the network request that loaded the save endpoint already failing?
