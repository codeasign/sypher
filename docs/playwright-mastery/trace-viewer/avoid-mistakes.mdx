# Avoid Mistakes with Trace Viewer

## When NOT to Use This

- **Every passing test in CI**: Enabling `trace: 'on'` for all tests in CI generates gigabytes of trace files per run. Storage costs add up and trace uploads slow down pipeline execution.
- **As your sole debugging tool**: Always read the test output and error message first. The trace is for deeper investigation — don't jump straight to the Trace Viewer for every failure.
- **Instead of assertions**: Traces are a debugging aid, not a substitute for well-written assertions. A good assertion pinpoints the failure immediately; a trace helps you understand why.
- **For performance tests**: Traces add overhead to test execution. Don't enable them in performance or load testing scenarios.

## The Pattern to Remember

- `retain-on-failure` for CI — you get traces for failures without paying for every passing test.
- `on` for local debugging when you're developing or investigating a specific issue.
- `off` for large stable suites in CI to keep artifact storage manageable.

## Common Mistakes

**1. Traces disabled in CI so failures can't be debugged**

You set `trace: 'off'` globally to save storage. A test fails in CI and you have no way to see what happened. You spend 30 minutes trying to reproduce locally, but the environment is different.

```typescript
// BAD: no traces in CI
export default defineConfig({
  use: { trace: process.env.CI ? 'off' : 'on' },
});

// GOOD: retain-on-failure gives you traces when you need them
export default defineConfig({
  use: { trace: 'retain-on-failure' },
});
```

**2. Traces enabled for every test fills disk space and slows CI**

A full test suite with 500 tests and `trace: 'on'` produces 500 trace files. Even small traces average 200-500 KB each, totaling 100-250 MB per run. Multiply by 10 parallel workers and you have multiple gigabytes.

```typescript
// BAD: all tests traced in CI
export default defineConfig({
  use: { trace: 'on' },
});

// GOOD: only failed tests traced
export default defineConfig({
  use: { trace: 'retain-on-failure' },
});
```

**3. Not knowing the Trace Viewer shortcuts**

You stare at the Trace Viewer clicking through actions one by one, missing that you can press arrow keys to step through actions, use the filter bar to find specific requests, and toggle the console with one click.

| Shortcut | Action |
|---|---|
| Up/Down arrows | Step through actions |
| Click action | Jump to that point in the timeline |
| Filter input | Search network requests by URL |
| Console tab | View all console output at that action |
| Before/After toggle | Compare DOM state between actions |

**4. Only looking at the last action, not the action that caused the failure**

A test clicks a button, waits for a response, and then asserts on an element. You look at the assertion action in the trace, but the real problem happened four actions earlier when the API call returned a 500. Always scroll back through the timeline to find the first unexpected state.

```typescript
// If the "Order Confirmed" assertion fails, go back in the trace to find:
// - Did the button click succeed? (Timeline shows it)
// - Did the API request fail? (Network tab)
// - Was the page still loading? (DOM snapshot)
// - Were there console errors? (Console tab)
```

**5. Not checking the network tab for failed API calls**

The DOM looks correct, the action succeeded, but the expected element never appeared. You're looking at the DOM snapshot when the answer is in the Network tab — the API call returned a 500 or timed out.

```typescript
// When debugging a missing element:
// 1. Check the Network tab for the API call that loads that data
// 2. Look for 4xx/5xx status codes or long response times
// 3. Check the response body — it might contain an error message
```

## Troubleshooting

| Symptom | Likely Cause | Solution |
|---|---|---|
| Trace file too large (50+ MB) | `screenshots: true` and `snapshots: true` in config | Use `trace: 'retain-on-failure'` with default settings; only enable screenshots for specific tests |
| Trace Viewer shows no actions | Trace recording started too late or the page context was created before tracing | Enable trace in `use` options (not per-test) so it applies to all contexts |
| Can't download trace from CI | CI artifact retention policy deletes test results | Configure artifact retention: `actions/upload-artifact` with `retention-days: 7` |
| DOM snapshot shows stale content | The snapshot was taken before the action completed | Look at the "After" state, not "Before" state; verify with `waitForSelector` before assertion |
| Trace Viewer is slow to load | Very large trace file with many actions | Filter the trace to only relevant actions; reduce screenshot/snapshot settings |
| Trace not generated at all | `trace` option not set in config, or set per-project with wrong project selected | Verify `trace` is set in `use` block of the correct project; check `playwright.config.ts` |

## Production Checklist

- [ ] Trace mode set to `'retain-on-failure'` in CI configuration
- [ ] Trace files are accessible from CI artifacts (uploaded and retained)
- [ ] Team knows how to use the action timeline and network tabs in Trace Viewer
- [ ] Local development workflow includes `trace: 'on'` for debugging
- [ ] Disk space monitoring is in place for trace file accumulation
- [ ] Team has a documented debugging workflow: read error → check trace → open network tab → inspect DOM snapshots
- [ ] Trace file cleanup is automated (CI artifact retention, local `.gitignore` for `test-results/`)
