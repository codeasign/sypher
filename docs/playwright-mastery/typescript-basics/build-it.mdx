# TypeScript Basics — Build It

In this hands-on lesson you will write real Playwright test files with TypeScript types, async/await patterns, interfaces for test data, and generics for collections. Follow each step and then build something real at the end.

## Step 1: Create a Test File with Basic Types

Create `tests/example.spec.ts` and declare Playwright types explicitly:

```typescript
import { test, expect, type Page, type Browser, type BrowserContext } from '@playwright/test';

let page: Page;
let browser: Browser;
let context: BrowserContext;

test.beforeAll(async ({ browser: b }) => {
  browser = b;
  context = await browser.newContext();
  page = await context.newPage();
});

test.afterAll(async () => {
  await context.close();
  await browser.close();
});

test('homepage loads', async () => {
  await page.goto('https://example.com');
  await expect(page).toHaveTitle(/Example/);
});
```

Notice the `type` keyword in the import. Playwright exports types separately so you can use them without bundling runtime code. The `page`, `browser`, and `context` variables all carry their exact Playwright types, and TypeScript validates every method call against those types.

## Step 2: Async/Await with Playwright

Every Playwright action returns a Promise. You must `await` every one of them:

```typescript
test('form submission', async ({ page }) => {
  await page.goto('https://example.com/login');

  // All of these return Promises — never skip await
  await page.locator('#username').fill('testuser');
  await page.locator('#password').fill('secret123');
  await page.locator('button[type="submit"]').click();

  // Wait for navigation after submit
  await page.waitForURL('**/dashboard');

  await expect(page.locator('h1')).toHaveText('Welcome');
});
```

## Step 3: Use Interfaces for Custom Test Data

Define interfaces for your test data so TypeScript catches missing fields or wrong types:

```typescript
interface User {
  name: string;
  email: string;
  role: 'admin' | 'user';
}

interface TestConfig {
  baseUrl: string;
  timeout: number;
  users: User[];
}

const config: TestConfig = {
  baseUrl: 'https://example.com',
  timeout: 30000,
  users: [
    { name: 'Alice', email: 'alice@example.com', role: 'admin' },
    { name: 'Bob', email: 'bob@example.com', role: 'user' },
  ],
};

test('admin can access settings', async ({ page }) => {
  const admin = config.users.find((u) => u.role === 'admin')!;

  await page.goto(`${config.baseUrl}/login`);
  await page.locator('#email').fill(admin.email);
  await page.locator('#password').fill('password123');
  await page.locator('button[type="submit"]').click();

  await page.locator('text=Settings').click();
  await expect(page).toHaveURL(/\/settings/);
});
```

If you misspelled `admin.email` as `admin.emial`, TypeScript would flag it immediately. The `!` after `find()` tells TypeScript you are sure the result is not undefined — use it only when you have validated the data exists.

## Step 4: Use Generics with Locator Collections

Playwright uses generics throughout its API. When you call `allTextContents()`, TypeScript infers `string[]`:

```typescript
test('list items have correct text', async ({ page }) => {
  await page.goto('https://example.com/items');

  // TypeScript infers string[] here
  const texts: string[] = await page.locator('.item-title').allTextContents();

  const numbers: number[] = texts
    .map((t) => parseInt(t, 10))
    .filter((n) => !isNaN(n));

  expect(numbers.length).toBeGreaterThan(0);
});
```

When you use `locator().all()` you get an array of `Locator` objects, each of which supports Playwright methods:

```typescript
test('each item has a delete button', async ({ page }) => {
  await page.goto('https://example.com/items');

  const items = await page.locator('.item').all();

  for (const item of items) {
    const deleteBtn = item.locator('.delete');
    await expect(deleteBtn).toBeVisible();
  }
});
```

TypeScript knows `items` is `Locator[]`, so `item.locator()` is fully typed.

## Step 5: Type Annotations in Fixture Destructuring

When you destructure fixtures from the test function parameter, Playwright's type system knows the types automatically. But you can also annotate explicitly for clarity:

```typescript
import { test, expect, type Page, type BrowserContext } from '@playwright/test';

test('fixture types are inferred', async ({ page, context }) => {
  // TypeScript knows page: Page and context: BrowserContext from the import
  const title: string = await page.title();
  const cookies = await context.cookies();
  expect(cookies.length).toBeGreaterThan(0);
});
```

You do not need to write `async ({ page }: { page: Page })` because Playwright exports typed test functions. The types flow through automatically. You only need explicit annotations when declaring standalone variables or function parameters outside the test callback.

## Build Something Real

Now bring everything together. Create a test file that uses typed test data, navigates a form, fills fields with typed values, submits, and asserts — with TypeScript catching mistakes at compile time.

Create `tests/user-signup.spec.ts`:

```typescript
import { test, expect } from '@playwright/test';

// --- Typed test data ---
interface SignupForm {
  firstName: string;
  lastName: string;
  email: string;
  password: string;
  confirmPassword: string;
}

const validUser: SignupForm = {
  firstName: 'Jane',
  lastName: 'Doe',
  email: 'jane.doe@example.com',
  password: 'SecureP@ss1',
  confirmPassword: 'SecureP@ss1',
};

const mismatchUser: SignupForm = {
  firstName: 'Jane',
  lastName: 'Doe',
  email: 'jane.doe@example.com',
  password: 'SecureP@ss1',
  confirmPassword: 'WrongPassword',
};

const invalidEmailUser: SignupForm = {
  firstName: 'Jane',
  lastName: 'Doe',
  email: 'not-an-email',
  password: 'SecureP@ss1',
  confirmPassword: 'SecureP@ss1',
};

// --- Helper function with typed parameters ---
async function fillSignupForm(page: import('@playwright/test').Page, data: SignupForm) {
  await page.locator('#firstName').fill(data.firstName);
  await page.locator('#lastName').fill(data.lastName);
  await page.locator('#email').fill(data.email);
  await page.locator('#password').fill(data.password);
  await page.locator('#confirmPassword').fill(data.confirmPassword);
  await page.locator('button[type="submit"]').click();
}

// --- Tests ---

test.describe('User signup flow', () => {
  test('successful signup with valid data', async ({ page }) => {
    await page.goto('https://example.com/signup');

    await fillSignupForm(page, validUser);

    // Wait for success redirect
    await page.waitForURL('**/welcome');
    await expect(page.locator('.success-message')).toHaveText(
      'Account created successfully!'
    );
  });

  test('shows error when passwords do not match', async ({ page }) => {
    await page.goto('https://example.com/signup');

    await fillSignupForm(page, mismatchUser);

    await expect(page.locator('.error-message')).toHaveText(
      'Passwords do not match'
    );
  });

  test('shows error for invalid email format', async ({ page }) => {
    await page.goto('https://example.com/signup');

    await fillSignupForm(page, invalidEmailUser);

    await expect(page.locator('.error-message')).toHaveText(
      'Please enter a valid email address'
    );
  });
});
```

TypeScript checks everything here: the `SignupForm` interface ensures you never forget a field or pass a wrong type. The `fillSignupForm` helper is fully typed with an explicit `Page` parameter. The test data objects are validated against the interface at compile time.

Try introducing a deliberate mistake — change `validUser.email` to `12345` — and run `npx tsc --noEmit`. TypeScript will report the error before you run a single test.
