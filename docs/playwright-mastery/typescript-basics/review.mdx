# TypeScript Basics — Review

Review everything you learned about TypeScript for Playwright, test your knowledge with a quiz, and prepare for interview questions.

## Key Takeaways

1. Every Playwright action returns a Promise and must be awaited. Missing `await` causes race conditions that are hard to debug.
2. Type annotations on `Page`, `Browser`, `BrowserContext`, and `Locator` catch method name typos and wrong argument types at compile time.
3. Interfaces define the shape of test data, preventing missing fields and type mismatches in test fixtures.
4. Playwright's generic methods (like `locator().all()`) infer collection types automatically, giving you full autocompletion.
5. A properly configured `tsconfig.json` with `"strict": true` and `"moduleResolution": "bundler"` is essential for Playwright type resolution.

## Recap

- **Type annotations** give you compile-time safety for all Playwright API calls.
- **Async/await** is the required pattern for every browser interaction in Playwright tests.
- **Interfaces and generics** structure test data and collection operations so the compiler validates your code before it ever runs.

## Quiz

### Multiple Choice Questions

**1. What is the return type of `page.locator('.my-class').allTextContents()`?**

- A) `Promise<string>`
- B) `Promise<string[]>`
- C) `string[]`
- D) `Locator[]`

<details>
<summary>Answer</summary>
B — `Promise<string[]>`. The method returns a promise that resolves to an array of strings.
</details>

**2. Which field in `compilerOptions` is required for Playwright type resolution in modern projects?**

- A) `"target": "ES5"`
- B) `"moduleResolution": "bundler"`
- C) `"jsx": "react"`
- D) `"sourceMap": true`

<details>
<summary>Answer</summary>
B — `"moduleResolution": "bundler"` resolves Playwright's export maps correctly.
</details>

**3. How do you import a type from `@playwright/test` without including it in the runtime bundle?**

- A) `import { Page } from '@playwright/test'`
- B) `import type { Page } from '@playwright/test'`
- C) `const Page = require('@playwright/test').Page`
- D) Types are always bundled; you cannot exclude them

<details>
<summary>Answer</summary>
B — The `type` keyword tells TypeScript to erase the import during compilation.
</details>

**4. What does TypeScript report when you write `page.goto('/')` without `await`?**

- A) No error — missing await is valid TypeScript
- B) `Type 'Promise<void>' is not assignable to type 'void'`
- C) `Cannot find name 'goto'`
- D) `'await' has no effect on this kind of expression`

<details>
<summary>Answer</summary>
B — TypeScript warns that the expression returns `Promise<void>` but you are treating it as `void`.
</details>

**5. Which operator should you use to handle `null` from `textContent()`?**

- A) Optional chaining `?.`
- B) Nullish coalescing `??`
- C) Logical OR `||`
- D) Ternary `? :`

<details>
<summary>Answer</summary>
B — Nullish coalescing (`textContent() ?? ''`) converts only `null` or `undefined`, preserving empty strings.
</details>

### Scenario Questions

**Scenario 1:** You have a helper function that gets the current user's display name from a profile page. It returns `string | null` but the calling code expects `string`. Without changing the return type of the helper, how do you handle this in the caller?

<details>
<summary>Answer</summary>
Use the nullish coalescing operator: `const name = (await getDisplayName(page)) ?? 'Guest';`. This provides a default value while keeping the helper's return type honest about the possibility of null.
</details>

**Scenario 2:** You are migrating a 50-test Playwright suite from JavaScript to TypeScript. Many tests use `page.$('.selector')` which is deprecated in favor of `page.locator('.selector')`. TypeScript warns on every deprecated call. What is the most efficient migration strategy?

<details>
<summary>Answer</summary>
Enable `noImplicitOverride` and address the deprecation warnings one file at a time. Start by replacing `$` with `locator` in the most frequently used selectors first, then run `npx playwright test` after each batch. Use `tsconfig.json` with `"strict": true` and the `"paths"` compiler option to incrementalize the migration if needed.
</details>
