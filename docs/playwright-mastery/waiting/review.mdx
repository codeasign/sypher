# Review — Waiting Strategies

## Key Takeaways

1. **Playwright auto-waits before every action** — it checks visibility, enabled state, stability (no animations), unobscured, and event reception. You do not need to add waits before clicks, fills, or other interactions.
2. **Use explicit waits for conditions outside the action flow** — `waitForURL` for navigation, `waitForResponse` for API calls, `waitFor({ state: 'hidden' })` for spinners and modals, and `waitForFunction` for custom JavaScript conditions.
3. **Always create `waitForResponse` promises before triggering the request** — if you create the promise after the action, the response may arrive before Playwright starts listening, causing a timeout.
4. **Never use `waitForTimeout` in production tests** — fixed sleeps are the primary cause of flaky tests. Every fixed sleep can be replaced with a deterministic wait for an actual condition.
5. **Avoid `waitForLoadState('networkidle')` on modern web apps** — pages with polling, WebSocket connections, or streaming data may never reach the idle state. Use `load` or `domcontentloaded` instead and wait for specific elements.

## Recap

In this lesson you learned how Playwright's auto-waiting mechanism handles the five actionability checks before every user action. ### Scenario Questions

**Scenario 1:** You have a test that clicks a "Generate Report" button. The button triggers an API call to `/api/report` that takes 5-15 seconds, then the page navigates to `/reports/{id}`. The API returns a JSON with the report ID. Your current test uses `page.waitForTimeout(20000)` and is flaky. How would you rewrite this test using deterministic waiting?

**Answer:** Create a `waitForResponse` promise for `**/api/report` before clicking the button. After the click, await the response promise, extract the report ID from the response body, then use `waitForURL` to wait for navigation to `**/reports/*`. This approach adapts to the actual timing regardless of whether the API takes 5 seconds or 15 seconds.

```typescript
const reportResponsePromise = page.waitForResponse(
  resp => resp.url().includes('/api/report') && resp.status() === 200
);
await page.getByRole('button', { name: 'Generate Report' }).click();
const reportResponse = await reportResponsePromise;
const { id } = await reportResponse.json();
await page.waitForURL(`**/reports/${id}`);
```

**Scenario 2:** Your test navigates to a dashboard that shows real-time stock ticker data. The page uses WebSocket connections for live updates and polls a health endpoint every 30 seconds. Your test hangs when using `waitForLoadState('networkidle')`. What alternative approach should you use, and why is it more reliable?

**Answer:** Replace `waitForLoadState('networkidle')` with `waitForLoadState('domcontentloaded')` or `'load'` combined with element-level waits. The `networkidle` state requires 500ms of zero network activity, which never occurs on a page with persistent WebSocket connections. By waiting for specific elements (like a dashboard title or a data table) to become visible, you verify the page is ready for interaction regardless of ongoing background network activity.
