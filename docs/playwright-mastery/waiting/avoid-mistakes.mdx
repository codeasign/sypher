# Avoid Common Waiting Mistakes

## When NOT to Use This

Avoid using explicit waiting in these scenarios:

- **Do not use `waitForTimeout` in production tests.** Fixed sleeps are always an anti-pattern. They make your tests slower on fast machines and flaky on slow ones. The only acceptable use is during debugging to inspect page state manually.
- **Do not call `waitForResponse` after the action that triggers the request.** If the API responds faster than Playwright registers the listener, the response is missed and the wait times out. Always create the promise before the trigger.
- **Do not use `waitForLoadState('networkidle')` on modern single-page applications or streaming pages.** The `networkidle` event waits until no network requests have been made for 500ms. Pages that poll for updates, maintain WebSocket connections, or stream analytics data may never reach this state, causing your test to hang.

## The Pattern to Remember

Let Playwright's auto-wait handle every user action (clicking, typing, selecting). Auto-wait already verifies that the target element is visible, enabled, stable, and unobstructed — you do not need to add explicit waits before actions.

Reserve explicit waits for conditions that auto-wait does not cover:

- **State changes on other elements** — `locator.waitFor({ state: 'hidden' })` to wait for a spinner to vanish
- **URL changes** — `page.waitForURL(pattern)` after navigation triggers
- **Network events** — `page.waitForResponse(predicate)` to coordinate with API data
- **Custom JavaScript conditions** — `page.waitForFunction(predicate)` for one-off browser-side checks

## Common Mistakes

Here are the five most common waiting mistakes in Playwright tests.

### Mistake 1: Using `waitForTimeout` Instead of a Real Condition

```typescript
// BAD — flaky and slow
await page.getByRole('button', { name: 'Load' }).click();
await page.waitForTimeout(5000);
await expect(page.getByText('Data loaded')).toBeVisible();

// GOOD — waits for the actual condition
await page.getByRole('button', { name: 'Load' }).click();
await expect(page.getByText('Data loaded')).toBeVisible({ timeout: 10000 });
```

The assertion `toBeVisible` already waits with its own timeout. By using `waitForTimeout`, you are both delaying the test unnecessarily on fast machines and still risking flakiness if 5 seconds is not enough on a slow runner.

### Mistake 2: Forgetting Auto-Wait Only Applies to Actions

Auto-wait only protects Playwright actions like `click`, `fill`, `selectOption`, `check`, and `dblclick`. It does NOT apply when you evaluate locators with methods like `locator.all()`, `locator.count()`, or `locator.evaluate()`.

```typescript
// BAD — locator.count() does not auto-wait
const buttonCount = await page.locator('.dynamic-button').count();
expect(buttonCount).toBe(5); // May fail if buttons haven't rendered yet

// GOOD — use assertion that auto-waits
await expect(page.locator('.dynamic-button')).toHaveCount(5);
```

Assertions from `expect` also auto-wait and retry, so use them instead of counting manually.

### Mistake 3: Setting Up Response Wait After Triggering the Action

```typescript
// BAD — race condition: response may arrive before the promise is created
await page.getByRole('button', { name: 'Search' }).click();
const response = await page.waitForResponse('**/api/search'); // May time out

// GOOD — create promise before the trigger
const responsePromise = page.waitForResponse('**/api/search');
await page.getByRole('button', { name: 'Search' }).click();
const response = await responsePromise;
```

The response promise must be registered in Playwright's internal listener list before the browser sends the request. If the API responds in under 50ms, the race is lost.

### Mistake 4: Not Setting Custom Timeouts for Slow Operations

The default timeout for wait methods is 30 seconds. Some operations — like file uploads, large data exports, or third-party widget initialization — may need more time, especially on CI.

```typescript
// BAD — default 30s may not be enough for a large export
await page.getByRole('button', { name: 'Export CSV' }).click();
await page.waitForURL('**/exports/*'); // Times out on slow CI

// GOOD — explicit timeout for known slow operation
await page.getByRole('button', { name: 'Export CSV' }).click();
await page.waitForURL('**/exports/*', { timeout: 60000 });
```

Conversely, if a condition should appear quickly (under 5 seconds), shorten the timeout so the test fails fast rather than waiting 30 seconds.

### Mistake 5: Using `waitForLoadState('networkidle')` on Streaming Pages

```typescript
// BAD — may hang on pages with polling or WebSocket connections
await page.goto('https://app.example.com/dashboard');
await page.waitForLoadState('networkidle');

// GOOD — wait for a concrete element instead
await page.goto('https://app.example.com/dashboard');
await expect(page.getByText('Dashboard Metrics')).toBeVisible({ timeout: 15000 });
```

Modern applications often maintain long-lived connections for real-time updates. The `networkidle` event expects a 500ms pause with zero network activity, which may never occur. Use `load` or `domcontentloaded` for the load state and wait for specific elements for readiness.

## Troubleshooting

| Symptom | Likely Cause | Solution |
|---|---|---|
| `waitForURL` times out | The navigation never happened, or the URL pattern does not match | Use `page.on('popup')` if navigation opens a new tab; check the actual URL with `console.log(page.url())` |
| `waitForResponse` never resolves | Response was sent before the promise was created, or the predicate does not match | Move the promise creation before the trigger action; add `console.log` in the predicate to debug matching |
| Element is visible but click still fails | Another element is covering the target (overlay, modal backdrop) | Check for overlapping elements; use `{ force: true }` only as a last resort after confirming the issue |
| `waitForLoadState` hangs on `networkidle` | Page has persistent connections (WebSocket, SSE, polling) | Switch to `load` or `domcontentloaded` and use element-level waits instead |
| Timeout on lazy content | Content loads only when scrolled into view or after a user gesture | Scroll the element into view first with `locator.scrollIntoViewIfNeeded()` |

## Production Checklist

Before considering your test suite ready for CI, verify these items:

- [ ] No `waitForTimeout` calls exist in any test file (use `grep -r 'waitForTimeout' tests/` to verify)
- [ ] Every explicit wait targets a specific condition (element state, URL pattern, network event, or JS condition)
- [ ] All wait promises are created before their triggering actions to avoid race conditions
- [ ] Timeout values are appropriate for CI environments (default 30s, custom higher values for known slow operations)
- [ ] Tests do not rely on `waitForLoadState('networkidle')` — they use element-level or URL waits instead
- [ ] Slow assertions have explicit `{ timeout: ... }` values rather than relying on the default
