# Waiting Strategies in Playwright

Difficulty: Intermediate.

## What You Will Learn

- Understand Playwright's auto-waiting mechanism and the five actionability checks it performs before every action
- Use explicit `waitFor` methods to synchronize with elements, navigation, and network events
- Leverage network-aware waits (`waitForRequest`, `waitForResponse`) to coordinate with backend API calls
- Eliminate flaky fixed-time sleeps from your tests by adopting deterministic waiting patterns

## Prerequisites

Before taking this lesson, you should have completed the following Playwright Mastery lessons:

- [Locators](../locators/overview) — selecting elements on the page
- [Actions](../actions/overview) — clicking, typing, and interacting with elements
- [Assertions](../assertions/overview) — verifying expected conditions

## Why This Matters

Incorrect waiting is the root cause of approximately 90% of flaky end-to-end tests. When you use `page.waitForTimeout(3000)` you are guessing how long something will take — and that guess will fail on a slow CI runner, a throttled network, or a machine under load. Playwright's built-in auto-waiting eliminates the guesswork by actively checking conditions before every action, but you need to understand how it works and when to reach for explicit waits to handle edge cases that auto-wait cannot cover.

## Quick Reference

| Method | Purpose | Common Use Case |
|---|---|---|
| `locator.waitFor({ state: 'visible' })` | Wait for element to reach a specific state | Waiting for a modal or spinner to appear or disappear |
| `page.waitForURL('**/dashboard')` | Wait for navigation to a matching URL | After clicking a link or submitting a form |
| `page.waitForLoadState('domcontentloaded')` | Wait for page load event | After navigation, when you need a specific load milestone |
| `page.waitForRequest(urlOrPredicate)` | Wait for a matching outgoing network request | Confirming an API call was made |
| `page.waitForResponse(urlOrPredicate)` | Wait for a matching incoming network response | Waiting for API data before asserting |
| `page.waitForTimeout(ms)` | Fixed sleep (debugging only) | Never use in production tests; use for manual debugging |
| `page.waitForFunction(predicate)` | Wait for a custom JavaScript condition | Polling `document.title`, a global variable, or DOM text |

Playwright's auto-waiting is not a simple timeout — it actively polls the page up to the configured timeout (default 30 seconds) until all five actionability checks pass. If any check fails, Playwright retries the entire sequence, so a brief animation or a delayed render does not cause a failure as long as it resolves within the timeout window.
