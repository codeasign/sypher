# Authentication — Review

## Key Takeaways

1. **Authentication should happen once, before tests** — use a setup project with `dependencies` to log in via API and save `storageState`. Tests load the saved state and start already authenticated.
2. **storageState captures cookies and localStorage** — after a successful login, call `page.context().storageState({ path })` to persist the session. Tests load this state automatically via `test.use({ storageState })`.
3. **Multi-role testing needs separate storage states** — create one setup file and one `.auth/` file per role (admin, user, guest). Apply the correct state per test file with `test.use()`.
4. **Credentials must never be in the codebase** — use environment variables, `.env` files (gitignored), and CI secret managers. Rotate secrets without code changes.
5. **Session expiry is a real concern in long suites** — plan for tokens that expire mid-run. Use short suites, refresh tokens in fixtures, or add re-login fallback logic to individual tests.

### Scenario Questions

**Scenario 1:** Your Playwright test suite has 150 tests and takes 25 minutes to run. Halfway through, tests start failing with 401 errors. Investigation shows the app's session tokens expire after 15 minutes.

**Question:** Propose two different solutions. For each, explain the trade-offs (complexity, speed, reliability).

**Answer:** 
- **Solution A: Refresh storageState mid-suite.** Add a global fixture that checks for 401 and re-runs the setup login. Pro: no structural changes. Con: adds complexity to fixtures, each test needs the check logic.
- **Solution B: Shard the suite so each shard runs in under 10 minutes.** With 4 shards, each runs 38 tests in about 6 minutes — well under the 15-minute token expiry. Pro: simple, no auth changes needed. Con: requires CI matrix changes, more CI minutes.

**Scenario 2:** A developer on your team committed `.auth/admin.json` to the repository. The file contains session cookies that grant admin access to your staging environment. A security scanner flagged it.

**Question:** Describe the immediate remediation steps and the long-term process change needed to prevent this from happening again.

**Answer:**
**Immediate:** (1) Rotate the staging app's session secret and all API tokens. (2) Force-push to remove the file from git history using `git filter-branch` or `bfg repo-cleaner`. (3) Notify team and update secrets.

**Long-term:** (1) Add `.auth/` to `.gitignore` in the project root and verify with a pre-commit hook (`git-secrets` or `husky` with a check). (2) Add a CI step that fails if any `.auth/` files or credential patterns appear in the commit. (3) Enforce code review checklist item: "No credentials or auth state files in source."
