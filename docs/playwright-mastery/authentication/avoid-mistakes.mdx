# Authentication — Avoid Mistakes

## When NOT to Use This

- **OAuth flows with redirects** — if your app uses OAuth (Google Login, GitHub Login) that requires browser redirects, `storageState` alone may not capture the full session. You need to handle the redirect flow in your setup script and save state after the full OAuth dance completes.
- **Credentials in the repo** — never commit usernames, passwords, API keys, or tokens to version control. Always use environment variables, CI secrets, or a secret manager. A leaked `.auth/` directory in git exposes your test infrastructure.
- **Short-lived tokens** — if your session tokens expire in under 5 minutes, `storageState` is not useful. Regenerate the state on a per-test basis or use API tokens that refresh automatically.

## The Pattern to Remember

API login to storageState to project dependency to authenticated tests.

```
Credentials (env vars)
    |
    ▼
auth.setup.ts (API login)
    |
    ▼
storageState file (.auth/user.json)
    |
    ▼
project.dependencies → ['setup']
    |
    ▼
test.use({ storageState: '.auth/user.json' })
    |
    ▼
Every test starts already authenticated
```

The key insight: your tests should never call a login function. Authentication happens before any test runs, via the setup project.

## Common Mistakes

### Mistake 1: Credentials Hardcoded in Test Files

Hardcoding credentials makes the codebase insecure and impossible to rotate secrets without a deploy.

```typescript
// BAD — credentials in source code
const USERNAME = 'admin@sypher.com';
const PASSWORD = 'SuperSecret123!';

// GOOD — credentials in environment variables
const USERNAME = process.env.TEST_USERNAME;
const PASSWORD = process.env.TEST_PASSWORD;
if (!USERNAME || !PASSWORD) {
  throw new Error('TEST_USERNAME and TEST_PASSWORD must be set');
}
```

Use a `.env` file (gitignored) and load it with `dotenv`:

```bash
npm install --save-dev dotenv
```

```typescript
// playwright.config.ts
import { defineConfig } from '@playwright/test';
import dotenv from 'dotenv';

dotenv.config(); // loads from .env file

export default defineConfig({
  // ...
});
```

### Mistake 2: storageState Not Generated Before Tests Run

If a test project runs before its setup project finishes (or never runs it), Playwright throws "storageState file not found." This happens when you forget the `dependencies` config.

```typescript
// BAD — no dependency declared
{
  name: 'chromium',
  use: { storageState: '.auth/user.json' },
  // Missing: dependencies: ['setup']
}

// GOOD — setup project runs first
{
  name: 'setup',
  testMatch: 'auth.setup.ts',
},
{
  name: 'chromium',
  dependencies: ['setup'],
  use: { storageState: '.auth/user.json' },
}
```

### Mistake 3: Session Expires During Test Suite

Some applications issue tokens that expire in 15-30 minutes. If your suite runs longer, tests that run late in the suite will get 401 errors.

```typescript
// Handle expired sessions gracefully
import { test, expect } from '@playwright/test';

test.use({ storageState: '.auth/user.json' });

test('long-running test with possible expiry', async ({ page }) => {
  await page.goto('/dashboard');

  // If session expired, re-login
  if (await page.locator('[data-testid="login-form"]').isVisible()) {
    await page.fill('[name="email"]', process.env.TEST_USERNAME!);
    await page.fill('[name="password"]', process.env.TEST_PASSWORD!);
    await page.click('[data-testid="login-button"]');
  }

  await expect(page.locator('[data-testid="dashboard"]')).toBeVisible();
});
```

For long suites, regenerate `storageState` mid-suite or use short-lived tokens with a refresh mechanism in a fixture.

### Mistake 4: storageState Shared Across Roles

Using the same storage state file for admin and regular user tests means admin-only pages may accidentally pass in user tests, or user-only pages may fail in admin tests.

```typescript
// BAD — single storage state for all roles
test.use({ storageState: '.auth/user.json' }); // admin tests also use this

// GOOD — different storage state per role
test.use({ storageState: '.auth/admin.json' }); // admin tests
test.use({ storageState: '.auth/user.json' });  // user tests
test.use({ storageState: '.auth/guest.json' }); // guest tests
```

Create a separate setup test and storage state file for each role, then use `test.use({ storageState })` per test file or describe block.

### Mistake 5: Not Handling MFA / 2FA in Auth Setup

If your application uses multi-factor authentication, your API login flow in `auth.setup.ts` will fail because it only sends username and password. MFA requires handling the second factor.

```typescript
// auth.setup.ts with MFA support
import { test as setup, expect } from '@playwright/test';
import crypto from 'crypto';

setup('authenticate with MFA', async ({ page }) => {
  // Step 1: Login with credentials
  const loginResponse = await page.request.post(
    'https://example.com/api/login',
    {
      data: {
        username: process.env.TEST_USERNAME,
        password: process.env.TEST_PASSWORD,
      },
    }
  );
  expect(loginResponse.ok()).toBeTruthy();

  // Step 2: Handle MFA challenge
  // Option A: Use a TOTP secret to generate the code
  const totpSecret = process.env.TEST_TOTP_SECRET;
  if (totpSecret) {
    const totpCode = generateTOTP(totpSecret); // your TOTP utility
    await page.request.post('https://example.com/api/mfa/verify', {
      data: { code: totpCode },
    });
  }

  // Option B: Use API endpoint that bypasses MFA for test users
  // (preferred — ask your backend team to provide this)

  await page.context().storageState({ path: '.auth/user.json' });
});

function generateTOTP(secret: string): string {
  // Simplified TOTP generation — use a library like otplib
  const time = Math.floor(Date.now() / 30000);
  const hmac = crypto
    .createHmac('sha1', Buffer.from(secret, 'base64'))
    .update(Buffer.from(time.toString(16).padStart(16, '0'), 'hex'))
    .digest();
  // Extract 6-digit code from HMAC (simplified)
  return (hmac.readUInt32BE(hmac.length - 4) % 1000000)
    .toString()
    .padStart(6, '0');
}
```

## Troubleshooting

| Symptom | Likely Cause | Solution |
|---|---|---|
| storageState file not found | Setup project ran after test project | Add `dependencies: ['setup']` to test projects |
| 401 Unauthorized during test | Session expired or never established | Check storageState path, token expiry, or regenerate state |
| Session expired mid-suite | Token TTL shorter than suite runtime | Add session refresh logic or regenerate storageState |
| Cross-origin cookies not saved | Login API and app are on different origins | Use `storageState` from the context after visiting the main origin |
| OAuth redirect not followed | Setup only calls API, not browser redirect | Perform full OAuth flow in the browser during setup |

## Production Checklist

- [ ] Credentials in environment variables or CI secrets, never committed
- [ ] Setup project declared with `dependencies` in all test projects
- [ ] Separate storageState files per role (admin, user, guest)
- [ ] Session refresh mechanism for suites longer than token TTL
- [ ] `.auth/` directory added to `.gitignore`
- [ ] MFA/2FA handled for test users (or bypass endpoint available)
- [ ] `dotenv` or similar loaded before config is evaluated
- [ ] Auth setup script handles API failures gracefully
