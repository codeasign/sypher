# Avoid Mistakes with Self-Healing Locators

## When NOT to Use This

- **Elements that must exist uniquely**: If a page should only ever have one "Delete" button, a single robust locator is better than `.or()` — the fallback might silently match a different element.
- **Stable, well-tested components**: Mature design systems with stable `data-testid` attributes don't need fallbacks. Over-engineering adds complexity without benefit.
- **Test setup pages**: Login forms, onboarding flows, and other rarely-changed pages don't need healing chains.
- **Accessibility-critical tests**: Use `getByRole` alone so test failures surface accessibility issues — fallbacks can mask missing roles.

## The Pattern to Remember

- `.or()` for handling UI variants (A/B tests, multi-brand apps).
- Custom selector engines for project-wide fallback chains registered once in global setup.
- Always order fallbacks from most specific to least specific (role > testid > text > CSS).

## Common Mistakes

**1. `.or()` matches multiple elements and the test clicks the wrong one**

When both sides of `.or()` match different elements on the page, Playwright picks the first one found in DOM order. This can silently click an unintended target.

```typescript
// BAD: both "Cancel" and "Delete" buttons match
const button = page.getByRole('button', { name: 'Cancel' })
  .or(page.getByRole('button', { name: 'Delete' }));
```

**2. Fallback masks a genuine failure that should be investigated**

A button should use `data-testid`, but the test passes because the CSS fallback matches. The missing testid never gets noticed.

```typescript
// BAD: the failing role is never flagged
const btn = page.getByTestId('order-btn')
  .or(page.locator('.btn-primary'));
// If getByTestId fails silently, no one adds the testid
```

**3. Custom selector engines are not cleaned up between tests**

Engines registered in one test persist across the file unless explicitly unregistered.

```typescript
// Register in test A
await selectors.register('my-engine', { query: ..., queryAll: ... });
// Test B may use the wrong engine or conflict
```

**4. Using `.or()` when a single robust selector works**

A well-chosen `getByRole` or `getByTestId` is simpler and more maintainable than a three-link chain.

```typescript
// BAD: unnecessary complexity
const btn = page.getByRole('button').or(page.getByText('Submit')).or(page.locator('#submit'));
// GOOD: single robust locator
const btn = page.getByRole('button', { name: 'Submit' });
```

**5. Not testing all fallback paths**

You wrote a chain of 4 fallbacks but only tested the first one. The second selector is broken and no one knows until it's the last resort in production.

```typescript
// If you never test each fallback independently, broken selectors hide
const btn = page.getByRole('button', { name: 'Save' })
  .or(page.locator('[data-testid="save-btn"]'))  // This could be stale
  .or(page.getByText('Save'));                     // This too
```

## Troubleshooting

| Symptom | Likely Cause | Solution |
|---|---|---|
| `.or()` times out after 5s | None of the fallback selectors match any element | Verify each selector independently with `locator.waitFor({ state: 'visible' })` |
| Custom selector not found | Engine not registered, or selector format doesn't match `query()` implementation | Add `console.log` inside `query()` and check terminal output |
| Multiple elements match OR condition | Both sides of `.or()` match different elements on the page | Narrow selectors with `{ name: /exact text/i }` or use `.first()` |
| `.or()` always matches the first variant | Playwright scans DOM in order; it's by design | Reorder selectors or use `.filter()` for precise matching |
| Custom engine works locally but not in CI | Engine registration depends on browser context teardown timing | Register in `globalSetup` or `test.beforeAll` at the top level |

## Production Checklist

- [ ] Fallback order is documented in the test file or a shared constants file
- [ ] Each fallback selector is tested independently (at least one test per variant)
- [ ] Custom engine registration lives in a shared setup file (`global-setup.ts` or `test.beforeAll`)
- [ ] `.or()` chains have at most 3 alternatives — beyond that, use a custom engine
- [ ] Team convention is agreed upon: when to use `.or()` vs custom engine vs single locator
