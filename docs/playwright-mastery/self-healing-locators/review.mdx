# Self-Healing Locators Review

## Key Takeaways

1. **Use `.or()` for UI variants**: Chain locators with `.or()` to handle A/B tests, multi-brand UIs, and evolving component text without rewriting tests.
2. **Custom selector engines for project-wide patterns**: Register an engine once in global setup and use it everywhere — the engine encapsulates your fallback strategy.
3. **Order fallbacks from most to least specific**: Role or testid first, text content second, CSS selector last. This ensures the most reliable strategy wins.
4. **Test every fallback path independently**: Each selector in your chain should have its own test case so broken selectors don't hide until production.
5. **Don't over-engineer**: Stable components with good `data-testid` attributes don't need fallback chains. Add complexity only when DOM changes are a real concern.

### Scenario Questions

**Scenario 1**: You have a login page with a "Sign In" button that sometimes renders as `<button>Sign In</button>` and other times as `<button data-testid="login">Sign In</button>`. Write a self-healing locator chain that handles both cases. What happens if a third variant appears with `<a class="btn-login">Sign In</a>`?

**Scenario 2**: Your team's custom selector engine "heal" is registered in `test.beforeAll` but intermittently fails in CI. Tests pass locally. What could be different between local and CI execution that causes the engine to be unavailable?

### Answer Key

**MCQ Answers**: 1-B, 2-C, 3-B, 4-D, 5-B

**Scenario 1**: The chain `page.getByRole('button', { name: /sign in/i }).or(page.getByTestId('login'))` handles the first two variants. For the anchor tag variant, you would add `.or(page.locator('.btn-login'))` — but first check if the `<a>` has `role="button"`, in which case `getByRole` may already match it.

**Scenario 2**: Possible causes: the `test.beforeAll` hook runs in a different worker and the engine registration doesn't propagate; the CI browser version differs and the engine's `query()` function behaves differently; or the global setup file isn't included in the CI config. Move registration to `playwright.config.ts` using `globalSetup` to ensure it runs once before all workers.
