# Avoid Common Mistakes: Playwright CLI Agent

## When NOT to Use This

Do not rely on generated code without reviewing it. Codegen records exactly what you do in the browser, but it does not understand the intent behind the actions. It cannot add assertions, handle error states, or choose the best locator strategy. Generated code is a starting point, not a finished test.

Also avoid using codegen for tests that require complex assertions or custom waiting patterns. Codegen records actions, not expectations. If your test needs to wait for a specific API response, verify a database state, or check an element's attribute value, you need to add those assertions manually.

## The Pattern to Remember

Use codegen for locator discovery and initial flow recording, then manually add assertions, edge cases, and error handling. The pattern is: generate the happy path with codegen, then harden the test by hand.

## Common Mistakes

**1. Using generated code as-is without adding assertions**

Codegen records page interactions but does not add `expect` calls. If you use the generated code directly, your test navigates and clicks but never verifies anything. A passing test that does not assert is worthless. Always add at least one assertion per test that validates the outcome of the action.

**2. Not setting the `--target` flag**

By default, codegen generates JavaScript, not TypeScript. If your project uses TypeScript, you must pass `--target=typescript` to get `.ts` output with proper types:

```bash
npx playwright codegen --target typescript -o tests/login.spec.ts
```

**3. Generating tests against production instead of a local or test environment**

If you run codegen against production, you risk modifying real data, triggering real emails, or interfering with real users. Always run codegen against a local or staging instance of your application. Use environment variables or configuration to switch between targets.

**4. Not cleaning up generated locators**

Codegen defaults to CSS selectors. These can be brittle. A CSS selector like `#__next > div > form > input:nth-child(2)` breaks on the smallest markup change. Refactor generated locators to use best practices: `getByRole`, `getByLabel`, `getByTestId`, or `getByText`.

**5. Not regenerating tests after UI changes**

If your application's UI changes, generated locators may no longer match. The CSS class that codegen recorded might have changed, or the element's text might have been updated. After any UI change, regenerate the affected tests with codegen and update locators accordingly.

## Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| codegen does not open a browser | Playwright browsers not installed | Run `npx playwright install` |
| Generated locators are CSS, not role-based | Default codegen behavior | Convert locators to `getByRole` or `getByLabel` manually; use `--locators=role` in newer Playwright versions |
| Codegen records clicks on wrong elements | Page interactions happen too fast; elements shift before click | Slow down your interactions; codegen records what Playwright sees, not what you intended |
| `--target=javascript` generates JS but project uses TypeScript | Flag omitted or mis-typed | Use `--target typescript` (or `--lang=TypeScript` depending on version) |
| Generated tests fail when re-run | Dynamic content, async loading, or element timing | Add `waitFor` assertions or `toBeVisible` checks; break long action sequences with verifications |

## Production Checklist

- [ ] Use `--target typescript` when generating tests in a TypeScript project
- [ ] Review every generated test before committing it
- [ ] Add assertions to validate that actions had the expected effect
- [ ] Convert CSS-based locators to role-based or testid-based locators
- [ ] Add error handling for timeouts and unexpected states
- [ ] Regenerate tests after significant UI changes
