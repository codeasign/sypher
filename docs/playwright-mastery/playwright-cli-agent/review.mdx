# Review: Playwright CLI Agent

## Key Takeaways

1. The `npx playwright codegen` command opens a browser and inspector side by side, recording every interaction as Playwright code.
2. Codegen does not add assertions by default — you must always add `expect` calls to validate behavior.
3. Generated locators default to CSS selectors; refactor them to `getByRole`, `getByLabel`, or `getByTestId` for more robust tests.
4. Use flags like `--device`, `--viewport-size`, `--target`, and `-o` to control codegen output.
5. Codegen is a starting point for test creation, not a finished product — always review, refactor, and harden generated code.

### Scenario Questions

**Scenario 1:** A developer on your team runs codegen against the production website, records a complete checkout flow, and commits the generated test without changes. What problems could arise?

**Answer:**
- The developer modified real production data (placed an order, created an account)
- The test has no assertions, so it validates nothing
- The generated CSS locators will break when the production UI is updated
- The test relies on production API behavior and real payment flows
- The test cannot run in CI against production safely
- The correct approach is to run codegen against a local or staging environment, add assertions, refactor locators, and only commit the hardened version

**Scenario 2:** Your codegen-generated test passes locally but fails in CI. The error is that the element cannot be found. What could cause this?

**Answer:**
- The CI environment uses a different viewport size, causing responsive layout changes that move the element
- The CI browser is a different engine (Chromium vs WebKit) with different rendering
- The CI navigates to a different base URL
- Dynamic content loads at different speeds in CI (slower network means elements appear later)
- The CI is running against a different data set, so expected text differs
- The generated CSS selectors matched locally but are not stable across environments
