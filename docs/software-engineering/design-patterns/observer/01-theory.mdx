## Intent

Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically. The Observer pattern decouples the subject (the object being observed) from its observers (the dependents), allowing both to vary independently. This is the foundational pattern for event-driven and publish-subscribe systems.

## The Problem Without This Pattern

Imagine you are building a stock market monitoring application. A `StockExchange` class tracks the latest prices for AAPL, GOOGL, and MSFT. Several other components need to react to price changes: a `PriceDisplay` that shows the latest price on screen, an `AlertSystem` that triggers when a stock moves more than 5%, a `PortfolioTracker` that recalculates total portfolio value, and a `Logger` that records every price change for audit.

Without the Observer pattern, the `StockExchange` class would need to know about every component that cares about price changes:

```typescript
// BROKEN: tight coupling — StockExchange must know all dependents
class StockExchange {
  private priceDisplay: PriceDisplay;
  private alertSystem: AlertSystem;
  private portfolioTracker: PortfolioTracker;
  private logger: Logger;

  setPrice(ticker: string, price: number) {
    this.data[ticker] = price;
    this.priceDisplay.update(ticker, price);
    this.alertSystem.check(ticker, price);
    this.portfolioTracker.recalculate(ticker, price);
    this.logger.log(ticker, price);
  }
}
```

Every time a new dependent component is added (e.g., a `ChartRenderer`), you must modify the `StockExchange` class. Dependents cannot be added or removed at runtime. The subject is tightly coupled to every concrete observer, making the system rigid and hard to test.

## Structure

The Observer pattern defines two core participants: the Subject (Observable) that maintains a list of observers, and the Observer interface that all dependents implement.

### Participants

1. **Subject** — Maintains a list of observers. Provides methods to attach and detach observers. Defines a `notifyObservers()` method that calls `update()` on every registered observer.
2. **Observer** — Interface or abstract class defining the `update()` method that the Subject calls when its state changes.
3. **ConcreteSubject** — Stores the actual state of interest. When its state changes, it calls `notifyObservers()` (inherited or implemented from Subject).
4. **ConcreteObserver** — Implements the Observer interface. Stores a reference to the ConcreteSubject (or receives the data via the `update()` call). Maintains state consistent with the subject's state.

## Pseudocode — Stock Market Example

The following pseudocode demonstrates the Observer pattern in a stock market domain. A `StockExchange` (ConcreteSubject) tracks stock prices. Multiple observers react to price changes: `PriceDisplay` shows prices, `AlertSystem` checks for threshold breaches, and `PortfolioTracker` updates portfolio values.

```
// PSEUDOCODE — Observer Pattern in Stock Market Context
// Participants:
//   Observer          — interface with update(ticker, price)
//   StockExchange     — ConcreteSubject
//   PriceDisplay      — ConcreteObserver
//   AlertSystem       — ConcreteObserver
//   PortfolioTracker  — ConcreteObserver

// ── Observer Interface ──
interface Observer:
    method update(ticker: string, price: float): void

// ── Subject ──
class StockExchange:
    // Instance variables
    observers: List<Observer>
    prices: Map<string, float>

    // Constructor
    method StockExchange():
        this.observers = new List()
        this.prices = new Map()

    // Registration
    method attach(observer: Observer):
        this.observers.add(observer)

    method detach(observer: Observer):
        this.observers.remove(observer)

    // Notification
    method notifyObservers(ticker: string, price: float):
        for each observer in this.observers:
            observer.update(ticker, price)

    // Business method — triggers notification
    method setPrice(ticker: string, price: float):
        this.prices[ticker] = price
        this.notifyObservers(ticker, price)

    // Query
    method getPrice(ticker: string) -> float:
        return this.prices.getOrDefault(ticker, 0.0)

// ── Concrete Observer 1: PriceDisplay ──
class PriceDisplay implements Observer:
    method update(ticker: string, price: float):
        print("DISPLAY: {ticker} is now ${price:.2f}")

// ── Concrete Observer 2: AlertSystem ──
class AlertSystem implements Observer:
    // Stores previous prices to detect changes
    previousPrices: Map<string, float>
    threshold: float

    method AlertSystem(threshold: float):
        this.previousPrices = new Map()
        this.threshold = threshold

    method update(ticker: string, price: float):
        if this.previousPrices.containsKey(ticker):
            oldPrice = this.previousPrices[ticker]
            change = abs(price - oldPrice) / oldPrice * 100
            if change >= this.threshold:
                print("ALERT: {ticker} moved {change:.1f}%")
        this.previousPrices[ticker] = price

// ── Concrete Observer 3: PortfolioTracker ──
class PortfolioTracker implements Observer:
    holdings: Map<string, int>  // ticker -> shares owned
    totalValue: float

    method PortfolioTracker():
        this.holdings = new Map()
        this.totalValue = 0.0

    method addHolding(ticker: string, shares: int):
        this.holdings[ticker] = shares

    method update(ticker: string, price: float):
        if this.holdings.containsKey(ticker):
            shares = this.holdings[ticker]
            // Recalculate total portfolio value
            this.totalValue = 0.0
            for each holding in this.holdings:
                this.totalValue += holding.shares * exchange.getPrice(holding.ticker)
            print("PORTFOLIO: Updated value is ${totalValue:.2f}")

// ── Main ──
method main():
    exchange = new StockExchange()
    display = new PriceDisplay()
    alerts = new AlertSystem(5.0)  // 5% threshold
    portfolio = new PortfolioTracker()

    // Register observers
    exchange.attach(display)
    exchange.attach(alerts)
    exchange.attach(portfolio)

    // Set up portfolio holdings
    portfolio.addHolding("AAPL", 10)
    portfolio.addHolding("GOOGL", 5)
    portfolio.addHolding("MSFT", 8)

    // Price changes trigger automatic notifications
    exchange.setPrice("AAPL", 150.00)
    exchange.setPrice("GOOGL", 2800.00)
    exchange.setPrice("MSFT", 300.00)
    exchange.setPrice("AAPL", 162.00)  // 8% change — triggers alert

    // Output:
    // DISPLAY: AAPL is now $150.00
    // DISPLAY: GOOGL is now $2800.00
    // DISPLAY: MSFT is now $300.00
    // PORTFOLIO: Updated value is $1500.00
    // DISPLAY: AAPL is now $162.00
    // ALERT: AAPL moved 8.0%
    // PORTFOLIO: Updated value is $1590.00
```

## How It Works — Step by Step

1. **Registration** — Each observer (PriceDisplay, AlertSystem, PortfolioTracker) calls `exchange.attach(observer)` to register itself with the StockExchange. The StockExchange stores each observer in its internal list.

2. **State Change** — Client code calls `exchange.setPrice("AAPL", 150.00)`. The StockExchange updates its internal `prices` map with the new price.

3. **Notification Trigger** — Immediately after storing the new price, `setPrice` calls `this.notifyObservers("AAPL", 150.00)`.

4. **Broadcast** — The `notifyObservers` method iterates through its list of registered observers and calls `observer.update("AAPL", 150.00)` on each one. The subject does not know what any observer does with the data — it only broadcasts.

5. **Observer Reaction (Display)** — PriceDisplay's `update` method simply prints the ticker and price to the console. This observer has no state — it is purely reactive.

6. **Observer Reaction (Alert)** — AlertSystem's `update` method compares the new price against the stored previous price. If the percentage change exceeds the threshold (5%), it prints an alert. It then stores the new price as the baseline for future comparisons.

7. **Observer Reaction (Portfolio)** — PortfolioTracker's `update` method checks if the changed ticker is in its holdings. If so, it recalculates the total portfolio value by querying each holding's current price and multiplying by the number of shares held.

8. **Dynamic Changes** — You can add or remove observers at any time without modifying the StockExchange class. For example, you could call `exchange.detach(alertSystem)` to silence alerts without affecting the display or portfolio tracking.

## When to Use

- When a change to one object requires changing others, and you do not know how many objects need to change.
- When an object should be able to notify other objects without making assumptions about who those objects are.
- When you want to avoid tight coupling between the subject and its dependents.
- When changes to the subject should trigger automatic updates in multiple, potentially unrelated, subsystems.
- When you need to support broadcast communication where the subject does not need a response from its observers.

## When NOT to Use

- When observers are tightly coupled to the subject and will never need to be added or removed independently — a simpler callback or direct method call suffices.
- When the notification chain becomes circular (observer updates trigger subject changes that trigger observer updates again) and you cannot guarantee termination.
- When the number of observers is very large and notification performance is critical — consider batching or event queues instead.
- When the subject's state changes very frequently and observers are expensive to update — consider throttling or a pull-based model.

## Real-World Occurrences

1. **Java Swing / AWT Event Listeners** — Every UI component (JButton, JTextField) is a subject that fires events to registered listener objects. The `EventListener` interface is the Observer, and listeners register via `addActionListener()`. This is a direct application of the Observer pattern.

2. **C# Events and Delegates** — The C# event system is a language-level implementation of the Observer pattern. The `event` keyword creates a multicast delegate that acts as the subject's observer list. Any method with the matching signature can subscribe to the event using `+=` and unsubscribe with `-=`.

3. **Node.js EventEmitter** — The core `EventEmitter` class in Node.js is a first-class Observer implementation. Objects emit named events, and listeners (observers) register via `.on()` and `.off()`. This is the foundation of Node.js's asynchronous, event-driven architecture.

4. **RxJS Observables** — The ReactiveX library for JavaScript (and other languages) extends the Observer pattern into a full reactive programming model. Observables are subjects that push values to Observers over time, with operators for transformation, filtering, and composition of event streams.

5. **Python asyncio Event Loop** — Python's `asyncio` event loop uses the Observer pattern for callback registration. Tasks and callbacks are registered as observers of specific events (socket readiness, timer expiration), and the event loop notifies them asynchronously when the event occurs.

6. **Rust event crates (event-listener, notify)** — While Rust does not have built-in observer support, crates like `event-listener` and `notify` provide Observer-pattern mechanics. The `notify` crate watches filesystem changes and notifies registered observers when files are modified.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Mediator** | Mediator centralizes communication between colleagues; Observer broadcasts from one subject to many observers. | When many objects communicate in complex, many-to-many ways and you need a central coordinator. |
| **Publisher-Subscriber** | Pub-sub adds a message broker/channel between publishers and subscribers; Observer has direct subject-to-observer notification. | When you need loose coupling across processes, networks, or asynchronous boundaries. |
| **Chain of Responsibility** | Chain passes a request along a chain of handlers until one handles it; Observer notifies all registered observers. | When exactly one handler should process a request, based on runtime conditions. |
| **Strategy** | Strategy encapsulates interchangeable algorithms; Observer defines a notification contract. | When you need to swap the behavior of a single object, not broadcast to many. |
| **Event Queue / Command** | Event queue stores events for later processing; Observer processes notifications immediately. | When you need decoupling in time (asynchronous processing) or need to replay events. |
