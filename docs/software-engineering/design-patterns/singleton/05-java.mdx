## Implementation

Below is a complete Java implementation of the Singleton pattern using a banking domain. The `BankConfiguration` class uses the Bill Pugh Initialization-on-Demand Holder idiom — the most thread-safe and efficient singleton pattern in Java. It leverages the JVM's class loading mechanism to guarantee a single instance without requiring explicit synchronization. Three client classes — `LoanDepartment`, `TransactionService`, and `ComplianceSystem` — all access the same `BankConfiguration` instance, ensuring consistent interest rates, fee structures, and regulatory settings across the banking application.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────┐
//   │   BankConfiguration (Singleton)      │  ← OOP: Encapsulation
//   ├──────────────────────────────────────┤     SRP: holds config data
//   │   <<inner static class>>             │
//   │   InstanceHolder                     │  ← JVM-level thread safety
//   │   ├──────────────────────────────────┤
//   │   │ + INSTANCE: BankConfiguration    │  ← created on first access
//   │   └──────────────────────────────────┤
//   ├──────────────────────────────────────┤
//   │   - baseInterestRate: double         │
//   │   - wireTransferFee: double          │
//   │   - ... (other config fields)        │
//   ├──────────────────────────────────────┤
//   │   + static getInstance(): BankConf   │  ← global access point
//   │   + getBaseInterestRate(): double    │
//   │   + getWireTransferFee(): double     │
//   └──────────────────────────────────────┘
```

#### OOP Concepts Applied

1. **Encapsulation** — The `BankConfiguration` class hides all fields as `private` and exposes only public getter methods. The private constructor prevents external instantiation. The inner `InstanceHolder` class is also private, encapsulating the singleton lifecycle completely.

2. **Abstraction** — Clients interact with `BankConfiguration` through a clear set of getter methods without knowledge of how configuration is loaded or how the singleton instance is managed.

3. **Polymorphism** — Not directly applicable to the Singleton pattern, but the three client classes all use the same `BankConfiguration` interface through its public getter methods.

#### SOLID Principles Applied

4. **SRP** — `BankConfiguration` is responsible only for holding configuration data. The `InstanceHolder` inner class is responsible for managing the singleton lifecycle, separating the two concerns.

5. **OCP** — New configuration properties can be added to `BankConfiguration` without modifying any client code. The system is open for extension.

6. **LSP** — While there is no inheritance hierarchy, the typed getter methods provide consistent behavior across all clients.

7. **ISP** — Each getter method is individually focused. A client that needs only the wire transfer fee does not depend on methods related to KYC endpoints.

8. **DIP** — High-level client classes depend on `BankConfiguration` through its public getter interface, not on the details of configuration loading or instance management.

### BankConfiguration (Singleton)

```java
// Singleton Pattern — Java — Banking example
// ==============================================
// Participants: BankConfiguration (Singleton),
// LoanDepartment, TransactionService, ComplianceSystem (Clients)
// OOP: encapsulation, abstraction
// SOLID: SRP, OCP, ISP, DIP
// ==============================================

// ── Singleton ──
class BankConfiguration {
    // Private fields — configuration data
    private final double baseInterestRate;
    private final double savingsInterestRate;
    private final double wireTransferFee;
    private final double maxTransactionLimit;
    private final double minBalanceRequirement;
    private final String kycApiEndpoint;
    private final String bankName;
    private final String swiftCode;
    private final double reserveRequirement;

    // Private constructor — prevents external instantiation
    private BankConfiguration() {
        // Load all configuration values
        this.baseInterestRate = 4.5;
        this.savingsInterestRate = 3.75;
        this.wireTransferFee = 25.00;
        this.maxTransactionLimit = 50000.00;
        this.minBalanceRequirement = 500.00;
        this.kycApiEndpoint = "https://api.globaltrustbank.com/kyc/v2";
        this.bankName = "Global Trust Bank";
        this.swiftCode = "GTBKUS44";
        this.reserveRequirement = 0.10;
    }

    // Bill Pugh Initialization-on-Demand Holder idiom
    // The inner class is loaded on first access to getInstance()
    // JVM class loading is inherently thread-safe
    private static class InstanceHolder {
        private static final BankConfiguration INSTANCE = new BankConfiguration();
    }

    // Public static method — global access point
    public static BankConfiguration getInstance() {
        return InstanceHolder.INSTANCE;
    }

    // Public getter methods
    public double getBaseInterestRate() {
        return baseInterestRate;
    }

    public double getSavingsInterestRate() {
        return savingsInterestRate;
    }

    public double getWireTransferFee() {
        return wireTransferFee;
    }

    public double getMaxTransactionLimit() {
        return maxTransactionLimit;
    }

    public double getMinBalanceRequirement() {
        return minBalanceRequirement;
    }

    public String getKycApiEndpoint() {
        return kycApiEndpoint;
    }

    public String getBankName() {
        return bankName;
    }

    public String getSwiftCode() {
        return swiftCode;
    }

    public double getReserveRequirement() {
        return reserveRequirement;
    }
}
```

### LoanDepartment (Client 1)

```java
// ── Client 1: LoanDepartment ──
class LoanDepartment {
    public void processLoanApplication(double amount, int creditScore) {
        BankConfiguration config = BankConfiguration.getInstance();
        double rate = config.getBaseInterestRate();
        if (creditScore > 700) {
            rate = rate - 1.5;
        }
        double annualInterest = amount * (rate / 100.0);
        System.out.printf("LoanDepartment: Processing $%,.2f loan application%n", amount);
        System.out.printf("  Interest rate: %.1f%%%n", rate);
        System.out.printf("  Annual interest: $%,.2f%n", annualInterest);
        System.out.printf("  Bank: %s%n", config.getBankName());
        System.out.printf("  SWIFT: %s%n", config.getSwiftCode());
    }
}
```

### TransactionService (Client 2)

```java
// ── Client 2: TransactionService ──
class TransactionService {
    public void processWireTransfer(double amount) {
        BankConfiguration config = BankConfiguration.getInstance();
        if (amount > config.getMaxTransactionLimit()) {
            System.out.printf("TransactionService: REJECTED — $%,.2f exceeds limit of $%,.2f%n",
                amount, config.getMaxTransactionLimit());
            return;
        }
        double fee = config.getWireTransferFee();
        double total = amount + fee;
        System.out.printf("TransactionService: Processing wire transfer of $%,.2f%n", amount);
        System.out.printf("  Wire fee: $%,.2f%n", fee);
        System.out.printf("  Total debit: $%,.2f%n", total);
        System.out.printf("  Bank: %s%n", config.getBankName());
    }

    public void checkSavingsRate() {
        BankConfiguration config = BankConfiguration.getInstance();
        double rate = config.getSavingsInterestRate();
        System.out.printf("TransactionService: Current savings interest rate is %.2f%%%n", rate);
    }
}
```

### ComplianceSystem (Client 3)

```java
// ── Client 3: ComplianceSystem ──
class ComplianceSystem {
    public void verifyCustomer(String customerId) {
        BankConfiguration config = BankConfiguration.getInstance();
        String endpoint = config.getKycApiEndpoint();
        double minBalance = config.getMinBalanceRequirement();
        System.out.printf("ComplianceSystem: Verifying customer %s%n", customerId);
        System.out.printf("  KYC endpoint: %s%n", endpoint);
        System.out.printf("  Minimum balance requirement: $%,.2f%n", minBalance);
        System.out.printf("  Reserve requirement: %.0f%%%n", config.getReserveRequirement() * 100);
    }
}
```

### Main / Driver

```java
// ── Main / Driver ──
public class Main {
    public static void main(String[] args) {
        System.out.println("=== Banking System Initialization ===\n");

        LoanDepartment loanDept = new LoanDepartment();
        TransactionService transactionService = new TransactionService();
        ComplianceSystem compliance = new ComplianceSystem();

        loanDept.processLoanApplication(250000.00, 720);
        System.out.println();
        transactionService.processWireTransfer(15000.00);
        System.out.println();
        transactionService.checkSavingsRate();
        System.out.println();
        compliance.verifyCustomer("CUST-4201");
        System.out.println();

        // Singleton identity verification
        System.out.println("=== Singleton Identity Verification ===");
        BankConfiguration ref1 = BankConfiguration.getInstance();
        BankConfiguration ref2 = BankConfiguration.getInstance();
        BankConfiguration ref3 = BankConfiguration.getInstance();
        System.out.println("ref1 == ref2: " + (ref1 == ref2));
        System.out.println("ref2 == ref3: " + (ref2 == ref3));
        boolean allSame = (ref1 == ref2) && (ref2 == ref3);
        System.out.println("All references point to the same instance: " + allSame);
    }
}
```

### Language-Specific Notes

- **Bill Pugh Initialization-on-Demand Holder** — This is the most widely recommended singleton pattern in Java. The `InstanceHolder` inner class is not loaded until `getInstance()` is called. The JVM guarantees that class loading is thread-safe, so no explicit `synchronized` keyword is needed. This approach combines lazy initialization, thread safety, and zero synchronization overhead after initialization.

- **`final` fields** — All configuration fields are declared `final`, making them immutable after construction. This ensures the singleton is safely published to all threads without additional synchronization. The Java Memory Model guarantees that `final` fields are visible to all threads after construction completes.

- **Reference comparison with `==`** — Java uses `==` for reference identity comparison, which is the correct way to verify that two variables point to the same object. The driver demonstrates this with three references from `getInstance()`.

- **Enum singleton alternative** — Java also supports the enum singleton pattern (`public enum BankConfiguration { INSTANCE }`). This is the simplest approach and provides built-in serialization safety. However, the Bill Pugh holder idiom is preferred when the singleton needs to extend a class other than `Enum`.

- **Serialization safety** — The Bill Pugh holder idiom does not automatically protect against serialization breaking the singleton. To make it serialization-safe, add `readResolve()` to return the singleton instance. The enum singleton avoids this issue entirely.
