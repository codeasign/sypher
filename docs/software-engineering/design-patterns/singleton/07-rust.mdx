## Implementation

Below is a complete Rust implementation of the Singleton pattern using a banking domain. The `BankConfiguration` struct uses `std::sync::OnceLock` — the standard library mechanism for thread-safe one-time initialization in Rust. Rust's ownership model and type system make singletons particularly interesting: instead of a global mutable instance, the configuration is initialized once and then accessed immutably by all clients. Three client structs — `LoanDepartment`, `TransactionService`, and `ComplianceSystem` — all access the same `BankConfiguration` instance, ensuring consistent interest rates, fee structures, and regulatory settings across the banking application.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────┐
//   │   BankConfiguration (Singleton)      │  ← OOP: Encapsulation
//   ├──────────────────────────────────────┤     SRP: holds config data
//   │   <<static OnceLock>>                │
//   │   CONFIG: OnceLock<BankConfiguration>│  ← Thread-safe init
//   ├──────────────────────────────────────┤
//   │   base_interest_rate: f64            │
//   │   wire_transfer_fee: f64             │
//   │   ... (other config fields)          │
//   ├──────────────────────────────────────┤
//   │   + fn get() -> &'static BankConfig  │  ← global access point
//   │   + fn base_interest_rate() -> f64   │
//   │   + fn wire_transfer_fee() -> f64    │
//   └──────────────────────────────────────┘
```

#### OOP Concepts Applied

1. **Encapsulation** — The `BankConfiguration` struct fields are private. Access is provided through public methods. The `OnceLock` is a private static item, and the initialization logic is encapsulated within the `get()` method.

2. **Abstraction** — Clients interact with `BankConfiguration` through its public methods without knowledge of how the configuration is initialized or stored.

3. **Polymorphism** — Not directly applicable to the Singleton pattern. In Rust, polymorphism is achieved through traits, which are not needed here since all clients access the same concrete type.

#### SOLID Principles Applied

4. **SRP** — `BankConfiguration` is responsible only for holding configuration data. The `OnceLock` mechanism handles thread-safe initialization as a separate concern.

5. **OCP** — New configuration methods can be added to `BankConfiguration` without modifying any client code. The system is open for extension.

6. **LSP** — While Rust does not use classical inheritance, the principle of consistent behavior applies: all clients rely on the same methods with the same guarantees.

7. **ISP** — Each method is individually focused. A client that needs only the wire transfer fee does not depend on methods related to KYC endpoints.

8. **DIP** — High-level client code depends on the `BankConfiguration` struct through its public methods, not on the details of initialization.

### BankConfiguration (Singleton)

```rust
// Singleton Pattern — Rust — Banking example
// ==============================================
// Participants: BankConfiguration (Singleton),
// LoanDepartment, TransactionService, ComplianceSystem (Clients)
// OOP: encapsulation, abstraction
// SOLID: SRP, OCP, ISP, DIP
// ==============================================

use std::sync::OnceLock;

// ── Singleton ──
struct BankConfiguration {
    base_interest_rate: f64,
    savings_interest_rate: f64,
    wire_transfer_fee: f64,
    max_transaction_limit: f64,
    min_balance_requirement: f64,
    kyc_api_endpoint: String,
    bank_name: String,
    swift_code: String,
    reserve_requirement: f64,
}

// Static OnceLock — the singleton storage
// OnceLock::get_or_init() ensures the closure runs exactly once
static CONFIG: OnceLock<BankConfiguration> = OnceLock::new();

impl BankConfiguration {
    // Private constructor — called only from get()
    fn new() -> Self {
        BankConfiguration {
            base_interest_rate: 4.5,
            savings_interest_rate: 3.75,
            wire_transfer_fee: 25.00,
            max_transaction_limit: 50000.00,
            min_balance_requirement: 500.00,
            kyc_api_endpoint: String::from("https://api.globaltrustbank.com/kyc/v2"),
            bank_name: String::from("Global Trust Bank"),
            swift_code: String::from("GTBKUS44"),
            reserve_requirement: 0.10,
        }
    }

    // Public static method — global access point
    // Returns a reference with static lifetime
    pub fn get() -> &'static BankConfiguration {
        CONFIG.get_or_init(|| BankConfiguration::new())
    }

    // Public getter methods
    pub fn base_interest_rate(&self) -> f64 {
        self.base_interest_rate
    }

    pub fn savings_interest_rate(&self) -> f64 {
        self.savings_interest_rate
    }

    pub fn wire_transfer_fee(&self) -> f64 {
        self.wire_transfer_fee
    }

    pub fn max_transaction_limit(&self) -> f64 {
        self.max_transaction_limit
    }

    pub fn min_balance_requirement(&self) -> f64 {
        self.min_balance_requirement
    }

    pub fn kyc_api_endpoint(&self) -> &str {
        &self.kyc_api_endpoint
    }

    pub fn bank_name(&self) -> &str {
        &self.bank_name
    }

    pub fn swift_code(&self) -> &str {
        &self.swift_code
    }

    pub fn reserve_requirement(&self) -> f64 {
        self.reserve_requirement
    }
}
```

### LoanDepartment (Client 1)

```rust
// ── Client 1: LoanDepartment ──
struct LoanDepartment;

impl LoanDepartment {
    fn process_loan_application(amount: f64, credit_score: u32) {
        let config = BankConfiguration::get();
        let mut rate = config.base_interest_rate();
        if credit_score > 700 {
            rate -= 1.5;
        }
        let annual_interest = amount * (rate / 100.0);
        println!("LoanDepartment: Processing ${:.2} loan application", amount);
        println!("  Interest rate: {:.1}%", rate);
        println!("  Annual interest: ${:.2}", annual_interest);
        println!("  Bank: {}", config.bank_name());
        println!("  SWIFT: {}", config.swift_code());
    }
}
```

### TransactionService (Client 2)

```rust
// ── Client 2: TransactionService ──
struct TransactionService;

impl TransactionService {
    fn process_wire_transfer(amount: f64) {
        let config = BankConfiguration::get();
        if amount > config.max_transaction_limit() {
            println!(
                "TransactionService: REJECTED — ${:.2} exceeds limit of ${:.2}",
                amount,
                config.max_transaction_limit()
            );
            return;
        }
        let fee = config.wire_transfer_fee();
        let total = amount + fee;
        println!("TransactionService: Processing wire transfer of ${:.2}", amount);
        println!("  Wire fee: ${:.2}", fee);
        println!("  Total debit: ${:.2}", total);
        println!("  Bank: {}", config.bank_name());
    }

    fn check_savings_rate() {
        let config = BankConfiguration::get();
        let rate = config.savings_interest_rate();
        println!("TransactionService: Current savings interest rate is {:.2}%", rate);
    }
}
```

### ComplianceSystem (Client 3)

```rust
// ── Client 3: ComplianceSystem ──
struct ComplianceSystem;

impl ComplianceSystem {
    fn verify_customer(customer_id: &str) {
        let config = BankConfiguration::get();
        let endpoint = config.kyc_api_endpoint();
        let min_balance = config.min_balance_requirement();
        println!("ComplianceSystem: Verifying customer {}", customer_id);
        println!("  KYC endpoint: {}", endpoint);
        println!("  Minimum balance requirement: ${:.2}", min_balance);
        println!("  Reserve requirement: {:.0}%", config.reserve_requirement() * 100.0);
    }
}
```

### Main / Driver

```rust
// ── Main / Driver ──
fn main() {
    println!("=== Banking System Initialization ===\n");

    LoanDepartment::process_loan_application(250_000.00, 720);
    println!();
    TransactionService::process_wire_transfer(15_000.00);
    println!();
    TransactionService::check_savings_rate();
    println!();
    ComplianceSystem::verify_customer("CUST-4201");
    println!();

    // Singleton identity verification
    // In Rust, references are compared by address using std::ptr::eq
    println!("=== Singleton Identity Verification ===");
    let ref1: &'static BankConfiguration = BankConfiguration::get();
    let ref2: &'static BankConfiguration = BankConfiguration::get();
    let ref3: &'static BankConfiguration = BankConfiguration::get();
    let eq12 = std::ptr::eq(ref1, ref2);
    let eq23 = std::ptr::eq(ref2, ref3);
    println!("ref1 == ref2: {}", eq12);
    println!("ref2 == ref3: {}", eq23);
    let all_same = eq12 && eq23;
    println!("All references point to the same instance: {}", all_same);
}
```

### Language-Specific Notes

- **`OnceLock` for thread-safe initialization** — `std::sync::OnceLock` was stabilized in Rust 1.70. The `get_or_init()` method takes a closure that runs exactly once, even if multiple threads call `get()` concurrently. This provides both lazy initialization and thread safety without external locks. Before `OnceLock` was stabilized, the community used `lazy_static!` or `once_cell` crates.

- **`&'static` lifetime** — The `get()` method returns `&'static BankConfiguration`, meaning the reference is valid for the entire program lifetime. This is safe because the `OnceLock` guarantees the value is initialized once and never dropped, and the `BankConfiguration` struct contains only immutable data after initialization.

- **No `unsafe` code** — Rust's standard library handles the interior mutability of `OnceLock` safely using `UnsafeCell` internally. The public API is safe, and the borrow checker ensures that no mutable references to the configuration exist after initialization.

- **Unit structs as clients** — The client types (`LoanDepartment`, `TransactionService`, `ComplianceSystem`) are defined as unit structs with no fields. Their methods are associated functions that take no `&self` parameter, since they only need to access the global singleton via `BankConfiguration::get()`.

- **`std::ptr::eq` for identity** — Rust references implement `PartialEq` for value comparison, not reference identity. To check whether two references point to the same memory location, use `std::ptr::eq()`. This is the Rust equivalent of Java's `==` or C#'s `ReferenceEquals` for reference comparison.

- **`f64` precision** — Rust's `f64` type is used for all monetary values. The `:.2` format specifier in `println!` ensures two decimal places for dollar amounts. In production banking code, a decimal type or fixed-point library would be preferred over `f64` to avoid floating-point rounding errors.
