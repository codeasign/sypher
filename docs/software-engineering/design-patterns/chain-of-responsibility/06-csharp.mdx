# Chain of Responsibility Pattern — C#

*Reading time: 10 minutes*

The Chain of Responsibility pattern lets a request travel through a chain of handlers until one of them processes it. In the stock market domain, trade orders must be approved by different trading roles based on the order's total value. Each role has a maximum approval limit, and orders exceeding that limit are passed to the next role in the chain.

## How the Implementation Works

### Class Diagram (Pseudocode)

```
// ── Participants ──────────────────────────────────────────
// IOrderApprover (interface)
//     - SetNext(IOrderApprover): IOrderApprover
//     - ApproveOrder(TradeOrder): string
//
// OrderApproverBase (abstract class, implements IOrderApprover)
//     - #next: IOrderApprover
//     - +SetNext(): IOrderApprover
//     - +ApproveOrder(): string  (delegates to next)
//
// JuniorTrader  (extends OrderApproverBase)  limit: $1,000
// SeniorTrader  (extends OrderApproverBase)  limit: $10,000
// PortfolioManager (extends OrderApproverBase) limit: $100,000
// RiskDirector  (extends OrderApproverBase)  limit: unlimited
//
// TradeOrder (record)
//     - Ticker: string
//     - Quantity: int
//     - PricePerShare: decimal
//     - OrderType: string
//     - TotalValue: decimal  (computed: Quantity * PricePerShare)
//
// ── OOP Roles ─────────────────────────────────────────────
// Handler         → IOrderApprover (interface)
// ConcreteHandler → JuniorTrader, SeniorTrader, PortfolioManager, RiskDirector
// Client          → Program.Main (builds chain and submits orders)
//
// ── SOLID Principles ──────────────────────────────────────
// SRP → each handler has one approval limit threshold
// OCP → new roles added without modifying existing handlers
// LSP → any handler can replace another in the chain
// ISP → IOrderApprover exposes only SetNext and ApproveOrder
// DIP → chain logic depends on the IOrderApprover abstraction
```

### OOP Concepts Applied

1. **Inheritance** — `OrderApproverBase` provides shared `SetNext` chaining and a default `ApproveOrder` that delegates to the next handler. Concrete handlers extend it without duplicating chain infrastructure.

2. **Polymorphism** — Each handler overrides `ApproveOrder` with its own approval threshold while sharing the same interface. The chain logic calls `ApproveOrder` polymorphically through the `IOrderApprover` reference.

3. **Encapsulation** — Each handler hides its approval limit and next-handler reference behind the `IOrderApprover` interface. Internal state (the `next` field) is not exposed to callers.

4. **Composition** — The chain is built by composing handler objects, each holding a reference to the next. This creates a flexible, runtime-configurable pipeline.

### SOLID Principles Applied

5. **Single Responsibility Principle** — Each handler class has exactly one responsibility: approving orders within its defined limit. No handler does more than one thing.

6. **Open/Closed Principle** — New approval roles (e.g., `ComplianceOfficer`) can be added by implementing `IOrderApprover` without modifying any existing handler classes.

7. **Liskov Substitution Principle** — Any handler can replace another in the chain. The chain logic works uniformly through the `IOrderApprover` interface regardless of which concrete handler is used.

8. **Interface Segregation Principle** — `IOrderApprover` exposes only `SetNext` and `ApproveOrder`. Clients are not forced to depend on methods they do not use.

9. **Dependency Inversion Principle** — The chain-building code in `Program.Main` depends on the `IOrderApprover` abstraction, not on concrete handler classes. High-level policy (the chain) is decoupled from low-level details (individual limits).

## Complete C# Implementation

### TradeOrder (Request Record)
```csharp
using System;

public record TradeOrder(string Ticker, int Quantity, decimal PricePerShare, string OrderType)
{
    public decimal TotalValue => Quantity * PricePerShare;
}
```

### IOrderApprover (Handler Interface)
```csharp
public interface IOrderApprover
{
    IOrderApprover SetNext(IOrderApprover approver);
    string ApproveOrder(TradeOrder order);
}
```

### OrderApproverBase (Abstract Base Handler)
```csharp
public abstract class OrderApproverBase : IOrderApprover
{
    private IOrderApprover? _next;

    public IOrderApprover SetNext(IOrderApprover approver)
    {
        _next = approver;
        return approver;
    }

    public virtual string ApproveOrder(TradeOrder order)
    {
        if (_next is not null)
            return _next.ApproveOrder(order);
        return $"Order for {order.Ticker} could not be approved: no approver available";
    }

    protected IOrderApprover? Next => _next;
}
```

### JuniorTrader (Concrete Handler)
```csharp
public class JuniorTrader : OrderApproverBase
{
    private const decimal Limit = 1_000m;

    public override string ApproveOrder(TradeOrder order)
    {
        if (order.TotalValue <= Limit)
            return $"JuniorTrader approved {order.OrderType} order for ${order.TotalValue:F2} of {order.Ticker}";
        return base.ApproveOrder(order);
    }
}
```

### SeniorTrader (Concrete Handler)
```csharp
public class SeniorTrader : OrderApproverBase
{
    private const decimal Limit = 10_000m;

    public override string ApproveOrder(TradeOrder order)
    {
        if (order.TotalValue <= Limit)
            return $"SeniorTrader approved {order.OrderType} order for ${order.TotalValue:F2} of {order.Ticker}";
        return base.ApproveOrder(order);
    }
}
```

### PortfolioManager (Concrete Handler)
```csharp
public class PortfolioManager : OrderApproverBase
{
    private const decimal Limit = 100_000m;

    public override string ApproveOrder(TradeOrder order)
    {
        if (order.TotalValue <= Limit)
            return $"PortfolioManager approved {order.OrderType} order for ${order.TotalValue:F2} of {order.Ticker}";
        return base.ApproveOrder(order);
    }
}
```

### RiskDirector (Concrete Handler)
```csharp
public class RiskDirector : OrderApproverBase
{
    public override string ApproveOrder(TradeOrder order)
    {
        return $"RiskDirector approved {order.OrderType} order for ${order.TotalValue:F2} of {order.Ticker}";
    }
}
```

### Program / Driver
```csharp
public class Program
{
    public static void Main()
    {
        // Build chain: JuniorTrader -> SeniorTrader -> PortfolioManager -> RiskDirector
        IOrderApprover junior = new JuniorTrader();
        IOrderApprover senior = new SeniorTrader();
        IOrderApprover pm     = new PortfolioManager();
        IOrderApprover director = new RiskDirector();

        junior.SetNext(senior)
              .SetNext(pm)
              .SetNext(director);

        // Test orders
        var orders = new[]
        {
            new TradeOrder("AAPL",  5,    150m,   "BUY"),   // $750.00    -> JuniorTrader
            new TradeOrder("GOOGL", 50,   175m,   "BUY"),   // $8,750.00  -> SeniorTrader
            new TradeOrder("MSFT",  1000, 300m,   "SELL"),  // $300,000.00 -> RiskDirector
        };

        foreach (var order in orders)
        {
            string result = junior.ApproveOrder(order);
            Console.WriteLine(result);
        }
    }
}
```

### Expected Output

```bash
JuniorTrader approved BUY order for $750.00 of AAPL
SeniorTrader approved BUY order for $8750.00 of GOOGL
RiskDirector approved SELL order for $300000.00 of MSFT
```

## Language-Specific Notes

- C# 12 `record` types provide value equality and a concise positional declaration syntax. `TradeOrder` uses a primary constructor, and `TotalValue` is a computed property with an expression-bodied member.
- The `IOrderApprover?` nullable annotation on the `_next` field signals that the tail of the chain has no successor. This is enforced by the compiler's nullable reference type analysis.
- `decimal` is the correct type for financial calculations in C#. Unlike `float` or `double`, it avoids floating-point rounding errors in monetary arithmetic.
- `SetNext` returns `IOrderApprover` to enable fluent chaining: `junior.SetNext(senior).SetNext(pm).SetNext(director)`. This makes chain construction concise and readable.
- Numeric literal separators (`1_000m`) improve readability for large constant values and are valid C# syntax since C# 7.0.
- The `RiskDirector` does not call `base.ApproveOrder` because it is the terminal handler. It always approves and never delegates, which matches the business requirement that the director has no upper limit.
