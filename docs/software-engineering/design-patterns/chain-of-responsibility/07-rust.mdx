## Implementation

Below is a complete Rust implementation of the Chain of Responsibility pattern using a stock market domain. A trade order flows through an approval chain where each approver checks whether the order's total value falls within their authority limit. If it does, they approve it; otherwise, they pass it to the next approver in the chain. The implementation demonstrates Rust's trait-based polymorphism via trait objects (`Box<dyn OrderApprover>`), and applies SOLID principles including SRP (each approver only checks one limit), OCP (new approver roles can be added without changing existing ones), and DIP (client code depends on the `OrderApprover` trait).

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌─────────────────────────────────────┐
//   │   <<trait>>                         │
//   │   OrderApprover (Handler)           │ ← OOP: Abstraction
//   ├─────────────────────────────────────┤    ISP: single method for
//   │ + set_next(Box<dyn OrderApprover>)  │     handling orders +
//   │ + approve_order(&TradeOrder) -> str │     chaining
//   └──────────────────┬──────────────────┘
//                      ▲ implements
//   ┌──────────────────┴─────────────────────────────────────┐
//   │                    │                    │              │
//   ┌────────────────┐ ┌────────────────┐ ┌────────────────┐ ┌──────────────┐
//   │ JuniorTrader   │ │ SeniorTrader   │ │PortfolioManager│ │RiskDirector  │
//   │ limit: 1,000   │ │ limit: 10,000  │ │ limit: 100,000 │ │ unlimited    │
//   └────────────────┘ └────────────────┘ └────────────────┘ └──────────────┘
//        LSP                 LSP                  LSP             LSP
//   Each concrete handler is substitutable for Box<dyn OrderApprover>
//
//   ┌─────────────────────────────────────┐
//   │ TradeOrder (Request)                │
//   ├─────────────────────────────────────┤
//   │ ticker: String, qty: u32           │
//   │ price: f64, order_type: OrderType  │
//   │ total_value(): f64                 │
//   └─────────────────────────────────────┘
//
//   Chain construction:
//     let junior = JuniorTrader::new();
//     let senior = SeniorTrader::new();
//     junior.set_next(Box::new(senior));
//     // Orders flow: junior -> senior -> pm -> director
```

#### OOP Concepts Applied

1. **Abstraction** — The `OrderApprover` trait defines the contract that all approvers must follow. Client code works against `Box<dyn OrderApprover>`, never concrete types.

2. **Encapsulation** — Each approver struct hides its approval limit and optional next handler as private fields. The `approve_order` method encapsulates the decision logic.

3. **Inheritance** — Simulated via the `BaseApprover` pattern. Each concrete approver struct contains an optional `next_approver: Option<Box<dyn OrderApprover>>` field and delegates to a shared `set_next` / fallback implementation.

4. **Polymorphism** — The trait object `Box<dyn OrderApprover>` allows any concrete handler to be placed in the chain. The driver treats all approvers uniformly through the trait interface.

#### SOLID Principles Applied

5. **SRP** — `JuniorTrader` only handles orders ≤ $1,000. `SeniorTrader` only handles orders ≤ $10,000. Each class has exactly one reason to change (its approval limit).

6. **OCP** — New approver roles (e.g., `ComplianceOfficer` with a $50,000 limit) can be added by implementing the `OrderApprover` trait. No existing code is modified — the system is open for extension, closed for modification.

7. **LSP** — Every concrete handler implements `OrderApprover` and can substitute for `Box<dyn OrderApprover>`. The client does not need to know which concrete type is in the chain.

8. **ISP** — The `OrderApprover` trait has only two methods (`set_next` and `approve_order`). No implementor is forced to provide behavior it doesn't need.

9. **DIP** — The chain builder depends on the `OrderApprover` trait, not on concrete handlers. The driver references `Box<dyn OrderApprover>` throughout, never a concrete type by name.

### OrderType (Enum)
```rust
use std::fmt;

#[derive(Debug, Clone)]
enum OrderType {
    Buy,
    Sell,
}

impl fmt::Display for OrderType {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            OrderType::Buy => write!(f, "BUY"),
            OrderType::Sell => write!(f, "SELL"),
        }
    }
}
```

### TradeOrder (Request)
```rust
#[derive(Debug, Clone)]
struct TradeOrder {
    ticker: String,
    quantity: u32,
    price_per_share: f64,
    order_type: OrderType,
}

impl TradeOrder {
    fn new(ticker: &str, quantity: u32, price_per_share: f64, order_type: OrderType) -> Self {
        Self {
            ticker: ticker.to_string(),
            quantity,
            price_per_share,
            order_type,
        }
    }

    fn total_value(&self) -> f64 {
        self.quantity as f64 * self.price_per_share
    }
}

impl fmt::Display for TradeOrder {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(
            f,
            "{} {} {} shares at ${:.2} (total: ${:.2})",
            self.order_type,
            self.ticker,
            self.quantity,
            self.price_per_share,
            self.total_value()
        )
    }
}
```

### OrderApprover (Handler Trait)
```rust
trait OrderApprover {
    /// Set the next handler in the chain. Returns the next handler for chaining.
    fn set_next(&mut self, next: Box<dyn OrderApprover>);

    /// Attempt to approve the given trade order.
    /// Returns a string describing which handler approved or rejected the order.
    fn approve_order(&self, order: &TradeOrder) -> String;

    /// Get the approval limit of this handler (in dollars).
    fn approval_limit(&self) -> f64;

    /// Get the role name of this handler.
    fn role_name(&self) -> &str;
}
```

### JuniorTrader (Concrete Handler)
```rust
struct JuniorTrader {
    next_approver: Option<Box<dyn OrderApprover>>,
}

impl JuniorTrader {
    fn new() -> Self {
        Self {
            next_approver: None,
        }
    }
}

impl OrderApprover for JuniorTrader {
    fn set_next(&mut self, next: Box<dyn OrderApprover>) {
        self.next_approver = Some(next);
    }

    fn approval_limit(&self) -> f64 {
        1_000.0
    }

    fn role_name(&self) -> &str {
        "JuniorTrader"
    }

    fn approve_order(&self, order: &TradeOrder) -> String {
        let total = order.total_value();
        if total <= self.approval_limit() {
            format!(
                "[JuniorTrader] Approved: {} (${:.2} within ${:.2} limit)",
                order,
                total,
                self.approval_limit()
            )
        } else if let Some(ref next) = self.next_approver {
            format!(
                "[JuniorTrader] Escalating {} (${:.2} exceeds ${:.2} limit) -> {}",
                order,
                total,
                self.approval_limit(),
                next.approve_order(order)
            )
        } else {
            format!(
                "[JuniorTrader] REJECTED: {} (${:.2}) — no further approver available",
                order, total
            )
        }
    }
}
```

### RiskDirector (Concrete Handler)
```rust
struct RiskDirector {
    next_approver: Option<Box<dyn OrderApprover>>,

impl RiskDirector {
    fn new() -> Self {
        Self {
            next_approver: None,
        }
    }
}

impl OrderApprover for RiskDirector {
    fn set_next(&mut self, _next: Box<dyn OrderApprover>) {
        // RiskDirector is the end of the chain — ignores set_next
    }

    fn approval_limit(&self) -> f64 {
        f64::INFINITY
    }

    fn role_name(&self) -> &str {
        "RiskDirector"
    }

    fn approve_order(&self, order: &TradeOrder) -> String {
        let total = order.total_value();
        format!(
            "[RiskDirector] Approved: {} (${:.2}) — final authority",
            order, total
        )
    }
}
```

### Main / Driver
```rust
fn main() {
    // Build the approval chain
    let mut junior = JuniorTrader::new();
    let mut senior = SeniorTrader::new();
    let mut pm = PortfolioManager::new();
    let director = RiskDirector::new();

    pm.set_next(Box::new(director));
    senior.set_next(Box::new(pm));
    junior.set_next(Box::new(senior));

    // Create test orders with realistic stock market data
    let orders = vec![
        TradeOrder::new("AAPL", 10, 150.0, OrderType::Buy),      // $1,500
        TradeOrder::new("GOOGL", 50, 175.0, OrderType::Buy),     // $8,750
        TradeOrder::new("MSFT", 1000, 200.0, OrderType::Sell),   // $200,000
    ];

    // Process each order through the chain
    for order in &orders {
        println!("{}", junior.approve_order(order));
        println!();
    }
}
```

### Language-Specific Notes

1. **Trait for Abstraction** — Rust uses `trait OrderApprover` to define the handler interface, analogous to an interface in Java or an abstract class in C++. Trait objects are Rust's primary mechanism for runtime polymorphism.

2. **`Box<dyn OrderApprover>` for Dynamic Dispatch** — Since handlers need to be stored in a heterogeneous chain, the next handler is stored as `Option<Box<dyn OrderApprover>>`. The `Box` heap-allocates the concrete handler, and the `dyn` keyword enables dynamic dispatch through the vtable.

3. **No Shared Mutable References Needed** — Each handler owns its next handler via `Box<dyn OrderApprover>`. This avoids the need for `Rc<RefCell<...>>` in the single-ownership case, since the chain is built linearly and ownership moves from link to link.

4. **`f64::INFINITY` as Unlimited Limit** — The `RiskDirector` sets `approval_limit()` to `f64::INFINITY` to represent unbounded authority, which makes the numeric comparison `total <= limit` always true without special-casing.

5. **Ownership and Mutability** — `set_next` takes `&mut self` to mutate the struct's `next_approver` field, while `approve_order` takes `&self` (shared reference) since it only reads state. This follows Rust's borrowing rules naturally.

6. **`impl fmt::Display` for Domain Types** — Both `OrderType` and `TradeOrder` implement the `Display` trait so they can be used with the `{}` format specifier. This is Rust's idiomatic equivalent of `toString()` in other languages.
