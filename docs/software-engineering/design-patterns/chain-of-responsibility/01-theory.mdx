## Intent

Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request. Chain the receiving objects and pass the request along the chain until an object handles it.

## Also Known As

CoR, Chain of Command, Responsibility Chain

## Motivation

Consider a stock trading firm that processes thousands of buy and sell orders every day. Each order must pass through an approval process before it can be sent to the exchange for execution. The approval rules are not simple: a junior trader may approve small orders under $1,000, a senior trader can handle orders up to $10,000, a portfolio manager can approve orders up to $100,000, and any order above that requires sign-off from a risk director.

The challenge is that the trading system does not know in advance which approver will handle a given order. The appropriate approver depends on the order's total value, which is computed from the quantity and price per share — and these values are only known at runtime when the order is created. You want the system to submit the order and let the approval hierarchy determine the correct handler, without the system knowing anything about the approval rules or the hierarchy structure.

Furthermore, approval limits change over time. A senior trader's limit might be raised from $10,000 to $15,000 after a performance review. A new Compliance Officer role might be introduced between the Senior Trader and the Portfolio Manager. Regulatory changes might require every order over $50,000 to be logged by a separate audit handler before approval. You need a design that accommodates these changes without modifying the existing approval logic.

The Chain of Responsibility pattern solves this problem by modeling each approval level as an independent handler object. Handlers are linked into a chain, and the order is passed along the chain until a handler takes responsibility.

## Problem

In a stock trading system, trade orders must be approved before they are executed. Different orders require different levels of authorization depending on factors such as total dollar value, order type, and market conditions. A junior trader should be able to approve small, routine buy orders, while a multi-million dollar block trade must be reviewed by senior management or a risk director.

If you hardcode these approval rules into a single monolithic function, you end up with a massive conditional block that is difficult to maintain, extend, or test. Every time a new approval tier is introduced, or an approval limit changes, you must modify that central function — violating the Open/Closed Principle.

Without the Chain of Responsibility pattern, your code might look like this:

```python
def approve_order(order, role):
    if role == "junior" and order.total_value <= 1000:
        return "Approved by Junior Trader"
    elif role == "senior" and order.total_value <= 10000:
        return "Approved by Senior Trader"
    elif role == "portfolio_manager" and order.total_value <= 100000:
        return "Approved by Portfolio Manager"
    elif role == "risk_director":
        return "Approved by Risk Director"
    else:
        return "Rejected: insufficient approval authority"
```

This approach is brittle. Adding a Compliance Officer role means editing the function. Adjusting the Senior Trader limit from 10,000 to 15,000 means changing the same block. Testing every combination of role and value becomes exponentially more complex as new roles are added. The function also violates the Single Responsibility Principle because it encodes the approval logic for every role in one place.

Specific problems with the hardcoded approach:

- **Rigidity.** Adding a new role, such as Compliance Officer, requires editing the central approval function. Every codebase change risks introducing regressions in existing approval paths.
- **Limited reusability.** The approval logic is embedded in a single function, making it impossible to reuse individual approval rules in other contexts, such as reporting or auditing.
- **Testing complexity.** Testing the approval function requires testing every combination of role and order value. As new roles are added, the test matrix grows combinatorially.
- **Poor separation of concerns.** The approval function must know about the approval limits and authority of every role in the organization. This creates an inappropriate coupling between the system and the organizational hierarchy.

## Solution

The Chain of Responsibility pattern turns each approval level into an independent handler object. Each handler holds a reference to the next handler in the chain. When a trade order arrives, the first handler in the chain decides whether it can approve the order. If it can, it processes the request and the chain stops. If it cannot, it forwards the order to the next handler.

This decouples the sender (the trading system or client) from the receiver (the approver). The sender knows only that it is submitting an order to the first handler in the chain. It has no knowledge of which handler will ultimately approve the order, how many handlers exist, or what the approval rules are at each level.

Key benefits of this approach:

- **Each handler is independent.** A handler contains only its own approval logic. It does not depend on other handlers or know about the organizational hierarchy.
- **The chain can be reconfigured at runtime.** You can add, remove, or reorder handlers without changing any handler code. A new Compliance Officer handler can be inserted into the chain by changing only the chain construction code.
- **Handlers are reusable.** The `JuniorTrader` handler can be used in multiple chains. For example, you might construct one chain for domestic equities and another for international equities, each with different handler combinations.
- **The sender remains ignorant.** The trading system that creates and submits orders does not need to know about approval rules, limits, or the organizational structure.

## Structure

The Chain of Responsibility pattern involves the following participants:

- **Handler** — An interface or abstract class that declares a method for setting the next handler in the chain and a method for processing the request. In the stock market example, `OrderApprover` is the Handler interface with `set_next()` and `approve_order()` methods.
- **ConcreteHandler** — Implements the Handler interface. Each ConcreteHandler stores a reference to its successor (the next handler in the chain). It implements its own approval logic in the `_can_approve()` method and decides whether to handle the request or forward it. The four ConcreteHandlers in this example are `JuniorTrader`, `SeniorTrader`, `PortfolioManager`, and `RiskDirector`.
- **Client** — Initiates the request and sends it to the first handler in the chain. In the example, the client is the `main()` function that builds the chain and submits trade orders. The client knows only the head of the chain — it has no visibility into the rest of the chain structure.
- **Request** — The object that flows through the chain, carrying the data needed for processing. In this example, the `TradeOrder` dataclass is the Request, containing ticker, quantity, price per share, order type, and a computed total_value property.

The `OrderApprover` interface defines two key methods:

- `set_next(approver)` — Links this handler to the next handler in the chain. Returns the next handler to enable fluent chaining syntax, such as `junior.set_next(senior).set_next(pm).set_next(director)`.
- `approve_order(order)` — The core method that either handles the request or forwards it. Each implementation checks whether it can handle the request. If yes, it processes and returns. If no, it calls `approve_order` on its successor. If there is no successor, it returns a rejection message.

### Handler Interface in Detail

The `OrderApprover` base class encapsulates the chain-walking logic so that concrete handlers only need to implement two small methods: `_can_approve()` and `_get_role_name()`. The base class handles the decision to forward or reject automatically:

```
approve_order(order):
    if _can_approve(order):
        return "Approved by " + _get_role_name()
    if next_handler exists:
        return next_handler.approve_order(order)
    return "Rejected: no approver could handle this order"
```

This template method pattern ensures that every handler follows the same protocol. You cannot accidentally skip the forwarding logic or forget to check the approval condition — the base class enforces the correct behavior.

### Concrete Handlers

Each concrete handler defines its own approval threshold:

- `JuniorTrader` — Approves orders where `total_value <= $1,000`. This covers small retail-sized orders, such as buying 5 shares of AAPL at $150.
- `SeniorTrader` — Approves orders where `total_value <= $10,000`. This covers medium-sized orders for individual stocks or small portfolios.
- `PortfolioManager` — Approves orders where `total_value <= $100,000`. This covers large institutional-sized orders for significant positions.
- `RiskDirector` — Approves any order. This is the terminal handler, ensuring every request receives a definitive outcome. In a real system, the Risk Director might still apply checks such as regulatory compliance or market impact analysis, but the handler never forwards the request.

Here is the complete step-by-step flow for a trade order of 500 shares of AAPL at $100 per share with a total value of $50,000:

**Step 1: Order creation and submission.** The trading system creates a `TradeOrder` object with ticker "AAPL", quantity 500, price_per_share $100, order_type "buy". The `total_value` property computes to $50,000 (500 times $100). The system holds a reference to the head of the approval chain — the `JuniorTrader` instance — and calls `approve_order(order)` on it.

**Step 2: Junior Trader evaluation.** The `JuniorTrader` receives the order and enters its `approve_order` method. It calls `_can_approve(order)`, which checks whether `total_value <= 1,000`. Since $50,000 is far above $1,000, the method returns `False`. The base class logic then checks whether a next handler exists (`self._next_approver`). It does — the `SeniorTrader`. The Junior Trader calls `self._next_approver.approve_order(order)` and the order moves to the next link in the chain.

**Step 3: Senior Trader evaluation.** The `SeniorTrader` receives the forwarded order. Its `_can_approve` method checks whether `total_value <= 10,000`. Again, $50,000 exceeds $10,000. The Senior Trader cannot approve the order. It finds its successor — the `PortfolioManager` — and forwards the order by calling `approve_order` on the Portfolio Manager.

**Step 4: Portfolio Manager evaluation.** The `PortfolioManager` receives the order. Its `_can_approve` method checks whether `total_value <= 100,000`. This time, $50,000 is within the limit. The method returns `True`. The base class logic constructs the approval message: `"Approved by Portfolio Manager"`. The Portfolio Manager returns this string to the caller (the Senior Trader).

**Step 5: Result propagation.** The approval string travels back through the call stack. The Senior Trader's call to `pm.approve_order(order)` returns the string, and the Senior Trader returns it to the Junior Trader, who returns it to the original caller. The trading system receives the message `"Approved by Portfolio Manager"` and proceeds to execute the order on the exchange.

**Step 6: Chain termination demonstration.** Consider what happens with a much larger order — 2,000 shares of AMZN at $300 per share, total value $600,000. This order passes through Junior Trader (declined, exceeds $1,000), Senior Trader (declined, exceeds $10,000), and Portfolio Manager (declined, exceeds $100,000). It reaches the Risk Director, whose `_can_approve` always returns `True`. The Risk Director approves the order. Without this terminal handler, the chain would return `"Rejected: no approver could handle this order"`, which demonstrates why every proper chain implementation needs a terminal handler that guarantees a definitive outcome.

## Pseudocode

The following pseudocode implements the trade order approval chain using the Chain of Responsibility pattern in Python:

```python
from dataclasses import dataclass
from abc import ABC, abstractmethod
from enum import Enum
from typing import Optional


class OrderType(Enum):
    BUY = "buy"
    SELL = "sell"


@dataclass
class TradeOrder:
    ticker: str
    quantity: int
    price_per_share: float
    order_type: OrderType

    @property
    def total_value(self) -> float:
        return self.quantity * self.price_per_share


class OrderApprover(ABC):
    """Handler interface — defines the chain contract."""

    def __init__(self) -> None:
        self._next_approver: Optional[OrderApprover] = None

    def set_next(self, approver: "OrderApprover") -> "OrderApprover":
        """Set the next handler in the chain. Returns the next
        handler to allow fluent chaining: junior.set_next(senior)."""
        self._next_approver = approver
        return approver

    @abstractmethod
    def _can_approve(self, order: TradeOrder) -> bool:
        """Subclasses implement their specific approval rule."""
        ...

    @abstractmethod
    def _get_role_name(self) -> str:
        """Return the human-readable role name for messaging."""
        ...

    def approve_order(self, order: TradeOrder) -> str:
        """Process the order or pass it to the next handler."""
        if self._can_approve(order):
            return f"Approved by {self._get_role_name()}"
        if self._next_approver is not None:
            return self._next_approver.approve_order(order)
        return "Rejected: no approver could handle this order"


class JuniorTrader(OrderApprover):
    """Approves orders with total value <= $1,000."""

    def _can_approve(self, order: TradeOrder) -> bool:
        return order.total_value <= 1_000.00

    def _get_role_name(self) -> str:
        return "Junior Trader"


class SeniorTrader(OrderApprover):
    """Approves orders with total value <= $10,000."""

    def _can_approve(self, order: TradeOrder) -> bool:
        return order.total_value <= 10_000.00

    def _get_role_name(self) -> str:
        return "Senior Trader"


class PortfolioManager(OrderApprover):
    """Approves orders with total value <= $100,000."""

    def _can_approve(self, order: TradeOrder) -> bool:
        return order.total_value <= 100_000.00

    def _get_role_name(self) -> str:
        return "Portfolio Manager"


class RiskDirector(OrderApprover):
    """Terminal handler — approves any order that reaches it."""

    def _can_approve(self, order: TradeOrder) -> bool:
        return True  # Terminal handler, always approves

    def _get_role_name(self) -> str:
        return "Risk Director"


# --- Client code ---

def build_approval_chain() -> OrderApprover:
    """Construct the chain: JuniorTrader -> SeniorTrader ->
    PortfolioManager -> RiskDirector."""
    junior = JuniorTrader()
    senior = SeniorTrader()
    pm = PortfolioManager()
    director = RiskDirector()

    junior.set_next(senior).set_next(pm).set_next(director)
    return junior  # Return the head of the chain


def main() -> None:
    chain_head = build_approval_chain()

    orders = [
        TradeOrder("AAPL", 5, 150.00, OrderType.BUY),       # $750
        TradeOrder("GOOGL", 20, 250.00, OrderType.BUY),     # $5,000
        TradeOrder("MSFT", 500, 100.00, OrderType.SELL),    # $50,000
        TradeOrder("AMZN", 2000, 300.00, OrderType.BUY),    # $600,000
    ]

    for order in orders:
        result = chain_head.approve_order(order)
        print(f"{order.ticker} ({order.order_type.value}, "
              f"${order.total_value:,.2f}): {result}")


if __name__ == "__main__":
    main()
```

When you run this code, the output is:

```
AAPL (buy, $750.00): Approved by Junior Trader
GOOGL (buy, $5,000.00): Approved by Senior Trader
MSFT (sell, $50,000.00): Approved by Portfolio Manager
AMZN (buy, $600,000.00): Approved by Risk Director
```

The chain handles each order at the correct approval level. The client never needs to know which handler approved the order — it simply submits to the head of the chain and gets a result.

### Key Design Decisions in the Pseudocode

1. **Fluent chaining.** The `set_next` method returns the next handler, not `self`. This allows the compact chain construction `junior.set_next(senior).set_next(pm).set_next(director)`. An alternative design returns `self` from `set_next`, enabling builder-style construction, but the fluent style is more readable for linear chains.

2. **Template method in the base class.** The `approve_order` method in `OrderApprover` follows the Template Method pattern. It defines the skeleton of the algorithm (check, forward, or reject) while delegating the specific check to subclass implementations of `_can_approve`. This eliminates code duplication across handlers.

3. **Terminal handler guarantee.** The `RiskDirector` always returns `True` from `_can_approve`, ensuring the chain never produces a rejection due to reaching the end without a handler. If you omit the terminal handler, the last handler's `_next_approver` is `None`, and the chain returns a rejection message. This is a design choice — some implementations prefer throwing an exception when no handler matches.

4. **Immutable request object.** The `TradeOrder` dataclass uses frozen-like immutability (Python dataclasses are mutable by default but should be treated as immutable). Handlers should never modify the request — they either approve it or forward it unchanged. This prevents subtle bugs where one handler modifies data that another handler relies on.

5. **Single responsibility per handler.** Each handler contains exactly one piece of business logic: its approval limit. This makes handlers easy to test in isolation, easy to reason about, and easy to modify without affecting other handlers.

## When to Use

Use the Chain of Responsibility pattern when:

- **More than one object may need to handle a request, and the handler is not known in advance.** In the stock market domain, you cannot predict whether a trade will need a junior trader or the risk director — the order value determines this at runtime. The chain resolves the correct handler without the client needing to know which one it is.

- **You want to issue a request to one of several objects without specifying the receiver explicitly.** The trading system submits the order to the chain head and lets the chain determine which handler processes it. This decoupling means the trading system does not need to be updated when approval rules change.

- **The set of handlers should be defined dynamically.** You can add, remove, or reorder approvers at runtime. For example, during a market volatility event, you could insert a Compliance Officer handler between the Senior Trader and the Portfolio Manager. During a trading blackout, you could replace the entire chain with a single handler that rejects all orders.

- **You want to avoid conditional logic sprawl.** Instead of a single function with many if-elif branches, each handler encapsulates its own approval logic independently. This makes the code easier to test, extend, and reason about.

- **You need to vary the chain of responsibility at runtime.** A chain for regular market hours might differ from a chain used during after-hours trading or during a company-wide trading blackout period. You can construct different chains for different contexts without changing any handler code.

- **You want to enforce separation of concerns between request processing steps.** Each handler focuses on one specific responsibility. The Junior Trader knows only about small orders. The Risk Director knows only about the terminal approval. No handler needs to understand the full approval hierarchy.

## When NOT to Use

Avoid the Chain of Responsibility pattern when:

- **Every request must be handled by a specific, known handler.** If you always know that orders under $1,000 go to the junior trader, and this never changes, a simple if statement is clearer and more efficient than setting up a chain of objects. The pattern adds indirection that is not justified when the handler is deterministic and fixed.

- **You need guaranteed handling by a particular handler.** The pattern only guarantees that the request travels through the chain — it does not guarantee which handler processes it. If an audit policy requires every trade to be reviewed by the Portfolio Manager regardless of value, the Chain of Responsibility alone cannot enforce this. You would need additional checks, such as a mandatory logging handler that wraps every approval.

- **Performance is critical and the chain is long.** Each link in the chain adds a method call overhead. For high-frequency trading systems processing millions of orders per second, even small overheads multiply into significant latency. In such cases, a direct lookup table or decision tree may be more appropriate.

- **The chain configuration changes infrequently.** If the approval hierarchy is fixed and defined by regulation, a straightforward function with well-defined approval rules is simpler to understand and maintain. The flexibility of the Chain of Responsibility comes with a cost in code complexity and indirection.

- **Request handling depends on complex interactions between multiple handlers.** The Chain of Responsibility processes requests sequentially. If approval requires consensus among multiple approvers simultaneously, consider the Mediator pattern instead. The Chain of Responsibility is inherently linear and cannot model parallel or branching approval workflows.

- **The chain is likely to have only one handler.** If you have only one approval level, the chain abstraction adds unnecessary complexity. Use a simple handler function or class directly.

## Real-World Occurrences

The Chain of Responsibility pattern appears in many software systems and real-world processes:

1. **Exception handling in programming languages.** When you wrap code in try-except blocks, the runtime walks up the call stack looking for a matching catch clause. Each exception handler is like a link in the chain — if it cannot handle the exception type, it passes control to the next handler up the stack. This continues until a handler catches the exception or the program terminates with an unhandled exception. This is the most widely used example of the Chain of Responsibility pattern in everyday programming.

2. **Web framework middleware pipelines.** Frameworks like Express.js (Node.js), Rack (Ruby), Django middleware (Python), and ASP.NET Core middleware (C#) process HTTP requests through a chain of middleware components. Each middleware layer can handle the request, modify it, or pass it to the next layer. A typical request passes through authentication middleware (check credentials), then logging middleware (record the request), then compression middleware (gzip the response), then routing middleware (find the controller), and finally the controller itself. If any middleware determines the request is invalid, it can short-circuit the chain and return an error response directly.

3. **Java Servlet filters.** The Java Servlet specification defines Filter objects that can intercept incoming HTTP requests. Filters are configured in a chain (either in web.xml or via annotations), and each filter can pre-process the request, delegate to the next filter in the chain via `chain.doFilter()`, and post-process the response. Common filters include authentication filters, logging filters, XSS sanitization filters, and compression filters. This is the canonical Chain of Responsibility example in enterprise Java and demonstrates how multiple concerns can be cleanly separated into independent filter objects.

4. **Logging frameworks (SLF4J, Log4j, Python logging).** Loggers in these frameworks are organized in a hierarchy. When a log message is emitted at a given logger, if that logger does not have an appender configured for the message's level, it passes the message to its parent logger. This continues up the hierarchy until a logger handles the message or the root logger is reached. For example, a debug message from `com.example.trading.orders.OrderService` might be handled by the `com.example.trading.orders` logger, or passed up to `com.example`, or finally to the root logger. This is a classic Chain of Responsibility design.

5. **Operating system interrupt handling.** When a hardware interrupt occurs, the OS interrupt dispatcher passes the interrupt to each registered interrupt handler in a chain. If the first handler does not recognize the interrupt (or declines to handle it), the next handler in the chain is tried. This continues until an appropriate device driver handles the interrupt. For example, a network interrupt might first be offered to the kernel's generic network stack, then to a virtual machine monitor, and finally to the specific network card driver.

6. **Stock exchange circuit breakers.** In major stock exchanges, trading halts follow a chain-like escalation pattern. Small price movements in individual stocks are handled by the Limit Up/Limit Down (LULD) mechanism at the single-stock level. If the market declines further, Level 1 market-wide circuit breakers trigger a 15-minute trading halt when the S&P 500 drops 7%. Level 2 triggers at a 13% decline, and Level 3 at a 20% decline, causing trading to halt for the remainder of the day. Each level only activates if the preceding level's conditions have already been triggered — a natural Chain of Responsibility structure.

7. **Help desk ticketing systems.** In IT support, a help desk ticket follows an escalation chain. A Level 1 support agent handles basic issues (password resets, account unlocks). If they cannot resolve the issue, they escalate to Level 2 (application-specific problems). If Level 2 cannot resolve it, they escalate to Level 3 (engineering team or vendor). Each level has authority to resolve certain categories of issues and passes the ticket up when it exceeds their scope.

8. **Government approval workflows.** Many government processes involve sequential approvals. A building permit application might first be reviewed by the zoning department, then passed to the building inspector, then to the fire marshal, and finally to the planning commission for final approval. Each department either approves within its jurisdiction and passes the application to the next department, or rejects it outright.

## Comparison with Related Patterns

| Pattern | How It Differs | Stock Market Example |
|---|---|---|
| **Chain of Responsibility** | Passes a request sequentially along a chain of handlers until one handles it. Each handler decides independently whether it can process the request. | Trade order flows through JuniorTrader, SeniorTrader, PortfolioManager, RiskDirector until one approves. |
| **Command** | Encapsulates a request as an object, allowing parameterization, queuing, logging, and undo. Focuses on the request as a first-class object rather than on the handler chain. | A `TradeOrder` object encapsulating all data needed to execute a trade, queued in an order book for later execution or rolled back if the market moves adversely. |
| **Composite** | Composes objects into tree structures to represent part-whole hierarchies. Clients treat individual objects and compositions uniformly through a common interface. | A trading team hierarchy where a team contains individual traders, and both the team and the individuals can receive and process orders through the same interface. |
| **Decorator** | Adds responsibilities to objects dynamically. Multiple decorators can wrap an object. Each decorator enhances or augments behavior before and after delegating to the wrapped object, and the request always passes through every decorator. | Adding real-time risk checks, compliance logging, fee calculation, and market impact analysis around a trade executor. Each decorator adds a concern, and all decorators execute — none can short-circuit the chain. |
| **Mediator** | Centralizes communication between objects. Instead of passing a request along a chain, the mediator coordinates multiple objects' responses to a single request in a centralized location. | A trading platform mediator that coordinates trade validation, margin checking, market data lookup, best-execution routing, and post-trade settlement — all in one place rather than scattered across a chain. |
| **Observer** | Defines a one-to-many dependency so that when one object changes state, all its dependents are notified. Observers receive events simultaneously, not sequentially. | A market data feed notifying all subscribed trading algorithms of a price change simultaneously. Each algorithm reacts independently rather than passing the event along a chain. |

The key distinction between Chain of Responsibility and Decorator is worth emphasizing because both patterns build linked object chains. Their intents are fundamentally different:

- **Chain of Responsibility** is about finding the right handler for a request. The request stops at the first handler that can process it. Handlers that come later in the chain never see the request.
- **Decorator** is about augmenting behavior. The request always passes through every decorator in the chain. Each decorator adds behavior before delegating, after delegating, or both.

Another important comparison is with the **Mediator** pattern. In Chain of Responsibility, each handler communicates only with its immediate successor. In Mediator, all communication goes through a central mediator object. CoR is decentralized; Mediator is centralized. CoR is ideal when the chain of communication is linear and fixed; Mediator is ideal when interactions are complex and many-to-many.

## Consequences

### Benefits

- **Decoupling of sender and receiver.** The client that submits a request has no knowledge of which handler will process it. This eliminates conditional dependencies and makes the client immune to changes in the handler hierarchy.

- **Simplified object responsibilities.** Each handler has a single responsibility: deciding whether it can handle the current request. This adheres to the Single Responsibility Principle and makes each handler easy to understand, test, and maintain.

- **Dynamic chain configuration.** The chain can be assembled, disassembled, or reordered at runtime. You can build different chains for different contexts without changing any handler implementation. For example, a chain for retail clients might use a lower approval threshold for the Junior Trader, while a chain for institutional clients might start with a higher threshold.

- **Open/Closed Principle compliance.** New handlers can be added without modifying existing handlers or the client. You can introduce a Compliance Officer, an Audit Tracker, or a Chief Risk Officer without touching a single line of existing handler code.

- **Focused testing.** Each handler can be tested independently. Testing the Junior Trader requires only test cases around the $1,000 threshold. Testing the Portfolio Manager requires only test cases around the $100,000 threshold. You no longer need combinatorial test cases that exercise every role against every order value.

### Liabilities

- **Unhandled requests.** If no handler in the chain can process a request, the request falls through without being handled. Every implementation must decide how to handle this case — return a default value, throw an exception, or guarantee a terminal handler.

- **No guaranteed handling.** The pattern guarantees that the request travels through the chain, but it does not guarantee which handler processes it. If a specific handler must always process certain requests, the Chain of Responsibility alone cannot enforce this.

- **Performance overhead.** Each link in the chain adds a method call. For short chains (3-5 handlers), the overhead is negligible. For long chains or high-frequency requests, the cumulative overhead can become significant.

- **Debugging difficulty.** The chain's behavior is determined at runtime by the chain configuration. Tracing which handler processed a request can be harder than reading a straightforward conditional statement, especially when the chain is long or dynamically configured.

- **Chain fragmentation.** If the chain becomes too long, it can be difficult to understand the overall approval flow. A chain with 20 handlers is harder to reason about than a chain with 4 handlers. Consider breaking long chains into sub-chains or using a composite handler structure.

## Variations

The Chain of Responsibility pattern has several common variations:

**CoR with a list of handlers.** Instead of each handler storing a reference to the next handler, the chain is stored as a list in a separate dispatcher object. The dispatcher iterates through the list and calls each handler until one handles the request. This variation makes it easier to add, remove, or reorder handlers at runtime because you only need to modify the list, not the handler references.

**CoR with a fallback handler.** Instead of a terminal handler, the chain stores a fallback handler that is called when no handler in the chain can process the request. This separates the handling logic from the fallback logic, making the chain more modular.

**CoR with event propagation.** Some implementations allow the request to continue through the chain even after it has been handled. This is useful when you want to log or audit the request at every level. For example, even after the Portfolio Manager approves the order, the Risk Director might still want to log the approval for compliance purposes. This variation blurs the line between Chain of Responsibility and Decorator.

**CoR with conditional branching.** Instead of a linear chain, the handler can decide which handler to forward to based on the request content. For example, a routing handler might forward buy orders to one sub-chain and sell orders to another sub-chain. This creates a tree-like structure that is more flexible than a strict linear chain.

## Code Example Walkthrough

Let us trace through the execution of the pseudocode with the four test orders to understand the full runtime behavior of the chain:

**Order 1: AAPL, 5 shares at $150 (total $750).**

The `main()` function calls `chain_head.approve_order(aapl_order)`. The chain head is the `JuniorTrader` instance. Inside `approve_order`, the base class calls `_can_approve(aapl_order)`. The Junior Trader checks `750 <= 1000`, which is `True`. The method returns `"Approved by Junior Trader"`. The chain stops here. The Senior Trader, Portfolio Manager, and Risk Director never see this order.

**Order 2: GOOGL, 20 shares at $250 (total $5,000).**

The `main()` function calls `chain_head.approve_order(googl_order)`. The Junior Trader checks `5000 <= 1000` — `False`. It forwards to the Senior Trader. The Senior Trader checks `5000 <= 10000` — `True`. The method returns `"Approved by Senior Trader"`. The Portfolio Manager and Risk Director never see this order.

**Order 3: MSFT, 500 shares at $100 (total $50,000).**

The Junior Trader checks `50000 <= 1000` — `False`, forwards to Senior Trader. Senior Trader checks `50000 <= 10000` — `False`, forwards to Portfolio Manager. Portfolio Manager checks `50000 <= 100000` — `True`. Returns `"Approved by Portfolio Manager"`. The Risk Director never sees this order.

**Order 4: AMZN, 2,000 shares at $300 (total $600,000).**

The Junior Trader forwards (exceeds $1,000). The Senior Trader forwards (exceeds $10,000). The Portfolio Manager forwards (exceeds $100,000). The Risk Director checks `_can_approve` — always returns `True`. Returns `"Approved by Risk Director"`. This demonstrates the terminal handler in action.

This walkthrough shows how the chain naturally routes each request to the appropriate handler. The client code is identical for all four orders — only the chain configuration determines which handler processes each request. This is the essence of the Chain of Responsibility pattern: the client does not know and does not care which handler ultimately approves the order.

## Implementation Considerations

When implementing the Chain of Responsibility pattern, consider the following:

**Representing the chain.** The simplest implementation uses object references: each handler stores a pointer to the next handler. This creates a singly linked list. More sophisticated implementations can use a list of handlers that is iterated internally, which allows runtime reordering and dynamic insertion. The linked-list approach is simpler; the list approach is more flexible.

**Terminal handler behavior.** Decide what happens when no handler in the chain can process the request. The options are: (a) return a default value like `null` or a rejection message, (b) throw an exception, or (c) guarantee a terminal handler that always handles the request. Option (c) is the safest but requires discipline to always include a terminal handler in the chain.

**Immutable requests.** Requests flowing through the chain should be immutable. If a handler could modify the request, it could introduce subtle bugs where later handlers receive corrupted data. If handlers need to add information, consider a separate context object that accumulates metadata without modifying the original request.

**Chain construction.** The chain can be constructed in client code (as shown in the pseudocode) or managed by a configuration system. For enterprise applications, you might define the chain in a configuration file or database, load it at startup, and allow runtime modifications through an admin interface.

**Thread safety.** If the chain is shared across threads, ensure that handlers are stateless (they carry only configuration, not per-request state). In the stock market example, each handler stores only its approval limit and the reference to the next handler — both are set during construction and never change, making the chain inherently thread-safe.

## Summary

The Chain of Responsibility pattern decouples request senders from receivers by building a linked chain of handlers. Each handler decides whether to process the request or forward it to the next handler in the chain. The sender knows only the head of the chain and has no knowledge of the chain's structure, size, or the specific handler that ultimately processes the request.

In the stock market trade approval system, this pattern models the natural escalation hierarchy found in real trading firms. A trade order starts at the lowest approval level (Junior Trader) and climbs through the chain — Senior Trader, Portfolio Manager, and finally Risk Director — until it reaches an approver with sufficient authority. Each handler encapsulates a single, well-defined approval rule, making the system easy to test, extend, and maintain.

The pattern's primary benefits are:

- **Decoupling.** The trading system does not depend on the approval hierarchy. Changes to approval rules or roles do not affect the code that creates and submits orders.
- **Flexibility.** New handlers can be added, existing handlers can be modified or removed, and the chain can be reordered — all without changing other handlers or the client code.
- **Testability.** Each handler is a small, focused class with a single responsibility. Testing a Junior Trader means testing only the $1,000 limit, not every combination of roles and limits.
- **Reusability.** Handlers can participate in multiple chains. The same `ComplianceOfficer` handler might appear in approval chains for equities, options, and fixed-income products.

The pattern is not a universal solution — it adds indirection that is not justified for simple, static hierarchies, and its linear nature makes it unsuitable for complex, multi-way approval workflows. But for systems with dynamic, extensible approval chains — whether in stock trading, help desk ticketing, or government permits — the Chain of Responsibility pattern provides a clean, maintainable architecture that evolves with the business rules it models.
