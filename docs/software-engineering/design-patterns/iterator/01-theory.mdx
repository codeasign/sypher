**Intent:** Provide a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

The Iterator pattern decouples traversal logic from the data structure it traverses. Instead of each collection type exposing its internal layout (array indices, map keys, tree nodes), it hands out an iterator object that knows how to walk its elements one at a time. The client calls `hasNext` and `next` -- it never sees the backing store.

In the stock market domain, this pattern is everywhere. A trading platform needs to page through time-series price data, scan an order book's bid and ask levels, walk a portfolio's positions, or stream trade confirmations from a message queue. Each of these collections has a different internal structure -- a sorted list for price history, a balanced BST for an order book, a hash map for a portfolio, a ring buffer for a trade feed -- but each can expose an iterator so the consuming code stays uniform. The trader's dashboard never needs to know whether it is walking a `PriceHistory`, an `OrderBook`, or a `PositionMap`; it just says `while it.hasNext(): process(it.next())`.

---

## The Problem Without This Pattern

Consider a brokerage application that must repeatedly display a list of recent stock prices, calculate a moving average, find the highest closing price, and export the data to a CSV file. Each of these tasks requires sequential access to the elements of a `PriceHistory` object.

Without the Iterator pattern, every consumer of `PriceHistory` must know exactly how the prices are stored internally. If the team stores prices in a simple Python list, every function indexes into it directly:

```python
# BAD -- every consumer couples to the internal storage
class PriceHistory:
    def __init__(self, symbol: str):
        self.symbol = symbol
        self.prices: list[StockPrice] = []

    def price_count(self) -> int:
        return len(self.prices)

    def get_price(self, index: int) -> StockPrice:
        return self.prices[index]


def display_prices(history: PriceHistory) -> None:
    for i in range(history.price_count()):
        p = history.get_price(i)
        print(f"{p.date} | {p.symbol}: ${p.close}")


def calc_moving_average(history: PriceHistory, window: int) -> float:
    total = 0.0
    count = min(window, history.price_count())
    for i in range(history.price_count() - count, history.price_count()):
        total += history.get_price(i).close
    return total / count
```

Every function duplicates the index-based loop. If the team later decides to store prices in a `deque` (for efficient appends to both ends) or a sorted `OrderedDict` keyed by date, every caller breaks. The `get_price(index)` method itself becomes ambiguous -- is the index zero-based insertion order, or chronological order, or sorted-by-symbol order?

The problem escalates with richer data structures. An `OrderBook` stores bids and asks in two separate balanced BSTs keyed by price level. A `Portfolio` maps ticker symbols to `Position` objects in a hash table. Each exposes a different iteration API:

```python
# BAD -- each collection has its own traversal interface
order_book = OrderBook()
for level in order_book.bid_levels():   # returns a specific list
    ...

portfolio = Portfolio()
for symbol in portfolio.symbols():      # returns keys
    ...

trades = TradeFeed()
while True:                             # blocking read
    trade = trades.read_next()
    if trade is None:
        break
```

The client code for scanning trades looks nothing like the client code for scanning the order book, even though both are doing the same thing: walk elements one by one. This inconsistency multiplies the surface area for bugs, makes code reuse harder, and forces every developer to learn each collection's bespoke traversal API.

## Structure

The Iterator pattern defines four participants that work together to encapsulate traversal:

| Participant | Role | Stock Market Analogy |
|---|---|---|
| **Iterator (interface)** | Declares `hasNext()` and `next()` operations | The contract any price traversal routine must satisfy |
| **ConcreteIterator** | Implements the iterator interface and tracks current position | A `PriceHistoryIterator` that walks a sorted price list |
| **Aggregate (interface)** | Declares `createIterator()` to produce an iterator | The contract any price collection must satisfy |
| **ConcreteAggregate** | Implements `createIterator()` and returns a new iterator for itself | A `PriceHistory` that creates a `PriceHistoryIterator` |

The client depends only on the `Iterator` and `Aggregate` interfaces. It never touches the concrete iterator or the internal structure of the aggregate. The aggregate is the factory that produces a matching iterator: an `OrderBook` produces an `OrderBookIterator`, a `Portfolio` produces a `PortfolioIterator`, and so on -- but the client treats them all as `Iterator`.

## Pseudocode -- Stock Market Example

The following pseudocode implements the Iterator pattern for stock price history traversal. Each participant is labeled with its role in brackets.

```
// PSEUDOCODE -- Iterator in Stock Market context
// Participants labeled with [ROLE]

[ITERATOR INTERFACE]
  interface StockIterator:
    method hasNext() -> bool
    method next() -> StockPrice

[CONCRETE ITERATOR]
  class PriceHistoryIterator implements StockIterator:
    private prices: list of StockPrice
    private position: int = 0

    method hasNext() -> bool:
      return position < prices.length

    method next() -> StockPrice:
      if not hasNext():
        throw NoSuchElementException
      price = prices[position]
      position++
      return price

[AGGREGATE INTERFACE]
  interface StockPriceCollection:
    method createIterator() -> StockIterator

[CONCRETE AGGREGATE]
  class PriceHistory implements StockPriceCollection:
    private prices: list of StockPrice

    constructor(symbol: string, prices: list of StockPrice):
      this.symbol = symbol
      this.prices = prices

    method addPrice(price: StockPrice):
      prices.add(price)

    method createIterator() -> StockIterator:
      return new PriceHistoryIterator(this.prices)

[CLIENT]
  class TradingDashboard:
    method displayPrices(history: StockPriceCollection):
      it = history.createIterator()
      while it.hasNext():
        price = it.next()
        print(f"{price.date} | {price.symbol}: ${price.close}")
```

The `TradingDashboard` client never imports `PriceHistoryIterator` or touches the internal `prices` list. It receives a `StockPriceCollection`, calls `createIterator()` to get a `StockIterator`, and walks the elements in a uniform loop. This decoupling means the internal storage of `PriceHistory` can change from a list to a `deque`, a database cursor, or a remote API paginator -- and the client code stays the same.

## How It Works -- Step by Step

The following walkthrough traces the interaction between the `TradingDashboard` client, the `PriceHistory` aggregate, and the `PriceHistoryIterator` when the dashboard calls `displayPrices`.

**Step 1: Client receives the aggregate.** The `TradingDashboard.displayPrices` method is called with a `StockPriceCollection` reference pointing to a `PriceHistory` instance that holds 10,000 daily close prices for AAPL.

**Step 2: Client requests an iterator.** `displayPrices` calls `history.createIterator()`. The `PriceHistory` object constructs a new `PriceHistoryIterator`, passing a reference to its internal `prices` list. The iterator's internal `position` is initialized to 0. The client receives the iterator and stores it as a `StockIterator` interface reference.

**Step 3: First call to `hasNext()`.** The client enters the while loop and calls `it.hasNext()`. Inside `PriceHistoryIterator`, this compares `position` (0) against `prices.length` (10,000). Since 0 < 10,000, it returns `true`.

**Step 4: First call to `next()`.** The client calls `it.next()`. The iterator checks `hasNext()` (true), retrieves `prices[0]` (the oldest AAPL daily close), increments `position` to 1, and returns the `StockPrice` object. The client prints the date, symbol, and closing price.

**Steps 5-10,003: Loop continues.** The client repeats the `hasNext`/`next` cycle. With each call to `next()`, the iterator advances `position` by one and returns the next `StockPrice` in chronological order. The client never knows whether it is walking a raw array, a linked list, or a database result set.

**Step 10,004: Final call to `hasNext()`.**
After 10,000 iterations, the iterator's `position` equals 10,000, which is not less than `prices.length`. `hasNext()` returns `false`.

**Step 10,005: Loop terminates.** The while condition evaluates to `false`. The client exits the loop and the method returns. The iterator object goes out of scope and is garbage collected. The aggregate's internal list is unchanged.

**Step 10,006: A second traversal starts fresh.** If `displayPrices` is called again on the same `PriceHistory`, a brand new `PriceHistoryIterator` is created with `position = 0`. Multiple iterators can coexist on the same aggregate without interfering -- one thread could be scanning prices forward while another scans backward.

## When to Use

Use the Iterator pattern when you recognize any of these signals in your stock market codebase:

- **You have multiple collection types that need uniform traversal.** Your codebase has `PriceHistory`, `OrderBook`, `Portfolio`, and `TradeFeed`, each with a different internal structure, and you want to write generic analysis functions that work on any of them.

- **You want to hide the internal representation of a collection.** The `PriceHistory` stores prices in a sorted list today, but you plan to migrate to a time-series database tomorrow. The iterator insulates every consumer from the change.

- **You need multiple simultaneous traversals over the same collection.** A trading algorithm computes a fast moving average (5-day window) and a slow moving average (200-day window) over the same `PriceHistory` in a single pass. Two iterators on the same aggregate, each at a different position, make this straightforward.

- **You want to provide multiple traversal strategies.** A `PriceHistory` might offer a forward iterator (oldest first), a reverse iterator (newest first), and a range iterator (only prices within a date window). Each is a separate `ConcreteIterator` class, and the aggregate's `createIterator` method can accept a parameter to select one.

- **You need lazy or deferred iteration over a large data set.** A stock exchange's historical trade feed contains terabytes of data. An iterator backed by a database cursor fetches one page at a time rather than loading the entire history into memory.

- **You are writing a generic function that processes elements without caring about the source.** A `calculateVWAP` (volume-weighted average price) function can accept any `StockPriceCollection` and walk it with an iterator, whether the source is a real-time feed, a backtest replay, or a CSV file.

---

## When NOT to Use

Avoid the Iterator pattern when the following conditions hold:

- **The aggregate is a simple, stable collection with a single traversal order.** If `PriceHistory` is always and forever a flat Python list that you iterate front-to-back, a plain `for price in history.prices` loop is clearer and faster. Adding an iterator class is unnecessary indirection.

- **The client must know the concrete collection type anyway.** If the `TradingDashboard` needs to call `history.latestPrice()` or `history.priceAt(date)` in addition to iteration, the abstract interface leaks. The client already depends on `PriceHistory` directly, so the iterator abstraction buys you nothing.

- **Performance is critical and the iterator allocation matters.** In a high-frequency trading engine that iterates over a price buffer millions of times per second, the overhead of allocating a new iterator object on every traversal can add up. A raw index-based loop avoids the allocation and the virtual dispatch.

- **The collection is empty or trivially small.** Iterating over a portfolio with three positions does not benefit from the Iterator pattern's abstraction. A simple `for` loop over the underlying data structure is more readable.

- **You control the caller and the callee in the same codebase, and neither changes often.** If the same team owns `PriceHistory` and all its consumers, and the storage format has been stable for years, there is no need for an abstract iterator. The abstraction tax (extra classes, interface indirection, harder debugging) outweighs the hypothetical benefit of future change.

- **Language idioms already provide a simpler iteration mechanism.** In Python, the `__iter__` and `__next__` dunder methods are the iterator pattern built into the language. Using them directly is idiomatic. Implementing a separate `StockIterator` interface on top of them is redundant -- just make `PriceHistory` implement `__iter__` and let `for price in history` work naturally. Same for JavaScript's `Symbol.iterator` and C#'s `foreach`/`IEnumerable`.

## Real-World Occurrences

The Iterator pattern is one of the most widely adopted patterns in standard libraries across every major programming language:

### Java: `java.util.Iterator`

Every `Collection` in Java (including `ArrayList`, `HashSet`, `TreeSet`, `LinkedList`, `ConcurrentLinkedQueue`) implements `Iterable`, which returns an `Iterator`. A `TreeSet` of stock tickers returns an iterator that walks elements in sorted order. A `LinkedList` of trade records returns an iterator that walks elements in insertion order. The client uses the same `Iterator` interface for both.

```java
// Java Iterator in practice with a stock portfolio
Set<String> tickers = new TreeSet<>();
tickers.add("AAPL");
tickers.add("MSFT");
tickers.add("GOOGL");

Iterator<String> it = tickers.iterator();
while (it.hasNext()) {
    String ticker = it.next();
    System.out.println(ticker);  // prints in alphabetical order
}
```

Java also supports the enhanced `for-each` loop, which desugars into an iterator:

```java
for (String ticker : tickers) {
    System.out.println(ticker);
}
```

The `ListIterator` subinterface adds bidirectional traversal (`previous()`, `hasPrevious()`) and positional operations (`nextIndex()`, `previousIndex()`), useful for navigating an ordered trade log in both directions.

### Python: `__iter__` and `__next__` Protocol

Python builds the Iterator pattern directly into the language with dunder methods. Any object that implements `__iter__` (returns an iterator) and `__next__` (returns the next element or raises `StopIteration`) can be used in a `for` loop.

```python
# Python __iter__ / __next__ protocol in a stock price history
class PriceHistory:
    def __init__(self, symbol: str):
        self.symbol = symbol
        self._prices: list[StockPrice] = []

    def add_price(self, price: StockPrice) -> None:
        self._prices.append(price)

    def __iter__(self):
        return PriceIterator(self._prices)


class PriceIterator:
    def __init__(self, prices: list[StockPrice]):
        self._prices = prices
        self._index = 0

    def __next__(self) -> StockPrice:
        if self._index >= len(self._prices):
            raise StopIteration
        price = self._prices[self._index]
        self._index += 1
        return price


history = PriceHistory("AAPL")
# ... populate ...
for price in history:           # uses the iterator implicitly
    print(f"{price.date}: ${price.close}")
```

Python also provides generator functions, which are a compact way to create iterators without writing an explicit iterator class:

```python
def price_iterator(prices: list[StockPrice]):
    for price in prices:
        yield price
```

### JavaScript: `Symbol.iterator`

Any JavaScript object can become iterable by defining a `[Symbol.iterator]` method that returns an iterator object with a `next()` method returning `{value, done}`.

```javascript
// JavaScript Symbol.iterator for a portfolio
const portfolio = {
    positions: new Map(),
    [Symbol.iterator]: function* () {
        for (const [symbol, position] of this.positions) {
            yield { symbol, ...position };
        }
    }
};

portfolio.positions.set("AAPL", { shares: 100, avgPrice: 175.20 });
portfolio.positions.set("MSFT", { shares: 50,  avgPrice: 380.15 });

for (const pos of portfolio) {
    console.log(`${pos.symbol}: ${pos.shares} shares`);
}
```

JavaScript arrays, maps, sets, strings, and `NodeList` all implement `Symbol.iterator`, making them usable with `for...of`, spread syntax (`[...collection]`), and `Array.from()`.

### C#: `IEnumerable` and `IEnumerator`

C# distinguishes the `IEnumerable<T>` interface (which returns an enumerator) from `IEnumerator<T>` (which tracks position). The `foreach` keyword consumes any `IEnumerable`.

```csharp
// C# IEnumerable for a stock price collection
public class PriceHistory : IEnumerable<StockPrice>
{
    private List<StockPrice> _prices = new();

    public void AddPrice(StockPrice price) => _prices.Add(price);

    public IEnumerator<StockPrice> GetEnumerator()
        => new PriceEnumerator(_prices);

    IEnumerator IEnumerable.GetEnumerator()
        => GetEnumerator();
}

public class PriceEnumerator : IEnumerator<StockPrice>
{
    private List<StockPrice> _prices;
    private int _index = -1;

    public StockPrice Current => _prices[_index];
    object IEnumerator.Current => Current;

    public bool MoveNext() => ++_index < _prices.Count;
    public void Reset() => _index = -1;
    public void Dispose() { }

    public PriceEnumerator(List<StockPrice> prices)
        => _prices = prices;
}

// Client code
var history = new PriceHistory();
// ... populate ...
foreach (var price in history)
{
    Console.WriteLine($"{price.Date}: {price.Close}");
}
```

### Rust: `Iterator` Trait

Rust's `Iterator` trait is defined in the standard library and provides a rich set of combinators (`map`, `filter`, `fold`, `take`, `skip`, `zip`) that all build on the core `next()` method.

```rust
// Rust Iterator trait for a trade feed
struct TradeFeed {
    trades: Vec<Trade>,
    index: usize,
}

impl Iterator for TradeFeed {
    type Item = Trade;

    fn next(&mut self) -> Option<Self::Item> {
        if self.index >= self.trades.len() {
            return None;
        }
        let trade = self.trades[self.index].clone();
        self.index += 1;
        Some(trade)
    }
}

// Client code uses combinator methods
let vwap: f64 = feed
    .take(1000)
    .map(|t| t.price * t.volume as f64)
    .sum::<f64>()
    / feed.take(1000).map(|t| t.volume).sum::<u64>() as f64;
```

Rust's iterators are zero-cost abstractions: the compiler inlines and optimizes the iterator chain into a tight loop with no allocation overhead.

### .NET LINQ: Deferred Execution with Iterators

Language Integrated Query (LINQ) in .NET is built on the Iterator pattern. When you write a LINQ query over an `IEnumerable<T>`, you are composing iterators that execute lazily:

```csharp
// LINQ uses iterator pattern under the hood for deferred execution
var aaplPrices = history
    .Where(p => p.Symbol == "AAPL")
    .OrderByDescending(p => p.Date)
    .Select(p => new { p.Date, p.Close });

// No iteration occurs here -- just iterator composition
foreach (var price in aaplPrices)
{
    // Actual iteration happens here, one element at a time
    Console.WriteLine($"{price.Date}: {price.Close}");
}
```

Each LINQ method (`Where`, `OrderBy`, `Select`) returns an iterator that wraps the previous one. `Where` yields elements that match the predicate, `OrderBy` buffers and sorts, `Select` transforms each element. The chain is evaluated one element at a time as the `foreach` loop pulls.

### Standard Library Examples Across Other Languages

- **C++:** STL iterators with `begin()` and `end()` for `vector`, `map`, `set`, `list`, `deque`, `unordered_map`. C++20 introduces ranges (`std::ranges::views`) that compose iterators similarly to LINQ.
- **Kotlin:** `iterator()` operator on `Iterable<T>`, with rich sequence processing via `Sequence<T>`. The `for` loop desugars to `while (it.hasNext())`.
- **Ruby:** `each` method on `Array`, `Hash`, `Range`, and custom enumerables via `Enumerable` mixin. `each` returns an `Enumerator` when called without a block.
- **Go:** No built-in iterator interface in the standard library, but Go 1.23 introduced iterator functions (`iter.Seq`) as a convention for ranging over custom sequences.
- **Swift:** `Sequence` and `IteratorProtocol` in the standard library. Any type conforming to `Sequence` can be used in a `for...in` loop.
- **Scala:** `Iterator` trait with `hasNext` and `next`, plus rich combinator methods. Collections like `List`, `Set`, `Map` are `Iterable`.

## Comparison with Related Patterns

The Iterator pattern is closely related to several other patterns and idioms. Knowing when to reach for Iterator versus its alternatives is key to clean design.

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Composite** | Composite lets you treat individual objects and compositions uniformly via a tree structure. Iterator walks the tree's elements. | Use Composite when you are building a tree of objects (e.g., a market data feed that aggregates multiple sub-feeds). Use Iterator when you need to traverse that tree without exposing its structure. A Composite often provides its own iterator (e.g., a recursive tree iterator). |
| **Visitor** | Visitor adds new operations to a set of classes without modifying them. Iterator provides sequential access to elements so a Visitor can process them. | Use Visitor when you need to perform many unrelated operations on elements of a fixed structure (e.g., calculate metrics, export to CSV, send alerts -- all on a `Portfolio`). Use Iterator when your primary need is traversal itself. Visitor and Iterator complement each other: an iterator walks the elements, a visitor processes each one. |
| **Factory Method** | Factory Method creates objects without specifying their concrete classes. In the Iterator pattern, the aggregate's `createIterator()` is itself a Factory Method. | Use Factory Method when you want subclasses of an aggregate to decide which iterator to return (e.g., an in-memory `PriceHistory` returns a sequential iterator; a database-backed `PriceHistory` returns a cursor iterator). The `createIterator()` method in the aggregate is the Factory Method. |
| **Strategy** | Strategy defines a family of interchangeable algorithms. Iterator defines a family of interchangeable traversal algorithms. | Use Strategy when the algorithm operates on the entire collection (e.g., a `ScreeningStrategy` that scans 10,000 stocks for a pattern). Use Iterator when the algorithm steps through elements one at a time and you might swap traversal order (forward, reverse, random-access). |
| **Memento** | Memento captures and externalizes an object's internal state without violating encapsulation. Iterator's position is a lightweight form of memento. | Use Memento when you need to snapshot an iterator's state and restore it later (e.g., pause and resume a backtest run). Some iterator implementations expose a "checkpoint" that captures the current position and can recreate an iterator at that point. |
| **Generator / Coroutine** | Generators are language features that produce iterators without writing an iterator class. They implement the Iterator pattern implicitly. | Use generators (Python `yield`, JavaScript `function*`, C# `yield return`) when you want the cleanest syntax for a simple, single-direction iterator. Use an explicit iterator class when you need bidirectional traversal, reset capability, or multiple independent iterator states on the same aggregate. |
| **Observer** | Observer notifies subscribers of state changes. Iterator pulls elements from a collection on demand. | Use Observer when you need real-time push notifications (e.g., alert when a stock price crosses a threshold). Use Iterator when the consumer controls the pace of data retrieval (e.g., batch processing historical prices). The two can merge: an observable data source can fan out to multiple iterators. |

## Practical Example: Order Book Iterator

To ground the Iterator pattern in a concrete stock-market scenario, consider an order book that maintains bid and ask levels. The order book manages two internal red-black trees: one for bids (sorted descending by price) and one for asks (sorted ascending by price). An iterator abstracts away the tree structure so that a matching engine can scan levels without touching the tree directly.

```python
from dataclasses import dataclass
from typing import Optional


@dataclass
class OrderBookLevel:
    price: float
    volume: int
    side: str  # "bid" or "ask"


class OrderBookIterator:
    """Iterates over order book levels in price priority order."""

    def __init__(self, levels: list[OrderBookLevel]):
        self._levels = sorted(
            levels,
            key=lambda x: (-x.price, x.price) if x.side == "bid" else x.price,
        )
        self._index = 0

    def __iter__(self):
        return self

    def __next__(self) -> OrderBookLevel:
        if self._index >= len(self._levels):
            raise StopIteration
        level = self._levels[self._index]
        self._index += 1
        return level


class OrderBook:
    """Aggregate that holds bid and ask levels."""

    def __init__(self, symbol: str):
        self.symbol = symbol
        self._bids: list[OrderBookLevel] = []
        self._asks: list[OrderBookLevel] = []

    def add_bid(self, price: float, volume: int) -> None:
        self._bids.append(OrderBookLevel(price, volume, "bid"))

    def add_ask(self, price: float, volume: int) -> None:
        self._asks.append(OrderBookLevel(price, volume, "ask"))

    def bid_iterator(self) -> OrderBookIterator:
        return OrderBookIterator(list(self._bids))

    def ask_iterator(self) -> OrderBookIterator:
        return OrderBookIterator(list(self._asks))

    def full_iterator(self) -> OrderBookIterator:
        """Return an iterator over all levels, bids first then asks."""
        return OrderBookIterator(list(self._bids + self._asks))


# Client: matching engine scans the order book
def find_best_bid(order_book: OrderBook) -> Optional[OrderBookLevel]:
    for level in order_book.bid_iterator():
        return level  # first level is the highest bid
    return None


def calculate_market_impact(order_book: OrderBook, order_volume: int) -> float:
    """Calculate the average fill price for a market order of given volume."""
    filled = 0
    total_cost = 0.0
    for level in order_book.ask_iterator():
        fill = min(level.volume, order_volume - filled)
        total_cost += fill * level.price
        filled += fill
        if filled >= order_volume:
            break
    return total_cost / filled if filled > 0 else 0.0
```

The matching engine uses the iterator to scan order book levels without ever calling `_bids` or `_asks` directly. If the internal storage changes from lists to a `sortedcontainers.SortedDict`, only the iterator changes -- the matching engine stays the same.

## Practical Example: Reverse Iterator for Price History

A price history is typically traversed oldest-first. But a trading dashboard also needs to display the most recent prices first. A reverse iterator provides this traversal without modifying the underlying collection.

```python
class ReversePriceIterator:
    """Iterates a PriceHistory from newest to oldest."""

    def __init__(self, prices: list[StockPrice]):
        self._prices = prices
        self._index = len(prices) - 1

    def __iter__(self):
        return self

    def __next__(self) -> StockPrice:
        if self._index < 0:
            raise StopIteration
        price = self._prices[self._index]
        self._index -= 1
        return price


class PriceHistory:
    """StockPrice collection with forward and reverse iterators."""

    def __init__(self, symbol: str):
        self.symbol = symbol
        self._prices: list[StockPrice] = []

    def add_price(self, price: StockPrice) -> None:
        self._prices.append(price)

    def __iter__(self):
        """Forward iterator: oldest first."""
        return PriceIterator(self._prices)

    def reverse_iterator(self):
        """Reverse iterator: newest first."""
        return ReversePriceIterator(self._prices)


# Dashboard renders prices newest-first
def render_price_table(history: PriceHistory) -> None:
    print(f"Recent prices for {history.symbol}:")
    for price in history.reverse_iterator():
        print(f"{price.date}: ${price.close}")
```

Having both `__iter__` (forward) and `reverse_iterator()` (backward) on the same aggregate illustrates how an aggregate can offer multiple iterator types, each encapsulating a different traversal strategy.

## Performance Considerations

The Iterator pattern introduces a few overheads that matter in latency-sensitive trading systems:

- **Allocation cost:** Each call to `createIterator()` allocates a new iterator object. In a Java garbage-collected environment, thousands of iterators created per second can pressure the young generation. Pooling iterators or reusing them can mitigate this.

- **Virtual dispatch:** Every call to `hasNext()` and `next()` goes through an interface dispatch (vtable lookup in C++, virtual method in Java, trait method in Rust). In tight loops processing millions of ticks, this dispatch overhead can be measurable. Rust's monomorphized iterators eliminate this cost at compile time.

- **Bounds checks:** Iterator implementations typically check `hasNext()` on every call. A raw indexed loop that trusts the developer to stay in bounds avoids one check per iteration. Safe languages enforce bounds checking anyway, so the advantage is minimal.

- **Boxing:** In languages with value types (C# structs, Java primitives), wrapping values in an iterator can introduce boxing overhead. C# `IEnumerator<T>` is a reference type; implementing it with a struct avoids heap allocation but requires careful implementation to avoid boxing at the `IEnumerator` interface boundary.

For non-latency-critical code (backtesting, reporting, dashboard rendering, compliance auditing), these overheads are negligible. The Iterator pattern's benefits in code clarity, separation of concerns, and abstraction vastly outweigh the micro-level costs.

## Summary

The Iterator pattern solves one problem -- "how do I walk the elements of a collection without knowing how it is built?" -- and solves it so well that every modern language embeds it in the standard library. In the stock market domain, where data comes in many shapes (time-series arrays, sorted trees, hash maps, paginated APIs, infinite streams), the Iterator pattern provides a uniform interface for traversal that decouples algorithms from data structures.

The pattern's longevity is a testament to its simplicity: two methods (`hasNext` and `next`) are enough to represent an infinite variety of collections and traversal strategies. Whether you are scanning a price history for a golden cross, paging through an order book to compute market impact, or streaming live trades into a monitoring dashboard, the Iterator pattern gives you a consistent, encapsulated, and testable way to move through your data one element at a time.
