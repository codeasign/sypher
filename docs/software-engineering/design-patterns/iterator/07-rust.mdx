The Iterator pattern is a fundamental concept in the Stock Market domain, where traversing price histories, trade logs, and portfolio snapshots must be both efficient and safe. Rust's approach to iteration combines zero-cost abstractions with strong compile-time guarantees. By implementing the `Iterator` trait (and its underlying `IntoIterator`), a stock price collection can support lazy, composable, and filterable traversal without allocation overhead. The pattern maps naturally to Rust's ownership and borrowing rules: iterators borrow collections immutably, consume them, or produce owned items depending on the desired semantics.

## How the Implementation Works

### Class Diagram (Pseudocode)

```
trait StockIterator {
    fn has_next(&mut self) -> bool;
    fn next(&mut self) -> StockPrice;
}

struct PriceHistoryIterator {
    prices: Vec<StockPrice>,
    position: usize,
}

impl StockIterator for PriceHistoryIterator {
    fn has_next(&self) -> bool;
    fn next(&mut self) -> StockPrice;
}

struct PriceHistory {
    symbol: String,
    prices: Vec<StockPrice>,
}

impl PriceHistory {
    fn create_iterator(&self) -> PriceHistoryIterator;
}

impl IntoIterator for PriceHistory { ... }  // Rustic iteration

struct HighVolumeFilter {
    inner: Box<dyn StockIterator>,
    min_volume: u64,
    buffer: Option<StockPrice>,
}

impl StockIterator for HighVolumeFilter { ... }

    inner: Box<dyn StockIterator>,
    start_date: String,

    buffer: Option<StockPrice>,
}

impl StockIterator for DateRangeFilter { ... }
```

### OOP Concepts Applied

- **Encapsulation** -- The `PriceHistoryIterator` hides its position counter and internal slice reference. The `PriceHistory` struct encapsulates its `Vec<StockPrice>` behind public methods like `add_price()` and `create_iterator()`.
- **Abstraction** -- The `StockIterator` trait defines a clean traversal contract. Consumers depend on the trait, not on concrete iterator types. `Box<dyn StockIterator>` enables runtime polymorphism.
- **Inheritance** -- Rust uses trait inheritance (`trait StockIterator: fmt::Debug`) and composition rather than class inheritance. Types implement traits to inherit behavior. `PriceHistoryIterator` inherits no data but implements the `StockIterator` contract.
- **Polymorphism** -- `HighVolumeFilter` and `DateRangeFilter` both implement `StockIterator` and can be used interchangeably. The `Box<dyn StockIterator>` trait object allows storing any iterator implementation behind a single pointer.

### SOLID Principles Applied

- **SRP (Single Responsibility)** -- `PriceHistory` is solely responsible for holding stock price data. `PriceHistoryIterator` handles traversal state. `HighVolumeFilter` and `DateRangeFilter` each handle one filtering concern.
- **OCP (Open/Closed)** -- New filtering iterators (e.g., `PriceRangeFilter`, `MovingAverageFilter`) can be added by implementing `StockIterator` without modifying any existing code. The system is open for extension, closed for modification.
- **LSP (Liskov Substitution)** -- Any `Box<dyn StockIterator>` can be substituted for another. The decorator iterators wrap an inner iterator and follow the same contract, so they can be nested or chained without the client knowing.
- **ISP (Interface Segregation)** -- The `StockIterator` trait is minimal (two methods). Clients depend only on the traversal interface. `IntoIterator` is a separate concern for Rust-native iteration support.
- **DIP (Dependency Inversion)** -- High-level code depends on the `StockIterator` abstraction rather than concrete implementations. Filter iterators accept `Box<dyn StockIterator>` via constructor injection.

### StockIterator (Iterator Trait)

```rust
// ── Iterator Trait ──
pub trait StockIterator: fmt::Debug {
    /// Returns true if there are more elements to visit.
    /// Takes &mut self because filter implementations may need to scan
    /// forward and buffer a candidate.
    fn has_next(&mut self) -> bool;
    /// Returns the next element. Panics if no more elements exist.
    fn next(&mut self) -> StockPrice;
}
```

### PriceHistoryIterator (Concrete Iterator)

```rust
// ── Concrete Iterator ──
#[derive(Debug)]
pub struct PriceHistoryIterator {
    prices: Vec<StockPrice>,
    position: usize,
}

impl PriceHistoryIterator {
    pub fn new(prices: Vec<StockPrice>) -> Self {
        Self {
            prices,
            position: 0,
        }
    }
}

impl StockIterator for PriceHistoryIterator {
    fn has_next(&mut self) -> bool {
        self.position < self.prices.len()
    }

    fn next(&mut self) -> StockPrice {
        if !self.has_next() {
            panic!("No more stock prices");
        }
        let price = self.prices[self.position].clone();
        self.position += 1;
        price
    }
}
```

### PriceHistory (Aggregate)

```rust
// ── Aggregate ──
#[derive(Debug)]
pub struct PriceHistory {
    pub symbol: String,
    prices: Vec<StockPrice>,
}

impl PriceHistory {
    pub fn new(symbol: &str) -> Self {
        Self {
            symbol: symbol.to_string(),
            prices: Vec::new(),
        }
    }

    pub fn add_price(&mut self, price: StockPrice) {
        self.prices.push(price);
    }

    pub fn create_iterator(&self) -> PriceHistoryIterator {
        PriceHistoryIterator::new(self.prices.clone())
    }

    pub fn price_count(&self) -> usize {
        self.prices.len()
    }
}

// IntoIterator implementation for Rustic iteration (consuming)
impl IntoIterator for PriceHistory {
    type Item = StockPrice;
    type IntoIter = std::vec::IntoIter<StockPrice>;

    fn into_iter(self) -> Self::IntoIter {
        self.prices.into_iter()
    }
}

// IntoIterator for references (borrowing iteration)
impl<'a> IntoIterator for &'a PriceHistory {
    type Item = &'a StockPrice;
    type IntoIter = std::slice::Iter<'a, StockPrice>;

    fn into_iter(self) -> Self::IntoIter {
        self.prices.iter()
    }
}
```

### HighVolumeFilter (Decorator: HighVolumeFilter (OCP))

```rust
// ── Decorator: HighVolumeFilter (OCP) ──
#[derive(Debug)]
pub struct HighVolumeFilter {
    inner: Box<dyn StockIterator>,
    min_volume: u64,
    buffer: Option<StockPrice>,
}

impl HighVolumeFilter {
    pub fn new(inner: Box<dyn StockIterator>, min_volume: u64) -> Self {
        Self {
            inner,
            min_volume,
            buffer: None,
        }
    }
}

impl StockIterator for HighVolumeFilter {
    fn has_next(&mut self) -> bool {
        // Scan forward and buffer the next matching element
        while self.buffer.is_none() && self.inner.has_next() {
            let candidate = self.inner.next();
            if candidate.volume >= self.min_volume {
                self.buffer = Some(candidate);
            }
        }
        self.buffer.is_some()
    }

    fn next(&mut self) -> StockPrice {
        // Ensure we have a buffered element (has_next() will scan forward)
        self.has_next();
        self.buffer
            .take()
            .expect("No more matching prices")
    }
}
```

### DateRangeFilter (Decorator: DateRangeFilter (OCP))

```rust
// ── Decorator: DateRangeFilter (OCP) ──
#[derive(Debug)]
pub struct DateRangeFilter {
    inner: Box<dyn StockIterator>,
    start_date: String,
    end_date: String,
    buffer: Option<StockPrice>,
}

impl DateRangeFilter {
    pub fn new(inner: Box<dyn StockIterator>, start_date: &str, end_date: &str) -> Self {
        Self {
            inner,
            start_date: start_date.to_string(),
            end_date: end_date.to_string(),
            buffer: None,
        }
    }
}

impl StockIterator for DateRangeFilter {
    fn has_next(&mut self) -> bool {
        // Scan forward and buffer the next matching element
        while self.buffer.is_none() && self.inner.has_next() {
            let candidate = self.inner.next();
            if candidate.date.as_str() >= self.start_date.as_str()
                && candidate.date.as_str() <= self.end_date.as_str()
            {
                self.buffer = Some(candidate);
            }
        }
        self.buffer.is_some()
    }

    fn next(&mut self) -> StockPrice {
        // Ensure we have a buffered element (has_next() will scan forward)
        self.has_next();
        self.buffer
            .take()
            .expect("No more prices in date range")
    }
}
```

### Language-Specific Notes

- **`IntoIterator` trait** -- Rust's `Iterator` pattern is governed by the `IntoIterator` trait, which allows a type to be converted into an iterator. Implementing `IntoIterator` for both `PriceHistory` (consuming) and `&PriceHistory` (borrowing) enables `for` loop support and integration with Rust's standard iterator adapters. This is equivalent to C#'s `IEnumerable<T>` but with ownership semantics encoded at the type level.
- **Trait objects and dynamic dispatch** -- Using `Box<dyn StockIterator>` enables runtime polymorphism for decorator iterators. However, trait objects come with a heap allocation and vtable indirection. In performance-critical stock market code, prefer generic impl Trait parameters or static dispatch with enums to avoid the allocation overhead.
- **No `has_next()` in standard `Iterator`** -- Rust's standard `Iterator` trait does not have a `has_next()` method; instead, `next()` returns `Option<Self::Item>`. The custom `StockIterator` trait with `has_next()` is provided to mirror the classic GoF pattern. Production Rust code should use the standard `Iterator` trait and its lazy adapter methods (`filter`, `map`, `take_while`, etc.).
- **Clone for stateful filters** -- The `StockIterator` trait requires `StockPrice` to implement `Clone` because `next()` returns an owned value. In a zero-copy production system, consider implementing an iterator that yields references (`Iterator<Item = &StockPrice>`) to avoid cloning each element.
- **No `yield` equivalent** -- Unlike C#'s `yield return`, Rust does not have compiler-generated iterators. Implementations must manually track state (like `position` in `PriceHistoryIterator`). For complex iteration logic, consider using crates like `genawaiter` or the nightly `gen` blocks, or restructure as a state machine.
- **Chaining filter decorators** -- The decorator iterators can be composed arbitrarily. The "MSFT Prices (Jun 3 - Jun 5, volume > 22M)" example chains `DateRangeFilter` inside `HighVolumeFilter`. The order matters: the inner filter runs first (date range), then the outer filter (volume). This composability is a direct application of the Decorator pattern working with Iterator.
