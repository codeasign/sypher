**Intent:** Define an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly, and it lets you vary their interaction independently.

The Mediator pattern introduces a central coordinator that manages communication between collaborating objects, known as colleagues. Instead of objects communicating directly with each other — creating a tangled web of dependencies — they communicate solely through the mediator. The mediator knows about all the colleagues it manages and routes messages, coordinates actions, and enforces communication rules.

In the stock market domain, the Mediator pattern maps directly to the role of a real-world stock exchange. A stock exchange does not allow buyers and sellers to negotiate directly with each other. Instead, all orders flow through the exchange: buyers submit bids, sellers submit asks, and the exchange's matching engine determines when a trade occurs. The exchange is the mediator, and the trading agents (buyers, sellers, market makers) are the colleagues.

Without a mediator, each trader would need to know about every other trader, every price they are willing to accept, and every order they have open. The complexity would grow quadratically with the number of traders. The exchange (mediator) centralizes this complexity and keeps traders decoupled from each other.

---

## The Problem Without This Pattern

Imagine a trading system where every agent communicates directly with every other agent. A BuyAgent that wants to purchase 100 shares of AAPL needs to contact every SellAgent to find a matching sell order. Each SellAgent needs to know about every BuyAgent. The RiskChecker needs to monitor every trade. The ComplianceMonitor needs to audit every communication. The MarketDataFeed needs to push prices to every agent. The PortfolioManager needs updates from every agent.

The result is a many-to-many communication graph where each object holds direct references to many other objects. Adding a new type of agent means updating every existing agent to know about it. Removing an agent requires removing references from potentially dozens of other objects. Testing any single agent requires setting up mocks for every other agent it communicates with.

Here is what this broken design looks like in pseudocode. Notice how every agent holds references to every other agent:

```
// ── Broken: Agents Communicate Directly ──

class BuyAgent {
    private sellAgents: List

    constructor() {
        this.sellAgents = new List()
    }

    method registerSellAgent(agent: SellAgent) {
        this.sellAgents.add(agent)
    }

    method placeBid(symbol: String, qty: Integer, price: Float) {
        // Must iterate over all sell agents to find a match
        for (sellAgent in this.sellAgents) {
            if (sellAgent.hasMatchingAsk(symbol, qty, price)) {
                trade = sellAgent.executeSell(symbol, qty, price)
                // Also need to notify risk checker, compliance, portfolio...
                this.riskChecker.logTrade(trade)
                this.complianceMonitor.auditTrade(trade)
                this.portfolioManager.updatePosition(symbol, qty)
                return trade
            }
        }
    }

    // Direct references to too many other agents
    private riskChecker: RiskChecker
    private complianceMonitor: ComplianceMonitor
    private portfolioManager: PortfolioManager
    private marketDataFeed: MarketDataFeed
}
```

The problems are clear:

- **Tight coupling:** Every agent must know the interface of every other agent.
- **Poor scalability:** Adding one new agent type requires modifying every existing agent.
- **Duplicated logic:** Every buyer and seller implements its own matching logic.
- **Testing nightmare:** Unit testing a BuyAgent requires instantiating SellAgent, RiskChecker, ComplianceMonitor, PortfolioManager, and MarketDataFeed.
- **No central control:** There is no single place to enforce trading rules, price limits, or compliance checks.

The Mediator pattern solves all of these problems by placing a single StockExchange mediator between every trading agent.

## Structure

The Mediator pattern in the stock market domain involves four participants:

| Participant | Role | Stock Market Analogy |
|---|---|---|
| **Mediator** (interface) | Defines the communication contract | StockExchange interface — methods for registering agents, submitting orders, and broadcasting notifications |
| **ConcreteMediator** | Coordinates colleague interactions | NasdaqExchange — maintains the order book, registers agents, matches bids and asks |
| **Colleague** (interface) | Defines the contract for participating objects | TradingAgent interface — methods for sending and receiving messages |
| **ConcreteColleagues** | Objects that communicate through the mediator | BuyAgent, SellAgent, MarketMakerAgent — each with specific trading behavior |

The key structural insight is that no colleague holds a reference to any other colleague. Every colleague holds only a reference to the mediator. When a colleague needs to communicate, it calls the mediator. The mediator routes the message to the appropriate recipient or recipients. This turns the many-to-many communication graph into a star topology with the mediator at the center.

The communication flow follows this pattern:

1. Colleague A calls `mediator.send(message, context)`.
2. The mediator processes the message, applies any business rules (e.g., price matching, compliance checks).
3. The mediator determines which colleague or colleagues should receive the message.
4. The mediator calls `colleague.receive(message)` on each target colleague.
5. Colleagues never know about each other — they only know the mediator.

## Pseudocode — Stock Market Example

The following pseudocode shows how the Mediator pattern works in a stock market trading system. The pseudocode is language-agnostic and focuses on the structural relationships between the mediator and its colleagues.

### Participants

- **StockExchange (Mediator interface):** Defines the contract for registering agents, placing bids and asks, and notifying agents of events.
- **NasdaqExchange (ConcreteMediator):** Maintains the agent registry, the order book, and the matching engine logic.
- **TradingAgent (Colleague interface):** Defines the contract that all trading participants must follow.
- **BuyAgent (Concrete Colleague):** Represents a buyer submitting purchase orders.
- **SellAgent (Concrete Colleague):** Represents a seller submitting sell orders.

### Complete Pseudocode

```
// ── Mediator Interface ──
// Defines the contract for all stock exchange mediators.
interface StockExchange {
    registerAgent(agent: TradingAgent) -> void
    placeBid(agent: TradingAgent, symbol: String, qty: Integer, price: Float) -> void
    placeAsk(agent: TradingAgent, symbol: String, qty: Integer, price: Float) -> void
    notify(agent: TradingAgent, event: TradeEvent) -> void
}


// ── Concrete Mediator ──
// The Nasdaq exchange implementation. Maintains an order book,
// matches bids and asks, and notifies agents of trade executions.
class NasdaqExchange implements StockExchange {
    private agents: Map

    // Order book: for each symbol, lists of open bids and asks.
    private bidBook: Map>
    private askBook: Map>
    private tradeLog: List

    constructor() {
        this.agents = new Map()
        this.bidBook = new Map>()
        this.askBook = new Map>()
        this.tradeLog = new List()
    }

    method registerAgent(agent: TradingAgent) -> void {
        agentName = agent.getName()
        this.agents.put(agentName, agent)
        print("Registered agent: " + agentName)
    }

    method placeBid(agent: TradingAgent, symbol: String, qty: Integer, price: Float) -> void {
        print(agent.getName() + " places BID: " + qty + " " + symbol + " @ " + price)

        // Check if any existing ask matches this bid
        if (this.askBook.containsKey(symbol)) {
            asks = this.askBook.get(symbol)
            for (ask in asks) {
                if (ask.price &lt;= price and ask.qty == qty) {
                    // Match found! Execute trade.
                    this.executeTrade(agent, ask.agent, symbol, qty, price)
                    this.askBook.get(symbol).remove(ask)
                    return
                }
            }
        }

        // No match — add bid to order book
        if (not this.bidBook.containsKey(symbol)) {
            this.bidBook.put(symbol, new List())
        }
        this.bidBook.get(symbol).add(
            new Order(agent, symbol, qty, price))
    }

    method placeAsk(agent: TradingAgent, symbol: String, qty: Integer, price: Float) -> void {
        print(agent.getName() + " places ASK: " + qty + " " + symbol + " @ " + price)

        // Check if any existing bid matches this ask
        if (this.bidBook.containsKey(symbol)) {
            bids = this.bidBook.get(symbol)
            for (bid in bids) {
                if (bid.price &gt;= price and bid.qty == qty) {
                    // Match found! Execute trade.
                    this.executeTrade(bid.agent, agent, symbol, qty, price)
                    this.bidBook.get(symbol).remove(bid)
                    return
                }
            }
        }

        // No match — add ask to order book
        if (not this.askBook.containsKey(symbol)) {
            this.askBook.put(symbol, new List())
        }
        this.askBook.get(symbol).add(
            new Order(agent, symbol, qty, price))
    }

    method notify(agent: TradingAgent, event: TradeEvent) -> void {
        agent.receive(event)
    }

    // Private: executes a matched trade and notifies both parties.
    private method executeTrade(
        buyer: TradingAgent, seller: TradingAgent,
        symbol: String, qty: Integer, price: Float
    ) -> void {
        trade = new Trade(buyer, seller, symbol, qty, price)
        this.tradeLog.add(trade)
        print("TRADE EXECUTED: " + qty + " " + symbol + " @ " + price)

        // Notify both parties through the mediator
        this.notify(buyer, new TradeEvent("BOUGHT", symbol, qty, price))
        this.notify(seller, new TradeEvent("SOLD", symbol, qty, price))
    }

    method getTradeHistory() -> List {
        return this.tradeLog
    }
}


// ── Colleague Interface ──
// All trading agents implement this interface.
interface TradingAgent {
    getName() -> String
    send(message: String) -> void
    receive(event: TradeEvent) -> void
}


// ── Concrete Colleague: BuyAgent ──
// Represents a buyer in the market.
class BuyAgent implements TradingAgent {
    private name: String
    private mediator: StockExchange
    private portfolio: Map // symbol -> qty held

    constructor(name: String, mediator: StockExchange) {
        this.name = name
        this.mediator = mediator
        this.portfolio = new Map()
        this.mediator.registerAgent(this)
    }

    method getName() -> String {
        return this.name
    }

    // Submit a bid to buy shares through the mediator.
    method buy(symbol: String, qty: Integer, maxPrice: Float) -> void {
        print(this.name + " wants to buy " + qty + " " + symbol)
        this.mediator.placeBid(this, symbol, qty, maxPrice)
    }

    method send(message: String) -> void {
        // The colleague sends messages through the mediator.
        // In a full implementation, this could be typed messages.
        print(this.name + " sends: " + message)
    }

    method receive(event: TradeEvent) -> void {
        // Update portfolio on receipt of a trade notification.
        if (event.type == "BOUGHT" or event.type == "SOLD") {
            current = this.portfolio.getOrDefault(event.symbol, 0)
            if (event.type == "BOUGHT") {
                this.portfolio.put(event.symbol, current + event.qty)
            } else {
                this.portfolio.put(event.symbol, current - event.qty)
            }
        }
        print(this.name + " received: " + event.toString())
    }

    method getHolding(symbol: String) -> Integer {
        return this.portfolio.getOrDefault(symbol, 0)
    }
}


// ── Concrete Colleague: SellAgent ──
// Represents a seller in the market.
class SellAgent implements TradingAgent {
    private name: String
    private mediator: StockExchange
    private inventory: Map // symbol -> qty available

    constructor(name: String, mediator: StockExchange) {
        this.name = name
        this.mediator = mediator
        this.inventory = new Map()
        this.mediator.registerAgent(this)
    }

    method getName() -> String {
        return this.name
    }

    // Add shares to available inventory.
    method addInventory(symbol: String, qty: Integer) -> void {
        current = this.inventory.getOrDefault(symbol, 0)
        this.inventory.put(symbol, current + qty)
    }

    // Submit an ask to sell shares through the mediator.
    method sell(symbol: String, qty: Integer, minPrice: Float) -> void {
        available = this.inventory.getOrDefault(symbol, 0)
        if (available &lt; qty) {
            print(this.name + " has insufficient " + symbol + " to sell")
            return
        }
        print(this.name + " wants to sell " + qty + " " + symbol)
        this.mediator.placeAsk(this, symbol, qty, minPrice)
    }

    method send(message: String) -> void {
        print(this.name + " sends: " + message)
    }

    method receive(event: TradeEvent) -> void {
        if (event.type == "SOLD") {
            current = this.inventory.getOrDefault(event.symbol, 0)
            this.inventory.put(event.symbol, current - event.qty)
        }
        print(this.name + " received: " + event.toString())
    }

    method getInventory(symbol: String) -> Integer {
        return this.inventory.getOrDefault(symbol, 0)
    }
}


// ── Supporting Types ──

class Order {
    public agent: TradingAgent
    public symbol: String
    public qty: Integer
    public price: Float

    constructor(agent: TradingAgent, symbol: String, qty: Integer, price: Float) {
        this.agent = agent
        this.symbol = symbol
        this.qty = qty
        this.price = price
    }
}

class Trade {
    public buyer: TradingAgent
    public seller: TradingAgent
    public symbol: String
    public qty: Integer
    public price: Float
    public timestamp: DateTime

    constructor(buyer: TradingAgent, seller: TradingAgent,
                symbol: String, qty: Integer, price: Float) {
        this.buyer = buyer
        this.seller = seller
        this.symbol = symbol
        this.qty = qty
        this.price = price
        this.timestamp = DateTime.now()
    }
}

class TradeEvent {
    public type: String    // "BOUGHT", "SOLD", "ORDER_MATCHED", "REJECTED"
    public symbol: String
    public qty: Integer
    public price: Float
    public message: String

    constructor(type: String, symbol: String, qty: Integer, price: Float) {
        this.type = type
        this.symbol = symbol
        this.qty = qty
        this.price = price
        this.message = type + " " + qty + " " + symbol + " at " + price
    }

    method toString() -> String {
        return this.message
    }
}


// ── Client / Driver Code ──
// Demonstrates the mediator in action.
function main() {
    // Create the mediator (the exchange)
    nasdaq = new NasdaqExchange()

    // Create trading agents (colleagues) and register with mediator
    buyer = new BuyAgent("AliceInstitutional", nasdaq)
    seller = new SellAgent("BobRetail", nasdaq)

    // Seller adds inventory
    seller.addInventory("AAPL", 200)
    seller.addInventory("GOOGL", 50)

    // Alice submits a bid — no match yet, goes to order book
    buyer.buy("AAPL", 100, 150.50)

    // Bob submits an ask that matches Alice's bid
    seller.sell("AAPL", 100, 149.00)

    // The mediator executes the trade and notifies both parties
    // Output:
    //   AliceInstitutional wants to buy 100 AAPL
    //   AliceInstitutional places BID: 100 AAPL @ 150.5
    //   BobRetail wants to sell 100 AAPL
    //   BobRetail places ASK: 100 AAPL @ 149.0
    //   TRADE EXECUTED: 100 AAPL @ 149.0
    //   AliceInstitutional received: BOUGHT 100 AAPL at 149.0
    //   BobRetail received: SOLD 100 AAPL at 149.0

    // Additional trades
    buyer.buy("GOOGL", 50, 185.00)
    seller.sell("GOOGL", 50, 183.50)

    // A non-matching order (price too low for existing asks)
    buyer.buy("MSFT", 10, 400.00)  // No seller, goes to order book
}
```

### Key Design Decisions in the Pseudocode

- **Single point of control:** All matching logic lives in `NasdaqExchange.executeTrade()`. If the matching algorithm needs to change (e.g., price-time priority instead of simple price matching), only the mediator changes. No colleague is affected.
- **Encapsulated order book:** The bid and ask books are private to the mediator. Colleagues have no visibility into the order book state. The mediator decides when to disclose information.
- **Event-driven notification:** The mediator publishes `TradeEvent` objects to colleagues. Colleagues react to events rather than polling for state changes. This is the Mediator pattern leaning on the Observer pattern internally.
- **Colleagues are lightweight:** BuyAgent and SellAgent have no knowledge of each other, no knowledge of the order book, and no matching logic. Their only job is to represent a trader and communicate through the mediator.

## How It Works — Step by Step

The following walkthrough traces a complete trading cycle from agent registration through trade execution, using the AAPL trade from the pseudocode above.

**Step 1: The NasdaqExchange Mediator Is Created**

The system initializes the stock exchange mediator. The mediator creates its internal data structures: an empty agent registry (a map from agent name to TradingAgent), an empty bid book (a map from symbol to list of bid orders), and an empty ask book (a map from symbol to list of ask orders). At this point, no agents are registered and no orders exist. The mediator is ready to accept registrations.

```
nasdaq = new NasdaqExchange()
// Internal state: agents={}, bidBook={}, askBook={}
```

**Step 2: AliceInstitutional (BuyAgent) Registers with the Mediator**

When `BuyAgent` is constructed, it calls `mediator.registerAgent(this)` inside its constructor. The mediator stores the agent in its agents map under the name "AliceInstitutional". The BuyAgent also stores a reference to the mediator in its own `mediator` field — this is the only external reference the agent will ever hold. The agent does not know about any other agent.

```
buyer = new BuyAgent("AliceInstitutional", nasdaq)
// mediator.agents: {"AliceInstitutional" -> buyer}
```

**Step 3: BobRetail (SellAgent) Registers with the Mediator and Adds Inventory**

BobRetail is constructed similarly, registering with the mediator. Then Bob calls `addInventory("AAPL", 200)` to record that he holds 200 shares of Apple. This inventory is local to Bob's agent — the mediator does not know about it. The mediator only knows about orders that have been placed, not about inventory that exists but is not yet listed.

```
seller = new SellAgent("BobRetail", nasdaq)
// mediator.agents: {"AliceInstitutional" -> buyer, "BobRetail" -> seller}
seller.addInventory("AAPL", 200)
// seller.inventory: {"AAPL" -> 200}
```

**Step 4: Alice Submits a Bid to Buy 100 AAPL at 150.50**

Alice calls `buyer.buy("AAPL", 100, 150.50)`. The BuyAgent delegates immediately to the mediator: `mediator.placeBid(this, "AAPL", 100, 150.50)`.

Inside the mediator, the `placeBid` method executes:

1. The mediator prints a log message: "AliceInstitutional places BID: 100 AAPL @ 150.5".
2. The mediator checks the ask book for AAPL. The ask book is empty — no one has placed an ask for AAPL yet.
3. Since no match is found, the mediator creates a new `Order(buyer, "AAPL", 100, 150.50)` and adds it to the bid book under the "AAPL" key.
4. The order sits in the order book, waiting for a matching ask.

```
// State after Step 4:
// bidBook: {"AAPL" -> [Order(Alice, AAPL, 100, 150.50)]}
// askBook: {}
```

**Step 5: Bob Submits an Ask to Sell 100 AAPL at 149.00**

Bob calls `seller.sell("AAPL", 100, 149.00)`. The SellAgent first checks its inventory — Bob holds 200 AAPL, which is enough to cover the 100-share sale. The agent then delegates to the mediator: `mediator.placeAsk(this, "AAPL", 100, 149.00)`.

Inside the mediator, the `placeAsk` method executes:

1. The mediator prints: "BobRetail places ASK: 100 AAPL @ 149.0".
2. The mediator checks the bid book for AAPL. There is a pending bid for 100 AAPL at 150.50 from Alice.
3. The mediator compares the prices. The rule is: a bid matches an ask if `ask.price &lt;= bid.price`. Bob's ask of 149.00 is less than or equal to Alice's bid of 150.50. The prices are compatible.
4. The mediator also checks that quantities match. Both are 100 shares. The trade is valid.

**Step 6: The Mediator Executes the Trade and Notifies Both Agents**

With a confirmed match, the mediator calls its private `executeTrade` method:

1. The mediator creates a `Trade` object recording the buyer (Alice), seller (Bob), symbol (AAPL), quantity (100), and price (149.00). The trade price is Bob's ask price (the lower of the two), which is standard market practice.
2. The mediator adds the trade to its internal `tradeLog`.
3. The mediator prints: "TRADE EXECUTED: 100 AAPL @ 149.0".
4. The mediator creates two `TradeEvent` objects. One with type "BOUGHT" for Alice, one with type "SOLD" for Bob.
5. The mediator calls `this.notify(buyer, boughtEvent)` which calls `buyer.receive(boughtEvent)`. Alice's agent updates its portfolio: AAPL holdings go from 0 to 100. Alice prints: "AliceInstitutional received: BOUGHT 100 AAPL at 149.0".
6. The mediator calls `this.notify(seller, soldEvent)` which calls `seller.receive(soldEvent)`. Bob's agent updates its inventory: AAPL holdings go from 200 to 100. Bob prints: "BobRetail received: SOLD 100 AAPL at 149.0".
7. The mediator removes Bob's matched ask from the ask book (it was fully consumed by the trade).

```
// Final state after trade:
// agents: {"AliceInstitutional" -> buyer, "BobRetail" -> seller}
// bidBook: {"AAPL" -> []}  // (empty — order was consumed)
// askBook: {}               // (empty — order was consumed)
// tradeLog: [Trade(Alice, Bob, AAPL, 100, 149.00)]
```

**Step 7: The Mediator Is Ready for More Orders**

The mediator continues accepting bids and asks. If Alice places another bid for a different symbol (e.g., GOOGL at 185.00), the process repeats. If no matching ask exists, the bid sits in the order book until a matching ask arrives. The order book is stateful — orders persist across trading sessions (in real exchanges, orders typically expire at end of day but can be day orders or good-til-canceled).

The same mediator can handle any number of agents and any number of symbols. The agent registry, bid book, and ask book all scale independently. Adding a new trading agent requires one line of code: `new BuyAgent("CharlieHedgeFund", nasdaq)`. No existing code changes. That is the power of the Mediator pattern.

## When to Use

Use the Mediator pattern when:

**Communication between objects is complex and creates a tangled web of dependencies.** In the stock market example, every trading agent would need to know about every other agent, the compliance system, the risk checker, the market data feed, and the portfolio manager. The mediator centralizes all of this into a single exchange object. If your system has objects that reference many other objects to route messages, coordinate actions, or share state, a mediator can simplify the architecture dramatically.

**You want to reuse individual objects independently.** A BuyAgent that communicates through a mediator can be reused in any exchange implementation that implements the StockExchange interface. No agent knows about other agents, so you can drop a BuyAgent into a new system without bringing along a web of dependencies. This reusability is the direct result of decoupling through the mediator.

**The interaction protocol between objects is complex and likely to change.** Trading rules change. Regulators impose new requirements. Matching algorithms are optimized. When all interaction logic lives in the mediator, changing the protocol (e.g., switching from price-time priority to pro-rata matching) changes only the mediator. The colleagues remain untouched. This localization of change is the primary practical benefit of the Mediator pattern.

**A set of objects communicate in well-defined but many-to-many ways.** The stock market is a textbook many-to-many communication problem: any buyer can trade with any seller, for any symbol, at any time. The mediator is the natural solution because it reduces the many-to-many graph to a star topology. Other examples include chat rooms (any user can message any other user) and airport control (any plane can communicate with any other plane about landing slots).

**You want to centralize control or add cross-cutting behavior like logging, auditing, or validation.** With the mediator, you add logging to one place (the mediator's `placeBid`, `placeAsk`, and `executeTrade` methods) rather than to every agent. Compliance checks, price circuit breakers, and trade limits are enforced in the mediator, guaranteeing they apply to every trade. If compliance rules change, only the mediator is updated — no agent changes are needed.

**The system needs a well-defined protocol for adding new participants at runtime.** The mediator's `registerAgent` method allows new trading agents to join the market dynamically. A hedge fund can connect at 9:30 AM and be trading by 9:30:01. Without a mediator, adding a new participant would require updating every existing participant's list of peers. The mediator provides a single registration point that integrates new participants instantly.

**Testing individual components is important.** Testing a BuyAgent in isolation is straightforward with the mediator pattern. You create a mock implementation of StockExchange, pass it to the BuyAgent constructor, and verify that the agent calls the expected mediator methods with the correct parameters. You do not need to instantiate SellAgents, MarketMakers, or any other colleague. The mediator is mocked, and the agent is tested in isolation. This is a significant improvement over the tightly coupled alternative where unit testing an agent requires the full object graph.

## When NOT to Use

Avoid the Mediator pattern when:

**You have a simple one-to-one or one-to-few communication pattern.** If object A talks only to object B and there are no plans to add C or D, a direct reference is simpler. The mediator would add unnecessary indirection. For example, if a single trading terminal communicates only with a single order management system, a direct API call is adequate. The mediator only pays off when the communication network has at least three participants that interact with each other.

**The mediator becomes a god object.** As the system grows, there is a risk that the mediator accumulates too much responsibility. The NasdaqExchange might start with order matching, then add compliance checks, then risk management, then market data distribution, then position keeping, then reporting. At some point, the mediator becomes a monolithic god object that knows everything and does everything. If the mediator grows beyond a manageable size, split it into multiple mediators, each responsible for a specific concern. For example, a TradingExchange for order matching, a ComplianceMediator for regulatory checks, and a MarketDataMediator for price distribution.

**Performance is critical and the mediator adds latency.** Every communication between colleagues must pass through the mediator. In a high-frequency trading system where microseconds matter, the extra hop through the mediator might be unacceptable. Direct point-to-point communication eliminates this overhead. However, for most systems (including retail trading platforms and institutional trading desks), the mediator's overhead is negligible compared to network latency and database access.

**Objects already communicate through a simpler mechanism like an event bus or publish-subscribe.** If your system already uses an event bus where objects publish events and subscribe to events without knowing about each other, adding a mediator on top would be redundant. The event bus IS a mediator — just with a different name and usually a more generic interface. Use whatever mechanism is already in place. The Mediator pattern codifies this approach with a specific interface and explicit colleagues, but the underlying concept is the same.

**The interaction protocol is fixed and will never change.** If the system is simple, stable, and not expected to evolve, the additional abstraction of a mediator adds complexity without benefit. The stock market scenario is inherently dynamic, but a simple single-purpose system might not need the pattern. As with all design patterns, apply the Mediator pattern where its benefits (decoupling, change localization, testability) are needed — not as a default for every system.

**Colleagues need to communicate asynchronously at high volume.** The Mediator pattern, in its classic form, involves synchronous method calls from the mediator to colleagues. For high-throughput asynchronous communication (thousands of orders per second), consider using a message queue or event streaming platform (Kafka, RabbitMQ) instead. These platforms act as mediators but handle backpressure, persistence, replay, and distributed processing. The classic Mediator pattern is best suited for in-process coordination with moderate throughput.

**You can use composition to delegate communication.** If colleague A has a reference to colleague B and delegates specific calls to it, that is composition, not a many-to-many web. Composition is a simpler and more direct mechanism than a mediator. Only introduce the mediator when you have multiple colleagues that all need to communicate with each other, creating a cross-cutting interaction pattern that composition cannot cleanly represent.

## Real-World Occurrences

The Mediator pattern appears in many software systems and real-world environments. Here are the most notable examples.

### 1. MVC Framework (Controller as Mediator)

The Model-View-Controller architecture is the most widely cited example of the Mediator pattern in software engineering. The Controller acts as the mediator between the Model (data and business logic) and the View (user interface). When the user interacts with the View, the View delegates to the Controller. The Controller updates the Model. The Model notifies the Controller of changes. The Controller updates the View.

The key insight is that the Model and View never communicate directly. The Model does not know which Views are displaying its data. The View does not know how to update the Model. The Controller mediates all interactions. This decoupling allows any number of Views to display the same Model data, and Views can be swapped without affecting the Model.

In web frameworks like Spring MVC, the Controller receives HTTP requests, calls service methods (which update the Model), and selects which View to render. The Model (often a POJO) has no reference to the View. The View (a JSP or template) reads from the Model but never writes to it directly. The Controller is the mediator.

### 2. Air Traffic Control (Tower as Mediator)

Air traffic control is the canonical real-world analogy for the Mediator pattern. Each airplane communicates with the control tower, and the tower communicates with each airplane. Planes do not communicate directly with each other — they do not need to know each other's altitude, speed, or heading. The tower has the complete picture and coordinates all movements.

A plane approaching an airport calls the tower, reports its position, and requests landing instructions. The tower checks the positions of all other planes in the area, calculates a safe landing sequence, and issues instructions to each plane. The plane does not need to know about the other planes. The tower (mediator) holds all the state and enforces the interaction rules.

This is structurally identical to the stock exchange mediator. The exchange (tower) coordinates trading agents (planes) without any agent knowing about any other agent. The exchange maintains the order book (airspace map) and enforces trading rules (flight safety rules). Adding a new trading agent (landing plane) requires registering with the exchange (tower), not updating every existing agent (plane).

### 3. Chat Room Servers

A chat room server is a mediator between chat participants. When a user sends a message, it goes to the server. The server determines which users should receive the message (all users in the room, or a specific recipient in a private chat) and forwards the message to each recipient's client.

Without the server (mediator), each chat client would need to know the network addresses of every other client and would need to send messages directly to each one. This is impractical for several reasons: clients behind NAT or firewalls may not be reachable, clients that go offline would disrupt message delivery, and group chats would require each client to fan-out messages to N-1 other clients. The server solves all of these problems by acting as the mediator.

Modern chat systems often use a publish-subscribe pattern internally, which is a close relative of the Mediator pattern. The server maintains topics (channels or rooms), clients subscribe to topics, and messages published to a topic are distributed to all subscribers. The pub-sub broker is the mediator.

### 4. Java's `java.util.concurrent.ExecutorService`

The `ExecutorService` in Java's standard library is a mediator for thread management. Instead of creating and managing threads directly, worker tasks submit themselves to the executor. The executor mediates between the tasks and the thread pool, deciding which task runs on which thread, how many threads are active, and what happens when tasks complete or fail.

A task that submits itself to an `ExecutorService` does not know about any other task. It does not know how many threads are in the pool, what other tasks are queued, or how the executor schedules work. The executor manages all of this internally. The task only knows its own logic and the executor interface.

This is the Mediator pattern applied to concurrency. The executor mediates the interaction between tasks and threads, just as the stock exchange mediates the interaction between buyers and sellers. The colleagues (tasks) are decoupled from each other and from the underlying infrastructure (threads), and the mediator (executor) provides a single point of control for scheduling, lifecycle management, and resource allocation.

### 5. Message Brokers (Kafka, RabbitMQ)

Enterprise message brokers like Apache Kafka and RabbitMQ are distributed mediators. Producers send messages to the broker on specific topics or queues. Consumers subscribe to topics or queues and receive messages from the broker. Producers do not know about consumers, and consumers do not know about producers.

In the stock market analogy, Kafka topics are like stock symbols. A producer publishing to the "AAPL" topic is analogous to a BuyAgent placing a bid through the exchange. A consumer subscribed to the "AAPL" topic receives all trades for AAPL, just as a MarketMakerAgent receives trade notifications from the NasdaqExchange.

The broker provides durability (messages persist on disk), ordering guarantees (messages on a partition are ordered), and scalability (partitions can be distributed across servers). These features go beyond the classic mediator pattern but demonstrate the same core principle: decoupling communicating parties through a central coordinator.

### 6. GUI Component Coordination (Mediator in UI Frameworks)

In complex user interfaces, multiple components often need to coordinate without direct coupling. A date picker, a time picker, and a calendar view all need to remain synchronized. When the user selects a date in the calendar, the date picker must update and the time picker must adjust available time slots.

A UI mediator manages this coordination. The mediator registers all relevant UI components, listens for events from each one, and updates the others as needed. The components never hold references to each other. They only know the mediator. This is the standard approach in complex UI frameworks and is often called the "mediator pattern" in UI development texts.

### 7. Ethereum Smart Contracts (Mediator for Decentralized Exchanges)

Decentralized finance (DeFi) applications on Ethereum use smart contracts as mediators. A decentralized exchange (DEX) smart contract mediates between liquidity providers and traders. Liquidity providers deposit tokens into the contract (like adding shares to an inventory). Traders submit swap requests to the contract (like placing orders). The contract matches trades, enforces prices through an automated market maker formula, and distributes tokens.

The smart contract is a transparent, immutable mediator. No trader needs to know about any other trader. No liquidity provider needs to negotiate terms with individual traders. The contract mediates all interactions according to its programmed rules, which are visible on the blockchain for anyone to audit.

## Comparison with Related Patterns

The Mediator pattern is often confused with patterns that share similar structural features. The following table clarifies the distinctions.

| Pattern | Purpose | Key Difference from Mediator |
|---|---|---|
| **Mediator** | Encapsulate how a set of objects interact, promoting loose coupling by keeping objects from referring to each other explicitly | Focuses on centralizing multi-way communication; colleagues are aware of the mediator but not of each other |
| **Observer** | Define a one-to-many dependency so that when one object changes state, all its dependents are notified | One subject pushes to many observers; observers do not communicate back through the subject. Mediator is bidirectional — any colleague can send and receive |
| **Facade** | Provide a unified interface to a set of interfaces in a subsystem | Facade simplifies a complex subsystem for a client; it is one-directional. Mediator manages bidirectional communication between multiple peers |
| **Command** | Encapsulate a request as an object, allowing parameterization, queuing, and undo | Command focuses on request encapsulation and decoupling sender from receiver. Mediator focuses on routing and coordinating between multiple objects |
| **Chain of Responsibility** | Pass a request along a chain of handlers until one handles it | Chain delivers a request to exactly one handler in sequence. Mediator can route to any subset of colleagues, possibly broadcasting to all |
| **Event Bus / Pub-Sub** | Decouple publishers from subscribers through an event channel | An event bus is a generic infrastructure component. Mediator is a domain-specific object with business logic. The event bus is the mechanism; the mediator is the policy |

### Mediator vs. Observer

These two patterns are closely related and are often used together. The Mediator pattern frequently uses Observer internally: when the mediator needs to notify colleagues of events, it calls a method on each colleague (which is the Observer pattern's notification mechanism — the mediator is the subject and the colleagues are the observers).

The key distinction is intent and structure:

- **Observer** defines a one-to-many relationship. One subject notifies many observers. The communication is unidirectional from subject to observer. The subject typically does not receive messages back from observers.
- **Mediator** defines a many-to-many relationship. Any colleague can communicate with any other colleague, but only through the mediator. The communication is bidirectional.

In the stock market example, the NasdaqExchange uses the Observer pattern when it calls `agent.receive(event)` on each trading agent. But the overall architecture is Mediator because agents also call the exchange to place orders. The communication flows in both directions.

### Mediator vs. Facade

The distinction between Mediator and Facade is one of direction and purpose:

- **Facade** provides a simplified, unified interface to a complex subsystem. The client calls the Facade, which delegates to subsystem components. The subsystem components do not call back to the Facade. Communication is one-directional from client through Facade to subsystem.
- **Mediator** enables multi-directional communication between peers. Colleagues call the mediator, and the mediator calls colleagues. Communication is bidirectional and collaborative.

A Facade for a trading system might wrap order placement, portfolio queries, and market data retrieval into a simple API. Clients of the Facade do not participate in the trading process — they just consume data and issue commands. A Mediator for the same system would involve the agents actively participating in the trading process, sending and receiving messages through the mediator as the market operates.

### Mediator vs. Command

The Command pattern is relevant to the Mediator discussion because mediators often execute commands:

- **Command** encapsulates an action as an object. The sender creates a command and passes it to an invoker, which executes it against a receiver. Command decouples the sender from the receiver but does not coordinate multiple objects.
- **Mediator** coordinates multiple objects and routes messages. A mediator might receive a Command object (like `PlaceOrderCommand` or `CancelOrderCommand`), execute it, and then notify other colleagues of the result.

In the stock exchange, a BuyAgent could send a `BidCommand` to the mediator. The mediator receives the command, processes it against the order book, and broadcasts `TradeExecutedEvent` to the affected colleagues. The Command pattern handles the request encapsulation; the Mediator pattern handles the routing and coordination.

### When Patterns Combine

In practice, these patterns are often combined:

- A **Mediator** uses **Observer** internally to notify colleagues of events.
- A **Mediator** processes **Command** objects submitted by colleagues.
- A **Facade** might be layered over a subsystem that internally uses **Mediator** for component coordination.
- An **Event Bus** acts as a generic, reusable mediator for publish-subscribe communication.

The Mediator pattern is not mutually exclusive with these patterns. It addresses a specific problem — coordinating many-to-many communication between objects — and coexists with other patterns that address different aspects of the same system.

## Summary

The Mediator pattern provides a clean solution to the problem of many-to-many communication between objects. By introducing a central coordinator that all objects communicate through, the pattern eliminates direct dependencies between collaborating objects and centralizes interaction logic in a single place.

In the stock market domain, the Mediator pattern maps naturally to the role of a real stock exchange. The exchange is the mediator. Trading agents (buyers, sellers, market makers) are the colleagues. Every order flows through the exchange, the exchange matches orders, and the exchange notifies agents of trade executions. No agent needs to know about any other agent.

The key benefits of the Mediator pattern are:

- **Decoupling:** Colleagues hold only a reference to the mediator, not to each other. This eliminates the tangled web of direct references that makes systems hard to maintain.
- **Centralized control:** Interaction logic (matching rules, compliance checks, auditing) lives in one place. Changing the rules changes only the mediator.
- **Extensibility:** Adding a new colleague type requires only implementing the colleague interface and registering with the mediator. No existing code changes.
- **Reusability:** Colleagues can be reused in different contexts by connecting them to different mediator implementations.
- **Testability:** Colleagues are tested in isolation with a mock mediator. The mediator is tested with mock colleagues.

The main risk of the Mediator pattern is that the mediator can become a god object — a monolithic class that knows everything and does everything. Mitigate this risk by keeping the mediator focused on coordination and routing, and by splitting complex systems into multiple specialized mediators.

The Mediator pattern is most valuable in systems where:
- Many objects communicate in complex, multi-directional patterns.
- The interaction protocol is likely to change.
- New types of participants are added regularly.
- Individual components must be independently testable.
- Centralized control over communication is desirable for auditing, logging, or compliance.

When applied appropriately, the Mediator pattern transforms a chaotic web of direct object references into a clean, maintainable star topology. The stock market is the perfect domain to illustrate this pattern because the parallel to real-world exchanges makes the pattern's structure and benefits immediately intuitive. Every time you buy or sell a stock through a brokerage, you are relying on a mediator — the exchange — that has been operating successfully for centuries.

## Checklist

When implementing the Mediator pattern, verify each point against your design:

- [ ] The Mediator interface defines clear methods for registration, message sending, and notification
- [ ] Colleagues hold only a reference to the mediator — no direct references to other colleagues
- [ ] Colleagues are registered with the mediator before any communication occurs
- [ ] All communication between colleagues passes through the mediator (no back channels)
- [ ] The mediator encapsulates all interaction logic (matching, routing, broadcasting)
- [ ] The mediator does not leak internal state (e.g., order book contents) to colleagues unless explicitly intended
- [ ] Adding a new colleague type requires no changes to existing colleagues
- [ ] Adding a new mediator behavior requires no changes to colleagues
- [ ] The mediator is testable with mock colleagues
- [ ] Each colleague is testable with a mock mediator
- [ ] The mediator has not grown into a god object — consider splitting if it has too many responsibilities
- [ ] Performance overhead of the mediator routing layer is acceptable for the use case
- [ ] The mediator correctly handles edge cases: unmatched orders, partial fills, order cancellations, concurrent submissions
- [ ] Error handling in the mediator does not propagate failures between unrelated colleagues
