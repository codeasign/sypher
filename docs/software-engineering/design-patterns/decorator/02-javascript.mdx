## Implementation

Below is a complete JavaScript implementation of the Decorator pattern using a car dealership domain. The `Car` interface defines the contract for description and cost. `BaseCar` is the concrete car without options. `CarFeature` is the abstract decorator that wraps a `Car` and delegates to it. `LeatherSeats`, `Sunroof`, `SportPackage`, and `NavigationSystem` are concrete decorators that add their own cost and description. The driver demonstrates stacking decorators to create different car configurations at runtime.

### How the Implementation Works

#### Class Diagram (Pseudocode)

#### OOP Concepts Applied

1. **Abstraction** — The `Car` base class defines the contract (`getDescription`, `getCost`) without specifying how they are computed. Clients interact with any decorated car through this abstract interface.

2. **Polymorphism** — `BaseCar` and all decorated variants are treated uniformly as `Car` objects. The driver calls `getDescription()` and `getCost()` on any instance without knowing the concrete type.

3. **Composition over Inheritance** — Instead of creating subclasses for every feature combination (e.g., `CamryWithLeatherAndSunroof`), decorators hold a reference to a `Car` (`has-a` relationship) and delegate to it. This makes feature mixing a runtime composition, not a compile-time hierarchy.

#### SOLID Principles Applied

4. **SRP** — `BaseCar` owns only the base model identity and price. Each decorator is responsible for exactly one feature's cost and description. No class knows about feature combinations beyond its own contribution.

5. **OCP** — New features are added by creating new decorator classes without modifying existing decorators or `BaseCar`. The system is open for extension and closed for modification.

6. **LSP** — Every decorator implements the `Car` interface and preserves its contract. A decorated car is substitutable for a plain `BaseCar` wherever a `Car` is expected.

7. **ISP** — The `Car` interface has only two methods (`getDescription`, `getCost`), both of which every client needs. No client is forced to depend on methods it does not use.

8. **DIP** — High-level client code (the driver) depends on the abstract `Car` interface, not on concrete `BaseCar` or decorator classes directly (beyond construction).

### Component (Base Class)

```javascript
class Car {
  getDescription() {
    throw new Error("getDescription() must be overridden by subclass");
  }

  getCost() {
    throw new Error("getCost() must be overridden by subclass");
  }
}
```

### ConcreteComponent

```javascript
class BaseCar extends Car {
  #model;
  #baseCost;

  constructor(model, baseCost) {
    super();
    this.#model = model;
    this.#baseCost = baseCost;
  }

  getDescription() {
    return this.#model;
  }

  getCost() {
    return this.#baseCost;
  }
}
```

### Decorator (Abstract)

```javascript
class CarFeature extends Car {
  #car;

  constructor(car) {
    super();
    if (!(car instanceof Car)) {
      throw new Error("CarFeature requires a Car instance");
    }
    this.#car = car;
  }

  getDescription() {
    return this.#car.getDescription();
  }

  getCost() {
    return this.#car.getCost();
  }
}
```

### ConcreteDecorator 1: LeatherSeats

```javascript
class LeatherSeats extends CarFeature {
  static #FEATURE_COST = 1500.00;

  constructor(car) {
    super(car);
  }

  getDescription() {
    return super.getDescription() + " + Leather Seats";
  }

  getCost() {
    return super.getCost() + LeatherSeats.#FEATURE_COST;
```

### Main / Driver

```javascript
function main() {
  console.log("=== Car Dealership Configuration System ===\n");

  // Create a base car
  console.log("--- Base Model ---");
  const baseCamry = new BaseCar("Toyota Camry", 28000.00);
  console.log(`  ${baseCamry.getDescription()}`);
  console.log(`  Base price: $${baseCamry.getCost().toFixed(2)}`);

  // Single feature: Leather Seats
  console.log("\n--- With Leather Seats ---");
  const withLeather = new LeatherSeats(baseCamry);
  console.log(`  ${withLeather.getDescription()}`);
  console.log(`  Total: $${withLeather.getCost().toFixed(2)}`);

  // Two features: Leather Seats + Sunroof
  console.log("\n--- With Leather Seats + Sunroof ---");
  const withLeatherAndSunroof = new Sunroof(new LeatherSeats(baseCamry));
  console.log(`  ${withLeatherAndSunroof.getDescription()}`);
  console.log(`  Total: $${withLeatherAndSunroof.getCost().toFixed(2)}`);

  // Three features: Leather Seats + Sunroof + Navigation
  console.log("\n--- With Leather Seats + Sunroof + Navigation ---");
  const withThreeFeatures = new NavigationSystem(
    new Sunroof(new LeatherSeats(baseCamry))
  );
  console.log(`  ${withThreeFeatures.getDescription()}`);
  console.log(`  Total: $${withThreeFeatures.getCost().toFixed(2)}`);

  // Fully loaded: all four features
  console.log("\n--- Fully Loaded ---");
  const fullyLoaded = new SportPackage(
    new NavigationSystem(
      new Sunroof(new LeatherSeats(baseCamry))
    )
  );
  console.log(`  ${fullyLoaded.getDescription()}`);
  console.log(`  Total: $${fullyLoaded.getCost().toFixed(2)}`);

  // Alternative configuration: Sport Package only
  console.log("\n--- Base Camry + Sport Package Only ---");
  const sportCamry = new SportPackage(baseCamry);
  console.log(`  ${sportCamry.getDescription()}`);
  console.log(`  Total: $${sportCamry.getCost().toFixed(2)}`);

  // Luxury configuration: Leather + Navigation only
  console.log("\n--- Luxury Config (Leather + Navigation) ---");
  const luxuryCamry = new NavigationSystem(new LeatherSeats(baseCamry));
  console.log(`  ${luxuryCamry.getDescription()}`);
  console.log(`  Total: $${luxuryCamry.getCost().toFixed(2)}`);

  // Verify interface compliance
  console.log("\n--- Type Check (Interface Preservation) ---");
  const configs = [baseCamry, withLeather, withLeatherAndSunroof, withThreeFeatures, fullyLoaded, sportCamry, luxuryCamry];
  configs.forEach((c, i) => {
    console.log(`  Config ${i + 1} is a Car: ${c instanceof Car}`);
  });
  console.log(`  All configurations implement the Car interface: ${configs.every(c => c instanceof Car)}`);

  // Summary table
  console.log("\n=== Configuration Summary ===");
  console.log("  Model                                | Features                                      | Price");
  console.log("  -------------------------------------|-----------------------------------------------|--------");
  configs.forEach(c => {
    const desc = c.getDescription().padEnd(37);
    const cost = `$${c.getCost().toFixed(2)}`;
    console.log(`  ${desc}| ${cost}`);
  });
}

main();
```

### Language-Specific Notes

- **No abstract class enforcement** — JavaScript does not have native abstract class support. The `Car` base class simulates abstract methods by throwing `Error` in the base implementation. If a subclass fails to override `getDescription()` or `getCost()`, the error is thrown at runtime rather than being caught by the compiler.

- **Private class fields (`#`)** — The implementation uses JavaScript's native private field syntax (`#model`, `#baseCost`, `#car`, `#FEATURE_COST`). These are truly private at the language level and cannot be accessed from outside the class even through reflection. Older JavaScript environments may require a transpiler such as Babel.

- **Static private fields for constants** — Feature costs are stored as `static #FEATURE_COST` on each decorator class. This prevents external modification and keeps the per-feature cost encapsulated within its decorator class. Each decorator accesses its own cost via the class name (e.g., `LeatherSeats.#FEATURE_COST`).

- **Type checking with `instanceof`** — The `CarFeature` constructor validates that the wrapped object is a `Car` instance using `car instanceof Car`. This catches programmer errors at construction time rather than causing cryptic failures later. The driver demonstrates this with a runtime type check on all configurations.

- **No interface keyword** — JavaScript has no `interface` keyword. The pattern is implemented with class inheritance instead of interface contracts. `Car` is a base class that defines the shape through method signatures, and all decorators extend it via the prototype chain.
