## Implementation

Below is a complete Java implementation of the Decorator pattern using a car dealership domain. Java's interface and abstract class features provide clean separation between the `Car` contract (interface), the `BaseCar` concrete component, the `CarFeature` abstract decorator, and concrete decorators. Java's type system enforces the pattern at compile time, and the `final` keyword prevents accidental mutation of feature costs and the wrapped reference.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────────────┐
//   │            <<interface>>                      │
//   │                 Car                           │  ← OOP: Abstraction
//   ├──────────────────────────────────────────────┤     ISP: 2 methods
//   │  + String getDescription()                    │

//        ┌─────┴──────┐    ┌──────┴──────────┐

//        │ implements │    │   <<abstract>>   │     DIP: depends on
//        │   Car      │    │   implements Car │     Car interface
//        └────────────┘    └──────┬───────────┘
//                                 │ extends
//            ┌────────────────────┼────────────────────┐
//            │                    │                    │

//   │  LeatherSeats  │  │   Sunroof    │  │  SportPackage    │
```

#### OOP Concepts Applied

1. **Abstraction** — The `Car` interface defines the contract without implementation details. Both `BaseCar` and all decorators implement this interface. Client code depends on `Car`, not on concrete types.

2. **Polymorphism** — A collection of `Car` references can hold `BaseCar`, `LeatherSeats`, `Sunroof`, or any decorated variant. The JVM dispatches `getDescription()` and `getCost()` calls dynamically based on the actual object type.

3. **Composition over Inheritance** — Features are combined by object composition at runtime, avoiding the combinatorial explosion of creating a subclass for every feature combination.

#### SOLID Principles Applied

4. **SRP** — `BaseCar` manages only the base vehicle identity. Each decorator manages exactly one feature. The driver's `main` method handles only configuration assembly and output.

5. **OCP** — A new feature (e.g., `HeatedSteeringWheel`) is added as a new decorator class extending `CarFeature`. No existing code is modified.

6. **LSP** — Decorators preserve the `Car` contract. A `SportPackage` wrapping a `BaseCar` can be used anywhere a plain `Car` is expected.

7. **ISP** — The `Car` interface is minimal with two methods. No client is forced to depend on unused methods.

8. **DIP** — All modules depend on the `Car` interface. `BaseCar` and decorators implement `Car`. High-level code in the driver interacts with `Car` references.

### Component interface

```java
interface Car {
    String getDescription();
    double getCost();
}
```

### ConcreteComponent

```java
class BaseCar implements Car {
    private final String model;
    private final double baseCost;

    public BaseCar(String model, double baseCost) {
        this.model = model;
        this.baseCost = baseCost;
    }

    @Override
    public String getDescription() {
        return model;
    }

    @Override
    public double getCost() {
        return baseCost;
    }
}
```

### Decorator (Abstract)

```java
abstract class CarFeature implements Car {
    protected Car car;

    public CarFeature(Car car) {
        this.car = car;
    }

    @Override
    public String getDescription() {
        return car.getDescription();
    }

    @Override
    public double getCost() {
        return car.getCost();
    }
}
```

### ConcreteDecorator 1: LeatherSeats

```java
class LeatherSeats extends CarFeature {
    private static final double FEATURE_COST = 1500.00;
    private static final String FEATURE_NAME = "Leather Seats";

    public LeatherSeats(Car car) {
        super(car);
    }

    @Override
    public String getDescription() {
        return super.getDescription() + " + " + FEATURE_NAME;
    }

    @Override
    public double getCost() {
        return super.getCost() + FEATURE_COST;
    }
}
```

### Main / Driver

```java
public class Main {
    private static final DecimalFormat CURRENCY = new DecimalFormat("$#,##0.00");

    public static void main(String[] args) {
        System.out.println("=== Java Car Dealership Configuration System ===");
        System.out.println();

        // Build configurations
        Car baseCamry = new BaseCar("Toyota Camry", 28000.00);
        Car withLeather = new LeatherSeats(baseCamry);
        Car withSunroof = new Sunroof(baseCamry);
        Car withLeatherSunroof = new Sunroof(new LeatherSeats(baseCamry));
        Car ecoLuxury = new NavigationSystem(new LeatherSeats(baseCamry));
        Car sportOnly = new SportPackage(baseCamry);
        Car fullyLoaded = new SportPackage(
            new NavigationSystem(
                new Sunroof(
                    new LeatherSeats(baseCamry)
                )
            )
        );

        // Store as a list of named configurations
        LinkedHashMap<String, Car> configs = new LinkedHashMap<>();
        configs.put("Base Model", baseCamry);
        configs.put("With Leather Seats", withLeather);
        configs.put("With Sunroof", withSunroof);
        configs.put("Leather + Sunroof", withLeatherSunroof);
        configs.put("Eco Luxury", ecoLuxury);
        configs.put("Sport Only", sportOnly);
        configs.put("Fully Loaded", fullyLoaded);

        // Print header
        System.out.printf("%-20s %-65s %s%n", "Configuration", "Description", "Price");
        System.out.println("------------------------------------------------------------------------------------------");

        // Print each configuration
        for (Map.Entry<String, Car> entry : configs.entrySet()) {
            Car c = entry.getValue();
            System.out.printf("  %-18s %-65s %s%n",
                entry.getKey(),
                c.getDescription(),
                CURRENCY.format(c.getCost()));
        }

        // Price breakdown for fully loaded
        System.out.println();
        System.out.println("--- Price Breakdown: Fully Loaded Camry ---");
        System.out.printf("  Base car:                       %10s%n", CURRENCY.format(baseCamry.getCost()));
        System.out.printf("  + Leather Seats:                %10s%n", CURRENCY.format(1500.00));
        System.out.printf("  + Sunroof:                      %10s%n", CURRENCY.format(1200.00));
        System.out.printf("  + Navigation System:            %10s%n", CURRENCY.format(2000.00));
        System.out.printf("  + Sport Package:                %10s%n", CURRENCY.format(3500.00));
        System.out.println("  -----------------------------------------");
        System.out.printf("  Total:                          %10s%n", CURRENCY.format(fullyLoaded.getCost()));

        // Interface compliance
        System.out.println();
        System.out.println("--- Interface Compliance ---");
        int i = 1;
        for (Car c : configs.values()) {
            System.out.printf("  Config %d: Car interface = %s%n", i++, c instanceof Car);
        }

        // Reuse with Honda Accord
        System.out.println();
        System.out.println("--- Honda Accord Configuration ---");
        Car baseAccord = new BaseCar("Honda Accord", 27000.00);
        Car accordLuxury = new NavigationSystem(new LeatherSeats(baseAccord));
        System.out.println("  " + accordLuxury.getDescription());
        System.out.println("  Total: " + CURRENCY.format(accordLuxury.getCost()));

        // Demonstrate decorator order independence
        System.out.println();
        System.out.println("--- Order Independence (additive features) ---");
        Car order1 = new Sunroof(new LeatherSeats(baseCamry));
        Car order2 = new LeatherSeats(new Sunroof(baseCamry));
        System.out.printf("  Sunroof(Leather(Camry)): %s — %s%n",
            order1.getDescription(), CURRENCY.format(order1.getCost()));
        System.out.printf("  Leather(Sunroof(Camry)): %s — %s%n",
            order2.getDescription(), CURRENCY.format(order2.getCost()));
        System.out.printf("  Same price: %b%n",
            Math.abs(order1.getCost() - order2.getCost()) < 0.001);
    }
}
```

### Language-Specific Notes

- **Interface vs abstract class** — Java separates the Component (`Car` interface) and Decorator (`CarFeature` abstract class) cleanly. The interface defines the contract; the abstract class provides the delegation boilerplate. Concrete decorators extend the abstract class, inheriting the `protected Car car` field and default delegation. This separation prevents the diamond problem that would arise if `CarFeature` were an interface.

- **`protected` access modifier** — The `car` field in `CarFeature` is `protected`, allowing subclasses to access it if needed. This is standard Java practice for abstract decorator base classes. The field is not `public`, so external code cannot modify the wrapped car reference.

- **`static final` constants** — Feature names and costs are declared `private static final` on each decorator. These are compile-time constants stored in the method area (not per instance). The `final` keyword ensures they cannot be reassigned.

- **`DecimalFormat` for currency** — The driver uses `java.text.DecimalFormat` with the pattern `$#,##0.00` for currency output. This handles locale-aware formatting and ensures consistent two-decimal-place output across all configurations.

- **`@Override` annotations** — Every method that overrides an interface or superclass method is annotated with `@Override`. This enables compile-time checking that the method signature matches a parent method, preventing subtle bugs from misspelled method names.

- **No reflection/annotation processing** — Unlike pattern implementations that use runtime reflection or annotation processors, this implementation is entirely compile-time static. The overhead of method calls through the decorator chain is minimal (one additional virtual method dispatch per decorator per method call).
