Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality. The Decorator pattern wraps an object with one or more layers of enhancement, each adding its own behavior, cost, or description, without altering the core object's interface. In the cars domain, this means starting with a base vehicle such as a Toyota Camry and stacking optional features — leather seats, a panoramic sunroof, a sport package, a premium navigation system — each contributing its own cost and description, all while preserving the same `Car` interface the dealership system expects.

## The Problem Without This Pattern

Consider an automobile dealership that sells car models with optional features. Customers can configure a car with any combination of features such as leather seats, a sunroof, a sport package, and a navigation system. Each combination has a unique total price and a composite description.

Without the Decorator pattern, the naive approach is to create a subclass for every possible combination of features:

```java
// PROBLEMATIC: Subclass explosion for every feature combination
class Camry { }
class CamryWithLeather extends Camry { }
class CamryWithSunroof extends Camry { }
class CamryWithLeatherAndSunroof extends CamryWithLeather { }
class CamryWithSportPackage extends Camry { }
class CamryWithLeatherAndSportPackage extends CamryWithLeather { }
class CamryWithSunroofAndSportPackage extends CamryWithSunroof { }
class CamryWithLeatherAndSunroofAndSportPackage extends CamryWithLeatherAndSunroof { }
// ... and so on for every permutation of 4+ features
```

This approach has several critical problems:

1.  **Class explosion** — With four optional features, you need 2^4 = 16 subclasses to cover all combinations. With five features, 32 subclasses. With ten features, over 1,000 subclasses. This growth rate is unsustainable.

2.  **Code duplication** — Each subclass duplicates the cost and description logic for its inherited features. Changing the price of leather seats means updating every subclass that includes leather seats.

3.  **Rigid at compile time** — Feature combinations are baked into the class hierarchy at compile time. There is no way to add features dynamically at runtime based on user preferences or inventory availability.

4.  **Open/closed principle violation** — Adding a new feature (e.g., heated steering wheel) requires adding new subclasses for every existing combination that includes the new feature. The system is not closed for modification.

5.  **Combinatorial complexity** — Testing requires verifying every subclass individually. With 32 subclasses for 5 features, the test suite grows unmanageably large.

6.  **Rigid pricing updates** — When the dealership runs a promotion that discounts the sunroof by $200, every subclass that includes a sunroof must be updated. With 16 subclasses for 4 features, half of them include the sunroof, meaning 8 classes need modification. The decorator pattern would change a single constant in the `Sunroof` class.

### A Concrete Comparison

Consider the table below showing the class count growth as features are added:

| Number of Features | Subclasses Required (Subclassing) | Classes Required (Decorator) |
|---|---|---|
| 2 | 4 | 4 (1 base + 1 decorator base + 2 features) |
| 3 | 8 | 5 (1 base + 1 decorator base + 3 features) |
| 4 | 16 | 6 (1 base + 1 decorator base + 4 features) |
| 5 | 32 | 7 |
| 10 | 1,024 | 12 |
| 15 | 32,768 | 17 |

With subclassing, the number of classes doubles with each new feature. With the Decorator pattern, each new feature adds exactly one class. For a dealership with 15 optional features, class count under subclassing exceeds 32,000 — an unmanageable codebase. Under the Decorator pattern, it is 17 clean, single-responsibility classes. This is the core motivation for the pattern.

### What Subclassing Looks Like in Practice

For a customer configuring a car, the dealership system must support every combination the customer might choose. With the subclassing approach, the system creates a class hierarchy like the following:

### How the Participants Interact

The ConcreteComponent (BaseCar) is the core object that may be wrapped. The Decorator (CarFeature) holds a reference to a Car and implements the same interface, forwarding calls to the wrapped object. ConcreteDecorators override one or more methods to add their own behavior before or after delegating to the wrapped component. Because both the core component and all decorators share the same `Car` interface, decorators can wrap other decorators recursively, building a stack of enhancements.

### Participant Responsibilities

Each participant in the Decorator pattern has a specific role:

- **Component (`Car`)** — Declares the interface for objects that can have responsibilities added to them dynamically. This is the common contract shared by both the core object and all decorators. In the car domain, `Car` declares `getDescription()` and `getCost()`.

- **ConcreteComponent (`BaseCar`)** — Defines the core object to which additional responsibilities can be attached. This is the innermost object in the decorator stack. `BaseCar` stores the car's model name and base price and provides the default implementations of `getDescription()` and `getCost()`.

- **Decorator (`CarFeature`)** — Maintains a reference to a `Component` object and defines an interface that conforms to the `Component`'s interface. The decorator delegates all operations to the wrapped component by default, but subclasses can override specific methods to add behavior before or after the delegation. This is the abstract class that all concrete decorators extend.

- **ConcreteDecorator (`LeatherSeats`, `Sunroof`, `SportPackage`, `NavigationSystem`)** — Adds responsibilities to the component by overriding the decorator's methods. Each concrete decorator adds its own contribution (feature name and cost) while delegating to the wrapped component for the base behavior. Concrete decorators can be stacked in any order, and each one only knows about its own contribution and the wrapped object.

### Call Flow Across the Decorator Stack

When a method is called on the outermost decorator, the call flows through each layer of the stack in a predictable pattern:

```text
1. SportPackage.getCost()             → delegates to NavigationSystem.getCost()
2.   NavigationSystem.getCost()       → delegates to Sunroof.getCost()
3.     Sunroof.getCost()              → delegates to LeatherSeats.getCost()
4.       LeatherSeats.getCost()       → delegates to BaseCar.getCost()
5.         BaseCar.getCost()          → returns 28000.00
6.       LeatherSeats.getCost()       → returns 28000.00 + 1500.00 = 29500.00
7.     Sunroof.getCost()              → returns 29500.00 + 1200.00 = 30700.00
8.   NavigationSystem.getCost()       → returns 30700.00 + 2000.00 = 32700.00
9. SportPackage.getCost()             → returns 32700.00 + 3500.00 = 36200.00
```

This flow is known as the "onion" or "layered" pattern: each layer adds its own contribution as the call returns up the stack. The same pattern applies to `getDescription()`, where each layer appends its feature name to the description string being built as the call returns.

## Pseudocode — Cars Example

The following pseudocode demonstrates the Decorator pattern in the context of configuring a car with optional features. The `Car` interface defines the contract; `BaseCar` is the concrete car without options; `CarFeature` is the abstract decorator; and `LeatherSeats`, `Sunroof`, `SportPackage`, and `NavigationSystem` are concrete decorators.

```text
// PSEUDOCODE — Decorator Pattern in Cars Domain
// Participants labeled with [ROLE]

// ── Component ──
interface Car
    method getDescription(): string
    method getCost(): double

// ── ConcreteComponent ──
class BaseCar implements Car
    private model: string
    private baseCost: double

    constructor(model: string, baseCost: double)
        this.model = model
        this.baseCost = baseCost

    method getDescription(): string
        return this.model

    method getCost(): double
        return this.baseCost

// ── Decorator (abstract) ──
abstract class CarFeature implements Car
    protected car: Car

    constructor(car: Car)
        this.car = car

    method getDescription(): string
        return car.getDescription()

    method getCost(): double
        return car.getCost()

// ── ConcreteDecorator 1: LeatherSeats ──
class LeatherSeats extends CarFeature
    private static FEATURE_COST: double = 1500.00
    private static FEATURE_NAME: string = "Leather Seats"

    constructor(car: Car)
        super(car)

    method getDescription(): string
        return super.getDescription() + " + " + FEATURE_NAME

    method getCost(): double
        return super.getCost() + FEATURE_COST

// ── ConcreteDecorator 2: Sunroof ──
class Sunroof extends CarFeature
    private static FEATURE_COST: double = 1200.00
    private static FEATURE_NAME: string = "Sunroof"

    constructor(car: Car)
        super(car)

    method getDescription(): string
        return super.getDescription() + " + " + FEATURE_NAME

    method getCost(): double
        return super.getCost() + FEATURE_COST

// ── ConcreteDecorator 3: SportPackage ──
class SportPackage extends CarFeature
    private static FEATURE_COST: double = 3500.00
    private static FEATURE_NAME: string = "Sport Package"

    constructor(car: Car)
        super(car)

    method getDescription(): string
        return super.getDescription() + " + " + FEATURE_NAME

    method getCost(): double
        return super.getCost() + FEATURE_COST

// ── ConcreteDecorator 4: NavigationSystem ──
class NavigationSystem extends CarFeature
    private static FEATURE_COST: double = 2000.00
    private static FEATURE_NAME: string = "Navigation System"

    constructor(car: Car)
        super(car)

    method getDescription(): string
        return super.getDescription() + " + " + FEATURE_NAME

    method getCost(): double
        return super.getCost() + FEATURE_COST

// ── Driver ──
function main()
    // Build a base car
    baseCar: Car = new BaseCar("Toyota Camry", 28000.00)
    print(baseCar.getDescription() + " — $" + baseCar.getCost())
    // Output: Toyota Camry — $28000.00

    // Add leather seats
    withLeather: Car = new LeatherSeats(baseCar)
    print(withLeather.getDescription() + " — $" + withLeather.getCost())
    // Output: Toyota Camry + Leather Seats — $29500.00

    // Add leather seats + sunroof
    withLeatherAndSunroof: Car = new Sunroof(new LeatherSeats(baseCar))
    print(withLeatherAndSunroof.getDescription() + " — $" + withLeatherAndSunroof.getCost())
    // Output: Toyota Camry + Leather Seats + Sunroof — $30700.00

    // Fully loaded: add all four features
    fullyLoaded: Car = new SportPackage(
        new NavigationSystem(
            new Sunroof(
                new LeatherSeats(baseCar)
            )
        )
    )
    print(fullyLoaded.getDescription() + " — $" + fullyLoaded.getCost())
    // Output: Toyota Camry + Leather Seats + Sunroof + Navigation System + Sport Package — $36200.00

    // Verify the interface is preserved
    print("All objects are Cars: " +
        (baseCar is Car) + ", " +
        (withLeather is Car) + ", " +
        (fullyLoaded is Car))
    // Output: All objects are Cars: true, true, true
```

Expected output:

```bash
Toyota Camry — $28000.00
Toyota Camry + Leather Seats — $29500.00
Toyota Camry + Leather Seats + Sunroof — $30700.00
Toyota Camry + Leather Seats + Sunroof + Navigation System + Sport Package — $36200.00
All objects are Cars: true, true, true
```

## How It Works — Step by Step

1.  **Create the core component** — The process begins by instantiating the `ConcreteComponent`, in this case a `BaseCar` with the model "Toyota Camry" and a base price of $28,000. This object is the innermost layer of the decorator stack and provides the default `getDescription()` and `getCost()` implementations.

2.  **Wrap with the first decorator** — A `LeatherSeats` decorator is constructed, receiving the `baseCar` as its argument. The `LeatherSeats` constructor stores the reference to the wrapped car in its `car` field. At this point, `LeatherSeats` exposes the same `Car` interface as `baseCar`, but its `getCost()` returns `car.getCost() + 1500.00` and its `getDescription()` returns `car.getDescription() + " + Leather Seats"`.

3.  **Chain additional decorators** — A `Sunroof` decorator wraps the `LeatherSeats` instance: `new Sunroof(leatherSeats)`. The `Sunroof` now holds a reference to `leatherSeats`, which in turns holds a reference to `baseCar`. Each decorator in the chain contributes to the final result when a method is called.

4.  **Call the outermost method** — When the client calls `fullyLoaded.getCost()`, the call is received by `SportPackage`, the outermost decorator. `SportPackage.getCost()` calls `super.getCost()` which delegates to its wrapped `NavigationSystem`, which in turn delegates through the chain until reaching `BaseCar.getCost()`, which returns 28000.00. Then each decorator unwinds, adding its own contribution: +2000 for Navigation, +1200 for Sunroof, +1500 for Leather, +3500 for SportPackage, for a total of $36,200.

5.  **Description string construction** — The same chain-of-delegation pattern applies to `getDescription()`. `SportPackage.getDescription()` calls `super.getDescription()`, which propagates all the way down to `BaseCar.getDescription()` returning "Toyota Camry". As the call stack unwinds, each decorator appends its feature name: " + Leather Seats", " + Sunroof", " + Navigation System", " + Sport Package".

6.  **Different combinations from the same base** — A second configuration, such as a "budget sporty Camry" with only `SportPackage` wrapped directly on `baseCar`, uses the same `SportPackage` class — no new subclass needed. The core component is reused, and decorators are mixed and matched freely.

7.  **Interface identity is preserved** — The driver verifies that every decorated object satisfies the `Car` interface. Client code written against `Car` works with any decorated variant without modification. This is the key benefit of the Decorator pattern: feature-rich objects conform to the same contract as the original.

8.  **Multiple configurations from a single base** — The same `BaseCar` instance can be reused as the foundation for multiple configurations. A Toyota Camry can be wrapped with `LeatherSeats` for one customer, with `SportPackage` for another, and with all four features for a third, all without creating additional `BaseCar` subclasses. The core object is shared while the decorations vary.

9.  **Feature removal is transparent** — Removing a feature is as simple as removing one decorator layer from the chain. If a customer changes their mind about the sunroof, the system removes the `Sunroof` decorator from the chain, and the remaining decorators continue to work. No subclass needs to be removed or recompiled.

10. **Nested method calls follow the call stack** — When `getCost()` is called on the outermost decorator, the call sequence is: `SportPackage.getCost()` → `NavigationSystem.getCost()` → `Sunroof.getCost()` → `LeatherSeats.getCost()` → `BaseCar.getCost()` → returns 28000 → adds 1500 → adds 1200 → adds 2000 → adds 3500 → returns 36200. This is identical to the way a call stack unwinds in a programming language, making the flow predictable and debuggable.

11. **Decorator order affects description order** — While the total cost is the same regardless of wrapping order (since all features are additive), the description string order follows the wrapping order. `Sunroof(LeatherSeats(car))` produces "Camry + Leather Seats + Sunroof" while `LeatherSeats(Sunroof(car))` produces "Camry + Sunroof + Leather Seats". The dealership system should present the description in a consistent order regardless of the internal wrapping order, which may require sorting the feature names before display.

## Consequences and Trade-offs

### Benefits

- **More flexibility than static inheritance** — The Decorator pattern provides a runtime-configurable alternative to inheritance. Features can be mixed, matched, added, and removed without recompiling any class. The system can even load decorator configurations from a database or configuration file at runtime.

- **Avoids feature-laden base classes** — Without the Decorator pattern, the `BaseCar` class might accumulate methods for every possible feature (`hasLeatherSeats()`, `hasSunroof()`, `hasSportPackage()`, `hasNavigationSystem()`, `getLeatherSeatsCost()`, `getSunroofCost()`, and so on). The Decorator pattern keeps the base class lean and focused on the core vehicle identity.

- **Single Responsibility is maintained** — Each decorator class has exactly one responsibility: representing one optional feature. No class needs to understand the interaction between features. The total cost is simply the sum of the base cost and all chosen decorator costs.

- **Layered behavior is explicit** — The decorator stack makes the layered nature of feature configuration visible in the code. A call like `new SportPackage(new NavigationSystem(new Sunroof(new LeatherSeats(baseCamry))))` clearly communicates that the car has four features stacked on top of the base model.

### Drawbacks

- **Many small objects** — A fully loaded car with ten features creates ten decorator objects plus the base object. While each object is small, the total number of objects can be large in a system that maintains many car configurations simultaneously. This increases memory usage and garbage collection pressure.

- **Type identity is obscured** — A decorated `BaseCar` is no longer an instance of `BaseCar` at the type level. Code that checks `car instanceof BaseCar` will not recognize a decorated car. The driver in the pseudocode section demonstrates this: `fullyLoaded is Car` returns true, but `fullyLoaded is BaseCar` returns false because the outermost object is a `SportPackage`, not a `BaseCar`.

- **Composition ordering dependencies** — If features have dependencies (e.g., "Sport Package requires the upgraded suspension, which is incompatible with the standard suspension"), the decorator pattern does not enforce these constraints. The dealership system must add validation logic outside the decorator chain.

- **Debugging complexity** — A deeply nested decorator chain produces a long call stack. Debugging a `getCost()` call that passes through ten decorators requires stepping through each layer. Modern debuggers mitigate this with stack trace navigation, but the cognitive load is higher than a single method call on a concrete class.

- **Identical interface, different behavior** — Because all decorated objects share the `Car` interface, it is not immediately obvious from the type system what features a particular car instance has. A method that accepts a `Car` parameter has no way to know at compile time whether the car is a base model or a fully loaded one. This is both a benefit (flexibility) and a drawback (lost type information).

## When to Use

- **Add features dynamically at runtime** — When a customer at the dealership configures a car by selecting options from a menu, the decorator pattern allows the system to wrap the base car with exactly the chosen features at runtime. No compile-time decisions are required.

- **Avoid a subclass explosion** — When the number of possible feature combinations grows exponentially, the decorator pattern reduces the class count from O(2^n) subclasses to O(n) decorator classes plus one base component. For ten features, this is 11 classes instead of 1,024.

- **Extend a class that is closed for modification** — When the core `Car` class or interface belongs to a third-party library and cannot be modified, decorators provide a way to add new behavior by wrapping the object externally.

- **Apply multiple, independent enhancements** — When features are orthogonal (e.g., leather seats and sunroof are independent), decorators compose naturally. Removing a feature is simply a matter of removing one decorator from the chain.

- **Adhere to the Open/Closed Principle** — New features (e.g., heated steering wheel, parking sensors, ambient lighting) can be added as new decorator classes without changing any existing decorator or the core component.

- **Feature pricing that tiers additively** — When each feature contributes a distinct, additive cost (or other numerical property), the decorator pattern mirrors this model directly. Each decorator adds its own cost to the chain's total.

- **When the system needs to support dynamically changing feature sets** — If the dealership periodically introduces new promotional packages (e.g., "Summer Special" with a discounted sunroof and free floor mats) and removes old ones, the decorator pattern handles this cleanly. New decorator classes are added for the promotion, and old ones remain in the codebase but are no longer instantiated by the configuration system.

- **When the configuration logic should be separated from the business domain** — The decorator pattern allows the car configuration logic (which features are available, their base prices, their descriptions) to be cleanly separated from the business logic (loan calculations, trade-in valuation, inventory management). The `Car` interface acts as the boundary between the two concerns.

- **When features need to be composed in a mix-and-match fashion** — If the dealership allows customers to build custom trims by selecting individual features rather than buying pre-defined trim levels (LE, XLE, SE, XSE), the decorator pattern is the natural fit. Each feature is a decorator, and the customer's selection is simply the set of decorators applied to the base model.

## When NOT to Use

- **When the order of decoration matters and is constrained** — If certain feature combinations are invalid (e.g., sunroof cannot be installed with a roof rack), the decorator pattern does not enforce these constraints by itself. Additional validation logic must be added.

- **When the core object itself needs to change its behavior** — The decorator pattern enhances an object from the outside. If the core `BaseCar` itself must change its intrinsic properties (e.g., changing its engine type), a decorator is not the right tool — consider the Strategy pattern for varying algorithms.

- **When there are many decorators that add the same kind of behavior** — If all decorators simply add a flat cost and a label, and there are dozens of them, the pattern introduces many small classes that each do almost the same thing. Configuration-driven approaches (e.g., a list of feature descriptors) might be simpler.

- **When excessive layering hurts performance or clarity** — A deeply nested decorator stack can make debugging difficult because the call flow jumps through many classes. For a dealership system with only a handful of features this is rarely an issue, but with dozens of decorators the indirection can become confusing.

- **When the client needs to access decorator-specific methods** — The decorator pattern works best when clients interact through the shared interface. If a client needs to call `hasLeatherSeats()` which is only available on `LeatherSeats` and not on `Car`, the pattern breaks down. In such cases, consider using the Visitor pattern or a query method on the component.

- **When features interact in complex, non-additive ways** — The decorator pattern assumes features are additive: each feature simply adds its cost and its description segment. If features interact in complex ways (e.g., "If the car has both leather seats and a sunroof, the total sunroof surcharge increases by $200 because the sunroof must be reinforced for the leather interior"), the decorator pattern has no built-in mechanism to model these interactions. Each decorator knows only about its own contribution and the base it wraps.

- **When the object creation code becomes unreadable** — Nesting decorator constructors ten levels deep (`new A(new B(new C(new D(new E(new F(new G(new H(new I(new J(base))))))))))`) produces code that is difficult to read, format, and maintain. For systems with many optional features, consider using a builder or fluent API that internally applies decorators but presents a cleaner interface to the caller.

- **When most clients need the fully decorated object** — If 90% of customers choose the "Fully Loaded" package, the decorator pattern introduces unnecessary complexity. A single `FullyLoadedCamry` subclass that includes all features would be simpler and more efficient. The Decorator pattern excels when there is genuine variability in feature selection, not when there is a single dominant configuration.

## Real-World Occurrences

1.  **Java I/O Streams (`java.io`)** — The Java standard library's I/O classes are the most widely cited example of the Decorator pattern. `InputStream` is the Component interface, `FileInputStream` (reading from a file) is a ConcreteComponent, and `BufferedInputStream` (adding buffering for performance) and `DataInputStream` (adding structured data reading methods) are concrete decorators. A call such as `new DataInputStream(new BufferedInputStream(new FileInputStream("data.bin")))` layers two decorators on top of a raw file stream, each contributing its own enhancement. The `FilterInputStream` abstract class serves as the Decorator base, exactly like `CarFeature` in the car domain. This design has been part of Java since version 1.0 and demonstrates the pattern's longevity and effectiveness.

2.  **Python Function Decorators (`@decorator`)** — Python's built-in decorator syntax (`@staticmethod`, `@property`, `@functools.lru_cache`, `@functools.wraps`) is a language-level embodiment of the pattern. While Python's syntactic decorators apply to functions or classes rather than objects, the underlying principle is identical: wrap a callable with additional behavior while preserving the original callable's interface. A function decorated with `@lru_cache(maxsize=128)` is still callable with the same arguments as the original but gains transparent memoization — results are cached, and repeated calls with the same arguments return instantly. The `@property` decorator wraps a method so it is accessed as an attribute rather than called explicitly. These decorators can be stacked: `@property` and `@lru_cache` can be combined, just like car features.

3.  **Express.js / Koa.js Middleware Pipelines** — In Node.js web frameworks, each middleware function wraps the next handler in the chain. A request passes through middleware A, which delegates to middleware B, which delegates to the final route handler. This is the Decorator pattern applied to request processing. Logging middleware, authentication middleware, body-parsing middleware, compression middleware, and error-handling middleware are all decorators around the core route handler. Removing a middleware is as simple as removing it from the stack, just as removing a decorator from the car feature chain. Express.js even calls the `next()` function to delegate to the next middleware, mirroring the `super.getCost()` delegation in the car example.

4.  **React Higher-Order Components (HOCs)** — In React, a higher-order component is a function that takes a component and returns a new component with enhanced behavior. `withLogging(WrappedComponent)`, `withAuth(ProtectedPage)`, `withRouter(MyComponent)`, and `connect(mapStateToProps)(MyComponent)` (from Redux) are decorators that add cross-cutting concerns while preserving the original component's interface. The wrapped component remains a valid React component, just as a decorated car remains a valid `Car`. React's newer hooks API has partially replaced HOCs, but the pattern remains prevalent in legacy React codebases and in libraries that need to compose behavior at the component level.

5.  **.NET Core Middleware Pipeline** — In ASP.NET Core, each middleware component can process an HTTP request before and after the next component in the pipeline. The `UseMiddleware<T>` extension method builds a decorator chain around the core request handler, with each middleware adding logging, authentication, compression, response-caching, or CORS behavior. The pipeline's `RequestDelegate` is the Component, the terminal middleware that produces the response is the ConcreteComponent, and each `UseMiddleware<T>` call adds a Decorator. This mirrors the car domain: the base car is the terminal handler, and each feature is a middleware component.

6.  **Rust Iterator Adapters** — Rust's iterator pattern uses combinator methods (`map`, `filter`, `take`, `skip`, `chain`, `zip`, `enumerate`, `rev`) that each return a new iterator wrapping the previous one. `vec.iter().map(f).filter(g).take(10)` creates a decorator chain where each adapter adds transformation or filtering behavior while preserving the `Iterator` trait. The key difference from classical OOP decorators is that Rust's iterator adapters are zero-cost abstractions — the compiler inlines the entire chain into a tight loop, eliminating any runtime overhead from the wrapping. This demonstrates that the Decorator pattern is not limited to object-oriented languages; it is a structural concept that applies wherever a wrapper layer can add behavior to a core contract.

7.  **Django/Python Framework Middleware** — Django's middleware framework processes HTTP requests through a chain of middleware classes. Each middleware can modify the request before passing it to the next middleware or view, and can modify the response before returning it. `AuthenticationMiddleware`, `SessionMiddleware`, `CsrfViewMiddleware`, and `GZipMiddleware` are decorators around the core view handler. Django's middleware is configured in `settings.py` as a list, where order matters — just as the order of decorators in the car domain affects the description string. Removing a middleware from the list removes that behavior from every request.

8.  **Java Swing `JScrollPane`** — In Java Swing (GUI toolkit), `JScrollPane` wraps a component (such as a `JTextArea` or `JTable`) and adds scrollbar functionality. The wrapped component remains fully functional and can be placed inside other containers. `JScrollPane` is a decorator that adds horizontal and vertical scrolling without modifying the wrapped component's code. Additional decorators like `JViewport` and `JLayer` can be composed to add viewport clipping, zooming, or custom painting on top of the base component.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Strategy** | Strategy encapsulates interchangeable algorithms in separate classes; Decorator wraps a core object to add new responsibilities. | Choose Strategy when the variation is in how something is done (e.g., different engine power calculations) rather than what additional features are present. |
| **Composite** | Composite composes objects into tree structures to represent part-whole hierarchies; Decorator aggregates one object to add behavior. Composite works with many children; Decorator wraps exactly one. | Choose Composite when representing a hierarchy of parts and assemblies (e.g., a car composed of engine, transmission, wheels). Decorator adds incidental features, not structural parts. |
| **Prototype** | Prototype creates new objects by cloning an existing instance; Decorator modifies an existing object by wrapping it. | Choose Prototype when the cost of constructing a fully configured object from scratch is high and cloning is cheaper — for example, cloning a pre-configured "loaded Camry" template instead of applying decorators each time. |
| **Adapter** | Adapter converts one interface to another; Decorator keeps the same interface and extends behavior. | Choose Adapter when integrating a third-party component with an incompatible interface into the dealership system. Decorator extends, while Adapter translates. |
| **Proxy** | Proxy controls access to an object (lazy loading, access control, logging); Decorator adds behavior transparently. Both have the same interface as the wrapped object. | Choose Proxy when the goal is lifecycle management or access restriction (e.g., a proxy that loads the car configuration from a remote server on demand). Choose Decorator when the goal is adding features or responsibilities. |
| **Chain of Responsibility** | Chain of Responsibility passes a request through a chain of handlers until one handles it; Decorator passes through every layer, with each layer contributing to the result. | Choose Chain of Responsibility when exactly one handler should process a request (e.g., routing a service request to the right department). Decorator accumulates contributions from all layers. |
| **Facade** | Facade provides a simplified interface to a complex subsystem; Decorator adds behavior while preserving the original interface. | Choose Facade when you need to hide the complexity of a car configuration system behind a simple `configureCar(options)` method. The Facade would internally apply decorators but present a clean API to the caller. |
| **Template Method** | Template Method defines the skeleton of an algorithm in a base class, with subclasses overriding specific steps. Decorator adds behavior externally via wrapping. | Choose Template Method when the car configuration process has a fixed sequence of steps with some optional variations (e.g., "apply base price, add features, apply discounts, compute taxes") where the steps are defined in the base class and subclasses refine specific steps. |
| **Flyweight** | Flyweight shares fine-grained objects to minimize memory usage; Decorator creates a chain of wrapper objects, each with its own state. | Choose Flyweight when many car configurations share the same features and the system needs to minimize memory usage. Feature objects could be shared across configurations, with only the unique combination tracked per car. |

## Testing Considerations

Testing decorator-based code requires a different approach than testing a subclass hierarchy.

### Unit Testing Individual Decorators

Each decorator should be tested in isolation against a mock or stub concrete component:

- Create a `MockCar` (a test double implementing `Car`) that returns known values from `getDescription()` and `getCost()`.
- Wrap the `MockCar` with a single decorator (e.g., `new LeatherSeats(mockCar)`).
- Verify that `getCost()` returns `mockCar.getCost() + 1500.00` and `getDescription()` returns `mockCar.getDescription() + " + Leather Seats"`.
- This tests each decorator's logic independently of other decorators and the real `BaseCar` implementation.

### Integration Testing Decorator Chains

After verifying individual decorators, test common chain configurations:

- Create a `BaseCar` with a known model and base price.
- Apply a known set of decorators (e.g., `LeatherSeats` + `Sunroof`).
- Verify that the description and cost are the cumulative result of all decorators in the chain.
- Test edge cases such as empty chains (no decorators), single decorator, and all decorators.

### Testing Decorator Order Independence

Since the decorator pattern for additive features should produce the same total cost regardless of wrapping order:

- Apply decorators in different orders and verify the cost is identical.
- Verify that the description, while potentially differing in feature order, contains all expected feature names.
- If the system requires a canonical description order, test that the display logic normalizes the order independently of the internal wrapping order.

### Common Testing Pitfalls

- **Testing type identity** — Do not use `instanceof` checks against concrete types in test assertions unless the test explicitly validates type identity behavior. Decorated objects are not instances of the core concrete type.
- **Testing through the decorator chain** — Avoid testing the inner decorator's behavior through the outer decorator. Each level of indirection makes test failures harder to diagnose. Test each decorator independently.
- **Mocking the wrong layer** — When testing a decorator, mock the wrapped `Car`, not the `CarFeature` base class. Mock at the interface contract level.

## Implementing the Pattern Step by Step

To implement the Decorator pattern in a new project, follow these steps:

1. **Define the Component interface** — Create the interface (or abstract class) that declares the common methods. In the car domain, this is `Car` with `getDescription()` and `getCost()`. Keep the interface minimal — only the methods that decorators need to override.

2. **Create the ConcreteComponent** — Implement the core object that will be wrapped. `BaseCar` implements `Car` and provides the baseline implementations. The concrete component should be simple and focused on its intrinsic state.

3. **Create the abstract Decorator class** — `CarFeature` implements `Car` and holds a protected reference to a `Car` instance (the wrapped object). The constructor takes a `Car` parameter. Default implementations of the `Car` methods simply delegate to the wrapped object. The constructor should validate that the wrapped object is not null.

4. **Create ConcreteDecorators** — For each feature that can be added to the car, create a decorator class that extends `CarFeature`:
   - Override `getDescription()` to append the feature name.
   - Override `getCost()` to add the feature's price.
   - Keep each decorator focused on exactly one feature.

5. **Wire decorators at runtime** — In the client code (the driver or configuration system), instantiate the `BaseCar` and wrap it with decorators as needed:
   ```java
   Car myCar = new Sunroof(new LeatherSeats(new BaseCar("Camry", 28000)));
   ```

6. **Use the decorated object through the Component interface** — All client code should interact with the car through the `Car` interface. This ensures that decorated and undecorated cars are interchangeable and that the system can be extended with new decorators without modifying existing client code.

## Summary

The Decorator pattern provides a flexible, runtime-configurable alternative to subclassing for adding responsibilities to objects. In the car dealership domain, it eliminates the combinatorial explosion of feature combination subclasses by representing each optional feature as a standalone decorator class that wraps the base car. Key takeaways:

- **Four participants**: Component (interface), ConcreteComponent (core object), Decorator (abstract wrapper), ConcreteDecorator (specific feature).
- **Composition over inheritance**: Features are combined by wrapping objects at runtime rather than creating a compile-time hierarchy.
- **Open/Closed Principle**: New features are added as new decorator classes without modifying existing code.
- **Transparent interface**: Clients interact with all car variants through the same `Car` interface, with no knowledge of which decorators are applied.
- **Predictable additive behavior**: Each decorator contributes its own cost and description, and the total is the sum of contributions from all layers in the chain.
- **Testing is straightforward**: Each decorator can be tested in isolation with a mock component, and chain configurations are tested as integration scenarios.

The Decorator pattern is one of the most widely used structural patterns in software engineering, appearing in Java I/O streams, Python decorators, web middleware pipelines, and iterator adapters across multiple languages. Its fundamental insight — that wrapping an object with behavior-preserving layers is more flexible than baking every combination into a class hierarchy — applies as readily to car features as to stream buffering, request processing, or data transformation.
