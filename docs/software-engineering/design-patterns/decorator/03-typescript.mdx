## Implementation

Below is a complete TypeScript implementation of the Decorator pattern using a car dealership domain. TypeScript's strong type system makes the pattern particularly clean: the `Car` interface defines the contract with full type safety, the abstract `CarFeature` class enforces the decorator structure, and each concrete decorator adds its own cost and description to the chain. The driver demonstrates stacking decorators at runtime to configure multiple car variants.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌──────────────────────────────────────────────┐
//   │            <<interface>>                      │
//   │                 Car                           │  ← OOP: Abstraction
//   ├──────────────────────────────────────────────┤     ISP: 2 methods
//   │  + getDescription(): string                   │
//   │  + getCost(): number                          │
//   └────────────────────┬─────────────────────────┘
//                        │ implements
//              ┌─────────┴─────────┐
//              │                   │
//        ┌─────┴──────┐    ┌──────┴──────────┐
//        │  BaseCar   │    │   CarFeature     │  ← OOP: Composition
//        │ implements │    │   <<abstract>>   │     DIP: depends on
//        │   Car      │    │   implements Car │     Car interface
//        └────────────┘    └──────┬───────────┘
//                                 │ extends
//            ┌────────────────────┼────────────────────┐
//            │                    │                    │
//            ▼                    ▼                    ▼
//   ┌────────────────┐  ┌──────────────┐  ┌──────────────────┐
//   │  LeatherSeats  │  │   Sunroof    │  │  SportPackage    │
//   │  OCP: new feat │  │ LSP: subst.  │  │ SRP: one feature │
//   └────────────────┘  └──────────────┘  └──────────────────┘
//        ┌──────────────────────────────────────┐
//        │           NavigationSystem            │
//        │           LSP: substitutable          │
//        └──────────────────────────────────────┘
```

#### OOP Concepts Applied

1. **Abstraction** — The `Car` interface defines the abstract contract. Client code written against `Car` never depends on concrete implementations. Every car variant — plain or fully loaded — is accessed through the same two-method interface.

2. **Polymorphism** — `BaseCar` and every decorated variant satisfy the `Car` interface. The driver iterates over a heterogeneous collection of `Car` objects and calls `getDescription()` and `getCost()` uniformly, with each object returning its correct, decorator-specific result.

3. **Composition over Inheritance** — Features are composed at runtime by wrapping objects, not by creating a combinatorial explosion of subclasses. The type system enforces that only `Car` instances can be wrapped, providing compile-time safety.

#### SOLID Principles Applied

4. **SRP** — `BaseCar` is responsible for the base vehicle identity only. Each decorator is responsible for exactly one optional feature. The driver is responsible for configuration assembly and output. No class does more than one thing.

5. **OCP** — Adding a new feature (e.g., `HeatedSteeringWheel`) requires creating a single new decorator class without modifying `BaseCar` or any existing decorator. The system is closed for modification, open for extension.

6. **LSP** — Every decorator's `getDescription()` and `getCost()` methods maintain the contract defined by the `Car` interface. A decorated car can replace a plain `BaseCar` in any client code without behavioral change.

7. **ISP** — The `Car` interface is minimal (two methods). No client is forced to depend on methods it does not use. If a client only needs the description, it still depends on both methods, but the interface is small enough that this is not a burden.

8. **DIP** — The driver, `CarFeature`, and all client code depend on the `Car` interface. `BaseCar` and concrete decorators implement `Car`. High-level modules are not coupled to low-level details.

### Component interface

```typescript
interface Car {
  getDescription(): string;
  getCost(): number;
}
```

### ConcreteComponent

```typescript
class BaseCar implements Car {
  private readonly model: string;
  private readonly baseCost: number;

  constructor(model: string, baseCost: number) {
    this.model = model;
    this.baseCost = baseCost;
  }

  getDescription(): string {
    return this.model;
  }

  getCost(): number {
    return this.baseCost;
  }
}
```

### Decorator (Abstract)

```typescript
abstract class CarFeature implements Car {
  protected car: Car;

  constructor(car: Car) {
    this.car = car;
  }

  getDescription(): string {
    return this.car.getDescription();
  }

  getCost(): number {
    return this.car.getCost();
  }
}
```

### ConcreteDecorator 1: LeatherSeats

```typescript
class LeatherSeats extends CarFeature {
  private static readonly FEATURE_COST: number = 1500.00;
  private static readonly FEATURE_NAME: string = "Leather Seats";

  constructor(car: Car) {
    super(car);
  }

  getDescription(): string {
    return `${super.getDescription()} + ${LeatherSeats.FEATURE_NAME}`;
  }

  getCost(): number {
    return super.getCost() + LeatherSeats.FEATURE_COST;
  }
}
```

### Utility: build a formatted summary string

```typescript
function formatConfig(car: Car): string {
  const desc = car.getDescription().padEnd(60);
  const cost = `$${car.getCost().toFixed(2)}`;
  return `  ${desc} ${cost}`;
}
```

### Main / Driver

```typescript
function main(): void {
  console.log("=== TypeScript Car Dealership ===");
  console.log();

  // Build configurations
  const baseCamry: Car = new BaseCar("Toyota Camry", 28000.00);
  const withLeather: Car = new LeatherSeats(baseCamry);
  const withSunroof: Car = new Sunroof(baseCamry);
  const withLeatherAndSunroof: Car = new Sunroof(new LeatherSeats(baseCamry));
  const luxurySport: Car = new SportPackage(
    new NavigationSystem(
      new Sunroof(new LeatherSeats(baseCamry))
    )
  );
  const ecoLuxury: Car = new NavigationSystem(new LeatherSeats(baseCamry));
  const sportOnly: Car = new SportPackage(baseCamry);

  // Print configurations
  const configurations: Car[] = [
    baseCamry,
    withLeather,
    withSunroof,
    withLeatherAndSunroof,
    ecoLuxury,
    sportOnly,
    luxurySport,
  ];

  console.log("Configuration                                                         Price");
  console.log("----------------------------------------------------------------------------");
  configurations.forEach(c => console.log(formatConfig(c)));

  // Calculate and display price breakdown
  console.log();
  console.log("--- Price Breakdown: Fully Loaded Camry ---");
  console.log(`  Base car:                    $${baseCamry.getCost().toFixed(2)}`);
  console.log(`  + Leather Seats:             $${(luxurySport.getCost() - new LeatherSeats(baseCamry).getCost() + baseCamry.getCost()).toFixed(2)}`);
  console.log(`  + Leather Seats contribution: $${LeatherSeats["FEATURE_COST"] ?? "$1500.00"}`);
  console.log(`  + Sunroof contribution:       $${(Sunroof as any).FEATURE_COST ?? 1200.00}`);
  console.log(`  + Navigation System:          $${(NavigationSystem as any).FEATURE_COST ?? 2000.00}`);
  console.log(`  + Sport Package:              $${(SportPackage as any).FEATURE_COST ?? 3500.00}`);

  // Interface compliance
  console.log();
  console.log("--- Interface Compliance ---");
  configurations.forEach((c, i) => {
    console.log(`  Config ${i + 1}: Car interface = ${c instanceof BaseCar || c instanceof CarFeature}`);
  });

  // Build a second car model to show reuse
  console.log();
  console.log("--- Honda Accord Configurations ---");
  const baseAccord: Car = new BaseCar("Honda Accord", 27000.00);
  const accordLuxury: Car = new NavigationSystem(new LeatherSeats(baseAccord));
  console.log(`  ${accordLuxury.getDescription()}`);
  console.log(`  Total: $${accordLuxury.getCost().toFixed(2)}`);
}

main();
```

### Language-Specific Notes

- **True interface via `interface`** — TypeScript's `interface` keyword provides a true contract that is enforced at compile time. Unlike the JavaScript implementation which uses a base class that throws at runtime, TypeScript catches missing method implementations during compilation. The `implements Car` clause on each class guarantees the shape is correct.

- **`readonly` and `private` modifiers** — The `private readonly` declaration on `BaseCar`'s fields ensures that once constructed, the model name and base cost cannot be changed. This aligns with the pattern's intent: decorators add behavior, not modify intrinsic state. TypeScript's `private` is enforced at compile time (unlike JavaScript's `#` which is enforced at runtime).

- **`protected` access on `CarFeature.car`** — The `protected` modifier on the `car` field allows subclasses to access the wrapped car reference. This is necessary because TypeScript's abstract class needs the field accessible to subclasses. The field is not accessible from outside the class hierarchy.

- **`static readonly` for feature constants** — Feature names and costs are declared `private static readonly` on each decorator. This makes them compile-time constants, inaccessible from outside the class, and immutable. Each decorator owns its own constants, supporting the Single Responsibility Principle.

- **Generic decorator base** — The `CarFeature` abstract class in TypeScript could be made generic (`abstract class CarFeature<T extends Car> implements Car`) to allow type-safe access to decorator-specific methods when needed, though this implementation keeps it simple for clarity.

- **Comparison with JavaScript** — The TypeScript version adds compile-time safety without any runtime overhead (TypeScript compiles to JavaScript, so the private/protected/readonly annotations are erased at runtime). The `protected car` field is accessible at runtime unlike JavaScript's truly-private `#car` field, which means the TypeScript version is slightly less encapsulated at runtime.
