## Intent

The Visitor pattern lets you define new operations on a hierarchy of objects without modifying the classes of those objects. It achieves this by separating the algorithm from the object structure on which it operates. The pattern models a double-dispatch mechanism: the operation that gets executed depends on both the type of the Visitor and the type of the Element it visits.

In the stock market domain, a portfolio contains various financial instruments -- stocks, bonds, ETFs, options -- each with its own data structure. You need to run different analyses across all instruments: calculate total portfolio value, compute dividend yield, assess risk exposure, generate tax reports. Without the Visitor pattern, each new analysis would require modifying every instrument class, violating the Open/Closed Principle and creating tight coupling between the analysis logic and the instrument hierarchy.

## The Problem Without This Pattern

Imagine you manage a stock portfolio containing `Stock` and `Bond` objects. You need four operations: calculate total value, compute annual dividends, assess risk level, and generate a summary report.

Without the Visitor pattern, you might add a method for each operation directly to every class:

```
class Stock:
    method calculate_value()    // added for feature 1
    method calculate_dividend() // added for feature 2
    method assess_risk()        // added for feature 3
    method generate_report()    // added for feature 4

class Bond:
    method calculate_value()    // added for feature 1
    method calculate_dividend() // added for feature 2
    method assess_risk()        // added for feature 3
    method generate_report()    // added for feature 4
```

This approach breaks down quickly:

- **Violates Open/Closed Principle:** Every new analysis forces you to modify every existing instrument class. After 20 operations, each class becomes bloated with 20+ unrelated methods.
- **Scattered logic:** The same calculation (e.g., total value) is spread across `Stock.calculate_value()`, `Bond.calculate_value()`, and so on. If the calculation changes, you must update every class.
- **Hard to add new types:** Adding a new instrument type like `Option` requires implementing all existing operations -- a high barrier to extension.
- **Poor separation of concerns:** A `Stock` class should model stock data, not host tax-reporting or risk-assessment logic. Mixing analysis algorithms into domain objects violates Single Responsibility Principle.

The Visitor pattern solves all of these: it keeps domain objects clean and lets you add new operations as separate Visitor classes without touching the Element hierarchy.

### Participants

1. **Visitor** (interface) -- Declares visit methods for each concrete Element type. Method names follow the convention `visit<ElementType>`.
2. **ConcreteVisitor** -- Implements each visit method to define the operation on that Element type. Each ConcreteVisitor encapsulates one algorithm.
3. **Element** (interface) -- Declares the `accept(visitor)` method that takes a Visitor as an argument.
4. **ConcreteElement** -- Implements `accept(visitor)` by calling the appropriate visit method on the visitor, passing itself as the argument: `visitor.visit(this)`.
5. **ObjectStructure** -- A collection of Elements. Provides a way to iterate over its elements and call `accept` on each one. Can be a simple list or a complex data structure like a portfolio tree.

### How Double Dispatch Works

The key insight of the Visitor pattern is **double dispatch**: which operation runs depends on both the Visitor type (first dispatch via the `visit` method called) and the Element type (second dispatch via the `accept` method that calls back the correctly typed visit method).

Without double dispatch, a client calling `visitor.visit(element)` would only dispatch on the Visitor's type. The `accept` method on each Element ensures the specific overloaded `visit<ElementType>` method is called, achieving dispatch on both types.

## Pseudocode -- Stock Market Example

```
// PSEUDOCODE -- Visitor Pattern in Stock Market context
// Participants labeled with [ROLE]

[ELEMENT INTERFACE]
interface FinancialInstrument {
    method accept(Visitor v)
}

[CONCRETE ELEMENTS]
class Stock implements FinancialInstrument {
    // State
    ticker: string
    companyName: string
    shares: int
    pricePerShare: float
    dividendYield: float  // e.g., 0.035 = 3.5%

    constructor(ticker, name, shares, price, yield) {
        this.ticker = ticker
        this.companyName = name
        this.shares = shares
        this.pricePerShare = price
        this.dividendYield = yield
    }

    [ACCEPT METHOD -- dispatches back to Visitor]
    method accept(Visitor v) {
        v.visitStock(this)
    }
}

class Bond implements FinancialInstrument {
    // State
    name: string
    faceValue: float
    couponRate: float  // e.g., 0.05 = 5%
    yearsToMaturity: int

    constructor(name, faceValue, rate, years) {
        this.name = name
        this.faceValue = faceValue
        this.couponRate = rate
        this.yearsToMaturity = years
    }

    method accept(Visitor v) {
        v.visitBond(this)
    }
}

[VISITOR INTERFACE]
interface Visitor {
    method visitStock(Stock stock)
    method visitBond(Bond bond)
}

[CONCRETE VISITOR 1 -- Value Calculator]
class ValueVisitor implements Visitor {
    totalValue: float = 0.0

    method visitStock(Stock s) {
        this.totalValue += s.shares * s.pricePerShare
    }

    method visitBond(Bond b) {
        this.totalValue += b.faceValue
    }

    method getTotalValue(): float {
        return this.totalValue
    }
}

[CONCRETE VISITOR 2 -- Dividend Calculator]
class DividendVisitor implements Visitor {
    totalDividend: float = 0.0

    method visitStock(Stock s) {
        // Only stocks pay dividends
        this.totalDividend += s.shares * s.pricePerShare * s.dividendYield
    }

    method visitBond(Bond b) {
        // Bonds pay coupon interest instead
        this.totalDividend += b.faceValue * b.couponRate
    }

    method getTotalDividend(): float {
        return this.totalDividend
    }
}

[CONCRETE VISITOR 3 -- Risk Assessor]
class RiskVisitor implements Visitor {
    results: List<string> = []

    method visitStock(Stock s) {
        // Stocks are inherently higher risk
        if s.dividendYield > 0.04 {
            this.results.add(s.ticker + ": LOW RISK -- established dividend payer")
        } else {
            this.results.add(s.ticker + ": HIGH RISK -- low or no dividend")
        }
    }

    method visitBond(Bond b) {
        // Government bonds are lower risk
        if b.yearsToMaturity > 10 {
            this.results.add(b.name + ": MEDIUM RISK -- long maturity")
        } else {
            this.results.add(b.name + ": LOW RISK -- short maturity")
        }
    }

    method getResults(): List<string> {
        return this.results
    }
}

[CONCRETE VISITOR 4 -- Report Generator]
class ReportVisitor implements Visitor {
    lines: List<string> = []

    method visitStock(Stock s) {
        value = s.shares * s.pricePerShare
        income = value * s.dividendYield
        this.lines.add("| " + s.ticker.ljust(6) + "| " +
                       s.companyName.ljust(20) + "| " +
                       s.shares.toString().rjust(5) + " | $" +
                       s.pricePerShare.toString().rjust(8) + " | $" +
                       value.toString().rjust(10) + " | $" +
                       income.toString().rjust(8) + " |")
    }

    method visitBond(Bond b) {
        income = b.faceValue * b.couponRate
        this.lines.add("| " + b.name.ljust(6) + "| " +
                       "Bond" + " " .repeat(16) + "| " +
                       b.yearsToMaturity.toString().rjust(2) + "yr | $" +
                       b.faceValue.toString().rjust(8) + " | $" +
                       b.faceValue.toString().rjust(10) + " | $" +
                       income.toString().rjust(8) + " |")
    }

    method getReport(): string {
        return "Portfolio Report:\n" +
               "| Ticker | Name" + " ".repeat(16) + "| Qty/Mat | Price/Val  | Mkt Value  | Income    |\n" +
               "|--------+----------------------+---------+-----------+------------+-----------|\n" +
               lines.join("\n") + "\n"
    }
}

[OBJECT STRUCTURE -- Portfolio]
class Portfolio {
    instruments: List<FinancialInstrument> = []

    method add(instrument: FinancialInstrument) {
        this.instruments.add(instrument)
    }

    method acceptAll(Visitor v) {
        for each instrument in this.instruments {
            instrument.accept(v)
        }
    }
}

[CLIENT CODE]
portfolio = new Portfolio()
portfolio.add(new Stock("AAPL", "Apple Inc.", 100, 175.50, 0.005))
portfolio.add(new Stock("GOOGL", "Alphabet Inc.", 50, 140.25, 0.004))
portfolio.add(new Stock("MSFT", "Microsoft Corp.", 80, 380.75, 0.007))
portfolio.add(new Bond("US10Y", 10000, 0.043, 10))
portfolio.add(new Bond("US30Y", 5000, 0.048, 30))

// ValueVisitor
valueCalc = new ValueVisitor()
portfolio.acceptAll(valueCalc)
print("Total Portfolio Value: $" + valueCalc.getTotalValue())

// DividendVisitor
divCalc = new DividendVisitor()
portfolio.acceptAll(divCalc)
print("Annual Income: $" + divCalc.getTotalDividend())

// RiskVisitor
riskCalc = new RiskVisitor()
portfolio.acceptAll(riskCalc)
for each result in riskCalc.getResults():
    print(result)

// ReportVisitor
reporter = new ReportVisitor()
portfolio.acceptAll(reporter)
print(reporter.getReport())
```

## How It Works -- Step by Step

1. **Client creates a ConcreteVisitor.** For example, `ValueVisitor` is instantiated to calculate the total portfolio value. The visitor starts with zero accumulated state.

2. **Client calls `portfolio.acceptAll(valueVisitor)`.** The Portfolio (ObjectStructure) begins iterating over its list of FinancialInstrument elements.

3. **First Element: Stock.** For the first element (e.g., AAPL), `Stock.accept(visitor)` is called. Within this method, the Stock calls `visitor.visitStock(this)`, passing itself as the argument. The runtime dispatches to `ValueVisitor.visitStock()`.

4. **ValueVisitor processes the Stock.** Inside `visitStock()`, the visitor reads the Stock's shares and pricePerShare, computes `shares * pricePerShare`, and adds the result to its internal `totalValue` accumulator. The Stock object exposes its data but does not participate in the calculation logic.

5. **Second Element: Bond.** The Portfolio moves to the next element (e.g., US10Y Bond). `Bond.accept(visitor)` is called, which in turn calls `visitor.visitBond(this)`. The runtime dispatches to `ValueVisitor.visitBond()`.

6. **ValueVisitor processes the Bond.** Inside `visitBond()`, the visitor reads the Bond's faceValue and adds it to `totalValue`. The visitor treats Bonds differently from Stocks without the Element classes knowing about the visitor's logic.

7. **Remaining elements processed.** The loop continues for GOOGL, MSFT, and US30Y. Each element's `accept()` dispatches to the correct visit method, and the visitor accumulates results.

8. **Client retrieves the result.** After iteration completes, the client calls `valueVisitor.getTotalValue()` to obtain the aggregated total.

9. **A different algorithm runs with a new Visitor.** The client repeats steps 1-8 with `DividendVisitor`, `RiskVisitor`, or `ReportVisitor`. Each visitor carries its own algorithm and accumulates different results. None of the Element classes needed modification.

10. **Double dispatch in action.** The critical guarantee: when `instrument.accept(v)` is called, the JVM/runtime dispatches to the correct `visitStock()` or `visitBond()` method. This works because `accept()` is implemented per Element type and calls the type-specific visit method. Without this double dispatch mechanism, the runtime would have to downcast or use `instanceof` checks.

## When to Use

- **You need many unrelated operations across a stable object hierarchy.** In a stock portfolio, instruments rarely change, but analysis requirements grow constantly. Each new analysis becomes a new Visitor class.
- **The Element classes are stable but operations are volatile.** If the instrument types (Stock, Bond, ETF) are well-established but you keep adding new reports, the Visitor pattern keeps the Element classes closed for modification and the operations open for extension.
- **Operations need to accumulate state across the object structure.** A `ValueVisitor` accumulates total portfolio value across all holdings. Stateful accumulation is natural in a Visitor but awkward as separate methods on each Element.
- **You want to centralize related logic.** All risk-assessment logic lives in one `RiskVisitor` class instead of being scattered across `Stock.assessRisk()`, `Bond.assessRisk()`, and so on. This makes the algorithm easier to understand, test, and change.
- **The object structure contains multiple disparate types with unrelated interfaces.** Stocks and Bonds share little common data (Stock has ticker and shares; Bond has face value and maturity). A Visitor accommodates each type's unique interface in separate visit methods.

## When NOT to Use

- **The Element hierarchy changes frequently.** If you regularly add new instrument types (e.g., adding Options, Futures, Commodities), every existing Visitor must be updated with a new `visitOption()` or `visitFuture()` method. This violates the Open/Closed Principle in the Visitor hierarchy. The pattern works best when Elements are stable and operations are what change.
- **A simple function or loop suffices.** If you only need one operation, writing a Visitor interface is over-engineering. `for each instrument: if instrument is Stock: ... else if instrument is Bond: ...` is simpler for a single operation.
- **The number of Element types is very large.** A Visitor interface with 50 visit methods becomes unwieldy. The pattern works best with a small, stable set of Element types (typically 3-8).
- **Operations must be added without knowing about the Element internals.** Visitors often need access to Element-specific data that could break encapsulation. If Elements must expose internal state only for Visitors, the pattern encourages a leaky abstraction.
- **You need to frequently add new operations AND new Element types.** The pattern does not handle simultaneous variation in both dimensions gracefully. Consider using a pattern-based dispatch table or pattern matching instead.

## Real-World Occurrences

- **Java Compiler / AST Visitors (Java):** The Java compiler's Abstract Syntax Tree uses the Visitor pattern extensively. `TreeVisitor` defines visit methods for each AST node type (variable declarations, method calls, expressions). New compiler passes (type checking, optimization, code generation) are implemented as visitors over the stable AST structure.
- **.NET Expression Trees (C#):** The `ExpressionVisitor` class in `System.Linq.Expressions` allows you to traverse and modify expression trees. Each expression node type (BinaryExpression, MethodCallExpression, ConstantExpression) has a corresponding `Visit` method. LINQ providers implement custom visitors to translate C# expressions into SQL or other target languages.
- **Roslyn / C# Compiler SyntaxWalker (C#):** Microsoft's Roslyn compiler platform provides `CSharpSyntaxWalker` and `SyntaxVisitor` classes. These let you traverse and analyze the syntax tree of C# code. Code analyzers and refactoring tools implement custom visitors to detect patterns or transform code.
- **Python's `ast` module (Python):** The `ast.NodeVisitor` class lets you walk Python's abstract syntax tree. You override `visit_ClassName` methods for each node type you care about. Tools like linters, formatters (Black, Ruff), and code analyzers use this pattern to inspect and transform Python source code.
- **Apache Wicket (Java):** The Wicket web framework uses a Visitor pattern in its component hierarchy. `IVisitor<T>` allows traversing the component tree. Different visitors handle rendering, validation, and state management across the page's component graph.
- **Rust's `syn` crate (Rust):** The `syn` crate for parsing Rust source code provides a `Visit` trait with visit methods for each AST node (item, expression, pattern, type). Procedural macros implement custom visitors to inspect and transform Rust syntax trees.

## Comparison with Related Patterns

| Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Iterator** | Iterator provides a way to traverse elements without exposing the underlying structure. Visitor adds operations to elements during traversal. | If you only need to walk the structure and read data, use Iterator. If you need to perform complex operations that differ per element type during traversal, use Visitor. |
| **Strategy** | Strategy lets you swap an algorithm at runtime, but the algorithm typically works on a single type. Visitor defines an algorithm per element type in a type hierarchy. | If your algorithm works uniformly on all elements (e.g., apply a discount percentage to all products), use Strategy. If the algorithm differs per element type (e.g., calculate value for stocks differently than for bonds), use Visitor. |
| **Command** | Command encapsulates a request as an object. It operates on a single receiver. Visitor encapsulates multiple operations across multiple receiver types. | If you need to parameterize clients with a single operation (e.g., "buy stock"), use Command. If you need to perform a family of operations across a type hierarchy (e.g., "value all portfolio holdings"), use Visitor. |
| **Composite** | Composite lets you treat individual objects and compositions uniformly. Visitor is often applied on top of a Composite structure to add operations. | If you build the element hierarchy itself (files and folders, tree nodes), use Composite as the structure. Add Visitor on top when you need to add operations without modifying the Composite classes. |

## When the Visitor Pattern Shines

The Visitor pattern is at its best when you have a well-defined, stable class hierarchy and an ever-growing set of operations you want to perform on that hierarchy. It is the pattern of compilers, linters, analyzers, and formatters -- any system where the data structure is stable but the processing logic evolves rapidly.

In the stock market domain, consider a trading platform that supports Stocks, Bonds, ETFs, Options, and Futures. The instrument types are stable -- they are defined by the exchange and rarely change. But the operations on them multiply rapidly: valuation, risk analysis, tax calculation, margin requirement, compliance checking, P&L reporting, scenario simulation, dividend projection, and more. The Visitor pattern allows each of these to be developed, tested, and deployed independently as a Visitor class, without ever touching the instrument classes.

The pattern does have a learning curve -- double dispatch is not intuitive for many developers -- and it introduces complexity that is justified only when operations genuinely outnumber element types. But in the right context, it produces systems that are remarkably extensible, maintainable, and well-organized. The cost of adding a new operation drops to zero changes in existing code: just write a new Visitor class.
