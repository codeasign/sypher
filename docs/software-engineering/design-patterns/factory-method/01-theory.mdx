## Intent

Define an interface for creating an object, but let subclasses decide which class to instantiate. The Factory Method pattern lets a class defer instantiation to subclasses. In the Banking domain, this means your core banking workflow code never needs to know the exact type of account it is creating — it delegates the creation decision to a dedicated factory method, keeping the system open for new account types without modifying existing code.

## The Problem Without This Pattern

Imagine you are building a banking application that opens different types of accounts. Without the Factory Method pattern, every piece of client code that needs an account must repeat the same creation logic. This leads to duplicated if-else chains scattered across your codebase, making the system brittle and hard to extend.

Consider a typical account opening workflow. A customer can open a SavingsAccount, a CheckingAccount, or a LoanAccount. Every service that needs to create an account — the onboarding service, the customer portal, the mobile app API, the batch processing system — must know how to construct each concrete type.

Here is what the broken approach looks like in pseudocode:

```python
# Problem: Account creation logic duplicated across the codebase
```

What breaks when you add a new account type, such as `MoneyMarketAccount` or `CertificateOfDeposit`?

- You must find every single location in the codebase that creates accounts.
- You must add a new branch to every if-else chain.
- The risk of missing one location is high, leading to runtime errors.
- The creation logic is tightly coupled to each client, making unit testing difficult.
- The code violates the Open/Closed Principle — the system is not open for extension without modification.

The Factory Method pattern solves this by encapsulating the creation logic behind a single abstraction, so that adding a new account type requires only one change: a new concrete product class and a new concrete factory.

The participants in this structure are:

- **Product (Account):** The interface or abstract class that defines the operations all concrete products must implement. In the banking example, Account declares methods such as `deposit()`, `withdraw()`, `get_balance()`, and `get_account_type()`.

- **Concrete Products (SavingsAccount, CheckingAccount, LoanAccount):** Concrete implementations of the Account interface. Each encapsulates the specific behavior, validation rules, and fee structures associated with that account type.

- **Creator (AccountFactory):** The abstract class that declares the factory method `create_account()`. The Creator may also provide a default implementation of the factory method, or it may delegate entirely to subclasses. The Creator contains business logic that operates on Product objects, relying on the factory method to produce the correct concrete type.

- **Concrete Creators (SavingsAccountFactory, CheckingAccountFactory, LoanAccountFactory):** Subclasses of the Creator that override the factory method to instantiate the specific Concrete Product. Each Concrete Creator is responsible for constructing and configuring one account type.

## Pseudocode — Banking Example

The following pseudocode demonstrates a complete implementation of the Factory Method pattern for a banking account creation system. Each participant is annotated with its role.

```python
# [PRODUCT] The interface all account types must implement
from abc import ABC, abstractmethod
from datetime import datetime
from decimal import Decimal


class Account(ABC):
    """Product interface for all bank account types."""

    def __init__(self):
        self.account_number = None
        self.customer = None
        self.balance = Decimal("0.00")
        self.created_at = datetime.now()
        self.is_active = False

    @abstractmethod
    def deposit(self, amount: Decimal) -> None:
        """Deposit funds into the account."""
        pass

    @abstractmethod
    def withdraw(self, amount: Decimal) -> None:
        """Withdraw funds from the account."""
        pass

    @abstractmethod
    def get_balance(self) -> Decimal:
        """Return the current account balance."""
        pass

    @abstractmethod
    def get_account_type(self) -> str:
        """Return a string identifying the account type."""
        pass

    def activate(self) -> None:
        """Activate the account after creation."""
        self.is_active = True
        print(f"Account {self.account_number} activated.")

    def deactivate(self) -> None:
        """Deactivate the account."""
        self.is_active = False


# [CONCRETE PRODUCT] Savings account implementation
class SavingsAccount(Account):
    """Concrete Product: Savings account with interest and minimum balance."""

    def __init__(self):
        super().__init__()
        self.interest_rate = Decimal("0.000")
        self.minimum_balance = Decimal("0.00")

    def deposit(self, amount: Decimal) -> None:
        if amount <= Decimal("0"):
            raise ValueError("Deposit amount must be positive.")
        self.balance += amount
        print(f"Deposited ${amount:.2f} into Savings Account {self.account_number}.")

    def withdraw(self, amount: Decimal) -> None:
        if amount <= Decimal("0"):
            raise ValueError("Withdrawal amount must be positive.")
        if self.balance - amount < self.minimum_balance:
            raise ValueError(
                f"Cannot withdraw. Minimum balance of ${self.minimum_balance:.2f} "
                f"must be maintained."
            )
        self.balance -= amount
        print(f"Withdrew ${amount:.2f} from Savings Account {self.account_number}.")

    def get_balance(self) -> Decimal:
        return self.balance

    def get_account_type(self) -> str:
        return "SAVINGS"

    def apply_interest(self) -> None:
        """Apply monthly interest based on the annual rate."""
        monthly_rate = self.interest_rate / Decimal("12")
        interest = self.balance * monthly_rate
        self.balance += interest
        print(f"Applied interest of ${interest:.2f} to Savings Account.")

    def set_interest_rate(self, rate: Decimal) -> None:
        self.interest_rate = rate

    def set_minimum_balance(self, min_bal: Decimal) -> None:
        self.minimum_balance = min_bal


# [CONCRETE PRODUCT] Checking account implementation
class CheckingAccount(Account):
    """Concrete Product: Checking account with overdraft protection."""

    def __init__(self):
        super().__init__()
        self.overdraft_limit = Decimal("0.00")
        self.monthly_fee = Decimal("0.00")
        self.transaction_count = 0

    def deposit(self, amount: Decimal) -> None:
        if amount <= Decimal("0"):
            raise ValueError("Deposit amount must be positive.")
        self.balance += amount
        self.transaction_count += 1
        print(f"Deposited ${amount:.2f} into Checking Account {self.account_number}.")

    def withdraw(self, amount: Decimal) -> None:
        if amount <= Decimal("0"):
            raise ValueError("Withdrawal amount must be positive.")
        available = self.balance + self.overdraft_limit
        if amount > available:
            raise ValueError(
                f"Withdrawal of ${amount:.2f} exceeds available balance of "
                f"${available:.2f} (balance: ${self.balance:.2f}, "
                f"overdraft: ${self.overdraft_limit:.2f})."
            )
        self.balance -= amount
        self.transaction_count += 1
        print(f"Withdrew ${amount:.2f} from Checking Account {self.account_number}.")

    def get_balance(self) -> Decimal:
        return self.balance

    def get_account_type(self) -> str:
        return "CHECKING"

    def apply_overdraft_fee(self) -> None:
        """Apply a fee if the balance is negative after a withdrawal."""
        if self.balance < Decimal("0"):
            overdraft_fee = Decimal("35.00")
            self.balance -= overdraft_fee
            print(f"Overdraft fee of ${overdraft_fee:.2f} applied.")

    def set_overdraft_limit(self, limit: Decimal) -> None:
        self.overdraft_limit = limit

    def set_monthly_fee(self, fee: Decimal) -> None:
        self.monthly_fee = fee


# [CONCRETE PRODUCT] Loan account implementation
class LoanAccount(Account):
    """Concrete Product: Loan account with principal, interest, and term."""

    def __init__(self):
        super().__init__()
        self.principal = Decimal("0.00")
        self.interest_rate = Decimal("0.000")
        self.loan_term_months = 0
        self.monthly_payment = Decimal("0.00")
        self.payments_made = 0

    def deposit(self, amount: Decimal) -> None:
        """Make a payment toward the loan."""
        if amount <= Decimal("0"):
            raise ValueError("Payment amount must be positive.")
        self.balance -= amount
        self.payments_made += 1
        print(f"Payment of ${amount:.2f} applied to Loan Account {self.account_number}.")

    def withdraw(self, amount: Decimal) -> None:
        """Loan disbursement — only allowed once at origination."""
        if self.payments_made > 0:
            raise ValueError("Cannot withdraw from a loan after disbursement.")
        if amount > self.principal:
            raise ValueError(
                f"Disbursement of ${amount:.2f} exceeds principal of "
                f"${self.principal:.2f}."
            )
        self.balance -= amount
        print(f"Loan disbursement of ${amount:.2f} for Loan Account {self.account_number}.")

    def get_balance(self) -> Decimal:
        """Return the outstanding loan balance (positive = owed)."""
        return abs(self.balance)

    def get_account_type(self) -> str:
        return "LOAN"

    def apply_penalty(self, days_overdue: int) -> None:
        """Apply a late payment penalty."""
        penalty_rate = Decimal("0.05")
        penalty = self.monthly_payment * penalty_rate * Decimal(str(days_overdue))
        self.balance -= penalty
        print(f"Late penalty of ${penalty:.2f} applied to Loan Account.")

    def set_principal(self, principal: Decimal) -> None:
        self.principal = principal
        self.balance = -principal

    def set_interest_rate(self, rate: Decimal) -> None:
        self.interest_rate = rate

    def set_loan_term_months(self, months: int) -> None:
        self.loan_term_months = months
        monthly_rate = self.interest_rate / Decimal("12")
        if monthly_rate == Decimal("0"):
            self.monthly_payment = self.principal / Decimal(str(months))
        else:
            rate_factor = (Decimal("1") + monthly_rate) ** Decimal(str(months))
            self.monthly_payment = (
                self.principal * monthly_rate * rate_factor
            ) / (rate_factor - Decimal("1"))


# [CREATOR] Abstract factory that declares the factory method
class AccountFactory(ABC):
    """Creator: Declares the factory method for creating accounts."""

    def create_account(self, customer, account_number: str) -> Account:
        """Factory method — subclasses override this to create specific accounts."""
        account = self._create_account_impl()
        account.customer = customer
        account.account_number = account_number
        account.created_at = datetime.now()
        self._configure_account(account)
        account.activate()
        return account

    @abstractmethod
    def _create_account_impl(self) -> Account:
        """Factory method implementation — subclasses provide the concrete type."""
        pass

    def _configure_account(self, account: Account) -> None:
        """Hook method — subclasses can override to configure the account."""
        pass


# [CONCRETE CREATOR] Creates SavingsAccount instances
class SavingsAccountFactory(AccountFactory):
    """Concrete Creator: Creates and configures SavingsAccount instances."""

    def __init__(self, interest_rate: Decimal, minimum_balance: Decimal):
        self.interest_rate = interest_rate
        self.minimum_balance = minimum_balance

    def _create_account_impl(self) -> Account:
        return SavingsAccount()

    def _configure_account(self, account: Account) -> None:
        if isinstance(account, SavingsAccount):
            account.set_interest_rate(self.interest_rate)
            account.set_minimum_balance(self.minimum_balance)


# [CONCRETE CREATOR] Creates CheckingAccount instances
class CheckingAccountFactory(AccountFactory):
    """Concrete Creator: Creates and configures CheckingAccount instances."""

    def __init__(self, overdraft_limit: Decimal, monthly_fee: Decimal):
        self.overdraft_limit = overdraft_limit
        self.monthly_fee = monthly_fee

    def _create_account_impl(self) -> Account:
        return CheckingAccount()

    def _configure_account(self, account: Account) -> None:
        if isinstance(account, CheckingAccount):
            account.set_overdraft_limit(self.overdraft_limit)
            account.set_monthly_fee(self.monthly_fee)


# [CONCRETE CREATOR] Creates LoanAccount instances
class LoanAccountFactory(AccountFactory):
    """Concrete Creator: Creates and configures LoanAccount instances."""

    def __init__(self, interest_rate: Decimal, loan_term_months: int):
        self.interest_rate = interest_rate
        self.loan_term_months = loan_term_months

    def _create_account_impl(self) -> Account:
        return LoanAccount()

    def _configure_account(self, account: Account) -> None:
        if isinstance(account, LoanAccount):
            account.set_interest_rate(self.interest_rate)
            account.set_loan_term_months(self.loan_term_months)


# [CLIENT] Code that uses the factory — no knowledge of concrete types
class AccountOpeningService:
    """Client: Uses AccountFactory without knowing concrete product types."""

    def __init__(self, savings_factory: AccountFactory,
                 checking_factory: AccountFactory,
                 loan_factory: AccountFactory):
        self.factories = {
            "savings": savings_factory,
            "checking": checking_factory,
            "loan": loan_factory,
        }

    def open_account(self, customer, account_type: str, account_number: str) -> Account:
        factory = self.factories.get(account_type)
        if factory is None:
            raise ValueError(f"Unsupported account type: {account_type}")
        return factory.create_account(customer, account_number)


# Example usage
if __name__ == "__main__":
    # Set up factories with configuration
    savings_factory = SavingsAccountFactory(
        interest_rate=Decimal("0.045"),
        minimum_balance=Decimal("500.00"),
    )
    checking_factory = CheckingAccountFactory(
        overdraft_limit=Decimal("1000.00"),
        monthly_fee=Decimal("12.99"),
    )
    loan_factory = LoanAccountFactory(
        interest_rate=Decimal("0.085"),
        loan_term_months=60,
    )

    # Client code — no if-else chains
    service = AccountOpeningService(savings_factory, checking_factory, loan_factory)

    customer = {"name": "Jane Doe", "id": "CUST-001"}

    savings = service.open_account(customer, "savings", "ACC-1001")
    checking = service.open_account(customer, "checking", "ACC-1002")
    loan = service.open_account(customer, "loan", "ACC-2001")

    # Use the accounts polymorphically
    for account in [savings, checking, loan]:
        print(f"  Type: {account.get_account_type()}, Balance: ${account.get_balance():.2f}")
```

## How It Works — Step by Step

The Factory Method pattern decouples object creation from client code. Here is the complete flow, traced through the banking example.

**Step 1 — The client requests an account.**

The `AccountOpeningService.open_account()` method receives a customer identifier and an account type string such as `"savings"`. The client does not know — and does not need to know — which concrete class implements the account.

**Step 2 — The client resolves the appropriate factory.**

The `AccountOpeningService` looks up the correct factory from its dictionary of registered factories. For `"savings"`, it retrieves the `SavingsAccountFactory` instance. This lookup replaces the if-else chain that would otherwise be necessary.

**Step 3 — The client calls the factory method.**

The client invokes `factory.create_account(customer, account_number)`. This call is polymorphic — the client does not know which factory subclass it is talking to. The `AccountFactory` base class provides the template for the creation workflow.

**Step 4 — The factory method creates the concrete product.**

Inside `create_account()`, the base class calls the abstract `_create_account_impl()` method. The `SavingsAccountFactory` override instantiates a `SavingsAccount` object. This is the moment of decoupling: the creation decision is deferred to the subclass.

**Step 5 — The factory configures and activates the product.**

After the product is created, the base class sets common properties such as `customer` and `account_number`. It then calls the hook method `_configure_account()`, which the `SavingsAccountFactory` uses to set the interest rate and minimum balance. Finally, the base class calls `activate()` on the account, which sets its status to active and logs the activation.

**Step 6 — The factory returns the product to the client.**

The fully constructed and configured `SavingsAccount` instance is returned to the client through the `Account` interface. The client interacts with the account using only the methods declared in the `Account` abstract class — `deposit()`, `withdraw()`, `get_balance()`, and `get_account_type()`.

**Step 7 — The client uses the product polymorphically.**

The client code can treat all accounts uniformly. When iterating over a collection of accounts, each one behaves according to its concrete type without the client needing to check the type explicitly. A `SavingsAccount` applies interest rules, a `CheckingAccount` enforces overdraft limits, and a `LoanAccount` tracks payments — all through the same interface.

**Step 8 — Adding a new type requires no changes to client code.**

When a new account type such as `MoneyMarketAccount` is introduced, you create a new `MoneyMarketAccount` class that implements `Account`, and a new `MoneyMarketAccountFactory` that extends `AccountFactory`. You register the new factory in the `AccountOpeningService` factory dictionary. None of the existing client code needs to change.

## When to Use

Use the Factory Method pattern when the following conditions apply to your Banking system.

- **A class cannot anticipate the class of objects it must create.** For example, a batch import process that reads account type indicators from a file and must construct the corresponding account objects without knowing the types at compile time.

- **A class wants its subclasses to specify the objects it creates.** For example, a base `ReportingService` that generates account statements but delegates the creation of the statement formatter to subclasses such as `PDFStatementService` or `CSVStatementService`.

- **You want to localize the knowledge of which concrete class gets created.** For example, all account creation logic is centralized in factory subclasses rather than scattered across the onboarding service, mobile API, teller workstation, and batch processor.

- **You are using a framework that provides reusable base classes, and you need to instantiate your own objects within the framework's workflow.** For example, a banking workflow engine that calls a factory method to create transaction handlers, allowing your application to supply `DomesticTransferHandler`, `InternationalTransferHandler`, and `WireTransferHandler` implementations.

- **You need to encapsulate complex construction logic that varies by product type.** For example, a `LoanAccountFactory` that calculates amortization schedules, validates debt-to-income ratios, and applies risk-based pricing during construction, while a `SavingsAccountFactory` performs simpler configuration.

- **You want to enforce a consistent creation workflow across all product types.** For example, ensuring that every account — regardless of type — goes through the same sequence of validation, configuration, activation, and notification steps.

## When NOT to Use

The Factory Method pattern can introduce unnecessary complexity when applied to situations that do not require it. Avoid the pattern in these scenarios.

- **You only have one concrete product type.** If your application currently creates only `SavingsAccount` objects and has no plans to support additional account types, a factory method adds an abstraction layer with no benefit. Introduce it when the second product type emerges.

- **The object creation logic is trivial.** If constructing the object requires only a simple constructor call with no configuration, validation, or instantiation logic, a factory method is overkill. A simple constructor call or a static factory method on the class itself is sufficient.

- **The product types never change.** If your domain model is stable and you are confident that no new account types will be added, the pattern adds indirection without corresponding value. The pattern shines when the set of product types is open for extension.

- **You need to create a small, fixed set of related objects.** If you have exactly two or three product types that are instantiated in a single location, the overhead of defining separate factory classes for each type may outweigh the benefits. A simple factory function or an in-line conditional may be more practical.

- **The pattern is applied prematurely for speculative future requirements.** Do not introduce a factory method hierarchy based on a guess that new product types might be needed someday. Apply the pattern when the actual requirement for a new product type emerges and the duplication in the existing code becomes apparent.

- **Your object creation is already handled by a dependency injection container.** In modern Banking applications using frameworks like Spring or ASP.NET Core, the DI container often handles object creation and lifecycle management. Adding a factory method on top of DI can create redundant abstraction layers.

## Real-World Occurrences

The Factory Method pattern appears throughout standard libraries, frameworks, and enterprise applications. Here are notable examples from different ecosystems.

**Java — java.util.Collection.iterator()**

The Java Collections Framework uses the Factory Method pattern through the `iterator()` method. Every collection class — `ArrayList`, `HashSet`, `TreeMap`, `LinkedList` — provides its own implementation of `iterator()` that returns a type-specific `Iterator` instance. The client code calls `iterator()` on a `Collection` reference without knowing whether it is iterating over a list, a set, or a map. This is the Factory Method pattern applied to iteration.

**Spring Framework — org.springframework.beans.factory.BeanFactory**

The Spring `BeanFactory` and its subinterfaces such as `ApplicationContext` are implementations of the Factory Method pattern. The `getBean()` method is a factory method that returns bean instances. Subclasses of `ApplicationContext` — `ClassPathXmlApplicationContext`, `AnnotationConfigApplicationContext`, `GenericWebApplicationContext` — provide different strategies for creating and configuring beans. The client code calls `getBean()` without knowing how the bean is constructed or which concrete implementation is returned.

**Python — datetime.date.today() and related constructors**

Python's `datetime` module uses the Factory Method pattern through class methods such as `date.today()`, `date.fromtimestamp()`, `date.fromordinal()`, and `datetime.fromisoformat()`. Each of these class methods is a factory method that creates a `date` or `datetime` instance using a different construction strategy. The `date` class itself is the Creator, and the class methods serve as factory methods that return concrete `date` instances.

**JavaScript / Node.js — ReadableStream.from()**

The Node.js stream module uses the Factory Method pattern through the `Readable.from()` and `Readable.fromWeb()` methods. These static factory methods create `Readable` stream instances from various input sources — arrays, iterables, async iterables, or web `ReadableStream` objects. The caller does not need to know which internal stream implementation is used; the factory method selects the appropriate one based on the input type.

**Java — javax.xml.parsers.DocumentBuilderFactory**

The Java XML parsing API uses the Factory Method pattern through `DocumentBuilderFactory.newInstance()` and `DocumentBuilderFactory.newDocumentBuilder()`. The `newInstance()` method is a factory method that creates a platform-specific `DocumentBuilderFactory` implementation. The `newDocumentBuilder()` method on the factory is itself a factory method that creates a `DocumentBuilder` instance. This layered application of the pattern allows the XML parsing code to be completely decoupled from the underlying parser implementation.

**.NET — System.Net.Http.HttpClientFactory**

The .NET `IHttpClientFactory` interface in the `Microsoft.Extensions.Http` namespace is an implementation of the Factory Method pattern. The `CreateClient()` method returns `HttpClient` instances. Different implementations of `IHttpClientFactory` — such as the default `DefaultHttpClientFactory` and the typed-client variants — provide different strategies for constructing and configuring `HttpClient` objects, including connection pooling, timeout policies, and handler pipeline configuration.

## Comparison with Related Patterns

The following table shows how the Factory Method pattern relates to other creational patterns and when to choose each one for your Banking application.

| Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Abstract Factory** | Creates families of related products instead of a single product. The factory itself has multiple factory methods, one for each product in the family. | Choose Abstract Factory when your Banking system needs to create sets of related objects — such as a complete account package including the account, a statement generator, and a fee calculator for each account type. |
| **Template Method** | Defines the skeleton of an algorithm, letting subclasses override specific steps. Factory Method is a special case of Template Method focused on object creation. | Choose Template Method when you need to define a multi-step algorithm — such as a loan approval workflow — and let subclasses customize individual steps without changing the overall structure. |
| **Prototype** | Creates new objects by cloning an existing prototype instance rather than calling a constructor. Avoids subclassing the Creator. | Choose Prototype when object creation is expensive and you want to copy an existing account configuration — such as creating a new savings account with the same interest rate and fee structure as an existing one. |
| **Builder** | Separates the construction of a complex object from its representation, allowing the same construction process to create different representations. | Choose Builder when constructing an account requires many configuration steps, optional parameters, and validation rules — such as building a custom loan product with a tailored interest rate, term, amortization schedule, and collateral details. |
| **Simple Factory** | A single factory class with a static method that creates products based on a parameter. Not a true GoF pattern; it violates the Open/Closed Principle. | Choose Simple Factory for small, stable product families where the set of account types is unlikely to grow. Introduce the full Factory Method when the Simple Factory's if-else chain becomes a maintenance burden. |
| **Singleton** | Ensures a class has exactly one instance and provides a global access point to it. Often used to implement factory classes. | Choose Singleton when you need exactly one instance of a factory — such as a single `AccountFactoryRegistry` that all account opening services share across the application. Combine Singleton with Factory Method by making the factory itself a Singleton. |
