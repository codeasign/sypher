The Factory Method pattern in Java 17+ leverages modern language features — sealed interfaces, records, and pattern matching — to deliver type-safe, extensible object creation. In this Banking domain implementation, an abstract `AccountFactory` declares the factory method `createAccount(String owner, BigDecimal balance)`, and concrete subclasses instantiate `SavingsAccount`, `CheckingAccount`, and `LoanAccount`. Each account type is a `record` that implements the sealed `Account` interface, guaranteeing exhaustive pattern matching at compile time and eliminating the need for error-prone `if-else` chains.

## How the Implementation Works

### Class Diagram (Pseudocode)

```text
sealed interface Account permits SavingsAccount, CheckingAccount, LoanAccount {
    String owner();
    BigDecimal balance();
    Account withdraw(BigDecimal amount);
    Account deposit(BigDecimal amount);
    Account monthlyMaintenance();
    default String statementLine() { ... }
}

record SavingsAccount(String owner, BigDecimal balance, double interestRate)
    implements Account { ... }

record CheckingAccount(String owner, BigDecimal balance,
                       BigDecimal overdraftLimit, BigDecimal monthlyFee)
    implements Account { ... }

record LoanAccount(String owner, BigDecimal balance,
                   double annualRate, BigDecimal minimumPayment)
    implements Account { ... }

abstract class AccountFactory {
    abstract Account createAccount(String owner, BigDecimal initialBalance);
    Account openAndProcess(String owner, BigDecimal amount) {
        Account acc = createAccount(owner, amount);
        System.out.println("  Statement: " + acc.statementLine());
        return acc;
    }
}

class SavingsAccountFactory extends AccountFactory {
    Account createAccount(String owner, BigDecimal initialBalance) {
        return new SavingsAccount(owner, initialBalance, 3.5);
    }
}

class CheckingAccountFactory extends AccountFactory {
    Account createAccount(String owner, BigDecimal initialBalance) {
        return new CheckingAccount(owner, initialBalance,
                                   new BigDecimal("500.00"),
                                   new BigDecimal("12.95"));
    }
}

class LoanAccountFactory extends AccountFactory {
    Account createAccount(String owner, BigDecimal initialBalance) {
        return new LoanAccount(owner, initialBalance, 7.2,
                               new BigDecimal("50.00"));
    }
}
```

### OOP Concepts Applied

| Concept | How It Is Applied |
|---|---|
| **Abstraction** | The `Account` interface exposes only `owner()`, `balance()`, `withdraw()`, `deposit()`, and `monthlyMaintenance()` — clients depend on this contract, not on any concrete record. The `AccountFactory` abstract class declares `createAccount()` without revealing how each product is constructed. |
| **Encapsulation** | Each `record` bundles its data (owner, balance, product-specific fields) with the behavior that operates on that data. Account records own their validation logic (`withdraw` checks sufficient funds before returning a new instance). |
| **Inheritance** | `SavingsAccountFactory`, `CheckingAccountFactory`, and `LoanAccountFactory` inherit the `openAndProcess()` template method from `AccountFactory` and override only `createAccount()`. |
| **Polymorphism** | The client code (`main`) holds a `Map<String, AccountFactory>` and selects a factory by key. The single line `factory.createAccount(owner, amount)` produces different product types depending on the factory's runtime class. Pattern matching in `switch` dispatches per-account behavior without explicit type checks. |

### SOLID Principles Applied

| Principle | How It Is Applied |
|---|---|
| **Single Responsibility** | Each factory class has exactly one reason to change: the construction logic for its specific product type. `SavingsAccountFactory` only knows how to build `SavingsAccount` records. Account records handle only their own immutable state transitions. |
| **Open/Closed** | Adding a new account type (e.g., `MoneyMarketAccount`) requires creating a new record and a new factory subclass. No existing code in `AccountFactory`, the product interface, or the driver's factory registry needs modification. |
| **Liskov Substitution** | Every concrete product record satisfies the `Account` contract. Every concrete factory's `createAccount()` returns an `Account` that passes all contract-implied invariants (non-null owner, non-negative balance where applicable). The driver treats all factories uniformly. |
| **Interface Segregation** | `Account` declares only banking-essential operations. No client is forced to depend on methods it does not use. Statement formatting is provided as a `default` method, keeping the core contract lean while offering convenience. |
| **Dependency Inversion** | The client depends on the abstract `Account` interface and the abstract `AccountFactory` class — never on `SavingsAccount`, `CheckingAccount`, or `LoanAccount` directly. Concrete product records are instantiated solely by their respective factory subclasses. |

### Account (Product / Sealed Interface)
```java
// OOP: Abstraction — clients depend only on this contract.
// SOLID: ISP — only essential banking methods; DIP — high-level
//        policy depends on abstraction.
sealed interface Account permits SavingsAccount, CheckingAccount, LoanAccount {

    String owner();
    BigDecimal balance();

    /**
     * Returns a NEW Account record with the updated balance.
     * Immutable — the original record is never modified.
     * OOP: Encapsulation — validation is inside the contract.
     * SOLID: SRP — the withdraw concern is here, not in a service class.
     */
    Account withdraw(BigDecimal amount);

    /**
     * Returns a new Account record with the amount added.
     */
    Account deposit(BigDecimal amount);

    /**
     * Monthly maintenance logic. Each concrete record knows its own rules.
     * Savings: no fee if balance >= 300.00, else 5.00 fee.
     * Checking: deducts monthlyFee regardless of balance.
     * Loan: adds accrued interest and requires minimum payment.
     */
    Account monthlyMaintenance();

    /**
     * SOLID: ISP-friendly convenience — a default method so all
     * records get a consistent statement line without repeating code.
     */
    default String statementLine() {
        return String.format("%-12s | %-20s | $%10.2f",
            getClass().getSimpleName().replace("Account", ""),
            owner(),
            balance());
    }
}
```

### SavingsAccount (Concrete Product)
```java
// OOP: Encapsulation — data + behavior in one immutable unit.
// SOLID: LSP — fulfills every Account contract requirement.
record SavingsAccount(String owner, BigDecimal balance,
                      double interestRate) implements Account {

    @Override
    public Account withdraw(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            System.out.println("  [SAVINGS] Withdrawal amount must be positive.");
            return this;
        }
        if (amount.compareTo(balance()) > 0) {
            System.out.printf("  [SAVINGS] Insufficient funds. Available: $%.2f%n",
                balance());
            return this;
        }
        BigDecimal newBalance = balance().subtract(amount)
            .setScale(2, RoundingMode.HALF_EVEN);
        System.out.printf("  [SAVINGS] Withdrew $%.2f. New balance: $%.2f%n",
            amount, newBalance);
        return new SavingsAccount(owner(), newBalance, interestRate());
    }

    @Override
    public Account deposit(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            System.out.println("  [SAVINGS] Deposit amount must be positive.");
            return this;
        }
        BigDecimal newBalance = balance().add(amount)
            .setScale(2, RoundingMode.HALF_EVEN);
        System.out.printf("  [SAVINGS] Deposited $%.2f. New balance: $%.2f%n",
            amount, newBalance);
        return new SavingsAccount(owner(), newBalance, interestRate());
    }

    /**
     * Savings: no maintenance fee if balance >= 300.00.
     * Otherwise a $5.00 fee is deducted.
     */
    @Override
    public Account monthlyMaintenance() {
        if (balance().compareTo(new BigDecimal("300.00")) >= 0) {
            System.out.println("  [SAVINGS] No fee — balance above $300.00 minimum.");
            return this;
        }
        BigDecimal fee = new BigDecimal("5.00");
        BigDecimal newBalance = balance().subtract(fee)
            .setScale(2, RoundingMode.HALF_EVEN);
        System.out.printf("  [SAVINGS] $5.00 maintenance fee deducted. Balance: $%.2f%n",
            newBalance);
        return new SavingsAccount(owner(), newBalance, interestRate());
    }
}
```

### CheckingAccount (Concrete Product)
```java
// OOP: Encapsulation — owns overdraftLimit, monthlyFee, and
//        validation logic.
// SOLID: LSP — satisfies Account contract; OCP — behavior is
//        sealed, no modification needed for new account types.
record CheckingAccount(String owner, BigDecimal balance,
                       BigDecimal overdraftLimit,
                       BigDecimal monthlyFee) implements Account {

    @Override
    public Account withdraw(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            System.out.println("  [CHECKING] Withdrawal amount must be positive.");
            return this;
        }
        BigDecimal maxAvailable = balance().add(overdraftLimit());
        if (amount.compareTo(maxAvailable) > 0) {
            System.out.printf("  [CHECKING] Exceeds overdraft limit of $%.2f. "
                    + "Available with overdraft: $%.2f%n",
                overdraftLimit(), maxAvailable);
            return this;
        }
        BigDecimal newBalance = balance().subtract(amount)
            .setScale(2, RoundingMode.HALF_EVEN);
        System.out.printf("  [CHECKING] Withdrew $%.2f. New balance: $%.2f%n",
            amount, newBalance);
        return new CheckingAccount(owner(), newBalance,
            overdraftLimit(), monthlyFee());
    }

    @Override
    public Account deposit(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            System.out.println("  [CHECKING] Deposit amount must be positive.");
            return this;
        }
        BigDecimal newBalance = balance().add(amount)
            .setScale(2, RoundingMode.HALF_EVEN);
        System.out.printf("  [CHECKING] Deposited $%.2f. New balance: $%.2f%n",
            amount, newBalance);
        return new CheckingAccount(owner(), newBalance,
            overdraftLimit(), monthlyFee());
    }

    /**
     * Checking: deducts the monthly fee unconditionally.
     * If balance goes negative, the overdraft limit still applies.
     */
    @Override
    public Account monthlyMaintenance() {
        BigDecimal newBalance = balance().subtract(monthlyFee())
            .setScale(2, RoundingMode.HALF_EVEN);
        BigDecimal maxAvailable = newBalance.add(overdraftLimit());
        if (maxAvailable.compareTo(BigDecimal.ZERO) < 0) {
            System.out.println("  [CHECKING] Overdraft exceeded after fee. Account frozen.");
            return this;
        }
        System.out.printf("  [CHECKING] Monthly fee of $%.2f deducted. Balance: $%.2f%n",
            monthlyFee(), newBalance);
        return new CheckingAccount(owner(), newBalance,
            overdraftLimit(), monthlyFee());
    }
}
```

### LoanAccount (Concrete Product)
```java
// OOP: Encapsulation — interest accrual and payment logic are
//        self-contained within the record.
// SOLID: LSP — maintains the Account contract. The balance
//        represents the outstanding debt (positive number owed).
//        A "deposit" is a payment that reduces the debt.
record LoanAccount(String owner, BigDecimal balance,
                   double annualRate,
                   BigDecimal minimumPayment) implements Account {

    @Override
    public Account withdraw(BigDecimal amount) {
        System.out.println("  [LOAN] Withdrawals not permitted on a loan account.");
        return this;
    }

    /**
     * A "deposit" on a loan is a payment — it reduces the
     * outstanding debt. If the payment exceeds the remaining
     * balance, the loan is paid off and the excess is refunded.
     */
    @Override
    public Account deposit(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            System.out.println("  [LOAN] Payment amount must be positive.");
            return this;
        }
        BigDecimal newBalance = balance().subtract(amount);
        if (newBalance.compareTo(BigDecimal.ZERO) < 0) {
            // Overpayment — refund the excess.
            BigDecimal refund = newBalance.abs();
            newBalance = BigDecimal.ZERO;
            System.out.printf("  [LOAN] Loan paid off. Refunding overpayment of $%.2f%n",
                refund);
        }
        newBalance = newBalance.setScale(2, RoundingMode.HALF_EVEN);
        System.out.printf("  [LOAN] Payment of $%.2f received. Remaining debt: $%.2f%n",
            amount, newBalance);
        return new LoanAccount(owner(), newBalance, annualRate(), minimumPayment());
    }

    /**
     * Loan: adds monthly interest (annualRate / 12) to the
     * outstanding debt, increasing what the customer owes.
     */
    @Override
    public Account monthlyMaintenance() {
        BigDecimal monthlyRate = BigDecimal.valueOf(annualRate() / 100.0 / 12.0);
        BigDecimal interest = balance().multiply(monthlyRate)
            .setScale(2, RoundingMode.HALF_EVEN);
        BigDecimal newBalance = balance().add(interest)
            .setScale(2, RoundingMode.HALF_EVEN);
        System.out.printf("  [LOAN] Interest accrued: $%.2f. Total debt: $%.2f%n",
            interest, newBalance);
        System.out.printf("  [LOAN] Minimum payment due: $%.2f%n", minimumPayment());
        return new LoanAccount(owner(), newBalance, annualRate(), minimumPayment());
    }
}
```

### AccountFactory (Abstract Creator)
```java
// OOP: Abstraction — declares the factory method contract.
// SOLID: DIP — client depends on this abstraction, not on
//        concrete subclasses.
static abstract class AccountFactory {

    /**
     * The factory method. Subclasses override this to instantiate
     * the specific Account type they are responsible for.
     * OOP: Polymorphism — the return type is Account, but the
     *      runtime type depends on the factory subclass.
     * SOLID: OCP — new products mean a new factory subclass, not
     *        modification of existing factories.
     */
    abstract Account createAccount(String owner, BigDecimal initialBalance);

    /**
     * Template method — calls the factory method to create the
     * account, then logs the initial statement.
     * SOLID: SRP — factory methods handle creation; this method
     *        handles the creation workflow.
     */
    Account openAndProcess(String owner, BigDecimal amount) {
        System.out.println("\n--- Opening account for " + owner + " ---");
        Account account = createAccount(owner, amount);
        System.out.println("  Initial: " + account.statementLine());
        return account;
    }
}
```

### Main / Driver
```java
// SOLID: DIP — every dependency is an abstract type (Account,
//        AccountFactory). No concrete product is referenced
//        by name in the orchestration logic.
public static void main(String[] args) {
    System.out.println("═══════════════════════════════════════════════════");
    System.out.println("  Factory Method Pattern — Banking Domain (Java)");
    System.out.println("═══════════════════════════════════════════════════");

    // Register concrete factories in a Map — DIP in action.
    // The driver never imports SavingsAccount, CheckingAccount, or
    // LoanAccount directly. It only knows Account and AccountFactory.
    Map<String, AccountFactory> factoryRegistry = new LinkedHashMap<>();
    factoryRegistry.put("savings", new SavingsAccountFactory());
    factoryRegistry.put("checking", new CheckingAccountFactory());
    factoryRegistry.put("loan", new LoanAccountFactory());

    // ── Alice opens a Savings account with $1,000.00 ──
    Account aliceSavings = factoryRegistry.get("savings")
        .openAndProcess("Alice Johnson", new BigDecimal("1000.00"));

    // Perform savings-specific operations
    System.out.println("\n  -- Alice's Monthly Operations --");
    aliceSavings = aliceSavings.withdraw(new BigDecimal("200.00"));
    aliceSavings = aliceSavings.monthlyMaintenance();
    System.out.println("  Final:   " + aliceSavings.statementLine());

    // ── Bob opens a Checking account with $2,500.00 ──
    Account bobChecking = factoryRegistry.get("checking")
        .openAndProcess("Bob Smith", new BigDecimal("2500.00"));

    // Perform checking-specific operations
    System.out.println("\n  -- Bob's Monthly Operations --");
    bobChecking = bobChecking.withdraw(new BigDecimal("2700.00"));
    bobChecking = bobChecking.deposit(new BigDecimal("500.00"));
    bobChecking = bobChecking.monthlyMaintenance();
    System.out.println("  Final:   " + bobChecking.statementLine());

    // ── Carol opens a Loan account with $50,000.00 principal ──
    Account carolLoan = factoryRegistry.get("loan")
        .openAndProcess("Carol Davis", new BigDecimal("50000.00"));

    // Perform loan-specific operations
    System.out.println("\n  -- Carol's Monthly Operations --");
    carolLoan = carolLoan.monthlyMaintenance();
    carolLoan = carolLoan.deposit(new BigDecimal("800.00"));
    System.out.println("  Final:   " + carolLoan.statementLine());

    // ── Summary using pattern matching (Java 17+ preview or 21+) ──
    System.out.println("\n═══════════════════════════════════════════════════");
    System.out.println("  End-of-Month Summary");
    System.out.println("═══════════════════════════════════════════════════");

    // The switch is exhaustive because Account is sealed — the
    // compiler knows there are exactly three concrete subtypes.
    for (Account acc : new Account[]{aliceSavings, bobChecking, carolLoan}) {
        String typeLabel = switch (acc) {
            case SavingsAccount s  -> "Savings Account";
            case CheckingAccount c -> "Checking Account";
            case LoanAccount l     -> "Loan Account";
        };
        System.out.printf("  %-20s | %s%n", typeLabel, acc.statementLine());
    }
    System.out.println("═══════════════════════════════════════════════════\n");
}
}
```

## Language-Specific Notes

- **Sealed types enforce exhaustive matching.** The `sealed interface Account` keyword restricts which records may implement it (`permits SavingsAccount, CheckingAccount, LoanAccount`). The compiler verifies that all `switch` patterns on `Account` are exhaustive — if a new record is added to the `permits` clause and not handled in every `switch`, the code will not compile. This is a compile-time safety net that runtime checks cannot match.

- **Records eliminate boilerplate.** Each concrete product is a `record` — the constructor, accessors (`owner()`, `balance()`), `equals()`, `hashCode()`, and `toString()` are synthesized by the compiler. The `monthlyMaintenance()` and `withdraw()` methods return a **new** record instance rather than mutating state, because records are immutable by design. This aligns with functional programming best practices and makes concurrency safe by default.

- **Pattern matching replaces `instanceof`/`cast` chains.** The driver's summary loop uses `switch` with type patterns (`case SavingsAccount s -> ...`) instead of the old `if (acc instanceof SavingsAccount)` / `(SavingsAccount) acc` idiom. The compiler verifies that the switch is exhaustive — because `Account` is sealed, no `default` branch is needed (though one can be added for a fallback).

- **`BigDecimal` for financial precision.** All monetary values use `java.math.BigDecimal` with `RoundingMode.HALF_EVEN` (banker's rounding) rather than `double` or `float`. Floating-point types cannot represent decimal fractions like $0.01 exactly, leading to rounding errors in financial applications. `BigDecimal` guarantees exact decimal arithmetic.

- **Default methods keep the interface lean.** `statementLine()` is a `default` method in the `Account` interface — concrete records inherit it automatically without being forced to implement it. This satisfies the Interface Segregation Principle (ISP): the core contract stays small, while convenience methods are provided as opt-in defaults.

- **The factory method is inherently Open/Closed.** To add a `MoneyMarketAccount`, write the record and a `MoneyMarketAccountFactory extends AccountFactory`. Register the new factory in the driver's `factoryRegistry` — no existing class is modified. The OCP is satisfied at the architectural level without frameworks or dependency injection containers.
