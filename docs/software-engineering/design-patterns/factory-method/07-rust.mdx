The Factory Method pattern in Rust uses traits to define the product interface and factory behavior, leveraging Rust's trait system and `Box<dyn Trait>` for runtime polymorphism. In this Banking domain example, `Account` is the product trait, three concrete structs implement it (`SavingsAccount`, `CheckingAccount`, `LoanAccount`), and `AccountFactory` is the creator that encapsulates instantiation logic behind a static method. Rust's ownership model and lack of classical inheritance make the Factory Method trait-based approach especially natural.

## How the Implementation Works

### Class Diagram (Pseudocode)

```text
trait Account {
    +account_number(): &str
    +account_holder(): &str
    +balance(): f64
    +account_type(): &str
    +deposit(amount: f64): Result<f64, String>
    +withdraw(amount: f64): Result<f64, String>
    +describe(): String
}

struct SavingsAccount {
    -account_number: String
    -account_holder: String
    -balance: f64
    -interest_rate: f64
}

struct CheckingAccount {
    -account_number: String
    -account_holder: String
    -balance: f64
    -overdraft_limit: f64
}

struct LoanAccount {
    -account_number: String
    -account_holder: String
    -balance: f64
    -interest_rate: f64
    -loan_term_months: u32
}

SavingsAccount ..|> Account
CheckingAccount ..|> Account
LoanAccount ..|> Account

enum AccountType { Savings, Checking, Loan }

struct AccountFactory

AccountFactory ..> AccountType : uses
AccountFactory ..> Account : creates

AccountFactory::create_account(type, holder, initial, number): Box<dyn Account>
```

### OOP Concepts Applied

| Concept | Rust Implementation |
|---|---|
| **Abstraction** | The `Account` trait defines a pure interface — no implementation details are exposed. Concrete structs must implement every method. |
| **Encapsulation** | Struct fields are private (default in Rust). Public accessor methods control read access. Business logic (`deposit`, `withdraw`) enforces invariants internally. |
| **Inheritance** | Rust has no class inheritance. Instead, structs implement traits (`impl Account for SavingsAccount`), providing behavior reuse through trait methods with default implementations. |
| **Polymorphism** | `Box<dyn Account>` enables runtime polymorphism. The factory returns a trait object, and callers invoke methods through the trait vtable without knowing the concrete type. |

### SOLID Principles Applied

| Principle | How It Is Applied |
|---|---|
| **SRP** | Each struct owns one responsibility: `SavingsAccount` manages savings rules, `CheckingAccount` manages checking rules, `LoanAccount` manages loan rules. `AccountFactory` owns only creation logic. |
| **OCP** | Adding a new account type (e.g., `MoneyMarketAccount`) requires only a new struct and a new `impl Account for MoneyMarketAccount` — no existing code is modified. The match arm is added to the factory. |
| **LSP** | Every struct that implements `Account` is substitutable for `dyn Account`. Callers operate on `Box<dyn Account>` and never depend on concrete types. `deposit` and `withdraw` behave consistently across all implementations. |
| **ISP** | The `Account` trait is narrowly focused on account operations. It does not mix in unrelated responsibilities like printing statements or sending notifications. |
| **DIP** | `AccountFactory` depends on the `Account` trait (abstraction), not on concrete struct types. The driver code depends on `Account` and `AccountFactory`, never on `SavingsAccount`, `CheckingAccount`, or `LoanAccount` directly. |

### Account (Product Trait)
```rust
/// Product: Defines the common interface for all account types.
/// Concrete structs implement this trait to provide account-specific
/// behavior for deposit, withdrawal, and description logic.
trait Account {
    /// Returns the unique account identifier.
    fn account_number(&self) -> &str;

    /// Returns the full name of the account holder.
    fn account_holder(&self) -> &str;

    /// Returns the current balance.
    fn balance(&self) -> f64;

    /// Returns a human-readable account type label, e.g. "Savings".
    fn account_type(&self) -> &str;

    /// Deposits funds into the account. Returns the new balance on success,
    /// or an error message if the deposit amount is invalid.
    fn deposit(&mut self, amount: f64) -> Result<f64, String>;

    /// Withdraws funds from the account. Returns the new balance on success,
    /// or an error message if the withdrawal exceeds allowed limits.
    fn withdraw(&mut self, amount: f64) -> Result<f64, String>;

    /// Returns a formatted multi-line description of the account.
    fn describe(&self) -> String;
}

/// Provides a consistent Display implementation for any Account trait object.
impl fmt::Display for dyn Account {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.describe())
    }
}
```

### SavingsAccount (Concrete Product)
```rust
/// A savings account that earns interest and has strict withdrawal rules.
/// Withdrawals are rejected if they would bring the balance below a
/// configurable minimum balance threshold (default: $50.00).
struct SavingsAccount {
    account_number: String,
    account_holder: String,
    balance: f64,
    interest_rate: f64,           // annual interest rate as a decimal (e.g. 0.035 for 3.5%)
    minimum_balance: f64,         // minimum allowable balance after withdrawal
}

impl SavingsAccount {
    /// Creates a new SavingsAccount with the given parameters.
    /// The minimum balance is set to $50.00 by default.
    fn new(account_number: &str, account_holder: &str, initial_deposit: f64) -> Self {
        SavingsAccount {
            account_number: account_number.to_string(),
            account_holder: account_holder.to_string(),
            balance: initial_deposit,
            interest_rate: 0.035,
            minimum_balance: 50.00,
        }
    }
}

impl Account for SavingsAccount {
    fn account_number(&self) -> &str {
        &self.account_number
    }

    fn account_holder(&self) -> &str {
        &self.account_holder
    }

    fn balance(&self) -> f64 {
        self.balance
    }

    fn account_type(&self) -> &str {
        "Savings"
    }

    fn deposit(&mut self, amount: f64) -> Result<f64, String> {
        if amount <= 0.0 {
            return Err(format!(
                "Deposit amount ${:.2} must be positive",
                amount
            ));
        }
        self.balance += amount;
        Ok(self.balance)
    }

    fn withdraw(&mut self, amount: f64) -> Result<f64, String> {
        if amount <= 0.0 {
            return Err(format!(
                "Withdrawal amount ${:.2} must be positive",
                amount
            ));
        }
        let projected_balance = self.balance - amount;
        if projected_balance < self.minimum_balance {
            return Err(format!(
                "Withdrawal of ${:.2} denied — maintaining ${:.2} minimum balance; \
                 projected ${:.2} is below threshold",
                amount, self.minimum_balance, projected_balance
            ));
        }
        self.balance = projected_balance;
        Ok(self.balance)
    }

    fn describe(&self) -> String {
        format!(
            "Account:        {}\n\
             Type:           {}\n\
             Holder:         {}\n\
             Balance:        ${:.2}\n\
             Interest Rate:  {:.1}%\n\
             Minimum Bal:    ${:.2}",
            self.account_number,
            self.account_type(),
            self.account_holder,
            self.balance,
            self.interest_rate * 100.0,
            self.minimum_balance,
        )
    }
}
```

### LoanAccount (Concrete Product)
```rust
/// A loan account that tracks the outstanding balance and interest rate.
/// Withdrawals (draws against the loan) increase the balance, and deposits
/// (payments) decrease it. The loan cannot be drawn beyond a maximum limit.
struct LoanAccount {
    account_number: String,
    account_holder: String,
    balance: f64,         // outstanding balance (positive means owed)
    interest_rate: f64,   // annual interest rate as a decimal (e.g. 0.068 for 6.8%)
    max_loan_amount: f64, // maximum amount that can be borrowed
    loan_term_months: u32,
}

impl LoanAccount {
    /// Creates a new LoanAccount with the given parameters.
    /// The interest rate is set to 6.8% and the loan term to 60 months by
    /// default. The max loan amount is set to the initial disbursement.
    fn new(account_number: &str, account_holder: &str, initial_disbursement: f64) -> Self {
        LoanAccount {
            account_number: account_number.to_string(),
            account_holder: account_holder.to_string(),
            balance: initial_disbursement,
            interest_rate: 0.068,
            max_loan_amount: initial_disbursement,
            loan_term_months: 60,
        }
    }
}

impl Account for LoanAccount {
    fn account_number(&self) -> &str {
        &self.account_number
    }

    fn account_holder(&self) -> &str {
        &self.account_holder
    }

    fn balance(&self) -> f64 {
        self.balance
    }

    fn account_type(&self) -> &str {
        "Loan"
    }

    /// A deposit here represents a loan payment, reducing the outstanding balance.
    fn deposit(&mut self, amount: f64) -> Result<f64, String> {
        if amount <= 0.0 {
            return Err(format!(
                "Payment amount ${:.2} must be positive",
                amount
            ));
        }
        if amount > self.balance {
            return Err(format!(
                "Payment of ${:.2} exceeds outstanding balance of ${:.2}; \
                 overpayment not allowed",
                amount, self.balance
            ));
        }
        self.balance -= amount;
        Ok(self.balance)
    }

    /// A withdrawal represents drawing additional funds against the loan.
    fn withdraw(&mut self, amount: f64) -> Result<f64, String> {
        if amount <= 0.0 {
            return Err(format!(
                "Draw amount ${:.2} must be positive",
                amount
            ));
        }
        let projected_balance = self.balance + amount;
        if projected_balance > self.max_loan_amount {
            return Err(format!(
                "Draw of ${:.2} denied — would push balance to ${:.2}, \
                 exceeding the max loan amount of ${:.2}",
                amount, projected_balance, self.max_loan_amount
            ));
        }
        self.balance = projected_balance;
        Ok(self.balance)
    }

    fn describe(&self) -> String {
        format!(
            "Account:         {}\n\
             Type:            {}\n\
             Holder:          {}\n\
             Outstanding:     ${:.2}\n\
             Interest Rate:   {:.1}%\n\
             Term:            {} months\n\
             Max Loan Amount: ${:.2}",
            self.account_number,
            self.account_type(),
            self.account_holder,
            self.balance,
            self.interest_rate * 100.0,
            self.loan_term_months,
            self.max_loan_amount,
        )
    }
}
```

### AccountFactory (Creator)
```rust
/// Enumeration of supported account types for the factory method.
enum AccountType {
    Savings,
    Checking,
    Loan,
}

/// The Creator participant. Encapsulates the instantiation logic for all
/// account types behind a single static factory method. Callers depend
/// only on the `Account` trait and `AccountType` enum.
struct AccountFactory;

impl AccountFactory {
    /// Factory method that returns a `Box<dyn Account>` for the requested
    /// account type. The caller receives a trait object and never needs to
    /// know the concrete struct that implements the behavior.
    ///
    /// # Arguments
    /// * `account_type` - The variant indicating which account to create
    /// * `holder` - Full name of the account holder
    /// * `initial_amount` - Initial deposit or disbursement amount
    /// * `account_number` - Unique account identifier string
    fn create_account(
        account_type: AccountType,
        holder: &str,
        initial_amount: f64,
        account_number: &str,
    ) -> Box<dyn Account> {
        match account_type {
            AccountType::Savings => {
                Box::new(SavingsAccount::new(account_number, holder, initial_amount))
            }
            AccountType::Checking => {
                Box::new(CheckingAccount::new(account_number, holder, initial_amount))
            }
            AccountType::Loan => {
                Box::new(LoanAccount::new(account_number, holder, initial_amount))
            }
        }
    }
}
```

### Main / Driver
```rust
fn main() {
    println!("═══════════════════════════════════════════════════");
    println!("  Bank Account Creation — Factory Method Pattern");
    println!("═══════════════════════════════════════════════════");
    println!();

    // Create three accounts of different types using the factory.
    // Alice opens a SavingsAccount with $1,000.00.
    // Bob opens a CheckingAccount with $2,500.00.
    // Carol draws a LoanAccount for $50,000.00.
    let mut accounts: Vec<Box<dyn Account>> = vec![
        AccountFactory::create_account(
            AccountType::Savings,
            "Alice Johnson",
            1000.00,
            "SAV-1001",
        ),
        AccountFactory::create_account(
            AccountType::Checking,
            "Bob Smith",
            2500.00,
            "CHK-2002",
        ),
        AccountFactory::create_account(
            AccountType::Loan,
            "Carol Williams",
            50000.00,
            "LN-3003",
        ),
    ];

    // Display initial state of every account.
    for account in accounts.iter() {
        println!("{}", account.describe());
        println!();
    }

    // ── Alice deposits $500.00 into her savings account ─────
    println!("--- Operations ---");
    println!();

    match accounts[0].deposit(500.00) {
        Ok(new_balance) => {
            println!(
                "Alice deposited $500.00. Savings balance: ${:.2}",
                new_balance
            );
        }
        Err(e) => {
            println!("Alice deposit failed: {}", e);
        }
    }

    // ── Bob withdraws $300.00 from his checking account ─────
    match accounts[1].withdraw(300.00) {
        Ok(new_balance) => {
            println!(
                "Bob withdrew $300.00. Checking balance: ${:.2}",
                new_balance
            );
        }
        Err(e) => {
            println!("Bob withdrawal failed: {}", e);
        }
    }

    // ── Carol makes a $1,000.00 loan payment ────────────────
    match accounts[2].deposit(1000.00) {
        Ok(new_balance) => {
            println!(
                "Carol paid $1,000.00. Loan outstanding: ${:.2}",
                new_balance
            );
        }
        Err(e) => {
            println!("Carol payment failed: {}", e);
        }
    }

    // ── Alice attempts to overdraw her savings ($15,000.00) ──
    match accounts[0].withdraw(15000.00) {
        Ok(new_balance) => {
            println!(
                "Alice withdrew $15,000.00. Savings balance: ${:.2}",
                new_balance
            );
        }
        Err(e) => {
            println!("Alice overdraw denied: {}", e);
        }
    }

    // ── Bob attempts to deposit a negative amount ───────────
    match accounts[1].deposit(-50.00) {
        Ok(new_balance) => {
            println!(
                "Bob deposited -$50.00 (should not happen). Balance: ${:.2}",
                new_balance
            );
        }
        Err(e) => {
            println!("Bob invalid deposit denied: {}", e);
        }
    }

    // ── Final account states ────────────────────────────────
    println!();
    println!("--- Final Account States ---");
    println!();
    for account in accounts.iter() {
        println!("{}", account.describe());
        println!();
    }

    println!("═══════════════════════════════════════════════════");
    println!("  All operations complete.");
    println!("═══════════════════════════════════════════════════");
}
```

## Language-Specific Notes

- **Traits replace interfaces and abstract classes**: Rust's `trait` system is the primary abstraction mechanism. Unlike Java or C++, a single Rust trait can provide default method implementations (similar to abstract classes) or define only method signatures (similar to interfaces). The `Account` trait in this example does both — it defines signatures that each concrete struct must implement, and it could also provide default methods if shared behavior existed.

- **`Box<dyn Trait>` enables runtime polymorphism**: Since Rust is a compiled language with no runtime type information by default, trait objects behind a `Box` (or `&` reference) enable dynamic dispatch through a vtable. The `AccountFactory` returns `Box<dyn Account>`, which erases the concrete type behind the trait interface. This is the idiomatic Rust equivalent of returning a base-class pointer in C++ or an interface reference in Java.

- **No classical inheritance — behavior reuse via composition**: Rust does not support class inheritance. Instead, the Factory Method pattern relies on trait implementation for shared interfaces and composition for shared behavior. Each concrete struct owns its fields independently, and any cross-cutting logic (e.g., validation helpers) would be provided as free functions or separate utility traits.

- **Enum-based dispatch is an alternative to trait objects**: While `Box<dyn Account>` provides open-ended polymorphism (any new struct implementing `Account` works without changing callers), an enum-based approach (`enum AnyAccount { Savings(SavingsAccount), Checking(CheckingAccount), Loan(LoanAccount) }`) offers better performance via static dispatch and avoids heap allocation. The tradeoff is that adding a new variant requires modifying the enum — violating the Open/Closed Principle. The Factory Method pattern in Rust commonly uses trait objects to preserve OCP.

- **Ownership and borrowing shape factory return types**: Returning `Box<dyn Account>` transfers ownership of the trait object to the caller. If the factory needed to return a reference, it would require lifetime annotations (`fn create_account(...) -> &'a dyn Account`). For long-lived collections of heterogeneous accounts, `Vec<Box<dyn Account>>` is the standard approach, as demonstrated in the driver code.

- **The `Result` type models business operation outcomes idiomatically**: Rust's `Result<T, E>` type replaces checked exceptions and error codes used in other languages. Each account operation (`deposit`, `withdraw`) returns a `Result<f64, String>`, where the `Ok` variant carries the new balance and the `Err` variant carries a descriptive error message. Callers must match or use `?` to handle errors, ensuring no failed operation goes unnoticed at compile time.
