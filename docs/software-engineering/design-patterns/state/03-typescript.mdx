The State pattern allows an object to alter its behavior when its internal state changes. This TypeScript implementation models a stock market order lifecycle, where an Order transitions through states such as New, Submitted, PartiallyFilled, Filled, Cancelled, and Rejected. Strict typing ensures that every state transition respects the contract, and the compiler catches mismatches between state operations and expected types at build time.

## How the Implementation Works

### Class Diagram in Pseudocode

```text
interface OrderState {
  handleSubmit(order: Order): string
  handleFill(order: Order, qty: number, price: number): string
  handleCancel(order: Order): string
  handleReject(order: Order, reason: string): string
  getStatusLabel(): string
  getAllowedOperations(): string[]
}

abstract class BaseOrderState implements OrderState {
  // Default implementations return error strings for every method.
  // Subclasses override only the operations they support.
}

class NewState extends BaseOrderState {
  handleSubmit()       вҖ” transition to SubmittedState
  handleCancel()       вҖ” transition to CancelledState
  getStatusLabel()     вҖ” "New"
  getAllowedOps()      вҖ” ["submit", "cancel"]
}

class SubmittedState extends BaseOrderState {
  handleFill(qty,prc)  вҖ” transition to PartiallyFilled or Filled
  handleCancel()       вҖ” transition to CancelledState
  handleReject(reason) вҖ” transition to RejectedState
  getStatusLabel()     вҖ” "Submitted"
  getAllowedOps()      вҖ” ["fill", "cancel"]
}

class PartiallyFilledState extends BaseOrderState {
  handleFill(qty,prc)  вҖ” transition to Filled if fully filled
  handleCancel()       вҖ” transition to CancelledState
  getStatusLabel()     вҖ” "PartiallyFilled"
  getAllowedOps()      вҖ” ["fill", "cancel"]
}

class FilledState extends BaseOrderState {     // terminal
  getStatusLabel()     вҖ” "Filled"
}

class CancelledState extends BaseOrderState {  // terminal
  getStatusLabel()     вҖ” "Cancelled"
}

class RejectedState extends BaseOrderState {   // terminal
  getStatusLabel()     вҖ” "Rejected"
}

class Order {
  private symbol, quantity, filledQuantity, price
  private state: OrderState
  submit()             delegates to state
  fill(qty, price)     delegates to state
  cancel()             delegates to state
  reject(reason)       delegates to state
  getStatus()          state.getStatusLabel()
  setState(s)          update internal state
  addFill(qty, price)  increment filled quantity
  isFullyFilled()      filledQuantity >= quantity
}
```

### OOP Concepts Applied

- **Abstraction** вҖ” The `OrderState` interface defines the contract; callers interact through the interface methods without knowledge of the concrete state class.
- **Encapsulation** вҖ” Each concrete state owns its transition rules and validation. The `Order` class uses TypeScript's `private` modifier to protect internal fields from external access.
- **Inheritance** вҖ” Concrete states extend a common abstract base class that provides default error behavior, overriding only the methods that represent valid transitions for that state.
- **Polymorphism** вҖ” The `Order` context delegates to its `state` field typed as the `OrderState` interface; each concrete state provides a unique implementation of the same method signatures.

### SOLID Principles Applied

- **Single Responsibility** вҖ” Every state class handles exactly the subset of lifecycle operations valid for its phase. `SubmittedState` manages fills, cancellations, and rejections; `NewState` only handles submission and pre-submission cancellation.
- **Open/Closed** вҖ” New states (e.g., `ExpiredState`, `StoppedState`) can be added by implementing the `OrderState` interface without modifying any existing state class or the `Order` context.
- **Liskov Substitution** вҖ” Any class implementing `OrderState` can replace another in the `Order` context without breaking correctness, because all implementations adhere to the same interface contract.
- **Interface Segregation** вҖ” The `OrderState` interface declares focused, operation-specific methods rather than a single monolithic method that branches on state type.
- **Dependency Inversion** вҖ” `Order` depends on the abstract `OrderState` interface, not on concrete state implementations. States are discovered and swapped at runtime.

### BaseOrderState (Abstract Base Class)

```typescript
// в”Җв”Җ Abstract Base Class в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
abstract class BaseOrderState implements OrderState {
  handleSubmit(order: Order): string {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot submit вҖ” order is already ${status}`;
  }
  handleFill(order: Order, quantity: number, price: number): string {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot fill вҖ” order is already ${status}`;
  }
  handleCancel(order: Order): string {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot cancel вҖ” order is already ${status}`;
  }
  handleReject(order: Order, reason: string): string {
    const status = this.getStatusLabel().toLowerCase();
    return `ERROR: Cannot reject вҖ” order is already ${status}`;
  }
  getStatusLabel(): string {
    return 'Unknown';
  }
  getAllowedOperations(): string[] {
    return [];
  }
}
```

### SubmittedState

```typescript
class SubmittedState extends BaseOrderState {
  handleFill(order: Order, quantity: number, price: number): string {
    const symbol = order.symbol;
    const remaining = order.remaining;
    if (quantity >= remaining) {
      order.addFill(remaining, price);
      order.setState(new FilledState());
      return `${symbol}: Filled ${order.filledQuantity} shares at $${price.toFixed(2)}`;
    }
    order.addFill(quantity, price);
    order.setState(new PartiallyFilledState());
    return `${symbol}: Partially filled ${order.filledQuantity}/${order.quantity} shares at $${price.toFixed(2)}`;
  }
  handleCancel(order: Order): string {
    const symbol = order.symbol;
    order.setState(new CancelledState());
    return `${symbol}: Submitted -> ${order.getStatus()}`;
  }
  handleReject(order: Order, reason: string): string {
    const symbol = order.symbol;
    order.setState(new RejectedState());
    return `${symbol}: Rejected вҖ” ${reason}`;
  }
  getStatusLabel(): string {
    return 'Submitted';
  }
  getAllowedOperations(): string[] {
    return ['fill', 'cancel'];
  }
}
```

### Order (Context)

```typescript
// в”Җв”Җ Order (Context) в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

class Order {
  private _symbol: string;
  private _quantity: number;
  private _filledQuantity: number;
  private _price: number;
  private _state: OrderState;

  constructor(symbol: string, quantity: number, price: number) {
    this._symbol = symbol;
    this._quantity = quantity;
    this._filledQuantity = 0;
    this._price = price;
    this._state = new NewState();
  }

  get symbol(): string {
    return this._symbol;
  }
  get quantity(): number {
    return this._quantity;
  }
  get filledQuantity(): number {
    return this._filledQuantity;
  }
  get price(): number {
    return this._price;
  }
  get remaining(): number {
    return this._quantity - this._filledQuantity;
  }

  setState(state: OrderState): void {
    this._state = state;
  }
  addFill(qty: number, price: number): void {
    this._filledQuantity += qty;
    this._price = price;
  }
  getStatus(): string {
    return this._state.getStatusLabel();
  }
  getAllowedOperations(): string[] {
    return this._state.getAllowedOperations();
  }
  isFullyFilled(): boolean {
    return this._filledQuantity >= this._quantity;
  }

  submit(): string {
    return this._state.handleSubmit(this);
  }
  fill(quantity: number, price: number): string {
    return this._state.handleFill(this, quantity, price);
  }
  cancel(): string {
    return this._state.handleCancel(this);
  }
  reject(reason: string): string {
    return this._state.handleReject(this, reason);
  }
}
```

## Language-Specific Notes

- TypeScript's `interface` keyword provides a first-class way to define the `OrderState` contract, and the compiler enforces that every concrete state implements all required methods with the correct signatures.
- The `abstract class` pattern (`BaseOrderState`) combined with an `interface` (`OrderState`) gives the best of both worlds: a compile-time contract and shared default behavior for terminal-state error handling.
- TypeScript's `private` access modifier on `Order` fields provides encapsulation at the type level вҖ” unlike JavaScript's `#` private fields, these are enforced only at compile time but are more familiar to developers from other typed languages.
- Strict type annotations on all method parameters and return types (`string`, `number`, `void`) make the state transition contracts explicit and self-documenting.
- The `readonly` modifier on constructor parameters could be used for `symbol`, `quantity`, and `price` to prevent accidental reassignment, keeping the design intent clear.
- Accessor methods (`get symbol()`, `get quantity()`) provide a clean read-only interface to private fields, which is essential since state objects need read access to Order data but should never mutate it directly.
