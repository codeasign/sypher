## Intent

The State pattern allows an object to alter its behavior when its internal state changes. The object will appear to change its class. In the Stock Market domain, this means a trading order changes how it responds to operations — submit, fill, cancel, reject — depending on its current lifecycle state, without any if-else chains in the order object itself. The pattern encapsulates each state's behavior into its own class, making state transitions explicit and state-specific logic easy to find, test, and modify.

## The Problem Without This Pattern

Imagine you are building a trading system that processes stock orders. An order can be New, Submitted, PartiallyFilled, Filled, Cancelled, or Rejected. Each state has different rules:

- A **New** order can be submitted or cancelled.
- A **Submitted** order can be filled, cancelled, or rejected.
- A **PartiallyFilled** order can be filled further or cancelled.
- **Filled**, **Cancelled**, and **Rejected** are terminal — no further operations allowed.

Without the State pattern, every order method uses a massive if-else chain checking the current status string. Every time a new state is added — such as `PendingApproval` — every single method must be updated to handle it, making the code fragile, hard to test, and prone to bugs.

The core problem is that state-specific behavior is scattered across multiple methods rather than being localized in one place. Adding a new state means reading through all methods to find every place the status is checked. Removing a state is equally risky because you might miss a branch that still references it. The status string itself is fragile — a typo in a string comparison ("Sumitted" vs "Submitted") silently changes behavior, and there is no compile-time safety.

Here is what the broken approach looks like in pseudocode:

```python
# Problem: All state-checking logic crammed into every method
# Every new state requires modifying all methods

class Order:
    def __init__(self, symbol: str, quantity: int, price: float):
        self.symbol = symbol
        self.quantity = quantity
        self.filled_quantity = 0
        self.price = price
        self.status = "New"  # "New", "Submitted", "PartiallyFilled",
                             # "Filled", "Cancelled", "Rejected"

    def submit(self):
        # [DUPLICATED] Status check duplicated in every method
        if self.status == "New":
            print("Submitting order...")
            self.status = "Submitted"
        elif self.status == "Submitted":
            raise Error("Order already submitted")
        elif self.status == "PartiallyFilled":
            raise Error("Order already submitted")
        elif self.status == "Filled":
            raise Error("Cannot submit — order is filled")
        elif self.status == "Cancelled":
            raise Error("Cannot submit — order was cancelled")
        elif self.status == "Rejected":
            raise Error("Cannot submit — order was rejected")
        else:
            raise Error(f"Unknown status: {self.status}")

    def fill(self, quantity: int, price: float):
        # [DUPLICATED] Same status check again
        if self.status == "New":
            raise Error("Cannot fill — order is not yet submitted")
        elif self.status == "Submitted":
            self.filled_quantity += quantity
            self.price = price
            if self.filled_quantity >= self.quantity:
                self.status = "Filled"
            else:
                self.status = "PartiallyFilled"
        elif self.status == "PartiallyFilled":
            self.filled_quantity += quantity
            if self.filled_quantity >= self.quantity:
                self.status = "Filled"
        elif self.status == "Filled":
            raise Error("Order already filled")
        elif self.status == "Cancelled":
            raise Error("Cannot fill — order was cancelled")
        elif self.status == "Rejected":
            raise Error("Cannot fill — order was rejected")

    def cancel(self):
        # [DUPLICATED] Third copy of the same status check
        if self.status == "New":
            self.status = "Cancelled"
        elif self.status == "Submitted":
            self.status = "Cancelled"
        elif self.status == "PartiallyFilled":
            self.status = "Cancelled"
        elif self.status == "Filled":
            raise Error("Cannot cancel — order already filled")
        elif self.status == "Cancelled":
            raise Error("Order already cancelled")
        elif self.status == "Rejected":
            raise Error("Cannot cancel — order was rejected")

    def reject(self, reason: str):
        # [DUPLICATED] Fourth copy
        if self.status == "New":
            raise Error("Cannot reject — order is not yet submitted")
        elif self.status == "Submitted":
            self.status = "Rejected"
        elif self.status == "PartiallyFilled":
            raise Error("Cannot reject — order is partially filled")
        elif self.status == "Filled":
            raise Error("Cannot reject — order already filled")
        elif self.status == "Cancelled":
            raise Error("Cannot reject — order was cancelled")
        elif self.status == "Rejected":
            raise Error("Order already rejected")


# Adding a new state like "PendingApproval" requires updating ALL
# four methods. One missed branch leads to a runtime bug.
```

This approach violates the Open/Closed Principle and has severe maintainability issues. Each method duplicates the same state-checking logic, adding a new state requires touching every method in the class, and the behavior for each state is scattered across multiple methods rather than being localized. The result is a class that grows without bound as states are added. Testing becomes harder because every test for a single state must navigate the if-else chains in every method, and the risk of introducing a regression in one state while fixing another is high.

In a real trading system, the consequences of a missed branch can be severe. If a `PendingApproval` state is added but the `fill()` method's if-else chain forgets to handle it, the method might silently fall through to the `else` clause and raise an `Error` — or worse, proceed with a fill on an order that should not be filled. The State pattern eliminates this entire class of bugs by making each state a first-class object with its own complete set of behavior methods.

## Structure

The State pattern organizes an object's state-dependent behavior into a set of discrete state classes. The key participants are:

- **Order (Context)**: Maintains a reference to the current `OrderState` object. It delegates all state-specific operations to this object. The Context is the object whose behavior changes as its state changes. It exposes a `setState()` method that concrete states call to trigger transitions.

- **OrderState (State interface)**: Defines the interface for all state-specific behavior. Each operation in the domain (submit, fill, cancel, reject) has a corresponding method in the interface. The interface also exposes query methods like `getStatusLabel()` and `getAllowedOperations()`.

- **Concrete States (NewState, SubmittedState, PartiallyFilledState, FilledState, CancelledState, RejectedState)**: Each concrete class implements the state-specific behavior for one lifecycle stage. Active states (NewState, SubmittedState, PartiallyFilledState) handle operations by executing domain logic and potentially transitioning to a new state. Terminal states (FilledState, CancelledState, RejectedState) reject all operations by raising errors.

## Pseudocode — Stock Market Example

The following pseudocode demonstrates the State pattern in the Stock Market domain. An `Order` (Context) delegates every operation to its current `OrderState` (State interface). Six concrete states implement the behavior for each order lifecycle stage: `NewState`, `SubmittedState`, `PartiallyFilledState`, `FilledState`, `CancelledState`, and `RejectedState`. Terminal states (Filled, Cancelled, Rejected) prevent any further operations.

The pattern works by having each concrete state object implement the full set of operations defined in the interface. When an operation is not valid for that state, the state raises an error. When an operation triggers a transition, the state calls `order.setState(new ConcreteState())` to switch to the next state. The Order class itself never checks its status — it simply forwards every call to the current state object. This delegation-based approach eliminates the need for any conditional logic in the Context class, making it a pure dispatcher.

The pseudocode below shows the complete implementation of the State pattern for the stock market order lifecycle:

```
// PSEUDOCODE — State in Stock Market domain
// Participants labeled with [ROLE]

[STATE] interface OrderState:
    method handleSubmit(order: Order): void
    method handleFill(order: Order, quantity: int, price: float): void
    method handleCancel(order: Order): void
    method handleReject(order: Order, reason: str): void
    method getStatusLabel(): str
    method getAllowedOperations(): List[str]

[CONCRETE_STATE] class NewState implements OrderState:
    method handleSubmit(order):
        print("Submitting order...")
        order.setState(new SubmittedState())
    method handleFill(order, quantity, price):
        raise Error("Cannot fill — order is not yet submitted")
    method handleCancel(order):
        print("Cancelling order...")
        order.setState(new CancelledState())
    method handleReject(order, reason):
        raise Error("Cannot reject — order is not yet submitted")
    method getStatusLabel(): return "New"
    method getAllowedOperations(): return ["submit", "cancel"]

[CONCRETE_STATE] class SubmittedState implements OrderState:
    method handleSubmit(order):
        raise Error("Order already submitted")
    method handleFill(order, quantity, price):
        order.addFill(quantity, price)
        if order.isFullyFilled():
            order.setState(new FilledState())
        else:
            order.setState(new PartiallyFilledState())
    method handleCancel(order):
        order.setState(new CancelledState())
    method handleReject(order, reason):
        print(f"Order rejected: {reason}")
        order.setState(new RejectedState())
    method getStatusLabel(): return "Submitted"
    method getAllowedOperations(): return ["fill", "cancel"]

[CONCRETE_STATE] class PartiallyFilledState implements OrderState:
    method handleSubmit(order):
        raise Error("Order already submitted")
    method handleFill(order, quantity, price):
        order.addFill(quantity, price)
        if order.isFullyFilled():
            order.setState(new FilledState())
    method handleCancel(order):
        order.setState(new CancelledState())
    method handleReject(order, reason):
        raise Error("Cannot reject — order is partially filled")
    method getStatusLabel(): return "Partially Filled"
    method getAllowedOperations(): return ["fill", "cancel"]

[CONCRETE_STATE] class FilledState implements OrderState:
    method handleSubmit(order):
        raise Error("Order is already filled — no further operations allowed")
    method handleFill(order, quantity, price):
        raise Error("Order is already filled — no further operations allowed")
    method handleCancel(order):
        raise Error("Order is already filled — no further operations allowed")
    method handleReject(order, reason):
        raise Error("Order is already filled — no further operations allowed")
    method getStatusLabel(): return "Filled"
    method getAllowedOperations(): return []

[CONCRETE_STATE] class CancelledState implements OrderState:
    method handleSubmit(order):
        raise Error("Order was cancelled — no further operations allowed")
    method handleFill(order, quantity, price):
        raise Error("Order was cancelled — no further operations allowed")
    method handleCancel(order):
        raise Error("Order was cancelled — no further operations allowed")
    method handleReject(order, reason):
        raise Error("Order was cancelled — no further operations allowed")
    method getStatusLabel(): return "Cancelled"
    method getAllowedOperations(): return []

[CONCRETE_STATE] class RejectedState implements OrderState:
    method handleSubmit(order):
        raise Error("Order was rejected — no further operations allowed")
    method handleFill(order, quantity, price):
        raise Error("Order was rejected — no further operations allowed")
    method handleCancel(order):
        raise Error("Order was rejected — no further operations allowed")
    method handleReject(order, reason):
        raise Error("Order was rejected — no further operations allowed")
    method getStatusLabel(): return "Rejected"
    method getAllowedOperations(): return []

[CONTEXT] class Order:
    fields: symbol, quantity, filledQuantity, price, state (OrderState)
    method __init__(symbol, quantity, price):
        self.state = new NewState()
    method submit():
        self.state.handleSubmit(self)
    method fill(quantity, price):
        self.state.handleFill(self, quantity, price)
    method cancel():
        self.state.handleCancel(self)
    method reject(reason):
        self.state.handleReject(self, reason)
    method getStatus():
        return self.state.getStatusLabel()
    method getAllowedOperations():
        return self.state.getAllowedOperations()
    method setState(newState):
        self.state = newState
    method addFill(quantity, price):
        self.filledQuantity += quantity
        self.price = price
    method isFullyFilled():
        return self.filledQuantity >= self.quantity

[CLIENT] class TradingSystem:
    method run():
        // Order 1: AAPL — New -> Submitted -> Filled
        order1 = Order("AAPL", 100, 175.50)
        order1.submit()
        order1.fill(100, 175.50)
        print(order1.getStatus())  // Filled

        // Order 2: GOOGL — New -> Submitted -> PartiallyFilled -> Cancelled
        order2 = Order("GOOGL", 50, 140.25)
        order2.submit()
        order2.fill(20, 140.25)
        order2.cancel()
        print(order2.getStatus())  // Cancelled

        // Order 3: MSFT — New -> Submitted -> Rejected
        order3 = Order("MSFT", 75, 378.80)
        order3.submit()
        order3.reject("Insufficient funds")
        print(order3.getStatus())  // Rejected
```

## How It Works — Step by Step

1. **An Order is created with a NewState.** The client creates an `Order("AAPL", 100, 175.50)`. The constructor initializes the `state` field to a `NewState` instance. All operations are now governed by `NewState`, which only allows submit or cancel. The Order object holds no status string — its behavior is entirely determined by which state object it references.

2. **The client calls order.submit().** The `Order` object does not check its status itself. Instead, it delegates to `self.state.handleSubmit(self)`, passing a reference to itself so the state can modify the order's state field. This delegation is the core mechanism of the State pattern — the Context never makes decisions about what behavior is appropriate; it simply forwards the request to the current state.

3. **NewState.handleSubmit transitions to SubmittedState.** Inside `NewState.handleSubmit`, the method prints "Submitting order..." and calls `order.setState(new SubmittedState())`. The Order now holds a reference to a `SubmittedState` instance. Its behavior has changed without any if-else checks in the Order class. The Order does not even know what state it is in — it only knows it has a state object that implements the `OrderState` interface.

4. **The client calls order.fill(100, 175.50).** The Order delegates to `self.state.handleFill(self, 100, 175.50)`. Since the current state is `SubmittedState`, the fill method executes the fill logic. It checks whether the order is fully filled after adding the fill quantity. This decision logic lives inside the state, not the Context.

5. **If fully filled, the state transitions to FilledState.** In this example, the order quantity was 100 and the fill quantity is 100. `SubmittedState.handleFill` calls `order.addFill(100, 175.50)`, then `order.isFullyFilled()` returns true, so it calls `order.setState(new FilledState())`. The Order is now in a terminal state. If the fill had been partial (e.g., 50 of 100 units), the state would have transitioned to `PartiallyFilledState` instead.

6. **Terminal states prevent further operations.** If the client attempts to call `order.fill()` again, the `FilledState.handleFill` method raises an error: "Order is already filled — no further operations allowed." The same applies to cancel and reject from `FilledState`. Terminal states are self-protecting — they unconditionally reject all operations. This is far more reliable than an if-else chain because the protection is guaranteed by the type system: the state object simply does not have any valid transition paths.

7. **Different lifecycle paths follow different transitions.** A second order might follow a different path: New -> Submitted -> PartiallyFilled -> Cancelled. Or New -> Submitted -> Rejected. Or New -> Cancelled (if the user cancels before submitting). Each path is defined entirely within the concrete state classes. The `Order` class itself contains zero state-transition logic — it simply delegates to whatever state it currently holds. The state transitions are completely decentralized, which is both the power and the complexity of the pattern.

8. **Adding a new state requires only one new class.** If a `PendingApprovalState` is needed, you create one new class implementing `OrderState`, define its transitions (which states it can transition to and which operations it allows), and wire up the incoming transitions from existing states (e.g., from `SubmittedState` if manual approval is needed before filling). No existing method in the `Order` class is modified. The Open/Closed Principle is preserved. The only change to existing code is modifying the states that can transition into the new state, which is a localized change.

9. **The client can query the current state without coupling to concrete states.** The `getStatus()` and `getAllowedOperations()` methods on the Order return state information without exposing the state object itself. This allows the client to display the current status, enable or disable UI buttons, or log state transitions without depending on concrete state classes. The `getAllowedOperations()` method is particularly useful in trading applications for dynamically constructing context menus or validating user actions before they are attempted.

10. **Removing or deprecating a state is equally isolated.** If `PartiallyFilledState` is no longer needed (due to a change in business rules), you remove its class and update the states that could transition into it (in this case, `SubmittedState.handleFill` would transition directly to `FilledState` regardless of fill quantity). No scattered if-else branches to find and update — the change is localized to one state transition method.

11. **The pattern naturally prevents illegal state combinations.** In the if-else approach, nothing prevents the status string from being set to an unexpected value or an invalid combination (e.g., status = "Filled" but filled_quantity = 0). With the State pattern, the state object controls what data modifications are possible. `SubmittedState.handleFill` ensures `addFill` is called correctly before transitioning. Terminal states prevent any data modification at all. This invariant enforcement is automatic and cannot be bypassed.

12. **The full system with three orders demonstrates three distinct paths.** The AAPL order (New -> Submitted -> Filled) shows a clean lifecycle that ends in a positive terminal state. The GOOGL order (New -> Submitted -> PartiallyFilled -> Cancelled) shows partial fills followed by manual cancellation. The MSFT order (New -> Submitted -> Rejected) shows the rejection path. All three paths use the same `Order` class and the same `OrderState` interface — only the concrete state instances differ. This demonstrates the core value of the State pattern: the same Context object exhibits completely different behavior depending on its current state, without any conditional logic in the Context itself.

## When to Use

- An object's behavior depends on its state and must change at runtime without conditional logic scattered across the class
- Operations have large, multipart conditional statements that depend on the object's current state, and you want to replace those conditionals with polymorphic dispatch
- State transitions are well-defined and relatively stable across the system, making the state machine feasible to model
- You want to make state-specific behavior explicit and localized in dedicated classes, each responsible for one state's complete behavior
- State-specific behavior changes frequently and needs to be maintained independently of the context, so changes to one state never affect others
- You want to avoid duplicated state-checking logic that appears in every method of a class, reducing the risk of a missed branch
- Each state has complex logic that is best tested in isolation from other states, and you want unit tests for each state class independently
- The number of states is expected to grow over time, and you want each addition to be isolated to a single new class
- In a trading system context: when order lifecycle management, trade settlement, or compliance workflows have well-defined state transitions that govern which operations are valid at each stage
- The Context class has grown large and complex due to state-checking logic, and you want to refactor it into smaller, focused classes
- You need to enforce that certain operations are only valid in certain states, and you want this enforcement to be automatic and guaranteed by the object structure rather than by conditional checks

## When NOT to Use

- There are only a few states with simple transitions that rarely change — a simple enum-based state machine is more appropriate
- State-specific behavior is minimal — just a few flag checks would suffice without the overhead of a class hierarchy
- The state transitions are highly dynamic and unpredictable at design time, making it impossible to model a fixed set of states
- The pattern introduces too many classes for what is essentially a simple finite state machine that could be implemented with a single switch statement
- A simple state machine implemented with enums and switch statements is sufficient for your use case and the number of states is small
- The state logic rarely changes and has only a couple of operations per state — the overhead of creating a class per state is not justified
- Performance is critical and the overhead of object delegation and dynamic dispatch matters (though this is rarely the bottleneck in practice)
- The state transitions are entirely data-driven and need to be configured at runtime, making a table-driven state machine a better fit
- The system already uses a well-established state machine library that handles all the complexity
- The state machine is shared across multiple Context types with different behaviors, suggesting a more generic state machine framework
- The team is not familiar with the State pattern, and the maintenance burden of the pattern exceeds the cost of the conditional logic

## Benefits and Trade-Offs

The State pattern offers several important benefits. It localizes state-specific behavior in dedicated classes, making each state easy to understand, test, and modify independently. It eliminates the massive if-else chains that would otherwise pervade every method of the Context class. It makes state transitions explicit — every transition is a `setState()` call inside a concrete state's method, making the flow of the state machine visible in the code. It follows the Open/Closed Principle, allowing new states to be added without modifying existing code (except for the states that transition into the new state).

However, the pattern also has trade-offs. It can lead to a large number of small classes, one for each state, which can make the codebase harder to navigate. It requires each state to handle all operations defined in the interface, even if most states raise errors for most operations — this means the terminal states in the order example must implement all four handle methods even though they all raise errors. The pattern ties the state machine structure to the class hierarchy, making it harder to change the state machine dynamically at runtime. For very simple state machines, the pattern can feel like over-engineering.

The key decision factor is whether the benefits of localized state behavior and the Open/Closed Principle outweigh the cost of additional classes. In trading systems where order lifecycle is critical and new states are added over time (regulatory changes, new order types, compliance requirements), the State pattern is almost always the right choice.

## Implementation Considerations

When implementing the State pattern in a trading system, several practical considerations arise that affect how the pattern is applied.

**State object creation and reuse.** In the stock market example, each state transition creates a new instance of the target state class (e.g., `new SubmittedState()`). For frequently transitioning orders, this creates many short-lived objects. If performance is a concern, states can be shared. Terminal states (FilledState, CancelledState, RejectedState) are stateless — they have no instance-specific fields — so a single shared instance can be used for all orders. This is a natural fit with the Flyweight pattern. Non-terminal states that carry no per-order data (NewState, SubmittedState) can also be shared if they do not need to store transition-specific state.

**State initialization and cleanup.** Some states may need setup or teardown logic. For example, transitioning to `SubmittedState` might log the submission timestamp, and transitioning out of `SubmittedState` might record the time spent in the submitted state. The State pattern can accommodate this by adding `onEnter(order)` and `onExit(order)` methods to the `OrderState` interface, called by the Context when transitioning between states. This is especially useful for auditing and compliance in financial systems.

**Determining state transitions.** In the stock market example, the state object itself determines the next state. This is the most common approach and keeps transition logic co-located with the behavior. However, it creates a dependency between concrete states — `SubmittedState` must know about `FilledState`, `PartiallyFilledState`, `CancelledState`, and `RejectedState`. An alternative is to centralize transition rules in the Context or in a separate transition table, which decouples the states from each other at the cost of scattering the transition logic.

**Handling invalid transitions.** The pattern naturally handles invalid operations by raising errors in the state methods. In a trading system, you might want more nuanced behavior — returning a result object with error codes, logging the attempt, or triggering a notification. The interface can be extended to return a result type rather than void, allowing states to communicate failure reasons back to the caller.

**Thread safety in concurrent systems.** In a multi-threaded trading engine, an order's state might change while an operation is being processed. The `setState` call in the Context must be thread-safe, typically by using a lock or atomic reference. The concrete state objects themselves should be immutable or stateless to avoid shared mutable state across threads.

**Testing state machines.** One of the strengths of the State pattern is testability. Each concrete state can be unit tested in isolation by creating a mock Order (Context) and verifying that the correct methods are called. Integration tests can verify that the full state machine produces the correct sequence of transitions for a given sequence of operations. In the trading domain, this means you can write tests for every valid path through the order lifecycle (New -> Submitted -> Filled) and every invalid path (New -> Fill without submit), ensuring the system handles all cases correctly. Because each state is a separate class, you can test its behavior independently without needing to navigate through other states first. For example, you can test `FilledState.handleFill` directly without first creating an Order, submitting it, and filling it — simply instantiate a `FilledState`, pass a mock Order, and verify the error is raised.

## Real-World Occurrences

- **Java's `javax.faces.lifecycle.Lifecycle`** — JavaServer Faces (JSF) lifecycle phases follow the State pattern. The six phases (RestoreView, ApplyRequestValues, ProcessValidations, UpdateModelValues, InvokeApplication, RenderResponse) each have specific behavior and transition rules. The `FacesServlet` acts as the Context, delegating request processing to the current lifecycle phase. Each phase is a state object that knows what to do and what phase comes next.

- **JavaScript's `XMLHttpRequest`** — the `readyState` property progresses through five states: UNSENT (0), OPENED (1), HEADERS_RECEIVED (2), LOADING (3), DONE (4). Each state governs which methods can be called. For example, `send()` can only be called from the OPENED state, and `abort()` is only valid after the request has been sent. Calling an invalid method raises an error, just like the State pattern's concrete states.

- **Python's `asyncio.Task`** — coroutine tasks in Python's asyncio library transition through states: PENDING, CANCELLED, FINISHED. The task's behavior for operations like `cancel()`, `result()`, and `exception()` depends on its current state. A cancelled task cannot be cancelled again, and a finished task's result is immediately available. The internal state machine is implemented using the State pattern internally.

- **.NET's `System.Net.WebRequest`** — web request lifecycle states govern which operations are available at each stage. The request progresses through states such as Created, RequestStreamObtained, ResponseReceived, and Error. Each state determines which methods (like `GetRequestStream()`, `GetResponse()`, `Abort()`) are valid.

- **Interactive Brokers API order states** — trading platform orders transition through a well-defined lifecycle: Inactive, PendingSubmit, PreSubmitted, Submitted, Filled, Cancelled, Inactive. Each state has different rules for what operations are permitted. The State pattern maps naturally to this domain because the order lifecycle is a canonical state machine.

- **TCP connection states** — the classic State pattern example. A TCP connection transitions through 11 states: LISTEN, SYN_SENT, SYN_RECEIVED, ESTABLISHED, FIN_WAIT_1, FIN_WAIT_2, CLOSE_WAIT, CLOSING, LAST_ACK, TIME_WAIT, and CLOSED. Each state has different rules for sending data, receiving data, and handling connection close. The TCP protocol specification is essentially a state machine diagram.

- **Vending machines** — a classic textbook example. A vending machine has states like Idle, HasMoney, Dispensing, SoldOut. Each state determines whether the machine accepts coins, dispenses products, or returns change. The State pattern models the machine's behavior cleanly, with each state handling the same operations (insert coin, dispense, return change) differently.

- **Java's `java.util.Scanner`** — the Scanner class for parsing input has internal states that govern which operations (hasNext, next, nextInt, close) are valid and how they behave. Once closed, all operations throw an IllegalStateException.

- **Workflow engines** — business process management systems like Camunda and Apache Airflow use the State pattern to model task and workflow lifecycle states. A task transitions through states like Created, Ready, Running, Suspended, Failed, and Completed. Each state determines which operations (retry, skip, suspend, resume) are valid and what happens when they are invoked.

- **Audio and video players** — media players typically have states such as Playing, Paused, Stopped, and Buffering. The same play button behaves differently depending on the current state: if stopped, it starts playback; if paused, it resumes; if playing, it does nothing (or restarts). This is a textbook State pattern example in GUI applications.

## Comparison with Related Patterns

| Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Strategy** | Strategy defines a family of interchangeable algorithms; the client chooses the strategy. State defines state-specific behavior; the context transitions between states automatically based on internal logic. | Choose Strategy when the algorithm varies independently of the state and the client selects which algorithm to use. Choose State when the transitions between behaviors are part of the domain logic and happen automatically based on operations. |
| **Chain of Responsibility** | Passes a request along a chain of handlers until one handles it. State dispatches a request to the current state object, which is the sole handler. | Choose Chain of Responsibility when you have multiple potential handlers for a single request and the handler is determined by the request itself. Choose State when exactly one handler is active at a time and transitions are deterministic. |
| **Flyweight** | Shares state objects across multiple contexts to save memory. State can use Flyweight internally to share state objects that have no instance-specific data. | Combine Flyweight with State when terminal states (Filled, Cancelled, Rejected) are stateless and can be shared across all order instances to reduce memory allocation overhead. |
| **Template Method** | Lets subclasses redefine specific steps of a fixed algorithm structure. State lets objects change their entire behavior by swapping state objects at runtime. | Choose Template Method when the algorithm structure is fixed but some steps vary by subclass. Choose State when the entire behavior changes per state and transitions occur at runtime based on operations. |
| **Command** | Encapsulates a request as an object for queuing, logging, or undo. State changes an object's behavior by swapping its current state object. | Choose Command when you need to parameterize, queue, log, or undo operations. Choose State when the operation's validity and behavior depend on the object's current lifecycle stage. |
| **Bridge** | Decouples an abstraction from its implementation, allowing both to vary independently. State decouples behavior from context by delegating to state objects. | Choose Bridge when you need to vary both the abstraction and its implementation independently. Choose State when the behavior variation is driven by internal state transitions. |
| **Mediator** | Centralizes communication between multiple objects. State concerns itself with the behavior of a single object across its lifecycle. | Choose Mediator when multiple objects need to coordinate their interactions. Choose State when a single object's behavior changes across its lifecycle states. |
| **Memento** | Captures and externalizes an object's internal state for later restoration. State manages internal state transitions. | Combine Memento with State when you need to save and restore the Context's state (e.g., undo order operations). Use Memento to capture the current state object, then restore it later. |
