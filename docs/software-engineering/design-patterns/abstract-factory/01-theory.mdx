## Intent

Provide an interface for creating families of related or dependent objects without specifying their concrete classes. The Abstract Factory pattern lets you produce objects that belong together as a cohesive family, ensuring that the products you create are compatible with each other. This is the pattern's defining characteristic: it is not just a factory of objects, but a factory of *families* of objects.

## The Problem Without This Pattern

Imagine you are building a banking system that supports two distinct customer tiers: Personal Banking and Business Banking. Each tier has different fee structures, different minimum balance requirements, and different account features. You need to create three account types per tier: Savings, Checking, and Loan.

Without the Abstract Factory pattern, you might write code like this:

```python
# BAD: Client code must know about every concrete combination
def create_personal_account(account_type, holder, amount):
    if account_type == "savings":
        fee = PersonalFee()
        return PersonalSavingsAccount(holder, amount, fee)
    elif account_type == "checking":
        fee = PersonalFee()
        return PersonalCheckingAccount(holder, amount, fee)
    elif account_type == "loan":
        fee = PersonalFee()
        return PersonalLoanAccount(holder, amount, fee)

def create_business_account(account_type, holder, amount):
    if account_type == "savings":
        fee = BusinessFee()
        return BusinessSavingsAccount(holder, amount, fee)
    elif account_type == "checking":
        fee = BusinessFee()
        return BusinessCheckingAccount(holder, amount, fee)
    elif account_type == "loan":
        fee = BusinessFee()
        return BusinessLoanAccount(holder, amount, fee)
```

This approach has several problems:

- **Duplication** — Every factory function repeats the same branching logic. Adding a new account type (e.g., Money Market) means updating every factory function.
- **Tight coupling** — Client code imports and instantiates concrete classes directly. Switching from Personal to Business banking requires changing call sites throughout the codebase.
- **Inconsistent families** — Nothing prevents a developer from accidentally pairing a `PersonalSavingsAccount` with a `BusinessFee`. The two product families are not grouped, so mismatched products can be combined at runtime.
- **No scalability** — Adding a third tier (e.g., Premium Banking) requires creating yet another factory function with the same branching structure, multiplied by the number of account types.

The core problem is that related products (SavingsAccount + PersonalFee, CheckingAccount + PersonalFee) have no grouping mechanism. The pattern needs a way to say: *these products belong together; use them as a set.*

## Structure

The Abstract Factory pattern organizes participants into two hierarchies: product families and factory families. The abstract factory declares creation methods for each abstract product type. Each concrete factory implements those methods to produce a specific family of compatible products.

### Participants

1. **AbstractFactory (BankFactory)** — Declares an interface for creating each abstract product type. The interface contains one creation method per product type in the family.
2. **ConcreteFactory (PersonalBankingFactory, BusinessBankingFactory)** — Implements the creation methods to produce a specific family of products. Each concrete factory produces a complete set of related products that are designed to work together.
3. **AbstractProduct (Account)** — Declares an interface for a type of product object. The Abstract Factory returns AbstractProduct references, not concrete types.
4. **ConcreteProduct (PersonalSavingsAccount, BusinessSavingsAccount, PersonalCheckingAccount, BusinessCheckingAccount, PersonalLoanAccount, BusinessLoanAccount)** — Implements the AbstractProduct interface. Each concrete product belongs to a specific family and is produced by the corresponding ConcreteFactory.
5. **Client** — Uses only the AbstractFactory and AbstractProduct interfaces. The client never instantiates concrete products directly; it receives them from the factory.

## Pseudocode — Banking Example

This pseudocode demonstrates the Abstract Factory pattern in a banking system. Two product families exist: PersonalBanking (low fees, personal account features) and BusinessBanking (higher fees, business account features). Each concrete factory creates a complete family of compatible accounts.

```
// PSEUDOCODE — Abstract Factory Pattern in Banking
// Participants labeled with [ROLE]

// ── Abstract Products ──
[ABSTRACT_PRODUCT] Account {
    abstract getDetails(): String
    abstract getBalance(): Decimal
    abstract getMonthlyFee(): Decimal
}

// ── Concrete Product Family 1: Personal Banking ──
[CONCRETE_PRODUCT] PersonalSavingsAccount implements Account {
    holder: String
    balance: Decimal
    fee: PersonalFee

    constructor(holder, deposit, fee) {
        this.holder = holder
        this.balance = deposit
        this.fee = fee
    }

    getDetails(): String
        return "Personal Savings[" + holder + "]: $" + format(balance, 2)

    getBalance(): Decimal
        return balance

    getMonthlyFee(): Decimal
        return fee.getMonthlyFee()
}

[CONCRETE_PRODUCT] PersonalCheckingAccount implements Account {
    holder: String
    balance: Decimal
    fee: PersonalFee

    constructor(holder, deposit, fee) {
        this.holder = holder
        this.balance = deposit
        this.fee = fee
    }

    getDetails(): String
        return "Personal Checking[" + holder + "]: $" + format(balance, 2) +
               ", OD limit: $" + format(fee.getOverdraftLimit(), 2)

    getBalance(): Decimal
        return balance

    getMonthlyFee(): Decimal
        return fee.getMonthlyFee()
}

[CONCRETE_PRODUCT] PersonalLoanAccount implements Account {
    holder: String
    principal: Decimal
    fee: PersonalFee

    constructor(holder, principal, fee) {
        this.holder = holder
        this.principal = principal
        this.fee = fee
    }

    getDetails(): String
        return "Personal Loan[" + holder + "]: $" + format(principal, 2) +
               " at " + fee.getAPR() + "% APR"

    getBalance(): Decimal
        return -principal

    getMonthlyFee(): Decimal
        return fee.getMonthlyFee()
}

// ── Concrete Product Family 2: Business Banking ──
[CONCRETE_PRODUCT] BusinessSavingsAccount implements Account { ... }
[CONCRETE_PRODUCT] BusinessCheckingAccount implements Account { ... }
[CONCRETE_PRODUCT] BusinessLoanAccount implements Account { ... }

// ── Abstract Factory ──
[ABSTRACT_FACTORY] BankFactory {
    abstract createSavingsAccount(holder, deposit): Account
    abstract createCheckingAccount(holder, deposit): Account
    abstract createLoanAccount(holder, principal): Account
}

// ── Concrete Factory 1: Personal Banking ──
[CONCRETE_FACTORY] PersonalBankingFactory implements BankFactory {
    createSavingsAccount(holder, deposit): Account
        return new PersonalSavingsAccount(holder, deposit, new PersonalFee())

    createCheckingAccount(holder, deposit): Account
        return new PersonalCheckingAccount(holder, deposit, new PersonalFee())

    createLoanAccount(holder, principal): Account
        return new PersonalLoanAccount(holder, principal, new PersonalFee())
}

// ── Concrete Factory 2: Business Banking ──
[CONCRETE_FACTORY] BusinessBankingFactory implements BankFactory {
    createSavingsAccount(holder, deposit): Account
        return new BusinessSavingsAccount(holder, deposit, new BusinessFee())

    createCheckingAccount(holder, deposit): Account
        return new BusinessCheckingAccount(holder, deposit, new BusinessFee())

    createLoanAccount(holder, principal): Account
        return new BusinessLoanAccount(holder, principal, new BusinessFee())
}

// ── Client ──
function main() {
    personalFactory = new PersonalBankingFactory()
    businessFactory = new BusinessBankingFactory()

    // Create personal banking accounts
    personalSavings = personalFactory.createSavingsAccount("Alice", 1000.00)
    personalChecking = personalFactory.createCheckingAccount("Bob", 2500.00)
    personalLoan = personalFactory.createLoanAccount("Carol", 50000.00)

    // Create business banking accounts
    businessSavings = businessFactory.createSavingsAccount("Acme Corp", 50000.00)
    businessChecking = businessFactory.createCheckingAccount("Globex Inc", 100000.00)
    businessLoan = businessFactory.createLoanAccount("Initech", 500000.00)

    // Display results
    display(personalSavings.getDetails())
    display(personalSavings.getMonthlyFee())
    display(businessSavings.getDetails())
    display(businessSavings.getMonthlyFee())
}
```

## How It Works — Step by Step

1. **The client selects a concrete factory.** The client decides which family of products it needs (PersonalBanking or BusinessBanking) and instantiates the corresponding concrete factory. The factory reference is stored as an `BankFactory` interface type, so the client never depends on the concrete factory class.

2. **The client calls a factory method for a specific product type.** The client calls `createSavingsAccount("Alice", 1000.00)` on the factory interface. Because the client holds a reference to an `BankFactory`, it does not know or care which concrete factory is handling the call.

3. **The concrete factory instantiates the concrete product.** The concrete factory (e.g., `PersonalBankingFactory`) creates a `PersonalSavingsAccount` object. It passes the account data along with the appropriate fee structure (a `PersonalFee` instance) to the constructor.

4. **The concrete factory returns the product as the abstract type.** The `PersonalSavingsAccount` is returned through the `Account` interface. The client sees only the `Account` type, not the concrete `PersonalSavingsAccount` type. This ensures the client does not depend on concrete classes.

5. **The client works with the product through the abstract interface.** The client calls `getDetails()`, `getBalance()`, and `getMonthlyFee()` on the `Account` reference. The correct implementation for the concrete product type executes polymorphically.

6. **The client can switch families by changing the factory.** To create a business banking account instead of personal, the client only needs to instantiate `BusinessBankingFactory` instead of `PersonalBankingFactory`. All three product types (Savings, Checking, Loan) automatically switch to the Business family. The account creation code itself does not change.

7. **The factory ensures product compatibility within a family.** Because `PersonalBankingFactory` always creates `PersonalSavingsAccount` with `PersonalFee`, and `BusinessBankingFactory` always creates `BusinessSavingsAccount` with `BusinessFee`, the products within a family are guaranteed to be compatible. A `PersonalSavingsAccount` can never be accidentally paired with a `BusinessFee`.

## When to Use

- **You need to create families of related products that must be used together.** In banking, a SavingsAccount, CheckingAccount, and LoanAccount from the same tier (Personal or Business) share consistent fee structures, interest rates, and minimum balance requirements. The Abstract Factory ensures that only compatible products are created together.

- **You want to enforce consistency across a product line.** When you have multiple product types (Savings, Checking, Loan) and multiple product families (Personal, Business, Premium), the Abstract Factory guarantees that all products from the same factory share the same family characteristics.

- **The system should be configured with one of multiple families of products.** Your application might need to support different banking tiers, different countries with different regulatory requirements, or different customer segments. The Abstract Factory lets you swap the entire product family by changing the factory at a single configuration point.

- **You want to isolate concrete classes from the client.** The client should not know which concrete account classes it is using. The Abstract Factory returns products through abstract interfaces, and the client code never imports concrete product types.

- **Adding a new family should not require changing existing client code.** When you add a `PremiumBankingFactory`, existing client code that works with `BankFactory` and `Account` interfaces can immediately use the new family without modification. This is the Open/Closed Principle in action.

## When NOT to Use

- **You only have one product type.** If you only need to create `SavingsAccount` objects (a single product type), use the Factory Method pattern instead. The Abstract Factory is designed for multiple related product types.

- **You do not expect new product families.** If your system only supports Personal Banking and adding Business Banking is not a requirement, the added abstraction of a factory interface is unnecessary overhead. A simple constructor or factory method suffices.

- **The product types in the family change frequently.** Every time you add a new product type (e.g., `MoneyMarketAccount`), you must update the `BankFactory` interface and every concrete factory. If product types change more often than families, the Abstract Factory adds maintenance burden rather than reducing it.

- **The products do not need to be compatible.** If there is no relationship between the products (e.g., SavingsAccount and LoanAccount are completely independent), grouping them in a factory serves no purpose. The Abstract Factory's value is ensuring compatibility within a family.

- **You need to create a single object.** The Abstract Factory is overkill for creating a single type of object. Use a simple factory function or the Factory Method pattern for single-product creation.

## Real-World Occurrences

1. **Java Swing Toolkit (javax.swing)** — Swing's `LookAndFeel` class is an abstract factory for creating GUI components. Each concrete look-and-feel (Metal, Windows, Motif, Nimbus) creates a family of related UI widgets (buttons, menus, text fields) that share consistent visual styling. Changing the look-and-feel swaps the entire UI family at once.

2. **.NET Framework — ADO.NET Provider Model** — The `DbProviderFactory` abstract class defines methods for creating a family of database-related objects: `DbConnection`, `DbCommand`, `DbDataAdapter`, `DbParameter`. Concrete factories like `SqlClientFactory`, `OracleClientFactory`, and `MySqlClientFactory` each produce a consistent set of objects for their respective database. Client code uses the abstract factory and never references concrete database types.

3. **Python — `unittest` / `mock` testing framework** — The `unittest.mock` module provides `Mock` and `MagicMock` classes that form a family of testing doubles. The `patch` decorator acts as an abstract factory, creating the appropriate mock objects based on the context. While not a textbook implementation, the pattern of creating a family of related testing objects (mocks, spies, stubs, fakes) that work together consistently follows the Abstract Factory idea.

4. **Rust — `winit` windowing library** — The `winit` library uses an abstract factory pattern for cross-platform window creation. The `EventLoop` and `Window` types are platform-specific products. The `EventLoop::new()` function acts as a factory that creates a family of related platform objects (Windows on Win32, X11 on Linux, AppKit on macOS). The caller receives abstract types and never handles platform-specific code.

5. **Qt Framework — `QStyle`** — Qt's `QStyle` is an abstract factory for creating platform-specific GUI elements. Each style (Fusion, Windows, macOS) creates a family of related visual elements: push buttons, scroll bars, sliders, and tree controls. The factory ensures all visual elements in a UI share the same appearance.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Factory Method** | Factory Method creates a single product type using inheritance; Abstract Factory creates multiple product types using composition. | Choose Factory Method when you need only one type of product (e.g., just creating SavingsAccount objects). Choose Abstract Factory when you need families of related products. |
| **Builder** | Builder constructs a complex object step by step and returns the final product; Abstract Factory returns products immediately. | Choose Builder when you need to construct a complex object with many configuration options (e.g., building a loan application with optional fields). Choose Abstract Factory when you need a family of related objects created as a unit. |
| **Prototype** | Prototype creates new objects by cloning existing ones; Abstract Factory creates new objects by calling factory methods. | Choose Prototype when object creation is expensive and you can cache and clone prototypes (e.g., pre-configured account templates). Choose Abstract Factory when you need to create families of objects from scratch. |
| **Singleton** | Singleton ensures a class has only one instance; Abstract Factory uses composition to create families of objects. | Choose Singleton to guarantee a single instance of a resource (e.g., a single database connection pool). The Abstract Factory is often combined with Singleton to ensure only one factory instance exists per family. |
