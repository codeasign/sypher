## Implementation

Below is a complete Python 3.11+ implementation of the Abstract Factory pattern using a banking domain. The `BankFactory` abstract class defines three creation methods, and two concrete factories (`PersonalBankingFactory` and `BusinessBankingFactory`) produce families of related account objects. Python's ABC (Abstract Base Class) module enforces the contract at runtime: any attempt to instantiate a concrete factory that has not implemented all creation methods raises a `TypeError`. The client code depends only on the `BankFactory` and `Account` abstract types.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌────────────────────────────────────────────────────────────┐
//   │   BankFactory (ABC — abstract base class)                 │
//   ├────────────────────────────────────────────────────────────┤
//   │ + create_savings_account(holder, amount) -> Account        │  ← OOP: Abstraction
//   │ + create_checking_account(holder, amount) -> Account       │     ISP: focused
//   │ + create_loan_account(holder, amount) -> Account           │     creation methods
//   └─────────────┬──────────────────────────┬──────────────────┘
//                 │ inherits                 │ inherits
//   ┌─────────────┴──────────────┐  ┌────────┴──────────────────┐
//   │ PersonalBankingFactory     │  │ BusinessBankingFactory    │
//   │ ── creates Personal        │  │ ── creates Business       │  ← OOP: Polymorphism
//   │    product family          │  │    product family          │     LSP: substitutable
//   └─────────────┬──────────────┘  └────────┬──────────────────┘
//                 │                          │
//                 ▼                          ▼
//   ┌─────────────────────────┐  ┌──────────────────────────┐
//   │ PersonalSavingsAccount  │  │ BusinessSavingsAccount   │
//   │ PersonalCheckingAccount │  │ BusinessCheckingAccount  │  ← OOP: Inheritance
//   │ PersonalLoanAccount     │  │ BusinessLoanAccount      │     (inherit Account)
//   └─────────────┬───────────┘  └────────┬─────────────────┘
//                 │                       │
//                 └────────┬──────────────┘
//                          │ inherits
//                   ┌──────┴──────────┐
//                   │ Account (ABC)   │  ← OOP: Abstraction
//                   ├─────────────────┤     ISP: minimal interface
//                   │ get_details()   │
//                   │ get_balance()   │
//                   │ get_monthly_fee │
//                   └─────────────────┘
```

#### OOP Concepts Applied

1. **Abstraction** — `BankFactory` and `Account` are ABCs (abstract base classes) that define the contracts using the `abc` module. Methods decorated with `@abstractmethod` must be implemented by subclasses.
2. **Encapsulation** — Python does not have truly private fields, but the convention of a leading underscore (`_holder`, `_balance`) indicates private implementation details. The public interface is limited to the methods defined in the ABC.
3. **Inheritance** — All concrete account classes inherit from the `Account` ABC. Both concrete factory classes inherit from the `BankFactory` ABC. Python's MRO (Method Resolution Order) handles method dispatch through the inheritance chain.
4. **Polymorphism** — The driver stores all accounts in a list typed as `list[Account]` and calls `get_details()` on each. Python's duck typing ensures the correct method runs per concrete type.

#### SOLID Principles Applied

1. **SRP** — `PersonalBankingFactory` creates only personal banking products. Each account class represents exactly one product type.
2. **OCP** — Adding a `PremiumBankingFactory` class requires no changes to the `BankFactory` ABC, existing factories, or existing account classes.
3. **LSP** — Every concrete account type can substitute `Account` without breaking the client. The driver treats all accounts uniformly through the `Account` interface.
4. **ISP** — The `Account` interface is minimal: three abstract methods. The `BankFactory` interface is similarly minimal. No class is forced to implement unused methods.
5. **DIP** — Client code depends on the `BankFactory` and `Account` abstract types, never on concrete types like `PersonalSavingsAccount`.

### Account

```python
# Abstract Factory Pattern — Python 3.11+ — Banking example
# ===================================================================
# Participants: BankFactory (AbstractFactory), PersonalBankingFactory
#   and BusinessBankingFactory (ConcreteFactory), Account (AbstractProduct),
#   PersonalSavingsAccount, PersonalCheckingAccount, PersonalLoanAccount,
#   BusinessSavingsAccount, BusinessCheckingAccount, BusinessLoanAccount
#   (ConcreteProducts)
# OOP: abstraction, encapsulation, inheritance, polymorphism
# SOLID: SRP, OCP, LSP, ISP, DIP
# ===================================================================

from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import final
# ── Abstract Product: Account ──
# OOP role: Abstraction + ISP (defines the contract for all account types)
# SOLID: ISP (minimal interface), DIP (clients depend on this)
class Account(ABC):
    """Abstract base class for all banking account products."""

    @abstractmethod
    def get_details(self) -> str:
        """Return a formatted string describing the account."""
        ...

    @abstractmethod
    def get_balance(self) -> float:
        """Return the current account balance."""
        ...

    @abstractmethod
    def get_monthly_fee(self) -> float:
        """Return the monthly maintenance fee for this account."""
        ...
```

### PersonalSavingsAccount

```python
# ── Concrete Product Family 1: Personal Banking Accounts ──
# OOP role: Inheritance (inherits Account) + Polymorphism
# SOLID: LSP (substitutable for Account), OCP (extensible), SRP (one responsibility)
@final
class PersonalSavingsAccount(Account):
    """Personal savings account with low fees and high interest."""

    def __init__(self, holder: str, initial_deposit: float) -> None:
        self._holder = holder
        self._balance = initial_deposit
        self._interest_rate = 3.5
        self._monthly_fee = 5.0

    def get_details(self) -> str:
        return (f"Personal Savings[{self._holder}]: "
                f"${self._balance:.2f} @ {self._interest_rate}% APY")

    def get_balance(self) -> float:
        return self._balance

    def get_monthly_fee(self) -> float:
        return self._monthly_fee
@final
```

### BankFactory

```python
# ── Abstract Factory: BankFactory ──
# OOP role: Abstraction + Encapsulation (encapsulates product family creation)
# SOLID: SRP (creates accounts only), DIP (clients depend on this abstraction)
class BankFactory(ABC):
    """Abstract factory that defines creation methods for account families."""

    @abstractmethod
    def create_savings_account(self, holder: str, amount: float) -> Account:
        """Create a savings account for the given holder."""
        ...

    @abstractmethod
    def create_checking_account(self, holder: str, amount: float) -> Account:
        """Create a checking account for the given holder."""
        ...

    @abstractmethod
    def create_loan_account(self, holder: str, amount: float) -> Account:
        """Create a loan account for the given holder."""
        ...
```

### Main / Driver

```python
# ── Main / Driver ──
def display_accounts(label: str, factory: BankFactory) -> None:
    """Display all account types created by the given factory."""
    print(f"=== {label} ===")
    accounts: list[Account] = [
        factory.create_savings_account("Alice", 1000.0),
        factory.create_checking_account("Bob", 2500.0),
        factory.create_loan_account("Carol", 50000.0),
    ]

    for account in accounts:
        print(f"  {account.get_details()}")
        print(f"    Balance: ${account.get_balance():.2f}, "
              f"Monthly Fee: ${account.get_monthly_fee():.2f}")
    print()


def main() -> None:
    personal_factory: BankFactory = PersonalBankingFactory()
    business_factory: BankFactory = BusinessBankingFactory()

    display_accounts("Personal Banking Accounts", personal_factory)
    display_accounts("Business Banking Accounts", business_factory)

    # Demonstrate family consistency — all accounts from the same factory
    # share the same fee structure and product family characteristics
    print("=== Cross-Family Comparison ===")
    personal_savings: Account = personal_factory.create_savings_account("Alice", 1000.0)
    business_savings: Account = business_factory.create_savings_account("Acme Corp", 50000.0)

    print(f"Personal Savings monthly fee: ${personal_savings.get_monthly_fee():.2f}")
    print(f"Business Savings monthly fee: ${business_savings.get_monthly_fee():.2f}")

    personal_loan: Account = personal_factory.create_loan_account("Carol", 50000.0)
    business_loan: Account = business_factory.create_loan_account("Initech", 500000.0)

    print(f"Personal Loan details: {personal_loan.get_details()}")
    print(f"Business Loan details: {business_loan.get_details()}")


if __name__ == "__main__":
    main()
```

### Language-Specific Notes

- **ABC module** — Python's `abc` module provides `ABC` (base class) and `@abstractmethod` (decorator) to define abstract base classes. Any class that inherits from an ABC and does not implement all abstract methods cannot be instantiated — the `TypeError` is raised at construction time. This provides runtime enforcement of the interface contract.
- **`@final` decorator** — Python 3.11+ introduced the `@final` decorator to prevent further subclassing. While not enforced at runtime by default, it signals design intent and can be checked by static type checkers like mypy and Pyright. This is useful for concrete product classes that should not be extended.
- **Type hints with `list[Account]`** — Python 3.11+ supports the `list[T]` generic syntax directly (no `from typing import List`). Type hints are used throughout to document the intended types, though they are not enforced at runtime. Tools like mypy validate them statically.
- **`__init__` pattern for encapsulation** — Python uses the `_single_underscore` convention for private attributes. This is a naming convention only — the attributes are still accessible from outside the class. True encapsulation relies on developer discipline rather than language enforcement. For true private fields, Python's `__name_mangling` (double underscore prefix) can be used, but the single underscore convention is idiomatic for most cases.
- **No interface keyword** — Python does not have an `interface` keyword. The Abstract Base Class (ABC) mechanism serves the same role. An ABC with only abstract methods is functionally equivalent to an interface in other languages. Python's ABCs can also provide default implementations, making them more flexible than Java or TypeScript interfaces.
