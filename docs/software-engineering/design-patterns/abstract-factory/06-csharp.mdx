## Implementation

Below is a complete C# 12 / .NET 8 implementation of the Abstract Factory pattern using a banking domain. The `IBankFactory` interface defines three creation methods, and two concrete factories (`PersonalBankingFactory` and `BusinessBankingFactory`) produce families of related account objects. C#'s type system enforces the contract at compile time: any class that implements `IBankFactory` must provide concrete implementations for all three creation methods. The client code depends only on the `IBankFactory` and `IAccount` interfaces.

### How the Implementation Works

#### Class Diagram (Pseudocode)

```
// ── Participant Diagram ──
//
//   ┌─────────────────────────────────────────────────────────────┐
//   │   <<interface>> IBankFactory                                │
//   ├─────────────────────────────────────────────────────────────┤
//   │ + CreateSavingsAccount(string, double): IAccount             │  ← OOP: Abstraction
//   │ + CreateCheckingAccount(string, double): IAccount            │     ISP: focused
//   │ + CreateLoanAccount(string, double): IAccount                │     creation methods
//   └─────────────┬───────────────────────────┬───────────────────┘
//                 │ implements                │ implements
//   ┌─────────────┴───────────────┐  ┌────────┴───────────────────┐
//   │ PersonalBankingFactory      │  │ BusinessBankingFactory     │
//   │ ── creates Personal         │  │ ── creates Business        │  ← OOP: Polymorphism
//   │    product family           │  │    product family           │     LSP: substitutable
//   └─────────────┬───────────────┘  └────────┬───────────────────┘
//                 │                           │
//                 ▼                           ▼
//   ┌──────────────────────────┐  ┌───────────────────────────┐
//   │ PersonalSavingsAccount   │  │ BusinessSavingsAccount    │
//   │ PersonalCheckingAccount  │  │ BusinessCheckingAccount   │  ← OOP: Inheritance
//   │ PersonalLoanAccount      │  │ BusinessLoanAccount       │     (implements IAccount)
//   └─────────────┬────────────┘  └────────┬──────────────────┘
//                 │                        │
//                 └──────┬─────────────────┘
//                        │ implements
//                  ┌─────┴───────────┐
//                  │ <<interface>>   │
//                  │ IAccount        │  ← OOP: Abstraction
//                  ├─────────────────┤     ISP: minimal interface
//                  │ GetDetails()    │
//                  │ GetBalance()    │
//                  │ GetMonthlyFee() │
//                  └─────────────────┘
```

#### OOP Concepts Applied

1. **Abstraction** — `IBankFactory` and `IAccount` are C# interfaces that define the contracts. Client code depends on these interfaces, never on the concrete classes.
2. **Encapsulation** — Each account class uses `private readonly` fields to hide internal state. C# also supports `record` types (used for one concrete product) which provide value-based equality and concise immutable state.
3. **Inheritance** — All concrete account classes implement the `IAccount` interface. Both concrete factory classes implement the `IBankFactory` interface.
4. **Polymorphism** — The driver stores accounts in a `List<IAccount>` and calls methods through the `IAccount` interface. C#'s virtual method dispatch ensures the correct implementation runs per concrete type at runtime.

#### SOLID Principles Applied

1. **SRP** — `PersonalBankingFactory` creates only personal banking products. `BusinessBankingFactory` creates only business banking products. Each account class represents exactly one product type.
2. **OCP** — Adding a `PremiumBankingFactory` class requires no changes to the `IBankFactory` interface, existing factories, or existing account classes.
3. **LSP** — Every concrete account type can substitute `IAccount` without breaking the client. The driver treats all accounts uniformly through the `IAccount` interface.
4. **ISP** — The `IAccount` interface is minimal: three methods. The `IBankFactory` interface has three creation methods. No class implements methods it does not need.
5. **DIP** — Client code depends on the `IBankFactory` and `IAccount` interfaces, never on concrete types like `PersonalSavingsAccount`.

### Account

```csharp
// Abstract Factory Pattern — C# 12 / .NET 8 — Banking example
// ============================================================
// Participants: IBankFactory (AbstractFactory), PersonalBankingFactory
//   and BusinessBankingFactory (ConcreteFactory), IAccount (AbstractProduct),
//   PersonalSavingsAccount, PersonalCheckingAccount, PersonalLoanAccount,
//   BusinessSavingsAccount, BusinessCheckingAccount, BusinessLoanAccount
//   (ConcreteProducts)
// OOP: abstraction, encapsulation, inheritance, polymorphism
// SOLID: SRP, OCP, LSP, ISP, DIP
// ============================================================
// ── Abstract Product Interface ──
// OOP role: Abstraction + ISP (defines the contract for all account types)
// SOLID: ISP (minimal interface), DIP (clients depend on this)
public interface IAccount
{
    string GetDetails();
    double GetBalance();
    double GetMonthlyFee();
}
```

### PersonalSavingsAccount

```csharp
// ── Concrete Product Family 1: Personal Banking Accounts ──
// OOP role: Inheritance (implements IAccount) + Polymorphism
// SOLID: LSP (substitutable for IAccount), OCP (extensible), SRP (one responsibility)
public sealed class PersonalSavingsAccount : IAccount
{
    private readonly string _holder;
    private readonly double _balance;
    private readonly double _interestRate = 3.5;
    private readonly double _monthlyFee = 5.0;

    public PersonalSavingsAccount(string holder, double initialDeposit)
    {
        _holder = holder;
        _balance = initialDeposit;
    }

    public string GetDetails() =>
        $"Personal Savings[{_holder}]: ${_balance:F2} @ {_interestRate}% APY";

    public double GetBalance() => _balance;

    public double GetMonthlyFee() => _monthlyFee;
}
```

### BankFactory

```csharp
// ── Abstract Factory Interface ──
// OOP role: Abstraction + Encapsulation (encapsulates product family creation)
// SOLID: SRP (creates accounts only), DIP (clients depend on this abstraction)
public interface IBankFactory
{
    IAccount CreateSavingsAccount(string holder, double amount);
    IAccount CreateCheckingAccount(string holder, double amount);
    IAccount CreateLoanAccount(string holder, double amount);
}
```

### PersonalBankingFactory

```csharp
// ── Concrete Factory 1: PersonalBankingFactory ──
// OOP role: Polymorphism (implements IBankFactory interface)

    public IAccount CreateSavingsAccount(string holder, double amount) =>
```

### Main / Driver

```csharp
// ── Main / Driver ──
public class AbstractFactoryDemo
{
    static void DisplayAccounts(string label, IBankFactory factory)
    {
        Console.WriteLine($"=== {label} ===");
        List<IAccount> accounts =
        [
            factory.CreateSavingsAccount("Alice", 1000.0),
            factory.CreateCheckingAccount("Bob", 2500.0),
            factory.CreateLoanAccount("Carol", 50000.0),
        ];

        foreach (var account in accounts)
        {
            Console.WriteLine($"  {account.GetDetails()}");
            Console.WriteLine($"    Balance: ${account.GetBalance():F2}, " +
                              $"Monthly Fee: ${account.GetMonthlyFee():F2}");
        }
        Console.WriteLine();
    }

    public static void Main(string[] args)
    {
        IBankFactory personalFactory = new PersonalBankingFactory();
        IBankFactory businessFactory = new BusinessBankingFactory();

        DisplayAccounts("Personal Banking Accounts", personalFactory);
        DisplayAccounts("Business Banking Accounts", businessFactory);

        // Demonstrate family consistency
        Console.WriteLine("=== Cross-Family Comparison ===");
        IAccount personalSavings = personalFactory.CreateSavingsAccount("Alice", 1000.0);
        IAccount businessSavings = businessFactory.CreateSavingsAccount("Acme Corp", 50000.0);

        Console.WriteLine($"Personal Savings monthly fee: ${personalSavings.GetMonthlyFee():F2}");
        Console.WriteLine($"Business Savings monthly fee: ${businessSavings.GetMonthlyFee():F2}");

        IAccount personalLoan = personalFactory.CreateLoanAccount("Carol", 50000.0);
        IAccount businessLoan = businessFactory.CreateLoanAccount("Initech", 500000.0);

        Console.WriteLine($"Personal Loan details: {personalLoan.GetDetails()}");
        Console.WriteLine($"Business Loan details: {businessLoan.GetDetails()}");
    }
}
```

### Language-Specific Notes

- **Naming convention — `I` prefix on interfaces** — C# uses the `I` prefix naming convention for interfaces (`IAccount`, `IBankFactory`). This is a .NET convention that distinguishes interfaces from classes at a glance. The prefix is not required by the language but is universally followed in the .NET ecosystem.
- **`sealed` classes** — The `sealed` keyword prevents further inheritance of a class. All concrete product and factory classes are marked `sealed` to prevent subclassing, following the pattern of composition over inheritance. This is the C# equivalent of Java's `final` and Python's `@final`.
- **Expression-bodied members** — C# 6+ supports expression-bodied members (`=>`) for methods and properties that consist of a single expression. Many methods in this example use expression-bodied syntax (`public string GetDetails() => ...`), which is idiomatic modern C#.
- **Collection expressions** — C# 12 introduced collection expressions using square brackets (`List<IAccount> accounts = [...]`). This is the concise syntax shown in the driver, replacing the older `new List<IAccount> { ... }` syntax. It is the idiomatic way to create collections in C# 12+.
- **String interpolation with `$`** — C# string interpolation (`$"Hello {name}"`) is more concise than composite formatting (`string.Format`). Interpolated strings also support format specifiers like `{_balance:F2}` for two decimal places, making the code readable and consistent with other languages' template literals.
