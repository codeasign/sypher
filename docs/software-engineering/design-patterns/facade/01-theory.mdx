## Intent

Provide a unified, simplified interface to a set of interfaces in a subsystem. The Facade pattern defines a higher-level interface that makes the subsystem easier to use by hiding its complexity behind a single entry point. In a cars domain, the `CarFacade` wraps dozens of interconnected subsystems — engine control, transmission, cooling, electrical, fuel system, exhaust, braking — and exposes a handful of intuitive methods like `startEngine()`, `drive()`, and `stopEngine()`. Instead of the driver having to orchestrate a dozen sequential subsystem commands in the correct order, the facade handles all the coordination internally.

The intent of the Facade pattern is not to add new functionality but to simplify access to existing functionality. A well-designed facade reduces the learning curve for a complex subsystem, minimizes coupling between clients and subsystem components, and promotes layering by defining a clear boundary between high-level operations and low-level implementation details.

## The Problem Without This Pattern

Consider what happens when a driver turns the ignition key in a modern car. Behind the dashboard, an elaborate sequence must execute before the engine fires:

```text
Step 1:  Verify battery voltage is above 12.0V
Step 2:  Close the starter relay circuit
Step 3:  Engage the starter motor pinion with the flywheel
Step 4:  Prime the fuel pump to build rail pressure to 55 PSI
Step 5:  Signal the Engine Control Unit (ECU) to begin ignition timing
Step 6:  Open the fuel injectors in cylinder-firing order
Step 7:  Monitor crankshaft position sensor for rotation
Step 8:  Verify oil pressure exceeds 20 PSI within 3 seconds
Step 9:  Disable the immobilizer via the transponder chip
Step 10: Disengage the starter motor once RPM exceeds 800
```

Without a facade, the driver — or worse, the calling code in an automotive software system — must know every one of these steps and execute them in precise order. A software system that controls a car without a facade might look like this:

```java
// PROBLEMATIC: Client code must know every subsystem detail
Battery battery = new Battery();
StarterMotor starter = new StarterMotor();
FuelPump fuelPump = new FuelPump();
ECU ecu = new ECU();
OilPump oilPump = new OilPump();
Immobilizer immobilizer = new Immobilizer();
CoolingSystem cooling = new CoolingSystem();

if (battery.getVoltage() >= 12.0) {
    starter.engagePinion();
    fuelPump.prime(55);
    ecu.initializeIgnitionTiming();
    ecu.openInjectors(CylinderOrder.FIRING_SEQUENCE_4_CYL);
    immobilizer.deactivate();
    starter.crank();
    oilPump.monitorPressure();
    if (ecu.getRPM() > 800) {
        starter.disengage();
    }
}
```

This approach has several critical problems:

1. **Tight coupling** — The client is coupled to every single subsystem class. Changing any subsystem's API (e.g., `fuelPump.prime(55)` becomes `fuelPump.buildPressure(55)`) forces changes throughout every client.

2. **Ordering responsibility** — The client must invoke subsystem operations in the exact correct sequence. A wrong order (e.g., opening injectors before priming the fuel pump) can cause mechanical damage.

3. **Code duplication** — Every piece of code that starts the car must duplicate this entire orchestration block. A software system with multiple control modules (dashboard UI, remote start, autonomous driving) repeats the same logic everywhere.

4. **Hard to test** — Testing the client requires mocking every subsystem. A change to the start sequence means updating every client test.

5. **Violates encapsulation** — Internal subsystem details (required fuel pressure, injector firing order, RPM threshold for disengagement) leak into the client layer.

### Concrete Scenario: Three Different Clients

To illustrate the scope of the problem, imagine an automotive software system with three distinct clients that all need to start the car:

- **Dashboard UI** — The driver presses the Start button on the touch screen.
- **Remote Start Module** — The driver presses the Lock button three times on the key fob.
- **Autonomous Driving Module** — The self-driving system needs to restart the engine after an idle stop at a traffic light.

Without a facade, each of these three clients must independently implement the entire ten-step startup sequence. If the sequence changes — and it will, as the ECU firmware is updated — all three clients must be modified and retested. In a large automotive codebase, the startup logic might be duplicated across a dozen modules, each with its own subtle variations and bugs.

### Concrete Scenario: Hybrid Vehicle Transition

Consider a car manufacturer transitioning from a purely internal-combustion vehicle to a hybrid electric vehicle. The hybrid adds new subsystems: a high-voltage battery pack, an electric motor/generator, a DC-DC converter, and a battery management system. The startup sequence now includes an additional precondition: the high-voltage contactor must close before the electric motor can assist the engine.

Without a facade, every client that implements the startup sequence must be modified to include the contactor check. With a facade, only the single `CarFacade.startEngine()` method changes. The three clients remain identical — they call `startEngine()` and the facade handles the rest.

The participants in the Facade pattern are:

1. **Facade (CarFacade)** — Knows which subsystem classes are responsible for a request and delegates client requests to the appropriate subsystem objects. The facade is aware of subsystem dependencies and orchestrates them in the correct order.

2. **Subsystem classes (Engine, Transmission, CoolingSystem, etc.)** — Implement subsystem functionality. They handle work assigned by the facade but have no knowledge of the facade. Subsystems can communicate with each other directly if needed.

3. **Client (Driver)** — Requests subsystem operations through the facade. The client has no direct knowledge of the subsystem objects.

The key insight is that the facade is not a layer that merely forwards calls — it is an orchestrator that adds value through coordination, precondition validation, ordering, and error handling. The subsystems remain independently usable by other clients that need fine-grained access.

## Pseudocode — Cars Example

The following pseudocode demonstrates the Facade pattern for a car control system. The `CarFacade` coordinates six subsystems to provide a simple start-drive-stop interface.

```text
// PSEUDOCODE — Facade Pattern in Cars Domain
// Participants labeled with [ROLE]

// ── Subsystem: Engine ──
class Engine
    private rpm: int = 0
    private oilPressure: float = 0.0
    private coolantTemp: float = 20.0

    method crank(): void
        print("Engine: cranking at 250 RPM")
        this.rpm = 250

    method injectFuel(): void
        print("Engine: injecting fuel into cylinders")

    method spark(): void
        print("Engine: firing spark plugs")

    method getIdleRPM(): int
        return 850

    method getRPM(): int
        return this.rpm

    method rev(amount: int): void
        this.rpm = this.rpm + amount
        print("Engine: revving to " + this.rpm + " RPM")

    method stop(): void
        this.rpm = 0
        print("Engine: stopped")

// ── Subsystem: Transmission ──
class Transmission
    private gear: string = "Park"

    method shiftTo(gearName: string): void
        this.gear = gearName
        print("Transmission: shifted to " + gearName)

    method getCurrentGear(): string
        return this.gear

// ── Subsystem: CoolingSystem ──
class CoolingSystem
    private coolantTemp: float = 20.0

    method runWaterPump(): void
        print("Cooling: water pump circulating coolant")

    method runRadiatorFan(): void
        print("Cooling: radiator fan activated")

    method getCoolantTemperature(): float
        return this.coolantTemp

    method setThermostat(temp: float): void
        this.coolantTemp = temp
        print("Cooling: thermostat set to " + temp + "C")

// ── Subsystem: ElectricalSystem ──
class ElectricalSystem
    private voltage: float = 12.6

    method getBatteryVoltage(): float
        return this.voltage

    method powerAccessories(): void
        print("Electrical: powering dashboard and infotainment")

    method powerECU(): void
        print("Electrical: Engine Control Unit powered on")

    method closeStarterRelay(): void
        print("Electrical: starter relay closed")

// ── Subsystem: FuelSystem ──
class FuelSystem
    private railPressure: float = 0.0

    method prime(pressure: float): void
        this.railPressure = pressure
        print("Fuel: rail pressure primed to " + pressure + " PSI")

    method getRailPressure(): float
        return this.railPressure

    method shutoff(): void
        this.railPressure = 0.0
        print("Fuel: fuel pump shut off")

// ── Subsystem: BrakingSystem ──
class BrakingSystem
    private engaged: bool = true

    method release(): void
        this.engaged = false
        print("Brakes: parking brake released")

    method apply(): void
        this.engaged = true
        print("Brakes: parking brake applied")

// ── Facade ──
class CarFacade
    private engine: Engine
    private transmission: Transmission
    private cooling: CoolingSystem
    private electrical: ElectricalSystem
    private fuelSystem: FuelSystem
    private brakes: BrakingSystem

    constructor()
        this.engine = new Engine()
        this.transmission = new Transmission()
        this.cooling = new CoolingSystem()
        this.electrical = new ElectricalSystem()
        this.fuelSystem = new FuelSystem()
        this.brakes = new BrakingSystem()

    method startEngine(): void
        print("--- Starting Engine ---")
        voltage = this.electrical.getBatteryVoltage()
        if voltage < 12.0
            print("ERROR: Battery voltage too low.")
            return
        this.electrical.powerECU()
        this.electrical.powerAccessories()
        this.electrical.closeStarterRelay()
        this.cooling.runWaterPump()
        this.fuelSystem.prime(55)
        this.engine.crank()
        this.engine.injectFuel()
        this.engine.spark()
        print("Engine started. Idle RPM: " + this.engine.getIdleRPM())
        this.cooling.runRadiatorFan()
        this.brakes.release()

    method drive(speed: int): void
        print("--- Driving at " + speed + " RPM ---")
        this.transmission.shiftTo("Drive")
        this.engine.rev(speed)
        targetTemp = 90.0
        this.cooling.setThermostat(targetTemp)
        print("Driving: cruising at " + speed + " RPM")

    method stopEngine(): void
        print("--- Stopping Engine ---")
        this.transmission.shiftTo("Park")
        this.brakes.apply()
        this.engine.stop()
        this.fuelSystem.shutoff()
        print("Engine stopped. All subsystems off.")

// ── Driver ──
function main()
    car = new CarFacade()
    car.startEngine()
    car.drive(2500)
    car.stopEngine()
```

## How It Works — Step by Step

1. **Client calls a high-level facade method** — The driver (or calling code) invokes `car.startEngine()` without knowing any subsystem details. The facade method name is intuitive and matches the user's mental model. The client does not need to know that starting an engine involves electricity, fuel, cooling, and mechanical rotation.

2. **Facade validates preconditions** — The facade checks the battery voltage. If the voltage is below 12.0V, it prints an error and returns immediately, preventing a weak battery from being drained further. This precondition check is performed once, in one place, rather than duplicated across every client.

3. **Facade initializes subsystems in correct order** — Power is sent to the ECU and accessories before the starter relay closes. The water pump circulates coolant before the engine cranks, ensuring the engine block is not dry. The fuel pump primes the rail to 55 PSI before the injectors open. Each ordering constraint is encoded in the facade's method body, not scattered across client code.

4. **Facade executes the start sequence** — The starter motor cranks the engine, fuel injectors spray into the cylinders, and the spark plugs fire. Each subsystem call happens in the exact sequence required for a successful start. The facade does not simply delegate; it actively coordinates. For example, `injectFuel()` must occur after `crank()` has started rotating the engine, and `spark()` must fire after the fuel-air mixture is present in the cylinder.

5. **Facade transitions to post-start state** — Once the engine is running at idle RPM, the radiator fan activates to begin active cooling. The parking brake releases so the car is ready to move. These post-start actions are easy to forget in ad hoc client code but are guaranteed by the facade.

6. **Facade provides a clean stop sequence** — When the driver calls `stopEngine()`, the facade shifts the transmission to Park, applies the parking brake, stops the engine, and shuts off the fuel pump. Every subsystem is returned to a safe idle state. The stop sequence is the inverse of the start sequence, ensuring symmetrical behavior that is hard to achieve when each client writes its own shutdown code.

### The Facade Method in Detail: `startEngine()`

The facade's `startEngine()` method is the most complex of the three. Let us examine each line for its purpose:

```text
// Step 1: Diagnostic summary header
print("--- Starting Engine ---")

// Step 2: Read battery voltage from the electrical subsystem
voltage = this.electrical.getBatteryVoltage()

// Step 3: Guard clause — abort if battery is too weak
if voltage < 12.0
    print("ERROR: Battery voltage too low.")
    return

// Step 4: Power on the car's brain (ECU) before anything else
this.electrical.powerECU()

// Step 5: Wake up the dashboard, radio, and climate controls
this.electrical.powerAccessories()

// Step 6: Close the high-current starter relay circuit
this.electrical.closeStarterRelay()

// Step 7: Start circulating coolant to prevent hot spots
this.cooling.runWaterPump()

// Step 8: Build fuel rail pressure for immediate injection
this.fuelSystem.prime(55)

// Step 9: Turn the engine over mechanically
this.engine.crank()

// Step 10: Introduce fuel into the combustion chambers
this.engine.injectFuel()

// Step 11: Ignite the compressed fuel-air mixture
this.engine.spark()

// Step 12: Confirm the engine is running at idle speed
print("Engine started. Idle RPM: " + this.engine.getIdleRPM())

// Step 13: Begin active cooling now that the engine is running
this.cooling.runRadiatorFan()

// Step 14: Release the parking brake so the car can move
this.brakes.release()
```

This fourteen-step sequence is a complete, ordered workflow. Every client that starts the car follows the same sequence, guaranteed by the facade.

## Benefits of the Facade Pattern

1. **Reduced client complexity** — The client interacts with one facade class instead of six or more subsystem classes. The client's learning curve drops from understanding the entire subsystem API to understanding three method names.

2. **Decoupling** — The client is decoupled from the subsystem classes. Subsystem internals can be refactored, upgraded, or replaced without affecting any client that uses the facade. This is especially valuable in the automotive industry, where ECU firmware updates occur frequently.

3. **Centralized cross-cutting concerns** — Logging, performance monitoring, fault tolerance, and security checks can be implemented once in the facade rather than scattered across every client. The facade's `startEngine()` method could emit telemetry to a cloud dashboard, log to the diagnostic bus, and record timing metrics — all transparently to the client.

4. **Enforced consistency** — The facade guarantees that all operations are performed in the correct order with the correct parameters. No client can accidentally skip the fuel priming step or invoke `spark()` before `crank()`.

5. **Layered architecture** — The facade defines a clear boundary between the high-level operations layer and the low-level subsystem layer. This layering allows teams to work independently: one team develops the facade's orchestration logic while another team develops the subsystem implementations.

6. **Change isolation** — When the car transitions from a 12V lead-acid battery to a 48V lithium-ion system, the facade's public API does not change. The internal voltage threshold and subsystem interactions are updated within the facade, and all clients continue to call the same `startEngine()` method.

## Consequences and Trade-offs

While the Facade pattern provides significant benefits, it also introduces trade-offs:

1. **Reduced flexibility** — The facade restricts clients to the operations it exposes. If a client needs to perform a subsystem operation that the facade does not support (e.g., read individual cylinder pressure), the facade becomes a bottleneck or must be extended. Mitigation: design the facade for the 90% use case and expose subsystem access for the remaining 10%.

2. **Risk of God Object** — The facade can grow into a monolithic "god object" that knows too much and does too much. This is a maintenance risk. Mitigation: split a large facade into multiple domain-specific facades (e.g., `PowertrainFacade`, `ClimateFacade`, `InfotainmentFacade`).

3. **Indirect access** — Every client request passes through the facade, adding a layer of indirection. For performance-critical paths (e.g., a real-time engine control loop running at 10 kHz), this indirection may introduce unacceptable latency. Mitigation: allow high-performance clients to bypass the facade and access subsystems directly.

4. **Transparency illusion** — The facade hides complexity from the client, but the complexity still exists. If a subsystem fails (e.g., the fuel pump cannot build pressure), the facade must surface meaningful error information without leaking subsystem details. Mitigation: design a clear error reporting protocol (typed errors, codes, or structured status objects) that provides enough context for the client to act.

## When to Use

- **You want to shield clients from subsystem complexity** — A modern car has dozens of ECUs, sensors, actuators, and mechanical subsystems. The facade provides a clean, minimal API that hides this complexity from the driver and from higher-level software modules (infotainment, telematics, autonomous driving stack).

- **There is tight coupling between clients and subsystem classes** — If changes to the fuel pump API force rewrites in five different control modules, introduce a facade. All five modules call `carFacade.startEngine()` instead of directly manipulating the fuel pump.

- **You want to introduce a layered subsystem architecture** — The facade defines a clear boundary between the high-level operations (start, drive, stop) and the low-level subsystem implementations. This decoupling allows each layer to evolve independently.

- **The subsystem is complex or undergoes frequent changes** — When internal combustion engines are replaced by hybrid or electric powertrains, only the facade's internal implementation changes. The client code that calls `startEngine()` remains identical.

- **You need a single entry point for cross-cutting concerns** — Logging, monitoring, security checks, and fault tolerance can be centralized in the facade rather than scattered across every subsystem call.

## When NOT to Use

- **The client needs fine-grained control over subsystems** — If the user is a professional mechanic or a diagnostic tool that needs to test fuel pressure independently of the engine start sequence, exposing subsystems directly is more appropriate. The facade hides exactly what you want hidden; it should not be the only way to access subsystems.

- **The facade becomes a God Object** — If the facade accumulates hundreds of methods covering every subsystem operation, it has lost its purpose. A bloated facade turns into a God Object that is itself complex and tightly coupled. Break it into multiple focused facades (e.g., `EngineFacade`, `ClimateFacade`, `EntertainmentFacade`).

- **Performance is critical and the facade adds unnecessary overhead** — If the calling code needs to start the car in under 50 milliseconds for a racing application, bypassing the facade's safety checks and logging may be warranted. However, this is rare — most systems benefit from the facade's safeguards.

- **The subsystem is already simple** — A three-function subsystem with a clean API does not need a facade wrapper. Adding a facade for a simple subsystem introduces accidental complexity and indirection.

## Real-World Occurrences

1. **Automotive Engine Control Unit (ECU)** — A modern ECU is the quintessential facade. It receives a single input (throttle position from the accelerator pedal) and internally orchestrates fuel injection timing, spark advance, variable valve timing, turbocharger boost pressure, and exhaust gas recirculation. The driver never interacts with these subsystems directly. The ECU itself is a Facade: the driver presses the accelerator, and the ECU translates that single action into dozens of coordinated subsystem commands, adjusting timing based on coolant temperature, air density, fuel octane rating, and engine load.

2. **Car Dashboard Instrument Cluster** — The instrument cluster facade receives simple CAN bus signals (vehicle speed, engine RPM, fuel level) and internally manages speedometer stepper motors, tachometer gauges, LCD displays, warning lights, and chime generators. The wiring harness does not route individual signals to each gauge; the cluster facade handles everything. A single CAN message for vehicle speed is translated into the correct stepper motor position on the analog speedometer, the correct digital readout on the LCD, and the correct trip computer calculation.

3. **Infotainment System API** — Apple CarPlay and Android Auto provide a facade over vehicle-specific infotainment hardware. The smartphone app calls abstract methods (`playAudio()`, `showNavigation()`) while the facade translates these into the specific protocols for the car's head unit, amplifier, and display. The smartphone developer does not need to know whether the car uses MOST bus, CAN bus, or Ethernet for audio streaming.

4. **Remote Start Module** — An aftermarket remote start system is a facade. It accepts a single signal from the key fob and internally sequences the ignition, fuel pump, starter motor, climate control, and security system disarm. The user presses one button and all subsystems activate in the correct order with proper safety interlocks. The remote start module also implements safety preconditions: the hood must be closed, the transmission must be in Park, and the parking brake must be engaged.

5. **OBD-II Diagnostic Interface** — The On-Board Diagnostics II port provides a standardized facade over dozens of proprietary ECU protocols. A single `PID 0x0D` (vehicle speed) request works uniformly across all car brands, while the OBD-II facade internally translates to manufacturer-specific CAN bus messages. The diagnostic tool developer writes one query that works on Ford, Toyota, BMW, and Mercedes vehicles, even though each manufacturer uses a different CAN message format for vehicle speed.

6. **Automatic Car Wash System** — The car wash control panel presents a single `startWash()` button that internally sequences the pre-soak, foam application, brush rotation, high-pressure rinse, wax spray, and blow-dry stages. The operator does not control each subsystem individually. The facade also implements safety interlocks: the conveyor belt cannot move unless the brushes are at their home position, and the high-pressure rinse cannot activate unless the foam stage has completed.

## Comparison with Related Patterns

| Pattern | Relationship with Facade | Key Difference |
|---------|--------------------------|----------------|
| **Adapter** | Both provide an interface to existing classes. | Adapter converts one interface to another that the client expects. Facade provides a simplified interface to a larger subsystem; it does not convert interfaces but simplifies access. |
| **Mediator** | Both coordinate communication between components. | Mediator abstracts communication between colleague objects that are aware of each other. Facade provides a simplified unidirectional interface to a subsystem; it does not manage peer-to-peer communication. |
| **Proxy** | Both act as intermediaries for another object. | Proxy controls access to a single object (lazy loading, access control, logging). Facade provides a simplified interface to a whole subsystem of objects. |
| **Abstract Factory** | Both can work together to create subsystem objects. | Abstract Factory focuses on creating families of related objects. Facade can use an Abstract Factory to obtain subsystem instances, but its role is coordinating those instances, not creating them. |
| **Singleton** | Facade is often implemented as a Singleton. | Singleton ensures a single instance. A Facade is frequently a Singleton (one car facade per vehicle), but the patterns address different concerns: instance control vs. interface simplification. |
| **Command** | Both can be used to encapsulate operations. | Command encapsulates a single operation as an object with an `execute()` method. Facade encapsulates an entire sequence of subsystem operations into a single high-level method. |

### Adapter vs. Facade — Detailed Distinction

The Adapter and Facade patterns are frequently confused because both sit between a client and an existing class or subsystem. The critical distinction is intent:

- **Adapter** is used when you have an existing class with the right functionality but the wrong interface. The adapter rewrites the interface to match what the client expects. In the cars domain, a 12V-to-5V USB adapter converts the car's cigarette lighter socket to a USB port that charges a phone. The functionality (power delivery) is the same; the interface changes.

- **Facade** is used when you have a complex subsystem with a valid interface, but the client should not need to understand the subsystem's complexity. The facade provides a simpler interface that still uses the subsystem's existing interfaces internally. In the cars domain, the `startEngine()` method does not convert interfaces — it coordinates existing interfaces in a specific sequence with specific parameters.

### Mediator vs. Facade — Detailed Distinction

The Mediator and Facade patterns both involve an intermediary that coordinates communication. The key distinction is directionality:

- **Mediator** manages peer-to-peer communication between colleague objects. The colleagues know about the mediator and communicate through it. The mediator reduces coupling between colleagues. In a car's climate control system, the mediator would manage communication between the temperature sensor, the HVAC actuator, the user's temperature dial, and the sunload sensor — all of which need to share data bidirectionally.

- **Facade** provides a unidirectional entry point from the client to the subsystem. The subsystem objects do not know about the facade. The facade reduces coupling between the client and the subsystem. In the same climate control system, the facade would expose `setTemperature(22)` and `setFanSpeed(3)` without the client needing to know about the temperature sensor or HVAC actuator.

## Implementation Considerations

When implementing the Facade pattern in an automotive context, consider the following:

### 1. Interface Design

A well-designed facade should expose operations at the level of the client's mental model. The three methods `startEngine()`, `drive()`, and `stopEngine()` map directly to what a driver thinks about when operating a car. Avoid exposing methods like `verifyBatteryVoltage()` or `primeFuelPump()` — these are implementation details that the facade should hide.

### 2. Error Handling Strategy

The facade is the natural place to implement error handling for the subsystem. Consider the following error scenarios and how the facade should respond:

- **Battery voltage too low** — Print an error and abort. Do not attempt to crank the engine with a weak battery, as this can damage the starter motor.
- **Fuel pump failure** — The facade should catch a `FuelPumpException` from the subsystem and translate it into a user-friendly message like "Fuel system error — service required."
- **Engine fails to start after 3 seconds** — The facade should cut power to the starter motor after a timeout to prevent overheating the starter winding.

Each error should be caught, logged, and translated into a meaningful message without leaking subsystem implementation details.

### 3. Subsystem Access

Consider whether the facade should be the only way to access subsystems. Two options exist:

- **Restricted access** — Subsystem classes are package-private or internal. All clients must go through the facade. This enforces consistency but limits flexibility.
- **Open access** — Subsystem classes are public. Clients can use the facade for 90% of operations and access subsystems directly for specialized needs (e.g., diagnostics, testing). This provides flexibility but risks inconsistent behavior.

The open access approach is generally preferred in automotive systems because diagnostic tools and test harnesses need direct subsystem access.

### 4. Facade Granularity

A single large facade may become unwieldy. Consider multiple focused facades:

- `EngineFacade` — start, stop, rev, diagnostic readouts
- `ClimateFacade` — set temperature, fan speed, air distribution
- `InfotainmentFacade` — play media, navigate, phone calls
- `SafetyFacade` — airbag status, tire pressure monitoring, collision detection

These facades can be composed by a higher-level `VehicleFacade` that orchestrates multiple domains.

## Complete Example: End-to-End Driving Scenario

The following scenario traces a complete driving session through the facade's three methods, showing the full subsystem interaction for each stage.

### Stage 1: Morning Start

The driver approaches the car, opens the door, and presses the start button. The facade executes `startEngine()`:

```text
1. electrical.getBatteryVoltage()       → 12.6V (passes check)
2. electrical.powerECU()                → ECU boots, loads calibration data
3. electrical.powerAccessories()         → Dashboard lights up, infotainment boots
4. electrical.closeStarterRelay()        → 12V delivered to starter solenoid
5. cooling.runWaterPump()               → Coolant begins circulating
6. fuelSystem.prime(55)                 → Fuel rail pressurized to 55 PSI
7. engine.crank()                       → Starter motor turns crankshaft at 250 RPM
8. engine.injectFuel()                  → Injectors spray fuel into cylinders 1-3-4-2
9. engine.spark()                       → Spark plugs fire, combustion begins
10. Engine reaches 850 RPM              → Idle speed achieved
11. cooling.runRadiatorFan()            → Fan activates to manage heat
12. brakes.release()                    → Parking brake disengaged
```

### Stage 2: Driving to Work

The driver shifts into Drive and accelerates. The facade executes `drive(2500)`:

```text
1. transmission.shiftTo("Drive")        → Torque converter locks, gear engaged
2. engine.rev(2500)                     → Throttle opens, RPM climbs to 2500
3. cooling.setThermostat(90.0)          → Thermostat opens at 90C for optimal operating temp
4. Engine produces adequate torque       → Car moves forward smoothly
```

### Stage 3: Park and Shut Down

The driver arrives at work, parks the car, and presses the stop button. The facade executes `stopEngine()`:

```text
1. transmission.shiftTo("Park")         → Parking pawl engages
2. brakes.apply()                        → Parking brake holds the car stationary
3. engine.stop()                         → Fuel injection stops, RPM drops to 0
4. fuelSystem.shutoff()                  → Fuel pump power cut, rail pressure drops
5. All subsystems in safe idle state     → Car is secure and powered down
```

This end-to-end scenario demonstrates the facade's value: the driver and the calling code think in terms of three high-level operations, while the facade translates each operation into the correct sequence of subsystem commands. Subsystem details are completely hidden, ordering is guaranteed, and preconditions are validated in one central place.
