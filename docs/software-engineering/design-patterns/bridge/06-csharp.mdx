The Bridge pattern in C# uses an `IEngine` interface to define the engine contract and an abstract `Car` base class for the car abstraction. The `IEngine` interface declares `Start()`, `Sound()`, and `FuelType()`. The `Car` abstract class holds a `protected readonly IEngine` reference and declares `Drive()` as abstract. C#'s strong typing and explicit interface implementation ensure compile-time safety. Three concrete engine classes implement `IEngine`, and three car subclasses extend `Car`. The `HybridEngine` demonstrates implementation-level composition by wrapping two `IEngine` references. C#'s property syntax and string interpolation make the engine specifications clean and readable.

## How the Implementation Works

C#'s `IEngine` interface defines the contract following the .NET naming convention (capital `I` prefix). The `Car` abstract class uses constructor injection to receive its `IEngine` reference and stores it as a `protected readonly` field — `readonly` prevents reassignment after construction, similar to Java's `final`. Each concrete engine class implements `IEngine` with private fields for its parameters. The `HybridEngine` accepts two `IEngine` parameters and conditionally delegates between them. C#'s string interpolation (`$"..."`) provides inline expression substitution. The `abstract` keyword on `Car.Drive()` enforces that all concrete car subclasses provide an implementation.

### OOP Concepts Applied

- **Abstraction** — `IEngine` interface defines the engine contract. `Car` abstract class defines the car contract. Both are abstractions decoupled from concrete types.
- **Encapsulation** — Each engine class uses `private readonly` fields to store its parameters. No public accessors expose internal engine state.
- **Polymorphism** — `Car.Drive()` calls `engine.Start()` and `engine.Sound()` polymorphically. C#'s virtual method dispatch routes calls to the correct concrete implementation.
- **Composition** — `Car` holds an `IEngine` reference via constructor injection. `HybridEngine` composes two `IEngine` references, demonstrating composition at the implementation level.

### SOLID Principles Applied

- **Single Responsibility Principle** — GasEngine handles only combustion. Sedan handles only sedan driving. Each class has one focused responsibility.
- **Open/Closed Principle** — New `IEngine` implementations (`DieselEngine`, `HydrogenEngine`) and new `Car` subclasses (`Convertible`, `Pickup`) can be added via new classes without modifying existing code.
- **Liskov Substitution Principle** — Any `IEngine` implementation can replace another in any `Car` subclass. `Car` interacts only through the `IEngine` interface.
- **Interface Segregation Principle** — The `IEngine` interface has exactly three methods. No engine is forced to implement unused operations.
- **Dependency Inversion Principle** — `Car` depends on the `IEngine` interface (abstraction), not on `GasEngine` or `ElectricEngine` (concretions).

### Participant 1: IEngine (Implementor Interface)

```csharp
// OOP: Abstraction — defines the contract all engines must satisfy
// SOLID: ISP — minimal three-method interface; DIP — Car depends on this
public interface IEngine
{
    void Start();
    string Sound();
    string FuelType();
}
```

### Participant 2: GasEngine (ConcreteImplementor)

```csharp
// OOP: Encapsulation — internal parameters are private readonly
// SOLID: SRP — implements only gas combustion logic
public class GasEngine : IEngine
{
    private readonly int _cylinders;
    private readonly double _displacement;

    public GasEngine(int cylinders, double displacement)
    {
        _cylinders = cylinders;
        _displacement = displacement;
    }

    public void Start()
    {
        Console.WriteLine("Gas engine fires: *vroom vroom*");
    }

    public string Sound()
    {
        var sb = new StringBuilder("VROOOOM");
        for (int i = 0; i < _cylinders; i++)
            sb.Append('.');
        return sb.ToString();
    }

    public string FuelType()
    {
        return $"Gasoline ({_displacement:F1}L {_cylinders}-cyl)";
    }
}
```

### Participant 3: ElectricEngine (ConcreteImplementor)

// OOP: Encapsulation — battery and motor specs are private readonly

### Participant 4: HybridEngine (ConcreteImplementor)

```csharp
// OOP: Composition — wraps both a gas and electric engine
// SOLID: SRP — coordinates delegation without duplicating engine logic
public class HybridEngine : IEngine
{
    private readonly IEngine _gasEngine;
    private readonly IEngine _electricEngine;
    private readonly int _batteryLevel;

    public HybridEngine(IEngine gasEngine, IEngine electricEngine, int batteryLevel)
    {
        _gasEngine = gasEngine;
        _electricEngine = electricEngine;
        _batteryLevel = batteryLevel;
    }

    public void Start()
    {
        _electricEngine.Start();
        if (_batteryLevel < 20)
        {
            Console.WriteLine("Battery low — gas engine kicks in.");
            _gasEngine.Start();
        }
        else
        {
            Console.WriteLine("Running on electric power — gas engine idle.");
        }
    }

    public string Sound()
    {
        if (_batteryLevel >= 20)
            return _electricEngine.Sound();
        return _gasEngine.Sound() + " then " + _electricEngine.Sound();
    }

    public string FuelType()
    {
        return $"Hybrid (Gas + Electric, battery at {_batteryLevel}%)";
    }
}
```

### Participant 5: Car (Abstraction)

```csharp
// OOP: Abstraction — defines the car interface and holds IEngine reference
// SOLID: DIP — depends on IEngine interface, not on concrete implementations
public abstract class Car
{
    protected readonly IEngine Engine;

    protected Car(IEngine engine)
    {
        Engine = engine;
    }

    public abstract string Drive();

    public virtual string GetSpecs()
    {
        return $"Car with {Engine.FuelType()}";
    }
}
```

### Participant 6: Sedan (RefinedAbstraction)

```csharp
// OOP: Polymorphism — overrides Drive() with sedan-specific behavior
// SOLID: SRP — handles only sedan driving mechanics
public class Sedan : Car
{
    private readonly double _trunkCapacity;

    public Sedan(IEngine engine, double trunkCapacity) : base(engine)
    {
        _trunkCapacity = trunkCapacity;
    }

    public override string Drive()
    {
        Engine.Start();
        return $"Sedan cruising smoothly on the highway. {Engine.Sound()}";
    }

    public string OpenTrunk()
    {
        return $"Trunk opens ({_trunkCapacity:F1} cu ft).";
    }

    public override string GetSpecs()
    {
        return $"Sedan with {Engine.FuelType()} | Trunk: {_trunkCapacity:F1} cu ft";
    }
}
```

### Participant 7: SUV (RefinedAbstraction)

```csharp
// OOP: Polymorphism — overrides Drive() with SUV-specific behavior
// SOLID: SRP — handles only SUV driving mechanics
public class SUV : Car
{
    private readonly double _groundClearance;
    private readonly bool _awd;

    public SUV(IEngine engine, double groundClearance, bool awd) : base(engine)
    {
        _groundClearance = groundClearance;
        _awd = awd;
    }

    public override string Drive()
    {
        Engine.Start();
        return $"SUV tackling off-road terrain. {Engine.Sound()}";
    }

    public string Engage4WD()
    {
        if (_awd)
            return "All-wheel drive engaged. Traction optimized.";
        return "This SUV does not have AWD.";
    }

    public override string GetSpecs()
    {
        string drivetrain = _awd ? "AWD" : "2WD";
        return $"SUV with {Engine.FuelType()} | Ground: {_groundClearance:F1}\" {drivetrain}";
    }
}
```

### Language-Specific Notes

- C# interfaces follow the `I` prefix naming convention (`IEngine` rather than `Engine`). This is a .NET standard that makes interfaces immediately distinguishable from classes.
- The `protected readonly` field in `Car` prevents the engine reference from being reassigned after construction. C#'s `readonly` modifier is enforced at compile time — any attempt to assign to the field outside the constructor produces a compilation error.
- C#'s `virtual` keyword on `Car.GetSpecs()` allows subclasses to optionally `override` the method. This provides more flexibility than Java's default non-virtual methods — subclasses can choose whether to replace or extend the base behavior.
- String interpolation (`$"..."`) is a first-class C# feature. Interpolated strings are compiled to efficient `String.Format` calls. Format specifiers like `:F1` (one decimal place) are embedded directly in the expression syntax.
- The `var` keyword in the driver code reduces type repetition. `var sedan = new Sedan(...)` is equivalent to `Sedan sedan = new Sedan(...)` — the compiler infers the type from the right-hand side.
- C#'s `StringBuilder` class is the idiomatic choice for building strings in loops, as demonstrated in `GasEngine.Sound()`. It avoids the quadratic copying of repeated string concatenation.
