The Bridge pattern in JavaScript uses composition and duck typing to decouple the Car abstraction hierarchy from the Engine implementation hierarchy. Three car types (Sedan, SUV, Truck) each hold a reference to an Engine object and delegate engine-specific behavior to it. Three engine implementations (GasEngine, ElectricEngine, HybridEngine) each implement the same method protocol. JavaScript's lack of formal interfaces makes this a convention-based bridge: any object with `start()`, `sound()`, and `fuelType()` methods is a valid engine. The client composes a car with an engine at runtime, and both hierarchies can grow independently.

## How the Implementation Works

The JavaScript Bridge pattern implements the Car abstraction as a base class that accepts an Engine object in its constructor. Each RefinedAbstraction (Sedan, SUV, Truck) extends Car and overrides `drive()` and `getSpecs()` to incorporate engine-specific behavior. The Engine implementations are plain classes that follow a convention: they all expose `start()`, `sound()`, and `fuelType()` methods. The HybridEngine demonstrates composition of implementations — it wraps both a GasEngine and an ElectricEngine and coordinates between them. JavaScript's dynamic dispatch ensures that `engine.start()` always calls the correct method for whichever engine object is passed in.

### OOP Concepts Applied

- **Abstraction** — Car defines the `drive()` and `getSpecs()` interface. Engine defines the `start()`, `sound()`, `fuelType()` interface. Both sides hide their internal complexity from each other.
- **Encapsulation** — Each engine class hides its internal parameters (cylinders, battery capacity, motor power) behind the Engine protocol methods. Car subclasses do not access engine internals directly.
- **Polymorphism** — Car subclasses call `this.engine.start()` without knowing the concrete engine type. GasEngine, ElectricEngine, and HybridEngine each respond to the same method calls with different behavior.
- **Composition** — Car holds an Engine reference (has-a) rather than inheriting from Engine. The HybridEngine composes two other engines, demonstrating composition at the implementation level too.

### SOLID Principles Applied

- **Single Responsibility Principle** — Each Car subclass is responsible for one car type's driving behavior. Each Engine subclass is responsible for one engine type's starting behavior. HybridEngine coordinates two engines but delegates the actual work to them.
- **Open/Closed Principle** — New car types (Convertible, Minivan) can be added without modifying existing Car subclasses or any Engine class. New engine types (HydrogenEngine, DieselEngine) can be added without modifying any Car subclass.
- **Liskov Substitution Principle** — Any Engine object can replace another in a Car instance as long as it exposes `start()`, `sound()`, and `fuelType()`. Car subclasses treat all engines uniformly through the Engine protocol.
- **Interface Segregation Principle** — The Engine protocol is minimal: three methods that every engine must support. No engine is forced to implement methods it does not need.
- **Dependency Inversion Principle** — Car depends on the Engine protocol (an abstraction), not on concrete engine classes. Concrete engines depend on the same protocol, not on Car subclasses.

### Participant 2: GasEngine (ConcreteImplementor)

```javascript
// OOP: Encapsulation — cylinders and displacement are private
// SOLID: SRP — implements only engine-specific behavior
class GasEngine {
  #cylinders;
  #displacement;

  constructor(cylinders, displacement) {
    this.#cylinders = cylinders;
    this.#displacement = displacement;
  }

  start() {
    console.log('Gas engine fires: *vroom vroom*');
  }

  sound() {
    return 'VROOOOM' + '.'.repeat(this.#cylinders);
  }

  fuelType() {
    return `Gasoline (${this.#displacement}L ${this.#cylinders}-cyl)`;
  }
}
```

### Participant 3: ElectricEngine (ConcreteImplementor)

// OOP: Encapsulation — batteryCapacity and motorPower are private

class ElectricEngine {

    return 'WHIIIIRRRR';

### Participant 4: HybridEngine (ConcreteImplementor)

```javascript
// OOP: Composition — wraps both a gas and electric engine
// SOLID: SRP — coordinates two engines without duplicating their logic
class HybridEngine {
  #gasEngine;
  #electricEngine;
  #batteryLevel;

  constructor(gasEngine, electricEngine, batteryLevel) {
    this.#gasEngine = gasEngine;
    this.#electricEngine = electricEngine;
    this.#batteryLevel = batteryLevel;
  }

  start() {
    this.#electricEngine.start();
    if (this.#batteryLevel < 20) {
      console.log('Battery low — gas engine kicks in.');
      this.#gasEngine.start();
    } else {
      console.log('Running on electric power — gas engine idle.');
    }
  }

  sound() {
    if (this.#batteryLevel >= 20) {
      return this.#electricEngine.sound();
    }
    return this.#gasEngine.sound() + ' then ' + this.#electricEngine.sound();
  }

  fuelType() {
    return `Hybrid (Gas + Electric, battery at ${this.#batteryLevel}%)`;
  }
}
```

### Participant 5: Car (Abstraction)

```javascript
// OOP: Abstraction — defines drive() and holds engine reference
// SOLID: DIP — depends on Engine protocol, not concrete engines
class Car {
  constructor(engine) {
    if (new.target === Car) {
      throw new TypeError('Car is abstract — instantiate a subclass');
    }
    this.engine = engine;
  }

  drive() {
    throw new Error('drive() must be implemented by subclass');
  }

  getSpecs() {
    return `Car with ${this.engine.fuelType()}`;
  }
}
```

### Participant 6: Sedan (RefinedAbstraction)

```javascript
// OOP: Polymorphism — overrides drive() and getSpecs()
// SOLID: SRP — responsible for sedan-specific behavior
class Sedan extends Car {
  #trunkCapacity;

  constructor(engine, trunkCapacity) {
    super(engine);
    this.#trunkCapacity = trunkCapacity;
  }

  drive() {
    this.engine.start();
    return `Sedan cruising smoothly on the highway. ${this.engine.sound()}`;
  }

  openTrunk() {
    return `Trunk opens (${this.#trunkCapacity} cu ft).`;
  }

  getSpecs() {
    return `Sedan with ${this.engine.fuelType()} | Trunk: ${this.#trunkCapacity} cu ft`;
  }
}
```

### Participant 7: SUV (RefinedAbstraction)

```javascript
// OOP: Polymorphism — overrides drive() and getSpecs()
// SOLID: SRP — responsible for SUV-specific behavior
class SUV extends Car {
  #groundClearance;
  #awd;

  constructor(engine, groundClearance, awd) {
    super(engine);
    this.#groundClearance = groundClearance;
    this.#awd = awd;
  }

  drive() {
    this.engine.start();
    return `SUV tackling off-road terrain. ${this.engine.sound()}`;
  }

  engage4WD() {
    if (this.#awd) {
      return 'All-wheel drive engaged. Traction optimized.';
    }
    return 'This SUV does not have AWD.';
  }

  getSpecs() {
    return `SUV with ${this.engine.fuelType()} | Ground: ${this.#groundClearance}" ${this.#awd ? 'AWD' : '2WD'}`;
  }
}
```

### Language-Specific Notes

- JavaScript has no `abstract` keyword or formal interfaces. The Car base class enforces abstraction through a constructor guard (`new.target === Car`) and a `drive()` method that throws `Error`. The Engine protocol is enforced by convention — any object with the right methods works.
- ES2022+ `#` private fields (`#cylinders`, `#batteryCapacity`) provide hard encapsulation, preventing external code from accessing engine internals directly. This is stronger than the underscore-prefix convention.
- The HybridEngine demonstrates that the Bridge pattern supports nested composition at the implementation level: an implementation can itself wrap multiple other implementations and delegate selectively.
- Fluent composition happens at construction time: `new Sedan(new GasEngine(6, 3.5), 15.5)`. This is the key moment where the abstraction and implementation are joined. The same GasEngine instance could theoretically be shared across multiple Car instances.
- JavaScript's prototypal inheritance means `instanceof` checks work correctly through the Car hierarchy. However, the Bridge pattern encourages programming to the interface rather than checking types.
- The driver demonstrates five configurations showing three car types with three engine types, including the same abstraction with different engines (Sedan + Electric, SUV + Gas) to prove the two hierarchies vary independently.
