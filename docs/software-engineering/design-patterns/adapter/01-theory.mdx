**Intent:** Convert the interface of a class into another interface that clients expect. Adapter lets classes work together that could not otherwise because of incompatible interfaces.

The Adapter pattern acts as a bridge between two incompatible interfaces. It wraps an existing class with a new interface so it becomes compatible with a client's expected interface. This pattern is essential when integrating legacy systems, third-party libraries, or components whose interfaces cannot be modified directly.

In the automotive domain, the Adapter pattern appears wherever two systems speak different protocols or use different physical connectors. The pattern allows them to interoperate without changing either system's internals.

---

## The Problem Without This Pattern

Imagine you own a Tesla Model 3. The car has a proprietary Tesla charge port that communicates using Tesla's own protocol over a unique physical connector. You are on a road trip and pull into a public charging station that only provides CCS (Combined Charging System) connectors. The CCS station expects a standard CCS plug and communicates over the CCS protocol (ISO 15118). Your Tesla cannot charge.

There are three options in this situation:

- **Option A:** Replace the Tesla charge port with a CCS port. This requires disassembling the car, replacing hardware, and rewiring high-voltage components. It voids the warranty and costs thousands of dollars.

- **Option B:** Modify the charging station to support Tesla's protocol. The station owner would need to replace the charging cable and electronics. This is impractical when hundreds of stations are already deployed.

- **Option C:** Use an adapter. A small physical and software adapter plugs into the CCS station on one side and into the Tesla port on the other. The adapter translates the CCS protocol into Tesla's protocol transparently. No car modification, no station modification.

Option C is the Adapter pattern. Without it, you face two rigid interfaces that cannot communicate. Every Tesla owner would be locked into the Tesla Supercharger network, and every CCS charging station would be inaccessible to Tesla vehicles. This fragmentation hurts consumers and slows EV adoption.

The problem is not limited to charging. Consider a mechanic who owns a diagnostic scan tool that speaks the OBD-II protocol. The mechanic plugs it into a 2024 Toyota Camry. The tool expects CAN bus messages at 500 kbps. The Camry's engine control unit (ECU) speaks CAN bus, but at a different baud rate with a different message format. Without an adapter layer in the scan tool, the mechanic cannot read diagnostic codes. The scan tool must include an adapter that translates between the tool's abstract PID (Parameter ID) requests and each manufacturer's specific CAN bus message format.

Another example: a fleet management company integrates telematics devices from multiple vendors into a single dashboard. Each vendor's device exposes a different API: one uses REST over HTTP, another uses MQTT over TCP, and a third uses a proprietary serial protocol. The dashboard defines a standard TelematicsDevice interface. For each vendor, the company writes an adapter that implements the standard interface while wrapping the vendor's specific API. The dashboard never changes. Only the adapters are swapped when the fleet adds a new device vendor.

The Adapter pattern is the solution to all of these scenarios. It provides a clean, maintainable way to bridge incompatible interfaces without modifying either side.

The same problem appears throughout software engineering:

- **Database drivers:** An application written for MySQL needs to talk to PostgreSQL. Without an adapter, the application must be rewritten.
- **Payment gateways:** An e-commerce site integrates with Stripe but needs to also support PayPal. Each has a different API. An adapter lets the site use a single interface.
- **Logging frameworks:** A library logs using Log4j, but the application uses SLF4J. An adapter bridges the two without modifying the library.
- **File formats:** A system reads XML but receives JSON data. An adapter converts between formats without changing either system.

In every case, the core issue is the same: two components that need to communicate expose incompatible interfaces. The Adapter pattern inserts a translation layer that makes them compatible.

## Structure

The Adapter pattern involves four participants:

| Participant | Role | Cars Analogy |
|---|---|---|
| **Client** | The class that uses the Target interface | The EV driver who wants to charge at a CCS station |
| **Target** | The interface the Client expects | The CCS charger standard (ISO 15118 protocol) |
| **Adapter** | Translates Target calls into Adaptee calls | The physical Tesla-to-CCS adapter dongle |
| **Adaptee** | The existing class with a different interface | The Tesla Supercharger port and protocol |

The key insight is that the Adapter implements the Target interface while holding a reference to the Adaptee. The Client sees only the Target interface and never knows it is talking through an adapter. The Adaptee is completely unaware that it is being adapted.

The flow of control is:

1. The Client calls a method on the Target interface (e.g., chargeDC(250)).
2. The Adapter receives the call and translates the parameters into the format the Adaptee expects (e.g., 400V and 625A).
3. The Adapter delegates the call to the Adaptee (e.g., sendCharge(400, 625)).
4. The Adaptee performs its operation using its native protocol.
5. The Adapter returns the result (possibly transformed) to the Client.

### Two Variants of the Adapter Pattern

The Adapter pattern has two distinct implementations:

**Object Adapter (composition-based):** The Adapter holds a reference to an Adaptee instance and delegates calls to it. This is the preferred variant because it uses composition over inheritance, works with any Adaptee subclass, and only requires one Adapter class. The TeslaToCCSAdapter in this guide is an Object Adapter.

**Class Adapter (inheritance-based):** The Adapter inherits from both the Target interface and the Adaptee class. This requires multiple inheritance, which is not supported in many languages (Java, C#, TypeScript). It is used in languages with multiple inheritance support (C++) or when the Adaptee's behavior must be overridden. The Class Adapter does not need to wrap an Adaptee instance because it inherits the Adaptee's methods directly.

The Object Adapter is the recommended approach in most modern languages. It is more flexible, easier to test, and follows the composition-over-inheritance principle.

## Pseudocode — Cars Example

The following pseudocode shows how the Adapter pattern works when a Tesla vehicle is charged at a CCS charging station. The pseudocode is language-agnostic and focuses on the structural relationships between the four participants.

### Architectural Overview

The pseudocode is organized into four logical sections, each corresponding to one participant in the Adapter pattern:

- **Target Interface (CCSCharger):** Defines the contract that the client expects. The client knows only this interface.
- **Adaptee (TeslaSupercharger):** The existing class with an incompatible interface. It is not modified.
- **Adapter (TeslaToCCSAdapter):** Implements the Target interface, wraps the Adaptee, and translates calls.
- **Client (ChargingStation):** Uses the Target interface to charge any vehicle. It never references the Adaptee or the Adapter directly.

### The Translation Logic

The core of the Adapter pattern is the translation performed by the adapter's `chargeDC` method. The translation converts a single parameter (power in kilowatts) into two parameters (voltage and amperage):

| CCS Parameter | Tesla Parameters | Calculation |
|---|---|---|
| powerKw = 250 | voltage = 400, amperage = 625 | I = P / V = 250000 / 400 = 625 |
| powerKw = 350 | voltage = 400, amperage = 875 | I = P / V = 350000 / 400 = 875 |
| powerKw = 750 | voltage = 800, amperage = 938 | I = P / V = 750000 / 800 = 938 |

The adapter also selects the voltage tier based on the power level. Powers under 350 kW use the standard 400V CCS tier. Powers of 350 kW or higher use the 800V ultra-fast charging tier. This voltage tier selection is part of the adapter's translation logic and is invisible to both the client and the adaptee.

### Complete Pseudocode

```
// ── Target Interface ──
// The interface the charging station client expects.
interface CCSCharger {
    chargeDC(powerKw: Integer) -> String
}

// ── Adaptee ──
// The Tesla-specific charging hardware that uses a proprietary protocol.
class TeslaSupercharger {
    method sendCharge(voltage: Integer, amperage: Integer) -> String {
        powerKw = (voltage * amperage) / 1000
        return "Tesla charging at " + powerKw + " kW (" + voltage + "V / " + amperage + "A)"
    }
}

// ── Adapter ──
// Wraps a TeslaSupercharger to make it work as a CCS charger.
class TeslaToCCSAdapter implements CCSCharger {
    private adaptee: TeslaSupercharger

    constructor(adaptee: TeslaSupercharger) {
        this.adaptee = adaptee
    }

    method chargeDC(powerKw: Integer) -> String {
        // Convert CCS standard call to Tesla proprietary call.
        // CCS typically operates at 400V (or 800V for newer stations).
        voltage = 400
        amperage = ceiling((powerKw * 1000) / voltage)
        return this.adaptee.sendCharge(voltage, amperage)
    }
}

// ── Client ──
// The charging station that only knows the CCS interface.
class ChargingStation {
    private name: String

    constructor(name: String) {
        this.name = name
    }

    method serviceVehicle(charger: CCSCharger, carModel: String, requiredKw: Integer) {
        print(this.name + ": Charging " + carModel)
        result = charger.chargeDC(requiredKw)
        print("  " + result)
    }
}

// ── Driver ──
// Example usage demonstrating the adapter.
function main() {
    teslaHardware = new TeslaSupercharger()
    adapter = new TeslaToCCSAdapter(teslaHardware)
    station = new ChargingStation("Highway CCS Station")

    // The station thinks it is talking to a standard CCS charger.
    // The adapter is translating everything transparently.
    station.serviceVehicle(adapter, "Tesla Model 3", 250)
    station.serviceVehicle(adapter, "Tesla Model Y", 250)
    station.serviceVehicle(adapter, "Tesla Model S", 200)
    station.serviceVehicle(adapter, "Tesla Cybertruck", 350)
}
```

Expected output:

```
Highway CCS Station: Charging Tesla Model 3
  Tesla charging at 250 kW (400V / 625A)
Highway CCS Station: Charging Tesla Model Y
  Tesla charging at 250 kW (400V / 625A)
Highway CCS Station: Charging Tesla Model S
  Tesla charging at 200 kW (400V / 500A)
Highway CCS Station: Charging Tesla Cybertruck
  Tesla charging at 350 kW (400V / 875A)
```

## How It Works — Step by Step

The Adapter pattern follows a precise sequence of interactions. Here are the detailed steps using the Tesla-to-CCS charging scenario:

**Step 1: Driver Initiates Charge Request**

The Driver (Client) approaches the ChargingStation and requests to charge a specific Tesla model. The driver specifies the vehicle model and the required charging power in kilowatts. The Driver knows only the ChargingStation interface — how to plug in and request a charge. The Driver does not know or care about the internal protocol used by the charging hardware.

```
driver.chargeAtStation(station, "Tesla Model 3", 250)
```

From the Driver's perspective, the station either works or it does not. The Driver should not need to know about CCS vs Tesla protocols. The adapter hides this complexity.

**Step 2: ChargingStation Delegates to the Target Interface**

The ChargingStation receives the request from the Driver. The station was designed to work with any CCSCharger. It has no knowledge of Tesla's proprietary protocol, no special case for Tesla vehicles, and no conditional logic for different car brands. It simply calls the standard CCSCharger interface that was provided to it during setup.

```
// Inside ChargingStation.serviceVehicle:
charger.chargeDC(250)
```

This call is the same whether the CCSCharger is native hardware or an adapter. The station treats all chargers uniformly. This is the key benefit of the Adapter pattern: the client code is clean, generic, and brand-agnostic.

**Step 3: Adapter Receives the Target Method Call**

The TeslaToCCSAdapter object implements CCSCharger, so it receives the chargeDC(250) call. From the station's perspective, this is just another CCSCharger doing its job. But the adapter knows that the underlying hardware does not understand CCS. The adapter must now perform its core responsibility: translation.

The adapter examines the incoming parameters. The CCS standard says "charge at 250 kW." The adapter needs to convert this into something the Tesla hardware can process.

**Step 4: Adapter Translates CCS Parameters to Tesla Parameters**

The adapter performs the translation math. The translation involves two decisions:

- **Voltage tier selection:** Is this a standard or ultra-fast charge? For powers up to 350 kW, CCS uses 400V nominal. For powers above 350 kW, CCS uses 800V nominal (sometimes called ultra-fast or HPC). The adapter selects accordingly.
- **Amperage calculation:** Power (W) = Voltage (V) * Current (A), so A = P / V. For 250 kW at 400V: amperage = 250000 / 400 = 625A. The result is rounded up using ceiling to guarantee the requested power level is met.

```
voltage = 400        // CCS nominal voltage for this power level
amperage = ceil(250000 / 400)  // = 625A
```

The adapter has now converted a single CCS parameter (250 kW) into a pair of Tesla parameters (400V, 625A). This is the complete translation for this simple case. Real adapters might also translate authentication tokens, charging curves, thermal limits, and safety handshake messages.

**Step 5: Adapter Delegates the Translated Call to the Adaptee**

With the translated parameters ready, the adapter calls the TeslaSupercharger's sendCharge method:

```
this.adaptee.sendCharge(400, 625)
```

This is a direct delegation. The adapter is not adding behavior or modifying the result. It is purely a translator. The TeslaSupercharger receives the call as if it came from a native Tesla controller. It has no idea that the call was originally a CCS request.

**Step 6: Adaptee Executes Its Native Charging Operation**

The TeslaSupercharger performs the charge operation using its native logic. It calculates the effective power from the voltage and amperage:

```
powerKw = (400 * 625) / 1000 = 250
```

It then manages the electrical flow: it engages the contactors, ramps up the current, monitors temperature and voltage, and maintains the charge session. It returns a confirmation string describing the operation.

```
return "Tesla charging at 250 kW (400V / 625A)"
```

**Step 7: Adapter Returns the Result to the Client**

The adapter receives the result string from the TeslaSupercharger. In this example, the string is returned directly with no additional transformation. In a more complex system, the adapter might also:

- Translate Tesla-specific error codes into CCS-standard error codes.
- Convert Tesla's charging state enum (e.g., CHARGING, FAULT, IDLE) into CCS equivalents.
- Map Tesla's thermal throttling warnings to CCS-standard power reduction messages.
- Log the translation for debugging or compliance purposes.

**Step 8: Client Observes a Seamless Charging Session**

The ChargingStation receives the result and reports success to the Driver. The station logged the session in its standard format. It never knew it was charging a Tesla through an adapter. The Driver sees a successful charge and unplugs.

From the station's perspective: it serviced a standard CCS-compatible vehicle. From the Tesla's perspective: it received a charge through its native protocol. The adapter is invisible to both sides.

This sequence repeats identically for every vehicle in the fleet. The same ChargingStation can service a Tesla Model Y (250 kW), a Tesla Model S (200 kW), and a Tesla Cybertruck (350 kW) by passing different requiredKw values. The adapter performs the appropriate translation for each power level, selecting the correct voltage tier and calculating the right amperage every time. The station code never changes.

## When to Use

Use the Adapter pattern when:

**You need to integrate a class whose interface does not match what your client expects.** This is the textbook case. You have a Tesla Supercharger and a CCS charging station. The adapter bridges them. In software, this happens constantly: your application uses one logging interface but the library exposes another; your frontend expects JSON but the legacy backend returns XML. The adapter is a thin translation layer that makes the two sides interoperable.

**You want to reuse an existing class but cannot modify it.** The class might be from a third-party library, a legacy system that nobody has touched in years, or a compiled binary. You cannot add implements CCSCharger to TeslaSupercharger because you do not own the source. The adapter wraps it without modification. This is one of the most common use cases for the pattern: you have working, tested code with the wrong interface, and you need to make it fit a new context without risk of regression.

**You want to create a reusable class that cooperates with unpredictable or future classes.** A well-designed client depends on an interface, not a concrete class. Anyone can write a new adapter to make their class work with your client. Tesla could build a CCS-to-Tesla adapter, and other EV manufacturers could build adapters too. The charging station just sees the interface. This is the "open for extension" principle in action: the client is closed for modification but open for new adapters.

**You need to add functionality to multiple unrelated classes.** Without a common interface, you would need to modify each class individually. With an adapter, you define the Target interface once and write one adapter per class. The payment gateway scenario is classic: StripeAdapter, PayPalAdapter, SquareAdapter all implement a common PaymentGateway interface. Adding a new payment provider means writing one new adapter class, not modifying the checkout system.

**You are working in a system that evolves independently on both sides of a boundary.** The charging station firmware is updated by the station manufacturer. The Tesla charge port firmware is updated by Tesla. Neither coordinates with the other. The adapter sits in the middle and handles the translation, absorbing changes on either side. If Tesla changes its protocol, only the adapter is updated. The charging station never changes. If the CCS standard evolves, the adapter is updated to match. The Tesla hardware never changes.

**You need to bridge between old and new versions of a system during a migration.** When migrating from a legacy system to a new one, the adapter can wrap the old system behind the new interface. Features can be migrated incrementally. Each feature that is moved to the new system simply removes the adapter and talks directly to the new implementation. This is a common pattern in database migration, API versioning, and hardware abstraction layers.

**You are writing a library or framework that needs to work with multiple third-party implementations.** The Java JDBC specification is the canonical example. The JDBC API defines a standard interface for database access. Each database vendor writes a JDBC driver (adapter) that translates JDBC calls into their database's wire protocol. Application code written against JDBC works with any database without modification.

## When NOT to Use

Avoid the Adapter pattern when:

**You control both interfaces and can modify one to match.** If you own the source for both the client and the service, it is simpler to change one interface to match the other. Adding an adapter when a simple refactor would work adds unnecessary complexity. For example, if you build both the charging station and the car, just use the same connector on both. The adapter is only valuable when you cannot or should not modify one of the interfaces.

**The interface mismatch is fundamental and cannot be bridged.** The adapter pattern assumes a straightforward translation between interfaces. If the two systems have fundamentally different data models, transaction semantics, or error handling, an adapter may introduce subtle bugs. An adapter cannot fix a semantic mismatch between "charge until full" and "charge for exactly 30 minutes." If the concepts on either side of the adapter are not equivalent, the adapter will produce incorrect results. The translation must be a faithful mapping; if it is not, the pattern is being misapplied.

**Performance is critical and the adapter adds latency.** Each adapter call adds a layer of indirection. The translation logic itself takes CPU time. In the charging scenario, microseconds of adapter overhead are irrelevant. But in a real-time engine control unit, an anti-lock braking system, or a high-frequency trading system, the extra function call and translation overhead might be unacceptable. Profile the adapter to confirm the overhead is within your latency budget.

**You need to adapt many methods or a complex hierarchy.** If the Adaptee exposes fifty methods and the Target needs all of them, the adapter becomes a fifty-method wrapper. Each method must be individually delegated and possibly transformed. This is tedious to write, hard to test, and fragile. Consider whether the Facade or Bridge pattern might serve better. The Facade pattern simplifies a complex interface rather than converting it. The Bridge pattern separates abstraction from implementation so they can vary independently.

**The system does not change over time.** If neither the client interface nor the adaptee interface will ever change, there is no future benefit to having the adapter. The indirection is pure overhead with no payoff. The adapter's value comes from absorbing change, not from the immediate functionality. If both sides are stable and the translation is straightforward, a direct integration is simpler and more maintainable.

**You are adapting a class that you will replace entirely soon.** Writing an adapter for a class that is scheduled for deprecation is wasted effort. The adapter will need to be removed when the adaptee is replaced. If the replacement is imminent, it is better to change the client to work with the new interface directly, bypassing the adapter layer entirely.

## Real-World Occurrences

The Adapter pattern appears throughout software engineering and everyday life:

### 1. Tesla-to-CCS Charging Adapters

This is the most direct real-world example. Tesla vehicles in regions using CCS (Europe, parts of Asia) come with a CCS-to-Tesla adapter. The adapter physically converts the CCS plug to the Tesla connector. On the software side, the adapter translates the CCS communication protocol (ISO 15118) into Tesla's protocol, including charging parameters, authentication, and safety handshakes. Multiple manufacturers sell these adapters, including Tesla's own official CCS Combo 1 adapter. The adapter is a small dongle that the driver carries in the car and plugs in when needed. The adapter knows nothing about the car's battery state or the station's power capacity — it is purely a translator.

In 2023, Tesla began opening its Supercharger network to non-Tesla EVs in North America. This required a reverse adapter: the Magic Dock, which is a CCS adapter built into the Supercharger stall itself. A non-Tesla EV with a CCS port can pull up to a Tesla Supercharger, use the Magic Dock, and charge. The Magic Dock is an Adapter pattern implementation: it presents a CCS interface to the vehicle while internally speaking Tesla's protocol to the Supercharger hardware.

### 2. Database Drivers (JDBC, ODBC, ADO.NET)

JDBC (Java Database Connectivity) is a textbook example of the Adapter pattern applied at scale. An application writes SQL queries against the JDBC interface. Each database vendor (MySQL, PostgreSQL, Oracle, SQL Server) provides a JDBC driver that adapts the JDBC interface to their proprietary wire protocol. The application never changes; only the adapter is swapped. This is identical in structure to the CCSCharger interface with different adapters for different car brands. The database driver architecture is one of the most successful examples of the Adapter pattern in the history of software engineering, enabling portable database access for decades.

ODBC (Open Database Connectivity) for C/C++ and ADO.NET for C# follow the same pattern. Each defines a standard data access interface, and each database vendor provides a driver that implements that interface. The application code is decoupled from the database engine, allowing organizations to switch databases by simply changing the connection string and driver.

### 3. The Java Collections `asList` and `asLifoQueue`

The Java standard library includes adapter methods that wrap one collection type to behave as another. `Arrays.asList()` wraps an array and makes it implement the `List` interface. `Collections.asLifoQueue()` wraps a `Deque` and makes it implement a `Queue` interface. These are pure adapter methods that convert one interface to another without modifying the underlying data structure. They are lightweight: `asList()` does not copy the array, it provides a view that delegates to the original array. This is the Object Adapter variant in its simplest form.

### 4. Payment Gateway Integrations

E-commerce platforms like Shopify, WooCommerce, and Stripe define a common payment processing interface. Each supported payment provider (Visa, PayPal, Apple Pay, Klarna) implements an adapter. The platform calls `charge(amount, currency)` on the interface; the adapter translates this into the provider's specific API calls. This lets merchants add new payment methods without changing their checkout flow. When the platform adds a new payment provider, the development team writes one adapter class. The rest of the checkout system remains unchanged. This is a textbook example of the Open/Closed Principle enabled by the Adapter pattern.

### 5. OBD-II Diagnostic Adapters

An OBD-II (On-Board Diagnostics) port exposes a standard interface to read vehicle diagnostic data. Different car manufacturers implement the OBD-II standard differently under the hood (some use CAN bus, some use K-Line, some use PWM). An OBD-II scan tool includes an adapter layer that speaks the standard OBD-II protocol to the mechanic's tablet while translating to each manufacturer's specific wiring and protocol. The mechanic sees a uniform interface regardless of whether the car is a Ford, Toyota, or BMW. The adapter layer is critical because the mechanic cannot be expected to carry a different scan tool for each car brand. The scan tool's adapter handles the translation invisibly, just like the TeslaToCCSAdapter in the running example.

### 6. Android's `ArrayAdapter` and `RecyclerView.Adapter`

The Android framework uses the Adapter pattern extensively in its UI toolkit. `ArrayAdapter` and `RecyclerView.Adapter` convert data sources (lists, arrays, database cursors) into view items that the UI can render. The `ListView` or `RecyclerView` (Client) calls methods on the `Adapter` interface to get the data for each position. The adapter translates between the raw data format and the View objects the UI expects. This is the same structural relationship as the CCSCharger pattern, applied to user interface rendering rather than charging infrastructure.

### 7. USB-C to HDMI / DisplayPort Adapters

A USB-C port exposes the DisplayPort Alt Mode standard. An HDMI monitor expects a different electrical and protocol interface. The adapter dongle translates between USB-C Alt Mode signals and HDMI signals. This is a physical-world adapter that follows the same structure: the laptop (Client) uses USB-C (Target), the dongle (Adapter) translates, and the monitor (Adaptee) receives standard HDMI. The adapter handles signal voltage levels, pin mapping, and protocol negotiation. Neither the laptop nor the monitor knows the adapter exists — each sees its expected interface.

### 8. Logging Framework Bridges (SLF4J, Apache Commons Logging)

SLF4J (Simple Logging Facade for Java) is an adapter layer that allows application code to write log statements against a generic API while delegating to any underlying logging framework (Logback, Log4j, java.util.logging). The application calls `logger.info("message")` on the SLF4J interface. SLF4J's adapter translates the call into the appropriate call for whichever logging framework is on the classpath. This is the Adapter pattern applied to the cross-cutting concern of logging. The application never imports Logback or Log4j directly; it only imports SLF4J. The adapter handles the translation.

## Comparison with Related Patterns

The following table compares Adapter with other structural patterns that might seem similar.

| Pattern | Purpose | Key Difference from Adapter |
|---|---|---|
| **Adapter** | Convert one interface to another so incompatible classes can work together | Focuses on interface incompatibility; wraps an existing object without changing its behavior |
| **Bridge** | Separate an abstraction from its implementation so both can vary independently | Designed up front to allow multiple implementations; Adapter is applied retroactively to make things work |
| **Facade** | Provide a simplified interface to a complex subsystem | Simplifies a complex interface; Adapter converts one exact interface to another without simplifying |
| **Decorator** | Add responsibilities to an object dynamically without changing its interface | Changes behavior while keeping the same interface; Adapter changes the interface without adding behavior |
| **Proxy** | Control access to another object, often for lazy loading, security, or logging | Keeps the same interface and controls access; Adapter changes the interface to enable interoperation |
| **Strategy** | Define a family of interchangeable algorithms | Provides different implementations of the same interface; Adapter makes two different interfaces work together |

The most common confusion is between Adapter and Bridge. Both involve an object that delegates to another. The distinction is intent and timing:

- **Adapter** is about making things work together that were not designed to work together. It is applied retroactively as a patch or integration layer.
- **Bridge** is about designing an abstraction that can have multiple implementations independently. It is planned in advance during architecture design.

A real car example makes this clear. The Tesla-to-CCS adapter is an **Adapter** — it was created after both Tesla and CCS existed, to bridge the incompatibility. A car's steering system is a **Bridge** — the steering wheel (abstraction) is designed from the start to work with multiple steering implementations (hydraulic, electric, steer-by-wire) that can vary independently.

The difference between Adapter and Facade is also worth highlighting. A Facade provides a simplified interface to a complex subsystem. It typically reduces the number of methods and hides complexity. An Adapter converts one interface to another without simplifying — it preserves the full functionality of the target interface. The Facade says "I will make this easier to use." The Adapter says "I will make this fit the expected interface."

Adapter and Decorator are sometimes confused because both involve wrapping an object. The distinction is that Decorator adds behavior while preserving the same interface. Adapter changes the interface without adding behavior. A Decorator on a CCSCharger might add logging, caching, or rate limiting — but it still implements CCSCharger. An Adapter changes CCSCharger to something else entirely.

### When Adapter Becomes the Wrong Pattern

There are cases where a design looks like an Adapter but is actually a different pattern in disguise:

- **Leaky Adapter:** The adapter translates the interface but leaks adaptee-specific concepts (error codes, parameter names, threading models) into the client. A real adapter must provide a clean translation. If the client has to handle Tesla-specific timeout values, the adapter is not doing its job.
- **God Adapter:** The adapter implements multiple unrelated interfaces for a single adaptee. This is a sign that the adapter is doing too much. Split it into multiple adapters, one per target interface.
- **Identity Adapter:** The adapter translates method calls without any actual transformation. The parameters and return types are identical. This is a sign that the adapter is unnecessary. Remove it and let the client call the adaptee directly.

## Summary

## Summary

The Adapter pattern solves a common and frustrating problem: two pieces of software (or hardware) that need to talk to each other but speak different protocols. The pattern introduces a translator object that implements the expected interface on behalf of an incompatible object.

In the car domain, the Adapter pattern appears everywhere: charging adapters for EVs, diagnostic adapters for OBD-II scan tools, infotainment adapters for phone projection, and CAN bus adapters for telematics. Each one follows the same structure: the Client sees a standard interface, the Adapter translates, and the Adaptee continues using its native protocol unchanged.

The Adapter pattern is a go-to tool for system integration. Whenever you encounter the words "we need to make this work with that," you are likely looking at an Adapter problem. The pattern does not make the system simpler, but it makes the incompatible parts work together without changing either one — and that is often the most practical solution available.

The key insights to remember about the Adapter pattern are:

- **Composition over inheritance:** The Object Adapter (composition) is preferred over the Class Adapter (inheritance) because it is more flexible, works with any adaptee at runtime, and does not require multiple inheritance.
- **Single Responsibility:** The adapter has one job: translation. It does not add behavior, modify the adaptee's internals, or manage the adaptee's lifecycle. Keeping the adapter focused on translation makes it easy to test and maintain.
- **Interface Segregation:** The target interface should be minimal. The adapter should only expose the methods the client actually needs. Exposing unnecessary methods increases the adapter's surface area and makes it harder to implement correctly.
- **Testability:** The adapter can be tested independently of the client by providing a mock adaptee. The client can be tested independently of the adapter by providing a mock target implementation. This decoupling is one of the pattern's main benefits.

The Adapter pattern is foundational to software engineering. It appears in nearly every large system, from database drivers and logging frameworks to UI toolkits and cloud SDKs. Understanding the Adapter pattern gives you a powerful tool for integrating systems that were not designed to work together — and that is a skill you will use constantly in real-world development.

## Checklist

When implementing the Adapter pattern, verify each point against your design:

- [ ] The Client depends only on the Target interface, never on the Adapter or Adaptee directly
- [ ] The Adapter implements the Target interface completely (all methods have real translation logic)
- [ ] The Adapter holds a reference to the Adaptee (composition) rather than inheriting from it
- [ ] The translation logic is correct for all expected parameter ranges and edge cases
- [ ] Error codes and exceptions from the Adaptee are mapped to equivalents in the Target interface
- [ ] The Adapter does not add behavior beyond translation (resisting scope creep into Decorator territory)
- [ ] Each Adaptee class gets its own Adapter class (one adapter per incompatible interface)
- [ ] The Adapter is tested independently from the Client to verify translation accuracy
- [ ] Performance overhead of the adapter layer is acceptable for the use case
- [ ] A simpler refactor of the Adaptee or Client interface is not available (for retroactive adapters)
