## Intent

Define a representation for a domain-specific language's grammar along with an interpreter that uses the representation to evaluate sentences in that language. The Interpreter pattern maps each grammatical rule to a class, composing expressions into tree structures that can be recursively evaluated. In the Stock Market domain, this enables you to build a flexible stock-filtering query language where traders can express complex criteria like "stocks with price above 150 AND volume above 1000000 OR ticker equals AAPL" without hardcoding each filter combination.

## The Problem Without This Pattern

Imagine you are building a stock screener application that lets traders filter stocks by various criteria. Without the Interpreter pattern, every new filter combination requires either new code or a tangled mess of conditionals.

Consider a typical screening workflow. A trader wants to find stocks where the price is above 150, the volume is above 1000000, and the ticker starts with "A". Without the pattern, you end up with deeply nested conditionals or combinatorial explosion of filter functions:

```python
# Problem: Every filter combination requires custom logic
# Adding new operators or conditions is error-prone

def filter_stocks(stocks, min_price=0, max_price=float('inf'),
                  min_volume=0, ticker_prefix=None, sector=None):
    results = []
    for stock in stocks:
        if stock.price > min_price and stock.price < max_price:
            if stock.volume > min_volume:
                if ticker_prefix is None or stock.ticker.startswith(ticker_prefix):
                    if sector is None or stock.sector == sector:
                        results.append(stock)
    return results

# Problem: What about OR conditions? What about NOT?
# Each new operator means new parameters and new nesting
# There is no way to dynamically compose filters at runtime

def filter_stocks_or(stocks, conditions):
    # This quickly becomes unmanageable
    # Mixing AND and OR means rewriting the whole function
    pass  # BAD: falls apart for complex boolean logic
```

The core problem is that the grammar of stock filter expressions is implicit — it lives inside nested if-statements rather than being represented as composable objects. Adding negation (NOT), grouping (parentheses), or new comparison operators requires invasive changes across the filtering code.

## Pseudocode — Stock Market Example

This pseudocode implements a stock screening query language where traders can compose filter expressions using AND, OR, and NOT operators. Each expression node knows how to evaluate itself against a stock data record.

```
// PSEUDOCODE — Interpreter Pattern in Stock Market Domain
// Participants: Expression (Abstract), PriceCheck, VolumeCheck,
// TickerMatch, SectorMatch (Terminal), AndExpr, OrExpr, NotExpr
// (Nonterminal), StockData (Context), StockScreener (Client)

[ABSTRACT EXPRESSION]
interface Expression {
    method interpret(stock: StockData) -> boolean
}

[CONTEXT]
class StockData {
    ticker: string
    price: float
    volume: integer
    sector: string
}

[TERMINAL EXPRESSIONS]
class PriceGreaterThan implements Expression {
    threshold: float

    method interpret(stock) -> boolean {
        return stock.price > self.threshold
    }
}

class VolumeGreaterThan implements Expression {
    threshold: integer

    method interpret(stock) -> boolean {
        return stock.volume > self.threshold
    }
}

class TickerEquals implements Expression {
    ticker: string

    method interpret(stock) -> boolean {
        return stock.ticker == self.ticker
    }
}

class SectorEquals implements Expression {
    sector: string

    method interpret(stock) -> boolean {
        return stock.sector == self.sector
    }
}

[NONTERMINAL EXPRESSIONS]
class AndExpr implements Expression {
    left: Expression
    right: Expression

    method interpret(stock) -> boolean {
        return left.interpret(stock) AND right.interpret(stock)
    }
}

class OrExpr implements Expression {
    left: Expression
    right: Expression

    method interpret(stock) -> boolean {
        return left.interpret(stock) OR right.interpret(stock)
    }
}

class NotExpr implements Expression {
    expr: Expression

    method interpret(stock) -> boolean {
        return NOT expr.interpret(stock)
    }
}

[CLIENT]
class StockScreener {
    method screen(stocks: List<StockData>, expression: Expression) -> List<StockData> {
        results = []
        for each stock in stocks:
            if expression.interpret(stock):
                results.append(stock)
        return results
    }
}

// ── Usage ──
stocks = [
    StockData("AAPL", 175.50, 85000000, "Technology"),
    StockData("GOOGL", 142.30, 32000000, "Technology"),
    StockData("MSFT", 378.90, 25000000, "Technology"),
    StockData("JPM", 198.40, 12000000, "Financial"),
    StockData("BAC", 39.20, 45000000, "Financial"),
]

screener = new StockScreener()

// Query: price > 150 AND volume > 30000000
expression = new AndExpr(
    new PriceGreaterThan(150.0),
    new VolumeGreaterThan(30000000)
)

results = screener.screen(stocks, expression)
// Expected: AAPL (175.50, 85M) and BAC (39.20, 45M) — wait, BAC price is 39.20 < 150
// Actually: AAPL (175.50, 85M) — GOOGL volume 32M > 30M but price 142.30 < 150
// MSFT price 378.90 > 150 but volume 25M < 30M
// Result: AAPL only

// Query: sector = "Financial" OR ticker = "MSFT"
expression2 = new OrExpr(
    new SectorEquals("Financial"),
    new TickerEquals("MSFT")
)

results2 = screener.screen(stocks, expression2)
// Expected: MSFT, JPM, BAC
```

## How It Works — Step by Step

1. **Define the grammar.** You identify the rules of your stock query language: comparisons (price > N, volume > N, ticker = X, sector = X) are terminal symbols; logical combinators (AND, OR, NOT) are nonterminal symbols that compose other expressions. Each grammar rule becomes a class.

2. **Build the expression tree (AST).** The client (StockScreener or the trader's code) composes expression objects into a tree. For example, `PriceGreaterThan(150)` and `VolumeGreaterThan(30000000)` are leaf nodes combined through an `AndExpr` internal node. The tree is the abstract syntax tree (AST) of the query.

3. **Interpretation begins at the root.** Calling `interpret()` on the root `AndExpr` triggers recursive evaluation. The root delegates to its left and right children.

4. **Terminal expressions evaluate against context.** `PriceGreaterThan.interpret()` reads `stock.price` from the StockData context and compares it to its stored threshold. It returns a boolean — either the stock passes this filter or it does not.

5. **Nonterminal expressions combine results.** `AndExpr.interpret()` calls `left.interpret(stock) && right.interpret(stock)` — if both sides return true, the AND passes. `OrExpr` uses `||`. `NotExpr` inverts.

6. **The client iterates and collects.** `StockScreener.screen()` loops through the stock list, calling `interpret()` on the root expression for each stock. Stocks that return true are collected into the result set. The entire query logic is embedded in the tree structure — adding a new operator means adding a new class, not modifying existing code.

## When to Use

- You need to evaluate expressions in a domain-specific language whose grammar is relatively simple and stable
- A stock screener, rules engine, or configuration filter needs to support composable, user-defined conditions
- The grammar can be represented as an abstract syntax tree with a small, fixed set of terminal and nonterminal symbols
- You want the Open/Closed Principle benefit: new expression types can be added without changing the interpreter or existing expressions
- The expression evaluation logic benefits from being decentralized into each expression class rather than centralized in a single switch statement

## When NOT to Use

- The grammar is large or complex (hundreds of rules) — a parser generator (ANTLR, PEG.js, Bison) or embedded DSL (LINQ, Pandas queries) is more appropriate
- Performance is critical for deeply nested expressions — the recursive tree walk adds overhead compared to compiled or JIT-compiled alternatives
- The grammar changes frequently — every grammar change means modifying or adding expression classes, which can become maintenance-heavy
- A simpler solution exists — many filtering problems can be solved with a map of lambdas, function composition, or a rules engine rather than a full interpreter
- The expressions are simple enough that a single function with parameters suffices (e.g., `filter(stocks, min_price=150, min_volume=30000000)`)

## Real-World Occurrences

- **Java `java.util.regex.Pattern`** — The standard regex engine uses the Interpreter pattern internally. Each regex construct (literal, character class, alternation, quantifier) is a subclass of `Pattern.Node`, and matching is the recursive `match()` method that walks the compiled pattern tree.
- **Python `ast` module** — Python's Abstract Syntax Tree module represents Python source code as a tree of `AST` node objects. While not an interpreter per se, the pattern's structure of a recursive tree of expression nodes maps directly to how Python compiles and inspects code.
- **.NET LINQ `Expression<T>`** — LINQ represents queries as expression trees (`System.Linq.Expressions.Expression`). Constant, Parameter, Binary (And, Or), MethodCall, and other expression types are subclasses that can be compiled or interpreted, enabling LINQ providers to translate C# queries to SQL, OData, and other targets.
- **Rust `pest` parser** — The pest parsing library represents grammar rules as an enum (`Rule`) and provides an iterator of parsed tokens. Users implement the `parse()` method that matches against rule variants, mirroring the Interpreter pattern's recursive descent approach.

## Comparison with Related Patterns

| Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Composite** | Composite composes objects into tree structures to represent part-whole hierarchies; Interpreter uses the same tree structure but adds an interpretation/execution dimension | Use Composite when you only need tree structure; use Interpreter when the tree represents a grammar that needs evaluation |
| **Strategy** | Strategy defines a family of interchangeable algorithms; Interpreter evaluates a grammar against a context | Use Strategy when you switch between whole algorithms; use Interpreter when you compose expressions from smaller building blocks |
| **Visitor** | Visitor separates an algorithm from the object structure it operates on; Interpreter embeds evaluation in the expression objects themselves | Use Visitor when expression types are stable but operations change; use Interpreter when operations are stable but expression types may grow |
| **Builder** | Builder constructs complex objects step by step; it is often used to build the AST that Interpreter then evaluates | Use Builder to construct the expression tree; use Interpreter to evaluate it |
