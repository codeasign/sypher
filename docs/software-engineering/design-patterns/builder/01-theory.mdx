## Intent

The Builder pattern separates the construction of a complex object from its representation so that the same construction process can create different representations. It lets you construct products step by step, vary the internal representation without changing the construction algorithm, and reuse the same construction code for different object configurations.

## The Problem Without This Pattern

Consider a banking system that processes loan applications. A loan application contains dozens of fields: applicant personal information, loan type, requested amount, repayment term, interest rate, collateral details, property appraisal information, vehicle identification, credit score ranges, and document attachments. Each loan type (PersonalLoan, Mortgage, AutoLoan) requires a different subset of these fields. Mortgages need property addresses and appraisal values. AutoLoans need vehicle VINs and dealership information. PersonalLoans need employment verification but no collateral.

Without the Builder pattern, developers resort to one of these anti-patterns:

### Telescoping Constructor Anti-Pattern

```python
# BAD: Telescoping constructors — every permutation needs a new constructor
class LoanApplication:
    def __init__(self, applicant_name, ssn, loan_type, amount, term):
        # Minimal constructor
        ...

    def __init__(self, applicant_name, ssn, loan_type, amount, term,
                 interest_rate):
        # Added interest rate — new constructor
        ...

    def __init__(self, applicant_name, ssn, loan_type, amount, term,
                 interest_rate, collateral_description, collateral_value):
        # Added collateral — yet another constructor
        ...

    # This grows combinatorially. A loan with 12 optional fields
    # would need thousands of constructor overloads.
```

The telescoping approach creates an explosion of constructor signatures. Developers must count positional arguments carefully, and adding a single new optional field forces new overloads across all consuming code.

### Mega-Factory Anti-Pattern

```python
# BAD: One massive method that does everything
class LoanFactory:
    def create_loan(self, loan_type, **kwargs):
        app = LoanApplication()
        app.applicant_name = kwargs.get("applicant_name")
        app.ssn = kwargs.get("ssn")
        app.loan_type = loan_type
        app.amount = kwargs.get("amount", 0)
        app.term = kwargs.get("term", 36)
        app.interest_rate = kwargs.get("interest_rate", 0.0)

        if loan_type == "mortgage":
            app.property_address = kwargs.get("property_address", "")
            app.appraisal_value = kwargs.get("appraisal_value", 0)
            app.down_payment = kwargs.get("down_payment", 0)
        elif loan_type == "auto":
            app.vehicle_vin = kwargs.get("vehicle_vin", "")
            app.vehicle_make = kwargs.get("vehicle_make", "")
            app.vehicle_model = kwargs.get("vehicle_model", "")
            app.dealership_name = kwargs.get("dealership_name", "")

        app.validate()
        return app
```

This mega-factory is fragile, unreadable, and untestable. A key mistyped as `"interest_rate"` instead of `"interest_rate"` silently defaults to zero. Adding a new loan type requires modifying the factory method and adding conditional branches. The object is created in an inconsistent state until all setters are called.

### What the Builder Pattern Provides

The Builder pattern replaces these anti-patterns with a fluent, step-by-step construction API. Each loan type gets its own builder that knows exactly which fields to configure. The director orchestrates the steps without knowing the concrete builder type. The result is readable, type-safe, and extensible.

### Participants

1. **Builder (interface)** — Declares the abstract steps for constructing a product: `setPersonalInfo()`, `setLoanTerms()`, `setCollateral()`, `addDocuments()`, `getResult()`.
2. **ConcreteBuilder** — Implements the Builder interface for a specific loan type. `PersonalLoanBuilder` sets only personal and loan fields. `MortgageBuilder` additionally sets property appraisal and collateral. `AutoLoanBuilder` sets vehicle information and dealership details.
3. **Director** — Defines the order of construction steps. `constructPersonalLoan()`, `constructMortgage()`, and `constructAutoLoan()` each call the appropriate builder steps in sequence.
4. **Product** — The complex object being built. `LoanApplication` is immutable after construction and exposes only `getSummary()` and read-only accessors.

## Pseudocode — Banking Example

```
// PSEUDOCODE — Builder Pattern in Banking Domain
// Participants labeled with [ROLE]

// ── [PRODUCT] ──
class LoanApplication:
    // The complex object constructed step by step
    fields: personalInfo dict, loanType string, amount float, term int,
            interestRate float, collateral dict, propertyAppraisal dict,
            vehicleInfo dict

    method getSummary():
        return formatted string with all fields
        ex: "Mortgage[Alice]: $350000 over 360mo @ 6.25% APR — 123 Main St"

// ── [BUILDER INTERFACE] ──
interface LoanApplicationBuilder:
    method setPersonalInfo(name, ssn, employment):
        // Store applicant personal details
    method setLoanTerms(loanType, amount, term):
        // Store loan amount and repayment period
    method setInterestRate(rate):
        // Store annual percentage rate
    method setCollateral(description, value):
        // Store collateral details (optional for personal loans)
    method addDocument(docType, reference):
        // Append a document reference
    method getResult():
        // Return the fully constructed LoanApplication
        returns LoanApplication

// ── [CONCRETE BUILDER: PersonalLoan] ──
class PersonalLoanBuilder implements LoanApplicationBuilder:
    private personalInfo, loanType, amount, term, interestRate
    private collateral = null  // Personal loans typically unsecured
    private documents = []

    method setPersonalInfo(name, ssn, employment):
        personalInfo = {name, ssn, employment}
    method setLoanTerms(loanType, amount, term):
        this.loanType = loanType
        this.amount = amount
        this.term = term
    method setInterestRate(rate):
        interestRate = rate  // Higher rate for unsecured loans
    method setCollateral(description, value):
        collateral = {description, value}  // Optional for secured personal loans
    method addDocument(docType, reference):
        documents.append({docType, reference})
    method getResult():
        return new LoanApplication(personalInfo, "personal", amount, term,
                                   interestRate, collateral, null, null,
                                   documents)

// ── [CONCRETE BUILDER: MortgageBuilder] ──
class MortgageBuilder implements LoanApplicationBuilder:
    private personalInfo, amount, term, interestRate
    private propertyAppraisal = {}
    private downPayment = 0
    private documents = []

    method setPersonalInfo(name, ssn, employment):
        personalInfo = {name, ssn, employment}
    method setLoanTerms(loanType, amount, term):
        this.amount = amount
        this.term = term  // Typically 360 months (30 years)
    method setInterestRate(rate):
        interestRate = rate  // Lower rate, secured by property
    method setPropertyInfo(address, appraisalValue, downPayment):
        propertyAppraisal = {address, appraisalValue}
        this.downPayment = downPayment
    method setCollateral(description, value):
        // For mortgages, collateral is the property itself
    method addDocument(docType, reference):
        documents.append({docType, reference})
    method getResult():
        ltv = (amount - downPayment) / propertyAppraisal.value * 100
        return new LoanApplication(personalInfo, "mortgage", amount, term,
                                   interestRate, propertyAppraisal,
                                   downPayment, ltv, documents)

// ── [CONCRETE BUILDER: AutoLoanBuilder] ──
class AutoLoanBuilder implements LoanApplicationBuilder:
    private personalInfo, amount, term, interestRate
    private vehicleInfo = {}
    private dealershipInfo = ""
    private documents = []

    method setPersonalInfo(name, ssn, employment):
        personalInfo = {name, ssn, employment}
    method setLoanTerms(loanType, amount, term):
        this.amount = amount
        this.term = term  // Typically 36-72 months
    method setInterestRate(rate):
        interestRate = rate  // Moderate rate, vehicle is collateral
    method setVehicleInfo(vin, make, model, year, value):
        vehicleInfo = {vin, make, model, year, value}
    method setDealership(name):
        dealershipInfo = name
    method addDocument(docType, reference):
        documents.append({docType, reference})
    method getResult():
        return new LoanApplication(personalInfo, "auto", amount, term,
                                   interestRate, vehicleInfo,
                                   dealershipInfo, documents)

// ── [DIRECTOR] ──
class LoanDirector:
    private builder

    method __init__(builder):
        this.builder = builder

    method constructPersonalLoan(name, ssn, employment, amount, term):
        builder.setPersonalInfo(name, ssn, employment)
        builder.setLoanTerms("personal", amount, term)
        builder.setInterestRate(11.99)  // Unsecured personal loan rate
        builder.addDocument("id", "DL-12345")
        builder.addDocument("income", "PAYSTUB-2024-01")
        return builder.getResult()

    method constructMortgage(name, ssn, employment, amount, term,
                             propertyAddress, appraisalValue, downPayment):
        builder.setPersonalInfo(name, ssn, employment)
        builder.setLoanTerms("mortgage", amount, term)
        builder.setInterestRate(6.25)
        builder.setPropertyInfo(propertyAddress, appraisalValue, downPayment)
        builder.addDocument("id", "DL-67890")
        builder.addDocument("income", "TAX-2023")
        builder.addDocument("appraisal", "APP-456")
        return builder.getResult()

    method constructAutoLoan(name, ssn, employment, amount, term,
                             vin, make, model, year, value, dealership):
        builder.setPersonalInfo(name, ssn, employment)
        builder.setLoanTerms("auto", amount, term)
        builder.setInterestRate(7.49)
        builder.setVehicleInfo(vin, make, model, year, value)
        builder.setDealership(dealership)
        builder.addDocument("id", "DL-11111")
        builder.addDocument("registration", "REG-789")
        return builder.getResult()

// ── CLIENT CODE ──
builder = new MortgageBuilder()
director = new LoanDirector(builder)
mortgage = director.constructMortgage(
    "Alice Johnson", "123-45-6789", "Software Engineer",
    350000.00, 360, "123 Main St, Springfield",
    450000.00, 70000.00
)
print(mortgage.getSummary())
// Output: Mortgage[Alice Johnson]: $350000.00 over 360mo
//         @ 6.25% APR — 123 Main St | LTV: 62.22%
```

## How It Works — Step by Step

1. **The client selects a concrete builder.** You choose `PersonalLoanBuilder`, `MortgageBuilder`, or `AutoLoanBuilder` based on the loan type you need to create. Each builder knows which fields are required for its loan category.

2. **The client passes the builder to the director.** `LoanDirector` accepts any `LoanApplicationBuilder` through its constructor. The director does not know which concrete builder it holds — it only knows the abstract builder interface.

3. **The director calls construction steps in sequence.** `constructPersonalLoan()` calls `setPersonalInfo()`, then `setLoanTerms()`, then `setInterestRate()`, then `addDocument()` in a fixed order. The same step sequence could produce different loans depending on which builder received the calls.

4. **Each step configures one aspect of the product.** `setPersonalInfo()` stores the applicant's name, SSN, and employment. `setLoanTerms()` sets the amount and repayment period. `setCollateral()` or `setPropertyInfo()` or `setVehicleInfo()` adds type-specific details. The builder accumulates state with each method call.

5. **The director requests the final product.** After all steps complete, the director calls `getResult()` on the builder. The builder assembles the `LoanApplication` object using the accumulated state and returns it.

6. **The product is immutable and ready to use.** `LoanApplication` provides read-only accessors and `getSummary()`. No further modification is possible, which guarantees consistency. The client receives a fully constructed, validated loan application.

## When to Use

- **Your object requires more than 4 constructor parameters.** Any class with 5 or more fields — especially optional ones — benefits from the Builder pattern. Loan applications with 15+ fields are a textbook case.

- **The same construction process must create different representations.** You process personal loans, mortgages, and auto loans through the same structure of steps but each produces a different configured product.

- **Object construction has invalidd intermediate states.** A partially constructed loan application should not be usable. The Builder pattern delays product creation until `getResult()` is called, ensuring the object is never exposed half-built.

- **You need to vary the internal representation without changing the client code.** Adding a `BusinessLoanBuilder` requires no changes to `LoanDirector` as long as it implements the builder interface.

- **Construction steps produce different types based on input.** A mortgage builder calculates loan-to-value ratio during construction. An auto loan builder sets dealership information. Each builder encapsulates its own construction logic.

## When NOT to Use

- **Your object has 2-3 simple fields with no optional parameters.** A builder adds unnecessary complexity for simple objects. Just use a constructor or a data class.

- **You need mutable, frequently modified objects.** The Builder pattern shines for immutable construction. If you frequently update fields after creation, a mutable object with setters or a simple data transfer object is more appropriate.

- **The number of product variants is small and stable.** If you only ever create two or three fixed configurations, factory methods or named constructors are simpler than a full builder hierarchy.

- **Performance is critical in a hot loop.** The Builder pattern adds object allocations for the builder, the product, and intermediary state. In garbage-collected languages, this overhead matters in tight loops allocating thousands of objects per second.

- **Your team is unfamiliar with the pattern and the object is simple.** Introducing a builder for a 3-field object confuses rather than clarifies. Apply the pattern when the construction complexity genuinely warrants it.

## Real-World Occurrences

1. **Java `StringBuilder` and `StringBuffer`** — The canonical example in the Java standard library. `StringBuilder` lets you construct a `String` by appending fragments step by step. `toString()` serves as the `getResult()` method. This predates the GoF book and remains one of the most widely used Builder implementations.

2. **.NET `System.Text.StringBuilder`** — The .NET equivalent of Java's `StringBuilder`. Used extensively in ASP.NET Core and Entity Framework for building SQL queries, JSON serialization, and string concatenation in performance-sensitive code paths.

3. **Python `sqlalchemy.sql.expression.select()`** — SQLAlchemy's query-building API is a fluent Builder. You chain `.select_from()`, `.where()`, `.join()`, `.group_by()`, and `.order_by()` calls to construct a SQL statement. The final `execute()` call returns the product (a result set).

4. **JavaScript/TypeScript `axios` request configuration** — The Axios HTTP library provides a builder-like fluent API for configuring requests. You set base URL, headers, params, timeout, and interceptors through chained configuration methods. The request is executed when you call `.then()` or `await`.

5. **Rust `std::process::Command`** — Rust's standard library `Command` type is a builder for spawning OS processes. You chain `.arg()`, `.env()`, `.current_dir()`, and `.stdout()` to configure the process, then call `.spawn()` or `.output()` to execute it and obtain the result.

6. **Kotlin `buildString {}` and `buildList {}`** — Kotlin standard library provides builder lambdas that accept a receiver (`StringBuilder`, `MutableList`) with a receiver lambda. Inside the lambda, you configure the object imperatively. The builder function returns the immutable result.

## Comparison with Related Patterns

| Related Pattern | Key Difference | When to Choose Instead |
|---|---|---|
| **Factory Method** | Factory Method creates one product through inheritance in a single method call. Builder creates a complex product through composition in multiple method calls. | Choose Factory Method when construction is a single step and you only vary the product type, not its internal configuration. |
| **Abstract Factory** | Abstract Factory returns a family of related products in one call. Builder constructs a single complex product step by step. | Choose Abstract Factory when you need families of related objects (e.g., different UI themes) and each family has multiple products. |
| **Prototype** | Prototype clones an existing object to create new instances. Builder constructs from scratch using configuration parameters. | Choose Prototype when the cost of constructing from scratch is high and a partially configured template is available to clone. |
| **Composite** | Composite builds tree structures of objects. Builder constructs a single aggregate object (not a tree). | Choose Composite when the product itself is a hierarchical tree (e.g., organizational chart, file system). Use Builder for flat but many-field objects. |
