## Implementation

The violation shows `NotificationSender` as a base class. `EmailSender` and `SmsSender` work, but `PushNotificationSender` requires a device registration token the base doesn't provide — causing it to throw. The correct version adds a separate `DeviceAwareSender` interface for senders that need device context.

```java
// LSP -- Java 17+ implementation
// Domain: notification system
import java.util.*;

// ============================================
// VIOLATION: NotificationSender requires a
// device token, but some senders don't use one
// ============================================

abstract class NotificationSender {
    protected String recipient;

    public NotificationSender(String recipient) {
        this.recipient = recipient;
    }

    public abstract void send(String message);

    public abstract String getDeliveryStatus();
}

class EmailSender extends NotificationSender {
    public EmailSender(String email) {
        super(email);
    }

    @Override
    public void send(String message) {
        System.out.println("Email to " + recipient + ": " + message);
    }

    @Override
    public String getDeliveryStatus() {
        return "Sent via email";
    }
}

class SmsSender extends NotificationSender {
    public SmsSender(String phone) {
        super(phone);
    }

    @Override
    public void send(String message) {
        System.out.println("SMS to " + recipient + ": " + message);
    }

    @Override
    public String getDeliveryStatus() {
        return "Sent via SMS";
    }
}

class PushNotificationSender extends NotificationSender {
    private String deviceToken;

    // VIOLATION: Requires device token, but base
    // class only accepts a general "recipient"
    public PushNotificationSender(String deviceToken) {
        super(deviceToken);
        this.deviceToken = deviceToken;
    }

    @Override
    public void send(String message) {
        // VIOLATION: Need a device token, but the base
        // class contract treats recipient as an email/phone
        if (!deviceToken.startsWith("dev-")) {
            throw new IllegalArgumentException("Invalid device token format");
        }
        System.out.println("Push notification sent to device " + deviceToken);
    }

    @Override
    public String getDeliveryStatus() {
        return "Pushed to device";
    }
}

class NotificationService {
    // VIOLATION: Assumes all senders work the same way
    public void sendAlert(NotificationSender sender, String message) {
        sender.send(message);
        System.out.println("Status: " + sender.getDeliveryStatus());
    }
}

// ============================================
// CORRECT: Use separate interfaces for
// device-aware and simple notification senders
// ============================================

interface MessageSender {
    void send(String message);
    String getDeliveryStatus();
}

interface DeviceAwareSender extends MessageSender {
    void setDeviceToken(String token);
    boolean validateDevice();
}

class EmailSender implements MessageSender {
    private final String email;

    public EmailSender(String email) {
        this.email = email;
    }

    @Override
    public void send(String message) {
        System.out.println("Email to " + email + ": " + message);
    }

    @Override
    public String getDeliveryStatus() {
        return "Email queued";
    }
}

class SmsSender implements MessageSender {
    private final String phone;

    public SmsSender(String phone) {
        this.phone = phone;
    }

    @Override
    public void send(String message) {
        System.out.println("SMS to " + phone + ": " + message);
    }

    @Override
    public String getDeliveryStatus() {
        return "SMS queued";
    }
}

class PushNotificationSender implements DeviceAwareSender {
    private String deviceToken;

    public PushNotificationSender(String deviceToken) {
        this.deviceToken = deviceToken;
    }

    @Override
    public void setDeviceToken(String token) {
        this.deviceToken = token;
    }

    @Override
    public boolean validateDevice() {
        return deviceToken != null && deviceToken.startsWith("dev-");
    }

    @Override
    public void send(String message) {
        if (!validateDevice()) {
            throw new IllegalStateException("Device not registered");
        }
        System.out.println("Push notification to " + deviceToken + ": " + message);
    }

    @Override
    public String getDeliveryStatus() {
        return "Push notification sent";
    }
}

// Functions that work with any message sender
class AlertService {
    public void sendAlert(MessageSender sender, String message) {
        sender.send(message);
        System.out.println("Status: " + sender.getDeliveryStatus());
    }
}

public class Main {
    public static void main(String[] args) {
        System.out.println("=== CORRECT IMPLEMENTATION ===");

        AlertService alertService = new AlertService();

        MessageSender email = new EmailSender("user@example.com");
        alertService.sendAlert(email, "Welcome!");

        MessageSender sms = new SmsSender("+1234567890");
        alertService.sendAlert(sms, "Your code is 1234");

        DeviceAwareSender push = new PushNotificationSender("dev-abc123");
        alertService.sendAlert(push, "New message received");
    }
}
```

### Key Language-Specific Notes

- Java enforces **nominal subtyping** -- a class must explicitly `implements` an interface or `extends` a class. This makes LSP contracts explicit at compile time, catching violations like missing methods immediately.
- Use `sealed` classes and interfaces (Java 17+) to restrict which types can extend a base class. This makes the set of substitutable types finite and known at compile time, reducing the risk of accidental LSP violations.
- The `@Override` annotation helps catch signature mismatches at compile time but does not enforce behavioral contracts (e.g., preconditions, postconditions). Document behavioral contracts in Javadoc and enforce with unit tests.
- Avoid abstract classes with a mix of concrete and abstract methods when the concrete methods assume invariants that not all subclasses can satisfy. Prefer narrow interfaces over broad abstract classes.
