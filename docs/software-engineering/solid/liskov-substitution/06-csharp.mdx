## Implementation

The violation shows a `DocumentProcessor` with `Read()` and `Write()` methods. `ReadOnlyDocument` must implement `Write()` but throws. The correct version separates `IReadable` and `IWritable` interfaces so read-only documents only implement what they need.

```csharp
// LSP -- C# 12 / .NET 8 implementation
// Domain: document reader

// ============================================
// VIOLATION: DocumentProcessor couples read
// and write, but read-only documents cannot
// support writing
// ============================================

abstract class DocumentProcessor
{
    protected string Content { get; set; } = "";

    public abstract string Read();
    public abstract void Write(string content);
}

class EditableDocument : DocumentProcessor
{
    public override string Read()
    {
        return Content;
    }

    public override void Write(string content)
    {
        Content = content;
        Console.WriteLine("Document written");
    }
}

class ReadOnlyDocument : DocumentProcessor
{
    public override string Read()
    {
        return "Read-only content";
    }

    // VIOLATION: ReadOnlyDocument cannot support writing,
    // but the base class contract promises it can
    public override void Write(string content)
    {
        throw new NotSupportedException("This document is read-only");
    }
}

class DocumentManager
{
    // VIOLATION: Assumes all documents are writable
    public void SaveDocument(DocumentProcessor doc, string content)
    {
        doc.Write(content);
        Console.WriteLine("Document saved");
    }
}

// ============================================
// CORRECT: Separate IReadable and IWritable
// ============================================

interface IReadable
{
    string Read();
}

interface IWritable
{
    void Write(string content);
}

// A document that supports both reading and writing
class EditableDocument : IReadable, IWritable
{
    public string Content { get; private set; } = "";

    public string Read()
    {
        return Content;
    }

    public void Write(string content)
    {
        Content = content;
        Console.WriteLine("Document written to memory");
    }
}

// A read-only document -- no Write method at all
class ReadOnlyDocument : IReadable
{
    private readonly string _content;

    public ReadOnlyDocument(string content)
    {
        _content = content;
    }

    public string Read()
    {
        return _content;
    }
}

// A write-only log
class WriteOnlyLog : IWritable
{
    public void Write(string content)
    {
        Console.WriteLine($"Log entry: {content}");
    }
}

class DocumentService
{
    public string ReadDocument(IReadable readable)
    {
        return readable.Read();
    }

    public void WriteDocument(IWritable writable, string content)
    {
        writable.Write(content);
    }
}

// Main driver
class Program
{
    static void Main()
    {
        Console.WriteLine("=== CORRECT IMPLEMENTATION ===");

        var service = new DocumentService();

        // Editable document works for both read and write
        var editable = new EditableDocument();
        service.WriteDocument(editable, "Hello, world!");
        string content = service.ReadDocument(editable);
        Console.WriteLine($"Read: {content}");

        // Read-only document only works for read
        var readOnly = new ReadOnlyDocument("Fixed content");
        string fixedContent = service.ReadDocument(readOnly);
        Console.WriteLine($"Read: {fixedContent}");

        // Write-only log
        var log = new WriteOnlyLog();
        service.WriteDocument(log, "Application started");

        Console.WriteLine("\nAll operations completed without LSP violations.");
    }
}
```

### Key Language-Specific Notes

- C# interfaces use **explicit interface implementation** to avoid naming conflicts when a class implements multiple interfaces with the same method signature. This is useful when different interfaces define contracts that a single class should fulfill.
- C# 12 introduces **primary constructors** for classes and structs, reducing boilerplate in small types that implement interfaces. Combined with `record` types, you can define immutable data carriers that trivially satisfy interface contracts.
- LINQ operates over `IEnumerable<T>`, which is a classic example of LSP in .NET. Any collection, array, or query result that implements `IEnumerable<T>` is substitutable in any LINQ method. The `yield return` pattern also produces `IEnumerable<T>` without creating a collection class.
- The `NotSupportedException` in a base class method is a strong signal of an LSP violation during code review. If you see it in an override, consider whether the interface hierarchy needs refactoring.
