## Implementation

The violation shows a `Storage` interface with `save()` and `load()`. A `ReadOnlyStorage` must implement `save()` but throws. The correct version separates `Readable` and `Writable` interfaces so read-only types can omit write methods.

```typescript
// LSP -- TypeScript implementation (strict mode)
// Domain: data storage

// ============================================
// VIOLATION: Storage interface couples read
// and write, but some storage is read-only
// ============================================

interface Storage {
  save(data: string): void;
  load(): string;
}

class FileStorage implements Storage {
  private data = "";

  save(data: string): void {
    this.data = data;
    console.log(`Saved to file: ${data}`);
  }

  load(): string {
    return this.data;
  }
}

class ReadOnlyStorage implements Storage {
  private data = "read-only content";

  // VIOLATION: ReadOnlyStorage cannot save,
  // but the Storage interface promises it can
  save(_data: string): void {
    throw new Error("This storage is read-only");
  }

  load(): string {
    return this.data;
  }
}

function writeData(storage: Storage, data: string): void {
  storage.save(data);
}

const fileStore = new FileStorage();
writeData(fileStore, "hello"); // OK

const readOnly = new ReadOnlyStorage();
// writeData(readOnly, "test"); // Throws!

// ============================================
// CORRECT: Separate Readable and Writable
// ============================================

interface Readable {
  load(): string;
}

interface Writable {
  save(data: string): void;
}

// A full read-write storage
class ReadWriteStorage implements Readable, Writable {
  private data = "";

  save(data: string): void {
    this.data = data;
    console.log(`Saved: ${data}`);
  }

  load(): string {
    return this.data;
  }
}

// Read-only storage -- no save() method at all
class ReadOnlyFileStorage implements Readable {
  constructor(private readonly filePath: string) {}

  load(): string {
    return `Content of ${this.filePath}`;
  }
}

// Write-only storage -- no load() method
class WriteOnlyCloudStorage implements Writable {
  save(data: string): void {
    console.log(`Uploaded ${data.length} bytes to cloud`);
  }
}

function readFrom(storage: Readable): void {
  console.log(`Read: ${storage.load()}`);
}

function writeTo(storage: Writable, data: string): void {
  storage.save(data);
}

// Main driver
console.log("=== CORRECT IMPLEMENTATION ===");

const rwStorage = new ReadWriteStorage();
writeTo(rwStorage, "important document");
readFrom(rwStorage);

const readOnlyFile = new ReadOnlyFileStorage("/etc/config.json");
readFrom(readOnlyFile);

const cloudStorage = new WriteOnlyCloudStorage();
writeTo(cloudStorage, "log entry");
```

```bash
Saved: important document
Read: important document
Read: Content of /etc/config.json
Uploaded 8 bytes to cloud
```

### Key Language-Specific Notes

- TypeScript's **structural type system** (duck typing at the type level) means any object with the correct shape satisfies an interface, even without an explicit `implements` clause. This makes LSP violations type-level errors when interfaces are properly separated.
- Use **interface segregation** combined with LSP: small, focused interfaces like `Readable` and `Writable` are easier to substitute than large monolithic interfaces. A `ReadOnlyFileStorage` that implements `Readable` is trivially substitutable for any `Readable`.
- The `never` type can signal unreachable code paths in strict LSP designs. A well-structured discriminated union with `never` exhaustiveness checking enforces substitutability at compile time.
- TypeScript `abstract` classes provide runtime enforcement of method contracts, though interfaces are preferred for LSP-correct designs because they carry no runtime footprint.
