Software entities should be open for extension but closed for modification. This principle, articulated by Bertrand Meyer in his 1988 book _Object-Oriented Software Construction_, is one of the five SOLID principles of object-oriented design. It prescribes that you should be able to add new functionality to a system without changing the existing, tested, and deployed code that implements that system.

A module conforming to the Open/Closed Principle meets two criteria:

1. **Open for extension** -- its behavior can be extended, typically by subclassing, implementing an interface, or providing a plugin/hook.
2. **Closed for modification** -- its source code is stable and should not be edited when new features are added.

The tension between "open" and "closed" is resolved through abstraction. The core logic depends on abstractions (interfaces or base classes), and concrete implementations supply the specifics. The core stays closed; new implementations extend it.

---

## Why It Exists

Modifying existing, tested code introduces risk. Every edit to a stable module carries the possibility of regressions -- breaking something that previously worked. Before the Open/Closed Principle was widely adopted, adding a new feature typically meant editing a switch statement or an if-else chain deep inside a core class. That single function or class would accumulate branches for every variant the system supported, growing longer and more fragile with each release.

Consider a typical evolution:

**Version 1:** The system supports credit card payments only. A single `processPayment` method handles the one case. Clean, simple, no branching needed.

**Version 2:** PayPal support is requested. The developer adds an `if` branch to `processPayment`. The method still works, but it now has two code paths.

**Version 3:** Bitcoin support. Another branch. The `processPayment` function is now three pages long.

**Version 4:** Apple Pay. The developer hesitates, because the last change introduced a subtle bug that took three days to find.

This is the violation pattern: each new feature requires modifying and retesting the same central piece of code. The OCP exists to break this cycle. By depending on abstractions rather than concrete types, the core logic stays stable. New features arrive as new files, not as edits to existing ones.

The benefit is not just about code quality -- it is about deployment safety. When you add a new class that implements an existing interface, you can deploy it without retesting every existing implementation. The interface contract guarantees that the core logic is unchanged.

## The Violation

The following example shows a typical OCP violation in a payment processing system. The `PaymentProcessor` class uses a switch statement to dispatch to different payment logic. Adding a new payment type requires editing this switch.

```javascript
// VIOLATION: Adding a new payment type means editing this switch statement,
// which is shared by every payment flow. A mistake here breaks all payments.

class PaymentProcessor {
  processPayment(amount, type) {
    // VIOLATION: switch on type enum -- every new type requires a new case
    // and re-testing of the entire method.
    switch (type) {
      case "credit":
        // 15 lines of credit card validation and processing
        console.log(`Processing credit card payment of $${amount}`);
        this.validateCardDetails();
        this.chargeCreditCard(amount);
        break;
      case "paypal":
        // 12 lines of PayPal API interaction
        console.log(`Processing PayPal payment of $${amount}`);
        this.redirectToPayPal(amount);
        this.confirmPayPalPayment();
        break;
      case "bitcoin":
        // 10 lines of blockchain interaction
        console.log(`Processing Bitcoin payment of $${amount}`);
        this.validateBitcoinAddress();
        this.broadcastBitcoinTransaction(amount);
        break;
      default:
        throw new Error(`Unknown payment type: ${type}`);
    }
    // After the switch, shared logic like receipt generation
    this.generateReceipt(amount, type);
    this.sendConfirmation(type);
  }

  validateCardDetails() { /* ... */ }
  chargeCreditCard(amount) { /* ... */ }
  redirectToPayPal(amount) { /* ... */ }
  confirmPayPalPayment() { /* ... */ }
  validateBitcoinAddress() { /* ... */ }
  broadcastBitcoinTransaction(amount) { /* ... */ }
  generateReceipt(amount, type) { /* ... */ }
  sendConfirmation(type) { /* ... */ }
}
```

The problem is structural: the `PaymentProcessor` class must know about every payment type that exists. When Apple Pay or Google Pay support is needed, a developer must:

1. Open `PaymentProcessor` and find the switch statement.
2. Add a new `case` block with the new payment logic.
3. Retest every existing case to ensure nothing was broken.
4. Deploy the modified `PaymentProcessor` -- a core class -- into production.

Step 3 is the most dangerous. In a real system with dozens of payment types, a single misplaced brace or missing `break` can cascade through every case below it.

---

## The Correct Implementation

The fix is to apply the **Strategy pattern**. Define an interface (or abstract base class) for payment processing, then implement that interface separately for each payment type. The core `PaymentProcessor` only depends on the interface, not on any concrete implementation.

```javascript
// CORRECT: Payment method is an interface. The processor depends on the
// abstraction, not on concrete types. New payment methods are added as
// new implementations without touching existing code.

// Interface contract -- each payment method must implement this
class PaymentMethod {
  process(amount) {
    throw new Error("Subclasses must implement process()");
  }
  getType() {
    throw new Error("Subclasses must implement getType()");
  }
}

class CreditCardPayment extends PaymentMethod {
  getType() { return "credit"; }

  process(amount) {
    console.log(`Validating credit card...`);
    console.log(`Charging $${amount} to credit card`);
    return { success: true, transactionId: "CC-" + Date.now() };
  }
}

class PayPalPayment extends PaymentMethod {
  getType() { return "paypal"; }

  process(amount) {
    console.log(`Redirecting to PayPal...`);
    console.log(`Completing PayPal payment of $${amount}`);
    return { success: true, transactionId: "PP-" + Date.now() };
  }
}

class BitcoinPayment extends PaymentMethod {
  getType() { return "bitcoin"; }

  process(amount) {
    console.log(`Validating Bitcoin address...`);
    console.log(`Broadcasting Bitcoin transaction of $${amount}`);
    return { success: true, transactionId: "BTC-" + Date.now() };
  }
}

// The factory maps types to strategy instances
class PaymentMethodFactory {
  constructor() {
    this.methods = new Map();
  }

  register(method) {
    this.methods.set(method.getType(), method);
  }

  get(type) {
    const method = this.methods.get(type);
    if (!method) {
      throw new Error(`Unknown payment type: ${type}`);
    }
    return method;
  }
}

// Core processor -- closed for modification
class PaymentProcessor {
  constructor(factory) {
    this.factory = factory;
  }

  processPayment(amount, type) {
    const method = this.factory.get(type);
    const result = method.process(amount);
    this.generateReceipt(amount, type);
    this.sendConfirmation(type);
    return result;
  }

  generateReceipt(amount, type) { /* ... */ }
  sendConfirmation(type) { /* ... */ }
}

// Setup and usage
const factory = new PaymentMethodFactory();
factory.register(new CreditCardPayment());
factory.register(new PayPalPayment());
factory.register(new BitcoinPayment());

const processor = new PaymentProcessor(factory);
processor.processPayment(100, "credit");
processor.processPayment(50, "paypal");
processor.processPayment(0.5, "bitcoin");

// Adding Apple Pay: create a new class, register it. No edits to existing code.
class ApplePayPayment extends PaymentMethod {
  getType() { return "apple-pay"; }
  process(amount) {
    console.log(`Processing Apple Pay payment of $${amount}`);
    return { success: true, transactionId: "AP-" + Date.now() };
  }
}
factory.register(new ApplePayPayment());
processor.processPayment(75, "apple-pay");
```

Now the `PaymentProcessor` class is **closed**: it never needs to change when a new payment type is added. The new type arrives as a brand new file -- no edits to tested code.

## How to Spot Violations in Code Review

When reviewing a pull request, look for these tell-tale signs that a codebase is violating the Open/Closed Principle:

### Switch Statements and If-Else Chains on Type Enums

A switch statement that branches on a string or numeric enum is the classic OCP violation. If a new value in the enum requires a new case in the switch, the code is not closed. The smell is strongest when the same enum appears in multiple switch statements across the codebase -- adding a new value means finding and editing every one of them.

### Long Conditional Blocks

A method that contains many `if-else` blocks, each handling a different variant of the same concept, likely violates OCP. Look for blocks that share the same structure but differ only in the concrete logic they execute. These are candidates for extraction into strategy implementations.

### `instanceof` Checks

Type-checking with `instanceof` and then branching based on the result is a sign that polymorphic dispatch is being bypassed. The language's virtual method dispatch already handles this -- if you find yourself writing `if (obj instanceof TypeA) { ... } else if (obj instanceof TypeB) { ... }`, you are fighting the type system instead of using it.

### Methods Named `handle*` with Type Dispatch

A method called `handleNotification`, `handleEvent`, `handleRequest`, or similar that internally dispatches on a type string or enum is likely violating OCP. The `handle` prefix often indicates a single entry point that grew organically as types were added.

### Modification Comments

Pay attention to comments like "add new type here" or "insert new case here in alphabetical order." These are apologies for a design that should have been open for extension from the start.

## Real-World Occurrences

The Open/Closed Principle appears throughout well-designed frameworks and libraries. Here are notable examples:

### Java Collections.sort() with Comparator

The `Collections.sort()` method (and `List.sort()`) is closed for the sorting algorithm but open for the ordering strategy. You provide a `Comparator` to define how elements are ordered:

```java
list.sort(Comparator.naturalOrder());
list.sort((a, b) -> a.getName().compareTo(b.getName()));
list.sort(Comparator.comparing(Employee::getSalary).reversed());
```

The sort implementation never changes. The comparison strategy is open for extension via the `Comparator` interface.

### Spring's HandlerInterceptor

Spring MVC allows you to add custom processing before and after request handling by implementing the `HandlerInterceptor` interface. The core MVC framework is closed -- it does not change when you add a new interceptor. You extend its behavior by registering a new implementation.

### Express Middleware Pipeline

Express.js middleware follows OCP naturally. The request pipeline is closed: it processes requests and responses through a chain. Each middleware is an extension point:

```javascript
app.use(logger);           // logging middleware
app.use(authenticate);     // auth check middleware
app.use(rateLimiter);      // rate limiting middleware
app.use(errorHandler);     // error handling middleware
```

Adding a new middleware does not require modifying Express core or any existing middleware.

### Rust's Trait Dispatch

Rust traits are a language-level embodiment of OCP. A function that accepts `impl Trait` or `Box<dyn Trait>` is closed for modification -- it calls trait methods without knowing the concrete type. Any struct that implements the trait extends the system.

```rust
pub fn process(items: Vec<Box<dyn Processor>>) {
    for item in items {
        item.process();  // closed: never changes
    }
}
```

### Plugin Systems

Nearly every plugin architecture implements OCP. The host application defines a plugin interface. Third-party developers implement that interface to extend the application. The host never changes when a new plugin is installed. Examples include VS Code extensions, Jenkins plugins, and WordPress hooks.

## When Strict Adherence Hurts

While OCP is valuable, applying it strictly in every situation adds complexity and can be counterproductive. Here is when to be pragmatic:

### Rarely Used Branches

If a code path is executed once in a blue moon -- for example, a migration script that runs during a one-time database upgrade -- creating a full strategy interface with abstract methods and concrete implementations is over-engineering. A simple conditional is fine.

### Rapid Prototyping

During early development, the shape of the abstraction is unknowable. Trying to design the perfect interface hierarchy before you understand the domain leads to premature abstraction. It is better to write a simple switch statement, flesh out the requirements, and then refactor to OCP once the pattern is clear.

### When Extension Points Are Unknowable

Sometimes you cannot predict what will need to be extended. Building extension points for every possible future change leads to a codebase full of unused interfaces and abstract classes. This is the **Speculative Generality** anti-pattern. A good heuristic: do not abstract until you have at least two concrete implementations, or until the third time you need to add a new variant.

### Performance-Critical Paths

Virtual method dispatch (interface calls) has a small runtime cost compared to direct function calls or inlined code. In hot paths -- rendering loops, real-time audio processing, game engine physics -- the overhead of strategy pattern dispatch may be unacceptable. In these cases, controlled conditional branches are often the pragmatic choice.

### The Rule of Three

A common guideline in the Extreme Programming community is the **Rule of Three**: the first time you implement something, just build it. The second time something similar appears, consider whether a pattern is emerging. The third time, refactor to use the abstraction. This prevents premature architecture while ensuring that genuine extension points are captured.

### Summary

The Open/Closed Principle is a design tool, not a dogma. Use it when you control the extension points and expect multiple implementations. Skip it when the domain is unstable, the variants are few, or the cost of abstraction outweighs the benefit of flexibility. Good design means knowing both when to apply a principle and when to set it aside.
