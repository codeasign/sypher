## Implementation

A JavaScript (ES2022+) implementation of ISP using a worker management system. The violation shows a `Worker` class with methods `work()`, `eat()`, and `sleep()` — a `Robot` class must implement all three even though it doesn't eat or sleep. The correct version segregates the capabilities.

```javascript
// ISP — JavaScript (ES2022+) implementation
// Domain: Worker management

// ============================================================
// VIOLATION: Worker class has eat() and sleep() methods that
// Robot must implement but cannot meaningfully use
// ============================================================
//
// class Worker {
//   work() { throw new Error("Must implement"); }
//   eat() { throw new Error("Must implement"); }
//   sleep() { throw new Error("Must implement"); }
// }
//
// class Human extends Worker {
//   work() { return "Human working"; }
//   eat() { return "Human eating"; }
//   sleep() { return "Human sleeping"; }
// }
//
// class Robot extends Worker {
//   work() { return "Robot working"; }
//   eat() { throw new Error("Robots don't eat!"); }
//   sleep() { throw new Error("Robots don't sleep!"); }
// }

// ============================================================
// CORRECT IMPLEMENTATION — Segregated capabilities
// ============================================================

// Each "interface" is a class that serves as a capability contract

class Workable {
  work() {
    throw new Error("Must implement work()");
  }
}

class Eatable {
  eat() {
    throw new Error("Must implement eat()");
  }
}

class Sleepable {
  sleep() {
    throw new Error("Must implement sleep()");
  }
}

class Human {
  constructor(name) {
    this.name = name;
  }

  work() {
    return `${this.name} is working`;
  }

  eat() {
    return `${this.name} is eating`;
  }

  sleep() {
    return `${this.name} is sleeping`;
  }
}

class Robot {
  constructor(name) {
    this.name = name;
  }

  work() {
    return `${this.name} is welding`;
  }

  charge() {
    return `${this.name} is recharging`;
  }
}

// Functions constrained to specific interfaces
function startShift(workers) {
  return workers.map(w => w.work());
}

function lunchBreak(eaters) {
  return eaters.map(e => e.eat());
}

function endOfDay(sleepers) {
  return sleepers.map(s => s.sleep());
}

// --- Main driver ---
function main() {
  const humans = [
    new Human("Alice"),
    new Human("Bob")
  ];

  const robots = [
    new Robot("R2-D2"),
    new Robot("C-3PO")
  ];

  console.log("=== Shift Start ===");
  startShift([...humans, ...robots]).forEach(msg => console.log(msg));

  console.log("\n=== Lunch Break ===");
  lunchBreak(humans).forEach(msg => console.log(msg));
  // robots don't participate — they are not Eatable

  console.log("\n=== End of Day ===");
  endOfDay(humans).forEach(msg => console.log(msg));
  // robots don't participate — they are not Sleepable

  console.log("\n=== Robot-specific ===");
  robots.forEach(r => console.log(r.charge()));
}

main();
```

Expected output:

```bash

Alice is working
Bob is working
R2-D2 is welding
C-3PO is welding

=== Lunch Break ===
Alice is eating
Bob is eating

=== End of Day ===
Alice is sleeping
Bob is sleeping

=== Robot-specific ===
R2-D2 is recharging
C-3PO is recharging
```

## Key Language-Specific Notes

- **Duck typing as ISP enforcement** — JavaScript doesn't have interfaces, but functions like `startShift(workers)` are implicitly constrained: any object with a `work()` method can participate. This is ISP via convention.
- **No `instanceof` checks needed** — The correct version never checks whether something is a Human or Robot; it just calls the method the function needs.
- **ES6 class syntax** provides a clean way to define capability contracts, even without true interface types.
- **Rest/spread operators** (`[...humans, ...robots]`) make it easy to compose workers of different types for operations they all support.
