The **Interface Segregation Principle (ISP)** states that "clients should not be forced to depend upon interfaces they do not use." Coined by Robert C. Martin, this is the fourth SOLID principle. It warns against "fat interfaces" — large interfaces that include methods irrelevant to some implementors. When a class implements an interface with methods it does not need, it must provide empty or throwing implementations for those methods, coupling itself to behaviors it does not use.

## Why It Exists

Before ISP was formalized, the common practice was to group related methods into a single interface so that implementors could pick and choose what they needed. This was especially problematic in the early days of COM (Component Object Model) and Java AWT, where interfaces grew to include dozens of methods to accommodate every potential client. An interface like `Container` might include `add()`, `remove()`, `getComponent()`, `paint()`, `repaint()`, `validate()`, `doLayout()`, and more — forcing every container implementation to provide stubs for behaviors it could not support.

The problem is that changing a fat interface affects every implementor, even those that don't use the changed method. Adding a new method to a shared interface forces every class that implements it to change — even classes where the new method makes no sense. This creates a coupling between unrelated clients and makes evolution painful.

## The Violation

Here is a realistic ISP violation. A `MultiFunctionPrinter` interface defines a set of office machine capabilities, and a `SimplePrinter` is forced to implement methods it cannot support:

```python
# VIOLATION: SimplePrinter depends on methods it does not use

from abc import ABC, abstractmethod

class MultiFunctionPrinter(ABC):
    """A fat interface — too many unrelated capabilities."""
    @abstractmethod
    def print(self, document: str) -> str:
        ...

    @abstractmethod
    def scan(self, document: str) -> str:
        ...

    @abstractmethod
    def fax(self, document: str) -> str:
        ...

    @abstractmethod
    def staple(self, pages: list[str]) -> str:
        ...

class SimplePrinter(MultiFunctionPrinter):
    """Must implement all methods, even the ones it cannot do."""

    def print(self, document: str) -> str:
        return f"Printing: {document}"

    def scan(self, document: str) -> str:
        raise NotImplementedError("SimplePrinter cannot scan")

    def fax(self, document: str) -> str:
        raise NotImplementedError("SimplePrinter cannot fax")

    def staple(self, pages: list[str]) -> str:
        raise NotImplementedError("SimplePrinter cannot staple")

class AllInOnePrinter(MultiFunctionPrinter):
    """Can legitimately implement all methods."""

    def print(self, document: str) -> str:
        return f"Printing: {document}"

    def scan(self, document: str) -> str:
        return f"Scanning: {document}"

    def fax(self, document: str) -> str:
        return f"Faxing: {document}"

    def staple(self, pages: list[str]) -> str:
        return f"Stapling {len(pages)} pages"
```

What breaks:

| Problem | Why It Happens |
|---|---|
| `SimplePrinter` throws on `scan()` | Any code calling `scan()` on a `MultiFunctionPrinter` fails when given a `SimplePrinter` |
| Cannot add new features | Adding `photocopy()` to the interface forces both classes to change |
| Testing is harder | Every test of `SimplePrinter` must account for methods it cannot use |
| Versioning is painful | A new interface version means all implementors must update |
| **Relationship semantics** | The interface name pretends all implementors are "multi-function" when some are not |

## The Correct Implementation

Split the fat interface into several smaller, client-specific interfaces:

```python
from abc import ABC, abstractmethod

class Printer(ABC):
    """Responsibility: printing only."""
    @abstractmethod
    def print(self, document: str) -> str:
        ...

class Scanner(ABC):
    """Responsibility: scanning only."""
    @abstractmethod
    def scan(self, document: str) -> str:
        ...

class FaxMachine(ABC):
    """Responsibility: faxing only."""
    @abstractmethod
    def fax(self, document: str) -> str:
        ...

class Stapler(ABC):
    """Responsibility: stapling only."""
    @abstractmethod
    def staple(self, pages: list[str]) -> str:
        ...

class SimplePrinter(Printer):
    """Only implements what it needs."""
    def print(self, document: str) -> str:
        return f"Printing: {document}"

class AllInOnePrinter(Printer, Scanner, FaxMachine, Stapler):
    """Implements all interfaces it supports."""
    def print(self, document: str) -> str:
        return f"Printing: {document}"

    def scan(self, document: str) -> str:
        return f"Scanning: {document}"

    def fax(self, document: str) -> str:
        return f"Faxing: {document}"

    def staple(self, pages: list[str]) -> str:
        return f"Stapling {len(pages)} pages"
```

Now `SimplePrinter` implements only `Printer` — no empty methods, no throwing stubs. `AllInOnePrinter` implements all four interfaces via multiple inheritance. Code that needs scanning works with `Scanner` — it cannot receive a `SimplePrinter` by accident. Each interface is a true contract.

## How to Spot Violations in Code Review

- **Interfaces with "I" prefix and five or more methods** — not always a violation, but worth examining whether all methods are genuinely needed by all implementors.
- **Classes that implement an interface with empty method bodies** — the implementor is confessing the interface is too broad.
- **`throw new UnsupportedOperationException()` in an implementation** — Java's standard library itself has this in some places (e.g., `java.util.Arrays.asList` returns a list where `add()` throws), but it is an anti-pattern.
- **Interfaces named `*Manager`, `*Service`, `*Handler`, `*Util`** — these names often signal the interface is collecting methods by destination rather than by client need.
- **Breaking interface changes** — if adding a method to an interface breaks multiple implementors, the interface was likely too fat.
- **`instanceof` checks on interface implementors** — if callers check concrete types before calling methods, the interface is providing more than clients need.

## Real-World Occurrences

| Library | ISP Application |
|---|---|
| **Java Collections** | `List`, `Set`, `Queue`, and `Deque` are separate interfaces — not one giant `Collection` that does everything. If a class only needs ordered access, it implements `List`, not a monolithic interface. |
| **Java I/O** | `InputStream`, `OutputStream`, `Reader`, `Writer` each cover exactly one direction of one data type. They are not merged into a single `Stream` interface. |
| **React** | Components have `props` interfaces that are specific to each component. There is no giant `ComponentProps` interface. Hooks each have small, specific interfaces (`useState`, `useEffect`, `useContext`). |
| **Rust Traits** | Rust's trait system naturally encourages small, focused traits. `Iterator`, `IntoIterator`, `FromIterator` are separate. `Read`, `Write`, `Seek` are separate. A type cannot be forced to implement methods it doesn't need. |
| **Python `collections.abc`** | Separate ABCs for `Container`, `Iterable`, `Sized`, `Callable`, `Hashable` — each covers one capability. A class can support `Iterable` without supporting `Sized`. |
| **C# LINQ** | `IEnumerable<T>`, `IQueryable<T>`, `IObservable<T>` — each interface covers one query pattern. A database driver implements `IQueryable` for SQL translation; a simple list only needs `IEnumerable`. |

## When Strict Adherence Hurts

**Very small interfaces.** A one-method interface is clean for ISP but can lead to a proliferation of micro-interfaces that make the type system hard to navigate. If every method becomes its own interface, consumers must understand and compose dozens of interfaces instead of one.

**Framework stability.** If an interface is published externally and never changes (a rare but real scenario), the cost of splitting it versus leaving it fat may not justify the change. For example, `java.lang.Iterable` has exactly one method (`iterator()`) — this is the ISP ideal. But once an interface reaches broad adoption, splitting it is a breaking change.

**Internal code with one implementation.** If an interface has exactly one implementor and is never exposed outside a small module, applying ISP is premature. The interface can be split later if a second implementor appears with different needs.

**Performance-critical code.** Interface dispatch has a small cost, and interface calls through many micro-interfaces can impact JIT compilation and inlining. In hot paths, a slightly larger interface that groups related methods can outperform many tiny interfaces.
