## Tier 1 -- Beginner

### Exercise 1: Rename a tool and verify the schema changes

**Task:** In your `weather_server.py`, rename the `get_forecast` tool to `fetch_city_forecast` and change its description to "Fetch the multi-day weather forecast for any city." Restart the server and use a test script to confirm the tool appears under the new name with the updated description.

**Hint:** Only the decorator arguments need to change. The function body stays the same.

#### Solution

Change the decorator:

```python
@server.tool(name="fetch_city_forecast", description="Fetch the multi-day weather forecast for any city")
async def get_forecast(
    city: str = "London",
    days: int = 7,
    units: str = "metric",
) -> str:
    # Body unchanged
```

Create a simple test script `test_tool_list.py`:

```python
import asyncio
from mcp.client.stdio import stdio_client
from mcp import ClientSession

async def main():
    async with stdio_client(["python", "weather_server.py"]) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()
            tools = await session.list_tools()
            for tool in tools:
                print(f"Tool: {tool.name} -- {tool.description}")

asyncio.run(main())
```

Run it:

```bash
python test_tool_list.py
```

```text
Tool: fetch_city_forecast -- Fetch the multi-day weather forecast for any city
Tool: submit_weather_report -- Submit a weather observation
```

The old name `get_forecast` no longer appears. The LLM will see only `fetch_city_forecast` in the tool list. This demonstrates that the decorator's `name` parameter is the authoritative identifier -- the Python function name is irrelevant to the MCP protocol. You can name your Python functions however you like, as long as the `name` parameter in the decorator is descriptive.

### Exercise 2: Add a description to a tool parameter

**Task:** The `submit_weather_report` tool has a `conditions` parameter but no description that explains what values are valid. Add a description using Python's `Annotated` type hint so the LLM knows it should use values like "Sunny", "Rainy", "Cloudy", "Windy", or "Partly Cloudy."

**Hint:** Use `from typing import Annotated` and annotate the parameter with `Annotated[str, ..., "description"]`. The `...` makes it required.

#### Solution

```python
from typing import Annotated

@server.tool(name="submit_weather_report", description="Submit a weather observation")
async def submit_weather_report(
    city: str,
    temperature: float,
    conditions: Annotated[str, ..., "Weather condition: Sunny, Rainy, Cloudy, Windy, or Partly Cloudy"],
    humidity: float = 50.0,
    wind_speed: float = 10.0,
) -> str:
    # Body unchanged
```

Run your test script again to inspect the schema:

```python
# Add to test_tool_list.py
for tool in tools:
    if tool.name == "submit_weather_report":
        import json
        print(json.dumps(tool.inputSchema, indent=2))
```

You will see the `conditions` parameter now includes the description:

```json
{
  "properties": {
    "conditions": {
      "type": "string",
      "description": "Weather condition: Sunny, Rainy, Cloudy, Windy, or Partly Cloudy"
    }
  }
}
```

The LLM now knows exactly what values to use for `conditions`. Without this description, the model would guess -- and it might guess "Overcast" instead of "Cloudy," or "Precipitating" instead of "Rainy." Parameter descriptions are the primary way you shape the LLM's input choices.

### Exercise 5: Create a country-level forecast template resource

**Task:** Add a second template resource `weather://country/{country_code}/forecast` that returns a summary forecast for an entire country. Use a two-letter country code as the parameter (e.g., `weather://country/US/forecast`). The response should include the capital city forecast and a general country-wide summary. Verify it works by reading `weather://country/JP/forecast`.

**Hint:** The URI template `weather://country/{country_code}/forecast` has a static segment `country` and `forecast` around the parameter. Make sure the static segments are unique enough to avoid matching conflicts with other templates.

#### Solution

```python
@server.resource(
    uri="weather://country/{country_code}/forecast",
    name="Country Forecast",
    mime_type="application/json",
)
async def get_country_forecast(country_code: str) -> str:
    countries = {
        "US": {"name": "United States", "capital": "Washington DC"},
        "UK": {"name": "United Kingdom", "capital": "London"},
        "JP": {"name": "Japan", "capital": "Tokyo"},
        "AU": {"name": "Australia", "capital": "Canberra"},
        "IN": {"name": "India", "capital": "New Delhi"},
    }
    country = countries.get(country_code.upper())
    if not country:
        return json.dumps({"error": f"Unknown country code: {country_code}"})

    temp = random.randint(10, 35)
    conditions = random.choice(["Sunny", "Cloudy", "Rainy", "Windy", "Partly Cloudy"])
    return json.dumps({
        "country": country["name"],
        "country_code": country_code.upper(),
        "capital_forecast": {
            "city": country["capital"],
            "temperature": temp,
            "unit": "C",
            "conditions": conditions,
        },
        "summary": f"Generally {conditions.lower()} conditions across {country['name']}.",
    })
```

Test it:

### Exercise 6: Add a tool that returns region metadata

**Task:** Create a new tool called `get_region_info` that takes a region name as a required parameter and returns the timezone, currency, and primary languages for that region. Use a hardcoded lookup table inside the handler. Define at least three regions: "uk", "us", and "eu". Add descriptions for both the tool and its parameter.

**Hint:** Use `Annotated` with a description for the region parameter. Structure the return value as JSON with clear keys.

#### Solution

```python
@server.tool(name="get_region_info", description="Get timezone, currency, and languages for a region")
async def get_region_info(
    region: Annotated[str, ..., "Region identifier: uk, us, eu, asia, or au"],
) -> str:
    regions = {
        "uk": {"timezone": "GMT/BST", "currency": "GBP", "languages": ["English"]},
        "us": {"timezone": "Multiple (ET/CT/MT/PT)", "currency": "USD", "languages": ["English", "Spanish"]},
        "eu": {"timezone": "CET/CEST", "currency": "EUR", "languages": ["Multiple"]},
        "asia": {"timezone": "Multiple (JST/CST/IST)", "currency": "Multiple", "languages": ["Multiple"]},
        "au": {"timezone": "AEST/AEDT", "currency": "AUD", "languages": ["English"]},
    }
    info = regions.get(region.lower())
    if not info:
        return json.dumps({"error": f"Unknown region: {region}. Valid: uk, us, eu, asia, au"})
    return json.dumps({"region": region, **info})
```

Restart and test by calling the tool from a test script or the MCP Inspector:

```python
async def main():
    async with stdio_client(["python", "weather_server.py"]) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()
            result = await session.call_tool("get_region_info", {"region": "uk"})
            print(result)

asyncio.run(main())
```

```text
text="{\"region\": \"uk\", \"timezone\": \"GMT/BST\", \"currency\": \"GBP\", \"languages\": [\"English\"]}"
```

This tool demonstrates a read-only operation that is better suited as a resource but intentionally implemented as a tool to highlight the distinction. The `region` parameter is a simple identifier -- this could just as easily be `weather://regions/{region}` as a resource. When you find yourself writing tools that only return data with no side effects, ask whether a resource would be more appropriate.

### Exercise 7: Add a new prompt template for severe weather warnings

**Task:** Create a new prompt template called `severe_weather_warning` that takes a `region` parameter and instructs the LLM to generate a severe weather warning bulletin. The prompt should tell the LLM to check the alerts resource, fetch the forecast, and format a public warning message with a severity level, affected areas, timing, and recommended actions.

**Hint:** Use the `@server.prompt()` decorator and have it instruct the LLM to use the tools and resources you have already defined.

#### Solution

```python
@server.prompt(

async def severe_weather_warning_prompt(region: str) -> str:

If no active alerts exist for {region}, state that no warnings are currently in effect and provide a general weather outlook instead.

Format the bulletin as plain text suitable for public broadcast.
"""
```

## Tier 3 -- Advanced

### Exercise 8: Design and implement a multi-resource documentation server

**Task:** Design a URI scheme for a documentation MCP server that exposes the following data:

- A list of all available documentation sections
- The content of a specific section by slug
- A search endpoint that finds documentation by keyword
- Version metadata showing when each section was last updated

Implement this as a new MCP server in a file called `docs_server.py`. Use hardcoded sample data for the documentation sections (at least 5 sections with realistic titles and markdown content). The search endpoint should be a tool (since it performs a lookup with a query parameter), and the rest should be resources.

**Hint:** For the search tool, accept a `query: str` parameter and return matching sections with their slugs and a relevance score. For the version metadata, consider whether it belongs on each section resource or as a separate resource.

#### Solution

```python
import json
from typing import Annotated
from mcp.server import Server
from mcp.server.models import InitializationOptions

server = Server("docs-server")

DOCS_SECTIONS = [
    {
        "slug": "getting-started",
        "title": "Getting Started",
        "content": "## Welcome\n\nThis guide walks you through the initial setup.",
        "updated": "2026-06-15",
    },
    {
        "slug": "configuration",
        "title": "Configuration Reference",
        "content": "## Configuration\n\nSet environment variables in `.env`.",
        "updated": "2026-06-20",
    },
    {
        "slug": "api-reference",
        "title": "API Reference",
        "content": "## API Endpoints\n\n### GET /users\nReturns a list of users.",
        "updated": "2026-07-01",
    },
    {
        "slug": "deployment",
        "title": "Deployment Guide",
        "content": "## Deployment\n\nDeploy using Docker or direct server install.",
        "updated": "2026-06-28",
    },
    {
        "slug": "troubleshooting",
        "title": "Troubleshooting",
        "content": "## Common Issues\n\n### Connection refused\nCheck that the server is running.",
        "updated": "2026-07-02",
    },
]

@server.resource(
    uri="docs://sections",
    name="Documentation Sections",
    mime_type="application/json",
)
async def list_sections() -> str:
    summary = [
        {"slug": s["slug"], "title": s["title"], "updated": s["updated"]}
        for s in DOCS_SECTIONS
    ]
    return json.dumps({"sections": summary})

@server.resource(
    uri="docs://sections/{slug}",
    name="Documentation Section Content",
    mime_type="text/markdown",
)
async def get_section(slug: str) -> str:
    for section in DOCS_SECTIONS:
        if section["slug"] == slug:
            return section["content"]
    return json.dumps({"error": f"Section '{slug}' not found"})

@server.resource(
    uri="docs://versions",
    name="Documentation Version Metadata",
    mime_type="application/json",
)
async def get_versions() -> str:
    versions = [
        {"slug": s["slug"], "title": s["title"], "updated": s["updated"]}
        for s in DOCS_SECTIONS
    ]
    return json.dumps({"versions": versions})

@server.tool(name="search_docs", description="Search documentation sections by keyword")
async def search_docs(query: Annotated[str, ..., "Search keyword to find in documentation"]) -> str:
    results = []
    query_lower = query.lower()
    for section in DOCS_SECTIONS:
        relevance = 0
        if query_lower in section["title"].lower():
            relevance += 10
        if query_lower in section["content"].lower():
            relevance += 5
        if relevance > 0:
            results.append({
                "slug": section["slug"],
                "title": section["title"],
                "relevance": relevance,
            })
    results.sort(key=lambda r: r["relevance"], reverse=True)
    return json.dumps({"query": query, "results": results})

if __name__ == "__main__":
    from mcp.server.stdio import stdio_server
    import asyncio

    async def main():
        async with stdio_server() as (read_stream, write_stream):
            await server.run(
                read_stream,
                write_stream,
                InitializationOptions(
                    server_name="docs-server",
                    server_version="0.1.0",
                ),
            )

    asyncio.run(main())
```

Test the server:

```bash

python docs_server.py
```

### Exercise 9: Add completion callbacks to a template resource

**Task:** Enhance the `weather://forecast/{city}` template resource in your weather server by adding a completion callback that suggests city names as the LLM types. The callback should return at least 5 city suggestions. Then test that the completion suggestions appear when using the MCP Inspector.

**Hint:** The MCP SDK supports resource template completion through a `completion_callback` parameter on the `ResourceTemplate` class. Use `ResourceTemplate` instead of the `@server.resource()` decorator for this.

#### Solution

First, refactor the template resource to use `ResourceTemplate`:

```python
from mcp.server.resources import ResourceTemplate

async def suggest_cities(missing: str) -> list[str]:
    cities = [
        "London", "Paris", "Tokyo", "New York", "Sydney",
        "Berlin", "Mumbai", "Toronto", "Seoul", "Dubai",
    ]
    if missing:
        return [c for c in cities if c.lower().startswith(missing.lower())]
    return cities

# Replace the @server.resource decorator with ResourceTemplate
forecast_template = ResourceTemplate(
    uri="weather://forecast/{city}",
    name="Weather Forecast for City",
    mime_type="application/json",
    completion_callback=suggest_cities,
)

server.register_resource_template(forecast_template)

# The handler function stays the same
async def get_forecast_resource(city: str) -> str:
    forecasts = []
    for day in range(1, 8):
        temp = random.randint(10, 35)
        conditions = random.choice(["Sunny", "Cloudy", "Rainy", "Windy", "Partly Cloudy"])
        forecasts.append({
            "day": day,
            "temperature": temp,
            "unit": "C",
            "conditions": conditions,
        })
    return json.dumps({"city": city, "forecast": forecasts})
```

Note that when using `ResourceTemplate`, the handler is not registered through the decorator. Instead, you register the template with `server.register_resource_template()` and define the handler separately. The handler function name must match the parameter name in the URI template (`city`).


In the Resources tab, when you type in the URI field for `weather://forecast/{city}`, the inspector will show completion suggestions. Start typing "L" and it should suggest "London." Type "Pa" and it should suggest "Paris." This auto-completion helps the LLM (or a human tester) discover valid parameter values without trial and error.

### Exercise 10: Add error classification and structured error responses

**Task:** Refactor your weather server so that all tools and resources return structured JSON error responses instead of plain strings. Define an error format with `error`, `error_code`, and `suggestion` fields. Then add a wrapper that catches unhandled exceptions and returns the structured format automatically.

**Hint:** Create a helper function that wraps any handler and catches exceptions. For validation errors (like out-of-range values), return error codes like `VALIDATION_ERROR`. For unexpected exceptions, return `INTERNAL_ERROR`.

#### Solution

```python
import json
import functools
from typing import Callable, Any

def structured_error(tool_func: Callable) -> Callable:
    """Decorator that wraps a tool handler with structured error handling."""

    @functools.wraps(tool_func)
    async def wrapper(*args, **kwargs):
        try:
            result = await tool_func(*args, **kwargs)
            # If the handler already returned an error string, parse it
            if isinstance(result, str):
                try:
                    parsed = json.loads(result)
                    if isinstance(parsed, dict) and "error" in parsed:
                        return result  # Already structured
                except (json.JSONDecodeError, TypeError):
                    pass
            return result
        except Exception as e:
            return json.dumps({
                "error": str(e),
                "error_code": "INTERNAL_ERROR",
                "suggestion": "This is an unexpected error. Please try again or contact support.",
            })

    return wrapper

# Define error response helper
def validation_error(message: str, suggestion: str) -> str:
    return json.dumps({
        "error": message,
        "error_code": "VALIDATION_ERROR",
        "suggestion": suggestion,
    })

# Apply to tools
@server.tool(name="get_forecast", description="Get the weather forecast for a city")
@structured_error
async def get_forecast(city: str = "London", days: int = 7, units: str = "metric") -> str:
    if days < 1 or days > 14:
        return validation_error(
            f"days must be between 1 and 14, got {days}",
            "Use a value between 1 and 14 for the forecast range.",
        )
    if units not in ("metric", "imperial"):
        return validation_error(
            f"units must be 'metric' or 'imperial', got '{units}'",
```

When the LLM receives a structured error, it can parse the `error`, `error_code`, and `suggestion` fields to decide what to do next:

```json
{
  "error": "days must be between 1 and 14, got 20",
  "error_code": "VALIDATION_ERROR",
  "suggestion": "Use a value between 1 and 14 for the forecast range."
}
```

The LLM can read the `suggestion` field and immediately retry with a corrected value, without needing to guess what went wrong. This reduces the number of failed tool calls and makes your server more robust against the LLM's imperfect reasoning.
