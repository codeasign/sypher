## The Problem Story

Imagine you are building an AI coding assistant. You have given the model access to read and write files in your project directory. The agent can open any file, search for function definitions, and even run tests. You feel good about how much ground it covers.

Then your teammate asks: "Can the agent look up the latest customer order in our database?" You freeze. The agent has no way to run a SQL query, no way to call your internal API, and no way to check whether the CI pipeline passed. The only thing it knows how to do is manipulate text files in a folder.

This is the fundamental limitation of giving an LLM raw file access: files are passive. They sit there waiting to be read or written, but they cannot represent live data, they cannot trigger side effects, and they cannot express the structured capabilities of a real system. Your database is alive. Your CI pipeline is alive. Your internal documentation portal is alive. And your agent is completely cut off from all of it.

You could, of course, write a custom Python script for every operation you need. You could wrap the database in a CLI tool that returns JSON. You could write a curl script that hits your internal API. But then you have a sprawling collection of bespoke integrations, each with its own invocation syntax, each returning data in a slightly different format, and none of them speaking a common protocol that the LLM understands.

What you need is a way to tell the LLM: "Here is a list of actions you can take, and here is a list of data sources you can read from. The actions have typed parameters. The data sources have well-known URIs. You can discover both at runtime without any hardcoded configuration."

That is exactly what the Model Context Protocol (MCP) gives you through its **tool** and **resource** primitives.

## The Concept

The MCP specification defines three core primitives that an LLM can use to interact with the outside world. Two of them are the subject of this page:

- **Tools** let LLMs perform actions. A tool is a function with a name, a description, and a typed input schema. When the LLM decides it needs to run a tool, the host forwards the call to the MCP server, which executes the action and returns a result. Tools are how agents affect the world: query a database, send an email, trigger a deployment, calculate a value.

- **Resources let LLMs read structured data.** A resource is a named data source identified by a URI. When the LLM needs information from a resource, it requests the contents at that URI, and the server returns the data along with a MIME type. Resources are how agents acquire context: read a configuration file, fetch a log snapshot, list available datasets, load a document.

- **Prompts** (the third primitive) are reusable templates. They are covered separately in the MCP topic on prompt templates.

Together, tools and resources form the **read/write surface** of an MCP server. Resources are the read side: the server exposes data that the LLM can inspect. Tools are the write side: the LLM invokes actions that change state or produce new results.

The distinction matters because it affects what the LLM can discover and how it decides what to do. Resources are listed and read passively. Tools are listed and called actively. The LLM chooses whether to read a resource or call a tool based on your descriptions, the current conversation context, and its own reasoning.

## API Reference

When you build an MCP server using the Python SDK, you use decorators to register tools and resources. Here are the key decorators, classes, and patterns you need to know.

| Decorator / Class | Purpose | Parameters | Example |
|---|---|---|---|
| `@server.tool()` | Register a tool function | name (str), description (str) | `@server.tool(name="get_forecast", description="Get weather forecast for a city")` |
| `@server.resource()` | Register a static resource | uri (str), name (str), mime_type (str) | `@server.resource(uri="weather://regions", name="Regions", mime_type="application/json")` |
| `@server.resource()` with template | Register a templated resource URI | uri template with `{param}` | `@server.resource(uri="weather://forecast/{city}")` |
| `json_schema` in tool signature | Define typed parameters via type annotations | Python type hints (str, int, list, dict) or explicit pydantic model | `city: str`, `days: int = 7`, `units: str = "metric"` |
| `URI` scheme | Namespace for resource identifiers | Custom scheme like `weather://`, `docs://`, `log://` | `weather://forecast/london` |
| `mime_type` | Content type of resource response | Standard MIME string | `"application/json"`, `"text/markdown"`, `"text/csv"` |
| `App` class | Create the MCP server application | name (str), instructions (str) | `app = App("weather-server")` |
| `server.run()` | Start the server transport | transport type (stdio, sse) | `server.run(transport="stdio")` |
| `Context` object | Access request metadata in tool/resource handlers | passed as first arg or injected | `ctx` in handler signature: `async def handler(ctx: Context)` |
| `ctx.report_progress()` | Send progress notifications for long operations | current (int), total (int) | `await ctx.report_progress(5, 10)` |
| `ctx.info()` | Send log messages to the client | message (str) | `await ctx.info("Fetching forecast data...")` |
| `ResourceTemplate` | Advanced URI template with additional options | uri, name, mime_type, completion_callback | `ResourceTemplate("weather://forecast/{city}", name="Forecast", completion_callback=suggest_cities)` |
| `CompletionCallback` | Provide auto-completion suggestions for URI template parameters | function returning list of strings | `async def suggest_cities(missing: str) -> list[str]` |

## How It Works

### Tools: Discovery and Invocation

When a client (like Claude Code or any MCP host) connects to your MCP server, it goes through a capability negotiation phase. During this phase, the client learns which primitives the server supports. If the server advertises tool support, the client proceeds with the tool lifecycle.

The lifecycle has four steps:

1. **Define.** You write a Python function and decorate it with `@server.tool()`. The SDK introspects the function signature, type annotations, and docstring to build a JSON Schema for the tool input. That schema tells the LLM exactly what parameters the tool accepts, what types they are, and whether they are required or optional.

2. **List.** The client sends a `tools/list` request to the server. The server responds with a list of all registered tools, including their names, descriptions, and input JSON Schemas. The LLM uses this information to decide which tool to call and with what arguments.

3. **Call.** The LLM decides it needs the tool. The client sends a `tools/call` request with the tool name and a JSON object of arguments. The server dispatches the call to your handler function, passing the arguments as Python keyword arguments.

4. **Respond.** Your handler runs and returns a result. The result is typically a string (rendered as text content) or a list of content items with MIME types. The client passes the result back to the LLM, which incorporates it into its reasoning.

### Resources: Discovery and Reading

Resources follow a similar lifecycle but with a different purpose. Instead of invoking an action, the LLM asks to read data from a known URI.

1. **Define.** You register a static resource with a fixed URI (like `weather://regions`) or a template resource with a URI pattern (like `weather://forecast/{city}`). The handler receives the URI and any extracted path parameters, and returns the data with a MIME type.

2. **List.** The client sends a `resources/list` request. The server responds with all registered resource URIs and their names. For template resources, the server lists the template pattern but not concrete instances (unless the server also implements resource subscriptions or dynamic discovery).

3. **Read.** The client sends a `resources/read` request with a specific URI. For static resources, the URI is the exact registered URI. For template resources, the client substitutes path parameters (e.g., `weather://forecast/london` matches `weather://forecast/{city}`, extracting `city=london`).

4. **Deliver.** The server returns the resource contents as a list of content items, each with a MIME type. The client presents this data to the LLM as if it were a file the model just read.

### JSON Schema for Tool Arguments

Every tool you register automatically gets a JSON Schema generated from your Python function signature. The SDK reads your type annotations and builds a schema that tells the LLM:

- What parameters are available
- What types they expect
- Which ones are required (those without defaults)
- The description of each parameter (from your docstring or inline annotation)

Consider this function:

```python
@server.tool(name="get_forecast", description="Get weather forecast for a city")
async def get_forecast(
    city: str = "London",
    days: int = 7,
    units: str = "metric",
) -> str:
    """Fetch forecast data."""
    # ...
```

The SDK generates this JSON Schema:

```json
{
  "$schema": "https://json-schema.org/draft/07/schema#",
  "type": "object",
  "properties": {
    "city": {
      "type": "string",
      "description": "City name",
      "default": "London"
    },
    "days": {
      "type": "integer",
      "description": "Number of forecast days",
      "default": 7
    },
    "units": {
      "type": "string",
      "description": "Temperature units: metric or imperial",
      "default": "metric"
    }
  }
}
```

When the LLM sees this schema, it knows it can call `get_forecast` with any subset of these parameters. It also knows the valid ranges and defaults, which helps it make good decisions without manual prompting.

## Cost, Limits, and Safety

### Tool Execution Costs

Every tool call has a real cost. The LLM spends tokens generating the tool call request (the function name and arguments), and it spends additional tokens processing the result. If your tool makes an external API call, you also pay for that API. If your tool queries a database, you pay for the query compute.

Consider a simple chain: the LLM calls `query_database(sql="SELECT * FROM orders LIMIT 100")`, which returns 100 rows of JSON. The input tokens for the tool call might be 20-50 tokens. The output tokens for the result might be 500-2000 tokens depending on the data size. If the LLM uses this result in its response, those tokens are billed at the model's output rate.

For expensive tools, always add clear descriptions that help the LLM decide judiciously. A tool description like `"Expensive: runs a full table scan on the orders table. Prefer get_recent_orders instead"` helps the model make cost-aware decisions.

### Sensitive Data Exposure

Resources expose data to the LLM. If your resource returns data that contains personally identifiable information (PII), API keys, internal IP addresses, or other sensitive fields, the LLM may include that data in its responses to users.

You have several strategies to mitigate this:

- **Filter at the source.** In your resource handler, strip sensitive fields before returning data. For example, if your `users://{id}` resource returns a user record, remove the `password_hash`, `ssn`, and `internal_notes` fields.

- **Use fine-grained resources.** Instead of one resource that returns everything, create separate resources for public and private data. The `weather://forecast/{city}` resource returns only forecast data, not the internal weather station configuration.

- **Log resource reads.** Use the `Context` object's `info()` method to log every resource access. This creates an audit trail you can review.

- **Apply the principle of least privilege.** Only register resources and tools that the LLM genuinely needs. If the LLM does not need to read user passwords, do not register a resource that exposes them.

### Input Validation

Tools accept arbitrary input from the LLM. While the LLM is generally well-behaved, you should still validate inputs as you would for any API endpoint. The LLM might extract incorrect arguments from the conversation, or a user might craft a prompt that causes the LLM to call your tool with unexpected parameters.

Validate:

- **Range constraints.** If `days` must be between 1 and 14, check this in your handler.
- **Allowed values.** If `units` must be `"metric"` or `"imperial"`, validate and return a clear error for anything else.
- **String length.** If a `city` name is too long, truncate or reject it.
- **SQL injection.** If your tool builds a SQL query from LLM-provided input, use parameterized queries, not string concatenation.

Here is a pattern for safe tool handlers:

```python
@server.tool(name="get_forecast")
async def get_forecast(city: str, days: int = 7, units: str = "metric") -> str:
    if days < 1 or days > 14:
        return f"Error: days must be between 1 and 14, got {days}"
    if units not in ("metric", "imperial"):
        return f"Error: units must be 'metric' or 'imperial', got '{units}'"
    # Safe: proceed with validated inputs
    data = await fetch_forecast(city=city, days=days, units=units)
    return json.dumps(data)
```

## When to Use Tools vs Resources

The distinction between tools and resources can feel blurry at first. Both let the LLM interact with your server. Both require async handler functions. Both return content. Here is a rule of thumb:

| Use a tool when ... | Use a resource when ... |
|---|---|
| The operation changes state (side effect) | The operation only reads data (idempotent) |
| The operation takes typed parameters | The data is identified by a URI |
| The result is computed fresh each time | The result can be cached or is static |
| The operation is an action the LLM performs | The operation is data the LLM reads |
| Examples: send email, create ticket, trigger build, insert record | Examples: read config, fetch user profile, list documents, load log |

A good mental model: **Resources are files; tools are functions.** A file server exposes files by path. You list available paths, then you read the file you want. A function server exposes functions by name. You list available functions, then you call the one you want with arguments.

Some operations do not fit neatly into either category. For example, querying a database with a SQL string: you could expose `db://query/{sql_encoded}` as a resource or `query_database(sql: str)` as a tool. The community convention is to use a tool here, because the operation is active (it runs a query), it is not idempotent (the same SQL can return different results over time), and it takes a complex parameter (the SQL string). Use your judgment and be consistent within your project.

## Mental Model Check

Test your understanding with these questions.

**Q1: You need to expose a list of available datasets in your data warehouse. Should you use a tool or a resource?**

A1: Use a resource with a static URI like `data://datasets`. This is read-only, the data changes infrequently, and the LLM just needs to browse available options. A tool would be overkill and would imply the LLM is "performing an action" rather than reading a listing.

**Q2: You need to let the LLM trigger a reindex of your search index. What do you use?**

A2: Use a tool like `reindex_search().` This has a side effect (it rebuilds the search index), it is not idempotent, and it is clearly an action the LLM is performing. A resource would be inappropriate because the LLM is not reading data; it is requesting a state change.

**Q3: How does the LLM discover what tools and resources are available on your server?**

A3: At connection time, the client sends `tools/list` and `resources/list` requests to the server. The server responds with all registered tools (including their names, descriptions, and JSON Schemas) and all registered resources (including their URIs, names, and MIME types). The LLM inspects this list to decide what to call or read. No hardcoded configuration on the client side is needed.

**Q4: Can the same URI pattern match multiple template registrations?**

A4: Yes, but the SDK resolves the most specific match first. If you register `weather://{region}` and `weather://forecast/{city}`, a request for `weather://forecast/london` matches the second template because `forecast` is a static path segment that distinguishes it from the first template. If you register two templates with identical static prefixes and only a parameter differs, the first registration wins and the second is shadowed. Always design URI templates with distinct static path segments to avoid ambiguity.

**Q5: What happens if the LLM calls a tool with invalid arguments?**

A5: The MCP SDK validates the arguments against the JSON Schema before calling your handler. If validation fails, the server returns an error response with a message describing what went wrong (e.g., "Expected type 'integer' for property 'days', got 'string'"). Your handler is never called with invalid data. However, semantic validation (like range checks or allowed values) is your responsibility in the handler.

## How It Actually Works: Protocol Mechanics

Understanding what happens at the wire level helps you debug when things go wrong. Every MCP interaction between client and server uses JSON-RPC 2.0 as its message format, sent over the transport layer (stdio or SSE).

### The Tools List Exchange

When a client first connects, it sends a `tools/list` request. The server responds with a JSON-RPC response containing the full tool definitions. Here is what the wire message looks like:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "get_forecast",
        "description": "Get the weather forecast for a city",
        "inputSchema": {
          "type": "object",
          "properties": {
            "city": {"type": "string", "description": "City name", "default": "London"},
            "days": {"type": "integer", "description": "Number of days", "default": 7},
            "units": {"type": "string", "description": "metric or imperial", "default": "metric"}
          }
        }
      }
    ]
  }
}
```

### The Tools Call Exchange

When the LLM decides to call a tool, the client sends a `tools/call` request:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "params": {
    "name": "get_forecast",
    "arguments": {"city": "London", "days": 3, "units": "metric"}
  }
}
```

Your handler receives `city="London"`, `days=3`, `units="metric"` as keyword arguments. It runs and returns a result. The server wraps the result in a JSON-RPC response:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\"city\": \"London\", \"forecast\": [...]}"
      }
    ]
  }
}
```

### The Resources List and Read Exchange

Resources follow the same pattern but use `resources/list` and `resources/read` methods. The read request includes a URI parameter:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "params": {
    "method": "resources/read",
    "params": {"uri": "weather://forecast/london"}
  }
}
```

The response includes MIME-typed content items that the client presents to the LLM.

## Cost, Limits, and Safety — Expanded

### Rate Limiting and Throttling

If your tools call external APIs that have rate limits, you need to protect those APIs from aggressive LLM usage. An LLM in a reasoning loop might call your tool multiple times in rapid succession, especially if it is trying different parameter combinations.

Implement rate limiting in your tool handlers:

```python
import time
from collections import defaultdict

_rate_limit_store = defaultdict(list)

async def check_rate_limit(key: str, max_calls: int = 10, window: float = 60.0):
    now = time.time()
    _rate_limit_store[key] = [t for t in _rate_limit_store[key] if now - t < window]
    if len(_rate_limit_store[key]) >= max_calls:
        return False
    _rate_limit_store[key].append(now)
    return True

@server.tool(name="call_external_api")
async def call_external_api(endpoint: str) -> str:
    allowed = await check_rate_limit("external_api")
    if not allowed:
        return "Rate limit exceeded. Try again in a few seconds."
    # Proceed with the call
```

### Resource Access Control

Not every resource should be readable by every user. If your MCP server serves multiple users (via SSE transport with authentication), check access in the resource handler:

```python
@server.resource(uri="internal://deploy-keys/{env}")
async def get_deploy_keys(ctx: Context, env: str) -> str:
    # Check that the requesting user has access to this environment
    user_role = ctx.metadata.get("role", "viewer")
    if env == "production" and user_role != "admin":
        return json.dumps({"error": "Access denied. Admin role required for production keys."})
    # Return the keys
    keys = await fetch_keys(env)
    return json.dumps({"keys": keys})
```

The `Context` object passed to your handler carries metadata about the requesting client when using SSE transport with authentication. On stdio, the context is simpler — the client is the local process that spawned the server — so access checks are typically unnecessary.

### Observability and Monitoring

Treat your MCP server like any production service. Add logging to every tool and resource handler:

```python
import logging

logger = logging.getLogger("mcp-weather-server")

@server.tool(name="get_forecast")
async def get_forecast(city: str, days: int = 7, units: str = "metric") -> str:
    logger.info("get_forecast called: city=%s, days=%d, units=%s", city, days, units)
    start = time.time()
    try:
        result = await fetch_forecast(city, days, units)
        elapsed = time.time() - start
        logger.info("get_forecast completed in %.2fs", elapsed)
        return result
    except Exception as e:
        logger.error("get_forecast failed: %s", str(e), exc_info=True)
        return json.dumps({"error": f"Forecast retrieval failed: {str(e)}"})
```

Key metrics to track:
- **Call frequency** per tool and per resource — identify which primitives the LLM uses most
- **Error rate** per tool — catch validation failures, external API errors, and unhandled exceptions
- **Latency** per call — detect slow handlers that might cause the LLM to time out
- **Data volume** per resource read — large resource responses consume many tokens

## Designing Your URI Scheme

The URI scheme you choose for your resources is your server's public API design. Good URI design makes your server intuitive for both the LLM and for humans reading your code.

### Scheme Naming Conventions

Use a scheme name that clearly identifies the domain:

```python
# Good — clear domain association
weather://regions
weather://forecast/{city}
weather://stations/{station_id}

# Also good
docs://api-reference
docs://tutorials/{slug}
docs://search?q={query}

# Avoid generic schemes
data://things/{id}       # What kind of data? Too vague
api://get-forecast       # Looks like a function call, not a data identifier
```

### Hierarchical URIs

Structure your URIs hierarchically to mirror the domain's natural organization:

```text
weather://regions                    # Top-level collection
weather://regions/{region_id}        # Single region detail
weather://regions/{region_id}/       # Stations within a region
weather://forecast/{city}            # Forecast data
weather://stations/{station_id}/     # Station metadata
weather://stations/{station_id}/     # Station readings over time
```

This hierarchy helps the LLM guess related URIs. If the model sees `weather://regions` in the resource list, it can infer that `weather://regions/{id}` might exist for individual region details, even if that URI was not explicitly listed.

### Avoiding URI Ambiguity

When designing URI templates, use static path segments to disambiguate between patterns:

```python
# Ambiguous — both start with weather://{param}
@server.resource(uri="weather://{region}")          # Matches any single segment
@server.resource(uri="weather://{city}/forecast")    # Never matched because {region} catches everything

# Unambiguous — static prefix distinguishes them
@server.resource(uri="weather://regions/{region_id}")
@server.resource(uri="weather://forecast/{city}")
```

The second approach is safer because the static segment (`regions` vs `forecast`) acts as a namespace qualifier that the template matcher can reliably distinguish.

## Additional Mental Model Check Questions

**Q6: You are designing a URI scheme for a code analysis tool. The LLM needs to read source files by path and also list available analysis modules. How would you structure these URIs?**

A6: Use a hierarchical scheme with clear static segments:
- `code://files/{filepath}` for source file access
- `code://analyzers` for listing available analysis modules
- `code://analyzers/{name}` for details about a specific analyzer

This keeps read operations (resources) separate from any analysis actions (tools). The `code://` prefix clearly identifies the domain, and the static segments `files/` and `analyzers/` disambiguate the two data categories.

**Q7: Your tool calls an external API that costs $0.01 per call and has a rate limit of 100 requests per minute. How would you communicate these constraints to the LLM?**

A7: Embed the cost and rate information in the tool description:
```python
@server.tool(
    name="search_products",
    description="Search product catalog. Cost: $0.01/call. Rate limit: 100/min. "
                "Prefer batch queries over individual lookups. Use the catalog:// "
                "resources for browsing instead."
)
```

The description is the only signal the LLM gets about when and how often to call the tool. Including cost and rate information in the description helps the model make informed decisions. You should also implement server-side rate limiting as a safety net, since the LLM may not always heed the description.

**Q8: You expose a `users://{id}` resource that returns user profiles. One profile field is an `email` address. Should you include it in the resource response?**

A8: It depends on whether the LLM needs email addresses to perform its tasks. Apply the principle of least privilege: only include fields the LLM genuinely needs. If the LLM's purpose is customer support, it probably needs the email to contact the user. If the LLM's purpose is internal analytics, it does not need the email and including it is unnecessary data exposure. When in doubt, create two resources: `users://{id}/public` (safe fields) and `users://{id}/private` (sensitive fields), and control access through authentication context in the handler.
