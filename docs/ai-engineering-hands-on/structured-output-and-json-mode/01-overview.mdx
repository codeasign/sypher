You build an app that extracts invoice data — vendor name, total amount, date, line items — from emailed PDFs. You send the invoice text to an LLM and ask it to "return the data as JSON." Most of the time it works. But sometimes the model returns `{"vendor": "Acme Corp", "total": 1234.56}` and other times `vendor: Acme Corp, total: 1234.56`. Sometimes it wraps the JSON in a markdown code block. Sometimes it says "Sure! Here is the JSON:" before the data. Your parser has to handle every variation. Then one day the model returns `{"error": "I cannot process this invoice"}` in JSON format — valid JSON, but not the schema you expected. Your parser crashes because the `total` field is missing.

This is the problem structured output solves. Instead of asking the model to "return JSON" and hoping it matches your schema, you define the exact schema upfront, and the model is constrained to produce output that conforms to it. If the model cannot produce a valid response matching the schema, it tells you so explicitly — no silent schema violations.

## The Concept

Structured output means constraining an LLM's response to follow a specific JSON schema that you define. Instead of the model freely generating text and you hoping to parse it, the model's output is guaranteed to match your schema — or the API returns an error. Think of it like a web form with field validation versus a blank text box. Without structured output, you ask "give me JSON" and get unpredictable results. With structured output, you define the fields, types, and constraints, and the API enforces them.

## API Reference

### OpenAI Structured Outputs

| Method / Parameter | What It Does | Example |
|---|---|---|
| `response_format={"type": "json_schema", "json_schema": {"name": "...", "schema": {...}}}` | Constrains output to a JSON schema | `response_format={"type": "json_schema", "json_schema": schema}` |
| `response_format={"type": "json_object"}` | Legacy JSON mode — returns valid JSON but no schema enforcement | `response_format={"type": "json_object"}` |
| `strict=True` (inside `json_schema`) | Forces the model to strictly follow the schema (recommended) | `"strict": true` |
| `json_schema.name` | Required name for the schema — used for the function/tool definition | `"name": "extract_invoice"` |
| `json_schema.schema` | The JSON Schema object defining the output structure | `{"type": "object", "properties": {...}}` |
| `json_schema.strict` | Boolean (default true) — enforces schema adherence | `"strict": true` |
| `json_schema.description` | Optional description of what the schema represents | `"description": "Extract invoice fields"` |
| `refusal` string | When the model refuses, the output `refusal` field is set instead of content | `if response.refusal:` |

### OpenAI Legacy JSON Mode (deprecated in favor of Structured Outputs)

| Parameter | What It Does | Example |
|---|---|---|
| `response_format={"type": "json_object"}` | Guarantees valid JSON but does not enforce a schema | `response_format={"type": "json_object"}` |
| Requires `system` or `developer` prompt mentioning JSON | The model needs instructions to produce JSON | `"You are a JSON generator..."` |

### Anthropic — No Native Structured Output

Anthropic does not have a built-in structured output mode. Instead, you use tool use (function calling) to constrain output. Define a tool with the desired JSON schema, and Claude will call it with arguments matching the schema:

| Method / Parameter | What It Does | Example |
|---|---|---|
| `tools` parameter in `client.messages.create()` | Define tools with JSON Schema inputs | `tools=[{"name": "extract", "input_schema": {...}}]` |
| `tool_choice={"type": "tool", "name": "..."}` | Force Claude to use a specific tool | `tool_choice={"type": "tool", "name": "extract"}` |
| `tool_choice={"type": "any"}` | Claude must use one of the provided tools | `tool_choice={"type": "any"}` |
| `tool_choice={"type": "auto"}` (default) | Claude decides whether to use tools | `tool_choice={"type": "auto"}` |
| Response `content[0].type == "tool_use"` | Check if response contains a tool call | `if block.type == "tool_use":` |
| `block.input` | The structured arguments matching the schema | `data = block.input` |

## How It Actually Works — Step by Step

### Step 1: Define Your Schema

You define a JSON Schema that describes the exact shape of the output you want. Every field name, type, and constraint is declared upfront.

```python
schema = {
    "name": "extract_invoice",
    "schema": {
        "type": "object",
        "properties": {
            "vendor_name": {
                "type": "string",
                "description": "The name of the vendor or company"
            },
            "invoice_date": {
                "type": "string",
                "description": "Date of the invoice in YYYY-MM-DD format"
            },
            "total_amount": {
                "type": "number",
                "description": "Total amount due on the invoice"
            },
            "currency": {
                "type": "string",
                "enum": ["USD", "EUR", "GBP", "JPY"],
                "description": "Three-letter currency code"
            },
            "line_items": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "description": {"type": "string"},
                        "quantity": {"type": "integer"},
                        "unit_price": {"type": "number"},
                        "total": {"type": "number"}
                    },
                    "required": ["description", "quantity", "unit_price", "total"],
                    "additionalProperties": False
                }
            }
        },
        "required": ["vendor_name", "invoice_date", "total_amount", "currency", "line_items"],
        "additionalProperties": False
    },
    "strict": True
}
```

### Step 2: Send the Schema with Your Request

With OpenAI's Structured Outputs, you pass the schema as part of the request:

```python
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "developer", "content": "Extract invoice data from the following text."},
        {"role": "user", "content": invoice_text}
    ],
    response_format={
        "type": "json_schema",
        "json_schema": schema
    }
)

result = json.loads(response.choices[0].message.content)
```

The model is constrained at the token generation level to produce output matching the schema. It cannot produce output with missing required fields, extra properties (if `additionalProperties: False`), or fields of the wrong type.

### Step 3: The Model Generates Constrained Output

Under the hood, constrained generation works differently than free-form text. The API:

1. Parses your schema and converts it into a context-free grammar (a set of allowed token sequences)
2. At each generation step, filters the token probability distribution to only include tokens that would produce valid JSON matching the schema
3. If no valid continuation exists, the model sets a `refusal` flag

This is fundamentally different from "generate JSON and validate after." The validation happens during generation, not after. Invalid tokens are never produced.

### Step 4: Parse the Response (Trivially)

Because the output is guaranteed to match your schema, parsing is trivial:

```python
import json

content = response.choices[0].message.content
data = json.loads(content)  # Always succeeds if structured output was used

print(f"Vendor: {data['vendor_name']}")
print(f"Total: {data['total_amount']}")
for item in data['line_items']:
    print(f"  - {item['description']}: {item['total']}")
```

No try-except for JSON parsing errors. No fallback regex patterns. The output either matches the schema or the model refuses — there is no third option.

### Step 5: Handle Refusals

The model can refuse if the input violates its content policy or if it cannot produce output matching the schema:

```python
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    response_format={"type": "json_schema", "json_schema": schema}
)

if response.choices[0].message.refusal:
    print(f"Model refused: {response.choices[0].message.refusal}")
else:
    data = json.loads(response.choices[0].message.content)
    # Process the data
```

Always check for `refusal` before processing the content. A refusal is not an error — it is the model correctly identifying that it cannot produce a valid response.

### Step 6: Anthropic — Structured Output via Tool Use

Anthropic does not have a native structured output mode. Instead, you define a tool with your desired schema and force Claude to call it:

```python
from anthropic import Anthropic

client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

response = client.messages.create(
    model="claude-sonnet-5",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "Extract invoice data: INVOICE #1234 from Acme Corp, $500 total."}
    ],
    tools=[
        {
            "name": "extract_invoice",
            "description": "Extract structured invoice data",
            "input_schema": {
                "type": "object",
                "properties": {
                    "vendor_name": {"type": "string"},
                    "total_amount": {"type": "number"},
                    "currency": {"type": "string", "enum": ["USD", "EUR", "GBP"]}
                },
                "required": ["vendor_name", "total_amount"]
            }
        }
    ],
    tool_choice={"type": "tool", "name": "extract_invoice"}
)

for block in response.content:
    if block.type == "tool_use":
        data = block.input
        print(f"Vendor: {data['vendor_name']}")
        print(f"Total: {data['total_amount']}")
```

## Cost, Limits, and Safety

### Token Costs

Structured output does not add token cost. The same per-token pricing applies. However, the schema itself is sent as part of the request (counted as input tokens), so larger schemas increase the input token count slightly. A typical schema adds 100–500 input tokens.

### Constraints and Limitations

OpenAI Structured Outputs supports:
- `string`, `number`, `integer`, `boolean`, `array`, `object` types
- `enum` constraints for strings
- `required` fields
- `additionalProperties: False` to prevent extra fields
- Nested objects up to 5 levels deep
- `description` fields for each property

OpenAI Structured Outputs does NOT support:
- `$ref` or `definition` references (inline schemas only)
- `anyOf`, `oneOf`, `allOf` (use inline optional fields instead)
- `default` values
- `null` types (make the field optional by removing it from `required`)
- Recursive schemas

### Safety Rules

**Always use `strict: true`.** Without strict mode, the model may deviate from the schema. Strict mode is the main value of Structured Outputs.

**Always check for `refusal`.** A model can refuse for valid content policy reasons. Your code must handle this gracefully — do not assume the response always contains valid data.

**Do not use Legacy JSON Mode for production.** OpenAI's `json_object` mode only guarantees valid JSON, not valid schema. Use `json_schema` with `strict: true` for production.

**Keep schemas focused.** A schema with 50+ fields or deeply nested objects (5+ levels) may reduce reliability. Split large schemas into sequential, smaller extractions.

**Use the `description` field on each property.** The model uses these descriptions to understand what to extract. Without descriptions, the model may populate fields incorrectly.

## When to Use This / When Not To

**Use structured output when:**
- You need to reliably parse LLM output into a programmatic data structure
- The output feeds into a database, API, or type-safe code
- You need the model to follow a specific format (e.g., extracting entities, classifying text, generating config)
- You are building a pipeline where downstream processing depends on field presence and types

**Do not use structured output when:**
- You only need free-form text (summaries, creative writing, conversational responses)
- The schema would be so large that it consumes most of your context window
- You are prototyping and the output format is still changing — use free text first, then lock down the schema
- You need capabilities not supported by the constrained generation (recursive types, complex `anyOf` logic)

## Mental Model Check

**Question 1:** What is the fundamental difference between "return JSON" and structured output?

  With "return JSON," the model generates free text that happens to be JSON — it can produce inconsistent formats, extra text, or invalid JSON. With structured output, the model is constrained at the token generation level to only produce output matching your schema.

---

**Question 2:** What does the `refusal` field indicate in a structured output response?

  It means the model could not or would not generate output matching the schema. This could be due to content policy violations, input that does not contain the requested information, or a schema that cannot be satisfied.

---

**Question 3:** How does Anthropic achieve structured output without a native structured output mode?

  By defining a tool with the desired JSON Schema as its `input_schema` and using `tool_choice={"type": "tool", "name": "..."}` to force Claude to call it. The tool call arguments are guaranteed to match the schema.

---

**Question 4:** Why does `strict: true` matter for OpenAI Structured Outputs?

  Without `strict: true`, the model may produce output that technically parses as JSON but does not match your schema — extra fields, wrong types, or missing required fields. Strict mode enforces exact schema adherence.

---

**Question 5:** What types of schemas are NOT supported by OpenAI Structured Outputs?

  `$ref` references, `anyOf`/`oneOf`/`allOf` combinators, `default` values, `null` types, and recursive schemas are not supported. Use inline schemas, optional fields (remove from `required`), and flat structures instead.
