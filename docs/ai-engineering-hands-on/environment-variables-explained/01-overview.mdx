# Environment Variables Explained from Zero

⏱ 15 minutes · Difficulty: Beginner

## What You Will Learn

By the end of this lesson you will be able to:

- Explain what an environment variable is and why it exists
- Read, set, and unset environment variables in your terminal
- Access environment variables from Python using `os.environ`
- Create a `.env` file to store secrets safely
- Understand why hardcoding API keys is dangerous and how env vars solve it
- Use the `python-dotenv` library to load variables from a `.env` file
- Describe the difference between temporary and persistent environment variables
- Verify that secrets are excluded from version control using `.gitignore`

## Prerequisites

You should already understand:

- What a terminal is and how to run a command
- What Python is and how to run a Python script
- The basic idea of a variable (a name that holds a value)
- How to create a file with a text editor

## The Problem Story

Imagine you just signed up for an AI API -- OpenAI, Anthropic, or any other. The website gives you a long secret key that looks like this:

```
sk-proj-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

You are told: "Keep this key secret. Anyone who has it can use your account and you will be billed for their usage."

Your first instinct is probably: "I will write it right into my Python script." So you create a file called `app.py` that looks like this:

```python
api_key = "sk-proj-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
print("My API key is:", api_key)
```

This works. The script runs. You feel good.

Then reality catches up.

### Problem 1: The GitHub leak

You put this script on GitHub so your teammate can see it. Now your secret key is public. Anyone who finds it can run up a massive bill on your account. Within hours, bots scrape GitHub for exposed keys. They use your key to generate images, run language models, and do everything the API allows. You get the bill at the end of the month and it is in the thousands of dollars.

You delete the key from the API provider's dashboard, but the damage is done. You have to explain to your team why the budget was blown. You learn a hard lesson about what "keep this key secret" actually means.

### Problem 2: Environment switching

You have two API accounts: one for testing (where you experiment and break things) and one for production (where real users depend on your app). Your script hardcodes the test key. When you deploy to production, you need to swap it for the production key.

You manually edit `app.py`:

```python
api_key = "sk-proj-PRODUCTION-KEY-HERE"  # changed from test key
```

Two weeks later, you deploy again. You forget to swap the key back. Now you are running production traffic against your test account. The test account has strict rate limits, so your users see errors. You scramble to fix it, and your team loses trust in your deployments.

### Problem 3: The stolen laptop

You are working from a coffee shop. You step away for a moment and your laptop is stolen. The thief now has your API key sitting in plain text in a file on the desktop. They do not need to crack any passwords or bypass any security -- they just open the file and read the key.

Even if you remotely wipe the laptop, the key was already exposed. You spend the next hour rotating every API key, every database password, and every secret you ever hardcoded.

### How environment variables solve all three

Environment variables fix every one of these scenarios:

1. **GitHub leak:** The secret is never in your source code, so it cannot be pushed to GitHub. The code says "read the key from the environment" but the key itself stays on your machine.

2. **Environment switching:** You set a different key in each terminal. Your dev terminal has the test key. Your production server has the production key. The same code works in both places with zero edits.

3. **Stolen laptop:** The env var vanishes when you close the terminal. Even if you persist it in a `.env` file, that file is never mixed into your source code. It is one file you can delete remotely, and the thief gets nothing useful.

## The Concept

An **environment variable** is a key-value pair that lives outside your program, in the shell session that runs it. Your program asks the operating system for the value when it starts, and the value is never written into your source code.

Think of it like a sticky note on the outside of a locked box. The box is your computer. The sticky note says "OpenAI key = sk-abc123". When your Python script needs the key, it reaches out of the box, reads the sticky note, and uses it -- but the key never gets written into the box itself (your code).

```
┌─────────────────────────────────────┐
│  Your Source Code (on GitHub)       │
│  import os                          │
│  key = os.environ["API_KEY"]        │
│  client = OpenAI(api_key=key)       │
│                                     │
│  No API key text anywhere in here   │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│  Your Shell Session (not on GitHub) │
│  API_KEY=sk-abc123                  │
│  python app.py                      │
└─────────────────────────────────────┘
```

The source code contains the instruction "read the key from the environment" but never the key itself. The key lives in the shell session that starts your program. When you push your code to GitHub, only the instruction goes up -- the key stays on your machine.

### How variables look in different shells

The way you interact with environment variables depends on which shell your terminal is running. On macOS and Linux, the default shell is usually `bash` or `zsh`. On Windows, the default shell is PowerShell. The concepts are identical across all three, but the syntax differs.

**macOS/Linux (bash/zsh):**

| Action | Command |
|--------|---------|
| Set a variable | `export MY_VAR=hello` |
| Read a variable | `echo $MY_VAR` |
| Unset a variable | `unset MY_VAR` |
| Print all variables | `env` |

**Windows (PowerShell):**

| Action | Command |
|--------|---------|
| Set a variable | `$env:MY_VAR = "hello"` |
| Read a variable | `echo $env:MY_VAR` |
| Unset a variable | `Remove-Item Env:MY_VAR` |
| Print all variables | `Get-ChildItem Env:` |

### Where environment variables come from

Environment variables come from several sources:

- **Your operating system sets them automatically.** Variables like `PATH`, `HOME` (macOS/Linux), or `USERPROFILE` (Windows) are set when your computer starts. They tell programs where to find executable files, where your home directory is, and so on.

- **Your shell startup files set them.** Files like `~/.bashrc`, `~/.zshrc`, or `~/.profile` run every time you open a terminal. If you put `export` commands in these files, those variables are available in every terminal session.

- **You set them manually.** When you type `export API_KEY=abc123` in the terminal, you are adding a new variable to the current session.

- **Your deployment platform sets them.** When you deploy to a cloud service like Heroku, AWS Elastic Beanstalk, or Vercel, you enter your secrets in a web dashboard, and the platform injects them as environment variables into your running application.

### Reference Table

This table covers every common operation you need with environment variables, across all three shell environments.

| Concept | Command / Code | What It Does |
|---------|---------------|--------------|
| Print all env vars | `env` (macOS/Linux) or `Get-ChildItem Env:` (PowerShell) | Shows every environment variable currently set in this session |
| Print one env var | `echo $API_KEY` (macOS/Linux) or `echo $env:API_KEY` (PowerShell) | Displays the value of a single named variable |
| Set a variable (temporary) | `export API_KEY=sk-abc` (macOS/Linux) or `$env:API_KEY="sk-abc"` (PowerShell) | Sets a variable for the current terminal session only -- gone when you close the window |
| Set a variable inline | `API_KEY=sk-abc python app.py` (macOS/Linux) | Sets a variable for one single command execution, then it vanishes |
| Unset a variable | `unset API_KEY` (macOS/Linux) or `Remove-Item Env:API_KEY` (PowerShell) | Removes a variable from the current session |

## How It Works

### Step 1: Every process inherits the environment of its parent

When you open a terminal, the terminal program starts as a **process** -- an instance of a running program. Every process has a block of memory called its **environment block**. This block is a list of key-value pairs, stored as strings.

When you run `python app.py` from the terminal, the terminal creates a new child process for Python. That child process automatically gets a complete copy of the terminal's environment block. Python can then read those variables using `os.environ`.

This inheritance chain goes all the way up. Your terminal inherited its environment from the shell that started it, which inherited from the login manager, which inherited from the operating system kernel at boot time.

### Step 2: Environment variables are session-scoped

An environment variable set in one terminal window is never visible in another. Each terminal is a separate process with its own environment block. If you open two terminal windows and set `API_KEY` to a different value in each, they operate independently and neither knows about the other.

This isolation is a deliberate safety feature. It means:

- You can run your development API key in terminal window A and your production API key in terminal window B, and they will never interfere.
- You can test changes to environment variables without affecting any other work you have running.
- When you close a terminal, all the variables you set in that session are discarded automatically -- no cleanup needed.

Try this yourself. Open two terminals. In the first, type `export MY_VAR=hello`. In the second, type `echo $MY_VAR`. You will see an empty line. The variable simply does not exist in the second terminal's environment.

### Step 3: Setting a variable lasts only as long as that terminal session

When you type `export API_KEY=sk-abc123` and press Enter, the variable exists in that terminal's environment block from that moment forward. When you close the terminal window (or exit the shell), every variable you set is gone.

This temporary nature is useful for several scenarios:

- **Testing:** You set a variable, run your script, see the result, and close the terminal. No secrets left on the system.
- **One-off commands:** Use the inline syntax `API_KEY=sk-abc python app.py` to set a variable for a single command. It exists only for the duration of that `python` process.
- **Avoiding startup file clutter:** If you only need a variable once, do not add it to `~/.bashrc`. Just set it in the terminal and let it vanish.

To make a variable persist across terminal sessions, you add the `export` command to your shell's startup file. On macOS with zsh, that file is `~/.zshrc`. On Linux with bash, it is `~/.bashrc`. On Windows with PowerShell, you use `$PROFILE` or the `setx` command.

```bash
# Add this line to ~/.zshrc or ~/.bashrc
export API_KEY="sk-abc1234567890abcdef"
```

After saving the file, either open a new terminal or run `source ~/.zshrc` to apply the changes to the current session.

### Step 4: Python reads variables via os.environ

Python provides a dictionary-like object called `os.environ` that mirrors the current process's environment block. You access it just like a regular dictionary -- by key name in square brackets.

```python
import os

api_key = os.environ["API_KEY"]
print(f"Using API key ending in ...{api_key[-4:]}")
```

If the variable does not exist, Python raises a `KeyError` and stops execution. This is a hard crash.

To handle the case where a variable might be missing, use the `.get()` method instead:

```python
import os

api_key = os.environ.get("API_KEY")
if api_key is None:
    print("API_KEY was not set. Using guest mode.")
    api_key = "guest"
```

The `.get()` method takes an optional second argument -- the default value to return if the key is not found:

```python
api_key = os.environ.get("API_KEY", "no-key-provided")
print(f"API key status: {api_key}")
```

Using `.get()` with a sensible default is the safest pattern. Your script continues running even if a particular variable was not set, and you can decide how to handle the missing value.

You can also check whether a variable exists without reading it:

```python
import os

if "API_KEY" in os.environ:
    print("API_KEY is set. Proceeding with authenticated mode.")
else:
    print("API_KEY is not set. Please create a .env file.")
```

The `in` operator checks for the key's existence without retrieving the value.

### Step 5: A .env file bridges the gap between temporary and persistent

Typing `export` every time you open a terminal is tedious and error-prone. A `.env` file stores your variables in a plain text file so you can load them on demand. The file follows a simple format:

```bash
# .env -- store your secrets here, one per line
API_KEY=sk-abc1234567890abcdef
DB_HOST=localhost
DB_PORT=5432
DB_NAME=mydatabase
DEBUG=true
```

Rules for writing a `.env` file:

- One variable per line
- Format is `KEY=VALUE` with no spaces around the `=`
- Keys are case-sensitive and conventionally written in `SCREAMING_SNAKE_CASE`
- Lines starting with `#` are comments and are ignored
- Values are stored as strings, even if they look like numbers or booleans
- If a value contains spaces, you can wrap it in quotes: `GREETING="Hello World"`
- Quotes are stripped when the value is loaded

This file is never committed to GitHub. You add `.env` to your `.gitignore` file so Git ignores it. We cover this in Step 6.

To load these variables into Python, you use the `python-dotenv` library. Install it first:

```bash
pip install python-dotenv
```

Then use `load_dotenv()` at the top of your script:

```python
import os
from dotenv import load_dotenv

load_dotenv()  # reads .env and adds everything to os.environ

api_key = os.environ["API_KEY"]
print(f"Loaded API key ending in ...{api_key[-4:]}")
```

The `load_dotenv()` function looks for a file named `.env` in the current working directory. If the file exists, it reads every `KEY=VALUE` pair and adds it to `os.environ`. If a key already exists in the environment (because you set it in the terminal, for example), `load_dotenv()` will not overwrite it -- the terminal-set value takes precedence.

You can also load a `.env` file from a different path:

```python
from dotenv import load_dotenv
load_dotenv("/path/to/custom.env")
```

### Step 6: The .gitignore file protects your secrets from GitHub

A `.gitignore` file tells Git which files to ignore. If a file is listed in `.gitignore`, Git will never stage, commit, or push it -- even if the file exists in your working directory.

Create a `.gitignore` file in the root of your project with this content:

```bash
# .gitignore
.env
```

The standard Python `.gitignore` template already includes `.env`. If you are using a template from GitHub or a framework like Django or FastAPI, check that `.env` is in the list. If it is not, add it immediately.

You can verify that a file would be ignored without risking a commit:

```bash
git check-ignore .env
```

If it prints `.env`, the file is safely ignored. If it prints nothing, your `.env` file would be committed to Git -- fix this before your first commit.

### Step 7: Environment variables in Python are always strings

Every value in `os.environ` is a string. Even if you set `DB_PORT=5432` in your `.env` file, `os.environ["DB_PORT"]` returns the string `"5432"`, not the integer `5432`.

This means you need to convert values to the correct type before using them:

```python
import os
from dotenv import load_dotenv

load_dotenv()

# All values are strings
db_port_str = os.environ.get("DB_PORT", "5432")
print(type(db_port_str))  # <class 'str'>

# Convert to the type you need
db_port_int = int(os.environ.get("DB_PORT", "5432"))
print(type(db_port_int))  # <class 'int'>

# Booleans are not automatically parsed
debug_str = os.environ.get("DEBUG", "false")
debug = debug_str.lower() in ("true", "1", "yes")
print(f"Debug mode: {debug}")  # Debug mode: False (unless DEBUG=true)
```

This is one of the most common sources of bugs when working with environment variables. Always convert explicitly.

### Step 8: You can override .env values from the terminal

A useful pattern: you can set a default value in `.env` and then override it from the terminal for testing. Because `load_dotenv()` does not overwrite existing environment variables, the terminal value always wins.

```bash
# Set a temporary override
export DB_HOST=test-server.example.com
python my_script.py
# The script sees DB_HOST=test-server.example.com, not whatever was in .env
```

This lets you test against different databases, different API keys, or different feature flags without editing any files. When you close the terminal, the override vanishes and the `.env` default is used again.

## Cost, Limits, and Safety

### Cost

Environment variables cost nothing. They are a fundamental feature of your operating system and have been part of every major OS since the 1970s. The `python-dotenv` library is free, open source, and maintained by the community. There are no usage charges, no API calls, and no subscriptions.

### Limits

Environment variables have several practical constraints you should know about:

- **Name rules:** Variable names can only contain letters (A-Z, a-z), digits (0-9), and underscores. Do not use hyphens, spaces, or special characters in names. `MY-VAR` is invalid in most shells. `MY_VAR` is correct.

- **Case sensitivity:** Names are case-sensitive on all modern systems. `API_KEY` and `api_key` are completely different variables. By convention, environment variable names are written in `SCREAMING_SNAKE_CASE` (all caps with underscores).

- **Size limits:** The environment block is a fixed-size memory region. On Linux, the combined size of all environment variables plus command-line arguments is typically limited to 2MB (the `ARG_MAX` limit), and practical per-variable limits are much lower. On Windows, the per-variable limit is 32,767 characters, and the total block has a practical limit around 256KB. These limits are generous for API keys, database URLs, and configuration flags, but you should not store file contents, large blobs, or binary data in environment variables.

- **Scope limits:** Environment variables set in a terminal session only reach programs launched from that same terminal. Graphical applications launched from the desktop (via the start menu, Finder, or desktop shortcuts) do not inherit terminal session variables. They do see system-level and user-level variables.

- **Persistence:** By default, environment variables are temporary. They vanish when the process that created them exits. Making them permanent requires explicit configuration (startup files on macOS/Linux, `setx` or system settings on Windows).

### Safety

- **Never commit a `.env` file to version control.** Add `.env` to your `.gitignore` before your first commit. If you accidentally commit it, rotate every secret it contained immediately.

- **Never hardcode secrets in source code.** A secret written in code stays in git history forever, even if you delete it in a later commit. Anyone who clones the repo can run `git log -p` and see the old commit with the secret in plain text.

- **Rotate exposed keys immediately.** If you suspect a secret was exposed -- even briefly -- regenerate it in the provider's dashboard. Do not assume it was safe. Bots scan GitHub, Pastebin, and public repositories constantly for exposed keys.

- **Use a linter to catch hardcoded secrets.** Tools like `truffleHog`, `git-secrets`, and `detect-secrets` scan your codebase for patterns that look like API keys. Run them in your CI/CD pipeline to catch mistakes before they reach production.

- **Be aware of shared machines.** On a shared or multi-user system, other users with sufficient privileges can read your process's environment variables through `/proc/[pid]/environ` (Linux) or via Process Explorer (Windows). Environment variables are safe from accidental commit, but they are not fully secure against a determined attacker on the same machine.

- **Use environment variables for configuration that changes between environments.** Your development, staging, and production deployments should each have their own set of environment variables. Never share production secrets with developers.

## When to Use

Use environment variables when:

- You need to store an API key, database password, or other secret that must not appear in source code
- You have different configuration values for development, testing, staging, and production
- You are deploying to a cloud platform -- Heroku, AWS, Vercel, Railway, Fly.io, and nearly every other platform injects secrets via environment variables
- You want to change behavior without modifying your code -- feature flags, log levels, timeouts, and toggles
- You need to configure an application in a containerized environment like Docker, where the container image is built once and configured at runtime via env vars
- You are working on a team and want to avoid the "works on my machine" problem -- each developer sets their own local values in a `.env` file that never gets committed

Do NOT use environment variables when:

- You have a large amount of configuration with many nested values -- use a structured config file (YAML, JSON, TOML) instead. Environment variables are flat key-value pairs with no nesting.
- The value is the same in every environment and is not secret -- just put it in your code. Constants like `APP_NAME=MyApp` that never change between dev and prod do not need to be env vars.
- You need to store structured or complex data like a list, dictionary, or configuration tree -- use a config file. Environment variables only support strings.
- You need to store binary data, large blobs, or file contents -- env vars have size limits and do not handle non-text data well.
- You need fine-grained access control or encryption at rest -- use a secrets manager (HashiCorp Vault, AWS Secrets Manager, etc.) instead of plain `.env` files.

## Questions You Might Have

### Why not just use a regular Python config file?

You could create a `config.py` file with all your settings and import it. The problem is that `config.py` gets committed to git, just like any other source file. Environment variables are not committed (when you use `.gitignore`). Also, cloud deployment platforms do not have your `config.py` -- they inject environment variables, so your code needs to read from `os.environ` to work in production.

### What if I have many environment variables?

Keep your `.env` file organized with comments and groups. Use separate `.env` files for different environments (`.env.development`, `.env.production`) and load the appropriate one. For large configurations, consider using a YAML or JSON config file instead, and store only the path to that file (or the secrets needed to decrypt it) in environment variables.

### Can I have spaces in environment variable values?

Yes. On macOS/Linux, wrap the value in quotes: `export GREETING="Hello World"`. In a `.env` file, values containing spaces should also be quoted. Python reads the value as-is, so `os.environ["GREETING"]` returns `"Hello World"` including the space.

### What happens if I forget to call load_dotenv()?

Your `os.environ.get("API_KEY")` call returns `None` (or raises `KeyError` if you used bracket access). The script runs without any secrets loaded. This is why checking for missing values is important -- a missing env var should produce a clear error message, not a mysterious failure later in the script.

### How do cloud platforms handle env vars?

Cloud platforms like Heroku, Vercel, and Railway provide a dashboard where you enter key-value pairs. When your application runs on their servers, those values are automatically injected into the environment. Your code reads them with `os.environ.get()` -- the same code works locally and in the cloud. The only difference is that locally you load from a `.env` file, and in production the platform sets the variables directly.

**Question 1:** You set `export DB_PASS=s3cret` in your terminal. Then you open a second terminal window and run `echo $DB_PASS`. What do you see, and why?

<details>
<summary>Click for answer</summary>

You see an empty line (nothing). The variable was set only in the first terminal's session. Each terminal process has its own independent environment block. Environment variables are never shared between processes unless explicitly inherited through the parent-child relationship.
</details>

**Question 2:** Your Python script has this code: `key = os.environ["API_KEY"]`. When you run it, Python raises a `KeyError: 'API_KEY'`. What went wrong?

<details>
<summary>Click for answer</summary>

The environment variable `API_KEY` was never set in the terminal session where you ran the Python script. The `os.environ` object only contains variables from the current process's environment block. You need to either set the variable first with `export API_KEY=...` (macOS/Linux) or `$env:API_KEY="..."` (PowerShell), or load it from a `.env` file using `load_dotenv()` before accessing it.
</details>

**Question 3:** You add `.env` to your `.gitignore` file. Then you commit all your changes. Will the `.env` file be pushed to GitHub?

<details>
<summary>Click for answer</summary>

No. If `.env` is listed in `.gitignore`, Git completely ignores the file. It will never appear in `git status`, never be staged with `git add`, and never be pushed to any remote repository. You can verify this by running `git check-ignore .env` -- if it prints `.env`, the ignore rule is working.
</details>

**Question 4:** What is the difference between `os.environ["KEY"]` and `os.environ.get("KEY", "fallback")` in Python?

<details>
<summary>Click for answer</summary>

`os.environ["KEY"]` raises a `KeyError` and crashes the program if the variable does not exist. `os.environ.get("KEY", "fallback")` returns the string `"fallback"` instead of raising an error -- it never crashes. The `.get()` method is safer when you are not certain the variable exists, or when you want to provide a sensible default value.
</details>

**Question 5:** You commit a file containing `password = "hunter2"` to git. Then you realize the mistake, delete the password, and commit again. Is your password safe?

<details>
<summary>Click for answer</summary>

No. Git stores every version of every file in its history. The password is still present in the git history even after you delete it in a later commit. Anyone who clones the repository can run `git log -p` to view every commit's diff, including the one that contained the password. To remove it, you would need to rewrite history using `git filter-branch` or a tool like `git-filter-repo`. Even then, assume the secret is compromised and rotate it immediately. This is why you add `.env` to `.gitignore` before the first commit -- before the secret ever touches git.
</details>
