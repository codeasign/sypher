You have a ticket to add a new API endpoint to an existing service. You have done this before. You know the pattern: a new route handler, a validation schema, a database query, a test. What should take an hour ends up taking three because you switch between your editor, the terminal, the browser, and the documentation page twenty times per step. You write the route handler, switch to the terminal to run the tests, they fail, switch back to the editor, find the typo, fix it, switch back to the terminal, run the tests again, the next test fails, repeat. Then you need to commit, push, open a PR, write a description, and tag a reviewer. By the time you are done, you have context-switched forty times and lost your train of thought at least four of those times.

There is a second, quieter pain. You have a task that is too big to hold in your head at once: refactor the entire authentication module, migrate a database schema, add a new feature across six files. You open the first file, make a change, open the second file, realize you need to change the first file differently, go back, lose your place, and the whole thing unravels. You end up with a half-finished refactor, a dirty working tree, and a sinking feeling that you have introduced a bug you will not find until the next deploy.

There is a third scenario that is even more common. You have a task that repeats every week — running a code quality report, updating dependencies, checking for security vulnerabilities. You have a script for it somewhere, but it is in a different directory, or it uses a different Python version, or the output format has changed. Every week you spend thirty minutes relearning the process and fixing the script. The task itself is simple, but the overhead of context-switching back to it makes it painful.

These are not problems of skill. They are problems of workflow. You have all the tools you need — a terminal, an editor, git, a test runner — but you are the one orchestrating every handoff between them. Each context switch costs you focus, time, and mental energy. What you need is a way to describe the whole task at once and let an assistant handle the orchestration while you review and approve the result.

## The Concept

Claude Code workflows are patterns for chaining Claude Code capabilities — slash commands, tools, permission gates, and session management — into repeatable processes that handle real development tasks from start to finish. Think of it as a foreman on a construction site: you describe the finished building, the foreman breaks it into jobs, assigns each job to the right crew, checks the work as it finishes, and comes back to you only when a decision is needed or the project is complete.

The key difference between a single Claude Code prompt and a workflow is structure. A single prompt is a one-shot request: "explain this function" or "find the bug in this file." A workflow is a multi-step process with a defined sequence: plan first, then implement, then review, then fix, then verify, then commit. Each step uses a different capability — `/plan` for design, editing tools for implementation, `/review` for quality checking, `/fix` for remediation, the verify skill for end-to-end testing, and git tools for version control.

Workflows are also repeatable. The same sequence of steps that works for adding one feature works for adding ten features. The same loop that monitors one deploy monitors every deploy. The same review process that checks one PR checks every PR. By codifying the sequence into a repeatable pattern, you eliminate the mental overhead of figuring out the process each time.

## CLI Reference

The table below covers every command, flag, and slash command that powers real Claude Code workflows. All syntax shown is current as of Claude Code v2.x.

| Command / Flag / Slash Command | What It Does | Example |
|---|---|---|
| `/plan` | Enter planning mode: Claude analyzes your request, proposes a structured plan with file-by-file changes, and waits for your approval before executing anything | `/plan add a health check endpoint to the API` |
| `/loop` | Run a task on a recurring interval. Claude executes the task, checks the result, and either loops or finishes based on the outcome | `/loop 5m "check the status of the deploy"` |
| `/review` | Review the current git diff for correctness bugs, security issues, and simplification opportunities. Specify effort level with `--effort` | `/review` or `/review --effort high` |
| `/fix` | Apply the findings from a review directly to the working tree. Can fix specific file paths or all findings | `/fix` or `/fix src/auth.py` |
| `claude --allowedTools` | Permit specific tools for the session without prompting, reducing permission friction during long workflows | `claude --allowedTools "Bash, Edit, Write, Read"` |
| `claude --model` | Switch between Claude models for different workflow needs. Opus for complex reasoning, Sonnet for speed | `claude --model opus` for planning, `claude` (default Sonnet) for execution |
| `claude -p "prompt"` | Run a one-shot prompt non-interactively. Useful for scripting Claude Code into automation pipelines | `claude -p "add error handling to all API routes"` |
| `claude execute "cmd"` | Run a shell command through Claude Code's context. Claude sees the output and can react to it | `claude execute "npm test -- --coverage"` |
| `/status` | Show current git status, active session info, pending approvals, and tool usage summary | `/status` |
| `/clear` | Clear the conversation history while keeping the session alive. Useful when context gets crowded during long workflows | `/clear` |
| `claude --resume` | Resume the most recent session. Workflows survive terminal restarts | `claude --resume` |
| `claude --verbose` | Show full tool-by-tool execution details. Essential for debugging complex workflows | `claude --verbose` |
| `claude --output-format json` | Output structured JSON for scripting and automated pipeline integration | `claude --output-format json -p "summarize the diff"` |
| `claude --config` | Open project-level settings to configure hooks, tool permissions, and model preferences | `claude --config` |

## How It Actually Works — Step by Step

Each step below reveals what happens at a different point in a real Claude Code workflow. Read them in order to build a complete mental model.

### Step 1: You Describe the Task

It starts with a natural language description of what you want. You can type this directly into the Claude Code terminal, pass it with `claude -p "..."`, or use a slash command like `/plan` to enter structured planning mode. The key difference from a simple prompt is that you are describing an outcome, not a sequence of commands.

Instead of saying "open auth.py, find the login function, change the token library import, then run the tests," you say "migrate the authentication module from opaque tokens to JWT." Claude handles the decomposition. This is the fundamental shift: you describe the what, Claude figures out the how.

The quality of the result depends heavily on the quality of the description. A vague description produces a vague plan. A precise description that includes the files involved, the desired outcome, and the constraints (test dependencies, style preferences, performance requirements) produces a plan that matches your intent on the first try.

Here are examples of good task descriptions versus vague ones:

| Vague Description | Precise Description |
|---|---|
| "Add error handling to the API" | "Add try/except blocks to all route handlers in src/routes/. Catch `ValueError` and `KeyError` and return 400 responses. Catch everything else as 500. Add tests for each error case." |
| "Refactor the auth module" | "Move the token validation logic from src/auth.py into a new module src/auth/validator.py. Keep the public API in auth.py by importing and re-exporting. Update all imports across the project. No behavior changes." |
| "Fix the slow query" | "Optimize the `get_user_orders` function in src/db/queries.py. It currently loads all orders into memory before filtering. Add a `LIMIT` clause and push the filter into the SQL query. Keep the existing test coverage." |

The more precise your description, the less time Claude spends guessing and the fewer iterations you need to get the right result.

### Step 2: Claude Reads the Codebase into Context

Once you describe the task, Claude reads the relevant parts of your codebase. It uses tools like `Grep` to search for patterns, `Read` to examine specific files, `Glob` to find files matching patterns, and `Bash` to run commands like `git log` or `git diff` to understand the current state.

The amount of context Claude can hold depends on the model. Claude Opus has a 200K token context window — enough to read the entire source of a medium-sized project. Claude Sonnet, the default, has a 100K token window. For most workflows, this is more than enough to hold the full set of files you need to change.

Claude does not read every file in the project. It reads strategically: the files you mention, the files that match search patterns, and the files that are imported or referenced by the files it already examined. This targeted reading keeps the context window focused on the relevant code.

The reading strategy matters for performance. If you ask Claude to "add a new endpoint to the orders API," it will start by finding the route files, the model files, and the test files related to orders. It will not read the entire project. It prioritizes breadth first — find all relevant files — then depth — read the most important ones fully. This means the first few seconds of a workflow are typically rapid searches followed by targeted reads.

### Step 3: Claude Proposes a Structured Plan

With the codebase context loaded, Claude formulates a plan. In `/plan` mode, this plan is explicit and structured: it lists every file that needs to change, describes the change for each file, and orders the changes in a dependency-aware sequence. Claude presents the plan to you and waits for approval before making any changes.

The plan includes:
- Which files to create, modify, or delete
- A summary of the change for each file
- The order of operations (files that depend on other files come later)
- Tests that should pass after the changes are applied
- Any commands that need to run (migrations, installs, builds)

You can approve the plan as-is, reject it and refine your description, or modify specific parts. This is the safety net: you never have to let Claude run unchecked. Every change is gated by your approval.

A real plan from `/plan` looks like this:

```text
Here is my plan for adding a health check endpoint:

1. Create src/routes/health.py
   - Add GET /health endpoint returning {"status": "ok", "timestamp": "..."}
   - Use the existing route pattern from src/routes/orders.py

2. Edit src/main.py
   - Import the health router
   - Register it at the "/api" prefix

3. Create tests/test_health.py
   - Test that GET /api/health returns 200
   - Test that the response contains "status" and "timestamp" fields

4. Run the test suite to confirm nothing is broken
```

The plan is specific enough that you can verify each step is correct before any code changes. If you notice something wrong — for example, the health endpoint should be at `/health` not `/api/health` — you can say "change the path to /health" before approving.

### Step 4: Claude Executes Tools and You Approve Each Action

Once the plan is approved, Claude starts executing. Each tool call — editing a file, writing a new file, running a command — requires your approval by default. The terminal shows you exactly what command Claude wants to run or what edit it wants to make, and you press Enter to approve, type a modification request, or reject.

This approval-per-action model is what makes Claude Code safe for real work. You see every change before it happens. You can catch mistakes, refine the approach, or stop the workflow entirely if it goes off track.

For workflows where you trust the pattern, you can reduce friction by pre-approving certain tools with `--allowedTools`. For example, if you are running a well-known refactoring pattern, you might pre-approve `Read, Edit, Grep, Glob` so Claude does not need to ask for permission to read files and make edits within the scope of the plan.

The approval system is not just a safety feature — it is also a learning tool. Each time you approve or reject an action, Claude learns your preferences. Over the course of a session, approvals become faster because Claude's proposals align more closely with your style.

There are three approval modes you will encounter:

**Per-action approval (default):** Every tool call pauses and waits for your input. This is the safest mode for unfamiliar tasks or when you are still learning the tool.

**Pre-approved tools with `--allowedTools`:** You specify which tools Claude can use without asking. Within those tools, Claude acts freely. Tools not in the list still require approval. This is the best balance for routine tasks.

**Full session approval via `claude --allowedTools "..."` at launch:** By setting permissions at session start, you avoid interruptions entirely for the approved tool set. Use this only when you are confident in the workflow pattern and have verified it on a smaller scale first.

### Step 5: Claude Verifies the Results

After executing the planned changes, Claude verifies the result. This is where the `/review` and `verify` skills come in. Claude runs the test suite, checks the diff for correctness, and reports whether the changes achieve the goal.

The verification step is critical because it catches errors that are invisible in the diff. A change might look correct in the editor but break a test you did not know existed. Or it might work in isolation but conflict with another part of the system. Claude runs the actual verification tools — tests, linters, type checkers, build steps — to confirm the change is sound.

The `verify` skill goes a step further: it exercises the changed code end-to-end by driving the actual flow. Instead of just checking that tests pass, it runs the application or script and observes the behavior. This catches integration issues that unit tests miss.

Here is what verification looks like in practice. After Claude edits the files, it runs:

```bash
python -m pytest tests/ -v --tb=short
```

If tests pass, it reports the results and moves on. If tests fail, it reads the error output, determines the root cause, and proposes a fix. This is the loop that makes the workflow self-correcting.

The `verify` skill adds a second layer. For the health endpoint example, after running tests, Claude would also:

```bash
python -c "
from src.routes.health import router
# Verify the route exists and has the right method
print('Health route registered successfully')
"
```

This second layer catches issues that tests do not cover — for example, a route that is registered correctly in the test fixture but not in the production app.

### Step 6: The Loop Decision — Finish or Iterate

The final step is the loop decision. Claude checks the verification results:

- If all checks pass, the workflow finishes. Claude can optionally commit the changes, create a branch, open a PR, or run any other completion steps you defined in the plan.
- If checks fail, Claude analyzes the errors, determines the root cause, and proposes a fix. With `/loop`, this iteration happens automatically on a timer. Claude runs the task, checks the result, fixes any issues, and runs again until the task succeeds or hits a limit.

The `/loop` command is the most powerful workflow tool for long-running or recurring tasks. You can set it to run every 5 minutes, every hour, or on a cron schedule. Each iteration, Claude executes the same prompt, checks the result, and decides whether to continue or stop.

### Step 7: Git Integration Finishes the Cycle

The final step of most workflows is git integration. Claude can create branches, commit changes, write commit messages, push to remotes, and open pull requests. This is where the workflow produces a shareable artifact — a commit, a PR, a release — that other team members can review and build on.

The git workflow is fully integrated into the approval system. Claude shows you the diff before committing, writes a commit message based on the changes, and asks for approval before running `git commit` or `git push`. You can customize the commit message, add co-authors, or request changes before the commit happens.

For PR workflows, Claude can open a PR on GitHub using the `gh` CLI. It writes the PR description based on the plan and the changes, adds reviewers, and links related issues. This turns a multi-step chore into a single approval.

Here is a typical git workflow sequence inside Claude Code:

```
You: Commit the changes and create a PR.
Claude: Shows the diff for your approval.
       Commit message: "Add health check endpoint with tests"
You: Approve.
Claude: git add . && git commit -m "Add health check endpoint with tests"
       git push origin feature/health-check
       gh pr create --title "Add health check endpoint" --body "..." --base main
You: PR URL: https://github.com/your-org/your-repo/pull/42
```

### Step 8: Managing Long-Running Tasks

Some workflows take longer than a single session. A batch refactor of 50 files, a data migration, or a large test suite can run for hours. Claude Code handles this through session management.

The `claude --resume` command restores your most recent session, including the conversation history and context. This means you can start a workflow, close the terminal, come back later, and pick up where you left off. The workflow state is preserved.

For background tasks, Claude Code can also run tasks in a detached mode using `run_in_background` on the Bash tool. This is useful for long-running operations like test suites or builds that you do not need to watch in real time. Claude checks the result when the task completes and reports back.

The key limitation is that Claude Code is a terminal tool — it does not run as a daemon. If you close the terminal, active background tasks stop. For truly persistent automation, combine Claude Code with your operating system's task scheduler (cron on Linux/macOS, Task Scheduler on Windows) or a CI/CD pipeline.

## Workflow Patterns for Common Tasks

### The Code Review Workflow

Every change should be reviewed before it reaches production. The Claude Code code review workflow makes this fast and thorough:

1. Make changes in your editor as usual.
2. Launch Claude Code in the project directory.
3. Run `/review --effort high` to get a thorough analysis of the diff.
4. Review the findings. Each finding includes a severity level, file location, and explanation.
5. Run `/fix` to apply the fixes you agree with.
6. Run `/review` again to confirm the fixes resolved the issues.
7. Commit and push.

This workflow catches issues that human reviewers miss — especially subtle bugs, inconsistent naming, missing edge cases, and security vulnerabilities. It also runs in seconds, so you can use it before every commit, not just before every PR.

### The Refactor Workflow

Refactoring is where Claude Code's workflow capabilities shine brightest. A refactor typically involves changes across many files, and the mechanical consistency of the changes is critical.

1. Use `/plan` to describe the refactor. Be specific about the before and after state.
2. Review the plan. Verify that every file that needs to change is listed.
3. Approve execution. Claude makes the changes file by file.
4. Run `/review` on the full diff to catch any inconsistencies.
5. Run the test suite to confirm behavior is preserved.
6. Commit with a descriptive message that explains what was refactored and why.

The refactor workflow is especially effective for well-known patterns: renaming symbols, extracting modules, changing import paths, adding type annotations, converting between data formats. These are tasks where the mechanical work is large but the judgment required is small.

### The Batch Processing Workflow

When you need to apply the same change to many files — adding a copyright header, updating a dependency version, fixing a recurring lint error — the batch processing workflow handles it efficiently:

1. Describe the change pattern: "Add `from __future__ import annotations` to every Python file that does not already have it."
2. Claude uses `Glob` to find all matching files and `Grep` to check each one.
3. Claude edits each file, showing the diff for approval.
4. For large batches, use `--allowedTools "Read, Edit, Grep, Glob"` to reduce approval friction.

The batch workflow is most efficient when the change is truly mechanical — the same edit applied to every matching file. If the edit needs different logic per file, the batch approach becomes slower because each file requires individual analysis.

### The Content Generation Workflow

For projects built on content — documentation, learning materials, blog posts, tutorials — Claude Code can generate entire sections using custom slash commands like `/add-topic-concept` or `/add-topic-git-github-actions`. These slash commands are defined in the project's `.claude/commands/` directory and encode the project's content conventions, including template structure, diagram requirements, and build checks.

This workflow is specific to knowledge-base or learning-site projects where content follows a template pattern. For example, on this learning site, the `/add-topic-concept` command knows to create pages with AsciiDiagram components, follow the sidebar JSON conventions, and include practice exercises. It also knows where to place the new content in the directory structure and how to update the sidebar.

The full content generation workflow looks like:

1. Launch Claude Code in the project directory.
2. Use a custom slash command like `/add-topic-concepts` or the project-specific skill.
3. Claude reads the existing content structure to understand patterns and conventions.
4. Claude generates the new content following the established template format.
5. Review the generated content and request edits where needed.
6. Run the build step (e.g., `npm run build`) to verify the content renders correctly.
7. Commit the new content.

This workflow is particularly powerful because it enforces consistency. Every new page follows the same template, uses the same diagram conventions, and passes the same build checks. The custom slash commands encode the project's content standards, so every contributor produces content that matches the existing style.

### The Onboarding Workflow

When you join a new project, Claude Code can accelerate the onboarding process significantly. Instead of spending days reading documentation and exploring the codebase manually, you can use a structured onboarding workflow:

1. Launch Claude Code in the project directory.
2. Ask Claude to explain the project structure: "Read the project README, package.json, and directory tree. Give me a high-level overview of this project."
3. Ask Claude to explain the architecture: "Read the main entry point and the router configuration. Explain how requests flow through the system."
4. Ask Claude to explain the testing strategy: "Find the test directory and explain the testing patterns used here."
5. Ask Claude to explain the deployment process: "Find the CI/CD configuration files and explain how this project is deployed."
6. Ask Claude to create a local onboarding document: "Create ONBOARDING.md with a summary of everything you have learned about this project."

Each question uses Claude's ability to read and synthesize information from across the codebase. The result is a personalized onboarding document that covers exactly what you need to know about this specific project.

### The Monitoring Workflow

The `/loop` command turns Claude Code into a monitoring tool for long-running processes:

1. Start a deploy, build, or test run.
2. Run `/loop 30s "Check the status of the deploy by running 'kubectl get pods' and report any changes."`
3. Claude polls every 30 seconds and reports when the status changes.
4. The loop stops when the deploy completes or hits the max iteration limit.

This pattern is useful for any task where you need to wait for an asynchronous result: CI/CD pipelines, database migrations, model training, file syncs, and service restarts.

## Cost, Limits, and Safety

### Session Costs

Every Claude Code session consumes tokens on both input and output. The cost of a workflow depends on its complexity:

- A simple code review (`/review`) typically costs $0.10-$0.30 in tokens, depending on the size of the diff.
- A multi-file refactor with 5-10 files might cost $0.50-$2.00, including the plan, execution, and verification steps.
- A `/loop` task running every 5 minutes for an hour costs roughly $1.00-$5.00, depending on the complexity of each iteration.
- A batch processing task touching 50 files might cost $2.00-$8.00, depending on the size of each file and the complexity of the edit.

### Managing Long-Running Sessions

For workflows that span hours or multiple sessions, use these strategies to manage costs:

- **Use `/clear` to reset context**: The conversation history grows with each interaction. A long session with hundreds of messages can become slow and expensive because every new message includes the full history. Run `/clear` periodically to reset the context while keeping the session alive. Claude will re-read the necessary files, but the conversation history is trimmed.

- **Use `claude --resume` to continue across terminal restarts**: If you need to close your terminal, the session is saved. Resume with `claude --resume` to pick up where you left off. This is useful for workflows that are interrupted by a reboot, a commute, or a meeting.

- **Break large tasks into smaller sessions**: A workflow that touches 100 files is better handled as 10 separate sessions of 10 files each. Each session has a focused context, lower cost, and faster execution. The session boundary is also a natural checkpoint — commit after each session, so you can roll back if needed.

- **Use `--model` to choose the right model for each phase**: Use Claude Opus (more expensive, more capable) for the planning and review phases where reasoning quality matters most. Use Claude Sonnet (faster, cheaper) for the execution phase where the plan is already defined and the work is mechanical. This tiered model approach reduces costs by 40-60 percent compared to using Opus for the entire workflow.

### Runaway Loop Safety

The most important safety precaution is preventing runaway loops. The `/loop` command is powerful, but without guardrails it can consume tokens indefinitely. Follow these rules:

- Always set a maximum iteration count: `/loop --max-iterations 5 "check deploy status"`
- Always define a clear success condition: the loop should stop when a condition is met, not run forever
- Set a timeout for long-running tasks: Claude Code has a default timeout per tool call, but `/loop` resets this each iteration
- Use `--allowedTools` to limit what tools the loop can use — if a loop only needs `Bash` and `Read`, do not permit `Edit` and `Write`

### Permission System Safety

The permission system is your primary safety control. Each tool type requires separate approval:

- `Read` — Read files and directories
- `Edit` — Modify existing files
- `Write` — Create new files
- `Bash` — Run shell commands
- `Glob` — Search for files by pattern
- `Grep` — Search file contents
- `WebFetch` — Fetch URLs
- `WebSearch` — Search the web

By default, every tool invocation requires approval. You can pre-approve tools for a session with `--allowedTools`, but you should only pre-approve tools that the workflow actually needs. A code review workflow needs `Read, Grep, Glob` but not `Edit, Write, Bash`. A refactoring workflow needs `Read, Edit, Write, Grep, Glob` but might not need `WebFetch`.

The permission system is per-session. Each time you launch Claude Code, tool permissions reset to the default (all tools require approval). This means you cannot accidentally carry over permissions from a previous session to a different task.

## When to Use This / When Not To

### Use Claude Code workflows when:

- You have a multi-step task that touches multiple files. A single `/plan` session replaces dozens of context switches between your editor, terminal, and documentation.
- You need to repeat the same task periodically. The `/loop` command turns a manual check into an automated monitor.
- You want a code review on every change before committing. The `/review` command runs in seconds and catches issues you would miss on a manual read-through.
- You are refactoring a well-understood pattern across many files. Claude handles the mechanical parts while you verify the correctness.
- You need to generate git commits, PRs, and branch management as part of the workflow. Claude handles the full git lifecycle.
- You are onboarding to a new codebase and need to understand the structure. Claude can read the entire project and explain the architecture.
- You have a batch change that applies the same transformation to many files. The batch processing workflow handles this efficiently.
- You need to monitor a long-running process (deploy, migration, test suite) and be notified when it completes. The `/loop` command handles this.

### Do not use Claude Code workflows when:

- The task is a single, trivial edit. Opening a file and changing one line is faster in your editor than launching a Claude Code session.
- You need to think deeply about architecture without writing code. The `/plan` mode helps with structure, but it is not a replacement for whiteboard design sessions.
- The codebase is enormous and the task requires understanding the entire system. Claude's context window is large but finite. Very large refactors may need to be broken into smaller chunks.
- You are working with highly sensitive data that cannot leave your machine. Claude Code makes API calls to Anthropic's servers. If your data cannot be sent to a third party, you must use a local alternative.
- You need pixel-perfect UI changes. Claude Code is a terminal tool — it cannot see rendered output. For visual changes, use a complementary tool in your editor.
- The task requires real-time collaboration with other developers. Claude Code is a single-user tool. For pair programming sessions, use your editor's collaboration features.
- You need to run the workflow on a server without internet access. Claude Code requires connectivity to Anthropic's API.

### Combining Workflows

The most powerful pattern is combining multiple workflows in sequence. A single development session might use three or four different workflows:

1. **Plan workflow**: Use `/plan` to design the approach for a new feature.
2. **Implementation workflow**: Execute the planned changes with approval gates.
3. **Review workflow**: Run `/review` on the complete diff.
4. **Fix workflow**: Apply the review findings with `/fix`.
5. **Git workflow**: Commit, push, and create a PR.

Each workflow feeds into the next. The plan determines what gets implemented. The implementation produces the diff that gets reviewed. The review produces the findings that get fixed. The fixed code gets committed. This chain of workflows turns a complex development task into a structured, repeatable process.

You can also combine workflows with custom slash commands. For example, a team might define a `/new-feature` command that runs the entire chain: plan, implement, review, fix, commit. This turns a 30-minute multi-step process into a single command that produces production-ready code.

### The Hybrid Workflow Pattern

The most effective approach is hybrid: use Claude Code for the orchestration and mechanical parts of a task, and use your editor for the parts that require deep focus. A typical hybrid workflow looks like:

1. Use `/plan` to design the approach and get a structured plan.
2. Let Claude execute the mechanical changes — imports, boilerplate, test scaffolding.
3. Switch to your editor for the critical logic — the parts where human judgment matters most.
4. Use `/review` on the final diff before committing.

This hybrid pattern gives you the speed of automation where it matters and the control of manual editing where it matters.

## Mental Model Check

These five questions will confirm whether you understand how to build and run Claude Code workflows. If you can answer all five, you are ready to try the practice exercise.

### Question 1
What is the difference between typing a task description directly into Claude Code and using `/plan` mode?

**Answer:** Typing a task description directly lets Claude start executing immediately. `/plan` mode forces Claude to first propose a structured plan listing every file change, the order of operations, and verification steps before executing anything. The plan must be approved before any changes are made. `/plan` is safer for multi-step tasks because it gives you a complete picture of what will happen before any changes are applied.

### Question 2
What does the `/loop` command do, and what safety guardrails should you set when using it?

**Answer:** `/loop` runs a task on a recurring interval. Claude executes the task, checks the result, and either continues or finishes based on the outcome. Safety guardrails include: a maximum iteration count (`--max-iterations`), a clear success condition, a timeout for the overall loop, and limited tool permissions (`--allowedTools`) so the loop can only use the tools it actually needs.

### Question 3
Why does every tool invocation require approval by default, and how can you reduce friction for trusted workflows?

**Answer:** Every tool invocation requires approval by default so you can see every change before it happens and catch mistakes, refine approaches, or stop the workflow entirely if it goes off track. To reduce friction for trusted workflows, use `--allowedTools` to pre-approve specific tool types for the session. For example, a read-only review workflow can pre-approve `Read, Grep, Glob` so Claude does not prompt for every search and file read.

### Question 4
What is the purpose of the verification step in a workflow, and what does the `verify` skill add beyond running tests?

**Answer:** The verification step runs tests, linters, type checkers, and build steps to confirm the changes are correct. The `verify` skill goes further by exercising the changed code end-to-end — it drives the actual application or script and observes the behavior. This catches integration issues that unit tests miss, such as incorrect runtime behavior that passes structural checks.

### Question 5
What is the hybrid workflow pattern, and when would you use it instead of a fully automated workflow?

**Answer:** The hybrid workflow pattern uses Claude Code for orchestration and mechanical changes (planning, imports, boilerplate, test scaffolding, git operations) and your editor for critical logic that requires human judgment. Use it when the task has both mechanical parts (where automation is fast and reliable) and high-judgment parts (where you want to think carefully about the implementation). The hybrid pattern gives you the speed of automation where it matters and the control of manual editing where it matters.
