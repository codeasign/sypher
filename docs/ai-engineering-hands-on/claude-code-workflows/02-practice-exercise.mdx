You will now build a real project using Claude Code workflows from start to finish. By the end of this exercise, you will have planned a feature, implemented it with Claude, reviewed the diff, fixed issues, and committed the result. Every step is laid out below.

## Setup

### Prerequisites

- **Claude Code installed**. Open a terminal and run `claude --version` to check. If it is not installed, follow the installation guide at docs.anthropic.com.
- **Node.js 18 or later** (if using JavaScript/TypeScript) or **Python 3.10 or later** (if using Python). Run `node --version` or `python --version` to check.
- **Git**. Run `git --version` to check. You need it for the git workflow parts of this exercise.
- **An Anthropic API key**. Set it as an environment variable: `export ANTHROPIC_API_KEY=sk-ant-...`

### Create the Project

Create a new directory and initialize it as a git repository:

```bash
mkdir workflow-practice
cd workflow-practice
git init
```

Create a `.gitignore` file:

```bash
echo "node_modules/" > .gitignore
echo ".env" >> .gitignore
```

Create a simple project that Claude Code will work with. If you are using Python, create `calculator.py` with this starter code:

```python
"""A simple calculator module for practicing Claude Code workflows."""

import os


def add(a: float, b: float) -> float:
    """Add two numbers together."""
    return a + b


def subtract(a: float, b: float) -> float:
    """Subtract b from a."""
    return a - b


def multiply(a: float, b: float) -> float:
    """Multiply two numbers."""
    return a * b
```

Create a corresponding test file `test_calculator.py`:

```python
"""Tests for the calculator module."""

from calculator import add, subtract, multiply


class TestCalculator:
    """Test suite for calculator operations."""

    def test_add(self):
        assert add(2, 3) == 5
        assert add(-1, 1) == 0
        assert add(0, 0) == 0

    def test_subtract(self):
        assert subtract(5, 3) == 2
        assert subtract(0, 5) == -5
        assert subtract(10, 10) == 0

    def test_multiply(self):
        assert multiply(2, 3) == 6
        assert multiply(0, 5) == 0
        assert multiply(-2, 3) == -6
```

Commit this starter code:

```bash
git add .
git commit -m "Initial calculator module with add, subtract, multiply"
```

### Verify Your Setup

Run the tests to confirm everything works before you bring in Claude Code:

```bash
python -m pytest test_calculator.py -v
```

Expected output:

```text
collected 3 items

test_calculator.py::TestCalculator::test_add PASSED
test_calculator.py::TestCalculator::test_subtract PASSED
test_calculator.py::TestCalculator::test_multiply PASSED
```

## The Build — Step by Step

### Step 1: Plan a New Feature with `/plan`

You need to add a `divide` function to the calculator. Instead of writing the code yourself, you will use Claude Code's `/plan` mode to design the change first.

Launch Claude Code in your project directory:

```bash
claude
```

Then type this command:

```
/plan Add a divide function to the calculator module. It should handle division by zero by raising a ValueError with a clear message. Add corresponding tests.
```

Claude will analyze the codebase, read `calculator.py` and `test_calculator.py`, and propose a structured plan. The plan will list exactly which files to change, what to add in each file, and the order of operations.

**Expected output** (varies slightly by version):

```text
I've analyzed your calculator project. Here is my plan:

1. Edit calculator.py — add `divide(a, b)` function with division-by-zero check
2. Edit test_calculator.py — add `test_divide` method with normal and edge cases
3. Run tests to verify everything passes

Shall I proceed with this plan? (Y/n)
```

Do not approve yet. Read the plan carefully. If it looks correct, type `Y` and press Enter. If you want to modify it, describe your changes.

### Step 2: Approve the Plan and Watch Execution

When you approve, Claude will execute each step. You will see tool invocations as they happen:

1. Claude reads `calculator.py` — the `Read` tool retrieves the current file content.
2. Claude reads `test_calculator.py` — same tool, second file.
3. Claude proposes an edit to `calculator.py` adding the `divide` function:

```python
def divide(a: float, b: float) -> float:
    """Divide a by b. Raises ValueError if b is zero."""
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b
```

4. Claude proposes an edit to `test_calculator.py` adding the test:

```python
def test_divide(self):
    assert divide(10, 2) == 5
    assert divide(3, 4) == 0.75
    assert divide(0, 5) == 0
    import pytest
    with pytest.raises(ValueError, match="Cannot divide by zero"):
        divide(5, 0)
```

Each edit requires your approval. Press Enter to approve each one.

After all edits are applied, Claude runs the tests:

```bash
python -m pytest test_calculator.py -v
```

Expected output:

```text
collected 4 items

test_calculator.py::TestCalculator::test_add PASSED
test_calculator.py::TestCalculator::test_subtract PASSED
test_calculator.py::TestCalculator::test_multiply PASSED
test_calculator.py::TestCalculator::test_divide PASSED
```

**Explanation:** The `/plan` command forced Claude to slow down and think about the task before making any changes. Without `/plan`, Claude might have started editing immediately and missed the need for division-by-zero handling or the import of `pytest` in the test file. The plan step catches these gaps before code changes are made.

### Step 3: Review the Diff with `/review`

Now that the feature is implemented, review the changes before committing. Still inside the Claude Code session, type:

```
/review
```

Claude will run its code review on the diff between the current state and the last commit. It checks for correctness bugs, simplification opportunities, and quality issues.

**Expected output** (varies slightly):

```text
Review of staged and unstaged changes:

calculator.py:
  - No issues found. The divide function correctly handles the edge case.

test_calculator.py:
  - Suggestion: The `import pytest` is inside the test method. Consider moving it to the top of the file.
  - The test covers normal cases (10/2=5, 3/4=0.75) and edge cases (0/5=0, division by zero).
```

The review catches the inline import issue. In Python, imports at the top of the file are the convention. A linting tool would also flag this, but the review catches it before you commit.

### Step 4: Fix the Issue with `/fix`

To apply the review finding automatically, type:

```
/fix
```

Claude will move the `import pytest` statement from inside the test method to the top of `test_calculator.py`. It shows you the edit before applying it:

```text
Proposed fix:
  File: test_calculator.py
  Remove "import pytest" from inside test_divide
  Add "import pytest" at the top of the file
```

Press Enter to approve.

After the fix, Claude re-runs the tests to confirm everything still passes:

```bash
python -m pytest test_calculator.py -v
```

All four tests should pass.

**Explanation:** The `/review` and `/fix` pair forms a complete quality loop. `/review` finds issues without changing anything. `/fix` applies only the findings you approve. This is safer than having Claude make changes directly, because you see the review report first and can decide which findings to act on.

### Step 5: Verify End-to-End Behavior with the Verify Skill

The tests pass, but do they test the right thing? The verify skill goes a step further by exercising the actual code and checking its behavior.

Type this command:

```
Add a file called verify_divide.py that imports and calls divide() with several inputs, printing results. Then run it and confirm the output.
```

Claude will create `verify_divide.py`:

```python
"""Verify the divide function works correctly."""

from calculator import divide

# Normal cases
result = divide(10, 2)
print(f"divide(10, 2) = {result}")

result = divide(3, 4)
print(f"divide(3, 4) = {result}")

# Edge cases
result = divide(0, 5)
print(f"divide(0, 5) = {result}")

# Error case
try:
    divide(5, 0)
except ValueError as e:
    print(f"divide(5, 0) raised ValueError: {e}")
```

Claude then runs it:

```bash
python verify_divide.py
```

Expected output:

```text
divide(10, 2) = 5.0
divide(3, 4) = 0.75
divide(0, 5) = 0.0
divide(5, 0) raised ValueError: Cannot divide by zero
```

**Explanation:** Tests can have bugs too. The verify step confirms that the code actually works when executed, not just that it passes structural tests. A test that passes but tests the wrong thing is worse than no test at all, because it gives false confidence.

### Step 6: Use `/loop` for a Repeated Task

The `/loop` command is useful for tasks that need repeated checking. For this exercise, imagine you want to verify that the calculator module stays correct after any future changes.

Type this command:

```
/loop 30s "Run python -m pytest test_calculator.py -v and report if any tests fail. Stop immediately if all tests pass."
```

Claude will run the tests every 30 seconds. Since all tests already pass, the loop will report success on the first iteration and stop.

If you had a test that was flaky or a feature that needed monitoring, `/loop` would keep checking on the interval you set and only report when something changed.

To stop a `/loop` manually, press Ctrl+C.

**Explanation:** The `/loop` command is not just for testing. You can use it to poll a deploy status, wait for a service to start, monitor a log file for errors, or check that a build artifact is ready. The iteration interval and the success condition define when the loop ends.

### Step 7: Commit and Create a PR

The final step is committing the changes through Claude Code. Type:

```
Commit the changes with a descriptive message. Include the divide function, tests, and verify script. Then show me the commit log.
```

Claude will show you the full diff, write a commit message:

```text
docs: Add divide function with division-by-zero handling

- Add divide() function to calculator module
- Add test_divide covering normal and edge cases
- Add verify_divide.py for end-to-end verification
- Fix inline import in test_calculator.py
```

Approve the commit. Then:

```
Show me the git log for this file.
```

Claude runs:

```bash
git log --oneline -- calculator.py
```

**Explanation:** Claude handles the entire git workflow: staging files, writing the commit message based on the changes, and committing. For a real project, you could extend this to create a branch, push, and open a PR with a single command. The git integration eliminates the context switch of leaving your terminal to use a separate git UI.

## Common Errors at This Stage

### Error 1: `/plan` Produces a Vague or Wrong Plan

```text
I'll modify the files to add a divide function.
```

**Cause:** You described the task too vaguely. Claude did not have enough detail about division-by-zero handling, test coverage, or the function signature.

**Fix:** Be specific in your `/plan` command. Include the function signature, error handling requirements, and test expectations. For example: `/plan Add a divide(a: float, b: float) -> float function that raises ValueError on division by zero. Add three test cases: normal division, zero numerator, and division by zero.`

### Error 2: `/review` Finds No Issues When You Know There Are Some

```text
No issues found in the diff.
```

**Cause:** The review is running at the default effort level (`low` or `medium`), which only catches obvious issues. The inline `import pytest` might not be flagged at low effort.

**Fix:** Increase the review effort level: `/review --effort high` or `/review --effort max`. Higher effort levels catch more subtle issues like import placement, variable naming, and edge cases in test coverage.

### Error 3: Permission Denied When Claude Tries to Run a Command

```text
I need to run: python -m pytest test_calculator.py
Permission denied: the shell command was not approved.
```

**Cause:** You did not approve the `Bash` tool call. Claude cannot run commands without your approval by default.

**Fix:** Press Enter to approve the command. If you plan to run many commands in a session, pre-approve `Bash` at startup: `claude --allowedTools "Bash, Read, Edit, Write, Grep, Glob"`.

### Error 4: `/loop` Runs Forever Without Stopping

```text
/loop 10s "Check the status"
# ... 10 seconds later ...
# ... 10 seconds later ...
# ... 10 seconds later ...
```

**Cause:** The loop task does not have a clear success condition. Claude cannot determine when the goal is met, so it keeps looping.

**Fix:** Always include a success condition in the prompt: `/loop 10s "Run tests and stop when all pass"` or use `--max-iterations`: `/loop --max-iterations 6 10s "Check deploy status"`. A maximum iteration count is a hard safety net that stops the loop even if the task never succeeds.

### Error 5: Git Commit Fails Due to Unstaged Files

```text
Error: Your local changes to the following files would be overwritten:
  calculator.py
Please commit your changes or stash them before you switch branches.
```

**Cause:** You tried to commit but there are untracked or unstaged files that git does not know about.

**Fix:** Claude should handle this automatically by running `git add` before `git commit`. If it does not, add the files explicitly: `git add calculator.py test_calculator.py verify_divide.py` and then ask Claude to commit again.

## What You Just Built

- **A complete Claude Code workflow** that went from planning to implementation to review to fix to verification to git commit — all without leaving the terminal. Each stage used a different Claude Code capability: `/plan` for structured design, `/review` for quality checking, `/fix` for applying fixes, the verify skill for end-to-end testing, `/loop` for automated repetition, and git integration for version control.

- **A repeatable process** for adding features. The same pattern works for any size change: plan, implement, review, fix, verify, commit. Large features are just many iterations of this loop.

- **A demonstration of the approval safety net**. Every edit, every command, every commit required your approval. At no point did Claude make changes without your oversight. This is the core safety property of Claude Code workflows.

- **A verified `divide` function** that was tested both structurally (unit tests) and behaviorally (verify script). Two levels of verification reduces the chance of undetected bugs.

- **A clean git history** with a descriptive commit message. The workflow produced a shareable artifact, not a dirty working tree.

- **An understanding of when to use each tool**: `/plan` for multi-step tasks, `/review` for quality gates, `/fix` for applying review findings, `/loop` for recurring checks, and direct prompts for ad-hoc work.
