In this exercise, you install the official filesystem MCP server from the `@modelcontextprotocol` package, connect it to Claude Code, and ask Claude to read a file from your project. By the end, you will have a working MCP server in your Claude Code session — the same kind of server any LLM application can discover and call through the Model Context Protocol.

## Prerequisites

You will need:

- **Node.js 18 or higher** — the filesystem server is distributed as an npm package and launched with `npx` (`node --version` to check)
- **Claude Code** installed and authenticated (`claude --version`)
- **A project directory** with at least one text file you do not mind Claude reading — a `README.md`, a `notes.txt`, or just the project you are already working in

If you do not have Node.js installed, download it from `https://nodejs.org` (the LTS version is fine) or install it through your package manager:

```bash
# macOS with Homebrew
brew install node

# Ubuntu / Debian
sudo apt install nodejs npm

# Windows (use the official installer from nodejs.org, or use winget)
winget install OpenJS.NodeJS.LTS
```

Confirm it is ready:

```bash
node --version
```

```text
v18.17.1
```

The exact version matters less than that it printed a number starting with `18` or higher. If it printed a `command not found` error instead, the installer did not add Node.js to your PATH — restart your terminal and try again.

## The Build — Step by Step

### Step 1: Create a test file Claude will read

Choose or create a text file in your current project that Claude has never seen before — this way you can confirm the read actually happens through the MCP server and not from a file Claude already knows about from earlier context.

Create a simple test file:

```bash
# On macOS / Linux
echo "Hello from the filesystem MCP server. If you can read this, MCP is working." > mcp-test.txt

# On Windows PowerShell
"Hello from the filesystem MCP server. If you can read this, MCP is working." | Out-File -Encoding utf8 mcp-test.txt
```

**Checkpoint:** confirm the file exists with the right content:

```bash
cat mcp-test.txt
```

```text
Hello from the filesystem MCP server. If you can read this, MCP is working.
```

Keep this file in your current working directory. You will ask Claude to read it in a later step.

### Step 2: Verify npx can fetch the server package

The filesystem MCP server is published on npm as `@modelcontextprotocol/server-filesystem`. Before registering it with Claude Code, confirm your system can download and run it by launching it briefly without Claude:

```bash
npx -y @modelcontextprotocol/server-filesystem --help
```

```text
Usage: server-filesystem [options] <directories...>

Options:
  -V, --version                  output the version number
  -h, --help                     output usage information

Connect the filesystem MCP server to direct read/write access to a set of directories on the local filesystem.
```

The `-y` flag tells `npx` to download the package automatically without prompting you for confirmation. The server printed its help text and exited — it is available and ready to use.

If you see an error like `npm ERR! 404 Not Found` or `npx: command not found`, check that Node.js and npm are correctly installed with `npm --version`. A version number of `8` or higher means npm is working.

**Checkpoint:** run the server one more time, this time with a concrete directory path, and confirm it starts without immediately crashing:

```bash
npx -y @modelcontextprotocol/server-filesystem .
```

```text
(no output — the server started and is waiting on stdin for JSON-RPC messages)
```

That silence is correct. A stdio MCP server does not print a banner or a startup message. It waits for a client to connect over its standard input and output. Press `Ctrl+C` (macOS/Linux) or `Ctrl+C` (Windows) to stop it.

### Step 3: Register the server with Claude Code

Now that the server works independently, register it with Claude Code so Claude can discover and call its tools during a conversation.

Run this command from your project directory, replacing `/absolute/path/to/your/project` with the actual absolute path:

```bash
pwd
```

```text
/home/you/projects/learn-mcp
```

Take that path and use it in the registration command:

```bash
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp
```

```text
Added stdio MCP server "filesystem" to local config
```

The server name `filesystem` is arbitrary — you can name it anything you like. The important part is everything after the `--` separator: that is the exact command Claude Code will run every time it needs to launch this server.

**Checkpoint:** verify the server registered and connected:

```bash
claude mcp list
```

```text
filesystem: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp - ✓ Connected
```

The green checkmark means the server launched, completed the MCP handshake, and reported its capabilities back to Claude Code. If you see `✗ Failed to connect` instead, proceed to the "Common Errors" section before continuing.

Look at the detailed server information to see what tools it exposes:

```bash
claude mcp get filesystem
```

```text
filesystem
  Status: ✓ Connected
  Command: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp
  Tools:
    read_file (Read the complete contents of a file from the file system)
    read_multiple_files (Read the contents of multiple files simultaneously)
    write_file (Write content to a file)
    create_directory (Create a new directory)
    list_directory (Get a listing of all files in a directory)
    move_file (Move or rename a file or directory)
    search_files (Recursively search for files and directories matching a pattern)
    get_file_info (Get file information and metadata)
    list_allowed_directories (Returns the list of directories that this server is allowed to work with)
```

Nine tools, each with a description the model will use to decide when to call which one. You have not written a single line of server code — you just installed a package and registered it.

### Step 4: Understand the scope restriction

Notice that you passed your project directory as an argument when registering the server: `npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp`. That path is the **allowed directory** — the server will refuse any operation outside of it. List the allowed directories to confirm:

```bash
claude mcp get filesystem
```

Look for this line in the output:

```text
  Allowed directories:
    /home/you/projects/learn-mcp
```

If you had passed multiple directories when registering, they would all appear here. The server simply refuses access to anything outside these boundaries, which is the safety mechanism described in the overview — a well-designed MCP server limits itself to explicit boundaries you define at configuration time.

### Step 5: Start a Claude Code session and ask Claude to read the file

Now the moment you have been building toward — a real MCP round trip from a natural language question through protocol messages to the filesystem and back.

Start an interactive Claude Code session:

```bash
claude
```

Once the session loads (you will see the Claude Code prompt), ask:

```text
Read the file mcp-test.txt in my project and tell me what it says.
```

Claude Code will recognize that the `filesystem` server has a `read_file` tool matching this request. Before the tool executes, you will see a permission prompt:

```text
Claude wants to use the filesystem server's read_file tool with these arguments:
  path: "/home/you/projects/learn-mcp/mcp-test.txt"

Allow this tool call?
  [Allow]  [Allow Always]  [Deny]
```

Choose **Allow**. Claude will run the tool and respond:

```text
Here is what mcp-test.txt contains:

Hello from the filesystem MCP server. If you can read this, MCP is working.
```

**Checkpoint:** this is the end state of the exercise — a full round trip from your question through Claude's reasoning, through the MCP protocol, to the filesystem, and back. If you see your file's content in Claude's response, the entire MCP stack is working.

### Step 6: Try a second tool call to confirm the server stays connected

The server process stays alive for the duration of your Claude Code session. Ask a follow-up question that uses a different tool on the same server:

```text
List the files in my project directory.
```

Claude will request to call `list_directory` on the `filesystem` server, and after your approval, return a listing of files in your project directory:

```text
Here are the files in your project directory:
- mcp-test.txt
- README.md
- src/
- package.json
- ...
```

### Step 7: Try a denied operation and observe what happens

Part of understanding MCP is also understanding what happens when you say no. Ask Claude to read a file outside the allowed directory:

```text
Read the file /etc/hostname on my system.
```

Claude will request to call `read_file` with path `/etc/hostname`. Allow the call. The tool will execute but return an error:

```text
Error: File /etc/hostname is not within allowed directories.
```

Claude will relay this back to you, explaining that the server is restricted to the allowed directory you configured. The tool-level error (`isError: true`) lets the model see the failure and react — it will explain what happened rather than silently failing.

Now choose **Deny** if Claude tries the operation again, and observe that the model receives a clear signal that the operation was blocked. Denial is a clean, predictable outcome, not a crash.

### Step 8: Disconnect and clean up

When you are done experimenting, exit the Claude Code session:

```text
/exit
```

Remove the server from your Claude Code configuration so it does not keep launching every time you start a new session:

```bash
claude mcp remove filesystem
```

```text
Removed stdio MCP server "filesystem" from local config
```

Confirm it is gone:

```bash
claude mcp list
```

```text
No MCP servers configured.
```

Optionally delete the test file:

```bash
rm mcp-test.txt
```

## Common Errors

**Error: `npx: command not found` when trying to run the server**

```text
npx: command not found
```

Cause: Node.js is not installed, or npm is not on your PATH. Run `node --version` and `npm --version` to diagnose which one is missing. Install Node.js from `https://nodejs.org`, then restart your terminal.

**Error: `EPERM: operation not permitted` or permission-related crashes**

```text
Error: EACCES: permission denied, stat '/home/you/projects/learn-mcp'
```

Cause: the path you passed to the filesystem server is not readable by your user, or the directory does not exist. Confirm the path is correct with `ls /home/you/projects/learn-mcp` and that your user owns the directory or has read permissions on it.

**Error: server shows `✗ Failed to connect` after `claude mcp add`**

```text
filesystem: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp - ✗ Failed to connect
```

Cause: the server process crashed during startup. This can happen if the path you passed to it does not exist — the filesystem server validates its allowed directories at startup and exits immediately if they are missing.

Fix: check the detailed error:

```bash
claude mcp get filesystem
```

```text
filesystem
  Status: ✗ Failed to connect
  Command: npx -y @modelcontextprotocol/server-filesystem /home/you/projects/learn-mcp
  Error: [the actual error message]
```

If the error mentions `directory not found` or `ENOENT`, double-check the absolute path you used in the registration command and run `claude mcp add` again with the corrected path.

**Error: server fails to start because the path contains spaces**

```text
Error: ENOENT, path does not exist
```

Cause: the allowed directory path has spaces in it (for example, `/home/you/My Projects/learn-mcp`), and `npx` split the argument at the space. The filesystem server only sees `/home/you/My` and `/Projects/learn-mcp` as two separate arguments.

Fix: quote the path when registering:

```bash
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem "/home/you/My Projects/learn-mcp"
```

**Error: server connected but Claude does not show the tool calls**

```text
I don't have a way to read files in this environment.
```

Cause: Claude may not be including the filesystem server's tools in its reasoning if the server registered after the session started. The server must be listed in `claude mcp list` before you start the session.

Fix: exit the Claude Code session, confirm `claude mcp list` shows the server as connected, and start a fresh session.

**Error: `npm ERR` during npx download**

```text
npm ERR! code E404
npm ERR! 404 Not Found - GET https://registry.npmjs.org/@modelcontextprotocol%2fserver-filesystem
```

Cause: you may be behind a corporate proxy or firewall that blocks npm registry access, or the package name may be typed incorrectly.

Fix: check the spelling matches `@modelcontextprotocol/server-filesystem` exactly, and if behind a proxy, configure npm's proxy settings or use a different network.

## What You Just Built

- A working MCP filesystem server installed from a public npm package and verified independently before connecting it to any AI client
- A server registered with Claude Code using `claude mcp add`, connecting a command-line AI application to a real local file system through the Model Context Protocol
- A confirmed round-trip tool call — a natural language question translated by the model into a `read_file` tool call, transmitted as a JSON-RPC `tools/call` message over stdio, executed by the server, and returned as structured content the model read and presented to you
- Exposure to the permission gate — see and approve every tool call before it touches your system, and observe what happens when a tool is blocked or denied
- Experience with the scope restriction — a server that refuses operations outside its configured allowed directories, showing how MCP servers enforce boundaries
