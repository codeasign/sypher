You are building an agent that needs to read files from a project, query a Postgres database, search the web, and call an internal ticketing API. For each of these, you write a bespoke integration: a wrapper function that formats the request, a parser that unpacks the response, error handling tuned to that one service's quirks, and a description string you hand-craft so the model knows the function exists. Four tools, four completely different pieces of glue code, none of which are reusable outside this one project.

Three months later, a teammate builds a different agent that also needs filesystem access and database queries. They cannot reuse your integrations because your file-reading wrapper is tangled up with your agent framework's specific tool-calling format, and your database wrapper hardcodes your project's connection logic. They write their own from scratch. Now there are two filesystem integrations and two database integrations in the company, both doing roughly the same thing, both maintained separately, both drifting out of sync as the underlying APIs change.

This is the problem every team building with LLMs eventually hits: connecting a model to the outside world does not scale by writing custom glue code per tool, per project, per framework. Every new data source is another one-off adapter. Every framework migration means rewriting every adapter again. What is missing is not more code — it is a shared interface that any tool can implement once and any LLM application can consume without custom work.

Take a moment to imagine what that world looks like. A developer in your company creates a new internal API — a knowledge base search, a deployment status checker, a customer lookup. Instead of writing an SDK, sending out documentation, and waiting for each team to integrate it their own way, they write one MCP server and publish the endpoint URL. Any LLM application in the company, from Claude Code to a custom dashboard to an IDE plugin, can now discover and use that API, without a single line of integration code being written. The API's author does not need to know who will consume it, and the consumers do not need to know how it was built.

That is the vision MCP was created to realize. The rest of this page is about how it gets there.

## The Concept

MCP (Model Context Protocol) is an open standard that lets LLM applications discover and call external tools through a unified interface, the same way USB lets you plug any peripheral into any computer.

Before USB, connecting a mouse, printer, or external drive to a computer meant matching serial ports, parallel ports, and proprietary connectors, each with its own driver and cabling. USB replaced all of that with one physical connector and one protocol: any USB device can describe itself to any USB host, and the host knows how to talk to it without custom wiring. MCP does the same thing for LLM applications and tools. An MCP server describes what it offers — tools it can run, data it can expose, prompt templates it provides — and any MCP client, whether that is Claude Code, a custom agent, or an IDE plugin, can connect to it, ask what is available, and use it, without the server and client having been written with each other in mind.

The people who benefit are on both sides of the interface. Tool builders write one MCP server and it works with every MCP-compatible client that now exists or ever will. Application builders write one MCP client integration and it works with every MCP server, whether that server wraps a filesystem, a database, a SaaS API, or a custom internal system. The protocol is the contract that makes both sides interchangeable.

There is a second, subtler benefit that shows up once you have more than one MCP server connected at the same time. Because every server describes its own tools using the same JSON Schema format, an LLM application can connect to a filesystem server, a database server, and a custom internal server simultaneously, merge all their tool catalogs into one list, and let the model pick freely across all of them in a single conversation. Nothing about the filesystem server needs to know the database server exists, and neither one needs to know anything about the application connecting to both. Composability like that is only possible because the interface is standardized rather than negotiated case by case.

MCP is deliberately not trying to be a new programming language, a new agent framework, or a replacement for the model's own reasoning. It is narrowly scoped to one job: describing what capabilities exist and carrying calls and results between two processes. Everything else — deciding when to call a tool, how to interpret the result, what to do next — stays exactly where it already was, inside the model and the application built around it.

### How This Differs from What You Already Know

If you have worked with REST APIs or gRPC, the idea of a protocol that sits between services will feel familiar. The critical difference is that MCP is designed for a caller that does not know ahead of time what it will need to call. A REST API client is written against a specific endpoint — it knows the URL, the HTTP method, the expected request body, and the response shape because those are documented and fixed. An MCP client, by contrast, discovers all of those details at runtime by asking the server what it offers. The client does not know whether the server exposes a single tool or fifty until it calls `tools/list` and sees the reply. That runtime discovery is the feature that makes MCP feel different from any API integration pattern you have used before.

### Why "Protocol" and Not "Library" or "Framework"

The word choice matters here. A library is code you import and call directly, inside your own process, in your own language. A framework dictates the overall shape of your application and expects you to build within its structure. A protocol is neither of those — it is an agreement about message formats and sequencing that two independent processes, possibly written in entirely different languages, can both implement without either one depending on the other's source code. MCP servers exist for Python, TypeScript, Go, and other languages, and a client written in any of them can talk to a server written in any other, because neither side needs to read the other's code. They only need to follow the same protocol.

This is exactly the same relationship HTTP has to web browsers and web servers. Nobody asks whether a browser and a server are written in the same language, because HTTP made that question irrelevant decades ago. MCP is trying to make the same question irrelevant for LLM applications and the tools they call.

## API Reference Table

MCP is defined as a set of primitives (the things a server can expose) and JSON-RPC messages (the wire format clients and servers exchange), layered on top of a transport (how the bytes actually move between processes). The table below covers every piece you need to understand any MCP interaction, whether you are using an existing server or building one.

| Primitive / Message | Exact Syntax | What It Does | One-Line Example |
|---|---|---|---|
| Tools | `tools/list`, `tools/call` | Executable functions the server exposes that the model can invoke, each with a name, description, and JSON Schema input | A `read_file` tool with an input schema requiring a `path` string |
| Resources | `resources/list`, `resources/read` | Addressable pieces of context (files, database rows, API responses) identified by a URI that the client can fetch | A resource at `file:///project/README.md` the client retrieves to add to context |
| Prompts | `prompts/list`, `prompts/get` | Reusable, parameterized prompt templates the server offers, so common workflows do not need to be re-written by every client | A `summarize-pr` prompt template that takes a `pr_number` argument |
| Roots | `roots/list` (client-exposed) | Filesystem or resource boundaries the client tells the server it is allowed to operate within | A client declaring `/home/you/project` as the only root a filesystem server may touch |
| `initialize` | `{"method": "initialize", "params": {"protocolVersion": "...", "capabilities": {...}, "clientInfo": {...}}}` | The first message a client sends; negotiates protocol version and declares what the client supports | Client asks to speak protocol version `2025-06-18` |
| `initialized` | `{"method": "notifications/initialized"}` | Notification the client sends after receiving the server's InitializeResult; final step of the handshake | Client tells the server it is ready to proceed |
| `tools/list` | `{"method": "tools/list", "params": {"cursor": "..."}}` | Client asks the server which tools it exposes, with optional pagination for servers with many tools | Server replies with an array of tool definitions including `inputSchema` |
| `tools/call` | `{"method": "tools/call", "params": {"name": "...", "arguments": {...}}}` | Client asks the server to execute a named tool with the given arguments | `{"name": "read_file", "arguments": {"path": "notes.txt"}}` |
| `resources/list` | `{"method": "resources/list"}` | Client asks the server which resources are currently available | Server replies with an array of `{uri, name, mimeType}` objects |
| `resources/read` | `{"method": "resources/read", "params": {"uri": "..."}}` | Client asks the server to return the contents of a specific resource | `{"uri": "file:///project/config.json"}` returns the file contents |
| `prompts/list` | `{"method": "prompts/list"}` | Client asks the server which prompt templates it offers | Server replies with prompt names and their argument definitions |
| `prompts/get` | `{"method": "prompts/get", "params": {"name": "...", "arguments": {...}}}` | Client asks the server to fill in a named prompt template with the given arguments | `{"name": "summarize-pr", "arguments": {"pr_number": "42"}}` |
| `notifications/tools/list_changed` | `{"method": "notifications/tools/list_changed"}` | Server-notification: the tool list has changed, the client should re-fetch | Server sends this when a tool is added or removed at runtime |
| `notifications/resources/list_changed` | `{"method": "notifications/resources/list_changed"}` | Server notification: the available resources have changed | Server sends this when a new resource becomes available |
| `notifications/cancelled` | `{"method": "notifications/cancelled", "params": {"requestId": "..."}}` | Either side cancels a pending request by its request ID | Client cancels a long-running `tools/call` that is no longer needed |
| `logging/message` | `{"method": "notifications/message", "params": {"level": "...", "data": {...}}}` | Server sends a log-level notification to the client for debugging | Server logs a warning when a tool call returns an unusual error |
| stdio transport | `command` + `args` in server config, communication over stdin/stdout | Runs the MCP server as a local child process; JSON-RPC messages are newline-delimited frames on stdin and stdout | `npx -y @modelcontextprotocol/server-filesystem /path` launched as a subprocess |
| SSE / HTTP transport | A URL the client connects to, e.g. `{"url": "https://.../sse"}` | Runs the MCP server as a network service; the client connects over HTTP with Server-Sent Events for the response stream | A shared team server running at an internal HTTPS endpoint |
| Streaming transport | Extended SSE for bidirectional streaming | Allows the server to stream partial results for a single `tools/call` before the final response is ready | A long-running database query streams rows as they arrive |

The three messages worth memorizing first are `initialize` (nothing works before this), `tools/list` (how the model learns what it can do), and `tools/call` (how anything actually happens). Resources and prompts follow the identical request-and-response pattern, just for different kinds of content. Once you understand the shape of one JSON-RPC exchange, you understand the shape of all of them — a method name, a params object, and a response that either succeeds with a result or fails with an error.

### Understanding JSON-RPC in MCP

Every message in MCP is a JSON-RPC 2.0 message. JSON-RPC is a lightweight, transport-agnostic protocol that defines three kinds of messages: requests (expect a response), notifications (no response expected), and responses (the reply to a request). MCP uses all three. A `tools/call` is a request — the client sends it and expects a `CallToolResult` back. A `notifications/initialized` is a notification — the client sends it and does not expect a reply. A response always includes the `id` that matched the request, which is how the client associates a response with the originating call when messages are in flight concurrently.

The `id` field deserves special attention. Because MCP messages are asynchronous — a client can send multiple requests to the same server before any of them gets a response — the `id` is the only way to match responses to their requests. The server copies the `id` from the request into the response verbatim, so the client knows exactly which outstanding call just completed. This is the same pattern used in AMPQ, WebSocket sub-protocols, and any other system that multiplexes concurrent calls over a single connection.

### A Real `tools/call` Request and Response

It helps to see the actual JSON that moves across the wire, rather than only the shape described in a table. Here is what a client sends when a tool named `read_text_file` is called with a specific path:

```json
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "read_text_file",
    "arguments": {
      "path": "notes.txt"
    }
  }
}
```

And here is a successful response from the server:

```json
{
  "jsonrpc": "2.0",
  "id": 7,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Meeting notes for Tuesday: discuss rollout plan."
      }
    ],
    "isError": false
  }
}
```

Notice that the `id` field matches between the request and the response. If the file did not exist, the response would still be a normal JSON-RPC result (not a JSON-RPC error), just with `isError: true` and a text explanation of what went wrong, so the model sees the failure and can react to it.

### A Real `resources/read` Response

Resources work the same way, but the response shape is slightly different because the server returns a blob of content that could be text or binary:

```json
{
  "jsonrpc": "2.0",
  "id": 12,
  "result": {
    "contents": [
      {
        "uri": "file:///project/README.md",
        "mimeType": "text/markdown",
        "text": "# Project Title\n\nThis is the readme."
      }
    ]
  }
}
```

Resources do not have an `isError` flag because reading a resource that does not exist is considered a protocol error — if the URI is valid and reachable, the read succeeds, and if the URI is unknown, the server returns a JSON-RPC error object with a code like `-32602` (invalid params). This is a subtle but important difference from tools, where failure is a legitimate outcome the model needs to handle. Resources are a simpler contract: they are either there or they are not.

The model itself never talks to the filesystem, the database, or the API directly. It talks to the MCP client, which is part of the LLM application. The client talks to the MCP server over a transport. The server is the only thing that touches the real system underneath. This separation is what makes MCP safe and swappable: you can replace the underlying system, or replace the LLM application, and as long as both sides still speak MCP, nothing else has to change.

## How It Actually Works — Step by Step

MCP is not magic. It is a well-defined sequence of messages between two processes, and understanding that sequence from start to finish is the best way to understand what the protocol actually does for you. The steps below walk through a complete interaction, from the first connection to handling errors and edge cases.

### Step 1: The Client Launches or Connects to the Server

For a local server, the MCP client starts the server as a child process using the configured command, for example `npx -y @modelcontextprotocol/server-filesystem /path/to/project`. For a remote server, the client opens an HTTP connection to a configured URL. Either way, once the transport is open, no tools are usable yet — the two sides have not agreed on anything.

This step is where most configuration mistakes surface. If the command in the server config points to a binary that is not installed, or a path that does not exist, the process launch itself fails, and you see a connection error before the protocol ever gets a chance to run. It is worth remembering that this step is pure operating-system process management — MCP has not entered the picture yet. Everything from here on is what happens once that process is alive and listening.

Under the hood, launching a stdio MCP server follows the same pattern as spawning any subprocess. The client allocates stdin and stdout pipes, starts the process, and begins reading newline-delimited JSON from stdout while writing JSON-RPC messages to stdin. If the process exits unexpectedly — segfault, uncaught exception, manual kill — the client sees the pipe close and knows the session is dead. Reconnection logic is up to the client; the MCP specification does not mandate automatic reconnection, though many clients implement it as a courtesy.

For an SSE server, the client opens a persistent HTTP connection and subscribes to the event stream. SSE connections can be interrupted by network issues, proxies that time out idle connections, or server restarts. A robust MCP client detects the stream ending and either attempts to reconnect (potentially with exponential backoff) or reports the disconnection to the user so they know the session is broken.

### Step 2: The Handshake — `initialize`

The client sends an `initialize` request containing the protocol version it wants to use, its own capabilities, and information about itself. The server responds with its own protocol version, its capabilities (does it support tools, resources, prompts), and information about itself. If the protocol versions are incompatible, the connection is refused here rather than failing confusingly later.

The `initialize` request is the only message the client can send before the handshake is complete. Every other message — `tools/list`, `resources/read`, `tools/call` — is invalid until the server has responded with an `InitializeResult` and the client has sent the `notifications/initialized` notification. This sequencing is deliberate: it forces both sides to declare what they support before any operation starts, preventing the kind of "I thought you supported that" bugs that happen when capabilities are assumed rather than negotiated.

### Step 3: Capability Negotiation

Capabilities are declared, not assumed. The server's `initialize` response says explicitly whether it supports tools, resources, prompts, and whether it will send change notifications when those lists update. The client only attempts operations the server has declared support for. If a server does not declare a `tools` capability, the client will not bother calling `tools/list` against it — there is nothing to list.

Capability negotiation also protects both sides from talking past each other as the protocol evolves. A newer client connecting to an older server will see that the server does not declare support for a newer feature and simply will not use it, rather than sending a request the server has no idea how to answer. This is the same reason web browsers negotiate which HTTP features a server supports before relying on them — declared capabilities avoid a whole category of "it should have worked" bugs.

The client has capabilities too, and it declares them in the same `initialize` request. A server can, for example, check whether the connecting client supports `roots` before it tries to ask the client which directories it is allowed to touch. Negotiation runs in both directions, not just from client to server.

Here is what a real `initialize` request and response look like with the details filled in:

```json
// Client sends:
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "roots": {
        "listChanged": true
      },
      "sampling": {}
    },
    "clientInfo": {
      "name": "my-custom-agent",
      "version": "1.0.0"
    }
  }
}

// Server responds:
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "tools": {
        "listChanged": true
      },
      "resources": {
        "subscribe": true,
        "listChanged": true
      },
      "prompts": {
        "listChanged": true
      }
    },
    "serverInfo": {
      "name": "filesystem-server",
      "version": "0.1.0"
    }
  }
}
```

Notice that the server declares `tools`, `resources`, and `prompts` capabilities, meaning it supports all three primitives. Each also declares `listChanged: true`, which means the server will send a notification when its list of available tools, resources, or prompts changes at runtime. The client can now decide to subscribe to those notifications or simply poll occasionally — the capability declaration gives the client the choice.

### Step 4: Tool Discovery — `tools/list`

Once the session is ready, the client sends `tools/list`. The server responds with an array of tool definitions. Each definition has a `name`, a human-readable `description`, and an `inputSchema` written in JSON Schema describing exactly what arguments the tool accepts. This is the moment the LLM application learns what it is capable of doing through this server — nothing is hardcoded on the client side about what a "filesystem server" or a "database server" can do.

The discovery step is what makes MCP fundamentally different from hardcoded tool integrations. When you hardcode a tool, you have to update the code every time the tool's interface changes — a new argument, a renamed parameter, a changed return format. With MCP, the server controls the interface definition, and the client learns it on every connection. You update the server, and every client that connects sees the new interface immediately, without any client-side code changes.

Discovery also handles pagination. A server that exposes a very large number of tools — hundreds or thousands — can return the first page of results with an optional `cursor` token that the client sends back to get the next page. This prevents a single `tools/list` response from being too large to handle, while still giving the client a complete picture of what is available.

### Step 5: The Model Decides to Call a Tool

The LLM application includes the discovered tool definitions in the model's context, formatted the way that model's API expects. When the model determines that answering the current request requires a tool — reading a file, running a query — it emits a structured tool-call request naming the tool and its arguments. This decision happens entirely inside the model's own reasoning; MCP does not influence when a tool gets called, only how the call is transmitted once the model decides to make one.

This is worth stating plainly because it is easy to assume MCP is somehow smarter than it is: the protocol carries descriptions and results, but it has no opinion about which tool is the right one for a given task. That judgment is entirely the model's. A poorly written tool description — vague, missing the constraints on valid inputs, silent about what happens on failure — leads directly to the model calling the wrong tool or calling the right tool with the wrong arguments. The quality of a tool's `description` and `inputSchema` is doing real work here, even though neither is executable code.

Consider two descriptions for the same tool:

| Bad Description | Good Description |
|---|---|
| "Read a file" | "Read the entire contents of a text file from the project directory. Returns the file content as a string. Fails if the file does not exist or is not within the allowed root. Max file size: 1 MB." |

The good description tells the model what the tool does, what it returns, when it fails, and what constraints apply. The model uses all of that information to decide not only whether to call this tool, but whether calling it is safe given the current context. Good descriptions are not nice-to-have documentation — they are part of the tool's logic, as critical as the implementation behind it.

### Step 6: The Call Executes — `tools/call`

The MCP client translates the model's tool-call request into a `tools/call` JSON-RPC message and sends it to the server over the transport. The server runs the corresponding handler, does whatever work the tool represents (reading a file, hitting an API, querying a database), and returns a result. The result contains a `content` array (usually text, but it can include images or embedded resources) and an `isError` flag.

The `content` array can hold multiple items of different types. A tool that generates a chart could return both a `text` item explaining the chart and an `image` item containing the chart itself. A tool that queries a database could return a `text` item with the query results and a `resource` item that references a URI the client can read for full details. Multiplexing content this way lets a single tool call produce rich, structured results without requiring the model to make follow-up calls for related data.

### Step 7: The Permission Gate

Most MCP clients built for interactive use, including Claude Code, insert a checkpoint between the model deciding to call a tool and the tool actually executing: your explicit approval. The first time a given tool is about to run, or the first time it is about to run with a particular class of argument, the client shows you the tool name and the exact arguments and waits for you to allow or deny it. Only after approval does the `tools/call` message in Step 6 actually get sent. This gate is not part of the MCP specification itself — it is a safety layer that responsible clients build on top of the protocol, and it is one of the main reasons MCP feels safe to use with tools that can modify real systems.

Approval choices are typically remembered per tool, so you are not prompted every single time for an operation you have already decided is safe. "Allow Always" tells the client to skip the prompt for that specific tool in future calls within the same project, while "Deny" blocks the call and returns control to the model, which then knows the operation did not happen and can decide what to do next — try something else, or simply tell you it was blocked.

Some clients implement more granular permission systems. You might be able to approve a tool for reading but not writing, or approve it when called with certain argument patterns (paths under `/tmp`) but deny it for others (paths under `/etc`). These are client-specific features, not part of the MCP specification, but they illustrate the flexibility the protocol gives clients to implement safety at whatever granularity they need.

### Step 8: Error Handling

Errors in MCP come in two forms. A tool-level error is when the tool ran but the operation it represents failed — a file did not exist, a query had bad syntax. This is returned as a normal `tools/call` response with `isError: true` and an explanation in the content, so the model sees it and can adjust, retry with different arguments, or tell you it failed. A protocol-level error is when the request itself was malformed or the method does not exist; this comes back as a JSON-RPC error object with a numeric code, and it usually indicates a bug in the client or server rather than something the model can reason its way around.

Here are the standard JSON-RPC error codes defined for MCP:

| Code | Name | Meaning |
|---|---|---|
| `-32700` | Parse Error | The JSON sent is not valid JSON |
| `-32600` | Invalid Request | The JSON is valid but not a valid JSON-RPC request |
| `-32601` | Method Not Found | The requested method (e.g. `tools/call`) is not supported by the server |
| `-32602` | Invalid Params | The method exists but the parameters are invalid |
| `-32603` | Internal Error | An unexpected error occurred inside the server |
| `0` to `-32000` | Server Error | Custom errors defined by the server implementation |

A well-written MCP server distinguishes carefully between tool-level and protocol-level errors. If a user calls a database query tool with invalid SQL, the tool should return `isError: true` with a clear message about what went wrong with the query. The server should not return a `-32602` Invalid Params error, because the tool's parameters (the query string) were valid JSON-RPC — it was the database operation itself that failed. Getting this distinction right is what allows the model to recover from failures intelligently rather than hitting a dead end.

### Step 9: The Transport Layer Underneath All of This

Every message above rides on a transport. With stdio, JSON-RPC messages are written as newline-delimited JSON on the server process's stdin and stdout — nothing else may be written to stdout, which is why servers log diagnostics to stderr instead. With SSE (Server-Sent Events over HTTP), the server runs as a persistent network process, the client posts JSON-RPC requests over HTTP, and receives responses (and any unsolicited server-to-client notifications) as an event stream. Both transports carry the exact same JSON-RPC message types — only the plumbing differs, which is precisely why a tool built for one transport can often be exposed over the other with no change to its logic.

The choice of transport is really a choice about where the server lives. stdio only makes sense when the client and server are on the same machine, because the client has to be able to spawn the process directly. SSE makes sense the moment you want a server that outlives any single client session, serves more than one client at a time, or needs to run somewhere other than your laptop — a shared internal server hosting a company's ticketing tool, for example, where every developer's MCP client connects to the same running instance instead of each spawning their own.

There is a third transport model emerging in the MCP ecosystem: the subprocess transport in reverse, where the server is a persistent daemon and the client connects to it over a local socket. This gives you the simplicity of local communication (no network config, no authentication for the transport layer itself) with the persistence of a long-running server. This is still less standardized than stdio and SSE, but it shows how the protocol's transport abstraction lets the community experiment with different plumbing without changing anything about the tools, resources, or prompts that run on top.

Here is what a stdio transport looks like when you configure it in an MCP client:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/home/you/projects/my-app"
      ]
    },
    "database": {
      "command": "node",
      "args": [
        "path/to/db-server.js"
      ],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    }
  }
}
```

And here is what an SSE transport looks like:

```json
{
  "mcpServers": {
    "team-ticketing": {
      "url": "https://mcp.internal.company.com/ticketing/sse"
    }
  }
}
```

The same client can have a mix of stdio and SSE servers. The transport is an implementation detail of each server individually. The client does not care how a particular server communicates — it only cares that JSON-RPC messages go out and responses come back.

### Step 10: Resources and Prompts Follow the Same Shape

Everything described above for tools applies almost identically to resources and prompts, which is part of what makes MCP easy to reason about once you understand one primitive. A client calls `resources/list` to discover what is available, then `resources/read` with a specific URI to fetch content — no arguments to shape, no side effects, just a read. Prompts work the same way: `prompts/list` to discover named templates, `prompts/get` with a name and arguments to retrieve the filled-in prompt text. The consistent shape — list to discover, then act on a specific named thing — is a deliberate design choice that keeps the protocol small enough to hold in your head.

Resources are particularly useful because they give the model access to up-to-date context without needing to call a tool first. A resource could represent the current state of a pull request, the latest test results, or the contents of a configuration file. Because resources are URI-addressable, the client can fetch them on demand, cache them, or subscribe to change notifications to keep them fresh. The resource abstraction makes context data first-class in MCP, rather than something you have to retrieve through a tool call every time.

Prompts serve a different purpose entirely. They are not about getting data into the model or executing operations — they are about packaging up commonly needed instructions so that any client can use them without re-inventing the pattern. A "code review" prompt template, for example, could take a diff as an argument and produce a complete review prompt with the right structure, tone, and output format. The client (or the model) can discover that template exists and use it without knowing anything about how review prompts should be structured. The server encapsulates that domain knowledge.

### Step 11: Change Notifications — When Things Update at Runtime

Servers that declare `listChanged: true` in their capabilities can send notifications when their tool, resource, or prompt lists change after the session is already established. This is important for servers whose capabilities change dynamically — a server that loads plugins at runtime, a CI server whose available actions change based on what is running, a database server whose available resources change as tables are created or dropped.

When the client receives a `notifications/tools/list_changed` notification, it knows the tool list it cached is stale and should call `tools/list` again. The same pattern applies to `notifications/resources/list_changed` and `notifications/prompts/list_changed`. Without this mechanism, the client would either never learn about changes (and keep using stale tool definitions) or have to poll the server periodically. Notifications let the server push changes as they happen, keeping the client's view up to date without constant polling.

## Cost, Limits, and Safety

MCP the protocol is free — it is a specification, not a paid service, and running an MCP server locally over stdio costs nothing beyond the compute it uses. The costs come from what sits behind the interface. A tool that calls a paid third-party API accrues that API's charges every time the model invokes it. A tool that hits a rate-limited service can get your whole session throttled if the model calls it too aggressively in a loop. And every tool call and its result becomes part of the model's context, which means large results add real token cost, and a long chain of tool calls can meaningfully eat into your context window and your bill on metered APIs.

Because a server is a separate process, credentials for whatever it wraps must never be written directly into a config file that gets checked into version control. Pass them as environment variables at launch, referenced by name in the server configuration, and set the actual values in your shell or a local, git-ignored file. If a server needs a database password or an API key, that value should exist in exactly one place: your environment, not a JSON file sitting next to your source code.

Safety in MCP rests on three mechanisms working together. First, sandboxing: a well-designed server restricts itself to explicit boundaries — a filesystem server should only touch directories you explicitly allow, a database server should only run the query types you permit. Second, user approval gates: MCP clients like Claude Code prompt you before running a tool call the first time you see it, so nothing executes silently on your machine or against your systems without your say. Third, no arbitrary code execution by design: a tool is a fixed, named operation with a fixed input schema — the model can supply arguments to a tool, but it cannot use a well-built MCP tool to execute code the tool's author did not intend it to run. That guarantee only holds if the tool itself is implemented carefully; a tool that shells out to a command built from unsanitized model-supplied input reintroduces exactly the risk the interface was supposed to prevent.

The practical safety habits: only connect to MCP servers whose source you have read or that come from a source you trust, start every new server with the narrowest possible permissions and widen them only as needed, never pre-approve destructive operations, and treat any server that asks for broad filesystem or network access with real suspicion.

### Rate Limits Belong to the Underlying System, Not MCP

It is worth being precise about where rate limits actually live. MCP itself has no built-in concept of a rate limit — it is a message format, not a traffic cop. If a tool wraps a third-party API that allows 60 requests per minute, that limit is enforced by the third-party API, not by MCP. The tool's handler will get a rate-limit error back from that API, and a well-written tool should translate that into a clear `isError: true` result so the model understands it needs to slow down or wait, rather than retrying immediately in a tight loop.

Some MCP clients add their own guardrails on top of this, independent of anything the server does. Claude Code, for instance, caps the number of sequential tool calls it will make within a single conversation turn, specifically to prevent a model from getting stuck calling tools in an unproductive loop. That cap is a client-side safety feature, not something the MCP specification requires — different clients are free to implement their own limits, or none at all.

### A Concrete Cost Example

Consider a tool that queries a database and returns full result rows with no limit. If the model asks a vague question like "show me all the users," and the table has ten thousand rows, the tool might return every row as text in the result. That entire blob becomes part of the model's context for the rest of the conversation, at a real token cost, and depending on the model and pricing tier you are using, that could mean a noticeably larger bill for what should have been a cheap operation. This is why the filesystem server example later in this course caps directory listings, and why a well-designed database tool should default to a sane row limit unless the model explicitly requests more.

### How to Store Credentials Safely

Here is the pattern for referencing credentials without hardcoding them:

```json
{
  "mcpServers": {
    "database": {
      "command": "python",
      "args": ["db-server.py"],
      "env": {
        "DB_PASSWORD": "${DB_PASSWORD}",
        "API_KEY": "${INTERNAL_API_KEY}"
      }
    }
  }
}
```

The `${VAR}` syntax tells the MCP client to read the value from the environment variable at launch time, not from the config file. You set those variables in your shell profile, a `.env` file that is listed in `.gitignore`, or your CI/CD pipeline's secrets store. The config file is safe to commit because it contains only variable references, not the actual secrets.

### What Happens When a Server Is Malicious

MCP does not include a built-in trust or verification layer. If you connect to an MCP server that you or your team did not build, you are trusting that server's author to implement the protocol honestly. A malicious server could:

- Declare a tool called `read_file` but actually execute a `rm -rf` command in its handler
- Return fabricated data to the model that leads the model to make incorrect decisions
- Send change notifications that trick the client into making repeated re-discovery calls (a denial-of-service vector)
- Log the arguments it receives for every tool call, potentially capturing sensitive data passed by the model

None of these are vulnerabilities in MCP itself — they are risks of running untrusted code. The same risks apply to installing any npm package, PyPI package, or binary from an untrusted source. The discipline is the same: only run servers from sources you trust, inspect the source code of servers you are unsure about, and use sandboxing (containers, restricted user accounts, read-only filesystem mounts) for servers you want to run but do not fully trust.

## When to Use This / When Not To

MCP earns its overhead when you need the same tool available across multiple LLM applications or frameworks, when you are building a tool you expect to reuse or share with a team, or when you want the underlying system swapped without touching the calling code. If you are writing an agent that needs filesystem access, database access, and a ticketing system, and you want that agent framework-agnostic, MCP servers are the right layer.

Direct API calls beat MCP when you have one specific integration, in one specific codebase, that will never be reused elsewhere and does not need to be dynamically discovered by a model. Writing a thin Python function that calls a REST endpoint and returning its result is simpler, faster to build, and easier to debug than standing up an MCP server for a single, one-off use. MCP adds a process boundary, a JSON-RPC layer, and a discovery step — real cost when none of that flexibility is needed.

MCP versus plugin systems is a question of scope. A plugin system tied to one application (a specific IDE, a specific chat product) only works inside that application. MCP is designed to work across applications, which matters if you expect more than one LLM application to need the same capability. If you are certain you will only ever use one application, a native plugin can be simpler to build and ship.

For agents versus simple scripts: an agent that reasons about which of many tools to call, possibly across several steps, benefits from MCP's discovery model — the model sees the tool catalog and decides. A simple script that always calls the same fixed sequence of operations does not need a model deciding anything, and wiring it through MCP just to get there adds indirection with no payoff. Use MCP when the calling side is a model making decisions, not when it is deterministic code that already knows exactly what to call and when.

### A Quick Decision Checklist

Ask yourself these questions before reaching for MCP:

- Will more than one LLM application, framework, or team need this same capability. If yes, MCP's reuse pays off quickly.
- Does the calling side need to discover what is available at runtime, rather than having it hardcoded. If yes, MCP's discovery model is the right shape.
- Is this a single, throwaway integration inside one codebase that nothing else will ever touch. If yes, a direct function call is faster to build and easier to debug.
- Does the tool need to run in a separate process for isolation, or does it need to run remotely. If yes, MCP's transport model already handles that; a direct call would need you to build that isolation yourself.
- Is the operation deterministic and always called the same way, with no model reasoning involved in deciding to call it. If yes, skip MCP and just call the function directly.
- Are you building something that will outlive the current LLM application landscape. If yes, MCP's protocol-based abstraction means your tools will work with whatever comes next, without being rewritten.

### MCP vs. Competing Approaches

You might wonder how MCP compares to other integration patterns you have seen or used. Here is a quick comparison:

| Approach | Discovery | Reuse | Isolation | Effort |
|---|---|---|---|---|
| Direct function call | None (hardcoded) | None (per-project) | None (same process) | Low |
| REST API call from code | None (hardcoded endpoint) | Per-codebase | Process boundary (HTTP) | Medium |
| Plugin system (IDE-specific) | Plugin manifest | Within one platform | Plugin sandbox | High |
| MCP | Runtime via `tools/list` | Across any MCP client | Process boundary (stdio/SSE) | Medium |
| Custom agent framework integration | Tool registry (framework-specific) | Per-framework | Framework-dependent | High |

MCP sits in a sweet spot: medium effort for setup, but high reuse and cross-platform compatibility. It is not the cheapest option for a single use case, but it is the most economical option for tools that need to work across multiple contexts.

## Mental Model Check

### Question 1

What problem does MCP solve that custom, per-tool glue code does not?

MCP replaces one-off integrations with a shared interface: a tool built once as an MCP server works with any MCP-compatible client, and a client built once works with any MCP server. Custom glue code only ever solves the problem for one tool in one project; MCP makes the solution reusable across tools, projects, and frameworks. Additionally, MCP's runtime discovery means a client can connect to a server it has never seen before and learn what that server offers, while custom glue code requires the client to know the interface in advance.

### Question 2

What are the three main primitives an MCP server can expose, and how do they differ?

Tools are executable functions the model can invoke with arguments. They cause actions and return results — reading a file, sending an email, running a query. Resources are addressable, readable pieces of context identified by a URI, like files or database rows. They are fetched by the client, not called by the model. Prompts are reusable, parameterized prompt templates for common workflows. Tools and resources bring data and actions to the model; prompts shape how the model processes them.

### Question 3

Why must `initialize` happen before any other MCP message, and what would go wrong if a client skipped it?

`initialize` negotiates the protocol version and declares capabilities on both sides. Skipping it means the client would not know what the server supports, could send messages the server does not understand, and would have no guarantee the two sides agree on message format at all. It would fail unpredictably rather than failing clearly at the handshake. In practice, skipping `initialize` means every subsequent message would get a JSON-RPC error response or be silently ignored, because the server does not consider the session established until the handshake completes.

### Question 4

Why is a tool-level error returned as part of a normal `tools/call` response instead of as a protocol error?

Because the model needs to see the failure to react to it — retry with different arguments, try a different tool, or explain the failure to you. If a tool-level error were a hard protocol failure (a JSON-RPC error response), the model would never see the actual problem and could not adjust its behavior. The `isError: true` pattern treats tool failure as a normal outcome that the model can reason about, not an exceptional condition that kills the conversation. This mirrors how human assistants work: you tell someone "that file does not exist" rather than ending the conversation when they make a mistake.

### Question 5

Why should credentials for an MCP server never be written directly into its configuration file?

Because that file often lives alongside project files that get committed to version control, and a hardcoded credential in that file becomes a leaked credential the moment it is pushed anywhere shared. Environment variables keep the actual secret value out of any file that could be committed, shared, or logged. Even in a private repository, configuration files tend to accumulate collaborators over time, and credentials in a config file create a permanent risk that grows with every person who gains access to the repository.

### Question 6

What is the difference between stdio transport and SSE transport, and when would you choose one over the other?

stdio transport runs the server as a child process with messages over stdin/stdout. It only works on the same machine and is tied to the client's lifecycle. SSE transport runs the server as a network service that multiple clients can connect to independently. Choose stdio for local development, personal tools, or anything that does not need to outlive a single session. Choose SSE for shared servers, production deployments, or any scenario where the server needs to be available independently of when a client connects.

### Question 7

A teammate says, "MCP is just a fancy way to call an API. I can do the same thing with a Python function and the `requests` library." Is your teammate right, and if not, what are they missing?

They are partially right: MCP does not let you do anything that is technically impossible with a Python function and HTTP calls. What they are missing is the standardization and discovery layer. With a raw Python function, every tool integration is a one-off that must be rewritten for every new client, every new framework, and every new project. With MCP, the tool author writes the server once, and the client handles discovery, transport, error translation, and result formatting automatically. The question is not whether you can achieve the same outcome with raw code — you can — but whether you want to pay the integration cost once or every time.

## Summary

MCP is a protocol that standardizes how LLM applications discover and call external tools, resources, and prompts. It is not a framework, not a library, and not a paid service — it is a shared contract that any tool builder and any LLM application can implement independently and still work together. The protocol uses JSON-RPC messages over stdio or SSE transports, with a handshake that negotiates capabilities before any operation starts. Tools, resources, and prompts all follow the same list-then-act pattern, which keeps the protocol small enough to hold in your head.

The key takeaways from this page:

- MCP replaces custom glue code with a standardized interface, making tools reusable across applications
- The handshake (`initialize` + `initialized`) must complete before anything else works
- Discovery (`tools/list`) is what makes MCP different from hardcoded integrations — the client learns at runtime
- Tool calls (`tools/call`) carry arguments and return structured results with an `isError` flag for recoverable failures
- Resources and prompts follow the same shape as tools but serve different purposes (context vs. instructions)
- The transport (stdio vs. SSE) is an implementation detail that does not change the protocol
- MCP is free, but the tools behind it have real costs in API usage and token consumption
- Safety relies on sandboxing, user approval gates, and careful server implementation — not on the protocol itself
- Use MCP when reuse, discovery, or cross-application compatibility matters; skip it for throwaway one-off integrations
