## The Concept

An MCP server is a lightweight program that exposes **tools**, **resources**, and **prompts** to LLM applications through the Model Context Protocol. Think of it as a purpose-built adapter: it sits between an AI assistant (like Claude Code) and the systems you already have — databases, APIs, file systems, or any service with an interface.

The server communicates over JSON-RPC, typically via standard input/output (stdio). The LLM client sends a request like "call the `get_forecast` tool with city=Tokyo," and your server runs the logic and returns the result. No HTTP setup, no authentication framework, no web server configuration is required for the basic case.

The Python MCP SDK (`mcp`) gives you everything you need. You define your tools as Python functions with type annotations, and the SDK handles the JSON-RPC serialization, request routing, and transport layer. A functioning server can be as short as thirty lines of code.

## API / SDK / CLI Reference

The Python MCP SDK provides these key classes and methods for building servers.

### Server

**Class: `mcp.server.Server`**

| Class / Method | Signature | Purpose | Example |
|---|---|---|---|
| `FastMCP` | `FastMCP(name: str, **kwargs)` | High-level entry point: auto-discovers tools and resources from decorated functions | `server = FastMCP("weather")` |
| `FastMCP.run()` | `run(transport: str = "stdio")` | Starts the server on the given transport | `server.run(transport="stdio")` |
| `Server` | `Server(name: str)` | Low-level server class for manual tool/resource registration | `srv = Server("weather")` |
| `Server.add_tool()` | `add_tool(fn, name, description)` | Registers a callable as a discoverable tool | `srv.add_tool(get_forecast, name="get_forecast")` |
| `Server.add_resource()` | `add_resource(uri, handler)` | Registers a resource template by URI pattern | `srv.add_resource("weather://{city}", handler)` |
| `Server.add_prompt()` | `add_prompt(name, fn)` | Registers a reusable prompt template | `srv.add_prompt("forecast", prompt_fn)` |
| `Tool` | `Tool(name, description, input_schema)` | Object describing a tool's name, description, and JSON Schema inputs | `Tool(name="get_forecast", description="...", input_schema={...})` |
| `Resource` | `Resource(uri, name, mime_type)` | Object describing a resource exposed at a URI | `Resource(uri="weather://tokyo", name="Tokyo Forecast")` |
| `Prompt` | `Prompt(name, description, arguments)` | Object describing a prompt template and its arguments | `Prompt(name="forecast", description="...", arguments=[...])` |

### Key Decorators (FastMCP)

| Decorator | Target | Purpose | Example |
|---|---|---|---|
| `@server.tool()` | Async function | Registers the function as a callable tool | `@server.tool() async def get_forecast(city: str): ...` |
| `@server.resource()` | Async function | Registers the function as a resource handler for a URI pattern | `@server.resource("weather://{city}") async def forecast(city: str): ...` |
| `@server.prompt()` | Async function | Registers the function as a prompt template | `@server.prompt() async def forecast_prompt(city: str): ...` |

### FastMCP Constructor Options

| Parameter | Type | Default | Purpose |
|---|---|---|---|
| `name` | `str` | required | Human-readable server name used in client UIs |
| `instructions` | `str` | `None` | System instructions the client sends to the LLM |
| `transport` | `str` | `"stdio"` | Transport layer: `"stdio"` or `"sse"` |
| `host` | `str` | `"0.0.0.0"` | Bind address for SSE transport |
| `port` | `int` | `8000` | Port for SSE transport |
| `log_level` | `str` | `"WARNING"` | Logging verbosity |

### Tool Input Schema Convention

Every tool's `input_schema` follows JSON Schema. The SDK infers it from your function signature if you use `FastMCP`:

```python
@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the weather forecast for a city."""
    ...
```

The SDK generates:

```json
{
  "type": "object",
  "properties": {
    "city": {"type": "string", "description": ""},
    "units": {"type": "string", "description": "", "default": "celsius"}
  },
  "required": ["city"]
}
```

## How It Actually Works

Building an MCP server follows a repeatable sequence of steps. Each step maps to a concrete piece of code or configuration.

### Step 1: Project Setup

Create a directory, a virtual environment, and a minimal `pyproject.toml`:

```bash
mkdir weather-mcp-server
cd weather-mcp-server
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
```

```toml
# pyproject.toml
[project]
name = "weather-mcp-server"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = ["mcp>=1.0.0"]
```

```bash
pip install "mcp>=1.0.0"
```

### Step 2: Server Initialization

Create `server.py` and instantiate the server:

```python
from mcp.server import FastMCP

# Each server needs a name — clients use this for display
server = FastMCP("WeatherServer")
```

The `FastMCP` constructor registers the server with the SDK. At this point the server has no tools, resources, or prompts. It exists but does nothing useful.

### Step 3: Register a Tool

A tool is a Python function that the LLM can invoke. The SDK automatically builds the JSON Schema from your type hints:

```python
@server.tool()
async def get_forecast(city: str, units: str = "celsius") -> str:
    """Get the weather forecast for a given city."""
    # In a real server, this would call an external API
    forecasts = {
        "tokyo": {"celsius": "18C, partly cloudy", "fahrenheit": "64F, partly cloudy"},
        "london": {"celsius": "12C, light rain", "fahrenheit": "54F, light rain"},
        "new york": {"celsius": "22C, sunny", "fahrenheit": "72F, sunny"},
    }
    city_key = city.lower().strip()
    data = forecasts.get(city_key)
    if not data:
        return f"No forecast available for {city}"
    return f"Forecast for {city}: {data[units]}"
```

Key details:
- The function must be `async def`.
- Type annotations generate the JSON Schema automatically.
- The docstring becomes the tool description that the LLM sees.
- Return values are serialized as JSON strings.

### Step 4: Register a Resource

A resource exposes data at a URI pattern. The LLM can read resources like files:

```python
@server.resource("weather://forecast/{city}")
async def forecast_resource(city: str) -> str:
    """Weather data for a city, accessible as a resource URI."""
    forecast = await get_forecast(city)
    return forecast
```

Resources are useful for data that the LLM should be able to "read" without calling a tool. The URI pattern `weather://forecast/{city}` defines a template. When the client requests `weather://forecast/tokyo`, the SDK extracts `city=tokyo` and calls your handler.

### Step 5: Transport Setup

The transport layer moves JSON-RPC messages between the client and your server. The default is `stdio` — the server reads from stdin and writes to stdout:

```python
if __name__ == "__main__":
    server.run(transport="stdio")
```

When you run `python server.py`, the server waits for JSON-RPC messages on stdin. Each message is a JSON object with `jsonrpc`, `method`, `id`, and `params` fields.

Example JSON-RPC request the client sends:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "get_forecast",
    "arguments": {"city": "Tokyo"}
  }
}
```

Example response the server sends back:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {"type": "text", "text": "Forecast for Tokyo: 18C, partly cloudy"}
    ]
  }
}
```

### Step 6: Running the Server

Start the server in one terminal:

```bash
python server.py
```

The server appears to hang — that is correct. It is listening for JSON-RPC messages on stdin. In development, you connect to it using the MCP Inspector or by configuring it as an MCP server in Claude Code.

### Step 7: Handling JSON-RPC Messages (Under the Hood)

The SDK handles this automatically, but understanding the flow helps when debugging:

1. Client sends a JSON-RPC `initialize` request.
2. Server responds with its capabilities (tools, resources, prompts).
3. Client sends `tools/list` to discover available tools.
4. Server responds with the tool list.
5. Client sends `tools/call` with the tool name and arguments.
6. Server invokes your function and returns the result.
7. Client sends `shutdown` when done.

Each message uses JSON-RPC 2.0 format. The SDK manages the request IDs, error codes, and response routing.

## Cost, Limits, and Safety

### Cost

Building and running a basic MCP server costs nothing beyond your development environment. The MCP SDK is open source. The server itself is a Python process running locally — no cloud infrastructure is required.

If you deploy the server on a remote host using SSE transport, you may incur hosting costs (a VM or container). These are typically minimal — an MCP server is not compute-intensive unless it wraps expensive operations.

API calls your server makes to third-party services (weather APIs, database queries, LLM calls) carry their own costs.

### Limits

The stdio transport binds the server to a single client process. If the client exits, the server terminates. For long-running or multi-client scenarios, use SSE transport.

The MCP protocol has a default message size limit. Very large tool outputs — megabytes of data — may need chunking or pagination.

Your server runs as the same user that started it. File system permissions, network access, and environment variables are inherited from the parent process.

### Safety

Input validation is essential. An LLM may send unexpected parameter values, including strings that could trigger injection attacks if your server forwards them unsanitized to a database or shell command.

Validate every parameter your tools receive. Do not construct shell commands from user-supplied strings. Use parameterized queries for databases. Treat tool arguments as untrusted input, even though they come from a language model rather than a human.

The server has access to whatever the parent process has access to. If Claude Code runs with your credentials, your server does too. Design your tools with the principle of least privilege: only expose operations that are safe for the LLM to trigger.

## When to Use / When Not To

### Build a Custom MCP Server When

- You have an internal API, database, or service that your AI assistant needs to query.
- You want to expose read-only data to the LLM (resources are ideal for this).
- You need custom logic before returning data (aggregation, formatting, caching).
- You want fine-grained control over which operations the LLM can perform.
- Your workflow involves multiple steps that an LLM should orchestrate through tool calls.

### Use an Existing MCP Server When

- A community server already exists for the service you need (databases, file systems, popular APIs).
- You need generic capabilities like reading files, searching the web, or querying SQL databases.
- You want to quickly evaluate whether MCP fits your workflow before committing to building.

### Use Python MCP SDK When

- Your team works in Python and you want minimal boilerplate.
- You need type-safe tool definitions with automatic schema generation.
- You want the FastMCP convenience layer for rapid prototyping.

### Consider TypeScript SDK When

- Your stack is Node.js/TypeScript and you want to keep the language consistent.
- You need to integrate with npm ecosystem tools.
- Your deployment runs on a Node.js runtime and you prefer not to add a Python dependency.

## Mental Model Check

**Q1: What problem does an MCP server solve?**

An MCP server bridges an AI assistant like Claude Code with external systems — APIs, databases, or internal services — that the assistant cannot reach on its own. It exposes those systems as tools and resources that the LLM can discover and invoke through a standard protocol.

**Q2: What is the difference between a tool and a resource?**

A tool is an action the LLM calls — it takes arguments and returns a result. Think of it as a function call. A resource is data the LLM reads at a URI — think of it as a file or an API endpoint that returns content. Tools are for operations; resources are for data access.

**Q3: How does the server communicate with the client?**

Over JSON-RPC, typically through standard input and output (stdio). The client writes a JSON-RPC request to the server's stdin. The server processes it and writes a JSON-RPC response to stdout. The SDK handles serialization and deserialization. For remote deployments, SSE (Server-Sent Events) transport is available.

**Q4: What happens when you run `server.run(transport="stdio")`?**

The server enters an event loop. It reads JSON-RPC messages from stdin one at a time. For each message, it checks the method field (`tools/list`, `tools/call`, `resources/read`, etc.), routes it to the appropriate handler, runs your Python function, and writes the JSON-RPC response to stdout. It continues until it receives a `shutdown` message.

**Q5: Why is input validation important in an MCP server?**

The LLM chooses which arguments to pass to your tools. It may pass unexpected values — empty strings, very long strings, special characters, or values outside the intended range. If your server forwards those arguments to a database query, a shell command, or a file system operation without validation, you risk SQL injection, command injection, or path traversal attacks. Validate every parameter as if it came from an untrusted source, because in the context of your backend systems, it effectively does.
