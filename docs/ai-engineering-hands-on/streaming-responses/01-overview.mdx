You build a chatbot that answers questions about your documentation. It works perfectly — you send a question, wait 4 seconds, and get the full answer printed. Your non-technical reviewer tries it and says, "It feels slow. Gmail's AI writes back to me as I type." You realize the problem: waiting 4 seconds for a complete response before seeing anything creates a perceptible delay. The user sees a blank screen wondering if the app is working. Streaming — showing tokens as they are generated, one by one — turns that 4-second wait into a 200-millisecond first-token experience. The user sees text appearing in real time, and the perceived speed improves dramatically.

This difference is not cosmetic. Streaming transforms the feel of an AI application from "ask and wait" to "converse and watch." Every major AI chat interface — ChatGPT, Claude.ai, GitHub Copilot — streams its responses. Non-streaming is the exception, used only when you need the complete response before proceeding (like structured output parsing or batch processing).

## The Concept

Streaming means the API sends the model's response as a sequence of chunks — each chunk containing one or more tokens — rather than waiting for the full generation to finish before sending anything. Think of it like watching a carpenter build a shelf versus going to the store to buy a pre-built one. Streaming lets you see each piece as it is added; non-streaming hides the process until the final product is ready.

## API Reference

### OpenAI Streaming

| Method / Parameter | What It Does | Example |
|---|---|---|
| `stream=True` in `client.chat.completions.create()` | Enables streaming on the request | `stream=True` |
| Response type with `stream=True` | Returns a `Stream[ChatCompletionChunk]` iterator instead of `ChatCompletion` | `for chunk in stream:` |
| `chunk.choices[0].delta.content` | The token text for this chunk (can be `None`) | `content = chunk.choices[0].delta.content` |
| `chunk.choices[0].delta.role` | The role field — sent once on the first chunk as `"assistant"` | `if chunk.choices[0].delta.role:` |
| `chunk.choices[0].finish_reason` | `"stop"`, `"length"`, or `None` (only set on final chunk) | `if chunk.choices[0].finish_reason:` |
| `chunk.choices[0].delta.tool_calls` | Tool call deltas for streaming tool use | `if chunk.choices[0].delta.tool_calls:` |
| `chunk.usage` | Token usage — only available on the final chunk (OpenAI) | `if chunk.usage: print(chunk.usage.total_tokens)` |
| `stream_options={"include_usage": True}` | Request usage data in the final streaming chunk | `stream_options={"include_usage": True}` |

### Anthropic Streaming

| Method / Parameter | What It Does | Example |
|---|---|---|
| `stream=True` in `client.messages.create()` | Enables streaming | `stream=True` |
| Response type with `stream=True` | Returns a `Stream[MessageStreamEvent]` iterator | `for event in stream:` |
| Event type `MessageStartEvent` | First event — contains the message metadata | `if event.type == "message_start":` |
| Event type `ContentBlockDeltaEvent` | Each content delta — the actual token text | `if event.type == "content_block_delta":` |
| `event.delta.text` | The token text in a `ContentBlockDeltaEvent` | `text = event.delta.text` |
| Event type `MessageStopEvent` | Final event — generation is complete | `if event.type == "message_stop":` |
| Event type `ContentBlockStartEvent` | Signals the start of a new content block (text or tool_use) | `if event.type == "content_block_start":` |

## How It Actually Works — Step by Step

### Step 1: The Client Opens a Server-Sent Events (SSE) Connection

When you set `stream=True`, the SDK does not wait for the full response. Instead, it opens an HTTP connection and keeps it open, reading the response body as a stream of Server-Sent Events (SSE). Each SSE event contains a JSON payload with one chunk of the response.

OpenAI sends events like:
```
data: {"id":"chatcmpl-...","object":"chat.completion.chunk","choices":[{"delta":{"content":"The"},"index":0}]}

data: {"id":"chatcmpl-...","choices":[{"delta":{"content":" capital"},"index":0}]}

data: [DONE]
```

Anthropic sends events in a different format:
```
event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"The"}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":" capital"}}

event: message_stop
data: {"type":"message_stop"}
```

### Step 2: Token Deltas Are Sent Incrementally

Each SSE event contains a "delta" — the difference from the previous state. Instead of sending the full accumulated text each time, the API sends only the new token(s) generated since the last event. Your code accumulates these deltas into the complete response.

```
Chunk 1: delta.content = "The"       → accumulated: "The"
Chunk 2: delta.content = " capital"  → accumulated: "The capital"
Chunk 3: delta.content = " of"       → accumulated: "The capital of"
Chunk 4: delta.content = " France"   → accumulated: "The capital of France"
Chunk 5: delta.content = " is"       → accumulated: "The capital of France is"
Chunk 6: delta.content = " Paris"    → accumulated: "The capital of France is Paris"
Chunk 7: delta.content = "."         → accumulated: "The capital of France is Paris."
Final:   finish_reason = "stop"      → streaming complete
```

### Step 3: Your Code Processes Each Chunk

With OpenAI, the streaming pattern looks like this:

```python
stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "What is the capital of France?"}],
    stream=True
)

for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        print(chunk.choices[0].delta.content, end="", flush=True)
```

Each `chunk` is a `ChatCompletionChunk` object. The `delta.content` field contains the new token text. Some chunks may have `delta.content = None` (for example, chunks that signal the start of a response or contain only metadata). Always check for `None` before using the value.

### Step 4: With Anthropic, Events Have Types

Anthropic's streaming API uses typed events rather than a unified chunk object:

```python
with client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=200,
    messages=[{"role": "user", "content": "What is the capital of France?"}],
    stream=True,
) as stream:
    for event in stream:
        if event.type == "content_block_delta":
            if event.delta.type == "text_delta":
                print(event.delta.text, end="", flush=True)
```

### Step 5: Accumulate the Full Response

In most applications, you want both the streaming display AND the complete response for storage, logging, or further processing. The pattern is to accumulate tokens during streaming:

```python
stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Tell me a short story."}],
    stream=True
)

collected = []
for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        content = chunk.choices[0].delta.content
        print(content, end="", flush=True)
        collected.append(content)

full_response = "".join(collected)
print(f"\n\n--- Complete response ({len(full_response)} chars) ---")
```

### Step 6: Token Usage with Streaming

Token usage is only available at the end of a stream. With OpenAI, you request it explicitly:

```python
stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Hi"}],
    stream=True,
    stream_options={"include_usage": True}
}

for chunk in stream:
    if chunk.choices[0].delta.content is not None:
        print(chunk.choices[0].delta.content, end="", flush=True)
    if chunk.usage:
        print(f"\n\nPrompt tokens: {chunk.usage.prompt_tokens}")
        print(f"Completion tokens: {chunk.usage.completion_tokens}")
        print(f"Total tokens: {chunk.usage.total_tokens}")
```

With Anthropic, usage comes in the `message_start` event:

```python
for event in stream:
    if event.type == "message_start":
        print(f"Input tokens: {event.message.usage.input_tokens}")
    elif event.type == "content_block_delta":
        if event.delta.type == "text_delta":
            print(event.delta.text, end="", flush=True)
```

## Cost, Limits, and Safety

### Cost Implications

Streaming does not change the total token cost. Whether you stream or not, you pay for the same tokens. The difference is purely in how the response is delivered. However, streaming can save costs in edge cases:

- **Early termination:** If you stream a response and detect a condition that means you do not need the rest (e.g., the first sentence already answers your question), you can close the connection early and stop billing for remaining tokens.
- **Timeouts:** With long generations, streaming lets you detect slow responses and cancel them before they reach `max_tokens`.

### Rate Limits

Streaming connections count toward the same rate limits as non-streaming. Each streaming request counts as one request, not one request per chunk. OpenAI and Anthropic both count streaming requests at the same rate as non-streaming.

### Safety Rules

**Always set a timeout on the streaming connection.** A stream that hangs indefinitely keeps your connection open. Set a reasonable timeout:

```python
client = OpenAI(
    api_key=os.environ["OPENAI_API_KEY"],
    timeout=30.0  # seconds
)
```

**Do not assume chunks arrive in order in all edge cases.** While streaming maintains token order within a single generation, if you are processing multiple streams concurrently, you may need to tag responses.

**Handle mid-stream disconnections.** If the network drops mid-stream, your code should handle the exception and resume or retry:

```python
try:
    for chunk in stream:
        process(chunk)
except APIConnectionError:
    print("Stream disconnected. Retrying...")
```

**Accumulate for logging, not just display.** If you only print tokens without accumulating them, you lose the full response for logging, moderation checks, or later analysis. Always collect the tokens as they arrive.

## When to Use This / When Not To

**Use streaming when:**
- Building any interactive chat interface where users expect real-time feedback
- The first-token latency matters more than total generation time
- You want to show progress indicators (typing dots, partial text)
- You need to display partial results while generation continues

**Do not use streaming when:**
- You need the complete response before doing anything (e.g., parsing structured output, validating against a schema)
- You are processing responses in batch (no user watching)
- Your downstream pipeline requires the full output before it can proceed
- The connection is unreliable — non-streaming with retries is more robust

## Mental Model Check

**Question 1:** What does `stream=True` actually change about the HTTP connection?

  It keeps the connection open and sends response chunks as Server-Sent Events (SSE) instead of waiting for the full response to be ready before closing the connection.

---

**Question 2:** Why does streaming improve perceived performance even though total generation time is the same?

  Because the user sees the first token in 200–500ms instead of waiting 4+ seconds for the complete response. The brain interprets immediate visual feedback as "it is working" rather than "it is broken."

---

**Question 3:** What is a "delta" in the context of streaming?

  The delta is the incremental difference — the new token(s) generated since the last chunk. Your code accumulates deltas to build the complete response text.

---

**Question 4:** How is token usage reported differently in streaming vs non-streaming?

  In non-streaming, usage is returned in the response object directly. In streaming, usage is either in the final chunk (OpenAI needs `stream_options={"include_usage": True}`) or in the initial `message_start` event (Anthropic). You must explicitly capture it.

---

**Question 5:** What happens if the network disconnects mid-stream?

  The stream raises a connection error (`APIConnectionError` in both SDKs). Your code should handle this exception with retry logic or graceful degradation — the partial text accumulated so far is still valid, but you may want to restart the request.
