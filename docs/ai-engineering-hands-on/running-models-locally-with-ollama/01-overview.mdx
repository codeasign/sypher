## The Problem Story

Sarah is building an AI-powered code review bot for her company. Every time a developer pushes a pull request, her bot sends the diff to a cloud LLM API. The pattern works, but the pain is mounting.

**API costs are unpredictable.** Each code review consumes between 2,000 and 8,000 tokens depending on the diff size. With 50 pull requests per day across the team, Sarah is spending roughly $150 per month on a single integration. Multiply that by the five AI features her team plans to build, and the cloud bill hits $750 per month before anyone has shipped a production product.

**Latency stings on every request.** Every API call requires a round trip across the internet. Even with fast providers, the 95th percentile response time sits at 4.7 seconds. For interactive use cases like chatting with documentation or iterating on prompt templates, those seconds add up to a sluggish experience.

**Privacy concerns create blockers.** Sarah company handles proprietary source code. The legal team requires a data processing agreement for every external API. Two of the five planned features involve sensitive customer data that cannot be sent to any third-party service at all.

**The offline gap is real.** During a recent internet outage, every AI feature in the application went dark. No fallback, no degraded mode, nothing.

Sarah needs a way to run models locally: on her laptop during development, and on internal servers for production features that handle sensitive data.

## The Concept

Ollama is a free, open-source tool that lets you download and run large language models on your own hardware. It wraps popular open-weight models into a simple command-line interface and a local REST API server.

The core idea is straightforward: instead of sending your prompts to a cloud provider over the internet, you run the model on your own machine. The model weights live on your disk. Inference happens in your CPU or GPU. Your data never leaves your computer.

Ollama handles the operational complexity behind the scenes:

- **Model management**: downloading, versioning, and storing models locally
- **Inference engine**: optimized inference using llama.cpp under the hood, with GPU acceleration where available
- **REST API**: a local server that exposes OpenAI-compatible endpoints
- **Model customization**: a Modelfile format for creating custom variants with system prompts, parameters, and templates

Available models span from small 1.5B parameter models that run on a Raspberry Pi to 70B+ parameter models that require a workstation. The Ollama library at ollama.ai hosts hundreds of models, including Llama, Gemma, Mistral, Qwen, Phi, and DeepSeek families.

## API / SDK / CLI Reference

### 1. ollama pull -- Download a Model

Downloads a model from the Ollama library to your local machine. Tags are optional and default to latest.

```bash
ollama pull gemma3:2b
ollama pull llama3.2:3b
ollama pull mistral:7b
```

The model is stored in the ~/.ollama/models/blobs/ directory by default. The download is resumable. If interrupted, running pull again picks up where it left off.

### 2. ollama run -- Start an Interactive Session

Starts an interactive chat session with a model. The model is loaded into memory, and you can type messages back and forth.

```bash
ollama run gemma3:2b
```

Inside the session, type prompts directly. Use /bye to exit, /help for available commands, and /set to adjust parameters.

For a single-shot prompt without entering interactive mode:

```bash
ollama run gemma3:2b "What is the capital of France?"
```

Useful flags:

| Flag | Purpose |
|------|---------|
| --verbose | Show timing and performance statistics |
| --format json | Request JSON-formatted output |
| --keepalive 1h | Keep the model loaded in memory for one hour |
| --nowordwrap | Disable word wrapping in output |

### 3. ollama list -- Show Downloaded Models

Lists all models currently stored on disk with their size and last-modified time.

```bash
ollama list
```

Output resembles:

```
NAME                   ID              SIZE      MODIFIED
gemma3:2b              5f1c7e3a8b2d    1.7 GB    2 days ago
llama3.2:3b            9f0b6a1c4d3e    2.3 GB    5 hours ago
mistral:7b             a2c8d4e6f0b1    4.9 GB    3 weeks ago
```

### 4. ollama serve -- Start the Local Server

Starts the Ollama HTTP server on localhost port 11434. On macOS and Linux, Ollama typically runs as a background service. On Windows, the Ollama app starts the server automatically.

```bash
ollama serve
```

You can configure the server with environment variables:

| Variable | Purpose |
|----------|---------|
| OLLAMA_HOST | Change the bind address (default: 127.0.0.1:11434) |
| OLLAMA_MODELS | Custom model storage directory |
| OLLAMA_KEEP_ALIVE | How long to keep models in memory (default: 5m) |
| OLLAMA_NUM_PARALLEL | Number of parallel requests to process |
| OLLAMA_MAX_LOADED_MODELS | Maximum models loaded simultaneously |

### 5. Python SDK -- ollama Package

The official Python package provides a synchronous Client and an asynchronous AsyncClient.

```python
from ollama import chat

response = chat(
    model='gemma3:2b',
    messages=[
        {'role': 'user', 'content': 'Explain recursion in one sentence.'}
    ]
)
print(response.message.content)
```

Streaming response:

```python
from ollama import chat

stream = chat(
    model='gemma3:2b',
    messages=[{'role': 'user', 'content': 'Count from 1 to 5.'}],
    stream=True
)

for chunk in stream:
    print(chunk['message']['content'], end='', flush=True)
```

Async client:

```python
import asyncio
from ollama import AsyncClient

async def main():
    client = AsyncClient()
    response = await client.chat(
        model='gemma3:2b',
        messages=[{'role': 'user', 'content': 'Hello'}]
    )
    print(response.message.content)

asyncio.run(main())
```

Key methods on the Client class:

| Method | Description |
|--------|-------------|
| chat() | Multi-turn conversation with messages array |
| generate() | Single-turn text generation from a prompt |
| embed() | Generate embeddings for text |
| pull() | Download a model from the registry |
| list() | List locally available models |
| show() | Show model metadata and configuration |
| delete() | Remove a model from disk |
| create() | Create a custom model from a Modelfile |
| ps() | Show models currently loaded in memory |

Error handling:

```python
from ollama import chat, ResponseError

try:
    response = chat(model='nonexistent-model', messages=[])
except ResponseError as e:
    print(f'Error {e.status_code}: {e.error}')
```

### 6. REST API Endpoint

Ollama exposes a REST API at http://localhost:11434/. The key endpoints are:

| Method | Path | Purpose |
|--------|------|---------|
| POST | /api/generate | Generate text from a prompt |
| POST | /api/chat | Multi-turn chat completion |
| POST | /api/embed | Create embeddings |
| GET | /api/tags | List local models |
| POST | /api/pull | Download a model |
| POST | /api/create | Create a model from Modelfile |

Example using curl:

```bash
curl -X POST http://localhost:11434/api/chat \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma3:2b",
    "messages": [
      {"role": "user", "content": "What is the speed of light?"}
    ],
    "stream": false
  }'
```

### 7. OpenAI-Compatible Endpoint

Ollama provides an OpenAI-compatible API at the /v1/ path. This lets you use any OpenAI SDK with Ollama by changing the base URL.

```python
from openai import OpenAI

client = OpenAI(
    base_url='http://localhost:11434/v1/',
    api_key='ollama'
)

response = client.chat.completions.create(
    model='gemma3:2b',
    messages=[{'role': 'user', 'content': 'What is machine learning?'}]
)
print(response.choices[0].message.content)
```

Supported endpoints:

| Endpoint | Purpose |
|----------|---------|
| POST /v1/chat/completions | Chat completions |
| POST /v1/completions | Text completions |
| POST /v1/embeddings | Embeddings |
| GET /v1/models | List available models |

### 8. Modelfile

A Modelfile is a configuration file for creating custom models. It defines the base model, system prompt, inference parameters, and template.

```modelfile
FROM gemma3:2b

PARAMETER temperature 0.7
PARAMETER num_ctx 4096
PARAMETER top_p 0.9

SYSTEM You are a helpful coding assistant that writes clean Python code.

TEMPLATE """{{ if .System }}<|start_header_id|>system<|end_header_id|>

{{ .System }}<|eot_id|>{{ end }}{{ if .Prompt }}<|start_header_id|>user<|end_header_id|>

{{ .Prompt }}<|eot_id|>{{ end }}<|start_header_id|>assistant<|end_header_id|>
"""
```

Create the custom model:

```bash
ollama create my-coding-assistant -f ./Modelfile
```

Then run it:

```bash
ollama run my-coding-assistant
```

Common PARAMETER directives:

| Parameter | Default | Description |
|-----------|---------|-------------|
| temperature | 0.8 | Higher values increase creativity |
| num_ctx | 2048 | Context window size in tokens |
| top_p | 0.9 | Nucleus sampling threshold |
| repeat_penalty | 1.1 | Penalty for repeated tokens |
| seed | 0 | Random seed for reproducibility |
| num_predict | -1 | Maximum tokens to generate |
| stop | (none) | Stop sequences as strings |

## How It Works -- Step by Step

### Step 1: Install Ollama

Ollama is distributed as a single installer for each platform. Download from ollama.ai and run the installer. The server registers itself as a background service that starts automatically.

On Linux, it is also available via package managers:

```bash
curl -fsSL https://ollama.ai/install.sh | sh
```

On macOS, the desktop app places an Ollama icon in the menu bar. On Windows, it runs as a system tray application.

Verification:

```bash
ollama --version
```

### Step 2: Pull a Model

The Ollama server does not ship with any model pre-installed. You must download at least one model before you can run inference.

Model identifiers follow the format family:tag. The latest tag is the default if you omit the tag.

```bash
ollama pull gemma3:2b
```

The download progress shows:

```
pulling manifest
pulling 5f1c7e3a8b2d... 100%
pulling 9f0b6a1c4d3e... 100%
verifying sha256 digest
writing manifest
success
```

### Step 3: Run the Model via CLI

Once the model is downloaded, you can interact with it directly from the terminal.

```bash
ollama run gemma3:2b
```

The prompt changes to indicate the model is ready:

```
>>>
```

Type a message:

```
>>> What is the difference between a list and a tuple in Python?
```

The model responds with generated text. The session remains open for follow-up messages, maintaining conversation context until you type /bye to exit.

1. The request is parsed and validated against the model schema
2. The input text is tokenized into token IDs using the model vocabulary
3. The token IDs are fed into the neural network (llama.cpp engine)

5. The output tokens are detokenized back into human-readable text
6. The response is streamed back to the client token by token or collected and returned as a single block

### Step 5: Use the Python SDK

For programmatic access, the Python SDK wraps the REST API into a clean interface.

```python
from ollama import chat

response = chat(
    model='gemma3:2b',
    messages=[
        {'role': 'user', 'content': 'Write a Python function to check if a string is a palindrome.'}
    ]
)
print(response.message.content)
```

The SDK automatically handles the HTTP connection, request serialization, and response parsing. The chat method accepts a list of message dictionaries with role and content keys, mirroring the OpenAI API format.

### Step 6: Use the OpenAI-Compatible Endpoint

Many existing tools and libraries are built to work with the OpenAI API. Ollama OpenAI-compatible endpoint lets you use those tools with zero code changes.

```python
from openai import OpenAI

client = OpenAI(
    base_url='http://localhost:11434/v1/',
    api_key='ollama'
)

response = client.chat.completions.create(
    model='gemma3:2b',
    messages=[
        {'role': 'system', 'content': 'You are a terse assistant.'},
        {'role': 'user', 'content': 'What is 2 + 2?'}
    ]
)
print(response.choices[0].message.content)
```

This compatibility means frameworks like LangChain, LlamaIndex, and AutoGen can use Ollama as a drop-in model provider.

## Cost, Limits, and Safety

### Hardware Requirements

Running models locally requires adequate hardware. The requirements scale with model size.

**RAM / VRAM estimates by model size (Q4 quantization):**

| Model Size | Memory Required | Example Models |
|------------|----------------|----------------|
| 2-3B parameters | 2-4 GB | Gemma 3 2B, Phi-3 mini |
| 7-8B parameters | 6-8 GB | Llama 3.2 3B, Mistral 7B |
| 13-14B parameters | 10-12 GB | Phi-4, Qwen 3 14B |
| 27-34B parameters | 18-24 GB | Gemma 3 27B, Qwen 3 32B |
| 70B parameters | 40-48 GB | Llama 3.1 70B |

A general formula: memory needed is approximately 0.6 GB per billion parameters at Q4 quantization, plus overhead for context.

**Disk space:**

| Model | Disk Space (Q4) |
|-------|-----------------|
| 2-3B model | 1.5-2 GB |
| 7-8B model | 4-5 GB |
| 70B model | ~40 GB |

An SSD is strongly recommended. Loading a large model from a mechanical drive can take minutes.

**CPU-only performance:** A 7B model at Q4 on a modern 8-core CPU produces roughly 5-15 tokens per second. The same model on an NVIDIA RTX 4090 achieves 40-80 tokens per second.

### Cost Comparison

| Factor | Local (Ollama) | Cloud API |
|--------|---------------|-----------|
| Per-token cost | $0 (electricity only) | $0.15 to $3.00 per million tokens |
| Hardware cost | One-time (existing or new) | $0 |
| Latency (p95) | 0.5-2 seconds | 3-8 seconds |
| Scaling cost | Buy more hardware | Pay per request forever |

The trade-off is clear: local inference has zero marginal cost per request but requires upfront hardware investment. For high-volume usage, local inference is dramatically cheaper.

### Privacy and Safety

**No data leaves your machine.** This is the most important safety property of local inference. When you run Ollama, your prompts, model inputs, and generated outputs stay entirely within your computer. There is no telemetry, no data collection, and no external API call.

**Network isolation.** You can run Ollama on a machine that has no internet connection at all, once the model is downloaded. This makes it suitable for air-gapped environments.

**Model security.** Open-weight models may have different safety alignment than commercial APIs. Test your chosen model for harmful outputs before deploying it in user-facing applications. Use a dedicated machine or container for production workloads.

**Resource limits.** A runaway generation loop or a very long prompt can exhaust system memory. Set num_predict to limit generation length, and monitor memory usage with ollama ps.

## When to Use

**Development and prototyping.** Iterate on prompts, test model behavior, and build application logic without incurring API costs. Local inference lets you experiment freely.

**Privacy-sensitive workloads.** Any application that processes proprietary code, personal data, or confidential documents benefits from keeping data on-premises.

**Offline and air-gapped environments.** Field deployments, secure facilities, and remote locations without internet access can still run AI features.

**High-volume or latency-sensitive applications.** When you need many short queries with predictable response times, local inference eliminates network variability.

**Cost-sensitive production.** If your application processes millions of requests per month, the marginal cost of local inference approaches zero.

## When Not to Use

**Larger models than your hardware supports.** If your use case requires a 70B model and you only have 16 GB of RAM, cloud inference is the only option.

**Multi-modal models with proprietary weights.** Some state-of-the-art multimodal models are not available as open weights. You may need cloud APIs for these.

**Rapidly changing model landscape.** If you need the latest model within hours of release, cloud providers typically offer new models faster than local downloads.

**Bursty workloads with unpredictable scale.** If your traffic spikes from 0 to 10,000 requests per minute, a cloud API can scale instantly. Local hardware has a fixed capacity.

**Team collaboration on shared models.** If multiple team members need to share the same model configuration, a cloud deployment with a single endpoint is simpler to manage than distributing model files to each machine.

## Mental Model Check

**Question 1:** You have a laptop with 16 GB of RAM and no dedicated GPU. Your team needs to run a 7B parameter model for a code review assistant. Is this feasible?

**Answer:** Yes, but with constraints. A 7B model at Q4 quantization needs roughly 6-8 GB of RAM, which fits within 16 GB total. CPU-only inference will be slower (5-15 tokens/sec) but usable for batch processing where latency is not critical. For interactive use, consider a smaller model like Gemma 3 2B or Phi-3 mini.

---

**Question 2:** A developer asks: "I need my application to work offline. Can I use Ollama?"

**Answer:** Yes, once the model is downloaded, the Ollama server runs entirely offline with no internet connectivity required. This is one of the primary advantages of local inference. You can even pre-download models on a connected machine and transfer them to an air-gapped system.

---

**Question 3:** A security engineer is concerned about data leakage. When using Ollama, does any prompt data leave the machine?

**Answer:** No. All inference happens locally on your machine. No prompt data, model inputs, or generated outputs are sent to any external service. The only network communication occurs when you explicitly run ollama pull to download a model from the registry.

---

**Question 4:** Your team is building a chatbot for customer support. The traffic pattern is unpredictable, with 10 requests on some days and 10,000 on others. Should you use Ollama for this?

**Answer:** Probably not. Local hardware has a fixed maximum throughput. For bursty workloads with orders-of-magnitude traffic variation, a cloud API that scales on demand is more appropriate. Ollama is better suited for predictable, steady-state workloads or for development and prototyping.

---

**Question 5:** You want to use LangChain with Ollama. Do you need to rewrite your existing OpenAI integration?

**Answer:** No. Ollama OpenAI-compatible endpoint at /v1/ allows you to use the OpenAI SDK with any LangChain component by changing the base_url. The same code that calls OpenAI can call Ollama with a single configuration change.

## Additional CLI Commands

### ollama ps -- Show Running Models

Displays models currently loaded in memory. This is useful for monitoring resource usage.

```bash
ollama ps
```

Output:

```
NAME            ID              SIZE    PROCESSOR       UNTIL
gemma3:2b       5f1c7e3a8b2d   1.7 GB  100% CPU        4 minutes from now
```

### ollama stop -- Unload a Model

Removes a model from memory without stopping the server.

```bash
ollama stop gemma3:2b
```

### ollama rm -- Delete a Model

Removes a model from disk entirely. This frees up storage space.

```bash
ollama rm gemma3:2b
```

### ollama show -- Inspect a Model

Displays model metadata including parameters, template, license, and system prompt.

```bash
ollama show gemma3:2b
```

### ollama cp -- Copy a Model

Creates a copy of a model under a new name. Useful for creating custom variants.

```bash
ollama cp gemma3:2b my-copy
```

Ollama supports several environment variables for advanced configuration.

| Variable | Default | Description |

| OLLAMA_MODELS | (platform-specific) | Directory for model storage |

```powershell
$env:OLLAMA_KEEP_ALIVE = "1h"
ollama serve
```

```bash
export OLLAMA_KEEP_ALIVE=1h
ollama serve
```

## Model Selection Guide

Choosing the right model for your hardware and use case is critical. Here is a practical guide.

### By Hardware Capability

**8 GB RAM or less:** Use 2-3B parameter models. The Gemma 3 2B and Phi-3 mini are good choices. Expect 8-15 tokens per second on CPU.

**16 GB RAM:** 7-8B parameter models work well. Llama 3.2 3B is very fast, while Mistral 7B provides better quality at slower speeds. You can run one 7B model comfortably.

**32 GB RAM:** You can run 13-14B parameter models like Phi-4 or Qwen 3 14B. Multiple smaller models can be loaded simultaneously.

**64 GB RAM or GPU with 24 GB VRAM:** 27-34B models like Gemma 3 27B or Qwen 3 32B are feasible. At this tier, model quality approaches cloud API levels.

### By Use Case

**Code generation:** Models fine-tuned for code (Llama 3.1 8B, Qwen 2.5 Coder 7B) produce better results than general-purpose models of the same size.

**Chat and conversation:** Gemma 3 and Mistral families excel at instruction following and dialogue.

**Summarization and extraction:** Smaller models like Phi-3 mini are surprisingly capable at structured extraction tasks due to their training.

**Creative writing:** Higher temperature settings (0.8-1.2) with models like Llama 3.2 produce more varied and creative output.

## Performance Optimization Tips

### 1. Use the Right Quantization

By default, Ollama uses Q4 (4-bit) quantization. This offers a good balance of quality and speed. Some models are available in multiple quantization levels.

```bash
ollama pull llama3.2:3b-q8_0
```

Higher quantization (Q8, Q6) preserves more quality but uses more memory and is slower. Lower quantization (Q3, Q2) uses less memory but degrades quality.

### 2. Adjust Context Window

The default context window is 2048 tokens. For longer conversations, increase it:

```bash
ollama run gemma3:2b --num-ctx 8192
```

Or in a Modelfile:

```modelfile
PARAMETER num_ctx 8192
```

Larger context windows use significantly more memory. A context window of 8192 tokens roughly doubles memory usage compared to 2048.

### 3. Use Batch Processing

For bulk processing, send prompts in batches rather than one at a time. The server handles queuing, and the model stays loaded between requests.

```python
prompts = [
    "Summarize: ...",
    "Summarize: ...",
    "Summarize: ...",
]

for prompt in prompts:
    response = chat(model='gemma3:2b', messages=[
        {'role': 'user', 'content': prompt}
    ])
    print(response.message.content)
```

### 4. Keep Models Loaded

The keepalive parameter controls how long a model stays in memory after the last request. Increase this for interactive use:

```bash
ollama run gemma3:2b --keepalive 30m
```

This prevents the model from being unloaded and reloaded between requests, which can take several seconds.

## Security Considerations

### Running as a Service

When running Ollama as a production service, follow these security practices:

- Bind to localhost only (127.0.0.1) to prevent external access
- Use a reverse proxy with authentication for remote access
- Set resource limits to prevent runaway processes from exhausting system memory
- Run Ollama in a container or dedicated user account with minimal permissions

### Model Provenance

Only download models from trusted sources. The official Ollama library at ollama.ai curates models from verified publishers. Community models may contain unexpected behavior or biases.

### Input Validation

Even local models can produce harmful output if prompted maliciously. Implement input validation and output filtering for user-facing applications, just as you would with a cloud API.

### Data Isolation

When running multiple tenants or applications on the same machine, ensure that prompt data is isolated. Ollama does not provide built-in multi-tenancy. Use separate processes, containers, or virtual machines for different security domains.
