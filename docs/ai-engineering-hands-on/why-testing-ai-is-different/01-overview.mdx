Your CI pipeline just turned green. Every unit test passed, every integration test passed, every lint rule was satisfied. You merge to production. Twenty minutes later, your AI chatbot tells a customer that your company's refund policy is "flexible" — which is false, because the policy explicitly says no refunds after 30 days. The test suite didn't catch it because the output was grammatically correct, well-structured, and on-topic. It just happened to be wrong.

This is the core challenge of testing AI systems. In traditional software, a test asserts that `add(2, 2)` returns exactly `4`. If it returns `5`, the test fails. But when you ask an LLM "What is your refund policy?", there is no single correct string to assert against. The model might answer in Spanish, in bullet points, with a polite opening, or as a limerick — and any of those could be acceptable. How do you write a test for something that can be correct in infinite ways and wrong in infinite more?

## The Concept

**Testing AI systems means shifting from assertion-based testing (exact match) to evaluation-based testing (measurement against criteria).** Think of it like grading a student essay instead of grading a multiple-choice exam. In a multiple-choice exam, there is exactly one right answer per question — that is traditional software testing. An essay can be well-argued, factually correct, and properly structured using different words and sentence arrangements — that is AI evaluation. You evaluate along dimensions (relevance, factual accuracy, tone, safety) rather than asserting exact equality.

## How It Actually Works — Step by Step

### Step 1: Define Evaluation Dimensions

Before you write a single test, you define what "good" looks like for your use case. For a customer support AI, you might care about correctness, tone, and whether the agent stays within policy. For a code generator, you care about syntactic validity, security, and whether the code compiles. Each dimension becomes a metric you will measure.

### Step 2: Collect a Golden Dataset

You gather a set of representative inputs and — unlike traditional testing — you may or may not have a single correct output for each. Instead, you have a rubric. For example, input `"What is your return policy?"` might have a rubric stating: "Must mention 30-day window, must not promise exceptions, should be polite."

### Step 3: Run Inference and Capture Outputs

You pass each golden input through your system and capture the raw output. For production-grade evaluation, you run each input multiple times (e.g., 3-5 calls at temperature 0.7) to get a distribution of outputs rather than a single datapoint.

### Step 4: Score Against Metrics

This is where AI testing diverges most sharply from traditional testing. You do not compare strings — you compute scores:

- **Embedding similarity:** Encode both the expected answer and the actual answer into vectors, compute cosine similarity.
- **LLM-as-judge:** Feed the input, output, and a rubric to a judge LLM (often a different model like GPT-4 evaluating a Llama-3 response) and ask for a score.
- **Programmatic checks:** Regex, JSON schema validation, length constraints, keyword presence.

### Step 5: Aggregate Scores Across the Dataset

A single test does not pass or fail — it contributes to a distribution. You look at the mean score, the variance, the percentile of low-scoring outputs. A regression is a statistically significant drop in the mean score, not a single failing assertion.

### Step 6: Set Thresholds and Alert

You set thresholds per metric: "Answer relevance must average above 0.85 across the golden set." If a new model version drops relevance to 0.82, your CI should flag it. But unlike a binary test, you can also monitor trends over time — is the model getting worse at following instructions as the conversation context grows?

## Cost, Limits, and Safety

**Token cost of evaluation:** Every LLM-as-judge call costs tokens. If you evaluate each of 100 prompts against 4 dimensions using a judge LLM, that is 400 API calls on top of your 100 inference calls. Budget for evaluation as a first-class cost — it can easily double or triple your total LLM API spend. For cost-sensitive pipelines, use a smaller judge model (e.g., Claude Haiku or GPT-4o-mini) and only escalate expensive judges (Claude Sonnet, GPT-4) for low-scoring samples.

**Rate limits:** Evaluation pipelines can trigger rate limits because they batch many LLM calls in quick succession. Build in exponential backoff and concurrency limits (e.g., 5 concurrent judge calls at a time).

**Safety evaluation:** Always evaluate for harmlessness and safety separately from quality metrics. A response can be factually correct and perfectly relevant but still contain harmful content. Use dedicated safety classifiers (e.g., Liama Guard, Azure Content Safety) as a distinct evaluation pass.

**Variance:** An LLM judge is itself non-deterministic. Run each evaluation call 2-3 times and average the scores. A single judge score is a data point, not a verdict.

**Data privacy:** If you send customer conversations to an external LLM for evaluation, you are exposing that data to a third party. Ensure your evaluation pipeline respects data residency and privacy requirements — use a local judge model (Ollama with Llama 3) for sensitive data.

## When to Use This / When Not To

**Use evaluation-based testing when:** You are validating LLM outputs against subjective or multi-dimensional criteria. Any system where a human would say "this answer is good" or "this answer is bad" based on multiple factors is a candidate for evaluation-based testing. This covers most LLM applications: chatbots, summarizers, content generators, code assistants.

**Do not use evaluation-based testing when:** You can write a deterministic assertion. If you are checking that an LLM output is valid JSON, assert on `json.loads()`. If you are checking that a number is in a range, assert numeric bounds. Use evaluation for what evaluation is good at — semantic quality — and assertions for what assertions are good at — structural correctness.

**Use evaluation-based testing alongside, not instead of, traditional tests.** Your CI for an AI system should have three layers:

1. **Traditional unit/integration tests** for the non-AI code (API routes, database queries, prompt template construction)
2. **Evaluation tests** for LLM output quality (relevance, factuality, tone, safety)
3. **Monitoring** in production (drift detection, user feedback correlation)

## Mental Model Check

**Q1:** Why can't you use exact string matching to test an LLM's response to "Write a poem about Python"?

**A1:** The same prompt with the same settings can produce different poems: different rhymes, different meters, different metaphors. All could be good poems. Asserting an exact string would flag all correct outputs as failures.

**Q2:** What is the minimum number of times you should evaluate a single prompt-output pair with an LLM judge?

**A2:** At least 2-3 times, because the judge LLM is itself non-deterministic. Averaging multiple judge calls gives a more reliable score than any single call.

**Q3:** If your evaluation pipeline scores a model at 0.82 average relevance this week and 0.79 next week, is that a regression?

**A3:** Not necessarily — the difference could be within the noise of the evaluation process itself. You need to track the variance of your measurements and use statistical significance (e.g., a t-test) before declaring a regression.

**Q4:** What is one situation where a traditional assertion is better than an AI evaluation?

**A4:** Checking that an LLM output is valid JSON, fits within a character limit, or contains all required sections (headers, intro, body) — these are structural checks where deterministic assertions are faster, cheaper, and more reliable than LLM evaluation.

**Q5:** If you have 200 test prompts and evaluate each on 3 dimensions using an LLM judge, how many API calls does the evaluation pipeline make (excluding inference)?

**A5:** At minimum, 600 judge calls (200 prompts x 3 dimensions). If you run each judge call 3 times for reliability, that is 1,800 calls.
