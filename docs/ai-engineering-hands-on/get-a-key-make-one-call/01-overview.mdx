You have read about LLMs and APIs. You understand the concept: send a prompt to a server, get a response back. But when you sit down to actually do it, you run into a wall of practical questions. Where do you get an API key? What does one look like? How do you give it to your code without typing it directly into the file where anyone could see it? What happens when you run the code for the first time — will it actually work, or will you get a cryptic error that you cannot decode?

The gap between understanding the idea and seeing it return real output is the hardest part of starting with AI engineering. Concepts are abstract. Keys are concrete. A missing character in your key string, a missing import statement, a missing `load_dotenv()` call — any of these can turn a three-line script into fifteen minutes of debugging. Every experienced AI engineer has a story about the first time they accidentally committed an API key to a public repository and woke up to a surprise bill.

This page bridges that gap. By the end, you will have seen every step from signing up for an account to printing a model response in your terminal. You will know what an API key looks like, how to keep it safe, how to set up your project so keys stay out of version control, and what the very first lines of code feel like when they work. You will also understand what happens inside the SDK and on the server during each part of the call.

## The Concept

An API key is a secret token that identifies you to the service provider. Think of it like a ticket you show at the door before entering a venue. The venue (the API server) checks your ticket, confirms it is valid, and then lets you in to make requests. Without the ticket, the door stays locked.

Every LLM provider issues API keys through a web console. When you create an account and generate a key, the provider stores a cryptographic hash of that key. When your code sends the key with a request, the server hashes it and compares it against stored hashes. If they match, your request is authenticated. The provider never stores the raw key on their servers — they store only the hash. This means that if their database were ever breached, the raw keys would not be exposed.

Making your first API call means writing a Python script that does three things: reads your key from a safe place (an environment variable), sends a prompt to the provider, and prints the response that comes back. That is it. Three actions. If you can do those three things, you can build anything on top of an LLM — chatbots, agents, RAG systems, code assistants, summarizers. Every advanced application starts with the same foundation.

## API Reference

The table below covers the essential methods, objects, and environment concepts you need to make your first call with the Anthropic Python SDK. These are the building blocks you will use in every subsequent lesson and project.

| Method / Object / Concept | What It Does | Example |
|---|---|---|
| `os.environ["ANTHROPIC_API_KEY"]` | Reads your API key from an environment variable so it never appears in source code | `api_key = os.environ["ANTHROPIC_API_KEY"]` |
| `anthropic.Anthropic(api_key)` | Creates an authenticated client that talks to the Anthropic API server | `client = anthropic.Anthropic(api_key=api_key)` |
| `client.messages.create(...)` | Sends a message to the model and returns the full response once generation completes | `response = client.messages.create(model="...", max_tokens=100, messages=[...])` |
| `model` parameter | Specifies which LLM to use; different models have different speed, cost, and capability | `model="claude-sonnet-4-20250514"` |
| `messages` parameter | A list of conversation turns, each with a `role` (user or assistant) and `content` (the text) | `messages=[{"role": "user", "content": "Say hello"}]` |
| `max_tokens` parameter | Limits how many tokens the model can generate in its response — controls both cost and response length | `max_tokens=200` |
| `response.content` | Accesses the model's generated text from the API response object | `print(response.content[0].text)` |
| `response.usage` | Shows how many tokens were used for input and output, which determines the cost of the call | `print(response.usage.input_tokens, response.usage.output_tokens)` |
| `pip install anthropic` | Installs the Anthropic Python SDK so your code can call the API from Python | `pip install anthropic` |
| `.env` file | A file that stores environment variables (API keys) locally and is never committed to version control | `ANTHROPIC_API_KEY=sk-ant-...` in `.env` |
| `load_dotenv()` | Loads variables from a `.env` file into `os.environ` so your code can read them | `load_dotenv()` after `from dotenv import load_dotenv` |
| `stream=True` parameter | Enables streaming mode where the response arrives token by token instead of all at once | `with client.messages.stream(...) as stream:` |

## How It Actually Works — Step by Step

Each step below covers one layer of what happens when you get a key and make your first call. Read them in order to build a complete mental model of the full system.

### Step 1: You Create an Account and Generate a Key

Every LLM provider — Anthropic, OpenAI, Google, Mistral — has a web console where you sign up and create API keys. The sign-up process normally requires an email address and phone verification. This is a one-time step that takes about five minutes.

Once logged in, you navigate to the API Keys section and click a button to create a new key. Different providers call this something slightly different — "Create Key", "Generate Key", or "New Secret Key" — but they all do the same thing. They generate a cryptographically random string and associate it with your account in their database.

The key is a long random string that the provider uses to identify you. When you make a request, the server checks this key to identify your account, enforce your rate limits, and bill your account for the tokens used. The key itself is not the account identifier — it is a credential that proves you are authorized to access that account.

The key looks like a string of letters, digits, and dashes. Anthropic keys start with `sk-ant-`. OpenAI keys start with `sk-`. The exact format and length varies by provider, but the concept is the same: a long, unguessable string that only you and the provider know.

The console shows the full key exactly once when you create it. If you navigate away or close the dialog before copying it, you cannot see it again. You would need to revoke that key and create a new one. This is a security measure — the provider does not store the raw key on their servers, only a cryptographic hash, so they literally cannot show it to you again. If their database were compromised, the attacker would find only hashes, not usable keys.

Treat this string like a password: if someone else gets it, they can make requests as you, and you will be billed for their usage.

### Step 2: You Store the Key in an Environment Variable

Hardcoding the key in your Python file is dangerous. If you commit that file to GitHub, the key is exposed permanently in the repository history. Attackers run automated scanners that search public repositories for API key patterns. They find exposed keys within minutes of a push and use them immediately to make calls charged to your account. This is not a theoretical risk — it happens constantly.

The safe practice is to store the key in an environment variable. An environment variable is a key-value pair that lives outside your source code, in the operating system's process environment. Your Python code reads it from `os.environ` at runtime. The key itself never appears in any file that gets committed to version control.

You can set an environment variable directly in your terminal session:

```bash
# macOS / Linux
export ANTHROPIC_API_KEY=sk-ant-api03-your-key-here

# Windows PowerShell
$env:ANTHROPIC_API_KEY = "sk-ant-api03-your-key-here"
```

This works but only for the current terminal session. If you open a new terminal, the variable is gone and you must set it again. This gets tedious fast.

A better approach for project work is to create a `.env` file in your project directory. The `.env` file is a plain text file with one KEY=VALUE pair per line:

```
ANTHROPIC_API_KEY=sk-ant-api03-your-key-here
```

Then use the `python-dotenv` library to load it at runtime:

```python
from dotenv import load_dotenv
load_dotenv()
import os
api_key = os.environ["ANTHROPIC_API_KEY"]
```

The `.env` file stays on your local machine and is never committed to version control because you list it in `.gitignore`:

```gitignore
# .gitignore
.env
```

This pattern is standard across the Python ecosystem and across AI engineering teams. It keeps your secrets local and your code portable. When you share your code on GitHub, other developers clone your repository, create their own `.env` file with their own key, and run the same code without any changes.

The environment variable approach also makes it easy to switch between different keys or accounts. You can have one `.env` file for development with a free-tier key and another for production with a paid-tier key. You just swap the file contents.

### Step 3: Your Code Reads the Key and Creates a Client

When you run your script, Python reads the environment variable using `os.environ`. This is a dictionary-like object that contains all environment variables available to the current process.

```python
import os
import anthropic

api_key = os.environ["ANTHROPIC_API_KEY"]
client = anthropic.Anthropic(api_key=api_key)
```

If you use a `.env` file, you add one line before reading the key:

```python
from dotenv import load_dotenv
load_dotenv()
```

This reads the `.env` file and merges its contents into `os.environ`. After this call, `os.environ["ANTHROPIC_API_KEY"]` works exactly as if you had set the variable in your terminal.

The `anthropic.Anthropic()` constructor creates a client object. This client holds your key internally and attaches it to every request it sends. You create the client once and reuse it for all your API calls in the same script. Creating multiple clients with the same key is unnecessary but harmless.

### Step 4: The Client Builds and Sends an HTTP Request

When you call `client.messages.create(...)`, the SDK does not do anything magical. It builds a standard HTTP POST request. HTTP is the same protocol your web browser uses to load web pages. The same protocol carries your API calls to the provider's server.

The SDK serializes your parameters into a JSON string. It adds your API key as an HTTP header called `x-api-key`. It adds other standard headers like `Content-Type: application/json` and `User-Agent: anthropic-python/0.x`.

Then it opens a TLS-encrypted connection to the server. TLS is the same encryption your banking website uses. Nobody between your computer and the server can read the contents of your request — not your internet service provider, not your network administrator, not anyone on the same Wi-Fi network.

The TLS connection also ensures that the server you are talking to really is the Anthropic API server, not an impostor. This is done through certificate verification, the same system that lets your browser verify that the Anthropic website is the real Anthropic website.

### Step 5: The Server Authenticates and Routes Your Request

The request arrives at the provider's server after a network round trip that typically takes 20-100 milliseconds, depending on your geographic distance from the server.

The server extracts the `x-api-key` header. It computes a cryptographic hash of the key and looks up the hash in its database. If it finds a match, it identifies your account, checks your rate limits (have you made too many requests recently?), and checks your billing status (do you have credits or a valid payment method?).

If any of these checks fail, the server returns an HTTP error response and the model never runs. Common failure modes:

- **401 Unauthorized**: The key is invalid, expired, or belongs to a different provider.
- **429 Too Many Requests**: You exceeded your rate limit. Wait and try again.
- **402 Payment Required**: Your account has no credits and no valid payment method.

If all checks pass, the server routes your request to the model you specified. The model identifier — `claude-sonnet-4-20250514` — tells the server which neural network weights to load and which inference configuration to use.

### Step 6: The Model Processes Input and Generates Output

The server loads the model you requested. It tokenizes your input — breaking your messages into tokens, roughly 3/4 of a word each. The tokenized input is fed into the neural network, which processes it through multiple layers of computation.

The model then begins generating tokens one at a time. At each step, it looks at everything so far (your input plus any tokens it has already generated) and predicts the most likely next token. This is called autoregressive generation.

### Step 7: The Response Travels Back and Your Code Consumes It

The server collects all generated tokens, deserializes them back into text, and wraps everything in a structured JSON response. The response includes the generated content, token counts, and a unique request ID.

```json
{
  "id": "msg_01abc123",
  "type": "message",
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "Hello! How can I help you today?"
    }
  ],
  "usage": {
    "input_tokens": 10,
    "output_tokens": 8
  }
}
```

The response travels back over the same encrypted TLS connection. The SDK receives the raw JSON, parses it into a Python object, and returns it from your `client.messages.create()` call. Your script resumes execution on the next line.

## Cost, Limits, and Safety

### Token Costs

Every API call costs money. Providers charge per token, separately for input (your prompt) and output (the model's response). As of mid-2026, here are approximate costs for common models:

- **Claude Sonnet 4**: roughly $3 per million input tokens, $15 per million output tokens
- **Claude Haiku 3.5**: roughly $0.80 per million input tokens, $4 per million output tokens
- **GPT-4o**: roughly $2.50 per million input tokens, $10 per million output tokens

A single hello-world call with 10 input tokens and 8 output tokens costs roughly 0.00003 cents. That is effectively free. You can make thousands of small calls before you accumulate a dollar in charges.

The cost scales with usage, not per-call. If you send a 50,000-token document as input and ask the model to generate a 10,000-token summary, that single call costs roughly $0.30 — still small, but orders of magnitude more than a hello-world call. Understanding token economics helps you design efficient prompts and avoid surprise bills.

### Rate Limits

API providers limit how many requests you can make in a given time window. This prevents any single user from overwhelming the shared infrastructure. Typical rate limits for free or low-tier plans are 5-50 requests per minute. Paid plans offer higher limits, sometimes thousands per minute.

If you exceed the rate limit, the server returns an HTTP 429 status code. The SDK raises a `RateLimitError` exception. Your script must handle this by waiting and retrying. A simple retry with a one-second delay is often enough to stay within the limit.

As your usage grows, you will need more sophisticated rate limiting strategies — exponential backoff, request queuing, and concurrent request management. These are covered in later sections of this course.

### Key Safety

Your API key is the only thing preventing someone else from using your account. Follow these rules strictly:

- Never paste your key into a code file that will be committed to git.
- Never share your key in a public chat, forum, or documentation.
- Never send your key over unencrypted channels like plain HTTP or unencrypted email.
- Use a `.env` file and add `.env` to `.gitignore` before creating the file.
- If you suspect your key is exposed, revoke it immediately in the provider console and generate a new one.
- Use different keys for development and production, so revoking one does not affect the other.
- Regularly rotate your keys — generate a new one, update your `.env` file, and revoke the old one.

Most providers also let you set key-level restrictions. For example, you can create a key that only works from certain IP addresses, or a key that has a monthly spending cap. These features add layers of protection.

### Setting a Budget

Most LLM providers let you set a spending limit on your account. Find the billing or usage limits section in the provider console and set a monthly budget, especially when you are learning. A $10-20 monthly cap is enough for extensive learning and experimentation.

When you hit the cap, the provider stops accepting requests until the next billing period or until you increase the cap. This prevents any runaway script from costing more than you expect.

### Writing Loop-Safe Code

The most common source of surprise bills is an infinite or near-infinite loop that calls the API repeatedly. Always cap loops that make API calls:

```python
# Safe: capped loop
for i in range(3):
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=100,
        messages=[{"role": "user", "content": "Tell me a fact"}]
    )
    print(response.content[0].text)
```

```python
# Dangerous: unbounded loop
while True:  # Never do this
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=2000,
        messages=[{"role": "user", "content": "Tell me a long story"}]
    )
```

The unbounded loop continues until you manually kill the script or your account runs out of money. A single session of a runaway loop can cost hundreds or thousands of dollars. Always use `for` loops with explicit ranges, or `while` loops with a counter and a maximum iteration check.

## When to Use This / When Not To

This topic covers the absolute minimum you need to make any API call. Use this knowledge every time you interact with an LLM from code.

### Use this approach when:
- You are making your first API call and need to verify that your key, your environment, and your SDK are all working together correctly.
- You are building a prototype that needs a single LLM response to test an idea.
- You are comparing the output of different models or providers by running the same prompt against each.
- You are learning a new SDK or provider and need to establish the authentication and request pattern before building more complex logic.

### Do not rely on this alone when:
- You are building a production application — you need error handling, retries, timeouts, and logging (covered later in this course).
- You need to stream responses for real-time display — use `stream=True` with the streaming API instead of waiting for the full response.
- You are making many calls in a loop — add rate limiting, cost tracking, and circuit breakers to prevent runaway usage.
- You are handling sensitive data that cannot leave your network — consider on-premise models or data processing agreements with your provider.
- You are building a multi-step agent that makes tool calls — you need conversation history management and structured output parsing in addition to basic API calls.

The key insight is that every complex LLM application is built from this same fundamental pattern. Master the single call first, then layer on complexity.

## Mental Model Check

These five questions will confirm you understand the full flow from getting a key to seeing output. If you can answer all five, you are ready to write code.

### Question 1

Where should you store your API key so it is never accidentally committed to a public repository?

**Answer:** Store it in an environment variable, preferably in a `.env` file. Add `.env` to your `.gitignore` file. Your code reads the key at runtime using `os.environ["ANTHROPIC_API_KEY"]` after calling `load_dotenv()`. The key never appears in your source code.

### Question 2

What does `anthropic.Anthropic()` do with your API key, and how does the key reach the server?

**Answer:** It creates an authenticated client object that stores your API key internally. When you call `client.messages.create()`, the client attaches the key as an HTTP header (`x-api-key`) on the outgoing request. The server reads this header to authenticate you. The key is never sent in the URL or the request body.

### Question 3

What three parameters does `client.messages.create()` require at minimum?

**Answer:** The `model` parameter (which LLM to use, e.g. `claude-sonnet-4-20250514`), the `messages` parameter (the conversation content as a list of role/content pairs), and `max_tokens` (the maximum number of tokens the model can generate). Without these, the request will fail with a `BadRequestError`.

### Question 4

If your API key is invalid or expired, what happens when you run your script, and are you charged?

**Answer:** The server returns an HTTP 401 error before the model ever runs. The SDK raises an `AuthenticationError` exception in Python. You are not charged for failed authentication attempts because the model never processed your request. Only requests that reach the model and generate tokens incur costs.

### Question 5

Your script calls the API in a loop and the billing page shows charges are accumulating fast. What is the most likely cause, and how do you fix it?

**Answer:** The most likely cause is an unbounded loop — a `while True` or a loop with no maximum iteration count. Every iteration generates a new API call and incurs token costs. Fix it by adding an explicit iteration cap with a `for` loop (e.g. `for i in range(10):`) or adding a counter that breaks after a maximum number of iterations (e.g. `if count >= 10: break`). Always test loops with a small cap before increasing it.
