Imagine you just heard about large language models for the first time. Everyone says they can write essays, answer questions, summarize books, generate code. You open ChatGPT, type a prompt, and get a perfectly reasonable answer. Magic. Then someone says, "You can call an LLM from your own Python code using an API." You open your editor, write three lines of Python, and... nothing. The error says something about authentication. You try to paste your key directly into the script, it works, you feel relieved, and you commit the file to GitHub. Three days later you wake up to a \$200 bill from someone who scraped your repository, extracted your key, and used it to generate a million tokens of marketing spam.

This is the moment most people give up on building with AI. Not because the concepts are hard — they are not — but because nobody showed them what is actually happening under the hood. The words "LLM" and "API" get thrown around as if everyone already knows what they mean. You are here to avoid that gap. By the end of this page, you will know exactly what an LLM is, what an API is, how they communicate, and how to use them safely.

There is a second common story. A developer hears about APIs, reads a tutorial, writes a script that asks the model to summarize a news article, and it works. Excited, they scale up: they write a loop that summarizes 200 articles from a CSV file. The loop starts fast, then slows down. After a few minutes, every call returns an error. The developer has no idea why. They try changing the model name, reinstalling the package, restarting their computer. Nothing helps. They give up. What actually happened is they hit the API rate limit — the provider allows 50 requests per minute on their plan, but the loop runs at 200 requests per minute. The server started returning HTTP 429 errors, but the developer never looked at the error code, only at the message, which said something vague about "too many requests."

The third story is the quietest. Someone builds an app that calls an LLM for every user action. A user leaves the app open overnight. The app generates 15,000 API calls while the user sleeps. The developer wakes up to a \$450 bill and a suspended account. The app had no cost guardrails — no token limit, no spending cap, no loop boundary. The model cheerfully answered the same question 15,000 times.

These are not rare edge cases. They are the normal failure modes of learning this technology without a mental model. Every single one is preventable when you understand what an LLM is, what an API is, and how they interact.

The second reason people struggle even when they avoid those pitfalls is that they jump straight to writing code without understanding the moving pieces. When an error comes back, they have no mental model to interpret it. An authentication error looks like random noise if you do not know that the server checked your key before running anything. A rate limit error looks like the server is broken if you do not know that providers throttle requests to protect their infrastructure. A model-not-found error makes no sense if you do not know that model names are versioned and change periodically. Every error becomes a dead end instead of a clue. Building your mental model first — what each part is and what job it does — turns those errors into meaningful signals.

## The Concept

An LLM — large language model — is a program that has been trained on a vast amount of text and can predict the next word in a sequence. That is the entire idea, stated plainly. It does not think. It does not understand the way a person understands. It is a statistical engine that has read so much text — billions of sentences from books, articles, code, conversations — that it can produce plausible continuations for nearly any prompt you give it.

Think of an LLM as an extremely advanced autocomplete. Your phone suggests the next word when you type a text message. An LLM does the same thing, but at a much larger scale. Where your phone might look at the last three words, an LLM can consider tens of thousands of words of context at once. Where your phone suggests one word, an LLM generates whole paragraphs, one token at a time, choosing each next token based on all the tokens that came before.

How does the model learn? Training happens in two phases. First, the model reads an enormous corpus of text — trillions of words from the public internet, books, academic papers, code repositories. During this phase, the model adjusts its internal parameters (billions of numbers) to minimize the difference between its predictions and the actual next word in every sentence it reads. After this phase, the model has a broad understanding of language patterns but no ability to follow instructions.

The second phase is instruction tuning. The model is fine-tuned on pairs of instructions and ideal responses written by human annotators. The model learns to recognize when it is being asked a question and to produce a response in a helpful tone. This is what makes the model feel like a conversation partner rather than a raw text predictor. Without instruction tuning, the model would just complete your sentence instead of answering your question.

An API — Application Programming Interface — is a waiter. You sit at a table. You tell the waiter what you want. The waiter carries your order to the kitchen. The chef cooks the food. The waiter brings it back to your table. You eat. You do not need to know how the kitchen works, what brand of stove they use, or where the chef trained. You only need to know how to place an order and how to receive it.

In the context of LLMs, your code is the customer. The LLM provider's API server is the waiter. The LLM itself is the chef. Your code sends a structured request over the internet. The API server authenticates you, routes your request to the right model, runs inference, and sends the result back. Your code receives the response and does something with it — prints it, saves it, uses it in a calculation, sends it to another system.

The API also handles errors. If you send a bad request — missing a required field, using an invalid model name, exceeding the max tokens — the API returns an error instead of a response. The error comes in a standard format with a code and a message. This is better than silent failure, because your code can check for errors and react appropriately. A missing field produces a 400 Bad Request error so you can fix your code. An invalid model name produces a 404 Not Found error. Each error type tells you exactly what went wrong.

The key insight is that the API is a contract. Both sides agree on a format. You send JSON in a specific shape — model name, messages, parameters. The server sends JSON back in a specific shape — generated text, token counts, stop reason. As long as both sides follow the contract, everything works. The contract is documented, stable, and versioned. When the provider updates the contract, they announce it in advance and give you time to update your code.

### What an LLM Can and Cannot Do

It helps to be precise about capabilities before you start using these tools. Some things LLMs are excellent at. Other things they are terrible at, and you need to know the difference.

| Capability | Good at | Bad at |
|---|---|---|
| Language | Writing, summarizing, translating, rephrasing, explaining concepts | Verifying factual claims, knowing when it does not know something |
| Code | Generating code from descriptions, explaining code, converting between languages | Running code, knowing whether its own generated code actually compiles or works |
| Math | Explaining concepts, working through well-known problem types | Novel arithmetic with many digits, exact calculations — it approximates |
| Logic | Following patterns, completing common reasoning chains | Detecting contradictions, handling novel logical structures it has not seen in training |
| Memory | Holding tens of thousands of tokens in context | Remembering what it said five conversations ago, tracking long-term state without assistance |
| Consistency | Producing similar outputs for similar inputs at the same temperature | Producing the exact same output every time (unless temperature is 0) |

The most important distinction: an LLM can generate text that looks like a confident answer, regardless of whether that answer is correct. The model has no internal sense of truth. It has an internal sense of plausibility. It generates the text that is most likely to follow your prompt given everything it has seen in training. If the most likely continuation is wrong but sounds correct, the model will produce it with full apparent confidence. This is called hallucination, and it is a fundamental property of how LLMs work, not a bug that will be fixed in the next version.

### The API Contract in More Detail

The API contract is defined in a specification called OpenAPI (formerly Swagger). The provider publishes a machine-readable JSON or YAML file that lists every endpoint, every parameter, every response field, and every possible error code. SDKs are often auto-generated from this spec, which is why SDK updates usually follow spec changes quickly.

The contract version is embedded in the URL path. The current version is `/v1/messages`. If the provider makes a breaking change, they release a new version (`/v2/messages`) while keeping the old version running for a transition period. This means your code will not break when the API updates — you control when to migrate.

## API Reference

The table below covers the most common parameters and methods for the Anthropic Messages API, one of the most widely used LLM APIs. Each entry shows the exact parameter name, what it controls, and a usage example.

| Method / Parameter | What It Does | Example |
|---|---|---|
| `POST /v1/messages` | Send a conversation to the model and receive a generated response. This is the primary API endpoint. | `client.messages.create(model="...", messages=[...])` |
| `model` | Identifies which version of the LLM to use. Different models have different capabilities, speed, and pricing. The string must match exactly. | `model="claude-sonnet-4-20250514"` |
| `messages` | The conversation history as a list of objects, each with a `role` (user or assistant) and `content` (the text). The list must alternate user/assistant/user/assistant. | `messages=[{"role": "user", "content": "Hello"}]` |
| `max_tokens` | The maximum number of tokens the model can generate in the response. Once this limit is reached, generation stops. This is your primary cost control. | `max_tokens=1024` |
| `system` | An optional system prompt that sets the model's behavior, tone, and constraints before any user messages. It is not part of the visible conversation. | `system="You are a helpful assistant."` |
| `temperature` | Controls randomness in the output. 0 produces nearly deterministic responses. Higher values (up to 1 or above) produce more varied and creative outputs. | `temperature=0.7` |
| `stream` | When true, the response is delivered incrementally as tokens are generated, rather than all at once at the end. This enables real-time display. | `stream=True` |
| `stop_sequences` | An array of strings that, when generated by the model, cause it to stop producing more tokens. Useful for delimiting structured output. | `stop_sequences=["END", "STOP"]` |
| `top_p` | An alternative to temperature. Nucleus sampling considers only the tokens whose combined probability reaches this value. A value of 0.9 means the model chooses from the smallest set of tokens whose summed probability is at least 90%. | `top_p=0.9` |

Every API call also requires an authentication header — typically `x-api-key` — which the SDK adds automatically when you pass your key to the client constructor.

### Response Object Structure

When the API responds, it returns a structured object with several fields. Here is what each field contains:

| Response Field | What It Contains | Example Value |
|---|---|---|
| `id` | A unique identifier for this API call, useful for debugging and billing verification | `"msg_01ABCDEF1234567890"` |
| `content` | A list of content blocks; each block has a `type` and `text` | `[{"type": "text", "text": "Hello"}]` |
| `model` | The model that generated the response (confirms which model you actually used) | `"claude-sonnet-4-20250514"` |
| `role` | The role of the responder, always `"assistant"` for LLM responses | `"assistant"` |
| `stop_reason` | Why the model stopped generating: `"end_turn"` (natural stop), `"max_tokens"` (hit limit), or `"stop_sequence"` (hit a custom stop string) | `"end_turn"` |
| `usage` | Token counts broken into `input_tokens` and `output_tokens` | `{"input_tokens": 10, "output_tokens": 5}` |

### How Your API Key Fits In

Your API key is the single secret that ties every API call to your account. The key is a long random string — typically 90-100 characters — that the provider generates for you. You create one from the provider's dashboard and copy it into your environment.

The SDK reads your key from an environment variable or from the client constructor. It never sends the key as part of the request body — only in the HTTP header. This is a security best practice: headers are less likely to be logged accidentally than request bodies.

When the server receives your request, it extracts the key from the header, hashes it, and looks up the hash in its authentication database. This means the provider never stores your raw key — only a cryptographic hash. If their database is breached, the keys cannot be recovered from the hashes.

## How It Actually Works — Step by Step

Each step below reveals what happens at a different layer of the system. Read them in order to build a complete mental model.

### Step 1: Your Code Starts an HTTP Conversation

When you call `client.messages.create(...)`, the SDK does not do anything magical. It builds a plain HTTP POST request. HTTP is the same protocol your web browser uses to load this page. The same protocol carries your API calls to the provider's server.

The SDK takes all of your parameters — `model`, `messages`, `system`, `max_tokens`, `temperature` — and serializes them into a JSON string. JSON is a text format for structured data that looks like a Python dictionary. It is the standard way that web services exchange data.

The SDK then adds an HTTP header called `x-api-key` with your secret key value. This header tells the server who you are. Without it, the server will not process your request.

You could build this HTTP request manually using Python's built-in `http.client` or the `requests` library. People did exactly that before SDKs existed. The SDK just saves you from writing boilerplate — it constructs the URL, adds the headers, serializes the body, sends the request, and parses the response.

### Step 2: The Request Travels over the Internet

The SDK sends the HTTP request to a URL like `https://api.anthropic.com/v1/messages`. Your computer first looks up the server address using DNS — the Domain Name System, which is the same system that turns google.com into a numeric IP address like 142.250.80.14. Once your computer knows the IP address, it opens a TCP connection.

That connection is encrypted using TLS — Transport Layer Security. This is the same encryption your browser uses when you visit a website with a padlock icon. The encryption ensures that nobody between you and the server can read your prompt, your API key, or the response. Not your internet provider, not the coffee shop WiFi, not anyone on the same network.

This encryption is not optional. Every modern API provider requires HTTPS. If you try to send a request over plain HTTP (without the 's'), the server will reject it. The SDK handles this automatically by using https:// in the URL.

### Step 3: The Server Authenticates Your Key

When the encrypted request arrives at the API server, the first thing the server does is look at the `x-api-key` header. It checks your key against its database of valid keys. It verifies that the key is active, that your account is in good standing, and that you have not exceeded your rate limit or quota.

If your key is invalid, expired, or missing, the server returns an HTTP 401 response with an error message saying authentication failed. The model never runs, and you are not charged. This check happens in milliseconds at the API gateway layer, before the request reaches the model servers.

If authentication succeeds, the server looks up your account tier — free, pro, enterprise — to determine your rate limits and priority. Higher-tier accounts get higher rate limits and faster routing.

This is why your API key is critically important. Anyone who has your key can make requests as you, and your account gets billed for those requests. A leaked key can cost you real money. Never hardcode it in your source code. Always read it from an environment variable or a secrets manager.

### Step 4: The Input Gets Tokenized

Once authenticated, the server takes your messages and breaks them into tokens. A token is a chunk of text — roughly 3/4 of a word in English, though the exact size varies by character and language.

The word "hello" might be one token. The word "unbelievable" might be split into three tokens: "un", "believ", "able". A space is often part of the preceding token. Punctuation attaches to the preceding word token. Chinese characters are typically one token each. Very long numbers get split into digit groups.

Every LLM has a fixed vocabulary of tokens it knows — typically 50,000 to 200,000 tokens. The model can only process text that can be represented using its vocabulary.

The total number of tokens in your input (the prompt) determines the input cost. The number of tokens the model generates determines the output cost. You pay for both, and output tokens are typically 3-5 times more expensive than input tokens per token, because generating output requires more computation than processing input.

### Context Window: The Model's Working Memory

Before the model starts generating, all of your input tokens are loaded into the model's context window. The context window is the maximum number of tokens the model can "see" at once. Think of it as the model's short-term memory. Every model has a fixed context window size — common sizes are 8K, 16K, 32K, 100K, or even 200K tokens.

If your prompt (including the system prompt, all message history, and any other content) exceeds the context window, the oldest tokens are dropped or the call fails. The model simply cannot see anything beyond its window.

This has direct implications for multi-turn conversations. Every turn you add to the conversation increases the token count. After enough turns, the conversation becomes longer than the context window. The model "forgets" the early part of the conversation because those tokens have been dropped. When building chatbots that need long-term memory, you need strategies to manage context — summarizing old messages, keeping only recent messages, or using external memory systems.

The context window also includes the tokens the model generates. A model with a 16K token window that receives a 12K token prompt can only generate 4K tokens of output before the window is full.

### Step 5: The Model Predicts Tokens One at a Time

The tokenized input passes into the neural network — the actual LLM. The model processes your entire prompt through its layers of computation. Each layer transforms the representation of the text, building up a rich internal state that captures meaning, context, and relationships between words.

After processing the full prompt, the model begins generating output one token at a time. At each step, it looks at all the tokens so far — both the input and whatever it has already generated — and assigns a probability score to every possible next token in its vocabulary. It picks one based on those probabilities (influenced by `temperature`) and outputs it.

This is called autoregressive generation. The model generates one token, appends it to the context, generates the next token, and repeats until it produces a stop token or reaches your max_tokens limit.

The temperature parameter changes how the model picks from its probability distribution. At temperature 0, the model always picks the single most likely token. At temperature 0.7, it sometimes picks a slightly less likely token, producing more varied output. At temperature 1.0 or higher, the variation increases further. This is why temperature 0 is good for factual answers (you want the most likely completion every time) and higher temperatures are good for creative writing (you want variety).

### Step 6: The Response Travels Back and Your Code Consumes It

The server collects all the generated tokens, deserializes them back into text, and wraps them in a JSON response. The response includes the generated content, the model identifier, why the model stopped generating, and the exact token usage breakdown.

The response travels back over the same encrypted connection. The SDK parses the JSON and gives you a response object with typed fields.

```python
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=100,
    messages=[{"role": "user", "content": "What is the capital of France?"}]
)

print(response.content[0].text)
# Output: The capital of France is Paris.
```

Behind the scenes, the SDK waits for the entire response before returning. Your code pauses at that `client.messages.create(...)` line and does not continue until the server has finished generating every token and the full HTTP response has arrived. For short responses this is nearly instant. For long responses — say, generating 2000 tokens — this can take 10-30 seconds depending on the model and network latency.

### Step 7 (Optional): Streaming Lets You See Tokens as They Arrive

If you set `stream=True`, the SDK does not wait for the full response. Instead, it opens a Server-Sent Events connection — a long-lived HTTP connection where the server pushes data as it becomes available. Your code receives each token the moment the model predicts it.

This is how ChatGPT streams its response word by word. The first token arrives in milliseconds, even if the full response will take many seconds. From a user perspective, streaming feels dramatically faster because you see the beginning of the answer immediately rather than staring at a spinner.

```python
with client.messages.stream(
    model="claude-sonnet-4-20250514",
    max_tokens=100,
    messages=[{"role": "user", "content": "Count to 5."}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
# Output (appears incrementally):
# Here is a count from 1 to 5:
# 1
# 2
# 3
# 4
# 5
```

The `flush=True` argument to `print()` forces Python to write text to the terminal immediately instead of buffering it. Without `flush=True`, your terminal might hold several tokens and print them in a batch, defeating the purpose of streaming.

## Cost, Limits, and Safety

### Token Costs

Every API call costs money based on the number of tokens in your prompt (input tokens) and the number the model generates (output tokens). Different models have different pricing. As of mid-2026, a capable model like claude-sonnet-4 costs roughly $3 per million input tokens and $15 per million output tokens.

A million tokens is roughly 750,000 words — more than three full novels. Most individual API calls cost fractions of a cent. A typical question-and-answer pair might cost $0.001 or less. Even heavy usage — hundreds of calls per day — would cost only a few dollars.

The real danger is not one call. The danger is a loop that calls the API thousands of times without your awareness. Consider this pattern:

```python
# This code will cost you real money — potentially a lot
while True:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=2000,
        messages=[{"role": "user", "content": "Tell me a long story"}]
    )
    print(response.content[0].text)
```

That loop would run until you kill it or your account runs out of credit. In one hour, a loop like this could make 3600 API calls, each generating 2000 tokens, costing roughly $108. Always set explicit limits on loops that call an LLM. A `for` loop with a fixed count is safer than a `while True` loop.

### Estimating Cost Before You Call

You can estimate the cost of an API call before making it by counting the tokens in your prompt. Every character matters — spaces, punctuation, line breaks all count toward the token total.

As a rough rule of thumb for English text:
- 100 tokens is about 75 words
- 1000 tokens is about 750 words
- 1 million tokens is about 750,000 words

To estimate input cost, count the approximate words in your prompt, convert to tokens (multiply by 1.33), then multiply by the per-token price. For example, a 500-word prompt is about 665 tokens. At $3 per million input tokens, that costs about $0.002 — two-tenths of a cent.

Output is harder to estimate because you do not know in advance how long the response will be. This is where `max_tokens` helps — it caps your maximum possible cost. A `max_tokens=500` call at $15 per million output tokens costs at most $0.0075, regardless of what the model actually produces.

Most API providers show your total spending in their dashboard. Check it regularly when you start experimenting, especially if you are using a pay-as-you-go account without spending limits.

### Rate Limits

API providers limit how many requests you can make per minute. The exact limits depend on your plan level — free tiers have lower limits, paid tiers have higher ones. If you exceed the limit, the server returns HTTP status 429 (Too Many Requests) with a message explaining when you can retry.

Your code must handle rate limits gracefully. The simplest approach is to add retry logic with a delay:

```python
import time

for attempt in range(3):
    try:
        response = client.messages.create(...)
        break
    except anthropic.RateLimitError:
        if attempt < 2:
            time.sleep(5)
        else:
            raise
```

The Anthropic SDK has built-in automatic retry for rate limits. By default it retries up to 2 times with exponential backoff. You can configure this when creating the client.

### Key Security

An API key is a secret string that proves your identity to the server. It looks something like `sk-ant-api03-xxxxxxxxxxxx`. If anyone else obtains this key, they can make API calls as you, and you will be billed for their usage.

The most common way keys get leaked is through accidental commits to public git repositories. Automated bots continuously scan GitHub for exposed API keys. When they find one, they start using it within minutes — often for cryptocurrency mining-related API calls or spam generation.

Never put your API key directly in your code:

```python
# NEVER DO THIS
api_key = "sk-ant-api03-xxxxxxxxxxxx"  # hardcoded in source code
client = Anthropic(api_key=api_key)
```

Always read it from an environment variable:

```python
import os
api_key = os.environ["ANTHROPIC_API_KEY"]  # read from environment
client = Anthropic(api_key=api_key)
```

Set the environment variable in your terminal or in a `.env` file. The `.env` file should be listed in `.gitignore` so it never gets committed to git. Most SDKs and AI projects use this pattern, and tools like python-dotenv make it easy to load `.env` files.

## When to Use This / When Not To

Choosing between an API-based LLM and alternatives (local models, rule-based systems, no AI at all) depends on your specific constraints. The table below maps common scenarios to the right choice.

| Scenario | Recommended Approach | Why |
|---|---|---|
| You are prototyping and exploring | Local model via Ollama | Free, no rate limits, no key management. Iterate freely, then switch to API for production. |
| You need the most capable model available | API (frontier model) | Frontier models require high-end GPUs that cost thousands per month to operate. API pricing is cheaper at low-to-moderate volume. |
| You handle medical, financial, or legal data | Local model on your own infrastructure | Compliance regulations (HIPAA, GDPR, SOC2) may prohibit sending data to third-party API servers. A local model keeps data on your network. |
| You need one simple classification (spam vs not spam) | Rule-based system or small local classifier | An API call adds latency, cost, and a network dependency for something a 50-line regex or a tiny ML model can handle in microseconds. |
| You are running millions of requests per day | Self-host a smaller model on your own GPUs | At high volume, per-token API pricing exceeds the total cost of ownership for dedicated GPU hardware running a distilled model. |
| Your application cannot tolerate network delays | Local model | API calls add 100-500ms of network latency before the model even starts generating. A local model responds in single-digit milliseconds. |
| You need the response immediately (latency-critical) | API with streaming | Streaming delivers the first token in milliseconds. Even with network latency, streaming feels instant to end users. |

### Use an LLM API when:
- You need access to a capable model that requires GPU hardware you do not own. Frontier models run on clusters of expensive GPUs that are not practical to self-host.
- Your application needs fast, reliable responses with uptime guarantees. Providers run redundant infrastructure with 99.9%+ uptime.
- You want the latest model capabilities as soon as they are released, without managing upgrades yourself.
- Your usage volume is moderate — up to thousands of requests per day. At this volume, API pricing is cheaper than the cost of GPUs and operations.

### Do not use an LLM API when:
- You are prototyping and experimenting with many rapid iterations. Use a local model with Ollama first — it is free and has no rate limits. Once your prompt and parameters are settled, switch to the API for production.
- You are handling sensitive data that cannot leave your network. Medical records, financial documents, proprietary code — if your compliance rules forbid sending data to third parties, you must use a local model.
- You only need a single fixed response — like classifying an email into two categories. A rule-based system or a small local classification model is cheaper, faster, and more consistent than an API call.
- You are running thousands of requests per minute. At high volume, self-hosting a smaller model on your own GPU infrastructure is more cost-effective than paying per-token API pricing.
- You need deterministic responses. APIs have network latency and connectivity dependencies that local models do not.
- You are building a feature where a simple lookup table or mathematical calculation would produce the same result with zero cost and zero latency. Always ask: does this problem genuinely require language generation, or can I solve it with simpler tools?

### The Hybrid Approach

Many production systems use a hybrid: a local model handles high-volume, low-stakes requests, and an API model handles complex or ambiguous requests that need a capable model. For example, a content moderation system might use a local model to filter obvious spam (95% of traffic) and route the remaining 5% of borderline cases to an API model. This keeps costs low while maintaining quality for the hard cases.

The same pattern applies within a single API provider. You might use a faster, cheaper model (like claude-3-haiku) for simple tasks and reserve the most capable model for complex reasoning. This tiered approach gives you the best balance of cost, speed, and quality.

## What You Have Learned

You now know that an LLM is a next-word prediction engine trained on billions of sentences, not a thinking mind. It generates one token at a time, choosing each next token based on the probability distribution over its vocabulary. The model has no sense of truth — it has a sense of plausibility. It does not know when it is wrong, which is why you must always verify its output.

You know that an API is a contract that lets your code send requests to that engine over the internet and receive structured responses. The contract specifies exactly what format the request must follow and what format the response will take. Parameters like `model`, `max_tokens`, `temperature`, and `system` are part of this contract — each one changes the behavior of the model in a predictable way.

You know the seven steps of an API call: SDK serialization of parameters into JSON, HTTPS encryption of the request, authentication of your key at the server, tokenization of your input, inference by the model, streaming or blocking response delivery, and deserialization back into a Python object. Each step is a concrete operation that either works or produces a specific error.

You know what tokens are — the atomic units the model processes — and how context windows bound how much text the model can see at once. You know that both input and output are metered in tokens, with separate pricing for each direction.

You have built the mental model that turns cryptic error messages into meaningful clues. An authentication error means the server did not recognize your key. A model-not-found error means the model string has changed. A rate limit error means you sent too many requests. Each error is a signal pointing to a specific problem with a specific fix.

## Mental Model Check

These five questions will confirm whether you can explain an LLM API to someone else. If you can answer all five, you are ready to write code. If any answer surprises you, re-read the corresponding section before moving to the practice exercise.

### Question 1
What does API stand for, and what job does it do in the context of an LLM?

**Answer:** API stands for Application Programming Interface. It is a contract that defines how your code communicates with the LLM server. Your code sends a structured request (JSON over HTTP) to the API endpoint, and the API returns a structured response containing the model's generated text and metadata. The API handles authentication, routing, and response formatting so you do not need to manage the underlying infrastructure.

### Question 2
When you set `stream=True` in an API call, what changes about how you receive the response?

**Answer:** Without streaming (`stream=False`, the default), your code pauses and waits for the entire response to be generated, then receives it as a single complete object. With streaming (`stream=True`), your code receives each token as soon as the model predicts it, via a persistent Server-Sent Events connection. This allows you to display output incrementally, making the application feel faster and more responsive. The first token arrives in milliseconds rather than waiting for the full generation.

### Question 3
What is a token, and why does it matter for cost?

**Answer:** A token is a chunk of text that the model processes as a single unit. In English, one token is roughly 3/4 of a word. Both the input (your prompt) and the output (the model's response) are measured in tokens. API pricing is per token, with separate rates for input and output tokens. Output tokens are typically priced 3-5 times higher than input tokens because generating output requires more computation. Knowing your token count lets you estimate and control costs.

### Question 4
Why should you never hardcode an API key in your source code?

**Answer:** If you hardcode an API key in source code and commit it to a public or even private repository, the key can be discovered. Automated bots scan code hosting platforms for exposed keys. Once compromised, the key can be used by anyone to make API calls on your account, generating charges that you are responsible for. The solution is to read the key from an environment variable at runtime and ensure the key file (like `.env`) is listed in `.gitignore`.

### Question 5
What does the `max_tokens` parameter control, and what happens if you set it too low?

**Answer:** `max_tokens` sets the maximum number of tokens the model is allowed to generate in its response. Once this limit is reached, the model stops generating, even if it has not finished its answer. If you set it too low, the model will produce truncated responses. For example, if you ask for a detailed explanation but set `max_tokens=10`, the model will stop after about 7 words, leaving the response incomplete. The `stop_reason` field in the response will say `max_tokens` instead of `end_turn`, indicating truncation.
