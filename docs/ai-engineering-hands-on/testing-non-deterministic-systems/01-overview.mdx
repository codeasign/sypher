You ship a chatbot that answers customer questions about shipping policies. Monday morning, a customer asks "How long does standard shipping take?" and gets "3-5 business days." Perfect. Tuesday, the same customer asks the exact same question and gets "Standard shipping typically arrives within 5-7 business days depending on your location." Your customer is confused — which is it? You debug and find no bug. Both answers are correct. The model just chose different words on different days.

This is non-determinism in action. In traditional software, the same input always produces the same output. In LLMs, the same prompt with the same settings can — and will — produce different outputs. Testing a system that does not produce consistent results requires a fundamentally different approach: you test the distribution of outputs, not individual outputs.

## The Concept

**Testing non-deterministic systems means measuring the statistical properties of a model's output distribution rather than asserting the correctness of any single output.** Think of it like testing a dice-rolling function. You do not write a test that says "the dice always returns 4." You write tests that say "over 1,000 rolls, each face appears roughly 16.7% of the time" and "the dice never returns 7." For LLMs, you test that outputs are "usually good" and "never catastrophic" rather than "always identical."

## Statistical Testing Methods Reference

| Method | What It Measures | How to Use It |
|---|---|---|
| `temperature` sampling (0.0 to 2.0) | Controls output variability. 0.0 = deterministic (argmax), higher = more diverse | Set `temperature=0.0` for deterministic tests, `temperature=0.7` for production variety |
| `top_p` (nucleus sampling) | Cumulative probability cutoff for token selection | `top_p=0.9` means sample from the smallest set of tokens whose probabilities sum to 90% |
| `n_samples` / `n` parameter | How many responses to generate per prompt | Pass `n=5` to get 5 different completions for the same input |
| `scipy.stats.kstest` | Kolmogorov-Smirnov test — compares two distributions | `ks, p = kstest(sample_scores, 'norm')` |
| `scipy.stats.ttest_ind` | Independent t-test — compares means of two groups | `t, p = ttest_ind(baseline, candidate)` |
| `numpy.percentile` | Quantile analysis — checks tail behavior | `np.percentile(scores, [5, 25, 50, 75, 95])` |
| `pytest.approx` with tolerance | Approximate equality for floating-point scores | `assert mean_score == pytest.approx(0.85, abs=0.05)` |
| `binomial test` | Tests if pass rate matches expected proportion | `scipy.stats.binomtest(passes, total, p=0.9)` |

## How It Actually Works — Step by Step

### Step 1: Understand Where Non-Determinism Comes From

LLMs are non-deterministic by design. At each token generation step, the model outputs a probability distribution over the entire vocabulary. The sampling strategy determines how the next token is chosen from this distribution. Even with temperature 0.0, floating-point arithmetic across different hardware or library versions can introduce tiny variations.

### Step 2: Run Multiple Samples Per Prompt

To test a non-deterministic system, you never run a single input once. You run it N times (typically 3-10 for quick checks, 20-50 for statistical rigor) and capture all outputs. This gives you a distribution of responses to analyze.

### Step 3: Compute Aggregate Statistics

For each dimension of quality, you compute statistics across the N samples: mean, standard deviation, percentiles, min, max. The key insight is that the mean score across 10 samples is a much more reliable signal than any single score.

### Step 4: Set Statistical Pass/Fail Criteria

Instead of `score >= 0.8`, your criteria become `mean score >= 0.8 with standard deviation <= 0.1` or `95th percentile score >= 0.7`. This accounts for the inherent variability of the system.

### Step 5: Run Statistical Significance Tests

When comparing two versions of a system (e.g., old prompt vs new prompt), you use statistical tests to determine whether the difference in scores is real or just noise. A t-test gives you a p-value: the probability that the observed difference could have occurred by chance.

### Step 6: Track the Distribution Over Time

A single evaluation run gives you a snapshot. The real power comes from tracking the distribution over time — is the mean stable? Is the variance increasing? Is the tail getting worse? These trends are early warning signs of system degradation.

## Cost, Limits, and Safety

**The cost of multiple samples:** Running each prompt 10 times means 10x the API cost. For a dataset of 100 prompts, that is 1,000 API calls instead of 100. Budget for this. The trade-off is reliability: a single-sample evaluation is noisy enough to be misleading, while a 10-sample evaluation gives you confidence in the result.

**Temperature matters:** Higher temperature produces more varied outputs, which means you need more samples to get a stable estimate of the mean. At temperature 0.3, 5 samples might be enough. At temperature 0.9, you might need 20+ samples. Choose your temperature based on the use case, and choose your sample count based on the temperature.

**Seed-based reproducibility:** Some providers support a `seed` parameter. With a fixed seed and temperature 0.0, you get deterministic outputs. This is useful for debugging but dangerous for testing — if you test only with seed=42, you are testing one specific path through the model, not the full behavior.

**Statistical significance is not practical significance:** A p-value of 0.001 means the difference is real, but the difference might be 0.82 vs 0.83 — practically meaningless. Always look at the effect size, not just the p-value.

## When to Use This / When Not To

**Use statistical testing when:** You are evaluating a live system that will serve many users, comparing model versions, or setting a quality baseline before deployment. The investment in statistical rigor pays off when the decision has real consequences (deploying to production, replacing a model version, changing a prompt strategy).

**Do not use statistical testing when:** You are debugging a single bad output, doing rapid prototyping, or performing a quick sanity check. For those, manual review of 3-5 outputs is more efficient. Save the statistical approach for the "should we deploy?" decision.

**Use temperature 0.0 for deterministic testing:** When you are writing unit tests for your prompt template logic (e.g., "does the template include the user's name?"), use temperature 0.0. This gives you deterministic outputs you can test with traditional assertions. Reserve non-deterministic testing for evaluating the quality of the model's output, not the structure.

## Mental Model Check

**Q1:** Why is a single test run on an LLM not reliable for determining output quality?

**A1:** Because the LLM is non-deterministic — the same prompt can produce different outputs across runs. A single sample might be unusually good or bad, giving a misleading impression of the system's true quality.

**Q2:** What is the relationship between temperature and required sample size?

**A2:** Higher temperature = more output variation = more samples needed to estimate the mean reliably. At temperature 0.0, 1 sample is enough. At temperature 0.7, aim for 5-10 samples. At temperature 1.0+, aim for 20+ samples.

**Q3:** If you run 10 samples and get scores of 0.9, 0.88, 0.92, 0.87, 0.91, 0.89, 0.88, 0.90, 0.91, 0.89, what is the standard deviation?

**A3:** The mean is 0.895. The standard deviation is approximately 0.016. This is very low variance — the system is performing consistently.

**Q4:** What statistical test would you use to compare two sets of eval scores from different model versions?

**A4:** An independent t-test (Welch's t-test, which does not assume equal variance). This tests whether the means of the two distributions are significantly different.

**Q5:** If you set temperature to 0.0 and seed to 42, are you guaranteed identical outputs everywhere?

**A5:** Not necessarily. Floating-point arithmetic differences across hardware (GPU vs CPU, different GPU architectures) and library versions (CUDA, PyTorch, ONNX) can introduce tiny variations. The outputs will be very close — often identical — but not guaranteed identical across all environments.
