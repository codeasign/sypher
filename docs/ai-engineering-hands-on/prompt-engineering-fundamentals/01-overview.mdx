You write a prompt: "Write a professional email declining a job offer." The model returns a long apology explaining why the candidate should take the job instead. You add "firm but polite." It becomes rude. You add "do not be rude." It becomes passive-aggressive. You try "make it sound like a Fortune 500 CEO wrote it." Now it uses jargon no human would say. You have spent 45 minutes getting worse results than your first attempt.

This is the universal experience of prompt engineering before you understand how it works. A good prompt feels like magic — you say the right thing and the model nails it. A bad prompt sends you down a spiral of adding instructions, each making things worse. The difference between the two comes down to understanding what the model actually "sees" and how its training shapes its responses. Prompt engineering is the skill of translating what you want into what the model can act on.

## The Concept

Prompt engineering is the practice of designing inputs to an LLM to consistently produce desired outputs. It is not "talking to an AI" — it is crafting structured instructions for a statistical text generator. The prompt is configuration, not conversation. A well-engineered prompt defines the task, the constraints, the output format, and the context explicitly, because the model will not infer anything you do not state.

## API Reference

### Prompt Structure Elements

| Element | What It Does | Example |
|---|---|---|
| System / Developer prompt | Sets the model's behavior, persona, and constraints before any user input | `{"role": "developer", "content": "You are a helpful coding assistant."}` |
| User prompt | The actual query or task from the user | `{"role": "user", "content": "Write a Python function to sort a list."}` |
| Assistant response | A prior response from the model — used in multi-turn to provide examples or context | `{"role": "assistant", "content": "Here is how..."}` |
| Few-shot examples | Giving the model 2-5 examples of desired input-output pairs before asking it to perform the task | `"Example 1: Q: What is 2+2? A: 4"` in the prompt |
| Instructions block | A standalone section of the prompt that states rules, constraints, and requirements | `"Follow these rules: 1) Be concise. 2) Use bullet points."` |
| Output format spec | Explicit description or schema of how the response should be structured | `"Return a JSON object with 'name' and 'age' fields."` |
| Context / Reference text | Information the model needs to answer correctly, placed before the question | `"The following is the documentation: ... Now answer:"` |
| Chain-of-thought | Asking the model to reason step by step before giving the final answer | `"Think step by step, then provide the final answer."` |

### Key Parameters

| Parameter | Effect on Prompting | Typical Range |
|---|---|---|
| `temperature` | Controls randomness. Lower values make output more deterministic and focused | 0.0 (precise) to 1.0 (creative) |
| `top_p` | Nucleus sampling — alternative to temperature. Lower values restrict token choices | 0.1 (focused) to 1.0 (diverse) |
| `max_tokens` | Maximum response length — forces the model to be concise when set low | 100 (short) to 4096 (long) |
| `presence_penalty` | Penalizes tokens that have appeared so far — encourages the model to discuss new topics | -2.0 to 2.0 |
| `frequency_penalty` | Penalizes tokens based on frequency — reduces repetition | -2.0 to 2.0 |
| `stop` | Stop sequences — tells the model when to stop generating | `["\n\n", "Human:"]` |

## How It Actually Works — Step by Step

### Step 1: Tokenization

Your prompt is not read as words. It is split into tokens — chunks of 1-4 characters. The model "sees" a sequence of token IDs, not text. This matters because:

- A word can be split across multiple tokens (e.g., "unbelievable" → `["un", "believ", "able"]`)
- The model's understanding of your prompt depends on how the tokenizer splits it
- Each token has a fixed position in the context window — 128K tokens means 128K token positions, not 128K words

### Step 2: Pattern Matching Against Training Data

The model has been trained on trillions of tokens from the internet, books, code, and other sources. When you send a prompt, the model computes which token sequences in its training best match the pattern of your input. It does not "understand" your request — it finds the statistically most likely completion.

This is why prompt engineering works: you are shaping the statistical pattern. A well-structured prompt activates the "correct" region of the model's training, while a vague prompt activates a broad, unpredictable region.

### Step 3: The System Prompt Sets the Frame

The system prompt (or `developer` role) sets the behavioral frame for the entire interaction. It is injected into the context before any user messages. The model treats it as a persistent instruction — not a separate "understanding" but a prefix that shapes all subsequent token predictions.

Effective system prompts are:
- **Specific:** "You are a Python expert" vs "You are helpful" — narrow scope prevents generic responses
- **Rule-based:** "Never include code comments" is more effective than "Be concise"
- **Format-focused:** "Respond in JSON format" sets the structural expectation early

### Step 4: Few-Shot Learning via In-Context Examples

Few-shot prompting means including examples of the desired input-output pattern in the prompt itself. The model uses these examples as a template for generating its own response.

```python
prompt = """
Classify the sentiment of each tweet as Positive, Negative, or Neutral.

Tweet: "I love this new phone!"
Sentiment: Positive

Tweet: "The battery died again."
Sentiment: Negative

Tweet: "It is raining today."
Sentiment: Neutral

Tweet: "This update broke everything!"
Sentiment:"""
```

The model sees three examples and learns the pattern: read the tweet, output one word. When it sees the fourth tweet, it continues the pattern by outputting "Negative".

Few-shot prompting works because the model treats your examples as part of the sequence it must continue. The more consistent your examples, the stronger the pattern.

### Step 5: Chain-of-Thought Reasoning

Chain-of-thought (CoT) prompting asks the model to reason step by step before giving the final answer. For complex tasks, CoT dramatically improves accuracy because the model has more "space" to compute intermediate results in its token predictions.

```python
prompt = """
If John has 5 apples and gives 2 to Mary, then buys 3 more, how many does he have?
Think step by step, then give the final answer.
"""
```

Without CoT, the model might guess "5 - 2 + 3 = 6" and output "6" directly. With CoT, it outputs intermediate steps: "John starts with 5. He gives 2 away → 5 - 2 = 3. He buys 3 more → 3 + 3 = 6. Final answer: 6."

### Step 6: Temperature and Sampling

The temperature parameter controls how "creatively" the model samples from its probability distribution at each token position:

- **Temperature 0.0:** The model always selects the highest-probability token. Output is deterministic — same prompt always produces the same response.
- **Temperature 0.3–0.7:** The model samples from the top tokens with some randomness. Output varies but stays in a reasonable range.
- **Temperature 0.8–1.0:** The model samples more broadly, including lower-probability tokens. Output is more creative but less reliable.

The temperature interacts with your prompt. A precise, detailed prompt at temperature 0.0 produces highly consistent output. The same prompt at 1.0 produces variations. A vague prompt at 0.0 still gives a generic "average" response.

## Cost, Limits, and Safety

### Token Costs

Prompt engineering directly affects token costs:
- Longer prompts cost more per call (input tokens are billed)
- Few-shot examples add input tokens — 5 detailed examples can add 500+ tokens
- Chain-of-thought increases output tokens (the reasoning steps are billed)
- System prompts are part of every call — keep them concise

A typical tradeoff: a well-engineered prompt may be 200 tokens longer but produces correct results on the first try, saving multiple retries. Spending tokens on a good prompt is usually cheaper than iterating on bad outputs.

### Limits

Context windows: Your entire prompt (system + examples + conversation history) must fit within the model's context window. With 128K token contexts, this is rarely a hard limit, but longer prompts slow down processing.

Prompt injection: When your prompt includes user-provided text, that text can override your instructions. Never put user content in the system prompt. Separate instructions from untrusted data.

### Safety Rules

**Never put sensitive data in prompts.** Your prompt is sent to the API provider's servers. Do not include passwords, API keys, or personal information in prompts.

**Separate instructions from data.** Structure your prompt so instructions come first and user data is clearly marked:

```python
messages = [
    {"role": "developer", "content": "You are a classifier. Respond with one word."},
    {"role": "user", "content": "---BEGIN DATA---\n" + user_text + "\n---END DATA---"}
]
```

**Be explicit about constraints.** The model will not infer unstated rules. If you need no markdown, say "Do not use markdown formatting." If you need JSON, say "Respond with valid JSON only, no other text."

**Test with temperature 0.0 first.** When developing a prompt, use `temperature=0.0` to get deterministic, repeatable results. Add temperature later for creative variation only after the prompt is reliable.

## When to Use This / When Not To

**Use prompt engineering when:**
- You need consistent, predictable output from an LLM
- You are building a system prompt for an application
- You want to reduce token waste from vague prompting
- You need the model to follow specific formats or rules
- You are debugging inconsistent model behavior

**Do not rely solely on prompt engineering when:**
- You need guaranteed structured output — use Structured Outputs (JSON schema) instead
- You need the model to learn new facts — use RAG or fine-tuning, not prompt instructions
- Your prompts are 5,000+ tokens — consider whether the task can be simplified or split
- Security is critical — prompt engineering alone cannot prevent prompt injection from user inputs

## Mental Model Check

**Question 1:** What is the fundamental difference between how a human reads a prompt and how an LLM processes it?

  A human reads words with understanding. An LLM tokenizes the text into numerical IDs and computes statistical patterns against its training data — it does not "understand" the words, it predicts the most likely token sequence based on similar patterns in its training.

---

**Question 2:** Why does adding "Think step by step" improve performance on complex reasoning tasks?

  It gives the model token "space" to compute intermediate results. Without it, the model must jump from the question directly to the answer in one step. With it, the model generates intermediate tokens that serve as scaffolding for the final answer.

---

**Question 3:** What is few-shot prompting, and why does it work?

  Few-shot prompting includes examples of desired input-output pairs in the prompt. It works because the model treats the examples as the beginning of a sequence it must continue, so it follows the pattern you established.

---

**Question 4:** What happens when you set `temperature=0.0` versus `temperature=1.0`?

  At 0.0, the model always selects the highest-probability token, producing deterministic output. At 1.0, it samples from a broader distribution, producing more varied (but less reliable) output.

---

**Question 5:** Why should user-provided content be separated from instructions in a prompt?

  To prevent prompt injection. If user content is placed in the system prompt or mixed with instructions, the user's text can override your behavioral instructions — telling the model to ignore your rules and follow the user's.
