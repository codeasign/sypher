You are three hours into debugging a production incident. The logs show a database connection pool is exhausting, but you cannot tell which service holds the most connections. You have Claude Code open in your terminal, and it can read your codebase, understand the architecture, and even suggest fixes. But every time you need to actually inspect the running system — check the database, query an API endpoint, read a configuration file from a remote server — you have to tab out, run a separate command, copy the output, and paste it back. The context switch breaks your flow. The insight you were chasing evaporates while you juggle five terminal windows.

Now imagine a different scenario. You are onboarding onto a new codebase. The project uses a PostgreSQL database, a Redis cache, and three microservices communicating over gRPC. To understand even a single feature, you need to inspect the database schema, check the cache keys, read the service logs, and trace the request flow. Without a connected toolchain, this means running a dozen separate commands across multiple terminal sessions, remembering connection strings, and manually correlating outputs. Each switch costs you time and mental energy.

Or consider the daily work of a platform engineer. You maintain a suite of internal tools: a deployment pipeline, a monitoring dashboard, a secrets manager, and a configuration database. Every time a developer asks for help, you switch contexts — open a new terminal, run a command, paste the output into Slack, and wait for the next question. With MCP connectors, those tools are directly available to Claude Code, and developers can ask questions in natural language that get answered using live data from the actual systems.

This is the gap that the Model Context Protocol, or MCP, fills. Instead of treating Claude Code as a code-only assistant that you manually feed with external data, MCP lets you connect Claude Code directly to the tools, databases, and APIs you already use. Your terminal AI assistant becomes the hub of your development workflow, not a passive participant. The protocol is open, extensible, and built into Claude Code — no plugins or third-party integrations required.

## The Concept

The Model Context Protocol (MCP) is an open standard that lets Claude Code connect directly to external tools, databases, and APIs through a unified interface.

Think of MCP like a universal power adapter for your AI assistant. Just as a travel adapter lets you plug any device into any wall socket regardless of the physical plug shape, MCP lets Claude Code connect to any tool or service regardless of how that tool exposes its functionality. The protocol defines a standard way for Claude Code to discover what tools are available, call them, and receive results — without you needing to know the details of each tool's underlying API.

MCP is not a new programming language or a complex framework. It is a JSON-RPC based protocol with a small set of message types: `listTools` for discovering capabilities, `callTool` for executing a tool, and a few lifecycle messages for initialization and shutdown. Any process that speaks this protocol can become an MCP server. This simplicity is intentional — it lowers the barrier to creating custom connectors.

## CLI Reference

The following table covers every command and flag for working with MCP connectors in Claude Code.

| Command / Flag | Syntax | What It Does | Example |
|---|---|---|---|
| /mcp | /mcp | Lists all connected MCP servers and their available tools | /mcp |
| /mcp [server] | /mcp server-name | Shows details and tools for a specific MCP server | /mcp filesystem |
| /mcp connect | /mcp connect server-name | Manually reconnects to a disconnected MCP server | /mcp connect filesystem |
| /mcp disconnect | /mcp disconnect server-name | Disconnects from a specific MCP server | /mcp disconnect database |
| /mcp restart | /mcp restart server-name | Restarts a connected MCP server | /mcp restart custom-server |
| `--mcp` | `claude --mcp "json-config"` | Passes MCP server configuration as a JSON string on launch | `claude --mcp '{}'` with JSON config string |
| --mcp-connect | claude --mcp-connect server-name | Auto-connects to a specific MCP server on launch | claude --mcp-connect filesystem |
| /mcp allow | /mcp allow tool-name | Pre-approves a specific MCP tool so it does not prompt for permission | /mcp allow read_file |
| /mcp deny | /mcp deny tool-name | Blocks a specific MCP tool from being used | /mcp deny write_file |
| /mcp logs | /mcp logs server-name | Shows the last N log lines from an MCP server | /mcp logs filesystem |
| /mcp config | /mcp config server-name | Shows the raw configuration for a specific MCP server | /mcp config filesystem |
| /mcp env | /mcp env server-name | Lists the environment variables passed to a specific MCP server | /mcp env sqlite-connector |

The three most important commands to remember are `/mcp` (list everything), `/mcp allow` (pre-approve a safe tool), and `/mcp logs` (debug a failing server). Most daily interaction with MCP servers does not require commands at all — Claude Code discovers and calls tools automatically. You only need the commands when you want to inspect the current state, manage permissions, or troubleshoot a connection issue.

The `/mcp` command without arguments is the first thing to run when you want to verify that your servers are connected. It shows each server name, its status (running or disconnected), and all tools it exposes. This is also useful for discovering what a new MCP server offers before you ask Claude Code to use it.

The diagram above shows the full flow. Claude Code does not talk directly to any external tool. Instead, it communicates through the MCP client built into Claude Code, which sends requests over the transport layer (either stdio for local servers or SSE for remote servers) to an MCP server process. The MCP server exposes a registry of tools, each of which wraps a specific capability: reading files, querying a database, calling a GitHub API, or running a Docker command. Each tool call returns structured results back through the same path.

The key insight is that the MCP server runs as a separate process. This means it can have its own dependencies, its own lifecycle, and its own security context. A crash in the MCP server does not crash Claude Code — it simply disconnects the server, and you can reconnect with `/mcp connect`.

## How It Actually Works — Step by Step

### Step 0: The MCP Protocol Handshake

Before any tools can be used, Claude Code and the MCP server perform a handshake. Claude Code sends an `initialize` request with its protocol version and capabilities. The server responds with its own protocol version and capabilities. If the versions are compatible, Claude Code sends an `initialized` notification, and the session begins. This handshake ensures that both sides understand the message format and can communicate reliably.

### Step 1: Server Configuration

MCP servers are defined in your project's `.claude/settings.json` file or in the global `settings.json`. Each server entry specifies the command to launch it, any arguments, and environment variables. When Claude Code starts, it reads this configuration and spawns each MCP server as a child process. Configuration can live at the project level (`.claude/settings.json` in your project root) or at the user level (in your home directory). Project-level settings merge with user-level settings, with project settings taking precedence for conflicting keys.

The spawning process is sequential. Claude Code processes each server entry one at a time, waiting for the handshake to complete before moving to the next server. This prevents race conditions and ensures that all servers are fully initialized before any conversation starts. If a server fails to start, Claude Code logs the error and continues with the remaining servers — a single failed server does not block the entire session.

### Step 2: Transport Layer Initialization

Each spawned MCP server communicates over a transport. There are two transport types: **stdio** (standard input/output) for local servers running as child processes, and **SSE** (Server-Sent Events) for remote servers that run on a different machine. The transport layer handles message framing, delivery, and error recovery. With stdio transport, each JSON-RPC message is a single line on stdout, and incoming messages arrive on stdin. With SSE transport, the server sends an HTTP response with a persistent connection and the client sends messages via HTTP POST.

The transport layer is also responsible for detecting disconnections. If a stdio server process exits unexpectedly, Claude Code detects the child process exit and marks the server as disconnected. For SSE servers, a dropped connection is detected when the HTTP stream ends or a heartbeat times out. In both cases, you can reconnect without restarting Claude Code.

Choosing between stdio and SSE depends on your use case. stdio is simpler and has lower latency because there is no HTTP overhead. For a local development MCP server that only you use, stdio is almost always the right choice. For a shared team server that provides access to a common database or API, SSE allows multiple developers to connect simultaneously without each running their own server process. It is the right choice for local development servers. SSE is appropriate when the MCP server runs on a different machine, when you want to share one server across multiple team members, or when the server needs to run persistently outside a Claude Code session.

### Step 3: Tool Discovery (ListTools)

Once the transport is established, the MCP client sends a `listTools` request to the server. The server responds with a list of all available tools, each containing a name, description, and input schema. This is how Claude Code learns what capabilities are available — it never guesses or assumes what tools exist. The input schema uses JSON Schema format, which allows Claude Code to understand what parameters each tool expects and what types those parameters should be.

Tool discovery happens automatically at startup, but it can also happen later if you add or remove tools from a running server. Claude Code caches the tool list for the duration of a session. If you modify your MCP server's tools while it is running, you can restart the server with `/mcp restart server-name` to trigger a fresh tool discovery cycle.

### Step 4: Tool Call Execution

When Claude Code decides to use a tool, it sends a `callTool` request with the tool name and parameters. The MCP server executes the tool and returns the result. The result can be text, structured data (JSON), images, or error messages. Claude Code then incorporates this result into its reasoning. The tool call is synchronous from Claude Code's perspective — it sends the request and waits for the response. The MCP server can perform any operation during this time: read files, query databases, call HTTP APIs, or run shell commands.

The tool handler function receives the arguments as a plain object and must return a structured response. The response content can include multiple parts of different types. For example, a database tool could return both a text summary and a JSON data blob in the same response. This flexibility allows you to design tools that serve both the conversational flow and the data analysis needs of Claude Code.

### Step 5: Permission Model and Safety

Before any MCP tool executes, Claude Code checks the permission model. By default, every tool call requires your explicit approval. You see a prompt in the terminal showing the tool name, parameters, and the MCP server it belongs to. You can approve once, approve always for that tool, or deny. This prevents Claude Code from accidentally running destructive operations. The permission model is granular at the tool level, not the server level — you can allow `read_file` while denying `write_file` on the same server.

The permission state is persistent across sessions for the same project. If you approve a tool with "Allow Always" today, it will still be approved tomorrow when you start a new Claude Code session in the same project directory. This means you only need to grant permissions once per tool, reducing friction over time without sacrificing safety upfront.

### Step 6: Result Processing and Context Integration

Once the tool result arrives, Claude Code integrates it into its ongoing reasoning. The result is treated as new context — it can inform the next question, the next tool call, or the final answer. This cycle repeats: Claude Code thinks, calls a tool, gets a result, thinks again, and potentially calls another tool. This is how multi-step workflows emerge naturally.

The context integration is not a simple append. Claude Code evaluates the tool result against its current plan and decides whether the result answers the question, raises new questions, or requires further investigation. This is the same reasoning process that Claude Code uses for code analysis — it treats tool results as first-class evidence, not as raw data to pass through. A tool result that contradicts an earlier assumption will cause Claude Code to revise its understanding, just as reading a conflicting source file would.

### Error Handling and Recovery

When an MCP server returns an error, Claude Code receives it as part of the tool result. The error includes a message string and an optional error code. Claude Code can use this information to retry the operation with different parameters, inform you of the failure, or try an alternative approach. If the MCP server process itself crashes or disconnects, Claude Code marks it as disconnected and shows a warning. You can restart it with `/mcp restart server-name` without restarting Claude Code itself.

There are two categories of errors you will encounter. **Tool-level errors** happen when the tool runs but encounters an application-level problem, such as a file not found or a database query failure. These errors are returned as part of the normal tool response and do not affect the server's connection status. **Server-level errors** happen when the MCP server process itself crashes, hangs, or loses its connection. These errors disconnect the server and require a restart. The `/mcp logs` command is useful for diagnosing both types of errors, as it shows the server's stderr output where most diagnostics are logged.

When a tool-level error occurs, Claude Code will typically inform you of the error and suggest alternatives. For example, if a file read tool returns "file not found," Claude Code might suggest checking the file path or listing the directory contents to find the correct filename. This feedback loop is part of what makes MCP connectors feel natural — Claude Code does not just report errors, it works around them.

### MCP Server Lifecycle Management

Each MCP server follows a defined lifecycle: started, connected, active, and disconnected. On startup, Claude Code reads the configuration and spawns all configured servers. Each server goes through the handshake and tool discovery phases. Once connected, the server remains active for the duration of the Claude Code session. When you exit Claude Code, all child process servers are terminated automatically.

You can manage the lifecycle during a session with these commands: `/mcp connect server-name` to start a server that was disconnected, `/mcp disconnect server-name` to gracefully stop a running server, and `/mcp restart server-name` to stop and restart a server. Restarting is useful when you modify the server code and want to pick up the changes without restarting Claude Code. The lifecycle is designed to be flexible — servers can come and go without affecting the core conversation.

### Debugging MCP Connections

When an MCP server fails to connect or behaves unexpectedly, you have several debugging tools. The first step is always `/mcp logs server-name`, which shows the last N log lines from the server process. These logs include stdout and stderr output from the server, which typically contains error messages, stack traces, and status information.

Common connection issues include:
- **Command not found**: The `command` field in settings.json points to a binary that does not exist on your system. Verify the command is installed and accessible from your shell.
- **Missing environment variables**: If a required `${VAR_NAME}` is not set in your environment, the server will fail to start. Run `/mcp env server-name` to see what variables were passed.
- **Port conflicts**: For SSE servers, the specified port may already be in use. Choose a different port or stop the process using the port.
- **Permission issues**: The server process may not have permission to read the files or directories it needs. Check the user that Claude Code runs as and the permissions of the target paths.

If the server starts but tools do not appear, the issue is likely in the `listTools` handler. Run the server standalone from your terminal to see its full output, including any errors during tool registration. For Node.js servers, you can run `node server.js` directly and watch for any error messages. Once a server is working standalone, it will work in Claude Code. The standalone test is the most reliable way to isolate server-side issues from connection issues.

## Cost, Limits, and Safety

### Cost Implications

Every MCP tool call consumes tokens. The tool call itself is a message sent to the model, and the tool result is additional context. A single tool call might use 500-2000 tokens depending on the size of the result. If you are using a paid API plan, each tool call adds to your token usage. With the Claude Code Pro plan, tool calls are included in your subscription, but excessively long chains of tool calls can still hit context window limits. A chain of 10 tool calls with large result sets could consume 10,000-20,000 tokens or more.

To minimize token consumption, design your tools to return only the data you need. A tool that returns a filtered or summarized result uses fewer tokens than one that returns the full raw output. For example, a database query tool should accept a LIMIT parameter and return column names rather than dumping entire tables. Streaming large results through multiple smaller tool calls is more expensive than one well-designed tool call that returns exactly what Claude Code needs.

### Rate Limits and Guardrails

MCP servers themselves do not have built-in rate limits — that is up to the server implementation. However, Claude Code imposes a maximum number of sequential tool calls per conversation turn. This prevents runaway loops where Claude Code keeps calling tools indefinitely. The default limit is 20 tool calls per turn. If you hit this limit, you will see a message indicating the tool call budget was exhausted, and you can continue the conversation with a new turn.

This limit is configurable via the `--max-tool-calls` flag or by setting the `ANTHROPIC_MAX_TOOL_CALLS` environment variable. You should only increase this limit when you are deliberately building long multi-step workflows and understand the token cost implications. For example, a data migration script that processes records in batches might legitimately need 50 tool calls. In that case, increase the limit temporarily and then restore it to the default when the migration is complete.

When you hit the tool call limit, you see a message like "Tool call budget exhausted" and Claude Code will summarize what it has learned so far. You can then start a new turn and continue. This is not a hard stop — it is a safety valve that prevents unbounded execution.

### Environment Variables

Sensitive credentials for MCP servers should always be read from environment variables, never hardcoded into settings.json. You can reference environment variables in your MCP server configuration:

```json
{
  "mcpServers": {
    "database": {
      "command": "npx",
      "args": ["-y", "@your-org/mcp-database-server"],
      "env": {
        "DB_HOST": "${DB_HOST}",
        "DB_PORT": "${DB_PORT}",
        "DB_USER": "${DB_USER}",
        "DB_PASSWORD": "${DB_PASSWORD}"
      }
    }
  }
}
```

Claude Code resolves `${VAR_NAME}` patterns from the environment at launch time. If a variable is not set, the server will fail to start, and you will see an error in the MCP server logs. You can verify which environment variables were passed to a server by running `/mcp env server-name`.

### Safety Best Practices

The permission model is your first line of defense, but it is not the only one. You should also design your MCP servers with safety in mind from the start.

- Use `/mcp allow` sparingly — only for read-only tools you trust completely
- Never pre-approve write or delete operations on production systems
- Run MCP servers with the minimum permissions needed (principle of least privilege)
- Use the `env` field in settings.json to pass credentials, never inline strings
- Test new MCP servers in a sandbox environment before connecting them to production
- Monitor MCP server logs with `/mcp logs` to detect unexpected behavior
- Disconnect unused MCP servers with `/mcp disconnect` to reduce attack surface
- Audit the source code of any MCP server from an untrusted source before running it
- Review the tool list returned by `/mcp` regularly to ensure no unexpected tools are registered
- Consider running MCP servers in a Docker container for additional isolation, especially for servers that need filesystem access
- For team-shared MCP servers, use SSE transport on an internal network and restrict access with firewall rules
- When building a custom MCP server, always validate inputs in the tool handler, never assume Claude Code will send valid parameters
- Use the `isError` flag in tool responses to indicate failures clearly, so Claude Code can report them accurately

## When to Use This / When Not To

### Use MCP Connectors When

- You need to inspect or modify files on your local system during a Claude Code session without switching terminals
- You want Claude Code to query a database directly instead of you running queries manually and pasting results
- You are building a custom tool or script that you want Claude Code to be able to invoke on demand
- You want to integrate Claude Code with your existing infrastructure — Docker, cloud APIs, monitoring tools, CI/CD pipelines
- You are debugging a multi-service system and need to check logs, configs, and databases across services from one place
- You want to automate repetitive tasks that involve multiple external tools — for example, deploying a service, checking its health, and reporting the status
- You are onboarding onto a new codebase and want to explore the database schema, configuration files, and service endpoints through conversation
- You need to troubleshoot a CI/CD pipeline failure and want Claude Code to inspect build logs, check environment variables, and query deployment status in one session
- You are running a code review and need to verify that database migrations match the schema expected by the application code
- You are writing documentation and want to include live examples from your actual database or API responses
- You are conducting a security audit and need to verify file permissions, database access controls, and API endpoint configurations

### Do Not Use MCP Connectors When

- The task is purely conversational and does not require external data — MCP adds unnecessary overhead and permission prompts
- The external tool has a high-latency API that would make the interaction feel sluggish (more than a few seconds per call)
- You are working with highly sensitive production data and the MCP server does not have proper access controls in place
- The tool you want to connect already has a built-in Claude Code integration (like git operations)
- Your environment has strict security policies that forbid running external processes from the AI assistant
- The MCP server you are considering is untrusted or from an unknown source — you should always audit the source code first
- You are prototyping something that will only run once and the setup cost of creating an MCP server exceeds the manual effort of running the commands yourself
- The tool depends on interactive input that cannot be provided through the tool's parameters — MCP tools are best for deterministic operations with well-defined inputs

The position here is clear: MCP connectors are transformative for development workflows, but they shift the responsibility of safety from the tool to you. The protocol is open and powerful, which means you must be deliberate about which servers you connect and what permissions you grant. Start with read-only tools, audit server code, and gradually expand as you build trust. The investment in setting up MCP servers pays off quickly when you find yourself debugging production issues without leaving your terminal.

MCP connectors are especially powerful when combined with the other capabilities of Claude Code that you have learned in this course. The slash commands, file editing, git integration, and code analysis features all work alongside MCP tools. You can ask Claude Code to read a configuration file via an MCP tool, analyze the code that uses that configuration, and suggest a fix — all in the same conversation. The connectors are not a replacement for Claude Code's built-in features; they are an extension that brings external systems into the same unified workflow.

A good rule of thumb is to ask yourself: "Would this task be faster if I could ask Claude Code to do it directly?" If the answer is yes, and the tool does not perform destructive operations, an MCP connector is likely the right choice. If the answer is no, or if the tool performs irreversible actions on critical systems, proceed with caution and always use the permission model to gate each operation.

As you build more MCP connectors, you will develop intuition for which tools are worth the setup effort. A simple read-only tool that saves you five terminal switches per day pays for itself within a week. A complex write tool that you use once a month may not be worth the maintenance cost. Start with the high-value, low-risk tools and expand outward.

## Mental Model Check

Test your understanding with these five questions. Try to answer each one before expanding the answer.

### Question 1

What is the MCP transport layer, and what are the two types of transports available?

You can think of the transport layer as the communication channel between Claude Code and the MCP server. The two transport types are **stdio** (standard input/output) for local servers running as child processes, and **SSE** (Server-Sent Events) for remote servers running on a different machine. stdio is simpler and faster for local use, while SSE allows MCP servers to run anywhere on your network as persistent services.

### Question 2

How does Claude Code discover what tools an MCP server provides?

Claude Code sends a `listTools` request to the MCP server over the transport layer. The server responds with a JSON-RPC message containing an array of tool definitions. Each tool definition includes a name, a human-readable description, and an input schema that specifies what parameters the tool accepts. Claude Code uses this information to decide which tool to call based on the task at hand. The discovery happens automatically at startup and can be triggered again if tools change.

### Question 3

What happens if you do not explicitly approve a tool call?

By default, every MCP tool call requires your approval. If you do not approve, the tool call does not execute. Claude Code will see the permission prompt and wait for your response. If you deny the request, the tool call is blocked and Claude Code receives an error, which it will communicate to you. You can also use `/mcp allow` to pre-approve specific tools so you are not prompted every time. Pre-approval is persistent across sessions for the same project.

### Question 4

How can you pass environment variables to an MCP server without hardcoding secrets?

You use the `env` field in the MCP server configuration inside `.claude/settings.json`. You reference environment variables with the `${VAR_NAME}` syntax. Claude Code resolves these patterns from your shell environment at launch time. For example, `"DB_PASSWORD": "${DB_PASSWORD}"` reads the password from the `DB_PASSWORD` environment variable rather than storing it in the settings file. If a variable is not set, the server will fail to start with a clear error message.

### Question 5

What is the maximum number of sequential tool calls Claude Code allows per conversation turn, and why does this limit exist?

The default limit is 20 sequential tool calls per conversation turn. This limit exists to prevent runaway loops where Claude Code might keep calling tools indefinitely, either due to a bug in its reasoning, a poorly designed tool that returns confusing results, or a malicious server that sends responses designed to trigger more calls. The limit ensures you always have a chance to intervene and redirect the conversation. You can increase this limit if needed, but you should understand the token cost implications first.

### Question 6

What is the difference between stdio and SSE transport, and when would you choose one over the other?

stdio transport uses standard input and output for communication between Claude Code and the MCP server. The server runs as a child process of Claude Code, and messages are JSON-RPC frames on stdin and stdout. SSE transport uses HTTP for communication, with the server running as a standalone HTTP server. Choose stdio for local development servers where latency matters and you want simple process management. Choose SSE when the MCP server needs to run on a different machine, serve multiple clients, or persist independently of any Claude Code session.
