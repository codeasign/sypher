## The Build -- Step by Step

You are going to build a research agent that, when asked a complex question, plans what information it needs, searches for each piece of information (simulated with mock data), synthesizes the results, and presents a structured comparison. The agent will use the planning and execution pattern you learned in the overview.

The core innovation is the **planning phase**. Before the agent performs any work, it creates a structured plan as a list of steps with dependencies. Then it executes each step, passing results from one step to the next. If a step fails, it re-plans.

### Step 1: Define the Plan Structure

Create `research_agent.py`. Start with the data structures that represent a plan:

```python
from anthropic import Anthropic
import json
import os
from typing import Optional

client = Anthropic()

class PlanStep:
    def __init__(self, step: int, task: str, depends_on: list[int], result: Optional[str] = None):
        self.step = step
        self.task = task
        self.depends_on = depends_on
        self.result = result

    def to_dict(self):
        return {
            "step": self.step,
            "task": self.task,
            "depends_on": self.depends_on,
        }

    def __repr__(self):
        deps = f"depends on {self.depends_on}" if self.depends_on else "no deps"
        status = f"result: {self.result[:50]}..." if self.result else "pending"
        return f"Step {self.step}: {self.task} ({deps}, {status})"
```

Each step has a step number, a task description, a list of step indices it depends on, and a placeholder for the result.

### Step 2: Build the Planner

The planner takes a goal and produces a structured plan. It uses the LLM to decompose the goal into sub-tasks with dependencies.

```python
def create_plan(goal: str) -> list[PlanStep]:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        messages=[
            {
                "role": "user",
                "content": f"""
You are a planning agent. Given a research goal, create a plan of sub-tasks.

Goal: "{goal}"

Create 4-6 sub-tasks that break this goal into logical steps.
Each sub-task must have:
- step: integer starting from 0
- task: what to do
- depends_on: list of step indices this task depends on (empty list for no deps)

Return ONLY a JSON array of objects with fields: step, task, depends_on.
"""
            }
        ]
    )

    raw = response.content[0].text
    steps_data = json.loads(raw)
    return [PlanStep(**s) for s in steps_data]
```

### Step 3: Build the Scheduler

The scheduler determines which steps can run in parallel. Steps with no dependencies run first. Steps whose dependencies are met run next.

```python
def schedule(plan: list[PlanStep]) -> list[list[int]]:
    """
    Returns batches of step indices that can execute in parallel.
    """
    batches = []
    completed = set()
    remaining = set(range(len(plan)))

    while remaining:
        batch = []
        for i in list(remaining):
            if all(dep in completed for dep in plan[i].depends_on):
                batch.append(i)
        if not batch:
            print("Warning: circular dependency detected")
            break
        for i in batch:
            remaining.remove(i)
        batches.append(batch)
        completed.update(batch)

    return batches
```

Test the scheduler:

### Step 4: Build the Mock Data Source

Since you do not have access to live pricing APIs, you will create a mock data source that returns realistic pricing data. This simulates the "search" step of the research agent.

```python
MOCK_PRICING_DATA = {
    "openai": {
        "gpt-4o": {"input": 2.50, "output": 10.00, "unit": "per 1M tokens"},
        "gpt-4o-mini": {"input": 0.15, "output": 0.60, "unit": "per 1M tokens"},
        "gpt-4-turbo": {"input": 10.00, "output": 30.00, "unit": "per 1M tokens"},
    },
    "anthropic": {
        "claude-sonnet-4-20250514": {"input": 3.00, "output": 15.00, "unit": "per 1M tokens"},
        "claude-haiku-3-5": {"input": 0.80, "output": 4.00, "unit": "per 1M tokens"},
        "claude-opus-4": {"input": 15.00, "output": 75.00, "unit": "per 1M tokens"},
    }
}

def search_pricing(provider: str) -> dict:
    """
    Simulates searching for pricing data for a given provider.
    """
    provider = provider.lower()
    if provider in MOCK_PRICING_DATA:
        return MOCK_PRICING_DATA[provider]
    else:
        return {"error": f"No pricing data found for {provider}"}
```

### Step 5: Build the Executor

The executor runs each step using the LLM, passing context from completed steps. For steps that involve searching, it calls the mock data source instead.

```python
def execute_step(step: PlanStep, context: dict) -> str:
    """
    Execute a single plan step. Returns the result as a string.
    """
    # Check if this step is a data-fetching task
    task_lower = step.task.lower()

    if "openai" in task_lower and "research" in task_lower:
        data = search_pricing("openai")
        return json.dumps(data, indent=2)

    if "anthropic" in task_lower and "research" in task_lower:
        data = search_pricing("anthropic")
        return json.dumps(data, indent=2)

    # For synthesis steps, use the LLM
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=2048,
        messages=[
            {
                "role": "user",
                "content": f"""
Task: {step.task}

Context from previous steps:
{json.dumps(context, indent=2)}

Execute this task using the context provided. Return your findings.
"""
            }
        ]
    )
    return response.content[0].text
```

### Step 6: Build the Main Agent Loop

Now you tie everything together. The agent creates a plan, schedules it, and executes each batch. After all steps complete, it verifies the results.

```python
def run_research_agent(goal: str) -> dict:
    print(f"=== Research Agent ===")
    print(f"Goal: {goal}\n")

    # Phase 1: Create the plan
    print("Phase 1: Creating plan...")
    raw_plan = create_plan(goal)
    plan = [PlanStep(**s) if isinstance(s, dict) else s for s in raw_plan]
    for p in plan:
        print(f"  {p}")
    print()

    # Phase 2: Schedule the plan
    print("Phase 2: Scheduling...")
    batches = schedule(plan)
    for i, batch in enumerate(batches):
        print(f"  Batch {i}: steps {batch}")
    print()

    # Phase 3: Execute
    print("Phase 3: Executing...")
    context = {}
    for batch_idx, batch in enumerate(batches):
        print(f"\n--- Batch {batch_idx} ---")
        for step_idx in batch:
            step = plan[step_idx]
            print(f"  Executing step {step.step}: {step.task}")
            result = execute_step(step, context)
            step.result = result
            context[step.step] = result
            print(f"  Result: {result[:100]}...")
            print()

    # Phase 4: Verify completion
    print("Phase 4: Verifying completion...")
    all_results = {s.step: s.result for s in plan}
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        messages=[
            {
                "role": "user",
                "content": f"""
Original goal: {goal}
All step results: {json.dumps(all_results, indent=2)}

Is the goal fully met? Return JSON:
{{"goal_met": bool, "missing_items": list, "summary": str}}
"""
            }
        ]
    )
    verification = json.loads(response.content[0].text)
    print(f"  Goal met: {verification['goal_met']}")
    print(f"  Summary: {verification['summary']}")

    return {
        "plan": plan,
        "batches": batches,
        "results": all_results,
        "verification": verification,
    }
```

### Step 8: Add Re-Planning on Failure

The agent above works when everything goes right. Now add error handling. Modify the executor to detect failures and trigger re-planning.

```python
def execute_with_replan(plan: list[PlanStep], goal: str) -> dict:
    """
    Execute the plan with re-planning on failure.
    """
    context = {}
    batches = schedule(plan)
    max_replans = 3
    replan_count = 0

    for batch_idx, batch in enumerate(batches):
        print(f"\n--- Batch {batch_idx} ---")
        for step_idx in batch:
            step = plan[step_idx]
            print(f"  Executing step {step.step}: {step.task}")

            try:
                result = execute_step(step, context)
                step.result = result
                context[step.step] = result
                print(f"  Result: {result[:80]}...")
            except Exception as e:
                error_msg = str(e)
                print(f"  FAILED: {error_msg}")

                if replan_count >= max_replans:
                    raise RuntimeError("Max re-plans exceeded. Aborting.")

                # Re-plan from this step
                print(f"  Re-planning... (attempt {replan_count + 1})")
                replan_count += 1

                response = client.messages.create(
                    model="claude-sonnet-4-20250514",
                    max_tokens=1024,
                    messages=[
                        {
                            "role": "user",
                            "content": f"""
Original plan: {json.dumps([p.to_dict() for p in plan], indent=2)}
Step {step.step} failed: {step.task}
Error: {error_msg}
Completed steps: {json.dumps(context, indent=2)}

Create a revised plan starting from step {step.step}.
Return a JSON array of {{"step": int, "task": str, "depends_on": list[int]}}.
"""
                        }
                    ]
                )
                new_steps = json.loads(response.content[0].text)
                # Replace remaining steps with new plan
                for s in new_steps:
                    new_step = PlanStep(**s)
                    if new_step.step not in context:
                        plan.append(new_step)
                # Recalculate batches
                batches = schedule(plan)
                break

    return context
```

## Common Errors

### Error 1: Plan Too Granular

A plan that breaks every tiny action into its own step wastes tokens and creates unnecessary overhead. If you ask the agent to "plan how to turn on a light" and it creates 20 steps (walk to switch, extend arm, grasp switch, rotate finger, etc.), the planning overhead exceeds the task value.

**Fix:** Set a minimum step granularity. Each step should represent a meaningful unit of work. A good rule: each step should take at least one LLM call or tool invocation to complete.

### Error 2: Plan Too Vague

A plan with steps like "do research" or "get data" is useless because the executor has no concrete instructions. The step must describe what to do, what tool to use, and what success looks like.

**Fix:** Each step task should include the action verb, the target, and the expected output format. Instead of "get data," write "call the pricing API for provider X and return the JSON response."

### Error 3: Re-Planning Loop

If the re-planning logic produces the same plan that just failed, the agent loops forever. This happens when the error is structural (e.g., the API key is invalid) but the re-planner tries to re-plan around it instead of reporting the failure.

**Fix:** Set a maximum re-plan count (3 is a good default). Track which steps have failed and prevent re-planning if the same step fails twice. If re-planning produces the same plan, abort.

```python
def detect_replan_loop(original_plan: list[dict], new_plan: list[dict]) -> bool:
    """Returns True if the new plan is effectively the same as the original."""
    original_tasks = [s["task"] for s in original_plan]
    new_tasks = [s["task"] for s in new_plan]
    return original_tasks == new_tasks
```

### Error 4: Task Dependency Not Detected

The LLM may miss implicit dependencies. For example, comparing pricing requires knowing both pricing sets first, but the LLM might create a plan that starts comparing before fetching both data sources.

**Fix:** Add a validation step after plan creation that checks for missing dependencies. You can ask the LLM to review its own plan and identify any missed dependencies.

```python
def validate_plan(plan: list[PlanStep]) -> list[str]:
    """Check for common dependency issues. Returns list of warnings."""
```

## What You Just Built

- **A planning agent** that decomposes a complex goal into sub-tasks before executing
- **A dependency-aware scheduler** that groups independent tasks for parallel execution
- **A step executor** that passes context from completed steps to dependent steps
- **A re-planning mechanism** that adjusts the plan when a step fails
- **A verification step** that checks goal completion after all steps execute
- **A mock data simulation** that lets you test the agent without live API calls

You now have a reusable research agent pattern. To use it with real data sources, replace the `search_pricing` mock function with actual API calls, web scraping, or database queries. The planning and execution infrastructure remains the same.
