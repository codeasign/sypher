## The Problem Story

It is 2:00 AM. Your first AI agent -- a support triage bot -- is running in
production. A customer submits a ticket: "My API key expired and I cannot
access the dashboard."

Your agent reads the ticket. It decides to call the `check_account_status`
tool. The tool returns "account active, API key expired yesterday." The agent
calls `send_password_reset` to fix it. Then it stops. It never checks whether
the reset email actually arrived. It never calls `verify_email_delivery`. It
never tells the customer what happened. One action, then silence.

On the other side of the spectrum is the runaway agent. You gave it a simple
task: "Find all unused cloud resources and delete them." It starts listing
resources. It deletes one. It lists again. It finds a related resource. It
deletes that too. It spirals. Three hundred and forty-two tool calls and
seventy-two dollars in API costs later, you manually kill the process. The
agent had no concept of "done."

Both failures share the same root cause: **the agent had no loop structure**.
The first agent had no loop at all -- it ran once and stopped regardless of
outcome. The second had a loop with no termination condition -- it ran forever
because nothing told it to stop.

These are not edge cases. They are the default behavior of naive agents. The
fix is the agent loop: a control structure that governs how an agent perceives
state, decides on action, executes, observes results, and repeats -- until the
task is complete or a clear termination condition is met.

## The Concept

The agent loop is the control structure that governs how an agent perceives
its state, decides on an action, executes it, observes the result, and repeats
-- until the task is complete or a termination condition is met.

Think of a chef preparing a sauce. The chef does not blindly follow a recipe
from start to finish without ever tasting. The chef tastes the sauce, decides
it needs more salt, adds salt, stirs, tastes again. Maybe it needs more acid
now. Add a squeeze of lemon. Taste again. The loop continues until the sauce
is balanced. Each cycle is one iteration of the taste-adjust-taste loop. The
agent loop works the same way: perceive the current state, decide on the next
action, execute it, observe what changed, and decide again.

A non-loop approach would be the chef who follows the recipe exactly as
written, never tastes, and serves whatever comes out. Sometimes it works.
Sometimes the sauce is bland or too salty and the chef never knows because
there was no feedback loop. That is an agent that takes one action and stops.

A runaway loop would be the chef who keeps adding salt, tasting, adding more
salt, tasting again, never deciding the sauce is good enough. That is the
agent that calls tools in an infinite spiral.

A well-structured agent loop has three critical pieces:

- **A decision mechanism**: the LLM decides what to do next based on the
  current state, the conversation history, and the available tools.
- **An execution step**: the application runs the tool the LLM chose and
  captures the result.
- **A termination condition**: something that tells the loop to stop -- the
  LLM produces a final answer, the loop hits a maximum number of iterations,
  token limits are reached, or a human intervenes.

Every production AI agent you have ever used is running some form of this
loop. The difference between a reliable agent and a frustrating one is how
well the loop is built.

## API / SDK Reference

The agent loop relies on a few core concepts in the Anthropic SDK and the
Claude API. These are the building blocks you will use when implementing your
own loop.

### Stop Reason

Every response from the API includes a `stop_reason` field that tells you why
the model stopped generating. This is the primary signal your loop uses to
decide what to do next.

| stop_reason | Meaning | Loop action |
|---|---|---|
| `end_turn` | The model finished its response normally | Return the final answer to the user |
| `tool_use` | The model wants to call a tool | Execute the requested tool and append the result |
| `max_tokens` | The response hit the token limit | Truncate, summarize, or continue with context |
| `stop_sequence` | The model hit a custom stop sequence | Return the response up to the stop point |

### Max Tokens per Turn

The `max_tokens` parameter controls how many tokens the model can generate in
a single response. For agent loops, this limits the length of a single
reasoning or tool-calling turn. Common values range from 1024 for simple tool
calls to 8192 for complex multi-step reasoning.

### Max Tool Calls per Turn

The API allows the model to call multiple tools in a single response (tool
use batching). The `max_tool_calls_per_turn` parameter controls how many tool
calls the model can make in one response. The default is 1. Setting this
higher allows the model to parallelize independent tool calls.

### Context Window

The context window is the total number of tokens the model can process in a
single request, including system prompt, conversation history, tool results,
and the model's response. For Claude Sonnet 4, this is 200,000 tokens. In an
agent loop, every iteration appends more tokens to the history, so the
context window fills over time. Managing this is critical for long-running
agents.

### Tool Definition Format

Tools are defined as JSON objects in the `tools` parameter. Each tool has a
name, description, and input_schema. The model uses the name and description
to decide when to call the tool. A clear description is essential for the
model to choose the right tool.

```json
{
  "name": "get_weather",
  "description": "Get the current weather for a location",
  "input_schema": {
    "type": "object",
    "properties": {
      "location": {
        "type": "string",
        "description": "City name, e.g. San Francisco"
      }
    },
    "required": ["location"]
  }
}
```

### Tool Result Format

When the model calls a tool, your application runs the tool function and
returns the result as a `tool_result` content block. The result must include
the `tool_use_id` from the model's request so the API can match the result to
the original call.

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_abc123",
      "content": "The current weather in San Francisco is 18 degrees and foggy."
    }
  ]
}
```

### Iteration Limit

A hard cap on the number of loop iterations prevents runaway agents. For most
tasks, 3 to 10 iterations is sufficient. The iteration limit is your safety
net -- it ensures the agent stops even if the termination logic fails.

### Token Usage Tracking

Each API response includes usage statistics in `response.usage`:
- `input_tokens`: tokens in the request
- `output_tokens`: tokens in the response
- `cache_creation_input_tokens`: tokens written to the cache (if using
  prompt caching)
- `cache_read_input_tokens`: tokens read from the cache (if using prompt
  caching)

Tracking these across iterations lets you monitor cost and detect runaway
token consumption before it becomes a problem.

## How It Actually Works -- Step by Step

Building an agent loop means understanding the flow at each level: the
messages array, the API call, the stop_reason check, the tool execution, and
the loop management. Here is the detailed breakdown.

### Step 1: Initialization

When a user sends a request, the agent loop initializes a messages array with
the user's input. This array is the single source of truth for the
conversation. Everything the LLM sees comes from this array, and everything
the loop does appends to it.

The initial messages array looks like this:

```json
[
  {
    "role": "user",
    "content": "What is the weather in San Francisco?"
  }
]
```

At this point, the loop counter is set to 0 and the maximum iterations (for
example, 10) is established. The loop is ready to begin.

### Step 2: The Think Phase (API Call)

The loop sends the messages array along with the tool definitions to the API.
This is the think phase -- the LLM processes the entire context and decides
what to do next.

```python
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=4096,
    messages=messages,
    tools=tools,
)
```

The LLM considers every message in the array, evaluates the available tools
based on their descriptions, and generates a response. The response includes a
`stop_reason` that tells your loop what happened.

### Step 3: The Act Phase (Stop Reason Check)

The loop checks the `stop_reason` field on the response. This is the critical
decision point that determines the next action.

```python
if response.stop_reason == "end_turn":
    # The model is done. Return the final text response.
    return response.content[0].text
elif response.stop_reason == "tool_use":
    # The model wants to call a tool. Extract the call details.
    for block in response.content:
        if block.type == "tool_use":
            tool_name = block.name
            tool_args = block.input
            tool_id = block.id
```

The stop_reason is the loop's compass. `end_turn` means the model has a
complete answer. `tool_use` means the model needs more information or wants
to take an action. `max_tokens` means the response was cut short.

### Step 4: The Observe Phase (Tool Execution)

When the model calls a tool, the loop must execute the corresponding function
in your application. The tool name from the API maps to a function in your
code. You extract the arguments, call the function, and capture the result.

```python
def execute_tool(tool_name, tool_args):
    if tool_name == "get_weather":
        location = tool_args["location"]
        result = get_weather(location)
        return result
    elif tool_name == "search_database":
        query = tool_args["query"]
        result = search_database(query)
        return result
```

The tool result is then formatted as a `tool_result` content block and
appended to the messages array. This is the observe phase -- the agent
observes the outcome of its action.

```python
messages.append({
    "role": "user",
    "content": [
        {
            "type": "tool_result",
            "tool_use_id": tool_id,
            "content": str(result),
        }
    ],
})
```

### Step 5: Loop Iteration

After appending the tool result, the loop counter increments and the loop
goes back to Step 2 -- the think phase. The LLM now sees the original
question, its own tool call, and the tool result in the conversation history.
It can decide what to do next.

This is the heart of the agent loop: each iteration adds to the context, and
the LLM builds on the accumulated information. The agent is not starting over
each time -- it is continuing a conversation with itself.

### Step 6: Termination

The loop terminates when one of these conditions is met:

1. **end_turn**: The model produces a stop_reason of `end_turn`, meaning it
   has a complete response. The loop returns the final text.

2. **Max iterations**: The loop counter reaches the maximum allowed
   iterations (for example, 10). The loop returns whatever the model last
   produced, along with a warning that the task may be incomplete.

3. **Max tokens**: A response hits the output token limit. The loop can
   either return the partial response or make one more call asking the model
   to continue.

4. **Human interruption**: A human stops the loop.

Here is how a complete loop looks in code:

```python
import anthropic
from anthropic import Anthropic

client = Anthropic()

messages = [{"role": "user", "content": "What is the weather in San Francisco?"}]
max_iterations = 10
iteration = 0

while iteration < max_iterations:
    iteration += 1
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=4096,
        messages=messages,
        tools=tools,
    )

    if response.stop_reason == "end_turn":
        print(f"Task complete after {iteration} iteration(s)")
        print(response.content[0].text)
        break

    elif response.stop_reason == "tool_use":
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                messages.append({
                    "role": "user",
                    "content": [{
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": str(result),
                    }],
                })
```

### Step 7: Context Window Management

The context window is not infinite. As the loop iterates, the messages array
grows with every tool call and tool result. For short tasks this is fine, but
for long-running agents the context window fills up.

When the context window is nearly full, you need a strategy. The most common
approach is to summarize older messages into a condensed form:

```python
def trim_context(messages, max_tokens=100000):
    """Summarize early messages when context gets too large."""
    total = count_tokens(messages)
    if total < max_tokens:
        return messages

    # Keep the most recent messages and a summary of the rest
    summary_prompt = {
        "role": "user",
        "content": "Summarize the conversation so far in 2-3 sentences."
    }
    summary_response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        messages=messages[:-5] + [summary_prompt],
    )

    summary = summary_response.content[0].text
    trimmed = [
        {"role": "assistant", "content": summary},
    ] + messages[-5:]

    return trimmed
```

## Cost, Limits, and Safety

Every loop iteration costs money. Understanding the cost structure is
essential before you run any agent loop in production.

### Token Cost Per Iteration

Each loop iteration involves one API call. That call sends the entire
messages array as input tokens and receives a response of output tokens. The
cost is:

```
iteration_cost = (input_tokens * input_price) + (output_tokens * output_price)
```

For Claude Sonnet 4, input is $3.00 per million tokens and output is $15.00
per million tokens. A typical iteration with 5,000 input tokens and 500
output tokens costs:

```
(5000 * 0.000003) + (500 * 0.000015) = $0.015 + $0.0075 = $0.0225 per iteration
```

Ten iterations costs about $0.22. One hundred iterations costs $2.25. One
thousand iterations costs $22.50. The cost scales linearly with iterations.

### The Blow-Up Scenario

An unconstrained loop amplifies cost. Consider a simple agent that deletes
files. It deletes one file, lists the directory, finds the next file, deletes
it, lists again. Each cycle adds more tokens to the context because the tool
results (file lists) grow with each deletion. The input cost per iteration
increases as the context grows. An unconstrained loop can burn through
hundreds of dollars in minutes.

### Safety Mechanisms

You must build at least three safety mechanisms into every agent loop:

1. **Maximum iterations**: A hard cap on the number of loop iterations. Set
   this to 10 for standard tasks, 25 for complex multi-step workflows. Never
   leave it unbounded.

2. **Maximum tokens**: Set `max_tokens` for each API call. Even if a
   iteration limit catches runaway behavior, individual responses should be
   bounded.

3. **Token usage monitoring**: Track total input and output tokens across all
   iterations. If cumulative token usage exceeds a threshold, stop the loop
   and log a warning.

```python
MAX_ITERATIONS = 10
MAX_TOTAL_TOKENS = 50000
total_tokens = 0

for iteration in range(MAX_ITERATIONS):
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=4096,
        messages=messages,
        tools=tools,
    )

    total_tokens += response.usage.input_tokens + response.usage.output_tokens

    if total_tokens > MAX_TOTAL_TOKENS:
        print(f"Token budget exceeded: {total_tokens} tokens used.")
        break

    if response.stop_reason == "end_turn":
        print(response.content[0].text)
        break
```

### Why max_tokens Is Not a Termination Strategy

Setting `max_tokens` on the API call limits each individual response, but it
does not stop the loop. If `max_tokens` is reached, the loop can keep going
by calling the API again with the truncated response in context. Without an
iteration limit, the loop continues indefinitely, each time hitting
`max_tokens` and continuing. This is why you always need both `max_tokens`
per call and a maximum iteration count on the loop.

## When to Use This / When Not To

### Use the Agent Loop When

- Your task requires multiple steps that depend on intermediate results. The
  agent needs to look up information, use a calculation result in a second
  query, or take actions in sequence.

- Your task involves external tools that return unpredictable results. Search
  queries, database lookups, file operations -- the result of each tool call
  affects what the agent should do next.

- You need the agent to verify its own work. The agent can make a change,
  then check whether the change had the expected effect, and correct if
  needed.

- Your task could be completed in a single call but you want robustness. A
  looped agent can retry after a tool failure, try an alternative approach if
  the first one fails, or ask for clarification if the instructions are
  ambiguous.

### Do Not Use the Agent Loop When

- A single LLM call is sufficient. If your task is a simple question-answer
  or a single transformation (translate, summarize, classify), adding a loop
  adds latency and cost with no benefit.

- You are constrained by strict latency requirements. Each loop iteration
  adds an API round trip (typically 1-5 seconds). If your application needs
  sub-second responses, a loop is not the right pattern.

- The tools available to the agent have destructive side effects. A loop that
  deletes production data, sends emails, or charges credit cards should have
  human approval at every step, not automated loop execution.

- You cannot afford the token cost of multiple iterations. If your budget is
  tight and a single call meets the requirements, skip the loop.

### Human-in-the-Loop Checkpoints

For high-stakes actions, you can add human approval steps inside the loop.
When the agent wants to take a destructive or costly action, pause the loop
and ask a human to approve:

```python
for iteration in range(MAX_ITERATIONS):
    response = client.messages.create(...)

    if response.stop_reason == "tool_use":
        for block in response.content:
            if block.type == "tool_use":
                if block.name in ["delete_resource", "send_email", "charge_card"]:
                    approved = input(f"Approve {block.name} with args {block.input}? (y/n): ")
                    if approved.lower() != "y":
                        messages.append({
                            "role": "user",
                            "content": [{
                                "type": "tool_result",
                                "tool_use_id": block.id,
                                "content": "Tool call cancelled by human operator.",
                            }],
                        })
                        continue
                # execute normally
                result = execute_tool(block.name, block.input)
                messages.append(...)
```

This pattern gives you the power of automated agents with human oversight at
the critical decision points.

### Single Tool vs Multiple Tools in a Loop

A common question is whether to define one complex tool that does many things
or multiple simple tools that the agent chains in the loop.

**Single complex tool**: Use when the operation is well-defined and always
needs the same sub-steps. Example: a `process_refund` tool that looks up the
customer, calculates the refund amount, and issues the credit. This reduces
loop iterations but the tool is harder to reuse.

**Multiple simple tools**: Use when the agent needs flexibility. Example:
separate `lookup_customer`, `calculate_refund`, `issue_credit` tools. The
agent can use only the ones it needs, skip unnecessary steps, and combine
them with other tools in the loop. This increases loop iterations but makes
the system more adaptable.

The rule of thumb: start with simple tools and compose them in the loop.
Refactor into combined tools only when you see the same sequence appearing
repeatedly in real usage.

## Mental Model Check

Test your understanding of the agent loop with these questions.

### Question 1

You build an agent loop without a maximum iteration limit. Your agent has a
single tool: `search_web(query)`. What happens when you ask it to "research
the history of cryptography"?

**Answer**: The agent calls `search_web` for "history of cryptography", gets
results, picks a subtopic from the results, searches that subtopic, gets more
results, picks another subtopic, and continues indefinitely. Each search
opens more avenues for investigation. The loop never terminates naturally
because there is always more to explore. The cost grows unbounded until you
manually stop the process or your API budget runs out.

### Question 2

Your agent loop checks `stop_reason` and only handles `end_turn` and
`tool_use`. What happens when the response hits `max_tokens`?

**Answer**: The loop has no branch for `max_tokens`, so it falls through
without handling the response. The model's partial output is lost. Depending
on the code structure, the loop either silently drops the response and tries
again (wasting the partial output), raises an unhandled exception, or
continues with an incomplete messages array. The correct approach is to
handle `max_tokens` by either returning the partial response with a warning
or making one more call with a continuation prompt.

### Question 3

You set `max_tokens = 100` on your API call and no iteration limit. What
happens when the agent needs to call a tool that requires detailed arguments?

**Answer**: Each response is cut off at 100 tokens. The model cannot finish
generating a tool call or a complete response. The loop keeps re-invoking the
API because `stop_reason` is `max_tokens`, not `end_turn`. This creates a
high-cost loop where each iteration pays for 100 tokens of output but the
model never completes a thought. This is why you set both `max_tokens`
(per-response limit) and a maximum iteration count (loop safety net).

### Question 4

Your agent has three tools: `read_file`, `write_file`, and `list_directory`.
You ask it to "find all Python files with syntax errors and fix them." The
agent runs for 15 iterations before hitting the limit and returning a partial
result. What went wrong?

**Answer**: The task is too complex for the given iteration limit, OR the
agent is being inefficient -- for example, listing the directory one file at
a time instead of listing all files in one call. Possible fixes: increase the
iteration limit, improve the tool design so the agent can operate on batches
of files, or add a `find_files(pattern)` tool that returns multiple results
at once.

### Question 5

You are building a code review agent. The agent should analyze code, suggest
fixes, and apply the fixes. At which points should you add human-in-the-loop
checkpoints?

**Answer**: Human approval should be required before applying any fix to the
codebase. The analyze and suggest phases can run automatically. The critical
checkpoint is the `write_file` or `apply_patch` tool call. Before executing
it, pause the loop and show the human what change will be made and to which
file. This prevents the agent from making unwanted changes while still
allowing the analysis loop to run without interruptions.
