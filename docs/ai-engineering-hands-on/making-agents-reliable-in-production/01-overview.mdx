Your agent is working perfectly in development. It searches a database, calls an API, formats a response — every step works. You deploy to production. Within an hour, the agent is stuck in a loop: it calls the search API, gets no results, rephrases the query, searches again, gets no results, rephrases again, and repeats 47 times before your wallet notices the API bill. This is not a bug in the traditional sense — the agent is doing exactly what it was designed to do: try to find the answer. It just does not know when to stop.

Making agents reliable in production means adding the systems and safeguards that keep an agent running correctly over thousands of real-world interactions — error handling, retry logic, timeout enforcement, cost tracking, state management, and observability.

## The Concept

**A production-reliable agent is one that fails gracefully, recovers from errors, stays within budget, and is observable at every step of its execution.** Think of it like a self-driving car. The car does not just need to drive well — it needs to handle a tire blowout, navigate a construction zone, recognize when its sensors are failing, and pull over safely when something goes wrong. The driving skill (the agent's reasoning) is only part of the system. The reliability infrastructure (error handling, monitoring, fallbacks) is equally important.

## Reliability Tools and Patterns Reference

| Technique | What It Prevents | How to Implement |
|---|---|---|
| `max_tokens` and `max_steps` | Runaway agents (infinite loops, unbounded generation) | `Agent(max_iterations=10)` or `token_limit=4000` |
| Exponential backoff with jitter | API rate limit errors and temporary failures | `time.sleep(min(60, base * 2**retry + random.uniform(0, 1)))` |
| Circuit breaker pattern | Repeated calls to a failing service | Break after N consecutive failures, retry after cooldown period |
| `try/except` with fallback | Unexpected API errors, network failures | `try: result = call_llm() except: result = fallback_response()` |
| State checkpointing | Agent crashes mid-execution | Save execution state to DB after each step |
| Token budget tracking | Cost overruns and surprise bills | `if total_tokens > budget: agent.stop("Budget exceeded")` |
| Structured logging | Debugging agent decisions mid-execution | Log each step as JSON: `{step, input, output, tokens, latency}` |
| Human-in-the-loop confirmation | High-stakes actions (deleting, purchasing, modifying) | `agent.pause_for_approval(action_id)` |
| Timeout enforcement | Hanging API calls, infinite loops | `asyncio.wait_for(call(), timeout=30)` |
| Tool output validation | Malformed tool responses causing agent bugs | `pydantic.validate(tool_output, expected_schema)` |

## How It Actually Works — Step by Step

### Step 1: Bound the Agent Before It Starts

Before the agent takes its first step, you set hard limits: maximum iterations (10 is a common default), maximum tokens (4,000-8,000 depending on the use case), and maximum cost ($0.05-$0.50 depending on the model). These are not guidelines — they are hard stops. If the agent hits any limit, execution terminates with a "budget exceeded" result. The user gets a partial result and a message that the agent could not complete the request within its budget.

### Step 2: Wrap Every Tool Call in Error Handling

The most common production failure is an API call that fails. The LLM provider might return a 429 (rate limited), the search API might return a 503 (unavailable), the database might time out. Every tool call must be wrapped in `try/except` with a meaningful fallback. The agent should know that the tool call failed and why, so it can decide whether to retry, use a different tool, or report the failure to the user.

### Step 3: Implement Retry with Exponential Backoff

When a tool call fails due to a transient error (rate limit, network timeout, service unavailable), the agent should retry — but not immediately. Exponential backoff means the first retry waits 1 second, the second waits 2 seconds, the third waits 4 seconds, up to a maximum. Adding jitter (randomizing the wait time slightly) prevents the "thundering herd" problem where many retries hit the API simultaneously. After 3 retries, the agent should give up and use a fallback.

### Step 4: Validate Every Tool Output

An LLM can call a tool with the wrong parameters, or a tool can return malformed data. Tool output validation checks that the result matches the expected schema before the agent uses it for reasoning. If the output is invalid, the agent should either retry the call with corrected parameters or report the error. This prevents the agent from "hallucinating" based on bad tool data.

### Step 5: Checkpoint State After Every Step

If the agent crashes mid-execution (server restart, deployment, OOM kill), the entire conversation is lost unless state is persisted. After each tool call and each reasoning step, save the agent's state to a database: the current step number, the conversation history, the tool results, and the accumulated cost. On restart, the agent can resume from the last checkpoint.

### Step 6: Log Everything in Structured Format

Every step of the agent's execution should produce a structured log entry: step number, the agent's reasoning, the tool called, the tool's result, the token usage, the latency, and any errors. This is not optional — without structured logging, debugging a production agent failure is nearly impossible. Each log entry should be a JSON object that can be ingested by your logging infrastructure (ELK, Datadog, Grafana Loki).

## Cost, Limits, and Safety

**The cost of production agent infrastructure:** The guardrails and error handling add minimal compute cost (regex checks, state persistence) but can add latency. Each retry delays the response by 1-7 seconds. Each structured log write adds milliseconds. Budget for a 10-20% overhead on top of the raw LLM cost for reliability infrastructure.

**The most expensive failure mode is the runaway agent:** An agent stuck in a loop calling an expensive API can rack up hundreds of dollars in minutes. Always set a hard cost limit per session. For a typical agent using gpt-4o-mini, a session should cost $0.01-$0.05. If the cost exceeds $0.10, something is likely wrong.

**State persistence storage:** If you checkpoint state after every step, you need storage. A typical agent session with 10 steps produces about 5-10 KB of state data. For 100,000 sessions, that is 500 MB to 1 GB. Use a cheap key-value store (Redis, SQLite, DynamoDB) for checkpoints.

**Observability infrastructure:** Structured logging requires a logging backend. For small deployments, writing to a local file with log rotation is sufficient. For larger deployments, use a centralized logging service (Datadog, Grafana Cloud, ELK). The cost of the logging infrastructure is typically less than 5% of the total infrastructure cost.

## When to Use This / When Not To

**Use production reliability patterns for every agent:** The minimum set — bounding, error handling, and structured logging — should be present in every agent, even internal prototypes. A prototype that costs $10 unexpectedly because of a runaway loop is not a prototype that teaches you anything useful.

**Do not over-engineer for low-stakes agents:** A simple text-rewriting agent that runs once per user request does not need state checkpointing or circuit breakers. Add complexity proportional to the cost of failure. If the agent failing means a user gets a bad response, keep it simple. If the agent failing means a database is corrupted or money is lost, invest in full production hardening.

**The human-in-the-loop pattern is non-negotiable for destructive actions:** Any agent that can delete data, make purchases, modify permissions, or send communications should pause for human confirmation before executing the action. This is not a reliability pattern — it is a safety requirement.

## Mental Model Check

**Q1:** What is the difference between a retry and a fallback?

**A1:** A retry repeats the same operation, hoping it succeeds this time (e.g., the API was temporarily unavailable). A fallback uses a different approach when the primary approach fails (e.g., use a cached response, use a different model, return a default answer).

**Q2:** Why is exponential backoff important for retries, rather than retrying immediately?

**A2:** Immediate retries are likely to hit the same error (the API is still rate limited, the service is still down). They also make the problem worse for the service under load. Exponential backoff gives the service time to recover and reduces the load on it.

**Q3:** What information should be in every structured log entry for an agent step?

**A3:** At minimum: session ID, step number, action taken (tool call or reasoning), tool name, input to the tool, output summary, token usage, latency, and any error messages. This allows you to reconstruct the full execution from the logs.

**Q4:** If an agent has a maximum of 10 steps and each step costs $0.005, what is the maximum cost per session?

**A4:** $0.05 for the agent steps, plus the initial prompt cost. The hard cost limit should be set at $0.10 to allow for the initial prompt and any retries. This is also the cost at which you should investigate whether the agent is behaving correctly.

**Q5:** What is the circuit breaker pattern and when would you use it in an agent system?

**A5:** The circuit breaker stops calling a failing service after N consecutive failures, then tries again after a cooldown period. Use it when an agent repeatedly calls a tool that is failing (e.g., a search API that returns 503 errors). Without it, the agent wastes retries and budget on a service that is known to be down.
