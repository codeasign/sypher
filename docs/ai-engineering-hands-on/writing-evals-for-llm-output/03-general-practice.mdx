## Tier 1 — Beginner

### Exercise 1.1: Add a toxicity eval

**Task:** Add a toxicity/toxicity safety evaluation to the test suite. Use deepeval's `ToxicityMetric` or build a simple keyword-based filter that flags responses containing known harmful terms. Run it on all test summaries and report which ones trigger it.

**Hint:** deepeval has a `ToxicityMetric` that uses an LLM judge. Pass it as an additional metric alongside faithfulness and relevance.

### Solution

```python
from deepeval.metrics import ToxicityMetric
from deepeval.test_case import LLMTestCase

def check_toxicity(summary: str) -> dict:
    metric = ToxicityMetric(
        threshold=0.5,
        model="gpt-4o-mini",
        include_reason=True,
    )
    test_case = LLMTestCase(
        input="Summarize",
        actual_output=summary,
    )
    metric.measure(test_case)
    return {
        "score": metric.score,
        "toxic": not metric.is_successful(),
        "reason": metric.reason,
    }


from summaries import GOOD_SUMMARIES, BAD_SUMMARIES

all_summaries = GOOD_SUMMARIES + BAD_SUMMARIES
labels = ["good_0", "good_1", "good_2", "bad_0", "bad_1", "bad_2"]

for label, summary in zip(labels, all_summaries):
    result = check_toxicity(summary)
    status = "⚠ TOXIC" if result["toxic"] else "✓ Safe"
    print(f"{label}: {status} (score: {result['score']:.3f})")
```

The test summaries in this exercise are all benign — but in real data, toxicity checks are essential.

### Exercise 1.2: Eval score over multiple runs

**Task:** Run the same faithfulness eval 5 times on the same test case. Record the scores. What is the mean and standard deviation? How consistent is the eval?

**Hint:** Call `metric.measure()` in a loop. deepeval metrics are designed for single use — create a new metric instance for each run.

### Solution

```python
from deepeval.metrics import FaithfulnessMetric
from deepeval.test_case import LLMTestCase
from summaries import SOURCE_DOCUMENTS, GOOD_SUMMARIES

key = "earnings"
source = SOURCE_DOCUMENTS[key]
summary = GOOD_SUMMARIES[0]

scores = []
for run in range(5):
    metric = FaithfulnessMetric(
        threshold=0.7, model="gpt-4o-mini"
    )
    test_case = LLMTestCase(
        input="Summarize",
        actual_output=summary,
        retrieval_context=[source],
    )
    metric.measure(test_case)
    scores.append(metric.score)
    print(f"  Run {run+1}: {metric.score:.3f} "
          f"[{'PASS' if metric.is_successful() else 'FAIL'}]")

from statistics import mean, stdev
print(f"\nMean: {mean(scores):.3f}")
print(f"Std dev: {stdev(scores):.3f}")
print(f"Range: {min(scores):.3f} - {max(scores):.3f}")
```

If the standard deviation is above 0.1, the eval may be too noisy for your use case. Consider switching to a more capable judge model or increasing the number of samples.

### Exercise 1.3: Threshold finder

**Task:** Write a script that tests multiple threshold values (0.3, 0.5, 0.7, 0.9) and reports how many good summaries pass and how many bad summaries fail at each threshold. Find the threshold that maximizes both.

**Hint:** For each threshold, compute: true positives (bad summaries correctly failing) and false positives (good summaries incorrectly failing). The best threshold maximizes both.

### Solution

```python
from deepeval.metrics import FaithfulnessMetric
from deepeval.test_case import LLMTestCase
from summaries import SOURCE_DOCUMENTS, GOOD_SUMMARIES, BAD_SUMMARIES, SOURCE_KEYS

thresholds = [0.3, 0.5, 0.7, 0.9, 0.95]

for threshold in thresholds:
    good_passes = 0
    bad_fails = 0

    for i, key in enumerate(SOURCE_KEYS):
        source = SOURCE_DOCUMENTS[key]

        # Good summary: should pass
        metric = FaithfulnessMetric(
            threshold=threshold, model="gpt-4o-mini"
        )
        case = LLMTestCase(
            input="Summarize",
            actual_output=GOOD_SUMMARIES[i],
            retrieval_context=[source],
        )
        metric.measure(case)
        if metric.is_successful():
            good_passes += 1

        # Bad summary: should fail
        metric2 = FaithfulnessMetric(
            threshold=threshold, model="gpt-4o-mini"
        )
        case2 = LLMTestCase(
            input="Summarize",
            actual_output=BAD_SUMMARIES[i],
            retrieval_context=[source],
        )
        metric2.measure(case2)
        if not metric2.is_successful():
            bad_fails += 1

    print(
        f"Threshold {threshold:.1f}: "
        f"good pass={good_passes}/3, "
        f"bad fail={bad_fails}/3"
    )
```

## Tier 2 — Intermediate

### Exercise 2.1: Cross-model eval comparison

**Task:** Run the same faithfulness eval using three different judge models: `gpt-4o-mini`, `gpt-4o`, and `claude-sonnet-4-20250514`. Compare the scores and the pass/fail results. Are there cases where one judge passes and another fails?

**Hint:** deepeval accepts a `model` parameter. For Anthropic models, set `ANTHROPIC_API_KEY` and use `model="claude-sonnet-4-20250514"`.

### Solution

```python
from deepeval.metrics import FaithfulnessMetric
from deepeval.test_case import LLMTestCase
from summaries import SOURCE_DOCUMENTS, BAD_SUMMARIES, SOURCE_KEYS

models = ["gpt-4o-mini", "gpt-4o", "claude-sonnet-4-20250514"]

print(f"{'Model':<30} {'Key':<15} {'Score':<8} {'Result':<8}")
print("-" * 65)

for model in models:
    for i, key in enumerate(SOURCE_KEYS):
        try:
            metric = FaithfulnessMetric(
                threshold=0.7, model=model
            )
            test_case = LLMTestCase(
                input="Summarize",
                actual_output=BAD_SUMMARIES[i],
                retrieval_context=[SOURCE_DOCUMENTS[key]],
            )
            metric.measure(test_case)
            result = "PASS" if metric.is_successful() else "FAIL"
            print(
                f"{model:<30} {key:<15} "
                f"{metric.score:<8.3f} {result:<8}"
            )
        except Exception as e:
            print(f"{model:<30} {key:<15} {'ERROR':<8} {str(e)[:30]}")
    print()
```

Note which models are stricter. In practice, using a cheaper model for bulk evaluation and escalating edge cases to a more expensive judge is a cost-effective strategy.

### Exercise 2.2: Eval-driven prompt improvement

**Task:** You have three candidate prompts for a summarization system. Run all three through the full eval suite (faithfulness + relevance + numeric) on the same source document. Which prompt produces the best scores? Which dimension is weakest across all prompts?

**Hint:** Create three different system prompts and use a real LLM to generate summaries for each. Then run the eval suite on all outputs.

### Solution

```python
import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

CANDIDATE_PROMPTS = [
    "Summarize the following text concisely. Focus on key facts.",
    "You are a professional editor. Summarize the text below, "
    "preserving all important numbers and data points.",
    "Read the following text and produce a one-paragraph summary "
    "that a busy executive could read in 30 seconds.",
]

from summaries import SOURCE_DOCUMENTS
from deepeval.metrics import FaithfulnessMetric, AnswerRelevancyMetric
from deepeval.test_case import LLMTestCase

source_text = SOURCE_DOCUMENTS["earnings"]

for i, system_prompt in enumerate(CANDIDATE_PROMPTS):
    print(f"\n--- Prompt {i+1} ---")
    print(f"System: {system_prompt[:50]}...")

    # Generate summary
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": source_text},
        ],
        temperature=0.3,
    )
    summary = response.choices[0].message.content
    print(f"Output: {summary[:80]}...")

    # Evaluate
    faithful = FaithfulnessMetric(
        threshold=0.7, model="gpt-4o-mini"
    )
    relevant = AnswerRelevancyMetric(
        threshold=0.7, model="gpt-4o-mini"
    )

    test_case = LLMTestCase(
        input="Summarize the earnings report",
        actual_output=summary,
        retrieval_context=[source_text],
    )

    faithful.measure(test_case)
    relevant.measure(test_case)

    print(f"  Faithfulness: {faithful.score:.3f} "
          f"[{'PASS' if faithful.is_successful() else 'FAIL'}]")
    print(f"  Relevance:    {relevant.score:.3f} "
          f"[{'PASS' if relevant.is_successful() else 'FAIL'}]")
```

### Exercise 2.3: Evaluation data quality check

**Task:** You suspect your golden dataset has issues — some prompts are ambiguous, some expected answers are wrong. Write a script that uses an LLM to audit the golden dataset itself. Flag any prompt-rubric pairs that are unclear, contradictory, or unanswerable.

**Hint:** Send each dataset entry to an LLM with a rubric-checking prompt. Ask the LLM to rate the clarity of the rubric on a scale of 1-5.

### Solution

```python
import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

DATASET_AUDIT_PROMPT = """You are auditing a golden dataset for LLM evaluation.
Review the following prompt and rubric, then rate:

1. Clarity (1-5): Is the rubric unambiguous?
2. Answerability (1-5): Can a knowledgeable person write a response
   that would meet this rubric?
3. Issues: List any problems (contradictions, vagueness, etc.)

Prompt: {prompt}
Rubric: {rubric}

Return JSON with "clarity", "answerability", and "issues" keys."""

golden_entries = [
    {
        "prompt": "What is 2+2?",
        "rubric": "Must give the correct sum of 2 and 2. Score 0-1.",
    },
    {
        "prompt": "Tell me about AI safety.",
        "rubric": "Response should be comprehensive and well-structured.",
        # Potential issue: "comprehensive" is vague — how comprehensive?
    },
]

for entry in golden_entries:
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "user", "content": DATASET_AUDIT_PROMPT.format(
                prompt=entry["prompt"],
                rubric=entry["rubric"],
            )},
        ],
        response_format={"type": "json_object"},
    )
    audit = response.choices[0].message.content
    print(f"Prompt: {entry['prompt'][:40]}")
    print(f"Audit: {audit}\n")
```

## Tier 3 — Advanced

### Exercise 3.1: Multi-turn conversation eval

**Task:** Design an eval framework for multi-turn conversations. Instead of evaluating a single response, evaluate a conversation of 3-5 turns. Metrics should include: consistency across turns (does the agent contradict itself?), context retention (does it remember details from earlier turns?), and progress (does it move toward resolution?).

**Solution approach:**

The key challenge is that single-response evals do not work for conversations — a response that is perfectly good in isolation might contradict something the model said three turns ago.

```python
from deepeval.test_case import LLMTestCase
from deepeval.metrics import FaithfulnessMetric
import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])


def evaluate_conversation(turns: list[dict]) -> dict:
    """
    Evaluate a multi-turn conversation.

    Each turn: {"role": "user"|"assistant", "content": str}
    """
    # Reconstruct the conversation
    messages = [
        {"role": t["role"], "content": t["content"]}
        for t in turns
    ]

    # Metric 1: Consistency — check that the last response
    # does not contradict the first response
    first_response = None
    for t in turns:
        if t["role"] == "assistant" and first_response is None:
            first_response = t["content"]
            break

    last_response = None
    for t in reversed(turns):
        if t["role"] == "assistant" and last_response is None:
            last_response = t["content"]
            break

    consistency_score = 0.5  # default
    if first_response and last_response:
        consistency_prompt = f"""Compare these two responses from the same AI:

First response: {first_response[:200]}

Latest response: {last_response[:200]}

Do they contradict each other? Rate consistency:
1.0 = perfectly consistent
0.5 = somewhat consistent but different emphasis
0.0 = direct contradiction

Return JSON with "score" and "reason"."""
        result = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": consistency_prompt}],
            response_format={"type": "json_object"},
        )
        import json
        data = json.loads(
            result.choices[0].message.content
        )
        consistency_score = data["score"]

    # Metric 2: Context retention
    # Check if the response references information from early turns
    context_prompt = f"""Here is the full conversation:
{chr(10).join(f'{t["role"]}: {t["content"][:100]}' for t in turns)}

Does the final assistant response demonstrate that it remembers
and correctly uses information from earlier in the conversation?
Rate 0.0-1.0. Return JSON with "score" and "reason"."""
    result2 = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": context_prompt}],
        response_format={"type": "json_object"},
    )
    import json
    context_data = json.loads(
        result2.choices[0].message.content
    )

    return {
        "consistency": consistency_score,
        "context_retention": context_data["score"],
    }


# Test with a conversation about travel recommendations
conversation = [
    {"role": "user", "content": "I am planning a trip to Japan."},
    {"role": "assistant", "content": "Great choice! Japan has "
     "amazing food, culture, and scenery. "
     "What time of year are you planning to go?"},
    {"role": "user", "content": "Cherry blossom season in April."},
    {"role": "assistant", "content": "April is perfect for cherry "
     "blossoms! Popular spots include Kyoto, Tokyo, and Osaka. "
     "Book accommodation 3-4 months in advance."},
]

scores = evaluate_conversation(conversation)
for metric, score in scores.items():
    print(f"  {metric}: {score:.3f}")
```

### Exercise 3.2: Automated eval regression triage

**Task:** Build a system that, when an eval fails, automatically runs additional diagnostics to identify the root cause. Was it a faithfulness failure (model hallucinated)? A relevance failure (model misunderstood the prompt)? Or a data quality issue (the eval rubric is wrong)? The system should output a recommended action for each failure.

**Solution approach:**

```python
import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])


def triage_eval_failure(
    prompt: str,
    response: str,
    source: str,
    failed_dimension: str,
    score: float,
) -> dict:
    """
    Diagnose why an eval failed and recommend an action.

    Returns diagnosis and recommended action.
    """
    triage_prompt = f"""An LLM evaluation failed. Diagnose the root cause.

Prompt: {prompt}
Response: {response[:300]}
Source text: {source[:300]}
Failed dimension: {failed_dimension}
Score: {score}

Possible root causes:
1. MODEL FAILURE: The model produced genuinely bad output
2. RUBRIC ISSUE: The eval rubric is too strict or misaligned
3. JUDGE ERROR: The judge model scored incorrectly
4. DATA ISSUE: The source or expected answer has problems

Return JSON with:
- "root_cause": one of the four options above
- "evidence": specific evidence from the text
- "recommended_action": what to do next

JSON:"""

    result = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": triage_prompt}],
        response_format={"type": "json_object"},
    )
    import json
    return json.loads(
        result.choices[0].message.content
    )


# Example: a borderline failure
diagnosis = triage_eval_failure(
    prompt="Summarize the earnings report.",
    response="Acme Corp revenue grew 12% to $2.4 billion.",
    source="Acme Corp reported Q3 revenue of $2.4 billion, up 12%.",
    failed_dimension="faithfulness",
    score=0.65,
)
for key, value in diagnosis.items():
    print(f"  {key}: {value}")
```

### Exercise 3.3: Evals as CI gates with automatic rollback

**Task:** Design a system where eval scores are tracked in a database, and a new model version is automatically rejected if its eval scores are statistically lower than the current production version. Include a rollback mechanism that switches back to the previous version if the new version underperforms in production.

**Solution approach:**

```python
import json
import sqlite3
from datetime import datetime
from statistics import mean, stdev
from scipy import stats


class EvalTrackingDB:
    """Tracks eval scores over time to detect regressions."""

    def __init__(self, db_path: str = "eval_history.db"):
        self.conn = sqlite3.connect(db_path)
        self.conn.execute("""
            CREATE TABLE IF NOT EXISTS eval_runs (
                id INTEGER PRIMARY KEY AUTOINCREMENT,
                model_version TEXT,
                dimension TEXT,
                mean_score REAL,
                std_score REAL,
                sample_count INTEGER,
                timestamp TEXT
            )
        """)
        self.conn.commit()

    def record_run(
        self,
        model_version: str,
        dimension: str,
        scores: list[float],
    ):
        """Record a completed eval run."""
        self.conn.execute(
            """INSERT INTO eval_runs
               (model_version, dimension, mean_score, std_score,
                sample_count, timestamp)
               VALUES (?, ?, ?, ?, ?, ?)""",
            (
                model_version,
                dimension,
                mean(scores),
                stdev(scores) if len(scores) > 1 else 0.0,
                len(scores),
                datetime.now().isoformat(),
            ),
        )
        self.conn.commit()

    def get_baseline(self, dimension: str, n_runs: int = 5):
        """Get the last n runs for a dimension as a baseline."""
        cursor = self.conn.execute(
            """SELECT mean_score FROM eval_runs
               WHERE dimension = ?
               ORDER BY timestamp DESC LIMIT ?""",
            (dimension, n_runs),
        )
        rows = cursor.fetchall()
        return [row[0] for row in rows] if rows else None


def evaluate_new_version(
    model_version: str,
    eval_scores: dict[str, list[float]],
    db: EvalTrackingDB,
    alpha: float = 0.05,
) -> tuple[bool, list[str]]:
    """
    Evaluate a new model version against historical baselines.

    Returns (approved, reasons_for_rejection).
    """
    rejections = []

    for dimension, scores in eval_scores.items():
        baseline = db.get_baseline(dimension)
        if baseline is None:
            # No historical data — approve but log baseline
            db.record_run(model_version, dimension, scores)
            continue

        current_mean = mean(scores)
        baseline_mean = mean(baseline)

        if len(scores) >= 5 and len(baseline) >= 3:
            t_stat, p_value = stats.ttest_ind(
                baseline, scores, equal_var=False
            )
            # One-tailed: check for significant DROP
            if t_stat > 0 and p_value < alpha:
                rejections.append(
                    f"{dimension}: mean dropped from "
                    f"{baseline_mean:.3f} to {current_mean:.3f} "
                    f"(p={p_value:.4f})"
                )

        db.record_run(model_version, dimension, scores)

    approved = len(rejections) == 0
    return approved, rejections


# Example usage
db = EvalTrackingDB()

# Simulate recording a baseline
db.record_run("gpt-4o-mini-v1", "faithfulness",
              [0.85, 0.88, 0.82, 0.90, 0.87])
db.record_run("gpt-4o-mini-v1", "relevance",
              [0.92, 0.90, 0.88, 0.94, 0.91])

# Evaluate a new version
new_scores = {
    "faithfulness": [0.72, 0.68, 0.75, 0.70, 0.73],
    "relevance": [0.91, 0.89, 0.90, 0.92, 0.88],
}

approved, reasons = evaluate_new_version(
    "gpt-4o-mini-v2", new_scores, db
)

print(f"Version approved: {approved}")
if reasons:
    print("Reasons:")
    for reason in reasons:
        print(f"  - {reason}")
```

**Discussion:** In practice, you would wrap this in a CI/CD pipeline. When a PR proposes a new model version or prompt change, the pipeline runs the eval suite and compares scores against the historical baseline stored in the database. If the scores drop significantly, the pipeline fails and the PR is blocked until the team reviews.
