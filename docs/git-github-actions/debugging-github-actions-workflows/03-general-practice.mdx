Complete these exercises on your own to prove you can diagnose, isolate, and fix GitHub Actions workflow failures using the full debugging toolkit.

## Tier 1 — Beginner

### Exercise 1: Enable Step Debug Logging

**Task:** You have a workflow that runs tests and the tests pass locally but fail on the runner. The error message says only "Process completed with exit code 1." Enable step-level debug logging so you can see every command and variable expansion the runner executes. Do this without modifying the workflow file.

**Hint:** Step debug logging is controlled through a repository secret, not a workflow configuration key. The secret name follows the pattern `ACTIONS_*_DEBUG`.

**Solution:**

Go to your repository Settings > Secrets and variables > Actions. Click "New repository secret." Set the name to `ACTIONS_STEP_DEBUG` and the value to `true`. Click "Add secret."

Re-run the failed workflow by going to the Actions tab, clicking on the failed run, and selecting "Re-run jobs" > "Re-run failed jobs." When the workflow runs again, open any step log. You will see additional lines prefixed with `##[debug]` that show the exact command strings, environment variable values, and working directory changes that the runner performed. This level of detail is almost always enough to spot a missing dependency, a wrong PATH, or an unexpected variable value.

### Exercise 2: Download Workflow Run Logs

**Task:** You need to share the full log output of a failed workflow run with a teammate who cannot access your repository. Download the raw logs from the Actions tab so you can send them as a file.

**Hint:** GitHub provides a "Download log archive" button on the workflow run page. The `gh` CLI also supports bulk log download.

**Solution:**

**Method A — Actions UI:** Go to the Actions tab in your repository. Click on the failed workflow run. Click the gear icon or the "..." menu in the upper-right corner of the log viewer. Select "Download log archive." GitHub downloads a zip file containing one file per job. Each file contains the complete raw log output including debug lines, timestamps, and exit codes.

**Method B — `gh` CLI:**

```bash
gh run view <run-id> --log > workflow.log
```

Find the `<run-id>` from the Actions tab URL (it is a number) or by running `gh run list`. The `--log` flag streams all job logs to stdout. Redirect to a file and share it. To filter to a specific job:

```bash
gh run view <run-id> --log --job build
```

This is useful when a workflow has multiple jobs and you only need the failing one.

### Exercise 3: Use `continue-on-error` to Allow a Flaky Step to Fail

**Task:** One of your workflow steps runs an integration test against an external API that occasionally times out. The timeout is a known issue and should not block the rest of the pipeline. Allow this step to fail without failing the entire job.

**Hint:** There is a single YAML key you add to the step that tells the runner to treat a failure as a warning instead of a hard stop.

**Solution:**

Add `continue-on-error: true` to the flaky step:

```yaml
- name: Integration tests
  continue-on-error: true
  run: npm run test:integration
```

When this step fails, the runner prints a warning and continues to the next step. The job appears as "Success with warnings" in the Actions UI. The step icon shows a yellow triangle instead of a red X. You can still see the failure in the step log, but it does not block the pipeline.

Remove `continue-on-error: true` after you fix the underlying issue. Using it permanently masks real failures.

### Exercise 4: Inspect the `github` Context by Printing It

**Task:** Your workflow behaves differently on `push` events versus `pull_request` events, and you need to understand what information is available in each case. Print the full `github` context to the log.

**Hint:** The `toJSON()` function converts any context object to a printable JSON string. Use it in an `echo` command inside a step.

**Solution:**

Add this step to your workflow:

```yaml
- name: Inspect github context
  run: echo '${{ toJSON(github) }}'
```

Push a commit and view the step output. You will see a large JSON object with fields like:

- `event_name` — the trigger event (`push`, `pull_request`, `workflow_dispatch`)
- `ref` — the branch or tag ref (`refs/heads/main`)
- `sha` — the commit SHA
- `actor` — the username who triggered the workflow
- `repository` — the full repository name
- `payload` — the full webhook event payload (different for each event type)

For a `pull_request` event, the payload includes `pull_request.head.ref`, `pull_request.base.ref`, and `pull_request.number`. For a `push` event, it includes `commits` array with details about each commit. Understanding these differences helps you write conditional steps that behave correctly for each trigger type.

## Tier 2 — Intermediate

### Exercise 5: Install and Run `nektos/act` to Simulate a Workflow Locally

**Task:** You are developing a new workflow with six steps and you expect three or four iterations to get it right. Pushing each iteration costs two minutes per round trip. Install `nektos/act` and run the workflow locally to reduce iteration time to seconds.

**Hint:** `act` requires Docker. Install Docker first, then install `act` using your operating system's package manager. Run `act -l` to list available jobs.

**Solution:**

Install Docker if you have not already. Then install `act`:

```bash
# macOS
brew install act

# Linux
curl -s https://raw.githubusercontent.com/nektos/act/master/install.sh | sudo bash

# Windows (any)
winget install nektos.act
# or
choco install act-cli
# or
scoop install act
```

From your repository root, list the available workflows and jobs:

```bash
act -l
```

Run a specific job from a specific workflow:

```bash
act -j build -W .github/workflows/ci.yml
```

`act` downloads the necessary Docker images on first run (the `ubuntu-latest` runner image is several hundred megabytes). Subsequent runs use the cached image. The workflow runs inside the container, and you see the same output you would see on GitHub.

For workflows that use secrets, create a `.secrets` file in the repository root:

```
MY_SECRET=local-value
ACTIONS_STEP_DEBUG=true
```

Then run `act` with `--secret-file .secrets` to pass these values to the simulated runner.

### Exercise 6: Use `if: failure()` to Send a Notification on Failure

**Task:** You have a deployment workflow. When the deploy step fails, you want to send a notification to a Slack webhook. The notification step should only run when the deploy fails, not when it succeeds.

**Hint:** The `if:` condition `failure()` evaluates to `true` when any previous step in the job has failed. Combine it with a step that posts to a webhook.

**Solution:**

Add a notification step with `if: failure()` at the end of your job:

```yaml
- name: Notify Slack on failure
  if: failure()
  run: |
    curl -X POST -H 'Content-type: application/json' \
      --data '{"text":"Deploy failed for ${{ github.repository }} on ${{ github.ref }}"}' \
      ${{ secrets.SLACK_WEBHOOK_URL }}
```

The `failure()` function returns `true` when any step in the job has failed. It returns `false` when all steps succeeded or when a step was skipped. This step runs only when the deploy fails.

You can also use `cancelled()` and `success()` alongside `failure()` for more granular conditions:

```yaml
- name: Always notify
  if: success() || failure()
  run: echo "This runs on success or failure, but not cancellation"
```

Note that `if: failure()` does not need `always()` — it evaluates to `true` only when there is a failure, not when the job is skipped or cancelled.

### Exercise 7: Add a `workflow_dispatch` Input for Debug Mode

**Task:** You want to be able to trigger your workflow manually with a debug mode toggle. When debug mode is on, the workflow prints extra diagnostic information and enables `ACTIONS_STEP_DEBUG`. When debug mode is off, the workflow runs normally with minimal logging.

**Hint:** Add `workflow_dispatch` to the `on:` trigger with an input field string. Check the input value with `${{ github.event.inputs.debug }}` and use it to conditionally set environment variables.

**Solution:**

Add `workflow_dispatch` to your workflow triggers with a `debug` input:

```yaml
name: Deploy with Debug

on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      debug:
        description: 'Enable debug logging'
        required: false
        default: 'false'
        type: boolean

env:
  ACTIONS_STEP_DEBUG: ${{ github.event.inputs.debug == 'true' && 'true' || 'false' }}
  DEBUG_MODE: ${{ github.event.inputs.debug == 'true' && '1' || '0' }}

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Print diagnostics in debug mode
        if: env.DEBUG_MODE == '1'
        run: |
          echo "Node version: $(node --version)"
          echo "npm version: $(npm --version)"
          echo "Disk space:"
          df -h

      - name: Run deploy
        run: ./deploy.sh
```

When you trigger the workflow manually from the Actions tab, a dialog appears with the `debug` checkbox. Check it to enable debug logging and diagnostics. When the workflow runs on `push`, `debug` is not set, so `ACTIONS_STEP_DEBUG` evaluates to `'false'` and the diagnostics step is skipped.

### Exercise 8: Use `runner.debug` to Conditionally Print Extra Diagnostics

**Task:** You have enabled `ACTIONS_STEP_DEBUG` as a repository secret, but you want to add additional diagnostic steps that only run when debug mode is active. Use the `runner.debug` context to conditionally execute these steps.

**Hint:** `runner.debug` returns `true` when either `ACTIONS_STEP_DEBUG` or `ACTIONS_RUNNER_DEBUG` is enabled. Use it in an `if:` condition.

**Solution:**

Add conditional steps that check `runner.debug`:

```yaml
- name: List workspace contents
  if: runner.debug == 'true'
  run: |
    echo "Workspace: $GITHUB_WORKSPACE"
    ls -la $GITHUB_WORKSPACE
    echo "Runner temp directory:"
    ls -la $RUNNER_TEMP

- name: Print environment variables
  if: runner.debug == 'true'
  run: env | sort

- name: Check tool availability
  if: runner.debug == 'true'
  run: |
    echo "Checking tools:"
    for tool in node python3 docker git ffmpeg; do
      if command -v $tool &> /dev/null; then
        echo "$tool: available ($($tool --version 2>&1 | head -1))"
      else
        echo "$tool: NOT FOUND"
      fi
    done
```

These steps only run when debug logging is active — either through the repository secret or through the `env` block. When debug mode is off, the steps are skipped and do not add noise to the log output. This approach keeps your workflow clean while providing deep diagnostics on demand.

## Tier 3 — Advanced

### Exercise 9: Debug a Runner-Specific Failure with tmate

**Task:** Your workflow runs perfectly on `ubuntu-latest` but consistently fails on `windows-latest`. The error message is vague — something about a file path not being found. You have tried enabling debug logging and running with `act` (which does not support Windows containers well), but the failure only occurs on the actual GitHub Windows runner. Set up a tmate debugging session to investigate the Windows runner environment interactively.

**Hint:** Add `mxschmitt/action-tmate@v3` as a step early in the job. Use `if: runner.os == 'Windows'` so tmate only activates for the failing OS. Use the tmate connection URL from the log to SSH in.

**Solution:**

Modify your workflow to include a conditional tmate step:

```yaml
jobs:
  test:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v4

      - name: Setup tmate session (Windows only)
        if: runner.os == 'Windows'
        uses: mxschmitt/action-tmate@v3

      - name: Run tests
        run: npm test
```

Push the workflow. When the `windows-latest` job reaches the tmate step, it pauses and prints a connection URL in the logs:

```
Connection established: ssh <hash>@nyc1.tmate.io
```

Open a terminal on your machine and connect:

```bash
ssh <hash>@nyc1.tmate.io
```

Inside the Windows runner, investigate:

```powershell
# Check current directory
Get-Location

# List workspace files
Get-ChildItem $env:GITHUB_WORKSPACE -Recurse

# Check PATH
$env:PATH -split ';'

# Test the failing command manually
npm test

# Check if expected files exist
Test-Path "some/file/path.js"
```

The file path issue will likely become obvious when you explore the runner's directory structure. Windows uses backslashes (`\`) while Linux uses forward slashes (`/`). A hardcoded path like `./config/settings.json` may work on Linux but fail on Windows if the configuration file does not exist at that location in the runner's workspace.

When you are done, type `exit` to disconnect. The tmate step completes and the workflow continues. Remove the tmate step after you identify the fix.

### Exercise 10: Debug a Matrix Failure with Conditional Debug Steps

**Task:** You have a matrix build that tests across three operating systems and four Node.js versions (12 combinations). One combination — `ubuntu-latest` with Node 18 — fails consistently, but all 11 other combinations pass. You need to add conditional debug steps that only activate when the failing combination runs, without disturbing the other jobs.

**Hint:** Use `matrix` variables in `if:` conditions to target the specific combination. Add a step that runs only when `matrix.os == 'ubuntu-latest'` and `matrix.node == '18'`.

**Solution:**

Add conditional debug steps that target the failing combination:

```yaml
name: Matrix Debug

on: [push]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [16, 18, 20, 22]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}

      - name: Debug — target failing combination
        if: matrix.os == 'ubuntu-latest' && matrix.node == '18'
        run: |
          echo "=== DEBUG: Failing combination ==="
          echo "OS: ${{ matrix.os }}"
          echo "Node: ${{ matrix.node }}"
          echo "Node path: $(which node)"
          echo "npm path: $(which npm)"
          echo "Node modules:"
          ls -la node_modules/ 2>/dev/null || echo "No node_modules yet"
          echo "Package lock:"
          cat package-lock.json 2>/dev/null | head -20
          echo "Env:"
          echo "NODE_ENV=$NODE_ENV"
          echo "CI=$CI"

      - name: Install dependencies
        run: npm ci

      - name: Debug — check after install
        if: matrix.os == 'ubuntu-latest' && matrix.node == '18'
        run: |
          echo "=== DEBUG: After install ==="
          ls -la node_modules/.package-lock.json 2>/dev/null
          npm ls --depth=0 2>/dev/null

      - name: Run tests
        run: npm test
```

The debug steps only execute for the failing combination. The other 11 jobs run without the diagnostic output, keeping their logs clean. When you find the cause (for example, a native module that does not support Node 18 on Ubuntu), fix it and then remove the debug steps.

You can also use `runner.debug` globally and target with matrix variables:

```yaml
- name: Debug diagnostics
  if: runner.debug == 'true' && matrix.os == 'ubuntu-latest'
  run: env | sort
```

This activates the diagnostics for all Ubuntu runners when debug mode is on, but keeps other OS runners quiet.

### Exercise 11: Build a Reusable Debugging Workflow

**Task:** Create a reusable debugging workflow that you can call from any other workflow in your organization. The workflow should accept a `run-id` input, download the logs for that run, analyze the logs for common error patterns (exit code 1, permission denied, resource not accessible, secret not found), and post a summary as a check run on the commit.

**Hint:** Use `workflow_call` with inputs to make the workflow reusable. Use the `gh` CLI inside the workflow to download logs and the GitHub API (via `GITHUB_TOKEN`) to create a check run. Use `grep` with patterns to analyze the logs.

**Solution:**

Create `.github/workflows/debug-analyzer.yml`:

```yaml
name: Debug Log Analyzer

on:
  workflow_call:
    inputs:
      run-id:
        required: true
        type: number
        description: 'The workflow run ID to analyze'
      job-name:
        required: false
        type: string
        default: ''
        description: 'Specific job name to analyze (empty = all jobs)'

jobs:
  analyze:
    runs-on: ubuntu-latest
    permissions:
      checks: write
      actions: read

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Download logs for the given run
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          mkdir -p logs
          if [ -n "${{ inputs.job-name }}" ]; then
            gh run view ${{ inputs.run-id }} --log --job ${{ inputs.job-name }} > logs/run.log
          else
            gh run view ${{ inputs.run-id }} --log > logs/run.log
          fi
          echo "Log size: $(wc -l < logs/run.log) lines"

      - name: Analyze logs for common errors
        id: analyze
        run: |
          LOG=logs/run.log
          ERRORS=""

          # Pattern 1: Exit code 1
          if grep -q "exit code 1" "$LOG"; then
            ERRORS="$ERRORS\n- exit_code_1: One or more steps exited with code 1"
          fi

          # Pattern 2: Permission denied
          if grep -qi "permission denied" "$LOG"; then
            ERRORS="$ERRORS\n- permission_denied: Permission denied error detected"
          fi

          # Pattern 3: Resource not accessible
          if grep -qi "resource not accessible" "$LOG"; then
            ERRORS="$ERRORS\n- resource_not_accessible: Integration permission issue"
          fi

          # Pattern 4: Secret not found
          if grep -qi "secret not found" "$LOG"; then
            ERRORS="$ERRORS\n- secret_not_found: A required secret is missing"
          fi

          # Pattern 5: Timeout
          if grep -qi "timeout" "$LOG"; then
            ERRORS="$ERRORS\n- timeout: A step timed out"
          fi

          echo "errors<<EOF" >> $GITHUB_OUTPUT
          echo -e "$ERRORS" >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT

      - name: Create check run with analysis summary
        uses: actions/github-script@v7
        with:
          script: |
            const errors = '${{ steps.analyze.outputs.errors }}';
            const summary = errors
              ? '## Debug Analysis Results\n\nFound the following issues:\n' + errors
              : '## Debug Analysis Results\n\nNo common error patterns detected in the logs.';

            await github.rest.checks.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              name: 'Debug Log Analyzer',
              head_sha: context.sha,
              status: 'completed',
              conclusion: errors ? 'neutral' : 'success',
              output: {
                title: 'Debug Log Analyzer Report',
                summary: summary,
                text: 'Full logs are available in the Actions artifacts for this workflow run.'
              }
            });
```

Call this workflow from any other workflow:

```yaml
name: CI

on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  debug-analysis:
    needs: build
    if: failure()
    uses: ./.github/workflows/debug-analyzer.yml
    with:
      run-id: ${{ github.run_id }}
      job-name: build
```

When your CI workflow fails, the debug analysis workflow runs automatically, downloads the logs from the failed run, scans for five common error patterns, and posts a summary as a check run. You see the analysis directly in the commit status without opening the workflow logs. Extend the analysis by adding more grep patterns for errors specific to your project.
