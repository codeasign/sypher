You have been working on a feature branch for three days. Your teammate has been committing to `main` in parallel. Now you need to bring your feature back into the main line — but your history has diverged, and you cannot just copy files by hand. Merging is Git's way of combining two divergent lines of history into one unified timeline.

## The Concept

A merge takes two branches, finds their point of divergence, and creates a new commit that combines both sets of changes into one snapshot. The result is a commit with two parents instead of one.

Think of two scientists writing separate sections of the same paper. Each works on their own chapter (branch) from the shared introduction (common ancestor). When both chapters are done, an editor (Git) reads both, combines them, and writes a new "merged" version that includes the introduction plus both chapters. The final version notes where each chapter came from — the two parent pointers.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git merge <branch>` | Merge `<branch>` into the current branch. | `git merge feature-login` |
| `git merge --no-ff <branch>` | Force a merge commit even if a fast-forward is possible. | `git merge --no-ff feature-login` |
| `git merge --squash <branch>` | Combine all of `<branch>`'s commits into one diff, staged but not committed. | `git merge --squash feature-login` |
| `git merge --abort` | Abort a merge that has conflicts and return to the pre-merge state. | `git merge --abort` |
| `git merge --continue` | Complete a merge after resolving conflicts (after `git add`). | `git merge --continue` |
| `git log --merge` | Show commits that touch conflicted files during a merge conflict. | `git log --merge` |
| `git diff --name-only --diff-filter=U` | List only files with merge conflicts (unmerged paths). | `git diff --name-only --diff-filter=U` |

**Windows note:** All merge commands are identical on Windows. When using `git merge --abort`, Git handles path separator differences automatically.

## How It Actually Works — Step by Step

### Step 1: Starting state — two divergent branches

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2))  ← main
                               │
                               └──▶ ((D0)) ──▶ ((D1))  ← feature
```

You are on `main` (HEAD → main → C2). The `feature` branch has two commits (`D0`, `D1`) that `main` does not. The merge base — the most recent common ancestor — is `C2`.

### Step 2: Git identifies the merge base

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2))  ← main  (also the merge base)
                               │
                               └──▶ ((D0)) ──▶ ((D1))  ← feature
```

Git runs `git merge-base main feature` internally. It finds `C2`. This is the commit from which both branches diverged. Git now has three snapshots to compare: the merge base (`C2`), the target (`main` at `C2` — no change), and the source (`feature` at `D1`).

### Step 3: Three-way merge calculation

Git compares three snapshots:

- **Base** (C2): the common ancestor
- **Ours** (main at C2): what the current branch has
- **Theirs** (feature at D1): what we are merging in

For each file, Git checks: Did *we* change it since the base? Did *they* change it? The logic:

| Ours changed? | Theirs changed? | Result |
|---|---|---|
| No | No | Keep base version |
| Yes | No | Use ours |
| No | Yes | Use theirs |
| Yes | Yes | Conflict — mark for resolution |

In this simple case, `main` has no commits since the base (`C2`), so all files take `feature`'s version. Git says "Already up to date" or performs a fast-forward (see Step 4).

### Step 4: Fast-forward merge (simple case)

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((D0)) ──▶ ((D1))  ← main
                                                        ← feature
                                                        ← HEAD
```

Because `main` has not diverged from `feature` — `main` is at `C2`, and `feature` started at `C2` — Git can simply move the `main` pointer forward to `D1`. This is called a **fast-forward merge**. No new commit is created. History stays linear.

### Step 5: True merge (divergent case)

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3))  ← main
                               │
                               └──▶ ((D0)) ──▶ ((D1))  ← feature
```

Now `main` has its own commit `C3` since the base. Git cannot fast-forward. It creates a new **merge commit** `M0`:

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3)) ──▶ ((M0))  ← main
                               │                /       ← HEAD
                               └──▶ ((D0)) ──▶ ((D1))  ← feature
```

`M0` is a special commit: it has **two parents** — `C3` (the previous tip of `main`) and `D1` (the tip of `feature`). Its snapshot is the combination of both histories.

### Step 6: After the merge

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3)) ──▶ ((M0))  ← main
                               │                /       ← HEAD
                               └──▶ ((D0)) ──▶ ((D1))  ← feature
```

The `feature` branch pointer remains at `D1` — merging does not delete the source branch. You can delete it with `git branch -d feature` once you verify the merge is correct. The merge commit's snapshot equals what you would get if you applied all of `feature`'s changes on top of `main`'s.

## When to Use This / When Not To

**Use merge when:**
- You want to preserve the full branching history — each feature's commits stay grouped on their branch even after integration
- You are collaborating and want other developers to see which commits came from which branch
- The feature branch has many small commits that tell a story you want to keep

**Do not use merge when:**
- You want a linear, clean history with no branch artifacts — use `git rebase` instead (covered in a later topic)
- You are merging a single-commit hotfix into `main` — a fast-forward or cherry-pick keeps history simpler
- The branch has not diverged from `main` and you want every commit to appear as if it was made on `main` — use `git merge --squash`

**Key trade-off:** Merge commits preserve history exactly as it happened but create a non-linear DAG. Tools like `git log --first-parent` let you follow just the mainline, skipping merge bubbles. Choose merge when collaboration context matters more than a flat timeline.
