Work through these exercises in a terminal. Tier 1 builds single-merge muscle memory. Tier 2 combines merging with prior Git branching skills. Tier 3 presents messy scenarios with multiple valid approaches.

## Tier 1 — Beginner

### Exercise 1

**Task:** Create a repo, make 2 commits on `main`, then create and switch to a branch `feature-a`. Add 1 commit on `feature-a`. Switch back to `main` and merge `feature-a`. What type of merge occurs?

<details>
<summary>Hint</summary>
Check `git log --oneline --graph` after the merge. If the history is linear, it was a fast-forward. If there is a merge commit with two parents, it was a true merge.
</details>

### Solution

```bash
mkdir -p ~/merge-beginner-1 && cd ~/merge-beginner-1
git init
echo "A" > f.txt && git add f.txt && git commit -m "Commit A"
echo "B" > f.txt && git add f.txt && git commit -m "Commit B"
git switch -c feature-a
echo "C" > f.txt && git add f.txt && git commit -m "Commit C"
git switch main
git merge feature-a
git log --oneline --graph
```

Since `main` had no commits after `feature-a` branched off, Git performed a fast-forward. The graph shows a linear chain: A → B → C.

### Exercise 2

**Task:** Create a repo with 3 commits on `main`. Create a branch `feature-b`, add 2 commits there. Before merging, add 1 more commit on `main`. Then merge `feature-b` into `main`. What type of merge occurs? How many parents does the merge commit have?

<details>
<summary>Hint</summary>
Use `git log --oneline --graph --all` to see the full topology. A merge commit has two parents visible as two lines joining.
</details>

### Solution

```bash
mkdir -p ~/merge-beginner-2 && cd ~/merge-beginner-2
git init
echo "1" > f.txt && git add f.txt && git commit -m "Commit 1"
echo "2" > f.txt && git add f.txt && git commit -m "Commit 2"
echo "3" > f.txt && git add f.txt && git commit -m "Commit 3"
git switch -c feature-b
echo "B1" > f.txt && git add f.txt && git commit -m "Feature B-1"
echo "B2" > f.txt && git add f.txt && git commit -m "Feature B-2"
git switch main
echo "4" > f.txt && git add f.txt && git commit -m "Commit 4"
git merge feature-b
git log --oneline --graph
```

This is a true merge — a merge commit with two parents appears. Run `git log -1 --format="%P"` to see the two parent hashes.

### Exercise 3

**Task:** Merge a branch using `--no-ff` even when fast-forward is possible. Compare the graph with a regular fast-forward merge. What is the practical difference?

<details>
<summary>Hint</summary>
Create a branch, add one commit, merge with `--no-ff`. Then repeat in a fresh repo without `--no-ff` and compare the graph shapes.
</details>

### Solution

```bash
mkdir -p ~/merge-beginner-3 && cd ~/merge-beginner-3
git init
echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c feature-c
echo "feature" > f.txt && git add f.txt && git commit -m "Feature C"
git switch main
git merge --no-ff feature-c
git log --oneline --graph
```

The graph shows a merge commit with two parents. Without `--no-ff`, the graph would be linear. The `--no-ff` commit explicitly marks the feature boundary in the DAG.

### Exercise 4

**Task:** Perform a squash merge of a branch with 3 commits. Confirm the target branch has exactly one new commit containing all changes.

<details>
<summary>Hint</summary>
Use `git merge --squash <branch>` which stages all changes without committing. Then `git commit -m "..."`. Compare commit counts.
</details>

### Solution

```bash
mkdir -p ~/merge-beginner-4 && cd ~/merge-beginner-4
git init
echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c multi-commit
echo "change 1" > a.txt && git add a.txt && git commit -m "Change 1"
echo "change 2" > b.txt && git add b.txt && git commit -m "Change 2"
echo "change 3" > c.txt && git add c.txt && git commit -m "Change 3"
git switch main
git merge --squash multi-commit
git commit -m "Squashed feature branch"
git log --oneline
```

`main` shows exactly two commits: "Base" and "Squashed feature branch". None of the three intermediate commit messages appear.

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a repo with 2 branches that each modify the same file but on different lines (one changes line 1, the other changes line 5). Merge them. Does Git auto-merge or report a conflict?

<details>
<summary>Hint</summary>
Git auto-merges changes on different lines. Use a file with at least 5 lines before branching. Each branch modifies a distinct line.
</details>

### Solution

```bash
mkdir -p ~/merge-inter-1 && cd ~/merge-inter-1
git init
printf "line1\nline2\nline3\nline4\nline5\n" > f.txt
git add f.txt && git commit -m "Base"
git switch -c branch-a
sed -i 's/line1/line1-modified/' f.txt
git add f.txt && git commit -m "Change line 1"
git switch main
git switch -c branch-b
sed -i 's/line5/line5-modified/' f.txt
git add f.txt && git commit -m "Change line 5"
git switch main
git merge branch-a
git merge branch-b
```

The first merge (`branch-a`) fast-forwards. The second merge (`branch-b`) auto-merges without conflict — Git sees that `branch-a` changed line 1 and `branch-b` changed line 5, which do not overlap.

### Exercise 2

**Task:** Merge a branch using `--no-ff`, then inspect the merge commit's parents with `git cat-file -p <hash>` and `git show --format="%P"`. Verify the commit has two parent hashes.

<details>
<summary>Hint</summary>
Capture the merge commit hash from `git log`. Use `git cat-file -p <hash>` — the lines starting with `parent` show the two parent hashes.
</details>

### Solution

```bash
mkdir -p ~/merge-inter-2 && cd ~/merge-inter-2
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c feat
echo "feat" > f.txt && git add f.txt && git commit -m "Feat"
git switch main
git merge --no-ff feat
MERGE_HASH=$(git log --format="%H" -1)
git cat-file -p $MERGE_HASH | head -10
```

The output shows two `parent` lines. `git show --format="%P" -s` prints just the two parent hashes.

### Exercise 3

**Task:** Create a branch `release/v1`, add 2 commits. Create `release/v2` from `release/v1`'s first commit. Add 1 commit to each branch. Now merge `release/v2` into `release/v1`. Then merge the result back into `main`. Does the merge-base change after the first merge?

<details>
<summary>Hint</summary>
Use `git merge-base` before and after the first merge to see how the common ancestor evolves.
</details>

### Solution

```bash
mkdir -p ~/merge-inter-3 && cd ~/merge-inter-3
git init && echo "start" > f.txt && git add f.txt && git commit -m "Start"
git switch -c release/v1
echo "v1-a" >> f.txt && git add f.txt && git commit -m "V1 change A"
echo "v1-b" >> f.txt && git add f.txt && git commit -m "V1 change B"
git switch main
git switch -c release/v2 <hash-of-Start-commit>
echo "v2-a" >> f.txt && git add f.txt && git commit -m "V2 change A"
git switch release/v1
git merge release/v2
git merge main
git merge-base main release/v1
```

The merge-base after the merge is the merge commit itself — merging establishes a new common ancestor for future merges.

## Tier 3 — Advanced

### Exercise 1: Merge with file-level conflicts

**Scenario:** Two developers each refactored the same configuration file on different branches. Both renamed keys and added new sections. Git reports conflicts on every changed line.

**Task:** Create this scenario deliberately. Then resolve it by picking the best of both versions, not just one branch's file. Write a script that sets up two branches with conflicting changes to the same YAML config file. Then merge and resolve by accepting specific lines from each branch.

<details>
<summary>Hint</summary>
Use a config file with named sections. Each branch renames one section and adds content to another. After the conflict, use `git mergetool` or manual editing to build a composite version.
</details>

### Solution — one valid approach

Setup:

```bash
mkdir -p ~/merge-adv-1 && cd ~/merge-adv-1
git init
cat > config.yaml << 'EOF'
database:
  host: localhost
  port: 5432
cache:
  engine: redis
  ttl: 300
EOF
git add config.yaml && git commit -m "Base config"
git switch -c dev-branch-a
sed -i 's/host: localhost/host: prod-db.internal/' config.yaml
sed -i 's/ttl: 300/ttl: 600/' config.yaml
git add config.yaml && git commit -m "Update DB host and cache TTL"
git switch main
git switch -c dev-branch-b
sed -i 's/port: 5432/port: 6432/' config.yaml
sed -i 's/engine: redis/engine: valkey/' config.yaml
git add config.yaml && git commit -m "Change port and cache engine"
```

Merge:

```bash
git switch main
git merge dev-branch-a   # fast-forward or merge
git merge dev-branch-b   # triggers conflict in config.yaml
```

Open `config.yaml` — conflict markers show both versions. Manually build the composite:

```yaml
database:
  host: prod-db.internal
  port: 6432
cache:
  engine: valkey
  ttl: 600
```

Stage and complete:

```bash
git add config.yaml
git merge --continue
```

**Alternate approaches:** Use `git checkout --ours config.yaml` to take one side entirely, then manually edit. Or use `git mergetool` with a visual diff tool like `vimdiff` or VS Code.

### Exercise 2: The accidental merge commit in a pull-request workflow

**Scenario:** Your team requires a linear history on `main` (no merge commits) to support `git bisect`. A junior developer merged `feature` into `main` using `git merge` without `--no-ff`, which created a merge commit. Now the history has a bubble.

**Task:** Create this scenario. Then remove the merge commit while preserving all changes from both branches. Use `git reset` and re-do the merge as a fast-forward or squash. Do not use `git rebase` — assume the branch was already merged and the merge commit is the only problem.

<details>
<summary>Hint</summary>
Use `git reset --soft HEAD~1` to undo the merge commit while keeping all changes staged. Then commit them as a single normal commit.
</details>

### Solution — one valid approach

Setup:

```bash
mkdir -p ~/merge-adv-2 && cd ~/merge-adv-2
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c feature
echo "feature work" > feature.txt && git add feature.txt && git commit -m "Feature work"
git switch main
echo "main work" > main.txt && git add main.txt && git commit -m "Main work"
git merge --no-ff feature   # creates the accidental merge commit
```

Now fix it:

```bash
git log --oneline --graph
```

You see `*   M0 Merge branch 'feature'`. Undo it:

```bash
git reset --soft HEAD~1
git status   # All changes from both branches are staged
```

Now commit normally:

```bash
git commit -m "Integrate feature and main changes"
git log --oneline --graph
```

The graph is now linear. All changes are preserved; only the merge topology is flattened.

**Alternative approach:** `git reset --mixed HEAD~1` (default) unstages the changes but keeps them in the working directory, letting you review before committing. Same end result.

### Exercise 3: Recreate a lost feature branch from a merge commit

**Scenario:** A feature branch was merged into `main` and then deleted. Two weeks later, you need to create a hotfix based on the feature's code, but the original branch is gone. The merge commit still exists in `main`'s history.

**Task:** Recreate a branch that starts at the feature's pre-merge tip — not at the merge commit itself. Verify it contains the feature's commits, not the merge bubble.

<details>
<summary>Hint</summary>
The merge commit has two parents. `git log --merges -1 --format="%P"` shows both. The second parent (index 1) is the feature branch's tip. Use `git branch recovered <hash>` with that hash.
</details>

### Solution — one valid approach

Setup:

```bash
mkdir -p ~/merge-adv-3 && cd ~/merge-adv-3
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c secret-feature
echo "secret1" > s1.txt && git add s1.txt && git commit -m "Secret part 1"
echo "secret2" > s2.txt && git add s2.txt && git commit -m "Secret part 2"
git switch main
echo "maintenance" >> f.txt && git add f.txt && git commit -m "Maint"
git merge --no-ff secret-feature
git branch -d secret-feature   # deleted!
```

Recreate:

```bash
# Get the second parent hash of the most recent merge commit
SECOND_PARENT=$(git log --merges -1 --format="%P" | awk '{print $2}')
git branch recovered $SECOND_PARENT
git log recovered --oneline --graph
```

The `recovered` branch shows "Secret part 2", "Secret part 1", "Base" — exactly the pre-merge feature branch history, without the merge bubble or subsequent `main` commits.

**Alternative approach:** If there are multiple merge commits, run `git log --merges --all --format="%H %P"` to list all merge commits and their parents, then pick the right second parent manually.
