You have been working on a feature branch for three days. You run `git push` and see a wall of text about rejected pushes, diverged histories, and non-fast-forward updates. Your teammate says "you forgot to pull first." You have no idea what your code has to do with their code, or where "origin" even lives.

A **remote** is a bookmark pointing to another copy of your repository — usually on a server like GitHub, GitLab, or Bitbucket. `origin` is the default name for the remote you cloned from, just as you might save a friend's address as "home" in your phone. You push commits to a remote so others can see them; you pull or fetch from a remote so you can see theirs.

## Command and Syntax Reference

| Command | What it does | Exact syntax | Common flags | One-line example |
|---|---|---|---|---|
| `git clone` | Download a repository from a remote for the first time | `git clone <url> [<directory>]` | `--depth 1` (shallow clone, faster for large repos) | `git clone https://github.com/user/project.git` |
| `git remote add` | Add a new remote URL to your local repository | `git remote add <name> <url>` | None | `git remote add upstream https://github.com/original/project.git` |
| `git remote -v` | List all remotes with their fetch and push URLs | `git remote -v` | `-v` (verbose, shows URLs) | `git remote -v` |
| `git remote remove` | Delete a remote bookmark | `git remote remove <name>` | None | `git remote remove upstream` |
| `git fetch` | Download new commits from a remote without merging | `git fetch [<remote>] [<branch>]` | `--prune` (delete stale remote-tracking refs) | `git fetch origin` |
| `git pull` | Fetch from a remote and merge into the current branch | `git pull [<remote>] [<branch>]` | `--rebase` (rebase instead of merge) | `git pull origin main` |
| `git push` | Upload your commits to a remote | `git push [<remote>] [<branch>]` | `-u` (set upstream), `--force` (overwrite remote history), `--dry-run` (show what would push) | `git push -u origin main` |
| `git remote rename` | Rename a remote bookmark | `git remote rename <old-name> <new-name>` | None | `git remote rename origin upstream` |
| `git push --delete` | Delete a branch on the remote | `git push <remote> --delete <branch>` | None | `git push origin --delete old-feature` |

On Windows, Git for Windows installs `git.exe` into your PATH. On macOS and Linux, git is typically available natively. All commands above work identically across platforms.

Your local repository tracks the remote's branch state through a **remote-tracking branch** — a read-only pointer named `origin/main` that marks where the remote's `main` branch was the last time you fetched or pulled. You never commit directly to `origin/main`; git updates it for you during fetch and pull.

## How Fetch and Push Actually Work — Step by Step

### Step 1: Cloning Creates the Remote

When you run `git clone https://github.com/user/project.git`, git performs four actions:

1. Creates a new directory and initializes an empty repository
2. Adds a remote called `origin` pointing to the URL
3. Downloads every commit from the remote's default branch
4. Checks out the latest commit into a working tree

Your local `main` branch and the remote-tracking `origin/main` now point at the same commit.

│  ┌────────────────────────────┐  │

│  origin → https://github.com/..  │
│  Remote-tracking: origin/main    │
└──────────────────────────────────┘

### Step 2: You Commit Locally; Remote Stays Put

You edit files, stage them, and run `git commit`. A new commit D appears on your local `main`. The remote is completely unaware — it still has commit B.

### Step 3: Fetch Gets Remote Changes Without Merging

While you were working, a teammate pushed commit C to `origin/main`. You run `git fetch origin`. Git contacts the remote, downloads commits it does not have (commit C), and moves `origin/main` to point at C. Your local `main` stays at D — fetch never touches your branches.

<AsciiDiagram id="git-remotes-push-pull/after-fetch" content={`
┌──────────────────────────────────┐
│  Local Repository                │
│  ┌────────────────────────────┐  │
│  │  Commit A ──> Commit B ──> C  │
│  │       │             │     │   │
│  │                        C: origin/main │
│  │                          │          │
│  │                       ┌──┘          │
│  │                       ▼             │
│  │                    Commit D          │
│  │                       │             │
│  │                  main: D            │
│  └────────────────────────────────────┘
│  origin/main updated to C              │
│  main still at D (not merged)          │
└──────────────────────────────────────────┘
`} alt="Diagram showing the local repository after a fetch, where origin/main has been updated to commit C while local main remains at commit D." />

Internally, fetch works by:
- Opening a connection to the remote via HTTPS or SSH
- Negotiating which commits each side has using the git protocol's "want" and "have" handshake
- Packing and transferring only the missing objects (delta compression reduces transfer size by 60-90 percent)
- Updating remote-tracking refs under `.git/refs/remotes/origin/`
- Never touching your local branch refs or working tree

### Step 4: Pull Combines Fetch and Merge

You run `git pull origin main`. Git does two things in sequence:

1. Runs `git fetch origin main` to download new commits
2. Runs `git merge origin/main` to merge the remote-tracking branch into your current branch

If no commits have arrived on the remote since your last fetch, step 1 downloads nothing and step 2 is a fast-forward — git simply moves `main` forward to match `origin/main`.

If both you and a teammate made commits (our scenario), git creates a merge commit E that has two parents: D (your change) and C (the teammate's change).

### Step 5: Push Uploads Your Commits

You run `git push origin main`. Git:

1. Connects to the remote
2. Checks that your local history includes all commits the remote has (the fast-forward check)
3. Packs and transfers commits D and E not yet on the remote
4. Asks the remote to advance its `main` ref to match yours
5. Updates your local `origin/main` to match

If the fast-forward check fails (someone pushed since your last pull), git rejects the push — you must pull first.

### Step 6: Remote-Tracking Branches Keep You Oriented

After every fetch, pull, or push that succeeds, git updates `origin/<branch>` to reflect the remote's state. These remote-tracking branches live in `.git/refs/remotes/origin/` and act like a snapshot of "what the server had last time you talked to it."

You can view them with:

```bash
git branch -r
```

Output:

```
origin/HEAD -> origin/main
origin/main
origin/develop
origin/feature/logins
```

You can inspect a remote-tracking branch without switching to it:

```bash
git log origin/main
git diff main origin/main
```

The upstream of a local branch is the remote branch it pushes to and pulls from by default. When you run `git push -u origin main`, git sets `main` to track `origin/main`. After that, simply running `git push` or `git pull` without arguments uses those defaults.

## The Upstream Remote Pattern

In open-source workflows, you often see two remotes:

| Remote name | URL | Purpose |
|---|---|---|
| `origin` | Your fork | You push here |
| `upstream` | The original repository | You fetch from here |

This lets you fetch the latest changes from the original project without pushing to it:

<AsciiDiagram id="git-remotes-push-pull/two-remote-workflow" content={`
┌─────────────────────────────────────────────┐
│  Original Repository (upstream)             │
│  main: latest stable code                   │
│       │   fetch / pull                      │
│       ▼                                     │
│  Your Fork (origin)                         │
│  main: behind upstream by N commits         │
│  feature: your new work                     │
│       │   push                              │
│       ▼                                     │
│  Your Local Machine                         │
│  origin/main → your fork's main             │
│  upstream/main → original project's main    │
│  feature → your new branch                  │
└─────────────────────────────────────────────┘
`} alt="Diagram illustrating the two-remote workflow, showing the relationship between the original upstream repository, the user's fork on GitHub, and the local machine." />

Set it up with:

```bash
git remote add upstream https://github.com/original/project.git
git fetch upstream
```

## When to Use This: Signals

**Push when:** You have working, tested commits on a branch and want to share them, back them up to the server, or trigger a CI pipeline. Push to `origin` early and often — it is not a deploy until you open a pull request.

**Pull when:** You are about to start work and want the latest changes, or when git tells you your push was rejected. A good habit is to pull before every coding session and before every push.

**Fetch when:** You want to see what is new on the remote without disrupting your working tree. Fetch is safe to run at any time — it never changes your files or branches. Run `git fetch` in the background periodically to keep your remote-tracking branches fresh.

**Clone when:** You are setting up a local copy of a repository for the first time.

## When Not to Use This

**Do not force-push** (`git push --force`) to a shared branch unless you are absolutely certain everyone has coordinated. Force push overwrites remote history, which can destroy your teammates' commits. Prefer `git push --force-with-lease` — it checks that your remote-tracking branch still matches the remote before overwriting, preventing accidental destruction of commits you have not seen.

**Do not pull with rebase** (`git pull --rebase`) on a branch others are working from unless your team agrees on a rebase workflow. Rebasing rewrites commit hashes, creating duplicates when someone else pulls.

**Do not push every local branch.** Push only branches you are actively collaborating on. Git will happily upload all 15 stale experiment branches if you run `git push --all`. Use explicit branch names instead.
