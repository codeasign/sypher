You have a Python script that took you three days to write. It works. You make one small change to test an edge case, the script breaks, and you cannot remember what the original looked like. There is no undo for file edits. There is no history. You are one Ctrl+S away from losing work you cannot reconstruct.

Git solves that by recording snapshots of your project over time. Each snapshot is a commit, and you decide which changes belong in each snapshot using the staging area.

## The Command Reference

| Command | What It Does | Syntax Example | Common Flags |
|---|---|---|---|
| `git init` | Creates a new Git repository in the current folder | `git init` | `--bare` for server repos, `--initial-branch main` to set branch name |
| `git status` | Shows which files are changed, staged, or untracked | `git status` | `--short` for compact output, `--ignored` to show ignored files |
| `git add` | Moves changes from working directory to staging area | `git add main.py` | `-A` stages all changes, `-p` stages interactively by hunk, `-n` dry run |
| `git commit` | Takes everything in staging area and creates a snapshot | `git commit -m "fix login bug"` | `-m "message"` inline message, `--amend` edit last commit, `--no-edit` keep message |
| `git log` | Displays the commit history | `git log` | `--oneline` compact one line per commit, `--graph` show branch topology, `-5` last 5 commits |
| `git diff` | Shows unstaged differences between versions | `git diff` | `--staged` show staged diff, `--word-diff` highlight changed words |
| `git commit -am` | Adds tracked files and commits in one step | `git commit -am "fix bug"` | Only works on already-tracked files; new files still need `git add` |
| `git restore` | Discards changes in working directory | `git restore main.py` | `--staged` unstage a file, `--source=HEAD` restore to last commit |

Windows users: Git Bash is recommended for a consistent experience. On macOS and Linux, git uses the system terminal directly. File paths on Windows use forward slashes (`src/main.py`) even on Git Bash, though backslashes work in PowerShell with escaping. Case-insensitive filesystems on Windows and macOS can cause subtle issues — a file named `Config.py` and `config.py` are the same file on disk but different objects in the Git index.

Changes flow left to right. The working directory is your editor buffer. The staging area is your review desk. The repository is the permanent archive.

## How It Actually Works

### Step 1: Creating a Repository with `git init`

When you run `git init`, Git creates a hidden `.git` folder inside your project directory. This folder contains the entire engine: object database, references, configuration, and hooks. Before this command, your folder is just a plain directory. After, it is a repository waiting for your first snapshot.

<AsciiDiagram id="git-init-add-commit/before-after-init" alt="Side-by-side comparison of a project directory before git init showing only main.py, and after git init showing the hidden .git directory with objects, refs, HEAD, config, and hooks subdirectories" content={`
 Before git init:                After git init:
 ┌──────────────────┐            ┌──────────────────────┐
 │  my-project/     │            │  my-project/         │
 │  └── main.py     │            │  ├── main.py         │
 │  (plain folder)  │            │  └── .git/           │
 └──────────────────┘            │      ├── objects/    │
                                 │      ├── refs/       │
                                 │      ├── HEAD        │
                                 │      ├── config      │
                                 │      └── hooks/      │
                                 └──────────────────────┘
`} />

The `.git` directory is around 30 KB on a fresh init. Inside `objects/`, Git stores every version of every file as content-addressable blobs. Inside `refs/`, it stores pointers to commits — branch names, tags, and HEAD. The `HEAD` file contains a single line: `ref: refs/heads/main`, telling Git which branch is currently active.

Run `git status` immediately after init and you see:

```
On branch main

No commits yet

nothing to commit (create/copy files and use "git add" to track)
```

### Step 2: Tracking Files with `git add`

When you create or modify a file and run `git add main.py`, Git computes a SHA-1 hash of the file contents, stores the content as a blob object in `.git/objects/`, and records the filename and blob reference in the staging area (the index file at `.git/index`).

The hash is a 40-character hex string like `e69de29bb2d1d6434b8b29ae775ad8c2e48c5391`. Git uses this hash as the object's filename inside `objects/`, split into a 2-character directory prefix and a 38-character filename. This content-addressing means identical files across different commits share the same object — Git never duplicates data.

You can add multiple files with one command: `git add .` stages everything in the current directory and subdirectories. `git add -A` stages all changes across the entire repository, including deletions. `git add -p` walks through each chunk of changed code and asks you whether to stage it — useful for splitting unrelated changes in the same file into separate commits.

### Step 3: Creating a Snapshot with `git commit`

When you run `git commit -m "initial commit"`, Git takes the current staging area and creates a commit object. This commit object stores:

- A tree object that maps every staged filename to its blob hash
- The commit message
- Your name and email (from git config)
- A timestamp
- Zero or more parent commit hashes (none for the first commit)

After commit, `git log --oneline` shows:

```
a1b2c3d (HEAD -> main) initial commit
```

The `HEAD` pointer now points to this commit. The branch `main` also points to this commit. They are currently identical.

### Step 4: The Second Commit — How the DAG Grows

The second commit is where the Directed Acyclic Graph (DAG) becomes visible. You edit `main.py`, run `git add main.py`, then `git commit -m "add error handling"`. The new commit object stores the previous commit hash as its parent.

<AsciiDiagram id="git-init-add-commit/commit-graph-two-commits" alt="Commit graph showing two commits with the second commit pointing back to the first as its parent, and both HEAD and main pointing to the latest commit" content={`
 ┌────────────┐     ┌────────────┐
 │ a1b2c3d    │     │ d5e6f7g8   │
 │ initial    │◀────│ add error  │
 │ commit     │parent│ handling   │
 └────────────┘     └────────────┘
                          │
                     ┌────┴────┐
                     │  HEAD   │
                     │  main   │
                     └─────────┘
`} />

Each commit points backward to its parent. The chain is one-way — you can walk backward through history but not forward. Git stores only the changes (technically, only new blob objects for changed files) plus metadata. If `main.py` changed but `README.md` did not, the second tree object reuses the same blob hash for `README.md`. This is how Git remains efficient even with thousands of commits.

### Step 5: Viewing History with `git log`

`git log` walks the parent chain from HEAD backward and displays each commit. With `--oneline --graph --all`, you see the branch structure visually:

```
* d5e6f7g (HEAD -> main) add error handling
* a1b2c3d initial commit
```

The `--graph` flag draws connecting lines. The `--all` flag shows every branch, not just the current one. You can filter by author (`--author="name"`), by date (`--after="2026-06-01"`), or by file path (`git log -- main.py`).

## When to Use This / When Not To

**Use `git init` when:**
- You start a new project from scratch
- You want version control on a local project without a remote
- You are experimenting and want the safety of snapshots

**Do not use `git init` when:**
- You want to clone an existing project — use `git clone <url>` instead, which runs init and fetches history in one command
- You already have a `.git` folder in the directory (running init again is harmless but pointless)

**Use `git add` when:**
- You want to include a file in the next commit
- You want to selectively stage only part of your changes with `-p`
- You need to track a new file Git has never seen before

**Do not use `git add` when:**
- The file is generated output (binaries, logs, build artifacts) — add it to `.gitignore` instead
- You are committing a secret or API key — once committed, it lives in history forever
- You want to commit every tracked change without reviewing — `git commit -am` skips the staging review but only works on already-tracked files

**Use `git commit` when:**
- You have a logically complete unit of work — one bug fix, one feature, one refactor
- You want a restore point you can return to

**Do not use `git commit` when:**
- Your code does not compile or tests do not pass (commit working code only)
- Your commit message is vague like "fix stuff" or "updates"
- You have unrelated changes in the same commit — split them so each commit does one thing

The staging area exists precisely so you can build clean commits. A commit that changes 2 files for a bug fix and 3 files for a new feature in one snapshot makes it impossible to revert one without the other. Use `git add -p` to stage only the relevant hunks, or keep unrelated work on separate branches.
