## Setup

Create a new directory and initialize a Git repository. Open your terminal (Git Bash on Windows, or the built-in terminal on macOS and Linux).

**Windows (Git Bash):**
```bash
cd /c/projects
mkdir recipe-manager
cd recipe-manager
git init
```

**macOS / Linux:**
```bash
cd ~/projects
mkdir recipe-manager
cd recipe-manager
git init
```

Verify the repository was created:

```bash
git status
```

Expected output:

```
On branch main

No commits yet

nothing to commit (create/copy files and use "git add" to track)
```

Now create two files in the `recipe-manager` directory. You can use any text editor or the terminal.

Create `recipes.txt` with these contents:

```
Recipe Manager
==============
- Pancakes
- Omelette
```

Create `README.md` with these contents:

```
# Recipe Manager
A simple project to learn Git.
```

## Configuring Your Identity

Before your first commit, set your name and email. Git embeds this information in every commit you create.

```bash
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

These settings apply only to this repository. To set them globally for all repositories, add the `--global` flag:

```bash
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
```

Check your settings:

```bash
git config --list
```

You should see your name and email in the output. This configuration is stored in `.git/config` for local settings or `~/.gitconfig` for global settings. Without it, Git refuses to create commits.

## The Walkthrough

### Step 1: Check the Repository Status

Run `git status` to see what Git knows about your files.

```bash
git status
```

Output:

```
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md
        recipes.txt

nothing added to commit but untracked files present (use "git add" to track)
```

Both files appear under "Untracked files." Git sees them on disk but has no record of them in the staging area or repository. They are invisible to Git until you explicitly add them.

**Checkpoint:** You should see exactly 2 untracked files. If you see different files, you may have extra files in your directory. Remove any extra files so you start clean.

### Step 2: Stage a Single File

Stage `recipes.txt`:

```bash
git add recipes.txt
```

Run status again:

```bash
git status
```

Output:

```
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   recipes.txt

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md
```

`recipes.txt` moved from "Untracked" to "Changes to be committed." The file is now in the staging area. `README.md` remains untracked.

 └──────────────────────┘
`} />

**Checkpoint:** `git status` shows exactly 1 staged file and 1 untracked file.

### Step 3: Stage the Remaining File

```bash
git add README.md
git status
```

Output:

```
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   README.md
        new file:   recipes.txt
```

Both files are now staged. The staging area contains the exact contents of both files at the moment you ran `git add`. If you edit a file after staging it, those new edits will not be in the next commit unless you stage again.

**Checkpoint:** "Changes to be committed" shows both files. There are zero untracked files.

### Step 4: Create Your First Commit

```bash
git commit -m "initial commit: add recipe list and readme"
git status
```

Output:

```
[main (root-commit) a1b2c3d] initial commit: add recipe list and readme
 2 files changed, 5 insertions(+)
 create mode 100644 README.md
 create mode 100644 recipes.txt

On branch main
nothing to commit, working tree clean
```

"Root-commit" means this is the first commit in the repository — it has no parent commit. The hash `a1b2c3d` is an abbreviated version of the full 40-character SHA-1 hash. The message "nothing to commit, working tree clean" means your working directory matches the staging area, which matches the most recent commit. All three trees are in sync.

**Checkpoint:** `git status` shows "nothing to commit, working tree clean."

### Step 5: View the Commit History

```bash
git log
```

Output:

```
commit a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0 (HEAD -> main)
Author: Your Name <your.email@example.com>
Date:   Fri Jul 3 10:00:00 2026 -0400

    initial commit: add recipe list and readme
```

Compact view:

```bash
git log --oneline
```

Output:

```
a1b2c3d (HEAD -> main) initial commit: add recipe list and readme
```

The `--oneline` flag shows each commit as a single line with the abbreviated hash and subject line. `HEAD -> main` means HEAD points to the `main` branch, which points to this commit.

**Checkpoint:** `git log --oneline` shows exactly 1 commit.

### Step 6: Make a Second Change

Edit `recipes.txt` to add a new recipe. Append this line:

```
- Spaghetti Bolognese
```

After saving, run `git status`:

```bash
git status
```

Output:

```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   recipes.txt

no changes added to commit (use "git add" and/or "git commit -a")
```

`recipes.txt` appears under "Changes not staged for commit." Git knows the file has changed because the file's current contents produce a different hash than the blob stored in the last commit.

**Checkpoint:** `git status` shows 1 modified file under "Changes not staged for commit."

### Step 7: Stage and Commit the Change

```bash
git add recipes.txt
git commit -m "add spaghetti bolognese recipe"
git log --oneline
```

Output:

```
d5e6f7g (HEAD -> main) add spaghetti bolognese recipe
a1b2c3d initial commit: add recipe list and readme
```

You now have 2 commits. The newest commit `d5e6f7g` is at the top. Its parent is `a1b2c3d`. The graph now has 2 nodes.

**Checkpoint:** `git log --oneline` shows 2 commits and `git status` shows clean.

### Step 8: See the Difference Between Commits

```bash
git diff a1b2c3d d5e6f7g
```

Output:

```
diff --git a/recipes.txt b/recipes.txt
index e69de29..f8a7b6c 100644
--- a/recipes.txt
+++ b/recipes.txt
@@ -2,3 +2,4 @@ Recipe Manager
 ==============
 - Pancakes
 - Omelette
+- Spaghetti Bolognese
```

The `diff` output shows exactly what changed between the two commits. Lines starting with `+` were added. Lines starting with `-` were removed. The `@@` line shows the context location and line count.

To see the most recent change without specifying hashes:

```bash
git diff HEAD~1 HEAD
```

`HEAD~1` means "one commit before HEAD." This notation is called relative refspecs and works with any depth: `HEAD~3` is 3 commits back.

**Checkpoint:** `git diff HEAD~1 HEAD` shows the addition of the Spaghetti Bolognese line.

### Step 9: Create a Third File and Stage in One Shot

Create a new file called `notes.txt` with any content you like. Then stage and commit all changes without separate commands for tracked files.

```bash
echo "Shopping list: eggs, flour, pasta, tomatoes" > shopping.txt
git add shopping.txt
git commit -m "add shopping list"
```

Now run the log with graph view:

```bash
git log --oneline --graph --all
```

Output:

```
* e8f9a0b (HEAD -> main) add shopping list
* d5e6f7g add spaghetti bolognese recipe
* a1b2c3d initial commit: add recipe list and readme
```

You have 3 commits in a straight line. The `*` at the start of each line is the graph marker — with branches, these connect with `|` and `\` characters to show divergence.

**Checkpoint:** `git log --oneline` shows 3 commits.

## Common Errors at This Stage

### Error 1: `fatal: not a git repository (or any of the parent directories): .git`

```
fatal: not a git repository (or any of the parent directories): .git
```

**Cause:** You ran a Git command in a directory that has no `.git` folder, or you deleted the `.git` folder accidentally. Git walks up the directory tree looking for `.git`; if it reaches the filesystem root without finding one, it fails.

**Fix:** Run `git init` to create a repository, or `cd` to the correct directory. Check your current directory with `pwd` (or `cd` on its own on Windows).

### Error 2: `Please tell me who you are`

```
Author identity unknown

*** Please tell me who you are.

Run

  git config --global user.email "you@example.com"
  git config --global user.name "Your Name"

to set your account's default identity.
Omit --global to set the identity only in this repository.
```

**Cause:** You tried to commit without setting your name or email. Git requires this information for every commit because commit authorship is permanent.

**Fix:** Run `git config user.name "Your Name"` and `git config user.email "you@example.com"`. Use `--global` if you want these settings to apply to every repository on your machine. After setting, run your commit command again.

### Error 3: `nothing added to commit but untracked files present`

```
On branch main
nothing added to commit but untracked files present (use "git add" to track)
```

**Cause:** You ran `git commit` but no files were in the staging area. The files exist on disk but Git has not been told to include them in the next snapshot.

**Fix:** Run `git add <filename>` to stage the files you want, then run `git commit` again. Alternatively, if the files are already tracked and you want to skip the staging step, use `git commit -am "message"` — but this only works for modified tracked files, not new untracked files.

### Error 4: `nothing to commit, working tree clean`

```
On branch main
nothing to commit, working tree clean
```

**Cause:** You ran `git commit` but the staging area matches the last commit exactly. There are no new changes to snapshot. This usually means you forgot to save a file, forgot to stage after editing, or you already committed everything.

**Fix:** Check if you actually saved your file changes. Open the file and verify. Then run `git status` to confirm the file shows as modified. If it does not, you probably edited but did not save. If it shows as modified, run `git add` to stage it, then commit.

### Error 5: `fatal: bad revision`

```
fatal: bad revision 'HEAD~5'
```

**Cause:** You tried to reference a commit that does not exist. `HEAD~5` means "5 commits before HEAD," but your repository only has 3 commits. Git cannot walk back further than the root commit.

**Fix:** Check your commit count with `git log --oneline`. Use a valid revision number. `HEAD~2` would work with 3 commits; `HEAD~0` or just `HEAD` is always valid as long as there is at least one commit.

## What You Just Learned

- `git init` creates a hidden `.git` directory that turns any folder into a Git repository, enabling version tracking for every file inside it
- `git status` shows the current state of your working directory, staging area, and repository in a single view — run it constantly while learning
- `git add` moves file changes from the working directory into the staging area, giving you control over what goes into the next commit
- `git commit` creates a permanent snapshot of the staging area, storing your name, email, timestamp, and message as metadata that lives in history forever
- `git log` displays the commit history, and with `--oneline --graph` it shows a compact topological view of how commits relate through parent links
- The three-tree model (working directory, staging area, repository) is the core mental model for every Git operation you will learn next
