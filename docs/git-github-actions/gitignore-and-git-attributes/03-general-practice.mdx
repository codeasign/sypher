## Tier 1 — Beginner

### Exercise 1: Ignore a Single File

**Task:** Create a `.gitignore` file in a fresh Git repository that ignores a file named `config.local.yaml` but does not affect any other files.

<details>
<summary>Hint</summary>

A `.gitignore` pattern is just the filename when you want to match that name in any directory. No wildcards needed for a single file.
</details>

<details>
<summary>Solution</summary>

```bash
git init
echo "config.local.yaml" > .gitignore
```

To verify:

```bash
echo "key: value" > config.local.yaml
git status
# config.local.yaml should NOT appear in untracked files
```

The pattern `config.local.yaml` matches that exact filename in any directory. If you placed it in a subdirectory — say `config/local.yaml` — this pattern would also match it. For a root-only match, use `/config.local.yaml` with a leading slash.
</details>

### Exercise 2: Ignore an Entire Directory and Its Contents

**Task:** Create a `.gitignore` that ignores all files inside a `temp/` directory, no matter where `temp/` appears in the project tree.

<details>
<summary>Hint</summary>

A trailing slash tells Git you mean a directory. The `**` pattern lets you match nested directories.
</details>

<details>
<summary>Solution</summary>

```bash
echo "temp/" > .gitignore
```

To verify:

```bash
mkdir -p temp/subdir
echo "cache" > temp/cache.dat
echo "data" > temp/subdir/data.dat
git status
# temp/ should not appear in untracked files
```

The pattern `temp/` matches a directory named `temp` at any level. All files inside `temp/` are implicitly ignored because Git does not recurse into ignored directories. If you had `src/temp/`, it would also be ignored. To match only the root `temp/`, use `/temp/`.
</details>

### Exercise 3: Use a Wildcard Pattern to Ignore File Types

**Task:** Create a `.gitignore` that ignores all `.log` files but does NOT ignore `.log` files inside a `release/` directory.

<details>
<summary>Hint</summary>

Combine a wildcard pattern with a negation pattern. The negation must specify the path structure.
</details>

<details>
<summary>Solution</summary>

```bash
cat > .gitignore << 'EOF'
*.log
!release/*.log
EOF
```

To verify:

```bash
echo "error" > server.log
mkdir -p release
echo "changelog" > release/notes.log
git status
# server.log should be ignored
# release/notes.log should appear as untracked
```

The `*.log` pattern matches every `.log` file. The `!release/*.log` pattern re-includes only `.log` files directly inside `release/`. Note that `release/sub/notes.log` would still be ignored — the `*` does not match across directory boundaries. Use `!release/**/*.log` to allow all `.log` files anywhere under `release/`.
</details>

### Exercise 4: Verify Why a File Is Ignored

**Task:** Given a repository with `*.env` in `.gitignore`, use a Git command to confirm that `.env.production` is ignored and show which pattern caused it.

<details>
<summary>Hint</summary>

The `git check-ignore` command has a verbose flag that shows the source of the match.
</details>

<details>
<summary>Solution</summary>

```bash
echo "*.env" > .gitignore
echo "TOKEN=abc" > .env.production
git check-ignore -v .env.production
```

Expected output:

```
.gitignore:1:*.env  .env.production
```

The `*.env` pattern matches any filename ending in `.env`, including `.env.production`. This is because `*` in a glob pattern matches any characters except a forward slash. If you want to match only the exact `.env` file, use `.env` (without a wildcard) or `/.env` (anchored to root).
</details>

## Tier 2 — Intermediate

### Exercise 1: Set Up Cross-Platform Line Ending Rules

**Task:** You are the lead on a new project with 12 developers — 8 on Windows, 4 on macOS. Create a `.gitattributes` file that:

1. Normalizes all text files to LF in the repository
2. Forces shell scripts (`.sh`) to use LF on checkout
3. Forces Windows batch files (`.bat`, `.cmd`) to use CRLF on checkout
4. Marks images (`.png`, `.jpg`, `.gif`) as binary
5. Sets Python diffs to show function headers

<details>
<summary>Hint</summary>

Each rule in `.gitattributes` is a pattern followed by attribute assignments. The `text` attribute controls normalization, `eol` controls the checkout line ending, `binary` is a shorthand, and `diff` sets the diff strategy.
</details>

<details>
<summary>Solution</summary>

```bash
cat > .gitattributes << 'EOF'
# Normalize all text files
* text=auto

# Shell scripts — LF only
*.sh text eol=lf

# Windows batch files — CRLF on checkout
*.bat text eol=crlf
*.cmd text eol=crlf

# Binary files — no modifications
*.png binary
*.jpg binary
*.gif binary

# Language-aware diffs
*.py diff=python
*.js diff=javascript
*.java diff=java
EOF
```

To verify the rules are applied:

```bash
# Check what attributes apply to a specific file
git check-attr -a -- main.py
# Output:
# main.py: text: auto
# main.py: diff: python
```

The `git check-attr` command is the counterpart to `git check-ignore` — it tells you which attribute rules match a given path. Use it whenever you are unsure what `.gitattributes` rules apply to a file.
</details>

### Exercise 2: Migrate a Project That Already Tracked Secrets

**Task:** A colleague committed `.env` with database credentials before adding it to `.gitignore`. Five commits later, the file is still tracked. You need to:

1. Remove it from tracking without deleting it from disk
2. Add it to `.gitignore` so it stays ignored
3. Verify the file still exists on disk
4. Confirm future changes to `.env` are not tracked

<details>
<summary>Hint</summary>

`git rm --cached` removes a file from the index while keeping the working copy. The `--cached` flag is the critical difference from `git rm` without it.
</details>

<details>
<summary>Solution</summary>

```bash
# Step 1: Remove from index only
git rm --cached .env

# Step 2: Verify file still exists on disk
ls -la .env
# Should show the file

# Step 3: Add to .gitignore if not already there
echo ".env" >> .gitignore

# Step 4: Commit the change
git add .gitignore
git commit -m "stop tracking .env and add to .gitignore"

# Step 5: Verify future changes are ignored
echo "NEW_SECRET=value" >> .env
git status
# .env should NOT appear in changes
```

The `git rm --cached` command writes a new tree entry to the index where `.env` is absent. The next commit will not include it. But the blob for `.env` from previous commits remains in the object database — anyone who clones your repository at an older commit will still see the secret. To fully purge a file from history, you need `git filter-branch` or `git filter-repo`, which rewrites the entire commit DAG. This is a much more invasive operation and should be done with clear team communication.
</details>

### Exercise 3: Set Up a Global Gitignore for Your Machine

**Task:** Configure your global Git installation to ignore:

- Vim swap files (`*.swp`, `*.swo`)
- macOS `.DS_Store` files
- Windows `Thumbs.db` files
- JetBrains IDE `.idea/` directories

The configuration should affect every repository you work on without requiring a `.gitignore` in each one.

<details>
<summary>Hint</summary>

Set `core.excludesFile` to a path, then create that file with the patterns. The file path can be anywhere on your system.
</details>

<details>
<summary>Solution</summary>

```bash
# Step 1: Configure the path
git config --global core.excludesFile ~/.config/git/ignore

# Step 2: Create the directory if it does not exist
mkdir -p ~/.config/git

# Step 3: Write the patterns
cat > ~/.config/git/ignore << 'EOF'
# Vim swap files
*.swp
*.swo
*.swn

# macOS metadata
.DS_Store

# Windows thumbnail cache
Thumbs.db

# JetBrains IDE
.idea/
EOF
```

To verify:

```bash
# Create a test repository
cd /tmp
mkdir -p test-global
cd test-global
git init

# Create a file that should be ignored globally
touch .DS_Store
git status
# .DS_Store should NOT appear
```

The global gitignore is the lowest-priority ignore source. It is checked after the project `.gitignore` and `.git/info/exclude`. If a project `.gitignore` explicitly includes `.DS_Store` with `!`, the global ignore cannot override it — the project-level rule wins.
</details>

### Exercise 4: Use .git/info/exclude for Local-Only Rules

**Task:** You are working on a team project. You have a personal `TODO.md` file in the project root that you do not want to commit, but adding it to `.gitignore` would affect everyone. Use the local-only exclude mechanism instead.

<details>
<summary>Hint</summary>

Every repository has a `.git/info/exclude` file that uses the same syntax as `.gitignore` but is never committed or pushed.
</details>

<details>
<summary>Solution</summary>

```bash
# Step 1: Create your personal file
echo "- [ ] review PR #42" > TODO.md

# Step 2: Add it to the local exclude file
echo "TODO.md" >> .git/info/exclude

# Step 3: Verify it is ignored
git status
# TODO.md should NOT appear

# Step 4: Confirm .gitignore was not modified
git diff .gitignore
# No output — .gitignore is unchanged
```

The `.git/info/exclude` file is part of the `.git` directory, which is never pushed to remotes. It is the right place for machine-specific paths, personal notes, and temporary experiments. The syntax is identical to `.gitignore` — all patterns, wildcards, and negation rules work the same way.
</details>

## Tier 3 — Advanced

### Exercise 1: The Case of the Vanishing Executable Permission

**Scenario:** Your team maintains a deployment script `deploy.sh`. On your machine, you set it as executable with `chmod +x deploy.sh` and commit. When a teammate clones the repository on Windows, the file exists but the executable bit is lost. When the Windows teammate pushes a change, the executable bit is still missing, and the CI pipeline fails because it cannot run `./deploy.sh`.

Your `.gitattributes` currently contains:

```
* text=auto
*.sh text eol=lf
```

**Problem:** Why is the executable bit being lost? How do you preserve it across all platforms?

**Approach:**

<details>
<summary>Solution Walkthrough</summary>

The executable bit is a file mode attribute stored in Git's index (as `100755` for executable, `100644` for non-executable). Git preserves whatever mode was set when the file was added to the index. There are two common causes for the disappearing executable bit:

1. **Windows does not have Unix-style executable permissions.** When a Windows user adds a file, Git for Windows may set mode `100644` (non-executable) because the file system does not report the executable bit. The file gets committed as non-executable.

2. **No one explicitly set the executable bit in the index.** Even if the file has `+x` on disk, Git only records the mode at `git add` time. If the file was added before `chmod +x`, the mode is `100644`.

**Solution — Three Parts:**

**Part A: Restore the executable bit in the index:**

```bash
git add --chmod=+x deploy.sh
git commit -m "fix: restore executable bit on deploy.sh"
```

The `--chmod=+x` flag tells Git to set the executable bit during `git add`, regardless of the file's actual filesystem permissions. This works on all platforms.

**Part B: Add a `.gitattributes` rule to preserve the mode on future adds:**

```
deploy.sh text eol=lf
```

Note: `.gitattributes` does not have a `mode` attribute. Instead, you rely on the file being added with `--chmod=+x` one time, after which the mode is part of the commit history. Future `git checkout` operations restore the mode from the stored tree entry.

**Part C: Educate the team to use `--chmod=+x` on Windows:**

Add a note to the project README:

```bash
# Windows users: add executable scripts with the --chmod flag
git add --chmod=+x scripts/deploy.sh
```

**Alternative approach — post-checkout hook:**

A `post-checkout` hook in `.git/hooks/post-checkout` can restore executable bits from a manifest file, but this is more complex and fragile. The `--chmod=+x` approach is simpler and more reliable.

**Alternative approach — core.fileMode config:**

Set `git config core.fileMode false` on Windows to tell Git to ignore filesystem permission differences. This prevents Git from reporting "modified" files whose only change is the executable bit, but it does not _fix_ the missing bit — it only hides the symptom. Combined with `--chmod=+x` on the initial add, this is a common team setup.
</details>

### Exercise 2: The Encrypted Config File That Should Not Be Diffed

**Scenario:** Your team uses `git-crypt` to encrypt configuration files. The `.gitattributes` contains:

```
*.secret filter=git-crypt diff=git-crypt
```

When a teammate runs `git diff` on a `.secret` file, the output shows the encrypted blob content — a wall of unreadable base64 text. This is noisy and exposes no useful information.

**Problem:** You want `git diff` to show `Binary files differ` for `.secret` files instead of the raw encrypted content. You also want the file to be encrypted at rest (as it is now) and decrypted on checkout (as it is now). You cannot change the encryption tool.

**Approach:**

<details>
<summary>Solution Walkthrough</summary>

The issue is that `diff=git-crypt` tells Git to use the `git-crypt` diff driver, which tries to decrypt the two sides and diff the plaintext. If the diff driver is not installed or configured, Git falls back to a text diff of the encrypted content.

**Three approaches exist, each with different trade-offs.**

**Approach A — Declare the files as binary (simplest):**

```
*.secret filter=git-crypt binary
```

The `binary` attribute is shorthand for `-text -diff -merge`. This disables text conversions, diffing, and auto-merge. `git diff` will show `Binary files differ`. The `filter=git-crypt` still runs, so encryption and decryption work as before.

The downside: you lose the ability to see meaningful diffs of the decrypted content. If the config file is small and changes infrequently, this is the right trade-off.

**Approach B — Configure a custom diff driver that returns "binary" (most correct):**

This requires running a command that Git can call as the diff driver. The command receives two file paths, and if it returns non-zero exit code or produces no output, Git falls back to `Binary files differ`.

**Approach C — Use `diff` attribute to disable diff explicitly:**

```
*.secret filter=git-crypt diff=
```

Setting `diff` to an empty string (or `-diff`) disables the diff attribute without setting `binary`. The file is still treated as text for line-ending purposes (if `text` is set), but `git diff` will not attempt to show a patch.

**Recommendation:**

Approach A is the best for most teams:

```
*.secret filter=git-crypt binary
```

The `binary` attribute is safe because:
- The files are encrypted — there is no meaningful text to diff
- The `filter=git-crypt` still runs independently of `binary`
- `git merge` will not attempt to auto-merge encrypted files (which would produce garbage)
- Line-ending conversion is irrelevant for encrypted content

If you later need decrypted diffs, you can add a custom diff driver that uses `git-crypt diff` as the command, but that is a more advanced setup that requires every team member to install the driver configuration.
</details>

### Exercise 3: The Monorepo with Mixed Ignore Rules

**Scenario:** Your monorepo has three subprojects with different build tools and languages:

```
project/
  ├── .gitignore                  # top-level rules
  ├── frontend/
  │   ├── .gitignore              # Node.js specific
  │   ├── node_modules/
  │   ├── dist/
  │   └── src/
  ├── backend/
  │   ├── .gitignore              # Python specific
  │   ├── __pycache__/
  │   ├── .venv/
  │   └── src/
  └── docs/
      ├── .gitignore
      └── _build/
```

The top-level `.gitignore` has:

```
*.log
.DS_Store
```

The `frontend/.gitignore` has:

```
node_modules/
dist/
.env
```

The `backend/.gitignore` has:

```
__pycache__/
.venv/
*.pyc
.env
```

The `docs/.gitignore` has:

```
_build/
```

**Problem 1:** A developer creates `frontend/dist/debug.log`. Which `.gitignore` patterns apply? Is it ignored?

**Problem 2:** A developer creates `backend/.env`. The top-level `.gitignore` does not mention `.env`, but `backend/.gitignore` does. Is it ignored?

**Problem 3:** A developer creates `docs/debug.log`. The top-level `.gitignore` has `*.log`, but `docs/.gitignore` does not mention logs. Is it ignored?

**Approach:**

<details>
<summary>Solution Walkthrough</summary>

**Answer to Problem 1 — `frontend/dist/debug.log`:**

Three patterns could match this file:

1. Top-level `.gitignore`: `*.log` — matches by basename, applies to all subdirectories
2. Top-level `.gitignore`: `dist/` — does NOT exist in the top-level file
3. `frontend/.gitignore`: `dist/` — matches the `dist/` directory inside `frontend/`

The `dist/` pattern in `frontend/.gitignore` matches the directory `frontend/dist/`, and all files inside an ignored directory are implicitly ignored. However, `*.log` from the top-level `.gitignore` also matches. Conflicts are resolved by the most specific rule, but since both say "ignore," there is no conflict.

Result: **Ignored.** Both patterns independently cause it to be ignored.

**Answer to Problem 2 — `backend/.env`:**

The top-level `.gitignore` does not mention `.env`. The `backend/.gitignore` does. `.gitignore` files apply to their own directory and all subdirectories. Since `backend/.env` is inside the `backend/` directory, the `backend/.gitignore` pattern `.env` applies.

Result: **Ignored.** The child `.gitignore` adds rules that the parent does not have.

**Answer to Problem 3 — `docs/debug.log`:**

The top-level `.gitignore` pattern `*.log` matches basenames in any directory. The `docs/.gitignore` does not mention logs. Git merges all `.gitignore` files from the root to the file's directory. The more specific file (closest to the path) wins for conflicting patterns, but there is no conflict here — the top-level `*.log` is never overridden by `docs/.gitignore`.

Result: **Ignored.** The top-level `*.log` pattern applies recursively to all subdirectories, including `docs/`.

**Key insight for monorepo management:**

Each `.gitignore` file only overrides patterns from _parent_ directories' `.gitignore` files for files _within that directory_. A child `.gitignore` cannot add patterns that affect sibling directories, and a parent `.gitignore`'s patterns are always in effect unless explicitly negated by a child. This hierarchical merging is the correct behavior for monorepos — the top-level `.gitignore` sets organization-wide policies (logs, OS metadata), and each subproject adds its own tool-specific ignores.

**Recommended top-level monorepo `.gitignore`:**

```
# Organization-wide ignores
*.log
.DS_Store
Thumbs.db
.idea/
.vscode/

# Catch-all for secrets (override in child .gitignore if needed)
.env
*.pem
```

Then each subproject only needs to add its build-tool-specific patterns, not re-declare the shared rules.
</details>

### Exercise 4: The .gitattributes That Changed Everything

**Scenario:** Your team has been working on a project for six months with no `.gitattributes`. Developers use Windows, macOS, and Linux. Every pull request shows hundreds of lines of whitespace-only changes — trailing spaces removed, line endings converted, blank lines at EOF changed. The diffs are impossible to review because real changes are buried in formatting noise.

You decide to add a `.gitattributes` file with `* text=auto` to normalize all text files. But after committing it, every file in the repository shows as modified in `git status`, even files you did not touch.

**Problem:** Why does every file show as modified? How do you apply the `.gitattributes` change without creating a massive "normalize everything" commit?

**Approach:**

<details>
<summary>Solution Walkthrough</summary>

**Root cause:**

When you add `* text=auto` to `.gitattributes`, Git re-evaluates every file in the index. Files that were stored with CRLF line endings (because they were committed from Windows without normalization) are now detected as "wrong" — Git expects LF in the object database, but the existing blobs contain CRLF. Git marks these files as modified because the working copy (after applying the new `.gitattributes` rules) differs from the index.

**The standard fix — a single normalization commit:**

```bash
# Step 1: Add .gitattributes
echo "* text=auto" > .gitattributes

# Step 2: Stage everything, including the re-normalization
git add --renormalize .

# Step 3: Commit
git commit -m "chore: add .gitattributes and normalize line endings"
```

The `--renormalize` flag (available since Git 2.16) tells Git to re-apply all text conversions to every file in the index. This produces a single commit that normalizes line endings across the entire repository.

**The safer approach — .gitattributes first, then normalize:**

Some teams prefer a two-step process to make the normalization commit easy to skip in `git blame`:

```bash
# Step 1: Commit .gitattributes alone
git add .gitattributes
git commit -m "chore: add .gitattributes"

# Step 2: Normalize all files
git add --renormalize .
git commit -m "chore: normalize line endings across repository"
```

**Important — after the normalization commit:**

Every team member must run:

```bash
git pull
git rm --cached -r .
git reset --hard
```

This forces Git to re-checkout every file with the new `.gitattributes` rules applied. Without this step, team members will have dirty working trees with phantom modifications.

**Alternative — use a .gitattributes file with selective normalization:**

If a full re-normalization is too disruptive (e.g., during a release cycle), you can add `.gitattributes` rules that only affect new files:

```
# Only normalize new files — existing files stay as-is
* text=auto
```

The `text=auto` attribute only affects files that are re-added to the index. Existing blobs are not re-normalized until `git add --renormalize .` is run. This lets you add the `.gitattributes` file now and normalize during the next sprint.

**The key lesson:**

Adding `.gitattributes` to an existing project is not a zero-cost change. It triggers a re-evaluation of every file's line endings. Plan for a normalization commit, communicate with the team, and ensure everyone runs the reset commands after pulling. The one-time pain of normalization is far less than the ongoing pain of whitespace-noise diffs.
</details>

## What You Practiced

- **Pattern syntax** — file names, wildcards, directory-only trailing slashes, and negation patterns with `!`
- **Multi-level ignore merging** — how project, local, and global ignore sources combine into a single rule set
- **Attribute-driven file processing** — line endings, binary detection, diff strategies, and filter pipelines
- **Diagnostic commands** — `git check-ignore -v` and `git check-attr -a` for debugging rule application
- **Real-world cleanup** — removing tracked secrets, normalizing line endings across a team, and handling encrypted files without diff noise
