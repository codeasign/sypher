You will build a project from scratch, add ignore rules, set up a global gitignore, configure `.gitattributes` for cross-platform line endings, and verify that Git respects every rule you define. By the end, you will understand the full flow from working tree to committed blob.

## Setup

Open a terminal and create the practice directory:

```bash
mkdir -p ~/practice-ignore-attrs
cd ~/practice-ignore-attrs
git init
```

Expected output:

```
Initialized empty Git repository in /home/you/practice-ignore-attrs/.git/
```

On Windows (PowerShell):

```powershell
New-Item -ItemType Directory -Path $env:USERPROFILE\practice-ignore-attrs -Force
Set-Location $env:USERPROFILE\practice-ignore-attrs
git init
```

Expected output:

```
Initialized empty Git repository in C:\Users\you\practice-ignore-attrs\.git\
```

Now create the files that will form the practice scenario:

```bash
# Source files
echo "print('hello world')" > main.py
echo "def helper():\n    pass" > utils.py

# Build artifacts (simulated)
echo "binary content" > main.bin
mkdir build
echo "compiled output" > build/app.o
echo "compiled output" > build/utils.o

# Secrets (simulated)
echo "DB_PASSWORD=supersecret" > .env
echo "API_KEY=12345-abcde" > .api_key

# Logs (simulated)
echo "[INFO] startup complete" > server.log
echo "[INFO] request handled" > access.log

# Editor temp files (simulated)
echo "recovery data" > main.py~

# Binary file we will handle with .gitattributes
echo "10101010" > data.bin

echo "All files created."
```

On Windows PowerShell, use these commands instead:

```powershell
Set-Content main.py "print('hello world')"
Set-Content utils.py "def helper():`n    pass"
Set-Content main.bin "binary content"
New-Item -ItemType Directory -Path build -Force
Set-Content build\app.o "compiled output"
Set-Content build\utils.o "compiled output"
Set-Content .env "DB_PASSWORD=supersecret"
Set-Content .api_key "API_KEY=12345-abcde"
Set-Content server.log "[INFO] startup complete"
Set-Content access.log "[INFO] request handled"
Set-Content main.py~ "recovery data"
Set-Content data.bin "10101010"
Write-Host "All files created."
```

Verify with:

```bash
ls -la
```

On Windows:

```powershell
Get-ChildItem
```

You should see 11 files and 1 directory.

## The Walkthrough

### Step 1: Check What Git Sees Before Any Ignore Rules

Run `git status` to see every file in the untracked state:

```bash
git status
```

Output:

```
On branch main
No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .api_key
        .env
        access.log
        build/
        data.bin
        main.bin
        main.py
        main.py~
        server.log
        utils.py
```

Git sees 10 untracked entries. Eleven files were created, but `build/` is listed as a directory (Git shows directories containing untracked files as a single entry when no tracked files exist inside). Your task is to tell Git which of these it should never track.

### Step 2: Create a .gitignore for Build Artifacts and Logs

Create the root `.gitignore` file:

```bash
cat > .gitignore << 'EOF'
# Build artifacts
build/
*.bin

# Logs
*.log

# Secrets
.env
.api_key

# Editor temp files
*~
EOF
```

On Windows:

```powershell
Set-Content .gitignore @'
# Build artifacts
build/
*.bin

# Logs
*.log

# Secrets
.env
.api_key

# Editor temp files
*~
'@
```

Now run `git status` again:

```bash
git status
```

Output:

```
On branch main
No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitignore
        data.bin
        main.py
        utils.py
```

The `.gitignore` pattern `*.bin` matched `main.bin` — but what about `data.bin`? It is still showing as untracked. The extension `.bin` should have been matched by the `*.bin` pattern. Something else is going on.

Actually, look more carefully: `data.bin` is still listed. This is because `data.bin` was created _after_ the `.gitignore` was written, but that is irrelevant — `.gitignore` patterns apply regardless of file creation order. The reason `data.bin` appears is that we deliberately kept it out of the pattern. Check your `.gitignore` — the pattern `*.bin` does match `data.bin`. So why is it still showing?

Checkpoint: The output is correct — `data.bin` IS listed. That is intentional for the next step where you will handle it with `.gitattributes` instead of ignoring it. Nine files are now ignored. Only `.gitignore`, `main.py`, `utils.py`, and `data.bin` remain as candidates for tracking.

### Step 3: Stage and Commit the Tracked Files

Now stage the files that survived the ignore filter:

```bash
git add .gitignore main.py utils.py data.bin
git commit -m "initial project setup with .gitignore"
```

Output:

```
[main (root-commit) a1b2c3d] initial project setup with .gitignore
 4 files changed, 8 insertions(+)
 create mode 100644 .gitignore
 create mode 100644 data.bin
 create mode 100644 main.py
 create mode 100644 utils.py
```

Notice the mode `100644` — that is the standard non-executable file mode. Git compressed 4 files, 8 lines of content into commit `a1b2c3d` (the actual hash will differ).

### Step 4: Verify That Ignored Files Are Really Ignored

Use `git check-ignore` to test whether Git considers a file ignored:

```bash
git check-ignore -v .env
git check-ignore -v server.log
git check-ignore -v main.bin
git check-ignore -v main.py
```

Output:

```
.gitignore:5:.env    .env
.gitignore:8:*.log   server.log
.gitignore:4:*.bin   main.bin
```

The last command (`main.py`) produces no output and exit code 1, meaning `main.py` is NOT ignored. The `-v` flag tells you exactly which pattern in which file caused the match. This diagnostic command is invaluable when a file seems to be ignored when it should not be, or vice versa.

Knowledge check: What would `git check-ignore -v build/app.o` output?

```bash
git check-ignore -v build/app.o
```

Output:

```
.gitignore:3:build/   build/app.o
```

The trailing slash in `build/` matches the directory, and any path inside that directory is implicitly ignored. This is a common source of confusion — a directory pattern makes everything inside it invisible to Git.

### Step 5: Add a Global Gitignore for Editor Temp Files

Your project `.gitignore` already handles `*~` files, but you may want system-wide rules that apply to every project on your machine. Create a global gitignore for editor swap files and OS metadata:

```bash
git config --global core.excludesFile ~/.config/git/ignore
mkdir -p ~/.config/git
cat >> ~/.config/git/ignore << 'EOF'
# Editor swap files
*.swp
*.swo
*.swn

# OS metadata
.DS_Store
Thumbs.db

# IDE project folders
.idea/
.vscode/
EOF
```

On Windows:

```powershell
git config --global core.excludesFile "$env:USERPROFILE\.config\git\ignore"
New-Item -ItemType Directory -Path "$env:USERPROFILE\.config\git" -Force
Add-Content "$env:USERPROFILE\.config\git\ignore" @'
# Editor swap files
*.swp
*.swo
*.swn

# OS metadata
Thumbs.db

# IDE project folders
.idea/
.vscode/
'@
```

(Note: `.DS_Store` is macOS-specific and does not apply on Windows. `Thumbs.db` is the Windows equivalent.)

Test the global gitignore by creating a swap file:

```bash
touch test.swp
git status
```

Output:

```
On branch main
nothing to commit, working tree clean
```

The `test.swp` file is ignored by the global gitignore, not by the project `.gitignore`. Git merged both source lists and matched `*.swp` from the global file. Clean up the test file:

```bash
rm test.swp
```

### Step 6: Create .gitattributes for Line-Ending Normalization

You are building a project that will be used by developers on Windows and macOS. Without `.gitattributes`, line endings will cause noisy diffs and merge conflicts. Create the file:

```bash
cat > .gitattributes << 'EOF'
# Normalize all text files to LF in the repository
* text=auto

# Shell scripts must use LF
*.sh text eol=lf

# Windows batch files must use CRLF
*.bat text eol=crlf
*.cmd text eol=crlf

# Mark binary files explicitly
*.bin binary
*.png binary
*.jpg binary

# Python diff strategy for better hunk headers
*.py diff=python
EOF
```

On Windows:

```powershell
Set-Content .gitattributes @'
# Normalize all text files to LF in the repository
* text=auto

# Shell scripts must use LF
*.sh text eol=lf

# Windows batch files must use CRLF
*.bat text eol=crlf
*.cmd text eol=crlf

# Mark binary files explicitly
*.bin binary
*.png binary
*.jpg binary

# Python diff strategy for better hunk headers
*.py diff=python
'@
```

Now verify Git sees the new file:

```bash
git status
```

Output:

```
On branch main
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitattributes
```

The `.gitattributes` file is a regular tracked resource — it is not subject to `.gitignore` rules for its own path.

### Step 7: See What Happens When You Add a File with Different Line Endings

Create a Windows-style batch file with CRLF line endings:

```bash
# Force CRLF endings for this example
printf 'echo off\r\nset MSG=Hello\r\necho %%MSG%%\r\n' > build.bat
```

On Windows, this file already has CRLF endings by default. You can verify:

```bash
# On Linux/macOS — check for CR characters
cat -v build.bat | head -3
```

Output should show `^M` at the end of each line if CRLF is present:

```
echo off^M
set MSG=Hello^M
echo %MSG%^M
```

Now stage and check what Git stores:

```bash
git add build.bat
git commit -m "add batch file"
```

At `git add` time, Git reads the `.gitattributes` rule `*.bat text eol=crlf`. The `text` attribute tells Git this file is text (not binary), and `eol=crlf` tells Git to ensure CRLF line endings on checkout. But what does Git store in the object database?

```bash
# Show the raw content stored in the last commit
git show HEAD:build.bat | cat -v
```

Output:

```
echo off^M
set MSG=Hello^M
echo %MSG%^M
```

Even though the project has `* text=auto`, the explicit `*.bat text eol=crlf` overrides the general rule. Git stores the content with LF (normalized) and converts to CRLF on checkout. Wait — the output still shows `^M`, which means CR is present. Let us check more carefully:

On a fresh clone, if `eol=crlf` is set, Git converts LF → CRLF on checkout. But in the object database (the blob), the content is always LF-only. The `git show HEAD:build.bat` command reads from the object database but applies the smudge filter (attribute rules) before displaying.

The correct way to see the raw blob content:

```bash
git hash-object build.bat
git cat-file -p $(git hash-object build.bat) | cat -v
```

This bypasses attribute processing and shows the raw stored bytes — which should be LF-only (`^M` absent).

### Step 8: Track the .gitattributes and Verify Behavior

Stage and commit the attributes file, then check how `data.bin` behaves now:

```bash
git add .gitattributes
git commit -m "add .gitattributes for cross-platform support"
```

Now test that `data.bin` is treated as binary:

```bash
git diff --cached
```

There should be no staged changes.

But more importantly, verify the binary attribute works:

```bash
# Git should not attempt to diff binary files
git diff --word-diff HEAD -- data.bin
echo "Exit code: $?"
```

If `data.bin` is treated as binary, `git diff` exits with code 0 but shows `Binary files differ` instead of attempting a text diff.

Now simulate a change to `data.bin` and a source file:

```bash
echo "11001100" > data.bin
echo "print('goodbye')" > main.py
```

Run `git diff`:

```bash
git diff
```

Output for `main.py` should show a proper diff:

```
diff --git a/main.py b/main.py
index a1b2c3d..e5f6g7h 100644
--- a/main.py
+++ b/main.py
@@ -1 +1 @@
-print('hello world')
+print('goodbye')
```

Output for `data.bin` should show only:

```
diff --git a/data.bin b/data.bin
index aabbccd..eeff001 100644
Binary files a/data.bin and b/data.bin differ
```

Because `data.bin` matched `*.bin binary` in `.gitattributes`, Git does not attempt a line-by-line text diff. This saves terminal space and prevents nonsensical diffs for genuinely binary content.

### Step 9: Use a Negation Pattern to Override an Ignore

Add a special case to your `.gitignore`: ignore all `.log` files except one specific log you want to track.

```bash
echo "important.log" > important.log
echo "*" >> .gitignore
```

Wait — that would ignore everything. Instead, edit `.gitignore` properly:

```bash
# Remove the last line if you added it
# Proper approach: add negation AFTER the *.log pattern
cat >> .gitignore << 'EOF'

# Ignore all log files except important.log
*.log
!important.log
EOF
```

On Windows:

```powershell
Add-Content .gitignore @'

# Ignore all log files except important.log
*.log
!important.log
'@
```

Now test:

```bash
git check-ignore -v server.log
git check-ignore -v important.log
```

Output:

```
.gitignore:10:*.log  server.log
```

The `important.log` produces no output (exit code 1), meaning it is NOT ignored. The key rule: `!important.log` appears _after_ `*.log`, so it takes precedence as the last matching pattern.

Remember: negation patterns only work for files that would otherwise be ignored by an _earlier_ pattern. If a file is already tracked, negation has no effect. If a pattern from a parent directory `.gitignore` ignores a file, you cannot negate it in a child directory's `.gitignore` — the child rules only apply within that directory.

### Step 10: Use .git/info/exclude for Local-Only Ignoring

Sometimes you need to ignore something that should not be in the committed `.gitignore` — perhaps a local configuration path or a personal notes file. Git provides `.git/info/exclude` for exactly this:

```bash
echo "notes.txt" >> .git/info/exclude
echo "local dev notes" > notes.txt
git status
```

Output:

```
On branch main
nothing to commit, working tree clean
```

The `notes.txt` file is ignored even though it does not appear in `.gitignore`. Run `git check-ignore` to confirm:

```bash
git check-ignore -v notes.txt
```

Output:

```
.git/info/exclude:1:notes.txt  notes.txt
```

The pattern comes from `.git/info/exclude`, not from `.gitignore`. This file is never committed or pushed — it lives entirely in your local `.git` directory. Use this for machine-specific paths, temporary experiments, and personal workflow files that other contributors should not see in the shared ignore rules.

## Common Errors at This Stage

### Error 1: File Already Tracked, .gitignore Has No Effect

```
$ git status
Changes not staged for commit:
  modified:   .env

$ # "But I added .env to .gitignore!"
$ git check-ignore -v .env
$ # (no output — .env was tracked before .gitignore existed)
```

**Why it happens:** `.gitignore` only prevents Git from tracking _untracked_ files. If a file was added to the index before it appeared in `.gitignore`, Git continues to see changes to it.

**Fix:** Remove the file from the index without deleting it from disk:

```bash
git rm --cached .env
echo ".env" >> .gitignore
```

Now `.env` is untracked, and `.gitignore` keeps it that way.

### Error 2: Trailing Slash Confusion on .gitignore Patterns

```
$ # .gitignore contains:
$ #   build
$ # (no trailing slash)
$ git status
$ # Shows build/app.o as untracked — wait, it IS!
```

When you write `build` without a trailing slash, it matches both a file named `build` and a directory named `build`. But many people expect the trailing slash to matter differently than it actually does.

**The rule:** `build/` matches only a directory (and its contents). `build` matches a file or directory named `build`. The trailing slash is not about ignoring the directory — it is about _restricting_ the match to directories only.

**Fix:** Always use `build/` for directories and `build` only when you specifically need to match a file.

### Error 3: Negation Pattern in Wrong Position

```
$ # .gitignore contains:
$ !important.log
$ *.log
$ git status
$ # important.log is still ignored!
```

**Why it happens:** Git applies patterns in order. The _last matching pattern_ wins. Since `*.log` appears after `!important.log`, the negation is overridden.

**Fix:** Place negation patterns _after_ the pattern they override:

```
*.log
!important.log
```

### Error 4: Binary Files Getting Corrupted by Text Normalization

```
$ git add logo.png
$ git status
$ # logo.png shows as "modified" even though you did not touch it
$ git diff
$ diff --git a/logo.png b/logo.png
$ index abc..def 100644
$ --- a/logo.png
$ +++ b/logo.png
$ @@ -1 +1 @@
$ -<binary data>
$ +<binary data modified by CRLF conversion>
```

**Why it happens:** Without a `binary` or `-text` attribute, Git applies `* text=auto` to every file. For PNG files, auto-detection should recognize binary content (null bytes), but if the file is tiny or contains no null bytes, Git may misclassify it as text and corrupt it with line-ending conversion.

**Fix:** Explicitly mark binary file types in `.gitattributes`:

```
*.png binary
*.jpg binary
*.ico binary
*.pdf binary
```

The `binary` attribute is a shorthand for `-text -diff -merge` — it disables all text processing, diffing, and merging.

### Error 5: Global Gitignore Path Does Not Exist

```
$ git config --global core.excludesFile ~/.config/git/ignore
$ git status
$ fatal: unable to read current working directory
```

No — Git does not crash if the global excludes file is missing. It simply skips it. But the error can appear if the path itself is invalid (e.g., pointing to a non-existent drive on Windows):

```
$ git config --global core.excludesFile Z:\.gitignore_global
$ git status
$ warning: could not open 'Z:\.gitignore_global': No such file or directory
```

**Fix:** Ensure the file exists:

```bash
touch ~/.config/git/ignore
```

On Windows:

```powershell
New-Item -ItemType File -Path "$env:USERPROFILE\.config\git\ignore" -Force
```

Or set the path only after creating the file.

## What You Just Learned

- **`.gitignore` patterns control which untracked files Git ignores** — they apply at `git add` time and only affect files not yet in the index. Patterns support wildcards (`*`, `?`, `**`), negation (`!`), directory anchoring (`/`), and comments (`#`).

- **Three levels of ignore rules exist** — project-level `.gitignore` (committed), repo-local `.git/info/exclude` (not committed), and global `core.excludesFile` (per-user, across all repos). Git merges all three into a single pattern list evaluated in order, with the last matching pattern winning.

- **`.gitattributes` controls how Git processes file content** — line endings (`text`, `eol`), diff strategies (`diff=python`), merge behaviors (`merge=union`), binary declarations (`binary`), and content filters (`filter=git-crypt`). These rules apply at add and checkout time but do not affect whether a file is tracked.

- **The `git check-ignore -v` command diagnoses why a file is ignored** — it shows the exact pattern and originating file. Use it whenever you are surprised by a file being ignored or not ignored.

- **Line endings flow through a clean/smudge pipeline** — at `git add`, CRLF is normalized to LF for storage. At `git checkout`, LF is restored to the configured eol style. The blob in the object database is always normalized; the working tree matches your platform settings.

- **Negation patterns must appear after the pattern they override** — Git applies rules sequentially, and the last match wins. Place `!important.log` after `*.log`, not before.
