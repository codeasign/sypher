You committed a credentials file containing your database password two weeks ago. Five other people have pulled that commit since then. Even after you delete the file and update `.gitignore`, the secret lives on in every teammate's local history, every CI clone, and every backup — because `.gitignore` cannot erase what Git has already started tracking. Three hours of rebasing and filter-branch later, you learn the hard way that ignoring must happen _before_ the first `git add`.

Ignoring and attribute rules are the gatekeepers Git consults before it decides what to track and how to treat each file. A `.gitignore` pattern tells Git "do not look at this path," while a `.gitattributes` rule tells Git "when you do look at this file, handle it this way" — for line endings, diff strategies, merge drivers, and identity expansion.

Think of `.gitignore` as a bouncer at a club entrance who prevents unwanted guests from ever walking in, and `.gitattributes` as a dress-code sheet handed to each admitted guest that specifies how they should behave on the dance floor.

## Command and Syntax Reference

| What It Does | Exact File / Pattern | Example | Notes |
|---|---|---|---|
| Ignore a single file | `filename.ext` | `secrets.env` | Matches `secrets.env` in any directory |
| Ignore a directory | `dirname/` | `node_modules/` | Trailing slash matches only directories |
| Wildcard any characters | `*` | `*.log` | Matches `.log` files at any depth |
| Wildcard single directory depth | `?` | `backup?.zip` | Matches `backup1.zip` but not `backup10.zip` |
| Wildcard any depth | `**` | `**/temp/` | Matches `a/temp/`, `a/b/temp/`, etc. |
| Negate a pattern | `!pattern` | `!important.env` | Re-include a file that would otherwise be ignored |
| Ignore all except | `*` + `!` combo | `*` then `!*.c` | Ignore everything except `.c` files |
| Comment | `#` at line start | `# build artifacts` | Line is ignored |
| Global gitignore | `git config core.excludesFile` | `~/.config/git/ignore` | Applies to every repo on your machine |
| Repository gitignore | `.git/info/exclude` | `.git/info/exclude` | Per-repo, not committed; same syntax as `.gitignore` |
| Set line endings | `* text=auto` in `.gitattributes` | `* text=auto` | Git auto-detects and normalizes line endings |
| Force CRLF | `text eol=crlf` | `*.bat text eol=crlf` | Windows batch files always get CRLF |
| Force LF | `text eol=lf` | `*.sh text eol=lf` | Shell scripts always get LF |
| Declare binary | `binary` | `*.png binary` | Git skips diff and line-ending conversion |
| Diff strategy | `diff=language` | `*.py diff=python` | Uses Python-aware hunk headers for diffs |
| Filter / smudge-clean | `filter=name` | `*.secret filter=git-crypt` | Runs external filter on checkout/commit |
| Keyword expansion | `ident` | `*.txt ident` | Expands `$Id$` to blob SHA-1 (rarely used) |

## How It Actually Works — Step by Step

### Step 1: Pattern Compilation on Repository Open

When you run `git status`, `git add`, or any command that scans the working tree, Git reads all active ignore files and compiles them into a single pattern list in memory. The search order is:

1. `.gitignore` in the current directory (and parent directories, walking up)
2. `.git/info/exclude` (per-repo, not committed)
3. Global gitignore from `core.excludesFile` (per-user, across all repos)

Git merges all patterns into one ordered list. Later patterns override earlier ones, and the _most specific_ match wins. A `.gitignore` in a child directory takes precedence over one in a parent directory for files under that child.

### Step 2: Path Matching Against Ignore Patterns

Every path Git encounters — whether from `git status`, `git add <path>`, or `git clean` — is checked against the compiled pattern list. Git walks the pattern list from first to last, applying each one. The _last matching pattern_ wins, which is why `!` negation patterns must appear _after_ the pattern they override.

Git matches patterns differently based on their structure:

- A pattern with a trailing slash (e.g., `build/`) matches only directories
- A pattern without a slash in the body matches basenames in any directory
- A pattern with a leading slash (e.g., `/build`) anchors to the root of the `.gitignore` file's directory
- `**` matches zero or more directories
- `*` matches anything except a slash

For example, given a repository with `*.log` in the root `.gitignore`:

```
debug.log                    → ignored (matches *.log)
src/app.log                  → ignored (matches *.log)
src/logs/access.log          → ignored (matches *.log)
```

But then if `src/.gitignore` contains `!access.log`, the result changes:

```
src/app.log                  → ignored (still matches *.log)
src/logs/access.log          → ignored (matches *.log from root)
src/access.log               → NOT ignored (!access.log in src/ overrides)
```

Yes, that last line is correct — the `!access.log` in `src/.gitignore` only applies to files _in or below_ `src/`, and Git's most-specific-wins rule means only `src/access.log` benefits from the negation.

### Step 3: File Path Checked Into the Index — Attributes Applied

Once a path passes the ignore check, Git consults `.gitattributes` to decide how to process the file content before storing it in the index. This step is silent — you do not see it unless `git add --verbose` is used — but Git runs through every matching attribute rule and applies transformations.

### Step 4: Clean Filter Runs Before Storing in the Index

The "clean" side of a filter transforms file content from your working copy into what Git stores. This runs at `git add` time. The "smudge" side reverses the transformation on `git checkout`, so your working copy has the original content. This is how tools like `git-crypt` and `git-lfs` work: they intercept the content at the boundary.

```
.gitattributes entry:    *.secret filter=git-crypt
.git/config section:     [filter "git-crypt"]
                            clean = git-crypt clean
                            smudge = git-crypt smudge
```

At `git add secret.env`, Git pipes the working-file content through `git-crypt clean` and stores the encrypted result. At `git checkout secret.env`, Git pipes the stored blob through `git-crypt smudge` and writes the decrypted plaintext to your working tree.

### Step 5: Stored in the Object Database with Attribute Metadata

The processed content becomes a blob object. Git does not store the attribute rules inside the blob — they are applied at read/write time, not baked into history. This means:

- Changing `.gitattributes` changes how existing files are read or written without requiring a new commit of those files
- Two clones of the same repo can have different `.gitattributes` rules and see different working-tree content for the same commit
- The blob in the object database always contains the clean (post-filter, normalized) content

### Step 6: Diff and Merge Drivers Change How Git Compares Content

The final attribute category affects how Git displays diffs and resolves merges. Without attribute rules, Git diffs files as byte sequences. With `diff` attributes, Git can produce language-aware diffs (hunk headers show function names) and even run external diff tools.

The `merge` attribute controls conflict resolution:
- `merge=union` — both sides of a conflict are included
- `merge=ours` — always keep the current branch's version
- `merge=binary` — never attempt auto-merge for binary files

For example, a `.gitattributes` for a team with differing operating systems might look like:

```
# Normalize all text files to LF in the repository
* text=auto

# Windows-specific files must stay CRLF on checkout
*.bat text eol=crlf
*.cmd text eol=crlf

# Shell scripts must stay LF
*.sh text eol=lf

# Binary files — no modifications
*.png binary
*.jpg binary
*.ico binary

# Language-specific diff strategies
*.py diff=python
*.js diff=javascript
*.java diff=java

# Lock files — use union merge (accept both sides)
package-lock.json merge=union
```

Without these rules, a team with Windows and macOS developers would create constant noise in every diff as Git auto-converts line endings differently on each platform.

## When to Use This / When Not To

**Add to .gitignore when:**
- Files are auto-generated (`node_modules/`, `build/`, `*.pyc`, `__pycache__/`)
- Files contain secrets or credentials (`.env`, `*.pem`, `config/keys/`)
- Files are personal or editor-specific (`*.swp`, `.vscode/`, `.idea/`)
- Files are large artifacts that should not be in the repo (`*.log`, `*.tar.gz`, `data/raw/`)

**Do not rely on .gitignore when:**
- A file is already tracked — `.gitignore` has no effect. Use `git rm --cached <file>` first
- You need to prevent a force push of a secret — `.gitignore` is advisory, not a security control
- You need different ignore patterns per branch — `.gitignore` is per-commit, not per-branch

**Add to .gitattributes when:**
- Your team works across Windows, macOS, and Linux — `* text=auto` prevents line-ending chaos
- You use Git LFS or git-crypt — the filter attribute wires the tool into add/checkout
- You want meaningful diffs for a specific language — `diff=python` shows function headers
- You need custom merge behavior for generated or lock files — `merge=union` avoids recurring conflicts

**Do not use .gitattributes when:**
- You need line-ending control for only a few files — a global `.gitconfig` and `core.autocrlf` may suffice
- You want to prevent a file from being tracked — use `.gitignore`, not `.gitattributes`
- You want to rename or move files — `.gitattributes` does not affect path operations
