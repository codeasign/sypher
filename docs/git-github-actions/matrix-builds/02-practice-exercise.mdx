You will create a matrix build that tests a library across multiple Node.js versions and operating systems. By the end of this exercise, you will have a CI pipeline that runs 6 parallel jobs — 3 Node versions × 2 OS targets — and reports the status of each.

## Setup

Create a new repository on GitHub called `matrix-builds-practice` and clone it:

```bash
git clone https://github.com/YOUR-USERNAME/matrix-builds-practice.git
cd matrix-builds-practice
```

Set up a simple project that behaves differently on different Node versions:

```bash
npm init -y
mkdir -p .github/workflows
```

Create `index.js`:

```javascript
function getFeatures() {
  const features = {
    nodeVersion: process.version,
    supportsFlatMap: typeof Array.prototype.flatMap === 'function',
    supportsAt: typeof Array.prototype.at === 'function',
  };
  return features;
}

module.exports = { getFeatures };
```

Create `index.test.js`:

```javascript
const { getFeatures } = require('./index');

test('returns Node version string', () => {
  const features = getFeatures();
  expect(features.nodeVersion).toMatch(/^v\d+/);
});

test('flatMap is available', () => {
  const features = getFeatures();
  expect(features.supportsFlatMap).toBe(true);
});

test('Array.at is available', () => {
  const features = getFeatures();
  expect(features.supportsAt).toBe(true);
});
```

Install the test runner:

```bash
npm install --save-dev jest
```

Update `package.json` to add a test script:

```json
"scripts": {
  "test": "jest"
}
```

Verify tests pass locally:

```bash
npm test
```

Expected output:

```
PASS  ./index.test.js
  ✓ returns Node version string
  ✓ flatMap is available
  ✓ Array.at is available
```

## The Walkthrough

### Step 1: Create a Basic Matrix Workflow

Create `.github/workflows/ci.yml`:

```yaml
name: CI Matrix

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [18, 20, 22]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

This workflow defines a matrix with one axis: `node: [18, 20, 22]`. Three jobs run in parallel, each on `ubuntu-latest` with a different Node version.

**Checkpoint:** Verify the file structure:

```bash
ls .github/workflows/
```

Output:
```
ci.yml
```

### Step 3: Add an OS Axis to the Matrix

Update the workflow to add an OS axis:

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

Now the matrix produces 2 × 3 = 6 jobs. Each job runs on a different OS-Node combination.

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add OS axis to matrix: ubuntu and windows"
git push origin main
```

### Step 4: Add an Exclude Rule

On Windows, Node 18 is a common combination that works fine, but you may want to exclude Windows + Node 22 for a specific use case. Add an `exclude`:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20, 22]
    exclude:
      - os: windows-latest
        node: 22
```

Now only 5 jobs run instead of 6:

```
CI Matrix / test (ubuntu-latest, 18) — ✅
CI Matrix / test (ubuntu-latest, 20) — ✅
CI Matrix / test (ubuntu-latest, 22) — ✅
CI Matrix / test (windows-latest, 18) — ✅
CI Matrix / test (windows-latest, 20) — ✅
(Windows + Node 22 is excluded)
```

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Exclude Windows + Node 22 from matrix"
git push origin main
```

**Checkpoint:** Verify only 5 jobs appear in the Actions tab.

### Step 5: Add an Include with Experimental Flag

Add an `include` entry to add an experimental configuration that is allowed to fail:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20, 22]
    exclude:
      - os: windows-latest
        node: 22
    include:
      - os: macos-latest
        node: 22
        experimental: true

  fail-fast: false

continue-on-error: ${{ matrix.experimental || false }}
```

The `include` adds a macOS + Node 22 job that is not part of the OS × Node cross-product. The `experimental` variable is set to `true` for this job, and `continue-on-error` allows it to fail without failing the whole workflow.

### Step 6: Add a Matrix-Specific Step

Add a step that runs only on Ubuntu (since Windows runners do not have `uname`):

```yaml
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: ${{ matrix.node }}

  - name: Show OS info
    run: uname -a
    if: runner.os == 'Linux'

  - name: Show Node version
    run: node --version

  - run: npm ci
  - run: npm test
```

The `if: runner.os == 'Linux'` condition ensures the `uname` step only runs on Ubuntu. Windows and macOS runners skip it. This is the pattern for writing matrix workflows that handle OS-specific behavior.

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add OS-conditional step to matrix"
git push origin main
```

### Step 7: Set max-parallel to Limit Concurrency

Update the strategy to limit concurrent jobs:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20, 22]
  max-parallel: 3
```

With `max-parallel: 3`, only 3 of the 6 jobs run at once. The remaining 3 wait in a queue. This prevents the CI from consuming too many runner minutes at once.

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Limit matrix to 3 parallel jobs"
git push origin main
```

Watch the Actions tab for the next run. You should see only 3 jobs active at a time, with the rest showing "Queued" or "Waiting" status.

### Step 8: Use the Matrix Output in a Downstream Job

Create a downstream job that aggregates matrix results. Update `ci.yml`:

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test

  summary:
    needs: test
    runs-on: ubuntu-latest
    if: always()
    steps:
      - name: Print matrix summary
        run: |
          echo "All matrix jobs completed."
          echo "Checked: Node 18, 20, 22 on Ubuntu and Windows"
```

The `summary` job runs after all matrix jobs finish (regardless of whether they passed or failed, because of `if: always()`). Downstream jobs can aggregate test results, publish reports, or send notifications.

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add summary job after matrix completes"
git push origin main
```

## Common Errors at This Stage

### Error 1: `runs-on` with literal `${{ matrix.os }}` not expanded

```
Error: .github/workflows/ci.yml (Line: 6, Col: 14) — Unrecognized named-value: 'matrix'
```

**Why it happens:** You referenced `${{ matrix.os }}` in `runs-on` but the `strategy.matrix` block is missing or misconfigured. The `matrix` context is only available inside a job that has a `strategy.matrix` definition.

**Fix:** Ensure the `strategy` block is correctly indented under the job:

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]
```

### Error 2: Jobs fail on Windows due to path separator differences

```
Error: Cannot find module '.\index.js'
```

**Why it happens:** Windows uses `\` as the path separator, while Linux and macOS use `/`. Some tools or scripts hardcode forward slashes.

**Fix:** Use cross-platform tools or normalize paths. For Node.js, `require('./index')` works cross-platform. For scripts, consider using `path.join()` or the `cross-env` package.

### Error 3: Large matrix hits the 256-job limit

```
Error: Matrix exceeds maximum number of jobs (256)
```

**Why it happens:** The Cartesian product of all matrix axes exceeds 256. GitHub Actions enforces a maximum of 256 matrix jobs per workflow.

**Fix:** Reduce the number of axes or values. Split into separate workflows if needed:

```yaml
# Instead of a 10 × 10 × 3 = 300 matrix, split into:
# Fast CI: 2 × 3 = 6 jobs
# Full compatibility: 10 × 1 × 3 = 30 jobs (weekly cron)
```

### Error 4: Matrix jobs use too many runner minutes

```
Workflow run cancelled — account minutes exhausted
```

**Why it happens:** Each matrix job consumes runner minutes independently. A 6-job matrix on GitHub-hosted runners uses 6 × runtime minutes. Free accounts have a monthly limit.

**Fix:** Use `max-parallel` to limit concurrency, use smaller runners, or add caching to reduce per-job runtime. For personal projects, consider excluding expensive OS combinations.

- You created a matrix build with Node version axes: `node: [18, 20, 22]`

- You limited concurrency with `max-parallel` to control runner minute usage
- You added OS-conditional steps using `if: runner.os == 'Linux'`
- You created a downstream summary job that runs after the matrix completes
