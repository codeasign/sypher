You have six commits on your branch: a typo fix, a commit with a debugging `print()` still in it, a commit whose message says "asdf", and three real changes mixed between them. You want to squash the fix into the commit that introduced the bug, remove the debug print, reword the "asdf" message, and keep the three real commits in order — all before anyone sees them in a pull request. Interactive rebase lets you edit, reorder, squash, fixup, reword, and drop commits before they become part of the permanent record.

## The Concept

Interactive rebase opens a text editor with a list of commits you want to rewrite, and lets you specify an action for each one: pick, reword, edit, squash, fixup, exec, or drop. Git then replays the commits in order, applying your instructions for each.

Think of writing a first draft of a report, then going back with a red pen before submitting it. You combine two short paragraphs into one (squash), fix a typo in a heading (reword), remove a footnote that no longer applies (drop), and reorder two sections that make more sense swapped. The final submission is clean, but the working draft had all the intermediate mess. Interactive rebase is the red pen.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git rebase -i HEAD~N` | Rebase the last N commits interactively. | `git rebase -i HEAD~5` |
| `git rebase -i <commit>` | Rebase all commits after `<commit>` interactively (`<commit>` itself is not included). | `git rebase -i a1b2c3d` |
| `git rebase -i --root` | Rebase every commit in the repository from the initial commit. | `git rebase -i --root` |
| `git commit --amend` | Edit the most recent commit's message or content (non-interactive but related). | `git commit --amend -m "Better message"` |
| `git rebase --continue` | Continue after resolving conflicts or completing an edit step. | `git rebase --continue` |
| `git rebase --abort` | Abort the interactive rebase and return to the original state. | `git rebase --abort` |
| `git rebase --edit-todo` | Re-open the interactive rebase instruction list during a paused rebase. | `git rebase --edit-todo` |

### Interactive rebase actions

| Action | Abbreviation | What it does |
|---|---|---|
| `pick` | `p` | Use the commit as-is (you can reorder picks to change commit order). |
| `reword` | `r` | Use the commit's changes but edit the commit message. |
| `edit` | `e` | Stop at this commit to make changes (amend content or message). |
| `squash` | `s` | Combine this commit into the previous one, merging both messages. |
| `fixup` | `f` | Combine this commit into the previous one, discarding this commit's message. |
| `drop` | `d` | Remove this commit entirely. |
| `exec` | `x` | Run a shell command (useful for running tests during a rebase). |
| `break` | `b` | Stop at this point (useful with `exec` to inspect state). |

**Windows note:** Interactive rebase opens your default Git editor. To use VS Code: `git config --global core.editor "code --wait"`. On Windows, VS Code is the most approachable editor for the rebase todo list. On macOS/Linux, the default is usually `vim` or `nano`.

## The Big Picture

```

│                                                         │
│  Before:                                                │
│  main ──▶ C0 ──▶ C1 ──▶ C2 ──▶ F0 ──▶ F1 ──▶ F2      │
│                                            │            │
│  Interactive rebase todo list:             │            │
│  pick F0  "Add feature"                    │            │
│  squash F1 "Fix typo in feature"          │            │
│  reword F2 "Add tests (was: asdf)"        ▼            │
│                                                         │
│  After:                                                 │
│  main ──▶ C0 ──▶ C1 ──▶ C2 ──▶ F0_squashed ──▶ F2'    │
│                                                         │
│  F0_squashed = F0 + F1 combined. F2' has new message.  │
│  3 commits became 2. Everything rewritten.              │
└────────────────────────────────────────────────────────┘
```

## How It Actually Works — Step by Step

### Step 1: Git opens the todo list in your editor

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((F0)) ──▶ ((F1)) ──▶ ((F2))  ← feature
```

You run `git rebase -i HEAD~3`. Git writes a todo file to `.git/rebase-merge/git-rebase-todo/`:

```
pick a1b2c3d Add feature
pick b2c3d4e Fix typo in feature
pick c3d4e5f asdf
```

Below the instructions, Git includes a comment block explaining each action. You edit this file, save, and close.

### Step 2: Git parses the todo and begins replay

You change the todo to:

```
pick a1b2c3d Add feature
squash b2c3d4e Fix typo in feature
reword c3d4e5f asdf
```

Git starts from the parent of the first listed commit. For each line, Git checks out the commit's parent and applies the specified action.

### Step 3: Squash — combining commits

Git applies "Add feature" normally (pick). Then, for "Fix typo in feature" (squash), Git:

1. Applies the diff of `b2c3d4e` on top of `a1b2c3d`
2. Opens your editor with a combined commit message from both commits
3. Creates a new commit with the combined message and both diffs

```
  ((C0)) ──▶ ... ──▶ ((C2)) ──▶ ((F0+F1 combined))  ← HEAD (detached)
```

### Step 4: Reword — changing the message

For the reword action:

1. Git applies the diff of `c3d4e5f` on top of the squashed commit
2. Instead of using the original message "asdf", Git opens your editor with that message
3. You change it to "Add tests" and save
4. Git creates a new commit with the new message

```
  ((C0)) ──▶ ... ──▶ ((C2)) ──▶ ((F0+F1 combined)) ──▶ ((F2' "Add tests"))  ← HEAD (detached)
```

### Step 5: Branch pointer update

Git moves the `feature` branch pointer to the new tip commit `F2'` and attaches HEAD:

```
  ((C0)) ──▶ ... ──▶ ((C2)) ──▶ ((F0+F1 combined)) ──▶ ((F2' "Add tests"))  ← feature
                                                                             ← HEAD
```

The original commits `F0`, `F1`, `F2` are unreachable. `git log --oneline` now shows 2 feature commits instead of 3.

## When to Use This / When Not To

**Use interactive rebase when:**
- You want to clean up a branch before submitting a pull request — squash fixup commits, reword unclear messages
- You need to reorder commits to make the history tell a coherent story
- You want to remove a commit with debugging code, a secret, or a mistake
- You need to split one commit into multiple smaller ones

**Do not use interactive rebase when:**
- The commits have been pushed to a shared branch (creates divergent history for everyone else)
- You have not yet committed and just want to change the last commit — use `git commit --amend` instead
- You want to keep the exact chronological record, warts and all, for audit purposes
- You are not confident you can resolve the conflicts that may arise during replay

Interactive rebase is the most powerful tool in Git for shaping commits before they become permanent. The key constraint is: rebase only commits that exist locally. Once pushed to a shared remote, a commit's hash is part of the shared history — rewriting it forces everyone who pulled it to reconcile the divergence.
