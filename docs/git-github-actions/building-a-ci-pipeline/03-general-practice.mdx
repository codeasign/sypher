Complete these exercises on your own to prove you can build, modify, and debug CI pipelines from scratch.

## Tier 1 — Beginner

### Exercise 1: Add a CI Pipeline to a Python Project

**Task:** Create a CI pipeline for a Python project that installs dependencies from `requirements.txt` and runs `pytest`. The pipeline should trigger on every push to `main` and on every pull request targeting `main`.

**Hint:** Use `actions/setup-python@v5` instead of `actions/setup-node@v4`. The install command is `pip install -r requirements.txt` and the test command is `pytest`.

**Solution:**

```yaml
name: Python CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install -r requirements.txt
      - run: pytest
```

The pipeline checks out code, installs Python 3.12, installs dependencies from `requirements.txt`, and runs `pytest`. Any test failure causes the pipeline to fail and report a red status on the commit.

### Exercise 2: Add a Lint Step to an Existing CI Pipeline

**Task:** You have a JavaScript CI pipeline that runs `npm test`. Add a lint step that runs `npm run lint` before the test step. The lint step should use `eslint` and fail the pipeline if there are lint errors.

**Hint:** Add `npm run lint` as a step before `npm test`. If the project does not have a lint script, add one to `package.json`:

```json
"scripts": {
  "lint": "eslint ."
}
```

**Solution:**

```yaml
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 20
  - run: npm ci
  - run: npm run lint
  - run: npm test
```

The lint step runs before tests. If linting fails, the test step is never reached, saving runner minutes. The commit shows which step failed — lint or test — making it easier to diagnose.

### Exercise 3: Add a Timeout to a CI Pipeline

**Task:** Your test suite sometimes hangs on flaky network tests. Add a 5-minute timeout to the CI job so that hung tests are automatically cancelled instead of running for the default 6-hour maximum.

**Hint:** Add `timeout-minutes: 5` at the job level.

**Solution:**

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

With `timeout-minutes: 5`, GitHub Actions terminates the job after 5 minutes. The job is marked as "Cancelled" rather than "Failed." This is important for teams that pay for runner minutes — a hung job running for 6 hours would burn 360 billable minutes.

### Exercise 4: Create a CI Pipeline with a `continue-on-error` Step

**Task:** Create a CI pipeline that runs tests and code coverage. The code coverage step should be allowed to fail without failing the pipeline. The test step should still block the pipeline.

**Hint:** Add `continue-on-error: true` to the coverage step.

**Solution:**

```yaml
name: CI with Coverage

on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

      - name: Code coverage
        continue-on-error: true
        run: npx jest --coverage
```

The test step must pass for the pipeline to succeed. The coverage step is informational — if coverage drops below the threshold (e.g., `< 80%`), the pipeline reports the failure but does not block merging. This is useful for teams that want to track coverage trends without enforcing a hard gate.

## Tier 2 — Intermediate

### Exercise 1: Multi-Job CI Pipeline

**Task:** Create a CI pipeline with two parallel jobs — `lint` and `test` — and a third job `report` that runs after both complete. The `report` job should print a summary of what was checked.

**Hint:** Use `needs: [lint, test]` on the `report` job. Both `lint` and `test` run in parallel, and `report` waits for both to finish.

**Solution:**

```yaml
name: Multi-Job CI

on: [push, pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  report:
    needs: [lint, test]
    runs-on: ubuntu-latest
    if: always()
    steps:
      - run: |
          echo "Lint and test completed."
          echo "Triggered by: ${{ github.event_name }}"
          echo "Branch: ${{ github.ref_name }}"
```

The `lint` and `test` jobs run in parallel, each on their own runner. The `report` job runs after both complete, regardless of success or failure (because of `if: always()`). This pattern is useful for sending notifications or publishing results after all checks finish.

### Exercise 2: CI Pipeline with Conditional Steps

**Task:** Create a CI pipeline that runs a full test suite on PRs but only runs linting on direct pushes to non-main branches. Use `if` conditions to control which steps run.

**Hint:** Check `github.event_name` to distinguish between push and pull_request events.

**Solution:**

```yaml
name: Conditional CI

on:
  push:
  pull_request:
    branches: [main]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint

      - name: Run tests
        if: github.event_name == 'pull_request' || github.ref == 'refs/heads/main'
        run: npm test
```

Tests run only on PRs and pushes to `main`. Direct pushes to feature branches only run linting, which is faster and sufficient for quick iterations. This saves runner minutes while still running the full suite before merging.

### Exercise 3: Pipeline That Reports to Pull Request

**Task:** Create a CI pipeline that posts a comment on the pull request with the test results summary. Use the `actions/github-script` action to comment.

**Hint:** Use `github-script` with the `context` and `github` objects to create a comment on the PR.

**Solution:**

```yaml
name: CI with PR Comment

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

      - name: Comment PR
        uses: actions/github-script@v7
        if: always()
        with:
          script: |
            const status = '${{ job.status }}';
            const runId = '${{ github.run_id }}';
            const body = `## CI Results\n- **Status:** ${status}\n- **Run:** [${runId}](https://github.com/${context.repo.owner}/${context.repo.repo}/actions/runs/${runId})`;
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: body
            });
```

The PR comment includes the CI status and a direct link to the workflow run. This is a more detailed alternative to the commit status checkmark — useful for teams that want to see the results in the PR conversation directly.

## Tier 3 — Advanced

### Exercise 1: Debug a Broken CI Pipeline

**Scenario:** You are onboarding to a team and find this CI pipeline in their repository. It is supposed to run tests but it is failing. Diagnose and fix all the issues.

```yaml
name: CI

on: push

jobs:
  build-and-test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm install
      - run: npm run build
      - run: npm run test
```

**Issues to identify:**

1. **Missing colon after job name:** `build-and-test` should be `build-and-test:` (YAML syntax error)
2. **`npm install` instead of `npm ci`:** `npm install` may resolve different versions than `package-lock.json`. Use `npm ci` for deterministic builds.
3. **Missing `--if-present`:** If the project has no build script, `npm run build` fails. Add `--if-present`.
4. **No timeout:** A hung test could run for 6 hours, consuming runner minutes.

**One valid fix:**

```yaml
name: CI

on: push

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build --if-present
      - run: npm test
```

### Exercise 2: Design a CI Pipeline for a Multi-Language Monorepo

**Scenario:** You have a monorepo with three directories: `frontend/` (Node.js), `backend/` (Python), and `shared/` (TypeScript). Design a CI pipeline that:
- Runs only the relevant checks for changed files (not all three on every commit)
- Uses separate jobs for each language
- Has a total timeout of 15 minutes
- Does not run more than 2 jobs in parallel

**Approach:**

There is no single correct answer, but a strong solution uses `paths` filters to trigger only the relevant jobs:

```yaml
name: Monorepo CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  frontend:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: |
          cd frontend
          npm ci
          npm test

  backend:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: |
          cd backend
          pip install -r requirements.txt
          pytest

  shared:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: |
          cd shared
          npm ci
          npm test
```

Alternative valid approaches include using `paths-filter` action to detect changed directories, or a single job with multiple working directories. The key trade-off is parallel speed vs. simplicity — separate jobs run in parallel but duplicate checkout and setup.

### Exercise 3: Migrate from Travis CI to GitHub Actions

**Scenario:** Your team uses this Travis CI configuration. Translate it to an equivalent GitHub Actions workflow:

```yaml
language: node_js
node_js:
  - 18
  - 20
script:
  - npm run lint
  - npm test
  - npm run build
notifications:
  email:
    on_failure: always
```

**One valid approach:**

```yaml
name: CI

on: [push, pull_request]

jobs:
  ci:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [18, 20]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build

      - name: Notify on failure
        if: failure()
        run: |
          echo "CI failed for Node ${{ matrix.node }}"
          echo "Run: https://github.com/${{ github.repository }}/actions/runs/${{ github.run_id }}"
```

The Travis `node_js` matrix becomes `strategy.matrix.node`. The `script` block becomes individual `run` steps. The email notification is replaced by a failure notification step (GitHub Actions sends email notifications on failure by default when the repository is configured to do so).
