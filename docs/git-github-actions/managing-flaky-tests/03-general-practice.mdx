## Tier 1 — Beginner

### Exercise 1.1: Identify the Flaky Root Cause

Read the following test and identify the root cause of its flakiness. Then write a fix.

```python
import tempfile
import os


def test_temp_file_cleanup():
    """Tests that a temporary file is created and then cleaned up."""
    tmp = tempfile.NamedTemporaryFile(delete=False)
    tmp.write(b"test data")
    tmp.close()

    # Verify the file exists
    assert os.path.exists(tmp.name)

    # Clean up
    os.remove(tmp.name)

    # Verify the file was deleted
    assert not os.path.exists(tmp.name)


def test_temp_file_count():
    """Tests that only one temporary file exists."""
    count = 0
    for f in os.listdir(tempfile.gettempdir()):
        if f.startswith("tmp"):
            count += 1
    assert count == 1
```

**Hint:** The second test depends on the state of the system, which is affected by the first test and by any other test that creates temporary files.

**Solution:** The root cause is test order dependency. `test_temp_file_count` assumes that the temporary directory contains only one temp file, but `test_temp_file_cleanup` already deleted its file. Worse, any other test running in the same session that creates a temp file will cause `test_temp_file_count` to fail. The fix is to isolate the second test from shared state:

```python
import tempfile
import os


def test_temp_file_cleanup():
    """Tests that a temporary file is created and then cleaned up."""
    tmp = tempfile.NamedTemporaryFile(delete=False)
    tmp.write(b"test data")
    tmp.close()

    assert os.path.exists(tmp.name)

    os.remove(tmp.name)

    assert not os.path.exists(tmp.name)


def test_temp_file_count_isolated():
    """Tests temp file counting using an isolated directory."""
    with tempfile.TemporaryDirectory() as tmpdir:
        # Create exactly one temp file in the isolated directory
        tmp = tempfile.NamedTemporaryFile(dir=tmpdir, delete=False)
        tmp.close()

        count = len([f for f in os.listdir(tmpdir) if f.startswith("tmp")])
        assert count == 1
```

### Exercise 1.2: Add Retry to a Flaky API Test

The following test intermittently fails because the external API is rate-limited. Add retry logic using `pytest-rerunfailures` without modifying the test code.

```python
import requests


def test_external_api():
    response = requests.get("https://api.github.com/zen", timeout=5)
    assert response.status_code == 200
    assert len(response.text) > 0
```

**Hint:** You do not need to change the test body. The retry configuration can be applied at the command line or in `pytest.ini`.

**Solution:** Run the test with the `--reruns` flag at the command line:

```bash
pytest test_external_api.py --reruns 3 --reruns-delay 2 -v
```

Or add the retry configuration to `pytest.ini` so it applies to all tests:

```ini
[pytest]
addopts = --reruns 3 --reruns-delay 2
```

The `--reruns 3` flag retries each failed test up to 3 times. The `--reruns-delay 2` adds a 2-second delay between retries, which gives the API rate limiter time to reset.

### Exercise 1.3: Configure a Non-Blocking Test Job

You have a test that is known to be flaky but you are not ready to quarantine it yet. Configure the GitHub Actions job so that the pipeline passes even if this test fails.

**Starting workflow:**

```yaml
name: CI
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install pytest
      - run: pytest tests/
```

**Hint:** There are two ways to make a test non-blocking in GitHub Actions. One is at the job level and one is at the step level. The step-level approach is more surgical.

**Solution (job-level approach):**

```yaml
name: CI
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest
    continue-on-error: true
    steps:
      - uses: actions/checkout@v4
      - run: pip install pytest
      - run: pytest tests/
```

**Solution (step-level approach, more precise):**

```yaml
name: CI
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install pytest
      - name: Run flaky tests
        continue-on-error: true
        run: pytest tests/flaky/
      - name: Run stable tests
        run: pytest tests/stable/
```

### Exercise 2.3: Create a Flaky Test Budget

Your team has agreed to a flaky test budget: no more than 2% of all test runs can be flaky failures. Design a workflow that enforces this budget by comparing the flake rate against the threshold and failing the pipeline if the budget is exceeded.

**Hint:** Track the total number of test runs and the total number of flaky failures. Compute the flake rate as `flaky_failures / total_runs`. If the rate exceeds 2%, fail the pipeline.

**Solution:**

```yaml
name: Flake Budget Enforcement
on:
  push:
    branches: [main]

jobs:
  enforce-budget:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest pytest-rerunfailures pytest-json-report
      - name: Run tests with flake tracking
        run: |
          pytest tests/ \
            --reruns 2 \
            --json-report \
            --json-report-file=report.json
      - name: Compute flake budget
        run: |
          python -c "
import json
with open('report.json') as f:
    report = json.load(f)

total_tests = len(report.get('tests', []))
total_retries = sum(t.get('rerun', 0) for t in report.get('tests', []))
flake_rate = total_retries / total_tests if total_tests > 0 else 0
budget = 0.02  # 2% budget

print(f'Total tests: {total_tests}')
print(f'Total retries (flaky indicators): {total_retries}')
print(f'Flake rate: {flake_rate:.2%}')
print(f'Budget: {budget:.0%}')

if flake_rate > budget:
    print(f'Flake budget exceeded! Rate {flake_rate:.2%} exceeds {budget:.0%}')
    exit(1)
else:
    print(f'Flake budget OK. Rate {flake_rate:.2%} is within {budget:.0%}')
"
```

## Tier 3 — Advanced

### Exercise 3.1: Build a Self-Healing Test Runner

Design a test runner that executes tests in three phases. In phase 1, it runs the full suite. In phase 2, it re-runs only the failed tests (up to 3 times). In phase 3, it classifies the persistently failing tests into three categories: flaky (passed on retry), broken (failed on all attempts), and new (failed on first run but code changed).

**Reasoning-based solution:**

The self-healing test runner needs to distinguish between tests that fail because of flakiness and tests that fail because of a real bug. The key insight is that a real bug produces consistent failures, while a flaky test produces intermittent failures. The runner uses a three-phase approach to make this classification.

```python
#!/usr/bin/env python3
"""
self_healing_runner.py — A self-healing test runner that classifies failures.
"""
import json
import subprocess
import sys
from collections import defaultdict


def run_pytest(args):
    """Run pytest with the given arguments and return the JSON report."""
    cmd = ["pytest", "--json-report", "--json-report-file=-"] + args
    result = subprocess.run(cmd, capture_output=True, text=True)
    return json.loads(result.stdout)


def classify_failures(phase1_report, phase2_report, phase3_report):
    """Classify each test based on its behavior across three phases."""
    classification = {}

    # Phase 1: all tests run once
    phase1_outcomes = {}
    for t in phase1_report.get("tests", []):
        phase1_outcomes[t["nodeid"]] = t["outcome"]

    # Phase 2: failed tests retried
    phase2_outcomes = {}
    for t in phase2_report.get("tests", []):
        phase2_outcomes[t["nodeid"]] = t["outcome"]

    # Phase 3: persistently failed tests retried again
    phase3_outcomes = {}
    for t in phase3_report.get("tests", []):
        phase3_outcomes[t["nodeid"]] = t["outcome"]

    # Classify each test
    all_test_ids = set(phase1_outcomes.keys())
    for test_id in all_test_ids:
        p1 = phase1_outcomes.get(test_id, "unknown")
        p2 = phase2_outcomes.get(test_id, "not-run")
        p3 = phase3_outcomes.get(test_id, "not-run")

        if p1 == "passed":
            classification[test_id] = "clean"
        elif p1 == "failed" and p2 == "passed":
            classification[test_id] = "flaky"
        elif p1 == "failed" and p2 == "failed" and p3 == "passed":
            classification[test_id] = "flaky-retry"
        elif p1 == "failed" and p2 == "failed" and p3 == "failed":
            classification[test_id] = "broken"
        else:
            classification[test_id] = "unknown"

    return classification


def main():
    print("Phase 1: Running full test suite...")
    phase1 = run_pytest(sys.argv[1:])

    failed_tests = [
        t["nodeid"] for t in phase1.get("tests", [])
        if t["outcome"] == "failed"
    ]

    if not failed_tests:
        print("All tests passed in phase 1. No retry needed.")
        print(json.dumps({"outcome": "passed", "classification": {}}))
        return

    print(f"Phase 2: Retrying {len(failed_tests)} failed tests...")
    phase2 = run_pytest(["-k", " or ".join(failed_tests)])

    still_failing = [
        t["nodeid"] for t in phase2.get("tests", [])
        if t["outcome"] == "failed"
    ]

    if not still_failing:
        print("All retries passed in phase 2. Tests are flaky.")
        classification = classify_failures(phase1, phase2, {"tests": []})
        report = {"outcome": "flaky-passed", "classification": classification}
        print(json.dumps(report, indent=2))
        return

    print(f"Phase 3: Final retry of {len(still_failing)} persistently failing tests...")
    phase3 = run_pytest(["-k", " or ".join(still_failing)])

    classification = classify_failures(phase1, phase2, phase3)
    broken = [t for t, c in classification.items() if c == "broken"]
    flaky = [t for t, c in classification.items() if "flaky" in c]

    report = {
        "outcome": "broken" if broken else "flaky-passed",
        "classification": classification,
        "broken_tests": broken,
        "flaky_tests": flaky,
    }

    print(json.dumps(report, indent=2))

    if broken:
        print(f"\n{len(broken)} tests are broken (not flaky):")
        for t in broken:
            print(f"  BROKEN: {t}")
        sys.exit(1)


if __name__ == "__main__":
    main()
```

This runner is self-healing because it automatically retries flaky tests without human intervention. The exit code is non-zero only when tests are classified as broken (consistently failing on all attempts). Flaky tests that pass on retry are reported but do not block the pipeline.

### Exercise 3.2: Build a Flaky Test Prediction Model

Design a system that predicts which tests are likely to become flaky before they actually fail in CI. The prediction should be based on test characteristics such as execution time variance, external dependencies, and shared state usage.

**Reasoning-based solution:**

Predicting flaky tests requires measuring test characteristics that correlate with flakiness. The most predictive characteristics are:

1. **Execution time variance:** A test that takes 0.1 seconds on one run and 2 seconds on another is likely flaky due to timing issues.
2. **External dependency count:** Tests that make HTTP requests or access databases are more likely to be flaky.
3. **Shared state usage:** Tests that modify global variables, files, or databases are more likely to be flaky due to test order dependencies.
4. **Skip rate:** Tests that are frequently skipped due to environment conditions are candidates for flakiness.

```python
#!/usr/bin/env python3
"""
flaky_predictor.py — Predicts flaky tests based on execution characteristics.
"""

import json
import subprocess
import statistics
from collections import defaultdict


def collect_test_metrics():
    """Run the test suite and collect per-test metrics."""
    result = subprocess.run(
        ["pytest", "--json-report", "--json-report-file=-", "-v", "--durations=0"],
        capture_output=True, text=True
    )
    report = json.loads(result.stdout)

    metrics = defaultdict(dict)
    for test in report.get("tests", []):
        nodeid = test["nodeid"]
        call = test.get("call", {})
        setup = test.get("setup", {})
        teardown = test.get("teardown", {})

        metrics[nodeid] = {
            "duration": call.get("duration", 0),
            "setup_duration": setup.get("duration", 0),
            "teardown_duration": teardown.get("duration", 0),
            "outcome": test["outcome"],
            "has_external_dep": "requests" in str(test) or "http" in str(test).lower(),
            "has_shared_state": "os.environ" in str(test) or "global" in str(test).lower(),
        }

    return metrics


def predict_flake_risk(metrics_history):
    """Predict flake risk based on execution characteristic variance."""
    predictions = {}

    for test_id, runs in metrics_history.items():
        durations = [r["duration"] for r in runs if r["duration"] > 0]
        outcomes = [r["outcome"] for r in runs]

        if len(durations) < 3:
            predictions[test_id] = {"risk": "insufficient-data", "score": 0}
            continue

        # Factor 1: Duration variance (coefficient of variation)
        mean_dur = statistics.mean(durations)
        stdev_dur = statistics.stdev(durations) if len(durations) > 1 else 0
        cv = stdev_dur / mean_dur if mean_dur > 0 else 0

        # Factor 2: Historical failure rate
        failures = outcomes.count("failed")
        failure_rate = failures / len(outcomes)

        # Factor 3: External dependency penalty
        has_external = runs[0].get("has_external_dep", False)
        has_shared = runs[0].get("has_shared_state", False)

        # Composite flake score (0-100)
        score = 0
        score += min(cv * 50, 40)  # Duration variance (up to 40 points)
        score += failure_rate * 40  # Historical failure rate (up to 40 points)
        score += 10 if has_external else 0  # External dependency (10 points)
        score += 10 if has_shared else 0  # Shared state (10 points)

        if score >= 30:
            risk = "high"
        elif score >= 15:
            risk = "medium"
        elif score >= 5:
            risk = "low"
        else:
            risk = "stable"

        predictions[test_id] = {
            "risk": risk,
            "score": round(score, 1),
            "duration_cv": round(cv, 3),
            "failure_rate": round(failure_rate, 3),
            "has_external_dep": has_external,
            "has_shared_state": has_shared,
        }

    return predictions


def main():
    # Collect 10 runs of test metrics
    print("Collecting test metrics over 10 runs...")
    all_metrics = defaultdict(list)
    for i in range(10):
        print(f"  Run {i + 1}/10...")
        metrics = collect_test_metrics()
        for test_id, data in metrics.items():
            all_metrics[test_id].append(data)

    # Predict flake risk
    predictions = predict_flake_risk(all_metrics)

    # Output results
    print(f"\nFlake Risk Predictions:")
    print(f"{'Test':<60} {'Risk':<10} {'Score':<8} {'Dur CV':<8} {'Fail Rate':<10}")
    print("-" * 96)

    for test_id, pred in sorted(predictions.items(), key=lambda x: x[1]["score"], reverse=True):
        print(
            f"{test_id:<60} {pred['risk']:<10} "
            f"{pred['score']:<8} {pred['duration_cv']:<8} "
            f"{pred['failure_rate']:<10}"
        )


if __name__ == "__main__":
    main()
```

This prediction model scores each test on a 0-100 scale. Tests with a score above 30 are flagged as high-risk and should be quarantined or rewritten before they start causing pipeline failures. The model is not perfect, but it catches the most common patterns that lead to flakiness before they manifest as actual flaky failures.

### Exercise 3.3: Design a Flaky Test SLA

Your team wants to define a Service Level Agreement (SLA) for flaky tests. The SLA has three tiers:

- **Tier 1 (Critical):** A flaky test in the main suite must be quarantined within 4 hours of detection.
- **Tier 2 (Standard):** A quarantined test must be diagnosed and fixed within 5 business days.
- **Tier 3 (Maintenance):** A test in quarantine for 30 days must be deleted.

Design a workflow that enforces this SLA by creating GitHub issues for each violation, escalating to the team lead if the SLA is breached, and automatically deleting tests that exceed the 30-day quarantine limit.

**Reasoning-based solution:**

The SLA enforcement requires three separate workflows running on different schedules. The escalation logic uses GitHub issues with labels and assignees to track SLA compliance.

Workflow 1 — Detection and Quarantine (runs every 4 hours):

```yaml

      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5

          python-version: "3.12"
      - run: pip install pytest pytest-rerunfailures pytest-json-report
      - name: Run tests with flake detection

          pytest tests/ \
            --reruns 3 \
            --reruns-delay 1 \
            --json-report \
```

Workflow 2 — SLA Escalation (runs daily):

```yaml
name: SLA Tier 2 Escalation
on:
  schedule:
    - cron: "0 9 * * *"
  workflow_dispatch:

jobs:
  escalate:
    runs-on: ubuntu-latest
    steps:
      - name: Check for SLA breaches
        run: |
          # Find issues labeled flaky and sla-tier1 that are older than 4 hours
          gh issue list \
            --label flaky,sla-tier1 \
            --json number,title,createdAt \
            --jq '.[] | select(.createdAt < (now - 14400)) | .number' \
            | while read issue; do
              gh issue comment "$issue" \
                --body "SLA Tier 1 breach: This flaky test has not been quarantined within 4 hours. Escalating to team lead."
              gh issue edit "$issue" --add-label sla-escalated
              gh issue edit "$issue" --remove-label sla-tier1
            done
```

Workflow 3 — Auto-Delete Quarantined Tests (runs weekly):

```yaml
name: SLA Tier 3 — Auto-Delete Stale Quarantined Tests
on:
  schedule:
    - cron: "0 10 * * 1"
  workflow_dispatch:

jobs:
  auto-delete:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
      - name: Find tests quarantined for 30+ days
        run: |
          git log --diff-filter=A --name-only --pretty=format:"%ad %H" --since="30 days ago" -- tests/quarantine/ \
            | grep "tests/quarantine/" \
            | while read file; do
              git rm "$file"
              echo "Deleted $file (quarantined for 30+ days without fix)"
            done
      - name: Create PR for deletions
        if: success()
        run: |
          git config user.name "Flake SLA Bot"
          git config user.email "sla-bot@example.com"
          git add -A
          git diff --staged --quiet || (
            git commit -m "Auto-delete quarantined tests exceeding 30-day SLA"
            git push origin main
          )
```

This SLA enforcement system ensures that flaky tests are detected quickly, quarantined within hours, fixed within days, and deleted if they cannot be fixed within a month. The combination of automated detection, human escalation, and automatic deletion creates a self-correcting system that maintains test suite reliability over time.
