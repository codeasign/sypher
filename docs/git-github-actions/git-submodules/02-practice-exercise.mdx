You will create a parent repository and a submodule repository, add the submodule, clone the parent with submodules, update the submodule, and remove the submodule. All labs are local-only — no GitHub account needed.

## Setup

Create two repositories: the "parent" and the "library" (which will be added as a submodule):

```bash
mkdir -p ~/submodule-lab
cd ~/submodule-lab
```

Create the library repository:

```bash
mkdir lib-repo
cd lib-repo
git init
git config user.name "Library Author"
git config user.email "lib@example.com"
echo "print('Library v1')" > utils.py
git add utils.py
git commit -m "Initial library"
echo "def helper():" >> utils.py
echo "    return 'helper v1'" >> utils.py
git add utils.py
git commit -m "Add helper function"
cd ..
```

Create the parent repository:

```bash
mkdir parent-repo
cd parent-repo
git init
git config user.name "Parent Author"
git config user.email "parent@example.com"
echo "# My Project" > README.md
echo "def main():" > main.py
echo "    pass" >> main.py
git add README.md main.py
git commit -m "Initial parent project"
cd ..
```

## The Walkthrough

### Step 1: Add the library as a submodule

```bash
cd parent-repo
git submodule add ../lib-repo lib/utils
```

Expected output:

```
Cloning into 'D:/.../parent-repo/lib/utils'...
done.
```

**Under the hood:** Git:
1. Cloned `lib-repo` into `lib/utils`
2. Created `.gitmodules` with the submodule URL and path
3. Staged `.gitmodules` and the gitlink entry for `lib/utils`

Verify:

```bash
cat .gitmodules
git submodule status
```

Expected output:

```
[submodule "lib/utils"]
    path = lib/utils
    url = ../lib-repo
 abc1234 lib/utils (heads/main)
```

The hash is the current commit of the library repo.

### Step 2: Commit the submodule reference

```bash
git add .gitmodules lib/utils
git commit -m "Add utils submodule"
git log --oneline
```

Expected output:

```
abc1234 (HEAD -> main) Add utils submodule
def5678 Initial parent project
```

### Step 3: Clone the parent repo and initialize submodules

```bash
cd ..
git clone parent-repo cloned-parent
cd cloned-parent
ls lib/utils/
```

Expected output — the directory exists but is empty:

```
(empty directory)
```

Initialize and update the submodule:

```bash
git submodule init
```

Expected output:

```
Submodule 'lib/utils' (D:/.../lib-repo) registered for path 'lib/utils'
```

```bash
git submodule update
```

Expected output:

```
Cloning into 'D:/.../cloned-parent/lib/utils'...
done.
Submodule path 'lib/utils': checked out 'abc1234...'
```

Now the submodule content is available:

```bash
cat lib/utils/utils.py
```

Expected output:

```
print('Library v1')
def helper():
    return 'helper v1'
```

**Checkpoint:** `git submodule status` shows the pinned commit hash with no prefix (no `+` or `-`).

### Step 4: Update the library and pin the parent to the new version

Make a change in the original library:

```bash
cd ../lib-repo
echo "def new_feature():" >> utils.py
echo "    return 'new stuff'" >> utils.py
git add utils.py
git commit -m "Add new feature"
cd ../parent-repo
```

Update the submodule in the parent to point to the latest commit:

```bash
cd lib/utils
git pull origin main
```

Expected output:

```
Updating abc1234..def5678
Fast-forward
 utils.py | 3 +++
 1 file changed, 3 insertions(+)
```

```bash
cd ../..
git status
```

Expected output:

```
Changes not staged for commit:
  modified: lib/utils (new commits)
```

The parent detects that the submodule pointer has changed. Commit the new pin:

```bash
git add lib/utils
git commit -m "Update utils submodule to latest"
```

### Step 5: Update all submodules to their latest remote commits

```bash
git submodule update --remote lib/utils
```

If there are newer commits in the library, this fetches and checks them out. If the library is up to date, it reports:

```
Already up to date.
```

### Step 6: Clone with submodules in one command

```bash
cd ..
git clone --recurse-submodules parent-repo full-clone
cd full-clone
cat lib/utils/utils.py
```

Expected output — the submodule is already populated with the latest pinned version:

```
print('Library v1')
def helper():
    return 'helper v1'
def new_feature():
    return 'new stuff'
```

### Step 7: Work across parent and submodule simultaneously

Make a change in the submodule that the parent needs:

```bash
cd ../parent-repo
cd lib/utils
echo "def critical_fix():" >> utils.py
echo "    return 'fixed'" >> utils.py
git add utils.py
git commit -m "Critical fix"
cd ../..
```

Now update the parent to point to the new submodule commit:

```bash
git add lib/utils
git commit -m "Pin to submodule with critical fix"
```

**Under the hood:** The submodule is a full Git repository. You can branch, commit, push, and pull inside it. The parent only cares about the commit hash — it does not care how the submodule got to that hash.

### Step 8: Remove the submodule

```bash
git submodule deinit lib/utils
```

Expected output:

```
Cleared directory 'lib/utils'
Submodule 'lib/utils' (../lib-repo) unregistered
```

```bash
git rm lib/utils
```

Expected output:

```
rm 'lib/utils'
```

Remove the cached submodule metadata:

```bash
rm -rf .git/modules/lib/utils
```

Commit the removal:

```bash
git add .
git commit -m "Remove utils submodule"
```

Verify:

```bash
ls lib/
```

Expected output — the `utils` directory is gone.

## Common Errors at This Stage

### Error 1: Submodule directory is empty after clone

You cloned the parent repo without `--recurse-submodules`. The submodule path exists but is empty because Git does not fetch submodules automatically.

**Fix:** Run `git submodule init` followed by `git submodule update`. Or, to fix a clone that is already in progress, `git submodule update --init --recursive` handles both initialization and update in one command.

### Error 2: `fatal: Path '<path>' is in submodule '<name>'`

You tried to add a file to the parent repo at a path that is already registered as a submodule. Git refuses because the submodule owns that path.

**Fix:** If you want to replace the submodule with a regular directory, remove the submodule first: `git submodule deinit <path>`, `git rm <path>`, then add your files. If you want to add files inside the submodule, change directory into the submodule and work there.

### Error 3: `fatal: No url found for submodule path '<path>' in .gitmodules`

The `.gitmodules` file is missing or corrupted. This can happen if you deleted `.gitmodules` manually or merged a branch that removed it.

**Fix:** Restore `.gitmodules` from the last commit: `git checkout HEAD -- .gitmodules`. If the submodule was added and committed, the file exists in the commit history. If you need to re-add the submodule, `git submodule add <url> <path>` again.

### Error 4: `fatal: remote error: upload-pack: not our ref`

The submodule is pinned to a commit that does not exist in the remote submodule repository. This happens when the submodule was updated to a commit that was later squashed, rebased, or force-pushed away.

**Fix:** The submodule's pinned commit is missing. If you have access to a local copy that has the commit, push it to the submodule's remote. Otherwise, update the pin to a valid commit: `git submodule update --remote` pulls the latest from the remote, or manually set the pin to a known-good commit hash.

## What You Just Learned

- A submodule is a reference from the parent repo to a specific commit in another repo
- `.gitmodules` stores the submodule URL and path mapping
- `git submodule add <url> <path>` adds a submodule
- `git clone --recurse-submodules` clones a repo with all submodules
- `git submodule update --remote` updates submodules to the latest remote commit
- After updating a submodule, commit the parent to record the new pin
- `git submodule deinit` + `git rm` removes a submodule
- Submodules are independent Git repos — you can branch, commit, and push inside them
