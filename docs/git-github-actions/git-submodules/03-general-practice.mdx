## Tier 1 — Beginner

### Exercise 1

**Task:** Create two repos: `lib-a` (2 commits) and `parent` (1 commit). Add `lib-a` as a submodule to `parent` at `deps/lib-a`. Commit the parent. Verify the submodule is listed with `git submodule status`.

<details>
<summary>Hint</summary>
Create `lib-a` with `git init`, make 2 commits. Create `parent` with `git init`, make 1 commit. Run `git submodule add <path-to-lib-a> deps/lib-a` from the parent.
</details>

### Solution

```bash
mkdir -p ~/sub-beginner-1 && cd ~/sub-beginner-1
mkdir lib-a && cd lib-a && git init
echo "v1" > f.txt && git add f.txt && git commit -m "Lib v1"
echo "v2" >> f.txt && git add f.txt && git commit -m "Lib v2"
cd ..
mkdir parent && cd parent && git init
echo "parent" > main.txt && git add main.txt && git commit -m "Parent init"
git submodule add ../lib-a deps/lib-a
git submodule status
git add . && git commit -m "Add lib-a submodule"
```

### Exercise 2

**Task:** Clone the parent repo from Exercise 1 with `--recurse-submodules`. Verify the submodule content is present without running `git submodule init` or `git submodule update`.

<details>
<summary>Hint</summary>
`git clone --recurse-submodules <source> <dest>`. Check `cat cloned/deps/lib-a/f.txt`.
</details>

### Solution

```bash
cd ~/sub-beginner-1
git clone --recurse-submodules parent cloned
cat cloned/deps/lib-a/f.txt
```

The file shows "v1" and "v2" — the submodule content is fully populated.

### Exercise 3

**Task:** Update the submodule to a new commit. In `lib-a`, add a third commit. In the parent, update the submodule to point to the new commit. Verify the parent's `git status` shows the change.

<details>
<summary>Hint</summary>
In `lib-a`, make a new commit. In the parent, `cd deps/lib-a && git pull origin main`. Then `cd ../..` and `git status` shows `modified: deps/lib-a`.
</details>

### Solution

```bash
cd ~/sub-beginner-1/lib-a
echo "v3" >> f.txt && git add f.txt && git commit -m "Lib v3"
cd ../parent
cd deps/lib-a && git pull origin main
cd ../.. && git status
git add deps/lib-a
git commit -m "Update lib-a to v3"
```

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a parent repo with two submodules: `lib-a` and `lib-b`. Use `git submodule foreach` to run `git status` in each submodule. Then use `git submodule foreach` to pull the latest changes in all submodules at once.

<details>
<summary>Hint</summary>
`git submodule foreach 'git status'` runs the command in each submodule. `git submodule foreach 'git pull origin main'` pulls in all submodules.
</details>

### Solution

```bash
mkdir -p ~/sub-inter-1 && cd ~/sub-inter-1
for lib in lib-a lib-b; do
  mkdir $lib && cd $lib && git init
  echo "$lib v1" > f.txt && git add f.txt && git commit -m "$lib init"
  cd ..
done
mkdir parent && cd parent && git init && echo "parent" > main.txt
git add main.txt && git commit -m "Parent init"
git submodule add ../lib-a deps/lib-a
git submodule add ../lib-b deps/lib-b
git add . && git commit -m "Add both submodules"
git submodule foreach 'git status'
git submodule foreach 'git log --oneline -1'
```

### Exercise 2

**Task:** Create a nested submodule scenario: repo `lib-a` itself has a submodule `lib-a-core`. Add `lib-a` as a submodule to `parent` with `--recursive`. Verify that `parent` sees both `lib-a` and `lib-a`'s submodule `lib-a-core`.

<details>
<summary>Hint</summary>
Create `lib-core`, add it as a submodule to `lib-a`. Then `git submodule add --recursive ../lib-a deps/lib-a` in the parent. Use `git submodule status --recursive` to see both levels.
</details>

### Solution

```bash
mkdir -p ~/sub-inter-2 && cd ~/sub-inter-2
mkdir lib-core && cd lib-core && git init
echo "core" > core.txt && git add core.txt && git commit -m "Core init"
cd ..
mkdir lib-a && cd lib-a && git init
echo "lib-a" > a.txt && git add a.txt && git commit -m "Lib A init"
git submodule add ../lib-core core
git add . && git commit -m "Add core submodule"
cd ..
mkdir parent && cd parent && git init
echo "parent" > p.txt && git add p.txt && git commit -m "Parent init"
git submodule add --recursive ../lib-a deps/lib-a
git submodule status --recursive
```

### Exercise 3

**Task:** Create a scenario where a submodule is on a detached HEAD. Fix it by checking out a branch in the submodule. Verify the parent's status shows the submodule is now on a branch.

<details>
<summary>Hint</summary>
`git submodule update` checks out the pinned commit in detached HEAD. To fix, `cd` into the submodule and `git switch main`. Then `git status` in the parent shows the submodule has changed.
</details>

### Solution

```bash
mkdir -p ~/sub-inter-3 && cd ~/sub-inter-3
mkdir lib && cd lib && git init
echo "v1" > f.txt && git add f.txt && git commit -m "V1"
echo "v2" >> f.txt && git add f.txt && git commit -m "V2"
cd ..
mkdir parent && cd parent && git init
echo "parent" > p.txt && git add p.txt && git commit -m "Parent init"
git submodule add ../lib deps/lib
git add . && git commit -m "Add submodule"
# Update submodule to latest
cd deps/lib && git checkout HEAD~1  # Detached HEAD
cd ../.. && git status
cd deps/lib && git switch main  # Fix detached HEAD
cd ../.. && git status
```

## Tier 3 — Advanced

### Exercise 1: Multi-repo feature branch workflow

**Scenario:** You need to make a change across two repositories: the parent and a submodule. The parent needs a new feature, and the submodule needs a new function to support it. Both changes must be committed and tested together before the parent can pin to the new submodule commit.

**Task:** Create this scenario. In the parent repo, create a branch `feature`. In the submodule, also create a branch `feature`. Make compatible changes in both branches. Test that the parent's code works with the submodule's new function. Then commit both changes — the submodule first, then update the parent's pin.

<details>
<summary>Hint</summary>
On the parent's `feature` branch, `cd` into the submodule, create a `feature` branch there, make the change, commit it. Then `cd` back to the parent, make the parent's change, commit it. The parent's commit includes the updated submodule pointer.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/sub-adv-1 && cd ~/sub-adv-1
mkdir lib && cd lib && git init
echo "def existing():" > lib.py && echo "    return 'old'" >> lib.py
git add lib.py && git commit -m "Initial lib"
cd ..
mkdir parent && cd parent && git init
echo "from lib import existing" > app.py && echo "print(existing())" >> app.py
git add app.py && git commit -m "Initial parent"
git submodule add ../lib deps/lib
git add . && git commit -m "Add lib submodule"

# Feature branch in parent
git checkout -b feature
# Feature branch in submodule
cd deps/lib && git checkout -b feature
echo "def new_func():" >> lib.py
echo "    return 'new'" >> lib.py
git add lib.py && git commit -m "Add new_func"
cd ../..
# Update parent to use new submodule
git add deps/lib
echo "from deps.lib.lib import new_func" >> app.py
echo "print(new_func())" >> app.py
git add app.py
git commit -m "Use new submodule feature"
# Test
python app.py
```

**Alternative approach:** Use a script that runs both repos' tests sequentially. The key principle is that submodule changes must be committed and pushed before the parent can pin to them.

### Exercise 2: Automate submodule updates with a CI script

**Scenario:** Your team has 5 submodules. Every week, you want to update all submodules to their latest commits, run the test suite, and create a PR if tests pass. You want to automate this with a script.

**Task:** Write a script that:
1. Updates all submodules to the latest remote commit
2. Records the old and new commit hashes
3. Stages the changes
4. Creates a commit with a message listing each submodule update
5. (Optional) Creates a PR using `gh`

<details>
<summary>Hint</summary>
Use `git submodule status` before and after `git submodule update --remote --recursive` to capture old and new hashes. Generate the commit message from the diff.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/sub-adv-2 && cd ~/sub-adv-2
# Setup: parent with multiple submodules
for lib in lib-a lib-b; do
  mkdir $lib && cd $lib && git init
  echo "$lib v1" > f.txt && git add f.txt && git commit -m "Init $lib"
  cd ..
done
mkdir parent && cd parent && git init && echo "parent" > main.txt
git add main.txt && git commit -m "Parent init"
git submodule add ../lib-a deps/a && git submodule add ../lib-b deps/b
git add . && git commit -m "Add submodules"
```

The update script:

```bash
cat > update-submodules.sh << 'EOF'
#!/bin/bash
echo "=== Updating submodules ==="
# Capture old hashes
OLD=$(git submodule status)
# Update
git submodule update --remote --recursive
NEW=$(git submodule status)
# Generate commit message
MSG="chore: update submodules to latest"
while IFS= read -r line; do
  OLD_HASH=$(echo "$line" | awk '{print $1}' | sed 's/^-//' | sed 's/^+//')
  NAME=$(echo "$line" | awk '{print $2}')
  NEW_HASH=$(git submodule status | grep "$NAME" | awk '{print $1}' | sed 's/^-//' | sed 's/^+//')
  if [ "$OLD_HASH" != "$NEW_HASH" ]; then
    MSG="$MSG"$'\n'"- $NAME: $OLD_HASH -> $NEW_HASH"
  fi
done <<< "$OLD"
# Commit
git add -A
git commit -m "$MSG" 2>/dev/null && echo "Committed: $MSG" || echo "No changes"
EOF
chmod +x update-submodules.sh
./update-submodules.sh
```

**Alternative approach:** Use Dependabot or Renovate for automated dependency updates. These tools handle submodule updates and can create PRs automatically with changelogs.

### Exercise 3: Migrate a submodule to a subtree

**Scenario:** Your team is frustrated with submodule complexity. You decide to migrate one submodule to use `git subtree` instead, which embeds the submodule's history directly into the parent repo.

**Task:** Create a repo with a submodule, then use `git subtree add` to replace it. The final state should have the submodule's files in the parent repo without the submodule indirection. Compare the workflow: with a submodule, you need `git submodule update`; with a subtree, the files are always present.

<details>
<summary>Hint</summary>
First `git subtree add --prefix=<path> <submodule-url> <branch>` to pull the submodule repo's history into the parent. Then remove the submodule with `git submodule deinit` and `git rm`. The subtree approach keeps the files in the parent's history.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/sub-adv-3 && cd ~/sub-adv-3
# Create the library
mkdir lib && cd lib && git init
echo "lib v1" > lib.py && git add lib.py && git commit -m "Init lib"
echo "lib v2" >> lib.py && git add lib.py && git commit -m "Lib v2"
cd ..
# Create parent with submodule
mkdir parent && cd parent && git init
echo "parent" > main.py && git add main.py && git commit -m "Init parent"
git submodule add ../lib deps/lib
git add . && git commit -m "Add lib as submodule"
# Now migrate to subtree
git subtree add --prefix=deps/lib ../lib main 2>/dev/null || true
# Remove the submodule reference
git submodule deinit deps/lib 2>/dev/null || true
git rm deps/lib 2>/dev/null || true
# Re-add files via subtree (if the above removed them)
git subtree add --prefix=deps/lib ../lib main --squash
# The files are now embedded in the parent
cat deps/lib/lib.py
git log --oneline
```

The parent's history now includes the library's commits (or a squashed summary if `--squash` was used). No `.gitmodules` file is needed. The library files are always present in every clone — no `git submodule update` required.

**Alternative approach:** Use `git filter-repo` to rewrite the submodule's history as part of the parent, then merge. This is more complex but produces a cleaner history where the submodule appears as if it was always part of the parent repo.
