Complete these exercises on your own to prove you can design, deploy, and debug CD pipelines.

## Tier 1 — Beginner

### Exercise 1: Deploy a Static Site to GitHub Pages

**Task:** Create a GitHub Actions workflow that deploys a static site (HTML files in a `public/` directory) to GitHub Pages. The workflow should trigger on every push to `main`.

**Hint:** Use `actions/upload-pages-artifact@v3` with `path: public` and `actions/deploy-pages@v4`.

**Solution:**

```yaml
name: Deploy to Pages

on:
  push:
    branches: [main]

permissions:
  pages: write
  id-token: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/upload-pages-artifact@v3
        with:
          path: public/
      - id: deployment
        uses: actions/deploy-pages@v4
```

The workflow uploads the `public/` directory as a Pages artifact and deploys it. The `permissions` block grants the necessary access. The `environment` block links the deployment to the `github-pages` environment.

### Exercise 2: Add a Smoke Test After Deploy

**Task:** Extend the GitHub Pages deployment from Exercise 1 with a post-deploy smoke test that verifies the deployed site returns HTTP 200.

**Hint:** Add a step after `deploy-pages` that runs `curl -s -o /dev/null -w "%{http_code}" $URL` and checks for 200.

**Solution:**

```yaml
steps:
  - uses: actions/checkout@v4
  - uses: actions/upload-pages-artifact@v3
    with:
      path: public/
  - id: deployment
    uses: actions/deploy-pages@v4

  - name: Smoke test
    run: |
      sleep 15
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" ${{ steps.deployment.outputs.page_url }})
      [ "$STATUS" = "200" ] && echo "✅ Site is live" || (echo "❌ Got $STATUS" && exit 1)
```

The smoke test waits 15 seconds for GitHub Pages to propagate, then issues a `curl` request. If the response is not 200, the step fails, marking the deployment as failed. This catches deployment issues that the deploy action itself might not detect.

### Exercise 3: Deploy with a Concurrency Guard

**Task:** Create a deployment pipeline that prevents two concurrent deployments to the same environment. If a new push happens during an active deployment, the new push should wait (not cancel the in-progress deployment).

**Hint:** Use `concurrency` with a group name and `cancel-in-progress: false`.

**Solution:**

```yaml
name: Safe Deploy

on:
  push:
    branches: [main]

concurrency:
  group: production-deploy
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

With `cancel-in-progress: false`, the second push waits for the first deployment to complete. This prevents two deployments from running simultaneously, which could cause database migration conflicts or inconsistent state.

## Tier 2 — Intermediate

### Exercise 1: CI/CD Pipeline with Build Artifact Passing

**Task:** Create a two-job CI/CD pipeline where the `build` job creates a compiled artifact and uploads it, and the `deploy` job downloads and deploys that exact artifact. The deploy job should only run on pushes to `main`.

**Hint:** Use `actions/upload-artifact@v4` in the build job and `actions/download-artifact@v4` in the deploy job.

**Solution:**

```yaml
name: CI/CD with Artifacts

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: build-output
          path: dist/
      - name: Deploy to server
        run: |
          echo "Deploying dist/ contents..."
          ls -la dist/
```

The `upload-artifact` step saves the `dist/` directory as a named artifact. The `deploy` job downloads the exact same artifact, ensuring the bytes deployed are the same ones that were built — not a different build from a different commit.

### Exercise 2: Conditional Deploy Based on Branch

**Task:** Create a workflow that deploys to staging from any branch, but only deploys to production from `main`. Use a single workflow file with conditional job execution.

**Hint:** Use `if: github.ref == 'refs/heads/main'` on the production deploy job.

**Solution:**

```yaml
name: Multi-Environment Deploy

on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh staging

  deploy-production:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://your-app.io
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

The staging deploy runs on every branch push. The production deploy only runs when the push is to `main`. The `if` condition is evaluated before the runner is provisioned, so production runner minutes are not wasted on non-main branches.

### Exercise 3: Add a Rollback Step to a Deployment Pipeline

**Task:** Create a deployment pipeline that, on success, records the current commit SHA to a file. Add a separate `rollback` workflow that accepts a commit SHA as input and redeploys that version.

**Hint:** Use `workflow_dispatch` with an `inputs.commit` parameter for the rollback workflow. Use `actions/checkout@v4` with `ref: ${{ github.event.inputs.commit }}` to check out the old version.

**Solution:**

**Deploy workflow (`.github/workflows/deploy.yml`):**

```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
      - name: Record successful deploy
        run: |
          echo "LAST_SUCCESSFUL_DEPLOY=${{ github.sha }}" > deploy-state.txt
          echo "Deploy time: $(date)" >> deploy-state.txt
```

**Rollback workflow (`.github/workflows/rollback.yml`):**

```yaml
name: Rollback

on:
  workflow_dispatch:
    inputs:
      commit:
        description: 'Commit SHA to rollback to'
        required: true

jobs:
  rollback:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.inputs.commit }}
      - run: ./deploy.sh production
      - name: Rollback confirmation
        run: |
          echo "✅ Rolled back to ${{ github.event.inputs.commit }}"
```

The rollback workflow is triggered manually from the Actions tab. It checks out the specified commit and deploys it. This is a safety net — if a deployment breaks production, anyone with write access can rollback by running the workflow and pasting the last known-good commit SHA.

## Tier 3 — Advanced

### Exercise 1: Debug a Broken CD Pipeline

**Scenario:** Your team has this CD pipeline. It is supposed to deploy to staging and then production, but production deployments are failing and no one is sure why.

```yaml
name: CD

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: npm test

  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh staging

  deploy-production:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

**Issues to identify:**

1. **Checkout is missing from the `test` job:** The `test` job runs `npm test` without checking out code first. The runner has no files to test.
2. **Production deploy depends on test, not staging:** The `deploy-production` job should `needs: deploy-staging` so it only runs after staging succeeds, not in parallel with it.
3. **No environment or approval gate:** Production deploys should use an environment with protection rules.
4. **No `if` condition on production deploy:** It deploys to production even from non-main branches (though the `on: push branches: [main]` trigger limits this).

**One valid fix:**

```yaml
name: CD

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://your-app.io
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

### Exercise 2: Design a Blue-Green Deployment Pipeline

**Scenario:** Design a deployment pipeline for a web application that uses a blue-green deployment strategy. The pipeline should:
1. Build and test the application
2. Deploy the new version to the inactive environment (green if blue is active, blue if green is active)
3. Run smoke tests against the new environment
4. Switch the load balancer to point to the new environment
5. Keep the old environment running for 15 minutes for rollback

**Approach:**

There is no single correct answer, but a strong approach uses environment variables to track the active color:

```yaml
name: Blue-Green Deploy

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test && npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/

  deploy:
    needs: build
    runs-on: ubuntu-latest
    outputs:
      active: ${{ steps.determine.outputs.active }}
      inactive: ${{ steps.determine.outputs.inactive }}
    steps:
      - name: Determine active environment
        id: determine
        run: |
          # Check which environment is currently active
          # In a real system, this would query a load balancer or state file
          echo "active=blue" >> $GITHUB_OUTPUT
          echo "inactive=green" >> $GITHUB_OUTPUT

      - uses: actions/download-artifact@v4
        with:
          name: build-output
          path: dist/

      - name: Deploy to inactive (${{ steps.determine.outputs.inactive }})
        run: |
          echo "Deploying to ${{ steps.determine.outputs.inactive }}..."
          # In a real system: push to the inactive server group

      - name: Smoke test inactive
        run: |
          echo "Smoke testing ${{ steps.determine.outputs.inactive }}..."
          # In a real system: curl the inactive server's health endpoint

      - name: Switch load balancer
        run: |
          echo "Switching traffic to ${{ steps.determine.outputs.inactive }}..."
          # In a real system: update load balancer target group

  cleanup:
    needs: deploy
    runs-on: ubuntu-latest
    steps:
      - name: Keep old env for 15 min
        run: |
          echo "Keeping old environment (${{ needs.deploy.outputs.active }}) ready for rollback"
          echo "Will terminate in 15 minutes"
```

The key design decisions are: (1) querying the active color dynamically so the pipeline works regardless of which color is active, (2) smoke testing the inactive environment before switching traffic, and (3) keeping the old environment alive for a rollback window.

### Exercise 3: Pipeline with Canary Deployment and Automated Rollback

**Scenario:** Design a canary deployment pipeline that:
1. Deploys the new version to 10% of traffic
2. Monitors error rates for 5 minutes
3. If error rate < 1%, gradually increases to 100%
4. If error rate >= 1%, rolls back immediately

**One valid approach:**

```yaml
name: Canary Deploy

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test && npm run build

  canary:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy canary (10% traffic)
        run: |
          echo "Deploying to canary group (10% of instances)..."
          # In a real system: update 10% of instances

      - name: Monitor for 5 minutes
        run: |
          for i in $(seq 1 10); do
            ERROR_RATE=$(curl -s https://monitoring.example.com/api/error-rate)
            echo "Minute $i: Error rate = $ERROR_RATE%"
            if (( $(echo "$ERROR_RATE >= 1.0" | bc -l) )); then
              echo "❌ Error rate exceeded 1% — rolling back!"
              # In a real system: redeploy previous version
              exit 1
            fi
            sleep 30
          done
          echo "✅ Canary passed — full rollout"

  full-rollout:
    needs: canary
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to 100% of instances
        run: |
          echo "Deploying to all instances..."
          # In a real system: update remaining 90% of instances
```

The critical design element is the monitoring loop: it polls the error rate every 30 seconds for 5 minutes. If the error rate exceeds the threshold at any point, it exits with code 1, triggering an automatic rollback. The `full-rollout` job only runs if the canary passes.
