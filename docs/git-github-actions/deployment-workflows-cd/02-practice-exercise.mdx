You will create a CI/CD pipeline that builds a static site, deploys it to GitHub Pages, and adds a staging-to-production promotion flow. By the end of this exercise, you will have a working CD pipeline that deploys your site on every green commit.

## Setup

Create a repository on GitHub called `cd-pipeline-practice` and clone it:

```bash
git clone https://github.com/YOUR-USERNAME/cd-pipeline-practice.git
cd cd-pipeline-practice
```

Set up a simple static site:

```bash
mkdir -p .github/workflows site
```

Create `site/index.html`:

```html
<!DOCTYPE html>
<html>
<head>
  <title>CD Pipeline Practice</title>
  <style>
    body { font-family: sans-serif; max-width: 600px; margin: 40px auto; }
    .badge { padding: 8px 16px; border-radius: 4px; }
    .green { background: #4caf50; color: white; }
  </style>
</head>
<body>
  <h1>CD Pipeline Practice</h1>
  <p class="badge green">✅ Deployed via GitHub Actions</p>
  <p>Version: 1.0.0</p>
  <p>Last deployed: <span id="deploy-time"></span></p>
  <script>
    document.getElementById('deploy-time').textContent = new Date().toISOString();
  </script>
</body>
</html>
```

Create a simple test for the HTML file:

```bash
npm init -y
npm install --save-dev html-validator-cli
```

Update `package.json`:

```json
{
  "scripts": {
    "test": "html-validator --file=site/index.html --format=text",
    "build": "echo 'Build complete'"
  }
}
```

Commit the initial setup:

```bash
git add .
git commit -m "Add static site files"
git push origin main
```

## The Walkthrough

### Step 1: Enable GitHub Pages

Go to your repository on GitHub: Settings > Pages > Source > GitHub Actions. This configures GitHub Pages to accept deployments from Actions workflows.

### Step 2: Create a CI-Only Workflow

Create `.github/workflows/ci.yml` — just the CI pipeline:

```yaml
name: CI

on: [push, pull_request]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test
```

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add CI pipeline"
git push origin main
```

Verify the CI pipeline passes in the Actions tab.

### Step 3: Add a Deploy-to-Pages Job

Create `.github/workflows/deploy.yml`:

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pages: write
      id-token: write

    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - uses: actions/checkout@v4

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: site/

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

This workflow:
- Triggers only on pushes to `main`
- Uses the `github-pages` environment (GitHub automatically creates this)
- Uploads the `site/` directory as a Pages artifact
- Deploys it to GitHub Pages

### Step 4: Combine CI and CD into One Pipeline

Create a unified `.github/workflows/ci-cd.yml` that combines both CI and CD with proper job dependencies:

```yaml
name: CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  deploy:
    needs: ci
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write

    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/upload-pages-artifact@v3
        with:
          path: site/
      - id: deployment
        uses: actions/deploy-pages@v4
```

The `deploy` job:
- `needs: ci` — only runs after CI passes
- `if: github.ref == 'refs/heads/main'` — only deploys from main
- `github.event_name == 'push'` — does not deploy on PR events

Commit and push:

```bash
git add .github/workflows/ci-cd.yml
git rm .github/workflows/ci.yml .github/workflows/deploy.yml
git commit -m "Combine CI and CD into unified pipeline"
git push origin main
```

### Step 5: Add a Staging Environment

Create a second workflow for staging deployments. This simulates deploying to a staging environment before production:

```yaml
name: Staging Deploy

on:
  push:
    branches: [develop]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy-staging:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.your-org.pages.dev
    steps:
      - uses: actions/checkout@v4
      - name: Build for staging
        run: |
          mkdir -p dist
          cp site/index.html dist/
          sed -i "s/Version: 1.0.0/Version: 1.0.0-staging/" dist/index.html
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist/
      - id: deployment
        uses: actions/deploy-pages@v4
```

Commit and push to a `develop` branch:

```bash
git checkout -b develop
git add .github/workflows/staging.yml
git commit -m "Add staging deployment workflow"
git push origin develop
```

### Step 6: Add a Concurrency Guard

Update the deployment workflow to prevent concurrent deployments:

```yaml
name: CI/CD

on:
  push:
    branches: [main]

concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: true

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  deploy:
    needs: ci
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write

    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/upload-pages-artifact@v3
        with:
          path: site/
      - id: deployment
        uses: actions/deploy-pages@v4
```

The `concurrency` block ensures that when a new push happens during an active deployment, the running deployment is cancelled and the new one starts immediately. This prevents the "last writer wins" race condition.

Commit and push:

```bash

```

### Step 7: Add a Smoke Test After Deploy

Add a smoke test step that validates the deployment was successful:

```yaml
deploy:
  needs: ci
  runs-on: ubuntu-latest
  environment:
    name: github-pages
    url: ${{ steps.deployment.outputs.page_url }}

  steps:
    - uses: actions/checkout@v4
    - uses: actions/upload-pages-artifact@v3
      with:
        path: site/
    - id: deployment
      uses: actions/deploy-pages@v4

    - name: Smoke test
      run: |
        sleep 10  # Wait for Pages to propagate
        curl -s -o /dev/null -w "%{http_code}" ${{ steps.deployment.outputs.page_url }} | grep 200
        echo "Smoke test passed — site is serving 200 OK"
```

The smoke test waits 10 seconds for the Pages CDN to propagate, then does a `curl` to verify the site returns HTTP 200. If the site returns anything else (404, 500), the step fails and the deployment is marked as failed even though the deployment itself succeeded.

Commit and push:

```bash
git add .github/workflows/ci-cd.yml
git commit -m "Add post-deploy smoke test"
git push origin main
```

### Step 8: Add a Rollback Step

Add a manual rollback trigger using `workflow_dispatch`:

```yaml
name: Rollback

on:
  workflow_dispatch:
    inputs:
      commit:
        description: 'Commit SHA to rollback to'
        required: true

jobs:
  rollback:
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write

    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.inputs.commit }}

      - uses: actions/upload-pages-artifact@v3
        with:
          path: site/

      - id: deployment
        uses: actions/deploy-pages@v4

      - name: Notify rollback
        run: |
          echo "Rolled back to ${{ github.event.inputs.commit }}"
          echo "URL: ${{ steps.deployment.outputs.page_url }}"
```

This rollback workflow can be triggered manually from the Actions tab. Pass the commit SHA of the previous stable deployment, and it redeploys that version.

Commit and push:

```bash
git add .github/workflows/rollback.yml
git commit -m "Add manual rollback workflow"
git push origin main
```

## Common Errors at This Stage

### Error 1: `deploy-pages` action fails with "Not authorized"

```
Error: Not authorized to deploy to GitHub Pages
```

**Why it happens:** The workflow is missing the `pages: write` and `id-token: write` permissions.

**Fix:** Add the permissions block at the job level:

```yaml
deploy:
  permissions:
    contents: read
    pages: write
    id-token: write
```

### Error 2: GitHub Pages returns 404 after deployment

```
curl: (22) The requested URL returned error: 404
```

**Why it happens:** GitHub Pages has a brief propagation delay (~1-2 minutes) after deployment. The `deploy-pages` action returns immediately after the deployment is queued, not after it is live.

**Fix:** Increase the sleep time in the smoke test to 30 seconds, or retry the curl:

```yaml
- name: Smoke test with retry
  run: |
    for i in 1 2 3 4 5; do
      status=$(curl -s -o /dev/null -w "%{http_code}" $URL || echo "000")
      [ "$status" = "200" ] && echo "Smoke test passed" && exit 0
      echo "Attempt $i: $status — waiting 15s..."
      sleep 15
    done
    echo "Smoke test failed after 5 attempts"
    exit 1
  env:
    URL: ${{ steps.deployment.outputs.page_url }}
```

### Error 3: Deploy job runs on PRs from forks

```
Deploy job ran on a fork PR and failed with "permission denied"
```

**Why it happens:** The `deploy` job runs on every push event, including PRs from forks. Fork PRs do not have access to deployment secrets or Pages permissions.

**Fix:** Add a condition to skip deploy on fork PRs:

```yaml
deploy:
  if: github.event.pull_request.head.repo.fork == false
```

### Error 4: "Upload-pages-artifact" fails with empty directory

```
Error: No files found at path: site/
```

**Why it happens:** The `site/` directory does not exist at the checkout root, or the `actions/checkout` step is missing.

**Fix:** Ensure `actions/checkout@v4` runs before the upload step, and verify the path is relative to the repository root. If the build step produces output in a different directory, adjust the path accordingly.
