You will simulate the full code review lifecycle using a local repository and the GitHub CLI (`gh`). If you do not have a GitHub account, the lab includes a local-only simulation mode. Every command works identically on Windows, macOS, and Linux.

## Setup

### Prerequisites

- A GitHub account and a repository you can push to
- GitHub CLI installed: `gh auth login` (run once)
- A local repository with at least one commit on `main`

If you do not have a GitHub repo, create one:

```bash
# Create a repo on GitHub (replace 'your-username' with your actual username)
gh repo create code-review-lab --public --clone
cd code-review-lab
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create the starting code:

```bash
cat > calculator.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b
EOF
git add calculator.py
git commit -m "Initial calculator"
git push origin main
```

## The Walkthrough

### Step 1: Create a feature branch with deliberate issues

```bash
git switch -c add-divide
```

Add a divide function with a known bug (no zero-division check):

```bash
cat > calculator.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b


def divide(a, b):
    return a / b
EOF
git add calculator.py
git commit -m "Add divide function"
git push origin add-divide
```

### Step 2: Create a pull request

```bash
gh pr create \
  --title "Add divide function" \
  --body "Adds a divide function to the calculator module.

**Changes:**
- New `divide(a, b)` function that returns `a / b`

**Testing:**
- Verified manually with print(divide(10, 2)) → 5.0

**Checklist:**
- [ ] Unit tests added
- [ ] Edge cases handled (zero division)" \
  --assignee @me
```

Expected output:

```
https://github.com/your-username/code-review-lab/pull/1
```

Open the URL in your browser or view with:

```bash
gh pr view 1
```

### Step 3: Review the PR yourself (simulating a reviewer)

Since you are the only person on this repo, you will simulate being the reviewer:

```bash
gh pr checkout 1
```

View the diff:

```bash
gh pr diff 1
```

Expected output:

```
diff --git a/calculator.py b/calculator.py
index abc..def 100644
--- a/calculator.py
+++ b/calculator.py
@@ -5,3 +5,7 @@ def subtract(a, b):
 
 
 def subtract(a, b):
     return a - b
+
+
+def divide(a, b):
+    return a / b
```

Now, as the reviewer, you notice the missing zero-division check. Submit a review requesting changes:

```bash
gh pr review 1 --request-changes \
  --body "The `divide` function will raise a `ZeroDivisionError` if `b` is 0. \
Please add a check for this case. Consider returning `None` or raising a \
`ValueError` with a descriptive message."
```

Expected output:

```
✓ Requested changes on pull request #1
```

### Step 4: Address the review feedback

Switch back to the feature branch:

```bash
git switch add-divide
```

Fix the function:

```bash
cat > calculator.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b


def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b
EOF
git add calculator.py
git commit -m "Add zero-division check"
git push origin add-divide
```

The PR automatically updates with the new commit. The "Request changes" status is still active — the reviewer needs to re-review.

### Step 5: Re-review and approve

```bash
gh pr checkout 1
gh pr diff 1
```

Verify the zero-division check is present. Then approve:

```bash
gh pr review 1 --approve --body "Zero-division check added. Looks good."
```

Expected output:

```
✓ Approved pull request #1
```

### Step 6: Merge the approved PR

```bash
gh pr merge 1 --squash --delete-branch
```

Expected output:

```
✓ Merged pull request #1 (add-divide)
✓ Deleted branch add-divide
```

Clean up locally:

```bash
git switch main
git pull origin main
git branch -d add-divide
```

### Step 7: Set up branch protection (if you have admin access)

If you own the repository, enable branch protection:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": null,
  "enforce_admins": true,
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "dismiss_stale_reviews": true
  },
  "restrictions": null
}
EOF
```

This requires at least 1 approval and dismisses approvals when new commits are pushed. You can also configure this in the GitHub web UI: Settings → Branches → Add rule.

### Step 8: Local-only simulation (no GitHub account)

If you do not have a GitHub account, simulate the review workflow locally:

```bash
mkdir -p ~/local-review-lab && cd ~/local-review-lab
git init
git config user.name "Author"
git config user.email "author@example.com"
echo "init" > code.py
git add code.py && git commit -m "Initial commit"

# Author creates a branch
git checkout -b feature
echo "buggy code" >> code.py
git add code.py && git commit -m "Add feature"
git log --oneline main..feature > diff.txt

# Simulate review by saving diff for inspection
echo "=== PR Simulation ==="
echo "Branch: feature"
echo "Diff:"
cat diff.txt
echo ""
echo "=== Reviewer Comments ==="
echo "1. Please add input validation"
echo ""
echo "=== Author Addresses Feedback ==="
echo "addressed" >> code.py
git add code.py && git commit -m "Address review: add input validation"

# Simulate approval
echo "=== Review Approved ==="
git checkout main
git merge feature
echo "PR merged successfully"
```

## Common Errors at This Stage

### Error 1: `! [rejected] main -> main (fetch first)`

You tried to push to main but the remote has commits you do not have locally. This happens when a PR was merged by someone else or via the web UI.

**Fix:** Pull the latest changes: `git pull origin main` (or `git pull --rebase origin main` to keep history linear).

### Error 2: `gh pr create: GraphQL: Base branch was not found`

The base branch you specified for the PR does not exist on the remote. Often this happens when you specify `main` but the remote uses `master`.

**Fix:** List branches on the remote: `git ls-remote --heads origin`. Use the correct branch name: `gh pr create --base master`.

### Error 3: `gh pr merge: Pull request #N is not in a mergeable state`

The PR has merge conflicts or required checks are failing. GitHub prevents merging until the conflicts are resolved or the checks pass.

**Fix:** Resolve conflicts locally: `git fetch origin main && git rebase origin/main`, fix conflicts, `git add`, `git rebase --continue`, `git push origin <branch> --force-with-lease`. The PR updates automatically.

### Error 4: Review approvals are dismissed after pushing new commits

When "Dismiss stale reviews" is enabled in branch protection, any new commit to the PR branch invalidates all previous approvals. This is by design — it ensures reviewers see the latest code.

**Fix:** After pushing changes, request re-review from the same reviewers. Use `gh pr review <number> --comment --body "Addressed all feedback. Ready for re-review."` to notify them.

## What You Just Learned

- `gh pr create` opens a pull request from the current branch
- `gh pr review` submits a review (approve, request changes, or comment)
- `gh pr checkout` checks out a PR locally for in-depth review
- `gh pr merge` merges an approved PR (with `--squash`, `--rebase`, or default merge)
- Branch protection enforces review requirements and status checks
- Address review feedback by pushing new commits to the same branch
- Keep PRs small (under 400 lines) for effective review
- Use `--dismiss-stale-reviews` to ensure reviewers see the latest code
