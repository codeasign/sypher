You push a branch with 300 lines of changes, assign two reviewers, and wait. One reviewer approves 20 minutes later. The other leaves no comments. You are not sure if they actually read the code or just clicked Approve. A week later, the change causes a production outage because a subtle logic error was missed. An effective code review workflow is not about rubber-stamping вҖ” it is a structured process with clear expectations, concrete feedback, and explicit sign-off.

## The Concept

A code review workflow is the structured process by which changes in a pull request are examined, discussed, approved, and merged. It includes the roles (author, reviewer, maintainer), the stages (open, review, revise, approve, merge), and the expectations at each stage.

Think of it like a manuscript peer review. The author submits a draft (PR). The editor assigns reviewers. Each reviewer reads the draft, marks up issues, and returns it with a verdict: accept, accept with minor changes, or reject with major revisions. The author revises and resubmits. The editor makes the final call on publication. The process is documented so every participant knows what is expected of them.

## Command / Syntax Reference

| Command / Action | What it does | Example |
|---|---|---|
| `gh pr create` | Create a pull request from the current branch. | `gh pr create --title "Fix" --body "..."` |
| `gh pr list` | List pull requests for the current repository. | `gh pr list --state open --author @me` |
| `gh pr checkout <number>` | Check out a PR locally for review. | `gh pr checkout 42` |
| `gh pr diff` | View the diff of a PR. | `gh pr diff 42` |
| `gh pr review <number> --approve` | Approve a PR. | `gh pr review 42 --approve` |
| `gh pr review <number> --request-changes` | Request changes on a PR. | `gh pr review 42 --request-changes --body "..."` |
| `gh pr review <number> --comment` | Add a comment without approving or rejecting. | `gh pr review 42 --comment --body "LGTM"` |
| `gh pr merge <number>` | Merge an approved PR. | `gh pr merge 42 --squash` |
| `gh pr comment <number> --body "<text>"` | Add a comment to an existing PR. | `gh pr comment 42 --body "Addressed in commit abc"` |
| `gh pr status` | Show status of PRs relevant to the current user. | `gh pr status` |

**GitHub branch protection settings (repository settings):**

| Setting | What it does |
|---|---|
| `Require pull request reviews` | Require at least one approved review before merging |
| `Dismiss stale reviews` | Old approvals are invalidated when new commits are pushed |
| `Require review from code owners` | Automatically require review from files matched in CODEOWNERS |
| `Require status checks` | Require CI checks to pass before merging |
| `Require linear history` | Reject merge commits вҖ” only rebase or squash merges allowed |
| `Include administrators` | Apply branch protection rules to admins too |

**Windows note:** The `gh` CLI commands work identically on all platforms. Branch protection settings are configured via the GitHub web UI at Settings вҶ’ Branches вҶ’ Add rule, or via `gh api repos/:owner/:repo/branches/:branch/protection` for scripted setup.

## How It Actually Works вҖ” Step by Step

### Step 1: Author creates a branch and opens a PR

```
  main в”Җв”Җв–¶ C0 в”Җв”Җв–¶ C1 в”Җв”Җв–¶ C2
                          в”Ӯ
                          в””в”Җв”Җв–¶ F0 в”Җв”Җв–¶ F1  вҶҗ feature/calculate
```

The author pushes the branch and runs `gh pr create`. GitHub creates a PR object with:
- A unique number (e.g., #42)
- The base branch (`main`) and head branch (`feature/calculate`)
- The diff of commits `F0` and `F1`
- Status: `open`

### Step 2: Reviewers are assigned and notified

GitHub (or the team's automation) assigns reviewers based on:
- **Manual assignment:** Author selects specific reviewers
- **CODEOWNERS file:** Automatic assignment based on file paths matched
- **Round-robin:** Automatic rotation among team members

Each reviewer receives a notification. The PR status shows:
- `Review required` вҖ” waiting for approval
- `Changes requested` вҖ” at least one reviewer requested changes
- `Approved` вҖ” all required reviewers have approved

### Step 3: Reviewer examines the diff

The reviewer runs (or clicks):

```bash
gh pr diff 42
```

Or checks out the branch locally:

```bash
gh pr checkout 42
git log main..HEAD
git diff main..HEAD --stat
```

The reviewer examines each changed file, looking for:
- Logic errors and edge cases
- Code style consistency
- Missing tests or documentation
- Security vulnerabilities
- Performance implications

### Step 4: Reviewer provides feedback

The reviewer submits comments, which can be:
- **Inline comments** on specific lines in the diff
- **General comments** on the PR as a whole
- **Review summary** with a verdict: Approve, Request changes, or Comment only

```
  gh pr review 42 --request-changes --body "The calculate function \
  does not handle negative numbers. Please add input validation."
```

### Step 5: Author revises and pushes

The author makes the requested changes:

```bash
git switch feature/calculate
# Make changes
git add .
git commit -m "Add input validation for negative numbers"
git push origin feature/calculate
```

The new commit appears in the PR automatically. If "Dismiss stale reviews" is enabled, previous approvals are invalidated and reviewers must re-review.

### Step 6: Merge and clean up

When all reviewers approve and all status checks pass:

```bash
gh pr merge 42 --squash --delete-branch
```

This:
1. Squash-merges the PR into the base branch
2. Closes the PR
3. Deletes the feature branch on GitHub

The author then cleans up locally:

```bash
git switch main
git pull origin main
git branch -d feature/calculate
```

## When to Use This / When Not To

**Use a structured code review workflow when:**
- The change affects production code
- Multiple team members need to understand the change
- The change could have security or compliance implications
- You want to maintain code quality standards across the team

**Do not use a full review workflow when:**
- The change is a trivial one-line fix (typo, comment) вҖ” a simple comment or direct commit is fine
- You are prototyping or exploring and the code is throwaway
- The repository is a personal project with only you as the contributor
- An emergency hotfix needs to go out immediately вҖ” push first, review after

**Code review best practices:**
- Keep PRs small вҖ” under 400 lines is ideal; under 1000 is acceptable. Large PRs are harder to review effectively
- Provide context in the PR description: what the change does, why, and how it was tested
- Review for correctness first, style second. Automated linters can catch style issues
- Be specific in feedback: "Line 42: rename `x` to `user_count`" is better than "this needs better naming"
- Respond to review comments within one business day. Stale PRs create merge conflicts
- Use "Request changes" sparingly вҖ” it blocks the PR. Use "Comment" for optional suggestions
