You will create pre-commit, pre-push, and commit-msg hooks, test them, learn to bypass hooks, and configure shared hooks. Every command works on Windows (Git Bash) and macOS/Linux.

## Setup

```bash
mkdir -p ~/hooks-lab
cd ~/hooks-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create a file to practice with:

```bash
cat > app.py << 'EOF'
def calculate(a, b):
    print("debug: calculating...")
    result = a + b
    return result
EOF
git add app.py
git commit -m "Initial commit"
```

## The Walkthrough

### Step 1: Create a pre-commit hook that checks for debug statements

```bash
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/sh
echo "=== Pre-commit check ==="
if grep -n "print(" *.py 2>/dev/null; then
    echo "ERROR: Found print() statements. Remove them before committing."
    exit 1
fi
echo "OK: No debug print statements found."
exit 0
EOF
```

On Windows (if using Git Bash), make the script executable:

```bash
chmod +x .git/hooks/pre-commit
```

**Windows note:** If you are using PowerShell instead of Git Bash, create the hook as `.git/hooks/pre-commit.ps1` with PowerShell syntax. Git for Windows includes Git Bash, which handles `#!/bin/sh` hooks natively.

### Step 2: Test the hook — it should reject the commit

```bash
git add app.py
git commit -m "Add calculate function"
```

Expected output:

```
=== Pre-commit check ===
app.py:2:    print("debug: calculating...")
ERROR: Found print() statements. Remove them before committing.
```

The commit was aborted. The changes remain staged:

```bash
git status
```

Expected output — shows `app.py` staged, ready to commit.

### Step 3: Fix the code and commit successfully

```bash
cat > app.py << 'EOF'
def calculate(a, b):
    result = a + b
    return result
EOF
git add app.py
git commit -m "Add calculate function"
```

Expected output:

```
=== Pre-commit check ===
OK: No debug print statements found.
[main abc1234] Add calculate function
 1 file changed, 2 insertions(+), 2 deletions(-)
```

The hook passed, and the commit was created.

### Step 4: Create a commit-msg hook that enforces message format

```bash
cat > .git/hooks/commit-msg << 'EOF'
#!/bin/sh
# Enforce commit message format: must start with a capital letter
# and be at least 10 characters long.
MSG=$(cat "$1")
LEN=${#MSG}
FIRST_CHAR=$(echo "$MSG" | cut -c1)
if [ "$LEN" -lt 10 ]; then
    echo "ERROR: Commit message is too short ($LEN chars). Minimum 10 characters."
    exit 1
fi
if echo "$FIRST_CHAR" | grep -q '[a-z]'; then
    echo "ERROR: Commit message must start with a capital letter."
    exit 1
fi
echo "OK: Commit message format is valid."
exit 0
EOF
chmod +x .git/hooks/commit-msg
```

Test with a short message:

```bash
echo "test" > test.txt
git add test.txt
git commit -m "fix"
```

Expected output — the hook rejects the short message:

```
ERROR: Commit message is too short (3 chars). Minimum 10 characters.
```

Test with a valid message:

```bash
git commit -m "Add test file for validation"
```

Expected output — the hook passes and the commit succeeds.

### Step 5: Create a pre-push hook that protects main

```bash
cat > .git/hooks/pre-push << 'EOF'
#!/bin/sh
# Reject pushes to main or master
protected="main master"
while read local_ref local_oid remote_ref remote_oid; do
    branch=${remote_ref#refs/heads/}
    for p in $protected; do
        if [ "$branch" = "$p" ]; then
            echo "ERROR: Direct push to $branch is not allowed."
            echo "Create a pull request instead."
            exit 1
        fi
    done
done
exit 0
EOF
chmod +x .git/hooks/pre-push
```

Test the hook by setting up a local remote:

```bash
cd ..
git init --bare remote.git
cd hooks-lab
git remote add origin ../remote.git
git push origin main  # This should be REJECTED by the hook
```

Expected output:

```
ERROR: Direct push to main is not allowed.
Create a pull request instead.
error: failed to push some refs to '../remote.git'
```

The push was blocked. Push to a feature branch instead:

```bash
git push origin main:feature/test-branch
```

This pushes the main branch's content to `feature/test-branch` on the remote. The hook inspects the remote ref (`feature/test-branch`), which is not protected, so it passes.

### Step 6: Bypass hooks with `--no-verify`

Even with hooks active, you can bypass them:

```bash
echo "print('debug')" >> app.py
git add app.py
git commit --no-verify -m "short"
```

Expected output — the commit succeeds without hook checks:

```
[main def5678] short
 1 file changed, 1 insertion(+)
```

**Under the hood:** `--no-verify` (or `-n`) tells Git to skip the `pre-commit` and `commit-msg` hooks. It does not affect `pre-push` — to bypass pre-push, use `--no-verify` with `git push`.

### Step 7: Configure shared hooks

Share hooks with your team by storing them in the repository:

```bash
mkdir -p .githooks
cp .git/hooks/pre-commit .githooks/
cp .git/hooks/commit-msg .githooks/
git add .githooks/
git commit -m "Add shared hooks"
```

Configure Git to use the shared hooks directory:

```bash
git config core.hooksPath .githooks
```

Verify:

```bash
git config core.hooksPath
```

Expected output:

```
.githooks
```

Now all hook scripts from `.githooks/` are active. New developers only need to run:

```bash
git config core.hooksPath .githooks
```

Or make it the default for all clones:

```bash
# Each developer runs this once
git config --global core.hooksPath ~/.githooks  # For global hooks
# OR per-repo: git config core.hooksPath .githooks
```

### Step 8: Create a hook that prevents committing large files

```bash
cat > .githooks/pre-commit << 'EOF'
#!/bin/sh
echo "=== Size check ==="
MAX_SIZE=100  # KB
for file in $(git diff --cached --name-only); do
    if [ -f "$file" ]; then
        size=$(du -k "$file" | cut -f1)
        if [ "$size" -gt "$MAX_SIZE" ]; then
            echo "ERROR: $file is ${size}KB (max ${MAX_SIZE}KB)"
            exit 1
        fi
    fi
done
echo "OK: All files under size limit."
exit 0
EOF
chmod +x .githooks/pre-commit
```

Test:

```bash
dd if=/dev/urandom of=bigfile.bin bs=1024 count=200 2>/dev/null
git add bigfile.bin
git commit -m "Add big file"
```

Expected output — the hook rejects the large file:

```
ERROR: bigfile.bin is 200KB (max 100KB)
```

## Common Errors at This Stage

### Error 1: `fatal: cannot run .git/hooks/pre-commit: No such file or directory`

Git found the hook file but cannot execute it. The shebang line is missing or the file is not executable.

**Fix:** Ensure the first line is `#!/bin/sh` (or `#!/usr/bin/env bash`) and the file is executable: `chmod +x .git/hooks/pre-commit`. On Windows with Git Bash, `chmod` works within the Git Bash environment.

### Error 2: Hook runs but passes when it should fail

The hook script has a logic bug — it does not properly detect the condition it was meant to check. This is the most common hook problem because hooks run silently unless they explicitly print output.

**Fix:** Add `echo` statements to debug the hook. Test the hook script directly by running it standalone: `bash .git/hooks/pre-commit`. Check what the hook's working directory is — hooks run from the repository root, not from the current directory.

### Error 3: `git commit --no-verify` still runs hooks

You are confusing `--no-verify` with something else. `--no-verify` (short: `-n`) skips `pre-commit` and `commit-msg` hooks only. For `pre-push`, use `git push --no-verify`.

**Fix:** Use the correct flag for the hook type you want to skip. For all hooks, there is no single "skip all" flag — you must use the specific flag for each Git command.

### Error 4: After configuring `core.hooksPath`, existing hooks in `.git/hooks/` are ignored

When `core.hooksPath` is set, Git ignores `.git/hooks/` entirely. This is by design — the setting overrides the default hooks directory.

**Fix:** Either remove `core.hooksPath` (`git config --unset core.hooksPath`) and use `.git/hooks/`, or move all hooks to the custom directory. If you want both, you cannot — choose one location and stick with it.

## What You Just Learned

- Git hooks are executable scripts in `.git/hooks/` that run at specific lifecycle events
- `pre-commit` runs before each commit — use for linting, secret detection, size checks
- `commit-msg` validates commit message format
- `pre-push` runs before each push — use for branch protection, test running
- `--no-verify` (or `-n`) bypasses hooks — hooks are convenience, not security
- Shared hooks via `.githooks/` directory + `core.hooksPath` configuration
- Hooks are local to each clone and not pushed by default
- Slow hooks frustrate developers — keep checks fast and move heavy checks to CI
