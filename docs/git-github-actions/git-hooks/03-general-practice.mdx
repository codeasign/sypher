## Tier 1 — Beginner

### Exercise 1

**Task:** Create a pre-commit hook that rejects commits containing `TODO` comments. Test it by committing a file with `# TODO: fix this`. Verify the commit is blocked.

<details>
<summary>Hint</summary>
Use `grep` in the hook to check staged files: `git diff --cached --name-only` lists staged files, then `grep "TODO"` each file.
</details>

### Solution

```bash
mkdir -p ~/hooks-beginner-1 && cd ~/hooks-beginner-1
git init
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/sh
for file in $(git diff --cached --name-only); do
  if [ -f "$file" ] && grep -n "TODO" "$file" 2>/dev/null; then
    echo "ERROR: $file contains TODO — resolve before committing"
    exit 1
  fi
done
exit 0
EOF
chmod +x .git/hooks/pre-commit
echo "# TODO: fix this later" > app.py
git add app.py
git commit -m "Add app"  # Should be rejected
```

### Exercise 2

**Task:** Create a pre-push hook that prevents force-pushing (rejects `--force`). Test by attempting to force-push.

<details>
<summary>Hint</summary>
The pre-push hook receives stdin with ref info. Check if `--force` was used by examining the `GIT_PUSH_OPTION_COUNT` or by checking if the local ref is `0000000000000000000000000000000000000000` (delete ref).
</details>

### Solution

```bash
mkdir -p ~/hooks-beginner-2 && cd ~/hooks-beginner-2
git init && echo "init" > f.txt && git add f.txt && git commit -m "Init"
git init --bare remote.git && git remote add origin ../remote.git
git push origin main
# Force push is allowed by default — but the hook can detect it
cat > .git/hooks/pre-push << 'EOF'
#!/bin/sh
while read local_ref local_oid remote_ref remote_oid; do
  # Detect force push: the remote_oid would be different than expected
  # or the force flag would be set
  if [ "$local_oid" = "0000000000000000000000000000000000000000" ]; then
    echo "ERROR: Force push detected and rejected by hook"
    exit 1
  fi
done
exit 0
EOF
chmod +x .git/hooks/pre-push
git push origin main --force 2>&1 || echo "Force push blocked"
```

### Exercise 3

**Task:** Create a post-commit hook that logs every commit to a file `.commit-log` with the timestamp, hash, and message. Make 3 commits and verify the log file contains all 3 entries.

<details>
<summary>Hint</summary>
`git log -1 --format="%H %s %ai"` gives the hash, subject, and author date. Append to a file with `>> .commit-log`.
</details>

### Solution

```bash
mkdir -p ~/hooks-beginner-3 && cd ~/hooks-beginner-3
git init
cat > .git/hooks/post-commit << 'EOF'
#!/bin/sh
git log -1 --format="%H %s %ai" >> .commit-log
EOF
chmod +x .git/hooks/post-commit
echo "A" > f.txt && git add f.txt && git commit -m "Commit A"
echo "B" >> f.txt && git add f.txt && git commit -m "Commit B"
echo "C" >> f.txt && git add f.txt && git commit -m "Commit C"
cat .commit-log
```

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a pre-commit hook that runs a linter (e.g., `ruff` or `pylint` or even `python -m py_compile`) on staged Python files. If the linter fails, the commit is aborted. If no Python files are staged, the hook passes silently.

<details>
<summary>Hint</summary>
Use `git diff --cached --name-only --diff-filter=ACM | grep '\.py$'` to find staged Python files. Run `python -m py_compile` on each to check syntax.
</details>

### Solution

```bash
mkdir -p ~/hooks-inter-1 && cd ~/hooks-inter-1
git init
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/sh
FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.py$')
if [ -z "$FILES" ]; then exit 0; fi
echo "Checking Python syntax..."
echo "$FILES" | while read file; do
  python -m py_compile "$file" 2>&1
  if [ $? -ne 0 ]; then
    echo "ERROR: $file has syntax errors"
    exit 1
  fi
done
exit 0
EOF
chmod +x .git/hooks/pre-commit
# Test with valid Python
echo "x = 1" > good.py && git add good.py && git commit -m "Good"
# Test with invalid Python
echo "x = " > bad.py && git add bad.py && git commit -m "Bad" 2>&1 || echo "Commit blocked"
```

### Exercise 2

**Task:** Set up shared hooks in `.githooks/` and configure a new repository clone to use them automatically via a setup script. Create three hooks: pre-commit (lint), commit-msg (format), and post-checkout (notify). Test that all three execute.

<details>
<summary>Hint</summary>
Create `.githooks/` directory with the hooks. Create a `setup.sh` that runs `git config core.hooksPath .githooks`. Test each hook with the appropriate Git command.
</details>

### Solution

```bash
mkdir -p ~/hooks-inter-2 && cd ~/hooks-inter-2
git init
mkdir -p .githooks
cat > .githooks/pre-commit << 'EOF'
#!/bin/sh
echo "[hook] pre-commit: checking for tabs..."
if git diff --cached | grep -E '^\+\t'; then
  echo "ERROR: Tabs detected — use spaces"
  exit 1
fi
exit 0
EOF
cat > .githooks/commit-msg << 'EOF'
#!/bin/sh
echo "[hook] commit-msg: checking length..."
MSG=$(cat "$1")
if [ ${#MSG} -lt 5 ]; then
  echo "ERROR: Message too short"
  exit 1
fi
exit 0
EOF
cat > .githooks/post-checkout << 'EOF'
#!/bin/sh
echo "[hook] post-checkout: Switched to $(git rev-parse --abbrev-ref HEAD)"
exit 0
EOF
chmod +x .githooks/*
cat > setup.sh << 'EOF'
#!/bin/sh
git config core.hooksPath .githooks
echo "Hooks configured. Run 'git commit' to test."
EOF
chmod +x setup.sh
./setup.sh
echo "test" > f.txt && git add f.txt && git commit -m "Test commit"
git checkout -b test-branch
```

### Exercise 3

**Task:** Create a pre-commit hook that checks for large files (>1MB) in the staged changes. If found, it lists the files and their sizes, then rejects the commit. Test by creating a 2MB file and verifying the hook blocks it.

<details>
<summary>Hint</summary>
Use `git diff --cached --name-only` to get staged file names. Use `wc -c` or `ls -l` to check size. Compare against 1048576 bytes (1MB).
</details>

### Solution

```bash
mkdir -p ~/hooks-inter-3 && cd ~/hooks-inter-3
git init
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/sh
MAX=1048576  # 1MB in bytes
FILES=$(git diff --cached --name-only)
for f in $FILES; do
  if [ -f "$f" ]; then
    SIZE=$(wc -c < "$f")
    if [ "$SIZE" -gt "$MAX" ]; then
      echo "ERROR: $f is ${SIZE} bytes (max $MAX)"
      exit 1
    fi
  fi
done
exit 0
EOF
chmod +x .git/hooks/pre-commit
dd if=/dev/zero of=small.bin bs=1024 count=500 2>/dev/null
git add small.bin && git commit -m "Small file"  # Should pass
dd if=/dev/zero of=big.bin bs=1024 count=2000 2>/dev/null
git add big.bin && git commit -m "Big file" 2>&1 || echo "Blocked as expected"
```

## Tier 3 — Advanced

### Exercise 1: Commit message enforcing conventional commits

**Scenario:** Your team uses [Conventional Commits](https://www.conventionalcommits.org/): `type(scope): description`. Types must be one of `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`. The message must have the colon and space after the type/scope.

**Task:** Write a `commit-msg` hook that validates the message matches the conventional commits format. The hook should:
1. Check that the message starts with a valid type
2. Optionally allow a scope in parentheses after the type
3. Require a colon and space after the type(scope)
4. Require at least 5 characters of description after the prefix
5. Allow merge commits (those starting with "Merge") to bypass validation

<details>
<summary>Hint</summary>
Use `grep` or `sed` with a regex pattern: `^(feat|fix|docs|style|refactor|test|chore)(\([^)]+\))?:\s.{5,}`. Skip validation if the message starts with "Merge".
</details>

### Solution — one valid approach

```bash
mkdir -p ~/hooks-adv-1 && cd ~/hooks-adv-1
git init
cat > .git/hooks/commit-msg << 'EOF'
#!/bin/sh
MSG=$(cat "$1")
# Allow merge commits
echo "$MSG" | grep -q "^Merge" && exit 0
# Conventional commit pattern
PATTERN='^(feat|fix|docs|style|refactor|test|chore)(\([^)]+\))?:\s.{5,}'
if echo "$MSG" | grep -qE "$PATTERN"; then
  exit 0
else
  echo "ERROR: Commit message does not follow Conventional Commits format."
  echo "Format: type(scope): description"
  echo "Types: feat, fix, docs, style, refactor, test, chore"
  echo "Example: feat(api): add user authentication endpoint"
  exit 1
fi
EOF
chmod +x .git/hooks/commit-msg
echo "test" > f.txt && git add f.txt
git commit -m "feat: add new feature"  # Should pass
git commit -m "fix(parser): handle edge case"  # Should pass
git commit -m "bad message"  # Should fail
git commit -m "Merge branch 'feature'"  # Should pass (merge exempt)
```

**Alternative approach:** Use a community tool like `commitlint` with a husky hook. This gives better error messages and supports additional configuration files (`commitlint.config.js`).

### Exercise 2: Branch naming enforcement with pre-push hook

**Scenario:** Your team has a branch naming convention: feature branches must start with `feature/`, bugfix branches with `fix/`, and hotfix branches with `hotfix/`. Direct pushes to `main`, `develop`, or `release/*` are only allowed via pull requests.

**Task:** Write a pre-push hook that:
1. Rejects pushes to `main` and `develop` entirely
2. Rejects pushes to `release/*` unless the pusher has admin access (simulate this with an environment variable)
3. Checks that new branches (branches that do not exist on the remote yet) follow the naming convention
4. Allows pushes to branches that already exist on the remote (existing work)

<details>
<summary>Hint</summary>
The pre-push hook receives stdin with lines: `<local-ref> <local-oid> <remote-ref> <remote-oid>`. If `remote-oid` is all zeros (`0000...`), the branch does not exist on the remote (new branch). Check branch names with `case` patterns.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/hooks-adv-2 && cd ~/hooks-adv-2
git init && echo "init" > f.txt && git add f.txt && git commit -m "Init"
git init --bare remote.git && git remote add origin ../remote.git
git push origin main
git checkout -b develop
git push origin develop
git checkout main
cat > .git/hooks/pre-push << 'EOF'
#!/bin/sh
ALLOW_RELEASE=${ALLOW_RELEASE_PUSH:-false}
while read local_ref local_oid remote_ref remote_oid; do
  branch=${remote_ref#refs/heads/}
  # Always reject main and develop
  if [ "$branch" = "main" ] || [ "$branch" = "develop" ]; then
    echo "ERROR: Direct push to '$branch' is not allowed. Use a PR."
    exit 1
  fi
  # Reject release/* unless ALLOW_RELEASE_PUSH is set
  if echo "$branch" | grep -q "^release/" && [ "$ALLOW_RELEASE" != "true" ]; then
    echo "ERROR: Push to '$branch' requires admin. Set ALLOW_RELEASE_PUSH=true to override."
    exit 1
  fi
  # Check naming convention for new branches
  if [ "$remote_oid" = "0000000000000000000000000000000000000000" ]; then
    case "$branch" in
      feature/*|fix/*|hotfix/*) ;;
      main|develop|release/*) ;;
      *)
        echo "ERROR: New branch '$branch' does not follow naming convention."
        echo "Use: feature/..., fix/..., hotfix/..."
        exit 1
        ;;
    esac
  fi
done
exit 0
EOF
chmod +x .git/hooks/pre-push
git push origin main 2>&1 || echo "Blocked as expected"
git checkout -b my-bad-branch
git push origin my-bad-branch 2>&1 || echo "Blocked as expected"
git branch -m my-bad-branch feature/good-branch
git push origin feature/good-branch
```

**Alternative approach:** Use server-side hooks (GitHub Actions or githooks server) for enforcement that cannot be bypassed. Client-side hooks can be skipped with `git push --no-verify`.

### Exercise 3: Auto-generate changelog with post-commit hook

**Scenario:** Your team wants a `CHANGELOG.md` that automatically appends each commit in the conventional commits format. The file should be updated after each commit to `main` (but not on feature branches).

**Task:** Create a `post-commit` hook that:
1. Detects whether the current branch is `main`
2. If on `main`, extracts the last commit message
3. Appends it to `CHANGELOG.md` with a timestamp
4. Stages and commits the changelog update automatically (be careful — this can cause infinite loops)
5. Uses a marker to avoid re-triggering itself

<details>
<summary>Hint</summary>
Check `git rev-parse --abbrev-ref HEAD`. Use `git log -1 --format="%s %ai"`. To avoid infinite loops, add a `[skip changelog]` marker in the auto-commit message and check for it in the hook.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/hooks-adv-3 && cd ~/hooks-adv-3
git init && echo "init" > f.txt && git add f.txt && git commit -m "Initial commit"
cat > .git/hooks/post-commit << 'EOF'
#!/bin/sh
BRANCH=$(git rev-parse --abbrev-ref HEAD)
if [ "$BRANCH" != "main" ]; then exit 0; fi
MSG=$(git log -1 --format="%s")
# Avoid infinite loop — skip if this commit was the changelog update
echo "$MSG" | grep -q "\[skip changelog\]" && exit 0
# Extract type and description
TYPE=$(echo "$MSG" | cut -d: -f1)
DESC=$(echo "$MSG" | cut -d: -f2- | xargs)
DATE=$(git log -1 --format="%ai" | cut -d' ' -f1)
ENTRY="- $DATE [$TYPE] $DESC"
# Append to changelog
echo "$ENTRY" >> CHANGELOG.md
git add CHANGELOG.md
git commit --no-verify -m "docs: update changelog [skip changelog]"
EOF
chmod +x .git/hooks/post-commit
echo "A" >> f.txt && git add f.txt && git commit -m "feat: add feature A"
echo "B" >> f.txt && git add f.txt && git commit -m "fix: resolve bug B"
cat CHANGELOG.md
```

The changelog contains:

```
- 2026-07-03 [feat] add feature A
- 2026-07-03 [fix] resolve bug B
```

**Alternative approach:** Use a GitHub Actions workflow on `push` to `main` that generates the changelog from commit history. This avoids the complexity of post-commit hooks and infinite loop prevention. The workflow approach is more robust for team environments.
