You will create a repository with divergent branches, perform a rebase, observe the rewritten history, handle a rebase conflict, and abort a rebase. Every command works identically on Windows, macOS, and Linux.

## Setup

```bash
mkdir -p ~/git-rebase-lab
cd ~/git-rebase-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create the base history:

```bash
echo "Line 1" > app.py
echo "Line 2" >> app.py
echo "Line 3" >> app.py
git add app.py
git commit -m "Initial commit"
echo "Line 4" >> app.py
git add app.py
git commit -m "Add line 4"
echo "Line 5" >> app.py
git add app.py
git commit -m "Add line 5"
```

Expected output:

```
[main (root-commit) a1b2c3d] Initial commit
[main c3d4e5f] Add line 4
[main d4e5f6a] Add line 5
```

## The Walkthrough

### Step 1: Create a feature branch with commits

```bash
git switch -c feature-rebase
sed -i 's/Line 3/Line 3 - modified by feature/' app.py
git add app.py
git commit -m "Update line 3 in feature"
echo "Feature line" >> app.py
git add app.py
git commit -m "Add feature line at end"
git log --oneline
```

Expected output:

```
e5f6a7b (HEAD -> feature-rebase) Add feature line at end
f6a7b8c Update line 3 in feature
d4e5f6a (main) Add line 5
c3d4e5f Add line 4
a1b2c3d Initial commit
```

```
  ((a1b2c3d)) ──▶ ((c3d4e5f)) ──▶ ((d4e5f6a))  ← main
                                                │
                                                └──▶ ((f6a7b8c)) ──▶ ((e5f6a7b))  ← feature-rebase
                                                                                   ← HEAD
```

### Step 2: Add new commits to main (creating divergence)

```bash
git switch main
echo "Main update A" >> app.py
git add app.py
git commit -m "Main update A"
echo "Main update B" >> app.py
git add app.py
git commit -m "Main update B"
git log --oneline
```

Expected output:

```
g7h8i9j (HEAD -> main) Main update B
h8i9j0k Main update A
d4e5f6a Add line 5
c3d4e5f Add line 4
a1b2c3d Initial commit
```

Now the branches have diverged further:

```
  ((d4e5f6a)) ──▶ ((h8i9j0k)) ──▶ ((g7h8i9j))  ← main
       │                                        ← HEAD
       └──▶ ((f6a7b8c)) ──▶ ((e5f6a7b))  ← feature-rebase
```

**Checkpoint:** `git log --oneline --graph --all` shows the fork shape.

### Step 3: Rebase the feature branch onto main

```bash
git switch feature-rebase
git rebase main
```

Expected output:

```
Auto-merging app.py
CONFLICT (content): Merge conflict in app.py
error: could not apply f6a7b8c... Update line 3 in feature
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
hint: You can instead skip this commit: run "git rebase --skip".
hint: To abort and get back to the state before "git rebase", run "git rebase --abort".
Could not apply f6a7b8c... Update line 3 in feature
```

A conflict occurred because both branches modified `app.py`. This is expected — rebase replays each commit as a patch, and patches can conflict just like merges.

### Step 4: Resolve the rebase conflict

View the conflict:

```bash
cat app.py
```

You see conflict markers around the part where both branches modified the file. Resolve by editing the file to keep the feature's line 3 change and main's additional lines:

```bash
# Accept the feature's version of line 3, keep main's new lines
sed -i 's/<<<<<<< HEAD/<<<<<<< HEAD\n# Keeping feature version/' app.py 2>/dev/null || true
# Instead, manually edit: replace the conflicted region
cat > app.py << 'EOF'
Line 1
Line 2
Line 3 - modified by feature
Line 4
Line 5
Main update A
Main update B
EOF
```

Stage the resolved file and continue:

```bash
git add app.py
git rebase --continue
```

Git opens your editor with a default commit message. Save and close:

```
[detached HEAD abc1234] Update line 3 in feature
 1 file changed, 1 insertion(+), 1 deletion(-)
```

The rebase continues to the next commit:

```
Auto-merging app.py
[detached HEAD def5678] Add feature line at end
 1 file changed, 1 insertion(+)
Successfully rebased and updated refs/heads/feature-rebase.
```

### Step 5: Verify the rebased history

```bash
git log --oneline
```

Expected output:

```
def5678 (HEAD -> feature-rebase) Add feature line at end
abc1234 Update line 3 in feature
g7h8i9j (main) Main update B
h8i9j0k Main update A
d4e5f6a Add line 5
c3d4e5f Add line 4
a1b2c3d Initial commit
```

The feature commits now appear AFTER main's commits, with NEW hashes (`def5678`, `abc1234`). The history is linear — no merge bubble.

```
  ((d4e5f6a)) ──▶ ((h8i9j0k)) ──▶ ((g7h8i9j)) ──▶ ((abc1234)) ──▶ ((def5678))  ← feature-rebase
                                                                                ← HEAD
```

**Under the hood:** The original commits `f6a7b8c` and `e5f6a7b` still exist in Git's object store but are no longer reachable from any branch. They will be removed by `git gc` eventually.

### Step 6: Aborting a rebase (practice)

Create a throwaway scenario to practice aborting:

```bash
git switch main
git switch -c abort-practice
echo "Conflict trigger" >> app.py
git add app.py
git commit -m "Change for abort practice"
git switch main
echo "Conflict trigger main" >> app.py
git add app.py
git commit -m "Main change for abort"
git switch abort-practice
git rebase main
```

When the conflict appears, abort:

```bash
git rebase --abort
```

Expected output — no error, you are back on `abort-practice` at the pre-rebase state:

```
git status
On branch abort-practice
nothing to commit, working tree clean
```

**Under the hood:** `git rebase --abort` deleted `.git/rebase-apply/` and restored HEAD, the index, and the working tree to match the pre-rebase commit.

### Step 7: Fast-forward merge after rebase

Now that `feature-rebase` is on top of `main`, merging is a simple fast-forward:

```bash
git switch main
git merge feature-rebase
```

Expected output:

```
Updating g7h8i9j..def5678
Fast-forward
 app.py | 2 ++
 1 file changed, 2 insertions(+)
```

No merge commit. The history stays linear:

```
git log --oneline --graph
* def5678 (HEAD -> main, feature-rebase) Add feature line at end
* abc1234 Update line 3 in feature
* g7h8i9j Main update B
* h8i9j0k Main update A
* d4e5f6a Add line 5
* c3d4e5f Add line 4
* a1b2c3d Initial commit
```

Linear history from start to finish. The feature's commits are in the record, but there is no trace of the branch that created them.

### Step 8: Using `git pull --rebase` (simulated)

Instead of `git pull` (which merges by default), you can rebase your local commits on top of fetched remote changes:

```bash
git config pull.rebase true
```

This sets the default for your current repository. Now `git pull` will behave like `git fetch` + `git rebase` instead of `git fetch` + `git merge`. You can verify the setting:

```bash
git config pull.rebase
```

Expected output:

```
true
```

## Common Errors at This Stage

### Error 1: `Cannot rebase: You have unstaged changes.`

You tried to rebase with uncommitted changes in the working directory. Git refuses because the rebase would overwrite those changes when it resets HEAD to the new base.

**Fix:** Commit your changes with `git add` + `git commit`, or stash them temporarily with `git stash push`, then run the rebase. After the rebase, run `git stash pop` to restore your changes.

### Error 2: `The following untracked working tree files would be overwritten by checkout:`

A file in your working directory exists as a tracked file in the target branch. Git refuses to overwrite it.

**Fix:** Move or delete the untracked file, or add it to `.gitignore`. Then re-run the rebase.

### Error 3: `Cannot rebase onto multiple branches`

You tried to use `git rebase` with a range that includes commits from multiple branches. Rebase works on a single linear sequence of commits.

**Fix:** Use `git rebase --onto <target> <upstream>` to specify exactly which commits to replay. For example, `git rebase --onto main feature-base` replays only commits that are in the current branch but not in `feature-base`.

### Error 4: After rebase, your branch is behind the remote

You rebased a branch that was already pushed to a remote, and now `git push` is rejected:

```
! [rejected]        feature-rebase -> feature-rebase (non-fast-forward)
```

**Fix:** You need to force-push: `git push --force-with-lease feature-rebase`. The `--force-with-lease` flag is safer than `--force` because it checks that your remote-tracking branch matches what you expect before overwriting. Only do this if you are the only person working on this branch.

## What You Just Learned

- `git rebase <base>` replays the current branch's commits on top of `<base>`, creating new commits with new hashes
- Rebasing produces a linear history with no merge bubbles
- Conflicts during rebase are resolved the same way as merge conflicts: edit, `git add`, `git rebase --continue`
- `git rebase --abort` returns to the pre-rebase state safely
- `git rebase --skip` discards the current commit during the rebase
- After rebasing, merging is a fast-forward — no merge commit needed
- `git pull --rebase` or `git config pull.rebase true` keeps your local history linear when pulling
- Never rebase commits that have been pushed to a shared branch
