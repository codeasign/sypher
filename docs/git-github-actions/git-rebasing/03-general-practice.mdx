Work through these exercises in a terminal. Tier 1 builds single-command muscle memory. Tier 2 combines rebasing with prior Git skills. Tier 3 presents real-world scenarios with multiple valid approaches.

## Tier 1 — Beginner

### Exercise 1

**Task:** Create a repo with 3 commits on `main`. Create a branch `feature` with 2 commits without diverging `main` further. Rebase `feature` onto `main`. What happens?

<details>
<summary>Hint</summary>
If `feature` is already a direct descendant of `main`'s tip, rebase has nothing to do — it reports "Current branch feature is up to date."
</details>

### Solution

```bash
mkdir -p ~/rebase-beginner-1 && cd ~/rebase-beginner-1
git init
echo "1" > f.txt && git add f.txt && git commit -m "1"
echo "2" > f.txt && git add f.txt && git commit -m "2"
echo "3" > f.txt && git add f.txt && git commit -m "3"
git switch -c feature
echo "A" > f.txt && git add f.txt && git commit -m "A"
echo "B" > f.txt && git add f.txt && git commit -m "B"
git rebase main
```

Git reports "Current branch feature is up to date" because `feature` is already a descendant of `main` — no commits to replay.

### Exercise 2

**Task:** Create a repo where `main` has 3 commits and `feature` branches off at commit 2 with 2 commits. Add 1 more commit to `main`. Rebase `feature` onto `main`. Verify the feature commits now appear after main's latest commit in the log.

<details>
<summary>Hint</summary>
Use `git log --oneline feature` after the rebase. The feature commits should have new hashes and appear after main's latest commit.
</details>

### Solution

```bash
mkdir -p ~/rebase-beginner-2 && cd ~/rebase-beginner-2
git init
echo "1" > f.txt && git add f.txt && git commit -m "Commit 1"
echo "2" > f.txt && git add f.txt && git commit -m "Commit 2"
echo "3" > f.txt && git add f.txt && git commit -m "Commit 3"
git switch -c feature
echo "A" > f.txt && git add f.txt && git commit -m "Feature A"
echo "B" > f.txt && git add f.txt && git commit -m "Feature B"
git switch main
echo "4" > f.txt && git add f.txt && git commit -m "Commit 4"
git switch feature
git rebase main
git log --oneline
```

The feature commits (`Feature A`, `Feature B`) now appear after `Commit 4` in the log.

### Exercise 3

**Task:** Rebase a branch, trigger a conflict, and abort the rebase. Verify you are back to the pre-rebase state.

<details>
<summary>Hint</summary>
Modify the same line of the same file on both branches. Run `git rebase main`. When the conflict appears, run `git rebase --abort`.
</details>

### Solution

```bash
mkdir -p ~/rebase-beginner-3 && cd ~/rebase-beginner-3
git init && echo "same line" > conflict.txt && git add conflict.txt && git commit -m "Base"
echo "main version" > conflict.txt && git add conflict.txt && git commit -m "Main change"
git switch -c feature -b feature HEAD~1
echo "feature version" > conflict.txt && git add conflict.txt && git commit -m "Feature change"
git rebase main
# Conflict appears
git rebase --abort
git status
git log --oneline
```

`git status` shows "nothing to commit, working tree clean." The log shows the two pre-rebase commits.

### Exercise 4

**Task:** Use `git rebase --skip` to skip a commit during a rebase. Verify the skipped commit's changes are not in the rebased branch.

<details>
<summary>Hint</summary>
Create a scenario where a commit on your branch causes a conflict that you want to discard entirely. Instead of resolving, use `git rebase --skip`.
</details>

### Solution

```bash
mkdir -p ~/rebase-beginner-4 && cd ~/rebase-beginner-4
git init
echo "base" > f.txt && git add f.txt && git commit -m "Base"
echo "main line" >> f.txt && git add f.txt && git commit -m "Main"
git switch -c feature HEAD~1
echo "feature line 1" >> f.txt && git add f.txt && git commit -m "Feature 1"
echo "feature line 2" >> f.txt && git add f.txt && git commit -m "Feature 2"
git rebase main
# Conflict on Feature 1
git rebase --skip   # Skip Feature 1
git rebase --continue  # Feature 2 should apply cleanly
cat f.txt
```

`f.txt` contains "base", "main line", and "feature line 2" — but NOT "feature line 1". The skipped commit was discarded entirely.

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a repo with 3 branches: `main`, `feature-a` (2 commits), and `feature-b` (1 commit, based on `feature-a`). Rebase `feature-b` directly onto `main` using `git rebase --onto`, skipping `feature-a`. Verify `feature-b`'s commit now sits on top of `main` without including `feature-a`'s changes.

<details>
<summary>Hint</summary>
Use `git rebase --onto main feature-a` when on `feature-b`. This replays commits in `feature-b` that are not in `feature-a`, directly onto `main`.
</details>

### Solution

```bash
mkdir -p ~/rebase-inter-1 && cd ~/rebase-inter-1
git init && echo "base" > f.txt && git add f.txt && git commit -m "Base"
git switch -c feature-a
echo "A1" >> f.txt && git add f.txt && git commit -m "Feature A - 1"
echo "A2" >> f.txt && git add f.txt && git commit -m "Feature A - 2"
git switch -c feature-b
echo "B1" >> f.txt && git add f.txt && git commit -m "Feature B - 1"
git switch main
echo "main update" >> f.txt && git add f.txt && git commit -m "Main update"
git switch feature-b
git rebase --onto main feature-a
git log --oneline
```

The log shows only "Feature B - 1" on top of "Main update". Feature A's commits are not included.

### Exercise 2

**Task:** Rebase a branch, then compare the `git log` output before and after. Note that the commit hashes change but the commit messages, authors, and diffs are identical. Use `git show` on a specific commit before and after to confirm the diff is the same but the hash differs.

<details>
<summary>Hint</summary>
Before rebasing, capture the commit hash with `git log --format="%H" -1`. After rebasing, find the same commit by message and compare with `git diff <old-hash> <new-hash>`.
</details>

### Solution

```bash
mkdir -p ~/rebase-inter-2 && cd ~/rebase-inter-2
git init
echo "1" > f.txt && git add f.txt && git commit -m "C1"
echo "2" > f.txt && git add f.txt && git commit -m "C2"
git switch -c feature
echo "F1" > f.txt && git add f.txt && git commit -m "Feature commit"
OLD_HASH=$(git log --format="%H" -1)
git switch main
echo "3" > f.txt && git add f.txt && git commit -m "C3"
git switch feature
git rebase main
NEW_HASH=$(git log --format="%H" -1)
echo "Old hash: $OLD_HASH"
echo "New hash: $NEW_HASH"
git diff $OLD_HASH $NEW_HASH
```

The diff output is empty (no differences in content), proving the hashes differ but the changes are identical.

### Exercise 3

**Task:** Configure `pull.rebase` to `true` in a repo, then simulate a pull scenario by fetching from a local remote. Confirm that `git pull` rebases instead of merging.

<details>
<summary>Hint</summary>
Create a bare repo as a "remote", push to it, clone it, make changes in both clones, then pull in one clone with `pull.rebase` set.
</details>

### Solution

```bash
mkdir -p ~/rebase-inter-3 && cd ~/rebase-inter-3
git init --bare remote.git
git init local && cd local
echo "start" > f.txt && git add f.txt && git commit -m "Start"
git remote add origin ../remote.git
git push -u origin main
cd ..
git clone local clone2
cd local
echo "local change" >> f.txt && git add f.txt && git commit -m "Local change"
cd ../clone2
echo "clone2 change" >> f.txt && git add f.txt && git commit -m "Clone2 change"
git push origin main
cd ../local
git config pull.rebase true
git pull origin main
git log --oneline --graph
```

The graph is linear — no merge commit. Without `pull.rebase true`, the pull would have created a merge bubble.

## Tier 3 — Advanced

### Exercise 1: Rebase a long-running feature branch across multiple upstream releases

**Scenario:** Your team has `main`, `release/v1`, and `release/v2`. Your feature branch was created from `release/v1` and has 10 commits. Now `release/v2` is the active line, and you need to rebase your feature onto `release/v2` — but only the 5 most recent commits belong to the feature; the first 5 were already released in `v1` and should not be replayed.

**Task:** Use `git rebase --onto` to rebase only the 5 commits that are not in `release/v1` onto `release/v2`. Verify the rebased branch has exactly 5 new commits on top of `release/v2`.

<details>
<summary>Hint</summary>
Find the merge base between `feature` and `release/v1` with `git merge-base`. Then `git rebase --onto release/v2 <merge-base>`.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/rebase-adv-1 && cd ~/rebase-adv-1
git init
for i in 1 2 3; do echo "$i" > f.txt && git add f.txt && git commit -m "Main $i"; done
git switch -c release/v1
for i in 1 2; do echo "v1-$i" > f.txt && git add f.txt && git commit -m "V1 release $i"; done
git switch main
echo "4" > f.txt && git add f.txt && git commit -m "Main 4"
git switch -c release/v2
for i in 1 2; do echo "v2-$i" > f.txt && git add f.txt && git commit -m "V2 release $i"; done
git switch main
git switch -c feature
# 5 commits on top of release/v1
for i in 1 2 3 4 5; do echo "feature-$i" > f.txt && git add f.txt && git commit -m "Feature $i"; done
# Now rebase only the last 5 commits onto release/v2
BASE=$(git merge-base feature release/v1)
git rebase --onto release/v2 $BASE
git log --oneline
```

The log shows exactly 5 feature commits on top of release/v2's commits. The V1 release commits are not included.

**Alternative approach:** Use `git rebase --onto release/v2 release/v1` — since `release/v1` is the upstream branch from which the feature diverged, this replays everything in `feature` that is not in `release/v1` directly onto `release/v2`.

### Exercise 2: Recover from a mistaken force-push after rebase

**Scenario:** You rebased a shared branch (`team-feature`) and force-pushed it. Another developer pulled the old version and made 2 commits on top of it. Now the remote has your rebased version, and the other developer has a divergent history. You need to help them rebase their work onto the new remote without losing their commits.

**Task:** Simulate this scenario: two local clones share a remote. One developer rebases and force-pushes. The other developer needs to rebase their local commits on top of the rewritten remote. Write the commands the second developer would run.

<details>
<summary>Hint</summary>
The second developer can use `git fetch` to get the new remote, then `git rebase --onto origin/team-feature team-feature@{1}` to replay their local commits on top of the rewritten remote. Or, simpler: `git fetch` then `git rebase origin/team-feature`.
</details>

### Solution — one valid approach

Setup (run in one terminal):

```bash
mkdir -p ~/rebase-adv-2 && cd ~/rebase-adv-2
git init --bare remote.git
git clone remote.git dev1 && cd dev1
echo "start" > f.txt && git add f.txt && git commit -m "Start"
git push -u origin main
git switch -c team-feature
echo "shared work" > f.txt && git add f.txt && git commit -m "Shared commit"
git push -u origin team-feature
cd ..
git clone remote.git dev2
```

Now in dev1, the rebase + force-push:

```bash
cd dev1
git switch main
echo "upstream" > f.txt && git add f.txt && git commit -m "Upstream change"
git switch team-feature
git rebase main
git push --force-with-lease origin team-feature
```

Now in dev2 (the other developer), they need to rebase their local work:

```bash
cd dev2
git switch team-feature
echo "my local work" >> f.txt && git add f.txt && git commit -m "My local commit"
git fetch origin
git rebase origin/team-feature
```

The rebase replays "My local commit" on top of the rewritten remote. The developer can now push normally.

**Alternative approach:** The second developer could also stash their changes, reset to `origin/team-feature`, then re-apply with `git stash apply`. But rebase is cleaner because it preserves the commit's authorship and message.

### Exercise 3: Split a large branch into two clean branches using rebase

**Scenario:** You have a branch with 6 commits that mix two independent features. Commits 1, 3, 5 are for feature A. Commits 2, 4, 6 are for feature B. You need to create two clean branches — one per feature — each rebased onto `main`.

**Task:** Create this scenario. Extract the feature A commits into one branch and feature B commits into another, each rebased on top of `main`. Use interactive rebase or `git rebase --onto` with careful cherry-picking. Do not use `git cherry-pick` manually for all 6 commits.

<details>
<summary>Hint</summary>
Use `git log --oneline --graph` to see the interleaved commits. Use `git rebase -i main` to reorder the commits so all feature A commits come first, then create a branch at that point, then continue the rebase for feature B. Or use `git rebase --onto` with `git branch` at the right points.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/rebase-adv-3 && cd ~/rebase-adv-3
git init
echo "base" > f.txt && git add f.txt && git commit -m "Base"
echo "base2" >> f.txt && git add f.txt && git commit -m "Base2"
git switch -c mixed-features
echo "A1" > a.txt && git add a.txt && git commit -m "Feature A part 1"
echo "B1" > b.txt && git add b.txt && git commit -m "Feature B part 1"
echo "A2" >> a.txt && git add a.txt && git commit -m "Feature A part 2"
echo "B2" >> b.txt && git add b.txt && git commit -m "Feature B part 2"
echo "A3" >> a.txt && git add a.txt && git commit -m "Feature A part 3"
echo "B3" >> b.txt && git add b.txt && git commit -m "Feature B part 3"
```

Now extract feature A:

```bash
# Create a branch at the base of mixed-features
git switch main
git switch -c feature-a
# Cherry-pick only the A commits
git cherry-pick mixed-features~5  # Feature A part 1
git cherry-pick mixed-features~3  # Feature A part 2
git cherry-pick mixed-features~1  # Feature A part 3
```

Extract feature B:

```bash
git switch main
git switch -c feature-b
git cherry-pick mixed-features~4  # Feature B part 1
git cherry-pick mixed-features~2  # Feature B part 2
git cherry-pick mixed-features~0  # Feature B part 3 (HEAD)
```

Verify:

```bash
git log --oneline feature-a
git log --oneline feature-b
```

Both branches have 3 commits each, cleanly separated, on top of `main`.

**Alternative approach:** Use `git rebase -i main` on `mixed-features` to reorder the commits interactively, grouping A together, then split into two branches. This is more advanced but avoids cherry-pick. The cherry-pick approach above is simpler and more explicit.
