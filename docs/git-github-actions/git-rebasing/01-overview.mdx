You have a feature branch with five commits, and while you were working, `main` moved forward with three new commits. You want to merge your feature, but a merge commit would clutter the history. You want your commits to appear as if they were written on top of the latest `main` — not as a separate branch that joined later. Rebasing rewrites your branch's foundation so your commits sit on top of a different starting point.

## The Concept

Rebasing takes a sequence of commits and replays them onto a new base commit, creating new commits with the same changes but different parents. The original commits become unreachable and are eventually garbage-collected.

Think of building a tower on a foundation that shifts. Instead of building a bridge from the old foundation to the new one (merge), you lift the tower, move it to the new foundation, and set it down again. The tower looks identical, but its base is now on solid ground. Every brick was relaid, so none of the original bricks remain — only copies that look exactly the same.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git rebase <branch>` | Replay the current branch's commits on top of `<branch>`. | `git rebase main` |
| `git rebase --onto <target> <base>` | Replay commits from the current branch that are not in `<base>`, onto `<target>`. | `git rebase --onto main feature-base` |
| `git rebase --abort` | Abort an in-progress rebase and return to the pre-rebase state. | `git rebase --abort` |
| `git rebase --continue` | Continue a rebase after resolving a conflict. | `git rebase --continue` |
| `git rebase --skip` | Skip the current commit during a rebase (discard its changes). | `git rebase --skip` |
| `git rebase main~3` | Rebase onto the commit three parents back from `main`. | `git rebase main~3` |
| `git rebase -i <commit>` | Start an interactive rebase from `<commit>` (not including it). | `git rebase -i HEAD~3` |
| `git pull --rebase` | Fetch and rebase local commits onto the fetched remote branch instead of merging. | `git pull --rebase origin main` |
| `git config pull.rebase true` | Set rebase as the default for `git pull` in this repository. | `git config pull.rebase true` |

**Windows note:** All rebase commands are identical on Windows. The `--rebase` flag for `git pull` works the same way. If you use Git Bash, the interactive rebase editor defaults to `vim` — you can change it with `git config core.editor "code --wait"` to use VS Code.

```
│                           │                              │
│                           └──▶ F0 ──▶ F1 ──▶ F2         │
│                                                          │
│  After `git rebase main` (from feature branch):          │
│                                                          │
│  main ──▶ C0 ──▶ C1 ──▶ C2 ──▶ C3                      │

│                                                          │

```

## How It Actually Works — Step by Step

### Step 1: Identify the commits to replay

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3))  ← main
                           │
                           └──▶ ((F0)) ──▶ ((F1)) ──▶ ((F2))  ← feature
                                                              ← HEAD
```

You are on `feature` (HEAD → feature → F2). You run `git rebase main`. Git calculates the set of commits that are reachable from `feature` but not from `main`. This is `git log main..feature` — commits `F0`, `F1`, `F2`. The merge base (`C2`) is irrelevant; Git uses the `main` tip as the new base.

### Step 2: Git saves the patches and detaches HEAD

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3))  ← main
                                            ← HEAD (detached)
```

Git saves the diff of each commit (F0, F1, F2) as patches. It then resets the working directory to match `main`'s tip (`C3`). HEAD is now detached — it points at `C3` directly, not at a branch.

Internally, Git stores the patches in `.git/rebase-apply/` and records the original branch name in `.git/HEAD` as a rebase in progress.

### Step 3: Replay each commit as a new patch

Git applies the first patch (`F0`'s diff) on top of `C3`:

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3)) ──▶ ((F0'))  ← HEAD (detached)
```

If the patch applies cleanly, Git creates a new commit `F0'` with the same author, message, and diff as `F0` — but with `C3` as its parent instead of `C2`. The hash is different because the parent hash is part of the commit's identity.

### Step 4: Replay remaining commits

Git applies the next patch:

```
  ((C0)) ──▶ ... ──▶ ((C3)) ──▶ ((F0')) ──▶ ((F1'))  ← HEAD (detached)
```

Then the last one:

```
  ((C0)) ──▶ ... ──▶ ((C3)) ──▶ ((F0')) ──▶ ((F1')) ──▶ ((F2'))  ← HEAD (detached)
```

### Step 5: Update the branch pointer

```
  ((C0)) ──▶ ... ──▶ ((C3)) ──▶ ((F0')) ──▶ ((F1')) ──▶ ((F2'))  ← feature
                                                                  ← HEAD
```

Git moves the `feature` branch pointer to `F2'` and attaches HEAD to it. The original commits `F0`, `F1`, `F2` are still in the object store but no branch points to them. They become dangling commits.

### Step 6: Clean up

Git removes `.git/rebase-apply/` and the rebase state files. The branch now has a linear history on top of `main`:

```
  git log --oneline
  F2' Third feature commit
  F1' Second feature commit
  F0' First feature commit
  C3  Latest main commit
  C2  Main commit
  ...
```

No merge commit. No branch divergence. The feature commits appear as if they were written after `C3`.

## When to Use This / When Not To

**Use rebase when:**
- You want a linear, clean history without merge bubbles
- You are updating a feature branch to integrate upstream changes before a pull request
- You are the only person working on the branch and you want to keep a tidy commit log
- You want to incorporate changes from the base branch without creating a merge commit

**Do not use rebase when:**
- The branch is shared with other developers — rebasing creates new hashes, which causes chaos for anyone else who has the old commits
- You want to preserve the exact historical record of when and in what order work was done
- You are rebasing a branch that has already been pushed to a shared remote (unless you are prepared to force-push and coordinate with the team)
- You need to distinguish between "this is where the branch was created" and "this is where it was merged"

**The golden rule of rebase:** Never rebase commits that have been pushed to a shared branch. Rebase is for local cleanup before pushing. If you rebase and force-push a shared branch, every collaborator who pulled the old commits will have a broken history.

Rebase and merge serve the same end goal — integrating changes — but they produce different histories. Rebase sacrifices historical accuracy for linear clarity. Merge sacrifices linear clarity for historical accuracy. Choose based on your team's workflow, not habit.
