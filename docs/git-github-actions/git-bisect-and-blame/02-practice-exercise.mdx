You will create a repository with a deliberate bug introduced in a specific commit, then use `git bisect` to find the exact commit that broke the behavior. Then use `git blame` to identify the specific line. Every command works identically on Windows, macOS, and Linux.

## Setup

```bash
mkdir -p ~/bisect-lab
cd ~/bisect-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create a script that will be correct at first, then broken later:

```bash
cat > math.sh << 'EOF'
#!/bin/bash
# Simple math utility
add() { echo $(($1 + $2)); }
subtract() { echo $(($1 - $2)); }
multiply() { echo $(($1 * $2)); }

# Run if executed directly
if [ "$1" = "add" ]; then add $2 $3; fi
if [ "$1" = "subtract" ]; then subtract $2 $3; fi
if [ "$1" = "multiply" ]; then multiply $2 $3; fi
EOF
chmod +x math.sh
git add math.sh
git commit -m "Initial math utility"
```

Now create a test script:

```bash
cat > test.sh << 'EOF'
#!/bin/bash
result=$(./math.sh multiply 3 4)
if [ "$result" = "12" ]; then
    exit 0
else
    echo "FAIL: 3*4 = $result (expected 12)"
    exit 1
fi
EOF
chmod +x test.sh
git add test.sh
git commit -m "Add multiplication test"
```

## The Walkthrough

### Step 1: Create a series of commits, one of which introduces a bug

We will make 8 commits, and commit #5 will introduce a bug in the multiply function:

```bash
echo "  echo 'Math utility v1.1'" >> math.sh
git add math.sh
git commit -m "Add version echo v1"
echo "  echo 'Math utility v1.2'" >> math.sh
git add math.sh
git commit -m "Bump version to v1.2"
echo "verbose=false" >> math.sh
git add math.sh
git commit -m "Add verbose flag"
echo "if [ \"\$verbose\" = true ]; then echo 'debug mode'; fi" >> math.sh
git add math.sh
git commit -m "Add debug mode output"
```

Commit #5 — the buggy one:

```bash
sed -i 's/multiply() { echo $(($1 \* $2)); }/multiply() { echo $(( $1 * $2 + 1 )); }/' math.sh
git add math.sh
git commit -m "Refactor multiply to use long form"
```

Continue with innocent commits:

```bash
echo "  echo 'Math utility v1.3'" >> math.sh
git add math.sh
git commit -m "Bump version to v1.3"
echo "log_level=info" >> math.sh
git add math.sh
git commit -m "Add log level config"
echo "if [ \"\$log_level\" = debug ]; then set -x; fi" >> math.sh
git add math.sh
git commit -m "Add debug logging support"
```

Verify the test fails on the latest commit:

```bash
./test.sh
```

Expected output:

```
FAIL: 3*4 = 13 (expected 12)
```

The bug exists. The test fails.

**Checkpoint:** `git log --oneline` shows 10 commits (initial + 8 changes + test).

### Step 2: Mark the good and bad commits

We know the test passes on the initial commit and fails on the latest:

```bash
# Find the initial commit hash
INITIAL_HASH=$(git log --oneline | tail -1 | awk '{print $1}')
git bisect start
git bisect good $INITIAL_HASH
git bisect bad HEAD
```

Expected output:

```
Bisecting: 4 steps remaining to test after this (roughly 4 commits)
<hash> is the first bad commit
```

Actually, Git will check out a midpoint commit and waits for you to test. Expected output is more like:

```
Bisecting: 4 steps remaining to test after this (roughly 4 commits)
```

### Step 3: Test the midpoint and mark it

Git has checked out a commit in the middle of the range. Run the test:

```bash
./test.sh
```

If the test passes (exit 0), mark it good:

```bash
git bisect good
```

If the test fails (exit 1), mark it bad:

```bash
git bisect bad
```

Repeat: test, mark, test, mark. Each time the range halves.

### Step 4: Continue until bisect finds the culprit

After 3-4 iterations, Git reports:

```
<hash> is the first bad commit
commit <hash>
Author: Your Name <your.email@example.com>
Date:   ...

    Refactor multiply to use long form

 math.sh | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)
```

This is the exact commit that introduced the bug. `git show <hash>` shows the diff:

```
- multiply() { echo $(($1 * $2)); }
+ multiply() { echo $(( $1 * $2 + 1 )); }
```

The `+ 1` is the bug.

### Step 5: Automate with `git bisect run`

Instead of testing manually each time, you can automate the entire process:

```bash
git bisect reset
git bisect start HEAD $INITIAL_HASH
git bisect run ./test.sh
```

Expected output — Git runs the test on each midpoint automatically:

```
running ./test.sh
Bisecting: 2 steps remaining to test after this (roughly 2 commits)
running ./test.sh
Bisecting: 1 steps remaining to test after this (roughly 1 commit)
running ./test.sh
<hash> is the first bad commit
```

The `bisect run` command:
1. Checks out a midpoint
2. Runs `./test.sh`
3. If exit code 0 → marks `good` and checks out the next midpoint
4. If exit code 1 → marks `bad` and checks out the next midpoint
5. When one commit remains, reports it as the first bad commit

**Windows note:** For `git bisect run` on Windows, use `git bisect run powershell.exe -Command ".\test.ps1; exit $LASTEXITCODE"` if your test is a PowerShell script. Batch files work with `git bisect run cmd.exe /c test.bat`.

### Step 6: Use `git blame` to identify the specific line

Reset the bisect and look at the file:

```bash
git bisect reset
git blame math.sh
```

Expected output:

```
^<hash1> (Your Name 2026-07-03 1) #!/bin/bash
^<hash1> (Your Name 2026-07-03 2) # Simple math utility
<hash3>  (Your Name 2026-07-03 3) add() { echo $(($1 + $2)); }
<hash3>  (Your Name 2026-07-03 4) subtract() { echo $(($1 - $2)); }
<hash5>  (Your Name 2026-07-03 5) multiply() { echo $(( $1 * $2 + 1 )); }
...
```

Line 5 shows the buggy `+ 1` was last changed in the commit that bisect identified.

Now blame a specific range:

```bash
git blame -L 5,6 math.sh
```

This shows only the lines 5-6, which is useful for large files.

### Step 7: Ignore whitespace-only changes

Create a commit that reformats whitespace, then see how `-w` helps:

```bash
sed -i 's/    /\t/' math.sh  # Replace 4 spaces with tab
git add math.sh
git commit -m "Convert indentation to tabs"
git blame -L 5,6 math.sh
```

The blame now points to the indentation commit, not the original author. Retry with `-w`:

```bash
git blame -w -L 5,6 math.sh
```

With `-w`, Git ignores whitespace changes and shows the last meaningful (non-whitespace) change — the buggy commit.

## Common Errors at This Stage

### Error 1: `You need to start a bisect session first`

You ran `git bisect good` or `git bisect bad` without first running `git bisect start`. Git has no session to record the mark.

**Fix:** Run `git bisect start` first, then mark the good and bad commits.

### Error 2: `bisect run failed: exit code 125`

Your test script exited with code 125, which Git interprets as "this commit cannot be tested." This is useful for commits that do not compile — you can `exit 125` in your script to skip them.

**Fix:** If you did not intend to skip, check your test script. Code 125 is reserved for "cannot test." Use 0 for good, 1-127 for bad (except 125).

### Error 3: `fatal: Needed a single revision`

You provided a bad commit reference that Git cannot resolve. For example, `git bisect bad HEAD~20` when the repo does not have 20 ancestors.

**Fix:** Use `git log --oneline` to find valid commit references within the range you want to bisect.

### Error 4: Bisect keeps skipping commits

If many commits in the range do not compile (you marked them with `skip`), Git has trouble converging. The binary search algorithm needs testable commits to narrow the range.

**Fix:** Start with a narrower range where you know every commit compiles. Use tags or release markers as the good boundary instead of an arbitrary commit from early in development.

## What You Just Learned

- `git bisect start`, `git bisect good`, `git bisect bad` perform a binary search through history
- Binary search finds the bad commit in log2(N) steps — 128 commits in 7 steps
- `git bisect run <script>` automates the process with a test script
- `git bisect reset` ends the session and restores your original HEAD
- `git blame <file>` shows the last commit that modified each line
- `git blame -L <start>,<end>` narrows to a line range
- `git blame -w` ignores whitespace changes to find the real author
