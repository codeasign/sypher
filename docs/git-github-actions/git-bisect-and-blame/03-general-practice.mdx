## Tier 1 — Beginner

### Exercise 1

**Task:** Create a repo with 10 commits. Introduce a deliberate bug in commit 7. Use `git bisect` manually (step by step, without `run`) to find the exact commit that introduced the bug. How many steps did it take?

<details>
<summary>Hint</summary>
Mark the first commit as good and the latest as bad. Test each midpoint by checking if the bug is present (e.g., by running a test command or checking a file's content). The number of steps should be ceil(log2(10)) = 4.
</details>

### Solution

```bash
mkdir -p ~/bisect-beginner-1 && cd ~/bisect-beginner-1
git init
for i in $(seq 1 10); do
  echo "Commit $i" > f.txt
  if [ $i -eq 7 ]; then echo "BUG" >> f.txt; fi
  git add f.txt && git commit -m "C$i"
done
git bisect start
git bisect good HEAD~9  # First commit
git bisect bad HEAD     # Latest commit
# Test each midpoint by checking if "BUG" is in f.txt
# If "BUG" is present → bad, otherwise → good
# Repeat until Git identifies commit 7
git bisect reset
```

The bisect should find commit 7 in 4 steps (log2 of 10 = ~4).

### Exercise 2

**Task:** Use `git blame` on a file with 20 lines, each written by a different commit. Identify which line was last modified by commit 15.

<details>
<summary>Hint</summary>
Create the file one line at a time across 20 commits. Run `git blame f.txt` and look for the commit hash that matches commit 15's message.
</details>

### Solution

```bash
mkdir -p ~/bisect-beginner-2 && cd ~/bisect-beginner-2
git init
for i in $(seq 1 20); do
  echo "Line $i written by commit $i" >> f.txt
  git add f.txt && git commit -m "Add line $i"
done
git blame f.txt
```

Identify which line has the hash matching commit 15. The `^` prefix on the first commit's hash indicates it is the boundary commit (root).

### Exercise 3

**Task:** Use `git blame -L` to show only lines 5-10 of a file in a repo with 30 commits. Then use `git blame -w` to ignore a whitespace-only reformatting commit that touched line 7.

<details>
<summary>Hint</summary>
After creating the file, add a commit that reformats whitespace (e.g., replace spaces with tabs). Run `git blame -L 5,10 f.txt` and `git blame -w -L 5,10 f.txt` to see the difference.
</details>

### Solution

```bash
mkdir -p ~/bisect-beginner-3 && cd ~/bisect-beginner-3
git init && echo "line1" > f.txt && git add f.txt && git commit -m "C1"
for i in 2 3 4 5 6 7 8 9 10; do echo "line$i" >> f.txt; git add f.txt && git commit -m "C$i"; done
sed -i 's/^line7/  line7/' f.txt  # Add whitespace
git add f.txt && git commit -m "Reformat line 7"
git blame -L 5,10 f.txt
# Line 7 shows the reformat commit
git blame -w -L 5,10 f.txt
# Line 7 now shows the original author
```

## Tier 2 — Intermediate

### Exercise 1

**Task:** Use `git bisect run` with a Python script to automatically find the buggy commit. Create a Python script that reads a file, checks a condition, and exits with 0 or 1. Run bisect with the script as the test command.

<details>
<summary>Hint</summary>
Create a Python test script: `import sys; content = open('f.txt').read(); sys.exit(0 if 'BUG' not in content else 1)`. Run `git bisect run python test.py`.
</details>

### Solution

```bash
mkdir -p ~/bisect-inter-1 && cd ~/bisect-inter-1
git init
for i in $(seq 1 12); do
  echo "Commit $i" > f.txt
  if [ $i -eq 8 ]; then echo "BUG" >> f.txt; fi
  git add f.txt && git commit -m "C$i"
done
cat > test.py << 'EOF'
import sys
content = open('f.txt').read()
if 'BUG' in content:
    sys.exit(1)  # bad
else:
    sys.exit(0)  # good
EOF
git bisect start HEAD HEAD~11
git bisect run python test.py
git bisect reset
```

### Exercise 2

**Task:** Create a scenario where a bug is introduced in a merge commit (merge of two branches). Use bisect to find the merge commit. Bisect handles merge commits by tracking the DAG structure — verify it can find the merge commit that introduced the bug.

<details>
<summary>Hint</summary>
Create two branches, each with commits. On one branch, introduce the bug. Then merge that branch into the other. The merge commit is the first bad commit. Run bisect from the merge commit (bad) back to the base (good).
</details>

### Solution

```bash
mkdir -p ~/bisect-inter-2 && cd ~/bisect-inter-2
git init && echo "ok" > f.txt && git add f.txt && git commit -m "Base"
echo "ok" >> f.txt && git add f.txt && git commit -m "C1"
echo "ok" >> f.txt && git add f.txt && git commit -m "C2"
git switch -c buggy
echo "BUG" >> f.txt && git add f.txt && git commit -m "Buggy commit"
echo "also ok" >> f.txt && git add f.txt && git commit -m "C4"
git switch main
echo "ok 3" >> f.txt && git add f.txt && git commit -m "C3"
git merge buggy -m "Merge buggy branch"
git bisect start HEAD HEAD~4
git bisect run bash -c "! grep -q BUG f.txt"
git bisect reset
```

Bisect should identify the merge commit (or the buggy commit itself, depending on the DAG) as the first bad commit.

### Exercise 3

**Task:** Use `git blame` to find the author of a specific line, then use `git log --since` to see all commits by that author after that date. This simulates a workflow where you find who changed a line and then review their recent work.

<details>
<summary>Hint</summary>
Capture the author and date from `git blame -L <line>,<line> f.txt`, then use `git log --author="<name>" --since="<date>"` to see their subsequent commits.
</details>

### Solution

```bash
mkdir -p ~/bisect-inter-3 && cd ~/bisect-inter-3
git init && echo "A" > f.txt && git add f.txt && git commit -m "Author A"
echo "B" >> f.txt && git add f.txt && git commit -m "Author B"
echo "C" >> f.txt && git add f.txt && git commit -m "Author C"
git blame -L 3,3 f.txt
# Output: hash (Author Name  date) line
# Extract author and date
AUTHOR=$(git blame -L 3,3 f.txt | awk '{print $2}')
DATE=$(git blame -L 3,3 f.txt | awk '{print $3, $4, $5}')
git log --author="$AUTHOR" --oneline
```

## Tier 3 — Advanced

### Exercise 1: Bisect with a flaky test

**Scenario:** Your test suite has a flaky test that fails about 30% of the time regardless of the code. When you run `git bisect run`, the flaky test may report a wrong good/bad result, causing bisect to find the wrong commit.

**Task:** Create a test script that simulates flakiness (randomly exits 0 or 1). Then write a wrapper that runs the test 3 times and exits with the majority result. Use this wrapper with `git bisect run` to robustly handle the flakiness.

<details>
<summary>Hint</summary>
Use a wrapper script that runs the test N times, counts passes vs failures, and exits with the majority. For 3 runs: `pass=0; for i in 1 2 3; do python test.py && pass=$((pass+1)); done; [ $pass -ge 2 ]`
</details>

### Solution — one valid approach

```bash
mkdir -p ~/bisect-adv-1 && cd ~/bisect-adv-1
git init
for i in $(seq 1 10); do
  echo "C$i" > f.txt
  if [ $i -eq 6 ]; then echo "BUG" >> f.txt; fi
  git add f.txt && git commit -m "C$i"
done
# Flaky test script
cat > flaky_test.py << 'EOF'
import sys, random
content = open('f.txt').read()
# 30% chance of wrong result
if random.random() < 0.3:
    sys.exit(0 if 'BUG' in content else 1)  # Wrong!
else:
    sys.exit(1 if 'BUG' in content else 0)  # Correct
EOF
# Robust wrapper
cat > robust_test.sh << 'EOF'
#!/bin/bash
pass=0
fail=0
for i in 1 2 3; do
  python flaky_test.py && pass=$((pass+1)) || fail=$((fail+1))
done
if [ $pass -ge 2 ]; then exit 0; else exit 1; fi
EOF
chmod +x robust_test.sh
git bisect start HEAD HEAD~9
git bisect run bash robust_test.sh
git bisect reset
```

**Alternative approach:** Use `git bisect skip` on commits that produce inconclusive results (run the test 5 times, if the result is not consistent, exit 125 to skip).

### Exercise 2: Blame across a file rename

**Scenario:** A file was renamed six months ago, and `git blame` on the current file shows only the rename commit for every line. You need to find the original authors of the code before the rename.

**Task:** Create a file, make commits, rename it, make more commits, then use `git blame -C` (detect moved lines) to track the origin of lines across the rename. Compare `git blame` with and without `-C`.

<details>
<summary>Hint</summary>
`git blame -C` detects lines that were moved or copied from other files. Use `-C` multiple times (`-C -C` or `-CCC`) for harder-to-find copies. Create the file, commit, rename with `git mv`, commit, then blame the renamed file.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/bisect-adv-2 && cd ~/bisect-adv-2
git init
echo "original author work" > old_name.py
git add old_name.py && git commit -m "Add old_name.py"
echo "more original work" >> old_name.py
git add old_name.py && git commit -m "Enhance old_name.py"
git mv old_name.py new_name.py
git commit -m "Rename old_name.py to new_name.py"
echo "new work after rename" >> new_name.py
git add new_name.py && git commit -m "Add new work"
# Without -C
git blame new_name.py
# Lines 1-2 show the rename commit, not the original author
# With -C
git blame -C new_name.py
# Lines 1-2 now show the original author from old_name.py
```

**Alternative approach:** `git blame -C -C -C` is the most aggressive detection level. It searches across all files in the repository, not just renames.

### Exercise 3: Range bisect with a performance regression

**Scenario:** A performance regression was introduced somewhere in the last 50 commits. The code compiles and passes tests, but a benchmark runs 3x slower. You cannot use a simple pass/fail test — you need to measure execution time and compare against a threshold.

**Task:** Create a performance test script that measures execution time of a command and exits with 0 (good) if it is under the threshold, 1 (bad) if over. Use `git bisect run` with this script to find the commit that introduced the slowdown.

<details>
<summary>Hint</summary>
Write a script that runs the benchmark multiple times, takes the median, and compares it to a threshold. Store the threshold in a file that is not tracked by Git. Use `time` command or Python's `time` module for measurement.
</details>

### Solution — one valid approach

```bash
mkdir -p ~/bisect-adv-3 && cd ~/bisect-adv-3
git init
# Create a slow script that gets progressively slower
cat > bench.py << 'EOF'
import time
# Simulated workload — gets slower as we add more iterations
total = 0
for i in range(10000):
    total += i * i
print(total)
EOF
git add bench.py && git commit -m "Initial benchmark"
# Commit 2: slightly slower
for i in 2 3 4 5 6 7 8; do
  cat > bench.py << 'EOF'
import time
total = 0
EOF
  echo "for i in range($((i * 10000))):" >> bench.py
  echo "    total += i * i" >> bench.py
  echo "print(total)" >> bench.py
  git add bench.py && git commit -m "Iteration $i"
done
# Performance test script
cat > perf_test.sh << 'EOF'
#!/bin/bash
# Run the benchmark and measure time
start=$(date +%s%N)
python bench.py > /dev/null
end=$(date +%s%N)
elapsed_ms=$(( (end - start) / 1000000 ))
# Threshold: 500ms
if [ $elapsed_ms -le 500 ]; then
    exit 0  # good — fast enough
else
    exit 1  # bad — too slow
fi
EOF
chmod +x perf_test.sh
git bisect start HEAD HEAD~7
git bisect run bash perf_test.sh
git bisect reset
```

**Note:** This approach requires that the benchmark is deterministic. For real performance regression hunting, run the benchmark multiple times at each commit and take the median. Set the threshold based on the known-good commit's measured performance plus a margin (e.g., 20% overhead).

**Alternative approach:** Instead of a hard threshold, compare against the good commit's performance dynamically: measure the good commit once, store the baseline, then bisect with a script that compares each commit's performance to the baseline.
