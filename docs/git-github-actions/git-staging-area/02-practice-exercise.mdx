This walkthrough simulates a real editing session. You will create files, modify them, stage selectively, unstage, and commit — all while watching the staging area change state.

## Setup

Open a terminal in a directory where you can create a new Git repository. Do not run this inside an existing repository.

```bash
# Create and enter a practice directory
mkdir git-staging-practice
cd git-staging-practice

# Initialize a new Git repository
git init
```

Expected output:
```
Initialized empty Git repository in .../git-staging-practice/.git/
```

Now create three files to work with:

```bash
# Create index.html
echo '<h1>Hello World</h1>' > index.html

# Create style.css
echo 'body { color: black; }' > style.css

# Create app.js
echo 'console.log("Hello");' > app.js
```

Make the initial commit so we have a baseline to compare against:

```bash
git add -A
git commit -m "Initial commit"
```

Expected output:
```
[main (root-commit) a1b2c3d] Initial commit
 3 files changed, 3 insertions(+)
 create mode 100644 index.html
 create mode 100644 app.js
 create mode 100644 style.css
```

Verify the state:

```bash
git status -s
```

No output means the working directory and index both match the last commit. Everything is clean.

**Windows users:** All commands above work identically in Git Bash, PowerShell, and Command Prompt. If you use PowerShell, note that `echo` is an alias for `Write-Output` — the output is the same. If you see `> 1` as part of the output in PowerShell, escape the redirect with `--%` or use `Set-Content` instead.

## The Walkthrough

### Step 1: Make Changes to Multiple Files

Edit all three files to simulate a real work session:

```bash
echo '<h1>Goodbye World</h1>' > index.html
echo 'body { background: blue; }' > style.css
echo 'console.log("Goodbye");' > app.js
```

Check the status:

```bash
git status -s
```

Expected output:
```
 M index.html
 M style.css
 M app.js
```

The space before each `M` means the first column is empty — nothing is staged. The `M` in the second column means the working directory has modifications compared to the index. The staging area still matches the last commit.

**Checkpoint:** Your index is clean. Your working directory has three modified files.

### Step 2: Stage Only One File

You decide to commit only the `app.js` change first because it is a simple, self-contained update.

```bash
git add app.js
```

Check the status again:

```bash
git status -s
```

Expected output:
```
M  app.js
 M index.html
 M style.css
```

Now `app.js` shows `M` in the first column — staged. The other two files still show the space-then-M pattern: unstaged. The staging area now has one file updated and two files matching the last commit.

**Checkpoint:** `app.js` is staged. `index.html` and `style.css` are not.

### Step 3: Review What Is Staged

Before committing, verify what will go into the commit:

```bash
git diff --staged
```

Expected output:
```
diff --git a/app.js b/app.js
index abc1234..def5678 100644
--- a/app.js
+++ b/app.js
@@ -1 +1 @@
-console.log("Hello");
+console.log("Goodbye");
```

Only the `app.js` change appears. The `index.html` and `style.css` changes are not shown because they are not staged. This confirms that the upcoming commit will contain exactly one change.

**Checkpoint:** `git diff --staged` shows only `app.js`.

### Step 4: Commit the Staged Change

```bash
git commit -m "Update app.js greeting"
```

Expected output:
```
[main 1234567] Update app.js greeting
 1 file changed, 1 insertion(+), 1 deletion(-)
```

Now check the status:

```bash
git status -s
```

Expected output:
```
 M index.html
 M style.css
```

`app.js` is gone from the status output — it now matches HEAD. The staging area for `app.js` was consumed by the commit, and the index entry for `app.js` now matches the new HEAD. The other two files remain unstaged.

### Step 5: Stage Selectively with Interactive Mode

The `style.css` change is simple. The `index.html` change is also simple. Stage them both:

```bash
git add style.css index.html
```

```bash
git status -s
```

Expected output:
```
M  index.html
M  style.css
```

Both files are now staged. The first column shows `M` for both, the second column is empty.

**Checkpoint:** All three files are staged and ready to commit.

### Step 6: Make a New Change After Staging

Edit `index.html` again without committing first:

```bash
echo '<h1>Hello Again World</h1>' > index.html
```

Now check the status:

```bash
git status -s
```

Expected output:
```
MM index.html
M  style.css
```

The `MM` output is important. The first `M` means the version in the index is different from HEAD (staged). The second `M` means the working directory version is different from the index (unstaged). You have two versions of the change: the staged version (`index.html` with "Goodbye World") and the unstaged version (`index.html` with "Hello Again World").

**Checkpoint:** `index.html` has both staged and unstaged changes.

### Step 7: Compare Staged vs Unstaged

Run `git diff --staged` to see what is staged:

```bash
git diff --staged
```

Expected output:
```
diff --git a/index.html b/index.html
index abc1234..def5678 100644
--- a/index.html
+++ b/index.html
@@ -1 +1 @@
-<h1>Hello World</h1>
+<h1>Goodbye World</h1>
```

Now run `git diff` (without `--staged`) to see what is unstaged:

```bash
git diff
```

Expected output:
```
diff --git a/index.html b/index.html
index def5678..ghi9012 100644
--- a/index.html
+++ b/index.html
@@ -1 +1 @@
-<h1>Goodbye World</h1>
+<h1>Hello Again World</h1>
```

The staged diff shows the change from the original to "Goodbye World." The unstaged diff shows the change from the staged version to "Hello Again World." Together they form a chain: original -> staged -> working directory.

**Checkpoint:** You can see two different diffs for the same file.

### Step 8: Unstage and Re-stage

You decide the "Goodbye World" version is wrong. You want to commit the "Hello Again World" version instead. First, unstage the current staged version:

```bash
git reset HEAD index.html
```

```bash
git status -s
```

Expected output:
```
 M index.html
M  style.css
```

`index.html` now shows ` M` — unstaged only. The staged version was replaced with the HEAD version (the original). But the working directory still has "Hello Again World."

Now stage the working directory version:

```bash
git add index.html
```

```bash
git status -s
```

Expected output:
```
M  index.html
M  style.css
```

Check the staged diff:

```bash
git diff --staged
```

Expected output:
```
diff --git a/index.html b/index.html
index abc1234..ghi9012 100644
--- a/index.html
+++ b/index.html
@@ -1 +1 @@
-<h1>Hello World</h1>
+<h1>Hello Again World</h1>
```

The staged diff now shows the direct change from original to "Hello Again World." The intermediate "Goodbye World" version was never in the index long enough to be committed.

### Step 9: Commit the Remaining Files

```bash
git commit -m "Update index.html and style.css"
```

Expected output:
```
[main 7890abc] Update index.html and style.css
 2 files changed, 2 insertions(+), 2 deletions(-)
```

```bash
git status -s
```

Expected output: (nothing — clean working directory)

```bash
git log --oneline
```

Expected output:
```
7890abc Update index.html and style.css
1234567 Update app.js greeting
a1b2c3d Initial commit
```

**Checkpoint:** Clean working directory. Two commits on top of the initial commit.

## Common Errors at This Stage

### Error 1: `fatal: pathspec 'file.txt' did not match any files`

```
$ git add nonexistent.py
fatal: pathspec 'nonexistent.py' did not match any files
```

**Why:** You tried to stage a file that does not exist in the working directory. The path could be misspelled, or the file was deleted without you realizing it.

**Fix:** Check the exact filename with `ls` or `dir`. If the file was deleted and you want to stage the deletion, use `git add -A` to stage all changes including deletions, or use `git rm` to stage the removal explicitly.

### Error 2: `MM` status and committing the wrong version

```
$ git status -s
MM index.html
```

You run `git commit -m "Fix"` and only the first staged version gets committed. The second unstaged change stays behind.

**Why:** Git commits only what is in the staging area. If you staged a file, then edited it again without re-staging, the commit captures the staged version, not the working directory version.

**Fix:** Always run `git status -s` before committing. If you see `MM`, the file has two layers of changes. Run `git add <file>` again to stage the latest version, or run `git diff --staged` to verify the staged version is what you intend to commit.

### Error 3: `git reset HEAD` without a file path unstages everything

```
$ git reset HEAD
Unstaged changes after reset:
M       index.html
M       style.css
```

This unstaged every file in the repository. If you intended to unstage only one file, you now have to re-stage the others.

**Why:** `git reset HEAD` (without a path) resets the entire index to match HEAD. This is the same as running `git reset HEAD <file>` for every file in the index.

**Fix:** Always specify a file path when you want to unstage a single file: `git reset HEAD <filename>`. If you accidentally unstage everything, re-stage the files you need with `git add <file1> <file2>`.

### Error 4: `git rm --cached` removes the file from tracking

```
$ git rm --cached config.yml
$ git status -s
D  config.yml
```

The file is now deleted from the index. If you commit now, the commit will record the deletion of `config.yml`. The file still exists on disk, but Git will stop tracking it.

**Why:** `git rm --cached` removes the file from the index entirely — it does not just unstage changes. This is different from `git reset HEAD`, which reverts the index entry to match HEAD but keeps the file tracked.

**Fix:** Use `git rm --cached` only when you want to stop tracking a file permanently (usually for files you then add to `.gitignore`). To simply unstage changes to a tracked file, use `git reset HEAD <file>` instead.

### Error 5: `git add -A` stages everything including unwanted files

```
$ git add -A
$ git status -s
M  index.html
M  style.css
M  app.js
M  debug.log
M  .env
```

You accidentally staged `debug.log` and `.env`, which should never be committed.

**Why:** `git add -A` stages every change in the working directory, including new files, modified files, and deleted files. It does not respect `.gitignore` if the files are already tracked or if `.gitignore` is not set up.

**Fix:** Before running `git add -A`, configure `.gitignore` to exclude files like `debug.log` and `.env`. If you already staged them, unstage individually with `git reset HEAD debug.log .env`.

## What You Just Learned

- `git add <file>` moves a file from the working directory into the staging area by updating the index with the file's current blob hash.
- `git status -s` shows two columns: the first column is staging area status, the second column is working directory status. A space in a column means "no difference from the reference."
- `git diff --staged` compares the index against HEAD, showing what will be committed. `git diff` (without flags) compares the working directory against the index, showing what is still unstaged.
- `git reset HEAD <file>` reverts the index entry for a file to match HEAD, unstaging changes without modifying the working directory.
- A file can have both staged and unstaged changes simultaneously, shown as `MM` in `git status -s`. This happens when you edit a file after staging it.
- `git rm --cached` removes a file from the index entirely, which stops Git from tracking it. This is different from `git reset HEAD`, which keeps the file tracked but reverts the staged content.
- Interactive staging with `git add -p` lets you stage individual hunks within a file, giving you fine-grained control over exactly what each commit contains.
- The staging area is a review gate, not a required formality. Use it when you need precision and skip it with `git commit -a` when the change is simple and atomic.
