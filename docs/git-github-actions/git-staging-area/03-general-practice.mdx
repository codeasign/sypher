## Tier 1 — Beginner

### Exercise 1: Stage a Single File

Create a new Git repository and two files. Stage only one of them.

**Task:**
1. `git init` a new repository.
2. Create `readme.md` with the text `# My Project` and `notes.txt` with the text `TODO: finish this later`.
3. Stage only `readme.md` and commit with the message "Add readme".
4. Verify that `notes.txt` is not tracked.

**Hint:**
<details>
<summary>Click to expand</summary>

Use `git add readme.md` to stage only that file. After committing, `git status` should show `notes.txt` as untracked. `git ls-files` will list only the tracked files.

</details>

**Solution:**

```bash
git init
echo '# My Project' > readme.md
echo 'TODO: finish this later' > notes.txt
git add readme.md
git commit -m "Add readme"
git status
```

Expected final status output:
```
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        notes.txt
```

`notes.txt` exists on disk but is not in the staging area and not in any commit.

### Exercise 2: Stage All Files, Then Unstage One

Create three files, stage all of them, then unstage exactly one.

**Task:**
1. `git init` a new repository.
2. Create `index.html`, `style.css`, and `app.js` with any content.
3. Stage all three files with a single command.
4. Unstage `app.js` so it returns to untracked status.
5. Commit the remaining two files.

**Hint:**
<details>
<summary>Click to expand</summary>

Use `git add -A` to stage everything. Then `git reset HEAD app.js` to unstage it. Note that `git reset HEAD` unstages changes but does not remove a file from tracking if it was already tracked — for a new file that was never committed, use `git rm --cached app.js` to fully unstage it.

</details>

**Solution:**

```bash
git init
echo '<h1>Home</h1>' > index.html
echo 'body { color: red; }' > style.css
echo 'console.log("init");' > app.js
git add -A
git rm --cached app.js
git status
```

Expected status:
```
Changes to be committed:
  new file:   index.html
  new file:   style.css

Untracked files:
  app.js
```

Commit the staged files:

```bash
git commit -m "Add index and styles"
```

### Exercise 3: Stage All Changes After Editing

Start from a repository with one committed file, edit it, then stage the modification.

**Task:**
1. `git init` a new repository.
2. Create `config.yaml` with the content `debug: false` and commit it.
3. Change the content to `debug: true`.
4. Stage the change.
5. Verify the staged diff shows exactly the key change.

**Hint:**
<details>
<summary>Click to expand</summary>

After editing, `git diff` shows the unstaged change. `git add config.yaml` stages it. Then `git diff --staged` shows the staged diff. Without `--staged`, `git diff` shows nothing because the working directory now matches the index.

</details>

**Solution:**

```bash
git init
echo 'debug: false' > config.yaml
git add config.yaml
git commit -m "Add config with debug disabled"
echo 'debug: true' > config.yaml
git add config.yaml
git diff --staged
```

Expected staged diff:
```
diff --git a/config.yaml b/config.yaml
index abc1234..def5678 100644
--- a/config.yaml
+++ b/config.yaml
@@ -1 +1 @@
-debug: false
+debug: true
```

### Exercise 4: Use Short Status to Check State

Practice reading the two-column `git status -s` output.

**Task:**
1. `git init` and create `a.txt`, `b.txt`, `c.txt`. Commit all three.
2. Edit `a.txt` and `b.txt` but not `c.txt`.
3. Stage only `a.txt`.
4. Edit `a.txt` again after staging.
5. Edit `b.txt` again.
6. Run `git status -s` and explain what each character means.

**Hint:**
<details>
<summary>Click to expand</summary>

The first column is staging area status. The second column is working directory status. `M` means modified compared to the reference. Space means matches the reference. The reference for the first column is HEAD. The reference for the second column is the index.

</details>

**Solution:**

```bash
git init
echo v1 > a.txt && echo v1 > b.txt && echo v1 > c.txt
git add -A && git commit -m "Initial"
echo v2 > a.txt && echo v2 > b.txt
git add a.txt
echo v3 > a.txt && echo v3 > b.txt
git status -s
```

Expected output:
```
MM a.txt
 M b.txt
```

Explanation:
- `MM a.txt`: First `M` means the index has a different version than HEAD (staged). Second `M` means the working directory has a different version than the index (unstaged).
- ` M b.txt`: Space in first column means the index matches HEAD (nothing staged). `M` in second column means the working directory is different from the index (unstaged changes).
- `c.txt` does not appear because it is clean in both columns.

## Tier 2 — Intermediate

### Exercise 1: Selective Commit with git add -p

You have a single file with two unrelated changes. Use interactive staging to commit only one of them.

**Task:**
1. `git init` and create `utils.py` with the following content and commit it:

```python
def add(a, b):
    return a + b

def multiply(a, b):
    return a * b
```

2. Add a new function `subtract` and a new function `divide` to the file. The file should now contain four functions.
3. Use `git add -p utils.py` to stage only the `subtract` function.
4. Commit the `subtract` addition.
5. Verify that `divide` remains unstaged.

**Hint:**
<details>
<summary>Click to expand</summary>

When you run `git add -p`, Git shows each hunk and asks what to do. Type `y` to stage the hunk, `n` to skip it. If both functions are in the same hunk, use `s` to split it into smaller hunks. You may need `e` to manually edit the hunk if splitting does not separate them.

</details>

**Solution:**

```bash
git init
cat > utils.py << 'EOF'
def add(a, b):
    return a + b

def multiply(a, b):
    return a * b
EOF
git add utils.py && git commit -m "Add add and multiply"
```

Now edit `utils.py` to add two new functions:

```bash
cat > utils.py << 'EOF'
def add(a, b):
    return a + b

def subtract(a, b):
    return a - b

def multiply(a, b):
    return a * b

def divide(a, b):
    return a / b
EOF
```

Run interactive staging:

```bash
git add -p utils.py
```

When Git prompts you with the first hunk (which may contain `subtract` or both functions), respond:
- If the hunk covers only `subtract`: type `y` to stage, then `n` for the `divide` hunk.
- If both are in one hunk: type `s` to split, then `y` for `subtract` and `n` for `divide`.
- If splitting does not separate them: type `e` to manually edit, delete the `divide` lines from the hunk, save, and exit.

```bash
git commit -m "Add subtract function"
git status -s
```

Expected output:
```
 M utils.py
```

The ` M` means `utils.py` has unstaged changes (the `divide` function).

### Exercise 2: Unstage and Discard Changes

You staged a file but then realize you want to discard both the staged version and the working directory changes.

**Task:**
1. `git init` and create `database.py` with `DB_NAME = "prod"` and commit it.
2. Edit the file to `DB_NAME = "dev"` and `DEBUG = True` and stage it.
3. Edit the file again to `DB_NAME = "staging"` and `DEBUG = False`.
4. Now decide you want to reset everything back to the committed version.
5. Unstage the file, then discard the working directory changes.
6. Verify the file content matches the original commit.

**Hint:**
<details>
<summary>Click to expand</summary>

First unstage with `git reset HEAD database.py`. Then discard working directory changes with `git checkout -- database.py`. These two commands are often chained: `git checkout HEAD database.py` does both at once.

</details>

**Solution:**

```bash
git init
echo 'DB_NAME = "prod"' > database.py
git add database.py && git commit -m "Set database to prod"
echo -e 'DB_NAME = "dev"\nDEBUG = True' > database.py
git add database.py
echo -e 'DB_NAME = "staging"\nDEBUG = False' > database.py
git status -s
```

Expected: `MM database.py`

Now reset everything:

```bash
git reset HEAD database.py
git checkout -- database.py
# Or in one step: git checkout HEAD database.py
cat database.py
```

Final content:
```
DB_NAME = "prod"
```

`git status -s` shows nothing — the file is clean.

### Exercise 3: Stage a Deletion

You want to commit the deletion of a file using the staging area.

**Task:**
1. Create a repository with `old.py` and `new.py`, both committed.
2. Delete `old.py` from disk.
3. Stage the deletion so that the commit will record it.
4. Commit the deletion.
5. Verify that `old.py` is no longer in the repository but `new.py` still is.

**Hint:**
<details>
<summary>Click to expand</summary>

You can stage a deletion with `git add old.py` (Git detects the file is gone) or `git rm old.py`. Both update the index to remove the file. Use `git rm` when you intend to delete; use `git add` when the deletion happened accidentally.

</details>

**Solution:**

```bash
git init
echo 'old code' > old.py
echo 'new code' > new.py
git add -A && git commit -m "Add old and new"
rm old.py
git rm old.py
git status -s
```

Expected output:
```
D  old.py
```

The `D` in the first column means "deleted from the index."

```bash
git commit -m "Remove old.py"
git ls-files
```

Expected output:
```
new.py
```

`old.py` is gone from the repository. `new.py` is still tracked.

### Exercise 4: Stage Changes for a Directory Subset

You have a project with two directories. Stage changes from only one directory.

**Task:**
1. Create a repository with `src/app.py` and `tests/test_app.py`, both committed.
2. Edit both files.
3. Stage only the changes in the `tests/` directory.
4. Verify that `src/app.py` changes are not staged.
5. Commit the test changes.

**Hint:**
<details>
<summary>Click to expand</summary>

`git add tests/` stages everything under the `tests/` directory. `git add src/` would stage the other directory. You can also use `git add tests/test_app.py` for a single file.

</details>

**Solution:**

```bash
git init
mkdir src tests
echo 'def main(): pass' > src/app.py
echo 'def test_main(): pass' > tests/test_app.py
git add -A && git commit -m "Initial structure"
echo 'def main(): return 42' > src/app.py
echo 'def test_main(): assert main() == 42' > tests/test_app.py
git add tests/
git status -s
```

Expected output:
```
 M src/app.py
M  tests/test_app.py
```

`src/app.py` has ` M` — unstaged. `tests/test_app.py` has `M ` — staged.

```bash
git commit -m "Update tests"
```

After commit, `git status -s` shows only ` M src/app.py`.

## Tier 3 — Advanced

### Exercise 1: The Two-Version Trap

You have a config file where you need to commit a specific change but preserve a later, unrelated change in the working directory.

**Scenario:**
You are working on `deploy.yml`. The file has three settings: `region`, `instance_count`, and `log_level`. You need to commit a change to `region` from `us-east-1` to `eu-west-1`. But you also changed `log_level` from `info` to `debug` for local testing, and you do not want to commit that yet.

**Task:**
1. Create a repository with `deploy.yml` containing:
   ```yaml
   region: us-east-1
   instance_count: 3
   log_level: info
   ```
   Commit it.
2. Edit the file to:
   ```yaml
   region: eu-west-1
   instance_count: 3
   log_level: debug
   ```
3. Stage only the `region` change.
4. Commit the region change.
5. Verify that the working directory still has `log_level: debug` and that the commit contains only the region change.

**Solution Walkthrough:**

This is a real scenario where `git add -p` is the right tool. The full-file `git add` would stage everything, including the `log_level` change you want to keep local.

```bash
git init
cat > deploy.yml << 'EOF'
region: us-east-1
instance_count: 3
log_level: info
EOF
git add deploy.yml && git commit -m "Initial deploy config"
```

Edit the file:

```bash
cat > deploy.yml << 'EOF'
region: eu-west-1
instance_count: 3
log_level: debug
EOF
```

Now stage only the region change:

```bash
git add -p deploy.yml
```

Git will present a hunk like:
```
diff --git a/deploy.yml b/deploy.yml
index abc1234..def5678 100644
--- a/deploy.yml
+++ b/deploy.yml
@@ -1,4 +1,4 @@
-region: us-east-1
+region: eu-west-1
 instance_count: 3
-log_level: info
+log_level: debug
```

This is one hunk. To stage only the region change, you need to split it:
- Type `s` to split.
- First split hunk: the `region` line change. Type `y` to stage.
- Second split hunk: the `log_level` line change. Type `n` to skip.

If `s` does not split cleanly (because the lines are too close together), type `e` to manually edit the hunk. Delete the `-log_level: info` and `+log_level: debug` lines from the hunk, then save and exit.

```bash
git commit -m "Change deploy region to eu-west-1"
git diff --staged
```

The staged diff shows only the region change:
```
diff --git a/deploy.yml b/deploy.yml
index abc1234..ghi9012 100644
--- a/deploy.yml
+++ b/deploy.yml
@@ -1,4 +1,4 @@
-region: us-east-1
+region: eu-west-1
 instance_count: 3
 log_level: info
```

The working directory still has `log_level: debug`:

```bash
git diff
```

Shows only the log level change. The region change is already committed.

**Note:** There are alternative approaches. You could manually edit the file to only change the region, commit that, then re-apply the log level change. You could use `git stash` to temporarily save the log level change, commit the region change, then pop the stash. The `git add -p` approach is the most direct because it avoids round-tripping through temporary states.

### Exercise 2: Split a Large Commit into Two

You made many changes at once and now realize they belong in two separate commits.

**Scenario:**
You have a `blog_post.py` file that contains both a bug fix and a new feature. You need to commit the bug fix first and the feature second.

**Task:**
1. Create a repository with `blog_post.py` containing:
   ```python
   def get_title():
       return "My Blog"
   
   def get_body():
       return "Content here"
   ```
   Commit it.
2. Edit the file to:
   ```python
   TITLE = "My Blog"
   
   def get_title():
       return TITLE.strip()
   
   def get_body():
       return "Content here with typo fixed"
   
   def get_word_count():
       return len(get_body().split())
   ```
3. Use the staging area to commit the changes in two steps:
   - First commit: the bug fix (the `strip()` call and the typo fix in `get_body`).
   - Second commit: the new feature (the `TITLE` constant and the `get_word_count` function).
4. Verify the commit history has two logical commits.

**Solution Walkthrough:**

This is a judgment call. The changes are mixed across the file, so you have to decide how to split them.

One approach: stage the file, then use `git add -p` to separate the hunks. But the bug fix lines and the feature lines may be in the same hunks and may not split cleanly.

A more reliable approach: stage the entire file, then use `git reset -p` to unstage the feature lines, leaving only the bug fix lines staged. `git reset -p` works like `git add -p` but in reverse — it walks through staged hunks and asks if you want to unstage each one.

```bash
git init
cat > blog_post.py << 'EOF'
def get_title():
    return "My Blog"

def get_body():
    return "Content here"
EOF
git add blog_post.py && git commit -m "Initial blog post"
```

Edit the file:

```bash
cat > blog_post.py << 'EOF'
TITLE = "My Blog"

def get_title():
    return TITLE.strip()

def get_body():
    return "Content here with typo fixed"

def get_word_count():
    return len(get_body().split())
EOF
```

Stage everything, then selectively unstage:

```bash
git add blog_post.py
git reset -p HEAD blog_post.py
```

Git will present each hunk. Unstage the feature-related hunks (the `TITLE` constant and the `get_word_count` function) and keep the bug fix hunks (the `strip()` call and the typo fix).

Respond to each prompt:
- For the hunk containing `TITLE = "My Blog"`: type `y` to unstage (meaning: do not include this in the first commit).
- For the hunk containing `return TITLE.strip()`: type `n` to keep staged (this is the bug fix).
- For the hunk containing the typo fix: type `n` to keep staged.
- For the hunk containing `get_word_count`: type `y` to unstage.

```bash
git commit -m "Fix title stripping and body typo"
git diff --staged
```

This shows the remaining staged changes (the feature). Commit them:

```bash
git add blog_post.py
git commit -m "Add TITLE constant and word count feature"
git log --oneline
```

Expected output:
```
2a3b4c5 Add TITLE constant and word count feature
1a2b3c4 Fix title stripping and body typo
abc1234 Initial blog post
```

**Note:** Other approaches exist. You could commit the bug fix directly from the editor, then re-add the feature. You could stash the feature, commit the fix, then pop and commit the feature. The `git add` / `git reset -p` approach keeps everything in the staging area and avoids temporary file states.

### Exercise 3: Recover from a Premature Commit — You Forgot to Stage a File

You committed but forgot to stage one file. The commit went through without it.

**Scenario:**
You have three files: `index.html`, `style.css`, and `app.js`. You edited all three, but you only staged `index.html` and `style.css` before committing. `app.js` was left unstaged — its changes remain in the working directory but were not included in the commit.

**Task:**
1. Create a repository with all three files committed.
2. Edit all three files.
3. Stage only `index.html` and `style.css` and commit.
4. Realize your mistake: `app.js` was supposed to be in the same commit.
5. Fix it without creating a separate commit message.

**Solution Walkthrough:**

You have a few options. The cleanest approach is to amend the commit.

```bash
git init
echo '<h1>Title</h1>' > index.html
echo 'body { color: black; }' > style.css
echo 'console.log("v1");' > app.js
git add -A && git commit -m "Initial commit"
```

Edit all three:

```bash
echo '<h1>Updated Title</h1>' > index.html
echo 'body { color: blue; }' > style.css
echo 'console.log("v2");' > app.js
```

Stage only two and commit:

```bash
git add index.html style.css
git commit -m "Update styles and content"
git status -s
```

Expected: ` M app.js` — the app.js change was left behind.

Now fix it:

```bash
git add app.js
git commit --amend --no-edit
```

`git commit --amend` replaces the most recent commit with a new commit that includes everything in the current staging area plus whatever was in the original commit. The `--no-edit` flag keeps the original commit message.

```bash
git log --oneline
```

Expected: one commit shown (the amended one, with a new hash), not two.

```bash
git diff --staged HEAD~1
```

This shows the combined diff of all three files, confirming `app.js` is now part of the commit.

**Warning:** Never amend a commit that has already been pushed to a shared branch. Amending rewrites history. If the commit is only local, amending is safe and clean.

**Note:** Alternative approaches include making a second commit with the message "Forgot to include app.js" (creates clutter in the log) or using `git reset --soft HEAD~1` to undo the commit while keeping the changes staged, then re-staging and re-committing (more steps, same result). The amend approach is the shortest path.
