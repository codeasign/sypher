## Tier 1 — Beginner

### Exercise 1: Trigger on tag push

**Task:** Create a workflow that only runs when you push a tag matching the pattern `v*.*.*` (e.g., `v1.0.0`, `v2.3.1`).

**Hint:** Use `tags:` under `push:` with a wildcard pattern.

<details>
<summary>Solution</summary>

```yaml
name: Tag Release
on:
  push:
    tags:
      - 'v*.*.*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Creating release for ${{ github.ref_name }}"
```

Test it by creating a tag locally and pushing it:
```bash
git tag v1.0.0
git push origin v1.0.0
```
</details>

### Exercise 2: Trigger on issue comment

**Task:** Create a workflow that runs when someone comments on an issue. The workflow should print the comment body and the issue number.

**Hint:** The `issue_comment` event triggers on both issue and PR comments. Use `github.event.issue.number` and `github.event.comment.body`.

<details>
<summary>Solution</summary>

```yaml
name: Comment Handler
on:
  issue_comment:
    types: [created]

jobs:
  respond:
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "New comment on issue #${{ github.event.issue.number }}"
          echo "Comment: ${{ github.event.comment.body }}"
```

To test, create an issue in your repository and leave a comment on it.
</details>

### Exercise 3: Trigger on release published

**Task:** Create a workflow that runs when a release is published. It should check out the code and print the release tag name.

**Hint:** Use the `release` event with `types: [published]`.

<details>
<summary>Solution</summary>

```yaml
name: Release Published
on:
  release:
    types: [published]

jobs:
  handle-release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Release ${{ github.event.release.tag_name }} published"
```

To test, create a release on GitHub: go to your repo, click Releases, then "Create a new release."
</details>

## Tier 2 — Intermediate

### Exercise 4: Combine multiple events with different filters

**Task:** Create a single workflow that:
- Runs on push to `main` or `develop`
- Runs on pull request to `main`
- Can be triggered manually
- Does NOT run when only `*.md` files are changed

**Hint:** Use a list of events under `on:` and combine `paths-ignore` with `push`.

<details>
<summary>Solution</summary>

```yaml
name: Multi-Event Quality
on:
  push:
    branches:
      - main
      - develop
    paths-ignore:
      - '**.md'
  pull_request:
    branches:
      - main
  workflow_dispatch:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Triggered by ${{ github.event_name }} on ${{ github.ref_name }}"
```

Test each trigger: push a `.py` file (runs), push a `README.md` (skipped), open a PR to main (runs), manually trigger (runs).
</details>

### Exercise 5: Use `workflow_run` with conclusion check

**Task:** Create two workflows. The first runs tests on push. The second runs only if the first workflow succeeded (not if it failed or was cancelled).

**Hint:** Use `types: [completed]` with an `if` condition checking `github.event.workflow_run.conclusion`.

<details>
<summary>Solution</summary>

File `.github/workflows/test.yml`:

```yaml
name: Test Suite
on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Running tests..."
      - run: exit 0  # Simulate success
```

File `.github/workflows/deploy-on-success.yml`:

```yaml
name: Deploy After Tests
on:
  workflow_run:
    workflows: ["Test Suite"]
    types: [completed]

jobs:
  deploy:
    runs-on: ubuntu-latest
    if: github.event.workflow_run.conclusion == 'success'
    steps:
      - run: echo "Tests passed — deploying!"
```

If the test workflow fails, the deploy workflow runs but is skipped (status: "Skipped"). If it succeeds, the deploy workflow runs all its steps.
</details>

### Exercise 6: Conditional path filter with negation

**Task:** Create a workflow that runs on push to any branch BUT only when files outside the `docs/` directory are changed. If only `docs/` files are changed, the workflow should be skipped.

**Hint:** Use `paths-ignore` under `push:`.

<details>
<summary>Solution</summary>

```yaml
name: Build Skip Docs
on:
  push:
    paths-ignore:
      - 'docs/**'

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Building because non-doc files changed"
```

To test: push a change to `docs/index.md` (skipped), then push a change to `src/main.py` (runs).
</details>

## Tier 3 — Advanced

### Exercise 7: Design a CI workflow with smart trigger decisions

**Scenario:** Your team has a monorepo with three components: `frontend/`, `backend/`, and `docs/`. Design a single workflow file that:

- Runs frontend tests only when `frontend/` files change
- Runs backend tests only when `backend/` files change
- Runs both when files in the root change
- Does not run anything when only `docs/` files change
- Can be manually triggered with a choice of which component to test
- Posts a summary of which tests ran

<details>
<summary>Solution</summary>

```yaml
name: Monorepo CI
on:
  push:
    paths-ignore:
      - 'docs/**'
  workflow_dispatch:
    inputs:
      component:
        description: 'Component to test'
        type: choice
        options:
          - all
          - frontend
          - backend

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      frontend: ${{ steps.check.outputs.frontend }}
      backend: ${{ steps.check.outputs.backend }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 2
      - id: check
        run: |
          if [ "${{ github.event_name }}" = "workflow_dispatch" ]; then
            echo "frontend=${{ github.event.inputs.component == 'all' || github.event.inputs.component == 'frontend' && 'true' || 'false' }}"
            echo "backend=${{ github.event.inputs.component == 'all' || github.event.inputs.component == 'backend' && 'true' || 'false' }}"
          else
            git diff --name-only HEAD~1 HEAD > changed-files.txt
            cat changed-files.txt
            echo "frontend=$(grep -c '^frontend/' changed-files.txt || true)" >> $GITHUB_OUTPUT
            echo "backend=$(grep -c '^backend/' changed-files.txt || true)" >> $GITHUB_OUTPUT
          fi

  frontend-tests:
    needs: detect-changes
    if: needs.detect-changes.outputs.frontend > 0
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running frontend tests..."

  backend-tests:
    needs: detect-changes
    if: needs.detect-changes.outputs.backend > 0
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running backend tests..."
```

This is one strong approach. Other valid designs include: using separate workflow files per component, using a matrix strategy with `fromJson()`, or using `paths:` filters in separate `on:` blocks.
</details>

### Exercise 8: Fix a workflow with incorrect triggers

**Scenario:** The following workflow is supposed to:
1. Run on push to `main` (full test suite)
2. Run on push to any feature branch (only lint)
3. Run on pull request to `main` (full test suite + deploy to staging)
4. Run every Monday at 3 AM (dependency audit)

Find and fix the errors:

```yaml
name: CI Pipeline
on:
  push:
    branches: [main]
  pull_request:
    branches:
    schedule:
      - cron: '0 3 * * 1'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Linting..."
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running tests..."
  deploy:
    needs: test
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying to staging..."
```

<details>
<summary>Solution</summary>

There are at least 3 issues:

1. **Missing `branches:` under `pull_request:`**: The `branches:` key has a value (`[main]`) but there is no `branches:` key shown — the `schedule:` key is parsed as a sibling instead of a separate event because `pull_request:` has no `branches:` key explicitly separating it. The YAML indentation makes `schedule` look like a sibling of `pull_request` which is correct, but actually looking closer... `branches:` is missing under `pull_request:`.

Actually, looking more carefully at the indentation:
```yaml
  pull_request:
    branches:
```
Here `branches:` is empty — there's no value. Then:
```yaml
    schedule:
      - cron: '0 3 * * 1'
```
`schedule` is indented at the same level as `branches`, making it a child of `pull_request` instead of a top-level event key. But `schedule` is not a valid sub-key of `pull_request`, so the parser would throw an error.

2. **Push to feature branches does not trigger**: The `push:` block only triggers on `main`. To trigger on feature branches, you need to add a wildcard.

3. **The deploy job runs on every PR trigger (push to PR)**: The `if: github.event_name == 'pull_request'` is correct but note that when you push to a PR branch, both `push` and `pull_request` events fire.

Fixed version:

```yaml
name: CI Pipeline
on:
  push:
    branches:
      - main
      - 'feature/**'
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 3 * * 1'

jobs:
  lint:
    if: github.event_name == 'push' && github.ref_name != 'main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Linting feature branch..."
  test:
    runs-on: ubuntu-latest
    if: github.event_name != 'schedule'
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running full tests..."
  deploy:
    needs: test
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying to staging..."
  audit:
    if: github.event_name == 'schedule'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running dependency audit..."
```
</details>

### Exercise 9: Design a release pipeline trigger strategy

**Scenario:** Design a triggering strategy for a production release pipeline that satisfies ALL of these constraints:

1. Deploy to staging when a PR is merged to `main` (after tests pass)
2. Deploy to production when a tag `v*.*.*` is pushed to the repository
3. Allow manual rollback to any previous version via `workflow_dispatch`
4. Run security audit every night at 2 AM
5. Do NOT deploy to production from a fork
6. Cancel in-progress deployments if a newer run starts for the same environment

<details>
<summary>Solution</summary>

```yaml
name: Release Pipeline
on:
  push:
    branches: [main]
    paths-ignore:
      - 'docs/**'
  push:
    tags:
      - 'v*.*.*'
  schedule:
    - cron: '0 2 * * *'
  workflow_dispatch:
    inputs:
      target:
        description: 'Deploy target'
        required: true
        default: 'production'
        type: choice
        options:
          - staging
          - production
      version:
        description: 'Version or commit SHA'
        required: true

concurrency:
  group: deploy-${{ github.event.inputs.target || 'staging' }}
  cancel-in-progress: true

jobs:
  security-audit:
    if: github.event_name == 'schedule'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running security audit..."

  test:
    if: github.event_name != 'schedule'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Running full test suite..."

  deploy-staging:
    needs: test
    if: github.event_name == 'push' && github.ref_name == 'main' && github.event.pusher.name != github.event.sender.login
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying to staging..."

  deploy-production:
    needs: test
    if: >
      (github.event_name == 'push' && startsWith(github.ref, 'refs/tags/v'))
      || (github.event_name == 'workflow_dispatch' && github.event.inputs.target == 'production')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying to production with version ${{ github.event.inputs.version || github.ref_name }}"

  deploy-staging-manual:
    needs: test
    if: github.event_name == 'workflow_dispatch' && github.event.inputs.target == 'staging'
    runs-on: ubuntu-latest
    steps:
      - run: echo "Manually deploying to staging with version ${{ github.event.inputs.version }}"
```

Note: The fork constraint (constraint 5) is handled implicitly — forks cannot push to the upstream repository's `main` branch or create tags, so only the `workflow_dispatch` pathway is relevant for forks. The concurrency group ensures constraint 6 by cancelling previous runs for the same target environment.
</details>
