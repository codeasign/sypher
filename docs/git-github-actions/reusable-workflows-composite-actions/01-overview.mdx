You maintain 20 microservice repositories, each with the same 50-line lint-test-deploy workflow copy-pasted from the last one. One day you need to add a new linting rule. You fix it in repo 1, fix it in repo 2, and by repo 5 you have already lost the will to live. By repo 20 you are seriously questioning your career choices. Copy-paste is not a deployment strategy, and in a CI/CD platform designed for automation, it is the worst possible approach.

## The Concept

Reusable workflows and composite actions are GitHub's answer to the "write once, use everywhere" problem. They are two different tools for two different levels of reuse.

A **reusable workflow** is a complete workflow file that another workflow can call using the `uses:` keyword. The reusable workflow lives in its own `.github/workflows/` file and exposes `on: workflow_call` as its trigger. The caller workflow treats it like a single job — it waits for the reusable workflow to complete, then continues with its own jobs. This is job-level reuse.

A **composite action** is a collection of steps packaged into a single action. Unlike a reusable workflow, a composite action runs inside the caller's job — it shares the same runner, the same file system, and the same environment. This is step-level reuse.

Think of a reusable workflow as a whole cooking show episode that you can embed in your TV channel. The episode runs in its own time slot, has its own host, and produces its own output. A composite action is a single recipe that you incorporate into your own cooking show episode — you are the host, and you just borrow the recipe steps.

There is a third option for reuse that you should know about: **local actions** (JavaScript or Docker actions published to the Marketplace). These are full-fledged actions written in TypeScript, JavaScript, or packaged as Docker containers. They are more powerful than composite actions but require more code to build and maintain. Composite actions are the simpler middle ground — they use the same YAML step syntax you already know from workflows, so there is no new language to learn.

### Composite Action Limitations

Composite actions have important constraints that set them apart from both reusable workflows and JavaScript actions:

- Every `run:` step inside a composite action must specify a `shell:` property. Workflow steps default to `bash` on Linux runners, but composite actions do not inherit a default shell.
- Composite actions cannot use `needs:`, `if:` at the job level, or `strategy:` — these are workflow-level constructs. Conditional logic goes on individual steps.
- A composite action cannot call another composite action directly unless the nested composite action is in the same repository and referenced with a relative path. Cross-repository nesting is not supported.
- Composite actions run with the caller's token permissions by default. You cannot scope down permissions inside the composite action itself — that must be done at the caller level.

## Command / Syntax Reference

| Key | Values | Example |
|-----|--------|---------|
| `on: workflow_call` | No value (marks the workflow as reusable) | `on: workflow_call` |
| `on: workflow_call: inputs:` | Map of input definitions | `on: workflow_call: inputs: node-version: { required: true, type: string }` |
| `on: workflow_call: secrets:` | Map of secret definitions | `on: workflow_call: secrets: api-token: { required: true }` |
| `on: workflow_call: outputs:` | Map of output definitions | `on: workflow_call: outputs: build-url: { value: ${{ jobs.build.outputs.url }}, description: "Build URL" }` |
| `jobs.<id>.uses:` | `owner/repo/.github/workflows/workflow.yml@ref` | `uses: my-org/shared-workflows/.github/workflows/ci.yml@v1` |
| `with:` | Map of input values passed to the reusable workflow | `with: node-version: 20` |
| `secrets: inherit` | Pass all caller secrets to the reusable workflow | `secrets: inherit` |
| `secrets:` | Map of specific secrets to pass | `secrets: api-token: ${{ secrets.API_TOKEN }}` |
| Composite `action.yml` | File at `.github/actions/<name>/action.yml` | `runs: using: composite steps: [...]` |
| Composite `runs.using: composite` | Required — tells GitHub this is a composite action | `runs: using: composite` |
| Composite `runs.steps` | List of steps (same as workflow steps but no `uses:` for other composites) | `runs: steps: - run: echo hello shell: bash` |
| Composite `inputs:` | Input definitions for the composite action | `inputs: greeting: { description: "Greeting", required: true, default: "Hello" }` |
| Composite `outputs:` | Output definitions for the composite action | `outputs: result: { description: "Result", value: ${{ steps.step1.outputs.result }} }` |

### Composite Action vs Reusable Workflow — Quick Comparison

| Aspect | Reusable Workflow | Composite Action |
|--------|-------------------|------------------|
| Scope | Whole job | Steps within a job |
| Runner | Separate runner instance | Same runner as caller |
| File location | `.github/workflows/*.yml` | `.github/actions/*/action.yml` |
| Trigger | `on: workflow_call` | `runs.using: composite` |
| Shares filesystem | No | Yes |
| Can use `actions/` marketplace | Yes | Yes |
| Can call other reusable workflows | Yes | No (nested composites supported) |
| Max nesting | 4 levels deep | 9 levels deep |
| Secrets support | Native via `secrets:` | Must use `inputs` with `${{ inputs.secret-name }}` |

## How It Actually Works

### Step 1: The caller workflow triggers a workflow_dispatch event to the reusable workflow

When GitHub encounters a job with `uses:` pointing to another workflow file, it does not execute that job locally. Instead, it creates a new workflow run for the reusable workflow using the `workflow_dispatch` event type. The caller passes inputs, secrets, and the GitHub token as part of this dispatch event.

### Step 2: The caller pauses and waits for the reusable workflow to complete

The caller job enters a "waiting" state. It does not consume a runner slot — it simply waits for the reusable workflow's run to complete. This is similar to how a `needs:` dependency works, but the dependency is an entire workflow run rather than another job in the same workflow.

During this waiting period, the caller's workflow run shows the job as "Expected — Waiting for workflow to complete." You can see the reusable workflow's run link in the caller's job UI. Clicking it takes you to the reusable workflow run in the shared repository, where you can see its individual jobs and step logs.

### Step 3: The reusable workflow runs on a separate runner instance

The reusable workflow gets its own runner, its own workspace, and its own checkout of the repository that contains the reusable workflow file (not the caller's repository). This is a critical distinction: the reusable workflow checks out the **workflow's repository**, not the caller's, unless you explicitly check out the caller's code using `actions/checkout` with the `repository` input.

### Step 4: Outputs propagate back to the caller

When the reusable workflow completes, it can publish outputs using `on: workflow_call: outputs:`. These outputs are defined at the workflow level and reference job outputs inside the reusable workflow. The caller accesses them via `needs.<job-id>.outputs.<output-name>`.

Outputs are defined as a two-level chain. First, the job inside the reusable workflow declares an `outputs:` block at the job level. Then, the `on: workflow_call: outputs:` block at the top level maps those job outputs to workflow-level outputs. This indirection exists because the caller does not know about individual jobs inside the reusable workflow — it only sees the workflow-level outputs.

```yaml
# Inside the reusable workflow
on:
  workflow_call:
    outputs:
      build-url:
        description: "URL of the build artifact"
        value: ${{ jobs.build.outputs.url }}

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      url: ${{ steps.upload.outputs.artifact-url }}
    steps:
      - id: upload
        run: echo "artifact-url=https://example.com/build/${{ github.run_id }}" >> $GITHUB_OUTPUT
```

The caller workflow then reads the output:

```yaml
jobs:
  ci:
    uses: org/shared/.github/workflows/ci.yml@v1
  deploy:
    needs: ci
    steps:
      - run: echo "Build URL: ${{ needs.ci.outputs.build-url }}"
```

### Step 5: Secrets are passed securely

Secrets can be passed in two ways. The `secrets: inherit` keyword passes every secret the caller has access to the reusable workflow. You can also pass specific secrets with `secrets: secret-name: ${{ secrets.SECRET_NAME }}`. GitHub encrypts secrets in transit and they are never exposed in logs, even when using `secrets: inherit`.

The `secrets: inherit` approach is convenient but comes with a security warning. When you use `inherit`, the reusable workflow can access every secret available to the caller — including secrets from environments, organizations, and repositories that the reusable workflow should not need. If the reusable workflow is compromised or logs the wrong thing, all those secrets could be exposed. Always prefer explicit secret passing, listing only the specific secrets the reusable workflow needs:

```yaml
# Explicit — only passes what is needed
jobs:
  ci:
    uses: org/shared/.github/workflows/ci.yml@v1
    secrets:
      npm-token: ${{ secrets.NPM_TOKEN }}
      docker-username: ${{ secrets.DOCKER_USERNAME }}

# Convenient but risky — passes everything
jobs:
  ci:
    uses: org/shared/.github/workflows/ci.yml@v1
    secrets: inherit
```

### Step 6: The `@ref` determines which version of the reusable workflow runs

The `@ref` suffix in the `uses:` line is critical. Every caller pins a specific version of the reusable workflow. You have three options for the ref:

- **Branch ref (`@main`, `@develop`)** — always uses the latest code on that branch. Callers pick up changes automatically. Convenient during development but dangerous in production — a broken push to `main` breaks every caller simultaneously.

- **Tag ref (`@v1`, `@v1.2.3`)** — points to a specific commit via a git tag. Tags are immutable (or at least should be treated as such). This gives callers stability: they upgrade explicitly by changing the tag in their `uses:` line.

- **Commit SHA (`@a1b2c3d4e5f6...`)** — the most specific ref. A full commit SHA (40 characters) pins the workflow to an exact version forever. No branch force-push or tag deletion can change what runs. This is the safest option for production workflows but requires updating every caller when you want to roll out a change.

The recommended strategy: tag your reusable workflow releases with semantic versioning (`v1.0.0`, `v1.1.0`, `v2.0.0`), move a major-version tag (`v1`) to point at the latest `v1.x.x` release, and have callers use `@v1` to get all patch and minor updates automatically without breaking changes.

### Step 7: Composite actions run inline in the caller's job

Unlike reusable workflows, composite actions execute in the same runner and workspace as the caller. The composite action's `action.yml` file defines `runs.using: composite` and a list of steps. These steps run as if they were defined directly in the caller's workflow. The composite action can access the caller's filesystem, environment variables, and even the `$GITHUB_OUTPUT` file.

## When to Use / When Not To

### Use reusable workflows when:

- You have the exact same job (same steps, same runner, same configuration) running in multiple repositories
- You want to enforce a standardized CI/CD pipeline across your organization
- You need to run a workflow in a separate security context (different token permissions, different environment)
- You want to version your workflows (using `@v1`, `@v2` tags) and roll out changes gradually

### Do not use reusable workflows when:

- You only need to reuse a few steps, not an entire job — a composite action is simpler
- The reusable workflow would have only one caller — there is no benefit over a regular workflow
- You need to access the caller's filesystem or workspace — reusable workflows run in a separate workspace

### Use composite actions when:

- You have a group of steps that are repeated across multiple jobs in the same workflow or across workflows
- The steps need to share the same runner environment with the caller
- You want to encapsulate complex setup logic (installing dependencies, configuring tools, setting up services)
- You want to create a reusable step that can be shared across repositories without requiring a separate workflow file

### Do not use composite actions when:

- You need to run steps on a different runner OS or architecture — use a reusable workflow instead
- Your reusable logic spans multiple jobs — composite actions only work within a single job
- You need to pass secrets securely — composite actions handle secrets through inputs, which is less secure than the `secrets:` mechanism in reusable workflows
- The action is simpler than a single `run:` command — do not over-abstract

### Use neither when:

- The duplicated code is a single `run:` command shared between two jobs — just use a YAML anchor instead
- The task is a one-off integration that will never be reused — premature abstraction creates maintenance overhead
- You are not sure if the pattern will be reused — wait until you have three instances of duplication before abstracting
