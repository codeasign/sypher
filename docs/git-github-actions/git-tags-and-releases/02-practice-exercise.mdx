You will create a repository, simulate a release cycle with tags, push tags to a remote, check out a tag, and use `git describe` for versioning. Every command works identically on Windows, macOS, and Linux.

## Setup

```bash
mkdir -p ~/tags-lab
cd ~/tags-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create the initial project:

```bash
cat > app.py << 'EOF'
VERSION = "0.1.0"


def greet(name):
    return f"Hello, {name}!"
EOF
git add app.py
git commit -m "Initial project structure"
```

## The Walkthrough

### Step 1: Create a lightweight tag

```bash
git tag v0.1
git tag
```

Expected output:

```
v0.1
```

**Under the hood:** Git created `.git/refs/tags/v0.1` containing the current commit's hash. The tag is a 41-byte file.

### Step 2: Make more commits and create annotated tags

```bash
sed -i 's/VERSION = "0.1.0"/VERSION = "0.2.0"/' app.py
echo "def farewell(name):" >> app.py
echo "    return f'Goodbye, {name}!'" >> app.py
git add app.py
git commit -m "Add farewell function"
git tag -a v0.2 -m "Release v0.2 with farewell function"
```

Verify the annotated tag:

```bash
git tag
git show v0.2 --stat
```

Expected output — the tag shows the tagger info, date, message, and the commit diff:

```
tag v0.2
Tagger: Your Name <your.email@example.com>
Date:   ...

Release v0.2 with farewell function
...
```

### Step 3: Create a tag at a specific commit

Add more commits, then tag an older commit:

```bash
sed -i 's/VERSION = "0.2.0"/VERSION = "0.3.0"/' app.py
echo "def add(a, b):" >> app.py
echo "    return a + b" >> app.py
git add app.py
git commit -m "Add add function"
echo "def subtract(a, b):" >> app.py
echo "    return a - b" >> app.py
git add app.py
git commit -m "Add subtract function"
```

Now tag the "Add add function" commit as v0.3 (not the latest):

```bash
HASH=$(git log --oneline | grep "Add add function" | awk '{print $1}')
git tag -a v0.3 $HASH -m "Release v0.3 with add function"
git log --oneline --decorate
```

Expected output — `v0.3` is attached to the "Add add function" commit, not the tip:

```
abc1234 (HEAD -> main) Add subtract function
def5678 (tag: v0.3) Add add function
ghi9012 (tag: v0.2) Add farewell function
jkl3456 (tag: v0.1) Initial project structure
```

**Checkpoint:** `git tag -l "v0.*"` lists all three tags.

### Step 4: List tags with patterns

```bash
git tag -l "v0.*"
git tag -l "v0.[23]*"
```

The first command lists all v0.x tags. The second lists only v0.2 and v0.3.

### Step 5: Check out a tag (detached HEAD)

```bash
git checkout v0.2
cat app.py
```

Expected output — the code from v0.2, without the add/subtract functions:

```
VERSION = "0.2.0"
...
```

HEAD is now detached:

```bash
git log --oneline -1
```

Expected output — shows the v0.2 commit with `(HEAD, tag: v0.2)`.

**Warning:** If you make changes here and commit, they will be lost when you switch away. To keep changes:

```bash
git switch -c hotfix-from-v0.2
```

Create a branch from the tag, make the fix, then tag a new release.

### Step 6: Delete a tag locally

```bash
git checkout main
git tag -d v0.1
```

Expected output:

```
Deleted tag 'v0.1' (was jkl3456)
```

The tag is removed from `.git/refs/tags/`. The commit itself is not affected.

### Step 7: Push tags to a remote

Set up a bare remote and push:

```bash
cd ..
git init --bare remote.git
cd tags-lab
git remote add origin ../remote.git
git push origin main
git push origin --tags
```

Expected output:

```
 * [new tag]         v0.2 -> v0.2
 * [new tag]         v0.3 -> v0.3
```

The `v0.1` tag was deleted locally, so it was not pushed. Verify tags on the remote:

```bash
git ls-remote --tags origin
```

Expected output — shows the tag refs and their hashes.

### Step 8: Delete a tag on the remote

```bash
git push origin --delete v0.2
```

Or:

```bash
git push origin :refs/tags/v0.2
```

Verify:

```bash
git ls-remote --tags origin
```

### Step 9: Use `git describe`

```bash
git describe
```

Expected output — something like:

```
v0.3-1-gabc1234
```

This means: the nearest tag is `v0.3`, 1 commit ahead of it, with hash prefix `abc1234`. The `g` prefix is a Git convention.

Now create a scenario where describe shows a different format:

```bash
git tag -a v0.4 -m "Release v0.4"
git describe
```

Expected output:

```
v0.4
```

When HEAD is exactly on a tag, `git describe` returns just the tag name.

Now add some commits past the tag:

```bash
echo "# TODO" > TODO.md
git add TODO.md
git commit -m "Add TODO file"
git describe
```

Expected output:

```
v0.4-1-g<abbrev-hash>
```

## Common Errors at This Stage

### Error 1: `fatal: tag 'v1.0' already exists`

You tried to create a tag with a name that already exists. Git refuses to overwrite a tag.

**Fix:** If you are certain you want to move the tag, delete it first with `git tag -d v1.0`, then re-create it. If the tag was already pushed to the remote, also delete it there: `git push origin --delete v1.0`. Never reuse a tag name that was part of a published release.

### Error 2: `fatal: 'v1.0' is not a valid tag name`

The tag name you provided contains invalid characters. Tag names must not contain spaces, and they should not start with a `-`. Valid characters are letters, digits, dots, dashes, underscores, and forward slashes (for hierarchical names like `release/v1.0`).

**Fix:** Use a valid name like `v1.0`, `release-1.0`, or `v1.0.0-beta`.

### Error 3: HEAD detached at v1.0 — where did my changes go?

You checked out a tag, made commits, and then switched to another branch. Your commits are no longer visible because they were not on any branch.

**Fix:** If you made commits in detached HEAD state, find them with `git reflog` and create a branch: `git branch rescued <hash>`. To avoid this, always create a branch before making changes: `git switch -c my-branch v1.0`.

### Error 4: `git push origin --tags` pushed tags you did not want

You ran `--tags` and pushed all local tags, including experimental ones like `testing`, `wip`, or `backup-tag`. These are now visible to everyone who clones the repo.

**Fix:** Push tags individually with `git push origin <tagname>` for release tags. Use `git tag -l` to review your local tags before pushing. Delete unwanted remote tags with `git push origin --delete <tagname>`.

## What You Just Learned

- Lightweight tags are 41-byte pointers in `.git/refs/tags/` — permanent, non-moving references
- Annotated tags (`git tag -a`) store the tagger, date, and message as a Git object
- Tag at specific commits: `git tag <name> <commit>`
- Check out a tag puts you in detached HEAD — create a branch to keep changes
- `git push origin --tags` pushes all local tags; `git push origin <tag>` pushes one
- `git tag -d <name>` deletes locally; `git push origin --delete <name>` deletes on remote
- `git describe` shows the nearest tag and how many commits ahead you are — useful for build versioning
