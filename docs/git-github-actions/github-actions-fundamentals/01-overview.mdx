You deploy a hotfix on a Friday afternoon. Two hours later, your staging environment is broken, the test suite was never run, and you have no idea which step in the pipeline failed because nobody wrote it down. Every team member has their own "deploy script" that works on their machine but not on anyone else's. You need a single, repeatable, automated system that runs the same steps every time — and GitHub Actions is that system.

## The Concept

GitHub Actions is an automation platform built into GitHub that lets you define workflows — sequences of steps triggered by events in your repository — that run on GitHub-hosted or self-hosted runners.

Think of it as a programmable assembly line for your code. When you push a commit (the trigger), the line starts moving: a worker picks up the raw material (your code on a fresh runner), runs each station (lint, test, build, deploy), and either passes the finished product to the next station or stops the line with a red alarm. Every station runs identically every time because each run starts from a clean machine.

## Key YAML Properties

Every workflow is defined in a `.yml` file inside `.github/workflows/`. Here are the top-level keys:

| Key | Required | Purpose |
|-----|----------|---------|
| `name` | No | Display name for the workflow in the Actions tab |
| `run-name` | No | Per-run name shown in the run list (can use expressions) |
| `on` | Yes | Event or events that trigger the workflow |
| `on.schedule` | No | Cron syntax for scheduled triggers |
| `on.workflow_dispatch` | No | Manual trigger with optional inputs |
| `env` | No | Environment variables available to all jobs |
| `defaults` | No | Default settings for `run` shell or working directory |
| `concurrency` | No | Ensures only one run at a time per concurrency group |
| `jobs` | Yes | Map of job IDs to job definitions |
| `jobs.<id>.runs-on` | Yes | Runner type or label (e.g., `ubuntu-latest`) |
| `jobs.<id>.steps` | Yes | Ordered list of steps to execute |
| `jobs.<id>.needs` | No | List of jobs that must complete before this one |
| `jobs.<id>.if` | No | Condition for running the job |
| `jobs.<id>.strategy` | No | Matrix build configuration |
| `jobs.<id>.env` | No | Job-level environment variables |
| `steps[*].name` | No | Display name for the step |
| `steps[*].uses` | Conditional | Action to run (e.g., `actions/checkout@v4`) |
| `steps[*].run` | Conditional | Shell command to execute |
| `steps[*].with` | No | Input parameters for a `uses` action |
| `steps[*].env` | No | Step-level environment variables |

## How It Actually Works — Step by Step

### Step 1: An event triggers the workflow

You push a commit, open a pull request, or create an issue. GitHub's event system detects the action and checks every workflow file in `.github/workflows/` for a matching `on:` trigger. If a workflow's trigger matches the event, GitHub creates a new workflow run and assigns it a unique run ID.

### Step 2: GitHub provisions a runner

GitHub takes the `runs-on` value from your workflow (e.g., `ubuntu-latest`) and provisions a fresh virtual machine. This VM has nothing on it except the base OS image and pre-installed tools. Every run starts from the same clean slate — no leftover files, no stale dependencies, no "works on my machine."

### Step 3: The workflow file is parsed and jobs are scheduled

GitHub reads your `.yml` file, validates the YAML syntax, and builds a dependency graph from the `needs` fields. Jobs with no `needs` dependencies start immediately. Each job gets its own runner (or shares one if you use `runs-on: [self-hosted]` labels).

### Step 4: Steps execute in order within each job

Inside a job, steps run sequentially. If a step fails (non-zero exit code), the job stops and is marked as failed — unless you add `continue-on-error: true`. Each step runs in the same working directory on the same runner, so later steps can use files created by earlier steps (like compiled binaries or test reports).

### Step 5: The workflow run completes

When all jobs finish (success or failure), GitHub records the final status, sends any configured notifications, and makes the full log available in the Actions tab. The runner VM is destroyed. Logs are retained for 90 days by default (adjustable in repository settings).

## When to Use GitHub Actions / When Not To

### Use when:
- You need automated CI/CD for a GitHub-hosted repository
- You want to run tests, linting, or security scans on every push
- Your team needs consistent, documented build and deploy processes
- You need scheduled tasks (daily backups, weekly dependency updates)
- You want to automate repository management (labeling issues, closing stale PRs)

### Do not use when:
- Your repository is not on GitHub — use GitLab CI/CD, Jenkins, or CircleCI instead
- You need millisecond-latency automation — Actions has a ~5-10 second startup overhead per job
- Your builds require specific on-premise hardware or network access you cannot expose to GitHub — use self-hosted runners carefully, or consider a different platform
- You need fine-grained control over the execution environment beyond what the runner image provides — consider Docker containers or dedicated CI infrastructure
