You will set up a small repository, create a realistic branching history, and practice every `git log`, `git show`, and `git diff` flag from the overview. By the end you will be able to investigate any project's history with precision.

## Setup

Open your terminal and run these commands exactly. They work on Windows (Git Bash or PowerShell), macOS, and Linux.

```bash
# Create and enter a practice directory
mkdir git-history-lab
cd git-history-lab

# Initialize a new Git repository
git init

# Configure local identity for this repo
git config user.name "Practice User"
git config user.email "practice@example.com"

# Create a starter file and make the first commit
echo "# Project Dashboard" > README.md
git add README.md
git commit -m "chore: initial project setup with README"

# Add the core module
mkdir src
echo "function greet(name) { return 'Hello, ' + name; }" > src/greet.js
git add src/greet.js
git commit -m "feat: add greet function"

# Add a second feature
echo "function add(a, b) { return a + b; }" > src/math.js
git add src/math.js
git commit -m "feat: add math utility functions"
```

Now create a branch and add commits to simulate branching history:

```bash
# Create and switch to a feature branch
git checkout -b feature/dashboard

# Make changes on the feature branch
echo "function renderDashboard() { return 'Dashboard loaded'; }" > src/dashboard.js
git add src/dashboard.js
git commit -m "feat: add dashboard render function"

echo "function loadWidgets() { return ['chart', 'table']; }" >> src/dashboard.js
git add src/dashboard.js
git commit -m "feat: add widget loader to dashboard"

# Switch back to main and make a parallel commit
git checkout main
echo "# Getting Started" > USAGE.md
git add USAGE.md
git commit -m "docs: add usage instructions"

# Merge the feature branch
git merge feature/dashboard --no-edit
```

Add more history to practice advanced filtering:

```bash
# Simulate a hotfix branch
git checkout -b hotfix/critical-bug
echo "fixed" > fix.txt
git add fix.txt
git commit -m "hotfix: resolve critical null-pointer in auth"

# Back to main, merge the hotfix
git checkout main
git merge hotfix/critical-bug --no-edit

# Add commits by different fake authors for author filtering practice
git config user.name "Alice Developer"
git commit --allow-empty -m "style: format all files with prettier"
git config user.name "Bob Tester"
git commit --allow-empty -m "test: add integration tests for dashboard"
git config user.name "Practice User"
git commit --allow-empty -m "chore: update dependencies"

echo "Setup complete. You now have a repository with 11 commits, 3 branches, and 2 merge points."
```

## The Walkthrough

### Step 1: View the Full History

```bash
git log
```

Expected output shows all 11 commits with full hashes, authors, dates, and messages, newest first:

```
commit a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b (HEAD -> main)
Author: Practice User <practice@example.com>
Date:   Thu Jul 2 14:30:00 2026 +0000
    chore: update dependencies
...
```

**Explanation:** Without flags, `git log` shows every detail. This is useful for reading commit messages but takes up a lot of screen space — use it when you need full context (author, timestamp, full message body).

**Checkpoint:** You should see at least 8 commits (your numbers may vary based on exact timing). Press `q` to exit the pager.

### Step 2: Use --oneline for a Compact View

```bash
git log --oneline
```

Expected output (hashes will differ):

```
a1b2c3d chore: update dependencies
b2c3d4e test: add integration tests for dashboard
c3d4e5f style: format all files with prettier
d4e5f6g Merge branch 'hotfix/critical-bug'
e5f6a7b hotfix: resolve critical null-pointer in auth
f6a7b8c Merge branch 'feature/dashboard'
a7b8c9d docs: add usage instructions
b8c9d0e feat: add widget loader to dashboard
c9d0e1f feat: add dashboard render function
d0e1f2a feat: add math utility functions
e1f2a3b feat: add greet function
f2a3b4c chore: initial project setup with README
```

**Explanation:** Each commit is a single line: 7-character abbreviated hash + subject. This is your everyday view for scanning history quickly.

**Checkpoint:** Count the commits. You should see approximately 11-12 (depending on merge commits). Notice how merge commits appear as their own entries.

### Step 4: Filter by Author

```bash
git log --oneline --author="Alice"
```

Expected output:

```
c3d4e5f style: format all files with prettier
```

**Explanation:** The `--author` flag does a substring match against the author name and email. It matches "Alice Developer" because the string "Alice" appears in the name.

Now try:

```bash
git log --oneline --author="Bob"
```

Expected output:

```
b2c3d4e test: add integration tests for dashboard
```

**Checkpoint:** Running both commands should give you exactly one commit each. If you get more, the repo may have additional authors from merge commits (which use the merger's identity, not the original author).

### Step 5: Filter by Date

```bash
git log --oneline --since="2026-06-01"
```

Since all commits are from today, this shows everything. Now simulate a time filter that narrows output:

```bash
git log --oneline --since="2 minutes ago"
```

**Explanation:** `--since` uses the author timestamp. Git accepts human-readable dates ("2 minutes ago", "yesterday", "last week") and ISO 8601 timestamps ("2026-06-30T14:00:00"). The `--until` flag works the same way but sets the upper bound.

**Checkpoint:** The `--since="2 minutes ago"` command should show only the commits you just made (0-2 commits depending on timing). This demonstrates how to narrow to a specific time window.

### Step 6: Search Commit Messages

```bash
git log --oneline --grep="feat"
```

Expected output (exact matches depend on your commit messages):

```
d0e1f2a feat: add math utility functions
e1f2a3b feat: add greet function
a7b8c9d feat: add widget loader to dashboard
b8c9d0e feat: add dashboard render function
```

**Explanation:** `--grep` searches the commit message (subject and body). Combine it with `-i` for case-insensitive search. Unlike `--author`, which matches a specific field, `--grep` matches anywhere in the message. Use `--all-match` when providing multiple `--grep` patterns to require all of them to match.

**Checkpoint:** Verify the count matches commits whose message contains "feat". If you included "feat" in merge commit messages, those will appear too.

### Step 7: View Full Patch Content

```bash
git log -p --oneline -2
```

Expected output shows the last 2 commits with their full diffs:

```
a1b2c3d chore: update dependencies
diff --git a/package.json b/package.json
...

b2c3d4e test: add integration tests for dashboard
diff --git a/tests/dashboard.test.js b/tests/dashboard.test.js
new file mode 100644
index 0000000..e69de29
```

**Explanation:** `-p` adds the complete diff for each commit. `-2` limits to the last 2 commits. This is useful when investigating exactly what changed in a specific commit but overwhelming for large histories — always combine `-p` with a limit (`-n`) or a path filter.

**Checkpoint:** You should see the commit metadata lines followed by `diff --git` headers and patch hunks (lines starting with `+`, `-`, or space). Each file change is shown as a unified diff.

### Step 8: Use git show to Inspect a Single Commit

```bash
git show HEAD~3
```

Expected output shows the commit 3 steps before HEAD: its full hash, author, date, message, and the complete diff of what changed in that commit.

```bash
git show main~2 --stat
```

Expected output shows the commit 2 steps before main's tip, but with `--stat` instead of the full diff — just the file summary.

**Explanation:** `HEAD~3` means "go back 3 parent links from HEAD." You can use any ref: branch names, tags, relative references like `main~5`, or absolute hashes. `git show` is your go-to for answering "what exactly was in that commit?" without filtering through `git log`.

**Checkpoint:** `HEAD~3` should show a commit near the middle of your history. The `--stat` variant shows only the file-change summary — compare the output lengths to see how much `-p` adds.

### Step 9: Compare Two Branches

```bash
git diff main..feature/dashboard
```

Since the dashboard branch is already merged, this may show an empty diff. For a meaningful comparison, compare the current state with the initial commit:

```bash
git diff HEAD~10..HEAD --stat
```

**Explanation:** `git diff` compares the tree objects of two references. The `..` syntax is optional (`git diff main feature/dashboard` works identically). Use `--stat` for a file-level summary or omit it for the full patch. Unlike `git log`, `diff` does not show intermediate commits — it shows the cumulative difference between two snapshots.

**Checkpoint:** `HEAD~10..HEAD --stat` should show every file in the repo with change counts. Compare this with `git log --oneline HEAD~10..HEAD` — the log shows the step-by-step history, while the diff shows only the beginning and end states.

### Step 10: Find Specific Changes with -S (Pickaxe)

```bash
git log -S"function greet" --oneline
```

Expected output:

```
e1f2a3b feat: add greet function
```

**Explanation:** `-S"function greet"` (the pickaxe option) searches for commits that changed the number of occurrences of the string "function greet". It finds the commit where this string first appeared AND any commit that later modified or removed it. This is more powerful than `--grep` because it finds structural changes regardless of commit message content.

**Checkpoint:** Search for a string that appears in multiple commits. Try `-S"Dashboard"` — it should match commits that reference "Dashboard" in code content, not just commit messages.

## Common Errors at This Stage

### Error 1: `fatal: your current branch 'main' does not have any commits yet`

```
fatal: your current branch 'main' does not have any commits yet
```

**Why it happens:** You ran `git log` in an empty repository with no commits, or you ran `git checkout --orphan` to create an orphan branch.

**Fix:** Make your initial commit first: `git add . && git commit -m "initial commit"`. If you are on an orphan branch and want to reset, run `git checkout main` or `git switch main`.

### Error 2: `ambiguous argument 'HEAD~10': unknown revision or path not in the working tree`

```
fatal: ambiguous argument 'HEAD~10': unknown revision or path not in the working tree
```

**Why it happens:** You referenced a commit that does not exist in the history. `HEAD~10` means "10 commits before HEAD" but your repo may have fewer than 10 commits.

**Fix:** Count your commits first: `git rev-list --count HEAD`. Then use a valid number: `git diff HEAD~N..HEAD` where N is less than the total count.

### Error 3: `--oneline is not a git command` (PowerShell on Windows)

```
git: '--oneline' is not a git command. See 'git --help'.
```

**Why it happens:** This occurs in PowerShell when you forget to add an argument value that PowerShell parses as a flag. More commonly, running `git log --oneline --since "7 days ago"` without the `=` sign can cause parsing confusion.

**Fix:** Always use the `=` syntax for values: `git log --oneline --since="7 days ago"`. Alternatively wrap the entire value in quotes: `git log --oneline --since "7 days ago"`. If you use Git Bash, this is not an issue.

### Error 4: `fatal: bad default revision 'HEAD'`

```
fatal: bad default revision 'HEAD'
```

**Why it happens:** You ran `git log` in a bare repository or a detached HEAD state where HEAD does not point to any commit.

**Fix:** Check your HEAD state with `git status` or `git symbolic-ref HEAD`. If detached, attach to a branch with `git checkout main`. In a bare repo, run `git log --all` to see all branches.

### Error 5: Pager freezing or `WARNING: terminal is not fully functional`

```
WARNING: terminal is not fully functional
```

**Why it happens:** Git paginates long output through `less`. In some terminal emulators (especially on Windows), `less` does not respond to the `q` key or keyboard input.

**Fix:** Disable the pager for a single command: `git --no-pager log --oneline`. Or disable it globally: `git config --global core.pager ""`. On Windows, consider using `git config --global core.pager "more"` for better compatibility.

### Error 6: `--author` returns no results but you know the author exists

```bash
git log --oneline --author="Practice User"
# (no output)
```

**Why it happens:** You might be running the command in a subdirectory that does not have commits, or the author name has different casing/spacing. Git's `--author` filter does substring matching but is case-sensitive by default.

**Fix:** Add `-i` for case-insensitive matching: `git log --oneline -i --author="practice"`. Or use the author's email: `git log --oneline --author="practice@example.com"`. Verify the actual author names: `git log --format="%an <%ae>" | sort -u`.

## What You Just Learned

- `git log --oneline` gives a compact, one-line-per-commit view for scanning history quickly
- `git log --graph --all --decorate` visualizes the full branch and merge DAG as an ASCII graph
- `git log --author`, `--since`/`--until`, `--grep`, and `-S` let you filter history by person, date, keyword, or code content
- `git log -p` shows the complete patch diff for each commit, and `--stat` shows the file-change summary
- `git show <ref>` inspects a single commit with metadata and diff in one command
- `git diff <ref1>..<ref2>` compares two tree snapshots, showing cumulative changes without intermediate commit history
- Relative refs like `HEAD~3`, `main~2` navigate the commit DAG and work with both `git show` and `git diff`
- The pickaxe option `-S"string"` finds commits that added or removed a specific code string, catching changes that `--grep` misses
