## Setup

You need a GitHub repository with GitHub Actions enabled, a Node.js project to apply gates to, and a terminal. This walkthrough uses a sample project that you can clone or create from scratch.

Clone the starter project:

```bash
git clone https://github.com/your-org/gate-demo-project.git
cd gate-demo-project
```

Or create a minimal project from scratch:

```bash
mkdir gate-demo-project
cd gate-demo-project
git init
npm init -y
npm install --save-dev eslint prettier typescript @types/node jest ts-jest
```

The project should have at least a `src/` directory with a few JavaScript or TypeScript files and a `tests/` directory with at least one test file. If you are starting from scratch, create a simple file:

```bash
mkdir src tests
cat > src/hello.js << 'EOF'
function greet(name) {
  return `Hello, ${name}!`;
}

module.exports = { greet };
EOF
cat > tests/hello.test.js << 'EOF'
const { greet } = require('../src/hello');

test('greets by name', () => {
  expect(greet('World')).toBe('Hello, World!');
});
EOF
```

You also need a GitHub repository created on github.com. Push the local project:

```bash
git add -A
git commit -m "Initial commit with sample project"
git branch -M main
git remote add origin https://github.com/your-org/gate-demo-project.git
git push -u origin main
```

## The Walkthrough

### Step 1: Create a linting gate

Create the workflow file `.github/workflows/ci.yml` with a single lint job. This is your cheapest gate and the first thing that runs on every push.

```bash
mkdir -p .github/workflows
```

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx eslint src/
      - run: npx prettier --check src/
```

Push and open a pull request. You should see the lint job run and pass.

Expected output in the GitHub Actions run:

```
Run npx eslint src/
✔ No problems found.
Run npx prettier --check src/
✔ All files match prettier formatting.
```

### Step 2: Add a type checking gate

Add a type checking job that runs after linting passes. This job uses the `needs` key to ensure it only runs if the lint gate succeeds.

```yaml
# Add inside jobs: in .github/workflows/ci.yml
  type-check:
    needs: [lint]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx tsc --noEmit
```

Commit and push. The pipeline now shows two jobs in sequence. If lint fails, type-check is skipped automatically.

Expected output:

```
Run npx tsc --noEmit
✔ No type errors found.
```

### Step 3: Add a unit test gate with parallel execution

Add a unit test job that runs after type checking. Use `jest --maxWorkers=2` to run tests across two parallel workers on the same runner.

```yaml
# Add inside jobs: in .github/workflows/ci.yml
  unit-tests:
    needs: [type-check]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx jest --maxWorkers=2 --verbose
```

Push and verify the pipeline. The test gate now runs in parallel with itself across two workers.

### Step 4: Add a medium-cost security scanning gate

Add a security scanning job that runs Dependabot-style vulnerability checking using `npm audit`. This job runs in parallel with unit tests, not after them, because it is independent and does not need code from previous steps.

```yaml
# Add inside jobs: in .github/workflows/ci.yml
  security-scan:
    needs: [lint]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm audit --audit-level=high
```

This job runs after linting but in parallel with the type-check and unit-test chain. The pipeline now has a fan-out structure.

### Step 5: Add an expensive E2E gate with continue-on-error

Add an end-to-end test gate that runs but does not block the merge. Use `continue-on-error: true` so that a flaky E2E test does not prevent a production fix from going out.

```yaml
# Add inside jobs: in .github/workflows/ci.yml
  e2e-tests:
    needs: [unit-tests, security-scan]
    if: success()
    continue-on-error: true
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx cypress run
```

This job runs only if both unit-tests and security-scan succeed. Even if it fails, the overall pipeline status is green and the PR can be merged.

### Step 6: Add concurrency to cancel redundant runs

Add a `concurrency` block at the workflow level to cancel in-progress runs when a new commit is pushed to the same branch. This prevents wasted CI minutes when you push multiple fixes.

```yaml
# Add at the top level of .github/workflows/ci.yml, same level as name and on:
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

The full workflow now looks like this:

```yaml
name: CI Pipeline

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx eslint src/
      - run: npx prettier --check src/

  type-check:
    needs: [lint]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx tsc --noEmit

  unit-tests:
    needs: [type-check]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx jest --maxWorkers=2 --verbose

  security-scan:
    needs: [lint]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm audit --audit-level=high

  e2e-tests:
    needs: [unit-tests, security-scan]
    if: success()
    continue-on-error: true
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx cypress run
```

### Step 7: Verify the full pipeline

Push the workflow file to your repository and open a pull request. The pipeline should show five jobs with the correct dependency chain. The lint job runs first. If it passes, type-check and security-scan run in parallel. If type-check passes, unit-tests run. If both unit-tests and security-scan pass, e2e-tests run. The e2e-tests job is marked with a yellow warning icon if it fails, but the overall pipeline status is green.

## Common Errors at This Stage

### Error 1: Job fails because `needs` references a non-existent job

```
Error: The workflow is not valid. .github/workflows/ci.yml (Line: 10, Col: 11) -
  Job type-check needs job lint, but lint is not defined in the jobs list.
```

**Fix:** Check that the job name in the `needs` array matches the job ID exactly. GitHub Actions is case-sensitive. The job ID is the key under `jobs:`. If you wrote `Lint` with a capital L, reference it as `needs: [Lint]`.

### Error 2: Security scan fails because of a high-severity vulnerability

```
=== npm audit security report ===
# Run  npm install --audit-audit-level=high  to resolve
high        Prototype Pollution
Package:   lodash
```

**Fix:** This is a real security finding, not a pipeline error. Run `npm audit fix` to update the vulnerable dependency. If the vulnerability cannot be fixed immediately, you can lower the audit level to `--audit-level=moderate` or use `npm audit --production` if the vulnerable package is only a dev dependency. The right fix is to patch the dependency, not to disable the gate.

### Error 3: E2E gate fails but the pipeline still shows as failed

```
Error: Process completed with exit code 1.
```

**Fix:** If you added `continue-on-error: true` and the pipeline still shows as failed, check that the `continue-on-error` key is indented at the same level as `runs-on` and `steps`, not inside the `steps` array. The correct indentation is:

```yaml
  e2e-tests:
    needs: [unit-tests, security-scan]
    if: success()
    continue-on-error: true  # <-- at job level, same as runs-on
    runs-on: ubuntu-latest
```

### Error 4: Concurrency cancels runs too aggressively

```
Canceling since run already in progress for ref: refs/heads/main
```

**Fix:** The `concurrency` group uses `github.ref` which includes the full ref path. For `push` events on `main`, every push cancels the previous run on `main`. This is usually the desired behavior for feature branches, but for `main` you may want to let all runs complete. Change the group to use `github.head_ref` for pull requests and `github.ref` for pushes:

```yaml
concurrency:
  group: ${{ github.head_ref || github.ref }}
  cancel-in-progress: true
```

This way, pull request runs get cancelled on new commits, but runs on `main` are never cancelled.

### Error 5: The pipeline runs even when no source files changed

```
Triggered: push to main (only changed README.md)
```

**Fix:** Use `paths` to restrict the workflow to relevant file changes. This prevents the pipeline from running on documentation-only changes:

```yaml
on:
  pull_request:
    branches: [main]
    paths:
      - 'src/**'
      - 'tests/**'
      - '.github/workflows/**'
```

## What You Just Learned

- You built a tiered CI pipeline with five quality gates, each with a specific purpose and cost profile.
- You used the `needs` key to chain jobs in sequence, preventing expensive gates from running on code that already failed a cheaper gate.
- You used `continue-on-error: true` to mark an expensive gate as informational, so a flaky E2E test does not block the merge.
- You used `concurrency` with `cancel-in-progress: true` to stop redundant CI runs when you push multiple commits to the same branch.
- You learned that the cheapest gates (lint, type check) should run first and the most expensive gates (E2E, performance) should run last with lower blocking power.
- You can now design a pipeline where most failures are caught in under sixty seconds, and the team trusts the results because the signal-to-noise ratio is high.
