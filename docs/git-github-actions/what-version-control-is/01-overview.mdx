You are working on a script that processes customer data. It took you three hours to write. You make one small change to fix a typo in a column name, save the file, and run it. Now nothing works. The output is garbage. You try pressing Ctrl+Z in your editor, but you already closed the file. You look for an auto-saved backup. Nothing. That three-hour block of working code is gone.

You try to remember what the original line looked like. You think it was `df["amount"]` but you are not sure. Maybe it was `df["Amount"]` with a capital A. You spend the next 45 minutes reconstructing the original logic from memory. You get most of it right, but you miss one edge case that you had fixed in the original version. You will not discover that mistake until the next time the script runs against a full dataset.

This scenario plays out thousands of times a day across the software industry. A developer makes a change, hits save, closes a file, and the previous version vanishes. If you have ever renamed a file to `script_final_v3_FINAL.py` or emailed a zip archive of your project to yourself, you have already felt the pain that version control exists to solve.

The problem is not just about losing work. It is about collaboration, experimentation, and accountability. When multiple people work on the same codebase, files get overwritten. Someone changes a function you depend on, and you do not find out until your entire pipeline breaks. You try a new approach to a problem, but you cannot go back to your starting point without manually remembering every line you changed. You ship a bug to production, and nobody can tell you what changed or who changed it.

Consider a more specific scenario. You and a teammate are working on the same file. You add error handling around a database call. Your teammate renames the database function from `get_user()` to `fetch_user()`. You both save your versions. Whose version survives? The answer depends on who saves last. The person who saves last overwrites the other person's work entirely. There is no merge. There is no warning. There is just a silent overwrite that you might not notice for days.

Now consider a different scenario. You are refactoring a payment processing function. You make 12 changes across 4 files. Halfway through, you realise the approach is wrong. The original design was better. To undo your work, you would need to remember every line you changed and manually revert each one. If you had version control, you would type one command and the entire project would snap back to its previous state.

Version control is the tool that eliminates all of these problems at once. It records every change to every file, keeps a complete history of who changed what and when, and lets you travel backward and forward through time in your project. It is the single most important tool in a professional developer's workflow.

Consider one more scenario to drive the point home. You ship a new version of your application to production. An hour later, your monitoring system alerts you that error rates have tripled. You need to know what changed in the deploy. Without version control, you would chase down every developer who committed code in the last week and ask them to list their changes. With version control, you run `git log --oneline --since="1 week ago"` and see every change in under a second. You find the problematic commit, look at the diff, and understand exactly what went wrong. You can then decide to fix forward or revert the commit. The revert itself is a single command: `git revert <hash>`. This entire workflow, from alert to resolution, takes minutes instead of hours.

Think about a situation where a deploy introduces a bug that corrupts user data. You identify the problematic commit hash `a1b2c3d` which changed the data transformation logic. Without version control, you would need to manually undo the changes, guess which lines to revert, and hope you did not miss anything. With version control, you run `git revert a1b2c3d` and Git automatically computes the inverse of the changes in that commit and applies them. The revert is precise, fast, and leaves an audit trail showing exactly when and why the reversion happened. If you later need to revisit that bug, the revert commit and the original commit together tell the complete story.

## The Concept

Version control is a system that tracks changes to files over time. It creates a series of snapshots that capture the state of your entire project at specific moments. You can think of each snapshot as a save point in a video game. In a game, you save your progress before a difficult boss fight. If you lose, you load your save and try again. Version control works the same way. You create a save point -- called a commit -- when your code is in a known good state. If you later break something, you can load an earlier commit and restore the entire project to the way it was.

A version control system, or VCS, does not just store the latest version of your files. It stores every version you tell it to remember, along with a message explaining what changed and why. This history becomes a complete record of your project's evolution. You can answer questions like: When was this line added? Who wrote it? What other changes were part of the same task? What bug report or feature request prompted this change?

Without version control, the only way to answer "what changed last week?" is to ask everyone on the team and hope they remember. With version control, you run a single command and get a precise, timestamped list of every change, every file, and every author.

Git, the most popular version control system, adds two important capabilities on top of basic version tracking. First, it is distributed. Every person who clones a repository has a full copy of the entire history on their own machine. There is no central server that must be online for you to access your project's history. You can browse commits, view diffs, and create new commits entirely offline. Second, Git uses branching to let you work on multiple versions of your code simultaneously. You can create a branch to try an experimental feature, commit changes there without affecting your main codebase, and either merge the branch back in or discard it entirely.

Other version control systems exist. Subversion and Mercurial are older alternatives. Subversion uses a centralised model where a single server holds the authoritative history and every operation requires a network connection. Mercurial is distributed like Git but uses a different internal data model. Git has become the dominant choice because of its speed, its branching model, and the ecosystem built around it, including platforms like GitHub, GitLab, and Bitbucket.

The word "version" in version control refers to the numbered iterations of your project over time. The first commit is version 1. The second commit is version 2. Git does not assign sequential version numbers by default, but you can tag specific commits with meaningful names like `v1.0.0` or `release-2026-06-15`. These tags serve as permanent bookmarks that let you reference a specific version of your project years later. The "control" part of the name refers to the ability to manage, compare, and navigate between these versions in a disciplined way rather than through ad-hoc file naming conventions.

The benefits of version control extend beyond the technical. When every commit is linked to a developer and a timestamp, teams develop a culture of accountability. You know that every change you make will be recorded and attributed to you. This encourages thoughtful commit messages, careful code review, and a willingness to document not just what changed but why. Over time, the commit history becomes a narrative of the project's evolution that new team members can read to understand how the codebase reached its current state.

There are three main types of version control systems. Local version control systems store versions on your local machine, often as simple patch files. Centralised version control systems like Subversion store the authoritative history on a single server and developers check out only the latest version. Distributed version control systems like Git give every developer a complete copy of the entire repository history. Git is distributed, which means every operation except pushing and pulling to a remote happens locally and is fast. You can commit, browse history, diff changes, and create branches entirely offline.

The distributed model has a critical advantage: it eliminates the single point of failure. If the central server in a centralised system goes down, nobody can commit or access history. If the server in a distributed system goes down, every developer still has a full copy of the repository. Restoring the server is a matter of copying any one developer's repository back to the server.

### A Brief History of Git

Git was created in 2005 by Linus Torvalds, the creator of Linux. The Linux kernel project had been using a proprietary distributed version control system called BitKeeper. When the free license for BitKeeper was revoked, Torvalds decided to build a replacement that met three requirements: it had to be fast, it had to support a distributed workflow, and it had to guarantee data integrity through cryptographic hashing. The first version of Git was written in about 10 days. Within months, the Linux kernel project had migrated to Git. Today, Git is used by millions of developers and is the foundation of GitHub, GitLab, Bitbucket, and countless other tools.

The name "Git" is British slang for an unpleasant person. Torvalds has said that he names his projects after himself, starting with Linux (Linus + Unix) and continuing with Git. The official Git documentation describes it as "the stupid content tracker," but the name has become a badge of pride in the open-source community.

### Key Terminology

Before moving to the command reference, it helps to have a shared vocabulary for the core concepts you will encounter in every Git operation.

- **Repository** -- a directory that contains your project files plus the `.git` directory that stores all version history and metadata. Every Git command you run operates within the context of a repository.
- **Commit** -- a snapshot of the entire project at a specific point in time. Each commit has a unique hash, an author, a timestamp, a message, and one or more parent commits.
- **Working directory** -- the directory on your filesystem where you edit files. Changes here are not automatically tracked by Git.
- **Staging area** -- also called the index. A middle area where you collect changes that will go into the next commit. It gives you control over exactly what each commit contains.
- **Branch** -- a movable pointer to a specific commit. The default branch is named `main` in modern Git repositories. Creating a new branch costs nothing in storage and allows parallel development.
- **HEAD** -- a special pointer that tells Git which branch you are currently on. When you make a commit, Git uses HEAD to determine which branch should move forward.
- **Remote** -- a copy of the repository hosted on another computer or server. Common remote names include `origin` for GitHub repositories.
- **Clone** -- the act of downloading a remote repository to your local machine. A clone contains the entire history, not just the latest version.
- **Push** -- sending your local commits to a remote repository so others can access them.
- **Pull** -- fetching commits from a remote repository and merging them into your local branch.
- **Merge** -- combining changes from two branches into one. Git creates a merge commit that has two parents.
- **Tag** -- a permanent label attached to a specific commit. Often used for release versions like `v1.0.0`.
- **SHA-1 hash** -- the 40-character hexadecimal string that uniquely identifies each commit, tree, and blob. The hash is computed from the content it identifies, so content integrity is guaranteed.

These terms will appear in every Git tutorial, every pull request, and every team discussion about version control. Learning them now will make the rest of your Git education significantly smoother.

## Command and Syntax Reference

The table below lists the most commonly used Git commands. Each entry shows the syntax, the purpose of common flags, and a minimal example.

| Command | Common Flags | Description | Example |
|---|---|---|---|
| `git init` | (none commonly used) | Creates a new Git repository in the current directory | `git init` |
| `git add` | `-A` or `--all`: stage all changes including deletions and untracked files | Stages file contents for the next commit | `git add main.py` |
| `git add` | `.`: stage everything in the current directory and its subdirectories | Shortcut to stage all changes from a directory | `git add .` |
| `git commit` | `-m "message"`: provide the commit message inline instead of opening an editor | Records the staged changes as a new commit | `git commit -m "Add login handler"` |
| `git commit` | `--amend`: edit the most recent commit instead of creating a new one | Fixes the commit message or adds forgotten changes to the last commit | `git commit --amend -m "Fixed message"` |
| `git commit` | `-a`: automatically stage all tracked files before committing | Combines `git add` and `git commit` for tracked files | `git commit -a -m "Update all configs"` |
| `git status` | `-s` or `--short`: compact output format | Shows which files are staged, unstaged, or untracked | `git status` |
| `git log` | `--oneline`: single-line format per commit | Displays the commit history | `git log --oneline` |
| `git log` | `--graph`: show branch structure as ASCII art | Visualises branching history | `git log --oneline --graph` |
| `git log` | `--since="2 days ago"`: filter commits by date range | Shows commits within a time window | `git log --since="2026-06-01"` |
| `git diff` | (no flags): shows unstaged changes | Shows line-by-line differences between working directory and staging area | `git diff` |
| `git diff` | `--staged` or `--cached`: shows staged changes | Shows what will be included in the next commit | `git diff --staged` |
| `git diff` | `HEAD~1 HEAD`: compare the last commit with the current one | Shows changes between two specific commits | `git diff HEAD~2 HEAD` |
| `git checkout` | `-b branch-name`: create and switch to a new branch | Switches to a different branch or restores working tree files | `git checkout -b feature-login` |
| `git checkout` | (branch name): switch to an existing branch | Moves HEAD to the specified branch | `git checkout main` |
| `git branch` | (no arguments): list local branches | Lists, creates, or deletes branches | `git branch` |
| `git branch` | `-d branch-name`: delete a branch | Removes a branch after its changes have been merged | `git branch -d old-feature` |
| `git branch` | `-m old-name new-name`: rename a branch | Renames the specified branch | `git branch -m bug-fix bugfix` |
| `git restore` | (no flags): restore a file from the last commit | Discards unstaged changes in the working directory | `git restore main.py` |
| `git restore` | `--staged file`: unstage a file without losing working directory changes | Removes a file from the staging area | `git restore --staged main.py` |

The commands `git checkout` and `git restore` overlap in functionality. The `git restore` command was introduced in Git version 2.23 to separate the two distinct uses of `git checkout`: switching branches versus restoring files. When you need to restore a file, prefer `git restore`. When you need to switch branches, use `git checkout` or `git switch`.

## The Big Picture

A Git repository history forms a directed acyclic graph of commits. Each commit points to its parent commit or commits, forming a chain that you can traverse forward and backward. When you create branches, the graph forks. When you merge branches, the graph merges back together. The graph is acyclic because a commit can never point to its own descendants -- there are no circular references.

If you are accustomed to a workflow where you copy an entire project directory to try a new idea, branching will feel liberating. Instead of maintaining a half-dozen copies of your project with names like `project-backup-may`, `project-experiment`, and `project-old`, you create a branch. The branch is weightless, it shares all the files that have not changed, and it merges back cleanly when you are done.

## How It Actually Works -- Step by Step

Understanding how Git stores data will help you reason about what each command does and why certain operations behave the way they do. Git's data model is elegant but unfamiliar to most new users. Taking the time to understand it now will save you hours of confusion later.

### Step 1: The Working Directory

Your working directory is the current state of your project on disk. This is the directory you see in your file explorer or code editor. When you modify a file, you change the contents of the working directory. Git sees these changes but does not record them automatically.

All Git commands operate on the three-tree system: the working directory, the staging area, and the commit history. You can visualise these three areas as separate storage locations that Git manages independently. Your task as a developer is to move changes intentionally from one area to the next.

To make Git aware of changes, you move files from the working directory into the staging area. You can think of the working directory as your workbench, the staging area as a packaging table, and the commit history as a storage room. You assemble your changes on the workbench, pack them on the packaging table, and store the package in the storage room.

The three-tree system is unique to Git. Other version control systems like Subversion use only two states: the working directory and the repository. Git's addition of the staging area gives you fine-grained control over what goes into each commit. Understanding the three trees will make every other Git concept easier to grasp.

### Step 2: Staging Changes with `git add`

The staging area, also called the index, is a middle ground between your working directory and the commit history. When you run `git add`, Git takes a snapshot of the file as it currently exists and stores a reference to that snapshot in the staging area. You can stage different parts of a file using interactive flags, or stage entire files at once.

The staging area is one of Git's most powerful features, and it is also one of the concepts that new users find most confusing. Why have a separate step between editing and committing? The reason is precision. Staging lets you build a commit in pieces.

Staging lets you build a commit in pieces. You might fix two unrelated bugs in the same file but only stage the changes for the first bug, commit them with a message like "Fix null pointer in user lookup", then stage and commit the second set of changes separately. The result is a clean history where each commit contains only related changes.

You can also unstage files without losing your edits. If you accidentally stage a file that is not ready, use `git restore --staged file` to remove it from the staging area while keeping your working directory changes intact.

```

The broken code on line 45 stays safely in your working directory, uncommitted, until you fix it.

### Step 3: Committing with `git commit`

When you run `git commit`, Git does the following:

1. It takes the contents of the staging area and creates a commit object.
2. The commit object contains a unique hash -- a 40-character hexadecimal string -- that identifies it forever. The hash is computed from the commit contents using SHA-1, so the same content always produces the same hash, and different content produces a different hash with extremely high probability.
3. The commit stores a tree object that represents the state of the entire project directory structure at that moment. The tree maps filenames to blob objects, which are the actual file contents.
4. The commit stores the hash of its parent commit or commits. The very first commit in a repository has no parent and is called the root commit. A normal commit has one parent. A merge commit has two parents.
5. The commit stores the author name, email, timestamp, and the commit message you provided.

This internal structure explains several important Git behaviours. First, because every commit stores a complete tree, checking out any commit is instantaneous -- Git just reads the tree and copies the blobs to disk. Second, because commits are identified by their content hash, you can verify that history has not been tampered with. Third, because commits store parent pointers, Git can detect when two branches have diverged and compute what needs to happen during a merge.

### Step 4: Snapshots, Not Deltas

A common misunderstanding is that Git stores changes -- diffs -- between file versions. It does not. Every commit is a full snapshot of every file in the project at that moment. Git compresses these snapshots using zlib and stores them efficiently. When files do not change between commits, Git reuses the same blob object for both commits -- the commit objects point to the same blobs -- so unchanged files take up no additional space.

<AsciiDiagram id="what-version-control-is/snapshots-over-time" title="Snapshots, Not Deltas" alt="Diagram showing how three commits store full file snapshots with shared blobs for unchanged files" caption="Git stores a full tree per commit but reuses blobs when files do not change" content={`┌──────────────────────────────────────────────────────────────┐
│  Snapshots, Not Deltas                                         │
│                                                              │
│  Commit 1:  ((C1))  ──  file_a: v1    file_b: v1            │
│                            │            │                    │
│  Commit 2:  ((C2))  ──  file_a: v1    file_b: v2            │
│                            │            │                    │
│  Commit 3:  ((C3))  ──  file_a: v2    file_b: v2            │
│                            │            │                    │
│  Blobs stored: file_a v1, file_a v2, file_b v1, file_b v2    │
│  Only 4 blobs for 3 commits. Unchanged files share blobs.    │
└──────────────────────────────────────────────────────────────┘`} />

This design makes checking out any previous commit extremely fast. Git does not need to replay a series of diffs; it just copies the blobs from the tree object into the working directory. By contrast, older version control systems like Subversion store the latest version of each file plus a reverse delta, meaning that checking out an old revision requires replaying every change in reverse. Checking out a revision that is 500 commits back in Subversion means applying 500 reverse deltas. Checking out the same revision in Git is a constant-time operation.

The snapshot model also makes branching cheap. Creating a branch in Git does not copy any files. It just creates a new pointer -- a 41-byte file -- that points to a commit. All the blobs are shared between branches until you start making divergent commits. This is why Git encourages creating many branches. There is no storage cost for branches that share most of their history.

To verify this for yourself, initialise a repository, create a file, commit it, then create a branch and compare the repository size before and after:

```bash
mkdir test-repo
cd test-repo
git init
echo "hello" > greeting.txt
git add greeting.txt
git commit -m "Add greeting"
du -sh .git
git branch experiment
du -sh .git
```

The two `du -sh .git` commands produce the same output. The branch cost exactly zero bytes of additional storage because no new objects were created. The branch file itself occupies 41 bytes, which is too small to register at the kilobyte resolution of `du -sh`.

### Step 5: Viewing History with `git log` and `git diff`

The `git log` command reads the commit graph starting from the commit you are currently on -- called HEAD -- and walks backward through the parent pointers. Each commit is displayed with its hash, author, date, and message. The `git diff` command, by contrast, compares file contents between any two points in the repository. You can diff the working directory versus the staging area, the staging area versus the last commit, or any two arbitrary commits.

These two commands form the foundation of code review and debugging. You use `git log` to find when a change was made and `git diff` to see exactly what changed. In a professional setting, code review tools like GitHub pull requests use `git diff` under the hood to generate the line-by-line comparison you see on screen.

The `git log` command accepts filters that let you narrow down the history. You can filter by author with `--author="Jane"`, by date with `--since="2026-01-01"`, by file path with `-- README.md`, or by keyword in the commit message with `--grep="fix"`. These filters make it practical to navigate repositories with thousands of commits.

You can also combine `git log` with `-p` to see the full diff for each commit in the output:

```bash
git log -p --oneline -3
```

This command shows the last 3 commits with their diffs, all in a single view. The `-p` flag stands for "patch" and shows the complete diff for each commit. This is useful for reviewing what changed across a series of commits without running separate commands.

The `git log` output format can be customised with the `--pretty` flag. The `--oneline` flag is a shortcut for `--pretty=oneline --abbrev-commit`. Other formats include `--pretty=format:"%h - %an, %ar : %s"` which displays the abbreviated hash, author name, relative date, and subject. This level of customisation is useful when you want to integrate Git history into scripts or reporting tools.

### Step 6: Branching and HEAD

A branch in Git is simply a movable pointer to a specific commit. When you create a new branch with `git branch feature-x`, Git creates a new pointer at the current commit. When you make a new commit on that branch, the pointer advances to the new commit while the original branch stays where it was. There is no duplication of files, no copying of directories, and no configuration overhead. A branch is a 41-byte text file containing the hash of the commit it points to. Creating a branch is one of the cheapest operations in computing.

The typical branching workflow looks like this:

1. Start on `main` with a working codebase.
2. Create and switch to a feature branch: `git checkout -b add-payment-api`.
3. Make changes, stage, and commit on the feature branch. These commits are isolated from `main`.
4. Push the feature branch to the remote: `git push origin add-payment-api`.
5. Open a pull request on GitHub for code review.
6. After approval, merge the feature branch into `main`.
7. Delete the feature branch locally and remotely: `git branch -d add-payment-api` and `git push origin --delete add-payment-api`.

This workflow keeps `main` stable while feature work happens in parallel. Multiple developers can each work on their own feature branches without interfering with each other.

The special pointer HEAD tells Git which branch you are currently on. HEAD typically points to a branch, and that branch points to a commit. When you run `git commit`, Git creates the new commit, sets its parent to whatever commit the current branch points to, and moves the branch pointer forward. If you switch branches with `git checkout`, HEAD moves to point at the new branch, and Git updates your working directory to match the commit that branch points to.

```
┌──────────────────────────────────────────────────────────────┐
│  Branches Are Pointers to Commits                            │
│                                                              │
│  Before creating feature-x                                   │
│                                                              │
│  main ──▶ C1 ──▶ C2 ──▶ C3  ◀── HEAD (points to main)       │
│                                                              │
│  After git branch feature-x                                  │
│                                                              │
│  main ──▶ C1 ──▶ C2 ──▶ C3  ◀── HEAD (still on main)        │
│                          │                                   │
│                          └──▶ feature-x                       │
│                                                              │
│  After git checkout feature-x and one commit                 │
│                                                              │
│  main ──▶ C1 ──▶ C2 ──▶ C3                                   │
│                          │                                   │
│                          └──▶ C4  ◀── feature-x, HEAD        │
│                                                              │
│  At this point, main still points to C3. feature-x points    │
│  to C4. HEAD points to feature-x. The branches have          │
│  diverged by one commit.                                     │
└──────────────────────────────────────────────────────────────┘
```

The diagram illustrates a critical concept: branches diverge over time. After creating `feature-x` and making one commit on it, `main` still points to C3 while `feature-x` points to C4. The two branches now contain different code. Changes made on one branch are invisible to the other until they are merged.

### Step 7: The .git Directory

Every Git repository contains a hidden directory called `.git` at the root. This directory is the repository itself. All of the objects, references, configuration, and logs that make up your version history live here. If you delete the `.git` directory, you delete the entire history. The working directory files remain, but they become ordinary files with no version history attached.

The `.git` directory structure includes:
- `objects/` -- the content database. Every blob, tree, and commit object is stored here as a compressed file named by its SHA-1 hash.
- `refs/` -- references to commits. This includes branches (in `refs/heads/`) and tags (in `refs/tags/`). Each reference is a text file containing a 40-character commit hash.
- `HEAD` -- a text file that contains the path to the current branch reference, like `ref: refs/heads/main`.
- `config` -- the repository-specific Git configuration file.
- `index` -- the staging area, a binary file that maps file paths to blob object hashes.

You never need to edit files inside `.git` by hand. All interaction with the repository goes through Git commands. However, knowing what is inside `.git` demystifies what Git is doing when you run commands. A branch is a tiny text file. A commit is a compressed object file. The staging area is a single binary index file. There is no magic, just files on disk.

You can explore the `.git` directory yourself with standard filesystem commands. From the root of any Git repository, run:

```bash
ls -la .git
cat .git/HEAD
ls .git/refs/heads/
```

The `HEAD` file shows which branch is currently checked out. The `refs/heads/` directory contains one file per local branch, and each file contains a 40-character commit hash. If you view one of these files with `cat .git/refs/heads/main`, you will see the hash of the latest commit on that branch. This is all a branch is: a text file containing a hash.`

### Step 8: Merging Branches

When you are ready to bring changes from one branch into another, you merge them. The `git merge` command creates a new commit that has two parents: the tip of the current branch and the tip of the branch being merged in. This merge commit combines the work from both lines of development.

If the branches have changed different files, the merge happens automatically. If they have changed the same lines in the same files, Git reports a merge conflict and asks you to resolve it manually. Merge conflicts are not errors. They are Git's way of telling you that it needs human judgment to decide which change to keep. When a conflict occurs, Git marks the conflicting regions in the affected files and pauses the merge. You edit the files to resolve the conflict, stage the resolved files, and run `git merge --continue` to complete the merge.

There are two types of merge in Git. A regular merge creates a new merge commit as shown in the diagram above. A fast-forward merge occurs when the branch being merged has not diverged from the current branch -- it is simply ahead by some commits. In that case, Git moves the current branch pointer forward to match the merged branch without creating a merge commit. Fast-forward merges produce a linear history, while regular merges preserve the branching structure. You can force a regular merge even when a fast-forward is possible with `git merge --no-ff feature-x`.

```
┌──────────────────────────────────────────────────────────────┐
│  Merge Commit Structure                                      │
│                                                              │
│  Before merge:                                               │
│                                                              │
│  main  ──▶ C1 ──▶ C2 ──▶ C3  (last common ancestor)         │
│                           │                                  │
│                           └──▶ C4 ──▶ C5  (feature-x)        │
│                                                              │
│  After git checkout main && git merge feature-x:             │
│                                                              │
│  main ──▶ C1 ──▶ C2 ──▶ C3 ──▶ C6 (merge commit)           │
│                           │       │                          │
│                           │       └── parent 2               │
│                           └──▶ C4 ──▶ C5 ──────┘             │
│                                                              │
│  C6 has two parents: C3 (from main) and C5 (from feature-x). │
│  The merge combines both lines of history.                   │
└──────────────────────────────────────────────────────────────┘
```

This snapshot model has a practical consequence for repository size. A project with 100 commits where every file changes in every commit will be roughly 100 times larger than a single checkout of the project. In practice, most files do not change in most commits, so repositories grow much more slowly. A typical web application with 500 files and 2,000 commits might have a `.git` directory of 30 to 50 MB -- smaller than a single high-resolution photograph. The Linux kernel repository, for example, has over one million commits and contains the entire history of the kernel since 2005. Its `.git` directory is around 3 GB -- large but manageable given the scale of the project. Most project repositories stay under 500 MB even with years of history.

### Step 9: Detached HEAD State

There is one more HEAD configuration you should understand. If you check out a specific commit instead of a branch -- for example, `git checkout a1b2c3d` -- Git enters a detached HEAD state. HEAD points directly to a commit rather than to a branch.

In detached HEAD state, any new commits you create belong to no branch. If you switch to a different commit or branch, these commits become unreachable and will eventually be cleaned up by Git's garbage collection. Detached HEAD is useful for inspecting old versions of a project, but you should create a branch before making new commits in this state.

```
┌──────────────────────────────────────────────────────────────┐
│  Detached HEAD State                                         │
│                                                              │
│  Normal state:                                               │
│                                                              │
│  main ──▶ C3  ◀── HEAD (HEAD points to main, main points    │
│                      to C3)                                  │
│                                                              │
│  Detached HEAD state after git checkout a1b2c3d:             │
│                                                              │
│  main ──▶ C1 ──▶ C2 ──▶ C3                                   │
│                           ▲                                   │
│                           │                                   │
│                      HEAD (HEAD points to C3 directly,       │
│                            not to any branch)                │
│                                                              │
│  Any commits made now will not advance any branch.           │
│  To keep them, run git branch new-branch before switching.   │
└──────────────────────────────────────────────────────────────┘
```

## When to Use This and When Not To

Use version control when:
- You are writing any amount of code that will exist for more than a few minutes. There is no minimum project size. A single Python script with 50 lines benefits from version control just as much as a multi-service application with 500,000 lines. The habit of version control should start with the first line of code you write.
- You are collaborating with other developers. Version control prevents overwrites, provides a communication channel through commit messages, and gives every team member a full copy of the project. Without version control, collaboration above two people quickly becomes unmanageable. With version control, teams of hundreds can work on the same codebase simultaneously.
- You want to experiment with changes without risking your working codebase. Branches let you try ideas in isolation. If the experiment fails, you delete the branch. If it succeeds, you merge it back. This freedom to experiment encourages better engineering because the cost of a failed idea is zero.
- You need to understand why something was changed months ago. The commit history provides accountability and context. A well-written commit message explains not just what changed, but why. This is invaluable when debugging a bug that was introduced weeks or months before it was discovered.
- You deploy code to any environment. Knowing exactly which version is deployed and what changed since the last deployment is critical for debugging production issues. Tagging releases with `git tag` gives you a permanent reference point that you can use to reproduce any deployed version.
- You work on more than one computer. Pushing and pulling your repository between machines keeps your work in sync without emailing files to yourself or using a USB drive. You can pick up exactly where you left off on any machine.
- You need to comply with audit or regulatory requirements. A complete, tamper-evident history of every change to a codebase provides an audit trail that manual processes cannot match. Each commit is cryptographically identified by its SHA-1 hash, and the chain of parent pointers makes it detectable if history has been altered.

Do not use version control -- or use a simpler tool -- when:
- You are editing binary files like images, videos, or compiled binaries. Git can store them, but it will not show meaningful diffs and the repository size will grow quickly with each version. For large binary assets, use Git LFS or a dedicated asset management system. If you must store binaries, keep them in a separate repository from your source code.
- You are working with files that contain secrets, API keys, or passwords. Version control keeps every version of a file forever, so a leaked secret cannot be removed by simply deleting it from the latest commit. You would need to rotate the secret and force-push a rewritten history, which is disruptive for everyone on the team. Use environment variables and secret management tools instead.
- You are managing database backups or log files. These change too frequently and are too large for Git to handle efficiently. A database backup that changes every hour will bloat the repository to gigabytes within weeks. Use dedicated backup tools for backups and log rotation utilities for log files.
- You need real-time collaborative editing like what Google Docs provides. Git is designed for intentional, packaged changes, not character-by-character syncing. Each commit is a deliberate snapshot, not a keystroke. If you need real-time collaboration, consider tools designed for that purpose alongside your version control workflow.
- You are prototyping a one-off script that you will run once and discard. If the script has no future, version control adds overhead without benefit. However, if your prototype turns into a real project, initialise a repository as soon as you decide to keep the code.

## Commit Message Conventions

A good commit message is as important as the code changes themselves. The commit message is how you communicate the intent of a change to your future self and to every other developer who will work on the project. A clear commit message can save hours of investigation when debugging a regression. A vague or misleading commit message wastes time.

The standard format for a Git commit message has two parts: a subject line and a body. The subject line is the first line of the message, limited to 50 characters. It should complete the sentence "If applied, this commit will `&lt;your subject line here&gt;`." The body follows after a blank line and provides additional context: why the change was made, what alternatives were considered, and any side effects to watch for.

Examples of good subject lines:

- `Add input validation to email field`
- `Fix null pointer in user lookup when session expires`
- `Refactor database connection pool configuration`
- `Remove deprecated payment gateway integration`

Examples of poor subject lines:

- `fix bug` -- does not say which bug or where
- `changes` -- completely uninformative
- `wip` -- meaningless to anyone except the author at the moment of writing
- `Update file` -- does not say which file or why

The body of the commit message is optional for trivial changes but essential for complex ones. When you write the body, focus on the why, not the what. The what is visible in the diff. The why is not. Here is an example of a complete commit message:

```
Refactor session timeout handling

Session timeouts were previously calculated from the server start time,
which caused all sessions to expire simultaneously at midnight after 24
hours of server uptime. Changed the calculation to measure from the
individual session creation timestamp.

This also fixes the edge case where sessions created after midnight
had negative remaining time.
```

This commit message tells the reader what changed (the subject line), why the old approach was wrong (the body), and what edge case was fixed (the final paragraph). Anyone reading this commit message in two years will understand both the change and its motivation without needing to read the full diff.

Consistent commit messages are a hallmark of professional software engineering teams. Many projects enforce commit message conventions through automated tooling or contributor guidelines. Adopting good habits now will serve you well throughout your career.
