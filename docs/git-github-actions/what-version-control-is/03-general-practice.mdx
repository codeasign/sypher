These exercises reinforce the core Git workflow you learned in the overview. They are organised into three tiers. Tier 1 focuses on single commands. Tier 2 combines multiple concepts. Tier 3 presents ambiguous scenarios you might encounter in real projects.

Each exercise follows the same structure: a task description, a hint if you get stuck, and a complete solution with explanation. Attempt each exercise on your own before looking at the solution. The exercises are designed to be completed in order, but you can skip ahead if you are comfortable with the material.

## Tier 1 — Beginner

### Exercise 1: Examine Repository State

**Task.** You are inside a Git repository. Run a single command that shows you the current branch, which files are modified, which files are staged, and whether the working tree is clean.

**Hint.** The command you need is the most frequently run Git command. It is safe to run at any time and has no side effects. You should run it before and after almost every other Git operation.

**Solution.**

```bash
git status
```

`git status` is the first command you should run when you are unsure about the state of your repository. It reports the current branch, lists untracked files, shows staged and unstaged changes, and tells you whether anything differs from the last commit. The output is designed to be human-readable and includes helpful hints like "(use `git add <file>...` to include in what will be committed)."

The command reads from the repository's index and working directory but never modifies them. You can run it hundreds of times per session without any side effects. Experienced developers run `git status` dozens of times per day.

### Exercise 2: Stage a Single File

**Task.** You have modified three files: `index.html`, `style.css`, and `app.js`. You want to commit only the changes to `style.css` and `app.js` together, leaving `index.html` out of the commit. Run the command to stage exactly those two files.

**Hint.** The `git add` command accepts one or more file paths as arguments. You do not need to use the `.` shortcut. You can list file paths separated by spaces.

**Solution.**

```bash
git add style.css app.js
```

`git add` takes a list of file paths. Each file listed is added to the staging area. Files you do not list remain unstaged, even if they have been modified. This is the mechanism that lets you build precise commits.

If you later want to stage `index.html` separately, you can run `git add index.html` and commit it with a different message. The result is two commits: one for the CSS and JS changes, and one for the HTML change. Each commit has a focused message that describes only the changes it contains.

### Exercise 3: Commit with an Inline Message

**Task.** You have staged several files. You want to commit them immediately with the message "Refactor database connection handling" without opening a text editor. Write the command.

**Hint.** Add a flag to `git commit` that lets you provide the message directly on the command line. The flag is a single letter.

**Solution.**

```bash
git commit -m "Refactor database connection handling"
```

The `-m` flag accepts the commit message as a string argument. The commit is created as soon as you press Enter. For longer messages that require multiple lines, omit the `-m` flag and Git will open your configured editor. When you save and close the editor, the commit is created with the message you wrote.

If you forget to include the `-m` flag and the message, Git opens your default text editor. If you are not comfortable with the terminal-based editor that opens (usually Vim or Nano), you can press Ctrl+C to abort the commit and retry with the `-m` flag.

### Exercise 4: View Commit History in Compact Form

**Task.** Your repository has 30 commits. You want to see them all listed on a single line each, from newest to oldest, with 7-character hashes and commit subjects. Run the command.

**Hint.** Add the `--oneline` flag to `git log`. This is the most common way to view history in a professional setting.

**Solution.**

```bash
git log --oneline
```

The compact form displays each commit as a single line showing the abbreviated hash and the commit subject. This is useful for getting a quick overview of the project history without scrolling through author, date, and full commit body fields. The output looks like this:

```
a1b2c3d Fix null pointer in user lookup
2b3c4d5 Add CSS and JavaScript files
3c4d5e6 Initial project setup
```

The most recent commit appears at the top. The hash shown is the first 7 characters of the full 40-character SHA-1 hash. This is long enough to be unique within nearly all repositories.

### Exercise 5: Check the Difference Before Committing

**Task.** You have made changes to `config.py` but have not staged them yet. Before staging, you want to verify that the changes are correct by seeing the exact lines that differ. Write the command.

**Hint.** Use `git diff` with no arguments. It compares the working directory against the staging area. Since the file is not staged, the staging area still matches the last commit.

**Solution.**

```bash
git diff config.py
```

You can also omit the filename to see diffs for all changed files:

```bash
git diff
```

Each diff shows the file header, the line numbers that changed, and the actual lines with `+` for additions and `-` for deletions. Reviewing the diff before staging is a good habit. It catches mistakes like accidentally modified whitespace, debugging print statements left in the code, or changes that belong in a different commit.

## Tier 2 — Intermediate

### Exercise 6: Stage, Unstage, and Verify

**Task.** You accidentally staged `debug.log`, a file that should never be committed because it contains temporary output. Perform the following sequence without writing any of the file contents to a permanent commit.

1. Create a file called `debug.log` with the content `temporary debug output`.
2. Stage the file.
3. Verify it is staged.
4. Unstage the file without deleting it from your working directory.
5. Verify it is no longer staged but still exists on disk.

**Hint.** To unstage a file, use `git rm --cached` for untracked files or `git restore --staged` for tracked files. Verify each step with `git status`. The `--cached` flag is critical -- it tells Git to remove the file only from the index, not from the filesystem.

**Solution.**

```bash
echo "temporary debug output" > debug.log
git add debug.log
git status
git rm --cached debug.log
git status
ls debug.log
```

After `git rm --cached debug.log`, the file is removed from the staging area but remains in the working directory. The `--cached` flag tells Git to remove the file only from the index, not from disk. The file `debug.log` will still appear as an untracked file when you run `git status`.

If `debug.log` had already been committed in a previous commit, you would use `git restore --staged debug.log` instead of `git rm --cached debug.log`. The `git restore` command is the modern way to unstage files that are already tracked.

### Exercise 7: Create a Multi-File Commit and Inspect

**Task.** You are starting a new Python project. Create the following files, stage them, commit them with a meaningful message, and then verify the commit was recorded correctly.

1. Create `main.py` with the content `def main():` on the first line and `print("hello")` on the second.
2. Create `requirements.txt` with the content `pytest==7.0.0`.
3. Create `README.md` with the content `# My Python Project`.
4. Stage all three files in a single command.
5. Commit with the message "Initial project setup with main script, dependencies, and readme".
6. Verify the commit appears in the log.

**Hint.** Use multiple `echo` commands, one per file. Chain the `echo` commands with line breaks using `>>` for multi-line files. Stage all files at once with `git add .` or list them explicitly separated by spaces.

**Solution.**

```bash
echo "def main():" > main.py
echo 'print("hello")' >> main.py
echo "pytest==7.0.0" > requirements.txt
echo "# My Python Project" > README.md
git add main.py requirements.txt README.md
git commit -m "Initial project setup with main script, dependencies, and readme"
git log --oneline
```

The `git log --oneline` output should show one new commit at the top with your message. The output confirms the commit was created and that all three files are now part of the repository history. You can also verify the file contents of the committed version with `git show HEAD` which displays the commit details along with the diff.

### Exercise 8: Stage a Subset of Changed Files

**Task.** You have made changes to four files, but only two of them are related to the bug fix you are working on. Stage exactly the two bug-fix files and commit them, leaving the other two files modified but unstaged.

1. Create four files: `bugfix.py`, `feature-x.py`, `notes.txt`, and `todo.md`, each with any content.
2. Stage and commit all four files together.
3. Modify `bugfix.py` and `feature-x.py` by adding a new line to each.
4. Modify `notes.txt` and `todo.md` by adding a new line to each.
5. Stage only `bugfix.py` and `notes.txt`.
6. Commit them with the message "Fix null pointer in user lookup and update notes".
7. Confirm that `feature-x.py` and `todo.md` remain modified but unstaged.

**Hint.** After step 6, run `git status` to see which files remain unstaged. The files you did not stage should appear under "Changes not staged for commit." The staged files should appear under "Changes to be committed."

**Solution.**

```bash
echo "def fix_bug(): pass" > bugfix.py
echo "def feature_x(): pass" > feature-x.py
echo "meeting notes" > notes.txt
echo "todo items" > todo.md
git add .
git commit -m "Initial files"
echo "print('fixed')" >> bugfix.py
echo "print('feature x')" >> feature-x.py
echo "new note" >> notes.txt
echo "new todo" >> todo.md
git add bugfix.py notes.txt
git commit -m "Fix null pointer in user lookup and update notes"
git status
```

The final `git status` shows `feature-x.py` and `todo.md` under "Changes not staged for commit." This demonstrates how staging lets you choose which changes belong in each commit. The `feature-x.py` changes are still in the working directory, waiting to be staged and committed separately with a message like "Add feature X implementation."

### Exercise 9: Compare Working Directory Against the Last Commit

**Task.** Modify a previously committed file and use a single command to see the exact lines you changed, before staging anything.

1. Reuse the repository from Exercise 7.
2. Add a new function to `main.py`: append the line `def helper(): pass` to the file.
3. Run the command that shows the difference between your working directory and the last commit.
4. Interpret the output. Confirm the `+` prefix indicates the line you added.

**Hint.** The command is `git diff` with no arguments. It compares the working directory against the staging area, which at this point matches the last commit because you have not staged anything since the last commit.

**Solution.**

```bash
echo "def helper(): pass" >> main.py
git diff
```

The output shows a `diff --git` header for `main.py`, followed by a hunk header like `@@ -1,2 +1,3 @@` indicating that the old file had 2 lines and the new file has 3 lines. The line `+def helper(): pass` is prefixed with `+`, confirming it is an addition. If you had deleted lines, they would appear with a `-` prefix. The hunk header `@@ -1,2 +1,3 @@` breaks down as follows: the left side (`-1,2`) means the old file starting at line 1 with 2 lines, and the right side (`+1,3`) means the new file starting at line 1 with 3 lines.

## Tier 3 — Advanced

### Exercise 10: You Changed Files but Do Not Know the Current State

**Scenario.** You joined a project three weeks ago. You have been making changes to several files, but you are not sure what state the repository is in. You need to answer these questions:

- Which branch am I on?
- Have I committed anything since yesterday?
- Are there unstaged changes in the working directory?
- How many commits exist in total on this branch?

**Task.** Write the sequence of commands that answers all four questions. Then describe what each command's output would tell you. Do not assume any specific file names or branch names.

**Solution.**

```bash
# Question 1: Which branch am I on?
git branch

# Question 2: Have I committed anything since yesterday?
git log --oneline --since="yesterday"

# Question 3: Are there unstaged changes?
git status

# Question 4: How many commits exist on this branch?
git log --oneline | wc -l
```

On Linux and macOS, use `wc -l`. On Windows PowerShell, use `Measure-Object -Line`:

```bash
git log --oneline | Measure-Object -Line
```

On Windows Command Prompt or Git Bash, `wc -l` is available:

```bash
git log --oneline | wc -l
```

These four commands give you a complete picture of the repository state. Run them in order whenever you sit down to work on a project you have not touched in a while. The `git branch` command shows your current branch with an asterisk. The `git log --since="yesterday"` shows commits made in the last 24 hours -- if this returns nothing, you have not committed today. The `git status` command shows any unstaged or untracked files. The `wc -l` count gives you a sense of the project's history depth.

### Exercise 11: Recover an Accidentally Deleted File

**Scenario.** You deleted a file called `config.yaml` from your working directory. You did not stage the deletion. The file was tracked in the most recent commit. You need to get it back.

**Task.** Determine the correct command or sequence of commands to restore `config.yaml` to the working directory without losing any other changes you might have made. Then execute the restoration.

**Hint.** The `git restore` command can recover a file from the last commit. The syntax is `git restore <file>`. If you are using an older version of Git that does not have `git restore`, you can use `git checkout -- <file>`. Both commands overwrite the working directory version with the committed version.

**Solution.**

```bash
# Modern Git approach (Git 2.23+):
git restore config.yaml

# Older Git approach:
git checkout -- config.yaml
```

Both commands replace the working directory version of `config.yaml` with the version from the most recent commit. Since the deletion was not staged, Git knows the working directory version is different from the committed version and restores the committed version. If you had staged the deletion with `git add config.yaml` before trying to restore, you would need to unstage it first with `git restore --staged config.yaml` and then restore the working directory copy.

If you want to restore a file from a specific commit other than the most recent, you can specify the commit hash:

```bash
git restore --source a1b2c3d config.yaml
```

This restores `config.yaml` as it existed in commit `a1b2c3d`, regardless of what the most recent commit contains.

### Exercise 12: Diagnose a Silent Repository

**Scenario.** You created a directory, ran `git init`, created a file called `app.py` with 50 lines of code, ran `git status`, and got this output:

```
On branch main

No commits yet

nothing to commit (create/copy files and use "git add" to track)
```

Wait. You created a file. Why does Git say there is nothing to commit? The file exists on disk. You can see it in your editor and in your file explorer. `app.py` has real content. Yet Git is telling you there is nothing to commit.

**Task.** Identify what happened, explain why Git gives this message, and write the commands that will successfully commit the file.

**Solution.**

**What happened:** You created `app.py` in the working directory but did not stage it. Git will never automatically stage or commit files. Every file starts as untracked. The message "nothing to commit" means the staging area is empty, not that no files exist. The parenthetical hint "(create/copy files and use `git add` to track)" is Git telling you exactly what to do next.

**Why Git gives this message:** Git distinguishes between "nothing to commit" (the staging area is empty) and "nothing added to commit but untracked files present" (there are files that Git sees but has not been told to track). If the output did not show an "Untracked files" section, the file might have been created outside the repository directory, or the filename might be listed in a `.gitignore` file.

**The fix:**

```bash
# Verify the file exists
ls app.py

# Check that git sees it
git status

# Stage and commit
git add app.py
git commit -m "Add app.py with initial implementation"
```

If `git status` still does not show `app.py` as untracked after these commands, check whether a `.gitignore` file exists in the repository that might be excluding it:

```bash
cat .gitignore
```

If `app.py` appears in `.gitignore`, either remove the matching pattern from `.gitignore` or use `git add -f app.py` to force-add it. The `.gitignore` file uses glob patterns -- one pattern per line -- to tell Git which files to ignore. A pattern like `*.py` would ignore all Python files, while `app.py` would ignore only that specific file.

### Exercise 13: Branch Navigation Without History Loss

**Scenario.** You are working on the `main` branch. You have two commits. You create a branch called `experiment` and make one commit on it. Then you switch back to `main`. When you run `git log --oneline`, you see only the two original commits. Your experiment commit appears to be missing.

**Task.** Confirm that the experiment commit still exists. Write the commands that prove it is still in the repository, and describe why `git log` on `main` does not show it.

**Solution.**

```bash
# Confirm the experiment branch still exists
git branch

# View the commits on the experiment branch
git log experiment --oneline

# Or view all commits across all branches
git log --oneline --all

# Switch to the experiment branch to see its working tree
git checkout experiment
git log --oneline
```

The reason `git log` on `main` does not show the experiment commit is that `git log` without any arguments shows only commits reachable from the current branch by following parent pointers backward. The commit you made on `experiment` is not an ancestor of `main`'s latest commit. It is reachable only from the `experiment` branch pointer. The `--all` flag overrides this behaviour and shows commits from all branches and all refs.

To bring the experiment changes into `main`:

```bash
git checkout main
git merge experiment
```

After merging, `git log --oneline` on `main` will include the experiment commit. The merge creates a new commit on `main` that has two parents: the previous tip of `main` and the tip of `experiment`. The `experiment` branch is now fully merged and can be deleted with `git branch -d experiment` if you no longer need it.

### Exercise 14: Commit Only Part of a Modified File

**Scenario.** You have a file called `server.py` that contains two unrelated changes. On line 15, you fixed a typo in a log message. On line 42, you added a new endpoint handler. You want to commit only the typo fix first, then commit the new endpoint in a separate commit.

**Task.** Use the `git add` interactive mode to stage only the change on line 15. Write the command that opens the interactive staging interface, and describe the keystrokes you would use to stage only the typo fix hunk.

**Hint.** The `-p` flag (short for `--patch`) on `git add` lets you select which hunks to stage. Git will show each hunk one at a time and ask what you want to do with it.

**Solution.**

```bash
git add -p server.py
```

Git displays the first hunk of changes. For each hunk, you can type:

- `y` to stage this hunk
- `n` to skip this hunk
- `s` to split the hunk into smaller hunks (if the changes are close together)
- `e` to manually edit which lines to stage
- `q` to quit

In this scenario, the two changes are on different lines in the file, so Git likely presents them as two separate hunks. For the first hunk (the typo fix on line 15), type `y` to stage it. For the second hunk (the new endpoint on line 42), type `n` to skip it. Then run `git diff --staged` to verify that only the typo fix is staged, and `git diff` to confirm the new endpoint remains unstaged.

After committing the typo fix, you can stage and commit the new endpoint separately:

```bash
git commit -m "Fix typo in log message"
git add server.py
git commit -m "Add new health check endpoint"
```

This produces a clean two-commit history where each commit contains only related changes. When another developer reviews your pull request, they can see the typo fix and the new endpoint as separate logical changes, making the review faster and more accurate.

To verify which hunks ended up in each commit, run `git log -p` after both commits. The `-p` flag shows the diff for each commit. You should see the typo fix diff under the first commit message and the endpoint addition diff under the second commit message.

### Exercise 15: Understanding the .gitignore File

**Scenario.** You create a file called `secrets.json` in your repository that contains database credentials. You do not want this file to be tracked by Git or accidentally committed. You also have a directory called `node_modules/` that you never want to track.

**Task.** Create a `.gitignore` file in your repository that ignores both `secrets.json` and the entire `node_modules/` directory. Verify that Git no longer shows these files as untracked.

**Hint.** The `.gitignore` file uses one pattern per line. To ignore a directory and everything inside it, use `directory-name/`. For a specific file, use its exact filename.

**Solution.**

```bash
echo "secrets.json" > .gitignore
echo "node_modules/" >> .gitignore
git add .gitignore
git commit -m "Add gitignore for secrets and dependencies"
```

After committing `.gitignore`, any file named `secrets.json` and any directory named `node_modules/` will be invisible to Git. They will not appear in `git status` output and cannot be accidentally staged with `git add .`. If you need to override `.gitignore` for a specific file, use `git add -f filename` to force-add it.

The `.gitignore` file itself should be committed to the repository so that every developer on the team benefits from the same ignore rules. Patterns in `.gitignore` support wildcards: `*.log` ignores all files ending in `.log`, and `build/` ignores a directory called `build` at any level. A leading slash like `/build` would ignore only the `build` directory at the repository root.
