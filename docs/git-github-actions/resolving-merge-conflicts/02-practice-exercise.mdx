You will deliberately create merge conflicts, resolve them manually, use `--ours` and `--theirs` strategies, abort a merge, and inspect the Git index during a conflict. Every command works identically on Windows, macOS, and Linux.

## Setup

```bash
mkdir -p ~/conflict-lab
cd ~/conflict-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create a file that both branches will modify in conflicting ways:

```bash
cat > greetings.py << 'EOF'
def greet(name):
    return f"Hello, {name}!"


def farewell(name):
    return f"Goodbye, {name}!"


if __name__ == "__main__":
    print(greet("World"))
    print(farewell("World"))
EOF
git add greetings.py
git commit -m "Initial greetings module"
```

## The Walkthrough

### Step 1: Create divergent branches with overlapping changes

```bash
git switch -c feature-uppercase
sed -i 's/Hello/HI THERE/' greetings.py
sed -i 's/Goodbye/SEE YOU/' greetings.py
git add greetings.py
git commit -m "Change greetings to uppercase style"
```

Expected output:

```
[feature-uppercase a1b2c3d] Change greetings to uppercase style
 1 file changed, 2 insertions(+), 2 deletions(-)
```

Now switch back to `main` and modify the same lines differently:

```bash
git switch main
sed -i 's/Hello/Hey/' greetings.py
sed -i 's/Goodbye/Cya/' greetings.py
git add greetings.py
git commit -m "Change greetings to casual style"
```

Expected output:

```
[main d4e5f6a] Change greetings to casual style
 1 file changed, 2 insertions(+), 2 deletions(-)
```

Your history now diverges with both branches changing the same lines:

```
  ((Initial)) ──▶ ((d4e5f6a))  ← main (casual style)
       │                       ← HEAD
       └──▶ ((a1b2c3d))  ← feature-uppercase (uppercase style)
```

**Checkpoint:** Run `git log --oneline --graph --all` to confirm the fork shape.

### Step 2: Attempt to merge and trigger a conflict

```bash
git merge feature-uppercase
```

Expected output:

```
Auto-merging greetings.py
CONFLICT (content): Merge conflict in greetings.py
Automatic merge failed; fix conflicts and then commit the result.
```

Git automatically merged what it could but found conflicting changes in `greetings.py`. The merge is paused.

**Under the hood:** Git stored three versions of `greetings.py` in its index:
- `:1:greetings.py` — the common ancestor (initial commit)
- `:2:greetings.py` — your version (casual style, on `main`)
- `:3:greetings.py` — their version (uppercase style, on `feature-uppercase`)

### Step 3: Inspect the conflict state

```bash
git status
git diff --name-only --diff-filter=U
```

Expected output:

```
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  both modified:   greetings.py
```

`greetings.py` is the only conflicted file. Now view the conflict markers:

```bash
cat greetings.py
```

Expected output:

```
def greet(name):
<<<<<<< HEAD
    return f"Hey, {name}!"
=======
    return f"HI THERE, {name}!"
>>>>>>> feature-uppercase


def farewell(name):
<<<<<<< HEAD
    return f"Cya, {name}!"
=======
    return f"SEE YOU, {name}!"
>>>>>>> feature-uppercase


if __name__ == "__main__":
    print(greet("World"))
    print(farewell("World"))
```

Both function bodies have conflict markers because each branch changed the same two lines differently.

### Step 4: View the three index stages

Inspect what each side contributed:

```bash
git show :1:greetings.py  # base (common ancestor)
```

Expected output — the original file with "Hello" and "Goodbye".

```bash
git show :2:greetings.py  # ours (main / HEAD)
```

Expected output — the file with "Hey" and "Cya".

```bash
git show :3:greetings.py  # theirs (feature-uppercase)
```

Expected output — the file with "HI THERE" and "SEE YOU".

These three versions are Git's internal staging slots. You can also use `git diff` to compare them:

```bash
git diff :2:greetings.py :3:greetings.py
```

This shows the difference between "ours" and "theirs" — what you must decide between.

**Checkpoint:** Run `git log --merge` to see commits from both branches that touch `greetings.py`:

```
d4e5f6a Change greetings to casual style
a1b2c3d Change greetings to uppercase style
```

### Step 5: Resolve manually — create the correct combined version

Edit `greetings.py` to produce a file that keeps the best of both. The intent was to have a friendly greeting and a farewell, not shouting. Remove the markers and pick the friendlier phrasing:

```python
def greet(name):
    return f"Hey, {name}!"


def farewell(name):
    return f"Goodbye, {name}!"


if __name__ == "__main__":
    print(greet("World"))
    print(farewell("World"))
```

On Windows, use any text editor (Notepad, VS Code). On macOS/Linux, use `nano` or `vim`:

```bash
# Example with sed — replace each conflicted region
sed -i 's/HI THERE/Hey/' greetings.py
sed -i 's/SEE YOU/Goodbye/' greetings.py
# Remove conflict markers
sed -i '/^<<<<<<< /d' greetings.py
sed -i '/^=======/d' greetings.py
sed -i '/^>>>>>>> /d' greetings.py
```

Verify the file has no markers and the content is correct:

```bash
cat greetings.py
```

Expected output — clean Python code with no `<<<<<<<` or `>>>>>>>` lines.

### Step 6: Stage the resolved file and continue the merge

```bash
git add greetings.py
git status
```

Expected output — the merge conflict is resolved, and Git is ready for the merge commit:

```
On branch main
All conflicts fixed but you are still merging.
  (use "git commit" to conclude merge)
```

Complete the merge:

```bash
git commit -m "Merge feature-uppercase: keep casual greeting, original farewell"
```

Expected output:

```
[main g7h8i9j] Merge feature-uppercase: keep casual greeting, original farewell
```

Verify the merge commit:

```bash
git log --oneline --graph --all
```

Expected output:

```
*   g7h8i9j (HEAD -> main) Merge feature-uppercase: keep casual greeting, original farewell
|\
| * a1b2c3d (feature-uppercase) Change greetings to uppercase style
* | d4e5f6a Change greetings to casual style
|/
* e5f6a7b Initial greetings module
```

### Step 7: Aborting a merge (practice)

Create another conflicting scenario and practice aborting:

```bash
git switch -c feature-abort-test
sed -i 's/greet/hello_func/' greetings.py
git add greetings.py
git commit -m "Rename greet function"
```

Switch back to `main` and create a different change:

```bash
git switch main
sed -i 's/greet/welcome/' greetings.py
git add greetings.py
git commit -m "Rename greet to welcome"
```

Attempt to merge and then abort:

```bash
git merge feature-abort-test
```

You see a conflict. Instead of resolving it:

```bash
git merge --abort
```

Expected output — no error, you are back to the pre-merge state:

```
git status
On branch main
nothing to commit, working tree clean
```

**Under the hood:** `git merge --abort` deleted `.git/MERGE_HEAD` and restored the working directory and index to match `HEAD` (the state before `git merge` was run).

### Step 8: Using `--ours` and `--theirs` strategies

Create one more conflict and resolve it by picking one entire side:

```bash
git switch -c feature-ours-theirs
sed -i 's/return f"Hey/return f"Howdy/' greetings.py
git add greetings.py
git commit -m "Change Hey to Howdy"
git switch main
sed -i 's/return f"Hey/return f"Hi/' greetings.py
git add greetings.py
git commit -m "Change Hey to Hi"
git merge feature-ours-theirs
```

Expected conflict output. Now resolve by accepting the entire incoming branch:

```bash
git checkout --theirs greetings.py
git add greetings.py
git commit -m "Accept their Howdy version entirely"
```

Verify:

```bash
grep Howdy greetings.py
```

The file now says "Howdy" — `--theirs` took the incoming branch's version completely, discarding `main`'s "Hi".

## Common Errors at This Stage

### Error 1: `fatal: Unable to create '<file>': File exists.` during merge

Git tried to write a conflicted file but a file with that exact name already exists in the working tree from a previous, unrelated operation.

**Fix:** Remove the stale file or rename it. Then re-run the merge. If the merge was aborted previously, run `git clean -fd` to remove untracked files before retrying.

### Error 2: `error: you need to resolve your current index first`

You ran `git merge --continue` or `git commit` before adding the resolved files. Git requires conflicted files to be staged (`git add`) before it can create the merge commit.

**Fix:** Run `git add <file>` for each resolved file, then `git merge --continue`.

### Error 3: `Merge conflict in <file>` even though you did not change that file

You may have run `git merge` from the wrong branch. The "theirs" side may have changes you did not expect. Alternatively, line-ending normalization (CRLF vs LF) or file permission changes can cause Git to report conflicts on every line.

**Fix:** First, confirm you are on the correct branch with `git branch`. If line endings are the cause, commit a `.gitattributes` file with `* text=auto` to normalize endings, then re-attempt the merge. Use `git merge --abort` first if needed.

### Error 4: After resolving, the merged file does not work even though there are no conflict markers

Semantic conflicts — where the code compiles or parses but behaves incorrectly — are the hardest to catch. Git sees no text conflict, but the combined logic is wrong.

**Fix:** Always run your test suite and do a quick manual check after any merge. A `git diff --check` flags whitespace errors but not logic errors. Add automated tests that exercise the merged area before marking the merge as complete.

## What You Just Learned

- Merge conflicts happen when two branches change the same lines of the same file
- Git stores three versions in the index during a conflict: base (`:1`), ours (`:2`), theirs (`:3`)
- Conflict markers `<<<<<<<`, `=======`, `>>>>>>>` show what each side contributed
- Resolve by editing the file, removing markers, staging with `git add`, then `git commit` or `git merge --continue`
- `git merge --abort` reverts to the pre-merge state safely
- `git checkout --ours` and `git checkout --theirs` accept one side completely
- `git log --merge` shows the conflicting commits from both branches
- Semantic conflicts (no markers, wrong logic) are your responsibility to catch with tests
