## Setup

Open your terminal and create a temporary directory for this exercise.

```bash
mkdir ~/git-stash-practice
cd ~/git-stash-practice
git init
```

On Windows (PowerShell), the same commands look like this:

```powershell
mkdir $env:USERPROFILE\git-stash-practice
cd $env:USERPROFILE\git-stash-practice
git init
```

On Windows (cmd.exe), use:

```cmd
mkdir %USERPROFILE%\git-stash-practice
cd /d %USERPROFILE%\git-stash-practice
git init
```

Now create a starting file and make an initial commit:

```bash
echo "# Payment Module" > payment.py
echo "DEBUG = False" > config.py
git add .
git commit -m "Initial commit"
```

Your starting state should look like this:

```bash
git log --oneline
```

Output:

```
abc1234 Initial commit
```

## The Walkthrough

### Step 1: Make some changes

Edit both files to simulate real work in progress.

```bash
echo "def process_payment(amount):" >> payment.py
echo "    return amount * 1.05" >> payment.py
echo "DEBUG = True" > config.py
```

Stage one file and leave the other unstaged:

```bash
git add config.py
```

Check your status:

```bash
git status
```

Output:

```
On branch main
Changes to be committed:
  modified:   config.py

Changes not staged for commit:
  modified:   payment.py
```

Your working directory now has a staged change (`config.py`) and an unstaged change (`payment.py`). This is a common real-world state: you staged one file because it is ready, but the other file still needs work.

### Step 2: Stash everything with a message

Your manager calls with an urgent request. You need to switch contexts immediately. Save everything to the stash.

```bash
git stash push -m "wip: payment surcharge"
```

Expected output:

```
Saved working directory and index state On main: wip: payment surcharge
```

Verify the working directory is clean:

```bash
git status
```

Output:

```
On branch main
nothing to commit, working tree clean
```

Both files have been reverted to their committed state. `payment.py` contains only `# Payment Module`, and `config.py` contains `DEBUG = False`. Your changes are safely stored in the stash.

### Step 3: List stashes and inspect

```bash
git stash list
```

Output:

```
stash@{0}: On main: wip: payment surcharge
```

The `stash@{0}` is the reference for this stash. The `0` means it is the most recent (top of the stack). If you create more stashes later, this number will increment and shift as stashes are added and removed.

Look at what is inside the stash:

```bash
git stash show stash@{0}
```

Output:

```
 payment.py | 2 ++
 config.py  | 2 +-
 2 files changed, 3 insertions(+), 1 deletion(-)
```

For a more detailed view showing the actual diff:

```bash
git stash show -p stash@{0}
```

Output:

```
diff --git a/config.py b/config.py
index abc1234..def5678 100644
--- a/config.py
+++ b/config.py
@@ -1 +1 @@
-DEBUG = False
+DEBUG = True
diff --git a/payment.py b/payment.py
index abc1234..def5678 100644
--- a/payment.py
+++ b/payment.py
@@ -1 +1,3 @@
 # Payment Module
+def process_payment(amount):
+    return amount * 1.05
```

The `-p` flag shows the patch (full diff) of what the stash contains. This is useful when you cannot remember what a particular stash holds.

### Step 4: Simulate doing urgent work

Pretend you fixed the urgent issue. Make a quick change and commit it on this branch:

```bash
echo "URGENT_FIX = True" > urgent.py
git add urgent.py
git commit -m "Urgent hotfix"
```

Check the log:

```bash
git log --oneline
```

Output:

```
def7890 Urgent hotfix
abc1234 Initial commit
```

The urgent fix is committed. Now you need to get back to your payment module changes.

### Step 5: Restore your changes with pop

```bash
git stash pop
```

Expected output:

```
Auto-merging config.py
On branch main
Changes not staged for commit:
  modified:   payment.py

no changes added to commit (use "git add" and/or "git commit -a")
```

Wait — `config.py` was auto-merged but is no longer staged. The pop restored your working tree changes but did not restore the staging state. This is a critical detail: `git stash pop` restores the working tree changes but does not re-stage them. You will need to stage files again after popping.

Check that the stash list is now empty:

```bash
git stash list
```

Output: (nothing, meaning the stash was removed by `pop`)

### Step 6: Create a second stash with keep-index

Let us try again with a different flag. First, reset to a clean state:

```bash
git checkout -- .
```

Now make changes again and stage one:

```bash
echo "TAX_RATE = 0.08" >> config.py
echo "def calculate_tax(amount):" >> payment.py
echo "    return amount * 0.08" >> payment.py
git add config.py
```

This time, use `--keep-index`:

```bash
git stash push --keep-index -m "wip: tax calculation"
```

Expected output:

```
Saved working directory and index state On main: wip: tax calculation
```

Check status:

```bash
git status
```

Output:

```
On branch main
Changes to be committed:
  modified:   config.py
```

The `--keep-index` flag told Git to stash only the unstaged changes (`payment.py`) and leave the staged changes (`config.py`) in place. This is useful when you want to commit the staged changes now but save the unstaged work for later.

### Step 7: Apply a stash without removing it

Commit the kept-index change:

```bash
git commit -m "Add tax rate to config"
```

Now bring back the stashed payment.py changes but keep the stash on the stack:

```bash
git stash apply
```

Expected output:

```
Changes not staged for commit:
  modified:   payment.py
```

Verify the stash is still there:

```bash
git stash list
```

Output:

```
stash@{0}: On main: wip: tax calculation
```

The stash remains because you used `apply` instead of `pop`. This is useful when you want to apply the same stash to multiple branches.

### Step 8: Create a stash branch

You realize the payment changes should have been on a feature branch all along. Create a new branch from the stash:

```bash
git stash branch feature/payment-tax
```

Expected output:

```
Switched to a new branch 'feature/payment-tax'
On branch feature/payment-tax
Changes not staged for commit:
  modified:   payment.py
```

What happened here? `git stash branch` creates a new branch from the commit where the stash was originally made, switches to it, and then applies the stash. The stash is also dropped after the branch is created. This is the cleanest way to turn a stash into proper branch-based work.

## Common Errors at This Stage

### Error 1: Stash pop with conflicts

**Error message:**

```
Auto-merging config.py
CONFLICT (content): Merge conflict in config.py
The stash entry is kept in case you need it again.
```

**Why:** The files in the stash conflict with the current state of the branch. This happens when the branch you are popping onto has different content in the same files.

**Fix:** Open the conflicted file, resolve the conflict markers, stage the file, and then manually run `git stash drop` to remove the stash. Git keeps the stash on purpose after a conflict — even if you used `pop` — because the pop could not complete cleanly.

### Error 2: No local changes to save

**Error message:**

```
No local changes to save
```

**Why:** You ran `git stash` but your working directory is clean — there are no modified tracked files to stash.

**Fix:** If you want to stash untracked files, add `--include-untracked` or `-u` flag. If you genuinely have no changes, there is nothing to stash. Run `git status` to confirm.

### Error 3: Stash apply on a dirty working directory

**Error message:**

```
Cannot apply stash: Your local changes would be overwritten by merge.
Please commit your changes or stash them before you apply stash.
```

**Why:** You tried to apply a stash while you have uncommitted changes in your working directory. Git refuses because the apply might overwrite your current work.

**Fix:** Either commit your current changes, stash them first (`git stash push -m "temp"` then apply the original stash), or discard your current changes if they are not needed.

### Error 4: stash@{N} not found

**Error message:**

```
stash@{3} is not a valid reference
```

**Why:** You referenced a stash index that does not exist. The stack only has 3 stashes (indices 0, 1, 2), so index 3 is invalid.

**Fix:** Run `git stash list` to see how many stashes exist. Use an index that exists, or omit the index entirely to use the top of the stack (`stash@{0}`).

## What You Just Learned

- `git stash push -m "message"` saves dirty tracked files to the stash stack and cleans the working directory.
- `git stash pop` applies the top stash and removes it from the stack; `git stash apply` applies it without removal.
- `--keep-index` stashes only unstaged changes, leaving staged files intact for immediate commit.
- `git stash show -p` displays the full diff inside a stash, helping you identify what is saved.
- `git stash branch` creates a new branch from the original stash commit — the preferred way to turn a stash into proper branch-based development.
- Stashes are local only and are never pushed to remote repositories. They are a temporary save point, not a replacement for branches and commits.
