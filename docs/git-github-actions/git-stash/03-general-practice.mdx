## Tier 1 — Beginner

### Exercise 1: Save your work and verify

**Task:** Initialize a git repo, create a file called `notes.txt` with the content "Draft notes", commit it, then edit the file to add "More notes". Stash this change with a message "wip: notes" and verify the working directory is clean.

<details>
<summary>Hint</summary>

Use `git stash push -m "wip: notes"` to stash. Use `git status` to verify the working directory is clean.

</details>

```bash
mkdir ~/stash-ex1 && cd ~/stash-ex1 && git init
echo "Draft notes" > notes.txt
git add notes.txt && git commit -m "Initial"
echo "More notes" >> notes.txt
git stash push -m "wip: notes"
git status
```

**Expected result:** `git status` shows `nothing to commit, working tree clean`.

---

### Exercise 2: List and inspect stashes

**Task:** Building on Exercise 1, create three separate stashes with different messages, then list all stashes and show the patch diff of the second stash.

<details>
<summary>Hint</summary>

Create three stashes by making changes and stashing them one at a time. Use `git stash list` to see all. Use `git stash show -p stash@{1}` to see the patch of the second stash.

</details>

```bash
# Create three stashes
echo "Change A" >> notes.txt && git stash push -m "change A"
echo "Change B" >> notes.txt && git stash push -m "change B"
echo "Change C" >> notes.txt && git stash push -m "change C"

# List all stashes
git stash list

# Show the patch of the second stash (index 1)
git stash show -p stash@{1}
```

**Expected result:** `git stash list` shows 3 stashes. The patch output shows only the lines from "Change B".

---

### Exercise 3: Pop and verify removal

**Task:** With the three stashes from Exercise 2 still on the stack, pop the top stash and verify it was removed from the list.

<details>
<summary>Hint</summary>

`git stash pop` applies and removes stash@{0}. Run `git stash list` after to confirm only 2 stashes remain.

</details>

```bash
git stash pop
git stash list
```

**Expected result:** The notes file now contains "Change C" (the top stash). Only 2 stashes remain in the list.

---

### Exercise 4: Apply without removal

**Task:** Apply the stash that contains "Change A" without removing it from the stack.

<details>
<summary>Hint</summary>

Use `git stash apply stash@{N}` with the correct index. Run `git stash list` after to confirm the stash count did not change.

```bash
git stash apply stash@{1}
git stash list
```

**Expected result:** The notes file now contains "Change A" in addition to "Change C". The stash list still shows 2 entries.

</details>

## Tier 2 — Intermediate

### Exercise 1: Keep-index workflow

**Task:** Stage a change in one file, leave another file unstaged, then stash only the unstaged changes using `--keep-index`. Commit the staged file, then apply the stash.

<details>
<summary>Hint</summary>

1. Create two files, commit both.
2. Edit file1 and stage it with `git add file1`.
3. Edit file2 (do not stage it).
4. Run `git stash push --keep-index -m "keep file1 staged"`.
5. Confirm file1 is still staged with `git status`.
6. Commit file1: `git commit -m "file1 ready"`.
7. Apply the stash: `git stash pop`.

</details>

```bash
# Setup
mkdir ~/stash-keepindex && cd ~/stash-keepindex && git init
echo "version 1" > file1.txt && echo "version 1" > file2.txt
git add . && git commit -m "Initial"

# Stage one, edit the other
echo "file1 staged change" > file1.txt && git add file1.txt
echo "file2 unstaged change" > file2.txt

# Stash only unstaged changes (file2)
git stash push --keep-index -m "save file2 only"

# Verify file1 is still staged
git status

# Commit the staged file
git commit -m "file1 update"

# Apply the stashed file2 changes
git stash pop
```

**Expected result:** After `--keep-index`, `git status` shows `file1.txt` staged and `file2.txt` clean (stashed away). After the commit and pop, `file2.txt` contains the unstaged change from earlier.

### Exercise 2: Stash, switch branches, and pop

**Task:** Create a feature branch, make changes, stash them, switch to main, make a commit, switch back to the feature branch, pop the stash.

<details>
<summary>Hint</summary>

1. Make a commit on main with `README.md`.
2. Create and switch to a feature branch: `git checkout -b feature/update`.
3. Edit `README.md` and stash the change.
4. Switch back to main: `git checkout main`.
5. Make a different change to `README.md` and commit it.
6. Switch back to feature: `git checkout feature/update`.
7. Pop the stash. Observe if there is a merge conflict.

</details>

```bash
mkdir ~/stash-switch && cd ~/stash-switch && git init
echo "Initial content" > README.md
git add README.md && git commit -m "Initial"

# Create feature branch and make changes
git checkout -b feature/update
echo "Feature work" >> README.md
git stash push -m "feature wip"

# Switch back to main and make a different change
git checkout main
echo "Main update" >> README.md
git add README.md && git commit -m "main update"

# Switch to feature and pop
git checkout feature/update
git stash pop
```

**Expected result:** There may be a merge conflict in `README.md` because both branches modified the same file. Resolve it if it appears. This demonstrates why stashing across branches can cause conflicts.

### Exercise 3: Stash untracked files

**Task:** Create a new file that is not tracked by git, then stash it together with your tracked changes using `--include-untracked`.

<details>
<summary>Hint</summary>

Regular `git stash` ignores untracked files. Use `git stash push --include-untracked -m "all changes"` or the shorthand `git stash push -u -m "all changes"` to include them.

</details>

```bash
mkdir ~/stash-untracked && cd ~/stash-untracked && git init
echo "tracked file" > tracked.txt
git add tracked.txt && git commit -m "Initial"

# Create both a tracked modification and a new untracked file
echo "modified" >> tracked.txt
echo "brand new file" > new.txt

# Try stashing without --include-untracked
git stash push -m "without untracked"
git stash list
git stash show stash@{0}
```

Note that `new.txt` is NOT in the stash and is still present in the working directory. Now clean up and do it correctly:

```bash
# Pop the stash to restore tracked.txt
git stash pop

# Now stash WITH untracked files
git stash push --include-untracked -m "with untracked"

# Verify new.txt is gone from the working directory
ls new.txt 2>/dev/null && echo "new.txt still exists" || echo "new.txt is gone"

# Pop to restore everything
git stash pop
```

**Expected result:** With `--include-untracked`, `new.txt` is stashed and removed from the working directory. Without it, `new.txt` stays behind.

### Exercise 4: Apply the same stash to multiple branches

**Task:** Apply one stash containing a useful utility function to two different feature branches.

<details>
<summary>Hint</summary>

1. Create and apply a stash with a reusable change on branch A.
2. Switch to branch B that was created from the same base commit.
3. Apply the same stash (it is still on the stack because you used `apply`, not `pop`).
4. Each branch now has the change without the stash being consumed.

</details>

```bash
mkdir ~/stash-reuse && cd ~/stash-reuse && git init
echo "base" > utils.py
git add utils.py && git commit -m "Initial"

# Create stash on main
echo "def helper(): pass" >> utils.py
git stash push -m "add helper function"

# Create two branches from the same base
git checkout -b feature/logging
git stash apply
git commit -am "logging with helper"

# Switch to second branch and apply same stash
git checkout main
git checkout -b feature/caching
git stash apply
git commit -am "caching with helper"

# Verify both branches have the helper
git log --oneline feature/logging
git log --oneline feature/caching
```

**Expected result:** Both branches have the helper function committed. The stash is still on the stack because `apply` does not remove it.

## Tier 3 — Advanced

### Exercise 1: Recover a dropped stash

**Scenario:** You have been working on a complex refactor for an hour. You stashed it, then accidentally ran `git stash drop` or `git stash clear`. All your work seems gone.

**Task:** Recover the dropped stash.

<details>
<summary>Hint</summary>

A dropped stash is not immediately garbage collected. Git keeps unreachable commits for a grace period (default 30 days). Use `git fsck --lost-found` to find dangling commits, or `git reflog show refs/stash` to see the stash reflog (if the stash was recently dropped).

</details>

```bash
mkdir ~/stash-recover && cd ~/stash-recover && git init
echo "important data" > data.txt
git add data.txt && git commit -m "Initial"
echo "1 hour of refactoring" >> data.txt

# Save and accidentally drop
git stash push -m "big refactor"
git stash drop stash@{0}
git stash list  # empty!
```

Approach 1 — check the stash reflog:

```bash
git reflog show refs/stash
```

If the stash was the only one and was just dropped, the reflog might show the old stash state. If it appears, you can `git stash apply` the commit hash shown.

Approach 2 — find dangling commits:

```bash
git fsck --lost-found | grep commit
```

This lists dangling commit objects. One or more of them is your stash. Inspect them with `git show <hash>` until you find the one with "1 hour of refactoring". Then create a branch from it:

```bash
git branch recovered-work <hash>
git checkout recovered-work
```

**What happened:** Git's garbage collection runs periodically, not on every command. When you dropped the stash, Git removed the `refs/stash` reference, but the actual commit objects remain on disk until `git gc` runs (usually after 30 days). The `git fsck --lost-found` command finds these orphaned commits.

### Exercise 2: Partial stash with pathspec

**Scenario:** You have modified 5 files but only want to stash 2 of them, leaving the other 3 modified in the working directory. The other 3 are not ready to be committed or stashed together.

**Task:** Stash only specific files without using `git add` or interactive staging.

<details>
<summary>Hint</summary>

`git stash push -- <pathspec>` stashes only the changes matching the pathspec. Files not matching the pathspec remain dirty in the working directory. This works with tracked files only.

</details>

```bash
mkdir ~/stash-pathspec && cd ~/stash-pathspec && git init

# Create 5 files
for i in 1 2 3 4 5; do echo "file $i" > "module_$i.py"; done
git add . && git commit -m "Initial"

# Modify all 5 files
for i in 1 2 3 4 5; do echo "change $i" >> "module_$i.py"; done

# Stash only module_1.py and module_3.py
git stash push -m "partial: modules 1 and 3" -- module_1.py module_3.py

# Verify: only module_1.py and module_3.py are clean
git status

# The other 3 files still have their changes
cat module_2.py
```

**Expected result:** `git status` shows `module_2.py`, `module_4.py`, and `module_5.py` as modified. `module_1.py` and `module_3.py` are clean — their changes are in the stash.

### Exercise 3: Stack management — reorder and clean

**Scenario:** You have accumulated 6 stashes over 2 days of work. The stashes are numbered 0 through 5. You need to reorder them and remove the ones that are no longer relevant. Git has no built-in `stash reorder` command.

**Task:** Design a strategy to inspect all 6 stashes, drop the ones you do not need, and apply the remaining ones in a specific order.

<details>
<summary>Solution and reasoning</summary>

Git's stash is a stack (LIFO — last in, first out), but real-world work is rarely linear. You might have stashed things in a random order: a fix for feature A, then an experiment for feature C, then a fix for feature B, then more work on feature A.

Git does not have `git stash reorder`. Here is one approach:

1. **Inspect every stash:** Run `git stash show -p stash@{N}` for each stash from 0 to 5. Take notes on what each contains.

2. **Drop unwanted stashes:** Run `git stash drop stash@{N}` for each stash you want to discard. Be careful — dropping a stash re-indexes every stash above it. For example, if you drop stash@{2}, the old stash@{3} becomes the new stash@{2}.

   A safer way to drop multiple stashes is to start from the highest index first and work downward:
   ```bash
   git stash drop stash@{5}  # drop last first
   git stash drop stash@{4}
   git stash drop stash@{2}
   ```

3. **Apply stashes in desired order:** Apply each remaining stash in the order you want your changes applied. Use `git stash apply` (not `pop`) to keep stashes on the stack in case the order causes conflicts and you need to retry.

   ```bash
   git stash apply stash@{2}  # Apply feature B first
   git stash apply stash@{0}  # Then feature A
   ```

4. **Alternatively, turn stashes into branches:** For a cleaner approach, turn each keeper stash into a branch:
   ```bash
   git stash branch feature/A-work stash@{0}
   git stash branch feature/B-work stash@{1}
   ```
   This gives each stash its own branch where you can commit the work properly, then merge or rebase in the order you want.

5. **Clean up leftover stashes:** Once all keeper stashes are applied or branched, run `git stash clear` to remove everything.

**Key insight:** A growing stash stack (more than 5 entries) is a signal that you should turn stashes into proper branches. Stash is a temporary holding area, not a long-term storage solution. The `git stash branch` command is the bridge between the two.

</details>
