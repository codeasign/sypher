You will create a workflow that uses environment variables at all three levels, dynamically computes values with `$GITHUB_ENV`, sets and consumes secrets, and passes outputs between jobs. By the end you will understand how to manage configuration and secrets securely in GitHub Actions.

## Setup

```bash
# Create a new repository for this practice
gh repo create env-secrets-practice --public --clone
```

```bash
# Create the workflows directory
mkdir -p .github/workflows
```

```bash
# Create a script that uses environment variables
cat > deploy.sh << 'EOF'
#!/bin/bash
echo "Deploying to: $DEPLOY_TARGET"
echo "Version: $APP_VERSION"
echo "Environment: $NODE_ENV"
echo "API endpoint: $API_ENDPOINT"

if [ -n "$API_KEY" ]; then
  echo "API key configured (${#API_KEY} chars)"
else
  echo "WARNING: No API key set"
fi

echo "Deploy complete."
EOF
chmod +x deploy.sh
```

```bash
# Create a simple application
cat > app.py << 'EOF'
import os

def main():
    env = os.getenv('NODE_ENV', 'development')
    log_level = os.getenv('LOG_LEVEL', 'info')
    print(f"Running in {env} mode with log level {log_level}")
    print(f"Database URL configured: {bool(os.getenv('DATABASE_URL'))}")

if __name__ == "__main__":
    main()
EOF

cat > requirements.txt << 'EOF'
EOF
```

## The Walkthrough

### Step 1: Set environment variables at workflow level

Create `.github/workflows/env-demo.yml`:

```yaml
name: Env Demo
on: workflow_dispatch

env:
  GLOBAL_VAR: "I am available everywhere"
  APP_VERSION: "1.0.0"

jobs:
  print-global:
    runs-on: ubuntu-latest
    steps:
      - name: Print global env var
        run: |
          echo "Global var: $GLOBAL_VAR"
          echo "App version: $APP_VERSION"
      - name: Print using expression syntax
        run: |
          echo "Global var: ${{ env.GLOBAL_VAR }}"
          echo "App version: ${{ env.APP_VERSION }}"
```

**Checkpoint:** Commit, push, and trigger manually. Both steps should show the same values.

```bash
git add .github/workflows/env-demo.yml
git commit -m "Add workflow-level env vars"
git push origin main
```

### Step 2: Override env vars at job and step levels

Replace the workflow content:

```yaml
name: Env Demo
on: workflow_dispatch

env:
  NODE_ENV: production
  LOG_LEVEL: info

jobs:
  staging:
    runs-on: ubuntu-latest
    env:
      NODE_ENV: staging  # Overrides workflow-level
      DATABASE_URL: postgres://localhost:5432/staging
    steps:
      - uses: actions/checkout@v4
      - name: Show job env
        run: |
          echo "NODE_ENV=$NODE_ENV"         # staging (from job)
          echo "LOG_LEVEL=$LOG_LEVEL"       # info (from workflow)
          echo "DATABASE_URL=$DATABASE_URL"  # postgres://.../staging
      - name: Override at step level
        env:
          NODE_ENV: staging-test            # Overrides even the job level
        run: |
          echo "NODE_ENV=$NODE_ENV"         # staging-test (from step)
          echo "DATABASE_URL=$DATABASE_URL"  # Still available from job

  production:
    needs: staging
    runs-on: ubuntu-latest
    env:
      NODE_ENV: production
      DATABASE_URL: postgres://prod-host:5432/production
    steps:
      - uses: actions/checkout@v4
      - name: Production env
        run: |
          echo "NODE_ENV=$NODE_ENV"         # production
          echo "DATABASE_URL=$DATABASE_URL"  # postgres://prod-host:.../production
```

**Checkpoint:** Push and observe that each job sees different env var values.

### Step 3: Use `$GITHUB_ENV` to set dynamic values

```yaml
name: Env Demo
on: workflow_dispatch

env:
  NODE_ENV: ci

jobs:
  dynamic:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Compute dynamic values
        run: |
          echo "TIMESTAMP=$(date +%s)" >> $GITHUB_ENV
          echo "BRANCH_NAME=${GITHUB_REF_NAME}" >> $GITHUB_ENV
          echo "SHORT_SHA=${GITHUB_SHA::7}" >> $GITHUB_ENV
      - name: Use dynamic values
        run: |
          echo "Build timestamp: $TIMESTAMP"
          echo "Branch: $BRANCH_NAME"
          echo "Short SHA: $SHORT_SHA"
      - name: Also available here
        run: |
          echo "Same timestamp: $TIMESTAMP"
```

**Checkpoint:** Push and verify the dynamic values are populated and used across steps.

### Step 4: Use `$GITHUB_OUTPUT` to pass values between jobs

```yaml
name: Env Demo
on: workflow_dispatch

jobs:
  compute:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.version.outputs.value }}
      timestamp: ${{ steps.date.outputs.value }}
    steps:
      - uses: actions/checkout@v4
      - id: version
        run: echo "value=2.0.0" >> $GITHUB_OUTPUT
      - id: date
        run: echo "value=$(date +%Y-%m-%d)" >> $GITHUB_OUTPUT

  consume:
    needs: compute
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "Version from compute: ${{ needs.compute.outputs.version }}"
          echo "Timestamp from compute: ${{ needs.compute.outputs.timestamp }}"
```

### Step 5: Set a repository secret

Create a secret in your repository. From your terminal:

```bash
# Generate a random API key for testing
API_KEY="sk-test-$(openssl rand -hex 16)"
```

```bash
# Set it as a repository secret
gh secret set API_KEY --body "$API_KEY"
```

```bash
# Set a plaintext variable
gh variable set DEPLOY_URL --body "https://api.example.com"
```

Verify the secret was set (it will show as masked):

```bash
gh secret list
```

### Step 6: Use the secret in a workflow

```yaml
name: Env Demo
on: workflow_dispatch

env:
  DEPLOY_URL: ${{ vars.DEPLOY_URL }}

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Use secret and variable
        env:
          API_KEY: ${{ secrets.API_KEY }}
        run: |
          echo "Deploying to $DEPLOY_URL"
          ./deploy.sh
      - name: Demonstrate masking
        run: |
          echo "Attempting to print secret: ${{ secrets.API_KEY }}"
```

**Expected output in logs:**

```
Deploying to https://api.example.com
Deploying to: https://api.example.com
Version: 1.0.0
Environment: ci
API endpoint: https://api.example.com
API key configured (35 chars)
Deploy complete.
Attempting to print secret: ***
```

Note that `${{ secrets.API_KEY }}` prints as `***` even though the step tries to echo it. GitHub's runner automatically masks secret values in log output.

### Step 7: Use GITHUB_TOKEN

The `GITHUB_TOKEN` is automatically available in every workflow. Use it to make API calls:

```yaml
name: Env Demo
on: workflow_dispatch

jobs:
  api-call:
    runs-on: ubuntu-latest
    permissions:
      issues: write
    steps:
      - name: Create an issue using GITHUB_TOKEN
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          gh issue create \
            --title "Auto-generated issue" \
            --body "This issue was created by a workflow run #${{ github.run_id }}"
```

**Checkpoint:** After running this workflow, check the Issues tab — a new issue should exist.

## Common Errors at This Stage

### Error 1: Secret appears as `***` even in intended output
**Message:** You want to print a partial secret value, but it is fully masked.

**Why:** GitHub masks secrets anywhere in the log output, including substrings. There is no way to reveal a secret in logs.

**Fix:** Do not try to print secrets. If you need to verify a secret is set, print its length:
```bash
echo "Secret length: ${#API_KEY}"
```

### Error 2: "Secret not found" or empty value
**Message:** `${{ secrets.MY_SECRET }}` evaluates to an empty string.

**Why:** The secret does not exist at the scope you are referencing. Either it was never set, it is set at the org level but you are looking at the repo level, or the secret name does not match.

**Fix:** Verify the secret exists:
```bash
gh secret list
```

Check for typos in the secret name. Secret names are case-insensitive but must match exactly.

### Error 3: Environment variable not available in a step
**Message:** A variable set in a previous step via `$GITHUB_ENV` is not available in the next step.

**Why:** You set the variable in a step that failed, or you are trying to read it in a different job (env vars set via `$GITHUB_ENV` are job-scoped, not workflow-scoped).

**Fix:** Use `$GITHUB_OUTPUT` for cross-job communication. For failed steps, check that the variable is set in a step with `continue-on-error: true`.

### Error 4: GITHUB_TOKEN permission denied
**Message:** `Resource not accessible by integration` when using `GITHUB_TOKEN`.

**Why:** The default `GITHUB_TOKEN` has limited permissions. You need to request specific permissions in the workflow.

**Fix:** Add a `permissions:` block:
```yaml
permissions:
  contents: write
  issues: write
  pull-requests: write
```

Or restrict to read-only:
```yaml
permissions:
  contents: read
```

## What You Just Learned

- Environment variables can be set at workflow, job, and step levels
- Inner scopes override outer scopes for the same variable name
- `$GITHUB_ENV` creates dynamic env vars available to subsequent steps in the same job
- `$GITHUB_OUTPUT` passes values between jobs
- Secrets are encrypted at rest, decrypted at runtime, and masked in log output
- Use `gh secret set` and `gh variable set` to manage secrets and config variables
- `GITHUB_TOKEN` is automatically available for API calls
