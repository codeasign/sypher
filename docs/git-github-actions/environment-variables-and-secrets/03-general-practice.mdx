## Tier 1 — Beginner

### Exercise 1: List all default env vars

**Task:** Create a workflow that prints every default GitHub Actions environment variable (`GITHUB_*` and `RUNNER_*`) in a table format.

**Hint:** Use `env | grep GITHUB_` and `env | grep RUNNER_` to list them.

<details>
<summary>Solution</summary>

```yaml
name: List Default Env Vars
on: workflow_dispatch

jobs:
  list:
    runs-on: ubuntu-latest
    steps:
      - name: Print default env vars
        run: |
          echo "| Variable | Value |"
          echo "|----------|-------|"
          env | grep '^GITHUB_' | sort | while IFS='=' read -r var val; do
            echo "| $var | $val |"
          done
          echo ""
          env | grep '^RUNNER_' | sort | while IFS='=' read -r var val; do
            echo "| $var | $val |"
          done
```

Expected output shows 15+ environment variables with their runtime values.
</details>

### Exercise 2: Pass a value between steps

**Task:** In the first step, set `MY_VAR=hello_world` using `$GITHUB_ENV`. In the second step, print the value. In the third step, verify it is still available.

<details>
<summary>Solution</summary>

```yaml
name: Pass Between Steps
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Set value
        run: echo "MY_VAR=hello_world" >> $GITHUB_ENV
      - name: Read value
        run: echo "MY_VAR = $MY_VAR"
      - name: Still available
        run: echo "Still have MY_VAR = $MY_VAR"
```

All three steps run in the same job, so `$GITHUB_ENV` values are available from step 2 onward.
</details>

### Exercise 3: Set a secret and verify it is masked

**Task:** Use `gh secret set` to store a test secret, then create a workflow that prints it (expect `***` in the log) and prints its length.

**Hint:** Use `${#VAR_NAME}` in bash to get string length.

<details>
<summary>Solution</summary>

```bash
# Set the secret
gh secret set TEST_TOKEN --body "my-super-secret-value-12345"
```

```yaml
name: Masked Secret
on: workflow_dispatch

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Use secret
        env:
          TEST_TOKEN: ${{ secrets.TEST_TOKEN }}
        run: |
          echo "Secret value: $TEST_TOKEN"
          echo "Secret length: ${#TEST_TOKEN}"
```

Expected log output:
```
Secret value: ***
Secret length: 27
```
</details>

## Tier 2 — Intermediate

### Exercise 4: Set env vars based on matrix values

**Task:** Create a matrix workflow that tests on `ubuntu-latest`, `windows-latest`, and `macos-latest`. For each OS, set an environment variable `PLATFORM` to a human-readable name using `$GITHUB_ENV`, then print a summary that includes the platform name.

**Hint:** Use `${{ matrix.os }}` to determine the platform name, or use `runner.os`.

<details>
<summary>Solution</summary>

```yaml
name: Platform Env
on: workflow_dispatch

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
    steps:
      - name: Set platform name
        run: |
          case "${{ runner.os }}" in
            Linux) echo "PLATFORM=Linux (Ubuntu)" >> $GITHUB_ENV ;;
            Windows) echo "PLATFORM=Windows Server" >> $GITHUB_ENV ;;
            macOS) echo "PLATFORM=macOS" >> $GITHUB_ENV ;;
          esac
        shell: bash
      - name: Print summary
        run: |
          echo "Running on: $PLATFORM"
          echo "OS: ${{ runner.os }}"
          echo "Matrix OS: ${{ matrix.os }}"
        shell: bash
```

Each matrix variant prints its own platform name.
</details>

### Exercise 5: Use org-level secrets

**Task:** Create a workflow that uses an organization-level secret (assume it is already set by your org admin). The workflow should check if the secret exists and print whether it is configured, without revealing the value.

**Hint:** Use an `if` condition or check the variable length.

<details>
<summary>Solution</summary>

```yaml
name: Org Secret Check
on: workflow_dispatch

jobs:
  check:
    runs-on: ubuntu-latest
    env:
      ORG_SECRET: ${{ secrets.ORG_DEPLOY_KEY }}
    steps:
      - name: Check if configured
        run: |
          if [ -n "$ORG_SECRET" ]; then
            echo "ORG_DEPLOY_KEY is configured (${#ORG_SECRET} characters)"
          else
            echo "ORG_DEPLOY_KEY is NOT configured"
          fi
```

If the org secret exists, the workflow reports its length. If not, it reports the secret is missing.
</details>

### Exercise 6: Environment-specific configuration

**Task:** Create a workflow with a `workflow_dispatch` input called `environment` (choices: `dev`, `staging`, `production`). Based on the input, set appropriate `DATABASE_URL`, `LOG_LEVEL`, and `API_ENDPOINT` values. Each environment should have different values.

**Hint:** Use a step that writes to `$GITHUB_ENV` with conditional logic based on the input.

<details>
<summary>Solution</summary>

```yaml
name: Environment Config
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: choice
        options:
          - dev
          - staging
          - production

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Set environment config
        run: |
          case "${{ github.event.inputs.environment }}" in
            dev)
              echo "DATABASE_URL=postgres://localhost:5432/dev" >> $GITHUB_ENV
              echo "LOG_LEVEL=debug" >> $GITHUB_ENV
              echo "API_ENDPOINT=https://dev-api.example.com" >> $GITHUB_ENV
              ;;
            staging)
              echo "DATABASE_URL=postgres://staging-db:5432/app" >> $GITHUB_ENV
              echo "LOG_LEVEL=info" >> $GITHUB_ENV
              echo "API_ENDPOINT=https://staging-api.example.com" >> $GITHUB_ENV
              ;;
            production)
              echo "DATABASE_URL=postgres://prod-db:5432/app" >> $GITHUB_ENV
              echo "LOG_LEVEL=warn" >> $GITHUB_ENV
              echo "API_ENDPOINT=https://api.example.com" >> $GITHUB_ENV
              ;;
          esac
      - name: Show config (without secrets)
        run: |
          echo "Environment: ${{ github.event.inputs.environment }}"
          echo "Database URL configured: $( [ -n '$DATABASE_URL' ] && echo yes || echo no )"
          echo "Log level: $LOG_LEVEL"
          echo "API endpoint: $API_ENDPOINT"
```

Each environment selection produces different configuration values.
</details>

## Tier 3 — Advanced

### Exercise 7: Design a secret rotation workflow

**Scenario:** Design a workflow that rotates a repository secret every 30 days. The workflow should:

1. Generate a new random value for `DEPLOY_KEY`
2. Update the repository secret using the GitHub API
3. Log the rotation event (without exposing the new value)
4. Notify the team via an issue comment
5. Only run on the first day of each month

**Constraints:** Use the `GITHUB_TOKEN` for API calls. The `GITHUB_TOKEN` must have `secrets: write` permission.

<details>
<summary>Solution</summary>

```yaml
name: Rotate Secrets
on:
  schedule:
    - cron: '0 0 1 * *'
  workflow_dispatch:

permissions:
  secrets: write
  issues: write

jobs:
  rotate:
    runs-on: ubuntu-latest
    steps:
      - name: Generate new secret
        id: generate
        run: |
          NEW_SECRET="deploy-key-$(openssl rand -hex 32)"
          echo "new_secret=$NEW_SECRET" >> $GITHUB_OUTPUT
          echo "Generated new secret (64 hex chars)"

      - name: Update the secret via API
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          echo "${{ steps.generate.outputs.new_secret }}" | gh secret set DEPLOY_KEY --repo ${{ github.repository }} --body -

      - name: Log rotation
        run: |
          echo "Secret DEPLOY_KEY rotated on $(date -u '+%Y-%m-%d %H:%M:%S UTC')"
          echo "Run ID: ${{ github.run_id }}"

      - name: Create notification issue
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          gh issue create \
            --title "Secret Rotation: DEPLOY_KEY" \
            --body "The DEPLOY_KEY secret was rotated on $(date -u '+%Y-%m-%d').\n\nPlease update any services that use this key.\n\nTriggered by workflow run #${{ github.run_id }}."
```

**Security note:** The new secret value passes through the workflow output. In a production system, consider generating and setting the secret in a single step using a script that reads from `/dev/urandom` directly, or using a dedicated secrets management service.
</details>

### Exercise 8: Debug a secret exposure risk

**Scenario:** Your teammate wrote this workflow. Find all the security issues:

```yaml
name: Risky Workflow
on: pull_request

jobs:
  debug:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Run tests with debug
        env:
          DEBUG: true
          DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
        run: |
          pytest -v --debug
          echo "DB password is: $DB_PASSWORD"
      - name: Print all env vars for debugging
        run: env
      - name: Store credentials in artifact
        run: |
          echo "password=$DB_PASSWORD" > credentials.txt
      - uses: actions/upload-artifact@v4
        with:
          name: debug-logs
          path: credentials.txt
```

<details>
<summary>Solution</summary>

There are at least 5 security issues:

1. **Echoing the secret in logs** (line 16): `echo "DB password is: $DB_PASSWORD"` — even though GitHub masks secrets, the mask is not guaranteed for partial matches or certain formats.

2. **Running on `pull_request` with secrets exposed**: PR workflows from forks run with limited secrets. But this workflow does not check for fork context, so the secret may be available to PRs from the same repository.

3. **Dumping all environment variables** (line 18): `env` prints everything, including `DB_PASSWORD`. Even if masked, this is a risk.

4. **Writing the secret to a file** (line 21): `echo "password=$DB_PASSWORD" > credentials.txt` stores the secret in plaintext.

5. **Uploading the secret as an artifact** (line 24): Anyone who can download the artifact (including outside collaborators on public repos) can read the credentials file.

Fixed version:

```yaml
name: Secure Workflow
on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Run tests
        env:
          DEBUG: true
          DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
        run: pytest -v --debug
      - name: Upload test logs (no secrets)
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: test-results.xml
```

Key fixes:
- Removed all `echo` statements that print secrets
- Removed `env` dump
- Removed credential file creation
- Artifact uploads only contain test results, not secrets
- Added explicit branch restriction to PR trigger
</details>

### Exercise 9: Design a multi-environment deployment config

**Scenario:** Design a workflow that deploys to three environments (dev, staging, production) with these requirements:

1. Each environment has its own set of secrets (`API_KEY`, `DB_PASSWORD`)
2. Dev deploys automatically on push to `develop`
3. Staging deploys on push to `main`
4. Production deploys manually via `workflow_dispatch` only
5. Production deploy requires a confirmation input
6. Each environment uses different configuration variable values (`MAX_CONNECTIONS`, `RATE_LIMIT`)
7. The workflow logs which environment was deployed and the configuration used — without exposing secrets

<details>
<summary>Solution</summary>

First, set up the secrets and variables:

```bash
# Dev secrets
gh secret set DEV_API_KEY --body "dev-key-123"
gh secret set DEV_DB_PASSWORD --body "dev-pass-456"

# Staging secrets
gh secret set STAGING_API_KEY --body "staging-key-789"
gh secret set STAGING_DB_PASSWORD --body "staging-pass-012"

# Production secrets
gh secret set PROD_API_KEY --body "prod-key-345"
gh secret set PROD_DB_PASSWORD --body "prod-pass-678"

# Configuration variables (not secrets — OK to show in logs)
gh variable set DEV_MAX_CONNECTIONS --body "10"
gh variable set DEV_RATE_LIMIT --body "100"
gh variable set STAGING_MAX_CONNECTIONS --body "50"
gh variable set STAGING_RATE_LIMIT --body "500"
gh variable set PROD_MAX_CONNECTIONS --body "200"
gh variable set PROD_RATE_LIMIT --body "1000"
```

Workflow file:

```yaml
name: Multi-Environment Deploy
on:
  push:
    branches:
      - develop
      - main
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: choice
        options:
          - production
      confirm:
        description: 'Type YES to confirm production deploy'
        required: true
        default: 'NO'

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ github.ref_name == 'develop' && 'dev' || github.ref_name == 'main' && 'staging' || github.event.inputs.environment }}
    env:
      # Resolve which environment is being targeted
      TARGET_ENV: ${{ github.ref_name == 'develop' && 'dev' || github.ref_name == 'main' && 'staging' || github.event.inputs.environment }}
    steps:
      - uses: actions/checkout@v4

      - name: Validate production confirmation
        if: env.TARGET_ENV == 'production' && github.event.inputs.confirm != 'YES'
        run: |
          echo "ERROR: Production deploy requires confirmation."
          echo "Set confirm input to 'YES' to proceed."
          exit 1

      - name: Set environment-specific secrets
        run: |
          case "${{ env.TARGET_ENV }}" in
            dev)
              echo "API_KEY=${{ secrets.DEV_API_KEY }}" >> $GITHUB_ENV
              echo "DB_PASSWORD=${{ secrets.DEV_DB_PASSWORD }}" >> $GITHUB_ENV
              echo "MAX_CONNECTIONS=${{ vars.DEV_MAX_CONNECTIONS }}" >> $GITHUB_ENV
              echo "RATE_LIMIT=${{ vars.DEV_RATE_LIMIT }}" >> $GITHUB_ENV
              ;;
            staging)
              echo "API_KEY=${{ secrets.STAGING_API_KEY }}" >> $GITHUB_ENV
              echo "DB_PASSWORD=${{ secrets.STAGING_DB_PASSWORD }}" >> $GITHUB_ENV
              echo "MAX_CONNECTIONS=${{ vars.STAGING_MAX_CONNECTIONS }}" >> $GITHUB_ENV
              echo "RATE_LIMIT=${{ vars.STAGING_RATE_LIMIT }}" >> $GITHUB_ENV
              ;;
            production)
              echo "API_KEY=${{ secrets.PROD_API_KEY }}" >> $GITHUB_ENV
              echo "DB_PASSWORD=${{ secrets.PROD_DB_PASSWORD }}" >> $GITHUB_ENV
              echo "MAX_CONNECTIONS=${{ vars.PROD_MAX_CONNECTIONS }}" >> $GITHUB_ENV
              echo "RATE_LIMIT=${{ vars.PROD_RATE_LIMIT }}" >> $GITHUB_ENV
              ;;
          esac

      - name: Deploy
        run: |
          echo "Deploying to $TARGET_ENV"
          echo "Max connections: $MAX_CONNECTIONS"
          echo "Rate limit: $RATE_LIMIT"
          echo "API key configured: $( [ -n '$API_KEY' ] && echo yes || echo no )"
          echo "DB password configured: $( [ -n '$DB_PASSWORD' ] && echo yes || echo no )"
          echo "Deployment to $TARGET_ENV complete!"

      - name: Post-deploy smoke test
        run: |
          echo "Running smoke tests against $TARGET_ENV..."
          echo "All smoke tests passed!"
```

This workflow makes each environment's secrets and configuration visible only during that specific deployment, and it enforces a confirmation gate for production.
</details>
