You commit a workflow that echoes `${{ secrets.API_KEY }}` for debugging. Five minutes later, you realize the secret value is visible in the Actions log for everyone to see. Secrets are masked — but only when used correctly. Meanwhile, your test script needs `DATABASE_URL` to point to a test database in CI but a production database in deployment, and you need a way to change this value without editing the workflow file every time.

## The Concept

Environment variables in GitHub Actions are key-value pairs available to step processes at runtime. They can be set at three levels (workflow, job, step) with inner levels overriding outer ones. Secrets are encrypted environment variables stored in the repository or organization settings — once set, they can never be read through the UI, and GitHub automatically masks their values in log output.

Think of environment variables as a stack of sticky notes on a whiteboard. The workflow-level notes are at the bottom (visible to everyone), job-level notes are stuck on top (visible to that job's steps), and step-level notes are on top of that (visible only during that step). Secrets are like notes written in invisible ink — you can use them, but if anyone tries to read them aloud, the words are automatically blurred.

## Key Properties and Commands

| Mechanism | Scope | How to Set | How to Read |
|-----------|-------|------------|-------------|
| `env` (workflow) | All jobs and steps | `env: KEY: value` at top level | `$KEY` or `${{ env.KEY }}` |
| `env` (job) | All steps in that job | `env: KEY: value` inside job | `$KEY` or `${{ env.KEY }}` |
| `env` (step) | Single step only | `env: KEY: value` inside step | `$KEY` or `${{ env.KEY }}` |
| `$GITHUB_ENV` | Subsequent steps | `echo "KEY=value" >> $GITHUB_ENV` | `$KEY` in later steps |
| `$GITHUB_OUTPUT` | Dependent jobs | `echo "key=value" >> $GITHUB_OUTPUT` | `${{ steps.id.outputs.key }}` |
| `secrets` (repo) | All workflows | Set in repo Settings > Secrets | `${{ secrets.NAME }}` |
| `secrets` (org) | All repos in org | Set in org Settings > Secrets | `${{ secrets.NAME }}` |
| `vars` (repo) | All workflows | Set in repo Settings > Variables | `${{ vars.NAME }}` |
| `github.token` | Current run | Auto-generated `GITHUB_TOKEN` | `${{ github.token }}` |
| Default env vars | Current run | Set by GitHub (see table below) | `${{ env.VAR }}` or `$VAR` |

### Default Environment Variables

| Variable | Description | Example |
|----------|-------------|---------|
| `GITHUB_WORKFLOW` | Workflow name | `CI Pipeline` |
| `GITHUB_RUN_ID` | Unique run ID | `1234567890` |
| `GITHUB_RUN_NUMBER` | Run number (increments per run) | `42` |
| `GITHUB_ACTION` | Current action name | `__run` |
| `GITHUB_ACTOR` | User who triggered the run | `octocat` |
| `GITHUB_REPOSITORY` | Owner and repo name | `octocat/hello-world` |
| `GITHUB_REF` | Branch or tag ref | `refs/heads/main` |
| `GITHUB_REF_NAME` | Short branch or tag name | `main` |
| `GITHUB_SHA` | Commit SHA | `abc123def456...` |
| `GITHUB_WORKSPACE` | Working directory on runner | `/home/runner/work/repo/repo` |
| `RUNNER_OS` | Runner OS name | `Linux`, `Windows`, `macOS` |
| `RUNNER_TEMP` | Temp directory path | `/home/runner/work/_temp` |

## How It Actually Works — Step by Step

### Step 1: The workflow YAML is parsed for env declarations

When a workflow run starts, GitHub collects all `env:` declarations from the workflow, job, and step levels. The effective environment for each step is computed by merging: workflow env vars + job env vars + step env vars. Lower levels override higher levels for duplicate keys.

### Step 2: Default environment variables are injected

GitHub injects its own default environment variables (`GITHUB_*`, `RUNNER_*`) into every step. These cannot be overridden by workflow-level `env:` declarations.

### Step 3: Secrets are decrypted at runtime

When a step references `${{ secrets.API_KEY }}`, GitHub's runner decrypts the secret from its encrypted storage and injects it into the step's environment. The plaintext value is never stored in the workflow file or the GitHub database — only the encrypted ciphertext is stored.

### Step 4: The `$GITHUB_ENV` file creates dynamic env vars

Within a step, you can write `KEY=value` to the `$GITHUB_ENV` file. After the step completes, the runner reads this file and makes those variables available to ALL subsequent steps in the same job.

```yaml
steps:
  - run: echo "VERSION=1.2.3" >> $GITHUB_ENV
  - run: echo "Version is $VERSION"  # Prints: Version is 1.2.3
```

### Step 5: `$GITHUB_OUTPUT` passes values between jobs

For cross-job communication, use `$GITHUB_OUTPUT`:

```yaml
steps:
  - id: compute
    run: echo "version=1.2.3" >> $GITHUB_OUTPUT
```

Then reference it: `${{ steps.compute.outputs.version }}` in the same job, or `${{ needs.other-job.outputs.version }}` across jobs.

### Step 6: Log masking happens in the runner

When a step that uses a secret produces output, the runner checks every printed line against the values of all secrets used in that step. If a match is found, the value is replaced with `***`. This masking is automatic but has limitations: secrets shorter than 4 characters are not masked, and secrets that appear in the middle of a long line may be partially masked.

## When to Use Each Mechanism / When Not To

### Use `env:` when:
- The value is non-sensitive and shared across multiple steps or jobs
- You want to set defaults that can be overridden at lower levels

### Use `$GITHUB_ENV` when:
- You need to compute a value in one step and use it in later steps
- The value depends on the runner environment (e.g., OS detection, dynamic paths)

### Use `secrets` when:
- The value is sensitive (API keys, passwords, tokens, certificates)
- Different environments (dev/staging/prod) need different credentials

### Do NOT use secrets when:
- The value is non-sensitive — use `vars` (plaintext configuration variables) instead
- The value is short (under 4 characters) — masking may not work reliably

### Use `vars` (configuration variables) when:
- You need non-sensitive configuration that changes per environment
- You want to avoid hardcoding values in workflow files

### Use `GITHUB_TOKEN` when:
- You need to authenticate API calls within the same repository
- You want the automatically-scoped, short-lived token for standard operations
