Your release process: bump the version number in `package.json`, write release notes, create a Git tag, push to GitHub, build the artifact, upload to the releases page, notify the team on Slack. Each step is a manual command, a copy-paste, or a button click. One Friday, you forget to tag the commit before pushing — now the release artifact points at the wrong commit, and your version tracking is off by one. Release automation turns this 20-minute sequence of error-prone steps into a single `git push --tags` that does everything automatically.

## The Concept

Release automation is a CI/CD workflow that creates software releases — version bumping, changelog generation, Git tagging, artifact building, and release publishing — triggered by a single event (like pushing a tag or clicking a workflow_dispatch button). It replaces a page of runbook instructions with a repeatable, auditable pipeline.

Think of it like a vending machine vs. a convenience store: you press one button and the machine dispenses exactly what you ordered (release), rather than walking every aisle, checking prices, standing in line, and bagging your items yourself. The machine never forgets a step.

## Core Release Automation YAML Keys

| Key | Purpose | Required | Example |
|-----|---------|----------|---------|
| `workflow_dispatch` | Manual trigger via GitHub UI button | No | `on: workflow_dispatch:` |
| `release` | Trigger on GitHub Release events | No | `on: release: [published]` |
| `github.ref` | The tag or branch ref that triggered the run | Yes | `if: startsWith(github.ref, 'refs/tags/v')` |
| `github.ref_name` | Short name of the tag or branch | No | `run: semver=${{ github.ref_name }}` |
| `github.sha` | Full commit SHA for the release | No | `run: echo ${{ github.sha }}` |
| `softprops/action-gh-release` | Create GitHub Release with assets | No | `uses: softprops/action-gh-release@v2` |
| `ncipollo/release-action` | Alternative release creation action | No | `uses: ncipollo/release-action@v1` |
| `goreleaser/goreleaser-action` | Go-specific release automation | No | `uses: goreleaser/goreleaser-action@v5` |

## How Release Automation Actually Works — Step by Step

### Step 1: Trigger the Release

A release workflow can be triggered in several ways:

**Tag push (most common):**
```yaml
on:
  push:
    tags:
      - 'v*'
```
Any push matching `v*` (e.g., `v2.1.0`, `v3.0.0-rc.1`) triggers the workflow.

**GitHub Release UI:**
```yaml
on:
  release:
    types: [published]
```
A user drafts a release in the GitHub UI and publishes it. The workflow then builds and attaches artifacts.

**Manual dispatch:**
```yaml
on:
  workflow_dispatch:
    inputs:
      version:
        description: 'Semver version (e.g., 2.1.0)'
        required: true
```
A user clicks "Run workflow" in the Actions tab and types a version number.

The tag-push pattern is most reliable because the tag itself records the release point in Git history — you can always look at the `v2.1.0` tag and know exactly what was released.

### Step 2: Extract Version from the Tag

The release version is embedded in the Git tag. Use `github.ref` or `github.ref_name` to extract it:

```yaml
- name: Extract version
  run: |
    echo "VERSION=${{ github.ref_name }}" >> $GITHUB_ENV
```

For a tag like `v2.1.0`, `github.ref_name` gives you `v2.1.0`. You can strip the `v` prefix:

```yaml
- name: Parse semver
  run: |
    echo "VERSION=${GITHUB_REF_NAME#v}" >> $GITHUB_ENV
```

This produces `2.1.0` — the version you can use in build outputs, Docker tags, and package registry publishing.

### Step 3: Build the Release Artifacts

Build the binaries, packages, or Docker images that will ship with the release:

```yaml
- name: Build
  run: |
    npm run build
    zip -r app-${{ env.VERSION }}.zip dist/

- name: Build Docker image
  run: |
    docker build -t myapp:${{ env.VERSION }} .
```

Use the extracted `VERSION` to tag build outputs so every artifact is clearly associated with the release.

### Step 4: Create the GitHub Release

Use a community action like `softprops/action-gh-release` to create the release on GitHub, attach artifacts, and auto-generate release notes:

```yaml
- name: Create Release
  uses: softprops/action-gh-release@v2
  with:
    name: Release ${{ env.VERSION }}
    body_path: CHANGELOG.md
    files: |
      app-${{ env.VERSION }}.zip
      dist/*
    generate_release_notes: true
    draft: false
    prerelease: ${{ contains(github.ref_name, 'rc') || contains(github.ref_name, 'beta') }}
```

Key parameters:
- `files:` — glob of artifacts to upload (max 10 GB total)
- `generate_release_notes: true` — auto-generates notes from merged PRs since the last release (GitHub reads PR titles and categorizes by labels)
- `draft: false` — publish immediately (set `true` for a review-before-publishing stage)
- `prerelease: true` — marks as pre-release in the GitHub UI (useful for release candidates)

### Step 5: Publish to Package Registry

If your release includes a package (npm, PyPI, Docker Hub, NuGet), the workflow publishes it:

```yaml
- name: Publish to npm
  run: npm publish
  env:
    NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

- name: Publish Docker image
  run: |
    docker tag myapp:${{ env.VERSION }} ghcr.io/myorg/myapp:${{ env.VERSION }}
    docker push ghcr.io/myorg/myapp:${{ env.VERSION }}
```

This step often uses the `NODE_AUTH_TOKEN`, `NPM_TOKEN`, or `GITHUB_TOKEN` secret — configure the appropriate token in your repository secrets.

### Step 6: Notify the Team

The final step notifies stakeholders that a release is available:

```yaml
- name: Notify Slack
  uses: slackapi/slack-github-action@v1
  with:
    payload: |
      {
        "text": "Release ${{ env.VERSION }} published!",
        "blocks": [
          { "type": "section", "text": { "type": "mrkdwn", "text": ":rocket: Release *${{ env.VERSION }}* is live!" } }
        ]
      }
  env:
    SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```

Notification channels can include Slack, email, Teams, or an entry in a changelog wiki.

## When to Use Release Automation / When Not To

**Use release automation when:**

- You release at least once per week — automation saves time and eliminates consistency errors
- Your release includes multiple artifacts — a tarball, a Docker image, and package registry entries should all be produced by one trigger
- You have more than one maintainer who publishes releases — automation ensures releases are consistent regardless of who runs them
- You need an audit trail for releases — the workflow run log shows exactly when, by whom, and from what commit each release was created

**Consider a simpler approach when:**

- You release less than once per month and produce a single binary — a simple `gh release create` command may suffice
- Your release process requires manual QA sign-off anyway — automation can build and stage, but the publish step should stay manual if every release needs human verification
- You are in early prototyping with no consumers — just use `git tag` and a note for yourself

The rule: automate releases when the cost of a mistake (wrong version, missing artifact, wrong tag) exceeds the cost of building the automation. For most projects with external consumers, that tipping point is reached at the second release.

## Full Release Automation Example

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      packages: write

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # needed for release notes generation

      - name: Extract version
        run: echo "VERSION=${GITHUB_REF_NAME#v}" >> $GITHUB_ENV

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci

      - run: npm run build

      - name: Package artifact
        run: zip -r app-${{ env.VERSION }}.zip dist/

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          name: Release ${{ env.VERSION }}
          files: app-${{ env.VERSION }}.zip
          generate_release_notes: true
          prerelease: ${{ contains(github.ref_name, 'rc') }}

      - name: Publish to npm
        run: npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Push a tag like `git tag v2.1.0 && git push origin v2.1.0` and this workflow builds, packages, publishes a GitHub Release with auto-generated notes, and publishes to npm — all from a single command.
