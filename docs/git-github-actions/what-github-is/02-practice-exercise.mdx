## Setup

Before you start this exercise, make sure you have the following ready:

1. A terminal or command prompt open on your computer
2. Git installed and configured on your machine
3. A web browser pointed to github.com
4. A local Git repository with at least two commits. If you do not have one, run these commands to create a test repository:

```bash
mkdir github-test
cd github-test
git init
echo "# My Test Project" > README.md
git add README.md
git commit -m "Initial commit: add README"
echo "print('hello github')" > hello.py
git add hello.py
git commit -m "Add hello script"
```

If Git is not configured with your identity yet, run these commands first:

```bash
git config --global user.name "Your Name"
git config --global user.email "your-email@example.com"
```

Your local repository is now ready with two commits. You can verify with:

```bash
git log --oneline
```

The output shows two lines, each with a commit hash and message.

## The Walkthrough

Follow these eight steps to connect your local repository to GitHub, push your code, pull changes, and clone a fresh copy.

### Step 1: Create a GitHub Account

Navigate to github.com in your browser. Click the Sign Up button in the top right corner. Enter your email address, create a password (at least 15 characters or 8 characters with a number and lowercase letter), and choose a username. GitHub sends a verification code to your email. Check your inbox, enter the six-digit code, and click Create Account.

After verification, GitHub asks about your experience level and goals. You can fill these in or click Skip Personalization at the bottom of the page. You land on the GitHub dashboard -- this is your home page when logged in.

### Step 2: Create a New Repository on GitHub

On the GitHub dashboard, click the plus icon in the top right corner (next to your profile picture). Select New Repository from the dropdown menu.

Fill in the form:

- **Repository name**: `github-test` (the same name as your local folder)
- **Description**: `My practice repository for learning GitHub` (optional)
- **Visibility**: Choose Public so you can see it from any browser. Private also works if you prefer.
- **Initialize this repository with**: Leave all checkboxes unchecked. You want an empty repository because you already have a local repository with content.

Click the green Create Repository button at the bottom.

GitHub shows you a page with instructions. Look for the section titled `...or push an existing repository from the command line`. It shows two commands:

```bash
git remote add origin https://github.com/your-username/github-test.git
git push -u origin main
```

Keep this page open. You need the remote URL for the next step.

### Step 3: Link Your Local Repository to the Remote

Copy the first command from the GitHub page and run it in your terminal. Make sure you are in the `github-test` directory.

```bash
git remote add origin https://github.com/your-username/github-test.git
```

Replace `your-username` with your actual GitHub username. This command tells Git that a remote repository exists at that URL and you want to call it `origin`.

Verify the remote was added correctly:

```bash
git remote -v
```

Expected output:

```
origin  https://github.com/your-username/github-test.git (fetch)
origin  https://github.com/your-username/github-test.git (push)
```

Both fetch and push URLs should point to your repository. If you see nothing, the remote was not added -- run the `git remote add` command again and check for typos.

### Step 4: Push Your Code to GitHub

Run the second command from the GitHub page:

```bash
git push -u origin main
```

Git asks for your GitHub username and password. Use your GitHub username and a personal access token instead of your account password. If you do not have a personal access token, create one:

1. On GitHub, click your profile picture → Settings
2. Scroll to Developer settings in the left sidebar
3. Click Personal access tokens → Tokens (classic)
4. Click Generate new token → Generate new token (classic)
5. Give it a name like `git-push-test`
6. Check the `repo` scope box
7. Click Generate token at the bottom
8. Copy the token (shown once) and use it as your password

After authentication, Git uploads your commits:

```

Counting objects: 100% (4/4), done.

Compressing objects: 100% (3/3), done.
Writing objects: 100% (4/4), 421 bytes | 421.00 KiB/s, done.

To https://github.com/your-username/github-test.git
 * [new branch]      main -> main
Branch 'main' set up to track remote branch 'main' from 'origin'.
```

### Step 5: Verify on GitHub Web UI

Go back to your browser and refresh the GitHub repository page. You should now see two files listed: `README.md` and `hello.py`. Below the file list, the commit history appears with both commits visible.

Click the `README.md` file. GitHub displays the rendered Markdown content: "My Test Project" displayed as a heading. Click the `hello.py` file to see the Python code highlighted with syntax coloring.

Click the commits link (above the file list, shows something like "2 commits"). You see the full log of commits with your commit messages, timestamps, and author name. Each commit is clickable and shows the diff of what changed.

### Step 6: Make a Local Change

Make a change to your local repository. Edit the `hello.py` file and add a second function:

```bash
echo "print('hello again')" >> hello.py
```

Or open the file in a text editor and add the second print statement. Then stage and commit the change:

```bash
git add hello.py
git commit -m "Add second print statement"
```

Your local repository now has three commits. The remote repository on GitHub still has only two. The local and remote histories have diverged.

```bash
git log --oneline
```

Expected output (your commit hash will differ):

```
a1b2c3d Add second print statement
d4e5f6g Add hello script
h7i8j9k Initial commit: add README
```

### Step 7: Push the Local Change to GitHub

Push the new commit to the remote repository. Since you already set the upstream tracking reference in Step 3, you can run the shortened form:

```bash
git push
```

Git detects that your local `main` branch is ahead of `origin/main` by one commit and sends the new commit to GitHub.

Return to your browser and refresh the repository page. The file count is still two, but the commit count has increased to three. Click on `hello.py` to see both print statements in the file.

### Step 8: Clone Into a Fresh Directory

Cloning simulates what happens when you set up a new computer or when a teammate downloads the project for the first time. Navigate to a different directory so you are outside the `github-test` folder:

```bash
cd ..
```

Now clone the repository into a new folder:

```bash
git clone https://github.com/your-username/github-test.git github-test-clone
```

The clone output:

```

Navigate into the cloned directory and verify the contents:

```

You should see both files and all three commits. The clone created a full copy of the repository with the remote `origin` already pointing to GitHub. You can run `git remote -v` to confirm.

## Common Errors

Here are four errors you are likely to encounter and exactly how to fix them.

### Error 1: remote origin already exists

```
fatal: remote origin already exists.
```

You see this when you try to add a remote named `origin` but one already exists. This happens if you already ran `git remote add origin` earlier in the same repository.

**Fix**: Either use the existing remote or remove it and add a new one.

To use the existing remote:

```bash
git remote -v
```

Check if the URL is correct. If it points to the right repository, proceed with `git push`. There is no need to add a new remote.

To replace the remote with a different URL:

```bash
git remote set-url origin https://github.com/your-username/correct-repo.git
```

To remove and re-add:

```bash
git remote remove origin
git remote add origin https://github.com/your-username/correct-repo.git
```

### Error 2: failed to push some refs

```
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'https://github.com/your-username/repo.git'
hint: Updates were rejected because the remote contains work that you do not
hint: have locally. This is usually caused by another repository pushing to
hint: the same ref. You may want to first integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
```

This happens when the remote branch has commits you do not have locally. Git refuses to overwrite work that exists on the remote but not in your local repository.

**Fix**: Pull the remote changes first, then push again.

```bash
git pull origin main
git push
```

If the pull creates a merge conflict, resolve it by editing the conflicted files, staging them with `git add`, and committing the merge. Then push again.

Only use `git push --force` if you are certain you want to overwrite the remote history and no one else depends on it. Force pushing can delete someone else's work.

### Error 3: could not read from remote repository

```
fatal: could not read from remote repository.

Please make sure you have the correct access rights
and the repository exists.
```

This error means Git cannot connect to the remote URL. Several things can cause it: the repository URL is wrong, the repository does not exist, you do not have permission to access it, or your network is blocking the connection.

**Fix**: Check each of these in order.

1. Verify the remote URL is correct:
```bash
git remote -v
```
The URL should match the one on your GitHub repository page. Copy it from the green Code button on GitHub.

2. Verify the repository exists by opening its URL in a browser:
```
https://github.com/your-username/your-repo-name
```
If the page shows a 404 error, the repository does not exist or you are logged into a different account.

3. If you use a personal access token for authentication, make sure the token is still valid. Tokens can expire or be revoked. Generate a new token in GitHub Settings (Settings → Developer settings → Personal access tokens).

4. Check your internet connection:
```bash
ping github.com
```
If ping fails, your network may be blocking GitHub. Try again from a different network.

### Error 4: repository not found

```
remote: Repository not found.
fatal: repository 'https://github.com/your-username/repo.git/' not found
```

This error appears when Git can reach GitHub but the repository at that URL does not exist or you do not have access to it.

**Fix**: Check the repository name and your permissions.

1. Confirm the repository name is spelled correctly. Repository names are case-sensitive. `My-Repo` and `my-repo` are different repositories.

2. If the repository is private, make sure you are authenticated with an account that has access. Use `gh auth status` to check which account is logged in with the GitHub CLI, or check your credential manager for stored credentials.

3. If the repository was deleted, create a new one on GitHub and update the remote URL:
```bash
git remote set-url origin https://github.com/your-username/new-repo-name.git
git push -u origin main
```

## What You Just Learned

- GitHub is a cloud platform that hosts Git repositories, providing backup and sharing capabilities beyond what local Git offers. Remote repositories store the same Git objects (commits, trees, blobs) as local ones but are bare -- they have no working tree.
- You can link any local Git repository to a GitHub remote using `git remote add origin <url>`. This creates a named reference that other commands use to communicate with the remote server.
- The `git push` command uploads your local commits to the remote repository, transferring the packfile of Git objects over HTTPS or SSH. The `-u` flag sets an upstream tracking reference so future pushes default to the same remote branch.
- The `git pull` command downloads remote commits and merges them into your local branch. It is equivalent to `git fetch` followed by `git merge`. If the remote has commits your local repository does not, the push is rejected until you pull.
- The `git clone` command creates a full copy of a remote repository on your machine, downloading all commits, branches, and tags. After cloning, the remote `origin` is automatically configured, and you can start working immediately.
- Four common errors have specific causes and fixes: an existing remote, a rejected push due to divergent history, a connection failure, and a missing or inaccessible repository. Each error message tells you exactly what went wrong.
