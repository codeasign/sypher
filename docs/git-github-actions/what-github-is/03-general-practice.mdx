These exercises build your muscle memory with GitHub remote operations. They are grouped into three tiers. Start at Tier 1 if this is your first time working with remote repositories. Move to Tier 2 once the basic commands feel automatic. Try Tier 3 when you are comfortable with the standard workflow and want to handle edge cases.

## Tier 1 -- Beginner

Each exercise in this tier is a single Git command. Run it and observe the output. The goal is to see each command succeed and understand what it tells you.

### Exercise 1: git remote add

Create a new local repository and add a remote to it.

```bash
mkdir tier1-exercise1
cd tier1-exercise1
git init
echo "remote practice" > README.md
git add README.md
git commit -m "Initial commit"
git remote add origin https://github.com/your-username/tier1-practice.git
```

Run `git remote -v` and verify the output shows both fetch and push URLs. The remote is now configured even though the repository on GitHub does not exist yet. The `git remote add` command only stores the URL locally. It does not verify the URL is valid or that the repository exists.

```bash
git remote -v
```

Expected output:

```
origin  https://github.com/your-username/tier1-practice.git (fetch)
origin  https://github.com/your-username/tier1-practice.git (push)
```

### Exercise 2: git clone

Clone any public repository from GitHub. Use a well-known project so you are sure it exists.

```bash
git clone https://github.com/octocat/Hello-World.git
```

This creates a directory called `Hello-World`. Navigate into it and inspect what you downloaded:

```bash
cd Hello-World
git log --oneline
git remote -v
```

The `git remote -v` output shows `origin` pointing back to the GitHub URL. When you clone, Git automatically sets up the remote for you. The `git log` shows the repository's commit history -- every commit that existed on the remote is now on your machine.

Remove the cloned directory when you are done:

```bash
cd ..
rm -rf Hello-World
```

### Exercise 3: git push to a real remote

For this exercise, create a repository on the GitHub website first. Name it `tier1-push-test`. Do not initialize it with any files. Then link and push from your local machine.

```bash
mkdir tier1-push
cd tier1-push
git init
echo "# Push Test" > README.md
echo "console.log('pushed')" > app.js
git add .
git commit -m "Initial commit with README and app"
git remote add origin https://github.com/your-username/tier1-push-test.git
git push -u origin main
```

Refresh the GitHub page for `tier1-push-test`. You should see `README.md` and `app.js` listed with your commit message. This confirms the push transferred your Git objects to the remote server.

### Exercise 4: git fetch

Clone a repository and then fetch from the remote to see how `git fetch` works without merging.

```bash
git clone https://github.com/your-username/tier1-push-test.git tier1-fetch-test
cd tier1-fetch-test
git fetch origin
```

Run `git log --oneline origin/main` to see the remote tracking branch. The `origin/main` reference shows the state of the remote `main` branch as of your last fetch. Since you just cloned, it is identical to your local `main` branch.

Make a change on GitHub directly: open the `README.md` file on GitHub, click the pencil icon to edit, add a new line, and commit the change through the web UI. Then back in your terminal, run:

```bash
git fetch origin
git log --oneline origin/main
```

The `origin/main` reference now includes the commit you made on GitHub. Your local `main` branch does not have it yet because `git fetch` does not merge. Run `git status` to confirm your local branch is behind the remote:

```bash
git status
```

You see a message like:

```
Your branch is behind 'origin/main' by 1 commit, and can be fast-forwarded.
```

The `git fetch` command downloaded the new commit objects but left your working tree and branch unchanged. This is the key difference from `git pull`.

## Tier 2 -- Intermediate

Each exercise in this tier involves multiple steps. Complete all steps in order without skipping.

### Exercise 1: Create repository, push code, verify on web, clone elsewhere

This exercise simulates the full workflow of publishing a new project.

1. On GitHub, create a new repository called `intermediate-exercise-1` with no initial files.
2. On your local machine, create a directory and initialize a Git repository:
```bash
mkdir intermediate-1
cd intermediate-1
git init
```
3. Create three files: `index.html`, `style.css`, and `config.yaml`. Each file should have at least one line of content.
4. Stage all files and make a commit:
```bash
git add .
git commit -m "Add initial project files"
```
5. Add the remote and push:
```bash
git remote add origin https://github.com/your-username/intermediate-exercise-1.git
git push -u origin main
```
6. Open the repository page on GitHub in your browser. Verify all three files are listed.
7. Navigate to a different directory on your machine and clone the repository:
```bash
cd ..
git clone https://github.com/your-username/intermediate-exercise-1.git intermediate-1-clone
```
8. Run `ls intermediate-1-clone` to confirm all three files are present.

### Exercise 2: Push, make remote change on GitHub, pull it back

This exercise simulates collaboration where someone else (or you from another computer) makes changes on GitHub and you pull them into your local repository.

1. Use the `tier1-push-test` repository from the beginner tier (or create a new one).
2. Push your local code to GitHub if you have not already:
```bash
git push -u origin main
```
3. On GitHub, open the repository and create a new file called `collab.txt`. Add the text `Added from GitHub web UI` as its content. Commit the new file directly on GitHub by clicking Commit new file at the bottom.
4. Back in your terminal, pull the change:
```bash
git pull origin main
```
5. Verify the file exists locally:
```bash
ls collab.txt
cat collab.txt
```

The output shows the content you typed on GitHub. The `git pull` command downloaded the new commit (the one you created on GitHub) and merged it into your local branch. Your local repository is now synchronized with the remote.

### Exercise 3: Clone a repo, make changes, push back

This exercise simulates contributing to a project you have just cloned. It demonstrates that a cloned repository is fully functional -- you can push changes back to the remote.

1. Clone your `tier1-push-test` repository into a fresh directory:
```bash
cd ..
git clone https://github.com/your-username/tier1-push-test.git push-back-test
cd push-back-test
```
2. Make a change to an existing file:
```bash
echo "// Added from cloned copy" >> app.js
```
3. Stage, commit, and push:
```bash
git add app.js
git commit -m "Update app.js from cloned directory"
git push
```
4. Refresh the GitHub page and verify the change appears in `app.js`.

Notice that `git push` worked without specifying the remote name or branch. The clone command set up the remote `origin` and the upstream tracking reference automatically.

### Exercise 4: Multiple remotes

This exercise shows how to add a second remote to a single local repository. This is useful when you fork a project on GitHub and want to keep your fork synchronized with the original project.

1. Create two repositories on GitHub: `multi-remote-a` and `multi-remote-b`. Leave both empty.
2. Create a local repository and add the first remote:
```bash
mkdir multi-remote
cd multi-remote
git init
echo "# Multi Remote" > README.md
git add README.md
git commit -m "Initial commit"
git remote add origin https://github.com/your-username/multi-remote-a.git
```
3. Add the second remote with a different name:
```bash
git remote add backup https://github.com/your-username/multi-remote-b.git
```
4. Push to both remotes:
```bash
git push -u origin main
git push -u backup main
```
5. Verify both remotes are configured:
```bash
git remote -v
```

Expected output:

```
backup  https://github.com/your-username/multi-remote-b.git (fetch)
backup  https://github.com/your-username/multi-remote-b.git (push)
origin  https://github.com/your-username/multi-remote-a.git (fetch)
origin  https://github.com/your-username/multi-remote-a.git (push)
```

Both GitHub pages now show the same `README.md`. The local repository can push to either remote using the name you assigned. The diagram below shows this relationship.

## Tier 3 -- Advanced

These exercises involve troubleshooting, infrastructure setup, and non-standard configurations. They require a solid understanding of how Git and GitHub interact.

### Exercise 1: Troubleshoot a rejected push

This exercise creates a deliberate push rejection so you can practice the recovery workflow.

1. Create a new repository on GitHub called `conflict-test`. Initialize it with a README file (check the box on GitHub).
2. On your local machine, create a directory and initialize a Git repository with a README that has different content:
```bash
mkdir conflict-test
cd conflict-test
git init
echo "Local version" > README.md
git add README.md
git commit -m "Local README"
```
3. Add the remote and try to push:
```bash
git remote add origin https://github.com/your-username/conflict-test.git
git push -u origin main
```

You see a rejection because the remote (which was initialized with a README) has a commit your local repository does not have.

4. Pull the remote changes and merge:
```bash
git pull origin main
```

Git downloads the remote commit and attempts to merge. The `README.md` file now has a merge conflict because both the local and remote versions modified it.

5. Open `README.md` in a text editor. You see conflict markers:
```
<<<<<<< HEAD
Local version
=======
# conflict-test
>>>>>>> <commit-hash>
```

6. Resolve the conflict by editing the file to keep both lines or a combined version:
```
Local version
# conflict-test
```

7. Stage the resolved file and complete the merge:
```bash
git add README.md
git commit -m "Merge remote README with local README"
```

8. Push the merged result:
```bash
git push
```

The push succeeds because you now have all the remote commits plus your local changes.

### Exercise 2: Manage a second remote for upstream tracking

This exercise is based on the fork workflow used in open-source contributions. You clone a repository, add the original project as a second remote (called `upstream`), and synchronize your fork with upstream changes.

1. Fork a small public repository on GitHub. Navigate to `https://github.com/octocat/Hello-World` and click the Fork button in the top right corner. This creates a copy of the repository under your GitHub account.
2. Clone your fork:
```bash
git clone https://github.com/your-username/Hello-World.git
cd Hello-World
```
3. Add the original repository as a remote named `upstream`:
```bash
git remote add upstream https://github.com/octocat/Hello-World.git
```
4. Verify both remotes:
```bash
git remote -v
```

Expected output:

```
origin    https://github.com/your-username/Hello-World.git (fetch)
origin    https://github.com/your-username/Hello-World.git (push)
upstream  https://github.com/octocat/Hello-World.git (fetch)
upstream  https://github.com/octocat/Hello-World.git (push)
```

5. Fetch changes from the upstream repository:
```bash
git fetch upstream
```

This downloads any new commits from the original project without affecting your local branches. To merge upstream changes into your local `main` branch:

```bash
git checkout main
git merge upstream/main
git push origin main
```

This workflow keeps your fork synchronized with the original project. The diagram below shows the three repositories involved.

### Exercise 3: Set up SSH key authentication

This exercise moves you from HTTPS authentication (where you enter a username and token each time) to SSH key authentication (where your machine is recognized automatically).

1. Generate an SSH key pair on your machine:
```bash
ssh-keygen -t ed25519 -C "your-email@example.com"
```

Press Enter to accept the default file location (`~/.ssh/id_ed25519`). Optionally enter a passphrase for extra security (you can press Enter twice for no passphrase).

2. Start the SSH agent and add your key:
```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
```

3. Copy your public key to the clipboard:
```bash
cat ~/.ssh/id_ed25519.pub
```

The output is a single line starting with `ssh-ed25519`. Select and copy the entire line.

4. On GitHub, go to Settings (click your profile picture) → SSH and GPG keys → New SSH Key. Paste your public key into the Key field, give it a title (like "My Laptop"), and click Add SSH Key.

5. Test the SSH connection:
```bash
ssh -T git@github.com
```

You see a success message like:

```
Hi your-username! You've successfully authenticated, but GitHub does not provide shell access.
```

6. Now create a repository on GitHub (empty, no initial files) and push to it using the SSH URL:
```bash
mkdir ssh-test
cd ssh-test
git init
echo "SSH test" > README.md
git add README.md
git commit -m "Initial commit"
git remote add origin git@github.com:your-username/ssh-test.git
git push -u origin main
```

Notice the URL format: `git@github.com:username/repo.git` instead of `https://github.com/username/repo.git`. The SSH protocol uses public-key cryptography to authenticate you without entering a password or token.

7. Verify SSH authentication works for clone as well:
```bash
cd ..
git clone git@github.com:your-username/ssh-test.git ssh-test-clone
cd ssh-test-clone
git log --oneline
```

The clone does not prompt for credentials. Your SSH key authenticates the connection automatically.

### Exercise 4: Diagnose a broken remote configuration

This exercise simulates a common real-world problem: the remote URL is wrong and all remote commands fail. The goal is to identify and fix the issue.

1. Create a new local repository with an incorrect remote URL:
```bash
mkdir broken-remote
cd broken-remote
git init
echo "Fix me" > fix.txt
git add fix.txt
git commit -m "First commit"
git remote add origin https://github.com/your-username/non-existent-repo.git
```
2. Attempt to push:
```bash
git push -u origin main
```

You see an error: `remote: Repository not found.` or `fatal: repository '...' not found`.

3. Diagnose the problem by checking the remote URL:
```bash
git remote -v
```

The output shows the remote URL. Notice the repository name is wrong or the URL is misspelled.

4. Check if the repository exists by visiting the URL in a browser. If the page shows a 404 error, the repository does not exist. Create a new repository on GitHub with the correct name.

5. Update the remote URL to the correct one:
```bash
git remote set-url origin https://github.com/your-username/correct-repo-name.git
```

6. Verify the fix:
```bash
git remote -v
git push -u origin main
```

The push succeeds because the URL now points to a real repository. This diagnostic process works for any remote-related error: check the URL, verify the repository exists, and update the remote configuration.
