You will build a workflow with multiple jobs, a matrix strategy, inter-job dependencies, and shared outputs. By the end you will understand how to structure complex pipelines that maximize parallelism while maintaining correct ordering.

## Setup

```bash
# Create a new repository for this practice
gh repo create jobs-steps-practice --public --clone
```

```bash
# Create the workflows directory
mkdir -p .github/workflows
```

```bash
# Create a multi-language project structure
mkdir -p src tests

cat > src/math.py << 'EOF'
def add(a, b):
    return a + b

def subtract(a, b):
    return a - b

def multiply(a, b):
    return a * b
EOF

cat > tests/test_math.py << 'EOF'
from src.math import add, subtract, multiply

def test_add():
    assert add(2, 3) == 5

def test_subtract():
    assert subtract(5, 3) == 2

def test_multiply():
    assert multiply(2, 3) == 6
EOF

cat > requirements.txt << 'EOF'
pytest>=7.0
pytest-cov>=4.0
EOF

cat > Makefile << 'EOF'
.PHONY: test lint

test:
	pytest tests/ -v --cov=src

lint:
	flake8 src/ tests/
EOF
```

## The Walkthrough

### Step 1: Create a single-job workflow

Start with a basic workflow that has one job and multiple steps:

```yaml
name: Single Job
on: workflow_dispatch

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Lint
        run: pip install flake8 && flake8 src/ tests/
      - name: Test
        run: pytest tests/ -v --cov=src
```

**Checkpoint:** Commit and trigger the workflow:
```bash
git add .
git commit -m "Add single-job workflow"
git push origin main
```

### Step 2: Split into separate jobs

Now split the single job into `lint` and `test` jobs that run in parallel:

```yaml
name: Parallel Jobs
on: workflow_dispatch

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install flake8
      - run: flake8 src/ tests/

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest tests/ -v --cov=src
```

**Checkpoint:** Update the workflow file and push:
```bash
git add .github/workflows/*.yml
git commit -m "Split into parallel lint and test jobs"
git push origin main
```

Observe in the Actions tab that both `lint` and `test` run simultaneously.

### Step 3: Add a deploy job that depends on both

```yaml
name: Sequential Jobs
on: workflow_dispatch

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install flake8
      - run: flake8 src/ tests/

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest tests/ -v --cov=src

  deploy:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        run: |
          echo "Both lint and test passed!"
          echo "Deploying to production..."
```

**Checkpoint:** Push and observe that `deploy` waits for both `lint` and `test` to complete before starting.

### Step 4: Add a matrix strategy for OS coverage

Expand the `test` job to run on multiple operating systems:

```yaml
name: Matrix Jobs
on: workflow_dispatch

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install flake8
      - run: flake8 src/ tests/

  test:
    needs: lint
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        python-version: ['3.10', '3.11']
      fail-fast: false
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install -r requirements.txt
        shell: bash
      - run: pytest tests/ -v --cov=src
        shell: bash
      - name: Show OS info
        run: |
          echo "OS: ${{ matrix.os }}"
          echo "Python: ${{ matrix.python-version }}"
          echo "Runner: ${{ runner.os }}"

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "All 6 matrix jobs passed — deploying!"
```

This produces 6 test jobs (3 OSes x 2 Python versions). The `fail-fast: false` setting means one failing variant does not cancel the others.

**Checkpoint:** Push and watch the Actions tab. You should see 6 `test` jobs in the matrix, plus `lint` and `deploy`.

### Step 5: Use job outputs to pass data between jobs

Add a `prepare` job that computes a version number and passes it to downstream jobs:

```yaml
name: Job Outputs
on: workflow_dispatch

jobs:
  prepare:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.version.outputs.value }}
      build_date: ${{ steps.date.outputs.value }}
    steps:
      - id: version
        run: echo "value=1.0.${{ github.run_number }}" >> $GITHUB_OUTPUT
      - id: date
        run: echo "value=$(date +%Y-%m-%d)" >> $GITHUB_OUTPUT
      - run: echo "Preparing build ${{ steps.version.outputs.value }}"

  build:
    needs: prepare
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          echo "Building version ${{ needs.prepare.outputs.version }}"
          echo "Build date: ${{ needs.prepare.outputs.build_date }}"
          echo "Build complete"

  deploy:
    needs: [prepare, build]
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "Deploying version ${{ needs.prepare.outputs.version }}"
          echo "Build dated ${{ needs.prepare.outputs.build_date }}"
```

**Checkpoint:** Push and verify the version number increments each time you run the workflow.

### Step 6: Use timeout and continue-on-error

Add a long-running step that would normally break the pipeline, but use `continue-on-error` and `timeout-minutes` to handle it gracefully:

```yaml
name: Timeout and Continue
on: workflow_dispatch

jobs:
  primary:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest tests/ -v

  long-task:
    needs: primary
    runs-on: ubuntu-latest
    timeout-minutes: 2
    continue-on-error: true
    steps:
      - name: This will timeout
        run: sleep 300
      - name: This never runs
        run: echo "I never execute"

  final:
    needs: [primary, long-task]
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "Primary passed."
          echo "Long task timed out (expected — that is fine)."
          echo "Final job runs regardless."
```

**Expected behavior:** The `long-task` job is cancelled after 2 minutes. The `final` job runs anyway because `long-task` has `continue-on-error: true`.

## Common Errors at This Stage

### Error 1: "No runner matching the specified labels"
**Message:** `No runner matching the specified labels was found: [self-hosted, my-label]`

**Why:** You specified a custom runner label in `runs-on` but no runner with that label is registered.

**Fix:** Either register a self-hosted runner with that label, or change to a GitHub-hosted runner:
```yaml
runs-on: ubuntu-latest  # Use a GitHub-hosted runner instead
```

### Error 2: "Matrix is empty"
**Message:** `Matrix must contain at least one vector of parameters.`

**Why:** The matrix definition is empty or has a syntax error that causes it to produce zero combinations.

**Fix:** Ensure the matrix has at least one valid variable:
```yaml
strategy:
  matrix:
    os: [ubuntu-latest]  # At least one value required
```

### Error 3: "Job output not found"
**Message:** `needs.build.outputs.version` is empty or undefined.

**Why:** The output was not set correctly in the upstream job. Either the step ID is missing, or the output was set in a step that failed.

**Fix:** Verify the step has an `id` and the output is set correctly:
```yaml
steps:
  - id: my-step
    run: echo "version=1.0.0" >> $GITHUB_OUTPUT
```
Then reference it: `${{ needs.prepare.outputs.version }}` (not `my-step`).

### Error 4: "Job skipped because of dependency failure"
**Message:** Job status shows "Skipped" with a grey icon.

**Why:** A job that has `needs: [other-job]` will be skipped if `other-job` fails. This is the default behavior.

**Fix:** If you want the downstream job to run even when an upstream job fails, add `if: always()` or `if: failure()`:
```yaml
cleanup:
  needs: [test]
  if: always()  # Run regardless of test outcome
  runs-on: ubuntu-latest
  steps:
    - run: echo "Cleanup running regardless of test status"
```

## What You Just Learned

- Jobs run in parallel by default; `needs` creates sequential dependencies
- Steps within a job run sequentially and share the working directory
- The `runs-on` key determines which OS and runner type executes the job
- Matrix strategies create multiple job variants from a single job definition
- Job outputs pass data between jobs via `$GITHUB_OUTPUT` and `needs.<id>.outputs`
- `timeout-minutes` and `continue-on-error` control job failure behavior
- `fail-fast` controls whether one matrix failure cancels the other variants
