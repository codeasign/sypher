Your CI pipeline builds a compiled binary, runs tests, and generates a coverage report. The tests pass, but the production deploy needs that binary. The binary is on the runner that ran the build job — but the deploy job runs on a different runner with a fresh filesystem. The binary is gone. You need a way to persist files between jobs and make them available for download after the workflow completes.

## The Concept

Artifacts are files or directories produced by a workflow run that are uploaded to GitHub storage and can be downloaded later — either by another job in the same workflow, or by a user from the Actions tab. Artifacts are stored as compressed archives and are retained for a configurable period (default 90 days).

Think of artifacts as a cargo hold on a ship. The build job is the factory that produces crates (binary, test reports, logs). The crates are loaded into the cargo hold (GitHub artifact storage). The deploy job is the distributor at the next port that picks up the crates from the hold. Without the cargo hold, each factory would need to hand-deliver its crates directly to the distributor — which is impossible when they operate on different ships (runners).

## Key Actions and Properties

| Action | Purpose | Key Inputs |
|--------|---------|------------|
| `actions/upload-artifact@v4` | Upload files from runner to GitHub | `name`, `path`, `if-no-files-found`, `retention-days`, `compression-level` |
| `actions/download-artifact@v4` | Download files from GitHub to runner | `name`, `path`, `github-token` |
| `actions/cache@v4` | Cache dependencies for speed (not persistence) | `path`, `key`, `restore-keys` |

### `upload-artifact` inputs

| Input | Description | Default |
|-------|-------------|---------|
| `name` | Artifact name (used for download) | `artifact` |
| `path` | File or directory to upload (supports globs) | `.` (entire workspace) |
| `if-no-files-found` | Action if no files match: `warn`, `error`, `ignore` | `warn` |
| `retention-days` | Override the default retention period | Repository default (90) |
| `compression-level` | Zip compression level (0-9, 0 = store, 9 = max) | `6` |

### `download-artifact` inputs

| Input | Description | Default |
|-------|-------------|---------|
| `name` | Specific artifact name to download | Downloads all artifacts |
| `path` | Directory to extract into | `.` (workspace root) |
| `github-token` | Token for authentication | `${{ github.token }}` |

## How It Actually Works — Step by Step

### Step 1: Files are created on the runner

During a job's steps, files are written to the runner's filesystem — compiled binaries, test reports, coverage data, logs. These files exist only on that specific runner's ephemeral disk.

### Step 2: The upload action compresses and uploads

When `actions/upload-artifact@v4` runs, it:
1. Globs the `path` pattern to find matching files
2. Compresses them into a zip archive
3. Uploads the archive to GitHub's artifact storage via a pre-signed URL
4. The upload is chunked (for large files) and retried on failure

### Step 3: The download action fetches and extracts

In a downstream job, `actions/download-artifact@v4`:
1. Makes an authenticated API call to GitHub to get the artifact download URL
2. Downloads the zip archive
3. Extracts it to the specified `path` (default: workspace root)
4. The extracted files are now available on the new runner's filesystem

### Step 4: Artifacts are available in the Actions UI

After the workflow completes, users can download artifacts from the Actions tab:
- Navigate to the workflow run
- Scroll to the "Artifacts" section at the bottom
- Click the artifact name to download the zip file

### Step 5: Artifact retention and cleanup

GitHub automatically deletes artifacts after the retention period (90 days by default). You can override this:
- Per-artifact: `retention-days: 7` in the upload step
- Repository-wide: Settings > Actions > General > Artifact and log retention

## When to Use Artifacts / When Not To

### Use artifacts when:
- You need to pass build outputs between jobs (binary → deploy)
- You want to make test reports or coverage data downloadable
- You need to store logs or screenshots from failed tests
- You want to archive build artifacts for later release

### Do not use artifacts when:
- The data is only needed within a single job — use the shared filesystem instead
- The data is sensitive (secrets, passwords) — artifacts are accessible to anyone with repo read access
- The data is large (multiple GB) — consider using external storage like S3 instead
- You need to share data between workflow runs — use releases or external storage instead

### Use `actions/cache` instead when:
- You want to speed up dependency installation (npm, pip, Maven)
- The data is disposable and can be regenerated
- You do not need to download the data manually from the UI
