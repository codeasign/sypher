## Tier 1 — Beginner

### Exercise 1.1: Identify the Gates in a Workflow

Read the following workflow and identify every quality gate. For each gate, write down the YAML key used and what it enforces.

```yaml
name: CI
on:
  pull_request:
    branches: [main]
    paths:
      - "src/**"
      - "!src/docs/**"

jobs:
  lint:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx eslint .

  test:
    needs: lint
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [16, 18, 20]
      fail-fast: true
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

**Hint:** Count every YAML key that restricts whether or when a job runs. There are at least four different gates beyond the basic `steps: run:` commands.

**Solution:** The workflow contains six quality gates:
1. `on.pull_request.branches: [main]` — Branch gate: only PRs targeting main trigger the pipeline.
2. `on.pull_request.paths` — Path gate: only changes to `src/` trigger it, and `src/docs/` changes are excluded.
3. `jobs.lint.timeout-minutes: 5` — Time gate: the lint job must complete within 5 minutes.
4. `jobs.test.needs: lint` — Dependency gate: the test job only runs if lint passes.
5. `jobs.test.strategy.matrix.node` — Matrix gate: the same tests run on three Node versions.
6. `jobs.test.strategy.fail-fast: true` — Fail-fast gate: if any matrix job fails, all in-progress jobs are cancelled.

### Exercise 1.2: Add a Path Gate to an Existing Workflow

You have a monorepo with two directories: `api/` and `web/`. Currently, pushing changes to either directory triggers the full CI pipeline. Modify the workflow so that changes to `api/` run only the API tests and changes to `web/` run only the web tests.

**Starting workflow:**

```yaml
name: CI
on:
  push:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

**Hint:** Use `paths` on the `on.push` trigger and consider using separate workflow files in `.github/workflows/`.

**Solution:**

Create two workflow files.

File `.github/workflows/api-ci.yml`:

```yaml
name: API CI
on:
  push:
    branches: [main]
    paths:
      - "api/**"
jobs:
  test:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: api
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

File `.github/workflows/web-ci.yml`:

```yaml
name: Web CI
on:
  push:
    branches: [main]
    paths:
      - "web/**"
jobs:
  test:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: web
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

### Exercise 1.3: Add a Timeout Gate

You have an integration test suite that sometimes hangs forever due to a third-party API that does not respond. Add a timeout gate to the workflow that kills the job after 10 minutes.

**Starting workflow:**

```yaml
name: Integration Tests
on:
  pull_request:
    branches: [main]
jobs:
  integration:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker compose up -d
      - run: npm run test:integration
      - run: docker compose down
```

**Hint:** You need exactly one additional YAML key at the job level. It is not a step-level key.

**Solution:**

```yaml
name: Integration Tests
on:
  pull_request:
    branches: [main]
jobs:
  integration:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - run: docker compose up -d
      - run: npm run test:integration
      - run: docker compose down
```

The `timeout-minutes: 10` key at the job level kills the entire job if it runs longer than 10 minutes. This prevents runaway containers from consuming runner minutes indefinitely.

## Tier 2 — Intermediate

### Exercise 2.1: Build a Conditional Deployment Pipeline with Manual Approval

Design a pipeline with four jobs: `lint`, `test`, `build`, and `deploy`. The `deploy` job must:
- Only run on pushes to `main` (not on pull requests).
- Only run if `lint`, `test`, and `build` all succeed.
- Use the GitHub `production` environment with required reviewers.
- Include a post-deployment health check step that fails and triggers a rollback.

**Hint:** Use `if` at the job level, `needs` for the dependency chain, and `environment` for the approval gate. Add a health check step with `if: failure()`.

**Solution:**

```yaml
name: Deploy Pipeline
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx eslint .

  test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build

  deploy:
    needs: build
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - name: Health check
        run: |
          curl --retry 3 --retry-delay 5 --retry-connrefused \
            https://api.example.com/health
      - name: Rollback
        if: failure()
        run: |
          echo "Rolling back to previous version..."
```

### Exercise 2.2: Gated Matrix with Conditional Skipping

You have a matrix of four operating systems: ubuntu, macos, windows. The Node 18 tests on ubuntu must pass before any other matrix job starts. The remaining matrix jobs run in parallel.

**Hint:** You cannot mix sequential and parallel execution in a single matrix block. Create two jobs: one non-matrix job for the required OS and Node version, then a matrix job that depends on it.

**Solution:**

```yaml
name: Gated Matrix
on:
  pull_request:
    branches: [main]

jobs:
  gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - run: npm ci
      - run: npm test

  matrix-tests:
    needs: gate
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node: [16, 20]
        exclude:
          - os: ubuntu-latest
            node: 18
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

The `gate` job acts as the quality gate. The `matrix-tests` job only runs if `gate` passes, then fans out across all OS and Node combinations.

### Exercise 2.3: Add a Security Scan Gate

Add a CodeQL security scan gate between the `build` stage and the `deploy` stage. The security scan must run on every pull request. If it finds any high-severity vulnerabilities, the pipeline must halt and the deploy must not run.

**Hint:** Use the `github/codeql-action/analyze` action. The dependency chain should be: lint → test → build → security → deploy.

**Solution:**

```yaml
name: CI with Security Gate
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx eslint .

  test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build

  security:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
        with:
          languages: javascript
      - uses: github/codeql-action/analyze@v3

  deploy:
    needs: security
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
```

## Tier 3 — Advanced

### Exercise 3.1: Design a Quality Gate Scorecard

Your team has four quality gates: lint, unit tests, integration tests, and security scan. Each gate produces a numeric score: lint produces a warning count (0 is perfect), unit tests produce a coverage percentage, integration tests produce a pass rate, and the security scan produces a vulnerability count. Design a pipeline that:

1. Runs all four gates in parallel.
2. Collects the numeric output from each gate.
3. Computes a composite quality score.
4. Only deploys if the composite score exceeds 90 out of 100.

**Reasoning-based solution:**

Parallel execution means each gate runs independently. To collect numeric outputs, each job must use `outputs` to expose its result. A final aggregation job collects all outputs and computes the score.

```yaml
name: Quality Scorecard
on:
  push:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    outputs:
      warning-count: ${{ steps.lint-report.outputs.warnings }}
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - id: lint-report
        run: |
          npx eslint . --format json --output-file eslint-report.json || true
          WARNINGS=$(jq '.[0].warningCount' eslint-report.json)
          echo "warnings=$WARNINGS" >> $GITHUB_OUTPUT

  unit-tests:
    runs-on: ubuntu-latest
    outputs:
      coverage: ${{ steps.coverage.outputs.pct }}
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - id: coverage
        run: |
          npx jest --coverage --silent
          PCT=$(jq '.total.lines.pct' coverage/coverage-summary.json)
          echo "pct=$PCT" >> $GITHUB_OUTPUT

  integration-tests:
    runs-on: ubuntu-latest
    outputs:
      pass-rate: ${{ steps.integration.outputs.pass_rate }}
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - id: integration
        run: |
          npx jest --testPathPattern=integration --json --outputFile=integration-results.json
          PASS=$(jq '.numPassedTests' integration-results.json)
          TOTAL=$(jq '.numTotalTests' integration-results.json)
          RATE=$(echo "scale=2; $PASS / $TOTAL * 100" | bc)
          echo "pass_rate=$RATE" >> $GITHUB_OUTPUT

  security:
    runs-on: ubuntu-latest
    outputs:
      vulnerabilities: ${{ steps.codeql.outputs.vulns }}
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
        with:
          languages: javascript
      - uses: github/codeql-action/analyze@v3
      - id: codeql
        run: |
          VULNS=$(jq '[.. | .severity? | select(. == "high" or . == "critical")] | length' codeql-results.sarif)
          echo "vulns=$VULNS" >> $GITHUB_OUTPUT

  scorecard:
    needs: [lint, unit-tests, integration-tests, security]
    runs-on: ubuntu-latest
    steps:
      - run: |
          LINT=${{ needs.lint.outputs.warning-count }}
          COVERAGE=${{ needs.unit-tests.outputs.coverage }}
          PASS_RATE=${{ needs.integration-tests.outputs.pass-rate }}
          VULNS=${{ needs.security.outputs.vulnerabilities }}
          LINT_SCORE=$(echo "scale=2; if ($LINT == 0) 100 else 100 - ($LINT * 2)" | bc)
          COV_SCORE=$COVERAGE
          INT_SCORE=$PASS_RATE
          SEC_SCORE=$(echo "scale=2; if ($VULNS == 0) 100 else 100 - ($VULNS * 10)" | bc)
          COMPOSITE=$(echo "scale=2; ($LINT_SCORE + $COV_SCORE + $INT_SCORE + $SEC_SCORE) / 4" | bc)
          echo "Composite quality score: $COMPOSITE"
          if (( $(echo "$COMPOSITE < 90" | bc -l) )); then
            echo "Scorecard gate: FAILED (score $COMPOSITE below 90)"
            exit 1
          fi
          echo "Scorecard gate: PASSED (score $COMPOSITE)"
```

This pipeline treats quality as a quantitative, multi-dimensional metric. Each gate contributes a weighted subscore. The final gate is a composite score threshold that blocks deployment when quality drops below the team's standard.

### Exercise 3.2: Build a Promotion Gate Pipeline

Design a pipeline that implements staged promotion through three environments: `dev`, `staging`, and `production`. Each environment is a separate quality gate with different requirements.

- Dev promotion requires lint and unit tests to pass.
- Staging promotion requires integration tests and a security scan to pass.
- Production promotion requires manual approval, a 24-hour soak time, and smoke tests.

**Reasoning-based solution:**

This requires multiple workflow runs or a single workflow with environment-based logic. The cleanest approach is a single workflow that uses job-level conditions and three GitHub environments.

```yaml
name: Staged Promotion Pipeline
on:
  push:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx eslint .

  unit-tests:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test -- --coverage

  deploy-dev:
    needs: unit-tests
    runs-on: ubuntu-latest
    environment: dev
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - run: echo "Deploying to dev..."

  integration-tests:
    needs: deploy-dev
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:integration

  security-scan:
    needs: deploy-dev
    runs-on: ubuntu-latest
    permissions:
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
        with:
          languages: javascript
      - uses: github/codeql-action/analyze@v3

  deploy-staging:
    needs: [integration-tests, security-scan]
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - run: echo "Deploying to staging..."

  smoke-tests:
    needs: deploy-staging
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "Running smoke tests against staging..."
          sleep 30
          echo "All smoke tests passed."

  soak-period:
    needs: smoke-tests
    runs-on: ubuntu-latest
    steps:
      - run: |
          echo "24-hour soak period starts now."
          echo "In a real setup, this would wait 24 hours."
          echo "For this exercise, we simulate with a 30-second wait."
          sleep 30

  deploy-production:
    needs: soak-period
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      - run: echo "Deploying to production..."
```

The promotion gate pattern ensures that each environment validates a different quality dimension. Dev validates basic correctness. Staging validates integration and security. Production adds human judgment and time-based verification before the final release.

### Exercise 3.3: Build a Self-Healing Pipeline with Retry Gates

Design a pipeline that retries failed stages up to 3 times before declaring failure. The retry logic must apply to the test stage only, not to the lint or build stages. If a test fails, the pipeline should isolate and re-run only the failed test subset.

**Reasoning-based solution:**

GitHub Actions does not have built-in retry for jobs. You implement retry logic at the step level using the `pytest-rerunfailures` plugin or a custom retry script with `continue-on-error`. The isolation of failed tests requires capturing the test results and feeding them back into the test runner.

```yaml
name: Self-Healing Pipeline
on:
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx eslint . --max-warnings 0

  build:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build

  test-with-retry:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - name: Run tests with automatic retry
        run: |
          MAX_RETRIES=3
          RETRY_COUNT=0
          EXIT_CODE=0
          while [ $RETRY_COUNT -lt $MAX_RETRIES ]; do
            npx jest --json --outputFile=test-results.json 2>&1 || true
            FAILED=$(jq '.numFailedTests' test-results.json)
            if [ "$FAILED" -eq "0" ]; then
              echo "All tests passed on attempt $((RETRY_COUNT + 1))"
              EXIT_CODE=0
              break
            fi
            RETRY_COUNT=$((RETRY_COUNT + 1))
            if [ $RETRY_COUNT -lt $MAX_RETRIES ]; then
              echo "Attempt $RETRY_COUNT failed with $FAILED failing tests. Retrying..."
              # Re-run only the failed tests on the next iteration
              FAILED_FILES=$(jq -r '.testResults[] | select(.status == "failed") | .name' test-results.json)
              echo "$FAILED_FILES" > failed-tests.txt
            else
              echo "All $MAX_RETRIES attempts exhausted. $FAILED tests still failing."
              EXIT_CODE=1
            fi
          done
          exit $EXIT_CODE
```

This approach gives flaky tests a chance to pass on retry while keeping the lint and build gates strict. The isolation of failed tests means the retry runs fewer tests each time, speeding up the feedback loop.
