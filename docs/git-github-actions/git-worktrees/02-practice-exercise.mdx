You will create a repository, add worktrees, work in multiple worktrees simultaneously, list them, and clean them up. Every command works identically on Windows, macOS, and Linux.

## Setup

```bash
mkdir -p ~/worktree-lab
cd ~/worktree-lab
git init main-project
cd main-project
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create the starting code:

```bash
cat > app.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b
EOF
git add app.py
git commit -m "Initial calculator"
```

## The Walkthrough

### Step 1: Create a feature branch and start working

```bash
git switch -c feature/multiply
```

Add a multiply function:

```bash
cat > app.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b


def multiply(a, b):
    return a * b
EOF
```

The file is modified but not staged. Now a production bug report comes in.

**Checkpoint:** `git status` shows `feature/multiply` with uncommitted changes.

### Step 2: Create a worktree for the hotfix without losing your place

```bash
git worktree add ../hotfix main
```

Expected output:

```
Preparing worktree (checking out 'main')
HEAD is now at abc1234 Initial calculator
```

A new directory `../hotfix` exists with `main` checked out. Your original directory is still on `feature/multiply` with the uncommitted changes.

```bash
ls ../hotfix/
```

Expected output:

```
app.py
```

The hotfix worktree has the clean `main` version of the code.

### Step 3: Fix the bug in the hotfix worktree

In a separate terminal window (or the same one, just change directory):

```bash
cd ../hotfix
```

Fix the bug:

```bash
cat > app.py << 'EOF'
def add(a, b):
    if a is None or b is None:
        return None
    return a + b


def subtract(a, b):
    if a is None or b is None:
        return None
    return a - b
EOF
git add app.py
git commit -m "Fix null input handling"
```

### Step 4: Verify the main worktree is untouched

```bash
cd ../main-project
git status
```

Expected output — `feature/multiply` with the uncommitted `multiply` function, exactly as you left it:

```
On branch feature/multiply
Changes not staged for commit:
  modified: app.py

no changes added to commit
```

No stashing, no lost work. The hotfix worktree is independent.

### Step 5: List all worktrees

```bash
git worktree list
```

Expected output:

```
D:/.../main-project   abc1234 [feature/multiply]
D:/.../hotfix         def5678 [main]
```

Each line shows the worktree path, the HEAD commit hash, and the checked-out branch.

### Step 6: Create a worktree with a new branch

```bash
git worktree add -b feature/divide ../divide
```

Expected output:

```
Preparing worktree (checking out 'feature/divide')
HEAD is now at abc1234 Initial calculator
```

The `-b` flag creates a new branch `feature/divide` and checks it out in `../divide/`. This is equivalent to `git branch feature/divide` + `git worktree add ../divide feature/divide`.

### Step 7: Work in all three worktrees

In the `divide` worktree:

```bash
cd ../divide
cat > app.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b


def divide(a, b):
    if b == 0:
        return None
    return a / b
EOF
git add app.py
git commit -m "Add divide function"
```

Meanwhile, in the `main-project` worktree, you can continue working on `feature/multiply`. The three worktrees are independent:

```
  main-project/  → feature/multiply  (uncommitted changes)
  hotfix/        → main              (committed the null check fix)
  divide/        → feature/divide     (committed the divide function)
```

### Step 8: Merge the hotfix and clean up

```bash
cd ../main-project
git fetch . ../hotfix/main  # Fetch the hotfix branch's commit
# Or simply merge from the main branch
git worktree list  # Identify the hotfix worktree
```

Merge the hotfix:

```bash
git switch main
git merge ../hotfix  # Or merge the specific branch
```

If your worktree is a linked worktree, you can reference it directly:

```bash
git fetch origin  # If the hotfix was pushed
git merge origin/main  # If the hotfix was merged upstream
```

For a local merge (since both worktrees share refs):

```bash
git log --all --oneline  # Hotfix commit is visible
git merge hotfix/urgent  # If the branch exists
```

Actually, the hotfix was created with `git worktree add ../hotfix main` — it checked out `main`, not a new branch. So the hotfix commit is on `main` in that worktree. Since the worktrees share refs, the hotfix commit is accessible from the main worktree:

```bash
git log --all --oneline
```

You should see the hotfix commit. Merge it:

```bash
git merge <hotfix-commit-hash>
```

Now clean up the hotfix worktree:

```bash
git worktree remove ../hotfix
```

Expected output:

```
Removing worktrees/hotfix
```

### Step 9: Prune stale worktrees

If you delete a worktree directory manually (without `git worktree remove`), Git still has the metadata:

```bash
rm -rf ../divide
git worktree list  # Still shows the divide worktree
git worktree prune
git worktree list  # Stale entry is gone
```

`git worktree prune` scans the worktree metadata and removes entries whose directories no longer exist.

## Common Errors at This Stage

### Error 1: `fatal: '<branch>' is already checked out at '<path>'`

You tried to check out a branch in one worktree that is already checked out in another worktree. Git prevents the same branch from being checked out in multiple worktrees to avoid conflicts.

**Fix:** Either work on the branch from its existing worktree, or use `git branch -f` to move the branch (if you are sure no one is using it). You can also create a new branch from that commit and work on the new branch.

### Error 2: `fatal: '<path>' already exists`

You tried to create a worktree at a path that already has a directory. Git refuses to overwrite an existing directory.

**Fix:** Remove the existing directory or choose a different path. Use `rm -rf <path>` if the directory is not needed, or rename it.

### Error 3: `fatal: could not open '.../.git/worktrees/<name>/HEAD': No such file or directory`

You deleted a worktree directory manually but the worktree metadata still exists in `.git/worktrees/`. Git tries to read the worktree's HEAD and fails.

**Fix:** Run `git worktree prune` to clean up stale metadata. If that does not work, manually remove the worktree directory from `.git/worktrees/`: `rm -rf .git/worktrees/<name>`.

### Error 4: Worktree shows incorrect branch in `git worktree list`

You used `git worktree add` with a branch name, but the branch was based on the wrong commit. The worktree shows the correct branch but the wrong starting point.

**Fix:** The worktree is just a checkout of the branch at the time you created it. To update the branch, `cd` into the worktree and `git pull` or `git rebase` to the desired commit. The worktree metadata updates automatically when the branch moves.

## What You Just Learned

- `git worktree add <path> <branch>` creates a new working directory with `<branch>` checked out
- `git worktree add -b <branch> <path>` creates a new branch and worktree in one command
- Each worktree has its own working directory, index, and HEAD
- All worktrees share the same Git object database and refs
- A branch can only be checked out in one worktree at a time
- `git worktree list` shows all active worktrees
- `git worktree remove <path>` removes a worktree safely
- `git worktree prune` cleans up metadata for manually deleted worktrees
