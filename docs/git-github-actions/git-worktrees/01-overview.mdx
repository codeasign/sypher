You are working on a feature branch with uncommitted changes when a production bug is reported. You need to switch to `main` to investigate, but you cannot — Git refuses because your working directory is dirty. You could stash your changes, but what if you need to reference them while fixing the bug? Or you could commit half-finished work, but that pollutes the history. Git worktrees let you check out multiple branches simultaneously in separate directories, all sharing the same repository.

## The Concept

A worktree is an additional working directory linked to the same Git repository. Each worktree has its own working directory, index, and HEAD, but they all share the same object database, refs, and configuration. You can have `main` checked out in one directory and a feature branch in another, both open at the same time.

Think of a worktree like having multiple monitors connected to one computer. Each monitor shows a different workspace (different branch), but they all access the same filesystem and applications. You can work on one monitor while looking at the other for reference. No switching, no stashing, no lost context.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git worktree add <path> <branch>` | Create a new worktree at `<path>` with `<branch>` checked out. | `git worktree add ../hotfix hotfix/urgent` |
| `git worktree add -b <branch> <path>` | Create a new branch AND a new worktree in one command. | `git worktree add -b new-feature ../feature` |
| `git worktree list` | List all worktrees with their paths, branches, and HEAD hashes. | `git worktree list` |
| `git worktree remove <path>` | Remove a worktree (requires clean working directory). | `git worktree remove ../hotfix` |
| `git worktree prune` | Clean up stale worktree metadata after a worktree directory was deleted manually. | `git worktree prune` |
| `git worktree lock <path>` | Lock a worktree to prevent it from being pruned. | `git worktree lock ../hotfix` |
| `git worktree unlock <path>` | Unlock a locked worktree. | `git worktree unlock ../hotfix` |

**Windows note:** Worktrees work identically on all platforms. The worktree directory can be anywhere on the filesystem — it does not have to be a sibling of the main working tree. On Windows, avoid paths with trailing spaces or special characters. Use forward slashes in paths: `git worktree add ../hotfix` works on all OSes.

## How It Actually Works — Step by Step

### Step 1: Create a worktree for a hotfix

You are on `main` with a clean working directory:

```bash
git worktree add ../hotfix hotfix/urgent
```

Git creates:
1. A new directory `../hotfix/` as a sibling of your current project
2. A new branch `hotfix/urgent` based on the current `HEAD` (if it does not exist)
3. The branch `hotfix/urgent` is checked out in `../hotfix/`
4. A worktree entry is added to `.git/worktrees/hotfix/`
5. A `.git` file in `../hotfix/` points back to the main repo's `.git/worktrees/hotfix/`

```
  Linked worktree: .git/worktrees/hotfix/
    HEAD           → ref: refs/heads/hotfix/urgent
    index          → hotfix's staging area
    commondir      → .. (refers to the main .git/)
```

### Step 2: Work in both worktrees simultaneously

In one terminal:

```bash
cd ~/hotfix
# Fix the bug
git add .
git commit -m "Fix production crash on null input"
git push origin hotfix/urgent
```

In another terminal:

```bash
cd ~/project
# Continue working on the feature
git status  # Shows feature branch, still on main, no stashing needed
```

Both worktrees share the same refs. When you push from the hotfix worktree, the refs are updated and visible from the main worktree.

### Step 3: Merge the hotfix back to main

From the main worktree:

```bash
cd ~/project
git fetch origin
git merge origin/hotfix/urgent
```

The main worktree's `main` branch now includes the hotfix. The hotfix worktree is still on `hotfix/urgent` — you can delete it once the fix is merged.

### Step 4: Remove the worktree

```bash
git worktree remove ../hotfix
```

If the worktree has uncommitted changes, Git refuses:

```
fatal: '../hotfix' contains modified or untracked files, use --force to delete it
```

Use `--force` if you are sure:

```bash
git worktree remove --force ../hotfix
```

Or, if you deleted the directory manually, prune the stale metadata:

```bash
rm -rf ../hotfix
git worktree prune
```

### Step 5: Create a worktree with a new branch

```bash
git worktree add -b feature/new-algorithm ../new-algorithm
```

This creates a new branch `feature/new-algorithm` from `HEAD` and checks it out in `../new-algorithm/`. You can start working immediately without touching the main worktree.

## When to Use This / When Not To

**Use worktrees when:**
- You need to context-switch between branches frequently without stashing or committing half-finished work
- You are reviewing a pull request and want to see the code side-by-side with your current work
- You need to work on a hotfix while keeping a long-running feature branch intact
- You want to run tests or builds on multiple branches simultaneously

**Do not use worktrees when:**
- You only need to switch branches for a few minutes — `git stash` is simpler
- The repository is large and cloning again is faster than adding a worktree (worktrees share objects, so they are usually faster)
- You are on a filesystem with limited space — each worktree duplicates the working tree files
- You need to work on the same branch from two worktrees (Git prevents this — a branch can only be checked out in one worktree at a time)

**Worktree vs. clone:** A worktree shares the object store and refs, so `git fetch` in one worktree is visible in all others. A separate clone duplicates the entire `.git/` directory. Worktrees are faster for operations that need to share state (e.g., reviewing a PR while working on a feature). Separate clones are better when you want completely isolated environments (e.g., different Git configurations).
