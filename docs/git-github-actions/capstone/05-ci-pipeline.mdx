# The CI Pipeline

This is the heart of the capstone. You will write a real GitHub Actions workflow, push it, watch it run, break a test on purpose to see the red X, then fix it and watch it turn green.

## What CI Must Verify

For urlcheck, CI must run three checks:

1. **Lint** -- verify code quality with `ruff` (or `flake8` as a simpler alternative)
2. **Test** -- run `pytest` and confirm all tests pass
3. **Build** -- verify the package can be installed (a dry-run install)

These three checks run in parallel on every pull request and must all pass before the PR can merge.

## The Full CI Workflow

Create the file `.github/workflows/ci.yml` with the following content:

```yaml
name: CI

on:
  pull_request:
    branches: [main]

concurrency:
  group: ci-${{ github.head_ref }}
  cancel-in-progress: true

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install ruff

      - name: Lint with ruff
        run: ruff check urlcheck.py test_urlcheck.py

  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.11", "3.12"]
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: Run tests
        run: pytest test_urlcheck.py -v

  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install build tools
        run: pip install build

      - name: Build package
        run: python -m build --sdist

      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
```

## Step-by-Step Walkthrough

### Trigger

```yaml
on:
  pull_request:
    branches: [main]
```

The workflow runs when a pull request targets `main`. This is the critical distinction: CI runs on the PR, not on every push to every branch.

### Concurrency

```yaml
concurrency:
  group: ci-${{ github.head_ref }}
  cancel-in-progress: true
```

If you push new commits while a previous CI run is still going, the old run is cancelled and the new one starts. This saves runner minutes.

### Lint Job

- Runs on `ubuntu-latest`
- Installs `ruff` (a fast Python linter)
- Runs `ruff check` on both source and test files

### Test Job

- Uses a **matrix** strategy: tests run on both Python 3.11 and 3.12
- Installs the project dependencies from `requirements.txt`
- Runs `pytest -v` for verbose output

### Build Job

- Uses the `build` package to create a source distribution
- Uploads the built artifact so it is available for later workflow steps or download

## Push the Workflow and Watch It Run

### Commit and Push the CI Workflow

```bash
# Switch to the feature branch (where the PR is pointing)
git checkout feature/add-urlcheck-cli

# Create the CI workflow file
touch .github/workflows/ci.yml
# (Paste the YAML content from above into this file)

# Commit and push
git add .github/workflows/ci.yml
git commit -m "ci: add CI workflow with lint, test (matrix), and build"
git push origin feature/add-urlcheck-cli
```

The push triggers a new commit on the feature branch, which updates the existing PR. GitHub Actions detects the `pull_request` event and starts the CI workflow.

### Watch in the Actions Tab

1. Go to your repo on GitHub.
2. Click the **Actions** tab.
3. You should see a new run titled **CI** with the commit message "ci: add CI workflow with lint, test (matrix), and build".
4. Click on the run to see the three jobs: lint, test, build.
5. Click on each job to expand its steps and see real-time log output.

### What a Successful Run Looks Like

Each job shows a green checkmark. The PR page shows a green **All checks have passed** banner and the merge button is clickable.

### What a Failing Run Looks Like

A failed job shows a red X. The log contains the error traceback or ruff output. The PR page shows a red **Some checks were not successful** banner and the merge button is disabled.

## Break It on Purpose

The lesson of CI is not that tests pass -- it is that CI stops bad code from merging. You will now experience that first-hand.

### Step 1: Break a Test

Edit `test_urlcheck.py` and make one assertion fail:

```python
def test_reachable_url(self) -> None:
    url, status, text = check_url("https://example.com")
    assert status == 999  # deliberately wrong expectation
    assert text == "OK"
```

### Step 2: Commit and Push

```bash
git add test_urlcheck.py
git commit -m "test: (deliberate) break test_reachable_url"
git push origin feature/add-urlcheck-cli
```

### Step 3: Watch It Fail

1. Go to the Actions tab.
2. See the test (3.11) job running.
3. Watch it turn red with an assertion error:

```
pytest test_urlcheck.py -v

FAILED test_urlcheck.py::TestCheckUrl::test_reachable_url -
AssertionError: assert 200 == 999
```

4. Go back to the PR page. See the red **Some checks were not successful** banner.
5. Notice the **Merge** button is greyed out.

### Step 4: Fix It

```python
def test_reachable_url(self) -> None:
    url, status, text = check_url("https://example.com")
    assert status == 200  # fixed back to correct value
    assert text == "OK"
```

### Step 5: Commit and Push

```bash
git add test_urlcheck.py
git commit -m "fix: correct broken assertion in test_reachable_url"
git push origin feature/add-urlcheck-cli
```

### Step 6: Watch It Go Green

1. Go to the Actions tab. The new run starts automatically.
2. All three jobs pass with green checks.
3. The PR page shows **All checks have passed**.
4. The **Merge** button is clickable (green) again.

You have now seen the full CI cycle: code pushed, checks run, failure blocked, fix applied, checks pass. This is the loop that keeps main branches stable.

## Why This Matters

Without CI, anyone can merge broken code. With CI, the pipeline becomes the gatekeeper. Every PR author knows their code has to pass before it can merge, so they run tests locally first -- not out of discipline, but because the pipeline enforces it.

## What Your PR Should Show Now

Navigate to your PR page. The **Checks** tab shows the full CI run history. You should see:

- One or more green runs (initial pass after adding CI)
- One red run (the deliberate break)
- One green run (the fix)

This commit history -- a red build followed by a green fix -- is exactly what a real delivery pipeline looks like. Leave it visible on the PR. It is the evidence that your pipeline works.
