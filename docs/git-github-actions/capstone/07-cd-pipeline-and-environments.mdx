# CD Pipeline and Environments

CI runs on every pull request to validate code. CD runs only after code is merged to `main`, because only merged code should reach production. Feature branches should never deploy.

## The Trigger Rule

```yaml
on:
  push:
    branches: [main]
```

This workflow fires when a commit lands on `main`, and only then. A push to any other branch is ignored.

## The Full CD Workflow

Create `.github/workflows/cd.yml`:

```yaml
name: CD

on:
  push:
    branches: [main]

concurrency:
  group: cd-main
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: Run tests before deploy
        run: pytest test_urlcheck.py -v

      - name: Deploy (simulated)
        run: |
          echo "========================================"
          echo "  Deploying urlcheck v$(git rev-parse --short HEAD)"
          echo "========================================"
          echo "  Environment: production"
          echo "  Artifact:    urlcheck package"
          echo "  Target:      simulated deployment"
          echo "  Status:      SUCCESS"
          echo "========================================"
```

## Walkthrough

### Environment

```yaml
environment: production
```

This single line does multiple things:
1. It creates or references a GitHub Environment named `production`.
2. It enables environment-level approval gates (which you configure next).
3. It surfaces the deployment in the GitHub Deployments dashboard.
4. It records the deployment history with commit SHA and timestamps.

### Concurrency

```yaml
concurrency:
  group: cd-main
  cancel-in-progress: false
```

Two pushes to `main` close together will queue rather than run in parallel, preventing race conditions during deployment. `cancel-in-progress: false` means a previous deployment finishes before the next one starts.

### Pre-Deploy Confidence

The workflow runs `pytest` again right before the deploy step. This is a belt-and-suspenders check: even though CI already passed on the PR, you verify again on the exact commit that was merged. In practice this catches environment-specific issues (e.g., a newly installed system library that conflicts with your dependencies).

### Simulated Deploy

Because this capstone does not have a real hosting target, the deploy step echoes success. In a real project this step would upload to a hosting provider or run a migration script.

## Configuring the Environment Approval Gate

Without an approval gate, CD deploys immediately after merge. For `main` this is fine; for `production` most teams add a manual approval step so a human can confirm the deployment is intentional.

### Step 1: Create the Environment

1. Go to your repo: **Settings** > **Environments** (left sidebar).
2. Click **New environment**.
3. Name it `production`.
4. Click **Configure environment**.

### Step 2: Add Required Reviewers

Under **Environment protection rules**:

| Field | Value |
|-------|-------|
| Required reviewers | Add your GitHub username |
| Prevent self-approval | Leave unchecked (you are the only person on this project) |
| Wait timer | 0 minutes (no delay needed) |
| Deployment branches | Selected branches: `main` |

1. In the **Required reviewers** box, type your GitHub username and select it from the dropdown.
2. Under **Deployment branches**, choose **Selected branches** and add the branch filter `main`.
3. Click **Save protection rules**.

### Step 3: Verify the Gate

1. After you create or update the environment, any CD workflow that references `environment: production` will pause at the deploy job and wait for approval.
2. When the workflow reaches the deploy job, GitHub sends a notification (and an email if configured) to the required reviewers.
3. The **Actions** tab shows the workflow with a yellow badge: **Waiting for review**.
4. The reviewer clicks **Review deployments**, types a comment, and clicks **Approve and deploy**.
5. Only then does the deploy job run.

## Watching the First Deployment

### Before You Merge

The CD workflow exists in the repo but has never run. You will not see it in the Actions tab until a push to `main` occurs.

### After Merge

1. Go to the **Actions** tab.
2. You should see a new run titled **CD**. The workflow run name is the commit SHA or workflow name.
3. Click on the run. The deploy job shows a yellow badge: **Waiting**.
4. Click **Review deployments**.
5. A dialog appears listing the `production` environment and the commit SHA.
6. Select **Approve and deploy** and confirm.
7. The deploy job turns green and the output shows the simulated deployment message.

### Verify in the Deployments Dashboard

1. Go to your repo.
2. Click the **Environments** section near the bottom of the repo home page (or **Settings** > **Environments** > **production**).
3. You should see a list of recent deployments with timestamps and commit SHAs.

## What You Have Now

- A CD workflow that runs only on push to `main`
- A `production` environment with a required approval gate
- A deployment that waits for manual approval before executing
- Deployment history visible in GitHub Environments
