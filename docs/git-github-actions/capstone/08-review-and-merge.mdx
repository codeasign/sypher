# Review and Merge

All the gates are in place: CI passes, branch protection is configured, and the CD pipeline is ready. Now you will go through the actual review and merge process.

## Requesting a Review

### Self-Review (Solo Project)

For a solo capstone, no one else can review your PR. GitHub allows you to approve your own pull request if you have write access to the repo:

1. Go to the **Files changed** tab on your PR.
2. Click **Review changes**.
3. Select **Approve**.
4. Add a comment (e.g., "LGTM, all checks passing").
5. Click **Submit review**.

### In a Real Team Context

On a team, you would request a review from a colleague:

```bash
# Via gh CLI
gh pr edit --add-reviewer @colleague-username

# Via the web UI
# PR page > Reviewers (right sidebar) > select a reviewer
```

The reviewer would:
1. Read the diff in **Files changed**.
2. Leave inline comments on specific lines.
3. Submit a review: **Comment** (general feedback), **Approve** (looks good), or **Request changes** (must fix before merge).

## Responding to Review Comments

### Adding New Commits (Responding to Feedback)

If a reviewer asks for changes, add new commits addressing the feedback:

```bash
# Make the requested change in your editor
git add urlcheck.py
git commit -m "fix: handle empty URL list in batch mode"
git push origin feature/add-urlcheck-cli
```

The new commits appear on the PR automatically. The reviewer sees exactly what changed since their last review.

### Amending Commits (Before Review)

If you catch a mistake before anyone has reviewed, you can fix up your last commit:

```bash
# Edit the file, then
git add urlcheck.py
git commit --amend --no-edit
git push origin feature/add-urlcheck-cli --force-with-lease
```

Use `--force-with-lease` (never `--force`) to overwrite the previous commit only if no one else has pushed to the same branch. Amending rewrites history, so do it before the PR is reviewed.

### When to Use Each

| Situation | Approach |
|-----------|----------|
| You find a typo before review | Amend your last commit |
| Reviewer asks for a minor change | Add a new commit |
| Reviewer asks for a structural change | Add new commits, possibly squash at merge |
| You want a clean history on main | Use squash merge (see below) |

## The Merge

You now have CI passing, an approving review, and no merge conflicts. Click the green **Merge pull request** button.

### Merge Strategy Decision

GitHub offers three merge methods:

| Method | What it does | History | Best for |
|--------|-------------|---------|----------|
| **Merge commit** | Creates a merge commit with all feature commits | Full history, every commit preserved | Teams that want to see every commit |
| **Squash and merge** | Collapses all feature commits into one | Single commit on main | Clean linear history |
| **Rebase and merge** | Applies each commit onto main without a merge commit | Linear history, each commit preserved | Feature branches with well-structured commits |

### What We Use Here

This capstone uses **Squash and merge**. Why:

- The feature branch has 4+ commits (scaffold, CI workflow, CODEOWNERS, deliberate break, fix). Squashing collapses these into a single meaningful commit on `main`.
- The individual commits on the feature branch include experiments and a deliberate break. You do not want those in `main`'s history.
- Squash and merge produces a linear, readable `main` branch where every commit represents a complete, working feature.

### Perform the Merge

1. Click the dropdown arrow on the **Merge pull request** button.
2. Select **Squash and merge**.
3. Edit the commit message to something clean:

```
feat: add urlcheck CLI tool with CI/CD pipeline

Includes:
- urlcheck CLI with single and batch URL checking
- Unit tests with pytest
- CI workflow with lint, matrix test, and build
- CD workflow with production environment approval gate
- Branch protection with required checks and review
- CODEOWNERS file for path-based ownership
```

4. Click **Squash and merge**.
5. Click **Confirm squash and merge**.

## Deleting the Feature Branch

After merge, GitHub shows a button: **Delete branch**. Click it.

Or from the CLI:

```bash
# Delete remote branch
git push origin --delete feature/add-urlcheck-cli

# Delete local branch (you are on a different branch first)
git checkout main
git pull origin main
git branch -d feature/add-urlcheck-cli
```

## Pull main Locally

```bash
git checkout main
git pull origin main
git log --oneline
```

You should see the squashed commit at the top of the log:

```
g7h8i9j feat: add urlcheck CLI tool with CI/CD pipeline
a1b2c3d chore: scaffold project structure
e5f6g7h Initial commit
```

## Watching CD Fire

Within seconds of the merge, the CD workflow triggers automatically.

1. Go to the **Actions** tab.
2. You should see a new run titled **CD**.
3. Because of the environment approval gate, the deploy job is paused with a yellow **Waiting** badge.
4. Click **Review deployments**.
5. Approve the deployment.
6. Watch the deploy job turn green with the simulated success message.

You have now completed the full cycle: feature branch, PR, CI, review, merge, CD, deploy. Every gate you configured was active. Nothing was optional.
