# Repo and Branch Strategy

## Step 1: Create the GitHub Repository

Choose one of the following methods.

### Option A: Using gh CLI (fastest)

```bash
# Log in if you have not already
gh auth login

# Create the repo with a Python .gitignore
gh repo create urlcheck --public --gitignore Python --clone
cd urlcheck
```

### Option B: Using the GitHub Web UI

1. Go to https://github.com/new in your browser.
2. Enter `urlcheck` as the repository name.
3. Set the visibility to **Public** (you will want this for your portfolio).
4. Check **Add a README file**.
5. Under **Add .gitignore**, select **Python**.
6. Click **Create repository**.
7. Clone it locally:

```bash
git clone https://github.com/YOUR-USERNAME/urlcheck.git
cd urlcheck
```

### Verify It Worked

```bash
git status
# Output: On branch main, nothing to commit, working tree clean

git log --oneline
# Output: one commit (the initial commit from GitHub)
```

## Step 2: Branch Strategy -- Trunk-Based with Short-Lived Feature Branches

For this project you will use **trunk-based development**: a single long-lived `main` branch, with short-lived feature branches that are merged back within hours or days. Every feature branch is deleted after merge.

### Why Not Git Flow?

Git Flow defines `main`, `develop`, `release/`, `hotfix/`, and `feature/` branches. It works for large projects with scheduled releases and multiple parallel versions. For a single-person project where the goal is to learn CI/CD, Git Flow adds overhead without benefit. Trunk-based development keeps things simple so you can focus on the pipeline mechanics that matter.

### The Rule

- `main` is always deployable.
- Every change happens on a topic branch named `feature/<short-description>`.
- No one (including you) pushes directly to `main`.
- All code enters `main` through a pull request that passes CI and review.

## Step 3: Create the Initial Repository Structure

Create the files that represent the project scaffold -- before you write the actual feature code. This establishes the project structure that the CI pipeline will later validate.

### Directory Layout

```
urlcheck/
├── urlcheck.py          # CLI entry point (placeholder for now)
├── test_urlcheck.py     # Test file (placeholder for now)
├── requirements.txt     # Dependencies
├── urls.json            # Sample batch config
├── .github/
│   └── workflows/
│       └── .gitkeep     # Placeholder for the CI workflow you will add later
├── README.md            # Already created by GitHub
└── .gitignore           # Already created by GitHub
```

### Create the Files

```bash
# Create the workflow directory structure
mkdir -p .github/workflows
touch .github/workflows/.gitkeep

# Create requirements.txt
echo "requests==2.31.0
pytest==8.0.0" > requirements.txt

# Create a placeholder for urlcheck.py
echo '"""urlcheck -- CLI tool to check URL reachability."""' > urlcheck.py

# Create a placeholder test file
echo '"""Tests for urlcheck."""' > test_urlcheck.py

# Create a sample batch config
echo '{
    "urls": [
        "https://example.com",
        "https://httpstat.us/404",
        "https://httpstat.us/500"
    ]
}' > urls.json

# Commit the scaffold
git add .
git commit -m "chore: scaffold project structure"
git push origin main
```


Your repo now has a clean `main` branch with the project scaffold. In the next step, you will create a feature branch, write the actual urlcheck code, and push it up.

## Key Takeaways

- You chose trunk-based development over Git Flow because it keeps the focus on CI/CD mechanics.
- Every change flows through a PR -- never a direct push.
- The `main` branch is always deployable because it only receives code that has passed CI and review.
- The scaffold commits establish the project structure before any feature code is written.
