These exercises progress from basic observation of required status checks to advanced multi-workflow gated pipelines. Work through them in order.

## Tier 1 -- Beginner

### Exercise 1.1: View Current Required Status Checks

**Task:** Inspect the branch protection settings on an existing GitHub repository and identify all currently configured required status checks.

**Hint:** Use `gh api` to query the branch protection endpoint. The `--jq` flag extracts just the status checks portion of the response.

**Solution:**

```bash
# Replace :owner and :repo with an actual repository you have access to
gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks'

# Example output:
# {
#   "url": "https://api.github.com/repos/octocat/hello-world/branches/main/protection/required_status_checks",
#   "strict": false,
#   "contexts": ["continuous-integration/travis-ci", "WIP"],
#   "contexts_url": "...",
#   "checks": [
#     { "context": "continuous-integration/travis-ci", "app_id": null },
#     { "context": "WIP", "app_id": null }
#   ]
# }
```

If the branch has no protection, you will get a `404 Not Found` error, meaning the branch has no protection rules at all.

**Observations:**

- `strict` tells you whether the branch requires up-to-date branches before merging
- `contexts` lists the exact check names that are required
- Each context must match a check run name or commit status context exactly, including case and spaces

### Exercise 1.2: Add a Required Status Check via the GitHub UI

**Task:** Navigate to a repository's Settings > Branches page and add one required status check called `"test"` to the main branch protection rule. Do not use the API.

**Hint:** The UI path is: Repository Settings > Branches > Branch protection rules > Edit (or Add rule) for main > Require status checks to pass before merging > Search for a check name. If no check named `"test"` exists yet, the field will accept it as a name that will be required in the future.

**Solution:**

1. Open your repository in a browser: `https://github.com/:owner/:repo/settings/branches`
2. Under "Branch protection rules," click "Add rule" or "Edit" next to `main`
3. Check the box: "Require status checks to pass before merging"
4. In the search box, type `test`
5. Select the check from the dropdown if it exists, or leave the typed name if it does not
6. Click "Save changes"

Once saved, the API will show the new required check:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks.contexts'
```

You should see `"test"` in the contexts array alongside any existing checks.

### Exercise 1.3: Trigger a Check Run and Observe Status

**Task:** Push a commit to a branch that has a CI workflow configured, then watch the status check appear on the commit in real time. Note how the check transitions from `queued` to `in_progress` to `completed`.

**Hint:** Use `gh run watch` to follow the workflow run, and use a second terminal or background command to poll the check run status via the API.

**Solution:**

First, ensure you have a simple workflow in `.github/workflows/observe.yml`:

```yaml
name: Observe Check
on: [push]
jobs:
  status-demo:
    name: Status Demo Check
    runs-on: ubuntu-latest
    steps:
      - run: sleep 10
      - run: echo "Check completed"
```

Push a commit to trigger the workflow, then watch it in real time:

```bash
# In terminal window 1: watch the workflow run
gh run list --limit 1 --json status,name,databaseId
```

```bash
# In terminal window 2: poll the check run state for the latest commit
gh api repos/:owner/:repo/commits/HEAD/check-runs \
  --jq '.check_runs[] | select(.name == "Status Demo Check") | {name, status, conclusion}'
```

Run the second command repeatedly (every 5-10 seconds) to see the transition:

```
# First poll -- check is queued
{ "name": "Status Demo Check", "status": "queued", "conclusion": null }

# Second poll -- check is running
{ "name": "Status Demo Check", "status": "in_progress", "conclusion": null }

# Third poll -- check completed
{ "name": "Status Demo Check", "status": "completed", "conclusion": "success" }
```

### Exercise 1.4: Remove a Required Status Check

**Task:** Remove one required status check from a branch protection rule, leaving the others intact. Verify the removal via the API.

**Hint:** You can only replace the entire `contexts` array, not remove individual elements. You must read the current list, remove the check you want to drop, and write the full list back.

**Solution:**

```bash
# Step 1: Read the current required status checks
CURRENT=$(gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks.contexts')

echo "Current checks: $CURRENT"
# Example: ["Unit Tests", "Lint", "Observe Check"]

# Step 2: Build a new list without the check you want to remove
# If you want to remove "Observe Check":
# CURRENT is ["Unit Tests", "Lint", "Observe Check"]
# You need to send: ["Unit Tests", "Lint"]

# Step 3: Write the updated list back
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": ["Unit Tests", "Lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF

# Step 4: Verify the removal
gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks.contexts'
# Expected: ["Unit Tests", "Lint"]
```

Note: The API replaces the entire `contexts` array. If you have many checks and only need to remove one, consider using `jq` to build the updated list programmatically:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks.contexts - ["Observe Check"]'
```

## Tier 2 -- Intermediate

### Exercise 2.1: Configure Required Status Checks via `gh api`

**Task:** Write a script that configures branch protection on a repository programmatically. The script should accept a list of required check names and a `strict` flag, then apply the protection rule. Include error handling for invalid input.

**Hint:** Accept input from environment variables or command-line arguments. Validate that the contexts array is not empty before calling the API.

**Solution:**

Create a script file `configure-checks.sh`:

```bash
#!/bin/bash
set -euo pipefail

# configure-checks.sh
# Usage: ./configure-checks.sh <owner/repo> <strict|loose> <check1> [check2] ...

if [ $# -lt 3 ]; then
  echo "Usage: $0 <owner/repo> <strict|loose> <check1> [check2] ..."
  exit 1
fi

REPO="$1"
MODE="$2"
shift 2

if [ $# -eq 0 ]; then
  echo "Error: At least one check name is required."
  exit 1
fi

if [ "$MODE" = "strict" ]; then
  STRICT=true
elif [ "$MODE" = "loose" ]; then
  STRICT=false
else
  echo "Error: MODE must be 'strict' or 'loose', got '$MODE'"
  exit 1
fi

# Build contexts array as JSON
CONTEXTS="["
FIRST=true
for CHECK in "$@"; do
  if [ "$FIRST" = true ]; then
    FIRST=false
  else
    CONTEXTS="$CONTEXTS, "
  fi
  CONTEXTS="$CONTEXTS\"$CHECK\""
done
CONTEXTS="$CONTEXTS]"

# Build the JSON payload and call the API
JSON=$(cat << EOF | tr -d '\n'
{
  "required_status_checks": {
    "strict": $STRICT,
    "contexts": $CONTEXTS
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
)

echo "Applying protection to $REPO (strict=$STRICT)..."
echo "Required checks: $CONTEXTS"

gh api "repos/$REPO/branches/main/protection" \
  --method PUT \
  --input - <<< "$JSON"

echo "Done."
```

Make it executable and run it:

```bash
chmod +x configure-checks.sh
./configure-checks.sh myorg/my-repo strict "Unit Tests" "Lint" "Integration Tests"
```

### Exercise 2.2: Set Up Strict Status Checks

**Task:** Configure branch protection with `strict: true` and at least two required checks. Then simulate a scenario where the PR branch falls out of date -- push a commit directly to the base branch while the PR is open -- and verify the merge is blocked.

**Hint:** After pushing to the base branch, use `gh pr view --json mergeable` to check the mergeability. The `"BLOCKED"` state indicates strict mode is working.

**Solution:**

```bash
# Step 1: Configure strict mode
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["Unit Tests", "Lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

```bash
# Step 2: Create a PR branch and push
git checkout -b feature/my-feature
echo "feature work" > feature.txt
git add -A && git commit -m "Add feature"
git push origin feature/my-feature
gh pr create --base main --head feature/my-feature --title "My feature" --body "PR to test strict mode"
```

Wait for checks to pass on the PR. Then push directly to `main` to simulate another PR merging:

```bash
git checkout main
git pull origin main
echo "other work" > other.txt
git add -A && git commit -m "Other PR merged first"
git push origin main
```

```bash
# Step 3: Check PR mergeability
git checkout feature/my-feature
git pull origin feature/my-feature
gh pr view --json mergeable
```

You should see:

```json
{
  "mergeable": "BLOCKED"
}
```

The only way to unblock it is to update the branch:

```bash
git merge main
git push origin feature/my-feature
gh pr merge --squash
```

### Exercise 2.3: Required vs. Optional Workflows

**Task:** Create two workflows in the same repository. Make one required (`required-ci.yml`) and the other optional (`optional-lint.yml`). Verify that a PR can merge when the optional workflow fails but the required one passes.

**Hint:** Only configure the required workflow's check name in the branch protection `contexts`. Leave the optional workflow's check name out of the list entirely.

**Solution:**

Create `.github/workflows/required-ci.yml`:

```yaml
name: Required CI
on: [pull_request]
jobs:
  required-build:
    name: Required Build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "This check is required to merge"
```

Create `.github/workflows/optional-lint.yml`:

```yaml
name: Optional Lint
on: [pull_request]
jobs:
  optional-lint:
    name: Optional Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: exit 1
```

Configure protection to require only `"Required Build"`:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": ["Required Build"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

Now create a PR. The `Optional Lint` check will fail (it always exits with 1), but because it is not in the required contexts list, the merge button will still be green as long as `Required Build` passes. Open the PR page in a browser to see the contrast: one red X (optional) and one green check (required) -- but the merge button is still clickable.

### Exercise 2.4: Pending vs. Failed Observation

**Task:** Write a workflow that intentionally sleeps for 60 seconds before completing. Push a commit, then immediately check the PR mergeability while the check is still pending. Watch the mergeability transition from `UNKNOWN` (pending) to `BLOCKED` (failure) or `MERGEABLE` (success).

**Hint:** Use `gh pr view --json mergeable` in a loop. The `--watch` flag on `gh run` can help track the workflow, but mergeability polling needs the API.

**Solution:**

Create `.github/workflows/slow-check.yml`:

```yaml
name: Slow Check
on: [pull_request]
jobs:
  slow:
    name: Slow Validation
    runs-on: ubuntu-latest
    steps:
      - run: sleep 60
      - name: Fail half the time
        run: |
          if [ $(( RANDOM % 2 )) -eq 0 ]; then
            echo "Check passed"
          else
            echo "Check failed"
            exit 1
          fi
```

Push the workflow, create a branch with a trivial change, and open a PR:

```bash
git checkout -b test/slow-check
echo "change" > test-file.txt
git add -A
git commit -m "Test slow check"
git push origin test/slow-check
gh pr create --base main --head test/slow-check --title "Slow check test" --body "Observing merge state transitions"
```

Add `"Slow Validation"` to the required checks:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": ["Slow Validation"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

Poll the mergeability in a loop:

```bash
for i in $(seq 1 15); do
  gh pr view --json mergeable
  sleep 5
done
```

You will observe:

```
# Initial state (check still queued/in_progress)
{ "mergeable": "UNKNOWN" }

# After check completes with failure
{ "mergeable": "CONFLICTING" }
```

Or:

```
# After check completes with success
{ "mergeable": "MERGEABLE" }
```

## Tier 3 -- Advanced

### Exercise 3.1: Multi-Stage Required Checks

**Task:** Design a pipeline where unit tests and integration tests are separate required checks. The unit test job must complete before the integration test job starts. If unit tests fail, the integration tests should not run at all. Both must be in the required status checks list.

**Hint:** Use the `needs:` keyword to sequence jobs. The integration test job should have `needs: [unit-tests]`. Configure the branch protection to require both `"Unit Tests"` and `"Integration Tests"`.

**Solution:**

Create `.github/workflows/multi-stage.yml`:

```yaml
name: Multi-Stage Checks

on:
  pull_request:
    branches: [main]

jobs:
  unit-tests:
    name: Unit Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - name: Run unit tests
        run: |
          echo "Running unit tests..."
          # Simulate unit tests
          echo "All unit tests passed"

  integration-tests:
    name: Integration Tests
    needs: [unit-tests]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
      - name: Run integration tests
        run: |
          echo "Running integration tests..."
          # Integration tests only run if unit tests passed
          echo "All integration tests passed"
```

Configure branch protection to require both check names:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["Unit Tests", "Integration Tests"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

Create a PR and confirm both checks appear in the UI. The integration tests will show a gray "Expected -- Waiting..." status until the unit tests complete. If unit tests fail, integration tests never start and the merge is blocked.

### Exercise 3.2: Matrix Strategy with Individual Required Checks

**Task:** Create a workflow that tests across a matrix of Node.js versions (16, 18, 20). Each matrix entry must appear as an individual required check. Configure branch protection to require all three matrix variant checks by name. Verify that a failure in any single matrix variant blocks the merge.

**Hint:** The check run names will be `"Node Tests (16)"`, `"Node Tests (18)"`, and `"Node Tests (20)"` when the job `name` is `"Node Tests"` and the matrix key is `node`. You must list all three names in the `contexts` array.

**Solution:**

Create `.github/workflows/matrix-checks.yml`:

```yaml
name: Matrix Checks

on:
  pull_request:
    branches: [main]

jobs:
  node-tests:
    name: Node Tests
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [16, 18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - name: Run tests on Node ${{ matrix.node }}
        run: |
          echo "Testing on Node ${{ matrix.node }}..."
          # Simulate a test that fails only on Node 20
          if [ "${{ matrix.node }}" = "20" ]; then
            echo "Test failed on Node 20"
            exit 1
          fi
          echo "Tests passed on Node ${{ matrix.node }}"
```

Configure branch protection to require all three:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": [
      "Node Tests (16)",
      "Node Tests (18)",
      "Node Tests (20)"
    ]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

Open a PR and check the merge state. You should see:
- `Node Tests (16)` -- green check
- `Node Tests (18)` -- green check
- `Node Tests (20)` -- red X
- Merge button -- blocked

Modify the workflow to make Node 20 pass, push again, and confirm the merge is now allowed.

### Exercise 3.3: Dynamic Check Name Pattern Matching

**Task:** Implement a merge gate that uses a naming convention for checks. Design a workflow where the check run name includes the module or service name (e.g., `"Check: auth"`, `"Check: api"`, `"Check: web"`). Configure branch protection to require any check that matches the pattern `"Check: *"`. Since GitHub does not support wildcard patterns for required checks, you must list every expected check name explicitly. Write a script that discovers the active check names from a running workflow and dynamically updates the branch protection configuration.

**Hint:** Use `gh run view --log` or the check runs API to extract the actual check run names after a workflow run completes, then use `gh api` to update the protection rule with the discovered names.

**Solution:**

Create `.github/workflows/dynamic-checks.yml`:

```yaml
name: Dynamic Checks

on:
  pull_request:
    branches: [main]

jobs:
  module-checks:
    name: Check
    runs-on: ubuntu-latest
    strategy:
      matrix:
        module: [auth, api, web, worker]
    steps:
      - uses: actions/checkout@v4
      - name: Check ${{ matrix.module }}
        run: |
          echo "Checking module: ${{ matrix.module }}"
          # Simulate a test
          sleep 5
          echo "Module ${{ matrix.module }} check passed"
```

Write a discovery script `discover-and-protect.sh`:

```bash
#!/bin/bash
set -euo pipefail

# discover-and-protect.sh
# Usage: ./discover-and-protect.sh <owner/repo>
# Discovers all check run names from the latest failed run
# and configures branch protection with those names

REPO="$1"

echo "Fetching latest check runs for HEAD commit..."

# Get check run names from the latest commit on main
CHECK_NAMES=$(gh api "repos/$REPO/commits/HEAD/check-runs" \
  --jq '[.check_runs[].name] | unique')

echo "Discovered checks: $CHECK_NAMES"

if [ "$(echo "$CHECK_NAMES" | jq 'length')" -eq 0 ]; then
  echo "No check runs found. Push a commit to trigger a workflow first."
  exit 1
fi

echo "Configuring branch protection with $CHECK_NAMES..."

# Build the payload with discovered check names
PAYLOAD=$(cat << PAYLOADEOF
{
  "required_status_checks": {
    "strict": true,
    "contexts": $CHECK_NAMES
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
PAYLOADEOF
)

gh api "repos/$REPO/branches/main/protection" \
  --method PUT \
  --input - <<< "$PAYLOAD"

echo "Done. Branch protection now requires: $CHECK_NAMES"
```

Run the script:

```bash
chmod +x discover-and-protect.sh
./discover-and-protect.sh myorg/my-repo
```

This approach is useful for teams that use matrix strategies or dynamically generated job names, where manually maintaining the list of required checks in the protection rule would be error-prone. You can run this script as part of an administrative workflow on a schedule or after significant changes to the CI pipeline.

## Summary of What You Practiced

| Exercise | Skill | API Used |
|----------|-------|----------|
| 1.1 View checks | Reading protection config | `GET /branches/{branch}/protection` |
| 1.2 Add check via UI | Manual configuration | Browser Settings |
| 1.3 Observe check transitions | Understanding check lifecycle | `GET /commits/{ref}/check-runs` |
| 1.4 Remove a check | Editing protection config | `PUT /branches/{branch}/protection` |
| 2.1 Scripted configuration | Automation | `PUT` with JSON payload |
| 2.2 Strict mode | Understanding up-to-date requirement | `strict: true` |
| 2.3 Required vs. optional | Selective enforcement | Partial contexts list |
| 2.4 Pending vs. failed | Observing merge blocking states | `PR mergeable` field |
| 3.1 Multi-stage checks | Sequential required jobs | `needs:` keyword |
| 3.2 Matrix required checks | Per-variant enforcement | Matrix check names |
| 3.3 Dynamic check discovery | Automated protection updates | Scripted API calls |
