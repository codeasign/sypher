A developer pushes a last-minute fix on a Friday afternoon. The fix looks safe -- just a one-line change to a configuration value. A teammate reviews it, approves, and clicks "Merge." Ten minutes later, the monitoring alert goes off: production is returning 500 errors for every request.

What happened? The one-line "fix" accidentally set a cache TTL to zero milliseconds instead of zero minutes. The unit test that would have caught this was never run. The reviewer did not catch it. There was no automated safety net between approval and merge. The PR sailed through because the only requirement was a human thumb -- not a green check mark from the test suite.

This story plays out in teams of every size. The fix is not to review harder; the fix is to automate the gate. Required status checks are that gate.

## The Concept

A required status check is a condition you configure on a protected branch that blocks merging until a specified check or checks report a successful conclusion. Until every required check passes, the merge button stays gray.

Think of it as a ride safety system at an amusement park. Before the ride starts, sensors check that every harness is locked, every gate is closed, and the track is clear. If any sensor reports a fault, the ride does not start -- regardless of whether the operator thinks everything looks fine. The sensors do not replace the operator; they enforce the preconditions mechanically. Required status checks work the same way: they mechanically enforce that your automated validation passes before code merges.

## Branch Protection API Parameters

The `required_status_checks` object in the GitHub Branch Protection API controls which checks are required and whether the branch must be up to date. Here are the key parameters:

| API Parameter | Type | Required | Description |
|---|---|---|---|
| `required_status_checks.strict` | boolean | Yes | When `true`, the branch must be up to date with the base branch before merging |
| `required_status_checks.contexts` | array of strings | Yes | List of check names that must pass. Each name must match a check run name or commit status name exactly |
| `required_status_checks.checks` | array of objects | Yes (modern) | Alternative to `contexts` with `context` and `app_id` fields for finer-grained matching |
| `required_status_checks.checks[].context` | string | Yes (modern) | The check name to require |
| `required_status_checks.checks[].app_id` | integer | No | Restrict to a specific GitHub App that created the check |
| `enforce_admins` | boolean | No | When `true`, rules apply to repository administrators too |
| `required_pull_request_reviews` | object | No | Add reviewer approval requirements alongside status checks |

### Strict vs. Loose Mode

| Mode | Behavior | Use Case |
|---|---|---|
| **Loose** (`strict: false`) | The branch only needs to pass required checks at the time the checks were last evaluated. The base branch can diverge after the checks passed and the PR can still merge. | Teams who want to move fast and accept that the base branch may change under the PR. |
| **Strict** (`strict: true`) | The branch must be up to date with the base branch AND pass required checks. If the base branch gets new commits, the PR branch must be updated (merged or rebased) and checks must re-run. | Teams who want to guarantee that every merge has been validated against the latest base branch. |

## Check Run vs. Commit Status

GitHub has two systems for reporting results on commits: commit statuses (legacy) and check runs (modern). Required status checks can enforce both, but the mechanics differ.

| Aspect | Commit Status | Check Run |
|---|---|---|
| API endpoint | `/repos/{owner}/{repo}/statuses/{sha}` | `/repos/{owner}/{repo}/check-runs` |
| Created by | Anyone with write access via the Statuses API | GitHub Actions, Apps, or the Checks API |
| State values | `pending`, `success`, `error`, `failure` | `queued`, `in_progress`, `completed` |
| Conclusion values | See state column | `success`, `failure`, `neutral`, `cancelled`, `timed_out`, `action_required`, `skipped` |
| Rich output | URL and description only | Annotations, images, markdown summary |
| Required check name | Matches the status context string | Matches the check run name |
| Retryable | No | Yes (re-run from the UI) |

### Check Run Conclusion Values

When a check run completes, it reports a conclusion. Required status checks evaluate this conclusion to determine pass or fail:

| Conclusion | Blocking? | Meaning |
|---|---|---|
| `success` | No | All checks passed. Merge allowed. |
| `failure` | Yes | One or more assertions failed. Merge blocked. |
| `neutral` | No | Check ran but neither passed nor failed (e.g., skipped due to condition). Merge allowed. |
| `cancelled` | Yes | The workflow run was cancelled before completion. Merge blocked. |
| `timed_out` | Yes | The check exceeded its timeout. Merge blocked. |
| `action_required` | Yes | The check needs manual intervention (review app, etc.). Merge blocked. |
| `skipped` | No | The check was explicitly skipped. Merge allowed. |

## How Status Check Names Work

The name of a check run or commit status is the string that appears in the GitHub UI next to the checkmark or red X. When you configure a required status check, you specify these names exactly.

For GitHub Actions workflows, the check run name follows this pattern:

- **Single job workflow:** Defaults to the job name(s) defined in the workflow.
- **Named job:** The job ID or `name` becomes the check run name.
- **Matrix strategy:** Each matrix combination creates a separate check run with the job name and matrix values appended.

```yaml
# Example: check run names for this workflow
jobs:
  test:
    name: Unit Tests
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [16, 18, 20]
    steps:
      - run: npm test
```

The above matrix produces three check runs named `"Unit Tests (16)"`, `"Unit Tests (18)"`, and `"Unit Tests (20)"`. If you require a single status check named `"Unit Tests"`, it will never match because each run has matrix-suffixed names. You must require each matrix entry individually or use a workflow-level check suite.

### Check Suite vs. Check Run

A **check suite** is a collection of check runs created by one GitHub App for one commit. When GitHub Actions runs a workflow, it creates a check suite containing one check run per job. You can require the entire suite to pass by requiring each individual check run -- there is no single "check suite" required status. The suite passes when all its runs pass.

## How It Actually Works -- Step by Step

### Step 1: A commit is pushed to a PR branch

The developer pushes a commit to a branch that has an open pull request targeting a protected branch (e.g., `main`). GitHub receives the push and records the new commit SHA. At this point, the PR is not yet blocked or allowed to merge -- GitHub has just noted that something changed.

### Step 2: GitHub delivers a webhook event

GitHub's event bus processes the push and triggers any workflows with matching `on:` conditions in their workflow YAML files. Common triggers for PR checks include:

```yaml
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]
```

The event system also updates the PR metadata: the "mergeable" flag is reset to `null` (unknown) because the commit changed and the previous check results are no longer valid.

### Step 3: The workflow runs on a runner

GitHub Actions provisions a runner (or picks an idle one), checks out the code at the commit SHA, and executes each step defined in the workflow. Steps run sequentially within a job, and jobs run in parallel unless `needs:` declares dependencies. Each job -- and each matrix variant of a job -- produces a check run.

```yaml
steps:
  - uses: actions/checkout@v4
  - name: Install dependencies
    run: npm ci
  - name: Run lint
    run: npm run lint
  - name: Run tests
    run: npm test
```

### Step 4: Check runs are created and updated

As each job executes, GitHub Actions reports check run status back to the repository via the Checks API. The sequence is:

1. Check run created with status `queued`
2. Check run updated to status `in_progress` when the job starts
3. Check run updated to status `completed` with a conclusion when the job finishes

The conclusion value (`success`, `failure`, `cancelled`, etc.) is what the branch protection rule evaluates.

### Step 5: Branch protection re-evaluates mergeability

Each time a check run completes, GitHub's branch protection system evaluates the PR against its configured rules. For required status checks, the logic is:

1. Gather all required check names from the branch protection configuration
2. Find the latest check run or commit status with each required name for the PR's head commit SHA
3. Check the conclusion of each required check
4. If all conclusions are `success`, `neutral`, or `skipped` -- set mergeable to `true`
5. If any conclusion is `failure`, `cancelled`, `timed_out`, or `action_required` -- set mergeable to `false`
6. If any check is still `queued` or `in_progress` -- set mergeable to `null` (pending)

The PR's merge button reflects this state in real time: it shows a green "Merge" button when all checks pass, a gray button with "Merging is blocked" when checks fail, and a spinning indicator when checks are pending.

### Step 6: The PR owner sees the result

On the PR page, the merge box at the bottom shows the status of required checks along with an action. The developer can click "Update branch" to pull in base branch changes (needed for strict mode), re-run failed checks, or view logs for debugging.

```
## Merge box states:

  All checks passed  ──>  Green "Merge pull request" button
  Some checks failed ──>  Gray button: "Required checks are failing"
  Checks pending     ──>  Gray button: "Waiting for status to be reported"
  Strict mode stale ──>  Gray button: "This branch is out of date"
```

## When to Use Required Status Checks

| Scenario | Recommendation |
|---|---|
| `main` or `master` branch | Always enable. No code should reach production without passing validation. |
| `release/*` branches | Always enable. Release branches should have the same or stricter gates as main. |
| `develop` or `integration` branches | Always enable. Integration branches accumulate work from multiple PRs and should be validated. |
| `feature/*` branches | Optional. Feature branches are short-lived and may not have a CI workflow configured yet. |
| Personal forks | Not applicable. Branch protection only applies to the target repository, not forks. |
| `gh-pages` or static content branches | Optional. If the branch only contains generated content, tests may not apply. |

## Command Reference: `gh` API for Branch Protection

You can configure required status checks from the command line using `gh api` against the Branch Protection API:

```bash
# Set required status checks on main branch (loose mode)
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": false,
    "contexts": ["test (16)", "test (18)", "test (20)", "lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

```bash
# Set required status checks with strict mode (branch must be up to date)
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["Unit Tests"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

```bash
# View current required status checks
gh api repos/:owner/:repo/branches/main/protection \
  --jq '.required_status_checks'
```

```bash
# Remove required status checks (set empty contexts)
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_status_checks": null,
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

## Common Status Check Context Names

When configuring required status checks, you need to know the exact name of the check. Here are common patterns:

| Source | Check Name Example | Notes |
|---|---|---|
| GitHub Action job | `build` | Uses the job ID if no `name` set |
| GitHub Action named job | `Build and Test` | Uses the job's `name` field |
| GitHub Action matrix job | `Test (ubuntu-latest, 18)` | Job name with matrix values in parentheses |
| Third-party CI (Jenkins) | `continuous-integration/jenkins/pr-merge` | Determined by the CI integration |
| Commit status | `ci/circleci: build` | Context string from the status API |
| CodeQL | `CodeQL (javascript)` | Auto-created by CodeQL analysis |
| Lint action | `lint` | Simple job name |
| Required (legacy) | `continuous-integration/travis-ci/push` | Travis CI context format |

## When Not to Use Required Status Checks

- **Solo projects or personal repos:** You are the only contributor and can enforce your own discipline. The overhead of configuring branch protection is unnecessary until you collaborate.
- **Experimental branches:** Branches named `wip/` or `experiment/` where you intentionally push broken code. Use branch name patterns to exclude these from protection.
- **Automated merge bots:** Some bots (dependabot, renovate) create PRs that may not need the same validation as human-authored changes. You can exempt dependency update PRs by using conditional workflows that produce `neutral` conclusions for irrelevant checks.
- **Documentation-only changes:** If your repo separates code from docs, you might skip checks for PRs that only touch `.md` files. This is achieved by configuring workflow-level path filters, not by disabling required status checks entirely.

The rule of thumb is simple: if you would be unhappy finding out that broken code was merged, you need required status checks. If you truly do not care about the branch's contents -- because it is ephemeral, personal, or auto-generated -- you can skip them.
