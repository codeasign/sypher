## Tier 1 -- Beginner

### Exercise 1

**Task:** Create a CODEOWNERS file in a new repository that assigns a single user as the default owner for all files. Verify the file is valid by committing it and checking that GitHub can parse it.

<details>
<summary>Hint</summary>
Create a `.github/CODEOWNERS` file with a single line: `* @your-username`. Commit it to the default branch. GitHub validates the syntax at commit time -- if the push succeeds, the file is valid.
</details>

```bash
mkdir -p ~/codeowners-beginner-1 && cd ~/codeowners-beginner-1
git init
gh repo create codeowners-beginner-1 --public --clone --push --source=.
mkdir -p .github src
echo "hello" > src/main.py
cat > .github/CODEOWNERS << 'EOF'
* @your-username
EOF
git add . && git commit -m "Add CODEOWNERS with default owner"
git push origin main
```

To verify, open a PR and check the reviewer is auto-requested:

```bash
git switch -c test-ownership
echo "new code" >> src/main.py
git add src/main.py && git commit -m "Test" && git push origin test-ownership
gh pr create --title "Test ownership" --body "Verify default owner" --base main
gh pr view 1 --json reviewRequests
```

### Exercise 2

**Task:** Add a team as the owner for a specific directory. Create a repository with a `src/backend/` directory, assign `@your-org/backend-team` as the owner, and verify that a PR changing files in that directory requests the team.

<details>
<summary>Hint</summary>
Add a line to `.github/CODEOWNERS` like `/src/backend/ @your-org/backend-team`. Teams must have read-level access to the repository.
</details>

```bash
mkdir -p ~/codeowners-beginner-2 && cd ~/codeowners-beginner-2
git init && gh repo create codeowners-beginner-2 --public --clone --push --source=.
mkdir -p src/backend .github
echo "def handler(): pass" > src/backend/api.py
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/backend/ @your-org/backend-team
EOF
git add . && git commit -m "Add backend team ownership" && git push origin main
```

If you do not have an organisation with teams, use a second GitHub username:

```bash
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/backend/ @second-username
EOF
git add .github/CODEOWNERS && git commit -m "Use second user as backend owner" && git push origin main
```

Create a PR that changes the backend file:

```bash
git switch -c backend-change
echo "def new_handler(): pass" >> src/backend/api.py
git add src/backend/api.py && git commit -m "Add new handler" && git push origin backend-change
gh pr create --title "Backend change" --body "Adds new handler" --base main
gh pr view 1 --json reviewRequests
```

```bash
cat > .github/CODEOWNERS << 'EOF'
* @your-username
EOF
git add .github/CODEOWNERS && git commit -m "Fix CODEOWNERS" && git push origin main
```

### Exercise 4

**Task:** Remove a code owner from the CODEOWNERS file. Remove a specific directory rule so that ownership falls back to the default owner.

<details>
<summary>Hint</summary>
Delete the line for the specific directory from the CODEOWNERS file. The fallback `*` rule then applies to files in that directory.
</details>

```bash
cd ~/codeowners-beginner-1
mkdir -p src/legacy && echo "old code" > src/legacy/old.py
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/legacy/ @legacy-team
/src/backend/ @backend-team
EOF
git add . && git commit -m "Add legacy team ownership" && git push origin main
```

Now remove the legacy team rule:

```bash
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/backend/ @backend-team
EOF
git add .github/CODEOWNERS && git commit -m "Remove legacy team, falls back to default" && git push origin main
```

Verify by creating a PR that changes a legacy file:

```bash
git switch -c legacy-change
echo "updated code" >> src/legacy/old.py
git add src/legacy/old.py && git commit -m "Update legacy" && git push origin legacy-change
gh pr create --title "Legacy change" --body "No specific owner" --base main
gh pr view 1 --json reviewRequests
```

The review request now shows `@your-username` (default owner) instead of `@legacy-team`.

## Tier 2 -- Intermediate

### Exercise 1

**Task:** Set up multi-directory ownership in a repository with three directories: `src/frontend/`, `src/backend/`, and `src/shared/`. Assign three different users. Verify that a PR changing a file in each directory requests all three owners.

<details>
<summary>Hint</summary>
Create a CODEOWNERS file with three directory rules. Create a branch that modifies one file in each directory. Open a PR and check the review requests.
</details>

```bash
mkdir -p ~/codeowners-inter-1 && cd ~/codeowners-inter-1
git init && gh repo create codeowners-inter-1 --public --clone --push --source=.
mkdir -p src/frontend src/backend src/shared .github
echo "frontend" > src/frontend/app.js && echo "backend" > src/backend/api.py
echo "shared" > src/shared/utils.py
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/frontend/ @frontend-user
/src/backend/ @backend-user
/src/shared/ @shared-user
EOF
git add . && git commit -m "Add multi-directory ownership" && git push origin main
```

Create a branch that changes all three directories:

```bash
git switch -c multi-dir-change
echo "update" >> src/frontend/app.js && echo "update" >> src/backend/api.py
echo "update" >> src/shared/utils.py
git add . && git commit -m "Update all directories" && git push origin multi-dir-change
gh pr create --title "Multi-directory" --body "Changes across all teams" --base main
gh pr view 1 --json reviewRequests
```

Expected output -- three review requests:

```json
{
  "reviewRequests": [
    {"requestedReviewer": {"__typename": "User", "name": "frontend-user"}},
    {"requestedReviewer": {"__typename": "User", "name": "backend-user"}},
    {"requestedReviewer": {"__typename": "User", "name": "shared-user"}}
  ]
}
```

### Exercise 2

**Task:** Combine CODEOWNERS with required reviewers in branch protection. Enable `require_code_owner_reviews` and `required_approving_review_count: 2`. Verify that a PR needs two approvals from code owners before it can merge.

<details>
<summary>Hint</summary>
Use `gh api` to set up branch protection with `required_approving_review_count: 2`. Create a PR, approve it once, try to merge, approve it again, then merge.
</details>

```bash
cd ~/codeowners-inter-1
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_pull_request_reviews": {
    "required_approving_review_count": 2,
    "require_code_owner_reviews": true,
    "dismiss_stale_reviews": true
  },
  "enforce_admins": true
}
EOF
```

Create a PR and try to merge with only one approval:

```bash
git switch -c test-two-approvals
echo "change" >> src/frontend/app.js
git add src/frontend/app.js && git commit -m "Test two-approval" && git push origin test-two-approvals
gh pr create --title "Two approvals needed" --body "Test" --base main
gh pr review 1 --approve --body "First approval"
gh pr merge 1 --squash
```

The merge is rejected because two approvals are required. The same user cannot approve twice. Reduce the count back to 1 to proceed:

```bash
gh api repos/:owner/:repo/branches/main/protection \
  --method PUT \
  --input - << 'EOF'
{
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "require_code_owner_reviews": true,
    "dismiss_stale_reviews": true
  },
  "enforce_admins": true
}
EOF
gh pr review 1 --approve --body "Approved"
gh pr merge 1 --squash --delete-branch
```

### Exercise 4

**Task:** Test file-level ownership overrides. Create a directory `src/config/` owned by `@core-team`, but make a specific file `src/config/secrets.yml` owned by `@security-team`. Verify that a PR changing `secrets.yml` requests `@security-team` instead of `@core-team`.

<details>
<summary>Hint</summary>
Place the more specific file-level rule after the directory rule. The last matching rule wins at the same specificity level. Use a full path like `/src/config/secrets.yml` for the specific file.
</details>

```bash
mkdir -p ~/codeowners-inter-3 && cd ~/codeowners-inter-3
git init && gh repo create codeowners-inter-3 --public --clone --push --source=.
mkdir -p src/config .github
echo "api_key: test" > src/config/secrets.yml && echo "app_name: myapp" > src/config/app.yml
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/config/ @core-team
/src/config/secrets.yml @security-team
EOF
git add . && git commit -m "Add file-level override" && git push origin main
```

Verify the override works:

```bash
git switch -c secret-change
echo "new_key: value" >> src/config/secrets.yml
git add src/config/secrets.yml && git commit -m "Update secrets" && git push origin secret-change
gh pr create --title "Secret update" --body "Test override" --base main
gh pr view 1 --json reviewRequests
```

The security team should be the reviewer, not the core team. Now verify the directory rule still applies to other files:

```bash
git switch -c config-change
echo "version: 2" >> src/config/app.yml
git add src/config/app.yml && git commit -m "Update app config" && git push origin config-change
gh pr create --title "Config update" --body "Test directory ownership" --base main
gh pr view 2 --json reviewRequests
```

The core team should be the reviewer for `app.yml`.

## Tier 3 -- Advanced

### Exercise 1: Design a CODEOWNERS Strategy for a Monorepo

**Scenario:** Your organisation has a monorepo with five teams and overlapping ownership requirements:

- `@frontend-team` owns `src/frontend/**`
- `@backend-team` owns `src/backend/**` and `src/api/**`
- `@platform-team` owns `src/**/*.config.*` (any config file in src/)
- `@devops-team` owns `deploy/**`, `Dockerfile`, and `docker-compose.yml`
- `@docs-team` owns `docs/**` and `**/*.md`
- `@security-team` must review any file named `secrets.*` or `*.key` at any depth
- `@core-team` is the default owner for everything else

The challenge: when a change touches `src/backend/config.yml`, both `@backend-team` and `@platform-team` should be requested. When a change touches `src/frontend/secrets.yml`, both `@frontend-team` and `@security-team` should be requested.

**Task:** Write the complete CODEOWNERS file that implements this policy. Create the directory structure and representative files. Verify with a PR that changes a config file in the backend directory and confirm both teams are requested.

<details>
<summary>Hint</summary>
Order matters in CODEOWNERS. Place general rules first and specific overrides last. Both `src/backend/` and `src/**/*.config.*` can match the same file -- each matched rule adds its owner as a required reviewer.
</details>

```bash
mkdir -p ~/codeowners-adv-1 && cd ~/codeowners-adv-1
git init && gh repo create codeowners-adv-1 --public --clone --push --source=.
mkdir -p src/frontend src/backend src/api deploy docs .github
echo "frontend" > src/frontend/app.js && echo "backend" > src/backend/api.py
echo "api" > src/api/routes.py && echo "config" > src/backend/config.yml
echo "config" > src/frontend/config.yml && echo "secrets" > src/frontend/secrets.yml
echo "deploy" > deploy/docker-compose.yml && echo "docs" > docs/readme.md
touch Dockerfile
cat > .github/CODEOWNERS << 'EOF'
* @core-team
/docs/ @docs-team
**/*.md @docs-team
/deploy/ @devops-team
/Dockerfile @devops-team
/docker-compose.yml @devops-team
**/secrets.* @security-team
**/*.key @security-team
src/**/*.config.* @platform-team
/src/backend/ @backend-team
/src/api/ @backend-team
/src/frontend/ @frontend-team
EOF
git add . && git commit -m "Add monorepo CODEOWNERS strategy" && git push origin main
```

Create a PR that changes a config file in `src/backend/`:

```bash
git switch -c backend-config-change
echo "new_config: true" >> src/backend/config.yml
git add src/backend/config.yml && git commit -m "Update backend config" && git push origin backend-config-change
gh pr create --title "Backend config update" --body "Changes backend/config.yml" --base main
gh pr view 1 --json reviewRequests
```

The PR requests both `@backend-team` (for `/src/backend/`) and `@platform-team` (for `src/**/*.config.*`). This is the overlapping ownership pattern working correctly.

### Exercise 2: Implement a CODEOWNERS-Based CI Check

**Scenario:** You want a CI workflow that checks every PR to ensure the PR has approval from the correct code owner, fails the workflow if not, and posts a comment listing which owners still need to approve.

**Task:** Create a GitHub Actions workflow file `.github/workflows/codeowners-check.yml` that implements this check. Trigger on `pull_request` events, use the GitHub API to check review status, and post a status check result.

<details>
<summary>Hint</summary>
Use `pull_request` trigger. Use `gh api` to fetch PR reviews. Use `jq` to parse the JSON responses. Write a step that checks whether the approvals match the expected CODEOWNERS.
</details>

```bash
mkdir -p ~/codeowners-adv-2 && cd ~/codeowners-adv-2
git init && gh repo create codeowners-adv-2 --public --clone --push --source=.
mkdir -p .github/workflows src/backend
echo "def handler(): pass" > src/backend/api.py
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/backend/ @backend-user
EOF
```

Create the CI workflow:

```bash
cat > .github/workflows/codeowners-check.yml << 'EOF'
name: CODEOWNERS Audit Check
on:
  pull_request:
    types: [opened, synchronize, reopened, review_requested]
permissions:
  contents: read
  pull-requests: write
  checks: write
jobs:
  check-codeowners:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check CODEOWNERS approval
        id: check
        run: |
          PR_NUM="${{ github.event.pull_request.number }}"
          REPO="${{ github.repository }}"
          CHANGED_FILES=$(gh api repos/$REPO/pulls/$PR_NUM/files \
            --jq '[.[].filename] | join(" ")')
          echo "Changed files: $CHANGED_FILES"
          REVIEWS=$(gh api repos/$REPO/pulls/$PR_NUM/reviews \
            --jq '[.[] | select(.state == "APPROVED") | .user.login] | unique')
          echo "Approved by: $REVIEWS"
          echo "approved_by=$REVIEWS" >> $GITHUB_OUTPUT
      - name: Generate report
        run: |
          echo "## CODEOWNERS Audit Report" >> $GITHUB_STEP_SUMMARY
          echo "PR #${{ github.event.pull_request.number }}" >> $GITHUB_STEP_SUMMARY
          echo "Approved by: ${{ steps.check.outputs.approved_by }}" >> $GITHUB_STEP_SUMMARY
EOF
git add . && git commit -m "Add CODEOWNERS audit workflow" && git push origin main
```

### Exercise 3: Create a CODEOWNERS Audit Workflow

**Scenario:** Over time, teams change, people leave, and CODEOWNERS rules become stale. You need an automated audit that scans the CODEOWNERS file weekly, reports unowned files and stale owners, and creates a GitHub Issue with the report.

**Task:** Create a GitHub Actions workflow that runs on a schedule (weekly) and performs the audit. Parse the CODEOWNERS file, check each owner against the GitHub API, and report any issues.

<details>
<summary>Hint</summary>
Use a `schedule` trigger with cron. Parse the CODEOWNERS file with `grep` to extract owner references. Use `gh api users/OWNER` and `gh api orgs/ORG/teams/TEAM` to verify each owner exists. Use `gh issue create` to report findings.
</details>

```bash
mkdir -p ~/codeowners-adv-3 && cd ~/codeowners-adv-3
git init && gh repo create codeowners-adv-3 --public --clone --push --source=.
mkdir -p .github/workflows src/backend src/frontend deploy
echo "api" > src/backend/api.py && echo "app" > src/frontend/app.js
echo "docker" > deploy/docker-compose.yml
cat > .github/CODEOWNERS << 'EOF'
* @your-username
/src/backend/ @backend-team
/src/frontend/ @frontend-team
/deploy/ @devops-team
EOF
```

Create the audit workflow:

```bash
cat > .github/workflows/codeowners-audit.yml << 'EOF'
name: Weekly CODEOWNERS Audit
on:
  schedule:
    - cron: '0 9 * * 1'
  workflow_dispatch:
permissions:
  contents: read
  issues: write
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Parse CODEOWNERS
        id: parse
        run: |
          if [ ! -f .github/CODEOWNERS ]; then
            echo "No CODEOWNERS file found"
            exit 0
          fi
          OWNERS=$(grep -oP '@[\w-]+(/[\w-]+)?' .github/CODEOWNERS | sort -u)
          STALE_OWNERS=""
          for OWNER in $OWNERS; do
            NAME="${OWNER:1}"
            HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" \
              -H "Authorization: Bearer ${{ secrets.GITHUB_TOKEN }}" \
              "https://api.github.com/users/$NAME")
            if [ "$HTTP_CODE" -eq 200 ]; then
              echo "  $OWNER is valid"
            elif echo "$NAME" | grep -q '/'; then
              ORG=$(echo "$NAME" | cut -d/ -f1)
              TEAM=$(echo "$NAME" | cut -d/ -f2)
              HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" \
                -H "Authorization: Bearer ${{ secrets.GITHUB_TOKEN }}" \
                "https://api.github.com/orgs/$ORG/teams/$TEAM")
              [ "$HTTP_CODE" -ne 200 ] && STALE_OWNERS="$STALE_OWNERS $OWNER"
            else
              STALE_OWNERS="$STALE_OWNERS $OWNER"
            fi
          done
          echo "stale_owners=$STALE_OWNERS" >> $GITHUB_OUTPUT
      - name: Report stale owners
        run: |
          STALE="${{ steps.parse.outputs.stale_owners }}"
          if [ -n "$STALE" ]; then
            echo "## CODEOWNERS Audit Report" > report.md
            echo "**Audit Date:** $(date -u)" >> report.md
            echo "### Stale Owners Found" >> report.md
            echo "$STALE" >> report.md
            gh issue create --title "CODEOWNERS Audit: Stale owners detected" \
              --body "$(cat report.md)" --label "maintenance"
          else
            echo "All owners are valid."
          fi
EOF
git add . && git commit -m "Add weekly CODEOWNERS audit workflow" && git push origin main
```
