You will simulate the full fork and pull request workflow using two local repositories: one acts as the "upstream" original, the other as your fork. You will fork, clone, add a remote, create a branch, push changes, simulate a PR, and sync with upstream changes. This lab uses only local repos — no GitHub account needed.

## Setup

```bash
mkdir -p ~/fork-pr-lab
cd ~/fork-pr-lab
```

Create the "upstream" original repository:

```bash
mkdir upstream-repo
cd upstream-repo
git init --bare
cd ..
```

Clone the "upstream" to create a working copy with some content:

```bash
git clone upstream-repo upstream-working
cd upstream-working
git config user.name "Original Owner"
git config user.email "owner@example.com"
```

Add initial content and push:

```bash
echo "# Project" > README.md
echo "print('Hello')" > main.py
git add README.md main.py
git commit -m "Initial commit"
git push origin main
cd ..
```

Now the upstream repo exists with one commit on `main`.

## The Walkthrough

### Step 1: Fork the upstream (simulated by cloning to a new location)

In the real GitHub workflow, forking creates a server-side copy. Here we simulate it by cloning the upstream to a second location that represents "your fork":

```bash
git clone upstream-repo your-fork
cd your-fork
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Your fork has the same initial commit, and its `origin` points to the same upstream. Let us fix that:

```bash
git remote -v
```

Expected output — both fetch and push point to `upstream-repo`:

```
origin  .../upstream-repo (fetch)
origin  .../upstream-repo (push)
```

This simulates a GitHub fork where your fork's `origin` points to your copy. In this local setup, `origin` pointing to the upstream is fine — we will add the "original" as a separate remote.

### Step 2: Add the upstream remote

```bash
git remote add upstream ../upstream-repo
git remote -v
```

Expected output:

```
origin    .../upstream-repo (fetch)
origin    .../upstream-repo (push)
upstream  ../upstream-repo (fetch)
upstream  ../upstream-repo (push)
```

**Under the hood:** In a real GitHub fork, the `origin` URL would point to `github.com/your-username/repo.git` and `upstream` would point to `github.com/original-owner/repo.git`. The remote strategy is the same.

### Step 3: Create a feature branch and make changes

```bash
git switch -c add-feature
echo "def add(a, b):" >> main.py
echo "    return a + b" >> main.py
echo "print(add(2, 3))" >> main.py
git add main.py
git commit -m "Add add function"
git push origin add-feature
```

Expected output:

```
 * [new branch]      add-feature -> add-feature
```

Your fork now has a branch `add-feature` that the upstream does not have.

```
  upstream/main ──▶ Initial commit
  origin/add-feature ──▶ Initial commit ──▶ Add add function
```

**Checkpoint:** `git log --oneline --all --graph` shows both branches.

### Step 4: Open a pull request (simulated)

Since we are not on GitHub, we simulate the PR by generating a diff:

```bash
git switch main
git diff main..add-feature
```

Expected output — shows the changes that the PR would propose:

```
diff --git a/main.py b/main.py
index ... .. ...
--- a/main.py
+++ b/main.py
@@ -1 +1,5 @@
 print('Hello')
+def add(a, b):
+    return a + b
+print(add(2, 3))
```

This diff represents what a pull request would propose to merge. On GitHub, you would run `gh pr create` here, or use the web UI.

### Step 5: Simulate upstream accepting the PR (merge the change)

Switch to the upstream working copy and pull the change:

```bash
cd ../upstream-working
git fetch origin
git checkout main
git merge origin/add-feature
git push origin main
```

Expected output:

```
Updating abc1234..def5678
Fast-forward
 main.py | 4 ++++
 1 file changed, 4 insertions(+)
```

The upstream now includes the feature.

```
  upstream/main ──▶ Initial commit ──▶ Add add function
```

### Step 6: Sync your fork with upstream

Now that the upstream has new commits (the merged PR), your fork is behind:

```bash
cd ../your-fork
git checkout main
git fetch upstream
git merge upstream/main
git push origin main
```

Expected output:

```
Updating abc1234..def5678
Fast-forward
 main.py | 4 ++++
 1 file changed, 4 insertions(+)
```

Your fork's `main` now matches the upstream. Delete the stale feature branch:

```bash
git branch -d add-feature
git push origin --delete add-feature
```

**Under the hood:** Syncing a fork is just `git fetch upstream` + `git merge upstream/main` + `git push origin main`. The fork is not special — it is a regular Git repository with two remotes.

### Step 7: Handle diverged history (sync with rebase)

Create a scenario where both the upstream and your fork's feature branch advance:

```bash
cd ../upstream-working
echo "def multiply(a, b):" >> main.py
echo "    return a * b" >> main.py
git add main.py
git commit -m "Add multiply function"
git push origin main
```

Now in your fork, create a branch that starts from an older commit:

```bash
cd ../your-fork
git fetch upstream
git switch -c new-feature upstream/main~1  # Start from the commit BEFORE multiply
echo "def divide(a, b):" >> main.py
echo "    return a / b" >> main.py
git add main.py
git commit -m "Add divide function"
```

Sync with rebase instead of merge:

```bash
git rebase upstream/main
```

If there are no conflicts, the rebase succeeds and your feature sits on top of the latest upstream:

```bash
git log --oneline
```

Expected output — your "Add divide function" commit appears after "Add multiply function".

Push the rebased branch:

```bash
git push origin new-feature --force-with-lease
```

Force-push is needed because rebasing rewrote the commit history. `--force-with-lease` is safer than `--force` because it checks that your remote-tracking branch matches what you expect.

### Step 8: Open a PR via GitHub CLI (simulated)

If you had `gh` authenticated, the command would be:

```bash
gh pr create \
  --base upstream/main \
  --head your-fork:new-feature \
  --title "Add divide function" \
  --body "Adds divide functionality to the calculator."
```

To simulate, create a PR description file:

```bash
cat > PR_DESCRIPTION.md << 'EOF'
## Title: Add divide function
## Body:
Adds divide functionality to the calculator.

**Changes:**
- Added `divide(a, b)` function
- Returns `a / b`

**Testing:**
- Called `print(divide(10, 2))` to verify output is `5.0`
EOF
echo "PR simulation file created (would run: gh pr create --fill)"
```

## Common Errors at This Stage

### Error 1: `ERROR: Permission to original/repo.git denied to <you>`

You tried to push directly to the original repository instead of your fork. You do not have write access.

**Fix:** Push to your fork (`origin`), not `upstream`. Verify your remotes with `git remote -v`. If `origin` points to the original repo, you forked incorrectly — fork on GitHub first, then clone your fork.

### Error 2: `fatal: remote upstream already exists.`

You tried to add an `upstream` remote but one already exists from a previous configuration.

**Fix:** Verify the existing upstream URL with `git remote get-url upstream`. If it is wrong, update it: `git remote set-url upstream <correct-url>`. If you want to start fresh, remove it: `git remote remove upstream`.

### Error 3: `! [rejected] main -> main (non-fast-forward)` when syncing fork

Your fork's main branch has diverged from upstream because you committed directly to your fork's main instead of using a feature branch.

**Fix:** Use `git fetch upstream` + `git rebase upstream/main` on your local main, then `git push origin main --force-with-lease` to sync your fork. In the future, never commit directly to your fork's main — always use feature branches.

### Error 4: Pull request shows commits you did not intend

Your fork's main is behind upstream, and your PR branch includes commits from upstream that you did not write. This happens when you created the feature branch from a stale main.

**Fix:** Rebase your PR branch onto the latest upstream: `git fetch upstream && git rebase upstream/main`. Force-push to update the PR: `git push origin <branch> --force-with-lease`. The PR updates automatically to show only your commits.

## What You Just Learned

- A fork is a server-side copy of a repository under your account
- Add the original repo as `upstream` remote: `git remote add upstream <url>`
- Sync your fork with `git fetch upstream` + `git merge upstream/main` (or `git rebase`)
- Always create feature branches from an up-to-date main
- Push feature branches to your fork, then open a PR to the upstream
- After a PR is merged, delete the feature branch both locally and on your fork
- Use `--force-with-lease` (not `--force`) when force-pushing a rebased branch
