Your team has been working on a new feature for three weeks. Twenty-three commits, four approved PRs, and a full QA cycle. On Friday at 4:47 PM, a junior developer types `git push origin main --force`, accidentally overwriting the entire branch with an old local copy. The commit history is gone. The CI pipeline that ran five minutes ago is now pointing at a broken state. The team lead spends the next four hours reconstructing commits from local clones and Slack messages. The feature ships two days late. A single `git push --force` on an unprotected branch caused all of it.

## The Concept

Branch protection rules are a set of conditions that GitHub enforces before allowing any operation on a specified branch, preventing destructive actions like force pushes, unprotected merges, and direct pushes to critical branches. Think of it like a library checkout system: you cannot walk out of the building with a book unless you have a valid library card (the PR), the book has been scanned by the librarian (status checks pass), and the exit gate confirms you are authorized to borrow it (required reviews). The librarian does not care who you are — the gate checks the rules, not the person.

## Branch Protection API / YAML Keys

The following keys are available when configuring branch protection rules through the GitHub REST API (the `PUT /repos/{owner}/{repo}/branches/{branch}/protection` endpoint). The same settings are available in the GitHub UI under Settings > Branches > Add rule.

| Key | What It Does | Values | Example |
|-----|-------------|--------|---------|
| Key | What It Does | Values | Example |
|-----|-------------|--------|---------|
| `required_status_checks` | Require specific CI checks (e.g., "Tests / build") to pass before merging | Object with `strict` (bool) and `contexts` (array of strings) | `{ "strict": true, "contexts": ["continuous-integration"] }` |
| `strict_required_status_checks_commits` | Require the branch to be up to date with the base branch before merging | Boolean | `true` |
| `required_pull_request_reviews` | Require one or more PR reviews before merging | Object with `dismiss_stale_reviews`, `require_code_owner_reviews`, `required_approving_review_count` | `{ "required_approving_review_count": 2 }` |
| `dismiss_stale_reviews` | Automatically dismiss approving reviews when new commits are pushed | Boolean | `true` |
| `require_code_owner_reviews` | Require at least one review from a code owner (from CODEOWNERS file) | Boolean | `true` |
| `enforce_admins` | Apply all rules to administrators (bypass disabled) | Boolean | `true` |
| `restrictions` | Restrict who can push to the branch (users, teams, apps) | Object with `users`, `teams`, `apps` arrays | `{ "users": ["alice"], "teams": ["devops"] }` |
| `required_linear_history` | Require a linear commit history (no merge commits) | Boolean | `true` |
| `allow_force_pushes` | Allow force pushes to the branch (override protection) | Boolean or Object with `enabled` | `false` |
| `allow_deletions` | Allow deletion of the branch | Boolean | `false` |
| `required_conversation_resolution` | Require all PR comments to be resolved before merging | Boolean | `true` |
| `lock_branch` | Lock the branch, making it read-only | Boolean | `true` |

The `strict` field in `required_status_checks` is particularly important. When `strict` is `true`, GitHub checks that the PR branch is up to date with the base branch before allowing a merge. If the PR branch is behind, the PR must be updated (typically by merging or rebasing against the base branch) before the merge button becomes active. Without strict mode, a PR can be approved while the base branch has moved ahead, and the merge will combine stale code with the latest changes — potentially introducing conflicts or breaking the build.

The `required_approving_review_count` under `required_pull_request_reviews` can be set to any integer from 1 to 6. Higher values give you more defense against a single bad review but increase the coordination overhead. Most teams use 1 for development branches and 2 for production branches.

The `restrictions` key is different from the others — it restricts push access to specific users, teams, or apps. When set, only those entities can push to the branch, even through PRs. This is a hard lock: if you restrict `main` to only the `devops` team, no one else can push to it, regardless of their role in the organization. Use this sparingly — it can block automated tools and CI systems from pushing tags or releases.

## How It Actually Works — Step by Step

Branch protection is not a single check. GitHub enforces it through a pipeline of internal systems that evaluate every git operation against the configured rules. You configure the rules once, and the enforcement is automatic — no manual intervention required after setup.

### Step 1: GitHub Receives the Push or PR Event

When you push to a branch or open a pull request, GitHub's webhook receiver parses the event. For pushes, it checks whether the target branch has protection rules. For PRs, it checks the target branch of the PR.

The webhook receiver identifies the branch name, the actor (who pushed), and the operation type (push, force push, delete, merge). It then loads the protection rules for that branch from the repository settings database. This happens before any data is written to the repository — the push or PR creation is blocked until the protection check completes.

The receiver handles three distinct event types differently:

- **Push events (direct):** Checked synchronously. If the branch is protected, the push is rejected before any objects are stored. The sender sees the `GH006` error immediately.
- **Push events (force):** Same synchronous check, but evaluated against `allow_force_pushes`. If force pushes are denied, the entire pack is rejected.
- **Pull request events:** Checked asynchronously. The PR is created, but the mergeability is calculated separately (Step 3). The PR exists in an "unmergeable" state until the protection engine finishes evaluating it.

### Step 2: Check Run Evaluation

When a PR is opened or updated, GitHub creates check runs for each CI workflow in the repository. The protection engine monitors these check runs in real time. It does not block the PR immediately — it waits for all required status checks to complete. This is a non-blocking wait: the PR exists and can be reviewed, but the merge button stays disabled.

The engine compares the list of completed check runs against the `required_status_checks.contexts` list. Each entry in `contexts` is a string matching a check run name or a status context name. For example, if your CI workflow is named "CI" and has a job named "test," the context is `CI / test`.

A check run can be in one of four states: `queued`, `in_progress`, `completed_success`, or `completed_failure`. The merge button stays disabled until every required context reports `completed_success`. If any check reports `completed_failure`, the merge is blocked and the PR shows a red "Some checks were not successful" banner.

If you enabled `strict_required_status_checks_commits` (the strict mode for required status checks), the engine also checks whether the PR branch is behind the base branch. If the base branch has new commits that the PR branch does not have, the engine requires the PR branch to be updated (by merging or rebasing against the base branch) before it re-evaluates checks. This prevents merging outdated code into a branch that has moved forward.

Check runs are evaluated by name, not by workflow file. If you rename a job in your workflow YAML, the old check name disappears and the new check name appears. Any protection rule referencing the old name will fail with "Required status check not found" until you update the rule. This is a common cause of broken merges after renaming CI jobs.

### Step 3: Mergeability Calculation

The mergeability check is the core of branch protection. GitHub calculates a boolean `mergeable` state for every PR. This is not a single variable — it is a composite of every protection rule evaluated as a logical AND:

```
mergeable = (
  required_status_checks_passed AND
  required_reviews_count_met AND
  code_owner_reviewed_if_required AND
  conversation_resolved AND
  linear_history_eligible AND
  branch_up_to_date_if_strict
)
```

If any of these conditions is `false`, the merge button is disabled. The PR displays a red banner with the specific failing rule. GitHub does not evaluate these lazily — it recalculates `mergeable` on every push, every review submission, every check run completion, and every status update. This means the mergeability state is always current, never stale.

The `mergeable` state has three possible values: `MERGEABLE` (all rules pass), `UNMERGEABLE` (one or more rules fail), and `UNKNOWN` (the calculation is still in progress). When you open a PR, the state starts as `UNKNOWN` and transitions to `MERGEABLE` or `UNMERGEABLE` within a few seconds. If the PR has been open for longer than 30 seconds and still shows `UNKNOWN`, a background job may have failed — you can force a recalculation by closing and reopening the PR.

### Step 4: Atomic Update Enforcement

When you click the merge button, GitHub does not skip the rules. It performs an atomic check immediately before the merge commit is created. This prevents a race condition where a rule passes, then a new push or force push invalidates it before the merge completes.

The atomic update sequence is:

1. Lock the branch row in the database (prevents concurrent merges)
2. Re-evaluate all protection rules against the current state of the branch
3. If all rules pass, create the merge commit and update the ref
4. If any rule fails, abort with an error message and roll back any changes
5. Release the branch lock

The branch lock is held for the entire duration of the atomic check, typically 50 to 200 milliseconds. This is fast enough that concurrent merges are not significantly delayed, but long enough to prevent race conditions. If a CI check completes between the time you click merge and the atomic check runs, the merge is still valid — the atomic check uses the latest state.

This atomicity means that even if a developer times their click perfectly between two check run updates, the merge will be denied if the intervening change broke a rule. The merge button is a guarantee, not a suggestion — GitHub re-verifies everything at the last possible moment.

### Step 5: Force Push and Deletion Blocking

Force pushes and branch deletions are evaluated differently from merges. These operations are checked at the git protocol level, not at the PR level. When GitHub receives a `git push --force` or `git push --delete`, the receiving endpoint checks the branch protection rules before accepting the pack data. The check happens before any objects are written to the repository.

If `allow_force_pushes` is `false` (the default for protected branches), the server rejects the force push with error `GH006: Push to protected branch`. The client sees:

```
remote: error: GH006: Push to protected branch "main"
remote: error: You cannot force push to a protected branch
To https://github.com/your-org/your-repo.git
 ! [remote rejected] main -> main (protected branch hook declined)
error: failed to push some refs to 'https://github.com/your-org/your-repo.git'
```

The same applies to deletions. If `allow_deletions` is `false`, `git push origin --delete main` is rejected before any data is transferred. The branch is effectively immutable through remote operations.

Note that `allow_force_pushes` can be set to `true` even when other protection rules are active. This is useful for branches like `feature/*` where you want PR reviews but also want to allow rebasing. The more common pattern is to set `allow_force_pushes` to `false` on `main` and `release/*` branches, and `true` on `feature/*` and `develop` branches.

### Step 6: Admin Enforcement and Bypass

The `enforce_admins` setting controls whether administrators are exempt from protection rules. When `enforce_admins` is `true` (the default for new rules), even repository administrators must follow all protection rules. No one can bypass the gate.

When `enforce_admins` is `false`, administrators see a "Bypass protection" option on the merge button. This is useful for emergency hotfix scenarios, but it creates a security gap: any admin can push directly to main without review. Most teams keep `enforce_admins` set to `true` and use a separate emergency process (like a dedicated hotfix branch with relaxed rules) for urgent fixes.

## API Configuration Example

You can configure branch protection rules entirely through the GitHub REST API. This is useful for scripting, infrastructure-as-code, and automated setup of new repositories. The following command creates a protection rule for `main` that requires 1 approving review, requires status checks, enforces admins, requires linear history, blocks force pushes, and blocks deletions:

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection \
  --method PUT \
  --input - <<EOF
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["CI / test", "Lint / lint"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "dismiss_stale_reviews": true,
    "require_code_owner_reviews": false
  },
  "restrictions": null,
  "required_linear_history": true,
  "allow_force_pushes": false,
  "allow_deletions": false,
  "required_conversation_resolution": true
}
EOF
```

The API returns a 200 OK with the full protection object if the rule was created successfully. A 404 means the branch does not exist. A 422 means the request body was invalid — usually because a required field is missing or a status check name does not match any existing check run.

You can also delete all protection rules for a branch by sending a `DELETE` request:

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection \
  --method DELETE
```

This removes all protection rules for the branch. Use this with caution — there is no undo. You must recreate the rules manually or through a script.

## When to Use Branch Protection / When Not To

**Use branch protection when:**

- You have a `main` or `master` branch that deploys to production — a direct push could take down your service and affect paying customers
- Your team has more than two developers — coordination requires rules, not trust, and a single mistake can affect everyone
- You need compliance with standards like SOC 2, PCI-DSS, or SOX that require change control and separation of duties between the developer and the reviewer
- You use CI/CD and want to ensure no code reaches production without passing tests — protection enforces the CI gate at the git level
- You have external contributors through forks — protection prevents malicious or accidental damage from unverified code, and it forces all changes through the PR workflow
- Your repository is part of a monorepo with multiple teams — different teams need different protections on different paths, and branch protection combined with CODEOWNERS gives you per-directory access control

**Consider skipping branch protection when:**

- You are working on a solo project with no collaborators — the overhead of PRs and status checks slows you down with no benefit because you are the only person pushing code
- Your repository is a documentation or configuration repo where direct pushes are the primary workflow — wikis, dotfiles repos, and personal notes repos do not benefit from PR overhead
- You are prototyping or in early exploration — protection rules add friction when you need to iterate quickly, and the code has no production impact yet
- You have a dedicated emergency branch pattern (like `hotfix/*`) with its own rules that allow fast pushes — use that instead of bypassing protection on `main`, and keep `main` fully protected
- Your CI pipeline is unreliable — requiring status checks that frequently fail due to flaky tests will block all merges and frustrate the team. Fix the CI first, then add the protection rule

**Emergency hotfix override process:** When you must bypass protection (a production outage), follow this process:

1. Do NOT disable protection on `main` — you will forget to re-enable it
2. Create a `hotfix/` branch with relaxed protection rules (allow force push, no reviews required)
3. Push the fix to `hotfix/`
4. Create a PR from `hotfix/` to `main` and merge it (the PR still requires status checks, but reviews are bypassed)
5. After the incident, review the audit trail and fix the root cause
6. Optionally prune the `hotfix/` branch

This process gives you an audit trail in the PR history, prevents the "forgot to re-enable protection" scenario, and keeps `main` protected even during an incident. The `hotfix/` branch acts as a controlled bypass — it is not protected, but only specific people know about it and can push to it.

The baseline: if a bad push would cost more than 30 minutes of recovery time, protect the branch. If you trust every team member completely and never make mistakes, you still need protection — mistakes happen at 5 PM on a Friday, not during a calm morning review session. Branch protection is not about trust; it is about physics. Humans make errors under pressure, and protection rules catch those errors before they reach production.
