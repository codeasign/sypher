You will create a GitHub repository, enable branch protection rules, configure required pull request reviews and status checks, then verify that the rules block direct pushes while allowing safe PR merges. You will also add required linear history and test rebase merging.

## Setup

You need a GitHub account and the GitHub CLI (`gh`) installed and authenticated.

**Install GitHub CLI:**

- **Windows:** Download from https://cli.github.com/ or run `winget install GitHub.cli`
- **macOS:** `brew install gh`
- **Linux:** `sudo apt install gh` (Ubuntu/Debian) or `sudo dnf install gh` (Fedora)

**Authenticate:**

```bash
gh auth login
```

Follow the prompts to authenticate with your GitHub account. Choose "HTTPS" as the protocol and "Y" when asked if you want to authenticate with your Git credentials.

**Verify authentication:**

```bash
gh auth status
```

You should see:

```
✓ Logged in to github.com as YOUR-USERNAME
```

**Create the practice directory:**

```bash
mkdir -p ~/branch-protection-practice
cd ~/branch-protection-practice
```

## The Walkthrough

### Step 1: Create a Test Repository with a File

Create a GitHub repository from the command line and initialize it with a file:

```bash
gh repo create branch-protection-practice --public --push --remote origin
echo "# Branch Protection Practice" > README.md
git add README.md
git commit -m "Initial commit with README"
git push origin main
```

You now have a repository at `https://github.com/YOUR-USERNAME/branch-protection-practice` with a single commit on `main`. The `main` branch is currently unprotected — anyone with push access can push directly to it.

### Step 2: Enable Branch Protection via the GitHub UI

Navigate to your repository on GitHub.com. Go to **Settings > Branches > Add branch protection rule**.

In the "Branch name pattern" field, enter `main`. This tells GitHub that the protection rule applies to the `main` branch. You can also use glob patterns like `release/*` or `v*` to protect multiple branches at once.

For now, check only the first option: **Require a pull request before merging**. Leave the required approvals at 1. Do not check any other boxes yet.

Click **Create** at the bottom of the page.

### Step 3: Verify the Rule Blocks a Direct Push

Try pushing directly to `main`:

```bash
echo "Trying to bypass protection" >> README.md
git add README.md
git commit -m "Direct push attempt"
git push origin main
```

You should see an error similar to:

```
remote: error: GH006: Push to protected branch "main"
remote: error: This branch must be reviewed before it can be merged.
To https://github.com/YOUR-USERNAME/branch-protection-practice.git
 ! [remote rejected] main -> main (protected branch hook declined)
error: failed to push some refs to 'https://github.com/YOUR-USERNAME/branch-protection-practice.git'
```

The push was rejected. GitHub did not even accept the data — it checked the protection rule before allowing the ref update. The commit exists in your local repository but was not pushed.

Reset the local commit so you can work from a clean state:

```bash
git reset HEAD~1 --hard
```

### Step 4: Create a Pull Request and Verify It Passes

Now create a branch with a change and open a pull request:

```bash
git checkout -b feature/add-description
echo "This repository demonstrates GitHub branch protection rules." >> README.md
git add README.md
git commit -m "Add description to README"
git push origin feature/add-description
```

Open a pull request from the command line:

```bash
gh pr create --title "Add description to README" --body "Adds a description line to the README file for clarity." --base main
```

Verify the PR was created:

```bash
gh pr view --json title,state,mergeable
```

The output should show `"mergeable": "MERGEABLE"` or `"mergeable": "UNKNOWN"` (it may take a few seconds for GitHub to calculate mergeability). The PR is open and targeting `main`.

Now merge the PR:

```bash
gh pr merge --merge --subject "Merge PR: Add description to README"
```

The merge succeeds because the PR exists, it targets `main`, and the only protection rule (require PR) is satisfied. You did not need a review because you set the required approvals to 1 but did not add a reviewer — GitHub checks that at least one review was submitted, and since you are the repository owner, GitHub may allow self-merge on a repo with no collaborators.

### Step 5: Require Status Checks Before Merging

Now configure a second protection rule: required status checks. Create a GitHub Actions workflow that produces a check run:

```bash
mkdir -p .github/workflows
```

Create `.github/workflows/ci.yml`:

```yaml
name: CI

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Verify README exists
        run: test -f README.md && echo "README exists" || echo "README missing"
      - name: Lint check
        run: |
          echo "Linting passed"
          exit 0
```

Commit and push the workflow:

```bash
git add .github/workflows/ci.yml
git commit -m "Add CI workflow for PR checks"
git push origin feature/add-description
```

Now go to the branch protection settings again. Under **Require status checks to pass before merging**, search for and select the check called `CI / test` (the `test` job in the `CI` workflow). Also check **Require branches to be up to date** to enable strict status checks.

### Step 6: Test That the PR Blocks When Checks Fail

Create a new branch and open a PR with a deliberately broken change:

```bash
git checkout main
git pull origin main
git checkout -b feature/breaking-change
echo "Broken" > broken-file.txt
git add broken-file.txt
git commit -m "Add broken file to test CI"
git push origin feature/breaking-change
```

Open a PR:

```bash
gh pr create --title "Add broken file" --body "This PR should fail the CI check." --base main
```

Wait for the CI workflow to run (about 30 seconds). Check the status:

```bash
gh pr checks feature/breaking-change
```

You should see:

```
CI / test  ...  pass
```

Actually, the CI workflow only checks that `README.md` exists and runs a lint step that always passes. The real test is that the PR cannot be merged unless the CI check passes. Since the check does pass, this PR is mergeable. But the branch is not up to date with `main` (because you created the branch from an older commit), so the strict status check rule blocks the merge.

Try to merge:

```bash
gh pr merge --merge --subject "Merge broken file"
```

The merge should fail because the branch is behind `main`. You need to update the branch first:

```bash
git pull origin main
git push origin feature/breaking-change
```

Now the CI workflow re-runs on the updated branch. Once it passes, the merge goes through. For a real test of a failing check, modify the workflow to fail on certain conditions, but for this exercise, the important behavior is that the merge is blocked until the check passes and the branch is up to date.

### Step 7: Add Required Linear History

Go back to the branch protection settings and check **Require linear history**. This prevents merge commits, forcing all merges to use rebase or squash.

Now create a new PR:

```bash
git checkout main
git pull origin main
git checkout -b feature/linear-test
echo "Linear history test" >> README.md
git add README.md
git commit -m "Test linear history requirement"
git push origin feature/linear-test
gh pr create --title "Test linear history" --body "Testing linear history requirement." --base main
```

Wait for the CI check to pass, then try to merge with `--merge` (which creates a merge commit):

```bash
gh pr merge --merge --subject "Merge: test linear history"
```

This should fail because merge commits are not allowed when linear history is required. The error message will say something like "Merge commits are not allowed on this branch."

Now try merging with `--rebase`:

```bash
gh pr merge --rebase --subject "Rebase: test linear history"
```

The rebase merge succeeds because it does not create a merge commit — it applies the commits on top of the base branch, maintaining a linear history.

### Step 8: Verify Enforcement with gh api

Read the current protection rules for `main`:

```bash
gh api repos/YOUR-USERNAME/branch-protection-practice/branches/main/protection
```

The output is a JSON object showing all configured rules. Look for these keys:

```json
{
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "dismiss_stale_reviews": false,
    "require_code_owner_reviews": false
  },
  "required_status_checks": {
    "strict": true,
    "contexts": ["CI / test"]
  },
  "required_linear_history": {
    "enabled": true
  },
  "enforce_admins": {
    "enabled": true
  },
  "allow_force_pushes": {
    "enabled": false
  },
  "allow_deletions": {
    "enabled": false
  }
}
```

This confirms that the protection rules are stored and enforced by GitHub. You can use this API endpoint to audit protection rules programmatically across your repositories.

## Common Errors at This Stage

### Error 1: "GH006: Push to protected branch"

```
remote: error: GH006: Push to protected branch "main"
remote: error: You cannot push directly to this branch
```

**Why it happens:** You tried to push directly to `main` while branch protection is enabled. The "Require a pull request before merging" rule blocks all direct pushes, regardless of who you are.

**Fix:** Push to a feature branch instead, then open a pull request:

```bash
git checkout -b feature/my-change
git add .
git commit -m "My change"
git push origin feature/my-change
gh pr create --base main
```

### Error 2: "Required status check not found"

```
remote: error: Required status check "CI / test" is not found.
remote: error: The status check was not found on this branch.
```

**Why it happens:** You configured a required status check named `CI / test` in the branch protection settings, but the CI workflow has not run for the current PR branch yet. GitHub requires at least one successful run of the check before it considers the PR mergeable.

**Fix:** Push a commit to the PR branch to trigger the workflow. The workflow must run and report a successful status for the exact check name listed in the protection rules. Check the exact check name in the Actions tab of your PR — it may be `CI / test (ubuntu-latest)` depending on your workflow configuration. If the names do not match exactly, update the protection rule to use the correct name.

### Error 3: "At least 1 approving review is required"

```
remote: error: At least 1 approving review is required by reviewers with write access.
```

**Why it happens:** You enabled "Require a pull request before merging" with 1 required approval, but no one has reviewed the PR. The merge button is disabled until someone with write access approves the PR.

**Fix:** If you are the only collaborator on the repository, you cannot approve your own PR (GitHub does not allow self-approval). You need to either:
- Add a collaborator to the repository who can review and approve
- Reduce the required approvals to 0 (not recommended for production repos)
- Use the `gh pr review` command from another account if you have one

For this practice exercise, consider working with a partner or reducing the review count to 0 temporarily.

### Error 4: "Merge conflict detected when trying to merge"

```
Merge conflict detected. This branch cannot be merged.
```

**Why it happens:** The PR branch has changes that conflict with the base branch. GitHub cannot automatically merge them because the same lines were modified in both branches.

**Fix:** Resolve the conflict locally:

```bash
git checkout feature/my-branch
git pull origin main
# Edit the conflicting files manually
git add .
git commit -m "Resolve merge conflicts with main"
git push origin feature/my-branch
```

The PR branch is now up to date with `main` and conflict-free. The CI checks will re-run automatically.

## What You Just Learned

- You created a GitHub repository and enabled branch protection rules on `main` through the Settings UI
- You verified that a direct push to a protected branch is rejected with error `GH006`
- You created a feature branch, opened a pull request, and merged it successfully through the protected branch
- You configured a GitHub Actions CI workflow and added it as a required status check in the branch protection rules
- You tested that the merge is blocked when the CI check does not exist or has not run for the PR branch
- You enabled required linear history and verified that merge commits are blocked while rebase merges are allowed
- You used the `gh api` endpoint to read the protection rules as JSON, confirming the rules are stored and enforced by GitHub
