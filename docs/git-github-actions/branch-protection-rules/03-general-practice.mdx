Complete these exercises on your own to prove you can configure, manage, and debug branch protection rules on GitHub.

## Tier 1 — Beginner

### Exercise 1: View Protection Rules on a Repository

**Task:** Use the `gh` CLI to view the current branch protection rules for the `main` branch of any repository you have access to. If the branch has no protection rules, the API returns a 404 — that is the expected answer.

**Hint:** Use `gh api repos/OWNER/REPO/branches/main/protection`. If the branch is not protected, the API returns `HTTP 404: Not Found`.

**Solution:**

```bash
gh api repos/github/docs/branches/main/protection
```

If the branch is protected, you will see a JSON response with keys like `required_status_checks`, `required_pull_request_reviews`, and `enforce_admins`. If the branch is not protected, the response is:

```
HTTP 404: Not Found
```

This is normal — it means the branch has no protection rules. You can also check the GitHub UI by navigating to **Settings > Branches** for the repository. The Branches page shows which branches have protection rules and which do not.

### Exercise 2: Enable a Single Rule — Require PRs

**Task:** Find a repository you own (or create a new one) and enable the "Require a pull request before merging" rule on the `main` branch. Set the required number of approving reviews to 1. Do not enable any other rules.

**Hint:** Go to Settings > Branches > Add rule. Enter `main` as the branch name pattern. Check "Require a pull request before merging" and set the required approvals to 1.

**Solution:**

1. Navigate to your repository on GitHub.com
2. Click **Settings** (top navigation bar)
3. Click **Branches** in the left sidebar
4. Click **Add branch protection rule**
5. In the "Branch name pattern" field, enter `main`
6. Check **Require a pull request before merging**
7. Set "Required approving reviews" to `1`
8. Click **Create**

To verify the rule is active, try pushing directly to `main`:

```bash
echo "test" >> test.txt
git add test.txt
git commit -m "Test direct push"
git push origin main
```

The push should be rejected with:

```
remote: error: GH006: Push to protected branch "main"
```

### Exercise 3: Disable Force Pushes on a Branch

**Task:** Create a branch protection rule for `main` that explicitly disallows force pushes. Test that `git push --force` is rejected.

**Hint:** In the branch protection rule settings, scroll down to the "Rules applied to everyone including administrators" section. Uncheck "Allow force pushes" or set it to "Deny."

**Solution:**

1. Go to Settings > Branches > Add rule (or edit the existing rule)
2. Enter `main` as the branch name pattern
3. Scroll to the bottom of the page
4. Under "Rules applied to everyone including administrators," find the **Allow force pushes** setting
5. Select **Deny** (or leave it unchecked — the default is to deny force pushes on protected branches)
6. Click **Save**

To test:

```bash
echo "force push test" >> forced.txt
git add forced.txt
git commit -m "Test force push"
git push origin main --force
```

Expected output:

```
remote: error: GH006: Push to protected branch "main"
remote: error: You cannot force push to a protected branch
```

### Exercise 4: Test That the Rule Blocks a Direct Push

**Task:** With the "Require PR" rule enabled, confirm that you cannot push directly to `main` from any local branch. Test with a new commit, an amended commit, and a force push.

**Hint:** Try `git push origin main`, `git commit --amend && git push origin main`, and `git push origin main --force`. All three should be rejected.

**Solution:**

Create a test commit on main locally:

```bash
git checkout main
echo "test" >> test.txt
git add test.txt
git commit -m "Test push"
```

Now try all three push methods:

```bash
# Standard push — should be rejected
git push origin main
# Output: GH006: Push to protected branch "main"

# Amended commit — should be rejected
git commit --amend -m "Amended test"
git push origin main
# Output: GH006: Push to protected branch "main"

# Force push — should be rejected
git push origin main --force
# Output: GH006: Push to protected branch "main"
```

All three fail because the "Require PR" rule applies to all push operations, not just standard pushes. Reset your local branch:

```bash
git reset HEAD~1 --hard
```

## Tier 2 — Intermediate

### Exercise 1: Configure Branch Protection via the API

**Task:** Use the GitHub REST API to enable branch protection on `main` for a repository. Configure it to require 1 approving review and block force pushes. Do not use the GitHub UI — use only `gh api`.

**Hint:** The endpoint is `PUT /repos/OWNER/REPO/branches/main/protection`. The request body needs `required_pull_request_reviews`, `enforce_admins`, and `restrictions` at minimum. You must include all required fields — GitHub will reject the request if any are missing.

**Solution:**

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection \
  --method PUT \
  --input - <<EOF
{
  "required_status_checks": null,
  "enforce_admins": true,
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "dismiss_stale_reviews": false,
    "require_code_owner_reviews": false
  },
  "restrictions": null,
  "required_linear_history": false,
  "allow_force_pushes": false,
  "allow_deletions": false,
  "required_conversation_resolution": false
}
EOF
```

The response should be a JSON object confirming the protection rule was created. To verify:

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection | jq '.required_pull_request_reviews.required_approving_review_count'
```

Output: `1`

### Exercise 2: Set Up Required Status Checks

**Task:** Create a GitHub Actions workflow in a repository, then configure a branch protection rule that requires the workflow's check to pass before merging. The workflow should run on `pull_request` events targeting `main`.

**Hint:** Create `.github/workflows/ci.yml` with a job that runs a simple test. In the branch protection settings, add the check name (e.g., `CI / test`) under "Require status checks to pass before merging."

**Solution:**

Create the workflow file:

```yaml
name: CI

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: |
          echo "Running tests..."
          echo "All tests passed"
          exit 0
```

Commit and push:

```bash
git add .github/workflows/ci.yml
git commit -m "Add CI workflow"
git push origin main
```

Now configure the branch protection rule to require the `CI / test` check. Go to Settings > Branches > Edit the rule for `main` > Check "Require status checks to pass before merging" > Search for and select `CI / test`.

To test, create a PR branch and push a change:

```bash
git checkout -b feature/test-ci
echo "test" >> test.txt
git add test.txt
git commit -m "Test CI check"
git push origin feature/test-ci
gh pr create --base main --title "Test CI" --body "Testing CI check"
```

The PR will show a yellow indicator while the CI check runs, then a green checkmark when it passes. The merge button is enabled only after the check passes.

### Exercise 3: Combine Require PRs + Require Status Checks + Dismiss Stale Reviews

**Task:** Configure a branch protection rule on `main` that combines three rules:
1. Require a pull request before merging (1 approval)
2. Require status checks to pass before merging (any check)
3. Dismiss stale pull request approvals when new commits are pushed

Test that pushing a new commit to a PR branch dismisses the existing approval.

**Hint:** All three settings are in the same branch protection rule. The "Dismiss stale reviews" checkbox is under "Require a pull request before merging." You need a collaborator to approve the PR, then push a new commit to see the approval get dismissed.

**Solution:**

Configure the protection rule:

1. Go to Settings > Branches > Edit rule for `main`
2. Check **Require a pull request before merging** — set approvals to 1
3. Check **Dismiss stale pull request approvals when new commits are pushed**
4. Check **Require status checks to pass before merging** — select `CI / test`
5. Click **Save changes**

Create a PR and have a collaborator approve it:

```bash
git checkout -b feature/stale-test
echo "stale test" >> test.txt
git add test.txt
git commit -m "Test stale review dismissal"
git push origin feature/stale-test
gh pr create --base main --title "Stale review test" --body "Testing stale review dismissal"
```

Ask a collaborator to approve the PR through the GitHub UI (or use `gh pr review --approve` if you have a second account):

```bash
gh pr review --approve
```

Immediately after the approval, push a new commit to the same branch:

```bash
echo "new commit" >> test.txt
git add test.txt
git commit -m "New commit after approval"
git push origin feature/stale-test
```

Now check the PR status:

```bash
gh pr view --json reviews
```

The previous approval should show but with a status of `DISMISSED` because new commits were pushed. The CI check re-runs on the new commit. The PR needs a fresh approval before it can be merged.

### Exercise 4: Use gh api to Check Current Protection State

**Task:** Write a shell script that checks the protection state of the `main` branch and reports whether each rule is enabled or disabled. The script should output a human-readable report.

**Hint:** Use `gh api` to fetch the protection data, then parse it with `jq`. Check for the presence of keys like `required_pull_request_reviews`, `required_status_checks`, `enforce_admins`, `required_linear_history`, `allow_force_pushes`, and `allow_deletions`.

**Solution:**

```bash
#!/bin/bash
# check-protection.sh — Inspect branch protection rules for a given repo

REPO="${1:-YOUR-USERNAME/YOUR-REPO}"
BRANCH="${2:-main}"

echo "=== Branch Protection Report for $REPO:$BRANCH ==="
echo ""

DATA=$(gh api repos/$REPO/branches/$BRANCH/protection 2>/dev/null)

if [ $? -ne 0 ]; then
  echo "Branch $BRANCH is NOT protected."
  exit 0
fi

echo "$DATA" | jq -r '
  "Required PR reviews: " + (
    if .required_pull_request_reviews != null then
      "YES (" + (.required_pull_request_reviews.required_approving_review_count | tostring) + " approvals)"
    else "NO" end
  ),
  "Required status checks: " + (
    if .required_status_checks != null then
      "YES (" + (.required_status_checks.contexts | join(", ")) + ")"
    else "NO" end
  ),
  "Strict status checks: " + (
    if .required_status_checks != null and .required_status_checks.strict then "YES"
    else "NO" end
  ),
  "Enforce admins: " + (
    if .enforce_admins != null and .enforce_admins.enabled then "YES"
    else "NO" end
  ),
  "Linear history: " + (
    if .required_linear_history != null and .required_linear_history.enabled then "YES"
    else "NO" end
  ),
  "Allow force pushes: " + (
    if .allow_force_pushes != null and .allow_force_pushes.enabled then "YES"
    else "NO" end
  ),
  "Allow deletions: " + (
    if .allow_deletions != null and .allow_deletions.enabled then "YES"
    else "NO" end
  ),
  "Conversation resolution: " + (
    if .required_conversation_resolution != null and .required_conversation_resolution.enabled then "YES"
    else "NO" end
  )
'
```

Run the script:

```bash
bash check-protection.sh YOUR-USERNAME/YOUR-REPO
```

Output:

```
=== Branch Protection Report for YOUR-USERNAME/YOUR-REPO:main ===

Required PR reviews: YES (1 approvals)
Required status checks: YES (CI / test)
Strict status checks: YES
Enforce admins: YES
Linear history: YES
Allow force pushes: NO
Allow deletions: NO
Conversation resolution: NO
```

## Tier 3 — Advanced

### Exercise 1: Emergency Hotfix Override

**Scenario:** Production is down. You need to push a critical fix to `main` immediately, but `main` has branch protection enabled with required reviews, status checks, and linear history. You cannot wait for a PR review. Design and execute a temporary override process that:
1. Allows you to push the fix within 60 seconds
2. Leaves an audit trail of what was done
3. Restores full protection immediately after the fix is pushed

**Task:** Create a temporary bypass by relaxing the protection rule, push the fix, then restore the original protection. Use the API, not the UI.

**Hint:** Use `gh api` with `PATCH` or `PUT` to update the protection rule. Remove the `required_pull_request_reviews` requirement, push the fix, then restore the original rule. Save the original protection rule JSON before modifying it.

**Solution:**

Save the current protection rules:

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection > original-protection.json
```

Temporarily remove the PR review requirement (keep status checks to prevent broken code):

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection \
  --method PUT \
  --input - <<EOF
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["CI / test"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": null,
  "restrictions": null,
  "required_linear_history": false,
  "allow_force_pushes": false,
  "allow_deletions": false,
  "required_conversation_resolution": false
}
EOF
```

Push the hotfix:

```bash
git checkout main
# Apply the fix
echo "fix" >> fix.txt
git add fix.txt
git commit -m "HOTFIX: Critical production fix"
git push origin main
```

Restore the original protection rules:

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection \
  --method PUT \
  --input original-protection.json
```

Clean up the saved file and verify:

```bash
rm original-protection.json
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection | jq '.required_pull_request_reviews.required_approving_review_count'
```

The output should show `1` — the original protection is restored. The `git log` on `main` shows the hotfix commit, and the API audit log (if you have GitHub Enterprise) records the protection rule changes.

### Exercise 2: Multi-Branch Protection Strategy

**Scenario:** Your team has three branch types: `main` (production), `develop` (integration), and `release/*` (release candidates). Each needs different protection rules. Write a script that configures protection for all three branch types.

**Task:** Create a shell script that uses `gh api` to set three different protection rule sets:
- `main`: Require 2 PR approvals, require status checks, enforce admins, require linear history, block force pushes
- `develop`: Require 1 PR approval, require status checks, enforce admins, allow force pushes
- `release/*`: Require 1 PR approval, require status checks, enforce admins, require linear history, block force pushes

**Hint:** Use a loop over the branch patterns. For `release/*`, the branch name pattern in the API call is `release/*` — GitHub applies the rule to all matching branches.

**Solution:**

```bash
#!/bin/bash
# configure-protection.sh — Set branch protection rules for multiple branches

REPO="${1:-YOUR-USERNAME/YOUR-REPO}"

protect_branch() {
  local pattern="$1"
  local approvals="$2"
  local linear="$3"
  local force_push="$4"

  echo "Configuring protection for pattern: $pattern"

  gh api repos/$REPO/branches/$pattern/protection \
    --method PUT \
    --input - <<EOF
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["CI / test"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": {
    "required_approving_review_count": $approvals,
    "dismiss_stale_reviews": true,
    "require_code_owner_reviews": false
  },
  "restrictions": null,
  "required_linear_history": {
    "enabled": $linear
  },
  "allow_force_pushes": {
    "enabled": $force_push
  },
  "allow_deletions": false,
  "required_conversation_resolution": true
}
EOF
  echo "Done: $pattern"
  echo ""
}

# Main branch — strictest rules
protect_branch "main" 2 true false

# Develop branch — moderate rules, allow force pushes
protect_branch "develop" 1 false true

# Release branches — strict rules, no force pushes, 1 approval
protect_branch "release/*" 1 true false

echo "All branch protection rules configured."
```

Run the script:

```bash
bash configure-protection.sh YOUR-USERNAME/YOUR-REPO
```

Verify each branch:

```bash
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection | jq '.required_pull_request_reviews.required_approving_review_count'
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/develop/protection | jq '.allow_force_pushes.enabled'
```

The `main` branch requires 2 approvals and blocks force pushes. The `develop` branch requires 1 approval and allows force pushes. The `release/*` pattern requires 1 approval and blocks force pushes. Each branch type has rules appropriate for its role in the development workflow.

### Exercise 3: Audit Protection Rule Changes

**Scenario:** Your security team needs to know when and how branch protection rules were changed on the `main` branch over the last 30 days. GitHub does not have a built-in audit log for protection rule changes in the free tier, but you can build a tracking system.

**Task:** Create a script that takes a snapshot of current protection rules and saves them to a file with a timestamp. Then create a diff script that compares two snapshots and reports what changed.

**Hint:** Use `gh api` to fetch the protection data, save it to a file named `protection-snapshot-YYYY-MM-DD.json`. Use `jq` to diff the `required_pull_request_reviews` and `required_status_checks` sections between two snapshots.

**Solution:**

Snapshot script:

```bash
#!/bin/bash
# snapshot-protection.sh — Save current protection rules to a dated file

REPO="${1:-YOUR-USERNAME/YOUR-REPO}"
DATE=$(date +%Y-%m-%d)
FILE="protection-snapshot-$DATE.json"

echo "Saving protection snapshot for $REPO to $FILE"

gh api repos/$REPO/branches/main/protection > "$FILE" 2>/dev/null

if [ $? -ne 0 ]; then
  echo "Error: Could not fetch protection rules for $REPO"
  exit 1
fi

echo "Snapshot saved. File size: $(wc -c < $FILE) bytes"
```

Diff script:

```bash
#!/bin/bash
# diff-protection.sh — Compare two protection snapshots

if [ $# -ne 2 ]; then
  echo "Usage: $0 snapshot1.json snapshot2.json"
  exit 1
fi

echo "=== Protection Rule Diff ==="
echo "From: $1"
echo "To:   $2"
echo ""

diff <(jq --sort-keys '{required_pull_request_reviews, required_status_checks, enforce_admins, required_linear_history, allow_force_pushes, allow_deletions, required_conversation_resolution}' "$1") \
     <(jq --sort-keys '{required_pull_request_reviews, required_status_checks, enforce_admins, required_linear_history, allow_force_pushes, allow_deletions, required_conversation_resolution}' "$2") \
     && echo "No changes detected." || echo "Changes detected above."
```

Run the scripts:

```bash
# Take a snapshot
bash snapshot-protection.sh YOUR-USERNAME/YOUR-REPO

# Make a change to protection rules (add linear history)
gh api repos/YOUR-USERNAME/YOUR-REPO/branches/main/protection \
  --method PUT \
  --input - <<EOF
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["CI / test"]
  },
  "enforce_admins": true,
  "required_pull_request_reviews": {
    "required_approving_review_count": 1,
    "dismiss_stale_reviews": true,
    "require_code_owner_reviews": false
  },
  "restrictions": null,
  "required_linear_history": {
    "enabled": true
  },
  "allow_force_pushes": false,
  "allow_deletions": false,
  "required_conversation_resolution": true
}
EOF

# Take a second snapshot
bash snapshot-protection.sh YOUR-USERNAME/YOUR-REPO

# Compare
bash diff-protection.sh protection-snapshot-2026-07-03.json protection-snapshot-2026-07-04.json
```

The diff output will show which fields changed. For example:

```
> "required_linear_history": {
>   "enabled": true
> }
```

This confirms that a linear history requirement was added between the two snapshots. For production use, schedule the snapshot script to run daily via a cron job or GitHub Actions scheduled workflow, and store the snapshots in a separate repository for audit compliance.
