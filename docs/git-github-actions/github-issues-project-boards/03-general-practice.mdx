## Tier 1 — Beginner

### Exercise 1

**Task:** Create a new issue with the title "Add documentation for API" and the label "documentation". View it, then close it.

<details>
<summary>Hint</summary>
Use `gh issue create --title "..." --label documentation`, then `gh issue view <number>`, then `gh issue close <number>`.
</details>

### Solution

```bash
cd ~/issues-lab
gh issue create --title "Add documentation for API" --label documentation
gh issue view 4
gh issue close 4
```

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a milestone, create 3 issues associated with it, then list all issues in the milestone using `gh issue list --milestone`.

<details>
<summary>Hint</summary>
Create the milestone with `gh api repos/:owner/:repo/milestones --method POST --field title="v2.0"`. Create issues with `--milestone "v2.0"`. List with `gh issue list --milestone "v2.0"`.
</details>

### Solution

```bash
gh api repos/:owner/:repo/milestones \
  --method POST \
  --field title="v2.0" \
  --field description="Next release"
for i in 1 2 3; do
  gh issue create --title "Issue $i for v2.0" --milestone "v2.0"
done
gh issue list --milestone "v2.0"
```

### Exercise 2

**Task:** Create a PR that references an issue with `Closes #N`. Before merging, verify the issue is open. After merging, verify the issue is closed. Use `gh issue view` before and after.

<details>
<summary>Hint</summary>
Create a branch, make a change, commit, push. Create PR with `--body "Closes #N"`. Before merging, `gh issue view N` shows `state: OPEN`. After merging, it shows `state: CLOSED`.
</details>

### Solution

```bash
git switch main
git switch -c close-issue-test
echo "test" > close_test.txt && git add close_test.txt && git commit -m "Test close"
git push origin close-issue-test
gh issue view 1 2>/dev/null && echo "Issue 1 is open" || echo "Issue 1 not found"
gh pr create --title "Test close" --body "Closes #1" --base main
gh pr merge 5 --squash --delete-branch
gh issue view 1
```

Issue 1 shows `state: CLOSED`.

## Tier 3 — Advanced

### Exercise 1: Automate issue triage with labels

**Scenario:** Your team receives 20+ issues per week. You want to automatically label new issues based on keywords in the title or body. For example, any issue with "crash", "error", or "bug" in the title should get the `bug` label. Any issue with "feature" or "request" should get the `enhancement` label.

**Task:** Write a GitHub Actions workflow that runs on `issues: opened` events, checks the title and body for keywords, and applies the appropriate label. Use `gh` CLI within the action to add labels.

<details>
<summary>Hint</summary>
Create `.github/workflows/auto-label.yml`. Use `github.event.issue.title` and `github.event.issue.body` in the workflow. Use `gh issue edit ${{ github.event.issue.number }} --add-label "bug"` to apply labels.
</details>

### Solution — one valid approach

```bash
mkdir -p .github/workflows
cat > .github/workflows/auto-label.yml << 'EOF'
name: Auto-label issues
on:
  issues:
    types: [opened]
jobs:
  label:
    runs-on: ubuntu-latest
    permissions:
      issues: write
    steps:
      - uses: actions/github-script@v7
        with:
          script: |
            const title = context.payload.issue.title.toLowerCase();
            const body = (context.payload.issue.body || '').toLowerCase();
            const labels = [];
            if (/crash|error|bug|fail/i.test(title + body))
              labels.push('bug');
            if (/feature|request|enhancement/i.test(title + body))
              labels.push('enhancement');
            if (/doc|readme|help/i.test(title + body))
              labels.push('documentation');
            if (labels.length > 0) {
              await github.rest.issues.addLabels({
                owner: context.repo.owner,
                repo: context.repo.repo,
                issue_number: context.issue.number,
                labels: labels
              });
            }
EOF
git add .github/workflows/auto-label.yml
git commit -m "Add auto-label workflow for issues"
git push origin main
```

**Alternative approach:** Use a GitHub App or Probot extension like `settings` to manage labels declaratively. The workflow approach is simpler and does not require additional permissions.

### Exercise 2: Create a sprint board with automated columns

**Scenario:** Your team uses a sprint board with 4 columns: Backlog, To Do, In Progress, Done. You want:
- New issues with the `priority-high` label go directly to "To Do"
- Issues with an open PR go to "In Progress"
- Issues with a merged PR go to "Done"
- All other new issues go to "Backlog"

**Task:** Create the project board with 4 columns. Write a script (or GitHub Actions workflow) that moves issues based on these rules. Then test it by creating a priority-high issue and verifying it lands in "To Do".

<details>
<summary>Hint</summary>
Create the project board via `gh project create`. Use `gh project field-list` to see available fields. For automation, either use GitHub Actions with a project automation action, or use the built-in "Auto-add to project" workflow. Use `gh project item-add` with the correct field values to place items in the right column.
</details>

### Solution — one valid approach

```bash
# Create the project board
gh project create --owner "@me" --title "Sprint Board" --repos issues-lab
```

Create the automation workflow:

```bash
cat > .github/workflows/sprint-automation.yml << 'EOF'
name: Sprint board automation
on:
  issues:
    types: [opened, labeled]
  pull_request:
    types: [opened, closed]
jobs:
  prioritize:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/add-to-project@v0.5.0
        with:
          project-url: https://github.com/users/<your-username>/projects/1
          github-token: ${{ secrets.GITHUB_TOKEN }}
      - if: github.event.label.name == 'priority-high'
        run: |
          echo "Priority-high issue goes to To Do"
          # The add-to-project action places it in the first column by default
          # GitHub project automation rules handle the column assignment
EOF
git add .github/workflows/sprint-automation.yml
git commit -m "Add sprint board automation"
git push origin main
```

**Alternative approach:** Use GitHub's built-in project board automation (no code needed). Go to Project → Workflows → "Item added to project" → choose the column. This is simpler for teams that prefer a UI.

### Exercise 3: Create a release issue checklist template

**Scenario:** Every release requires the same steps: update version number, update changelog, run tests, create tag, push to package registry, deploy to production. You want a reusable issue template that creates a checklist for each release.

**Task:** Create a release checklist issue template at `.github/ISSUE_TEMPLATE/release-checklist.md`. The template should include:
- Version number input
- Checklist of release steps
- Space for the release manager to sign off
- A note that the issue should be closed only after the release is complete

Then create a script that, given a version number, creates a new release issue with the checklist pre-filled.

<details>
<summary>Hint</summary>
Use YAML frontmatter in the template. Use `gh issue create` with the template's filename. Use `gh issue create --template release-checklist.md` or pass the body via `--body-file`.
</details>

### Solution — one valid approach

```bash
cat > .github/ISSUE_TEMPLATE/release-checklist.md << 'EOF'
---
name: Release checklist
about: Track a release from start to finish
title: "[RELEASE] "
labels: release
assignees: ''
---

## Release Checklist

### Version: `{{VERSION}}`

### Pre-release
- [ ] Version number updated in all files
- [ ] Changelog updated with release notes
- [ ] All tests pass: `pytest`
- [ ] Linter passes: `ruff check .`
- [ ] Security audit: `pip audit`

### Release
- [ ] Tag created: `git tag -a v{{VERSION}} -m "Release v{{VERSION}}"`
- [ ] Tag pushed: `git push origin v{{VERSION}}`
- [ ] Package published to registry
- [ ] Release notes published on GitHub

### Post-release
- [ ] Production deployed
- [ ] Smoke tests pass on production
- [ ] Release announced to team

### Sign-off
**Release Manager:** __________________
**Date:** __________________

> Close this issue only after all steps are complete.
EOF
```

Create a script to instantiate the template:

```bash
cat > create-release.sh << 'EOF'
#!/bin/bash
VERSION=${1:?Usage: $0 <version>}
TEMPLATE=$(cat .github/ISSUE_TEMPLATE/release-checklist.md)
BODY="${TEMPLATE//\{\{VERSION\}\}/$VERSION}"
gh issue create --title "[RELEASE] v$VERSION" --label release --body "$BODY"
EOF
chmod +x create-release.sh
./create-release.sh 1.0.0
```

**Alternative approach:** Use GitHub Actions to create the release issue automatically when a milestone is closed. The action can parse the milestone name as the version number and populate the checklist.
