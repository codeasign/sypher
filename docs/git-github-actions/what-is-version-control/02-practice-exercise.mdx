This walkthrough takes you from a blank directory to a fully version-controlled project with a complete commit history. You will create a repository, make changes, stage them, commit them, and view the history. By the end you will have experienced every step of the version control workflow.

## Setup

Before you start, make sure Git is installed and configured. Open a terminal and run:

```bash
git --version
```

You should see output similar to `git version 2.40.0` or higher. If you see an error about a missing command, install Git using your operating system's package manager:

- **Windows:** Download from https://git-scm.com or run `winget install Git.Git` in PowerShell
- **macOS:** Run `brew install git` or download from https://git-scm.com
- **Linux (Ubuntu/Debian):** Run `sudo apt install git`
- **Linux (Fedora):** Run `sudo dnf install git`

Next, configure your name and email. Git attaches this information to every commit you make. Run these two commands with your own name and email:

```bash
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
```

Now create a new directory for this walkthrough:

```bash
mkdir ~/git-first-repo
cd ~/git-first-repo
```

**Windows users:** If you are using Command Prompt or PowerShell, use `mkdir %USERPROFILE%\git-first-repo` and `cd %USERPROFILE%\git-first-repo`. If you are using Git Bash, the `~/` path works as shown above.

**macOS and Linux users:** The `~/` path expands to your home directory. All commands in this walkthrough use the current directory, so you can also use any directory you prefer.

## The Walkthrough

You will follow a realistic scenario: you are building a simple project and you want to track every change with Git.

### Step 1: Initialize a Repository

Turn your project directory into a Git repository.

```bash
git init
```

**Expected output:**
```
Initialized empty Git repository in /Users/you/git-first-repo/.git/
```

**Explanation:** `git init` creates a `.git` subdirectory. This directory contains all of Git's internal data structures — the object database, references, configuration, and staging area. Your project directory is now a Git repository, but it contains no commits and no tracked files. The repository is empty, ready for you to add content.

**Checkpoint:** Verify the repository was created:

```bash
ls -la .git
```

You should see files and directories inside `.git` including `HEAD`, `config`, `description`, `hooks/`, `info/`, `objects/`, and `refs/`. These are Git's internal plumbing — you will rarely touch them directly, but they are the engine of version control.

### Step 2: Check the Status of an Empty Repository

See what Git knows about your project right now.

```bash
git status
```

**Expected output:**
```
On branch master

No commits yet

nothing to commit (create/copy files and use "git add" to track)
```

**Explanation:** `git status` shows the current state of the repository. The output tells you three things: you are on the branch named `master`, there are no commits in the history, and there is nothing to commit because the working directory is empty. This is the baseline state of a freshly initialized repository.

**Checkpoint:** Run `git status` any time you are unsure about the state of your repository. It is a read-only command — it never modifies anything. Use it as your compass.

### Step 3: Create Your First File and Check Status

Create a file to track. This simulates writing your first piece of code.

```bash
echo "# My First Project" > README.md
git status
```

**Expected output:**
```
On branch master

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md

nothing added to commit but untracked files present (use "git add" to track)
```

**Explanation:** Git sees `README.md` but it is listed as "untracked." Untracked means Git has detected a file that does not exist in any previous commit and has not been added to the staging area. Git tells you about it but does not do anything with it until you tell it to. This is version control's safety mechanism — you control what gets tracked.

**Checkpoint:** The file exists on disk but Git is not tracking it yet. You can verify with `cat README.md` to see its contents.

### Step 4: Stage the File

Tell Git to start tracking `README.md` by adding it to the staging area.

```bash
git add README.md
git status
```

**Expected output:**
```
On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   README.md
```

**Explanation:** `git add README.md` copies the current version of the file from your working directory into the staging area (the index). The staging area is a file at `.git/index` that records what will go into the next commit. The output now shows `README.md` under "Changes to be committed" — it is staged and ready to be committed.

The staging area is one of Git's most important concepts. It lets you build a commit incrementally: you can stage some files, stage parts of files, or stage all files, and then commit exactly what you have prepared.

**Checkpoint:** The file is now staged. Any changes you make to the working directory copy will not affect the staged version until you `git add` again.

### Step 5: Commit the Staged File

Record the staged snapshot as a permanent commit in the repository.

```bash
git commit -m "Initial commit: add README.md"
```

**Expected output:**
```
[master (root-commit) a1b2c3d] Initial commit: add README.md
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
```

**Explanation:** `git commit` takes everything in the staging area and creates a permanent snapshot. The `-m` flag provides the commit message. The output shows:
- The branch name `master`
- The label `(root-commit)` — this is the first commit in the repository
- The commit hash `a1b2c3d` — a unique SHA-1 identifier for this commit
- A summary of changes: 1 file changed, 1 line inserted

Internally, Git creates a tree object representing the directory structure, a blob object for the file contents, and a commit object that ties them together with the author, timestamp, and message. These objects are stored in `.git/objects/`.

**Checkpoint:** The commit is permanent. Git will never lose it unless you deliberately delete the `.git` directory. Run `git status` to confirm the working directory is clean:

```
On branch master
nothing to commit, working tree clean
```

### Step 6: Modify the File and View the Diff

Edit the file and see what changed before committing.

```bash
echo "# My First Project" > README.md
echo "" >> README.md
echo "This project demonstrates version control with Git." >> README.md
git status
```

**Expected output:**
```
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   README.md

no changes added to commit (use "git add" and/or "git commit -a")
```

**Explanation:** Git detects that `README.md` has changed compared to the version in the last commit. The file is listed as "modified" under "Changes not staged for commit." This means the working directory version is different from the committed version.

Now view the exact changes:

```bash
git diff README.md
```

**Expected output:**
```
diff --git a/README.md b/README.md
index a1b2c3d..e5f6g7a 100644
--- a/README.md
+++ b/README.md
@@ -1 +1,3 @@
 # My First Project
+
+This project demonstrates version control with Git.
```

**Explanation:** `git diff` shows the lines that differ between the working directory and the last commit. Lines starting with `+` are additions. Lines starting with `-` are deletions. The `@@` line tells you the position and number of lines affected. This is how you inspect changes before staging them.

**Checkpoint:** The file is modified but not staged. The diff shows exactly what you added. You can undo the changes with `git restore README.md` if you decide you do not want them.

### Step 7: Stage and Commit the Changes

Add the modified file to the staging area and commit it.

```bash
git add README.md
git commit -m "Add project description to README"
```

**Expected output:**
```
[master e5f6g7a] Add project description to README
 1 file changed, 2 insertions(+)
```

**Explanation:** You repeated the stage-and-commit cycle. This is the fundamental workflow of Git: edit, stage, commit. Each commit creates a permanent snapshot in the repository's history. The commit hash `e5f6g7a` is different from the first commit's hash. Each commit gets a unique identifier derived from its content, author, timestamp, and parent hash.

**Checkpoint:** The working tree is clean again. Run `git status` to confirm.

### Step 8: View the Commit History

See the full history of your project.

```bash
git log
```

**Expected output:**
```
commit e5f6g7a (HEAD -> master)
Author: Your Name <your.email@example.com>
Date:   Thu Jul 3 14:30:00 2026 -0400

    Add project description to README

commit a1b2c3d
Author: Your Name <your.email@example.com>
Date:   Thu Jul 3 14:25:00 2026 -0400

    Initial commit: add README.md
```

**Explanation:** `git log` displays the commit history in reverse chronological order — the most recent commit appears first. Each entry shows the commit hash, author, date, and message. The most recent commit is marked with `HEAD -> master`, meaning `HEAD` (your current position) points to `master`, which points to this commit.

Now view a compact version:

```bash
git log --oneline
```

**Expected output:**
```
e5f6g7a (HEAD -> master) Add project description to README
a1b2c3d Initial commit: add README.md
```

**Explanation:** The `--oneline` flag condenses each commit to a single line with the abbreviated hash and message. This is the most common way to scan history.

**Checkpoint:** You have two commits in your history. Your repository now has a complete, versioned record of your project's evolution.

### Step 9: View the Graph of Your History

See how commits relate to each other.

```bash
git log --oneline --graph
```

**Expected output:**
```
* e5f6g7a (HEAD -> master) Add project description to README
* a1b2c3d Initial commit: add README.md
```

**Explanation:** The `--graph` flag adds an ASCII visualization of the commit graph. The `*` characters show each commit. With a linear history like this, the graph is a straight line. When you start using branches, the graph will show branching and merging lines.

**Checkpoint:** The `*` characters trace the chain of commits. Each commit points to its parent, forming the DAG described in the overview.

### Step 10: Create a Second File and See the Full Workflow

Repeat the workflow with a new file to reinforce the cycle.

```bash
echo "def hello():" > hello.py
echo "    print('Hello, version control!')" >> hello.py
git status
git add hello.py
git commit -m "Add hello.py script"
git log --oneline
```

**Expected output:**
```
On branch master
Untracked files:
  hello.py

[master c8d9e0f] Add hello.py script
 1 file changed, 2 insertions(+)
 create mode 100644 hello.py

c8d9e0f (HEAD -> master) Add hello.py script
e5f6g7a Add project description to README
a1b2c3d Initial commit: add README.md
```

**Explanation:** You now have three commits. Each commit is a permanent snapshot. You can view any commit's contents with `git show <hash>`, compare any two commits with `git diff <hash1>..<hash2>`, or restore the entire project to any earlier state with `git checkout <hash>`.

## Common Errors at This Stage

### Error 1: fatal: not a git repository

**Error message:**
```
fatal: not a git repository (or any of the parent directories): .git
```

**Why it happens:** You ran a Git command (like `git status` or `git commit`) in a directory that is not a Git repository. Either you never ran `git init` or you are in the wrong directory.

**How to fix:** Run `git init` to create a repository in the current directory, or use `cd` to navigate to the correct directory where your `.git` folder exists.

### Error 2: nothing to commit, working tree clean

**Error message:**
```
On branch master
nothing to commit, working tree clean
```

**Why it happens:** You ran `git commit` or `git status` but there are no changes staged or modified. This happens when you forget to edit files, forget to save changes, or forget to stage changes with `git add`.

**How to fix:** Make sure you have edited and saved files. Run `git status` to verify files are modified, then run `git add` to stage them, then commit.

### Error 3: fatal: refusing to merge unrelated histories

**Error message:**
```
fatal: refusing to merge unrelated histories
```

**Why it happens:** You tried to merge two repositories that do not share a common ancestor commit. This is common when you create a local repository with `git init` and then try to pull from a remote that already has commits.

**How to fix:** If you are intentionally merging unrelated histories, use the `--allow-unrelated-histories` flag. In most cases, you should clone the remote repository instead of initializing a new one.

### Error 4: error: pathspec did not match any files

**Error message:**
```
error: pathspec 'filename.py' did not match any files
```

**Why it happens:** You ran `git add filename.py` but the file does not exist in the working directory. You may have mistyped the filename, the file may be in a different directory, or you may have deleted the file.

**How to fix:** Check the exact filename with `ls` (or `dir` on Windows). Verify the path is correct. Git is case-sensitive even on Windows, so `README.md` and `readme.md` are different files.

### Error 5: Please tell me who you are

**Error message:**
```
*** Please tell me who you are.

Run

  git config --global user.email "you@example.com"
  git config --global user.name "Your Name"

to set your account's default identity.
Omit --global to set the identity only in this repository.
```

**Why it happens:** You tried to make a commit without configuring your name and email. Git requires this information for every commit because each commit stores the author's identity.

**How to fix:** Run the two `git config` commands shown in the error message. Use `--global` to set it for all repositories on your machine, or omit `--global` to set it only for the current repository.

## What You Just Learned

- You can create a Git repository with `git init` and verify its state with `git status`.
- Files must be explicitly staged with `git add` before they can be committed. Git does not track anything automatically.
- A commit is a permanent snapshot of every staged file, stored in the `.git/objects/` database with a unique SHA-1 hash.
- The `git log` command displays the commit history, and `--oneline` gives a compact view suitable for scanning.
- The `git diff` command shows the exact lines that changed between your working directory and the last commit.
- The core Git workflow is a three-step cycle: edit files in the working directory, stage them with `git add`, and commit them with `git commit`.
- `git status` is a safe, read-only command that you should run frequently to understand the current state of your repository.
- Every commit stores the author, timestamp, and a message — this forms a complete, auditable record of your project's evolution.
