You finish editing `config.yaml` at 10:23 AM. At 11:00 AM the build breaks. You need the old version back, but your editor undo history is gone. You check your project folder and find `config_v2.yaml`, `config_v2_reverted.yaml`, `config_FINAL.yaml`, `config_actually_final_v3.yaml`, and `config_backup_working.yaml`. None of them is the exact version you need. You spend twenty minutes hunting through email attachments, Slack messages, and USB backup folders asking coworkers if they have a copy. This is the reality of working without version control — lost work, conflicting edits, untracked changes, and hours wasted every week reconciling files by hand. Version control eliminates this entire category of problems.

## The Concept

Version control is a system that records changes to files over time so you can recall specific versions later. Think of it as a time machine for your project: you can travel backward to any saved state, see exactly what changed and who changed it, and restore any file to any previous version with a single command. It is the same principle a scientist uses when logging every step of an experiment — record the exact conditions, timestamp the observation, annotate the result, and never lose a data point.

## Command and Syntax Reference

| Command | Purpose | Syntax | Common Flags | Example |
|---------|---------|--------|-------------|---------|
| `git init` | Create a new repository | `git init [directory]` | `--bare` for server repos | `git init my-project` |
| `git clone` | Copy an existing repository | `git clone <url> [dir]` | `--depth 1` for shallow clone | `git clone https://github.com/user/repo.git` |
| `git status` | Show working tree state | `git status` | `--short` for compact output | `git status --short` |
| `git log` | Display commit history | `git log [options]` | `--oneline --graph --all` | `git log --oneline -5` |
| `git add` | Stage changes for commit | `git add <pathspec>` | `-p` for interactive staging | `git add src/app.py` |
| `git commit` | Record staged snapshot | `git commit -m "message"` | `--amend` to edit last commit | `git commit -m "fix: handle null input"` |
| `git diff` | Show unstaged differences | `git diff [path]` | `--staged` for staged diff | `git diff --staged` |

**Windows note:** All commands above work identically in Git Bash, PowerShell, and Command Prompt. The only difference is that Windows uses `\` in paths while macOS and Linux use `/`. Git accepts `/` on all platforms, so you can use forward slashes consistently.

**macOS and Linux note:** Git is pre-installed on most systems. If not, install via `brew install git` (macOS) or `apt install git` (Ubuntu/Debian) or `dnf install git` (Fedora).

## How It Actually Works: Step by Step

Version control evolved through three generations of architecture. Each generation solved a limitation of the previous one. Understanding this evolution explains why Git works the way it does.

### Step 1: The Problem of Manual Versioning

Before version control existed, developers managed versions by copying files. This approach has four fundamental problems:

1. **No history** — You cannot see what changed between copies or who made the change.
2. **No single source of truth** — Multiple copies diverge, and you never know which one is current.
3. **No atomicity** — A partial edit can leave files in an inconsistent state with no way to roll back.
4. **No collaboration** — Two people editing the same file must merge changes by hand, line by line.

### Step 2: Local Version Control

The first generation of version control tools, such as RCS (Revision Control System), stored a patch database on your local machine. Instead of copying entire files, they stored the original file plus a series of diffs — the changes between versions. To reconstruct any version, the tool applied the diffs in order.

This approach solved the "too many copies" problem but introduced a single point of failure. If your hard drive failed, you lost every version of every file.

### Step 3: Centralized Version Control

The second generation introduced a central server. Tools like CVS and Subversion (SVN) stored the master repository on a shared server. Every team member checked out files from this server and checked in changes. The server was the single source of truth.

This approach solved the collaboration problem, but it introduced a new single point of failure. If the server went down, nobody could collaborate or access history. If the server's database became corrupted and backups failed, the entire project history was lost forever.

### Step 4: Distributed Version Control

Git is a distributed version control system. Every developer has a full copy of the entire repository — not just the latest snapshot, but the complete history of every file, every branch, and every commit. There is no single point of failure. If the server goes down, any developer's machine has the complete repository and can restore it.

The key innovation is that every clone is a full backup. You can work entirely offline: commit locally, browse history, create branches, and diff files. When you reconnect to the network, you synchronize with the remote repository.

### Step 5: Git Stores Snapshots, Not Diffs

This is the most important technical detail to understand about Git. Most other version control systems (SVN, CVS, Perforce) store changes as a list of file-based diffs. They think of version history as a set of files and the changes made to each file over time.

Git thinks of version history as a series of snapshots. Each commit stores a complete copy of every file in the project at that moment. If a file has not changed between two commits, Git does not duplicate the file — it links to the previous identical copy. This gives you the safety of full snapshots with the storage efficiency of diffs.

Internally, Git computes a SHA-1 hash of each file's content and uses that hash as the object identifier. Identical files always produce the same hash, so Git automatically deduplicates them.

<AsciiDiagram id="git-github-actions/what-is-version-control/overview/snapshots" title="Git Stores Snapshots, Not Diffs" content={`

### Step 6: Git Models History as a Directed Acyclic Graph

Each commit points to its parent commit, forming an immutable chain. This structure is called a Directed Acyclic Graph (DAG). The DAG guarantees that history is append-only — you cannot change a commit without creating a new commit that branches from it.

Each commit object stores:
- A pointer to the project tree (the directory listing of all files)
- One or more parent commit pointers (none for the first commit, one for normal commits, two for merge commits)
- The author name, email, and timestamp
- The committer name, email, and timestamp (can differ from author for rebased commits)
- A commit message

<AsciiDiagram id="git-github-actions/what-is-version-control/overview/dag" title="Git's Commit DAG" content={`

  │   └──┬──┘     └──┬──┘     └──┬──┘     └──┬──┘           │
  │      │           │           │           │               │
  │      ▼           ▼           ▼           ▼               │
  │   ┌──────────────────────────────────────────────────┐   │
  │   │  Each commit stores:                              │   │

  │  • Verifiable — any corruption is detected via hash       │

### Step 7: The Three Trees of Git

Git tracks your work across three data structures at all times. Understanding these three trees is the key to understanding every Git command.

1. **Working Directory** — The files on your filesystem that you edit with your editor or IDE. This is the only tree you interact with directly.
2. **Staging Area (Index)** — A file at `.git/index` that records what will go into the next commit. It is a buffer between your working directory and the repository.
3. **Repository** — The `.git/objects/` directory containing all commits, trees, and blobs. This is the permanent history.

The flow is always the same: you edit files in the working directory, you use `git add` to promote changes to the staging area, and you use `git commit` to freeze the staging area as a permanent snapshot in the repository.

## Key Commands Explained

These four commands form the foundation of daily Git usage. They are the first commands you should learn and the ones you will use most frequently.

### git init

Creates a new Git repository in the current directory. It creates a `.git` subdirectory that contains all of Git's internal data structures — the object database, references, configuration, and index. You run this once per project, at the beginning.

```bash
cd my-project
git init
# Output: Initialized empty Git repository in /path/to/my-project/.git/
```

After `git init`, your project is a Git repository, but it contains no commits. You have to add files and commit them to start tracking history.

### git clone

Creates a full copy of an existing remote repository. It downloads every commit, every branch, and every tag from the remote and sets up a local repository that is an exact mirror of the remote. This is how you start working on an existing project.

```bash
git clone https://github.com/username/project.git
# Output: Cloning into 'project'...
# Output: remote: Enumerating objects: 152, done.
# Output: Receiving objects: 100% (152/152), done.
```

### git status

Shows the current state of your working directory and staging area. It tells you which files are modified, which are staged, and which are untracked. Run this command constantly — it is the safest command in Git and never modifies anything.

```bash
git status
# Output: On branch main
# Output: Changes not staged for commit:
# Output:   modified:   src/app.py
```

### git log

Displays the commit history of your repository. Each entry shows the commit hash, author, date, and message. The `--oneline` flag condenses each commit to a single line, making it easy to scan the history.

```bash
git log --oneline
# Output: a1b2c3d fix: handle null pointer in payment
# Output: e5f6g7a feat: add user authentication
# Output: h8i9j0k Initial commit
```

## When to Use Version Control and When Not To

**Use version control when:**
- You are writing any amount of code, even a single script or configuration file
- You need to experiment with changes without losing a known-good state
- You collaborate with other people on the same files
- You need to diagnose when a bug was introduced by reviewing history
- You deploy software and need to know exactly which version is running on each environment
- You want to annotate each change with a rationale for future reference

**Do not use version control when:**
- You are editing large binary files that change entirely on every save (video files, 3D models, design assets over 100 MB) — Git can track them but the repository grows rapidly and every clone becomes slow
- You need fine-grained access control per file or directory — Git's permission model is per-repository, not per-file
- You are working with secrets or credentials — never commit passwords, API keys, tokens, or private certificates to a repository. Use environment variables, secret managers, or vault tools instead

Version control is not optional for professional software development. It is the foundation that code review, deployment, testing, and collaboration build on. Mastering the concepts in this section gives you the mental model you need to understand every Git command that follows.
