You will create a repository, make some commits, create branches, switch between them, and observe how the working directory changes. Every command works identically on Windows, macOS, and Linux.

## Setup

Open a terminal and run these commands to create a practice directory and initialize a Git repository:

```bash
mkdir -p ~/git-branching-lab
cd ~/git-branching-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Expected output:

```
Initialized empty Git repository in /home/you/git-branching-lab/.git/
```

(Windows shows the path with backslashes: `C:\Users\you\git-branching-lab\.git\`.)

Now create the starting files:

```bash
echo "Welcome to the branching lab" > README.md
git add README.md
git commit -m "Initial commit"
```

Expected output:

```
[main (root-commit) a1b2c3d] Initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
```

## The Walkthrough

### Step 1: Create your first branch

```bash
git branch first-feature
git log --oneline
```

Expected output:

```
a1b2c3d (HEAD -> main, first-feature) Initial commit
```

Both `main` and `first-feature` point at the same commit. `HEAD` is still on `main`. The branch was created but not switched to.

**Checkpoint:** Run `git branch` — you should see `first-feature` listed below `* main`.

### Step 2: Switch to the new branch

```bash
git switch first-feature
git log --oneline
```

Expected output:

```
a1b2c3d (HEAD -> first-feature, main) Initial commit
```

Now `HEAD` points at `first-feature`. Both branches are at the same commit, so no files changed on disk.

**Under the hood:** Git updated `.git/HEAD` from `ref: refs/heads/main` to `ref: refs/heads/first-feature`. No content was written or copied.

### Step 3: Make commits on your branch

```bash
echo "Feature file" > feature.txt
git add feature.txt
git commit -m "Add feature file"
echo "More feature work" >> feature.txt
git add feature.txt
git commit -m "Complete feature work"
git log --oneline
```

Expected output:

```
b2c3d4e (HEAD -> first-feature) Complete feature work
c3d4e5f Add feature file
a1b2c3d Initial commit
```

Your branch has moved forward. `main` still points at `a1b2c3d`.

```
  ((a1b2c3d))  ← main
       │
       └──▶ ((c3d4e5f)) ──▶ ((b2c3d4e))  ← first-feature
                                          ← HEAD
```

### Step 4: Switch back to main and see the isolation

```bash
git switch main
cat feature.txt
```

Expected output for `cat feature.txt`:

```
cat: feature.txt: No such file or directory
```

**This is the core of branching.** When you switched to `main`, Git rewrote your working directory to match `main`'s snapshot — which has no `feature.txt`. Your commits are safe on `first-feature`; they are not gone.

**Windows note:** On Windows, `type feature.txt` instead of `cat` gives `The system cannot find the file specified.`

### Step 5: Make a conflicting commit on main

```bash
echo "Main update" > main-update.txt
git add main-update.txt
git commit -m "Add main update"
git log --oneline
```

Expected output:

```
d4e5f6a (HEAD -> main) Add main update
a1b2c3d Initial commit
```

Now the branches have diverged:

```
  ((a1b2c3d)) ──▶ ((d4e5f6a))  ← main
       │                      ← HEAD
       └──▶ ((c3d4e5f)) ──▶ ((b2c3d4e))  ← first-feature
```

### Step 6: Switch back to your feature branch

```bash
git switch first-feature
ls
```

Expected output:

```
README.md   feature.txt
```

`main-update.txt` is gone; `feature.txt` is back. Each branch has exactly the files its history created.

**Checkpoint:** Run `git log --oneline --all --graph` to see both branches in one view:

```
* b2c3d4e (HEAD -> first-feature) Complete feature work
* c3d4e5f Add feature file
| * d4e5f6a (main) Add main update
|/
* a1b2c3d Initial commit
```

The `|/` shows where history diverged.

### Step 7: Create a second branch from the feature branch

```bash
git switch -c nested-feature
echo "Nested work" >> feature.txt
git add feature.txt
git commit -m "Nested feature work"
```

Expected output:

```
[first-feature e5f6a7b] Nested feature work
 1 file changed, 1 insertion(+)
```

Wait — the output says `first-feature` not `nested-feature`? No — let's check:

```bash
git branch
```

Expected output:

```
  first-feature
* nested-feature
```

You created `nested-feature` from `first-feature`'s tip, then switched to it. The commit message says `first-feature` because that was the branch name when you ran `git switch -c` — the parent branch name shown in the commit message is cosmetic, not structural.

```
  ((a1b2c3d)) ──▶ ((d4e5f6a))  ← main
       │
       └──▶ ((c3d4e5f)) ──▶ ((b2c3d4e))  ← first-feature
                                     │
                                     └──▶ ((e5f6a7b))  ← nested-feature
                                                       ← HEAD
```

### Step 8: Rename a branch

```bash
git branch -m nested-feature deeper-feature
git branch
```

Expected output:

```
  first-feature
* deeper-feature
```

The branch was renamed. `deeper-feature` still points at `e5f6a7b`. This only changes the label, not the history.

### Step 9: Delete a branch

```bash
git switch main
git branch -d deeper-feature
```

Expected output:

```
Deleted branch deeper-feature (was e5f6a7b).
```

Git refused? Actually, let's check. The `-d` flag only deletes if the branch is fully merged into the current branch. Since `deeper-feature` has commits not in `main` (`c3d4e5f`, `b2c3d4e`, `e5f6a7b`), `git branch -d` will refuse:

```
error: The branch 'deeper-feature' is not fully merged.
If you are sure you want to delete it, run 'git branch -D deeper-feature'.
```

That is a safety guard. Since you created this branch for practice, force-delete it:

```bash
git branch -D deeper-feature
```

Expected output:

```
Deleted branch deeper-feature (was e5f6a7b).
```

The commits `c3d4e5f`, `b2c3d4e`, and `e5f6a7b` still exist in Git's object store — they are not immediately deleted. They become unreachable from any named branch. Git's garbage collector (`git gc`) will eventually remove them.

## Common Errors at This Stage

### Error 1: `fatal: A branch named 'feature' already exists.`

You tried to create a branch with a name that already exists. Git branch names must be unique within your local repository.

**Fix:** List branches with `git branch` to see existing names, then pick a different one.

### Error 2: `fatal: The branch 'X' is not fully merged.` when deleting

You used `git branch -d` on a branch that has commits not present in the current branch. Git refuses to let you delete history silently.

**Fix:** Either merge the branch first, or use `git branch -D` (capital D) to force-delete. Only force-delete if you are certain you do not need those commits.

### Error 3: Working directory files mysteriously appear and disappear after `git switch`

You switched branches and files you were editing are suddenly gone (or files you never created appear). This is expected — `git switch` rewrites your working directory to match the target branch's commit snapshot.

**Fix:** Commit or stash your changes before switching. Run `git status` before and after switching to confirm the state. If you need to keep uncommitted work across a switch, use `git stash push`.

### Error 4: `fatal: 'origin/main' is not a commit and a branch 'main' cannot be created from it`

You tried to create a local branch from a remote-tracking branch before fetching. The remote reference does not exist locally.

**Fix:** Run `git fetch origin` first to update remote-tracking branches, then create your local branch.

## What You Just Learned

- A branch is a 41-byte pointer to a commit, stored in `.git/refs/heads/`
- `git branch <name>` creates a branch without switching to it
- `git switch <name>` moves `HEAD` and rewrites the working directory
- `git switch -c <name>` creates and switches in one command
- Branches let you isolate work-in-progress from the stable line
- `git branch -d` refuses to delete unmerged branches; `-D` force-deletes
- `git branch -m` renames a branch
- `git log --oneline --all --graph` visualizes the branching structure
