Work through these exercises in a terminal. The first tier reinforces single commands. The second tier combines branching with prior Git skills. The third tier presents messy real-world scenarios — there is more than one valid approach.

Create a fresh practice directory for each tier so you start clean.

---

## Tier 1 — Beginner

### Exercise 1

**Task:** Create a new directory, initialize a Git repo, make one commit with a file called `hello.txt`, then create a branch named `experiment` without switching to it. Verify both branches point at the same commit.

<details>
<summary>Hint</summary>
Use `git branch` to create the branch and `git log --oneline` to see both branches side by side.
</details>

### Solution

```bash
mkdir -p ~/branch-beginner-1 && cd ~/branch-beginner-1
git init
echo "Hello" > hello.txt
git add hello.txt
git commit -m "Initial commit"
git branch experiment
git log --oneline
```

Expected output shows both `main` and `experiment` at the same commit hash.

---

### Exercise 2

**Task:** On the `experiment` branch, create a file called `data.csv` with three lines of CSV data, commit it, then switch back to `main`. Confirm that `data.csv` is not present when you are on `main`.

<details>
<summary>Hint</summary>
Use `git switch experiment` to move to the branch, then create and commit the file. Then `git switch main` and list files.
</details>

### Solution

```bash
git switch experiment
echo -e "id,name,score\n1,Alice,95\n2,Bob,87" > data.csv
git add data.csv
git commit -m "Add data file"
git switch main
ls
```

On `main`, `data.csv` should not appear. The working directory reflects only `main`'s snapshot.

---

### Exercise 3

**Task:** List all branches including any remote-tracking references. Then rename the `experiment` branch to `analysis`.

<details>
<summary>Hint</summary>
`git branch -a` lists everything. `git branch -m <old> <new>` renames.
</details>

### Solution

```bash
git branch -a
git branch -m experiment analysis
git branch
```

The renamed branch `analysis` now appears in the list and `experiment` is gone.

### Exercise 4

**Task:** Create a branch called `temp-work`, switch to it, make one commit, switch back to `main`, then delete `temp-work` using the safe delete flag. What happens?

<details>
<summary>Hint</summary>
If the branch has commits not in `main`, `git branch -d` will refuse. Read the error message carefully.
</details>

### Solution

```bash
git branch temp-work
git switch temp-work
echo "Temporary" > temp.txt
git add temp.txt
git commit -m "Temp commit"
git switch main
git branch -d temp-work
```

Git refuses with: `error: The branch 'temp-work' is not fully merged.` Use `git branch -D temp-work` to force-delete (or merge first).

## Tier 2 — Intermediate

### Exercise 1

**Task:** Create a repo with 3 commits on `main`. Then create a branch `feature-1`, add 2 more commits on it. Switch back to `main` and create another branch `feature-2` with 1 commit. Use `git log --oneline --all --graph` to confirm the branches form a fork shape. How many commits are reachable only from `feature-1`?

<details>
<summary>Hint</summary>
Count commits that appear in `git log feature-1` but not in `git log main`. You can also use `git log main..feature-1` to see the exclusive set.
</details>

### Solution

```bash
mkdir -p ~/branch-inter-1 && cd ~/branch-inter-1
git init
echo "1" > f.txt && git add f.txt && git commit -m "Commit 1"
echo "2" > f.txt && git add f.txt && git commit -m "Commit 2"
echo "3" > f.txt && git add f.txt && git commit -m "Commit 3"
git branch feature-1
git switch feature-1
echo "A" > f.txt && git add f.txt && git commit -m "Feature A"
echo "B" > f.txt && git add f.txt && git commit -m "Feature B"
git switch main
git branch feature-2
git switch feature-2
echo "X" > f.txt && git add f.txt && git commit -m "Feature X"
git log --oneline --all --graph
```

The graph should show three branches diverging from commit 3. `git log main..feature-1` shows exactly 2 commits (Feature A and Feature B).

### Exercise 2

**Task:** Start a repo with `main` having 2 commits. Create a branch `bugfix`, make 2 commits there. Switch back to `main` and create 2 more commits. Now create a third branch `integration` from `bugfix`'s second commit — not from `main` or `bugfix`'s tip. Confirm the new branch shares the same parent as `bugfix`'s second commit.

<details>
<summary>Hint</summary>
Use `git log bugfix` to find the hash of the second commit, then `git branch integration <hash>` to create the branch at that specific commit.
</details>

### Solution

```bash
mkdir -p ~/branch-inter-2 && cd ~/branch-inter-2
git init
echo "base" > f.txt && git add f.txt && git commit -m "Base"
echo "second" > f.txt && git add f.txt && git commit -m "Second"
git branch bugfix
git switch bugfix
echo "fix1" > f.txt && git add f.txt && git commit -m "Fix 1"
echo "fix2" > f.txt && git add f.txt && git commit -m "Fix 2"
git switch main
echo "main3" > f.txt && git add f.txt && git commit -m "Main 3"
echo "main4" > f.txt && git add f.txt && git commit -m "Main 4"
git log bugfix --oneline
# Pick the hash of "Fix 1" — let's say it is abc1234
git branch integration <hash-of-Fix-1>
git log --oneline integration
```

`integration` should show only commits up to and including "Fix 1" — no "Fix 2", no "Main 3" or "Main 4".

### Exercise 3

**Task:** Create a branch, make commits, then rename it to a more descriptive name. Then rename the current branch without specifying the old name (hint: `git branch -m <newname>` renames the current branch when given only one argument). Verify with `git branch`.

<details>
<summary>Hint</summary>
`git branch -m <new-name>` — omitting the old name renames whatever branch you are currently on.
</details>

### Solution

```bash
mkdir -p ~/branch-inter-3 && cd ~/branch-inter-3
git init && echo "start" > f.txt && git add f.txt && git commit -m "Start"
git switch -c poorly-named
echo "work" >> f.txt && git add f.txt && git commit -m "Work"
git branch -m well-named
git branch
```

The current branch is now `well-named`; `poorly-named` no longer exists.

## Tier 3 — Advanced

### Exercise 1: Rescue an orphaned branch

**Scenario:** You open a project and realize the previous developer left a branch's work in the repo but deleted the branch reference. The commits exist but are unreachable from any named branch.

**Task:** Use `git log` or similar commands to find the "orphaned" commits (hint: `git reflog` shows previous HEAD positions even after branch deletion). Recreate a branch at the latest orphaned commit.

<details>
<summary>Hint</summary>
Run `git reflog` to see all HEAD movements. Look for checkouts and commits that happened after the branch was deleted. The `@{N}` syntax or the commit hash lets you recreate the branch: `git branch rescued <hash>`.
</details>

### Solution — one valid approach

First, set up the scenario (if you do not have one):

```bash
mkdir -p ~/branch-adv-1 && cd ~/branch-adv-1
git init && echo "start" > f.txt && git add f.txt && git commit -m "Start"
git switch -c doomed
echo "important" > data.txt && git add data.txt && git commit -m "Important work"
git switch main
git branch -D doomed
```

Now the rescue:

```bash
git reflog
```

Find the line showing the commit on `doomed`, e.g.:

```
abc1234 HEAD@{0}: checkout: moving from doomed to main
def5678 HEAD@{1}: commit: Important work
```

Create a branch at that commit:

```bash
git branch rescued def5678
git log rescued --oneline
```

You should see both "Important work" and "Start".

**Other valid approaches:**
- `git fsck --lost-found` finds dangling commits Git has not garbage-collected
- `git log --all --reflog` lists every commit reachable from any reflog entry

### Exercise 2: Design a branching strategy for a release cycle

**Scenario:** You maintain a library with three active versions. Version 1 is in maintenance (only critical bugfixes). Version 2 is the current stable release. Version 3 is in development on a long-lived feature branch. A critical security bug is found that affects all three versions.

**Task:** You do not need to actually fix the bug. Design the branch operations to:

1. Create a fix branch that diverges from the oldest common ancestor of all three versions
2. Apply the fix to all three active lines
3. Clean up the fix branches

Write the exact `git branch` and `git switch` commands you would run, in order. There is no single correct answer — explain your reasoning.

<details>
<summary>Hint</summary>
Think about tagging the release points. `git merge-base` finds the common ancestor commit between any two branches.
</details>

### Solution — one valid approach

```bash
# Find the common ancestor of all three lines
git merge-base v1 v3   # returns hash, say ABC
# Create a fix branch from that ancestor
git switch -c security-fix ABC
# Apply the fix
echo "patch" > fix.patch && git add fix.patch && git commit -m "Security fix"
# Switch to v1 and cherry-pick or merge
git switch v1
git merge security-fix   # fast-forwards v1 to security-fix if v1 was at ABC
# Switch to v2
git switch v2
git merge security-fix   # normal merge if v2 has diverged
# Switch to v3
git switch v3
git merge security-fix   # merge into development line
# Clean up
git branch -d security-fix
```

**Why this approach:** Starting from `merge-base` ensures the fix applies cleanly to all branches with minimal conflict risk. Cherry-picking (using `git cherry-pick <hash>`) is another valid approach that avoids creating a shared merge commit but requires repeating the fix on each branch.

### Exercise 3: Untangle a confused working directory

**Scenario:** You have been working on `main` for two hours and have three uncommitted files: two belong on a new feature branch, the third is a quick hotfix that should go to `main`. You have not staged anything yet.

**Task:** Without losing any changes, create a feature branch containing only the two feature files, commit them, switch back to `main`, and commit the hotfix there.

<details>
<summary>Hint</summary>
`git stash` can selectively stash files. Or you can create the branch first (which carries uncommitted changes with you), then selectively stage and commit.
</details>

### Solution — one valid approach

```bash
# Create a feature branch — uncommitted changes follow the switch
git switch -c new-feature
git add feature-file-1.py feature-file-2.py
git commit -m "Add feature files"
# Uncommitted hotfix file is still there, still unstaged
git switch main
git add hotfix.py
git commit -m "Hotfix"
```

**Why this works:** When you switch to a branch that has no conflicting changes with your working tree, Git lets you carry uncommitted changes across the switch. The two feature files were staged and committed on `new-feature`; the hotfix file was never part of that commit, so it remained in the working directory when you switched back to `main`.

**Alternative approach:** Stash the feature files, switch to `main`, commit the hotfix, then pop the stash on a new branch. Same result, more commands.
