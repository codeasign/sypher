# Security Scanning with CodeQL and Dependabot — Practice Exercises

These exercises are organized by difficulty. Start with the Beginner tier to build confidence, then move to Intermediate and Advanced as you gain experience.

## Tier 1 — Beginner

### Exercise 1.1: Enable Dependabot for npm Dependencies

**Task:** Create a `.github/dependabot.yml` file that enables Dependabot for npm dependencies in the root directory, running on a weekly schedule, with a limit of 3 open pull requests.

**Hint:** The `package-ecosystem` for npm is `npm`, not `node`. The `directory` is `/` for the root of the repository. The `schedule.interval` accepts `daily`, `weekly`, or `monthly`.

**Solution:**

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
    open-pull-requests-limit: 3
```

### Exercise 1.2: Run CodeQL Analysis on Default Setup

**Task:** Create a GitHub Actions workflow that runs CodeQL analysis on JavaScript code. The workflow should trigger on pushes to `main` and on pull requests targeting `main`. Use the default query suite (do not specify `queries`).

**Hint:** You need three actions: `actions/checkout@v4`, `github/codeql-action/init@v3`, and `github/codeql-action/analyze@v3`. The `autobuild` step is also needed. The workflow needs `security-events: write` permission.

**Solution:**

```yaml
# .github/workflows/codeql.yml
name: CodeQL Analysis

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  analyze:
    name: Analyze
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write

    strategy:
      fail-fast: false
      matrix:
        language: [javascript]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
```

### Exercise 1.3: View Dependabot Alerts in the Security Tab

**Task:** After Dependabot has run at least once on your repository, navigate to the Dependabot alerts and find an alert for a vulnerable dependency. Describe what you see: the severity, the affected version range, the patched version, and the remediation recommendation.

**Hint:** The Dependabot alerts are at `https://github.com/YOUR-USERNAME/YOUR-REPO/security/dependabot`. Each alert shows the CVE identifier, the CVSS score, the package name, the vulnerable version range, and the fixed version. You can click an alert to see the full advisory.

**Solution:**

Navigate to the Security tab of your repository. Click **Dependabot alerts** in the left sidebar. You will see a list of alerts, each showing:

- **Package name** (e.g., `lodash`)
- **Severity** (Critical, High, Medium, Low)
- **CVE identifier** (e.g., `CVE-2020-8203`)
- **Affected version range** (e.g., `< 4.17.20`)
- **Fixed version** (e.g., `4.17.20`)

Click into an alert. The detail view shows:
- A description of the vulnerability
- The CVSS vector string and score
- The path to the manifest file that declares the dependency
- A **Create Dependabot security update** button that opens a PR with the fix
- The GitHub advisory with references and affected products

### Exercise 1.4: View CodeQL Alerts in the Security Tab

**Task:** After a CodeQL workflow run has completed, find the CodeQL alerts in the Security tab. Filter alerts by severity and by query type. Identify one alert, read its description, and locate the affected source code line.

**Hint:** CodeQL alerts are under **Security** → **Code scanning alerts**. You can filter by severity (Critical, High, Medium, Low, Error) and by query name. Each alert shows the file path, line number, and a description of the vulnerability. Click the alert to see the data flow path from source to sink.

**Solution:**

Navigate to the Security tab of your repository. Click **Code scanning** in the left sidebar. You will see a list of alerts grouped by query. Each alert card shows:

- **Rule name** (e.g., `Command injection`, `SQL injection`, `Cross-site scripting`)
- **Severity** (Error, Warning, Note)
- **File path** and **line number** (e.g., `src/index.js:42`)
- **Status** (Open, Fixed, Dismissed)

Click an alert. The detail view shows:
- The affected code snippet with the problematic line highlighted
- A data flow path from the source (user input) to the sink (dangerous function call)
- A description of the vulnerability class
- Remediation guidance

Use the filter bar at the top to narrow results:
- Filter by severity: click the severity label
- Filter by branch: select the branch from the dropdown
- Filter by query: type the query name in the search box

## Tier 2 — Intermediate

### Exercise 2.1: Configure Dependabot for Two Ecosystems

**Task:** Create a Dependabot configuration that monitors both npm dependencies and Docker base images. The npm updates should run weekly on Monday. The Docker updates should run monthly. Both should use labels.

**Hint:** The `updates` array accepts multiple entries, each with its own `package-ecosystem`, `directory`, and `schedule`. The Docker ecosystem identifier is `docker`. You can set different schedules for each entry.

**Solution:**

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
      day: monday
      time: "09:00"
    labels:
      - dependencies
      - npm

  - package-ecosystem: docker
    directory: /
    schedule:
      interval: monthly
      day: monday
      time: "07:00"
    labels:
      - dependencies
      - docker
```

### Exercise 2.2: Add Ignore Rules to Skip Certain Dependency Updates

**Task:** Extend the Dependabot configuration to ignore updates for `express` (any version 5.x) and for all `@types/*` packages. Configure it so Dependabot will still alert on security vulnerabilities for these packages but will not open PRs for non-security version bumps.

**Hint:** The `ignore` block accepts `dependency-name` with optional `versions` using semver range syntax. Use `">=5.x"` for express and `"@types/*"` with the wildcard pattern. Note that Dependabot still opens security PRs for ignored dependencies if a vulnerability is found.

**Solution:**

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
    ignore:
      - dependency-name: express
        versions:
          - ">=5.x"
      - dependency-name: "@types/*"
```

### Exercise 2.3: Customize CodeQL Queries to Run `security-extended` Only

**Task:** Create a CodeQL workflow that runs the `security-extended` query suite instead of the default or `security-and-quality` suites. Use a matrix for Python and JavaScript languages. Add a comment explaining why you chose `security-extended` over the default.

**Hint:** Pass `queries: security-extended` in the `with` block of the `init` step. The `security-extended` suite includes more queries than the default but fewer than `security-and-quality`. It is a good middle ground for teams that want broader coverage without the noise of experimental quality rules.

**Solution:**

```yaml
# .github/workflows/codeql.yml
name: CodeQL Security Extended

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  analyze:
    name: Analyze (${{ matrix.language }})
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write

    strategy:
      fail-fast: false
      matrix:
        language: [javascript, python]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          # security-extended provides more coverage than the default
          # suite while producing less noise than security-and-quality
          queries: security-extended

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
```

### Exercise 2.4: Set Up Dependabot Auto-Approve Workflow

**Task:** Create a GitHub Actions workflow that automatically approves Dependabot pull requests for both patch and minor version updates. The workflow should only run when the PR author is `dependabot[bot]`.

**Hint:** Use `pull_request_target` as the trigger event. Use `dependabot/fetch-metadata@v2` to get the update type. Check `steps.metadata.outputs.update-type` for `version-update:semver-patch` or `version-update:semver-minor`. The `GITHUB_TOKEN` must have `pull-requests: write` permission.

**Solution:**

```yaml
# .github/workflows/dependabot-auto-approve.yml
name: Dependabot Auto-Approve

on: pull_request_target

permissions:
  pull-requests: write

jobs:
  approve:
    runs-on: ubuntu-latest
    if: github.actor == 'dependabot[bot]'
    steps:
      - name: Fetch Dependabot metadata
        id: metadata
        uses: dependabot/fetch-metadata@v2

      - name: Approve patch and minor updates
        if: |
          steps.metadata.outputs.update-type == 'version-update:semver-patch' ||
          steps.metadata.outputs.update-type == 'version-update:semver-minor'
        run: gh pr review --approve "$PR_URL"
        env:
          PR_URL: ${{github.event.pull_request.html_url}}
          GH_TOKEN: ${{secrets.GITHUB_TOKEN}}
```

## Tier 3 — Advanced

### Exercise 3.1: Design a Security Pipeline — Dependabot PR Blocked by CodeQL

**Task:** Design a workflow where:
1. Dependabot creates a PR with a dependency update.
2. CodeQL scans the PR branch.
3. If CodeQL finds a critical severity alert, the PR is automatically blocked (a failing status check).
4. A label is applied to the PR based on the highest severity found: `security-critical`, `security-high`, or `security-low`.

You do not need to implement this fully, but provide the complete workflow YAML and explain the logic.

**Hint:** This requires two workflow files. The first triggers on `pull_request_target` from Dependabot. The second triggers on `push` to any branch that starts with `dependabot/`. For the label logic, you can use the `github/codeql-action/analyze` output or query the code scanning alerts API after the analysis completes. Use `gh api` to fetch the alerts and determine severity.

**Solution:**

This is a multi-workflow strategy. Here is the design:

**Workflow 1** — Trigger a CodeQL scan on Dependabot PR branches:

```yaml
# .github/workflows/codeql-dependabot-pr.yml
name: CodeQL Scan Dependabot PR

on:
  push:
    branches:
      - dependabot/**

jobs:
  analyze:
    name: Analyze Dependabot PR
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write
      pull-requests: write

    strategy:
      matrix:
        language: [javascript]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          ref: ${{ github.ref }}

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: security-and-quality

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3

      - name: Fetch CodeQL alerts and apply label
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          # Get the PR number from the Dependabot branch name
          PR_NUMBER=$(echo "${{ github.ref }}" | grep -oP 'dependabot/\K\d+')
          if [ -z "$PR_NUMBER" ]; then
            echo "Could not determine PR number from branch name"
            exit 0
          fi

          # Fetch all code scanning alerts for this PR
          ALERTS=$(gh api \
            /repos/${{ github.repository }}/code-scanning/alerts \
            --jq '[.[] | select(.state == "open") | .rule.severity]')

          # Determine highest severity
          if echo "$ALERTS" | grep -q "error"; then
            LABEL="security-critical"
          elif echo "$ALERTS" | grep -q "warning"; then
            LABEL="security-high"
          elif echo "$ALERTS" | grep -q "note"; then
            LABEL="security-low"
          else
            echo "No alerts found"
            exit 0
          fi

          # Apply label to the PR
          gh pr edit "$PR_NUMBER" --add-label "$LABEL"

          # If critical, add a comment
          if [ "$LABEL" = "security-critical" ]; then
            gh pr comment "$PR_NUMBER" \
              --body "CodeQL found critical severity alerts on this branch. Review before merging."
          fi
```

**Workflow 2** — Block PRs with critical alerts using branch protection:

You cannot block a PR from a workflow alone. Instead, configure a **branch protection rule** on your default branch:
- Go to **Settings** → **Branches** → **Add rule**
- Set **Branch name pattern** to `main`
- Enable **Require status checks to pass before merging**
- Add `Analyze Dependabot PR / Analyze (javascript)` as a required check

If CodeQL finds a critical vulnerability, the workflow fails because you can add a step that exits with a non-zero code:

```yaml
      - name: Block PR if critical alerts found
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          ALERTS=$(gh api \
            /repos/${{ github.repository }}/code-scanning/alerts \
            --jq '[.[] | select(.state == "open") | .rule.severity]')
          if echo "$ALERTS" | grep -q "error"; then
            echo "::error::Critical security alerts found in this PR branch"
            exit 1
          fi
```

### Exercise 3.2: Create a Custom CodeQL Query

**Task:** Write a custom CodeQL query that detects `console.log()` statements in production code. Run this query alongside the default `security-and-quality` suite. The query should flag any call to `console.log`, `console.warn`, `console.error`, or `console.info` outside of test files.

**Hint:** Create a `custom-queries/` directory at the root of your repository. Write a `.ql` file. For JavaScript/TypeScript, the query uses the `javascript` QL library. You can reference the CodeQL standard library for method calls. To run it alongside default queries, pass the path to the custom query file in the `queries` field of the `init` step.

**Solution:**

Create the custom query directory and file:

```bash
mkdir -p custom-queries
```

Create `custom-queries/no-console-log.ql`:

```ql
/**
 * @name No console.log in production code
 * @description Flags console.log, console.warn, console.error, and console.info calls
 * @kind problem
 * @severity warning
 * @id custom/no-console-log
 */

import javascript

from MethodCallExpr call
where
  call.getMethodName() = "log" or
  call.getMethodName() = "warn" or
  call.getMethodName() = "error" or
  call.getMethodName() = "info" or
  call.getMethodName() = "debug"
  and call.getReceiver().(VarAccess).getName() = "console"
  and not call.getFile().getRelativePath().matches("%test%")
  and not call.getFile().getRelativePath().matches("%__tests__%")
select call, "Console output should not be left in production code."
```

Update your CodeQL workflow to include the custom query:

```yaml
# .github/workflows/codeql.yml
name: CodeQL with Custom Queries

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  analyze:
    name: Analyze
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write

    strategy:
      matrix:
        language: [javascript]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: security-and-quality,./custom-queries/no-console-log.ql

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
```

### Exercise 3.3: Multi-Repo Security Monitoring

**Task:** Design a strategy to aggregate SARIF results from multiple repositories into a central monitoring dashboard. You have 10 microservice repositories, each with CodeQL enabled. You want a single place to view all security alerts across all services.

**Hint:** You can use a central orchestrator repository that runs a workflow on a schedule, fetches SARIF files from each microservice repo, and uploads them to GitHub's Security tab. Alternatively, you can use GitHub's organizational-level code scanning or a third-party SIEM that ingests SARIF. The key challenge is cross-repo aggregation.

**Solution:**

This is a multi-repo strategy. Here is the design:

**Step 1: In each microservice repo**, upload SARIF as a workflow artifact after CodeQL analysis:

```yaml
# In each microservice: .github/workflows/codeql.yml
name: CodeQL Analysis

on:
  push:
    branches: [main]
  schedule:
    - cron: "0 6 * * 1"

jobs:
  analyze:
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write

    steps:
      - uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: [javascript]
          queries: security-and-quality

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3

      - name: Upload SARIF as artifact
        uses: actions/upload-artifact@v4
        with:
          name: codeql-results-${{ github.repository }}
          path: results/*.sarif
```

**Step 2: In a central monitoring repo**, create a workflow that runs weekly, downloads SARIF from each microservice, and uploads them:

```yaml
# In central-monitoring repo: .github/workflows/aggregate-sarif.yml
name: Aggregate Security Results

on:
  schedule:
    - cron: "0 8 * * 1"
  workflow_dispatch:

jobs:
  aggregate:
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write

    env:
      GH_TOKEN: ${{ secrets.ORG_PAT }}
      # This PAT needs read access to all microservice repos
      # and write access to the central monitoring repo

    steps:
      - name: Checkout monitoring repo
        uses: actions/checkout@v4

      - name: Download SARIF from all microservices
        run: |
          REPOS=(
            "org/auth-service"
            "org/payment-service"
            "org/user-service"
            "org/notification-service"
            "org/inventory-service"
            "org/order-service"
            "org/search-service"
            "org/gateway-service"
            "org/analytics-service"
            "org/file-service"
          )

          mkdir -p sarif-results

          for REPO in "${REPOS[@]}"; do
            echo "Downloading SARIF from $REPO"
            # Get the latest completed workflow run ID
            RUN_ID=$(gh api \
              /repos/$REPO/actions/workflows/codeql.yml/runs \
              --jq '.workflow_runs[] | select(.status == "completed") | .id' \
              | head -1)

            if [ -n "$RUN_ID" ]; then
              # Download the artifact
              gh run download "$RUN_ID" \
                --repo "$REPO" \
                --dir "sarif-results/$REPO"
            else
              echo "No completed run found for $REPO"
            fi
          done

      - name: Upload aggregated SARIF
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: sarif-results
          category: aggregated-codeql

      - name: Generate summary report
        run: |
          echo "## Security Scan Summary" >> $GITHUB_STEP_SUMMARY
          echo "" >> $GITHUB_STEP_SUMMARY
          echo "| Repository | Alerts |" >> $GITHUB_STEP_SUMMARY
          echo "|---|---|" >> $GITHUB_STEP_SUMMARY

          for REPO_DIR in sarif-results/*/; do
            REPO_NAME=$(basename "$REPO_DIR")
            ALERT_COUNT=$(find "$REPO_DIR" -name "*.sarif" -exec \
              python3 -c "
import json, sys
data = json.load(open(sys.argv[1]))
runs = data.get('runs', [])
total = sum(len(r.get('results', [])) for r in runs)
print(total)
" {} \;)
            echo "| $REPO_NAME | $ALERT_COUNT |" >> $GITHUB_STEP_SUMMARY
          done
```

**Alternative: Organizational-level code scanning**

If your organization is on GitHub Enterprise Cloud, you can use the organizational code scanning API to query alerts across all repositories:

```bash
# List all code scanning alerts across the organization
gh api /orgs/ORG/code-scanning/alerts --paginate \
  --jq '.[] | "\(.repository.name): \(.rule.description) (\(.rule.severity))"'
```

This approach avoids the need for a central monitoring repo entirely. The organizational API aggregates results from all repos that have code scanning enabled. You can integrate this into a scheduled workflow, a Slack notification system, or a custom dashboard.

## Summary

After completing these exercises, you should be comfortable with:

- **Configuring Dependabot** for one or more package ecosystems with custom schedules, labels, and ignore rules
- **Setting up CodeQL workflows** with different query suites and path filters
- **Viewing and triaging alerts** in both the Dependabot and Code scanning sections of the Security tab
- **Automating Dependabot PR handling** with auto-approve and auto-merge workflows
- **Writing custom CodeQL queries** to enforce project-specific rules
- **Designing multi-repo security monitoring** strategies for organizational-scale deployments
