# Security Scanning with CodeQL and Dependabot — Let's Work Together

In this hands-on exercise, you will set up Dependabot and CodeQL for a Node.js project from scratch. You will create a repository, introduce a real security vulnerability, watch CodeQL catch it, and configure automated dependency updates.

## Setup

You need a GitHub account and a local environment with the following:

- git installed and configured
- Node.js 18 or newer (for the sample project)
- npm (ships with Node.js)
- A GitHub personal access token (classic or fine-grained) with repo scope

Create a working directory to hold your practice project:

```bash
mkdir security-practice
cd security-practice
git init
```

### Sample Project Files

Create a minimal Node.js project with some dependencies. One of these dependencies will have a known vulnerability so you can see Dependabot in action.

**package.json:**

```json
{
  "name": "security-practice",
  "version": "1.0.0",
  "private": true,
  "description": "A practice project for security scanning",
  "main": "index.js",
  "scripts": {
    "start": "node index.js"
  },
  "dependencies": {
    "express": "^4.17.1",
    "lodash": "4.17.15",
    "axios": "^0.21.1",
    "ejs": "^3.1.6"
  }
}
```

**index.js:**

```javascript
const express = require('express');
const { exec } = require('child_process');
const app = express();

app.use(express.json());

app.get('/health', (req, res) => {
  res.json({ status: 'ok' });
});

app.post('/greet', (req, res) => {
  const name = req.body.name;
  res.send('<h1>Hello ' + name + '</h1>');
});

app.post('/run', (req, res) => {
  const cmd = req.body.command;
  exec(cmd, (error, stdout) => {
    if (error) {
      res.status(500).send(error.message);
    } else {
      res.send(stdout);
    }
  });
});

app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

This `index.js` intentionally contains security issues: an unstripped XSS pattern in the `/greet` endpoint and a command injection vulnerability in the `/run` endpoint. CodeQL will flag these when you run analysis.

**Create the project, then initialize it:**

```bash
npm init -y
npm install
```

## Walkthrough

### Step 1: Create the Repository and Push

Create a new repository on GitHub. You can do this through the GitHub web UI or using `gh` CLI.

```bash
# Using gh CLI
gh repo create security-practice --public --source=. --remote=origin --push

# If you did not use gh, connect the remote manually:
# git remote add origin https://github.com/YOUR-USERNAME/security-practice.git
# git add .
# git commit -m "Initial commit with vulnerable project"

```

Once pushed, go to your repository on GitHub. Under the **Insights** tab, click **Dependency graph**. You will see the dependencies from your `package-lock.json` listed there.

### Step 2: Enable Dependabot

Create the Dependabot configuration file.

```bash
mkdir -p .github
```

Create `.github/dependabot.yml`:

```yaml
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
      day: monday
      time: "09:00"
    open-pull-requests-limit: 5
    labels:
      - dependencies
      - security
    assignees:
      - YOUR_GITHUB_USERNAME
```

Replace `YOUR_GITHUB_USERNAME` with your actual GitHub username.

Commit and push:

```bash
git add .github/dependabot.yml
git commit -m "Enable Dependabot for npm dependencies"
git push
```

### Step 3: Check Dependabot Runs

Dependabot will not trigger immediately because it runs on the schedule you configured (weekly, on Monday at 09:00). However, you can see that it is active.

Navigate to your repository on GitHub and go to:
- **Insights** tab → **Dependency graph** → **Dependabot**

Here you will see the Dependabot configuration status. You can also manually trigger a check by going to the **Security** tab → **Dependabot alerts** — GitHub may scan on demand after you commit the configuration.

If Dependabot has not run yet, you will see a message indicating when the next scheduled check occurs. For the purposes of this exercise, knowing your configuration is correct is sufficient.

### Step 4: Create the CodeQL Workflow

Create `.github/workflows/codeql.yml`:

```bash
mkdir -p .github/workflows
```

```yaml
name: CodeQL Analysis

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: "0 8 * * 1"

jobs:
  analyze:
    name: Analyze
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write

    strategy:
      fail-fast: false
      matrix:
        language: [javascript]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: security-and-quality

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
        with:
          category: "/language:${{matrix.language}}"
```

Commit and push:

```bash
git add .github/workflows/codeql.yml
git commit -m "Add CodeQL analysis workflow with security-and-quality queries"
git push
```

### Step 5: Observe the CodeQL Analysis Run

On your GitHub repository, go to the **Actions** tab. You should see a workflow run titled **CodeQL Analysis** that was triggered by your latest push.

Click into the run to see the steps:
1. **Checkout repository** — clones your code
2. **Initialize CodeQL** — downloads CodeQL CLI and sets up the database
3. **Autobuild** — automatically detects the build system (for JavaScript, it runs `npm install`)
4. **Perform CodeQL Analysis** — runs the `security-and-quality` query suite against your code

### Step 6: Introduce a Security Flaw and Verify Detection

Now make the vulnerability more obvious so CodeQL definitely catches it. Update `index.js` with a blatant SQL injection pattern (even though there is no actual database, CodeQL will flag the pattern):

```javascript
const express = require('express');
const { exec } = require('child_process');
const app = express();

app.use(express.json());

app.get('/health', (req, res) => {
  res.json({ status: 'ok' });
});

app.post('/login', (req, res) => {
  const username = req.body.username;
  const password = req.body.password;
  // Dangerous: direct string interpolation into a query
  const query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'";
  console.log('Executing query:', query);
  res.send('Login attempted');
});

app.post('/run', (req, res) => {
  const cmd = req.body.command;
  exec(cmd, (error, stdout) => {
    if (error) {
      res.status(500).send(error.message);
    } else {
      res.send(stdout);
    }
  });
});

app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

Commit and push:

```bash
git add index.js
git commit -m "Add login endpoint with SQL injection pattern"
git push
```

Go back to the **Actions** tab. You will see a new CodeQL workflow run. When it completes, go to the **Security** tab → **Code scanning**. You should see new alerts:

- **SQL injection** — flagged in `/login` endpoint because the query string is built with string concatenation from user input
- **Command injection** — flagged in `/run` endpoint because `exec()` receives user-controlled input

Click any alert to see the affected lines, the data flow path from user input to the sink, and a description of the vulnerability.

### Step 7: Configure Dependabot Auto-Approve and Auto-Merge

Create `.github/workflows/dependabot-auto-merge.yml`:

```yaml
name: Dependabot Auto-Merge

on: pull_request_target

permissions:
  pull-requests: write
  contents: write

jobs:
  auto-merge:
    runs-on: ubuntu-latest
    if: github.actor == 'dependabot[bot]'
    steps:
      - name: Fetch Dependabot metadata
        id: metadata
        uses: dependabot/fetch-metadata@v2
        with:
          alert-lookup: true

      - name: Approve patch update PRs
        if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr review --approve "$PR_URL"
        env:
          PR_URL: ${{github.event.pull_request.html_url}}
          GH_TOKEN: ${{secrets.GITHUB_TOKEN}}

      - name: Enable auto-merge for patch updates
        if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr merge --auto --squash "$PR_URL"
        env:
          PR_URL: ${{github.event.pull_request.html_url}}
          GH_TOKEN: ${{secrets.GITHUB_TOKEN}}

      - name: Approve minor update PRs
        if: steps.metadata.outputs.update-type == 'version-update:semver-minor'
        run: gh pr review --approve "$PR_URL"
        env:
          PR_URL: ${{github.event.pull_request.html_url}}
          GH_TOKEN: ${{secrets.GITHUB_TOKEN}}
```

This workflow auto-approves and auto-merges Dependabot PRs for patch updates, and auto-approves (but does not auto-merge) minor updates. You can adjust the logic based on your risk tolerance.

Commit and push:

```bash
git add .github/workflows/dependabot-auto-merge.yml
git commit -m "Add Dependabot auto-approve and auto-merge for patch updates"
git push
```

### Step 8: Configure Query Filters for CodeQL

You may want to exclude test files or vendor directories from CodeQL analysis to reduce noise. Update your CodeQL workflow to add path filters:

```yaml
      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: security-and-quality

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
        with:
          category: "/language:${{matrix.language}}"
          paths: src
          paths-ignore: |
            src/vendor
            **/*.test.js
            **/tests/**
```

Commit and push:

```bash
git add .github/workflows/codeql.yml
git commit -m "Add path filters to exclude test and vendor files from CodeQL"
git push
```

Your CodeQL workflow now ignores test files and vendor directories, focusing analysis only on your application source code.

## Common Errors

### Error 1: CodeQL analysis failed — no code found

**Symptom:** The CodeQL analysis step completes without finding any results, even though you know your code has vulnerabilities.

**Cause:** The `autobuild` step may not have detected the correct build system. For compiled languages (Java, C++, Go), CodeQL needs to intercept the compilation to build its database. If autobuild fails silently, the database will be empty.

**Fix:** Explicitly specify the build step. For Maven projects, add:

```yaml
- name: Build with Maven
  run: mvn clean compile

- name: Perform CodeQL Analysis
  uses: github/codeql-action/analyze@v3
```

For JavaScript and Python, autobuild usually works, but if you see no results, verify that `npm install` completed successfully by adding an explicit install step.

### Error 2: Dependabot YAML is invalid

**Symptom:** You push `.github/dependabot.yml` but Dependabot does not start. No alerts appear. No PRs are created.

**Cause:** The YAML file has a syntax error or uses an unsupported `package-ecosystem` value. Common mistakes include:
- Using a tab instead of spaces for indentation
- Misspelling `package-ecosystem` (e.g., `npm` is correct, `node` is not)
- Missing the `version: 2` line at the top

**Fix:** Validate your YAML locally before pushing:

```bash
# Install a YAML linter
npm install -g yaml-lint

# Validate the file
yaml-lint .github/dependabot.yml

# Or use Python
python -c "import yaml; yaml.safe_load(open('.github/dependabot.yml'))"
```

You can also check the **Insights** tab → **Dependency graph** → **Dependabot** tab for error messages. GitHub displays YAML parsing errors there.

### Error 3: SARIF upload failed — unauthorized

**Symptom:** The CodeQL analysis runs successfully, but the SARIF upload step fails with a 403 or 401 error.

**Cause:** The workflow does not have sufficient permissions to write security events. The `GITHUB_TOKEN` used by the workflow needs `security-events: write` permission.

**Fix:** Add the correct permissions at the job level:

```yaml
jobs:
  analyze:
    permissions:
      actions: read
      contents: read
      security-events: write
```

If you are using a custom token instead of `GITHUB_TOKEN`, ensure it has the `repo` scope and the `security_events` write permission.

### Error 4: Language not supported by CodeQL

**Symptom:** The CodeQL workflow fails during initialization with an error like `CodeQL language 'typescript' is not supported`.

**Cause:** You specified a language that CodeQL does not support, or you used the wrong name for a supported language.

**Fix:** Use one of these supported language identifiers:
- `cpp` (C and C++)
- `csharp` (C#)
- `go`
- `java` (also covers Kotlin)
- `javascript` (also covers TypeScript)
- `python`
- `ruby`
- `swift`

Note that TypeScript analysis is handled by the `javascript` language identifier. Do not use `typescript` as a separate entry.

### Error 5: Dependabot PRs are never created

**Symptom:** Dependabot is configured and running (you can see status under Insights), but no pull requests are created even though you know you have vulnerable dependencies.

**Cause:** Several possibilities:
- The `open-pull-requests-limit` is set to 0
- The vulnerable dependency is listed in the `ignore` list
- The vulnerable version range does not have a known fix in the Advisory Database
- Dependabot has already created PRs for other dependencies and reached the limit

**Fix:** Check the Dependabot logs under **Insights** → **Dependency graph** → **Dependabot** → **Last checked** time. Click the time to see the log output. Look for entries about why specific dependencies were skipped. Common reasons:
- "Dependency already up to date" — the current version is patched
- "Skipped due to ignore rule" — check your `ignore` configuration
- "No new versions available" — no fix exists yet
