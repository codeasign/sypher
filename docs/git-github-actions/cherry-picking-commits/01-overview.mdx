A hotfix was committed on the `staging` branch, but `main` needs only that one fix — not every other change on `staging`. The fix is a single commit buried in a branch with 15 unrelated changes. Cherry-pick lets you grab that one commit and apply it to any branch, ignoring everything else on the source branch.

## The Concept

Cherry-picking takes the diff from a specific commit and applies it as a new commit on the current branch, creating a copy of the changes with a new hash.

Think of copying a single paragraph from one document into another. You do not copy the entire source document — just the paragraph you need, pasted at the right spot in your document. The pasted paragraph looks the same but has a new position, new context, and (in Git's case) a new identity hash.

## Command / Syntax Reference

| Command | What it does | Example |
|---|---|---|
| `git cherry-pick <commit>` | Apply the changes from `<commit>` as a new commit on the current branch. | `git cherry-pick a1b2c3d` |
| `git cherry-pick <commit1> <commit2>` | Cherry-pick multiple commits in order. | `git cherry-pick a1b2c3d e5f6a7b` |
| `git cherry-pick <start>..<end>` | Cherry-pick all commits between `<start>` (exclusive) and `<end>` (inclusive). | `git cherry-pick main..feature` |
| `git cherry-pick <start>^..<end>` | Cherry-pick including `<start>` through `<end>`. | `git cherry-pick main^..feature` |
| `git cherry-pick -n <commit>` | Apply the changes but do not create a commit (--no-commit). Staged for you to commit later. | `git cherry-pick -n a1b2c3d` |
| `git cherry-pick -x <commit>` | Append the original commit's hash to the new commit message. | `git cherry-pick -x a1b2c3d` |
| `git cherry-pick --continue` | Continue a cherry-pick after resolving a conflict. | `git cherry-pick --continue` |
| `git cherry-pick --abort` | Abort the cherry-pick and return to the pre-pick state. | `git cherry-pick --abort` |
| `git cherry-pick --skip` | Skip the current commit during a cherry-pick sequence. | `git cherry-pick --skip` |

**Windows note:** All cherry-pick commands work identically on Windows. The `-x` flag is especially useful for tracking origins in audit logs — it appends `(cherry picked from commit <hash>)` to the message.

## How It Actually Works — Step by Step

### Step 1: Identify the commit to cherry-pick

```
  ((C0)) ──▶ ((C1)) ──▶ ((C2)) ──▶ ((C3))  ← release
                                           │
  ((D0)) ──▶ ((D1)) ──▶ ((FIX)) ──▶ ((D3)) ← staging
```

You are on `release` (HEAD → release → C3). The fix commit `FIX` is on `staging`. It is a single commit with hash `a1b2c3d` and message "Fix login timeout".

You run `git cherry-pick a1b2c3d`.

### Step 2: Git computes the diff

Git runs `git diff a1b2c3d^..a1b2c3d` internally. This produces a patch: the exact set of line additions, deletions, and modifications that `FIX` introduced. The parent (`a1b2c3d^` = `D1`) is the starting point for the diff.

### Step 3: Git applies the patch to HEAD

Git applies the computed patch to the current HEAD (`C3`) using Git's merge machinery. If the patch applies cleanly, Git stages the changes.

```
  ((C0)) ──▶ ... ──▶ ((C3)) ──▶ [[ staged: FIX's diff ]]  ← release
                                                            ← HEAD (detached during cherry-pick)
```

### Step 4: Git creates a new commit

Git creates a new commit with:
- **Tree:** the staged snapshot (C3's files + FIX's diff)
- **Parent:** C3 (the current HEAD, not FIX's original parent D1)
- **Author:** the original commit's author (preserved from FIX)
- **Committer:** you (the person running cherry-pick)
- **Message:** the original commit's message, with `(cherry picked from commit a1b2c3d)` if `-x` was used
- **Date:** the current time (not the original commit's date)

```
  ((C0)) ──▶ ... ──▶ ((C3)) ──▶ ((FIX'))  ← release
                                           ← HEAD
  FIX' has hash f8e7d6c — completely different from a1b2c3d.
```

### Step 5: Handle conflicts (if any)

If the patch cannot apply cleanly because `C3` and `D1` have diverged, Git pauses with conflict markers:

```
error: could not apply a1b2c3d... Fix login timeout
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git cherry-pick --continue".
```

Resolve the conflict like any merge: edit the file, `git add`, `git cherry-pick --continue`.

```
  ((C0)) ──▶ ... ──▶ ((C3)) ──▶ ((FIX' resolved))  ← release
                                                     ← HEAD
```

## When to Use This / When Not To

**Use cherry-pick when:**
- A bugfix was applied to one branch and needs to be ported to another (e.g., from `staging` to `release`)
- You need a specific feature commit from a branch whose other changes are not ready
- You want to backport a critical fix to a long-term support branch without merging the entire feature
- You accidentally committed something to the wrong branch and want to copy it before resetting

**Do not use cherry-pick when:**
- You want all commits from a branch — use `git merge` or `git rebase` instead
- The cherry-picked commit depends on earlier commits in its source branch that are not being picked — the result may compile but fail at runtime
- You need to track that the same change exists on multiple branches — use `git merge` for shared history, or always use `-x` to annotate cherry-picks
- The commit is a merge commit — cherry-picking a merge commit applies only the diff, not the branching structure

**Key concern:** Cherry-picked commits are copies. If the same fix is cherry-picked to three branches, each has a different hash. A developer looking at one branch cannot tell by the hash alone that the fix exists on the other branches. The `-x` flag mitigates this by recording the original hash in the message.
