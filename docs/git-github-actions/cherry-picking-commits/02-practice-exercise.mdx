You will create two branches, make a specific fix on one branch, cherry-pick it to another, handle a conflict during cherry-pick, and cherry-pick a range of commits. Every command works identically on Windows, macOS, and Linux.

## Setup

```bash
mkdir -p ~/cherry-pick-lab
cd ~/cherry-pick-lab
git init
git config user.name "Your Name"
git config user.email "your.email@example.com"
```

Create the base structure:

```bash
cat > calculator.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b
EOF
git add calculator.py
git commit -m "Initial calculator"
```

## The Walkthrough

### Step 1: Create divergent branches

```bash
git switch -c development
echo "def multiply(a, b):" >> calculator.py
echo "    return a * b" >> calculator.py
git add calculator.py
git commit -m "Add multiply function"
echo "def divide(a, b):" >> calculator.py
echo "    return a / b" >> calculator.py
git add calculator.py
git commit -m "Add divide function"
```

Now switch to `main` and commit a different change:

```bash
git switch main
echo "'''Calculator module'''" > calculator.py
echo "def add(a, b):" >> calculator.py
echo "    return a + b" >> calculator.py
git add calculator.py
git commit -m "Add module docstring"
```

Branch state:

```
  ((Initial)) ──▶ ((Docstring))  ← main
       │                         ← HEAD
       └──▶ ((Multiply)) ──▶ ((Divide))  ← development
```

**Checkpoint:** `git log --oneline --graph --all` shows the fork.

### Step 2: Cherry-pick a single commit

Pick just the "Add divide function" commit onto `main`:

```bash
git log development --oneline
```

Note the hash of "Add divide function" (second commit on development). Then:

```bash
git cherry-pick <hash-of-divide-commit>
```

Expected output:

```
[main abc1234] Add divide function
 Date: ...
 1 file changed, 2 insertions(+)
```

Verify:

```bash
git log --oneline -3
```

Expected output:

```
abc1234 (HEAD -> main) Add divide function
bcd2345 Add module docstring
cde3456 Initial calculator
```

The "Add divide function" changes are now on `main`. The "Add multiply function" change is NOT on `main` — it stayed on `development`.

```
  ((Initial)) ──▶ ((Docstring)) ──▶ ((Divide cherry-picked))  ← main
       │                                                     ← HEAD
       └──▶ ((Multiply)) ──▶ ((Divide original))  ← development
```

**Under the hood:** The cherry-picked commit has a different hash from the original. Its parent is the docstring commit, not the multiply commit.

### Step 3: Cherry-pick with the `-x` flag

Cherry-pick another commit with tracking information:

```bash
git cherry-pick -x <hash-of-multiply-commit>
```

Expected output:

```
[main def5678] Add multiply function
 Date: ...
 1 file changed, 2 insertions(+)
```

View the commit message to see the tracking note:

```bash
git log -1 --format="%B"
```

Expected output:

```
Add multiply function

(cherry picked from commit <original-hash>)
```

The `-x` flag appends the original commit hash. This is invaluable for traceability in production branches — you can always find the source commit if you need more context.

### Step 4: Cherry-pick with conflict

Create a conflict scenario deliberately:

```bash
git switch development
echo "def add(a, b):" > calculator.py
echo "    return a + b + 1  # intentional bug" >> calculator.py
echo "" >> calculator.py
echo "def subtract(a, b):" >> calculator.py
echo "    return a - b" >> calculator.py
git add calculator.py
git commit -m "Add off-by-one to add function"
```

Now cherry-pick this commit onto `main`:

```bash
git switch main
git cherry-pick <hash-of-bug-commit>
```

Expected output:

```
Auto-merging calculator.py
CONFLICT (content): Merge conflict in calculator.py
error: could not apply <hash>... Add off-by-one to add function
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git cherry-pick --continue".
```

A conflict occurs because both `main` and `development` modified `calculator.py` in incompatible ways.

### Step 5: Resolve the cherry-pick conflict

View the conflict:

```bash
cat calculator.py
```

Resolve by keeping the `main` version of the add function (no bug) while allowing other changes:

```bash
cat > calculator.py << 'EOF'
def add(a, b):
    return a + b


def subtract(a, b):
    return a - b


def multiply(a, b):
    return a * b


def divide(a, b):
    return a / b
EOF
git add calculator.py
git cherry-pick --continue
```

Git opens your editor with the commit message. Save and close.

Expected output:

```
[main f8e7d6c] Add off-by-one to add function
 1 file changed, 2 insertions(+), 2 deletions(-)
```

The commit went through with the resolved content. **Note:** The actual diff still says "off-by-one" in the message, but the resolved code is correct. In practice, you would reword the message during `--continue` to reflect what actually changed.

### Step 6: Cherry-pick a range of commits

Create a scenario with multiple commits to cherry-pick at once:

```bash
git switch -c feature-range
echo "def square(n):" >> calculator.py
echo "    return n * n" >> calculator.py
git add calculator.py
git commit -m "Add square function"
echo "def cube(n):" >> calculator.py
echo "    return n ** 3" >> calculator.py
git add calculator.py
git commit -m "Add cube function"
echo "def sqrt(n):" >> calculator.py
echo "    return n ** 0.5" >> calculator.py
git add calculator.py
git commit -m "Add sqrt function"
```

Now cherry-pick all three commits onto `main` in one command:

```bash
git switch main
git cherry-pick <hash-of-square>..<hash-of-sqrt>
```

Or use the `main..feature` syntax:

```bash
git cherry-pick main..feature-range
```

Expected output — three separate cherry-picks execute in sequence:

```
[main abc1111] Add square function
[main abc2222] Add cube function
[main abc3333] Add sqrt function
```

Verify:

```bash
git log --oneline -5
```

You should see the three new cherry-picked commits on top of `main`.

### Step 7: No-commit cherry-pick

Cherry-pick changes without creating a commit (useful when you want to combine multiple cherry-picks into one commit):

```bash
git switch -c feature-nc
echo "def factorial(n):" >> calculator.py
echo "    if n <= 1: return 1" >> calculator.py
echo "    return n * factorial(n-1)" >> calculator.py
git add calculator.py
git commit -m "Add factorial"
echo "def gcd(a, b):" >> calculator.py
echo "    while b:" >> calculator.py
echo "        a, b = b, a % b" >> calculator.py
echo "    return a" >> calculator.py
git add calculator.py
git commit -m "Add gcd"
```

Pick both changes as a single commit on `main`:

```bash
git switch main
git cherry-pick -n <hash-of-factorial>
git cherry-pick -n <hash-of-gcd>
git status
```

The `-n` flag leaves both sets of changes staged but not committed. You can now commit them together:

```bash
git commit -m "Add factorial and GCD functions"
```

Your `main` branch has one commit containing both changes instead of two separate commits.

## Common Errors at This Stage

### Error 1: `fatal: bad revision`

You provided a commit reference that Git cannot find. This usually means the commit hash is wrong, or you typed it from memory incorrectly.

**Fix:** Copy the full hash from `git log`. Use tab completion if available. You only need the first 7-8 characters as long as they are unique.

### Error 2: `error: could not apply <hash>... <message>`

The cherry-pick patch could not apply cleanly to the current HEAD because the files diverged too much from the original context.

**Fix:** Resolve the conflict markers manually, `git add`, and `git cherry-pick --continue`. If the commit is no longer needed, `git cherry-pick --abort` to undo.

### Error 3: Cherry-pick succeeds but the code does not compile

The cherry-picked commit depends on code introduced in an earlier commit on its source branch that was not cherry-picked. Git applied the diff correctly, but the surrounding context is missing a function, variable, or import that the cherry-picked code assumes exists.

**Fix:** Cherry-pick the dependency commits first, then the target commit. To find dependencies, examine the commit's diff with `git show <hash>` and identify any references to code that does not exist on your branch.

### Error 4: `git cherry-pick main..feature` picks too many or too few commits

The `..` range operator is exclusive of the starting point. `main..feature` picks everything reachable from `feature` that is not reachable from `main` — which is exactly what you want for feature branches, but may include merge commits or intermediate commits you did not intend.

**Fix:** Use explicit hashes for precise control: `git cherry-pick <hash1> <hash2> <hash3>`. For ranges, verify the range with `git log --oneline <start>..<end>` before cherry-picking.

## What You Just Learned

- `git cherry-pick <hash>` applies a specific commit's diff as a new commit on the current branch
- The new commit has a different hash, parent, and date from the original
- `git cherry-pick -x <hash>` appends the original hash to the commit message for traceability
- `git cherry-pick -n <hash>` applies the diff without committing (useful for combining picks)
- Conflicts during cherry-pick are resolved with `git add` + `git cherry-pick --continue`
- Cherry-picking a range: `git cherry-pick <start>..<end>` or `git cherry-pick main..feature`
- Cherry-picks are copies, not moves — the original commit remains on its source branch
