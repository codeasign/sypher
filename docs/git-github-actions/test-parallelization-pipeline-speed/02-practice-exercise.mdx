## Setup

You need a Python project with pytest and a GitHub Actions workflow to parallelize. This walkthrough uses a generated test suite that produces a realistic number of tests so you can see the speedup.

Create a new project directory:

```bash
mkdir parallel-demo
cd parallel-demo
git init
python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate
pip install pytest pytest-xdist pytest-split
```

Generate a test suite with fifty test files. Each file has a few tests with varying durations to simulate a real project.

```bash
mkdir tests
```

```bash
for i in $(seq 1 50); do
  sleep_time=$(echo "scale=2; $RANDOM / 5000" | bc)
  cat > "tests/test_${i}.py" << PYEOF
import time
import pytest

def test_${i}_a():
    time.sleep(${sleep_time})
    assert True

def test_${i}_b():
    time.sleep(${sleep_time})
    assert True
PYEOF
done
```

On Windows (PowerShell):

```powershell
1..50 | ForEach-Object {
  $sleep = [math]::Round((Get-Random -Minimum 1 -Maximum 30) / 10, 2)
  @"
import time
import pytest

def test_${_}_a():
    time.sleep($sleep)
    assert True

def test_${_}_b():
    time.sleep($sleep)
    assert True
"@ | Out-File -FilePath "tests\test_${_}.py" -Encoding utf8
}
```

Create a `.gitignore`:

```bash
echo "venv/" > .gitignore
echo "__pycache__/" >> .gitignore
echo ".pytest_cache/" >> .gitignore
```

Commit and push to a GitHub repository:

```bash
git add -A
git commit -m "Add parallel demo test suite"
git branch -M main
git remote add origin https://github.com/your-org/parallel-demo.git
git push -u origin main
```

## The Walkthrough

### Step 1: Run the suite sequentially and measure the baseline

Create a workflow that runs all tests on a single runner without any parallelization. This gives you the baseline time.

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest
      - run: pytest tests/ --verbose --durations=10
```

Push and run the workflow. Record the total time. It should be between thirty seconds and two minutes depending on the random sleep values.

Expected output (approximate):

```
collected 100 items
tests/test_01.py ..                                                [  2%]
tests/test_02.py ..                                                [  4%]
...
tests/test_50.py ..                                                [100%]

===================== 100 passed in 85.3s =====================
```

### Step 2: Add parallel workers on a single runner

Install `pytest-xdist` and update the workflow to use `-n auto` for parallel execution within a single runner.

```yaml
# Update the test step in .github/workflows/ci.yml
      - run: pip install pytest pytest-xdist
      - run: pytest tests/ -n auto --verbose --durations=10
```

The `-n auto` flag tells pytest to spawn one worker per CPU core. GitHub-hosted `ubuntu-latest` runners have two cores, so you get two workers.

Push and compare the time. You should see a significant speedup.

### Step 3: Split the suite across multiple runners with a matrix

Now distribute the test load across multiple GitHub Actions runners. Use a matrix strategy to create four shards. Each shard runs on its own runner.

```yaml
name: CI Pipeline

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        shard: [1, 2, 3, 4]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest
      - run: pytest tests/ --verbose
```

This creates four parallel jobs, each running the full test suite. That is not what you want. You need to split the tests so each shard runs a different subset.

### Step 4: Split tests by file with `pytest-split`

Install `pytest-split` and use the `--splits` and `--group` flags to distribute tests across shards.

```bash
pip install pytest-split
```

```yaml
name: CI Pipeline

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        shard: [1, 2, 3, 4]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest pytest-xdist pytest-split
      - run: pytest tests/ --splits 4 --group ${{ matrix.shard }} --verbose
```

The `--splits 4` flag tells pytest to split the test suite into four groups. The `--group ${{ matrix.shard }}` flag tells each job which group to run. The `--store` flag stores timing data from each run so future splits are more balanced.

Push and observe the pipeline. You should see four parallel jobs, each running approximately one quarter of the tests.

### Step 5: Add fail-fast and max-parallel controls

Add `fail-fast` and `max-parallel` to control how aggressively the matrix runs. This prevents all runners from being consumed by a single PR and limits the blast radius of a failing shard.

```yaml
    strategy:
      fail-fast: false
      max-parallel: 4
      matrix:
        shard: [1, 2, 3, 4]
```

`fail-fast: false` ensures that if shard 1 fails, shards 2 through 4 still complete and report their results. `max-parallel: 4` limits the number of concurrent runners to four, which is already the maximum for this matrix.

### Step 6: Add concurrency to cancel redundant runs

Add a `concurrency` block to cancel in-progress runs when a new commit is pushed to the same branch.

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

The full workflow:

```yaml
name: CI Pipeline

on:
  pull_request:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      max-parallel: 4
      matrix:
        shard: [1, 2, 3, 4]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest pytest-xdist pytest-split
      - run: pytest tests/ --splits 4 --group ${{ matrix.shard }} --verbose
```

### Step 7: Add timing-based balancing

After several runs, `pytest-split` stores timing data in a `.test_durations` file. Commit this file so all runs use the stored timing data for balanced splits.

```bash
git add .test_durations
git commit -m "Add test timing data for balanced sharding"
git push
```

Update the workflow to use the stored timing data:

```yaml
      - run: pytest tests/ --splits 4 --group ${{ matrix.shard }} --store --verbose
```

The `--store` flag saves timing data from each run. Future runs automatically use the stored data to create balanced shards.

## Common Errors at This Stage

### Error 1: `pytest-xdist` workers hang or crash

```
INTERNALERROR> AssertionError: worker crashed while running test_23.py
```

**Fix:** This usually happens when tests share state (a database, a file, a global variable). Add `--dist worksteal` to xdist to use a work-stealing scheduler that handles uneven test durations better:

```bash
pytest tests/ -n auto --dist worksteal --verbose
```

If tests still crash, mark the offending test as serial with the `@pytest.mark.serial` decorator or run it in a separate job.

### Error 2: Shards are wildly unbalanced

```
Shard 1: 45 tests, 2 seconds
Shard 2: 5 tests, 40 seconds
```

**Fix:** The default split is by test file name, not by duration. Install `pytest-split` and use `--splits` and `--group` instead of manual file splitting. For the first run, you can also shuffle the test files manually:

```bash
python -c "
import os, random
files = [f for f in os.listdir('tests') if f.startswith('test_')]
random.shuffle(files)
for i, f in enumerate(files):
    print(f'Shard {i % 4 + 1}: {f}')
"
```

### Error 3: The matrix creates too many runners at once

```
Error: The workflow will run 16 parallel jobs, which exceeds the limit of 10.
```

**Fix:** Use `max-parallel` to cap the number of concurrent runners:

```yaml
    strategy:
      max-parallel: 4
      matrix:
        shard: [1, 2, 3, 4, 5, 6, 7, 8]
```

This limits the pipeline to four concurrent runners even though the matrix has eight shards.

### Error 4: `pytest-split` cannot find the timing file

```
pytest-split: No timing data found. Running tests without splitting.
```

**Fix:** Run the test suite once with `--store` to generate the `.test_durations` file, then commit it to the repository:

```bash
pytest tests/ --splits 4 --group 1 --store
git add .test_durations
git commit -m "Add test timing data"
```

On subsequent runs, the timing file is used automatically.

### Error 5: The workflow runs all tests on every shard instead of splitting

```
Runner 1: 100 passed
Runner 2: 100 passed
Runner 3: 100 passed
Runner 4: 100 passed
```

**Fix:** You forgot to add the `--splits` and `--group` flags. Without these flags, every runner runs the full test suite. The matrix creates four parallel jobs, but they all do the same work. Update the run command:

```bash
pytest tests/ --splits 4 --group ${{ matrix.shard }}
```

## What You Just Learned

- You measured your baseline test suite time with a sequential run and recorded the total wall clock time.
- You used `pytest-xdist` with `-n auto` to parallelize across CPU cores on a single runner, gaining a two-times speedup.
- You used a GitHub Actions matrix strategy with `pytest-split` to distribute tests across four parallel runners, gaining a nearly four-times speedup.
- You added `fail-fast: false` and `max-parallel` to control the blast radius and cost of parallel execution.
- You added `concurrency` with `cancel-in-progress: true` to stop redundant runs when you push multiple commits.
- You stored and committed timing data with `--store` so that future runs use balanced shards based on actual test durations instead of file name order.
