## Tier 1 -- Beginner

### Exercise 1: Calculate the speedup from parallelization

Your test suite has 200 tests. Each test takes an average of 1.2 seconds to run. You have a single runner with 4 CPU cores.

**Task:** Calculate the expected wall time running sequentially versus running with `pytest -n auto` on a 4-core runner. Assume perfect parallelization with no overhead.

<details>
<summary>Hint</summary>

Sequential time is the number of tests multiplied by the average test time. Parallel time assumes the work is split evenly across the available cores. With 4 cores and 200 tests, each core gets 50 tests.
</details>

<details>
<summary>Solution</summary>

Sequential time: 200 tests times 1.2 seconds = 240 seconds = 4 minutes.

Parallel time: 200 tests divided by 4 cores = 50 tests per core. 50 tests times 1.2 seconds = 60 seconds = 1 minute.

Expected speedup: 4x.

In practice, the speedup is usually 2.5x to 3.5x due to overhead from process spawning, I/O contention, and unequal test durations. The theoretical maximum 4x is rarely achieved, but 3x is realistic.
</details>

### Exercise 2: Identify the bottleneck shard

You have four shards running in parallel. Their durations are: shard 1 = 8 minutes, shard 2 = 14 minutes, shard 3 = 7 minutes, shard 4 = 9 minutes.

**Task:** What is the wall time for the full pipeline? Which shard is the bottleneck? By how much would the pipeline improve if you optimized the bottleneck shard to run in 9 minutes?

<details>
<summary>Hint</summary>

Wall time is determined by the slowest shard, not the average. The bottleneck is the shard that takes the longest. If you improve it, the new wall time is whichever shard is now the slowest.
</details>

<details>
<summary>Solution</summary>

Wall time: 14 minutes (the duration of the slowest shard, which is shard 2).

If shard 2 improves to 9 minutes, the new wall time is 9 minutes (determined by shard 4 at 9 minutes, which was previously the second-slowest).

Improvement: 14 minutes to 9 minutes = 5 minutes saved = 36 percent reduction.

The key insight: optimizing the bottleneck only helps until the next-slowest shard becomes the new bottleneck. This is why timing-balanced sharding is important -- the gap between the fastest and slowest shard should be as small as possible.
</details>

### Exercise 3: Add a matrix strategy to an existing workflow

You have the following workflow that runs tests on a single Python version:

```yaml
name: CI
on: [pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest
      - run: pytest tests/
```

**Task:** Modify this workflow to test on Python 3.10, 3.11, and 3.12 in parallel, with a maximum of 2 concurrent runners. Do not run tests for Python 3.10 if 3.11 or 3.12 fails.

<details>
<summary>Hint</summary>

Use `strategy.matrix` with a `python-version` key. Set `fail-fast` to `true` so that a failure on one version stops the others. Use `max-parallel` to limit concurrency.
</details>

<details>
<summary>Solution</summary>

```yaml
name: CI
on: [pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: true
      max-parallel: 2
      matrix:
        python-version: ["3.10", "3.11", "3.12"]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install pytest
      - run: pytest tests/
```

This creates three parallel jobs, one for each Python version. The `max-parallel: 2` setting ensures only two runners run at the same time. The `fail-fast: true` setting cancels all remaining jobs if any Python version fails.
</details>

### Exercise 4: Read concurrency behavior from a pipeline log

You see the following log output from a GitHub Actions run:

```
Run 1 (commit abc123) started at 10:00:00
  - lint job: started
  - test job: started
Run 2 (commit def456) triggered at 10:02:00
  - Run 1: Cancelling...
  - lint job: Cancelled
  - test job: Cancelled
Run 2 (commit def456) started at 10:02:05
  - lint job: started
  - test job: started
```

**Task:** What is the concurrency group configuration that produces this behavior? How would you change it so that runs on `main` are never cancelled?

<details>
<summary>Hint</summary>

The current configuration cancels any in-progress run when a new commit is pushed to the same branch. The `${{ github.ref }}` variable includes the branch name. For `main`, you want to preserve previous behavior and not cancel runs.
</details>

<details>
<summary>Solution</summary>

The current configuration is:

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

To prevent cancellation on `main`, modify the `cancel-in-progress` value to be conditional:

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: ${{ github.ref_name != 'main' }}
```

This setting cancels in-progress runs on feature branches but allows all runs on `main` to complete normally. For `main`, `cancel-in-progress` evaluates to `false`, so new pushes wait for the previous run to finish.
</details>

## Tier 2 -- Intermediate

### Exercise 1: Design a sharding strategy for a large Python monorepo

You have a monorepo with the following structure:

```
services/
  auth/    (200 tests, avg 0.5s each)
  api/     (800 tests, avg 1.2s each)
  worker/  (100 tests, avg 0.3s each)
libs/
  shared/  (50 tests, avg 0.1s each)
```

Your goal is to achieve a total wall time of under 5 minutes for the entire test suite. You have GitHub Actions runners with 2 CPU cores each.

**Task:** Determine how many shards you need and how to allocate tests to each shard. Show the expected wall time with your allocation.

<details>
<summary>Hint</summary>

Calculate the total test time for each service and for the whole suite. Then determine how many shards are needed to bring the largest service (api) under 5 minutes. Consider whether to shard by service or to split tests across all services for better balance.
</details>

<details>
<summary>Solution</summary>

Total test times by service:
- auth: 200 tests times 0.5s = 100 seconds = 1.67 minutes
- api: 800 tests times 1.2s = 960 seconds = 16 minutes
- worker: 100 tests times 0.3s = 30 seconds = 0.5 minutes
- shared: 50 tests times 0.1s = 5 seconds
- Total: 1095 seconds = 18.25 minutes

The api service alone takes 16 minutes sequentially. To get under 5 minutes, shard the api tests first.

With xdist on a single runner (2 cores):
- api with `-n auto`: 16 minutes divided by 2 = 8 minutes (still over 5)

With 4 shards across 4 runners:
- api tests split across 4 shards: 16 minutes divided by 4 = 4 minutes (under 5!)

Recommended configuration:
- 4 shards total
- Shard 1: 200 api tests (approx 4 minutes)
- Shard 2: 200 api tests (approx 4 minutes)
- Shard 3: 200 api tests + 50 shared tests (approx 4.2 minutes)
- Shard 4: 200 api tests + 200 auth tests + 100 worker tests (approx 4.5 minutes)

Alternatively, use `pytest-split` with `--splits 4` and let the tool balance by timing data automatically. The tool distributes tests so each shard has approximately 273 seconds (1095 divided by 4 = 274 seconds) of test time.

Expected wall time with `pytest-split --splits 4`: approximately 5 minutes per shard.

To guarantee under 5 minutes, use 6 shards:
- Each shard: approximately 1095 divided by 6 = 183 seconds = 3 minutes.
- This is safer and leaves room for test suite growth.
</details>

### Exercise 2: Combine matrix testing with sharding

You need to test a Python package across three Python versions (3.10, 3.11, 3.12) and you want to shard the test suite into 4 groups for each version. The total test suite has 600 tests.

**Task:** Write the YAML for the `strategy` section that creates 12 total jobs (3 versions times 4 shards). Use `fail-fast: false` and limit to 6 concurrent runners.

<details>
<summary>Hint</summary>

Use a matrix with two dimensions: `python-version` and `shard`. The total number of jobs is the product of the matrix dimensions. Control concurrency with `max-parallel`.
</details>

<details>
<summary>Solution</summary>

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      max-parallel: 6
      matrix:
        python-version: ["3.10", "3.11", "3.12"]
        shard: [1, 2, 3, 4]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pip install pytest pytest-xdist
      - run: pytest tests/ -n auto --splits 4 --group ${{ matrix.shard }}
```

This creates 12 jobs: each of the 3 Python versions gets 4 shards. The `max-parallel: 6` setting limits concurrent runners to 6, so the jobs run in two waves of 6. The `fail-fast: false` setting ensures that a failure in one Python version does not cancel the other versions.

Total expected wall time: the slowest shard across all Python versions, which should be under 5 minutes with balanced sharding.
</details>

### Exercise 3: Optimize a Jest test suite with sharding

Your team uses Jest for a React project. The test suite has 500 test files that take 25 minutes to run sequentially. Each test file has different durations.

**Task:** Write the CI workflow that splits the Jest test suite across 5 runners using `--shard`. Each shard should run 100 test files. Add concurrency to cancel redundant runs.

<details>
<summary>Hint</summary>

Jest has built-in sharding with `--shard=N/M` where N is the shard number and M is the total number of shards. Jest shards by file name by default, not by timing. Use `matrix.shard` to generate the N values.
</details>

<details>
<summary>Solution</summary>

```yaml
name: CI

on:
  pull_request:
    branches: [main]

concurrency:
  group: ci-jest-${{ github.ref }}
  cancel-in-progress: true

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      max-parallel: 5
      matrix:
        shard: [1, 2, 3, 4, 5]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx jest --shard=${{ matrix.shard }}/5 --maxWorkers=2
```

Jest's `--shard=1/5` tells Jest to run the first fifth of test files (sorted alphabetically). Each shard runs approximately 100 files. The `--maxWorkers=2` flag limits parallel execution within each shard to 2 workers to avoid memory exhaustion.

Expected wall time: approximately 5 to 8 minutes (25 sequential minutes divided by 5 shards, with some overhead for the slowest shard).

For better balance, consider using Jest's `--shard` with a custom test sequencer that reads timing data, or split tests by directory rather than relying on alphabetical file ordering.
</details>

## Tier 3 -- Advanced

### Exercise 1: Build a custom test splitting script for non-standard test runners

Your team uses a custom test runner that does not support sharding or parallel execution. The runner takes a list of test files as arguments and runs them sequentially.

```bash
custom-runner test_file1.py test_file2.py test_file3.py
```

You have 200 test files. The runtimes vary from 0.5 seconds to 30 seconds per file, with a total suite runtime of 45 minutes.

**Task:** Write a Python script that reads a timing file (JSON format with `{filename: seconds}`), splits the files into N balanced groups using a greedy algorithm, and outputs the group for a given shard index.

<details>
<summary>Hint</summary>

A greedy algorithm sorts files by duration descending, then assigns each file to the shard that currently has the smallest total duration. This is the classic multiprocessor scheduling approximation. It produces near-optimal balance without the complexity of full optimization.
</details>

<details>
<summary>Solution</summary>

```python
import json
import sys

def load_timings(timing_file: str) -> dict:
    """Load timing data from a JSON file."""
    with open(timing_file) as f:
        return json.load(f)

def split_into_groups(timings: dict, num_groups: int) -> list:
    """Split test files into balanced groups using greedy algorithm."""
    # Sort by duration descending (longest tests first)
    sorted_files = sorted(timings.items(), key=lambda x: -x[1])

    # Initialize groups: each is a list of (filename, duration)
    groups = [[] for _ in range(num_groups)]
    group_durations = [0.0] * num_groups

    for filename, duration in sorted_files:
        # Find the group with the smallest current total duration
        min_group = min(range(num_groups), key=lambda i: group_durations[i])
        groups[min_group].append((filename, duration))
        group_durations[min_group] += duration

    return groups

def main():
    timing_file = sys.argv[1]
    num_shards = int(sys.argv[2])
    shard_index = int(sys.argv[3])  # 0-based

    timings = load_timings(timing_file)
    groups = split_into_groups(timings, num_shards)

    group = groups[shard_index]
    total_duration = sum(d for _, d in group)
    filenames = [f for f, _ in group]

    print(f"Shard {shard_index + 1}/{num_shards}: "
          f"{len(filenames)} files, "
          f"{total_duration:.1f} seconds")
    print(" ".join(filenames))

if __name__ == "__main__":
    main()
```

Usage in a GitHub Actions workflow:

```yaml
    - name: Split tests
      run: |
        python split_tests.py test_timings.json ${{ strategy.job-total }} ${{ strategy.job-index }}
      id: split
    - name: Run tests
      run: custom-runner ${{ steps.split.outputs.files }}
```

The greedy algorithm ensures that the longest tests get placed first, preventing a single long test from becoming the bottleneck at the end of the balancing process. The result is groups that differ by no more than the duration of the longest test file.
</details>

### Exercise 2: Design a cost-optimized parallelization strategy for a tight budget

Your team has a monthly CI budget of $200. Each GitHub Actions runner costs $0.008 per minute. Your test suite takes 30 minutes to run sequentially. You run CI on average 20 times per day, 22 days per month.

**Task:** Calculate the monthly cost for sequential runs. Then design a parallelization strategy that brings the wall time under 10 minutes while staying under the $200 budget. Include the number of shards, the new wall time, and the new monthly cost.

<details>
<summary>Hint</summary>

Calculate the current monthly cost first. Then calculate the cost per parallel run. Remember that more shards means more total compute minutes (cost) but lower wall time. You need to find the sweet spot where wall time is under 10 minutes and cost is under $200.
</details>

<details>
<summary>Solution</summary>

Current sequential cost:
- 30 minutes per run times $0.008 per minute = $0.24 per run
- 20 runs per day times 22 days = 440 runs per month
- 440 runs times $0.24 = $105.60 per month

Budget remaining for parallelization: $200 minus $105.60 = $94.40

Now consider parallel strategies:

**Option A: 2 shards**
- Each run: 2 runners for 15 minutes = 30 compute minutes
- Cost per run: 30 minutes times $0.008 = $0.24
- Monthly cost: 440 runs times $0.24 = $105.60 (same as sequential!)
- Wall time: 15 minutes (still over 10-minute target)

**Option B: 3 shards**
- Each run: 3 runners for 10 minutes = 30 compute minutes
- Cost per run: 30 minutes times $0.008 = $0.24
- Monthly cost: 440 runs times $0.24 = $105.60
- Wall time: 10 minutes (at the target)

**Option C: 4 shards**
- Each run: 4 runners for 8 minutes = 32 compute minutes
- Cost per run: 32 minutes times $0.008 = $0.256
- Monthly cost: 440 runs times $0.256 = $112.64
- Wall time: 8 minutes (under target)

**Option D: 6 shards**
- Each run: 6 runners for 5 minutes = 30 compute minutes
- Cost per run: 30 minutes times $0.008 = $0.24
- Monthly cost: 440 runs times $0.24 = $105.60
- Wall time: 5 minutes (well under target)

If you add xdist within each shard (2 workers per shard), you can further reduce wall time without increasing cost:
- 4 shards with 2 workers each: 8 minutes divided by 2 = 4 minutes
- Same cost because runners are the same (4 runners, not 4x2)

**Recommendation: 4 shards with xdist -n auto = 4 minutes wall time, $112.64 monthly cost.** This stays under the $200 budget with $87.36 remaining for future growth.

If the test suite grows by 50 percent in the next quarter, increase to 6 shards, which keeps the cost at $105.60 and the wall time under 5 minutes.
</details>

### Exercise 3: Design a flaky test detection and quarantining system

Your team has 8 shards running in parallel. About 5 percent of tests are flaky -- they fail intermittently due to timing, ordering, or environment issues. Each flaky failure causes the entire shard (and its tests) to re-run on retry, wasting 10 minutes of CI time per flake.

**Task:** Design a workflow that:
1. Detects which tests failed in each shard
2. Re-runs only the failed tests (not the entire shard)
3. If the re-run passes, marks those tests as flaky and does not block the merge
4. If the re-run fails, blocks the merge normally
5. Tracks flaky tests in a GitHub issue or artifact

<details>
<summary>Hint</summary>

Use `pytest` with `--junitxml` to produce structured test results. Use the `--last-failed` flag with `pytest-xdist` to re-run only failed tests. Use a combination of `continue-on-error` on the first run and conditional re-run jobs. Store flaky test names in an artifact for tracking.
</details>

<details>
<summary>Solution</summary>

```yaml
name: CI with Flaky Test Detection

on:
  pull_request:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      max-parallel: 8
      matrix:
        shard: [1, 2, 3, 4, 5, 6, 7, 8]
    continue-on-error: true  # Allow flaky failures without blocking
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install pytest pytest-xdist pytest-split
      - name: Run tests
        id: run_tests
        run: |
          pytest tests/ \
            --splits 8 \
            --group ${{ matrix.shard }} \
            -n auto \
            --junitxml=results-${{ matrix.shard }}.xml \
            --verbose
      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results-${{ matrix.shard }}
          path: results-${{ matrix.shard }}.xml

  flaky-retry:
    needs: [test]
    if: always()
    runs-on: ubuntu-latest
    strategy:
      matrix:
        shard: [1, 2, 3, 4, 5, 6, 7, 8]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - name: Download test results
        uses: actions/download-artifact@v4
        with:
          name: test-results-${{ matrix.shard }}
      - name: Re-run failed tests only
        id: rerun
        run: |
          pip install pytest pytest-xdist pytest-split
          # pytest --last-failed runs only tests that failed in the previous run
          # --cache-clear ensures we start fresh
          if pytest tests/ \
            --splits 8 \
            --group ${{ matrix.shard }} \
            -n auto \
            --last-failed \
            --junitxml=rerun-results-${{ matrix.shard }}.xml \
            --verbose; then
            echo "Flaky tests detected and passing on retry"
            echo "flaky=true" >> $GITHUB_OUTPUT
          else
            echo "Tests still failing after retry"
            exit 1  # This blocks the merge
          fi

  report-flaky:
    needs: [flaky-retry]
    if: always()
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Aggregate flaky test results
        run: |
          # Collect all rerun results and extract flaky test names
          # Post a summary comment on the PR with flaky test details
          echo "## Flaky Test Report" > flaky-report.md
          echo "The following tests failed on first run but passed on retry:" >> flaky-report.md
          # Parse rerun XML files for test names that went from fail to pass
          # (Implementation depends on your test framework and tooling)
      - name: Create GitHub Issue for tracked flaky tests
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            // Check if flaky tests exist and create/update tracking issue
            // This helps the team track flaky test debt over time
```

This design detects flaky tests without blocking the pipeline. The `continue-on-error: true` on the first run allows flaky failures. The `flaky-retry` job re-runs only the failed tests using `--last-failed`. If the re-run passes, the tests are marked as flaky and the pipeline succeeds. If the re-run also fails, the `exit 1` on the retry job blocks the merge. The `report-flaky` job aggregates results and can create GitHub issues for tracking.

The key advantage: your pipeline gates are not weakened by flaky tests, but developers also do not spend 10 minutes re-running entire shards.
</details>
