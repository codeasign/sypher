Complete these exercises on your own to prove you can configure, manage, and debug GitHub Environments and approval gates.

## Tier 1 — Beginner

### Exercise 1: Create Two Environments with Different Protection Levels

**Task:** Create a GitHub Actions workflow with two environment-referencing jobs: `deploy-staging` (no protection rules) and `deploy-production` (requires 1 reviewer). The workflow should deploy to staging first, then production.

**Hint:** Use `environment: staging` for the first job and `environment: production` for the second. Add `needs: deploy-staging` to the production job.

**Solution:**

```yaml
name: Deploy with Environments

on:
  push:
    branches: [main]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.example.com
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://example.com
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

In the GitHub UI, configure the `production` environment with 1 required reviewer. The `deploy-production` job will pause and wait for approval. The `deploy-staging` job runs automatically.

### Exercise 2: Environment with a Wait Timer

**Task:** Configure the `production` environment with a 2-minute wait timer. Create a deployment workflow that triggers on push to `main` and deploys to production. Verify the wait timer pauses the deployment for 2 minutes before it becomes available for approval.

**Hint:** The wait timer is configured in Settings > Environments > production > Protection rules > Wait timer. Set it to 2 minutes.

**Solution:**

Settings configuration:
1. Go to Settings > Environments > production
2. Under Protection rules, check "Wait timer"
3. Enter `2` minutes
4. Click Save

The workflow itself does not change — the wait timer is a server-side setting. When the workflow runs, the `deploy-production` job shows "Waiting" with a countdown timer. After 2 minutes, the Approve/Reject buttons become available. The wait timer is useful for:
- Giving team members a chance to abort a deployment they notice is wrong
- Allowing monitoring systems to detect anomalies before the deploy happens
- Creating a mandatory cooldown period for irreversible deployments

### Exercise 3: Reference an Environment Secret in a Deploy Step

**Task:** Create a deployment workflow that uses an environment-specific secret called `DEPLOY_KEY` to authenticate a deployment step. The secret should only be available to the `production` environment job.

**Hint:** Add the secret in Settings > Environments > production > Secrets. Reference it with `${{ secrets.DEPLOY_KEY }}` in the production job's steps.

**Solution:**

```yaml
name: Deploy with Secret

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - name: Deploy with key
        run: ./deploy.sh production
        env:
          DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}
```

The `DEPLOY_KEY` secret is stored in the `production` environment. Only jobs that reference `environment: production` can access it. The `test` and `build` jobs (if they existed) would not have access to this secret, preventing accidental exposure of production credentials in non-production jobs.

## Tier 2 — Intermediate

### Exercise 1: Multi-Environment Pipeline with Different Secrets per Environment

**Task:** Create a pipeline that deploys to three environments — `dev`, `staging`, and `production` — each with its own API key secret. The `dev` environment should have no protection rules, `staging` should have a branch restriction (only `main`), and `production` should require 1 reviewer.

**Hint:** Create three environments in the GitHub UI. Add `API_KEY` as a secret in each environment with different values. Reference `${{ secrets.API_KEY }}` in each job.

**Solution:**

```yaml
name: Multi-Environment Deploy

on:
  push:
    branches: [main]

jobs:
  deploy-dev:
    runs-on: ubuntu-latest
    environment: dev
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh dev
        env:
          API_KEY: ${{ secrets.API_KEY }}

  deploy-staging:
    needs: deploy-dev
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh staging
        env:
          API_KEY: ${{ secrets.API_KEY }}

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
        env:
          API_KEY: ${{ secrets.API_KEY }}
```

Each environment has its own `API_KEY` secret with a different value. The `dev` environment deploys immediately. The `staging` environment enforces the branch restriction. The `production` environment requires approval. This is the standard pattern for a progressive delivery pipeline where each environment has increasing security.

### Exercise 2: Deployment Branch Restriction for a Specific Environment

**Task:** Configure the `production` environment to only accept deployments from the `main` branch. Create a workflow that runs on push to any branch, but the production deploy should only execute when the push is to `main`.

**Hint:** Configure the branch restriction in Settings > Environments > production > Deployment branches > Selected branches. Add `main`. Also add an `if: github.ref == 'refs/heads/main'` condition to the production job.

**Solution:**

```yaml
name: Branch-Restricted Deploy

on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

The `if` condition prevents the job from even being created on non-main branches. The environment's branch restriction is a second layer of defense — if someone removes the `if` condition, the environment protection still blocks deployments from non-main branches. This defense-in-depth approach ensures production deployments are always from `main`.

### Exercise 3: Use Environment for Deployment Tracking

**Task:** Create a deployment pipeline that uses the `environment` block to provide a deployment URL and description. After deployment, verify the URL appears in the GitHub Environments page.

**Hint:** Set `environment: { name: production, url: https://your-app.io }` in the job. The URL will be clickable in the GitHub UI.

**Solution:**

```yaml
name: Tracked Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://your-app.io
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
      - name: Set dynamic URL
        run: |
          echo "DEPLOY_URL=https://${{ github.sha }}.deploy.example.com" >> $GITHUB_ENV
```

After the workflow runs, go to the repository's Environments page (Settings > Environments or the Deployments section of the repo). You should see a deployment entry for `production` with the URL and a link to the workflow run. Each deployment is recorded in the audit log, showing which commit was deployed, by whom, and at what time.

## Tier 3 — Advanced

### Exercise 1: Debug an Environment Protection Issue

**Scenario:** Your team created this production deployment workflow. The production deploy job is always being skipped with "Environment protection rule not met," even on pushes to `main`. Diagnose and fix the issue.

```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm test

  deploy-production:
    needs: build
    runs-on: ubuntu-latest
    environment: Production
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

**Issues to identify:**

1. **Environment name is case-sensitive:** The YAML says `environment: Production` but the environment in GitHub Settings is `production` (lowercase). Environment names must match exactly.
2. **Missing checkout in build job:** The `build` job runs `npm test` without checking out code first. The runner has no files to test.
3. **Missing `if` condition:** Even though the trigger is restricted to `main`, the environment's branch restriction may be configured for a different branch name. Add an explicit `if` condition.

**One valid fix:**

```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy-production:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

### Exercise 2: Design a Secure Deployment Pipeline for PCI Compliance

**Scenario:** Your company processes credit card payments and must comply with PCI DSS requirements. Design a deployment pipeline that enforces:
1. Separation of duties: the person who merges code cannot approve the production deployment
2. Two-person approval for production deployments
3. All secrets are scoped to the production environment and never exposed to lower environments
4. A complete audit trail showing who approved what and when

**Approach:**

There is no single correct answer, but a strong approach uses GitHub Environments with these configurations:

**Environment configuration (GitHub UI):**
- `production` environment: 2 required reviewers, deployment branch = `main`
- Add two teams as reviewers: `security-team` and `devops-team`
- Store production secrets only in the `production` environment
- Do not add secrets to `dev` or `staging` environments

**Workflow:**

```yaml
name: PCI-Compliant Deploy

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy-staging:
    needs: build
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://example.com
    steps:
      - uses: actions/checkout@v4
      - name: Deploy with PCI secrets
        run: ./deploy.sh production
        env:
          PCI_API_KEY: ${{ secrets.PCI_API_KEY }}
          DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
```

**Audit trail:** GitHub automatically records:
- Who triggered the workflow (the pusher)
- Who approved the production deployment (from the approval log)
- The exact commit SHA deployed
- The timestamp of every action

**Separation of duties:** The person who merges the PR cannot be the same person who approves the deployment if you configure the environment to require approval from a different team. GitHub does not enforce this natively, but you can use a CODEOWNERS file or branch protection rules to ensure the merger and approver are different people.

### Exercise 3: Implement a "Deploy Freeze" with Environment Protection

**Scenario:** Your team has a production freeze during the holiday season (December 15 to January 5). Design a mechanism that automatically blocks all production deployments during this period, without requiring any manual intervention.

**Approach:**

There are several valid approaches. Here is one using a workflow-level condition with date checking:

```yaml
name: Deploy with Freeze

on:
  push:
    branches: [main]

jobs:
  check-freeze:
    runs-on: ubuntu-latest
    outputs:
      is-frozen: ${{ steps.freeze-check.outputs.is-frozen }}
    steps:
      - id: freeze-check
        run: |
          TODAY=$(date +%s)
          FREEZE_START=$(date -d "2026-12-15" +%s)
          FREEZE_END=$(date -d "2027-01-05" +%s)
          if [ $TODAY -ge $FREEZE_START ] && [ $TODAY -le $FREEZE_END ]; then
            echo "is-frozen=true" >> $GITHUB_OUTPUT
            echo "❌ Deploy freeze active: Dec 15 - Jan 5"
          else
            echo "is-frozen=false" >> $GITHUB_OUTPUT
            echo "✅ No deploy freeze in effect"
          fi

  deploy:
    needs: check-freeze
    if: needs.check-freeze.outputs.is-frozen == 'false'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh production
```

The `check-freeze` job runs first and checks the current date against the freeze window. If the date falls within the freeze period, the `deploy` job is skipped entirely. An alternative approach uses GitHub Environments with a "deploy freeze" toggle — but since GitHub Environments do not have a built-in freeze feature, the date-check approach is more reliable.

Alternative valid approaches include:
- Using a repository variable `DEPLOY_FREEZE_ENABLED` that is set to `true` during the freeze period
- Using an environment that has no approved reviewers (removing all reviewers effectively blocks deployments)
- Using a workflow-level `if` condition that checks a GitHub environment variable
