The following exercises progress from basic workflow testing to advanced pipeline-as-code testing strategies. Each exercise includes a hint to guide you and a full solution to check against. You will need `act` (nektos/act), Docker, and the `gh` CLI installed to complete these exercises.

## Tier 1 — Beginner

These exercises focus on the fundamental skills of validating workflows locally before pushing to GitHub.

### Exercise 1.1: Validate a Workflow with `act --dry-run`

You have been given the following workflow file. Before running it on GitHub, validate it with `act --dry-run` to check for syntax errors, missing variables, and matrix expansion issues.

**Workflow file:**
```yaml
name: Data Pipeline
on: [push]
jobs:
  process:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        dataset: [users, orders, products]
        format: [csv, json]
    steps:
      - uses: actions/checkout@v4
      - name: Process ${{ matrix.dataset }} as ${{ matrix.format }}
        run: |
          echo "Processing dataset: ${{ matrix.dataset }}"
          echo "Format: ${{ matrix.format }}"
          mkdir -p output/${{ matrix.dataset }}
          echo "processed" > output/${{ matrix.dataset }}/result.${{ matrix.format }}
      - name: Upload result
        uses: actions/upload-artifact@v4
        with:
          name: ${{ matrix.dataset }}-${{ matrix.format }}
          path: output/${{ matrix.dataset }}/result.${{ matrix.format }}
```

**Hint:** Run `act --dry-run -W .github/workflows/your-workflow.yml` from the directory containing the `.github/workflows/` folder. The dry run will show the expanded matrix as separate jobs.

**Solution:**
```bash
# Save the workflow to .github/workflows/data-pipeline.yml
mkdir -p .github/workflows
cat > .github/workflows/data-pipeline.yml << 'EOF'
name: Data Pipeline
on: [push]
jobs:
  process:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        dataset: [users, orders, products]
        format: [csv, json]
    steps:
      - uses: actions/checkout@v4
      - name: Process ${{ matrix.dataset }} as ${{ matrix.format }}
        run: |
          echo "Processing dataset: ${{ matrix.dataset }}"
          echo "Format: ${{ matrix.format }}"
          mkdir -p output/${{ matrix.dataset }}
          echo "processed" > output/${{ matrix.dataset }}/result.${{ matrix.format }}
      - name: Upload result
        uses: actions/upload-artifact@v4
        with:
          name: ${{ matrix.dataset }}-${{ matrix.format }}
          path: output/${{ matrix.dataset }}/result.${{ matrix.format }}
EOF

# Run dry run
act --dry-run -W .github/workflows/data-pipeline.yml
```

Expected output:
```
[Data Pipeline/process (csv, users)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Data Pipeline/process (csv, orders)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Data Pipeline/process (csv, products)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Data Pipeline/process (json, users)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Data Pipeline/process (json, orders)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Data Pipeline/process (json, products)] 🚀  Start image=catthehacker/ubuntu:act-22.04

6 jobs to run. 0 total steps. (dry-run)
```

The dry run confirms that the workflow has 6 jobs (3 datasets x 2 formats = 6 matrix combinations). If there were a syntax error, the dry run would fail with an error message pointing to the exact line.

### Exercise 1.2: Run a Single Job Locally

Your workflow has three jobs: `lint`, `test`, and `deploy`. You want to test only the `lint` job locally to verify it works before running the full pipeline. Use `act` to run just the `lint` job.

**Workflow file:**
```yaml
name: CI
on: [push]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  deploy:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - run: echo "Deploying..."
```

**Hint:** Use `act -j lint` to run only the `lint` job. The `-j` flag specifies which job to execute. Dependent jobs (test, deploy) are not run.

**Solution:**
```bash
# Save the workflow
cat > .github/workflows/ci.yml << 'EOF'
name: CI
on: [push]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  deploy:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - run: echo "Deploying..."
EOF

# Run only the lint job
act -j lint -W .github/workflows/ci.yml
```

By running only the `lint` job, you save time and resources. The `test` and `deploy` jobs are skipped entirely. This is especially useful when you are iterating on a specific job and want fast feedback.

### Exercise 1.4: Trigger a Workflow with `gh` CLI

You have pushed a workflow to GitHub and want to trigger it manually with custom inputs. Use the `gh` CLI to trigger the workflow with specific parameters.

**Workflow file:**
```yaml
name: Manual Deploy
on:
  workflow_dispatch:
    inputs:
      environment:
        description: "Target environment"
        type: choice
        options:
          - staging
          - production
        default: staging
      version:
        description: "Version to deploy"
        type: string
        required: true
```

**Hint:** Use `gh workflow run <workflow-name> -f <key>=<value>` to pass inputs. The `-f` flag is used for each input parameter.

**Solution:**
```bash
# Trigger the workflow
gh workflow run "Manual Deploy" -f environment=staging -f version=1.2.3

# Watch the run
gh run watch

# View the logs after completion
gh run view --log
```

After running, you can verify the workflow was triggered with the correct inputs by checking the GitHub Actions tab in your browser or using `gh run list`.

### Exercise 2.1: Test Matrix Strategy with Include/Exclude

Your workflow uses a matrix strategy with both `include` and `exclude` rules. Before running the full workflow, validate the matrix expansion to ensure the correct combinations are generated.

**Workflow file:**
```yaml
name: Matrix Test
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        os: [ubuntu, windows, macos]
        node: [16, 18, 20]
        exclude:
          - os: windows
            node: 16
          - os: macos
            node: 20
        include:
          - os: ubuntu
            node: 20
            coverage: true
    steps:
      - run: echo "OS=${{ matrix.os }}, Node=${{ matrix.node }}, Coverage=${{ matrix.coverage }}"
```

**Hint:** Run `act --dry-run` and carefully count the number of jobs. Calculate the expected number manually: (3 OS x 3 Node = 9) - 2 exclusions + 1 inclusion = 8 jobs. The dry run output should show 8 jobs.

**Solution:**
```bash
# Save the workflow
cat > .github/workflows/matrix-test.yml << 'EOF'
name: Matrix Test
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        os: [ubuntu, windows, macos]
        node: [16, 18, 20]
        exclude:
          - os: windows
            node: 16
          - os: macos
            node: 20
        include:
          - os: ubuntu
            node: 20
            coverage: true
    steps:
      - run: echo "OS=${{ matrix.os }}, Node=${{ matrix.node }}, Coverage=${{ matrix.coverage }}"
EOF

# Run dry run
act --dry-run -W .github/workflows/matrix-test.yml
```

Expected output:
```
[Matrix Test/build (ubuntu, 16)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (ubuntu, 18)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (ubuntu, 20)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (windows, 18)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (windows, 20)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (macos, 16)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (macos, 18)] 🚀  Start image=catthehacker/ubuntu:act-22.04
[Matrix Test/build (ubuntu, 20, coverage)] 🚀  Start image=catthehacker/ubuntu:act-22.04

8 jobs to run. 0 total steps. (dry-run)
```

The dry run shows exactly 8 jobs. The `windows` + `node 16` and `macos` + `node 20` combinations are excluded. The `ubuntu` + `node 20` + `coverage: true` combination is included. If the calculation were wrong, the dry run would reveal the discrepancy immediately.

### Exercise 2.2: Test a Reusable Workflow Interface

You have created a reusable workflow that accepts inputs and produces outputs. Test the interface locally to verify that the inputs are correctly received and the outputs are properly set.

**Reusable workflow (`.github/workflows/reusable-build.yml`):**
```yaml
name: Reusable Build
on:
  workflow_call:
    inputs:
      node-version:
        required: true
        type: string
      build-command:
        required: false
        type: string
        default: "npm run build"
    outputs:
      build-status:
        value: ${{ jobs.build.outputs.status }}
        description: "Build result"

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      status: ${{ steps.result.outputs.status }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci
      - name: Run build
        run: ${{ inputs.build-command }}
      - id: result
        run: echo "status=success" >> $GITHUB_OUTPUT
```

**Caller workflow (`.github/workflows/caller.yml`):**
```yaml
name: Caller
on: [push]
jobs:
  call-build:
    uses: ./.github/workflows/reusable-build.yml
    with:
      node-version: "20"
      build-command: "npm run build"
  after-build:
    runs-on: ubuntu-latest
    needs: call-build
    steps:
      - run: echo "Build status: ${{ needs.call-build.outputs.build-status }}"
```

**Hint:** Run the caller workflow with `act -W .github/workflows/caller.yml`. The reusable workflow will be resolved from the local path. Check that the `after-build` job correctly receives the output.

**Solution:**
```bash
# Save the reusable workflow
cat > .github/workflows/reusable-build.yml << 'EOF'
name: Reusable Build
on:
  workflow_call:
    inputs:
      node-version:
        required: true
        type: string
      build-command:
        required: false
        type: string
        default: "npm run build"
    outputs:
      build-status:
        value: ${{ jobs.build.outputs.status }}
        description: "Build result"

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      status: ${{ steps.result.outputs.status }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci
      - name: Run build
        run: ${{ inputs.build-command }}
      - id: result
        run: echo "status=success" >> $GITHUB_OUTPUT
EOF

# Save the caller
cat > .github/workflows/caller.yml << 'EOF'
name: Caller
on: [push]
jobs:
  call-build:
    uses: ./.github/workflows/reusable-build.yml
    with:
      node-version: "20"
      build-command: "npm run build"
  after-build:
    runs-on: ubuntu-latest
    needs: call-build
    steps:
      - run: echo "Build status: ${{ needs.call-build.outputs.build-status }}"
EOF

# Run the caller
act -W .github/workflows/caller.yml
```

Expected output should show that the `call-build` job completes successfully and the `after-build` job prints "Build status: success". This confirms that the reusable workflow interface (inputs and outputs) is correctly wired.

### Exercise 2.3: Test with Custom Event Payload

Your workflow behaves differently based on the `action` field of the `pull_request` event. You need to test all three scenarios: `opened`, `synchronize`, and `closed`. Create three event payload files and test each one.

**Workflow file:**
```yaml
name: PR Handler
on:
  pull_request:
    types: [opened, synchronize, closed]

jobs:
  handle-pr:
    runs-on: ubuntu-latest
    steps:
      - name: PR action
        run: |
          echo "PR #${{ github.event.number }}"
          echo "Action: ${{ github.event.action }}"
          if [ "${{ github.event.action }}" == "opened" ]; then
            echo "New PR — run full CI"
          elif [ "${{ github.event.action }}" == "synchronize" ]; then
            echo "PR updated — run quick checks"
          elif [ "${{ github.event.action }}" == "closed" ]; then
            echo "PR closed — cleanup"
          fi
```

**Hint:** Create three separate JSON files, each with a different `action` value. Run `act` with each file using `act pull_request -e <file>.json`.

**Solution:**
```bash
# Create event payloads
cat > pr-opened.json << 'EOF'
{
  "action": "opened",
  "number": 42,
  "pull_request": {
    "head": { "ref": "feature-branch" },
    "base": { "ref": "main" }
  }
}
EOF

cat > pr-synchronize.json << 'EOF'
{
  "action": "synchronize",
  "number": 42,
  "pull_request": {
    "head": { "ref": "feature-branch" },
    "base": { "ref": "main" }
  }
}
EOF

cat > pr-closed.json << 'EOF'
{
  "action": "closed",
  "number": 42,
  "pull_request": {
    "head": { "ref": "feature-branch" },
    "base": { "ref": "main" },
    "merged": true
  }
}
EOF

# Test each scenario
echo "=== Testing PR opened ==="
act pull_request -e pr-opened.json -W .github/workflows/pr-handler.yml

echo "=== Testing PR synchronize ==="
act pull_request -e pr-synchronize.json -W .github/workflows/pr-handler.yml

echo "=== Testing PR closed ==="
act pull_request -e pr-closed.json -W .github/workflows/pr-handler.yml
```

Each run should produce different output based on the `action` field. This validates that your conditional logic correctly handles each event type without having to open and close real PRs on GitHub.

### Exercise 2.4: Verify Artifact Content Between Jobs

You have a workflow that builds an application in one job and deploys it in another. The build output is passed between jobs using artifacts. Test this workflow locally to verify that the artifact content is correct.

**Workflow file:**
```yaml
name: Build and Deploy
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Generate build
        run: |
          mkdir -p build
          echo "version: 1.0.0" > build/version.txt
          echo "commit: $(git rev-parse HEAD)" >> build/version.txt
          echo "built: $(date -u)" >> build/version.txt
      - name: Upload build
        uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: build/

  deploy:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Download build
        uses: actions/download-artifact@v4
        with:
          name: build-output
          path: build/
      - name: Verify content
        run: |
          echo "Verifying build artifact..."
          cat build/version.txt
          if grep -q "version: 1.0.0" build/version.txt; then
            echo "Version check passed"
          else
            echo "Version check failed"
            exit 1
          fi
      - name: Deploy
        run: echo "Deploying build from $(date)..."
```

**Hint:** Run the full workflow with `act`. If `act` does not fully support artifact upload/download, use the `--artifact-server-path` flag to specify a local directory for artifacts.

**Solution:**
```bash
# Run the workflow with artifact server path
act -W .github/workflows/build-deploy.yml --artifact-server-path /tmp/artifacts
```

Expected output should show both jobs succeeding. The `deploy` job should print the contents of `version.txt` and confirm the version check. After the run, you can inspect the artifacts on disk:

```bash
ls -la /tmp/artifacts/
```

If the artifact path is correctly configured, you should see the `build-output` directory with the `version.txt` file inside.

### Exercise 3.1: Build a Local Testing Harness for a Multi-Job Workflow

Your team has a complex workflow with 5 jobs in a dependency chain: `lint -> build -> test -> security-scan -> deploy`. Each job takes 2-10 minutes on GitHub. You want to build a local testing harness that lets you:

1. Run each job in isolation to test it independently.
2. Run the full chain to verify dependencies.
3. Inject failures into specific jobs to test error handling.
4. Generate a test report that shows which jobs passed and which failed.

Design a local testing approach using `act` and shell scripts.

**Reasoning-based solution:**

```bash
#!/bin/bash
# local-test-harness.sh — Test the multi-job workflow locally

WORKFLOW_FILE=".github/workflows/ci.yml"
RESULTS_DIR="test-results/$(date +%Y%m%d-%H%M%S)"
mkdir -p "$RESULTS_DIR"

echo "=== Multi-Job Workflow Test Harness ==="
echo "Workflow: $WORKFLOW_FILE"
echo "Results: $RESULTS_DIR"
echo ""

# Test 1: Run each job in isolation
echo "=== Test 1: Isolated Job Tests ==="
for JOB in lint build test security-scan deploy; do
  echo "Running job: $JOB (isolated)"
  act -j "$JOB" -W "$WORKFLOW_FILE" > "$RESULTS_DIR/$JOB-isolated.log" 2>&1
  if [ $? -eq 0 ]; then
    echo "  ✅ $JOB passed (isolated)"
    echo "$JOB:passed" >> "$RESULTS_DIR/summary.txt"
  else
    echo "  ❌ $JOB failed (isolated)"
    echo "$JOB:failed" >> "$RESULTS_DIR/summary.txt"
  fi
done

# Test 2: Run the full chain
echo ""
echo "=== Test 2: Full Chain Test ==="
act -W "$WORKFLOW_FILE" > "$RESULTS_DIR/full-chain.log" 2>&1
if [ $? -eq 0 ]; then
  echo "  ✅ Full chain passed"
else
  echo "  ❌ Full chain failed"
fi

# Test 3: Inject failure into lint (simulate lint error)
echo ""
echo "=== Test 3: Failure Injection Test ==="
echo "Injecting failure into lint job..."
# Temporarily modify the workflow to add a failing step
sed -i '/npm run lint/a\      - run: "exit 1"' .github/workflows/ci.yml
act -W "$WORKFLOW_FILE" > "$RESULTS_DIR/failure-injection.log" 2>&1
# Restore the workflow
git checkout .github/workflows/ci.yml
if grep -q "failure" "$RESULTS_DIR/failure-injection.log"; then
  echo "  ✅ Failure injection correctly detected"
else
  echo "  ❌ Failure injection was not detected"
fi

# Generate report
echo ""
echo "=== Test Report ==="
echo "Generated: $(date)"
echo "Location: $RESULTS_DIR"
cat "$RESULTS_DIR/summary.txt"
```

This harness runs three types of tests:
1. **Isolated job tests** — each job runs independently, so you can identify which specific job has a problem.
2. **Full chain test** — runs the complete dependency chain to verify that job outputs are correctly passed between jobs.
3. **Failure injection test** — intentionally breaks a job to verify that the error handling works correctly.

The harness generates a test report with timestamps and results, which you can archive for historical comparison.

### Exercise 3.2: Create a Pipeline-as-Code Test Suite

Your CI/CD pipeline is complex and changes frequently. You need a test suite that validates the pipeline itself, not just the application it builds. Design a pipeline-as-code test suite that:

1. Validates YAML syntax of all workflow files.
2. Runs `act --dry-run` on every workflow and checks that the expected number of jobs matches.
3. Runs each job in isolation and verifies it completes within a time budget.
4. Generates a pipeline health report and uploads it as an artifact.

**Reasoning-based solution:**

```yaml
name: Pipeline-as-Code Test Suite
on:
  push:
    paths:
      - ".github/workflows/**"
  pull_request:
    paths:
      - ".github/workflows/**"
  workflow_dispatch:

jobs:
  validate-yaml:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install yamllint
        run: pip install yamllint
      - name: Validate all workflow YAML files
        run: |
          echo "Validating YAML syntax..."
          for file in .github/workflows/*.yml .github/workflows/*.yaml; do
            if [ -f "$file" ]; then
              echo "Checking $file..."
              yamllint "$file"
              if [ $? -eq 0 ]; then
                echo "  ✅ $file is valid"
              else
                echo "  ❌ $file has syntax errors"
                exit 1
              fi
            fi
          done

  dry-run-all:
    runs-on: ubuntu-latest
    needs: validate-yaml
    steps:
      - uses: actions/checkout@v4
      - name: Install act
        run: |
          curl -s https://raw.githubusercontent.com/nektos/act/master/install.sh | sudo bash
      - name: Dry run all workflows
        run: |
          echo "Running dry-run on all workflows..."
          for file in .github/workflows/*.yml; do
            if [ -f "$file" ]; then
              WORKFLOW_NAME=$(basename "$file")
              echo "=== Dry run: $WORKFLOW_NAME ==="
              act --dry-run -W "$file" > /tmp/dryrun-$WORKFLOW_NAME.log 2>&1
              if [ $? -eq 0 ]; then
                echo "  ✅ $WORKFLOW_NAME dry run passed"
                # Count jobs from dry run output
                JOB_COUNT=$(grep -c "🚀" /tmp/dryrun-$WORKFLOW_NAME.log || true)
                echo "  Jobs: $JOB_COUNT"
              else
                echo "  ❌ $WORKFLOW_NAME dry run failed"
                cat /tmp/dryrun-$WORKFLOW_NAME.log
                exit 1
              fi
            fi
          done

  run-critical-jobs:
    runs-on: ubuntu-latest
    needs: dry-run-all
    strategy:
      matrix:
        workflow: [ci.yml, build.yml]
        job: [lint, build]
    steps:
      - uses: actions/checkout@v4
      - name: Install act
        run: |
          curl -s https://raw.githubusercontent.com/nektos/act/master/install.sh | sudo bash
      - name: Run ${{ matrix.job }} from ${{ matrix.workflow }}
        run: |
          echo "Running ${{ matrix.job }} from ${{ matrix.workflow }}..."
          timeout 300 act -j ${{ matrix.job }} -W .github/workflows/${{ matrix.workflow }}
          if [ $? -eq 0 ]; then
            echo "✅ ${{ matrix.job }} passed"
          else
            echo "❌ ${{ matrix.job }} failed"
            exit 1
          fi

  generate-report:
    runs-on: ubuntu-latest
    needs: [validate-yaml, dry-run-all, run-critical-jobs]
    if: ${{ always() }}
    steps:
      - name: Generate pipeline health report
        run: |
          echo "# Pipeline Health Report" > pipeline-health.md
          echo "" >> pipeline-health.md
          echo "## Summary" >> pipeline-health.md
          echo "- YAML validation: ${{ needs.validate-yaml.result }}" >> pipeline-health.md
          echo "- Dry run: ${{ needs.dry-run-all.result }}" >> pipeline-health.md
          echo "- Critical jobs: ${{ needs.run-critical-jobs.result }}" >> pipeline-health.md
          echo "" >> pipeline-health.md
          echo "## Run Details" >> pipeline-health.md
          echo "- Workflow: ${{ github.workflow }}" >> pipeline-health.md
          echo "- Run ID: ${{ github.run_id }}" >> pipeline-health.md
          echo "- Trigger: ${{ github.event_name }}" >> pipeline-health.md
          echo "- Branch: ${{ github.ref_name }}" >> pipeline-health.md
          echo "- Commit: ${{ github.sha }}" >> pipeline-health.md
      - name: Upload pipeline health report
        uses: actions/upload-artifact@v4
        with:
          name: pipeline-health-report-${{ github.run_id }}
          path: pipeline-health.md
```

This pipeline-as-code test suite treats your CI/CD configuration as production code:
- **YAML validation** catches syntax errors before they reach production.
- **Dry run** validates the job graph, matrix expansion, and variable resolution.
- **Critical job execution** runs the most important jobs in Docker containers to verify they work.
- **Health report** generates a summary with pass/fail status for each validation step.

### Exercise 3.3: Build a Local Development Loop for Pipeline Code

Your team workflow changes frequently, and each change requires a push to GitHub to test. Design a local development loop that lets you edit a workflow file and immediately re-run it with `act` without leaving your editor. The loop should:

1. Watch the `.github/workflows/` directory for changes.
2. Automatically run `act --dry-run` when a file changes.
3. If the dry run passes, run the full workflow.
4. Show the results in the terminal.

**Reasoning-based solution:**

```bash
#!/bin/bash
# pipeline-dev-loop.sh — Local development loop for pipeline code

WATCH_DIR=".github/workflows"
INTERVAL=2

echo "=== Pipeline Development Loop ==="
echo "Watching: $WATCH_DIR"
echo "Press Ctrl+C to stop"
echo ""

# Function to run the pipeline
run_pipeline() {
  local file="$1"
  echo ""
  echo "=== Change detected: $file ==="
  echo "Timestamp: $(date)"
  echo ""

  # Step 1: Validate YAML
  echo "--- Step 1: YAML Validation ---"
  if command -v yamllint &> /dev/null; then
    yamllint "$file"
    if [ $? -ne 0 ]; then
      echo "❌ YAML validation failed"
      return
    fi
    echo "✅ YAML valid"
  else
    echo "⚠️  yamllint not installed, skipping validation"
  fi

  # Step 2: Dry run
  echo ""
  echo "--- Step 2: Dry Run ---"
  act --dry-run -W "$file" > /tmp/dry-run-latest.log 2>&1
  if [ $? -ne 0 ]; then
    echo "❌ Dry run failed:"
    cat /tmp/dry-run-latest.log
    return
  fi
  echo "✅ Dry run passed"
  echo "Jobs:"
  grep "🚀" /tmp/dry-run-latest.log || true

  # Step 3: Run the workflow
  echo ""
  echo "--- Step 3: Full Run ---"
  act -W "$file" > /tmp/run-latest.log 2>&1
  if [ $? -eq 0 ]; then
    echo "✅ Pipeline passed"
  else
    echo "❌ Pipeline failed"
    echo "Last 20 lines of output:"
    tail -20 /tmp/run-latest.log
  fi

  echo ""
  echo "--- Ready for next change ---"
}

# Watch for changes
echo "Last known modification times:"
declare -A last_mtimes
for file in "$WATCH_DIR"/*.yml "$WATCH_DIR"/*.yaml; do
  if [ -f "$file" ]; then
    last_mtimes["$file"]=$(stat -c %Y "$file" 2>/dev/null || stat -f %m "$file" 2>/dev/null)
    echo "  Watching: $file"
  fi
done
echo ""

while true; do
  for file in "${!last_mtimes[@]}"; do
    if [ -f "$file" ]; then
      current_mtime=$(stat -c %Y "$file" 2>/dev/null || stat -f %m "$file" 2>/dev/null)
      if [ "$current_mtime" != "${last_mtimes[$file]}" ]; then
        last_mtimes["$file"]=$current_mtime
        run_pipeline "$file"
      fi
    fi
  done
  sleep "$INTERVAL"
done
```

This development loop gives you instant feedback on pipeline changes:
1. Edit the workflow file in your editor.
2. The loop detects the change within 2 seconds.
3. It validates YAML syntax.
4. It runs a dry run to check the job graph.
5. It executes the full workflow locally.
6. Results appear in the terminal — no push to GitHub required.

Run it with:
```bash
chmod +x pipeline-dev-loop.sh
./pipeline-dev-loop.sh
```

Then in another terminal or editor tab, edit your workflow files. Each save triggers a new test cycle. This is the fastest way to iterate on pipeline code — the feedback loop is measured in seconds, not minutes.
