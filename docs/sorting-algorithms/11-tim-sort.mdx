## Summary

Tim Sort is a "best of both worlds" method that mixes two other ideas from this course: it finds small stretches of the list that are already in order, tidies up any short stretch that isn't, and then merges all those stretches together the same way Merge Sort does. It's not just a classroom idea вҖ” it's the real method used behind the scenes by many popular programming languages' built-in "sort" button, including Python and Java, because it's genuinely fast on real-world data. It's reliably quick even in the worst case, and often much faster than that when the data already has some order to it.

## Pseudocode

```text
1. Look through the list and find short stretches that are already in
   increasing order вҖ” call each one a "run."
2. If a stretch isn't already in order (or is too short), tidy it up using
   Insertion Sort (the same one-at-a-time method from an earlier module) so
   it becomes one clean, sorted run.
3. Now you have a collection of sorted runs. Merge two runs together at a
   time, the same way Merge Sort does: compare the front of each run, take
   the smaller one, repeat until both runs are used up.
4. If one run keeps "winning" the comparison over and over, skip ahead and
   copy a whole block from it at once instead of comparing one item at a
   time вҖ” this speeds things up when one run is clearly running ahead.
5. Keep merging runs together, two at a time, until only one big sorted run
   is left. That's your sorted list.
```

## How It Works

- Look at the diagram: the list `[2, 5, 8, 1, 4, 9]` already contains two stretches that are each in increasing order on their own вҖ” `2, 5, 8` and `1, 4, 9`. Tim Sort spots these instead of ignoring them.
- Since both stretches are already sorted, there's no need to tidy them up first вҖ” Tim Sort jumps straight to merging them.
- Merging works by comparing the front of each stretch and picking the smaller one each time: `1` beats `2`, so `1` goes first, then `2` beats `4`, and so on, until every number has been placed in the new list.
- If one stretch kept winning many times in a row, Tim Sort would jump ahead and grab a whole chunk from it at once instead of comparing one number at a time вҖ” this is what makes it fast on data that already has a lot of order in it.
- This method needs extra room to build the merged list вҖ” it doesn't rearrange the original list directly in place.
- If two numbers are exactly equal, this method keeps them in the same left-to-right order they started in, because merging always takes from the left stretch first when there's a tie.
- How fast is it? If the list already has long sorted stretches in it, this is very fast вҖ” barely more work than reading through the list once. Even in the worst case, with no helpful order at all, it's never worse than Merge Sort's pace (`O(n log n)`), so it never turns painfully slow the way some simpler methods can.
