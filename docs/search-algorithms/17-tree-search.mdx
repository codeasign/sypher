## Summary

Tree Search looks for a value in a branching tree structure by exploring one branch all the way down before backing up and trying the next one, keeping track of the exact trail of values it followed. Think of it like exploring a family tree, checking a person, then all of their descendants down one side, before moving to the other side. Because the values aren't arranged in any particular order, there's no shortcut — the search has to check nodes one at a time until it finds a match, so it gets slower the more nodes the tree has.

## Pseudocode

```text
1. Build the tree first: turn the flat list of numbers into a tree of connected nodes, where each number's two "children" come from specific positions further along in the list.
2. Start at the root node. Add its value to your trail.
3. If this node's value is the target, stop — you found it, and the trail so far is your answer.
4. Otherwise, try going left: repeat this whole process on the left child.
5. If going left found the target, stop and report the trail.
6. Otherwise, try going right: repeat this whole process on the right child.
7. If going right found the target, stop and report the trail.
8. If neither side found it, remove this node's value from the trail (you're backing out of a dead end) and report "not found" back up to whoever called you.
9. If you run out of tree (an empty spot), that spot has nothing to check — immediately report "not found."
```

## How It Works

- Start at the root, `5`, in the diagram. Add it to the trail: `[5]`. It's not `4`, so go left to `3`.
- Add `3` to the trail: `[5, 3]`. Not a match, so go left again, to `1`.
- Add `1` to the trail: `[5, 3, 1]`. Not a match, and `1` has no children to check — so this branch is a dead end. Remove `1` from the trail, leaving `[5, 3]`, and go back up to try `3`'s other child instead.
- Now try `3`'s right child, `4`. Add it to the trail: `[5, 3, 4]`. It matches. Stop here — `[5, 3, 4]` is the answer.
- Notice the tree isn't sorted by value — there's no rule saying smaller numbers go left and bigger numbers go right. That's why the search can't skip a branch just by comparing numbers; it has to actually check both children of every node it visits.
- How fast is it? If the target happens to be the very first node checked (the root), it's instant. If the target is missing entirely, or hiding in the last branch checked, the search has to visit every single node in the tree before it can be sure (this is `O(n)` time, where n is the number of nodes) — there's no ordering trick to skip ahead.
