## Summary

Interpolation Search finds something in a sorted list of numbers by guessing where the target probably is, based on its value, instead of always guessing the middle. Think of a phone book: if you're looking for "Zamora," you'd flip near the very end, not the middle, because you know the alphabet is spread out evenly. This algorithm does the same math trick with numbers. When the numbers are spread out evenly, it's even faster than Binary Search — but if the numbers are bunched up unevenly, its guesses get bad and it can slow down a lot.

## Pseudocode

```text
1. Set "low" at the start and "high" at the end of the list.
2. While low hasn't passed high:
3. If every remaining number in the range is the same value, just check whether it equals the target directly, and stop.
4. Otherwise, guess a position based on how close the target's value is to the low end versus the high end of the current range — like estimating a page number by value.
5. If that guessed position falls outside the current range, report "not found."
6. If the item at the guessed position is the target, stop and report its position.
7. If it's smaller than the target, keep searching to the right of the guess.
8. If it's bigger than the target, keep searching to the left of the guess.
9. If low ever passes high without a match, report "not found."
```

- Comparison: Binary Search always guesses the middle no matter what the values are. Interpolation Search uses the target's actual value to make a smarter guess.

## How It Works

- We're looking for `90` in the ten boxes above, which run from `10` up to `100`. Since `90` is close to the high end of that range, we guess the target is near the end of the list too — not in the middle like Binary Search would.
- The guess lands on index 8, which holds `90` — a direct match. We stop and report position 8.
- If the guess had been wrong, we'd narrow the range just like Binary Search does: throw away the side that can't contain the target, and guess again with the smaller range.
- This only works if the list is sorted and holds numbers, and it only works *well* if those numbers are spread out fairly evenly. If a few numbers are huge outliers (say, mostly numbers under 100 but one number in the millions), the guesses can go very wrong and land far from the target. There's also a special case: if every number left in the range is identical, the algorithm just checks that value directly instead of trying to guess a position.
- How fast is it? On evenly spread-out numbers, this is often faster than Binary Search — each guess tends to land very close to the real answer (this is called `O(log log n)` time, an even smaller number of steps than Binary Search's `O(log n)`). But on unevenly spread-out or clustered numbers, the guesses can be repeatedly bad, and in the worst case it slows all the way down to checking almost every item, one at a time (`O(n)` time, same as Linear Search).
