## Summary

Jump Search finds something in a sorted list by leaping forward in fixed-size chunks (blocks) instead of checking every item, and then walking normally through the one block where the target must be hiding. Imagine flipping through a phone book by jumping ahead 20 pages at a time until you overshoot the name you want, then flipping page by page from there. It's slower than Binary Search, but it only ever moves forward — never jumps backward — which matters for some kinds of data. It's faster than checking every item one by one, but not as fast as Binary Search.

## Pseudocode

```text
1. If the list is empty, report "not found" right away.
2. Pick a block size — roughly the square root of the list's length.
3. Starting from the beginning, leap forward by one block size at a time, each time checking the last item in the block you just landed on.
4. Keep leaping forward as long as that last item is still smaller than the target.
5. Once a block's last item is no longer smaller than the target (or you reach the end of the list), stop leaping — the target must be in this block, if it's anywhere.
6. Walk through that one block item by item, checking each one against the target.
7. If you find a match, report its position. If you reach the end of the block with no match, report "not found."
```

## How It Works

- We're hunting for `13` in the nine boxes above, split into three blocks of three. The block size (3) is picked to be roughly the square root of how many items there are — that's the size that keeps the leaping and the walking each about equally cheap.
- We check the last item of block 0, which is `5`. Since `5` is smaller than `13`, the target can't be in this block — we leap ahead to block 1.
- We check the last item of block 1, which is `11`. Still smaller than `13` — leap ahead again to block 2.
- Block 2's last item is `17`, which is not smaller than `13`, so we stop leaping. The target must be somewhere inside block 2 (indices 6 through 8), if it's anywhere at all.
- Now we walk through block 2 item by item: index 6 holds `13` — a match. We stop and report position 6.
- This only works if the list is sorted — the "still smaller, keep leaping" trick depends on values only ever going up as you move right.
- How fast is it? Best case, the target sits right at the front and you find it almost immediately. Worst case, you leap through most of the blocks and then walk through one whole block — but because the block size is tuned to the square root of the list's length, that worst case is still much better than checking every item (this is called `O(sqrt n)` time — slower than Binary Search's `O(log n)`, but faster than Linear Search's `O(n)`).
