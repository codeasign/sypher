# Algorithm Basics

New to algorithms? Start here. This page explains the handful of ideas every
module in this course assumes you already know — what an algorithm is, how we
talk about "fast" and "slow," and what it means for data to be "sorted." None
of this is search-specific yet; it's the vocabulary the rest of the course is
written in.

If you've taken a data structures class before, skim this page and jump to
[Module 1: Linear Search](/docs/search-algorithms/linear-search).

## What Is an Algorithm?

An algorithm is just a precise, step-by-step recipe for solving a problem —
the same way a cooking recipe tells you exactly what to do, in what order, so
that anyone following it gets the same result. "Search algorithms" are
recipes for one specific problem: given a collection of items, find whether a
particular value is in there, and if so, where.

Every module in this course gives you the same three things for one recipe:
**pseudocode** (the recipe written in plain steps), an **explanation** of why
it works, and **runnable code** so you can try it yourself.

## Why Speed Matters: Reading Big-O Notation

You'll see notation like `O(n)` or `O(log n)` in every module. This is called
**Big-O notation**, and all it measures is: *if the input gets bigger, how
much more work does the algorithm have to do?* It's not a stopwatch time —
it's a growth pattern.

Here's what the ones you'll actually see mean, with an everyday comparison:

- **`O(1)` — constant time.** The work doesn't change no matter how big the
  input is. Like knowing a friend's phone number by heart — it takes the same
  half-second whether your contacts list has 10 entries or 10,000.
- **`O(log n)` — logarithmic time.** The work grows *very* slowly as the
  input grows, because each step eliminates half of what's left. Like
  guessing a number between 1 and 1,000 when someone tells you "higher" or
  "lower" after each guess — you can find it in about 10 guesses, not 1,000.
  (This is exactly how Binary Search, Module 2, works.)
- **`O(n)` — linear time.** The work grows in direct proportion to the input.
  Twice the input, twice the work. Like checking every locker in a hallway
  one by one until you find the one that's unlocked — no shortcuts, you just
  keep going. (This is Linear Search, Module 1.)
- **`O(n^2)` — quadratic time.** The work grows with the *square* of the
  input. Ten times the input means a hundred times the work. This shows up
  when an algorithm compares every item to every other item.

The table below makes the difference concrete — it shows roughly how many
steps each pattern needs as the input grows:

| Input size (`n`) | `O(1)` | `O(log n)` | `O(n)` |
|---|---|---|---|
| 10 | 1 | ~3 | 10 |
| 1,000 | 1 | ~10 | 1,000 |
| 1,000,000 | 1 | ~20 | 1,000,000 |

The lesson: an algorithm that's `O(log n)` barely slows down as data grows,
while an `O(n)` algorithm slows down in direct proportion. Neither is "wrong"
— Module 1 (Linear Search) and Module 2 (Binary Search) both solve the same
problem, and the tradeoff between them is one of the first things this course
walks through.

## Why Memory Matters: Space Complexity

Alongside time, every module also lists a **space** complexity — how much
*extra* memory the algorithm needs beyond the input itself. `O(1)` space
means the algorithm only ever needs a small, fixed number of extra variables
(like a single counter), no matter how large the input gets. `O(n)` space
means it needs to build something roughly proportional in size to the input
(like a second array, or a set to track visited items).

## Sorted vs. Unsorted Data

One idea comes up constantly in this course: some algorithms require the
data to already be **sorted** (arranged smallest to largest, or by some
consistent rule), and some don't care at all.

Linear Search (Module 1) works on data in *any* order, sorted or not — that's
its whole appeal. Binary Search (Module 2) is much faster, but only works
*because* it can assume the data is already sorted; if you handed it unsorted
data, it would give wrong answers. Watching for "does this algorithm require
sorted input?" is one of the first questions worth asking about any new
algorithm you meet in this course.

## How This Course Is Organized

Every module from here on follows the same shape:

1. **Summary** — a couple of plain-English sentences on what the algorithm
   does, usually with an everyday analogy, so you know the big idea before
   anything gets technical.
2. **Pseudocode** — the algorithm as a short, language-neutral recipe.
3. **ASCII Diagram** — a picture that walks through the algorithm running on
   a real example, step by step.
4. **How It Works** — a plain-English breakdown of the diagram: what
   happens, why it works, and roughly how fast it is and how much memory it
   needs (the Big-O costs, explained above) — no formal proofs, just the
   intuition.
5. **Code** — a real, runnable implementation in a browser code editor,
   supporting nine different languages (Python, C, C++, Java, C#,
   JavaScript, TypeScript, Rust, and Go). You only need to know *one* of them
   — pick whichever you're most comfortable with and ignore the rest. Python
   is selected by default.
6. **Solutions** — a reference implementation in every language, in case you
   get stuck or just want to compare notes after writing your own.
7. **Practice** — five test cases you can run directly in the editor to check
   that your own implementation works.

Read the summary and pseudocode first, look at the diagram until the example
genuinely makes sense to you, then read "How It Works" — *then* write the
code. Trying to write code before understanding the idea behind it is the
hard way to learn an algorithm.
