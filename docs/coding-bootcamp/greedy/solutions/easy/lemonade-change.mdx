# Lemonade Change — Solution

## Problem Overview

Each lemonade costs **$5**. Customers pay with **$5**, **$10**, or **$20** bills. You must provide correct change to each customer. You start with no change on hand. Return `true` if you can give every customer correct change, `false` otherwise.

## Brute Force Approach

**Idea:** Simulate all possible sequences of giving change. For each $20 bill, try both giving one $10 + one $5, or three $5s. Backtrack if a choice leads to a dead end.

**Algorithm:**
1. Maintain counts of $5 and $10 bills
2. For each customer, try every valid change combination recursively
3. If a path succeeds, return `true`; if all paths fail, return `false`

**Complexity Analysis:**
- **Time:** O(2^k) where k is the number of $20 bills — each $20 creates a branching choice
- **Space:** O(n) — recursion stack depth

**Why this could be better:** The brute force approach explores unnecessary branches. The greedy strategy always picks the optimal change combination immediately, avoiding backtracking entirely.

## Optimal Approach: Greedy Bill Selection

**Idea:** Track the count of $5 and $10 bills. For each transaction, give change using the **largest denominations possible first** — this preserves $5 bills for future customers.

**Algorithm:**
1. Initialize `five = 0`, `ten = 0`
2. For each `bill` in `bills`:
   - **$5 bill:** increment `five` (no change needed)
   - **$10 bill:** if `five > 0`, give one $5 as change, increment `ten`; otherwise return `false`
   - **$20 bill:** prefer giving one $10 + one $5 (saves $5 bills); if not possible, try three $5s; if neither works, return `false`
3. Return `true` if all customers were served

**Why This Works:** The greedy choice of using a $10 bill before three $5s when giving change for $20 is optimal because $10 bills are useless for change when a customer pays with $10 (they need $5 back) or $5 (no change needed). Only $5 bills provide the flexibility needed for all transaction types. By preserving $5 bills whenever possible, we maximize our ability to make change.

**Complexity Analysis:**
- **Time:** O(n) — single pass through the bills array
- **Space:** O(1) — only two integer counters

## Deep Dive: How the Greedy Strategy Works

### Visual Walkthrough

For `bills = [5, 5, 5, 10, 20]`:

```
Start:    five=0, ten=0

Bill $5:  five=1, ten=0   (no change needed)
Bill $5:  five=2, ten=0   (no change needed)
Bill $5:  five=3, ten=0   (no change needed)
Bill $10: five=2, ten=1   (give back $5)
Bill $20: five=1, ten=0   (give back $10 + $5 — prefer this over 3 × $5)
Result: true
```

For `bills = [5, 5, 10, 10, 20]`:

```
Start:    five=0, ten=0

Bill $5:  five=1, ten=0
Bill $5:  five=2, ten=0
Bill $10: five=1, ten=1   (give back $5)
Bill $10: five=0, ten=2   (give back $5)
Bill $20: five=0, ten=2   → need $15 change
          Try: one $10 + one $5  → fail (no $5)
          Try: three $5s         → fail (no $5)
Result: false
```

### Edge Cases

- **Single customer with $5**: Always works — `five` becomes 1, return `true`
- **Single customer with $10**: Fails immediately — no $5 to give as change
- **Single customer with $20**: Fails immediately — no change available
- **All $5 bills**: Always works — `five` just keeps increasing
- **Large $20 followed by $5s**: If we used three $5s when we had a $10 available, we'd be out of $5s for $10 customers later. The greedy choice of `$10 + $5` first prevents this.

### Why Not Track $20 Bills?

$20 bills are never given as change (customers only pay with $5, $10, or $20 — maximum change required is $15). Tracking them would waste memory and add no value.

### Greedy Choice Proof

At a $20 transaction, if we have both a $10 and enough $5s, choosing `$10 + $5` is always at least as good as `3 × $5`. After `3 × $5`, we lose three $5s; after `$10 + $5`, we lose one $5 and one $10. Since $5s are needed for both $10 and $20 transactions while $10s are only useful for $20 transactions, preserving $5s strictly dominates preserving $10s. No optimal solution ever needs to use three $5s when a $10 is available.

```python
def lemonade_change(bills: list[int]) -> bool:
    five = 0  # count of $5 bills
    ten = 0   # count of $10 bills

    for bill in bills:
        if bill == 5:
            five += 1  # no change needed, just collect the $5
        elif bill == 10:
            if five == 0:
                return False  # cannot give $5 change
            five -= 1  # give one $5 as change
            ten += 1   # collect the $10
        else:  # bill == 20
            # Prefer to give one $10 + one $5 (preserves $5 bills)
            if ten > 0 and five > 0:
                ten -= 1
                five -= 1
            # Fall back to three $5s if no $10 available
            elif five >= 3:
                five -= 3
            else:
                return False  # cannot make $15 change
    return True  # all customers received correct change
```

## Key Takeaways

- **Greedy choice**: when giving change for $20, use a $10 bill first (along with a $5) rather than three $5 bills. This preserves $5 bills which are needed for other transactions.
- **Only $5 and $10 matter**: $20 bills are never given as change, so we don't track them.
- **Early exit**: return `false` as soon as change cannot be made — no need to process remaining customers.
- **Proof of optimality**: $5 bills are the most versatile (needed for change on both $10 and $20 payments), so saving them whenever possible is always optimal.
- **Edge cases**: single customer, all $5 bills, starting with a $10 or $20.
