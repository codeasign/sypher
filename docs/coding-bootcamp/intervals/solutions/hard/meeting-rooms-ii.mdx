# Meeting Rooms II — Solution

## Problem Overview

Given an array of meeting intervals, find the minimum number of conference rooms required to host all meetings without conflicts. This is equivalent to finding the maximum number of overlapping intervals at any point in time — the peak concurrency tells us how many rooms are needed.

## Brute Force Approach

**Idea:** For each meeting, count how many other meetings are active at the same time. Track the maximum count.

**Algorithm:**
1. For each interval `i`, count how many other intervals `j` overlap with it
2. Two intervals `[a, b]` and `[c, d]` overlap if `max(a, c) < min(b, d)`
3. Track the maximum overlap count across all intervals
4. Return the maximum count

**Complexity Analysis:**
- **Time:** O(n²) — we compare each pair of intervals
- **Space:** O(1) — no extra space

**Why this could be better:** The brute force works but is O(n²). For n up to 10⁴, we need an O(n log n) solution.

## Optimal Approach 1: Two Pointers (Chronological Ordering)

**Idea:** Separate start times and end times into two arrays. Sort both. Use two pointers to track rooms in use: when a meeting starts, we need a room; when a meeting ends, a room frees up.

**Algorithm:**
1. Create arrays `starts` and `ends` from the intervals
2. Sort both arrays
3. Initialize `start_ptr = 0`, `end_ptr = 0`, `rooms = 0`
4. While `start_ptr < n`:
   - If `starts[start_ptr] < ends[end_ptr]`: a meeting starts before the earliest ends → need a new room. Increment `rooms` and `start_ptr`
   - Else: a meeting has ended → free a room. Increment `end_ptr` and `start_ptr`
5. Return `rooms`

**Why This Works:** By processing events in chronological order, we simulate the flow of meetings occupying and releasing rooms. The two-pointer technique lets us process all events in O(n) after sorting, giving us O(n log n) overall.

**Complexity Analysis:**
- **Time:** O(n log n) — dominated by sorting
- **Space:** O(n) — for the two arrays

## Optimal Approach 2: Min-Heap (Priority Queue)

**Idea:** Sort intervals by start time. Use a min-heap to track the end times of meetings currently using a room. For each meeting, free rooms whose end time has passed, then assign a new room.

**Algorithm:**
1. Sort intervals by start time
2. Initialize a min-heap (priority queue)
3. For each interval `[start, end]`:
   - Remove all end times from the heap that are `<= start` (rooms freed)
   - Add `end` to the heap (a new room is booked)
   - Track the maximum heap size (peak concurrent rooms)
4. Return the maximum heap size observed

**Complexity Analysis:**
- **Time:** O(n log n) — sorting + heap operations
- **Space:** O(n) — for the heap

## Deep Dive: How the Two Pointers Approach Works

### Visual Walkthrough

For the input `intervals = [[0, 30], [5, 10], [15, 20]]`:

```
Step 1: Create and sort start/end arrays
        starts = [0, 5, 15]
        ends   = [10, 20, 30]

Step 2: Two pointer traversal
        start_ptr=0, end_ptr=0, rooms=0

        starts[0]=0 < ends[0]=10 → new room needed
        rooms=1, start_ptr=1

        starts[1]=5 < ends[0]=10 → new room needed
        rooms=2, start_ptr=2

        starts[2]=15 < ends[0]=10? No → room freed
        end_ptr=1, start_ptr=3

        start_ptr=3, start_ptr >= n → stop

        Maximum rooms = 2
```

### Visual Walkthrough: Heap Approach

For the same input:

```
Sorted by start: [(0,30), (5,10), (15,20)]

Step 1: Process (0,30)
        Heap = []
        No rooms to free (heap empty)
        Add 30 → Heap = [30]
        Max rooms = 1

Step 2: Process (5,10)
        Heap = [30]
        Check top: 30 <= 5? No → no room freed
        Add 10 → Heap = [10, 30]
        Max rooms = 2

Step 3: Process (15,20)
        Heap = [10, 30]
        Check top: 10 <= 15? Yes → free room
        Heap = [30]
        Check top: 30 <= 15? No
        Add 20 → Heap = [20, 30]
        Max rooms = 2

Result: 2 rooms
```

### Edge Cases

- **Empty input (n = 0):** No meetings, 0 rooms needed
- **Single meeting:** 1 room needed
- **No overlaps (sequential meetings):** 1 room — just reuse the same room
- **All meetings overlapping:** n rooms needed — each meeting gets its own room
- **Back-to-back meetings:** `[0, 5], [5, 10]` — 1 room, since the first ends exactly when the second starts
- **Negative times:** Handled naturally since sorting works on all integers

### Why Two Pointers vs. Heap

The two-pointer approach is simpler and slightly faster (no heap overhead), but works only because we only care about the peak count — not which specific rooms are assigned. The heap approach is more general and can be extended to actually assign specific rooms to meetings.

import java.util.Arrays;

class Solution {
    public int minMeetingRooms(int[][] intervals) {
        if (intervals.length == 0) return 0;

        int n = intervals.length;
        int[] starts = new int[n];
        int[] ends = new int[n];

        // Separate start and end times
        for (int i = 0; i < n; i++) {
            starts[i] = intervals[i][0];
            ends[i] = intervals[i][1];
        }

        // Sort both arrays
        Arrays.sort(starts);
        Arrays.sort(ends);

        int rooms = 0;
        int endPtr = 0;

        // Process each start time
        for (int startPtr = 0; startPtr < n; startPtr++) {
            if (starts[startPtr] < ends[endPtr]) {
                // A meeting starts before the earliest ends → need a new room
                rooms++;
            } else {
                // A meeting has ended → free that room
                endPtr++;
            }
        }
        return rooms;
    }
}

## Key Takeaways

- **Minimum rooms = maximum concurrency** — the answer is the peak number of overlapping intervals at any point in time
- **Two pointers (chronological ordering)** is an elegant O(n log n) solution: separate starts and ends, sort both, and sweep
- **Min-heap approach** is more general and can be adapted for room assignment, not just counting
- The overlap condition `start < end_ptr_end` determines when a new room is needed
- This problem builds directly on Meeting Rooms and Merge Intervals — it is the hardest of the three classic interval problems
