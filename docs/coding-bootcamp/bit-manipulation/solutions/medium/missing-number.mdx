# Missing Number — Solution

## Problem Overview

Given an array `nums` containing `n` distinct numbers in the range `[0, n]`, find the one number that is missing from the range.

## Brute Force Approach

**Idea:** Use a hash set to track which numbers are present, then check which number from 0 to n is missing.

**Algorithm:**
1. Add all numbers from `nums` into a hash set
2. For each number from 0 to n:
   - If it's not in the set, return it
3. Return -1 (should never reach here)

**Complexity Analysis:**
- **Time:** O(n) — building the set and checking all numbers
- **Space:** O(n) — storing n elements in the hash set

**Why this could be better:** The hash set approach works but uses O(n) extra space. Both the math and XOR approaches solve it in O(1) space.

## Optimal Approach Using XOR

**Idea:** XOR all numbers from 0 to n with all numbers in the array. The pairs cancel out, leaving only the missing number.

**Why This Works:**
- XOR of a number with itself is 0: `x ^ x = 0`
- We XOR the indices (0 to n) with the values (nums)
- The missing number appears only once (as an index), while all others appear twice (once as index, once as value)
- The result is the missing number

**Algorithm:**
1. Initialize `missing = n` (since the range goes up to n)
2. For each `i` from 0 to n-1:
   - `missing ^= i ^ nums[i]`
3. Return `missing`

**Trace for `nums = [3, 0, 1]`:**
```
missing = 3  (n = 3)
i=0: missing ^= 0 ^ 3 = 3 ^ 0 ^ 3 = 0
i=1: missing ^= 1 ^ 0 = 0 ^ 1 ^ 0 = 1
i=2: missing ^= 2 ^ 1 = 1 ^ 2 ^ 1 = 2
Result: 2  ← the missing number
```

**Complexity Analysis:**
- **Time:** O(n) — single pass
- **Space:** O(1) — only one integer variable

## Alternative Optimal Approach: Sum Formula

**Idea:** Compute the expected sum `n * (n + 1) / 2` and subtract the actual sum of the array.

**Algorithm:**
1. Compute `expected = n * (n + 1) // 2`
2. Compute `actual = sum(nums)`
3. Return `expected - actual`

**Complexity Analysis:**
- **Time:** O(n) — single pass to compute the sum
- **Space:** O(1) — only two integer variables

**Caveat:** This approach can overflow for large n in languages with fixed-width integers. The XOR approach avoids overflow entirely.

## Deep Dive

### Why XOR Is Better Than Sum

For very large `n` (up to 10⁴ in this problem, but potentially larger), the sum formula `n * (n + 1) / 2` can overflow a 32-bit integer. For example, with `n = 65536`, the expected sum is `65536 * 65537 / 2 ≈ 2.1 × 10⁹`, which fits in 32 bits, but for `n = 100000`, the sum is `~5 × 10⁹`, which exceeds the signed 32-bit limit.

The XOR approach never overflows because XOR operates on bits individually and never produces a carry beyond the bit width.

### Edge Cases

- **Missing 0:** When 0 is the missing number, the XOR result will be 0 (since `n ^ n ^ ... ^ 0 = 0`)
- **Missing n (last element):** When the last element in range is missing, the XOR will produce n
- **Single element array:** `nums = [0]` → missing = 1; `nums = [1]` → missing = 0

class Solution {
    public int missingNumber(int[] nums) {
        int n = nums.length;
        int missing = n;                    // Start with n (the last index)
        for (int i = 0; i < n; i++) {       // Iterate through array

## Key Takeaways

- **XOR with index** is a clever extension of the XOR cancellation pattern
- The XOR approach avoids integer overflow that the sum formula can encounter
- The same pattern works for any problem where you need to find a missing element in a range
- The solution is O(n) time and O(1) space — optimal
