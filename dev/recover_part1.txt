## Internal Architecture of Consensus

Consensus protocols solve the fundamental problem of distributed consistency: how do multiple nodes agree on a value when the network can fail, nodes can crash, and messages can be delayed or reordered?

The Raft consensus algorithm is the most widely deployed protocol. Its architecture breaks down into three layers.

### Layer 1: Leader Election

Every node in a Raft cluster starts in the Follower state. Followers expect periodic heartbeats from a Leader. If a Follower receives no heartbeat within an election timeout (150-300ms), it transitions to Candidate, requests votes from all other nodes, and if a majority votes for it, becomes the Leader.

### Layer 2: Log Replication

Once a node is Leader, all write requests flow through it. The Leader appends each command to its log as an entry, then sends AppendEntries RPCs to all Followers.

### Layer 3: Commitment and Application

The Leader tracks which log entries have been replicated to a majority of Followers. When a majority has acknowledged, the Leader advances the commit index and applies the entry to its state machine. A future Commit RPC tells Followers the entry is committed, and they apply it too.

### Paxos vs Raft: Architectural Comparison
